## Summary

The reporter raises a usability/clarity concern with the **DiagnosticReport.code**
element as profiled by the HL7 **Laboratory Report (FHIR)** IG. In that IG,
`DiagnosticReport.code` is bound to a *Laboratory Report Types* value set that
contains a single allowed value — "Laboratory Report" — while the **category**
slice on the same profile is what actually carries the specific report types
and is intended to be used for filtering. Because `code` therefore conveys no
discriminating information beyond "this is a laboratory report" (which is
already implied by the profile and the category slice), the reporter asks
whether the element should simply be **fixed** to the single allowed concept
("Laboratory Report") with the definition updated accordingly, rather than
left as a bound-but-effectively-single-valued element that implementers must
populate redundantly.

The ticket has **no linked Jira tickets** and no Jira/Zulip/GitHub items
appear in the cross-reference index — only the FHIR artifact
`DiagnosticReport.code` is cross-referenced. There is, however, a substantial
body of related community discussion about how `DiagnosticReport.code` should
be used in lab contexts (Argonaut/US Core, IPS, EU Lab, AU, IT Lab Report,
UK), summarized below; the IT Lab Report IG already takes precisely the
"fix to 11502-2 Laboratory report" approach this ticket suggests.

## Details

**Description:**

> The code attribute is associated with the Laboratory Report Types value set
> which has one allowed value "Laboratory Report".  It is unclear how this
> attribute is to be used especially since the category slice contains the
> specific types of reports and, according to the value set definition,
> should be used for filtering.  If the intent of the attribute is to capture
> the type of DiagnosticReport, should the attribute be changed to a fixed
> value of Laboratory Report and the definition updated accordingly?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **Laboratory Report (FHIR)** IG, modify the `DiagnosticReport`
profile so that `DiagnosticReport.code` is **fixed** to the single concept
allowed by the current "Laboratory Report Types" value set — i.e., LOINC
**`11502-2` "Laboratory report"** — using a `patternCodeableConcept` (or
`fixedCodeableConcept`) on the profile element. Drop the value-set binding
on the profile in favour of the fixed value, and update the element
**short** and **definition** to read along the lines of:

> "The fixed code identifying this resource as a Laboratory Report. The
> *kind* of laboratory report (e.g., chemistry, microbiology, pathology)
> is conveyed via `DiagnosticReport.category`; specific tests/panels are
> carried via `DiagnosticReport.result` Observation `code`s and may also
> be referenced via `Observation.hasMember`."

Add a short usage note on the profile page explaining that filtering on
report sub-type is intended to use `category` (not `code`), and that
implementers populating `.code` can simply set `11502-2`.

#### Justification

- The current binding offers no implementation choice (single allowed
  value), so consumers cannot filter or differentiate on `.code`; the
  filtering signal lives on `category`, as the IG already states.
- This is a precedented pattern: the **Italian Lab Report IG** already
  fixes `DiagnosticReport.code = 11502-2` ("Laboratory report"), per the
  O&O Zulip thread above. Aligning with that approach harmonises lab
  report typing across IGs.
- Fixing the value removes a class of avoidable validation / data-quality
  errors (implementers selecting arbitrary LOINCs in a single-valued
  binding) and makes the IG self-describing.
- The change is documentation-bearing and tightening — implementations
  already supplying "Laboratory report" remain valid; those supplying
  other codes were not actually conformant under the current binding.

### Disposition B: Alternative Approach

#### Proposal

Keep `DiagnosticReport.code` as a coded element but **rebind** it to the
broader **LOINC document/report type set** appropriate for laboratory
documents (e.g., the LOINC ordinal of report-level codes used by
`Composition.type` for laboratory documents — `11502-2` plus its
sub-classes such as `11526-1` "Pathology study", `18717-9` "Genetic
analysis report", `26436-6` "Laboratory studies (set)" etc.), and
explicitly document that:

- `DiagnosticReport.code` carries the **report-document-type** axis
  (analogous to `Composition.type`),
- `DiagnosticReport.category` carries the **service category** used for
  filtering (LAB and its specialties),
- `DiagnosticReport.result.code` carries the **specific tests / panels**
  performed.

Tighten the cardinality/usage guidance accordingly and align the binding
with the corresponding IPS / EU Lab Report / Composition.type guidance.

#### Justification

- Preserves an actual coding axis on `.code` (rather than collapsing it
  to a fixed value), which several implementer threads — UK, EU, and the
  Argonaut / US Core thread — argue is useful for distinguishing common
  varieties of lab report (clinical chemistry vs. pathology vs. genomics)
  at the document level.
- Aligns the Lab Report IG with the broader FHIR/IHE/IPS pattern of
  using `Composition.type` for document typing and keeps the IG
  composable with IPS lab sections and EU Laboratory Report.
- Avoids the (small) compatibility risk of a hard fixed value for
  systems that today emit, for example, `11526-1` "Pathology study" in
  `DiagnosticReport.code` for pathology cases and rely on it
  downstream.

### Disposition C: Decline

#### Proposal

Leave `DiagnosticReport.code` and its value-set binding unchanged in the
Laboratory Report (FHIR) IG. Address the reporter's concern purely
through **non-normative documentation**: add a usage note stating that
`.code` is constrained to a single concept ("Laboratory Report") because
the IG profiles the resource as a laboratory report, and that
sub-classification and filtering are intended to occur on
`.category`/`.result.code`. Optionally close the ticket as
"Persuasive with Modification — documentation only".

#### Justification

- A `Required` binding to a single-valued value set is functionally
  equivalent to a fixed value at validation time; downstream behaviour
  does not actually change with Disposition A.
- Keeps the IG model open to future expansion of the value set (e.g.,
  to add report-document-type codes later) without a normative profile
  change.
- Aligns with a "minimum-change" stance for a balloted IG and avoids
  reopening cross-IG harmonisation discussions (US Core, IPS, IT, EU,
  AU, UK) that the Zulip threads show are unsettled.

---

### Recommendation

**Recommended disposition:** **A — Accept As Requested**

The reporter is correct that, as written, `DiagnosticReport.code` provides
no implementation latitude and no discriminating information in this IG —
the single allowed value, the category-as-filter intent, and the explicit
question on the ticket all point in the same direction. Fixing the value
to LOINC `11502-2` and rewriting the definition to make the
`code`/`category`/`result.code` division of labour explicit:

- Eliminates the ambiguity the ticket flags (the entire reason it was
  filed and labelled `block-vote-5`).
- Aligns the HL7 Laboratory Report IG with the **Italian Lab Report IG**,
  which already takes this exact approach — this is the strongest
  precedent surfaced by the related Zulip discussion (O&O WG stream).
- Is a tightening change with no new constraints implementers cannot
  meet today (any system populating "Laboratory report" remains
  conformant; any other code was already non-conformant under the
  current binding).
- Is small, locally-contained, and does not require the IG to take a
  position on the cross-IG `code` vs. `category` debate that
  Disposition B would inevitably reopen.

Disposition B is the right *long-term* answer if the workgroup decides
the IG should carry an active report-document-type axis, but it should
be staged as a follow-up after coordinating with IPS / EU Lab / US Core
rather than bundled into this ticket. Disposition C technically
resolves the validator-level ambiguity but leaves the user-facing
confusion intact — exactly the situation the reporter is trying to
escape.

A code-level implementation plan should be deferred until the HL7
Laboratory Report IG repository is added to the FHIR Augury GitHub
source configuration and a `repo-analysis` briefing is produced (see
*Repo Context*, gap noted).
