## Summary

The reporter flags an internal contradiction on the FHIR Vital Signs profile
page (section 10.1.17, *observation-vitalsigns.html*) regarding the cardinality
of the systolic and diastolic components on the Blood Pressure profile. The
third bullet of the introductory list says that *"if you have a blood pressure
observation, you must have both a systolic and a diastolic component, though
one or both may have dataAbsentReason instead of a value."* The Blood Pressure
Panel row in the table further down the same page states that the BP panel
*"...contains at least one component (systolic and/or diastolic)."* The two
sentences cannot both be true, and the formal `bp` StructureDefinition agrees
with the bullet (component is `min=2`, with both `Observation.component:SystolicBP`
and `Observation.component:DiastolicBP` slices set to `min=1`). The reporter
asks that the page text be made internally consistent and consistent with the
profile.

This is not a new observation. The same contradiction has been raised at least
twice before:

- [FHIR-28259](https://jira.hl7.org/browse/FHIR-28259) — *Clarity required for
  BP measurements* (Craig Newman, CIMI) — quotes the same two sentences
  verbatim, was resolved **Persuasive** and **Published**.
- [FHIR-20833](https://jira.hl7.org/browse/FHIR-20833) — *Difference between BP
  components* (Craig Newman) — questions why one component was historically
  `1..1` and the other `0..1`; resolved **Persuasive / Applied** and the
  StructureDefinition was tightened so both slices are now `min=1`.

So the formal model has been corrected, but the narrative on the introduction
page still carries the legacy "systolic and/or diastolic" wording, which is
what FHIR-54508 is now catching.

The reporter has a closely related, currently-Triaged companion ticket
[FHIR-54503](https://jira.hl7.org/browse/FHIR-54503) on the same page (binding
strength contradiction between the bullet list and the per-vital-sign profiles),
which suggests a broader audit of section 10.1.17 narrative is warranted.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit `source/observation/observation-vitalsigns-profile-introduction.xml` so
the *Blood pressure panel* row in the vital-signs table aligns with the
mandatory-components bullet and the formal `bp` StructureDefinition.
Specifically, on line 213 replace

> *"This is a component observation. It has no value in
> Observation.value[x] and contains at least one component (systolic
> and/or diastolic). ..."*

with wording such as

> *"This is a component observation. It has no value in
> Observation.value[x] and contains both a systolic and a diastolic
> component (one or both of which may carry `dataAbsentReason` instead
> of a value). ..."*

Leave the existing supplementary-component sentence (8478-0 mean BP, 8357-6
method, etc.) intact. No change to `structuredefinition-profile-bp.xml` is
required (`Observation.component` is already `min=2` with both slices
`min=1`). Sweep `observation-vitalsigns-profile-notes.xml` for the same
"and/or" wording and update consistently. Mark this as a *correction-only*
follow-up to FHIR-28259 (Persuasive, Published) and FHIR-20833 (Persuasive,
Applied) so reviewers can see why the narrative still drifted.

#### Justification

This is a documentation defect, not a model change: the formal profile
already requires both components, and the bullet immediately above the table
already says so. Aligning the table cell is the minimal change that removes
the contradiction the reporter identified, restores consistency between the
narrative and the StructureDefinition, and closes the loop on the prior
Persuasive dispositions (FHIR-28259, FHIR-20833) whose model fix never
propagated into the table prose. It also addresses the implementer
confusion that surfaced repeatedly on Zulip (*Vital Signs Profile - clarity
on applicability*, *Send only systolic blood pressure*).

---

### Disposition B: Alternative Approach — Relax the Profile to Match the Table

#### Proposal

Instead of fixing the prose, change the formal model: edit
`source/observation/structuredefinition-profile-bp.xml` so that
`Observation.component` is `min=1` (not `2`) and the `DiastolicBP` slice is
`min=0` (or, symmetrically, change both slices to `0..1` and add an
invariant that at least one is present and that whichever is present must
have a value or `dataAbsentReason`). Then update the bullet on line 75 of
`observation-vitalsigns-profile-introduction.xml` to read *"if you have a
blood pressure observation, it must contain at least one of the systolic or
diastolic components"*. This makes the table the source of truth and
narrows the bullet rather than tightening the table.

#### Justification

There is documented implementer demand for sending a single component (see
the *Send only systolic blood pressure* Zulip thread; FHIR-20833 also
reflects historical asymmetry between the two slices). Some devices and
home-monitoring scenarios genuinely capture only systolic. A `0..1 / 0..1`
+ "at least one present" invariant is more permissive and avoids forcing
implementers to fabricate a `dataAbsentReason` on a component they were
never asked to measure.

The trade-off versus Disposition A is significant: this is a normative
profile change in an STU/Normative artifact that has been published with
both components mandatory for years (FHIR-20833 was specifically resolved
to *tighten* `bp` in this direction, and downstream IGs — notably US Core
and IPS — already require both components). Loosening it now would create
a backwards-compatibility hazard and contradict prior workgroup consensus,
which is why this should be considered an alternative *to be argued
explicitly*, not the default path.

---

### Disposition C: Decline as Already Resolved (Persuasive — No Further Action)

#### Proposal

Mark FHIR-54508 as a duplicate of [FHIR-28259](https://jira.hl7.org/browse/FHIR-28259)
(Persuasive / Published) and close as *Not Persuasive — Considered for
Future Use* / *Already Addressed*, on the basis that the model has already
been corrected (per FHIR-20833) and the bullet on line 75 already reflects
the correct requirement.

#### Justification

The substantive question — *are both components required?* — has already
been answered authoritatively in the StructureDefinition (`min=2`). A
reviewer can resolve the apparent contradiction by reading the bullet
together with the formal profile.

This disposition is included for completeness but is **weak**: the
contradiction in the published prose is real and verifiable in the
master branch today, FHIR-28259 was supposed to have addressed it and
demonstrably did not, and the same reporter has filed a sibling defect
(FHIR-54503) against the same bullet list. Declining a second time would
likely guarantee a third filing.

---

### Recommendation

**Recommended disposition:** A — Accept As Requested.

The formal model already says exactly what the reporter expects, and
prior Persuasive dispositions (FHIR-28259, FHIR-20833) have already
established the workgroup's position that both BP components are
mandatory. The defect FHIR-54508 actually reports — that the table cell
text was never re-aligned with the model after FHIR-20833 was applied —
is a small, surgical narrative edit in one well-known authored file
(`source/observation/observation-vitalsigns-profile-introduction.xml`),
with no cross-repo impact and no risk to downstream IGs that already
assume both components are present. Disposition B (loosening the
profile) would re-open a settled normative question and break US Core /
IPS expectations; Disposition C is unsafe given that FHIR-28259 was
already a Persuasive close on the same defect and the prose still
drifted. The workgroup should also bundle FHIR-54503 (same reporter,
same bullet list, binding-strength contradiction) into the same
narrative pass to avoid touching the file twice.
