## Summary

The reporter argues that `Observation.component` limits data reuse — because the
component values are bound into a single Observation envelope, they cannot be
reused or reasoned about independently. He asks Orders & Observations to (a)
allow blood pressure vital signs to be expressed using `Observation.hasMember`
(grouping into separate Observations) instead of, or in addition to, the
existing `Observation.component`-based representation, and (b) provide explicit
spec guidance on when `.hasMember` is preferred over `.component`, with Apgar
score called out as a likely candidate for `hasMember`. The reporter and other
implementers have offered six additional use cases (visual acuity OD/OS/OU,
spirometry FEV1/FVC/PEF, vital capacity battery, blood-gas cartridge pH/pCO₂/pO₂/lactate,
pacemaker remote-session events, multi-lead Holter) to demonstrate that the
"BP" case is not unique. The ticket also surfaces an implementer
counter-argument that components keep a single business identifier intact, and
that splitting into multiple Observations creates identifier-reuse problems.

The request is centred on the long-running Orders & Observations debate
captured in
[Grouping options for Observation/DiagnosticReport](https://chat.fhir.org/#narrow/stream/Orders%20and%20Observation%20WG/topic/Grouping%20options%20for%20Observation%2FDiagnosticReport),
which has been on the agenda for several recent UV Lab calls (2025-04-15,
2025-04-29, 2025-08-19, 2026-01-13, 2026-01-16) leading into R6 ballot
reconciliation. The workgroup is also weighing this against earlier rulings:
[FHIR-32788](https://jira.hl7.org/browse/FHIR-32788) (Applied) clarified that
*additional* BP codes should be carried as components rather than separate
Observations; [FHIR-49736](https://jira.hl7.org/browse/FHIR-49736) (Applied)
removed the `MS` constraint on `component` from the base Vital Signs profile so
that it only applies in the BP profile; [FHIR-46099](https://jira.hl7.org/browse/FHIR-46099)
(Published) corrected Apgar examples that mis-used `hasMember`; and
[FHIR-39659](https://jira.hl7.org/browse/FHIR-39659) (Published) reconciled
conflicting BP `dataAbsentReason` guidance — all of which presuppose the
component-based BP shape that this ticket asks to relax.

## Details

**Description:**
> While trying to maximize data reuse, we see that observation.component limits
> the reuse of the values, as the enire observation is shared. Does blood
> pressure justify that case? or are there others? We think that even Apgar
> scores may be used with grouped (hasMember) observations - is this correct?
> Please provide guidance

**Comments:**

### costateixeira (2026-01-18)

> Note that if we want to clarify that when components are provided by
> different participants, or inferred from previous measurement, or given at
> different times (which is not that common), then we must abandon component
> and use .hasmember. So hasMember has simply a better hold for a variety of
> cases.

### costateixeira (2026-02-09)

> Feedback from implementers: In our system we have Observations that are one
> 'thing' and have one identifier for the Observation that includes
> components. For blood pressure, wound care, and many others we have one
> identifier. If we move to different Observations instead of components, we
> cannot reuse that identifier, because different resources will share the
> same identifier (which from what I understand is bad practice). … Another
> note is that as an implementer I'm not the biggest fan of exceptions, I
> would prefer a generic approach. Now we will need to implement both with
> and without components anyways, because of these exceptions.

### costateixeira (2026-02-09)

> more implementer comment: As discussed in the workgroup, I would add some
> use cases to this issue to demonstrate the BP case isn't unique. … Use case
> 1: visual acuity exam (OD/OS/OU); Use case 2: spirometry (FEV1, FVC, PEF);
> Use case 3: Vital Capacity (CPV ratio, grip strength, fatigue resistance);
> Use case 4: Blood gas cartridge (pH, pCO2, pO2, lactate); Use case 5:
> Pacemakers (event per component during remote session); Use case 6: Holter
> monitoring (3 to 12 lead capture, multiple leads per session).

## Related Zulip Discussions

### [Orders and Observation WG > Grouping options for Observation/DiagnosticReport](https://chat.fhir.org/#narrow/stream/Orders%20and%20Observation%20WG/topic/Grouping%20options%20for%20Observation%2FDiagnosticReport)

This is the canonical, multi-year (May 2023 → Feb 2026) thread that
underpins FHIR-54710. It started from an AU diagnostics question about how
to represent panels (CBC/FBC) and grew into a broader debate about when
implementers should use `Observation.component`, `Observation.hasMember`,
or `DiagnosticReport.result` for grouping. The thread documents repeated UV
Lab / OO call cycles (2025-04-15, 2025-04-29, 2025-08-19, 2026-01-13,
2026-01-16, 2026-01-23) explicitly aimed at producing a workgroup
consensus before R6 ballot reconciliation. Forest's question — *"is it
appropriate to represent BP as a panel Obs with two member Obs linked via
hasMember?"* — is essentially the same question FHIR-54710 asks. **Bryn
Rhodes** answered with a citation to
[Vital Signs §10.1.17](https://hl7.org/fhir/R4/observation-vitalsigns.html#10.1.17):
*"when a system supports a vital sign, it SHALL use the profile listed
here to exchange that result"*, i.e., the existing BP profile (component
shape) is mandatory and a `hasMember`-only BP would be non-conformant.
**Elliot Silver** offered the practical decision rule: *"would this stand
alone? would someone search for only one of these values?"*, concluding
that systolic without diastolic has limited stand-alone use, so component
is reasonable for BP but `hasMember` is a better fit when items are
collected together but each is independently meaningful (BP + heart rate
+ SpO₂). **Riki Merrick** noted that susceptibility testing complicates
that rule because results don't stand alone without the bug being tested.
The WGM Confluence pages
([2023-09-11](https://confluence.hl7.org/pages/viewpage.action?pageId=171450118),
[2025-04-15](https://confluence.hl7.org/spaces/OO/pages/324974287/2025-04-15+OO+-+LAB+UV),
[2025-04-29](https://confluence.hl7.org/spaces/OO/pages/345539523/2025-04-29+OO+-+LAB+-+Observation+Grouping),
[2025-08-19](https://confluence.hl7.org/spaces/OO/pages/358269198/2025-08-19+OO+-+LAB+-+Observation+Grouping),
[Grouping Option Analysis](https://confluence.hl7.org/spaces/OO/pages/403871266/Grouping+Option+Analysis))
referenced from the thread record the structured option-analysis the WG
has been doing.

Participants: Alexander Henket ⚒️, Andrea Pitkus PhD MLS(ASCP)CM FAMIA,
Angus Millar, Becky Gradl, Bryn Rhodes, David McKillop, Elliot Silver,
Forest, Grahame Grieve, Hans Buitendijk, Jay Lyle, JD Nolen, John Moehrke,

### [implementers > Observation.component vs Observation.hasMember](https://chat.fhir.org/#narrow/stream/implementers/topic/Observation.component%20vs%20Observation.hasMember)

Older but directly on-point thread. **Lloyd McKenzie** articulated the
working rule still cited today: *"If the information makes sense and is
useful by itself, then it should be a separate Observation (possibly
linked by 'hasMember'). If it qualifies another Observation or doesn't
make sense based on common medical convention to ever interpret separately
from another Observation, then component is appropriate."* He explicitly
listed Apgar pieces and BP systolic/diastolic/cuff-size as canonical
*component* cases, and CBC parts as canonical *hasMember* cases — i.e.,
the stated rationale already treats Apgar as a `.component` case, contrary
to the conjecture in FHIR-54710's description.

Participants: Lloyd McKenzie (and broader implementers thread).

### [Orders and Observation WG > Observation for Panel (like a CBC)](https://chat.fhir.org/#narrow/stream/Orders%20and%20Observation%20WG/topic/Observation%20for%20Panel%20%28like%20a%20CBC%29)

**Rob Hausam** restated the same rule with an Observation-Grouping spec
citation: `Observation.component` is for *"any supporting result that
cannot reasonably be interpreted and used outside the scope of the
Observation it is a component of"*, and explicitly affirmed that
**BP and Apgar fit `.component`** while CBC-style panels fit `.hasMember`.
He acknowledged the line is not entirely black-and-white for some lab
panels, supporting the FHIR-54710 reporter's call for clearer guidance
even while contradicting the proposed remedy of switching BP to
`hasMember`.

Participants: Rob Hausam, Gay (and panel discussion participants).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify the FHIR Core Vital Signs profile pack so that the **Blood Pressure**
profile permits an alternate representation in which a parent BP Observation
uses `Observation.hasMember` to reference two child Observations (one
systolic, one diastolic) instead of (or in addition to) the current
mandatory two-component shape. Concretely:

1. In `observation-vitalsigns.html` §10.1.17, relax the "SHALL use the
   profile listed here" statement to allow either the component-based BP
   profile or a new `bp-hasmember` variant.
2. Define the new profile (`bp-hasmember`) with `Observation.hasMember`
   sliced into `systolic` and `diastolic` slices that target Observations
   bound to LOINC `8480-6` and `8462-4` respectively, and forbid
   `Observation.component` on this variant.
3. Add a normative §10.1.5.3 paragraph that **Apgar score** MAY be
   represented using either the existing component-based example
   (`observation-example-5minute-apgar-score.html`) or a `hasMember`-based
   parent/children example, and add a worked second example.
4. Add the same flexibility statement and a generic worked example for
   the use cases enumerated in the ticket comments (visual acuity,
   spirometry, vital capacity, blood-gas cartridge, pacemaker remote
   session, Holter leads).

#### Justification

This is the most direct response to the implementer pain in the ticket —
"we will need to implement both with and without components anyways" — by
*authorising* the second pattern instead of leaving it as a de facto
deviation. It also aligns the BP profile with the practical reality
acknowledged in the Zulip threads that some assessment-scale panels and
multi-instrument captures do not fit the
"single method, single observation, single performer, single device,
single time" preconditions in §10.1.5.3.2. The reporter's six additional
use cases (visual acuity, spirometry, vital capacity, blood gas,
pacemaker, Holter) materially weaken the "BP is uniquely justified"
defence of the current rule.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** change the BP vital-signs profile. Instead, address the ticket
by publishing **explicit decision guidance** in `observation.html`
§Observation Grouping (`obsgrouping`) and a cross-reference from
`observation-vitalsigns.html` §10.1.17:

1. Promote Lloyd McKenzie's "stand-alone vs. only-meaningful-together"
   rule (already cited in the Zulip threads) to normative-strength
   guidance text, with a worked decision tree.
2. State explicitly that the BP, Apgar, and other R4 component-based
   examples were chosen because their parts *do not stand alone* clinically,
   and that the test for `.component` vs. `.hasMember` is the stand-alone
   question — *not* identifier reuse, *not* "different participants /
   different times" (those are reasons to use a *new* Observation, not to
   change the grouping mechanism).
3. Add the bullet already proposed in the
   [Grouping options thread](https://chat.fhir.org/#narrow/stream/Orders%20and%20Observation%20WG/topic/Grouping%20options%20for%20Observation%2FDiagnosticReport) /
   FHIR-51724:
   *"Laboratory results are not expected to be represented using
   Observation.component. Typically the 'hasMember' relationship is used
   for grouping laboratory results (e.g., panels)."*
4. Add a non-normative table walking through the reporter's six use cases
   and stating which mechanism the WG recommends for each (e.g., visual
   acuity OD/OS/OU → `.component`; spirometry FEV1/FVC/PEF → `.hasMember`
   because the values *can* be reported in isolation; blood-gas
   cartridge → `.hasMember`; multi-lead Holter → `.hasMember`).
5. Cross-reference the new guidance from
   [FHIR-46099](https://jira.hl7.org/browse/FHIR-46099)'s Apgar
   correction so the example and the guidance are consistent.

#### Justification

This addresses the underlying ask ("Please provide guidance") without
breaking the 100s of existing BP implementations that conform to the
current vital-signs profile, and without re-opening the
profile-conformance rule that just came up in adjacent tickets
([FHIR-32788](https://jira.hl7.org/browse/FHIR-32788),
[FHIR-49736](https://jira.hl7.org/browse/FHIR-49736),
[FHIR-39659](https://jira.hl7.org/browse/FHIR-39659)). The Zulip threads
already show that the WG-experienced commenters (Bryn Rhodes, Rob Hausam,
Lloyd McKenzie, Elliot Silver) consider BP and Apgar to be *correctly*
modelled as `.component`. The implementer counter-argument in the
ticket's own comments ("we cannot reuse that identifier… we'd prefer a
generic approach") is a real concern, but it is one *for IGs to address*
when they need separate-Observation reporting, and it is what `hasMember`
already supports for non-vital-sign cases. Allowing a second BP shape in
core would force every server, validator, and downstream profile (US Core,
IPS, AU Base) to support both, multiplying the very "exception" surface
the reporter is complaining about — outcome opposite to the goal.

---

### Disposition C: Decline

#### Proposal

Decline the change. Reply with text along the lines of: *"The Vital Signs
Blood Pressure profile is intentionally normative and intentionally
mandates `Observation.component` for systolic and diastolic, as stated in
[Vital Signs §10.1.17](https://hl7.org/fhir/R4/observation-vitalsigns.html#10.1.17).
The use case of representing each measurement as a separately-reusable
Observation is already covered for non-vital-sign panels (e.g., CBC) by
`Observation.hasMember`, and the existing rule in §10.1.5.3.2 — that
`.component` is for results that cannot reasonably be interpreted outside
the scope of the parent Observation — already explains why BP and Apgar
qualify. No change to core."* Mark resolution **Not Persuasive**.

#### Justification

Defensible because (a) the Vital Signs BP profile is normative and
changing it is a breaking change for every existing BP implementer,
profile (US Core BP, IPS BP, AU Base BP), validator, and example; (b) the
ticket's stated motivation — identifier reuse and "different
participants / different times" — is not actually a `.component`
limitation: an implementer who genuinely needs to report systolic and
diastolic from different participants/times must report them as
*separate* Observations (not as `.hasMember` children of a parent BP),
and that scenario was never covered by the BP profile in the first
place; and (c) the recent adjacent tickets
([FHIR-32788](https://jira.hl7.org/browse/FHIR-32788),
[FHIR-49736](https://jira.hl7.org/browse/FHIR-49736),
[FHIR-39659](https://jira.hl7.org/browse/FHIR-39659)) all *tightened*
the component-based BP design rather than loosening it; reversing course
in R6 would be inconsistent.

---

### Recommendation

**Recommended disposition:** B

The ticket asks two things — *"allow `.hasMember` for BP"* and *"provide
guidance"*. The first is at odds with normative §10.1.17 and with
positions consistently expressed by experienced WG members in the
canonical [Grouping options](https://chat.fhir.org/#narrow/stream/Orders%20and%20Observation%20WG/topic/Grouping%20options%20for%20Observation%2FDiagnosticReport) thread (Bryn Rhodes,
Rob Hausam, Elliot Silver, Lloyd McKenzie), and would force every
downstream profile and validator to support a second BP shape. The second
is real, well-motivated, and exactly what the WG has been working toward
across the 2025–2026 UV Lab calls and the
[Grouping Option Analysis](https://confluence.hl7.org/spaces/OO/pages/403871266/Grouping+Option+Analysis)
Confluence page. Disposition B captures the legitimate guidance ask
(including the reporter's six additional use cases and the FHIR-51724
"laboratory ≠ component" bullet), keeps faith with the just-Applied
[FHIR-32788](https://jira.hl7.org/browse/FHIR-32788),
[FHIR-49736](https://jira.hl7.org/browse/FHIR-49736), and
[FHIR-46099](https://jira.hl7.org/browse/FHIR-46099) decisions, avoids the
breaking-change cost of Disposition A, and is more responsive than
Disposition C — which dismisses a valid request for clearer guidance the
WG itself has acknowledged is overdue. Resolution language should be
*Persuasive with Modification*.
