## Summary

FHIR-55275 asks Orders & Observations to add an "unknown", "other", or
equivalent escape-valve concept to the value set bound to
`ServiceRequest.intent`. The element is `1..1` with a required binding,
so when an implementer genuinely cannot determine the intent of a
service request (e.g., when forwarding a record from another system or
ingesting partial data), there is no conformant code to use and the
resource cannot be populated. The reporter (Lloyd McKenzie) is explicit
that the ask is to **extend the bound value set**, not to change the
binding strength.

The technical complication, surfaced in OO discussion (comments on
2026-01-29 and 2026-02-12), is that `ServiceRequest.intent` ultimately
binds to the cross-resource value set
[`request-intent`](https://build.fhir.org/valueset-request-intent.html)
backed by the code system
[`request-intent`](https://build.fhir.org/codesystem-request-intent.html).
Neither artifact is owned by O&O — the value set is owned by FHIR-I and
the code system by CDS — and neither currently carries an `unknown` /
`other` concept. The same value set is referenced by sibling
"escape-valve" tickets in O&O ([FHIR-55262](https://jira.hl7.org/browse/FHIR-55262)
DeviceRequest, [FHIR-55267](https://jira.hl7.org/browse/FHIR-55267)
NutritionOrder) and by the Patient Care–owned sibling
[FHIR-55322](https://jira.hl7.org/browse/FHIR-55322)
(CommunicationRequest), so a single change to the FHIR-I-owned value
set / code system would resolve the family at once. A FHIR-I rollup
ticket ([FHIR-56142](https://jira.hl7.org/browse/FHIR-56142)) was
subsequently opened to do exactly that.

OO previously voted "Persuasive with Modification" (Hausam / Gaudet,
5-0-0, 2026-02-20) to (1) relax `ServiceRequest.intent` cardinality to
`0..1` and (2) change the binding to extensible — but that resolution
was **reverted** by motion on 2026-02-24 (David / Rob, 6-0-0) once it
was pointed out that `.intent` is a `code`-typed element and `code`
elements cannot carry an extensible binding. The ticket therefore needs
a fresh disposition.

## Details

required binding, but appears to have neither an 'unknown' or 'other'
concept, which can make the containing element impossible to populate

## Proposed Dispositions

The substantive change cannot be made by O&O: the value set
`request-intent` is owned by FHIR-I and the code system `request-intent`
is owned by CDS. O&O's disposition is therefore really a question of
what to **recommend** and **whom to refer to** for this O&O resource.
The previously-voted change (relax `.intent` cardinality to `0..1`,
change binding to extensible) is no longer viable as written, because
`.intent` is a `code`-typed element and `code` elements cannot carry an
extensible binding (per the 2026-02-23 / -24 discussion).

### Disposition A: Accept As Requested

#### Proposal

**Persuasive with Modification.** Recommend that the `request-intent`
code system (CDS) add an `unknown` concept (and, if CDS prefers,
`other`), and that FHIR-I incorporate it into the `request-intent`
value set. O&O makes no change to `ServiceRequest.intent` itself
(cardinality stays `1..1`, binding stays required). Re-assign the
ticket to FHIR-I (value-set owner) with a note asking FHIR-I to
coordinate the code-system change with CDS. Pin this ticket as
"awaiting upstream" against the FHIR-I rollup
[FHIR-56142](https://jira.hl7.org/browse/FHIR-56142) so it auto-resolves
when the value set is updated. Apply the same pattern to the O&O
siblings [FHIR-55262](https://jira.hl7.org/browse/FHIR-55262) and
[FHIR-55267](https://jira.hl7.org/browse/FHIR-55267) (already on the
same trajectory).

#### Justification

This is exactly what the reporter asked for ("add 'unknown', 'other',
or some equivalent escape valve concept to the value set") and is the
clarification Lloyd McKenzie gave on 2026-02-23 ("the request isn't to
change the binding strength, it's to extend the value set"). It
preserves the required binding — which is desirable for a
cross-resource workflow code that drives behaviour — and avoids the
`code`-element / extensible-binding incompatibility that defeated the
earlier "relax to 0..1 + extensible" resolution. It also routes the
change to the work groups that actually own the artifacts, in line
with the OO main-call consensus on 2026-02-12. Resolving the value set
once fixes all four sibling tickets (`ServiceRequest`, `DeviceRequest`,
`NutritionOrder`, `CommunicationRequest`) by inheritance, which is the
explicit observation in the Zulip thread.

---

### Disposition B: Alternative Approach

#### Proposal

**Persuasive with Modification (cardinality variant).** Recommend
that the **request pattern** loosen `Request.intent` cardinality from
`1..1` to `0..1`, and that `ServiceRequest.intent` follow the pattern
to `0..1`. Keep the binding required and the value set unchanged.
Re-assign the ticket to FHIR-I (which owns the request pattern), again
linked to the rollup [FHIR-56142](https://jira.hl7.org/browse/FHIR-56142),
and explicitly note that this is an alternative to extending the value
set if CDS / FHIR-I judge the code-system change too costly. Note that
this is essentially the surviving half of the reverted 2026-02-20
vote (drop the unworkable "extensible binding" half).

#### Justification

A `code`-typed element with a required binding genuinely cannot use an
extensible binding, so the only ways to give implementers an "I don't
know" path are (a) extend the value set (Disposition A) or (b) make
the element optional. Patient Care already flagged on the FHIR-55322
discussion (2026-03-05) that Rob Hausam thought `Request.intent` might
loosen to `0..1`, suggesting cross-WG appetite for this option. It is
also less risky from a terminology-governance perspective: it does not
require CDS to add a concept to a workflow code system.

The downside, and the reason this is B rather than A: dropping
`.intent` to `0..1` weakens the request pattern's ability to drive
behaviour (the workflow stack reasonably expects every request to
declare whether it's a `proposal`, `plan`, `order`, etc.), risks
silent under-population in real systems, and is a substantive,
backward-affecting change that ripples to every Request-pattern
resource. Disposition A captures the same use case with a much
narrower change.

---

### Disposition C: Decline

#### Proposal

**Not Persuasive (for O&O scope).** O&O closes its review on
FHIR-55275 with no recommendation to change `ServiceRequest`. Rationale:
the cross-resource `request-intent` value set / code system are owned
outside O&O (FHIR-I / CDS), and a sibling FHIR-I rollup ticket
[FHIR-56142](https://jira.hl7.org/browse/FHIR-56142) already exists to
address the underlying problem at the request-pattern level. Re-assign
this ticket to FHIR-I as a duplicate-of / blocked-by FHIR-56142 and
take no further O&O action.

#### Justification

This is defensible on jurisdictional grounds — O&O really cannot make
the substantive change, and the upstream owner already has an open
ticket. However, "decline" understates O&O's responsibility to make a
recommendation: the OO 2026-02-12 consensus was to "complete our
review … note our recommendation, and then re-assign", and the
reporter's clarification makes a specific, viable recommendation
(extend the value set) easy to state. Choosing this disposition risks
losing O&O's voice in the FHIR-I / CDS deliberation about which fix to
apply, which matters because Disposition A and Disposition B have
materially different downstream impacts on `ServiceRequest`
implementers.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest match to what the reporter actually
asked for, the only one that survives the `code`-element /
extensible-binding constraint that killed the prior 2026-02-20 vote,
and the one that aligns O&O's recommendation with both the Patient
Care position on FHIR-55322 (wait for upstream, prefer inheritance)
and the FHIR-I rollup [FHIR-56142](https://jira.hl7.org/browse/FHIR-56142)
that Michelle Miller opened on 2026-03-20. It keeps `.intent`
mandatory and required-bound — preserving the workflow guarantees
the request pattern is built on — while giving implementers the
escape valve they need for genuinely-unknown cases. Disposition B
is worth raising in the FHIR-I conversation as a fallback if CDS
declines to add the concept, but should not be O&O's primary
recommendation. Disposition C abdicates the recommendation O&O
explicitly committed to make on 2026-02-12 and is not advised.
