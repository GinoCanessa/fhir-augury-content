## Summary

Eric Haas reports an inconsistency in the Observation resource introduced
when the new `Observation.interpretationContext` element was added at the
root of Observation (per [FHIR-51529](https://jira.hl7.org/browse/FHIR-51529),
which has been Applied with resolution Persuasive). The root resource now
carries both `interpretation` and a sibling `interpretationContext`, but
`Observation.component` still exposes only `interpretation` — there is no
`Observation.component.interpretationContext`. Because individual components
inside a panel/grouped Observation can have their own clinically relevant
context (e.g., posture, fasting state, oxygen delivery, body site) that is
distinct from — or finer-grained than — the parent Observation's context,
the reporter argues `interpretationContext` should be mirrored on
`component` for symmetry with `interpretation` and with the design intent
of the new element.

Background context that is not formally cross-linked but is highly
relevant: [FHIR-51529](https://jira.hl7.org/browse/FHIR-51529) is the
originating change that added `Observation.interpretationContext` as
`0..* CodeableReference(Any)` ("Other preceding or concurrent information
that is critical to understand the context and significance of the
observation"); [FHIR-53508](https://jira.hl7.org/browse/FHIR-53508)
(retracted) was an earlier complaint that `interpretation` appeared to be
duplicated, in fact the second appearance was the new
`interpretationContext` — confirming the new element is already in the
build and paired with `interpretation` at the root only;
[FHIR-55269](https://jira.hl7.org/browse/FHIR-55269) tracks the value-set
work for `observation-interpretation-context`. The current ticket
(FHIR-54461) is therefore the natural follow-on parity question for the
component element.

## Details

**Description:**

> **Issue:** The root Observation has a new `interpretationContext` element,
> but `component` does not. Components may require different interpretation
> contexts than the parent observation.
>
> **Propose:** Add `interpretationContext` element to component.

**Comments:** No comments have been posted on FHIR-54461 yet.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add `Observation.component.interpretationContext` to the Observation
StructureDefinition `<differential>` in `source/observation/`, mirroring
the existing `Observation.interpretationContext`:

- Element name: `Observation.component.interpretationContext`
- Cardinality: `0..*`
- Type: `CodeableReference(Any)` (same as `Observation.interpretationContext`)
- Definition: "Other preceding or concurrent information that is critical
  to understand the context and significance of this component
  observation" (parameterised wording of the parent element's definition).
- Binding: extensible to
  `http://hl7.org/fhir/ValueSet/observation-interpretation-context`
  (re-using the binding established by FHIR-51529 and tracked in
  FHIR-55269).
- Mappings: copy the `rim` mapping pattern used on
  `Observation.interpretationContext` (per the RIM mapping added in
  PR #3834, commit `64027da`); flag for SME review as comment 60615 on
  FHIR-51529 already requested confirmation that the RIM mapping was
  appropriate at the root — the same review applies on `component`.
- Update the Observation narrative page(s) under `source/observation/`
  to add a short paragraph explaining when to use the
  component-level element vs. the root-level element (component-level
  context applies only to *that* component; root-level context applies
  to the Observation as a whole or to all components uniformly).

Edit the `<differential>` only; allow the publisher to regenerate
`<snapshot>`, JSON, schemas, and `qa/`. Target version: R6 (matches
FHIR-51529 scope).

#### Justification

The request restores element-level symmetry that already exists for
`interpretation`: `Observation.interpretation` and
`Observation.component.interpretation` are both present, and the new
`interpretationContext` was paired with `interpretation` at the root
(per FHIR-51529 / FHIR-53508). Failing to mirror it on `component`
breaks that symmetry and forces implementers using grouped/panel
Observations to either (a) lift component-specific context up to the
root — which is wrong, because it implies the context applies to the
whole Observation — or (b) fall back to extensions for something the
core spec already considers in-scope. The repo briefing confirms this is
a low-risk, in-pattern change (single `<differential>` edit plus prose),
with no cross-repo touch points required. The concerns raised by
apitkus on FHIR-51529 about panel/AOE explosion actually *support*
component-level placement: a single root `interpretationContext` cannot
be applied per-result inside a panel, while a component-level field
can.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a new element to `Observation.component`. Instead, publish
formal usage guidance on the Observation page (under `source/observation/`)
that directs implementers to one of the existing mechanisms when
component-specific interpretation context is needed:

1. Use a separate child Observation linked via
   `Observation.hasMember` (or `Observation.derivedFrom`) and put the
   `interpretationContext` on that child Observation at the root.
2. Where a child Observation is unjustified, define a profile-level
   extension on `Observation.component` (mirroring the pre-FHIR-51529
   `servicerequest-precondition` / `interpretation-context` extension
   pattern) and publish it from `HL7/fhir-extensions`.

Add a worked example to the Observation examples set illustrating
pattern (1) for a panel where one component needs distinct context
(e.g., one analyte measured fasting in a metabolic panel).

#### Justification

This avoids growing the BackboneElement footprint of `Observation.component`
(which is already a wide structure: `code`, `value[x]`,
`dataAbsentReason`, `interpretation`, `referenceRange`) and keeps the
component a relatively flat record of a single measurement. The
`hasMember` / child-Observation pattern is already the canonical way to
express per-result metadata that does not fit on a component (e.g.,
per-result performer, per-result effective time), so reusing it for
context is consistent. It also preserves backward compatibility with
existing R6 ballots and lets profile authors layer in component-level
context only where their use case demands it. The trade-off vs.
Disposition A is real: implementers who already use `component` heavily
(vital-signs panel, lab panels) will see this as friction compared to a
single new optional element.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54461 as **Not Persuasive – existing mechanism**: do not add
`Observation.component.interpretationContext`. Direct implementers to
`Observation.hasMember` / `Observation.derivedFrom` for cases where
per-component context is needed, and treat any remaining gap as
profile-specific work.

#### Justification

`Observation.component` was deliberately scoped narrowly to capture
"part of a single observation" and avoids most of the metadata fields
that live on the root resource (no `effective[x]`, no `performer`, no
`bodySite`, etc.). Adding `interpretationContext` to `component` would
break that established design boundary and invite future tickets to
restore further root-only fields on `component` (effective time,
performer, body site) for parallel reasons. The root-level
`Observation.interpretationContext` from FHIR-51529 is brand new
(Applied late October 2025) and there is no implementer report yet of
the pattern failing in real deployments — only a symmetry argument from
the reporter. Decline-with-guidance preserves the design intent of
`component` until concrete implementer evidence shows the existing
mechanisms are insufficient.

---

### Recommendation

**Recommended disposition:** A

The symmetry argument is strong and well-grounded in the resource as it
stands today: `interpretation` is already mirrored on
`Observation.component`, and `interpretationContext` was explicitly
introduced as the contextual peer of `interpretation` in FHIR-51529.
Leaving the pair split across the two levels (root has both, component
has only `interpretation`) is harder to explain to implementers than
either having both elements at both levels or having neither pair on
`component`. The panel/AOE concerns that apitkus raised on FHIR-51529
also cut in favour of a component-level field, not against it: a single
root-level `interpretationContext` cannot express that, in a metabolic
panel, only the *glucose* component was collected fasting, while a
component-level field can. Disposition B's `hasMember` workaround is
viable but adds resource fan-out for what should be a one-line piece of
metadata, and Disposition C trades a small spec edit for ongoing
implementer friction. The repo-analysis briefing confirms the change is
low-risk: a single `<differential>` edit in `source/observation/` plus
narrative, no cross-repo work required, value-set binding re-uses the
one already established by FHIR-51529 / FHIR-55269. The workgroup
should still confirm the RIM mapping (the same SME confirmation
requested on FHIR-51529 comment 60615) before applying.
