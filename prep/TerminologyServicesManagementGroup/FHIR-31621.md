## Summary

The ticket ([FHIR-31621](https://jira.hl7.org/browse/FHIR-31621)) raises a
governance concern about the parallel and currently inconsistent maintenance
of OIDs and URIs in HL7 artifacts. The reporter observes that V3/CDA relies
on HL7's meticulously maintained OID Registry, while FHIR has introduced URIs
— frequently for the same underlying code systems and value sets — that are
created ad hoc and decentrally per work group, drift over time, and are not
reliably tied back to the OID that already identifies the same concept. The
reporter argues this is not about readable identifiers per se, but about
"on par, in tandem" governance: if HL7 is going to scale V3 down to IG
creation only, then every new FHIR CodeSystem URI must also have an OID
assigned (examples cited: ICD-11, IDMP) and there must be a single,
version-stable, computable mapping between OID and URI. The ticket includes
a substantial data analysis showing, for R4, 17 OIDs with more than one URI
(e.g. `1.3.160` mapping to both `http://terminology.hl7.org/CodeSystem/gtin`
and `https://www.gs1.org/gtin`) plus numerous cases where a single OID has
been reused across V2 and V3 ValueSets that are, in fact, different objects,
and where CodeSystem and ValueSet share the same OID. The cross-reference
index confirms the ticket touches `ValueSet.compose`, `CodeSystem.valueSet`,
and `CodeSystem.url`. The request is, at its core, a policy/governance
ask: move OID↔URI administration to a single, central, version-aware home
(effectively THO/UTG + a proper NamingSystem regime, under TSMG stewardship)
and stop treating the OID Registry and the URI namespace as independent
artifacts.

## Details

**Description:**

HL7 has an OID Registry which is absolutely vital to V3 (CDA). As of FHIR,
HL7 also has URIs, in many cases for the same things that OIDs already
existed for. The reporter has argued many times against readable
alternative identification for the same thing — that is not what this
issue is about. This issue is about on par, in tandem governance for URIs
and OIDs.

Terminology is sort of centralized into THO/UTG. Identifier maintenance
also needs a proper home. While the OID Registry has been meticulously
maintained and OIDs have a tendency to be stable, the URIs more or less
have a tendency to be created in an ad hoc and decentralized way, and also
to be changed over time.

This means that many URIs exist for the same purpose. That in itself would
be OK-ish, but it is in many cases impossible to tell which is leading for
a given version of FHIR, unless you have a reliable NamingSystem setup
that marks one and only one URI/OID as the preferred one for a given
version of FHIR, and uses that consistently in the FHIR Core specs and
examples.

It is not possible to switch preferred URI/OID mid-FHIR-version as that
constitutes a breaking change in implementations. It is also not possible
to mix and match different URIs for the same thing within the same version
of FHIR. It should also not be possible to change the preferred OID/URI in
a patch or minor update — again, it is a breaking change.

HL7 has said it will scale down V3 support to only supporting IG creation.
The only way that holds is if OIDs and URIs bear a meaningful and
computable relationship. That also means that new CodeSystem URIs SHALL
have an OID assigned, e.g. ICD-11 and IDMP; otherwise they are unavailable
for CDA and the world is not ready for full-scale FHIR conversion.

So the administration of OIDs and URIs should be done centrally/in tandem
and not in ad hoc/per-working-group fashion. This is a governance issue.
To support why that is necessary, the reporter included substantial
analysis:

- **17 OIDs with more than one URI for R4**, including:
  - `1.3.160` → `http://terminology.hl7.org/CodeSystem/gtin` and
    `https://www.gs1.org/gtin`
  - `1.3.6.1.4.1.19376.1.2.3` → `http://ihe.net/fhir/ValueSet/IHE.FormatCode.codesystem`
    and `urn:oid:1.3.6.1.4.1.19376.1.2.3` (found in `ValueSet.compose`)
  - `2.16.840.1.113883.1.11.10228` → `v3-Confidentiality` and
    `v3-ConfidentialityClassification`
  - `2.16.840.1.113883.1.11.20428` → `v2-0719` and
    `v3-InformationSensitivityPolicy`
  - `2.16.840.1.113883.1.11.78` → `v2-0078` and
    `v3-ObservationInterpretation`
  - `2.16.840.1.113883.3.1937.98.11.8` → `template-status-code` and
    `urn:oid:…` (found in `CodeSystem.valueSet`)
  - `2.16.840.1.113883.3.912` → `cancer.sanger.ac.uk/…/cosmic` and
    `hl7.org/fhir/ValueSet/cosmic`
  - `2.16.840.1.113883.4.642.1.0` → `audit-event-outcome` and
    `applicability`
  - `2.16.840.1.113883.4.642.3.466` → `audit-event-outcome` and
    `audit-source-type`
  - `2.16.840.1.113883.6.280/.281/.282/.284` → various NCBI / genenames /

- Cases where the same OID has been assigned to both a V2 and a V3
  ValueSet (i.e. different objects), and cases where a CodeSystem and a
  ValueSet share the same OID.

The reporter also flags, in comments, that published THO/UTG content
(e.g. the `NamingSystem-icd10PCS` build page) marks *both* an OID and a
URI as `preferred = true`, which begs the question: preferred for what?
Preferred URIs legitimately differ between STU3 and R4, and a single
`preferred` boolean cannot express that — it should arguably be
replaced/supplemented by a version-scoped code (v2, v3/CDA, fhir-dstu2,
fhir-stu3, fhir-r4, …).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt, under TSMG stewardship, a formal policy that OID and URI
identification for HL7-governed terminology artifacts must be
administered centrally and in tandem, with the following concrete
elements:

1. Require that every HL7-published `CodeSystem` (and, where applicable,
   `ValueSet`) have a corresponding OID in the HL7 OID Registry, and
   that every newly registered HL7 terminology OID have a corresponding
   canonical URI. Apply this to new HL7 work (e.g. ICD-11 and IDMP
   bindings) and to THO/UTG additions going forward.
2. Move identifier administration into THO/UTG as the single source of
   truth, realised through `NamingSystem` resources that bind
   `CodeSystem.url` (and the OID form found in `ValueSet.compose` /
   `CodeSystem.valueSet`) to the authoritative OID.
3. Replace (or complement) the current single `NamingSystem.uniqueId.preferred`
   boolean with a version-scoped indicator — e.g. a code enumerating
   `v2`, `v3`/`CDA`, `fhir-dstu2`, `fhir-stu3`, `fhir-r4`, `fhir-r5`,
   `fhir-r6` — so that the "preferred" URI/OID for a given product
   family and version can be expressed unambiguously. File this as an
   R6 change request against `NamingSystem` with a TSMG-owned
   companion policy.
4. Codify as policy that the preferred URI/OID for a published FHIR
   version is immutable for that version (no changes in patch or minor
   releases), consistent with the reporter's breaking-change argument.
5. Produce a remediation backlog (TSMG-owned) covering the 17 R4 OIDs
   with multiple URIs enumerated in the ticket, plus the V2/V3
   collisions, and drive them through THO/UTG as NamingSystem fixes
   rather than spec changes that would themselves break implementations.

#### Justification

This is what the ticket literally asks for and it aligns with the
comments from Ted Klein and Rob McClure that the problem is real,
cross-family, and long-standing. Centralising under THO/UTG + TSMG is
consistent with HL7's stated direction of scaling V3 down to IGs,
which — as the reporter correctly observes — only works if OID↔URI
mapping is computable. The version-scoped "preferred" indicator
addresses the concrete bug that Alexander Henket and Ted Klein both
identified on the `NamingSystem-icd10PCS` page, where both an OID and a
URI are simultaneously flagged `preferred = true` without any statement
of *for which product family and version*.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than pursuing a single, broad governance ticket, split the
request into two tracks:

1. **Policy track (TSMG):** TSMG publishes a short, targeted
   governance policy that (a) requires OID assignment for any new
   HL7-published CodeSystem, (b) designates THO/UTG as the authoritative
   registry of OID↔URI bindings, and (c) forbids changing the preferred
   URI/OID for a published FHIR version. This is handled as a TSMG
   policy document, not a FHIR Core spec change.
2. **Technical track (FHIR Core, R6):** File a focused FHIR Core change
   against `NamingSystem` (and associated documentation) to add a
   version/product-family scope to the "preferred" designation — either
   as a new element on `NamingSystem.uniqueId` (e.g. `period` already
   exists; consider a coded `scope` extension) or as guidance that the
   existing `NamingSystem.uniqueId.period` MUST be populated with the
   applicable FHIR version range. The cleanup of the 17 duplicate-URI
   OIDs remains THO/UTG tickets, not Core changes.

Close FHIR-31621 itself as "addressed by" those follow-ups once they
are filed, rather than trying to resolve it directly.

#### Justification

The current ticket mixes governance policy (which FHIR Core cannot
implement) with a concrete spec gap (the single `preferred` boolean).
TSMG's 2025-11-03 comment — "the OID registry covers more than just
terminology OIDs" and "if there is no appetite for making these
changes right now, please consider closing this ticket" — suggests
that keeping this as a single monolithic ask is exactly why it has
sat for 4+ years. Splitting it lets the policy half be owned by TSMG
and the technical half be handled as a tractable R6 `NamingSystem`
change, each with its own tracking ticket. This also avoids the
unbounded-scope cleanup of every historical duplicate URI being a
blocker on the policy and on the `NamingSystem` change.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive but Not Actioned** / **Not Persuasive with Mod**.
The underlying observations are accepted (OID/URI inconsistency is
real; duplicate URIs per OID exist; the single `preferred` boolean is
under-specified), but the ticket as filed is a governance/policy ask
against FHIR Core where the actionable mechanism — NamingSystem
entries in THO/UTG — already exists and is being used. Decline on the
grounds that (a) FHIR Core is not the right venue for cross-family
HL7 identifier governance, (b) TSMG can pursue the policy
independently of this ticket, and (c) the existing `NamingSystem`
resource with its `uniqueId.preferred` + `period` elements, combined
with THO's per-CodeSystem NamingSystem pattern, is already sufficient
when populated correctly. Encourage the reporter to open discrete
THO/UTG tickets against any specific duplicate-URI OIDs they want
cleaned up.

#### Justification

This disposition reflects TSMG's own 2025-11-03 posture: agreement
with the problem statement, but no concrete appetite for a
FHIR-Core-level action and explicit willingness to close. The OID
Registry is broader than terminology, so a TSMG-led fix cannot fully
resolve it. FHIR's `NamingSystem` already has `uniqueId.period`,
which — if authors populated it — expresses the version-scoped
preferred relationship the reporter wants; the defect is largely one
of authoring discipline in THO/UTG, not of the FHIR specification.
Declining is defensible because accepting as written would commit
TSMG and THO/UTG to an open-ended cleanup and a cross-organisation
policy project that no one has resourced in 4+ years.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right balance. The ticket is important — Ted
Klein called it "critical" and Rob McClure called it "top priority" —
and the data in the description (17 duplicated-URI OIDs, V2/V3
collisions on the same OID, CodeSystem/ValueSet OID reuse) is too
concrete to close without a trace, which is what Disposition C
effectively does. But Disposition A is too ambitious to land in a
single ticket: it mixes a cross-HL7 governance policy, a FHIR Core
`NamingSystem` change, and an open-ended THO/UTG cleanup backlog,
which is precisely why this ticket has been open since 2021. Splitting
the work into a TSMG policy document, a targeted R6 `NamingSystem`
change for version-scoped "preferred", and per-duplicate THO/UTG
tickets gives each piece an owner and a realistic scope, answers
Jessica Snell's 2025-11-03 question ("do you have advice on how we
can improve our OID/URI consistency?") with a concrete plan, and lets
FHIR-31621 itself be closed as "addressed by" the spin-off tickets —
preserving the analysis while unblocking triage.
