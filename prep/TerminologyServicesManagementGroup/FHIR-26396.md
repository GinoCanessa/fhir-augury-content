## Summary

The ticket reports that the OID displayed in the summary section of the
`MedicationRequest.intent` value set (`ValueSet/medication-request-intent`,
OID `2.16.840.1.113883.4.642.3.1321`) does not appear in the HL7 OID
Registry. Because the OID is not registered, implementers have no
authoritative way to verify that the OID rendered on the FHIR specification
page actually corresponds to that value set. The reporter asks that
HL7 either register the OID in the HL7 OID Registry or update the
specification to show the correct (registered) OID. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced for this
ticket; the only follow-up on file is a Vocab/Terminology comment
acknowledging the gap and committing to register the OID.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Register OID `2.16.840.1.113883.4.642.3.1321` in the HL7 OID Registry
as the canonical OID for `ValueSet/medication-request-intent`
(`http://hl7.org/fhir/ValueSet/medication-request-intent`), then leave
the FHIR specification rendering of that OID unchanged. As part of the
registration, ensure the registry entry points back at the FHIR
canonical URL and the responsible work group (Pharmacy as the
content owner; TSMG as the registry steward), and confirm the same
OID/URL pairing in any THO mirror once the value set is moved to THO.
Carmela Couderc's 2021-02-25 comment already commits Vocab/TSMG to
performing this registration; this disposition formalises that work as
the resolution.

#### Justification

This is exactly what the reporter asked for and is the
lowest-disruption path: it preserves every existing OID reference in
implementations and downstream IGs while restoring the ability to
verify the OID against the HL7 OID Registry. It is also consistent
with the working-group commitment captured in the ticket comment, and
it directly addresses the underlying registry-hygiene gap that TSMG
has acknowledged is a systemic issue rather than a one-off. No spec
text or value set artifact needs to change.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this ticket as the trigger for a broader, programmatic fix
rather than a single OID registration: (1) register
`2.16.840.1.113883.4.642.3.1321` for `medication-request-intent` as in
Disposition A, (2) generate a reconciliation report that compares
every OID rendered in the FHIR Core value set summary sections against
the HL7 OID Registry, (3) bulk-register any FHIR Core value set OIDs
that are missing from the registry, and (4) document the OID
registration expectation in the TSMG/Vocab process so that newly
published FHIR value sets are registered as part of normative
publication. Track the bulk work under a TSMG epic and close
FHIR-26396 once the medication-request-intent OID is registered as
part of that effort.

#### Justification

The 2021-02-25 comment explicitly identifies that "Not all OIDs are in
the OID registry" and that Vocab has put this on its backlog as a
process gap. Resolving only the single value set named in this ticket
leaves the systemic issue intact and guarantees a stream of nearly
identical tickets. Bundling the fix with a reconciliation pass is more
work up front but eliminates the recurring reporter burden, makes OID
verification reliable across FHIR Core, and gives TSMG a defensible
process artifact. Trade-off vs. A: more coordination cost and a
longer time-to-close, but a much higher-leverage outcome.

---

### Disposition C: Decline

#### Proposal

Decline on the basis that the FHIR specification's rendering of the
OID is correct and unambiguous (the value set's canonical URL
`http://hl7.org/fhir/ValueSet/medication-request-intent` is the
authoritative identifier in FHIR), and that absence from the HL7 OID
Registry is a registry-process issue, not a defect in the FHIR
specification or the value set definition. Close as "Not Persuasive"
or "Already Addressed by Existing Mechanism," directing implementers
to use the canonical URL as the primary identifier and treating OID
registration as a separate operational task outside the FHIR ballot
process.

#### Justification

FHIR's identification model treats the canonical URL — not the OID —
as the primary identifier for a value set, so implementations that
need to verify the artifact can do so against the FHIR registry today.
The OID is provided for cross-paradigm interoperability (CDA, V2)
where it remains useful, but its absence from the OID Registry does
not invalidate any FHIR-conformant exchange. Declining is defensible
on scope grounds: the requested action is a registry administrative
task, not a change to the FHIR specification. The weakness of this
position is that the responsible work group has already publicly
committed (in the 2021-02-25 comment) to performing the registration,
which makes a hard decline awkward to defend.

---

### Recommendation

**Recommended disposition:** B

The ticket has been open since 2020 and the 2021-02-25 comment from
Vocab/TSMG concedes that the missing OID is a symptom of a broader
process gap, not a one-off. Disposition A discharges the literal
request but leaves the systemic problem (and the steady drip of
similar tickets) untouched; Disposition C contradicts a public
commitment already made by the responsible work group and is
politically hard to justify. Disposition B respects the commitment
in the comment, closes this ticket on its own terms (the
medication-request-intent OID does get registered), and uses the
ticket as the lever to fix the underlying registry-hygiene issue.
TSMG is the right work group to own that bulk reconciliation, so the
incremental coordination cost falls inside the group that already
agreed the work is needed.
