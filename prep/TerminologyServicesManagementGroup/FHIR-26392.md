## Summary

The reporter (John Snyder) asserts that the OID `2.16.840.1.113883.1.11.16621`,
identified in FHIR as the OID for the v3 **Act Substance Admin Substitution Code**
value set, is not resolvable in the HL7 OID Registry — the registry URL
`https://www.hl7.org/Oid/index.cfm?Comp_OID=2.16.840.1.113883.1.11.16621`
returns a 500 error. The request is to "correct the OID information on this
value set." The reporter speculates that the correct OID may be
`2.16.840.1.113883.11.19717` (note the different arc — `.11.` instead of
`.1.11.`). Comments confirm that THO (terminology.hl7.org) does in fact list
`2.16.840.1.113883.1.11.16621` as the value set's identifier; vocab
acknowledged that the OID is genuinely missing from the HL7 OID Registry, that
this reflects a wider gap in OID-registration practice, and committed to
register the OID. The ticket therefore is less about a wrong OID in FHIR/THO
than about an unregistered OID in the HL7 OID Registry, with no
cross-referenced Jira tickets, Zulip threads, or GitHub items found in
Augury.

**Comments:**

- **tedhl7geek (2021-12-20):** Notes this looks like an error in the OID
  information in the FHIR R4 resource. Because it was published in the last
  regular coremif from the "old" harmonization process in December 2019, the
  OID for this value set in THO should match the 12/2019 publication. Other
  copies of the object elsewhere in HL7 may or may not share that identifier
  and may or may not be the same value set.
- **carmela_couderc (2021-02-25):** Confirms that THO lists
  `2.16.840.1.113883.1.11.16621`. Acknowledges that not all OIDs are in the
  OID registry and that Vocab has identified a deficiency in how OID
  registration expectations and rendering are communicated. States that
  Vocab will register the OID.

## Proposed Dispositions

Note: Augury surfaces no GitHub cross-references for this ticket, so each
disposition below names the conceptual locus of change (FHIR R5 ValueSet
resource OID identifier, the THO publication of the same value set, and/or
the HL7 OID Registry) rather than a specific repo path. The HL7 OID
Registry itself is not an HL7 source spec under Augury's repo coverage.

### Disposition A: Accept As Requested

#### Proposal

Treat the request literally as "correct the OID information on this value
set." Concretely:

1. Confirm via THO maintainers that `2.16.840.1.113883.1.11.16621` is the
   intended canonical OID for the v3 Act Substance Admin Substitution Code
   value set (per Carmela Couderc's comment and per the December-2019
   coremif publication noted by tedhl7geek).
2. **Register** the OID `2.16.840.1.113883.1.11.16621` in the HL7 OID
   Registry so that
   `https://www.hl7.org/Oid/index.cfm?Comp_OID=2.16.840.1.113883.1.11.16621`
   resolves with the correct metadata (name, type=ValueSet, custodian,
   responsible body, status). This is the action the Vocab/TSMG comment
   thread already committed to.
3. Leave the FHIR R5 value-set artifact's `identifier`/OID unchanged,
   because the OID asserted by FHIR is correct — it is the registry that
   is missing the entry, not FHIR that is publishing the wrong OID.
4. Close the ticket once the registry entry resolves.

#### Justification

The two comments converge on the same finding: THO does carry
`2.16.840.1.113883.1.11.16621` as the value set's OID, which means FHIR's
copy is consistent with the canonical source. The reporter's symptom (a
500 from the OID Registry lookup) is caused by the OID never having been
registered, not by FHIR or THO publishing a wrong identifier. Registering
the OID resolves the user-visible defect without churning any
specification artifact, preserves stability for existing implementers
already using `2.16.840.1.113883.1.11.16621`, and aligns directly with
Vocab's stated remediation ("Vocab will register the OID").

---

### Disposition B: Alternative Approach

#### Proposal

Treat the registration as necessary but insufficient, and pair it with a
THO/FHIR-side cleanup that prevents the next instance of this defect:

1. Register `2.16.840.1.113883.1.11.16621` in the HL7 OID Registry
   (as in Disposition A).
2. In addition, add a TSMG/Vocab-owned **publication-time check** that
   every OID asserted by a THO value set (and by the FHIR R5 ValueSet
   resources that mirror them) is resolvable in the HL7 OID Registry
   before publication, with unregistered OIDs flagged. Carmela's comment
   explicitly identifies this as a known deficiency ("there is a
   deficiency in communication about OID registration expectations and
   rendering ... It has been added to the list of items Vocab must
   address.").
3. Document, in the THO style guide and in the FHIR ValueSet narrative
   for v3 value sets, that the canonical OID lives in THO and that the
   HL7 OID Registry entry is the authoritative human-resolvable
   reference.
4. Explicitly reject the reporter's speculative replacement OID
   `2.16.840.1.113883.11.19717` — adopting it would silently break
   conformance for every implementer who already binds against
   `2.16.840.1.113883.1.11.16621`.

#### Justification

This addresses the underlying class of defect (unregistered OIDs masquerading
as a "wrong OID" complaint) rather than just the one symptomatic OID. Because
TSMG already owns the OID-registration workflow and the comment thread
already acknowledged the systemic gap, a process fix has a much higher
return on effort than another one-off ticket cycle. It also avoids the
risk that Disposition A's narrow fix simply defers the next FHIR-26392 by
a few months. The added cost is modest: a checklist item or CI check
inside the existing publication pipeline.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed**. Close the ticket with a comment noting
that:

- THO and FHIR both publish the value set with OID
  `2.16.840.1.113883.1.11.16621`, which is correct per the
  December-2019 coremif harmonization output.
- The reporter's candidate replacement OID
  `2.16.840.1.113883.11.19717` is not the OID of this value set; using it
  would be a breaking change.
- The remaining gap — that the OID is not yet visible in the HL7 OID
  Registry — is a registry-side operational issue, not a FHIR change
  request, and Vocab has already committed to register it.

#### Justification

Strictly read, the ticket asks for "correction" of the OID in the value
set, but the investigation in the comments shows there is nothing to
correct on the FHIR side: the OID is right and matches THO. The change
request as filed therefore has no FHIR-side action, and decline-as-already-addressed
is defensible. The downside is that the user-visible symptom (500 from the
OID Registry lookup) persists until Vocab actually performs the
registration, so this disposition is only honest if there is also an
out-of-band tracking item against the registry workflow.

---

### Recommendation

**Recommended disposition:** B

The data on the ticket is unambiguous on two points: the OID FHIR/THO
publishes is correct, and the symptom is an unregistered OID rather than a
wrong OID. Disposition A fixes only this one OID; Disposition C closes the
ticket without acting on the user-visible defect; Disposition B does both —
it registers `2.16.840.1.113883.1.11.16621` (which is what the reporter
ultimately needs) and turns Carmela's already-acknowledged systemic gap
("a deficiency in communication about OID registration expectations and
rendering ... added to the list of items Vocab must address") into a
publication-time check. Given that the ticket has been triaged since 2020
and was last touched in 2022, doing the structural fix now also
discharges the standing commitment Vocab made in-thread, rather than
re-litigating the same one-OID conversation the next time an
ActSubstanceAdminSubstitutionCode-style OID surfaces.
