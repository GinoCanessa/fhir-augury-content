## Summary

The ticket asks the FHIR specification to provide explicit guidance on how
to represent *private codes* — codes that a code system steward formally
reserves for implementer-defined meanings (the canonical example is
[DICOM 7.8 Private Data Elements](https://dicom.nema.org/dicom/2013/output/chtml/part05/sect_7.8.html),
and similar patterns appear in older messaging code systems such as
ISO 11073 / IEEE x73 device codes). At present, exchanging such codes in
a `Coding` or `CodeableConcept` is awkward: validators reject them
because the published code system does not enumerate them, and there is
no agreed convention for naming a private namespace in the `system`
URI.

The reporter proposes that FHIR state (a) private codes from a code
system that explicitly defines a private space MAY be exchanged in FHIR,
(b) implementations SHOULD also include a more descriptive coding
alongside any private code (and, for `CodeableConcept`, retain the
private code), and (c) a CodeSystem extension should be defined to
declare a *range* of codes that share a single definition, so the
private space does not have to be enumerated as thousands of entries.
The single comment on the ticket (Elliot Silver) asks for matching
guidance on the `system` URI itself — whether private codes belong inside
the standard system URI or under a separate, implementer-scoped one.

The "extension on CodeSystem" half of this request was carried forward
into the closely related ticket
[FHIR-50851](https://jira.hl7.org/browse/FHIR-50851) ("Define an
extension for CodeSystem to allow defining private code system spaces"),
which the Terminology Infrastructure WG resolved 2025-09-08 as
*Persuasive with Modification*: a complex extension `reserved-code-space`
(regex + markdown description, 0..* on CodeSystem) was approved. That
resolution closes the *mechanism* part of FHIR-49671 inside the
Extensions Pack, but leaves the *normative spec guidance* — when to use
private codes, how to pair them with descriptive codings, what `system`
values are appropriate, and what validators are expected to do — still
open for TSMG (and TI) to address here.

## Details

**Description:**

Several (existing, older, or communication related) code systems have the
concept of a *private code*. A *private code* is a code for which the
code system indicates that it does not define the meaning and the field
can be used by implementers as they wish.

An example of such code includes the
[DICOM 7.8 Private Data Elements](https://dicom.nema.org/dicom/2013/output/chtml/part05/sect_7.8.html).

They typically appear in codes that used in messages where the code
system is implicit.

The FHIR specification should provide guidance on:

- whether private codes can be expressed in FHIR and
- if so, how.

There has been a discussion on this topic in
[#IG creation > supporting private codes in code system - FHIR Community - Zulip](https://chat.fhir.org/#narrow/channel/179252-IG-creation/topic/supporting.20private.20codes.20in.20code.20system).

Options mentioned in discussion are:

- Do not represent — when topic is expressed in FHIR an appropriate system
  needs to be provided that scopes the private code.
- Include the code without code system when meaning is not known.
- Include the code as a normal code in the CodeSystem (might require an
  extension on code system to link a code to a range of codes).

The reporter proposes that FHIR states that:

- FHIR resources can use private codes from code systems that define them.
- Implementations SHOULD:
  - For `CodeableConcept`:
    - add a more descriptive code whenever a private code is used.
    - include the original private code.
  - For `Coding`:
    - add a more descriptive code whenever a private code is used.

Additionally, an extension should be defined on `CodeSystem` allowing
expression that a range of codes has the same definition (preventing
adding thousands of codes with the same definition).

**Comments:**

- **Elliot Silver (esilver), 2025-01-28** — Notes that guidance on use of
  `system` values is also needed: can private codes live inside a
  standard code system URI, or does each set of private codes need its
  own corresponding code system? He suggests he knows the answers but
  they should be clearly documented as part of the resolution.

## Related Jira Tickets

### [FHIR-50851: Define an extension for CodeSystem to allow defining private code system spaces](https://jira.hl7.org/browse/FHIR-50851)

Filed by Grahame Grieve in May 2025 and resolved by Terminology
Infrastructure on 2025-09-08 as *Persuasive with Modification*. It
defines a complex extension `reserved-code-space` on `CodeSystem`
(0..*), with sub-elements `reservedCodeSpace` (string regex, 1..1) and
`description` (markdown, 0..1), so a code system can declare that codes
matching a regex are private/user-defined. Validators are expected not
to flag matching codes as invalid on the basis of the code system
definition. The Extensions Pack is the publication target; the
disposition explicitly asked the submitter to file a separate ticket
for the validation-behaviour work, which suggests TSMG/TI guidance for
*using* the extension (and for the `system` URI question raised here)
remains an open downstream task — exactly what FHIR-49671 covers.

Users involved: Grahame Grieve (reporter), Carmela Couderc, Oliver Egger,
JIRAUSER17326.

## Related Zulip Discussions

### [IG creation > supporting private codes in code system](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/supporting%20private%20codes%20in%20code%20system)

The discussion is the direct origin of this ticket. It was started in
`#implementers` in November 2024 and moved to `#IG creation` by Lloyd
McKenzie. Participants debate three approaches: (1) refuse to represent
private codes in FHIR and require implementers to mint their own
`system` URI; (2) send the code with no system when the meaning is
unknown; (3) treat private codes as legitimate codes in the published
code system, supported by a CodeSystem extension that declares a range
(regex) as a reserved private space. Grahame Grieve advocates for
option (3) and notes the regex approach is attractive because it could
also let stewards attach properties to the entire private range. John
Moehrke and Elliot Silver push back on whether a *consumer* can
unilaterally redefine a steward's code system (they cannot — only the
steward can declare a private space) and worry about open-ended regexes
(`[\s\S]*`) effectively whitelisting any code. The thread converged
toward option (3) and led directly to FHIR-50851 (mentioned in the
later message by Carmela Couderc on 2025-12-01); it did not resolve the
parallel `system`-URI guidance that this ticket is asking TSMG to
address.

Participants: Bas van den Heuvel, Lloyd McKenzie, Grahame Grieve, John
Moehrke, Elliot Silver, Carmela Couderc.

## Proposed Dispositions

> Note on grounding: because there are no GitHub cross-references, no
> per-repo briefing constrains the change targets below. The
> dispositions reference the FHIR Core specification (the spec
> domain implied by the ticket's `Specification = FHIR Core (FHIR)`
> field) and, where relevant, the already-approved `reserved-code-space`
> extension in the FHIR Extensions Pack.

### Disposition A: Accept As Requested

#### Proposal

Add a new normative subsection to the FHIR Core terminology guidance
(under `terminologies.html` / "Using Codes in Resources") titled
"Private Codes" that states:

- A FHIR `Coding` MAY carry a private code defined inside a published
  code system *iff* that code system declares a reserved private space.
  In FHIR R6+, a steward declares such a space using the
  `reserved-code-space` extension on `CodeSystem` (FHIR-50851).
- The `Coding.system` for a private code SHALL be the canonical URI of
  the publishing code system (not an implementer-minted URI). Codes
  outside any declared reserved space remain invalid against that
  system.
- For `CodeableConcept`, implementers SHOULD include both the original
  private `Coding` and a second, more descriptive `Coding` (from a
  shared standard code system or an implementer code system) when one
  is available, so receivers that cannot interpret the private code can
  still act on the descriptive one.
- For a single `Coding` (no `CodeableConcept`), implementers SHOULD
  prefer a descriptive code from a standard system if available;
  otherwise the private code MAY stand alone.
- Validators SHALL NOT raise an `unknown-code` error for a code that
  matches a `reserved-code-space` regex on the referenced `CodeSystem`,
  but MAY still warn that no `display` / definition is available.

This packages the guidance the reporter requested, ties it to the
already-approved extension, and answers Elliot Silver's `system`-URI
question explicitly.

#### Justification

This is the most direct way to make existing, real-world private-code
patterns (DICOM 7.8, ISO 11073) representable in FHIR without forcing
each implementer to invent their own conventions. It composes cleanly
with FHIR-50851 (which already defines the steward-side mechanism) and
with the existing `CodeableConcept` / `Coding` model — no new resource
or datatype changes are required, only normative prose plus the
existing extension. The "include a descriptive coding alongside" SHOULD
matches established FHIR practice (e.g., translation codings, the
`text` fall-back), so implementers will recognise the pattern.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a narrower change: publish the guidance as a non-normative
**Implementation Notes** page (or as a section in the Terminology
module's narrative), rather than adding new SHALL/SHOULD statements to
core terminology rules. The page would:

- Document the three options surfaced in the Zulip thread and the
  trade-offs of each.
- Recommend the steward-led pattern (FHIR-50851's `reserved-code-space`
  extension) as the preferred approach when the code system steward
  cooperates.
- Document a fallback pattern for *uncooperative* legacy systems:
  publish a sibling `CodeSystem` resource with `content = supplement`
  (or a separately-minted system URI in an implementer-controlled
  namespace) that scopes the private codes, and reference it from the
  `Coding.system`.
- Defer the `system`-URI rule and the validator-behaviour rule to TI
  through follow-up tickets (the validator question is already flagged
  by FHIR-50851's resolution).

#### Justification

The reporter's request mixes three concerns: (1) representation of
private codes, (2) `system` URI conventions, and (3) validator
behaviour. Each has implications for tooling (HAPI, `org.hl7.fhir.core`
validator, terminology servers) and for cross-WG agreement
(TI owns CodeSystem semantics; CDS and FHIR-I own resource-level
validation behaviour). Encoding all three as normative SHALLs in one
release risks shipping rules that TI's `reserved-code-space` work has
not yet operationalised. A narrative recommendation gets implementers
unstuck immediately, lets TSMG land FHIR-49671 in this cycle, and
preserves room for TI to refine the normative validator rules
afterwards. It also gives a documented path for *legacy* code systems
whose stewards will never publish a `reserved-code-space` extension —
something Disposition A does not address.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-49671 *Not Persuasive* on the grounds that the underlying
representation question is already resolved: FHIR-50851 has approved
the `reserved-code-space` extension, and the existing `Coding` /
`CodeableConcept` rules already permit any code from a referenced code
system once that code system says the code is valid. Forward Elliot
Silver's `system`-URI sub-question to a new TI ticket and close
FHIR-49671 with a pointer to FHIR-50851.

#### Justification

A defensible read of the ticket is that the bulk of the ask — "should
FHIR allow private codes, and is there a CodeSystem mechanism for
declaring them" — has been answered by FHIR-50851. Adding parallel
normative guidance in FHIR Core risks two normative voices on the same
mechanism (the extension definition vs. the core spec prose). Declining
keeps the spec lean, avoids re-litigating the Zulip thread, and routes
the genuinely-open sub-question (`system` URI conventions) to the WG
that owns CodeSystem semantics.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying problem is real and the community discussion
clearly converged toward a steward-led "reserved code space" pattern,
which FHIR-50851 already implements as an extension. What is still
missing is *guidance for implementers* on when to use that mechanism,
how to dual-code with a descriptive coding, and what `system` to put on
the wire — and Elliot Silver's comment shows that the `system` question
in particular is not yet settled. Disposition A would answer all of
this in one go, but it bakes new SHALL/SHOULD rules into FHIR Core
ahead of the validator-behaviour ticket that TI itself asked Grahame to
file separately, and it offers no path for legacy code systems whose
stewards will not adopt `reserved-code-space`. Disposition C
under-delivers — it leaves the `Coding`-pairing recommendation, the
legacy fallback, and the `system`-URI question all unanswered, even
though TSMG is the right WG to take a position on those.

Disposition B captures the practical guidance the reporter and
commenters asked for, ties it to the already-approved extension,
explicitly handles the legacy / steward-uncooperative case, and routes
the still-open normative sub-questions to TI as scoped follow-ups —
without committing FHIR Core to validator rules that the related TI
ticket has not yet defined.
