## Summary

The reporter identifies that the OID listed in the summary section of the
`MedicationStatus` value set (subsequently confirmed in the comment thread as
`2.16.840.1.113883.4.642.3.376`) does not appear in the HL7 OID registry.
Because the registry has no entry for this OID, implementers cannot
independently verify that the OID published in the FHIR specification is in
fact the correct identifier for this value set. The ticket asks the
Terminology Services Management Group (Vocab) to either (a) register the
OID in the HL7 OID Registry so it is discoverable and authoritative, or
(b) correct the specification to use whatever OID is actually registered.
A Vocab response on the ticket (carmela_couderc, 2021-02-25) acknowledges
that not all FHIR-published OIDs have been registered, that the value set
`MedicationStatus` is also missing from THO, and commits Vocab to register
the OID and to address the broader gap in OID-registration expectations and
rendering. No cross-referenced Jira tickets were surfaced for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Have the Terminology Services Management Group (Vocab) register OID
`2.16.840.1.113883.4.642.3.376` in the HL7 OID Registry against the
canonical `MedicationStatus` value set
(`http://hl7.org/fhir/ValueSet/medication-status`), and migrate the value
set itself into THO so its summary block, OID, and canonical URL all
resolve to a single authoritative source. No change to the FHIR Core
specification's value-set definition or content is required — only the
registry entry and the THO publication are added. Once registered, the
existing summary-section OID rendering continues to be valid and is now
verifiable by implementers via the OID Registry lookup.

#### Justification

This is exactly what the reporter asked for and what Vocab committed to
in the 2021 comment ("Vocab will register the OID"). It preserves the
already-published OID, so no implementer who has bound to
`2.16.840.1.113883.4.642.3.376` is broken. It also aligns with the
broader Vocab work item the comment references — closing the gap in OID
registration and rendering expectations — by treating this ticket as one
concrete instance of that programme. Because the value set is owned by
Vocab and lives in the Medication module's terminology surface, the
authoring change is registry/THO metadata only and carries no
downstream code or profile breakage.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than treating this as a one-off registration, fold FHIR-26397 into
a Vocab-wide audit: enumerate every `ValueSet` (and `CodeSystem`)
published in FHIR Core whose summary-section OID is not present in the
HL7 OID Registry, register them as a batch, and add a build-time check
that fails publication when a summary-section OID has no corresponding
registry entry. For this specific ticket, register
`2.16.840.1.113883.4.642.3.376` as part of that batch and migrate the
`MedicationStatus` value set to THO at the same time. Document the OID
registration expectation in the Vocab authoring guidance so future value
sets cannot ship with an unregistered OID.

#### Justification

The 2021 Vocab comment explicitly identifies this as a systemic issue
("Not all OIDs are in the OID registry … added to the list of items
Vocab must address"). A one-ticket fix solves the symptom but not the
class of defect, and the workgroup will keep receiving identical tickets
for other value sets. Adding a publication-time guard is the only
durable way to prevent recurrence and is consistent with how FHIR Core
already enforces other terminology-publication invariants. The trade-off
versus Disposition A is more upfront effort for Vocab and tooling, but
the marginal cost of registering one more OID inside an audited batch is
small.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the OID is already declared in
the FHIR Core specification's value-set summary, that THO publication
and OID-registry registration are operational/governance activities
outside the scope of a per-resource Change Request, and that no
specification text needs to change. Close the ticket as "Not
Persuasive" with a pointer to the broader Vocab OID-registration
backlog item referenced in carmela_couderc's 2021 comment.

#### Justification

Strictly read, this ticket asks for an administrative action (register
an OID) rather than a normative spec change, and Vocab has already
acknowledged the underlying programme. Declining as a discrete ticket
keeps the Jira queue focused on specification changes and consolidates
the work into the existing Vocab backlog. The defensible rationale is
"already addressed by an existing mechanism" — namely, Vocab's
in-flight OID-registration cleanup. The downside is that the reporter's
verifiability concern remains unaddressed until the broader programme
lands, and the OID continues to look unverifiable in the meantime.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the literal commitment Vocab already made on the
ticket and is the right floor for action; however, the comment thread
itself frames this issue as one instance of a recurring class
("deficiency in communication about OID registration expectations and
rendering"). Doing the work as a Vocab-wide audit plus a publication
guard (Disposition B) discharges Disposition A's promise for this
specific OID while preventing the same ticket being re-filed against
every other unregistered value-set OID. Disposition C is hard to
justify given that Vocab has explicitly accepted the work on the public
record, and would leave a verifiability gap that genuinely impedes
implementers who try to validate the OID against the HL7 registry.
Recommend the workgroup adopt B and treat FHIR-26397 as the tracking
ticket for the `MedicationStatus`-specific slice of the broader
registration cleanup.
