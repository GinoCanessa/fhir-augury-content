## Summary

Lin Zhang asks how an HL7 v3 OID that identifies a `ValueSet` should be
expressed in FHIR — specifically, which kind of identifier on the
`ValueSet` resource the OID corresponds to. FHIR `ValueSet` carries
multiple identifier-like elements (`ValueSet.url`, `ValueSet.identifier`,
`ValueSet.version`), and the ticket title implies the reporter is unsure
which of these is the correct home for a v3 OID such as
`2.16.840.1.113883.x.y.z`. The ticket has no description body — only the
question in the title — and was triaged but never resolved. The single
comment (tedhl7geek, 2021-12-20) is "Do not understand the question",
indicating the workgroup needs clarification from the reporter before it
can be acted on. No related Jira tickets, Zulip threads, or GitHub items
were surfaced via cross-references or search.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a documentation / guidance request and add explicit
guidance to the `ValueSet` resource page (and to the FHIR Identifier
Registry page where OID conventions are described) clarifying how to
represent an HL7 v3 OID that identifies a value set:

- `ValueSet.url` (canonical URL) **MAY** carry the OID using the
  `urn:oid:` URI form (e.g.
  `urn:oid:2.16.840.1.113883.4.642.3.1`) when no http(s) canonical URL
  exists; otherwise the `urn:oid:` form should appear in
  `ValueSet.identifier`.
- `ValueSet.identifier` (an `Identifier` datatype) is the preferred home
  for a v3 OID that *identifies the same logical value set* but is not
  its canonical URL. Use `identifier.system =
  "urn:ietf:rfc:3986"` and `identifier.value =
  "urn:oid:2.16.840.1.113883.x.y.z"`.
- `ValueSet.version` is **not** the place for an OID; it carries the
  business version of the value set definition.

Add a short example to the `ValueSet` page showing both options.

#### Justification

This is consistent with the existing FHIR convention for representing
OIDs as URIs (`urn:oid:`) and with the established pattern that
`Identifier.system = "urn:ietf:rfc:3986"` is used when the identifier
value is itself a URI. It directly answers the reporter's question
without changing the resource model and improves the on-ramp for
implementers migrating v3 vocabularies into FHIR — a common scenario
flagged repeatedly to TSMG.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only documenting the convention, ask the reporter to
clarify the use case and then, if confirmed, define a small
`ValueSet`-level guidance block that distinguishes three cases and
points at canonical URI patterns already in use elsewhere in the
specification (e.g., the Identifier Registry):

1. The OID **is** the canonical identifier for the value set → put it
   in `ValueSet.url` as `urn:oid:…`.
2. The OID is an *additional* business identifier (e.g., a registry
   entry that is distinct from the FHIR canonical URL) → put it in
   `ValueSet.identifier` with `system = "urn:ietf:rfc:3986"` and
   `value = "urn:oid:…"`.
3. The OID is a v3 *code system* OID that the value set draws from →
   it does not belong on `ValueSet` at all; it belongs on the
   `CodeSystem` resource (or in `ValueSet.compose.include.system` as a
   `urn:oid:…` URI).

Defer any normative wording until the reporter confirms which of
(1)/(2)/(3) they actually meant — the comment trail shows the workgroup
already had to ask.

#### Justification

The existing comment ("Do not understand the question") shows the ticket
is genuinely ambiguous: a v3 OID could identify the value set itself, a
contained code system, or a registry entry. Producing guidance for all
three cases avoids re-opening the ticket later and is more useful to
implementers than answering only one interpretation. It also keeps
TSMG's editorial scope tight — no schema changes, only clarifying prose
plus an example block — and matches how identifier/OID guidance is
already structured elsewhere in the spec.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — Question, Not Change Request**
(or **Not Persuasive with Modifier — Insufficient Information**),
noting that:

- The existing FHIR Identifier Registry and the documentation of the
  `Identifier` datatype already describe how to represent OIDs as URIs
  (`urn:oid:…`) and how to use `system = "urn:ietf:rfc:3986"` for
  URI-valued identifiers.
- `ValueSet.url`, `ValueSet.identifier`, and `ValueSet.version` are
  each defined in the resource page with descriptions sufficient to
  answer this question.
- The reporter never responded to the 2021 request for clarification,
  and there are no cross-references, linked tickets, or Zulip
  discussions indicating an active community need.

Invite the reporter to re-file with a concrete proposed change if
existing guidance is insufficient.

#### Justification

The ticket has been open since 2019, has no description body, was
explicitly questioned by the workgroup in 2021, and received no
follow-up. There is no community signal (no cross-references, no Zulip
threads, no linked tickets) that the underlying mechanism is unclear
to other implementers — the existing Identifier Registry and `ValueSet`
documentation cover the question. Declining is consistent with TSMG's
practice for stale, single-reporter "what does X mean?" questions that
do not request a specific spec change.

---

### Recommendation

**Recommended disposition:** B

The ticket asks a real question that recurs for v3 → FHIR migrators,
but as written it is ambiguous in exactly the way the 2021 comment
flagged: a "v3 OID of a ValueSet" can mean the canonical identifier of
the value set, an additional registry identifier, or the OID of a code
system the value set draws from. Disposition A picks one
interpretation and risks being wrong for the reporter's actual case;
Disposition C throws away a useful documentation opportunity. B
threads the needle: it acknowledges the question is real and worth
answering, defers normative wording until the reporter confirms intent,
and — if confirmed — produces guidance that covers all three plausible
readings. This is a cheap, editorial-only change that improves the
`ValueSet` page without committing TSMG to a model change.
