## Summary

The reporter (Lloyd McKenzie) requests that HL7's publishing machinery
enforce that `CodeSystem.content` is always set to `complete` for any
CodeSystem whose canonical URL is rooted at `hl7.org` (including
`terminology.hl7.org` and `hl7.org/fhir`). The rationale is that every
code system HL7 itself defines is — by definition — one for which HL7
holds the complete authoritative set of codes, so values like
`fragment`, `example`, `not-present`, or `supplement` should never
appear on an HL7-defined CodeSystem instance. The ticket explicitly
notes this is a tooling change but asks TSMG first for the policy
endorsement before tooling work proceeds. No related tickets, Zulip
threads, or GitHub items were surfaced through cross-references; the
only cross-referenced item is the FHIR artifact `CodeSystem.content`
itself, confirming that this ticket is about constraining how that
specific element is populated for HL7-published code systems.

## Details

**Description:**

The publishing machinery should enforce that CodeSystem.content for
code systems with a canonical URL with a base of "hl7.org" (i.e.
terminology.hl7.org, hl7.org/fhir, etc.) must always be 'complete'.

All code systems that HL7 defines are ones that, by definition, we
define the complete set of codes in. Therefore, there's never any
situation where the CodeSystem instances we produce would be anything
other than the complete set of codes. Similarly, there's no reason
for HL7 to ever supplement its own code systems. If we want something
in one of our code systems, we should put it there.

This is a tooling change, but before the tooling change can occur, we
first need the blessing of the TSMG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

TSMG endorses the rule and authorizes the tooling change: the FHIR
publishing machinery (IG publisher / core build) shall reject any
`CodeSystem` resource whose `CodeSystem.url` begins with
`http://hl7.org/`, `http://terminology.hl7.org/`,
`https://hl7.org/`, or `https://terminology.hl7.org/` unless
`CodeSystem.content = complete`. The check should fire as a build
error (not a warning) for HL7-published artifacts, with a clear error
message naming the offending CodeSystem and its current `content`
value. Concretely: add an invariant-style check in the publisher's
CodeSystem validation pass and surface it through the standard QA
report. Existing HL7-published CodeSystems should be audited as part
of the rollout, and any that currently declare `fragment`,
`example`, `not-present`, or `supplement` should be either corrected
to `complete` (with the missing codes added) or moved off an
`hl7.org`-rooted canonical URL.

#### Justification

The proposal is consistent with HL7's role as the authoritative
source for the code systems it publishes: when HL7 owns the
namespace, anything other than `complete` is by definition
under-stating what HL7 has authoritatively defined, and a
`supplement` of an HL7 code system by HL7 itself is incoherent (HL7
should just edit the source). Both Lloyd (reporter) and the TSMG
discussion on 2025-11-03 converged on this position. Enforcing the
rule in tooling — rather than relying on author discipline — closes
a gap that currently allows mis-stated `content` values to slip into
HL7-published terminology, which downstream terminology servers and
validators rely on to know whether expansion is exhaustive.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the rule but stage it more gently: implement the check first as
a **warning** in the publisher's QA report for one publication cycle,
along with an inventory of every HL7-published CodeSystem whose
`content` is not `complete`. After the cycle, with the inventory
remediated and any genuine exceptions documented, promote the check
to a build error. Optionally, define an explicit, narrow allow-list
mechanism (e.g., a publisher configuration entry) for any
edge-case HL7 CodeSystem that legitimately cannot be `complete` — for
example, code systems that wrap an externally-maintained vocabulary
distributed under HL7's namespace for historical reasons. The allow-
list would be reviewed and approved by TSMG.

#### Justification

This achieves the same end state as Disposition A but reduces the
risk that the first build under the new rule fails noisily across
many IGs and core. Given the ticket has been open since 2022, there
may be existing HL7 CodeSystem instances that quietly declare a
non-`complete` content mode; turning the check straight to error
without an inventory pass could block publications until those are
fixed. A short warning phase plus a TSMG-governed exception list
preserves the policy intent while giving authors a path forward in
edge cases nobody has yet identified.

---

### Disposition C: Decline

#### Proposal

Do not adopt a tooling-enforced rule. Instead, document the
expectation in the terminology authoring guidance for HL7-published
CodeSystems (i.e., a normative-sounding "SHOULD use
`content = complete`" statement in the CodeSystem page or TSMG
authoring guide) and leave enforcement to reviewer judgement during
the standard ballot/publication process.

#### Justification

A decline is defensible only if TSMG believes there are legitimate
HL7-published CodeSystems whose `content` should not be `complete`
(e.g., HL7 republishing a snapshot of an externally-maintained
vocabulary, or a deliberately partial enumeration), and that case-by-
case judgement is preferable to a hard tooling rule. It is also the
lowest-cost path because it requires no publisher changes. However,
the ticket explicitly frames this as a tooling enforcement request,
and the reporter and TSMG comments to date both endorse enforcement,
so this disposition would only make sense if the workgroup discovers
counter-examples during review.

---

### Recommendation

**Recommended disposition:** B

The substantive policy is not in dispute — both Lloyd and the TSMG
discussion on 2025-11-03 endorse it, and the design rationale is
sound: HL7-published CodeSystems are by construction `complete`, and
HL7 should not be supplementing its own code systems. The risk in
Disposition A is purely operational: because the ticket has been open
since 2022, nobody has done a current inventory of HL7-published
CodeSystem instances, and flipping the check straight to a build
error could fail core / IG publications on day one without warning.
Disposition B preserves the full policy intent, gives TSMG a
defensible audit trail (the warning-phase inventory), and provides a
narrow, governed escape hatch for any genuine edge case that surfaces
during the warning cycle — after which the rule is promoted to a hard
error, matching what Disposition A would have done. Disposition C is
not recommended because the workgroup has already signalled
agreement with enforcement; relying on reviewer judgement is what got
us to a state where the rule needs to be written down in the first
place.
