## Summary

The ticket reports that the published FHIR specification page for HL7
Version 2 Table 0004 (section 4.3.15.4 — `v2/0004`) renders the
value-set/code-system summary block with `??` in the OID field instead
of a real OID. The reporter requests that the OID be populated, pointing
to an OID that Ted Klein registered in the HL7 OID Registry under branch
`2.16.840.1.113883.12` (the V2 table branch). A subsequent comment from
the v2 table steward pushes back: those `…12.x` OIDs identify the
printed PDF artifact for a V2 table, not the FHIR CodeSystem/ValueSet
that FHIR Core renders, and the authoritative table metadata (including
the correct table OID and any associated value-set / code-system OIDs)
already lives in THO's `v2tables.xml`. The reporter agrees the THO
content is correct and that the FHIR Core page is simply out of sync
with what UTG/THO publishes, with the sync gap to be tracked under a
separate ticket. There are no cross-referenced Jira tickets, Zulip
threads, or GitHub items recorded for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Populate the OID field on the FHIR Core spec page for HL7 V2 Table 0004
(`v2/0004`, section 4.3.15.4) so that the summary block no longer
renders `??`. Specifically, fill the value with the OID Ted Klein
registered under the V2 table OID branch
(`2.16.840.1.113883.12.4` for table 0004). The change would land in
whatever generator inputs feed the v2 table summary on the FHIR Core
spec page, and the published HTML would then show that OID for table
0004's value set/code system.

#### Justification

The reporter's stated goal is purely cosmetic-but-real: the published
spec page advertises an OID slot and currently renders a `??`
placeholder, which looks like a defect to implementers and gives them
no usable identifier. The HL7 OID Registry does carry an entry under
the V2 table branch for table 0004, so the OID is not invented — it
exists and is publicly reachable. Filling it would close the visible
gap on the page that this ticket was filed against.

---

### Disposition B: Alternative Approach — Resync the FHIR Core v2 Table Page From THO

#### Proposal

Treat the missing OID as a symptom of the FHIR Core `v2/0004` page
being out of sync with `terminology.hl7.org`, rather than as a missing
field to hand-populate. Concretely:

1. Confirm that THO's `v2tables.xml` carries the authoritative OID(s)
   for table 0004 — the table OID itself, plus any associated
   ValueSet/CodeSystem OIDs — as the v2 steward asserts.
2. Update the FHIR Core spec generator that emits the v2 table summary
   pages (the section-4.3.15.x family) to read OIDs from the THO source
   of truth instead of from whatever local store currently yields `??`.
3. Regenerate `v2/0004` (and the other v2/000x summary pages) so they
   render the THO-sourced OIDs consistently, and verify that the
   correct OID — which per the v2 steward is *not* necessarily
   `…12.4` — appears.

If THO is in fact missing or wrong for table 0004, file a UTG ticket
against THO first and let the FHIR Core page pick up the corrected
value when regenerated.

#### Justification

The v2 table steward has explicitly stated that the OIDs under
`2.16.840.1.113883.12.x` identify the printed PDF artifacts of V2
tables, not the FHIR CodeSystem/ValueSet resources that the FHIR Core
page is summarizing. Hand-pasting `…12.4` per the literal request
would therefore publish a semantically wrong identifier under a
heading that implementers will reasonably interpret as the
CodeSystem/ValueSet OID. The reporter has already conceded in-thread
that THO is correct and that the Core page is simply out of sync, so
the durable fix is to pull from THO rather than to patch a single
field. This also fixes the same class of defect for any sibling v2
tables that render `??` for the same reason, without requiring a
ticket per table.

---

### Disposition C: Decline / Close as Superseded

#### Proposal

Decline to make a content change under this ticket on the grounds that
(a) the literal suggested fix is incorrect per the v2 table steward,
and (b) the reporter has acknowledged that the underlying problem is a
THO↔FHIR-Core sync gap that "we will address in another ticket." Close
`FHIR-26384` as superseded / not-persuasive, with a pointer to the
THO-sync work item (or asking the reporter to file one if it does not
yet exist).

#### Justification

The ticket has been Triaged for years with no cross-references, no
Zulip discussion, and no GitHub work, and its own comment thread
already documents two reasons not to act on it as written: the
proposed OID is the wrong identifier, and the real fix belongs in a
different work stream (the THO↔Core sync). Acting on the literal
request would publish a wrong OID; acting on the underlying issue
duplicates work that the reporter has explicitly deferred to another
ticket. Declining keeps the spec from being changed under a ticket
whose recommended action has been refuted on the record.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the only path that actually fixes the user-visible
defect (`??` in the OID slot on `v2/0004`) without publishing the
wrong identifier. The v2 table steward's comment is decisive against
Disposition A: `2.16.840.1.113883.12.4` identifies the printed PDF
artifact of table 0004, not the FHIR CodeSystem/ValueSet rendered on
the spec page, so accepting the literal request would replace one
defect (`??`) with a worse one (a confidently-wrong OID). Disposition
C is defensible procedurally — the ticket is stale and the reporter
deferred the real fix — but it leaves the `??` rendering in place and
relies on an unfiled follow-up ticket that has not materialised in the
years since. Disposition B routes the fix through THO, which both
sides of the comment thread already agree is the source of truth, and
generalises to the rest of the v2/000x summary pages. The workgroup
should confirm whether the THO→Core regeneration tooling exists today
or needs to be built, and whether table 0004 specifically is correct
in `v2tables.xml` before regenerating.
