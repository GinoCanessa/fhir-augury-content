## Summary

The reporter (Alexander Henket) lists 24 V3-derived ValueSets whose
authoritative HL7 V3 OIDs are *not* surfaced on the corresponding FHIR
R4 ValueSet pages. The asserted V3 OIDs are provided for each entry
(e.g., `AcknowledgementDetailCode = 2.16.840.1.113883.1.11.19638`,
`Race = 2.16.840.1.113883.1.11.14914`,
`MaritalStatus = 2.16.840.1.113883.1.11.12212`,
`OrderableDrugForm = 2.16.840.1.113883.1.11.14411`). The ask is that
these OIDs be added to the FHIR ValueSet metadata so consumers
(particularly V3/CDA implementers) can correlate the FHIR copies with
their canonical V3 identifiers.

The comment thread shifts the framing from "FHIR-spec defect" to "spec
synchronization defect": Jessica Bota (`jsnell`) confirmed the OIDs are
present and correct on terminology.hl7.org (THO), Carmela Couderc noted
Vocab considered this a UTG ticket, and Ted Klein (`tedhl7geek`) added
that ValueSets that were migrated to THO should have been merged with
THO content from the V3 coremif during migration; any miss is "an error
that needs to be manually fixed", and the V3-derived FHIR R4 copies
were not retro-fitted with the V3 OIDs. The ticket has therefore sat in
**Triaged** as essentially a UTG/THO sync question rather than a
substantive R5 spec change.

There are no recorded cross-references, no Zulip discussions, and no
GitHub items linked to this ticket; related context is limited to other
OID-housekeeping tickets surfaced by keyword similarity (see Related
Jira Tickets).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an `identifier` element with `system = urn:ietf:rfc:3986` and
`value = urn:oid:<oid>` to each of the 24 listed FHIR ValueSet
resources, using the V3 OID values supplied by the reporter (e.g.,
`urn:oid:2.16.840.1.113883.1.11.19638` on `AcknowledgementDetailCode`,
`urn:oid:2.16.840.1.113883.1.11.14914` on `Race`, …). Verify each
candidate OID against the corresponding entry on
terminology.hl7.org / UTG before merging — Jessica Bota's comment
indicates the OIDs are already correct in THO, so the change is
effectively a copy-down from THO into the FHIR-side ValueSet
definitions for the FHIR Core publication. Apply to the next FHIR
release line (R6, given the R5 label is now historic) and *not* to R4.

#### Justification

The reporter's underlying need — that FHIR copies of V3 ValueSets be
correlatable with their canonical V3 identifiers — is legitimate and
unambiguously fixed by surfacing the existing THO OIDs on the FHIR
ValueSet resources. THO already holds the authoritative values
(`jsnell` confirmed and sampled them), so the change carries effectively
zero terminology risk; it is a metadata-only addition, fully
backward-compatible, and directly supports V3↔FHIR interoperability.
Restricting the change to the next release line (rather than
back-patching R4) honours Ted Klein's observation that there is no
decision to amend the older spec.

---

### Disposition B: Alternative Approach

#### Proposal

Do not modify the FHIR Core ValueSet resources. Instead, treat this as a
UTG/THO synchronization defect (consistent with Jessica Bota's and
Carmela Couderc's framing) and address it by ensuring that, for any
V3-derived ValueSet, the FHIR Core publication renders the OID *from
THO* on the ValueSet HTML page (e.g., in the rendered "Identifiers"
block) rather than embedding it in the FHIR resource itself. Open (or
re-target) a UTG ticket to: (a) confirm each of the 24 OIDs is present
and correct in THO, (b) ensure the publication tooling reads the
identifier from THO when generating the FHIR ValueSet page. Close
FHIR-20376 with a pointer to the UTG ticket and to FHIR-31621 for the
broader governance question.

#### Justification

This avoids duplicating identifier metadata in two places (a recurring
maintenance hazard the comments explicitly flag — "out of sync with
UTG"), keeps THO as the single source of truth for V3-derived
ValueSets, and aligns with how HL7 has been progressively migrating V3
content out of FHIR Core. It also respects the precedent set by
FHIR-26412 and FHIR-26418, where TSMG / FHIR Mgmt declined to embed
OID registrations into FHIR-native artifacts. The trade-off vs.
Disposition A is implementer convenience: consumers who only read the
FHIR ValueSet resource (not the HTML page) still won't see the OID
unless they resolve to THO, but that is consistent with the broader
direction of travel.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive – No Change**, citing that (1) the OIDs are
already correct and present in THO per `jsnell`'s investigation, (2)
the FHIR R4 copies of these V3 ValueSets are frozen and not being
retrofitted per `tedhl7geek`'s comment, and (3) HL7 has consistently
declined to register/embed OIDs for FHIR-native ValueSet artifacts
(precedents: FHIR-26412 *Resolved – No Change*, FHIR-26418
*Resolved – No Change*). Direct the reporter to use the THO canonical
URL (`http://terminology.hl7.org/ValueSet/...`) when correlating with
V3 OIDs.

#### Justification

The defensible argument is that the request was already satisfied by
the V3-to-THO migration: the authoritative artifacts (and their OIDs)
now live in THO, and FHIR Core has been deliberately divesting V3
content. Re-embedding OIDs in FHIR Core ValueSet resources would
recreate the very out-of-sync condition that motivated the migration in
the first place. The two same-shape tickets (FHIR-26412, FHIR-26418)
both closed *No Change* on this exact governance ground.

---

### Recommendation

**Recommended disposition:** B

The comment record on this ticket already converges on Disposition B's
framing: this is a UTG/THO ↔ FHIR-publication synchronization issue,
not a FHIR-resource defect. Disposition C is technically defensible on
precedent (FHIR-26412, FHIR-26418) but is unsatisfying — the reporter's
underlying need is real, the data exists in THO, and TSMG owns the
machinery to surface it. Disposition A solves the symptom but
re-introduces dual-maintained OIDs across THO and FHIR Core, exactly
what `jsnell`'s "out of sync with UTG" comment warns against.
Disposition B keeps THO authoritative, removes the "FHIR page doesn't
show the OID" pain that drove the ticket, dovetails with FHIR-31621's
broader OID/URI governance push from the same reporter, and is
implementable as a publication-tooling change rather than a normative
content change. TSMG should pair the disposition with a check that all
24 listed OIDs are in fact present and correct in THO before closing.
