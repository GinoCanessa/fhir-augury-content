## Summary

A comparison of the `PatientContactRelationship` value set (bound to
`Patient.contact.role`) between R4 and R5 shows that the R5 expansion
includes four additional codes from v2 table 0131, at least one of which
duplicates an existing code. The reporter (Lorraine Constable) suspects
the R5 expansion is silently surfacing codes that v2 considers deprecated,
which is confusing for implementers because the deprecation is not visible
in the expansion. Investigation by FHIR-I, PA, Terminology Infrastructure,
and the THO maintainers traced the root cause to the underlying HL7 v2
code system [`v2-0131`](https://terminology.hl7.org/7.1.0/en/CodeSystem-v2-0131.html),
which marks deprecated concepts using a custom THO property
(`http://terminology.hl7.org/CodeSystem/utg-concept-properties#v2-table-deprecated`)
rather than the standard FHIR concept property
[`status=deprecated`](http://hl7.org/fhir/concept-properties#status) (and/or
`inactive=true`). Standard expansion logic does not understand the custom
property, so those concepts are neither suppressed from expansions nor
flagged as deprecated in the rendered value set. The most recent comment
(2026-04-13) records that FHIR-I has agreed the change requires TSMG
approval so that a follow-up UTG/UP ticket can be raised against THO. A
related ticket [FHIR-39206](https://jira.hl7.org/browse/FHIR-39206)
("Deprecated concepts are not indicated or suppressed") was filed against
PA for the same problem on `PatientContactRelationship` and has been
closed as a Duplicate of FHIR-32232.

## Details

**Description:**

A comparison of the ValueSet Patient Contact Relationship in R4 and R5
indicates that R5 is listing 4 additional codes, at least one of which is
duplicative with the R4 codes. Referencing Table 0131 in v2.9 did not show
these 4 codes.

Best guess is that the current R5 expansion is including deprecated codes?
If so, these additions will be confusing to implementers unless they are
marked as such.

### [terminology / utg > deprecating concepts](https://chat.fhir.org/#narrow/stream/terminology%20%2F%20utg/topic/deprecating%20concepts/near/422554042)

Brian Postlethwaite, while processing UTG tickets, asked which of the
standard concept properties (`status`, `inactive`, `deprecationDate`)
should be used to deprecate concepts and explicitly tied the question to
FHIR-32232. Jessica Bota explained that
[`status=deprecated`](http://hl7.org/fhir/concept-properties#status) plus
`deprecationDate` is the conventional pair, that deprecated concepts
remain active, and pointed at
[`v3-hl7ApprovalStatus`](https://terminology.hl7.org/CodeSystem-v3-hl7ApprovalStatus.html)
as a worked example. John Moehrke contributed that
`concept-properties#inactive` does not exist in R4 (so R4-targeting IGs
can't use it), and warned — based on his experience fixing IHE FormatCodes
— that once you add the `status` property to one concept the publisher
expects it on all concepts in the code system or it silently ignores the
ones missing it. The thread reached a working consensus on the
*mechanism* (use `status`/`deprecationDate`, declare them in
`CodeSystem.property`) but flagged ergonomic issues (must tag every
concept; copyright handling for codes that have moved between SDOs). Marc
Duteau confirmed at the end that other v2 content (e.g., `v2-0066`)
suffers from the same root cause — using
`utg-concept-properties#v2-table-deprecated` instead of the standard
`status` property — which is exactly the fix FHIR-32232 is asking TSMG to
authorise.

Participants: Brian Postlethwaite, John Moehrke, Jessica Bota, Marc
Duteau, Grahame Grieve (mentioned).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Direct THO/UTG (via a follow-up UP ticket — Marc Duteau already offered
to raise one) to migrate the v2 code system
[`v2-0131` (Patient Contact Relationship)](https://terminology.hl7.org/7.1.0/en/CodeSystem-v2-0131.html)
from the custom THO property
`http://terminology.hl7.org/CodeSystem/utg-concept-properties#v2-table-deprecated`
to the standard FHIR concept properties
[`http://hl7.org/fhir/concept-properties#status`](http://hl7.org/fhir/concept-properties#status)
(value `deprecated`) and
[`http://hl7.org/fhir/concept-properties#deprecationDate`](http://hl7.org/fhir/concept-properties#deprecationDate)
for the four codes implicated in the comparison (and any siblings in the
table currently flagged via the custom property). Specifically:

1. In `v2-0131`, declare the two standard properties in the
   `CodeSystem.property` collection (with their canonical URIs and
   types), populate `status=deprecated` and a best-effort
   `deprecationDate` on each formerly-`v2-table-deprecated` concept, and
   add `status=active` to every other concept (because the UTG/THO
   publisher silently drops concepts missing the property once it is
   declared — see John Moehrke's warning in the UTG Zulip thread).
2. Retire the custom `utg-concept-properties#v2-table-deprecated`
   property usage on `v2-0131` (and, as a follow-on TSMG work item,
   sweep the rest of THO's v2-* code systems that use the same custom
   property — confirmed by Marc Duteau to affect e.g. `v2-0066`).
3. Re-publish `v2-0131`; the existing `PatientContactRelationship`
   value set will then expand with the deprecated codes either
   suppressed (if the binding/expansion parameters require active-only)
   or visibly flagged, restoring parity with R4.

This disposition formally records TSMG's approval so the UP/UTG change
can move forward, which is exactly what FHIR-I asked for in the most
recent comment.

#### Justification

The standard FHIR concept-property model is the single supported
mechanism for terminology-server expansion logic to recognise
deprecation; the custom `v2-table-deprecated` property cannot be honoured
by general-purpose expanders, as Grahame Grieve and Lloyd McKenzie both
confirmed in comments. Aligning v2-0131 with
`concept-properties#status` is consistent with how other HL7 code
systems (e.g., `v3-hl7ApprovalStatus`) already mark deprecation, removes
the R4↔R5 expansion divergence the reporter flagged, and unblocks an
existing pattern Marc Duteau is willing to implement once TSMG approves.
The known gotcha (must populate `status` on *all* concepts) is
addressable in the same UP ticket and is a one-time cost.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) altering the THO source, change the
`PatientContactRelationship` value set definition to filter out
concepts whose property `v2-table-deprecated = true` (using a `filter`
inside `compose.include`), and have the value-set rendering layer in the
publisher emit a deprecation badge for any concept carrying the custom
property. Concretely:

1. Update `ValueSet/PatientContactRelationship` so its include filters
   the `v2-0131` code system on
   `property = utg-concept-properties#v2-table-deprecated, op = !=,
   value = true`, restoring the R4 behaviour for this binding without
   touching `v2-0131` itself.
2. Add publisher/IG-template support so that any value set whose
   underlying code system carries the custom property renders the
   concept with a "deprecated" indicator equivalent to the one used for
   `status=deprecated`.

#### Justification

This avoids a wide-blast change across many v2 code systems and any
downstream THO consumers that may already rely on the custom property
shape. It also restores R4-equivalent expansion behaviour for the
specific binding the reporter complained about with a narrowly-scoped
edit. The trade-off is significant: it preserves a non-standard property
that no third-party terminology server will understand, leaves the
underlying THO data quality issue in place (Marc Duteau has already
acknowledged it affects other v2 tables), and creates a long-term
maintenance burden in the publisher to keep the special-case rendering
in step with `concept-properties#status` semantics. It is best treated
as a stopgap if Disposition A cannot land in time for R6.

---

### Disposition C: Decline

#### Proposal

Decline the request as a spec change and close the ticket: the value set
`PatientContactRelationship` is correct in including all v2-0131 codes
that exist; the visibility/suppression of deprecated codes is a
*publishing/data-quality* concern in THO, not a FHIR Core spec defect,
and is already tracked through normal UTG processes.

#### Justification

There is a defensible reading that the value set is doing exactly what
its definition asks (include the code system) and that the user-visible
problem is entirely in how `v2-0131` declares deprecation. However, this
disposition is weak because: (a) FHIR-I has explicitly asked TSMG for
approval rather than another bounce, (b) the duplicate FHIR-39206 shows
real implementer pain (Highest priority from a PA reviewer), and
(c) declining without authorising the THO fix leaves the underlying
problem in place across multiple v2 code systems.

---

### Recommendation

**Recommended disposition:** A

This ticket has been bounced between PA, FHIR-I, and TI for nearly five
years precisely because no one workgroup owned the THO-side fix. The
2026-04-13 FHIR-I comment is unambiguous that TSMG approval is the
remaining gate, and Marc Duteau has already volunteered to raise the UP
ticket to migrate `v2-0131` to the standard
`concept-properties#status` + `deprecationDate` properties as soon as
that approval is recorded. Disposition A is the only path that
permanently aligns HL7 v2 deprecation with the rest of the FHIR
ecosystem, generalises cleanly to the other v2 tables Marc has flagged
(e.g., `v2-0066`), and resolves the original R4↔R5 divergence the
reporter raised. The publisher-side gotcha that John Moehrke surfaced
(must populate `status` on every concept once declared) is real but
mechanical, and is best handled inside the same UP work rather than used
as a reason to stay on a non-standard property. Disposition B should be
held in reserve only if R6 timing forces a stopgap; Disposition C is not
defensible given the explicit ask from FHIR-I.
