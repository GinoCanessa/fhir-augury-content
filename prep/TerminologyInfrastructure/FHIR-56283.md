## Summary

The ticket asks that the registry of CodeSystem "defined properties"
currently published in the FHIR core specification at
`codesystem.html#defined-props` be moved out of the core spec and into
HL7 Terminology (THO), where it can evolve between FHIR releases. The
reporter (Lloyd McKenzie) argues that this content does not meet the
bar for living in core because (a) it is essentially a registry that
naturally accretes new entries and (b) holding it to the FHIR release
cadence is too restrictive. The request explicitly requires that the
canonical URL remain stable across the move so that existing
references and instances are not broken.

Context from cross-referenced work: the related ticket
[FHIR-56275](https://jira.hl7.org/browse/FHIR-56275) — "Add
artifact-authoritativeSource extension" — was raised at TSMG out of
the same discussion. It addresses the companion problem that, once a
CodeSystem's source of truth is THO rather than the FHIR spec, the
CodeSystem resource itself ought to expose where its real source of
truth lives (since the canonical URL alone implies "FHIR spec"). That
ticket has been disposed *Persuasive with Modification* to add an
`artifact-authoritativeSource` extension on `DomainResource`. In
addition, Bryn Rhodes filed UTG ticket UP-799 (referenced in
comments) to actually add the code system to THO; that ticket should
be linked to FHIR-56283 so that the spec-side change and the THO-side
change land together.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the contents of `codesystem.html#defined-props` (the registry of
defined CodeSystem properties — `parent`, `child`, `synonym`,
`comment`, `notSelectable`, `deprecated`, `effectiveDate`,
`itemWeight`, `status`, etc., together with their codes, types,
URIs and descriptions) out of the FHIR core specification and into
HL7 Terminology (THO) as a managed CodeSystem owned by Terminology
Infrastructure / TSMG. Coordinate the move with the THO-side ticket
**UP-799**. Specifically:

1. In THO, create the CodeSystem at the existing canonical URL used
   by the property `uri`s (i.e., the `http://hl7.org/fhir/concept-properties`
   namespace currently published from the core build). Keep every
   existing property URI byte-for-byte so that no existing
   `CodeSystem.property.uri` value in the wild breaks.
2. In FHIR core, replace the inline registry on
   `codesystem.html#defined-props` with (a) a short narrative
   explaining that the registry lives in THO, (b) a stable link to
   the THO-published CodeSystem, and (c) a non-normative table that
   is *generated from* the THO content rather than authored in
   core, so that the page continues to render the current list for
   readers but is no longer the source of truth.
3. Establish in the FHIR publication tooling that the
   `defined-props` page is rebuilt from the THO release, not from
   in-repo source, so that updates in THO surface in the next FHIR
   build without a core spec change.
4. Cross-link FHIR-56283 to UP-799 so the spec-side and THO-side
   changes ship together.

#### Justification

The page is a textbook registry: the codes have URIs, they are added
incrementally by the terminology community, and several of them
(`itemWeight`, `effectiveDate`) have already been added between
releases through ad-hoc spec-build mechanisms. Putting it in THO
matches how other terminology-owned registries are managed and lets
Terminology Infrastructure govern additions on the THO cadence
rather than the FHIR release cadence. The reporter's stability
constraint — "keep the URL the same for the move" — is achievable
because CodeSystem property URIs are already absolute canonicals
that don't depend on the FHIR core publishing path; they only need
to resolve to the same content. The companion work on
`artifact-authoritativeSource` (FHIR-56275) directly supports this
move by giving consumers of any future profiled / republished
CodeSystem a standard way to point back at the THO authoritative
copy.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the registry's *definitional* content (the property URIs and
their semantics) in FHIR core, but split out the *catalog* — the
list of currently-recognised properties, examples, and notes — into
THO. Concretely:

1. Retain in FHIR core only the framing rules: how
   `CodeSystem.property` is structured, the requirement that
   `property.uri` be a canonical URI, and a small set of
   "infrastructure" properties whose semantics are tightly bound
   to FHIR itself (e.g., `notSelectable`, `status`, `deprecated`,
   `parent`, `child`).
2. Move all "domain" properties (`itemWeight`, `effectiveDate`,
   `synonym`, `comment`, future additions, etc.) into a THO-owned
   CodeSystem of concept properties. Keep every URI stable.
3. On `codesystem.html#defined-props`, render two sections: one
   listing the small core-resident set, and one transcluded from
   THO listing the domain properties.
4. Coordinate with UP-799 only for the domain-property subset.

#### Justification

This is a less disruptive move that addresses the underlying
concern (registry growth between releases) while keeping the
properties most tightly coupled to CodeSystem semantics under the
same change-control discipline as CodeSystem itself. It also
reduces the risk that a THO-only change inadvertently changes the
behaviour of a property that FHIR-defined operations or invariants
depend on (e.g., `notSelectable`'s effect on expansion). The cost
is a slightly more complex publication pipeline and an ongoing
classification decision (core vs THO) for each new property.

---

### Disposition C: Decline

#### Proposal

Decline the move. Leave the defined-properties registry in
`codesystem.html#defined-props` as part of FHIR core. Address the
"evolve between releases" concern by allowing additions to that
page through the existing technical-correction / continuous-build
process rather than waiting for a numbered FHIR release, and use
the new `artifact-authoritativeSource` extension (FHIR-56275) to
expose THO provenance on individual `CodeSystem` resources where
that is appropriate.

#### Justification

`CodeSystem.property` is a normative element of a normative
resource, and the set of recognised property URIs has direct
semantic effects (e.g., expansion behaviour for `notSelectable`,
hierarchy semantics for `parent`/`child`). Moving the registry to
THO splits ownership of CodeSystem semantics across two
governance bodies and two repositories, which raises the risk of
drift and complicates the publication chain. If the only real
problem is cadence, that can be solved within the FHIR core
process without restructuring ownership.

---

### Recommendation

**Recommended disposition:** A

The properties registry behaves like a registry, not like
specification text: it grows over time, individual entries are
authored by terminology stakeholders, and several entries have
already been added between FHIR releases under pressure. THO is
the right home for that, both organisationally (Terminology
Infrastructure already owns this surface) and technically
(THO already manages stable canonical URIs and publishes on a
faster cadence than FHIR core). The reporter's URL-stability
constraint is satisfiable, and the companion work on
`artifact-authoritativeSource` in FHIR-56275 plus the THO-side
ticket UP-799 means the supporting machinery is already lined up.
Disposition B is a defensible compromise if FHIR-I is uneasy
about handing over properties whose semantics drive resource
behaviour (`notSelectable`, `parent`/`child`); the workgroup may
want to fall back to B if it cannot agree which properties are
"infrastructure". Declining (C) does not address the underlying
cadence problem and leaves a registry living in normative spec
text that does not warrant that status.
