## Summary

Brian Postlethwaite reports that the `issues` output parameter of the
terminology validation operations (`ValueSet/$validate-code` and
`CodeSystem/$validate-code`, which return an `issues` parameter typed as
`OperationOutcome`) contains issue entries whose path / expression values are
invalid or unclear. The specification does not state which Parameters
instance the FHIRPath expressions on `OperationOutcome.issue.expression` are
anchored to — the request's input `Parameters` resource, or the response's
output `Parameters` resource — and there is no documented convention for how
to report an issue when the input was supplied via the simple `code`/`system`
form rather than as a `coding` or `codeableConcept`. The ticket asks the
Terminology Infrastructure WG to (a) clarify the binding of
`OperationOutcome.issue.expression` paths in the validate-code response, (b)
specify how to report issues against the scalar `code` input form, and (c)
add one or two worked examples showing expected usage of the issue
`expression` property. No cross-referenced Jira tickets, Zulip threads, or
GitHub items were surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the documentation of the `issues` output parameter for both
`ValueSet/$validate-code` and `CodeSystem/$validate-code` (in the operation
definitions and the operation narrative pages) to:

1. **Anchor expressions to the input Parameters resource.** State explicitly
   that any `OperationOutcome.issue.expression` returned in the `issues`
   output parameter is a FHIRPath expression evaluated against the **input**
   `Parameters` resource the client sent (or, for GET form, against the
   logical input Parameters that would have been constructed from the query
   parameters). Also clarify the corresponding behaviour of
   `OperationOutcome.issue.location` (deprecated) so implementations are not
   left to guess.
2. **Define the scalar input form.** For input supplied via the
   `code`/`system`/`version`/`display` parameters (no `coding` /
   `codeableConcept`), specify that issues should use expressions such as
   `Parameters.parameter.where(name='code').value`,
   `Parameters.parameter.where(name='display').value`, etc. For
   `coding`/`codeableConcept` input, expressions should descend into the
   parameter value, e.g.
   `Parameters.parameter.where(name='codeableConcept').value.coding[0].display`.
3. **Add worked examples.** Add two worked examples to the operation page —
   one for the scalar `code`+`system`+`display` form (e.g., a display
   mismatch reported against
   `Parameters.parameter.where(name='display').value`) and one for the
   `codeableConcept` form (e.g., a member-not-in-VS issue reported against
   `Parameters.parameter.where(name='codeableConcept').value.coding[1]`) —
   each showing the full request Parameters and the resulting `issues`
   output Parameter with populated `expression`.

#### Justification

This is a documentation and conformance clarification, not a structural
change: the operations already return `issues` typed as `OperationOutcome`,
and `OperationOutcome.issue.expression` is already the recommended
machine-readable locator (FHIRPath) per R5+ guidance that deprecates
`location`. Anchoring expressions to the input Parameters is the only
interpretation that lets the client correlate issues back to data it
controls — the output Parameters is server-authored and clients have no
stable structure to point at. The `code`-form question is real because the
Parameters input model has no nested element to address; naming the parent
parameter is the natural FHIRPath answer and keeps every input shape
addressable. Worked examples are explicitly requested by the reporter and
align with how other operations (e.g., `$validate`) document
expression-based issue reporting.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than anchoring `OperationOutcome.issue.expression` to the input
Parameters, define a small **virtual addressing model** for validate-code
issues and document it on the operation page. Concretely:

- Reserve a short set of well-known logical names —
  `code`, `system`, `version`, `display`, `coding`, `codeableConcept`,
  `valueSet`, `url` — and define that
  `OperationOutcome.issue.expression` is a FHIRPath expression rooted at a
  virtual node whose children are these logical names regardless of whether
  the client sent the scalar form, a `coding`, or a `codeableConcept` (e.g.,
  `code`, `coding[0].display`, `codeableConcept.coding[1].system`).
- Optionally also populate `OperationOutcome.issue.details.coding` with a
  WG-defined CodeSystem of issue subjects (e.g., `display-mismatch`,
  `code-not-in-valueset`, `inactive-code`) so clients can react without
  parsing FHIRPath at all.
- Add the same two worked examples requested in Disposition A, but using the
  virtual-root expressions.

#### Justification

The "expression rooted at the input Parameters" answer in Disposition A is
correct but exposes the wire shape of the request to clients that may have
constructed the call from a friendly API — they then have to translate
`Parameters.parameter.where(name='codeableConcept').value.coding[0]` back to
their own model. A virtual addressing model gives clients a stable,
shape-independent vocabulary that is identical for GET and POST callers and
identical regardless of which input form was used; it also gives the WG a
natural place to standardise issue codes alongside the locators. The
trade-off is more documentation surface, the introduction of a non-standard
FHIRPath root (which must be clearly flagged), and the loss of the
"expression evaluates against a real resource" property that
`OperationOutcome.issue.expression` otherwise enjoys.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by existing mechanisms**: the
base `OperationOutcome` definition documents `issue.expression` as a
FHIRPath locator, and by long-standing convention across FHIR operations
that take a `Parameters` input the expression is evaluated against that
input Parameters. No change to the validate-code operation pages is
required; the reporter's questions can be answered on the Zulip
`#terminology` stream.

#### Justification

`OperationOutcome.issue.expression` is generic; the Terminology
Infrastructure WG has historically chosen not to re-document base behaviour
on every operation page to avoid drift. Adding examples and bespoke
guidance to validate-code creates a precedent that every operation page
needs the same treatment, and risks contradicting a future cross-cutting
clarification at the OperationOutcome level. The `code`-form question is
arguably answered by FHIRPath itself
(`Parameters.parameter.where(name='code').value`).

---

### Recommendation

**Recommended disposition:** A

Accept as requested. The reporter is asking for clarification of behaviour
that is genuinely under-specified on the validate-code operation pages
today — there is no normative statement that
`OperationOutcome.issue.expression` is rooted at the input Parameters, and
the scalar `code` input form has no documented addressing convention at
all. Disposition C overstates how clearly the base `OperationOutcome`
definition resolves these questions for a Parameters-based operation, and
leaving this to tribal knowledge on Zulip is exactly what produced the
ticket. Disposition B is attractive in the abstract but introduces a
non-standard FHIRPath root that other operations would not share, which is
a larger conformance change than the ticket warrants and one that should
not be made unilaterally by Terminology Infrastructure. Disposition A is
the smallest change that fully answers all three of the reporter's
questions (anchoring, scalar form, examples), keeps validate-code aligned
with how `$validate` and other Parameters-based operations behave, and
gives implementers concrete examples to test against. If the WG later
wants to standardise issue codes (the Disposition B sidecar idea), that
can be done as a follow-on ticket without disturbing the addressing
clarification proposed here.
