## Summary

The reporter raises three editorial/normative gaps in the newly added §4.8.16
"Implicit Code Systems" of the FHIR Core CodeSystem page. (1) The section
tells readers how to *create* an implicit code system but never *defines*
the term — Webster-style definitions are offered as a starting point.
(2) The section gives no guidance on how a consumer (human or system) can
recognise that a given `CodeSystem` resource (or a URL pointing to one) is
acting as an implicit code system, i.e., there is no described marker,
property, or detection rule. (3) The illustrative example "(e.g.,
subsetting using ValueSets)" links to the `ValueSet` resource page, but
that page does not actually describe value-set–based subsetting, leaving
the example dangling.

The cross-reference index returned no other Jira, Zulip, or GitHub items
that explicitly mention FHIR-54109. However, Zulip search and the
contemporaneous chat history show the underlying topic is actively under
discussion in `terminology > Implicit CodeSystems`, where Grahame Grieve
acknowledges that the section "mashed two different things together
without really explaining how they were different — implicit code systems
for single StructureDefinitions and Questionnaires, and implicit code
system for systems containing Medications," and proposes "we should only
bring back StructureDefinition and Questionnaire." Related ballot work
referenced in that thread includes [FHIR-50704](https://jira.hl7.org/browse/FHIR-50704),
[FHIR-40432](https://jira.hl7.org/browse/FHIR-40432) and
[FHIR-39265](https://jira.hl7.org/browse/FHIR-39265). FHIR-54109 is the
narrower editorial counterpart to that broader scoping debate.

## Details

**Description:**

#1

Great detail in this newly added section, however it does not include a
definition of implicit code system - it should. The text describes how to
create an implicit code system, but does not define.

Definitions of implicit from Websters - could possibly be incorporated
into a definition of implicit code system.

1. Capable of being understood from something else though not clearly or directly stated
2. Implied

#2

If a system or person is evaluating a CodeSystem resource, how would
it/they know they are looking at an implicit code system?

#3

The example provided, (e.g., subsetting using ValueSets) provides a link
to the ValueSet resource. Subsetting is not described on that page.
Suggest providing another example, or adding information to ValueSet
about subsetting using ValueSets.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit `source/codesystem/codesystem.html` (§4.8.16 "Implicit Code
Systems") in the FHIR Core spec to address all three sub-points:

1. **Add a definition paragraph** at the top of the section, immediately
   under the heading. Wording along the lines of: *"An **implicit code
   system** is a set of codes whose membership and meaning are derived
   from another FHIR resource (e.g., a `StructureDefinition` or
   `Questionnaire`) rather than being declared by an instance of the
   `CodeSystem` resource. The codes are *implied* by the structure of
   the source resource and need not be enumerated separately. The
   canonical URL of the source resource is reused as the
   `system` for those codes."*
2. **Add an "Identifying an Implicit Code System" subsection** that
   states the detection rules: the `system` URL resolves to a
   non-`CodeSystem` resource; if a `CodeSystem` resource exists at the
   same URL it is generated, and `CodeSystem.content` is set to
   `supplement`/`fragment`/`not-present` as appropriate; the
   `CodeSystem` (when present) carries a marker (e.g., a property such
   as `http://hl7.org/fhir/CodeSystem/implicit` or a documented
   convention) that consumers can test.
3. **Replace the dangling "subsetting using ValueSets" example** with
   either (a) a worked example that uses `ValueSet.compose.include`
   referencing the implicit code system URL, or (b) leave the brief
   mention but add a forward link to a new "Subsetting" subsection on
   `valueset.html` that describes value-set–based subsetting in concrete
   terms.

#### Justification

All three points are editorial/normative-clarity asks against text the
working group itself authored, and they are consistent with FHIR's
documentation principle that any defined term used normatively must be
defined where it is first introduced. The Zulip thread shows that even
the section's author (Grahame) considers the current text under-explained
and conflated; an explicit definition + identification rules would
directly address that. The ValueSet cross-link fix is a low-risk doc
patch.

---

### Disposition B: Alternative Approach

#### Proposal

Combine FHIR-54109 with the broader scoping discussion in
[FHIR-50704](https://jira.hl7.org/browse/FHIR-50704) and the prior
task-force tickets ([FHIR-40432](https://jira.hl7.org/browse/FHIR-40432),
[FHIR-39265](https://jira.hl7.org/browse/FHIR-39265)), and rewrite §4.8.16
end-to-end rather than patching it. Specifically:

1. **Narrow the scope** of "implicit code system" to the two cases
   Grahame proposes keeping — `StructureDefinition` and `Questionnaire`
   — and explicitly call out (and exclude) the Medication-style "system
   of systems" case from this section, deferring it to a separate
   section or future work.
2. **Add a new top-level subsection** on `valueset.html` covering
   "Subsetting Using ValueSets" so the §4.8.16 example actually has a
   target.
3. Carry forward Carmela's three sub-points (definition, identification,
   example) as a checklist that the rewrite must satisfy, but do them as
   part of the larger restructuring rather than as standalone edits.

#### Justification

The Zulip thread makes clear the section has a deeper scoping problem
than just the three editorial gaps in FHIR-54109. Patching the text in
isolation (Disposition A) risks codifying a definition for a concept
whose scope the working group is still debating, and may need to be
re-litigated when FHIR-50704 is dispositioned. A coordinated rewrite
avoids two rounds of churn on the same section, and aligns the editorial
fix with the substantive scoping decision. The trade-off is more work
and a longer cycle time before the reporter's concrete asks land.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54109 as **already addressed by existing mechanisms** —
specifically, declare that §4.8.16 sufficiently describes implicit code
systems in the surrounding paragraphs, that the meaning of "implicit"
follows ordinary English usage, and that identification is implicit in
the URL-resolution rules already documented elsewhere on
`codesystem.html`. No spec change.

#### Justification

This is the weakest of the three positions and is included for
completeness. It is defensible only if the working group judges that
adding a formal definition and detection rules would over-specify a
deliberately flexible mechanism. The Zulip thread strongly contradicts
this position — the section's author concedes it is under-explained —
so this disposition would likely be unpopular.

---

### Recommendation

**Recommended disposition:** B

The Zulip discussion is unambiguous that §4.8.16 has a scoping problem
the workgroup has not yet resolved (Medication-style aggregate code
systems vs. per-StructureDefinition/Questionnaire implicit systems), and
that a parallel ballot comment ([FHIR-50704](https://jira.hl7.org/browse/FHIR-50704))
is being weighed against the implicit-CodeSystem framing. Accepting
FHIR-54109 as written (Disposition A) lands the reporter's three asks
quickly but bakes a definition into a section the working group is
likely to restructure anyway, creating churn. Declining (Disposition C)
ignores legitimate editorial defects acknowledged by the section's own
author. Disposition B uses FHIR-54109 as the editorial backbone of a
single coordinated rewrite that also closes out FHIR-50704 and honours
the prior task-force work captured on FHIR-40432 / FHIR-39265 — fewer
spec churns, one consistent story, and the reporter's three concrete
asks become acceptance criteria of the rewrite rather than competing
edits. The cost is calendar time, which is acceptable given the priority
is "Highest" but the resolution is still "Unresolved" with no assignee.
