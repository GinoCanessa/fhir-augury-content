## Summary

The reporter is asking the Terminology Infrastructure WG to clarify the
terminology binding situation for `CodeSystem.property.code` (and the
parallel `CodeSystem.concept.property.code`) in FHIR R6 ballot 4. In the
element table the definition reads "Reference to `CodeSystem.property.code`
or a FHIR defined concept-property", but no binding is declared and no
value set is linked. The detailed definition page links to the FHIR
defined concept-properties registry, yet that registry does not make it
obvious what the literal value of `property.code` should be in
implementations (e.g. `status` vs. the full URI
`http://hl7.org/fhir/concept-properties#status`). The summary
"terminology bindings" table on the CodeSystem page lists
`CodeSystem.property.type` (bound to `concept-property-type`) but omits
`CodeSystem.property.code`. The reporter asks whether `property.code`
should carry a required or at least extensible binding to the FHIR
defined concept-properties value set, or whether each CodeSystem author
is free to invent their own codes here. There are no cross-referenced
Jira tickets — only cross-references to the FHIR artifacts themselves
(`CodeSystem.property.code`, `CodeSystem.concept.property.code`,
`CodeSystem.property.type`). The same question and adjacent design
problems have surfaced repeatedly in chat.fhir.org going back to 2020,
so the recommendation here should be informed by that prior
conversation.

## Related Zulip Discussions

### [terminology > codesystem and concept status](https://chat.fhir.org/#narrow/stream/terminology/topic/codesystem%20and%20concept%20status)

A 2020 thread that asks the same question almost verbatim ("should
codesystem.property.code have a binding?"). Jose Costa Teixeira
proposed an extensible binding to the FHIR-defined concept-properties so
authors could pick from `status`, `parent`, `child`, `retirementDate`,
`deprecationDate`, `selectable`, etc. and still extend with their own.
Forest and Jim Steel pointed Jose at the existing CodeSystem.property
mechanism and the `status` documentation. Grahame Grieve answered the
binding question directly: **"no. but you should consider using a code
carefully."** He clarified that `property.code` is an internal
reference within the code system that does not have meaning outside it;
the optional `property.uri` is what gives a `property.code` non-arbitrary
meaning by tying it to a formally-defined property such as the HL7
`status` property. No consensus was reached that a binding should be
added; the resolution was a clarification of the existing model, not a
spec change.

Participants: Jose Costa Teixeira, Forest, Jim Steel, Michael Lawley,
Grahame Grieve.

### [terminology > Constraining CodeSystem.property.type](https://chat.fhir.org/#narrow/stream/terminology/topic/Constraining%20CodeSystem.property.type)

A January 2025 thread in which Russell Ott explicitly asks for the same
binding the current ticket is requesting, motivated by wanting LOINC
fragments / supplements to use a consistent set of property codes
(matching the THO LOINC properties guidance) without slicing 80+
properties in a profile. Grahame Grieve pushed back: he does not see
`property.code` as something that should be bindable across code
systems, and proposed instead that the *validator* should enforce that
"property definitions in fragments and supplements cannot redefine
properties" of a code system whose authoritative representation is
known. The thread did not reach a definitive disposition but moved
toward a validator-rule plus a normative sentence in the spec rather
than a value-set binding on `property.code`.

Participants: Russell Ott, Grahame Grieve.

### [terminology > Filter for "only leaves" in hierarchical CS](https://chat.fhir.org/#narrow/stream/terminology/topic/Filter%20for%20%22only%20leaves%22%20in%20hierarchical%20CS)

In a 2020 side-discussion Michael Lawley wondered whether there should
be "some kind of (extensible) binding for `CodeSystem.property.code`" to
help document property semantics across code systems — i.e. an early
voice on the same side as the current reporter, suggesting at minimum
an extensible binding rather than the no-binding status quo.

Participants: Michael Lawley (and the broader filter-discussion group).

### [terminology > CodeSystem.concept effective | expiration](https://chat.fhir.org/#narrow/stream/terminology/topic/CodeSystem.concept%20effective%20%7C%20expiration)

In 2021 Michael Lawley summarised the core difficulty: "one challenge
with a binding here is that the `CodeSystem.property.code` codes are
effectively relative" — the same `code` token can mean different things
in different code systems, so a system-wide value-set binding is hard
to justify even if some property codes (`status`, `parent`, etc.) are
universally re-used.

Participants: Michael Lawley (and the broader effective/expiration
discussion).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an **extensible** binding on both `CodeSystem.property.code` and
`CodeSystem.concept.property.code` to a new value set
`http://hl7.org/fhir/ValueSet/concept-properties` that enumerates the
codes from the FHIR defined concept-properties registry
(`http://hl7.org/fhir/concept-properties` — `status`, `parent`,
`child`, `notSelectable`, `deprecationDate`, `synonym`, etc.). At the
same time:

1. Add a row for `CodeSystem.property.code` (and the parallel
   `concept.property.code`) to the "Terminology Bindings" summary table
   on `codesystem.html#tx`, pointing at the new value set.
2. Add a sentence next to the binding in `codesystem-definitions.html`
   clarifying that the literal value of `property.code` is the short
   token (`status`), and that the corresponding registry URI
   (`http://hl7.org/fhir/concept-properties#status`) is conveyed via
   `CodeSystem.property.uri`, not duplicated into `property.code`.
3. Update the "Defined Concept Properties" prose on
   `codesystem.html#defined-props` to make explicit which column is the
   value to put in `property.code`.

#### Justification

This directly answers the reporter's complaint that the spec is
ambiguous about whether implementers may invent property codes freely
or are expected to reuse the registry. An *extensible* binding (rather
than required) preserves the long-standing position that
`property.code` is an internal-to-the-CodeSystem identifier — authors
can still coin code-system-specific property codes like `loinc-class`
that do not appear in the registry — while also nudging authors toward
the registered codes when one applies, and giving validators something
to warn on. The accompanying tx-summary-table fix is a pure
documentation defect: there is no defensible reason for
`CodeSystem.property.type` to appear there and `CodeSystem.property.code`
to be omitted when both have community-relevant terminology semantics
under this proposal. Michael Lawley's 2020 suggestion of an extensible
binding and Russell Ott's 2025 LOINC use case both line up with this
disposition.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a binding to `CodeSystem.property.code`. Instead, treat
this ticket as a documentation-and-tooling clarification:

1. Add a row to the `codesystem.html#tx` terminology bindings table for
   `CodeSystem.property.code` whose binding column reads
   *"No binding — internal CodeSystem reference; see
   `CodeSystem.property.uri`"*, so the table no longer appears to
   silently omit the element.
2. Expand `codesystem.html#defined-props` to state explicitly that
   `property.code` is a free-form, code-system-local identifier; that
   reuse of registry tokens (`status`, `parent`, `child`, …) is
   strongly recommended where the meaning matches; and that the
   *binding to the registry is conveyed via `property.uri`*, which
   should be set to the registry URI (e.g.
   `http://hl7.org/fhir/concept-properties#status`) when the
   property is one of the FHIR-defined concept-properties.
3. Add the explicit invariant Grahame proposed in the 2025 thread:
   "Property definitions in CodeSystem fragments and supplements
   cannot redefine properties already declared by the authoritative
   representation of the same code system."  Encode this as a
   validator rule rather than as a profile/binding constraint.
4. Optionally clarify in the same prose that `property.code` values
   used inside `concept.property.code` MUST exist in the enclosing
   `CodeSystem.property` collection — restating the existing
   structural rule so the question "does each system invent its own?"
   has an unambiguous answer ("yes, but they must be declared first").

#### Justification

This matches the position consistently taken by the design owner of
the structure (Grahame Grieve) in both the 2020 and 2025 chat
threads: `property.code` is intentionally an internal token whose
*external* meaning is carried by `property.uri`, and the registry is a
recommendation rather than a vocabulary. A binding — even an
extensible one — risks making implementers think the registry tokens
are reserved or owned by HL7 in a way that constrains code-system
authors who legitimately need code-system-local properties (e.g.
`loinc-class`, `snomed-moduleId`). The fix to the tx summary table
and to the defined-properties prose still resolves the reporter's core
complaint (the spec is currently ambiguous), but does so without
imposing a value set that the design intentionally avoids. Ott's
LOINC use case is addressed by the validator rule plus the explicit
"fragments cannot redefine properties" invariant rather than by
binding `property.code`.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive — No Change**: the existing model is
correct as-designed. Respond with a comment that:

- `CodeSystem.property.code` is intentionally not bound because it is
  a code-system-local internal reference; binding it would
  misrepresent the design.
- The FHIR-defined concept-properties registry is a *registry of
  recommended URIs*, not a value set; the URI is carried by
  `CodeSystem.property.uri`, and the literal value of `property.code`
  is whatever local token the code system author chooses
  (conventionally the short name, e.g. `status`).
- Code system authors are free to invent their own `property.code`
  values; they are encouraged to align with the registry when the
  semantics match.

Make no change to the spec, the binding table, or the defined-properties
prose.

#### Justification

Defensible on the grounds that the design has been deliberate and
publicly defended by the workgroup (Grahame, 2020 and 2025) and that
the registry-vs-value-set distinction is the intended mechanism. A
"clarification" ticket that results in no change is appropriate when
the right answer is "the current behaviour is correct" and the gap is
purely the questioner's understanding.

The weakness of this disposition is that the reporter has identified
two genuine spec defects independent of the binding question — the
tx-summary table omission and the lack of guidance on what literal
goes into `property.code` — both of which are real and were also
raised five years ago without being fixed. Declining outright leaves
those defects in place to be re-asked again.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right balance. It honours the consistent
TI-WG / design-owner position that `CodeSystem.property.code` is an
internal reference that should not be bound to a value set (Grahame's
explicit "no" in 2020, restated in 2025), and it directly fixes the
two concrete defects the reporter found: the missing row in the
tx-summary table and the ambiguity about what literal token goes into
`property.code`. Disposition A would be substantively easier to
defend to the reporter — it gives them what they asked for — but it
re-opens a debate the workgroup has already settled twice and risks
implementer confusion about whether code-system-local properties
(e.g. `loinc-class`, `snomed-moduleId`) are now non-conformant.
Disposition C resolves the principle but leaves the documentation
defects in place, which guarantees a third repeat of this thread in
2027. Disposition B addresses the principle, the documentation, and
(by way of the proposed "fragments/supplements cannot redefine
properties" invariant) Russell Ott's adjacent LOINC use case from the
2025 Zulip thread, which is the strongest implementation-side
motivation behind the current question.
