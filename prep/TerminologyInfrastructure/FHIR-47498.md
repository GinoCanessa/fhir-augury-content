## Summary

This ticket reports that 14 CodeSystems and 14 ValueSets are defined in
**both** the FHIR Extensions Pack and HL7 Terminology (THO), based on a
duplicate-resource-id analysis. An attached spreadsheet enumerates the
28 duplicates and links each occurrence in the two specifications. The
reporter is asking Terminology Infrastructure (in coordination with
FMG) to decide which copy is canonical for each artefact, remove or
deprecate the redundant copy, and define a policy that prevents future
duplication between the Extensions Pack and THO.

The work has since been folded into a successor / consolidation ticket,
[FHIR-55033](https://jira.hl7.org/browse/FHIR-55033) — *"Address
duplicate terminology artefacts (e.g. value sets and code systems)
occurring between the FHIR core specification, the FHIR Extensions
Pack, and HL7 Terminology (THO)"*. FHIR-55033 broadens the scope to
include the FHIR core specification, links to the same Reuben/Jessica
tracking spreadsheet, and is being driven through R6 QA review (Nov–Dec
2025) with anything not closed during QA being addressed as negative
ballot votes. A Sept-2025 WGM motion (Grahame/Lloyd, 19-0-1) tabled
the resolution to a subgroup; Lloyd's stated principle for that
subgroup is *"things bound as code should live in the Extensions Pack;
things not bound as code should live in THO,"* mirroring how the core
spec is split. The most recent comment (carmela_couderc, 2026-04-13)
proposes that this ticket be closed as **Persuasive** because the
remaining work is now tracked in FHIR-55033.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Resolve the ticket as **Persuasive** and close it in favour of
[FHIR-55033](https://jira.hl7.org/browse/FHIR-55033), which is the
active, broader-scope tracking ticket for duplicate
CodeSystem / ValueSet artefacts across FHIR Core, the FHIR Extensions
Pack, and THO. Add a closing comment that:

1. Names the canonical home for each of the 28 duplicates per Lloyd's
   WGM-Sep-2025 rule (*"things bound as code → Extensions Pack;
   things not bound as code → THO"*), using the Reuben/Jess Google
   spreadsheet as the row-by-row decision record.
2. For each duplicate, deprecates (does **not** retract) the
   non-canonical copy in its source spec and adds a `replaced-by` /
   `predecessor` reference to the canonical copy, so existing
   implementers' canonical URLs continue to resolve.
3. Confirms in the comment that any items that cannot be cleared
   during the R6 QA window will be carried forward via FHIR-55033 (and
   tracked through ballot reconciliation if needed), exactly as
   jsnell's 2025-11-24 comment commits to.

#### Justification

This is what every recent commenter — Reuben, Jess, Carmela — is
already converging on. The Persuasive resolution and the explicit
"close as duplicate of FHIR-55033" recommendation in carmela_couderc's
2026-04-13 comment make this the path of least friction and the one
that preserves the cleanup work Reuben and Jess have already done.
Lloyd's *bound-as-code → Extensions Pack / otherwise → THO* rule was
adopted by motion (19-0-1) and gives the subgroup an objective basis
for the per-artefact decisions, which is the key thing the original
Zulip thread was missing. Deprecating rather than retracting the
non-canonical copies addresses Grahame's earlier concern that each
case needed author-level investigation, by guaranteeing that historical
resolutions of either URL keep working while the canonical version is
re-pointed.

---

### Disposition B: Alternative Approach

#### Proposal

Keep FHIR-47498 **open** as the Extensions-Pack-scoped tracking ticket
and **do not** fold it into FHIR-55033. Instead:

1. Re-scope FHIR-47498 narrowly to the 28-artefact Extensions-Pack ↔
   THO list (its original scope), apply Lloyd's *bound-as-code →
   Extensions Pack; otherwise → THO* rule artefact-by-artefact, and
   close it only when the Extensions-Pack-side cleanup is verifiably
   complete in the R6 build.
2. Use FHIR-55033 strictly for the Core-vs-Extensions-Pack-vs-THO
   triangle that emerged after FHIR-47498 was filed.
3. Add a Terminology Expectations policy update (the FMG work jsnell
   refers to) as an explicit blocker on FHIR-47498 so the policy and
   the artefact moves land together.

#### Justification

FHIR-47498 has the clearest, narrowest scope (one duplicate-id list of
28 artefacts in two specs) and the best paper trail (WGM Sep 2024 +
WGM Sep 2025 motion + the Reuben/Jess spreadsheet). FHIR-55033 was
filed later, has lower priority (Medium vs. Highest here), is
specified against *FHIR Core* not *Extensions Pack*, and broadens the
scope to a three-way comparison. Folding the Extensions-Pack-only
work into a Core-scoped ticket risks losing both the existing TI
visibility and the FMG governance hook that the *fmg > Extensions
Pack* Zulip topic provides. Keeping this ticket alive gives TI a
concrete, closeable deliverable for R6 QA rather than an open-ended
cross-spec umbrella.

---

### Disposition C: Decline

#### Proposal

Decline the request as **Not Persuasive** on the grounds that
duplication between the Extensions Pack and THO is by-design during
the R6 ballot transition and that the Terminology Expectations policy
work (in flight at FMG) is the appropriate vehicle, not per-artefact
ticket churn against a moving target. Close the ticket without
making the artefact-level moves.

#### Justification

The honest case for Decline is narrow: Grahame's *"it totally
depends — each case has to be investigated with the author"* answer
in the originating Zulip thread, plus jsnell's note that the
resolutions are *tightly coupled* with as-yet-undrafted FMG policy
updates, mean that hand-deciding 28 artefacts before that policy
lands is premature and could create a second round of moves. If the
WG believes the FMG Terminology Expectations rewrite will produce
different rules than Lloyd's interim *bound-as-code* heuristic,
deferring is defensible. This is the weakest of the three options
because the WGM Sep 2025 motion already chose to act, and the
Persuasive resolution on the ticket reflects that choice.

---

### Recommendation

**Recommended disposition:** A

This ticket has been overtaken by events. The reporter and assignee's
own delegate (jsnell), the WG co-chair commenter (carmela_couderc),
and the most recent comment all converge on closing FHIR-47498 in
favour of FHIR-55033, and the resolution field is already set to
*Persuasive*. The substantive cleanup work is happening on the
spreadsheet and through R6 QA review under FHIR-55033's umbrella, so
keeping FHIR-47498 open (Disposition B) creates duplicate process for
no incremental coverage — which is ironic given the ticket's subject.
Disposition C is hard to defend after the WGM Sep 2025 motion
(19-0-1) and the resolution field. Disposition A is therefore the
clean answer: close as Persuasive, point at FHIR-55033, and make sure
the closing comment captures the *bound-as-code → Extensions Pack;
otherwise → THO* rule plus the deprecate-don't-retract guidance so the
subgroup driving FHIR-55033 inherits a sharp, executable rule rather
than a freshly-empty inbox.
