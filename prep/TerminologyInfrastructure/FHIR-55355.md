## Summary

This ticket reports that `ConceptMap.group.element.target.relationship` is a
mandatory element with a required binding to the `ConceptMapRelationship`
value set, but that value set contains only "positive" relationship codes
(`related-to`, `equivalent`, `source-is-narrower-than-target`,
`source-is-broader-than-target`, and `not-related-to`). It does not include an
`unknown` or `other` "escape valve" concept. The reporter — Lloyd McKenzie,
via imported comment 319 — argues that this can make the containing element
impossible to populate in legitimate cases (for example, when an authoring
system knows that a target concept is the right one to map to but cannot
classify the relationship along the existing axis), and asks that an
`unknown`, `other`, or equivalent fallback code be added to the bound value
set. There are no other Jira tickets cross-referenced to this one in the
data source; no GitHub PRs/issues are linked.

## Details

**Description:**

The element ConceptMap.group.element.target.relationship is a mandatory
element with a required binding, but appears to have neither an 'unknown'
or 'other' concept, which can make the containing element impossible to
populate in some cases. Please add 'unknown', 'other', or some equivalent
escape valve concept to the value set.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an explicit "escape valve" concept to the `ConceptMapRelationship`
value set / underlying code system used by
`ConceptMap.group.element.target.relationship`, and keep the binding
`required`. Concretely:

- Define a new code (preferred: `unknown`; alternatively `other`, or both)
  in the FHIR-defined `ConceptMapRelationship` CodeSystem, with a
  definition along the lines of: *"The relationship between the source and
  target concept is known to exist, but its nature cannot be characterised
  using any of the other defined relationship codes."*
- Add that code to the bound `ConceptMapRelationship` ValueSet so it is
  selectable under the existing `required` binding.
- Update the element definition / comments on
  `ConceptMap.group.element.target.relationship` to call out when the
  fallback code should be used (e.g., authoring systems that have
  identified a mapping target but cannot classify the directional
  relationship), and to discourage routine use as a workaround for poor
  modelling.

#### Justification

The argument in the ticket is a standard FHIR design pattern: when an
element is **mandatory** *and* has a **required** binding, the bound value
set must be exhaustive enough that conformant authors are never forced
into an unrepresentable state. FHIR has used `unknown` / `other` /
`null-flavour-style` fallbacks in many other required value sets
(e.g., `AdministrativeGender` includes `unknown`/`other`,
`AllergyIntoleranceCriticality` and others use `unable-to-assess`). Adding
a single `unknown` (or `other`) code preserves the strength of the binding
and the validation guarantees it provides, while removing the
"impossible-to-populate" failure mode the reporter identifies. This is the
least disruptive option for downstream implementers: existing instances
remain valid, and the only change for tooling is recognising one new code.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) extending the value set, change the
binding from `required` to `extensible` on
`ConceptMap.group.element.target.relationship`. This lets the existing
`ConceptMapRelationship` codes remain the recommended set, but allows
implementers to use a code from another code system (including a project-
or jurisdiction-specific "unknown" / "other" / fine-grained relationship
code) when no listed concept fits. The element remains mandatory; only the
binding strength relaxes.

Optionally combine with a small extension of the value set to add a single
`unknown` code so that the typical fallback case has a normative answer
without forcing implementers to mint their own codes.

#### Justification

A `required` binding to a closed list of relationship semantics is a
strong commitment for an element whose semantics are still maturing — the
2020 Zulip thread already shows confusion about reading direction and
completeness of the value set. An `extensible` binding addresses both the
specific gap the ticket identifies *and* the broader risk that other
relationship semantics (e.g., partial overlap, contextual equivalence)
emerge later. The trade-off is a real loss of validation strength and
interoperability: receivers can no longer assume the relationship code is
drawn from a known, comparable set, which weakens ConceptMap's value as a
machine-actionable artefact. For that reason this disposition is best
paired with adding a documented fallback code rather than used on its own.

---

### Disposition C: Decline

#### Proposal

Do not modify the value set or the binding. Resolve as **Persuasive with
Mod / Not Persuasive**, on the grounds that the existing
`ConceptMapRelationship` value set already provides a fallback in
`related-to` (the most generic positive relationship) and `not-related-to`
(the explicit negative), and that requiring authors to commit to one of
those is a feature, not a bug: it forces concept maps to express a
defensible relationship rather than masking unknowns behind a
null-flavour-style code that downstream consumers cannot act on.

#### Justification

`related-to` is already the deliberately broad fallback in this value set:
the FHIR definition is essentially "the concepts are related; the nature
of the relationship is not specified." Adding `unknown` or `other`
duplicates that semantics and risks fragmenting practice (some authors
will pick `related-to`, others `unknown`, for indistinguishable cases),
which makes ConceptMap content harder to compare across systems. For the
narrower case of "we can't even say whether they are related," the
appropriate modelling answer is usually to omit the mapping target
entirely rather than persist a `relationship = unknown` row.

---

### Recommendation

**Recommended disposition:** A

The ticket identifies a real and well-known FHIR design rule violation —
mandatory element + required binding without an `unknown`/`other` escape
— and the reporter is the same person who has historically driven the
addition of fallback codes elsewhere in the spec (e.g., `unknown` in
gender and similar value sets). Disposition C overstates how much
`related-to` covers: `related-to` asserts that *some* directional
relationship exists, which is a stronger claim than "we know the target
concept is the right one but cannot classify the relationship." Disposition
B (relaxing the binding) is a much larger interoperability cost than the
problem warrants and would weaken ConceptMap's machine-actionability for
every implementer, not just the edge cases. Adding a single `unknown` code
(Disposition A) is a small, additive, backward-compatible change that
removes the "impossible to populate" failure mode while preserving the
required binding and the comparability guarantees that come with it. The
workgroup should confirm the preferred code (`unknown` vs `other` vs
both) and the exact definition wording, but the direction is clear.
