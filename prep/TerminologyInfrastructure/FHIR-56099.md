## Summary

This Change Request asks the Terminology Infrastructure WG to align the
`TerminologyCapabilities.codeSystem.version` backbone with the version
modelling now used on `CodeSystem` (and other canonical resources) in R5+.
Two related changes are proposed:

1. **Rename** `TerminologyCapabilities.codeSystem.version.code` to
   `TerminologyCapabilities.codeSystem.version.value`. The reporter argues
   that the supplied value is a free-form version identifier, not a code,
   so the element name `code` is misleading. The element's definition would
   also be updated from "Version identifier for this version" to
   "Code System business version" so it parallels the `CodeSystem.version`
   wording.
2. **Add** a new optional element `TerminologyCapabilities.codeSystem.version.algorithm`
   modelled after `CodeSystem.versionAlgorithm[x]` (see
   [build.fhir.org definition](https://build.fhir.org/codesystem-definitions.html#CodeSystem.versionAlgorithm_x_)),
   with a tightened description (the current "how to compare versions"
   wording is considered imprecise and should be revised).

Background discussion is recorded in the VOC Confluence page
[9-Mar-2022 — R5 FHIR Terminology Server Minimum Requirements Task Force Call](https://confluence.hl7.org/spaces/VOC/pages/94636347/9-Mar-2022+%E2%80%94+R5+FHIR+Terminology+Server+Minimum+Requirements+Task+Force+Call?focusedTaskId=6).
The 6-Apr-2026 WG call concluded that the `versionAlgorithm` addition is
acceptable, but more community input — particularly from terminology
service implementers — is needed before renaming the existing element,
because the rename is a breaking change to a normative-track structure.
A separate Zulip thread is to be opened to canvass that input. No
cross-referenced FHIR Jira tickets surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply both requested changes to the `TerminologyCapabilities` resource in
the FHIR Core spec:

1. Rename `TerminologyCapabilities.codeSystem.version.code` →
   `TerminologyCapabilities.codeSystem.version.value` (`0..1 string`,
   cardinality unchanged). Update the short and definition to "Code
   System business version" to mirror `CodeSystem.version`.
2. Add `TerminologyCapabilities.codeSystem.version.algorithm[x]`
   (`0..1`, types `string | Coding`) modelled exactly on
   `CodeSystem.versionAlgorithm[x]`, bound to the same
   `version-algorithm` value set with the same binding strength
   (`extensible`). Replace the inherited "how to compare versions"
   wording with an element-specific definition such as: "Indicates the
   mechanism used to compare versions of this code system, so that a
   client can correctly order versions advertised by the server."
3. Add a publication-notes / change-log entry flagging the rename as a
   breaking change, and produce a `StructureDefinition`-level rename
   mapping so generators can emit a deprecation note in IGs that snapshot
   the previous element name.

#### Justification

The reporter's core argument is correct: the value carried in
`codeSystem.version.code` is a free-form business version string, not a
`code` data type token, so the element name is misleading and is
inconsistent with `CodeSystem.version`. Adding `versionAlgorithm` at the
backbone level closes a real gap — a server may legitimately advertise
multiple supported versions of a code system that use different
ordering schemes (SemVer for one, calendar versioning for another), and
without a per-version algorithm a client cannot interpret them. Modelling
both elements after `CodeSystem` keeps the canonical resource family
internally consistent, which is a recurring design principle the WG has
applied across R5 (see the Freeze-Time thread above for prior
applications of the same pattern).

---

### Disposition B: Alternative Approach — Add `versionAlgorithm`, but resolve the rename without breaking the wire format

#### Proposal

Split the request into two independently-tracked changes:

1. **Accept the additive change immediately.** Add
   `TerminologyCapabilities.codeSystem.version.algorithm[x]` exactly as in
   Disposition A. This is non-breaking and matches the WG's 6-Apr-2026
   conclusion.
2. **Do not rename the existing element.** Instead, leave
   `TerminologyCapabilities.codeSystem.version.code` in place but:
   - update its **short** and **definition** to "Code System business
     version" (matching `CodeSystem.version`), and
   - add an explicit note in the element comments that the element name
     is retained for backward compatibility, with a forward pointer to
     `CodeSystem.version` for the canonical naming.
   Optionally, add a `replacedBy`-style mapping or a publication note
   recommending implementers treat the element as if named `value`. If
   the WG later decides the rename is essential, do it through the normal
   substantive-change ballot once the Zulip implementer canvass closes.

Also adopt mjlawley's open question: clarify in the new
`algorithm[x]` definition that the cardinality is `0..1` per advertised
version, and that a code system whose versions use multiple algorithms
should advertise each `version` entry separately (one per
`(version, algorithm)` pair). If that is judged insufficient, raise the
cardinality to `0..*` rather than introducing a new sub-structure.

#### Justification

Renaming an element on a Maturity-Level / normative-track resource
breaks every existing `CapabilityStatement`-style consumer that parses
`TerminologyCapabilities` JSON/XML by element name, including
auto-generated client SDKs and validators. The 6-Apr-2026 call
explicitly flagged this risk and deferred to a Zulip canvass; this
disposition respects that deferral while still delivering the agreed
additive improvement and the documentation clarification the reporter
actually needs (the *meaning* of the element, not just its name).
mjlawley's comment about multiple algorithms also has a clean, no-spec-
change resolution if it is addressed in the new element's narrative.

---

### Disposition C: Decline the rename (and treat `versionAlgorithm` separately)

#### Proposal

Decline the rename of
`TerminologyCapabilities.codeSystem.version.code`. State explicitly that
the element name will be retained because (a) the element is part of an
already-published structure and (b) the cosmetic improvement does not
justify a breaking change for terminology-server vendors. If the WG
wishes, accept only the `versionAlgorithm[x]` addition under a separate
disposition / ticket — the additive change is uncontroversial and
already approved per the 6-Apr-2026 minutes.

#### Justification

`TerminologyCapabilities` is consumed mainly by tooling (registries,
conformance dashboards, server probes) that almost universally binds
generated types to element names. A rename that yields no functional
benefit imposes real migration cost across the ecosystem: every
generated POJO/POCO/TypeScript model, every FHIRPath expression, every
JSON Schema, and every IG-published `CapabilityStatement` example would
need to change. The reporter's stated rationale — that "the version is
not a code" — is addressable purely through the element's *definition*,
which can be updated without renaming. Declining the rename also avoids
a precedent for other "the name is misleading" renames against
already-published structures.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the closest match to the WG's own 6-Apr-2026
conclusion: it accepts the uncontroversial, additive
`versionAlgorithm[x]` change immediately while explicitly deferring the
rename pending the planned Zulip implementer canvass. It still gives the
reporter the substantive outcome they asked for — a clearer definition
that aligns with `CodeSystem.version` and a way to advertise version
ordering — without a breaking rename that the WG itself flagged as
needing more input. It also folds in a concrete answer to mjlawley's
"multiple algorithms" question, which any version of the change will
need to address. If the subsequent Zulip canvass returns broad
implementer support for the rename, the WG can revisit the rename as a
follow-up ticket through the normal substantive-change path; if it does
not, the spec still gets the algorithm element and the improved
definition, which is the underlying need.
