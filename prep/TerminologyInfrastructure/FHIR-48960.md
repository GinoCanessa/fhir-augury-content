## Summary

Elliot Silver asks the Terminology Infrastructure WG to clarify, in the
CodeSystem and Terminology Service specifications, the rules and
expectations around **multiple fragments of the same code system** and
the **interaction between fragments and supplements**. Specifically,
he wants the spec to address: (a) whether a server may hold multiple
fragments of the same CodeSystem or whether fragments are expected to
be combined into a single synthesized CodeSystem; (b) whether fragments
are disjoint, may overlap, or may be subsets of one another; (c)
whether there is any way to indicate that a set of fragments together
represent the *entire* code system (so they can be used for negative
validation); (d) any rules about combining fragments across different
versions of a code system; and (e) whether a supplement must align with
a single fragment or may cover concepts spanning multiple fragments.
He further asks for guidance on what happens when fragments — or a
fragment plus a complete/example code system — disagree about the same
concept, and that the ValueSet resource be updated to reflect any
impact.

The closely related ticket [FHIR-48959](https://jira.hl7.org/browse/FHIR-48959)
(*Clarify ValueSet behavior with CodeSystem fragments*), filed by the
same reporter on the same day and also resolved "Persuasive with
Modification", focuses on the ValueSet expansion side — how
`ValueSet.compose` references fragments, what an expansion looks like
when only fragments are available, and how the consumer is informed
that a fragment was used. Together the two tickets cover the
definitional (this ticket) and the consumption (FHIR-48959) sides of
the same underlying problem.

The TI WG has converged on a "Persuasive with Modification" resolution
that intentionally does **not** prescribe server behavior for many of
the edge cases. Instead it (1) adds a SHOULD on
`CodeSystem.version` for fragments drawn from a versioned code system,
(2) introduces a new Terminology Service section "4.7.4 Fragments and
Supplements" with narrative rules (complete CS wins over fragment;
supplements MAY be used with fragments; supplements SHOULD assert the
version they apply to), (3) adds two new repeating markdown elements
**`TerminologyCapabilities.implementation.fragmentSupport`** and
**`TerminologyCapabilities.implementation.supplementSupport`** (each
required-when-supported via constraint) so servers self-document their
fragment/supplement handling, and (4) adds a Note to Balloters asking
implementers how they actually handle multiple fragments today. The
3rd sub-bullet of the original request (a way to assert that a set of
fragments constitutes the *whole* code system) is explicitly noted as
**not addressed** by the proposed resolution, and the 4th sub-bullet
(cross-version fragment combinations) is deferred to FHIR-48959's
Zulip discussion. Grahame Grieve has flagged that the proposed
FHIRPath warning constraint on `CodeSystem.version` cannot actually be
written in FHIRPath, which is an open issue against the resolution.

## Details

**Description:**

> Please update the discussion of CodeSystem fragments and supplements
> to clarify (or indicate that there are no requirements) around:
>
> - The relationship between multiple fragments of the same code system:
>   - whether a system is allowed to have multiple fragments of the same
>     code system, or whether an import facility (or similar) is expected
>     to synthesize a single combined code system;
>   - whether fragments are expected to be disjoint, may overlap, may be
>     subsets of each other;
>   - whether there is a way to indicate that a set of fragments
>     represent the entire code system (and therefore could be used to
>     validate that a code is not in the system);
>   - any expectations around fragments of different code system versions
>     (e.g., can a valueset expansion combine multiple fragments from
>     different versions), different versions of the same fragment, etc.
> - The relationship between fragments and supplements:
>   - whether supplements need to align with fragments or can cover
>     concepts from across multiple fragments (which may not be present
>     on the system).
>
> Also, clarify behavior when multiple fragments (or a fragment code
> system, and a complete or example code system) contain differing
> information on the same concept.
>
> Update the ValueSet resource where necessary to reflect the impact of
> any of this.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the TI WG's full 2026-03-31 resolution package essentially as
written, with one technical fix:

1. **CodeSystem resource (fragment section):** add a SHOULD
   conformance statement — "If a fragment is drawn from a versioned
   code system, the fragment SHOULD provide a value for
   `CodeSystem.version`." **Drop the FHIRPath warning constraint**
   (per Grahame Grieve's 2026-04-14 comment, the rule cannot be
   expressed in FHIRPath because the constraint cannot determine
   whether the source code system is versioned). Replace it with
   prose-only guidance, or with an `inv-` invariant only on the
   property the fragment exposes (e.g., when `content = fragment`
   and a `valueSet`/`url` points to a CS with versions present in
   the server's TS).
2. **Terminology Service spec:** insert a new §4.7.4 "Fragments and
   Supplements" carrying:
   - "If a complete code system and a fragment for the same code
     system are both present, the server SHALL use the complete code
     system. A fragment SHOULD only be used when it is the only
     representation of the code system available on the server."
   - "Supplements MAY be used with fragments. Supplements SHOULD
     declare the version of the code system they supplement when the
     code system is versioned."
3. **Terminology Service §7.4 (TerminologyCapabilities):** require
   servers that support fragments to populate
   `TerminologyCapabilities.implementation.fragmentSupport`, and
   servers that support supplements to populate
   `TerminologyCapabilities.implementation.supplementSupport`.
4. **TerminologyCapabilities resource:** add two new repeating
   markdown elements
   - `TerminologyCapabilities.implementation.fragmentSupport :
     markdown [0..*]`
   - `TerminologyCapabilities.implementation.supplementSupport :
     markdown [0..*]`
   each with a constraint that the element MUST be present and
   non-empty whenever the server supports fragments / supplements
   respectively. Element comments should give the example axes
   already enumerated in the resolution (multi-fragment handling,
   disjoint vs overlapping, conflict resolution; supplements of
   complete CS vs of fragments).
5. **CodeSystem (fragment section):** add a "Note to Balloters" box
   listing the behaviors deliberately left open and asking
   implementers for input, with a link to the new
   `TerminologyCapabilities.implementation.fragmentSupport`
   definition.
6. Carry forward the resolution's explicit non-handling of (a) the
   "set of fragments == whole code system" assertion and (b)
   cross-CS-version fragment combination (the latter is deferred to
   FHIR-48959).

#### Justification

This is the disposition that the WG actually proposed and that the
2026-04-20 Zulip thread is soliciting comment on; it reflects roughly
five months of triage and is directly traceable to the
August/September 2025 implementer discussion. Crucially, it follows
established TI practice of using TerminologyCapabilities to *describe*
implementation behavior rather than prescribing it in the core spec —
which matches Michael Lawley's position that today there is no defined
behavior and any one prescription would invalidate existing servers.
Dropping the FHIRPath constraint addresses Grahame Grieve's concrete
objection without weakening the SHOULD on `CodeSystem.version`, which
multiple Zulip participants identified as the single most painful
missing piece. The "Note to Balloters" preserves the option to
tighten the rules in a future cycle once implementer responses are in.

---

### Disposition B: Alternative Approach

#### Proposal

Apply only the **narrative and capability** changes now, and defer
the new TerminologyCapabilities elements to a separate change
request once implementer responses are in.

Concretely:

1. Add the §4.7.4 "Fragments and Supplements" section to the
   Terminology Service spec with the four narrative rules from
   Disposition A items 2–3 (complete CS wins, fragment-only fallback,
   supplements MAY accompany fragments, supplements SHOULD declare
   version).
2. Add the SHOULD on `CodeSystem.version` for fragments of versioned
   code systems, **without** a FHIRPath constraint.
3. Add the "Note to Balloters" box on the CodeSystem fragment
   section soliciting implementer input on the open behaviors.
4. **Do not** add `TerminologyCapabilities.implementation.fragmentSupport`
   or `…supplementSupport` in this ticket; instead, reference an
   existing free-text mechanism (`TerminologyCapabilities.implementation`
   already carries a `description`, and the wider `codeSystem` /
   `expansion` blocks already let servers describe handling) and file
   a follow-up tracker against TerminologyCapabilities once the
   community has converged on what the new fields should distinguish.
5. As part of the same release, file the cross-version-fragment and
   "set of fragments == whole CS" sub-bullets back to FHIR-48959 (or
   into a fresh ticket) so they are not silently dropped.

#### Justification

The two new TerminologyCapabilities elements are the most invasive
part of the proposal: they add R6 schema surface, change generated
StructureDefinitions, conformance tooling, and validators, and they
do so before the implementer community has actually said *what* it
wants those markdown blocks to capture. The 2026-04-20 Zulip thread
has zero responses so far and the TI Tracker vote is scheduled only
two weeks out. Disposition B locks in the parts that the existing
discussion already supports (versioning SHOULD, complete-CS-wins rule,
supplement-with-fragment narrative) while keeping the schema surface
stable; it also avoids adding *required-when-supported* constraints
on fields that no current server populates. The trade-off versus
Disposition A is real — without the new capability fields servers
have nowhere structured to declare their fragment behavior — but the
risk of locking in an under-specified element shape is greater than
the cost of one additional ballot cycle.

---

### Disposition C: Decline

#### Proposal

Decline to make any normative changes in R6, on the grounds that the
substantive behaviors (multi-fragment handling, supplement coverage,
cross-version fragment combination, fragment-vs-complete conflict
resolution) are an *implementation governance* problem rather than a
specification gap. Close the ticket as "Not Persuasive" with a
narrative-only erratum to the existing CodeSystem fragments page
re-stating the already-present guidance ("Publishing a code system in
multiple fragments can create confusion … Code System authorities
that publish fragments should be careful and communicate their intent
clearly") and pointing implementers at the existing free-text
`TerminologyCapabilities.implementation.description` to describe their
behavior.

#### Justification

The original Zulip thread (Forest 2024-12-05; Lloyd McKenzie
2024-12-06) explicitly framed the problem as a governance issue, and
Michael Lawley stated that the FHIR spec does not — and arguably
should not — define behavior here. Several implementers (Lawley,
Lindström) hold *different* preferred behaviors (union of fragments
vs. reject-on-conflict vs. server-picks-one), so any normative rule
beyond "complete CS wins over fragment" risks invalidating one or
more existing server implementations. The "Note to Balloters" in
Disposition A is itself an admission that the WG is not yet ready to
prescribe behavior; declining now and re-filing once the community
has actually responded would give the next pass real evidence to work
from instead of leaving the new TerminologyCapabilities markdown
fields under-defined.

---

### Recommendation

**Recommended disposition:** A (with the modification noted)

Disposition A is the recommendation: adopt the WG's resolution as
proposed, but drop the unsatisfiable FHIRPath warning per Grahame
Grieve's comment. The clarifications it adds — the
complete-CS-wins rule, the SHOULD on `CodeSystem.version` for
versioned-CS fragments, and the structured
`fragmentSupport`/`supplementSupport` capability fields — are the
minimum needed to let consumers reason about whether a given server's
fragment behavior is safe for their use case, which is the actual
pain point reported by Lindström and reflected in both the
"Fragments and closure" and "Adding codes to fragment CS" Zulip
threads. Disposition B is tempting because the 2026-04-20
implementer-feedback thread is empty, but the new TerminologyCapabilities
elements are deliberately typed as `markdown [0..*]` precisely so
they can absorb whatever shape the implementer responses turn out to
take — there is no schema risk in adding them now and refining the
*comments* in a later cycle. Disposition C concedes too much: the
existing "be careful and communicate clearly" prose has been on the
page for years and demonstrably has not been enough, which is why
both Elliot Silver and Carmela Couderc kept driving toward a
structured resolution. The known open issues — the
"set-of-fragments == whole CS" assertion (deferred), cross-version
combination (deferred to FHIR-48959), and the FHIRPath constraint
(removed) — should be called out explicitly in the WG vote so the
ballot record is unambiguous about what is and is not in scope.
