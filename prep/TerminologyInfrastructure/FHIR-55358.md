## Summary

`NamingSystem.kind` is a mandatory (1..1) element on the `NamingSystem`
resource, bound with **required** strength to the `namingsystem-type`
value set (codes: `codesystem`, `identifier`, `root`). The reporter
(Lloyd McKenzie) observes that the value set has no `unknown`, `other`,
or equivalent "escape valve" concept. As a result, when the kind of an
identifier system is not actually one of those three (or is genuinely
unknown to the author), the producer cannot legally populate
`NamingSystem.kind` and therefore cannot create a conformant
`NamingSystem` instance at all. The ticket asks Terminology
Infrastructure to add an `unknown` and/or `other` concept to the value
set so the resource is always populatable.

The ticket has no linked or cross-referenced Jira issues, no Zulip
threads tied to it, and no associated GitHub PRs/issues. The single
cross-reference recorded by Augury is to the FHIR artifact
`NamingSystem.kind` itself (the element being discussed). The note
"Comment 322 - imported by: Lloyd McKenzie" in the description suggests

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an `unknown` concept to the `namingsystem-type` value set (and the
underlying `namingsystem-type` code system) in FHIR Core, with a
definition along the lines of: *"The kind of naming system is not
known."* Keep the binding on `NamingSystem.kind` at **required**
strength; the new code is the escape valve so author tooling can always
emit a conformant instance. Mark the change in the `NamingSystem`
maintenance notes and add a short usage note on the resource page
explaining when `unknown` is appropriate (i.e., when the author has a
genuine identifier/code-system/root naming construct but cannot classify
which of the three applies).

In repository terms this means an edit to the `namingsystem-type` code
system + value set definitions in `HL7/fhir` (or, if HL7 governance has
already migrated this terminology, the corresponding entry in
`HL7/UTG`), plus a short prose addition under
`source/namingsystem/` on the resource's notes/intro page.

#### Justification

This is the minimum change that resolves the populatability defect the
reporter identifies. FHIR has a long-standing convention of providing
`unknown` (and sometimes `other`) escape-valve concepts on required
bindings precisely so a 1..1 element with a closed value set never
becomes impossible to populate (e.g., `AdministrativeGender#unknown`,
`MaritimeStatusOfHealth`-style `unknown` codes, the `data-absent-reason`
pattern more generally). Following that convention here keeps
`NamingSystem.kind` mandatory while removing the trap. It is also
backward-compatible: existing instances using `codesystem`, `identifier`,
or `root` remain valid, and consumers that don't recognise `unknown`
fall back to "I don't know what kind this is" — which is the correct
semantics.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing three-code value set unchanged, but **relax the
binding strength** on `NamingSystem.kind` from `required` to
`extensible`. Producers that genuinely fall outside `codesystem` /
`identifier` / `root` would then be permitted to use a code from another
code system (e.g., a local code), while implementers who do fit one of
the three are unaffected. Optionally, add an `unknown` code in a
companion HL7-defined code system (e.g., reuse
`http://terminology.hl7.org/CodeSystem/data-absent-reason#unknown`) and
recommend it in the resource notes as the canonical escape valve when
the author simply doesn't know the kind.

In repository terms this is a binding-strength change to the
`NamingSystem` StructureDefinition (under `source/namingsystem/` in
`HL7/fhir`) plus updated resource-notes prose; the value set itself need
not change.

#### Justification

Relaxing to `extensible` solves the populatability problem at the
*binding* layer rather than the *value-set* layer, and acknowledges the
underlying issue: the three existing kinds are not exhaustive. It avoids
a debate about what `unknown` would semantically mean (is the author
saying "I don't know" or "it's none of these three"?) by letting
implementers choose an appropriate code when they have a real one, and
fall back to `data-absent-reason#unknown` when they don't. The trade-off
versus Disposition A is that it weakens interoperability slightly (two
systems may pick different "fourth" codes) and is a larger conformance
change than just adding a code, which may make it more contentious in
ballot review.

---

### Disposition C: Decline

#### Proposal

Decline the change. Treat the three existing values (`codesystem`,
`identifier`, `root`) as exhaustive of the kinds of naming construct
`NamingSystem` is intended to represent, and document explicitly in the
`NamingSystem.kind` element comments and resource intro that any
naming construct that does not classify as one of these is **out of
scope** for `NamingSystem` and should be expressed via a different
mechanism (e.g., `CodeSystem`, `Identifier.system` on a tracker
resource, an Implementation-Guide-defined construct). Close the ticket
as "Persuasive with mod – clarify scope" or "Not Persuasive" depending
on how the workgroup reads the original comment.

#### Justification

`NamingSystem.kind` was deliberately authored with a closed three-value
set to enforce a clean partition: a `NamingSystem` instance describes
either a code system, an identifier system, or a root naming authority.
If a producer cannot classify a construct into one of those three, the
right answer may be that the construct does not belong in `NamingSystem`
in the first place, rather than that the value set needs an escape
valve. Adding `unknown` risks degrading the data quality of the registry
of NamingSystems by making "I don't know" a routine answer for what is
intended to be a deliberate classification by the publisher of the
naming system.

---

### Recommendation

**Recommended disposition:** A

Adding an `unknown` concept (Disposition A) is the smallest,
lowest-risk, and most consistent-with-FHIR-convention change that fully
addresses the reporter's defect. The reporter's framing — *"can make
the containing element impossible to populate in some cases"* — is the
exact problem the FHIR project routinely solves with an `unknown`
escape-valve code on otherwise-required bindings; doing the same here
keeps `NamingSystem.kind` mandatory (so authoring tooling and validators
keep their current behaviour for the common case) while eliminating the
trap. Disposition B (loosen the binding) is a defensible alternative
but is a strictly larger conformance change and weakens interoperability
without a clear use-case driving the relaxation. Disposition C is
unlikely to be persuasive: the reporter is a long-standing core
contributor flagging a real populatability gap, and "use a different
resource" is not a satisfying answer when the author *does* have a
naming construct but simply cannot classify it. Workgroup discussion
should focus on whether to add just `unknown`, or also `other`, and on
the exact definition wording and the cross-link from the resource
notes.
