## Summary

This is the Terminology Infrastructure (TI) split of the broader R6 ballot 4
deprecation cleanup originally raised in
[FHIR-53722](https://jira.hl7.org/browse/FHIR-53722) ("Deprecated elements
should be withdrawn where applicable in R6"). Lloyd McKenzie observes that
several elements remain in R6 ballot 4 that are intended to be marked as
deprecated, and argues it is inappropriate to ship a major version aimed at
normativity while still carrying elements that explicitly should not be used.
He notes that the deprecation flag is not actually rendering in the R6
ballot 4 StructureDefinitions, but the deprecation is still present in the
underlying GitHub source files and in field descriptions, which makes the
problem hard to detect.

For TI specifically, this ticket calls out four resource elements that were
either Trial Use or Deprecated in R5 and therefore meet the parent ticket's
withdrawal criteria: **CodeSystem.topic**, **ConceptMap.topic**,
**NamingSystem.topic**, and **ValueSet.topic**. The reporter's stated rule
of thumb is binary: if the element is truly obsolete it should be removed,
and if it is to be retained for backwards compatibility it should not be
flagged as deprecated.

Important context already exists from the parent:
[FHIR-53722](https://jira.hl7.org/browse/FHIR-53722) was resolved by FHIR
Infrastructure on 2026-01-26 as **Persuasive with Modification**. FHIR-I
chose to remove `MetadataResource.topic` from the abstract resource (and to
keep the `topic` SearchParameter, repointed at `useContext` with code
`topic`), and explicitly noted that *"all of the derived resources will need
to have the element removed (or definition updated to no longer be derived
if they talk to us and explain why) at the same time."* The four `.topic`
elements TI is responsible for are exactly those derived elements.

Michael Donnelly subsequently created discrete per-element child tickets
covering the TI-owned resources:
[FHIR-55432](https://jira.hl7.org/browse/FHIR-55432) (CodeSystem.topic),
[FHIR-55433](https://jira.hl7.org/browse/FHIR-55433) (ConceptMap.topic),
[FHIR-55440](https://jira.hl7.org/browse/FHIR-55440) (NamingSystem.topic),
and [FHIR-55446](https://jira.hl7.org/browse/FHIR-55446) (ValueSet.topic).
Sibling roll-ups exist for other work groups —
[FHIR-55564](https://jira.hl7.org/browse/FHIR-55564) for CDS and
[FHIR-55566](https://jira.hl7.org/browse/FHIR-55566) for OO — and
`Observation.bodySite` is being handled by OO under
[FHIR-55441](https://jira.hl7.org/browse/FHIR-55441), so it is out of TI's
scope here.

## Details

**Description:**

A number of elements remain in R6 ballot 4 that are (intended to be) marked
as deprecated.

It seems odd to still have deprecated elements in a new major version
intended for normativity. Doing so requires FHIR models & consumers to
consider an element that is explicitly indicated as something that should
not be used.

As far as I can tell, except for `Observation.bodySite`, the following
elements (or resources) were either Trial Use or Deprecated in R5, which
seems to meet the criteria for withdrawing them in R6 (per definition of
Trial Use or per http://hl7.org/fhir/versions.html#deprecation).

If the element is truly obsolete, it should be removed. Or if it is
expected to be maintained for backwards compatibility, it should not be
considered deprecated.

**Note:** the deprecation flag doesn't seem to be rendered into the
R6-ballot4 StructureDefinitions, but field descriptions still indicate
these as being deprecated, and the deprecation still exists in the raw
github files. This makes it harder to find all deprecated fields, but here
is a hopefully complete list:

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the parent ticket's guidance verbatim and **remove all four TI
`.topic` elements from R6**: drop `CodeSystem.topic`, `ConceptMap.topic`,
`NamingSystem.topic`, and `ValueSet.topic` from their respective
StructureDefinitions, and remove the corresponding entries from the
generated profiles, schemas, examples, and search parameters that target
the element directly. Track the actual edits under the four already-open
child tickets (FHIR-55432, FHIR-55433, FHIR-55440, FHIR-55446) so that
each removal has its own audit trail. As part of this disposition, also
ensure that the resource-level `topic` SearchParameter is repointed at
`useContext` with code `topic` (mirroring the resolution applied to
`MetadataResource.topic` under FHIR-53722) so that consumers that were
searching by topic on these terminology resources continue to function.

#### Justification

Each of the four TI `.topic` elements satisfies the parent's removal
criteria: they all carry the deprecation comment, all four resources are
maintenance-status Trial Use (or were in R4), and the `topic` concept is
already inherited from the abstract `MetadataResource`. FHIR-I has
already removed `MetadataResource.topic` and was explicit in the FHIR-53722
resolution that *"all of the derived resources will need to have the
element removed... at the same time"* — meaning leaving the four TI
`.topic` slices in place would actually break the abstract derivation
contract. Repointing the search parameter at `useContext` mirrors what
FHIR-I already did for the abstract resource and preserves discoverability
without re-introducing the deprecated element. There is no dissenting
position on the parent thread; the only debate was per-element, and TI's
four elements all land on the "remove" side of the parent's decision tree.

---

### Disposition B: Alternative Approach

#### Proposal

**Remove three, retain one with an explicit deprecated flag.** Remove
`CodeSystem.topic`, `ConceptMap.topic`, and `NamingSystem.topic` (all of
which are clearly Trial Use derivatives whose only purpose is to inherit
`MetadataResource.topic`), but retain `ValueSet.topic` in R6 with a
properly rendered `deprecated` flag and a description pointer to
`useContext` with code `topic`. ValueSet is by far the most-implemented
terminology resource and is closest to normative; preserving the slot
gives implementers a one-release transition window on what is in practice
the most widely-deployed of the four. Carry the work under the three
"remove" child tickets and use FHIR-55446 to record the "retain with
deprecated flag" decision plus a rendering-fix follow-up so that the
deprecated badge actually appears in the published StructureDefinition
(addressing Lloyd's note that the flag isn't currently rendering).

#### Justification

The parent's guidance is explicit but not absolute: an element that is
deprecated *but does not meet any of the removal criteria* should "add the
deprecated flag" rather than be removed. The disposition above accepts
that all four elements technically satisfy the removal criteria, but
notes that ValueSet has the largest installed base of any TI resource
and that breaking the derivation cleanly inside one ballot cycle imposes
real migration cost on terminology servers and FHIR registries. Splitting
the disposition lets TI honour the spirit of FHIR-I's resolution (no
deprecated elements in normative content) for the three lower-impact
resources while giving ValueSet a documented one-release runway. The
trade-off is that it diverges from the parent ticket's "all derived
resources... at the same time" instruction and would need a quick check-
in with FHIR-I.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-55565 as duplicative and close it as **Not Persuasive –
duplicate of FHIR-53722 / per-element child tickets**. The substantive
decision was already made under FHIR-53722 (Persuasive with Modification),
the per-element work is already tracked under FHIR-55432, FHIR-55433,
FHIR-55440, and FHIR-55446, and TI has no element-level discretion that
hasn't already been delegated. Direct any TI-specific debate to the open
child tickets.

#### Justification

This disposition is defensible if the workgroup judges that FHIR-55565
adds no information beyond the parent and the four child tickets, and
that re-opening the underlying question would risk relitigating a
decision FHIR-I has already taken. Declining the roll-up would not change
the spec — the four child tickets continue independently — but it would
clean up the WG's queue and avoid voting twice on the same substance. The
risk is purely procedural: if a stakeholder reads only FHIR-55565 (e.g.,
because they searched the R6 ballot tracker for `R6-Normative-Ballot`),
declining without a substantive comment may look like TI is rejecting the
removal, when in fact the work is happening under different keys.

---

### Recommendation

**Recommended disposition:** A

The parent FHIR-53722 has already been resolved as Persuasive with
Modification by FHIR Infrastructure, with explicit guidance that derived
resources must remove the element in lockstep with the abstract
`MetadataResource.topic` removal. All four TI `.topic` elements
unambiguously satisfy the removal criteria in that guidance, the
per-element child tickets are already open, and the search parameter
repoint to `useContext` is a clean, well-precedented mitigation. Disposition
B is conceptually defensible but introduces a one-resource exception that
breaks the abstract derivation contract FHIR-I has just established —
which would generate cross-WG churn out of proportion to the migration
relief it provides. Disposition C is procedurally tidy but leaves the
ballot reviewer without a TI-level statement on a ticket that explicitly
calls out TI in its title. Accepting as requested is the cleanest outcome
for the R6 normative ballot and the easiest to defend to implementers.
