## Summary

This ticket bundles a long list of editorial / clarification changes to the
FHIR R6 Terminology Service specification page (section 4.7). The changes
were captured during a Terminology Infrastructure (TI) co-chair content
review (Confluence page 256510055, "FHIR R6 Terminology Content Review")
and Jessica Bota filed this Jira ticket on the group's behalf so the
edits can be tracked through the normal change-request workflow.

The proposed edits are predominantly text-only and fall into a few
patterns:

- **Restructure Basic Concepts** — promote the "Basic Concepts" sub-section
  from 4.7.3 to 4.7.1, refresh its bulleted "see also" list (Operations
  Framework, Terminology Fundamentals, terminology.hl7.org), and drop the
  "In addition…" trailing block.
- **Terminology hygiene** — systematically replace the loose terms
  "code", "codes", and "codes/concepts" with the more precise "concept
  code(s)" throughout 4.7.4, 4.7.5, and 4.7.6, while explicitly leaving
  "code" alone where it is correct (e.g., "code system", "error code").
- **Operation naming** — switch bare verbs ("the translate operation",
  "the expansion operation") to the canonical operation-handle form
  ("`$translate`", "`$expand`") and add a link from the Subsumption
  Testing intro to `$subsumes`.
- **Validate-Code wording** — rewrite the CodeableConcept guidance and
  the result-parameter sentence in 4.7.6 to align with the actual
  `$validate-code` output (`result` parameter, `issues`, optional
  display from the value-set expansion).
- **Deletions tied to a sibling ticket** — remove section 4.7.3.1
  "External Code Systems" entirely; this is contingent on the
  deprecation of the `CapabilityStatement Supported CodeSystem` extension
  proposed in [FHIR-53787](https://jira.hl7.org/browse/FHIR-53787).
- **Small editorial fixes** — punctuation, capitalisation ("Subsumption
  Testing"), CapabilityStatement / TerminologyCapabilities example
  bullet wording, and column-header rename in the operations table
  ("Required input parameters" → "Input parameters that must be
  supported").

The cross-reference index surfaces only one related item — the Confluence
review page that motivated the ticket. There is no inbound or outbound
GitHub linkage yet, and no Zulip discussion was found that references the
ticket key or its specific text edits.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the full edit list verbatim to `4.7 Terminology Service` on the
FHIR Core spec page in a single change request, treating it as one
coordinated editorial pass. Concretely:

1. Promote the existing **4.7.3 Basic Concepts** content to **4.7.1**,
   renumbering subsequent sections, refresh its "see also" bullets to
   the three URLs called out in the ticket (Operations Framework,
   Terminology Fundamentals, terminology.hl7.org), and drop the
   "In addition…" trailing block. For "Terminology Fundamentals" use a
   plain-text reference (no hyperlink) until that document is
   officially published, with an editorial note to add the link in a
   later release.
2. Apply the per-section text rewrites listed in the description
   (4.7.1.1, 4.7.1.2, 4.7.1.3, 4.7.3.2, 4.7.3.3, 4.7.4, 4.7.5,
   4.7.5.1, 4.7.6, 4.7.7, 4.7.9) exactly as worded.
3. Replace bare operation names with linked operation handles
   (`$expand`, `$lookup`, `$validate-code`, `$subsumes`, `$translate`)
   using the existing operation-page anchors.
4. Defer the deletion of **4.7.3.1 External Code Systems** until
   [FHIR-53787](https://jira.hl7.org/browse/FHIR-53787) is approved;
   resolve FHIR-53809 with a note that this single sub-bullet is
   tracked alongside FHIR-53787 and will be applied in the same R6
   build.

#### Justification

These are TI-vetted editorial changes with no semantic impact on the
Terminology Service operations themselves. Moving Basic Concepts to the
front aligns the spec page with the way the rest of the FHIR Core
operation pages are organised (overview → REST → operations →
maintenance), and the systematic "code → concept code" rewording closes
a long-standing source of reader confusion that surfaces repeatedly in
implementer questions. Tying the External Code Systems removal to
FHIR-53787 keeps the spec internally consistent: the prose only stops
making sense once the underlying extension is deprecated.

---

### Disposition B: Alternative Approach — Split into a tracked editorial work plan

#### Proposal

Accept the substance of the request but break it into three smaller,
independently-trackable tickets so each can be reviewed and merged on
its own merits:

- **B1 — Structural change:** promote 4.7.3 to 4.7.1 and refresh its
  bullet list. (One PR, easily reviewed, easy to revert.)
- **B2 — Terminology hygiene pass:** all the "code → concept code"
  rewrites in 4.7.4 / 4.7.5 / 4.7.6 plus the operation-handle linking
  (`$expand`, `$lookup`, `$validate-code`, `$subsumes`, `$translate`).
- **B3 — Validate-Code & contingent removals:** the 4.7.6 wording
  realignment (`result`, `issues`, ValueSet-expansion-display SHOULD)
  and the 4.7.3.1 removal, the latter explicitly blocked on
  [FHIR-53787](https://jira.hl7.org/browse/FHIR-53787).

Track FHIR-53809 as the umbrella; close it once B1–B3 are resolved.

#### Justification

The current ticket bundles three different *kinds* of change —
structural reorganisation, sweeping terminology rewording, and
normative-adjacent rewording (the `$validate-code` SHOULD) — into one
unit. Bundling makes review harder: a reviewer who disagrees with a
single bullet either has to block the whole change or accept it. The
"code → concept code" pass in particular is the highest-risk piece
because the ticket itself flags that "code" must *not* be replaced where
it is correct (code system, error code, the entire 4.7.5.1 sub-section);
isolating it in its own PR makes it easier to grep / diff / spot
mistakes. The structural promotion of Basic Concepts is also worth
isolating because it renumbers every downstream anchor on the page and
will invalidate inbound links from other spec pages.

---

### Disposition C: Decline

#### Proposal

Reject the ticket as filed and close it as **Out of Scope (process)**.
Direct the TI co-chairs to apply the editorial fixes via the normal
spec-editor channel (direct PR against `HL7/fhir`'s
`source/terminology-service.html`) rather than through a single
omnibus Jira change request, on the grounds that pure editorial /
typo / linking changes do not require change-request balloting.

#### Justification

Most of the bullets here are typographical or stylistic (capitalising
"Testing", removing a stray period, switching bare operation names to
backtick-wrapped handles, joining a paragraph) and do not change the
specification's meaning. The HL7 process generally allows editorial
fixes to be applied directly by the work group's spec editors without
requiring each one to be tracked as a Change Request, which would
otherwise consume disproportionate disposition cycles for what amounts
to copy-edits.

---

### Recommendation

**Recommended disposition:** **B**

Disposition A is workable but creates a single, sprawling PR that will
be hard to review and almost certain to need re-work on the
high-volume "code → concept code" pass; that risk argues against
treating these as one atomic edit. Disposition C undersells the
ticket — the move of Basic Concepts to 4.7.1, the deletion of 4.7.3.1
"External Code Systems", and the `$validate-code` rewording in 4.7.6
all touch on substantive normative wording (SHOULD, output parameter
names, deprecation linkage) and warrant proper Change-Request review,
not silent spec-editor fixes. Disposition B keeps the substance of the
TI co-chair review intact, gives reviewers a digestible diff per
sub-ticket, and naturally lines up the External Code Systems removal
with [FHIR-53787](https://jira.hl7.org/browse/FHIR-53787) so the two
land together in the same R6 build. The umbrella ticket structure also
mirrors how prior R6 documentation passes have been handled by TI.
