## Summary

The reporter notes that several Terminology Module operation parameters that
identify versionable terminology resources (value sets, code systems,
structure definitions) are currently typed as `uri`, which provides no
first-class way to convey a specific version of the referenced resource. The
ticket asks the Terminology Infrastructure work group to review these
parameters and decide whether to retype them to `canonical` (which embeds an
optional `|version` segment) or to add a companion version element. The
parameters explicitly called out are, on `ConceptMap/$translate`, the
`sourceScope`, `targetSystem` (when identifying a value set), and
`targetSystem` (when identifying a code system) input parameters, and on
`ValueSet/$validate-code`, the `context` input parameter (which identifies a
StructureDefinition). The reporter explicitly cross-references
[FHIR-53312](https://jira.hl7.org/browse/FHIR-53312) — a Highest-priority
Technical Correction (Resolution: Persuasive, status Resolved - change
required) that retyped `ConceptMap/$translate`'s `match.originMap` output
parameter from `uri` to `canonical` for the same versioning reason — and
positions this ticket as the broader sweep of remaining `uri`-typed
parameters that the comment on FHIR-53312 (carmela_couderc) flagged as
needing follow-up tickets.

## Details

**Description:**

The following operation input and output parameters currently have a **uri**
datatype, and may require an update to the specification to more clearly
support versions (i.e. change the datatype to **canonical** or add a separate
version element):

`$translate` operation
- *sourceScope* input parameter (identifies a value set)
- *targetSystem* input parameter (identifies a value set)
- *targetSystem* input parameter (identifies a code system)

`$validate-code` on ValueSet
- *context* input parameter (identifies a StructureDefinition)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Retype the four enumerated input parameters from `uri` to `canonical` in the
Terminology Module operation definitions, mirroring the FHIR-53312
treatment:

- `ConceptMap/$translate` — `sourceScope` (input): `uri` → `canonical`
  (target: ValueSet).
- `ConceptMap/$translate` — `targetSystem` (input, value-set sense): `uri` →
  `canonical` (target: ValueSet). If retaining the existing single
  `targetSystem` causes ambiguity with the code-system sense, split into
  `targetScope` (canonical[ValueSet]) and keep `targetSystem` for the code
  system, or document the polymorphism explicitly.
- `ConceptMap/$translate` — `targetSystem` (input, code-system sense): `uri`
  → `canonical` (target: CodeSystem).
- `ValueSet/$validate-code` — `context` (input): `uri` → `canonical`
  (target: StructureDefinition).

In each case, update the operation's parameter description to state that the
canonical reference MAY include a `|version` segment, and refresh the
operation's narrative and the corresponding examples accordingly.

#### Justification

This is the minimum-surprise outcome: it directly addresses the gap the
reporter identifies, and it carries the same precedent already adopted by
the work group in FHIR-53312 (Persuasive, change required). `canonical` was
designed exactly for this use — versionable references to canonical
resources — and unifying these parameters under `canonical` removes a small
but real interoperability hazard where a server cannot honour a client's
intent to pin a specific value-set, code-system, or structure-definition
version. carmela_couderc's comment on FHIR-53312 explicitly flagged these
remaining parameters as candidates for follow-up tickets, so accepting as
requested completes the work group's stated intent rather than reopening
it.

---

### Disposition B: Alternative Approach

#### Proposal

Keep each parameter typed as `uri` (preserving wire-format compatibility for
existing clients) and add, alongside each one, an optional sibling version
parameter:

- `ConceptMap/$translate`: add `sourceScopeVersion` (string, 0..1),
  `targetSystemVersion` (string, 0..1; applies whether `targetSystem`
  identifies a value set or a code system).
- `ValueSet/$validate-code`: add `contextVersion` (string, 0..1).

Document in each operation's narrative that, when a version sibling is
supplied, servers MUST resolve the canonical reference at that version, and
that the `|version` form on the URI is also accepted for clients that
prefer it; conflicts between the two forms SHALL be an error.

#### Justification

A datatype change from `uri` to `canonical` is technically backwards-
compatible at the JSON wire level (both are strings), but it is a schema-
level change in StructureDefinitions and downstream tooling (validators,
generated client SDKs, profile slicing). Disposition B avoids touching the
existing parameter datatypes entirely and gives clients an explicit, easily
documented way to pin versions, matching the pattern already used elsewhere
in FHIR (e.g. `valueSetVersion` alongside `url` on ValueSet expansion). The
trade-off versus Disposition A is parameter-surface bloat and the
inconsistency with FHIR-53312, which moved the same module's
`match.originMap` to `canonical` rather than introducing a sibling version
field.

---

### Disposition C: Decline

#### Proposal

Decline the request and document on each affected operation page that
clients needing version pinning should encode it in the existing `uri`
parameter using the `|version` convention already understood by canonical
references, with no formal datatype change.

#### Justification

This is the lowest-impact path and is defensible only on the grounds that
implementers can already pass `http://example.org/ValueSet/foo|1.2.0` in a
`uri`-typed parameter when servers cooperate. However, it is materially
weaker than the FHIR-53312 precedent, where the work group explicitly chose
the `canonical` retype rather than relying on the `|version` convention on
a `uri`. Declining would also leave carmela_couderc's flagged follow-up
work unresolved and would create inconsistency between input parameters
(still `uri`) and at least one output parameter (`match.originMap`, now
`canonical`) on the same operation.

---

### Recommendation

**Recommended disposition:** A

The work group already adopted the `uri` → `canonical` retype on the same
operation family in FHIR-53312 with a Persuasive resolution, and the
comment thread on that ticket explicitly anticipated this follow-up. Taking
the matching action on `ConceptMap/$translate.sourceScope`,
`ConceptMap/$translate.targetSystem` (both senses), and
`ValueSet/$validate-code.context` keeps the Terminology Module internally
consistent, reuses an established design pattern, and avoids growing the
operation parameter surface with parallel `*Version` siblings (Disposition
B) that the work group has so far declined to introduce. The dual-sense
overloading of `targetSystem` should be addressed in the same change —
either by an explicit narrative note that the canonical may target either a
ValueSet or a CodeSystem, or by splitting it — but that is a minor scoping
decision inside Disposition A rather than a reason to prefer B or C.
