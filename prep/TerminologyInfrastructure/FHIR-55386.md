## Summary

Michael Lawley requests that `ConceptMap` be enhanced so a map author can
computably declare which subset of `concept-map-relationship` codes are
permitted on `ConceptMap.group.element.target.relationship` for a given
ConceptMap. Today, although `ConceptMap` declares its `property`,
`additionalAttribute`, and `sourceScope`/`targetScope` computably, there
is no analogous declaration that constrains the allowed relationship
codes — even though, in practice, many ConceptMaps only use a single
relationship (e.g., only `equivalent`). The reporter proposes adding
`ConceptMap.allowedRelationships` (`code`, `0..*`, bound to
`http://hl7.org/fhir/ValueSet/concept-map-relationship`) to declare the
permitted subset. No related Jira tickets, Zulip discussions, or GitHub
items were surfaced via cross-references or search.

## Details

**Description:**

When creating / maintaining a ConceptMap, it is common for only a subset
of the available `target.relationship` types to be considered valid —
e.g., only "equivalent".

Just as properties, additionalAttributes, source and target scope are
declared computably, so should this subset.

I propose the addition of the code-typed `ConceptMap.allowedRelationships`
element `0..*` that is bound to
<http://hl7.org/fhir/ValueSet/concept-map-relationship> that limits the
set of valid values for `ConceptMap.group.element.target.relationship`.

**Comments:**

No comments on the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new element `ConceptMap.allowedRelationships` to the base
`ConceptMap` resource with the following definition:

- **Cardinality:** `0..*`
- **Type:** `code`
- **Binding:** required, to `http://hl7.org/fhir/ValueSet/concept-map-relationship`
  (the same value set used by `ConceptMap.group.element.target.relationship`)
- **Short:** "The relationship codes that may be used in this ConceptMap"
- **Definition:** "A subset of the relationship value set whose members
  are the only relationship codes that MAY appear on
  `ConceptMap.group.element.target.relationship` within this
  ConceptMap. If absent, all codes from the bound value set are permitted."
- **Comment:** Empty (`0..*` with zero entries) is equivalent to absent
  — i.e., all codes are permitted.
- **Position:** Place alongside the existing computable scope/property
  declarations in the ConceptMap header (near `property`,
  `additionalAttribute`, `sourceScope`, `targetScope`), not under
  `group`.

Add a FHIRPath invariant on `ConceptMap` such as:

```
allowedRelationships.empty() or
group.element.target.relationship.all($this in %context.allowedRelationships)
```

(or equivalently expressed with `subsetOf`) so that any
`target.relationship` used in any group is one of the declared allowed
relationships when the list is non-empty.

This change would be made in the FHIR Core specification — i.e., in the
`HL7/fhir` repo under `source/conceptmap/` (resource definition,
structure XML, and the resource intro/notes pages), and the resource
would be re-published through the bespoke HL7 publisher.

#### Justification

The proposal mirrors a well-established pattern already present on
`ConceptMap`: `property`, `additionalAttribute`, `sourceScope`, and
`targetScope` all computably declare what may appear inside the
`group.element` content. Adding `allowedRelationships` extends that
pattern to the relationship codes themselves, which closes a real
authoring gap: today a server or client can pre-validate that a
property is recognised but cannot pre-validate that a relationship
code is one the map intends to use. A required binding to the same
canonical value set used by the per-target `relationship` element
ensures consistency, and the `0..*` cardinality with "absent ==
unconstrained" preserves backward compatibility for every existing
ConceptMap. Adding a FHIRPath invariant also gives validators a
deterministic check, consistent with how other computable scope
elements are enforced on `ConceptMap`.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a new element to base `ConceptMap`. Instead, address the
underlying need via two non-breaking mechanisms that already exist:

1. **Standard extension** — define a new HL7 standard extension,
   e.g., `http://hl7.org/fhir/StructureDefinition/conceptmap-allowedRelationship`,
   `0..*` of type `code` bound to
   `http://hl7.org/fhir/ValueSet/concept-map-relationship`, applicable
   at `ConceptMap` root. Document its semantics identically to
   Disposition A (absent or empty = all codes allowed).
2. **Profile guidance** — show a worked example/profile in the
   ConceptMap intro (e.g., an "equivalence-only" profile) that uses
   the extension plus a `constraint` invariant binding
   `target.relationship` to a narrower required value set, so
   profile-aware validators can enforce the subset without any base
   resource change.

If the workgroup later observes wide uptake, the extension can be
promoted to a core element in a future release using the standard
extension-promotion process.

#### Justification

`ConceptMap` is a high-traffic resource and any new core element ripples
into every conformance artifact (StructureDefinition, JSON/XML
schemas, generated code, validators, tooling, snapshots in dependent
IGs). Extensions are the FHIR-sanctioned mechanism for "useful in
specific authoring contexts but not universally needed" semantics, and
this request — restricting the relationship subset for an individual
ConceptMap — is by the reporter's own description a common-but-not-
universal authoring concern. Using an extension also avoids creating
a second redundant binding site for `concept-map-relationship` codes
on the same resource, which can confuse implementers about precedence
when both the per-target binding and the new constraint disagree.
Profile-based narrowing of `target.relationship` already gives strict
enforcement when desired without any base spec change.

---

### Disposition C: Decline

#### Proposal

Do not adopt the change. Close as "Persuasive with Mod / Not Persuasive"
with the rationale that the existing `target.relationship` required
binding to `http://hl7.org/fhir/ValueSet/concept-map-relationship`
already constrains the allowed codes, and any further narrowing for a
particular ConceptMap can be expressed today either by (a) profiling
`ConceptMap` to bind `target.relationship` to a narrower required
value set, or (b) authoring an organisational invariant. No core
element change is necessary.

#### Justification

FHIR's general design preference is to push optional, author-specific
constraints into profiles rather than the base resource, especially
when the constraint is fundamentally a binding-narrowing operation —
which is exactly what profiling required bindings is for. Adding
`ConceptMap.allowedRelationships` introduces a second, parallel
mechanism for constraining relationship codes that could disagree with
a profile's narrowed binding (e.g., the element lists `equivalent`
only, but a profile narrows to `equivalent | source-is-narrower-
than-target`), forcing implementers to define precedence rules. The
reporter has not cited interoperability failures attributable to the
absence of this declaration, and there is no cross-referenced Zulip
or GitHub discussion indicating broader community demand. The cost of
a new core element on a widely-implemented resource is not yet
justified.

---

### Recommendation

**Recommended disposition:** A

The request is small, additive, fully backward compatible (`0..*`,
absent == unconstrained), and follows a pattern already established on
`ConceptMap` itself — `property`, `additionalAttribute`,
`sourceScope`, and `targetScope` all do exactly this kind of
computable declaration of what is allowed inside `group.element`.
Disposition C's "use profiles" answer is technically true but
misaligned with the existing resource design: the other scope
declarations were not pushed to profiles either, and asymmetry between
them and `relationship` is a usability papercut for ConceptMap
authors and tooling. Disposition B (extension first, promote later)
is a defensible conservative path, but the analogous elements on the
same resource were never extensions, so starting `allowedRelationships`
as an extension would create exactly the inconsistency the reporter
is asking to remove. Pair acceptance with the FHIRPath invariant
described in Disposition A so that the constraint is machine-checkable
from day one, and confirm with the reporter and the Vocabulary /
Terminology Infrastructure WG whether the binding strength should be
`required` (recommended) or `extensible`.
