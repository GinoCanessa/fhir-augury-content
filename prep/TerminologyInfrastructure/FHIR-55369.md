## Summary

FHIR-55369 asks the Terminology Infrastructure work group to clarify, in the
normative description of the `ValueSet/$expand` operation, the relationship
between the `includeDesignations` (Boolean) input parameter and the
`designation` (token, repeating) input parameter. The ticket was spun out of
[FHIR-43919](https://jira.hl7.org/browse/FHIR-43919), which originally asked
for documentation explaining how `includeDesignations`, `designation`, and
`displayLanguage` interact when expanding a value set (e.g., "return all
French designations for each concept"). Discussion on the 2025-09-29 TI
tracker call concluded that guidance is needed, but the group could not
agree whether the `designation` filter should only take effect when
`includeDesignations=true`, when it should be honoured regardless, or
whether a non-empty `designation` should implicitly enable designation
output. Reuben Daniels filed FHIR-55369 to track that single open question
so FHIR-43919 (the broader doc work) can be unblocked.

## Proposed Dispositions

The ticket targets only the prose / `OperationDefinition` for
`ValueSet/$expand` in the FHIR Core specification. Because no GitHub
cross-references exist, no repo briefings have been consulted; any of the
dispositions below would be implemented in the FHIR Core authoring source
for the `ValueSet/$expand` operation page (typically the
`valueset-expand.xml`/`.md` operation definition and the
`valueset-operations.html` narrative), and the workgroup should confirm
the exact paths against the current `HL7/fhir` clone before drafting a PR.

### Disposition A: Accept As Requested

#### Proposal

Add a new "Parameter interaction" subsection to the `ValueSet/$expand`
operation page that normatively states:

1. `displayLanguage` always applies to
   `ValueSet.expansion.contains.display`, **independent** of
   `includeDesignations`.
2. When `includeDesignations` is `true` (or absent — default `true` per
   the existing definition), `designation` acts as a **filter**: only
   designations whose `use` or `language` matches one of the supplied
   tokens are returned in `ValueSet.expansion.contains.designation`. If
   `designation` is absent, server discretion applies (status quo).
3. When `includeDesignations` is **explicitly `false`**, `designation`
   SHALL be ignored and no `ValueSet.expansion.contains.designation`
   elements are returned, even if `designation` filter tokens are
   supplied. Servers MAY return an `OperationOutcome` warning indicating
   that `designation` was ignored.
4. Add a worked example demonstrating
   `?includeDesignations=true&designation=urn:ietf:bcp:47|fr` returning
   only French designations per concept.

This codifies the consensus reached on the 2018 terminology thread
(Peter Jordan's interpretation, agreed by Jack Bowie) and resolves
Michael Lawley's "I'm not sure what should happen" case in the most
intuitive direction.

#### Justification

This matches the principle of least surprise (`includeDesignations=false`
literally means "don't include designations", so a filter on designations
is moot), is consistent with the existing implementer behaviour
documented in the HAPI thread, and avoids changing the wire format or
parameter cardinality. It is purely documentation/clarification, which
matches the `Operation` label and the parent ticket's framing as a
documentation request. The 2018 Zulip consensus, while informal, has
stood for ~7 years without serious objection.

---

### Disposition B: Alternative Approach — Treat a non-empty `designation` as implicitly enabling output

#### Proposal

Same documentation additions as Disposition A for items 1, 2, and 4,
but invert item 3:

3'. If `designation` is supplied with one or more values, designations
matching the filter SHALL be returned **regardless of the value of
`includeDesignations`**. An explicit `includeDesignations=false` combined
with a non-empty `designation` is treated as a request for those
specific designations only. If both are absent, the existing default
applies.

Optionally, deprecate `includeDesignations` in a future version in favour
of `designation` alone, since `includeDesignations=true` with no
`designation` filter could be expressed as "designation matches any".

#### Justification

This reading argues that supplying a `designation` filter is an
unambiguous user signal — the caller went to the trouble of naming a
language/use, so silently dropping that filter because of a default
(`includeDesignations`) value violates the principle of least astonishment
in the opposite direction. It also keeps `designation` useful in clients
that don't explicitly set `includeDesignations` and avoids requiring two
parameters to express a single intent. Trade-off vs. A: it makes the
parameter set slightly more complex to reason about (a "false" can become
"partially true"), but it reduces the number of round-trips for callers
who only want, say, French designations.

---

### Disposition C: Decline

#### Proposal

Close FHIR-55369 as "Persuasive with Mod – No Change Required" and
resolve FHIR-43919 with a pointer to the existing prose. Argue that
the current text ("if no designation is specified, it is at the server
discretion which designations to return") combined with the literal
reading of `includeDesignations` already covers both cases by server
discretion, and that codifying one interpretation will break
implementations that have chosen the other.

#### Justification

Multiple implementations exist in the wild on both sides of the
question (per the implementer/HAPI threads), and a normative
clarification in either direction risks invalidating conformant
behaviour. Server discretion is a defensible status quo for an edge
combination (`includeDesignations=false` plus a `designation` filter)
that has not produced visible interoperability failures in the 7+ years
since the 2018 thread.

---

### Recommendation

**Recommended disposition:** A

Disposition A captures the only interpretation the community has ever
reached informal consensus on (Peter Jordan's 2018 proposal, agreed on
the thread and not contradicted since), aligns with the literal English
reading of `includeDesignations=false`, and is the smallest change that
unblocks the parent FHIR-43919. Disposition B is internally coherent but
introduces a new concept ("a parameter that overrides another parameter
when non-empty") that isn't used elsewhere in `$expand`, and it would
break servers that currently honour `includeDesignations=false`
strictly. Disposition C leaves the question open and forces FHIR-43919
to either ship with a known ambiguity or be closed without satisfying
the original request — neither outcome is workgroup-friendly. Adopting
A also sets up a clean future deprecation path (B) if the workgroup
later decides server discretion needs to be tightened further.
