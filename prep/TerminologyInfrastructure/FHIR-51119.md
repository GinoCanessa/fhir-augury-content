## Summary

FHIR-51119 proposes extending `ValueSet.compose.include.filter` so that a
filter can contain nested child filters in place of its `value`. Specifically,
when the filter's `op` is one that already accepts a list of codes or a value
set (e.g. `in`, `not-in`, and the new `of` operator from
[FHIR-51118](https://jira.hl7.org/browse/FHIR-51118)), the spec would allow
zero-or-more nested `filter` elements as a substitute for `value`. The set of
codes consumed by the parent operator is then defined as the **intersection**
of the codes selected by the nested filters. As a related change, the
cardinality of `filter.value` would relax from `1..1` to `0..1`, with a new
invariant requiring that either `value` or `filter` be present. The reporter
explicitly notes that this is *not* a breaking change: every existing
`ValueSet.compose.include.filter` instance remains valid, but authors can now
express more complex intersections inline rather than decomposing them into
multiple chained value sets.

The ticket sits in the middle of a coordinated cluster of TI tracker items
filed by Grahame Grieve on 2025-06-17 in the wake of the long-running VCL
("ValueSet Compose Language") discussion (see
[FHIR-51117](https://jira.hl7.org/browse/FHIR-51117) and
[FHIR-51118](https://jira.hl7.org/browse/FHIR-51118)). FHIR-51117 clarifies
how `filter.value` is interpreted and lets `in`/`not-in` reference a value
set. FHIR-51118 adds the new `of` operator for navigating property-of
relationships. FHIR-51119 builds on both: with `in`, `not-in`, and `of` all
able to take a value set, nested filters become the natural inline equivalent
of pulling a small intersection of code-set logic into the parent filter
without spawning an auxiliary `ValueSet` resource. The ticket has been
triaged as Persuasive; the resolution text says the proposed structural
changes and constraint will be made and elaborated when voted on.

## Details

**Description:**

> Courtesy of [FHIR-51117](https://jira.hl7.org/browse/FHIR-51117) and 51118,
> we are able to compose pretty complex value sets that meet a lot of the use
> cases that have been identified as limitations in the existing FHIR
> ValueSet compose syntax.
>
> But while we can compose some very complicated value sets, it's a bit clunky
> because we keep having to break our logic across multiple value sets. It'd
> be a cleaner tighter syntax if a filter could contain other filters in place
> of a value. That is, specifically, that if the operator allows for value
> set, it can also allow for nested filters in place of the value. So
> proposal is to add
>
> `filter : @filter 0..*`  Use in place for value if value can be a list of codes
>
> and a formal definition of:
>
> > one or more filters are allowed in place of a value if (and only if) the
> > operator allows for a list of codes/and or a value set. If one or more
> > filters are present, the intersection of the filters specifies the set of
> > codes that are the 'value' the operator acts on.
>
> Related change: make `filter.value` `0..1` instead of `1..1`, and add an
> invariant that either `value` or `filter` SHALL be present.
>
> This is not a breaking change — all the existing value sets continue to be
> valid, but it allows for new value sets that have more complex inline
> definitions without being decomposed into component value sets.

**Resolution Description (Persuasive):**

> Make the proposed changes to the structure of the value set resource and
> add the constraint as specified. This disposition will be elaborated on
> further when the ticket is voted on.

**Comments:** No comments were recorded on the ticket.

### [FHIR-51117: Clarify the Definition of ValueSet.compose.include.filter.value](https://jira.hl7.org/browse/FHIR-51117)

Tightens the definition of `ValueSet.compose.include.filter.value`: cleans up
the markdown, makes it explicit that for `in`/`not-in` the value is a
comma-separated list of codes, lets the code system govern interpretation
(important for SNOMED CT reference sets), and — most relevant to FHIR-51119 —
allows the `in`/`not-in` value to be a value set canonical (with `,`/`%`
percent-encoded). Status: **Applied** (Persuasive with Modification). Without
51117 enabling value-set values on `in`/`not-in`, the nested-filter pattern
proposed by 51119 has fewer operators to attach to.

Users involved: Grahame Grieve (reporter), carmela_couderc, reuben.daniels,
Michael Lawley (mjlawley).

### [FHIR-51118: Add new operator 'of' for ValueSet.compose.include.filter.operator](https://jira.hl7.org/browse/FHIR-51118)

Adds a new `property-value-of` operator (alias `of`) to the
`filter-operator` code system that selects "the codes that are the property
values of the codes in the provided value (comma-separated list or value
set)." Status: **Resolved – change required** (Persuasive with Modification).
This is the third operator that takes a list of codes / value set, and
together with `in`/`not-in` it forms the set of operators against which
FHIR-51119's nested-filter mechanism would be defined.

Users involved: Grahame Grieve (reporter).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply exactly the changes described in the ticket to the
`ValueSet` resource StructureDefinition for the next FHIR Core release:

1. Add a new repeating element
   `ValueSet.compose.include.filter.filter : BackboneElement 0..*` whose
   type is the same backbone as the parent `filter` (i.e. a self-reference
   producing nested `filter` instances). Each nested `filter` carries the
   usual `property` / `op` / `value` (and now `filter`) children.
2. Relax `ValueSet.compose.include.filter.value` from `1..1` to `0..1`.
3. Add an invariant on `ValueSet.compose.include.filter` (e.g. `vsd-N`)
   stating: "Either `value` or `filter` SHALL be present, but not both"
   (or "at least one"; the WG should pick the exact wording when voting).
4. Add a second invariant restricting nested `filter` to operators that
   accept a list of codes or a value set — i.e. those operators whose
   `filter-operator` code system entries are flagged as taking
   list/value-set values (currently `in`, `not-in`, and the new
   `property-value-of` from FHIR-51118). Operators such as `=`, `regex`,
   `is-a`, `descendent-of`, `exists` etc. would still require `value`.
5. Add narrative text on the ValueSet page (the same place the existing
   filter/operator semantics are described) defining the runtime
   semantics: "If one or more nested `filter` elements are present, the
   set of codes the parent operator acts on is the intersection of the
   code sets selected by each nested filter, evaluated against the same
   `system` as the enclosing `include`/`exclude`."
6. Add at least one worked example to the ValueSet examples set
   (e.g. an intersection of two LOINC property filters) to anchor the
   semantics.

#### Justification

This is what the ticket asks for, what the reporter has been steering the
TI tracker cluster toward, and what the resolution text already commits to
("make the proposed changes ... and add the constraint as specified"). The
underlying design is sound:

* It is **non-breaking**. Existing value sets all still satisfy the new
  schema because `value` was `1..1` and remains permitted; the new
  `filter` child is `0..*`.
* It is **internally consistent** with the rest of the cluster:
  FHIR-51117 makes `in`/`not-in` accept a value set, FHIR-51118 adds an
  `of`/`property-value-of` operator that also accepts a value set or
  list of codes, and FHIR-51119 lets that "value-set-shaped" slot be
  replaced inline by an intersection of further filters. Without 51119,
  many real-world VCL expressions would still need to spawn auxiliary
  `ValueSet` resources just to express an intersection.
* It directly addresses the alignment goal Grahame Grieve articulated in
  the `terminology > VCL` thread: that everything VCL can express must
  have a clean transform to `ValueSet.compose`. Nested filters close
  the most-cited gap (inline intersection / nested sub-expression).

The Zulip thread shows broad TI/Vocab community interest in making
`ValueSet.compose` more expressive in exactly this way, and TI's tracker
disposition has already classified the ticket as Persuasive.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a recursive `filter.filter` child, define
nested-filter logic as a **contained / inline anonymous `ValueSet`**
referenced from `filter.value`. Concretely:

* Keep `filter.value` `1..1` (no cardinality change, no new invariant on
  `value` vs `filter`).
* Add a new sibling element
  `ValueSet.compose.include.filter.valueSet : BackboneElement 0..1` whose
  shape is `compose.include` (i.e. the same `system` + `concept` +
  `filter` machinery), to be used **only** when `op` allows a list of
  codes or value set.
* Define semantics: when `filter.valueSet` is present, the parent
  operator is evaluated against the codes that the nested
  `compose.include` selects; `filter.value` MUST be absent in that case.
* Add a single invariant tying presence/absence of `value` vs
  `valueSet` to whether `op` is one of the value-set-accepting
  operators.

This gives equivalent expressive power (intersection comes out of the
nested `compose.include`'s own `filter` cardinality) but routes the
recursion through the existing `compose.include` shape rather than
introducing a new self-referential backbone on `filter`.

#### Justification

* Self-references on backbone elements are unusual in FHIR and tooling
  (validators, generators, IG publisher diagrams) sometimes handles
  them awkwardly. Routing through `compose.include` avoids that.
* It composes with everything `compose.include` already supports
  (`system`, `version`, `concept`, multiple `filter`s, `valueSet`
  references), giving authors uniform mental model: "a filter can
  carry an inline mini-compose."
* It is still non-breaking on existing instances.

Trade-offs vs. A: the surface area is larger (a whole nested
`compose.include` shape rather than just nested `filter`s), and the
"intersection" semantics become "whatever the nested include selects"
which is a superset of what FHIR-51119 actually asks for (you'd also
inherit `concept` and other things the ticket explicitly does not
mention). It is also a bigger spec change to validate, and it doesn't
match how Michael Lawley's VCL grammar mirrors the structure (VCL
sub-expressions look more like nested `filter`s than nested
`compose.include`s).

---

### Disposition C: Decline

#### Proposal

Decline FHIR-51119 on the grounds that the intersection it expresses
can already be obtained by listing **multiple `filter` elements on the
same `compose.include`** (which are AND-ed together, per the existing
`ValueSet` semantics), or by referencing a separate `ValueSet`
canonical in the (now value-set-capable) `in` / `not-in` /
`property-value-of` operators (per FHIR-51117 and FHIR-51118). Therefore
no structural change to `ValueSet.compose.include.filter` is needed;
authors who want "intersection inside a value-set-accepting operator"
should define an auxiliary `ValueSet`.

#### Justification

* It is the most conservative option: no schema change, no new
  invariant, no new tooling work for validators / IG publisher /
  reference servers / language SDKs.
* It avoids introducing a recursive structure on a backbone element,
  which historically has caused friction in code generators.
* The functional outcome is reachable today (post-51117/51118), just
  with one extra named `ValueSet` per intersection.

The weakness — and the reason this is a defensible-but-not-recommended
position — is that it is exactly the "clunky" pattern Grahame Grieve
calls out in the description, and the WG has already triaged the
ticket as Persuasive. The Zulip thread also documents repeated calls
from Lawley/Steel/Mandel to make `compose` more expressive.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested).

Disposition A matches the existing TI triage outcome (Persuasive), is
internally consistent with the two sibling tickets (FHIR-51117 and
FHIR-51118) that have already advanced through TI, and is explicitly
non-breaking. The "intersection of nested filters" semantics is the
minimal addition needed to give `ValueSet.compose` parity with VCL's
nested sub-expressions — which is the alignment Grahame Grieve is
driving in the `terminology > VCL` thread, and which Michael Lawley
and Jim Steel have argued for since 2023. Disposition B's nested-
`compose.include` shape is plausible and worth raising in discussion,
but it is a larger spec change than the ticket asks for and pulls in
features (notably `concept` enumeration) that the ticket explicitly
does not need; it should be considered only if the WG dislikes the
self-referential `filter.filter` element. Disposition C is the
"do nothing structural" path and is documented mainly so the WG can
weigh it; it leaves authors with the same composition pain the ticket
was filed to fix and contradicts the existing Persuasive triage. The
WG should also lock down two small wording questions when voting on A:
(1) whether the new invariant is "exactly one of value or filter" or
"at least one"; and (2) which operators in the `filter-operator` code
system are formally tagged as accepting a list of codes / value set
(driving the second invariant in A.4).
