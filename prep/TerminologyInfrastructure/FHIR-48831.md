## Summary

[FHIR-48831](https://jira.hl7.org/browse/FHIR-48831) requests that the
definition of `ValueSet.compose.include.filter.value` be rewritten so it
accurately covers every datatype a CodeSystem property/filter can take
(code, Coding, string, integer, boolean, dateTime, decimal) and clearly
specifies the string serialization for each — particularly Coding, which
the reporter (Michael Lawley) suggests should use FHIR Search **token
syntax** (`[system]|[code]`). It also asks that for the `=` operator the
value be permitted as the simple string form of any supported type, and
that for numeric types FHIR Search comparator prefixes (e.g., `lt10`,
`gt100`) be allowed so equality‑style operators can express inequality
without resorting to hand‑written regex.

The ticket explicitly cites
[FHIR-38104](https://jira.hl7.org/browse/FHIR-38104) (the original
clarification request, now Applied/Persuasive) as having only partially
addressed the underlying problem. In comments the reporter further links
the ticket to [FHIR-51117](https://jira.hl7.org/browse/FHIR-51117) —
which has since been Applied (Persuasive with Modification) and added
support for the `in`/`not-in` operators and ValueSet‑canonical values —
and to [FHIR-44866](https://jira.hl7.org/browse/FHIR-44866) ("ValueSet
filter.value should define syntax for different data types"), whose
proposed approach has been **pre‑applied** to the build (per the
[filter‑values
section](https://build.fhir.org/valueset.html#filter-values)) and which
the original reporter on FHIR‑44866 (Josh Mandel) accepted on
2026‑02‑20.

The reporter's most recent comments (2025‑12‑08, 2026‑01‑15) state that
this ticket can be closed because the underlying concern has been
resolved by those other tickets, but no formal disposition has yet been
voted by the WG and the validator behaviour observed on Zulip
(`VALUESET_BAD_FILTER_VALUE_CODED_INVALID` thrown against valid
token‑syntax filter values) suggests there may still be open
implementation gaps even if the spec text now reflects the change.

## Details

**Description:**

> Ticket FHIR-38104 requested clarification of the valid values for
> ValueSet.compose.filter.value but the result did not properly address
> the issue.
>
> The set of valid types for filters extends beyond code, string, and
> boolean.
>
> Furthermore based on the following, which includes the "=" operator:
>
> > For all other filter operations
> > (ValueSet.compose.include.filter.op) the value
> > (ValueSet.compose.include.filter.value) represents a code.
>
> The only way to do a simple equality match on these other types is to
> use the "regex" operator!
>
> The description should be updated to make clear that for the "="
> operator the value can be the simple string representation of any of
> the supported types.
>
> It should also clearly indicate how a Coding should be represented in
> string form: I would suggest "token syntax" — `[system]|[code]`.
>
> Furthermore, for numeric values it would also be very helpful if the
> Search syntax were also adopted to allow inequality testing. For
> example, for the integer filter "size", allow the following:
>
> ```json
> {"filter": [
>   { "property": "size", "op": "=", "value": "lt10" }
> ]}
> ```

### [FHIR-51117: Clarify the Definition of ValueSet.compose.include.filter.value](https://jira.hl7.org/browse/FHIR-51117)

Now **Applied / Persuasive with Modification**. Filed by Grahame Grieve
in June 2025; targets the same definition text. Three concrete changes
were accepted: (1) when `op = in`/`not-in` the value is a comma‑separated
list of codes (with the SNOMED CT reference‑set carve‑out documented on
the SCT CodeSystem page); (2) interpretation of the codes is under the
control of the CodeSystem; (3) a ValueSet canonical may be used as the
filter value (new functionality, non‑breaking). Discussed at the
2025‑07‑14 TI Trackers call and the September 2025 WGM with FHIR‑I.
This is the ticket Michael Lawley credits with resolving FHIR‑48831's
core ask around the `=`/`in` semantics.

Reporter: Grahame Grieve. Commenters: Carmela Couderc, Reuben Daniels,
Michael Lawley.

### [FHIR-44866: ValueSet.compose.include.filter.value should define syntax for different data types](https://jira.hl7.org/browse/FHIR-44866)

Status: **Waiting for Input → Persuasive** (changes pre‑applied at
[`build.fhir.org/valueset.html#filter-values`](https://build.fhir.org/valueset.html#filter-values)
and at the
[element definition](https://build.fhir.org/valueset-definitions.html#ValueSet.compose.include.filter.value)).

Reporter/Assignee: Josh Mandel. Commenters: Carmela Couderc, Reuben

## Related Zulip Discussions

All three cross‑referenced Zulip messages are from a single thread on
the terminology stream.

### [terminology > Defining Relationships Between Codes from Different CodeSyst](https://chat.fhir.org/#narrow/stream/terminology/topic/Defining%20Relationships%20Between%20Codes%20from%20Different%20CodeSyst)

Started 2024‑11‑12 by Tayeb merabti asking whether FHIR can express
cross‑CodeSystem relationships (beyond ConceptMap) and use them as
ValueSet filter properties. Rutt Lindström confirmed that a CodeSystem
property of type Coding works in principle but reported being unable to
get a ValueSet filter on a Coding‑typed property to work. Michael Lawley
explained that filters on Coding‑typed properties need **token syntax**
for `value`, and recommended avoiding Coding‑typed properties when a
single CodeSystem is implicit (reserving them for cross‑system cases or
when subsumption operators like `is-a` need to know the target system).
He gave a working example using `system#code`; Gino Canessa corrected
this to **pipe syntax** (`system|code`) per
`build.fhir.org/search.html#token`. Michael Lawley then opened
FHIR‑48831 (his message of 2024‑11‑14 in this thread links to it
alongside FHIR‑38104).

The thread re‑surfaced in mid‑2025 with Abdelali Boussadi reporting that
the validator emits `VALUESET_BAD_FILTER_VALUE_CODED_INVALID` for a
ValueSet whose filter uses the documented pipe‑form
`system|code` value (pointing at element location
`ValueSet.compose.include[0].filter[0]`), and asking whether FHIR‑48831
covers the validator side or only the spec text. No definitive answer
was given in the thread; Boussadi notes "you already discussed this
point in this ticket … but I'm not sure you took a decision about the
syntax and/or impact on the validator." So while the spec direction is
now reasonably settled (pipe‑separated token syntax for Coding),
**implementer experience suggests at least one major validator is still
rejecting valid input**, which the WG should weigh before closing this
ticket as a duplicate.

Participants: Tayeb merabti, Rutt Lindström, Michael Lawley, Gino
Canessa, Abdelali Boussadi (and indirectly, via @-mentions, Grahame
Grieve).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept all three asks of FHIR‑48831 by extending the now‑pre‑applied
text at
[`build.fhir.org/valueset.html#filter-values`](https://build.fhir.org/valueset.html#filter-values)
(authored under the `hl7/fhir` core spec, ValueSet definition + the
`#filter-values` markdown block) to:

1. **State explicitly** that for the `=` operator the `value` is the
   string serialization of whichever datatype the referenced
   CodeSystem property/filter declares (code, Coding, string, integer,
   boolean, dateTime, decimal).
2. **Normatively define** the Coding serialization as FHIR Search
   token syntax `[system]|[code]`, with an explicit note that the
   `system|version#code('display')` form Grahame Grieve described is
   accepted by some validators but **not** the canonical wire form,
   and align validator messages
   (`VALUESET_BAD_FILTER_VALUE_CODED_INVALID`) accordingly.
3. **Adopt FHIR Search number/date prefix syntax** (`eq`, `lt`, `le`,
   `gt`, `ge`, `ne`, `ap`, plus the date‑specific forms) for numeric
   and dateTime properties when `op = '='`, with worked examples for
   integer, decimal and dateTime properties.

Cross‑link to `codesystem.html#fhir-search` so the rules live in one
place rather than being scattered between CodeSystem and ValueSet pages.

#### Justification

FHIR‑44866 has already been accepted with the per‑datatype syntax pinned
to FHIR Search conventions and FHIR‑51117 fixed the `in`/`not-in`/list
semantics. What is still missing is exactly what FHIR‑48831 asks for:
(a) the explicit `=` rule for non‑code datatypes, (b) a single canonical
Coding serialization, and (c) numeric/date inequality without forcing
authors to write regular expressions over decimals (which the FHIR‑38104
discussion already acknowledged is error‑prone). Aligning with FHIR
Search keeps the terminology and search modules consistent and minimises
implementer surprise. The validator gap reported on the terminology
Zulip thread is direct, recent evidence that the spec change without an
accompanying clarification on the wire format is insufficient.

---

### Disposition B: Alternative Approach

#### Proposal

**Mark FHIR‑48831 as Persuasive‑Resolved By** the combination of
[FHIR‑38104](https://jira.hl7.org/browse/FHIR-38104),
[FHIR‑51117](https://jira.hl7.org/browse/FHIR-51117) and
[FHIR‑44866](https://jira.hl7.org/browse/FHIR-44866) — but only after
performing a focused **gap audit**:

1. Diff the current text at
   [`build.fhir.org/valueset.html#filter-values`](https://build.fhir.org/valueset.html#filter-values)
   against the three asks of FHIR‑48831 (ranged numeric/date values,
   normative Coding token syntax, explicit `=`‑operator rule). For each
   ask not already covered, file a small, narrowly‑scoped follow‑up
   tracker against the next R6 ballot cycle rather than re‑opening the
   broader request here.
2. File a separate **validator alignment** ticket (TI + FHIR‑I joint)
   referencing the
   `VALUESET_BAD_FILTER_VALUE_CODED_INVALID` behaviour reported on the
   terminology Zulip thread, so that core‑validator behaviour catches up
   with the spec text rather than leaving FHIR‑48831 as a catch‑all.

Close FHIR‑48831 with a note linking to the gap‑audit results and the
new validator ticket.

#### Justification

The reporter himself stated on 2025‑12‑08 and 2026‑01‑15 that the issue
"can be closed" because it has been resolved by other tickets. Treating
FHIR‑48831 as a tracking placeholder rather than re‑deriving the same
clarifications risks duplicating work and producing inconsistent text.
However, simply closing it without checking would lose two real signals
the ticket carries: the numeric‑comparator request (which FHIR‑44866
addresses only partially via the example) and the validator behaviour
gap (which is purely an implementation issue, not a spec issue). A
small, audited close‑out with one or two narrowly‑scoped follow‑ups is
cheaper to maintain than rolling everything back into a broad ticket.

---

### Disposition C: Decline

#### Proposal

Close FHIR‑48831 as **Not Persuasive — Already Addressed**, with a
canonical pointer to
[`build.fhir.org/valueset.html#filter-values`](https://build.fhir.org/valueset.html#filter-values)
and to FHIR‑38104, FHIR‑51117, FHIR‑44866 as the resolving tickets. Do
not open follow‑ups; rely on existing FHIR Search documentation
(`search.html#token`, `search.html#number`, `search.html#date`) for the
per‑datatype syntax, since CodeSystem/ValueSet already cross‑reference
those.

#### Justification

Every concrete ask in the original ticket has been addressed by one of
the three related tickets, all of which have been Applied or
pre‑applied: per‑datatype syntax (FHIR‑44866), `=`/`in`/list/ValueSet‑URI
semantics (FHIR‑51117), and the cleaner element definition (FHIR‑38104).
The reporter has explicitly agreed it can be closed. Re‑opening the
specification text under FHIR‑48831 risks introducing inconsistencies
with text that has already been balloted/applied. The validator
complaint on Zulip is implementer behaviour, not a spec defect, and the
TI WG's tracker isn't the right venue to track HAPI/Java‑validator
issues.

---

### Recommendation

**Recommended disposition:** **B**

A is appealing because it produces a single, complete clarification, but
it duplicates work the WG has already done across FHIR‑44866 and
FHIR‑51117 — and the editor risk of re‑touching text that was just
applied is real. C is the cleanest paperwork outcome and matches the
reporter's stated wish to close, but it ignores two pieces of useful
signal that FHIR‑48831 carries: (1) the numeric inequality use case that
FHIR‑44866's resolution covers more by example than by normative rule,
and (2) the validator gap surfaced on the terminology Zulip thread,
which is current implementer pain. B captures both: it honours the
reporter's request to close, it avoids re‑editing freshly‑applied text,
and it converts the still‑open concerns into narrowly‑scoped tickets
where they can be tracked and disposed properly — including a joint
validator/spec ticket that the FHIR‑I and TI WGs can co‑own. The
expected effort is one short gap‑audit pass at a TI Trackers call plus
filing 0–2 follow‑up trackers, which is materially less than re‑opening
the spec edit under A.
