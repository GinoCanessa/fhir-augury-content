## Summary

Vadim Peretokin reports that the narrative text describing `designation`
elements is nearly identical in the
[CodeSystem](http://hl7.org/fhir/codesystem.html#designations) and
[ValueSet](http://hl7.org/fhir/valueset.html#designations) resource pages,
yet the two resources play very different roles when handling translations.
Specifically, when translating a third-party CodeSystem into another
language, the additional language designations belong on a CodeSystem
**supplement**, not on a ValueSet. The current spec text gives readers no
hint of this distinction, so without consulting Zulip an implementer is
likely to put translation designations on the wrong resource — described in
the ticket as a "footgun."

The ticket asks for two things:

1. **Spec clarification** — improve the guidance on the CodeSystem and
   ValueSet pages so readers understand when translation designations
   belong on a CodeSystem supplement versus a ValueSet (where designations
   are typically used for non-language purposes such as the LOINC short /
   long display example).
2. **Tooling check** — have the IG publisher emit a warning/info message
   when **language** designations are attached to a ValueSet, since that is
   almost certainly a misuse.

A linked Zulip discussion ("Better guidance for translating codesystems",
terminology stream) confirms the use case with Grahame Grieve, who
endorsed filing the ticket but cautioned against a hard validator rule:
language designations on a ValueSet are *not wrong*, just much narrower in
purpose. A subsequent comment from Michael Lawley on the ticket points
out that the existing example should reference the
`http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra#preferredForLanguage`
designation use code, and suggests adding that code to the
[`designation-use`](http://hl7.org/fhir/ValueSet/designation-use)
ValueSet.

There are no cross-referenced Jira tickets and no GitHub items linked to
this issue in the FhirAugury index.

## Details

**Description:**

In the context of translating codesystems, the [CodeSystem](http://hl7.org/fhir/codesystem.html#designations)
resource says about designations:

> In addition to the display and definition, a concept can have one or
> more `designation` elements. The display is equivalent to a special
> designation with an implied `designation.use` meaning "primary
> code/designation" and a language equal to the
> [Resource Language](http://hl7.org/fhir/resource.html#language). The
> designations can provide additional displays for other language, as
> well as designations for other purposes. When using concepts,
> applications use the `display` and `designation` unless the language
> or usage in context provides a reason to use one of the designations.

The [ValueSet](http://hl7.org/fhir/valueset.html#designations) resource
says:

> In addition to the display, a concept can have one or more
> `designation` elements. The display is equivalent to a special
> designation with an implied `designation.use` of "primary code" and a
> language equal to the
> [Resource Language](http://hl7.org/fhir/resource.html#language). The
> designations can provide additional displays for other languages, as
> well as designations for other purposes. When using concepts,
> applications use the `display` unless the language or usage in
> context provides a reason to use one of the designations.

The text is pretty much the same, which is unfortunate, because there
is a very big difference between the two. When translating 3rd party
codesystems, designations should be
[added to the codesystem supplements](https://chat.fhir.org/#narrow/channel/179252-IG-creation/topic/.E2.9C.94.20Translating.203rd.20party.20codesystems),
**not** valuesets. This is a bit of a footgun: without consulting
Zulip, it's not possible to know the right way to use translation
designations.

It would be great if the spec was clearer about this. Additionally,
the IG publisher should give a warning/information message when adding
**language designations** to valuesets — which is something you are
unlikely to do. I imagine designations added to a valueset would be
non-language ones, just like the LOINC code example in the spec. See
also the associated
[Zulip discussion](https://chat.fhir.org/#narrow/channel/179202-terminology/topic/Better.20guidance.20for.20translating.20codesystems).

**Comments:**

- **mjlawley (2025-08-03):** That text should really point the reader
  to the designation use code
  `http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra#preferredForLanguage`,
  and that code should probably also be included in the
  [`designation-use`](http://hl7.org/fhir/ValueSet/designation-use)
  ValueSet.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Make two coordinated changes:

1. **Spec text — CodeSystem § Designations and ValueSet § Designations.**
   Rewrite both blocks so the divergent intent is explicit:
   - On the **CodeSystem** page, state that additional language
     designations are how a CodeSystem (or, for third-party code systems
     a CodeSystem **supplement**) carries translations of its concepts'
     displays. Cross-link to the supplements page and to the
     [`designation-use`](http://hl7.org/fhir/ValueSet/designation-use)
     ValueSet (and to
     `http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra#preferredForLanguage`
     per mjlawley's comment).
   - On the **ValueSet** page, state that designations on a ValueSet
     are intended for context-specific or use-restricted overrides
     (e.g., the LOINC short/long display example) and that **language
     translations belong on a CodeSystem (supplement), not on the
     ValueSet**. Add a "Note / Warning" callout pointing readers to
     the CodeSystem supplements pattern when their goal is translation.
2. **`designation-use` ValueSet.** Add
   `http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra#preferredForLanguage`
   to
   [`http://hl7.org/fhir/ValueSet/designation-use`](http://hl7.org/fhir/ValueSet/designation-use)
   so the recommended code is discoverable from the spec.
3. **IG publisher.** Emit an informational (not error) message when a
   `ValueSet.compose.include.concept.designation` (or
   `ValueSet.expansion.contains.designation`) carries a `language`
   that does not match the ValueSet's resource language and no
   `designation.use` is set or `use` is purely linguistic — pointing
   the author at CodeSystem supplements as the recommended place for
   translations. Per Grahame's comment in the Zulip thread, this stays
   *informational*, not a validation error.

#### Justification

This is the literal request, and it is well-scoped: the ticket is
labelled `TEXTONLY` for the prose change, with a small additional ask
to enrich `designation-use` and to add a soft IG-publisher message.
The Zulip thread shows the reporter and Grahame both agree the spec
text needs to clarify intent; mjlawley's ticket comment provides the
exact terminology code to point at. Keeping the IG-publisher signal at
"info / warning" — not an error — directly addresses Grahame's concern
that language designations on a ValueSet are not wrong, just much more
narrowly useful. All changes are additive and backward compatible.

---

### Disposition B: Alternative Approach — Spec-only clarification, defer tooling

#### Proposal

Adopt the spec-text changes from Disposition A (rewrite the
designations sections on CodeSystem and ValueSet, add the
`preferredForLanguage` code to `designation-use`, cross-link to
CodeSystem supplements), but **defer** the IG-publisher
warning/information message to a separate tooling ticket against the
HL7/fhir-ig-publisher project. The current ticket would be resolved
once the spec text and `designation-use` ValueSet are updated.

#### Justification

The IG-publisher behaviour is a tooling change owned by a different
codebase and review path, and it carries the soft-vs-hard policy
question Grahame raised on Zulip. Splitting the work lets Terminology
Infrastructure dispose the spec/text portion in this ticket window
without blocking on a tooling design discussion (what counts as a
"language-only" designation, what severity, opt-out story for IGs that
deliberately attach translation hints to a ValueSet). It also keeps
the Terminology Infrastructure work group's disposition narrowly
within the spec and the THO `designation-use` ValueSet. The spec
clarification alone removes most of the footgun by telling readers
where translations belong before they ever get to the publisher.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Modification → Not Persuasive** on the
grounds that (a) the existing CodeSystem and ValueSet specs already
state that designations can carry "additional displays for other
languages" and that the appropriate use depends on context, and (b)
the canonical guidance to use CodeSystem supplements for third-party
translations is documented elsewhere (terminology guidance pages, the
referenced Zulip discussion, and the THO supplements material). The
`designation-use` ValueSet remains as-is; no IG-publisher behaviour
change.

#### Justification

A defensible "decline" rests on the position that the current text is
not technically incorrect — both resources *do* allow language
designations, and which one to use depends on the modelling intent.
However, the ticket's central evidence is exactly that this technical
correctness has been failing implementers ("a footgun: without
consulting Zulip, it's not possible to know the right way"), and the
Zulip thread shows broad agreement that the spec should help. So this
disposition is the weakest of the three and is included primarily as
the contrasting option for the work group's discussion.

---

### Recommendation

**Recommended disposition: A (Accept As Requested)** — with the
explicit caveat from Grahame Grieve's Zulip reply that the IG-publisher
signal must be **informational**, not a validation error.

The change is genuinely text-and-tooling guidance, not a structural or
breaking change. The Zulip discussion converged on "yes, please file
this," mjlawley's ticket comment supplies the concrete THO code to
point at, and the ask is consistent with how the FHIR spec already
treats nearby clarifications (cross-link the relevant pattern,
enumerate the canonical designation-use codes). Splitting off the
IG-publisher work (Disposition B) is reasonable if the tooling owners
need their own design pass, but the spec-side substance is identical;
the recommendation is therefore A with a willingness to fall back to B
if the publisher change cannot be scheduled in the same cycle.
Disposition C undersells the reporter's evidence — the very fact that
the canonical guidance lives on Zulip rather than on the resource
pages is the bug being reported.
