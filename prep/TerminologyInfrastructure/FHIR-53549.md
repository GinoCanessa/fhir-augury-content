## Summary

The ticket reports an inconsistency in the FHIR Core specification text on
the `CodeSystem` page regarding the recommended datatype for coded
`CodeSystem.concept.property` values. The CodeSystem Supplements narrative
correctly tells authors that when a supplement defines new concepts/codes
for use as property values, those values should use the `Coding` datatype
(paired with a possibly-contained CodeSystem) so consumers — most
importantly the IG publisher — know which system the codes come from.
Immediately afterward, the Concept Properties section uses language that
appears to contradict this: it says "The values for properties of type
'code' (`CodeSystem.concept.property.valueCode`) are drawn from the codes
defined in the CodeSystem instance," which (in the reporter's reading)
suggests `code` is the appropriate datatype for coded property values
even when those codes come from the CodeSystem itself. The reporter is
asking the workgroup to align the two passages so that authors are
consistently directed toward `Coding` (`valueCoding`) for coded property
values, rather than `code` (`valueCode`).

There are no cross-referenced sibling Jira tickets. The single GitHub
cross-reference (commit `cfb8c4e…`) is a false positive: the commit
message contains the literal string "FHIR-53549" but the embedded link
and the actual subject ("Added observation patterns supporting 3
consecutive negative stick tests") point to FHIR-53459 — a different,
unrelated ticket.

## Details

**Description:**

> If a Codesystem defines coded properties, the datatype of the
> properties should be Coding, not code. Otherwise, the IG publisher
> will not know what system the codes are coming from.
>
> While the text for Codesystem supplements make this obvious:
>
> > If a supplement needs to define new concepts/codes to use as
> > property values, it can be paired with a new (possibly contained)
> > Code System and use the `Coding` type for the property values.
>
> The text that follows this one, in the Concept Properties section,
> seems to confuse this and refer to the code datatype instead of
> Coding:
>
> > The type of the property value. The values for properties of type
> > "code" (`CodeSystem.concept.property.valueCode`) are drawn from the
> > codes defined in the CodeSystem instance.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the **Concept Properties** narrative on the CodeSystem page in the
FHIR Core spec (`source/codesystem/…` page in `HL7/fhir`, the spec page
that renders as `codesystem.html`) so that the example used to describe
`CodeSystem.concept.property.type` no longer treats `code` as the
canonical example for "values drawn from the CodeSystem instance."
Concretely:

- Reword the offending sentence so it mirrors the Supplements guidance:
  property values that are themselves codes drawn from a CodeSystem
  SHOULD use `Coding` (`CodeSystem.concept.property.valueCoding`),
  paired with a (possibly contained) CodeSystem so the system is
  unambiguous to consumers such as the IG publisher.
- Retain `code` (`valueCode`) only as a legacy / constrained option,
  with an explicit note that it is discouraged for new content because
  the system is implicit and tooling cannot reliably resolve the code
  out of context.
- Add a forward-pointer from the Concept Properties section to the
  Supplements paragraph so the two passages read as a single coherent
  recommendation.
- Make no schema or StructureDefinition change: both `valueCode` and
  `valueCoding` already exist on `CodeSystem.concept.property.value[x]`;
  this is documentation-only.

#### Justification

This is the lowest-risk way to remove the inconsistency the reporter
identified, and it aligns the Concept Properties prose with guidance
that the workgroup has already accepted in the immediately-adjacent
Supplements paragraph. The IG-publisher concern raised in the ticket
description is real: a bare `code` value with no companion `system` is
ambiguous to downstream tooling that does not assume "the enclosing
CodeSystem" as the implicit system, and the Supplements text explicitly
calls out `Coding` as the fix. Documentation alignment also avoids any
risk of breaking existing instances (both datatypes remain valid),
which is appropriate for a Triaged / Medium-priority Change Request
with no community-wide redesign discussion behind it.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than editing only the Concept Properties paragraph in isolation,
treat this as a small documentation-quality work item and:

1. Lift the datatype guidance (Coding vs. code) into a dedicated
   subsection ("Choosing the property value datatype") on the CodeSystem
   page that is referenced by both the Concept Properties section and
   the CodeSystem Supplements section, so a single normative paragraph
   governs both. Both call-sites then link into it instead of restating
   it.
2. In that subsection, document the decision matrix explicitly:
   - `valueCoding` — REQUIRED when the values come from any external
     CodeSystem and RECOMMENDED whenever the values come from a
     contained / sibling CodeSystem (so the system is explicit).
   - `valueCode` — permitted only when the property's binding is a
     ValueSet bound at the property definition level so the system is
     statically derivable, and even then a non-normative note that
     `valueCoding` is preferred for tooling interoperability.
3. Add a short "Migration note" pointing existing CodeSystem authors at
   the new guidance without invalidating prior content.

#### Justification

The reporter's underlying complaint is not just one sentence — it is
that the page presents two different recommendations side by side
without acknowledging the relationship between them. Fixing only the
Concept Properties sentence (Disposition A) leaves the Supplements
paragraph as a duplicate of the new normative text and risks the
inconsistency re-emerging the next time either section is edited.
Consolidating into a single "Choosing the property value datatype"
subsection is consistent with how the FHIR Core spec already factors
out shared design notes (e.g., extension vs. modifier extension
guidance) and gives implementers a single canonical place to cite. The
trade-off versus A is more editorial churn on a single page and a
slightly larger review surface; nothing in the change touches the
schema or instance validity.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive — sentence is consistent with the schema**.
The Concept Properties paragraph is describing what the datatype `code`
*does* (its values are drawn from the enclosing CodeSystem), not
recommending `code` over `Coding`. The Supplements paragraph is layered
guidance for the supplement-authoring case. Close with a comment that
no change to the spec text is required, and that authors selecting a
property datatype should consult the `value[x]` definition on
`CodeSystem.concept.property` directly.

#### Justification

A defensible read of the current text is that the two paragraphs are
answering different questions: "what does `valueCode` mean?" versus
"what should a supplement use?" Under that read, the apparent
contradiction is a reading-order artifact rather than a real
inconsistency, and the workgroup may prefer to avoid touching normative
narrative for what is fundamentally a clarity preference. Declining
also avoids the risk that re-wording the Concept Properties section
narrows the legitimate uses of `valueCode` (e.g., where a property is
bound to a strongly-bound ValueSet whose system is statically
discoverable) and creates a new class of complaints from CodeSystem
authors who have been using `valueCode` correctly.

---

### Recommendation

**Recommended disposition:** A (with a friendly nudge toward B if the
workgroup has appetite for slightly broader editorial work).

The reporter's concern is grounded: the IG publisher and other
consumers genuinely cannot resolve a `valueCode` to a system without
out-of-band knowledge that the CodeSystem in hand is the source, and
the Supplements paragraph already concedes this is the right pattern.
Disposition C ("the text is fine, you're misreading it") is unlikely to
satisfy the reporter and leaves the inconsistency in the spec for the
next reader to trip over. Disposition A delivers exactly what was
asked, is documentation-only, carries no instance-level breaking risk,
and is small enough for a single PR against the CodeSystem page in
`HL7/fhir`. If during drafting the editor finds that the two paragraphs
naturally collapse into one shared subsection, escalating in-flight to
Disposition B is cheap; doing B up-front is only worthwhile if the
workgroup wants to formalise the decision matrix at the same time.
Either way, the implementation PR should be paired with a
clone-current `repo-analysis` briefing for `HL7/fhir` before the edit
lands.
