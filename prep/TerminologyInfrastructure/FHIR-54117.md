## Summary

FHIR-54117 asks the Terminology Infrastructure work group to remove or
otherwise resolve a "dragon warning" embedded in the Code System
supplements / value set expansion discussion in FHIR Core. The current
text states that the impact of Code System supplements on value set
expansion (and therefore on value set validation) is "subject to ongoing
experimentation and implementation testing, and further clarification
and additional rules might be proposed in future versions of this
specification." The reporter argues that an open-ended "dragons here"
caveat is not appropriate for a section that is being advanced as
normative, and that the specification must commit to defined behavior
before publication.

The reporter explicitly relates this to
[FHIR-48960](https://jira.hl7.org/browse/FHIR-48960) — *Clarify fragment
and supplement relationships* — which is the broader, already
"Persuasive with Modification" ticket that proposes concrete
clarifications around how supplements (and fragments) interact with
complete code systems, with value set expansion, and with terminology
server capability declarations. FHIR-54117 is effectively the normative
follow-up: even after FHIR-48960's clarifications land, the residual
"dragon" hedge in the supplements-and-expansion section needs to be
either deleted or replaced with a definitive (even if narrowly scoped)
normative statement.

## Linked Jira Tickets

### [FHIR-48960: Clarify fragment and supplement relationships](https://jira.hl7.org/browse/FHIR-48960)

This Change Request (Medium priority, resolved as *Persuasive with
Modification*) asks for definitive guidance on how multiple fragments of
a code system relate to each other, how fragments relate to
supplements, and what happens when a fragment / supplement / complete
code system disagree. The proposed resolution adds a new section
(terminology-service.html §4.7.4 "Fragments and Supplements"), adds a
SHOULD constraint that fragments of versioned code systems carry
`CodeSystem.version`, and introduces
`TerminologyCapabilities.implementation.fragmentSupport` and
`TerminologyCapabilities.implementation.supplementSupport` so servers
can declare their behavior. It is the substantive ticket of which
FHIR-54117 is the editorial / normative-readiness counterpart: once
FHIR-48960's text lands, the lingering "dragon" hedge in the
supplements/expansion section is what FHIR-54117 wants removed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Delete the dragon warning paragraph from the "Code System supplements
and value set expansion" subsection of the CodeSystem / ValueSet
narrative (the sentence beginning *"The impact of Code System
supplements on value set expansion ... is subject to ongoing
experimentation and implementation testing ..."*). Replace it with
normative text that pins down the behavior the work group is willing to
commit to today. At minimum:

1. State unambiguously that an `$expand` on a value set whose included
   code system has one or more supplements MUST take those supplements
   into account when (a) the supplement's `CodeSystem.url` and
   `CodeSystem.version` (where present) match the resolved base code
   system, and (b) the supplement is known to the terminology server
   performing the expansion.
2. Define the consequence for `$validate-code` (and therefore
   value-set validation): a code is considered valid in the expansion
   if it is defined in the base code system; supplement content that
   adds designations, properties, or filterable properties does not
   widen or narrow membership unless the value set composition
   explicitly references those properties.
3. Require servers to advertise their supplement-handling behavior via
   `TerminologyCapabilities.implementation.supplementSupport` (the
   element being introduced under the FHIR-48960 resolution), so that
   any residual implementation variation is discoverable rather than
   silent.

This change would touch the FHIR Core specification source — the same
authoring tree that produces `codesystem.html` (Code System
Supplements section), `valueset.html` (Expansion), and
`terminology-service.html` (the new §4.7.4 introduced by FHIR-48960).

#### Justification

A normative section cannot ship with an open-ended "we may add rules
later" caveat — that is precisely the kind of forward-incompatible hook
that normative status is meant to forbid. FHIR-48960 has already done
most of the conceptual work for fragments and supplements (Persuasive
with Modification), and its resolution introduces the
`supplementSupport` capability declaration that gives servers a clean
way to surface any remaining implementation variability. Removing the
dragon and replacing it with a definite rule (plus a discoverable
capability declaration) is consistent with how FHIR has handled other
late-normative cleanups: commit to the rule, and let
`TerminologyCapabilities` carry the residual variation.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the substance of the warning but reshape it so it is not a
"dragon" in a normative section. Specifically:

- Remove the speculative "further clarification and additional rules
  might be proposed in future versions" wording from the normative
  prose.
- Move the *advisory* portion ("supplement handling in expansion is an
  area of active implementation experience") into a clearly marked
  non-normative box (e.g. an "Implementation note" or "Note to
  implementers" callout) immediately following the normative section,
  using the same callout style used elsewhere in FHIR Core for
  non-normative guidance.
- In the normative text itself, state the minimum behavior the work
  group is confident in (e.g. "supplements MUST be applied to
  designations and properties returned in the expansion when the
  server holds them; supplements MUST NOT alter value set membership
  except via composition rules that explicitly reference supplement
  properties"), and defer the contested edge cases (e.g. cross-server
  supplement discovery) to `TerminologyCapabilities`.

#### Justification

This preserves the work group's honesty about an area where
implementation experience is genuinely still maturing, without
embedding that hedge in the normative text. FHIR has used non-normative
callouts elsewhere (e.g. implementation guidance boxes around search
parameter handling) for exactly this purpose. It carries less risk than
Disposition A of overcommitting to a behavior that a later
implementation round trips on, while still resolving the reporter's
core complaint that a normative section should not contain an
open-ended "rules might change" warning. The cost is editorial
complexity: the work group must be disciplined about which sentences
move to the callout and which stay normative.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave the existing wording in place, on the
basis that the paragraph is a candid and accurate description of an
area where implementation experience is still accruing, and that the
work group is not yet in a position to commit to a single normative
rule for supplement-aware expansion. Mark the resolution as *Not
Persuasive*, with the rationale that the warning is informational, not
a normative escape clause, and that the underlying clarification work
is being tracked under FHIR-48960.

#### Justification

This would be defensible only if the work group genuinely believes the
supplement / expansion interaction is too unsettled to specify, and
that the alternative (committing to a rule that some implementations
cannot yet meet) is worse than the current hedge. The risk is real:
forcing premature normative wording can lock in behavior that
implementers later cannot follow, which is itself a ballot hazard.
However, the reporter's substantive point — that "rules might be
proposed in future versions" is incompatible with normative status —
is hard to refute on its face, so this disposition is the weakest of
the three.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right starting point for the work group. The
reporter's complaint is editorially correct — a normative section
cannot promise that more rules might appear later — but Disposition A
risks overcommitting to supplement-and-expansion behavior that
FHIR-48960 has only partially settled and that implementers are still
exercising. Disposition B removes the offending hedge from the
normative text (which is what the ticket actually demands), preserves
the work group's transparency by relocating the implementation-maturity
language into a clearly marked non-normative callout, and leans on the
new `TerminologyCapabilities.implementation.supplementSupport` element
(landing via FHIR-48960) to carry any remaining server-by-server
variation. If the work group later finds that the residual variation
has narrowed, it can tighten the normative text in a follow-up cycle
without having had to walk back an over-strong commitment.
