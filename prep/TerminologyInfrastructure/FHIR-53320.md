## Summary

The reporter (Elliot Silver) flags an inconsistency in the
`ValueSet-reference` search parameter on the FHIR Core ValueSet resource.
The human-readable description states the parameter searches code systems
that are *included or excluded* in the value set (or in an imported value
set), but the published FHIRPath expression only walks
`ValueSet.compose.include.system` — it never visits
`ValueSet.compose.exclude.system`, nor does it follow imported value sets
via `ValueSet.compose.include.valueSet` / `ValueSet.compose.exclude.valueSet`.
The reporter additionally questions the parameter name `reference`, calling
it a poor choice and asking whether compatibility rules permit a rename.

The TI tracker call on 2026-04-13 agreed the documentation/expression are
inconsistent and proposed a "Persuasive with Modification" resolution that
goes beyond the literal request: change the parameter type from `uri` to
`canonical`, rewrite the description to make clear the parameter returns
ValueSets that reference a code system *or value set* (in either include
or exclude) — directly or via an included/excluded value set — and extend
the expression to add the three missing paths (`compose.exclude.system`,
`compose.include.valueSet`, `compose.exclude.valueSet`). The reporter
accepts the broader scope but raises a substantive-vs-text-only labelling
concern; carmela_couderc clarifies that `TEXTONLY` is a tracking tag and
the final disposition will be re-tagged appropriately. The proposed name
change has not been resolved — Elliot suggested `composeSystem` but no
follow-up rename decision is recorded.

There are **no cross-referenced Jira tickets, Zulip threads, or GitHub
issues/PRs** linked to FHIR-53320 in the indexed sources; the only
cross-references are internal mentions of three FHIR ValueSet element
paths.

## Details

**Description:**

> The description indicates that the parameter searches includes or
> **excluded** code systems, but the FHIRpath only searches included
> codesystems.
>
> Also "reference" is a poor name choice for search param, and should be
> changed if compatibility rules permit.

**Resolution Description (proposed by TI):**

Agree with submitter that the documentation is inconsistent.

- *Current text:* Parameter `reference` : `uri` — "A code system included
  or excluded in the value set or an imported value set"
- *Change to:* Parameter `reference` : `canonical` — "All value sets on
  the server which reference a code system or value set included or
  excluded in the value set. It is up to the server whether to return
  information about the references to a code system or value set in
  included or excluded value sets."
- Update the expression to add three expressions to the existing one:
  `ValueSet.compose.exclude.system`, `ValueSet.compose.include.valueSet`,
  `ValueSet.compose.exclude.valueSet`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the WG-proposed "Persuasive with Modification" exactly as recorded
on the 2026-04-13 TI tracker call, in the FHIR Core ValueSet resource
under `HL7/fhir` (authoring root `source/`, ValueSet folder
`source/valueset/`, in the search-parameters bundle that defines the
`ValueSet-reference` SearchParameter):

1. Change `SearchParameter.type` from `uri` to `canonical` for
   `ValueSet-reference`.
2. Replace the description with: *"All value sets on the server which
   reference a code system or value set included or excluded in the
   value set. It is up to the server whether to return information
   about the references to a code system or value set in included or
   excluded value sets."*
3. Replace the existing FHIRPath expression
   (`ValueSet.compose.include.system`) with the union of all four paths:
   `ValueSet.compose.include.system | ValueSet.compose.exclude.system | ValueSet.compose.include.valueSet | ValueSet.compose.exclude.valueSet`.
4. Update any narrative on `valueset-search.html` (the rendered
   `ValueSet-reference` row) so the published table reflects the new
   description, type, and expression.
5. Re-label the ticket from `TEXTONLY` to substantive when applied (per
   carmela_couderc's note), and treat it as a normative-affecting change
   in change-tracking.
6. Defer the parameter rename to a separate, later ticket. The current
   resolution does not include a rename.

#### Justification

This is the option the workgroup has already converged on, and the
reporter has explicitly accepted the broader scope (adding `valueSet`
paths) while deferring on `uri` vs `canonical`. The four-path expression
makes the implementation match the documentation in both directions
(include/exclude) and across the only two ValueSet composition
mechanisms (`system` and `valueSet`). Switching from `uri` to `canonical`
is the right type for a parameter that targets `CodeSystem.url` /
`ValueSet.url` (both canonicals); the existing `uri` typing has always
been over-broad for what's actually being matched. Deferring the rename
keeps backwards compatibility for every existing implementation and
client query — renaming a search parameter on a maturity-5 resource is a
breaking change that needs its own discussion and use-case justification
(see Disposition B).

The HL7/fhir FhirCore briefing flags two things worth respecting in the
edit: (a) only `<differential>` content in StructureDefinitions is
hand-edited — the search-parameters bundle is hand-edited XML, so the
edit applies cleanly; and (b) generated areas and the publisher harness
are out of scope. No publisher / build configuration changes are needed.

### Disposition B: Alternative Approach

#### Proposal

Apply the four-path expression and description rewrite from Disposition
A (which everyone agrees on), but **also resolve the rename now** rather
than punting it. Concretely:

1. Keep the existing `reference` parameter as a deprecated alias with
   the corrected description and expression, so existing clients keep
   working.
2. Add a new search parameter on ValueSet — proposed name
   `compose-reference` (clearer than the reporter's earlier
   `composeSystem`, since the expression now covers both `system` and
   `valueSet`) — typed as `canonical`, with the new description and the
   four-path expression.
3. Mark `reference` as deprecated in the SearchParameter narrative,
   pointing readers to `compose-reference`, and target removal in a
   future major release per the FHIR maturity / breaking-change rules.
4. File a follow-up ticket on the FHIR maintenance backlog to retire
   `reference` once its replacement reaches the same maturity level.

#### Justification

The reporter's *original* complaint had two parts: a documentation/
expression mismatch (which Disposition A fixes) and the parameter name
itself, which Disposition A simply defers. Doing the rename now — in a
backwards-compatible way via deprecation — closes the loop on the full
ticket, preserves every existing query (no breakage), and lets the
workgroup advertise the better name immediately rather than carrying a
known-bad name indefinitely. The trade-off is that introducing a second
search parameter on the same resource doubles the search-parameter
surface for ValueSet implementers and creates an explicit "which one do
I use" question that the WG would need to answer in narrative. There is
also no current Zulip / community discussion demanding the rename, so
the cost/benefit is real but not urgent.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as not actionable in its current form: the requested
behavioural change (have the parameter also match excluded systems and
valueSet imports) silently broadens the result set of every existing
`reference` query, and the requested rename is a breaking change. Close
as "Not Persuasive" and ask the reporter to refile two narrower tickets:

1. A documentation-only ticket clarifying that the existing parameter
   *only* matches `compose.include.system`, leaving the expression
   unchanged.
2. A separate ticket — with concrete use cases and consensus from
   implementers — proposing either a rename or an additional parameter
   for the broader semantics.

#### Justification

This option captures the conservative read: search parameter expression
changes alter query semantics for every existing client, even when no
schema element changes. An implementer who today depends on
`?reference=http://snomed.info/sct` returning only ValueSets that
**include** SNOMED would, after the change, also see ValueSets that
**exclude** SNOMED — which may be the documented intent but is, in
practice, a behavioural change. Declining keeps the published behaviour
stable until the WG has explicit feedback from terminology server
implementers (none surfaced in Zulip or related tickets) and forces the
naming question to come back with use cases.

---

### Recommendation

**Recommended disposition: A (Accept As Requested).**

Disposition A is the option the TI workgroup has already discussed twice
(2025-12-15 and 2026-04-13) and that the reporter has explicitly endorsed
on 2026-04-14, so it carries clear consensus. It corrects a real
inconsistency (the description and the FHIRPath disagree today, and that
disagreement is bug-shaped, not opinion-shaped), uses the correct
SearchParameter `type` (`canonical` matches `CodeSystem.url` /
`ValueSet.url`), and keeps the well-known parameter name `reference`
intact — preserving every existing query at every existing terminology
server. Disposition B's rename, while attractive in the abstract,
introduces churn (two parameters with overlapping semantics) without any
community pressure for it in the indexed Jira/Zulip/GitHub history;
better to defer to a dedicated rename ticket where the case can be made
on its own. Disposition C overweights the behavioural-change risk: the
current expression provably contradicts its own description, so any
implementer who relied on the narrow-include-only behaviour was relying
on a documented bug. The substantive-vs-text-only labelling concern
raised by the reporter should be honoured at apply time per
carmela_couderc — re-label the ticket to substantive so change tracking
reflects the real impact.
