## Summary

The reporter asks Terminology Infrastructure to retire the FHIR-Core‑maintained
"External Code Systems" listing on `terminologies-systems.html` (rendered as
§4.3.0.1 in the R6 ballot) because the same content is now authoritatively
maintained in HL7 Terminology (THO) under the "Using xxx with HL7 Standards"
guidance. Keeping a parallel list inside FHIR Core risks divergence and
double‑maintenance. Concretely, the ticket requests: (a) deletion of
§4.3.0.1's table of external code systems and the "Don't see what you're
looking for here?" pointer, (b) replacement of the surviving "subset of a
much larger list…" paragraph with a single pointer to terminology.hl7.org
that explicitly notes per‑system "Use with HL7 Standards" guidance is now
hosted there, and (c) tightening of the page's "Important Notes" — drop one
bullet and rename the tab reference from "Defined Externally" to "External
Content" in the bullet that survives. No comments, cross‑references, Zulip
discussions, or GitHub items were located via fhir-augury for this key.

## Details

**Description:**

The information in this section repeats content in THO. Now that the
"Using xxx with HL7 Standards" has moved to THO, there is no need to
maintain in two places.

Suggested changes:

- [Remove section 4.3.0.1 from this page.](https://hl7.org/fhir/6.0.0-ballot4/terminologies-systems.html)
- Remove the sentence: *Don't see what you're looking for here: See Registering Code Systems for further advice.*
- Change the existing text:
  - From: *This is a subset of a much larger list of known code systems, and only details code systems for which FHIR specific usage documentation is provided. For other code systems, see [terminology.hl7.org](https://terminology.hl7.org/external_code_systems.html).*
  - To: *For code systems defined external to HL7, see [terminology.hl7.org](https://terminology.hl7.org/external_code_systems.html). Some code system publishers have provided information about how to use their code system with HL7 standards. You will find that information at the same link.*
- Important Notes changes:
  1. Remove the first bullet.
  2. Change the second bullet:
     - From: *The external terminology content provided in the Defined Externally tab is provided for convenience and should not be considered authoritative nor normative.*
     - To: *The external terminology content provided in the External Content tab is provided for convenience and should not be considered authoritative nor normative.*

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir`, edit `source/terminologies-systems.html` to:

1. Delete the §4.3.0.1 block: the `<a name="external"> </a>` anchor on line 48, the `<h3>External Code systems</h3>` heading on line 49, and the entire `<table class="grid">` of external code systems that follows (through line ~141).
2. Replace the trailing paragraph (lines 143–147) with the wording specified in the ticket: *"For code systems defined external to HL7, see [terminology.hl7.org](https://terminology.hl7.org/external_code_systems.html). Some code system publishers have provided information about how to use their code system with HL7 standards. You will find that information at the same link."*
3. Delete the "Don't see what you're looking for here?" paragraph on line 45 (and its link to the *Request Content in External Terminologies* Confluence page).
4. In the "Important Notes" `<ul>`, drop the first bullet (line 35, "This list of names is incomplete and subject to change…") and re‑word the surviving bullet on line 36 to read *"…provided in the External Content tab…"* in place of *"…provided in the Defined Externally tab…"*.
5. Housekeeping required by the literal request but not spelled out in the ticket: remove the now‑orphaned duplicate `<a name="external"> </a>` anchor on line 150 (or move the surviving anchor onto the `Internal Code systems` heading deliberately), and rename the tab itself from "Defined Externally" to "External Content" wherever the tab label is defined (likely in the `txheader` / `tx-include` machinery referenced by `<%txheader systems%>`). Run a repo‑wide search for `terminologies-systems.html#external` and `terminologies-systems.html#http` before merging and update any callers that targeted rows in the deleted table.

#### Justification

The ticket's stated rationale is sound: the per‑system "Use with HL7 Standards" column is being maintained by THO going forward, and the FhirCore playbook for narrative edits in this repo is exactly "edit `source/*.html` directly" — there is no generated artifact to regenerate. Removing the duplicate listing eliminates a known divergence risk and centralises external‑terminology guidance with the work group (Terminology Infrastructure / Vocabulary) that owns it in THO. The wording change in the second bullet keeps the page consistent with the renamed tab, and dropping the first "incomplete and subject to change" bullet is reasonable now that the authoritative list lives elsewhere. The required anchor / tab‑label cleanup is a small but mandatory follow‑on that any reviewer will catch, and it is well within the scope a single workgroup vote can authorise.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the ticket's intent but stop short of deleting the §4.3.0.1 table outright in this ballot cycle. Specifically:

1. Replace the body of §4.3.0.1 with the new pointer paragraph from the ticket (text in Disposition A item 2), and convert the existing `<table class="grid">` into a short read‑only "Quick reference" table containing only the URI and a link‑out to the matching THO `external_code_systems.html#…` row — i.e., delete the *Source*, *Use with HL7 Standards*, and *OID* columns from FHIR Core and let THO own them. Keep the `<a name="external">` anchor and the heading so external deep links continue to resolve.
2. Apply the "Important Notes" edits exactly as in Disposition A (drop bullet 1, rename "Defined Externally" → "External Content" in bullet 2) and delete the "Don't see what you're looking for here?" paragraph.
3. File a follow‑on ticket against THO/UTG to confirm every URI currently listed on this page has a corresponding row in `terminology.hl7.org/external_code_systems.html` before the FHIR R7 cycle, then delete the residual table in R7.

#### Justification

The literal request is a one‑shot deletion of a long‑standing table that implementers and IGs have likely deep‑linked into for years (each row carries its own `<a name="…">` anchor, e.g., `terminologies-systems.html#http://snomed.info/sct`). Briefing‑level guidance for this repo flags exactly this class of breakage on the analogous `mapping-language.html#api` anchor cleanup. A two‑step migration — point at THO now, prune the residual stub table after THO has been verified to fully cover the inventory — preserves working deep links during R6 ballot, gives Vocab/TI a clean coverage check before deletion, and is materially less risky than an in‑ballot delete. The cost is one extra paragraph of stub content for one cycle.

---

### Disposition C: Decline

#### Proposal

Reject the deletion request. Keep §4.3.0.1 and its table on `source/terminologies-systems.html`, on the basis that (a) FHIR Core has historically been the canonical place implementers find the HL7‑endorsed *URI* for external code systems used in FHIR, (b) the ticket itself acknowledges THO is the authoritative source while still asking FHIR Core to delete the table that names the URIs implementers paste into `Coding.system`, and (c) no Zulip thread, no other Jira ticket, and no GitHub PR were found that demonstrate broader community agreement to remove the listing in the R6 ballot. Apply only the smallest, lowest‑risk subset of the request: delete the "Don't see what you're looking for here?" paragraph (line 45) and re‑word the second "Important Notes" bullet to use "External Content tab" once the tab is actually renamed, but leave the §4.3.0.1 heading and table in place pending broader discussion.

#### Justification

The current page is more than a duplicate of THO: each row pins an HL7‑endorsed FHIR URI to a specific external code system, and those URIs are normative inputs to `Coding.system` across the entire ecosystem. Moving them to THO is reasonable in principle but not without a documented migration plan and a clean deep‑link preservation story; cross‑references for this ticket returned zero hits, so the workgroup has no community signal to act on at the scale of an outright deletion. Declining at this time keeps implementers' deep links working during R6 ballot and forces the conversation back to a properly scoped TI / Vocab discussion that pulls in THO maintainers.

---

### Recommendation

**Recommended disposition:** B

The reporter is right that maintaining the per‑system "Use with HL7 Standards" guidance in two places is a divergence risk, and the FhirCore playbook clearly supports a direct `source/*.html` edit — Disposition A is mechanically straightforward. However, the literal deletion in A would silently break per‑URI deep links of the form `terminologies-systems.html#http://snomed.info/sct` that downstream IGs and implementer documentation are very likely to have used, and would also orphan the `#external` fragment because of the duplicate `<a name="external">` anchor on line 150. The briefing for this repo specifically warns about this exact anchor‑cleanup hazard for analogous edits. Disposition C over‑corrects in the other direction: the underlying request is sound and the change really does belong in this work group's lane. Disposition B accepts the editorial direction the reporter is pushing for — pointer paragraph, tab rename, "Important Notes" tightening — while keeping a one‑cycle stub that preserves anchor stability and gives Terminology Infrastructure and THO maintainers a chance to verify URI coverage in `terminology.hl7.org/external_code_systems.html` before the table is fully retired in R7.
