## Summary

The reporter asks the CRMI IG to extend the discussion around `ManifestParameters`
and the `ManifestLibrary` profile so that it references a forthcoming
`expansion-parameters` extension (to live in the FHIR Extensions Pack). That
extension is intended to record *the parameters that were actually used to
expand the value sets in the IG at publication time* — i.e., an "as-published"
record of expansion inputs. The ticket carefully distinguishes this new
extension from two adjacent, already-defined extensions:

- `cqf-expansionParameters` — parameters that *should* be used when expanding
  value sets in the IG (a *recommendation* to consumers).
- `cqf-inputParameters` — input expansion parameters used as input to the
  release / publication process.

The ticket has been Triaged with a `Persuasive` resolution: agree to update the
"Canonical Resolution with a Manifest" section so that `cqf-expansionParameters`
SHOULD be used to convey expansion defaults, and `expansion-parameters` MAY be
used (when `cqf-expansionParameters` is absent) to convey the as-published
expansion parameters. A clarifying note about the semantic difference between
the two extensions is also part of the proposed wording.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
recorded in the FhirAugury index for this ticket.

## Details

**Description:**

> Add to the ManifestParameters discussion and the ManifestLibrary profile a
> reference to the new "expansion-parameters" extension that will be in the
> extensions pack and is used to indicate "the parameters that were used to
> expand value sets in the IG"
>
> This is different than the cqf-expansionParameters extension, which is
> specifically "the expansion parameters that should be used to expand value
> sets in the IG", and different again from the "cqf-inputParameters"
> extension, which identifies the "input expansion parameters used as input to
> the release/publication process"

**Resolution Description (Persuasive, recorded 2026-02-09):**

Agreed, update the language in the
[Canonical Resolution with a Manifest](https://build.fhir.org/ig/HL7/crmi-ig/version-manifest.html#canonical-resolution-with-a-manifest)
section.

- *Current:* "In addition, if the operation specifically involves value set
  expansion, the expansionParameters extension of the manifest library can be
  used to provide values for the parameters to the expand (such as activeOnly
  and system-version)."
- *New:* "In addition, if the operation specifically involves value set
  expansion, the cqf-expansionParameters extension of the manifest library
  SHOULD be used to provide values for the parameters to the expand (such as
  activeOnly and default-system-version). If no cqf-expansionParameters is
  present, expansion-parameters MAY be used to provide the 'as-published'
  expansion parameters."
- *Note:* `cqf-expansionParameters` defines default values that SHOULD be used
  for `$expand` input parameters. `expansion-parameters` defines the parameter
  values that were used during the publication expansion process.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply exactly the wording change captured in the resolution to the
"Canonical Resolution with a Manifest" section of the CRMI IG
(`version-manifest.html`):

1. Replace the current sentence about `expansionParameters` with the new
   two-sentence form that:
   - States `cqf-expansionParameters` SHOULD be used to provide expand
     parameters (default-system-version, activeOnly, …).
   - States `expansion-parameters` MAY be used (only when
     `cqf-expansionParameters` is absent) to convey the "as-published"
     expansion parameters.
2. Add the clarifying note distinguishing the *should-be-used* semantics of
   `cqf-expansionParameters` from the *was-actually-used* semantics of
   `expansion-parameters`.
3. Add a corresponding bullet to the `ManifestLibrary` profile description
   listing `expansion-parameters` alongside `cqf-expansionParameters` and
   `cqf-inputParameters` so all three extensions are documented in one place.
4. Defer merging until the `expansion-parameters` extension actually exists in
   the FHIR Extensions Pack (per the 2026-02-23 comment), and link the IG text
   to its canonical URL once published.

#### Justification

This is a pure documentation / profile-wording change in the CRMI IG; the
resolution is already recorded as Persuasive and the exact text is agreed.
Adding the third extension closes a real semantic gap — there is currently no
sanctioned way to publish "what expansion parameters were *actually* used" as
distinct from "what should be used" or "what was input to the release
process". The SHOULD/MAY layering preserves the primacy of
`cqf-expansionParameters` for consumer-facing behaviour while letting
publishers record provenance. Carmela's comment is naturally accommodated by
gating the merge on the extension's publication, not by changing the
disposition.

---

### Disposition B: Alternative Approach

#### Proposal

Take the same wording change in spirit, but consolidate the three
expansion-parameter extensions into a single, structured discussion in the
CRMI IG (a small "Expansion Parameters in a Manifest" subsection) that
explicitly tabulates:

| Extension | Semantics | Conformance |
|-----------|-----------|-------------|
| `cqf-expansionParameters` | Defaults that SHOULD be used by consumers when `$expand`-ing | SHOULD |
| `expansion-parameters` | The parameters that *were* used at publication time | MAY (when `cqf-expansionParameters` absent) |
| `cqf-inputParameters` | Inputs to the release/publication process | informative |

The `ManifestLibrary` profile would then point readers at this single section
rather than embedding the rules inline in "Canonical Resolution with a
Manifest". Implementation again waits for `expansion-parameters` to be
published in the Extensions Pack.

#### Justification

The three extensions are easy to confuse — the ticket itself spends most of
its body distinguishing them. A short table in one place is significantly
easier for implementers to reason about than three sentences scattered across
two contexts. The trade-off vs. Disposition A is a slightly larger edit
surface (a new subsection plus cross-references) and a small amount of
re-flow in the existing narrative; the upside is a durable reference that
reduces future tickets of the same shape.

---

### Disposition C: Decline

#### Proposal

Do not adopt the change at this time. Rationale category: *premature —
referenced artifact does not yet exist*. Close the ticket as "Not
Persuasive — Defer" and re-open (or file a fresh ticket) once the
`expansion-parameters` extension is actually published in the FHIR
Extensions Pack and has a stable canonical URL.

#### Justification

The 2026-02-23 comment from carmela_couderc records that the
`expansion-parameters` extension does not yet exist. Publishing CRMI IG text
that references a non-existent canonical URL would create a dangling reference
in a normative-track IG, with no way for implementers to resolve it. Declining
(or at minimum deferring) protects the IG from shipping an unresolvable
pointer and forces the dependency ordering (publish the extension first, then
reference it) that the workgroup already informally agreed to.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already taken the substantive decision (Resolution:
Persuasive) and the wording is fully specified. The only real concern raised
in discussion — that the `expansion-parameters` extension does not yet exist —
is a *sequencing* problem, not a *correctness* problem, and is best handled by
holding the merge of the IG change until the Extensions Pack publishes the
extension, rather than by reopening the wording debate. Disposition B is
attractive as a follow-up cleanup but is larger than what was agreed and
risks re-litigating settled text; it would be better filed as a separate
editorial ticket if the workgroup wants the consolidated subsection.
Disposition C overreacts to a fixable sequencing issue.
