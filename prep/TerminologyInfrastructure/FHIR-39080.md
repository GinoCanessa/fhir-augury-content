## Summary

The VHIE Clinical Data Quality Team (via reporter Ioana Singureanu) is asking
that FHIR provide a standardized way for a terminology server to declare its
support for implicit value sets — i.e., the URL pattern that lets a single URI
identify a value set defined by all (or a subset of) the codes in a given code
system. Today, an implementing client cannot reliably discover, from a
server's published metadata, whether that server supports the implicit
value-set URL pattern for any given code system, nor what the URL pattern is.
The reporter's recommendation is to surface this capability somewhere
machine-readable (the discussion has converged on `TerminologyCapabilities`)
so that human implementers and automated terminology clients (including the
Tx Registry) can determine which servers can expand which implicit value
sets.

There are no Jira tickets explicitly linked to or cross-referenced from this
ticket; the only cross-reference Augury surfaced is to the FHIR artifact
`CodeSystem.valueSet`, which is the existing element implementers reach for
today and which the discussion has already noted is an inadequate substitute
(see Peter Jordan's comment).

## Proposed Dispositions

The dispositions below all touch the FHIR Core specification — specifically
the `TerminologyCapabilities` resource and the surrounding "implicit value
sets" narrative on the `valueset.html` page. Because no GitHub
cross-references were produced, no per-repo briefing is available; any
disposition that proceeds with a code/spec change would touch the FHIR Core
authoring root for `TerminologyCapabilities` (resource definition + fhirpath
invariants + StructureDefinition) and the `valueset.html` narrative page.
The TI WG should confirm those paths against the current `HL7/fhir`
briefing before opening a PR.

### Disposition A: Accept As Requested

#### Proposal

Add a new repeating element to `TerminologyCapabilities.codeSystem` that
declares implicit value-set support for that code system. Concretely:

- `TerminologyCapabilities.codeSystem.implicitValueSet` (0..\*) — a
  BackboneElement with:
  - `uriPattern : uri` (1..1) — the URL pattern the server recognises (e.g.,
    `http://snomed.info/sct?fhir_vs=ecl/...`),
  - `description : markdown` (0..1) — human-readable description of what
    the pattern selects,
  - `documentation : url` (0..1) — link to vendor / IG documentation for
    the pattern.
- Add a top-level boolean
  `TerminologyCapabilities.implicitValueSet : boolean` (0..1) to declare
  blanket support for the generic implicit value-set mechanism across any
  code system the server hosts (covers Carmela's first discovery
  question: "any code system?").
- Update the `valueset.html` "Implicit Value Sets" narrative to point at
  these new elements as the discovery surface.

#### Justification

This directly addresses the three discovery questions Carmela enumerated on
2023-08-21 (any code system, specific code systems, the URL pattern) and
the September 2024 joint TI / FHIR-I conclusion that "there is a place to
support this feature, but it doesn't exist yet." It lands the change in
`TerminologyCapabilities`, where the WG has already converged it should
live (2024-07-29, 2024-07-25 comments), and explicitly captures what
`CodeSystem.valueSet` cannot — multiple URL patterns per code system, plus
the fact that `CodeSystem.valueSet` is not guaranteed to even *be* an
implicit value set (Peter Jordan, 2024-07-29). It is also additive
(0..\* / 0..1), so it is non-breaking for R6 — consistent with the R6
label on the ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than modelling URL-pattern strings directly, adopt the
2024-09-26 joint-meeting suggestion: define a *canonical identifier* for
each implicit value-set pattern (independent of any specific server) and
have `TerminologyCapabilities` reference those identifiers.

Concrete shape:

- Publish (in FHIR Core or a TI-owned IG) a registry of named implicit
  value-set patterns — e.g., `loinc-answer-list`,
  `snomed-ecl`, `snomed-isa`, `cpt-modifier-set` — each with a canonical
  URL, the URL-pattern template, and a description.
- Add `TerminologyCapabilities.codeSystem.implicitValueSet : canonical[]`
  (0..\*) referencing entries in that registry.
- Optionally allow servers to extend with vendor-specific patterns via a
  fallback `extension` slot.

#### Justification

This responds directly to the 2024-09-26 discussion that "we have a URL
pattern now, not an identifier" and that the answer "should not be server
specific." It separates the *capability catalogue* (community-owned,
versionable, citable) from the *server claim* (lightweight reference),
which makes Tx Registry discovery (mjlawley's question, 2024-07-23) a
trivial filter rather than a string-pattern match. Trade-off vs. A: it
requires standing up and maintaining a registry, which is a larger
commitment than just adding two elements to `TerminologyCapabilities`,
and a registry-bootstrap is itself out of scope for a single
`TerminologyCapabilities` change request — so feasibility hinges on TI
being willing to sponsor that catalogue.

---

### Disposition C: Decline

#### Proposal

Decline the change request as currently scoped, on the grounds that
between 2023-08-21 and the latest 2025-01-29 update the reporter has not
supplied the concrete client/server use cases the WG asked for three
times, and that `CodeSystem.valueSet` plus existing
`TerminologyCapabilities.expansion`/`codeSystem` declarations already
provide a partial discovery surface for the most common case. Recommend
the reporter resubmit (or the WG re-open) once the use cases for
*automated* client behaviour are documented.

#### Justification

Carmela explicitly warned on 2024-02-12 that without engagement the WG
would ask for the ticket to be withdrawn; the use-case detail mjlawley
asked for in 2024-07-23 was never supplied; and Peter Jordan's question
("what does this tell a client beyond `CodeSystem.valueSet`?") has not
been answered with a concrete scenario. Without that, any modelling
choice between A and B is guesswork, and adding optional metadata that
no implementer has committed to populating is the kind of churn TI has
historically tried to avoid in `TerminologyCapabilities` (already noted
to be a relatively immature resource — JIRAUSER28585, 2024-07-25).

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right starting point. The WG has already done the
analysis: the joint TI / FHIR-I meeting in September 2024 concluded that
the feature belongs somewhere in `TerminologyCapabilities`, the
2024-07-29 recap ruled out `CodeSystem.valueSet` as a substitute, and the
three discovery questions Carmela enumerated in 2023-08-21 map cleanly
onto the proposed `implicitValueSet` boolean (any-code-system) plus the
per-codeSystem `implicitValueSet` backbone (specific code systems plus
URL pattern). Disposition B is conceptually cleaner but materially
larger — standing up a community-owned registry of implicit-value-set
patterns is a separate body of work and a separate ticket, and blocking
A on B leaves implementers (including the Tx Registry use case mjlawley
flagged) with no discovery surface at all in R6. Disposition C is
defensible procedurally, but the WG's own discussion has already
substantively answered the questions that justified the original
"insufficient use case" concern, so declining now would discard that
work. Recommend Accept (A) for R6, with a follow-up ticket against TI to
explore the registry pattern from B as an evolutionary step in a later
release.
