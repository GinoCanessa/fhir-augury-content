## Summary

The submitter, Bas van den Heuvel, observes that a substantial set of
ValueSets and CodeSystems used by extensions in the **FHIR Extensions
Pack** are also defined in **HL7 Terminology (THO)**, but under
different canonical URLs (e.g., `http://hl7.org/fhir/ValueSet/implantStatus`
in the Extensions Pack vs. `http://terminology.hl7.org/ValueSet/implantStatus`
in THO). The same pattern holds for the corresponding CodeSystems. The
ticket asks the project to pick a single canonical home for each
artifact, deprecate the duplicate where one is retained for historical
reasons, and otherwise align URLs and content so that implementers do
not face two divergent definitions of the same terminology.

The follow-up work has been consolidated into two related tickets:
[FHIR-47498](https://jira.hl7.org/browse/FHIR-47498) — opened by
Jessica Bota two days after FHIR-47270 — catalogues the duplicate set
(14 CodeSystems and 14 ValueSets identified by resource id) in an
attached spreadsheet, and
[FHIR-55033](https://jira.hl7.org/browse/FHIR-55033) — opened by
Reuben Daniels in January 2026 — replaces both as the active workstream
and tracks per-artifact resolutions in a shared spreadsheet. The
Resolution Description on FHIR-47270 explicitly closes this ticket as
**Persuasive**, redirecting any remaining work to FHIR-55033.

## Details

**Description:**

There are many extensions in the extension pack use coded values. The
binding of the values refers to ValueSets. The codes in these ValueSets
refer to codes from CodeSystems.

When accessing the html/json/xml of these ValueSets and CodeSystems it
returns html pages and definition that are stored in the extension pack.

The same code systems and value sets are also defined in THO.

The urls used for the ValueSets and CodeSystems in THO are different
from the ones used in the extension pack.

E.g. ValueSet `ImplantStatus`

- Extension pack: <http://hl7.org/fhir/ValueSet/implantStatus>
- THO: <http://terminology.hl7.org/ValueSet/implantStatus>
- CodeSystem `ImplantStatus`
- Extension pack: <http://hl7.org/fhir/implantStatus>
- THO: <http://terminology.hl7.org/CodeSystem/implantStatus>

Is this correct (I assume not)? FHIR should decide on one location for
these ValueSets and CodeSystems and should not duplicate them. If one
of these is there due to historical reasons, it should be clearly
marked as deprecated. If there is a good reason to duplicate them, then
the URLs and content should be aligned.

**Resolution Description:**

Many changes have already been applied — close this ticket as the
changes listed in
[FHIR-55033](https://jira.hl7.org/browse/FHIR-55033) address the
remaining issues.

## Proposed Dispositions

### Disposition A: Accept As Requested (Persuasive — close, defer to FHIR-55033)

#### Proposal

Resolve FHIR-47270 as **Persuasive** and close it, recording that the
underlying problem (duplicated terminology between the FHIR Extensions
Pack and THO with divergent canonical URLs) has been accepted as a real
defect and is being addressed under
[FHIR-55033](https://jira.hl7.org/browse/FHIR-55033), which carries
the consolidated duplicate inventory and the per-artifact resolution
plan tracked in the shared Google Sheet. Apply the same disposition
text used by Carmela Couderc on FHIR-47498 ("Many changes have already
been applied — close this ticket as the changes listed in FHIR-55033
address the remaining issues."). No spec-level change is made by this
ticket; the per-artifact URL/master-copy decisions live with FHIR-55033
and the FMG Terminology Expectations policy update.

#### Justification

This is the path the work group has effectively already chosen: both
this ticket and FHIR-47498 carry the `DupResources` label, the
**Persuasive** resolution is already recorded, and Carmela and Jessica
explicitly proposed to close both in favour of FHIR-55033 (see
FHIR-55033 comments 2026-03-16 and 2026-03-17). Closing here keeps
ownership of the substantive cleanup in one place — important because
each duplicated artifact requires a per-artifact decision per Grahame's
position in the Zulip thread ("each case has to be investigated with
the author") and per Lloyd's WGM rule recorded on FHIR-47498 ("things
bound as code should live in the extension pack; things not bound as
code should live in THO"). Splitting the work across three tickets
risks divergent decisions on the same artifact.

---

### Disposition B: Persuasive With Mod — capture the policy and the gap before closing

#### Proposal

Close FHIR-47270 as **Persuasive With Modification**, but only after
adding two specific outputs to the resolution comment so the close does
not lose information:

1. **Policy reference.** Cite the FMG Terminology Expectations policy
   rule the subgroup is applying (Lloyd McKenzie, WGM Sep 2025
   Wednesday Q1, recorded on FHIR-47498): "things bound as code should
   live in the extension pack; things not bound as code should live in
   THO." Make this the explicit decision rule that FHIR-55033 enforces.
2. **Gap list.** Carry forward the artifacts Bas listed in the
   `fhir/infrastructure-wg` Zulip thread that Carmela noted are *not*
   on the FHIR-47498 spreadsheet — at minimum
   `CodeSystem/pronouns` and `ValueSet/name-assembly-order` — into
   FHIR-55033's tracking sheet so they are not silently dropped when
   FHIR-47498 and FHIR-47270 close.

Then redirect remaining work to FHIR-55033 as in Disposition A.

#### Justification

Disposition A is correct but loses two things on close that the
submitter and the discussion explicitly surfaced: the *rule* that
governs which side wins, and a known *delta* between the catalogues.
The FHIR-47498 attached spreadsheet was generated by id matching, while
Bas's Zulip enumeration was suffix matching — Carmela documented at
least two artifacts visible to the latter but not the former. If
FHIR-55033 inherits only FHIR-47498's spreadsheet, those artifacts
remain duplicated. This disposition costs minutes in the resolution
comment but eliminates the gap and makes the policy citable from the
ticket itself, which helps R6 QA reviewers and ballot voters who land
on FHIR-47270 from a search.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Decline FHIR-47270 on the grounds that the request — "FHIR should
decide on one location for these ValueSets and CodeSystems" — is too
broad to act on as a single change request, and the duplicates the
submitter listed are already being addressed individually under
FHIR-47498 and FHIR-55033 with different (per-artifact) outcomes.

#### Justification

Defensible only as a procedural close: the ticket as written asks for a
blanket policy plus 28+ individual artifact moves, which is not a
shape Jira tickets resolve well, and the substantive work is in fact
happening elsewhere. However, this framing throws away the value of
having a clearly-marked submitter complaint (a non-HL7 implementer
hitting two divergent definitions of the same terminology) and risks
implying the WG does not regard the underlying problem as real — which
it clearly does, given the Persuasive resolution already on record and
the active FHIR-55033 workstream. Recommended only if the WG wants to
draw a hard line that FHIR-47270 was a bug report, not a change
request.

---

### Recommendation

**Recommended disposition:** B

Disposition A reflects the WG's stated direction and is the path of
least resistance, but the close as currently drafted on FHIR-47270 and
FHIR-47498 references FHIR-55033 without citing the policy rule that
makes the cleanup possible (Lloyd's "bound-as-code → extension pack;
not bound-as-code → THO") and without acknowledging the artifact gap
Carmela explicitly flagged on the Zulip thread (`pronouns`,
`name-assembly-order`). Disposition B is a small superset of A — it
still closes here and redirects to FHIR-55033 — but it preserves the
two pieces of context the conversation produced. That matters because
FHIR-55033's analysis lives in a Google Sheet rather than in Jira;
without those artifacts being explicitly added, the next person who
re-discovers `pronouns` will file a fourth duplicate ticket. Disposition
C is not justified given the work group's prior Persuasive vote and
the active follow-on workstream.
