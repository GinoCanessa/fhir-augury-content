## Summary

KyleBennett requests two clarifications to the guidance in the FHIR Core
terminology page (section 4.1.4.2.1, "Required binding strength") for
elements typed `code`:

1. The current guidance describes the exception case using an
   "`additional` binding with `purpose = maximum`". Because
   [FHIR-46350](https://jira.hl7.org/browse/FHIR-46350) deprecated the
   `maximum` additional-binding purpose in favor of placing the
   `required` binding directly on `binding.strength` (and added a best-
   practice warning when a `required` additionalBinding is used without
   `usage`), the example in 4.1.4.2.1 now references a deprecated
   construct. The reporter asks that the guidance use a non-deprecated
   binding shape.
2. The same paragraph states that exceptions "need to be approved by
   the FHIR-I work group which will document the patterns that qualify
   as exceptions," but those documented patterns are not linked from
   4.1.4.2.1. The reporter asks that they be linked (or, per esilver's
   comment, that the text simply enumerate the qualifying patterns
   rather than describing the internal HL7 process).

The change is editorial/clarification scope on the terminology page;
the underlying normative rule (required binding strength SHALL be used
for `code`-typed elements with documented exceptions) is not in
dispute.

## Details

**Description:**

> In the guidance for required binding strength 4.1.4.2.1 it is noted
> that "Required binding strength SHALL be used for elements with type
> code (unless the element has a context-free (i.e., no
> binding.additional.usage present) [additional binding](https://build.fhir.org/elementdefinition-definitions.html#ElementDefinition.binding.additional)
> with purpose = maximum, a combination which is strongly discouraged
> as of R6)". However, as of [FHIR-46350](https://jira.hl7.org/browse/FHIR-46350)
> the maximum binding strength has been deprecated in favor of
> required. Although this is a discouraged combination, it would make
> sense to use a non-deprecated binding.
>
> Additionally, after that it is noted "Exceptions to this rule need to
> be approved by the FHIR-I work group which will document the patterns
> that qualify as exceptions." It would be helpful for those exceptions
> to be linked here.

**Comments:**

- **esilver (2026-01-28):** Suggests internal HL7 governance ("approved
  by FHIR-I") should not appear in the standard text. Recommends
  rewriting as a direct enumeration:
  > Exceptions to this rule are:
  > - \<pattern A\>
  >
  > rather than describing the approval process.

## Related Jira Tickets

### [FHIR-46350: Bindings for when using Common/Max should be consistent](https://jira.hl7.org/browse/FHIR-46350)

Applied change (resolution: Not Persuasive with Modification, FHIR
Infrastructure WG) that established the current R6 policy for stacking
required, maximum, and preferred/starter bindings. The applied
resolution: (1) deprecates the `maximum` use of
`ElementDefinition.binding.additional` because it is equivalent to
placing `required` on `binding.strength`; (2) adds a best-practice
warning when an `additionalBinding` carries `use = required` with no
`usage` while the primary `binding` is not required ("Required
bindings SHOULD appear as the primary binding"); and (3) adds a note in
`terminology.html#bindings` about multiple required bindings to
non-overlapping value sets when the type is `CodeableConcept` or
`CodeableReference`. This is the exact change that left the
"additional binding with purpose=maximum" wording in section 4.1.4.2.1
out of date, and it is the source of the inconsistency FHIR-54009
flags.

Users involved: Gino Canessa (reporter), Lloyd McKenzie (mckenzie),
Josh Mandel (jmandel), Grahame Grieve, Corey Spears, Jeff Brown.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update `terminology.html` section 4.1.4.2.1 ("Required binding
strength") on the FHIR Core spec to:

1. Replace the parenthetical exception example so it no longer
   references the deprecated `maximum` additional-binding purpose.
   Restate the exception in terms of the post-FHIR-46350 model — i.e.,
   describe it as an element whose primary `binding.strength` is
   `required` to a maximal value set together with a context-free
   (no `binding.additional.usage`) `additional` binding of a weaker
   strength (e.g., `preferred` / `starter`) — matching the pattern
   FHIR-46350 endorsed.
2. Add an explicit, inline list of (or hyperlinks to) the exception
   patterns that have already been approved by FHIR-I (e.g., the
   `Resource.language` / All-Languages + Common-Languages pattern
   called out in FHIR-46350), so readers of 4.1.4.2.1 do not have to
   chase governance minutes to know which patterns qualify.

Source-of-truth file: `source/terminology.html` (or the equivalent
markdown/page source) on `HL7/fhir`. No schema, value-set, or
StructureDefinition changes are required.

#### Justification

The reporter has correctly identified that 4.1.4.2.1 cites a deprecated
construct. FHIR-46350 (Applied) made `maximum` redundant with a
`required` primary binding and added a best-practice warning steering
authors away from it; continuing to teach the exception in those
deprecated terms is actively misleading. Inline linking of the
approved exception patterns also addresses esilver's reasonable
concern that internal HL7 process language ("need to be approved by
FHIR-I") does not belong in the normative text — readers want to know
which patterns are allowed, not how the WG decides.

---

### Disposition B: Alternative Approach

#### Proposal

Make the same factual fix as Disposition A (drop the deprecated
`maximum` example, add the up-to-date exception shape), but, per
esilver's comment, restructure the paragraph to **enumerate** the
exception patterns directly rather than linking out:

> Required binding strength SHALL be used for elements with type
> `code`. The following patterns are the only permitted exceptions:
>
> - **Pattern 1:** *(e.g., `Resource.language`-style — a `required`
>   binding to a maximal value set plus a context-free `additional`
>   binding of weaker strength)*
> - **Pattern 2:** *(any further FHIR-I-approved patterns)*
>
> Any new exception requires explicit approval from FHIR Infrastructure
> (FHIR-I) and an update to this list.

This keeps the governance constraint visible (so new exceptions still
require FHIR-I sign-off) but moves the *current* approved patterns
into the spec text itself. Coordinate with FHIR-I to confirm the
authoritative list before publishing; Terminology Infrastructure
should not enumerate patterns FHIR-I has not approved.

#### Justification

esilver's point is well-founded: the standard should describe what
implementers must do, not how a work group governs itself. Enumerating
the patterns in-line is more useful to spec readers than a hyperlink
into governance artifacts that may move. The trade-off versus
Disposition A is maintenance burden: every newly-approved exception
requires a spec edit instead of just adding a row to the linked list.
Because new exceptions are intentionally rare (they require FHIR-I
approval), that maintenance cost is small and acceptable. This
approach also lets Terminology Infrastructure correct the
`maximum`-binding wording immediately while inviting FHIR-I to ratify
the enumerated list during disposition.

---

### Disposition C: Decline

#### Proposal

Decline as already-addressed clarification: rule that the "exception"
paragraph in 4.1.4.2.1 will be *removed* (rather than rewritten),
because the FHIR-46350 best-practice warning on `ElementDefinition`
already steers authors to put `required` on the primary `binding`,
and the `terminology.html#bindings` text added by FHIR-46350 already
covers multiple required bindings for `CodeableConcept` /
`CodeableReference`. Under this disposition, 4.1.4.2.1 would simply
say required bindings SHALL be used for `code`-typed elements, and
direct readers to `terminology.html#bindings` for the post-R6
binding-stacking guidance.

#### Justification

This is defensible if the WG views the "exception" sentence as a
historical artifact that survived FHIR-46350's edit pass and no
longer carries any rule that isn't already covered elsewhere. It
avoids re-litigating which patterns count as exceptions and keeps
4.1.4.2.1 minimal. The downside is real: if there *are* legitimate
patterns where a `code`-typed element does not get a `required`
primary binding, removing the exception language without a
replacement leaves implementers with no spec-level cover for those
patterns, and tooling enforcing 4.1.4.2.1 strictly could raise false
positives.

---

### Recommendation

**Recommended disposition:** B

Disposition B fixes the concrete defect the reporter identified
(stale reference to a deprecated construct), addresses esilver's
substantive comment (don't bake HL7 governance language into normative
text), and produces a section that is genuinely useful to spec
readers rather than a pointer chain into governance minutes.
Disposition A is a strict superset of the literal request and is a
fine fallback if Terminology Infrastructure cannot get an
authoritative exception list from FHIR-I in time, but any rewrite
that keeps the "approved by the FHIR-I work group" phrasing leaves
the second half of esilver's concern unaddressed. Disposition C
should be declined unless FHIR-I confirms there are zero qualifying
exception patterns under the post-FHIR-46350 model — given that
FHIR-46350 itself was triggered by `Resource.language`'s
required-to-maximal pattern, that confirmation seems unlikely.
Because the exception list ultimately belongs to FHIR Infrastructure
even though the page lives in Terminology Infrastructure's WG, this
ticket should be progressed in coordination with FHIR-I (cf. the
FHIR-46350 history on the same paragraph).
