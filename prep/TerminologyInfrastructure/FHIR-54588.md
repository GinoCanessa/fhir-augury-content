## Summary

FHIR-54588 asks the Terminology Infrastructure work group to disentangle two
concepts that the R6 ballot text for **Additional Bindings** currently
intermingles in the same narrative and table: **binding strength** (`required`,
`extensible`, `preferred`, `example` — which drive conformance and validator
behavior) and **binding purpose** (e.g., `component`, `minimum`, `current`,
`ui`, `starter` — which describe *why* an additional value set is supplied and
how it is intended to be used). The ticket argues these are orthogonal axes,
that conflating them produces inconsistent IG modeling, unclear validator
behavior when additional bindings are layered on top of extensible base
bindings, and implementer confusion about whether a given additional binding is
enforceable, informational, or application-level guidance. The requested change
is to revise the Terminology Bindings section so that strength and purpose are
clearly separated (e.g., distinct tables or subsections) and to state
explicitly that they are independent concepts, with their relationships called
out where they exist.

The ticket explicitly cites **[FHIR-54128](https://jira.hl7.org/browse/FHIR-54128)
— "Additional Bindings Documentation is under-specified"** (reporter: Eric
Haas; assignee: Grahame Grieve; status: *Waiting for Input*) as a "see also"
dependency. FHIR-54128 takes the same observation further and proposes that
*for each* purpose code the spec should define (a) whether it must be a
superset, subset, or disjoint from the main binding, (b) which main bindings it
can be combined with, (c) general guidance on its use, and (d) examples
(reporter has stubbed a HackMD draft). The 2026-03-23 TI tracker call
discussed both tickets together; the WG agreed FHIR-54588's resolution is
dependent on the broader clarification work being scoped under FHIR-54128 and
that Grahame Grieve will start a Zulip discussion to converge on what should
land in the spec, including support for the US Core extensible-binding pattern.

## Details

**Description:**

> The current R6 ballot text describing **additional bindings** conflates two
> distinct concepts:
>
> 1. **Binding strength** (required / extensible / preferred / example), which
>    defines conformance and validation behavior, and
> 2. **Binding purpose** (e.g., component, minimum, current, ui, starter),
>    which describes why an additional value set is supplied and how it is
>    intended to be used.
>
> These concepts are orthogonal, but are currently intermingled in the same
> narrative and table, making it difficult for IG authors, implementers, and
> tooling developers to understand which aspects affect conformance versus
> documentation or guidance.
>
> **Why this matters.** The lack of clear separation leads to:
>
> - inconsistent modeling patterns across IGs,
> - unclear validator behavior when additional bindings are layered on top of
>   extensible base bindings, and
> - implementer confusion about whether a given additional binding is
>   enforceable, informational, or application-level guidance.
>
> **Requested change.** Please revise the Terminology Bindings section to:
>
> - explicitly separate **binding strength** from **binding purpose** (e.g.,
>   distinct tables or clearly delineated subsections), and
> - state clearly that binding strength and binding purpose are independent
>   concepts, with defined relationships where applicable.
>
> See Also: <https://jira.hl7.org/browse/FHIR-54128>

**Comments:**

- **carmela_couderc — 2026-03-23 (TI Tracker Call notes):**

    represent binding strength; purpose is separate from binding strength,
    though there is an interaction.
  - The table is manually authored.
  - In §4.1.4.2 Binding Strengths, conformance applies only to `required` and
    `extensible`; the other "strengths" in that table are documentation-only.
  - **Agreement:** rename the table heading to reflect that the rows are not
    all binding strengths, and add information indicating where the

  - Some purpose values overlap; a free-text comment alongside `purpose` may be
    needed so IG authors can explain *why* they chose a particular purpose.
  - Interplay between primary binding and additional bindings can produce
    nonsense combinations; the spec should describe relationships and impact,
    or clearly state when they are independent.
  - Discussion drifted into specific binding strengths/expectations.
  - **The change to be proposed for this ticket is dependent on FHIR-54128.**

## Linked Jira Tickets

### [FHIR-54128: Additional Bindings Documentation is under-specified](https://jira.hl7.org/browse/FHIR-54128)

The "See Also" referenced by FHIR-54588. Argues that the current Additional
Bindings prose (in `terminologies.html#binding`, `terminologies.html#strength`,
the `additional-binding-purpose` value set, and the `ElementDefinition.binding.additional`
definitions) does not provide rules and guidance sufficient for consistent
interpretation, and proposes that *for each* purpose code the spec should
specify whether the value set is a superset/subset/disjoint from the main
binding, which main bindings it can be paired with, general guidance, and
examples. The 2026-03-23 TI Tracker Call agreed (a) to take the conversation to
Zulip (Grahame), (b) that the US Realm pattern for extensible additional
bindings is not yet supported but could be, (c) that one Coding can satisfy
multiple additional bindings to the same/overlapping value sets, and (d) that
extensible/preferred strengths cannot be tested for conformance. *Status:
Waiting for Input — not ready for proposed resolution.* FHIR-54588's structural
"separate strength from purpose" change is treated as a sub-deliverable of the
broader clarification scope tracked here.

Reporter: Eric Haas. Assignee: Grahame Grieve. Commenters: carmela_couderc,
reuben.daniels.

The cross-reference index returned no Zulip hits for `FHIR-54588`, but a

### [IG creation > Implementiing Additional Bindings for each purpose unclear](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/Implementiing%20Additional%20Bindings%20for%20each%20purpose%20unclear)

Eric Haas opened the thread (2025-10-22) with the same complaint that drives
FHIR-54588/-54128: the spec leaves open which strengths can be combined with
which purposes, whether additional bindings must be subsets of the main
binding, and there are no isolated examples. Grahame Grieve and Lloyd McKenzie
clarified the intended model in detail: **all bindings always apply**; strength
governs conformance weight (`example`/`preferred`/`starter`/`ui` carry no
validator force; `extensible` allows the "no concept fits" out; `required` is
hard); additional bindings on a `CodeableConcept` are conceptually `AND`ed (you
satisfy them collectively across multiple `Coding`s, not by `OR`); slicing
codings to express AND is now an anti-pattern in favor of additional bindings;
and the `usage`/context qualifier means a binding only applies inside that
context (so it is "satisfied" by falling out of context — language Gino Canessa
flagged as awkward). Robert McClure and Pieter Edelman pressed on edge cases
(must additional-binding codes be in the main value set? What does
`purpose=required` + `usage` mean together?). Eric Haas reported that at WGM
the group agreed to add a matrix of worked examples in the IG Guidance IG and
that he and Gay Dolin (the reporter of FHIR-54588) are building an explicit
use-case matrix for US Core. **This thread is the principal evidence base for
both tickets and shows broad consensus that the spec text and the current
combined strength/purpose table are confusing implementers in practice.**

Participants: Eric Haas, Grahame Grieve, Lloyd McKenzie, Robert McClure, Pieter
Edelman, Gino Canessa, Gay Dolin (referenced).

### [implementers > Additional Bindings validation (R4)](https://chat.fhir.org/#narrow/stream/implementers/topic/Additional%20Bindings%20validation%20%28R4%29)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revise the Terminology Bindings narrative in `terminologies.html` (the page
that today contains the combined "additional bindings" prose plus §4.1.4.2
Binding Strengths) so that strength and purpose are presented as two
independent axes:

1. **Split the existing single "binding properties" / strengths table** into
   two tables: a **Binding Strength** table containing only the four
   `BindingStrength` codes (`required`, `extensible`, `preferred`, `example`)
   with conformance/validator semantics in their own column, and a **Binding
   Purpose** table containing the `additional-binding-purpose` codes
   (`maximum`, `minimum`, `required`, `extensible`, `candidate`, `current`,
   `preferred`, `ui`, `starter`, `component`, `conformance`, …) with a
   "Conformance impact" column (none / documentary / behavioral) and a
   "Typical strength(s) it pairs with" column.
2. **Rename the heading** of §4.1.4.2 from "Binding Strengths" to make it
   explicit that the rows in the additional-bindings table are *not* all
   strengths (per carmela_couderc's 2026-03-23 note).
3. **Add a short normative paragraph** that states binding strength and
   binding purpose are independent concepts, that strength always governs the
   validator's enforcement weight, and that purpose describes intent/use of
   the additional value set; cross-link the two tables.
4. **Leave the underlying value sets and `ElementDefinition.binding.additional`
   structure unchanged** — this is a documentation/structuring change, not a
   model change.
5. **Defer the per-purpose subset/superset rules and worked examples to
   FHIR-54128** (which is the broader clarification ticket), but link forward
   to it from the new prose.

#### Justification

This matches what the TI WG agreed on the 2026-03-23 tracker call: the table
is manually authored, the heading is misleading, only `required` and
`extensible` actually drive conformance, and the WG explicitly accepted that
"the table heading will be changed to reflect that the rows do not represent
binding strength in all cases." The change is small in surface area, does not
touch the model or any value set, and resolves the precise complaint the
reporter raised. The Zulip evidence (Lloyd McKenzie's repeated "all bindings
always apply; strength tells you the obligation; purpose tells you why" formula
in the *IG creation > Implementiing Additional Bindings for each purpose
unclear* thread) is exactly the framing this disposition would land in the
spec. Scoping the per-purpose semantic rules to FHIR-54128 respects the WG's
finding that those rules are "not ready for a proposed resolution" yet.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-54588 as a **ballot comment to be folded into FHIR-54128**, and
resolve it as a structural sub-task once FHIR-54128's per-purpose rules and
matrix are agreed. Concretely:

1. Mark FHIR-54588 as a *Persuasive with Mod* whose change is "carried by
   FHIR-54128", with the clarifying agreement that any resolution to
   FHIR-54128 must include a documentation restructure that visibly separates
   strength from purpose.
2. As part of FHIR-54128, deliver a single new subsection in `terminologies.html`
   — for example *"Binding Strength and Binding Purpose"* — that introduces
   the two axes, gives the per-purpose rules (subset/superset of main binding,
   compatible main strengths, conformance weight, examples), and replaces the
   current combined table with the two-axis matrix Eric Haas and Gay Dolin are
   already drafting in the Guidance IG.
3. Add a small number of normative worked examples (US Core extensible-binding
   pattern, "send SNOMED AND ICD-10" via additional bindings rather than
   slicing, `usage`-scoped additional binding) directly in the spec rather
   than only in the Guidance IG.

#### Justification

The 2026-03-23 TI call explicitly tied FHIR-54588's resolution to FHIR-54128
and Grahame committed to a Zulip discussion to figure out the spec changes
together. Resolving FHIR-54588 in isolation risks landing a documentation
restructure that has to be redone once the per-purpose rules in FHIR-54128 are
finalized — implementers would then read two passes of changes in two ballot
cycles for the same area. Folding the structural change into FHIR-54128 also
gives the WG a single coherent edit (one PR against `terminologies.html`) and
addresses Pieter Edelman's and Robert McClure's open questions in the same
pass. The trade-off vs Disposition A is timing: this ties a quick, agreed
clean-up to a still-open ticket that is "Waiting for Input." If FHIR-54128
slips, FHIR-54588 slips with it.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-54588 as **Not Persuasive** on the grounds that the existing prose
in `terminologies.html#binding`, `terminologies.html#strength`, and the
`ElementDefinition.binding.additional` definition page already distinguishes
the two concepts: `ElementDefinition.binding.strength` is a separate property
from `ElementDefinition.binding.additional.purpose`, and the strength VS
(`BindingStrength`) and purpose VS (`additional-binding-purpose`) are distinct
artifacts. Direct any remaining clarification work to the Guidance IG that Eric
Haas and Gay Dolin are already authoring, rather than restructuring core spec
prose.

#### Justification

Defensible only on the narrow grounds that strength and purpose *are* already
separate at the structural/model level — the underlying `ElementDefinition`
shape, the two value sets, and the validator behavior already treat them as
orthogonal. The complaint is purely about how the spec narrative and the
manually authored table present them.

---

### Recommendation

**Recommended disposition:** **A**, executed in coordination with FHIR-54128.

Disposition A is what the TI WG already agreed to in substance on 2026-03-23
(rename the heading, mark the rows that aren't really strengths, add prose
that strength and purpose are independent), it is mechanically small (it
touches `terminologies.html` only and changes no value sets, no model, no
validator behavior), and it directly resolves the reporter's complaint without
waiting for FHIR-54128's per-purpose semantic matrix to be finalized.
Disposition C is hard to defend in light of the multi-year Zulip evidence
(2022 → 2026) showing that implementers, IG authors, and even spec renderers
keep tripping over the combined table — the complaint is real and recurrent.
Disposition B is appealing for editorial coherence, but it makes the easy,
agreed clean-up hostage to a ticket that is still *Waiting for Input* and that
Grahame has only just committed to seed a Zulip discussion for. The pragmatic
path is to land the structural separation now (A), explicitly note in the new
prose that per-purpose subset/superset rules and worked examples are tracked
under FHIR-54128, and let FHIR-54128 fill in the cells of the new purpose
table when the WG converges. That sequencing also gives implementers something
correctable to react to in the next ballot rather than another deferral.
