## Summary

The ticket asks Terminology Infrastructure to extend the core
`ValueSet/$expand` and `ValueSet/$validate-code` operations so that a
caller can supply expansion-manifest information directly, in two
related ways:

1. Add a new `manifestParameters 0..1 Parameters` input that carries a
   `Parameters` resource inline. It is mutually exclusive with the
   existing `manifest` parameter but has identical semantics — i.e. the
   same set of expansion-rule parameters, just passed by value rather
   than by reference.
2. Broaden the existing `manifest` parameter so that it can reference a
   `Library` **or** an `ImplementationGuide`. Both of those resources
   already carry the `cqf-expansionParameters` extension, which holds
   the `Parameters` resource that should be used to drive the
   expansion.

The proposal is the FHIR Core counterpart of work that has already
landed in the CRMI IG. [FHIR-51088](https://jira.hl7.org/browse/FHIR-51088)
("Clarify manifest parameters, versus manifest library") is the parent
discussion: it defined the distinction between a "manifest parameters"
(a non-persisted `Parameters` resource of pinning entries) and a
"manifest library" (a persisted `Library` whose contained
`Parameters` plus `relatedArtifact.depends-on` give a full pinning),
introduced the `CRMIManifestParameters` profile, and added
`manifestParameters` to the CRMI `ValueSet/$expand` /
`ValueSet/$validate-code` operations — already **Applied** via
[HL7/crmi-ig PR #75](https://github.com/HL7/crmi-ig/pull/75).
[FHIR-51055](https://jira.hl7.org/browse/FHIR-51055) explored letting an
`ImplementationGuide` carry the manifest library; it was retracted by
the submitter, but it explains why the present ticket suggests pointing
`manifest` at an `ImplementationGuide` (relying on
`cqf-expansionParameters` rather than introducing a new IG extension).
[FHIR-53319](https://jira.hl7.org/browse/FHIR-53319) is a downstream
clarification about how `cqf-expansionParameters` and the new
`expansion-parameters` extension interact in a manifest library.

The Triaged + Persuasive disposition recorded in Jira indicates the
sub-group has already agreed in principle that the manifest-parameters
mechanism should be lifted from CRMI into the core `ValueSet`
operations; what remains is to confirm the exact wording and the
reference-target list for the `manifest` parameter.

## Details

**Description:**

> Add support for passing a Parameters resource directly:
>
> manifestParameters 0..1 Parameters, exclusive with manifest, but with the same semantics as that parameter
>
> Also allow the manifest to be a reference to a Library | ImplementationGuide, as both those resource can have a cqf-expansionParameters that contains the Parameters resource to be used.
>
> Propose that these changes be made to both ValueSet/$expand and ValueSet/$validate-code

**Resolution Description (recorded in Jira):**

> Add support for passing a Parameters resource directly:
>
> manifestParameters 0..1 Parameters — Specifies manifest parameters that provide expansion rules for the operation. See the description of manifest parameters for a complete description of how manifest values are used to provide defaults for expansion parameters. Parameters specified directly in an $expand operation override behaviors specified by the manifest parameter. It is an error to supply both a manifest and a manifestParameters.
>
> Update the existing manifest parameter to allow reference to an ImplementationGuide as well.

No comments are present on this ticket.

### [FHIR-53319: Add "used" parameters](https://jira.hl7.org/browse/FHIR-53319)

Terminology Infrastructure ticket (CRMI spec, **Triaged**, Persuasive)
that clarifies the layered semantics of the three relevant extensions
on a manifest library: `cqf-expansionParameters` (defaults that
**should be used** by `$expand`), `cqf-inputParameters` (parameters
fed into the publication process), and a new `expansion-parameters`
extension (records the parameters that **were used** during
publication). This matters to FHIR-51089 because the proposed allowance
to point `manifest` at a `Library` / `ImplementationGuide` depends on
servers knowing which extension to read; the resolution agreed to make
`cqf-expansionParameters` the SHOULD source and let
`expansion-parameters` provide an "as-published" fallback. Carmela
Couderc noted the `expansion-parameters` extension does not yet exist
in the extensions pack.

*Participants:* Bryn Rhodes (reporter), Carmela Couderc (commenter).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution wording already recorded on the ticket to the
core `ValueSet/$expand` and `ValueSet/$validate-code`
OperationDefinitions:

1. Add a new input parameter:
   - Name: `manifestParameters`
   - Cardinality: `0..1`
   - Type: `Parameters`
   - Use: `in`
   - Documentation: "Specifies manifest parameters that provide
     expansion rules for the operation. See the description of manifest
     parameters for a complete description of how manifest values are
     used to provide defaults for expansion parameters. Parameters
     specified directly in an $expand (or $validate-code) operation
     override behaviors specified by the manifestParameters. It is an
     error to supply both a `manifest` and a `manifestParameters`."
2. Update the existing `manifest` parameter on both operations so that
   the allowed reference targets are
   `Reference(Library | ImplementationGuide)` (currently `Library`
   only). Document that a server reads
   `cqf-expansionParameters` from whichever target is supplied.
3. Add an OperationDefinition-level invariant (or prose constraint)
   stating that `manifest` and `manifestParameters` are mutually
   exclusive.
4. Update the `ValueSet/$expand` and `ValueSet/$validate-code`
   narrative pages with a "Manifest parameters" subsection that
   matches the CRMI version-manifest narrative and points readers to
   `CRMIManifestParameters` as the recommended profile.

#### Justification

This is exactly what the reporter asked for, and Jira already records
the resolution as Persuasive with concrete normative wording, so the
group has effectively pre-agreed to the substance. The mechanism has
been implemented and shipped in the CRMI IG (FHIR-51088, Applied via
[crmi-ig PR #75](https://github.com/HL7/crmi-ig/pull/75)), which
provides a working reference implementation, an example
(`Parameters-crmi-manifestparameters-example.json`), and a profile
(`CRMIManifestParameters`) that core can point to without re-inventing
the shape. Lifting `manifestParameters` into core lets vanilla
terminology servers honour CRMI-style pinning without requiring
clients to depend on the CRMI IG, and broadening `manifest` to accept
`ImplementationGuide` reuses the `cqf-expansionParameters` extension
that already exists on both `Library` and `ImplementationGuide`,
avoiding the new IG-level extension that FHIR-51055 retracted.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt only the inline `manifestParameters` input on both operations,
and keep the `manifest` parameter as `Reference(Library)` only. Cover
the "manifest from an ImplementationGuide" use case by documenting
that callers should pass the IG's contained / referenced manifest
`Library` directly, or by adding a clarifying note that a server MAY
resolve `cqf-expansionParameters` from the IG when the caller passes
the IG canonical via the existing `context` / `system-version`
mechanisms — without changing the cardinality or target type of
`manifest`. Defer the `ImplementationGuide` target change until
FHIR-53319 (`expansion-parameters` extension) and the related "which
extension wins" question are settled.

#### Justification

`ImplementationGuide` and `Library` have very different roles
(packaging vs. logic/manifest container), and FHIR-51055 already
showed the working group was uneasy about coupling
`ImplementationGuide` more tightly to manifest semantics — it was
retracted partly on the "ImplementationGuide is a package" Madrid
resolution. Layered on top, FHIR-53319 explicitly notes that the
`expansion-parameters` extension that the new IG-as-manifest path
would interact with does not yet exist, and that the precedence rules
between `cqf-expansionParameters`, `cqf-inputParameters`, and
`expansion-parameters` are still being clarified. Splitting the
ticket — accept the uncontroversial inline `manifestParameters` half
now, defer the `ImplementationGuide` target until those upstream
ambiguities resolve — gives implementers the immediate value (no need
to round-trip a `Parameters` resource through a server-stored
`Library`) without locking core `ValueSet` operations into a behaviour
that may need to change once FHIR-53319 lands.

---

### Disposition C: Decline

#### Proposal

Resolve as "Not Persuasive – Already Addressed by CRMI IG". Direct
implementers who need manifest-driven expansion semantics to use the
CRMI `ValueSet/$expand` / `ValueSet/$validate-code` operations
(`OperationDefinition/crmi-valueset-expand`,
`OperationDefinition/crmi-valueset-validate-code`) plus the
`CRMIManifestParameters` profile. Keep core `ValueSet/$expand` and
`ValueSet/$validate-code` unchanged in this area.

#### Justification

The Manifestable interface and the `manifestParameters` input were
deliberately introduced in CRMI as a layered extension over the base
operations; FHIR-51088 has already shipped them in the CRMI IG. The
core operations are widely implemented and historically conservative
about adding new inputs, especially ones (like a full `Parameters`
resource) that materially change request shape and parser
expectations. Pushing the mechanism into core forces every base
terminology server to reason about manifest precedence, mutual
exclusivity with `manifest`, and the
`Library`/`ImplementationGuide` resolution path — concerns that
today only manifest-aware (CRMI-conformant) servers need to handle.
If the gap is really that clients want a vendor-neutral on-ramp, that
can be solved by referencing the CRMI operations from the core
narrative without modifying the base OperationDefinitions.

---

### Recommendation

**Recommended disposition:** A

The ticket's resolution is already recorded as Persuasive with
normative-quality wording; the underlying mechanism has been
implemented, peer-reviewed, and merged in the CRMI IG (FHIR-51088 →
Applied, with PR and example artifacts in `HL7/crmi-ig`); and the only
real risk that Disposition B tries to mitigate — the
`ImplementationGuide` target — is squarely within the
`cqf-expansionParameters` extension that already exists on
`ImplementationGuide`, so adding it as a permitted reference target
does not depend on FHIR-53319 landing. Disposition C is hard to
defend now that CRMI has demonstrated the pattern works and that
implementers want it available without taking a CRMI dependency.
Disposition A, mirroring the CRMI shape verbatim into core, is the
cleanest outcome: it preserves a single normative definition of
`manifestParameters`, keeps CRMI as a profile/usage layer rather than
a divergent operation, and gives Terminology Infrastructure a clean
hand-off point for FHIR-53319's "used vs. should-be-used" extension
clarifications when those land.
