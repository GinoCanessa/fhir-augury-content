## Summary

The reporter notes that the ConceptMap resource description today restricts
`source` and `target` to CodeSystems, while the CodeSystem resource defines a
mechanism whereby other canonical artifacts (StructureDefinitions,
Questionnaires, etc.) can be treated as *implicit* code systems
(see https://build.fhir.org/codesystem.html#implicit-code-systems). Because
ConceptMap does not call out this connection, implementers are unsure whether
they may legitimately use ConceptMap to express mappings such as logical model
→ resource, Questionnaire → Questionnaire, or profile → logical model. The
ticket asks for two things: (1) add explicit language to the ConceptMap
resource page that points to the implicit-code-systems section and clarifies
that mappings to/from such artifacts are supported, and (2) ensure the
canonical FHIR tooling actually accepts these canonical URLs as ConceptMap
endpoints (the reporter notes prior tooling failures).

The ticket explicitly relates to [FHIR-43032](https://jira.hl7.org/browse/FHIR-43032)
("Mechanism (extension or other) for associating a profile with the logical
model that represents it"), which was resolved Persuasive with Modification.
Its resolution adds a new "Approaches for Mapping between models" section to
the StructureDefinition notes that already endorses using ConceptMap "with
each of the structural models expressed as implicit code systems" as one of
the recommended mapping approaches. FHIR-54769 effectively asks Terminology
Infrastructure to mirror that same guidance from the ConceptMap side and to
make sure tooling does not reject it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the ConceptMap resource page in FHIR Core to explicitly call out the
implicit-code-systems mechanism. Concretely:

1. In the ConceptMap resource description (the prose surrounding
   `ConceptMap.sourceScope[x]` / `ConceptMap.targetScope[x]` and
   `ConceptMap.group.source` / `.target`), add a paragraph stating that the
   `uri`/`canonical` value is not restricted to CodeSystem resources: any
   canonical URL that resolves to an artifact with an "implicit code system"
   per the rules at
   `codesystem.html#implicit-code-systems` (currently StructureDefinition
   and Questionnaire) is a legal source/target. Link to that section.
2. Add a short "Mapping between structural models" sub-section on the
   ConceptMap page (or in `conceptmap.html#usage`) with worked examples:
   logical model → resource (e.g., the cross-version packages already
   shipping such ConceptMaps), Questionnaire → Questionnaire, and
   profile → logical model (cross-referencing the FHIR-43032 "Approaches for
   Mapping between models" note in StructureDefinition).
3. File a companion tooling task against the validator and IG publisher to
   accept StructureDefinition / Questionnaire canonical URLs anywhere a
   ConceptMap source/target is expected, returning a non-error (information
   or hint) when the canonical resolves to an artifact with an implicit
   code system. This mirrors the rendering work already requested for
   FHIR-43032 in HL7/fhir-ig-publisher#1220.

#### Justification

The resolution of FHIR-43032 already commits FHIR Infrastructure to telling
implementers to express profile/logical-model mappings as ConceptMaps "with
each of the structural models expressed as implicit code systems". Without
the matching language on the ConceptMap page, an implementer reading
ConceptMap in isolation will conclude (as the terminologist in the
"IG creation" Zulip thread did) that the practice is wrong. The
"terminology > Implicit CodeSystems" thread shows the use case is already
in production (cross-version packages, hl7-eu/hdr) and that the relevant
authors (Grieve, Lawley, Costa Teixeira) treat this as the intended design.
The change is therefore a documentation alignment, not a model change, and
falls cleanly inside Terminology Infrastructure's remit for the ConceptMap
resource. The companion tooling task is necessary because the reporter
notes prior failures, and Jose's question on Zulip ("would StructureDef
canonical URLs be accepted as a CodeSystem URL for ConceptMap?") confirms
the gap is real.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding prose to ConceptMap, formalise the contract by:

1. Editing the type/binding text on `ConceptMap.sourceScope[x]`,
   `ConceptMap.targetScope[x]`, `ConceptMap.group.source`, and
   `ConceptMap.group.target` so the element definitions themselves state
   that the canonical may identify a CodeSystem **or** any artifact that
   defines an implicit code system per `codesystem.html#implicit-code-systems`.
2. Adding a normative note in CodeSystem's "Implicit Code Systems" section
   that explicitly enumerates the consumers (ConceptMap, ValueSet
   compose.include.system, $translate, $validate-code) so the contract is
   declared in one place and pointed at from many.
3. Defining a small invariant or computable hint (e.g.,
   `cnl-conceptmap-canonical`) that validators can use to recognise the
   pattern and emit a hint rather than an error, and ship it together with
   the documentation change so tooling has a single anchor to implement
   against.
4. Treat the documentation paragraphs in Disposition A as a follow-on PR
   that references this normative anchor.

#### Justification

The reporter explicitly mentions that "tooling supports this as well — we
had issues on this in the past". Documentation alone (Disposition A) leaves
each implementation free to interpret the prose differently, and the Zulip
thread shows experienced terminologists are already pushing back against
the practice. Anchoring the contract in element-level text plus a named
invariant gives both the spec authors (Lloyd's "decision tree" suggestion
on FHIR-43032) and tooling vendors a single, testable definition. It also
matches Grahame's observation in the "Implicit CodeSystems" thread that
the original section "mashed two different things together" — formalising
the contract is the cleanest way to disentangle them. The cost is more
ConceptMap surface change and another invariant to maintain.

---

### Disposition C: Decline

#### Proposal

Decline the request as already addressed. The CodeSystem "Implicit Code
Systems" section already states that StructureDefinition and Questionnaire
*are* code systems for the purpose of ConceptMap and similar resources, and
the FHIR-43032 resolution adds explicit guidance on the StructureDefinition
side. Implementers should be directed to those existing sections; no
additional ConceptMap-side language or tooling work is required.

#### Justification

The mechanism exists, the cross-version packages already use it in
production, and FHIR-43032 has agreed text that names ConceptMap as the
preferred concept-level mapping mechanism. From a strict
"is-the-spec-already-correct?" angle, the answer is yes; adding prose to
ConceptMap risks duplicating (and eventually drifting from) the canonical
text in CodeSystem and StructureDefinition. The April 2026 "IG creation"
Zulip thread can be addressed by community education / a clarifying FAQ
rather than a normative change.

---

### Recommendation

**Recommended disposition:** A

Disposition C understates the problem: the most recent Zulip thread (April
2026) shows that even within the IG-creation community the practice was
challenged by a terminologist, and the reporter explicitly documents
historical tooling failures. "The mechanism exists" is not the same as
"the mechanism is discoverable from where implementers read." Disposition B
is technically the strongest answer, but it expands the change from a
clarification into an element-definition edit plus a new invariant, which
materially increases ballot risk and slows what is fundamentally a doc
fix the FHIR Infrastructure side has already committed to via FHIR-43032.

Disposition A delivers exactly what the reporter asks for and exactly what
the cross-referenced Zulip discussions show is missing: discoverable
ConceptMap-side language and a tracked tooling task. It is consistent with
the resolution path already taken by FHIR-43032, keeps Terminology
Infrastructure as the owner of the ConceptMap edit, and leaves the door
open to promote it to Disposition B's normative anchor in a future cycle
if the doc-only approach proves insufficient. Recommend Persuasive with
Modification along the lines of A, with the tooling sub-task spun off so
it can be tracked against the validator and IG publisher independently.
