## Summary

Rob McClure (Terminology Infrastructure) is asking that the FHIR Core
"Implicit Code Systems" section in `codesystem.html` be rewritten so that
it actually defines what an implicit code system *is* and, equally
important, what *cannot* be expressed as one. As written, the section
introduces the concept only by way of a single reference to its use in
ConceptMap, and leaves a working terminologist (Rob himself, in this
case) unable to draft replacement text — which he treats as evidence that
the section is too unclear to be left as-is. He also asks that one or two
worked examples be added.

The request sits on top of two earlier rounds of editorial work in the
same area: [FHIR-39265](https://jira.hl7.org/browse/FHIR-39265) (Carmela
Couderc) flagged the same section as confusing and led to the implicit
code systems content being removed; [FHIR-40432](https://jira.hl7.org/browse/FHIR-40432)
(Grahame Grieve) then asked for the section to be restored because real,
useful functionality was being lost. Grahame's own retrospective on
chat.fhir.org acknowledges that the original section "mashed two different
things together" — (1) implicit code systems for single
`StructureDefinition` and `Questionnaire` instances, and (2) implicit
code systems backing resources like `Medication`, `Location`, and
`ObservationDefinition` — and that those need to be teased apart. A
related but distinct request,
[FHIR-50704](https://jira.hl7.org/browse/FHIR-50704) (Giorgio Cangioli),
proposes a different mechanism (extending ConceptMap to map model-to-model
without going through implicit code systems at all). FHIR-54701 should
therefore be read as a request to *finish* the FHIR-39265 / FHIR-40432
arc with prose that is precise enough for a terminologist to use.

## Details

**Description:**

> The short text introducing 'Implicit Code Systems' does not define what
> an implicit code system is. It only includes a reference to one use in
> concept maps. This section needs a detailed definition as to what this
> artifact is and boundaries as to what *cannot be* represented as an
> implicit code system. As a terminologist I don't understand this well
> enough to suggest need text, which demonstrates how confusing this
> thing is.
>
> In addition, one or two examples should be included.

### [FHIR-40432: Restore section about implicit code systems](https://jira.hl7.org/browse/FHIR-40432)

Grahame Grieve filed this immediately after FHIR-39265 was resolved by
removal, arguing that "much of the discussion around this is
mis-informed" and that real, useful functionality (notably implicit
code systems backing single `StructureDefinition` and `Questionnaire`

not be resolved without talking to him. Resolution was *Persuasive* and
the section was reinstated — but, by Grahame's own admission on Zulip,

which is what FHIR-54701 is now trying to clean up.

Reporter: Grahame Grieve; participants: Jose Costa Teixeira, Michael

### [FHIR-50704: Improve the support for model to model maps](https://jira.hl7.org/browse/FHIR-50704)

Giorgio Cangioli's ticket proposes adapting `ConceptMap` so that
model-to-model concept relationships (where the "concepts" live in
`StructureDefinition` logical models, not in a `CodeSystem`/`ValueSet`)
can be expressed natively. On Zulip, Giorgio offered this as an
alternative to relying on implicit code systems at all; Grahame
characterised it as "a different approach that is less empowering than
describing implicit code systems because it works across the terminology
eco-system." Status: Waiting for Input / Unresolved. Relevant here
because any disposition that re-scopes implicit code systems narrowly
needs to acknowledge what FHIR-50704 would (or would not) absorb.

Reporter: Giorgio Cangioli; participants: Grahame Grieve, Jose Costa
Teixeira, Michael Lawley.

### [terminology > Implicit ValueSets collision](https://chat.fhir.org/#narrow/stream/terminology/topic/Implicit%20ValueSets%20collision)

Orlando Osorio raised an operational case where two distinct
`CodeSystem` resources declared the same `CodeSystem.valueSet` URL,
causing `$expand` / `$validate-code` to be ambiguous. Grahame Grieve
agreed two code systems claiming the same implicit value set is "an
error" and was "fixed in R5"; he advised preferring an explicit
`ValueSet` over the implicit one when both are present. Michael Lawley
pushed back on edge cases — explicit `ValueSet` versions that don't pin

resolution and version pinning. Relevant here because it is a worked
example of the kind of boundary condition ("when does an implicit value
set / code system not behave well") that FHIR-54701 is asking the spec
to make explicit.

Participants: Orlando Osorio, Grahame Grieve, Michael Lawley, Brian
Postlethwaite.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rewrite the "Implicit Code Systems" section in FHIR Core's
`codesystem.html`, taking the diagnosis from the May 2025
`terminology > Implicit CodeSystems` Zulip thread as the editorial
brief. Specifically:

1. Replace the current short introductory paragraph with a normative
   definition: an *implicit code system* is a `CodeSystem` whose
   concept set is determined by reference to another resource at
   resolution time rather than enumerated in `CodeSystem.concept`,
   identified by a canonical URL that resolves to that backing
   resource.
2. Split the section into clearly-labelled subsections, one per
   supported pattern. Based on Grahame's enumeration: (a) implicit
   code systems backed by a single `StructureDefinition`
   (parameterised profiles, conditional bindings — the case Jose
   Costa Teixeira's team is exercising), (b) implicit code systems
   backed by a single `Questionnaire`, (c) implicit code systems
   backed by resources like `Medication`, `Location`, and
   `ObservationDefinition`. For each, state the canonical URL form,
   what the "codes" are, what `CodeSystem.valueSet` means, and what
   `$lookup` / `$validate-code` are expected to return.
3. Add an explicit "Cannot be represented as an implicit code system"
   subsection enumerating the boundaries Rob asks for (e.g., systems
   that need post-coordination, hierarchical filters, version
   negotiation distinct from the backing resource's version, or
   externally-curated relationships).
4. Add at least two worked examples drawn from real published
   artifacts — one `StructureDefinition`-backed (the
   conditional-bindings pattern Jose linked) and one resource-backed
   (`Medication` or `Location`), each showing the canonical URL, the
   backing resource fragment, and the resulting `$lookup` / `$expand`
   behaviour.
5. Cross-link to the "Implicit Value Sets" section so that the
   collision issues raised in
   `terminology > Implicit ValueSets collision` (two code systems
   claiming the same `CodeSystem.valueSet` URL, version pinning) are
   surfaced as known constraints rather than discovered in the wild.

#### Justification

This is exactly what the ticket asks for, and it has a ready-made
editorial outline: Grahame's own Zulip post identifies the conflation
that makes the current text confusing, names the patterns that need to
be separated, and flags `ObservationDefinition` as previously
under-described. Carmela Couderc's original FHIR-39265 critique — that
the lead sentence conflates ConceptMap-of-ValueSets with mapping
between code systems, and that the table doesn't define "implicit" —
maps directly to fixes (1) and (2). Jose Costa Teixeira's "intensively
using" usage gives a concrete, in-production example for (4). And
Rob's "as a terminologist I don't understand this" bar is the right one
to write to: a terminologist should be able to read the section and
decide, for any given artefact, whether an implicit code system is the
correct mechanism — which is exactly what (3) enables.

---

### Disposition B: Alternative Approach

#### Proposal

Narrow the spec rather than expand the prose. Restrict the "Implicit
Code Systems" section to *only* the `StructureDefinition`- and
`Questionnaire`-backed patterns — Grahame's "so I think we should only
bring back StructureDefinition and Questionnaire" position from the
Zulip thread — and explicitly deprecate (or move to an Informative
appendix) the `Medication` / `Location` / `ObservationDefinition`
patterns. Then write the focused definition and examples Rob asks for
against that smaller, internally-consistent surface, and add a short
forward-pointer that says "for model-to-model relationships between
logical models, see [ConceptMap improvements proposed in
FHIR-50704](https://jira.hl7.org/browse/FHIR-50704) when those land."

#### Justification

The reason the section is hard to write clearly is that it documents
several mechanisms with different semantics under one heading.
Disposition A papers over that by writing more prose; Disposition B
attacks the underlying cause by reducing the surface to the cases that
are demonstrably in use and well-understood (Jose's parameterised
profiles, Reuben's task-force work, Questionnaire answer codes). It
also avoids re-litigating the FHIR-39265 → FHIR-40432 cycle: the
reinstated content stays, but only for the patterns Grahame is willing
to defend in writing. The trade-off vs. A is real — implementers
relying on the `Medication`/`Location` form would need a deprecation
window — but the workgroup gets a section it can actually finish, and
the deprecation question is one Terminology Infrastructure already
needs to answer to close FHIR-50704 cleanly.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54701 as out of scope for the current cycle on the basis
that the underlying mechanism is already under active redesign via
FHIR-50704 ("Improve the support for model to model maps") and that any
text written now will need to be rewritten once that ticket resolves.
Mark FHIR-54701 as *Not Persuasive — see FHIR-50704* and re-open it
once the ConceptMap / model-to-model work has been dispositioned.

#### Justification

The defensible version of declining rests on avoiding wasted editorial
work: if Terminology Infrastructure adopts a richer model-to-model
mechanism in `ConceptMap`, the role of implicit code systems in
mapping use cases changes materially, and prose written today would
need to be revisited. Declining now and bundling the rewrite with
whatever FHIR-50704 produces gives a single coherent narrative instead
of two.

The argument against this disposition is that Grahame himself called
FHIR-50704's approach "less empowering" than implicit code systems and
indicated they solve different problems, so the rewrite is unlikely to
become moot. It is included here because the workgroup may still want
to weigh "wait for the bigger picture" against "fix the obvious
confusion now."

---

### Recommendation

**Recommended disposition:** B

Disposition B is the recommendation because the diagnosis is no longer
in dispute: the section reads poorly because it documents two (really
three) mechanisms under one heading, and the senior author of the
original text has said on chat.fhir.org that he would only defend the
`StructureDefinition` and `Questionnaire` cases. Writing more prose
around an internally-inconsistent surface (Disposition A) is what
produced the current text in the first place — Carmela's FHIR-39265
critique and Rob's FHIR-54701 critique are essentially the same
critique three years apart. Reducing the scope, then writing the
definition / boundaries / two worked examples Rob asks for against
that reduced scope, gives Terminology Infrastructure a section it can
actually close out. Disposition C waits on FHIR-50704, which by
Grahame's own assessment doesn't subsume the use cases Jose Costa
Teixeira's team is depending on today, so the wait would not pay off.
If the workgroup is unwilling to deprecate the `Medication` /
`Location` form in this cycle, Disposition A is an acceptable
fallback — but the editorial brief should still be the
"split into named subsections" outline, not a single rewritten
paragraph.
