## Summary

This ticket requests a series of editorial text updates to the FHIR Core
**Terminology Module** landing page (§4.0 *terminologies.html*). The changes
were collected by the Terminology Infrastructure (TI) co-chairs as part of
the *FHIR R6 Terminology Content Review* (Confluence page 256510055) and
entered into Jira by Jessica Bota on behalf of that review. The edits span
four sub-sections of the page:

- **4.0.1 Introduction** — replace the opening sentence with a tighter
  definition of the Terminology Module as a foundational component of core
  FHIR, and rephrase the lead-in to the diagram of terminology resource
  types.
- **4.0.2 Index** — rename the "Operations" section to "Terminology
  Specific Operations" and add a NamingSystem column with links to the
  `$preferred-id` and `$translate-id` operations.
- **4.0.4 Common Use Cases** — reword the value-set, record-data, validate/
  look-up, mapping, and capability-statement bullets so that several
  related bullets collapse into single, sharper statements (e.g., merge the
  three "Record data" bullets into "Record codes, expressions and
  translations"; merge "Validate" and "Look up" into a single bullet).
- **4.0.5 Development Roadmap** — update the wording to reflect that all
  terminology resources are now normative, and consolidate the
  resource-status lists into a single list.

All edits are content-only on the module landing page; no schema, profile,
operation definition, or example artifact is being changed. There are no
linked or cross-referenced Jira tickets — the only cross-reference is the
Confluence review page that is the source of these proposals.

## Details

**Description:**

In **4.0.1 Introduction** make the following changes:

1. Update the first sentence from:

   > "The Terminology Module provides an overview and guide to the FHIR
   > resources, operations, coded datatypes and externally-defined standard
   > and FHIR-defined terminologies that are used for representing and
   > communicating coded, structured data in the FHIR core specification
   > and profiles."

   to:

   > "The Terminology Module is a foundational component of core FHIR
   > specification.  Any terminology related resource type, operation or
   > datatype is part of the Terminology Module."

2. Update the paragraph in 4.0.1 Introduction from:

   > "The primary terminology-related structures and their relationships
   > are shown below:"

   to

   > "The diagram below shows the relationship between terminology
   > resource types."

In **4.0.2 Index**, make the following changes:

1. Change name of section from "Operations" to "Terminology Specific
   Operations".

2. Add a column in the Operations section for NamingSystem and add links
   to the following operations:
   - https://build.fhir.org/namingsystem-operation-preferred-id.html
   - https://build.fhir.org/namingsystem-operation-translate-id.html

In **4.0.4 Common Use Cases**, make the following changes:

1. Replace "Create or reference a value set" with: "Create, reference or
   expand a value set."
2. Replace the 3 "Record data" bullet points with one bullet point that
   states: "Record codes, expressions and translations".
3. Collapse the "Validate" and "Look up" bullets into one bullet point
   that states: "Validate or look up a display term for a code".
4. Replace "Map data between different terminologies" with: "Create or
   reference a concept map".
5. Replace "Declare the capabilities of a terminology service" with:
   "Operate a terminology service".

In **4.0.5 Development Roadmap**, make the following changes:

1. Since all resources in FHIR are normative, update the text to say:
   "The following terminology resources have been tested and are being
   used in production tooling are normative".
2. All resources should be listed together in one list.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply each of the editorial changes verbatim to the Terminology Module
landing page (§4.0 of the FHIR Core spec, source for
`terminologies.html`):

1. **§4.0.1 Introduction** — replace the opening sentence and the
   "primary terminology-related structures" lead-in with the wording
   given in the description.
2. **§4.0.2 Index** —
   - Rename the "Operations" sub-heading to "Terminology Specific
     Operations".
   - Add a `NamingSystem` column to the operations matrix and populate
     it with rows for `$preferred-id` and `$translate-id` (linked to
     `namingsystem-operation-preferred-id.html` and
     `namingsystem-operation-translate-id.html`).
3. **§4.0.4 Common Use Cases** — replace the five bullets exactly as
   specified (value-set, record-data, validate/look-up, mapping,
   capabilities).
4. **§4.0.5 Development Roadmap** — replace the lead-in sentence with
   the new normative-status wording and consolidate the resource lists
   into a single list.

No code, schema, operation definition, profile, or example is changed —
this disposition is entirely a `terminologies.html` content edit, plus
the corresponding source in the FHIR Core spec authoring tree. The PR
should be opened by the TI work group and reviewed against the
Confluence R6 Terminology Content Review page.

#### Justification

The changes are editorial and bring the page into line with the TI
co-chair review (Confluence 256510055), which is the cited authority for
all five edits. The new Introduction wording is shorter and clearer; the
"Terminology Specific Operations" rename and the new NamingSystem column
correct a real gap (NamingSystem operations are currently absent from
the index even though they exist in build.fhir.org); the consolidated
Common Use Cases bullets remove redundancy that has accumulated over
multiple ballot cycles; and the Development Roadmap update reflects the
factual change in normative status of the terminology resources for R6.
None of the edits change semantics, conformance, or downstream tooling.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substantive content edits but split the work into three
narrower changes and tighten the wording on two of the proposed edits
before applying:

1. **Treat 4.0.1, 4.0.2, and 4.0.4 as one editorial PR** (uncontroversial
   text + index update).
2. **Defer the 4.0.5 Development Roadmap rewrite to a second PR** that
   first verifies the current normative status of every listed
   terminology resource against the FHIR Core maturity matrix, and
   updates the wording to be grammatical (the proposed sentence,
   *"The following terminology resources have been tested and are being
   used in production tooling are normative"*, has a missing conjunction
   — suggest *"The following terminology resources have been tested,
   are being used in production tooling, and are normative."*).
3. **Tighten 4.0.4 bullet 1** from *"Create, reference or expand a value
   set."* to *"Create, reference, or expand a value set"* (Oxford comma,
   no trailing period for parallelism with the other bullets).

#### Justification

The goal of the ticket — to clean up the Terminology Module landing page
in line with the TI co-chair review — is sound, but two of the proposed
strings are imperfect as written. The Development Roadmap sentence as
quoted in the ticket is grammatically broken and asserts a normative
status for *all* listed resources without naming them; splitting that
edit into a second PR lets TI verify the maturity matrix and produces a
sentence that will not need to be re-edited in the next ballot. Splitting
also shrinks the review surface for the bulk of the changes (4.0.1,
4.0.2, 4.0.4), which are uncontroversial copy edits, so they can land
quickly without being held up by the maturity-status discussion.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as filed and ask the TI co-chairs to either (a)
re-open the underlying R6 Terminology Content Review on Confluence and
land the wording changes directly via the spec PR process, or (b)
re-file the ticket as separate change requests per page section so that
each substantive change (especially the §4.0.5 normative-status claim)
can be tracked and balloted independently.

#### Justification

This is defensible only if the work group judges that bundling four
distinct editorial decisions into a single ticket is itself a process
problem. The §4.0.5 change in particular asserts that *all* listed
terminology resources are normative, which is a factual claim about
maturity status that should be verified against the resource maturity
matrix before being baked into the landing page; mixing that with
uncontroversial copy edits in §4.0.1/4.0.2/4.0.4 risks the ticket being
held up indefinitely. Declining and asking for resubmission would force
that separation. However, this disposition is weak in practice: the edits
are small, the source of truth (the Confluence review) already exists,
and TI can address any wording concerns inside the PR review.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the substantive intent of the TI co-chair review
without locking in two minor defects (the broken §4.0.5 sentence and the
small punctuation inconsistency in §4.0.4). Splitting the §4.0.5
normative-status edit into a second PR is also methodologically safer:
the claim that "all terminology resources are normative" is a factual
assertion about maturity status that ought to be checked against the
maturity matrix before being committed to the landing page. The bulk of
the changes (§4.0.1, §4.0.2, §4.0.4) can land immediately as a single
editorial PR, satisfying the ticket's intent without a multi-cycle
back-and-forth. Disposition A is acceptable if TI is willing to fix the
two wording issues in PR review; Disposition C should only be invoked if
the work group wants to enforce a stricter "one ticket per section"
practice going forward.
