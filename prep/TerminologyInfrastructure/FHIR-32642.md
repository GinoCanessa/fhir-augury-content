## Summary

The ticket asks Terminology Infrastructure to fold three parameters from the
CRMI / QM-IG `ValueSet/$expand` extension — `manifest`, `expansion` (a.k.a.
`expansionIdentifier`), and `includeDraft` — into the core
`ValueSet/$expand` `OperationDefinition` so that authoring and measure-
calculation workflows can get consistent expansion behaviour across servers
without having to lean on an IG-specific extension.

The conversation has narrowed considerably since the ticket was filed in
2021. The `manifest` parameter has been split off into its own ticket
[FHIR-45986](https://jira.hl7.org/browse/FHIR-45986) (status *Applied*,
resolution *Persuasive*), so manifest is no longer in scope here. The
`expansion` / `expansionIdentifier` parameter was extensively debated on
Zulip ([terminology > Proposed parameters for ValueSet/$expand](https://chat.fhir.org/#narrow/stream/terminology/topic/Proposed%20parameters%20for%20ValueSet%2F%24expand))
without producing implementer consensus and remains unresolved. The
`includeDraft` strand was reworked through 2024–2025: an early
*Persuasive with Modification* (May 2024) added a `considerStatus` code
parameter, but Grahame Grieve, Michael Lawley and others objected to its
name and definition, and the resolution was reverted in March 2025. After
discussion in Madrid Q3 2025 and on the
[Terminology Service Test Cases > includeDraft](https://chat.fhir.org/#narrow/stream/Terminology%20Service%20Test%20Cases/topic/includeDraft)
thread, Bryn proposed (and Grahame independently suggested) a single
boolean `includeUnreleasedContent` parameter, with default behaviour at
server discretion. That is the form currently captured in the Resolution
Description, and Carmela Couderc has placed the ticket on the 2026-02-23
TI Tracker agenda for confirmation, linking it to FHIR-54108 (review of
terminology operation parameters) per
[terminology > TI Tracker Agenda for 2026-02-23](https://chat.fhir.org/#narrow/stream/terminology/topic/TI%20Tracker%20Agenda%20for%202026-02-23).

## Details

**Description:**

Consider adding parameters for `manifest`, `expansion`, and `includeDraft`
to the `$expand` operation definition, as defined in the extension of the
`$expand` operation in the QM IG:
http://hl7.org/fhir/us/cqfmeasures/2021May/OperationDefinition-ValueSet-expand.html

**Resolution Description (current):**

> To support authoring systems that provide access to both "draft" and
> "released" content in the same repository, add the following parameter
> to the `$expand` operation:
>
> `includeUnreleasedContent` boolean 0..1 — `true` indicates that when
> resolving which version of a value set definition should be used during
> an expansion, the server SHALL consider unreleased content. `false`
> indicates that the server SHALL only consider released content during
> the expansion. If this parameter is not supplied, the behaviour is at
> the server's discretion. What constitutes released content is at the
> server's discretion, such as publication status or workflow status.
> Servers MAY choose to throw an exception if asked to include
> unreleased content and they do not support this capability.

### [Terminology Service Test Cases > includeDraft](https://chat.fhir.org/#narrow/stream/Terminology%20Service%20Test%20Cases/topic/includeDraft/near/505522151)

Opened by Grahame Grieve on 2025-03-13 immediately after the TI call
that reverted the `considerStatus` resolution. The thread (~116 messages)
focuses on what the parameter actually needs to do and how each server
behaves today. Grahame and Michael Lawley both objected to the
`considerStatus` name and to coupling the parameter to the artifact's
`PublicationStatus`. Bryn explained that VSAC, by default, only returns
the latest *active* version unless `includeDraft=true` is supplied —
i.e., the proposal is to let other servers preserve their current
behaviour while giving clients a portable way to opt in or out of draft
content. Bart Decuypere reported that Snowstorm has no equivalent
behaviour, and Alexander Kiel described a multi-key
(status → SemVer → lastUpdated → id) resolution strategy in his server
and questioned whether the discussion actually belongs to general
canonical resolution rather than `$expand` alone. Michael Lawley argued
for a more general "set of acceptable statuses" (or even a
tag/partition-based) mechanism, and noted that for `CodeSystem` you also
want to exclude `content = not-present` / `example`. The thread
converges on Grahame's suggestion of an intent-oriented boolean,
`includeUnreleasedContent = true | false`, with server-specific
semantics for what counts as "released"; Bryn agrees, and Grahame
re-pings the thread on 2025-05-12 to feed the result back to committee.
Open question: whether the same mechanism should be lifted up to
canonical-version resolution generally rather than living solely on
`ValueSet/$expand`.

Participants: Alexander Kiel, Bart Decuypere, Bryn Rhodes, Grahame
Grieve, Joshua Wiedekopf, Michael Lawley.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the current Resolution Description as-is: add a single new
parameter to the core `OperationDefinition/ValueSet-expand` —

> `includeUnreleasedContent : boolean [0..1]` — When `true`, the server
> SHALL consider unreleased content when resolving which version of a
> value set definition (and any value sets it imports/excludes by
> unversioned canonical reference) is used during the expansion. When
> `false`, the server SHALL consider only released content. When the
> parameter is omitted, behaviour is server-defined; what counts as
> "released" is also server-defined (e.g., `ValueSet.status`, an
> external workflow state, a tag). Servers MAY return an
> `OperationOutcome` error if `true` is requested and they cannot
> distinguish unreleased content.

Drop `manifest` and `expansion` / `expansionIdentifier` from this
ticket: `manifest` is already tracked and *Applied* under
[FHIR-45986](https://jira.hl7.org/browse/FHIR-45986), and
`expansionIdentifier` did not reach implementer consensus on Zulip.
Update the operation's narrative to describe the parameter's intent
(authoring servers vs. publication servers) and to note that the
parameter's behaviour applies transitively through unversioned
canonical references encountered during the expansion.

#### Justification

This is the form Bryn Rhodes proposed and Grahame Grieve independently
arrived at on the *includeDraft* Zulip thread, and the form Bryn
restated in the 2025-09-15 comment after the Madrid TI discussion. It
sidesteps the problems the committee had with the 2024-05
`considerStatus` proposal — confusing name, ambiguous "version of a
value set" wording, ambiguous transitivity through `import` — by
narrowing scope to a single boolean intent flag and explicitly leaving
"what is released?" to server discretion. It also preserves the current
default behaviour of every server in the discussion (VSAC, Snowstorm,
Alexander Kiel's server, Joshua Wiedekopf's package-driven workflow),
which protects backward compatibility — the concern Paul Denning
originally raised about `includeDraft` defaulting to `false`.

---

### Disposition B: Alternative Approach

#### Proposal

Replace the boolean with a code-list parameter that names the artifact
statuses a client is willing to accept during canonical-version
resolution, and lift the mechanism above `ValueSet/$expand` so the same
parameter is honoured wherever a terminology operation resolves an
unversioned canonical:

- Add `acceptableArtifactStatus : code [0..*]` (bound to
  `PublicationStatus`, extensible) to `ValueSet/$expand`,
  `ValueSet/$validate-code`, `CodeSystem/$lookup`,
  `CodeSystem/$validate-code`, `ConceptMap/$translate`, and the
  corresponding system-level operations. Default (parameter absent) =
  server-defined.
- Document the parameter once on a shared "canonical resolution"
  section of the terminology module rather than copy-pasting on every
  OperationDefinition.
- Keep `includeUnreleasedContent` *out* of the spec; the intent it
  expresses is the same as `acceptableArtifactStatus = active`.
- Continue to handle `manifest` via the already-Applied
  [FHIR-45986](https://jira.hl7.org/browse/FHIR-45986).

#### Justification

This is the direction Michael Lawley and (with caveats) Alexander Kiel
pushed for in the *includeDraft* thread — explicit status selection is
more expressive than a single boolean (`retired` and `unknown` are not
covered by `includeUnreleasedContent`), and Michael's observation that
the requirement applies generally to canonical resolution is well
founded: the same draft/released question shows up for `CodeSystem`
references inside an expansion (e.g., to exclude
`CodeSystem.content = not-present`). The trade-off is that this is a
larger and slower change with more coordination cost across the TI
operations; it also re-opens the `considerStatus` naming and
transitivity questions that the committee already wanted to retire.

---

### Disposition C: Decline

#### Proposal

Decline to add any new parameter to `ValueSet/$expand` for this
purpose. Keep the CRMI extension as the home for `includeDraft` /
`includeUnreleasedContent`. Implementers who need draft-vs-released
control would reference the CRMI OperationDefinition in their
CapabilityStatement, and clients who need cross-server portability
would resolve and request the appropriate value-set *version*
explicitly rather than relying on a status-based hint.

#### Justification

This is the position TI attendees voiced on the 2023-10-16 TI Tracker
call: each of the originally-requested parameters can be approximated
by existing mechanisms (request a specific `valueSetVersion`, lock
versions via a manifest, retrieve a known expansion by `search` on
`ValueSet.expansion.identifier`). Layering an authoring-workflow
concept ("draft vs released") onto the core operation introduces
behaviour that is already explicitly server-defined when the parameter
is omitted, which arguably gives implementers no more interoperability
than they had before. It also avoids the residual disagreement about
whether the parameter belongs on `$expand` specifically versus on
canonical resolution generally.

---

### Recommendation

**Recommended disposition:** A

The Madrid Q3 2025 discussion, the *includeDraft* Zulip thread, and the
2025-09-15 comment all point to the same minimal change:
`includeUnreleasedContent : boolean [0..1]` with server-defined default
and server-defined definition of "released". It is the only
formulation in the conversation that has support from both Bryn Rhodes
(the original requester) and Grahame Grieve (the most active objector
to `considerStatus`) and that does not break any of the server
behaviours that Bart Decuypere, Alexander Kiel, and Joshua Wiedekopf
reported. Disposition B is intellectually attractive — Michael Lawley
is right that the requirement is broader than `$expand` — but it would
re-open the naming and transitivity arguments the committee just
finished closing and would expand the scope from one ticket to a
cross-operation re-design, which is unlikely to land in R6. Disposition
C ignores the substantive use case (authoring servers like VSAC that
already maintain a draft/released split and want clients to be able to
ask for the same behaviour on any conformant server) and would leave
that capability stranded in CRMI. Apply Disposition A now, and treat
"is the same parameter needed on `$validate-code`, `$lookup`, etc.?"
as a follow-up under FHIR-54108.
