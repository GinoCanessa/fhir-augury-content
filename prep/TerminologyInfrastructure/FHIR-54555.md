## Summary

Reporter Jessica Bota requests an editorial pass across thirteen
terminology-related FHIR Core specification pages to replace bare uses of the
word "code" with the more precise phrase "concept code" wherever the context
warrants the disambiguation. The intent is purely to improve clarity for
readers — distinguishing the human-meaningful identifier of a concept (a
"concept code", e.g. `1234-5` from LOINC) from the more general / overloaded
use of "code" elsewhere in FHIR (which can also refer to the `code` data type,
the `code` element on Coding/CodeableConcept, syntactic codes, status codes,
etc.). The work was identified during the Terminology Infrastructure ballot
content clean-up. The companion ticket
[FHIR-53809](https://jira.hl7.org/browse/FHIR-53809) ("Text Updates for
Terminology Service Page") is explicitly carved out: any analogous changes on
`terminology-service.html` are tracked there, not here. No cross-references,
linked Jira tickets, Zulip threads, or GitHub items were surfaced by Augury
for this ticket; the only Jira context is the sibling FHIR-53809 named in the
description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a targeted editorial pass across the thirteen `HL7/fhir`
authoring pages enumerated in the description, replacing the bare word
"code" with "concept code" wherever the surrounding sentence is
talking about the human-meaningful identifier of a concept within a
code system (i.e., the value carried in `Coding.code` /
`CodeableConcept.coding.code` when used as a terminology lookup key).
Concretely, in `HL7/fhir/source/`:

- `terminologies-valuesets.html`, `terminologies-conceptmaps.html`,
  `identifier-registry.html` — top-level pages, edit in place.
- `codesystem/codesystem.html`, `codesystem/codesystem-operation-lookup.html`,
  `codesystem/codesystem-operation-validate-code.html`,
  `codesystem/codesystem-operation-subsumes.html` — narrative prose
  only; do not touch the generated parameter tables.
- `valueset/valueset.html`, `valueset/valueset-operation-expand.html`,
  `valueset/valueset-operation-validate-code.html` — same rule.
- `conceptmap/conceptmap.html`,
  `conceptmap/conceptmap-operation-translate.html` — same rule.
- `terminologycapabilities/terminologycapabilities.html` — same rule.

Explicitly preserve all uses of "code" that refer to (a) the `code`
FHIR primitive data type, (b) the `code` element on `Coding` or
`CodeableConcept`, (c) operation/parameter names whose canonical name
is `code` (e.g. the `code` input parameter on `$lookup`,
`$validate-code`, `$subsumes`, `$translate`), or (d) status / mode
codes. Do not touch `terminology-service.html` (covered by
[FHIR-53809](https://jira.hl7.org/browse/FHIR-53809)).

#### Justification

The change is exactly what the reporter — and the TI ballot
content-clean-up review — asked for. It is a narrative-only,
backwards-compatible editorial change with zero impact on schemas,
StructureDefinitions, OperationDefinitions, examples, or downstream
artifacts. The FhirCore briefing confirms that prose pages under
`source/*.html` and `source/<resource>/*.html` are authored directly
and that "Update narrative / prose" is the correct change recipe. The
disambiguation of "code" vs. "concept code" is consistent with how the
FHIR community already speaks about terminology in informal contexts
(Zulip, ballot calls) and removes a known reader-confusion source
between the `code` data type / element and the value of a coded
concept. Limiting the scope to bare prose (not generated tables or
parameter names) honours the briefing's gotcha that operation pages
contain regenerated content.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than replacing each occurrence inline, add a single, prominent
**glossary / definitional callout** at the top of
`source/terminologies.html` (and link it from each of the thirteen
listed pages with a one-line "*In this section, 'code' refers to a
**concept code** — the identifier of a concept within a code system.*"
note near the start of each page). Leave the body prose largely
untouched, except for the small number of sentences where "code" is
genuinely ambiguous *within a single sentence* (e.g., "the code is
returned as a code" → "the concept code is returned as a `code`"). A
companion FHIRPath-style typographic convention — `code` (monospace)
for the data-type/element use and "concept code" (plain text) for the
terminology meaning — would be documented once in the callout and
applied only at points of real ambiguity.

#### Justification

A pure search-and-replace across thirteen pages risks producing
stilted prose ("the concept code system defines the concept code …")
and may force the editor to make many subjective judgement calls
about which "code" is which. A definitional callout plus targeted
edits achieves the reader-clarity goal with a much smaller diff,
keeps the ballot/normative text stable, and creates a reusable
convention that future TI editors and other work groups can apply
without re-litigating individual sentences. It also reduces the risk
of accidentally replacing a `code`-the-element use with "concept
code" and introducing a new error. The trade-off vs. Disposition A is
that some readers will still see the bare word "code" and have to
infer the meaning from context — but the up-front callout, plus the
typographic convention, makes the inference explicit.

---

### Disposition C: Decline

#### Proposal

Decline the request. Take no action on the thirteen listed pages.
Document the reason on the ticket as: "Editorial preference; the
existing prose is unambiguous in context, and a sweeping
search-and-replace risks introducing inconsistencies between core
terminology pages and the rest of the FHIR specification, which
continues to use 'code' throughout. Reader clarity is better served
by the existing data-type / element / parameter naming conventions
and the surrounding narrative."

#### Justification

The term "code" is pervasive throughout FHIR Core (the `code` data
type, `Coding.code`, `CodeableConcept.coding.code`, the `code`
parameter on multiple terminology operations, status `code`s on most
resources). Renaming uses of the bare word selectively on thirteen
pages, while leaving every other page and every schema/definition
artifact unchanged, creates a vocabulary inconsistency that may
confuse readers more than it helps. There is no community Zulip
thread or implementer report (none surfaced by Augury) demonstrating
that the ambiguity is causing real-world implementation problems —
the ticket originates from an internal ballot clean-up rather than
from external feedback. Declining preserves uniformity with the rest
of the spec at the cost of a small clarity gain on a subset of
pages.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested), with the explicit
guard-rails called out in the proposal.

The change is editorial-only, narrowly scoped, and exactly matches the
FhirCore "Update narrative / prose" change recipe. It is one of two
sibling clean-ups (with [FHIR-53809](https://jira.hl7.org/browse/FHIR-53809))
identified by the same reviewer during TI ballot prep; rejecting it would
leave that clean-up half-done and would not address the ambiguity the
reviewer has already identified. Disposition B's glossary-callout idea
is attractive in principle, but it leaves the per-page prose unchanged,
so the same sentences the reviewer flagged would still be ambiguous on
first read; B is best treated as a *complement* to A rather than a
substitute, and could be raised as a separate follow-up if the workgroup
agrees the convention is worth formalising. Disposition C
under-weights the fact that the ambiguity has been flagged from inside
the work group and that the proposed change is non-breaking.

The main thing the workgroup should sanity-check before accepting A is
the editor's interpretation of "appropriate instances": the briefing's
gotchas (do not touch generated parameter tables, do not rename uses of
`code` as data-type / element / parameter name) must be enforced
sentence-by-sentence on each of the thirteen pages, not by a blind
find-and-replace. With that constraint, the change is low-risk and
high-clarity.
