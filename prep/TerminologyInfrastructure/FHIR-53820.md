## Summary

The ticket asks the Terminology Infrastructure work group to refresh the
conceptual diagram in §4.0.1 *Introduction* of the FHIR Terminology Module
page. The reporter wants the boxes renamed to use the canonical resource
type names (CodeSystem, ValueSet, NamingSystem, ConceptMap, ElementDefinition),
inline definitions stripped (because the boxes will hyperlink directly to
the resource pages), and the example content tightened (e.g., a clearer
SNOMED CT / ICD-10 ConceptMap example, italicised concept designations).
It also asks for two new boxes — `TerminologyCapabilities` and `FHIR
Terminology Service` — with explicit "References" and "Conforms to"
relationships, a relabelled "References" edge from the coded data type to
CodeSystem, and visual cleanup (color-coded resource categories with a
legend, removal of dotted lines around Element Definition, and clickable
boxes that link to the corresponding resource pages). The supplied
guidance points implementers at the PlantUML clickable-diagram pattern
already used on `datatypes.html`. No cross-referenced Jira tickets, GitHub
items, or Zulip threads were surfaced for this issue, so the request
stands on its own.

## Details

**Description:**

Make the following changes to the diagram in **4.0.1 Introduction** (note
that definitions are being removed because we are linking directly to the

1. In the Code System box, change label to "CodeSystem", add a couple of

5. In the Element (instance) Coded Data Type box: remove "(instance)",
   remove `code/Coding/CodeableConcept`, add a couple of blank lines

   <https://build.fhir.org/elementdefinition.html>.
7. Change the relationship line between the Element Coded Data Type and
   CodeSystem to "References" (from "Refers to").
8. Add a new box for `TerminologyCapabilities` with "References"
   relationships to CodeSystem, ValueSet, NamingSystem, and ConceptMap.
9. Add a new box for `FHIR Terminology Service` with a "Conforms to"

10. Other cleanup:
    1. Add links from boxes to the resource pages (CodeSystem in the box

       <https://build.fhir.org/ig/FHIR/ig-guidance/diagrams-plantuml.html>

    3. Add a legend describing the colors used.
    4. Remove the dotted lines around Element Definition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the diagram embedded in §4.0.1 *Introduction* of
`terminology-module.html` (FHIR Core) to match every change in the
ticket:

- Rename the boxes to `CodeSystem`, `ValueSet`, `NamingSystem`,
  `ConceptMap`, "Element Coded Data Type" (drop the "(instance)"
  qualifier and the inline `code/Coding/CodeableConcept` list), and
  "Element Definition".
- Remove the inline definitions and the dotted border around the
  Element Definition box; keep one example per box separated by blank
  lines (SNOMED CT International for CodeSystem, *SNOMED CT Fracture
  disorder concepts* for ValueSet, the existing OID/URI for
  NamingSystem, an italicised "*S52.2 Fracture of shaft of ulna
  (ICD-10)*" target on the ConceptMap edge labelled `equivalent`).
- Relabel the Element Coded Data Type → CodeSystem edge from "Refers
  to" to "References".
- Add a new `TerminologyCapabilities` box with `References` edges to
  `CodeSystem`, `ValueSet`, `NamingSystem`, and `ConceptMap`, plus a
  new `FHIR Terminology Service` box with a `Conforms to` edge to
  `TerminologyCapabilities`.
- Re-author the diagram as a clickable PlantUML diagram following the
  pattern at
  `https://build.fhir.org/ig/FHIR/ig-guidance/diagrams-plantuml.html`
  (mirroring how `datatypes.html` does it). Each box should link to its
  resource page (`codesystem.html`, `valueset.html`, `namingsystem.html`,
  `conceptmap.html`, `terminologycapabilities.html`, `elementdefinition.html`).
- Apply the requested color scheme — one color for the four terminology
  resource boxes, a second color for non-terminology boxes (Element
  Coded Data Type, Element Definition), and a third color for the FHIR
  Terminology Service box — and add a legend explaining the palette.

The change is purely narrative/diagrammatic: it touches the
`terminology-module.html` source page (and the associated diagram
source if the diagram is moved into a `.plantuml`/`.puml` file under
the source tree). No structure-definition, value-set, or
schema artefact changes; no FMM-impacting normative content.

#### Justification

The reporter's design is internally consistent: removing inline
definitions while making the boxes hyperlinks avoids drift between the
diagram and the resource pages, and the colour scheme + legend
materially improve scannability of which boxes are conformance
resources vs. data-model elements vs. service-layer concepts. Adding
`TerminologyCapabilities` and `FHIR Terminology Service` closes a
genuine gap — the current diagram shows the conformance resources but
omits the service contract that ties them together, even though
`TerminologyCapabilities` has been a first-class resource for several
ballot cycles. The PlantUML/clickable-diagram pattern is already an
established FHIR convention (the example reference in the ticket,
`datatypes.html`, ships with it today), so adopting it for
terminology-module aligns with house style rather than introducing a
new mechanism. The Zulip thread surfaced above also reinforces that
module-level diagrams are intended to be conceptual and navigable —
exactly what the proposed redesign delivers.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substantive changes (rename boxes, drop inline definitions,
add `TerminologyCapabilities`, add `FHIR Terminology Service`, relabel
"Refers to" → "References", clickable links, colour legend) but defer
two pieces of styling for a follow-up:

1. Keep the current diagram-rendering technology (e.g., the existing
   SVG/PNG asset) for this update and only *migrate* to a clickable
   PlantUML source if the maintainer doing the work confirms the
   PlantUML pipeline supports the desired layout cleanly. If PlantUML
   rendering fights the requested layout, ship an updated SVG with
   `<a xlink:href="…">` link wrappers instead — the user-visible
   "clickable boxes" outcome is the same.
2. Treat the `code/Coding/CodeableConcept` list inside the "Element
   Coded Data Type" box as informative and move it to a short caption
   below the diagram rather than dropping it outright, so readers
   landing on the page from a search engine still see which datatypes
   the box represents without having to click through.

All other changes — including the new `TerminologyCapabilities` and
`FHIR Terminology Service` boxes, the relationship relabelling, the
ConceptMap example refresh, the colour scheme, the legend, and removal
of the dotted Element Definition border — are applied as requested.

#### Justification

This disposition delivers the reporter's intent (cleaner, link-rich,
service-aware diagram) while leaving the rendering pipeline decision to
whoever holds the editor's pen. PlantUML is the strategic direction for
diagrams in HL7 specifications and `datatypes.html` proves it works,
but a few module diagrams still ship as hand-authored SVG/PNG; if
re-authoring in PlantUML balloons the ticket scope, an SVG with
embedded links is a credible Plan B that still meets the reader's
needs. Keeping `code/Coding/CodeableConcept` as a caption preserves the
context that the current diagram conveys without re-cluttering the
box, which matches the spirit of "remove inline definitions while
keeping a useful example".

---

### Disposition C: Decline

#### Proposal

Decline to make any of the requested changes; close as Not Persuasive
or Considered – No Change.

#### Justification

The only defensible reason to decline is that this is a non-normative
narrative refresh of a single conceptual diagram on a module
introduction page, with no comments, no community discussion, and no
cross-referenced tickets indicating broader interest. A workgroup that
has limited editorial bandwidth could argue the request is purely
cosmetic and that the existing diagram, while dated, conveys the
essential relationships. This rationale is weak: the ticket *does*
identify a real content gap (`TerminologyCapabilities` and `FHIR
Terminology Service` are absent from a diagram that purports to
introduce the terminology module), so a flat decline would leave a
documentation defect in place.

---

### Recommendation

**Recommended disposition:** A

The ticket is well-scoped, the changes are concentrated in one
non-normative diagram on a module introduction page, and the substantive
additions (the `TerminologyCapabilities` and `FHIR Terminology Service`
boxes, the "References"/"Conforms to" relationship labels, and the
colour-legend treatment) close a genuine gap in how the module is
introduced to readers. The PlantUML/clickable-diagram pattern called
out in the ticket is already in use elsewhere in the spec
(`datatypes.html`), so there is an existing template to follow rather
than a new tooling commitment. Disposition B is a reasonable fallback
if the editor finds the PlantUML pipeline awkward for the requested
layout, but it should be invoked only as a tactical adjustment during
implementation rather than as the disposition of record. Disposition C
would leave a documented, uncontested defect in place and is not
recommended.
