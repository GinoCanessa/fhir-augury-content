## Summary

The reporter (Gay Dolin) observes that the CRMI IG's "Artifact Status" list
does not include a "Deprecated" value, even though VSAC supports both
"Retired" and "Deprecated", "Deprecated" has historical use in C-CDA, and
the CRMI IG itself uses "Deprecated" elsewhere when describing its own
artifacts. She requests adding "Deprecated" as a distinct artifact status,
defined as an artifact that is "still available/valid, but discouraged for
use" — signalling implementers that a better/newer/safer alternative
exists and that they should migrate, while the artifact continues to work
for backward compatibility. The intent is to clearly distinguish
"Deprecated" from "Retired".

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for FHIR-52593 by the augury index. The Jira comment thread,
however, makes the design problem explicit: the CRMI IG is reusing the
canonical resources' base `status` element, whose value set
(`PublicationStatus`) is fixed by core FHIR — adding a new code there
would be an R6 base-spec change touching every canonical resource, and
"retired" already loosely covers "should not be used". The work group
therefore considered an extension-based approach (`standards-status`,
`template-status-code`, a new `lifecycleStatus` / `releaseStatus`) rather
than a code-system change, which the resolution ultimately adopts: the
set of base lifecycle statuses will not be modified, but ShareableValueSet
will gain the existing `valueset-workflowStatusDescription` extension,
with documented usage for "deprecated" and "retired".

## Details

**Description:**

The artifact Status list (https://hl7.org/fhir/uv/crmi/2025Sep/artifact-lifecycle.html#artifact-status) appears to be missing the status **"Deprecated."**

VSAC supports both **Retired** and **Deprecated**, and "Deprecated" has also been used historically in C-CDA. To align with VSAC and common IT usage, please consider adding **"Deprecated"** as a distinct status and clarifying its meaning in contrast to **"Retired."** Of note is that you also use "Deprecated" as a status in the CRMI IG artifacts.

**Proposed definition of Deprecated:**

- **Definition:** Still available/valid, but discouraged for use.
- **Purpose:** Signals to implementers that there is a better, newer, or safer alternative, and they should migrate away from the deprecated item.
- **Status:** The feature, function, or standard continues to exist and usually still works for backward compatibility.

**Resolution (Persuasive with Modification):**

> We cannot modify the set of lifecycle statuses established by base FHIR for artifacts generally, but we can support identifying when an artifact has an additional status of deprecated. For value sets specifically, this is done with the valueset-workflowStatusDescription extension. We will add this extension to the ShareableValueSet pattern and document its use for describing "deprecated" and "retired" status.

**Comments:**

- **bryn.rhodes (2025-09-24):** Notes the underlying constraint — CRMI is just reusing the canonical resource `status` element, so adding "deprecated" would require an R6 base-spec change across all canonical resources. Observes that `PublicationStatus` "retired" ("should not be used") already overlaps with "deprecated", and suggests this might instead be indicated with a separate flag.
- **bryn.rhodes (2025-09-24):** Points out that CRMI already uses the core spec's `standards-status` extension to mark CRMI features as "deprecated" (links to GitHub code search in HL7/crmi-ig), suggesting an extension with a separate status for deprecation may be the right approach.
- **bryn.rhodes (2025-09-24):** Floats a `releaseStatus` extension with values such as "pre-release", "release", "deprecated".
- **bryn.rhodes (2025-09-24):** Suggests considering existing extensions `ValueSet-template-status-code` and `ValueSet-standards-status`, possibly a `lifecycleStatus`.
- **bryn.rhodes (2025-10-22):** Asks Gay Dolin to clarify the distinction between "deprecated" and "retired" given her proposed definition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new `deprecated` code to the CRMI Artifact Status enumeration on
the Artifact Lifecycle page, alongside the existing values, with the
reporter's proposed definition: "Still available/valid, but discouraged
for use; a better, newer, or safer alternative exists and implementers
should migrate. The artifact continues to exist and typically still works
for backward compatibility." Update the Artifact Lifecycle narrative in
the CRMI IG (`HL7/crmi-ig`) to define `deprecated` and contrast it
explicitly with `retired` (e.g., retired = should not be used at all;
deprecated = still usable but discouraged), and ensure any CRMI
ShareableX patterns that reference artifact status reflect the new
value.

#### Justification

This matches what the reporter asked for, aligns CRMI's vocabulary with
VSAC and historical C-CDA usage, and removes the inconsistency the
reporter flagged where CRMI itself describes its own features as
"deprecated" but does not expose that as an artifact status. It also
matches a recurring community ask (see FHIR-31028, FHIR-17214,
FHIR-44608).

However, as bryn.rhodes pointed out in the comment thread, the artifact
status surfaced on canonical resources is the base FHIR `status` element
bound to `PublicationStatus`, which is a *required* binding fixed by
core FHIR. CRMI cannot unilaterally add `deprecated` to that code
system; doing so is an R6 base-spec change touching every canonical
resource. The most this disposition can honestly do inside CRMI is add
an additional CRMI-specific lifecycle vocabulary that lives *alongside*
`status`, which leads naturally to Disposition B.

---

### Disposition B: Alternative Approach (the resolution actually adopted)

#### Proposal

Do not modify the base lifecycle status set. Instead, attach the existing
`valueset-workflowStatusDescription` extension to the ShareableValueSet
pattern in the CRMI IG (`HL7/crmi-ig`) and add narrative on the Artifact
Lifecycle page documenting how to use it to express that an artifact is
"deprecated" or "retired" (with the reporter's "still valid, discouraged
for use, prefer alternative" definition for "deprecated", and a clear
contrast with "retired" = should not be used). Specifically:

1. In the ShareableValueSet StructureDefinition / pattern definition,
   declare `valueset-workflowStatusDescription` as an applicable
   extension (must-support / pattern-level guidance, not a new element).
2. On the Artifact Lifecycle page, add a "Deprecated and retired"
   subsection that defines both terms and shows worked examples using
   the extension.
3. Mention the existing `standards-status` extension as the analogous
   mechanism CRMI already uses to mark its own IG features as
   "deprecated", and clarify when authors should reach for which
   extension.

This is exactly what the recorded resolution describes ("Persuasive with
Modification: …add this extension to the ShareableValueSet pattern and
document its use for describing 'deprecated' and 'retired' status").

#### Justification

This addresses the underlying need — letting publishers signal "this
value set is on its way out, prefer the replacement" — without breaking
the FHIR-wide invariant that the canonical `status` element uses the
required `PublicationStatus` binding. It avoids an R6-only change that
would block CRMI delivery, reuses an extension that already exists in
core FHIR (so no new artifact has to be invented or balloted), and stays
consistent with how CRMI already marks deprecation in its own IG via
`standards-status`. It also lets the work group ship in CRMI's own
release cycle rather than waiting on a base-spec change.

The trade-off vs. Disposition A is that "deprecated" is not surfaced as a
top-level status code, so tools that only look at `ValueSet.status` will
still see `active`. The narrative guidance and pattern-level
must-support on the extension are intended to mitigate this.

---

### Disposition C: Decline

#### Proposal

Decline the request to add `deprecated` as a CRMI artifact status, on
the grounds that (a) the artifact status enumeration is the base FHIR
`PublicationStatus` value set with a required binding, which CRMI is not
permitted to extend, and (b) `retired` already covers "should not be
used", which subsumes the practical effect of deprecation for a value
set. Direct the reporter to FHIR-31028 (the open base-spec request to
add `deprecated` to `PublicationStatus`) as the correct venue.

#### Justification

This is the cleanest answer if the work group's view is that artifact
status must remain a single, FHIR-wide vocabulary and that CRMI should
not introduce a parallel CRMI-only lifecycle vocabulary via an
extension. It also avoids the risk of two competing signals
(`ValueSet.status` vs. an extension) disagreeing on the same artifact.

The downside is real: it leaves the inconsistency the reporter flagged
unresolved (CRMI calls some of its own things "deprecated" but offers
publishers no standard way to do the same on their value sets), and it
forces the question onto a much slower R6 base-spec timeline. The
community has been asking for "deprecated" semantics across multiple
tickets (FHIR-17214, FHIR-31028, FHIR-44608), suggesting that simply
declining is unlikely to make the ask go away.

---

### Recommendation

**Recommended disposition: B**

Disposition B is the right call and matches what the work group has
already resolved this ticket as ("Persuasive with Modification"). It
honours the substantive ask — give publishers a standard way to say
"deprecated, prefer the replacement" on a value set — without
overreaching into base FHIR's required `PublicationStatus` binding,
which is the technical blocker bryn.rhodes correctly identified in the
Jira thread. It also reuses an existing core extension
(`valueset-workflowStatusDescription`) rather than minting new
machinery, which keeps the change small, ballotable inside CRMI's own
cycle, and consistent with CRMI's existing use of `standards-status` for
its own deprecations. Disposition A is the better answer in principle
but is not deliverable in CRMI alone; the work group should route that
broader fight to FHIR-31028. Disposition C under-serves a recurring
community request and leaves the inconsistency the reporter flagged in
place.
