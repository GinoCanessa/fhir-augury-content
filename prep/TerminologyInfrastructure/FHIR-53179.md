## Summary

The ticket requests that FHIR document (or define) an extension on
`TerminologyCapabilities.codeSystem` that lets a terminology server advertise
its **default SNOMED CT Edition**, in addition to the default *version* that
`TerminologyCapabilities` already supports. SNOMED CT is unusual among code
systems in that the substantive content varies not only by version (a date)
but also by Edition (International, Australian, Dutch, US, etc.), each
identified by its module URI under `http://snomed.info/sct/{moduleId}`.
Ontoserver already publishes such an extension at
`http://ontoserver.csiro.au/fhir/StructureDefinition/defaultsnomededition`
with a `valueUri` carrying the edition module URI (e.g.
`http://snomed.info/sct/32506021000036107` for the AU Edition). The reporter
asks that this be adopted as-is, or that an equivalent extension be defined
in the `fhir.org` namespace so all servers and clients have a single,
standard way to negotiate the default Edition.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the augury index for this ticket, so the request stands on its
own at this point.

## Details

**Description:**

Currently the TerminologyCapabilities resource allows a TX server to specify
the default version for any given CodeSystem, but in the case of SNOMED CT
there may also be the concept of a default Edition (e.g., the International
or Australian or Dutch or ...).

Ontoserver uses an extension as shown below to indicate this. It would be
great if this extension could be adopted by others, or define an equivalent
extension in the fhir.org namespace.

```json
{
  "codeSystem": [{
    "extension": [{
      "url": "http://ontoserver.csiro.au/fhir/StructureDefinition/defaultsnomededition",
      "valueUri": "http://snomed.info/sct/32506021000036107"
    }],
    "uri": "http://snomed.info/sct",
    "version": [ ... ]
  }]
}
```

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new standard extension in the FHIR core namespace,
e.g. `http://hl7.org/fhir/StructureDefinition/terminologycapabilities-defaultSnomedEdition`,
with `Extension.value[x] = uri`, intended to carry a SNOMED CT Edition
module URI under `http://snomed.info/sct/{sctid}`. Bind the extension
to `TerminologyCapabilities.codeSystem` (with cardinality 0..1,
applicable only when `codeSystem.uri = http://snomed.info/sct`). Add an
explicit usage note to the SNOMED CT section of the FHIR terminology
documentation describing how `version` (the version date) and the new
default-Edition extension interact, and reference the existing
Ontoserver extension as the prior art the new extension is intended to
replace. Mention the extension on the `TerminologyCapabilities` page and
in the SNOMED CT terminology guidance.

#### Justification

This is the most direct way to satisfy the underlying need: clients
that issue `$expand`, `$validate-code`, and `$lookup` against a SNOMED
CT code system without specifying an Edition module need a deterministic
way to discover what the server will assume. `TerminologyCapabilities`
is the right place because it already advertises the default *version*
for each code system; default *Edition* is the natural sibling concept
for SNOMED CT specifically. Standardising the URL under `hl7.org/fhir`
(rather than only documenting Ontoserver's extension) avoids vendor
lock-in and lets every server and every client toolchain rely on a
single canonical URL.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a new extension, extend the SNOMED CT version
representation itself. The SNOMED CT version URI grammar already
supports concatenating an Edition module with a version date as
`http://snomed.info/sct/{moduleId}/version/{date}`. Update the FHIR
SNOMED CT documentation and `TerminologyCapabilities` guidance to state
that, for SNOMED CT, the value of
`TerminologyCapabilities.codeSystem.version.code` MAY be either a bare
date (current behaviour) **or** a fully-qualified
`{moduleId}/version/{date}` form, and that the server's default version
entry SHOULD use the fully-qualified form when an Edition is being
advertised. No new extension is required; clients that already parse
the SNOMED CT URI grammar will get the Edition for free.

#### Justification

This avoids growing the extension surface for a problem that the SNOMED
CT URI scheme already encodes. It keeps `TerminologyCapabilities`
slim, reuses the same machinery the server uses on every other SNOMED
CT operation (`system|version` parameters, expansion identifiers, etc.),
and aligns with how SNOMED International itself recommends identifying
Editions. The trade-off vs. Disposition A is that older clients
inspecting `TerminologyCapabilities` may not realise the version string
encodes both Edition and date; that is a documentation problem, not a
modelling one.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for FHIR Core terminology metadata, on the
grounds that the SNOMED CT URI grammar
(`http://snomed.info/sct/{moduleId}[/version/{date}]`) is already
sufficient to identify any SNOMED CT Edition, and that any server
wishing to advertise a default Edition can already do so by emitting
that fully-qualified URI in `TerminologyCapabilities.codeSystem.version`.
Servers (such as Ontoserver) that want a separate, explicit extension
remain free to publish one in their own namespace, exactly as they do
today.

#### Justification

The case for a new core extension rests on explicitness, not on missing
expressive power. FHIR has historically been reluctant to add SNOMED-CT-
specific extensions to general resources when the SNOMED CT URI grammar
already covers the case. Declining keeps `TerminologyCapabilities` free
of code-system-specific extension proliferation and avoids the
maintenance cost of a parallel mechanism that would have to be kept in
sync with the SNOMED CT URI documentation.

---

### Recommendation

**Recommended disposition:** A

A first-class core extension is the cleanest fit for the actual
implementer experience. `TerminologyCapabilities` is already the
discovery surface clients use to learn server defaults; adding a
SNOMED-CT-specific `defaultSnomedEdition` extension makes the default
Edition machine-readable in the same place implementers already look
for the default version, with no ambiguity about whether a given
`version` string is "just a date" or "Edition plus date". Disposition B
is technically sufficient but bets that every client correctly parses
the long form of the SNOMED CT URI, which experience with existing
implementations suggests is fragile. Disposition C is defensible from a
purity standpoint but ignores the practical reality that Ontoserver has
already shipped this extension and others are likely to clone it under
their own URLs if FHIR Core does not standardise one. Standardising
under `hl7.org/fhir` once now is cheaper than reconciling several
vendor extensions later.
