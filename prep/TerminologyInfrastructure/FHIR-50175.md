## Summary

The reporter (Alexander Kiel) asks that `ValueSet/$validate-code` gain a
`system-version` parameter so it mirrors `ValueSet/$expand`, which already
defines `system-version`, `check-system-version`, and `force-system-version`.
The request originated in a Zulip exchange where Grahame Grieve agreed that
the parameter "should be listed" on `$validate-code` and explicitly invited a
ballot ticket. While this ticket is in `Triaged`/Not Persuasive state, the
substantive change has in fact already been delivered by the duplicate
[FHIR-52907](https://jira.hl7.org/browse/FHIR-52907) ("Clarification of use
of system parameters in expansion parameters in $validate-code") raised by
Grahame Grieve in September 2025. FHIR-52907 was voted Persuasive with
Modification at the September 2025 WGM and Applied: it adds `system-version`,
`check-system-version`, and `force-system-version` to `ValueSet/$validate-code`
with definitions copied from `$expand`, and adds wording to distinguish the
new `system-version` from the existing `systemVersion` parameter on
`$validate-code`. TI's most recent tracker discussions (2025-12-08 and
2025-12-15) confirm 50175 is a duplicate but note that "Duplicate" is not
available as a resolution and the request can no longer be rejected, so they
are routing it to a "Not Persuasive" disposition with explanatory text. A
secondary concern raised on the 2025-12-08 call is that the parameter
descriptions on `$expand` (and now copied to `$validate-code`) are of poor
quality and could benefit from rewrite plus a table describing the
combinations of code-system and value-set version pinning - that work is
queued for the January 2026 WGM joint with FHIR-I.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add `system-version`, `check-system-version`, and `force-system-version` as
input parameters to `ValueSet/$validate-code`, with definitions copied from
`ValueSet/$expand`. Add wording to disambiguate the new `system-version`
parameter from the existing camelCase `systemVersion` parameter on
`$validate-code`. Note (without spec change) that `force-system-version`
does not override `Coding.version`.

#### Justification

This is precisely what the reporter asks for and aligns `$validate-code` with
`$expand`, which already supports the triplet. It also matches Grahame
Grieve's own position in the Zulip discussion ("it should be listed"). The
change has, in substance, already been applied via FHIR-52907 (resolved
Persuasive with Modification at the September 2025 WGM), so accepting this
ticket affirms the work and the rationale. This treatment also matches what
TI previously proposed at the May 2025 WGM (per Carmela's 2025-12-08 note)
before duplication was discovered.

---

### Disposition B: Alternative Approach

#### Proposal

Mark FHIR-50175 as **Persuasive with Modification** (rather than Not
Persuasive) and use the resolution to:

1. Acknowledge that the requested parameters have been added via FHIR-52907
   (resolution text references 52907 as the implementing change);
2. Commit to a follow-up editorial pass on the `$expand` parameter
   descriptions for `system-version`, `check-system-version`, and
   `force-system-version` (replacing the boilerplate "Specifies a version to
   use for a valueset" and the misleading "Note that this is similar to the
   force-system-version parameter but applied to valuesets" sentence) and
   then mirror the improved text into `$validate-code`;
3. Add a small explanatory table to both operation pages covering the
   combinations of code-system and value-set version pinning (default /
   check / force), so implementers don't have to read three near-identical
   parameter definitions to understand the relationships;
4. Reconsider `default-system-version` as a clearer alias name (per Bryn
   Rhodes's 2025-06-12 question) - or explicitly close that question with
   "no, we keep `system-version` for consistency with `$expand`".

This bundles the literal request and the descriptive-quality concern that TI
flagged on 2025-12-08, and routes the editorial work to the planned January
2026 WGM joint with FHIR-I.

#### Justification

The descriptive-quality issue is real and was explicitly raised by attendees
on the 2025-12-08 TI Tracker call; closing 50175 as Not Persuasive risks
losing that thread. Marking it Persuasive with Modification keeps both
strands - the parameter addition (already done) and the description rewrite
(still to do) - tied to a single ticket, which is more discoverable than a
new follow-up ticket. It also avoids the awkward optics of a "Not
Persuasive" disposition on a request that has, in practice, been adopted.
Trade-off vs. A: requires the workgroup to commit, on the record, to the
description rewrite work; if that work is not actually planned, this
overstates the resolution.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** with a resolution note pointing to FHIR-52907
as the ticket that actually delivered the requested parameters; defer any
work on parameter description quality to the January 2026 WGM agenda item
already scheduled (joint with FHIR-I). This is essentially the disposition
TI already drafted on 2025-12-15 after discovering that "Duplicate" is not a
selectable resolution and that the request can no longer be marked
Rejected.

#### Justification

Defensible because the change exists in the spec via the duplicate ticket;
re-applying it under 50175 would either be a no-op or risk unintended drift.
The Jira workflow simply does not offer a clean "Duplicate / Already
Applied" outcome at this stage, and Not Persuasive with a clear pointer to
52907 is the pragmatic alternative. Trade-off: the disposition reads as if
TI thinks the request is wrong, when in fact TI agrees with it; the
resolution text must do the heavy lifting to avoid implementer confusion.
Also leaves the description-quality work entirely outside this ticket, which
is fine if the January WGM session reliably produces a follow-up ticket -
but loses traceability if it doesn't.

---

### Recommendation

**Recommended disposition:** B

Disposition B (Persuasive with Modification, with an editorial follow-up)
best matches the actual record: the parameters Alexander Kiel asked for have
been added via FHIR-52907, and the workgroup itself flagged on 2025-12-08
that the bigger remaining problem is the poor quality of the parameter
descriptions (and the absence of a table describing the version-pinning
combinations). A "Persuasive with Modification" outcome captures both
realities, ties the editorial work to a discoverable ticket, and avoids the
optics problem of voting "Not Persuasive" on a request that the spec has
quietly already adopted. Disposition A is essentially a re-statement of the
already-applied change and risks duplicate edits; Disposition C reflects
where TI landed on 2025-12-15 only because the Jira workflow did not allow
"Duplicate" - a workflow constraint, not a substantive disagreement. If
Joshua Procious can enable a "Duplicate" resolution path before vote, that
becomes a clean tie-breaker between B and C; absent that, B is the safer
choice because it preserves the description-rewrite commitment.
