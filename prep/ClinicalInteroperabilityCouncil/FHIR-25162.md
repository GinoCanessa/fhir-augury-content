## Summary

The ticket (titled "Initial Report - BC #113") proposes adding a dedicated
section for an "Initial Report" to the US Breast Cancer Radiology Reporting
(FHIR) IG. The reporter (Susan Matney) observes that the BreastRadiology
profile already exposes a `PriorReports` slot that references prior
BreastRadiology reports, but there is no symmetric "Initial" report slot
to identify or reference the first/baseline report in a series. The
request is to add such an "Initial Report" section/slot so an Initial
report can be explicitly named and referenced alongside the existing
prior-reports mechanism. The ticket has been triaged and a resolution
note ("Need more info. Will discuss in meeting.") indicates the
workgroup wants additional clarification before deciding. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the augury index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an explicit "Initial Report" section/slot to the BreastRadiology
profile in the US Breast Cancer Radiology Reporting IG, mirroring the
existing `PriorReports` mechanism. Concretely, this would be a new
slice on `DiagnosticReport.result` (or, depending on the IG's current
modelling, a new section/entry on the underlying Composition/Bundle)
that explicitly identifies the initial/baseline BreastRadiology report
in a longitudinal series. The slice would be `0..1`, typed as a
`Reference(BreastRadiology)`, and given a fixed code/identifier (e.g.,
`initial-report`) parallel to the code used by the existing prior-reports
slice. Add narrative guidance on when to populate "Initial Report"
versus "Prior Reports", and update example instances to demonstrate
both.

#### Justification

This directly satisfies the reporter's observation that the profile is
asymmetric: it already supports referring back to prior reports but has
no first-class way to mark/refer to the initial one. Making "initial"
explicit is consistent with breast-imaging clinical workflow, where
comparison against the baseline study is a recurring and clinically
meaningful operation distinct from "any prior". A bounded `0..1`
addition with a fixed code is low-risk, preserves backward compatibility
for senders that don't populate it, and follows the same slicing pattern
already used for `PriorReports`.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a new section/slot, address the use case via the
existing `PriorReports` mechanism by introducing a small extension or a
fixed `code`/`extension` on each prior-report reference that flags one
of them as the *initial* report (e.g., a boolean extension
`isInitialReport` on the prior-report entry, or a coded role with values
such as `initial | interim | most-recent`). Document that the initial
report is conveyed as a tagged member of the existing prior-reports
collection. Provide guidance and an example demonstrating the tag.

#### Justification

This avoids introducing a second top-level structural element for what
is conceptually one ordered list of related reports — the initial
report is, by definition, the earliest member of "prior reports".
Tagging is cheaper to implement for senders that already populate
`PriorReports`, and it scales naturally if the workgroup later wants to
flag other roles (e.g., "most-recent comparison", "interim"). The
trade-off versus Disposition A is discoverability: consumers must know
to inspect the tag instead of reading a dedicated slot. This is the
preferable approach if the workgroup believes "initial" is just one of
several semantic roles within prior reports.

---

### Disposition C: Decline

#### Proposal

Decline the change. The current BreastRadiology profile already permits
referencing any prior BreastRadiology report via `PriorReports`; the
initial report is reachable as the chronologically earliest entry in
that collection (or by following each report's own `PriorReports`
references back to the root). No structural change is required, and
implementation guidance can be added to the IG narrative (without a
profile change) explaining how to identify the initial report from
existing references.

#### Justification

A defensible decline if the workgroup judges that (a) the use case is
sufficiently served by ordering/traversal of `PriorReports`, (b) adding
an "Initial Report" slot risks ambiguity when senders disagree about
which report is "initial" (e.g., baseline at this institution vs.
absolute first ever), and (c) the IG is already at a stage where
profile changes have a high implementer-cost relative to the marginal
benefit. The existing resolution note ("Need more info. Will discuss in
meeting.") leaves room for declining once the workgroup determines that
the use case can be met with guidance rather than a new structural
element.

---

### Recommendation

**Recommended disposition:** A

The reporter's observation is concrete and structural: `PriorReports`
exists, but its semantic counterpart for "the first/baseline report"
does not, leaving consumers to infer initiality by walking back through
the prior-report chain. A bounded, optional `Reference(BreastRadiology)`
slot (Disposition A) makes "Initial Report" first-class, matches the
existing prior-reports pattern (low cognitive load for implementers),
and is backward-compatible for senders that don't populate it.
Disposition B (tagging via extension) is reasonable but defers the same
work onto every consumer and obscures a clinically important concept
behind an extension lookup. Disposition C is hard to defend given that
the reporter explicitly asked for the slot and the resolution note
("Need more info. Will discuss in meeting.") signals the workgroup is
open to the change pending clarification — that clarification is the
right place to confirm cardinality (`0..1` vs. `0..*`) and the exact
fixed code, not to walk away from the request entirely.
