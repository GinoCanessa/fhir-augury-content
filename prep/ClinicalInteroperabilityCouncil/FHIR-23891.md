## Summary

The reporter (Craig Newman) raises a structural/IG-organization complaint
against the **2019SEP ballot of the US Breast Cancer Radiology Reporting
(FHIR)** Implementation Guide. Three concrete problems are called out:

1. **Broken/ambiguous reference target** — `Composition.relatesTo.targetReference`
   points at an `obf-Composition` profile that does not resolve. The
   reporter asks whether the intended target is the IG's own
   `breastrad-BreastRadiologyDocument`, and notes that if `obf-Composition`
   is real, its hyperlink should work.
2. **Profiles index page is incomplete** — the IG's profiles index omits
   several profiles that are reachable in the build but not listed:
   `obf-Condition`, `obf-ActionRequest`, `obf-Procedure`, and
   `obf-MedicationStatement`. The reporter argues these must appear in
   the index because they are referenced from the Composition and its
   related resources.
3. **`obf-Observation` is under-described** — the profile that
   presumably carries the Patient Risk observations has no narrative
   guidance on intended use. The reporter requests narrative text (or
   constraints) sufficient for an implementer to use it correctly.

A follow-up comment from the reporter generalises the complaint: many
profiles carry generic descriptions inherited from the underlying logical
model (e.g., `Condition` listed with the entire Condition value-set
gloss), the `ConditionAbsent` reference is unexplained, and constraints
such as `Condition.stage` reduced from `0..*` to `0..1` are imposed
without guidance on intended use.

The cross-reference scan found no other Jira tickets, Zulip threads, or
GitHub items linked to this issue. The ticket was resolved
**Persuasive** on 2019-11-06 with the resolution note: *"This is an
artifact of the CIMPL logical model. For the next ballot we will not be
using these logical models and instead adhere more closely to the base
FHIR resources."* — i.e., the Work Group accepted the underlying
critique and addressed it by removing the CIMPL-derived logical-model
layer for the subsequent ballot rather than by patching individual
profiles.

## Proposed Dispositions

> Note: the ticket has already been resolved **Persuasive** by the work
> group (the resolution narrative is reproduced in **Details**). The
> dispositions below are written as if the ticket were still open, so
> the work group can sanity-check that the chosen path matches one of

### Disposition A: Accept As Requested

#### Proposal

Patch the **2019SEP ballot** of `us/breast-radiology` in place to address
each defect the ticket lists, without changing the underlying CIMPL-derived
logical-model architecture:

1. **Fix `Composition.relatesTo.targetReference`** in
   `StructureDefinition-breastrad-BreastRadiologyDocument` (and any
   sibling Composition profile in the IG): replace the dangling
   `obf-Composition` reference with the canonical of the actual target
   profile (`breastrad-BreastRadiologyDocument` if self-referential, or
   the correct OBF profile canonical if one exists). Re-publish so the
   hyperlink resolves in the rendered IG.
2. **Add the missing profiles to the Profiles index page** of the IG
   (`profiles.html` / `artifacts.html`) so that `obf-Condition`,
   `obf-ActionRequest`, `obf-Procedure`, and `obf-MedicationStatement`
   are listed, grouped under the OBF library section, with descriptions
   that explain their role in the Breast Radiology Composition.
3. **Author IG-specific narrative for `obf-Observation`** describing
   how it carries Patient Risk observations: which `code`s are expected,
   the cardinality and usage of `value[x]`, `interpretation`, `bodySite`,
   and how it is referenced from the Breast Radiology Composition's
   sections. Apply the same narrative-uplift pattern to `obf-Condition`
   (clarify the intended condition scope), explicitly link or remove
   the `ConditionAbsent` reference, and document the rationale for
   `Condition.stage 0..1` plus guidance on the (still-example) value
   set.

#### Justification

This is the most direct response to what the reporter asked for: it
fixes broken navigation, completes the index, and replaces generic
boilerplate with IG-specific guidance — all hallmarks of an
implementer-ready IG. It preserves the existing artifact set, so
implementations that have already started against the 2019SEP ballot
remain valid, and it keeps the CIMPL/OBF tooling pipeline that the WG
has already invested in. From a FHIR-IG-quality perspective, the
profile-page/profile-narrative defects called out here are exactly the
class of issue ballot reconciliation is meant to catch.

---

### Disposition B: Alternative Approach (the path actually taken)

#### Proposal

Withdraw the CIMPL-derived `obf-*` logical-model layer for the **next
ballot** of `us/breast-radiology` and re-author the IG's profiles
**directly against the base FHIR resources** (`Composition`,
`Condition`, `Procedure`, `MedicationStatement`, `Observation`, …),
keeping only the breast-radiology-specific constraints, slices, and
narrative. As part of that re-author:

- Replace `obf-Composition` references with a directly-authored
  `BreastRadiologyComposition` (or equivalent) profile and ensure all
  `Composition.relatesTo.targetReference` targets resolve to profiles
  shipped in this IG.
- Curate the Profiles index page from the directly-authored set so
  there is no possibility of a profile being reachable but unlisted.
- Write IG-specific narrative for every profile up front (no inherited
  generic logical-model descriptions), and document constraints such as
  `Condition.stage 0..1` with their clinical rationale and bound value
  set.

#### Justification

The reporter's individual defects (broken link, missing profiles,
generic descriptions, unexplained constraints) are symptoms of a single
root cause: the IG is being projected out of a CIMPL logical model whose
generated artifacts (`obf-*`) are leaking into the published IG with
their generic, model-driven descriptions and their own intra-model
references. Patching each symptom (Disposition A) leaves that root
cause intact and guarantees similar defects in the next ballot. Cutting
over to direct FHIR authoring fixes the class of defect, aligns with
common WG practice for US-realm IGs (US Core, mCODE, etc., which
author profiles directly against base resources), and gives reviewers
a single, curated artifact set to review. The cost is non-trivial —
the IG team must re-author every profile and lose the CIMPL
round-tripping — but the trade-off is favorable for a still-pre-STU
IG. **This is what the WG actually did**, per the resolution note.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** on the grounds that the cited defects
are limitations of the CIMPL logical-model tooling that the IG depends
on, and that the IG team has no in-scope mechanism to fix them within
the 2019SEP ballot cycle. Document the reporter's points in the IG's
known-issues page and ask implementers to refer to the underlying OBF
StructureDefinitions when in doubt about a profile's intent.

#### Justification

Defensible only as a stop-gap: it acknowledges that the broken
hyperlink, the incomplete index, and the generic descriptions are
genuine, but argues they cannot be fixed without abandoning the CIMPL
pipeline (which is itself a separate, larger decision). It avoids
churn in the 2019SEP ballot and keeps the OBF tooling investment
visible. It is weak on its merits because the reporter's complaints
are concrete and actionable independently of the tooling story (the
profiles index page is hand-curated, narrative can be added per profile
without regenerating from CIMPL, and a broken canonical reference is
always fixable), so "we cannot fix this" overstates the constraint.

---

### Recommendation

**Recommended disposition:** B

The reporter is not really asking for one missing link and one missing
index entry — they are asking *why the IG reads as a generic projection
of a logical model rather than as a curated FHIR IG*. Disposition A
treats the symptoms one by one, which works for this ballot but leaves
the WG patching the same class of defect again at the next reconciliation.
Disposition B addresses the root cause: by re-authoring directly against
base FHIR resources, the IG gains a single, hand-curated artifact set
where the index page, the profile narrative, and every cross-profile
reference are authored deliberately. That is also the path the WG
actually chose (the resolution explicitly says "for the next ballot we
will not be using these logical models and instead adhere more closely
to the base FHIR resources"), so this disposition aligns the public
record with the action taken. Disposition C should be rejected — the
defects are too concrete to label "Not Persuasive."
