## Summary

The reporter (Marti Velezis), commenting on the US Breast Cancer Radiology
Reporting (FHIR) IG, asserts that the IG's current use of the Device resource
is too sparse for proper identification of a medical (imaging) device. Today
the IG profile surfaces only `Device.id` and `Device.deviceName` (name/type).
The ticket asks the IG to additionally require/permit the FHIR Device elements
needed for unambiguous device identification per UDI conventions, namely:

- `Device.udiCarrier.deviceIdentifier` at cardinality `1..1` (the UDI-DI),
- a UDI-PI bundle, requiring at least one of `Device.manufactureDate`,
  `Device.expirationDate`, `Device.lotNumber`, `Device.serialNumber`, and
  `Device.distinctIdentifier`, and
- supplemental device descriptors `Device.deviceName`, `Device.modelNumber`,
  and `Device.version` when present.

The ticket carries the imported note "Comment 54 — imported by Jean Duteau",
indicating it originated as a spreadsheet ballot comment rather than as a
free-form Jira issue. It was resolved **Persuasive with Modification** on
2021-07-27 with a deferral rationale: implementer software backing the
Breast Imaging FHIR IG does not currently retain that level of device
detail, and the workgroup intends to align with the forthcoming
Unique Device Identification (UDI) Exchange FHIR IG once it is published.
The ticket is now flagged Block-Vote-5 for confirmation. No related Jira,
Zulip, or GitHub items were surfaced by cross-reference or search.

## Proposed Dispositions

The ticket has already been resolved **Persuasive with Modification** by
the workgroup; it now appears under Block-Vote-5 for confirmation. The
dispositions below frame that confirmation as a fresh choice for the
workgroup so the original decision can be reaffirmed, narrowed, or
overturned.

### Disposition A: Accept As Requested

#### Proposal

Update the Device profile in the US Breast Cancer Radiology Reporting
(FHIR) IG so that, when a Device instance is exchanged, it carries the
UDI-shaped element set described in the ticket:

- Slice/constrain `Device.udiCarrier` to `1..1` and require
  `Device.udiCarrier.deviceIdentifier` (`1..1`, must-support) — the
  UDI-DI.
- Mark the UDI-PI elements `Device.manufactureDate`,
  `Device.expirationDate`, `Device.lotNumber`, `Device.serialNumber`,
  and `Device.distinctIdentifier` as must-support, with an
  invariant requiring **at least one** of them to be populated when a
  UDI is asserted (capturing the "1..* of the following" language from
  the ticket).
- Mark the descriptive elements `Device.deviceName`,
  `Device.modelNumber`, and `Device.version` as must-support so that, if
  present in the source system, they are conveyed.
- Update the IG narrative for the Device profile to spell out the UDI
  expectation and to cite FDA UDI / GS1 conventions for UDI-DI vs.
  UDI-PI.

#### Justification

This is the literal request and is consistent with the broader FHIR
Devices direction: `Device.udiCarrier` plus the PI-style attributes are
the canonical home for device identification, and the must-support
mechanism (rather than raising minimum cardinalities) lets the IG signal
intent without forcing every receiving system to invent data it does not
hold. It also keeps the breast-imaging Device profile aligned with how
imaging modalities are typically tracked in clinical and regulatory
environments where UDI is already in scope.

---

### Disposition B: Alternative Approach (Persuasive with Modification — defer to UDI Exchange IG)

#### Proposal

Reaffirm the existing resolution. Specifically:

- Do **not** raise cardinalities or add must-support flags for the UDI-DI,
  UDI-PI, or descriptive Device elements in this ballot of the US Breast
  Cancer Radiology Reporting IG.
- Instead, add a non-normative note to the Device profile page stating
  that richer device identification (UDI-DI via
  `Device.udiCarrier.deviceIdentifier`, UDI-PI elements, model/version)
  is anticipated and **SHOULD** be populated when available in the
  source system, with a forward reference to the forthcoming UDI
  Exchange FHIR IG.
- Track follow-up work via a placeholder ticket so that, when the UDI
  Exchange IG publishes, this profile is updated to reference it
  (and tightens cardinality/must-support to align with that IG's
  conformance expectations).

#### Justification

The workgroup's recorded rationale — that current implementer systems
backing the Breast Imaging IG do not retain the requested fields, and
that a dedicated UDI Exchange IG is in flight — remains the strongest
trade-off argument. Promoting elements to must-support that no producing
system can populate creates conformance failures without improving
patient-safety outcomes; deferring keeps the IG implementable today
while preserving the path to richer device identification once the UDI
work lands. The "Persuasive with Modification" resolution captures this
exactly: agree with the intent, modify the timeline.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Not Persuasive** for this IG: the
breast-imaging Device profile would continue to carry only `Device.id`
and `Device.deviceName`, and the additional UDI-DI / UDI-PI / model /
version elements would not be added, even as must-support, with no
forward commitment.

#### Justification

A Decline could be defended on scope: this IG is a clinical reporting
IG, not a device registry, and detailed UDI exchange is arguably the
job of the UDI Exchange FHIR IG and base Device guidance rather than a
disease-specific imaging IG. If the workgroup judged that **any**
mention of UDI obligations here would invite implementer confusion
about which IG is authoritative for device identification, declining
keeps the boundary clean.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the recorded resolution and remains the right answer
for Block-Vote-5. The reporter's request is technically correct — those
are the elements the FHIR Device resource exposes for proper UDI-based
identification — but the workgroup has already validated that producing
systems for breast-imaging reports do not yet hold that data, and a
dedicated UDI Exchange FHIR IG is the natural home for the conformance
machinery. Disposition A would create must-support obligations ahead of
implementer readiness and risks duplicating (and later contradicting)
the UDI Exchange IG; Disposition C throws away an in-scope
patient-safety improvement that the workgroup already agreed is
desirable. Reaffirming "Persuasive with Modification" — accept the
intent, defer the normative changes to alignment with the UDI Exchange
IG, and add a forward-looking note to the Device profile narrative —
keeps the IG implementable today and on a credible path to richer
device identification tomorrow.
