## Summary

Ballot comment #52 against the US Breast Cancer Radiology Reporting (FHIR)
Implementation Guide reports broken navigation on the IG's "Where to
Focus" page. Specifically, the reporter found that the "logical models"
link resolves to an empty page, the "reference model" link returns a
404, and an additional link intended to reach the "Breast Radiology
Report section" should instead point to the IG's modeling-approach and
logical-model material. The ticket is a Technical Correction (link
hygiene only) — no normative content change is requested. The ticket has
already been triaged with a resolution of **Persuasive**: the
work-group's recorded resolution comment commits to repairing the links
and improving the link descriptions in the next ballot.

There are no cross-referenced Jira tickets, Zulip threads, GitHub
issues/PRs, or Confluence pages associated with FHIR-25076 in the
indexed corpus, so the ticket stands on its own.

## Proposed Dispositions

The ticket is a documentation/navigation fix against the BCRR IG. No
GitHub repository for that IG was surfaced through cross-references, so
the dispositions below are framed in terms of the published IG's
authoring conventions (Where to Focus page, logical-model pages,
modeling-approach page) without naming a specific repo path. If the IG
source repository is later identified, the Accept dispositions can be
re-anchored to the concrete file in that repo.

### Disposition A: Accept As Requested

#### Proposal

Repair the broken in-page links on the BCRR IG's "Where to Focus" page
so that:

1. The text currently linked as "logical models" points to the IG's
   actual logical-models index page (the page that lists the IG's
   logical model definitions) instead of the empty/placeholder page it
   currently targets.
2. The text currently linked as "reference model" points to the IG's
   reference-model page instead of returning a 404.
3. The link from "logical models" that currently lands on the
   "Breast Radiology Report section" be re-targeted to the IG's
   modeling-approach page and the corresponding logical-model page (per
   the reporter's clarification "should be changed to go to the modeling
   approach and logical [model]").

In addition to fixing the targets, update the visible link text /
surrounding sentence on the "Where to Focus" page so the link label
matches its destination (the resolution comment specifically calls out
"improve … description of the links"). Republish the page in the next
ballot of the IG, and add the corrected URLs to the IG's regression
checklist so the same links are verified on each subsequent build.

#### Justification

This matches both the reporter's request and the recorded resolution
("We agree, and will improve the links and description of the links in
the next ballot"), and the resolution is **Persuasive**, so the
work-group has already endorsed this direction. It is purely a
navigation / documentation fix — no normative artefact (resource,
profile, value set, extension) is touched, so there is no
backward-compatibility cost and no implementer impact beyond
documentation usability. Fixing the link text alongside the targets
addresses the underlying readability problem (the reporter's confusion
came in part from the link labels themselves being ambiguous), which is
a small additional change that prevents the same comment from
re-appearing in a later ballot.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substance of the request, but generalise the fix beyond the
specific links called out:

1. Audit *all* in-page links on the BCRR IG's narrative pages
   (introduction, Where to Focus, modeling approach, conformance) using
   the IG publisher's link-checker output and/or an external
   broken-link scanner against the published site.
2. Replace every broken or empty-target link surfaced by the audit, not
   just the three the reporter happened to land on.
3. As part of the same pass, normalise the link labels to follow a
   single convention (e.g., every reference to a logical model uses the
   logical-model name, not the umbrella phrase "logical models").
4. Add a CI step (or a checklist item in the IG release process) that
   runs the link-checker on each build and fails the build on broken
   internal links.

#### Justification

The reporter found three problems on one page; it is unlikely those are
the only broken links in the IG, since the underlying cause (whatever
generated the empty page and the 404) probably affected sibling pages
too. Doing a single sweep is only marginally more work than fixing the
three named links, and adding a build-time check prevents this exact
ballot comment from being re-filed against the next version. The
trade-off vs. Disposition A is scope: this disposition does more work
than the ticket strictly requires, and may surface additional problems
that need their own dispositions, which can slow the next ballot
slightly. It is the better engineering answer; Disposition A is the
better minimal-change answer.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as Not Persuasive / Withdraw on the grounds that the
referenced page ("Where to Focus") is no longer present, or that the
named link targets ("logical models", "reference model", "Breast
Radiology Report section") no longer exist in the current draft of the
IG, so there is nothing to repair.

#### Justification

This is only defensible if a current review of the BCRR IG draft
confirms the relevant page and links have already been removed or
restructured since 2019, in which case the ticket is moot. Given the
recorded resolution is **Persuasive** and explicitly commits to the
fix, declining now would contradict the work-group's prior position
without new evidence, so this disposition is included for completeness
only.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been triaged Persuasive and the recorded
resolution comment is essentially Disposition A verbatim ("we agree …
will improve the links and description of the links in the next
ballot"). The change has no normative impact, no implementer cost, and
no controversy — it is a maintenance fix that the work-group has
already endorsed. Disposition B is a strictly better engineering
outcome and the work-group should consider adopting it informally
during the same editorial pass, but it should not block closing this
ticket: the minimum action that satisfies the reporter and matches the
recorded resolution is Disposition A. Disposition C should only be
revisited if the page has since been removed from the IG entirely.
