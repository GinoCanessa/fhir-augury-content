## Summary

The ticket asks the mCODE IG to improve the quality of NCI Thesaurus
value-set presentations by (1) surfacing a definition for each concept
directly in the IG and (2) linking each code to its specific NCI
Thesaurus concept page rather than to the generic NCI Thesaurus browser.
The reporter notes this is consistent with how other terminologies such
as SNOMED CT are already rendered. The workgroup previously voted
"Persuasive with Modification" (2024-09-25) and then "Considered for
Future Use" (2024-10-24); both resolutions were subsequently reverted
and the ticket was re-opened post-STU4.0.0 publication, so it is now
awaiting fresh disposition. The stated technical blocker is that the IG
Publisher relies on an ontology-server-hosted terminology to render
concept definitions and deep links, and NCI Thesaurus is not currently
available from the ontology server tx.fhir.org (or any equivalent the
Publisher is wired to consume). No other Jira tickets cross-reference
this one; a single Zulip topic
(`#IG-creation > NCI Thesaurus -- lack of hyperlinks and expansion`)
was surfaced via the workgroup's previous comment and captures the
same operational constraint.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the mCODE IG so that every binding to an NCI-Thesaurus-based
ValueSet (primarily those used by the cancer staging profiles and the
oncology-specific body-site / morphology bindings) renders:

1. A per-concept definition, authored in the mCODE IG itself as part of
   each ValueSet / CodeSystem fragment (or as an `extension` on the
   binding's description in the StructureDefinition), sourced from the
   NCI Thesaurus release notes / the terminology owner.
2. A deep hyperlink to the specific NCI Thesaurus concept (e.g.,
   `https://ncit.nci.nih.gov/ncitbrowser/ConceptReport.jsp?dictionary=NCI_Thesaurus&code=<CODE>`)
   on every code display, in place of the generic NCI Thesaurus browser
   link.

Implement this by shipping a supplementary CodeSystem (or per-code
`definition` entries on the ValueSet expansions) in the mCODE IG source
so that the IG Publisher has local material to render, bypassing the
missing ontology-server support.

#### Justification

This is what the reporter asked for and what the 2024-09-25 vote
("Persuasive with Modification", 13-0-3) was already tracking. It
delivers concrete parity with SNOMED CT for mCODE implementers and
removes the most common complaint from downstream mCODE consumers —
that NCI-coded fields are opaque unless a reader manually copies the
code into the NCI browser. Adding concept definitions directly in
mCODE also protects the IG from future changes in tx.fhir.org's
coverage. The main cost is authoring effort for the IG editors and
the ongoing maintenance burden of keeping the mCODE-side copies in
sync with NCI releases.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the NCI Thesaurus bindings as-is but address the reporter's
underlying need through a two-part narrative-plus-linking solution
that does **not** require hand-authoring per-concept definitions in
the mCODE IG:

1. In the mCODE IG narrative for each profile that binds to an
   NCI-Thesaurus ValueSet, add an explicit "How to read NCI
   Thesaurus codes" paragraph that names the staging system, explains
   what the codes mean at a *system* level (e.g., "AJCC 8th edition
   prognostic stage group"), and links once, prominently, to the NCI
   Thesaurus browser's search landing page.
2. Ship a small CodeSystem supplement with per-code URLs (using
   `CodeSystem.concept.property` with a `source` property pointing at
   the concept's canonical NCI page), so the IG Publisher can render a
   deep link per code even without ontology-server support, while
   leaving concept definitions to the upstream source.

This is close to the reverted "Persuasive with Modification"
resolution but generalised across the IG.

#### Justification

The deep-link half of the ticket is genuinely cheap and high-value
once you accept that the Publisher can render `concept.property`
values. The definition half is the expensive half and is the one that
has blocked progress twice. By separating them, the workgroup can
ship the high-value part now, get out of the STU4 block-vote queue,
and come back to per-concept definitions when NCI Thesaurus is
available from a terminology server (at which point both halves
become automatic). This also avoids mCODE taking on an unbounded
maintenance obligation for NCI concept text.

---

### Disposition C: Decline

#### Proposal

Persuasive — Not Persuasive / Considered for Future Use with a clear
statement that mCODE will add NCI Thesaurus definitions and deep-links
automatically once NCI Thesaurus is supported by a terminology server
consumed by the IG Publisher (e.g., tx.fhir.org). No changes to the
mCODE IG at this time.

#### Justification

This is the substance of the 2024-10-24 vote (7-0-0) that was
subsequently reverted only so the ticket could be re-visited post-STU4
publication. The technical statement is accurate: the IG Publisher's
rendering of SNOMED CT definitions and deep links is not a piece of
content maintained by mCODE — it is an emergent property of having the
terminology on tx.fhir.org. Duplicating that content inside the mCODE
source would create a maintenance liability without changing the
underlying gap. Deferring is defensible as long as mCODE publishes an
explicit "known limitation" note in the IG so implementers understand
why NCI-coded values lack the same affordances as SNOMED-coded ones.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best starting point for the workgroup. Disposition
C captures the real technical constraint honestly but produces no
reader-facing improvement and has already been rejected once by the
re-open; leaving implementers without deep links is a tangible pain
point the ticket was filed to fix. Disposition A is the cleanest
outcome for readers but imposes a recurring definition-maintenance
burden on the mCODE editors that the group has been visibly reluctant
to take on (evidenced by the two reverted resolutions). Disposition B
threads the needle: it delivers per-concept deep links — the cheap,
purely mechanical half of the reporter's request — via an in-IG
CodeSystem supplement that the IG Publisher can already render, while
explicitly deferring concept-definition text until the upstream NCI
Thesaurus is served by an ontology server the Publisher consumes. The
workgroup should confirm with the IG Publisher maintainers that
`CodeSystem.concept.property` with a URL-typed property is an
acceptable deep-link mechanism, and then scope the authoring effort
to the staging-profile ValueSets that motivated the original ticket.
