## Summary

The ticket asks the mCODE IG editors to consider **anchoring (publishing) the
IG's value sets in THO (terminology.hl7.org)** rather than defining them inside
the mCODE IG itself, on the rationale that the value sets could be reused
beyond mCODE. The request was originally imported as "Comment 4" by reporter
Ron G. Parker and was originally dispositioned during STU4 ballot reconciliation
as *Considered for Future Use* (vote 5–0–4) with the explanation that mCODE
would stay consistent with other widely-used FHIR IGs (e.g., US Core) that keep
value sets anchored in the IG or base FHIR. After STU 4.0.0 publication the
ticket was re-opened by lynn_laakso for fresh consideration, which is what
brings it back to the work group now.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were returned
from the augury data sources for this ticket key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the mCODE-authored value sets out of the mCODE IG and publish them in
**terminology.hl7.org (THO)**. Concretely:

1. Identify the in-scope value sets — those whose composition is general
   oncology terminology that is plausibly reusable outside mCODE (e.g.,
   cancer disorder, primary cancer condition, secondary cancer condition,
   cancer body location, cancer staging, cancer-related procedure, cancer
   genomic biomarker, performance status, etc.). Exclude value sets that are
   purely structural to mCODE profiles or that are simply re-bindings of
   existing US Core / FHIR base value sets.
2. For each in-scope value set, transfer authoring to THO (HL7 Terminology):
   re-host the resource at a `http://terminology.hl7.org/ValueSet/...`
   canonical, retire the mCODE-hosted canonical, and update mCODE profiles
   so each binding now references the THO canonical.
3. Publish a backwards-compatibility note in the mCODE IG: the previous
   `http://hl7.org/fhir/us/mcode/ValueSet/...` canonicals continue to
   resolve (deprecated retirement), and implementers are directed to the new
   THO canonicals going forward.
4. File a companion ticket against the **Vocabulary** work group / THO
   submission queue to track ingestion, since THO content is curated by
   Vocabulary, not by CIC.

#### Justification

The ticket's stated rationale — that mCODE's oncology value sets could be
useful to other IGs (e.g., genomics IGs, cancer registry IGs, oncology quality
measures, mCODE-derived national IGs) — is consistent with the long-standing
HL7 direction that re-usable terminology artifacts should live in THO so they
are discoverable, version-managed, and citable independent of any single IG.
Publishing in THO also reduces the risk that downstream IGs each fork their
own near-identical copy. The "Highest" priority and the `stu4-block-vote-1`
label indicate the WG already considers this important enough to re-evaluate
post-STU4 publication.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of moving the value sets to THO wholesale, take a **two-tier hybrid
approach**:

1. **Keep value sets anchored in the mCODE IG** for normal authoring and
   publication, preserving the current canonical URLs and avoiding immediate
   churn for existing implementers.
2. **Mirror or re-publish a curated subset to THO** as derivative artifacts
   with their own THO canonical, marked as "Sourced from mCODE STU4". The
   mCODE IG remains the source of truth; THO copies are downstream
   publications intended for cross-IG reuse.
3. Add an editorial note to the mCODE IG and to each mirrored value set
   describing the relationship and the canonical that downstream IGs should
   prefer when reuse is the goal.
4. Defer a full migration to THO until the mCODE WG, the Vocabulary WG, and
   downstream IG authors have agreed on which value sets are genuinely
   reusable vs. mCODE-internal.

#### Justification

This addresses the underlying need (cross-IG reuse) without forcing an
immediate canonical change on every existing mCODE implementer, which is the
main concern raised in the original 2024-06-27 disposition ("stay consistent
with US Core, which anchors in the IG"). It also aligns with how some HL7
content has historically migrated to THO — incrementally, after the artifacts
have stabilized. The downside vs. Disposition A is duplication and the
possibility of drift between the mCODE-anchored copy and the THO copy, which
needs an explicit governance answer (who edits, who re-publishes, how versions
align).

---

### Disposition C: Decline

#### Proposal

Decline the request and **keep mCODE value sets anchored in the mCODE IG**,
on the grounds that:

- The current pattern is consistent with the most widely-used FHIR IGs
  (US Core, IPS, etc.), which is exactly the rationale the WG used in its
  original 2024-06-27 disposition (vote 5–0–4, *Considered for Future Use*).
- Migrating canonicals is a breaking change for STU4 implementers, and the
  ticket presents no concrete external IG that is currently blocked by the
  current anchoring.
- Cross-IG reuse is technically already possible today: any other IG can bind
  to the existing `http://hl7.org/fhir/us/mcode/ValueSet/...` canonicals.
- THO submission and curation introduces additional cross-WG dependency
  (Vocabulary) without a demonstrated near-term consumer.

If declined, close as **Considered for Future Use** (the same outcome as the
original 2024-06-27 vote) so the request can be re-opened once a concrete
downstream IG dependency is identified.

#### Justification

The original WG vote (5–0–4) already weighed this trade-off and chose
consistency with US Core. Re-opening post-publication is appropriate, but the
ticket itself contains no new evidence — no new requesting IG, no new use
case, no implementer pain — that would justify a STU-level breaking change
to canonicals. Declining (and re-classifying as "Considered for Future Use")
is defensible while leaving the door open.

---

### Recommendation

**Recommended disposition:** B

The original WG concern — consistency with US Core and avoiding churn for
existing STU4 implementers — is real and was the basis for the 5–0–4 vote
that accepted *Considered for Future Use* in 2024. At the same time, the
ticket has been deliberately re-opened post-STU4 publication, which signals
that the WG does not consider the question fully settled, and the reporter's
underlying point (mCODE oncology terminology is plausibly reusable outside
mCODE) is well-founded. A pure Disposition A migration breaks existing
canonicals for no immediate consumer; a pure Disposition C decline ignores
the reason the ticket was re-opened. The hybrid in Disposition B keeps
existing STU4 bindings stable, lets the WG curate which value sets are
genuinely cross-IG candidates, and creates a concrete deliverable
(THO-mirrored value sets) without committing to a full migration before the
Vocabulary WG and downstream IG authors have weighed in. The work group
should override this if there is a known external IG already waiting for
THO-anchored mCODE value sets — in that case Disposition A becomes the
better choice.
