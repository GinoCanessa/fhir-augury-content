## Summary

The reporter asks that mCODE value sets that draw codes from the NCI
Thesaurus carry the same display affordances that SNOMED-backed value
sets get in the published IG: a per-concept definition (sourced from
NCI Thesaurus) and a deep hyperlink to the specific concept (e.g.,
`https://ncit.nci.nih.gov/ncitbrowser/ConceptReport.jsp?code=Cxxxxx`)
rather than a link to the NCI browser homepage. The example cited is
the codex-radiation-therapy `radiotherapy-adverse-event-termVS` value
set, where today the IG publisher emits no expansion ("No Expansion
for this valueset (Unknown Code System)") and no per-concept
hyperlinks. NCI reviewers raised the gap during the mCODE ballot, so
the request is grounded in concrete reviewer feedback rather than
abstract concern. The ticket was previously resolved "Considered for
Future Use" (Hill/Leftwich, 7-0-0, 2024-10-24) on the rationale that
the IG Publisher derives expansions and per-concept hyperlinks from an
ontology server and NCI Thesaurus is not currently served by one in
the FHIR ecosystem; that resolution was later reverted and the ticket
was re-opened post STU 4.0.0 publication so the workgroup can revisit
it. There are no cross-referenced Jira tickets; the only cross-ref is
to a Zulip thread in `#IG creation` titled "NCI Thesaurus -- lack of
hyperlinks and expansion" which captures the technical discussion
about why this is hard and what would unblock it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the request as in-scope for mCODE and add per-concept NCI
Thesaurus definitions and concept-specific hyperlinks (form:
`https://ncit.nci.nih.gov/ncitbrowser/ConceptReport.jsp?code=<NCIT
code>`) to every mCODE value set whose composition draws from the
NCI Thesaurus code system (`http://ncithesaurus-stage.nci.nih.gov`),
and emit a usable expansion. Concretely, this means standing up — or
arranging for a partner to stand up — a public Ontoserver instance
that hosts NCI Thesaurus (built with the
[aehrc/fhir-owl](https://github.com/aehrc/fhir-owl) OWL → CodeSystem
converter as Michael Lawley described), registering that server with
the FHIR Tx ecosystem so the HL7 IG Publisher will route NCIT lookups
to it, and then republishing mCODE so the value set pages render
"logical-definition-cld" hyperlinks and an "expansion" section the
same way they do today for SNOMED-backed value sets. No mCODE source
artifacts (profiles, value sets) themselves need to change for this
to work — the gap is purely in the terminology supply chain that
feeds the IG Publisher.

#### Justification

This is what the ticket actually asks for and what the NCI ballot
reviewers want. The Zulip discussion confirms there is no
IG-Publisher code change needed: the existing pattern that gives
SNOMED CT its hyperlinks and expansions is the correct pattern, and
applying it to NCI Thesaurus is just a matter of registering an
ontology server that holds the code system. Doing so improves data
quality and reviewability for every IG that binds to NCIT, not just
mCODE, and it removes the recurring "Unknown Code System" warning on
published value set pages — a consistency-with-existing-patterns
argument that is hard to refuse on its merits.

---

### Disposition B: Alternative Approach

#### Proposal

Acknowledge the request but reframe it as a terminology-services
issue rather than an mCODE specification issue. Mark the ticket
"Persuasive with Modification": (a) leave mCODE's NCIT bindings
unchanged, (b) add a normative-flavored note in the mCODE Terminology
section explaining that NCI Thesaurus expansions and per-concept
hyperlinks depend on the value set being resolved against an NCIT-
hosting terminology server (citing Ontoserver + the OWL → CodeSystem
converter as one available path), (c) provide an explicit
implementer-guidance paragraph that lists the canonical NCI Thesaurus
concept-report URL pattern
(`https://ncit.nci.nih.gov/ncitbrowser/ConceptReport.jsp?code=<code>`)
so downstream consumers can resolve concepts manually, and (d) file a
companion issue against tx.fhir.org / HL7 Terminology Authority asking
that NCI Thesaurus be hosted on a registered public ontology server
that the IG Publisher will use. Coordinate that companion issue with
Vocabulary WG.

#### Justification

The actual fix is upstream of mCODE — neither the mCODE editors nor
the IG Publisher author can produce the desired output without an
NCIT-hosting registered terminology server. Closing the mCODE-side
ticket with explicit guidance and pointers (rather than holding it
open indefinitely on infrastructure that mCODE doesn't own)
acknowledges the reviewer feedback, gives implementers a concrete
manual workaround, and pushes the real work to the group with the
authority and resources to host terminology. It also avoids putting
the Clinical Interoperability Council on the hook for a
terminology-hosting commitment they cannot directly fulfill.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" with the rationale "Out of scope for
mCODE; no action required in this specification." The substance of
the resolution is: per-concept NCI Thesaurus hyperlinks and value-set
expansions are produced by the HL7 IG Publisher when it can resolve
the code system against a registered terminology server, and NCI
Thesaurus is not currently served that way; the mCODE source
artifacts already declare the correct code system URI and the correct
NCIT concept codes, so there is nothing to change in mCODE itself.
Direct the reporter to the IG Publisher / Tx ecosystem governance.

#### Justification

This is essentially the prior resolution ("Considered for Future Use"
/ "Once the NCI Thesaurus concepts are hosted in an ontology server
… we will be happy to add them") that the workgroup voted 7-0-0 in
October 2024, and Grahame Grieve's Zulip messages corroborate the
technical reasoning. The risk of this disposition is purely
optical: it leaves the reviewer-visible "Unknown Code System" /
no-hyperlink behavior in place on mCODE value set pages even though
mCODE's own content is correct, which is what got the resolution
reverted in the first place. If the workgroup is unwilling to commit
to chasing the upstream fix, this disposition is honest about what
mCODE can and cannot do.

---

### Recommendation

**Recommended disposition:** B

The Zulip thread leaves no doubt that this is a terminology-supply
problem rather than a mCODE editorial problem — the codes and the
code-system URI in mCODE's value sets are already correct, and the
IG Publisher will do the right thing the moment NCI Thesaurus is
served by a registered ontology server. Disposition C is technically
defensible but is exactly the resolution that Lynn Laakso reverted on
2025-02-17, so re-stating it without movement is unlikely to satisfy
the workgroup or the NCI reviewers who triggered the ticket.
Disposition A would solve the visible problem but commits the
Clinical Interoperability Council to terminology-hosting and
ecosystem-registration work that lives outside the mCODE charter and
that no one in the Zulip thread agreed to own. Disposition B threads
the needle: it closes the mCODE-side issue with a Persuasive-with-
Modification result (clarifying the NCIT concept-link URL pattern in
mCODE's terminology guidance and explaining the dependency on a
hosted ontology server), gives implementers something concrete to
use today, and explicitly hands the structural fix to Vocabulary WG /
the Tx ecosystem via a companion ticket — which is where Grahame and
Michael Lawley both said the work belongs. The workgroup should
expect pushback on whether mCODE can really walk away from the
ballot reviewer experience, but the appropriate counter is the
companion ticket, not an open-ended commitment in mCODE itself.
