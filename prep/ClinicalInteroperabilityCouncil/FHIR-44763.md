## Summary

The submitter argues that the mCODE `GleasonGradeGroup` profile is misclassified
as derivative of `CancerStage`. Per the AJCC 8th Edition staging rules, Gleason
Grade Group is a *non-anatomical* prognostic category that — alongside PSA
categorical value, TNM components, and other site-specific markers — feeds the
overall Prostate Cancer Stage Group, but is not itself a stage group. The
submitter proposes introducing a new `StageCategory` parent profile to capture
non-TNM staging *categories* (e.g., G and PSA for prostate; ER, PR, HER2,
OncotypeDx for breast; HPV for oropharynx). `StageCategory` profiles would
reference the underlying lab/genomic Observation profiles where appropriate,
since onco-specific EMRs typically capture the categorical value even when the
raw lab value is not available. The ticket is currently `Triaged` against the
mCODE specification with the `preSTU4-block-vote` label, having been re-opened
post-STU4.0.0 publication after an earlier disposition (Considered for Future
Use, 2024-03-28) was reverted to allow re-evaluation in the STU4 reconciliation
process.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the mCODE IG, introduce a new abstract parent profile **`StageCategory`**
on `Observation` that represents non-anatomical / non-TNM cancer staging
categories. Re-base the existing **`GleasonGradeGroup`** profile (and any
analogous categorical staging profiles such as a future `PSACategory`,
`ERStatus`, `PRStatus`, `HER2Status`, `OncotypeDxScore`, `HPVStatus`) so they
derive from `StageCategory` rather than from `CancerStage` / the
TNM-style stage profile.

Concrete shape of `StageCategory`:

- `Observation.code` bound to a value set of categorical staging factors (G,
  PSA category, ER, PR, HER2, OncotypeDx, HPV, …).
- `Observation.value[x]` typed `CodeableConcept`, bound per derived profile to
  the appropriate categorical value set.
- `Observation.derivedFrom` (0..*) referencing the underlying lab / genomic
  Observation profile when present (e.g., a quantitative PSA lab Observation,
  or a genomic biomarker Observation) — but not required, since onco-specific
  EMRs may carry only the category.
- Update `CancerStageGroup` (and any per-cancer stage group profiles such as
  Prostate / Breast / Oropharynx) so their `hasMember` slice for non-TNM
  categories targets `StageCategory` rather than `CancerStage`.
- Update the IG narrative pages on cancer staging to describe the
  TNMCategory + StageCategory + StageGroup model and the AJCC alignment.

Mark this as a STU4 reconciliation change; flag the impact on existing
`GleasonGradeGroup` instances (the resource type is unchanged, but the
declared profile chain shifts).

#### Justification

The submitter's reading of AJCC 8th Edition is correct: Gleason Grade Group is
a *prognostic / non-anatomical category* that contributes to a Prostate Stage
Group rather than being a Stage Group itself. Having `GleasonGradeGroup`
derive from `CancerStage` overloads the term "stage" and conflates two
concepts that AJCC explicitly separates. The proposed `StageCategory` parent
generalizes cleanly to other site-specific non-TNM categories (PSA, ER, PR,
HER2, OncotypeDx, HPV), which mCODE will need anyway as it broadens beyond
prostate. The new profile also gives implementers a predictable place to hang
`derivedFrom` linkage to the underlying lab/genomic Observations without
forcing those linkages where the EMR only stores the categorical result —
matching the data-availability pattern the submitter describes.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `GleasonGradeGroup` as a stand-alone Observation profile but **do not**
create a new `StageCategory` parent profile. Instead:

1. Re-parent `GleasonGradeGroup` directly on a base mCODE Observation profile
   (e.g., the same parent used by other categorical onco-Observations such as
   tumor marker / biomarker profiles) and remove any inheritance from
   `CancerStage`.
2. Add a clarifying narrative section in the mCODE staging guidance that
   distinguishes anatomical stage (TNM) from non-anatomical staging factors
   and lists Gleason Grade Group, PSA category, ER/PR/HER2, OncotypeDx, HPV
   as examples.
3. Use `Observation.category` (with a mCODE-defined code such as
   `staging-category` / `non-anatomic-staging-factor`) plus an extension on
   `CancerStageGroup.hasMember` to mark which member Observations are
   non-anatomical staging factors, rather than introducing a new profile
   hierarchy.

#### Justification

This addresses the underlying conceptual error (Gleason Grade Group is not a
stage group) without adding a new profile layer that the IG would need to
maintain. Profile proliferation has a real cost: every new abstract parent
adds documentation, validation, FSH, and IG-build surface area, and forces
every downstream re-publish of mCODE to re-validate inheritance chains. Using
`Observation.category` plus a categorization extension is the same mechanism
mCODE already uses elsewhere (cancer-related categories, tumor markers) and
keeps the change small enough to be safely landed in STU4 reconciliation
without inviting scope creep into "let's also add ER, PR, HER2, OncotypeDx,
HPV" — each of which has its own value-set and binding decisions that have
not been worked through here.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped. Rationale category: **insufficiently
scoped change with cross-cutting impact, and underlying need partially
addressed by existing mechanisms.** Close the ticket with the suggestion that
the submitter open a more narrowly-scoped follow-up specifically targeting the
`GleasonGradeGroup` profile parentage, and a separate enhancement request
covering the broader `StageCategory` family once concrete value sets and use
cases for PSA category, ER, PR, HER2, OncotypeDx, and HPV have been gathered.

#### Justification

The current ticket bundles two distinct asks: (1) fix the parentage of one
profile (`GleasonGradeGroup`), and (2) introduce a new abstract profile family
(`StageCategory`) that mCODE has not yet scoped value sets, bindings, or
member profiles for. The first is a small reconciliation-style fix; the
second is a substantive structural change that touches multiple cancer types
and would benefit from a dedicated design discussion with the mCODE
modeling subgroup, ideally with concrete profiles drafted for the cited
examples (PSA category, ER, PR, HER2, OncotypeDx, HPV). The earlier
"Considered for Future Use" disposition (Tcheng/Bray 6-0-0, 2024-03-28)
captured this concern well; there is no new community feedback in the ticket
or in cross-referenced threads (none were found) that resolves the open
design questions.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the submitter's correct conceptual point — Gleason
Grade Group is not a stage group — while keeping the change scope appropriate
for STU4 reconciliation. The ticket comes from an external reviewer flagging a
modeling inconsistency with AJCC; declining outright (C) leaves the
inconsistency in the published IG and will likely re-surface in future ballot
cycles. Accepting as requested (A) is conceptually attractive but ships a new
abstract profile family without the value-set work for ER/PR/HER2/OncotypeDx/
HPV/PSA-category, which risks landing a `StageCategory` parent that is
under-specified and rarely used. Re-parenting `GleasonGradeGroup` off
`CancerStage` and clarifying the staging narrative addresses the immediate
defect, leaves a clean path to introduce `StageCategory` later when the
broader family of non-anatomical staging factors is genuinely ready, and
honors the original 2024-03-28 working group concern about scope creep. If
the WG would prefer to do the bigger restructuring now, A is a defensible
upgrade path; the recommendation should be revisited with the mCODE modeling
subgroup before STU4 reconciliation closes.
