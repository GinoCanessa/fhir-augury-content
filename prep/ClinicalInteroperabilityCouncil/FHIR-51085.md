## Summary

The reporter argues that the HRex definition of Must-Support for Data Sources is overly absolute. As written, it requires a Data Source to be "capable of populating" any MS element when sharing a profile, and explicitly declares any system "incapable of ever sharing the element" to be non-conformant. The reporter contends this fails to recognize legitimate cases where (a) the element is genuinely not relevant to a particular kind of system (e.g., a form-author whose forms never need `Questionnaire.item.extension:itemHidden`), (b) optional elements such as `Patient.birthSex`, `Address.period`, or `PractitionerRole.specialty` are not realistically expected to be implemented by every conformant system, or (c) regulatory or privacy rules permanently prevent population. The request is to either soften the HRex Must-Support language to permit "never present" for some elements/systems, or to restate the per-element expectations using FHIR Obligations so different actors can have different conformance bars on a per-element basis. No directly-linked Jira tickets surfaced via cross-reference, but the originating Zulip thread (referenced below) makes clear that Lloyd McKenzie — author of much of the HRex MS language — agrees that expectations vary across HRex profiles and across elements within a profile, and proposes Obligations as the principled long-term fix.

## Details

**Description:**

The definition of Must-Support for an implementation acting as a data source is as follows:

> Data Sources **SHALL** be capable of populating the data element when sharing resources compliant with the profile. I.e. the system must be able to demonstrate the population and sharing of the element, but it is acceptable to omit the element if the system doesn't have values in a particular instance. A system that is incapable of ever sharing the element would be non-conformant.

This does not correctly enable the nuance that should be available in implementations choosing how to support different elements. For example, in DTR (because DTR references the HRex definition of Must-Support), form authors may have no need for some of the elements due to their business rules, and there is no reason to place a conformance expectation on supporting those elements regardless. Lloyd had these examples from HRex:

> Looking at the profiles in HRex, the expectations actually vary. For something like Consent in $member-match, or Data Request Task the expectation is absolutely "if you don't have this capability in your software, you must write code to add it". On the other hand, for something like Patient in $member-match or PractitionerRole, it's not clear the same expectation holds. E.g. If you don't support birthSex, address period, or PractitionerRole.specialty, it's probably not the intent for implementers to write code to add support for that.

In addition, some elements may never be available due to regulatory or privacy concerns applying to the data source.

Either the language should be adjusted to be less restrictive and allow for the possibility that elements will never be available in a proper and complete implementation, or these expectations should be expressed using obligations (with which I am unfamiliar but are apparently more work to implement).

## Related Zulip Discussions

### [Da Vinci HRex > Must-support definition](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20HRex/topic/Must-support%20definition)

This 10-message thread (2025-06-11 to 2025-06-12) is the origin of the ticket. November Valentine opened by quoting the HRex MS definition for Data Sources and arguing it conflicts with the parallel Data Consumer rule, which already accepts that elements may be absent for "privacy or other business reasons"; they argue a system whose business will never produce a particular MS element should still be considered compliant. Lloyd McKenzie initially defended the strict reading — implementers don't get to opt out of MS by declaring an element "not relevant" — but conceded that adding language for regulatory prohibition is reasonable, and that whether the strict default is appropriate "across all Da Vinci guides is certainly something we could review." Valentine pressed with the DTR `Questionnaire.item.extension:itemHidden` example: a form author whose forms never need hiding shouldn't be forced to implement population code; Lloyd agreed the "must be able to populate" rule should not apply to form authors in DTR, while clients still need to be able to consume `itemHidden` from any partner. Lloyd then proposed Obligations as the principled fix but flagged that it is "not a small undertaking." After Valentine pushed for, at minimum, an interim language edit, Lloyd reviewed HRex's own profiles and conceded the expectations really do vary — Consent in `$member-match` and Data Request Task warrant the strict bar, while Patient `$member-match` and PractitionerRole (citing `birthSex`, `Address.period`, `PractitionerRole.specialty`) do not — and asked Valentine to file the ticket, which became FHIR-51085. Consensus: the current single MS definition is too coarse; moving to Obligations is the right destination, with softened language as a stop-gap.

Participants: November Valentine, Lloyd McKenzie.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the HRex "Must Support" conformance section so that the Data Source expectation explicitly tolerates elements that a system will never populate when (a) the system's business scope does not produce values for that element or (b) regulatory or privacy rules prevent population. Concretely, replace the current absolute clause —

> A system that is incapable of ever sharing the element would be non-conformant.

— with language such as:

> A Data Source MAY be unable to ever populate a Must-Support element when the element is outside the scope of the system's business function, or when regulatory or privacy rules prohibit its disclosure. Such a system remains conformant provided it documents the affected elements (e.g., in its CapabilityStatement) and the rationale, and continues to satisfy the consumer-side expectation of treating absent elements as "not present in the source."

Pair this with a normative note that the bar is still strict for elements the IG specifically calls out as required-to-populate (e.g., elements driving `$member-match` matching or Data Request Task workflow).

#### Justification

This is the minimum-risk change that resolves the conflict the reporter identifies between the Data Source and Data Consumer rules in HRex (the consumer rule already presupposes "privacy or other business reasons" can suppress an element permanently, but the source rule denies the same possibility). It preserves the Must-Support mechanism, preserves backward compatibility for existing HRex-conformant systems, and aligns with Lloyd McKenzie's explicit Zulip concession that the strict bar should not apply to elements like `Patient.birthSex`, `Address.period`, or `PractitionerRole.specialty`. It is a documentation/conformance-language change only, with no profile/StructureDefinition impact, so it can be delivered in the next HRex publication cycle without an Obligations-level effort.

---

### Disposition B: Alternative Approach

#### Proposal

Migrate HRex's per-element conformance expectations from a single textual MS definition to FHIR R5 **Obligations** (`http://hl7.org/fhir/tools/StructureDefinition/obligation`). For each profile in HRex, classify each MS element against the actor(s) that must populate, must consume, or are merely permitted to populate it (e.g., `SHALL:populate` for Data Source on `$member-match` matching elements; `SHOULD:populate` or no obligation for `PractitionerRole.specialty`; `SHALL:able-to-process` for Data Consumer on `Questionnaire.item.extension:itemHidden`). In the interim, ship the textual softening from Disposition A in the same release so implementers are not stranded while the Obligations migration is staged. Document the actor inventory (Data Source, Data Consumer, Form Author, Form Filler, etc.) in a single HRex page that other Da Vinci IGs (DTR, PDex, CRD) can reuse.

#### Justification

This is the principled long-term answer that Lloyd identified in the Zulip thread. It directly addresses the reporter's underlying concern — that one MS definition cannot fairly describe the expectations across all profiles and all elements — by giving each element its own machine-readable expectation per actor. It also benefits every downstream Da Vinci IG that currently inherits HRex's MS definition by reference (notably DTR, where the form-author vs form-filler distinction is exactly the case the reporter raises). The trade-off vs Disposition A is significant scope: Lloyd flagged Obligations as "not a small undertaking," it requires R5-tooling-aware publication infrastructure, and every HRex profile needs review. That cost is why Disposition A is offered as a parallel near-term measure rather than as a competitor.

---

### Disposition C: Decline

#### Proposal

Decline the change. Retain the current HRex Must-Support definition unchanged on the basis that (a) the existing definition is consistent with the long-standing Da Vinci interpretation that MS implies a baseline implementation obligation, (b) the Data Consumer "missing element" allowance is intended for instance-level absence and not as an open door to permanent non-population, and (c) implementers who cannot populate an element have the option to use a profile that does not require it.

#### Justification

Defensible only if the workgroup believes the current language is being correctly applied and the reporter's concerns are isolated to misunderstanding rather than a real interoperability problem. It avoids any churn in HRex and any propagation of changes into downstream IGs. However, this position is hard to sustain in the face of the Zulip discussion, where the original author of the language conceded that the strict bar genuinely does not fit several elements in HRex's own profiles and asked the reporter to file this ticket. It also does nothing to resolve the regulatory/privacy edge case that even the existing Data Consumer rule already acknowledges.

---

### Recommendation

**Recommended disposition:** A, with a commitment to follow on with B.

The Zulip thread already establishes that the HRex authors agree the current MS definition is too coarse — Lloyd McKenzie explicitly walks through HRex's own profiles and identifies elements (`Patient.birthSex`, `Address.period`, `PractitionerRole.specialty`) for which the strict "must be able to populate" bar is not the actual intent, and he asked the reporter to file this ticket precisely so HRex could "do something here." Given that, declining (C) would contradict the workgroup's own stated reading and is not recommended. Pure-Obligations migration (B) is the right destination but is, in Lloyd's own words, "not a small undertaking" and would delay a fix the community can use immediately. Accepting (A) — softening the Data Source clause to explicitly tolerate "never populated" for out-of-scope or regulatorily-restricted elements while keeping the strict bar where the IG has called specific elements out — is a low-risk, immediately publishable change that resolves the reporter's compliance gap and sets the stage for the full Obligations rework in a subsequent cycle. The workgroup should explicitly schedule (B) as a follow-on rather than letting (A) become the permanent answer.
