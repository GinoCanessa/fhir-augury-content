## Summary

The submitter (NCI reviewer Robinette Renner) reports that mCODE value sets
that draw from the NCI Thesaurus lack two pieces of information that
implementers and reviewers rely on when reading the published IG: (a) a
human-readable definition for each enumerated NCI Thesaurus concept, and
(b) a hyperlink that resolves to the specific concept page in the NCI
Thesaurus browser (e.g., `https://ncit.nci.nih.gov/ncitbrowser/ConceptReport.jsp?code=Cxxxxxx`)
rather than just the generic browser landing page. The reporter notes
that this is the treatment SNOMED CT enjoys today and asks for parity for
NCI Thesaurus, since the data is readily available from the source.

Subsequent triage and the cross-referenced Zulip thread on `IG creation >
NCI Thesaurus -- lack of hyperlinks and expansion` clarified that this is
not a content-authoring problem in mCODE: the IG Publisher derives those
definitions, hyperlinks, and value-set expansions by calling out to a
terminology / ontology server (tx.fhir.org), and the NCI Thesaurus is not
currently hosted on any registered ontology server in the FHIR ecosystem.
Grahame Grieve confirmed that the NCI Thesaurus *can* be hosted on
Ontoserver via the AEHRC `fhir-owl` OWL→CodeSystem converter
(https://github.com/aehrc/fhir-owl), so the practical fix is to get
NCI Thesaurus stood up on a public, registered ontology server rather
than to change mCODE or the Publisher.

The ticket has had two prior dispositions reverted — "Persuasive with
Modification" (2024-09-25) and then "Considered for Future Use"
(2024-10-24) — and was re-opened post STU4.0.0 publication on
2025-02-17 by Lynn Laakso. It is now back in `Triaged` status awaiting
a fresh disposition for the next mCODE block vote (`stu4-block-vote-3`).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the request as a content gap that mCODE itself should close: in
the next mCODE publication, augment every ValueSet that binds to NCI
Thesaurus concepts (notably the performance status value sets and any
NCIt-bound value sets reused from `codex-radiation-therapy`) with
explicit per-code narrative blocks that (a) restate the NCI Thesaurus
preferred term and definition for each code and (b) provide a direct
hyperlink of the form `https://ncit.nci.nih.gov/ncitbrowser/ConceptReport.jsp?code={NCIt code}`
for each enumerated concept. This would be done by adding
`ValueSet.compose.include.concept[].display`, `…designation`, and
extension-driven narrative, plus authored ValueSet-level narrative
text (or a Jinja/markdown include during IG build) so the published
HTML matches the visual treatment SNOMED CT enjoys today.

#### Justification

This is the literal ask in the ticket and matches the previously-voted
"Persuasive with Modification" plan (Dave Hill / James McClay, 13-0-3)
to re-organize the performance status profiles and add more detailed
narrative on what each value means. It does not depend on
infrastructure outside mCODE's control (no ontology-server hosting,
no Publisher change), so it can ship in the next mCODE release without
external blockers, and it directly addresses the NCI reviewers'
ballot feedback. The downside — that the definitions are duplicated
in mCODE rather than served live from a terminology server — is
acceptable because NCI Thesaurus concept definitions are stable and
publicly cited by the canonical browser URL.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve the ticket as **Persuasive — Forwarded to Vocabulary / FHIR-I /
Ontoserver community**, with a concrete two-track plan rather than a
content change in mCODE itself: (1) coordinate with AEHRC / CSIRO and
NCI to host the NCI Thesaurus as a `CodeSystem` on a public ontology
server (using `fhir-owl`, https://github.com/aehrc/fhir-owl) and
register it with `tx.fhir.org` so the IG Publisher resolves it the
same way it resolves SNOMED CT; and (2) once the NCIt code system is
reachable from `tx.fhir.org`, simply re-publish mCODE — no value-set
edits required. The mCODE ticket would be closed as resolved by that
infrastructure work, with a tracking note (and ideally a companion
ticket on the publishing pipeline) so the dependency is visible.

#### Justification

This is the path Grahame Grieve and Michael Lawley converged on in
the cross-referenced Zulip thread, and it is strictly better in the
long run: every IG that binds to NCI Thesaurus (codex-radiation-therapy,
genomics work, future oncology IGs) gets correct expansions, hyperlinks,
and definitions automatically, with no per-IG narrative duplication
and no risk of the embedded definitions drifting from the authoritative
NCI source. It also avoids the maintenance burden of hand-curating
NCIt definitions inside mCODE, which would otherwise need to be
re-checked each NCIt release. The trade-off is timing: it depends on
an external party (an Ontoserver host willing to publish NCI Thesaurus
and have it registered), and mCODE cannot guarantee a date.

---

### Disposition C: Decline

#### Proposal

Disposition: **Not Persuasive — Out of Scope for mCODE**. The
underlying limitation (no per-code hyperlinks or expansions for code
systems not served by a registered ontology server) is a property of
the FHIR IG Publisher's terminology-server pipeline, not of the mCODE
specification. The NCI Thesaurus is not currently hosted on a
registered ontology server in the FHIR ecosystem; once it is, the
desired behavior will appear automatically. Close the ticket with
that explanation and a pointer to the IG-creation Zulip thread, and
leave the wider hosting/registration work to the FHIR-I / vocabulary
infrastructure track.

#### Justification

This is essentially the previously-voted "Considered for Future Use"
position (Dave Hill / Russ Leftwich, 7-0-0). It is defensible because
mCODE has correctly bound to a published external code system, and
the missing presentation is genuinely a tooling concern. However,
since the ticket has now been *re-opened* post STU4.0.0 specifically
because that disposition was felt to be inadequate, this option is
included for completeness rather than as the expected outcome — the
reporter's complaint (NCI reviewers seeing a noticeably worse
experience than SNOMED CT in the published IG) is real and recurring.

---

### Recommendation

**Recommended disposition:** A (with a B-shaped follow-up).

The ticket has already been declined-in-spirit twice (Considered for
Future Use, then reverted; Persuasive with Modification, then
reverted) and is back open under a block-vote label
(`stu4-block-vote-3`) precisely because option C is no longer flying
with NCI reviewers and the work group. Disposition B is the
architecturally correct long-term answer, but mCODE cannot ship it on
its own schedule — it depends on an external Ontoserver host
publishing NCI Thesaurus and on `tx.fhir.org` registering it. Choosing
B alone risks a third revert when the next ballot cycle arrives and
the infrastructure has not landed. Disposition A is fully under the
work group's control, matches the original "persuasive with
modification" intent, and gives the reporter and NCI reviewers a
concrete improvement in the next mCODE publication: per-code NCIt
definitions and direct `ConceptReport.jsp?code=…` hyperlinks rendered
in the value-set narrative for the affected (notably performance
status) value sets. The work group should adopt A now and open or
attach a companion tracking item for B so the duplicated narrative
can be retired once NCI Thesaurus is reachable through `tx.fhir.org`.
