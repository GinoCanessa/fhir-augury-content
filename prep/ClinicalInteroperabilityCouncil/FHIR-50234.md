## Summary

The reporter requests that the Da Vinci HRex Organization profile be aligned
with the CARIN Blue Button (C4BB) Organization profile by adding the EDI
"payerid" identifier as one of the recognized payer identifiers. The motivation
is that a unique payer identifier will likely be important for the payer-to-payer
registration process, although the reporter notes uncertainty over whether the
NAIC identifier or the EDI payer id should be the preferred identifier. There
are no cross-referenced Jira tickets; the only cross-reference surfaced is a
mention of the FHIR core `Organization` resource. The ticket has already been
resolved as "Not Persuasive" by the work group, with the rationale that the EDI
payerid is not yet clearly defined as distinct from NAIC or TIN identifiers
(both of which are already permitted in HIPAA-covered claims) and that endpoint
discovery, rather than a new identifier type slug, is the recommended mechanism
for a payer to advertise which identifier system it wishes to be known by.

## Details

**Description:**

It would be beneficial to align the specified payer identifiers with the CARIN
IG. In particular, could we add the (EDI) payerid identifier as seen in the
[CARIN Organization profile](https://build.fhir.org/ig/HL7/carin-bb/StructureDefinition-C4BB-Organization.html)?

We are thinking having a unique payer identifier will be important in the
payer-to-payer registration process, but it's unclear whether NAIC or the EDI
payer id will be the preferred identifier.

**Resolution Description (Not Persuasive):**

The Blue Button code will need to move from their IG into a more official
space soon. At present, it's not clearly defined as being distinct from NAIC
or TIN identifiers — which are also allowed to be used in HIPAA-covered claims.
If we were going to have a new identifier type, it would need to (a) be clearly
understood by all parties, and (b) be distinct from the kinds of identifiers
that already exist.

Given that we're recommending the use of the endpoint discovery mechanism to
find out what identifier a payer wants to be known by, it's not clear that
having a new identifier type will add additional value. Every identifier on an
Organization that represents a payer is, by definition, a "payer identifier".
The purpose of this type seems to be to encompass all identifiers that aren't
NAIC or TIN identifiers, and it's easy enough to do that now by saying "is the
system something that isn't one of those other two?".

If there's a convincing use-case with a clear definition distinct from other
identifiers that comes forward in the future, we could consider that, but we
don't see that now.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the HRex `Organization` profile to add a sliced `identifier` entry that
mirrors the CARIN C4BB Organization profile's payer identifier slice. Concretely:

- Add an `identifier` slice named `payerid` to the HRex Organization profile.
- Bind `identifier.type` to a code identifying "EDI Payer Identifier" (reusing
  the C4BB-defined coding so the slice is wire-compatible with C4BB
  Organization instances).
- Set `identifier.system` to the canonical URI used by C4BB for the EDI payerid
  registry (currently namespaced under the CARIN IG; coordinate with CARIN to
  promote that system URI into a more durable namespace before HRex normative
  publication).
- Leave the existing NAIC and TIN identifier slices unchanged so existing
  implementations continue to validate.
- Add narrative to the HRex Organization profile page explaining when each
  identifier slice (NAIC, TIN, EDI payerid) is expected and how it is consumed
  in payer-to-payer registration.

#### Justification

Aligning HRex's Organization profile with C4BB's makes Organization instances
exchanged across Da Vinci and CARIN use cases interoperable without
profile-specific re-mapping, which directly supports the reporter's
payer-to-payer registration use case. Reusing C4BB's identifier slice
definition (rather than inventing a new one) avoids divergent slugs for the
same concept across the HL7 payer ecosystem.

### Disposition B: Alternative Approach

#### Proposal

Instead of pinning a single payer identifier slice into the HRex Organization
profile, document the **endpoint discovery** pattern as the normative way for a
payer to advertise which identifier(s) it wishes to be known by, and relax the
HRex Organization identifier slicing to be **open** (allowing NAIC, TIN, EDI
payerid, or any future payer identifier system without a profile change). In
narrative:

- Keep the existing NAIC / TIN slices as informative examples.
- Add a "Payer Identifier" guidance section to the HRex Organization page
  describing how a consumer should use the discovered Endpoint plus
  `Organization.identifier[*].system` to resolve which identifier the payer
  considers authoritative.
- Cross-reference the C4BB Organization profile as a compatible specialization
  for plans that have already adopted EDI payerid.

#### Justification

This honors the work group's resolution rationale: every identifier on a payer
Organization is by definition a payer identifier, and what implementers really
need is a way to know *which* identifier system the receiver expects, not a
new well-known identifier-type code. Open slicing lets HRex absorb whatever
identifier system the broader industry settles on (EDI payerid, a future
federal payer registry, etc.) without another normative cycle, while the
endpoint-discovery guidance gives implementers a deterministic resolution
algorithm. This is also less coupled to a CARIN-namespaced system URI that the
work group has flagged as needing to migrate to a more official space.

### Disposition C: Decline

#### Proposal

Decline the request as **Not Persuasive**, with the rationale that:

1. The EDI payerid system as currently published lives in the CARIN IG
   namespace and has not yet been clearly defined as distinct from NAIC or TIN
   identifiers, both of which are already permitted on a payer Organization in
   HIPAA-covered claims contexts.
2. The HRex specification already recommends the endpoint discovery mechanism
   as the way for a payer to advertise its preferred identifier, so adding
   another named identifier slice does not provide additional implementer
   value.
3. The reporter themselves notes uncertainty about whether NAIC or EDI payerid
   should be preferred, suggesting the underlying interoperability question is
   not yet settled enough to encode into a normative profile.

Leave the door open: if a future submission supplies a use case that is
clearly distinct from NAIC/TIN and uses a payerid system URI in a stable,
non-CARIN-IG-internal namespace, the work group will reconsider.

#### Justification

This matches the work group's existing resolution and is defensible on three
grounds: namespace stability (the CARIN payerid system URI is acknowledged as
needing to move), redundancy (NAIC and TIN already cover the HIPAA payer
identifier space, and any other identifier can be carried as a generic
`Organization.identifier` today), and existing mechanism (endpoint discovery
already solves the "which identifier do I use to talk to this payer?" problem
that motivates the request). Declining now avoids baking a transient
identifier-system URI into an HRex normative artifact.

---

### Recommendation

**Recommended disposition:** C

The work group has already landed on "Not Persuasive" for this ticket, and the
resolution rationale is sound: the EDI payerid system is not yet clearly
distinct from NAIC/TIN, the canonical system URI still lives inside the CARIN
IG and is expected to move, and HRex's endpoint-discovery mechanism already
covers the implementer need that motivates the request. Disposition A would
hard-code a CARIN-internal system URI into HRex and create a churn risk when
that URI migrates; Disposition B is conceptually appealing but is broader than
what this ticket actually asks for and would deserve its own targeted request
with implementer input. Confirming Disposition C — while explicitly inviting
the reporter to resubmit once the EDI payerid system has a stable home and a
use case clearly distinct from NAIC/TIN — is the cleanest and lowest-risk path,
and matches the guidance the work group has already provided.
