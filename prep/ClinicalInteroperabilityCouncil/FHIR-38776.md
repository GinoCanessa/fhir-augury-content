## Summary

Karen Thullner is requesting that example instances be authored for the
profiles defined in the US Breast Cancer Radiology Reporting (FHIR)
implementation guide. The ticket is filed against the Clinical
Interoperability Council work group with the title "Make Examples" and a
one-line description ("Create profile examples"). It asks for editorial /
authoring work on the IG itself rather than a normative change to the
FHIR core specification: implementers reading the IG need concrete
example resources that conform to each profile so they can validate
their own implementations and understand intended usage. No
cross-referenced Jira tickets were surfaced for this ticket, so its
scope is entirely self-contained to the Breast Cancer Radiology
Reporting IG's profile set.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the change request and produce at least one conformant example
instance per profile defined in the US Breast Cancer Radiology Reporting
(FHIR) IG. Concretely:

1. Enumerate every `StructureDefinition` of `kind=resource` (profile)
   shipped by the IG.
2. For each profile, author an `Example` resource (FSH `Instance: ...`
   with `Usage: #example`, or an equivalent JSON/XML example file)
   that:
   - asserts `meta.profile` to the profile's canonical URL,
   - populates all `MS` (must-support) and `1..*` cardinality
     elements with realistic breast-cancer-radiology values, and
   - validates clean against the IG publisher.
3. Reference each new example from the profile's intro page so it
   appears under the profile's "Examples" tab in the published IG.
4. Republish the IG and verify QA shows no new errors and that example
   counts on each profile page are non-zero.

#### Justification

Implementation guides without examples are notoriously hard to adopt;
implementers rely on examples to disambiguate slicing, value-set
bindings, and intended use of must-support elements. HL7's published IG
guidance and the FHIR IG publisher's QA reports both treat
"profile-with-no-example" as a quality smell. The reporter has not
asked for any structural change — only for editorial content the IG is
already expected to carry — so accepting as requested simply closes a
known gap rather than altering conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than producing a separate hand-authored example per profile,
author one or two end-to-end **clinically realistic patient scenarios**
(e.g., a screening mammogram with BI-RADS 2 finding, and a diagnostic
follow-up with biopsy-proven invasive ductal carcinoma) as `Bundle`
examples, and let the contained / referenced resources collectively
exercise every profile in the IG. Cross-link each profile page to the
scenario(s) that demonstrate it.

#### Justification

Scenario-based examples carry more implementer value than per-profile
"shape" examples because they show how the profiles compose in a real
report (Composition → DiagnosticReport → Observations → Imaging study
references). They also reduce duplication: a Patient, Practitioner, and
Organization defined once in a Bundle can satisfy the example
requirement for several profiles. The trade-off vs. Disposition A is
authoring effort per profile drops, but tooling that surfaces "examples
of this profile" on the IG profile page may need extra wiring (e.g.,
`exampleCanonical` entries on multiple `ImplementationGuide.definition.resource`
nodes pointing at the same Bundle members).

---

### Disposition C: Decline

#### Proposal

Decline the ticket as not actionable in its current form: the request
is a single-line editorial ask with no acceptance criteria, no
nominated profiles, no example data sources cited (e.g., real BI-RADS
reports or de-identified cases), and the IG is in an early authoring
state. Close as "Not Persuasive – Insufficient Detail" and ask the
reporter to refile with a prioritized list of profiles needing
examples and at least one source narrative report to draw realistic
values from.

#### Justification

A change request is normally expected to identify the artifact(s) it
targets and the change to make. "Create profile examples" against an
entire IG is an open-ended editorial backlog, not a discrete change
the WG can dispose of. Declining-with-feedback keeps the tracker honest
and avoids the ticket lingering in Triaged for years (it has already
been open since 2022-10-03 with only an empty placeholder comment).

---

### Recommendation

**Recommended disposition:** A

The request is unambiguous in intent — the IG is missing examples and
implementers need them — even if it is light on specifics. Declining
(C) protects tracker hygiene but does nothing for the actual gap, and
the gap is real: an IG with profiles but no conformant examples is
materially harder to implement and reflects poorly on ballot-readiness
for the US Breast Cancer Radiology Reporting (FHIR) specification. The
scenario-bundle alternative (B) is attractive and the WG should
strongly consider folding it into the execution of A — i.e., accept A
as the disposition, and let the editor decide per-profile vs.
scenario-bundle authoring as an implementation detail. Either way, the
work is editorial and bounded to this one IG, so the cost of accepting
is low and the implementer payoff is high.
