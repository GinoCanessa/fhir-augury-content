## Summary

The ticket reports an inconsistency on the "Primary extensions defined as
part of this Implementation Guide" page within the US Breast Cancer
Radiology Reporting (BCRR) FHIR IG. The page heading implies the
extensions listed are first defined in BCRR, but in practice the page
also enumerates extensions that originate elsewhere — notably from the
mCode IG (e.g., `LandmarkLocation`,
`LandmarkToBodyLocationDirection`, `Laterality`). The reporter
proposes either renaming the heading to something more accurate (e.g.,
"Extensions used in this Guide") and adding a column showing each
extension's source/basis, or restricting the listing to only those
extensions actually created in this IG. The ticket has been triaged
with a resolution of *Persuasive* and a stated intent to make the
change in the next ballot. There are no cross-referenced Jira tickets,
Zulip threads, or GitHub items in the Augury index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the BCRR IG extensions index page to honor the existing heading
"Primary extensions defined as part of this Implementation Guide" by
restricting the listing to only those extensions whose canonical URL is
defined in the BCRR IG itself. Remove (or move to a separate "Extensions
re-used from other IGs" section) any entries whose canonical resolves
outside BCRR — including the mCode-defined extensions called out in the
ticket: `LandmarkLocation`, `LandmarkToBodyLocationDirection`, and
`Laterality`. No element-level definitional change is required; this is
narrative/template scoping on the IG's extensions index.

#### Justification

This option preserves the existing heading wording (which implementers
have already encountered) and aligns the page contents with that
heading by tightening what gets listed. It is the minimum-impact
change that fixes the reported inconsistency, matches the second
option the reporter suggested, and cleanly separates "ours" from
"borrowed" extensions — a common expectation for IG readers reviewing
provenance.

---

### Disposition B: Alternative Approach

#### Proposal

Rename the section heading from "Primary extensions defined as part of
this Implementation Guide" to "Extensions used in this Guide" (or
equivalent) and add a "Source" / "Defined in" column to the table that
shows, for each extension, the originating IG (e.g., BCRR, mCode, FHIR
core). Keep the unified listing so readers see in one place every
extension a BCRR conformant resource may carry. Optionally, sort or
group rows by source IG so locally-defined extensions are visually
distinct from re-used ones.

#### Justification

This is the first option proposed by the reporter and addresses the
reader's likely real need: understanding which extensions appear in
BCRR-conformant instances and where each one is defined. A combined
table with provenance is more useful than two separate lists, avoids
forcing readers to flip between pages, and makes re-use of mCode
extensions explicit — a positive signal about ecosystem alignment.
The cost is slightly more template work than Disposition A, but no
normative content changes.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" on the grounds that the page in question is
auto-generated from the IG Publisher's standard extensions index, that
the existing heading is conventional across HL7 IGs, and that any
listed extension referenced by a BCRR profile is, in a working sense,
"part of" this IG even when its canonical lives elsewhere.

#### Justification

This stance would only be defensible if the heading text were
fixed by tooling and could not be overridden, or if the workgroup
believed the current wording is already understood by implementers as
"all extensions you'll see in this IG." The ticket itself is already
recorded with a resolution of *Persuasive* and a commitment to fix in
the next ballot, so this disposition is the least consistent with the
existing record and is included primarily for completeness of the
trade-off discussion.

---

### Recommendation

**Recommended disposition:** B

Disposition B most directly answers the reader's underlying question —
"where does each of these extensions come from?" — without losing the
single-page overview of every extension that may appear in a
BCRR-conformant instance. Adding a "Source" column is a small
narrative/template change with no impact on profiles, value sets, or
conformance, and it explicitly surfaces BCRR's reliance on mCode for
the landmark/laterality extensions called out in the ticket. It also
matches the reporter's first-stated preference and is consistent with
the ticket's existing *Persuasive* resolution. Disposition A is a
reasonable fallback if the workgroup prefers minimal template changes
and is willing to lose the at-a-glance "what extensions might I see in
BCRR?" view; Disposition C is not recommended given the ticket has
already been agreed to in principle.
