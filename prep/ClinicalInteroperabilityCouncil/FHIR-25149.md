## Summary

The reporter (Susan Matney) filed this ballot comment (tracked as
"BC #100") against the US Breast Cancer Radiology Reporting (FHIR) IG,
requesting that an element currently typed as the generic
`Reference(Resource)` be tightened so that it must be able to carry —
at minimum — a breast radiology report. In other words, the concern
is that a reference to "any resource" is too permissive for the
semantic being modeled, and the IG should constrain the target type(s)
so that a breast radiology report (typically represented as a
`DiagnosticReport` profiled for breast imaging) is required as one of
the allowed reference targets. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were located for this ticket, so the
analysis rests entirely on the ticket text itself and on standard FHIR
modeling patterns for references in IGs.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Locate the element in the US Breast Cancer Radiology Reporting (FHIR)
IG whose type is currently `Reference(Resource)` and tighten its type
to include, at minimum, a breast radiology report. Concretely, change
the element's reference target from `Reference(Resource)` to
`Reference(DiagnosticReport)` — and, if the IG defines a breast
imaging `DiagnosticReport` profile (e.g., a "Breast Radiology Report"
profile), further constrain the reference to
`Reference(<that profile's canonical URL>)`. Document the narrower
target in the element definition and add an invariant or short
description noting that the referenced resource is expected to be a
breast radiology report.

#### Justification

`Reference(Resource)` is the most permissive reference type in FHIR
and is widely discouraged in IGs because it provides no semantic
guardrails for implementers and no validation hook against the
intended payload. Constraining the type to `DiagnosticReport` (and, if
present, the IG's breast radiology profile) matches the ballot
comment's explicit ask, aligns with the general FHIR style guide's
preference for the narrowest reasonable reference target, and lets
profile-aware validators confirm that a conforming instance actually
points at a breast radiology report rather than an arbitrary resource.
This is consistent with how other oncology / imaging IGs (e.g.,
mCODE, US Core DiagnosticReport for Report and Note exchange)
constrain their report references.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of narrowing the reference to a single target type, change the
element from `Reference(Resource)` to a **choice** of references that
covers the realistic range of breast-imaging artifacts — for example
`Reference(DiagnosticReport | ImagingStudy | DocumentReference)` —
and then **slice** the reference so that at least one slice requires
the IG's breast radiology `DiagnosticReport` profile (using
`min = 1` on that slice). This preserves the ability to also carry
the underlying imaging study or a document-style report, while still
guaranteeing that a conforming instance includes a breast radiology
report.

#### Justification

In real-world breast-cancer reporting workflows, the clinically useful
artifact is sometimes an `ImagingStudy` (raw DICOM metadata), a
`DiagnosticReport` (structured radiology report), or a
`DocumentReference` (narrative PDF/CDA). A single-target
`Reference(DiagnosticReport)` (Disposition A) risks forcing
implementers to re-model existing data just to conform. Slicing the
reference lets the IG state "you MUST include a breast radiology
report" — satisfying the ballot comment — while also acknowledging
that existing systems may additionally reference the imaging study or
a document-based report. The cost is additional profile complexity
and slightly more involved validation.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** / **Will Not Fix** on the
grounds that the affected element is intentionally typed as
`Reference(Resource)` to let the IG cover the full range of imaging
artifacts a breast cancer reporting workflow may attach, and that the
expectation of "at least a breast radiology report" is already
conveyed in the IG's narrative / usage notes rather than in a type
constraint.

#### Justification

This disposition is defensible only if the IG's authors determine
that the element was deliberately left open because the IG is a
reporting shell that references heterogeneous evidence, and that
enforcing a specific target type in the base profile would break
existing implementers who currently point at
`ImagingStudy`, `DocumentReference`, or custom resources. In that
case the existing narrative guidance is argued to be sufficient, and
the ballot comment is treated as a preference rather than a defect.
The weakness of this disposition is that it leaves `Reference(Resource)`
in the wire format, which provides no machine-checkable guarantee
that the referenced resource is a report at all, let alone a breast
radiology report — exactly the gap the reporter is flagging.

---

### Recommendation

**Recommended disposition:** B

Disposition B directly answers the ballot comment ("must include at
least a breast radiology report") by requiring a sliced reference to
the IG's breast radiology `DiagnosticReport` profile, while avoiding
the implementer burden and backward-compatibility risk of forcing a
single narrow target type as in Disposition A. `Reference(Resource)`
is essentially always an anti-pattern in a published IG, so
Disposition C is hard to justify on its own merits; the ticket's
resolution note ("We will discuss this.") also signals the authors
already consider the status quo debatable. Slicing gives CIC a
forward-compatible answer: the MUST constraint the reporter is
asking for is present, implementers who already reference an
`ImagingStudy` or `DocumentReference` can keep doing so as
additional slices, and the IG gains a validator-checkable guarantee
that every conforming instance carries a breast radiology report.
Before finalizing, the workgroup should confirm which specific
element (and which profile) the ballot comment targets, since the
ticket text does not name it explicitly.
