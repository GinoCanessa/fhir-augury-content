## Summary

Ron G. Parker requests that the value sets defined within the mCODE
(US Minimal Common Oncology Data Elements) FHIR Implementation Guide
be anchored in the HL7 Terminology server (THO, terminology.hl7.org)
rather than being scoped to the IG's own canonical namespace. The
rationale stated by the reporter is reuse: many of the oncology value
sets defined in mCODE are likely to be useful beyond mCODE itself, and
hosting them in THO would make them discoverable, citable, and reusable
by other IGs (e.g., other CIC-sponsored guides, registries, EHR
profiles) without forcing those consumers to depend on mCODE. The
ticket was originally dispositioned during the STU4 cycle as
"Considered for Future Use" (5-0-4 vote, Tcheng / Potteiger, 2024-06-27)
on the basis that other widely-used FHIR IGs (notably US Core) anchor
their value sets in the IG or in base FHIR rather than in THO, and
mCODE wished to remain consistent with that pattern. Lynn Laakso
re-opened the ticket post-STU4.0.0 publication on 2025-02-17 and
reverted the prior resolution so that the working group can revisit the
question for the next ballot cycle. No Jira, Zulip, or GitHub
cross-references were surfaced for this ticket via fhir-augury.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate mCODE-defined value sets (and, where applicable, the
CodeSystems they intension) from the mCODE IG canonical namespace
(`http://hl7.org/fhir/us/mcode/ValueSet/...`) to the HL7 Terminology
namespace (`http://terminology.hl7.org/ValueSet/...`). Concretely, for
each ValueSet currently authored in the mCODE IG sources (FSH under
the IG's authoring root, typically `input/fsh/`):

1. Identify the value sets that are plausibly reusable outside mCODE
   (e.g., disease-status value sets, treatment-intent, performance
   status, staging systems anchored on third-party code systems such as
   AJCC, SNOMED CT, LOINC, ICD-O-3).
2. Submit a UTG (Unified Terminology Governance) proposal to move each
   such ValueSet to THO, assigning it a `terminology.hl7.org`
   canonical and the appropriate UTG steward.
3. In the mCODE IG, replace the in-IG ValueSet definitions with
   references to the new THO canonicals; keep `bind`s in the
   StructureDefinitions pointing at the new THO canonicals.
4. Pin a specific THO release version in mCODE's
   `dependencies` / `sushi-config.yaml` so that mCODE remains
   reproducible across THO point releases.
5. Provide a redirect / deprecation note for the old mCODE-scoped
   canonicals so that existing implementers' content continues to
   resolve through at least one publication cycle.

#### Justification

THO exists precisely so that broadly-useful terminology artifacts
have a single, IG-neutral home. Several mCODE value sets — particularly
those scoped to staging, histology, and treatment intent — are already
candidates for reuse by registries (NAACCR, CDC), EHR vendors building
oncology profiles, and other HL7 IGs (e.g., Cancer Pathology Data
Sharing, Genomics Reporting). Anchoring them in THO removes mCODE as a
hard dependency for consumers and aligns with the long-stated UTG
direction that "terminology lives in THO." It also reduces the risk of
parallel, divergent definitions appearing in sibling IGs. The reporter
(Ron Parker) is well placed to identify which specific value sets meet
the "useful beyond this IG" bar.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a **selective, criteria-driven** migration rather than a
wholesale move:

1. Define a written "promotion policy" inside the mCODE IG (e.g.,
   under `input/pagecontent/`) that states the criteria a ValueSet
   must meet to be hosted in THO instead of in the IG: (a) the
   ValueSet is intensional/extensional over a code system already
   anchored in THO or externally maintained (SNOMED CT, LOINC,
   ICD-O-3, AJCC), (b) at least one non-mCODE consumer (HL7 IG,
   registry, vendor) has expressed need, and (c) the steward is willing
   to accept ongoing UTG maintenance.
2. For ValueSets that pass the criteria, file individual UTG tickets
   and migrate as in Disposition A.
3. For ValueSets that **fail** the criteria (e.g., mCODE-internal
   composition value sets, profile-specific tightenings, or
   short-lived sets used only in a single mCODE Observation profile),
   leave them anchored in the IG canonical namespace.
4. Document the rationale for each ValueSet's home (THO vs IG) in the
   IG's terminology section so the policy is auditable.

#### Justification

This addresses the reporter's underlying concern (reuse) without
incurring the overhead of moving every ValueSet — including the many
that are genuinely IG-internal "shape" sets that no other consumer
will ever need. It is also consistent with the precedent the prior
disposition cited (US Core anchors most of its value sets in the IG):
US Core does in fact host some terminology in THO and some in-IG, on
similar criteria. A criteria-driven split lets mCODE stay aligned
with US Core's actual practice while still surfacing the reusable
artifacts. It additionally creates a defensible written policy that
future tickets in this category can be dispositioned against without
re-litigating the question.

---

### Disposition C: Decline

#### Proposal

Persist the prior position: keep all mCODE value sets anchored in the
mCODE IG canonical namespace
(`http://hl7.org/fhir/us/mcode/ValueSet/...`). Resolve the ticket as
**Not Persuasive** (or restore the prior **Considered for Future Use**)
with the rationale that consistency with US Core and other widely-used
FHIR IGs outweighs the reuse benefit at this time, and that consumers
who wish to reuse a specific ValueSet may do so today by referencing
the mCODE canonical directly.

#### Justification

This is the position the workgroup adopted in 2024 (5-0-4) and the
reasoning has not materially changed: most major US-realm IGs anchor
their value sets in the IG, FHIR consumers are accustomed to that
pattern, and a wholesale move to THO would create a multi-cycle
migration burden (UTG submissions, dependency-version pinning,
breaking-canonical changes for any current implementer). If no
concrete reuse demand has been documented in the year since the
original disposition, the case for moving has not strengthened.

---

### Recommendation

**Recommended disposition:** B

The original 5-0-4 vote was not a strong endorsement of "leave
everything in the IG"; it was effectively a "not now" with an explicit
"reconsider in the future" hook, which is exactly why Lynn Laakso
re-opened it post-STU4.0.0. Disposition A goes further than the
ticket actually requires and risks churn for genuinely IG-internal
value sets. Disposition C ignores the re-open and the reporter's
unrebutted reuse argument. Disposition B captures the spirit of the
request — reusable terminology should live in THO — while giving the
workgroup an objective, documentable policy that mirrors what US Core
and other mature IGs already do in practice. It also produces a
deliverable (the written promotion policy) that future tickets in
this family can be dispositioned against cheaply, which is valuable
to CIC given that mCODE will continue to grow its terminology
footprint.
