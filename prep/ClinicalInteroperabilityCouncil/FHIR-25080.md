## Summary

The submitter (Claude Nanjo) is responding to a question raised by the
ballot submission team for the US Breast Cancer Radiology Reporting (BC
#57) IG: should a Diagnostic Report be modeled as a `Composition` or as a
`DiagnosticReport`? The submitter argues that `Composition` is the right
choice, and goes further to suggest that `DiagnosticReport` should either
be eliminated as a resource or redefined as a profile on `Composition`,
with all resources that currently reference `DiagnosticReport` rebound to
the corresponding `Composition` profile. This is a base-FHIR-level
suggestion raised in the context of an IG ballot comment rather than a
narrow IG fix. The ticket has been triaged with a resolution of
"Not Persuasive with Modification": the work group agreed that
`Composition` would ultimately be a better solution but could not
restructure the IG within the ballot cycle, and instead committed to
using `DiagnosticReport` as the root with `Observation`s acting as
section headers, pending feedback over subsequent ballot cycles. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the data sources for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure the US Breast Cancer Radiology Reporting (BC #57) IG so the
report-level artifact is a profile on `Composition` rather than a
profile on `DiagnosticReport`. Specifically:

- Replace the BC radiology report profile's base resource with
  `Composition`, modelling the narrative report as `Composition.section`
  entries (e.g., findings, impression, recommendations) and binding each
  section's `Composition.section.entry` to the appropriate breast-cancer
  observation profiles.
- Update every IG-defined profile and example that currently references
  the BC `DiagnosticReport` profile so that it instead references the
  new BC `Composition` profile.
- Carry the request forward as a base-spec / O&O conversation: file a
  companion change request against the FHIR core spec proposing that
  `DiagnosticReport` be either retired or redefined as a profile on
  `Composition`, with downstream references migrated accordingly.

#### Justification

The submitter's argument aligns with FHIR's documented document/section
pattern: when a reporter is conveying an authored, structured narrative
with sections and a signed attestation, `Composition` is the
canonically-correct backbone. The work group's own resolution text
explicitly concedes that "`Composition` would be a better solution",
which makes Disposition A consistent with the work group's stated
design preference. Choosing it now (rather than deferring again) avoids
compounding the mismatch over additional ballot cycles and gives
implementers a single, durable shape rather than a `DiagnosticReport`
shape that is signalled as transitional.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `DiagnosticReport` as the root of the BC radiology report for this
ballot cycle, exactly as the existing resolution describes — i.e.,
"Diagnostic Report as the root and Observations as the section headers"
— but make the transitional intent explicit and falsifiable:

- Add an IG-level note to the BC report profile declaring that the
  long-term direction is a `Composition`-based model, that the current
  `DiagnosticReport`-based shape is a ballot-cycle compromise, and that
  re-evaluation is expected based on ballot feedback.
- Pre-define, in the IG's design notes, the mapping from each
  `DiagnosticReport` element used by BC #57 to its eventual
  `Composition` equivalent (e.g., `DiagnosticReport.result` →
  `Composition.section.entry`, `DiagnosticReport.conclusion` →
  `Composition.section` for an impression section), so that a future
  migration is mechanical rather than a re-design.
- Track the base-spec conversation with O&O as a separate work item
  rather than blocking the IG's ballot.

#### Justification

This matches what was actually applied in the existing resolution
("Not Persuasive with Modification") but raises its rigour: the IG
ships on schedule, implementers are not whipsawed mid-ballot, and the
work group preserves a clean migration path. It also respects the
practical reality flagged in the resolution — that the change touches
referring resources and requires O&O coordination — while still moving
the design conversation forward in a structured way. Trade-off vs.
Disposition A: implementers will eventually need to migrate, but the
migration cost is bounded by the documented mapping.

---

### Disposition C: Decline

#### Proposal

Decline the request to restructure the BC radiology report around
`Composition` (and the broader request to retire / re-base
`DiagnosticReport`). Retain `DiagnosticReport` as the report-level
artifact in BC #57 and leave the FHIR core `DiagnosticReport` resource
unchanged. Rationale category: out of scope for the IG-level ticket and,
for the base-spec portion, a breaking change to a widely-implemented
core resource that is not justified by a single ballot comment.

#### Justification

`DiagnosticReport` is one of the most widely-implemented FHIR
resources, with extensive deployment across labs, radiology, and
pathology, and a large set of resources that already reference it.
Re-basing it on `Composition` (or eliminating it) would be a sweeping
backwards-incompatible change to the core spec — well outside the
scope a single IG-level ballot comment can drive. Within the IG itself,
`DiagnosticReport` is already the conventional choice for a clinical
report, and switching to `Composition` mid-ballot would diverge from
peer radiology IGs and impose unnecessary implementer churn.

---

### Recommendation

**Recommended disposition:** B

The current resolution on file is effectively Disposition B, and the
work group's own resolution text is candid that `Composition` is the
better long-term shape but cannot be delivered in the current ballot
window without coordinated changes to referring resources and O&O
buy-in. Disposition A is directionally right but, taken now, would
either slip the ballot or ship an unreviewed restructure that affects
core resource references — neither of which is responsible for an IG
that implementers are already piloting. Disposition C is too strong:
the work group has explicitly agreed `Composition` is preferable, so
declining outright would misrepresent the design intent. Disposition B
preserves the ballot, locks in the transitional intent in writing, and
pre-stages the migration so the eventual move to `Composition` (driven
through the proper base-spec / O&O channel) is mechanical rather than
speculative.
