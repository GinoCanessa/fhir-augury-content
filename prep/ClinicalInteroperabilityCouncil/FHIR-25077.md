## Summary

The reporter (Claude Nanjo) raises a concern that the US Breast Cancer
Radiology Reporting IG mixes logical model definitions and FHIR resource
profiles without clearly distinguishing them. The example given is
`Observation.bodySite`: in the IG it is presented with sub-attributes
such as *laterality* and *anatomicalLocation*, but in R4 the underlying
`Observation.bodySite` element is a single `CodeableConcept`. The
ticket requests that the IG explicitly delineate which artifacts are
logical structure definitions (CIMPL-style models) and which are
profiles constraining real FHIR resources, so implementers are not
misled into thinking the logical-model shape is the on-the-wire FHIR
shape. The ticket has been marked **Persuasive** and the recorded
resolution states the next ballot will drop the CIMPL logical models
and adhere more closely to the base FHIR resources.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the US Breast Cancer Radiology Reporting IG so that every
artifact page makes its kind unambiguous. Concretely:

1. In each artifact's narrative header, add a one-line "Artifact kind"
   label — either *Logical Model (CIMPL-derived)* or *FHIR Profile on
   `<BaseResource>`* — with a short sentence explaining that a logical
   model is a domain-modeling abstraction and is **not** instantiable
   as a FHIR resource on the wire.
2. Group artifacts in the IG's table of contents and the "Artifacts"
   index page into two sections — "Logical Models" and "Profiles" —
   instead of listing them in a single flat list.
3. For elements like `Observation.bodySite` where the logical model
   exposes sub-attributes (laterality, anatomicalLocation, …) that do
   not exist on the base FHIR element, add an inline note on the
   element page stating that on the wire these sub-attributes must be
   conveyed via the corresponding base-FHIR mechanism (e.g.,
   `bodySite` as `CodeableConcept` plus `bodyStructure`/`Location`
   references, or component observations), not as direct children of
   `Observation.bodySite`.

#### Justification

This matches exactly what the reporter asked for and is the lowest-risk
intervention: it adds clarifying documentation and reorganizes the
artifact index without changing any StructureDefinitions, examples, or
conformance expectations. It directly addresses the root cause of the
confusion — readers cannot distinguish a logical model from a profile —
and is consistent with general FHIR IG authoring guidance that logical
models and profiles should be visually and structurally separated.
Because the ticket was filed for the BC ballot review, accepting it as
requested also gives reviewers something concrete to verify in the next
publication.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only relabeling the existing artifacts, restructure the IG
so that the published conformance artifacts are **FHIR profiles on the
base resources** and the CIMPL logical models are demoted to a
non-normative "Domain Analysis Model" appendix. Specifically:

1. For every CIMPL-derived structure that is currently presented as if
   it were a profile, author a real FHIR profile on the corresponding
   base resource (e.g., a `BreastCancerBodySite` profile that
   constrains `Observation.bodySite` as a `CodeableConcept` bound to a
   breast-anatomy ValueSet, with laterality / anatomicalLocation
   represented via established mechanisms such as `bodyStructure`,
   component observations, or standard extensions where no base
   element exists).
2. Move the CIMPL logical models into an appendix labeled "Domain
   Analysis Model — Informative" with a banner stating they are not
   conformance artifacts.
3. Update examples and the conformance section to reference the new
   profiles, not the logical models.

This is essentially the direction the recorded resolution already
points to ("for the next ballot we will not be using these logical
models and instead adhere more closely to the base FHIR resources").

#### Justification

The reporter's underlying concern is not really a documentation
problem; it is that the IG's conformance surface looks like FHIR but
isn't. Fixing only the labels (Disposition A) leaves implementers
parsing logical-model shapes that they cannot put on the wire. Rebuilding
the conformance artifacts as real profiles aligns the IG with FHIR
authoring norms, removes a long-standing source of implementer
confusion, and is consistent with the resolution the work group has
already endorsed. The cost is higher — it is a substantive ballot's
worth of work — but it converts the ticket from a doc fix into a
durable structural correction.

---

### Disposition C: Decline

#### Proposal

Decline to make any change in the current ballot cycle on the grounds
that the issue is already fully addressed by the recorded resolution
("next ballot will not use the CIMPL logical models and will adhere
more closely to the base FHIR resources"). Close the ticket as
**Persuasive — Will Be Addressed In Next Ballot** with no further
narrative or structural edits to the present publication.

#### Justification

Since the work group has already committed to dropping the CIMPL
logical models in the next ballot, layering a relabeling pass on top
of artifacts that are about to be replaced wholesale would consume
editorial effort that is better spent on the replacement profiles.
Declining further work here keeps the change log clean and avoids
publishing intermediate guidance that will itself be obsolete in one
ballot cycle.

---

### Recommendation

**Recommended disposition:** B

The ticket is marked Persuasive and the resolution text already commits
the work group to moving away from CIMPL logical models toward profiles
on base FHIR resources, which is exactly Disposition B. Disposition A
addresses only the symptom (mislabeling) and would leave the IG
publishing logical-model shapes that don't match the on-the-wire FHIR
they appear to constrain — the same problem the reporter raised, just
with a clearer warning sticker. Disposition C aligns with the recorded
resolution but produces no visible artifact for the work group or
implementers, and risks the next ballot arriving without the
restructuring actually planned. Disposition B operationalizes the
recorded resolution: it produces real profiles on base resources
(starting with the `Observation.bodySite` example called out in the
ticket), demotes the logical models to an informative appendix, and
gives implementers a conformance surface that matches FHIR. The work
group can scope which artifacts to convert first, but the direction —
profiles on base resources, logical models as informative — should be
adopted as the disposition of this ticket.
