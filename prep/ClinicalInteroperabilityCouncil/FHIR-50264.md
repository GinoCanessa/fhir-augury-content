## Summary

The ticket asserts that the Da Vinci HRex IG is mature enough that its
"temporary" placeholder code systems should be retired in favour of
permanent homes for each set of codes. Specifically:

- **UMB** (the "Unique Member Identifier" identifier-type code used in
  member match on the Patient profile) should be moved to the HL7 v2
  identifier type code system. Lloyd notes this is being tracked
  separately as **UP-666** in the HL7 UTG/UP project.
- The **Task codes** (used in CDex and CRD on the Task profile to
  request data) should be promoted into the FHIR core specification
  and from there to UTG. This work is being handled in parallel under
  [FHIR-50258](https://jira.hl7.org/browse/FHIR-50258), which has
  since been resolved Persuasive (codes `data-request`, `data-query`,
  `data-code`, `data-value` added to `task-code` and
  `task-inputoutput-parameter-type`) and applied via
  [HL7/fhir PR #3655](https://github.com/HL7/fhir/pull/3655).
- The **endpoint discovery codes** (used in the HRex logical model
  defining the JSON syntax for endpoint discovery, only consumed
  inside HRex today) were originally proposed to live in an
  IG-specific code system. The agreed resolution is more ambitious:
  move them to **THO** (Terminology.HL7.org) so new codes can be
  added without re-publishing HRex and so the codes can potentially
  align with FAST.

In all three cases, the existing HRex temporary codes will be retained
but **deprecated** to allow a smooth transition for current
implementers. The disposition was reviewed and approved on the
**14 May 2025 CDex call**.

## Proposed Dispositions

### Disposition A: Accept As Requested (i.e. accept the agreed resolution verbatim)

#### Proposal

Apply the resolution that was reviewed and approved on the 14 May
2025 CDex call:

1. **UMB** — Stop publishing UMB as an HRex-defined identifier-type
   code. In the HRex member-match Patient profile, re-bind the
   relevant `Patient.identifier.type.coding` slice to the v2
   identifier type code system (`http://terminology.hl7.org/CodeSystem/v2-0203`),
   using the new code being added under [UP-666](https://jira.hl7.org/browse/UP-666).
   Mark the existing HRex UMB code as **deprecated** (status
   `deprecated` in the HRex CodeSystem, narrative pointer to the v2
   replacement) so existing implementations keep validating during
   the transition.
2. **Task codes** — Replace HRex's temporary `data-request` /
   `data-query` / `data-code` / `data-value` codes with references
   to the now-Applied core codes from
   [FHIR-50258](https://jira.hl7.org/browse/FHIR-50258):
   `http://hl7.org/fhir/CodeSystem/task-code#data-request` and the
   three core `task-inputoutput-parameter-type` codes. Re-bind the
   HRex Task profile (and any CDex/CRD bindings inherited from
   HRex) accordingly. Deprecate the HRex copies.
3. **Endpoint discovery codes** — Define a new permanent code system
   on **THO** (Terminology.HL7.org) for the HRex endpoint-discovery
   codes (the codes used in the member-match logical model that
   defines the JSON syntax for endpoint discovery). Re-point the
   HRex logical model to that THO code system. Keep the existing
   HRex codes published but **deprecated** so legacy
   implementations continue to validate.
4. Add migration guidance to HRex (Notes / Migration page) describing
   the old → new code mappings and the deprecation timeline; flag
   the change in the HRex changelog.

The endpoint-discovery code migration has already been kicked off
in HRex master commit `71ed163`; this disposition formalises that
work and pulls the rest of the resolution along with it.

#### Justification

This is the disposition the responsible workgroup (CDex / CIC) has
already reviewed and approved, and it is consistent with FHIR's
standard practice of homing widely-reusable codes in their natural
authority (v2 identifier types live in `v2-0203`; cross-IG
workflow codes live in core / UTG; cross-version IG codes live in
THO so they can evolve out-of-band). The approach explicitly
preserves backward compatibility by keeping deprecated stubs in
HRex, which protects current member-match, CDex, and CRD
implementers. Aligning the endpoint-discovery codes with FAST is a
real near-term need that an IG-local code system would block.

---

### Disposition B: Alternative — keep endpoint-discovery codes in an HRex-owned code system (the original ticket text)

#### Proposal

Apply parts (1) and (2) above (UMB → v2-0203 via UP-666; Task codes
→ FHIR core via FHIR-50258), but **do not** move the endpoint
discovery codes to THO. Instead, formalise them in a permanent
HRex-owned `CodeSystem` (e.g.
`http://hl7.org/fhir/us/davinci-hrex/CodeSystem/endpoint-discovery-type`),
matching what the original ticket text requested. Continue to
publish them inside the HRex IG package and re-bind the HRex
member-match logical model to the new HRex code system. Deprecate
the temporary codes as in Disposition A.

#### Justification

Pros: simpler operational model — HRex retains full editorial
control of codes that are (today) only used inside HRex; no
dependency on THO publication cadence; one less cross-repo
coordination point. Cons: every new code requires a new HRex
publication, and any future FAST / cross-IG alignment forces a
larger restructure later. This is essentially the path the ticket
*originally* asked for; the workgroup considered it during the
14 May discussion and chose Disposition A's THO route instead.

---

### Disposition C: Decline (no migration; keep the HRex temporary code systems as-is)

#### Proposal

Close the ticket with no changes. The HRex IG continues to publish
the UMB, Task, and endpoint-discovery codes from its existing
"temporary" code systems. UP-666 and FHIR-50258 may still proceed
independently in their own projects, but HRex makes no binding
changes and issues no deprecations.

#### Justification

The only defensible argument for Decline is implementation
disruption: deployed member-match, CDex, and CRD implementations
already validate against the existing HRex codes, and re-binding
profiles can ripple through downstream IGs and conformance
infrastructure. However, the resolution already mitigates this by
keeping deprecated stubs in HRex, so this rationale is weak.
Declining would also leave HRex permanently dependent on
"temporary" code systems that the IG itself acknowledges are no
longer appropriate, and would block the FAST-alignment use case
behind the THO move.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already vetted and approved this exact path on the
14 May 2025 CDex call, the parallel core/UTG work
([FHIR-50258](https://jira.hl7.org/browse/FHIR-50258)) is Applied,
and the HRex-side endpoint-discovery migration is partly underway in
master commit `71ed163`. Disposition A finishes a coherent piece of
work with low implementer risk (deprecated stubs preserve current
behaviour) and unlocks future FAST alignment. Disposition B
re-opens a question the workgroup already settled and would have to
be undone as soon as a second IG wants to reuse the
endpoint-discovery codes. Disposition C leaves a mature IG sitting
on temporary code systems indefinitely and is hard to defend on
governance grounds. The workgroup should ratify A and convert the
ticket into the implementation tickets needed against the HRex
repo (and confirm the THO publication path for the
endpoint-discovery code system).
