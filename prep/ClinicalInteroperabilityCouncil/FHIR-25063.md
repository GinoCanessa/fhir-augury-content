## Summary

Gary Dickinson submitted ballot comment "BC #11" against the US Breast Cancer
Radiology Reporting (FHIR) implementation guide, observing that it is not
clear if or how provenance is represented in the IG. The reporter argues that
provenance — capturing **who, what, when, where, and why** — is essential both
at the **point of origination** (when content is first captured) and at every
**point of update** (where prior content and its provenance must be
preserved). The reporter also notes that provenance binding may need to be
expressed at varying granularities, from the whole dataset down to individual
elements. The ticket asks the work group to clarify how the IG conveys this
information so that originators, performers, authors, organizations,
locations, devices, and rationales for both clinical action and information
capture can be unambiguously identified.

No related Jira tickets, Zulip threads, or GitHub items were found via the
`fhir-augury-cli` cross-reference or search calls, so the request stands on
its own ballot text. The ticket has already been triaged with a resolution of
**Persuasive** and a resolution comment indicating the work group agreed the
provenance elements are not well defined in the balloted version and intend
to define them more thoroughly in the next ballot.

> Comment:
>
> Not clear if/how provenance is represented in this FHIR Implementation
> Guide. Provenance (who, what, when, where, why) is essential at: 1) point
> of origination (at the source, for newly captured content), 2) point of
> update (for newly updated content, preserving previous content and its
> provenance).
>
> - Who = subject of care/information (typically the patient)
> - Who = performer of action taken
> - Who = author of information captured or updated
> - Who = organization
> - What = action taken
> - When = date/time of action taken
> - When = date/time of information capture or update
> - Where = physical location of action taken
> - Where = physical location of information capture or update
> - Where = network address and/or device ID where information captured or updated
> - Why = rationale, purpose of action taken
> - Why = rationale, purpose of information capture or update
>
> Granularity of provenance (e.g., binding authorship to content) may be at
> the dataset or element level.
>
> Summary: Clarify provenance.

## Proposed Dispositions

> Note: Because no GitHub cross-references were found, the dispositions below
> name the affected IG by specification rather than by repo path. The IG in
> question is the **US Breast Cancer Radiology Reporting (FHIR)** guide.

### Disposition A: Accept As Requested

#### Proposal

Add an explicit "Provenance" section to the US Breast Cancer Radiology
Reporting (FHIR) IG that defines, for each profiled artifact (e.g., the
DiagnosticReport / Observation / Composition profiles used to convey the
report), how each of the five W's is conveyed:

- **Who (subject)** — `subject` reference to `Patient`.
- **Who (performer of action)** — `performer` / `Observation.performer` /
  `DiagnosticReport.performer`.
- **Who (author of information)** — a profiled `Provenance` resource with
  `Provenance.agent` slices for `author` and `informant`, referencing
  `Practitioner`/`PractitionerRole`/`Device`.
- **Who (organization)** — `Provenance.agent.onBehalfOf` referencing
  `Organization`, plus IG-level guidance to populate
  `Practitioner.qualification` / `PractitionerRole.organization`.
- **What** — `Provenance.activity` bound to a value set covering
  origination vs. update.
- **When (action)** — the resource's clinically meaningful timestamp
  (`effective[x]` / `issued`).
- **When (capture/update)** — `Provenance.recorded` and
  `Provenance.occurred[x]`.
- **Where (physical)** — `Provenance.location` referencing `Location`.
- **Where (device/network)** — `Provenance.agent` of type `device` plus a
  `Device` reference; capture network address via a documented IG extension
  if required.
- **Why** — `Provenance.reason` bound to an appropriate value set.

Mandate that conformant systems SHALL produce a `Provenance` resource that
targets the report bundle/composition at point of origination and a new
`Provenance` (with `Provenance.activity = update`) at every subsequent
update, preserving prior versions via standard FHIR versioning.

#### Justification

This is the disposition the work group already signalled in the recorded
resolution ("We agree that these elements are not well defined in this
ballot, and intend to define them more thoroughly in the next ballot"). FHIR
already defines a fit-for-purpose `Provenance` resource that maps cleanly to
the reporter's who/what/when/where/why list, and reusing it keeps the IG
aligned with existing IGs (US Core, IPS) that have settled on `Provenance`
for the same need. Specifying granularity (one `Provenance` per
DiagnosticReport vs. per Observation) inside the IG resolves the reporter's
concern about dataset- vs. element-level binding without inventing new
machinery.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring bespoke profiles and conformance text inside the US
Breast Cancer Radiology Reporting IG, **adopt the US Core `Provenance`
profile by reference** and add a narrow IG-specific extension only where the
breast-cancer reporting workflow demands something US Core does not cover
(for example, a structured "network address / device ID where information
was captured" element or a value set of breast-cancer-specific update
activities). Add an IG page that:

- Cites the US Core Provenance profile as the normative dependency.
- Documents the expected `Provenance.target` granularity for this IG
  (e.g., target the DiagnosticReport at origination; target the updated
  resource at update).
- Provides one or two worked examples (origination + update) using the IG's
  existing example resources.

#### Justification

The US Core `Provenance` profile already encodes most of the who/what/when
list the reporter calls out, and it is the de-facto US realm baseline that
implementers of a US-realm radiology reporting IG will already be coding to.
Re-profiling Provenance from scratch inside this IG risks divergence from
US Core and forces implementers to support two near-identical profiles.
Trade-off vs. Disposition A: this approach is cheaper to author and easier
for implementers, but it inherits any US Core gaps (notably weaker coverage
of "where = network address / device ID" and of element-level granularity).
Those gaps can be closed with one targeted extension instead of a full
profile suite.

---

### Disposition C: Decline

#### Proposal

Mark the request as out of scope for the current ballot cycle of the US
Breast Cancer Radiology Reporting IG and direct the reporter to the base
FHIR `Provenance` and `AuditEvent` resources, which already define the
who/what/when/where/why semantics he enumerates. No IG text changes.

#### Justification

This is the weakest of the three options. FHIR `Provenance` and
`AuditEvent` do exist and are normatively defined, so a strict reading lets
the IG simply rely on the base spec. However, the work group has already
recorded a "Persuasive" resolution acknowledging that the IG's current
treatment of provenance is inadequate, which directly contradicts a
"decline" outcome. Declining would also leave implementers without IG-level
guidance on which Provenance profile to use, which targets to bind, or
whether origination + update Provenance is required vs. optional — exactly
the ambiguity the ticket reports. Declining is only defensible if the work
group decides provenance belongs in a separate, future companion guide
rather than in this IG.

---

### Recommendation

**Recommended disposition:** B

The recorded resolution is already "Persuasive", so Disposition C is
inconsistent with the work group's own position and should not be the
opening offer. Between A and B, Disposition B better serves US-realm
implementers: the US Core `Provenance` profile is the established baseline
and is what radiology-reporting systems in the US are most likely to
already implement. Adopting US Core by reference, plus a small IG-specific
extension for the "network address / device ID" capture point and explicit
granularity guidance (origination Provenance targeting the
DiagnosticReport, update Provenance targeting the updated resource),
addresses every concern in the reporter's list while minimising the
authoring burden on the work group and the conformance burden on
implementers. If the work group later finds the US Core profile is too
loose for this IG's workflow, they can tighten it with derived profiles in
a future ballot — but starting from the US Core baseline keeps this IG
aligned with the rest of the US realm.
