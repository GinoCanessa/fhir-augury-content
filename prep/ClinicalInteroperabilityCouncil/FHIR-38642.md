## Summary

FHIR-38642 reports that the Questionnaire instances shipped with the ICHOM
Patient Centered Outcomes Measure Set for Breast Cancer IG (in particular
`Questionnaire-PatientReportedYear3and4.json`) fail `.NET`-based validation
(Simplifier / Firely Server) against the narrative invariants `txt-1`
(allowed XHTML elements/attributes only) and `txt-2` (narrative must contain
non-whitespace content). Investigation in the ticket comments traced the
root cause to the IG-generated narrative: one of the Questionnaire items has
`linkId = "EORTC QLQ-LMC21"`, and the IG Publisher embeds that `linkId`
verbatim as a table-cell `id` attribute (`<td id="item.EORTC QLQ-LMC21">`),
which is invalid XHTML because `id` values may not contain spaces. Other
Questionnaires in the IG use `linkId`s such as `N/A-Clinical` that are
similarly problematic (forward slashes).

Because the failure mode is the combination of a permissive FHIR datatype
and a brittle tooling choice, Ward Weistra opened the companion ticket
[FHIR-38654](https://jira.hl7.org/browse/FHIR-38654) against FHIR
Infrastructure asking whether `Questionnaire.item.linkId` should be
constrained (e.g., made type `id`). FHIR-38654 has since been **Published
Persuasive** with the explicit resolution that `linkId` may contain
whitespace and slashes by design, and that tooling **must not** directly
embed `linkId` values as XHTML element IDs. That disposition pushes the
remaining action for FHIR-38642 back into the ICHOM IG (fix the instances)
and/or the IG Publisher (sanitize generated IDs).

A comment by the IG editor (`vadim_peretokin`, 2023-01-31) states that a
fix has been applied in the ICHOM IG, but the ticket itself is still in
`Triaged` status with `Unresolved` resolution and no labels, so it is
awaiting formal WG disposition.

### [FHIR-38654: Questionnaire.item.linkId allows spaces, should it?](https://jira.hl7.org/browse/FHIR-38654)

Companion FHIR Infrastructure Technical Correction raised directly from
FHIR-38642's discussion. It asks whether `Questionnaire.item.linkId` should
be constrained (e.g., retyped to `id` or given a regex invariant) to prevent
the XHTML-unsafe characters that triggered the ICHOM validation failure, or
whether the IG Publisher should instead transform `linkId`s when embedding
them as element IDs. Resolution is **Persuasive / Published**: "linkIds can
have whitespaces and slashes by design. Tooling should not rely on linkIds
being valid XHTML element IDs, and should not directly embed them as such."
That outcome scopes the remaining remediation for FHIR-38642 to the ICHOM
IG instances and/or IG Publisher narrative generation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as an instance-level defect in the ICHOM Breast Cancer IG and
close it as *Persuasive with Modification*:

1. In the ICHOM Breast Cancer IG sources, rewrite every
   `Questionnaire.item.linkId` that contains a space, slash, or other
   non-`id`-safe character so that the resulting value would be a valid
   `id` token. Known offenders called out in the ticket comments:
   - `EORTC QLQ-LMC21` → e.g. `EORTC-QLQ-LMC21`
   - `N/A-Clinical` → e.g. `NA-Clinical`
   Apply the same rule across all Questionnaires in the IG, not only
   `Questionnaire-PatientReportedYear3and4`.
2. Re-run the IG Publisher and re-validate the resulting bundle in
   Simplifier and Firely Server to confirm that `txt-1` and `txt-2` pass.
3. Record in the ticket that the fix was applied (the 2023-01-31 comment
   from the IG editor already states this) and close as resolved.

#### Justification

This is the minimal, behaviour-preserving fix that unblocks downstream
.NET-based validators today. The defect is fully explained: the IG
Publisher copies `linkId` into a table-cell `id`, and XHTML `id` values
cannot contain whitespace — which is why both Simplifier and Firely
Server reject the generated narrative with `txt-1`/`txt-2`. Because
companion ticket [FHIR-38654](https://jira.hl7.org/browse/FHIR-38654) was
resolved **Persuasive / Published** with the explicit position that
`linkId` may contain whitespace or slashes and that tooling must not embed
them as XHTML IDs, the core spec will not be tightened, and ICHOM cannot
wait indefinitely for IG Publisher tooling to catch up. Fixing the IG
content also matches the IG editor's stated intent ("The fix for this has
been applied", 2023-01-31). No implementers are harmed: `linkId` values
are internal identifiers used for Questionnaire/QuestionnaireResponse
linkage and are not mandated to carry human-facing text.

---

### Disposition B: Alternative Approach

#### Proposal

Close FHIR-38642 as *Not Persuasive with Modification*: keep the
ICHOM `linkId` values as-authored (`EORTC QLQ-LMC21`, `N/A-Clinical`,
etc.) because they are meaningful instrument codes, and instead address
the root cause in the IG Publisher's narrative generator. Specifically:

1. File a follow-up issue against the IG Publisher (HL7/fhir-ig-publisher)
   requesting that the narrative-rendering code base64- or percent-encode
   (or otherwise sanitize) `Questionnaire.item.linkId` before using it as
   the value of an XHTML `id` / `name` attribute in generated tables, so
   that arbitrary legal `linkId`s round-trip to valid XHTML.
2. Until that tooling fix ships, stop regenerating narratives for the
   affected Questionnaires: author a minimal `Questionnaire.text.div`
   manually (or via `generateNarrative=false`) that passes `txt-1`/`txt-2`.
3. Re-validate the IG build with Simplifier and Firely Server, and link
   this ticket to the IG Publisher issue before closing.

#### Justification

FHIR-38654's resolution is unambiguous: `linkId`s with whitespace and
slashes are *legal by design* and tooling must not assume otherwise, so
any IG that rewrites authored `linkId` values loses fidelity with the
underlying clinical instrument (e.g., the literal EORTC QLQ-LMC21 code)
just to placate a validator. This alternative keeps the IG semantically
faithful, pushes the fix to the layer that is actually wrong (the
narrative generator), and aligns Clinical Interoperability Council's
position with the published FHIR-I guidance. The main trade-off is
timeline — the IG will continue to fail .NET validation until the IG
Publisher or a local override ships — which is why the proposal also
includes a short-term hand-authored narrative workaround.

---

### Disposition C: Decline

#### Proposal

Close FHIR-38642 as *Not Persuasive*, with the rationale that the FHIR
Infrastructure work group has already disposed of the underlying design
question in [FHIR-38654](https://jira.hl7.org/browse/FHIR-38654), and
that the concrete IG-instance defect is being tracked and fixed in the
ICHOM IG repository rather than on this spec ticket. No spec change is
required in the ICHOM Breast Cancer IG's normative content, and no
further CIC action is needed.

#### Justification

The ticket as filed is a validator bug report against an IG build
artefact, not a change request against the IG's normative content
(profiles, value sets, examples). The structural question was carved off
into FHIR-38654 and resolved Published-Persuasive: "linkIds can have
whitespaces and slashes by design. Tooling should not rely on linkIds
being valid XHTML element IDs, and should not directly embed them as
such." The remaining work is either (a) edit IG source data or (b) fix
IG Publisher — both of which are repo-level engineering tasks, not
work-group dispositions. The comment trail shows the IG editor already
reporting the fix applied (2023-01-31). Leaving the ticket open gives
the false impression that CIC owes a design answer that FHIR-I has
already provided.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest path forward for Clinical Interoperability
Council. The facts are not in dispute: Ward Weistra's root-cause analysis
identifies the exact mechanism (spaces/slashes in `linkId` → invalid
XHTML `id` in the generated narrative), the IG editor has publicly
stated the fix was applied, and FHIR-I has already closed the structural
question via [FHIR-38654](https://jira.hl7.org/browse/FHIR-38654) in a
way that puts responsibility squarely on the IG content and/or tooling
rather than the core datatype. Of the three options, A is the only one
that both (i) actually clears the validation errors today against the
.NET-based validators that real implementers are using, and (ii)
produces a visibly closed ticket on the ICHOM IG's spec trail.
Disposition B is worth pursuing **in parallel** as an IG Publisher
issue, but predicating closure of this CIC ticket on an external
tooling fix leaves the IG failing validation for an unknown period and
risks re-litigating FHIR-38654. Disposition C is defensible but is
weaker from an implementer-experience standpoint because it closes the
ticket without confirming the IG artefacts are actually fixed.
