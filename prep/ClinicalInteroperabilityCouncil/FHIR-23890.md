## Summary

The ticket asks the US Breast Cancer Radiology Reporting IG authors to
clarify whether `Patient` and `RelatedPerson` are valid sources of
authoring/attestation data on several profile elements within the IG
(specifically `Composition.author` / `Composition.attester`,
`Condition.asserter` / `Condition.recorder`, and the `performer` on the
breast abnormality `Observation` profile). The reporter argues the IG's
introduction does not say what data sources are expected in a breast
radiology report, so it is unclear whether patient-supplied content
(e.g., self breast exam observations) is in scope. The ticket asks the
work group to either further constrain the profiles to exclude
`Patient` and `RelatedPerson` references, or, if those references are
intended, to add narrative describing the valid scenarios so
implementers know how to use them.

The ticket has already been triaged as **Persuasive** with a resolution
note indicating the work group agrees these elements are not well
defined in the current ballot and intends to define them more
thoroughly in the next ballot. No related Jira tickets, Zulip threads,
Confluence pages, or GitHub items were surfaced for this ticket.

## Details

**Description:**

There are several resource author or attester elements which allow
Patient or RelatedPerson references. It's not clear from the
introduction what sources of data are expected to be included in a
breast radiology report. Should the authors and attesters in
Composition allow Patient and RelatedPerson? Should the asserter or
recorder in Condition allow Patient and RelatedPerson? Should the
performer in the breast abnormality Observation allow Patient and
RelatedPerson? If these aren't valid scenarios, the profiles should be
further constrained. If they are valid scenarios (eg. Self breast exam
data), then a description of the valid scenarios should be included
for implementers.

**Resolution Description:**

We agree that these elements are not well defined in this ballot, and
intend to define them more thoroughly in the next ballot.

No comments are recorded on this ticket.

## Proposed Dispositions

The dispositions below are scoped to the **US Breast Cancer Radiology
Reporting (FHIR)** IG profiles for `Composition`, `Condition`, and the
breast abnormality `Observation`. No GitHub repo briefing is available
for this IG in the cache, so file paths cannot be named with certainty;
each disposition references the affected profile by name.

### Disposition A: Accept As Requested

#### Proposal

Treat the request as written and resolve it in two coordinated edits to
the IG (consistent with the existing **Persuasive** disposition):

1. **Constrain the affected profile elements** so they no longer
   permit `Patient` or `RelatedPerson` as reference targets in the
   provider-authored radiology workflow. Concretely:
   - On the IG's `Composition` profile, restrict
     `Composition.author` and `Composition.attester.party` to
     `Practitioner | PractitionerRole | Organization | Device` (drop
     `Patient` and `RelatedPerson` from the allowed targets).
   - On the IG's `Condition` profile, restrict `Condition.asserter`
     and `Condition.recorder` to `Practitioner | PractitionerRole`
     (drop `Patient` and `RelatedPerson`).
   - On the IG's breast abnormality `Observation` profile, restrict
     `Observation.performer` to `Practitioner | PractitionerRole |
     Organization` (drop `Patient` and `RelatedPerson`).
2. **Add an "Authors, Attesters, and Performers" subsection to the
   IG introduction** that explicitly states the radiology report is
   produced by the imaging service and is not a vehicle for patient-
   or family-reported content; point implementers wanting to capture
   self-exam findings to a separate Observation outside the report
   Composition.

#### Justification

The IG's stated scope is *radiology reporting* — the provider-authored
artifact produced by an imaging service. The reporter's concern is
that the profiles, by inheriting the broad reference targets from the
base resources, silently permit patient-authored content with no
implementer guidance. Constraining the targets and documenting the
expectation in the introduction removes the ambiguity in one
direction (the simpler one) and is consistent with the way other
provider-attested IGs (e.g., diagnostic report profiles in US Core)
narrow `author`/`performer` to provider actors. The resolution note on
the ticket already commits the work group to defining these elements
"more thoroughly in the next ballot," which this disposition operationalizes.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Patient` and `RelatedPerson` permitted on the affected elements
but **make the ambiguity explicit** rather than removing it:

1. Leave `Composition.author`/`attester.party`,
   `Condition.asserter`/`recorder`, and the breast abnormality
   `Observation.performer` reference targets unchanged from base FHIR.
2. Add a normative narrative subsection to the IG introduction titled
   "Patient- and RelatedPerson-Sourced Content" that:
   - Lists the elements that may carry `Patient`/`RelatedPerson`
     references.
   - Enumerates the valid scenarios (e.g., self breast exam findings
     contributed alongside the provider report, family history
     asserted by a relative).
   - States that when `Patient`/`RelatedPerson` is the asserter or
     performer, implementers SHALL also populate
     `Observation.method`/`Condition.evidence` (or an IG extension)
     identifying the source as patient- or family-reported, so
     consumers can distinguish provider- vs. patient-asserted findings.
3. Optionally add a `MustSupport` flag on the asserter/performer
   element so implementers know reading the source is required.

#### Justification

This preserves flexibility for the legitimate use cases the reporter
raises (self breast exam data, family-asserted history) without
silently permitting them. It is more aligned with FHIR's general
preference to constrain through narrative and slicing rather than to
forbid resource references that may be clinically meaningful. The
trade-off vs. Disposition A is implementation cost: every consumer of
the IG must now branch on the source actor type, which is exactly
what Disposition A avoids by ruling the cases out entirely. The
strength of B is that it leaves the door open for a future
patient-contribution profile without a breaking change.

---

### Disposition C: Decline

#### Proposal

Close the ticket without a profile or narrative change, on the basis
that base FHIR's reference targets for `Composition.author/attester`,
`Condition.asserter/recorder`, and `Observation.performer` already
include `Patient` and `RelatedPerson`, and the IG inherits that
behavior intentionally. Refer implementers to the base resource
definitions for guidance on when those reference targets are
appropriate.

#### Justification

This is the lowest-effort path and is technically defensible —
nothing in base FHIR is being violated, and the IG could argue
provider-actor narrative belongs in implementation guidance rather
than the standard itself. However, it directly contradicts the work
group's existing **Persuasive** resolution on this ticket ("we agree
that these elements are not well defined in this ballot, and intend
to define them more thoroughly in the next ballot"), so it is
included only for completeness as the "do nothing" baseline.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested).

The ticket has already been resolved **Persuasive** by the work group,
with a commitment to define these elements more thoroughly in the next
ballot — Disposition C contradicts that commitment and should be
discarded. Between A and B, Disposition A is the right starting point
for a *radiology reporting* IG: the artifact is provider-authored by
construction, and silently allowing patient/family-asserted content on
`Composition.author`, `Condition.asserter`, and the breast abnormality
`Observation.performer` creates an interoperability hazard (consumers
cannot tell whether a finding is clinician- or patient-asserted
without inspecting reference targets at runtime). Disposition B's
more permissive model is appealing in principle but it imposes an
ongoing burden on every consumer to disambiguate the source actor,
which is disproportionate for the narrow self-exam use case the
reporter cited. If the work group later decides to formalize patient
contributions to the report, that is best handled by a separate
patient-reported `Observation` profile referenced from the
`Composition` rather than by relaxing the author/attester/performer
targets on the existing profiles.
