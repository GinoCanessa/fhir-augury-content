## Summary

FHIR-25135 (“Verify Hyperlinks - BC #86”) is a change request against the
**US Breast Cancer Radiology Reporting (FHIR)** implementation guide,
flagging that a hyperlink in the IG that references the Clinical
Information Modeling and Profiling Language (CIMPL) is broken: the
underlying GitHub-hosted URL has moved. The reporter proposes updating
the link to the then-current CIMPL documentation location (for example,
`http://standardhealthrecord.org/cimpl-doc/`). More broadly, the ticket
asks the authors to verify hyperlinks in the IG and update any that
point at stale GitHub/CIMPL locations. The ticket was resolved
**Persuasive** with a commitment to validate and fix the links in the
next publication cycle. There are no linked Jira tickets, no
cross-referenced Jira items, no Zulip discussions, and no associated
GitHub items surfaced by the augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update every narrative reference to CIMPL (Clinical Information Modeling
and Profiling Language) in the US Breast Cancer Radiology Reporting IG
so that the hyperlink targets the current CIMPL documentation location
rather than the old GitHub URL. Specifically: replace the existing
anchor targets with `http://standardhealthrecord.org/cimpl-doc/`
(verified at publication time), leaving the visible anchor text
(“Clinical Information Modeling and Profiling Language (CIMPL)”)
unchanged. As a companion editorial pass, audit all other external
hyperlinks in the IG narrative pages, identify any that 404 or redirect
to unrelated content, and update them to current canonical URLs. No
conformance artifact (profile, extension, value set, code system) is
changed.

#### Justification

The request is purely editorial: a stale link in prose that reduces the
IG's usability without affecting conformance. Accepting as requested is
consistent with the reporter's intent and with the workgroup's own
resolution statement (“In the next cycle we will make sure that all
links to existing tooling will be valid.”). It is also consistent with
HL7 IG publication practice, where broken external references are
routinely corrected during the republication cycle. The change carries
no interoperability risk and no implementer burden.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than updating only the CIMPL link, remove the inline CIMPL URL
from the IG narrative and replace it with a single, stable entry in a
consolidated “References / Tooling” section at the end of the IG (or a
dedicated `references.html` page). Each external tool or document
(CIMPL, any modeling tooling, source repositories, etc.) gets one entry
there with its current URL, and in-body text refers to the entry by
name. Going forward, only the references page needs to be maintained
when upstream URLs change.

#### Justification

This reduces the long-term maintenance burden and the surface area for
link-rot: the CIMPL URL has already moved once, and external project
URLs (especially those hosted on `github.io` or organizational domains)
are prone to further moves. Centralising references also makes it
easier for reviewers and implementers to audit what tooling the IG
depends on. The trade-off versus Disposition A is slightly more initial
editorial work and a minor change in reading flow (readers now follow
an internal anchor before reaching the external URL), which is
generally acceptable in a reference-style IG.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as not actionable in the current IG version on the
grounds that (a) CIMPL itself is a tooling reference rather than a
normative part of the US Breast Cancer Radiology Reporting IG, and (b)
the IG is expected to be republished on a cycle at which time any
external URLs will be re-verified as part of routine editorial QA.
Close without a targeted edit.

#### Justification

Declining is defensible only in a narrow sense: the ticket describes a
link-rot issue that is typically handled through the normal
publication-cycle editorial pass rather than as a standalone change
request. If the workgroup's publication process already guarantees
external-link verification before each release, a dedicated disposition
may be redundant. The risk, however, is that the specific CIMPL link
(and any similarly stale links) is forgotten if there is no explicit
action item; the reporter's own evidence is that this exact link had
already gone stale at the time of reporting.

---

### Recommendation

**Recommended disposition:** A

Accept as requested. The ticket is narrow, editorial, and has already
been resolved “Persuasive” by the workgroup with an explicit commitment
to fix the links in the next cycle — Disposition A simply executes on
that commitment. The change is low-risk, requires no conformance
artifact modifications, and directly improves the IG's usability for
implementers who follow the CIMPL reference. Disposition B (centralised
references page) is attractive as a longer-term hygiene improvement but
is out of proportion to a single broken link and would expand the scope
of what was reported; it can be proposed separately if the workgroup
wants to adopt it as a general IG convention. Disposition C should not
be selected: the reporter has demonstrated that the current editorial
process did in fact miss this link, so an explicit fix is warranted.
