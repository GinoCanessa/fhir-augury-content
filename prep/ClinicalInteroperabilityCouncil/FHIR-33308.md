## Summary

The ticket reports an XPath syntax defect in one of the constraint
expressions on the `Condition` resource (the constraint commonly known as
`con-3`, which requires `clinicalStatus` to be present when the condition
is on the problem list and is not `entered-in-error`). The current XPath
references `category` without the FHIR namespace prefix, while the rest
of the same expression correctly uses the `f:` prefix
(e.g., `f:clinicalStatus`, `f:verificationStatus/f:coding/f:code/@value`).
The reporter proposes a literal fix: replace the unprefixed
`category[@value='problem-list-item']` step with the prefixed
`f:category[@value='problem-list-item']` so the whole expression evaluates
in the FHIR namespace as intended.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items in the local index for this ticket, so the request stands on its
own.

## Details

**Description:**

> XPath syntax issue (`'f:category'` instead of `'category'`)

**Resolution Description (proposed by reporter):**

> Change XPath from:
>
> `exists(f:clinicalStatus) or exists(f:verificationStatus/f:coding/f:code/@value='entered-in-error') or not(exists(category[@value='problem-list-item']))`
>
> to:
>
> `exists(f:clinicalStatus) or exists(f:verificationStatus/f:coding/f:code/@value='entered-in-error') or not(exists(f:category[@value='problem-list-item']))`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the literal change requested by the reporter. In the `Condition`
resource StructureDefinition (authored under `source/condition/` in
`HL7/fhir`), locate the invariant on `Condition` whose XPath currently
reads:

```
exists(f:clinicalStatus) or
exists(f:verificationStatus/f:coding/f:code/@value='entered-in-error') or
not(exists(category[@value='problem-list-item']))
```

and change the unprefixed `category` step to `f:category`, yielding:

```
exists(f:clinicalStatus) or
exists(f:verificationStatus/f:coding/f:code/@value='entered-in-error') or
not(exists(f:category[@value='problem-list-item']))
```

Edit only the `<differential>` element's `<constraint>`/`<xpath>` value;
do not hand-edit `<snapshot>`. Verify the fix with the publisher's
QA/invariant pass.

#### Justification

The current expression mixes prefixed and unprefixed steps, which is
internally inconsistent. FHIR XML instances place the `Condition`
element and its children in the `http://hl7.org/fhir` namespace, which
is bound to `f:` in every other constraint XPath in the resource and
across the spec. An unprefixed step in an XPath evaluated against a
namespaced document binds to the *no-namespace*, so
`exists(category[...])` will always be false against any conformant
FHIR XML instance — meaning the `not(exists(...))` branch is always
true and the constraint silently never fires for the
problem-list-item case. Adding the `f:` prefix makes the XPath match
the rest of the spec's authoring convention and restores the
constraint's intended behaviour.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the XPath as doubly broken and fix both defects in the same
technical correction:

1. Add the missing `f:` prefix on the `category` step (the reporter's
   ask), **and**
2. Fix the step itself, because `Condition.category` is a
   `CodeableConcept`, not a primitive — `f:category[@value='...']` will
   never match. The intended check is on the coded value inside the
   CodeableConcept's coding, e.g.:

   ```
   exists(f:clinicalStatus) or
   exists(f:verificationStatus/f:coding/f:code/@value='entered-in-error') or
   not(f:category/f:coding[f:system/@value='http://terminology.hl7.org/CodeSystem/condition-category' and f:code/@value='problem-list-item'])
   ```

Make the matching update to the FHIRPath form of the invariant if it
has the same defect, so XPath and FHIRPath remain semantically aligned.
Edit only the `<differential>` constraint entry under
`source/condition/` and re-run the publisher invariant pass.

#### Justification

A pure prefix fix (Disposition A) restores namespace correctness but
leaves the expression evaluating an `@value` attribute on an element
that does not have one, so the constraint still cannot fire for the
problem-list-item case. Since the ticket is filed as a Technical
Correction, the workgroup has standing to fix both layers of the same
defect in one change rather than ship a fix that is syntactically clean
but semantically still inert. The trade-off is a slightly larger diff
and the obligation to keep the FHIRPath form in sync, but it avoids a
predictable follow-up ticket.

---

### Disposition C: Decline (As Obsolete)

#### Proposal

Close the ticket as obsolete on the grounds that XPath-form invariants
have been deprecated and removed from later FHIR versions in favour of
FHIRPath-only invariants. Document in the closing comment that:

- The current authoring convention in `HL7/fhir` is FHIRPath-first;
  XPath expressions on invariants are no longer the normative form.
- Implementers validating XML use schema + FHIRPath via the validator,
  not the legacy XPath strings, so the practical impact of the defect
  is limited.
- If the FHIRPath form of the same invariant has the analogous defect,
  a new ticket should be opened scoped to FHIRPath only.

#### Justification

If the workgroup has already retired XPath as a normative invariant
form for the version line where this would otherwise be fixed, the
correct disposition is to stop maintaining the XPath rather than patch
it. Investing review and publisher cycles in correcting a
non-normative, deprecated artifact spends time the workgroup could
spend on the FHIRPath surface that implementers actually consume.

---

### Recommendation

**Recommended disposition:** B

The reporter's analysis of the prefix bug is correct, and a Technical
Correction that ships a still-broken XPath would be a poor outcome.
The `Condition.category` element is a `CodeableConcept`, so any
predicate of the form `category[@value='problem-list-item']` — prefixed
or not — cannot match a conformant instance, and the
`not(exists(...))` branch will always evaluate true. Disposition A is
mechanically safe but leaves the constraint inert; Disposition C is
defensible only if the workgroup has confirmed that XPath invariants
are no longer maintained in the target version. Disposition B fixes
both layers in one tightly-scoped edit to the `Condition`
StructureDefinition's `<differential>` constraint entry, keeps XPath
and FHIRPath semantically aligned, and is in keeping with the
"Technical Correction" classification of the ticket. The workgroup
should confirm the canonical category coding (system + code) before
merging.
