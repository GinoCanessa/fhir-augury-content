## Summary

This ballot comment (BC #50) against the US Breast Cancer Radiology
Reporting FHIR IG asks the authors to explain an apparently recursive
reference in the BreastRadiologyReport diagram, where both the start
and end nodes are labelled `BreastRadiologyReport` (distinguished only
by color and shape). The reporter suggests the ambiguity could be
resolved with a diagram key. The ticket has already been triaged with
a resolution of **Persuasive**: the work group accepted the comment
and committed to improving the diagramming and documentation to
clarify how prior Breast Radiology Reports can be children of a new
Breast Radiology Report. No cross-referenced Jira tickets, Zulip
discussions, or GitHub items were found, so this prep is based on the
ticket content alone.

## Proposed Dispositions

> Note: The ticket already carries a resolution of **Persuasive** with
> a commitment to improve diagramming and documentation. The
> dispositions below are framed as concrete options the work group
> can confirm or revisit when implementing that resolution.

### Disposition A: Accept As Requested

#### Proposal

Update the BreastRadiologyReport diagram in the US Breast Cancer
Radiology Reporting FHIR IG to make the self-reference unambiguous by:

1. Adding an explicit **legend / key** to the diagram that calls out
   what the differing colors and shapes mean (e.g., "filled box =
   current report; outlined box = prior report being referenced").
2. Labelling the two `BreastRadiologyReport` nodes with role-based
   names (e.g., `BreastRadiologyReport (current)` and
   `BreastRadiologyReport (prior)`) rather than relying solely on
   visual styling.
3. Adding narrative text adjacent to the diagram explaining that
   `BreastRadiologyReport` may reference one or more prior
   `BreastRadiologyReport` instances (i.e., the recursion models the
   "prior report" relationship, not a true cycle), and naming the
   element / reference path that carries that link.

#### Justification

The reporter explicitly suggested a key would resolve the confusion,
and the work group has already agreed the change is persuasive. A
legend plus role-qualified node labels is the lowest-risk way to
preserve the existing model while removing the perceived circularity.
This is a documentation-only change, so it has no impact on
conformance, instances, or downstream implementers — it only improves
how the existing structure is explained. It is also consistent with
how other FHIR IGs disambiguate self-referential resources (e.g.,
`Observation.hasMember`, `List.entry.item`, `Composition.relatesTo`).

---

### Disposition B: Alternative Approach

#### Proposal

Replace the recursive node in the diagram with an explicit
**"Prior BreastRadiologyReport"** abstraction — visually rendered as
a stack/aggregate of report icons with a `0..*` cardinality back to
the current report — and move the formal description of the
self-reference into a dedicated section that documents:

- The element/path carrying the prior-report reference (e.g., a
  reference to `Composition` / `DiagnosticReport` representing prior
  studies).
- Cardinality and Must Support flags.
- Expected behavior when the chain is long (e.g., do implementers
  walk all prior reports, or only the immediate predecessor?).

Combine this with the legend from Disposition A, but treat the
narrative section — not the diagram — as the normative explanation.

#### Justification

The original diagram conveys *that* there is a self-reference but not
*why* or *how deep*. A reviewer's first instinct ("this is a cycle")
is a sign that the diagram is doing work that prose would do better.
Promoting the explanation to a dedicated narrative section gives
implementers a precise contract (path, cardinality, traversal
expectations) and reduces the chance that future readers raise the
same comment again. The trade-off vs. Disposition A is more authoring
effort and a larger diff against the IG, but the result is more
durable and more useful to implementers.

---

### Disposition C: Decline

#### Proposal

Decline to change the diagram on the grounds that the self-reference
is a faithful depiction of the underlying model and that the
differing colors and shapes already distinguish the two roles.

#### Justification

This disposition is included for completeness only and is **not
defensible** here: the work group has already agreed the comment is
persuasive, and a single reviewer flagging the diagram as ambiguous
is direct evidence that the current visual encoding is insufficient.
Declining would also leave the underlying confusion (current vs.
prior report) unaddressed for future readers. The only reason to
record this option is to make explicit that "do nothing" was
considered and rejected.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly delivers what the reporter asked for (a key /
clarification), aligns with the work group's already-recorded
"Persuasive" resolution, and is a documentation-only change with no
conformance impact. Disposition B is a stronger long-term answer but
is meaningfully more work and risks scope creep beyond a single
ballot comment; the work group can pursue B opportunistically (or in
a future ballot cycle) if reviewers continue to find the
self-reference confusing after the legend is added. Disposition C is
inconsistent with the existing resolution and should not be adopted.
