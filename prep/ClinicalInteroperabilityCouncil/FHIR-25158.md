## Summary

FHIR-25158 is ballot comment "BC #109" against the **US Breast Cancer
Radiology Reporting (FHIR)** implementation guide. The reporter (Susan
Matney) asks whether the value set used in this IG to convey
"malignancy interpretation" needs to be breast-cancer-specific or
whether it could be defined more generically so that it would be
reusable for malignancy interpretation across other cancer / radiology
reporting domains. The ticket carries the `Discuss-at-WGM` label, has
been triaged, and is currently `Unresolved` with the resolution-
description text "Will discuss" — i.e., the work group has agreed it is
a real question worth deciding but has not yet picked a direction. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the cross-reference index for this key, so the request
stands on its own.

## Proposed Dispositions

> Note: the ticket text does not name the specific value set under
> review. The dispositions below are framed against "the value set the
> IG uses to convey malignancy interpretation" generically; before
> formally voting, the work group will need to confirm which value set
> in the US Breast Cancer Radiology Reporting IG is being referenced
> (most likely the one bound to the malignancy-interpretation
> Observation profile).

### Disposition A: Accept As Requested

#### Proposal

Generalize the value set so that its name, scope, and intent describe
"malignancy interpretation" across solid-tumor / radiology contexts
rather than breast cancer specifically. Concretely:

1. Rename the existing value set (e.g., `MalignancyInterpretationVS`),
   move its canonical URL out of the breast-cancer-specific namespace,
   and reauthor its description as oncology-general.
2. Re-evaluate every code currently in the value set; remove codes that
   are demonstrably breast-cancer-only and add the general malignancy-
   interpretation codes (e.g., the SNOMED CT `Malignant`, `Benign`,
   `Indeterminate`, `Suspicious for malignancy`, `In situ` family) that
   are already in routine radiology use.
3. Keep the binding on the breast-cancer Observation profile pointing
   at the new generalized value set (with the same binding strength),
   so the IG continues to validate.
4. Either retire the original breast-specific value set as deprecated
   with a pointer to the generalized successor, or retain it as a
   subset bound elsewhere if any breast-only codes are required.

#### Justification

This is what the reporter literally asks for, and it follows the FHIR
terminology design principle of defining value sets at the broadest
clinically coherent scope so they can be reused. Malignancy
interpretation as a concept is not breast-specific — the same codes
appear in pathology, radiology, and oncology reporting across tumor
sites — so a generic value set reduces duplication and makes the IG's
binding immediately reusable by adjacent IGs (e.g., other organ-system
radiology reporting guides). The Discuss-at-WGM tag and the "Will
discuss" resolution suggest the work group is already inclined to
revisit scope rather than defend the breast-specific framing.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the breast-cancer-specific value set in the IG, but factor it as
an **extension of a generic base**:

1. Author (or adopt, if one already exists in a CDS / oncology IG) a
   generic `MalignancyInterpretation` value set at a neutral canonical
   URL, owned at a level above this IG.
2. Define the breast-cancer-specific value set as a subset / extension
   of that base (intensional definition that includes the base value
   set plus any breast-specific codes the IG actually needs).
3. Bind the IG's breast-cancer Observation profile to the breast-
   specific subset, so the IG retains its tight, validation-friendly
   binding.
4. Document the layering in the IG so other IGs can pick up the
   generic base without inheriting breast-specific codes they do not
   need.

#### Justification

This preserves the IG's existing implementer contract (no breaking
change to the bound value set on the breast-cancer profile) while
still answering the reporter's underlying need: that "malignancy
interpretation" as a concept be available in a reusable form. It is
the standard FHIR pattern for "shared-but-specialized" terminology and
avoids the tension between "broad enough to reuse" and "tight enough
to validate one IG cleanly". It is also lower-risk for downstream
implementers who have already conformed to the existing binding.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the value set scoped to breast-cancer
radiology reporting as published, with a rationale of "out of scope
for this IG; generalization belongs to a cross-cutting oncology
terminology effort, not to a single-domain radiology IG."

#### Justification

The US Breast Cancer Radiology Reporting IG was scoped specifically to
breast cancer; broadening one of its value sets implicitly turns this
IG into the steward of a cross-cutting oncology terminology, which is
beyond its charter. If a generic malignancy-interpretation value set
is needed, the right place to author it is an oncology / pathology /
clinical-genomics terminology workstream where multiple cancer
domains can negotiate the contents, not as a side effect of a breast-
cancer ballot comment. Declining here does not block the broader
work; it just keeps responsibility where it belongs.

---

### Recommendation

**Recommended disposition:** B

The reporter is making a real terminology-design point — malignancy
interpretation is not a breast-cancer-specific concept and ought to be
reusable — so straight decline (C) under-serves the request and leaves
the same question to be re-asked the next time a cancer-domain IG
needs to express the same idea. Pure acceptance (A) is the most
satisfying answer in principle, but it pushes a single IG into
publishing terminology that other domains will have to live with, and
it risks a breaking change to anyone already conforming to the
existing breast-specific binding. Disposition B threads that needle:
it gives the community a generic, reusable
`MalignancyInterpretation` value set at a neutral home, keeps the IG's
own profile binding stable for current implementers, and creates an
explicit place for cross-domain codes to accumulate without enlarging
this IG's scope. Confirm at WGM which value set in the IG the
reporter is pointing at, then proceed with the layered base + subset
authoring.
