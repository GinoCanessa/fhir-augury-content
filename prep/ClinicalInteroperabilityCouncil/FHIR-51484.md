## Summary

The reporter notes that, during US-Core cross-version analysis for Da Vinci
HRex, the working group missed the fact that the cardinality of the NPI and
CLIA identifiers changed across US Core versions: they are `0..1` in
US Core 3.1.1 but were relaxed to `0..*` in US Core 6.1.0. Because HRex
profiles attempt to align with multiple US Core versions, this divergence
makes it impossible for a single HRex profile constraint to be conformant
with both base versions simultaneously — anything that asserts `0..*` will
fail validation against US Core 3.1.1, and anything that asserts `0..1`
contradicts US Core 6.1.0. The ticket asks the work group to choose among
three resolution paths: (1) tighten HRex to `0..1` (US Core 3.1.1 behavior),
(2) confirm an errata exists for US Core 3.1.1 that effectively allows
additional repetitions and document/ignore the resulting validation
warning, or (3) split the affected HRex profile into version-specific
variants (one for US Core 3.1.1, one for 6.1.0+) and reference both
profiles wherever the original is used. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced for this ticket.

## Details

**Description:**

In our US-Core cross-version analysis, we missed the fact that the
cardinality of the NPI and CLIA identifiers were 0..1 in US Core 3.1.1,
but were relaxed to 0..* in US Core 6.1.0.

We need to decide whether to:

1. impose the 0..1 constraint in Da Vinci
2. find out whether there is an errata that allows additional repetitions
   in 3.1.1 and if so, just note it and ignore the error that says we're
   not complying with 3.1.1
3. Split the profile into two — one for US Core 3.1.1 and one for 6.1.0+
   and reference both profiles everywhere

## Proposed Dispositions

> Note: With no GitHub cross-references and no persisted briefing
> consulted, the file-path specifics below are described at the IG /
> profile level rather than at the source-file level. A reviewer should
> confirm the exact StructureDefinition file in the HRex IG (e.g., the
> HRex Practitioner / Organization profile, or the slice on the
> identifier element where the NPI/CLIA `system` is constrained) before
> implementation.

### Disposition A: Accept As Requested — Constrain to 0..1

#### Proposal

In the HRex IG, tighten the cardinality on the NPI and CLIA identifier
slices (the `Practitioner.identifier` slice constrained to
`http://hl7.org/fhir/sis/us-npi` and the `Organization.identifier` slice
constrained to `urn:oid:2.16.840.1.113883.4.7`) from `0..*` to `0..1`.
Apply the same change anywhere HRex re-asserts these slices. Document in
the IG that this is a deliberate intersection of the US Core 3.1.1 and
US Core 6.1.0 cardinalities so that HRex instances are valid against
both base versions.

#### Justification

Profiles legitimately further constrain (narrow) cardinality, and `0..1`
is a strict subset of `0..*`, so this remains conformant to US Core
6.1.0 while also satisfying US Core 3.1.1. It produces a single profile
(easier for implementers, no profile-selection logic required upstream)
and removes the validation-error / warning path entirely. The cost is
forbidding the (rare in practice) case of a Practitioner or
Organization reporting more than one NPI or more than one CLIA — a
constraint that matches real-world identifier semantics for these
specific identifier systems. This is the simplest, most consistent
resolution and avoids duplicating profiles across the IG.

---

### Disposition B: Alternative Approach — Split into Version-Specific Profiles

#### Proposal

Create two parallel HRex profiles for each affected resource: one
derived from US Core 3.1.1 (with NPI/CLIA cardinality `0..1`) and one
derived from US Core 6.1.0+ (with NPI/CLIA cardinality `0..*`). Update
every HRex artifact that currently references the single profile to
instead reference both via `targetProfile` choices (or via separate
named profiles per use case), and add IG narrative explaining when each
applies. Optionally keep an "umbrella" abstract profile that both
extend, for readability.

#### Justification

This is the literal "honor the source profile" approach and preserves
the full cardinality of US Core 6.1.0 for systems on that version,
while not falsely asserting compatibility with 3.1.1 for repeating
identifiers. It is the most semantically honest approach across
versions. The trade-off is significant: every reference site in HRex
must be updated to accept both profiles, downstream Da Vinci IGs that
reference HRex must do likewise, and implementers gain a profile
selection problem they did not have before. Given that HRex's value is
partly to provide a single, common profile across exchange partners,
this option is heavyweight and tends to push complexity onto every
consuming IG.

---

### Disposition C: Decline / Document and Ignore (option 2 from the ticket)

#### Proposal

Decline to change the cardinality in HRex. Instead, confirm whether US
Core 3.1.1 has published an errata that effectively permits additional
repetitions of the NPI/CLIA slices; if such an errata exists, add a
note to the HRex IG describing the situation, explicitly acknowledge
the cross-version validation warning against unmodified 3.1.1, and
suppress / ignore that specific QA error. If no such errata exists,
still document the deliberate divergence from US Core 3.1.1 and treat
the warning as expected.

#### Justification

This is defensible only if (a) the working group is confident that
US Core 3.1.1 implementations in the wild already tolerate repeating
NPI/CLIA, or (b) HRex consumers are effectively all on US Core 6.1.0+.
It avoids touching profile structure or splitting artifacts, but it
leaves a known validation issue in place and pushes interpretation of
the warning onto every implementer and tester. It is the option with
the least change but the highest ongoing support cost, and it depends
on an errata that has not been confirmed in this ticket.

---

### Recommendation

**Recommended disposition:** A

Disposition A (constrain to `0..1`) is the cleanest resolution and the
one most consistent with how FHIR profiling is normally used to bridge
versions: profiles narrow, they don't broaden. It keeps HRex as a
single, predictable profile that is conformant against both US Core
3.1.1 and US Core 6.1.0+, removes the cross-version validation
problem outright, and aligns with the real-world semantics of NPI and
CLIA (an entity has one of each). Disposition B is the textbook
"correct" answer but imposes profile-selection complexity on every
HRex-consuming IG, which is disproportionate to the underlying
problem. Disposition C is only viable if a 3.1.1 errata is confirmed,
and even then it institutionalises a known QA warning rather than
fixing it. The work group should adopt A unless it has concrete use
cases for multiple NPI or CLIA values per Practitioner/Organization
that would be lost.
