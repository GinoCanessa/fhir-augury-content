## Summary

The reporter (Eric Haas) asks the Da Vinci HRex IG to clarify what its
section on modifier elements actually requires of senders and receivers,
particularly with respect to elements that are mandatory in the base FHIR
spec but are *not* flagged Must Support in HRex profiles. The ticket
proposes adding explicit conformance language that tells receivers they
**must** detect unexpected modifier extensions / modifier elements and
either reject the instance or demonstrate that they can safely process it
(allowing limited tolerance when the modifier is nested inside content
the receiver does not care about, or when the instance is for human
review only — otherwise rejection is the only safe action). The reporter
points to the US Core 6.1.0 "Modifier Elements" guidance
(https://hl7.org/fhir/us/core/general-requirements.html#modifier-elements)
as the model wording HRex should adopt or adapt. No related Jira tickets,
Zulip threads, or GitHub items were surfaced through the cross-reference
index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new "Modifier Elements" subsection to the HRex IG's general
guidance / conformance pages that mirrors US Core 6.1.0's
[Modifier Elements](https://hl7.org/fhir/us/core/general-requirements.html#modifier-elements)
text, adapted to HRex's exchange context. The new text would state, in
normative-style language, that:

1. Senders **SHALL NOT** include modifier extensions or `isModifier`
   elements that are not declared in the relevant HRex profile unless
   they are confident the receiver supports them.
2. Receivers **SHALL** inspect every received instance for unknown
   modifier extensions and `isModifier` elements (recursively, within
   contained / nested resources).
3. If an unknown modifier is encountered, the receiver **SHALL** either
   (a) reject the instance with an `OperationOutcome` of
   `severity=error`, `code=not-supported`, citing the offending
   element's path, or (b) demonstrate that the modifier can be safely
   ignored — limited to the cases described in US Core (modifier nested
   inside content the receiver does not consume, or instance intended
   for human review only).
4. Cross-link the new section from the HRex "Must Support" page so the
   distinction between Must Support, mandatory, and modifier elements is
   clear.

Because no GitHub repository for HRex is currently surfaced in the
augury cross-references, the actual file path (e.g., the IG's
`input/pagecontent/general-guidance.md` or equivalent) should be
confirmed with the publisher repo (`HL7/davinci-ehrx`) before the
workgroup commits the change.

#### Justification

Aligning HRex with US Core 6.1.0 keeps the two most widely-implemented
US realm IGs consistent on a topic where divergent guidance is actively
harmful to implementers — many systems implement HRex *and* US Core in
the same engine. The US Core text has been through ballot, has
established community understanding, and gives the workgroup a
well-tested template, so the editorial cost is low. The label
`3.1.1Maintenance` indicates the change is already scoped to a
maintenance release, which is the right vehicle for clarifying (not
changing) conformance expectations.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than copying the US Core text verbatim into HRex, **reference**
the FHIR core "[Modifier Elements](https://hl7.org/fhir/R4/extensibility.html#modifierExtension)"
rules and the FHIR R4 [Conformance rules for `isModifier`](https://hl7.org/fhir/R4/conformance-rules.html)
as the normative source, and add a short HRex-specific paragraph
covering only what is unique to HRex's exchange model:

- Clarify that, in HRex exchanges, "non-Must-Support mandatory
  elements" are still required to appear in instances (because base
  cardinality `1..*` is unaffected by Must Support flags) but receivers
  are not required to *use* them — and modifiers attached to such
  elements still trigger the standard reject-or-safely-ignore rule.
- Add a single normative SHALL aimed at receivers, pointing back to
  base-spec rules instead of restating them.

#### Justification

This avoids text drift between HRex, US Core, and base FHIR — three
sources of subtly different wording on the same topic is exactly what
created the confusion the ticket is trying to fix. It also keeps HRex
slim (the IG's value is exchange semantics, not re-explaining base
conformance). The trade-off vs. Disposition A is reader convenience:
implementers reading HRex in isolation will need to follow links to
base FHIR / US Core, which the workgroup may judge unacceptable given
HRex's audience.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — already addressed by base FHIR." Rationale:
the base FHIR specification's
[`modifierExtension` rules](https://hl7.org/fhir/R4/extensibility.html#modifierExtension)
already require receivers to either understand or reject unknown
modifier extensions; HRex inherits these rules without restating them.
Restating base-spec normative content in every IG creates maintenance
burden and risks divergence. If the reporter believes specific HRex
profiles are ambiguous, ask for follow-up tickets naming the offending
sections.

#### Justification

The base FHIR rule is unambiguous on its face, and the workgroup has
historically resisted restating base conformance in derived IGs. Several
related Zulip threads (e.g., "Why do IGs redefine search parameters
from the base spec?") show recurring community pushback against IGs
duplicating base normative text. Declining is defensible if the
workgroup judges the existing base text adequate.

---

### Recommendation

**Recommended disposition:** A

US Core has already proven that implementers do *not* find the base
FHIR `modifierExtension` text sufficient in practice — that is precisely
why US Core 6.1.0 added the "Modifier Elements" section the reporter
cites. HRex sits in the same regulatory and implementation neighborhood
as US Core and is consumed by the same engines, payers, and providers,
so leaving HRex silent (Disposition C) effectively forces implementers
to guess whether HRex inherits US Core's interpretation or only base
FHIR's. Disposition B's "link, don't duplicate" instinct is principled
but in this specific case the duplication has been deliberately chosen
by the larger US Core community, and matching that choice avoids
creating a third, slightly different rendering. Adopting the US Core
language verbatim (with attribution) is the lowest-risk way to give
HRex implementers actionable, consistent guidance, and it fits cleanly
into the `3.1.1Maintenance` scope already attached to the ticket. The
workgroup should ask the assignee to coordinate with the US Core editors
to keep the wording aligned across future revisions.
