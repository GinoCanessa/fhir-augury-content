## Summary

Lloyd McKenzie has filed a cross-cutting documentation request against the
Da Vinci HRex implementation guide (and, by extension, the broader family of
Da Vinci guides). Most Da Vinci guides reuse FHIR data structures whose
element definitions describe what each element means, but those definitions
are written as descriptive prose rather than `SHALL` conformance statements.
The reporter argues there is nonetheless an implicit expectation that
implementers must populate those elements with data that actually matches
the stated definitions, and that this implicit expectation should be made
explicit in the guide(s). The request also asks the guide(s) to clarify that
different actors (for example, data producers vs. consumers vs.
intermediaries) carry different levels of responsibility for honouring those
definitions. No cross-referenced Jira tickets, Zulip threads, or GitHub
items were surfaced for this ticket. The ticket has already been triaged on
the Apr. 15 BR call with a proposed resolution of **Not Persuasive** — the
work group's position being that no additional language is needed to state
these implicit expectations.

## Details

**Description:**

> Most Da Vinci guides leverage data structures that define data elements.
> These definitions typically aren't framed as 'SHALL' statements, however
> there's an implicit expectation that systems need to populate the data
> structures with data that matches the definitions. We need to make that
> implicit expectation explicit - and to make clear that there are different
> levels of responsibility here.

**Resolution (proposed):** Not Persuasive — "We do not feel the need to add
language that states these sort of implicit statements."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative "Conformance" subsection to the HRex IG (and propagate the
pattern to other Da Vinci guides) that explicitly states: where this guide
references a profile or data structure, conforming systems **SHALL**
populate any element they include with data that matches that element's
definition (cardinality, type, value-set binding, and prose semantics).
Pair this with an actor-responsibility table distinguishing the obligations
of (a) data producers (must ensure populated values match the definition),
(b) data consumers (may rely on populated values matching the definition,
but should still validate), and (c) intermediaries (must not silently
transform values in a way that violates the definition). The new section
would live in HRex's "Conformance" or "General Guidance" page.

#### Justification

This makes a real, frequently-encountered ambiguity explicit. Implementers
testing against Da Vinci guides regularly disagree about whether a stated
element definition imposes any obligation when the surrounding profile is
silent on cardinality or invariants. Spelling out the implicit expectation
removes that ambiguity and aligns Da Vinci with how other HL7 work groups
(e.g., US Core's "Must Support" and conformance verbs guidance) have chosen
to handle the same problem. A dedicated actor-responsibility table also
helps testers and certification programs target their assertions correctly.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding new normative `SHALL` text in HRex, add a short
non-normative "Interpreting Element Definitions" subsection to HRex's
general guidance that (i) reminds readers that FHIR's base specification
already imposes structural conformance (data must conform to the element's
declared type, cardinality, and bindings) and (ii) points to the existing
FHIR conformance and "Must Support" pages. Combine this with a single
sentence reminding implementers that producers, consumers, and
intermediaries each carry the obligations defined in their respective HRex
actor sections. No new SHALL statements are introduced.

#### Justification

This addresses the underlying confusion the reporter raises without
manufacturing new normative content that overlaps with what the FHIR core
spec already says. FHIR R4/R5 already make element-definition conformance
binding via the spec's own conformance rules; restating it as a Da Vinci
SHALL risks creating redundant — and potentially divergent — normative
text. A pointer-style clarification keeps HRex aligned with core FHIR while
still making the expectation visible to readers who skim the IG without
reading the base spec.

---

### Disposition C: Decline (matches the proposed resolution)

#### Proposal

Resolve the ticket as **Not Persuasive** with no spec change. The HRex IG
already inherits FHIR's conformance machinery: any profile or data
structure it references is governed by FHIR's existing rules about element
definitions, cardinality, types, bindings, and invariants. Actor-specific
responsibilities are already enumerated in the HRex actor and capability
statement sections. No new prose is added, and the implicit expectations
remain implicit — the same as in every other FHIR-based guide.

#### Justification

This is consistent with the work group's existing position (recorded in
the Apr. 15 BR call) that the requested language would restate
expectations that are already enforced by the FHIR base specification.
Adding Da Vinci-specific SHALL statements about element definitions would
set a precedent that every IG must restate base-spec conformance rules,
which inflates the spec without changing real-world behaviour. Furthermore,
the ticket does not cite a concrete interoperability failure attributable
to the missing language, and there is no Zulip or GitHub discussion in the
augury index suggesting an active implementer pain point that this change
would resolve.

---

### Recommendation

**Recommended disposition:** C

The ticket as filed describes a real ambiguity but proposes a remedy
(explicit SHALL-style language plus an actor-responsibility statement)
whose substance is already covered by the FHIR base specification's
conformance rules and HRex's existing actor sections. There is no
cross-referenced Jira ticket, Zulip thread, or GitHub item in the augury
index pointing to a concrete implementation problem caused by the absence
of this language, which weakens the case for normative change. Disposition
C also matches the proposed disposition the BR call already accepted on
Apr. 15, so adopting it preserves continuity. If the work group later sees
evidence that implementers are genuinely treating element definitions as
optional guidance, Disposition B (a non-normative clarification pointing at
the FHIR base spec) is a low-cost fallback that avoids the over-reach
introduced by Disposition A.
