## Summary

The ticket flags a noun/verb cohesion problem in the US Breast Cancer
Radiology Reporting (FHIR) implementation guide: the element
`breastrad-PriorReports` is named in the plural but is currently constrained
to a single value (cardinality `0..1`). The reporter asks whether the
cardinality should be widened to `0..*` so the element name and its allowed
multiplicity match. The ticket has already been resolved as **Persuasive**,
with the work group agreeing that `breastrad-PriorReports` should be
`0..*`, to be applied in the next ballot. No cross-referenced Jira tickets
were returned for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the cardinality of `breastrad-PriorReports` in the US Breast Cancer
Radiology Reporting (FHIR) profile from `0..1` to `0..*`, leaving the
element name unchanged. Republish the affected profile (and the
corresponding StructureDefinition / snapshot/differential) in the next
ballot of the IG so the plural element name correctly reflects that
multiple prior reports can be referenced.

#### Justification

This is exactly what the resolution text already endorses ("We agree.
breastrad-PriorReports should have cardinality is 0\*. This change will be
made in the next ballot."). FHIR naming guidance is that plural element
names imply repeatable elements; aligning the cardinality with the name
removes a real semantic mismatch and matches the clinical reality that a
breast imaging report frequently references more than one prior report.
Widening the upper bound from `1` to `*` is a non-breaking change for any
existing instances (a single value still validates), so the upgrade path
for implementers is clean.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the underlying cardinality at `0..1` but rename the element to the
singular form (e.g., `breastrad-PriorReport`) so the name matches the
existing constraint. Update any bindings, invariants, search parameters,
and example instances to use the singular name, and document the rename
in the IG change log.

#### Justification

If the original design intent was deliberately to constrain a breast
imaging report to a *single* prior-report reference (for example, "the
most recent prior"), then renaming preserves that constraint and still
removes the noun/verb mismatch that the ticket flags. The trade-off is
significant: renaming is a breaking change for any implementer who has
already built against `breastrad-PriorReports`, whereas Disposition A is
additive. This option is only defensible if the work group can confirm
the singular cardinality was intentional.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave both the name (`breastrad-PriorReports`,
plural) and the cardinality (`0..1`) as-is, on the grounds that the
mismatch is purely cosmetic and existing implementations are already in
the field.

#### Justification

This is the weakest option. The work group's own resolution rejects it
("We agree…"), there is a real semantic mismatch (plural name vs.
single-value constraint), and broadening from `0..1` to `0..*` is
backward-compatible. There is no published reason to keep the constraint
narrow, so a "decline" position would be hard to defend to implementers
who have already noticed the inconsistency.

---

### Recommendation

**Recommended disposition:** A

The ticket is already resolved Persuasive with an explicit commitment to
widen `breastrad-PriorReports` to `0..*` in the next ballot, and the
change is non-breaking for existing instances. Disposition A delivers
exactly that with minimum disruption to implementers. Disposition B
(rename to singular) would force a breaking change on every consumer of
the profile and is only worth pursuing if the work group has concrete
evidence that the original `0..1` cardinality was deliberate — nothing in
this ticket suggests that. Disposition C contradicts the work group's
existing resolution and should not be entertained.
