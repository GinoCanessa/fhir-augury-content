## Summary

The reporter (Ron G. Parker) asks the mCODE editors to consider anchoring
the value set referenced in this ticket — and, by extension, value sets
in mCODE that have utility beyond the IG itself — in the HL7 Terminology
server (THO, `terminology.hl7.org`) rather than publishing them inside
the mCODE IG. The rationale is that THO-anchored value sets are
discoverable and reusable by other specifications and implementers, so a
mCODE-defined value set with cross-IG utility would benefit a wider
audience if it were published in THO. The ticket was originally voted
"Considered for Future Use" on 2024-06-27 (Jimmy Tcheng / Caroline
Potteiger, 5-0-4) on the basis that the most broadly used FHIR IGs
(e.g., US Core) keep their value sets in the IG or in core FHIR and
mCODE wished to stay consistent. It was then re-opened post-STU4.0.0
publication by Lynn Laakso for fresh consideration. No cross-referenced
Jira tickets, Zulip threads, or GitHub artifacts were found in the
indexed sources.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Identify the specific mCODE value set called out by Comment 6 (and any
sibling value sets the CIC editors agree have cross-IG utility, e.g.
the cancer-staging, cancer-disease-status, and TNM category /
modifier value sets that are commonly cited as candidates) and migrate
their canonical home from the mCODE IG to THO. Concretely, for each
selected value set:

- Submit a UTG / THO proposal to publish the value set under
  `http://terminology.hl7.org/ValueSet/<name>` (and, where the codes
  themselves are mCODE-defined, the underlying CodeSystem under
  `http://terminology.hl7.org/CodeSystem/<name>`).
- Update the mCODE IG sources so the affected profiles bind to the
  THO canonical URL instead of the IG-local URL, and remove the
  IG-local ValueSet/CodeSystem definitions (or retain them only as
  retired aliases pointing to the THO canonical).
- Add a migration note to the mCODE IG history / change log calling
  out that the canonical URLs have moved, since this is a
  binding-canonical change implementers will need to absorb.

#### Justification

Anchoring in THO is the model HL7 endorses for terminology with reuse
potential outside a single IG: it gives the value set a stable,
authority-managed canonical URL, enables independent versioning, and
makes it discoverable to other IGs without forcing them to take a
dependency on the mCODE IG package. The original 2024-06-27 vote did
not dispute the technical merit of THO publication; it relied on a
consistency argument with US Core. That consistency rationale is the
weakest part of the prior reasoning — US Core's choices were driven by
its own constraints (largely US-Realm-locked, regulatory-cycle-bound
content) and don't necessarily transfer to oncology terminology that
the wider community (CodeX, ICAREdata, registries, payer projects) is
already reusing.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a **selective / criteria-based** policy rather than a blanket
move. Specifically:

1. CIC ratifies a short rule for mCODE value sets: a value set is
   anchored in THO when (a) at least one non-mCODE artifact (HL7 IG,
   CodeX IG, external project) is known to reuse it, **or** (b) the
   value set is composed entirely of codes from external code systems
   (LOINC, SNOMED CT, ICD-O, etc.) and contains no mCODE-authored
   concepts. All other value sets remain in the mCODE IG.
2. For value sets that meet the criteria, file UTG proposals to move
   them to THO and rebind the mCODE profiles, as in Disposition A.
3. For value sets that don't (yet) meet the criteria, leave them
   IG-anchored but document the policy in the mCODE IG so future
   editors and reusers know the rule and can promote a value set when
   reuse is demonstrated.

#### Justification

This avoids the consistency objection raised in the original vote (we
are not unilaterally diverging from the US Core pattern for everything
mCODE publishes) while still unlocking the specific reuse benefit the
reporter is asking for. It is also operationally cheaper than a
wholesale migration: each THO submission requires a UTG ticket, a
Vocabulary WG review, and coordinated rebinding in the IG, so scoping
the move to value sets with demonstrated reuse keeps editorial cost
proportional to the benefit. The written policy also gives future
ballot reconcilers a defensible answer to similar comments without
having to re-litigate the principle each time.

---

### Disposition C: Decline

#### Proposal

Persisted Not Persuasive (or Considered for Future Use, again).
Maintain the current arrangement in which mCODE value sets are
published inside the mCODE IG, with the canonical URL on the mCODE
canonical base. Document in the response that mCODE follows the same
pattern as US Core and the majority of HL7 FHIR IGs and that
implementers needing the value set should consume it from the mCODE
IG package.

#### Justification

This is the position the workgroup already arrived at on 2024-06-27
(5-0-4). The justification then — internal consistency with the way
the most broadly used HL7 FHIR IGs publish their terminology, and the
operational simplicity of keeping value-set authoring co-located with
the profiles that bind to them — has not changed. Moving canonical
URLs is also a non-trivial breaking change for downstream
implementers: every binding URL changes, IG package consumers may need
to re-resolve dependencies, and validators tied to specific canonicals
need to be reconfigured. Without a concrete external consumer named in
the ticket, the cost-to-benefit ratio favours leaving the status quo.

---

### Recommendation

**Recommended disposition:** B

The original "consistency with US Core" argument is real but not
strong enough on its own to override a request that has clear
implementer benefit, and the reopening by the publication facilitator
signals the workgroup is willing to revisit. A blanket "move
everything to THO" (Disposition A) is overreach for what was filed
as a single ballot comment and would create a lot of UTG / rebinding
work without first scoping which value sets actually justify it. A
flat decline (Disposition C) just re-runs the 2024 vote and leaves
the underlying reuse problem in place, which is why the ticket was
re-opened. A criteria-based policy (Disposition B) lets CIC say *yes*
to the reporter's underlying ask — make genuinely reusable mCODE
value sets discoverable in THO — while preserving the consistency
argument for the value sets that really are mCODE-internal. It also
hands future reconcilers a written rule to apply, which is the durable
fix.
