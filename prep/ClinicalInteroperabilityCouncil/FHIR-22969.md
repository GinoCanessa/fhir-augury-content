## Summary

The ticket reports a documentation defect in the FHIR Core specification:
the Visio diagram illustrating the "Solicited Communication (With Task)"
workflow is missing its second page. The first page contains a "Go to A"
connector pointing to a continuation labeled "A" on a second page, but
that second page is absent from the published diagram, leaving the
workflow visually incomplete and ambiguous for implementers trying to
follow the solicited request flow that uses Task. The reporter
(Lisa R. Nelson) is asking the work group that owns the relevant
content (now CIC, which owns CDex / the solicited communication
guidance) to publish the missing page so the diagram can be read end to
end. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items to widen the picture.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a confirmed documentation defect and republish the
"Solicited Communication (With Task)" workflow diagram with both pages
present. Concretely: locate the source Visio (and any derived SVG/PNG)
in the FHIR Core authoring tree under the Communication workflow page
(the diagram referenced from the Communication resource's workflow
section), restore the second page that contains the "A" connector
target, and regenerate the embedded image artifacts so the published
HTML shows a complete, two-page flow with the "Go to A" link resolving
to a visible target. No resource, profile, or constraint changes are
needed — this is a content/asset fix only.

#### Justification

The ticket describes a clear, narrow, verifiable defect: a connector
labelled "Go to A" with no corresponding "A" target is, on its face,
broken documentation. Restoring the missing page is consistent with
FHIR's general practice of treating diagrams as normative-adjacent
explanatory material that must be self-consistent, and it is the
minimum change that fulfills the reporter's request. There is no
community pushback recorded against the fix itself.

---

### Disposition B: Verify-Then-Close (Alternative Approach)

#### Proposal

Before changing anything, confirm whether the diagram has already been
repaired by intervening edits — as suggested by the May 2025 comment
from OO ("OO thinks the subsequent edits may have fixed this issue").
The reviewer (or assignee) should open the current published
Communication workflow page in the in-progress build, inspect the
"Solicited Communication (With Task)" diagram, and verify that every
"Go to X" connector resolves to a labelled target. If the diagram is
now complete, close the ticket as **Persuasive – Already Resolved** with
a comment pointing at the commit/PR that fixed it. If a gap remains,
fall back to Disposition A and patch only the still-missing portion.

#### Justification

This avoids spending authoring effort re-fixing something that may
already be correct, and produces a defensible audit trail for closing a
six-year-old ticket. It also respects the OO comment on the record
rather than overriding it. The trade-off vs. Disposition A is one extra
verification step, which is cheap relative to round-tripping a Visio
edit and regenerating image assets.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that, in the six
years since it was filed, the underlying diagram and surrounding
guidance have been substantially reworked, and no implementer has
re-raised the concern. Treat the original "missing page" as superseded
by the current state of the workflow documentation and CDex guidance.

#### Justification

The ticket has been open since 2019 with only routing/triage comments,
no implementer follow-up, and an OO note suggesting the issue is moot.
Spending work-group cycles on a stale documentation cosmetic when no
one is currently blocked is hard to justify against the rest of the
backlog.

---

### Recommendation

**Recommended disposition:** B

The May 2025 OO comment ("subsequent edits may have fixed this issue")
is the single most important new fact on this ticket and it points
directly at a verify-then-act path. Disposition C ignores a concrete,
verifiable defect report and leaves a potentially still-broken diagram
in normative-adjacent content. Disposition A risks redoing a fix that
is already in place. Disposition B costs only one inspection of the
current build: if the diagram is whole, the ticket closes cleanly with
a citation; if it is still missing the "A" page, the fix is the same
narrow asset edit described in A. CIC should ask the assignee (once
named) to perform that inspection against the current ci-build of the
Communication workflow page and report back before the next call.
