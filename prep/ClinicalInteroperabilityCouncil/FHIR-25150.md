## Summary

The ticket reports that LOINC code **10193-1 "Physical findings of breast
narrative"** is being used in the US Breast Cancer Radiology Reporting
(FHIR) IG profile as the binding code for what is intended to be a
structured, multi-element observation. As its LOINC name indicates,
10193-1 represents a single narrative result (a free-text finding) and
therefore does not fit the complex, panel-style structure described in
the profile. The reporter (Susan Matney) recommends locating a different
code — most plausibly a LOINC panel/group code, or a set of component
codes — that semantically matches a structured set of physical breast
findings rather than a single narrative blob. The ticket has been
triaged with a **Persuasive** resolution: the work group agrees the
value sets and codes are under-specified in the current ballot and
plans to define them more thoroughly in the next ballot, including
creating the required LOINC / SNOMED / SOLOR codes as needed. There are
no cross-referenced Jira tickets, Zulip threads, or GitHub items for
this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Breast Cancer Radiology Reporting (FHIR) IG, replace the
binding to LOINC **10193-1 "Physical findings of breast narrative"** on
the structured "physical findings of breast" Observation profile with a
LOINC **panel/grouping** code (or a small value set of such codes) that
properly represents a structured, multi-component finding. Move 10193-1
to a clearly-labelled narrative-only slot (e.g.,
`Observation.valueString` on a separate "narrative" profile, or as a
DiagnosticReport-level conclusion code) so the narrative concept is
still available where appropriate, but is no longer bound to the
structured panel profile. Add the per-finding LOINC component codes
required by the profile to the `Observation.component.code` value set,
and file LOINC submissions for any concepts that do not yet exist.

#### Justification

This is exactly what the reporter asked for and aligns with how LOINC
distinguishes narrative result codes (single text observations) from
panel codes (groupings whose members are individual observations). The
work group's own resolution note already accepts the underlying
critique ("we agree the value sets/codes are not well defined in this
ballot") and commits to defining them more thoroughly in the next
ballot, including creating required LOINC / SNOMED / SOLOR codes as
necessary. Accepting as requested formalises that commitment as a
concrete change to the profile bindings.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than swap a single code, restructure the breast physical
findings model in the IG into **two coordinated artefacts**:

1. A **narrative** Observation profile that retains LOINC 10193-1 and
   is intended for unstructured/historical content where only a free
   text impression is available.
2. A **structured** "physical findings of breast" panel profile bound
   to a new LOINC panel code (submitted to LOINC if none exists), with
   `Observation.hasMember` (or `Observation.component`) entries bound
   to a curated value set of per-finding LOINC / SNOMED CT codes
   covering the elements the existing profile enumerates (e.g., mass,
   skin changes, nipple discharge, lymphadenopathy, etc.).

Provide guidance in the IG on when each profile applies, and add
ConceptMap or invariant-level rules to discourage use of 10193-1 inside
the structured profile.

#### Justification

This preserves implementations that may already be sending narrative
findings under 10193-1 while also giving radiology systems a clean
structured target. It addresses the same root cause as Disposition A
but adds explicit modelling for the narrative case, which is a real
clinical pattern (dictated impressions). The cost is a larger,
multi-profile change rather than a single binding swap, and it commits
the IG to LOINC submission work — but that submission work is already
implied by the work group's resolution note.

---

### Disposition C: Decline

#### Proposal

Decline to change the binding at this time. Rely on the existing IG
text and let implementers choose appropriate component-level codes
locally; revisit code selection as part of a broader IG refresh rather
than a per-code ballot fix.

#### Justification

Defensible only if the IG were close to a final publication freeze and
the work group judged that ad-hoc per-code corrections risk
destabilising existing trial implementations. However, this rationale
is weak here: the IG has already accepted the comment as Persuasive
and explicitly committed to better terminology in the next ballot, so
declining contradicts the current resolution stance. Listed for
completeness only.

---

### Recommendation

**Recommended disposition:** A

The work group has already marked this comment **Persuasive** and the
resolution note commits to redefining the value sets and codes in the
next ballot, including new LOINC / SNOMED / SOLOR concepts as needed.
Disposition A is the most direct, lowest-friction way to operationalise
that commitment: replace the misused narrative code with a panel /
grouping code on the structured profile and tighten the component
bindings. Disposition B is an attractive richer model and may be the
right destination, but it expands scope beyond what the ticket asked
for and beyond what the resolution note promises; it should be
considered a follow-on improvement once the immediate binding fix from
Disposition A lands. Disposition C contradicts the existing
Persuasive resolution and should not be chosen absent new information.
