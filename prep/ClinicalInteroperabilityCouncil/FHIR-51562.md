## Summary

The reporter raises a real-world interoperability problem with the Da Vinci
HRex security guidance (HRex §7.4): the spec offers four authorization
approaches — (a) SMART App Launch, (b) mutual TLS / client cert, (c) FAST
UDAP, and (d) SMART Backend Services — and only requires that one or more
SHOULD be implemented. As a payer that implemented UDAP (Option C), they
could not exchange with EHRs (cited Epic and MEDITECH) that had implemented
other options, so end-to-end testing of CRD/DTR/PAS stalled. They argue
options (a) and (d) are oriented to third-party clients reading EHR data,
while provider↔payer is a B2B exchange that needs richer organizational
context (Organization, Location, Practitioner, PractitionerRole) which is
naturally carried by UDAP. Their proposal is to mandate FAST UDAP (the
ONC-recommended profile) for both providers and payers across the
HRex-dependent Da Vinci IGs.

There are no Jira cross-references, but the ticket is the trigger for an
active Zulip thread in `#smart` ("Identifying Organization or User details
in SMART Backend") opened by Alex Kontur, which broadens the discussion to
whether the SMART community should standardize a way to convey
organization/user context inside SMART Backend Services (e.g., by
adopting the FAST `hl7-b2b` authorization extension object). The
resolution recorded on the ticket is **Not Persuasive**: Da Vinci IGs
draw on SMART App Launch, CDS Hooks, and other specifications that have
their own security requirements, the community has not yet committed to
UDAP everywhere, and trust frameworks covering all Da Vinci participants
do not exist; individual IGs may still elect to require UDAP.

## Details

**Description:**

Our Da Vinci IGs like CRD, DTR, PAS and other similar guides are pointing
to HREX security guide. However 7.4 section Bullet 2 is indicating that
only one or more approaches SHOULD be implemented. As payer, we
implemented UDAP (Option C). But when we tried to test with EHRs like
EPIC and MEDITECH, they did not implement UDAP but other approaches. So
we could not move forward with our testing. One of these two parties
must implement support for all approaches to make integration possible.

We need to put clarity that if Providers, payers or both parties need
to implement ALL four approaches.

Also we see option a (Smart App Launch Authorization) and option d
(SMART back-end services) are more focusing on third party clients
accessing data from EHR or Providers.

Also Provider to Payer relationship is more business to business
interaction and we need lot of additional information about Provider
Org, Location, Practitioner and Practitioner role information. All this
information is only available when we go for UDAP option.

So it seems easiest option is that both Provider and Payers security
should be FAST UDAP (recommended by ONC) only to keep things simple.

Please address this issue.

**Resolution (Not Persuasive):**

Da Vinci IGs draw on a number of different technologies, including SMART
app launch, CDS Hooks, and potentially other specifications that have
specific (non-UDAP) security requirements. And at this point, the
community has not evolved to the point where they are ready to commit to
using UDAP across all use-cases. As well, mandating UDAP would require
there to be trust entities that cover the full set of participants and
use-cases, and that's certainly not yet the case for all Da Vinci
use-cases. It is certainly possible that certain Da Vinci implementation
guides could agree to limit their security requirements to UDAP only, but
it's not something we can do across all Da Vinci IGs.

## Related Zulip Discussions

### [smart > Identifying Organization or User details in SMART Backend](https://chat.fhir.org/#narrow/stream/smart/topic/Identifying%20Organization%20or%20User%20details%20in%20SMART%20Backend/near/541269762)

Alex Kontur opened the thread on 2025-09-24 explicitly to take the
FHIR-51562 question to the SMART community: payers in the Da Vinci
Burden Reduction work need to identify the user and/or organization
behind a SMART Backend Services exchange, but SMART Backend has no
standard way to convey that context — some deployments register one
client ID per (app × organization), while others register a single
client ID for the product across many customers. The proposal under
discussion is to require SMART Backend Services plus the FAST Security
IG `hl7-b2b` extension object on the authentication JWT, which Kontur
warns would create a third overlapping OAuth profile (SMART + FAST + Da
Vinci) and risks fragmenting from any future SMART solution. Cooper
Thompson (Epic) strongly supports solving the problem and prefers
putting it in the SMART core spec rather than only in Da Vinci, citing
real-world cases where one client ID is shared by both payer and
provider tenants and the server cannot tell which is calling (e.g., for
self-pay filtering). Michele Mottini takes a hard line that each
distinct entity should have its own client ID + private key — that
multiplexed client IDs are essentially a security smell — while
acknowledging the approach does not scale. Grahame Grieve's terse
"evil_bit" link (RFC 3514) signals scepticism that a self-asserted
identity field meaningfully strengthens trust. Cooper later reframes
the design tension: the client ID conflates "product" and
"organization", and a single field cannot carry both. Grant Smith
(MiHIN) revives the thread in 2026-02 endorsing the UDAP/FAST/SSRAA B2B
access-token JWT extension as a promising standardization path,
suggesting `subject_role`/`purpose_of_use` could partially carry
context and that a `product_name` field could be advocated for; he
plans to raise it on Trebuchet calls. No consensus was reached; the
open questions are (i) where the solution lives (SMART core vs. FAST
vs. a Da Vinci-only profile), (ii) whether self-asserted org/user
claims carry meaningful trust, and (iii) how to represent both the
product and the deploying organization without overloading client ID.

Participants: Alex Kontur, Michele Mottini, Cooper Thompson, Grahame
Grieve, Grant Smith.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Amend HRex §7.4 to mandate FAST UDAP (Option C) as the **single
required** provider↔payer authorization profile for all HRex-dependent
Da Vinci IGs (CRD, DTR, PAS, and others that import HRex security).
Concretely: change the bullet that currently allows "one or more" of
options (a)–(d) to a SHALL on Option C for B2B provider↔payer hops, and
remove or downgrade options (a), (b), and (d) to informative for that
exchange direction. Add a normative requirement that the UDAP
authorization-extension object (`hl7-b2b`) be populated with
Organization (TIN-preferred), Practitioner, PractitionerRole, and a
`purpose_of_use` aligned with the IG (`TREATDS`/`HPAYMT` for CRD/DTR;
`COVAUTH` for PAS). Patient-facing and third-party-app flows would
continue to use SMART App Launch as today.

#### Justification

This directly addresses the reporter's interoperability failure —
parties that picked different "SHOULD" options cannot exchange — and
aligns with the ONC-recommended FAST profile, which is the only one of
the four that natively carries the organization/practitioner context
the provider↔payer use case demands. It also matches what Grant Smith
(MiHIN) and Cooper Thompson (Epic) describe as a real implementation
need on the Zulip thread.

### Disposition B: Alternative Approach (Pluralism + UDAP-style B2B claims layer)

#### Proposal

Keep HRex §7.4 plural (continue to allow SMART App Launch, mTLS, UDAP,
and SMART Backend Services), but tighten it as follows for
provider↔payer B2B hops in CRD/DTR/PAS:

1. Each IG SHALL designate a **floor** mechanism for B2B hops — namely
   SMART Backend Services — so that at least one common profile is
   guaranteed on both sides. This matches the draft mckenzie posted on
   2025-08-08.
2. When SMART Backend Services is used, systems SHALL include the FAST
   `hl7-b2b` authorization-extension object on the authentication JWT
   to convey Organization (TIN if available), PractitionerRole, and
   `purpose_of_use` (`TREATDS`/`HPAYMT` for CRD/DTR; `COVAUTH` for PAS).
3. FAST UDAP dynamic client registration SHOULD be supported,
   particularly when exchanging in a trust network; networks may add
   their own expectations (including endpoint discovery, which UDAP
   does not natively convey beyond authz endpoints).
4. Document explicitly that SMART App Launch is for EHR-mediated DTR
   user flows, not for the B2B server-to-server hops.

#### Justification

This is the disposition the Burden Reduction call effectively approved
on 2026-04-15: it solves the interoperability failure (a single
mandatory floor profile + a standard claims layer) without forcing
every Da Vinci participant into UDAP trust-framework membership, which
the resolution text correctly notes does not yet exist for all
participants. It addresses Alex Kontur's endpoint-discovery concern
honestly (acknowledging UDAP DCR doesn't carry FHIR endpoint info, so
networks/directories must) and gives Cooper Thompson's
"who-is-calling" use case a concrete answer via the `hl7-b2b`
extension. It does not pre-empt the SMART community's parallel work on
conveying organization/user context — if SMART core later
standardizes this, Da Vinci IGs can converge.

### Disposition C: Decline (record the as-filed disposition)

#### Proposal

Decline the request as filed — i.e., do not amend HRex to mandate FAST
UDAP across all Da Vinci IGs. Record the existing "Not Persuasive"
resolution: HRex sits beneath IGs that depend on SMART App Launch, CDS
Hooks, and other specifications with their own security models;
trust-framework coverage for all Da Vinci use-cases does not yet
exist; and individual Da Vinci IGs remain free to narrow their security
requirements (e.g., to UDAP-only) where their community is ready to do
so.

#### Justification

This is defensible for three reasons. First, scope: HRex is a shared
substrate, and the reporter's evidence is specifically about the
provider↔payer hop in CRD/DTR/PAS — a UDAP-only mandate at the HRex
level would force the constraint onto IGs (e.g., patient-facing flows,
CDS Hooks calls) where UDAP is the wrong tool. Second, dependency:
mandating UDAP requires a credentialing/trust authority that covers all
Da Vinci participants, and that infrastructure is not in place. Third,
existing mechanism: each downstream IG can already tighten HRex's
SHOULD into a SHALL where its community has consensus, which is the
appropriate venue.

### Recommendation

**Recommended disposition:** B

The reporter's underlying problem — that HRex's "one or more SHOULD"
permits non-overlapping choices and breaks interoperability — is real
and is independently corroborated on the SMART Zulip thread by both
Epic and MiHIN. Pure Disposition A overshoots: it would impose a
trust-framework requirement that the Da Vinci ecosystem cannot
currently satisfy and would conflict with the SMART/CDS Hooks
foundations many IGs already depend on, which is exactly what the
recorded "Not Persuasive" resolution is reacting to. Pure
Disposition C technically closes the ticket but leaves the
interoperability gap unaddressed and ignores both the active SMART
community work and the BR call's own draft language. Disposition B
takes the disposition the BR call already approved on 2026-04-15 —
SMART Backend as the floor, FAST `hl7-b2b` extension to carry
Organization/Practitioner/`purpose_of_use`, FAST DCR as a SHOULD for
network exchanges — and packages it as an HRex-level normative change
that solves the reporter's concrete failure while preserving the
heterogeneity the resolution rationale defends. The remaining open
issues mckenzie listed (JWT cert issuance, organization-id selection
when no network is present, CDS Hooks JWT language, EHR access-token
scaling) should be tracked as follow-up tickets so they don't block
this one.
