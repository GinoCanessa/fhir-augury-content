## Summary

The submitter (Ron G. Parker) asks the mCODE IG authors to consider
**anchoring the IG's value sets in THO (terminology.hl7.org)** instead of
defining and hosting them inside the mCODE IG itself. The request applies
broadly to value sets defined in mCODE that could reasonably be shared
across HL7 specifications. The ticket was originally dispositioned as
*Considered for Future Use* during the STU4 ballot cycle (Tcheng /
Potteiger, 5-0-4) on the grounds that the most-used FHIR IGs (e.g., US
Core) also keep value sets in-IG and the mCODE WG wanted to stay aligned
with that convention. After STU4.0.0 publication, lynn_laakso re-opened
the ticket and reverted that resolution so the WG can reconsider the
question on its merits. No cross-referenced Jira tickets, Zulip threads,
or GitHub items were surfaced for this ticket by Augury.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

For the next mCODE STU release, **migrate mCODE-defined value sets (and
their supporting CodeSystems where applicable) into THO** at canonicals
of the form `http://terminology.hl7.org/ValueSet/<id>` and
`http://terminology.hl7.org/CodeSystem/<id>`. Concretely:

- Inventory the value sets currently defined under
  `http://hl7.org/fhir/us/mcode/ValueSet/...` and classify each as
  (a) cross-IG-reusable terminology, (b) mCODE-specific composition of
  external code systems, or (c) thin wrappers around a single external
  code system.
- For class (a) and (c), submit the artifacts to the appropriate THO
  steward (Vocabulary WG) for hosting in THO; in mCODE, replace the
  in-IG definitions with binding references to the new THO canonicals
  and update profile bindings (`Observation`, `Condition`,
  `Procedure`, `MedicationRequest`, etc. in mCODE) accordingly.
- For class (b), keep them in mCODE but document the rationale.
- Provide a backwards-compatibility note mapping each retired
  `hl7.org/fhir/us/mcode/ValueSet/...` canonical to its new THO
  canonical, and (where possible) leave the old canonical in place as
  a deprecated alias for one cycle to give implementers a migration
  window.

#### Justification

THO is the HL7-recommended home for value sets and code systems that
are reusable across specifications, and the original 2024-06 vote
explicitly framed the deferral as "we will reconsider in the future"
rather than a principled rejection. mCODE's terminology (cancer
disease status, treatment intent, staging, etc.) is increasingly cited
by other oncology and quality-measure IGs, so anchoring in THO reduces
duplication and version skew, and aligns mCODE with HL7's terminology
governance direction. The reporter is Ron G. Parker, and the ticket's
"Highest" priority and inclusion in the `stu4-block-vote-1` label set
indicate the WG already considers it material.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a **hybrid / criteria-driven** approach instead of a blanket
move. Define a written mCODE terminology-placement policy that:

- **Keeps in mCODE** value sets that are tightly coupled to mCODE
  profile semantics, change at mCODE's cadence, or that compose
  external code systems in mCODE-specific ways.
- **Migrates to THO** only those value sets that meet explicit reuse
  criteria (used by ≥1 other published or in-development HL7 IG,
  stable for at least one STU cycle, no mCODE-specific composition
  logic).
- Ratifies the policy with Vocabulary WG, then runs the inventory from
  Disposition A and migrates only the qualifying subset, leaving the
  rest in mCODE with the policy as documented justification.

#### Justification

A wholesale migration carries non-trivial implementation cost: every
relocated canonical changes for downstream implementers, validators,
and terminology servers, and any value set that gets retired from
mCODE-as-source must still be governed somewhere. The 2024-06
resolution's reasoning — consistency with US Core and other major
IGs that keep value sets in-IG — is still valid for value sets with
no demonstrated cross-IG reuse. A criteria-driven policy gets the
benefits of THO anchoring (shared governance, deduplication) for the
artifacts that actually need it, while avoiding gratuitous canonical
churn for the ones that don't, and gives the WG a defensible basis to
say "yes" or "no" to similar future requests without re-litigating
the principle each time.

---

### Disposition C: Decline

#### Proposal

Decline the request as **Considered, no change** (effectively
re-affirming the original 2024-06 disposition). Document that mCODE
will continue to define value sets under
`http://hl7.org/fhir/us/mcode/ValueSet/...` for the foreseeable
future, citing alignment with US Core and other major US Realm IGs
and the absence of a concrete cross-IG reuse case in the ticket.

#### Justification

The ticket as written contains only a one-sentence ask with no
specific value sets, no enumerated reusers, and no analysis of the
migration cost. The original WG vote (5-0-4) already weighed
consistency with major IGs against the abstract benefit of THO
anchoring and concluded that the cost wasn't justified at the time.
Without new evidence — e.g., another IG that needs to bind to a
specific mCODE value set, or a Vocabulary WG request to take
ownership — the same reasoning still applies, and declining preserves
canonical stability for STU4 implementers.

---

### Recommendation

**Recommended disposition:** B

A blanket "yes" (Disposition A) creates real implementer churn for
artifacts that may never be reused outside mCODE, and the original WG
explicitly cited consistency with US Core as a reason to keep value
sets in-IG — that argument hasn't gone away. A blanket "no"
(Disposition C) ignores the fact that lynn_laakso re-opened the
ticket post-STU4 specifically to revisit the question, and it leaves
mCODE without a principled answer the next time a cross-IG reuse case
appears. A criteria-driven policy (Disposition B) gives the WG a
defensible, repeatable rule, lets it move the genuinely
cross-IG-reusable value sets to THO where they belong, and avoids
gratuitous canonical churn for the rest. The recommended next step is
a joint conversation with Vocabulary WG to ratify the criteria and
then run the inventory.
