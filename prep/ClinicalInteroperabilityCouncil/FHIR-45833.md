## Summary

The ticket asks the mCODE IG to consider anchoring its value sets in
[terminology.hl7.org](https://terminology.hl7.org) (THO) rather than
defining them locally inside the IG, on the rationale that the value
sets in question are likely to be useful beyond mCODE. The original
disposition (2024-06-27) was *Considered for Future Use* (vote
5 - 0 - 4, Tcheng / Potteiger), with the workgroup arguing that broadly
used FHIR IGs such as US Core also keep their value sets anchored
inside the IG (or in base FHIR) and that mCODE preferred to stay
consistent with that practice. Lynn Laakso re-opened the item on
2025-02-17 ("Re-open post STU4.0.0 publication") and reverted the
prior resolution, so the workgroup must take a fresh disposition for
post-STU4 work. No cross-referenced Jira tickets, Zulip threads, or
GitHub items were surfaced by the search and cross-reference indexes
for this key.

### Disposition A: Accept As Requested

#### Proposal

Migrate the in-scope mCODE value sets from the mCODE IG canonical
namespace to the THO canonical namespace
(`http://terminology.hl7.org/ValueSet/...`). Concretely:

1. Define the in-scope set as those mCODE value sets that (a) are
   not tightly bound to a single mCODE profile's semantics and
   (b) reference code systems already maintained in THO
   (e.g., LOINC, SNOMED CT, ICD-O-3, NCI Thesaurus). Cancer-specific
   groupers that have plausible reuse outside oncology
   (e.g., performance status, generic staging components, body
   site groupings) are first candidates.
2. For each such ValueSet:
   - Move the authoring artifact to the THO source repository under
     its standard authoring layout, keeping the same logical content
     and compose rules.
   - Reissue under
     `http://terminology.hl7.org/ValueSet/<id>` with the THO
     versioning policy.
   - In the mCODE IG, replace local `ValueSet` definitions with
     references to the THO canonicals from the relevant
     `StructureDefinition` `binding.valueSet` slots.
3. Coordinate with the THO maintainers (Vocabulary WG) on naming,
   stewardship, and publication timing so the THO release lines up
   with the mCODE post-STU4 publication.
4. Add IG release notes and a migration appendix that maps the old
   mCODE canonical URLs to the new THO canonicals so implementers
   can update references.

#### Justification

The reporter's premise — that some mCODE value sets are useful
beyond oncology — is consistent with HL7's general direction of
consolidating broadly reusable terminology in THO so that multiple
IGs can bind to a single, versioned canonical instead of each IG
re-publishing near-duplicates. Anchoring in THO reduces fragmentation
across cancer-adjacent IGs (e.g., genomics reporting, registries,
quality measures), gives the value sets an explicit terminology
governance home (Vocabulary WG), and avoids the situation where the
authoritative copy of a clinically generic value set lives inside an
oncology-specific IG.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a **two-tier policy** rather than a wholesale migration:

- **Keep in mCODE:** value sets whose semantics are oncology-specific
  or that exist primarily to constrain a single mCODE profile
  (mCODE-internal groupers, oncology-discipline-specific code
  selections). These remain at
  `http://hl7.org/fhir/us/mcode/ValueSet/...`.
- **Promote to THO:** only those value sets that meet a documented
  reuse test — used or plausibly used by ≥1 IG outside mCODE,
  composed of code systems already governed in THO, and stable
  enough to commit to THO's versioning cadence.

Implement the policy by:

1. Adding a short "Value Set Hosting Policy" section to the mCODE IG
   that states the criteria and lists which value sets are
   THO-anchored vs. IG-anchored, with rationale.
2. Promoting an initial small batch (e.g., 2–4 high-reuse value
   sets) to THO in this cycle, leaving the rest in mCODE.
3. Establishing a per-cycle review (CIC + Vocabulary WG) to promote
   additional value sets as cross-IG demand is demonstrated.

#### Justification

This honours the spirit of the request without taking on the cost
and risk of moving every value set at once. The original 2024-06-27
disposition cited consistency with US Core, which keeps value sets
in-IG; a published policy with an explicit reuse test answers that
concern by making the choice principled rather than ad hoc. It also
limits churn for mCODE implementers — only the demonstrably
cross-cutting value sets change canonicals — and gives the
Vocabulary WG a manageable intake instead of a bulk transfer.
Trade-off vs. Disposition A: less aggressive consolidation, but
sustainable and reversible.

---

### Disposition C: Decline

#### Proposal

Decline the request and confirm the existing pattern: mCODE
value sets remain anchored at
`http://hl7.org/fhir/us/mcode/ValueSet/...`. Add a short note in
the IG explaining that mCODE follows the prevailing US-realm IG
convention (consistent with US Core) of co-locating value sets with
the bindings that use them, and that any future externalization
will be considered case-by-case if and when concrete cross-IG reuse
is demonstrated.

#### Justification

This is the position the workgroup actually held in 2024-06-27
(5 - 0 - 4) — consistency with US Core and other broadly used
US-realm IGs, plus the implementer expectation of finding mCODE
value sets at mCODE canonicals. Migration to THO has real cost:
canonical URL churn for downstream implementers, dual stewardship
coordination with Vocabulary WG, and risk that THO's publication
cadence does not align with mCODE's. Without concrete external
consumers identified in the ticket, the request is speculative and
the existing mechanism (publish in mCODE; promote later if a real
reuse case appears) already covers the underlying need.

---

### Recommendation

**Recommended disposition:** B

The ticket's underlying point is sound — some mCODE value sets are
clinically generic and would benefit downstream IGs if hosted
centrally — but the 2024-06-27 vote also captured a real concern
about consistency, implementer expectations, and the cost of
canonical URL churn. Disposition B threads both: it commits the
workgroup to a documented hosting policy, promotes a small,
defensible initial batch to THO this cycle, and leaves the bulk of
mCODE-specific value sets where implementers already expect them.
Disposition A is preferable in principle but is high-risk to land
cleanly inside one cycle without a prior repo-analysis of the mCODE
IG source and explicit Vocabulary WG buy-in. Disposition C re-states
the prior position and ignores the fact that Lynn Laakso explicitly
re-opened the ticket post-STU4.0.0 to reconsider — declining outright
would not honour that re-opening. Before implementation, a
`repo-analysis` of the mCODE IG repository should be run so the
specific authoring paths for the promoted value sets and their
binding `StructureDefinition`s can be named.
