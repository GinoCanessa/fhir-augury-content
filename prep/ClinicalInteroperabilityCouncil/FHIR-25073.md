## Summary

The ticket — a balloter comment ("BC #49") against the *US Breast Cancer
Radiology Reporting (FHIR)* IG — asks the publication's diagrams to
include a key (legend) explaining what each color and shape represents.
The ballot comment is brief: the reviewer noted the diagrams use color
and shape coding without a legend, and proposed adding one so readers
can interpret the figures unaided. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were returned by Augury for this key, so
the request stands on its own. The ticket has already been triaged with
a *Persuasive* resolution: the work group agreed and indicated the
diagramming would be improved in a number of ways.

## Proposed Dispositions

> Note: this ticket already carries a *Persuasive* resolution from the
> 2019 ballot cycle. The dispositions below are framed as the choices
> that *would* have been available to the work group at disposition
> time, and remain useful as a confirmation/audit pass.

### Disposition A: Accept As Requested

#### Proposal

Add an explicit legend ("Color and Shape Key") to the IG's narrative
content immediately adjacent to the first figure that uses color/shape
encoding, and reference it from each subsequent diagram. The legend
should enumerate every color and shape used in the published diagrams
(e.g., resource boxes vs. profile boxes vs. extension boxes; required
vs. must-support vs. optional element styling) with a short textual
description of each. Where the IG publisher generates diagrams (e.g.,
PlantUML / generated class diagrams), update the source so the legend
is regenerated alongside the figures rather than maintained as a
separate static image.

#### Justification

Aligns directly with the balloter's request and matches the work
group's stated resolution ("we agree…"). Adding a legend is a
documentation-only change with no impact on conformance, schemas, or
implementer behavior, and it materially improves the figures'
self-containedness — a standard expectation for HL7 ballot-quality
narrative. It also addresses an accessibility concern: readers who
cannot reliably distinguish certain color pairs benefit from the shape
encoding being called out explicitly in text.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) inserting a per-page legend, adopt a
single canonical "Diagram Conventions" page in the IG's front matter
that documents every color, shape, line style, and arrow used across
the publication, and link to it from a short caption under each
figure. Where feasible, also annotate the diagrams themselves with
inline labels so the encoding is self-explanatory without bouncing to
the conventions page. If the IG inherits diagram styling from a shared
HL7 template, push the conventions page upstream so other IGs benefit
from the same key.

#### Justification

A single conventions page is cheaper to keep in sync than per-figure
legends and avoids legend drift as new diagrams are added. Inline
labels reduce the need to consult the key at all, which is the most
robust accessibility outcome. The trade-off vs. Disposition A is one
extra click for the reader on the first encounter with a figure;
that's usually acceptable for a reference IG. Pushing conventions
upstream is the highest-leverage variant if the diagrams are produced
by a shared toolchain.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the figures are illustrative
rather than normative, that the surrounding narrative already names
each depicted artifact, and that adding/maintaining a legend imposes
ongoing editorial cost without changing conformance.

#### Justification

Defensible only if the diagrams are genuinely redundant with the
narrative and the work group judges the ballot comment to be a stylistic
preference rather than a clarity defect. In practice this is a weak
position for a ballot reconciliation: balloters rarely raise legend
requests unless they actually struggled to read the figures, and
declining a low-cost documentation improvement tends to attract
negative votes on subsequent ballot cycles.

---

### Recommendation

**Recommended disposition:** A (consistent with the existing
*Persuasive* resolution).

The request is small, low-risk, and entirely within the IG's narrative
layer — there is no schema, profile, or terminology impact to weigh.
The work group already agreed in 2019 ("We agree, and will be
improving our diagramming in a number of ways"), and Disposition A is
the most direct path to closing that commitment. Disposition B is a
strictly better long-term answer if the IG team is willing to invest
in a shared conventions page and/or inline diagram labels, and can be
folded into the *"in a number of ways"* portion of the resolution
without changing the disposition itself. Disposition C should be
avoided: declining a clarity-improvement ballot comment with no
implementer cost is rarely worth the goodwill it burns.
