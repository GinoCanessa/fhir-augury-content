## Summary

The reporter observes that the `breastrad-recommendations` profile in the
US Breast Cancer Radiology Reporting (BCRR) FHIR IG has an entry typed as
`obf-ActionRequest`, which is not a FHIR core resource. It is unclear from
the IG whether `obf-ActionRequest` is intended as a logical model or as a
FHIR core resource. The reporter asks why the IG does not instead use
`PlanDefinition`, which is a FHIR core resource defined for representing
plans/recommendations of clinical actions. The ticket therefore proposes
replacing the use of `obf-ActionRequest` in `breastrad-recommendations`
with a FHIR-core resource that fits the recommendation use case.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the `obf-ActionRequest` reference on the `breastrad-recommendations`
profile's `entry` slice with `PlanDefinition`. Concretely, in the BCRR IG:

- Update the `Bundle` profile `breastrad-recommendations` so that the
  recommendation `entry` slice targets `PlanDefinition` (FHIR core)
  instead of the CIMPL-derived `obf-ActionRequest` logical model.
- Adjust supporting StructureDefinitions, examples, narrative, and
  ImplementationGuide resource list to drop `obf-ActionRequest` from
  this slot.
- Provide at least one revised example `PlanDefinition` representing a
  follow-up imaging recommendation (e.g., "6-month follow-up
  mammogram").

#### Justification

The literal request is to use a FHIR core resource for what is
semantically a clinical recommendation/plan, and `PlanDefinition` is the
core resource explicitly intended for "definition of a clinical or
business action plan", including diagnostic imaging follow-up
recommendations. Using a core resource (rather than a CIMPL-derived
logical-model reference) removes implementer ambiguity about whether
`obf-ActionRequest` is a real wire-format resource, improves
interoperability with non-CIMPL implementers, and aligns with FHIR
modelling guidance to prefer core resources before introducing
profile/logical-model variants.

---

### Disposition B: Alternative Approach

#### Proposal

Do not use `PlanDefinition`. Instead, replace `obf-ActionRequest` on
`breastrad-recommendations` with `ServiceRequest` (and optionally
`Appointment`) as the entry type, matching what the resolution text on
the ticket describes as the project's actual direction:

- Change the `breastrad-recommendations` recommendation entry to allow
  `ServiceRequest` (primary) and `Appointment` (secondary, where a
  scheduled follow-up is being conveyed).
- Retain `PlanDefinition` only at the *definitional* layer (e.g., a
  protocol for "BI-RADS category 3 → 6-month follow-up"), and have the
  per-patient recommendation be a `ServiceRequest` instantiating that
  `PlanDefinition`.
- Update profiles, examples, and narrative accordingly; remove all
  `obf-ActionRequest` references from the IG.

#### Justification

`PlanDefinition` is a *definitional* resource — it represents the
template/protocol, not the patient-specific request. A radiology
recommendation produced for a specific patient ("Patient X should
receive a 6-month follow-up ultrasound") is operationally a
`ServiceRequest` (optionally with an `Appointment` if a slot has been
booked). This is also what the workgroup committed to in the ticket's
resolution text ("we currently intend to use ServiceRequest and/or
Appointment"). It addresses the reporter's underlying concern (no more
non-core `obf-ActionRequest`) while keeping the per-patient artifact in
the right resource class, and leaves room to layer a `PlanDefinition`
in for protocol-level reuse.

---

### Disposition C: Decline

#### Proposal

Decline the request as filed (i.e., decline a like-for-like swap to
`PlanDefinition`) on the grounds that the underlying issue is already
being addressed in the next ballot cycle by removing `obf-ActionRequest`
in favour of `ServiceRequest`/`Appointment`, per the ticket's recorded
resolution.

#### Justification

The reporter's substantive concern — that `obf-ActionRequest` is not a
FHIR core resource and is ambiguous to implementers — is acknowledged
and is being remediated by the project's stated next-ballot plan. A
formal "Accept" of the literal `PlanDefinition` proposal would commit
the IG to the wrong resource class for a per-patient recommendation
(`PlanDefinition` is definitional). Declining the literal ask while
documenting the agreed `ServiceRequest`/`Appointment` direction in the
resolution avoids over-committing the IG to a modelling choice that
does not match `PlanDefinition`'s intent.

---

### Recommendation

**Recommended disposition:** B

The reporter is right that an IG bundle slot pointing at a non-core
`obf-ActionRequest` is a real interoperability problem and should be
fixed, so a flat decline understates the issue. However, `PlanDefinition`
is the wrong target for a *patient-specific* recommendation entry — it
is a definitional resource, not an instance request — so accepting the
literal proposal would trade one modelling problem for another. Option B
matches the resolution text already recorded on the ticket
("ServiceRequest and/or Appointment"), removes `obf-ActionRequest` from
the IG entirely (which is what the reporter actually cares about), and
preserves the option to use `PlanDefinition` at the protocol layer if
the workgroup later wants to formalise BI-RADS-driven follow-up
protocols. This gives the workgroup a concrete, FHIR-idiomatic path
forward without overloading `PlanDefinition`.
