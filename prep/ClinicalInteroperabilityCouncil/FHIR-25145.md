## Summary

The ticket (Breast Cancer ballot comment #96) reports that the
"patient history" element in the US Breast Cancer Radiology Reporting
FHIR profiles is typed as a `Reference(Observation)` only, and that this
target type is too narrow for the kinds of historical clinical
information a radiology report actually needs to cite. The reporter
notes that prior history relevant to a breast cancer radiology report
can also live in `DiagnosticReport` instances (e.g., prior imaging or
pathology reports), History & Physical notes, or other prior pathology
documents — none of which can currently be linked from this slot. The
request is to broaden the allowed reference targets so that history can
point at the full range of artifacts that carry it, not just discrete
`Observation` resources. There are no cross-referenced Jira tickets,
Zulip threads, or GitHub items recorded in Augury for this issue, so
context is limited to the ticket itself; the reviewer's response on
the ticket is "We agree, and need to discuss this," and the resolution
was set to **Persuasive**.

### Disposition A: Accept As Requested

#### Proposal

Update the US Breast Cancer Radiology Reporting profile that carries
the "patient history" element so the reference target list is broadened
from `Reference(Observation)` to
`Reference(Observation | Condition | DiagnosticReport | DocumentReference | Composition)`.
This captures the four artifact families the reporter calls out:
discrete observations, prior diagnostic reports (including prior
imaging and pathology reports rendered as `DiagnosticReport`), H&P /
narrative notes (`Composition` or `DocumentReference`), and standalone
past pathology documents (`DocumentReference`). `Condition` is added so
that an actual problem-list entry from the patient's history (e.g.,
prior breast cancer) can be referenced directly without forcing it to
be re-stated as an `Observation`. No cardinality change is implied;
the slot stays optional/repeating as today.

#### Justification

The reporter's complaint is fundamentally that the profile mis-models
clinical reality: "patient history" in a breast cancer radiology
report is heterogeneous, and FHIR already has resources designed for
each of those flavours. Constraining the target to `Observation` alone
forces implementers to either fabricate `Observation` wrappers around
`DiagnosticReport` or `DocumentReference` payloads, or to drop the
information entirely — both of which degrade interoperability. The
expanded target set is consistent with how other US-realm clinical IGs
(e.g., US Core's history- and document-style references) already model
historical context, so this aligns the BC profile with established
patterns. The ticket's own resolution note ("We agree, and need to
discuss this") shows the workgroup has already conceded the substance;
Disposition A simply concretes the agreement into a profile change.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than widening the existing `patient history` element in place,
introduce a small extension (or repurpose `supportingInfo`-style
slicing if the underlying resource provides one) named something like
`bc-patient-history-reference` whose `value[x]` is
`Reference(Observation | Condition | DiagnosticReport | DocumentReference | Composition)`,
and leave the original `Reference(Observation)` element untouched for
backward compatibility. The IG narrative would be updated to direct
implementers to use the extension when the prior history is carried in
a non-`Observation` artifact, and the original element would be marked
as the canonical slot only when the history really is a discrete
`Observation`.

#### Justification

This preserves wire-level backward compatibility for any early
implementer that already populates the existing `Reference(Observation)`
element — important because the IG is realm-specific and may already
have validators / tooling pinned to the narrow target list. It also
sidesteps the "is this a breaking change?" debate that a direct
target-type widening will raise at ballot, by additively introducing
new capability instead of altering the published constraint. The cost
is ergonomic: implementers face two slots that do almost the same
thing, and consumers must look in both places. That trade-off is
defensible only if the workgroup believes the installed base is large
enough to matter; for a still-evolving radiology IG, it usually is
not, which is why this is offered as the alternative rather than the
recommendation.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive (with mod)** on the grounds that
the existing `Reference(Observation)` slot, combined with FHIR's
general ability to carry references to prior artifacts elsewhere on
the report (e.g., `DiagnosticReport.result`, `DiagnosticReport.basedOn`,
or a generic `supportingInfo`/`extension` mechanism on the host
resource), already provides a path to convey prior history without
changing the BC profile's target list. The resolution would direct
implementers to use those existing slots for `DiagnosticReport`,
`DocumentReference`, and `Composition` references, and to reserve the
`patient history` element for true `Observation`-shaped history items.

#### Justification

A defensible "decline" rests on two claims: (1) the ticket conflates
"history" with "anything previously recorded," and (2) the host
resource (Observation/DiagnosticReport, depending on which BC profile
this element lives on) already exposes generic linkage points that can
carry prior `DiagnosticReport` / `DocumentReference` references without
profile change. If both hold, broadening the target list adds
implementation surface for no semantic gain. This disposition is
included for completeness, but it sits in tension with the workgroup's
own recorded resolution ("We agree…"), which is why it is unlikely to
prevail.

---

### Recommendation

**Recommended disposition:** A

The ticket is a narrowly-scoped, well-justified profile fix on a
realm-specific IG, and the workgroup has already recorded agreement on
the substance ("We agree, and need to discuss this"; resolution
**Persuasive**). Disposition A turns that agreement into a concrete,
minimal change — widening the `patient history` reference target list
to include `DiagnosticReport`, `DocumentReference`, `Composition`, and
`Condition` alongside `Observation` — that directly addresses every
artifact the reporter calls out (prior diagnostic reports, H&P notes,
past pathology documents). Disposition B (extension-based) buys
backward compatibility the IG most likely does not need at this stage
and adds two-slot ambiguity for implementers; Disposition C is hard
to defend against the workgroup's own resolution note. The remaining
work is mechanical: identify the exact profile and element path in the
US Breast Cancer Radiology Reporting IG, edit the profile's `type`
slice, and regenerate the IG. A follow-up `repo-analysis` on the host
IG repository will let `ticket-plan` pin the change to specific files.
