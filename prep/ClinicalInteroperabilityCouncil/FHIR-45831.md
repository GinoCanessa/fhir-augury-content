## Summary

The ticket asks the mCODE work group to consider anchoring the IG's value
sets in the HL7 Terminology server (THO, `terminology.hl7.org`) rather than
publishing them inside the mCODE IG itself, on the rationale that the value
sets could be reusable beyond mCODE. The reporter (Ron G. Parker) raised it
during STU4 ballot. The mCODE team initially resolved it as "Considered for
Future Use" on 2024-06-27 (vote 5–0–4), arguing that the most broadly used
FHIR IGs (e.g., US Core) anchor their value sets in the IG or in base FHIR
and that mCODE wished to stay consistent with that pattern. That resolution
was reverted on 2025-02-17 by Lynn Laakso so the request can be reconsidered
after STU4.0.0 publication, which is the state in which it now sits. There
are no cross-referenced Jira tickets, no related Zulip threads, and no
related GitHub items recorded for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move mCODE's IG-defined value sets (and, where appropriate, the
underlying code systems mCODE owns) out of the mCODE IG and publish them
in the HL7 Terminology (THO) project at `terminology.hl7.org`. In the
mCODE IG, replace the in-IG `ValueSet` definitions with references to
the THO-hosted canonicals (same canonical URL, version pinning per
mCODE's terminology-binding rules), and update profile bindings to point
at those THO canonicals. Coordinate with the Vocabulary work group and
the THO maintainers on naming, ownership, and the STU/normative status
of each migrated value set, and document the migration (and any URL
changes) in the mCODE IG's terminology page and changelog.

#### Justification

This directly addresses the reporter's concern that mCODE's value sets
are potentially reusable beyond the IG: hosting them in THO makes them a
first-class HL7 terminology asset, gives them an identity independent of
the mCODE publication cadence, and lets other IGs bind to them without
taking a dependency on mCODE. The reconsideration was explicitly
deferred to "post STU4.0.0 publication", which is the current state, so
the timing the work group set for itself has arrived. THO is the
designated home for cross-IG terminology artifacts, so this aligns with
the broader HL7 architectural direction even though it differs from US
Core's current pattern.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a hybrid policy rather than a wholesale move: keep mCODE-internal
value sets (those that are tightly coupled to mCODE profiles and
unlikely to be consumed elsewhere) in the mCODE IG, and migrate to THO
only those value sets that meet a documented "cross-IG reuse" bar —
e.g., value sets that already have external implementer interest, that
draw from widely used external code systems (SNOMED CT, LOINC, ICD-O)
without mCODE-specific scoping, or that the Vocabulary WG agrees are
candidates. Capture the criteria in the mCODE IG's terminology guidance
so future value sets are placed correctly from the start, and produce a
short candidate list for STU5 with the Vocabulary WG.

#### Justification

This preserves the consistency-with-US-Core argument from the original
2024-06-27 resolution for the bulk of mCODE's bindings while still
honouring the reporter's underlying need for the genuinely reusable
value sets. It also avoids a single large migration that would churn
canonical URLs and require every mCODE implementer to revalidate
bindings at once. By engaging the Vocabulary WG in setting the bar, the
work group keeps governance of cross-IG terminology where it belongs
without ceding mCODE-specific terminology decisions.

---

### Disposition C: Decline

#### Proposal

Re-affirm the 2024-06-27 resolution: keep mCODE's value sets anchored in
the mCODE IG, citing consistency with US Core and other widely used
FHIR IGs that publish their value sets in-IG. Close the ticket as
"Persuasive with Mod" / "Not Persuasive" (whichever the work group
prefers procedurally) with a note that the policy will be revisited only
if a concrete external IG identifies a value set it wants to bind to
directly.

#### Justification

Declining is defensible on three grounds. First, the original vote
(5–0–4) was on the same substantive question and the underlying
landscape — US Core's pattern of in-IG value sets — has not changed.
Second, no cross-referenced ticket, Zulip discussion, or GitHub issue
has surfaced demonstrating concrete external demand for the mCODE value
sets, so the "useful beyond this IG" claim is currently speculative.
Third, a migration carries non-trivial cost (canonical-URL governance,
re-balloting through the Vocabulary WG, version-pinning updates across
all profiles) that is hard to justify without a named consumer.

---

### Recommendation

**Recommended disposition:** B

The reporter's point — that some mCODE value sets are plausibly
reusable — is a real one, and the work group itself signaled openness
by reverting the prior resolution and asking for reconsideration
post-STU4.0.0. At the same time, Disposition A is too blunt: it forces
churn on every binding and abandons the consistency-with-US-Core
argument without evidence that all mCODE value sets warrant THO
hosting. Disposition C ignores the explicit re-open. Disposition B
splits the difference in a way that is governable: it keeps existing
bindings stable, defers actual migrations until a concrete reuse case
is identified, and produces a written placement policy with the
Vocabulary WG so future value sets land in the right place from the
start. That gives the workgroup something concrete to vote on without
committing STU5 to a large terminology refactor.
