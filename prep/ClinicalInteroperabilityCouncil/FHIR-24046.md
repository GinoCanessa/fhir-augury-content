## Summary

The ticket asks the authors of the US Breast Cancer Radiology Reporting IG
to refine several `Reference(...)` targets on the `BreastAbnormality`
profile (a profile of `Observation`):

1. `BreastAbnormality.partOf` currently references `obf-MedicationStatement`
   and `obf-Procedure` (CIMPL/`obf` framework profiles). The reporter
   argues these references should target the unprofiled core resources
   `MedicationStatement` and `Procedure` instead, and offers to demonstrate
   the CIMPL "No profile" flag that produces that effect.
2. `BreastAbnormality.subject` should be constrained to `Patient` only
   (the broader allowance ŌĆö `Group`, `Location`, etc. inherited from
   `Observation.subject` ŌĆö does not make clinical sense for a breast
   abnormality observation).
3. `BreastAbnormality.performer` should be constrained to `Practitioner`
   only (other allowed targets such as `Organization`, `CareTeam`,
   `RelatedPerson`, etc. similarly do not make sense in this context).

The work group has already accepted the ticket as **Persuasive**; the
resolution recorded by the WG states "We agree. In the next ballot we
will be revisiting each of those issues." There are no cross-referenced
Jira tickets, GitHub items, or focused Zulip threads tied directly to
this change request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the `BreastAbnormality` profile of the US Breast Cancer Radiology
Reporting IG, tighten the three reference target lists:

- **`BreastAbnormality.partOf`** ŌĆö replace
  `Reference(obf-MedicationStatement | obf-Procedure)` with
  `Reference(MedicationStatement | Procedure)` (i.e., the unprofiled
  core resources). In CIMPL source this is the "No profile" flag on
  the `partOf` mapping rule.
- **`BreastAbnormality.subject`** ŌĆö constrain target type to
  `Reference(Patient)` only (remove `Group`, `Device`, `Location`).
- **`BreastAbnormality.performer`** ŌĆö constrain target type to
  `Reference(Practitioner)` only (remove `PractitionerRole`,
  `Organization`, `CareTeam`, `Patient`, `RelatedPerson`).

These are zero-cost narrowing constraints on an already-restrictive
profile, applied in the next ballot publication of the IG.

#### Justification

The work group has already marked the ticket Persuasive ("We agree. In
the next ballot we will be revisiting each of those issues."). The
requested refinements are textbook profile-tightening: removing
reference targets that are not clinically meaningful for a breast
abnormality observation, and dropping a now-unnecessary intermediate
profile (`obf-*`) on `partOf`. Narrowing a `Reference(...)` target
list to a subset of the parent is a backward-compatible profile
operation per the FHIR profiling rules. The reporter has also offered
direct guidance on the CIMPL mechanism ("No profile" flag), so
implementation risk for the IG authors is minimal.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the three refinements, but rather than dropping the `obf-*`
intermediate profile on `partOf` outright, audit the `obf-*` framework
usage across the IG and decide as a single design action whether to
retain or remove the `obf-` profile layer for *all* reference-typed
elements (not just `partOf`). At the same time, document the chosen
target-type narrowing rationale in the profile's intro/markdown so
implementers understand why core resources are referenced directly.

#### Justification

The `obf-*` profiles come from the CIMPL Objective FHIR framework and
are typically applied uniformly across an IG to keep semantics
consistent. Removing them ad hoc on a single element risks an
inconsistent reference graph (some elements pointing to `obf-*`
profiles, others to core). A single coordinated decision ŌĆö either
"the IG uses `obf-*` for referenced resources" or "the IG references
core" ŌĆö produces a cleaner, more reviewable profile set than a
piecemeal change. The `subject`/`performer` narrowing is uncontroversial
and proceeds either way.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave the existing reference target lists in
place: `partOf` continues to reference `obf-MedicationStatement` /
`obf-Procedure`, and `subject` / `performer` retain the full set of
target types inherited from `Observation`.

#### Justification

A "decline" position is hard to defend here. The only colorable
argument is that the IG is built on the CIMPL `obf-*` framework and
that breaking that consistency for `partOf` (Disposition A's literal
form) introduces an asymmetry. But that argument is already addressed
by Disposition B without rejecting the underlying request, so a flat
decline is not justified. Including this disposition only for
completeness.

---

### Recommendation

**Recommended disposition:** A

The work group has already disposed of this ticket as Persuasive and
recorded that it would be addressed in the next ballot, so the
question for the WG is really *how* to act on it, not *whether*. The
three refinements are routine, backward-compatible profile-tightening
steps that improve clinical fidelity (`subject` to `Patient`,
`performer` to `Practitioner`) and remove an unnecessary profiling
layer on `partOf` that the reporter explicitly flagged as solvable via
CIMPL's "No profile" flag. Disposition B is a sensible discipline if
the WG wants to revisit `obf-*` usage IG-wide, and that conversation
can be picked up as a separate item; it should not gate this ticket.
Disposition C has no real basis. Recommend Accept As Requested and
fold any IG-wide `obf-*` audit into a follow-up work item.
