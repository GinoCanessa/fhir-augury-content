## Summary

This ticket — titled "Add sections - BC #90" — is a consolidated set of
ballot-style comments from Steven Nichols against the **US Breast Cancer
Radiology Reporting (FHIR)** implementation guide. The reporter raises
several gaps in the IG's current report model and asks the work group
to consider adding or restructuring content to cover: a patient
**history** section (symptoms, hormones, family history); **clinical
sign observations** (scar, implant, birthmark) that are not currently
represented; whether the model accommodates **double-reading**;
distinct structures for **screening vs. diagnostic** reporting; an
**intervention** section (e.g., biopsy); and the scope of **dose
reporting** (noting FHIR does not currently provide a dedicated dose
resource). The reporter also challenges the modeling approach itself,
asking that **ACR BI-RADS** be referenced (in whole or in relevant
sections) and pointing out that the IG does not appear to consider the
long-established **DICOM Breast Imaging Report Templates**.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned for this ticket key, so the request must be evaluated on its
own merits and against the current state of the IG.

## Proposed Dispositions

Because no repo briefings are available, the dispositions below speak

adopts an approach that requires source changes, the matching IG

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as an umbrella request and split it into an explicit,
trackable work plan against the US Breast Cancer Radiology Reporting
IG. Concretely:

1. **History section** — add a new `Composition.section` with a
   LOINC-bound code for "History" that allows
   `Observation` (symptoms, hormone use), `Condition` (relevant prior
   conditions), and `FamilyMemberHistory` entries.
2. **Clinical signs** — extend the IG's observation value sets (or add
   profiled `Observation` instances) for visible/palpable signs such
   as scar, implant, and birthmark, with an extensible binding so
   sites can add local concepts.
3. **Double-reading** — model the second read as an additional
   `Composition.attester` (mode `professional`) plus a profiled
   `DiagnosticReport.performer` slice, and document the workflow in
   the IG narrative.
4. **Screening vs. diagnostic** — introduce two distinct
   `Composition` profiles (or a single profile with a required
   `Composition.type` slice) so the report's intent is explicit, with
   conformance rules that enable/require different sections per type.
5. **Intervention section** — add a `Composition.section` permitting
   `Procedure` references (e.g., biopsy) and a profiled
   `Procedure.code` value set anchored to BI-RADS-aligned procedure
   concepts.
6. **Dose reporting** — add an optional `Composition.section` carrying
   `Observation` resources profiled for absorbed/effective dose,
   modality, and exposure metadata, until/unless a dedicated FHIR
   resource emerges.
7. **BI-RADS and DICOM SR alignment** — add an explicit IG narrative
   section mapping each `Composition.section` and key observation to
   the corresponding ACR BI-RADS lexicon entry and DICOM Breast
   Imaging Report Template row, with value-set bindings to BI-RADS
   codes where they exist.

File these as separate JIRA child tickets so each can be tracked,
balloted, and closed independently.

#### Justification

The reporter's bullets describe substantive gaps that align with how
mature radiology-reporting projects (ACR BI-RADS structured reporting,
DICOM SR Breast Imaging Report Templates) actually capture breast
imaging studies. Splitting the umbrella into discrete change requests
preserves the reporter's intent while letting the work group prioritise
and ballot each change individually rather than as one unreviewable
block. The proposed mechanisms (Composition sections, profiled
Observations, attester slices) are consistent with how other
diagnostic-reporting IGs compose their reports.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the US Breast Cancer Radiology Reporting IG's
own model to cover every gap, **realign the IG as a thin breast-imaging
profile layer on top of an existing structured-radiology baseline** and
explicitly reuse, rather than re-invent, the ACR BI-RADS lexicon and
DICOM Breast Imaging Report Templates:

1. Adopt (or normatively reference) a generic radiology
   `DiagnosticReport`/`Composition` profile (e.g., the work being done
   in Imaging Integration / DICOM-aligned radiology IGs) as the base.
2. Add only **breast-specific** profiles and value sets on top —
   BI-RADS assessment categories, breast composition, lesion
   descriptors — and bind them to the ACR-published code system.
3. Express **history**, **clinical signs**, **intervention**, and
   **double-reading** by profiling the existing radiology-base
   sections rather than defining new ones, so other body-site IGs can
   share the same machinery.
4. For **dose reporting**, defer to whatever emerges from the broader
   FHIR imaging community (e.g., a future dose Observation profile or
   the Imaging WG's work) and leave a single extension point in the
   IG.
5. Replace any narrative re-statement of BI-RADS / DICOM SR concepts
   with explicit mappings (`ConceptMap`) to those external standards.

#### Justification

The reporter's most pointed criticism is that the IG appears to model
breast reporting in isolation from ACR BI-RADS and DICOM SR — both of
which are already normative in clinical practice. Re-issuing the IG as
a breast-specific overlay on a shared radiology base avoids
permanently embedding breast-only design decisions into the FHIR
report model, reduces divergence from the standards radiologists
already use, and keeps cross-cutting concerns like dose reporting in
one place. The trade-off versus Disposition A is more upfront
restructuring and a dependency on a baseline IG that may itself be in
flight, but it produces a smaller, more maintainable surface area
long-term.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as a single change against US Breast Cancer
Radiology Reporting. Close it as **"Not Persuasive — split required"**
and request that the reporter (or sponsor) re-file each bullet as a
discrete, scoped change request with a concrete proposal and use case.
If the IG itself is no longer being actively maintained (the ticket
has been Triaged since 2019 and the spec has shown little movement),
escalate to the sponsoring work group to confirm IG status before any
of the bullets are pursued.

#### Justification

As written, the ticket is a list of high-level "consider" comments
without proposed structure or scope, which is hard to ballot or
disposition coherently — each bullet is large enough to be its own
work item (history modeling, intervention modeling, dose reporting,
double-reading workflow, BI-RADS alignment, DICOM SR alignment), and
they have different affected artifacts and different stakeholders. The
absence of any cross-referenced Jira tickets, Zulip activity, or
GitHub work in the four+ years since the ticket was filed suggests
that addressing it as one umbrella will not produce useful work
product. Declining-and-resplitting clears the queue and forces the
real conversations to happen in scoped tickets.

---

### Recommendation

**Recommended disposition:** B

The reporter's specific bullets are individually reasonable, but the
underlying problem he is naming is structural: the IG appears to model
breast radiology reporting in isolation from ACR BI-RADS and DICOM SR.
Disposition A treats each bullet as a local fix, which risks burning
work-group cycles re-implementing things that ACR and DICOM already
specify, and Disposition C punts on the substantive critique. Realigning
the IG as a breast-specific overlay on a structured-radiology base
(Disposition B) directly answers the modeling-approach concern, makes
the history/intervention/double-reading/dose questions tractable on
inherited machinery, and produces a smaller, more durable spec. The
work group should treat this as a starting point: confirm the IG's
maintenance status, identify the right radiology-base profile to
inherit from, and then re-file the reporter's specific bullets as
narrowly scoped child tickets against that restructured model.
