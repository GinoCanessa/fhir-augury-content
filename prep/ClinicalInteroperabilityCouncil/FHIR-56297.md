## Summary

The Da Vinci HRex Implementation Guide, in section 7.4 ("Exchange Security"),
contains the requirement "TLS SHALL be implemented as per RFC8705". Matt
Randall (relayed by Matthew Beermann) points out that RFC8705 — *"OAuth 2.0
Mutual-TLS Client Authentication and Certificate-Bound Access Tokens"* — does
not provide normative guidance for implementing TLS itself, so it is the
wrong reference for that sentence. A separate sentence later in the same
section, "When mutual TLS is used, it SHALL be done in accordance with
RFC8705", correctly cites RFC8705, which strongly suggests the first citation
is a copy-and-paste error. The reporter suggests RFC9325 ("Recommendations
for Secure Use of Transport Layer Security (TLS) and Datagram Transport Layer
Security (DTLS)") as the likely intended reference for the TLS-implementation
requirement. There are no cross-referenced Jira tickets; the only related
artifact is the originating Zulip thread in #Da Vinci HRex.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the HRex Implementation Guide, section 7.4 ("Exchange Security"), replace
the first occurrence of the citation in the sentence
*"TLS SHALL be implemented as per RFC8705"*
with a reference to **RFC9325** ("Recommendations for Secure Use of Transport
Layer Security (TLS) and Datagram Transport Layer Security (DTLS)"), so it
reads *"TLS SHALL be implemented as per RFC9325"*. Leave the later sentence
*"When mutual TLS is used, it SHALL be done in accordance with RFC8705"*
unchanged, since that citation is correct (RFC8705 normatively defines OAuth 2.0
mTLS client authentication and certificate-bound access tokens). Add RFC9325
to the IG's references list if not already present. Coordinate with Da Vinci
CRD (per @hbuitendijk's comment) to confirm no CRD-side narrative needs to be
adjusted in lock-step or that interim CRD guidance is unnecessary because the
fix is purely editorial.

#### Justification

This is the literal interpretation of the reporter's request and matches the
internal evidence in the spec itself: a later sentence in the same section
already cites RFC8705 correctly for mTLS, so the earlier "TLS SHALL be
implemented as per RFC8705" is almost certainly a copy/paste error. RFC9325 is
the current IETF BCP for TLS/DTLS usage (BCP 195) and is the canonical
normative target for a "TLS SHALL be implemented as per …" statement, so the
substitution makes the requirement actually testable and aligned with current
IETF guidance. No backwards-compatibility risk: implementers conforming to
RFC8705 mTLS today are not relying on RFC8705 for general TLS configuration,
so tightening the wrong citation does not change real conformance behavior.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the typo fix as the floor, but take the opportunity to align HRex's
TLS/mTLS conformance language with the broader FHIR security normative stack
rather than minting an HRex-specific citation. Concretely: in section 7.4,
replace *"TLS SHALL be implemented as per RFC8705"* with a reference to the
FHIR core Security page's TLS requirements (which already point to current
IETF TLS BCPs, including RFC9325) — for example, *"TLS SHALL be implemented in
accordance with the FHIR Security guidance for transport-layer security (which
incorporates RFC9325 / BCP 195 by reference)"*. Keep the existing mTLS sentence
referencing RFC8705. Optionally clarify that the mTLS sentence applies only
when the mTLS client-authentication option from §7.4 is selected, to remove
any remaining ambiguity about which RFC governs which layer. Coordinate the
change with Da Vinci CRD so any CRD narrative that quotes the HRex sentence is
updated in the same ballot cycle.

#### Justification

This addresses the same underlying defect (wrong RFC cited for "implementing
TLS") but avoids HRex independently pinning a specific TLS BCP, which can
drift as IETF updates BCP 195. Delegating to FHIR core Security keeps HRex
consistent with every other HL7 IG that layers on FHIR, reduces future
maintenance, and removes the risk of two HL7 specs citing different TLS BCPs.
Trade-off vs. Disposition A: it is a slightly larger editorial change and
introduces an indirection that some implementers may find less crisp than a
direct RFC9325 citation; it also requires confirming the FHIR core Security
page actually says what we want it to say at the time of publication.

---

### Disposition C: Decline

#### Proposal

Decline the change as **Not Persuasive — already addressed by existing
mechanism**, on the theory that the surrounding paragraph (which correctly
cites RFC8705 for mTLS) plus the standard expectation that conformant
deployments use current IETF TLS BCPs (RFC9325 / BCP 195) makes the
requirement adequately clear in context, and that fixing it requires a
republication of HRex purely for an editorial nit.

#### Justification

This is the defensible "do nothing" path: the ticket is a citation error in a
single sentence, and the surrounding text plus general industry practice
arguably make the actual TLS expectations unambiguous. Declining avoids a
republication cycle for an editorial fix and keeps the IG stable for downstream
guides (notably CRD) that pull HRex in transitively. The honest weakness is
that the sentence is *normative* ("SHALL"), so leaving an obviously wrong
citation in normative text undermines testability and invites exactly the kind
of implementer question that produced this ticket — which is why this option
is documented for completeness rather than recommended.

---

### Recommendation

**Recommended disposition:** A

The internal evidence is unusually clean: the same section already cites
RFC8705 correctly two sentences later, the reporter's analysis of RFC8705's
actual scope is correct, and there is no Zulip pushback — Lloyd McKenzie's
only response was "file a Jira so we can take this up on a call." That makes
this a textbook **Persuasive with Modification** style fix, and the literal
substitution to RFC9325 is the smallest correct change that restores a
testable normative requirement. Disposition B is attractive long-term but
expands the scope from "fix a typo" to "restructure HRex's TLS conformance
narrative," which is harder to justify mid-ballot for a single sentence.
Disposition C should be rejected because the offending text is normative SHALL
language; leaving an admittedly-wrong citation in a SHALL clause is exactly
the situation a Change Request process exists to fix. Per @hbuitendijk's
comment, the workgroup should confirm with Da Vinci CRD that this purely
editorial correction does not require a coordinated CRD update, but that
should be a coordination check, not a blocker for accepting the fix in HRex.
