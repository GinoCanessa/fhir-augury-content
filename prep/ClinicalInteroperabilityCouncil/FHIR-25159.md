## Summary

The reporter (Susan Matney) raises a naming concern against the US Breast
Cancer Radiology Reporting (BreastRad) FHIR IG: the artifact named
`breastrad-PatientHistory` carries a `breastrad-` prefix even though the
content has not been specialized beyond a generic "patient history" section
(the LOINC code in use is the standard patient-history code). The
recommendation is to drop the IG-specific prefix and rename the artifact to
`PatientHistory`, and to apply the same treatment consistently to the IG's
other sections that are not actually specialized. No related Jira tickets,
Zulip threads, or GitHub items were surfaced by the cross-reference index,
so the ticket stands on its own. The ticket already carries a resolution of
"Persuasive with Modification" with a recorded explanation that the
`breastrad-` prefix is appended automatically by the IG tooling in use at
the time, that the next ballot will use different tooling that does not
append the prefix automatically, and that PatientHistory will most likely be
further constrained — so reusing the unconstrained generic FHIR resource is
not viable.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Breast Cancer Radiology Reporting (BreastRad) IG, rename the
section profile / Composition.section artifact currently identified as
`breastrad-PatientHistory` to `PatientHistory` (drop the IG-specific
`breastrad-` prefix), and apply the same rename to every other BreastRad
section artifact that is not actually specialized beyond what the base FHIR
resource and the section's LOINC code already convey. Update the section
slice names, profile `id`/`url`, narrative, and any inbound references in
the IG (Composition profile section slices, IG resource list, examples,
narrative pages) to use the new ids.

#### Justification

The reporter's complaint is well-formed: when an artifact's only
"specialization" is the IG's own id prefix, the prefix becomes noise that
implies a specialization that does not exist. Dropping the prefix aligns
with FHIR naming guidance to keep ids descriptive of content rather than of
the publishing IG, and matches the LOINC-coded "patient history" semantics
already in use. Applying the rename uniformly across other unspecialized
sections prevents the inconsistency the reporter flagged ("Similar for all
sections.").

---

### Disposition B: Alternative Approach

#### Proposal

Keep the artifacts in place but address the underlying concern at the
tooling and constraint layer rather than by renaming individual artifacts:

1. Switch the IG to publication tooling that does **not** auto-prepend an
   IG-code prefix to artifact ids (the resolution note already commits to
   this for the next ballot).
2. For each section currently flagged as unspecialized, decide between
   (a) genuinely specializing the profile (cardinalities, must-support,
   bound value sets, fixed LOINC code, narrative requirements) so the
   `breastrad-`-prefixed id is semantically justified, or (b) removing the
   profile entirely from the IG and referencing the base FHIR resource /
   LOINC-coded section directly from the Composition profile.

#### Justification

The recorded resolution already notes that the prefix is a tooling artifact
and that PatientHistory will most likely be further constrained, so the
"correct" answer depends on whether each section will actually be
specialized. A blanket rename now (Disposition A) risks a second rename
after the next round of constraints lands. Letting the new tooling handle
the prefix and either truly specializing or removing each profile gives a
durable result and avoids churn for downstream implementers who have
already bound to the existing ids in the current ballot.

---

### Disposition C: Decline

#### Proposal

Decline the rename request as written. Treat the `breastrad-` prefix as an
intentional, IG-scoped namespace marker on artifact ids and leave the
existing ids in place. Document in the IG's "Conventions" page that all
BreastRad-published artifacts carry the `breastrad-` prefix regardless of
the degree of specialization, so the prefix conveys provenance, not
specialization.

#### Justification

Renaming published artifact ids is a breaking change for any implementer
who has already validated against the current ballot or referenced the
canonical urls. If the prefix is documented as a namespace rather than a
specialization claim, the reporter's concern (that the prefix implies
specialization that does not exist) is addressed by documentation rather
than by churn in the artifact ids. This is the lowest-cost path for
implementers but does not address the underlying tooling-coupling issue.

---

### Recommendation

**Recommended disposition:** B

The ticket is already resolved on Jira as "Persuasive with Modification",
and the recorded resolution text is essentially Disposition B in miniature:
the IG owners acknowledge the prefix is a tooling artifact, commit to new
tooling that won't auto-append it, and signal that PatientHistory will be
further constrained (so the generic FHIR resource cannot simply be reused).
Disposition A is responsive to the literal request but risks a second
rename once the new constraints land; Disposition C ignores the legitimate
"prefix implies specialization that doesn't exist" critique. Disposition B
captures the tooling fix the IG has already committed to, and forces a
per-section decision (truly specialize, or drop the profile and reference
base FHIR + LOINC directly), which is what the reporter was really driving
at when they said "Similar for all sections."
