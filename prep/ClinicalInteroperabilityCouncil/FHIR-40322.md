## Summary

The reporter, submitting on behalf of The Clinician team, provides feedback
from a proof-of-concept PROMs collection project for breast cancer patients
on the ZEDOC platform that uses the ICHOM Patient Centered Outcomes Measure
Set for Breast Cancer (FHIR) IG. Two concerns are raised about the
individual questionnaire questions:

1. Most date questions are modelled as a paired boolean ("is the date
   known?") followed by the actual date item, whereas the underlying ICHOM
   documentation expresses each as a single date question. The reporter
   suspects the boolean was added to drive `enableWhen` UI behaviour, and
   suggests this conditional-display logic should be left to the
   implementer rather than baked into the IG's Questionnaire definitions.
2. Several variable names in the IG drift from the ICHOM Breast Cancer data
   dictionary (e.g. the IG uses `EORTCQLQBR23_Q31` while the dictionary
   uses `EORTCQLQBR45_Q31`, and similar through Q53).

There are no cross-referenced Jira tickets, Zulip discussions, or GitHub
items associated with this ticket in the FHIR Augury index. The ticket has
already been triaged by the work group with a resolution of **Not
Persuasive**, with the published rationale that the source ICHOM set
requires the date question to permit either a specific date or a "date not
known" answer (motivating the boolean+date pattern), and a commitment to
fix the variable-name drift.

## Details

**Description:**

Feedback is based on a proof-of-concept PROMs collection project for breast
cancer patients on ZEDOC.

- Most of the date questions first request a boolean response, then
  followed by a request for the actual date
  - E.g. *Is the date if the histological diagnosis known? (If yes) →
    Indicate the initial date of histological diagnosis*
  - In the ICHOM documentation, this is only represented by one question
    e.g. *The initial date of histological diagnosis*
  - We noted that the boolean questions may have been added for
    navigational / UI purposes. For example, to enable the display of
    relevant questions depending on the information pulled in from an EHR.
    Instead of using this `enableWhen` workflow as part of the
    Questionnaire in the IG, it may be more desirable to leave this logic
    for the implementer to achieve on UI.
  - We think this approach was taken as the only way in FHIR Questionnaire
    resource to implement this behaviour is dependent on an answer to
    another question.
  - In our system we set up a Custom Behaviour to hide and conditionally
    reveal using a formula that evaluates a variable value (not a question)
    which we can set via integration.
- Some of the variable names in the ICHOM Breast Ca data dictionary are
  different from the variable names in the FHIR implementation guide
  - e.g. `EORTCQLQBR`**`23`**`_Q31` in the FHIR implementation guide is
    `EORTCQLQBR`**`45`**`_Q31` in the data dictionary (similar up to Q53)

Submitted on behalf of The Clinician team.

**Resolution Description (already recorded on the ticket):**

> The source ICHOM set for date questions requires either the date or an
> answer specifying that the date is not known. This is why the questions
> were modelled this way: either the date is unknown, or it is known and
> this is the specific date.
>
> Re: variable names — we'll update those, thank you!

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the ICHOM Breast Cancer (FHIR) IG so that:

1. Each date item in the IG's Questionnaire(s) is collapsed to a single
   `date` (or `dateTime`) question matching the underlying ICHOM source
   item — for example, the histological-diagnosis-date pair becomes one
   question "Initial date of histological diagnosis" with cardinality
   `0..1`. The companion boolean "Is the date known?" item and its
   `enableWhen` linkage are removed, and the convention that absence of an
   answer represents "not known" is documented in the Questionnaire's
   guidance text.
2. The `linkId` / variable names for the EORTC QLQ-BR23 items are
   re-aligned with the ICHOM Breast Cancer data dictionary (rename
   `EORTCQLQBR23_Q31` → `EORTCQLQBR45_Q31`, applying the same Q31..Q53
   shift to every affected item, and updating any cross-references in the
   IG narrative, Questionnaire, ValueSets, and example
   QuestionnaireResponses).

#### Justification

This option matches the reporter's request literally and reduces the
complexity of the published Questionnaires. Removing the boolean+date
pattern keeps the IG closer to the ICHOM source artifact, which is the
authority the IG normatively cites, and shifts UX-only concerns
(conditional display) out of the conformance artifact and into implementer
space — consistent with the FHIR principle that Questionnaire structure
should describe the data captured rather than presentation. Re-aligning the
EORTC variable names is uncontroversial: the reporter has identified a
straightforward labelling defect and the work group has already
acknowledged on the ticket that the rename is desired.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the boolean + date pairing in the published Questionnaires (so the
"date not known" state remains a first-class, structured answer rather than
a missing item), but make two narrower changes:

1. Replace today's two-question encoding with a single date item that
   carries an `ordinalValueSet` / answer-option extension permitting either
   a `date` or a coded "date not known" response — e.g. by attaching a
   `data-absent-reason` / "asked-but-unknown" answer option, or by using a
   `dataAbsentReason` extension on the `QuestionnaireResponse.item.answer`
   value. Document that implementers MAY render this as two UI controls
   while the wire format remains a single item.
2. Apply the EORTC QLQ-BR23 → BR45 variable-name re-alignment exactly as in
   Disposition A.

#### Justification

This honours the work group's stated rationale on the ticket — the source
ICHOM set explicitly allows "date not known" as a valid response — while
addressing the reporter's concern that two literal Questionnaire items are
heavier than the underlying single ICHOM data point. It avoids losing the
ability to distinguish *unanswered* from *known to be unknown*, which is
clinically meaningful for outcomes measurement and is something a
"missing-item-means-unknown" convention silently throws away. Trade-off vs.
A: more spec-side complexity (an extension or curated answer option per
date item) but better fidelity to ICHOM and to FHIR's general guidance to
distinguish "not asked" from "asked and unknown".

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** for the boolean+date concern, while keeping the
EORTC QLQ-BR23 → BR45 variable-name re-alignment as a separate, accepted
fix (tracked either by reopening this ticket scoped to the rename, or by
filing a follow-up ticket). The boolean+date encoding remains in the IG;
publish guidance in the Questionnaire's notes section explaining that the
boolean exists so the "date is not known" answer can be captured
explicitly, and that implementers are free to render the pair as a single
UI control.

#### Justification

This is the path the work group has already taken on the ticket: the
resolution text rejects the request to collapse the date items because the
ICHOM source allows "unknown" as an answer, and accepts the variable-name
fix. The boolean+date pattern is a conscious modelling choice that
preserves answer semantics; FHIR Questionnaire `enableWhen` is the
sanctioned mechanism for exactly this kind of conditional follow-up, and
moving the logic out of the conformance artifact would push variability
into every implementer. The reporter's UI concern is best addressed by
implementer-side rendering choices, not by removing structured answers
from the IG.

---

### Recommendation

**Recommended disposition:** C

The work group has already triaged this ticket with a resolution of *Not
Persuasive* on the modelling concern and a tacit acceptance of the
variable-name correction, and the published rationale (the ICHOM source
permits "date not known" as an answer) is sound. Disposition C aligns the
formal disposition with that resolution while making the variable-name fix
explicit and trackable. Disposition B is the strongest fallback if the
work group later decides the boolean+date pair is heavier than necessary —
it preserves the "asked but unknown" semantics that motivated the original
modelling — and should be revisited if implementer feedback continues to
push back on the two-question pattern. Disposition A is not recommended,
because collapsing the items to a single optional date silently conflates
"not asked" with "asked and unknown" and weakens the outcomes-measurement
fidelity the IG exists to support.
