## Summary

Eric Haas reports that the Da Vinci CDex Task Data Request profile
(`cdex-task-data-request`) added a `questionnaire-url` input parameter so
consumers can request data directly using a Questionnaire (i.e., a
data-request form). Because the HRex Task Data Request profile's
constraint **`tdr-1`** ("Must have exactly one input that is data-query
or data-code") does not allow a `questionnaire`-typed input slice, CDex
cannot legally derive its profile from HRex and was forced to recreate
the Task Data Request profile from scratch. The ticket proposes adding
a `questionnaire-url` (or `questionnaire`) input slice to the HRex Task
Data Request profile with cardinality `0..1` or `0..*` and relaxing
`tdr-1` so derived profiles (CDex first) can offer Questionnaire-based
data requests without rebuilding the profile.

This ticket is part of a broader thread of HRex Task Data Request
refactoring requests opened by the same reporter — see the
keyword-related tickets below — most of which target the same `tdr-*`
invariants and input-slice cardinalities. There are no formal
cross-references recorded in Augury (Jira/Zulip/GitHub), so the
"related" tickets below were surfaced by keyword similarity rather than
by an explicit link.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the HRex `Task Data Request` profile (StructureDefinition
`hrex-task-data-request`):

1. Add a new `Task.input` slice named `data-questionnaire` (or
   `data-questionnaire-url`) discriminated on `type.coding.code` (the
   same discriminator used by the existing `data-query` and `data-code`
   slices) with:
   - `Task.input.type` fixed to a new HRex code (e.g.,
     `data-questionnaire-url`) in the HRex `task-input-type` value set;
   - `Task.input.value[x]` constrained to `valueUrl` (consistent with
     the FHIR-46459 direction in CDex; URL accommodates canonical, FHIR
     Endpoint, and other endpoint types);
   - cardinality `0..*` (the more permissive of the two options the
     ticket suggests, to also cover the multi-questionnaire use case
     raised in FHIR-45981).
2. Modify invariant `tdr-1` from "Must have exactly one input that is
   data-query or data-code" to also accept the new
   `data-questionnaire(-url)` slice — i.e., "Must have at least one
   input that is data-query, data-code, or data-questionnaire-url".
3. Update HRex Task Data Request narrative to describe the new slice,
   when to use it vs. `data-query` / `data-code`, and the security /
   purpose-of-use considerations for fetching a Questionnaire by URL.
4. Remove the bespoke Task profile in CDex
   (`cdex-task-data-request`) and have CDex *derive* from the updated
   HRex profile, slicing further as needed.

#### Justification

This is exactly what the ticket asks for and removes the duplication
that motivated CDex to recreate the profile in the first place. HRex's
stated role is to be the common Task substrate for downstream Da Vinci
IGs; today `tdr-1` makes that promise unenforceable for any IG that
wants Questionnaire-driven data requests. Choosing `0..*` rather than
`0..1` aligns with the resolution of FHIR-45981 ("change required") on
the CDex side and avoids re-litigating the multiplicity question later.
Choosing `url` (not `canonical`) for the value type follows the
direction proposed in FHIR-46459 and makes the slice useful for
endpoints / bundles, not just Questionnaires hosted as canonical
artifacts.

The principal risk is that FHIR-37688 and FHIR-37689 — earlier requests
to relax the same family of `tdr-*` invariants — were both resolved as
**No Change**. Disposition A would partially reverse those decisions and
should be discussed against the original rationale (likely "keep HRex
Task strict; let derived profiles loosen via further slicing"), which
turns out to be unworkable given that derived profiles cannot
weaken a parent invariant.

---

### Disposition B: Alternative Approach

#### Proposal

Leave the `hrex-task-data-request` profile and its `tdr-1` invariant
unchanged in HRex, but publish a **new HRex companion profile** —
`hrex-task-data-request-questionnaire` — that derives from
`hrex-task-data-request` and:

1. Adds a `data-questionnaire-url` slice on `Task.input` typed as
   `url`, cardinality `0..*`;
2. *Replaces* the `tdr-1` invariant on the derived profile with a
   relaxed form that also accepts `data-questionnaire-url` (using a
   FHIRPath override + `severity = warning`/`error` as appropriate).

CDex (and any other downstream IG) then derives from this new HRex
companion profile when it needs Questionnaire-based requests, and
continues to derive from the strict `hrex-task-data-request` when it
does not. Optionally, formalize this as the HRex pattern: one strict
"core" profile and one "extended" profile per family of request inputs.

#### Justification

Keeps the strict HRex baseline that the workgroup has twice chosen to
preserve (FHIR-37688, FHIR-37689 — both **No Change**), while still
giving CDex (and future IGs) a conformant derivation point so they no
longer need to recreate the profile from scratch. This addresses the
*actual* harm reported in this ticket — duplicated profiles — without
forcing every existing HRex consumer to revisit their `tdr-1`
expectations. The trade-off is profile proliferation in HRex and a
slightly more complex story for implementers ("which Task Data Request
profile do I claim conformance to?"). It also relies on FHIR's profile
mechanics tolerating an invariant override on a derived profile, which
the workgroup should sanity-check.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressable via existing FHIR
mechanisms**. Recommend that CDex either (a) continue maintaining its
own `cdex-task-data-request` profile that does not derive from HRex, or
(b) carry the Questionnaire URL via an **extension** on `Task.input`
rather than as a new typed slice on the base HRex profile. Document
this guidance in the HRex narrative ("HRex Task Data Request is
intentionally limited to data-query / data-code; richer request shapes
should use extensions or be modelled in derived IGs").

#### Justification

Consistent with the disposition history of the closely-related
FHIR-37688 ("Change `query`/`code` slice max to `*`") and FHIR-37689
("Remove `tdr-3`"), both of which the WG resolved as **No Change**.
Holding the line keeps `hrex-task-data-request` semantically narrow and
predictable: every conforming Task is guaranteed to carry a
`data-query` or `data-code` input, which simplifies validation,
routing, and audit. The cost is real — CDex has to keep maintaining a
parallel profile — but that cost has already been borne, and any IG
needing richer inputs has the extension mechanism available.

The weakness of this disposition is that it does not actually solve
the problem the ticket reports (CDex *had* to fork the profile because
derivation was impossible) and it makes HRex less useful as a common
substrate.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the one most likely to survive WG discussion intact.
It directly fixes the harm in the ticket — CDex no longer has to
recreate the Task Data Request profile — without overturning the WG's
two prior **No Change** decisions on the same `tdr-*` invariants
(FHIR-37688, FHIR-37689). It also lets the WG defer the harder
question — "should `tdr-1` be permanently relaxed for everyone?" —
until there is a second non-CDex IG requesting the same relaxation.
Disposition A is the cleanest design but asks the WG to reverse those
prior decisions in one step; it is the right answer if the WG is now
ready to do so. Disposition C is defensible only if the WG is willing
to formally bless CDex's parallel profile and close this family of
tickets permanently — and the trajectory of FHIR-45981 ("change
required") suggests it is not.
