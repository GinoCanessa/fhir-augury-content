## Summary

The ticket, filed against the mCODE IG, objects to mCODE defining its own
value set for specimen-related concepts (body sites and fluids likely to
appear in tumor-marker or genomic testing) when, in the reporter's view,
existing HL7 terminology value sets already cover that space. The reporter
(Ron G. Parker) points to three candidate reusable value sets:
`v2-0070` (Source of Specimen), `v2-0487` (Specimen Type), and
`v3-SpecimenEntityType`. The request is either to reuse one of those value
sets directly or, if the mCODE value set is intentionally narrower
(scoped to tumor/genomic specimens), to rename it so that scope is
explicit rather than colliding with the existing general-purpose specimen
value sets. No cross-referenced Jira tickets, Zulip threads, or GitHub
items were found for this ticket, so the record stands on its own.

## Details

**Description:**

> If this valueset is specific for body sites and fluids likely to be
> used in tumor marker or genomic testing, then the name of the valueset
> should be updated to make that distinction. There are already
> valuesets for Specimen types.
>
> - https://terminology.hl7.org/ValueSet-v2-0070.html
> - https://terminology.hl7.org/ValueSet-v2-0487.html
> - https://terminology.hl7.org/ValueSet-v3-SpecimenEntityType.html
>
> (Comment 3 — imported by: Ron G. Parker)

**Resolution Description (Not Persuasive):**

> We looked into using ValueSet-v3-SpecimenEntityType as proposed, but
> that value set lacks an important code: that for "TUMOR", so we really
> can't use it as-is.
>
> The existing mCODE value set uses http://terminology.hl7.org/ValueSet/v2-0487
> but it removes elements that are completely inappropriate for a
> specimen from the human body. For example, environmental samples (e.g.,
> soil) and things unlikely to be related to cancer (e.g., intestinal
> worm). In short, we already use the v2-0487 code system but filter
> down to appropriate codes. The binding is extensible, just in case
> there is an unexpected type of sample.
>
> We therefore propose no change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rename the mCODE specimen-scope value set so the name signals its
oncology-specific intent (for example, rename the current
"specimen type" value set to something like
`mCODE Cancer-Related Specimen Type Value Set` with a matching
`name`/`title`/`description` and a URL that reflects the oncology
scope). Keep the current composition (a filtered subset of
`v2-0487` with extensible binding) intact. Optionally add a
`ValueSet.purpose` note explaining that the set is a human-specimen,
cancer-focused narrowing of `v2-0487` and pointing to
`v2-0070`, `v2-0487`, and `v3-SpecimenEntityType` as the broader
value sets that remain available for non-oncology use.

#### Justification

This directly responds to the reporter's concern, which is primarily
one of naming clarity: implementers should not have to open the
ValueSet to discover that it is a cancer-narrowed subset of
`v2-0487`. Renaming is a low-risk editorial change that preserves
the existing code content, existing bindings, and existing extensible
semantics (so the "TUMOR" use case that blocks adopting
`v3-SpecimenEntityType` is unaffected). It also aligns with HL7's
general preference for value-set names that convey scope, and it
avoids the implication that mCODE is silently "redefining" a generic
specimen value set.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the value set as-is but add explicit documentation — in the
ValueSet resource's `description`/`purpose` and in the mCODE IG
narrative that references it — that (a) the value set is a
deliberate, oncology-scoped filter over
`http://terminology.hl7.org/ValueSet/v2-0487`, (b)
`v3-SpecimenEntityType` was evaluated and rejected because it lacks
a `TUMOR` code, and (c) the binding is extensible so implementers may
use any `v2-0487` code (or another specimen code) when an oncology
specimen doesn't match the curated list. No change to the value set
URL, name, or composition.

#### Justification

This preserves stability of the mCODE artifact (no URL or name churn
for existing implementers, profiles, or IG references) while
addressing the underlying concern that the value set looks like it
competes with existing generic specimen value sets. Most of the
reporter's concern is informational — "there are already value sets
for specimen types" — and is fully answered by surfacing the
rationale already captured in the resolution. Trade-off vs. A: it
leaves the value set's name unchanged, so readers who skim only the
name (not the description) may still be surprised by the scope; but
it avoids breaking any downstream dependency on the existing
canonical URL.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** (the disposition currently recorded on
the ticket): make no changes to the mCODE specimen value set. The
ticket's literal ask — reuse an existing HL7 value set — is not
feasible because `v3-SpecimenEntityType` lacks the `TUMOR` code
required by mCODE, and `v2-0487` in unfiltered form includes codes
(environmental samples, parasitic organisms) that are inappropriate
for a human oncology specimen. The existing value set already reuses
the `v2-0487` code system and narrows it for cancer use, and the
binding is extensible so outlier cases are still expressible.

#### Justification

The workgroup has already evaluated each candidate value set against
mCODE's requirements and documented why reuse in the literal sense
would regress coverage or introduce irrelevant codes. The current
value set is itself a disciplined reuse of an existing HL7 code
system plus a curated filter, with an extensible binding to
accommodate edge cases — which is the pattern the community
generally recommends when an existing value set is *close* but not
exact. Declining also matches the Block-Vote-1 outcome (approved
8-0-0 on 2023-07-13).

---

### Recommendation

**Recommended disposition:** B

Disposition C (the current resolution) answers the feasibility half
of the reporter's concern but leaves the naming/clarity half
unresolved; reviewers who see "specimen type value set" next to
`v2-0487` can reasonably still ask the same question a year from now.
Disposition A fixes that, but at the cost of a rename — which, for a
published IG value set, is a non-trivial ripple through profiles,
bindings, and any downstream implementers that reference the
canonical URL. Disposition B preserves the work already done and the
Block-Vote-1 outcome while closing the gap the reporter actually
identified: it moves the rationale (cancer scope, `v2-0487` filter,
`v3-SpecimenEntityType` gap, extensible binding) from the Jira
resolution into the ValueSet and IG where future readers will
actually see it. If the workgroup judges the rename cost to be low
(few downstream dependencies), upgrading to Disposition A is a
reasonable escalation; otherwise B is the best balance of
responsiveness and stability.
