## Summary

The reporter (Ron G. Parker) requests that the value sets defined inside the
mCODE IG be anchored in HL7's Terminology server (THO,
[terminology.hl7.org](https://terminology.hl7.org)) instead of being published
solely from within the mCODE IG. The premise is that several mCODE value sets
are likely to be useful beyond mCODE itself, and hosting them in THO would make
them discoverable, reusable, and governed under HL7's cross-IG terminology
infrastructure. The ticket was originally dispositioned as "Considered for
Future Use" on 2024-06-27 (vote 5-0-4) on the grounds that other widely used
FHIR IGs (e.g., US Core) also keep their value sets local for consistency. That
resolution has since been reverted by Lynn Laakso and the ticket re-opened
post-STU 4.0.0 publication for fresh consideration. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were surfaced for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate mCODE-defined value sets that are reasonably reusable outside of an
oncology context to HL7 Terminology (THO), republishing them under
`http://terminology.hl7.org/ValueSet/...` canonicals and retiring (or
forwarding via `replacedBy`) the mCODE-local canonicals. Concretely:

1. Inventory every `ValueSet` resource currently authored in the mCODE IG and
   classify each as (a) clearly oncology-specific, (b) general-purpose
   clinical, or (c) terminology-server-friendly utility set.
2. For categories (b) and (c), submit Jira/UTG tickets to publish the value
   sets in THO and rebind mCODE profiles to the THO canonicals.
3. Update mCODE profile `binding.valueSet` references to the new THO
   canonicals and add release-notes guidance for implementers about the
   canonical change.
4. Coordinate with the Terminology Authority (Vocab WG / UTG) on the
   submission process, governance, and stewardship of the migrated sets.

#### Justification

THO is the designated home for cross-IG terminology artifacts. Any value set
the mCODE workgroup expects to be cited by other IGs (e.g., other CIC IGs,
research/quality measures, registries) is better governed centrally so that
implementers do not have to depend on the mCODE IG just to consume the value
set. This aligns with the long-running HL7 direction of consolidating
reusable terminology under THO and reduces the risk of duplicate, divergent
value sets being created downstream.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a *targeted* migration policy rather than a blanket move:

1. Keep oncology-specific value sets (those whose semantics are meaningful
   only in cancer care - e.g., cancer staging, disease-status, treatment
   intent) anchored in the mCODE IG, where authoring agility and oncology
   stewardship are highest.
2. Define and publish in the mCODE IG a documented criterion ("reusable
   beyond mCODE") and apply it once. Migrate only the value sets that meet
   the criterion to THO, leaving the rest in place.
3. For each migrated value set, provide a stable redirect / `replacedBy`
   note in mCODE so existing implementers are not silently broken.
4. Capture the policy in the mCODE IG's "Conformance" or "Terminology"
   section so future authors know when to route a new value set to THO vs.
   keep it local.

#### Justification

A wholesale migration is high-effort and risks breaking implementer code
that already binds to mCODE-canonical value sets, while the value of moving
sets that are *only* used by mCODE is low. A criterion-driven split
preserves backward compatibility for the bulk of mCODE bindings, addresses
the reporter's underlying need (make broadly-useful sets first-class THO
citizens), and keeps mCODE consistent with the precedent the original
2024-06-27 vote relied on (US Core's local-anchoring practice) where local
anchoring is appropriate.

---

### Disposition C: Decline

#### Proposal

Reaffirm the original 2024-06-27 disposition: keep mCODE value sets anchored
inside the mCODE IG. Document in the ticket that the workgroup considered
re-opening post-STU4 and chose to remain consistent with comparable
US-realm IGs (notably US Core), and close the ticket as
"Persuasive with Mod" only to the extent of recording the policy in the
IG's terminology guidance.

#### Justification

The 2024-06-27 vote (5-0-4) reflected a deliberate alignment with US Core
and base FHIR practice: implementers benefit from predictable canonical
locations and from terminology being versioned alongside the IG that uses
it. Moving value sets to THO introduces a second governance surface (UTG /
Vocab WG), a new release cadence, and additional coordination cost that has
historically been a drag on IG publication. Without concrete external
consumers asking to bind to mCODE value sets from outside mCODE, the
benefit case remains speculative and the cost case (bindings churn,
canonical breakage, dual-publication confusion) is concrete.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the reporter's actual concern - that some mCODE
value sets are useful beyond oncology - without committing the workgroup
to a disruptive blanket migration. It honors the original 2024-06-27 vote's
reasoning (alignment with US Core) where that reasoning still applies
(oncology-specific sets), while opening a clean, criterion-driven path to
THO for the subset that genuinely warrants cross-IG governance. It also
forces mCODE to articulate, in writing, when a new value set should be born
in THO vs. in the IG, which prevents this question from being re-litigated
ticket-by-ticket. Disposition A is the "right answer in principle" but is
operationally heavy and likely to break existing bindings without enough
external demand to justify the churn; Disposition C ignores the fact that
the workgroup deliberately re-opened the ticket post-STU 4.0.0, which
signals that simply reaffirming the prior vote is unlikely to satisfy the
reviewers who pushed for reconsideration.
