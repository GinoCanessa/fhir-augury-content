## Summary

This ticket reports a structural rendering / modeling defect in the
**US Breast Cancer Radiology Reporting (FHIR)** implementation guide.
The reporter (Susan Matney) observes that two elements,
`summarybiradcode` and `breastexamtechnique`, are presented as
**backbone elements** in the published artifact when they should
instead be presented as **extensions** on the underlying resource.
The ticket asks the Clinical Interoperability Council to correct the
modeling so the two concepts appear as extensions rather than nested
backbone structures.

The ticket has been **triaged with a Resolution of "Persuasive"** and
the recorded resolution text states the workgroup agreed this is a
tooling artifact that will be corrected in the next ballot. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the FHIR Augury services for this key, so the request
stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the request as **Persuasive** (matching the existing
resolution): in the US Breast Cancer Radiology Reporting (FHIR) IG,
re-model `summarybiradcode` and `breastexamtechnique` so they are
authored and rendered as **FHIR extensions** on the relevant profile
(e.g., the breast imaging DiagnosticReport / Observation profile they
were intended to attach to) rather than as backbone elements of a
nested differential. Concretely, define two `StructureDefinition`
resources of `type = Extension` (one per concept) bound to the
appropriate value set (BI-RADS assessment categories for
`summarybiradcode`; the breast exam technique value set for
`breastexamtechnique`), then reference them from the host profile via
`extension` slices instead of inlining a backbone element. Confirm the
publisher tooling renders them under the "Extensions" section of the
profile rather than as inline backbone structure rows, and republish in
the next ballot of the IG.

#### Justification

This matches the recorded workgroup resolution ("We agree … will be
corrected in the next ballot") and aligns with FHIR's core authoring
guidance: backbone elements are reserved for genuine new nested
structures within a resource definition, while domain-specific coded
concepts that decorate an existing resource are conventionally carried
as extensions. Re-modeling as extensions also makes the elements
reusable across other breast-imaging profiles and avoids the appearance
that the IG is defining new resource structure where none was intended.
The reporter's framing ("displayed as backbone element … should
display as extensions") indicates the underlying authoring intent was
already extensions, and the publisher rendering simply followed how the
elements were declared.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the issue as primarily a **profile-authoring fix** rather than a
new modeling decision: leave the conceptual content unchanged but
correct the profile differentials so that, instead of introducing
`summarybiradcode` / `breastexamtechnique` as new backbone children,
the profile slices the existing `extension` element of the host
resource and binds each slice to a small reusable `Extension`
StructureDefinition. As part of the same fix, add narrative in the IG
explaining why these are extensions (so future authors don't
re-introduce them as backbones), and add publisher QA checks (e.g.,
an `ig.ini` lint or a profile review checklist item) to catch
inadvertent backbone-style declarations on subsequent edits.

#### Justification

This reaches the same end state as Disposition A but frames the work
as a tooling/authoring hygiene fix rather than a content change, which
matches the workgroup's stated belief that this is a tooling artifact.
The added narrative and QA step reduce the likelihood of regression in
future ballots and give downstream implementers a clearer signal that
these elements are extensions by design. The trade-off is slightly
more work up front (process/QA additions) for a more durable fix.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave `summarybiradcode` and
`breastexamtechnique` rendered as backbone elements, on the grounds
that the existing structure is functional, downstream implementers can
still populate the values, and the cost of re-publishing the IG to fix
a presentation issue outweighs the benefit.

#### Justification

This disposition is included for completeness only. It is weak: the
reporter is a recognized domain expert (Susan Matney), the workgroup
has already agreed the current rendering is incorrect, and leaving
backbone-style elements where extensions were intended creates
long-term confusion for implementers about whether these are new
structural elements of the host resource or modular decorations.
Declining would also contradict the recorded "Persuasive" resolution.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly executes the change the workgroup has already
agreed to ("Persuasive" / "will be corrected in the next ballot") and
gets `summarybiradcode` and `breastexamtechnique` published as
extensions, which is how the reporter and the workgroup both believe
they were intended. Disposition B is a reasonable elaboration and its
QA additions could be folded into A as a follow-up, but treating this
ticket as anything more than a straightforward modeling correction
risks delaying a fix that the workgroup has already endorsed.
Disposition C is not defensible given the recorded agreement and the
clarity of the modeling guidance.
