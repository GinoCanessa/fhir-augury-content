## Summary

The reporter (Mitra Rocca, FDA) suggests that the mCODE specification leverage
the NCI Enterprise Vocabulary Services (EVS) — and, by extension, the Cancer
Data Standards Registry and Repository (caDSR) — when mCODE is used to support
cancer research. EVS maintains the NCI Thesaurus (NCIt) and related ontologies
that are widely used by FDA and cancer-research stakeholders, while caDSR
publishes Common Data Elements (CDEs) built as question/answer pairs grounded
in NCIt concepts. The request is essentially that mCODE either adopt NCI
terminology directly in its value sets or provide concept-map bridges from its
existing (predominantly SNOMED CT / LOINC / ICD) bindings out to NCIt so the
same mCODE-shaped data captured clinically can be reused for research without
manual recoding. Comment history shows the mCODE and NCI EVS teams met on
2023-06-13 and reached consensus to address this via concept maps from mCODE
value sets to NCIt terms, delivered separately from the STU3 publication
timeline (e.g., in a 3.x update); a prior "Considered for Future Use" vote
(8-0-0) was reverted on 2023-10-26 and the ticket re-opened post-STU3 so the
work could be tracked formally. There are no cross-referenced Jira tickets,
Zulip threads, or GitHub items on file for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt EVS/caDSR as a first-class secondary terminology surface for mCODE in a
post-STU3 update (target 3.1). Concretely:

- Publish a new HL7-maintained `ConceptMap` resource per priority mCODE value
  set, mapping each member code (predominantly SNOMED CT, LOINC, and ICD-O-3
  morphology/topography) to the equivalent NCIt concept code, with `source`
  set to the mCODE value set canonical and `target` set to the NCIt code
  system (`http://ncithesaurus.nci.nih.gov`, OID `2.16.840.1.113883.3.26.1.1`).
- Seed the priority list with the mCODE value sets already covered by NCI's
  existing STU2 mappings at <https://evs.nci.nih.gov/ftp1/mCode/> — primarily
  cancer disorder, primary cancer condition, secondary cancer condition,
  cancer-related medications, treatment intent, performance status, and the
  cancer staging value sets — and confirm/extend that list with the NCI EVS
  team (Renner, Rocca) per the 2023-06-13 meeting.
- Add a normative narrative section to the mCODE IG (e.g., a new "Use with
  Cancer Research and the NCI Enterprise Vocabulary Services" page under the
  "Background and Context" / "Conformance" group) describing the intended
  workflow: capture clinically in mCODE's primary bindings, translate to
  NCIt via the published `ConceptMap`s for downstream research use (clinical
  trials matching, FDA submissions, caDSR-aligned CDEs).
- Where caDSR Common Data Elements correspond directly to mCODE elements,
  cite the caDSR public ID alongside the element definition (informative
  mapping table), without changing the element's binding strength.

#### Justification

This is what the ticket and the 2023-06-13 mCODE/NCI meeting actually agreed
to: keep clinical capture in SNOMED CT/LOINC and bridge to NCIt with
`ConceptMap`s, delivered after STU3. Using `ConceptMap` rather than rebinding
preserves backward compatibility for existing mCODE implementers while giving
research consumers (FDA, NCI, trial-matching systems) a maintained, normative
bridge instead of relying on the informal STU2 NCI FTP drop. The approach
follows established FHIR terminology practice (separation of binding from
translation) and reuses content the NCI EVS team has signaled willingness to
own jointly, which is how the ticket was re-opened in the first place.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than HL7 publishing and maintaining the `ConceptMap`s inside the mCODE
IG, formalize the partnership but locate the mappings on the NCI side and
reference them from mCODE:

- NCI EVS continues to host the mCODE↔NCIt mappings (modernizing the existing
  `evs.nci.nih.gov/ftp1/mCode/` drop into a stable, versioned FHIR
  `ConceptMap` package published from EVS as the authoritative source).
- The mCODE IG adds an informative "External Terminology Bridges" page that
  links to the NCI-published `ConceptMap` package, names the supported
  source value sets, and pins a tested NCI release per mCODE version.
- Optionally, mCODE ships a thin CI test that resolves a sample of the NCI
  `ConceptMap` entries against the mCODE value sets at build time so the
  link does not silently rot between releases.

#### Justification

NCI EVS — not HL7 — is the terminology authority for NCIt and caDSR and is
already publishing mCODE-flavored mappings. Co-locating the mappings with
their source of truth avoids duplicate maintenance, removes a release-cycle
coupling between mCODE STU updates and NCIt's quarterly publication cadence,
and keeps HL7's mCODE IG focused on the clinical-care data model. It still
delivers the requested capability — a discoverable, FHIR-shaped bridge from
mCODE to NCIt — and is consistent with how other HL7 IGs reference externally
governed terminology (e.g., RxNorm, LOINC) without re-publishing it. The
trade-off vs. Disposition A is reduced HL7 control over the mappings'
versioning and review process; the build-time test is intended to mitigate
that.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** (or **Considered for Future Use**, restoring the
original 2023-07-13 resolution) on the grounds that mCODE's normative
bindings already use the terminologies in routine clinical use (SNOMED CT,
LOINC, ICD-O-3, RxNorm), that translation to NCIt for research is a
downstream concern outside the mCODE IG's scope, and that a stable mapping
already exists informally at <https://evs.nci.nih.gov/ftp1/mCode/> for parties
that need it. Document this decision in a release note and point implementers
at the NCI EVS resource.

#### Justification

mCODE is positioned as a clinical-care minimal common data set; layering
research-only terminology bridges into the IG arguably broadens its scope
and creates a maintenance obligation HL7 may not be best placed to carry, as
NCIt is governed and versioned by NCI on its own cadence. The original
ballot vote (8-0-0 for Considered for Future Use) reflects exactly this
hesitation. Implementers who need NCIt today already have a working asset on
the NCI FTP site, so declining does not block any current use case.

---

### Recommendation

**Recommended disposition:** B

The 2023-06-13 meeting and the 2023-10-26 re-open both make clear the
workgroup intends to deliver this capability — Disposition C contradicts that
direction and would have to re-litigate a decision the mCODE and NCI EVS
teams have already reached jointly. Between A and B, Disposition B better
reflects terminology-governance realities: NCI is the steward of NCIt and
caDSR, already publishes mCODE-aligned mappings, and updates that content on
its own quarterly cycle; asking the mCODE IG to vendor those mappings would
duplicate work and tie mCODE patch releases to NCIt errata. Disposition B
keeps the mCODE IG as the clinical anchor, gives implementers a single
authoritative `ConceptMap` package they can resolve at runtime, and still
satisfies the reporter's underlying request — that mCODE-shaped data be
reusable in cancer research via NCI vocabularies. If the workgroup judges
that an external dependency on NCI is too loose for a normative IG, it can
fall back to Disposition A and bring the mappings in-house for STU 3.1
without changing any of the analysis above.
