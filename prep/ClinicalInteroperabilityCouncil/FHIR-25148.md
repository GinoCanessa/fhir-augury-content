## Summary

The reporter (Susan Matney) raised a comment against the US Breast Cancer
Radiology Reporting FHIR IG ballot ("BC #99") asserting that the
`ImagingStudy` resource is missing fields needed to capture a radiology
result, an impression, or an "insertion." The ticket asks for a place
within `ImagingStudy` to record a coded result/impression. No related
Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-references or search, so this ticket appears to stand alone in
the current corpus.

The CIC has already triaged this with a tentative resolution of
**Not Persuasive**, on the grounds that result/impression content
belongs in the radiology report's findings/impressions sections (i.e.,
the `DiagnosticReport`/Composition narrative path used by the IG),
rather than in `ImagingStudy`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend `ImagingStudy` (or, more realistically for a US-realm IG, the
US Breast Cancer Radiology Reporting profile of `ImagingStudy`) with a
mechanism to carry coded results and impressions directly on the study.
Concretely, this would mean defining one of:

- A profile-level extension on `ImagingStudy` (e.g.,
  `imagingstudy-codedFinding`, `imagingstudy-codedImpression`) typed as
  `CodeableConcept`, bound to a value set such as RadLex / SNOMED CT
  finding concepts and to the BI-RADS impression/assessment value set
  used elsewhere in the IG; and
- Guidance (in the IG's ImagingStudy profile narrative) on when to use
  these extensions versus the report-level findings/impressions in the
  associated `DiagnosticReport`/`Observation` resources.

#### Justification

This is the literal request: it gives the reporter the place they say
is missing on `ImagingStudy`. Using extensions (rather than asking for
a base-spec change to `ImagingStudy`) keeps the change inside the IG's
authoring scope and avoids touching the `ImagingStudy` resource
definition owned by Imaging Integration. It also matches the standard
FHIR pattern of using profile-level extensions when a US-realm IG
needs domain-specific data on a base resource.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change `ImagingStudy`. Instead, document and tighten the
existing report-side path: the IG already uses
`DiagnosticReport.conclusion` / `DiagnosticReport.conclusionCode` and
report sections (findings, impressions) carried via `Observation`
resources referenced from the report. Address the underlying need by:

- Confirming the IG's `DiagnosticReport` profile binds
  `conclusionCode` to the BI-RADS assessment value set;
- Adding an `Observation`-based "coded finding" profile and an
  "impression" profile (or reusing existing US Core / IHE radiology
  profiles), referenced from the report's `result`/section entries; and
- Linking the radiology report back to the relevant `ImagingStudy` via
  `DiagnosticReport.imagingStudy` / `Observation.partOf` so consumers
  can navigate from the study to the coded result without that data
  being stored on `ImagingStudy` itself.

#### Justification

This matches the resolution the work group has already drafted on the
ticket: results, impressions, and codes belong in the findings /
impressions sections, not on `ImagingStudy`. `ImagingStudy` is
intentionally a metadata-about-the-acquisition resource (series,
instances, modality, performer), not a clinical-interpretation
resource. Putting coded findings on `ImagingStudy` would duplicate
data that already lives in `DiagnosticReport` / `Observation`, create
two competing places to look for the same fact, and diverge the IG
from the broader FHIR pattern for diagnostic reporting. This
alternative satisfies the reporter's underlying need (a place to
record coded results/impressions for breast imaging) without
overloading `ImagingStudy`.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** with the rationale already drafted on the
ticket: `ImagingStudy` is not the appropriate resource for clinical
findings, results, or impressions, and the IG's existing
`DiagnosticReport`-plus-`Observation` mechanism is the intended home
for that content. Close the ticket with a brief pointer to the
relevant IG sections (findings section, impressions section) and to
`DiagnosticReport.imagingStudy` as the linkage back to the study.

#### Justification

The base FHIR design separates *what was acquired* (`ImagingStudy`)
from *what was concluded* (`DiagnosticReport` and its referenced
`Observation`s). Adding result/impression fields to `ImagingStudy` —
even via extensions in this IG — would set a precedent at odds with
that separation and would not improve interoperability, because
consumers already know to look at the diagnostic report for
interpretive content. The ticket also provides no use case that the
existing report-side path cannot satisfy. There are no cross-references
or Zulip discussions in the corpus that argue for revisiting this
position.

---

### Recommendation

**Recommended disposition:** C

The ticket as written conflates two FHIR resources with deliberately
different responsibilities. `ImagingStudy` describes the acquisition
(modality, series, instances, performer); coded findings and
impressions belong on the diagnostic report and its `Observation`
children, which is exactly how the US Breast Cancer Radiology
Reporting IG is structured. The resolution text already on the ticket
makes that case correctly, and there is no community discussion
(Zulip, GitHub, or linked Jira) pushing back on it. Disposition B is a
reasonable secondary if the work group feels the IG documentation
needs to be tightened so that future readers don't make the same
assumption — but that is a documentation tweak rather than a reason to
re-open the modeling question. Disposition A should be avoided
because it would diverge from the resource's intent and create a
second, redundant home for interpretive content.
