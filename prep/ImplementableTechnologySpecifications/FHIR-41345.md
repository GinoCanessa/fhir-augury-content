## Summary

The ticket reports a naming error in the FHIR specification's registry of
HL7-defined OIDs for FHIR code systems. OID
`2.16.840.1.113883.4.642.4.1930` is currently published with the name
"Transport Status", but the reporter (Nick Radov) states it should be
"Transport Status Reason". Based on the FHIR R4B/R5 OID assignment scheme
for FHIR-defined CodeSystems, OID `…642.4.1930` corresponds to the
CodeSystem `transport-status-reason` (the reason a `Transport` resource has
its current status), which is a distinct artifact from the
`transport-status` CodeSystem (the status itself). The mismatch therefore
appears to be a clerical/labelling error in the published OID registry
table rather than a substantive change to either CodeSystem definition.

No cross-referenced Jira tickets were found by Augury, but a related
Zulip discussion in `committers > Checking the current release content`
(Oct 2022) independently called out that the spec ships two CodeSystems
— `transport-status-reason` and `transport-status` — that share the
duplicate `name: "TransportStatus"`. That observation is consistent with
this ticket's claim that the wrong name is being published against the
`transport-status-reason` OID, and suggests the underlying root cause is
that `transport-status-reason.name` was mistakenly authored (or
regenerated) as `TransportStatus` instead of `TransportStatusReason`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a technical correction to the FHIR Core specification's HL7-defined
OID registry so that OID `2.16.840.1.113883.4.642.4.1930` is published
with the name **"Transport Status Reason"** (and corresponding
machine-readable name `TransportStatusReason`) instead of "Transport
Status". Concretely, this is the entry for the
`http://hl7.org/fhir/CodeSystem/transport-status-reason` CodeSystem in
the OID-to-name mapping table that drives the
`terminologies-systems` page. As part of the same fix, update
`CodeSystem/transport-status-reason.name` from `TransportStatus` to
`TransportStatusReason` (with `title` "Transport Status Reason") so the
authored CodeSystem and the OID registry agree, eliminating the
duplicate-name collision with `CodeSystem/transport-status` that was
also flagged on Zulip.

#### Justification

The OID `…642.4.1930` is, by the registry's own assignment scheme, the
identifier for the `transport-status-reason` CodeSystem, not for
`transport-status`. Publishing it under the name "Transport Status"
makes the registry inconsistent with the actual CodeSystem it points at
and with the `Transport.statusReason` element it backs, and also
produces a duplicate `name` collision with `CodeSystem/transport-status`
— a defect Jose Costa Teixeira independently surfaced on the
`committers` Zulip stream and that Grahame Grieve agreed should not
exist. As a Technical Correction this is a non-breaking,
documentation-level fix (the OID and canonical URL do not change), and
it brings the published artifact into line with FHIR's general
convention that CodeSystem `name` is a unique, machine-friendly mirror
of `title`.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this as the trigger for a broader editorial sweep rather than a
single one-line edit. Specifically: (1) make the corrections from
Disposition A, and additionally (2) audit the full HL7-defined OID
registry table on `terminologies-systems` against the underlying
authored CodeSystems for `name`/`title`/canonical-URL agreement, using
the duplicate-name list Jose Costa Teixeira posted on Zulip as the
starting work list, and (3) add a publication-time check (in the FHIR
publisher / qa) that fails the build if two FHIR-Core CodeSystems share
a `name`, or if a registered OID's published name does not match its
target CodeSystem's `title`.

#### Justification

The Zulip audit showed FHIR-41345 is one symptom of a class of defects
(many duplicate CodeSystem names, several mismatches between the
published registry and the underlying artifact). Fixing the single OID
in isolation will recur the next time someone clones a CodeSystem or
re-runs the registry generator. Adding a build-time invariant is the
durable fix and is consistent with Grahame's "needs an editorial sweep"
comment in the same Zulip thread. The trade-off versus Disposition A is
scope: this expands a Medium technical-correction ticket into editorial
+ tooling work, and so warrants either being split into a follow-up
ticket or being scheduled against the next FHIR Core publication
window.

---

### Disposition C: Decline

#### Proposal

Close FHIR-41345 as **Not Persuasive — already addressed elsewhere**, on
the basis that the OID-to-name table on `terminologies-systems` is
generated from the CodeSystem resource's `title`/`name`, and any
discrepancy should be raised against the CodeSystem artifact (or the
publisher's registry-generation logic) rather than treated as a
standalone registry bug.

#### Justification

This is the weakest of the three options and is included only for
completeness. The ticket's underlying claim — that the published name
for OID `…642.4.1930` is wrong — is verifiable and was independently
confirmed on Zulip. Declining without action would leave a known
mislabelled OID in the published spec and would leave the duplicate
`name: TransportStatus` collision between two distinct CodeSystems in
place, which is hostile to implementers doing terminology lookups by
OID or by `name`. Decline would only be defensible if ITS concluded the
registry table is purely derived and that the fix must be re-routed to
a different ticket against the CodeSystem itself; even then, Augury
shows no such replacement ticket exists.

---

### Recommendation

**Recommended disposition:** A

This is a textbook Technical Correction: the ticket points at a single
mislabelled row in the HL7-defined OID registry, the correct value is
unambiguous from the OID's assignment scheme and from the existing
`transport-status-reason` CodeSystem, and the change is non-breaking
(the OID, canonical URL, and code values are untouched). The Zulip
discussion in `committers > Checking the current release content`
corroborates the defect and shows it has already been observed by the
core team, so ITS can act on it with confidence. Disposition B
(broader editorial sweep + a publisher invariant) is the right
*follow-up* and the workgroup should consider opening a separate
tracker for it, but it should not block this specific correction.
Disposition C is not defensible given the independent confirmation on
Zulip.
