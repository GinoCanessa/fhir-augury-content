## Summary

The ticket asks the FHIR Core specification to replace the existing — and
admittedly insufficient — JSON canonicalization rule on the `json.html`
page ("no whitespace other than single spaces … and order properties
alphabetically") with a normative reference to **IETF RFC 8785 (JSON
Canonicalization Scheme, JCS)**. Because RFC 8785 itself does not define
a URI for its canonicalization method, FHIR would adopt
`http://hl7.org/fhir/canonicalization/json` as the identifier for the
JCS-based canonicalization in a FHIR context, and would extend the
existing family of `#data`, `#document`, `#narrative`, `#static` (etc.)
URIs alongside it.

Reporter Grahame Grieve flags that, strictly, this is a **breaking
technical change** for how numbers are canonicalized, but argues it is
unavoidable: JSON signatures cannot be implemented reliably without it,
and known implementations are already doing this. Comments from Eric
Haas additionally request that the long-standing claim "These
canonicalization algorithms do not work for enveloped signatures. This
will be researched and addressed in a future release." be revisited at
the same time, since the Signature page already requires detached JWS
signatures, and provides proposed wording aligned with JCS.

The ticket sits inside a larger "what are we actually signing"
clean-up that the Security and Privacy WG and FHIR-I are coordinating;
related Jira items raised in the same Zulip review include
[FHIR-31566](https://jira.hl7.org/browse/FHIR-31566) (canonicalization
violates our own recommendations), [FHIR-41522](https://jira.hl7.org/browse/FHIR-41522)
(replace section with RFC 8785 — ITS), [FHIR-45023](https://jira.hl7.org/browse/FHIR-45023)
(canonicalization for Bundle that excludes `.signature` — ITS),
[FHIR-51133](https://jira.hl7.org/browse/FHIR-51133) (reorganize all
signature guidance — Sec), [FHIR-51112](https://jira.hl7.org/browse/FHIR-51112)
(misuse of "enveloping"), [FHIR-50934](https://jira.hl7.org/browse/FHIR-50934)
(allow exclusion of `Composition.section.text` from signatures), and
[FHIR-46788](https://jira.hl7.org/browse/FHIR-46788) (XML narrative
canonicalization). FHIR-51115 is the foundational JSON-canonicalization
piece on which several of the others depend.

## Details

**Description:**

The Json page says:

> This specification defines the following method for canonicalizing
> FHIR resources, when represented as JSON: "No whitespace other than
> single spaces in property values and in the xhtml in the Narrative +
> Order properties alphabetically"

But that's not actually sufficient. The section should be rewritten to
say:

> This specification adopts the method defined in RFC 8785 for JSON
> canonicalisation. Since RFC 8785 does not describe a formal URL to
> describe the method, this canonicalization method is identified in
> the context of FHIR by the URI
> `http://hl7.org/fhir/canonicalization/json`. The following additional
> canonicalization URIs are also defined: …

**Comments:**

- **grahamegrieve (2025-06-17):** Notes that this is, strictly speaking,
  a breaking technical change in how numbers are canonicalized, but
  that JSON signatures cannot be implemented reliably without it and
  all known implementations are already doing this — therefore the
  change can and must be made.

- **ehaas (2025-06-18):** Asks that the existing assertion *"These
  canonicalization algorithms do not work for enveloped signatures.
  This will be researched and addressed in a future release"* be
  reconsidered, because (1) it is unclear why JCS would not work for
  both detached-content and enveloped JWS signatures, and (2) the
  Signature page already specifies that signatures are *detached*.

- **ehaas (2025-06-18):** Proposes concrete replacement wording: *"This
  specification uses the IETF's JSON Canonicalization Scheme (JCS, RFC
  8785) to standardize JSON data for consistent processing. Since RFC
  8785 does not specify a formal URL to identify the canonicalization
  method, this specification adopts the URI
  http://hl7.org/fhir/canonicalization/json to represent the JCS method
  within the FHIR context. Additional JCS canonicalization URIs are
  defined as follows: …"*

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rewrite the JSON Canonicalization section on the `json.html` page of
FHIR Core (FHIR R6) to **normatively adopt IETF RFC 8785 (JCS)** as the
JSON canonicalization method for FHIR, substantially per Eric Haas's
proposed wording:

> This specification uses the IETF's JSON Canonicalization Scheme (JCS,
> RFC 8785) to standardize JSON data for consistent processing. Since
> RFC 8785 does not specify a formal URL to identify the
> canonicalization method, this specification adopts the URI
> `http://hl7.org/fhir/canonicalization/json` to represent the JCS
> method within the FHIR context. Additional JCS canonicalization URIs
> are defined as follows: …

Concretely:

1. Replace the current "no whitespace other than single spaces … order
   properties alphabetically" prose with a normative reference to RFC
   8785 and a statement that `http://hl7.org/fhir/canonicalization/json`
   identifies the FHIR-context binding of JCS.
2. Retain (and re-express in JCS terms) the existing variant URIs that
   are already documented elsewhere in the spec —
   `…/json#data`, `…/json#document`, `…/json#narrative`, `…/json#static`
   — clarifying for each what is included/excluded relative to the base
   JCS form (e.g., `#document` excludes `Bundle.signature` so that
   signatures over a Bundle can be verified).
3. Strike or rewrite the standing claim *"These canonicalization
   algorithms do not work for enveloped signatures. This will be
   researched and addressed in a future release"* to reflect (a) that
   the Signature page already mandates detached JWS signatures and
   (b) that JCS is fit for purpose for both detached and enveloped
   variants.
4. Add an explicit STU/breaking-change note acknowledging that strict
   number canonicalization changes vs. R5, and cross-link from
   `datatypes.html#Signature` and `datatypes.html#JSON`.

#### Justification

- This is the textbook fit: RFC 8785 *is* the IETF standardized JSON
  canonicalization scheme and is what existing FHIR signing
  implementations have converged on (per the worked Bundle/JWS example
  in the Signature Zulip thread).
- The current FHIR rule is demonstrably insufficient — the Signature
  thread documents Steve Swinsburg's signature failing
  interoperability tests precisely because the current rule does not
  pin number formatting, xhtml normalization in Narrative, or several
  other corner cases that JCS resolves.
- Reporter explicitly accepts the "breaking technical change" framing
  for number canonicalization and asserts that all known
  implementations are already doing this — so adoption is closer to
  *codifying current practice* than introducing churn.
- This change is the foundation that several other open tickets
  (FHIR-31566, FHIR-41522, FHIR-45023, FHIR-50934, FHIR-46788,
  FHIR-51133) depend on; deferring it blocks the Security and
  Privacy / FHIR-I joint cleanup that Grahame Grieve and John Moehrke
  have already scheduled.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt RFC 8785 **by reference for the *base* canonicalization** (per
Disposition A), but split the change into two coordinated artifacts so
the breaking number-canonicalization change does not silently land
inside an editorial-looking section update:

1. **In FHIR Core R6:** rewrite `json.html#canonical` to point at RFC
   8785 and assign `http://hl7.org/fhir/canonicalization/json` to JCS,
   *and* add a clearly-marked "Backwards-incompatibility Note" callout
   that enumerates exactly which prior R5 outputs would change (number
   formatting, xhtml whitespace handling in Narrative, removal of
   `Bundle.signature` for `#document`, etc.).
2. **As a coordinated package** with the related ITS/Sec/FHIR-I
   tickets (FHIR-31566, FHIR-41522, FHIR-45023, FHIR-46788, FHIR-50934,
   FHIR-51112, FHIR-51133), produce a single base-spec PR that lands
   the wording change *and* the new/clarified canonicalization URIs
   (`#data`, `#document`, `#narrative`, `#static`, plus any new
   `…/json.sig`-style form proposed in the Zulip thread for excluding
   either the entire `signature` or only `signature.data`). Optionally
   pre-publish a non-normative IG demonstrating signed-Bundle worked
   examples (per Eric Haas's R4-based JCS write-up).

#### Justification

- Same end state as A on the spec text, but addresses a real risk
  raised by the reporter: the change *is* technically breaking for
  number canonicalization, and a one-paragraph swap could be missed by
  implementers reading the diff as editorial.
- Bundling the change with the other open canonicalization/signature
  tickets matches the WG's already-agreed plan (Grahame Grieve in the
  "Signing and narrative" thread proposed exactly this: move the block
  to FHIR-I, dispose together, single PR, single resolution slot) and
  prevents the family of variant URIs from drifting.
- Worked examples are repeatedly cited in the Zulip threads as the
  thing that finally aligned implementer understanding (Grahame's JWS
  Bundle example, Eric's R4 JCS write-up, John's XML XAdES example);
  capturing them alongside the wording change reduces the chance of a
  third round of "what are we actually signing" questions.

---

### Disposition C: Decline

#### Proposal

Do **not** adopt RFC 8785 / JCS in the FHIR Core JSON Canonicalization
section. Leave the existing wording in place for R6 and re-route the
underlying need into either (a) a non-normative implementer guidance
page or (b) an IG owned by Security and Privacy.

#### Justification

The defensible (though, in this author's view, unpersuasive) basis
would be:

- The change is acknowledged to be a *breaking technical change* in
  number canonicalization. A Triaged "Medium" ticket against an STU
  → Normative trajectory may not be the right vehicle for a normative
  algorithm swap that materially changes byte-level outputs.
- A purely editorial replacement of the canonicalization rule risks
  being adopted by some implementers and not others if the
  Implementation Guides that actually exercise signatures (e.g.,
  CDex) do not also align — potentially creating *two* incompatible
  "JCS-ish" canonicalizations in the wild.
- Several adjacent decisions (whether `targetFormat` carries the
  canonicalization URI as a MIME parameter, whether a new `0..*`
  element is introduced, what `…/json.sig` should look like, how
  `DomainResource.text` with `additional` interacts with `#data`) are
  unresolved in Zulip; declining now would let those settle first.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is recommended. The substantive answer is unambiguous —
the current FHIR JSON canonicalization rule is insufficient for
real-world signatures, every active implementer in the Signature
thread has converged on JCS / RFC 8785, and the reporter (a Core
editor) is explicitly proposing the change. So Disposition C is not
defensible: declining would leave the spec in a state where the
documented canonicalization cannot be used to verify signatures
produced by tools that follow the rest of the Signature page.

Choosing B over A is a matter of *delivery hygiene*, not direction:
this ticket is one of a coordinated cluster (FHIR-31566, FHIR-41522,
FHIR-45023, FHIR-46788, FHIR-50934, FHIR-51112, FHIR-51133) that the
Security and Privacy and FHIR-I groups have already agreed to dispose
of as a block with a single base-spec PR and a worked example. Landing
FHIR-51115 alone, without the explicit breaking-change callout and
without the coordinated `#document` / `…/json.sig` URI clarifications,
would leave the spec internally inconsistent during the gap. If the
WG would prefer to take it in two steps anyway, Disposition A captures
the same wording outcome and is acceptable as a fallback.
