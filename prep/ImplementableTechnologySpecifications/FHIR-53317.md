## Summary

Grahame Grieve reports that the definition of `MessageDefinition.event[x]`
("Event code or link to the EventDefinition.") is inconsistent with the
declared types. The element is a choice between `Coding` and `uri`, but
neither the wording nor the `uri` type matches what the definition implies.
He proposes one of two corrections: (a) if the current intent stands —
that the link is to an `EventDefinition` resource — then the `uri` type
should be replaced with `canonical` (presumably `canonical(EventDefinition)`);
or (b) if the intent is broader than just `EventDefinition`, then the
choice should be extended to include `canonical(EventDefinition)` (in
addition to the existing types) and the definition text should be
clarified accordingly. No cross-referenced Jira tickets were surfaced
for this ticket; the cross-references are limited to the FHIR artifacts
`MessageDefinition.event` and `MessageDefinition.event[x]` themselves.

## Details

**Description:**

> The definition of MessageDefinition.event[x] is:

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the existing definition as authoritative ("Event code or link to
the EventDefinition.") and tighten `MessageDefinition.event[x]` to
`Coding | canonical(EventDefinition)`, replacing the current
`uri` choice. Concretely, in the FHIR Core spec
(`MessageDefinition` resource StructureDefinition):

- Change `event[x]` choice from `Coding | uri` to
  `Coding | canonical(EventDefinition)`.
- Keep cardinality 1..1.
- Leave the existing definition text largely intact (it already names
  `EventDefinition`), and add a short clarifying sentence: "When given
  as a `canonical`, the value SHALL be a canonical reference to an
  `EventDefinition` resource."
- Update the corresponding search parameter and any examples that
  currently use a non-EventDefinition URI.

#### Justification

- The current `uri` type does not match the stated definition; FHIR
  reserves `canonical` precisely for canonical references to conformance
  resources like `EventDefinition`.
- This aligns `MessageDefinition.event[x]` with patterns used elsewhere
  in the spec (e.g., `PlanDefinition.action.definition[x]` uses
  `canonical(...)` for analogous "link to definition" cases).
- The 2017–2018 history (committers > tracker-item GF#13848,
  fhir-svn / master notification) shows `uri` was chosen before
  `canonical` was a first-class FHIR data type; this ticket fixes a
  legacy modelling artifact, not a design decision that was ever
  re-affirmed.
- Note: this is a breaking change for any implementer relying on
  arbitrary (non-EventDefinition) `uri` values in `event[x]`. That
  risk is the main argument for Disposition B instead.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the current definition as too narrow and broaden the choice
rather than narrowing it. Change `MessageDefinition.event[x]` from
`Coding | uri` to `Coding | uri | canonical(EventDefinition)`, and
rewrite the definition to:

> "Event code (Coding) for ad-hoc events, a canonical reference to an
> EventDefinition for events defined as conformance resources, or a
> URI identifying the event when neither of the above applies."

Keep cardinality 1..1. Mark the `uri` form as the legacy/escape-hatch
option in the comments, and add invariants/examples showing the three
intended uses.

#### Justification

- Backwards compatible: existing `uri` content remains valid, so no
  implementer who relies on a plain URI is broken.
- Adds the missing `canonical(EventDefinition)` form that the original
  definition wording always implied but never actually offered as a
  type.
- Addresses the second branch Grahame himself proposes ("if the
  definition is too narrow, then the types should be extended by adding
  canonical(EventDefinition) and the definition should be clarified").
- The cost is increased modelling complexity (a 3-way choice) and
  validator/tooling updates to recognise the new `canonical` option;
  past Zulip discussion (`methodology > ConceptReference`) shows the
  community has long viewed `Coding | uri` `event[x]` as awkward, so
  some additional complexity is acceptable in exchange for cleaner
  semantics.

---

### Disposition C: Decline

#### Proposal

Decline the change as a "definition clarification only" outcome.
Update the description text on `MessageDefinition.event[x]` to make
clear that the `uri` form is intended to carry an identifier for the
event (not necessarily an `EventDefinition` canonical), and that
EventDefinition linkage is out of scope for this element. Make no
change to the type list. Rationale category: existing mechanism is
adequate; type change would be breaking without a demonstrated
implementer use case.

#### Justification

- `MessageDefinition.event[x]` has shipped as `Coding | uri` since
  early versions; changing the type list is a breaking change for any
  implementer whose `MessageDefinition` instances carry non-canonical
  URIs.
- The ticket presents no concrete implementer pain — it is a modelling
  consistency observation, not a defect tied to a real-world failure.
- A definition-only fix preserves wire-compatibility while removing
  the misleading "link to the EventDefinition" wording that drove the
  ticket.
- If demand for `canonical(EventDefinition)` materialises later, it
  can be added via an extension on `MessageDefinition` without
  changing the core type list.

---

### Recommendation

**Recommended disposition:** B

The ticket exposes a real defect: the current definition explicitly
mentions `EventDefinition`, yet the type list cannot actually carry a
canonical reference to one. Disposition C papers over that by editing
the description, which keeps the modelling inconsistency in place and
will keep generating tracker items. Disposition A is the cleanest
target state but is a breaking change to a long-standing element with
no migration story for implementers carrying plain URIs today.
Disposition B fixes the actual gap (no way to express
`canonical(EventDefinition)`) while preserving backwards compatibility,
matches the alternative branch the reporter himself offered, and is
consistent with how FHIR Core has historically resolved similar
"need a canonical here too" issues on choice elements (additive rather
than replacement). The ITS work group should treat the breaking-change
risk as the deciding factor and adopt B unless there is concrete
evidence that the `uri` form is unused in the wild — in which case
upgrading to A is preferable.
