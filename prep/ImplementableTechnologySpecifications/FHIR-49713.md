## Summary

FHIR-49713 is a Technical Correction filed by David Booth against the FHIR
Core RDF (Turtle) serialization page (`rdf.html`). In the predecessor
ticket [FHIR-37937](https://jira.hl7.org/browse/FHIR-37937) — "1 of 5
Usability improvements to RDF (Turtle): Shortening property names" — the
ITS / W3C HCLS RDF subgroup agreed to drop the datatype suffix from
choice-property names (e.g., use `fhir:value` instead of
`fhir:valuePeriod`) and instead carry the datatype on an explicit
`rdf:type` arc (`a fhir:Period ;`). The prose on `rdf.html` was updated
when FHIR-37937 was applied, but the **yellow Turtle template /
schema** illustrated on that same page was not — the example still
showed the old `fhir:effectiveDateTime`, `fhir:effectivePeriod`, etc.
form. FHIR-49713 asks that the yellow template be brought into
alignment with the agreed-upon shortened-property-with-type-arc syntax
so the example does not contradict the surrounding prose.

The ticket has already been resolved as Persuasive: pull request
[HL7/fhir#3528](https://github.com/HL7/fhir/pull/3528) ("Fixed RDF
Turtle template for issue FHIR-49713") was opened by David Booth on
2025-07-04 and merged by Grahame Grieve into `master` on 2025-09-21.
The PR is a minimal 4-addition / 4-deletion edit to `source/rdf.html`,
rewriting the four `effective[x]` lines in the `turtle-template`
section from `fhir:effectiveDateTime [ fhir:v "<dateTime>" ];` style to
`fhir:effective [ a fhir:dateTime ; fhir:v "<dateTime>" ];` style.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the fix already implemented and merged as
[HL7/fhir#3528](https://github.com/HL7/fhir/pull/3528): in
`source/rdf.html`, inside the `turtle-template` section
(`<a name="turtle-template"></a>Turtle Template for Resources`),
replace the four `effective[x]` example lines

```
fhir:effectiveDateTime [ fhir:v "<dateTime>" ];
fhir:effectivePeriod [ Period ];
fhir:effectiveTiming [ Timing ];
fhir:effectiveInstant [ fhir:v "<instant>" ];
```

with the shortened-property + explicit-type-arc form agreed in
FHIR-37937:

```
fhir:effective [ a fhir:dateTime ; fhir:v "<dateTime>" ];
fhir:effective [ a fhir:Period ; Period ];
fhir:effective [ a fhir:Timing ; Timing ];
fhir:effective [ a fhir:instant ; fhir:v "<instant>" ];
```

Mark the ticket Applied / Closed with the merge SHA of PR #3528 and
record the resolution as Persuasive (already set).

#### Justification

This is the literal request in the ticket and it has already been
implemented, reviewed, and merged to `master`. The change is
intrinsically minimal (one file, 4 ± 4 lines, no anchor rename, no
generated-area touches) and brings the example into agreement with the
surrounding prose, which currently contradicts it. Accepting the
request closes a documented inconsistency in a Normative-track
serialization specification, removes a footgun for implementers reading
the yellow template as authoritative, and discharges an outstanding
follow-up that the ITS RDF subgroup explicitly tracked from
FHIR-37937. No new design decisions are required — the convention was
already accepted in FHIR-37937; this ticket only fixes the missed
example. The FhirCore briefing confirms `source/rdf.html` is
hand-authored content, so no publisher / generation gymnastics are
involved.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the fix in PR #3528 *and* extend the same audit to the rest of
`source/rdf.html` (and any sibling RDF documentation pages, e.g.
sections that include a `turtle-template`-style yellow example) to
catch any other choice-element examples — `value[x]`,
`onset[x]`, `deceased[x]`, etc. — that may still be using the
pre-FHIR-37937 `fhir:<element><Type>` form. File any additional
hits as a follow-up technical-correction ticket (or roll them into a
single PR), and add a short note at the top of the
`turtle-template` section warning future editors to keep the example in
sync with the type-arc convention.

#### Justification

The bug FHIR-49713 documents is "we updated the prose but forgot to
update the example." That same class of oversight is likely to apply to
any other choice-element example on the page. PR #3528 fixes only the
`effective[x]` case that the reporter happened to notice. A wider sweep
costs little additional reviewer time, but materially reduces the risk
that another implementer files the same ticket against a different
choice element next ballot cycle. The briefing's gotcha that the
`turtle-template` anchor must remain stable is satisfied because this
disposition still edits in place and does not rename anchors.
Trade-off vs. Disposition A: slightly larger PR scope and an
additional commit, but the change remains documentation-only on
`source/rdf.html` and stays inside the same authoring root.

---

### Disposition C: Decline

#### Proposal

Close FHIR-49713 as Not Persuasive on the grounds that the
yellow Turtle template is an illustrative example and the surrounding
prose already describes the agreed FHIR-37937 syntax, so an
implementer following the prose will get the right answer without the
template change.

#### Justification

This disposition is **not** defensible here and is included only for
completeness. The ticket reporter is the original author of the
FHIR-37937 convention; the inconsistency is real, demonstrably visible
on a Normative-track page, and the fix is a 4-line edit with zero
behavioural impact (RDF-serialization-only, no JSON/XML/model
impact — explicitly noted in FHIR-37937). The PR is already merged.
Reverting or declining at this point would knowingly leave a
self-contradicting normative example in the published spec, which is
the exact failure mode HL7 ITS uses Technical Correction tickets to
prevent.

---

### Recommendation

**Recommended disposition:** A

The ticket is a textbook Technical Correction: a small, scope-bounded
documentation fix to a single hand-authored `source/*.html` page,
already implemented exactly as described by the reporter, already
reviewed, and already merged into `master` by the FHIR product
director. The originating design decision (FHIR-37937) was settled
years ago in the ITS RDF subgroup and is Published; this ticket only
discharges a known follow-up. Disposition B (broader audit) is a
reasonable belt-and-braces add-on, but it is properly a *separate*
technical-correction ticket — bundling it risks delaying the closure
of a fix that is already in `master`. Disposition C is not credible
given the merged PR. The workgroup should ratify Disposition A,
record PR #3528's merge SHA on the ticket, and (optionally) open a
follow-up to perform the wider yellow-template audit suggested in
Disposition B.
