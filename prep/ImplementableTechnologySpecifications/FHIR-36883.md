## Summary

The reporter ([Mohamed Omar](https://jira.hl7.org/browse/FHIR-36883)) observes
that the legend explaining the icons / flags shown alongside element names in
the FHIR specification (e.g., `R!`, `C?`, the lock/summary icon, etc.) appears
to be incomplete on the JSON page when compared against the XML page. In
triage, [mckenzie](https://jira.hl7.org/browse/FHIR-36883) confirmed that
`R!` and the lock icon are documented appropriately, but the `C?` flag — used
to mark elements that carry conditional constraints / invariants — is **not**
defined in the legend block on the JSON page. The ticket was transferred from
FHIR Infrastructure to **Implementable Technology Specifications (ITS)**
because ITS owns the JSON page documentation. The fix is purely editorial:
add a `C?` entry (and any other missing flag entries) to the JSON page
legend so that it matches the legend rendered on the XML page.

This request sits in a small family of similar legend-documentation tickets:
[FHIR-3516](https://jira.hl7.org/browse/FHIR-3516) ("Document what the 'lock'
icon means in the xml comments", resolved Persuasive) added the original
lock-icon explanation to the formats page, and
[FHIR-14520](https://jira.hl7.org/browse/FHIR-14520) ("Typo in legend 'Key to
Type Icons and Flags'", resolved Persuasive) corrected the `<D>` (Draft) flag
description in the same legend. FHIR-36883 is the next iteration of that same
clean-up — synchronising the JSON-page legend with the XML-page legend so the
two representation pages give equivalent guidance.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **"Key to Type Icons and Flags"** legend on the JSON
representation page (`json.html`) of the FHIR Core specification to add
the missing **`C?`** entry (and audit the same legend on `xml.html` and
the formats overview page for parity). The `C?` entry should describe it
as: *"This element has additional conditions / constraints (invariants)
defined elsewhere — see the element's invariant table or the 'Constraints'
section of the resource page."* Verify that every flag rendered by the
spec build into the JSON page tree (`∑`, `?!`, `I`, `C`, `C?`, `TU`,
`N`, `D`, `R!`, lock, etc.) is defined in the legend, and add any other
entries found to be missing. No resource, profile, structure, or
machine-readable artifact changes are required — this is an editorial
fix to the prose / legend block authored under the JSON-page source in
`HL7/fhir` (ITS-owned).

#### Justification

The reporter's observation is factually correct and was confirmed by an
ITS reviewer (mckenzie) at triage: `C?` is used in the rendered tree on
the JSON page but is absent from that page's legend. Adding the entry
costs nothing in terms of spec content or implementer impact, removes a
real documentation gap, and is fully consistent with the precedent set
by [FHIR-3516](https://jira.hl7.org/browse/FHIR-3516) (added the lock
icon definition) and [FHIR-14520](https://jira.hl7.org/browse/FHIR-14520)
(corrected the `<D>` legend entry), both of which ITS / FHIR-I accepted
as Persuasive editorial corrections. The change strictly improves the
spec's internal consistency between the XML and JSON representation
pages, which is exactly the kind of cross-format parity ITS is chartered
to maintain.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than patching the JSON-page legend in isolation, **factor the
legend out into a single shared include** that is rendered identically
on `xml.html`, `json.html`, and any future representation page (e.g.,
RDF/Turtle). The include would be the canonical source of truth for all
type-icon / flag definitions (`∑`, `?!`, `I`, `C`, `C?`, `TU`, `N`,
`D`, `R!`, lock, etc.). As part of this work, populate the include with
the currently-missing `C?` entry, fixing the original ticket and
preventing the same drift from recurring. This is still an editorial
change, but it is structural: it eliminates the class of bug rather
than the single instance.

#### Justification

The history of the legend (FHIR-3516, FHIR-14520, and now FHIR-36883)
shows the same family of defects recurring every few years because the
legend exists in two near-duplicates that drift apart. A shared include
is the standard authoring fix and matches how other cross-page boilerplate
is handled in `HL7/fhir`. The trade-off versus Disposition A is that B
requires touching the publication tooling / templating layer, has a wider
review surface, and is harder to backport to in-flight ballot
publications, where Disposition A is a few lines of edited prose. Pick B
only if the workgroup wants to invest publishing-pipeline effort to stop
the bleeding permanently.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the underlying
information is already discoverable: every constraint/invariant flagged
with `C?` resolves to an entry in the per-element invariant table on the
resource page, and the meaning of constraint flags is described in the
formats overview / profiling pages. The legend on the JSON page is a
quick-reference and is not contractually required to enumerate every
glyph the renderer emits.

#### Justification

This position is technically defensible — readers who are unfamiliar
with `C?` can usually trace its meaning from context — but it is hard to
sustain given that (a) the XML page already documents the equivalent
glyphs, so declining creates an asymmetric, lower-quality user
experience on the JSON page; (b) the workgroup has historically
accepted nearly identical editorial corrections (FHIR-3516, FHIR-14520);
and (c) the cost of fixing is trivial. Decline is included for
completeness but is the weakest of the three options.

---

### Recommendation

**Recommended disposition:** A

The reporter's observation is correct, the in-triage ITS comment from
mckenzie already concedes the fix is needed and accepts ownership, and
the precedent in FHIR-3516 and FHIR-14520 is unambiguous: missing or
incorrect entries in this legend are routinely accepted as Persuasive
editorial changes. Disposition B (shared include) is technically
appealing and would prevent recurrence, but it is a substantially larger
change to the publishing pipeline and should be tracked as its own
infrastructure ticket rather than gating this fix. Disposition C is
inconsistent with both the workgroup's prior handling of identical
tickets and with mckenzie's own triage note. Accept as requested, add
the `C?` entry (plus any other missing glyphs surfaced by a quick audit)
to the JSON page legend, and consider opening a follow-up ticket against
ITS / publishing tooling to pursue the shared-include refactor described
in Disposition B.
