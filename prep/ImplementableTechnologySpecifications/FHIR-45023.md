## Summary

The ticket asks ITS to define a canonicalization algorithm — for both the
JSON and XML wire formats — that can be used to compute and verify a
digital signature carried in `Bundle.signature`. Today neither
`json.html#canonical` nor `xml.html#canonical` supports an "enveloped"
signature use: both pages explicitly note that the existing
canonicalizations "do not work for enveloped signatures" and defer the
problem to a future release. Because of that gap, a Bundle that carries
its own signature has no portable way to compute the bytes that the
signature actually covers. The reporter (John Moehrke) frames the need
as a generic one — any resource that embeds a `Signature` element and
signs over itself (Bundle today, Provenance plausibly tomorrow) — and
flags that the Bundle "document" canonicalization needs the same
treatment.

The ticket sits at the centre of the broader Bundle/Provenance signature
work currently being driven through FHIR-I and Security & Privacy. The
cross-references show no other Jira tickets directly linking
FHIR-45023, but the related Zulip threads show it is being treated as
the canonicalization prerequisite for the larger signature rewrite for
R6, alongside (separately tracked) tickets such as **FHIR-51143** (the
overarching signature rework), **FHIR-51115** (point JSON
canonicalization at RFC 8785), **FHIR-50934** (canonicalization handling
of `text.status = additional` narratives), **FHIR-31566**, **FHIR-41522**,
**FHIR-46788**, **FHIR-47197**, **FHIR-51099** (cardinality of
`Bundle.signature`) and **FHIR-51112** (use of "enveloping"). Those
tickets are *not* listed as cross-references on this ticket but are the
context the ITS workgroup needs to dispose of FHIR-45023 coherently.

The unresolved design question that the workgroup must settle as part of
the disposition is **what to exclude** from the signed bytes:

- John Moehrke's position: exclude the entire `Signature` element
  (`Bundle.signature`) so none of the Signature datatype's metadata is
  covered by the signature.
- Grahame Grieve's draft (`build.fhir.org/branches/2025-07-gg-digital-signatures1/signatures.html`,
  via Zulip): exclude only `Signature.data` so `.type`, `.when`,
  `.who`, and `.onBehalfOf` are signed and thus tamper-evident.

Either way, ITS owns the canonicalization page change.

## Details

**Description:**

> Both XML and JSON do not have a canonicalization that can be used with
> a `Bundle.signature` digital signature. There is a need for a
> canonicalization that excludes the `.signature` element so that the
> signature across the bundle can be calculated prior to inserting that
> signature.
>
> This is needed for any kind of Bundle, but might also need to be added
> to the `#document` canonicalization.

## Proposed Dispositions

> core specification (`HL7/fhir`) based on the URLs called out in the

> `xml.html#canonical`, `signatures.html`), but the workgroup should

### Disposition A: Accept As Requested

#### Proposal

Define **two new canonicalization URIs** in FHIR core — one for JSON,
one for XML — whose only difference from the existing
`json.html#canonical` / `xml.html#canonical` URIs is that they
**exclude the entire `Signature` element value** when the signature
element is the embedded signature for the resource being canonicalized
(i.e., `Bundle.signature` for Bundle today, and the equivalent path
for any other resource that embeds a self-signature, such as
`Provenance.signature`). Specifically:

- Add the JSON variant on `json.html#canonical` (likely
  `http://hl7.org/fhir/canonicalization/json+embedded-signature` or
  similar), defined as: apply the existing JSON canonicalization
  (RFC 8785-aligned per FHIR-51115), but with the named embedded
  Signature element omitted prior to canonicalization.
- Add the matching XML variant on `xml.html#canonical`.
- Update `signatures.html` to (a) state that these are the
  canonicalizations to be used when a signature is to be embedded in
  the resource being signed, (b) clarify that they apply to Bundle and
  to other resources with embedded `Signature` elements, and (c)
  remove the existing "do not work for enveloped signatures … will be
  researched and addressed in a future release" disclaimer on the
  json/xml canonicalization pages.
- Extend the Bundle `#document` canonicalization to also exclude
  `Bundle.signature` so signed Documents work end-to-end without a
  separate ceremony.

#### Justification

This is exactly what the reporter asked for and what implementers
already do by hand (see davinci-cdex's worked Python example, which
strips `.signature` and `.meta` before signing). It closes the
publicly-acknowledged gap in `json.html#canonical` /
`xml.html#canonical`. It also generalises cleanly to
`Provenance.signature`, which the reporter explicitly called out.
Excluding the *entire* Signature element matches Moehrke's stated
position and avoids the operational hazard Grahame raised in the
"What to sign?" thread that `Signature.when` "must be affixed before
signing" (a one-tick race that breaks signature verification across
parties); with the whole element excluded, no field of the embedded
Signature contributes to the signed bytes and that race goes away.

### Disposition B: Alternative Approach

#### Proposal

Define **one new canonicalization URI per format** (JSON, XML) that
excludes only **`Signature.data`** (and any other purely-derived
fields such as `Signature.sigFormat` if the wg agrees), leaving the
Signature metadata (`type`, `when`, `who`, `onBehalfOf`) inside the
signed bytes. Apply the same canonicalization recursively to any
embedded Signature in the resource (Bundle, Provenance). Update
`signatures.html` to specify the order of operations: populate every
Signature field except `.data`, canonicalize, sign, then write the
signature bytes back into `.data`. Keep the `Bundle` `#document`
canonicalization aligned with this rule.

This is the approach already prototyped by Grahame on
`build.fhir.org/branches/2025-07-gg-digital-signatures1/signatures.html`.

#### Justification

It satisfies the practical need (a stable canonicalization that works
for embedded signatures, including in `#document`) while keeping the
human-readable Signature metadata cryptographically protected against
tampering. That gives applications a defensible reason to rely on
`Bundle.signature.who`/`.when`/`.type` without parsing the inner
JWS/CMS — which the "What to sign?" thread surfaced as a real
implementer expectation. The cost is the operational discipline
Grahame and Gino discussed: callers must populate `.when`, `.who`,
etc. *before* signing, and tooling must enforce that, otherwise the
signature breaks. The justification therefore must address that
gotcha by also requiring the spec text to call out the strict ordering
and to state that `.data` MUST be the only Signature field absent at
signing time.

### Disposition C: Decline

#### Proposal

Decline to add a Bundle-/embedded-signature canonicalization in core.
Instead, document on `signatures.html` that implementations needing a
signed Bundle MUST use a non-embedded mechanism — a detached signature
delivered out-of-band, a JWS over the FHIR document at the transport
layer, or a `Provenance` resource that carries the signature *for* the
Bundle rather than *in* it. As a defensive companion change, mark
`Bundle.signature` as deprecated for the next release on the grounds
that the past decade of FHIR has not produced an interoperable use of
it, and that the absence of a defined canonicalization is, per the
reporter's own observation, "strong evidence that there has been no
implementation".

#### Justification

If the workgroup agrees with Lloyd's and John's repeated observation
that virtually no one has actually shipped `Bundle.signature` and that
the people who have (e.g., davinci-cdex) had to invent their own
canonicalization rules anyway, then standardising a canonicalization
*now* — only weeks before targeting `Bundle.signature` for normative
in R6 — risks codifying a mechanism that no implementer needs.
Provenance-based signing is the direction the Security & Privacy
thread is converging on for any "real" multi-party / counter-signature
scenario, and a detached JWS is what most modern profiles
(IHE DSG, davinci-cdex JWS profile, JAdES) actually use in practice.

### Recommendation

**Recommended disposition:** **B**

Disposition B (exclude only `Signature.data`) is the right answer
because (1) it's the only path that keeps `Bundle.signature` viable
for normative R6 — Decline (C) is the cleanest engineering answer but
politically very costly given Grahame's stated R6 normative goal and
the active draft on `build.fhir.org`; (2) it matches the implementation
already prototyped and exercised end-to-end with the IG publisher in
the Signature Zulip thread, so its operational cost is known rather
than speculative; and (3) it preserves the property implementers
actually want from `Bundle.signature.who`/`.when`/`.type` — that
those values are tamper-evident — which Disposition A throws away.

The "must populate `.when`/`.who` before signing" gotcha that John
flagged is a real concern, but it is bounded: it lives entirely inside
a single signing ceremony executed by one party, and the spec can
require it in normative language. By contrast, Disposition A
permanently gives up the integrity of `Signature.who` for everyone.

Disposition A is a defensible second choice if the workgroup decides
the operational simplicity of "exclude the whole element" outweighs
the loss of metadata integrity, particularly because it generalises to
`Provenance.signature` without further thought. The workgroup should
explicitly rule on this in conjunction with FHIR-I, since the same
choice has to be reflected in `signatures.html` and in the
yet-to-be-merged FHIR-51143 disposition.
