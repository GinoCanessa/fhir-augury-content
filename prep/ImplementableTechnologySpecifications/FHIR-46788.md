## Summary

John Moehrke reports that the description of the XML narrative
canonicalization on the FHIR Core `xml.html#narrative` page is visually
truncated (per the attached screenshot) and that, even when read in
full, it does not make clear that this canonicalization is intended to
be the counterpart of the JSON narrative canonicalization on
`json.html#narrative`. The ticket asks ITS to fix the truncation and
clarify the parallelism between the two narrative canonicalizations.
There are no Jira tickets explicitly cross-referenced from this issue,
but the broader Security/Signature work since mid-2025 (Grahame
Grieve's `2025-06-gg-digital-signatures` branch and the
FHIR-51133 "Reorganize all of the signature guidance" effort) has
consolidated the XML, JSON, and `Signature` data-type content into a
single `signatures.html` page. In the Security and Privacy Zulip
thread on 2025-10-13, Eric Haas explicitly proposed that FHIR-46788
"can be withdrawn" because the new signature page defines the XML and
JSON canonicalizations together, which is the exact remedy the ticket
was asking for.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit `xml.html#narrative` (the XML page section that documents
`http://hl7.org/fhir/canonicalization/xml#narrative`) so the
description is no longer truncated and explicitly mirrors the
`json.html#narrative` text. Specifically: (1) restore the full prose
of the rule (XHTML narrative canonicalized per
`http://hl7.org/fhir/canonicalization/xml` after stripping
unnecessary whitespace; only `xhtml:` prefix and `xml:lang`
attributes preserved; etc.), (2) add a "see also" cross-link from
the XML page to the JSON page and vice-versa, and (3) make the two
sections use parallel section structure and headings so the
correspondence is visually obvious. Leave the rest of the
canonicalization framework as-is.

#### Justification

This is the smallest, most literal fix to the defect Moehrke filed.
The XML page is normative ITS-owned content today, and the truncation
is unambiguously a bug. Until / unless the consolidated
`signatures.html` page actually lands in a published FHIR release,
implementers reading `xml.html` will continue to hit the truncated
text. A surgical fix is cheap and is reversible if the consolidated
page later supersedes it.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve FHIR-46788 by folding the narrative canonicalization
description into the consolidated `signatures.html` page being
authored on the `2025-06-gg-digital-signatures` branch (and tracked
by FHIR-51133 — "Reorganize all of the signature guidance"). Replace
the prose on `xml.html#narrative` and `json.html#narrative` with a
short stub that points to the single authoritative section in
`signatures.html`, and ensure the consolidated section makes the
parallel XML/JSON canonicalization rules explicit and side-by-side
(which the current draft already does in part). Mark FHIR-46788 as
"persuasive with modification — addressed by FHIR-51133."

#### Justification

This is the direction the Security/ITS/FHIR-I cross-WG conversation
has actually taken (see the Zulip threads above). Doing a one-off
fix on `xml.html#narrative` (Disposition A) without coordinating
with the consolidation work risks introducing wording that diverges
from the consolidated page, which would re-create the very
"these two pages don't obviously line up" problem Moehrke is
complaining about. Folding the fix into FHIR-51133 keeps the
canonicalization rules in a single owning location, makes the
XML/JSON parallelism structurally guaranteed (rather than something
two separately-maintained pages have to keep in sync), and aligns
ITS work with the FHIR-I/Security collaboration model Grahame
proposed. Trade-off: it ties this ticket's resolution to the larger
signature rewrite landing, which may slip; if FHIR-51133 stalls,
this ticket would need to fall back to Disposition A.

---

### Disposition C: Decline (Withdraw / Already Addressed)

#### Proposal

Close FHIR-46788 as "withdrawn / already addressed" on the basis
that the rewritten `signatures.html` page on the
`2025-06-gg-digital-signatures` branch already defines the XML and
JSON narrative canonicalizations together, which is the relief the
ticket was asking for. Cite FHIR-51133 (Reorganize all of the
signature guidance) as the umbrella ticket and the Zulip exchange
of 2025-10-13 in which the original co-discussant (Eric Haas)
explicitly recommended withdrawing FHIR-46788 for this reason.

#### Justification

This matches the only on-record disposition proposal in the Zulip
threads and is the lowest-effort path. It is defensible because the
underlying defect (truncated text + unclear parallelism with JSON)
is already remedied in the in-flight rewrite, and because the
reporter has not pushed back on Eric Haas's withdrawal suggestion.
The risk is that the rewrite is still on a branch, not yet merged
to the main spec, so closing the ticket today leaves a window in
which the published page is still defective; this risk is
mitigated by FHIR-51133 carrying the actual remediation.

---

### Recommendation

**Recommended disposition:** B

The Zulip record makes clear that the workgroup's centre of gravity
is the signature-page consolidation tracked under FHIR-51133, and
that the relevant subject-matter participants (Moehrke, Grieve,
Haas) all support moving canonicalization narrative into one place
rather than maintaining two parallel descriptions on
`xml.html` and `json.html`. Pure withdrawal (Disposition C) is
attractive but premature: the consolidated `signatures.html` lives
on a branch and has not yet shipped in a published FHIR release, so
closing this ticket without an explicit linkage to FHIR-51133 risks
losing the trail when the umbrella ticket is dispositioned.
A surgical edit to `xml.html#narrative` alone (Disposition A)
solves today's defect but locks in the very split-page structure the
broader effort is trying to retire. Disposition B captures the
benefits of both: it commits ITS to the consolidated outcome while
keeping FHIR-46788 attached as the documentation defect that the
consolidated page must demonstrably fix. If FHIR-51133 is delayed
beyond this ballot cycle, the workgroup can fall back to
Disposition A as an interim patch.
