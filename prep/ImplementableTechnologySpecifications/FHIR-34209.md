## Summary

The reporter (Jens Villadsen) observes that the FHIR Identifier Registry
(`https://www.hl7.org/fhir/identifier-registry.html`) is US-centric and asks
that the GLN (Global Location Number) registry — identified by OID `1.3.88`
or by the URI `https://www.gs1.org/gln` — be added as a "Known Identifier
System" globally. GLN is an internationally used GS1 identifier (sister key
to GTIN, which is already listed) and has implementers outside the US who
need a stable, blessed `Identifier.system` value.

The closely related HTA ticket [HTA-40](https://jira.hl7.org/browse/HTA-40)
("External Code System Identifier Request for Global Location Number
(GLN)") was opened independently by the Vaccine Credentialing project and
is now **Complete / Done** — HTA approved GLN on 2021-12-02, and the
identifier system was published as a NamingSystem in HL7 Terminology (THO)
at `https://terminology.hl7.org/NamingSystem-GLN.html`. Per the Zulip
discussion, the broader plan from terminology leadership is that the
FHIR-core `identifier-registry.html` table itself is being retired in
favour of THO as the single source of truth for identifier systems, and
FHIR-I is being coordinated with to remove the table and point users to
THO. So the substantive ask of FHIR-34209 has effectively been delivered
through THO/HTA-40, and what remains is a process/registry-location
question rather than a missing identifier.

## Linked Jira Tickets

### [HTA-40: External Code System Identifier Request for Global Location Number (GLN)](https://jira.hl7.org/browse/HTA-40)

The HTA ticket through which GLN was formally requested (originally by
the Vaccine Credentialing project) and which directly satisfies the
substantive ask of FHIR-34209. It is **Complete / Done** (resolved
2022-01-13): HTA approved GLN on 2021-12-02, and Jessica Bota
(jsnell) confirmed that GLN — and similarly GTIN — are being modelled
as **NamingSystems** in the Identifier System area of THO (now visible
at `https://terminology.hl7.org/NamingSystem-GLN.html`). The thread
also captures Jens Villadsen's recommendations that GLN be modelled as
a NamingSystem (not a CodeSystem), that the page must not be marked
"GS1 US" because GS1 is international, and that a checksum (mod-10)
invariant should be applied to GLN values.

Participants: Carmela A. Couderc (reporter), Caroline Macumber, Paul
Denning (mitrep9g), Max Masnick (JIRAUSER17262), Jens Villadsen
(jkiddo), Jessica Bota (jsnell), Christian Hay (mentioned).

The originating thread for FHIR-34209. Jens Villadsen asks how a
request to add GLN (OID `1.3.88`) to the FHIR Identifier Registry
would be received and notes that the registry is US-centric. Lloyd
McKenzie responds that an OID would not be well-received and that HL7

international. Jose Costa Teixeira asks that SSCC also be added.
Consensus: the right home is THO (via HTA), the URI should be
`gs1.org`-rooted, and GLN was indeed added.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new row for GLN to the FHIR Identifier Registry page
(`hl7.org/fhir/identifier-registry.html`) with:

- **System URI:** `https://www.gs1.org/gln`
- **OID form:** `urn:oid:1.3.88` (the HTA-endorsed and OID-registry
  value for GLN; note that some realms — e.g., Switzerland — also use
  `2.51.1.3` under the GS1 root `2.51`, which the IG Publisher accepts
  without OID-validation warnings, but `1.3.88` is the value HTA
  blessed in HTA-40).
- **Description:** Global Location Number (GS1) — identifies physical,
  legal and functional locations; international scope.
- **Reference:** link to `https://www.gs1.org/standards/id-keys/gln`
  and to the THO NamingSystem
  `https://terminology.hl7.org/NamingSystem-GLN.html`.

Sibling entries for **SSCC** (`https://www.gs1.org/sscc`, OID
`2.51.1.2`) raised in the same Zulip threads should be filed as a
separate HTA request before being added here, but the GLN row should
land in the next FHIR publication.

#### Justification

GLN is internationally used, was explicitly approved by HTA in HTA-40,
and is the sister key to GTIN, which is already listed on this page.
The request from Jens Villadsen is precisely to remove the US-centric
gap in the registry. Listing a GLN row is consistent with the existing
GTIN row's URI pattern (`https://www.gs1.org/gtin`) and with Lloyd
McKenzie's stated preference (in the Zulip thread) for resolvable
`gs1.org`-rooted URIs over OIDs. HTA-40 has already done the upstream
governance work, so adoption here is mechanical.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new row to the FHIR-core identifier-registry table.
Instead, finish the migration that Jessica Bota describes in the
`#terminology > GS1 NamingSystems` thread: deprecate
`hl7.org/fhir/identifier-registry.html` as a content page, replace its
table with a redirect / pointer to the THO NamingSystem index
(`https://terminology.hl7.org/`), and reference the GLN NamingSystem
specifically (`https://terminology.hl7.org/NamingSystem-GLN.html`).
Add a brief normative note in FHIR-core that THO is the authoritative
registry of identifier systems, and that all "Known Identifier
Systems" should be sourced from THO going forward. As a follow-up,
file a separate HTA ticket for SSCC so the THO catalogue is complete
for GS1 supply-chain identifiers.

#### Justification

Per Jessica Bota (THO/HTA), the identifier systems on the FHIR-core
registry page have already been moved to THO, and FHIR-I has been
asked to remove the table and point users to THO. Maintaining two
registries — one in FHIR-core and one in THO — guarantees they will
drift, and the GLN ask is itself an example of the FHIR-core list
falling behind. Aligning with the THO-as-source-of-truth direction
delivers the underlying need (a globally-blessed `Identifier.system`
value for GLN, which already exists in THO) and prevents a recurrence
of the same kind of ticket for the next missing GS1 / international
identifier system. It also respects Lloyd McKenzie's note in the
ticket comments that "HTA owns requests for identifier registrations."

---

### Disposition C: Decline (Already Addressed)

#### Proposal

Resolve FHIR-34209 as **Persuasive with Modification — Already
Implemented**, with the modification being: the work has been done
through HTA-40 and THO, not through a change to the FHIR-core
identifier registry. Close the ticket with a pointer to
`https://terminology.hl7.org/NamingSystem-GLN.html` and to the
in-progress retirement of the FHIR-core `identifier-registry.html`
table (per the `#terminology > GS1 NamingSystems` discussion).
Implementers needing a globally-blessed GLN `Identifier.system` URI
should use `https://www.gs1.org/gln` as published in THO.

#### Justification

The substantive ask is satisfied: HTA-40 is **Complete/Done**, GLN is
published in THO with the requested `gs1.org` URI, and HTA — not the
FHIR Core specification — owns identifier-system registrations going
forward (mckenzie's own comment on this ticket). Adding a row to the
FHIR-core registry page now would duplicate authoritative content
that lives in THO and would be undone by the planned migration of the
page to a THO pointer. Declining (in the "already addressed" sense)
records the outcome accurately and avoids creating a dual-source-of-
truth maintenance burden.

---

### Recommendation

**Recommended disposition:** C (Persuasive with Modification — Already
Implemented), trending toward B as a follow-up.

The reporter's actual need — a globally-blessed `Identifier.system`
value for GLN — has been delivered through HTA-40 and is live in THO
at `https://terminology.hl7.org/NamingSystem-GLN.html`. The on-record
position of HTA (mckenzie on this ticket; Jessica Bota in the
terminology Zulip thread) is that identifier-system registrations are
HTA's responsibility and that THO is the single source of truth, with
the FHIR-core registry page being retired in favour of a pointer to
THO. Reopening this ticket as an "add a row to identifier-registry.html"
change (Disposition A) would re-introduce the duplication that the
THO migration is explicitly trying to remove. The cleanest action for
the workgroup is therefore to **close FHIR-34209 as already addressed
via HTA-40**, citing the THO NamingSystem, and — separately — to
follow through on the page-retirement work described in Disposition B
so that future "missing identifier" tickets like this one are routed
to HTA from the start. SSCC, raised in the same Zulip threads, should
be tracked as a fresh HTA request, not under FHIR-34209.
