## Summary

FHIR-18223 asks the IPS work to move the **IPSNoImmunizationInfo** value set
— which today is built from an IPS-internal code system used to flag the
absence of immunization information — onto **SNOMED CT**, in line with
"the interest in an HL7 free subset." The ticket was originally voted
*Considered for Future Use* at the September 2018 WGM (Hausam/Beebe,
14-0-3) on the basis that no globally usable free SNOMED CT subset then
existed; lynn_laakso re-opened it on 2022-11-22 ("re-open post STU
update publication") so it can be reconsidered now that such a subset is
available.

The wider IPS migration to free SNOMED CT content has, in fact, already
landed via [FHIR-23805](https://jira.hl7.org/browse/FHIR-23805)
("Add SNOMED CT free set value sets as appropriate in IPS profiles",
*Persuasive*, Published) — that ticket applied GPS (SNOMED CT Global
Patient Set) slices and ECL-based SNOMED valuesets across the IPS
profiles, and a 2024-07 comment (jdamore) notes the IG has since moved
to the **IPS Terminology sub-ontology of SNOMED CT** for SNOMED-bound
value sets. FHIR-18223 is the immunization-section counterpart to that
work: the IPSNoImmunizationInfo value set was not explicitly listed in
the FHIR-23805 application notes, so reviewers should decide whether
the immunization "no information" codes have already been brought under
the IPS Terminology / GPS pattern, or whether an explicit re-binding is
still outstanding. The sister ticket
[FHIR-18247](https://jira.hl7.org/browse/FHIR-18247) (move
Condition.severity binding from a LOINC answer list to SCT) shares the
same vote, the same November 2022 re-open, and the same underlying
"now that GPS exists, revisit the LOINC/IPS-internal bindings"
motivation.

## Proposed Dispositions

> Important framing: the IPS IG has, since FHIR-18223 was filed,
> already migrated most SNOMED-bound IPS value sets to ECL-based
> definitions over the IPS Terminology sub-ontology of SNOMED CT
> (per FHIR-23805's 2024-07 comment). The dispositions below should be
> read against that current state.

### Disposition A: Accept As Requested — Re-bind IPSNoImmunizationInfo to SNOMED CT (IPS Terminology / GPS)

#### Proposal

Replace the IPS-internal code system currently used by the
`IPSNoImmunizationInfo` value set with a SNOMED-CT-based definition
authored against the IPS Terminology sub-ontology (the same mechanism
FHIR-23805 standardised on). Specifically:

- Define the value set as an ECL expression over the IPS Terminology
  edition that selects the SNOMED CT situation/finding codes for
  "no known history of immunization" / "no information about prior
  immunization", e.g. `1149447003 |No information about prior
  immunization (finding)|` and the equivalent "No history of
  immunization (situation)" concept.
- Keep the existing IPS-internal codes only as a deprecated
  alternate slice (or remove them) so existing implementers have a
  documented migration path.
- Update the IPS Immunization profile binding to point at the
  re-defined value set; preserve the value-set canonical URL.
- Mirror the `core-...-gps-uv-ips` naming and copyright pattern that
  FHIR-23805 introduced for other IPS value sets.
- Disposition this ticket together with FHIR-18247 (LOINC →
  SCT for Condition.severity) so the "post-STU revisit of 2018
  holds" arrives as one coherent change.

#### Justification

This is exactly what the ticket asked for and is consistent with the
established IPS pattern. The original 2018 vote explicitly conditioned
its "Considered for Future Use" outcome on the eventual existence of a
"globally usable subset of SNOMED CT concepts" — that condition has
been satisfied (GPS / IPS Terminology) and was the basis for
lynn_laakso re-opening the ticket post-STU update. SNOMED CT already
has fit-for-purpose codes for the "no known / no information about
prior immunization" concepts these slots are intended to express,
which removes the IP-availability concern that drove the original
hold. This disposition aligns FHIR-18223 with FHIR-23805 (Persuasive,
Published) so the IPS terminology story stays consistent across all
sections. Risk to weigh: implementers using the current IPS-internal
codes will need a migration note in the change log.

---

### Disposition B: Alternative Approach — Use FHIR's existing absence semantics (DataAbsentReason / "no known" pattern), with an optional SCT slice

#### Proposal

Rather than re-binding `IPSNoImmunizationInfo` to a SCT-only set,
restructure the immunization "no information" representation to use
the FHIR-wide absence pattern:

- For the empty-section case, lean on the IPS "Empty Section" /
  `emptyReason` mechanism (already discussed by John Moehrke in the
  IPS Zulip thread above) instead of carrying a placeholder
  Immunization with a "no information" code.
- Where a coded "no known immunization" assertion is genuinely
  needed inside the section, model it via the existing FHIR
  "no-known-X" pattern (e.g., AllergyIntolerance's `no known
  allergies` precedent) and bind to a small value set whose default
  slice is SNOMED CT (IPS Terminology) and which optionally allows
  the existing IPS-internal codes for backward compatibility.
- Retire the bespoke IPS code system over the next IPS publication
  cycle.

#### Justification

This addresses the underlying need — being able to assert "no
immunization information" — without minting an IPS-specific binding
just for one value set, and it aligns IPS more tightly with the wider
FHIR ecosystem's handling of absence (DataAbsentReason, "no known"
profile patterns). It also respects the precedent set by FHIR-18222
(*Resolved - No Change*), where the work group declined a wholesale
"consolidate everything onto one code system" reshape. Trade-offs vs.
A: more design work, broader scope, and potentially a breaking
change for implementers who currently rely on the IPS-internal codes
in `IPSNoImmunizationInfo`. Stronger long-term hygiene; weaker
alignment with the literal text of the ticket.

---

### Disposition C: Decline as already addressed by FHIR-23805

#### Proposal

Close FHIR-18223 as **Persuasive with Modification** — *already
addressed by FHIR-23805*. State that the IPS migration to free
SNOMED-CT-based value sets (GPS, then the IPS Terminology
sub-ontology) accomplished the request in spirit at the value-set
layer, and that no further per-value-set re-binding action is
required for IPSNoImmunizationInfo specifically. Add a forward
pointer to FHIR-23805 in the resolution note. If, on re-inspection,
the IPS Immunization section's "no information" value set is
genuinely *not* yet covered by the IPS Terminology pattern, fall
back to Disposition A.

#### Justification

FHIR-23805 explicitly applied the GPS / IPS-free-set pattern across
IPS profiles and the 2024 follow-up moved the IG onto IPS Terminology.
If IPSNoImmunizationInfo is already being expressed via that
mechanism — or has been folded into a GPS-aligned value set — then
re-opening the 2018 hold creates churn rather than substantive
change. Defensible if a quick check of the current IPS publication
shows the immunization "no information" codes are already SNOMED-CT
backed. Carries one clear risk: if the re-check is skipped, this
disposition silently leaves an outdated IPS-internal binding in
place, which is precisely what the original 2018 vote committed to
revisit.

---

### Recommendation

**Recommended disposition: A.**

The ticket is small, narrow, and has already been triaged twice on its
merits — the only reason it sat unresolved was the 2018 unavailability
of a free SNOMED CT subset, and lynn_laakso's 2022 re-open explicitly
acknowledges that condition has been removed. The cleanest path is to
do exactly what was asked: re-define `IPSNoImmunizationInfo` as a
SNOMED-CT-backed value set authored against the IPS Terminology /
GPS, in the same shape FHIR-23805 (Persuasive, Published) used for the
rest of the IPS value sets, and to disposition FHIR-18247 in the same
batch so the immunization and Condition.severity holds clear together.
Disposition C is tempting and may be partly true, but it relies on an
inspection of `HL7/fhir-ips` that this prep was not able to perform
(no repo briefing in the cache); using C without that check risks
silently leaving the original IPS-internal binding behind. Disposition
B is a stronger long-term answer but exceeds the scope of a single
2018 change request and re-opens design questions the work group has
previously declined (FHIR-18222) — better filed as a separate ticket
if the work group wants to pursue it.
