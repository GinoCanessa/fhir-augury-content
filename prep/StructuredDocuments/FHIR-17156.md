## Summary

FHIR Implementation Guide to the FHIR core specification are broken. The

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Fix the broken external links to the FHIR core specification throughout
the published US C-CDA on FHIR IG. Concretely:

1. In the IG source for US C-CDA on FHIR, audit every link that targets
   the FHIR core spec (links of the form `http://hl7.org/fhir/...` or
   `{{site.data.fhir.path}}...` style template variables).
2. Identify links that render as `http://hl7.org/hl7.org/...` in the
   published HTML — the doubled host segment is the visible symptom of
   a relative-link being incorrectly resolved against a base href, or
   of a template variable being concatenated to an already-absolute URL.
3. Normalize all such links to use a single, absolute, versioned base
   for FHIR core (e.g., `http://hl7.org/fhir/R4/`) — preferring an IG
   Publisher dependency-resolved page reference over hand-rolled URLs
   where possible.
4. Re-publish the IG and verify with a link checker that no
   `hl7.org/hl7.org` (or other doubled-host) URLs remain.

#### Justification

The ticket reports a clear publication defect: a malformed URL that no
browser can resolve to FHIR core content. The IG's value depends on its
references to FHIR core being navigable, so fixing the rendered links is
mandatory editorial maintenance, not a design change. Using a single
absolute, versioned base for FHIR core links is consistent with how
other HL7 IGs reference the core spec and avoids regressions when the
IG is re-published or moved.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) hand-fixing the affected links, switch
the IG's link-rendering strategy to use the IG Publisher's built-in
cross-spec dependency mechanism (declare FHIR core as a dependency in
the IG `dependsOn` and reference pages via the publisher's resolution
helpers / Markdown autolinking) so that the publisher emits correct
absolute URLs at build time. As part of this change, remove any
hard-coded `http://hl7.org/...` strings in narrative templates that
were the source of the doubled-host concatenation.

#### Justification

A mechanical "find and replace" fixes today's symptom but leaves the
underlying templating bug in place, so the same class of breakage can
recur the next time the site base URL or template changes. Letting the
IG Publisher own URL construction for cross-spec links eliminates the
root cause, gives the IG a stable upgrade path when FHIR core
republishes, and matches current authoring guidance for HL7 FHIR IGs.
Trade-off vs. Disposition A: this is more work up front and may touch
more files, but it is durable.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive / No Action** on the grounds that the ticket
targets a 2018-era publication of US C-CDA on FHIR; the current
published IG should be re-verified, and if the reported broken links no
longer appear in the current build, no change is required.

#### Justification

The ticket is from May 2018 and was last updated in November 2019. URL
construction in the IG Publisher and HL7-hosted IGs has changed
substantially since then, and the specific symptom
(`hl7.org/hl7.org/...`) is consistent with a build-time bug that may
already have been resolved by intervening publisher upgrades. If a
current re-publish does not reproduce the defect, declining (rather
than fabricating a fix for a no-longer-present symptom) is defensible
and avoids unnecessary churn in the IG.

---

### Recommendation

**Recommended disposition:** B

The reported URL pattern (`http://hl7.org/hl7.org/...`) is a templating
defect, not a one-off typo, so a purely surface-level fix (Disposition
A) is likely to leave latent breakage in other narrative pages and to
recur on the next republish. Disposition C is tempting given the
ticket's age, but the workgroup should confirm current state with a
fresh build/link-check before closing without action; if breakage
persists, only Disposition B addresses the underlying cause.
Disposition B also positions the IG to track FHIR core version updates
cleanly via the IG Publisher's dependency mechanism, which is the
direction other HL7 FHIR IGs have moved. The workgroup should adopt B,
but in practice that work begins with the same audit Disposition A
calls for, so A's steps are not wasted effort.
