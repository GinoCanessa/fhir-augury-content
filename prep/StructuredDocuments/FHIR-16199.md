## Summary

The reporter (Giorgio Cangioli) requests that the IPS Implementation
Guide's `profiles.html` page (`http://hl7.org/fhir/uv/ips/profiles.html`)
be split into several shorter pages to improve readability. As authored
in 2018, this page enumerated all of the IPS profiles (Composition,
Patient, AllergyIntolerance, Condition, MedicationStatement, etc.) on a
single, long HTML page. The ticket is purely an editorial / IG-structure
request — it does not propose any change to the underlying profile
content, cardinalities, bindings, or invariants. There are no
cross-referenced Jira tickets and no substantive Zulip discussion;
the only cross-reference is an automated FHIR-Bot post that announced
the tracker item to the `committers` stream when it was filed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure the IPS IG's single `profiles.html` page into a small set
of grouped, shorter pages within the IG's authoring sources. A
reasonable split (to be confirmed by the IPS authoring team) is:

- `profiles-composition.html` — IPS Composition + section/structure
  profiles.
- `profiles-clinical.html` — Allergies, Conditions, Medications,
  Immunizations, Procedures, Results, Devices, etc.
- `profiles-administrative.html` — Patient, Practitioner, Organization,
  Encounter, etc.
- Keep a thin `profiles.html` (or rename to `profiles-index.html`) as a
  landing page that introduces the profile set and links into each
  group, preserving the existing URL as a redirect / index for
  in-bound links from prior IPS publications.

The change is editorial: profile definitions, narratives, examples,
constraints and bindings remain exactly as published; only the IG page
that aggregates / introduces them is partitioned and re-linked.

#### Justification

Since the request is purely about IG navigation/readability of a long
aggregator page, splitting it is in line with how other large HL7 IGs
(e.g. US Core) organize per-resource profile pages. It is the
literal action requested by the reporter, the original 2018 vote was
only a deferral (not a substantive rejection), and lynn_laakso
explicitly re-opened the ticket post-STU-update so it could be
revisited. Doing the split as a set of grouped pages — rather than one
page per profile — keeps the change small and reviewable while
materially improving readability.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of physically splitting `profiles.html` into multiple
authored pages, retain it as a single canonical page but improve its
in-page navigability:

- Add a sticky / collapsible table of contents grouped by
  Composition / clinical / administrative profiles.
- Ensure each profile section has a stable anchor (`#patient-uv-ips`,
  `#allergyintolerance-uv-ips`, etc.) and surface those anchors in the
  IG's left-hand navigation.
- Add per-profile summary cards at the top of the page so readers can
  scan the profile set without scrolling the full content.

This preserves the existing single canonical URL
(`profiles.html`) and avoids any risk to inbound links from prior STU
publications, balloted material, derived IGs, or external citations.

#### Justification

The reporter's stated motivation is *readability*, not navigation per
se. A well-structured single page with anchors and a TOC can deliver
the same readability win as a multi-page split, while keeping the
URL surface stable — which matters for an internationally-referenced
IG where the `profiles.html` URL has been cited in literature, derived
national IPS IGs, and tooling for years. It is also strictly less
work for IPS publication tooling, since no template, sushi-config, or
menu changes are required beyond markdown/HTML edits.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** / **No Change**, on the basis
that (a) the request is a subjective editorial preference rather than
a defect or interoperability concern, (b) splitting the page would
break stable inbound URLs that have been cited since the original IPS
STU publication, and (c) the ticket has been outstanding since 2018
without any follow-up advocacy, additional use cases, or supporting
discussion (the only cross-reference is an automated bot post).

#### Justification

The IPS IG has been balloted, published, and updated multiple times
with the current `profiles.html` layout, and no implementer has
filed a follow-up ticket arguing it is a barrier to adoption. The
original 2018 workgroup vote was 16-0-3 to defer it (effectively
unopposed but unprioritised), and seven years on there is still no
cross-referenced Zulip discussion or linked GitHub change. Spending
publication-cycle effort on an unprioritised editorial change carries
non-zero risk (broken anchors, stale links from derived IGs) without
a clear implementer benefit on the record.

---

### Recommendation

**Recommended disposition:** B

Disposition B best matches the reporter's stated goal (readability)
while protecting the substantial install base that already cites
`http://hl7.org/fhir/uv/ips/profiles.html` directly — including
derived national IPS IGs and external clinical-informatics literature.
A clean in-page TOC, grouped sections, and stable per-profile anchors
deliver the navigation improvement the reporter is asking for without
requiring a multi-page split, new menu entries, or redirect handling
in the IPS publication pipeline. If, during implementation, the
authoring team finds the page is still unmanageable after the TOC and
grouping work, the workgroup can revisit Disposition A as an
incremental follow-up — but starting with B is the lower-risk,
higher-leverage move.
