## Summary

The reporter (Giorgio Cangioli) flags that the IPS profile
`attachment-uv-ips` — which constrains `Observation` so that the value is
expressed as `valueAttachment` — ships with effectively no narrative
explaining how it is meant to be used. The ticket asks the IPS authors
to add explanatory text to the profile so implementers understand the
intent, scope, and expected usage of an Observation whose value is an
embedded attachment (e.g., a scanned report, image, or document carried
inline as part of an IPS bundle). The ticket was opened in 2018, given
a 2018-08-02 disposition of *Considered for Future Use* (vote 16-0-3,
deferred to a later release), and then **reverted to Triaged on
2022-11-22 by Lynn Laakso** to be re-opened post the STU update
publication, so it is currently awaiting a fresh disposition by
Structured Documents.

There are no linked Jira tickets and no GitHub cross-references, so the
work is wholly an editorial / IG-content change. There is, however,
historical context: the sibling ticket **GF#16207** (raised one day
earlier by the same reporter) reported "wrong content shown" on the same
profile, and a July-2018 IG build entry on Zulip indicates the project
discussed **replacing `observation-attachment-uv-ips` with
`observation-media-uv-ips` (a Media-resource-based profile)**. Whether
that replacement actually shipped in the published IPS — and therefore
whether `attachment-uv-ips` still exists as an authored artifact today —
materially affects how this ticket should be dispositioned.

## Proposed Dispositions

> Note on grounding: the cross-reference index returned no GitHub items
> for this ticket, and no `repo-analysis` briefing applies, so the
> dispositions below name the IPS publication artifacts in conceptual
> terms (profile + intro/notes page) rather than concrete repo paths.
> The dispositions assume `attachment-uv-ips` is still an authored
> artifact in the IPS IG; if Structured Documents has in fact retired
> it in favour of `observation-media-uv-ips`, Disposition C becomes the
> correct framing.

### Disposition A: Accept As Requested

#### Proposal

Adopt the ticket as a documentation-only change against the IPS IG.
Specifically, add narrative content to the `attachment-uv-ips` profile
(its intro / notes / description page in the IPS publication) covering:

1. **Intent.** Why the profile exists: to carry, inside an IPS bundle,
   observation results that are inherently document-shaped (scanned
   reports, PDFs of legacy results, images embedded inline) and where
   neither a coded `valueCodeableConcept`, a `valueQuantity`, nor a
   `valueString` is sufficient.
2. **When to use vs. when not to.** Contrast with (a) the
   `observation-media-uv-ips` / `Media` route for image-style content
   and (b) `DocumentReference` for separately-managed documents that
   should not be embedded.
3. **Expected `Attachment` population.** Required/expected fields
   (`contentType`, `data` or `url`, `title`, `creation`, optionally
   `language`, `hash`, `size`) and guidance on inline vs.
   externally-referenced attachments inside an IPS context.
4. **Code, category, effective[x], status.** Guidance on minimal
   required Observation context (which `code`, which `category`, what
   `effective[x]` should reflect for a captured artifact).
5. **Example(s).** At least one inline example Observation
   demonstrating the expected shape.

This requires no StructureDefinition changes — only intro/notes-page
narrative, and optionally one or two example resources.

#### Justification

The ticket is, by its own wording, a request for explanatory text — not
a structural change. IPS profiles that constrain a polymorphic element
(`Observation.value[x]`) without telling implementers when to reach for
that variant routinely produce inconsistent implementation, which is
exactly the kind of feedback IPS publications absorb in subsequent
ballots. Disposition A is the lowest-risk, narrowest-scope way of
honouring the request and is consistent with how Structured Documents
normally addresses "missing implementer guidance" tickets in the IPS
IG. There is no community opposition recorded — the only Zulip activity
is bot traffic — and the 2018 deferral was procedural ("defer to next
release"), not substantive.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the ticket as the trigger for a small **rationalisation pass** on
the IPS "non-coded observation result" pattern, not just a one-profile
narrative fix. Concretely:

1. Audit `attachment-uv-ips` against `observation-media-uv-ips` and any
   sibling profiles for non-coded results, and document — once, on a
   shared IPS page — the **decision tree** for an implementer choosing
   between `valueAttachment`, `Media`-based observations, and
   `DocumentReference`.
2. Add a short "see also" cross-link to that decision tree from each
   participating profile's intro page (so the explanation lives in one
   place).
3. On `attachment-uv-ips` specifically, layer the
   profile-specific guidance items from Disposition A on top of the
   shared decision tree.
4. Optionally, if the audit shows `attachment-uv-ips` is genuinely
   redundant with `observation-media-uv-ips` for every realistic IPS
   use case, schedule formal deprecation in a follow-up ticket rather
   than silently leaving both profiles to drift.

#### Justification

The Zulip evidence — that the IPS authors at one point intended to
*replace* `observation-attachment-uv-ips` with
`observation-media-uv-ips` — strongly suggests there is unfinished
editorial work on this whole pattern, not just one missing paragraph.
Adding text only to `attachment-uv-ips` (Disposition A) risks codifying
guidance that contradicts what the sibling profile says, and leaves
implementers to discover the choice on their own. A shared decision
tree is more durable, scales to future IPS revisions, and avoids the
common IG anti-pattern where each profile re-litigates the same
question in its own intro. The trade-off versus A is scope: B is a
larger editorial change and requires Structured Documents to commit to
a position on the Attachment-vs-Media split, which the workgroup may
prefer to do in a dedicated ticket.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** (or **Already Addressed**, depending on
current IPS content) on the grounds that either (a) the
`attachment-uv-ips` profile has effectively been superseded by
`observation-media-uv-ips` in the published IPS, in which case the
correct fix is to retire the profile rather than expand its narrative,
or (b) the existing IPS profile pages already describe the intended
use for Attachment-valued observations at the level appropriate for an
international IG, and no implementer feedback has materialised in the
seven years the ticket has been open to suggest the current text is
insufficient.

#### Justification

The ticket is supported by no Zulip discussion, no linked tickets, no
GitHub references, and no implementer commentary in the cross-reference
index. The 2018 vote chose deferral, and the 2022 reopening was an
administrative reset rather than new substantive interest. If
Structured Documents finds during triage that `attachment-uv-ips` is
either (i) effectively obsolete or (ii) already adequately documented,
declining is defensible: persisting and expanding narrative on a
profile the workgroup may wish to retire is editorial debt, and IPS
ballots have a finite carrying capacity for Change Requests with no
demand signal behind them.

---

### Recommendation

**Recommended disposition:** **B**

The ticket is genuine — `attachment-uv-ips` *does* sit in an
under-explained corner of the IPS profile set — but the historical
record (sibling ticket GF#16207 about wrong content on the same
profile, plus the 2018 IG-build attempt to replace the Attachment
profile with a Media-based one) shows the underlying issue is not a
missing paragraph but an unresolved editorial decision about how IPS
should carry non-coded observation results. Disposition A would
relieve the symptom on this one profile while leaving the broader
inconsistency in place, and Disposition C is hard to defend without
Structured Documents first confirming that `attachment-uv-ips` has
actually been superseded. Disposition B addresses the root cause with a
small shared decision-tree page plus targeted profile-specific text,
keeps options open for either retiring or retaining the profile, and
gives implementers a single canonical explanation to point at — which
is what the reporter was asking for in 2018.
