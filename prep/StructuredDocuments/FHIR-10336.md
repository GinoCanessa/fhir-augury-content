## Summary

The ticket reports that in the **C-CDA on FHIR Consultation Note Composition**
profile, the `Composition.encounter` element does not constrain its
`Reference` to the DAF Encounter profile. The reporter (Corey Spears) asks
that the profile add a `<type><profile value="http://hl7.org/fhir/StructureDefinition/daf-encounter"/></type>`
declaration on `Composition.encounter` so that consumers know the referenced
`Encounter` resource is expected to conform to the DAF Encounter profile.

The companion ticket [FHIR-10340](https://jira.hl7.org/browse/FHIR-10340)
raises the identical concern against the **C-CDA on FHIR Transfer Summary
Composition** profile (same author, filed three minutes later). Both
tickets were originally filed in July 2016 and were re-opened by
lynn_laakso in March 2024 with the comment *"re-open post STU1.2
publication"*, signalling that disposition was deferred during the
STU1.2 cycle and the request should now be considered for the next
revision of US C-CDA on FHIR.

The substantive complication, not raised in the ticket itself, is that
the `daf-encounter` profile referenced in the request belongs to the
retired Data Access Framework (DAF) implementation guide. DAF was
superseded by **US Core**, whose modern equivalent is
`http://hl7.org/fhir/us/core/StructureDefinition/us-core-encounter`.
Any disposition has to decide whether to honour the literal request,
update it to the current US Realm profile, or decline.

### Disposition A: Accept As Requested

#### Proposal

Add a target-profile constraint on `Composition.encounter` in the
**C-CDA on FHIR Consultation Note Composition** profile (and, by
parity, FHIR-10340 against the Transfer Summary Composition profile).
The constraint would be expressed in the StructureDefinition
differential as a `Reference` `type` with a `targetProfile` of
`http://hl7.org/fhir/StructureDefinition/daf-encounter`, exactly as the
reporter requested.

#### Justification

This is the literal interpretation of the ticket and is consistent with
the FHIR mechanism for narrowing a `Reference` to a specific profile:
ElementDefinition.type.targetProfile. It records the original authoring
intent (DAF as the encounter profile of choice in 2016) into the
machine-readable profile.

The serious caveat is that DAF is no longer maintained and the
`daf-encounter` URL does not resolve to a published profile in the
current US Realm landscape. Adopting Disposition A as written would
bake a dangling profile reference into a current US C-CDA on FHIR
publication, which validators will not be able to resolve — so this
disposition is only defensible if the ticket is treated as a literal
correction to a historical artefact rather than a forward-looking
constraint.

---

### Disposition B: Apply the Constraint Using US Core Encounter

#### Proposal

Apply the spirit of the request but replace the obsolete DAF profile
with the current US Realm equivalent: add
`targetProfile = "http://hl7.org/fhir/us/core/StructureDefinition/us-core-encounter"`
on `Composition.encounter` in the Consultation Note Composition profile
(and on the equivalent element in Transfer Summary Composition for
FHIR-10340). Pin to the US Core version that aligns with the current
US C-CDA on FHIR base dependency, and add a short narrative note in the
profile's intro page explaining that the original request named DAF
Encounter, which has been superseded by US Core Encounter.

If desired, allow `Reference(Encounter | us-core-encounter)` so that
non-US-Realm content remains valid while US Realm consumers receive
the stronger constraint.

#### Justification

This addresses the underlying need — telling implementers what kind of
`Encounter` to expect at `Composition.encounter` — while staying
consistent with the rest of the US C-CDA on FHIR IG, which already
profiles other compositional references against US Core. It also
prevents publishing a dead profile URL.

The trade-off is a tighter dependency on the US Core release line
(US C-CDA on FHIR is already pinned to a specific US Core version, so
this is mostly a matter of confirming the version pin) and a small
risk of breaking implementers who today bind `Composition.encounter` to
a non-US-Core Encounter resource. Allowing the union type
(`Encounter | us-core-encounter`) mitigates that risk while still
giving US Realm validators the stronger check the ticket asked for.

---

### Disposition C: Decline (Persistent Resolution)

#### Proposal

Resolve as **Persuasive with Modification — Decline literal change /
No further action** with a comment on the ticket explaining that:

1. The DAF Encounter profile referenced in the request has been
   retired; the equivalent constraint today would point at US Core
   Encounter.
2. The C-CDA on FHIR Composition profiles intentionally leave
   `Composition.encounter` typed as a generic `Reference(Encounter)`
   so that the IG can be used in non-US-Realm contexts and so that
   downstream profiles (e.g., US Core, QI-Core, jurisdictional IGs)
   can apply their own target-profile constraints without conflict.
3. Implementations that want the US Realm constraint can compose the
   US C-CDA on FHIR Composition profile with US Core Encounter at the
   instance level, or define a derived profile.

Close FHIR-10336 and FHIR-10340 with the same disposition.

#### Justification

This is defensible if the working group's authoring policy is to keep
C-CDA on FHIR Composition profiles realm-neutral on resource
references. Pinning `Composition.encounter` to a US Realm encounter
profile in the base IG would force every conformant document to use
US Core Encounter, even in deployments that have a stronger or
parallel Encounter profile (e.g., QI-Core). Declining keeps the IG
composable.

It also avoids re-opening a 2016 ticket whose substance has been
overtaken by the US Core / DAF transition.

---

### Recommendation

**Recommended disposition: B — Apply the Constraint Using US Core Encounter.**

The ticket's underlying complaint — that
`Composition.encounter` gives implementers no machine-readable hint
about which Encounter profile to expect — is legitimate, and the
March 2024 re-open by lynn_laakso signals the work group wants to act
on it now rather than carry it into another cycle. Disposition A, as
written, would publish a dead `daf-encounter` URL into a current US
Realm IG and is therefore not defensible. Disposition C is reasonable
only if the work group has an explicit policy of realm-neutral
references in the base C-CDA on FHIR Composition profiles; absent that
policy, leaving the constraint open in a US Realm IG is a missed
opportunity.

Disposition B threads the needle: it honours the ticket's intent,
aligns the constraint with the rest of the US C-CDA on FHIR / US Core
ecosystem, and — by allowing the union form
`Reference(Encounter | us-core-encounter)` — minimises the chance of
breaking implementers who already serialise plain `Encounter`
references. It should be applied in lockstep to FHIR-10340 (Transfer
Summary Composition), and the work group should review whether any
other C-CDA on FHIR Composition profiles (Discharge Summary, Procedure
Note, etc.) need the same treatment for consistency.
