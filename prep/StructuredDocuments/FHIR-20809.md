## Summary

The reporter (Craig Newman) flags a series of inconsistencies between the
generated XML and HTML/Content-tab renderings of the
`hai-questionnaire-opc-ssi-event` Questionnaire instance in the May 2019
publication of the US Healthcare Associated Infections (HAI) FHIR IG. The
XML appears to be missing the **Event Type** item that is present in the
HTML; the **Required** flag for `risk-factor-wound-class` does not agree
between the XML and the Content tab; the display text for
`risk-factor-weight` incorrectly reads "height" instead of "weight"; and
the **CPT Procedure Code** item is typed as `choice` but does not
reference a value set — the reporter suggests
`http://hl7.org/fhir/ValueSet/cpt-all` as a candidate. The same defects
are noted to occur on related Questionnaire instance pages within the IG.
The ticket has no cross-referenced Jira tickets, Zulip threads, or GitHub
items in Augury, so the analysis below is grounded in the ticket text
itself.

## Proposed Dispositions

> Note: Because no GitHub cross-references were available, the
> dispositions below cannot name a specific authoring root or repo path.
> They are framed in terms of the canonical Questionnaire instance file
> within the HL7 US HAI IG (whichever repository currently hosts the
> publication source for the May 2019 release of `hl7.fhir.us.hai`).

### Disposition A: Accept As Requested

#### Proposal

Update the source `Questionnaire/hai-questionnaire-opc-ssi-event` (and any
sibling Questionnaire instances flagged with the same defects, including
the `Questionnaire-hai-questionnaire-opc-ssi-event` page referenced in
the ticket) so that the XML, HTML and Content-tab renderings are in
sync. Specifically:

1. Add the missing **Event Type** `Questionnaire.item` to the XML
   instance, mirroring the `linkId`, `text`, `type`, `required` and
   `answerOption`/`answerValueSet` shown in the HTML/Content tab.
2. Reconcile `Questionnaire.item[linkId="risk-factor-wound-class"].required`
   so that the XML value matches the Content-tab table (set both to the
   same value — pick the one the workgroup considers authoritative for
   the SSI Event use case).
3. Fix the `text` for the `risk-factor-weight` item from "height" to
   "weight".
4. For the **CPT Procedure Code** item, keep `type = choice` and add
   `Questionnaire.item.answerValueSet` (or `answerOption`) referencing a
   concrete value set. The reporter proposed
   `http://hl7.org/fhir/ValueSet/cpt-all`; if licensing/scope rules out
   the full CPT-All set, bind to a HAI-specific subset value set that
   the IG defines (or create one).
5. Re-run the IG publisher so the regenerated HTML, XML and Content-tab
   views all derive from the corrected source.

#### Justification

The ticket describes straightforward errata in a single Questionnaire
instance: a missing item, a contradictory `required` flag, a copy-paste
error in display text, and a `choice` item with no answer set. None of
these are design questions — they are defects that violate the
Questionnaire invariant that `type = choice|open-choice` items SHOULD
have either `answerValueSet` or `answerOption` populated, and that the
published renderings of the same source should agree. Fixing them in
the source instance and re-publishing addresses every bullet the
reporter raised, requires no normative or invariant changes, and keeps
the IG aligned with the base FHIR Questionnaire guidance.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the ticket as two separable issues and resolve them differently:

- **Errata (items 1–3):** Apply the same source corrections as
  Disposition A — add the missing Event Type item, reconcile the
  `required` flag for `risk-factor-wound-class`, and correct the
  `risk-factor-weight` text. These are uncontroversial source fixes.
- **CPT Procedure Code binding (item 4):** Rather than binding to
  `http://hl7.org/fhir/ValueSet/cpt-all`, change the item from
  `type = choice` to `type = string` (or `Coding` with a CPT
  `system`-only binding and `binding strength = example`). CPT is a
  licensed code system whose full enumeration cannot always be
  distributed with HL7 IGs, so an unbounded `cpt-all` value set may not
  be a usable binding for all implementers. Document in the
  Questionnaire item description that responses should be CPT codes and
  cite `http://www.ama-assn.org/go/cpt` as the system.

#### Justification

The errata fixes are non-controversial and should land regardless. The
CPT binding, however, is a known sore spot in HL7 IGs because CPT is
not freely redistributable; binding to `ValueSet/cpt-all` gives the
appearance of a closed enumeration that the publisher cannot actually
materialize, and downstream tooling often cannot expand it. Loosening
the binding (or switching the item type) avoids shipping a binding the
IG cannot back with content, while still telling implementers what
codes are expected. Trade-off vs. Disposition A: Disposition A is more
literal to the reporter's request and keeps `type = choice` (which is
friendlier for form-renderer tooling); Disposition B is more honest
about CPT's licensing constraints.

---

### Disposition C: Decline

#### Proposal

Decline as **out of scope** on the grounds that the ticket targets the
**May 2019** publication of the US HAI IG, which is a frozen historical
release. If the Structured Documents WG is no longer actively
republishing this IG, the appropriate response is to close the ticket
as "Persuasive with Mod – not applied to a frozen ballot/publication"
and, if a future revision of the HAI IG is undertaken, carry the
defects forward as inputs to that work.

#### Justification

The ticket is from April 2019 and was last touched in November 2019; it
has been triaged but never resolved, has no comments, no cross-references,
no Zulip discussion, and no linked GitHub PR or issue. That pattern is
consistent with an IG that the workgroup is no longer actively
maintaining at the version the reporter saw. If that is the case,
spending review cycles re-opening and re-publishing a six-year-old IG
release is hard to justify, and a "won't fix in this version" close is
defensible.

---

### Recommendation

**Recommended disposition:** A

Items 1–3 are unambiguous defects in the published Questionnaire that
mislead implementers (a missing item, a contradictory required flag, a
weight item labeled "height"). Item 4 is a legitimate completeness gap
— a `choice` item with no answer set is a Questionnaire authoring
error. All four can be fixed in a single edit to the source
Questionnaire instance and a republish, with no normative or
backward-compatibility risk. Disposition B is a reasonable hedge if the
WG concludes that `ValueSet/cpt-all` is not an acceptable binding, in
which case the CPT bullet from A should be swapped for B's CPT bullet
while keeping the rest of A. Disposition C should only be selected if
the WG confirms that the HAI IG is not being republished, in which case
the disposition note should explicitly say so to give the reporter
closure.
