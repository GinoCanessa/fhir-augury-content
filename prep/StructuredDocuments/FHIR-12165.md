## Summary

NHS Digital (UK) is profiling a "CDA on FHIR" composition header for the UK
realm in the same shape as the US C-CDA on FHIR
`ccda-us-realm-header-composition` profile. The reporter notes that the US
profile reuses `ccda-participant-extension` to carry a CDA `participant`,
but the extension only allows the participant to be a `RelatedPerson` — it
does not let implementers reference a more general document participant
qualified by the CDA `participant.functionCode` (e.g. urgent-notification
contact, next of kin, caregiver, Power of Attorney, usual GP). NHS Digital
is currently working around this by referencing `PractitionerRole` and
hanging a complex extension off it to carry `functionCode`. The ticket
asks whether the C-CDA on FHIR project intends to broaden the participant
mechanism on `ccda-us-realm-header-composition` so that arbitrary
participants — qualified by a `functionCode` — can be referenced
natively, removing the need for non-US implementers (and the US profile
itself) to invent ad-hoc complex extensions.

The request is closely related to several other Structured Documents
tickets that have shaped how `Composition` carries CDA header concepts:
[FHIR-9665](https://jira.hl7.org/browse/FHIR-9665) explicitly enumerated
`participant`, `participant.functionCode` and the rest of the CDA header
roles as Composition extensions that needed to be defined for C-CDA on
FHIR (resolved Persuasive with Modification, 2018);
[FHIR-13635](https://jira.hl7.org/browse/FHIR-13635) argued that most of
those C-CDA extensions are really CDA-header-level (i.e. universal) and
should live on the generic Clinical Document profile rather than the
US-realm IG, but explicitly excluded `Participant` and `Performer` from
that promotion because they were "not well defined";
[FHIR-13488](https://jira.hl7.org/browse/FHIR-13488) added
`PractitionerRole` as a target for Composition.author / attester.party at
the core level, which is exactly the workaround NHS Digital is using; and
[FHIR-49767](https://jira.hl7.org/browse/FHIR-49767) is a recent (2025)
analogue asking that `Composition.information_recipient_extension` be
broadened to include `Organization` and `RelatedPerson` (Deferred — out
of scope for that ballot, "may be revisited"). Together these show a
recurring need to broaden the reference targets and qualifier mechanisms
on C-CDA-derived Composition extensions.

## Details

**Description:**
NHS Digital (UK) is looking to build a CDA on FHIR composition header
profile similar to the `ccda-us-realm-header-composition` profile. The
`ccda-us-realm-header-composition` has support for the related person
entity/resource within the `ccda-participant-extension` but there is no
provision for a reference to a participant that can be further detailed
using the CDA `participant` attribute `functionCode` (e.g. urgent
notification contacts participating specifically as a next of kin, a
caregiver, a Power of Attorney, usual GP, etc.). For now we are using
`practitioner role` with a complex extension to support `functionCode`.
Does the CCDA on FHIR project have plans to support the referencing of a
similar document participant within `ccda-us-realm-header-composition`
profile?

**Comments:**
None — the ticket has been in `Triaged` status with no Jira comments
since it was filed in 2016.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US C-CDA on FHIR IG, broaden `ccda-participant-extension` (used by
`ccda-us-realm-header-composition` on `Composition.extension`) so that a
participant can be either:

1. The current `RelatedPerson` participant, **or**
2. A reference to `Practitioner | PractitionerRole | Patient |
   RelatedPerson | Organization | Device` qualified by the CDA
   `participant.typeCode` and `participant.functionCode`.

Concretely, restructure `ccda-participant-extension` as a complex extension
with sub-extensions:

- `participant` : `Reference(Practitioner | PractitionerRole | Patient |
  RelatedPerson | Organization | Device)` — 1..1
- `typeCode` : `CodeableConcept` bound to the CDA `ParticipationType` value
  set (already used by `ccda-participant-extension`) — 0..1
- `functionCode` : `CodeableConcept` bound to the CDA `ParticipationFunction`
  value set (e.g. `RoleCode` / `ParticipationFunction`) — 0..1

Keep the existing `RelatedPerson`-only form available (or migrate it to the
new structure with `participant` constrained to `RelatedPerson`) so existing
US Realm Header instances remain valid.

#### Justification

This is exactly the missing capability the reporter and a current real-world
implementer (NHS Digital) need, and it follows the precedent set by
[FHIR-9665](https://jira.hl7.org/browse/FHIR-9665), which already named
`participant.functionCode` as a CDA header concept that needed to be carried
through Composition extensions. It also aligns with
[FHIR-13488](https://jira.hl7.org/browse/FHIR-13488), which broadened
participant target types on Composition by adding `PractitionerRole`, and
removes the need for downstream profiles to invent ad-hoc complex
extensions on `PractitionerRole` to carry `functionCode`. Because the
ticket targets the C-CDA on FHIR IG (Structured Documents authoring root,
the IG's `input/` Composition profile sources and the
`ccda-participant-extension` StructureDefinition), the change is contained
to a single IG and can be done without core-spec coordination.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than widen `ccda-participant-extension` in place, define the
participant + `functionCode` mechanism at a higher level so non-US realms
(NHS Digital, CH-Core, AU-Core, German national IGs, etc.) can use it
without reaching into a US-specific extension:

1. Promote a new `Composition.participant` complex extension to the generic
   Clinical Document profile (or as a universal Composition extension owned
   by Structured Documents), with sub-extensions `participant` (broad
   reference set), `typeCode`, and `functionCode` as in Disposition A.
2. Re-base `ccda-participant-extension` on that universal extension via
   profiling, so the US Realm Header continues to work but the underlying
   definition is reusable.

This re-opens the explicit `Participant` carve-out in
[FHIR-13635](https://jira.hl7.org/browse/FHIR-13635) now that the
participant shape is better understood. Coordinate with
[FHIR-49767](https://jira.hl7.org/browse/FHIR-49767) so the
information-recipient broadening is handled with the same pattern.

#### Justification

The reporter's underlying need is "I want to build a UK realm header
profile that does what the US one does." Solving FHIR-12165 only inside
the US C-CDA on FHIR IG forces every other realm to either (a) copy
the US-specific extension URL (semantically wrong) or (b) define their
own near-identical extension (extension proliferation). FHIR-13635
already established that most C-CDA Composition extensions are really
CDA-header-level; and FHIR-49767 shows the pattern recurring. A
universal participant extension solves all three at once. Trade-off:
larger scope, requires Structured Documents to own a new universal
artifact and coordinate naming/value-set bindings, and the C-CDA on
FHIR profile would need a re-publish to re-base on it.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by an existing mechanism**.
Implementers who need to qualify a non-RelatedPerson participant with a
CDA `functionCode` can:

- Use `Composition.author` / `Composition.attester.party` references to
  `PractitionerRole` (per [FHIR-13488](https://jira.hl7.org/browse/FHIR-13488))
  for clinician participants whose role is the qualifier of interest, and
- Carry any additional CDA `functionCode` via a local complex extension on
  the referenced resource (the workaround the reporter is already using).

Direct the reporter to the Clinical Document profile and note that any
realm-specific qualifiers belong in realm-specific extensions, not in the
US C-CDA on FHIR IG.

#### Justification

The C-CDA on FHIR IG is explicitly a *US-realm* mapping. Extending its
extensions to support a UK use case stretches the IG's scope. The
reporter already has a working mechanism (`PractitionerRole` + complex
extension), and the same pattern is what FHIR-49767 was deferred under
("not in scope for this ballot — document-construction concern"). This
disposition keeps the IG narrow and pushes the broader question into a
Clinical Document / universal-extension conversation.

---

### Recommendation

**Recommended disposition:** B

The request is real and well-grounded — `participant.functionCode` is a
CDA header concept Structured Documents already accepted should be
carryable on Composition (FHIR-9665), and the workaround the reporter
describes (`PractitionerRole` + ad-hoc complex extension) is exactly the
sort of extension proliferation the work group has criticised in
FHIR-13635 and seen recur in FHIR-49767. Solving it only inside the US
C-CDA on FHIR IG (Disposition A) would unblock NHS Digital but does
nothing for the next realm (CH, AU, DE) that hits the same wall, and
declining (Disposition C) freezes a workaround that the work group has
already identified as undesirable. Promoting a universal
`Composition.participant` complex extension with `participant` /
`typeCode` / `functionCode` sub-extensions, and re-basing
`ccda-participant-extension` on it, is the smallest design that closes
FHIR-12165, sets a clean pattern for FHIR-49767, and revisits the
explicit `Participant` carve-out in FHIR-13635 with the design
information that has accumulated since 2017. The work group should
expect this to require coordination on value-set bindings
(ParticipationType / ParticipationFunction) and a re-publish of the C-CDA
on FHIR profile to consume the universal extension.
