## Summary

The reporter, Simon Knee, observes that the C-CDA on FHIR profile
`ccda-us-realm-header-composition` declares no profile reference for the
`ccda-authorization-extension` it carries, so it is unclear which profile (or
which underlying resource) implementers are expected to use as the value of
that extension. The ticket asks the Structured Documents work group to clarify
the intended approach for the `ccda-authorization-extension` — specifically,
what content (likely a `Consent`-like resource or profile thereof) should be
referenced from the extension on the C-CDA Composition profile, and whether
the C-CDA on FHIR IG should publish a `ccda-consent` profile to fill the gap
the title alludes to ("ccda-consent (no profile found)"). No Jira tickets are
cross-referenced, and no GitHub items are cross-referenced; the only
cross-reference is an automated GForge tracker mirror posted to Zulip when the
ticket was filed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the C-CDA on FHIR IG, update the `ccda-authorization-extension`
StructureDefinition so that its `value[x]` (a `Reference`) explicitly targets
a profile, and publish that target profile alongside it. Concretely:

1. Define a new `ccda-consent` profile on the FHIR `Consent` resource that
   represents the C-CDA *Authorization* / *Consent* act (CONF rules from the
   C-CDA US Realm Header §2.1.x Authorization template — purpose of consent,
   policy reference, performer/authenticator, and effective time).
2. Change `ccda-authorization-extension.value[x]` (Reference) so that
   `targetProfile` points at the new `ccda-consent` profile (and, where
   appropriate, at `Contract` if the work group decides scope should include
   directives carried as Contract).
3. Add narrative to the extension's `purpose`/`comment` and to the
   `ccda-us-realm-header-composition` profile's invariants/guidance making the
   intended `ccda-consent` reference explicit, with an example instance.

#### Justification

The reporter's complaint is that the extension has no `targetProfile` and the
IG provides no guidance, so two interoperating implementers will pick
different resources (Consent vs. Contract vs. DocumentReference) and will not
exchange. The C-CDA Authorization template maps semantically to FHIR
`Consent`, which is the established FHIR pattern for authorization/consent
acts; constraining the extension to a published `ccda-consent` profile is the
minimum change that turns the extension from "free-form Reference" into an
interoperable contract. This is consistent with how other C-CDA on FHIR
extensions in the IG bind their references to companion profiles (e.g.,
participant/section profiles referenced from the same Composition header).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting a new `ccda-consent` profile, **retire the
`ccda-authorization-extension` from `ccda-us-realm-header-composition` and
replace it with `Composition.event` carrying a `Reference(Consent)` (and/or
`Composition.attester` where the C-CDA semantics align)**. Concretely:

1. Remove `ccda-authorization-extension` from the
   `ccda-us-realm-header-composition` profile (or mark it deprecated for one
   cycle).
2. Add a slice on `Composition.event` constrained to authorization-purpose
   coding, with `event.detail` typed as `Reference(Consent)` (using the
   un-profiled base `Consent`, or a very thin profile only if a CONF rule
   actually requires it).
3. Provide an IG page mapping the C-CDA Authorization template to this
   slicing pattern, with an example.

#### Justification

`Composition.event` is the FHIR-native place to record acts that the
Composition documents — including authorizations — and avoids inventing a
new extension/profile pair. Reusing the base `Consent` resource (rather than
publishing a thin `ccda-consent` profile) keeps the IG surface small and lets
implementers reuse `Consent` profiles already published by other US Realm IGs
(e.g., consent-on-FHIR work and US Core's evolving consent guidance). The
trade-off versus Disposition A is a (potentially breaking) change to the
header profile — but the extension is currently unusable because of the
missing `targetProfile`, so few conforming implementations can exist yet, and
the breakage is largely theoretical.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Mod / Already Addressed by Existing
Mechanism** without changing the `ccda-authorization-extension` definition.
Add a single note to the IG saying: "The `value[x]` Reference on
`ccda-authorization-extension` is intentionally unconstrained; implementers
SHOULD reference a `Consent` instance whose semantics match the C-CDA US
Realm Header Authorization template. No US Realm Consent profile is mandated
by this IG."

#### Justification

C-CDA on FHIR is a semantic mapping IG and not the venue for normative
US-realm Consent profiling — that work belongs to projects such as
Consent-on-FHIR and Privacy & Security. The ticket has sat at *Triaged* since
2016 with no implementer follow-up, no comments, and no Zulip discussion,
which is weak evidence of real-world demand. A documentation-only
clarification preserves implementer flexibility and avoids publishing a
profile that may diverge from whatever the Consent-focused work groups
ultimately produce.

---

### Recommendation

**Recommended disposition:** A

The ticket is squarely about an interoperability gap inside the C-CDA on
FHIR IG: an extension whose `Reference` has no `targetProfile` cannot be
implemented consistently. Disposition C papers over that with prose and
leaves the same ambiguity the reporter flagged in 2016. Disposition B is
architecturally cleaner but is a breaking change to the published US Realm
Header profile and depends on cross-WG alignment on a base `Consent` profile
that doesn't yet exist in the C-CDA on FHIR ecosystem. Disposition A is the
smallest change that closes the reporter's actual gap — it adds a
`ccda-consent` profile scoped to the C-CDA Authorization template's CONF
rules and pins the extension's `targetProfile` to it — and it leaves the
door open for the work group to later refactor toward `Composition.event`
(Disposition B) if Consent profiling consolidates elsewhere. Given the
ticket's age and lack of implementer pushback, the work group should aim
for the change that is most likely to be adopted by the next C-CDA on FHIR
publication cycle, which is Disposition A.
