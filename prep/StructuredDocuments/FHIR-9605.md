## Summary

The reporter requests that two SNOMED-CT concepts be added to the
`daf-observation-CCDAVitalSignResult` value set in the **US C-CDA on FHIR**
implementation guide so that the value set can be used to satisfy CMS
Eligible Hospital Clinical Quality Measures (eCQMs). Specifically:

- **SNOMED-CT 412726003** "Length of gestation at birth" — needed for
  **CMS9v4 (Exclusive Breast Milk Feeding)**.
- **SNOMED-CT 444135009** "Estimated fetal gestational age at delivery" —
  needed for **CMS113v4 (Elective Delivery)**.

These two concepts describe gestational-age observations recorded around
the time of birth/delivery. The reporter's framing treats them as vital
sign results so that a single CCDA-on-FHIR Vital Signs profile / value set
can carry the data required by the named eCQMs. No cross-referenced
tickets, Zulip threads, or GitHub items were surfaced for this ticket, so
the request must be evaluated on its face.

The originating value set name carries the legacy `daf-` prefix
(Data Access Framework), which the US C-CDA on FHIR IG inherited; this
historical context matters for the dispositions because the DAF program
has since been retired and the US Realm has shifted observation/vital-sign
expectations toward US Core and the eCQM/QI-Core stack.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add the two SNOMED-CT concepts to the
`daf-observation-CCDAVitalSignResult` ValueSet in the US C-CDA on FHIR
IG, as concrete `concept` entries (system =
`http://snomed.info/sct`):

- `412726003` — *Length of gestation at birth*
- `444135009` — *Estimated fetal gestational age at delivery*

Republish the ValueSet (bump version, refresh expansion against the
current SNOMED-CT US Edition), and update the corresponding Vital Signs
profile/example narrative to mention that gestational-age observations
recorded as vital signs are in scope for the named eCQMs.

#### Justification

The request is narrow, additive, and fully backward-compatible: extending
a value set with two additional concepts does not invalidate existing
instances. Both concepts are well-established SNOMED-CT pre-coordinated
concepts in routine perinatal use, and they directly enable the named
CMS eCQMs (CMS9v4 Exclusive Breast Milk Feeding, CMS113v4 Elective
Delivery) without requiring any structural change to the Vital Signs
profile. Honouring the literal request is the lowest-friction way to
unblock the reporter's quality-measure scenario.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the legacy `daf-observation-CCDAVitalSignResult`
value set in place, address the underlying need by:

1. Aligning the US C-CDA on FHIR Vital Signs concept binding with the
   **US Core Vital Signs** profile family and the **QI-Core**
   eCQM-oriented observation profiles (which already host the eCQM
   content); and
2. If the gestational-age concepts are not already covered there, raising
   the addition against the value set that US Core / QI-Core actually
   uses for vital-sign / pregnancy-related observations, and having the
   C-CDA on FHIR IG re-bind (or compose) to that value set instead of
   maintaining a parallel `daf-`-prefixed list.

Where retiring the `daf-` value set is not feasible for ballot reasons,
add the two concepts there *and* file a parallel change against the
upstream US Core / QI-Core value set so that the two stay aligned.

#### Justification

The `daf-` prefix is a vestige of the now-retired Data Access Framework
program; carrying eCQM content forward inside that namespace perpetuates
a fork between the C-CDA on FHIR vocabulary and the US Realm's current
observation stack (US Core + QI-Core), which is what implementers
actually deploy for eCQM reporting today. Routing the change through the
upstream value set keeps a single source of truth, reduces value-set
drift between IGs, and means future eCQM updates only have to be made
once. The trade-off vs. Disposition A is a longer review path and
cross-WG coordination (CIMI / CQI / Cross-Group Projects), which is why
the parallel-add fallback is offered for the immediate ballot cycle.

---

### Disposition C: Decline (as out-of-scope for the legacy value set)

#### Proposal

Do not modify `daf-observation-CCDAVitalSignResult`. Resolve the ticket
as **Persuasive with Mod / Not Persuasive** with a directive that
gestational-age observations for CMS9v4 and CMS113v4 should be sourced
from the US Core / QI-Core / eCQM value sets that already exist for that
purpose, and that the C-CDA on FHIR IG will not maintain its own
DAF-derived eCQM content.

#### Justification

The DAF program has been retired and the eCQM ecosystem has
standardised on QI-Core-based Observation profiles and their associated
value sets; adding new content to a `daf-`-prefixed value set in 2026
risks signalling that the C-CDA on FHIR IG is the place to extend
eCQM-supporting vocabulary, which is no longer the design intent. The
specific concepts are also arguably better modelled outside the Vital
Signs umbrella (gestational-age observations are obstetric/perinatal
findings rather than routine vital signs), so declining-with-pointer
preserves the integrity of the Vital Signs binding.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach), with Disposition A
as a fallback for the current ballot cycle.

The reporter has a real, concrete clinical-quality-measure use case, and
simply declining (Disposition C) leaves them without an actionable path
in the very IG they filed against. At the same time, accepting the
literal request (Disposition A) entrenches the legacy `daf-`-prefixed
vocabulary inside the C-CDA on FHIR IG and creates a parallel
eCQM-vocabulary surface that will diverge from US Core / QI-Core over
time. Disposition B addresses the reporter's need *and* keeps the US
Realm's observation vocabulary converging on a single owner. If the
upstream coordination cannot land in the same ballot window, the
workgroup should fall back to Disposition A's two-concept add — but
explicitly minute that the value set is on a deprecation/realignment
path so the same ticket does not recur for the next eCQM update. Given
how old this ticket is (filed 2016, last updated 2019, no comments, no
cross-references), the workgroup should also confirm with the reporter
that the CMS9v4 / CMS113v4 measure versions cited are still the targets
they care about before committing engineering effort.
