## Summary

Robert Hausam requests that the `DiagnosticReport` and `Composition`
resources be aligned so that `DiagnosticReport` can also serve as the
first/anchor resource of a FHIR Document `Bundle`. Today, only a
`Composition` may anchor a FHIR Document, and `DiagnosticReport.composition`
points to a `Composition` so the report can borrow that resource's
section-based structuring without duplicating it. In jurisdictions
(notably Europe) where a laboratory report must be a legally validated
"document", IGs such as the EU Base IG and the UV Laboratory Report IG
have to keep both `Composition` and `DiagnosticReport` at the "top level"
of the bundle — one to satisfy the FHIR Document rules, the other to
support DiagnosticReport-style search and result reporting. The request
is to formalize the overlap (likely via a shared FHIR "interface"
pattern) and to extend the FHIR Document machinery so that a
`DiagnosticReport` can stand in for the `Composition` at entry zero of a
document bundle.

The ticket is closely tied to **[FHIR-54513](https://jira.hl7.org/browse/FHIR-54513)**
(the Documents page should also describe DiagnosticReport's role) and
**[FHIR-46693](https://jira.hl7.org/browse/FHIR-46693)** (resolved
Persuasive in the Clinical Documents IG; SDWG explicitly committed to
"raise a FHIR R6 issue to gain clarity" on the
`Composition` / `DiagnosticReport` overlap and on the use of
`DiagnosticReport.composition`). FHIR-55368 is effectively that R6
issue.

## Details

**Description:**

Align DiagnosticReport and Composition and allow DiagnosticReport to
serve as the base for a FHIR Document bundle. There is a need,
particularly in some jurisdictions (e.g., Europe) for a laboratory
report to be treated and validated as a full-fledged "document".
Currently in FHIR, being recognized as a "FHIR Document" requires a
Composition resource as the first resource in a Bundle. Also,
DiagnosticReport has a 'composition' element which references a
Composition resource in order to provide structuring of the report
content (without having to fully duplicate the Composition structures).

If a DiagnosticReport was also able to serve as the base for a FHIR
Document bundle, this would considerably simplify the design of the
EU (https://build.fhir.org/ig/hl7-eu/base/index.html) and UV
(https://build.fhir.org/ig/HL7/uv-lab-rep-ig/) laboratory report IGs,
which now use a somewhat awkward design with both Composition and
DiagnosticReport considered to be at the "top level" of the report.
This design has been worked out in an attempt to be able to treat the
Bundle both as a valid FHIR document and to allow searching for
laboratory reports via a FHIR DiagnosticReport endpoint. This could be
significantly simplified if a single resource was able to serve both
purposes.

This could be done if additional alignment was done between the
DiagnosticReport and Composition resources, especially if they both
resources derived from or were compliant with a common pattern (most
likely an "interface" pattern in FHIR), and if both resources were
recognized as able to be used as the base of a valid FHIR Document.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-55368 as the SDWG-promised R6 alignment ticket and adopt the
following coordinated changes in the **FHIR Core** specification (FMG
sign-off required because both resources are normative-track and the
`document` Bundle rules are core machinery):

1. Define a new logical "interface" (pattern) — provisional name
   `DocumentDefinition` — in `documents.html` / a new pattern page,
   capturing the elements both `Composition` and `DiagnosticReport`
   already share or near-share: `identifier`, `status`, `subject`,
   `encounter`, `date`/`effective[x]`, `author`/`performer`,
   `category`/`type`, `attester` (where applicable), `custodian`
   (Composition-only today), `event`, `section`/`presentedForm`,
   `relatesTo`/`composition`.
2. Mark `Composition` and `DiagnosticReport` as compliant with that
   pattern, adjusting element names/cardinalities only where strictly
   necessary to satisfy the pattern (no breaking renames; use
   `obligations` / pattern-mapping rather than renames where possible).
3. Update `bundle.html`'s rules for `Bundle.type = document` so that
   the first entry MAY be **either** a `Composition` **or** a
   `DiagnosticReport` (anything that conforms to the new
   `DocumentDefinition` pattern). Update the `bdl-9` / document-bundle
   invariants and the "FHIR Document" definition accordingly.
4. Update `documents.html` to describe the two permissible anchors and
   to deprecate the
   "Composition + DiagnosticReport both at the top of the bundle" workaround
   pattern that the EU Base IG and the UV Lab Report IG currently use.
5. Re-evaluate `DiagnosticReport.composition` once the alignment lands —
   in many flows it becomes redundant when the `DiagnosticReport`
   itself is the document anchor.

#### Justification

This is the literal scope SDWG promised in the FHIR-46693 resolution
("SDWG will raise a FHIR R6 issue to gain clarity"). It removes the
double-anchor workaround that the EU Base IG and UV Lab Report IG
explicitly call out as awkward, eliminates the need for hand-rolled
identifier-matching invariants (see the
`europe > Invariant for Composition/DiagnosticReport.identifier equal`
Zulip thread), and uses an established FHIR mechanism (logical
patterns / interfaces) rather than inventing a new one. R6 is the
appropriate vehicle because the change touches the normative
`document` Bundle definition.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the `Bundle.type = document` rules. Instead, keep
`Composition` as the sole document anchor, but:

1. Define the same shared "interface" pattern (Disposition A, step 1)
   so that `Composition` and `DiagnosticReport` are demonstrably
   aligned at the element level; this addresses FHIR-54513 and the
   gdolin "specialization" question structurally without changing
   document-bundle semantics.
2. In `documents.html`, formalize the existing EU/UV pattern:
   `Composition` is entry 0 (and the document anchor), with a
   `Composition.section.entry` that references the `DiagnosticReport`
   carrying the lab content. Provide a tiny Composition profile (e.g.,
   `LaboratoryReportComposition`) and an invariant that
   `Composition.identifier` equals
   `DiagnosticReport.identifier` so the bundle can still be located via
   either identifier.
3. Optionally, mark `DiagnosticReport.composition` as
   trial-use-deprecated in favour of the
   Composition-points-to-DiagnosticReport direction.

#### Justification

The existing EU Base IG and UV Lab Report IG already work — implementer
tooling, validators, search behavior, and CDA-on-FHIR mappings all
assume `Composition` is the document anchor. Letting either resource be
the anchor (Disposition A) creates two equally valid representations of
the same business artifact, which complicates every downstream consumer
(servers' `_type` filters, IG validators, document-handling APIs). A
conformance-pattern alignment plus a clearly described
"Composition-anchored lab document" recipe gets most of the integration
benefit at a fraction of the spec churn, and avoids relitigating the
normative `document` Bundle definition for R6. It is also consistent
with the implementer guidance already given on the
`implementers > DiagnosticReport - Composition/Bundle boundaries`
thread.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-55368 **Not Persuasive — Considered for Future Use**. The
change is out of scope for R6 because (a) it would alter the normative
definition of the `document` Bundle, (b) the existing
`Composition` + `DiagnosticReport` "double anchor" pattern, while
verbose, is fully expressive and is already deployed in IGs, and
(c) implementers needing to search lab reports via a DiagnosticReport
endpoint can already do so today by indexing the
`DiagnosticReport` entry inside the document bundle.

#### Justification

`Composition` and `DiagnosticReport` were intentionally separated:
`Composition` organizes any human-presentable content,
`DiagnosticReport` summarizes a specific result set. Collapsing them
risks losing that distinction across hundreds of existing profiles and
servers. The IG awkwardness called out in the ticket (EU Base, UV Lab
Report) is real but is fundamentally a documentation problem
(addressable by FHIR-54513) rather than a normative one. R6 already has
significant Structured Documents work in flight; declining keeps that
scope manageable.

---

### Recommendation

**Recommended disposition:** A

FHIR-55368 is the explicit follow-on that SDWG committed to in the
FHIR-46693 ballot resolution ("SDWG will raise a FHIR R6 issue to gain
clarity"). Two real ballot-grade IGs (EU Base, UV Lab Report) and the
EHDS work documented on Zulip have already designed around the gap with
a known-awkward double-anchor pattern, and implementers are writing
hand-rolled invariants (see the `europe` thread on
`Composition`/`DiagnosticReport.identifier` equality) to compensate. A
shared interface plus an extended `document` Bundle anchor rule
addresses the root cause cleanly using a mechanism FHIR already has,
while honoring the resource distinction by keeping both resources in
place. Disposition B is a reasonable fallback if FMG declines to touch
the normative document Bundle in R6, and Disposition C is hard to
defend given the explicit prior commitment.
