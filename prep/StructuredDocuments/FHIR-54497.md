## Summary

The ticket reports that the diagram intended to illustrate "typical transitions
of clinical status for the Composition resource" is missing on the Composition
resource page. The introduction (`source/composition/composition-introduction.xml`,
line 109) references an image (`composition-state-machine.png`) immediately
after the prose describing how `Composition.status` moves from
`registered`/`preliminary` to `final` and possibly `amended`, but according to
the reporter the diagram is not actually shown / is not meaningful.

Inspection of the cached HL7/fhir clone (HEAD `db79dcd`) shows the file
`images/composition-state-machine.png` is present but is only **795 × 113
pixels** (≈10 KB) — a thin horizontal strip that does not look like a real
state-machine diagram, especially compared with the Composition-grade peers in
the same `images/` folder (e.g., `devicealert-state-machine.svg` ≈ 103 KB,
`request-state-machine.svg` ≈ 12 KB, `researchsubject-state-machine.png`
≈ 80 KB). The image was added in 2023 by commit `13d402a` ("FHIR-15738 - Add
state transition diagram"). So either the original PNG was a placeholder that
was never replaced with the intended artwork, or the file became corrupted /
truncated between then and now. Either way, from a reader's perspective the
diagram promised by the prose and `alt` text is not delivered.

There are no cross-referenced Jira tickets, Zulip threads, Confluence pages,
or GitHub items recorded for FHIR-54497 in FhirAugury at the time of
preparation. The only directly-related historical ticket is the Jira key
embedded in the original commit message — **FHIR-15738** ("Add state
transition diagram") — which is not currently in the FhirAugury Jira cache
(`get` returned 404). The fix is therefore self-contained in the FHIR Core
specification.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a "technical correction": replace the broken / placeholder
asset at `images/composition-state-machine.png` with a correct state-machine
diagram that visually depicts the transitions described in the surrounding
prose:

```
registered ──┐
             ├──► preliminary ──► final ──► amended
             │                         └──► entered-in-error (any state)
             └──► final
```

Concretely:

1. Author a new `images/composition-state-machine.svg` (preferred; matches
   recent peers like `devicealert-state-machine.svg`,
   `request-state-machine.svg`, `task-state-machine.svg`) containing the
   states `registered`, `preliminary`, `final`, `amended`,
   `entered-in-error`, with directed arrows showing the legal forward
   transitions and the "to `entered-in-error` from any state" edge.
2. Update `source/composition/composition-introduction.xml` line 109 to
   reference the new asset and wrap it in `<figure>` / `<figcaption>` to
   match the DeviceAlert pattern, e.g.:
   ```xml
   <figure>
     <img src="composition-state-machine.svg"
          alt="State machine showing typical transitions of Composition.status
               between registered, preliminary, final, amended, and
               entered-in-error."
          style="max-width:100%; height:auto;"/>
     <figcaption>Typical transitions of Composition.status.</figcaption>
   </figure>
   ```
3. Delete (or leave untouched if Git history hygiene is preferred) the old
   PNG once nothing else references it. A repo-wide grep for
   `composition-state-machine.png` should be performed before deletion.
4. Confirm via a local publisher run that the diagram now appears on the
   built `composition.html`.

#### Justification

- Matches exactly what the reporter is asking for (the prose claims a
  diagram exists; deliver one).
- Aligns Composition with the established pattern used by other resources
  that document state transitions (DeviceAlert, Request, Task,
  ResearchStudy, ResearchSubject), which all ship a real diagram alongside
  the introduction.
- SVG is preferred for new content (scales cleanly, smaller diff churn,
  and accessible alt text), and the FhirCore briefing's "Update narrative
  / prose" recipe covers the kind of edit needed here.
- The change is isolated to `HL7/fhir` under the documented authoring
  root (`source/` + `images/`), with no cross-repo or publisher
  configuration impact, so risk is minimal.

---

### Disposition B: Alternative Approach

#### Proposal

Drop the image entirely and replace it with an inline, machine-rendered
state-transition table or a Mermaid-style block (whichever the publisher
already supports — Mermaid is used elsewhere in the FHIR build). Specifically:

1. Remove the `<img src="composition-state-machine.png" .../>` line in
   `source/composition/composition-introduction.xml`.
2. In its place, insert a concise transitions table:

   | From         | Allowed next states              |
   |--------------|----------------------------------|
   | registered   | preliminary, final, entered-in-error |
   | preliminary  | final, entered-in-error          |
   | final        | amended, entered-in-error        |
   | amended      | amended, entered-in-error        |

3. Optionally also add a Mermaid `stateDiagram-v2` block (if the
   publisher's Mermaid support is enabled in this area) to provide a
   visual rendering generated at build time rather than as a binary asset.

#### Justification

- A textual / generated representation cannot rot the way a binary PNG
  has rotted here: every implementer reading the spec sees the same,
  authoritative content, and there is no separate asset to maintain.
- It reduces the Composition page's dependency on hand-drawn artwork that
  must be re-exported each time the value-set evolves.
- Trade-off vs. Disposition A: a static diagram is more glanceable for
  reviewers and clinical readers, and is consistent with the visual style
  used by other resource introductions. Removing the image is a more
  invasive editorial change and would set a precedent that other
  resources may not want to follow.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive / Already Addressed**: the image file
`images/composition-state-machine.png` exists in the repo and has been
present since 2023 (commit `13d402a`, FHIR-15738). The `<img>` tag in
`source/composition/composition-introduction.xml` correctly references it
using the same relative-path convention as other resource pages. If the
diagram is not appearing on the published page for the reporter, that is a
publisher / hosting issue rather than a content defect, and should be
re-filed against the FHIR build infrastructure.

#### Justification

- The asset is on disk and is wired up the same way as peer diagrams that
  do render correctly.
- Defensible only if Structured Documents inspects the published
  Composition page and confirms the diagram does in fact render and is
  semantically meaningful.
- This disposition is **weak** in light of the actual file inspection: a
  795 × 113 px / 10 KB PNG is implausibly small for a real state-machine
  diagram, so "already addressed" is hard to defend without first opening
  the image and confirming its content.

---

### Recommendation

**Recommended disposition:** A

The ticket is filed as a "Highest"-priority Technical Correction by the
Structured Documents work group, and direct inspection of the cached
`HL7/fhir` clone confirms the reporter's claim is real: the referenced
image is a 795 × 113-pixel, ≈10 KB PNG that has been in the repo
unchanged since 2023 and is far smaller than every other state-machine
diagram in the `images/` folder. The prose explicitly promises "a diagram
showing typical transitions of clinical status for the Composition
resource", and the introduction page is otherwise in good shape — so the
right fix is to actually deliver the diagram.

Disposition A is also the cheapest in implementation terms: it is a
single asset replacement (plus an optional one-line `<img>`-tag tweak in
`source/composition/composition-introduction.xml`) inside the documented
FhirCore authoring root, with no cross-repo, value-set, or
StructureDefinition impact, and the FhirCore briefing's gotchas
(publisher mechanics, generated `<snapshot>` blocks, value-set churn) do
not apply to this change. Disposition B is reasonable but introduces an
editorial precedent and removes a visual aid that several peer resources
intentionally provide; it should be reserved for a future, broader
"narrative diagrams as code" cleanup. Disposition C is not defensible
once the image file's actual size is examined, so it should be rejected.
