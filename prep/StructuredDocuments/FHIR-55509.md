## Summary

The ticket questions the necessity and clarity of the `Composition.name`
element, which the resource definition characterises as "computer friendly".
The reporter (Katie Reynolds) observes that `Composition` already carries
both `Composition.type` (a coded classification of the document) and
`Composition.identifier` (a stable business identifier), and asks which
use cases cannot be satisfied by those two elements. The request is not
to remove `Composition.name` outright but to **clarify the description**
so that the intended use of the element — and what "computer friendly"
actually means in this context — is unambiguous to implementers. There
are no cross-referenced Jira tickets, Zulip threads, or GitHub items in
the FHIR Augury index, so the picture is drawn entirely from the ticket
itself and the existing FHIR Core definition of `Composition.name`.

## Details

**Description:**

> Why do we need the "computer friendly" Composition.name element when we
> have Composition.type and Composition.identifier? What use cases cannot
> be met without Composition.name?
>
> I'd suggest that the description be updated to clarify the intended use
> for this element.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the element description for `Composition.name` in the FHIR Core
Composition resource definition to (a) explicitly state the intended
purpose of the element, (b) clarify what "computer friendly" means in
this context, and (c) distinguish it from `Composition.type` and
`Composition.identifier`. Concretely:

- Replace "computer friendly" with language such as "A short, machine-
  processable name (token-style, no whitespace) intended for use as a
  technical/file label or in URLs and references — analogous to a
  symbolic identifier rather than a coded classification or business
  identifier."
- Add an explicit note that `Composition.name` is **not** a
  human-readable display title (that role is `Composition.title`), is
  **not** a coded type (that is `Composition.type`), and is **not** a
  globally-unique business identifier (that is `Composition.identifier`).
- Add 1–2 short usage examples (e.g., a short token used as a file/URL
  slug for the composition snapshot) so the boundary against `title`,
  `type`, and `identifier` is concrete.
- No structural change (no cardinality change, no removal, no rename); the
  edit is to the element definition text and short description only.

#### Justification

The ticket itself only asks for clarification, not removal — the
priority "Highest" with label "DISCUSS" reflects that confusion among
implementers is the actual harm. Clarifying the description is a
non-breaking, low-risk change that respects the R5 design decision to
add the element while addressing the stated issue. gdolin's comment
explicitly signals that the workgroup should "look back at why
composition.name was added in R5" — that historical context can be
distilled directly into the element description, which is exactly what
this disposition delivers.

---

### Disposition B: Alternative Approach

#### Proposal

Treat `Composition.name` as redundant with the combination of
`Composition.title` (human-readable), `Composition.type` (coded
classification), and `Composition.identifier` (stable business
identifier), and **deprecate** the element rather than re-explain it:

- Mark `Composition.name` with a standards-status downgrade and a
  deprecation note pointing implementers to:
  - `Composition.title` for human-readable naming,
  - `Composition.identifier` for a stable, machine-routable identifier,
  - `Composition.type` for classification.
- Keep the element in the schema for backwards compatibility with R5
  instances but discourage population in new content.
- Add migration guidance in the Composition resource notes.

#### Justification

If the workgroup, on revisiting the R5 rationale, cannot identify a use
case that is not already covered by `title` / `type` / `identifier`,
then "clarifying the description" only entrenches a redundant element
and pushes the burden onto every implementer to decide which of three
near-synonymous slots to populate. Deprecation is the more honest
answer to "what use cases cannot be met without Composition.name?" when
the answer is "none we can articulate." The trade-off vs. Disposition A
is a heavier lift (deprecation process, downstream IG impact) and a
small backwards-compatibility cost — but the long-term simplification
of the resource is real.

---

### Disposition C: Decline

#### Proposal

Decline the change request on the basis that `Composition.name` was
intentionally introduced in R5 to align Composition with the same
`name` / `title` pattern used across canonical and definitional
resources (e.g., `StructureDefinition.name` vs.
`StructureDefinition.title`), and that the existing element definition
— while terse — is consistent with how `name` is documented on those
peer resources. Close as "Not Persuasive" with a short explanatory
note pointing the reporter at the established `name` vs. `title` vs.
`identifier` convention.

#### Justification

If the R5 rationale (per gdolin's comment) is in fact "be consistent
with the canonical-resource `name`/`title` pattern", then the question
is already answered by the cross-resource convention, and per-resource
re-explanation risks divergence. Declining keeps the element definition
short and consistent with peer resources. The risk is that the reporter's
core complaint — implementer confusion — is left unaddressed, which is
why this disposition is the weakest of the three.

---

### Recommendation

**Recommended disposition:** A

The ticket is a documentation/clarity request from a careful reviewer
(reporter Katie Reynolds, label `DISCUSS`, priority `Highest`), and the
single workgroup comment so far (gdolin) treats it as a question to
answer rather than a change to reject. Accepting as requested — i.e.,
updating the `Composition.name` element description to explicitly
delineate its role against `title`, `type`, and `identifier`, and to
replace the vague "computer friendly" phrase with concrete language —
is the lowest-risk, lowest-cost path that directly resolves the stated
problem. Disposition B (deprecation) is worth keeping in reserve if the
workgroup's R5-rationale review concludes there really is no
distinguishing use case, but that is a larger conversation that should
not block fixing the description in the meantime. Disposition C
under-serves the reporter and the broader implementer community given
the explicit `DISCUSS` flag.
