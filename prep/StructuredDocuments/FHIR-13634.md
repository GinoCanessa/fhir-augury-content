## Summary

Melva Peters reported that the US C-CDA on FHIR implementation guide
contains a broken hyperlink in the `InFulFillmentOf` extension. The link
points at the DSTU2-era DAF Core "DiagnosticRequest" profile
(`http://hl7.org/fhir/us/daf/2016Sep/daf-core-diagnosticrequest.html`),
which no longer resolves to a meaningful target because the
`DiagnosticRequest` resource was merged into `ProcedureRequest` (and
later `ServiceRequest`) shortly after DSTU2. The ticket therefore asks
the Structured Documents work group to repair the cross-reference in the
CCDA-on-FHIR IG so that readers of the `InFulFillmentOf` extension can
get to a current, authoritative description of the resource being
fulfilled. There are no cross-referenced Jira tickets; the only
cross-reference surfaced by FhirAugury is an automated `[committers] >
tracker-item` Zulip post that announced the original tracker item and
contains no substantive discussion. A commenter (`ehaas`) noted that DAF
itself is part of the frozen DSTU2 publication and cannot be edited, so
any fix must happen on the CCDA-on-FHIR side rather than in DAF.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Repair the broken hyperlink in the CCDA-on-FHIR `InFulFillmentOf`
extension so that it points at a working, current target. Concretely:

- Locate the `InFulFillmentOf` extension definition and any narrative
  page in the CCDA-on-FHIR IG that references
  `http://hl7.org/fhir/us/daf/2016Sep/daf-core-diagnosticrequest.html`
  (typically the extension's `StructureDefinition` definition/comments
  text and the extension intro page in the IG narrative).
- Replace that URL with the current canonical target for the resource
  the extension is meant to point at. The most accurate replacement is
  the present-day successor of DSTU2 `DiagnosticRequest` → DSTU2.1/STU3
  `ProcedureRequest` → R4 `ServiceRequest`
  (`http://hl7.org/fhir/ServiceRequest.html`), aligning with whatever
  FHIR version the published CCDA-on-FHIR IG is built against.
- If the prose around the link names "DiagnosticRequest" by name,
  update the prose alongside the URL so the wording matches the new
  target resource (`ServiceRequest` / `ProcedureRequest`).
- Rebuild the IG and verify the link resolves in the published HTML.

#### Justification

This is exactly what the reporter asked for and exactly what
`ehaas`'s comment points toward: the upstream DAF DSTU2 page is
frozen and unfixable, so the only viable repair is to retarget the
link from CCDA-on-FHIR. Pointing at the current core resource page
(`ServiceRequest` in R4) gives readers a stable, authoritative
definition of the resource the extension is meant to reference,
matches FHIR's "use the latest canonical" guidance for narrative
links, and keeps the change scoped to the IG that owns the broken
reference.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than swapping the URL for a single new target, audit the
`InFulFillmentOf` extension end-to-end and:

1. Replace the dangling DAF DSTU2 reference with a versioned link to
   the FHIR core resource that matches the publication version of
   CCDA-on-FHIR (e.g., `ProcedureRequest` for STU3, `ServiceRequest`
   for R4), using a `|<version>` canonical or a versioned URL so the
   link does not silently rot again when the next FHIR release ships.
2. Add a short normative note in the extension's narrative explaining
   the historical lineage (DAF DSTU2 `DiagnosticRequest` →
   `ProcedureRequest` → `ServiceRequest`) so implementers landing on
   older CDA-derived documents understand why the type evolved.
3. While the IG editor is in this file, sweep the rest of the IG for
   other `hl7.org/fhir/us/daf/2016Sep/...` links and fix any other
   instances in the same PR.

#### Justification

The literal request (Disposition A) fixes one symptom but does not
prevent recurrence. The DAF DSTU2 publication is the kind of frozen
artifact whose links are most prone to rot, and CCDA-on-FHIR is
unlikely to contain only one such reference. Fixing the lineage in the
narrative also helps the C-CDA → FHIR mapping audience, who often
arrive with older terminology in mind. The trade-off is a slightly
larger edit and a slightly longer review, but the change stays scoped
to the same IG and the same extension family.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Mod / Not Persuasive — already
addressed by current IG link practice**, on the grounds that (a) the
DAF DSTU2 page is part of a frozen ballot publication that HL7 cannot
edit, (b) the ticket dates from 2017 and the CCDA-on-FHIR IG has been
republished since, and (c) the work group should confirm via a quick
spot-check of the current published IG before committing to an edit
that may already have been applied during a prior republication.

#### Justification

Defensible only if Structured Documents verifies that the current
published CCDA-on-FHIR build no longer contains the dangling DAF
DSTU2 link — in which case the ticket is moot and should be closed
without further work. If the link is still present, declining is not
defensible because the issue is a clear, low-risk documentation defect
that the IG owners can fix unilaterally. This disposition exists
mainly to force the work group to confirm the current state of the
published IG before spending editing effort.

---

### Recommendation

**Recommended disposition:** B

The ticket is, on its face, a trivial broken-link fix (Disposition A),
but the underlying cause — link rot to a frozen DSTU2 publication — is
likely to affect more than one place in CCDA-on-FHIR, and the
substantive comment from `ehaas` shows that the change also requires a
small terminology update (`DiagnosticRequest` no longer exists). Doing
the broader sweep in one pass (Disposition B) is only marginally more
work than the narrow fix, removes a class of latent defects, and
produces narrative that will age better as FHIR continues to version.
Disposition C is only appropriate if the link has already been silently
fixed in a later republication, which the work group should verify
first; if it has, close as already-resolved, otherwise proceed with B.
