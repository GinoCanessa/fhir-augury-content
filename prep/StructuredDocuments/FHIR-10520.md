## Summary

This ticket asks the C-CDA on FHIR specification to add inline examples for
the DAF (Data Access Framework) profiles that are referenced from C-CDA on
FHIR Composition sections. The intent is to make it easier for implementers
to see what the referenced clinical data resources (e.g., DAF Patient,
DAF AllergyIntolerance, DAF Medication, etc.) actually look like when
embedded alongside Composition examples, instead of having to chase
references out to the DAF IG. The ticket was originally filed in 2016,
disposed in 2016 as "Considered for Future Use" (the WG declined inline
examples on Composition resources because Composition entries are designed
to be references, but agreed to consider supplying full Document Bundle
examples that bundle the DAF resource instances together with the
Composition examples), and was re-opened in March 2024 by lynn_laakso
post-STU 1.2.0 publication so that the topic could be reconsidered.
No related Jira tickets were surfaced via cross-references; only the
original gForge tracker bot post on Zulip is linked.

## Proposed Dispositions

> Note: No GitHub cross-references were surfaced for this ticket, so no

> IG is maintained by HL7 Structured Documents in the
> `HL7/ccda-on-fhir` IG repository (authoring root: typical HL7 IG
> layout under `input/` with examples under `input/examples/` and
> profile/Composition pages under `input/pagecontent/`). Concrete file
> paths below are illustrative of the standard HL7 IG layout that the
> Structured Documents WG already uses for this IG; the editor should
> confirm exact paths in the current repo before acting.

### Disposition A: Accept As Requested

#### Proposal

Add inline example resources for every DAF (now US Core / DAF-aligned)
profile that is referenced from a C-CDA on FHIR Composition section.
Concretely, for each section profile in the IG (e.g.,
`AllergiesSection`, `MedicationsSection`, `ProblemSection`,
`ImmunizationsSection`, `ProceduresSection`, `EncountersSection`,
`ResultsSection`, `VitalSignsSection`, etc.), author a fully-populated
example instance of each referenced resource (e.g., `AllergyIntolerance`,
`MedicationStatement`/`MedicationRequest`, `Condition`, `Immunization`,
`Procedure`, `Encounter`, `Observation`) that conforms to the DAF /
US-Core profile cited in the Composition entry, and render that example
inline on the Composition section's narrative page so a reader does not
need to leave the IG to see the referenced payload.

#### Justification

This is what the reporter literally asked for and would meaningfully
improve implementer experience: the current Composition examples
reference resources that the reader cannot see without navigating to the
DAF / US-Core IG and constructing matching example payloads themselves.
Inlining the referenced examples is consistent with how other HL7
companion IGs (e.g., US Core, IPS) present rich worked examples adjacent
to their structural definitions, and it directly serves the educational
purpose of the C-CDA on FHIR IG (mapping CDA narrative sections to FHIR
Composition + referenced clinical resources). The original 2016 decline
was partly because Composition entries are designed as *references* —
that design constraint does not prevent the IG from rendering the
referenced resource example inline in the page narrative; it only
prevents embedding the resource as a contained child of the Composition,
which is not what is being proposed here.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of attaching inline examples to each individual Composition
section page, satisfy the underlying need by publishing a small set of
**end-to-end Document Bundle examples** — exactly the path the WG
floated in the 2016 vote when it said it "will consider supplying full
Document Bundle examples that include the DAF resource instances as
well as the Composition examples." Each Document Bundle would include:
(a) one Composition instance using the C-CDA on FHIR Composition
profile, (b) every section's referenced resource instance conforming to
the relevant DAF / US-Core profile, and (c) the supporting Patient,
Practitioner, Organization, and Encounter resources. Render these
bundles on a new "Worked Examples" page in `input/pagecontent/`, and
cross-link from each Composition section page ("see *Continuity of Care
Document — full bundle example* for an inline AllergyIntolerance
matching this section's reference").

#### Justification

This preserves the original architectural intent of Composition entries
as references (the 2016 concern), avoids per-section duplication of the
same example resource across multiple section pages, and produces
artifacts (full Bundles) that are directly usable as conformance test
fixtures and as inputs to validators and reference implementations. It
also matches how C-CDA itself ships sample documents (CCD, Discharge
Summary, etc.), which is a familiar pattern for the C-CDA on FHIR
audience. The trade-off vs. Disposition A is that a reader on a single
Composition section page does not see the referenced example *next to*
the section text without one extra click; in exchange, the IG gets a
smaller number of canonical, internally-consistent, end-to-end examples
that are easier to maintain than dozens of per-section snippets.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — already addressed." Take the
position that the referenced DAF profiles (and their successors,
US Core profiles) already publish authoritative examples in their own
IGs, and that C-CDA on FHIR should continue to *reference* those
examples (by canonical URL or hyperlink) rather than re-author and
maintain duplicate copies inline. Update the C-CDA on FHIR Composition
section pages, where they cite a DAF / US-Core profile, to add a direct
link to that profile's example page in the upstream IG.

#### Justification

The original 2016 disposition rests on a sound architectural argument
that has not changed: Composition entries are references, and the
referenced profiles live in IGs whose maintainers own their examples.
Re-publishing those examples inside C-CDA on FHIR creates a
maintenance burden and a risk of drift every time DAF / US Core
revises a profile. With US Core having effectively replaced DAF in the
US realm since this ticket was filed, the cleanest path is to ensure
the cross-references to upstream examples are correct and discoverable,
not to duplicate them. The Justification weakens, however, if the WG
believes implementer feedback shows that the extra navigation hop is a
real adoption barrier.

---

### Recommendation

**Recommended disposition:** B

The ticket has been open for nearly a decade and was explicitly
re-opened post-STU 1.2.0 to reconsider the original "decline." The
2016 vote already pre-negotiated the alternative path ("will consider
supplying full Document Bundle examples that include the DAF resource
instances as well as the Composition examples"), so adopting it now is
both a continuation of prior WG intent and a direct response to the
reporter's underlying need. Disposition B preserves the architectural
purity that motivated the original decline (Composition entries remain
references, not containers), avoids the maintenance/drift risk of
Disposition A (duplicating per-section example snippets that must be
kept in lock-step with the upstream DAF / US Core IGs), and still
delivers a tangible artifact — full worked Document Bundles — that
implementers can validate against and use as test fixtures. Note that
"DAF" as a label is largely historical at this point; the WG should
treat the request as "inline examples for the *currently referenced*
profiles" (US Core in the US realm) when scoping the work.
