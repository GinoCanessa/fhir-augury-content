## Summary

The ticket asks the Structured Documents Work Group to consider using
`Composition.section.emptyReason` in **C-CDA on FHIR** for sections that, in
classic C-CDA, are required to carry coded entries but in practice may have
no data. The motivating concern is to avoid the C-CDA pattern of populating
those sections with several `nullFlavor`-flavored placeholder entries when
there is genuinely nothing to report. The original 2016 description is
brief and informal ("…because Brett says so"); a single 2024 comment from
**lynn_laakso** simply notes the issue should be re-opened post-STU1.2.0
publication. There are no explicitly linked Jira tickets and no cross-
referenced Jira tickets — the only artifact reference is the FHIR element
`Composition.section.emptyReason` itself, and the only meaningful Zulip
context is a 2020 `cda to fhir > No Information` thread (see below) in
which the ticket's own reporter, Rick Geimer, gives a definitive answer:
in C-CDA on FHIR all section entries were made optional, so an empty
section can simply carry narrative `Section.text` (e.g., "No
Information") and no entries, and `emptyReason` should generally be left
off — he further notes a (then-)pending FHIR-core proposal to remove the
element.

## Related Zulip Discussions

### [cda to fhir > No Information](https://chat.fhir.org/#narrow/stream/cda%20to%20fhir/topic/No%20Information)

Gay Dolin asked (2020-05-21) how to represent the C-CDA "No Information"
pattern in C-CDA on FHIR, specifically whether `Composition.section.emptyReason`
was the right mechanism (noting that the IG link to "Section Empty Reason"
404'd). Rick Geimer — the reporter of FHIR-12300 — answered (2020-05-22)
that in C-CDA on FHIR **all section entries were made optional**, since
`Composition.section` has no `nullFlavor` analog; therefore an empty
section is best expressed by populating `Section.text` (e.g., "No
Information") and leaving the entries blank. He explained that
`Composition.section.emptyReason` is a hold-over from when section entries
were modeled with a `List` resource, that if it must be populated the
`http://hl7.org/fhir/valueset-list-empty-reason.html` value set applies,
and that SDWG had voted to **remove the element in a subsequent FHIR
release** — recommending implementers simply leave it off. No dissent was
recorded; the thread closed with that single response.

Participants: Gay Dolin, Rick Geimer (Brett Marquard was @-mentioned but
did not reply in this thread).

### Disposition A: Accept As Requested

#### Proposal

In the C-CDA on FHIR IG, formally describe and recommend
`Composition.section.emptyReason` as the mechanism for representing
sections that in classic C-CDA require coded entries but have no data.
Concretely:

- Add IG narrative (in the Composition profile / "Sections" guidance
  page) stating that, where a C-CDA template requires entries but no
  data exists, the corresponding FHIR section SHALL include
  `Composition.section.emptyReason` instead of populating the section
  with nullFlavor-bearing placeholder entries.
- Bind `Composition.section.emptyReason` (in the C-CDA on FHIR
  Composition profile) to `http://hl7.org/fhir/ValueSet/list-empty-reason`
  (preferred or extensible binding, matching the base FHIR binding
  strength).
- Mirror the C-CDA `nullFlavor` → `list-empty-reason` mapping table
  (NI / NA / UNK / MSK / NASK …) in the IG mapping page so the
  round-trip from C-CDA "no information" sections is unambiguous.
- Add at least one IG example instance showing an empty section that
  uses `emptyReason` with the appropriate code and a `Section.text`
  narrative.

#### Justification

- Directly addresses the reporter's stated concern about avoiding the
  "bag of nullFlavors" anti-pattern when migrating C-CDA documents.
- `Composition.section.emptyReason` is the element FHIR provides for
  exactly this purpose; using it preserves the *reason* a section is
  empty (no information vs. unknown vs. masked vs. not asked), which a
  pure narrative-only approach loses.
- An explicit binding plus a mapping table improves
  C-CDA ↔ FHIR round-trip fidelity, which is one of the IG's primary
  value propositions.

### Disposition B: Alternative Approach — Narrative-only empty section, with optional `emptyReason`

#### Proposal

Adopt the pattern Rick Geimer described in the `cda to fhir > No
Information` Zulip thread, and document it as the IG's normative
guidance:

- Update the IG's "Sections" guidance to state that an empty section is
  represented by populating `Composition.section.text` with an
  appropriate narrative (e.g., "No Information") and **omitting all
  entries**. All entries in C-CDA on FHIR are already optional, so this
  is conformant today.
- Keep `Composition.section.emptyReason` available but mark it
  **optional / discouraged** in the C-CDA on FHIR Composition profile,
  with IG narrative noting that (a) the element is a holdover from when
  section entries were modeled as a `List`, and (b) implementers MAY
  populate it (using `http://hl7.org/fhir/ValueSet/list-empty-reason`)
  when they need to convey *why* the section is empty, but it is not
  required for the no-data case.
- Provide one example for each style (narrative-only; narrative +
  `emptyReason`).

#### Justification

- This is exactly the answer the original reporter (Rick Geimer) gave
  the community when the question came up in practice in 2020 — i.e.
  the de-facto interpretation already in the field.
- It avoids over-prescribing a sender behavior that the IG has
  intentionally relaxed (entries-optional), and it preserves the option
  to use `emptyReason` for senders that genuinely need to convey a
  *reason*.
- Lower implementer burden than Disposition A: no new conformance
  obligations on existing senders; receivers must already accept
  sections without entries.

#### Gotcha addressed

The same Zulip exchange flagged that SDWG (FHIR-core) had at one point
voted to **remove** `Composition.section.emptyReason` in a future FHIR
release. Disposition B explicitly handles that risk: by treating
`emptyReason` as optional/discouraged guidance rather than a SHALL, the
IG remains forward-compatible if/when the base element is removed,
while still giving implementers a documented path today. Before
finalizing, the workgroup should confirm the current status of the
element in the latest FHIR core (R5 / R6 ballot) — if it has in fact
been removed or deprecated, Disposition A becomes infeasible and
Disposition B is the only viable "use this mechanism" option.

### Disposition C: Decline

#### Proposal

Close FHIR-12300 as **Persuasive with Modification → No Change**, on
the grounds that:

- C-CDA on FHIR has already addressed the underlying need by making all
  `Composition.section` entries optional, so a "no information" section
  is representable today using `Section.text` alone.
- `Composition.section.emptyReason` is a vestigial element from an
  earlier FHIR design and is reportedly slated for removal in a future
  base FHIR release; baking it into IG conformance would create
  forward-compatibility debt for a problem that is already solved.
- The ticket's prose is informal ("…because Brett says so"), there are
  no follow-on use cases in Jira, and the only substantive community
  discussion (the `cda to fhir > No Information` thread) concluded that
  no IG change is needed.

Resolution comment should point implementers to the
narrative-only-section pattern and to
`http://hl7.org/fhir/ValueSet/list-empty-reason` as an *informative*
option for senders that wish to convey a reason.

#### Justification

- The mechanism the ticket asks the IG to adopt may be removed from
  base FHIR, so adopting it normatively could create churn.
- The reporter himself, in 2020, recommended *not* using `emptyReason`.
- Declining keeps the IG aligned with current implementation practice
  and avoids new conformance burden.

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach — narrative-only empty section, with optional `emptyReason`)**

Disposition B is the best fit because it captures the de-facto answer
the IG's own community has been giving for years — the original
reporter, Rick Geimer, told another implementer in the
`cda to fhir > No Information` thread that the right pattern is a
narrative-only section, with `emptyReason` left off. Promoting that
guidance to documented IG narrative resolves the ticket without
imposing new conformance obligations and without prematurely declining
a request that does have a real interoperability concern behind it
(round-tripping C-CDA "no information" sections).

It is preferable to **A** because Composition.section.emptyReason has
been flagged for possible removal from base FHIR; mandating its use in
the IG would create forward-compatibility risk for a problem that is
already solvable with `Section.text`. It is preferable to **C**
(outright decline) because the ticket is asking a legitimate
representational question that deserves an explicit answer in the IG
rather than an undocumented "ask on Zulip" outcome — and because
keeping `emptyReason` documented as an *optional* signal preserves
value for senders that need to convey *why* a section is empty.

Before finalizing, the workgroup should (a) confirm the current status
of `Composition.section.emptyReason` in the latest FHIR core ballot to
make sure the "optional, discouraged" framing is still accurate, and
(b) decide whether the IG narrative should also include the explicit
C-CDA `nullFlavor` → `list-empty-reason` mapping table for the senders
that do choose to populate `emptyReason`.
