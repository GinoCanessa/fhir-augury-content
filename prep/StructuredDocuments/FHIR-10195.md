## Summary

The C-CDA on FHIR Implementation Guide's Immunization Section profile, as
authored against the Composition resource, does not declare an
`entry` slice/reference that points at the DAF-Immunization profile.
Corey Spears, the reporter, expected the Immunization section's
`Composition.section.entry` element to be constrained so that it
SHOULD contain zero-or-more references to U.S. DAF Immunization
(`http://hl7.org/fhir/StructureDefinition/daf-immunization`), mirroring
the pattern other C-CDA-on-FHIR sections use to bind their entries to
domain-specific U.S. profiles. The ticket asks the Structured Documents
work group to add that constraint to the Immunization Section profile.
The ticket was filed in June 2016, originally tied to the
`build.fhir.org/ccda` build (per Lynn Laakso's later comment), and was
re-opened in March 2024 with the note "re-open post STU1.2 publication"
so that the fix can land in the next C-CDA on FHIR cycle.

There are no cross-referenced Jira tickets — the only non-FHIR
cross-reference is an automated `FHIR Bot` tracker-item announcement on
Zulip from when the issue was originally filed. No related GitHub PRs,
issues, or commits are recorded in Augury for this ticket.

## Proposed Dispositions

> Note: Augury surfaces no related GitHub repos for this ticket, so the
> repo and authoring root for the change cannot be cited from a
> persisted briefing. The C-CDA on FHIR IG is published by Structured

### Disposition A: Accept As Requested

#### Proposal

Update the Immunization Section profile in the C-CDA on FHIR IG
(`Composition` profile for the C-CDA Immunization Section) so that
`Composition.section.entry` is sliced/constrained with an
`Immunization`-typed entry whose `type.profile` is
`http://hl7.org/fhir/StructureDefinition/daf-immunization`, with
cardinality `1..*` and a `SHOULD` definition matching the wording in the
ticket. Apply the same edit pattern the IG already uses for other
section profiles that bind entries to U.S. profiles (Problem Section →
DAF Condition, Medication Section → DAF MedicationStatement, etc.) so
that Immunization aligns with its peers.

#### Justification

The reporter is pointing out a consistency gap: every other C-CDA on
FHIR section profile binds its `Composition.section.entry` to the
relevant U.S./DAF profile, and the Immunization Section was simply
missed. Adding the slice is mechanical, additive (it tightens an
otherwise-open `entry`), and aligns the IG with its own established
pattern, which is exactly what the work group's editorial guidance for
the C-CDA on FHIR IG calls for. Lynn's 2024 re-open note ("post STU1.2
publication") signals the work group has already accepted that this
should be done — they were just deferring it past the STU 1.2 freeze.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than hard-binding the Immunization Section entry to the legacy
DAF-Immunization profile (`http://hl7.org/fhir/StructureDefinition/daf-immunization`),
slice `Composition.section.entry` to a `Reference(Immunization)` and
bind the profile to the **current** US-Core Immunization profile
(`http://hl7.org/fhir/us/core/StructureDefinition/us-core-immunization`),
keeping cardinality `1..*` and a `SHOULD` definition. If the work group
still wants DAF compatibility, declare an additional slice for the DAF
profile rather than making it the only allowed profile.

#### Justification

The DAF IG has been effectively superseded by US-Core in U.S.
implementer practice; pinning a brand-new C-CDA on FHIR slice to
`daf-immunization` in 2024+ would freeze a reference to an IG most
implementers no longer target, and would diverge from the direction
other Structured Documents profiles have taken when refreshed
post-STU 1.2. Sliced/multi-profile entries are a well-supported FHIR
mechanism and let the IG honor the original intent ("there should be a
typed Immunization entry here") without re-anchoring on a stale U.S.
profile. This is the "do the right thing now" path — it costs more
review time than Disposition A but avoids needing a follow-up ticket to
swap DAF for US-Core a release later.

---

### Disposition C: Decline

#### Proposal

Decline the change as written, on the grounds that the requested edit
references the deprecated DAF-Immunization profile and that the
Immunization Section's current open `Composition.section.entry`
already permits a DAF-Immunization (or US-Core Immunization) reference
without an explicit slice. Close the ticket as "Persuasive with
modification → Not Persuasive" if the work group does not also want to
adopt Disposition B.

#### Justification

A pure decline is defensible only if the work group concludes that
(a) the DAF profile path the ticket asks for is no longer the right
target, and (b) tightening the section profile is unnecessary because
implementers can already include the right reference under the open
`entry`. This is the weakest of the three options because it leaves
the documented inconsistency between Immunization and the IG's other
section profiles unresolved, and it ignores the 2024 re-open note that
treated the work as deferred-but-needed.

---

### Recommendation

**Recommended disposition:** B

Disposition A faithfully implements the ticket but bakes a reference to
the now-obsolete DAF-Immunization profile into a 2025+ C-CDA on FHIR
release; that buys a guaranteed follow-up ticket the moment the IG is
next reviewed against U.S. realm guidance. Disposition C ignores the
fact that Lynn Laakso explicitly re-opened this in 2024 to be addressed
after STU 1.2, which reads as work-group acceptance that **something**
needs to change. Disposition B honors the reporter's underlying intent
("the Immunization Section should pin its entry to the U.S. Immunization
profile, like the other sections do"), aligns with the U.S. realm's
move from DAF to US-Core, and uses standard FHIR slicing so the work
group can still keep a DAF slice if any consumer is known to depend on
it. The work group should confirm the target profile (US-Core
Immunization version) and then apply the standard section-entry
slicing pattern already used elsewhere in the IG.
