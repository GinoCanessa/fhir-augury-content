## Summary

Nick Radov (the reporter) is requesting that the **US C-CDA on FHIR** IG add
explicit guidance on how to map a CDA section's narrative text to FHIR. He
believes the intent is that the narrative block of each CDA `<section>`
should be carried over to the equivalent `Composition.section.text` element
in FHIR, and he is asking the Structured Documents work group to state that
mapping rule unambiguously in the IG. The ticket also notes a process
issue: the comment was raised against IG version 1.1.0 but the Jira "Raised
in Version" field only offered 1.0, and the reporter is asking that 1.1.0
be added as a selectable value. There are no cross-referenced Jira
tickets, no Zulip discussions, and no GitHub PRs/issues linked to this
ticket — the only cross-reference surfaced is a FHIR element mention of
`Composition.section.text`, which is the very element the reporter is
proposing as the mapping target.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the US C-CDA on FHIR IG to add an explicit mapping statement that
the narrative content of each CDA `<section>` (the section's `<text>`
narrative block) maps to the corresponding FHIR
`Composition.section.text` (an XHTML `Narrative`) when transforming a CDA
document into a FHIR `Composition`. Specifically:

- Add a sentence (and ideally a small example) in the IG's
  general/section-mapping guidance page making the
  `CDA section/text → Composition.section.text` mapping explicit and
  normative-styled (SHOULD / SHALL as the WG decides).
- Note that the `Narrative.status` should be set per FHIR rules
  (e.g., `additional` when other structured data in `section.entry`
  also conveys the same content, `generated` only when the narrative is
  re-generated from entries).
- Address the reporter's secondary request by adding `1.1.0` (and any
  later published versions) to the Jira "Raised in Version" picklist
  for this project so future comments can be filed against the version
  that actually contains the issue.

#### Justification

This matches the existing FHIR design intent: `Composition.section.text`
is defined as a human-readable `Narrative` for the section, which is the
direct semantic counterpart of the CDA section's `<text>` narrative
block. Documenting it explicitly in the IG removes ambiguity for
implementers building CDA→FHIR transforms, prevents divergent
interpretations (some implementers today drop section narrative entirely,
others stuff it into `Composition.text`), and is consistent with how
CDA-on-FHIR equivalents are described elsewhere in the spec. The change
is small, additive, and non-breaking. The Jira picklist update is a
trivial admin fix that improves the quality of future ballot/tracker
comments.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding a one-line statement, publish a dedicated
"Section narrative mapping" subsection in the IG that covers the full
range of cases the reporter's question implies:

1. Default rule: CDA `section/text` → `Composition.section.text`
   (with `Narrative.status` guidance as in Disposition A).
2. Handling of CDA narrative `<reference value="#id"/>` links to entries
   — recommend retaining those references in the migrated XHTML so that
   `Composition.section.entry`-referenced resources can be cross-linked
   back to the narrative.
3. Handling of styling / tables / lists / `<renderMultiMedia>` —
   describe which CDA narrative constructs are preserved as XHTML, which
   are normalized, and which (if any) are dropped.
4. Round-trip considerations: when converting FHIR back to CDA, where
   the `Composition.section.text` content lands and how
   `Narrative.status` is interpreted.
5. Apply the same Jira "Raised in Version" picklist fix from
   Disposition A.

#### Justification

The reporter's concrete ask is narrow, but the underlying need
("how do I map CDA section narrative to FHIR?") is broader than a single
sentence can answer, and implementers consistently get tripped up by the
edge cases above (entry references, multimedia, status). Spending the
authoring effort once on a proper subsection avoids a stream of
follow-on tickets and is more useful than a one-liner. The trade-off
versus Disposition A is more authoring effort and longer time to
publish; the benefit is materially better implementer guidance and
fewer interoperability defects in CDA-on-FHIR transforms.

---

### Disposition C: Decline

#### Proposal

Decline to add new guidance, on the basis that the mapping the reporter
is asking the IG to state explicitly is already implied by the base FHIR
definition of `Composition.section.text` (a `Narrative` representing the
human-readable content of the section) and by the general CDA-on-FHIR
principle that CDA narrative is preserved in the corresponding FHIR
`Narrative` element. Close the ticket as "Not Persuasive" with a
pointer to the existing definitions, and separately raise the Jira
"Raised in Version" picklist update with the tracker administrators as
an admin task rather than an IG change.

#### Justification

Declining is defensible if the WG judges that the existing FHIR
definitions plus general CDA-on-FHIR conventions are sufficient and that
adding restating-the-obvious text inflates the IG without changing
implementer behavior. However, this position is weak in practice because
the reporter is an experienced implementer who still found the mapping
non-obvious, which is itself evidence that the existing text is not
sufficient.

---

### Recommendation

**Recommended disposition:** A (with a stretch goal of B if authoring
capacity allows in the next ballot cycle).

The ask is small, low-risk, additive, and directly improves implementer
clarity on a question that an experienced implementer (the reporter)
explicitly hit. Disposition A pays that down immediately with minimal
churn and unblocks anyone writing CDA→FHIR transforms today. Disposition
B is the better long-term answer because the surrounding edge cases
(narrative-to-entry references, multimedia, `Narrative.status`,
round-tripping) are exactly where current CDA-on-FHIR implementations
diverge in practice — but it is more authoring work and shouldn't block
the simple statement the reporter actually requested. Disposition C
should be rejected: the ticket itself is evidence that the current IG
text leaves the mapping ambiguous. The Jira "Raised in Version"
picklist fix should be applied regardless of which disposition the WG
chooses.
