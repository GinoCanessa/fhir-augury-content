## Summary

Giorgio Cangioli proposes that the International Patient Summary (IPS) FHIR
implementation guide publish dedicated, section-specific pages for each IPS
section. Today the IPS specification documents the sections largely through
the composite Composition profile and the section table, which can make it
hard for readers to find, in one place, the narrative description, the
required/optional content, the entries permitted in that section, the
relevant terminology bindings, and the example assets associated with a
single IPS section. The ticket asks the work group to add per-section pages
that pull these assets together to improve readability and navigability of
the IG. There are no cross-referenced Jira tickets that broaden or modify
the request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a dedicated page in the IPS IG for each Composition section defined in
the IPS Composition profile (e.g., Allergies and Intolerances, Medication
Summary, Problem List, Immunizations, History of Procedures, Medical
Devices, Diagnostic Results, Vital Signs, Past History of Illness,
Functional Status, Plan of Care, Social History, Pregnancy History,
Advance Directives). Each page would include: (a) the section's purpose
and narrative description, (b) cardinality and Must Support flags as
applied in IPS, (c) the entry profile(s) referenced from the section, (d)
the value sets / code systems bound for the section's `code` and entry
elements, (e) IPS-relevant guidance/usage notes, and (f) links to the
canonical IPS examples that exercise the section. Index these pages from
the Composition profile's section table and from the IG's left
navigation, so each row in the section table links to its dedicated page.

#### Justification

Section-specific pages bring together what is currently scattered across
the Composition profile, the entry profiles, the value-set artifacts,
and the examples folder. This is a pure documentation/structure
enhancement that does not change conformance and is consistent with how
other IGs (e.g., US Core's per-profile pages, IPA's per-profile pages)
present assets per logical unit. The reporter's stated motivation —
readability — is directly addressed.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding standalone section pages, enhance the existing IPS
Composition profile page so that each row of the section table expands
in-place (or links to an in-page anchor) to a structured block showing
the entry profile(s), bound value sets, narrative guidance, and example
links for that section. Pair this with a single new "IPS sections at a
glance" reference page that lists all sections in a uniform table with
direct deep-links into the entry profile and value-set pages. This keeps
the section content anchored to the Composition profile (which is the
normative carrier) and avoids duplicating content across many new pages.

#### Justification

The information the reporter wants is mostly already authored elsewhere
(entry profiles, value sets, examples). Duplicating it on per-section
pages risks drift between the section page and the underlying profile
content as the IG evolves. An enhanced section table plus a single
reference page achieves the readability goal with substantially less
authoring and maintenance burden, and it keeps the Composition profile
page as the single source of truth for section semantics.

---

### Disposition C: Decline — Not Persuasive / Already Addressed

#### Proposal

Decline the request and close as "Not Persuasive". The IPS IG already
documents each section through the Composition profile's section table,
the per-entry profiles, and the IPS examples; readers can navigate from
the Composition profile to the bound entry profile and from there to its
value sets and examples. No new page artifacts are introduced.

#### Justification

The original work-group disposition (2018-08-02, 16-0-3) was effectively
"defer as enhancement" rather than a substantive design objection, and
in the years since the IG has matured its Composition profile and
profile-per-section structure. Adding parallel section pages would
duplicate content already produced by the IG publisher and create an
ongoing maintenance tax for a benefit that is largely cosmetic. If
specific navigability gaps remain, they can be addressed with targeted
edits rather than a new page family.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the reporter's underlying need — making it easy to
see, per IPS section, the profile, terminology, and examples in one
place — while avoiding the duplication and maintenance cost flagged in
Disposition C. It also preserves the Composition profile as the single
normative anchor for section semantics, which is important because the
IPS section list is itself part of the conformance surface. Disposition A
is acceptable if the work group judges that authoring per-section pages
is worth the long-term upkeep, but the alternative achieves most of the
readability win at a fraction of the editorial cost. Outright decline
(Disposition C) is hard to defend given that the work group explicitly
re-opened the ticket post-STU.
