## Summary

The ticket asks the IPS (International Patient Summary) authors to add
narrative guidance to the IPS MedicationStatement / Medication profiles
explaining when to populate `medication[x]` as a `CodeableConcept`
(`medicationCodeableConcept`) versus as a `Reference` to a `Medication`
resource (`medicationReference`). The reporter, Giorgio Cangioli (an IPS
co-editor), notes that today the IPS spec exposes the choice element
without describing how an implementer should choose between the two
forms in an IPS context. No related Jira tickets, GitHub issues, or
substantive Zulip discussions surfaced through the cross-reference
search; the only cross-reference is the automated FHIR Bot tracker
announcement on `#committers > tracker-item`. Note that R4 dropped the
`[x]` choice on `MedicationStatement` and replaced it with a single
`medication : CodeableReference` element, so the disposition needs to
account for which IPS publication (R4 / R4B / R5) the guidance is being
added to.

## Proposed Dispositions

Because no GitHub cross-references exist, none of the dispositions
below name a specific repo path. They instead target the IPS
specification source (HL7/fhir-ips on GitHub, currently outside the
cached repo set). The IPS authoring artifacts in scope are the
`MedicationStatement-uv-ips` profile and its companion
`Medication-uv-ips` / `MedicationRequest-uv-ips` profiles, plus the
narrative section of the IPS guide that introduces the medications
section.

### Disposition A: Accept As Requested

#### Proposal

Add a normative-tone "Usage" paragraph to the IPS
`MedicationStatement-uv-ips` profile page (and a parallel paragraph on
`MedicationRequest-uv-ips`) that documents the expected use of the
`medication[x]` choice (R4 / R4B IPS) and the `medication`
`CodeableReference` (R5 IPS) as follows:

- **Use `medicationCodeableConcept` (or `CodeableReference.concept` in
  R5) when** the medication can be sufficiently identified by a code
  from one of the IPS-required value sets (e.g., the IPS
  pharmaceutical product / substance value sets, ATC, SNOMED CT,
  RxNorm, EDQM) and no further structured information about strength,
  form, ingredients, batch, or manufacturer is needed to convey the
  patient's medication.
- **Use `medicationReference` (or `CodeableReference.reference` in R5)
  to a contained `Medication-uv-ips` instance when** the medication
  must be described in more structured detail than a single code
  allows — for example, compounded products, products that need
  explicit `Medication.ingredient`, products requiring `form`,
  `strength`, `batch`, or non-coded brand/manufacturer information, or
  when the IPS-required value sets do not yield an adequate code.
- Clarify that an out-of-bundle `Reference` to a server-side
  `Medication` resource is **not appropriate** for an IPS document
  bundle (which must be self-contained); references must resolve
  inside the bundle, typically via `contained`.
- Cross-link to the IPS "absent and unknown data" guidance for the
  case where no medication code is available.

#### Justification

This is exactly what the reporter (an IPS co-editor) asked for, and it
matches the long-standing FHIR convention for `[x]` choice elements
where the codeable form is preferred for the simple case and the
reference form is reserved for the structured case. The IPS
specification already requires a self-contained document Bundle, so
restricting `medicationReference` to contained `Medication` instances
preserves the IPS conformance model. Adding the guidance in the
profile narrative (rather than as a constraint) keeps the change
non-breaking and consistent with how IPS handles other choice elements
such as `Observation.value[x]`.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding prose, also tighten the profile in a
machine-checkable way: on `MedicationStatement-uv-ips` (and siblings),
add an `obligation` / `mustSupport` style annotation plus a profile
invariant that says "if `medication.ingredient`, `medication.form`,
`medication.batch`, or `medication.amount` would be needed to convey
the patient's medication, the `medicationReference` form **SHALL** be
used pointing at a contained `Medication-uv-ips`; otherwise
`medicationCodeableConcept` SHOULD be used with a code from the
IPS-required value sets." Keep R4/R4B using the `[x]` choice and, for
the R5 IPS, write the equivalent rule against
`CodeableReference.concept` vs `CodeableReference.reference`. Add the
same usage paragraph from Disposition A as the human-readable companion.

#### Justification

The ticket's underlying need is to reduce implementer ambiguity. Pure
narrative guidance (Disposition A) is easy to ignore and is hard to
test in IPS validators. A profile-level rule, even a `SHOULD`-strength
one, gives validator authors and IG QA something concrete to check and
gives consumers a clearer expectation of which form they will see.
Trade-offs vs. A: it is more work to author and review (especially
across the R4 / R4B / R5 IPS variants), it has a small risk of
flagging existing conformant instances as warnings, and it depends on
the WG agreeing that the "structured detail required" criterion is
crisp enough to invariant-ize. R5's `CodeableReference` makes this
cleaner because the shape is uniform.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Mod / Not Persuasive — guidance lives
upstream**: decline to add IPS-specific guidance on the grounds that
the choice between `medicationCodeableConcept` and
`medicationReference` is governed by the base FHIR
`MedicationStatement` / `MedicationRequest` resource pages and by the
Medications module guidance, not by individual IGs. Recommend that the
reporter raise the request against the base FHIR resource (FHIR Infra
/ Pharmacy WG) so all IGs benefit from one consistent answer, and
mark FHIR-16287 as superseded once that work is filed.

#### Justification

A defensible reason to decline is *scope*: choice-element selection
guidance for `medication[x]` is not unique to IPS, and adding it only
to IPS risks divergence from how Pharmacy / FHIR-Infra describes the
same element for other IGs. There is also a partial existing mechanism
— R5 collapsed the choice into `CodeableReference`, which itself
encodes the design intent — so for the R5 IPS the guidance is largely
already implicit in the type definition. The downside of declining is
that the IPS reader (a clinician or IPS implementer who is not
expected to read base FHIR pages) is left without the answer in the
guide they actually consult.

---

### Recommendation

**Recommended disposition:** A

The reporter is an IPS editor, the ticket has been deliberately
re-opened post-STU-update by the WG (lynn_laakso, 2022-11-22), and the
ask is narrowly scoped to documentation. Disposition A delivers
exactly that with no breaking change and no validator churn, and it
fits the existing IPS pattern of using profile narrative to explain
intended use of choice elements. Disposition B is attractive long-term
but should be a follow-up ticket — turning the prose into an invariant
benefits from first having the prose agreed and reviewed in the IG,
and from coordinating with Pharmacy WG on whether the same rule
belongs upstream. Disposition C under-serves IPS implementers, who
look to the IPS guide rather than the base spec for the "what do I put
here" answer; even if Pharmacy WG also writes guidance, the IPS guide
should still answer the question in context. Recommend accepting and
authoring the paragraph against both the R4/R4B IPS profiles
(`medication[x]`) and the R5 IPS profile (`CodeableReference`), with a
forward-pointer to a separate ticket if the WG later wants to make the
rule machine-checkable.
