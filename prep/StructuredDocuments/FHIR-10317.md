## Summary

FHIR-10317 reports a systemic identifier-style problem in the **US C-CDA on
FHIR** implementation guide: every place that the guide references the LOINC
code system, it uses the OID-based URN form (i.e. `urn:oid:2.16.840.1.113883.6.1`)
rather than the FHIR-canonical URL `http://loinc.org`. The reporter (Corey
Spears) notes that this is inconsistent with FHIR conventions for code-system
identifiers and that the issue is repeated throughout the entire guide — not
isolated to a single profile or example. A 2024-03-12 comment from
lynn_laakso indicates the ticket should be re-opened post STU1.2 publication
and points at one example resource
(`build.fhir.org/ccda/ccda-referral-note-composition.html`) where the URL form
should appear instead of the OID URN. There were no related Jira tickets
returned by the cross-reference search; the only cross-reference is an
automated tracker-bot post on Zulip from 2016 (no human discussion).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Perform a guide-wide sweep of the **US C-CDA on FHIR** IG and replace every
LOINC code system reference of the form
`urn:oid:2.16.840.1.113883.6.1` with the FHIR-canonical
`http://loinc.org`. Concretely:

1. **StructureDefinitions / profiles** — for any element with a fixed,
   pattern, or required-binding code system value (e.g. `Coding.system`,
   `CodeableConcept.coding.system`, terminology-binding `system` slices),
   change `urn:oid:2.16.840.1.113883.6.1` → `http://loinc.org`.
2. **ValueSet / ConceptMap / CodeSystem resources** authored in the IG —
   change any `compose.include.system` / `concept.system` /
   `targetSystem` references to LOINC over to `http://loinc.org`.
3. **Examples** — update every `Coding.system` / `Identifier.system` in
   example resources (e.g. the Referral Note Composition example called
   out in the 2024-03-12 comment) to `http://loinc.org`.
4. **Narrative pages** — update the guide narrative, mapping tables, and
   any `system` snippets shown in code blocks so documentation matches
   the resources.

Treat the URN form as a non-canonical representation that the
publisher/validator may still recognise but that the IG no longer
authors. Add a brief migration note in the change-log so implementers
know the URN form is being retired.

#### Justification

The FHIR specification consistently designates `http://loinc.org` as the
canonical code-system URL for LOINC, and the FHIR validator, LOINC
content modules, and the `fhir.tx.fhir.org` terminology service all
expect that URL. Authoring against the URN form makes profiles and
examples inconsistent with the rest of the FHIR ecosystem (US Core, QI
Core, Da Vinci IGs, the base spec's terminology bindings), undermines
ValueSet expansion against tx.fhir.org, and is exactly the inconsistency
the reporter raised. The 2024 follow-up comment indicates the WG already
intends to address this post-STU1.2.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt `http://loinc.org` as the canonical LOINC URL going forward
(same target end-state as A), but **stage the change** rather than doing
a one-shot sweep:

1. In the next ballot cycle, change all **profile** bindings and
   **value-set / code-system resource** references to `http://loinc.org`
   (highest-impact for downstream conformance).
2. Update **examples** in the same cycle and add a short migration note
   in the IG describing the OID-URN → URL change and pointing implementers
   at `http://hl7.org/fhir/terminologies-systems.html`.
3. Defer purely **narrative / illustrative snippet** updates to a
   follow-up technical-correction publication if the bulk of the work
   exceeds the available editorial bandwidth.
4. Optionally, ship a `ConceptMap` (or document in the IG's mapping
   page) declaring `urn:oid:2.16.840.1.113883.6.1` and `http://loinc.org`
   as the same code system, so legacy CDA-derived instances remain
   interpretable.

#### Justification

The end-state is identical to Disposition A, so the FHIR-consistency
benefit is preserved, but staging reduces the risk of an editorial sweep
introducing collateral breakage in profiles or examples that the WG
hasn't time to re-verify before publication. Explicit equivalence
between the URN and the URL also helps tools that round-trip C-CDA
documents into the FHIR representation, which is a real concern for a
"C-CDA on FHIR" IG specifically.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep `urn:oid:2.16.840.1.113883.6.1` as the
authored LOINC code-system identifier in US C-CDA on FHIR, on the
rationale that the IG's purpose is to express CDA semantics in FHIR and
CDA inherently uses OID-based identifiers. Add a clarifying note in the
IG's terminology section that the URN form is intentional for this IG.

#### Justification

A defensible decline argument is that **CDA is an OID-based world** and
"C-CDA on FHIR" sits at the seam between the two — preserving the OID
URN keeps a 1:1 correspondence with the source CDA documents and avoids
introducing a transformation step (URN → URL) for every code element
when ingesting CDA. Some downstream consumers that already round-trip
between CDA and the IG may be relying on the URN form.

---

### Recommendation

**Recommended disposition:** A (with the equivalence note from B as a small
add-on)

The FHIR ecosystem has a single canonical URL for LOINC
(`http://loinc.org`), and every other US-realm IG (US Core, QI Core,
Da Vinci, mCODE, etc.) authors against it. Continuing to author the URN
form in C-CDA on FHIR makes the IG an outlier, breaks easy reuse of
profiles and value sets across IGs, and degrades validation/expansion
against tx.fhir.org — exactly the consistency problem Corey Spears
raised and that lynn_laakso flagged for re-open after STU1.2. The CDA
provenance argument behind Disposition C is real but is best handled by
the publisher/validator's existing OID↔URL equivalence handling and, if
needed, a small `ConceptMap` borrowed from Disposition B, rather than
by perpetuating an inconsistency in every Coding.system reference. The
work is editorial sweep + example regeneration; staging it (B) is a
sensible execution tactic but should not change the recommended target
state, which is full adoption of `http://loinc.org` throughout the IG.
