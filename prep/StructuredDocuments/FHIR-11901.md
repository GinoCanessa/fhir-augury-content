## Summary

The ticket asks the Structured Documents work group to add a **computable
mapping from C-CDA to CDA on FHIR** in the US C-CDA on FHIR
implementation guide. As filed by Sean Muir in 2016 (originally GForge
ccda #86), the request is brief: provide explicit, machine-processable
mappings between C-CDA (CDA R2) constructs and the FHIR resources /
profiles defined by the CDA on FHIR IG, so that implementers can drive
transformations and conformance checks from the IG itself rather than
recreating the mapping per-implementation.

The ticket was originally dispositioned **Considered for Future Use**
(Sarah Gaunt / Lisa Nelson, 9-0-0, 2016-11-10) on the grounds that
comprehensive mappings were out of scope for the initial work, with the
commenter encouraged to contribute. Lynn Laakso re-opened the ticket on
2024-03-12 ("re-open post STU1.2.0 publication"), explicitly reverting
that prior resolution so it can be reconsidered now that STU1.2.0 of
US C-CDA on FHIR has shipped. There are no other linked Jira tickets
and no substantive Zulip or GitHub discussion attached to the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add computable mappings from C-CDA (CDA R2) templates to the CDA on FHIR
profiles in the **US C-CDA on FHIR** IG. Concretely:

- For each C-CDA template referenced by the IG (CCD, Care Plan, Discharge
  Summary, plus the section- and entry-level templates such as
  Allergies / Medications / Problems / Results / Vital Signs / Immunizations /
  Procedures / Encounters / Plan of Treatment), publish a
  **StructureMap** (FHIR Mapping Language) from the C-CDA template to
  the corresponding US C-CDA on FHIR `Bundle` / `Composition` /
  resource profile.
- Use **`StructureDefinition.mapping`** entries on each
  US C-CDA on FHIR profile to declare the C-CDA template OID as a
  mapping source, and add element-level `mapping` slices pointing at
  the C-CDA XPath / element for each mapped element.
- Publish the supporting **ConceptMaps** (e.g., C-CDA value sets ↔ FHIR
  value sets used by the IG) so the StructureMaps can call
  `$translate`.
- Render the mappings into the IG narrative (per-profile "Mappings"
  tab) and ship the StructureMap / ConceptMap resources in the IG
  package so they are downloadable, executable, and testable against
  reference C-CDA documents.
- Treat the mapping set as **normative content of the IG** for each
  template that has a corresponding US C-CDA on FHIR profile; mark
  any template without a current FHIR counterpart as out-of-scope for
  this iteration.

#### Justification

The reporter's underlying need — that implementers should not have to
re-derive the C-CDA → CDA-on-FHIR mapping by hand — is exactly the
problem FHIR's mapping infrastructure (`StructureMap` /
`StructureDefinition.mapping` / `ConceptMap`) was designed to solve, and
it is the mechanism US Core, IPS, and other major IGs use to publish
computable mappings. Now that STU1.2.0 has shipped (the explicit
trigger Lynn Laakso cited when re-opening), the profile set is stable
enough to map against without churn. Doing this inside the IG also
keeps the mappings versioned with the profiles they bind to, which is
the only way to keep them trustworthy over time.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than commit the IG itself to publishing a complete normative
StructureMap set, **publish a "Mapping Companion" profile pack /
informative annex** that:

- Adds **`StructureDefinition.mapping`** entries (identity = the C-CDA
  template OID) to each US C-CDA on FHIR profile, with element-level
  `mapping.map` strings pointing at the C-CDA XPath. This is
  lightweight, human-readable, renders for free in the IG, and creates
  no executable-conformance burden.
- Ships an **informative** set of StructureMaps and ConceptMaps as a
  **separate companion package** (or as IG examples), explicitly marked
  non-normative, covering the highest-value templates first (CCD
  header + Allergies, Medications, Problems, Results, Vital Signs).
- Establishes a community contribution process (the original 2016
  disposition's "encouraged to help" language) so that additional
  templates can be added incrementally without each addition requiring
  a ballot cycle.
- Leaves the door open to promote the companion to normative in a
  future major version once it is implementation-tested.

#### Justification

A complete, normative C-CDA → CDA-on-FHIR StructureMap set is a large
piece of work — historically the original committee declined it on
exactly that basis ("not in scope for the initial work … if time and
resources are available"). The specification text alone does not change
that resourcing reality. Disposition B preserves the implementer
benefit (every profile gets traceable mapping rows + a starter
StructureMap) while keeping conformance scope honest, avoiding the risk
of shipping incomplete or wrong mappings as normative, and explicitly
inviting the community help the original disposition called for. This
also matches how other IGs (e.g., IPS, mCODE) have introduced cross-
standard mappings without taking on full normative responsibility on
day one.

---

### Disposition C: Decline

#### Proposal

Persistent Not Persuasive — keep the original 2016-11-10 disposition.
The work group does not have committed resources to author and maintain
a comprehensive computable C-CDA → CDA-on-FHIR mapping, and no
specific, scoped proposal (which templates, which mapping format,
maintenance owner) has been put forward in the eight years the ticket
has been open. Mapping work, where it occurs, can be done in
implementer-led projects (e.g., open-source transformers) and
contributed back to a future version of the IG via concrete pull
requests rather than via this open-ended request.

#### Justification

The ticket as written is a one-line aspiration with no scope, no
proposed mechanism, and no committed contributor. Re-opening it
without a specific proposal effectively re-asks the same question the
2016 vote answered 9-0-0. Declining is defensible as long as the work
group communicates the path forward: that a concrete, scoped pull
request against the IG (StructureMap + ConceptMap + mapping rows for a
named template set) would be welcomed under a fresh ticket. This
avoids carrying an open-ended commitment that the IG cannot deliver in
the current cycle.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most realistic path that still honours Lynn
Laakso's reason for re-opening the ticket post-STU1.2.0. Disposition A
is technically correct and is what the reporter asked for, but it
commits the work group to authoring and maintaining a complete
normative StructureMap set across the full C-CDA template space — the
same scope problem that drove the 2016-11-10 "Considered for Future
Use" vote, and the reason the ticket sat open for eight years. Nothing
in the re-open comment indicates new resourcing. Disposition C is the
status quo and gives implementers nothing new despite STU1.2.0 having
shipped.

Disposition B threads the needle: it lands the high-leverage,
low-risk pieces immediately — `StructureDefinition.mapping` rows on
every profile (so every element has a documented C-CDA origin) plus a
starter StructureMap / ConceptMap companion for the highest-value
templates — while explicitly inviting the community contributions the
original disposition asked for. It also creates a clean upgrade path
to Disposition A in a future version once the companion has been
implementation-tested. The work group should set this direction and
ask the reporter / interested implementers to bring forward concrete
PRs against the named template list.
