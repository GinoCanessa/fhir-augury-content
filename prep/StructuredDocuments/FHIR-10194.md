## Summary

The reporter notes that in the C-CDA on FHIR Implementation Guide, the
**Family History** section of the CCD Composition profile does not declare
an `entry` reference to the **DAF FamilyMemberHistory** profile, which the
reporter expected based on the pattern used by other sections. The ticket
proposes adding a `Composition.section.entry` element to the Family
History section that references the DAF FamilyMemberHistory profile
(`http://hl7.org/fhir/StructureDefinition/daf-familymemberhistory`) with
cardinality `1..*` and `SHOULD` strength.

The ticket was originally filed in 2016 against the early C-CDA on FHIR
work and was triaged but not resolved. A March 2024 comment from
`lynn_laakso` notes "re-open post STU1.2 publication" and re-records the
original related URL
(`http://build.fhir.org/ccda/ccda-ccd-composition.html`), indicating SDWG
intends to re-evaluate this defect against the current C-CDA on FHIR
content rather than the 2016 ballot draft.

There are no linked Jira tickets and no GitHub cross-references. The only
cross-reference in the ingested data is an automated FHIR Bot
"tracker-item" announcement on Zulip from 2016, which contains no
substantive discussion.

## Details

**Description:**

The Family History section of the C-CDA on FHIR Implementation guide does
not provide an entry reference for the DAF-FamilyMemberHistory as I expect
it would.

I anticipate the following to appear as the Family History
`Composition.section.entry` element:

```xml
<element>
    <path value="Composition.section.entry" />
    <short value="entry" />
    <definition value="SHOULD contain zero or more [0..*] U.S. DAF FamilyMemberHistory Profile (identifier: http://hl7.org/fhir/StructureDefinition/daf-familymemberhistory)." />
    <min value="1" />
    <max value="*" />
    <type>
        <code value="Reference" />
        <profile value="http://hl7.org/fhir/StructureDefinition/daf-familymemberhistory" />
    </type>
</element>
```

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the C-CDA on FHIR `CCD-on-FHIR` Composition profile (and/or the
dedicated Family History section profile, depending on how the IG is
structured today), add (or unsuppress) a slice on
`Composition.section.entry` for the Family History section whose `type`
constrains `Reference.targetProfile` to the DAF FamilyMemberHistory
profile (`http://hl7.org/fhir/StructureDefinition/daf-familymemberhistory`),
with cardinality `0..*` and a `SHOULD`-level binding in the section's
narrative requirements (matching the IG's existing pattern for
section-to-profile entry references). Update the Family History section
narrative page to reference DAF FamilyMemberHistory in the same way that
sibling sections reference their DAF profiles.

#### Justification

This restores symmetry with the other sections of the CCD-on-FHIR
Composition, which generally do declare profile-typed `entry` slices for
their corresponding DAF resource profiles. Without the slice, the section
is effectively under-constrained and gives no discoverable hint to
implementers about which FamilyMemberHistory profile to use, which is the
defect the reporter raised. The original ticket text already provides a
ready-to-paste differential and the patch is local to one section.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than referencing the legacy DAF FamilyMemberHistory profile
(`http://hl7.org/fhir/StructureDefinition/daf-familymemberhistory`),
update the Family History section of CCD-on-FHIR to reference the current
US-realm successor profile — i.e., the **US Core FamilyMemberHistory**
profile (or, if SDWG prefers an unprofiled target for portability, the
base FHIR `FamilyMemberHistory` resource) — as the
`Composition.section.entry` target. The slice cardinality and
`SHOULD`-level requirement remain as the reporter proposed, but the
`targetProfile` is updated to the actively-maintained US-realm profile,
and the section narrative is updated accordingly.

#### Justification

The DAF IG (and its FamilyMemberHistory profile) has effectively been
superseded in US-realm work by US Core; pinning a 2024-era C-CDA on FHIR
section to a deprecated DAF profile risks shipping a normative pointer to
a profile that consumers no longer implement. This option still satisfies
the reporter's underlying need — making the section's expected entry
profile explicit — while keeping CCD-on-FHIR aligned with the profile
ecosystem implementers actually use. It is also a smaller risk surface
for the next ballot cycle than re-endorsing DAF.

---

### Disposition C: Decline

#### Proposal

Mark the request as **Persuasive with Mod / Not Persuasive** and decline
to add a DAF-FamilyMemberHistory–typed entry slice to the Family History
section. Document in the resolution that (a) DAF FamilyMemberHistory is
no longer the active US-realm profile, (b) the current C-CDA on FHIR STU
already addresses the Family History section through its general
`Composition.section.entry` rules, and (c) implementers should bind to
the US Core FamilyMemberHistory profile via implementation guidance
rather than via a normative slice on the section.

#### Justification

The ticket is from 2016 and was filed against an early build of the IG.
The literal request — add a slice referencing
`http://hl7.org/fhir/StructureDefinition/daf-familymemberhistory` — has
been overtaken by events: that profile is no longer the recommended
US-realm FamilyMemberHistory profile, so honoring the request as written
would bake a stale reference into the published IG. If the only goal is
to acknowledge that Family History entries should use a known profile,
that can be handled in narrative guidance without changing the
StructureDefinition.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying defect is real and worth fixing — the Family
History section of CCD-on-FHIR should make its expected entry profile
explicit, the same way sibling sections do. However, accepting the
request literally (Disposition A) would lock the IG to the deprecated
DAF FamilyMemberHistory profile, which contradicts where US-realm
profiling has moved since 2016 and would create a fresh maintenance
problem the next time the section is touched. Disposition C leaves the
documented defect unaddressed and forces implementers to infer the
intended profile from narrative, which is exactly what the reporter
flagged as missing. Disposition B fixes the defect, follows the IG's
existing slicing pattern, and re-targets the slice at the
actively-maintained US Core FamilyMemberHistory profile (or the
unprofiled base resource if SDWG prefers neutrality), which is the
combination most likely to survive the next ballot cycle without
re-introducing the same ticket.
