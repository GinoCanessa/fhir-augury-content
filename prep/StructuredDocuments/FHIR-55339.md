## Summary

This ticket is one of an automated batch (originating from "Comment 303"
imported by Lloyd McKenzie) flagging value sets in the Composition
resource that are candidates for migration to the Terminology.HL7 (THO)
common terminology repository. Specifically it calls out the
`composition-attestation-mode` CodeSystem (`CompositionAttestationMode`)
and its bound ValueSet `composition-attestation-mode`, used by
`Composition.attester.mode`. Migration to THO is not strictly required,
but is recommended when (a) the value set is plausibly reusable in CDA,
V2, or other product families, or (b) the definition is likely to need
updates outside the FHIR release cadence. The ticket asks Structured
Documents to consciously decide whether to migrate. The ticket has
already been triaged as **Persuasive** with a stated resolution to move
the `composition-attestation-mode` ValueSet into THO. No related Jira,
Zulip, or GitHub items were surfaced by cross-reference; the only
incoming cross-reference is the artifact mention of
`Composition.attester.mode`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the `composition-attestation-mode` artifacts from the FHIR Core
specification to the Terminology.HL7 (THO) repository:

- Move CodeSystem `composition-attestation-mode`
  (`CompositionAttestationMode`, canonical
  `http://hl7.org/fhir/composition-attestation-mode`) into THO under the
  appropriate THO namespace, preserving canonical URL where THO policy
  permits (or otherwise registering a redirect / OID equivalence so
  existing bindings continue to resolve).
- Move ValueSet `composition-attestation-mode` (canonical
  `http://hl7.org/fhir/ValueSet/composition-attestation-mode`) into THO
  alongside the CodeSystem.
- Update `Composition.attester.mode`'s binding in FHIR Core to reference
  the THO-hosted ValueSet (canonical unchanged from the consumer's point
  of view if THO retains the `hl7.org/fhir` namespace, otherwise update
  the binding URL and add release notes).
- Confirm there is no existing TSMG exemption for this CodeSystem
  before proceeding; if one exists, document it on the artifact and
  close this ticket without migration.
- Coordinate with the Vocabulary work group on naming, namespace, and
  publication cadence in THO.

This matches the recorded resolution ("We will move
ValueSet/composition-attestation-mode into THO") and the spirit of the
original automated review.

#### Justification

The originating analysis (mckenzie) and the SD lead (gdolin) both
concluded that this CodeSystem/ValueSet is a reasonable THO candidate:
the concept of "attestation mode" is plausibly reusable beyond FHIR
(CDA documents already model attestation, and V2 has analogous
concepts), and hosting in THO removes a future need for off-cycle FHIR
releases just to update the value set. The ballot direction (BlockVote1)
already endorses moving terminology to THO where appropriate, and the
ticket's resolution has been set to **Persuasive** with this exact
proposal — Disposition A simply executes that decision.

---

### Disposition B: Alternative Approach

#### Proposal

Move only the **ValueSet** to THO and leave the **CodeSystem** in FHIR
Core (or vice versa, depending on TSMG guidance). Concretely:

- Migrate `ValueSet/composition-attestation-mode` to THO so the
  enumeration can be revised independently of the FHIR release cycle and
  reused by CDA / V2 IGs.
- Retain the underlying `CodeSystem/composition-attestation-mode` in
  FHIR Core (with an explicit note that the codes are FHIR-defined) if
  there is no clear cross-family use of the codes themselves, or if a
  TSMG exemption already applies.
- Re-bind `Composition.attester.mode` to the THO ValueSet canonical and
  document the split in the value-set narrative and FHIR Core release
  notes.

This is the minimal disturbance option that still gains the off-cycle
update benefit, while avoiding a CodeSystem move that may be redundant
if the codes are unlikely to be referenced outside this single
ValueSet.

#### Justification

mckenzie's comment explicitly separates the two questions for value
sets ("could they be used in another product family" and "could they
need updates sooner than the next FHIR release"). The ValueSet
migration alone is enough to address question 2 — implementers consume
the bound ValueSet, so updates flow from there. Splitting the move is
also defensible if TSMG has previously granted (or is likely to grant)
an exemption keeping the CodeSystem in FHIR Core. The trade-off vs.
Disposition A is operational complexity (a CodeSystem in one repo and a
ValueSet that includes it in another), which Vocabulary/THO has
tooling for but which still adds a small maintenance cost.

---

### Disposition C: Decline

#### Proposal

Decline migration. Keep both `CodeSystem/composition-attestation-mode`
and `ValueSet/composition-attestation-mode` in FHIR Core, and resolve
this ticket as **Not Persuasive** on the grounds that:

- The codes (`personal`, `professional`, `legal`, `official`) are stable
  and have not changed materially across FHIR versions.
- The value set is bound only to a single FHIR element
  (`Composition.attester.mode`); there is no demonstrated cross-product
  reuse in CDA or V2 today.
- Migration carries non-zero coordination cost with Vocabulary/THO,
  potential canonical-URL churn, and downstream IG impact for a value
  set that does not appear to need off-cycle updates.

#### Justification

Migration to THO is explicitly described in the ticket as "not
required." The originating comment is from an automated flagging
process, not a concrete implementer pain point, and the ticket records
no implementer demand for off-cycle revisions. mckenzie's own framing
("In the end, it's up to the work group to make the decision") leaves
declining as a defensible answer when the work group judges the
migration cost outweighs the speculative benefit.

---

### Recommendation

**Recommended disposition:** A

The work group has already triaged this ticket as **Persuasive** with a
resolution that matches Disposition A verbatim, and both the SD lead
(gdolin) and the original commenter (mckenzie) have publicly aligned on
moving `composition-attestation-mode` to THO. Disposition A converts
that recorded decision into action: migrate both the CodeSystem and
ValueSet, rebind `Composition.attester.mode`, and verify no TSMG
exemption is in force. Disposition B is a reasonable fallback only if
Vocabulary/TSMG advises that the CodeSystem should remain in FHIR Core;
Disposition C contradicts the existing resolution and should only be
revisited if new evidence emerges that migration is materially harmful
to implementers. The next concrete step is for SD to coordinate with
Marc Duteau (per gdolin's 2026-02-19 ask) and Vocabulary to schedule
the move and update the binding in the next FHIR Core publication.
