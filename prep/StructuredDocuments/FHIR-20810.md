## Summary

FHIR-20810 reports four defects in the rendered "Questionnaire Items"
tables that the US Healthcare Associated Infections (HAI) IG publishes
alongside its Questionnaire instances (e.g.
`Questionnaire-hai-questionnaire-opc-ssi-event`):

1. The `Type` for the **Procedure Details** item is shown as `g` but
   should be `group`.
2. The **CPT Procedure Code** row is duplicated in the Questionnaire
   Items table.
3. The `Type` for **Procedure Duration** is shown as `time`, while the
   underlying XML uses `quantity`. The reporter notes that "time" is
   misleading if the value represents a duration (minutes/hours) rather
   than a clock time.
4. The **NHSN comment** that is present in the XML form of the item is
   missing from the rendered Questionnaire Items table.

The ticket sits next to two sibling change requests filed by the same
reporter on the same day against the same HAI publication, so it should
be triaged together with them:

- [FHIR-20809](https://jira.hl7.org/browse/FHIR-20809) â€” broader set of
  HTML-vs-XML inconsistencies for Questionnaire instances in this IG
  (missing Event Type item in XML, divergent `required` for
  `risk-factor-wound-class`, wrong text for `risk-factor-weight`, CPT
  Procedure Code declared `choice` with no value-set pointer). FHIR-20810
  is effectively a narrower instance of the same class of defect.
- [FHIR-20808](https://jira.hl7.org/browse/FHIR-20808) â€” `group`-typed
  questionnaire items in the XML/JSON instances are missing the
  `required` element, asking whether it should be present and how
  required vs. optional groups are otherwise distinguished. This shares
  the same Questionnaire-instance authoring/rendering pipeline.

All three tickets target the published HAI 2019May build and remain in
**Triaged / Unresolved** state with no assignee, reporter activity, or
comments since 2019.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-20810 as a four-part errata against the
`hai-questionnaire-opc-ssi-event` Questionnaire (and any sibling HAI
Questionnaire instances exhibiting the same patterns) and apply the
following targeted fixes in the next HAI publication:

1. **Procedure Details item type** â€” change the rendered `Type` cell
   from `g` to `group`. Confirm the underlying Questionnaire resource
   has `Questionnaire.item.type = group`; if so, the defect is in the
   tooling/template that emits the truncated `g`. Either way, the
   published table must read `group`.
2. **Duplicate CPT Procedure Code row** â€” remove the duplicate row
   from the Questionnaire Items table. Verify the source Questionnaire
   does not contain two `item` entries with the same `linkId`/text and
   correct it if it does.
3. **Procedure Duration item type** â€” keep the underlying
   Questionnaire as `quantity` (it represents a duration, not a clock
   time) and fix the table so it shows `quantity` rather than `time`.
   Add a short item-level note clarifying the expected unit (e.g.
   minutes) so the value is unambiguous.
4. **NHSN comment** â€” surface the NHSN comment that is present in the
   XML form (carried as an extension or `Questionnaire.item.text` /
   helper text) in the rendered Questionnaire Items table so the HTML
   and XML are content-equivalent.

#### Justification

All four findings are simple errata against a single HAI Questionnaire
instance and its rendered table. They do not change the meaning of the
data being collected, do not affect conformance, and align the
human-readable view with the canonical XML â€” a stated requirement of
HL7 publication QA. The reporter has documented each defect precisely;
each fix is mechanical. Accepting matches Structured Documents'
typical handling of "rendering does not match underlying resource"
errata in published IGs and clears the way to also apply the broader
twin-ticket fix in FHIR-20809.

### Disposition B: Alternative Approach

#### Proposal

Resolve FHIR-20810 as a duplicate-of / rolled-into a single composite
disposition that fixes FHIR-20808, FHIR-20809, and FHIR-20810 together
under one HAI Questionnaire-instance cleanup pass. Concretely:

- Audit every Questionnaire instance in the HAI IG (not just
  `hai-questionnaire-opc-ssi-event`) for: truncated `type` values
  (`g`, `t`, etc.) in the Items table; duplicated rows; mismatches
  between rendered `Type` and the underlying `Questionnaire.item.type`;
  missing `Questionnaire.item.required` on `group`-typed items; and
  NHSN comments / helper text present in XML but absent from the
  table.
- Fix the underlying `Questionnaire` resources where the source data
  is wrong (e.g., truncated type, duplicate item, missing `required`),
  and fix the rendering template where the source data is correct but
  the table is wrong (e.g., `quantity` mis-displayed as `time`, NHSN
  comment dropped).
- Close FHIR-20808, FHIR-20809, and FHIR-20810 with a shared
  cross-reference to the resulting change.

#### Justification

The three tickets were filed minutes apart by the same reporter
against the same artifact and describe the same underlying problem
class â€” drift between the source `Questionnaire` resources and their
rendered Items table â€” manifesting in slightly different fields. Fixing
them in isolation risks repeated review cycles and inconsistent
treatment (e.g., fixing `g`â†’`group` in one questionnaire while leaving
`t`â†’`time` lurking in another). One audit pass is cheaper than three
separate spot fixes and produces a single, coherent change that the
work group can review and the publisher can run regression on. The
trade-off is slightly more scope than what FHIR-20810 literally asks
for, but the additional work is bounded to one IG and is a strict
superset of Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-20810 on the grounds that the US HAI (FHIR) IG is a
historical 2019May publication that has not been actively maintained
(both this ticket and its siblings have been Triaged/Unresolved with no
activity since 2019-11-21), and that no current ballot or publication
cycle is queued against this IG. Suggest the reporter reopen if and
when the HAI IG is republished, at which point the rendering and
content fixes can be applied as part of the new build's normal QA.

#### Justification

The HAI IG in question is dated `2019May` and the ticket has had no
movement in over five years. The Structured Documents work group has
finite review capacity, and "rendered table mismatches the XML in a
five-year-old IG that no one is republishing" has very low downstream
impact: implementers building against the HAI IG today are reading the
canonical XML, not the rendered table, and any consumer relying on the
HTML rendering can fall back to the XML. Declining frees the work group
to focus on artifacts in active publication.

---

### Recommendation

**Recommended disposition:** B

These three tickets (FHIR-20808, FHIR-20809, FHIR-20810) are siblings
reporting variants of the same authoring/rendering drift in the HAI
Questionnaire instances. Disposing of FHIR-20810 in isolation
(Disposition A) is technically cheap but leaves the work group having
the same review conversation three times for one underlying problem.
Declining (Disposition C) is defensible given the IG's dormancy, but
the fixes are small, well-scoped, and improve the quality of a
published HL7 artifact that is still indexable on hl7.org â€” closing as
"not worth fixing" sets a poor precedent for errata against published
material. Disposition B applies a single audit-and-correct pass across
the HAI Questionnaire instances, fixes all three tickets together, and
produces one coherent change for the work group to approve. If the
work group prefers a narrower scope, fall back to Disposition A and
process the three tickets as a coordinated batch rather than
independently.
