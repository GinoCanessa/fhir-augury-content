## Summary

Corey Spears reported that, in the C-CDA on FHIR Composition profiles
(specifically the Transfer Summary Composition referenced in the
ticket's related URL), the slicing `discriminator.rules` value is set
to `openAtEnd` where it should be `open`. The ticket title contains a
typo (`opeAtEnd`) but the description and the original related URL
make the intent clear: the slicing rule on `Composition.section` (and
likely the parallel slicings on the other C-CDA on FHIR Composition
profiles) is too restrictive — `openAtEnd` only allows additional
slices to appear *after* the defined ones, whereas `open` allows
additional slices to appear *anywhere* in the list, which the reporter
argues is the appropriate behaviour for these Composition section
slicings. A `re-open post STU1.2 publication` comment from Lynn Laakso
in March 2024 indicates the WG deferred the change but did not reject
it; it is awaiting reconsideration in a post-STU 1.2 cycle.

There are no linked or cross-referenced Jira tickets, no substantive
Zulip discussion threads (only a FHIR-Bot tracker-item announcement),
and no GitHub PRs/issues recorded for this item.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the C-CDA on FHIR IG (work group: Structured Documents,
specification "US C-CDA on FHIR (FHIR)"), change
`Composition.section.slicing.rules` from `openAtEnd` to `open` on the
Transfer Summary Composition profile (the URL named in the ticket) and
on every other C-CDA on FHIR Composition profile that currently uses
`openAtEnd` for `Composition.section`. The change is a single
`<rules value="open"/>` edit in each affected `StructureDefinition`
differential (typically authored as the `slicing` block on
`Composition.section`). All slice definitions and discriminator paths
remain unchanged; only the `rules` value moves from `openAtEnd` to
`open`. Republish the affected profiles, regenerate the rendered
profile pages, and add a brief change-log entry noting that
implementers may now interleave non-defined sections among the
profile-defined sections.

#### Justification

`openAtEnd` is the most restrictive of the "permits extra slices"
options: it only allows additional, undefined slices *after* all of
the named slices, in the order they were declared. For a clinical
document Composition, sections are not strictly ordered in clinical
practice, and implementers regularly include sections that are not
named by a given C-CDA on FHIR template (e.g., site-specific or
optional C-CDA sections that are not profiled). `open` reflects the
real-world behaviour of C-CDA documents, which permit additional
sections in any position. This is consistent with how comparable
Composition profiles in other IGs (e.g., IPS) use `open` slicing on
`Composition.section`. Lynn Laakso's 2024 comment to "re-open post
STU1.2 publication" indicates the WG already views this as a valid
defect to address in the next cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the choice between `open` and `openAtEnd` as a design decision
that should be made consistently across the entire C-CDA on FHIR IG
rather than ticket-by-ticket, and resolve this ticket by adopting an
IG-wide rule. Specifically: audit every `Composition.section.slicing`
in the C-CDA on FHIR IG, document the chosen rule in the IG's
"Conventions" or "Profiling approach" page, and then apply that rule
uniformly. The recommended uniform value would be `open` (matching
Disposition A) for all C-CDA on FHIR Composition profiles, with a
narrative paragraph explaining that section ordering in C-CDA-derived
documents is not normative and that implementers MAY include
additional sections in any position. As part of the same change,
verify that no profile is inadvertently relying on
`openAtEnd`-implied ordering in its `mustSupport` or invariant
definitions.

#### Justification

Changing one profile in isolation risks leaving the IG internally
inconsistent — some Composition profiles permitting interleaved extra
sections (`open`) and others only trailing extras (`openAtEnd`) — which
is hard for implementers to reason about and likely to generate
follow-up tickets identical to this one. A single documented
convention is cheaper to maintain than per-profile decisions. The
trade-off versus Disposition A is scope: this is a larger edit, so it
takes more WG time, but it produces a more coherent IG and forecloses
a class of repeat tickets.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" on the basis that `openAtEnd`
already permits the additional sections implementers care about, and
that any implementer needing to add sections elsewhere in the list
can do so by deriving a further profile or by relaxing the rule
locally. State explicitly that the C-CDA on FHIR IG intends the
profile-defined sections to appear first, in declared order, with any
implementer-specific sections appended.

#### Justification

`openAtEnd` is a deliberate constraint that communicates a section
ordering expectation to implementers: profile-named sections come
first, additional sections come after. If the WG considers that
ordering desirable for downstream tooling (e.g., human-readable
narrative rendering, document navigation), then preserving
`openAtEnd` is defensible. The ticket as filed is a one-line
assertion with no concrete implementer pain reported and no community
discussion to weigh, which weakens the case for a normative change.

---

### Recommendation

**Recommended disposition:** A

The ticket is narrow, the requested change is a one-token edit per
affected profile, and Lynn Laakso's 2024 comment ("re-open post STU1.2
publication") shows the WG already expects to revisit it. The
substantive argument in Disposition A — that real C-CDA documents do
not enforce section ordering and `openAtEnd` therefore over-constrains
the profile — is straightforward and aligns with the looser slicing
posture other Composition-based IGs have adopted. Disposition B is
attractive but should not block this fix: the WG can adopt the
IG-wide convention as a follow-up once Disposition A has settled the
immediate defect. Disposition C is hard to defend: there is no
recorded use case for enforcing section ordering via `openAtEnd`, and
declining would simply recycle the ticket. Accept as requested,
applying the change to the Transfer Summary Composition (the profile
named in the ticket) and to any sibling C-CDA on FHIR Composition
profiles that share the same `openAtEnd` slicing on
`Composition.section`.
