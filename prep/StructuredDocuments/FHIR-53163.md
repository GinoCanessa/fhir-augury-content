## Summary

The ticket asks Structured Documents to expand the target list of the new
`Composition.basedOn` element so that `DocumentReference` is an allowed
target alongside the request resources already permitted there. The
motivation is that, in real-world workflows, a paper order may have been
scanned and represented as a `DocumentReference`; clinicians then need a
way to assert that the events captured in a `Composition` were performed
in fulfillment of that scanned/paper order. Without `DocumentReference`
as an allowed target on `Composition.basedOn`, there is no clean way to
point a clinical document back at the paper-order artifact it is
documenting fulfillment of.

The request is closely tied to [FHIR-53164](https://jira.hl7.org/browse/FHIR-53164)
("Add DocumentReference to the Requests pattern list"), which proposed
to add `DocumentReference` to the FHIR Requests pattern. As commenter
Benjamin pointed out on FHIR-53163, if FHIR-53164 had been accepted and
`Composition.basedOn` were typed as `Reference(Request)`, the change
requested here would have been picked up automatically. FHIR-53164 was
itself resolved *Not Persuasive* by FHIR Infrastructure (DocumentReference
maps best to the Event pattern, not Request — its contents may follow
*any* pattern), which removes the automatic-inheritance path and forces
this ticket to be decided on its own merits. The ticket description also
references FHIR-22636 as the originating change, but the cited issue's
title is unrelated to Composition (it concerns `$translate` output) and
the augury orchestrator returned 404 for it, so it appears to be a stale
or mistyped link in the description.

The current SD-side recommendation captured on the ticket is to instead
use `Composition.relatesTo`. Commenter ehaas pushes back that
`Composition.relatesTo` lacks an equivalent of *"Fulfills plan,
proposal, or order"*, so accepting that recommendation would still
require either extending the `relatesTo` type value set with a
`based-on` concept or living with a semantic mismatch — which, he
argues, would be an obvious duplication of `Composition.basedOn`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FHIR Core specification, extend the target type list of
`Composition.basedOn` to include `Reference(DocumentReference)`
alongside the existing request-resource targets (CarePlan,
ServiceRequest, MedicationRequest, etc., as defined when
`Composition.basedOn` was introduced). Update the Composition resource
definition, structure definition, and any element-table narrative to
reflect the new allowed target. Add a usage note explaining that the
`DocumentReference` use case is specifically for scanned/paper orders
or other request-shaped artifacts whose only FHIR representation is a
`DocumentReference`.

#### Justification

This is the most direct path to the reporter's clinical scenario:
asserting that a `Composition` documents events performed in fulfillment
of a paper order represented as a scanned `DocumentReference`.
`Composition.basedOn` already carries the precise semantics being
sought ("fulfills") and is shaped exactly for this kind of linkage, so
reusing it avoids the value-set extension and "duplication of purpose"
concern raised by ehaas. It is also additive: existing instances remain
valid, and implementers who do not have the paper-order use case are
unaffected. The principal cost is that `DocumentReference` is not
itself a Request-pattern resource (per FI's resolution of
[FHIR-53164](https://jira.hl7.org/browse/FHIR-53164)), so the
disposition would have to be defended as a deliberate, narrowly scoped
exception on `Composition.basedOn` rather than as adherence to the
Request pattern.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Composition.basedOn` typed strictly to Request-pattern resources
and instead address the reporter's need on `Composition.relatesTo` by
extending the `v3-DocumentRelationshipType` (or successor) value set
used for `Composition.relatesTo.type` with a new concept equivalent to
*"Fulfills plan, proposal, or order"* (e.g., `based-on` /
`fulfills`). Document a normative pattern that, when the antecedent
artifact is a `DocumentReference` (e.g., a scanned paper order), the
`relatesTo.type` SHALL use this new concept and `relatesTo.target`
SHALL reference the `DocumentReference`. The element comments on
`Composition.basedOn` should add an explicit cross-reference pointing
implementers to this `relatesTo` pattern for `DocumentReference`
targets.

#### Justification

This preserves the Request-pattern purity of `Composition.basedOn`
(consistent with FI's reasoning on FHIR-53164 that `DocumentReference`
is best modeled as Event, not Request), and aligns with the SD
committee's recorded recommendation on the ticket to "use
`Composition.relatesTo`." It directly answers ehaas's objection by
adding the missing concept rather than leaving implementers to choose
an inappropriate code, which is the actual gap in the recommendation as
written. Trade-off vs. Disposition A: it requires a value-set change
and an additional implementer convention, and ehaas's "duplication of
purpose" critique still has merit — implementers will have two
syntactically valid ways to assert "fulfills" depending on the target
type, which can be confusing. Mitigated by clear authoring guidance on
both elements.

---

### Disposition C: Decline

#### Proposal

Resolve as *Not Persuasive* (the resolution already on the ticket),
with the rationale that `Composition.basedOn` is intentionally typed
against Request-pattern resources, that `DocumentReference` is an
Event-pattern resource (per FI's resolution of FHIR-53164), and that
the workflow described — linking a clinical document to a scanned paper
order — is adequately served today by `Composition.relatesTo` together
with whatever existing `relatesTo.type` concept best fits the
implementer's intent (e.g., `transforms`, `replaces`, or a derived
concept), or by referencing the underlying real-world order through
`Provenance` / `DocumentReference.relatesTo` rather than through
`Composition.basedOn`.

#### Justification

This is consistent with the pattern decisions already taken by FI on
FHIR-53164 and avoids creating a precedent for putting non-Request
resources into `basedOn` slots across other Event-pattern resources
(which is the secondary risk Benjamin called out). It also respects
the FHIR design principle that pattern conformance should drive target
typing rather than ad hoc per-resource exceptions. The weakness — and
the reason it requires careful framing — is that ehaas has correctly
identified that the current `Composition.relatesTo` value set does not
contain a "fulfills" concept, so a literal "use `relatesTo`" answer is
incomplete; declining without at minimum acknowledging that gap will
likely produce a reconsideration request.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need — being able to record that a
`Composition` documents events in fulfillment of a scanned paper order
— is real and recurring (Portable Medical Orders are the obvious, and
explicitly-cited, example). Disposition C alone leaves that need
unaddressed, because the current `Composition.relatesTo` value set
genuinely lacks a "fulfills" concept, which is exactly the hole ehaas
identified in the existing resolution text. Disposition A solves the
problem cleanly but requires SD to defend an explicit Request-pattern
exception on `Composition.basedOn`, immediately after FI rejected
FHIR-53164 on pattern grounds; that is fightable but invites further
ticket churn. Disposition B threads both: it keeps
`Composition.basedOn` aligned with the Request pattern (honoring the FI
position), it converts the SD committee's already-recorded
"`Composition.relatesTo`" recommendation into something actually
usable by implementers by closing the value-set gap, and it gives
ehaas a substantive answer to the "duplication of purpose" concern via
explicit authoring guidance. SD should bring Disposition B as the
recommendation into the Zulip discussion gdolin requested, and only
fall back to A if FI signals openness to a narrowly-scoped
`Composition.basedOn` exception.
