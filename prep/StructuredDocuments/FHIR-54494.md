## Summary

The reporter notes that the Composition page in FHIR Core contains a quoted
reference to a "Clinical Document profile" that "constrains Composition to
specify a clinical document (matching CDA)". The hyperlink behind that phrase,
however, does not point to an actual `StructureDefinition` of kind `resource`
constraining `Composition`; instead it lands on a set of extensions
(historically the FHIR Clinical Document Extensions IG, now superseded by the
HL7 Clinical Document IG which does publish a real `clinical-document-composition`
profile). The reporter proposes either removing the broken textual reference
or repointing it to a genuine profile. There are no cross-referenced Jira
tickets, no related GitHub items, and no Zulip discussions explicitly tied to
this ticket key, although adjacent Zulip activity in `fhir/documents`
discussing the recently published FHIR Clinical Document IG provides useful
context.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested — Repoint the link to the new Clinical Document IG profile

#### Proposal

In the FHIR Core `Composition` resource page (the introductory/scope
narrative that mentions the "Clinical Document profile constrains Composition
to specify a clinical document (matching CDA). See also the comparison with
CDA."), repoint the hyperlink behind "Clinical Document profile" so that it
resolves to the actual profile published by the HL7 Clinical Document IG —
`http://hl7.org/fhir/uv/fhir-clinical-document/StructureDefinition/clinical-document-composition`
(human-readable rendering at
`https://hl7.org/fhir/uv/fhir-clinical-document/STU1/StructureDefinition-clinical-document-composition.html`).
Verify the surrounding sentence still reads correctly ("constrains Composition
to specify a clinical document (matching CDA)") given that the new IG profile
is what now plays that role, and update the "comparison with CDA" link target
if it has likewise drifted.

#### Justification

The reporter's complaint is precisely that the anchor text says "profile" but
the link resolves to a set of extensions. With the HL7 Clinical Document IG
now publishing a genuine `Composition`-constraining profile, fixing the
hyperlink fully satisfies the request without touching any normative content
or breaking existing implementer expectations. This also keeps the editorial
intent of the Composition page intact (acknowledging that a clinical-document
profile exists in the FHIR uv space) and aligns Core narrative with the
artifact the Structured Documents WG itself maintains, which the
`fhir/documents` Zulip thread shows is being actively curated.

---

### Disposition B: Alternative Approach — Remove the textual reference from Core

#### Proposal

Delete the sentence "The Clinical Document profile constrains Composition to
specify a clinical document (matching CDA). See also the comparison with CDA."
from the `Composition` resource page (and any equivalent sidebar/section
referencing "the Clinical Document profile"). Retain only the generic
"comparison with CDA" pointer, if it is still accurate and self-contained.
This removes the dependency on any particular external IG link.

#### Justification

This is the second half of the reporter's own suggestion ("remove the
reference or add the profile"). It avoids embedding a Core-spec link to an
external IG that lives on its own release cadence (and, per the related
Zulip discussion, is still receiving STU-level corrections), eliminating a
recurring source of broken-link / version-skew tickets like this one. It is
also the lowest-risk option editorially: no claim is made about an external
artifact whose semantics or location may change.

---

### Disposition C: Decline — Mark as "Persuasive with Mod / No Change Needed"

#### Proposal

Decline the change on the basis that the link is editorial/informative only
and the underlying intent of the Composition page is preserved; any reader
following the link still lands on a related Structured Documents artifact
about clinical documents, even if it is technically a set of extensions
rather than a single profile.

#### Justification

This is a defensible position only if the WG concludes that the existing link
is "good enough" as an informative pointer and that maintaining it is not
worth committee bandwidth. However, the reporter's observation is factually
correct (anchor text says "profile" → target is not a profile), so this
disposition is essentially asking implementers to tolerate a known
documentation defect; it is the weakest of the three options.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a concrete, verifiable defect: the word "profile"
links to something that is not a profile. Now that the HL7 Clinical Document
IG (under the same Structured Documents work group) publishes a real
`clinical-document-composition` profile, Disposition A delivers exactly what
the anchor text already promises with minimal editorial change and no
normative impact. Disposition B is acceptable as a fallback if SDWG decides
they do not want Core to point at a uv-space IG at all, but it loses useful
context for implementers coming from a CDA background. Disposition C should be
avoided because it leaves a known-incorrect link in a high-traffic Core
resource page.
