## Summary

The reporter, Lisa R. Nelson, identifies a gap in the **C-CDA on FHIR US Realm Header Profile**: there is no documented way to represent that a Patient has a **legal guardian** who makes medical decisions on the patient's behalf. This capability exists in CDA (and explicitly in the C-CDA US Realm Header template), so it represents a regression for implementers migrating from C-CDA to C-CDA on FHIR. The lone ticket comment cross-references the FHIR Core ticket [FHIR-15841](https://jira.hl7.org/browse/FHIR-15841), which dealt with the same underlying root cause at the base-resource level: `Patient.contact` lists "guardian" as a use case, but the bound value set for `Patient.contact.relationship` did not include a guardian code. FHIR-15841 was resolved as **Persuasive with Modification** (status **Applied**, label **R6**) by Patient Administration in 2022 — splitting `Patient.contact` into a `relationship` element (bound to `v3-PersonalRelationshipRoleType`, which includes guardian concepts) and a new `role` element. This means the upstream FHIR Core gap that prevented a clean answer at the time FHIR-31068 was filed has been closed in R6, and the C-CDA on FHIR profile now needs to surface and document that pathway in the US Realm Header.

## Details

**Description:**

> We have use cases where the Patient is not able to make decisions about their own medical care. The Patient has a legal guardian and it needs to be present in the header that this relationship to the person and situation for the patient exists.
>
> I could not see how to represent this role in the US Realm Header Profile.
>
> This is currently possible in CDA and in the C-CDA US Realm Header Template.

### [FHIR-15841: Patient.contact lists guardian as a use case, but code system for Patient.contact.relationship does not include a code for guardian.](https://jira.hl7.org/browse/FHIR-15841)

This Patient Administration ticket against FHIR Core is the upstream root cause referenced by FHIR-31068's only comment. After several rounds of discussion (including a reopen in February 2022), it was resolved **Persuasive with Modification** and is now **Applied** (label `R6`). The resolution splits `Patient.contact` into two preferred-bound elements — `Patient.contact.relationship` (now bound to `v3-PersonalRelationshipRoleType`, which includes guardian concepts) and a new `Patient.contact.role` (bound to `relatedperson-relationshiptype`) — and applies a parallel split to `RelatedPerson`. This change makes guardian a first-class concept in core FHIR R6, which directly enables the C-CDA on FHIR US Realm Header Profile to address FHIR-31068 without inventing a US-specific extension.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **US Realm Header Profile** in **C-CDA on FHIR** to explicitly support representing a Patient's legal guardian, with the following concrete changes:

1. Add narrative guidance to the US Realm Header Profile page describing how to populate `Patient.contact` for a guardian, with a worked example showing `Patient.contact.relationship` set to a guardian concept from `http://terminology.hl7.org/ValueSet/v3-PersonalRelationshipRoleType` (e.g., `GUARD` from RoleClass OID `2.16.840.1.113883.11.11555`, aligning with the C-CDA template).
2. In the `us-realm-header-patient` profile (or the US Realm Header `Patient` slice), add a `mustSupport` flag on `Patient.contact`, `Patient.contact.relationship`, `Patient.contact.name`, and `Patient.contact.telecom`, and document the expectation that a guardian SHALL be conveyed through `Patient.contact` when one is known and relevant to the document.
3. Add an explicit C-CDA → FHIR mapping row showing `responsibleParty/assignedEntity` (and the C-CDA Guardian template `2.16.840.1.113883.10.20.22.4.32`) mapping to `Patient.contact` with the guardian relationship code.

#### Justification

This is exactly the gap the reporter identified: a known C-CDA capability has no documented path in the C-CDA on FHIR US Realm Header. With **FHIR-15841 now Applied in R6**, `Patient.contact.relationship` is bound to `v3-PersonalRelationshipRoleType`, which already contains the `GUARD` concept the C-CDA template uses. So the IG can adopt this disposition without waiting on any upstream change and without fabricating a US-specific code system. Adding `mustSupport` plus a worked example follows the pattern other US Realm Header elements use (Race, Ethnicity, Birth Sex), giving implementers a clear, testable target. This also closes a parity gap with C-CDA, which is the IG's primary value proposition.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) using `Patient.contact`, recommend representing the guardian as a **separate `RelatedPerson` resource** referenced from the document, with `RelatedPerson.relationship` carrying the guardian code, and require the C-CDA on FHIR Composition to include that `RelatedPerson` as an `informant` or via a documented extension on `Composition.attester` / `Composition.author` where the guardian is acting as decision-maker for a specific clinical event. Add narrative in the US Realm Header Profile that distinguishes:

- **Standing relationship** (guardian as a contact for the patient) → `Patient.contact` with relationship `GUARD`.
- **Acting decision-maker for the encounter/document** → `RelatedPerson` with relationship `GUARD`, referenced from the relevant Composition role (e.g., `Composition.author`, `Composition.attester`, or a section's `Provenance`/`informant`).

#### Justification

`Patient.contact` is intentionally lightweight — it does not have its own resource identity and cannot be referenced from elsewhere in the bundle, which limits its utility when a guardian is consenting to or attesting a specific document. C-CDA's guardian template lives on the patient, but C-CDA also distinguishes "responsibleParty" and "informant" roles that act on the document itself; a richer FHIR mapping recognizes that distinction. Using `RelatedPerson` aligns with how other IGs (e.g., US Core) treat persons with their own identity who participate in care, and it scales to scenarios where the guardian must be an addressable, query-able resource (e.g., to attach a `Consent`). The trade-off is more implementation surface area than Disposition A, and `RelatedPerson` is not currently profiled by C-CDA on FHIR's US Realm Header — adopting this would require introducing or referencing a US Realm `RelatedPerson` profile.

---

### Disposition C: Decline

#### Proposal

Decline as **Already Addressed by Existing Mechanism**, citing that R6 `Patient.contact.relationship` (post-FHIR-15841) is bound to a value set that includes guardian, and that the base FHIR `Patient` resource therefore already supports the requested capability. The Structured Documents work group could close FHIR-31068 as **Not Persuasive** with a pointer to FHIR-15841 and to the `Patient.contact` documentation, noting that no profile-level change is required.

#### Justification

The literal request — "I could not see how to represent this role in the US Realm Header Profile" — could be read as a documentation/discoverability complaint rather than a missing feature. With FHIR-15841 applied in R6, the mechanism exists; an implementer reading `Patient.contact.relationship` will now see guardian-capable codes in the bound value set. Declining keeps the IG lean and avoids piling on `mustSupport` flags. The downside is significant: the reporter's actual concern is C-CDA parity in **the US Realm Header IG**, not in base FHIR. Pointing to a base resource without IG-level guidance leaves implementers to guess, perpetuates the discoverability gap, and provides no conformance basis for validators.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest match to the reporter's request and is now low-risk because the upstream FHIR Core dependency (FHIR-15841) has been resolved and applied in R6 — the C-CDA on FHIR IG can simply adopt the existing `Patient.contact.relationship` binding to `v3-PersonalRelationshipRoleType`, mark the relevant elements `mustSupport`, and add a worked guardian example with the `GUARD` code that aligns to the C-CDA template. This delivers the C-CDA parity the IG promises, requires no new extensions or value sets, and gives implementers a concrete, validatable target. Disposition B's `RelatedPerson`-based pattern is worth flagging in the same narrative as a forward-looking option for the "decision-maker on the document" use case, but it should not block resolution of the original ticket; the work group can consider it as a follow-on if implementer feedback shows the standing-contact representation is insufficient. Disposition C should be rejected because the ticket is fundamentally about IG-level guidance and conformance, not about whether the base resource permits the representation.
