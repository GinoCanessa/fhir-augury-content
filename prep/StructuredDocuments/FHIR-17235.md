## Summary

This change request, raised by Giorgio Cangioli during the WGM Cologne (Wed
Q3) discussion, asks the IPS work to align the allowed status codes of the
CDA IPS MedicationStatement entry and the FHIR IPS `MedicationStatement`
profile so the two flavors of the International Patient Summary convey
medication status consistently. Concretely the ticket proposes (a) adding an
invariant on the FHIR IPS `MedicationStatement` profile that excludes the
codes `entered-in-error` and `intended` from `MedicationStatement.status`,
and (b) optionally publishing a `ConceptMap` between the CDA medication
status vocabulary and the FHIR `medication-statement-status` value set so
implementers can transform between the two summaries deterministically.

A `giorgio` comment from 2018-11-14 notes that an interim version added a
narrative note excluding `entered-in-error`, but the formal invariant and
the ConceptMap were never landed. The ticket was reopened by `lynn_laakso`
on 2022-11-22 "post STU update publication" so that the alignment work can
be picked up against the current STU of IPS. No related Jira tickets were
found via cross-reference; the only Zulip reference is the automated
`#committers > tracker-item` posting from FHIR Bot announcing the original
tracker submission, not a discussion thread.

## Proposed Dispositions

> Note: Because no GitHub cross-references are recorded for this ticket, the
> dispositions below name the IPS authoring artifacts by their well-known
> identifiers (`StructureDefinition/MedicationStatement-uv-ips`,
> the `medication-statement-status` value set, and a new IPS `ConceptMap`)
> rather than concrete repo paths. The Structured Documents / IPS work
> group should map these to the current IPS authoring repo when scheduling
> the change.

### Disposition A: Accept As Requested

#### Proposal

Make two coordinated changes to the FHIR IPS publication:

1. **Add an invariant on the IPS `MedicationStatement` profile**
   (`StructureDefinition/MedicationStatement-uv-ips`) that constrains
   `MedicationStatement.status` to exclude `entered-in-error` and
   `intended`. Concretely, add a `constraint` element on the
   `MedicationStatement` root with an expression equivalent to
   `status.exists() and status != 'entered-in-error' and status != 'intended'`,
   `severity = error`, a stable `key` (e.g. `ips-medstmt-status-1`), and
   human text such as *"For an IPS Medication Summary, MedicationStatement.status
   SHALL NOT be 'entered-in-error' or 'intended'."*. Retain the existing
   narrative note Giorgio added in 2018 as supporting prose, but make the
   invariant the normative constraint.
2. **Publish a `ConceptMap` resource** named e.g. `cda-fhir-medication-status`
   that maps the CDA medication status vocabulary (the values used in the
   CDA IPS MedicationStatement entry — typically `active`, `suspended`,
   `aborted`, `completed` from `ActStatus`/HL7 status codes) to the
   corresponding FHIR `medication-statement-status` codes (`active`,
   `on-hold`, `stopped`, `completed`, plus `not-taken`/`unknown` as
   appropriate), and reference it from the Medication Summary narrative.
   Mark `entered-in-error` and `intended` as having no source mapping in
   the CDA → FHIR direction so the constraint in (1) is reinforced by the
   mapping table.

#### Justification

This is exactly the action the reporter recorded against the WGM
discussion. Excluding `entered-in-error` and `intended` from a *patient
summary* is consistent with the IPS design intent: the IPS conveys the
clinically relevant medications a patient is or has been on, not draft
intentions or known data-quality errors, and the equivalent CDA entry has
no parallel for either status. Promoting the existing narrative note to a
machine-checkable invariant is a low-risk profile-only change that lets
validators surface non-conformant instances. Publishing the `ConceptMap`
gives bidirectional CDA ↔ FHIR transformers a single normative source of
truth and removes the recurring "what does CDA `aborted` map to?" question
that motivated the WGM agenda item.

---

### Disposition B: Alternative Approach

#### Proposal

Express the alignment without adding a profile invariant. Instead:

1. **Bind `MedicationStatement.status` to a tightened, IPS-specific
   value set** (e.g. `medication-statement-status-ips`) that enumerates
   only `active`, `completed`, `on-hold`, `stopped`, `not-taken`, and
   `unknown`, with `strength = required`. This achieves the same
   exclusion of `entered-in-error` and `intended` through a value-set
   binding rather than a FHIRPath constraint, which is the more idiomatic
   way to restrict a coded element in a FHIR profile.
2. **Publish the same CDA ↔ FHIR `ConceptMap`** as in Disposition A, but
   target the new IPS-specific value set so the mapping and the binding
   are mutually consistent.
3. Keep the narrative note Giorgio added in 2018; do not add an invariant.

#### Justification

Profile invariants on a coded element duplicate work that the FHIR
binding machinery does natively, and they are easier to miss in tooling
that focuses on bindings and slicing. A required binding to a
purpose-built IPS value set is the pattern the IPS IG already uses for
several other coded elements (e.g. allergy clinical status), so this
keeps the profile stylistically consistent and gives implementers a
single value set URL to consult. The trade-off vs. Disposition A is
churn: a new value set must be created, registered, and version-managed
in the IPS terminology, whereas an invariant is a single addition to the
existing profile. If the IPS terminology pipeline is already
publishing IPS-scoped value sets, this is the lower-friction long-term
choice; if not, the invariant in Disposition A is faster.

---

### Disposition C: Decline

#### Proposal

Decline the formal invariant and the new `ConceptMap`. Rely on the
existing narrative note (added by `giorgio` in 2018) plus implementer
guidance that `entered-in-error` and `intended` are out of scope for an
IPS, and treat CDA ↔ FHIR status alignment as an implementer concern to
be handled in mapping projects (e.g. C-CDA-on-FHIR companion guides)
rather than inside the IPS IG.

#### Justification

The ticket has been open since 2018 and the only published response to
date has been the narrative note, which suggests the work group has not
felt sufficient implementer pain to invest in either a normative
invariant or a new ConceptMap. `MedicationStatement.status` is already a
required-bound coded element; a clinical IPS receiver that finds
`entered-in-error` in a summary has every reason to treat the entry as
data-quality noise regardless of whether an IPS-specific invariant fires.
A CDA ↔ FHIR status `ConceptMap` arguably belongs alongside the broader
C-CDA-on-FHIR mappings, not inside the IPS IG, because IPS itself does
not perform the transformation. Declining keeps the IPS profile minimal
and avoids introducing a cross-paradigm normative artifact whose
maintenance owner is unclear.

---

### Recommendation

**Recommended disposition:** A

The reporter spelled out a concrete two-part action at WGM Cologne — add
the invariant, publish the `ConceptMap` — and the work has been sitting
unresolved for years with only a narrative note to show for it. The
invariant approach (Disposition A) is the most direct path to closing the
ticket as written, costs only a single profile edit plus a small
`ConceptMap`, and produces validator-checkable conformance for IPS
implementers immediately. Disposition B is technically cleaner but trades
fast resolution for a new IPS-scoped value set and the associated
terminology-pipeline overhead, which has historically been a bottleneck
for IPS — that overhead is the main reason this ticket has lingered.
Disposition C concedes that the alignment is a real implementer need
(otherwise the WGM discussion and the 2022 reopen would not have
happened) and is hard to defend on substance. The work group can later
migrate the invariant in Disposition A into a tightened value-set
binding if/when the IPS terminology infrastructure makes that cheap; in
the meantime the invariant is the right starting point.
