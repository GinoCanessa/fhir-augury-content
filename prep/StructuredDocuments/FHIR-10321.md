## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define and publish a `ValueSet` resource in the C-CDA on FHIR IG with
canonical URL
`http://hl7.org/fhir/ValueSet/2.16.840.1.113883.1.11.20.2.3` (or
republish under the IG's own canonical namespace and update the
`Composition.type` binding on the Referral Note Composition profile
accordingly). The ValueSet content should mirror the C-CDA template's
"Referral Document Type" / referral-note Composition.type code(s) —
typically the LOINC code(s) used to identify a Referral Note Composition
(e.g., `57133-1` "Referral note") drawn from the LOINC document type
ontology used elsewhere in C-CDA on FHIR. Wire the ValueSet into the
Referral Note Composition profile's `type` element binding (strength
preserved from the existing profile, expected `extensible` per C-CDA on
FHIR convention) so the published binding dereferences cleanly.

#### Justification

The reported defect is a broken canonical reference in a published
profile binding — implementers cannot resolve, validate against, or
expand the bound value set, which undermines conformance tooling for
Referral Note Compositions. Publishing the missing ValueSet is the
minimum change that restores integrity to the existing profile and is
consistent with how every other Composition.type binding in C-CDA on
FHIR is realised (a published ValueSet with a stable canonical URL).
The reporter's request, the reopener's note, and the originally linked
page all converge on this artifact, and no Zulip or cross-reference
evidence suggests scope creep is warranted.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting a new ValueSet at the legacy OID-shaped canonical
`http://hl7.org/fhir/ValueSet/2.16.840.1.113883.1.11.20.2.3`, retire
that canonical and rebind `Composition.type` on the C-CDA on FHIR
Referral Note Composition profile to an existing, actively maintained
document-type ValueSet — preferring a US Realm Header / C-CDA-aligned
ValueSet already published either in C-CDA on FHIR itself or referenced
from US Core (the LOINC-based document-type value set used across
Composition profiles in the IG). Add a brief note in the profile narrative
explaining the historical OID and documenting the migration so existing
implementers can find the new binding.

#### Justification

The OID-shaped `hl7.org/fhir/ValueSet/<oid>` canonical pattern is a
2010-era artefact; the C-CDA on FHIR IG has, in its more recent
revisions, standardised on human-readable canonical URLs scoped to the
IG. Reusing an existing, maintained ValueSet avoids creating a
single-purpose terminology resource whose membership would have to be
curated independently and re-balloted on every C-CDA template revision.
This carries a small backward-compatibility cost (any implementer who
hard-coded the old canonical must update) but yields a cleaner, more
maintainable terminology surface and aligns Referral Note with the
binding pattern used by sibling Composition profiles.

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Mod / Not Persuasive – Already Addressed**:
since the ticket pre-dates STU1.2, verify in current STU (STU1.2+) that
the Referral Note Composition profile no longer references the
`2.16.840.1.113883.1.11.20.2.3` canonical, and if the binding has been
silently re-pointed to a working ValueSet during a subsequent
publication, document that and close without further normative change.

#### Justification

The ticket has been triaged for almost a decade and was reopened only to
preserve the historical URL after STU1.2 shipped. If the Referral Note
Composition profile's published binding has already been migrated to a
valid canonical (which is plausible given the IG's STU1.2 republication
cycle), then no normative work remains and the right outcome is to
record that fact and close. Spending publication-cycle effort
re-defining a value set that no live binding still references would be
out of scope for the current Structured Documents backlog.

---

### Recommendation

**Recommended disposition:** A

The reporter's complaint is a concrete, narrowly scoped publication
defect: a binding canonical that does not resolve. Disposition A
restores integrity to the published Referral Note Composition profile
with the smallest possible change and matches the pattern used by every
other Composition.type binding in the IG. Disposition B is attractive
in the abstract but expands the ticket from "publish a missing artifact"
to "redesign a binding," which deserves its own ballot-cycle discussion
rather than being folded in here. Disposition C should only be invoked
if a quick check of the current STU build confirms the binding has
already been silently fixed; absent that confirmation, declining a
ticket whose reopener explicitly asked for post-STU1.2 follow-through
would be hard to defend. The workgroup should adopt A and, in the
course of authoring the ValueSet, evaluate whether to also flag the
canonical-naming question raised by B for a separate, IG-wide tracker.
