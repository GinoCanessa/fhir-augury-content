## Summary

The C-CDA on FHIR Implementation Guide binds `Composition.type` (on the
History and Physical composition profile, and likely on additional C-CDA
on FHIR composition profiles) to the value set canonical
`http://hl7.org/fhir/ValueSet/2.16.840.1.113883.1.11.20.22`. That
canonical URL does not resolve to a published FHIR ValueSet — the OID
refers to the HL7 CDA value set "HITSP C80 Clinical Document Type" /
"DocumentType" (OID `2.16.840.1.113883.1.11.20.22`), but no
corresponding FHIR ValueSet resource has been published under
`hl7.org/fhir/ValueSet/...` with that identifier. The reporter (Corey
Spears) asks that a real, resolvable ValueSet be added so the binding
becomes valid. Lynn Laakso confirmed in 2024 that the issue should be
re-opened post-STU 1.2 publication, and pointed at the
History-and-Physical composition page
(`http://build.fhir.org/ccda/ccda-history-and-physical-composition.html`)
as the original context. There are no cross-referenced sibling Jira
tickets; the only Zulip cross-reference is the automated `FHIR Bot`
tracker-item announcement in `#committers`, not a substantive
discussion.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Publish a real FHIR `ValueSet` resource with canonical
`http://hl7.org/fhir/ValueSet/2.16.840.1.113883.1.11.20.22` (preserving
the OID-based canonical so existing artifacts that reference it
continue to resolve) inside the C-CDA on FHIR IG. The ValueSet should
faithfully represent the HL7 CDA "HITSP C80 / DocumentType" value set
that the OID denotes — that is, the LOINC document-type codes
historically used for that binding — and be registered in the IG's
terminology resources so it appears in the published artifact list. All
C-CDA on FHIR composition profiles whose `Composition.type` currently
binds to this URL (starting with the History and Physical composition
referenced in the comment, and any siblings that share the binding)
keep the same canonical reference; nothing in the profiles needs to
change. Add a short narrative on the ValueSet page noting its
provenance from the CDA OID.

#### Justification

Accepting as requested fixes the immediate validation/resolution
defect: today the canonical does not resolve, so any tool that
dereferences the binding (validators, IG publishers, terminology
servers) reports a broken link. Preserving the OID-shaped canonical
keeps existing instances and external references stable, which is
consistent with FHIR's general guidance that canonical URLs are
identifiers and should not change once published. Re-using the CDA OID
semantics also keeps the C-CDA on FHIR mapping faithful to the source
CDA definition, which is the whole point of the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than republishing the OID-shaped canonical, retire the broken
binding and re-bind `Composition.type` on the affected C-CDA on FHIR
composition profiles to a properly-named, IG-local ValueSet such as
`http://hl7.org/fhir/us/ccda/ValueSet/composition-type` (or the
appropriate per-document-type value set already maintained by SDWG /
the LOINC Document Ontology, e.g. the FHIR core "FHIR Document Type
Codes" value set). The new ValueSet would be defined as an intensional
include of the relevant LOINC document-type codes — for History and
Physical, the LOINC codes used by the C-CDA H&P template. Update each
C-CDA on FHIR composition profile to point at the new canonical, and
publish a redirect / deprecation note on the
`hl7.org/fhir/ValueSet/2.16.840.1.113883.1.11.20.22` URL so old
references degrade gracefully.

#### Justification

The current canonical embeds an OID inside the FHIR `ValueSet` URL
namespace, which is awkward stylistically and out of step with how
other US Realm IGs name terminology artifacts; it also conflates "FHIR
core URL space" (`hl7.org/fhir/...`) with C-CDA on FHIR IG content,
which more recent IG guidance discourages. Re-binding to an IG-local,
human-readable canonical makes the binding discoverable in the IG TOC,
makes governance clear (the C-CDA on FHIR IG owns it), and allows the
value set to evolve (e.g., add new LOINC document codes) without
touching a URL pinned to a CDA OID. The trade-off vs. Disposition A is
a one-time disruption to any consumer that hard-coded the OID
canonical.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the binding can be addressed
by changing the binding target rather than minting a new ValueSet at
the broken canonical: simply re-bind `Composition.type` on the
affected profiles to an existing, already-published ValueSet (e.g., the
FHIR core `http://hl7.org/fhir/ValueSet/doc-typecodes` "FHIR Document
Type Codes" value set, or the equivalent LOINC document-ontology value
set already used elsewhere in C-CDA on FHIR), and remove all references
to the OID-shaped canonical from the IG. No new ValueSet resource is
created.

#### Justification

The reporter asked for a missing ValueSet to be "added", but the
underlying need — that `Composition.type` have a resolvable, sensible
binding — is already met by ValueSets that exist in FHIR core / the
LOINC ecosystem. Creating a new C-CDA-on-FHIR-owned ValueSet at the
old OID-shaped canonical perpetuates an awkward URL and adds
maintenance burden for an artifact that duplicates content already
governed elsewhere. Declining (in the "address by re-binding, not by
creating") sense is defensible if SDWG's intent is to align C-CDA on
FHIR with FHIR-native terminology rather than mirroring CDA OID-based
value sets.

---

### Recommendation

**Recommended disposition:** B

The defect is real and long-standing (open since 2016, re-opened in
2024 after STU 1.2), so doing nothing is not viable. Disposition A is
the smallest possible fix and preserves stability, but it bakes a CDA
OID into the FHIR core URL namespace, which is a long-term governance
liability that future C-CDA on FHIR maintainers will have to keep
publishing forever. Disposition C is cleanest but discards the explicit
CDA→FHIR semantic mapping that is the IG's reason for existing —
H&P-specific document codes are not necessarily the same set as the
generic FHIR document type codes. Disposition B threads the needle: it
gives `Composition.type` a resolvable, IG-owned binding; names it in a
way SDWG can govern; keeps the CDA-derived semantics (because the
ValueSet is defined intensionally over the same LOINC codes the OID
denoted); and the one-time canonical change is acceptable given that
the current canonical is already broken and therefore cannot have
many compliant consumers in the wild. SDWG should confirm the exact
LOINC code set with the H&P template authors before publishing.
