## Summary

Rick Geimer proposes that the `Composition.identifier` element be optional
(cardinality `0..1`) in the **US Realm Header** profile of the *C-CDA on
FHIR* implementation guide. His rationale is that `Composition.identifier`
is the FHIR equivalent of CDA's `setId`, and `setId` is optional in CDA
itself — so constraining the FHIR profile to require an identifier is
inconsistent with the CDA semantics the profile is meant to mirror.
Graham Grieve's brief comment ("it already is?") suggests the underlying
`Composition` resource is already `0..1`, while Rick's clarifying comment
makes clear that the request targets the *US Realm Header profile*
specifically — i.e., a profile-level slicing/cardinality constraint that
tightens the base resource. The cross-reference index links this ticket
to the FHIR artifact `Composition.identifier` and to a single
auto-generated `[committers] > tracker-item` Zulip notification; no other
Jira tickets, GitHub PRs/issues, or substantive Zulip threads are
associated with the request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the *C-CDA on FHIR* IG, change the cardinality of
`Composition.identifier` in the **US Realm Header** profile (and any
profiles that derive from it, e.g., the document-type-specific headers
such as CCD, Discharge Summary, etc.) from `1..1` back to the base
resource's `0..1`. Update the profile's StructureDefinition differential
accordingly, regenerate the snapshot, and update the corresponding row
in the profile's element table / mapping page so the CDA-to-FHIR mapping
explicitly notes that `Composition.identifier` mirrors the optional
`ClinicalDocument.setId` (rather than the required
`ClinicalDocument.id`).

#### Justification

The reporter's rationale is internally consistent with the
CDA-to-FHIR mapping the IG is built on: `ClinicalDocument.setId` is
optional in CDA R2, and the IG's stated design intent for
`Composition.identifier` is to carry that `setId` value. Tightening the
profile to `1..1` therefore creates a constraint that has no analogue in
the source standard and forces implementers to mint an identifier that
CDA producers were never required to supply, which is awkward when
re-publishing existing CDA documents as FHIR Compositions. Aligning with
the base resource's `0..1` keeps the profile faithful to CDA semantics
without losing any expressivity for senders that *do* have a stable
document set identifier.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `Composition.identifier` at `1..1` in the US Realm Header profile,
but (a) document explicitly in the profile's narrative and the
CDA-to-FHIR mapping that when the source `ClinicalDocument.setId` is
absent, implementers SHOULD populate `Composition.identifier` by
echoing `Composition.id`/`ClinicalDocument.id` (or another stable
business identifier), and (b) add a guidance note distinguishing
"document instance identifier" (`Composition.id`) from "document set
identifier" (`Composition.identifier`). Optionally relax the cardinality
to `0..1` only in a future major version of the IG with an explicit
migration note.

#### Justification

In practice, most C-CDA documents in production carry a `setId`, and the
FHIR community generally encourages that `Composition.identifier` be
populated so that successive versions of a document can be correlated
across systems — the same role `setId` plays in CDA. Keeping the
constraint preserves a useful interoperability invariant for downstream
consumers (e.g., XDS-style document registries) and avoids a
backward-compatibility flag day for implementers who built against the
existing profile. The cost of Disposition A is non-trivial because
existing validators, test data, and conformance reports will all start
treating documents that were previously *required* to carry an
identifier as conformant without one.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the request is based on a
misreading of the IG and that no implementer has reported a real-world
production issue caused by the existing constraint. Close as "Not
Persuasive" / "Already Sufficient" and point the reporter at Graham's
comment that the base `Composition.identifier` is already `0..1`, with a
note that the profile-level tightening is a deliberate design choice.

#### Justification

The ticket has been in *Triaged* status since 2018 with no follow-up
activity, no Zulip discussion, no linked PRs, and no other tickets
piling on the same concern — a signal that the constraint is not
causing field problems. The only substantive discussion artifact is
Graham's one-line "it already is?", which suggests the workgroup was
not convinced at the time that a profile change was warranted. Without
a concrete implementer use-case showing harm from the `1..1`
constraint, declining preserves the IG's stability and avoids a
breaking conformance change.

---

### Recommendation

**Recommended disposition:** A

The reporter's argument is grounded in the CDA-to-FHIR mapping that
underpins the entire *US Realm Header* profile: if
`Composition.identifier` is the FHIR carrier for `ClinicalDocument.setId`,
and `setId` is optional in CDA, then a `1..1` constraint on the FHIR side
introduces a requirement that does not exist in the source standard.
Disposition B's "leave it tight, document a workaround" path effectively
asks implementers to fabricate identifiers for documents whose CDA
originals had none, which weakens the semantics of
`Composition.identifier` as a *set* identifier. Disposition C is
defensible only because the ticket has gone stale, not because the
underlying argument is wrong. Relaxing to `0..1` is a non-breaking
change for senders (anything previously valid remains valid) and a
modest one for receivers (who already had to handle the optional
element on the base resource), so the cost-to-correctness ratio favours
acceptance. The workgroup should confirm during disposition that the
same relaxation is propagated to any document-type-specific profiles
that inherit from the US Realm Header.
