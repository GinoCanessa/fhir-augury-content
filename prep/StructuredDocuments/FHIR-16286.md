## Summary

The ticket asks the IPS (International Patient Summary, FHIR) work group to
consider how medicinal-product **package information** should be carried in
an IPS FHIR document when a common product identifier is not available —
the typical situation in cross-border exchange. In the IPS CDA the textual
description of the product also includes package data (e.g., package size,
form, units per pack), but the corresponding FHIR R4 `Medication` resource
no longer carries the `Medication.package` backbone that existed in
DSTU2/STU3. The reporter (Giorgio Cangioli) asks the work group to
evaluate whether a future IPS FHIR release should rely on another resource
— `MedicationKnowledge` is named explicitly — to convey package data so
that IPS FHIR documents can match the expressivity of the IPS CDA. There
are no cross-referenced Jira tickets and no substantive Zulip discussion
to incorporate.

## Proposed Dispositions

> Note: With no GitHub cross-references and therefore no repo briefings,
> the dispositions name candidate authoring repos by convention (the IPS
> FHIR IG lives in `HL7/fhir-ips`; the base spec lives in `HL7/fhir`),
> but no concrete authoring-root or file-path commitments are made — the
> work group should pick a path during disposition and a follow-up
> `repo-analysis` run should anchor the eventual implementation plan.

### Disposition A: Accept As Requested

#### Proposal

In the next IPS-FHIR update, extend the IPS profile of `Medication` (and
the IPS `MedicationStatement` profile that references it) so that
package-level information can be conveyed when no common product
identifier is available. Concretely:

1. Add an IPS-specific extension `medication-package` on the IPS
   `Medication` profile carrying a small structured payload
   (`type` : CodeableConcept for container/package type bound to
   the relevant SNOMED CT / EDQM packaging value set,
   `quantity` : SimpleQuantity for amount per package,
   `count` : positiveInt for number of packages) — this restores the
   STU3 `Medication.package` semantics that the IPS CDA narrative
   currently carries.
2. Allow the same extension to appear on the contained `Medication`
   referenced from `MedicationStatement.medicationReference` so that the
   IPS document can convey package data without depending on a Medication
   master file.
3. Update the IPS narrative-generation guidance so that, when present,
   the package extension is rendered into the human-readable section
   alongside the product description, matching the IPS CDA narrative.

#### Justification

This is the lowest-impact way to achieve parity with the IPS CDA
narrative without a base-spec change: extensions are the FHIR-sanctioned
mechanism for adding data IPS needs but the core `Medication` resource
does not currently carry. It keeps the change inside the IPS IG
(authoring repo `HL7/fhir-ips`) rather than reopening core
`Medication`, and it directly answers the reporter's stated need —
conveying package data in the cross-border, identifier-less case.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than extend `Medication`, model the medicinal product (including
package information) using `MedicationKnowledge` and reference it from
the IPS `MedicationStatement` via an extension on `medicationReference`
(or by allowing a contained `MedicationKnowledge` instance in the IPS
profile). Specifically:

1. Define an IPS `MedicationKnowledge` profile that constrains
   `MedicationKnowledge.packaging` (type + quantity), `code`,
   `doseForm`, and `ingredient` to the IPS bindings.
2. Add an IPS extension `medication-knowledge-reference` on the IPS
   `MedicationStatement` (and IPS `MedicationRequest`/`MedicationAdministration`
   profiles, where in scope) that points to the IPS
   `MedicationKnowledge` instance.
3. Document the cross-border usage pattern in the IPS narrative section
   for medications, including how the package data flows from
   `MedicationKnowledge.packaging` into the human-readable narrative.

#### Justification

This route follows the FHIR R4+ design intent: `Medication` is the
*administrable* concept; product-knowledge data (including packaging,
manufacturer, regulatory information) lives in `MedicationKnowledge`.
Using `MedicationKnowledge` aligns the IPS with how other communities
(e.g., national medication-knowledge IGs, IDMP-aligned work) are already
modelling product data, and avoids accreting IPS-only extensions on
`Medication` that re-introduce removed STU3 semantics. The trade-off
versus Disposition A is heavier: implementers must populate and
reference an additional resource, and `MedicationKnowledge` itself is
still maturing, which could affect IPS conformance stability across
cross-border partners.

---

### Disposition C: Decline

#### Proposal

Decline the request and instead document the existing pattern: when a
common product identifier is unavailable, IPS implementers should convey
the product description — including any package text — in
`Medication.code.text` and/or in the `MedicationStatement.note` /
narrative `<text>` block, exactly as the IPS CDA does today. No
structured package element or extension would be added; the IPS template
narrative guidance would be tightened so that the package portion of the
description is consistently rendered.

#### Justification

`Medication.package` was deliberately removed from R4 because the work
group concluded that packaging is product-knowledge information rather
than administration data. Re-adding it via an IPS extension risks
diverging from base-spec direction, and the cross-border use case can be
served by the existing free-text mechanisms that the IPS CDA itself
uses. Declining preserves consistency with the R4 design and keeps the
IPS profile minimal; the cost is that the package data remains
unstructured in IPS-FHIR (queryable only via narrative), which is a
known limitation but is consistent with current IPS-CDA practice.

---

### Recommendation

**Recommended disposition:** B

`MedicationKnowledge` is the resource the FHIR community deliberately
chose for product-level information (including `packaging`) when
`Medication.package` was retired in R4, and the reporter himself
nominates it in the original ticket text. Re-introducing package data
via an IPS-only extension on `Medication` (Disposition A) would solve
the immediate cross-border narrative parity but would re-create exactly
the conflation between *administrable* and *product-knowledge* data
that R4 worked to remove, and would saddle IPS with a divergent pattern
that other medication IGs do not share. Declining outright (Disposition
C) leaves a real cross-border interoperability gap — the IPS CDA *does*
carry structured package text today, and IPS-FHIR should be able to
match it. Disposition B accepts a heavier modelling cost in exchange
for staying on the FHIR roadmap and re-using `MedicationKnowledge.packaging`
as it was intended; the work group should weigh this against the
maturity level of `MedicationKnowledge` in the target IPS-FHIR release.
