## Summary

The reporter notes that, on the C-CDA on FHIR *Transfer Summary
Composition* profile, `Composition.encounter` does not constrain the
referenced `Encounter` to the DAF Encounter profile
(`http://hl7.org/fhir/StructureDefinition/daf-encounter`). The
expectation, expressed via a `<profile value="…/daf-encounter"/>`
fragment in the description, is that the profile slot under
`Composition.encounter.type.profile` should reference the DAF Encounter
StructureDefinition so that conforming Transfer Summary Compositions
explicitly require a DAF-conformant Encounter.

The ticket was originally filed against the DSTU2 era of C-CDA on FHIR
in 2016 and was reopened by Lynn Laakso post-STU 1.2 publication
(2024-03-12) for re-triage. No cross-referenced Jira tickets were
returned, so there is no formally linked sibling work; the only
non-Jira reference is an auto-generated FHIR Bot Zulip announcement of
the tracker item itself, with no substantive discussion attached.

Because the DAF profiles named in the original request have since been
deprecated and superseded by US Core (DAF Encounter was retired and
replaced by `us-core-encounter`), the disposition has to choose between
honoring the original intent (constrain `Composition.encounter` to a
specific Encounter profile, now US Core) versus declining to add a
cross-IG profile lock at the Composition level.

## Proposed Dispositions

### Disposition A: Accept As Requested (modernized to US Core)

#### Proposal

On the C-CDA on FHIR *Transfer Summary Composition* (and, for
consistency, the analogous CCD / Discharge Summary / Referral Note /
Progress Note Compositions where they include `Composition.encounter`),
constrain the `Composition.encounter` element so that the
`Reference(Encounter)` target's `type.profile` is fixed (or required)
to the current US-realm Encounter profile, i.e.:

- `targetProfile = http://hl7.org/fhir/us/core/StructureDefinition/us-core-encounter`

Because DAF Encounter (`.../StructureDefinition/daf-encounter`) was
retired and superseded by US Core Encounter, the literal request — bind
to `daf-encounter` — should be honored in spirit by binding to its
successor profile rather than to the dead DAF URL. The change is a
single `targetProfile` addition on `Composition.encounter` in the
relevant Composition StructureDefinition(s) under the C-CDA on FHIR IG.

#### Justification

This is what the ticket actually asks for, updated for the current
state of US-realm profiles. C-CDA on FHIR is a US-realm IG and already
declares a dependency on US Core in current STU work, so naming
`us-core-encounter` as a `targetProfile` is consistent with the rest of
the IG and gives implementers an unambiguous Encounter shape when an
encounter context is asserted on a Transfer Summary. It also resolves
the original 2016 complaint without leaving the spec pointing at a
deprecated DAF artifact.

---

### Disposition B: Alternative Approach — Narrative Guidance, No `targetProfile` Lock

#### Proposal

Do not add a `targetProfile` constraint on `Composition.encounter`.
Instead, add a *Notes / Usage* paragraph to the Transfer Summary
Composition page (and the IG's Composition guidance) stating that
*when* a Composition references an `Encounter`, the referenced
Encounter SHOULD conform to US Core Encounter
(`us-core-encounter`). Optionally, reinforce this with a
`Composition`-level `obligation`/`mustSupport` note rather than a hard
profile constraint.

#### Justification

Compositions in C-CDA on FHIR are document-header artifacts and are
often produced by tooling that already binds Encounter conformance at
the Bundle/Encounter resource level. Locking `targetProfile` at the
Composition level can create double-validation friction and
cross-profile compatibility issues for senders that legitimately use a
non-US-Core Encounter (e.g., an international Encounter or a
specialty profile that derives from `us-core-encounter` but uses a
different canonical URL). Guidance accomplishes the original intent
(make it clear which Encounter shape is expected) without introducing
a breaking constraint and without creating an STU-versus-publication
churn cycle each time the US Core Encounter canonical URL is rev'd.

---

### Disposition C: Decline — Persistent Effect Already Achieved Elsewhere

#### Proposal

Decline the change. Rationale: the original request named a profile
(`daf-encounter`) that has been retired, the C-CDA on FHIR IG already
either constrains the included Encounter resource directly or inherits
US Core Encounter conformance through its IG dependency graph, and
adding a `Composition.encounter` `targetProfile` constraint risks
breaking existing conformant senders for negligible additional
clarity. Close as *Not Persuasive — Superseded* with a note pointing
implementers to the IG's Encounter conformance statement.

#### Justification

The ticket is from 2016 and predates the deprecation of DAF and the
adoption of US Core as the canonical US-realm Encounter profile.
`Composition.encounter` is a `Reference(Encounter)`; conformance to a
specific Encounter shape is conventionally asserted on the Encounter
resource itself, not at the Composition reference site, and this is
how the rest of C-CDA on FHIR handles patient/author/etc. Adding a
`targetProfile` here would be inconsistent with the IG's existing
pattern and would add a constraint that produces no new validation
power that the current Encounter profile constraint doesn't already
produce.

---

### Recommendation

**Recommended disposition:** A (modernized to US Core)

The original 2016 complaint is still substantively valid:
`Composition.encounter` on the Transfer Summary Composition profile
does not name an Encounter profile, and a reader of that page cannot
tell which Encounter shape is expected. The 2024 reopen by Lynn Laakso
post-STU1.2 explicitly preserves the issue for re-triage, which
suggests the workgroup has not considered it resolved by the move from
DAF to US Core. Adding a single `targetProfile = us-core-encounter`
on `Composition.encounter` is a small, well-scoped change that brings
the Transfer Summary Composition into line with how other US-realm
references are handled in the IG and that costs implementers little
beyond what they're already doing for US Core conformance. Disposition
B is a reasonable fallback if the workgroup is concerned about
profile-lock churn; Disposition C should only be chosen if the
workgroup confirms the constraint is already enforced elsewhere in the
IG.
