## Summary

The ticket reports a stale editorial comment on `Composition.title`. The
element's cardinality was relaxed from `1..1` (in R4) to `0..1` in the
current FHIR Core build, but the element's accompanying comment text
still tells readers that `title` is mandatory and invites feedback on
that requirement during the trial use period. The reporter (Ryan
Zoellner) asks that the comment be cleaned up so it no longer
contradicts the element's actual cardinality. There are no
cross-referenced Jira tickets, Zulip discussions, or GitHub items —
only an artifact-level cross-reference to `Composition.title` itself.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `comment` (element-level comments) on `Composition.title`
in the FHIR Core specification so that it no longer asserts that the
element is mandatory and no longer solicits trial-use feedback on a
mandatory cardinality. Specifically, replace the current sentence —
"...Note that CDA does not make title mandatory, but there are no
known cases where it is useful for title to be omitted, so it is
mandatory here. Feedback on this requirement is welcome during the
trial use period." — with text that reflects the current `0..1`
cardinality. A reasonable replacement: "CDA does not make `title`
mandatory, and FHIR likewise leaves it optional (`0..1`); however,
implementers are strongly encouraged to populate `title` because
there are very few situations where omitting it is useful." No
structural / cardinality / binding change is implied — this is a
documentation-only fix on the existing element.

#### Justification

The stated text is factually inconsistent with the element's
cardinality in the current build. Element comments are normative
guidance and should not contradict the cardinality printed
immediately above them. This is exactly the class of editorial drift
that the "Technical Correction" issue type exists to handle, so the
change is low-risk, requires no ballot, and removes a reader-facing
contradiction. Retaining the "feedback during trial use" sentence is
also misleading because the original requirement (mandatory `title`)
no longer exists to receive feedback against.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than rewriting the comment in place, refactor the guidance
into two separate sentences and move the stronger "should populate"
guidance into a `usage note` or short `requirements` paragraph on
`Composition.title`. The element `comment` would be reduced to a
single neutral sentence describing the relationship to CDA's
optional `title` (e.g., "CDA does not make `title` mandatory; FHIR
likewise leaves it optional."). The implementer-facing
recommendation ("populate `title` whenever possible — omission is
rarely useful") would live in `Composition.title.requirements` (or
a short note in the resource-level narrative for `Composition`),
where prose guidance is more discoverable to humans skimming the
resource page than dense comment text.

#### Justification

This better separates the two concerns currently tangled in the same
sentence: (1) a factual statement about CDA's cardinality, which is
reference material, and (2) an implementer recommendation, which is
guidance. Splitting them makes both easier to maintain (e.g., a
future change to how strongly we recommend populating `title`
doesn't require re-touching CDA-relationship text) and is consistent
with how other `Composition` elements separate `comment` from
`requirements`. Trade-off vs. Disposition A: it touches one extra
field (`requirements`) and so is a slightly larger edit, but it
leaves the spec in a tidier long-term state.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as "Not Persuasive — Editorial". Leave the existing
comment in place on the grounds that (a) the comment can be read as
historical context for the cardinality change rather than a current
normative assertion, and (b) the cost of re-editing element comments
for every cardinality relaxation outweighs the clarity benefit on a
self-evident contradiction.

#### Justification

This is the weakest of the three options and is included for
completeness. The only defensible rationale for declining is
work-prioritisation: editorial fixes consume editor time that could
go to substantive issues. However, the ticket is unambiguous, the
fix is small, and the contradicting text is visible on every render
of the `Composition` resource — so "decline" is hard to defend on
the merits.

---

### Recommendation

**Recommended disposition:** A

The ticket is a textbook Technical Correction: the element's printed
cardinality (`0..1`) and its prose comment ("…it is mandatory
here…") are in direct contradiction, and there is no community
disagreement to weigh (no Zulip threads, no linked tickets, no PRs).
Disposition A removes the contradiction with the smallest possible
change and preserves the existing structure of `Composition.title`'s
documentation. Disposition B is a defensible refinement and the
workgroup may choose to bundle the comment cleanup with a small
rewrite of `requirements`, but that is a stylistic preference rather
than a substantive improvement. Disposition C is not recommended —
leaving a self-contradicting comment in the published spec is a poor
outcome for implementers reading the resource page for the first
time.
