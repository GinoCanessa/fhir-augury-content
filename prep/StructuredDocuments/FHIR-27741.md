## Summary

The reporter observes that the published US C-CDA on FHIR Implementation
Guide carries a broken / misdirected set of navigation links. Two
specific problems are called out: (1) the top menu links on the IG do
not behave consistently — it is unclear whether they are intended to be
generic FHIR core links or links scoped to this IG; and (2) the
"propose a change" link in the page footer still points to the
long-retired GForge tracker rather than to the HL7 Jira project where
ballot comments and change requests are now filed. Because the
cross-reference graph is empty, no related Jira tickets, Zulip threads,
or GitHub items were surfaced by FhirAugury for this issue, so the
request stands on its own.

### Disposition A: Accept As Requested

#### Proposal

Patch the US C-CDA on FHIR IG so that:

1. The footer "Propose a change" link is repointed from the legacy
   GForge tracker URL to the canonical HL7 Jira intake URL for this IG
   (the standard `https://jira.hl7.org/secure/CreateIssueDetails!init.jspa?...`
   form, prefilled with `pid` for the FHIR project, the appropriate
   Specification component for "US C-CDA on FHIR (FHIR)", and the
   current page URL).
2. The top-of-page menu is reviewed and corrected so each entry either
   (a) resolves to the intended IG-local page, or (b) is explicitly
   labelled and styled as a link out to the FHIR core spec when that
   is the desired behaviour. Entries that are dead or duplicative are
   removed.

The change lands in the IG's publisher inputs — typically `menu.xml`
(or the equivalent `input/menu.xml`) for the menu, and the
publication template's footer/jira link configuration (commonly the
`jira-spec` / `propose-changes` value in `ig.ini` or the IG's
`publication-request.json` / template includes) for the footer link.

#### Justification

This is a straightforward technical correction, exactly as the ticket
type ("Technical Correction") indicates. GForge has been retired for
years; leaving the footer pointed at it strands implementers who try
to file feedback and silently loses change proposals. Aligning the
menu with the rest of the HL7 IG family is consistent with the
publishing conventions the IG publisher already enforces for newer
guides, and does not change any normative content.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than editing the IG inputs in this repository in isolation,
adopt the current shared HL7 IG template / publisher defaults wholesale
for US C-CDA on FHIR:

- Bump the IG's template reference (in `ig.ini`) to the current
  `hl7.fhir.template` (or the C-CDA-on-FHIR-specific descendant) that
  already wires the footer "Propose a change" link to Jira via the
  standard macro and renders a consistent menu.
- Re-run the IG publisher and accept the regenerated menu / footer,
  only overriding entries where the IG genuinely needs an
  IG-specific item.

#### Justification

The literal request fixes two visible symptoms, but the underlying
cause is that this IG is on an older template that pre-dates the
current Jira-based feedback wiring. Re-templating fixes both reported
problems and prevents the same drift from recurring the next time HL7
changes its feedback URL. The trade-off is a larger diff and the need
to re-publish the IG, but it brings US C-CDA on FHIR in line with the
rest of the HL7 IG estate and reduces ongoing maintenance.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — superseded by republication." If the IG has
since been republished on a current template (or is slated to be in
the next ballot cycle) and the live published version already shows a
working Jira "Propose a change" link and a coherent menu, the
underlying defect no longer exists and no further action is warranted.

#### Justification

The ticket was filed in May 2020 against a published artifact and has
sat in `Triaged` with no comments for the entire intervening period.
If a more recent publication of US C-CDA on FHIR has already moved to
the standard template (which is what fixed the same class of defect
across most HL7 IGs), reopening the source to "fix" something that no
longer reproduces would be churn for no benefit. Declining is only
defensible after verifying against the current published IG, however —
otherwise it strands the original report.

---

### Recommendation

**Recommended disposition:** B

The reporter is right on the facts — pointing implementers at GForge
in 2020+ is a real defect, and an inconsistent menu actively confuses
readers about whether they are inside the IG or in the core spec. A
narrow fix (Disposition A) would clear both symptoms, but the root
cause is template drift; the same defect will reappear the next time
HL7's feedback infrastructure changes if this IG continues to carry
its own bespoke footer/menu wiring. Re-templating onto the current
shared HL7 IG template (Disposition B) fixes both reported problems,
restores parity with the rest of the HL7 IG family, and pays down the
maintenance debt that produced the ticket in the first place.
Disposition C is only appropriate if the workgroup can confirm the
defect no longer reproduces in the latest publication, which the
ticket as filed does not establish.
