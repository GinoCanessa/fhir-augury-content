## Summary

The reporter (Giorgio Cangioli) observes that the differential and snapshot
tree views rendered by the FHIR IG Publisher omit useful information that
profile authors have asserted on individual elements — for example, the
meaning of an `Identifier` slice in the Device profile. The reporter notes
that comparable tooling (Simplifier) does surface this information, and
asks the IG Publisher / IPS team to evaluate how to expose those
profile-authored details to readers of the published guide. The ticket is
filed against the International Patient Summary (FHIR) specification but,
per the most recent comment, the underlying issue is an IG Publisher
rendering gap rather than an IPS content gap. There are no
cross-referenced Jira tickets and no related GitHub items; the only
cross-reference is the automated FHIR Bot announcement of the tracker
item in the `#committers > tracker-item` Zulip topic.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as an IG Publisher rendering enhancement (consistent with
Lynn Laakso's 2022-11-22 note) and have the Structured Documents WG file
a forwarded request against the IG Publisher to extend the differential
and snapshot tree renderers so that profile-authored, element-level
annotations are visible inline in the rendered tree. Specifically, for
each `ElementDefinition` (and slice) in a `StructureDefinition`, surface
in the tree row (or a hover/expand control adjacent to it) the
profile-supplied values for at least:

- `ElementDefinition.label`
- `ElementDefinition.short`
- `ElementDefinition.definition`
- `ElementDefinition.comment`
- `ElementDefinition.requirements`
- For sliced elements, the slice name plus
  `ElementDefinition.slicing.discriminator` summary and the
  slice-specific `short` / `definition` (the "meaning of the identifier"
  case the reporter calls out).

The IPS guide would then automatically benefit, as would every other IG
that builds with the same publisher version. Close the IPS-side ticket
once the upstream IG Publisher change is merged and IPS rebuilds against
it.

#### Justification

The ticket text and the most recent WG comment both attribute the gap
to the IG Publisher's tree renderer, not to any missing content in the
IPS profiles. Profile authors are already populating the relevant
`ElementDefinition` fields; the data exists in the StructureDefinition
resources. Pushing the fix into the IG Publisher means every HL7 IG
(not only IPS) gets parity with the comparable Simplifier rendering the
reporter cites, which is consistent with FHIR's "fix it once, in the
shared tooling" pattern. It also avoids per-IG narrative duplication
that would otherwise drift over time.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the in-tree renderer, address the readability gap
at the IPS level by enriching the per-profile narrative pages: for each
IPS profile (Device, Patient, Condition, etc.), include a dedicated
"Element Details" / "Slice Details" section under the snapshot/diff
view that lists the profile-authored `short`, `definition`, and slice
discriminator metadata for any element where the profile sets a
non-trivial value. This can be produced today using the IG Publisher's
existing `{% include ... %}` / template hooks and an authored fragment
per profile, without any IG Publisher code change.

#### Justification

This delivers reader value to IPS users now, without waiting on an
upstream IG Publisher feature and without depending on the IG
Publisher maintainers' roadmap. It also gives the IPS WG editorial
control over which annotations are highlighted (the reporter's "Device
identifier meaning" example would be explicitly called out rather than
buried in a generic tree expander). The trade-off is duplication of
content already present in the StructureDefinition, ongoing maintenance
burden as profiles evolve, and the fact that the fix is local to IPS
while every other IG continues to suffer the same rendering gap.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive — wrong tracker / out of scope for the IPS
specification**. The request is a tooling enhancement against the IG
Publisher, not a change to the IPS specification or its profiles. The
correct path is for the reporter (or the WG on the reporter's behalf)
to file an issue at <https://github.com/HL7/fhir-ig-publisher>
describing the desired snapshot/diff tree behavior, and to close
FHIR-16226 with a pointer to that upstream issue.

#### Justification

The ticket is filed against "International Patient Summary (FHIR)" but
neither the description nor the WG comment identifies any IPS profile
content to add, modify, or remove. The 2022-11-22 comment from
lynn_laakso explicitly states "This is likely an IG Publisher issue."
Keeping the ticket open against IPS implies an action the IPS WG cannot
deliver on its own, and it competes for triage attention with tickets
that do require IPS content changes. Routing it to the IG Publisher
tracker puts it in front of the people who can actually act on it.

---

### Recommendation

**Recommended disposition:** A

Disposition A (forward to the IG Publisher as a rendering enhancement)
best matches both the reporter's intent — make profile-authored element
metadata visible to readers in every guide that uses the snapshot/diff
tree — and the WG's own 2022-11-22 assessment that this is an IG
Publisher concern. It scales: a single change in the publisher benefits
IPS plus every other HL7 IG, eliminating the per-IG narrative
duplication that Disposition B would create. Disposition C is
operationally close to A but loses the WG's sponsorship of the request;
because Structured Documents has already invested in re-opening this
ticket post-STU, the cleaner outcome is for the WG to formally route
it to the IG Publisher project (Disposition A's mechanism) and track
its closure there, rather than dropping the ticket on the reporter to
re-file. Disposition B should be held in reserve as a fallback only if
the IG Publisher team declines or defers the upstream enhancement
beyond an acceptable horizon for the next IPS publication.
