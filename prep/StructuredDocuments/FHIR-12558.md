## Summary

The ticket reports that the `Composition.attester` slicing on the
**C-CDA on FHIR US Realm Header** profile contains an invalid second
slice (id
`Composition:c-cdaonfhirusrealmheader.attester:attester_slice_pos17`)
that defines a second `attester.mode` slice without specifying any
fixed/pattern values to discriminate it. Because the slicing on
`Composition.attester` is already declared as **open**, a second
unconstrained slice is meaningless — additional attesters are already
permitted by the open slicing — and Eric Haas requests that the second
slice be removed.

The ticket has been resolved **Persuasive with Modification**: the
Structured Documents work group agreed to remove the redundant second
slice and to add narrative documentation clarifying that the profile
constrains the realm header to a single *legal* attester while still
allowing additional attesters via the open slice. There are no
explicitly-linked Jira tickets, and the only cross-reference is an
auto-generated tracker-bot post on Zulip, so the ticket is essentially
self-contained.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **C-CDA on FHIR US Realm Header** Composition profile
(`StructureDefinition` for `Composition` constrained for the US C-CDA
on FHIR realm header), delete the second `Composition.attester` slice
identified as
`Composition:c-cdaonfhirusrealmheader.attester:attester_slice_pos17`.
Retain the first `attester` slice that fixes `attester.mode` to
`legal` (with cardinality 1..1) and leave the slicing declaration
`@mode, open` unchanged so that additional attesters can still appear
via the open slice. Add a short narrative paragraph to the profile's
intro / `Composition.attester` element definition stating that exactly
one *legal* attester is required and that other attesters
(e.g. professional, informational) are permitted through the open
slicing.

#### Justification

This is exactly what the reporter asked for and matches the recorded
resolution ("Persuasive with Modification"). FHIR's slicing rules
require each non-default slice to carry a discriminating value; a
second `attester.mode` slice with no fixed/pattern value is
semantically a no-op that confuses tooling and validators rather than
constraining behaviour. Because the slicing is already `open`,
removing the redundant slice causes no loss of capability — additional
attesters remain valid — while the added narrative makes the
single-`legal`-attester intent explicit for implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of deleting the second slice, replace it with a properly
discriminated second slice that fixes `attester.mode` to a specific
non-`legal` value the realm header expects implementers to support
(for example, a `professional` slice with `mode = professional`, 0..*).
Keep the slicing as `@mode, open`, document each named slice in the
profile intro, and leave room for further mode-specific slices in
later versions.

#### Justification

If Structured Documents wants to express affirmative guidance for
additional attester roles (not just "anything goes via open
slicing"), a second properly-discriminated slice communicates that
intent in the differential rather than in prose. Trade-off vs.
Disposition A: this expands scope beyond what the ticket actually
asked for, requires WG agreement on which additional `mode` values to
call out, and risks over-constraining implementations that legitimately
attest with other modes. Choose this only if the WG has consensus on
the additional named modes; otherwise the open slice already covers
the use case.

---

### Disposition C: Decline

#### Proposal

Leave the profile unchanged on the basis that an extra open-equivalent
slice is harmless and that any change risks invalidating downstream
artefacts (IGs, validators, examples) built against the published
slice id `attester_slice_pos17`.

#### Justification

A defensible decline would rest on stability — published artefacts may
reference the slice id, and removing it is a (minor) breaking change
to the profile differential. However, this argument is weak: the slice
carries no constraint, so removing it changes nothing about which
instances validate, and FHIR validators flag undiscriminated slices as
warnings/errors. Decline is therefore hard to justify on substance.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the correct action and matches what the WG already
recorded as the resolution. The reported defect is a genuine slicing
error: a second `attester.mode` slice with no discriminating fixed
value violates FHIR slicing semantics and is redundant given the open
slicing already in place. Removing it plus adding narrative that
exactly one *legal* attester is required (others permitted via the
open slice) preserves all existing implementer capability, removes a
validator-confusing artefact, and clarifies intent. Disposition B
expands scope without a stated need, and Disposition C cannot be
defended on substance because the deleted slice contributes no
constraint.
