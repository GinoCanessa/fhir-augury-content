## Summary

The ticket asks the Structured Documents work group to clean up the
`StructureDefinition` profiles in the **US C-CDA on FHIR** IG by stripping the
Trifolia-generated CONF identifiers (e.g., `(CONF:2219-146)`) that appear at
the end of `differential.element.definition` text. These tokens are an
artifact of how the source C-CDA templates were exported from the Trifolia
authoring tool — they were originally meant to anchor numbered conformance
statements in Word/PDF output and downstream conformance test harnesses, but
they carry no defined meaning in a FHIR `StructureDefinition` and instead
clutter the human-readable definitions seen by IG readers and tooling. The
issue applies to "almost all" of the C-CDA profiles in the IG. The reporter
notes that Rick Geimer has the historical context for these statements. The
ticket was originally raised in 2016 (as GForge #10192), is still in `Triaged`
state, and was annotated by Lynn Laakso in March 2024 to be re-opened
post-STU 1.2 publication. No related Jira tickets, GitHub items, or
substantive Zulip discussions surfaced via cross-reference search — only the
auto-generated 2016 FHIR-Bot tracker-item announcement in `#committers`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Strip the Trifolia-generated `(CONF:NNNN-NNN)` tokens from the trailing end
of every `StructureDefinition.differential.element.definition` value across
the C-CDA on FHIR profile set. Operationally:

1. Identify the source pipeline that converts the Trifolia exports into the
   IG's `StructureDefinition` JSON/XML resources. Update that
   transformation step to drop any trailing `\s*\(CONF:[0-9-]+\)\s*$`
   pattern (and any whitespace it leaves behind) from each `definition`
   string, while preserving any text that precedes it.
2. If the IG no longer round-trips through Trifolia and the
   `StructureDefinition` resources are now hand-maintained, perform a
   one-time sweep of the profile resources in the IG source applying the
   same regex, and remove the tokens directly.
3. Re-publish the affected `StructureDefinition` resources, regenerate the
   IG narrative pages, and verify by spot-checking several of the rendered
   profile pages on `build.fhir.org/ccda/` that the CONF tokens no longer
   appear in element definition prose.
4. Coordinate with Rick Geimer (per the reporter) to confirm no downstream
   consumer of the IG's profile JSON/XML still relies on the embedded CONF
   tokens for traceability before the cleanup is published.

Schedule the change for the post-STU 1.2 cycle, as Lynn Laakso noted in the
2024 comment.

#### Justification

The reporter's analysis is correct on the FHIR side: a token like
`(CONF:2219-146)` has no defined semantics in a `StructureDefinition`. The
`element.definition` field is meant to be human-readable prose describing
the meaning of the element, and a bare conformance ID with no surrounding
explanation actively reduces clarity for IG readers. Removing the tokens
brings the C-CDA on FHIR profiles in line with how every other published
HL7 IG presents element definitions, and is a strictly narrative/text-only
change — it does not alter cardinality, types, bindings, slicing, or any
machine-processable conformance constraint, so the risk of breaking a
conforming implementation is essentially zero. The fact that the comment
explicitly defers the work to "post STU 1.2 publication" already
acknowledges this is a low-risk cleanup that the WG agrees should happen.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than deleting the Trifolia CONF identifiers outright, **relocate**
them out of `element.definition` prose and preserve them as machine-readable
traceability metadata:

1. For each affected element, strip the trailing `(CONF:NNNN-NNN)` token
   from `definition` (so the human-facing prose is clean, as in
   Disposition A).
2. Capture the same identifier in a structured location, either:
   - an `ElementDefinition.mapping` entry with a dedicated mapping
     `identity` (e.g., `cda-conf`) declared at the StructureDefinition
     level and a uri/name describing it as "C-CDA template conformance
     identifier", and the `map` field set to the CONF value; **or**
   - a small custom extension on `ElementDefinition` (e.g.,
     `http://hl7.org/fhir/us/ccda/StructureDefinition/conf-id`) whose
     value is the CONF string.
3. Document the mapping/extension in the IG so implementers and tooling
   that previously relied on the inline CONF tokens can continue to map
   FHIR profile elements back to the originating C-CDA template
   conformance statements.

#### Justification

The reporter notes that the CONF identifiers were originally used by
"conformance testing tools." Even if no current consumer is known, simply
deleting the tokens loses the only existing bridge between the FHIR profile
elements and the numbered conformance statements in the underlying C-CDA
templates. Moving them into `mapping` (the FHIR-native mechanism for
exactly this kind of cross-standard traceability) keeps that linkage
available to anyone who needs it while still cleaning up the prose. The
trade-off vs. Disposition A is more authoring effort (one mapping or
extension entry per affected element) and a marginally larger profile
footprint, but the work is mechanical and can be done by the same
transformation pass that strips the tokens, so the incremental cost is
modest.

---

### Disposition C: Decline

#### Proposal

Leave the `(CONF:NNNN-NNN)` tokens in place in
`StructureDefinition.differential.element.definition`. Close the ticket
as "Not Persuasive" on the basis that the tokens, while admittedly
FHIR-meaningless, are harmless trailing annotations that preserve
provenance back to the originating C-CDA template conformance
statements, and that mass-editing the profile set carries more risk than
the cosmetic benefit warrants.

#### Justification

Defensible only on a "do no harm" reading: the CONF tokens are visible
clutter but do not affect machine-readable conformance, validation, or
implementer behavior, and they encode a non-trivial piece of provenance
(which numbered statement in the C-CDA template each FHIR element
corresponds to). If the WG has no bandwidth and no clear consumer
demand, leaving them in place keeps that provenance intact. This is the
weakest of the three options, however, because the originator, the
work-group note from 2024, and the FHIR design expectation for
`element.definition` all point the other way.

---

### Recommendation

**Recommended disposition:** B

Disposition A cleanly resolves the reporter's complaint and is what Lynn
Laakso's 2024 "re-open post STU 1.2" note seems to anticipate, but it
discards the only existing trace from FHIR profile elements back to the
numbered C-CDA template conformance statements — and the reporter himself
flags that those identifiers were used by downstream conformance tooling
and recommends checking with Rick Geimer first. Disposition B gets the
same narrative-cleanup outcome (the prose stops being polluted with bare
CONF tokens) while moving the identifiers into a FHIR-native mapping
slot, which is exactly what `ElementDefinition.mapping` exists for and
costs only a mechanical extension of the same transformation pass that
would do Disposition A's deletion. Disposition C is hard to defend given
that the WG has already implicitly agreed the tokens should go. The
work-group should adopt B unless Rick Geimer (or another implementer)
confirms that no consumer of the C-CDA on FHIR IG cares about the
identifiers, in which case falling back to A is acceptable.
