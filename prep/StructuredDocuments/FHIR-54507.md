## Summary

The submitter is reading the `Comments` text on `Composition.status` in the
FHIR Core specification and is confused by the phrase **"document series"**.
The element comment currently says, in part, that if a composition is marked
as withdrawn, "the compositions/documents in the series, or data from the
composition or **document series**, should never be displayed to a user
without being clearly marked as untrustworthy." The reporter notes that the
term *document series* is not defined anywhere in the FHIR Composition
documentation, asks what it is meant to denote, and requests that
Structured Documents either clarify the term in place or replace it with a
different (defined) term.

There are no other Jira tickets, Zulip threads, or GitHub items
cross-referenced from this ticket. The only cross-reference returned by
FhirAugury is to the FHIR element `Composition.status` itself (a textual
mention, not a linked artifact).

## Details

**Description:**

> Compostion.status
>
> | | |
> |-|-|
> | Comments | If a composition is marked as withdrawn, the compositions/documents in the series, or data from the composition or **document series**, should never be displayed to a user without being clearly marked as untrustworthy. The flag "entered-in-error" is why this element is labelled as a modifier of other elements. Some reporting work flows require that the original narrative of a final document never be altered; instead, only new narrative can be added. The composition resource has no explicit status for explicitly noting whether this business rule is in effect. This would be handled by an extension if required. |
>
> What is a document series? Please clarify or use a different term.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `Comments` text on `Composition.status` in the Composition
resource (FHIR Core) to remove the undefined phrase **"document
series"** and replace it with an explicit, in-line definition of what
is meant. Concretely, reword the first sentence of the comment along
the lines of:

> If a composition is marked as `entered-in-error`, neither this
> composition/document nor any other composition/document in the same
> *document lineage* (i.e., other versions/revisions of the same
> logical document — those sharing the same `Composition.identifier`,
> or referenced by `Composition.relatesTo` with `code = replaces` /
> `appends` / `transforms`) should be displayed to a user without being
> clearly marked as untrustworthy.

Apply the same wording change consistently anywhere else in the
Composition narrative or element comments where the bare phrase
"document series" currently appears, so the resource uses one defined
term throughout.

#### Justification

The reporter's complaint is purely editorial: the comment uses a term
("document series") that is not defined anywhere in the Composition
resource's narrative or in the FHIR glossary, while clearly intending
to refer to the related set of documents addressed by
`Composition.identifier` and `Composition.relatesTo`. Tying the comment
to those existing, defined element-level mechanisms keeps the guidance
unambiguous, costs nothing in terms of conformance (definitions and
cardinalities are unchanged), and aligns with the FHIR convention that
element comments should be self-explanatory or refer to other defined
constructs in the resource. There is no Zulip controversy or
cross-ticket disagreement to weigh against the change.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the term **"document series"** but promote it to a defined term in
the Composition resource. Specifically:

1. Add a short paragraph to the Composition resource's *Notes* /
   *Background and Context* section defining "document series" as the
   set of compositions that share a logical identity — i.e., those
   sharing the same `Composition.identifier`, plus any compositions
   linked via `Composition.relatesTo` with `code = replaces`, `appends`,
   or `transforms`.
2. Link the first occurrence of "document series" in the
   `Composition.status` element comment (and anywhere else it appears
   on the Composition resource) to that anchor.

The element comment text otherwise remains unchanged.

#### Justification

If "document series" is already informally used in implementer
discussions or downstream IGs, defining the term once and linking to
it preserves existing language while still answering the reporter's
question. This reduces the diff against the current spec and avoids
churn in any IG that may already quote the existing comment verbatim.
The trade-off versus Disposition A is that readers must follow a link
to understand a single phrase in a comment that they would otherwise
expect to be self-contained, and Structured Documents would be taking
on the maintenance of a defined term that is not actually used in any
element name or value-set binding.

---

### Disposition C: Decline

#### Proposal

Decline the request as **Not Persuasive — already addressed in
context**. The element comment uses "document series" in apposition
("the compositions/documents in the series, or data from the
composition or document series"), and the surrounding text plus
`Composition.relatesTo` semantics make the meaning recoverable from
context. Recommend the reporter rely on `Composition.identifier` and
`Composition.relatesTo` (`replaces` / `appends` / `transforms`) as the
authoritative description of how documents relate, and close the
ticket with no spec change.

#### Justification

Defensible only if the workgroup judges the wording change too minor to
spend ballot cycles on, and is comfortable that the comment is
understandable in context to a careful reader. The downside is real:
the reporter (an active implementer) was confused enough by the
existing wording to file a ticket, which is direct evidence that the
text is not self-explanatory; declining leaves that confusion in
place. There is no compatibility, scope, or design-principle argument
that strongly favours leaving the wording untouched.

---

### Recommendation

**Recommended disposition:** A

The ticket is purely editorial and has zero conformance impact: it
asks the workgroup to either define a term it is already using or
replace it with one that is already defined. Disposition A does the
latter by leaning on existing, well-understood Composition machinery
(`Composition.identifier`, `Composition.relatesTo`), which is exactly
what implementers will reach for when they need to act on the
guidance. Disposition B is a reasonable second choice if the
workgroup wants to preserve the phrase, but it adds a defined term
that is not used anywhere else in the resource and creates a
maintenance burden out of proportion to the problem. Disposition C
ignores direct implementer feedback that the current text is
confusing, which is the weakest position to take on a `Highest`
priority change request whose only ask is "please clarify".
