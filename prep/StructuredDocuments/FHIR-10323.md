## Summary

The ticket reports that, in the **US C-CDA on FHIR** IG, the
`Composition.type` element on the (Consultation Note) Composition profile
is not bound to any value set. The reporter asks the IG to point the
binding at the C-CDA "DocumentType" value set identified by the OID
`2.16.840.1.113883.11.20.9.31` (referenced via the FHIR-style URI
`http://hl7.org/fhir/ValueSet/2.16.840.1.113883.11.20.9.31`) and to make
sure that value set is actually included in the IG build. The ticket was
originally filed against the Consultation Note composition profile
(`ccda-consultation-note-composition.html`) but the underlying gap —
`Composition.type` having no terminology binding — applies more broadly
to the C-CDA on FHIR composition profiles.

There are no cross-referenced Jira tickets and no related GitHub PRs or
issues. The only Zulip cross-reference is the original GForge tracker
notification bot post from 2016, not a substantive discussion. The
ticket was re-opened post STU 1.2 publication per Lynn Laakso's 2024-03-12
comment, so the ask is still outstanding for the next ballot/publication
of US C-CDA on FHIR.

### Disposition A: Accept As Requested

#### Proposal

In the **US C-CDA on FHIR** IG, bind `Composition.type` on the
**Consultation Note Composition** profile (and, by symmetry, on the other
C-CDA on FHIR composition profiles that currently leave the binding
undeclared) to the C-CDA "DocumentType" value set identified by OID
`2.16.840.1.113883.11.20.9.31`, exposed at the FHIR canonical
`http://hl7.org/fhir/ValueSet/2.16.840.1.113883.11.20.9.31`. Use binding
strength `extensible` (matching the base R4 `Composition.type` binding)
and add the value set definition to the IG so it is included in the IG
build (publication artefacts: ValueSet resource, expansion, and HTML
page). Where a C-CDA document type has a single fixed LOINC code (e.g.
`11488-4` for Consultation Note), retain the existing per-profile
fixed/`patternCodeableConcept` slice on `Composition.type.coding` so the
binding documents the broader allowed set while the slice pins the
specific document code.

#### Justification

The ticket identifies a real gap: `Composition.type` in C-CDA on FHIR
should be terminologically constrained to the same set of LOINC document
type codes that govern `ClinicalDocument/code` in C-CDA R2.1, and that
set is exactly OID `2.16.840.1.113883.11.20.9.31`. Binding to that value
set keeps the FHIR profile aligned with its CDA source of truth, which
is the core value proposition of the IG. Including the value set in the
build (rather than leaving it as a dangling URL) ensures implementers,
validators, and the publisher can resolve and expand it. The reporter
explicitly asked for this and Lynn Laakso re-opened the ticket
post-STU 1.2 specifically so it could be addressed in the next
publication cycle, indicating WG intent.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than binding to the legacy C-CDA OID-based value set verbatim,
publish a **new IG-owned `ValueSet`** in the C-CDA on FHIR IG (e.g.
`ValueSet/CCDAonFHIR-DocumentType`) that enumerates the LOINC document
type codes used by C-CDA R2.1 documents, and bind `Composition.type` to
that value set with strength `extensible`. The new value set's
definition would be expressed via LOINC `compose.include.concept` (or a
LOINC filter equivalent), and its narrative would cite OID
`2.16.840.1.113883.11.20.9.31` as the source. Per-profile composition
slices (Consultation Note → `11488-4`, Discharge Summary → `18842-5`,
etc.) would continue to fix the specific code via
`patternCodeableConcept`. Optionally, ship a `ConceptMap` from the new
value set to the legacy OID for traceability.

#### Justification

The OID-based canonical
`http://hl7.org/fhir/ValueSet/2.16.840.1.113883.11.20.9.31` is a
synthetic shim that re-uses the HL7 OID namespace as a FHIR canonical;
it is not maintained as a first-class FHIR `ValueSet` resource.
Publishing an IG-owned value set lets the IG (a) control publication and
versioning, (b) ensure the expansion is stable and visible in the IG
build, (c) remove the dependency on a value set that may not be present
in current terminology infrastructure, and (d) align with how other FHIR
US-realm IGs (e.g. US Core) handle LOINC-based document type bindings.
Implementation cost is modest — a single `ValueSet` resource — and the
result is a cleaner, more maintainable artefact than reaching back to
the legacy OID URI.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the underlying need is already satisfied by
the **per-profile fixed `Composition.type` coding** that each C-CDA on
FHIR composition profile applies (e.g. Consultation Note pins
`Composition.type.coding` to LOINC `11488-4`). Because each profile
already constrains `Composition.type` to the single correct LOINC code
for that document type, an additional value-set binding adds no
constraint that implementers are not already required to honour, and
introduces a maintenance burden (keeping the value set in sync with
LOINC and with C-CDA R2.1 revisions).

#### Justification

If, after review, the WG confirms that **every** C-CDA on FHIR
composition profile already pins `Composition.type` to a fixed LOINC
code via `patternCodeableConcept` or `fixedCoding`, then the value-set
binding is redundant for the conformant case: a resource that satisfies
the profile's slice is automatically a member of the proposed value set.
In that case, the ticket can be closed as "Persuasive with mod / no
change required", documenting the existing per-profile slices as the
satisfying mechanism. This also avoids importing a legacy OID-keyed
value set whose terminology server availability is uncertain.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need is real — `Composition.type` should be
terminologically scoped to the LOINC document type codes used by
C-CDA — and Lynn Laakso re-opened the ticket specifically so it could
be acted on after STU 1.2, so Disposition C (decline) does not match WG
intent. Between A and B, B better serves implementers: an IG-owned
`ValueSet` resource is published, expanded, and validated as part of
the IG build, instead of pointing at a synthetic OID-based canonical
that may not resolve on the terminology servers implementers actually
use. The per-profile fixed slices on `Composition.type.coding` are
preserved either way, so document-specific conformance does not
weaken. Disposition A remains a reasonable fallback if the WG prefers
to mirror the C-CDA OID URI verbatim for traceability with CDA R2.1.
