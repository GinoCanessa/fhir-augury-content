## Summary

The ticket is a long-standing C-CDA on FHIR profile-design request from
Corey Spears (originally filed in 2016 as GF#10315, re-opened by Lynn
Laakso in March 2024 for post-STU1.2 follow-up). The reporter argues
that the C-CDA on FHIR Composition profiles should not constrain section
codes (and other coded values defined "in the composition") by
fixing/patterning the entire `CodeableConcept`. Instead, the profiles
should fix the underlying leaf elements — `Coding.system` and
`Coding.code` (and where appropriate `Coding.version`) — on the
`coding` slice. The example artifact called out in the comments is the
Consultation Note Composition profile
(`ccda-consultation-note-composition.html`), but the request is
intended to apply consistently across the C-CDA on FHIR composition
profiles that today use `fixedCodeableConcept` / `patternCodeableConcept`
on `Composition.type` and on `Composition.section.code`.

There are no other Jira tickets cross-referenced to this issue. The
single cross-reference is an automated FHIR Bot post on the
`#committers > tracker-item` Zulip stream that simply announced the
new tracker — i.e., there is no human discussion thread on Zulip to

### Disposition A: Accept As Requested

#### Proposal

In every C-CDA on FHIR Composition profile that today fixes the section
or document type via `fixedCodeableConcept` (or `patternCodeableConcept`)
on `Composition.type` or on `Composition.section.code`, replace that
constraint with leaf-level constraints on a single named slice of
`coding`. Concretely, for each affected element:

1. Change cardinality on `coding` to `1..*` and add a slice
   discriminated by `value` on `system` (and, where useful, `code`).
2. On the named slice (e.g., `loinc`):
   - Set `coding.system` to `fixed` = `http://loinc.org`.
   - Set `coding.code` to `fixed` = the specific LOINC code that today
     appears inside the `fixedCodeableConcept`.
   - Leave `display` unconstrained (or `mustSupport`-only) so source
     systems can supply locale/version-appropriate display strings.
3. Remove the `fixedCodeableConcept` / `patternCodeableConcept` from
   the parent `CodeableConcept`.
4. Apply the same transformation to any other coded leaf in the
   Composition profile family that follows the same pattern (e.g., any
   header-level coded fields the IG fixes today).

The Consultation Note Composition profile referenced in the 2024
comment (`ccda-consultation-note-composition.html`) is the canonical
worked example.

#### Justification

Leaf-level binding is the more permissive and forward-compatible
pattern in FHIR profiling: it forces the system+code that the
specification cares about while allowing additional translations,
display variants, and `userSelected` to flow through unmolested. A
fixed `CodeableConcept` requires the *entire* JSON/XML object — every
property of the contained `Coding` plus any `text` — to match exactly,
which historically has caused validator strictness problems and
interop friction with source systems that legitimately add additional
coding translations. This is the same modelling rationale used in
US Core and IPS for terminology constraints and is consistent with the
guidance the FHIR core team has given on `fixed[x]` vs. `pattern[x]`
for `CodeableConcept`. The change is additive in semantics (any
instance valid under the old fixed pattern will remain valid under the
new sliced pattern), so existing conformant data is preserved.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of changing the `fixedCodeableConcept` to a fully sliced
`coding` constraint on every element, switch from `fixedCodeableConcept`
to **`patternCodeableConcept`** containing a single `coding` whose
`system` and `code` are populated. This keeps the constraint at the
parent `CodeableConcept` level but only requires that *one* of the
codings match the pattern — additional translations and a `text` value
are then permitted. Document this convention once in the IG narrative
(e.g., a "Coded Element Conventions" section) and apply it uniformly
across the Composition profiles, rather than introducing slices into
every coded leaf.

#### Justification

`patternCodeableConcept` was introduced precisely to address the
"fixed is too strict" problem the reporter is describing, and it does
so without the structural overhead of declaring slices, discriminators,
and 1..* cardinality on `coding` for every coded element. The result
is materially smaller diffs, simpler StructureDefinitions, simpler
generated snapshots, and IG pages that remain easy to read. For
implementers who only need to assert "this composition section is
LOINC X", pattern-on-CodeableConcept is functionally equivalent to the
sliced approach and avoids the readability cost of slicing every
section. Trade-off vs. Disposition A: it does *not* let the IG
constrain anything more than presence — it cannot, for example, also
fix `version` or attach a `mustSupport` flag specifically to the LOINC
coding — so if Structured Documents anticipates needing per-coding
constraints later, A is the more future-proof choice.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Persuasive with Modification — No Change" (or "Not
Persuasive" if the workgroup disagrees with the premise), on the
grounds that (a) `fixedCodeableConcept` is a valid FHIR profiling
mechanism and the IG's choice to use it is consistent with how many
HL7 IGs constrain Composition section codes, (b) the ticket has been
open since 2016 with no implementer feedback or vote pile-on, (c) the
single 2024 re-open comment provides no new use cases or evidence of
field problems, and (d) re-cutting every Composition profile risks
breaking downstream profiles, examples, and conformance test suites
that already depend on the current shape. Recommend the workgroup
revisit only if a specific implementer reports a concrete validation
or interop failure traceable to the current pattern.

#### Justification

Declining is defensible because the reporter's complaint is a style
preference rather than a bug: a `fixedCodeableConcept` constraint
*works*, and instances that match it are correct C-CDA on FHIR
documents. The ticket carries no Zulip discussion, no linked
implementer issues, and no GitHub PRs — i.e., there is no demonstrated
demand. Re-profiling every Composition section to slice `coding` is a
non-trivial editorial change that will churn the IG, perturb examples,
and require regression-testing every published instance, all without
delivering a user-facing capability. If the workgroup prefers to defer
work on a 2016-era ticket with no community engagement, declining (and
inviting a fresh ticket if a real problem appears) is a coherent
position.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — switch
`fixedCodeableConcept` to `patternCodeableConcept`)

Disposition B captures everything the reporter actually needs (the IG
no longer requires a byte-exact `CodeableConcept` match; implementers
can supply additional translations, `text`, and `userSelected`) while
keeping the editorial cost of the change small and the resulting
profiles readable. Disposition A is technically the cleanest model and
would be the right answer if Structured Documents anticipates needing
to constrain additional `Coding` properties (version, mustSupport per
slice) — but absent that signal, the slicing overhead in every
Composition profile is hard to justify for a 2016 ticket with no
implementer pile-on. Disposition C is intellectually consistent but
ignores the legitimate, well-known usability problem with
`fixedCodeableConcept` that the FHIR community has repeatedly flagged;
since the fix is cheap (a mechanical search-and-replace from
`fixed` → `pattern` in the affected profiles, with regenerated
snapshots and example revalidation), declining outright leaves a
known-rough edge in the IG. The workgroup should adopt B and, where a
specific section is known to need richer constraints, escalate that
section individually to A.
