## Summary

The reporter (Corey Spears) asks the C-CDA on FHIR work group to stop using
`fixedCodeableConcept` to lock down code-bearing elements in the C-CDA on FHIR
Composition profiles. The objection is that `fixedValue[x]` on a
`CodeableConcept` requires implementers to send the *exact same*
`CodeableConcept` shape — which forecloses on supplying an optional
`display`, an additional `coding` translation, or `text`. The request is to
instead pin the value at the leaf `Coding` level (e.g., `coding.system` and
`coding.code` fixed via slicing or `pattern`/`fixed` on the leaf) so that
display strings, translations, and text remain available to authors. The
reporter notes this same pattern repeats across **all** C-CDA on FHIR
Composition profiles **except Transfer Summary**, implying a sweep across the
profile set rather than a one-off fix. Lynn Laakso has marked it for re-open
post the STU 1.2 publication and supplied the original related URL
(`http://build.fhir.org/ccda/ccda-referral-note-composition.html`). There are
no explicitly linked Jira tickets and no substantive Zulip discussion — the
only cross-reference is an automated FHIR Bot post that mirrored the original
GForge tracker item into the `#committers > tracker-item` stream in 2016.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Sweep the C-CDA on FHIR Composition profiles — explicitly **all Composition
StructureDefinitions except Transfer Summary** (e.g., Referral Note,
Discharge Summary, Consultation Note, History and Physical, Progress Note,
Operative Note, Procedure Note, Care Plan, Continuity of Care Document,
etc.) — and replace every leaf-on-CodeableConcept lock-down currently
expressed as `fixedCodeableConcept` (or `patternCodeableConcept` collapsed
to a single coding) on `Composition.type`, `Composition.category`, and
`Composition.section.code` with a leaf-level constraint:

- Slice `coding` on `system`, require a slice with
  `fixed` / `pattern` for `coding.system` (e.g., `http://loinc.org`) and
  `coding.code` (e.g., the section's LOINC document code), `min = 1`, but
  do **not** fix `coding.display`, do **not** forbid additional `coding`
  slices (translations), and do **not** forbid `CodeableConcept.text`.
- Equivalently, replace `fixedCodeableConcept` with `patternCodeableConcept`
  containing only the required `coding.system` + `coding.code`, **and**
  re-slice `coding` to be open so additional translations remain valid.

Align Transfer Summary with the same pattern for consistency once the
sweep is done (the ticket's exclusion of Transfer Summary appears to be
a description of current state, not a design preference).

#### Justification

`fixedCodeableConcept` requires the instance's serialized
`CodeableConcept` to match exactly — implementers cannot add an
optional `coding.display`, cannot supply translation `coding`s, and
cannot include `CodeableConcept.text`. That conflicts with FHIR's own
guidance that `display` is informative and that `CodeableConcept` is
designed to carry multiple `coding`s plus narrative text. Pinning the
constraint at `coding.system` + `coding.code` (the identity of the
concept) preserves the IG's intent (the section *is* this LOINC code)
while honoring `CodeableConcept`'s extensibility contract. The change is
also consistent with how most other HL7 IGs (US Core, IPS, QI-Core)
constrain coded elements. The reporter confirms the issue is
profile-wide, so a single coordinated sweep is more maintainable than
per-section patches.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than sweeping every Composition profile by hand, introduce (or
reuse) a single **abstract parent Composition profile** for C-CDA on
FHIR that establishes the leaf-level slicing pattern on
`Composition.type` and `Composition.section.code`
(`coding` sliced by `system`, `min=1` LOINC slice with fixed
`coding.system` only; per-section profiles add a `pattern` for
`coding.code`). All section-specific Composition profiles then derive
from this parent and only pin their own `coding.code`. Keep the
existing `fixedCodeableConcept` definitions only where the IG genuinely
requires a single, fully-specified concept with no display variation
(and document why).

#### Justification

A single derivation point makes the constraint machinery uniform across
the IG, reduces duplication, and prevents drift the next time an
implementer flags the same problem on a newly-added section. It also
keeps the per-section profiles small and reviewable. The trade-off
versus Disposition A is that callers walking the differential view need
to climb one more derivation step to see the slice definition, and
existing implementers' validators must be re-tested — but the resulting
profile set is materially easier to maintain and audit. This pattern is
also closer to how IPS Composition is structured.

---

### Disposition C: Decline

#### Proposal

Decline the change and document in the IG that `fixedCodeableConcept`
is intentional: the C-CDA on FHIR Composition section codes are
identity tokens for the section, not human-presentation strings, so
implementers should populate `display` / translations / text on
**other** coded elements (e.g., entries within the section), not on
`Composition.section.code` itself. Add a narrative paragraph to the
IG's "Conformance" page explaining why and pointing implementers to
slicing-based subclassing if they need to carry additional coding
metadata locally.

#### Justification

This is defensible only if the work group concludes that
`Composition.section.code` (and the parallel `Composition.type`) are
purely machine-routing identifiers in C-CDA on FHIR's design and that
the loss of `display`/`text` is by intent. However, this rationale is
weak because (a) FHIR's `CodeableConcept` semantics explicitly support
`display` and `text` even on identity-bearing codes, (b) implementers
have already raised this as a real authoring obstacle, and (c) the IG
itself excludes Transfer Summary from the same pattern, which suggests
the current design is not a deliberate, IG-wide stance. Decline should
only be considered if the work group does not have bandwidth to reissue
the profile set.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures everything Disposition A delivers (implementers
regain `display`, translations, and `text` without losing the IG's intent
that the section *is* a specific LOINC code) while collapsing the change
to a single inheritance point. Given the reporter's own observation that
the issue spans every Composition profile except one, a one-shot abstract
parent will be cheaper to implement and maintain than touching each
StructureDefinition independently, and it mirrors how IPS and US Core
manage analogous constraints. Disposition A is acceptable as a fallback
if the work group prefers minimal architectural change; Disposition C is
not recommended because the underlying complaint is technically correct
and the IG's own exclusion of Transfer Summary undermines a "fixed by
design" defense. Either A or B should also tidy up Transfer Summary so
the IG ends up internally consistent.
