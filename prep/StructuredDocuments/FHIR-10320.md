## Summary

The ticket reports that, in the US C-CDA on FHIR Composition profiles, the
slices on `Composition.section.entry` that are intended to constrain entries
to specific U.S. DAF (Data Access Framework) profiles only document the
target profile in human-readable `definition` text — they do not declare it
machine-readably via a `type.profile` reference. As a result, an instance
validator cannot enforce that an entry referenced from the section actually
conforms to the named DAF profile (e.g., `daf-allergyintolerance`). The
reporter requests that the affected slices add the appropriate
`<type><code value="Reference"/><profile value="…"/></type>` declaration so
that the slice is a proper, validatable profile reference rather than only
prose. The issue is called out across the Allergies and Intolerances,
Medications, Problems, Results, Family History, Immunizations, Vital Signs,
and Procedures sections.

There are no cross-referenced Jira tickets to incorporate. The single
related Zulip cross-reference is an automated FHIR Bot tracker
notification carrying the original GForge item (GF#10320) and contains no
discussion content.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

For each of the affected section profiles in the US C-CDA on FHIR IG
(Allergies and Intolerances, Medications, Problems, Results, Family History,
Immunizations, Vital Signs, and Procedures), update every
`Composition.section.entry` slice that today only names a DAF profile in
its `definition` text. For each such slice, add a `type` element of the
form:

```xml
<type>
  <code value="Reference"/>
  <profile value="http://hl7.org/fhir/StructureDefinition/daf-{resource}"/>
</type>
```

so that the slice machine-readably declares which profile the referenced
resource must conform to. Because the work is mechanical and applies the
same shape across all eight sections, do it as a single sweep across the
section Composition profiles and validate that the slice discriminator
(typically `type = profile` on `resolve()`) still resolves correctly after
the change. Note that the URLs as written in the original ticket point at
the original DAF profiles (`http://hl7.org/fhir/StructureDefinition/daf-*`);
before merging, confirm whether the current C-CDA on FHIR IG should bind
to the historic DAF URLs or to their successor US Core / DAF-Core
canonicals — the structural change is the same either way, only the
canonical URL value differs.

#### Justification

This is exactly the conformance gap the reporter raised: prose in a
`definition` cannot be validated, so today an instance can put any
`Reference(Resource)` into the entry slot and pass schema/profile
validation as long as it satisfies the base `Composition.section.entry`
constraint. Adding `type.profile` makes the binding normative and
checkable by any FHIR validator, which is the standard FHIR pattern for
constraining slice targets and matches how other US Realm IGs (US Core,
QI-Core) constrain their Composition/Bundle entry slices. The change is
additive at the structural level (introduces a constraint that was
already promised in prose) and should not invalidate any instance that
was actually meeting the documented intent.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than binding each entry slice to the historic DAF profile URLs,
re-target the slices to the current US-Realm successor profiles (US Core
for AllergyIntolerance, MedicationStatement / MedicationRequest,
Condition, Observation Lab / Vital Signs, Immunization, Procedure, and
FamilyMemberHistory where applicable), and add the same
`type.profile` declaration the reporter asks for, but pointing at those
URLs. Where a single C-CDA section can carry multiple kinds of entries
(e.g., Medications can be MedicationStatement *or* MedicationRequest),
encode that as multiple slices on `Composition.section.entry`, each with
its own `type.profile`, using a `type = profile` discriminator on
`resolve()`. Update the slice `definition` text to match the new target
canonical and remove the now-obsolete CONF:#### identifiers if Structured
Documents agrees they no longer carry meaning in the FHIR IG.

#### Justification

The DAF profiles named in the original 2016 ticket are effectively
deprecated; the US Realm has consolidated on US Core, and most current
C-CDA on FHIR consumers expect US Core resources. Accepting the request
literally (Disposition A) re-introduces a binding to a profile family
that is no longer maintained, which would create a follow-up cleanup
ticket almost immediately. Doing the structural fix and the canonical
update in one pass yields a single coherent change, removes a known
source of implementer confusion, and aligns the IG with the rest of the
US Realm. The trade-off is broader scope and the need to confirm the
correct US Core target for each section (and to handle multi-target
sections via multiple slices), which makes this larger than the
mechanical change in A.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — Out of Scope for current cycle**,
on the grounds that (a) the reporter is calling out a specific authoring
choice in the StructureDefinition that the work group made deliberately
when the IG was written against DAF, (b) the issue has sat in Triaged
state since 2016 with no implementer follow-up other than the 2024
re-open note, and (c) the affected sections will need wholesale re-work
when C-CDA on FHIR is realigned with US Core, at which point the entry
slice profiles will be revisited as part of that broader effort.
Recommend the reporter raise a new ticket against the next major
revision of C-CDA on FHIR rather than patching the published STU.

#### Justification

A defensible reason to decline is scope: the published artifact is
stable, no implementer has reported breakage in eight years, and any fix
that just re-asserts a binding to deprecated DAF URLs (Disposition A)
buys little real validation value because few current systems claim
conformance to those DAF profiles. Doing the more useful US-Core-aligned
fix (Disposition B) is arguably a normative content change that belongs
to a larger republication, not a maintenance ticket. Declining keeps the
STU stable and defers the structural work to the next planned revision.

---

### Recommendation

**Recommended disposition:** B

The underlying defect the reporter identified is real and correctly
diagnosed: a slice that only names its target profile in `definition`
text is not a profile constraint that any FHIR validator can enforce, so
the IG silently fails to deliver the conformance it promises in prose.
Disposition C leaves that gap in place indefinitely, which is hard to
defend now that the ticket has been explicitly re-opened post-STU1.2.
Disposition A fixes the structural gap but binds the IG to DAF URLs
that the US Realm has effectively retired, which almost guarantees a
second pass in the next cycle. Disposition B does the same structural
fix the reporter asks for and updates the canonical targets to the
profiles that current implementers actually use, so the IG ends up
internally consistent and validator-checkable in one change. The added
scope (confirming the right US Core / DAF-Core target per section, and
splitting Medications-style sections into multiple slices) is the work
that needs to happen anyway before this constraint is meaningful, so
folding it into this ticket is the most efficient path. Recommend
Structured Documents adopt B, with confirmation from the C-CDA on FHIR
editors on the per-section target canonicals before the slice edits are
made.
