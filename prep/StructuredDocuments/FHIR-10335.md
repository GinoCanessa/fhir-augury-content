## Summary

The ticket reports that the C-CDA on FHIR Implementation Guide's
Consultation Note Composition profile does not carry over a constraint
that C-CDA R2.1 imposes on the underlying CDA template: every
Consultation Note must reference at least one prior order that the note
fulfills (C-CDA R2.1 §1.1.4.2 `inFulfillmentOf`, CONF:1198-8382,
1198-29923, 1198-29924). In CDA, the constraint is `inFulfillmentOf`
1..*, each containing exactly one `order` with at least one `id`. The
reporter (Corey Spears) is asking the C-CDA on FHIR guide to add an
equivalent FHIR-side constraint on the Consultation Note Composition
profile so that round-trippability with C-CDA R2.1 is preserved and the
"complex" template-level constraints are not silently dropped.

The single Zulip cross-reference (`committers > tracker-item`,
[message 153838154](https://chat.fhir.org/#narrow/stream/committers/topic/tracker-item/near/153838154))
is just the automated FHIR Bot announcement of the tracker item being
filed in 2016 — there is no substantive community discussion attached.
Lynn Laakso's 2024 comments simply re-open the issue post-STU1.2
publication and preserve the original related URL
(`http://build.fhir.org/ccda/ccda-consultation-note-composition.html`),
implying the SD work group still considers it an open gap to address in
the next C-CDA on FHIR cycle.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the C-CDA on FHIR Consultation Note Composition profile
(`ccda-consultation-note-composition`) to add a SHALL-level constraint
that mirrors C-CDA R2.1 §1.1.4.2 `inFulfillmentOf`. Concretely:

- Constrain `Composition` (Consultation Note profile) so that the FHIR
  representation of `inFulfillmentOf` — i.e., a reference to one or
  more `ServiceRequest` resources that the note fulfills — is required
  with cardinality 1..*. In current C-CDA-on-FHIR practice this maps
  cleanly to constraining `Composition.event.detail` (slice
  `inFulfillmentOf`) to `1..*`, each `Reference(ServiceRequest)`,
  encoding CONF:1198-8382.
- Slice each fulfilled-order reference to require exactly one referenced
  `ServiceRequest` (CONF:1198-29923) and require that the referenced
  `ServiceRequest` carry at least one `ServiceRequest.identifier`
  (CONF:1198-29924). Add an invariant on the Consultation Note profile
  asserting `event.detail.resolve().identifier.exists()` for the
  inFulfillmentOf slice.
- Cross-link the Patient Referral Act guidance to the same
  `ServiceRequest.identifier` so the "same id as the referral" note in
  C-CDA carries through.
- Update narrative + examples in the Consultation Note Composition
  page, and add at least one example bundle that includes a fulfilled
  `ServiceRequest`.

#### Justification

Mirrors C-CDA R2.1 SHALL-level conformance verbatim, which is the
stated design goal of the C-CDA on FHIR IG: a Consultation Note
expressed in FHIR should not silently relax constraints that are
mandatory in CDA. Round-trip between CDA and FHIR Consultation Notes
remains lossless for downstream consumers (HIEs, payers) that today
rely on `inFulfillmentOf/order/id` being present. It also closes the
explicit gap identified by Corey Spears and re-affirmed by Lynn Laakso
when re-opening the ticket post-STU1.2.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than tightening the base Consultation Note Composition profile
to 1..*, keep `Composition.event.detail` (inFulfillmentOf slice) at
0..* on the base Consultation Note profile and instead publish a
**conformance variant** — a derived profile (e.g.,
`ConsultationNote-fromCCDA`) — that adds the SHALL-level
`inFulfillmentOf` 1..*, the single-`ServiceRequest` constraint, and
the `ServiceRequest.identifier` 1..* invariant. The base profile would
add must-support on the slice and narrative guidance pointing
implementers to the derived profile when they are converting from or
to C-CDA R2.1 documents.

#### Justification

The original 2016 ticket scope is "complex constraints have not been
addressed", but C-CDA on FHIR is also used in greenfield FHIR-native
flows where a Consultation Note is authored without a pre-existing
order (e.g., walk-in tele-consultations recorded directly in a FHIR
EHR). Forcing 1..* on the base profile would either block those use
cases or push implementers to invent placeholder `ServiceRequest`
resources, both of which are worse than the current under-specification.
A derived "round-trip with C-CDA R2.1" profile preserves the strict
constraint exactly where it is needed (CDA↔FHIR conversion) without
breaking FHIR-native authors. This pattern is already used elsewhere
in the IG to layer CDA-strict constraints on top of FHIR-flexible
bases.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive — no change needed**, on the basis
that (a) the C-CDA on FHIR IG is not normatively required to carry
every CONF rule from C-CDA R2.1 forward, (b) the existing
`Composition.event` slot in the FHIR Composition resource already
allows referencing fulfilled `ServiceRequest` resources for
implementers who need it, and (c) the IG's general conformance
statement already says implementers converting from C-CDA R2.1
documents must preserve mandatory CDA constraints on the FHIR side.
Add a single sentence of narrative to the Consultation Note
Composition page pointing implementers at C-CDA R2.1 §1.1.4.2 and the
`Composition.event.detail` slot.

#### Justification

The ticket has sat triaged since 2016 with no recorded community
demand beyond the original report and a 2024 procedural re-open. There
is no Zulip discussion, no linked GitHub PR or issue, and no other
Jira tickets cross-referenced — i.e., no evidence of implementer pain
or HIE round-trip failure attributable to this gap. Adding a SHALL
1..* constraint at this stage risks invalidating existing C-CDA on
FHIR Consultation Note instances in production, which is a real
breaking change for an IG already past STU1.2. Documenting the
mapping is a low-risk way to satisfy the spirit of the request without
forcing a normative tightening.

---

### Recommendation

**Recommended disposition:** B

C-CDA on FHIR's value proposition is faithful round-trip with C-CDA
R2.1, and silently dropping a SHALL-level constraint as visible as
`inFulfillmentOf` undermines that promise — so straight Decline (C) is
too weak, especially since Lynn Laakso explicitly re-opened the ticket
in 2024 to keep it on the post-STU1.2 work list. At the same time,
the eight-year absence of any community escalation, plus the
non-trivial risk that 1..* on the base profile breaks FHIR-native
Consultation Note authors who never had a prior order, argues against
a hard Accept-As-Requested (A). A derived "CDA-strict" profile (B)
restores the C-CDA SHALL exactly where conversion correctness matters,
keeps the base profile usable for greenfield flows, and gives the
work group a concrete artifact to validate against in the next
C-CDA-on-FHIR ballot cycle. The work group should confirm the slice
mapping (`Composition.event.detail` vs. `Composition.basedOn`) early
in the discussion, since that choice drives everything downstream.
