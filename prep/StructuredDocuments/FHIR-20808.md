## Summary

The reporter notes that in the published XML and JSON serializations of
the example `Questionnaire` instances in the US Healthcare Associated
Infections (HAI) IG, items whose `type` is `group` consistently lack a
`required` element. Because `required` is optional on
`Questionnaire.item`, an absent value collapses to `required=false`, so
there is currently no way for a downstream reader of the HAI example
Questionnaires to tell which groups are intended to be mandatory. The
reporter notes this affects "some of the other questionnaires" in the
IG as well.

This ticket sits alongside two other Craig Newman reports filed the same
day against the same HAI Questionnaire material:
[FHIR-20809](https://jira.hl7.org/browse/FHIR-20809) flags inconsistencies
between the HTML and XML representations of a Questionnaire instance —
explicitly including a case where the `Required` value for
`risk-factor-wound-class` differs between the XML and the human-readable
content tab — and
[FHIR-20810](https://jira.hl7.org/browse/FHIR-20810) flags table errors
in the rendered Questionnaire instance pages (a `g` instead of `group`
type, a duplicated CPT row, a misleading `time` vs. `quantity` type,
and a missing `NHSN comment`). Together the three tickets describe a
quality-control gap in the HAI IG's published Questionnaire examples
rather than a defect in the base FHIR `Questionnaire` resource.

## Related Zulip Discussions

The cross-reference index returned no Zulip threads explicitly tagged
with FHIR-20808. The two threads below were surfaced by content search
on "Questionnaire group required item" and provide directly applicable
authoritative interpretation of `Questionnaire.item.required` on group
items.

### [questionnaire > Meaning of item.required on questions and groups](https://chat.fhir.org/#narrow/stream/questionnaire/topic/Meaning%20of%20item.required%20on%20questions%20and%20groups)

Short thread (Nov 2019) in which Miriam quotes the
`Questionnaire.item.required` definition and Lloyd McKenzie clarifies
that `required` is meaningful on group-typed items: when `required=true`
on a group, the group must be present in a completed
`QuestionnaireResponse` and at least one of its child questions must
have an answer. Confirms that `required` is a real authoring decision
that must be made per group, not an element that can simply be omitted.

Participants: Miriam, Lloyd McKenzie.

### [questionnaire > item.required](https://chat.fhir.org/#narrow/stream/questionnaire/topic/item.required)

Long-running thread (Apr 2020 – May 2021) between Paul Lynch, Lloyd
McKenzie, Yunwei Wang, Jing Tang, and Brian Postlethwaite. Establishes
the consensus interpretation that (a) when `required` is absent the
meaning is the same as `required=false`; (b) `required` is always
evaluated in the scope of the parent — a required child does not
implicitly make its parent group required; and (c) a required group
means the group must appear in the response and at least one child
question must be answered. The thread spawned tracker items FHIR-32151
and FHIR-32665 to clarify the spec language. The discussion directly
supports FHIR-20808's premise: because absent ≡ false, the HAI examples
that omit `required` on a group are silently asserting that no groups
are mandatory, which contradicts the IG's narrative tables.

Participants: Paul Lynch, Lloyd McKenzie, Yunwei Wang, Jing Tang, Brian
Postlethwaite.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the affected example `Questionnaire` instance(s) under the HAI
IG so that every `Questionnaire.item` with `type = "group"` carries an
explicit `required` element. For each group, set `required` to the
value implied by the IG's authoring intent (cross-checked against the
narrative tables called out by FHIR-20809) — typically `required=false`
for optional groupings of related questions, and `required=true` for
groups whose presence is mandatory in a completed QuestionnaireResponse.
Apply the same pass to the other HAI Questionnaire example instances
the reporter notes are "similarly affected" (the
`hai-questionnaire-opc-ssi-event` family and its siblings cited in
FHIR-20809). Re-render the HTML/JSON/XML so the published serializations
agree with the narrative tables.

#### Justification

The Zulip "item.required" thread establishes that an absent `required`
on a `Questionnaire.item` collapses to `required=false`, so the current
HAI examples are not silent on the question — they are actively
asserting that no group is mandatory, which contradicts the IG's own
content tables (FHIR-20809 documents at least one such mismatch). Making
`required` explicit on every group is consistent with how the
Questionnaire community interprets the element and removes the
ambiguity for downstream implementers reading the example payloads. The
change is scoped to authored example resources in a single IG and does
not touch the base `Questionnaire` resource definition.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) hand-editing each example, address the
root cause: add a publication-level QA step for the HAI IG that
(1) lints every example `Questionnaire` to require an explicit
`required` flag on each group item, and (2) asserts that the rendered
narrative tables and the source XML/JSON agree on the `required` value
for every item. Bundle the resulting fixes for FHIR-20808, FHIR-20809
and FHIR-20810 into a single HAI Questionnaire-examples cleanup ballot
item, since all three tickets describe the same underlying class of
defect (out-of-sync example resources and rendered tables).

#### Justification

FHIR-20808, FHIR-20809 and FHIR-20810 were filed by the same reporter
on the same day against the same artifacts and all describe authoring
inconsistencies in the HAI Questionnaire examples. Fixing them
one-by-one is brittle: a future regeneration of the IG could easily
reintroduce the same omissions because nothing in the build catches
them. A linter / publication check makes the fix durable and avoids a
piecemeal disposition that closes FHIR-20808 while leaving the
demonstrably related FHIR-20809 inconsistency open. The trade-off is a
larger up-front change and a small ongoing build-time cost, in exchange
for protecting future revisions of the IG from the same defect class.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — works as designed". Document in the
resolution that `Questionnaire.item.required` is `0..1` with a
documented default of `false`, that absence is therefore semantically
equivalent to `required=false`, and that the HAI example
Questionnaires are technically valid as published. Point readers at the
`Questionnaire.item.required` definition for the rule.

#### Justification

`Questionnaire.item.required` is optional in the base resource and the
spec is explicit that an absent value behaves as `false`, so the
example instances are not invalid FHIR. A reader who applies the spec
literally will get a deterministic answer for every group. Declining
keeps the IG editors from having to make an explicit `true`/`false`
authoring decision for every group on legacy 2019 example content that
may no longer reflect current intent.

---

### Recommendation

**Recommended disposition:** B

Disposition C is hard to defend on the merits. FHIR-20809 — filed by
the same reporter on the same day — already documents at least one
case where the `Required` value in the HAI narrative table disagrees
with the value in the XML, so the examples are not merely "silent on a
default", they are internally inconsistent. Disposition A correctly
fixes FHIR-20808 in isolation but leaves the related FHIR-20809 /
FHIR-20810 defects to be reasoned about and fixed independently, with
no protection against the same class of defect re-emerging the next
time the IG is regenerated. Disposition B treats the three tickets as
the single underlying problem they really are (out-of-sync HAI
Questionnaire examples + rendered tables) and adds a lightweight build
check so the fix sticks. The Zulip community discussion supports
making `required` an explicit, deliberate authoring decision on every
group item, which is exactly what the linter step in B enforces.
