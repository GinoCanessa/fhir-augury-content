## Summary

The reporter flags an editorial defect in the Composition / documents
narrative: prose currently refers to `Composition.section.content`, but
the element on `Composition.section` that holds the references the
attester is attesting is named `entry`, not `content`. The ticket asks
that all such occurrences of `Composition.section.content` be replaced
with `Composition.section.entry`. The reporter quotes a paragraph from
the documents page describing what attesters attest ("...the resources
referred to in the **Composition.section.content** references...") to
illustrate the wrong element name in the wild. This is filed as a
Technical Correction (Highest priority) against FHIR Core, with no
linked or cross-referenced Jira tickets, no related GitHub items, and
no substantive Zulip discussion.

## Details

**Description:**

> - The attester attests contents of the document resource, the subject
>   resource and the resources referred to in the
>   **Composition.section.content** references. Because documents are
>   often derived. Compositions and the attestation from the composition
>   is held to apply to the document, the method for [presenting a
>   document](https://build.fhir.org/documents.html#presentation) should
>   be used when/if attesters review
> - the content of the composition.

Composition.section.content should be Composition.section.entry

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the Composition / documents narrative on `documents.html` (the
"Presentation of a document" section that defines what an attester
attests) and any other prose in the FHIR Core spec that refers to
`Composition.section.content`, replacing each occurrence with
`Composition.section.entry`. Concretely:

- On `documents.html#presentation`, change "the resources referred to in
  the **Composition.section.content** references" to "the resources
  referred to in the **Composition.section.entry** references".
- Grep the FHIR Core sources for any other occurrence of
  `Composition.section.content` (e.g., on `composition.html`,
  `composition-definitions.html`, attestation-related notes,
  documents.html, security/provenance discussion) and apply the same
  substitution where the intent is clearly the references on a
  `Composition.section`.

This is a documentation / narrative change only. The
`Composition.section.entry` element is unchanged; no StructureDefinition,
schema, ValueSet, example, or invariant is touched.

#### Justification

The element on `Composition.section` that carries the `Reference(Any)`
list of resources making up the section's content is named `entry` in
the FHIR Core StructureDefinition. There is no `Composition.section.content`
element — that path simply does not exist in the resource. Prose that
uses that path is therefore incorrect and misleading to implementers,
who may legitimately wonder whether they are looking at an outdated
draft or a different element. The attestation paragraph the reporter
cites is in fact describing the references in `Composition.section.entry`,
so the substitution preserves the original meaning while making it
accurate. As a Technical Correction with no behavioural impact, this
fits the standard editorial-fix pathway and carries essentially no risk
to implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying defect, but generalise the fix: rather than a
literal find-and-replace of `Composition.section.content` →
`Composition.section.entry`, rewrite the affected sentences so the
attestation prose talks about "the resources **referenced from each
section** (`Composition.section.entry`)" and links the element name to
the formal `Composition.section.entry` definition page. While doing so,
audit nearby prose for the same class of error (e.g., places that use
the informal word "content" where they really mean "the entries", or
that conflate `Composition.section.text` (the narrative for the
section) with the entry references) and clarify each.

#### Justification

A pure string substitution fixes the reporter's example but leaves the
door open to the same confusion recurring, because the word "content"
is used informally throughout the documents narrative to mean "what is
in the document". Rewriting the sentences to (a) link the element name
to its definition, and (b) explicitly distinguish `section.text`
(narrative) from `section.entry` (referenced resources) gives readers a
durable mental model and reduces the chance that a future editor types
`section.content` again. The cost is a slightly bigger PR and a more
careful review, but the change is still narrative-only and still ships
as a Technical Correction.

---

### Disposition C: Decline

#### Proposal

Close the ticket as Not Persuasive on the grounds that
`Composition.section.content` is recognisable shorthand for "the
content (entries) of a Composition section", and that the surrounding
context already makes the intended meaning clear; no spec change is
required.

#### Justification

This is the defensible "do nothing" position if one believes the prose
is colloquial rather than referring to a formal element path. It is the
weakest of the three options because the FHIR specification is the
authoritative source for FHIR element paths, and using a non-existent
path in normative-adjacent prose undermines that authority. Decline is
listed for completeness so the workgroup can explicitly reject it.

---

### Recommendation

**Recommended disposition:** B

Disposition A is correct and low-risk and would close the ticket
exactly as filed; if the workgroup wants the smallest possible change,
A is fine. Disposition B is recommended because it costs only marginally
more effort, fixes the same defect, and additionally hardens the
attestation/presentation prose against the recurring confusion between
`Composition.section.text` (narrative) and `Composition.section.entry`
(the referenced resources the attester is actually attesting). Given
this is a Highest-priority Technical Correction with no behavioural
risk, and given the editor will already be in the same paragraph,
spending the extra paragraphs to clarify the model is the more
durable fix. Disposition C should be rejected: there is no
`Composition.section.content` element in FHIR, so prose that names it is
factually wrong, not just informal.
