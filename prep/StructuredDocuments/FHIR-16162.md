## Summary

The ticket asks the IPS (International Patient Summary) editors to expand the
set of example instances shipped with the FHIR IPS Implementation Guide.
Specifically, the reporter (Giorgio Cangioli) suggests adding examples that
illustrate edge / boundary conditions of an IPS document beyond the existing
"Bundle example": at minimum a *minimal* IPS, a *no-information* IPS (for
patients about whom essentially no clinical data is available), and a
*healthy patient* IPS (a patient with no significant findings). The purpose
is to give implementers concrete reference instances for the common "thin"
cases that the current example set does not cover well.

The ticket was originally dispositioned in 2018 as **Considered for Future
Use** (Hausam / Dolin vote 16-0-3) and was deferred. In November 2022,
Lynn Laakso re-opened it post-STU update so that it can be re-considered
against the current published IPS. There are no linked Jira tickets and no
GitHub PRs/issues recorded against it; the only cross-reference is an
automated FHIR Bot post in Zulip announcing the original tracker item, which
does not contain any substantive discussion. Several keyword-related IPS
tickets (notably [FHIR-39272](https://jira.hl7.org/browse/FHIR-39272),
[FHIR-25004](https://jira.hl7.org/browse/FHIR-25004), and
[FHIR-16161](https://jira.hl7.org/browse/FHIR-16161)) have already been
resolved Persuasive and indicate the WG's recent practice of expanding /
correcting IPS examples — they provide a useful template for how this
ticket could be handled.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Author and ship three new IPS example instances in the IPS IG and reference
them from the IG's Examples / Artifacts pages:

1. **`Bundle-IPS-no-information`** — an IPS Bundle for a patient about
   whom no clinical data is recorded. The Composition's required
   sections (Allergies & Intolerances, Medication Summary, Problem List)
   each carry a single entry with `dataAbsentReason = unknown` (or use
   the IPS-published `absentOrUnknown*` value-set entries where
   profiled), with a narrative `text` element explaining "no information
   available". Conforms to the IPS Bundle and IPS Composition profiles
   and uses `exampleCanonical` to point at the IPS Bundle profile (per
   the convention set by FHIR-39272).
2. **`Bundle-IPS-healthy-patient`** — an IPS Bundle for a patient with
   no significant clinical findings: required sections present with
   either `emptyReason` (Composition.section.emptyReason) on each
   section or single "no known"-style entries (e.g. AllergyIntolerance
   with `code = no-known-allergies`, Condition with `code = no-known-
   problems`, MedicationStatement with `code = no-known-medications`
   from the IPS-recommended value sets).
3. **`Bundle-IPS-minimal`** — a stripped-down but still
   profile-conformant IPS Bundle illustrating only the absolutely
   required sections / mustSupport elements, intended as a
   "starter template" for implementers.

For each example: add the resource under the IPS IG's authoring root
(typically the `input/examples/` and `input/resources/` directories of
the HL7 IPS IG project), register the example in the IG's `ImplementationGuide`
resource (`definition.resource[]` with `exampleCanonical` pointing at
the appropriate IPS profile), and add a short prose entry to the
Examples page explaining the intent of each example. Also re-confirm the
`fullUrl` discipline established by FHIR-16161 (absolute URLs or
`urn:uuid:` form) and the `exampleCanonical` discipline established by
FHIR-39272.

#### Justification

This is precisely what the reporter asked for, has clear implementer
value (the three named cases — minimal, no-info, healthy — are the
classic edge cases new IPS implementers stumble over), and is fully
consistent with how the WG has recently dispositioned similar
example-curation tickets (FHIR-16161, FHIR-39272, FHIR-25004 all
**Persuasive / Published**). The cost is bounded — three new example
instances and a small IG manifest change — and the same authoring root
that hosts the existing "IPS Bundle example - minimal" can host the new
ones, so the change is low risk to the published IPS profiles.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) shipping three hand-crafted example
Bundles, add a single new **`no-information`** IPS Bundle as the only
new example, and meet the rest of the underlying need by adding a new
**"Examples and Edge Cases"** narrative page to the IPS IG that
documents — with code/value-set references and short JSON snippets —
how an implementer should construct *minimal* and *healthy patient* IPS
documents using existing IPS-published mechanisms (`emptyReason`,
`absentOrUnknown*` codes, `dataAbsentReason`, etc.). The single new
no-information example would be authored under the IPS IG's
`input/examples/` directory; the narrative page would live alongside
the existing Examples / Guidance pages.

#### Justification

The "minimal" and "healthy patient" cases are largely *patterns* rather
than distinct profile-conformance situations — they share the same
structural shape as the existing minimal IPS Bundle example and differ
only in which "no known" / `absentOrUnknown` codes are used. Documenting
the patterns in narrative form (with snippets) gives implementers
broader, more transferable guidance than three near-identical example
files, and avoids the maintenance burden of keeping three near-duplicate
Bundles aligned with every future IPS revision. The genuinely distinct
"no-information" case (where `dataAbsentReason` semantics drive the
shape) is preserved as a concrete instance because it is the hardest
edge case to derive from prose alone.

---

### Disposition C: Decline

#### Proposal

Decline the request as **Not Persuasive — sufficient examples and
guidance already exist**. The IPS IG already ships the "IPS Bundle
example - minimal" instance (re-pointed correctly per FHIR-39272), the
IPS profiles already define `mustSupport` and slicing that constrain
the shape of a conformant minimal/healthy/no-info Bundle, and the
IPS-published value sets (`absentOrUnknown*`, "no known" codes) already
provide the building blocks for the "healthy" and "no-info" cases.
Implementers can construct those documents from the existing example
plus the existing profile + value-set guidance.

#### Justification

This disposition is defensible primarily on scope and maintenance
grounds: the ticket has been open since 2018, the WG explicitly chose
*Considered for Future Use* once already, and example instances do not
ship with normative force — they are illustrative. Adding more examples
expands the surface area the IG editors must keep in sync across
publications without changing what conformant IPS implementations are
required to do. If implementers are demonstrably struggling to build the
edge-case documents, that is better addressed in a future ticket with
concrete reported issues to point at. (Note: this disposition is the
weakest of the three in light of the WG's recent acceptance of similar
example-curation tickets.)

---

### Recommendation

**Recommended disposition:** A

The reporter's request is narrowly scoped, the WG has consistently
accepted analogous IPS example-curation tickets as *Persuasive*
(FHIR-16161, FHIR-39272, FHIR-25004), and the three named cases —
minimal, no-info, healthy — are the exact edge cases that drive the
most "how do I encode X?" questions on the IPS Zulip stream and at
connectathons. Three discrete example instances are concretely
referenceable from issue trackers, test scripts, and validator fixtures
in a way that a narrative page (Disposition B) is not, and the
maintenance overhead is small because the IPS IG's example tooling
already handles the existing minimal Bundle. Disposition B remains a
reasonable fallback if the WG decides the "minimal" and "healthy"
shapes are too close to justify two separate files — in that case, ship
*no-information* as a new example and consolidate the other two cases
into an Examples-and-Edge-Cases narrative page.
