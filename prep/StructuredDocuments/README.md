#### Topic: Reuse US Core / DAF profiles in C-CDA on FHIR references

Each of these tickets asks the C-CDA on FHIR profiles to reference an existing US Core / DAF profile rather than the unprofiled base resource. They cover the Family History section's encounter reference (DAF Encounter), Composition.encounter on the Transfer Summary (DAF Encounter again), and the Plan of Treatment / Medical Equipment sections (US Core CarePlan and US Core Device). Together they push C-CDA on FHIR to compose with the established US realm profile set instead of duplicating constraints.

#### Topic: Composition.section slicing and constraint completeness

Both tickets concern how Composition profiles in C-CDA on FHIR slice and constrain `Composition.section`. One reports that the more complex C-CDA constraints (for example the H&P combined Chief Complaint / Reason for Visit and Assessment sections, mutually exclusive children, and so on) have not yet been carried into the FHIR profiles. The other reports that the slicing discriminator on Composition.section uses `openAtEnd` where the intent is `open`, which causes profile validators to reject conformant section orderings.

#### Topic: Replace fixedCodeableConcept with leaf-level coding in Composition profiles

These tickets — which share a title and were filed against the same Composition profiles — argue that the C-CDA on FHIR profiles should not pin `Composition.type`, `Composition.category`, and `Composition.section.code` with `fixedCodeableConcept`. Instead the system and code should be fixed at the leaf `Coding.system` / `Coding.code` level so that translations, additional codings, and downstream binding work correctly.
