## Summary

The reporter (Rob McClure) requests that the IPS profile change the binding
for `Condition.severity` away from a LOINC answer list (the LA-code set
chosen for the IPS Problems Section) to a SNOMED CT (SCT) value set. The
ticket acknowledges that LOINC was used because of intellectual-property
considerations but argues SCT codes would be more appropriate, and
internally consistent with the parallel request that `Condition.category`
also be aligned to a more appropriate SCT code rather than the IPS LOINC
"Problem" code. The reporter qualifies the request: changing
`Condition.severity` to SCT only makes sense if the underlying SCT IP
issue (the availability of a SNOMED CT free-set covering severity codes)
can be resolved.

The ticket was originally dispositioned in 2018 as **Considered for Future
Use** (Hausam/Beebe motion, 14-0-3), with the IPS team noting that "in
absence of a currently available SCT free-set, these LOINC codes meet the
IPS needs and principles" and reserving the right to reconsider later.
Lynn Laakso re-opened the ticket on 2022-11-22 ("re-open post STU update
publication"), reverting the prior resolution so the IPS team can revisit
the binding choice in a future revision of the IPS FHIR IG.

The cross-reference index links this ticket to the FHIR artifact
**`Condition.severity`** (the element whose binding is in scope). No other
Jira tickets, GitHub items, or Confluence pages are surfaced as
cross-references. Related Zulip discussion in `IPS > Condition Category
Code` (Oct 2020) covers the same IPS terminology choice space and produced
follow-up tracker FHIR-29409 (which is not currently cached locally).

## Related Zulip Discussions

### [IPS > Condition Category Code](https://chat.fhir.org/#narrow/stream/IPS/topic/Condition%20Category%20Code)

This 14-message thread (Sept-Oct 2020) discusses the closely-related
question of why IPS profiles `Condition.category` to a LOINC code
(`75326-9` "Problem") rather than the FHIR core `problem-list-item` code,
and explicitly notes that the same situation applies to
`Condition.severity`. Rob Hausam explains the IPS team's reasoning: at
the time the IPS was authored, the FHIR core binding for
`Condition.category` was only **example**, the LOINC "Problem" code
aligned with the parallel IPS CDA guide (Giorgio Cangioli notes the IPS
team deliberately kept CDA and FHIR value sets aligned), and SNOMED CT
codes were not used because "LOINC is universally available." Michelle
(Moseman) Miller surfaces FHIR-18833, which changed the core
`Condition.category` binding to **extensible** in final R4, technically
making the published IPS non-conformant. Patient Care WG agreed (2020-10-22
call) that R4 implementers should use **dual coding** (CDA LOINC + FHIR
`problem-list-item`) and revisit the invariant in R5. Rob Hausam
explicitly addresses `Condition.severity` in the closing messages: "It
isn't strictly necessary to do it for `Condition.severity`, as using the
LOINC code alone is still conformant and the validation output for that
is an Information message (not a Warning)... I'm not sure if we will
want to add the SNOMED CT severity codes in the technical correction, as
we intentionally did not use the SNOMED CT codes and instead chose LOINC
(as it is universally available) - but we could discuss that." He then
opens follow-up tracker FHIR-29409.

Participants: Peter Jordan, Rob Hausam, Giorgio Cangioli, Michelle
(Moseman) Miller.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the IPS `Condition.severity` binding from the IPS-defined LOINC
answer-list value set
(`http://hl7.org/fhir/uv/ips/ValueSet/condition-severity-uv-ips`) to a
SNOMED CT-based value set drawn from the SNOMED CT international free-set
(e.g., descendants of `272141005 |Severities (qualifier value)|` such as
`24484000 |Severe|`, `6736007 |Moderate|`, `255604002 |Mild|`). Update
the IPS Condition profile (`Condition-uv-ips`) so the `severity` element
points at the new SCT value set, regenerate the corresponding ValueSet
artifact (`ValueSet-condition-severity-uv-ips`) and CodeSystem references,
update IPS examples to use the new codes, and align the IPS CDA guide's
severity terminology if cross-guide consistency is to be preserved.
Confirm with the SNOMED-on-FHIR team that the proposed concepts are in
the SNOMED International free-set before locking the change in.

#### Justification

This is what the reporter literally asked for, and SNOMED CT is the
HL7-preferred clinical terminology for clinical findings, qualifiers, and
severity. Using SCT for `severity` is internally consistent with FHIR
core's own non-normative recommendation
(`http://hl7.org/fhir/ValueSet/condition-severity` already includes SCT
severity codes) and would let IPS instances avoid the validator's
"information" message about being outside the FHIR core preferred binding.
The reporter explicitly conditions the request on the SNOMED CT IP issue
being resolved; with the SNOMED International free-set now established and
broadly licensed for IPS-style use, the original 2018 blocker is
substantially weaker than it was when the ticket was first triaged.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing LOINC answer-list as the primary binding, but extend
the IPS `condition-severity-uv-ips` value set to **also** include the
SNOMED CT severity concepts (mirroring the dual-coding pattern already
adopted for `Condition.category`). The binding strength stays
**preferred**, the LOINC LA codes remain the canonical IPS choices for
backward compatibility, and IPS examples are updated to demonstrate the
recommended dual-coding pattern (LOINC + SCT in a single
`CodeableConcept`). Document in the IPS Condition profile narrative that
implementers MAY supply SCT codes either alongside or instead of the
LOINC LA codes.

#### Justification

This addresses the reporter's underlying concern (SCT codes should be
usable for `Condition.severity`) without breaking implementations already
producing LOINC LA codes per the published IPS R4 IG, and without
re-opening the IP question for jurisdictions that lack a SNOMED CT
license. It is exactly the pattern Patient Care WG and the IPS team
agreed to for `Condition.category` in the 2020-10-22 PC call (per the
Zulip thread above), so it is consistent with established IPS practice.
It also keeps CDA/FHIR alignment - the IPS CDA guide can continue to use
LOINC as the lead terminology while the FHIR profile additionally permits
SCT, which is closer to how Hausam framed the open question in the Zulip
thread ("we could discuss that").

---

### Disposition C: Decline (Persistent: Considered for Future Use)

#### Proposal

Re-affirm the 2018 disposition: keep the IPS `Condition.severity` binding
on the LOINC answer-list, on the explicit basis that LOINC is universally
available without licensing constraints and that the validator
classification for the current binding is only an *information* message,
not a warning or error. Mark the ticket **Persistent / Considered for
Future Use** so it remains visible for re-evaluation if a future SNOMED
CT licensing change makes a clean SCT-only binding viable across all IPS
deployment jurisdictions.

#### Justification

The IPS team's original 2018 reasoning (universal availability of LOINC,
non-blocking validator behaviour, alignment with the IPS CDA guide) is
still operationally valid, and Rob Hausam's 2020 reading in the Zulip
thread was that the SCT change was not required for conformance and
should be discussed before being adopted. Changing the binding outright
would impose churn on existing IPS implementations and break alignment
with the IPS CDA guide unless that is also re-aligned in lock-step. If
the workgroup is not prepared to commit to the cross-guide and tooling
work needed for a clean SCT-only binding right now, deferring is more
honest than promising a change without a plan.

---

### Recommendation

**Recommended disposition:** **B** (extend the value set to allow dual
coding with SNOMED CT alongside the existing LOINC LA codes)

Disposition B is the most defensible path. The reporter's concern is
real - SNOMED CT severity codes are the natural choice for clinical
severity, and several IPS implementers want to use them - but
Disposition A (full SCT switch) is too aggressive on its own: it would
break the deliberate CDA/FHIR alignment the IPS team established, churn
existing R4 IPS implementations, and re-open the SNOMED CT licensing
question that the original 2018 vote treated as a hard blocker.
Disposition C (re-affirm "Considered for Future Use") was already tried
and was explicitly reverted by Lynn Laakso in 2022 ("re-open post STU
update publication") - the workgroup has signalled it does not want to
keep deferring. Disposition B mirrors the Patient Care WG /
SDWG-IPS-team consensus already reached for the closely related
`Condition.category` issue (per the IPS Zulip thread above): allow
dual-coding via the value set, keep the binding strength stable, leave
LOINC as the primary code for cross-guide and tooling continuity, and
let implementers add SCT codes where they prefer or where downstream
consumers demand them. That outcome answers the reporter without
forcing either a breaking change or another deferral, and it gives the
IPS team a clean future path to promote SCT to primary if/when SNOMED CT
free-set coverage and adoption justify it.
