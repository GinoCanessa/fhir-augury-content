## Summary

The ticket asks Structured Documents to update the **US C-CDA on FHIR**
implementation guide so that its "Consent" section is built on the **FHIR
`Consent` resource** rather than the legacy **`Consent` profile on
`Contract`**. The reporter, David Pyke, points out that the
profile-on-`Contract` representation has been unmaintained since 2014 and
historically had no work-group ownership (later picked up by CBCP), making
it a poor foundation for a published IG. The request is to migrate to the
first-class `Consent` resource "as soon as is practical." The ticket was
triaged with resolution **Persuasive** ("Will fix"), assigning the work to
Rick Geimer for incorporation into the C-CDA on FHIR IG.

No other Jira tickets were surfaced as cross-references for FHIR-17367, so
the request stands alone in the available data; however, it is a
direct consequence of the broader retirement of the
`StructureDefinition/consent` profile on `Contract` in core FHIR in favor
of the dedicated `Consent` resource.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the C-CDA on FHIR IG's Consent representation from a profile on
the **`Contract`** resource to a profile on the **`Consent`** resource.
Concretely:

- Author a new `StructureDefinition` (e.g., `consent-ccda` or
  `Consent-CCDAonFHIR`) constraining `Consent` with the elements C-CDA's
  Consent section needs: `Consent.status`, `Consent.scope` (typically
  `treatment` or `patient-privacy`), `Consent.category` (bound to a
  C-CDA-aligned value set / LOINC consent-document codes),
  `Consent.patient`, `Consent.dateTime`, `Consent.performer`,
  `Consent.sourceAttachment`/`sourceReference`, and a constrained
  `Consent.provision` block where the legacy CDA Consent entries are
  mapped.
- Remove or deprecate the existing `Contract`-based Consent profile from
  the IG and update every Composition profile, narrative example, value
  set binding, and CDA→FHIR mapping table that points at it.
- Refresh the section-level guidance (the "C-CDA on FHIR Consent"
  section) to describe the new resource, including a CDA→`Consent`
  element-level mapping replacing the prior CDA→`Contract` mapping.
- Coordinate with **CBCP** (current owner of the legacy Consent profile
  on Contract) so the deprecation aligns with their plans for the
  unmaintained artifact.

#### Justification

This is exactly what the reporter asked for, what the work group
triaged as **Persuasive ("Will fix")**, and what the broader FHIR
ecosystem already does: `Consent` is a first-class, actively-maintained
normative-track resource designed for exactly this use case, while the
`Consent`-on-`Contract` profile has been unmaintained since 2014 and
sits in a resource (`Contract`) whose semantics are a poor fit for a
patient-privacy/treatment consent document. Continuing to publish
guidance against an abandoned profile actively misleads implementers
and creates interoperability churn each time a new `Consent`
implementation appears in the wild. Aligning C-CDA on FHIR with the
core resource also lets the IG inherit future improvements (terminology
bindings, provision modeling, normative stability) for free.

---

### Disposition B: Alternative Approach — Dual-publish with a deprecation window

#### Proposal

Publish a **new `Consent`-based profile** as the recommended
representation in the next ballot of C-CDA on FHIR, but **retain the
existing `Contract`-based Consent profile as DEPRECATED** for one
publication cycle. Specifically:

- Add the new `Consent`-based profile and section guidance as in
  Disposition A.
- Mark the legacy `Contract`-based profile with `status = retired` (or an
  IG-level "DEPRECATED — use `Consent` profile instead" callout) and
  cross-link it to the replacement.
- Provide a CDA→FHIR migration note and (optionally) a structure map or
  worked example showing how an existing `Contract`-based Consent
  instance maps to the new `Consent`-based instance.
- Remove the legacy profile entirely in the publication after next.

#### Justification

This honors the request while acknowledging that some early implementers
of C-CDA on FHIR may already have produced documents using the
`Contract`-based profile. A one-cycle deprecation window gives those
implementers a documented, reviewable migration path rather than a hard
break, and it lets validators continue to recognize legacy instances
during the transition. The trade-off vs. Disposition A is a slightly
larger, more complex IG for one cycle in exchange for materially less
disruption to existing users — a familiar pattern HL7 has used when
retiring profiles in published IGs.

---

### Disposition C: Decline

#### Proposal

Do **not** migrate the C-CDA on FHIR Consent section to the `Consent`
resource. Keep the existing `Contract`-based Consent profile and
instead coordinate with CBCP to either revive maintenance of the
profile-on-`Contract` or to add a non-normative pointer in the IG noting
that implementers may also represent consent using the `Consent`
resource at their discretion.

#### Justification

The defensible grounds for declining are narrow: (a) the IG has
significant deployed use of the `Contract`-based profile that would be
disrupted by a switch, or (b) the work group lacks bandwidth to do the
migration cleanly in the available publication window. Neither is
suggested by the available data — the ticket itself was triaged as
**Persuasive**, the reporter explicitly notes the legacy artifact is
unmaintained, and the larger FHIR community has long since standardized
on `Consent`. Declining therefore preserves a known-bad state and is
hard to defend.

---

### Recommendation

**Recommended disposition: B (Alternative Approach — dual-publish with a
deprecation window).**

The technical answer is unambiguously Disposition A: the `Contract`-based
Consent profile is an unmaintained 2014-vintage artifact, the FHIR
`Consent` resource is the right home for this section, and the work
group has already triaged this as Persuasive. The only reason to prefer
B over A is operational: C-CDA on FHIR has been published for several
years, so any implementer who has built against the legacy profile
deserves an explicit, reviewable deprecation step rather than a silent
swap. Disposition B delivers the same end-state as A within one
additional publication cycle, with materially lower risk of breaking
deployed instances, and it gives CBCP a natural coordination point for
formally retiring the legacy profile in core. Disposition C should be
rejected — keeping a section of a published IG anchored to an
unmaintained 2014 profile cannot be justified now that `Consent` exists.
