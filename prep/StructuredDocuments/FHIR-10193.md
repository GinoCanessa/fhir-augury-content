## Summary

The reporter notes that the Encounters section of the **C-CDA on FHIR**
Implementation Guide does not constrain `Composition.section.entry` to
reference a profiled Encounter resource. At the time of filing (June
2016), they expected the entry slice to point at the
`http://hl7.org/fhir/StructureDefinition/daf-encounter` profile (the
US DAF Encounter profile, then in active use), and supplied an
`ElementDefinition` snippet showing `Composition.section.entry` typed
as `Reference(daf-encounter)` with cardinality `1..*`.

The ticket was originally dispositioned in 2017 as **Considered for
Future Use** (Geimer/Beebe, 16-0-0) on the grounds that DAF-Encounter
had been removed from US-Core and the section would instead link to an
un-profiled `Encounter`. In March 2024 (after the STU 1.2 publication)
Lynn Laakso reverted that resolution and re-opened the issue, leaving
it in **Triaged** state with the underlying concern still open: the
Encounters section's `entry` slice does not reference the
US-jurisdictional Encounter profile that has since taken DAF's place
(US Core Encounter), and the workgroup needs to decide what (if
anything) to constrain it to in the next publication.

No related Jira tickets, GitHub PRs/issues, or substantive Zulip
discussions were surfaced by the cross-reference index — only the
automated FHIR-Bot tracker-item announcement that fired when the ticket
was first posted.

## Details

**Description:**

The Encounters section of the C-CDA on FHIR Implementation guide does
not provide an entry reference for the DAF-Encounters as I expect it
would.

I anticipate the following to appear as the Encounters
`Composition.section.entry` element.

```xml
<element>
    <path value="Composition.section.entry" />
    <short value="entry" />
    <definition value="SHOULD contain zero or more [0..*] U.S. DAF Encounter Profile (identifier: http://hl7.org/fhir/StructureDefinition/daf-encounter)." />

    <max value="*" />
    <type>
```

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **C-CDA on FHIR** IG so that the **Encounters Section**
profile (the SectionDefinition / Composition profile that constrains
the Encounters section of the CCD Composition profile) slices
`Composition.section.entry` and adds a slice typed as
`Reference(<US-jurisdictional Encounter profile>)` with cardinality
`0..*` (or `1..*` if the section is required to be populated when
present). Because the originally-requested `daf-encounter` profile was
retired with DAF and no longer exists, substitute the current
US-jurisdictional Encounter profile — **US Core Encounter**
(`http://hl7.org/fhir/us/core/StructureDefinition/us-core-encounter`)
— as the slice target, and update the `definition` text on the slice
to mirror the requester's wording (e.g., *"SHOULD contain zero or
more US Core Encounter Profile entries"*).

#### Justification

This is the substantive fix the requester originally asked for and is
consistent with the pattern already used elsewhere in C-CDA on FHIR:
each section's `entry` slice is profiled to the corresponding
US-jurisdictional resource profile so that a CCD Composition
generated from US Core resources round-trips against the IG without
unprofiled `Reference`s. Lynn's 2024-03-12 revert explicitly notes
that the original "link to un-profiled Encounter" resolution is no
longer the right answer post-STU 1.2, and US Core Encounter is the
natural successor to the now-retired `daf-encounter` profile that the
ticket originally cited. Constraining to US Core Encounter also
matches what implementers building C-CDA on FHIR Compositions from US
Core data will already be producing.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of hard-binding `Composition.section.entry` of the Encounters
section to US Core Encounter, leave the `entry` typed as
`Reference(Encounter)` (un-profiled) and add a **conformance
statement** in the Encounters Section narrative of the C-CDA on FHIR
IG that says *"When the source data conforms to US Core, the
referenced Encounter SHALL conform to US Core Encounter"*. Optionally
introduce two named slices on `entry` (`us-core-encounter` and
`other-encounter`) using a profile-discriminated `slicing`, both
`0..*`, so a single Composition can carry US Core Encounters and
non-US-Core encounters side-by-side.

#### Justification

C-CDA on FHIR is sometimes used outside strict US Core contexts (e.g.,
documents converted from legacy CDA where the source Encounter
doesn't yet conform to US Core, or where a vendor profile is
preferred). A hard `Reference(us-core-encounter)` slice with `min=1`
would block those documents from validating, which is a meaningful
backward-compatibility risk for the published STU 1.2 content. A
narrative conformance rule plus optional profile slicing preserves
the requester's intent (US Core Encounter is *the* expected target in
the US realm) without tightly coupling the Composition profile to a
specific version of US Core, which is helpful given that US Core
versions independently of C-CDA on FHIR.

---

### Disposition C: Decline

#### Proposal

Decline the change and re-close the ticket with the rationale that the
Encounters section already permits `Reference(Encounter)` and that
implementers needing US-jurisdictional constraints should compose C-CDA
on FHIR with US Core via the standard "downstream IG" mechanism rather
than have C-CDA on FHIR import US Core as a hard dependency.

#### Justification

C-CDA on FHIR is structured to remain a base IG that doesn't pull in
US-jurisdictional profiles directly — that is exactly the reasoning
captured in the original 2017 resolution (DAF was removed from US Core,
so the section was deliberately left pointing at an un-profiled
`Encounter`). Re-opening the ticket only changes which profile would
be substituted (US Core Encounter for `daf-encounter`); it does not
change the architectural argument that made the original disposition
defensible. Implementers who require US Core conformance can express
that via a derived profile or a separate "C-CDA on FHIR + US Core"
binding IG, without the base IG taking on a US Core dependency.

---

### Recommendation

**Recommended disposition:** B

Disposition A directly answers the requester's intent and matches what
most US-realm implementers will actually produce, but it carries
real cost: pinning `Composition.section.entry` to a specific US Core
Encounter profile creates a hard cross-IG dependency and a
release-cadence coupling between C-CDA on FHIR and US Core that the
workgroup deliberately avoided in 2017. Disposition C re-asserts that
2017 reasoning, but Lynn's March-2024 revert is a clear signal from
the workgroup that "un-profiled Encounter" is no longer the desired
end-state. **Disposition B** threads that needle: it documents US
Core Encounter as the expected target in the US realm (which is what
the requester asked for in spirit) and supports it with optional
profile-discriminated slicing, while leaving the Composition
validatable for non-US-Core inputs and avoiding a hard version-pin to
US Core. If the workgroup later decides a hard binding is acceptable,
upgrading from B's optional slice to A's required slice is a minor,
non-breaking change; the reverse is much harder. Recommend taking B
to the workgroup as the proposal and asking explicitly whether the
group wants to escalate to A.
