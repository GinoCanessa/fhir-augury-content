## Summary

This DEQM change request proposes to remove the `stratifier` and `supplementalData`
input parameters from the existing `Measure/$evaluate` (a.k.a. `$evaluate-measure`)
operation invocation pattern and instead address those concerns through a separate,
purpose-built operation tentatively named `$compose-measure`. As the supplied
example invocation shows, callers today bundle a complex tree of `stratifier`
(with `groupId`, `component`, `code`, `description`, `criteria` (CQL/FHIRPath),
`valueSet`, `unit`) and `supplementalData` (with `code`, `usage`, `description`,
`criteria`) parts inside the `$evaluate` `Parameters` resource alongside
operational inputs (`measureUrl`, `periodStart`, `periodEnd`, `subject`,
`reporter`, `location`, `manifest`, `parameters`, `lastReceivedOn`,
`excludeEvaluatedResources`). The reporter argues this overloads `$evaluate`
with measure-authoring/composition concerns that should be resolved before
evaluation. The request is to factor those concerns out so that `$evaluate`
receives an already-composed (or referenced) Measure and `$compose-measure`
becomes the single, reusable mechanism for layering stratifiers and supplemental
data on top of one or more base Measures (with `groupId` targeting and
component-level criteria/value-set bindings preserved). The cross-reference
index surfaced no related Jira tickets, Zulip threads, or GitHub items for
this issue beyond an automatic mention of the FHIRPath element `Patient.gender`
that appears inside the example payload.

### Disposition A: Accept As Requested

#### Proposal

Introduce a new `Measure-compose` (`$compose-measure`) OperationDefinition
in the US Da Vinci DEQM IG, scoped to `Measure` (type-level and
instance-level), whose inputs include the union of measure-composition
parts currently carried inline by `$evaluate`:

- `measureUrl` (canonical, 1..*) â€” the base Measure(s) to compose against.
- `stratifier` (BackboneElement-shaped Parameters part, 0..*) with
  `groupId` (0..*), `component` (1..*) â†’ `code`, `description`,
  `criteria` (Expression), `valueSet` (canonical, 0..*), `unit`
  (string, 0..1) â€” mirroring the structure shown in the example
  comment.
- `supplementalData` (0..*) with `measureUrl` (0..1 targeting),
  `code` (CodeableConcept), `usage` (CodeableConcept, bound to
  `measure-data-usage`), `description` (markdown), `criteria`
  (Expression).

The operation returns a composed `Measure` (or a `Bundle` of composed
Measures, one per `measureUrl`) with `Measure.group.stratifier` and
`Measure.supplementalData` populated from the inputs. `$evaluate` is
amended (in DEQM, not in core R5/R6) to remove the inline `stratifier`
and `supplementalData` input parts and to instead consume the canonical
reference to the composed Measure (or accept it inline via a
`measure` Parameters part). `$evaluate` retains all operational
parameters (`periodStart`, `periodEnd`, `subject`, `reporter`,
`location`, `manifest`, `parameters`, `lastReceivedOn`, `reportType`,
`excludeEvaluatedResources`).

#### Justification

This separation aligns with the FHIR design principle that operations
should have a single, well-scoped responsibility. Composition (mutating
the Measure definition) and evaluation (running the resulting Measure
against data) are conceptually distinct: composition is idempotent and
cacheable, evaluation is per-period / per-subject. Splitting them
enables servers to cache composed Measures, lets clients reuse the
same composition across many `$evaluate` calls (a common DEQM pattern
when an MCO runs the same set of stratifiers across every reporting
period and every member-group), and produces a persistable artifact
(the composed Measure) that is auditable for regulatory reporting.
The example payload in the comment demonstrates that the inline form
is genuinely heavy â€” three stratifier blocks and three supplementalData
blocks for two Measures â€” and that complexity scales linearly with
both stratifier count and Measure count, making the case for factoring
it out compelling.

### Disposition B: Alternative Approach

#### Proposal

Rather than minting a new `$compose-measure` operation, define a
**`MeasureComposition` profile / extension pattern** on the `Measure`
resource (and a small `composedFrom`/`appliesStratifiers` extension
chain) and update the DEQM `$evaluate` guidance to require that
stratifier and supplemental data customisation be expressed as a
*separately-authored Measure* (either persisted on the server or
passed inline as a `measure` parameter) that derives from the base
Measure(s) via `Measure.derivedFrom` (already in core). `$evaluate`
keeps its current parameter list â€” including, in a backward-compatible
deprecated state, the inline `stratifier` and `supplementalData` parts
â€” but DEQM marks them `deprecated` for removal in the next major DEQM
version and the IG narrative directs implementers to the derive-and-pass
pattern. Optionally publish a `compose-measure` *Library/PlanDefinition*
example showing how to perform the composition client-side with no new
operation.

#### Proposal trade-offs vs. A

Avoids adding a new operation surface to DEQM, leverages mechanisms
that already exist in core FHIR (`Measure.derivedFrom`, transactional
PUT of a composed Measure, `measure` parameter on `$evaluate`), and
preserves backward compatibility for existing clients during a
deprecation window.

#### Justification

DEQM has historically been cautious about adding new operations when
existing FHIR mechanisms can carry the same intent, because each new
operation is an interoperability burden on every conforming server.
The composition described in the ticket is essentially "build a Measure
that derives from one or more base Measures and adds stratifier
components / supplementalData entries" â€” which is already legal in
the Measure resource itself. A profile + IG-narrative approach
documents the pattern without requiring server vendors to implement a
new operation, while still solving the underlying readability /
reusability problem the reporter is highlighting. The cost is that
clients must perform the composition themselves (or rely on a
PlanDefinition), and there is no single round-trip endpoint that
both composes and persists.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the current `$evaluate` parameter set,
including the inline `stratifier` and `supplementalData` parts shown
in the example. Rationale category: **insufficient demonstrated need
beyond a single reporter, and request is at the wrong layer** â€”
`$evaluate-measure` is owned by the FHIR Clinical Reasoning module
(core) and DEQM constrains/profiles it; introducing a DEQM-only
`$compose-measure` would fork the operation surface from core CR
implementations and create a divergence that other CR-aware tooling
(CQL engines, measure repositories) would not honour.

#### Justification

The ticket has no linked tickets, no Zulip discussion, no GitHub
implementation issues, no community co-signers, and is currently
`Triaged / Unassigned / Medium`. The example invocation, while
verbose, is functional today and is the established pattern for
several reference implementations. Adding a new operation creates
permanent maintenance burden and conformance-test surface for every
DEQM-conformant server, and the same readability concern can be
addressed by clients structuring their requests with helper code or
templated `Parameters` resources. If the underlying concern is really
about caching/reuse, that can be solved by persisting a Measure (with
the desired stratifiers baked in via `Measure.group.stratifier` and
`Measure.supplementalData`) on the server and passing its canonical
URL to `$evaluate` â€” no new operation is needed.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right starting point for the workgroup discussion
because it solves the reporter's actual problem (the inline payload is
unwieldy and not reusable) without minting a new operation that DEQM
would have to coordinate with the core Clinical Reasoning module and
with every CR-conformant server vendor. FHIR already provides the
building blocks â€” `Measure.derivedFrom`, `Measure.group.stratifier`,
`Measure.supplementalData`, and the existing `measure` (inline) /
canonical-reference inputs to `$evaluate-measure` â€” so the missing
piece is normative IG guidance and an example pattern, not a new
operation surface. If the workgroup discovers, during the discussion,
that a meaningful number of implementers want a single round-trip
"compose then persist" endpoint and are willing to bear the
implementation cost, the recommendation should escalate to Disposition
A (and the new operation should be raised against Clinical Reasoning,
not just DEQM, to avoid forking). Disposition C is not recommended
because the example payload genuinely demonstrates a usability problem
worth addressing in some form.
