## Summary

Bryn Rhodes is requesting that the **Measure Reporting** section of the
US Da Vinci DEQM IG be expanded to provide a concrete walkthrough of how
`MeasureReport` resources are constructed when calculating quality
measures. Today the section delegates almost entirely to the FHIR Quality
Measure IG by reference, which leaves implementers without a worked
example of how the reporting flow actually plays out. The ticket asks
for a step-by-step example of a proportion measure in both subject-based
and non-subject-based forms, plus walkthroughs of supplemental data,
stratifiers, value-based stratifiers, and continuous-variable values,
each linked back to the section of the Quality Measure IG that defines
the underlying semantics.

The ticket is explicitly paired with [FHIR-56278](https://jira.hl7.org/browse/FHIR-56278)
("Clarify population counting semantics for non-subject-based measures"),
which is **Resolved – change required (Persuasive)** against the Quality
Measure IG. FHIR-56278 tightens Conformance Requirement 3.11 to state
that for subject-based measures a null result is interpreted as false,
and for non-subject-based measures a null result is interpreted as an
empty list with null elements in that list not contributing to the
result. FHIR-56279 wants the new DEQM walkthrough to show this
interpretation in action so that DEQM implementers reach the same
counting outcomes as a plain Quality Measure IG implementation.

## Details

**Description:**

> The Measure Reporting section provides guidance on how MeasureReport
> resources are constructed by referring to the Quality Measure IG
> calculation descriptions. This section should be expanded by providing
> a simple walkthrough for a proportion measure for a subject-based and
> non-subject-based measure to illustrate how this process is followed.
>
> In particular, clarity should be provided on how to interpret results
> of an expression, consistent with the related tracker
> https://jira.hl7.org/browse/FHIR-56278 for the Quality Measure IG:
>
> > For subject-based measures, a null result is interpreted as false.
> > For non-subject-based measures, a null result is interpreted as an
> > empty list, and null elements within the resulting list do not
> > contribute to the result.
>
> In addition, a walkthrough of calculating supplemental data,
> stratifiers, value-based stratifiers, and continuous-variable values
> should be provided, with links from each area back to the section of
> the quality measure IG where the semantics are defined.

## Proposed Dispositions

(No GitHub repository was surfaced for this ticket, so the proposals
below name the affected IG by specification rather than by an
authored-source path. Implementation will most naturally land in the
DEQM IG source repository under its `input/pages/` (or equivalent)
authoring root for the Measure Reporting page; an editor with write
access to the DEQM source should confirm the exact file before
committing.)

### Disposition A: Accept As Requested

#### Proposal

Expand the **Measure Reporting** narrative page in the US Da Vinci DEQM
IG (specification "US Da Vinci DEQM (FHIR)") with a new "Worked
Examples" subsection that walks through, end-to-end, the construction of
a `MeasureReport`:

1. A **subject-based proportion measure** example, showing one subject
   passing through Initial Population, Denominator, Denominator
   Exclusions, Numerator, and Numerator Exclusions, including at least
   one expression that evaluates to null and is therefore counted as
   `false` per FHIR-56278.
2. A **non-subject-based proportion measure** example using a
   list-returning population expression, including a case where the
   list contains a null element to demonstrate that null elements do
   not contribute to the count.
3. A **supplemental data** walkthrough that produces the corresponding
   `MeasureReport.group.stratifier` / supplemental-data resources from
   the example population.
4. A **stratifier** walkthrough (single-component) and a
   **value-based stratifier** walkthrough (multi-component / value
   expression).
5. A **continuous-variable measure** walkthrough showing aggregation of
   the measure-observation expression across the measure population.

Each subsection ends with an inline link back to the corresponding
normative section of the FHIR Quality Measure IG (the
Calculation/Conformance pages, with the population-counting paragraph
deep-linked to Conformance Requirement 3.11 as updated by FHIR-56278).
No conformance requirements are added in DEQM itself; the change is
purely informative.

#### Justification

This is exactly what the reporter asked for, and it directly addresses a
known interoperability gap: implementers consistently report that DEQM's
"see the Quality Measure IG" pointer is too thin to ground a concrete
implementation. Aligning the worked examples with the new wording from
FHIR-56278 (Persuasive, change required) keeps DEQM consistent with its
upstream IG before the DEQM ballot, avoiding a window where the two
documents teach different null semantics. Because the change is
informative narrative on a single page, it is low risk, requires no
schema or profile changes, and does not impact existing implementers'
conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than maintaining a parallel set of worked examples in DEQM, add
the worked examples to the **FHIR Quality Measure IG** (under the
Calculation chapter) as part of the FHIR-56278 follow-up work, and have
DEQM update the Measure Reporting section with a short "How to read this
section" preamble plus deep links into the new Quality Measure IG
examples. DEQM would still gain the DEQM-specific bits (e.g., DEQM
profiles applied to the example `MeasureReport`, `MeasureReport.evaluatedResource`
expectations under DEQM), but the underlying scoring math would live in
exactly one place.

#### Justification

Score calculation, null handling, stratifiers, and continuous-variable
aggregation are all defined normatively in the Quality Measure IG; DEQM
layers reporting expectations on top. Putting the worked examples in
DEQM risks them drifting out of sync with the base IG over time
(especially after FHIR-56278's wording change), and it doubles the
maintenance burden when a future tracker tightens any of the rules.
Centralising the examples upstream means the FHIR-56278 author can
guarantee the examples match the new wording, and DEQM's contribution
becomes a thin, DEQM-flavoured wrapper that is easier to keep accurate.
The trade-off is a tighter cross-IG dependency and the need for a
parallel CQI agreement to land the examples in the Quality Measure IG.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the Measure
Reporting section already delegates calculation semantics to the FHIR
Quality Measure IG by design, and the soon-to-land FHIR-56278 wording
change to Conformance Requirement 3.11 will itself be sufficient
clarification of the null-handling rule that motivated this ticket.
Implementers who need worked examples can be directed to the existing
calculation examples in the Quality Measure IG and to the
[CQF-Ruler](https://github.com/cqframework/cqf-ruler) reference
implementation.

#### Justification

DEQM is intentionally an exchange profile on top of the Quality Measure
IG and tries to avoid restating its calculation rules. Adding a full
example matrix (proportion subject-based, proportion non-subject-based,
supplemental data, stratifier, value-based stratifier, continuous
variable) is a non-trivial editorial effort that risks drift from the
upstream IG. If the underlying complaint is just the null-handling
rule, FHIR-56278 already addresses that at the normative layer.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly closes the gap that the reporter and downstream
implementers are hitting: DEQM's "go read the other IG" hand-off is the
single most common source of confusion in measure reporting, and a
worked walkthrough is the cheapest, lowest-risk way to fix it. The
change is purely informative, scoped to one page, and carries no
conformance impact, so it is unlikely to provoke ballot blockers. The
reporter is the same author behind FHIR-56278, which makes it
straightforward to keep the DEQM examples and the new Quality Measure
IG wording consistent. Disposition B is intellectually cleaner but
requires coordinated edits in two IGs on the same cycle, which CQI may
not have appetite for; the workgroup can reasonably pivot to B if a
volunteer steps up to own the upstream change. Disposition C should be
rejected — the status quo of "see the Quality Measure IG" has
demonstrably not been enough.
