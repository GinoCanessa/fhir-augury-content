## Summary

The ticket questions the conformance strength used in the HL7 FHIR Clinical
Quality Measures (fhir-cqm) IG for naming population criteria when a measure
defines multiple populations and/or multiple population groups. Specifically,
Conformance Requirement 3.16 mandates (SHALL) a specific naming scheme for
population/group names in that multi-population case, while the more general
Conformance Requirement 3.9 — which says a population criteria expression's
name SHOULD always match the criteria type — is only a SHOULD. The reporter
argues the inconsistency is odd: it is unclear why the multi-population case
should be strictly stronger than the single-population case, and asks the
workgroup to reconsider whether 3.16 truly warrants SHALL. No related Jira,
Zulip, or GitHub items were surfaced via cross-reference, so the request
stands on its own. The reporter notes in the single comment that this tracker
will be folded into STU5 should a future errata be published.

## Details

**Description:**

Conformance Requirement 3.16 — https://build.fhir.org/ig/HL7/fhir-cqm/measure-conformance.html#conformance-requirement-3-16:

> When specifying multiple populations and/or multiple population groups the following naming scheme **SHALL** be used

This seems odd to make this requirement a SHALL especially when Conformance
Requirement 3.9
(https://build.fhir.org/ig/HL7/fhir-cqm/measure-conformance.html#conformance-requirement-3-9)
is a SHOULD:

> The name of an expression specifying a population criteria within a measure **SHOULD** always be the name of the criteria type

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Weaken Conformance Requirement 3.16 in the HL7 FHIR Clinical Quality
Measures IG (`measure-conformance.html#conformance-requirement-3-16`) from
**SHALL** to **SHOULD**, so that the multi-population/multi-group naming
scheme has the same normative strength as Conformance Requirement 3.9
(single-criteria naming). Keep the recommended naming pattern (criteria
type plus group-qualifying suffix) verbatim, and add a sentence clarifying
that tooling MAY rely on this naming convention and that authors SHOULD
follow it to enable generic measure processing. Apply the edit in the
authoring source for that narrative page in `HL7/fhir-cqm` and republish
in the next STU5 errata cycle, as noted by the reporter.

#### Justification

This is the lowest-risk path and directly resolves the inconsistency the
reporter raised: if naming the single-criteria expression after its
criteria type is a SHOULD (CR 3.9), it is difficult to defend a stricter
SHALL for the multi-population variant of the same convention (CR 3.16),
which is just CR 3.9 generalized. Downgrading to SHOULD preserves the
guidance and tooling expectation without rendering otherwise-valid
measures non-conformant, which is especially appropriate given that 3.16
is a naming convention rather than a semantic requirement. The change is
errata-class (normative-strength downgrade, no structural change), which
the reporter's own comment explicitly contemplates for STU5.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both requirements normative but **promote CR 3.9 to SHALL** (aligning
upward) rather than weakening CR 3.16, and add explicit conformance
language that a Measure's population-criteria expression name MUST match
the criteria type (singular form for a single population; the CR 3.16
pattern for multi-population/multi-group measures). Simultaneously,
introduce a single-sentence "invariant" style note and, if feasible,
express the naming rule as a `Measure` profile invariant
(FHIRPath/constraint on `Measure.group.population.code` paired with the
expression `name`) so that validators can flag violations mechanically.
Updates would land in the fhir-cqm IG authoring sources for the
measure-conformance narrative and, if the invariant path is taken, on the
IG's Measure profile.

#### Justification

The inconsistency the reporter noted can be resolved either by lowering
3.16 or raising 3.9. Raising 3.9 is defensible because consistent
expression naming is what enables generic measure engines, quality
reporting tooling, and cross-measure analytics to discover populations
without measure-specific glue code. A uniform SHALL makes the convention
enforceable in validation and removes a common source of interoperability
friction. The trade-off is a potentially breaking change for existing
measures in the wild whose expression names do not match the criteria
type — which is why this disposition should only be taken if the WG
accepts that breakage as part of STU5 and signals it clearly in a
migration note.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** / decline the change. Retain CR 3.16 at
SHALL and CR 3.9 at SHOULD, and add a short clarifying paragraph in the
measure-conformance narrative explaining why the two requirements use
different normative strengths: CR 3.9 is a general stylistic recommendation
applied to any single population-criteria expression, whereas CR 3.16
defines a disambiguation scheme that is *only* meaningful when multiple
populations and/or groups coexist in the same `Measure` — in that case a
deterministic, machine-parseable naming scheme is essential for tooling
to distinguish which expression belongs to which population/group, hence
the stricter SHALL.

#### Justification

The asymmetry the reporter flags is defensible on design grounds: CR 3.9
is about readability for humans and generic tooling (nice-to-have,
SHOULD), while CR 3.16 is the mechanism that makes multi-population
measures unambiguously machine-addressable (needed-to-work, SHALL).
Weakening 3.16 would reintroduce ambiguity where a single `Measure` has
two "numerator" expressions across two groups and tooling has no
conformant way to tell them apart. Declining preserves existing
implementations and validation behavior; the clarifying narrative
addresses the reporter's "this seems odd" concern without altering
conformance semantics.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest fit for an STU5 errata, which is the vehicle
the reporter themselves identified. The ticket is fundamentally about
normative-strength consistency between two closely related naming
requirements, and downgrading 3.16 to SHOULD resolves the inconsistency
with minimal implementer impact: existing measures following the scheme
remain conformant, and measures that deviated under the prior SHALL are
no longer retroactively non-conformant. Disposition B (raise 3.9 to
SHALL) is more principled if machine-enforceable naming is a long-term
goal, but it is a breaking change that should not be introduced via
errata without broader WG signal and an implementer consultation we have
no evidence of in Zulip or linked trackers. Disposition C is defensible
on design grounds but leaves the reporter's concern unaddressed beyond a
narrative clarification, which the WG can still add in addition to A.
Recommend accepting A and, if the WG wants the stronger guarantee later,
opening a separate ticket to pursue B for a future normative revision.
