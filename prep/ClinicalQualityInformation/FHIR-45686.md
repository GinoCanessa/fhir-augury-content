## Summary

The reporter notes that on the **CQFM Composite Measure** profile (IG section
17.46.1.1, part of the *US FHIR Guidance — Quality Reporting* IG, a.k.a.
`cqf-measures`), the **short descriptions** attached to several differential
elements do not actually describe what the element represents. The elements
called out are:

- `Measure.scoringUnit`
- `Measure.relatedArtifact:component.groupId`
- `Measure.relatedArtifact:component.weight`
- `Measure.group.scoringUnit`
- `Measure.group.component`
- `Measure.group.component.groupId`
- `Measure.group.component.weight`
- `Measure.group.population`

The request is editorial: each of those `ElementDefinition.short` values on the
CQFM Composite Measure StructureDefinition should be rewritten so it conveys
the element's purpose (rather than, for example, restating the element name,
echoing the base resource's short, or being left as a slice-defaulted value
that happens to mislead readers in the composite-measure context). No model,
cardinality, or binding change is being requested.

There are no cross-referenced Jira tickets, Zulip threads, Confluence pages, or
GitHub items — the only cross-reference surfaced is the FHIR artifact mention
of `Measure.scoringUnit` itself, which is the same element listed in the
description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as an editorial fix to the **CQFM Composite Measure** profile
in the `HL7/cqf-measures` IG (US FHIR Guidance — Quality Reporting). Rewrite
the `ElementDefinition.short` value on each of the eight called-out
differential entries so that each short reflects the element's purpose
*within a composite measure*. Suggested rewrites (workgroup to ratify exact
wording):

- `Measure.scoringUnit` → "Unit of the composite score (e.g., proportion, ratio, score points)."
- `Measure.relatedArtifact:component.groupId` → "Identifier of the group within the referenced component measure that contributes to this composite."
- `Measure.relatedArtifact:component.weight` → "Weight applied to the referenced component measure when combining into the composite score."
- `Measure.group.scoringUnit` → "Unit of the score reported for this composite measure group."
- `Measure.group.component` → "A component measure (and group) that contributes to this composite group's score."
- `Measure.group.component.groupId` → "Identifier of the group within the component measure used as input to the composite calculation."
- `Measure.group.component.weight` → "Weight applied to this component measure group when combining into the composite group score."
- `Measure.group.population` → "Population criteria for the composite group (typically derived from contributing component measures)."

The change is confined to the Composite Measure StructureDefinition's
differential (authoring root expected at
`input/resources/measure/StructureDefinition-composite-measure-cqfm.*` or
the equivalent FSH source if the IG has been migrated to FSH). No examples,
narrative pages, or terminology need to change. Republish the IG; no
substantive change.

#### Justification

`ElementDefinition.short` is required by FHIR conformance guidance to be a
plain-language summary of the element. When a profile slices or constrains
an element, the short on the differential is what implementers see in the
generated profile table, so leaving an unhelpful short defeats the very
purpose the differential exists. The fix is purely editorial — no behavioral,
schema, or binding impact — so it carries no implementer burden and no
backward-compatibility risk. Aligning the shorts with the composite-measure
semantics is also consistent with how the base `Measure` resource handles
shorts on its own scoring/population elements.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the editorial intent, but expand the scope to a **single coordinated
pass** over the Composite Measure profile rather than only the eight elements
the reporter happened to flag. Specifically:

1. Audit every differential element on
   `StructureDefinition/composite-measure-cqfm` and on any sibling profiles
   it inherits from (e.g., a base `measure-cqfm`) for `short`, `definition`,
   and `comment` strings that were inherited verbatim from base FHIR or
   left as slice defaults.
2. Where `short` is misleading, fix it; where `definition`/`comment` are
   missing the composite-specific rationale, add one or two sentences.
3. Capture the result in a single PR against `HL7/cqf-measures` and link
   it back to FHIR-45686 (and any sibling tickets discovered during the
   audit).

#### Justification

The reporter's list reads like the result of one reading pass over section
17.46.1.1 — it is plausible that other elements on the same profile
(`group.stratifier`, `group.measureScore`, `supplementalData`, etc.) have
similar issues. Doing a one-shot audit avoids paying the
edit-republish-review cost repeatedly for the same profile, and it gives
implementers a coherent, rather than spotty, set of descriptions. The
trade-off vs. Disposition A is scope: more work up front, and a slightly
larger review surface for the work group, but a markedly better outcome for
implementers reading the generated profile page.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that `ElementDefinition.short` on a profile
differential is permitted to be brief and that the canonical
`ElementDefinition.definition` (inherited from the base `Measure` resource)
already conveys the element's meaning. Close the ticket as **Not
Persuasive** and direct readers to the underlying base-resource definition
where they need a longer description.

#### Justification

This is the defensible "do nothing" path: the base `Measure` resource's
element definitions are normative and implementers can always consult them.
However, this rationale is weak — the whole point of profiling and slicing
is to communicate the *profile-specific* meaning, and the IG's published
profile page is what implementers actually read. Decline is included for
completeness so the work group has the option to push back if it judges the
underlying shorts to be acceptable as-is.

---

### Recommendation

**Recommended disposition:** B

The fix the reporter is asking for is unambiguously correct (Disposition A
would already be accepted on its merits), and there is essentially zero
implementer cost to fixing element shorts. But once an editor is already
opening the Composite Measure profile to fix eight shorts, the marginal
cost of an end-to-end audit of the same profile's differential is small,
and it heads off a steady drip of "and also this short is wrong" tickets
against the same section. Disposition B preserves the reporter's intent
verbatim while delivering a more useful result to implementers, and it
leaves a single, reviewable PR for the work group to approve. Decline (C)
is not defensible here: the reporter is correct that the shorts as written
are unhelpful, and "look at the base resource" is not a good answer for
profile-specific semantics.
