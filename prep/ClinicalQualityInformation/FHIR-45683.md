## Summary

FHIR-45683 is a Change Request filed against the **CQFM (Quality Measure)
implementation guide**, section **17.51.1.1 CQFM Proportion Measure**. The
reporter, Aaron Nusstein, observes that the **slice short descriptions on
`Measure.group.population`** in the CQFM Proportion Measure profile are
not clear or descriptive of what each slice (population component —
e.g., initial-population, denominator, denominator-exclusion,
denominator-exception, numerator, numerator-exclusion) actually represents.
While the meanings are "somewhat intuitive", the reporter argues the
short descriptions should be reworked to better help readers of the
profile and rendered IG. The ticket carries no comments, no explicit
Jira links, and no Zulip/GitHub cross-references in the Augury index;
the only cross-reference surfaced is the implicit mention of the FHIR
element `Measure.group.population`. The request is editorial/clarifying
in nature — it does not propose adding, removing, or re-cardinalising
populations, only improving the human-readable short text on the
existing slices.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **CQFM Proportion Measure profile** (`proportion-measure`
StructureDefinition in `HL7/cqf-measures`, authored under `input/` —
likely the FSH or `StructureDefinition-proportion-measure-*` source
file referenced above) so that each slice on `Measure.group.population`
carries a clear, descriptive `ElementDefinition.short`. Concretely,
rewrite the `short` for the six standard slices roughly as follows
(exact wording to be ratified by CQI):

- **initial-population** — "Patients/cases eligible for measurement
  before any further filtering."
- **denominator** — "Subset of the initial population to which the
  measure outcome applies."
- **denominator-exclusion** — "Cases removed from the denominator
  because they should not be measured at all (e.g., wrong condition,
  hospice)."
- **denominator-exception** — "Cases removed from the denominator only
  when they fail the numerator, due to a documented clinical reason
  (proportion measures only)."
- **numerator** — "Cases in the denominator that meet the desired
  measurement outcome."
- **numerator-exclusion** — "Cases erroneously counted in the
  numerator that must be removed (used in ratio-style proportion
  measures)."

Also re-render the IG section **17.51.1.1 CQFM Proportion Measure** so
that the snapshot/differential table picks up the improved short text.

#### Justification

The change is purely editorial: it clarifies human-readable slice
descriptions on an existing profile without altering cardinality,
slicing discriminators, value sets, invariants, or the profile's
structural contract. It is therefore safe under FHIR / IG
backward-compatibility expectations and squarely matches the
reporter's stated need. Better short descriptions also improve the
rendered IG page (17.51.1.1) and any downstream tooling that surfaces
`ElementDefinition.short` (e.g., profile differentials, IG validators'
error messages, authoring tools), which is a recurring CQI request
pattern around measure population semantics. Cost is low and fits the
"editorial polish" bucket of CQI ballot dispositions.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) editing each slice's
`ElementDefinition.short` in isolation, address the underlying
readability problem by **layering documentation in two places**:

1. Keep `short` text terse but unambiguous (one sentence, ~120
   chars), aligned with the QualityMeasure / CQL specification's
   canonical names for population components.
2. Move the longer explanatory prose (worked examples, the
   denominator-exception vs. denominator-exclusion distinction, the
   numerator-exclusion-only-applies-to-ratio note) into the IG
   narrative page **17.51.1.1 CQFM Proportion Measure** itself, with
   a small table cross-walking each slice to its plain-language
   meaning and a worked example.

This produces a single canonical place for the long explanation
(narrative page) and keeps the StructureDefinition `short` values
crisp enough to render well in differential tables and tooltips.

#### Justification

`ElementDefinition.short` is rendered in many constrained UIs (tables,
tooltips, validator messages) where a long sentence wraps poorly or
gets truncated. Putting the depth into the IG narrative page — which
already exists at 17.51.1.1 and is the page the reporter cited —
addresses readability for the human audience without bloating the
profile's element table. It also avoids duplicating explanatory prose
across six slice definitions that will drift over time. Trade-off vs.
Disposition A: requires editing both the profile and the narrative
page (slightly more authoring work) and keeps `short` somewhat terse,
so a reader who only sees the profile (e.g., via FHIR tooling, not the
IG site) gets less context than under Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the request as **"Not Persuasive — already addressed"**, on
the basis that the slice names themselves (`initial-population`,
`denominator`, `denominator-exclusion`, `denominator-exception`,
`numerator`, `numerator-exclusion`) are the canonical population
identifiers defined by the Measure resource and the QualityMeasure /
CQL specifications, and that their meanings are documented in the
base FHIR spec (`http://hl7.org/fhir/measure-population`) and in the
CQFM IG narrative. The reporter themselves notes the meanings are
"somewhat intuitive". No specific slice is identified as actively
misleading.

#### Justification

A defensible "decline" rests on three points: (1) the population codes
are normative across CQL/QualityMeasure and the codes' display text
in the value set is the authoritative short definition; (2) the
ticket does not flag any specific incorrect or harmful wording, only
a general feeling that the descriptions could be better; and (3)
opening editorial revisions on these slices invites bikeshedding and
risk of inconsistency with the base `measure-population` value set
display strings. If declined, the reviewer should still note where in
the IG narrative readers can find the long-form explanation, so the
ticket is not perceived as ignored.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is the strongest balance between the reporter's clearly
reasonable concern (the current short descriptions don't actually
explain the slices, especially the easily-confused
denominator-exclusion vs. denominator-exception pair) and the
practical realities of profile editing and IG rendering. Disposition A
is acceptable but risks producing `short` strings that are too long
for tooling display, and it duplicates explanatory prose in six
places where it will be hard to keep in sync. Disposition C
under-serves the reporter: the request is editorial and low-risk,
"already adequate" is unconvincing when the IG itself is the canonical
home for this kind of explanation. Disposition B explicitly anchors
the long explanation in the IG narrative at 17.51.1.1 (where the
reporter was already looking) and tightens the profile `short`
strings to be unambiguous without overflowing UI surfaces. Before
work begins, run `repo-analysis HL7/cqf-measures` so the editor can
confirm the FSH source path, slice IDs, and any IG-publisher
gotchas around regenerating section 17.51.1.1.
