## Summary

The reporter notes that the link to "CQLLibrary" in section 7.2 "Library
Profile Usage" of the US FHIR Guidance - Quality Reporting (FHIR)
specification is dead. The link in the published guide points at
`http://hl7.org/fhir/uv/cql/StructureDefinition/cql-library`, which no
longer resolves. The reporter asks that the link be updated to the
functional canonical URL for the CQL Library profile so that readers of
section 7.2 can navigate from the Library Profile Usage discussion to
the underlying StructureDefinition. No related Jira tickets were
surfaced via cross-references, and no Zulip or GitHub items were found

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the hyperlink in section 7.2 "Library Profile Usage" of the US
FHIR Guidance - Quality Reporting (FHIR) specification so that the
"CQLLibrary" reference resolves to the current canonical URL of the CQL
Library profile in the international Using CQL With FHIR (CQL) IG. The
edit is a single anchor target change in the section 7.2 narrative
markdown / xhtml; no profile, value set, or example content is
affected. Verify the replacement target by following the canonical URL
currently published for the CQL Library StructureDefinition in the
latest released version of `http://hl7.org/fhir/uv/cql` and use that
URL (with a stable version suffix if cross-IG link guidance for this
spec calls for one). Add the corrected link to the IG's link-check
fixture (if one exists) so that future renames of the cross-referenced
profile are caught at build time rather than in ballot.

#### Justification

This is the literal request on the ticket and matches its
classification as a Technical Correction. A dead cross-IG link
materially harms readability of the Library Profile Usage section,
which exists specifically to point readers at the upstream profile
definition. Fixing the URL is low-risk, has no normative impact, and
restores parity with the rest of the spec's cross-IG link conventions.
The reporter has already supplied the broken URL, so the diagnosis and
scope are unambiguous.

---

### Disposition B: Alternative Approach

#### Proposal

Replace the bare cross-IG hyperlink with a versioned, structured
reference: render the target as the CQL Library profile's canonical URL
pinned to a specific published version of the CQL IG (e.g.,
`http://hl7.org/fhir/uv/cql/<version>/StructureDefinition-…`), and at
the same time sweep section 7.2 (and the rest of the Library Profile
Usage area) for any other unversioned pointers into
`http://hl7.org/fhir/uv/cql`. Where the IG's tooling supports it,
convert the inline links to the IG-publisher cross-IG link macros (e.g.
`[[CQL Library]]`-style references) so that the target is resolved at
build time against the dependency declared in the IG's
`dependsOn` / `package-list.json`, eliminating this whole class of
defect.

#### Justification

The root cause of FHIR-45687 is not just a single stale URL; it is that
the section 7.2 narrative hard-codes a cross-IG link rather than using
the IG publisher's resolver, so any rename or restructuring upstream
silently breaks readers. A scoped sweep + adoption of the publisher's
cross-IG link mechanism for the CQL IG dependency is only modestly
more work than Disposition A, prevents recurrence, and aligns the
Library Profile Usage section with the link style most other CQI/CQF
guides use. The trade-off is slightly more reviewer effort in this
ballot cycle in exchange for materially lower maintenance cost across
future cycles.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the broken target represents a
relocation in the upstream international CQL IG that the CQI work
group does not own, and ask the reporter to file the defect against
the international CQL IG so that the upstream URL is reinstated (or so
that an explicit redirect / migration table is published) before the
US Quality Reporting guide is patched downstream.

#### Justification

The dead URL lives at `hl7.org/fhir/uv/cql/...`, which is the
international CQL IG's canonical, not the US Quality Reporting guide's.
If the upstream profile was renamed without a redirect, mechanically
patching every downstream guide is whack-a-mole; the durable fix is
upstream. This disposition is defensible only if the upstream change is
recent and reversible — otherwise the reporter is left with a broken
guide while the upstream conversation plays out, which is a poor
reader experience for a published US-realm guidance document.

---

### Recommendation

**Recommended disposition:** B

The ticket is correctly classified as a Technical Correction and the
narrowest fix (Disposition A) would close it, but the underlying
failure mode — an unversioned, hand-written cross-IG hyperlink in a
section whose entire purpose is to direct readers at a profile owned
by a different IG — will recur the next time the upstream CQL IG
renames or republishes. Disposition B keeps the editorial scope tight
(section 7.2 and immediately adjacent Library Profile Usage links)
while fixing the class of defect, costs only marginally more reviewer
time than A, and leaves the guide measurably more durable. Disposition
C is hard to justify because the reporter's complaint is that the
published US guide is broken today; deferring to upstream does not
relieve CQI of the responsibility to keep its own published narrative
navigable.
