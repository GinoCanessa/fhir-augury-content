## Summary

The reporter (Aaron Nusstein) observes that in section **17.0.9 "Example
Instances"** of the *US FHIR Guidance — Quality Reporting* IG, several of
the published Example Instances carry only the placeholder description
"This is an example only." rather than a meaningful narrative. The ticket
asks the workgroup to replace those placeholders with more substantive,
in-depth descriptions so that readers of the IG can understand what each
example is intended to demonstrate (clinical scenario, profile being
exercised, expected use, etc.). The request is editorial / documentation
in nature; it does not propose any changes to the underlying StructureDefinitions,
ValueSets, or instance JSON/XML — only to the human-readable descriptions
attached to the examples on the Artifacts / Example Instances page.

The cross-reference index returned no incoming or outgoing links for this
ticket (no related Jira issues, Zulip threads, or GitHub items were
recorded). Keyword-similarity searches surface a few thematically similar
tickets that the workgroup may wish to consider together (see *Related
Jira Tickets* below), notably **FHIR-47427** (Structured Documents) which
makes the same observation against another IG and is a useful precedent
for the kind of remediation expected.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the ticket as a Persuasive technical/editorial correction.
For every Example Instance on section 17.0.9 of the US FHIR Guidance
for Quality Reporting IG whose narrative is the placeholder
"This is an example only.", replace the description with a 2–4
sentence narrative that explicitly states:

1. The clinical / quality-reporting scenario the example illustrates
   (e.g., "An eCQM individual MeasureReport for a 67-year-old patient
   meeting the numerator of CMS122v12").
2. The principal profile(s) being exercised (e.g., `QICore-Patient`,
   `MeasureReport-deqm-individual`).
3. Why a reader of this IG would consult this example (what
   construction pattern or constraint is being demonstrated).

The change is limited to the example narrative metadata (the IG
publisher's `description` field on each example resource and/or the
intro text rendered above each example on the Artifacts page); no
changes to the example payloads, profiles, or value sets are required.
Identify the affected examples by grepping the IG sources for the
literal string "This is an example only."

#### Justification

This matches the literal request and is consistent with the workgroup's
own remediation pattern in the analogous FHIR-47427 ticket (Structured
Documents accepted the same complaint as Persuasive and applied it).
The change is low-risk (description text only — no normative impact,
no breaking changes for implementers, no schema or terminology churn),
high-value for IG readability, and aligns with HL7's general
expectation that published examples carry meaningful narrative for
ballot reviewers and downstream implementers. The only cost is editorial
authoring time.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying need but address it structurally rather than as a
one-off edit pass. Specifically:

1. Add an editorial rule to the IG's contribution / style guide that
   bans the placeholder string "This is an example only." (and any
   equivalent boilerplate) for any Example Instance, requiring instead
   a structured description containing *scenario*, *profile(s)
   exercised*, and *reader takeaway*.
2. Add a CI / publication check (e.g., a Sushi or IG-publisher
   post-processing step, or a simple FSH lint script in the IG repo)
   that fails the build if any example resource has a description
   matching the banned placeholder pattern or is shorter than a
   threshold (e.g., 80 characters).
3. As part of the same change, fix the existing offending descriptions
   so the new check passes.

#### Justification

Disposition A fixes today's instance of the problem but does not
prevent regression — new examples added by future contributors can
re-introduce the same placeholder, and historically (per the
keyword-similar FHIR-47427 ticket) the same complaint has surfaced
across multiple IGs. Encoding the rule as both a documented
contributor expectation *and* a build-time check makes the fix
durable and self-policing, at the cost of a one-time tooling/CI
investment in the IG's source repository. This trade is usually
worth it for an IG that publishes regularly and accepts contributions
from multiple authors.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as **Not Persuasive — editorial-only, no normative
impact**, with a note that the workgroup welcomes a pull request from
the reporter (or any contributor) supplying the suggested improved
descriptions for the specific examples in question, but will not
prioritise a workgroup-driven editorial pass given current ballot and
errata workload.

#### Justification

The placeholder text, while ugly, does not cause incorrect
implementation behaviour, does not affect conformance, and does not
block use of the examples (the example resources themselves are
present and parseable). A workgroup with limited editorial bandwidth
might reasonably defer this to a community contribution rather than
spending CQI cycles on it. This disposition is defensible only if the
workgroup is content with the IG continuing to ship visible
"This is an example only." text on its Artifacts page in the interim
— which is likely to attract repeat tickets (cf. FHIR-47427) and
erode reviewer confidence.

---

### Recommendation

**Recommended disposition:** B

The substance of the ticket is uncontroversial — placeholder descriptions
are objectively a documentation defect and should be fixed (so
Disposition C should not be the answer; declining a self-evident
documentation bug invites a repeat filing, exactly as we already see in
FHIR-47427 against a different IG). Between A and B, Disposition B
delivers the same fix as A but additionally hardens the IG against
recurrence by codifying the rule and enforcing it in the build. Given
that the same problem has independently surfaced in at least one other
HL7 IG, treating "no placeholder example descriptions" as a structural
quality gate — rather than a single editorial sweep — is the higher-
leverage answer for the Clinical Quality Information workgroup. If
tooling capacity is genuinely unavailable in this cycle, the workgroup
can fall back to Disposition A for the immediate publish and carry the
CI-check work as a follow-up tracking item.
