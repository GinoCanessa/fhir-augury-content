## Summary

The QI-Core `Coverage` profile currently documents the `type` data element as
drawing from an SOP (Source of Payment Typology) code system whose steward is
the "US Public Health Data Consortium." The reporter notes that this steward
attribution is incorrect: the SOPT code system is in fact stewarded by the
Centers for Disease Control and Prevention, National Center for Health
Statistics (CDC/NCHS). The correct steward is already reflected in HL7
Terminology (THO) v7.1.0 (see the External Code Systems page) and is tracked
upstream by HL7 ticket UP-631 ("SOPT – Additional Identifier and Other
Changes"). The resolution recorded on the ticket proposes removing the
QI-Core-specific description sentence that re-asserts the incorrect steward
("Categories of types of health care payor entities as defined by the US
Public Health Data Consortium SOP code system") and instead inheriting the
description from US Core, so that when US Core is corrected QI-Core will
automatically reflect the updated wording. No separate cross-referenced
tickets were surfaced in FhirAugury for FHIR-51998; context from the related
UP-631 issue referenced in the description is used for background only.

## Details

**Description:**

Coverage profile, type data element has incorrect SOP code system steward
information. SOPT code system steward is Centers for Disease Control and
Prevention, National Center for Health Statistics (CDC/NCHS), not the US
Public Health Data Consortium.

Referenced links:
- [Source of Payment Typology - NEW DRAFT – Terminology Authority (Confluence)](https://confluence.hl7.org/spaces/TA/pages/220698394/Source+of+Payment+Typology+-+NEW+DRAFT)
- [UP-631 — SOPT - Additional Identifier and Other Changes](https://jira.hl7.org/browse/UP-631)

**Resolution (Persuasive):**

There is a ticket to start the process of correcting the owner of the SOP
value set. The first step is to correct THO, then the correction can roll
through to US Core and other IGs. Version 7.1.0 has the right owner:
[External Code Systems - HL7 Terminology (THO) v7.1.0](https://terminology.hl7.org/en/external_code_systems.html).

To correct QI-Core, the reporter would like to remove the statement added in
the description of the `type` data element:

> Categories of types of health care payor entities as defined by the US
> Public Health Data Consortium SOP code system

QI-Core should instead inherit the description from US Core. Then, when US
Core is fixed, QI-Core will reflect that update.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the fix described in the resolution to the QICore-Coverage profile:
in the QI-Core Coverage profile's description of the `Coverage.type`
element, remove the QI-Core-added sentence —

> Categories of types of health care payor entities as defined by the US
> Public Health Data Consortium SOP code system

— and let `Coverage.type` inherit the `short` / `definition` text from the
upstream US Core Coverage profile. In authoring terms this means editing
the QI-Core `Coverage` StructureDefinition (typically the authored FSH or
JSON/XML source for the `QICore-Coverage` profile) so that the element
differential on `Coverage.type` no longer overrides `short` or `definition`
with the incorrect steward attribution. No change is made to the bound
value set or to the SOPT code system itself within QI-Core; QI-Core relies
on the upstream THO correction tracked by
[UP-631](https://jira.hl7.org/browse/UP-631) and the subsequent US Core
update to propagate the correct steward wording.

#### Justification

This is the minimal, behaviour-preserving fix that the reporter asked for
and that the resolution already endorses. It follows the established
QI-Core design principle of only deviating from US Core when QI-Core has a
substantive constraint to add — stating the same thing as US Core (or
worse, a stale copy of it) adds no information and actively misleads
implementers when the upstream text changes. Removing the local override
lets the single authoritative description (eventually sourced from THO via
US Core) flow through automatically, which is exactly the layered
correction path captured in the resolution and consistent with the
"Persuasive" disposition already marked on the ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of silently dropping the QI-Core override and waiting for the
upstream THO/US Core fix to land, fix the steward text directly in the
QI-Core `Coverage.type` description now — replacing "US Public Health Data
Consortium" with "Centers for Disease Control and Prevention, National
Center for Health Statistics (CDC/NCHS)" — and leave a short implementer
note referencing UP-631. Once US Core is corrected and QI-Core rebases on
a US Core version that carries the correct wording, remove the QI-Core
override as in Disposition A.

#### Justification

This trades a two-step fix for an immediate one. It gets the correct
steward attribution in front of QI-Core implementers on the next QI-Core
publication rather than waiting for THO (UP-631) → US Core → QI-Core to
cascade. The downside is maintaining a QI-Core-local duplicate of text
that is really US Core / THO's to own, which is exactly the drift that
Disposition A eliminates. It is the right choice only if the QI-Core
publication window is materially ahead of US Core's pickup of the
corrected THO text and implementers are actively being misled in the
meantime.

---

### Disposition C: Decline

#### Proposal

Close FHIR-51998 as "Not Persuasive" on the grounds that QI-Core inherits
`Coverage.type` semantics from US Core, and steward metadata for the SOPT
code system is the responsibility of HL7 Terminology (tracked by UP-631),
not QI-Core. Direct the reporter to UP-631 and to US Core, and make no
change in the QI-Core source.

#### Justification

The only defensible ground for declining is jurisdictional: QI-Core is not
where the SOPT code system's stewardship metadata lives, so arguably QI-Core
should not be restating it at all. However, this argument is weak in
practice — the incorrect steward statement *is* physically present in
QI-Core's `Coverage.type` description and *does* mislead QI-Core
implementers today. Declining leaves that error in place until some future
rebase incidentally removes it, which is why the resolution already tags
this ticket "Persuasive." Recorded here for completeness.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest fix, matches the resolution already attached
to the ticket, and embodies the QI-Core principle of not re-stating US Core
text unless QI-Core is adding a real constraint. The error is a description
override that QI-Core never needed to add in the first place; removing it
both corrects the immediate mistake and structurally prevents the same
drift from recurring when THO (via UP-631) and US Core update their
wording. Disposition B is a reasonable tactical fallback only if the
QI-Core ballot / publication timeline diverges significantly from US
Core's, in which case the workgroup can apply B as an interim and then
converge on A at the next QI-Core/US Core rebase. Disposition C should be
rejected: the incorrect wording lives in QI-Core today and QI-Core is the
right place to remove it.
