## Summary

The reporter proposes that two additional use cases be added to the
Point of Care Devices (PoCD) FHIR implementation guide. The first use
case is a patient in critical-but-stable condition who must be
transferred between facilities, where FHIR-based access to contextual
and device-produced data can connect baseline monitoring before,
during, and after transfer into a single care journey. The second use
case is ambulatory clinics that already have EMR integration in place
and can reduce implementation cost by reusing FHIR transports rather
than standing up additional system or network infrastructure for
device data. The ticket is small in scope: it asks the workgroup to
incorporate these scenarios into the IG's narrative use-case material,
not to change any resource, profile, or operation. No related Jira
tickets were surfaced through cross-references; the only external
linkage is to the Devices WG meeting minutes that re-opened this item
for the upcoming September ballot.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add the two suggested scenarios to the PoCD IG overview / use-cases
narrative as new bullet items (or short subsections) alongside the
existing use cases:

1. **Inter-facility transfer of a critical-but-stable patient** —
   describe the continuity-of-monitoring need (pre-transfer baseline,
   in-transit, post-arrival) and call out FHIR access to device data
   plus contextual resources (Patient, Encounter, Location, Device,
   DeviceMetric, Observation) as the enabler that ties the journey
   together.
2. **Ambulatory clinic reusing existing EMR/FHIR infrastructure** —
   describe the cost-reduction motivation: where an EMR already exposes
   a FHIR endpoint, device data can flow over the same transport rather
   than requiring a parallel device-specific channel.

No resource, profile, extension, value set, or operation changes are
implied. The change is narrative-only and would land in the IG's
overview page (the file Devices WG identifies as the PoCD overview
during ballot prep).

#### Justification

This matches what the Devices WG already agreed to on the 2026-04-08
main call ("these use cases could easily be added to the overview
page") and what mcros recorded when re-opening the item for the
September ballot. Both scenarios are clearly in scope for PoCD —
inter-facility transfer and ambulatory device integration are
mainstream device-data use cases — and adding them strengthens the
IG's positioning narrative without altering any normative content.
The original 2018 vote (9-0-0, "Impact: Non-substantive; Category:
Enhancement") supports this disposition; the only thing that changed
is that the WG now wants to actually do the work rather than defer it.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent, but rather than dropping the two scenarios in as
free-standing bullets, fold them into a structured "Use Cases" section
of the PoCD overview that uses a consistent template (Actor / Setting
/ Trigger / FHIR Resources Involved / Outcome). Refactor any existing
use-case prose in the overview to the same template so the IG presents
one uniform list rather than a mix of legacy paragraphs and new
bullets. Cross-link each use case to the resources it exercises
(Device, DeviceMetric, Observation, Encounter, Location).

#### Justification

The literal request adds two bullets next to whatever exists today,
which can leave the overview inconsistent. A template-driven section
yields a more reviewable and extensible artifact: future use-case
contributions (and there will be more, given how broad PoCD is) can
slot in cleanly, and implementers can scan a uniform structure to find
the scenario closest to their deployment. The downside is more editing
work than Disposition A and a small risk of touching narrative that
other tickets are also editing for the September ballot, so it would
need to be sequenced carefully against parallel ballot-prep changes.

---

### Disposition C: Decline

#### Proposal

Close as "Considered for Future Use" or "Not Persuasive": leave the
PoCD overview page as-is and do not add the two suggested scenarios.

#### Justification

Defensible only on the narrow grounds that the suggested scenarios are
already implicit in the IG's general framing of device data exchange,
so calling them out adds words without adding normative value.
However, this rationale is weak: the WG already explicitly disagreed
with it on 2026-03-26, 2026-03-30, and 2026-04-08, and the original
2018 vote was unanimous in favour of inclusion. Declining now would
reverse a position the WG has held consistently across two ballot
cycles, so this disposition is included for completeness only.

---

### Recommendation

**Recommended disposition:** A

The Devices WG has already converged on accepting the request — the
2026-04-08 minutes record agreement to incorporate the use cases for
the September ballot, and mcros explicitly re-opened the ticket and
flagged it for the block vote. The change is narrative-only,
non-substantive, and carries effectively zero risk to existing
implementations. Disposition B is attractive in principle but expands
scope beyond what this ticket asks for and risks colliding with other
overview-page edits in the ballot-prep window; if the WG wants the
template treatment, that is better tracked as a separate ticket.
Disposition C contradicts the WG's standing position and the original
unanimous vote, and should not be adopted. Accept as requested, land
the two scenarios in the overview page, and clear the item in the
block vote.
