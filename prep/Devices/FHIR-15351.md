## Summary

FHIR-15351 (originally raised as 2018-Jan PoCD #18) asks that the
`DeviceMetric.color` field be re-bound to a coding scheme based on the
HTML color model with RGB valuations, on the grounds that an RGB-based
scheme would be more enforceable, more relatable, and would better
support plug-and-play interoperability without requiring receivers to
interpret named colors. The original 2018 disposition was *Considered
for Future Use* (S. Karl / Reinhold 13-0-0), explicitly because the
ticket was deemed substantially a duplicate of
[FHIR-15342](https://jira.hl7.org/browse/FHIR-15342), which was
accepted. In April 2026 that resolution was reverted by the Devices WG
so the ticket could be re-considered ahead of the September ballot
([2026-04-08 Devices minutes](https://confluence.hl7.org/spaces/HCD/pages/453903324/2026-04-08+Devices+Main+Call+Agenda+and+Minutes)).

Since the original request, the underlying issue has been addressed by
a chain of related Devices tickets:
[FHIR-39430](https://jira.hl7.org/browse/FHIR-39430) (*Published*,
Persuasive with Modification) relaxed the previously over-strict
binding; [FHIR-54464](https://jira.hl7.org/browse/FHIR-54464)
(*Resolved – change required*, Persuasive) changed the data type to
`CodeableConcept`; and
[FHIR-55085](https://jira.hl7.org/browse/FHIR-55085) (*Resolved –
change required*) split the underlying value set into the THO-candidate
`color-names` (`ColorCodes`) and `color-rgb` (`ColorRGB`) code systems.
As Erik Moll noted on 2026-03-26, in R6 `DeviceMetric.color` is bound to
[Color Codes (Names or RGB Values)](https://build.fhir.org/valueset-color-codes.html),
which already permits both common English color names and RGB values —
i.e., the substantive change requested by FHIR-15351 has already been
made.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-bind `DeviceMetric.color` to an HTML/RGB-style color coding scheme.
Concretely, in the Point of Care Devices specification: change the
binding of `DeviceMetric.color` so that the value set is restricted to
the `color-rgb` (`ColorRGB`) code system only — i.e., 6-digit
hexadecimal RGB triplets in the HTML `#RRGGBB` form — and remove the
`color-names` (`ColorCodes`) code system from the bound value set.
Retain a *required* binding strength so that all senders express color
with a single, machine-interpretable coding scheme, matching the
original ticket's plug-and-play intent.

#### Justification

This literally implements the request as written: an HTML color scheme
with RGB valuations that is "more enforceable, relatable and
facilitates plug-and-play without the need of interpretation". RGB
hex values are unambiguous, do not require lexical interpretation of
named colors, and align with display tooling that consumes hex color
strings directly. The `color-rgb` code system already exists (per
FHIR-55085), so the only change required is in the value set
composition for `DeviceMetric.color`.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the current R6 `DeviceMetric.color` design — `CodeableConcept`
bound (required) to [Color Codes (Names or RGB Values)](https://build.fhir.org/valueset-color-codes.html),
which combines `color-names` (`ColorCodes`) and `color-rgb`
(`ColorRGB`) — and address the spirit of FHIR-15351 with a
documentation/usage-guidance change instead of a binding change. In
the Point of Care Devices narrative for `DeviceMetric.color`, add an
explicit usage note that:

1. RGB hex codings from `color-rgb` SHOULD be used when interoperable,
   exact display rendering is required;
2. Named codings from `color-names` are appropriate for human-readable
   labelling and where the receiving system is expected to map names to
   display colors;
3. When both are known, both codings MAY be supplied on the same
   `CodeableConcept` (which is now possible because of FHIR-54464).

Optionally, also recommend that implementers populate `text` on the
`CodeableConcept` with the human-readable color name when an RGB
coding is provided, to preserve the relatability the ticket
emphasises.

#### Justification

The substantive value-set work the ticket asks for has already
shipped: FHIR-39430 relaxed the over-strict binding, FHIR-54464 moved
the data type to `CodeableConcept`, and FHIR-55085 produced the
combined named/RGB value set. What the original ticket actually lacks
in the current design is *guidance* on which coding to use when, not a
new code system. A guidance change is non-breaking, preserves the
flexibility implementers have come to rely on, and avoids the
plug-and-play ambiguity FHIR-15351 was concerned about without
invalidating any existing instances. It also avoids dropping the
named-color code system, which is useful in human-facing contexts and
is being considered for THO migration (FHIR-55085).

---

### Disposition C: Decline

#### Proposal

Mark FHIR-15351 as *Not Persuasive — Already Addressed*. Close the
ticket with a resolution comment that points to the cumulative
changes already made:

- [FHIR-39430](https://jira.hl7.org/browse/FHIR-39430) — relaxed the
  over-strict required binding (Published);
- [FHIR-54464](https://jira.hl7.org/browse/FHIR-54464) — changed the
  data type to `CodeableConcept` (Resolved – change required);
- [FHIR-55085](https://jira.hl7.org/browse/FHIR-55085) — established
  the `color-names` and `color-rgb` THO-candidate code systems and the
  combined `color-codes` value set (Resolved – change required).

Make no further normative or value-set changes for FHIR-15351 itself.

#### Justification

The R6 `DeviceMetric.color` element is a `CodeableConcept` bound
(required) to a value set that already permits both RGB hex values and
common English color names — exactly the HTML/RGB-capable scheme the
ticket asks for, plus the named-color flexibility implementers also
need. Erik Moll's 2026-03-26 comment on this very ticket documents
that state of affairs and concludes "no further action needed". The
2018 disposition that originally treated FHIR-15351 as a duplicate of
the accepted [FHIR-15342](https://jira.hl7.org/browse/FHIR-15342) was
correct in substance; the recent revert was procedural (to allow
re-consideration in the September ballot cycle), not a finding that
the underlying request remained unmet.

---

### Recommendation

**Recommended disposition:** C

The literal request in FHIR-15351 — "use an HTML color scheme with
RGB valuations" — has been satisfied by the chain of work in
FHIR-39430, FHIR-54464, and FHIR-55085. R6 `DeviceMetric.color` is
already a `CodeableConcept` bound to a value set that includes
`color-rgb` hex values, so any sender that wants the
plug-and-play, interpretation-free behaviour the ticket asks for can
already obtain it today by sending an RGB coding. Disposition A would
remove the `color-names` code system that other implementers rely on
and would be a breaking change to existing R6 instances for no
additional expressive power. Disposition B is reasonable and could be
adopted as a low-cost follow-up if the WG hears that implementers are
confused about which coding to use, but it is not what FHIR-15351
itself requested and can be raised as a separate guidance ticket.
Closing this ticket as *Already Addressed* — and recording the three
related tickets as the resolution trail — gives the WG a clean answer
for the September ballot and avoids re-litigating settled value-set
decisions.
