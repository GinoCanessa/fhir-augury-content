## Summary

The reporter (Malcolm Clarke) asks the Devices Work Group to simplify the
`PhdCoincidentTimeStampObservation` profile defined in section 4.10.1 of the
Personal Healthcare Devices (PHD) FHIR Implementation Guide. As originally
conceived, this observation is a simple side-by-side comparison of two
timestamps — the current time reported by the Personal Health Device (PHD)
and the current time of the Personal Health Gateway (PHG) — together with
each timestamp's status, providing an audit trail for any time translation
applied to subsequent measurement timestamps. The reporter contends the
profile has grown well beyond that purpose, carrying many fields whose use
is not explained, and asks that it be reduced to the minimum needed:
two timestamps and their respective time-source statuses. The Devices
Work Group has already triaged the issue and deferred it to the next
revision of the PHD IG (post STU2 publication), per the 2025-11-12 Devices
Main Call (Confluence page 404097645). No related Jira tickets, Zulip
threads, or GitHub items were found in the indexed sources.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reduce the `PhdCoincidentTimeStampObservation` profile in PHD IG section
4.10.1 to its minimal originally-intended shape. Concretely, retain only:

1. Two `Observation.component` entries (or two paired `valueDateTime`
   slices) carrying the **PHD-reported current time** and the
   **PHG current time** captured at coincident moments.
2. For each timestamp, a companion component (or extension) carrying the
   **time-source / time status** code (e.g., the IEEE 11073-10101 time
   status flags such as `relative-time`, `epoch-time`, `time-synchronized`,
   `time-set-by-user`, etc.) so consumers can audit any subsequent time
   translation applied to measurement timestamps.

Remove all other components / extensions whose purpose is not documented
in the profile narrative. Ensure the profile narrative explicitly states
that the sole purpose of the resource is to provide an auditable
"time-pair snapshot" used to interpret time translations applied
elsewhere in the data set. Schedule the change for the next revision of
the PHD IG (post STU2), as already agreed by the Devices WG on
2025-11-12.

#### Justification

This matches the reporter's stated original intent and the Devices WG's
existing decision to defer the simplification to the next PHD IG
revision. A minimal profile (two timestamps + two status codes) is
faithful to FHIR's design principle of keeping profiles narrow and
purpose-built, removes elements that have no documented use (and are
therefore a source of implementer confusion and inconsistent data), and
preserves the only behaviour anyone is known to depend on — the audit of
time translation between PHD and PHG clocks. Because the change is
scheduled post-STU2, current STU2 implementers are not destabilised.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing `PhdCoincidentTimeStampObservation` structure but
**document each currently-undocumented element**, mark elements that are
not part of the minimal "time-pair audit" use case as `0..1` and
"informative / optional" with a short rationale, and add a profile-level
narrative that distinguishes the **core minimal pattern** (PHD time,
PHG time, two statuses) from the **optional extras**. Optionally, add a
companion "minimal" profile (e.g., `PhdCoincidentTimeStampObservation-Min`)
that constrains the existing profile down to just the two timestamps and
their statuses, so consumers and validators that only need the audit
behaviour can target the minimal profile while existing implementers are
not required to change.

#### Justification

This addresses the underlying complaint — undocumented, hard-to-use
elements — without removing any current capability or breaking
implementations that may already populate the extra fields. It is a
softer change that can land within a normal IG revision cycle and avoids
a debate over which fields are "really" unused. The cost is that the
profile remains larger than the reporter prefers, and consumers still
have to look past the optional fields to see the audit pattern.

---

### Disposition C: Decline

#### Proposal

Decline to change `PhdCoincidentTimeStampObservation`, on the grounds
that (a) the profile is already published in STU2 and removing fields is
a breaking change for any implementer relying on them, and (b) the
ticket does not enumerate which specific fields are undocumented or
unused — making it difficult to act on without speculation. Close as
"Not Persuasive" pending a more specific proposal that lists the exact
elements to remove and any known implementations affected.

#### Justification

Removing elements from a published STU profile is a non-trivial step,
particularly without an enumerated list of the elements in question or
evidence that no implementer is using them. FHIR's compatibility
guidance discourages dropping elements once published unless there is a
clear harm. If the underlying problem is documentation rather than
structure, that is better handled by a documentation-only ticket.

---

### Recommendation

**Recommended disposition:** A

The Devices WG has **already** triaged this ticket, accepted the
underlying concern, and explicitly deferred the simplification to the
next revision of the PHD IG (per the 2025-11-12 Devices Main Call and
Lynn Laakso's 2026-03-31 comments re-opening the ticket post-STU2).
Disposition A simply formalises that decision into a concrete change
proposal: restrict the profile to two timestamps and two time-status
codes, with narrative that explains the audit purpose. This is faithful
to the reporter's intent, matches the WG's prior decision, and is safe
to do in a non-STU2 release where breaking changes to profile cardinality
are expected. Disposition B is a reasonable fallback if, on review, the
WG finds that some "undocumented" fields actually have implementers
depending on them; in that case, demoting them to documented-optional
preserves behaviour while still cleaning up the profile. Disposition C
is not recommended because the WG has already agreed the change is worth
making — the open question is scope, not whether to act.
