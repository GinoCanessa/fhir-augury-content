## Summary

The ticket asks the Devices work group to apply the FHIR `mustSupport` flag to
selected elements in the Point-of-Care Device (PoCD) implementation guide —
specifically calling out the `identifier` elements on the DeviceComponent
profiles from the 2018-Jan PoCD ballot. Originally filed during the 2018-Jan
PoCD ballot cycle (tracker item #45), the request was once rejected as Not
Persuasive but has since been revived: on the 2026-04-08 Devices Main Call the
work group agreed to reopen the item in preparation for the September 2026
ballot. The ticket carries one cross-reference to a Confluence page (the
2026-04-08 Devices WG agenda/minutes recording that decision) and one Zulip
cross-reference — an automated `fhir_bot` announcement in `#committers >
tracker-item` when the ticket was originally filed in 2018, not a human
discussion thread. No GitHub items are linked.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the FHIR `mustSupport = true` flag to the `identifier` element on the
DeviceComponent-family profiles in the Point-of-Care Devices (PoCD) IG
(`http://hl7.org/fhir/uv/pocd/`) — at minimum the DeviceComponent and any
derived component profiles referenced from
`DeviceComponentProfiles.html`. Include the updated `mustSupport` flags in
the September 2026 ballot of the PoCD IG and document the intended meaning
of `mustSupport` for identifiers (how conformant implementers must behave
when populating, persisting, and surfacing Device identifiers) in the
profile narrative.

#### Justification

Devices WG formally agreed on 2026-04-08 to revisit this item for the
September ballot, and commenter erikmoll (2026-03-25) notes the profiles
have since been updated with `mustSupport` on selected elements including
identifiers — so Option A largely ratifies work already done and ensures
the ballot text is consistent with the profiles. FHIR's core guidance
(`mustSupport` is meaningful only when the IG defines what "support"
entails) requires accompanying narrative, which this disposition makes
explicit. Identifiers are a reasonable candidate because PoCD devices
routinely need to be correlated across systems (serial numbers, asset
tags, UDI), making population and preservation of `identifier` a
practical interoperability requirement.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change `mustSupport` on `identifier` in the base PoCD profiles.
Instead, follow the nuance raised by mcros (2026-04-16) and jrhoads
(2018-03-28): (1) leave `identifier` as-is in the core PoCD profiles, (2)
apply `mustSupport = true` only to the elements for which the WG has a
concrete, documented support semantic — the currently suggested candidates
are `operationalStatus` on DeviceComponent and DeviceMetric — and (3) add a
new section to the PoCD IG explaining the WG's policy for when
`mustSupport` will and will not be asserted, plus the expected implementer
behavior for each flagged element. Implementers that need stricter
identifier semantics can layer a derived profile.

#### Justification

This approach honors the 2018 PoCD Tech consensus (case-by-case
application of `mustSupport`, avoid unnecessary interoperability
constraints) while still delivering the underlying intent — making it
clear which elements a conformant implementation must handle. It avoids
the hazard mcros flagged: an unspecified `mustSupport` on `identifier`
creates ambiguity for implementers. It also leaves room for more targeted
profiling work (e.g. UDI-specific profiles) without committing the base
IG to a blanket rule.

---

### Disposition C: Decline

#### Proposal

Close FHIR-15378 as **Not Persuasive with Modification / Already Addressed**.
Record that: (a) the original 2018 resolution (Not Persuasive, Brian/S. Karl
10-0-0) captured the substantive position that `mustSupport` should be
applied case-by-case and not broadly to identifiers, and (b) per erikmoll
(2026-03-25) the current PoCD profiles already carry `mustSupport` on
selected elements including identifiers, so the outstanding request has
been overtaken by events. No further profile changes are required for the
September 2026 ballot on the basis of this ticket.

#### Justification

The request as literally written ("apply `mustSupport` to identifiers") is
thin on rationale and was rejected in 2018 on the substantive grounds that
`mustSupport` without per-profile semantics risks over-constraining
interoperability. Those grounds have not changed. Meanwhile, the practical
need has reportedly already been met in the current profiles. Reopening
only to re-assert a change that is already present adds churn to the
ballot without delivering new implementer value.

---

### Recommendation

**Recommended disposition:** B

Disposition B best reflects the current signal in the ticket. The 2026-04-08
WG decision re-opened the item because `mustSupport` deserves considered
treatment before the September ballot, but mcros's own 2026-04-16 comment
and the 2018 PoCD Tech consensus are explicit that a blanket `mustSupport`
on `identifier` is not well-motivated and that `mustSupport` without a
profile-level support definition is harmful. Disposition A ratifies a
change the WG has historically hesitated over and would ship `mustSupport`
flags without the accompanying narrative the WG itself has said is
mandatory. Disposition C ignores the fact that the WG has actively chosen
to reopen the item and that there is at least one genuinely promising
candidate element (`operationalStatus`) that the original commenters and
the 2026 reviewers both flagged. Disposition B delivers the substantive
improvement — targeted `mustSupport` flags with documented semantics — and
leaves the `identifier` decision deferrable to concrete use cases and
derived profiles. The WG can override by promoting A if it confirms that
the current `mustSupport` choices on identifiers are backed by clear
profile narrative; if that narrative is not yet written, B is the safer
ballot-ready option.
