## Summary

The HL7 Point of Care Devices (PoCD) IG currently provides a guidance page on
RESTful Transfer but does not provide any equivalent guidance for implementers
who want to use the FHIR Messaging exchange paradigm. The reporter (Javier
Espina) notes that the FHIR core spec page on messaging
(`build.fhir.org/messaging.html`) does not by itself answer
PoCD‑specific questions — in particular, how references and resources should
(or should not) persist across successive message bundles in a streaming /
device‑observation context. The ticket asks the Devices WG to add a Messaging
guidance page to the PoCD IG with examples and explicit guidance on
persistence semantics. The ticket is marked Resolution: **Persuasive**, the
resolution text is "Use the text in the messaging.md file attached by Erik
Moll", and corresponding draft content has already been landed on the IG's
1.1 branch (commits `a2eff20` and `0b2aab4` in `HL7/uv-pocd`, previewable at
the 1.1 branch build link above).

No other Jira tickets surface as cross-references; the only related items are
the two HL7/uv-pocd commits that implement the proposed change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already on the table: merge the `messaging.md` content
authored by Erik Moll (commits `a2eff20` and `0b2aab4` on the
`version-1.1.0` branch of `HL7/uv-pocd`) into the next published version of
the PoCD IG. Concretely:

- Add `input/pagecontent/messaging.md` as a new "Messaging" guidance page
  parallel to `transfer.md`.
- Register the page in `input/PoCD.xml` and add the menu entry in
  `input/includes/menu.xml`.
- Add the cross-link from `input/pagecontent/transfer.md` to the new page so
  readers landing on RESTful Transfer can jump to the Messaging guidance.
- The page must explicitly cover the originally-requested topics: at least
  one worked example, and explicit guidance on whether/how references and
  resources persist (or do not persist) across successive message Bundles
  in a device observation flow.

#### Justification

This is exactly what the ticket requests, the resolution is already marked
Persuasive, the reporter (Javier Espina) has stated on-ticket that "the
content seems OK", and a rendered preview has already been published on the
1.1 branch. Adopting the content closes a real documentation gap that
implementers have asked about — `build.fhir.org/messaging.html` is generic
and does not answer PoCD-specific persistence questions. Following the
existing IG convention (a sibling page to `transfer.md`, wired into
`menu.xml` and `PoCD.xml`) keeps the IG internally consistent and minimises
review overhead.

---

### Disposition B: Alternative Approach — Accept with Editorial Review and Provenance Note

#### Proposal

Accept the change in principle (same files and same place in the IG as
Disposition A), but before merging the page into the published IG version
require:

1. A human editorial pass on `messaging.md` to confirm that every normative
   statement (especially around reference / resource persistence across
   Bundles) is consistent with the FHIR core `messaging.html` rules and
   with the in-flight v2-to-FHIR work (per `mfaughn`'s comment). The author
   explicitly flagged the draft as AI-generated, so this pass should treat
   correctness — not prose quality — as the bar.
2. An explicit scope statement at the top of the page clarifying that this
   guidance is about device-observation messaging in the PoCD IG and does
   *not* speak to v2-to-FHIR messaging, with a forward link to the
   v2-to-FHIR IG once it ballots.
3. At least one fully-worked example (Bundle of `MessageHeader` plus the
   relevant device/observation resources) to be included in
   `input/examples/` and referenced from the page, rather than relying
   solely on inline narrative.

#### Justification

The substance of Disposition A is right, but the draft was AI-generated and
the comment thread surfaces two real coordination concerns: (a) overlap with
the v2-to-FHIR messaging IG, and (b) the absence of an attached worked
Bundle example. A short editorial pass and an explicit scope note address
both without delaying the page beyond the next publication cycle, and they
materially reduce the risk of having to issue an erratum after ballot.

---

### Disposition C: Decline (or Defer)

#### Proposal

Do not add a PoCD-specific messaging guidance page in this cycle. Instead,
add a single short paragraph (and a link) on the existing
`transfer.md` page that points readers to the FHIR core
`messaging.html` and to the (forthcoming) v2-to-FHIR messaging IG, and
close the ticket as "Not Persuasive" or "Persuasive with Modification —
Defer".

#### Justification

The narrowest reading of the request can be satisfied by a pointer rather
than a new IG page: FHIR core already documents the messaging paradigm, and
a dedicated v2-to-FHIR messaging IG is expected to ballot in the same cycle.
Standing up a parallel PoCD-specific page risks duplicating — or worse,
diverging from — guidance that will live elsewhere, and the draft on the
table is AI-generated and not yet editorially vetted.

---

### Recommendation

**Recommended disposition:** **B**

The reporter, the assignee-in-practice (Erik Moll), and the WG chair-thread
participants have already converged on adding the page, and a working draft
plus a rendered preview exist — declining (Disposition C) would discard real
work and leave the documented implementer gap open. However, the draft is
explicitly flagged as AI-generated, the comment thread raises an unresolved
coordination question with the v2-to-FHIR messaging work, and there is no
attached Bundle example in `input/examples/`. Disposition B keeps the
agreed-on outcome (a Messaging guidance page in PoCD, in the same shape and
location as `transfer.md`) while adding two low-cost guardrails — a human
correctness pass and an explicit scope note — that the comment history
already implicitly asks for. This gives the WG a clean path to publish in
the current cycle without committing to draft prose that has not yet been
human-checked against the core spec.
