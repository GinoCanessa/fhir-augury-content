## Summary

This Devices WG change request asks for clarifying text and (optionally) a
new constraint on the `alertDetection` extension's `limitRange`
subextension in the FHIR Extensions Pack. The extension was created by
[FHIR-43708](https://jira.hl7.org/browse/FHIR-43708) ("Add to
DeviceMetric - alertDetection", Applied / Persuasive) which also defined
the `limitRange` subextension; during that work a comment suggested
guidance was needed to discourage `limitRange` when the extension is
applied to `Device` (where `alertCode` is permitted to be absent), but
no such text or constraint was added. The reporter clarifies that the
real intent is to discourage `limitRange` whenever `alertCode` is
absent (not specifically when the context is `Device`), since a numeric
threshold is meaningless without identifying which alert it bounds.
The ticket additionally asks that the `limitRange` definition text —
currently written as if the extension were always on `DeviceMetric` —
be made inclusive of the `Device` context, with explicit guidance that
units, when present, should match `DeviceMetric.unit` and, when absent,
are implied to match it.

## Details

**Description:**

In the discussion of [FHIR-43708](https://jira.hl7.org/browse/FHIR-43708)
(which created the `alertDetection` extension), a comment was made about
whether additional guidance was needed to discourage the `limitRange`
subextension when the extension is applied to `Device`.

The current extension doesn't have text to that effect, but the reporter
doesn't recall a discussion against adding it. He asks the WG to consider
whether additional text is needed.

The reporter believes the intent of the original comment was to
discourage `limitRange` when an alert code is omitted — something which
is only allowed, but not required, when the extension context is
`Device`. Any added text should match the actual intent.

It may be possible to add a constraint that if the context is `Device`,
then `limitRange` must be absent. (Alternatively, with the
"if-alert-code-is-absent" intent, the precondition would be "if alert
code is absent".)

The current `limitRange` definition assumes the extension is applied to
`DeviceMetric`. The reporter does not think that assumption is necessary
and proposes more inclusive language, e.g.:

> "...if the alert activation state were active. The range should be in
> the same units as when the associated value is reported. When this
> extension is applied to DeviceMetric, the units, if present, should
> match `DeviceMetric.unit`; if absent, the units are implied to be the
> same as `DeviceMetric.unit`."

**Comments:**

- **esilver (2025-10-17)** — Proposes a concrete disposition:
  - Add a note: "Since the `limitRange` element of this extension records
    the values outside of which an active alert of the specified
    priority is detected, it may be populated only if an `alertCode` is
    specified. (It doesn't make sense to specify a limit of 65–100°C
    without identifying which alert has that limit.)"
  - Update the `limitRange` definition to: "The range of values outside
    of which an alarm condition of the (single) specified priority would
    be triggered for the specified alert code, if the alert activation
    state were `active`. Limits may include upper or lower values, or
    both. **The range should be in the same units as when the associated
    value is reported. When this extension is applied to DeviceMetric,
    the units, if present, should match `DeviceMetric.unit`; if absent,
    the units are implied to be the same as `DeviceMetric.unit`.**"
    (replacing: "If units are present, they should be the same as the
    DeviceMetric. If absent, units are assumed to be the same as the
    DeviceMetric.")
  - Add constraint **alrtdet-2**: "Subextension limitRange applies only
    to a specific alert code." — `extension('limitRange').exists()
    implies extension('alertCode').exists()`.
- **javier_espina (2026-01-22)** — Notes that re-discovered background
  for the original concern lives in section **1.e.iii** of the
  Confluence page
  [2024-01-16 to 18 FHIR Device Alerting Workshop 2](https://confluence.hl7.org/spaces/HCD/pages/216241953/2024-01-16+to+18+FHIR+Device+Alerting+Workshop+2).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the package of edits the reporter has already drafted in his
2025-10-17 comment to the `alertDetection` extension in the FHIR
Extensions Pack:

1. **Replace the `limitRange` definition text** so it is no longer
   `DeviceMetric`-only and explicitly states the unit rule:
   "The range of values outside of which an alarm condition of the
   (single) specified priority would be triggered for the specified
   alert code, if the alert activation state were `active`. Limits may
   include upper or lower values, or both. The range should be in the
   same units as when the associated value is reported. When this
   extension is applied to DeviceMetric, the units, if present, should
   match `DeviceMetric.unit`; if absent, the units are implied to be the
   same as `DeviceMetric.unit`."
2. **Add a usage note** to the `limitRange` element: "Since `limitRange`
   records the values outside of which an active alert of the specified
   priority is detected, it may be populated only if an `alertCode` is
   specified."
3. **Add invariant `alrtdet-2`** on the `alertDetection` extension:
   - Human: "Subextension `limitRange` applies only to a specific alert
     code."
   - Severity: `error`.
   - Expression:
     `extension('limitRange').exists() implies extension('alertCode').exists()`.

The change is limited to the `alertDetection` StructureDefinition (and
its narrative/intro) in the FHIR Extensions Pack and does not touch
core R6 Device or DeviceMetric.

#### Justification

This precisely matches the reporter's stated intent (constrain on the
*absence of `alertCode`*, not on the `Device` context per se), reflects
the substantive feedback already on the ticket from javier_espina
(pointing back to the Devices Alerting Workshop 2 minutes for the
original rationale), and aligns with the FHIR pattern of using a
FHIRPath invariant rather than narrative-only guidance to enforce a
real semantic dependency between sibling subextensions. The unit
clarification removes an over-specific assumption (that the parent is
always `DeviceMetric`) without breaking existing `DeviceMetric`-context
usage, since the `DeviceMetric.unit` rule is preserved verbatim as a
context-conditional clause.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the textual clarifications from Disposition A (definition rewrite
+ usage note) but **scope the invariant to context rather than to a
sibling extension**, by either:

- (B1) Adding `alrtdet-2` as
  `(%context is DeviceMetric) implies extension('limitRange').empty() or extension('alertCode').exists()`
  — i.e., on `DeviceMetric` `alertCode` is already 1..1 so the rule is
  trivially satisfied, and on `Device` `limitRange` is forbidden
  outright; or
- (B2) Splitting the constraint into per-context profiles: a
  `Device`-context profile that prohibits `limitRange` (`limitRange`
  cardinality `0..0`) and a `DeviceMetric`-context profile that leaves
  it `0..1`, instead of a single cross-context invariant.

#### Justification

Compared with Disposition A this is closer to the *original* workshop
comment (which targeted the `Device` context, not the absence of
`alertCode`) and makes the rule statically checkable from the profile
shape alone, which can be friendlier to validators and to implementers
who reason about the two contexts separately. The trade-off is that B1
re-introduces context-specific logic that Disposition A consciously
avoids, and B2 doubles the artifact count for what is otherwise a
single sub-extension; both are also a step further from the reporter's
stated intent, which explicitly favors keying off `alertCode`
presence rather than the parent resource.

---

### Disposition C: Decline

#### Proposal

Decline the request as **Not Persuasive – existing mechanism is
adequate**: leave the `alertDetection` extension as published, on the
basis that

- the `limitRange` definition's existing wording ("…the same as the
  DeviceMetric") is understood by Devices WG implementers in practice,
- a meaningless `limitRange` without an `alertCode` is implausible
  enough that no implementer is expected to author it, and
- adding an invariant after publication imposes a (small) downstream
  validation cost on existing implementations of the Extensions Pack.

#### Justification

This option is defensible only if the WG concludes that the
clarification is editorial-only, no deployed implementations are
ambiguous in practice, and that the cost of touching a published
Extensions Pack artifact (republication, regression risk for any
downstream IGs that pin the extension) outweighs the benefit. It does
not address the reporter's substantive concern that the current text
is silently `DeviceMetric`-shaped, and it leaves the structural
trap (`limitRange` without `alertCode`) reachable by conformant
instances.

---

### Recommendation

**Recommended disposition:** A

The reporter has done the work of drafting both the narrative and the
FHIRPath, the change is internally consistent (one definition rewrite,
one usage note, one invariant), and it is faithful both to the
re-surfaced workshop rationale (per javier_espina's pointer to the
2024-01 Device Alerting Workshop 2 minutes) and to the actual semantic
problem — that `limitRange` is meaningless without an `alertCode`,
regardless of whether the parent is `Device` or `DeviceMetric`.
Disposition B is a reasonable fallback if the WG prefers
context-shape-driven validation, but it pays in artifact complexity
for a constraint that Disposition A expresses in one line of FHIRPath.
Disposition C should only be selected if the WG positively concludes
the current text is unambiguous in deployed practice, which the
reporter and the workshop history both contradict.
