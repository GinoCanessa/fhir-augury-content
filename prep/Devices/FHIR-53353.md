## Summary

Malcolm Clarke (Devices WG) reports that Section 2.9.7 of the Personal
Healthcare Devices (PHD) FHIR Implementation Guide treats "relative time"
the same as absolute or base-offset time with respect to time
synchronisation. Because relative time is, by definition, not
synchronised to any external reference, the IG should not require — or
even meaningfully accept — a time synchronisation method property for a
device reporting in relative time. The ticket therefore asks that the
time-synchronisation-method property be removed (or marked
not-applicable) for the relative-time case in §2.9.7. The ticket has no
linked or cross-referenced Jira tickets and no Zulip discussion was
located; the only cross-reference is to the Devices WG meeting-minutes
Confluence page where disposition was discussed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In §2.9.7 of the Personal Healthcare Devices (FHIR) IG, change the
guidance so that when a device reports time using the *relative time*
representation, the time synchronisation method property is **not
applicable** and MUST NOT be populated (or, if expressed via a
profile/extension, its cardinality is constrained to `0..0` in the
relative-time case). Update the narrative in §2.9.7 and any associated
mapping/example fragments (e.g., MDS time-capabilities mapping into the
PHD-Device profile) so that:

1. The time synchronisation method property is described as required
   only for absolute and base-offset time.
2. For relative time, the IG explicitly states the property is omitted
   because no external time reference exists to synchronise against.
3. Examples that combine relative time with a synchronisation method
   are removed or corrected.

#### Justification

This matches the underlying IEEE 11073 model on which the PHD IG is
built: relative time is monotonic device-local time with no external
reference, so a "method" of synchronising it to a reference is
semantically meaningless. The change is a technical correction, not a
behavioural one — implementers reading §2.9.7 today must already infer
this, and the lynn_laakso comment trail confirms the Devices WG already
agreed to carry the fix into the next PHD revision rather than dispute
the substance. Removing the property in the relative-time case
eliminates a source of invalid-but-passing instances and aligns the IG
with the source standard.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than forbidding the time synchronisation method property
outright for relative time, retain it in the model but add a normative
constraint and an `invariant` (or `obligation`) in §2.9.7 stating that
when the time representation is `relative`, the synchronisation method
SHALL be `none` (or an equivalent fixed code drawn from the existing
PHD time-sync method value set). Update §2.9.7 narrative accordingly
and add a worked example showing relative time + `method = none`.

#### Justification

This keeps the property structurally uniform across all time
representations, which simplifies parser/validator logic and avoids
introducing a representation-dependent cardinality. It also leaves a
machine-readable assertion in instances ("this device is explicitly
*not* synchronised") rather than relying on absence-as-meaning, which
is generally weaker in FHIR. The trade-off versus Disposition A is a
small amount of redundant data on every relative-time observation, but
that cost is minor compared to the interoperability benefit of a fixed
shape.

---

### Disposition C: Decline

#### Proposal

Close the ticket as *Not Persuasive — already addressed*, on the basis
that §2.9.7 (read together with the surrounding text on time
representations) does not actually mandate a synchronisation method for
relative time, and no implementer has reported a concrete validation
or interoperability failure. Direct the reporter to raise a more
specific change request if a particular sentence, profile element, or
example is misleading.

#### Justification

The ticket description is short and points to a section header rather
than a specific element or constraint that demands the property. If the
IG already treats the synchronisation method as conditional on time
representation elsewhere, the fix may be a narrative clarification that
does not require a normative change. Declining preserves STU2-published
text and avoids editorial churn for a defect that has not been shown to
cause harm in conformant implementations.

---

### Recommendation

**Recommended disposition:** A

The Devices WG has already agreed (per the 2025-11-12 minutes
referenced in the ticket) to defer this fix to the next PHD IG
revision, which is an implicit acceptance that the current text needs
to change — Disposition C is therefore inconsistent with the WG's own
position. Between A and B, Disposition A more faithfully reflects the
IEEE 11073 semantics: the synchronisation-method property is
*meaningless* (not merely *fixed-valued*) for relative time, and FHIR
generally prefers to model meaningless elements as absent rather than
as a sentinel value. Disposition B should be kept on the table only if
the Devices WG concludes that downstream tooling benefits from a
uniform shape; otherwise A is the cleaner technical correction and
matches the ticket as filed.
