## Summary

The reporter (Malcolm Clarke) flags that the worked examples in the
Personal Healthcare Devices (PHD) FHIR IG — specifically section 4.61.1
(also numbered 4.66.1 in another rendering) — are inconsistent with the
text around them: both timestamps in the example are encoded as UTC
("Zulu") `instant`/`dateTime` values, whereas the surrounding narrative
calls for one of them to represent an *absolute* time (i.e., a wall‑clock
time with an explicit local offset, per the IEEE 11073‑10206 / PHD
mapping of "Absolute Time" vs. "UTC time" / "Base Offset Time"). In
other words, the example does not demonstrate the distinction the
section is trying to teach.

No other Jira tickets, GitHub PRs/issues, or Zulip threads cross‑reference
this ticket in the Augury index. The only cross‑reference is to the
Devices Main Call minutes from 2025‑11‑12, where the WG decided to
**defer the fix to the next revision of the PHD IG** (post‑STU2
publication) rather than block the in‑flight publication on it. Lynn
Laakso subsequently re‑opened the ticket post‑STU2 to keep it on the
backlog for the next cycle.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persisted (next-revision) — produce a technical correction to PHD IG
section 4.61.1 / 4.66.1 so that the worked example actually demonstrates
the **Absolute Time** encoding rather than emitting both timestamps in
UTC ("Z") form. Concretely: in the example `Observation` (or whichever
PHD profile the section walks through), change the timestamp that is
described in the surrounding narrative as "absolute time" so that it is
serialised as a `dateTime`/`instant` with an **explicit non-Z timezone
offset** (e.g., `2025-11-05T08:14:32-05:00`), matching the local
wall-clock time the example narrative posits. Leave any timestamp that
is genuinely meant to be UTC (e.g., a "Base Offset Time" reference or
gateway-side `issued`) in `…Z` form, and add a one-sentence caption
calling out which is which so the example visibly contrasts the two
encodings. Apply the same fix to any duplicate of the example that
appears under the alternate section number (4.66.1).

#### Justification

This is the literal request from the reporter, and the WG already
agreed at the 2025-11-12 Devices Main Call that the fix is correct in
substance — they only deferred timing to the next IG revision rather
than reject it. PHD is a domain whose entire value proposition is
faithful round-tripping of 11073's time model (Absolute Time vs.
Base Offset Time vs. UTC time); a teaching example whose JSON
contradicts its own caption actively misleads implementers and is
exactly the class of erratum a "Technical Correction" issue type is
designed for. There is no spec-text change required and no normative
impact — only example narrative + example instances.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only patching the two timestamps in the existing example,
**replace the example with a paired pair of mini-examples** under
section 4.61.1 / 4.66.1: (1) one that shows a measurement carrying an
*Absolute Time* (`dateTime`/`instant` with an explicit local offset)
and (2) one that shows the same measurement carrying a *UTC time*
(`…Z`), with a short table mapping each to the corresponding 11073
attribute (Absolute-Time-Stamp vs. Base-Offset-Time / Time-Stamp-Absolute
in BTLE). Also add a short normative-style note (or strengthen the
existing one) stating that PHD-conformant systems SHALL preserve the
timezone information from the source PHD when populating
`Observation.effective[x]`, and SHOULD NOT silently normalise to UTC.

#### Justification

The defect is fundamentally pedagogical: the original example failed
because a single timestamp in isolation cannot demonstrate the
*distinction* the section is trying to teach. A paired example makes
the contrast self-evident and prevents the same erratum from being
re-introduced when the IG is regenerated. The cost is modest (one extra
small example block plus a sentence of guidance) and it strengthens the
IG against the most common implementer mistake on PHD timestamps —
gateways converting everything to UTC at the gateway layer and losing
the device's local-offset information. The trade-off vs. Disposition A
is more editorial work and a slightly larger diff for the next PHD
publication.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persistent / Not Reproducible** on the grounds that the
section in question is illustrative narrative, the JSON in the example
is syntactically valid FHIR, and PHD already documents the
Absolute-Time vs. UTC-time distinction normatively elsewhere — i.e.,
the example is sub-optimal but not *wrong*, and editorial polish does
not warrant a tracked technical correction.

#### Justification

This is the weakest position and is presented for completeness. It is
defensible only if one treats "the example does not match its caption"
as an editorial nit. In practice the WG has already implicitly
rejected this disposition: they reverted the original "Considered for
Future Use" resolution and explicitly chose to keep the ticket open for
the next revision (per Lynn Laakso's 2026-03-31 comment), which is the
opposite of declining.

---

### Recommendation

**Recommended disposition:** B

The WG has already committed (via the 2025-11-12 Devices Main Call
minutes and the post-STU2 re-open) to fixing this in the next PHD
revision, so Disposition C is off the table in practice. Between A and
B, Disposition B costs only marginally more editorial effort but
delivers the actual teaching value the section was trying to provide:
a side-by-side contrast between *Absolute Time* (with local offset)
and *UTC time* (`…Z`) encodings, anchored to the 11073 attributes they
map to. Given that mis-handling of PHD timestamps (gateways
normalising to UTC and dropping local offset) is one of the most common
implementer errors on PHD measurements, the slightly larger diff is
worth it. If the editor's bandwidth is constrained for the next
revision, fall back to Disposition A as a strict subset of B that is
still a complete fix.
