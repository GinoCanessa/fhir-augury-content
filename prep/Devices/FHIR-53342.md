## Summary

The reporter (Malcolm Clarke) is asking the Devices WG to clarify the
narrative text in **Section 3.3 "Observation objects in ACOM"** of the
Personal Healthcare Devices (PHD) Implementation Guide. The current
wording does not adequately explain how the *observation objects* that
exist on a Personal Health Device (PHD) – which are the entities that
*generate* observations – should be represented when those observations
are forwarded through a Personal Health Gateway (PHG). The reporter
notes that mapping PHD observation objects into the PHG can be done
either as a **transient instantiation** (only present while observations
are being relayed) or as a **permanent representation** (kept across
observations so that attribute values that are not repeated in every
measurement can still be carried). The ticket asks the IG to spell out
which approach (or which choice criteria) applies, so that PHG
implementers know how to model the observation object on the gateway
side.

There are no cross-referenced Jira tickets or Zulip threads. The only
external reference is a link in the comment trail to the **2025-11-12
Devices Main Call** minutes, where the WG decided to defer the change
to the next revision of the PHD IG (post STU2 publication).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add clarifying narrative to **Section 3.3 "Observation objects in
ACOM"** of the PHD IG that explicitly addresses how a PHD's single
*observation object* maps onto the PHG side. Concretely:

1. State that for each PHD observation object the PHG SHALL maintain a
   logical representation of that object for the duration of the
   forwarding session.
2. Specify two allowed instantiation patterns and when each applies:
   - **Transient instantiation** – the PHG materialises the object only
     while a measurement is being forwarded; appropriate when every
     reported observation carries the full set of attribute values
     needed for interpretation.
   - **Permanent (session-spanning) instantiation** – the PHG retains
     the object across observations; required when the PHD reports
     attributes only on first publication / on change (e.g., unit,
     reference range, measurement-status flags) and subsequent
     observations rely on those previously-published values.
3. Add a short example illustrating the permanent case (attribute
   reported once at session start, then carried into every
   `Observation` resource the PHG emits).
4. Cross-link the new text to the existing PHG mapping rules in the IG
   so implementers know which Observation/Device elements receive the
   carried attributes.

#### Justification

The reporter is the long-standing PHD IG author and is calling out a
real interoperability gap: without this clarification two conformant
PHG implementations could legitimately drop or duplicate attributes
from a PHD whenever the PHD chooses not to re-emit them on every
measurement. The change is purely **narrative** – no resource
structure, profile, or constraint changes – so it is low-risk and
matches the ticket type (*Technical Correction*). The Devices WG has
already agreed (2025-11-12 minutes) that the change is wanted but
should land in the next PHD IG revision rather than STU2; Disposition A
is the work the WG has effectively pre-committed to.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than describing two allowed patterns, **mandate the permanent
(session-spanning) instantiation** in the PHG and require the PHG to
fill in any non-repeated attributes from the most recently received
value for that observation object. Update Section 3.3 to:

1. State that the PHG MUST keep an internal representation of each PHD
   observation object for the lifetime of the PHD's session with the
   gateway.
2. Require that every `Observation` resource emitted by the PHG carry
   the *effective* attribute values (i.e., last-known values from the
   PHD), regardless of whether the underlying PHD frame repeated them.
3. Optionally describe the transient case only as an *implementation
   optimisation* that is invisible at the FHIR boundary.

#### Justification

This eliminates the ambiguity the reporter is concerned about by
removing the choice altogether: downstream consumers of the
PHG-emitted `Observation` resources see a uniform, fully-populated
object every time, which is the more defensible behaviour for a FHIR
endpoint. Trade-off vs. Disposition A: it is a slightly stronger
normative statement (MUST vs. descriptive) and may surprise existing
PHG implementations that today take the transient-only approach, so
it carries a small backward-compatibility risk and would benefit from
an explicit migration note. It also better aligns with the general
FHIR principle that resource instances should be self-describing.

---

### Disposition C: Decline

#### Proposal

Close the ticket as *Not Persuasive – existing text is sufficient* on
the grounds that the underlying ACOM specification, together with the
existing PHG mapping section, already implies the required behaviour
(the PHG must preserve PHD context across measurements), and that
adding more narrative risks contradicting ACOM rather than clarifying
it.

#### Justification

The mechanism the reporter describes (carrying attributes that are
reported only once across subsequent observations) is arguably already
required by ACOM's own state model, and a FHIR IG should generally
defer to the source standard rather than re-state it. Declining keeps
the IG smaller and avoids the risk of the FHIR text drifting from the
underlying IEEE 11073 / ACOM definitions over time.

---

### Recommendation

**Recommended disposition:** A

The ticket is a *Technical Correction* raised by the IG's primary
domain expert, the requested change is purely narrative, and the
Devices WG has already agreed on the 2025-11-12 main call to make the
change in the next PHD IG revision (the resolution was reverted from
*Considered for Future Use* specifically so the work can land
post-STU2). Disposition B is attractive in principle but turns a
clarification into a normative behavioural change and should be
considered separately if the WG wants to tighten PHG conformance;
piggy-backing it onto a clarification ticket would understate the
impact. Disposition C ignores the fact that a sitting WG member is
telling us the current text is unclear in practice. Accept as
requested, with the caveat that the new text should describe both
instantiation patterns and the criteria for choosing between them as
laid out in Disposition A's proposal.
