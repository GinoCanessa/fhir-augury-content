## Summary

The reporter (Malcolm Clarke) is questioning section 3.3.3 of the Personal
Healthcare Devices (PHD) FHIR IG, "Generating the PHD Reported Timestamp
Identifier". The reporter finds the section unclear: it is not obvious what
this identifier is, what its purpose is, or why it is needed at all given
that FHIR already represents timestamps in a `dateTime` format. In short,
the section reads as "nonsensical" without further context. The assignee
(Erik Moll) has responded that the section is in fact about an identifier
used as the `Observation.identifier` value to support conditional-create
(deduplication) behavior described in detail in the `PhdBaseObservation`
profile, and proposes accept-with-modification text that adds a forward
pointer to the profile's "Conditional-create Identifier" section so the
purpose is no longer left implicit. The Devices WG had previously deferred
the change to the next revision of the PHD IG, but it has been reopened
post-STU2 publication for proper disposition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Take the reporter's question at face value and rewrite section 3.3.3
"Generating the PHD Reported Timestamp Identifier" of the PHD IG so
that it is self-explanatory without requiring the reader to first read
the `PhdBaseObservation` profile. Concretely:

1. Open the section with a one-sentence "What and why": this identifier
   is a value used in `Observation.identifier` (not in any timestamp
   element) for the sole purpose of acting as the selection criterion
   in a conditional-create transaction, so that the same physiological
   measurement is not duplicated on the server when a PHG re-uploads
   it.
2. Explicitly clarify that this identifier has no semantic meaning to
   consumers of the `Observation` and must not be used for
   informational content.
3. Clarify that this is **independent** of how PHD timestamps are
   carried in FHIR (which use `dateTime`), addressing the reporter's
   confusion about timestamp formats.
4. Add a forward link to
   `PhdBaseObservation` →
   "Conditional-create Identifier — Prevention of Data Duplication"
   for the full algorithm.

#### Justification

This is the change the assignee already proposed (accept with
modification). The reporter's confusion is exactly the failure mode
this rewrite fixes: the section name talks about a "Timestamp
Identifier", which sounds like a timestamp, but the content is
actually about an `Observation.identifier` used for deduplication.
That mismatch is a documentation defect, and the ticket type
("Technical Correction") fits a narrative-only fix. Doing it now —
rather than deferring again — also closes the ticket cleanly post-STU2
publication, consistent with lynn_laakso's "re-open post STU2"
comment.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only rewriting section 3.3.3, **consolidate** the
Conditional-create Identifier material so it is defined in exactly one
place and referenced from the others. Specifically:

- Treat `PhdBaseObservation` →
  "Conditional-create Identifier — Prevention of Data Duplication" as
  the single normative description.
- Reduce section 3.3.3 to a short pointer that names the mechanism
  ("Conditional-create Identifier"), states it is carried in
  `Observation.identifier`, states its sole purpose
  (deduplication via conditional create), and links to the
  `PhdBaseObservation` section for the algorithm.
- Rename section 3.3.3 from "Generating the PHD Reported Timestamp
  Identifier" to "Conditional-create Identifier for PHD Observations"
  (or similar) so the heading no longer implies a timestamp value.

#### Justification

The root cause of the reporter's confusion is two-fold: (a) the
heading is misleading, and (b) the same identifier is described in
two places at two different levels of detail, which invites
inconsistency. Disposition A fixes (a) but leaves (b) intact;
Disposition B fixes both, at the cost of slightly larger editorial
churn and a section rename that some readers may have bookmarked.
This is the more durable fix and reduces the chance the same ticket
gets refiled later.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive / Already Addressed**: the section already
states that the identifier "is part of the `Observation.identifier`
whose sole purpose is to perform a conditional create transaction"
and that "the identifier is not to be interpreted by a consumer of
the resource and it should not be used for informational content".
The PHD IG also defines the full mechanism in the
`PhdBaseObservation` profile under "Conditional-create Identifier —
Prevention of Data Duplication". No spec change is required.

#### Justification

This option is defensible only if the WG considers the existing text
sufficient and the reporter's confusion idiosyncratic. It avoids
re-publication churn for a narrative-only issue. However, the fact
that an experienced PHD contributor (Malcolm Clarke) read this section
and was unable to determine its purpose is itself evidence that the
text is not, in practice, self-explanatory — which weakens this
disposition.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses both the surface symptom (the misleading
"Timestamp Identifier" heading that prompted the ticket) and the
underlying duplication of normative material between section 3.3.3
and the `PhdBaseObservation` profile. The assignee's proposal in
Disposition A is correct as far as it goes and is a strict subset of
B; promoting it to a consolidate-and-link edit costs little extra
work in this revision and prevents a near-identical ticket from being
filed again. If the WG prefers the smallest possible diff for
post-STU2, fall back to Disposition A — but Disposition C should be
avoided, because the ticket itself is direct evidence that the
current narrative does not communicate the section's purpose to its
target audience.
