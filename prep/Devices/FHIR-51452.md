## Summary

The reporter (Julian Sass) observes that the Point of Care Devices (PoCD)
IG's `metric-availability` extension currently defines its `value[x]` as
`code` with an **extensible** binding, which conflicts with FHIR core
guidance that elements of type `code` should normally use a **required**
binding (per the R4 Terminologies page). The ticket asks the Devices WG
to either (a) tighten the binding strength to `required`, or (b) change
the value type from `code` to `Coding` so that an extensible binding is
appropriate.

The proposed resolution (Persuasive with Modification) leans on a third
path informed by FHIR R6: in R6 the `DeviceMetric.availability` element
uses **`CodeableConcept`** with an **extensible** binding to the
`metric-availability` value set. The WG plans to align the R4 PoCD IG
extension with that R6 shape (i.e., switch `value[x]` to
`CodeableConcept` while keeping the extensible binding) and to retain
the extension in the IG for the R4 cycle. The intended canonical URL
for the extension is
`http://hl7.org/fhir/StructureDefinition/metric-availability`,
anticipating its publication as a backport extension when R6 ships.

The resolution is tightly coupled to [FHIR-33760](https://jira.hl7.org/browse/FHIR-33760)
("Defining URL (Canonical URL) should be in THO not hl7.org/fhir.4"),
which is already **Applied** and which moved the underlying CS/VS
canonical URLs from `hl7.org/fhir` into THO. That ticket also
established the broader strategy of removing the extension from R6
(where the resource carries the element natively) and using the THO
code system in the R4 IG. FHIR-51452's binding-shape change should be
made on top of the URLs landed by FHIR-33760.

## Details

**Description:**

The Metric availability extension currently defines its value with:

- type: *code*
- binding strength: *extensible*

According to the [FHIR core spec](http://hl7.org/fhir/R4/terminologies.html#required),
for elements of type *code*, a *required* binding strength is recommended:

> *In the standard, this is generally used for elements where the value
> needs to be strictly controlled so that everyone can interpret it
> with confidence. Generally, this is used for elements with type code*

Please consider updating the extension to either:

- change the binding strength to *required* if the set of codes from
  the Metric availability value set is meant to be strictly controlled, or
- change the type from *code* to *Coding*, which allows for extensible
  binding to be more appropriate.

**Comments:**

  resource has an `availability` data element of type `CodeableConcept`
  with extensible binding to

  following this approach in the R4 extension, keeping the extension in
  the IG, with corresponding CS/VS in the IG that use THO URLs —
  anticipating a move of these CS/VS from FHIR Core to THO.

**Resolution Description (Persuasive with Modification):**

In FHIR R6 the `DeviceMetric` resource has an `availability` data
element of type **CodeableConcept** with **Extensible** binding to the
metric-availability value set. The proposed disposition is to follow
this approach in the extension for the R4 PoCD IG and to keep the
extension in the IG for R4. The extension URL will be
`http://hl7.org/fhir/StructureDefinition/metric-availability` (or
similar), anticipating that it will become available there as part of
the "backport" extensions for R6. Corresponding CS/VS in the IG should
use THO URLs (anticipating a move from FHIR Core to THO — see
[FHIR-33760](https://jira.hl7.org/browse/FHIR-33760)).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the binding-strength tightening literally requested by the
reporter: keep `value[x]` typed as **`code`** on the
`metric-availability` extension and change the binding strength from
**extensible** to **required**, bound to the (now THO-rooted) Metric
availability value set landed by FHIR-33760. No changes to the
extension's canonical URL or to the value set / code system content
beyond the URL rebasing already done in FHIR-33760.

#### Justification

This is the most literal reading of the FHIR R4 Terminologies guidance
the reporter cites: elements of type `code` should generally have a
required binding so that all implementers interpret the value
identically. Because the value set is small and Devices-controlled,
locking it down avoids the ambiguity that an extensible `code`
introduces (extensible bindings on `code` are essentially
unimplementable — there is no syntactic room for an out-of-VS code).
It is also the smallest possible diff to the published R4 IG: only the
extension's `binding.strength` flips, and the existing typed `code`
value continues to validate against current consumers.

---

### Disposition B: Alternative Approach (Align with R6 — recommended in the WG resolution)

#### Proposal

Apply the resolution as written: change the extension's `value[x]` from
`code` to **`CodeableConcept`** and keep the binding **extensible**,
bound to the THO-rooted `metric-availability` value set established by
FHIR-33760. Republish the extension under
`http://hl7.org/fhir/StructureDefinition/metric-availability` (or the
agreed backport-pack URL) so that its identity matches the R6
`DeviceMetric.availability` element and a future "R6 backport
extensions" pack can host it without further URL churn. Retain the
extension in the PoCD IG for the R4 release; plan to remove it for the
R6 release where `DeviceMetric.availability` is native.

#### Justification

This aligns the R4 extension with R6's `DeviceMetric.availability`
shape (`CodeableConcept` + extensible binding), making R4↔R6 mapping
mechanical and giving implementers room to convey local availability
codes (and translations / display text) without breaking core
validation — which is exactly the use case that "extensible" exists
for. It is consistent with the reporter's *second* option ("change the
type from code to Coding, which allows for extensible binding to be
more appropriate"), upgraded to `CodeableConcept` to match R6 and to
retain text/translation capability. It also dovetails with the already
applied FHIR-33760 (THO URLs) and with the planned R6 backport-extension
pack, which means the URL change is paid once and the extension's
identity stays stable across the R4→R6 transition.

---

### Disposition C: Decline

#### Proposal

Decline the change for the current PoCD R4 release: leave `value[x]` as
`code` with an extensible binding. Document the discrepancy in the
extension's narrative as known technical debt and defer all
binding-shape changes to the R6 publication, where
`DeviceMetric.availability` is native and the extension will be removed
entirely.

#### Justification

The extension is already in the field on R4 implementations. Both
Disposition A (strength change to required) and Disposition B (type
change to CodeableConcept) are non-backward-compatible at the
StructureDefinition level: A rejects any instance carrying an
out-of-VS code, and B changes the JSON/XML shape of the extension's
value. Given that R6 obsoletes the extension entirely, it is
defensible to absorb the R4 inconsistency rather than churn existing
implementers twice (once for the R4 fix, once again at R6).

---

### Recommendation

**Recommended disposition:** B

Disposition B is the disposition the WG has already resolved to
(Persuasive with Modification) and it is the right call. The reporter's
core complaint — an extensible binding on a `code`-typed element is
inconsistent with FHIR core guidance — is best resolved by changing the
*type*, not the *strength*, because the underlying use case (vendor /
local availability codes plus display text) genuinely benefits from
extensible behavior. Aligning on `CodeableConcept` rather than `Coding`
matches R6's `DeviceMetric.availability` exactly, which makes R4↔R6
mapping a no-op and lets the extension be hosted unchanged in the
forthcoming R6 backport-extension pack under
`http://hl7.org/fhir/StructureDefinition/metric-availability`. It also
composes cleanly with FHIR-33760's THO URL move: the binding target
already lives at the right canonical, so this ticket only has to
change the extension's value type. Disposition A is implementer-hostile
(an out-of-VS code becomes a hard validation failure) and Disposition C
ships known wrong guidance for the entire R4 lifetime.
