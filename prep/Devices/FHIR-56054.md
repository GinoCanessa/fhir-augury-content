## Summary

This change request asks the Devices work group to correct several
documentation issues in §1.2.3.3 ("The Time Stamp: Observation.effective[x]")
of the *Personal Healthcare Devices (PHD) IG — Consuming the PHD Profile
FHIR Resources* page. The issues were originally reported against the
Point of Care Devices (PoCD) IG via [FHIR-33254](https://jira.hl7.org/browse/FHIR-33254)
and were applied to PoCD; the same wording exists in the PHD IG and now
needs the parallel fix. The reported defects are: (1) a wrong data-type
reference — the sentence "The 'instant' data type is not used" should
read "The 'Time' data type is not used", because the surrounding text
describes timestamps that are an "instant in time" referencing the
FHIR `Time` datatype, not the FHIR `instant` datatype; and (2) an
awkward phrasing — "even though the measurement is taken in Japan"
should be reworded as "even if the measurement was taken in Japan".
The reporter (Erik Moll) has already committed in a comment to fixing
this in the R6 release of the PHD IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the exact narrative edits requested by the reporter to the PHD IG
*"Consuming the PHD Profile FHIR Resources"* page, §1.2.3.3 "The Time
Stamp: Observation.effective[x]":

1. Replace the sentence "The 'instant' data type is not used" with
   **"The 'Time' data type is not used"** so the wording correctly
   references the FHIR `Time` datatype rather than the FHIR `instant`
   datatype.
2. Reword the example clause "even though the measurement is taken in
   Japan" to **"even if the measurement was taken in Japan"** in the
   subsequent example paragraph.
3. While editing the section, sanity-check every other use of "instant
   in time" in §1.2.3.3 to make sure each one refers to a point on the
   time continuum (the colloquial sense) and not to the FHIR `instant`
   datatype, to avoid re-introducing the same confusion the reporter
   flagged. Disposition: **Persuasive — apply for the next PHD IG
   release (the reporter has committed to R6).**

#### Justification

This is a straightforward technical correction in IG narrative —
exactly the resolution category that was already used for the
identical wording in the PoCD IG via [FHIR-33254](https://jira.hl7.org/browse/FHIR-33254)
(Resolution: Persuasive; Status: Applied). The current PHD wording is
demonstrably misleading: it uses the lowercase phrase "instant in
time" colloquially in one sentence and then names the *FHIR datatype
called* `instant` in the very next sentence, which conflates two
different concepts. The reporter is also the active maintainer of the
PHD IG and has already stated in-comment that he will land the fix in
R6, so adopting as-requested has the lowest friction and the highest
alignment with the parallel fix already accepted by the work group.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the *intent* of the request, but replace the targeted edits with
a small editorial rewrite of §1.2.3.3 that:

1. Introduces a one-sentence clarification at the top of §1.2.3.3
   distinguishing the colloquial phrase "instant in time" from the FHIR
   `instant` datatype, e.g., *"In this section the phrase 'instant in
   time' is used in its everyday sense (a single point on the time
   continuum) and is unrelated to the FHIR `instant` datatype, which
   is not used by PHD profiles."*
2. Replaces "instant" with "Time" wherever the FHIR datatype is meant,
   and applies the "even if … was taken" rewording requested in
   Disposition A.
3. Adds (or links to) the resolved value-set / datatype table in the
   Observations section so that the reader can see at a glance which
   FHIR primitive types PHD actually allows for `Observation.effective[x]`
   (`dateTime`, `Period`) and which it explicitly excludes (`instant`,
   `Time`). Disposition: **Persuasive with Modification.**

#### Justification

The reporter's literal change fixes the immediate wording bug, but the
root cause — overloading the word "instant" within a single paragraph
to mean both an English-language concept and a FHIR datatype — is a
foot-gun that will trip the next reader, too. A short editorial
rewrite plus a pointer to the allowed datatypes addresses the
underlying clarity problem rather than just the two flagged sentences.
Trade-off vs. Disposition A: slightly larger diff and one extra round
of editorial review, but it makes the section self-explanatory and
reduces the chance of a future "typos in §1.2.3.3" ticket. It is still
non-normative and does not change any profile, so it remains a
technical-correction-class change rather than a substantive content
change.

---

### Disposition C: Decline

#### Proposal

Mark the change request as **Not Persuasive** on the grounds that
§1.2.3.3 is non-normative narrative whose intent is unambiguous in
context (the reader can tell the difference between the colloquial
"instant in time" and the FHIR `instant` datatype), and reject the
edits as cosmetic. The phrasing "even though the measurement is taken
in Japan" would likewise be left as-is.

#### Justification

There is a defensible-but-weak argument for declining: the section is
informative narrative, no profile constraint is wrong, and the
ambiguity has not, to date, generated a stream of implementer
confusion (only one reported instance via the parallel PoCD ticket).
Declining would also avoid any cost of regenerating the IG for what
is purely a wording polish.

The argument is weak because (a) the same defect was *not* declined in
the parallel PoCD ticket — it was accepted and applied — so declining
in PHD would be inconsistent within the same work group's body of
work, and (b) the reporter, who is the PHD IG editor, has already
volunteered to do the fix.

---

### Recommendation

**Recommended disposition:** A

This is a tiny, low-risk, narrative-only correction in an IG that the
work group already accepted as a real defect on the sister
specification ([FHIR-33254](https://jira.hl7.org/browse/FHIR-33254) was
Persuasive / Applied against PoCD). The PHD IG editor, Erik Moll, has
explicitly committed to making the fix in R6, so the disposition that
matches both established work-group practice and stated maintainer
intent is **Accept As Requested**. Disposition B (the broader
editorial rewrite) is a reasonable amendment if the work group wants
to harden the section against future "what does 'instant' mean here"
confusion, but it should be treated as an optional add-on rather than
a reason to delay the literal fix; it can be folded into the same R6
edit. Disposition C should not be selected — declining a narrative
typo fix that the maintainer has already agreed to make and that was
already applied to the parallel PoCD spec would be both inconsistent
and wasteful of the implementer feedback that produced the ticket.
