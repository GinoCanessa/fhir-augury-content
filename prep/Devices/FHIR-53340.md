## Summary

The reporter (Malcolm Clarke) is asking for a technical correction to
Section 3.2 of the Personal Healthcare Devices (PHD) FHIR Implementation
Guide. The current wording frames the IEEE 11073-10206 model as primarily
defining the Personal Health Device (PHD), and asserts that "all PHDs based
upon the IEEE 11073-10206 model … can be handled by the PHG without
requiring the PHG to view the values of the attributes; only the type of
the attribute is used." The reporter argues the broader and more accurate
framing is that 10206 is actually defining the objects and nomenclature
that live in the **Personal Health Gateway (PHG)** — the PHG-side objects
are what get mapped to FHIR, irrespective of the PHD source. The request
is therefore to re-orient the language in Section 3.2 (and any consistent
text downstream) to refer to the PHG rather than the PHD when describing
where 10206 objects are created and mapped to FHIR.

The ticket was originally triaged by the Devices Work Group and resolved
as "Considered for Future Use," deferred to the next revision of the PHD
IG (per the [2025-11-12 Devices Main Call](https://confluence.hl7.org/spaces/HCD/pages/404097645/2025-11-12+Devices+Main+Call+Agenda+and+Minutes)).
It was re-opened post-STU2 publication and is currently in Triaged status
awaiting fresh disposition. There are no cross-referenced Jira tickets,
Zulip threads, or GitHub items in Augury at this time — the only outbound
cross-reference is to the Devices Main Call minutes Confluence page.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit Section 3.2 of the PHD IG narrative so that the description of the
IEEE 11073-10206 object model is anchored on the **PHG** rather than the
**PHD**. Concretely, rewrite the existing sentence ("all PHDs based upon
the IEEE 11073-10206 model … can be handled by the PHG without requiring
the PHG to view the values of the attributes; only the type of the
attribute is used") to something along the lines of:

> The IEEE 11073-10206 model defines the objects and nomenclature that
> are instantiated in the Personal Health Gateway (PHG). Any PHD whose
> data is normalized into 10206 objects in the PHG can therefore be
> mapped to FHIR by the PHG without the PHG having to interpret attribute
> values — only the attribute *type* is required. The mapping is thus
> independent of the specific PHD that produced the data.

Sweep the rest of the narrative (Sections 1–3 in particular, and any
"how PHDs map to FHIR" prose) for the same PHD-vs-PHG conflation and
align the wording so 10206 objects are consistently described as living
in the PHG.

#### Justification

This is exactly the correction the reporter — a long-standing IEEE
11073 SME — is asking for, and it is a pure narrative/technical-correction
change with no schema or conformance impact. The reframing is technically
more accurate: 10206 specifies the gateway-side object model, and FHIR
mappings in this IG operate on those PHG-side objects. The change makes
the IG easier to reconcile with the underlying IEEE specification and
removes a recurring source of reader confusion that has already prompted
at least one ticket. Because it touches only narrative pages, the risk
of regressions in profiles, examples, or generated artifacts is
negligible.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than rewriting Section 3.2 in isolation, treat this ticket as the
trigger for a small, scoped "PHD vs. PHG terminology" pass across the
PHD IG:

1. Add a short subsection (e.g., "PHD, PHG, and the 10206 Object Model")
   early in Section 3 that explicitly defines PHD and PHG, names IEEE
   11073-10206 as the gateway-side object/nomenclature model, and states
   the invariant that FHIR mapping in this IG is performed against
   PHG-side 10206 objects.
2. Replace each PHD/PHG misuse in the existing narrative with a link or
   reference back to that subsection, instead of paraphrasing the
   distinction in every place.
3. Apply the Section 3.2 wording fix from Disposition A as part of this
   pass.

#### Justification

The reporter's underlying concern is not just one sentence but a
systemic conflation of PHD and PHG in the IG narrative. A point fix in
Section 3.2 (Disposition A) addresses the immediate complaint but leaves
the same ambiguity elsewhere, which is likely to generate future
tickets. A small terminology pass anchored on a single authoritative
subsection is more durable, keeps individual sections shorter, and gives
the WG one canonical place to maintain the PHD-vs-PHG framing as the
10206 model evolves. The cost is modest — still narrative-only — but it
requires a wider editorial review than a single sentence change.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive / No Change**. Keep the existing
Section 3.2 wording and document the rationale in the resolution note:
the existing text is intentionally written from the implementer's
("PHD-centric") perspective because that is how most readers approach
the IG, and the technical accuracy concern can be handled by the
existing 10206 normative reference.

#### Justification

This is defensible only if the WG concludes that the current wording is
"close enough" and that the reporter's framing — while technically more
precise — would make the IG harder for typical implementers (device
vendors, app developers) who think first about a PHD and only secondarily
about the gateway. It also avoids re-opening narrative that has already
been balloted and published in STU2. However, the previous WG action
already moved away from a hard decline (it was filed as "Considered for
Future Use" and has since been re-opened), and the change requested is
genuinely a technical accuracy issue rather than a stylistic preference,
so a flat decline is hard to defend on the merits.

---

### Recommendation

**Recommended disposition:** B

The reporter's correction is technically right and the WG has already
signalled (by re-opening post-STU2) that the previous "defer to next
revision" resolution is no longer adequate, so Disposition C is not
viable. Disposition A fixes the immediate sentence and would be a
perfectly reasonable outcome, but the reporter's commentary ("the
greater point is …") explicitly identifies a systemic framing problem,
not a single typo. Disposition B captures that broader intent at modest
incremental cost: one new short subsection plus consistent edits, all
within PHD IG narrative pages with no schema impact. It also gives the
Devices WG a single anchor for future PHD/PHG questions, which should
reduce the rate of similar tickets against the next revision of the
PHD IG.
