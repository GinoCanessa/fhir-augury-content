## Summary

The ticket asks the Devices WG to loosen the binding on a set of code-bearing
elements in the Point-of-Care Device (PoCD) profiles. The original PoCD
profiles fixed `coding.system` to the IEEE 11073 MDC code system on
`Device.type`, `DeviceComponent.type`, `DeviceMetric.type`,
`DeviceMetric.unit`, `Observation.code`, and `Observation.component.code`.
The reporter (Todd Cooper) requests that those fixed-value constraints be
replaced with an **extensible binding** to an 11073 MDC value set so that
implementers can supply an alternative code (e.g., a proposed or vendor-private
11073 code) when the published MDC code system does not yet cover a needed
concept.

The ticket was originally voted "Considered for Future Use" in 2018 and has
since been re-opened for the upcoming September 2026 PoCD ballot. Two recent
comments give conflicting signals: one Devices WG comment (erikmoll, 2026-03-26)
asserts that the bindings to MDC nomenclature are "all extensible now" and that
no further action is needed, while a later comment (mcros, 2026-04-16) reverts
the prior "Considered for Future Use" resolution and queues the item for ballot
preparation per the 2026-04-08 Devices main call decision.

There are no cross-referenced Jira tickets and no related GitHub items. The
only Zulip cross-reference is the original FHIR-bot tracker-item announcement
post; no substantive Zulip discussion has been indexed.

## Proposed Dispositions

Note: All dispositions concern the **Point of Care Devices (PoCD) FHIR**
specification. No GitHub cross-references were surfaced, so no specific
authoring root or repo path can be cited from a `repo-analysis` briefing —
the WG should resolve the change against the current PoCD profile/StructureDefinition
sources used to publish the PoCD IG.

### Disposition A: Accept As Requested

#### Proposal

Update the PoCD profiles so that the following elements use an **extensible
binding** to an IEEE 11073 MDC value set rather than a fixed-value pattern on
`coding.system`:

- `Device.type` (PoCD Device profile)
- `DeviceComponent.type` (only if the profile still includes DeviceComponent;
  see Justification — DeviceComponent is no longer a normative resource and
  its functionality has been folded into `Device`/`Device.definition` since
  R4)
- `DeviceMetric.type`
- `DeviceMetric.unit` (bound extensibly to the 11073 MDC units value set)
- `Observation.code` (PoCD Observation profile)
- `Observation.component.code`

Concretely, in each StructureDefinition, remove any `pattern[x]` /
`fixedUri` constraint that pins `Coding.system` to
`urn:iso:std:iso:11073:10101`, and replace it with an `ElementDefinition.binding`
of `strength = extensible` referencing the appropriate MDC-scoped ValueSet
(e.g., a value set whose includes filter the MDC code system on the
relevant nomenclature partition for each element).

#### Justification

This is exactly the mechanism FHIR provides for the situation the reporter
describes: an extensible binding is the published guidance for "MDC SHALL be
used when a suitable code exists, but implementers MAY use another code when
no MDC code is available." It eliminates the well-known interop trap of a
hard fixed-system constraint, which forces non-conformance whenever an
implementer needs a vendor-private or yet-to-be-published 11073 nomenclature
code. The 2026-04-08 Devices main call already agreed to address this for
the September ballot, and the prior "Considered for Future Use" disposition
has been formally reverted, so accepting is procedurally clean. Accepting
also aligns the PoCD profiles with the binding-strength conventions used
elsewhere in FHIR for terminology-heavy resources (Observation, Condition,
etc.).

---

### Disposition B: Alternative Approach

#### Proposal

Verify the current state of the PoCD profiles first, then apply the minimum
delta needed:

1. For each named element, inspect the published StructureDefinition. Where
   the binding strength is **already extensible** (per the comment from
   `erikmoll` on 2026-03-26), close that element as "already addressed" with
   no normative change, recording the SD URL and binding ValueSet in the
   resolution note.
2. For any element that still carries a `fixedUri` / `pattern` constraint on
   `coding.system`, replace **only that** constraint with an extensible
   binding to the corresponding 11073 MDC value set (as in Disposition A).
3. Drop `DeviceComponent.*` from scope entirely on the basis that
   `DeviceComponent` was retired as a separate resource after STU3; redirect
   that part of the request into the modern `Device` / `Device.definition`
   shape, or note explicitly that PoCD no longer profiles DeviceComponent.
4. Where the IG still publishes "fixed system, free-text display" patterns,
   add an **invariant** or guidance paragraph stating that MDC SHALL be used
   when an applicable MDC code exists, to preserve the original intent that
   motivated the fixed-value pattern.

#### Justification

This addresses the underlying interop need (allow private/proposed 11073
codes when MDC has no suitable concept) while honouring the WG comment that
the bindings may already be extensible — avoiding a no-op normative change
that would still trigger ballot churn and reviewer load. It also avoids
accepting a request that names a deprecated resource (`DeviceComponent`) at
face value, which would otherwise force the WG to interpret what "change
DeviceComponent.type" should mean in the current information model. The
explicit "MDC SHALL be used when one exists" guidance preserves the
original constraint's intent and discourages implementers from defaulting
to vendor-private codes whenever convenient.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Persuasive with Mod / No Change Required** (or
"Already Implemented"), citing the comment from `erikmoll` on 2026-03-26 that
"the mentioned bindings to MDC nomenclature are all extensible now." The
resolution would record the StructureDefinition URLs and binding ValueSet
canonicals demonstrating that each enumerated element on the PoCD profiles
already uses `binding.strength = extensible` against an 11073-MDC value
set, and that the historical `coding.system` fixed-value pattern is no
longer present.

#### Justification

A 2018-era ticket whose substance has been superseded by intervening PoCD
publications should not drive a normative change that re-litigates work
already done. If the bindings are in fact extensible today, accepting the
ticket as a normative change creates ballot noise (line-item votes, comment
threads) without changing the published artifact. Declining as
"already addressed" is defensible if and only if the PoCD profiles can be
shown to satisfy the request — otherwise this disposition is not safe.

---

### Recommendation

**Recommended disposition:** B

The two recent comments are in direct tension: `erikmoll` says the bindings
are already extensible (which would justify Disposition C), while `mcros` has
formally reverted the prior closure and queued the ticket for ballot
preparation (which suggests Disposition A is expected). Disposition B
resolves that tension by making the resolution data-driven: the WG inspects
the current StructureDefinitions, leaves alone the elements that already
satisfy the request, fixes only the ones that don't, and explicitly handles
the awkward `DeviceComponent` element that the original 2018 wording named
but which no longer exists as a standalone resource. This delivers exactly
the implementer benefit the reporter asked for — the ability to use a
private/proposed 11073 code when MDC has no suitable concept — without
re-balloting elements that already carry the desired binding. It also
produces a resolution note that is easy for ballot reviewers and downstream
implementers to verify against the published profiles.
