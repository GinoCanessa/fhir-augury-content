## Summary

The reporter (Terrie Reed) asks the editors of the Point of Care Devices
(PoCD) FHIR specification to reference the **Unique Device Identifier (UDI)
Domain Analysis Model (DAM)** and the **UDI Pattern Implementation Guide**
in the PoCD spec, and to cite related UDI guidance documents in any future
publications. The existing wording the comment was filed against reads
"Editor: UDI Pattern IG and any other IGs.", and the change request is to
upgrade that to an explicit reference to the UDI DAM and the UDI Pattern
IG (and successor IGs). The ticket originated as PoCD ballot comment #26
from the 2018-Jan cycle, was originally dispositioned **Not Persuasive /
Deferred** in 2018-2019, and was **re-opened on 2026-04-01** by motion
(Erik Moll / Monroe Patillo, 13-0-1) for re-disposition. No related Jira
tickets were surfaced through cross-references, and the only Zulip hit is
an automated `fhir_bot` tracker-item announcement (no human discussion).

## Proposed Dispositions

The proposals below are constrained: this ticket has no GitHub
cross-references, so no repo-analysis briefing is in scope. The
implementation surface named below is the published Devices /
Point-of-Care-Devices (PoCD) IG content and the Devices resource
mapping pages on the FHIR core spec; an implementer running these
dispositions should still anchor the actual file edits in the
`HL7/fhir` (core) and the relevant PoCD IG repos via repo-analysis
before opening a PR.

### Disposition A: Accept As Requested

#### Proposal

Honour the 2026-04-01 motion to re-open and the workgroup's stated
"Accepted - will add the requested content" decision. Concretely,
amend the Point of Care Devices (FHIR) specification text that today
reads "Editor: UDI Pattern IG and any other IGs." to a substantive,
reader-facing sentence in the PoCD IG's introductory / references
section that:

1. Names and links the **UDI Domain Analysis Model (UDI DAM)** as a
   normative-informative reference for the UDI concepts used by the
   `Device` and PoCD profiles.
2. Names and links the **UDI Pattern Implementation Guide** as the
   canonical pattern for representing UDI in FHIR (covering
   `Device.udiCarrier`, `Device.identifier`, and the
   issuer/jurisdiction model).
3. Adds a forward-looking clause: "and any subsequent
   HL7-published UDI implementation guides", so the reference does not
   age out as new UDI IGs are balloted.

The references should appear at least in the PoCD IG's
"References" / "Dependencies" page, and a short `udi` callout should
be added near the `Device` resource discussion in the PoCD IG so the
reader is pointed at the UDI DAM/IG before they try to model UDI
inline.

#### Justification

This is exactly what the reporter (Terrie Reed, who is the FDA's
historical UDI subject-matter expert) asked for, and the workgroup has
already voted to accept it (mfaughn, 2026-04-01: "Accepted - will add
the requested content", with a 13-0-1 motion to re-open). It is a
documentation/reference change, not a normative wire-format change, so
it is **Compatible / substantive / Enhancement** as the original
disposition noted, and carries no implementer breakage risk. It also
respects the FHIR design principle of pointing implementers at the
authoritative pattern artifact (the UDI Pattern IG) rather than
re-stating UDI guidance inline in every device-related spec.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent of the request, but broaden the scope along the
lines mcros suggested in the 2026-03-30 comment: rather than only
adding a reference to the UDI DAM and UDI Pattern IG in the PoCD IG,
also surface UDI guidance on the **Device resource mapping pages** in
the FHIR core spec — specifically the **V2**, **DIM (IEEE 11073
Domain Information Model)**, and **SDC MDS Device** mapping pages —
and from each of those pages link out to the UDI DAM / UDI Pattern IG
for the canonical UDI representation. In parallel, add the same
DAM/IG references to the PoCD IG as in Disposition A. This produces a
single, consistent UDI reference appearing wherever a reader is
already looking at a device-mapping context, and avoids leaving UDI
guidance only in the PoCD IG.

#### Justification

mcros's comment makes the substantive point that the UDI mapping is
currently only discoverable via the SDC MDS Device profile mapping,
and that the same content is relevant to the V2 and DIM mappings.
Disposition A addresses the literal ballot comment but leaves that
inconsistency in place; Disposition B uses the re-open as an
opportunity to fix it. The trade-off vs A is scope: it touches the
core-spec mapping pages as well as the PoCD IG, which (a) requires
coordination with whoever owns the Device resource mapping pages in
the core repo, and (b) needs a repo-analysis briefing for `HL7/fhir`
and the PoCD IG repo before edits land — this skill cannot supply
those briefings. Worth doing if the workgroup is willing to absorb
the wider edit; otherwise, fall back to A and file B as a follow-up.

---

### Disposition C: Decline

#### Proposal

Re-affirm the 2018/2019 disposition of **Not Persuasive / Deferred**
on the grounds that the request is editorial only, that PoCD readers
who need UDI guidance can reach the UDI DAM and the UDI Pattern IG
through the HL7 product catalogue, and that adding informative
references is not a sufficient justification to consume Devices WG
ballot bandwidth. Close as **Not Persuasive — Considered for Future
Use**.

#### Justification

This is the easiest path to defend on the basis that nothing in the
PoCD spec is technically wrong without the references; the UDI DAM
and UDI Pattern IG are independently published and discoverable.
However, this disposition flatly contradicts the **2026-04-01 motion
to re-open and the workgroup's recorded "Accepted - will add the
requested content"** position, so its main use here is as the
explicit "do nothing" alternative for the workgroup to weigh against
A and B. Recommending C would also leave the documented inconsistency
mcros raised (UDI mapping only visible from the SDC MDS profile)
unaddressed.

---

### Recommendation

**Recommended disposition:** **B**

The Devices WG has already reached consensus to accept this ticket
(13-0-1 motion to re-open, "Accepted — will add the requested
content"), so Disposition C is effectively off the table. Between A
and B, mcros's 2026-03-30 comment supplies the workgroup's own
rationale for going broader: the UDI references are useful not only
in the PoCD IG but on the V2, DIM, and SDC MDS Device mapping pages,
and addressing them all together produces one coherent UDI story
across the Device-related artifacts. Disposition B implements
Disposition A as a strict subset, so picking B costs the WG nothing
in terms of fulfilling the ballot comment, while also discharging the
cross-mapping concern in a single editorial pass. Before edits begin,
the implementer should run the `repo-analysis` skill against the
`HL7/fhir` core repo and the PoCD IG repo to anchor the actual file
paths and authoring roots — none were available to this prep run
because no GitHub cross-references exist on this ticket.
