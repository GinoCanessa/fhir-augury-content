## Summary

The reporter (Malcolm Clarke) flags that Section 3.3.2.0.1 of the Personal
Healthcare Devices (PHD) FHIR IG instructs implementers to populate the
`CodeableConcept.text` element, but observes that ISO/IEEE 11073‑10101
(the nomenclature standard the PHD IG depends on) does not provide an
agreed human‑readable text string for the codes being represented. The
ticket asks the Devices Work Group to clarify what value should be used
in the `text` field when 10101 does not supply one.

The ticket has been triaged and reverted from a previous resolution
("Considered for Future Use") and was deferred to the next revision of
the PHD IG by the Devices WG (see the 2025‑11‑12 Devices Main Call). One
WG member (Erik Moll) has proposed rejection on the grounds that
`CodeableConcept.text` is optional and may be filled with an
implementation‑defined string. There are no Jira, Zulip, or GitHub
cross‑references; only a single Confluence link to the WG meeting
minutes was found.

### Disposition A: Accept As Requested

#### Proposal

Update Section 3.3.2.0.1 of the PHD IG to provide explicit guidance for
populating `CodeableConcept.text` when ISO/IEEE 11073‑10101 does not
supply a human‑readable string. Concretely, add a normative
sub‑paragraph that states implementations SHOULD populate
`CodeableConcept.text` with the 10101 reference identifier (e.g.,
`MDC_PRESS_BLD_NONINV_SYS`) when no agreed display text exists, and MAY
substitute a locally‑translated label provided the original 10101
reference identifier is preserved in `CodeableConcept.coding.display`.
Cross‑link this guidance from any examples in §3.3.2 that currently
populate `text`.

#### Justification

The reporter is correct that the IG today instructs implementers to
populate `text` without telling them what to put there when 10101 has
no agreed display string. Ambiguity in this slot directly harms
interop: two conformant PHDs could legitimately emit different `text`
values for the same MDC code, defeating the point of the field for
human review. Naming the 10101 reference identifier as the default is
deterministic, round‑trippable, and consistent with how other 11073‑
based profiles (e.g., point‑of‑care device data) handle missing
display text.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mandating a single fallback string, soften §3.3.2.0.1 to
make `CodeableConcept.text` explicitly optional in the PHD context, and
add an *informative* (not normative) note that lists acceptable
strategies when 10101 provides no agreed text:

1. Omit `text` entirely.
2. Populate `text` with the 10101 reference identifier
   (`MDC_PRESS_BLD_NONINV_SYS`).
3. Populate `text` with an implementation‑defined human‑readable label,
   provided the 10101 code is also present in `coding`.

Update any example narratives in §3.3.2 to mirror at least two of these
strategies so implementers can see the spread.

#### Justification

This matches Erik Moll's reading of the FHIR data type: `text` is
optional by design, and the IG should not invent a normative
constraint where the base spec is intentionally permissive. Listing
strategies (rather than picking one) avoids creating a backward‑
incompatible expectation for systems that already populate `text` with
a locally‑translated label, while still answering Malcolm Clarke's
underlying question ("what should I do?"). It also keeps the IG
consistent with general FHIR `CodeableConcept` guidance.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" with the rationale that
`CodeableConcept.text` is optional in base FHIR and may be populated
with any implementation‑defined string (or omitted), and that
§3.3.2.0.1 already conveys this by not constraining the value.
Recommend no spec change; if implementers want guidance, they should
follow base‑spec patterns for `CodeableConcept.text`.

#### Justification

Erik Moll's comment captures this view: §3.3.2 is sufficiently clear
that `CodeableConcept.text` is optional, and adding IG‑level guidance
risks contradicting or duplicating the FHIR core specification. The
Devices WG already deferred this item to a future revision, suggesting
it is not blocking publication, and the cost of a normative change
(re‑ballot, implementer churn) outweighs the benefit of clarifying an
already‑optional field.

---

### Recommendation

**Recommended disposition:** B

The ticket raises a real readability problem — implementers genuinely
do not know what to do — but the WG should not paper over the fact
that base FHIR makes `text` optional, which is the strongest argument
in Disposition C. Disposition B threads that needle: it answers
Malcolm Clarke's question by enumerating the realistic strategies
(including "omit"), it does not invent a normative constraint where
none exists, and it keeps the door open for a future, more opinionated
revision once the WG has implementer feedback. It also aligns with
Lynn Laakso's note that the item was deferred to the next IG revision
rather than rejected outright, signalling that some guidance — just
not a hard rule — is the WG's preferred direction.
