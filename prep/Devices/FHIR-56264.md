## Summary

Erik Moll questions the temporal semantics of the `DeviceAlert` resource.
Several elements — `DeviceAlert.presence` (mandatory boolean), the
`signal.activationState` code (`on | off | paused`), and `signal.presence`
(`on | latched | off | ack`) — are described as conveying the "current"
state, while `occurrence[x]` is an optional, separate timestamp for when
the alert condition occurred or is occurring. Because FHIR resources are
not real-time and a consumer may read a `DeviceAlert` instance minutes,
hours, or years after it was written, the ticket argues it is unclear at
what moment the "current" elements are being asserted, and whether
`presence`/`signal.*` could end up inconsistent with `occurrence[x]`.
The reporter proposes adding a `currentTime` (`dateTime`) element that
asserts the validity of the state-bearing elements at a specific instant,
adding constraints that prevent `presence` from conflicting with
`occurrence[x]`, and adding an `occurrence[x]`-style element to the
`signal` backbone so the timing of the activation/presence combination is
explicit. The ticket also flags an apparent copy/paste error in the
descriptions of the `derived-from` and `signal-presence` SearchParameters,
both of which read "Whether the alert is currently occurring." There are
no cross-referenced Jira tickets, Zulip threads, or GitHub items
associated with FHIR-56264 in the augury index, so the ticket stands on
its own at this point.

## Details

**Description:**

The DeviceAlert seems to assume that it is kept up to date in real-time
since it has a number of elements that use "currently" or just assume the
value is valid for the current moment in time:

- DeviceAlert.presence — Whether the alert condition is currently active
  — mandatory boolean
- DeviceAlert.signal.activationState — `on | off | paused`
- DeviceAlert.signal.presence — `on | latched | off | ack`

In addition there is an optional element `occurrence[x]` — When the alert
condition occurred / is occurring.

How is a consumer of this resource supposed to interpret these elements?
For example, what should be considered "current" if this resource is read
a year or a minute after the `occurrence[x]` time with `presence` being
true or false?

Similarly, is the signal activation state and presence assumed to be that
from the "current" time as well? Or is "current" the latest time the
resource was uploaded to the FHIR server? That would not be a robust
approach and since FHIR is not real-time, this SDC-like state-modelling
seems incomplete in FHIR context.

Some changes to this resource are needed to resolve this IMHO. Something
like:

- Add a `currentTime` element of type `dateTime` that asserts the
  validity of these data elements at that moment in time.
- Add constraints that prevent conflicting values in `DeviceAlert.presence`
  and `DeviceAlert.occurrence[x]`.
- Add an `occurrence[x]` element to the `signal` element structure to
  capture the moment or period that the combination of signal activation
  state, presence, etc., was valid.

Other solutions / clarifications may be possible as well of course.

Also note that the description of the SearchParameters `derived-from` and
`signal-presence` is "Whether the alert is currently occurring", which
seems a copy-paste error.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the structural changes the reporter outlines, scoped to the
`DeviceAlert` resource in FHIR Core:

1. Add a new element `DeviceAlert.currentTime` of type `dateTime`,
   `0..1`, defined as "The instant at which the values of `presence`,
   `signal.activationState`, and `signal.presence` are asserted to be
   correct." Mark the element as `mustSupport`-relevant for real-time
   producers.
2. Add an element `DeviceAlert.signal.occurrence[x]` (`dateTime |
   Period`, `0..1`) capturing the moment or interval over which the
   combination of `signal.activationState` / `signal.presence` /
   `signal.code` was valid, mirroring `DeviceAlert.occurrence[x]` at the
   resource level.
3. Add invariants on `DeviceAlert`:
   - `dal-X`: when `presence = true`, `occurrence[x]` SHALL either be
     absent or its end SHALL NOT be earlier than `currentTime` (i.e. the
     period is not closed in the past).
   - `dal-Y`: when `presence = false`, if `occurrence[x]` is a `Period`
     it SHALL have an `end` (the alert condition is no longer active).
4. Fix the SearchParameter descriptions: change `DeviceAlert-derived-from`
   to "The DeviceAlert from which this DeviceAlert was derived (e.g. an
   aggregated or filtered DeviceAlert)" and change
   `DeviceAlert-signal-presence` to "The signal presence state (`on |
   latched | off | ack`) of this DeviceAlert."

#### Justification

The reporter's analysis is correct: a `boolean` `presence` flag has no
implicit timestamp in a non-real-time exchange, and the only existing
temporal anchor (`occurrence[x]`) is optional and applies to the alert
condition rather than to the state-bearing elements. Adding an explicit
`currentTime` makes the assertion auditable and aligns with the
established FHIR pattern of pairing state with `effective[x]` /
`issued` / `recorded` (e.g. `Observation.effective[x]` /
`Observation.issued`, `Condition.recordedDate`, `Flag.period`). The
`signal.occurrence[x]` addition resolves the parallel ambiguity inside
the backbone. The SearchParameter description fix is plainly editorial
and should land regardless of which structural disposition is chosen.

---

### Disposition B: Alternative Approach

#### Proposal

Solve the temporal-semantics problem without adding new elements:

1. Reuse the existing infrastructure already on every resource:
   `Resource.meta.lastUpdated` and `DeviceAlert.occurrence[x]`. Update
   the narrative on `DeviceAlert.presence`, `signal.activationState`, and
   `signal.presence` to state explicitly that these elements assert the
   state **as of `DeviceAlert.occurrence[x]` (when present) and
   otherwise as of `Resource.meta.lastUpdated`**, and that consumers
   reading the resource at a later time MUST treat the values as
   historical, not as "live" telemetry.
2. Promote `DeviceAlert.occurrence[x]` from `0..1` to `1..1` (or add an
   invariant requiring it whenever `presence = true`) so that there is
   always a defined temporal anchor.
3. Add the same two cross-element invariants from Disposition A
   (`presence` vs `occurrence[x]` consistency).
4. Apply the SearchParameter description fix from Disposition A.
5. For `signal`, add narrative — but not a new element — clarifying that
   the signal state is asserted at the same instant as
   `DeviceAlert.occurrence[x]`. If implementers report this is
   insufficient, revisit `signal.occurrence[x]` in a later cycle.

#### Justification

`DeviceAlert` is currently in an early maturity state, and adding two new
elements (`currentTime`, `signal.occurrence[x]`) before there is broad
implementation experience risks locking in a model that will need
revising. Most of the ambiguity the reporter describes can be removed by
making `occurrence[x]` mandatory (or conditionally mandatory) and
clarifying in the resource narrative which timestamp scopes the
state-bearing elements — an approach FHIR uses elsewhere
(`Observation.effective[x]` already plays this role for `Observation`).
The trade-off is that a single `occurrence[x]` cannot distinguish "when
the alert condition occurred" from "when the signal state was sampled,"
which is a real limitation for devices that latch or acknowledge
asynchronously; if the WG concludes that distinction is essential, A is
preferable.

---

### Disposition C: Decline

#### Proposal

Decline the structural changes and treat FHIR-56264 as a documentation /
errata ticket only:

1. Update the description of `DeviceAlert.presence` and the `signal.*`
   coded elements to make explicit that the value is the state asserted
   by the producer at the time the resource was authored (i.e.
   `Resource.meta.lastUpdated`), and that consumers MUST NOT assume the
   resource is being updated in real time.
2. Fix the two SearchParameter descriptions (`derived-from`,
   `signal-presence`) as in A and B.
3. Take no structural change to the resource.

#### Justification

`DeviceAlert` has only just been introduced and there is no recorded
implementer experience (in this index) reporting actual interoperability
failures arising from the current model. FHIR resources in general carry
state that is "current as of when the producer wrote it"; that is the
universal contract conveyed by `Resource.meta.lastUpdated` and is not
unique to `DeviceAlert`. Adding `currentTime` and
`signal.occurrence[x]` introduces redundancy with `meta.lastUpdated` and
`occurrence[x]`, raises new consistency questions (which timestamp wins
when they disagree?), and is hard to justify without concrete failure
cases. The SearchParameter copy-paste fix is editorial and should ship
regardless.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the real ambiguity the reporter has correctly
identified — there is no defined moment at which `presence` /
`signal.activationState` / `signal.presence` are being asserted — without
introducing two new elements before the WG has implementation feedback to
size them correctly. Making `occurrence[x]` the single, conditionally
mandatory temporal anchor and stating that explicitly in the element
definitions follows the precedent set by `Observation.effective[x]` and
keeps `DeviceAlert` aligned with the rest of the spec. If the WG, after
discussion, decides that the timing of the alert *condition* and the
timing of the *signal* state genuinely need to be separable (for latched
or acknowledged signals), Disposition A is the correct fallback and the
two changes can be added without breaking B's narrative. Disposition C is
not recommended: the reporter's question is answerable only by appealing
to `meta.lastUpdated`, which is exactly the "not robust" answer the
ticket is pushing back on. Regardless of which structural disposition is
adopted, the SearchParameter description fix should land as an editorial
cleanup.
