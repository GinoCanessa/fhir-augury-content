## Summary

The reporter (Malcolm Clarke) flags an inconsistency in section 2.9.7.2 of
the Personal Healthcare Devices (PHD) FHIR Implementation Guide. The
guide currently treats every PHD "capability" as an event that the
Personal Health Gateway (PHG) only reports when the underlying device
indicates the capability is present. The reporter argues that the **time
capability** is a fundamentally different beast: it is a persistent
**state** that a PHD always reports (unless defaults are assumed), and
in practice the PHG does not relay the time capability itself onward —
it consumes the capability locally to know how to talk to the PHD (for
example, to set the device clock). What downstream consumers actually
care about is the **status of the clock** (e.g., whether it is set,
synchronized, drifting), not the existence of the time capability.

The ticket is therefore asking the Devices WG to clarify in section
2.9.7.2 that the "treat every capability as an event" rule does not
apply to the time capability, and to elevate clock **status** as the
information that should be conveyed by the PHG. The ticket has only
one cross-reference: the 2025-11-12 Devices Main Call meeting minutes
on Confluence (page 404097645), where the WG deferred it to the next
revision of the PHD IG. There are no related GitHub items and no
Zulip discussions linked to it.

## Proposed Dispositions

Note: the ticket's only cross-reference is the 2025-11-12 Devices Main
Call meeting minutes on Confluence (page 404097645), where the WG
already chose to defer the change to the next revision of the PHD IG.
The dispositions below take that as the practical baseline and propose
how to land the change when work resumes.

### Disposition A: Accept As Requested

#### Proposal

Update PHD IG section 2.9.7.2 to carve the **time capability** out of
the general "capability is an event" rule. Concretely:

1. In the narrative of 2.9.7.2, add a paragraph stating that the time
   capability is a **persistent state** of the PHD (always reported by
   the PHD unless defaults are assumed) and is normally consumed by
   the PHG itself — for example, to set or correct the device clock —
   rather than relayed onward as an Observation/event by the PHG.
2. Introduce explicit guidance that the information PHGs SHOULD
   surface to downstream FHIR consumers is the **clock status** of the
   PHD (e.g., set / not set, synchronized / unsynchronized, drift,
   reference source) rather than the bare presence of the time
   capability.
3. Where the IG currently illustrates the time capability with an
   event-style example, replace or supplement it with a clock-status
   example (mapped onto the existing Device / DeviceMetric /
   Observation profiles already used by the IG for PHD state).
4. Cross-link 2.9.7.2 to whichever downstream section discusses Device
   metadata / time so the state-vs-event distinction is visible from
   both ends.

#### Justification

This is a Technical Correction filed by a long-standing PHD subject
matter expert against a specific section that, on its own terms,
mis-describes how PHGs actually behave with respect to time. The
existing wording ("most PHGs will only report the capability if the
PHD indicates it has the capability") is factually inaccurate for the
time capability and risks misleading implementers into emitting
spurious capability events. Aligning the narrative with deployed PHG
behavior — and surfacing clock status, which is the information
clinicians and downstream systems actually consume — is consistent
with the IG's general approach of mapping IEEE 11073 PHD semantics
onto FHIR resources without inventing new event semantics. The Devices
WG already accepted, in the 2025-11-12 minutes, that the change is
warranted and should land in the next PHD revision; this disposition
operationalises that decision.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than singling out the time capability as a special case in
2.9.7.2, refactor the section to introduce a **general distinction
between PHD capabilities that are events and PHD capabilities that are
states**, and then list the time capability as the canonical example
of a state capability. As part of this:

1. Add a short subsection (e.g., 2.9.7.2.1 "Event capabilities vs.
   state capabilities") that defines the two classes and gives the
   PHG-side reporting rules for each.
2. Move the "treat each capability as an event" guidance under the
   event-capability heading, so it is no longer presented as a
   universal rule.
3. Document the time capability under the state-capability heading,
   together with the recommendation that PHGs surface **clock status**
   (set / synchronized / drift / reference source) rather than the
   capability itself.
4. Leave room for additional state capabilities to be enumerated in
   future PHD IG revisions without further structural change.

#### Justification

Disposition A fixes the immediate inaccuracy but leaves the IG one
"special case" away from the next state-vs-event question. PHD
specifications already distinguish persistent device state from
transient events at the IEEE 11073 layer, so introducing the same
distinction in the FHIR IG narrative is more faithful to the source
standard and reduces the chance that future tickets re-litigate the
same wording for other state-like capabilities. The cost is a slightly
larger narrative edit and a new sub-heading, but it is still a
narrative-only change with no profile or terminology impact, and it
gives the WG a stable place to document clock-status guidance instead
of bolting it onto an event-centric paragraph.

---

### Disposition C: Decline

#### Proposal

Close the ticket without changing section 2.9.7.2, on the grounds that
(a) the existing wording uses "most PHGs" and therefore already
acknowledges exceptions, and (b) clock status is more naturally a
topic for Device / DeviceMetric guidance elsewhere in the IG than for
2.9.7.2. Resolve as "Not Persuasive" or "Considered for Future Use"
and direct any clock-status work to a separate, scoped enhancement
ticket.

#### Justification

A defensible decline rests on two points: the section's "most PHGs"
language is a hedge that arguably already covers state capabilities,
and the substantive request — surfacing clock status — is a new piece
of guidance, not a correction to the existing paragraph. Bundling a
new guidance item into a Technical Correction risks scope creep and
delays the surrounding STU2 errata. Declining (or splitting the
ticket) keeps 2.9.7.2 stable for STU2 readers while giving clock
status its own discoverable artifact in a future revision.

---

### Recommendation

**Recommended disposition:** B

The Devices WG has already agreed (per the 2025-11-12 minutes) that
the change is warranted and should land in the next PHD IG revision,
which rules out Disposition C as the practical outcome. Between A and
B, the underlying issue is not actually about the time capability in
isolation — it is that 2.9.7.2 conflates two different classes of PHD
capability. Fixing only the time case (Disposition A) leaves the
section structurally mis-shaped and invites the same conversation the
next time a state-like capability surfaces. Disposition B costs only
slightly more narrative work, is consistent with how PHD / IEEE 11073
already separates state from events, and gives clock-status guidance a
durable home. Implementers benefit because the rule they need to apply
is now explicit ("is this capability a state or an event?") rather
than implicit in a list of exceptions.
