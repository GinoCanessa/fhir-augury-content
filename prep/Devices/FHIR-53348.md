## Summary

This ticket against the Personal Healthcare Devices (PHD) FHIR IG flags a
factual overstatement in §2.1.1.3, which currently asserts that "all
timestamps with resolutions finer than a day contain the offset to UTC."
The reporter (MalcolmClarke) points out that this statement assumes the
Personal Health Gateway (PHG) always has a UTC offset available, when in
practice a PHG (or the underlying Continua/IEEE 11073 device) may only
report **UTC time**, **local time**, or **no time at all** — a UTC offset
is not guaranteed. The Devices Work Group reviewed the issue on the
2025-11-12 Main Call (Confluence page 404097645) and elected to defer the
correction to the next revision of the PHD IG; the ticket was then
re-opened post-STU2 publication so the change can be tracked against the
following ballot. There are no other linked Jira tickets, no Zulip
discussion threads, and no GitHub PRs/issues currently cross-referenced
to this item.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revise PHD IG §2.1.1.3 to drop the absolute claim that sub-day timestamps
"contain the offset to UTC" and instead enumerate the three time states a
PHG may actually deliver:

1. UTC time **with** a known offset to local time → represent as a FHIR
   `dateTime`/`instant` carrying the offset (e.g.,
   `2026-04-15T13:22:11-05:00`).
2. UTC time **without** a known offset (PHG knows the absolute instant
   but not the user's local zone) → represent as a UTC-Z `dateTime`
   (`2026-04-15T18:22:11Z`) and document that no local offset was
   available.
3. Local time only (no UTC anchor) → represent without a timezone
   designator and document the ambiguity, or carry the value in an
   extension that flags it as "unzoned local".
4. No time available from the PHG → either omit the timestamp element or
   carry a `dataAbsentReason = unknown` extension on
   `Observation.effective[x]`.

Update the surrounding normative language so that the requirement reads
"Timestamps with sub-day resolution **SHOULD** carry the UTC offset
**when the PHG has it**; if the PHG only has UTC, only local time, or no
time, the timestamp SHALL be represented as described in cases 2–4
above."

#### Justification

This is what the reporter actually asked for and what the Devices WG
already agreed to on the 2025-11-12 Main Call when they deferred it to
the next PHD revision rather than declining it. The current text is
demonstrably wrong against IEEE 11073-10206/Continua device behaviour
(many medical-grade PHGs have no real-time clock with timezone
awareness), so leaving it as-is risks implementers either fabricating
offsets or dropping observations entirely. Aligning the section with
FHIR's own `dateTime`/`instant` rules (which permit `Z`, offset, or
unzoned forms depending on context) is consistent with how
`Observation.effective[x]` is handled elsewhere in the core spec.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than rewriting §2.1.1.3 prose alone, introduce a small structured
mechanism to make the time-quality explicit on the wire: define a PHD
extension (e.g., `phd-time-quality`) on `Observation.effective[x]` (and
on `Device.lastSystemChange` where relevant) with a code drawn from a
new value set covering the four states above (`utc-with-offset`,
`utc-no-offset`, `local-only`, `no-time`). §2.1.1.3 would then be
rewritten to point implementers at this extension and to give the
representation rule per code, instead of an ungrounded SHALL on
"contains the offset to UTC."

#### Justification

The underlying problem is not just textual — downstream consumers
currently cannot distinguish "this timestamp is in UTC because the PHG
knew UTC" from "this timestamp is in UTC because we forced `Z` onto a
local-only value." Disposition A fixes the prose but still leaves the
quality of the time-stamp implicit; receivers have to infer from
context. An extension makes the provenance explicit, is purely additive
(safe for any later STU), and matches the way other Devices-WG IGs
(e.g., PoCD) flag measurement-quality. The trade-off is greater
implementation cost: PHG vendors and downstream FHIR servers all need to
populate/consume the extension, and a new value set has to be balloted.

---

### Disposition C: Decline

#### Proposal

Close FHIR-53348 as **Persuasive with Modification — already addressed**:
note that the WG has already deferred the textual change to the next PHD
revision via the 2025-11-12 minutes, and that no further action is
required against the current STU2 publication. The ticket would be
re-opened (or superseded) at the start of the next PHD ballot cycle as a
ballot-tracking item rather than an active correction.

#### Justification

The Devices WG has explicitly chosen not to apply this change to STU2.
Carrying it as an open Triaged ticket against the published spec implies
work that is not actually planned for the current cycle. Declining (with
a clear note that it will be picked up at the next revision) is
defensible because: (a) the issue is non-breaking text, not a wire
incompatibility; (b) implementers reading STU2 will find unchanged
guidance regardless; and (c) the WG already has the 2025-11-12 minutes
as a durable record of intent. The risk is that the issue gets lost
between cycles if no follow-up ticket is filed against the new PHD
version.

---

### Recommendation

**Recommended disposition:** A

The WG has already, in effect, agreed with the reporter — the only
reason this isn't already in the spec is that STU2 had shipped and they
chose to defer rather than re-open the ballot. Disposition A captures
exactly that intent and gives the editors concrete prose-level guidance
to drop into §2.1.1.3 in the next PHD revision. Disposition B's
extension is attractive but introduces ballotable artefacts (a new
extension and a new value set) that the WG didn't ask for; it should be
considered as a follow-on enhancement once the prose correction lands,
not as a substitute for it. Disposition C understates what the WG
actually decided — the resolution they reverted was "Considered for
Future Use," and they explicitly chose to **defer**, not **decline**, so
a Decline disposition would mis-record their intent. Accept-As-Requested
keeps the ticket aligned with the deferral note and gives the next-cycle
editor an actionable change.
