## Summary

The reporter (Malcolm Clarke) is asking that the Personal Healthcare Devices
(PHD) IG, in §4.10.1 ("timestamp status"), expose the *status* of the
timestamp held by both the Personal Health Device (PHD) and the Personal
Health Gateway (PHG) so that consumers can *inspect* it. In IEEE 11073-based
PHD/PHG behaviour, the device-side time clock can be in a number of states
(e.g., relative time only, set/synchronised, accuracy known, drift suspect,
etc.); today the PHD IG describes the timestamp itself but does not surface a
machine-readable indication of *what state that timestamp is in* on either the
PHD or the upstream PHG. Without that, a downstream system cannot tell
whether a measurement's `effectiveDateTime` was recorded against an
authoritatively-set clock, a free-running relative clock, or a clock whose
sync was lost — which materially affects clinical interpretation.

The ticket carries no explicit Jira links and no related Zulip threads. Its
only cross-reference is to the **2025-11-12 Devices Main Call** Confluence
minutes, where the WG initially elected to defer this request to the next
revision of the PHD IG (after STU2). That deferral was subsequently reverted
and the ticket was re-opened post-STU2 publication for active disposition,
which is the state in which it now sits in front of the work group.

## Proposed Dispositions

> Note: because the ticket has no GitHub cross-references, the dispositions
> below describe the change in terms of the PHD IG narrative section
> (§4.10.1) and the FHIR resources/extensions the PHD IG already profiles
> (Device, DeviceMetric, Observation), rather than naming a specific repo
> path. If the WG accepts a normative change, the editor should confirm the
> authoring location in the PHD IG repository before implementation.

### Disposition A: Accept As Requested

#### Proposal

Add a normative requirement in PHD IG §4.10.1 that both the PHD and the PHG
**SHALL** convey the *status* of their timestamp source so that a consumer
can inspect it. Concretely:

- Define (or adopt) a `time-status` extension on `Device` (and `DeviceMetric`
  where appropriate) carrying a coded value drawn from the IEEE 11073 time
  capability flags — e.g., values along the lines of
  `relative-time-only`, `absolute-time-set`, `synchronised`,
  `sync-lost / drift-suspect`, `unknown`. Bind it to a new
  `TimestampStatus` ValueSet defined in the PHD IG.
- Require the PHG, when forwarding measurements, to populate this status for
  *itself* and to surface (or pass through) the PHD-reported status, so a
  receiver can distinguish "PHG clock is good but PHD clock was free-running"
  from "both clocks were synchronised".
- Update §4.10.1 prose to describe the new element, give an example
  Observation/Device pair showing the status populated on both sides, and
  reference the new ValueSet.

#### Justification

This is what the reporter literally asked for and it closes a real
clinical-safety gap: an `effectiveDateTime` from an unsynchronised PHD clock
is not interchangeable with one from a synchronised clock. The IEEE 11073
device model already carries equivalent "Mds-Time-Info" / time capability
attributes, so an IG-level extension is a faithful FHIR projection rather
than new clinical content. Treating it as a Technical Correction (the type
the reporter chose) is appropriate because the IG today describes the
timestamp without making its provenance inspectable, which is arguably an
omission rather than a new feature.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than defining a PHD-specific `time-status` extension, **reuse existing
FHIR mechanisms** to convey timestamp quality, and clarify §4.10.1 to require
their use:

- Use `Observation.dataAbsentReason` / `Observation.effective[x]` precision
  semantics together with the existing `Device.property` /
  `DeviceMetric.calibration` slots to record clock state, and add a small
  set of `DeviceMetric.calibration` codings (or a `Device.property` slice)
  for "time set", "time synchronised", "time uncertain", scoped both to PHD
  and PHG.
- For the PHG specifically, recommend recording its own clock state via a
  `Provenance` resource attached to the measurement bundle, with
  `Provenance.agent` distinguishing PHD vs PHG and a coded
  `Provenance.signature` / extension for clock state.
- Update §4.10.1 with an explicit pattern (and example bundle) showing how
  these existing slots are profiled in the PHD IG.

#### Justification

This avoids minting a new PHD-only extension for something — clock /
timestamp quality — that is plausibly cross-cutting (lab devices, vitals
monitors, even non-PHD home devices have the same problem). Reusing
`DeviceMetric.calibration` and `Provenance` keeps the PHD IG aligned with
how the rest of FHIR talks about measurement provenance and reduces the
implementer burden of learning yet another extension. The trade-off vs.
Disposition A is expressiveness: `calibration` was designed for measurement
calibration, not strictly for clock state, so reviewers should weigh whether
this is a clean fit or a stretch.

---

### Disposition C: Decline

#### Proposal

Decline the request for STU2.x and document the rationale: the existing
PHD IG already records the timestamp, and consumers that need clock
provenance can derive it from the PHG/PHD `Device` resources and the
transport-level metadata (e.g., the time the PHG received vs. forwarded the
measurement). Keep §4.10.1 unchanged and re-defer to a future major
revision once cross-WG (Devices + Orders & Observations) alignment on a
generic "measurement time quality" pattern exists.

#### Proposal (continued — rationale category)

Category: **Already addressable / cross-WG dependency**. Not declined on
merit, but declined as a *PHD-IG-only* change.

#### Justification

This was the WG's first instinct on 2025-11-12 (defer to next revision), and
there is a defensible argument that PHD shouldn't unilaterally invent a
timestamp-status pattern that other device IGs would then have to harmonise
against. However, the fact that the ticket was explicitly re-opened
post-STU2 publication signals that the WG no longer considers pure deferral
acceptable, which weakens this disposition.

---

### Recommendation

**Recommended disposition:** **A**

The ticket was deliberately re-opened after STU2 was published, which means
the WG has already rejected pure deferral (Disposition C) once. Disposition B
is attractive in principle — reusing `DeviceMetric.calibration` /
`Provenance` would be more idiomatic FHIR — but in practice
`DeviceMetric.calibration` is awkward for clock state, and `Provenance` adds
a resource that PHG implementers do not otherwise need to emit, raising the
bar for what is supposed to be a Technical Correction. Disposition A is
narrowly scoped to §4.10.1, mirrors the IEEE 11073 time-capability semantics
PHDs already implement, and closes the clinical-safety gap (consumers being
unable to tell a synchronised timestamp from a free-running one) that the
reporter raised. The WG should ratify A and ask the editor to draft the
`time-status` extension and `TimestampStatus` ValueSet against the PHD IG,
with a note inviting O&O / Devices cross-review so the same extension can
later be promoted out of the PHD IG if other device IGs want to adopt it.
