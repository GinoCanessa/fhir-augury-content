## Summary

the Point of Care Devices (PoCD) FHIR IG) requests that diagrams — and at

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Confirm the reopened resolution: add use-case diagram(s) and at least one
simple worked example to the Use Cases section of the Point of Care
Devices (FHIR) IG ahead of the September 2026 ballot. Concretely:

- In the PoCD IG's narrative source (the use-cases page that backs the
  rendered "Use Cases" tab), embed at least one diagram per primary use
  case (e.g., a sequence/activity diagram for the device-to-gateway flow
  and one for gateway-to-clinical-system flow), authored as SVG/PNG plus
  source (PlantUML/draw.io) committed alongside the page.
- Add one minimal worked example per use case showing the
  `DeviceMetric`/`Observation`/`Device` resources actually exchanged
  (FHIR JSON in the IG's `examples/` area, surfaced inline on the
  use-case page).
- Cross-link from the new diagrams to the existing 11073-10201 overview
  diagram already added (per the 2026-03-30 comment) so the IG presents
  a consistent visual story.
- Mark the change `Impact: Non-substantive` (consistent with the
  reopened resolution) and roll the ticket into the Devices block vote
  for the September ballot.

#### Justification

This is exactly what the ticket asks for, what the WG already agreed to
on 2026-04-08, and what the 2026-03-30 comment indicates is partially
done (11073-10201 overview diagram). The change is documentation-only
(narrative + examples), labeled `R1NFAN` and explicitly marked
non-substantive, so it is low-risk for ballot. Diagrams + simple
examples are a long-standing pattern across HL7 IGs and demonstrably
help reviewers understand device data flows, which is the friction this
ticket has been calling out since 2018.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the *intent* of the ticket but scope the deliverable to a single
canonical "Devices Information Flow" diagram plus a pointer to a
small set of curated examples that already exist elsewhere in the IG,
rather than authoring a diagram-and-example pair per use case. Specifically:

- Add one overview diagram on the IG's Home/Use Cases landing page
  showing the canonical PoCD actors (medical device, gateway, EHR /
  clinical system, archive) and the resources that flow between them.
- For each existing use-case bullet, add a short "See example: …" link
  to one already-published example resource in the IG (re-using the
  examples that ship today rather than creating new ones).
- Defer per-use-case diagrams to a follow-up ticket if reviewer
  feedback at ballot indicates the single overview is insufficient.

#### Justification

The ticket text is brief ("need a diagram, and simple examples") and
does not actually mandate one diagram per use case. A single overview
plus example pointers delivers most of the comprehension benefit at a
fraction of the authoring and maintenance cost, and avoids the risk
that hand-drawn per-use-case diagrams drift out of sync with the
narrative each time a use case is edited. It also keeps the change
unambiguously non-substantive and minimizes ballot surface area, which
matters given that the September window is close. The WG can always
expand coverage in a future cycle if reviewers ask for more.

---

### Disposition C: Decline

#### Proposal

Re-close the ticket as "Persuasive with Mod / Already Applied" on the
basis that the requested content (a new diagram in the 11073-10201
overview) has already been added per the 2026-03-30 comment, and that
no further use-case diagrams are needed because the existing IG
narrative plus the published examples already cover the use cases
adequately. Do not author additional diagrams or examples in this
ballot cycle.

#### Justification

The ticket dates to January 2018 and was originally dispositioned
"Considered for Future Use" with no objections (9-0-0). The most recent
substantive comment (`mcros`, 2026-03-30) states that the 11073-10201
overview "has a new diagram", which arguably satisfies the original
request. Declining (or marking already-applied) avoids reopening
documentation work for a ticket that has sat dormant for eight years
without a sustained user demand signal — there is no Zulip discussion,
no linked tickets, and no GitHub activity tied to it. If a reviewer
files fresh feedback at ballot, a new, scoped ticket can be created
then.

---

### Recommendation

**Recommended disposition:** A

The Devices WG has already taken a recorded vote on 2026-04-08 to reopen
this ticket and add the requested content for the September 2026
ballot, and partial work (the 11073-10201 overview diagram) has been
applied. Disposition A simply executes that decision in full and
delivers what the original reporter asked for; the change is explicitly
non-substantive (R1NFAN) and the editorial cost is bounded. Disposition
B is reasonable as a fallback if authoring bandwidth is tight, but
under-delivers against a commitment the WG already made on the call.
Disposition C is the weakest option: it would effectively reverse the
2026-04-08 WG decision and leave the IG in the same "diagrams missing"
state that prompted the ticket in the first place.
