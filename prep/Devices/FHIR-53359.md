## Summary

The ticket, filed against the **Personal Healthcare Devices (PHD) FHIR IG**
(section 2.9.12), notes that a *transparent* Personal Health Gateway (PHG) —
i.e., a PHG that simply forwards data observed from an IEEE 11073 Personal
Health Device without enriching the payload — would typically not populate the
`display` or `text` fields on `CodeableConcept` / `Coding` elements that
appear in the resulting FHIR resources. The IG examples and surrounding text
appear to imply (or assume) that those human-readable fields will be present,
which is at odds with what a transparent uploader actually emits. The
reporter is asking the IG authors to clarify, in the example narratives and
any normative text in 2.9.12, that `display` / `text` are optional and may
legitimately be absent in PHG-produced resources.

The ticket has only one cross-reference — to the Devices WG meeting minutes
of 2025-11-12 (Confluence page 404097645) — where the WG decided to defer
the change to the next revision of the PHD IG (post STU2). There are no
related Jira tickets, no related Zulip threads, and no related GitHub items
in Augury's index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next revision of the PHD IG, revise section **2.9.12** (and any
other sections whose examples render `CodeableConcept.text` or
`Coding.display`) so that:

1. The narrative text explicitly calls out that, for a *transparent* PHG,
   `text` and `display` are expected to be **absent** when the source 11073
   protocol does not carry a human-readable string for the code. Add a
   one-paragraph clarification immediately under the 2.9.12 heading.
2. The example resources in 2.9.12 are split (or annotated) into two
   variants:
   - a **transparent-PHG** example that omits `text` / `display` on
     code-bearing elements, and
   - an **enriching-PHG** example that includes them, clearly labelled as
     such.
3. A short editorial sweep is performed across the IG examples to remove
   `text` / `display` from any sample that is supposed to represent a
   transparent-PHG upload, so the examples stop implicitly suggesting that
   those fields are required.

#### Justification

This addresses the reporter's literal complaint and aligns the examples
with the IG's existing normative statement (per **erikmoll**'s comment)
that filling those fields is optional for the uploader. It is a
documentation/example correction only — no profile, value set, or
constraint changes — so it carries no compatibility risk for existing
implementers and matches the WG's decision to defer it to the next PHD IG
revision rather than treat it as out-of-scope.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than rewriting the 2.9.12 examples in place, add a new dedicated
subsection — e.g., **"2.9.12.x Human-readable fields in transparent vs.
enriching PHGs"** — that:

- States the rule once (transparent → omit `text` / `display`; enriching →
  populate from a trusted source such as the 11073 nomenclature),
- Cross-references it from each example caption rather than duplicating
  prose, and
- Optionally reinforces it with a short conformance bullet list (e.g.,
  *"A transparent PHG SHALL NOT invent `display` strings; an enriching PHG
  SHOULD source them from IEEE 11073-10101"*).

Leave the existing example resources unchanged in structure, but annotate
them with a "transparent" / "enriching" label that ties back to the new
subsection.

#### Justification

This keeps the change scoped to additive narrative + labels and avoids
churning every example, which reduces editorial review surface and the
risk of accidentally breaking unrelated examples (cross-resource refs,
ids, narrative round-tripping). It also creates a single normative
"home" for the rule that future tickets can point at, instead of relying
on readers to infer the rule from a pattern in the examples. Trade-off
vs. A: less visually obvious to a casual reader who only scans 2.9.12's
examples without reading the surrounding subsection.

---

### Disposition C: Decline

#### Proposal

Close as **Already addressed by existing IG text** — per
**erikmoll**'s comment, the PHD IG already states that filling
`text` / `display` is optional for the uploader, including the
transparent-PHG case. Mark no change required, with a note pointing
implementers at the existing optionality language.

#### Justification

If the optionality is already stated in the IG, the examples are
illustrative and not conformance-bearing, so the apparent over-population
in 2.9.12 does not actually impose a requirement. Declining avoids
example churn and keeps editorial budget for substantive issues.
Defensible only if the WG is satisfied that no implementer has read the
current 2.9.12 examples as implying that `text` / `display` are
required — which the existence of this ticket suggests is not the case.

---

### Recommendation

**Recommended disposition:** A

The reporter's underlying concern is that the **examples** in 2.9.12 are
sending the opposite signal from the IG's normative optionality language,
and erikmoll's own comment concedes that an editorial sweep of the
text/display elements is warranted. Disposition C ignores that mismatch
and effectively bets that no implementer will mis-read the examples,
which the ticket itself contradicts. Disposition B is a reasonable
middle ground but adds a new subsection without fixing the
specifically-cited examples, which leaves the original complaint only
half-addressed. Disposition A — clarify the prose **and** correct the
examples in the next PHD IG revision — is the smallest change that
actually resolves the reported issue, is consistent with the WG's
existing decision to defer (not decline) the ticket post-STU2, and
introduces no compatibility risk.
