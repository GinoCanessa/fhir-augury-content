## Summary

The submitter, writing from the Personal Healthcare Devices (PHD/PHG)
context, observes that FHIR's `dateTime` primitive requires a numeric
time-zone offset whenever hours and minutes are present (`YYYY-MM-DDThh:mm:ss+zz:zz`,
or the special `Z` suffix for UTC). Personal Health Devices and Gateways
frequently know an absolute UTC instant (for example, when their clock
was synchronized via NTP) but do **not** know the local civil time-zone
offset. The PHD community has been using `-00:00` (negative-zero offset,
which RFC 3339 §4.3 reserves for "time in UTC where the local offset is
unknown") to signal this case, but that convention is not described
anywhere in the FHIR specification. The ticket asks FHIR to explicitly
define and bless a way to represent "UTC without a known local
time-zone", so PHD implementers can quote a normative spec rule rather
than a local convention.

No related Jira tickets, GitHub items, or Zulip threads were surfaced
by the cross-reference index. The only related artifact is the
2025-11-12 Devices WG meeting minutes (Confluence page 404097645) where
the WG decided to revert a prior "Considered for Future Use" resolution
and defer the change to the next revision of the PHD IG.

## Details

**Description:**

The dateTime of FHIR is defined as:

> A date, date-time or partial date (e.g. just year or year + month) as
> used in human communication. The format is YYYY, YYYY-MM, YYYY-MM-DD
> or YYYY-MM-DDThh:mm:ss+zz:zz, e.g. 2018, 1973-06, 1905-08-23,
> 2015-02-07T13:28:17-05:00 or 2017-01-01T00:00:00.000Z. If hours and
> minutes are specified, a time zone SHALL be populated. Seconds must be
> provided due to schema type constraints but may be zero-filled and may
> be ignored at receiver discretion. Dates SHALL be valid dates.

Frequently a PHD and PHG will not have information on time zone, and so
cannot be populated.

There does not appear to be a method to support UTC alone, ie when the
TZ is unknown (eg if acquired from NTP) - we have defined
`YYYY-MM-DDThh:mm:ss-00:00` for this purpose. FHIR should have such a
definition clearly provided in this section.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FHIR core specification, amend the `dateTime` (and, for
consistency, `instant`) primitive type definition to explicitly
recognise the RFC 3339 §4.3 "unknown local offset" convention. Specifically:

- Update the narrative under `datatypes.html#dateTime` (and
  `datatypes.html#instant`) to state that an offset of `-00:00` is a
  valid representation meaning "the instant is in UTC, but the local
  time-zone offset is unknown". Contrast it with `+00:00` / `Z`, which
  mean "the instant is in UTC and the local offset is known to be
  zero".
- Confirm the existing `dateTime` / `instant` regex already allows
  `-00:00` (it does: the offset production is
  `(\\+|-)\\d{2}:\\d{2}`), so no schema change is needed.
- Add an implementer note to PHD-style use cases (devices that know an
  absolute UTC instant from NTP but not a civil offset).

This is a Technical Correction filed against the PHD IG, but the change
properly belongs in FHIR core because `dateTime` / `instant` are core
primitives. The PHD IG would then add a short profile-level note that
references the new core guidance for PHG/PHD time stamps.

#### Justification

The `-00:00` convention is already part of RFC 3339, which FHIR's
`dateTime` definition implicitly inherits. Documenting it costs nothing
in the wire format, breaks no existing instances (every conforming
parser already accepts `-00:00` as a valid offset), and immediately
gives device implementers a quotable rule. It also lines FHIR up with
the broader IETF/standards ecosystem (HTTP `Date`, RFC 9557 extended
date-time, etc.) that already distinguishes "UTC, unknown offset" from
"UTC, known-zero offset".

---

### Disposition B: Alternative Approach

#### Proposal

Leave the FHIR core `dateTime` / `instant` text unchanged and instead
solve the problem inside the PHD IG. Specifically:

- In the next revision of the Personal Healthcare Devices IG, add a
  page (or extend an existing time-handling section) that says: when a
  PHD/PHG knows only a UTC instant and not a civil time-zone offset,
  populate the time stamp as `YYYY-MM-DDThh:mm:ss-00:00`, and treat
  this as semantically distinct from `…Z` / `…+00:00`.
- Optionally, define a small extension (e.g.,
  `http://hl7.org/fhir/uv/phd/StructureDefinition/timezone-unknown`)
  that can be hung off `Observation.effective[x]`,
  `DeviceMetric.measurementPeriod`, etc., to mark a time stamp as
  "offset is `-00:00` because it is unknown, not because it is
  literally zero", for receivers that don't want to rely on the textual
  distinction.
- Reference RFC 3339 §4.3 from the IG so implementers have an upstream
  citation.

This matches what the WG actually decided on the 2025-11-12 call:
defer to the next PHD IG revision rather than push a core change.

#### Justification

The minutes show the WG already chose this direction (deferred to next
PHD IG revision). The PHD IG is the only specification that has a
concrete need today, and IG-level guidance is much faster to publish
than a core-spec change. Keeping the change inside the IG avoids
opening a discussion across every workgroup that uses `dateTime`
(which is essentially all of them). The downside — that
non-PHD implementers won't see the guidance — is mitigated by the fact
that `-00:00` is already syntactically valid and the semantic
distinction matters mainly for device telemetry.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — already covered". Position: FHIR's
`dateTime` regex already accepts `-00:00`, RFC 3339 already defines its
meaning, and FHIR generally inherits RFC 3339 semantics. Implementations
that need to distinguish "UTC, offset unknown" from "UTC, offset known
zero" can do so today without any spec change. If the only concern is
discoverability, the PHD IG (not FHIR core) is the right place to call
it out, because PHD is where the use case lives.

#### Justification

The ticket does not request a behavioural change — only a clarification
in prose. The wire format already supports the distinction, and
RFC 3339 (which FHIR cites/follows) already defines the semantics.
Adding extra prose to a heavily-used primitive type carries a non-zero
risk of confusing the much larger population of implementers who do
not have this problem (e.g., people who interpret `-00:00` as just
another way to spell `Z`). For most receivers the two are
operationally identical.

---

### Recommendation

**Recommended disposition:** B

The Devices WG already recorded this direction on the 2025-11-12 call —
"deferred to next revision of PHD IG" — so the recommendation is to
ratify that decision rather than re-litigate it. Disposition B keeps
the change in the specification that actually has the use case (PHD),
lets the WG ship guidance on its own publication cadence, and preserves
the option of later promoting the convention to FHIR core if other WGs
report the same need. Disposition A (core change) is defensible and
arguably cleaner long-term, but it has a much wider blast radius and
will draw FHIR-I attention to a primitive-type prose change for a
single-IG concern. Disposition C is too dismissive: the lack of
documented guidance is exactly what bit the PHD implementers, and
"RFC 3339 already says so" is unsatisfying when the FHIR text doesn't
echo it. The strong recommendation is therefore B, with a follow-up
hook to revisit core-spec wording if non-Devices WGs raise the same
issue.
