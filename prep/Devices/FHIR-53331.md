## Summary

The reporter (Malcolm Clarke) asks that **§4.11.1 PhdDevice** of the
Personal Healthcare Devices (PHD) FHIR IG also support reporting the
device system identifier as an **IEEE EUI‑64**. The ticket is a
companion to [FHIR-53351](https://jira.hl7.org/browse/FHIR-53351)
("EUI‑64", §2.9), which raises the same support request against the
narrative section of the IG. Both tickets were triaged at the
2025‑11‑12 Devices Main Call and **deferred to the next revision of
the PHD IG**; an earlier resolution of "Considered for Future Use"
was reverted on 2026‑03‑31 by Lynn Laakso so that the issue can be
re‑opened **post STU2 publication**.

Substantively, the underlying support already exists in the IG: PHD
profiles slice `Device.identifier` so that one slice represents an
IEEE EUI‑64 system id (alongside Bluetooth/Ethernet MAC slices), and
[FHIR-48202](https://jira.hl7.org/browse/FHIR-48202) (Persuasive,
Published) documents the expected EUI‑64 hex formatting on both the
profile-consumer page and the PhdBaseObservation profile.
[FHIR-48236](https://jira.hl7.org/browse/FHIR-48236) (Persuasive with
Modification, Published) further loosened the slicing on PhdDevice /
PhgDevice so that the EUI‑64 / MAC slices permit additional codings.
The remaining concern from this ticket is therefore most likely
about **promoting EUI‑64 from "one of several allowed slices" to a
first-class, explicitly required-or-recommended identifier** in
§4.11.1, and ensuring the narrative reflects the IEEE 11073‑20601
requirement that the SYSID be an EUI‑64 with an IEEE‑assigned OUI.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **PHD IG** so that §4.11.1 PhdDevice (and the parallel
PhdDeviceProfile structure definition) explicitly calls out reporting
of the device SYSID as an **IEEE EUI-64**:

- In §4.11.1 narrative, add a paragraph stating that the SYSID
  identifier slice of `PhdDevice.identifier` SHALL/SHOULD be an
  IEEE EUI-64 (24-bit IEEE-assigned OUI + 40-bit manufacturer-defined
  ID, per IEEE 11073-20601:2019 Table 3) when the source device
  follows IEEE 11073-20601, and MAY be EUI-64 when the source follows
  IEEE 11073-10206 (ACOM).
- Reference the canonical `system` URI
  `http://hl7.org/fhir/sid/eui-64` for that slice and link to
  `PhdDeviceProfile.html#system-identifier`.
- Carry the same EUI-64 hex formatting rule already documented under
  FHIR-48202 (capitalized hex, dash-separated per two characters).
- Mirror this change in §2.9 to close the companion FHIR-53351, so
  the conformance narrative and the profile page stay consistent.
- Schedule the change for the **next PHD IG revision** (post STU2),
  matching the Devices WG triage decision recorded on 2026-03-31.

#### Justification

Jacob Andersen's Zulip explanation and IEEE 11073-20601:2019 Table 3
make clear that the SYSID is normatively an EUI-64, so making this
explicit in §4.11.1 simply documents existing reality rather than
introducing new conformance. The slice structure is already in place
(FHIR-48236, Published) and the formatting rules are already
published (FHIR-48202, Published), so this is an additive
clarification, not a breaking change. It also satisfies the WG's
own triage outcome — defer to the next PHD revision, but address —
and is symmetrical with FHIR-53351.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding more normative text in §4.11.1 itself, treat the
EUI-64 support request as a **profile/documentation-clarification**:

- In `PhdDevice` (and `PhgDevice` where applicable), add an
  `identifier:eui-64` slice definition (or tighten the existing
  SYSID slice) with `system = http://hl7.org/fhir/sid/eui-64`,
  `type` patterned to the appropriate ContinuaDeviceIdentifiers /
  HL7 v2-0203 code, and a `^short` / `^definition` that states the
  EUI-64 expectation and OUI requirement.
- Add a worked Example resource on the PhdDevice profile page
  showing an EUI-64 SYSID populated from an IEEE 11073-20601 device.
- Cross-link from §4.11.1 and §2.9 to the new slice + example
  rather than duplicating the normative text inline.

#### Justification

This puts the EUI-64 semantics where implementers actually look
(StructureDefinition + example) and keeps the narrative sections
short. It also avoids mixing -20601 and -10206 conformance language
in §4.11.1, which is already noted in the Zulip thread as a real
conformance distinction. Trade-off vs. A: less visible to a casual
reader of §4.11.1, but more actionable for a profile consumer and
less brittle to renumbering of narrative sections.

---

### Disposition C: Decline (Not Persuasive — Already Supported)

#### Proposal

Decline FHIR-53331 with a rationale of **"already addressed by an
existing mechanism"**, on the basis that:

- `PhdDevice.identifier` already has a slice for the IEEE EUI-64
  SYSID, established by FHIR-48236 (Persuasive with Modification,
  Published).
- EUI-64 hex formatting is already documented on the PhdDevice
  profile page and PhdBaseObservation, established by FHIR-48202
  (Persuasive, Published).
- `PhdDeviceProfile.html#system-identifier` already explains the
  EUI-64 expectation (per the Zulip thread).

Suggest the reporter file a more specific ticket if a concrete gap
remains (e.g., "§4.11.1 narrative does not name EUI-64 although the
profile slice does").

#### Justification

A literal reading of the ticket — "should also support reporting of
EUI-64" — is technically already true for PhdDevice. Declining
avoids editing a published STU2 IG for a clarification that the
profile already permits, and it pushes the reporter to identify the
specific narrative gap instead of triggering a spec change on
ambiguous wording.

---

### Recommendation

**Recommended disposition:** **A (Accept As Requested)** — scheduled
for the next PHD IG revision, executed in the manner described in
Disposition B (slice/example + narrative cross-link).

The Devices WG already chose to defer rather than decline (the
"Considered for Future Use" resolution was deliberately reverted on
2026-03-31 with the note "re-open post STU2 publication"), so
Disposition C is inconsistent with the recorded WG intent. Disposition
A on its own risks being purely cosmetic narrative; Disposition B on
its own risks being invisible to the reporter who is reading
§4.11.1. Combining them — explicit EUI-64 statement in §4.11.1 and a
sharpened slice + example on PhdDevice — directly answers the
reporter's request, leverages the work already done in FHIR-48202 /
FHIR-48236, and stays consistent with the IEEE 11073-20601 normative
position summarized by Jacob Andersen on the `devices` Zulip stream.
FHIR-53351 should be batched into the same PR for §2.9.
