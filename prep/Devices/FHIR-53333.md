## Summary

The reporter (Malcolm Clarke) asks the Personal Healthcare Devices (PHD)
IG to relax the requirement in Section 4.10.1 ("Timestamp coding") that
implementers explicitly convey the timestamp coding (local vs. UTC).
The argument is that this duplication is unnecessary because the FHIR
`dateTime` / `instant` lexical format already conveys offset
information: a value with a `Z` suffix or an explicit `±hh:mm` offset
unambiguously identifies UTC vs. local time. The ticket therefore
proposes removing or softening the IG-level "timestamp coding" obligation
in 4.10.1.

There are no linked Jira tickets and no GitHub items associated with
this ticket. The single cross-reference is to the 2025-11-12 Devices
Main Call agenda/minutes page, where the work group reviewed the issue
and deferred it to the next revision of the PHD IG. The disposition was
subsequently reverted from "Considered for Future Use" and re-opened
post-STU2 publication for Dev WG to address in the next revision.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next revision of the Personal Healthcare Devices IG, edit
Section 4.10.1 ("Timestamp coding") to remove the explicit obligation
that implementers separately encode whether a timestamp is local or
UTC. Replace the prescriptive coding text with a note that the FHIR
`dateTime` and `instant` primitive lexical forms already carry that
information via the trailing `Z` (UTC) or explicit `±hh:mm` offset,
and require that PHD producers populate timestamps with a fully
qualified offset (no naive local-time values). Add a worked example
contrasting `2025-11-05T11:19:49Z` and `2025-11-05T11:19:49+02:00`.

#### Justification

The FHIR core specification's `dateTime` and `instant` types are based
on ISO 8601, and an offset (or `Z`) is sufficient to distinguish UTC
from local time without a separate coding flag. Maintaining a parallel
PHD-specific "timestamp coding" requirement is redundant, increases
implementation burden, and creates a risk of the two sources of truth
disagreeing. Removing the redundancy keeps the PHD IG aligned with
core FHIR primitive semantics and is consistent with the reporter's
observation. The Devices WG has already agreed in the 2025-11-12 call
to defer this to the next revision, indicating in-principle support
for revisiting the requirement.

---

### Disposition B: Alternative Approach

#### Proposal

Keep Section 4.10.1 but reframe it from a "must encode timestamp
coding" rule into guidance that mandates *use of the FHIR primitive's
offset capability* rather than a separate coding mechanism.
Specifically: (a) require all PHD timestamps to be expressed as either
`instant` or `dateTime` with a mandatory time-zone offset (forbidding
naive local timestamps with no offset); (b) where the originating
device provides only naive local time, define a small extension (e.g.,
`phd-timestamp-source-zone`) that records the offset/zone reported by
the device alongside the FHIR-format timestamp; (c) remove any
language that implies a separate "coding" element is needed when the
offset is already known.

#### Justification

This addresses the underlying concern (avoid redundant coding) while
preserving information that some PHD devices genuinely produce —
namely, that a measurement was reported in *device-local* time without
a known offset. Pure removal (Disposition A) would lose that
distinction for legacy devices that only report naive timestamps and
rely on the gateway to attach an offset. An extension is the standard
FHIR mechanism for capturing implementation-specific provenance
without polluting the core primitive, and it leaves the door open for
profiles that only target offset-aware devices to ignore the
extension entirely.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Persuasive with Modification — No Change" (or
equivalent), retaining Section 4.10.1 as written. Add a brief
clarifying note in the IG that the timestamp coding requirement is
intended to capture the *device-reported* time semantics (which can
include naive local time without an offset), and is not a duplicate
of the FHIR primitive's offset.

#### Justification

PHD devices frequently emit timestamps in device-local time with no
embedded offset; the IG-level "timestamp coding" mechanism exists
precisely to capture that the value, as reported by the device, was
*local* even when the gateway later normalises it to UTC for
transport. The FHIR `dateTime` offset reflects the *FHIR-resource*
encoding, not necessarily the device's original semantics. Removing
the requirement risks erasing that provenance distinction and would
be a breaking change for STU2 implementers who already conform to
4.10.1.

---

### Recommendation

**Recommended disposition:** B

Disposition B best balances the reporter's valid observation with the
practical reality of personal healthcare devices. The reporter is
correct that, for any timestamp expressed as a FHIR `dateTime` /
`instant` with an offset, a separate coding element is redundant —
which makes a strict "as-written" reading of 4.10.1 hard to defend.
However, pure removal (Disposition A) would lose information about
devices that *only* produce naive local timestamps, which is a real
class of PHD hardware. Declining outright (Disposition C) ignores
the redundancy the reporter has correctly identified and leaves the
IG out of step with how core FHIR primitives already handle offsets.
Disposition B keeps the PHD IG aligned with FHIR primitive semantics
(no redundant coding when an offset is present) while introducing a
small, optional extension for the legacy "naive local time" case.
The Devices WG has already deferred this to the next IG revision,
which is the natural window to introduce a small structural change
of this kind. The recommendation should be revisited if the WG
identifies additional device classes whose timestamp semantics
cannot be captured by an offset plus a single provenance extension.
