## Summary

The reporter (Malcolm Clarke) objects to a normative requirement in §3.3.2.0.1
of the *Personal Healthcare Devices (PHD) FHIR* IG that compels uploaders to
map IEEE 11073-10101 nomenclature unit codes to their equivalent UCUM codes.
He argues that emitting UCUM should not be a hard requirement on PHD-FHIR
uploaders. The Devices Work Group initially dispositioned this as
*Considered for Future Use* and deferred it to the next revision of the PHD
IG, then re-opened it after STU2 publication so that it can be revisited.
A WG comment from Erik Moll recommends rejecting the change on the grounds
that UCUM is the FHIR-preferred coding system, all currently-used 10101 units
have a defined UCUM equivalent, and requiring the mapping materially improves
downstream interoperability and usability of uploaded observations.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket; the only cross-reference is to the Devices WG
meeting minutes on Confluence (page 404097645, 2025-11-12) where the
deferral was decided.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the normative requirement in PHD-FHIR §3.3.2.0.1 that uploaders
MUST map IEEE 11073-10101 nomenclature unit codes to the equivalent UCUM
code. Specifically, downgrade the conformance verb in the affected
sentence from "requires" / SHALL to SHOULD, so that PHD uploaders that
cannot or do not perform the UCUM lookup may emit `Observation.value[x]`
as a `Quantity` whose `system` and `code` reference the 11073-10101
unit (URN `urn:iso:std:iso:11073:10101`) without also populating a
parallel UCUM-coded representation. Update any conformance language,
narrative examples, and capability statement expectations elsewhere in
the IG that depend on the SHALL.

#### Justification

This is the literal reading of the reporter's request. Mapping logic
inside the uploader adds implementation burden for device gateway and
PHG vendors, and the IG already conveys the underlying measurement
semantics through the 11073-10101 code. FHIR's general guidance is that
UCUM should be used when units are exchanged as `Quantity`, but it does
not categorically forbid other unit code systems, so a SHOULD better
matches the spirit of the base spec while preserving the IG's preference
for UCUM where uploaders can supply it.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the requirement that the *consumed* PHD `Observation.valueQuantity`
ultimately carries a UCUM `system`/`code`, but move the mapping
responsibility off the uploader. Two complementary changes in
§3.3.2.0.1:

1. Allow the uploader to emit only the 11073-10101-coded `Quantity`
   together with the MDC unit code in an extension or
   `Quantity.translation`-style mechanism (or a dedicated
   `valueCodeableConcept`/coding pair on a profile element) when it
   cannot perform the lookup.
2. Define (or reference) a server-side conformance expectation — and an
   IG-published ConceptMap from MDC units to UCUM — so that a conformant
   PHD receiver/post-processor performs the canonicalisation to UCUM
   before the data is exposed to consumers.

#### Justification

This preserves Erik Moll's interoperability argument (downstream
consumers always see UCUM) while removing the burden the reporter
objects to from constrained device-side uploaders, who are the actors
least able to maintain a UCUM mapping table. It also makes the IG's
mapping table normative and reusable rather than implicit, which is
generally better practice than relying on every uploader to
re-implement the same lookup. The trade-off vs. Disposition A is added
spec surface (the ConceptMap and a new receiver conformance section)
and a more complex profile shape; vs. Disposition C it concedes that
the uploader-only requirement is in fact heavier than necessary.

---

### Disposition C: Decline

#### Proposal

Persuasive — Not Persuasive. Retain the existing requirement in
§3.3.2.0.1 that uploaders map 11073-10101 nomenclature unit codes to
their UCUM equivalent before populating `Observation.valueQuantity`.
No spec change.

#### Justification

This mirrors the WG comment from Erik Moll. UCUM is the unit code
system FHIR expects on `Quantity`; for the set of units actually used
by 11073-10101 device observations a complete UCUM mapping already
exists, so the implementation cost is bounded and one-time. Allowing
PHD uploaders to skip the mapping would push that work onto every
downstream consumer and fragment the data: some PHD `Observation`
resources would carry UCUM, others 11073-10101, and consumers would
need either both code systems or a translation step to do basic
arithmetic (e.g., unit conversion, range checks, charting). That
defeats a primary value proposition of producing FHIR from PHDs in the
first place. The ticket presents no concrete use case in which an
uploader cannot perform the mapping.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The reporter's underlying concern — that
the IG places a non-trivial mapping burden on the most resource-constrained
actor (the device-adjacent uploader) — is legitimate, but Disposition A
addresses it by giving up the property Erik Moll correctly identifies as
the chief reason to do this in FHIR at all: that downstream consumers
get a uniform UCUM-coded `Quantity` and don't need to know about
11073-10101 to use the data. Disposition C accepts that cost only because
"the mapping table is small," which is true today but does not justify
embedding the lookup in every uploader implementation forever.
Disposition B keeps the normative UCUM outcome that Devices WG clearly
wants, publishes the MDC-to-UCUM ConceptMap as a first-class IG
artifact, and lets uploaders that can't carry the table defer the
canonicalisation to a receiver. It is also the disposition most
consistent with the WG's prior decision to defer this to the next
revision of the PHD IG, since it requires real spec work rather than a
simple verb downgrade.
