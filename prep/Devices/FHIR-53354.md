## Summary

The reporter (Malcolm Clarke) observes that Section 2.9.7 ("Time
synchronisation") of the **Personal Healthcare Devices (PHD) FHIR IG**
currently treats time synchronisation as a required behaviour for
conformant gateways/devices, but argues that time synchronisation is
not essential to the core value proposition of PHD measurements and
should therefore be **optional** rather than mandatory. The ticket is a
Technical Correction against the PHD IG and is scoped narrowly to the
conformance language in Section 2.9.7.

The Devices WG triaged the item on the 2025-11-12 Devices Main Call and
**deferred it to the next revision of the PHD IG** (a previous
"Considered for Future Use" resolution was reverted on 2026-03-31 and
the ticket was re-opened post STU2 publication). No cross-referenced
Jira tickets, Zulip threads, or GitHub PRs/issues were surfaced; the
only cross-reference is to the Devices WG meeting-minutes Confluence
page that recorded the deferral.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit Section **2.9.7 "Time synchronisation"** of the Personal
Healthcare Devices (PHD) FHIR IG to relax the conformance language so
that time synchronisation between the sensor device and the
gateway/PHG is **OPTIONAL** rather than **SHALL/required**. Concretely:

- Replace any normative `SHALL`/`SHOULD` statements requiring the PHG
  to time-synchronise the personal health device with `MAY`.
- Add a short paragraph stating that, when time synchronisation is not
  performed, the PHG SHALL still record `Observation.effectiveDateTime`
  using its own clock at the time of measurement reception, and SHOULD
  flag (e.g., via an extension or a `Device.property` entry) that the
  measurement timestamp was assigned by the gateway rather than by the
  device.
- Update any examples and the conformance checklist in 2.9.7 to match.

#### Justification

The reporter's observation is concrete and bounded: many PHD use cases
(spot-check vitals, trend monitoring at clinically-relevant resolution
of minutes, not seconds) do not require sub-second device/gateway
clock alignment, and a hard requirement excludes otherwise-conformant
low-cost devices that do not implement an IEEE 11073 time-sync
exchange. Making it optional with a documented fallback (gateway
timestamp + provenance flag) is a small, backward-compatible loosening
that matches what implementers report doing in practice. The Devices
WG already accepted the principle of revisiting this in the next
revision (per the 2025-11-12 minutes), so this disposition simply
codifies that intent.

---

### Disposition B: Alternative Approach

#### Proposal

Keep Section 2.9.7 normatively requiring time synchronisation as the
**default** behaviour, but introduce a clearly-named **conformance
profile / capability flag** that allows a PHG to declare itself as
operating in a "**non-time-synchronised**" mode. Specifically:

- Define an extension on `Device` (or a `Device.property` code in the
  PHD code system) such as `phd-time-sync-mode` with values
  `synchronised` / `gateway-timestamped`.
- In Section 2.9.7, retain the SHALL for synchronised mode but add a
  parallel paragraph: "A PHG MAY operate in `gateway-timestamped`
  mode, in which case it SHALL set the extension and SHALL include a
  `Provenance` entry attributing the timestamp to the gateway."
- Optionally split the conformance checklist into two columns
  (synchronised PHG vs. gateway-timestamped PHG).

#### Justification

This preserves the existing safety/quality posture for clinical use
cases that genuinely depend on accurate device-side timestamps
(continuous monitoring, multi-device fusion, arrhythmia analysis)
while giving low-end and consumer-grade devices a sanctioned escape
hatch. Implementers and reviewers can tell at the resource level
which mode produced a given Observation, which is more useful than a
silent relaxation. Trade-off vs. Disposition A: more spec surface to
maintain (a new extension and Provenance pattern), but better
downstream interoperability because the time-sync state is queryable
rather than implicit.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep Section 2.9.7's existing conformance
language unchanged. Resolution category: **Not Persuasive — existing
mechanism adequate**, with a note that implementers who cannot
perform IEEE 11073 time synchronisation may already attach gateway
timestamps via standard FHIR Provenance and should not need a spec
change.

#### Justification

Time accuracy is foundational to the clinical reliability of
longitudinal device data, and quietly making it optional risks
producing a class of "PHD-conformant" data whose timestamps cannot be
trusted for any time-sensitive analysis. The Continua / IEEE 11073
ecosystem on which PHD is built treats time sync as a design
requirement, and divergence from that baseline at the FHIR-IG layer
would create a conformance gap between the PHD IG and the underlying
device standards. The ticket contains a single-sentence assertion
without supporting use cases or implementer evidence, which is a thin
basis for relaxing a normative requirement.

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach)**

The reporter's underlying concern is real — a hard SHALL on time
synchronisation does exclude legitimate low-end PHD implementations —
but Disposition A relaxes the requirement without leaving any audit
trail in the resulting Observations, which is exactly the kind of
silent quality drift that downstream analytics teams (and certifying
bodies) push back on. Disposition C ignores the implementer signal
and the WG's own decision to revisit this in the next PHD revision.

Disposition B threads the needle: it honours the reporter's request
(time sync is no longer mandatory in all cases), it preserves the
safety properties for use cases that need them, and it makes the
chosen mode explicit and queryable through a small `Device`
extension plus a `Provenance` pattern that is already idiomatic in
FHIR. Because the WG has already deferred this to the next PHD IG
revision, the workgroup has runway to specify the extension and
update Section 2.9.7's checklist properly rather than land a hasty
text change. Reviewers should still confirm with the
IEEE 11073 / Continua liaison that introducing a non-synchronised
conformance mode does not conflict with any upstream device-side
requirement before the change is balloted.
