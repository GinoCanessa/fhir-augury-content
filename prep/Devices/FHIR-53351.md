## Summary

Reporter Malcolm Clarke is asking the Personal Healthcare Devices (PHD) IG
to add explicit support for the IEEE **EUI-64** (64-bit Extended Unique
Identifier) in Section 2.9 of the IG (the *System Identifier* section of
`PhdDeviceProfile`). EUI-64 is the canonical IEEE-administered globally
unique identifier used by Personal/Continua-style Personal Health Devices
(e.g., the IEEE 11073 SYSID), and IG users have been asking for clear
guidance on how to encode it on `Device.identifier` for years.

The ticket is a single sentence ("Section 2.9 It would be helpful to
support EUI-64") with no design proposal attached, so the PHD WG needs
to decide *how* to support it — typically by registering a stable
`identifier.system` URL (the long-discussed
`http://hl7.org/fhir/sid/eui-64`) and documenting its use in the PHD
profile, and/or by adding/extending a coded type in the
`ContinuaDeviceIdentifiers` code system.

There are no linked Jira tickets and no related GitHub items in
FhirAugury's cross-reference index. The only formal cross-reference is
to the **2025-11-12 Devices Main Call** Confluence page, where (per the
most recent comment by Lynn Laakso on 2026-03-31) the WG decided to
**defer to the next revision of the PHD IG** and re-opened the ticket
post-STU2 publication. The previous resolution had been *Considered for
Future Use*.

### [devices > ContinuaDeviceIdentifiers codesystem](https://chat.fhir.org/#narrow/stream/devices/topic/ContinuaDeviceIdentifiers%20codesystem)

Most-recent and most-relevant thread (Jan 2025). Jose Costa Teixeira
asks where the `ContinuaDeviceIdentifiers` code system used in the PHD
device example is defined; Forest links to the published code system at
`hl7.org/fhir/uv/phd/CodeSystem-ContinuaDeviceIdentifiers.html`; **Jacob

**SYSID-type identifiers are normally EUI-64 strings**. This thread is
the clearest recent statement that EUI-64 is already the de-facto
encoding for the PHD `SYSID` identifier type — it just isn't called out

Participants: Jose Costa Teixeira, Forest, Jacob Andersen.

### [IG creation > system values for transport addresses](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/system%20values%20for%20transport%20addresses)

Long thread (2018–2021) that produced the original community consensus
on identifier `system` URLs for IEEE addresses on devices: forms like
`http://hl7.org/fhir/sid/eui-64/zigbee` and `…/eui-48/ethernet` were
proposed by Grahame Grieve, with later debate (Lloyd McKenzie, Peter
Jordan, Ted Klein) about whether these belonged as NamingSystems or
CodeSystems, and how to register them via the HTA / UTG vocabulary
process. **Brian Reinhold** ultimately drove a UTG NamingSystem
submission (tracked under UP-177) so EUI-64-based transport/identifier
systems would have stable, resolvable URLs; the proposal reached
consensus review in May 2021. Conclusion: the community agrees these
identifier systems should exist; the open work has always been to land
them as registered NamingSystems and reference them from the PHD IG.

Participants: Brian Reinhold, Lloyd McKenzie, Grahame Grieve, Ted Klein,
Jessica Bota, Peter Jordan, John Moehrke, Robert McClure, Carol
Macumber.

### [IG creation > Null pointer exception](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/Null%20pointer%20exception)

Earlier (2018) discussion in which Stefan Karl and Brian Reinhold
debate canonical encodings for EUI-64 and EUI-48 ("...it would greatly
improve interoperability if everyone who is encoding EUI-64 or EUI-48
does it the same…"), referencing IEEE's guidance document, and again
asking what it would take to define `http://hl7.org/fhir/sid/eui-64` as
a recognized identifier system. Same theme, same unresolved request:
people want a single, documented way to put an EUI-64 on
`Device.identifier`.

Participants: Stefan Karl, Brian Reinhold, Lloyd McKenzie.

### [implementers > Multiple identifiers](https://chat.fhir.org/#narrow/stream/implementers/topic/Multiple%20identifiers) and [implementers > system values for BT addresses](https://chat.fhir.org/#narrow/stream/implementers/topic/system%20values%20for%20BT%20addresses)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next revision of the PHD IG, extend **Section 2.9 (System
Identifier)** of `PhdDeviceProfile` to explicitly call out **EUI-64** as
a supported identifier and document its encoding:

1. State that the IEEE 11073 `SYSID` value SHALL be expressed as a
   `Device.identifier` whose `system` is the registered EUI-64
   NamingSystem URL **`http://hl7.org/fhir/sid/eui-64`** and whose
   `value` is the canonical hyphen-separated 8-octet form
   (e.g., `01-04-03-02-f0-00-00-00`), matching the pattern Grahame
   Grieve and Brian Reinhold settled on in the 2017–2018 implementer
   discussions.
2. Add a `Device.identifier.type` binding to the appropriate
   `ContinuaDeviceIdentifiers` code (e.g., the existing `SYSID` code) so
   the EUI-64 identifier is unambiguously typed.
3. Add an example fragment to §2.9 showing the EUI-64 identifier
   alongside the existing examples, and link to the IEEE EUI-64
   guidelines document for canonical encoding rules.
4. If `http://hl7.org/fhir/sid/eui-64` is not yet an HL7-registered
   NamingSystem (this is the historical UP-177 work and the WG should
   confirm its current status), file/track a UTG NamingSystem proposal
   so the IG can reference a real, resolvable system URL rather than an
   invented one.

#### Justification

This is exactly what the reporter asked for and aligns with the most
recent statement by Jacob Andersen in the *devices* stream that
"SYSID-type identifiers are normally EUI-64 strings" — the IG already
*assumes* this; the change just makes it normative and example-backed.
It also ratifies the long-running community convention (Grahame Grieve,
Brian Reinhold, Stefan Karl) of using `http://hl7.org/fhir/sid/eui-64`
as the system URL, removing a recurring source of implementer confusion
about how to encode IEEE 11073 SYSID values on `Device.identifier`.
Because the WG has already decided (per the 2026-03-31 comment) to
defer this to the next PHD revision, the change is well-timed.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than singling out EUI-64 in §2.9, generalise the section to
cover the full family of IEEE-administered identifiers used by PHDs and
their transports — **EUI-64**, **EUI-48** (Ethernet/BT MAC), and the
ZigBee/IEEE OUI variants — using a single normative pattern:

1. Define §2.9 as "Globally-unique device and transport identifiers"
   with one paragraph per identifier family (SYSID/EUI-64, BT-MAC/EUI-48,
   ZigBee, USB VID/PID).
2. For each family, name the registered `identifier.system` URL (e.g.,
   `http://hl7.org/fhir/sid/eui-64`, `…/eui-48/ethernet`,
   `…/eui-64/zigbee`) — the exact set Grahame proposed and Brian's
   UP-177 NamingSystem submission was intended to register.
3. Bind `Device.identifier.type` to the matching
   `ContinuaDeviceIdentifiers` codes for each family so the EUI-64
   "ask" is satisfied as part of a coherent identifier story rather
   than as a one-off.
4. Add a single combined example showing a Device with multiple typed
   identifiers (SYSID/EUI-64 + BT-MAC/EUI-48), reusing the example
   pattern from the historical Zulip threads.

#### Justification

The implementer discussions over many years repeatedly hit the same
problem: people want guidance not just for EUI-64 but for the full set
of IEEE addresses they have to put on a Device, and they want them
modelled the same way across implementations. Solving §2.9 only for
EUI-64 leaves the EUI-48 / ZigBee questions open and almost guarantees
follow-up tickets. This costs slightly more authoring effort than
Disposition A but is closer to what the *system values for transport
addresses* and *Multiple identifiers* threads were actually asking for,
and it gives `ContinuaDeviceIdentifiers` a clear role beyond the
current `SYSID` code.

The main risk to weigh: any system URL the IG references must be
registered (HTA/UTG NamingSystem), and the historical UP-177 process
was painful. The WG should confirm the current registration status of
each `eui-*` NamingSystem before committing to listing all of them in
§2.9.

---

### Disposition C: Decline (or close as already-supported)

#### Proposal

Resolve as *Not Persuasive — already supported* and close. Document in
the resolution that:

- The PHD IG already permits any IEEE-style identifier on
  `Device.identifier` via the open `ContinuaDeviceIdentifiers` code
  system; in particular, the `SYSID` typed identifier is in practice an
  EUI-64 (per Jacob Andersen's January 2025 statement on the *devices*
  stream).
- No spec change is required — implementers can already populate
  `Device.identifier` with an EUI-64 value today.
- Any remaining "where do I document the system URL?" question is a
  vocabulary/UTG concern (UP-177), not a PHD IG change.

#### Justification

The literal request ("support EUI-64") is arguably already met: the IG
does not forbid EUI-64, the `ContinuaDeviceIdentifiers` `SYSID` code
exists, and a working example payload can be written today. The
strongest argument for declining is scope hygiene — keeping
NamingSystem/UTG concerns out of the PHD IG and letting the vocabulary
process own them.

The weakness is real, however: declining would not address the
documented implementer pain (multiple Zulip threads, recurring
questions in 2017, 2018, 2021, 2025) about *which* `system` URL to use
and *how* to encode the value. That is why the WG already reverted the
earlier "Considered for Future Use" resolution.

---

### Recommendation

**Recommended disposition: A (with elements of B)**

Disposition A is the right starting point because it directly answers
the reporter's request, lines up with the WG's already-recorded
decision to *defer to the next PHD revision* (so the change can land
without disturbing STU2), and matches the de-facto practice Jacob
Andersen described in January 2025. Disposition C is hard to defend in
light of nearly a decade of repeated implementer questions on Zulip
about how to put an EUI-64 on `Device.identifier`; "it's already
allowed" has not been good enough.

The WG should, however, seriously consider folding **Disposition B**'s
broader treatment into the same revision: if §2.9 is being opened up
anyway, covering EUI-48 and the ZigBee/transport variants alongside
EUI-64 prevents an obvious wave of follow-up tickets and finally cashes
in the work that was started under UP-177. Before either A or B can
ship, the WG must confirm the registration status of
`http://hl7.org/fhir/sid/eui-64` (and any sibling `eui-*`
NamingSystems) so the IG isn't pointing at an unregistered system URL —
that is the single feasibility risk worth flagging in the disposition
discussion.
