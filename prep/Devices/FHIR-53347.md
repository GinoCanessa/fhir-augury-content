## Summary

FHIR-53347 ("MDC LOINC mapping") is a Technical Correction filed by Malcolm
Clarke against the **Personal Healthcare Devices (PHD) FHIR IG**. The
reporter notes that the IG's Section 2.1.1.1 — a table mapping IEEE 11073
MDC nomenclature codes to LOINC codes — is **quite limited** and currently
includes few codes that are actually relevant to PHD devices. The reporter
flags that work is already underway to produce a new, more complete MDC↔LOINC
mapping table aimed specifically at PHD use cases, and the implication is
that the IG's table should be revised (or replaced) once that new mapping
is available.

The ticket has no formally linked Jira tickets and the only cross-reference
captured by Augury is the **Devices Main Call agenda/minutes** Confluence
page (HCD/404097645) for 2025-11-12, where the Devices WG reviewed the
ticket and decided to defer the change to the next revision of the PHD IG
(post-STU2). The previous resolution ("Considered for Future Use") was
reverted on 2026-03-31 so the ticket could be re-opened post-STU2 and
carried forward into the next IG cycle.

## Proposed Dispositions

Per the most recent Devices WG comment on the ticket, the workgroup has
already agreed to defer this to the next revision of the PHD IG and re-open
post-STU2. The dispositions below are framed for that next-revision
review, where the workgroup will need to choose how the MDC↔LOINC mapping
in Section 2.1.1.1 is restructured.

### Disposition A: Accept As Requested

#### Proposal

In the next PHD IG revision, replace the existing MDC↔LOINC mapping table
in Section 2.1.1.1 with the new, expanded PHD-focused MDC↔LOINC mapping
that is being prepared (per the reporter's note that "work is ongoing to
provide a new table for PHDs"). Concretely:

1. Land the new mapping content as a maintained ConceptMap resource (e.g.,
   `ConceptMap/MDC-to-LOINC-PHD`) bound to the IG, with `sourceUri` =
   IEEE 11073 MDC and `targetUri` = LOINC, scoped to the PHD-relevant
   observation types (vital signs, glucose, weight, SpO₂, ECG-derived
   parameters, activity, sleep, etc.).
2. Rewrite Section 2.1.1.1 to render that ConceptMap (rather than a hand-
   maintained markdown table) and to explain the methodology and
   maintenance cadence of the mapping.
3. Where a PHD-relevant MDC code has no LOINC equivalent yet, mark those
   rows explicitly ("no LOINC equivalent at time of publication") so
   implementers do not silently drop them.

#### Justification

This is what the reporter literally asked for and aligns with the WG's
already-recorded direction to address the gap in the next revision.
Promoting the mapping from a static markdown table to a published
ConceptMap brings several benefits: it is machine-consumable by validators
and tooling, it can be versioned independently, and it allows third-party
mapping work (LOINC's PHD-aligned mapping efforts, the IEEE 11073 side,
and existing Devices WG drafts) to be merged in via standard terminology
maintenance rather than ad-hoc edits to the IG. The Zulip background
(Cooper, Reinhold, Andersen, Haas) makes clear that "devices emit MDC,
consumers want LOINC" is the durable interoperability problem this table
exists to solve, so a more complete table is high-value.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than maintaining the MDC↔LOINC mapping inside the PHD IG itself,
**externalize** it: keep Section 2.1.1.1 as a short normative pointer and
publish the actual mapping as a standalone, jointly-stewarded ConceptMap
(e.g., hosted by the Devices WG or co-published with LOINC), referenced
from the IG by canonical URL and version. The PHD IG would then ship with
only:

- A normative requirement that PHD Observation resources whose `code`
  contains an MDC Coding SHOULD also include the corresponding LOINC
  Coding when one is published in the referenced ConceptMap version.
- A pinned canonical reference (e.g.,
  `http://hl7.org/fhir/uv/phd/ConceptMap/mdc-to-loinc|<version>`) so
  consumers can resolve the current authoritative mapping.
- A small illustrative subset of mappings retained inline for readability.

#### Justification

The MDC↔LOINC mapping is genuinely a terminology artifact, not an IG-
specific data model decision. Coupling it tightly to the PHD IG's
publication cycle means the table goes stale every time LOINC adds
device-relevant codes (which happens between IG ballots). An externally
versioned ConceptMap lets the mapping be updated and re-issued without
re-balloting the IG, lets POCD and other Devices specs reuse the same
mapping (the Argo Write thread shows POCD has the same need), and gives
LOINC's own PHD mapping work a natural integration point. The trade-off
versus Disposition A is that implementers must resolve the canonical
reference rather than reading the table inline — but PHD implementers
already deal with terminology resolution, so this is a small added cost.

---

### Disposition C: Decline

#### Proposal

Decline to expand Section 2.1.1.1 in this cycle and instead leave the
existing limited MDC↔LOINC table in place, marking it explicitly as
**informative and non-exhaustive**. Implementers needing a fuller mapping
would be directed to the LOINC- and IEEE 11073-published mappings as they
mature, with no normative dependency added to the PHD IG.

Rationale category: **defer to upstream terminology stewards**.

#### Justification

The reporter notes that "work is ongoing to provide a new table for PHDs"
— that work is being done elsewhere and may not yet be at a stable enough
point to be normatively bound from a published HL7 IG. Embedding (or
referencing) an immature mapping risks the IG having to issue technical
corrections every time the upstream mapping changes. Declining to act now
keeps the IG aligned with what is currently stable, and the WG can revisit
once the external mapping is mature. This is the weakest of the three
options, since the WG has already explicitly agreed to address the gap in
the next revision (per the 2025-11-12 Devices Main Call minutes), but it
is the defensible "do nothing structural" position if the new mapping is
not ready in time.

---

### Recommendation

**Recommended disposition:** **B**

The underlying problem — devices emit MDC codes, downstream systems want
LOINC — is a terminology problem with a multi-decade history (visible in
the 2017 implementers Zulip threads as well as the 2021 Argo Write and
2023 wearables threads). Embedding a hand-maintained mapping table in
Section 2.1.1.1 is what produced the current "quite limited" state the
reporter is complaining about; doing the same thing again with a bigger
table (Disposition A) just defers the same staleness problem to the next
ballot cycle. Publishing the mapping as a versioned, externally-referenced
ConceptMap (Disposition B) addresses the reporter's concern, gives the
Devices WG a single mapping artifact reusable across PHD and POCD, and
matches the WG's already-recorded intent to handle this in the next IG
revision. Disposition C is not recommended because the WG has already
committed in the 2025-11-12 minutes to addressing the gap. If the external
mapping work is not ready in time for the next ballot, the WG can fall
back to Disposition A as a transitional step, but the strategic
recommendation is B.
