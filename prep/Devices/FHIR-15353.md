## Summary

The reporter (Ali Nakoulima, January 2018 PoCD batch, item #20) asks
that `DeviceMetric` carry a first-class `dataType` field rather than
relying on `meta.profile` to communicate the kind of metric data the
resource instance represents. The motivation is implementer ergonomics:
a typed element on the resource is easier to dispatch on than a profile
URL inspected at runtime. The original Devices on FHIR PoCD subgroup
(2018-04-16: S. Karl, K. Matsumoto, G. Dixon, M. Faughn, J. Garguilo)
agreed in principle but recommended naming the element `class` rather
than `dataType` and binding it to IEEE 11073 nomenclature, which already
distinguishes object classes such as Numeric, RealTimeSampleArray, and
Enumeration. The ticket was previously dispositioned "Considered for
Future Use" (2018-04-18, S. Karl / B. Reinhold, 7-0-0; impact:
compatible/substantive; category: enhancement). In April 2026 the
Devices WG (per the 2026-04-08 main call) reverted that resolution and
re-triaged the ticket for action in preparation for the September 2026
ballot. There are no linked or cross-referenced Jira tickets, no
substantive Zulip discussion (only the original `fhir_bot`
tracker-item announcement on 2018-02-08), and no GitHub PRs/issues
attached.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new first-class element `DeviceMetric.class` (using the
2018-04-16 subgroup's preferred naming over the original `dataType`)
of type `code`, cardinality `0..1`, bound (required) to a value set
drawn from IEEE 11073-10201 / 10207 object-class nomenclature —
minimally the values `numeric`, `realtime-sample-array`,
`enumeration`, `complex-metric` (and a handful of related classes such
as `distribution-sample-array` if PoCD requires them). Anchor the value
set in `http://hl7.org/fhir/ValueSet/devicemetric-class` backed by a
new `CodeSystem` (or by an externally-maintained 11073 system if
already published). The element documents the *shape/kind* of metric
data the instance carries, complementing `DeviceMetric.type` (what is
being measured, e.g. an MDC_DIM_* concept) and `DeviceMetric.category`
(measurement / setting / calculation / unspecified). Mark
forwards-compatible: existing instances without `class` remain valid.

#### Justification

Aligns with the original requester's intent and the 2018-04-16 PoCD
subgroup consensus (7-0-0). A typed code element is materially easier
for implementers to dispatch on than `meta.profile` URLs, which are
optional, free-form, and not validated by most receivers. IEEE 11073
already defines exactly this discriminator (object class), so reuse —
not invention — is the right move; this honours Devices' long-standing
policy of reflecting 11073 modelling in FHIR. The change is
backward-compatible (new optional element), substantive but
non-breaking, and lands cleanly in a September 2026 ballot.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a new core element. Instead, (1) define a standard
extension `http://hl7.org/fhir/StructureDefinition/devicemetric-class`
of type `Coding` bound to the same IEEE 11073 object-class value set,
and (2) tighten the guidance on `DeviceMetric.type` so that, where a
single 11073 nomenclature concept already encodes both the measured
parameter and its object class (which is common in 11073-10101), the
type coding is sufficient and the extension is only used when the type
concept is ambiguous about object class. Document the discriminator
recipe in the DeviceMetric narrative and in the Devices on FHIR IG.

#### Justification

Erik Moll's 2026-03-26 comment ("DeviceMetric has a `type` element —
no further action required") reflects a real tension: in 11073
nomenclature, `type` and object-class are entangled, and adding a
parallel core element risks under-specified or contradictory
combinations (e.g., `type` says "blood pressure waveform" while
`class` says `numeric`). An extension keeps the door open for
implementers who genuinely need machine-readable dispatch without
committing the base resource to a second axis that could become
under-used or incoherent. Lower normative risk for a base resource
that has already gone through several FMM uplift cycles, and easier to
retract if September ballot feedback rejects the modelling.

---

### Disposition C: Decline (Persuasive)

#### Proposal

Mark "Persuasive with Mod" only insofar as documentation is concerned:
add narrative to DeviceMetric explaining how IEEE 11073 object class
is conveyed via `DeviceMetric.type` (and, where present, via the
device's `Device.type` / Definition linkage) and explicitly state that
implementers should not rely on `meta.profile` for dispatch. Do *not*
add a new element or extension. Close the ticket once the narrative
lands.

#### Justification

`DeviceMetric.type` (CodeableConcept, required-bound to the device
metric type value set) and `DeviceMetric.category` already carry the
information the requester needs when 11073 nomenclature is used
correctly. Adding a second axis duplicates information for the common
case and risks contradiction in the long tail. The original 2018-04-18
disposition ("Considered for Future Use", 7-0-0) reflected the same
caution; eight years on, no implementer report has surfaced that
showed `type` to be insufficient — only the original requester's
ergonomic preference. Backward-compatibility is best preserved by
clarifying existing mechanisms rather than introducing a new optional
element that vendors may or may not populate, leaving consumers worse
off than today.

---

### Recommendation

**Recommended disposition:** A

The 2018-04-16 PoCD subgroup consensus is unusually clear and well
documented (7-0-0, named participants, explicit naming and binding
guidance), and the WG explicitly *reverted* the 2018 "Considered for
Future Use" closure on 2026-04-08 in order to act for the September
2026 ballot — that signal should not be re-litigated absent new
substantive opposition. Disposition A delivers exactly what the WG
already agreed to: a `class` code bound to IEEE 11073 object-class
nomenclature, complementing rather than replacing `type`. Erik Moll's
"`type` is sufficient" objection (Disposition C / B's premise) is
worth recording, but the WG's repeated position has been that
`DeviceMetric.type` answers "what is being measured" and not
"what shape is the data" — a distinction that 11073 itself preserves
and that implementers benefit from when wiring up Numeric vs.
RealtimeSampleArray vs. Enumeration handlers. Disposition B (extension
only) hedges, but extensions are routinely ignored by receivers and
would leave Devices in the same ergonomic hole the ticket describes.
Recommend the WG ratify A, finalise the value-set scope (especially
which 11073 object classes are in/out for FHIR purposes), and target
the September 2026 ballot.
