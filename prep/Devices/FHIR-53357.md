## Summary

The ticket requests a technical correction to Section 2.9.9 ("Time Ticks")
of the Personal Healthcare Devices (PHD) FHIR Implementation Guide. The
Time Ticks mechanism was added to accommodate simple devices that report
elapsed time directly from a microprocessor clock — typically at 1024 Hz
— rather than using a wall-clock timestamp. The reporter recommends that
the IG explicitly guide implementers to convert any tick-based timestamp
to milliseconds (or finer) resolution before exposing it through FHIR,
so downstream consumers receive a standard, well-defined time unit
rather than device-specific tick counts.

No other Jira tickets are cross-referenced. The only cross-reference
surfaced by Augury is the Confluence page for the **2025-11-12 Devices
Main Call Agenda and Minutes**, which is where the WG reviewed this
ticket and, per Lynn Laakso's comment, agreed to defer the change to the
next revision of the PHD IG (post-STU2). The ticket was subsequently
reopened "post STU2 publication" so that it can be picked up in the
next PHD IG cycle.

## Proposed Dispositions

Because no GitHub cross-references exist, these dispositions are
grounded in the PHD IG narrative content rather than a specific
authoring path in a cached repo. The WG should confirm the exact
authoring location when acting on the ticket; the PHD IG publication
narrative is maintained within the HL7 PHD IG project and Section 2.9.9
("Time Ticks") is the specific page/anchor the change would touch.

### Disposition A: Accept As Requested

#### Proposal

Amend Section 2.9.9 ("Time Ticks") of the PHD IG so that it contains a
normative recommendation that implementations convert tick-based
timestamps to milliseconds (or finer) resolution before surfacing them
in FHIR. Concretely:

- Replace or supplement the current informative wording with a clear
  **SHOULD** statement: "A system that receives ticks (typically at
  1024 Hz from a simple device microprocessor clock) SHOULD convert
  them to milliseconds or better resolution before representing the
  time in a FHIR Observation (e.g., in `Observation.effective[x]` or
  `Observation.component.value[x]` where a duration is represented)."
- Add a short worked example showing the conversion from a 1024 Hz tick
  counter to milliseconds (e.g., `ms = ticks * 1000 / 1024`, rounded to
  the implementation's chosen precision).
- Add an informative note calling out that raw tick values MUST NOT be
  carried through as if they were milliseconds, and that the PHD IG
  does not define a code system for "ticks at 1024 Hz" for direct
  FHIR use.

#### Justification

This is what the reporter asked for and it matches the PHD IG's
existing pattern of mapping device-native units into FHIR-native units
(seconds/milliseconds, UCUM) before exposing them to consumers. It
removes an ambiguity where a gateway might otherwise pass through a
tick count with a second- or millisecond-style label. Keeping the rule
at SHOULD rather than SHALL preserves backwards compatibility for
existing gateways that already emit millisecond timestamps, while
giving new implementers a clear target. The Devices WG already
effectively endorsed the direction in the 2025-11-12 call minutes by
deferring rather than declining the change.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding a narrative SHOULD in Section 2.9.9, address the
underlying need structurally:

- Clarify in the PHD IG that time-base conversion is the **gateway**'s
  responsibility and belongs in the gateway-to-FHIR mapping layer
  rather than in the FHIR instances themselves, and move the guidance
  there.
- Where ticks genuinely need to be represented (e.g., for device
  diagnostics), require them to be expressed as a `Quantity` with a
  concrete unit (UCUM `/s` or a named code system for the device clock
  rate) rather than as a bare number, so the time base is always
  explicit.
- Leave Section 2.9.9 as a description of the device-side mechanism and
  add a cross-reference to the new gateway-mapping guidance.

#### Justification

The literal ask (add a recommendation in 2.9.9) fixes the symptom; the
alternative fixes the cause by putting the unit-conversion rule where
the conversion actually happens and by forbidding unit-less tick
numbers anywhere in the FHIR payload. This is consistent with the PHD
IG's general stance that FHIR Observations carry clinically meaningful,
unit-qualified values, and it avoids the risk that a downstream system
"implements" the SHOULD by relabeling ticks as milliseconds. The
trade-off is a larger edit surface than Disposition A and a longer
review window, which conflicts with the WG's stated intent to land this
quickly in the next PHD IG revision.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive / already covered**: the PHD IG already
treats Section 2.9.9 as informative background on a device-side
mechanism, and the broader IG already expects FHIR instances to carry
time values in FHIR-native types (`instant`, `dateTime`,
`Quantity` with UCUM `ms`/`s`). A tick count is, by construction, not
one of those types, so an implementer who emitted raw ticks would
already be non-conformant.

#### Justification

Adding an explicit SHOULD risks implying that the absence of such
wording in other sections means ticks were previously acceptable — a
reading the WG does not want. If the WG considers the existing type
system and narrative sufficient, declining avoids churn in the PHD IG
narrative and keeps the STU2 baseline stable. This is weak, however,
because the reporter is one of the PHD IG's long-standing contributors
and has explicitly asked for the clarification; the existing narrative
clearly isn't clear enough in practice.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the lowest-risk, highest-signal change and is what the
Devices WG has already tacitly agreed to by deferring (not declining)
the ticket in the 2025-11-12 call and by reopening it "post STU2
publication" for the next PHD IG revision. The reporter is asking for
exactly one narrow edit in one numbered section, the fix is a SHOULD
plus a worked example, and it does not break any existing conformant
implementation. Disposition B is attractive in principle but expands
scope beyond what the WG signed up for and would delay landing the
correction; it can be tracked as a follow-up if the WG wants to
formalise gateway-side mapping guidance more broadly. Disposition C is
not defensible given that the ticket has already been reopened by the
WG and the reporter's concern is clearly a real ambiguity in current
Section 2.9.9.
