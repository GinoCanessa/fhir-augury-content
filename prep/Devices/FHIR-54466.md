## Summary

The ticket asks the Devices work group to move several `DeviceMetric`-related
CodeSystems (and possibly their bound ValueSets) out of the FHIR core
specification and into THO (Terminology.HL7.org). The stated rationale is
the standing FHIR policy that terminology artifacts intended to evolve
independently of the FHIR release cycle should live in THO so they can be
maintained while the FHIR specification itself is frozen between releases.

Closely related ticket [FHIR-54042](https://jira.hl7.org/browse/FHIR-54042)
("Move DeviceMetric terminology to THO", marked **Duplicate**) names the
specific code systems in scope —
`http://hl7.org/fhir/metric-category`,
`http://hl7.org/fhir/metric-availability`, and
`http://hl7.org/fhir/metric-calibration-type` — and notes a TSMG
discussion on 2026-01-12 where it was further proposed that none of these
need to retain the FHIR canonical URL because they were `required`-bound
in R5 (so the code system URL was not part of any implementation surface).
The DeviceAlert analog [FHIR-49183](https://jira.hl7.org/browse/FHIR-49183)
sets a precedent for this kind of move (driven by the rule that
extensions cannot reference core ValueSets), although that ticket itself
was resolved **Not Persuasive** — the work happened under a different
ticket trail. Sibling tickets
[FHIR-54464](https://jira.hl7.org/browse/FHIR-54464) and
[FHIR-54465](https://jira.hl7.org/browse/FHIR-54465) (DeviceMetric.color
and DeviceMetric.calibration.state) both call out that their proposed
changes would also enable moving the bound code systems to THO,
reinforcing that this ticket is the umbrella terminology-relocation work
for DeviceMetric.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the following DeviceMetric CodeSystems from FHIR core to THO via
the standard UTG process, retiring the FHIR canonical URLs (per
FHIR-54042's TSMG note that they were `required`-bound in R5 and so
were not part of any implementation surface):

- `http://hl7.org/fhir/metric-category` →
  `http://terminology.hl7.org/CodeSystem/metric-category`
- `http://hl7.org/fhir/metric-availability` →
  `http://terminology.hl7.org/CodeSystem/metric-availability`
- `http://hl7.org/fhir/metric-calibration-type` →
  `http://terminology.hl7.org/CodeSystem/metric-calibration-type`

Move the corresponding ValueSets alongside them (`metric-category`,
`metric-availability`, `metric-calibration-type`) and re-point the
`DeviceMetric` element bindings (`category`, `operationalStatus`,
`calibration.type`) at the new THO ValueSet canonicals. Update the
extensions pack to reference the new THO canonicals as well. Do **not**
mint backward-compatible FHIR canonicals on the THO side, since the R5
required bindings mean no implementer should have been carrying the
code system URL forward as a stable identifier.

Track the work as three companion tickets per the FHIR-49183 template:
(1) add to THO via UTG, (2) update core spec bindings, (3) update
extensions pack.

#### Justification

This is the path the original requester, the duplicate sibling
(FHIR-54042), and the assignee comment from `mfaughn` all converge on,
and it matches the explicit FHIR policy that mutable terminology
should live in THO so it can evolve while FHIR is frozen. The TSMG
already discussed and recommended dropping the FHIR canonical URLs on
2026-01-12, so doing so is not a unilateral decision. The DeviceAlert
work (FHIR-49183 and downstream) provides a working procedural
template, including the requirement to update the extensions pack so
extensions are no longer blocked from referencing the value sets.

---

### Disposition B: Alternative Approach

#### Proposal

Move the CodeSystems and ValueSets to THO **but mint THO canonicals
that the core spec then references via dual-canonical aliases** — i.e.,
keep `http://hl7.org/fhir/metric-category` etc. as `compose.include`
references in legacy R5 ValueSet snapshots and provide an explicit
FHIR-canonical → THO-canonical mapping in the THO publication, so any
implementation that *did* hard-code the FHIR URLs continues to
resolve. Concretely:

1. Publish the three CodeSystems and ValueSets in THO via UTG with the
   `http://terminology.hl7.org/...` canonicals.
2. In the FHIR core spec, replace the in-spec definitions with a
   pointer page that documents the move and shows the new canonical.
3. Retain the `http://hl7.org/fhir/...` URLs as *historical* canonicals
   in the THO record (e.g., via `identifier` slices or
   `OtherElement`-style metadata), so terminology services that already
   cache the old URL can be redirected.
4. Bound from sibling tickets: also bring `metric-color` and
   `metric-calibration-state` over while DeviceMetric is being touched
   for FHIR-54464 / FHIR-54465.

#### Justification

The TSMG argument for dropping the FHIR URL ("nobody could have used
it because it was `required`-bound") is plausible but not provably
true — a terminology server, a validator, or a downstream IG could
still cache canonical URLs even when a ValueSet binding is `required`.
A redirect/alias strategy gives the safety of Disposition A's clean
THO move without forcing every implementer to re-canonicalize lookups
in the same release cycle. Bundling in `metric-color` and
`metric-calibration-state` (driven by FHIR-54464 / FHIR-54465) avoids
having to re-open the core spec a second time.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Not Persuasive** (mirroring FHIR-49183's
disposition). State that the DeviceMetric code systems will remain in
FHIR core for this release because: (a) the bindings are `required`
and the code lists are stable enough that frozen-FHIR evolution is not
yet a practical problem, (b) the move induces churn for terminology
servers and validators with no immediate consumer driving it, and
(c) the related `CodeableConcept`-widening work in FHIR-54464 and
FHIR-54465 is the more important pre-requisite — once those land and
the bindings are extensible, a future ticket can do the THO move with
fewer downstream consequences.

#### Justification

The ticket is explicitly hedged ("possibly valuesets") and there is no
named extension or IG currently blocked by these bindings (unlike the
DeviceAlert case in FHIR-49183, which was driven by a concrete
extension that needed to reference the value sets). With sibling
tickets actively reshaping `DeviceMetric.color` and
`DeviceMetric.calibration.state`, doing the THO move now risks
re-doing it after those land. Declining preserves the option to move
later from a more stable base.

---

### Recommendation

**Recommended disposition:** A

The Devices work group already has alignment on the substance
(FHIR-54042 was treated as a duplicate of this ticket, the assignee
comment from `mfaughn` confirms intent to move, and the TSMG endorsed
the move and the URL-retirement on 2026-01-12). FHIR-49183 has
already proven the three-step UTG → core-spec → extensions-pack
workflow, so the procedural risk is low. Disposition B's
alias/redirect scheme is appealing in principle but adds permanent
THO bookkeeping for a problem that, per TSMG, doesn't actually exist
in implementations because the R5 bindings were `required`. Disposition
C is the most defensible fall-back if the WG wants to wait for
FHIR-54464 and FHIR-54465 to land first, but those are scoped to
individual elements and don't actually conflict with the THO move —
they make it cleaner — so blocking on them is unnecessary.

The recommended path is therefore Disposition A, with the small
amendment from Disposition B's last point: while DeviceMetric is on the
table, scope the move to include `metric-color` /
`metric-calibration-state` if FHIR-54464 / FHIR-54465 are accepted, so
the core spec is only re-opened once.
