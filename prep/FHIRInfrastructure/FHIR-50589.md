## Summary

James Jahns reports an internal inconsistency in the FHIR Core
specification's documentation of the `_list` search parameter. On
`search.html`, the worked example escapes the leading `$` of a named
list token with a backslash:

```
GET [base]/AllergyIntolerance?patient=42&_list=\$current-allergies
```

but the corresponding example on `lifecycle.html#current` uses the
unescaped form `$current-allergies`. The reporter's assumption is that
the `search.html` example (with the `\$`) is the incorrect one and
should be corrected to match `lifecycle.html`. This is filed as a
Medium-priority Technical Correction against FHIR Infrastructure.

No cross-referenced Jira, Zulip, or GitHub items were returned by the
`cross-referenced` query for `FHIR-50589`. A wider Zulip search did
surface an older 2022 `fhir/infrastructure-wg` thread
("Question about escaping and _list") in which Gino Canessa raised the
same tension — `search.html#escaping` lists `$` as a character that
must be escaped, yet `search.html#list` originally showed the bare
`$current-allergies` form — and Grahame Grieve answered simply "you
can just escape them." The current ticket is, in effect, the reverse
direction of that earlier exchange: the escape that was presumably
added in response to Grahame's answer now disagrees with the
`lifecycle.html` example.

## Details

**Description:**

On search.html, the example for _list escapes the $ with a \ :

`GET [base]/AllergyIntolerance?patient=42&_list=\$current-allergies`

This doesn't match the example at
[http://hl7.org/fhir/6.0.0-ballot3/lifecycle.html#current](http://hl7.org/fhir/6.0.0-ballot3/lifecycle.html#current).

I assume that the escaping on search.html is the incorrect one.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat `search.html` as the incorrect example and update it to match
`lifecycle.html`. Concretely, in the `search.html#list` example block,
change:

```
GET [base]/AllergyIntolerance?patient=42&_list=\$current-allergies
```

to:

```
GET [base]/AllergyIntolerance?patient=42&_list=$current-allergies
```

Leave `lifecycle.html#current` unchanged. Do not modify the
`search.html#escaping` rules themselves.

#### Justification

The ticket's reading is defensible: `$current-allergies` is not a
user-supplied search parameter *value* in the sense that the escaping
rules target — it is a well-known logical list identifier whose leading
`$` is part of its name, analogous to an operation name. Treating it
as a literal token (the form used on `lifecycle.html`) keeps the two
pages consistent and matches how named operations are written
throughout the rest of the specification. It also matches the form
implementers are most likely to recognize.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the escape (`\$current-allergies`) on `search.html` because it is
consistent with the normative escaping rule in
`search.html#escaping`, and instead fix `lifecycle.html#current` to
show the escaped form. Additionally, add a one-sentence note next to
the `_list` example on `search.html` explaining that the leading `$`
of a named list must be escaped per the escaping rules, since readers
coming from `lifecycle.html` will otherwise be surprised.

#### Justification

`search.html#escaping` explicitly lists `$` as a reserved character
that must be escaped in search parameter values, and the prior
workgroup guidance on the 2022 Zulip thread from Grahame Grieve was
"you can just escape them" — which is presumably why the `\$` form is
on `search.html` today. Removing the escape (Disposition A) reintroduces
the inconsistency that thread tried to close and can be read by
implementers as permission to ignore the escaping table for any token
whose `$` is "part of the name." Fixing `lifecycle.html` instead
preserves the single normative rule and only changes a non-normative
example.

---

### Disposition C: Decline

#### Proposal

Decline the technical correction on the grounds that both forms are
acceptable: `$` only needs escaping where it would otherwise be
ambiguous with a search modifier or operator, and for the `_list`
parameter — which expects a list identifier, never a modifier — either
`$current-allergies` or `\$current-allergies` round-trips to the same
value. Close as "Not Persuasive" with a note that the two examples are
deliberately tolerant of either form.

#### Justification

This can be defended if the workgroup reads the escaping rules as
applying only where ambiguity is possible. For a parameter like `_list`
that is typed as a list reference, no search modifier/operator can
legally occupy that position, so there is no ambiguity to escape
against. This is the lowest-churn option and avoids touching either
page.

---

### Recommendation

**Recommended disposition:** B

The 2022 Zulip thread is directly on point: the workgroup chair
answered the same question with "you can just escape them," which is
how `search.html` came to show `\$current-allergies` in the first
place. Disposition A undoes that decision on the basis of a single
comparison with one other example page, and it weakens the escaping
table on `search.html#escaping` by implying that a listed reserved
character does not actually need to be escaped in certain contexts.
Disposition C hand-waves the inconsistency away without fixing either
page. Disposition B keeps the normative escaping rule intact, resolves
the reporter's concern by making the two pages agree, and does so by
changing a non-normative example on `lifecycle.html` rather than a
normative-adjacent example that was deliberately brought into line
with the escaping table. A short clarifying sentence on `search.html`
next to the `_list` example will also pre-empt the next round of this
ticket.
