## Summary

The ticket reports a defect in the XML example shown in
[`narrative.html#linking`](https://hl7.org/fhir/6.0.0-ballot4/narrative.html#linking)
of the FHIR Core 6.0.0-ballot4 specification. The example shows a
`Composition.section.text` element containing only a `<div>` child, but
omits the mandatory `<status>` child of `Narrative` (which is required
to be one of `generated | extensions | additional | empty`). Because
`Narrative.status` is a 1..1 element across FHIR, the example as
written would not validate — and worse, it teaches readers an invalid
shape for narrative. The reporter proposes the trivial fix of inserting
`<status value="generated"/>` immediately before the `<div>` inside the
sample's `<text>` block.

No cross-references to other Jira tickets, GitHub issues, PRs, or
Confluence pages were returned by the cross-reference index for
FHIR-54251, and no Zulip thread directly cites the ticket key. Related
prior community discussion exists in the `implementers` stream around
how to choose a value for `Composition.section.text.status`, but the
ticket itself is a narrow Technical Correction to a code sample, not a
re-opening of the semantic question.

## Details

**Description:**

> https://hl7.org/fhir/6.0.0-ballot4/narrative.html#linking
>
> The XML sample provided seems missing status in the text field under
> section in the Composition:
>
> ```xml
>      <section>
>         ...
>         <text>
>           <div xmlns="http://www.w3.org/1999/xhtml">
>             ...
>             <p>There is a history of <span id="a1">Asthma</span></p>
>             ...
>           </div>
>         </text>
>       </section>
> ```
>
> should be
>
> ```xml
>       <section>
>         ...
>         <text>
>           <status value="generated"/>
>           <div xmlns="http://www.w3.org/1999/xhtml">
>             ...
>             <p>There is a history of <span id="a1">Asthma</span></p>
>             ...
>           </div>
>         </text>
>       </section>
> ```

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit `source/narrative.html` in `HL7/fhir`. Locate the
`Composition.section` XML example beneath the *Linking inside the
narrative* heading (anchor `#linking`) and insert
`<status value="generated"/>` as the first child of the `<text>`
element, immediately before `<div xmlns="http://www.w3.org/1999/xhtml">`,
with the indentation used by the surrounding sample. If the same page
shows the snippet in JSON form, add `"status": "generated"` as the
sibling of `"div"` in that example as well. No specification text,
StructureDefinition, invariant, or schema changes are required —
this is a pure example-fix Technical Correction.

While in the file, perform a `source/`-wide grep for the broken
pattern (`<text>\s*<div xmlns=`) to catch identical defective
samples on other pages (`composition.html`,
`composition-examples.html`, and any other resource page that
illustrates `Resource.text` with an XML snippet). Fix in place.

#### Justification

`Narrative.status` is cardinality 1..1; an XML sample that omits it
is invalid against the schema and against the definition of the
`Narrative` datatype. Showing an invalid example in normative
narrative documentation actively misleads implementers and undermines
the rule the spec itself enforces. The fix is a one-line insertion
that exactly matches the reporter's diff, carries zero semantic risk,
zero backward-compatibility risk, and zero cross-repo impact (per the
HL7/fhir briefing). The change recipe is the standard *update
narrative / prose* recipe for FhirCore — a plain `source/*.html`
edit picked up by the bespoke publisher. A repo-wide sweep for the
same defective pattern is the only meaningful additional cost and is
explicitly justified by the "examples are often copy-pasted" gotcha
flagged in the briefing.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the inline fix in Disposition A, **and** add an example-quality
guardrail so this class of defect is harder to reintroduce. Two
options that achieve this without code changes to the publisher:

1. Run the existing FHIR validator over each XML/JSON snippet that the
   publisher emits from `source/*.html` example blocks as part of the
   `qa/` build step (the current QA harness already validates resource
   instances under `source/<resource>/`; this would extend that to
   inline page examples).
2. Add a short authoring note to `narrative.html` itself reminding
   contributors that every `<text>` block in any sample must include
   `<status>` before `<div>`, and link to `Narrative` from the
   *Linking inside the narrative* prose so the requirement is
   visually adjacent to the example.

#### Justification

The literal reporter request (Disposition A) is correct but narrow —
it fixes one occurrence and leaves the underlying authoring failure
mode in place. The implementers Zulip thread *When is
Composition.section.text "generated"?* shows that `Narrative.status`
is already a frequent point of confusion; broken examples in the spec
itself amplify that confusion. Adding a lightweight guard means the
next reporter never has to file the same Technical Correction. The
trade-off is scope: option (1) is a real (small) addition to the QA
harness and would need FMG / Infrastructure agreement on the validator
profile to apply; option (2) is editorial and can ship with the
inline fix. A pragmatic compromise is to do the inline fix now and
file a follow-up tracker for option (1).

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that the ellipses (`...`) in the
sample mean "details elided" and a reader is expected to infer that
elided content includes the required `<status>` element; therefore
the sample is illustrative, not normative, and no fix is required.

#### Justification

This is the only defensible decline framing, and it is weak. The
ellipses in the sample sit *inside* the `<div>`, not before it —
they elide xhtml content, not narrative-level children. There is no
plausible reading under which a `<text>` element that begins
`<text><div xmlns="...">...` implicitly contains a `<status>`
sibling. Declining would also be inconsistent with how prior
"missing required element in code sample" Technical Corrections in
FhirCore have been treated. Recording this disposition for
completeness; it is not the recommended path.

---

### Recommendation

**Recommended disposition:** A

The ticket is a textbook Technical Correction: a short, unambiguous,
one-line fix to an XML sample that is currently invalid against the
very datatype it illustrates. The reporter has supplied the exact
diff, the change is confined to a single FhirCore authoring file
(`source/narrative.html`) with no cross-repo touch points and no
semantic impact, and the HL7/fhir briefing confirms the standard
"update narrative / prose" recipe applies cleanly. Disposition B's
guardrail idea is a good follow-up but should not block this fix —
file it as a separate FHIR Infrastructure or FMG ticket rather than
expanding the scope of FHIR-54251. Disposition C is recorded only
to make the trade-off explicit; the ellipses argument does not
survive even a casual reading of the snippet. Recommend accepting,
applying the inline fix, and doing the cheap `source/`-wide sweep
for the same pattern as part of the same commit.
