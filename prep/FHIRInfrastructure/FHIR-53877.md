## Summary

DanielSchuster is asking the FHIR Infrastructure work group to clean up
the first paragraph of the `delete` interaction in `http.html` of the
FHIR R6 ballot. The paragraph mixes two different conversations — the
HTTP status that a server returns to the `DELETE` request itself
(`200 OK`, `204 No Content`, `202 Accepted`) and the HTTP status that
subsequent `read` requests should return for the now-deleted resource
(`410 Gone`, or another code per Access Denied Response Handling) — and
the reporter found that ordering hard to follow. The reporter raises
three concrete points: (1) the read-versus-delete responses should be
visually separated so a reader does not have to re-parse the paragraph,
(2) the conformance level of "return `410 Gone` on subsequent reads"
should be made explicit (informative suggestion vs. `SHOULD` vs.
`SHALL`), and (3) the spec should acknowledge that some implementations
do not physically remove the row but instead flip a status element, and
that those servers may legitimately want to keep returning the resource
on a normal read (or even via search). No cross-referenced tickets or
GitHub items were surfaced by the FHIR Augury cross-reference index for
this ticket, but a closely related historical ticket
[FHIR-16198](https://jira.hl7.org/browse/FHIR-16198) ("Add delete/purge
REST operation", Applied/Persuasive with Modification) and several
Zulip threads on `#implementers` cover the same ground and should be
considered when wording the clarification.

## Details

**Description:**

A few comments on the first paragraph of the delete interaction description:

> "A delete interaction means that the resource can no longer found
> through a [search](https://hl7.org/fhir/6.0.0-ballot4/http.html#search)
> interaction. Subsequent
> [non-version specific reads](https://hl7.org/fhir/6.0.0-ballot4/http.html#read)
> of the resource return a `410 Gone` HTTP status code when the server
> wishes to indicate that the resource is deleted. For security reasons,
> the server may return other status codes as defined under
> [Access Denied Response Handling](https://hl7.org/fhir/6.0.0-ballot4/security.html#AccessDenied).
> Upon successful deletion, or if the resource does not exist at all,
> the server should return either a `200 OK` if the response contains a
> payload, or a `204 No Content` with no response payload. Servers using
> the [Asynchronous Interaction Request Pattern](https://build.fhir.org/ig/HL7/api-incubator/async-bundle.html)
> may return a `202 Accepted`, if the server wishes to be non-committal
> about the outcome of the delete."

- I had to read a few times to realize that the HTTP 410 response was
  for subsequent reads, whereas the 200, 204, and 202 responses were
  for the delete request itself. I wonder if moving the second and
  third sentences to the end of the paragraph would clear it up.
- It is unclear whether returning 410 by default (given no security
  considerations) is a SHALL or SHOULD level directive, or merely an
  informative suggestion.
- Some implementations will merely change elements on the resource (eg
  marking a status element as deleted) rather than actually deleting
  the resource from the database. It is feasible that some workflows
  may want to perform a standard read on a resource with status=deleted
  (or even return it in a search).

## Related Zulip Discussions

### [implementers > Purge and Delete History Operations (FHIR-16198)](https://chat.fhir.org/#narrow/stream/implementers/topic/Purge%20and%20Delete%20History%20Operations%20%28FHIR-16198%29)

Multi-month design discussion attached to FHIR-16198. The thread
explicitly settled the conformance question that FHIR-53877 is now
re-asking: returning `410 Gone` on a subsequent read of a deleted
resource is **implementer choice** — `404 Not Found` is also valid,
and "servers are under no obligation to return a 410 over a 404."
Gino Canessa quoted the same paragraph FHIR-53877 cites and noted
"this ticket does not change that." Alexander Kiel argued that the
choice should at least be **mentioned** in the spec the same way
`read` mentions it, and pushed for a parallel clarification on `vread`
behaviour for "delete entries" (suggesting `204` instead of `410` for
the empty placeholder version). No one in the thread argued for
making `410` mandatory. The thread did **not** explicitly address the
"soft delete / status=deleted" pattern that FHIR-53877 raises.

Participants: Alexander Kiel, Sonat Suer, Gino Canessa, Michael Lawley.

### [implementers > Version Id of a deleted resource](https://chat.fhir.org/#narrow/stream/implementers/topic/Version%20Id%20of%20a%20deleted%20resource)

Short implementer Q&A from 2023. Bohdan Holyshevskyi asked how to
discover the `versionId` of a deleted resource given that `vread` of a
delete entry returns `410 Gone` and the resource body is absent.
Michael Lawley pointed at `_history`, and Lloyd McKenzie clarified
that in the history Bundle the `entry.request.method` will be `DELETE`
and the version is carried in `entry.response.etag`. Reinforces that
implementers already treat the `410` text as descriptive of one
acceptable behaviour rather than a `SHALL`.

Participants: Bohdan Holyshevskyi, Michael Lawley, Lloyd McKenzie.

### [implementers > _history query when Resources are deleted](https://chat.fhir.org/#narrow/stream/implementers/topic/_history%20query%20when%20Resources%20are%20deleted)

Older (2017) thread initiated by John Moehrke asking what
`_history` should return when resources have been deleted. Brian
Postlethwaite walked through the expected behaviour and explicitly
called out the implementer choice: `base/Patient/45` is `410 Gone`
for a deleted resource, but `_history` "might = 404 if server doesn't
support versioning" — i.e. the community has been treating `410` as
the recommended-but-not-mandatory response on deleted-resource reads
for nearly a decade.

Participants: John Moehrke, Lloyd McKenzie, Brian Postlethwaite.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure the first paragraph of the `delete` interaction in
`http.html` so that the two concerns are textually separated, and add
explicit conformance language and a soft-delete carve-out:

1. **Paragraph 1 (response to the DELETE request itself):** "Upon
   successful deletion, or if the resource does not exist, the server
   **SHALL** return either `200 OK` (with a payload, e.g. an
   `OperationOutcome`) or `204 No Content` (with no payload). Servers
   implementing the
   [Asynchronous Interaction Request Pattern](https://build.fhir.org/ig/HL7/api-incubator/async-bundle.html)
   **MAY** return `202 Accepted` instead, when the server wishes to be
   non-committal about the outcome of the delete."
2. **Paragraph 2 (effect on subsequent interactions):** "After a
   successful delete, the resource **SHALL NOT** be returned by a
   `search` interaction. A subsequent non-version-specific
   [read](#read) of the resource **SHOULD** return `410 Gone` to
   signal that the resource is deleted, but **MAY** return `404 Not
   Found` instead; servers **MAY** return other status codes as
   described under [Access Denied Response Handling](#AccessDenied)
   when security considerations apply."
3. **Paragraph 3 (logical / soft delete):** Add an explicit note that
   "some servers implement deletion logically (e.g. by setting a
   status element on the resource rather than removing the row). Such
   servers MAY continue to return the resource on `read` and in
   `search` results provided that the resource's status correctly
   indicates the deleted state; the `delete` interaction's
   requirements above apply to the resource as exposed by the FHIR
   API, not to the underlying storage."

#### Justification

This is exactly the wording change the reporter is asking for and it
maps cleanly onto the existing community consensus already documented
in the FHIR-16198 Zulip thread: `410` is recommended-but-not-required
(so `SHOULD` is the right verb), and the choice has been treated as
implementer choice in practice for years. Splitting the two response
families into two short paragraphs eliminates the "I had to read it a
few times" problem without changing any normative behaviour. Naming
the soft-delete pattern explicitly aligns the spec with how a large
class of implementations actually behave today and avoids implying
that those implementations are non-conformant.

---

### Disposition B: Alternative Approach

#### Proposal

Make a smaller editorial fix only:

- Re-order the existing sentences (move the `200/204/202` sentence to
  the front, then the `410 Gone` / Access Denied sentence about
  subsequent reads), keep the existing "may return" and "should
  return" wording verbatim, and add a single explanatory sentence:
  "The choice between `410 Gone` and `404 Not Found` on subsequent
  reads is left to the server, mirroring the
  [read interaction](#read)."
- Do **not** add a new normative `SHOULD`/`SHALL` for the read
  response and do **not** add a soft-delete paragraph; instead, file a
  follow-up ticket against the Security work group to address the
  soft-delete question separately, since it touches consent,
  auditability, and access control as well as `delete`.

#### Justification

This minimises the normative surface change in a ballot cycle. The
ordering fix unambiguously addresses the reporter's first point. The
"left to the server" sentence resolves the second point without
forcing the work group to debate the `SHOULD` vs. `MAY` distinction
mid-ballot. Splitting the soft-delete clarification into its own
ticket is defensible because the FHIR-16198 thread never reached
consensus on it and Michael Lawley's question about SMART scopes for
`delete-history` shows that the soft-delete conversation pulls in
work outside FHIR-I.

---

### Disposition C: Decline

#### Proposal

Decline as "Not Persuasive – already addressed". Argue that the
existing paragraph, read together with the `read`, `vread` and
`search` sections and with the Access Denied Response Handling
section, already conveys the intended behaviour, and that
FHIR-16198's resolution explicitly settled the `410 Gone` vs.
`404 Not Found` question as implementer choice. The soft-delete
pattern is out of scope for the `delete` interaction text because
the spec describes the behaviour of the FHIR API surface, not the
storage mechanism.

#### Justification

A defensible "decline" rationale exists: nothing the reporter raises
is incorrect in the current spec, and reworking ballot wording for a
purely editorial readability complaint risks inviting fresh ballot
comments at a late stage of the R6 cycle. The conformance ambiguity
the reporter perceives was discussed at length in FHIR-16198 and
left intentionally open.

---

### Recommendation

**Recommended disposition: A**

The reporter's three points are all real and have been raised
independently by other implementers in the Zulip threads cited above.
The first point (ordering) is purely editorial and cost-free. The
second point (conformance level) is a long-standing source of
implementer confusion that FHIR-16198's resolution already settled
informally as "implementer choice"; promoting that to a clear
`SHOULD … MAY return 404 instead` in the spec text removes ambiguity
without changing what conformant servers actually have to do. The
third point (soft delete) is the most contentious, but acknowledging
the pattern in a short note is materially less work than fielding the
same ticket again every ballot cycle, and it does not require any
new normative machinery — just an explicit statement that the spec
governs API behaviour, not storage. Disposition B is acceptable as a
fallback if the work group wants to ship the ordering fix immediately
and defer the soft-delete language to a separate ticket. Disposition
C is hard to defend given that the reporter is pointing at genuine
ambiguity that the community has already had to resolve in chat.
