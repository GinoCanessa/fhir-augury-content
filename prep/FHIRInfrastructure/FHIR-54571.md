## Summary

The reporter (Chris Moesel) observed that the **Version Diff** tab on the
Condition resource in the R6 ballot publication lists the removal of codes
from `Condition.clinicalStatus` (recurrence, relapse, remission, resolved)
and `Condition.verificationStatus` (provisional, differential). Those codes
are in fact still present in the bound value sets for R6, so the diff is
misleading. The reporter notes that every code flagged as "removed" is a
level-2 (child) code in its value-set hierarchy, suggesting a systematic
bug in the diff-generation tooling rather than any real spec change.

The ticket was filed against Patient Care but, per discussion on the
`[committers] R6 Diff for codes` Zulip thread and
[michelle.m.miller's comment](https://jira.hl7.org/browse/FHIR-54571),
ownership was transferred to FHIR Infrastructure and the ticket was
flagged as a duplicate of the earlier, already-resolved
[FHIR-54489](https://jira.hl7.org/browse/FHIR-54489) ("Version Diff for
Codes Broken"). FHIR-54489 is resolved **Persuasive** with the root cause
identified as a tooling bug in kindling; a follow-up has been raised as
[HL7/kindling#212](https://github.com/HL7/kindling/issues/212) and the
work group has called out that the diff view is not reliable and not
subject to ballot. The reporter also added in a comment that he sees the
same pattern on StructureDefinition (where `fhirVersion` shows many
codes as removed that are not in fact removed), implying the bug is
widespread across resources that bind to hierarchical value sets.

## Details

**Description:**

The Version Diff tab lists the following differences (for R4/R4B to R6
and R5 to R6):

- Condition.clinicalStatus: Remove codes recurrence, relapse, remission, resolved
- Condition.verificationStatus: Remove codes provisional, differential

This is incorrect, as each of those codes remains in the bound value
sets for R6. Is this possibly an error in the tooling that generates the
diffs? I've noticed that all of the codes flagged as "removed" are child
codes (i.e., they are level 2 in the code hierarchy).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-54571 as a duplicate of
[FHIR-54489](https://jira.hl7.org/browse/FHIR-54489) and mirror its
resolution: resolve **Persuasive** with a resolution note that the
Version Diff is a **kindling tooling defect** (tracked under
[HL7/kindling#212](https://github.com/HL7/kindling/issues/212)) affecting
codes at level >1 in hierarchical value sets, that no spec change to
`Condition.clinicalStatus` or `Condition.verificationStatus` is being
made (the codes remain in the bound value sets and the bindings are
unchanged), and that the diff view is not reliable and is not subject to
ballot. Add the reporter's observation about
`StructureDefinition.fhirVersion` to FHIR-54489 (or a follow-up on
kindling#212) so the tooling fix covers the StructureDefinition case as
well.

#### Justification

The reporter is substantively correct, the same determination was
already made on FHIR-54489, and the root cause is outside the Condition
spec — it is in the kindling diff-generation pipeline. Resolving
Persuasive-as-duplicate records the issue as legitimate, preserves an
audit trail for balloters, and avoids a spurious Condition spec change
that would re-introduce churn in a stable value-set binding. The
ownership transfer from Patient Care to FHIR-I has already occurred via
the `[committers] R6 Diff for codes` Zulip thread, so FHIR-I is the
correct resolver.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve Persuasive-as-duplicate **and** open (or link to) a companion
tracker that expands the scope of the kindling fix explicitly to
`StructureDefinition.fhirVersion` and any other resources whose status
or coded-element bindings exhibit the level-2 false-removal pattern —
e.g., by enumerating the affected resources (Condition, DocumentReference,
Observation, StructureDefinition, …) in
[HL7/kindling#212](https://github.com/HL7/kindling/issues/212) or in a
new FHIR-I Jira tracking ticket. Include a regression check: for every
required-bound coded element whose value set has a hierarchy, assert
that the set of codes in the Version Diff "removed" bucket is empty
unless the binding or value set actually changed.

#### Justification

FHIR-54489's resolution fixes the immediate Condition/DocumentReference/
Observation symptoms, but Chris Moesel's comment here flags
StructureDefinition as another affected resource, which suggests the
defect is general to hierarchical-code diffs rather than limited to the
three resources originally reported. Capturing the wider scope and a
regression test prevents the same issue from recurring on future
ballots and gives publishers a concrete gate before the next release.
This option still avoids any Condition spec change and still closes
FHIR-54571 itself as a duplicate — it only adds breadth to the tooling
follow-up.

---

### Disposition C: Decline

#### Proposal

Close the ticket **Not Persuasive** on the grounds that the Version Diff
tab is explicitly called out by the publishing process as not reliable
and not subject to ballot, and therefore no work-group action is
warranted on the ballot comment.

#### Justification

This is defensible only on the narrow basis that the Version Diff is not
normative, so a mis-rendered diff does not represent a spec defect. The
spec artifacts themselves (the Condition resource, its bindings, and the
target value sets) are correct.

---

### Recommendation

**Recommended disposition:** **A** (with the spirit of B captured on
kindling#212).

Disposition A is the correct and minimally disruptive action: the
duplicate ticket [FHIR-54489](https://jira.hl7.org/browse/FHIR-54489)
has already been resolved **Persuasive** by FHIR-I with the identical
root-cause analysis, ownership of FHIR-54571 has already been
transferred from Patient Care to FHIR-I on the
`[committers] R6 Diff for codes` Zulip thread, and the real fix lives
in kindling rather than in the Condition spec. Resolving this ticket as
a persuasive duplicate keeps the ballot audit trail consistent and
avoids a pointless spec edit. The reporter's useful additional
observation about `StructureDefinition.fhirVersion` should be added as a
comment on [HL7/kindling#212](https://github.com/HL7/kindling/issues/212)
so the tooling fix is scoped broadly enough to prevent a recurrence —
this captures what Disposition B adds without requiring a separate Jira
tracker. Disposition C is not recommended: even though the Version Diff
isn't balloted, silently declining a correct observation from two
independent reviewers would undermine confidence in the ballot-comment
process.
