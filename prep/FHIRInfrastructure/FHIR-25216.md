## Summary

The ticket asks the FHIR specification (specifically the FHIRPath
"Using FHIR types in expressions" section) to clarify what happens when
a FHIRPath expression compares two `FHIR.Quantity` values whose units
are *not* UCUM-coded. The current text says FHIR primitives are
auto-converted to FHIRPath system types, but also says the mapping from
`FHIR.Quantity` to `System.Quantity` only applies when the FHIR
Quantity has a UCUM code (`system = http://unitsofmeasure.org`) plus a
code. That leaves a gap: invariants such as Range's **rng-2**
(`low.empty() or high.empty() or (low <= high)`) actively compare two
Quantities and will be evaluated even when those Quantities carry
non-UCUM coded units. The reporter proposes that the spec state two
`FHIR.Quantity` values are comparable in FHIRPath when (and possibly
only when) they share the same `system` and `code`, and asks the WG to
enumerate which FHIRPath operations are well-defined over FHIR
Quantities — equality (likely yes), ordered comparison (likely yes),
and arithmetic (likely problematic because the resulting unit is
generally undefined). No related Jira tickets, Zulip threads, or
GitHub items were surfaced by the cross-reference index for this
ticket.

## Details

**Description:**

> [https://build.fhir.org/fhirpath.html#types](https://build.fhir.org/fhirpath.html#types)
> indicates that the FHIRPath evaluation engine will "automatically
> convert the value of FHIR types representing primitives to FHIRPath
> types when they are used in expressions".
>
> However,
> [https://build.fhir.org/fhirpath.html#quantity](https://build.fhir.org/fhirpath.html#quantity)
> indicates that "The Mapping from FHIR Quantity to FHIRPath
> System.Quantity can only be applied if the FHIR Quantity has a UCUM
> code - i.e. a system of `[http://unitsofmeasure.org,|]` and a code is
> present."
>
> This begs the question "what if a quantity is used in an expression
> and cannot be mapped to a FHIRPath quantity?"
>
> This is not just hypothetical because we have constraints like
> **rng-2** `low.empty() or high.empty() or (low <= high)` which
> compare two quantities.
>
> I think it would be good for
> [https://build.fhir.org/fhirpath.html#quantity](https://build.fhir.org/fhirpath.html#quantity)
> to clarify that two `FHIR.Quantity` objects are comparable in
> FHIRPath if (and only if?) they have the same system and code (coded
> unit). The case where both are UCUM is already covered by
> auto-converting to `System.Quantity` types and comparing as
> described in FHIRPath. However if they have non-UCUM units, we
> should clarify that they can still be evaluated in certain cases.
>
> Question: which FHIRPath functions and operations can be performed
> over FHIR Quantities?
> * Equality? probably yes
> * Comparison? probably yes
> * Math? probably not since we cannot compute the resulting unit

**Comments:**

- **johntimm (2019-11-13):** "We should also be able to support
  addition and subtraction of two non-UCUM `FHIR.Quantity` values with
  the same coded units."
- **bryn.rhodes (2019-11-14):** "Doesn't the definition for equality
  here follow from the definition of equality for objects?" — i.e.,
  equality of two FHIR Quantities can fall back to structural object
  equality without needing the UCUM mapping.
- **lmsurprenant (2019-11-15):** Agrees that object-equality semantics
  make sense for the equality operators specifically.
- **brian.postlethwaite (2026-04-02):** Suggests the spec could add
  definitions/overloads for what FHIR Quantity comparison looks like
  with FHIR types (in the same way FHIRPath already overloads
  `equality`, `ofType()`, and `toString()` for FHIR types). Notes that
  addition/subtraction of same-unit Quantities is well-defined, and
  that multiplication is also defined when one operand is the UCUM
  dimensionless unit `'1'` (since that does not change the unit).

## Proposed Dispositions

> Note: with no GitHub cross-references or repo briefings to anchor
> against, the dispositions below name the FHIRPath specification page
> (`source/fhirpath.html` in the FHIR Core spec, section "Using FHIR
> types in expressions > Quantity") as the implied authoring target.
> The actual code change would land in HL7/fhir (FhirCore) and be
> validated against `Range.rng-2` and similar Quantity-comparing
> invariants.

### Disposition A: Accept As Requested

#### Proposal

Update the FHIRPath spec section "Using FHIR types in expressions >
Quantity" (https://build.fhir.org/fhirpath.html#quantity) to add an
explicit subsection on operations over `FHIR.Quantity` values that
cannot be mapped to `System.Quantity` (i.e., non-UCUM-coded
quantities). The new normative text should:

1. State that two `FHIR.Quantity` values are **comparable for
   equality** (`=`, `~`, `!=`, `!~`) using the standard FHIRPath
   object-equality rules — they are equal iff `system`, `code`,
   `unit`, and `value` (decimal-equal) all match. This handles
   bryn.rhodes's point that equality already follows from object
   equality.
2. State that two `FHIR.Quantity` values are **comparable for
   ordering** (`<`, `<=`, `>`, `>=`) **if and only if** they share the
   same non-empty `system` and `code`; in that case the comparison is
   performed on `value`. If `system` or `code` differs (or is absent
   on either side), the result is `{ }` (empty), per the existing
   FHIRPath rule for incomparable quantities.
3. State that **arithmetic** (`+`, `-`) on two `FHIR.Quantity` values
   with the same non-empty `system` and `code` is defined and yields a
   `FHIR.Quantity` with that same `system`/`code` and the
   sum/difference of the values. Multiplication/division by the UCUM
   dimensionless unit `'1'` is defined and preserves the other
   operand's unit (per Brian's comment). All other arithmetic
   combinations on non-UCUM quantities yield `{ }`.
4. Add a worked example using `Range.rng-2`
   (`low.empty() or high.empty() or (low <= high)`) showing the
   intended behaviour when `low` and `high` are non-UCUM coded
   Quantities with matching `system`/`code`.

#### Justification

This is what the reporter asked for and what the comment thread
converged on — equality via object equality (Bryn), addition/subtraction
of same-unit non-UCUM Quantities (John), and overloads modeled on the
existing `equality`/`ofType()`/`toString()` overloads for FHIR types
(Brian). It also closes a real interoperability hole: `Range.rng-2`
and similar invariants are currently under-specified for non-UCUM
Quantities, which means different validators may return different
results for the same data. Tying the comparable-set to "same `system`
and `code`" is consistent with how Quantity equivalence (`~`) already
treats coded units and avoids inventing a new comparability model.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the FHIRPath spec with a new
non-UCUM-Quantity algebra, take a smaller, defensive change:

1. In `fhirpath.html#quantity`, clarify only the **conversion** rule:
   when a `FHIR.Quantity` cannot be mapped to `System.Quantity`
   (because it lacks a UCUM `system`+`code`), it remains a
   `FHIR.Quantity` value. Equality and equivalence between two
   `FHIR.Quantity` values fall through to the generic FHIRPath object
   equality/equivalence rules (so Bryn's point becomes the only
   normative statement on equality).
2. For **ordered comparison** and **arithmetic** between two
   non-mapped `FHIR.Quantity` values, declare the result is `{ }`
   (empty) unless an implementation guide or profile defines
   otherwise.
3. Audit the FHIR Core invariants that compare Quantities (notably
   `Range.rng-2`, but also any `MoneyQuantity`/`SimpleQuantity`
   constraints) and either (a) restrict them with an explicit
   precondition (e.g., `low.system = high.system and low.code =
   high.code`) or (b) replace the comparison with a structural check
   that does not depend on Quantity ordering.

#### Justification

Defining a partial algebra over `FHIR.Quantity` (Disposition A) adds
surface area to FHIRPath that other languages/profiles may not
implement consistently, and it risks under-specifying edge cases (e.g.,
case sensitivity of `code`, treatment of `comparator`, treatment of
`unit` text without a `code`). Constraining the spec to "no implicit
ordering/arithmetic without UCUM, and tighten the invariants that
assume otherwise" produces deterministic behaviour with much less new
normative text. It also localises the impact: invariants get fixed in
place rather than every FHIRPath engine having to ship a new Quantity
overload set. Trade-off vs. A: implementers who legitimately compare
non-UCUM coded Quantities (e.g., scoring scales, currency-as-coded)
must do it themselves outside FHIRPath, and `Range` over coded
Quantities loses its ordering check.

---

### Disposition C: Decline

#### Proposal

Decline the change and close as "already addressed by existing
mechanism." The FHIRPath spec already says (a) `FHIR.Quantity` is
mapped to `System.Quantity` only when UCUM-coded, and (b) FHIRPath
operations on values that cannot be evaluated return `{ }` (empty).
Together these mean: comparing two non-UCUM `FHIR.Quantity` values in
FHIRPath yields `{ }`, and any invariant that compares such Quantities
either passes (because the comparison is empty and the constraint is
written with `or`) or is, by construction, not applicable. Recommend
that authors who need ordering/arithmetic on coded units convert to
UCUM or use a profile-specific extension.

#### Justification

Defensible on the grounds that the change is out of scope for the core
FHIRPath spec — Quantity semantics for arbitrary code systems is a
domain problem, not a path-language problem, and FHIRPath already has
a well-defined "empty result" fallback for operations it cannot
perform. The risk is that this leaves `Range.rng-2` under-specified
in practice (different validators may treat the empty-result fallback
differently), and it ignores the fact that the ticket has been open
for six years precisely because implementers do not find the current
text clear. This is the weakest of the three options.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right answer because the underlying ambiguity is
real (rng-2 is a live example), the comment thread already converged
on the shape of the fix (object-equality for `=`/`~`, same-unit rule
for ordering and additive arithmetic, dimensionless-`'1'` rule for
multiplication), and the change pattern — overloading FHIRPath
operations for FHIR types — is one the FHIRPath spec already uses for
`equality`, `ofType()`, and `toString()`. Disposition B is a tempting
minimisation but it requires editing every Quantity-comparing
invariant in FHIR Core, which is a larger and more error-prone change
surface than just specifying the operations once in FHIRPath.
Disposition C is not credible after six years of the ticket sitting
open: the empty-result fallback exists, but implementers clearly do
not read the current text as licensing them to rely on it. FHIR
Infrastructure should adopt the Disposition A clarification, with
Bryn's object-equality rule and Brian's overload framing called out
explicitly in the new text, and verify the result against `Range.rng-2`
as a worked example.
