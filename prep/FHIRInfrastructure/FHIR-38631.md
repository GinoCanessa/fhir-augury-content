## Summary

The reporter (Lee Surprenant) observes that the FHIR / FHIRPath integration
page at <http://hl7.org/fhir/fhirpath.html#types> declares that "primitive
data types, data types, resources" are made available in the `FHIR` namespace
for FHIRPath, but does not actually enumerate the types that belong to that
namespace. As a result, two questions are not clearly answered by the spec:

1. Are profiled datatypes such as `SimpleQuantity` and `MoneyQuantity`
   included as `FHIR.*` types (i.e., is `quantity.is(FHIR.SimpleQuantity)`
   meaningful)?
2. Are pattern / interface names (e.g., the abstract patterns the FHIR
   methodology defines) included?

The ambiguity matters for the FHIRPath `is` and `as` operators, which the
FHIRPath specification defines as returning true when the left-hand value's
type is the named type "or a subclass thereof". The ticket asks the FHIR
spec to be explicit about what counts as a subclass within the FHIR type
hierarchy (notably for primitive datatypes, the `Quantity` family, and any
profiled datatypes), because most current FHIRPath engines handle this
inconsistently. The ticket explicitly references the linked Zulip thread
in `#fhirpath` and notes that the FHIRPath conformance test suite should
be improved to cover these cases.

There are no cross-referenced Jira tickets. The single cross-reference is
to the originating Zulip discussion (see below).

## Related Zulip Discussions

### [#fhirpath > FHIRPath `is` and `as` with FHIR types](https://chat.fhir.org/#narrow/stream/fhirpath/topic/FHIRPath%20%60is%60%20and%20%60as%60%20with%20FHIR%20types/near/298799885)

Lee opened the thread by asking whether the FHIRPath `is` / `as` operators
should be aware of inheritance among FHIR datatypes (e.g., should
`d.is(Quantity)` be `true` for a `Duration`?). Josh Mandel and Grahame
Grieve both immediately said yes — that is the intent. Lee then re-read the
FHIRPath specification of `is` and acknowledged that the "or a subclass
thereof" wording already covers the inheritance case, but that the
*scope* of "subclass" in FHIR (primitive type hierarchy, the `Quantity`
family, profiled datatypes such as `SimpleQuantity` / `MoneyQuantity`,
abstract patterns) is still under-specified — particularly because most
existing engines (he cites Brian's FHIRPath playground) get it wrong
despite the spec wording. Josh agreed this points to a need for more
test cases. Grahame asked for volunteers to add them to
[`fhir-test-cases`](https://github.com/FHIR/fhir-test-cases) under
`r4/fhirpath/tests-fhir-r4.xml` and `r5/fhirpath/tests-fhir-r5.xml`. Lee
also wondered aloud whether a non-inheritance variant of `is` was needed,
then concluded `type().namespace` / `type().name` is usually sufficient
(though `%resource.id.type().name` only worked in fhirpath.js at the
time). Prasanna Hegde (working with Lee on LinuxForHealth FHIR) then
contributed
[`FHIR/fhir-test-cases#119`](https://github.com/FHIR/fhir-test-cases/pull/119)
adding `is` / `as` inheritance tests, and Grahame later added
`testTypeA4` to the R5 file. A separate sub-thread (Brian Postlethwaite,
March 2023) raised the `boolean` vs `Boolean` casing question; Grahame
explained that `FHIR.boolean` and `System.Boolean` are both in scope and
distinct, that he accepts either casing in the engine because the
distinction is subtle, but that **FHIR invariants should always use the
lowercase FHIR-namespace form (`is boolean`, not `is Boolean`)**.

Participants: Lee Surprenant (reporter), Josh Mandel, Grahame Grieve,
Prasanna Hegde, Brian Postlethwaite.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the "Types" section of `fhirpath.html` (the FHIR / FHIRPath
integration page in the FHIR Core spec) to:

1. **Enumerate (or programmatically generate) the membership of the
   `FHIR` namespace.** Make explicit that the namespace contains every
   resource type, every complex datatype, and every primitive datatype
   defined by the FHIR core spec. Be explicit about whether the
   following are also `FHIR.*` types:
   - The abstract / pattern types (`Element`, `BackboneElement`,
     `BackboneType`, `DataType`, `PrimitiveType`, `Resource`,
     `DomainResource`, `CanonicalResource`, `MetadataResource`,
     plus the methodology patterns).
   - The constrained `Quantity` profiles (`SimpleQuantity`,
     `MoneyQuantity`, `Age`, `Count`, `Distance`, `Duration`).
2. **Define the FHIR-specific type hierarchy used by `is` / `as` /
   `ofType()`.** State that the "subclass" relation referenced by
   FHIRPath corresponds, for FHIR types, to the
   `StructureDefinition.baseDefinition` chain of the type, and recurses
   through `Resource → DomainResource → ConcreteResource`,
   `Element → DataType → ConcreteDataType`, and through the `Quantity`
   profiles. Profiled datatypes that constrain (but do not derive new
   structure from) `Quantity` are treated as subtypes of `Quantity` for
   `is` / `as`.
3. **Resolve `FHIR.boolean` vs `System.Boolean`** explicitly: in the
   `FHIR` default namespace, `is boolean` resolves to `FHIR.boolean`;
   only `System.Boolean` resolves the System primitive. State that
   FHIR-authored invariants MUST use the lowercase FHIR-namespace form,
   matching Grahame's conclusion in the Zulip thread.
4. **Cross-link the conformance fixtures.** Reference
   `fhir-test-cases/r{4,5}/fhirpath/tests-fhir-r{4,5}.xml` (and the
   `testTypeA4` family added per the Zulip thread) as the canonical
   conformance set for this behaviour, and call for additional cases
   covering profiled `Quantity` types and abstract patterns.

#### Justification

The reporter's complaint — that the integration page tells implementers
the `FHIR` namespace exists without telling them what is in it — is
factually correct, and Josh / Grahame both agreed in chat that
inheritance-aware `is` / `as` is the intended behaviour. Codifying the
membership rules and the "subclass = `baseDefinition` chain" mapping
turns under-specified tribal knowledge into something engines can
implement and conformance tests can verify. Tying the explanation to
`baseDefinition` keeps the prose lined up with how the spec is actually
generated, which avoids a second source of truth that would drift. The
casing clarification (`is boolean`) directly resolves Brian
Postlethwaite's later question and gives invariant authors a single
correct form.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) prose enumeration, address the request
primarily through **machine-readable artefacts plus a much smaller spec
clarification**:

1. Publish a generated "FHIRPath type list" artefact alongside the spec
   build (similar in spirit to the published lists of resources /
   datatypes), listing the full membership of the `FHIR` namespace and
   each type's `baseDefinition` parent. The integration page links to
   this artefact instead of trying to keep an in-prose list current.
2. On `fhirpath.html`, add a single normative paragraph stating: "For
   FHIR types, `is` and `as` use the `StructureDefinition.baseDefinition`
   chain to determine the subclass relation. Profiled datatypes
   (`SimpleQuantity`, `MoneyQuantity`, `Age`, `Count`, `Distance`,
   `Duration`) ARE `FHIR.*` types and ARE subtypes of their constrained
   base. Abstract patterns are NOT in the `FHIR.*` namespace for the
   purpose of `is` / `as`." (Or the inverse policy on patterns, if the
   group prefers — the key is to pick one and write it down.)
3. Bulk up `fhir-test-cases` rather than the spec page: extend
   `tests-fhir-r{4,5}.xml` with an explicit suite that pins down
   `quantity.is(FHIR.SimpleQuantity)`, `code.is(FHIR.string)`,
   `Duration.is(Quantity)`, `boolean` casing, and the pattern question.

#### Justification

The "list the types" request is a maintenance trap: the FHIR datatype /
resource list grows every release, profiled datatypes can be added,
and a hand-curated list in `fhirpath.html` is virtually guaranteed to
fall out of sync. Generating the list from the same metadata used to
build the rest of the spec keeps it correct by construction. Anchoring
the inheritance rule to `baseDefinition` is also the change that most
helps engine implementers, because it tells them which already-published
metadata to consume rather than asking them to hard-code a hierarchy.
Test cases — which Josh, Grahame, Prasanna, and Lee all converged on in
chat — are in practice what makes engines converge; the spec page only
needs to be precise enough to make the test cases unambiguous.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as *Persuasive — Already Addressed*: the FHIRPath
specification already defines that `is` / `as` operate on subclasses
(quoted by the reporter himself), `fhir-test-cases` already includes
the `testTypeA4` family added by Grahame as a result of the Zulip
discussion, and Grahame's existing Java engine implements the intended
behaviour. No spec change required; close as "informational, behaviour
is as specified, see referenced test cases".

#### Justification

The reporter explicitly notes mid-thread that the FHIRPath specification
itself is "not as underspecified as I thought" once he located the
`is` / `as` definition. The substantive disagreements that follow
(profiled datatypes, abstract patterns, `boolean` casing) are
genuinely about FHIR conformance, but `fhir-test-cases` is the
designated home for those clarifications and has already absorbed the
PR resulting from this discussion. From a pure "request as filed"
perspective — "describe what subclass means" — the FHIRPath spec
already does, and one could argue the appropriate response is to
point readers at the existing definition and the existing test cases
rather than enlarging the FHIR Core spec.

---

### Recommendation

**Recommended disposition:** B

The ticket is genuinely valid — the integration page promises a
namespace and then doesn't define its membership — but Disposition A's
"enumerate it in prose" path is brittle and creates a second source of
truth that will drift out of sync at every release. Disposition C is
too thin: even if the FHIRPath spec text is technically sufficient, the
on-the-ground evidence in the Zulip thread (multiple engines diverging,
Brian's later `boolean`/`Boolean` question recurring six months later)
demonstrates that "read the spec more carefully" has not produced
interoperability. Disposition B threads the needle: a single normative
paragraph on `fhirpath.html` that pins down the three contested cases
(profiled `Quantity` subtypes, pattern membership, FHIR vs System
namespace casing), a generated machine-readable list of `FHIR.*`
namespace members so the spec stays correct without manual upkeep, and
a deliberate expansion of `fhir-test-cases` to lock in the behaviour.
That gives implementers exactly what the Zulip participants were
asking for — conformance traction — without adding an editorially
expensive list to the prose. The workgroup should also explicitly
decide the abstract-patterns question one way or the other before
merging, since the chat thread never resolved it.
