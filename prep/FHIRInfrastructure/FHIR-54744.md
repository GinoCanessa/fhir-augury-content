## Summary

The ticket asks the FHIR Infrastructure work group to clarify, on the
`fhirpath.html` page, which of the FHIR-specific extensions to FHIRPath
(custom functions, environment variables, type handling, navigation
shortcuts, the General Service API, etc.) are **required** for any
conformant FHIRPath implementation used in FHIR, and which are
**optional**. Where a feature is optional but is nevertheless required
to evaluate certain resources or constraints, the reporter asks the WG
to define one or more additional named profiles — analogous to the
existing "Restricted Subset" used for discriminators — so that
implementers can declare and consumers can detect the level of FHIRPath
support that is actually expected in a given context.

There are no Jira-side cross-references on this ticket, but it is
clearly part of a small WGM cluster from the same reporter aimed at
trimming or stratifying the FHIRPath chapter (see also
[FHIR-54742](https://jira.hl7.org/browse/FHIR-54742), which targets the
General Service API). Earlier persuasive work — most notably
[FHIR-13711](https://jira.hl7.org/browse/FHIR-13711) (define limited
FHIRPath subsets for discriminator / SearchParameter / GraphDefinition)
and [FHIR-19545](https://jira.hl7.org/browse/FHIR-19545) (clarify `$this`
in the restricted subset) — set the precedent that "named subsets"
already exist in the FHIRPath chapter, and this ticket extends that
pattern to the FHIR-specific extensions overall. A parallel approach
from SDC, [FHIR-49590](https://jira.hl7.org/browse/FHIR-49590), shows
the work group's recent direction of using feature-negotiation /
CapabilityStatement features rather than profiles to declare which
expression languages and dialects an implementer supports.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update `fhirpath.html` (the FHIR-specific use of FHIRPath chapter) to:

1. **Add a "Conformance" subsection** at the top of the page that, for
   each FHIR-specific FHIRPath addition, marks it as **REQUIRED** or
   **OPTIONAL** for a conformant FHIRPath-in-FHIR implementation.
   At minimum, classify:
   - **Functions:** `extension(url)`, `hasValue()`, `getValue()`,
     `resolve()`, `ofType(type)`, `htmlChecks()`, `conformsTo(uri)`,
     `memberOf(uri)`, `subsumes(code)`, `subsumedBy(code)`,
     `hasTemplateIdOf(uri)`, `comparable(quantity)`, and any other
     FHIR-defined functions presently in the chapter.
   - **Environment variables:** `%resource`, `%rootResource`,
     `%context`, `%ucum`, `%sct`, `%loinc`, `%"vs-…"`, `%"ext-…"`.
   - **Navigation / typing rules:** polymorphic value\[x\] handling,
     primitive-extension navigation, Reference resolution behaviour.
   - **General Service API** (cross-link to FHIR-54742; whatever
     remains in core after that ticket resolves should be classified
     here).
2. **Generalise the existing "Restricted Subset" pattern** by giving
   each named profile a stable identifier and a complete feature
   matrix. Today the chapter already documents discriminator,
   SearchParameter, invariant, GraphDefinition, and mapping subsets
   (per the resolution of [FHIR-13711](https://jira.hl7.org/browse/FHIR-13711));
   this ticket asks that those be promoted to a single normative
   conformance table on this page rather than scattered prose, and
   that any newly-required combination get its own named profile (e.g.,
   "Validation Subset", "Resolution-Capable Subset").
3. **Add a per-element note** on each spec element that drives FHIRPath
   evaluation (e.g., `ElementDefinition.constraint.expression`,
   `SearchParameter.expression`, `StructureDefinition.context`,
   `GraphDefinition.link.path`, `Questionnaire.item.enableWhen.*`)
   stating which named subset its expressions MUST satisfy.

#### Justification

This is exactly what the reporter asks for and is the natural
continuation of the work already done under
[FHIR-13711](https://jira.hl7.org/browse/FHIR-13711) (Persuasive) and
[FHIR-19545](https://jira.hl7.org/browse/FHIR-19545) (Persuasive). The
named-subset pattern is established, well-understood, and maps cleanly
onto how implementers actually build FHIRPath engines today (many
support only what discriminators and SearchParameters need, not the
full FHIRPath surface). Making the required/optional split explicit
in a normative conformance table removes ambiguity that today forces
implementers to reverse-engineer expectations from the resources that
happen to use a given function. It is also low-risk in the sense that
it documents existing behaviour rather than changing engine semantics.

---

### Disposition B: Alternative Approach — Conformance via Feature Negotiation

#### Proposal

Rather than defining additional FHIRPath "profiles" inside the
FHIRPath chapter, introduce **FHIRPath conformance feature codes**
declared on `CapabilityStatement` (mirroring the SDC approach taken in
[FHIR-49590](https://jira.hl7.org/browse/FHIR-49590)). Concretely:

- Define a small set of feature codes published in the FHIR core
  feature registry, e.g.
  `fhirpath.subset.discriminator`,
  `fhirpath.subset.searchparam`,
  `fhirpath.subset.invariant`,
  `fhirpath.functions.terminology` (covers `memberOf` / `subsumes` /
  `subsumedBy`),
  `fhirpath.functions.resolve`,
  `fhirpath.environment.terminology` (covers `%ucum` / `%sct` /
  `%loinc`),
  `fhirpath.api.generalService`.
- On the FHIRPath chapter, keep a single "Required for any FHIR
  implementation" core list and document the rest only as feature
  codes; remove the implication that anything outside the core list
  is uniformly required.
- Update the `CapabilityStatement` examples and the conformance page
  to show how a server / client / validator declares which of these
  features it supports, and recommend that profiles which depend on a
  non-core feature reference the corresponding feature code.

#### Justification

Feature negotiation is the work group's most recent answer to
"required-vs-optional" questions for expression languages (see
[FHIR-49590](https://jira.hl7.org/browse/FHIR-49590)), and it scales
better than minting more named profiles: future FHIRPath additions can
ship as a new feature code without re-cutting the chapter. It also
sidesteps a real risk of Disposition A — that a normative
"required vs optional" table on the FHIRPath page hardcodes today's
reality and becomes the source of breaking changes whenever a new
function is added. Trade-off: it is somewhat heavier to author against
(implementers must consult the feature registry rather than read one
table), and it leaves the question "what does an unmarked existing
resource expect?" implicitly answered "everything in the core list,"
which still requires the WG to define that core list anyway.

---

### Disposition C: Decline — already addressed by existing subsets

#### Proposal

Close as **Not Persuasive** on the grounds that the FHIRPath chapter
already documents named subsets for the contexts where restriction
matters (discriminator, SearchParameter, GraphDefinition, mapping,
invariants — per
[FHIR-13711](https://jira.hl7.org/browse/FHIR-13711)) and that, outside
those contexts, the FHIR-specific extensions described in
`fhirpath.html` are **all** required of a FHIR-conformant FHIRPath
implementation. Optionally, add a one-paragraph clarification to that
effect, but introduce no new profiles or conformance tables.

#### Justification

A defensible reading of the current spec is that the existing named
subsets *are* the answer the reporter is asking for, and that further
fragmentation would create a combinatorial mess of "profiles of
FHIRPath profiles." Implementers who only need a small subset can
simply implement one of the existing named subsets; implementers who
process arbitrary resources must support the full FHIR-specific
extension set. Declining preserves a single, predictable conformance
target.

---

### Recommendation

**Recommended disposition:** A, executed in a way that leaves room for B.

The ticket is well-grounded: today the page genuinely does not state
which FHIR-specific extensions are required of a conformant FHIRPath
implementation, and the WG has a clear, persuasive history of
documenting subsets exactly for situations like this
([FHIR-13711](https://jira.hl7.org/browse/FHIR-13711),
[FHIR-19545](https://jira.hl7.org/browse/FHIR-19545)). Disposition C
under-serves implementers who today are forced to infer requirements
from incidental resource usage. Disposition B is the more modern
mechanism and is clearly where the WG is heading on related expression
questions ([FHIR-49590](https://jira.hl7.org/browse/FHIR-49590)), but
adopting *only* B leaves the FHIRPath chapter without the explicit
"required vs optional" classification the reporter is asking for —
feature codes are useless without an enumerated list of features to
negotiate over. The pragmatic recommendation is therefore to adopt A
(produce the conformance table and named subsets the reporter asks
for), and *as part of the same edit* register the resulting
classifications as feature codes per B so that
`CapabilityStatement`-based negotiation continues to work. This should
also be coordinated with the resolution of
[FHIR-54742](https://jira.hl7.org/browse/FHIR-54742) so that the
General Service API's status is consistent across both tickets.
