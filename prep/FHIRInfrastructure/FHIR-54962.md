## Summary

The ticket asks the FHIR Infrastructure work group to improve the FHIR
Mapping Language (FML) / StructureMap specification text in the "type
wrangling" section. Specifically the reporter (Brian Postlethwaite)
wants:

1. Additional explanation of why the *string representation* of a value
   is (or could be) relevant to type conversions.
2. A definition of what an engine must do when a type cannot be mapped —
   is it an error that fails the whole map, or should the engine log
   and skip the property, and is this behaviour a configurable option
   per engine.
3. Explicit mention that type wrangling can also be satisfied by groups
   that map between two types — groups carrying the `<<types>>`
   stereotype — with a reference to the default mapping groups section.
4. A restructure of the spec so that *Default mapping groups* becomes a
   child of the *Groups* section, letting the type-wrangling prose
   reference it backwards rather than forwards; the reporter notes that
   the difference between `types` and `type+` is defined there.
5. Cross-references from the *target transform* section back into type
   wrangling and default mapping groups, so that the narrative of how
   data is copied from source to target is complete.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced by the cross-reference service, so the request stands alone in
the current data set.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the FML / StructureMap specification narrative to action each of
the five requests verbatim:

1. In the **Type Wrangling** section, add a paragraph that explains the
   role of the string (canonical / lexical) representation in type
   conversion — i.e., that conversions between primitives and between
   primitive-like datatypes commonly route via their string form, and
   that this is what governs things like preservation of precision,
   timezone, and leading zeros.
2. In the same section, add a normative paragraph on failure mode: if
   the engine cannot find a conversion for a source type into the
   target type, the default behaviour is that the map fails (error),
   unless the engine offers a conformance-declared option to log and
   skip the property. Flag this as an engine-level configuration point
   and add a conformance bullet.
3. Add to *Type Wrangling* a subsection stating that before falling
   back to built-in conversions, the engine considers user-defined
   groups with the `<<types>>` stereotype whose input/output datatypes
   match, and that the **Default Mapping Groups** section defines the
   built-in set of such groups (including the `types` vs `type+`
   distinction).
4. Restructure: move *Default Mapping Groups* to be a child section of
   *Groups* so that *Type Wrangling* (which follows) can reference it
   backwards rather than forwards. Update all internal anchors.
5. In the *Target Transform* section, add cross-references to *Type
   Wrangling* and *Default Mapping Groups* explaining how copy-style
   rules implicitly invoke type wrangling when source and target
   datatypes differ.

#### Justification

All five items are clarifications and structural improvements to
normative narrative that already exists. They do not change the
semantics of FML — they make the existing semantics explicit (what
happens on unmapped types, how `<<types>>` groups participate, how the
default mapping groups fit in). Clarifying engine behaviour on unmapped
types is a genuine interoperability win: today two conformant engines
can legitimately disagree on whether an unmapped type aborts the map
or is silently skipped. Making this a named configuration option is
consistent with how other FHIR engines document conformance choices
(e.g., FHIRPath strict-mode, validator best-practice levels).

---

### Disposition B: Alternative Approach

#### Proposal

Accept items 1, 3, 4, and 5 (prose expansion, forward → backward
reference fix, structural move of *Default Mapping Groups* under
*Groups*, and target-transform cross-references) but split item 2 — the
behaviour on unmapped types — into a dedicated follow-up ticket for
broader community review.

For item 2 specifically, mark the spec with a "this is engine-defined
today; a normative rule is under discussion" box and link to the new
follow-up ticket, rather than inventing a normative default and a new
conformance-declared configuration option in a triage-level change
request.

#### Justification

Items 1, 3, 4, and 5 are editorial / structural and have no downstream
implementation impact — they can be merged with low risk. Item 2, by
contrast, establishes normative engine behaviour and introduces what
amounts to a new engine conformance switch. Fixing that via a ticket
that currently has no comments, no Zulip discussion, and no
cross-references carries the risk of baking in a default (fail vs.
skip) that existing engines don't match. Splitting lets the work group
ship the uncontroversial improvements immediately while giving the
engine-behaviour question the discussion it needs.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as scoped. Close with a note that FML / StructureMap
narrative issues of this breadth should be opened as separate, focused
tickets — one per concrete change — rather than as a single omnibus
"clarifications" request, because the dispositions and voting records
need to be traceable per change.

#### Justification

This ticket bundles one restructure (move *Default Mapping Groups*),
two cross-reference additions, one prose expansion, and one normative
behavioural question (engine failure mode on unmapped types) into a
single "clarifications" change request with no comments and no linked
discussion. That makes tracking, voting, and back-references hard, and
it risks the normative item riding in on the back of the editorial
ones. Declining with a request to re-file as discrete tickets is a
defensible hygiene action and does not reject any of the underlying
requests on their merits.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the reporter's intent for the editorial and
structural items — which are clear wins and low-risk — while
recognising that the "what happens on unmapped types" question is a
genuinely normative decision that should not be settled inside an
otherwise editorial ticket without community discussion. Accepting the
bulk of the ticket keeps the reporter's momentum and improves the spec
now; splitting out the engine-behaviour question protects existing
implementers and invites the broader Zulip / work-group conversation
that a new conformance-affecting rule deserves. If the work group
wants to keep the ticket atomic, Disposition A is the next-best
choice; Disposition C is recommended only if the group prefers a
stricter one-change-per-ticket policy going forward.
