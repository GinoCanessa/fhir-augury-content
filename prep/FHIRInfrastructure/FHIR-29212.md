## Summary

The reporter asks how a server can declare, in its `CapabilityStatement`, that
it supports asynchronous interactions (the FHIR async request pattern), and
what granularity such a declaration should have — specifically, that it
should be expressible at the server level, per-resource, and per-operation.
The current `CapabilityStatement` resource in R4/R4B/R5 has no first-class
mechanism for advertising async support, so implementers cannot
machine-discover whether `Prefer: respond-async` is honoured for a given
endpoint, resource, or operation.

No other Jira tickets are explicitly linked or surfaced as cross-references by
Augury. The ticket is, however, part of the broader CapabilityStatement2
work-stream (label `CapabilityStatement2`) — the most recent FHIR-I action
(2025-10-27, jmandel) reverts an earlier 2021 "Not Persuasive with
Modification" disposition and instead defers the substantive solution to the
App Feature IG (which will define the general feature-discovery mechanism)
and the API Incubator IG (which will define the concrete async-related
features). The ticket is to be brought back to FHIR-I once those IGs are
published. The Zulip references are all FHIR-I housekeeping threads (agenda,
status-report) that simply list this ticket as outstanding work.

## Details

**Description:**

> How do I state in the CapabilityStatement that async calls are supported?
> At what level can I state this? It should be possible on server, resource
> and operation level.

**Comments:**

- **mckenzie (2021-01-27):** Asks the reporter to experiment with the
  proposed new "feature" approach to describing capabilities in
  CapabilityStatement2 to see whether it would meet the requirement.

- **bvdh (2021-06-14):** Reports back to Lloyd McKenzie that he has looked
  at the mechanism and it would work.

- **bvdh (2021-06-21):** Concrete proposal — add features (codes in the
  `capability-features` valueset) called `synchronous` and `asynchronous`,
  each taking the values `true`/`false`, allowing servers to be searched
  for sync vs. async support. The feature should be allowed at `rest`,
  `rest.resource`, `rest.interaction`, `rest.operation`,
  `rest.resource.interaction`, and `rest.resource.operation` levels, with
  the deepest occurrence overriding shallower ones (e.g., a feature on
  `rest.resource.interaction` overrides one on `rest.interaction`).

- **jmandel (2025-10-27):** Reverts the previous 2021-06-28 "Not
  Persuasive with Modification" resolution (Ewout Kramer / Josh Mandel,
  vote 11-0-0; impact: compatible substantive; category: enhancement) which
  had committed to adding Bas's proposed codes to the CapabilityStatement2
  `capability-features` code system.

- **jmandel (2025-10-27):** After four years FHIR-I has a better
  understanding of how to resolve this — the expectation is that the App
  Feature IG will define a general feature mechanism and the API Incubator
  IG will define the async-related features.

- **jmandel (2025-10-27):** Tags Corey Spears (the assignee) and asks that
  once the App Feature IG is published and the mechanism for externally
  defined features is clear, the ticket be brought back to FHIR-I. Notes
  general agreement that features for discovery (including discovery of
  async support for specific operations or general resource queries) should
  live inside the API Incubator IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the concrete mechanism Bas proposed in 2021 inside the App Feature
IG / API Incubator IG (per the 2025-10-27 FHIR-I direction): define two
boolean features, `synchronous` and `asynchronous`, in (or referenced
from) the `capability-features` code system, and allow them to be asserted
on `CapabilityStatement.rest`, `rest.resource`, `rest.interaction`,
`rest.operation`, `rest.resource.interaction`, and
`rest.resource.operation`. Specify that the deepest applicable feature
overrides shallower ones (a feature on `rest.resource.interaction`
overrides one on `rest.interaction`, and so on). Resolve FHIR-29212 once
those IGs publish the codes; the resolution text in core FHIR points
implementers at the App Feature IG / API Incubator IG for async
discovery.

#### Justification

This is what the reporter asked for, what FHIR-I voted 11-0-0 to do in
2021, and what the 2025-10-27 comment thread realigns toward (with the
home moved from "CapabilityStatement2" to "App Feature IG + API Incubator
IG"). The granularity matches real implementations: async is often
operation-specific (e.g., bulk-data `$export`, `$apply`) rather than a
server-wide property, so per-operation granularity is necessary; the
override rule keeps simple cases compact (declare once at `rest`) while
allowing precise opt-in/opt-out at deeper levels. Doing this in an IG
rather than core FHIR also avoids re-opening a normative resource and
matches the precedent set by other capability-features work.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of two booleans (`synchronous`, `asynchronous`), define a single
multi-valued feature — e.g., `async-support` with codes such as
`not-supported`, `supported`, `required` (and possibly `preferred`) — and
permit it on the same `rest.*` paths Bas listed, with the same
"deepest-wins" override rule. Pair this with a separate feature naming
the *async pattern* in use (e.g., `async-pattern` ∈
{`fhir-r5-async`, `bulk-data`, …}) so that consumers can distinguish the
core `Prefer: respond-async` flow from the SMART Bulk Data flow. Both
features live in the App Feature IG with concrete codes contributed by
the API Incubator IG.

#### Justification

Two parallel booleans are awkward in practice — a server that supports
both sync and async would assert both `synchronous=true` and
`asynchronous=true`, which is verbose and leaves the third real-world
state (async required, sync refused) ambiguous. A single tri-/quad-state
feature collapses the matrix and lets a server signal "async only" for
endpoints like Bulk Data. Splitting the *capability* (does it support
async at all?) from the *pattern* (which async protocol?) also avoids
conflating "FHIR async API" with "Bulk Data" — a real source of confusion
today. This better honours Lloyd McKenzie's earlier nudge to use the
`feature` mechanism rather than adding ad-hoc booleans, and it scales as
new async patterns appear without renaming the underlying capability.

---

### Disposition C: Decline

#### Proposal

Close FHIR-29212 with a "Persuasive – Will Not Fix" disposition: the
async request pattern is already documented in core FHIR
(`async.html` / `Prefer: respond-async`), and capability discovery for
async is being addressed by parallel, IG-level mechanisms — the App
Feature IG (general feature-discovery framework) and the SMART Bulk Data
IG (which has its own well-defined capabilities advertisement). Direct
the reporter to use the App Feature IG / Bulk Data CapabilityStatement
guidance once published, and do not add a core-FHIR mechanism.

#### Justification

The ticket has been open since 2020 with no consensus on a core-spec
mechanism, and the 2025-10-27 comments effectively concede that the right
home is an IG, not core FHIR. Declining (rather than deferring) avoids
keeping a long-running CR open against R5/R6 for work that will be
delivered elsewhere, and acknowledges that two of the three needed
async-discovery surfaces (Bulk Data; per-operation features such as
`$export`) are already covered by existing guidance.

---

### Recommendation

**Recommended disposition:** B

The 2025-10-27 FHIR-I commentary explicitly endorses moving this work
into the App Feature IG / API Incubator IG, so Disposition C (outright
decline) is at odds with current direction. Disposition A is the literal
request and is defensible, but the dual-boolean shape Bas proposed
captures the matrix (server supports sync, supports async, both, or
async-only) less cleanly than a single state-valued feature, and it does
not address the real-world fact that "async" today means at least two
different protocols (the FHIR async pattern and SMART Bulk Data).
Disposition B keeps everything Disposition A proposes — same locations,
same override rule, same IG home — but lands a more durable coding that
won't need a follow-up ticket the first time someone asks "but which
async?". The work is small (two coded features in the App Feature IG /
API Incubator IG with documented placement and override semantics) and
fits cleanly inside the framework FHIR-I has now committed to.
