## Summary

The reporter observes that `Element.id` is currently typed as `string` in the
FHIR Core specification, even though its description constrains the value to
not contain spaces — i.e., it behaves like the `id` primitive type (regex
`[A-Za-z0-9\-\.]{1,64}`) rather than a free-form `string`. The ticket
requests that the type of `Element.id` be changed from `string` to `id` so
that the type definition matches the documented constraints. No related
Jira tickets were surfaced by the cross-reference index, but the question
has been raised repeatedly on Zulip across multiple FHIR versions, and a
prior partial fix in R4B is acknowledged by the spec leads (see "Related
Zulip Discussions" below).

## Details

**Description:**

The type of `Element.id` should be changed to `id`. Although the
description says it should not contain spaces, the `string` type definition
violates the specification.

**Comments:**

- **brian.postlethwaite (2026-01-22):** The `id` type can't contain `:`
  characters, which are used in StructureDefinition's element ids. It
  could however have an invariant added to ensure there are no spaces in
  it.

## Related Zulip Discussions

### [implementers > Element.id type](https://chat.fhir.org/#narrow/stream/implementers/topic/Element.id%20type)

Nick George raised the issue in December 2022, noting that in R4 the
`Element.id` definition uses `"type": [{ "extension": [{ "url":
"http://hl7.org/fhir/StructureDefinition/structuredefinition-fhir-type",
"valueUrl": "string" }], "code": "http://hl7.org/fhirpath/System.String"
}]` — i.e., it relies on the `structuredefinition-fhir-type` "compiler
magic" extension to declare the FHIR type as `string`. He pointed out
that R4B silently changes this to `id`, while the human-readable spec
page still describes the field as `string`. Grahame Grieve confirmed
that "R4B is broken, and may be fixed in technical correction," and
Nick (along with others) argued that the "compiler magic" extension
mechanism is itself a recurring pain point. No firm resolution was
reached in the thread, but the spec lead acknowledged the inconsistency.

Participants: Nick George, Grahame Grieve, Lloyd McKenzie, Ankit Kumar
Agarwal.

### [implementers > Element.id](https://chat.fhir.org/#narrow/stream/implementers/topic/Element.id)

Earlier (2017) thread observing the same issue between STU3 and DSTU2 —
`Element.id` was typed `string` in STU3 but had been `id` in DSTU2. The
discussion established that the change was deliberate but never well
explained, and is the historical root of the current confusion.

Participants: Michele Mottini and others.

### [implementers > Resource.id Data Type](https://chat.fhir.org/#narrow/stream/implementers/topic/Resource.id%20Data%20Type)

Grahame Grieve clarified: "Element.id is different — it's allowed to be
any valid string with no spaces." This explicitly contradicts the
machine-readable type (`string`, which permits spaces) and supports the
reporter's claim that the documented constraint and the declared type
disagree.

Participants: Grahame Grieve and others.

### [fhir/infrastructure-wg > why is there only an extension for element.id type](https://chat.fhir.org/#narrow/stream/fhir%2Finfrastructure-wg/topic/why%20is%20there%20only%20an%20extension%20for%20element.id%20type)

A 2018 FHIR-I conversation in which Eric Haas hit a code-generation
exception because `Element.id`'s type is encoded only via the
`structuredefinition-fhir-type` extension rather than a normal type code.
Grahame's response — "this is where the compile magic lives: Element.id
doesn't have extensions" — explains the historical reason: `Element.id`
sits at the bottom of the type hierarchy and cannot itself carry the
extension that would otherwise narrow it. This is directly relevant to
why "just change the type to `id`" is non-trivial.

Participants: Eric Haas, Grahame Grieve.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the declared type of `Element.id` (and, for symmetry, the
`id` element on every datatype that inherits from `Element` —
including `BackboneElement.id`, `Extension.id`, etc.) from the
current "code = `http://hl7.org/fhirpath/System.String`,
`structuredefinition-fhir-type` extension = `string`" pattern to the
equivalent pattern declaring `id` as the FHIR type. In practice this
means updating the source `element.profile.xml` /
`element.profile.json` (and the BackboneElement equivalents) so that
the `Element.id` ElementDefinition.type carries the
`structuredefinition-fhir-type` extension with value `id` instead of
`string`. Regenerated published artifacts (StructureDefinitions,
schema, JSON schema, type-generation outputs) follow automatically.

#### Justification

This makes the machine-readable definition match the prose
constraint that `Element.id` "should not contain spaces." It is also
consistent with the partial change already made in R4B that
brian.postlethwaite and Grahame acknowledged on Zulip ("R4B is
broken, and may be fixed in technical correction"), and would
remove a long-standing confusion that has surfaced in at least four
distinct Zulip threads since 2017. Because this is logged as a
*Technical Correction*, it is in scope for a non-substantive fix
that brings the definition in line with the documented intent.

---

### Disposition B: Alternative Approach — Keep `string` type, add an invariant

#### Proposal

Leave the declared FHIR type of `Element.id` as `string` (preserving
the current FHIRPath `System.String` mapping and avoiding any change
to derived schemas or generated code), but add an explicit invariant
to the `Element` StructureDefinition requiring that `Element.id`
match a permissive pattern that allows the characters actually used
in StructureDefinition element ids — at minimum
`[A-Za-z0-9\-\.\:]{1,64}` (i.e., the `id` regex plus `:` for
slicing). Update the `Element.id` description to point at the
invariant rather than relying on prose.

#### Justification

This is the path explicitly suggested by brian.postlethwaite in the
ticket comment: "It could however have an invariant added to ensure
there are no spaces in it." Crucially, it accommodates the
real-world usage — element ids in StructureDefinition slicing
(e.g. `Observation.value[x]:valueQuantity`) contain `:` characters,
which are **not** permitted by the standard `id` primitive regex
(`[A-Za-z0-9\-\.]{1,64}`). Switching the type to `id` (Disposition
A) would therefore make every existing `StructureDefinition.snapshot`
and `.differential` element technically invalid against its own
declared type. An invariant captures the actual constraint
(no-spaces, length ≤ 64, allow `:`) without breaking conforming
content or forcing changes to tooling. It is also the lower-risk
technical correction.

---

### Disposition C: Decline

#### Proposal

Decline the change. Mark the ticket "Not Persuasive — Will Not Fix"
on the grounds that the existing definition is intentional: the
`structuredefinition-fhir-type` extension already carries the
narrower FHIR type when needed, the human-readable description
documents the no-spaces constraint, and the surface area at risk
(every `id` element on every Element-derived datatype, in every
published StructureDefinition) outweighs the benefit of cleaning up
the type name. Recommend the reporter rely on the prose constraint
as today.

#### Justification

The current behaviour has stood since DSTU2 → STU3 and is
well-known to implementers; tooling has been written against it.
Switching the type to `id` would make existing
StructureDefinition ids that contain `:` non-conforming
(brian.postlethwaite's point), and any change — even adding an
invariant — has a non-trivial impact on validators. If neither A
nor B is willing to accept the breakage / churn, declining is the
defensible status-quo option.

---

### Recommendation

**Recommended disposition:** B

Disposition A (the literal request) cannot be applied without
breaking the very content it is trying to describe: every
`StructureDefinition.snapshot.element.id` and
`differential.element.id` containing a slice marker uses `:`, which
the `id` primitive regex forbids. Disposition C ignores the
legitimate complaint that the declared type contradicts the
documented constraint. Disposition B captures the real intent —
"no spaces, bounded length, allow the punctuation that actually
appears in StructureDefinition element ids" — without invalidating
existing content, requires no change to generated schemas or type
hierarchies, and is exactly what the FHIR-I co-chair (Brian
Postlethwaite) suggested in the ticket comment. It is the smallest
technical correction that resolves the reporter's concern, and the
workgroup should pursue it unless a strong case is made for
preserving the status quo.
