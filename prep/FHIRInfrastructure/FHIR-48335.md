## Summary

The ticket proposes adding a **definition-based pre-population** technique
to the Structured Data Capture (SDC) IG, as a counterpart to the existing
definition-based *extraction* mechanism. The new technique would reuse the
`itemPopulationContext` extension as the input data (a complete resource
or set of resources, not a fragment), and then walk the `Questionnaire.item.definition`
references to pull values directly from the context resource â€” instead of
requiring the form author to author an `initialExpression` (FHIRPath) for
every leaf item that simply mirrors what the `definition` already points
at.

Brian Postlethwaite (reporter) clarifies in comments that the proposal is
**additive** â€” `initialExpression` would still be available, and
definition-based population would be a `MAY`. The motivation is to avoid
having implementations evaluate "essentially duplicate expressions" when
the leaf path is already implied by the item's `definition`.

The ticket was originally voted **Considered for Future Use** (Postlethwaite/Hay, 4-0-0)
on 2024-11-28 with the comment "Will take this up in a future release."
On 2026-03-26 Lynn Laakso reverted that resolution and re-opened it
**post STU4 publication** so SDC can pick it up in the next release cycle.

There are no linked or cross-referenced Jira tickets and no GitHub
artifacts associated with this issue. Three Zulip threads on the
`#questionnaire` stream provide community context: implementer interest
in a definition-based mechanism, an existing Clinical Reasoning module
implementation by Brenin Rhodes (on which Smile/CDS work depends), and a
related design debate about whether `itemPopulationContext` makes sense
on non-repeating items â€” a debate whose answer turns directly on whether
definition-based population is on the roadmap.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a **definition-based pre-population** mechanism to the SDC IG's
`populate.html` page as a `MAY`-conformance peer of expression-based
population. Specification points:

1. **Trigger.** When an `item` (or ancestor `item` / Questionnaire
   root) carries an `itemPopulationContext` whose expression returns
   one or more FHIR resources or fragments, and the item or its
   descendants carry `Questionnaire.item.definition` pointing into a
   resolvable `StructureDefinition` rooted in (or compatible with) the
   context resource type, the populate engine MAY walk the `definition`
   path against the context to obtain the value rather than requiring
   an `initialExpression`.
2. **Composability with `initialExpression`.** Both extensions remain
   valid on the same item. If both are present, `initialExpression`
   wins (consistent with current "expression takes precedence" patterns
   in SDC). Document this precedence explicitly.
3. **Type mapping.** Reuse and *extend* the existing definition-based
   extraction type-mapping rules (per Postlethwaite's 2024-09-21
   comment) so that the same FHIR-type â†’ Questionnaire-item-type table
   serves both directions. Make the rules a shared subsection that
   `populate.html` and `extraction.html` both reference.
4. **Repetition semantics.** When the context returns multiple
   instances and the item repeats, populate one repetition per
   instance (mirroring expression-based population). When the context
   returns multiple instances and the item does not repeat, raise the
   same error already specified for `itemPopulationContext`.
5. **Description clarification.** Update the `itemPopulationContext`
   extension description to make explicit (per Postlethwaite 2025-04-01)
   that the expression may legitimately return *fragments* (e.g.,
   `Patient.contact`, `Patient.identifier`), not only complete
   resources.

#### Justification

Every implementer who has weighed in (Postlethwaite, Rhodes, Vanraes)
wants this; one production implementation already exists in the Clinical
Reasoning module, and Smile has "a cut of this." Specifying the existing
behaviour reduces implementation divergence and unlocks the
StructureDefinition-driven Questionnaire-authoring patterns that
Interactive CDS already depends on. Keeping it `MAY` and additive
preserves backward compatibility with every existing populate
implementation. Reusing the extraction type-mapping table avoids
inventing a parallel mapping and keeps populate â‡„ extract symmetric, a
core SDC design principle.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a new pre-population *technique*, **define a
separate `definition-based-populate` profile of Questionnaire** (paralleling
the eventual `definition-based-extract` profile that Lloyd McKenzie
alluded to in the `repeats=false` thread). Within that profile:

- `itemPopulationContext` is **required** at the root (or at every
  group whose children carry `definition`).
- Items that participate in definition-based population MUST carry
  `Questionnaire.item.definition`; they MUST NOT carry
  `initialExpression` (mutual exclusion within the profile).
- The base SDC `populate` page gains only a short note pointing to the
  profile, and the `itemPopulationContext` description is clarified to
  cover fragments.
- Engines advertise support via a new capability (e.g., a
  `populate-mode` parameter on `$populate`).

#### Justification

A profile-based approach gives implementers a clean conformance target
("does this server do definition-based population?") rather than a
silent `MAY` whose presence is unobservable to form authors. It also
sidesteps the precedence question raised by Postlethwaite (no need to
pick a winner between `initialExpression` and definition walk because
they cannot coexist within the profile). It matches Lloyd McKenzie's
suggestion in the `repeats=false` thread to clean up the root-level
`itemPopulationContext` once a definition-based profile exists.
Trade-off: profile creation is heavier work, may delay availability,
and existing implementations (Clinical Reasoning module, Smile) would
need a small migration to claim conformance to the profile rather than
to the base SDC populate behaviour.

---

### Disposition C: Decline (re-defer)

#### Proposal

Re-instate the prior **"Considered for Future Use"** resolution and
defer until a subsequent SDC release after the population-related
issues raised in `itemPopulationContext and repeats=false?` (and
Postlethwaite's "variations on re-populate") are scoped together. In
the interim, do not change the spec; implementations that have
already shipped the behaviour (Clinical Reasoning module, Smile)
continue as vendor extensions.

#### Justification

The original 2024-11-28 vote (4-0-0) was unanimous in deferring this,
and the only material change since then is STU4 publication. Several
adjacent design questions are still unsettled (root-level
`itemPopulationContext`, fragment vs. resource semantics,
`initialExpression` precedence, capability advertisement). Specifying
definition-based population in isolation risks locking in a design
that conflicts with the broader populate / re-populate work
Postlethwaite has flagged as "first on my list." Declining now is
defensible; a coherent design package later is preferable to
piecemeal additions.

---

### Recommendation

**Recommended disposition: A**

The ticket has been re-opened *specifically* because STU4 is now out
and the work group committed to picking it up next cycle. There is a
real implementation in the Clinical Reasoning module, a second one in
Smile, demonstrated implementer demand on Zulip (Vanraes 2025-12), and
a clear, narrow design proposal from the reporter. Disposition A also
unblocks the `repeats=false` debate (Lynch / Rhodes / McKenzie /
Postlethwaite) by giving `itemPopulationContext` a documented
non-`variable` job to do, which is the very thing that thread was
waiting on.

Disposition B (profile) is attractive long-term and worth keeping on
the table as a follow-on, but standing up a new conformance profile
should not be a precondition for documenting behaviour two
implementations already ship â€” that is exactly the pattern SDC has
historically followed (specify the `MAY`, profile later if needed).
Disposition C (re-defer) ignores the explicit Laakso/Postlethwaite
intent to act on this in the next release and would leave the
`repeats=false` thread permanently in limbo.

The work group should adopt A as the starting point and treat the
`itemPopulationContext` description clarification (fragments allowed)
and the shared populate/extract type-mapping subsection as in-scope
companion edits, since both are referenced in the existing comments.
