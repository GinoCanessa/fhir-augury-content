## Summary

The reporter notes that the FHIR Resource Type page links out to the new
"FHIR Testing incubator" — the relocated home of `TestScript` and `TestPlan`
(now also `TestReport`) after they were moved out of FHIR core into the
`fhir-testing-ig` additional-resource IG. The URL used in the build,
`https://build.fhir.org/ig/hl7/fhir-testing-ig/index.html`, returns
HTTP 404. The same defect was independently filed ten days later as
[FHIR-54820](https://jira.hl7.org/browse/FHIR-54820) (Highest priority,
also Triaged, FHIR Infrastructure), reporting a broken "see also Testing
Resources" link with the same target URL — strongly suggesting a single
underlying cause and a single fix that resolves both tickets. The Zulip
record (committers > "Additional Resources", testing > "Testing IG")
shows that the testing IG was the first pilot of the new
"additional-resources IG" model managed by Grahame Grieve, and that
working links to its content actually live under the localised path
`https://build.fhir.org/ig/HL7/fhir-testing-ig/en/…` (e.g.
`en/StructureDefinition-TestPlan.html`). The bare `/index.html` target
configured in `source/fhir.ini` does not match that publication layout,
which is consistent with the 404.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir`, edit `source/fhir.ini` line 9 (in the `[additional-ig-list]`
section) to point at the actually-published landing page of the FHIR
Testing IG. Concretely, change:

```
testing=https://build.fhir.org/ig/hl7/fhir-testing-ig/index.html
```

to the working form observed in production usage and in the sister
file `source/ballot-intro.html`:

```
testing=https://build.fhir.org/ig/HL7/fhir-testing-ig/en/index.html
```

Then run a repo-wide grep of `source/` for the literal
`fhir-testing-ig/index.html` and `fhir-testing-ig` (without `/en/`) and
update any other authored pages found to use the `…/en/…` form. Close
[FHIR-54820](https://jira.hl7.org/browse/FHIR-54820) as a duplicate
fixed by the same change.

#### Justification

The ticket is a verifiable technical correction — the URL 404s today and
the working URL (`…/fhir-testing-ig/en/…`) is already used elsewhere in
the same repo (`source/ballot-intro.html` lines 92–94) and is the form
cited by an external implementer in the testing > "Testing IG" Zulip
thread. The change is one-line, fully contained within
`HL7/fhir/source/`, hits the bespoke-publisher's authored config (the
briefing confirms `source/fhir.ini` is hand-authored, not generated),
and resolves both FHIR-53898 and the higher-priority duplicate
FHIR-54820 simultaneously. There is no design question to debate.

---

### Disposition B: Alternative Approach

#### Proposal

Same surgical fix as Disposition A, but resolve the link to a
**version-stable canonical** rather than the `build.fhir.org` continuous
build URL. Set `source/fhir.ini`'s `[additional-ig-list]` `testing`
entry to the HL7 published canonical for the testing IG — e.g.
`https://hl7.org/fhir/uv/testing/` (or whatever canonical FMG / FHIR-I
agree on for the first balloted publication of `fhir-testing-ig`) — and
update `source/ballot-intro.html` lines 92–94 to the matching
versioned `StructureDefinition-Test{Plan,Report,Script}.html` paths
under that canonical. Where no balloted publication yet exists, leave
the `build.fhir.org/.../en/` form in place but add a TODO comment in
`fhir.ini` so it is updated at the next ballot cut.

#### Justification

R6 ballots are versioned artifacts; their outbound links should not
silently rot when the testing IG re-publishes or moves its CI URL. The
committers > "Additional Resources" thread shows the additional-resources
IG model is brand new and that work groups (and their published URLs) may
shift. Pointing the Resource Type page at a canonical hl7.org URL
insulates the FHIR core ballot from churn in `build.fhir.org` paths and
from locale-segment changes (the very issue that broke the link in the
first place). Trade-off vs. A: requires confirming the canonical with
FMG, and may have to wait for the testing IG's first balloted
publication before it can be applied; A is faster and unblocks the 404
today.

---

### Disposition C: Decline

#### Proposal

Decline as not the FHIR Infrastructure work group's responsibility —
the broken target lives in the `fhir-testing-ig` IG repository, not in
`HL7/fhir`, and the IG owners should re-publish so that
`/fhir-testing-ig/index.html` resolves.

#### Justification

This disposition is recorded for completeness. It is **not** defensible
on the facts: the offending URL is *authored in `HL7/fhir`*
(`source/fhir.ini` line 9 in the cached clone) and is what the FHIR
core publisher emits onto the Resource Type page. The broken link is
therefore directly under FHIR Infrastructure's editorial control as
the owner of `source/fhir.ini` and `source/ballot-intro.html`. Pushing
the fix to the testing-IG owners would also leave FHIR-54820 (Highest
priority) unresolved.

---

### Recommendation

**Recommended disposition:** A

The defect is real, reproducible, and entirely within FHIR-I's
authoring surface. Disposition A is a one-line edit to
`source/fhir.ini` (with a small grep-and-fix sweep) that closes both
FHIR-53898 and the higher-priority duplicate FHIR-54820 with no design
risk and no cross-repo coordination. Disposition B's canonical-URL
argument is sound in principle and should be tracked separately as a
follow-up once `fhir-testing-ig` has a balloted publication, but
holding the 404 fix on that decision penalises ballot reviewers who
hit the broken link today. Disposition C should be rejected outright.
