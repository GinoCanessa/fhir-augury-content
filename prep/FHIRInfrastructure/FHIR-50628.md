## Summary

The reporter (Weiyu Zhang) flags a minor inconsistency in the FHIR Core
Subscriptions Framework page: in section 3.9.8, the Subscription channel
type is rendered as `REST-hook` (uppercase `REST`) inside a security
best-practices paragraph, whereas the rest of the specification — and the
canonical channel-type code — uses lowercase `rest-hook`. The reporter
characterises the issue as "very minor / almost not worth changing" but
asks that the spec text be aligned with the canonical token. There are no
cross-referenced Jira tickets, Zulip discussions, or GitHub items
recorded in FhirAugury for this issue, so it stands on its own as an
isolated editorial / Technical Correction request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the `HL7/fhir` repo, in the Subscriptions Framework authoring page
under `source/` that renders as section 3.9.8 of the published spec
("channels — security best-practices" paragraph), change the single
occurrence of `<tt>REST-hook</tt>` to `<tt>rest-hook</tt>`. The edited
sentence becomes:

> When implementing channels, implementers should be aware of
> security best-practices for those protocols. For example, when
> implementing the `rest-hook` channel type, it is strongly recommended
> to use HTTPS instead of HTTP. …

No other text in the paragraph changes; no anchors, IDs, or links are
affected; no StructureDefinition, value set, or example needs updating.
Resolve as Persuasive — Technical Correction.

#### Justification

The token `rest-hook` is the canonical `Subscription.channel.type` code
used everywhere else in the Subscriptions Framework page and in the
underlying ValueSet/CodeSystem for channel types. Codes in FHIR are
case-sensitive, so rendering the same channel type as `REST-hook` in one
spot creates a real (if small) risk that a reader copies the
mis-capitalised form into a `Subscription.channel.type` element — which
would not validate. The FhirCore briefing's "Update narrative / prose"
recipe applies cleanly: it is a single `source/*.html` edit with no
generated artefacts to regenerate and no cross-repo impact, satisfying
the bar for a Technical Correction.

---

### Disposition B: Alternative Approach

#### Proposal

Make the literal capitalisation fix above **and** do a pass over
section 3.9 (the whole Subscriptions Framework chapter) for any other
inconsistent renderings of channel-type codes (`rest-hook`,
`websocket`, `email`, `message`) — especially places that wrap them in
`<tt>` / `<code>` and may have been hand-typed rather than templated.
File the fix as Persuasive with Modification, with the broadened scope
documented in the resolution. The change is still a single-file (or
small handful of files) edit under `source/` on `HL7/fhir`.

#### Justification

The reporter explicitly notes "almost not worth changing" for a single
token, which suggests the underlying concern — implementer confusion if
they copy a mis-cased code — is bigger than this one instance. A small
audit of section 3.9 costs little once the page is open in the editor
and prevents a future near-duplicate ticket. Trade-off vs. Disposition
A: slightly more reviewer time and a marginally larger diff, but no
substantive risk because every change is a code-string normalisation
that can be verified mechanically against the channel-type CodeSystem.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** on the grounds that channel-type code
values are case-sensitive at the data layer (a `Subscription` instance
must use the exact lowercase code regardless of how prose renders it),
the surrounding paragraph is plainly informative narrative, and the
reporter themselves describes the issue as "almost not worth changing".
Close without a spec change.

#### Justification

The defensible position is that this is a purely cosmetic prose
inconsistency that does not affect interoperability: `Subscription.
channel.type` validation is driven by the bound CodeSystem, not by how
the narrative happens to capitalise the token in one sentence.
Declining keeps Technical Correction throughput focused on issues that
change conformance, examples, or normative text. This rationale is
weaker than for many declines because the fix is genuinely trivial,
but it is the standard answer for "purely typographical" tickets.

---

### Recommendation

**Recommended disposition:** A

This is the textbook case for a quick Persuasive — Technical Correction
resolution. The change is one token in one paragraph of one
`source/*.html` page in `HL7/fhir`; the FhirCore briefing's "Update
narrative / prose" recipe applies directly; there are no
cross-references, no Zulip debate, no linked tickets, no generated
artefacts, and no cross-repo touch points to coordinate. The reporter's
own "almost not worth changing" caveat is exactly the kind of friction
the Technical Correction process is designed to absorb so that
implementers see consistent, copy-pasteable codes. Disposition B's
broader audit is reasonable but is best handled as a follow-up if the
editor notices additional cases while making the fix; bundling it
up-front risks scope creep for a ticket the reporter intentionally
flagged as minor. Disposition C is hard to defend given how cheap the
fix is and the small but real risk of a reader copying `REST-hook` into
a `Subscription.channel.type` field.
