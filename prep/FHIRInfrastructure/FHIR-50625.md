## Summary

The reporter notes an internal contradiction in the FHIR Core
`SubscriptionStatus` resource page regarding `SubscriptionStatus.notificationEvent.eventNumber`.
The "Event Numbering" section's table marks `eventNumber` as **Required** for
both *best-effort* and *guaranteed* delivery channels (with different semantics:
a global monotonic index vs. a relative within-notification index). The
"Notification Types" section, however, uses example-row prose stating that
`eventNumber` is "Required for subscriptions with best-effort delivery,
optional for subscriptions with guaranteed delivery" and links to the Event
Numbering section for details. The reporter asks whether the Event Numbering
section should be updated to align with the (more permissive) examples — i.e.,
make `eventNumber` optional under guaranteed delivery.

Important context comes from the Zulip conversation that triggered this work
and the related Backport-IG ticket [FHIR-48808](https://jira.hl7.org/browse/FHIR-48808),
which was filed against the R5 Subscriptions Backport IG for the same
inconsistency. FHIR-48808 was resolved **Persuasive** with a clear direction
from the resource owner: the cardinality stays **1..1** (still required), but
the *meaning* of `eventNumber` is allowed to be a notification-local index
(e.g., 1, 2, 3 …) when the channel uses guaranteed delivery, and channels
**SHOULD** only allow deviation from the global monotonic interpretation when
reliable / guaranteed delivery is in use. The Event Numbering table on the
core page already encodes that two-meaning rule; what conflicts is the
"Notification Types" example prose, which incorrectly downgrades the element
to optional. So the underlying fix is most naturally framed as aligning
`subscriptionstatus-notes.xml` (and the example rows) with the resolved
position rather than changing cardinality.

## Details

**Description:**

> The event numbering section mentions eventNumber as required for both best
> effort and guaranteed delivery channels:
>
> | Element | Channel | Conformance | Comment |
> |---------|---------|-------------|---------|
> | `SubscriptionStatus.notificationEvent.eventNumber` | Best Effort | Required | Event numbers SHALL match the index in the subscription event sequence. E.g., the highest-numbered event in the notification will match the number in `eventsSinceSubscriptionStart`. |
> | `SubscriptionStatus.notificationEvent.eventNumber` | Guaranteed | Required | A relative index of events in a single notification, e.g., 1, 2, 3, etc.. |
>
> However, the examples in the Notification Types section all have the same
> blurb for event number, with it being optional for guaranteed delivery:
>
> | Element | Channel | Conformance | Comment |
> |---------|---------|-------------|---------|
> | `SubscriptionStatus.notificationEvent.eventNumber` | Special | Required for subscriptions with best-effort delivery, optional for subscriptions with guaranteed delivery. See [Event Numbering](https://build.fhir.org/subscriptionstatus.html#event-number) for details. |
>
> Should the event numbering section be updated?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Event Numbering section of `source/subscriptionstatus/subscriptionstatus-notes.xml`
in `HL7/fhir` so that the conformance table marks
`SubscriptionStatus.notificationEvent.eventNumber` as **Optional** under
*Guaranteed* delivery (matching the existing per-notification-type example
prose: "Required for subscriptions with best-effort delivery, optional for
subscriptions with guaranteed delivery"). Leave the *Best Effort* row as
**Required**, and re-word the comment column on the *Guaranteed* row to
explain that, when supplied, it is a relative index of events within the
notification (e.g., 1, 2, 3). No StructureDefinition cardinality change
is implied by this option, since `eventNumber` is page-level conformance
text rather than ElementDefinition cardinality.

#### Justification

This is the literal reading of the reporter's question ("Should the event
numbering section be updated?") and is the simplest way to make the page
internally consistent. It also reduces the implementation burden on
guaranteed-delivery channels by removing any obligation to populate
`eventNumber` at all.

### Disposition B: Alternative Approach

#### Proposal

Keep `SubscriptionStatus.notificationEvent.eventNumber` **Required** in both
delivery modes (the existing Event Numbering table) and instead **fix the
"Notification Types" example prose** to align with the resolved policy from
[FHIR-48808](https://jira.hl7.org/browse/FHIR-48808). Concretely, in
`source/subscriptionstatus/subscriptionstatus-notes.xml`:

1. Replace every occurrence of *"Required for subscriptions with best-effort
   delivery, optional for subscriptions with guaranteed delivery"* in the
   per-notification-type example tables with wording such as: *"Required.
   For best-effort channels this SHALL be the global, monotonically
   increasing event index (matching `eventsSinceSubscriptionStart`). For
   guaranteed-delivery channels this MAY instead be a relative index of
   events within this notification (e.g., 1, 2, 3) when the channel type
   describes and supports that semantic; channels SHOULD only deviate from
   the global monotonic interpretation when reliable / guaranteed delivery
   is in use."*
2. Add a single sentence to the Event Numbering section pointing readers at
   the channel-type specification for which interpretation applies.
3. File a follow-on ticket against `HL7/fhir-subscription-backport-ig`
   (or confirm FHIR-48808's edits) so the Backport IG description text and
   the core text stay in lock-step.

This option does **not** require touching the StructureDefinition
differential — it is confined to the authored `subscriptionstatus-notes.xml`
narrative. Edits land in `HL7/fhir` under the `source/subscriptionstatus/`
authoring root; the bespoke publisher will pick them up directly with no
sushi/IG-publisher involvement.

#### Justification

This matches the position the resource owner articulated on Zulip and that
was accepted (Persuasive) in FHIR-48808: cardinality stays 1..1, but the
*meaning* of `eventNumber` is allowed to vary by channel. Making the
element optional in core (Disposition A) would diverge from the Zulip-and-
ticket-resolved policy, would silently break any consumer that relies on
`eventNumber` always being present in a guaranteed-delivery notification
(it currently is — and one reason it stays required is so that multiple
events in one notification remain unambiguously ordered/identifiable), and
would create a needless gap between Core and the Backport IG, which is the
problem FHIR-48808 set out to close. The two pieces of guidance the
reporter found in conflict can be reconciled by fixing the prose, not the
conformance.

### Disposition C: Decline

#### Proposal

Decline as already-addressed. Mark the ticket as duplicate / persuasive of
[FHIR-48808](https://jira.hl7.org/browse/FHIR-48808), whose resolution
already states the canonical policy (`eventNumber` remains required;
local-only usage allowed under guaranteed delivery; channels SHOULD only
deviate from global monotonic semantics under reliable delivery), and close
without making any change to `source/subscriptionstatus/`. The Workgroup
would rely on the existing Event Numbering section being read as
authoritative and the example-table prose as informational shorthand.

#### Justification

The substantive policy question ("is `eventNumber` actually optional under
guaranteed delivery?") was already resolved in FHIR-48808. From a
conformance perspective the table in the Event Numbering section is the
normative statement, and the per-notification-type example tables are
illustrative. Closing without a spec edit avoids any risk of regressing
the FHIR-48808 outcome.

### Recommendation

**Recommended disposition:** B

The reporter has correctly identified a real internal inconsistency on
`subscriptionstatus.html`, but the resolution that the workgroup already
adopted in FHIR-48808 was explicit: keep `eventNumber` required and clarify
the *meaning*. Disposition A would diverge from that resolution and break
parity with the Backport IG; Disposition C leaves the contradictory text
in place, which the next reader will hit again. Disposition B is a
narrow, prose-only edit to `source/subscriptionstatus/subscriptionstatus-notes.xml`
in `HL7/fhir` that brings the example tables into agreement with the Event
Numbering section and the FHIR-48808 resolution, and it is a clean
"Update narrative / prose" change under the FhirCore playbook with no
StructureDefinition or `<snapshot>` implications. The Workgroup should
pair the edit with a quick check that the Backport IG language matches.
