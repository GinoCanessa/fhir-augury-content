## Summary

The reporter notes that the SDC `$populate` documentation describes a
"re-populate" use case (refreshing an in-progress or previously
completed `QuestionnaireResponse` with current data), but the
`OperationDefinition` for `Questionnaire/$populate` has no input
parameter that lets a caller pass in the existing `QuestionnaireResponse`.
The request is to extend `$populate` with a new IN parameter (proposed
name: `response`) carrying the existing `QuestionnaireResponse`, so that
servers and clients have a defined mechanism for re-population.

The ticket sat with a 2024 SDC vote of "Considered for Future Use" (4-0-0,
Postlethwaite/Beda) on the grounds that there was insufficient
implementation experience to ballot before STU4. lynn_laakso re-opened
it on 2026-03-26 ("Re-open post STU4 publication") so that re-populate
can be considered for the next SDC ballot cycle. There are no
explicitly-linked Jira tickets and no GitHub cross-references; all
cross-referenced context lives in two Zulip threads on the
`#questionnaire` stream and a newer Confluence page
(`SDC RePopulate`, FHIRI space, page id 413044258) that Brian
Postlethwaite stood up in late 2025 to collate the design discussion.

### [#questionnaire > ✔ Need a questionnaire-prior-answer extension.](https://chat.fhir.org/#narrow/stream/questionnaire/topic/%E2%9C%94%20Need%20a%20questionnaire-prior-answer%20extension./near/465553843)

Lloyd McKenzie cross-references FHIR-46271 from a separate thread about
tracking prior answers. He notes a key constraint relevant to any
`$populate` re-population design: an external `$populate` server does
not necessarily share the calling app's permissions to re-query the EHR
for current source data — tokens are typically scoped to the app, not
shareable — which limits how much of the "re-populate" workflow can be
delegated to the operation server. He suggests that if someone wants
explicit "prior answer" tracking on individual items, that's a separate
CRD-style change request; the current plan is only to track where the
answer came from and whether it was overridden.

Participants: Lloyd McKenzie (cross-reference into the FHIR-46271
discussion).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new IN parameter `response` (cardinality `0..1`, type
`Resource` constrained to `QuestionnaireResponse`) to the
`Questionnaire/$populate` `OperationDefinition` in the SDC IG. The
parameter carries the existing `QuestionnaireResponse` whose answers
should be retained / merged when the server produces the populated
response. Update `populate.html` "Population Operations" so the
re-populate narrative explicitly references the new parameter, and add
non-normative guidance covering: (1) item-level merge semantics
(server preserves prior answers unless authoritative source data
overrides them), (2) handling of items whose `enableWhen` evaluation
changes after re-population, (3) what to do with answers whose source
items have been removed from the `Questionnaire` or whose source data
is no longer retrievable, and (4) author-controlled opt-out (e.g., an
extension on `Questionnaire.item` to mark answers as "user-entered, do
not overwrite").

#### Justification

This matches the literal request and removes a documented contradiction
between `populate.html` and the `OperationDefinition`. The parameter
itself is additive (`0..1`, optional), so existing clients and servers
that don't implement re-populate are unaffected, and there is no wire
break for current STU4 conformance. The Zulip thread already enumerates
the design questions Brian Postlethwaite raised in 2024-08, and Brenin
Rhodes' Confluence proposals plus Brian's fhirpath-lab/HAPI sample give
the SDC group concrete material to write the guidance against. Lloyd's
constraint from the prior-answer thread (external `$populate` servers
may not be able to re-query the EHR) is accommodated by making the
parameter optional and the merge guidance explicit about which
responsibilities fall on the client vs. server.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** overload the existing `Questionnaire/$populate` operation.
Instead, define a sibling `Questionnaire/$repopulate` operation in the
SDC IG with its own `OperationDefinition` whose IN parameters
explicitly include `response : QuestionnaireResponse (1..1)` plus the
re-populate-specific controls Brian Postlethwaite called out in the
Zulip thread (e.g., `mode : code` for full/merge/refresh-changed-only,
`preserve : string` for FHIRPath expressions identifying user-entered
items to leave untouched, and `since : instant` so the server can
limit the work to source data changed after a given timestamp). Keep
`$populate` strictly for first-time population. Also add an
`Questionnaire.item.extension` (or reuse the in-flight prior-answer
extension referenced in the second Zulip thread) so re-populate can
record provenance per-answer.

#### Justification

The 2024-11 SDC vote and the recurring objections in the thread
("we don't have enough implementation experience", "more guidance is
likely necessary", Paul Lynch's "is this even a good idea") show that
the WG sees re-populate as a meaningfully different operation, not a
small addition to `$populate`. Splitting the operations keeps
`$populate`'s contract stable (a real consideration given STU4 just
published), avoids overloading a single OperationDefinition with mode
flags, and lets the SDC group ballot `$repopulate` as draft / Trial Use
on its own publication cadence as implementation experience grows.
This matches Lynn Laakso's reopen note, which explicitly contemplates
"draft guidance" alongside experience.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Mod / Not Persuaded — keep the
existing approach.** Resolution: do not change the
`Questionnaire/$populate` `OperationDefinition` or define a new
operation. Instead, clarify `populate.html` to make explicit that
"re-populate" means *generating a new `QuestionnaireResponse` that the
client then merges client-side with the prior `QuestionnaireResponse`*,
and remove or soften any wording that implies the operation itself
takes a prior `QuestionnaireResponse` as input.

#### Justification

This is the disposition that mirrors the original 2024-11 vote
("Considered for Future Use", 4-0-0). The objections raised on Zulip
are substantive and unresolved: Paul Lynch questioned whether
re-population is even desirable; Lloyd noted the security model
problem (external `$populate` servers can't necessarily re-query the
source EHR); the FHIR Infrastructure / SDC group already published
STU4 without it; and the merge semantics, `enableWhen` re-evaluation,
deleted-source-data behaviour, and user-overwrite protection are all
unresolved design problems where a wrong default can cause patient-safety
issues (silently overwriting a clinician's correction). Documentation
fix-up is the lowest-cost way to remove the contradiction the reporter
identified without committing the WG to a contract change.

---

### Recommendation

**Recommended disposition:** B

The Zulip thread, the Confluence page Brian Postlethwaite stood up
(`SDC RePopulate`, FHIRI/413044258), and the fhirpath-lab + HAPI
samples make it clear that the SDC group has decided re-populate is in
scope for the next cycle — Lynn Laakso's reopen comment confirms that
explicitly. Disposition C therefore feels like a regression from where
the discussion already is. Disposition A gets the contradiction out of
the IG quickly, but it bakes the design choices ("merge vs. refresh",
"preserve vs. overwrite", "client- vs. server-driven re-query") into
the same `OperationDefinition` everyone is already implementing for
first-time populate, which raises the conformance cost for existing
implementers and concentrates risk in one operation. Disposition B
keeps `$populate` stable for the implementers who just shipped against
STU4, gives the SDC sub-group room to ballot `$repopulate` as draft /
Trial Use with the additional controls Brian enumerated, and lines up
with the workgroup's stated preference for "draft guidance + experience
before commitment." It also makes Lloyd's external-server token
constraint a first-class concern of the new operation rather than a
footnote on `$populate`. The reporter (VitalyBanchenko) gets a defined
mechanism that satisfies his original request; SDC gets to evolve the
re-populate semantics without breaking `$populate`'s STU4 contract.
