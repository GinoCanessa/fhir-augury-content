## Summary

The reporter responds to the explicit balloter‑feedback request added by
[FHIR-51587](https://jira.hl7.org/browse/FHIR-51587) ("We are seeking input
on approaches to managing the balance between expressiveness and performance
impact on servers from the use of various FHIRPath features in search
parameter expressions"). They propose that the FHIR core specification
**normatively define a limited subset of FHIRPath that authors SHOULD use
in `SearchParameter.expression`**, with two stated benefits:

1. Server implementers (like the reporter, who built a partial FHIRPath
   evaluator scoped to search parameters) would not have to reverse‑engineer
   the actual subset in use by manually parsing core SearchParameters.
2. Future IGs would be steered toward authoring custom search parameters
   that are reasonably computable on existing servers.

A third benefit — consistency — is illustrated with `Observation.value-date`,
whose expression has three functionally equivalent but syntactically
different forms across STU3, R4, and R6. The reporter further notes that
even within R6 the spec is not fully self‑consistent (e.g.,
`ActivityDefinition.subject-canonical` vs. `Task.focus-canonical`) and that
some uses of `as {Resource}` would be more consistent expressed via
`ofType()`.

The ticket is tightly coupled to two recently‑resolved FHIR‑I tickets:
[FHIR-51587](https://jira.hl7.org/browse/FHIR-51587) (Persuasive with
Modification, applied — added text saying the spec "restricts expressions
to the simple subset", plus the balloter‑feedback note that prompted this
ticket) and [FHIR-49784](https://jira.hl7.org/browse/FHIR-49784) (Persuasive
with Modification, applied — added `repeat()` to several core search
parameters). Per the related Zulip thread, those two resolutions are
**internally inconsistent**: `repeat()` is not part of the FHIRPath
"simple" subset, so the very subset FHIR-51587 names cannot be the one in
use. FHIR-53907 essentially asks FHIR‑I to nail down which subset is
actually intended and to publish it as SHOULD‑level guidance.

## Details

**Description:**

> This is in response to this request:
>
> > Balloter feedback requested: We are seeking input on approaches to
> > managing the balance between expressiveness and performance impact on
> > servers from the use of various FHIRPath features in search parameter
> > expressions.
>
> In my opinion, it would be very helpful for servers if FHIR defined a
> limited subset of FHIRPath that one SHOULD use for
> `SearchParameter.expression`.
>
> For my FHIR server implementation, we've implemented limited filtering
> capabilities based on SearchParameter FHIRPaths. For performance reasons,
> we do not implement a full FHIRPath engine, but rather only the subset
> needed for search parameters. This ends up requiring a *very* limited
> subset of FHIRPath, but we could only derive this subset by manually
> parsing/evaluating the SearchParameter definitions.
>
> Indicating the subset of FHIRPath in use by the base spec would have
> made this easier.
>
> More importantly, providing SHOULD guidance on what subset of FHIRPath
> to use would encourage future IGs to design reasonably computable custom
> search parameters.
>
> Additionally, there's a few variations to communicate the same
> functional concept in FHIRPath, so SHOULD guidance would also help in
> standardizing that. For example, compare Observation's `value-date`
> expression in [STU3](https://www.hl7.org/fhir/STU3/observation.html) vs
> [R4](https://www.hl7.org/fhir/R4/observation.html) vs
> [R6](http://build.fhir.org/observation.html). This parameter has 3
> different variations in syntax that are all functionally identical.
>
> (in fact, even R6 is not fully self‑consistent. Compare
> `ActivityDefinition.subject-canonical` with `Task.focus-canonical`)
>
> For reference, beyond direct paths, these are other FHIRPath functions
> in use by FHIR core as far as I know, although this may be incomplete:
>
> | FHIRPath function | Example search param |
> |---|---|
> | `.where(resolve() is {Resource})` | Procedure.patient |
> | `.where({field}='{constant}')` | Patient.email |
> | `.ofType({Type})` | Patient.death-date |
> | comparators (`!=`) | Patient.deceased |
> | `.exists()` | Patient.deceased |
> | Or logic (`\|`) | AllergyIntolerance.code |
> | `.extension('{url}')` | CareTeam.name |
> | `.repeat({field})` | CodeSystem.code |
> | `.combine({field})` | QuestionnaireResponse.linkId |
> | `as {Resource}` | Resource type params (Bundle.composition) |
> | `.where(extension('{url}').exists())` | QuestionnaireResponse.item-subject |
>
> (there are also some uses of "as" that should probably instead use
> "ofType()" for consistency)

## Related Jira Tickets

### [FHIR-51587: Core spec FHIRpath should be validated](https://jira.hl7.org/browse/FHIR-51587)

The direct precursor. FHIR-I voted "Persuasive with Modification" and
agreed to add a new "Use of FHIRPath in Expressions" section in Search
Parameters and an additional comment on `SearchParameter.expression`
stating that "While `SearchParameter.expression` allows for the full use
of FHIRPath, this specification restricts expressions to the 'simple'
subset" plus guidance restricting `resolve()` to type narrowing. The
disposition also added the **balloter‑feedback note that FHIR-53907
quotes verbatim**. The motion was re‑opened on 2025‑11‑03 (Grieve /
Postlethwaite, 15‑0‑0) precisely because the "simple subset" wording
conflicts with how core search parameters are actually written, which is
exactly the gap FHIR-53907 is trying to close.

Reporter: Elliot Silver. Commenters: Brian Postlethwaite, Grahame Grieve
(via re‑open motion), mckenzie (clerk), Gino Canessa.

### [FHIR-49784: Definition of SearchParams on nested elements](https://jira.hl7.org/browse/FHIR-49784)

The companion ticket whose resolution introduced `repeat()` into core
SearchParameter expressions (so that a parameter on a nested element such
as `CodeSystem.concept.code` reaches deeper concept levels). Because
`repeat()` is **not** part of the FHIRPath simple subset that FHIR-51587
named, the two dispositions contradict each other — Grahame Grieve
flagged this on Zulip. Any disposition of FHIR-53907 has to reconcile the
two by either expanding the named subset (to include `repeat()`,
`where()`, `ofType()`, `extension()`, etc.) or by replacing "simple
subset" with a freshly‑defined, search‑specific subset.

Reporter: Jonas Schön.

### [FHIR-37591: Patient.deceased does not work with `:missing`](https://jira.hl7.org/browse/FHIR-37591), [FHIR-46005: Change Patient.deceased description](https://jira.hl7.org/browse/FHIR-46005), [FHIR-41283: SearchParameter 'reason' on ImagingStudy does not contain a FHIRPath expression](https://jira.hl7.org/browse/FHIR-41283), [FHIR-54161: Search Parameter date Has Empty Expression](https://jira.hl7.org/browse/FHIR-54161)

Surfaced by keyword similarity. They are not directly about subsetting,
but each is a concrete example of the underlying problem this ticket
raises: SearchParameter expressions in core that are inconsistent,
incomplete, or rely on FHIRPath features whose expected server semantics
are unclear. They reinforce the case for explicit normative guidance on
what authors may rely upon.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new normative subsection to the **Search Parameters** page of FHIR
Core (sibling to the "Use of FHIRPath in Expressions" section added by
FHIR‑51587) titled "FHIRPath subset for SearchParameter.expression". In
that subsection, define the **closed list of FHIRPath constructs that
authors SHOULD use** when writing a `SearchParameter.expression`,
covering at minimum the constructs already in use by core (drawn from the
table the reporter supplied):

- Direct path navigation
- `.where(resolve() is {Resource})` — restricted to type narrowing
- `.where({field} = '{constant}')` (and `!=`)
- `.ofType({Type})`
- `.exists()`
- Or logic (`A | B`)
- `.extension('{url}')`
- `.repeat({field})`
- `.combine({field})`

Update `SearchParameter.expression`'s element comment to point to this
subsection, replace the existing reference to the FHIRPath
`fhirpath.html#simple` "simple subset" with a link to the new
search‑specific subset, and add a SHOULD requirement that custom
SearchParameters in IGs use only constructs from this list (with a
defined extension hook for declaring out‑of‑subset usage). At the same
time, sweep core SearchParameters and normalize the inconsistent
formulations the reporter flagged (e.g., rewrite stray `as {Resource}`
to `ofType({Resource})`; align
`ActivityDefinition.subject-canonical` with `Task.focus-canonical`; pick
one canonical form for `Observation.value-date`).

#### Justification

This is exactly what the balloter‑feedback note added by FHIR‑51587
solicited, and it directly addresses the contradiction Grahame Grieve and
Brian Postlethwaite identified on Zulip between FHIR‑51587 and
FHIR‑49784: the "simple subset" link is wrong because `repeat()` is not
in it, and the spec then permits `resolve()` for type narrowing anyway.
A purpose‑built, enumerated subset (a) gives server implementers like the
reporter a stable target for partial FHIRPath engines, (b) gives IG
authors a clear yardstick for "computable" search parameters, and (c)
lets the FHIRPath validation tooling promised by FHIR‑51587 actually
flag out‑of‑subset usages instead of mis‑pointing at the
`fhirpath.html#simple` definition. Sweeping core SearchParameters into a
single canonical syntax also removes the cross‑version drift the
reporter documented across STU3 / R4 / R6 of `value-date`.

---

### Disposition B: Alternative Approach

#### Proposal

Do not embed the subset directly in the core spec text. Instead:

1. Add an **`http://hl7.org/fhir/StructureDefinition/searchparameter-fhirpath-features`** code‑valued extension on `SearchParameter` (or reuse an
   existing capability mechanism) whose values come from a new
   `CodeSystem`/`ValueSet` enumerating the FHIRPath features used by the
   expression (e.g., `direct-path`, `resolve-type-narrow`, `where-equals`,
   `repeat`, `extension`, `combine`, `or`, `ofType`, `exists`,
   `comparator`).
2. Have the publisher tooling (kindling/IG publisher) populate this
   extension automatically by parsing each `SearchParameter.expression`,
   and fail the build for any feature outside an allow‑list maintained
   per‑spec.
3. Update `CapabilityStatement.rest.searchParam` (or its referenced
   `SearchParameter`) so that servers can advertise *which* feature codes
   they support, letting clients negotiate parameter use rather than
   relying on a single global SHOULD.
4. Replace the "simple subset" wording in
   `SearchParameter.expression` and the new "Use of FHIRPath in
   Expressions" section with a link to that ValueSet plus narrative on
   the recommended baseline (effectively the same set as Disposition A,
   but expressed as data rather than prose).

#### Justification

This shifts the answer from a single static subset (which will inevitably
be re‑opened every time a real server wants `repeat()` or every time an
IG legitimately needs an unusual feature) to a machine‑readable,
extensible declaration. It directly solves the reporter's primary pain
point — "we could only derive this subset by manually parsing the
SearchParameter definitions" — by making the feature set discoverable
from the artifact itself, which is far cheaper for partial FHIRPath
engines to consume than freeform spec prose. It also handles the Zulip
disagreement gracefully: `repeat()` and type‑narrowing `resolve()` can be
flagged feature codes rather than special‑case carve‑outs in the
narrative. The downside is more machinery (a new extension, a new
ValueSet, publisher work) and a longer runway than Disposition A.

---

### Disposition C: Decline

#### Proposal

Mark the request "Not Persuasive". Take the position that the existing
text added by FHIR‑51587 (linking to `fhirpath.html#simple` plus the
`resolve()` carve‑out) is sufficient guidance, that the inconsistencies
across STU3/R4/R6 noted by the reporter are historical artefacts that
will be cleaned up incrementally via point tickets (e.g., FHIR‑54161,
FHIR‑41283), and that the balloter‑feedback note was a request for
discussion rather than a commitment to publish a new normative subset.
Encourage the reporter to file individual change requests against
specific SearchParameters whose expressions are inconsistent and to
contribute to FHIRPath SDC/clinical‑reasoning work where a richer
profile of FHIRPath is being defined.

#### Justification

A normative SHOULD‑subset locks FHIR‑I into ongoing curation and
tooling work for what is essentially an authoring quality issue. The
"simple subset" already exists in the FHIRPath spec and the
`resolve()` carve‑out was just added; layering a third subset risks
spec churn and new contradictions. However, this disposition does **not
resolve** the FHIR‑51587/FHIR‑49784 contradiction the Zulip thread
identified, and it leaves implementers with no normative way to know
which features they must support — so it is the weakest option on the
substantive merits.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the smallest change that actually answers the
balloter‑feedback question FHIR‑51587 explicitly opened, and it directly
removes the FHIR‑51587 ↔ FHIR‑49784 inconsistency that Grahame Grieve and
Brian Postlethwaite agreed on Zulip is a real defect. The reporter has
already done most of the legwork — the table of FHIRPath constructs
currently in use by core SearchParameters is essentially the first draft
of the subset to publish. Disposition B is the right end‑state for IG
tooling and CapabilityStatement negotiation, but it is materially larger
in scope (new extension, new ValueSet, publisher rework) and should not
block the core‑spec correction; FHIR‑I can take A now and use it as the
declarative seed for B in a later cycle. Disposition C leaves a known
contradiction in the spec at ballot time and provides no help to the
implementers who prompted the balloter‑feedback note in the first
place, so it should not be selected.
