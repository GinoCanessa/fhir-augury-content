## Summary

The ticket requests that the FHIR HTTP page be extended with a new
section that describes the behavior of **reverse proxies** in front of
FHIR servers. Reverse proxies are now commonplace in production
deployments, and the request is that the specification provide basic
guidance covering (a) the forwarding headers (notably
[`Forwarded`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Forwarded)
and the conventional `X-Forwarded-*` set) and (b) the general principle
that a reverse proxy should be "invisible" to the FHIR client — that is,
the URLs, hostnames, and scheme surfaced by the server (in `Bundle.link`,
`fullUrl`, `Location` headers, OperationOutcome, search self/next links,
etc.) should reflect the public-facing endpoint rather than the internal
one behind the proxy.

There are no linked or cross-referenced Jira tickets, so the scope is
self-contained. The only related discussion lives in the
`implementers > Multi-Tenant FHIR Server` Zulip topic, where the
reporter acknowledged the open ticket and a participant indicated such
guidance would be useful. Two prior WG comments are material: one
(2022) asked the reporter to draft specific proposed language before
approval, and a later one (2023) observed that
[`http.html#custom`](https://build.fhir.org/http.html#custom) already
covers most of what is needed and that anything beyond that should wait
for R6. The most recent comment (2025) from the reporter proposes
relocating the work to the **API Incubation IG** so the guidance can be
iterated on and tested before being folded into FHIR Core, which is
consistent with the ticket's `MovingToGuide` label.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level section to the FHIR Core HTTP page (`http.html`),
sited adjacent to the existing `#custom` section, titled "Reverse
Proxies." The section would:

1. Define reverse proxy in the FHIR context (TLS termination, multi-
   tenant routing, rate-limiting/WAF, aggregation in front of one or
   more FHIR servers).
2. State the "invisibility" principle: clients SHOULD see only the
   public-facing scheme, host, port, and base path; the server behind
   the proxy SHALL NOT leak internal hostnames/ports into
   `Bundle.link`, `Bundle.entry.fullUrl`, `OperationOutcome`
   diagnostics, `Location` response headers, search `self`/`next`/`prev`
   links, or conformance `CapabilityStatement.implementation.url`.
3. Describe the `Forwarded` HTTP header (RFC 7239) and the de-facto
   `X-Forwarded-For` / `X-Forwarded-Host` / `X-Forwarded-Proto` set,
   recommending that FHIR servers deployed behind a reverse proxy be
   configured to honor these headers when computing the server base
   URL, and that proxies be configured to set them.
4. Note interactions with existing sections (`#custom`, `#summary`,
   `#paging`) where server-generated URLs appear.

#### Justification

This is what the ticket literally requests and what the Zulip
participant said would be useful. Reverse proxies are ubiquitous in
real-world FHIR deployments and the current HTTP page offers no
direct guidance, which forces every implementer to rediscover the
"public URL" problem. The WG's 2023 observation that `#custom`
"covers most of what's needed" is correct for custom headers but does
not address the specific `Forwarded` / `X-Forwarded-*` family or the
URL-leak hazard; a short dedicated section closes that gap without
disturbing the rest of the HTTP page.

---

### Disposition B: Alternative Approach

#### Proposal

Develop the reverse-proxy guidance in the **API Incubation IG** rather
than FHIR Core, per the reporter's 2025 comment and the ticket's
`MovingToGuide` label. Concretely: author a page in the API
Incubation IG that covers the same material listed under Disposition A
(invisibility principle, `Forwarded` / `X-Forwarded-*` handling,
server-base-URL computation, URL-leak checklist for `Bundle.link`,
`fullUrl`, `Location`, paging links, `CapabilityStatement`). Keep the
guidance non-normative while it incubates; once implementer feedback
stabilizes it, promote the settled material into the FHIR Core HTTP
page in a future release (R6 or later). Leave the FHIR Core HTTP page
unchanged for now except for an informational link to the incubator
page.

#### Justification

The 2023 WG comment explicitly deferred further HTTP-page changes to
R6, and the 2025 reporter comment plus the `MovingToGuide` label
signal that the WG's current intent is to iterate on this in a guide
rather than in Core. Incubating first lets the guidance pick up
implementer review (e.g., from the multi-tenant discussion on Zulip)
and concrete examples before taking a normative slot in Core, which
reduces the risk of landing under-specified text. It also avoids a
late R5-era Core change on a page that has historically been sensitive
to churn.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — already addressed." Rationale:
the HTTP page's existing
[`#custom`](https://build.fhir.org/http.html#custom) section already
covers custom-header behavior generically, server base-URL semantics
are implicit in the existing rules for server-generated links in
`Bundle.link`, `fullUrl`, `Location`, and paging, and the specific
headers the ticket calls out (`Forwarded`, `X-Forwarded-*`) are IETF
constructs that are adequately defined outside FHIR. A dedicated
FHIR-Core section is not required to deploy a FHIR server behind a
reverse proxy correctly.

#### Justification

This mirrors the WG's 2023 position. Adding normative-looking text on
a purely deployment-topology concern risks either duplicating RFC 7239
or accidentally constraining legitimate proxy topologies. The absence
of any Jira cross-references, any linked tickets, and any substantive
Zulip design discussion over three years suggests the implementer pain
point has not escalated to a level that warrants a Core change.

---

### Recommendation

**Recommended disposition:** B

Disposition B aligns with both the ticket's `MovingToGuide` label and
the reporter's most recent (2025) comment proposing relocation to the
API Incubation IG, and it respects the 2023 WG decision to defer
further HTTP-page changes to R6. Disposition A is substantively
correct but would land normative text directly in Core without the
"specific proposal for language" the WG asked for in 2022 and without
the benefit of implementer iteration — a Core HTTP-page change is
expensive to get wrong. Disposition C understates the real
implementer gap: `#custom` is about custom HTTP headers, not about the
invisibility principle or the `Forwarded` family, and the Zulip
multi-tenant thread shows the audience exists. Routing this through
the API Incubation IG preserves the intent of the ticket, produces
something concrete the workgroup can review, and leaves a clean path
to fold the settled guidance into FHIR Core in a later release.
