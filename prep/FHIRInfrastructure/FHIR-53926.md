## Summary

The reporter asks the FHIR Infrastructure work group to add HTTP `303 See
Other` to the list of "Common HTTP Status codes returned" documented for the
**Create** interaction in the FHIR Core RESTful API specification. Today, the
spec already endorses returning `303 See Other` from the Create interaction as
part of the duplicate-detection guidance ("Servers that determine that a POST
would result in a duplicate MAY return a `303 See Other` pointing to the
existing (unchanged) record …"), but `303` is not enumerated in the Create
interaction's bulleted list of common HTTP responses, while it does appear in
the analogous list for the Update interaction. The request is therefore an
internal-consistency edit: the duplicate-detection prose already prescribes
`303` for Create, so the surrounding status-code list should mention it too.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket; the request stands on the text of the spec alone.

## Details

**Description:**

> For the create interaction, common HTTP status codes are listed under the
> phrase "Common HTTP Status codes returned (in addition to normal HTTP errors
> related to security, header and content type negotiation issues):"
>
> HTTP 303 (See Other) could also be listed here, as the guidance for the
> update interaction that
>
> *"Servers that determine that a POST would result in a duplicate MAY return
> a `303 See Other` pointing to the existing (unchanged) record and
> indicating that the record was not created because the specified resource
> already exists at the specified location"*
>
> may apply to both the create and the update interaction.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir` (authoring root `source/`), edit `source/http.html` to add a
new bullet for `303 See Other` to the Create interaction's "Common HTTP
Status codes returned …" list, mirroring the entry already present in the
Update interaction's list. The bullet should make explicit the same semantic
the spec already gives in prose: that a server MAY respond with `303 See
Other` and a `Location` header pointing to the existing resource when it
determines that the POST would create a duplicate of a record that already
exists at the indicated location (i.e., the duplicate-detection /
deduplication case). No StructureDefinitions, value sets, or examples need
to change; the edit is confined to the HTML narrative on the Create
interaction section of `http.html`.

#### Justification

The spec is internally inconsistent today: the Create interaction's prose
already prescribes `303 See Other` for the duplicate-detection case (the
exact passage the ticket quotes), and the Update interaction enumerates `303
See Other` in its own status-code list, but the Create interaction's
status-code list omits it. Listing `303 See Other` for Create is therefore
strictly clarifying — it documents behaviour the normative narrative already
permits, which helps server implementers (who scan the bullet list) and
client implementers (who need to know which 3xx codes to handle on POST).
The change is isolated to a single authored HTML page, uses the bespoke
publisher's normal HTML edit recipe from the briefing, and carries no
cross-repo or schema impact.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding the bullet to the Create list, perform a small
consistency sweep on `source/http.html`: (1) add `303 See Other` to the
Create interaction's status-code list as in Disposition A; (2) cross-link
the new bullet to the existing duplicate-detection paragraph (anchor on the
"Servers that determine that a POST would result in a duplicate …"
sentence), and to the conditional-create section, so that readers can
navigate from the status code to the behaviour that produces it; and (3)
review the Update, Conditional Update, and Conditional Create sub-sections
to confirm `303 See Other` appears in each status-code list where the
narrative already references it, fixing any other gaps in the same edit.
All changes remain in `HL7/fhir:source/http.html`; no other authoring files
are touched.

#### Justification

The same root cause (status-code lists drifting out of sync with the
narrative they accompany) almost certainly affects more than just the Create
interaction, because Create, Update, Conditional Create, and Conditional
Update all share the duplicate-detection / "resource already exists"
branching. Bundling a quick consistency sweep avoids a second ticket later,
and the cross-links make the relationship between the status code and the
duplicate-detection prose discoverable. The cost is small — still a single
HTML page in `source/`, no schema or generated artefacts — and matches the
briefing's "Update narrative / prose" recipe. Trade-off vs. A: marginally
larger review surface, but the FHIR Infrastructure WG can review all the
status-code lists in one pass instead of revisiting them later.

---

### Disposition C: Decline

#### Proposal

Decline the request as redundant. The Create interaction already documents,
in prose, that a server MAY return `303 See Other` for the duplicate-detection
case, and the status-code list is explicitly described as "Common" rather
than exhaustive. No spec change is made; the ticket is closed as "Persuasive
with mod" → "no change required" or "Not Persuasive".

#### Justification

The list is prefaced as "Common HTTP Status codes returned (in addition to
normal HTTP errors related to security, header and content type negotiation
issues)", which signals it is illustrative rather than normative. `303 See
Other` is already permitted by the surrounding narrative, so a conformant
implementer reading the page in full has the information they need. Adding
every status code referenced in nearby prose risks turning the bullet list
into a duplicate of the narrative.

---

### Recommendation

**Recommended disposition:** A

This is a small, low-risk documentation edit that removes a real
inconsistency between two parts of the same page: the Create interaction's
narrative already authorises `303 See Other`, the Update interaction's list
already enumerates it, and only the Create interaction's list omits it.
Disposition A fixes exactly that, with a single bullet added to a single
HTML page in `HL7/fhir:source/http.html` — the briefing's standard
"Update narrative / prose" recipe, with no cross-repo impact and no risk
to generated artefacts. Disposition B is a reasonable expansion if the WG
has appetite for a quick sweep of the other interaction sections, but it
should not block A; it can be folded in during review or filed as a
follow-up if the sweep surfaces nothing. Disposition C under-serves
implementers: scanning the status-code list is the natural way readers find
out which codes a server might return, and leaving `303` out when the
adjacent prose explicitly endorses it is the kind of inconsistency that
generates support questions and conformance ambiguity.
