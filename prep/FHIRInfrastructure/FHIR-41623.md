## Summary

The ticket reports that `SearchParameter.name` carries invariant **cnl-0**,
which constrains the value to the regex
`^[A-Z]([A-Za-z0-9_]){1,254}$` — i.e., the name must begin with an
uppercase letter. The reporter (Christiaan Knaap, Firely) observes that
many — arguably none — of the SearchParameters in the core specification
actually conform to this rule, and proposes relaxing the regex to
`^[A-Za-z]([A-Za-z0-9_]){1,254}$` so that lowercase initial characters
are permitted.

A subsequent FHIR-I discussion (recorded by Josh Mandel as a comment)
broadened the question: should the constraint on Canonical names be
relaxed more generally (allowing `_` and lowercase initial characters,
and is that relaxation breaking?); should the constraint instead be on
`SearchParameter.code` rather than `name`; and do SearchParameters
really need both a `name` and a `code`?

The currently recorded resolution is **Not Persuasive with
Modification**: the work group's position is that `name` is intended to
be a programming-language-safe class-style identifier (UpperCamelCase),
that the existing non-conformant SearchParameter `name`s in core are
*errors* in the spec, and that the appropriate fix is to correct those
`name`s rather than weaken `cnl-0`. Firely (per `wardweistra`'s comment)
has confirmed they have no problem with that correction.

The cross-reference graph attaches the ticket to two FHIR elements
(`SearchParameter.name` and `SearchParameter.code`); no related Jira
tickets, Zulip threads, or GitHub items were surfaced.

## Details

**Description:**

> SearchParameter.name has constraint cnl-0, with the regex
> `name.exists() implies name.matches('^[A-Z]([A-Za-z0-9_]){1,254}$')`
>
> That means that the name should start with a capital character. But
> afaik none of the SearchParameters in the core specification adhere to
> that rule.
>
> I think this regex should be `^[A-Za-z]([A-Za-z0-9_]){1,254}$`

**Resolution Description (currently recorded):**

> The intent is that the `.name` element is a proper class name that's
> legal across languages. Starting with a lower-case letter isn't legal
> in many languages.
>
> The SearchParameter names that are not expressed as UpperCamelCase
> (whether starting with lower-case or containing dashes) were
> erroneous.
>
> Proposal is to correct all of the `.name` elements to follow the
> constraint. We will check that this doesn't negatively impact existing
> implementers as some of our publications have erroneously suggested
> that some SearchParameters were normative.

**Comments:**

- **jmandel — 2023-09-14:** Captures three open questions from the
  FHIR-I discussion: (1) Can the constraint on Canonical be relaxed to
  allow `_` and lowercase letters in the leading position, and would
  that be considered breaking? (Tagged Grahame Grieve.) (2) Do we need
  constraints on `SearchParameter.code` instead of (or in addition to)
  `name`? (3) Do SearchParameters really need both a `name` and a
  `code`?
- **wardweistra — 2026-01-07:** Reports back on behalf of Firely that
  they have no problem with fixing the erroneous `name` elements in
  core SearchParameters to make them proper UpperCamelCase.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax invariant **cnl-0** on `SearchParameter.name` (and on the
underlying Canonical-name pattern wherever it is reused for
SearchParameter) from
`name.exists() implies name.matches('^[A-Z]([A-Za-z0-9_]){1,254}$')`
to
`name.exists() implies name.matches('^[A-Za-z]([A-Za-z0-9_]){1,254}$')`.
Leave the existing non-UpperCamelCase SearchParameter `name`s in the
core specification unchanged. Note in the change log that a leading
lowercase letter is now explicitly permitted for `SearchParameter.name`.

#### Justification

This matches what implementations have de facto been producing and
consuming, since (per the reporter) the bulk of core-spec
SearchParameters never conformed to the original regex. It is a
non-breaking change for instances: anything that satisfied the old
pattern also satisfies the new one, and previously non-conformant
content becomes conformant. It also avoids the risk and churn of
renaming SearchParameters that downstream profiles, IGs, and registry
entries may already reference by `name`.

---

### Disposition B: Alternative Approach (Not Persuasive with Modification — current resolution direction)

#### Proposal

Keep **cnl-0** as written (`^[A-Z]([A-Za-z0-9_]){1,254}$`) on
`SearchParameter.name`, and instead **fix the data** in the core
specification: rename every core SearchParameter whose `name` does not
satisfy cnl-0 to a proper UpperCamelCase identifier, removing dashes
and uppercasing the leading character. Audit other published
artifacts (e.g., R4/R4B/R5 SearchParameter bundles, IG-published
SearchParameters in the HL7 portfolio that were carried into core)
during the same pass. Track the renames in the technical-corrections /
change-log so implementers can update consumers. Separately consider
whether the FHIR-I follow-on questions — relaxing Canonical name
generally, moving validation to `SearchParameter.code`, or collapsing
`name`/`code` — warrant their own tickets.

#### Justification

This reflects the work group's stated intent, captured in the
resolution description: `name` is meant to be a class-style identifier
that is portable across programming languages (many of which disallow a
leading lowercase letter or `_` for type names), so weakening cnl-0
removes a useful guarantee for tooling that generates code from the
spec. Firely (`wardweistra`) has confirmed they are comfortable with
the fix-the-data approach. Because the resolution description also
notes that some non-conformant SearchParameters were "erroneously"
suggested to be normative, treating the renames as corrections of
errors — rather than normative breaks — is defensible. Implementers
that match SearchParameters by `code` (the search-time identifier) or
by canonical URL are unaffected; only those that match by `name` need
to update.

---

### Disposition C: Decline

#### Proposal

Reject the proposed regex change and make no change to either
`SearchParameter.name`, `cnl-0`, or the existing core SearchParameter
instances. Close as Not Persuasive.

#### Justification

This is hard to defend on its own: the ticket has demonstrably
identified a real inconsistency between the constraint and the
specification's own content, and Disposition C resolves neither side of
that inconsistency. It would be appropriate only if the work group
concludes that the ticket is duplicative of separate, larger work on
Canonical names (the FHIR-I discussion's first bullet) and prefers to
consolidate there — but in that case Disposition B is a strictly better
expression of the same intent.

---

### Recommendation

**Recommended disposition:** B

Disposition B (fix the data, keep cnl-0) is consistent with the
resolution already recorded on the ticket
(*Not Persuasive with Modification*) and with the work group's stated
rationale that `SearchParameter.name` is intended to be a
language-portable class-style identifier. It also has on-record
implementer support from Firely, who are arguably the most exposed
party to a rename. Disposition A is tempting because it is
zero-effort, but it permanently erodes a constraint whose purpose
(safe code-generation across languages) is still valid; once relaxed,
it would be very hard to re-tighten. The follow-on FHIR-I questions
about Canonical-name policy and the `name`/`code` duality are real and
worth pursuing, but they are properly the subject of separate tickets
rather than a justification to weaken cnl-0 here. The work group
should confirm that no production implementer is critically dependent
on the current `name` values of the affected core SearchParameters
(per the resolution's own caveat about prior normative claims) and, if
so, proceed with the rename as a technical correction.
