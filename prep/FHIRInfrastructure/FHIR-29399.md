## Summary

The ticket asks the FHIR Shorthand (FSH) specification authors to document
how to constrain a `value[x]` choice element so that it is restricted to
`Reference(Any)` — i.e., to any reference target without narrowing the set
of allowed targets. The reporter notes that the FSH idiom
`* value[x] only Reference` accomplishes this, but observes that the FSH
documentation does not currently call this out, leaving authors to guess
or experiment. The request is purely documentation-oriented; no change to
FSH grammar or compiler behaviour is implied. A linked Zulip thread on
`#shorthand` triggered the ticket and contains the working pattern that
should be captured in the docs.

## Related Zulip Discussions

### [shorthand > Constrain datatype choice to Reference(Any)](https://chat.fhir.org/#narrow/stream/shorthand/topic/Constrain%20datatype%20choice%20to%20Reference%28Any%29/near/214175146)

Jean Duteau asked how to constrain a `value[x]` choice (specifically
`Claim.supportingInfo.value[x]`) so that only `valueReference` is allowed
while keeping the reference target open to any resource. He reported that
`Reference(Any)`, `Reference(any)`, and `Reference()` all failed and that
the FSH documentation did not cover the case. Chris Moesel suggested
`* value[x] only Reference`, reasoning that the underlying `Reference` in
the choice is already to any resource, so the bare `only Reference` form
preserves that openness while removing the other choice types. Jean
confirmed this worked for him and asked whether the same idiom would apply
to an extension's `value[x]`; Chris confirmed it does, since
`Extension.value[x]`'s Reference is also to any by default. Mark Kramer
then said this should be documented and filed FHIR-29399. There is no
disagreement in the thread — consensus is that `* value[x] only Reference`
is the correct pattern and the gap is purely documentation.

Participants: Jean Duteau, Chris Moesel, Mark Kramer.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Shorthand specification's "Rules" / "Type Constraint"
documentation (the section covering the `only` keyword as applied to
choice elements such as `value[x]`) to add an explicit subsection or
worked example for "constraining a choice to `Reference(Any)`". The new
content should:

- State that to restrict a choice element (e.g., `value[x]`) to a
  Reference that may still point to any resource, FSH authors should
  write `* value[x] only Reference` (with no parenthesised target list).
- Explain that supplying `Reference(Any)`, `Reference(any)`, or
  `Reference()` is **not** valid FSH syntax for this case.
- Note that this idiom preserves whatever target types are already
  allowed by the underlying StructureDefinition (e.g., "any" for
  `Extension.value[x]`) and is therefore not a widening of scope.
- Include a short example using `Claim.supportingInfo.value[x]` (the
  scenario in the Zulip thread) and a parallel `Extension.value[x]`
  example.

#### Justification

The request is a low-risk, documentation-only clarification of an
existing, working FSH idiom. The Zulip thread shows that experienced FSH
users hit the gap in practice and resolved it only by chance via chat
support. Documenting the pattern aligns with FSH's stated goal of being a
human-friendly authoring language and removes a recurring source of
implementer confusion. Because the change does not touch grammar,
compiler, or output semantics, it carries no compatibility risk.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying need but address it more broadly than the literal
request. Rather than adding a single `Reference(Any)` example, restructure
(or add) a "Constraining choice elements" subsection in the FSH spec that:

- Tabulates how to (a) drop choices, (b) restrict a choice to a single
  primitive type, (c) restrict a choice to `Reference` keeping the
  underlying target list, and (d) restrict a choice to `Reference(Type1
  | Type2)` to narrow the targets.
- Calls out the exact syntactic difference between `only Reference` and
  `only Reference(Type)` and warns that `Reference(Any)` / `Reference()`
  are not legal FSH.
- Cross-links from the `Extension.value[x]` and observation/claim
  authoring tutorials so readers hit the guidance from the most common
  entry points.

#### Justification

The Zulip thread shows that the confusion is not specific to
`Reference(Any)` — users routinely struggle with how `only` interacts
with choice elements in general. A targeted-but-narrow note (Disposition
A) closes this one ticket but leaves adjacent gaps. A short table costs
little additional effort over a single example and substantially reduces
the chance of follow-up tickets on neighbouring patterns. Trade-off
versus A: slightly more authoring effort and a slightly larger review
surface, but materially better long-term documentation quality.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Persuasive with Mod" or "Not Persuasive" on the grounds
that the existing FSH spec already documents the `only` keyword for type
constraints, and that the working idiom (`* value[x] only Reference`)
follows directly from the documented general rule that `only <Type>`
restricts a choice to that type while preserving its existing reference
targets. Close without a documentation change.

#### Justification

A defensible declination would argue that FSH cannot enumerate every
combination of `only` and choice elements, that the general rule already
covers this case, and that workgroup time is better spent on substantive
language changes than restating implications of existing rules. This is
the weakest of the three positions because the Zulip thread is direct
empirical evidence that the rule is *not* discoverable from the current
docs even by experienced FSH users.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because the underlying confusion in the
Zulip thread is broader than the literal `Reference(Any)` question — it
is really about how `only` composes with choice elements and reference
target types. A small, well-scoped "Constraining choice elements" table
or subsection costs only marginally more than the single example proposed
in A but pre-empts the most common neighbouring questions (drop a
choice, restrict to a single type, narrow reference targets) that
otherwise tend to surface as follow-up tickets. Disposition A is an
acceptable fallback if editor bandwidth is tight; Disposition C is not
recommended because the empirical evidence (a multi-message thread with
two experienced FSH contributors) shows the current documentation does
not communicate the pattern, regardless of whether it is "implied" by
the general rule.
