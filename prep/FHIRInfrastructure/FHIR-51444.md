## Summary

Elliot Silver reports that the FHIR Core resources `Condition` and `Encounter`
appear to model the same clinical association from both sides, creating what
looks like a circular reference. `Condition.encounter` points to "the
Encounter during which this Condition was created", while `Encounter.reason`
(which may reference a `Condition`) and `Encounter.diagnosis.condition` ("the
diagnosis relevant to the encounter") both point back at `Condition`. An
implementer he consulted was unclear whether the relationship should be
recorded on the `Condition`, on the `Encounter`, or on both. The ticket
asks the work group either to clarify the element definitions so the
intended usage is unambiguous and circularity is avoided in practice, or —
if circular references really are intended — to revisit the model, since
circular references are against FHIR design best practice. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced
by the cross-reference service.

## Details

**Description:**

> Condition contains `encounter` ("The Encounter during which this Condition was created).
>
> Encounter contains `reason.value` which can reference Condition, and `diagnosis.condition` ("The diagnosis relevant to the encounter").
>
> Although Encounter.reason would appear to point to an existing condition, it seems likely that Condition.encounter and Encounter.diagnosis.condition would reference each other. Discussion with an implementer indicated confusion about whether they should record the relationship in one direction or the other or both.
>
> If these elements are not intended to create circular references, please clarify the definitions so that it the uses are clearer and it is easier to understand why circular references wouldn't be created.
>
> If these elements are expected to create circular references, please revisit modelling as circular references are against FHIR design best practice.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Clarify the element definitions in FHIR Core so that the intent of each
of the three elements — `Condition.encounter`, `Encounter.reason`, and
`Encounter.diagnosis.condition` — is unambiguous, and so that a circular
pair of references is not the expected shape. Concretely:

- Tighten `Condition.encounter` to state that it captures the
  encounter in which the condition was **first recorded / established**
  (a provenance / context link), not a redundant mirror of
  `Encounter.diagnosis`.
- Tighten `Encounter.diagnosis.condition` to state that it captures
  conditions the encounter is **specifically addressing** (billing /
  clinical focus), with guidance that the set of diagnoses for an
  encounter is normally curated rather than auto-populated from every
  condition whose `encounter` points at it.
- Tighten `Encounter.reason` (and `Encounter.reason.value` when it is a
  `Reference(Condition)`) to clarify that it captures the **reason the
  encounter was initiated**, which may be a pre-existing condition and
  is distinct from `diagnosis`.
- Add a short explanatory block (implementer note or usage note on
  both `Condition` and `Encounter`) describing the three elements
  together, stating which side is authoritative for which use case,
  and explicitly noting that populating all three directions for the
  same clinical fact is not required and generally not expected.

The definitions would be revised in the FHIR Core specification
(`HL7/fhir` authoring sources for `Condition` and `Encounter`), with no
change to cardinalities, types, or invariants.

#### Justification

This is the minimally-disruptive path: it addresses the reported
confusion without breaking any existing instances or profiles. The three
elements already have distinct intended semantics (provenance of the
condition, reason the encounter was booked, diagnoses addressed during
the encounter); the problem is that the current short definitions do
not make those distinctions clear enough for a cold-read implementer.
Clarifying definitions is exactly the kind of change FHIR Infrastructure
and Patient Care regularly coordinate on, and it respects the comment
from `corey_spears` that this crosses work groups — the clarifications
can be drafted by FHIR-I and reviewed by the owning WG of each resource.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of only clarifying prose, **make one direction canonical** and
mark the others as secondary / derived:

- Declare `Encounter.diagnosis.condition` the canonical link between an
  encounter and the conditions it addresses clinically.
- Declare `Encounter.reason` (when a `Reference(Condition)`) the
  canonical link for the reason the encounter was booked.
- Change `Condition.encounter` to an explicit **provenance-only** link
  ("the encounter in which this Condition was first recorded") and add
  usage guidance (and, optionally, a non-normative invariant on
  `Condition` or an `Encounter` SearchParameter hint) stating that
  systems should not rely on `Condition.encounter` for enumerating an
  encounter's diagnoses — they should traverse
  `Encounter.diagnosis.condition` instead.

No element is removed, no cardinality changes, but the definitions and
usage notes are opinionated about which side is authoritative for which
query, removing the need for implementers to populate both directions.

#### Justification

Disposition A removes the confusion in prose but leaves the three
elements semantically overlapping; implementers can still reasonably
populate all three for the same fact, and server developers still have
to decide which to trust when they disagree. Picking a canonical
direction per use case gives implementers a single place to look and
removes the "do I write it once or twice?" question the ticket
describes. It is still non-breaking (no schema change), but it is more
prescriptive than A. The trade-off is that it may conflict with
existing IG guidance (e.g. US Core, IPS) that happens to populate
`Condition.encounter`; that guidance would need to be reviewed for
consistency, which is why B is heavier than A.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as **not-persuasive / working-as-designed**: the
three elements describe different clinical facts (provenance of the
`Condition` record, booked reason for the `Encounter`, and curated
diagnoses addressed during the `Encounter`), and the apparent circle is
only a circle at the reference-graph level, not at the semantic level.
FHIR Infrastructure would respond that bidirectional references are
permitted in FHIR when they carry different semantics, and that
existing definitions — combined with the general FHIR guidance on
`Reference` — are sufficient.

#### Justification

Circular reference *graphs* are not in themselves forbidden by FHIR;
FHIR best practice warns against circular *definitional* dependencies
and against modelling the same fact twice, neither of which is strictly
what is happening here. If the work group judges that the distinctions
are already adequately documented (and that the implementer confusion
is an education issue rather than a specification issue), declining
avoids committing editorial effort and IG-review cost for a change that
does not alter conformance. This is the weakest option given the
reporter explicitly requested clarification, but it is defensible if
the WG believes the current definitions are already precise.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers what the reporter asked for — clearer
definitions so implementers stop having to guess — at the lowest cost
and with no risk to existing instances or IGs. The comment from
`corey_spears` already flags that Condition and Encounter are owned by
different work groups and suggests FHIR-I initiate the discussion;
coordinating a prose clarification across FHIR-I and the owning WGs
(Patient Care for Condition, Patient Administration for Encounter) is
exactly the kind of change that fits naturally into a cross-WG review.
Disposition B is attractive in principle but materially heavier:
declaring a canonical direction may conflict with established IG
guidance and deserves its own ticket once A has landed. Disposition C
ignores the reporter's evidence of actual implementer confusion and is
not recommended. Start with A, and if reviewers find that prose alone
cannot eliminate the ambiguity, promote the work to the B path in a
follow-up ticket.
