## Summary

The ticket reports a defect in the FHIR Core specification's table that maps
FHIR primitive types to their corresponding FHIRPath/System types. Today the
table maps `FHIR.date` to `System.DateTime`, but FHIRPath now defines a
distinct `System.Date` type, so `FHIR.date` should map directly to
`System.Date`. Reporter Brian Postlethwaite also notes in a follow-up
comment that the FHIR `xhtml` primitive is missing from the mapping table
entirely. The ticket has been triaged with a Persuasive resolution
("Update mapping as requested.") and is queued by the reporter as part of
a proposed FHIR-I block vote covering several FHIRPath-page corrections
(see the Zulip thread referenced below).

This issue is essentially a re-raise of the older
[FHIR-28617](https://jira.hl7.org/browse/FHIR-28617) ("Translation for
Primitive Types misses System.Date"), filed by Alexander Kiel in 2020,
which made exactly the same observation: with the introduction of
`System.Date` in FHIRPath, `FHIR.date` (and only `FHIR.date`, not
`FHIR.dateTime` or `FHIR.instant`) should translate to `System.Date`.
FHIR-28617 was closed as Duplicate of this newer ticket but carries the
same Persuasive resolution, reinforcing that the underlying mapping is
agreed to be wrong and should be corrected.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR primitive → FHIRPath/System type mapping table in the FHIR
Core specification (the FHIRPath-usage page that enumerates how each FHIR
primitive is exposed to FHIRPath) so that:

- `FHIR.date` maps to `System.Date` (not `System.DateTime`).
- `FHIR.dateTime` continues to map to `System.DateTime`.
- `FHIR.instant` continues to map to `System.DateTime`.
- A new row is added for `FHIR.xhtml`, mapping to `System.String` (the
  natural FHIRPath representation for an xhtml-bearing primitive whose
  `value` is a string), per Brian's follow-up comment.

Apply this change as a Technical Correction to FHIR Core, since it aligns
the spec with the FHIRPath type system as it actually exists today and
matches the Persuasive resolution already recorded on the ticket and on
the duplicate FHIR-28617.

#### Justification

The current table predates the introduction of `System.Date` in FHIRPath.
Mapping `FHIR.date` (which has no time component, by FHIR's own primitive
definition) to `System.DateTime` is internally inconsistent: it forces
implementers to invent a time portion that the source value does not
have, and it loses the date-only semantic that both FHIRPath and FHIR
have separately defined. The fix restores type-system parity between
FHIR and FHIRPath, is non-breaking for any implementation that already
respects date-only semantics, and is the change the reporter, the
duplicate ticket FHIR-28617, and the FHIR-I Zulip thread (with a proxy
vote from Grahame in favour) all agree on. Adding the missing `xhtml`
row is a documentation completeness fix and addresses an obvious gap
called out in the comment.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the `FHIR.date` → `System.Date` correction (the core ask), but
split the `xhtml` follow-up into a separate technical-correction ticket
rather than bundling it. Additionally, when fixing the row, also audit
the rest of the table in the same edit pass against the current FHIRPath
spec (e.g., `FHIR.string` vs `System.String`, `FHIR.boolean` vs
`System.Boolean`, decimal/integer ranges, the precision footgun
documented in FHIR-50939) and either confirm or correct each row, so the
FHIR-I block vote disposes of the whole table consistently rather than
patching one cell.

#### Justification

The reporter is already gathering a cluster of FHIRPath-page corrections
for a single FHIR-I block vote (see the Zulip thread). Treating the
mapping table as a single object of review — rather than approving a
one-cell edit and a comment-driven row addition in one motion — produces
a cleaner audit trail and reduces the chance that the next reader files
a third ticket against the same table. The xhtml addition is a
reasonable change but is technically a different defect (missing row vs.
incorrect mapping); separating it keeps each ticket's resolution
self-contained and easier to revert if implementer feedback surfaces a
problem with the chosen `xhtml` mapping.

---

### Disposition C: Decline

#### Proposal

Decline to change the mapping and instead add a non-normative note to
the table explaining that `FHIR.date` is shown as mapping to
`System.DateTime` for historical/back-compat reasons and that
implementations may treat it as `System.Date` where the time component
is absent. Leave the table itself unchanged.

#### Justification

This option is included for completeness only. It would be defensible
solely on a "do not perturb a published mapping that downstream tools
may have hard-coded" theory of change. There is no technical argument
in either the ticket, the duplicate FHIR-28617, or the FHIR-I Zulip
thread in favour of preserving the current mapping; the resolution
already recorded ("Persuasive — update mapping as requested") explicitly
points the other way.

---

### Recommendation

**Recommended disposition:** A

The mapping is wrong on its face: FHIRPath has had `System.Date` for
years, FHIR.date is by definition date-only, and the current table
mis-states the relationship. The same defect was filed and agreed to in
2020 (FHIR-28617) and re-filed in 2026 (this ticket), both with
Persuasive resolutions; the FHIR-I Zulip agenda thread shows no
technical opposition and an explicit proxy vote from Grahame Grieve in
favour. Disposition B's "audit the whole table" instinct is sound
hygiene but should not block the fix — the workgroup can take B's audit
as a follow-up action item while still applying A now, and the `xhtml`
addition is small enough and clearly attributable to the same comment
trail that bundling it into A is reasonable. Declining (C) would leave
the spec demonstrably inconsistent with FHIRPath and would have to
overrule two prior Persuasive resolutions, which is not warranted on
this record.
