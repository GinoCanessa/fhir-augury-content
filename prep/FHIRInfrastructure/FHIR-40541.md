## Summary

Lloyd McKenzie observes that, conceptually, the Subscriptions framework
fits the FHIR workflow pattern triad: a `Subscription` is effectively a
**request** for information, expressed against a **definition**
(`SubscriptionTopic`), and satisfied by **events** (carried in
`SubscriptionStatus` notifications). `SubscriptionTopic` is already
treated as a Definition resource, but neither `Subscription` (as a
Request) nor `SubscriptionStatus` (as an Event) is currently mapped to
the corresponding workflow pattern. Lloyd notes that these resources
also behave quite differently from the canonical Request/Event resources
(e.g., ServiceRequest, Observation, Procedure) and asks whether adding
the workflow-pattern mappings would actually be valuable.

The ticket has no Jira cross-references and no comments, and no Zulip or
GitHub items in the FhirAugury index reference it directly. It does,
however, sit very close to the already-Published change in
[FHIR-34092](https://jira.hl7.org/browse/FHIR-34092) ("Correct
'Definition' mappings"), which removed the Definition mapping from
`CanonicalResource`, individually propagated Definition mappings to
`SubscriptionTopic` (alongside ActivityDefinition, PlanDefinition, etc.),
and added an **Event** pattern mapping to `SubscriptionStatus`. That
prior change effectively addresses two of the three legs Lloyd's question
implies, so FHIR-40541 reduces in practice to: *should `Subscription`
itself be mapped to the **Request** pattern?* The related ticket
[FHIR-15125](https://jira.hl7.org/browse/FHIR-15125) shows the historical
cost of fitting a resource to the Event pattern (status,
statusReason, reasonReference, performer.actor alignment, etc.) and is
useful precedent for reasoning about how forced a Subscription→Request
mapping would feel.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add formal workflow-pattern mappings to the two Subscription-family
resources that currently lack them, in `HL7/fhir` (FHIR Core, R6):

1. On **`Subscription`**, add a mapping group to the **Request**
   pattern (`workflow`/`request`), aligning at minimum:
   - `Subscription.id` → `Request.identifier` (conceptual)
   - `Subscription.status` → `Request.status`
   - `Subscription.topic` → `Request.instantiatesCanonical`
   - `Subscription.end` → `Request.occurrence[x]` (end of validity)
   - `Subscription.reason` → `Request.reasonCode` (textual)
   - `Subscription.contact` → `Request.requester` (party to contact)
   - Document gaps (no `intent`, no `priority`, no `authoredOn`,
     no `performer`) explicitly in the mapping comments rather than
     adding new elements.
2. Confirm and (if needed) tighten **`SubscriptionStatus` →
   Event** mapping that was added under
   [FHIR-34092](https://jira.hl7.org/browse/FHIR-34092) so that
   `SubscriptionStatus.status` aligns with `Event.status` and the
   notification semantics are documented.
3. Add `Subscription` and `SubscriptionStatus` to the resource list on
   the workflow page (`workflow.html`) under Request and Event
   respectively, mirroring the treatment SubscriptionTopic already
   receives as a Definition.

#### Justification

Mapping Subscription/SubscriptionStatus to the workflow patterns makes
the Subscriptions framework discoverable from the workflow module and
documents the Request/Definition/Event triad that already informs the
design. It is consistent with the precedent set by
[FHIR-34092](https://jira.hl7.org/browse/FHIR-34092), which deliberately
chose to add Event-pattern mappings to SubscriptionStatus rather than
leave it pattern-less. Adding the corresponding Request mapping to
Subscription closes the obvious symmetry gap and, per the reporter,
reflects how the resources actually behave conceptually. R6 is the right
window because the labels already mark this as `R6`.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a full pattern mapping. Instead, in the FHIR Core spec
add a short, normative-style narrative section to the `Subscription`
resource page (and to the workflow module overview in `workflow.html`)
that:

1. Explicitly states the conceptual alignment — Subscription is a
   Request, SubscriptionTopic is its Definition, SubscriptionStatus is
   the Event — without mapping individual elements.
2. Lists the deliberate divergences from the Request and Event patterns
   (no `intent`, no `priority`, no `authoredOn`, no `performer`,
   notification-shaped Event rather than per-occurrence Event, etc.) so
   implementers understand why the patterns are not formally claimed.
3. Cross-links from the Workflow page's Request and Event sections to
   the Subscription resource page under a "Related but not pattern-
   conformant" heading.

#### Justification

The reporter explicitly notes that Subscription and SubscriptionStatus
"are quite different from most of the resources that implement the
workflow patterns." A formal mapping risks suggesting more conformance
than actually exists: Subscription has no `intent`, no `priority`, no
`authoredOn`, and no `performer`; SubscriptionStatus is a per-
notification snapshot, not a per-occurrence Event. The precedent in
[FHIR-15125](https://jira.hl7.org/browse/FHIR-15125) shows that
seriously claiming the Event pattern usually triggers element additions
(status codes, statusReason, reasonReference, performer alignment) — a
breaking-change cost the Subscriptions framework should not pay just to
flag a conceptual analogy. A documentation-only alignment delivers the
discoverability benefit without forcing element churn or implying
conformance the resources do not meet.

---

### Disposition C: Decline

#### Proposal

Reject the proposal to add Request and Event mappings to Subscription
and SubscriptionStatus. Note in the resolution that
[FHIR-34092](https://jira.hl7.org/browse/FHIR-34092) already
established the Definition mapping for SubscriptionTopic and an Event
mapping for SubscriptionStatus, and that further pattern alignment is
out of scope for the Subscriptions framework, which is intentionally
modelled as its own pub/sub mechanism rather than as a workflow
participant.

#### Justification

The Subscriptions framework is a transport / notification mechanism, not
a clinical or operational workflow artefact in the sense of
ServiceRequest, Procedure, or Observation. Forcing it into the
Request/Event pattern vocabulary has limited implementer value
(Subscriptions are configured by clients and admins, not "ordered" by
performers) and a real cost in mapping noise and reviewer confusion.
The change is also low-priority (Medium, Triaged, no commenters, no
Zulip discussion, no GitHub references), suggesting limited community
demand.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — documentation-
only alignment).

The reporter himself flags that Subscription and SubscriptionStatus
"are quite different from most of the resources that implement the
workflow patterns," and the precedent in
[FHIR-15125](https://jira.hl7.org/browse/FHIR-15125) shows that
formally claiming the Event pattern typically triggers element churn
the Subscriptions framework cannot absorb in R6 without breaking
implementers (Subscription has no `intent`, `priority`, `authoredOn`,
or `performer`; SubscriptionStatus is per-notification rather than per-
occurrence). At the same time, simply declining (Disposition C) leaves
the asymmetry the reporter identified — SubscriptionTopic is mapped to
Definition (per FHIR-34092), SubscriptionStatus is mapped to Event (per
FHIR-34092), and only Subscription is left without any workflow
positioning — which is exactly the inconsistency that motivated the
ticket. Disposition B captures the conceptual alignment in narrative
form, points implementers from the workflow module to the Subscriptions
framework, and explicitly catalogues the deliberate divergences,
without committing the work group to additive R6 changes that would
otherwise be required to honour a literal Request-pattern mapping. If
the work group later decides the Request-pattern conformance is worth
the implementer cost, Disposition A is the natural follow-on and
remains compatible with B as a stepping-stone.
