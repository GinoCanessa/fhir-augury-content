## Summary

Sean Fong (CSIRO) requests that the `Endpoint` resource
(`https://hl7.org/fhir/R4/endpoint.html`) be added as a standard code to the
Structured Data Capture (SDC) `launchContext` ValueSet
(`http://hl7.org/fhir/uv/sdc/ValueSet/launchContext`) and the companion
CodeSystem (`http://hl7.org/fhir/uv/sdc/CodeSystem/launchContext`), with the
`type` invariant (`inv-1` on the `sdc-questionnaire-launchContext`
extension) updated accordingly so that an `endpoint` code maps to the
`Endpoint` resource type. The driving use case is a SMART-App-Launch-style
"launch within a launch": CSIRO is passing a launch endpoint URL into a
Questionnaire via `fhirContext`, then using a `cqf-expression` to render a
link. The request grew out of a Zulip discussion on `#questionnaire >
Launch context extensibility` where Lloyd McKenzie confirmed that
implementers can already define their own launch-context codes (inv-1 only
constrains the standard ones) but encouraged filing a change request to
add additional standard codes. The ticket was originally resolved
"Considered for Future Use" on 2025-08-14 (Postlethwaite / Rhodes, 3-0-0)
because there was insufficient time to discuss before the STU4 lock, and
was re-opened on 2026-03-26 post-STU4 publication for reconsideration in
the next release. No cross-referenced FHIR Jira tickets were found.

## Details

**Description:**

Add **Endpoint** resource (https://hl7.org/fhir/R4/endpoint.html) to the
launch context ValueSet.

_Background:_ Based on the Zulip thread
https://chat.fhir.org/#narrow/channel/179255-questionnaire/topic/Launch.20context.20extensibility/with/530003519.

Within the context of SMART App Launch, CSIRO is experimenting with
performing a launch within a launch. The idea is to pass a launch endpoint
URL (via `fhirContext`) into a Questionnaire, then use a `cqf-expression`
to render the link. Theoretically the `cqf-expression` should be able to
access the endpoint URL via a `launchContext` defined as follows:

```json
{
  "url": "http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-questionnaire-launchContext",
  "extension": [
    {
      "url": "name",
      "valueCoding": {
        "system": "http://hl7.org/fhir/uv/sdc/CodeSystem/launchContext",
        "code": "endpoint"
      }
    },
    { "url": "type", "valueCode": "Endpoint" },
    {
      "url": "description",
      "valueString": "An AusCVDRisk-i launch endpoint to pre-populate a cqf-expression in the Absolute cardiovascular disease risk calculation section."
    }
  ]
}
```

## Related Zulip Discussions

### [questionnaire > Launch context extensibility](https://chat.fhir.org/#narrow/stream/questionnaire/topic/Launch%20context%20extensibility/near/530107640)

Sean Fong (CSIRO) opened the thread asking whether the SDC `launchContext`
ValueSet's five predefined codes (`patient`, `encounter`, `location`,
`user`, `study`) are intended to be extensible, citing that the binding
appears extensible. He then noted that `inv-1` on the
`sdc-questionnaire-launchContext` extension enumerates allowed resource
types per code (e.g., `patient` → Patient, `user` → Patient /
Practitioner / PractitionerRole / RelatedPerson, `encounter` → Encounter,
`location` → Location, `study` → ResearchStudy) and asked how that
reconciles with extension. Lloyd McKenzie replied that `inv-1` only
enforces rules for the standard codes and that implementers are free to
define their own codes, but encouraged Sean to submit a change request to
add additional standard codes (Lloyd's example was `ImagingStudy`). Sean
then filed FHIR-51537, scoped to adding `Endpoint` (his actual use case)
rather than `ImagingStudy`. No disagreement was expressed; consensus in
the short thread was that adding standard codes via a change request is
the appropriate path.

Participants: Sean Fong, Lloyd McKenzie.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new concept `endpoint` to the SDC `launchContext` CodeSystem
(`http://hl7.org/fhir/uv/sdc/CodeSystem/launchContext`), extend the
`launchContext` ValueSet
(`http://hl7.org/fhir/uv/sdc/ValueSet/launchContext`) to include it, and
amend invariant `inv-1` on `StructureDefinition/sdc-questionnaire-launchContext`
so that `name = endpoint` requires `type = Endpoint`. Update the narrative
on the launchContext extension page to describe the intended use (passing
an `Endpoint` reference into a Questionnaire, typically in support of
SMART-App-Launch "launch within a launch" scenarios and pre-population
via `cqf-expression`). Include a short example mirroring the CSIRO
AusCVDRisk-i payload in the ticket.

#### Justification

`Endpoint` is a first-class R4/R5 resource that naturally represents the
kind of launch target the reporter describes, and the existing
`launchContext` binding is explicitly extensible — Lloyd McKenzie has
already confirmed on Zulip that implementers may define their own codes,
so standardising a widely-useful one is consistent with the pattern
already used for `patient`, `encounter`, `location`, `user`, and `study`.
Adding the code to the CodeSystem / ValueSet and extending `inv-1` is
additive and backward-compatible: existing Questionnaires continue to
validate, and consumers gain a standard identifier instead of each vendor
minting their own. The ticket also carries a concrete, implemented use
case (CSIRO's AusCVDRisk-i pre-population), which was the bar the
previous 2025-08-14 vote ("need more discussion/experience") was waiting
on — re-opening post-STU4 is the right point to act on that experience.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding only `Endpoint`, open a single coordinated change that
expands the SDC `launchContext` CodeSystem / ValueSet with the set of
standard codes the community has been asking for in this thread and
elsewhere — at minimum `endpoint` (→ `Endpoint`) and `imagingstudy` (→
`ImagingStudy`, as explicitly suggested by Lloyd in the same Zulip
thread), and review whether `organization`, `device`, or similar
candidates should be batched in. Update `inv-1` once with all new
(code, type) pairs, publish a consolidated example set, and add a brief
note in the SDC launchContext narrative clarifying that the binding
remains extensible and documenting the pattern for implementer-defined
codes (so future one-off additions don't each require a change request).

#### Justification

The Zulip thread that prompted FHIR-51537 is the same thread in which
Lloyd encouraged submitting a change request to add `ImagingStudy` as a
standard code; `Endpoint` is effectively the first of a predictable
series of such requests. Publishing additions one-at-a-time produces
repeated churn on the CodeSystem, ValueSet, invariant, narrative, and IG
examples for what is conceptually the same change shape. A single
batched disposition amortises that cost, gives implementers one
coherent "expanded launchContext" section to read, and reduces the risk
that successive point additions drift in naming conventions (casing of
codes, phrasing of invariant, style of description). The trade-off vs.
Disposition A is a small delay to canvas other candidate resources, which
is reasonable given that the work group has explicitly chosen to defer
this to post-STU4 anyway.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing mechanism already
supports the reporter's use case: the `launchContext` binding is
extensible, `inv-1` only constrains the standard codes, and implementers
— including CSIRO — are free to define their own `endpoint` code today
without any specification change. Close as "Not Persuasive" / "Already
Supported", and instead add (or strengthen) guidance in the SDC
`launchContext` narrative that describes how to mint implementer-defined
launch-context codes (naming conventions, the fact that `inv-1` doesn't
apply to custom codes, example payload).

#### Justification

This position is directly supported by Lloyd McKenzie's answer in the
Zulip thread — he confirmed that defining custom codes is already
allowed. Every resource type is potentially a legitimate launch-context
target (Endpoint, ImagingStudy, Device, Organization, Task, …), and
standardising each one as a new code indefinitely expands the SDC
CodeSystem without adding meaningful interoperability, because consumers
of a Questionnaire have to understand every launch-context code they
encounter anyway. Declining keeps the CodeSystem small and predictable
while documentation improvements solve the real problem (implementers
not being confident they may extend). The gap here is primarily
discoverability of an existing capability, not a missing capability.

---

### Recommendation

**Recommended disposition:** B

Disposition A is reasonable and gives the reporter what he asked for,
and Disposition C is defensible on purely technical grounds — but both
miss the signal in the originating Zulip thread. The same conversation
that produced this ticket also explicitly suggested `ImagingStudy` as a
standard code; absent a change in approach, the work group will be back
here shortly for a near-identical ticket. Disposition B treats
FHIR-51537 as the forcing function for a small, batched expansion of the
SDC `launchContext` CodeSystem / ValueSet plus a one-time clarification
that implementer-defined codes remain valid. That addresses the
reporter's concrete AusCVDRisk-i use case, honours Lloyd's Zulip
suggestion, and heads off the next several "please add X to
launchContext" tickets — which aligns with the 2026-03-26 decision to
revisit this post-STU4 with more implementer experience in hand. If the
work group wants to keep scope tight, falling back to Disposition A is a
clean second choice.
