## Summary

FHIR-55143 is one of a large family of "Spelling/abbreviation issues on
page: …" technical-correction tickets opened by Lloyd McKenzie against
FHIR Core. This particular instance targets the **Bundle** resource page
(`bundle.html`). An automated check flagged four tokens that the editor
considers either misspellings, undesirable contractions, or
unexplained abbreviations and asks that they be corrected or fully
spelled out:

- `doesn't` (contraction)
- `fullURL` (abbreviation/casing)
- `fullURLs` (plural form)
- `fullURLs,` (the same token followed by a comma — same underlying
  occurrence)

There are no cross-references to other Jira tickets, Zulip threads, or
GitHub items for this ticket. It is one of a series of essentially
identical tickets covering many other Core pages (documentation,
graphql, languages, managing, operations, pushpull, services, updates,
versioning, json, etc. — see the Related Jira Tickets section), so the
disposition chosen here is likely to set the pattern for the whole
batch.

The complication unique to **Bundle** is that `fullUrl` (note the
lowercase `u`) is the **literal JSON/XML element name** on
`Bundle.entry.fullUrl`. The flagged tokens are about the casing and
the prose form of references to that element, not about the element
itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the Bundle resource narrative page (`bundle.html`, generated from
the Bundle resource's intro/notes source files in `HL7/fhir`) so that
each flagged token is corrected or expanded:

- Replace the contraction **`doesn't`** with **`does not`**.
- Replace the abbreviation **`fullURL`** / **`fullURLs`** in prose
  with **`full URL`** / **`full URLs`** wherever the text is
  describing the *concept* of an entry's full URL.
- Where the prose is referring to the actual element
  `Bundle.entry.fullUrl` (lowercase `u`, single word), wrap it in
  `<code>fullUrl</code>` (or the page's existing element-name markup
  convention) so it is unambiguously the element name and not an
  abbreviation. Plural usages (e.g. "the entries' `fullUrl` values")
  should be rephrased so the element name is not pluralised inside
  the code span.
- Re-run the spelling/abbreviation check against the page to confirm
  zero remaining flags.

No normative behaviour, schema, or constraints change; this is
prose-only editorial work on a single page.

#### Justification

This is a Highest-priority technical-correction ticket from the editor
explicitly to clean up prose. Each flagged token has a clean, low-risk
fix that matches existing FHIR house style: the spec already prefers
expanded forms over contractions in normative text, and existing pages
already mark FHIR element names with code formatting. Distinguishing
"the concept of a full URL" from "the `fullUrl` element" also removes
a real source of confusion for implementers reading the page, and is
consistent with how other resource pages (e.g., MessageHeader,
Subscription) format element references.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this ticket as one slice of the larger batch (FHIR-55036,
-55105, -55109, -55111, -55116, -55125, -55129, -55131, -55132,
-55248, FHIR-55143, …) and resolve them under a single editorial
sweep:

1. Establish/confirm a Core-wide style rule:
   - No contractions in normative or descriptive prose ("does not",
     "cannot", "will not", …).
   - FHIR element names always rendered in `<code>` and never
     "abbreviation-corrected" — i.e., `fullUrl` stays `fullUrl` when
     marked as code; `fullURL` in plain prose is always rewritten to
     either the code form or to "full URL".
2. Add the rule (or a pointer to an existing one) to the FHIR
   contributor / editorial conventions page so future authors do not
   re-introduce the same issues.
3. Apply the fixes from Disposition A to `bundle.html`, and ask the
   editor (or run the same automated check) to roll equivalent fixes
   across the sibling pages in the batch.
4. Resolve FHIR-55143 with a reference to the umbrella editorial fix
   and close the sibling tickets the same way.

#### Justification

The 10+ keyword-similar siblings make it clear that this is a
systemic editorial issue, not a one-page slip. Fixing each ticket in
isolation risks inconsistent treatment (one page expands `fullURL`
to "full URL"; another wraps it as `<code>fullUrl</code>`) and burns
significant editor time per ticket. A single style-rule pass plus a
batch edit produces a uniform result and makes the spelling/abbreviation
check a useful long-term gate. The trade-off vs. Disposition A is
extra coordination cost up front, in exchange for not having to
re-litigate the same questions on every sibling ticket.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as **Not Persuasive** (or **Not Persuasive with
Mod**) on the grounds that:

- `fullUrl` is the literal name of the element `Bundle.entry.fullUrl`
  in the FHIR specification and cannot be "spelled out". The
  spelling/abbreviation checker is producing a false positive when
  it flags the element name.
- `doesn't` is a normal English contraction and is widely tolerated in
  FHIR's narrative pages; rewriting it does not change meaning.

Resolution comment would clarify that prose references to the element
should already be code-formatted, and propose closing the ticket
without an editorial change to `bundle.html`.

#### Justification

This is a defensible rejection if the editors decide the cost of
sweeping prose-style changes outweighs the benefit, and if existing
occurrences of `fullUrl` on the Bundle page are already inside `<code>`
spans (in which case the checker is the thing that needs adjusting,
not the page). It also avoids precedent-setting work across the
sibling tickets. The downside is that the ticket comes from the
project editor at Highest priority, so a Decline disposition needs a
strong justification — primarily that the underlying text is already
correct and the tool is over-flagging.

---

### Recommendation

**Recommended disposition:** B

The pattern of ten-plus near-identical sibling tickets (all opened by
the same reporter at Highest priority, all using the same template,
all flagging the same kinds of tokens) strongly suggests a
spec-wide editorial-style sweep is the right unit of work, not
ten-plus independent page edits. Disposition B keeps the user-visible
outcome of Disposition A on `bundle.html` (contractions expanded,
`fullURL`/`fullURLs` either rewritten to "full URL" or wrapped as the
`fullUrl` code span), but anchors it in a written convention so the
sibling tickets resolve consistently and future contributions do not
regress. Disposition C is hard to defend given the editor explicitly
asked for these fixes; Disposition A solves the immediate ticket but
leaves the sibling tickets to discover the same answers
independently.
