## Summary

The reporter notes that the prose introduction to Search Prefixes in the
FHIR Core specification (section 3.2.1.5.1 of `search.html`, on the
6.0.0-ballot4 build) describes prefixes as being "allowed on the
`number` search type", with `number` as the only example called out.
This is an under-statement of the actual rules: in normative content
elsewhere on the same page, prefixes are also permitted on `date` and
`quantity` search parameter types. The reporter asks that the
introductory sentence be updated to enumerate the full set of search
parameter types on which prefixes can be used (at minimum `number`,
`date`, and `quantity`) so that the narrative description matches the
normative table and avoids misleading implementers who only read the
intro paragraph.

There are no Jira cross-references, no Zulip discussions referencing
this ticket, and no linked GitHub items in Augury's index, so the scope
is the documentation change in isolation.

## Details

**Description:**

> Finally, a [Search Prefix](https://hl7.org/fhir/6.0.0-ballot4/search.html#prefix),
> allowed on the `number` search type, can be used to change the test to
> something like a 'greater than' (i.e., `{element value}` is greater than

Considering adding `date` and `quantity` to this description as search

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the introductory sentence in `search.html` section 3.2.1.5.1 so
that it enumerates the search parameter types on which prefixes may be
used, matching the normative table in `search.html#prefix`. Concretely,
replace text of the form

> Finally, a Search Prefix, allowed on the `number` search type, can be
> used to change the test to something like a 'greater than' …

with text of the form

> Finally, a Search Prefix, allowed on the `number`, `date`, and
> `quantity` search parameter types, can be used to change the test to
> something other than equality — for example, a 'greater than'
> comparison (i.e., `{element value}` is greater than `{input value}`).

No changes are needed to the prefix table, the prefix definitions, or
any structure definition; this is a pure narrative correction in the
search page intro.

#### Justification

The change is a strict documentation clarification with no normative
impact. The normative content on the same page already permits prefixes
on `number`, `date`, and `quantity` (and gives type-specific rules for
each). Aligning the intro sentence with the normative table removes a
contradiction that can mislead implementers who skim the intro and
conclude that prefixes only apply to `number` parameters. The change is
consistent with FHIR's general editorial practice that narrative
descriptions should not under-specify what the normative tables allow.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than editing only the one sentence in 3.2.1.5.1, restructure the
intro so the prefix description does not enumerate types inline at all.
Replace the "allowed on the number search type" clause with a back-link
to the prefix subsection — e.g.

> Finally, a [Search Prefix](#prefix) can be used on parameters whose
> search type supports comparison (see the prefix table for the
> applicable types and per-type semantics) to change the test to
> something other than equality — for example, a 'greater than'
> comparison.

This avoids duplicating the list of supported types in two places, so
the intro cannot drift out of sync with the normative table again if
the set of prefix-supporting types changes in a future version.

#### Justification

The root cause of the reported defect is duplicated information: the
intro paragraph repeats a fact (the set of prefix-supporting types)
that is authoritatively defined a few sections later. Fixing the wording
only (Disposition A) solves the symptom for 6.0 but leaves the same
maintenance hazard in place. A single source of truth (the prefix
table) plus a cross-reference is the more robust editorial pattern and
matches how other "applies to which types" facts are handled elsewhere
in `search.html`. The trade-off is a slightly less self-contained
sentence — readers must follow the link to see the full type list.

---

### Disposition C: Decline

#### Proposal

Reject the change on the grounds that the existing intro is illustrative
rather than exhaustive: it uses `number` only as an example to motivate
the concept of a comparison operator, and the authoritative list of
prefix-supporting types is given in the prefix subsection on the same
page. Close as "Not Persuasive — already addressed by existing
content".

#### Justification

The contested sentence sits in an introductory paragraph whose purpose
is to convey the *idea* of prefixes via a single concrete example
(`number` plus 'greater than'). The full normative list is one click
away in the same document, so an attentive reader is not actually
misled. Spec text frequently uses one search type as a representative
example without enumerating every type that supports a feature.

---

### Recommendation

**Recommended disposition:** A

The reported defect is real and low-risk: the intro sentence states a
set ("allowed on the number search type") that is narrower than the
set defined by the normative table on the same page. That is a
documentation bug, not stylistic preference, and the fix is mechanical.
Disposition C under-weights the harm — implementers who only read the
intro can reasonably conclude that `gt`/`lt`/`ge`/`le`/etc. are
illegal on `date` and `quantity`, which is incorrect. Disposition B is
editorially nicer, but it expands the scope from a one-sentence fix to
a small restructure that touches phrasing the reporter did not
complain about and risks ballot churn for an HL7 Infrastructure WG
review on a Low-priority ticket. Accepting as requested — explicitly
naming `number`, `date`, and `quantity` in the intro — fixes the
specific contradiction the reporter identified, keeps the change
surface to a single sentence in `search.html` §3.2.1.5.1, and leaves
the door open to apply Disposition B's editorial cleanup later if the
WG wishes.
