## Summary

The reporter flags a small grammatical typo on the R6 HTTP page, in the
"Transactional Integrity" section. The published sentence reads:

> "A server MAY to choose to maintain the information that would be lost, but
> there is no defined way for a server to determine whether the client
> omitted the information because it wasn't supported (perhaps in this case)
> or whether it wishes to delete the information."

The phrase "MAY to choose" is ungrammatical — either "MAY choose" or "MAY
opt to choose" is correct. The reporter is asking for a copy-edit to fix
the wording. The substantive normative content ("MAY") and the surrounding
guidance about a server's discretion to retain information are not
disputed; only the editorial slip is in scope. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were returned for this issue, so
the change is a self-contained editorial correction.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a one-word editorial fix in the R6 HTTP RESTful API page,
"Transactional Integrity" section. Change:

> "A server MAY **to** choose to maintain the information that would be
> lost..."

to:

> "A server MAY choose to maintain the information that would be lost..."

The edit removes the stray "to" and leaves the normative force of "MAY"
unchanged. No semantic, conformance, or interoperability impact — this is
a pure copy edit on the FHIR Core specification source for R6.

#### Justification

The reporter has correctly identified a grammatical error introduced into
the prose. "MAY" is an RFC 2119 / RFC 8174 conformance verb and combines
directly with a bare infinitive ("MAY choose"); "MAY to choose" is
ungrammatical and never the intended phrasing in HL7 specifications.
Because the surrounding sentence already explains *why* a server might
preserve the information, no rewording beyond removing "to" is needed.
Technical Correction is the appropriate ticket type for this class of
fix, and accepting it costs essentially nothing while removing a small
distraction from a normative paragraph.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying fix but take the opportunity to lightly reword the
whole bullet for readability while keeping the normative meaning. For
example:

> "A server MAY retain information that would otherwise be lost. However,
> there is no defined way for a server to determine whether the client
> omitted the information because the client did not support it (as may
> be the case here) or because the client wishes to delete it."

This still removes "MAY to choose" but also breaks the run-on sentence
into two and tightens the parenthetical aside.

#### Justification

The current bullet is long, parenthetical-heavy, and reads awkwardly even
once the typo is removed. A modest rewrite improves comprehension for
implementers without changing conformance. The trade-off is that any
rewording — even editorial — invites re-review by the workgroup and
slightly enlarges the diff vs. the surgical Disposition A change. If the
WG wants to minimise re-review surface, prefer A; if the WG is willing to
spend a small amount of editorial review time, B leaves the spec
measurably clearer.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Not Persuasive" / "Not Related" on the grounds that
this is a near-trivial editorial defect that can be folded into a future
batch of editorial cleanups rather than tracked individually, or
alternatively close as a duplicate if a broader editorial-pass ticket
already covers the same paragraph.

#### Justification

Declining is only defensible if there is an existing editorial-cleanup
ticket against the R6 HTTP page that will sweep this paragraph anyway, or
if the workgroup has an explicit policy of bundling sub-sentence typos.
Otherwise, declining a clearly-correct typo report sends a poor signal to
the reporter (the issue is real and the fix is one character pair) and
leaves a normative-flavoured sentence visibly broken in the published R6
build. There is no design or interoperability risk that would justify
declining on the merits.

---

### Recommendation

**Recommended disposition:** A

This is the textbook case the "Technical Correction" type exists for: a
clear, non-substantive editorial defect in a normative paragraph, with no
linked tickets, no Zulip discussion, and no GitHub work surfaced by the
cross-reference index. Disposition A applies the minimum diff that
removes the typo, carries zero conformance risk, and is fast to review.
Disposition B is a reasonable secondary if the workgroup wants to clean
up the surrounding sentence at the same time, but it should be treated
as an *additive* improvement on top of A rather than a reason to delay.
Disposition C is hard to defend in the absence of a known broader
editorial sweep that already covers the paragraph.
