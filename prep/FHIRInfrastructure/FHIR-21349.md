## Summary

The submitter (Bas van den Heuvel) asks whether a `Questionnaire.item` can
include a *reference* to another item — drawn either from the current
Questionnaire or from a different Questionnaire — so it is "inserted" at
that position. The motivation is the population of resources that contain
recursive / iterative elements such as `Questionnaire.item.item`,
`PlanDefinition.action.action`, and `RequestGroup.action.action`. Because
those structures nest themselves, a generator that derives a Questionnaire
from such a resource/profile needs a way to express recursion in the
resulting form. Lloyd McKenzie's comments on the ticket frame the question
as: "If we're generating a Questionnaire from a resource or profile, how
do we handle situations where the resource/profile has a recursive element
(e.g. PlanDefinition)?" and suggest the recursive element could be modeled
as a *child Questionnaire* with a pointer back to a Questionnaire of the
same type.

The ticket has been around since 2019. It was previously dispositioned
*Considered for Future Use* (vote 10-0-0, 2020-04-20) on the rationale
that Grahame was planning to generate questionnaires from profiles as
part of the IG publisher and the WG would revisit. lynn_laakso re-opened
it post-STU4 publication on 2026-03-26 and applied `Ready-For-Vote`, so
it is now back in front of FHIR Infrastructure for fresh disposition.

There is one cross-reference of any kind — a Zulip block-vote roster
posting — and no GitHub artifacts touch the ticket.

### Disposition A: Accept As Requested

#### Proposal

Define an explicit "item reference" mechanism on `Questionnaire.item`
in the SDC IG that allows one item to be replaced (or augmented) at
runtime by the contents of another item — drawn either from the same
Questionnaire (by `linkId`) or from a different Questionnaire (by
canonical URL plus `linkId`). Concretely, add a new SDC extension —
e.g. `http://hl7.org/fhir/uc/sdc/StructureDefinition/sdc-questionnaire-itemReference` —
of type `Reference(Questionnaire) | uri | code` whose value carries:

1. an optional `canonical` to the target Questionnaire (default: the
   current Questionnaire),
2. a required `linkId` of the target item, and
3. an optional cardinality marker so a population engine knows whether
   to inline the item once or once per occurrence of an iterative
   element (`Questionnaire.item.item`, `PlanDefinition.action.action`,
   `RequestGroup.action.action`).

The SDC narrative (form-behavior page and population page) gains a
section "Recursive and shared items" that specifies how a populating
client must resolve and inline the referenced item before continuing
population, and how an authoring tool / IG-publisher questionnaire
generator must emit such references when it encounters a recursive
element in the source profile.

#### Justification

This is the literal answer to the ticket's question. It gives
implementers a single, named mechanism for both intra-questionnaire
reuse and cross-questionnaire reuse, and it gives the IG-publisher
questionnaire generator (the tool Lloyd referenced in the 2020 vote
discussion and which lynn_laakso flagged in re-opening the ticket) a
deterministic emission target for recursive profile elements. Because
the mechanism is an extension on `Questionnaire.item`, it does not
break existing Questionnaire instances and can be adopted incrementally.

### Disposition B: Alternative Approach

#### Proposal

Do **not** invent a new "item reference" extension. Instead:

1. Document, in the SDC IG, that the existing
   `http://hl7.org/fhir/uc/sdc/StructureDefinition/sdc-questionnaire-subQuestionnaire`
   extension is the supported way to "insert" the items of another
   Questionnaire at a given position — exactly Lloyd's 2019-06-27
   suggestion of "treat the repeating element as a child Questionnaire
   and have the Questionnaire point to a Questionnaire of the same type."
2. Add an explicit SDC narrative section on **recursive form
   structures** that:
   - Describes how to model a profile / resource recursive element
     (`Questionnaire.item.item`, `PlanDefinition.action.action`,
     `RequestGroup.action.action`) as a self-referential
     `subQuestionnaire`.
   - Sets a maximum-depth / loop-prevention obligation on the
     populating client, so a self-referential subQuestionnaire does not
     populate infinitely.
   - Updates the IG-publisher's profile-to-Questionnaire generator
     guidance to emit a `subQuestionnaire` reference (rather than a
     fully-expanded item) at recursion points.
3. If reuse of a *single* item (not a whole subtree) inside the same
   Questionnaire is also required, prefer aliasing via
   `Questionnaire.item.definition` pointing at the same
   `ElementDefinition` rather than introducing a second extension.

#### Justification

`sdc-questionnaire-subQuestionnaire` already exists and already covers
"items defined elsewhere are inserted here." The substantive gap that
FHIR-21349 actually exposes is that the IG does not currently *say* how
that extension should be applied to recursive structures, and that the
IG-publisher generator does not currently emit it for them. Closing
that documentation/tooling gap costs much less than introducing a new
extension and avoids two parallel mechanisms (subQuestionnaire vs.
itemReference) that implementers would have to choose between. It also
honours the rationale of the 2020 *Considered for Future Use* vote —
which was explicitly waiting on Grahame's profile-to-Questionnaire
work — by turning that work product into normative SDC guidance.

### Disposition C: Decline

#### Proposal

Reaffirm the 2020 *Considered for Future Use* disposition and close the
ticket as **Persuasive with Mod** of "no change required". Rationale
category: **already addressed by an existing mechanism**
(`sdc-questionnaire-subQuestionnaire`) plus **insufficient
implementer demand** — there are no GitHub issues, no PRs, no
substantive Zulip threads, and no other Jira tickets cross-referencing
FHIR-21349 in seven years.

#### Justification

The ticket is a question, not a defect. The mechanism it asks about
exists today (`subQuestionnaire`). The 2020 WG vote was 10-0-0 to
defer, and the only post-2020 activity is a procedural re-open by the
co-chair — not new use-case material. Adding a second extension or new
narrative section without implementer pull-through risks SDC churn and
two competing patterns. Declining keeps the SDC surface stable and
asks the requester to come back with a concrete failing implementation
or use case if `subQuestionnaire` does not in fact suffice.

---

### Recommendation

**Recommended disposition: B (Alternative Approach).**

Disposition C is defensible on procedural grounds but ignores the
substantive content of Lloyd McKenzie's 2019 comments, which are
effectively a yes-and-here-is-how design sketch — meaning the WG
already has a worked answer and just hasn't written it down. Disposition
A is more than is needed: it introduces a brand-new extension that
duplicates the semantics of `sdc-questionnaire-subQuestionnaire` and
forces a "which extension do I use?" decision on every SDC implementer.
Disposition B is the smallest change that actually closes the gap the
ticket exposes — recursive-structure population guidance and an
IG-publisher generator rule — by leveraging the extension that already
ships in SDC. It is also the disposition most consistent with the 2020
vote's rationale (wait for Grahame's profile-to-Questionnaire work,
then formalize). The workgroup should still confirm two open
sub-questions before voting: (1) the depth / cycle-detection
obligation on populating clients, and (2) whether single-item reuse
needs `definition` aliasing called out alongside the `subQuestionnaire`
guidance.
