## Summary

FHIR-53788 is a single-line technical-correction ticket reporting that the
word "processsing" (with three s's) is misspelled in the FHIR Core
specification. The ticket itself does not name the affected page or file;
it is reported against FHIR Core (FHIR) under the FHIR Infrastructure
work group. Cross-reference lookups against the augury indexes returned
no linked Jira tickets, Zulip threads, GitHub PRs, or commits for this
key. However, a content search across Jira surfaces two clearly related
items: FHIR-55172 ("Spelling/abbreviation issues on page: list"), filed
two weeks later by Lloyd McKenzie against the same FHIR Core `list`
page and explicitly listing `"processsing"` as one of the misspellings,
and FHIR-41588 ("Typo in Search Parameter Processsing Mode Valueset
Description"), an older Persuasive technical correction in the same
work group whose resolution note shows the FHIR Infrastructure WG has
historically accepted and fixed this exact typo class. A GitHub code
search confirms the only remaining occurrence of `processsing` in
HL7/fhir lives in `source/list/list-notes.xml`, which is consistent
with FHIR-55172's "page: list" framing. FHIR-53788 is therefore
effectively a narrower duplicate of FHIR-55172, scoped to just the
spelling typo and excluding FHIR-55172's contraction-expansion
requests.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-53788 as a non-substantive technical correction: change the
single occurrence of `processsing` to `processing` in the FHIR Core
narrative for the `list` resource (the only remaining occurrence in
HL7/fhir, per the code search above is in `source/list/list-notes.xml`).
Confirm by grep that no other `processsing` occurrences remain in the
FHIR Core authoring source before closing. Resolve as **Persuasive –
Non-substantive** and link to FHIR-55172 so the spelling portion of
that ticket can be marked as already addressed.

#### Justification

The typo is unambiguous, has zero semantic impact, and the FHIR
Infrastructure WG has already accepted essentially the same correction
on FHIR-41588 (Persuasive) and FHIR-23492 (Published). Coordinating
with FHIR-55172 avoids two independent edits to the same page during
ballot prep and keeps the audit trail consistent with how FMG normally
handles spelling fixes (single Persuasive disposition, single commit).

---

### Disposition B: Alternative Approach

#### Proposal

Resolve FHIR-53788 as a **duplicate of FHIR-55172** without making any
edits under FHIR-53788's key. Do the actual `list` page cleanup
(`processsing` → `processing`, plus the contraction expansions
`doesn't` / `isn't` / `wasn't`) as a single coordinated change tracked
on FHIR-55172, which already names the affected page and bundles all
copy-edits the editor needs to make in one place.

#### Justification

FHIR-55172 is a strict superset of FHIR-53788's request (same page,
includes the same typo, plus other copy-edits), and consolidating
copy-edits into one ticket reduces ballot churn. The risk of editing
twice or producing two near-identical errata entries is real when two
near-duplicate tickets remain open against the same authoring file.
Using FHIR-55172 as the canonical ticket also picks up Lloyd
McKenzie's broader page-level review, which a narrow `processsing`
fix would otherwise miss.

---

### Disposition C: Decline

#### Proposal

Reject FHIR-53788 as **Not Persuasive** on the grounds that the
report is not actionable as filed: the ticket gives no page,
section, or context for the misspelled word, and the WG should
require submitters to identify *where* a typo lives before opening a
correction ticket. Direct the submitter to refile against the
specific page if the issue persists.

#### Justification

This stance is internally consistent with how some HL7 WGs handle
under-specified change requests, and it puts the burden of
localisation on the submitter rather than the editor. However, in
this case the missing context is trivially recoverable (one grep on
the source repo finds the single offending file), so this option is
weak on the merits and only listed for completeness.

---

### Recommendation

**Recommended disposition:** B

FHIR-55172 was filed two weeks after FHIR-53788 by Lloyd McKenzie, a
core FHIR Infrastructure participant, and it explicitly identifies the
`list` page, lists `processsing` as one of several issues on that
page, and is already prioritised Highest. Resolving FHIR-53788 as a
duplicate of FHIR-55172 is the cleanest path: it preserves the
correction (so the submitter's concern is addressed), avoids two
editors touching `source/list/list-notes.xml` for the same paragraph,
and lets the WG handle all of that page's copy-edits in one
ballot-prep commit. Disposition A is a perfectly defensible fallback
if the WG would rather close FHIR-53788 with the fix attached and
trim the spelling bullet out of FHIR-55172. Disposition C should be
avoided — declining a one-character spelling fix where the location
is recoverable in seconds is hard to justify against the WG's
existing Persuasive precedent on FHIR-41588 and FHIR-23492.
