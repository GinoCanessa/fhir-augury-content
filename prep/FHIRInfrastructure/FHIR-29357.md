## Summary

The reporter asks how a client connecting to a SMART Web Messaging (SMW) host
discovers which messaging events the host actually supports. SMW defines a set
of optional event types (e.g., `ui.done`, `scratchpad.*`, `status.*`), but at
the time the ticket was filed the specification offered no machine-readable
way for an embedded app to know in advance which of those events the host will
honor. The ticket originated as ballot feedback against the SMW STU1 ballot
and was deferred during 2021-02-19 ballot reconciliation (vote 8-0-0,
Gino Canessa / Bas van den Heuvel) under the resolution "Considered for
Future Use", with the work group agreeing that the new handshake protocol
already provides per-session, contextually-aware discovery and that more
formal discovery should wait for implementation experience after STU1. The
ballot disposition recorded two concrete future actions: (1) add a forward
note to the handshake documentation indicating discovery will be expanded in
a later release, and (2) define a `smart-web-messaging` capability value that
servers MAY include in the `capabilities[]` array of their
`.well-known/smart-configuration` document. Lynn Laakso reverted the
"Considered for Future Use" resolution on 2022-05-06 with the note
"re-open after publication of STU1", returning the ticket to Triaged so that
SMW STU2 work can address it. No cross-referenced Jira tickets, Zulip threads,
or GitHub items were surfaced for this ticket in FHIR Augury.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Specify a discovery mechanism in SMART Web Messaging STU2 that lets a
launched app enumerate the SMW event types the host supports. Concretely:

1. Add a new `smwCapabilities` (or equivalently named) object to the SMW
   handshake response. The object would contain a single required member,
   `supportedMessageTypes`, whose value is an array of message-type strings
   drawn from the SMW message-type registry (e.g., `["ui.done",
   "scratchpad.create", "scratchpad.update", "status.*"]`). Hosts MAY also
   advertise per-message-type sub-capabilities (e.g., `{"messageType":
   "scratchpad.update", "supportedTargetTypes": ["DocumentReference",
   "Composition"]}`) where granularity is meaningful.
2. Define a SMW conformance rule that clients SHALL NOT send a message whose
   `messageType` is absent from `supportedMessageTypes`, and that hosts SHALL
   respond to such a message with the existing `messaging.handshake` /
   error-response pattern.
3. Register the discovery object in the SMW StructureDefinition (or its
   prose equivalent) and add normative narrative + a worked example to the
   handshake page.
4. Carry forward the two ballot-reconciliation actions already recorded on
   this ticket: register `smart-web-messaging` in the `capabilities[]` array
   of `.well-known/smart-configuration` (so launchers can discover *whether*
   a server speaks SMW at all, before the handshake), and remove the
   "future release" forward note added in STU1 once the in-handshake
   discovery is normative.

#### Justification

This is the literal request: a client today has no machine-readable way to
know which event types the host honors, and the work group already
acknowledged in 2021-02-19 reconciliation that richer discovery should land
"in a future SMW release, when we have some implementation experience to
build on". STU1 is now published, so the deferral condition has been met,
and Lynn Laakso explicitly re-opened the ticket on 2022-05-06 to drive STU2
work. Putting the catalog inside the existing handshake response keeps
discovery contextual (per-session, per-launch, post-authorization) — which
was the work group's stated reason for preferring the handshake over a
static manifest in the first place — while still answering the reporter's
question deterministically. Reusing the SMW message-type registry ensures
the advertised values are the same identifiers the wire protocol already
uses, avoiding a parallel taxonomy.

---

### Disposition B: Alternative Approach

#### Proposal

Address discovery in two layers and avoid extending the handshake payload:

1. **Coarse, pre-launch discovery:** Adopt only ballot-reconciliation
   action #2 — register a `smart-web-messaging` capability token in the
   SMART App Launch capabilities list so that hosts MAY advertise SMW
   support in their `.well-known/smart-configuration`. This tells a client
   that SMW is available before launch, which is the part of the question
   that the handshake cannot answer.
2. **Fine-grained, runtime discovery:** Standardize on a `messaging.ping`
   (or `capabilities.query`) SMW message that any client MAY send
   immediately after the handshake. The host's response carries the same
   `supportedMessageTypes` array described in Disposition A. Clients that
   do not need full discovery (most production launches will hard-code the
   message types they care about) pay no extra payload cost on the
   handshake; clients that do need it issue one extra round-trip.

#### Justification

This split honors the work group's original concern — "We don't want to
over-specify at this stage" — by leaving the handshake response stable for
existing implementers and confining the new surface area to (a) one capability
token already accepted in principle by the 2021-02-19 vote, and (b) one new
optional message type that follows the existing SMW request/response shape.
It also preserves backward compatibility: hosts that don't implement
`messaging.ping` simply return the existing "unsupported messageType" error,
which clients already have to handle. The cost is one extra round-trip for
discovery-aware clients and a slightly less obvious "where do I look?" story
for implementers, since the answer is split across the SMART configuration
document and an SMW message.

---

### Disposition C: Decline

#### Proposal

Close as "Persuasive with Mod" / "Considered for Future Use" again, with the
single concrete change being ballot-reconciliation action #1: amend the SMW
handshake page to state that detailed discovery is intentionally out of
scope for the current release and will be standardized when implementation
experience justifies a specific shape. Take no normative action on the
handshake payload or the `.well-known/smart-configuration` capabilities
list, and ask the reporter to bring back a concrete proposal once they (or
another implementer) have built against SMW STU1 and can describe a
discovery shape they actually need.

#### Justification

The work group already declined the request in essentially this form on
2021-02-19 (8-0-0). The re-open note — "re-open after publication of STU1" —
records a procedural gate, not new substantive evidence that the original
reasoning has changed. The handshake protocol still provides per-session,
contextually-aware capability negotiation, and the SMW error-response model
already lets a client probe support by attempting a message and handling a
documented failure. There is no Zulip discussion, no GitHub PR, and no
linked Jira ticket showing implementer demand for a richer discovery
mechanism, and no recorded use case where probe-and-handle is unworkable.
Standardizing a discovery shape now risks codifying the wrong abstraction;
deferring keeps SMW's published surface stable while leaving the door open.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best fit because it directly executes the two concrete
actions the work group already agreed to in the 2021-02-19 ballot
reconciliation — register a `smart-web-messaging` capability token in
`.well-known/smart-configuration`, and provide a richer in-band discovery
mechanism — without committing STU2 to embedding a potentially over-grown
capability object in the handshake payload that every host would have to
maintain. It answers the reporter's literal question ("how does a client
know what events are supported by the host?") in two complementary places:
pre-launch (`.well-known/smart-configuration`) and post-handshake
(`messaging.ping` / `capabilities.query`), which matches how SMART App Launch
and SMW already partition discovery responsibility. Disposition A goes
further but risks the over-specification the work group warned against in
2021, especially since there is no implementer evidence on file showing
the granularity of discovery they actually need. Disposition C is
defensible procedurally but ignores the fact that STU1 has now shipped and
the reporter (and Lynn Laakso, on behalf of the work group) explicitly
re-opened the ticket to drive STU2 work — the deferral condition has been
discharged. If the work group wants to start narrow, B can be staged: ship
the `.well-known/smart-configuration` capability token in a STU2 ballot
first and iterate on the `messaging.ping` shape based on STU2 implementer
feedback.
