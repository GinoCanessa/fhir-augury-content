## Summary

The ticket asks the FHIR Core specification to clarify that the FHIRPath
expression supplied in `ElementDefinition.slicing.discriminator.path` must be
evaluated **relative to the sliced element**, and must not be written as an
absolute path that starts at the resource root or any other upstream node.
Concretely, for slicing defined on `Patient.identifier`, expressions like
`Patient.identifier.type` (absolute) should be disallowed, while `type` (and,
by implication, `$this.type`) is the correct relative form. The request was
prompted by, and is consistent with, the chat.fhir.org discussion
"[conformance] Absolute resource paths as discriminator path", in which Grahame
Grieve and Lloyd McKenzie confirmed that the path must be relative and asked
the reporter to file a change request so that the spec says so explicitly.
There are no cross-referenced Jira tickets and no related GitHub items.

## Details

**Description:**

> Please clarify that the FHIRPath expression in the
> `ElemenentDefinition.slicing.discriminator.path` element should be a path
> relative to the sliced element and not start at any other node (such as the
> resource root).
>
> The specification should state that expressions of pattern
>
> ```
> Path.to.sliced.element.path.to.discriminatingValue
> ```
>
> are not allowed while
>
> ```
> $this.path.to.discriminatingValue
> ```
>
> or
>
> ```
> path.to.discriminatingValue
> ```
>
> are. Or rather that the provided expression is evaluated from the sliced
> element.

## Related Zulip Discussions

### [conformance > Absolute resource paths as discriminator path](https://chat.fhir.org/#narrow/stream/conformance/topic/Absolute%20resource%20paths%20as%20discriminator%20path/near/541969816)

Paul Behrend asked whether it is valid in FHIR R4 to use absolute resource
paths (e.g. `ResourceType.element1.[...].slicedElement.path.to.value`) as
values of `ElementDefinition.discriminator.path`, rather than forms relative
to the sliced element such as `$this.path.to.discriminatedValue`. Lloyd
McKenzie answered that it must be relative and pinged Grahame Grieve for
confirmation. After Lloyd clarified the question with a concrete
`Patient.identifier` example, Grahame confirmed "has to be relative". Lloyd
then asked Paul to file a change request to make that explicit in the spec,
which Paul agreed to — this ticket (FHIR-52999) is the resulting request.
Consensus in the thread is unambiguous: discriminator paths are relative; the
spec should say so.

Participants: Paul Behrend, Lloyd McKenzie, Grahame Grieve.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the narrative description of `ElementDefinition.slicing.discriminator.path`
in the FHIR Core specification (the element's definition/comments in the
`ElementDefinition` resource page and the "Slicing" / "Profiling" narrative
pages) to state explicitly that:

1. The value is a FHIRPath expression that is **evaluated in the context of the
   sliced element** — i.e. it is relative to the element identified by
   `ElementDefinition.path` on which `slicing` is declared, not to the
   containing resource.
2. Expressions that begin with the resource type or any ancestor path (e.g.
   `Patient.identifier.type` when slicing `Patient.identifier`) are **not
   permitted**.
3. Both a bare relative path (e.g. `type`, `value.ofType(CodeableConcept)`) and
   a `$this`-rooted path (e.g. `$this.type`) are permitted and equivalent.

Add one or two worked examples (e.g. slicing `Patient.identifier` by `type`)
showing a correct and an incorrect form. Optionally tighten the invariant(s)
on `ElementDefinition` so that validators can flag absolute paths, but at
minimum the narrative must be unambiguous.

#### Justification

The Zulip thread shows that both the chief editor (Grahame Grieve) and a lead
methodologist (Lloyd McKenzie) agree the path must be relative, and that the
current spec text does not make this clear enough to answer an implementer's
direct question. The change is purely clarifying — it does not alter any
intended behavior, does not break any correctly-authored profile, and aligns
the text with how the published FHIR tooling (e.g. the validator, Forge,
Sushi/FSH) already treats the field. Consistency with how other FHIRPath-
valued elements are described (e.g. `ElementDefinition.condition`,
`invariant.expression` evaluating in the element's own context) is also
improved.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to the narrative clarification in Disposition A, **add a normative
constraint (FHIRPath invariant) on `ElementDefinition`** that enforces the
rule mechanically. For example, an invariant on `ElementDefinition.slicing`
of the form:

> `discriminator.all(path = '$this' or not(path.startsWith(%context.path & '.')) and not(path.matches('^[A-Z][A-Za-z0-9]*\\..*')))`

(the exact FHIRPath would need refinement, but the intent is: the
discriminator path must not start with a resource-type token and must not
re-state the sliced element's path). Pair this with the narrative update and
worked examples from Disposition A.

#### Justification

A pure narrative change (Disposition A) relies on implementers reading the
prose; an invariant also lets the validator flag non-conformant profiles
automatically, which is how other structural rules on `ElementDefinition` are
enforced. The trade-off is authoring cost (writing a correct, robust
FHIRPath invariant that doesn't misfire on legitimate expressions using
`.where(...)`, `.ofType(...)`, `resolve()`, etc.) and the risk that
previously-published IGs containing incorrect absolute paths will start
failing validation. That last point is a feature, not a bug — such profiles
were already broken by the intended semantics — but it does have an
ecosystem cost that needs to be weighed.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — already adequately specified", on the
grounds that FHIRPath expressions on `ElementDefinition` children (invariants,
conditions, discriminators) are by convention evaluated in the context of the
element on which they are declared, and that the existing slicing narrative
plus the worked examples in the spec already demonstrate relative paths.

#### Justification

This is the weakest option because the Zulip discussion itself demonstrates
that the current wording is not clear enough: an implementer read the spec,
couldn't tell, and asked — and a spec editor agreed the text should be made
explicit. Declining would leave a documented ambiguity in place for no
corresponding benefit. This disposition is included for completeness only.

---

### Recommendation

**Recommended disposition:** A

Disposition A (accept as requested) is the right starting point: it directly
resolves the ambiguity that prompted the ticket, it is what Grahame and Lloyd
endorsed on Zulip, and it is a pure-narrative change with no backward-
compatibility risk. Disposition B is attractive if the workgroup wants the
fix to be machine-enforceable, but the invariant is non-trivial to get right
and will fail previously-published profiles that embedded absolute paths,
which is a separate policy call better made once the normative clarification
in A is in place. Declining (C) is not defensible given that the spec editor
explicitly requested this ticket be filed. The workgroup should accept A now
and, if desired, open a follow-up ticket to add the validator-level invariant
as a discrete piece of work.
