## Summary

The ticket asks the FHIR Infrastructure work group to clarify a special-case
rule baked into the FHIRPath Equivalence (`~`) operator as it is applied to
FHIR data: the rule that `id` elements are ignored when comparing two
elements for equivalence. Brian Postlethwaite (reporter) flags that, while
ignoring `Element.id` (the local XML/JSON id used for narrative anchors and
contained references) is reasonable for most FHIR types, `ElementDefinition.id`
carries semantically significant meaning (it encodes the slice/path of an
element inside a StructureDefinition's differential/snapshot and is the basis
for resolving references such as `#...` paths in slicing and constraint
authoring). Treating two `ElementDefinition`s as `~`-equivalent while ignoring
`id` would silently equate elements that are not actually the same, which is
almost certainly not the intended behaviour.

The ticket also asks for an explicit note that the "ignore `id`" behaviour is
a FHIR-type-aware extension to the FHIRPath operator (i.e. it is part of the
FHIR profile of FHIRPath, not the intrinsic FHIRPath operator). If the
evaluation engine does not know it is operating on FHIR-typed data, this
special rule does not apply.

There are **no cross-referenced** Jira tickets, Zulip threads, or GitHub items
recorded for this ticket. Topically related (but not formally linked) work
includes [FHIR-55473](https://jira.hl7.org/browse/FHIR-55473) ("FHIRPath
String Equivalence Ambiguities", same reporter community, same operator,

("FHIRPath and CQL string equivalence not fully aligned", the historical
ticket that introduced the "normalizing whitespace" wording). These are noted

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR specification's description of how the FHIRPath `~`
(Equivalence) operator behaves on FHIR-typed data (the page documenting
"Using FHIRPath" / FHIR's profile of FHIRPath, where the "ignore `id`
elements" rule lives) to:

1. **Carve out `ElementDefinition`** from the "ignore `id`" rule: when
   comparing two `ElementDefinition` instances with `~`, `ElementDefinition.id`
   **is** considered, because the id encodes the element's path/slice
   identity inside the StructureDefinition and is not a discardable local
   anchor. State this as a narrow, named exception alongside the general
   rule for `Element.id`.
2. **Add a clarifying note** that the "ignore `id`" behaviour is part of
   FHIR's profile of FHIRPath and is therefore conditional on the evaluation
   engine knowing the operand is a FHIR type. If the engine is operating on
   untyped data (or non-FHIR-typed data), it must fall back to the intrinsic
   FHIRPath `~` semantics; nothing in this clarification changes the base
   FHIRPath operator definition itself.

#### Justification

The carve-out is consistent with how the spec already distinguishes
"plumbing" identifiers (narrative/anchor `id`s on ordinary elements) from
identifiers with semantic meaning. `ElementDefinition.id` is special: it is
how slicing, differentials, snapshots and `#...` references inside a
StructureDefinition are resolved, and two ElementDefinitions that differ
only in `id` are emphatically not the same element. Silently equating them
under `~` would produce wrong answers for tooling that uses `~` to compare
structure definitions (profile diffing, conformance checks). The
"FHIR-type-aware" note mirrors the framing already used for FHIR's other
extensions to FHIRPath (e.g. resolve(), the FHIR-specific implicit
conversions) and matches the style used in the recent FHIRPath clarification
[FHIR-55473](https://jira.hl7.org/browse/FHIR-55473), where the work group
preferred a narrow, explicitly-scoped extension over a silent broadening.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than carving out `ElementDefinition` as a one-off exception, restate
the rule in terms of *intent*: the FHIR profile of FHIRPath ignores `id`
elements that are **purely local anchors** (i.e. `Element.id` used for
narrative cross-references and `contained` resolution) but **does not**
ignore `id` properties whose definition gives them semantic identity inside
the resource. Then enumerate the known semantic-identity `id` properties
(at minimum `ElementDefinition.id`; reviewers should also confirm whether
`Bundle.entry`'s fullUrl-keyed identity, `Questionnaire.item.linkId`-style
identifiers, or any other `*.id`-typed paths warrant the same treatment).
Pair this with the same FHIR-type-aware note proposed in Disposition A.

#### Justification

This avoids the brittleness of having to re-open the spec every time someone
discovers another resource where `id` actually matters semantically — a
real risk given that `id` appears on every Element. It keeps the rule
principled rather than a list of exceptions, which is consistent with how
FHIRPath generally prefers semantically-grounded definitions. The trade-off
versus Disposition A is more upfront editorial work (the work group must
decide and document which `id`s are "semantic"), and the resulting wording
is harder for an implementer to mechanically check than a flat "ignore
`Element.id` except for `ElementDefinition.id`" rule. Existing tools that
already implement the strict literal "ignore all `id`" rule would also need
broader audit work than under Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave the current "ignore `id` elements" wording in
place. Optionally add a single sentence noting that the rule is part of
FHIR's profile of FHIRPath (the type-aware clarification), but do not
introduce any carve-out for `ElementDefinition.id`.

#### Justification

A defensible decline rests on the position that FHIRPath `~` is an
*equivalence* operator for *value comparison* of element trees and was never
intended as a semantic identity check between StructureDefinition elements;
profile-diffing and conformance tooling should be using purpose-built
comparators, not `~`. Under this reading, the "significant meaning" of
`ElementDefinition.id` is a concern of the StructureDefinition processing
model, not of the generic `~` operator, and overloading `~` with
type-specific exceptions risks fragmenting its semantics over time. The
weakness of this position is that it leaves the spec literally saying that
two ElementDefinitions differing only in `id` are equivalent, which most
implementers would read as a bug rather than a feature.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest fit for what the reporter actually asked, and
it matches the work group's recent practice on the same operator
([FHIR-55473](https://jira.hl7.org/browse/FHIR-55473)) of making narrow,
explicit, FHIR-typed clarifications to FHIRPath behaviour rather than either
re-architecting the rule (Disposition B) or refusing to clarify
(Disposition C). The carve-out for `ElementDefinition.id` is narrow,
demonstrably correct (slicing and snapshot resolution provably depend on
it), and easy for implementers to encode as a one-line exception. The
type-aware note is essentially free editorial work and prevents downstream
confusion about whether non-FHIR FHIRPath engines need to implement the
exception. If the work group discovers further `id` properties that warrant
the same treatment during review, those can be folded in as additional
named exceptions without changing the framing — effectively a controlled
on-ramp to Disposition B, should a future ticket justify it.
