## Summary

Isaac Vetter is asking the SMART on FHIR specification to drop the
requirement that authorization servers support the "JWKS Set directly"
client-key registration method. Today SMART mandates that servers support
two duplicative ways for a client to publish its public key — by URL
(`jku` / `jwks_uri`) and by direct in-band JWKS at registration — while
the spec itself notes that the URL-based mechanism is the superior
approach. The reporter argues that the original justification (clients
that cannot host a public endpoint, e.g., mobile / SPA / firewalled
backend) no longer holds: such clients also cannot protect a client
secret or private key, and SMART's adjacent `x5c` (X.509-based) flow,
already required by HTI-2 / TEFCA, covers the same niche without
requiring a public endpoint. The request is to remove the server
"SHALL support JWKS Set directly" requirement so that servers may push
clients toward `jwks_uri`. No cross-referenced Jira tickets were
returned for this ticket.

## Details

**Description:**

Currently SMART mandates servers to support two duplicative JWKS
registration methods, while also explaining why one of those two methods
("JWKS Set directly") is inferior to the other.

Please remove the requirement that servers must support the inferior
method ("JWKS Set directly"). While the ability for servers in the US
now have wide support for this capability (due to this requirement
being tested by Inferno for HTI-1 certification of SMART v2.0), servers
should be enabled to push clients to use the superior method (JKU).

One argument against removing the JWKS-Set-Directly requirement is to
enable clients whose architecture prohibits hosting a publicly
accessible endpoint, such as mobile apps and SPAs.

1. Apps that can't host a public endpoint are also generally unable to
   protect a secret, such as the private key corresponding to the public
   key in the JWKS.
2. Servers are also being mandated to support `x5c` with X.509 certs as
   part of HTI-2 and TEFCA requirements, which is largely duplicative of
   both the SMART JKU and JWKS-Set-Directly requirements; however, the
   `x5c` mechanism does not require the app to have an accessible
   endpoint.

Please remove the requirement for servers to support the duplicative
(triplicative?) JWKS-Set-Directly registration capability.

Isaac Vetter (the same reporter) opened this thread in 2020 questioning
whether SMART truly requires servers to support both `jwks_uri` and the
in-band JWK Set, noting the spec text places its SHALL language on
clients rather than on servers. Josh Mandel and Dan Gottlieb (SMART
editors) confirmed the intent was that servers MUST support both
methods, with `jwks_uri` preferred and direct-JWKS retained as a
"less-common special case" for clients that cannot host keys online
(e.g., apps running entirely behind a firewall). Yunwei Wang agreed
with that interpretation. Isaac pushed back that the normative text
does not actually say so and that the use case for non-publicly-hosted
keys is weak; Dan acknowledged the Bulk Data spec might need to be
tightened. The thread closed with Josh pointing to clarifying PR
[HL7/bulk-data#79](https://github.com/HL7/bulk-data/pull/79). The
discussion is the direct historical antecedent of this ticket: the
2024 ticket asks the workgroup to revisit the same trade-off now that
HTI-1 has forced wide server support and HTI-2 / TEFCA add the `x5c`
alternative.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the SMART App Launch / SMART Backend Services IG client-authentication
section ("Asymmetric ('private key JWT') authentication"), change the
server conformance for the JWK Set publication mechanism so that:

- Servers **SHALL** support the `jku`-based mechanism (resolving the
  client's `jwks_uri` to fetch the JWK Set over TLS).
- Servers **SHOULD** (was SHALL) support accepting a JWK Set provided
  directly at client registration time.
- Clients that cannot host a public `jwks_uri` are directed to the
  asymmetric `x5c` profile (X.509-based JWS header), which already
  covers the offline / firewalled-client niche under HTI-2 and TEFCA.
- Add a non-breaking note for clients: clients SHOULD prefer
  `jwks_uri`; if they cannot host one, they SHOULD use `x5c` rather
  than direct JWK Set publication.
- Update the registration / dynamic-registration narrative to remove
  language that implies a server MUST accept both forms, and update
  CapabilityStatement / discovery guidance accordingly (the
  `.well-known/smart-configuration` capabilities list around
  `client-public` / `client-confidential-asymmetric`).

This is a normative change targeted at the next SMART App Launch
release; no FHIR-core resource changes are implied.

#### Justification

The spec itself characterises direct-JWKS as inferior to `jku`, which
makes a server SHALL on the inferior path hard to defend on FHIR's own
"prefer the better mechanism" design principle. The historical
justification for keeping direct-JWKS — clients that cannot host a
public endpoint — is now demonstrably handled by the `x5c` mechanism
that HTI-2 and TEFCA already mandate, so removing the SHALL does not
strand any real client population. The Zulip thread shows the SMART
editors themselves treated direct-JWKS as a "less-common special case",
and the reporter has been consistent for four years that the normative
text is over-strong for the use case. Demoting SHALL → SHOULD preserves
the option for servers that wish to keep accepting direct-JWKS (e.g.,
to remain backward-compatible with HTI-1 Inferno suites) without
forcing it on every server going forward.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both mechanisms in the IG but split the conformance along a
capability axis instead of removing it outright:

- Define two named server capabilities (already partially present via
  `client-confidential-asymmetric`): one indicating support for
  `jwks_uri`-based key publication and one indicating support for
  direct-JWKS at registration.
- Require servers to advertise at least the `jwks_uri` capability
  (SHALL); the direct-JWKS capability becomes an advertised, optional
  capability (MAY).
- Require clients to inspect the server's
  `.well-known/smart-configuration` and choose a supported mechanism;
  if neither `jwks_uri` nor direct-JWKS is acceptable to the client,
  fall back to the asymmetric `x5c` path.
- Add explicit guidance for HTI-1-certified servers that want to keep
  accepting direct-JWKS for backward compatibility — they simply
  continue to advertise the optional capability.

#### Justification

This addresses the same underlying request — servers should not be
forced to maintain an inferior mechanism — but does so without
introducing an implicit breaking change for downstream regulations
(HTI-1 Inferno suites currently test direct-JWKS support). Because the
capability is advertised in `.well-known/smart-configuration`, clients
gain a deterministic way to discover what a given server accepts,
which is a cleaner long-term design than relying on universal SHALLs.
It also avoids the awkwardness of US-Realm conformance (HTI-1) being
stricter than the base SMART IG, since both regimes can coexist on the
same capability surface. The cost is more discovery surface area and
slightly more client logic.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" and retain the current SHALL on
direct-JWKS support. Optionally add a non-normative note clarifying
that `jwks_uri` is the preferred mechanism and pointing implementers
at `x5c` for clients that cannot host a public endpoint.

#### Justification

A defensible decline rests on three points. First, the SHALL on
direct-JWKS was deliberately set by the SMART editors (Josh Mandel,
Dan Gottlieb) in the Zulip thread cited above precisely so that
clients without a public endpoint always have a server-supported
fallback, and that population (firewalled backend services, certain
mobile architectures) has not disappeared. Second, HTI-1 certification
already requires every certified server to support direct-JWKS;
demoting it to SHOULD risks a divergence between US-Realm conformance
and base SMART, which causes more implementer confusion than the
status quo. Third, `x5c` is not a drop-in equivalent — it carries CA /
PKI assumptions that some deployments explicitly avoid — so treating
it as a complete substitute is over-stated. On this view the cost of
the duplicative requirement is low (servers have already implemented
it) and the cost of removing it is real interop churn.

---

### Recommendation

**Recommended disposition:** A

The reporter's analysis holds up: the SMART specification itself ranks
`jku` above direct-JWKS, the original "clients that can't host a
public endpoint" rationale is now covered by the `x5c` flow that
HTI-2 / TEFCA already require servers to support, and the cited Zulip
thread shows the SHALL was inferred from intent rather than written as
a clean normative requirement. Demoting the requirement to SHOULD
(Disposition A) gives servers room to deprecate the inferior path
without breaking any server that still wants to accept direct-JWKS for
HTI-1 backward-compatibility, and it avoids the discovery-surface
expansion that Disposition B would add. Disposition C is plausible if
the workgroup believes the HTI-1/US-Realm divergence cost outweighs
the design cleanup, but the reporter's `x5c` argument is strong
enough that the workgroup should be pushed to address it on the
record before defaulting to "no change."
