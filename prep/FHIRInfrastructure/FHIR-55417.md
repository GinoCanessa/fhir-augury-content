## Summary

Josh Mandel reports a documentation defect on the core FHIRPath page
(`https://build.fhir.org/fhirpath`): a worked example shown in an attached
screenshot is missing a `.where` invocation before the predicate `(id = 23)`,
so the snippet as written is not valid FHIRPath. The ticket asks for the
example text to be corrected so that the filter is expressed as
`...where(id = 23)` instead of `...(id = 23)`. The change is purely editorial
— it does not alter FHIRPath semantics, the FHIR specification's normative
content, or any resource definition. The ticket has already been triaged with
a "Persuasive" resolution noting "Update the example as noted." It was
included by Brian Postlethwaite in a block-vote candidate list of small
FHIRPath documentation issues on the FHIR Infrastructure Zulip channel.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the editorial fix exactly as the ticket requests: on the core FHIRPath
page (`build.fhir.org/fhirpath`), locate the example where the screenshot
shows `...(id = 23)` immediately following a collection-valued navigation
step, and insert the missing `.where` so the example reads
`...where(id = 23)`. Confirm the surrounding paragraph and any narrative that
references the example still flow correctly after the edit. No
schema/StructureDefinition/IG resource changes are required; this is a
narrative-page correction only.

#### Justification

The ticket is unambiguous, the requested change makes an otherwise invalid
FHIRPath snippet valid, and there is no semantic risk: the corrected form
(`where(...)`) is the canonical FHIRPath idiom for collection filtering. The
ticket has already been resolved "Persuasive" with the resolution comment
"Update the example as noted.", and the FHIR-I Zulip thread treats it as a
candidate for a block vote — i.e., the WG already views it as low-risk and
ready to apply.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying correction, but take the opportunity to audit the
entire FHIRPath narrative page for the same class of typo (filter
expressions written without `.where(...)`, plus the related case-of-System
type names that other tickets in the same Zulip block — FHIR-55495,
FHIR-37861 — also report). Bundle FHIR-55417's fix into a single editorial
PR against the FHIRPath page so the example is corrected and any
neighbouring inconsistent examples are normalised at the same time, rather
than applying isolated one-line edits ticket-by-ticket.

#### Justification

The Zulip thread explicitly proposes a block vote on a cluster of small
FHIRPath documentation tickets, which suggests the WG is already thinking
about handling them as a batch. Doing the audit once is cheaper than
re-touching the same page repeatedly, reduces the chance of leaving similar
typos in place, and keeps the build.fhir.org/fhirpath page internally
consistent. The trade-off vs. Disposition A is slightly more reviewer effort
and a marginally larger diff, but the change remains editorial and
non-normative.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave the example unchanged, on the grounds that the
FHIRPath specification itself (not the FHIR core spec) is the normative
source for FHIRPath syntax, and that the build.fhir.org/fhirpath page is a
pass-through reference whose examples should not be edited piecemeal in the
FHIR core repo.

#### Justification

This is the weakest option and is included only for completeness. The
example appears on a FHIR Core page maintained under the FHIR-I work group's
remit; it is a clear typo rather than a semantic disagreement, and there is
no community pushback on the Zulip thread or on the ticket itself. Declining
would leave an invalid FHIRPath snippet in published FHIR documentation with
no offsetting benefit.

---

### Recommendation

**Recommended disposition:** A

The fix is a one-line editorial correction that has already been resolved
"Persuasive" by triage, has implicit support on the FHIR-I Zulip thread
(including a proxy "yes" from Grahame Grieve), and carries no semantic or
backward-compatibility risk. Disposition B is reasonable if the WG wants to
batch the FHIRPath documentation tickets together at edit time, but that is
an execution detail rather than a different disposition — the
correctness call on FHIR-55417 itself is "apply the fix as requested."
Recommend voting Persuasive / Accept and applying the `.where` insertion as
part of the next FHIR Core publication build.
