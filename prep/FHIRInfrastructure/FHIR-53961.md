## Summary

Brian Postlethwaite is asking the FHIR Infrastructure WG to clarify the
specification of the FHIR-specific FHIRPath function `elementDefinition()`.
The current spec text (in the FHIRPath section of FHIR Core) describes the
function only briefly, and the reporter notes several open questions that
make it hard for engines and authors of invariants/profiles to use it
predictably:

- **What is returned** — Is the result a collection of
  `FHIR.ElementDefinition` objects? When the input is a property whose type
  is itself a complex datatype (e.g. `Patient.identifier`), is the returned
  `ElementDefinition` the one for the property in `Patient` (i.e. the
  containing resource's element) or the root element of the referenced
  datatype (`Identifier`)? For deeper paths such as `Patient.identifier.system`
  the assumption appears to be that the result comes from the `Identifier`
  type, but the spec does not say so.
- **Where it can be used** — Because the function references `%profile`,
  the reporter assumes it is only available in invariant/constraint
  contexts. The spec should state the allowed contexts explicitly.
- **Completeness of the returned `ElementDefinition`** — Is it a full
  `ElementDefinition`, or a defined subset (e.g. `path`, `type`,
  cardinality, `name`)? Engines need to know what callers can rely on.
- **Framing as reflection** — The reporter suggests the spec should
  acknowledge that `elementDefinition()` is effectively a partial
  reflection mechanism in FHIRPath, and frame it that way for readers.

The cross-reference index links the ticket only to two FHIR artifacts —
`Patient.identifier` and `Patient.identifier.system` — which are the
illustrative examples in the description rather than independent items
of work. There are no linked Jira tickets and no related GitHub PRs,
issues, or commits indexed against this ticket.

### [fhirpath > narrow the scope of fhirpath](https://chat.fhir.org/#narrow/stream/fhirpath/topic/narrow%20the%20scope%20of%20fhirpath/near/153982686)

Nikolai Ryzhikov argued that `resolve()`, `elementDefinition()`, and
`conformsTo()` expand FHIRPath's scope too much because they pull
FHIR-resolution, FHIR metadata, and a profile validator into the engine,
and proposed splitting the spec so these would live in a separate FHIR
extension layer. Bryn Rhodes confirmed that these are already part of
the FHIR-specific FHIRPath extensions (not the core FHIRPath spec), but
agreed it would be valuable to clarify how heavily they are actually
relied on in invariants. Grahame Grieve added that `resolve()` is used
"heaps" but the other two — including `elementDefinition()` — are not
used much in practice. The thread did not reach a formal resolution but
established two points relevant here: (a) `elementDefinition()` is a
FHIR extension to FHIRPath, not a core FHIRPath function, and (b) it is
sparsely used, which is consistent with the under-specified definition
that FHIR-53961 is now asking to tighten.

## Proposed Dispositions

Because there are no GitHub cross-references and no per-repo briefing
to anchor a code/IG path, the proposals below reference the canonical
authoring location of the affected text — the FHIRPath section of the
FHIR Core specification (typically `source/fhirpath.html` and the
function/extension index in HL7/fhir) — without naming a specific
repo path. A repo-analysis briefing for `HL7/fhir` should be loaded
before any of these dispositions is acted on.

### Disposition A: Accept As Requested

#### Proposal

Expand the normative description of `elementDefinition()` in the FHIR
Core FHIRPath section to specify, at minimum:

1. **Return type and shape** — explicitly state that the function
   returns a (possibly empty) collection of `FHIR.ElementDefinition`
   nodes, and define the rule for which `ElementDefinition` is returned
   for each input:
   - For an element whose type is a primitive or complex datatype, the
     returned `ElementDefinition` is the one for that element **in its
     containing StructureDefinition** (e.g. `Patient.identifier`
     returns the `ElementDefinition` for `Patient.identifier` in the
     `Patient` StructureDefinition).
   - For navigation **into** a complex datatype (e.g.
     `Patient.identifier.system`), the returned `ElementDefinition` is
     the one defined by the **datatype's** StructureDefinition
     (`Identifier.system`).
   - When `%profile` is in scope, the returned `ElementDefinition`
     SHOULD be the snapshot/diff entry from the profile, not the base
     resource, where applicable.
2. **Allowed contexts** — state that `elementDefinition()` is part of
   the FHIR-specific FHIRPath extensions and is only guaranteed to be
   available in contexts where FHIR profile/StructureDefinition
   metadata is in scope (notably `ElementDefinition.constraint.expression`
   and other invariant-style contexts). Engines outside those contexts
   MAY raise an error.
3. **Completeness** — clarify that the returned node is a full
   `ElementDefinition` instance as defined by the resource, and that
   callers can rely on at least `path`, `type`, `min`, `max`, and
   `definition` being populated when present in the source profile.
4. **Framing** — add a non-normative paragraph noting that
   `elementDefinition()` is a partial reflection facility and should
   be treated as such by tooling.
5. Add the two worked examples from the ticket
   (`Patient.identifier`, `Patient.identifier.system`) verbatim, with
   the resolved `ElementDefinition.path` for each.

These edits land in the FHIRPath authoring page in HL7/fhir (and any
mirrored function index) and do not require schema or resource
changes.

#### Justification

The change is purely clarifying — it tightens spec language without
altering behaviour that any reasonable engine would already implement.
Brian's questions are exactly the kind that have to be answered for
any implementer who is wiring `elementDefinition()` into a profile
validator, and the asymmetry between "element on the containing
resource" vs "root element on the datatype" is a real ambiguity that
will produce divergent behaviour between engines if left implicit.
Aligning the text with the way `%profile` is referenced (only in
invariant contexts) matches Bryn Rhodes's confirmation on Zulip that
this is a FHIR-specific extension and not part of core FHIRPath.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the current `elementDefinition()` description but introduce a
**new dedicated FHIRPath reflection section** (e.g. "FHIR Reflection
Functions") in the FHIR Core FHIRPath page. Move
`elementDefinition()` into that section alongside related FHIR-only
functions, and document the same clarifications listed under
Disposition A as section-wide rules:

- All reflection functions are scoped to invariant/constraint
  contexts with `%profile` in scope.
- All reflection functions return collections of resource/type
  metadata nodes (`ElementDefinition`, `StructureDefinition`, …).
- A worked-example block at the section level shows the
  property-vs-datatype resolution rule once and is cross-linked from
  each function.

The function-level text for `elementDefinition()` then becomes a
short pointer to the section rules plus the two `Patient.identifier`
examples.

#### Justification

This addresses the underlying need — implementers not knowing the
contract — without bolting all of the clarifications onto a single
function definition. It also creates a natural home for the
"reflection" framing that Brian explicitly asked the spec to
acknowledge, and lets future FHIR-specific reflection helpers (or a
later split-out of the FHIR FHIRPath extensions, as Nikolai suggested
on Zulip) reuse the same scaffolding. The trade-off vs Disposition A
is a larger editorial reshuffle and a slightly higher risk of
disturbing cross-references from other parts of the spec into the
FHIRPath page.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive — Not Persuasive" / "Already addressed",
on the grounds that the existing FHIRPath text plus the
`ElementDefinition` and `StructureDefinition` resource definitions
already, by reference, define what `elementDefinition()` returns and
where it may be used. No spec change is made; FMG/FHIR-I would
instead point implementers to the FHIR-specific FHIRPath extensions
section and to `%profile` to infer scope.

#### Justification

A defensible (if weak) argument: the function is sparsely used in
practice — Grahame stated on Zulip that `elementDefinition()` is "not
so much" used compared with `resolve()` — so the cost of leaving the
ambiguity in place may be small. The mechanism (returning
`ElementDefinition` nodes) is already implied by the function name
and the way it is referenced from invariant contexts. This option
should be considered only if the WG concludes that further
clarification risks over-specifying behaviour that engines have
already converged on de-facto.

---

### Recommendation

**Recommended disposition:** A

The ticket is a clear request for editorial precision in a spec
section that is already known to be under-specified, and Brian's
specific questions (property vs datatype resolution, allowed
contexts, completeness of the returned node, reflection framing) all
have unambiguous, low-risk answers. Disposition A delivers those
answers in place and lets implementers act on them immediately, with
minimal disruption to the rest of the FHIRPath page. Disposition B
is a reasonable longer-term refactor and could be folded in later if
FHIR-I decides to formalise a "FHIR Reflection" section, but it
should not block this fix. Disposition C is hard to defend now that
the WG has been told in writing that two engines/profile authors
disagree on the basic semantics — Grahame's "not used much" comment
from 2018 argues for cheap clarification, not for leaving the gap
open.
