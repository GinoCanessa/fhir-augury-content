## Summary

The reporter (Nick George) found that the FHIR Core ValueSet
`http://hl7.org/fhir/ValueSet/security-labels` (used by `Meta.security`)
includes-by-reference the canonical
`http://terminology.hl7.org/ValueSet/v3-ConfidentialityClassification`,
which does not exist in THO. The reporter further observes that on the
rendered page for `security-labels` this canonical is the only one not
turned into a hyperlink, confirming the publisher cannot resolve it.
The reporter proposes that the include should instead point to
`http://terminology.hl7.org/ValueSet/v3-Confidentiality`, which is the
correct THO canonical for the same OID-bearing value set.

A subsequent terminology Zulip discussion (cross-referenced below) and
later Jira comments (Marc Duteau, John Moehrke, Carmela Couderc)
confirmed the diagnosis: when the v3 content was migrated from FHIR
into THO/UTG, FHIR's historical synthetic name `v3-ConfidentialityClassification`
was replaced by THO's canonical name `v3-Confidentiality`, but the FHIR
R4 and R4B editions of `security-labels` were not updated. R5 has been
fixed, so this is a Triaged R6-labelled change request that is being
held for confirmation/closure rather than further design work. The
broader (separate) question of how to back-fix the broken reference for
already-published R4/R4B remains a THO/UTG governance topic.

## Related Zulip Discussions

### [terminology > ISsue with http://terminology.hl7.org/ValueSet/v3-Confidenti](https://chat.fhir.org/#narrow/stream/terminology/topic/ISsue%20with%20http%3A%2F%2Fterminology.hl7.org%2FValueSet%2Fv3-Confidenti)

This is the originating thread (Dec 2022) where Nick George first
reported that `http://hl7.org/fhir/ValueSet/security-labels` (transitively
used by `Meta.security`) includes the unresolvable
`http://terminology.hl7.org/ValueSet/v3-ConfidentialityClassification`,
visible on the R4B render but apparently expandable on the R4 render
because the R4 build embedded an expansion. John Moehrke and Mohammad
Jafari quickly agreed the right target is `v3-Confidentiality` in THO,
and Ted Klein provided the historical analysis: when v3 content was
imported into FHIR from the V3 coremif, the URL was constructed
algorithmically as `v3-<name>`, but FHIR at some point (Aug 2020,
"phase 1 removing UTG content from core build") renamed/aligned to
THO's `v3-Confidentiality`. The R4/R4B published artifacts were left
referencing the obsolete name.

The thread converged on a remediation plan: rather than mutating THO
(which would break OID/canonical invariants and other consumers), THO
should publish a thin pass-through ValueSet at the legacy id
`v3-ConfidentialityClassification` (no OID, or a different OID) that
simply `compose.include`s `v3-Confidentiality`, with a description
discouraging future use and a retirement plan. Grahame Grieve agreed
this was acceptable and asked someone to file a THO proposal. John
Moehrke asked Jessica Bota for help walking through the THO proposal
process; Jessica indicated willingness but said the concrete content
fix still needed to be agreed. The thread ends without a confirmed THO
proposal having been filed. A separate but analogous breakage was
flagged by Nick George for `http://hl7.org/fhir/ValueSet/usps-state`
(now resolved by US Core) but was set aside.

Participants: Nick George, John Moehrke, Mohammad Jafari, Ted Klein,
Grahame Grieve, Jessica Bota.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In FHIR Core (R6), update the definition of
`http://hl7.org/fhir/ValueSet/security-labels` so that the
`compose.include[].valueSet` entry that currently points to
`http://terminology.hl7.org/ValueSet/v3-ConfidentialityClassification`
points instead to `http://terminology.hl7.org/ValueSet/v3-Confidentiality`,
matching the THO canonical for the underlying OID-bearing v3
Confidentiality value set. Confirm via the John Moehrke comment that R5
already carries this fix; mark the ticket Persuasive / Already Applied
for R6, with an explicit cross-check that the R6 source for
`security-labels` indeed contains the corrected canonical. No change is
made to R4/R4B (those are immutable published normative artifacts);
the broader R4/R4B back-fix is delegated to a THO proposal as discussed
in the Zulip thread (out of scope for this FHIR-I ticket).

#### Justification

This is exactly what the submitter asked for and matches the
unanimous diagnosis of the terminology workgroup discussion: the legacy
canonical `v3-ConfidentialityClassification` was a FHIR-side
construction that never existed in THO/UTG, and `v3-Confidentiality`
is the THO-managed value set with the same OID and identical
definition (per Marc Duteau and Ted Klein). Pointing
`security-labels` at the canonical that actually resolves restores
publisher expansion, removes the lone non-hyperlinked entry on the
rendered page, and is internally consistent with how R5 already
references THO. Because the OID and definition match, there is no
semantic change for implementers. The R6 label on the ticket and
John Moehrke's "fixed since R5" confirmation indicate this is at most
a confirmation-and-close action for R6.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the include-URL change for R6 as in Disposition A, **and**
additionally have FHIR Infrastructure formally request that THO
publish a thin pass-through ValueSet at the legacy id
`http://terminology.hl7.org/ValueSet/v3-ConfidentialityClassification`
whose only `compose.include` is `v3-Confidentiality` (with no OID or a
distinct non-conflicting OID, and a description marking it as a
deprecated alias retained solely so that already-published R4 and R4B
artifacts continue to expand). Track that as a UP/THO ticket and link
it from this Jira ticket; close FHIR-39671 once the FHIR Core change
is confirmed for R6 and the THO ticket is opened.

#### Justification

Disposition A resolves the R6 problem but leaves the published R4 and
R4B `security-labels` value sets pointing at a dead canonical, which
is exactly the experience that prompted Nick George to file the
ticket in the first place (against R4B). The Zulip thread converged on
this pass-through-ValueSet pattern as the only viable way to avoid
either mutating an already-published THO value set or duplicating the
OID. Grahame Grieve explicitly endorsed making a THO proposal to do
this. Filing the THO request as part of dispositioning this ticket
ensures the action item is not lost (the Zulip thread ended without a
THO ticket being created, and Carmela Couderc's 2023 comment also
suggested the submitter open a UP ticket — which apparently did not
happen). The trade-off vs. A is a small amount of cross-WG
coordination effort in exchange for actually closing the loop for
implementers stuck on R4/R4B.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — Already Resolved." Per John
Moehrke's 2025-06-14 comment, the issue has already been corrected in
R5 and the current FHIR Core build, and there is therefore nothing
left for FHIR-I to do in R6. Any remaining R4/R4B remediation is a
THO/UTG governance matter and outside the scope of a FHIR Core change
request.

#### Justification

If the R6 source for `security-labels` already references
`v3-Confidentiality`, there is no FHIR Core change to make and the
ticket is purely procedural. The R4 and R4B value sets are normative
published artifacts that FHIR-I cannot rewrite in place; the only
viable remediation lives in THO (per Ted Klein and Grahame Grieve in
the Zulip thread). On that reading, FHIR-39671 is a duplicate of work
already done and a misfile against the wrong tracker, and closing it
keeps the FHIR-I R6 ballot prep focused on substantive changes.

---

### Recommendation

**Recommended disposition:** B

The community discussion is unambiguous about the diagnosis and the
two needed actions: (1) make sure FHIR Core (R6) references the
correct THO canonical, and (2) get THO to publish a pass-through
ValueSet at the legacy canonical so that already-published R4/R4B
artifacts stop being broken in the wild. Disposition A handles only
half of that and risks repeating the same gap that this ticket is
already stuck in (Carmela Couderc's 2023 suggestion to file a UP
ticket was never acted on, and the Zulip thread similarly ended with
"someone should write a THO proposal" and silence). Disposition C is
defensible if R6 is verifiably already correct, but it leaves
implementers on R4/R4B with the exact unresolvable include the
submitter reported. Disposition B costs FHIR-I very little — confirm
the R6 fix, then formally hand the THO pass-through request to UTG via
a UP ticket linked from this one — and is consistent with both Grahame
Grieve's explicit endorsement of the pass-through pattern and John
Moehrke's stated goal of "making the mistake less painful" for R4/R4B
consumers.
