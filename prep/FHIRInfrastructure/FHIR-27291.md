## Summary

The ticket asks the FHIR Shorthand (FSH) grammar to support a list-literal
syntax for fields whose cardinality is greater than one, so that an author
can write a single assignment like
`* measure = #missing, #text, #not, #in, #not-in, #below, #above, #ofType`
instead of repeating the path with explicit indices
(`* modifier[0] = #missing`, `* modifier[1] = #text`, …). The proposed
wording in the ticket points at the FSH ANTLR grammar
(`FHIR/sushi/blob/master/antlr/src/main/antlr/FSH.g4`) as the place where
the change would land, implying a grammar/parser-level extension to the
existing assignment rule rather than a new rule, structure, or extension in
the FHIR spec proper. No related Jira tickets or GitHub items were
surfaced via cross-references; the only related discussion is a 2024 Zulip
thread (`#shorthand > shorthand for lists`) that explicitly re-raises this
ticket.

## Details

**Description:**

> I want to be able to say: `* field = a, b, c` for fields which can take an array.
>
> - `modifier[0] = #missing`
> - `modifier[1] = #text`
> - `modifier[2] = #not`
> - `modifier[3] = #in`
> - `modifier[4] = #not-in`
> - `modifier[5] = #below`
> - `modifier[6] = #above`
> - `modifier[7] = #ofType`
>

> `measure = #missing, #text, #not, #in, #not-in, #below, #above, #ofType`

## Related Zulip Discussions

### [#shorthand > shorthand for lists](https://chat.fhir.org/#narrow/stream/shorthand/topic/shorthand%20for%20lists/near/472734049)

A community implementer (Kippi Bordowitz) asked on 2024-09-24 whether
there is a shorter syntax for assigning multiple codes to a repeating
element, e.g. `* base = (#MedicationRequest and #MedicationAdministration
and #MedicationDispense)`, observing that the `* base[0] = … * base[+] =
…` pattern feels overly verbose. Chris Moesel (FSH/SUSHI maintainer)
responded the next day confirming FSH has no list shorthand today, noting
that the same idea was raised in the very first FSH ballot and deferred
(explicitly citing FHIR-27291), and inviting other implementers to weigh
in. Elliot Silver pushed back mildly, asking how often lists of *basic*
datatypes actually appear and suggesting most repeating elements are of
complex types where the savings would be smaller. Chris agreed that is
probably why the Jira issue never progressed, but pointed out that since
FSH codings can be assigned to `CodeableConcept` and `Coding`, the
shorthand would still help for common arrays such as
`Observation.category` and `CodeableConcept.coding`. The thread ends
without consensus and without a concrete grammar proposal.

Participants: Chris Moesel, Kippi Bordowitz, Elliot Silver.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the FSH grammar (`FHIR/sushi`'s `antlr/src/main/antlr/FSH.g4`) and
the SUSHI compiler to accept a comma-separated value list on the
right-hand side of an assignment to a repeating element, semantically
equivalent to a sequence of soft-indexed assignments. Concretely, add a
new alternative to the assignment rule along the lines of:

```
fixedValue : path '=' value (',' value)* ;
```

with the semantics that `* path = v1, v2, …, vN` desugars to
`* path[0] = v1`, `* path[+] = v2`, …, `* path[+] = vN`. The values
permitted in the list are exactly the value forms FSH already accepts on
the right of `=` (codes, codings, strings, numbers, booleans, references,
quantities), and the rule errors out at compile time if the path's
cardinality upper bound is `1`. Update the FSH reference documentation
(rules pages on `build.fhir.org/ig/HL7/fhir-shorthand/`) and SUSHI's
parser/exporter accordingly, with worked examples for `Observation.category`
and the `modifier` example from the ticket.

#### Justification

The change is purely additive syntactic sugar over an already-supported
semantic operation (assigning multiple values to a repeating element via
soft indexing), so it does not introduce new modeling capability or
break any existing FSH document. The pattern is consistent with how
several other DSLs (CQL, ANTLR-based config languages) already permit
inline lists for set-like assignments, and it directly addresses a
recurring ergonomic complaint surfaced both at the original 2020 STU1
review and again in the 2024 Zulip thread. Because FSH grammar changes
live in `FHIR/sushi`, the change is contained to one tool and one
specification page rather than touching the FHIR core spec. The
re-opening note from lynn_laakso explicitly held the suggestion "for
potential implementation" after STU1, indicating WG appetite to
revisit it once the base language stabilised.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a new grammar production for inline lists, add a
narrower piece of syntactic sugar tied to the existing `[+]` soft-index
operator: allow a single assignment whose right-hand side is a
parenthesised list to be unrolled, e.g.

```
* base = (#Patient | #Practitioner | #RelatedPerson)
* category = (#vital-signs | #laboratory)
```

(or a comma-separated equivalent inside parentheses), and restrict the
sugar to value types that are unambiguous in FSH today (codes, codings,
quantities, primitives). Defer broader list literals (references,
nested CodeableConcepts, Quantities with units) to a follow-up ticket
once SUSHI has telemetry on which forms implementers actually want.
Document the construct as "list assignment for repeating coded
elements" rather than as a general list literal.

#### Justification

This addresses Elliot Silver's concern in the Zulip thread that the
benefit is concentrated on coded repeating elements (e.g.,
`Observation.category`, `CodeableConcept.coding`,
`MedicationRequest.category`, `Patient.communication.language.coding`)
and may not justify the additional grammar surface for arbitrary value
types. A parenthesised, restricted form is easier to disambiguate from
existing FSH constructs (no risk of a stray comma in a free-text
description being parsed as a list separator), keeps the parser
changes smaller, and gives the project an incremental upgrade path:
extend the allowed inner value types in a future minor release once
real-world use is observed. It also avoids committing the language to
a delimiter (`,`) before the WG has discussed how it interacts with
other comma-bearing FSH constructs (parameters in profiles, slicing
expressions).

---

### Disposition C: Decline

#### Proposal

Resolve as **Persuasive with Modification — keep current syntax**:
decline the new grammar and instead document the existing
soft-indexing idiom (`* path[0] = v1`, `* path[+] = v2`, …) as the
canonical way to assign multiple values to a repeating element, with a
worked example in the FSH reference for `CodeableConcept.coding` and
`Observation.category`. Optionally, ask SUSHI to emit a hint when it
sees three or more sequential indexed assignments to the same path,
pointing the author at the documented idiom rather than introducing
new syntax.

#### Justification

The capability already exists; the request is purely about typing
convenience. Adding a grammar production has a perpetual maintenance
cost (parser, formatter, linter, syntax highlighting in every editor
extension that targets FSH, plus interaction with future grammar
additions), whereas the existing `[+]` soft-index operator is already
documented and unambiguous. The 2020 WG vote (Kramer/Rhodes 13-0-0)
originally placed the ticket in "Considered for Future Use", and the
2024 Zulip discussion did not surface any new use cases beyond the
original 2020 framing — only one implementer asking and one maintainer
noting nobody had pushed it. With STU1 published and no significant
field demand having materialised in four years, the cost/benefit
tilts toward stability of the language.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best starting point for the workgroup because it
captures the real ergonomic win — assigning multiple codes/codings to
common repeating elements like `Observation.category` and
`CodeableConcept.coding` — without committing the FSH grammar to a
fully general list literal whose interactions with other
comma-bearing constructs have not been worked through. It directly
answers Elliot Silver's concern in the 2024 Zulip thread (most
beneficial for coded arrays) while still delivering what Chris Moesel
identified as the recurring pain point. Disposition A is a defensible
end-state but is larger than what implementers have actually asked for
and risks debate over delimiter choice and value-type scope; doing it
in two steps (B now, A later if demand grows) is lower-risk for a
language at FSH's stage of adoption. Disposition C is hard to defend
given that the ticket has now been explicitly re-raised by the
community four years after re-opening — declining without delivering
any improvement is likely to invite the same request a third time.
