## Summary

Brian Postlethwaite reports that the `defaultValue[x]` element on
`StructureMap.group.rule.source` has been narrowed to a single `string`
datatype, whereas in R4 the same element accepted a wide choice of
datatypes (`defaultValue[x]` as a `*` choice). He asks that the other
allowed datatypes be restored — at minimum the set that maps to the
FHIRPath-compatible literal types, since those are the types that the
FHIR Mapping Language (FML) text grammar can serialize as literals in
rule sources. As written, the change makes the StructureMap resource
unable to round-trip an FML map whose `default` clause uses any literal
other than a string (e.g., integer, decimal, boolean, date, dateTime,
time, Quantity).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restore `StructureMap.group.rule.source.defaultValue[x]` to a choice
type (`*`) — or, at minimum, expand it from the current `string`-only
binding to the set of primitive datatypes that correspond to the
literal forms expressible in the FML grammar:

- `string`
- `integer`
- `decimal`
- `boolean`
- `date`
- `dateTime`
- `time`
- `Quantity`

The change would be made on the StructureMap resource definition
(element `StructureMap.group.rule.source.defaultValue[x]`), restoring
the broader choice type. Update the element documentation to note
that the value SHALL be expressible as a FHIRPath literal so that the
JSON/XML form round-trips with the `default` clause in FML text
serialization.

#### Justification

The R4 definition allowed the same element to take any datatype, and
the FML text grammar already accepts non-string literals in `default`
clauses. Narrowing the element to `string` only creates a silent
data-loss path between FML text and the StructureMap resource: an FML
file with `default 0` (integer) or `default true` (boolean) cannot be
captured losslessly in StructureMap. Restoring the choice — at least
to the FHIRPath literal types — re-aligns the resource with both the
prior major version and the companion FML grammar that the WG
maintains. The constraint to "FHIRPath-compatible literals" matches
what implementations actually need to serialize today and avoids
re-introducing exotic types nobody used.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the typed element as `defaultValueString` (1..1 choice of one),
but add a sibling `defaultValueExpression` of type `Expression` (or
reuse `string` typed as a FHIRPath literal expression with a
documented parsing rule) so that non-string default values are
expressed by their FHIRPath literal form rather than a typed FHIR
value. Document in the StructureMap narrative and the FML mapping
guide that any non-string default is conveyed via the
expression/literal path, and that consumers parse it using FHIRPath
literal rules.

#### Justification

This avoids reintroducing a `*` choice on a resource where only a
small subset of types is actually meaningful, and keeps the JSON
shape stable for tooling that already targets `defaultValueString`.
It also future-proofs the element against new literal forms by
delegating to the FHIRPath literal grammar instead of enumerating
datatypes. The trade-off vs. Disposition A is that round-trip with
existing R4 instances that carried, e.g., `defaultValueInteger`
becomes a transformation rather than a structural match, and
implementers must run a tiny parser to recover the typed value.

---

### Disposition C: Decline

#### Proposal

Leave `StructureMap.group.rule.source.defaultValue[x]` as `string`
only, on the basis that the narrowed shape was a deliberate
simplification and that any non-string default value can be
represented today by writing the literal as a string (e.g., `"0"`,
`"true"`) and letting downstream evaluation in the rule cast it.
Recommend updating the FML serializer/parser guidance to emit and
accept string forms.

#### Justification

The case for declining rests on (a) implementer simplicity — a single
typed element is easier to bind, validate, and search than a `*`
choice; (b) the fact that `defaultValue` is consumed by a mapping
engine that already has to coerce string inputs in many other rule
positions; and (c) that no production complaint beyond round-trip
elegance has been recorded. If the WG accepts a small lossy step
between FML text (typed literal) and StructureMap JSON (string), no
spec change is required.

---

### Recommendation

**Recommended disposition:** A

The reporter's argument — that the FML text grammar already admits
typed literals and that the resource form should be able to capture
them losslessly — is the strongest design constraint in play. R4
already supported the broader choice, so Disposition A is the
backwards-aligned option and removes a regression rather than adding
new surface area. Scoping the choice to the FHIRPath literal types
(string, integer, decimal, boolean, date, dateTime, time, Quantity)
keeps the change tightly bounded and avoids reintroducing rarely-used
types. Disposition B works, but it pushes parsing into every consumer
and complicates validation; Disposition C accepts a permanent
round-trip gap between FML and StructureMap that the WG will keep
having to explain. The WG should confirm the exact type list and
whether `Coding`/`CodeableConcept` should also be admitted given that
FML has token-style literals for codes.
