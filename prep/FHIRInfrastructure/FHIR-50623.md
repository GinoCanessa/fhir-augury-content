## Summary

Weiyu Zhang raises a combined documentation correction and design question about
the `$event` operation on `Subscription` in FHIR Core (R6).

1. **Documentation correction.** The `content` parameter on `Subscription.$event`
   is documented as drawing its codes from `backport-content-value-set`. In R6
   core, this should almost certainly point at the R6-native content value set
   (i.e., the equivalent of `SubscriptionPayloadContent`) rather than the
   backport value set, which is an artifact of the R4/R4B Subscriptions
   Backport IG.
2. **Design question.** Can/should servers honour a `content` value on
   `$event` that differs from the `content` the client originally negotiated
   when creating the `Subscription` — for example, sending notifications as
   `empty` or `id-only` and only materialising `full-resource` payloads when a
   client explicitly asks via `$event`? And if `full-resource` is returned via
   `$event`, is the resource the *snapshot* at the time of the original
   notification event, or the *current* state of the resource (i.e., equivalent
   to a fresh `read`)?

The cross-reference, Zulip, and GitHub searches against the Augury index
returned **no related items**, so there is no surfaced prior community
discussion to anchor the answer; the question appears to be fresh.

## Proposed Dispositions

> Note: with no cross-referenced GitHub PRs/issues or Zulip threads to anchor
> repo paths, the dispositions below name the FHIR Core artifact families
> (Subscription resource page, `$event` `OperationDefinition`, content value
> set) rather than specific file paths. A planning pass against `HL7/fhir`

### Disposition A: Accept As Requested

#### Proposal

Treat this as two coupled changes against the FHIR Core (R6) `Subscription`
material:

1. **Doc fix (uncontroversial).** Edit the `OperationDefinition` for
   `Subscription.$event` so that the `content` parameter's `documentation`
   stops referring to `backport-content-value-set` and instead binds (and
   names in prose) the R6-native content value set used elsewhere on
   `Subscription` / `SubscriptionTopic` (e.g., `SubscriptionPayloadContent`,
   with codes `empty`, `id-only`, `full-resource`). Confirm the
   `binding` element on the `content` parameter actually points at that R6
   value set; replace the URL if it is still the backport one.
2. **Behavioural clarification (the substantive change).** Add normative
   text to the `$event` operation page stating that:
   - Clients **MAY** request a `content` level on `$event` that differs from
     the level negotiated on the underlying `Subscription`.
   - The parameter is a *hint*; the server **MAY** honour, downgrade, or
     ignore it (consistent with the existing "MAY be ignored" wording).
   - When a server returns `full-resource` payloads via `$event`, those
     resources represent the **current** state of the resource at the time
     `$event` is invoked (i.e., semantically equivalent to a `read`), **not**
     a frozen snapshot from when the original notification was emitted.
     Servers wishing to expose historical state should direct clients to
     `vread` / `_history`.

#### Justification

The doc reference to `backport-content-value-set` is almost certainly a
copy-from-backport-IG slip — the R4B Subscriptions Backport IG uses that value
set name precisely because the R4/R4B core spec lacks a native equivalent,
and R6 does have one. Fixing the binding aligns `$event` with the rest of the
R6 Subscription material and removes a normative pointer from core into an
IG, which is a layering bug.

The behavioural clarification matches the way `$event` is already framed (a
client-driven replay/refresh affordance) and matches how implementers
generally reason about `read`-style endpoints: returning the current state is
the cheap, well-defined option and is consistent with the rest of the RESTful
API. Calling out "current, not snapshot" up front avoids a class of
interoperability surprises where two servers diverge on the same parameter.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the documentation/binding fix from Disposition A (#1) unchanged, but
take a **narrower** position on the behavioural question:

- Keep the `content` parameter on `$event` purely as the existing *hint* with
  no new normative semantics about "differ from the Subscription's
  content level".
- Add only a non-normative clarification that the resources returned by
  `$event` reflect the **current** state of those resources at the time of
  the call, with an explicit note that "snapshot at notification time" is
  out of scope and is the job of resource versioning (`vread` / `_history`)
  if the server elects to support it.
- Defer the "ask for full-resource on demand even though the Subscription
  was negotiated as `id-only`" pattern to a follow-up ticket once
  implementer feedback exists.

#### Justification

The substantive part of the ticket — "is this a way to send notifications
with empty/id-only and only serve up full-resource if queried?" — is really a
new usage pattern for `$event`, not a clarification of existing intent.
Blessing it normatively in R6 without implementer feedback (Augury surfaces
no Zulip / GitHub discussion) risks locking in a contract that servers have
not actually agreed to. A narrower fix removes the immediate documentation
bug and the snapshot/current ambiguity (which *is* clearly in scope for
`$event`) while leaving the cross-level-content question open for the
Subscriptions community to weigh in on first.

---

### Disposition C: Decline

#### Proposal

Decline the substantive behavioural clarification, and merge only the
strictly-editorial doc fix (replace the `backport-content-value-set`
reference with the correct R6 value set) as a non-substantive correction
under the WG's editorial process. Close the question portion of the ticket
as **Not Persuasive – Already Addressed**, on the grounds that the existing
"this is a hint to the server… and MAY be ignored" wording already permits
servers to do (or not do) exactly what the reporter is asking about.

#### Justification

The reporter's question is already answerable from the current normative
text: a `content` value on `$event` is a hint, the server MAY ignore it,
which means a server is already free to honour a different level than was
negotiated, or to refuse to do so. The snapshot-vs-current question is
likewise governed by the resource being returned — `$event` does not promise
historical reconstruction. Adding new normative text risks over-specifying
an operation that was deliberately written as best-effort.

---

### Recommendation

**Recommended disposition:** A

The doc fix is uncontroversial regardless of which disposition the WG
ultimately picks (Disposition C also concedes it as editorial), so the only
real question is how much to say about behaviour. The snapshot-vs-current
ambiguity in particular is a real interoperability hazard: two conformant
servers reading today's text could plausibly land on opposite answers (one
returning the resource version that existed at notification time, the other
returning current state), and the cost of saying "current state, use
`vread`/`_history` for snapshot semantics" is essentially zero. Pairing that
with an explicit "client MAY request a different content level; server MAY
honour or ignore" statement formalises a pattern (`empty`/`id-only` push +
on-demand `full-resource` pull) that several Subscriptions deployments are
already moving towards, without imposing any new server obligations.
Disposition B's caution is reasonable but leaves the snapshot question
half-answered; Disposition C under-delivers given that the reporter has
flagged a concrete documentation bug that points into the wrong IG.
