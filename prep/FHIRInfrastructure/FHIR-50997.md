## Summary

The reporter (TakuyaShinozu) flags a defect in the published example
`Bundle/bundle-response` (the JSON rendering at
`bundle-response.json.html`). One of the response entries — the one
illustrating a conditional `DELETE` outcome — sets
`response.status` to the literal string `"DELETE"`. Per the
`Bundle.entry.response.status` definition, that element is required to
contain the HTTP response status code (as a string, e.g., `"200 OK"`,
`"204 No Content"`), not the HTTP method. The example therefore
contradicts the spec it is supposed to demonstrate. The reporter
hypothesizes a copy/paste-style mistake where the HTTP verb was written
where the HTTP status belongs. No related Jira tickets, Zulip threads,
or GitHub items were surfaced by cross-reference lookup, so this is a
narrow, self-contained Technical Correction against a single example
file.

## Details

**Description:**

The example at [Example Bundle/bundle-response (JSON)](https://www.hl7.org/fhir/bundle-response.json.html)
shows the following response entries for DELETE operations:

```json
{
  "response": {
    "status": "DELETE"
  }
}
```

The use of `"DELETE"` as a value for `response.status` is not compliant
with the FHIR specification. According to the
[Bundle - FHIR v5.0.0](https://www.hl7.org/fhir/bundle-definitions.html#Bundle.entry.response.status)
definition, the value must be a valid HTTP status code string.

Using `"DELETE"` in this context appears to be a mistake — possibly a
misinterpretation of the HTTP method instead of the HTTP response
status.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a Technical Correction to `HL7/fhir` at
`source/bundle/bundle-response.xml`. Change the offending entry — the
conditional-delete response, currently:

```xml
<response>
  <!-- delete matching resource - but you can't find out what was
       deleted - an inherent limitation in a condition delete -->
  <status value="DELETE"/>
</response>
```

to use a real HTTP status code. The natural choice, consistent with
the sibling simple-delete response in the same file (which uses
`"202 Accepted"`), is:

```xml
<status value="204 No Content"/>
```

`204 No Content` matches the canonical successful-delete pattern on
the [HTTP interactions page](https://www.hl7.org/fhir/http.html#delete);
`202 Accepted` is also defensible if the workgroup wants to mirror the
adjacent simple-delete entry verbatim. While in the file, also
normalise the two `"200 ok"` occurrences to `"200 OK"` for casing
consistency with the other entries (optional but cheap). No code, no
StructureDefinitions, no narratives, and no other example files need
to change; the publisher will regenerate the `.json.html`, `.xml.html`,
and `.ttl.html` renderings from the corrected XML.

#### Justification

The current value violates the very invariant the example is intended
to illustrate (`Bundle.entry.response.status` MUST be the HTTP
response status, per the element definition the reporter cites).
Examples in the spec are routinely treated as normative-by-illustration
by implementers, and a plainly wrong value here actively misleads
implementers writing transaction/batch processors. The fix is a
one-line, backward-compatible content edit confined to a single
authoring file under the briefing's documented authoring root
(`source/`); it touches no generated areas, no schemas, and no other
resources, so the regression surface is essentially nil. As a
Technical Correction it requires no ballot.

---

### Disposition B: Alternative Approach

#### Proposal

Same edit to `source/bundle/bundle-response.xml`, but pair it with a
small expansion: replace the single conditional-delete entry with two
worked entries that show both the documented outcomes — one with
`status="204 No Content"` (success, nothing deleted or one match
deleted) and one with `status="200 OK"` plus an `OperationOutcome`
in the entry resource (for the case where the server wants to report
which resource(s) were affected). Add a one-line XML comment cross-
referencing `http.html#3.1.0.7.1` (conditional delete). Optionally
take the same opportunity to normalise the casing of the existing
`"200 ok"` values in the file to `"200 OK"`.

#### Justification

The reporter's literal request is to fix the wrong status value, but
the underlying reason the original author wrote `"DELETE"` is
plausibly that conditional delete has more than one legitimate
response shape and the example was trying to gesture at "this is the
delete case" rather than commit to a specific HTTP status. Showing
both shapes makes the example genuinely instructive instead of just
correcting it to one value, and it pre-empts the obvious follow-up
"why didn't you use 200 OK?" question. Trade-off: it is a slightly
larger edit than a strict Technical Correction warrants, and grows
the rendered example, so a reviewer who prefers minimal-scope fixes
will favour Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the change as out-of-scope cosmetic/example cleanup, on the
basis that the surrounding XML comments ("response to the conditional
delete operation", "delete matching resource …") make the intent of
the entry clear to a reader and that
`Bundle.entry.response.status`'s normative definition is unambiguous
elsewhere in the spec.

#### Justification

This is the weakest of the three options and is included for
completeness. The only honest argument for declining is process — that
example files are not normative — and that argument does not survive
contact with the briefing's "examples are authored content under
`source/`, edits go in directly" guidance and the reality that the
example contradicts the very invariant it is meant to demonstrate.
There is no breaking-change, no maintenance-burden, and no
backward-compatibility justification available here.

---

### Recommendation

**Recommended disposition:** A

This is the textbook shape of a Technical Correction: a single
authoring file under `source/bundle/`, one obviously-wrong literal,
no normative impact, no cross-repo ripple, and no community
disagreement to weigh (the cross-reference and Zulip lookups returned
nothing substantive). Disposition B is attractive but expands the
scope beyond what the reporter asked for and beyond what a Technical
Correction conventionally covers; if the workgroup wants the richer
two-entry illustration, that is better tracked as a follow-up
documentation enhancement rather than bundled into this fix.
Disposition C has no real defence. Apply Disposition A — change
`<status value="DELETE"/>` to `<status value="204 No Content"/>` in
`source/bundle/bundle-response.xml` — and close.
