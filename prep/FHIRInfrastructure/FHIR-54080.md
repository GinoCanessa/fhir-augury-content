## Summary

Eric Haas requests that the **XML Digital Signatures** section of the FHIR Core
`signatures.html` page be rewritten so that it focuses on how the FHIR
`Signature` / `Provenance.agent` elements should be populated when an XML
Digital Signature (XML-DSig) or an XAdES profile is used, rather than on
restricting the underlying signature technology. The proposal removes three
prescriptive constraints currently in the spec — the `SHALL` requiring an
X.509 certificate, the `SHALL` mandating a Detached signature, and a
copy/paste error tying `Signature.type` value-set behaviour to the (CMS-only)
`srCms` element — on the grounds that these are implementation choices that
do not belong in the FHIR base standard and risk colliding with use-case
profiles. It also removes a duplicate sentence about canonicalization that
the reporter says is already covered in a preceding section. The proposed
replacement text reframes every constraint as `SHOULD`-level guidance on
mapping XML-DSig artifacts onto FHIR `Signature.type`/`when`/`who`/
`onBehalfOf`/`targetFormat`/`sigFormat`/`data` (and the `Provenance.agent.*`
equivalents).

This ticket is part of a coordinated set of signature-section rewrites the
same reporter has filed against the same page:
[FHIR-53835](https://jira.hl7.org/browse/FHIR-53835) rewrites the page
introduction to cover all electronic-signature types,
[FHIR-54025](https://jira.hl7.org/browse/FHIR-54025) applies the same
"focus on FHIR content, not the signature standard" treatment to the JWS
section, and [FHIR-54035](https://jira.hl7.org/browse/FHIR-54035) adds a
parallel CMS section. The reporter explicitly references the JWS and CMS
tickets as the model this XML-DSig rewrite mirrors.
[FHIR-53891](https://jira.hl7.org/browse/FHIR-53891) (canonicalization
edits) and [FHIR-54066](https://jira.hl7.org/browse/FHIR-54066)
(`Signature.targetFormat` alignment with the DataTypes page) touch the
same page and need to be reconciled with this proposal.

## Details

**Description:**

> Similar to my comments on CMS and JWS signatures focus on the FHIR content rather than the signature content.
>
> - To keep the FHIR Standard as a flexible base standard that avoids potential conflicts with use case-specific profiles. Do not introduce the XMLDSig standard requirements.
>
> Remove:
> - "When FHIR Resources are signed, the signature is across either the Canonical XML form or Canonical JSON form of the resource(s) as indicated in the targetFormat element." — This topic is fully explained in a preceding section and is redundant here.
> - "The Signature SHALL contain a X.509 Certificate" — Why? Although X.509 is commonly supported, this is an implementation decision and there is no need for FHIR to restrict the choice here.
> - "The signature is a Detached Signature (where the content that is signed is separate from the signature itself)" — Again why? I found no technical reason to restrict the choices. This is an implementer decision.
> - "The Signature.type SHALL contain the same values as the srCms element." — this is a copy/paste error.
>
> Proposed text (source at https://github.com/Healthedata1/fhir/edit/master/source/signatures.html):
>
> ### XML Digital Signatures
>
> When the digital signature is an XML Signature Syntax and Processing (XML Digital Signature) ([Version 1.0](https://www.w3.org/TR/2002/REC-xmldsig-core-20020212/), [Version 1.1](https://www.w3.org/TR/xmldsig-core/), or [Version 2.0](https://www.w3.org/TR/xmldsig-core2/)) or a profile built upon it, such as [XAdES (XML Advanced Electronic Signature)](https://www.w3.org/TR/XAdES/), the following guidelines and rules apply:
>
> - `Signature.type` or `Provenance.agent.type`: This element SHOULD be populated with the codes from [Signature Type Codes](https://hl7.org/fhir/6.0.0-ballot4/valueset-signature-type.html).
>   - The Purpose can be the action being attested to, or the role associated with the signature.
>   - It SHOULD agree with any "purpose" element in the XML Digital Signature (for example, the "CommitmentTypeId" element defined in the XAdES profiles).
> - `Signature.when` or `Provenance.agent.occurredDateTime`: This element SHOULD be populated with the signature timestamp.
>   - It SHOULD agree with the "signingTime" element or "signatureTimeStamp" property in the XML Digital Signature.
> - `Signature.who` or `Provenance.agent.who`: This element SHOULD be populated with a `Reference.reference` or `Reference.identifier` of who signed the content.
>   - It SHOULD match the signer's identity found in the digital certificate (for example, the Subject Name (SN), or one of the Subject Alternative Names (SAN) in an X.509 certificate).
> - `Signature.onBehalfOf` or `Provenance.agent.onBehalfOf`: When applicable, this element SHOULD be populated with a `Reference.reference` or `Reference.identifier` of the party represented by the signer.
> - `Signature.targetFormat`: This element SHOULD be populated, and the value SHOULD contain a canonicalization URL from one of those defined above or defined by other specifications.
> - `Signature.sigFormat`: This element SHOULD be populated with the value `application/xml` for XML Digital Signature and profiles.
> - `Signature.data`: The base64 encoded XML Digital Signature.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's replacement text verbatim for the **XML Digital
Signatures** subsection of `source/signatures.html` in the FHIR Core
build. Concretely:

1. Delete the redundant canonicalization sentence ("When FHIR Resources
   are signed, the signature is across either the Canonical XML form
   or Canonical JSON form…") — coordinate with FHIR-53891 to ensure
   the preceding Canonicalization section still carries the message.
2. Remove the three current `SHALL` constraints — the X.509 certificate
   requirement, the Detached-signature requirement, and the copy/paste
   sentence claiming `Signature.type` "SHALL contain the same values
   as the srCms element". The `srCms` reference is unambiguously a
   bug because `srCms` is a CMS-header concept and has no place in an
   XML-DSig section.
3. Replace the section with the proposed `SHOULD`-level mapping
   bullets covering `Signature.type` / `when` / `who` / `onBehalfOf`
   / `targetFormat` / `sigFormat` / `data` and the corresponding
   `Provenance.agent.*` paths, including the explicit cross-references
   to XAdES `CommitmentTypeId`, `signingTime`, and
   `signatureTimeStamp`, and the W3C XML-DSig 1.0/1.1/2.0 normative
   links.
4. Apply the same edit pattern in lockstep with FHIR-54025 (JWS) and
   FHIR-54035 (CMS) so the three subsections stay structurally
   parallel.

#### Justification

The three `SHALL`s being removed are technology-policy choices, not
FHIR conformance choices — XML-DSig itself permits Enveloped, Enveloping,
and Detached forms, and permits non-X.509 key material via
`KeyValue`/`RetrievalMethod`. Forcing X.509 + Detached in the FHIR
base spec narrows the standard without enabling any FHIR interop
behaviour, and use-case IGs that genuinely need those constraints
(e.g., government-grade signing) can still impose them via profiles.
The `srCms` sentence is plainly wrong for an XML-DSig section and
should be removed regardless of how the rest of the disposition is
handled. The replacement `SHOULD`-level mapping is consistent with
how the page already treats the equivalent JWS bullets and matches
the reporter's coordinated rewrites in FHIR-54025 / FHIR-54035 /
FHIR-53835, so accepting it preserves intra-page symmetry.

---

### Disposition B: Alternative Approach (Persuasive with Modification)

#### Proposal

Adopt the **substance** of the rewrite but with three modifications:

1. **Keep one `SHALL` on the copy/paste error fix only.** Delete the
   `srCms` sentence outright (it is a bug), but explicitly say
   "`Signature.type` SHALL be populated when the FHIR `Signature`
   element is used to carry an XML Digital Signature" — i.e., retain
   a minimum conformance hook so that `Signature` instances carrying
   XML-DSig payloads remain machine-classifiable. Everything else
   relaxes to `SHOULD` as proposed.
2. **Reconcile `Signature.targetFormat` with FHIR-54066 before merging.**
   FHIR-54066 flags that the DataTypes page describes
   `Signature.targetFormat` as a MIME type while the signatures page
   (and this rewrite) treat it as a canonicalization URL. Disposition
   of FHIR-54080 should be paired with a decision on FHIR-54066;
   either the DataTypes page is corrected to match (preferred) or
   the bullet here is reworded to describe the MIME-type usage and
   move the canonicalization-URL guidance into a separate bullet
   referencing the canon header that FHIR-54066 mentions.
3. **Pull the canonicalization-deletion into FHIR-53891.** Don't
   delete the duplicate sentence in this ticket; instead make the
   deletion a sub-task of FHIR-53891 so a single editorial pass
   guarantees the message is preserved exactly once on the page.

#### Justification

This keeps the reporter's main correction (drop unjustified `SHALL`s,
fix the `srCms` bug, restore base-spec flexibility) but avoids two
foreseeable problems. First, fully removing every `SHALL` leaves no
required conformance hook tying a `Signature` instance to its
underlying technology, which makes downstream profiling more painful
than necessary. Second, FHIR-54066 is an unresolved conflict between
two normative pages; merging FHIR-54080 without resolving it would
re-introduce that conflict in the rewritten text. Sequencing the
canonicalization deletion with FHIR-53891 prevents an editorial gap
where both copies of the sentence get removed.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Decline the rewrite as a single change. Acknowledge the `srCms`
copy/paste error and fix it as an editorial correction, but retain
the existing `SHALL` constraints on X.509 and Detached form on the
basis that (a) every known production deployment of XML-DSig with
FHIR uses X.509 + Detached, and (b) the FHIR Infrastructure WG has
historically chosen interoperability-by-restriction over
implementer-flexibility on signature mechanics (see the precedent
set by FHIR-47211 and FHIR-51143).

#### Justification

The reporter's argument is principled — base specs should not
over-constrain — but in the specific case of digital signatures the
combinatorial space of XML-DSig variants is large enough that
unrestricted optionality can render two compliant implementations
unable to verify each other's signatures. The current `SHALL`s
function as a de-facto interoperability profile baked into the base
spec; removing them shifts the burden onto every downstream IG to
re-state the same constraints. If the WG's standing position is that
signature interop trumps flexibility, declining is defensible.

---

### Recommendation

**Recommended disposition:** **B (Persuasive with Modification).**

Disposition A is directionally correct — the three `SHALL`s really
are technology-policy choices, the `srCms` sentence is unambiguously
a bug, and the rewrite restores symmetry with the JWS and CMS
sections that the same reporter is rewriting in lockstep
(FHIR-54025, FHIR-54035, FHIR-53835). Disposition C overstates the
interoperability cost: the FHIR `Signature` datatype is already a
thin wrapper that does not normalise across signature technologies,
so the X.509 / Detached `SHALL`s do not actually buy cross-vendor
interop in any meaningful way, and use-case IGs are the right place
to lock those choices down.

What pushes the recommendation off "pure A" is the open
reconciliation with FHIR-54066 on `Signature.targetFormat` and the
sequencing risk on the canonicalization-sentence deletion versus
FHIR-53891. Both are real and both will surface in WG review even if
A is otherwise persuasive. Modification B captures the rewrite the
reporter wants, fixes the `srCms` bug, and gates merge on resolving
the two cross-ticket dependencies — which is the cleanest path to a
self-consistent `signatures.html` after the full rewrite series
lands.
