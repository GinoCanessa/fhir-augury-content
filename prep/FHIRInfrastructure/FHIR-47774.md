## Summary

The ticket requests that the `HumanName` data type's HL7 v2 mapping be
expanded so it also documents the mapping to the v2 `XCN` (Extended
Composite ID Number and Name for Persons) data type. Today the v2 mapping
for `HumanName` only references `XPN` (Extended Person Name), which is the
correct v2 type for a patient's name (`Patient.name`). However, when
`HumanName` is used on `Practitioner.name`, the corresponding v2 structure
is the name portion of `XCN`, not `XPN`. The reporter — Linda Michaelsen —
asks that the mapping table include the `XCN` equivalents so implementers
doing v2↔FHIR translation for practitioners have explicit guidance.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
returned by FHIR Augury for this issue, so the scope is exactly as
described above: a documentation/mapping addition to the `HumanName`
data type in FHIR Core.

### Disposition A: Accept As Requested

#### Proposal

Add an `XCN` mapping alongside the existing `XPN` mapping on the
`HumanName` data type in FHIR Core. Concretely:

- Register an additional v2 mapping URI/identity on the `HumanName`
  StructureDefinition (or extend the existing `http://hl7.org/v2`
  mapping target) so each element can carry both an `XPN.*` and an
  `XCN.*` reference.
- Populate `mapping` entries on `HumanName` elements with their `XCN`
  counterparts:
  - `HumanName.family` → `XCN.2.1` (Family Name → Surname)
  - `HumanName.given` → `XCN.3` (Given Name) and `XCN.4` (Second and
    Further Given Names or Initials Thereof)
  - `HumanName.prefix` → `XCN.6` (Prefix, e.g. DR)
  - `HumanName.suffix` → `XCN.5` (Suffix, e.g. JR or III) and
    `XCN.7` (Degree, e.g. MD)
  - `HumanName.use` → `XCN.10` (Name Type Code)
  - `HumanName.period` → `XCN.19`/`XCN.20` (Effective/Expiration Date)
- Update the prose in the `HumanName` mapping section to explain that
  `XPN` is the v2 equivalent when `HumanName` appears on `Patient` (and
  similar non-provider contexts), while `XCN`'s name components are the
  v2 equivalent when `HumanName` appears on `Practitioner` /
  `PractitionerRole` / other provider contexts.
- Cross-link from `Practitioner.name`'s mapping note to the new `XCN`
  rows on `HumanName`.

#### Justification

This is precisely the technical correction the reporter requests, and it
is consistent with how FHIR documents v2 mappings elsewhere — multiple
v2 source paths often appear on the same FHIR element when a single
FHIR concept is reused across several v2 data types. `XCN` is
unambiguously the v2 carrier for a practitioner's name (per HL7 v2
Chapter 2A), so the absence of an `XCN` mapping on `HumanName` is a
genuine documentation gap that hampers v2-to-FHIR mapping work
(including the v2-to-FHIR IG). Adding the mapping is non-breaking: it
adds rows in a mapping table without changing element definitions,
cardinalities, terminology bindings, or invariants.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding `XCN` mappings on `HumanName` itself, document the
practitioner-name v2 mapping at the resource level on `Practitioner`
(and `PractitionerRole.practitioner` users such as `STF`/`PRA` segments)
and/or in the v2-to-FHIR Implementation Guide. Specifically:

- Leave `HumanName`'s mapping table focused on `XPN` (the canonical
  "person name" data type in v2).
- On `Practitioner.name`, add a mapping note that the v2 equivalent is
  the name portion of `XCN` (with the component-level breakdown listed
  above), and on `Practitioner.identifier` note `XCN.1` / `XCN.9` /
  `XCN.13`.
- Defer the full `XCN` ↔ FHIR mapping to the v2-to-FHIR IG, which
  already mediates between v2 segments (`STF`, `PRA`, `ROL`, `PRD`) and
  FHIR `Practitioner` / `PractitionerRole`.

#### Justification

`XCN` is a composite that combines a name *and* an identifier (plus
assigning authority, name context, etc.). Mapping all of `XCN` onto
`HumanName` is structurally awkward because several `XCN` components
(`XCN.1`, `XCN.9`, `XCN.13–15`) belong on `Practitioner.identifier`,
not on `HumanName` at all. Putting only the name-bearing `XCN`
components on `HumanName` risks implying that `HumanName` "covers"
`XCN` when it does not. Documenting the split at the resource level
(Disposition B) keeps `HumanName`'s mapping clean and pushes the full
composite mapping to the place that already handles v2 segment-level
context — the v2-to-FHIR IG. Trade-off: implementers reading
`HumanName` in isolation would still not see the `XCN` story, which is
the gap the reporter is flagging.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the v2-to-FHIR Implementation
Guide is the canonical home for v2 segment- and composite-level
mappings, and that the existing `XPN` row on `HumanName` is sufficient
for the data type itself. Close as "Not Persuasive — already addressed
by the v2-to-FHIR IG," with a pointer to the relevant `XCN` mapping in
that IG.

#### Justification

Defensible because the v2 mappings sprinkled through FHIR Core data
types are documentary aids, not the authoritative v2 conformance
artifact. The v2-to-FHIR IG is explicitly chartered to handle the full
combinatorial mapping problem (including `XCN`, `XPN`, `CNN`, `CWE`,
etc.) and is where implementers building v2 bridges should look.
However, this rationale is weak in practice: the same argument would
justify removing *all* v2 mappings from FHIR Core data types, which the
spec has not done. So Decline is the least defensible option here.

---

### Recommendation

**Recommended disposition:** A

Accepting as requested is the right call. The change is small,
documentation-only, non-breaking, and directly fills a real gap that
implementers hit whenever they map `Practitioner.name` to/from v2
`STF`/`PRA`/`ROL` segments. FHIR already mixes multiple v2 source data
types in `HumanName`-adjacent mappings elsewhere (e.g. `Address` ↔
`XAD`/`AD`), so adding an `XCN` row is consistent with house style.
Disposition B is a reasonable refinement and could be combined with A
(add the `XCN` rows on `HumanName` *and* add a resource-level pointer
on `Practitioner.name`); the workgroup may want to do both. Disposition
C should be rejected because the v2 mappings on FHIR Core data types
are an established convention and removing or refusing to extend them
inconsistently would be more confusing than helpful.
