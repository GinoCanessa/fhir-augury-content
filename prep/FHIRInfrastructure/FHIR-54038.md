## Summary

FHIR-54038 is a Technical Correction filed against the FHIR Core
specification (FHIR Infrastructure work group) noting that the `AllOk`
example for the `OperationOutcome` resource still uses the older
severity/code values and should be updated to use the newly introduced
`success` value for both `OperationOutcome.issue.severity` and
`OperationOutcome.issue.code`. The reporter's intent is purely
illustrative correctness — the example bundled with the resource page
should demonstrate the recommended pattern for a successful outcome
under the newer codes rather than continuing to use the legacy
`information` severity / `informational` code shown today. The
companion ticket [FHIR-54029](https://jira.hl7.org/browse/FHIR-54029)
("Clarify severity usage in OperationOutcome") is a related Change
Request from the same reporter that motivates the underlying work: it
asks the work group to clarify ambiguous severity values (notably
`fatal` vs `error`) and proposes a constraint that `severity` and
`code` both be `success` on a successful outcome. FHIR-54038 is the
example-only follow-on to the conceptual change tracked in
FHIR-54029.

## Proposed Dispositions

> Note: No GitHub repository was surfaced via cross-references for this
> ticket, so the dispositions below describe the intended FHIR Core
> change in spec-relative terms. The work group will route the change
> to the appropriate `HL7/fhir` authoring location when the disposition
> is applied.

### Disposition A: Accept As Requested

#### Proposal

Update the `AllOk` example instance for `OperationOutcome` so that the
single `issue` entry uses `severity = success` and `code = success`
(i.e., set `OperationOutcome.issue.severity` to `success` and
`OperationOutcome.issue.code` to `success`, replacing the current
`information` / `informational` pairing). No structural change to
`OperationOutcome` itself, and no value-set change is required — this
is purely an example update that depends on `success` already being a
valid code in the `IssueSeverity` and `IssueType` value sets bound to
those elements. Refresh the rendered example on the OperationOutcome
resource page (and any examples list referencing the `AllOk` example
ID) so the published narrative and JSON/XML/Turtle serializations all
show the new values.

#### Justification

The ticket is a Technical Correction, not a normative change: it asks
the published example to track the same recommendation that
[FHIR-54029](https://jira.hl7.org/browse/FHIR-54029) is formalising for
successful outcomes. Keeping the `AllOk` example aligned with the
recommended pattern is consistent with FHIR's general practice that
shipped examples should illustrate the preferred usage, particularly
for resources that implementers copy from. The change is low-risk and
backward-compatible: existing implementations that still emit
`information` / `informational` for a success continue to validate;
only the example narrative changes.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the example update as in Disposition A, but defer the change
until [FHIR-54029](https://jira.hl7.org/browse/FHIR-54029) is resolved
and (if accepted) the proposed constraint
`severity = success when code = success` is in place. Track FHIR-54038
as a dependent ticket, and at the same time refresh **all** stock
`OperationOutcome` examples (not just `AllOk`) so that every example
representing a successful outcome uses the `success` severity/code
pair, while error/warning/information examples continue to use their
existing values. Optionally add a short note in the OperationOutcome
narrative pointing implementers at the recommended pattern.

#### Justification

Updating only the `AllOk` example in isolation risks creating
inconsistency with the broader severity-clarification work being done
under FHIR-54029. Bundling the example refresh with that ticket means
the published spec changes once, in a coherent way, and gives the work
group an opportunity to ensure examples and (potential) constraints
stay in sync. The alternative also addresses the implicit risk that
some readers may treat the current `information` / `informational`
example as authoritative guidance for "what a successful outcome
looks like", since several published IGs derive their own examples
from the core resource page.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54038 as "Persuasive with Modification — Not Persuasive for
this ticket" with the rationale that the existing `AllOk` example
(using `severity = information`, `code = informational`) remains
valid, that both old and new values are members of the bound value
sets, and that the work group prefers to keep historical examples
stable to avoid churn in implementations that diff against published
artifacts. Direct the reporter's underlying concern to FHIR-54029 for
discussion of severity clarification.

#### Justification

A defensible "decline" is grounded in (a) the example as it stands is
not technically incorrect — `information` / `informational` is still a
valid representation of a successful outcome under the current value
set bindings — and (b) churn in published examples can cascade into
test suites and IG-derived examples. If the work group also declines
the constraint proposed in FHIR-54029, then there is no driver for
refreshing the example, and FHIR-54038 becomes redundant.

---

### Recommendation

**Recommended disposition:** B

The change requested in FHIR-54038 is small and clearly correct in
direction, but it is downstream of the substantive question raised in
[FHIR-54029](https://jira.hl7.org/browse/FHIR-54029). Updating the
`AllOk` example before that ticket settles risks either (i) shipping
an example that the work group later walks back, or (ii) leaving the
remaining `OperationOutcome` examples out of step with the new
guidance. Bundling the example refresh with the FHIR-54029 disposition
gives implementers a single, coherent set of changes — updated
examples plus (if accepted) a `severity = success when code = success`
constraint — and avoids piecemeal edits to the same resource page. If
FHIR-54029 is rejected, the work group can fall back to Disposition A
and still apply the example change on its own merits.
