## Summary

The reporter observed that on the FHIR R5 [Resource List page](https://hl7.org/fhir/resourcelist.html), the `DocumentReference` resource is listed under two distinct module groupings (per the attached screenshot, in both the Clinical → Diagnostics group and the Documents group). The reporter notes that `DocumentReference` is the only resource that appears in two groups and asks whether this is intentional or a mistake. The proposed remedy is a clarifying note at the bottom of the page explaining the duplication if it is intentional. No cross-referenced Jira tickets, Zulip threads, or GitHub items were found for this ticket.

## Details

**Description:**

> In the list of resources ([Resourcelist](https://hl7.org/fhir/resourcelist.html)—FHIR v5.0.0), DocumentReference appears in two groups (see screenshot). Since it's the only resource with this behavior, it's unclear if it's on purpose or a mistake. A note at the bottom of the page could help understand this caveat.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a brief clarifying footnote to `resourcelist.html` (the FHIR R5 Resource List page) stating that `DocumentReference` is intentionally cross-listed in both the *Clinical → Diagnostics* group and the *Documents* group because it is used both as the standard wrapper for clinical/diagnostic report documents (alongside `DiagnosticReport`) and as the general index resource for any document-style content (alongside `Composition`, `List`, `Library`). The note would appear at the bottom of the page as a single sentence such as: "Note: `DocumentReference` is the only resource intentionally listed in two groups, reflecting its dual role as a diagnostic report carrier and as a general document index."

#### Justification

This is the literal request and the lowest-cost intervention. It is a documentation-only change to a single rendered page, with no impact on schemas, validators, or downstream tooling. The reporter has correctly identified a real reader-confusion issue: when every other resource appears exactly once, an unexplained duplicate naturally reads as a typo. A short footnote preserves the existing dual-categorization (which conveys useful semantic information about the resource's two valid uses) while removing the ambiguity. Technical Correction tickets of this scope routinely settle on a documentation clarification.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) a footnote, render the duplicate `DocumentReference` entry on `resourcelist.html` with a visual marker — for example italic styling plus a tooltip/`title` attribute, or a small superscript dagger keyed to a single legend line — that explicitly flags the second occurrence as a deliberate cross-listing rather than a duplication bug. The page generator would emit the marker on any resource that appears in more than one module group, so the same mechanism would automatically cover any future cross-listed resources without requiring per-resource prose.

#### Justification

A footnote at the bottom of the page is easy to miss, especially for readers who jump directly to a specific section via anchor link. An inline visual marker localizes the explanation to the exact spot where the confusion arises and degrades gracefully (the page still reads correctly without the tooltip). Driving it from the page generator rather than hand-authored prose also avoids the maintenance hazard of having a footnote drift out of sync if the cross-listing set ever changes. The trade-off versus Disposition A is a small amount of generator work in the publication tooling versus a one-line edit to the page source.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Modifier" or "Not Persuasive" on the grounds that the dual listing is intentional, semantically meaningful, and self-evident in context: the page's section headings (*Diagnostics*, *Documents*) make clear which role is being highlighted in each group, and no normative spec text is affected. Take no editorial action.

#### Justification

This is defensible on the principle that the Resource List is an index/navigation aid, not a normative artifact, and that adding meta-commentary about the index's own structure tends to accrete noise over time. The reporter's confusion is real but localized to a single thoughtful reader; the duplication is discoverable in one click (the two entries link to the same page). Declining avoids setting a precedent that every navigational quirk in published spec pages requires an inline explanation.

---

### Recommendation

**Recommended disposition:** A

A one-sentence footnote on `resourcelist.html` resolves the ambiguity at essentially zero cost, with no risk to implementers, no schema impact, and no need to change the publication generator. Disposition B is a nicer end-state but is disproportionate to the size of the problem — investing in a generic "cross-listed" mechanism for a single resource is hard to justify, and the work spans the page generator rather than the spec source. Disposition C dismisses a legitimate documentation gap that the reporter took the time to surface; the fact that `DocumentReference` is the *only* resource exhibiting this behavior is precisely what makes a clarifying note valuable. Accept the ticket and add the footnote; if a future ticket adds a second cross-listed resource, the workgroup can revisit the generator-driven approach from Disposition B at that point.
