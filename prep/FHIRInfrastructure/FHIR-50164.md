## Summary

The ticket asks FHIR Infrastructure to add an invariant on the `Identifier`
datatype enforcing that whenever `Identifier.value` is a UUID (i.e., a
`urn:uuid:` value), `Identifier.system` MUST be `urn:ietf:rfc:3986`. The
request grew out of an `#ihe` Zulip discussion (see "Fixed system values in
MHD identifiers") in which Grahame Grieve stated that, in his reading of the
spec, a value such as `{"system":"http://abc.com","value":"urn:uuid:6eec…"}`
is invalid because UUIDs cannot be assigned to a sub-namespace. The reporter
disagrees with that interpretation, but having heard Grahame state it, has
filed this ticket to make the rule explicit through a machine-checkable
constraint rather than leaving it to prose interpretation.

The same Zulip thread references two earlier, already-resolved core-spec
tickets that cover essentially the same ground from the opposite angle:

- **[FHIR-45479](https://jira.hl7.org/browse/FHIR-45479)** ("Clarify use of
  namespaces for globally unique identifiers") — Applied, Persuasive with
  Modification. This change was specifically aimed at clarifying that
  **custom systems** for globally-unique values (including UUIDs) **are**
  permitted, i.e. the converse of what FHIR-50164 wants to lock down by
  invariant.
- **[FHIR-41478](https://jira.hl7.org/browse/FHIR-41478)** ("Clarify when
  OID and GUID identifiers, etc. must use RFC 3986 system and format") —
  Applied; explicitly superseded by / merged with FHIR-45479.

A reviewer therefore needs to read FHIR-50164 not in isolation but as an
attempt to re-open territory the workgroup very recently settled in
FHIR-45479, this time with a stronger (invariant-level) restriction.

## Related Jira Tickets

### [FHIR-45479: Clarify use of namespaces for globally unique identifiers](https://jira.hl7.org/browse/FHIR-45479)

Reporter Morten Ernebjerg argued the existing Identifier datatype
documentation was unclear on whether custom `Identifier.system` URLs
could be used for values that happen to be globally unique (including
UUIDs). The ticket was resolved **Persuasive with Modification** /
**Applied** by FHIR Infrastructure on 2025-02-11, with the outcome
landing as a clarification that **using globally-unique values
(including UUIDs) inside custom identifier namespaces IS allowed**, and
clarifying when `urn:ietf:rfc:3986` is in fact required (random URI
values prefixed with `urn:uuid:` / `urn:oid:`). This is directly
relevant — and largely in tension with — FHIR-50164, which wants to
machine-enforce the opposite rule.

Users involved: Morten Ernebjerg (reporter), Grahame Grieve (applier),
Elliot Silver, Jens Villadsen.

### [FHIR-41478: Clarify when OID and GUID identifiers, etc. must use RFC 3986 system and format](https://jira.hl7.org/browse/FHIR-41478)

Reporter Elliot Silver asked the spec to clarify when OID/GUID values
must use the `urn:ietf:rfc:3986` system and the `urn:oid:` /
`urn:uuid:` value format, and whether implementations or IGs may
declare uses of those values that fall outside that system. The ticket
was **superseded by FHIR-45479** and merged with its outcomes
(Persuasive with Modification, Applied 2025-02-11). It establishes that
the workgroup has already considered the precise question raised in
FHIR-50164 — and chose to clarify in prose rather than constrain via
invariant.

Users involved: Elliot Silver (reporter), Grahame Grieve (applier),
JIRAUSER24126.

## Related Zulip Discussions

### [#ihe > Fixed system values in MHD identifiers](https://chat.fhir.org/#narrow/stream/ihe/topic/Fixed%20system%20values%20in%20MHD%20identifiers)

A 180-message thread that is the genesis of this ticket. It begins as a
question from Rokas Grabys about why IHE MHD's `EntryUUID.Identifier`
profile fixes `system` to `urn:ietf:rfc:3986`, with John Moehrke and
Elliot Silver explaining that `EntryUUID` must be a UUID for XDS / XCA /
XDR interoperability and that fixing `system` follows the FHIR rule for
naturally-unique URIs. Jens Villadsen and Rokas Grabys push back that
the FHIR-level rule itself is too strong: a system that issues
identifiers happening to be UUIDs should be allowed to put its own URL
in `system`, both for searchability and to disambiguate multiple
`use=official` identifiers from different sources on a single
`DocumentReference`. Grahame Grieve states clearly that, in his reading
of the current spec, a value like
`{"system":"http://abc.com","value":"urn:uuid:…"}` is **not** valid —
"you can't assign a system to a subset of UUIDs that are random
values, so UUIDs should always be in `urn:ietf:rfc:3986`" — and that
solving the MHD-side concern (distinguishing `EntryUUID` vs `uniqueId`)
should be done through `Identifier.use` plus distinct search parameters
in MHD itself, not by relaxing the core rule. Jens disagrees on the
merits but accepts that, if Grahame's interpretation is normative, an
invariant should make it explicit — which is what this ticket
proposes. Daniel Vazquez Rivera briefly suggests `Identifier.assigner`
as the right hook for "which system issued this identifier"; Quentin
Ligier reads along but does not advance a position. **No consensus was
reached on the underlying interpretation** — the thread effectively
concludes with Jens promising to file a change proposal (this ticket).

Participants: Rokas Grabys, John Moehrke, Elliot Silver, Jens Villadsen,
Grahame Grieve, Daniel Vazquez Rivera, Quentin Ligier.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a FHIRPath invariant to the `Identifier` datatype (in
`build/source/identifier/identifier.xml` in the core spec authoring
tree) that fires whenever `Identifier.value` is a syntactic UUID and
requires `Identifier.system` to be `urn:ietf:rfc:3986`. Concretely
something like:

- `key`: `idr-1` (next free Identifier invariant id)
- `severity`: `error`
- `human`: "If Identifier.value is a UUID (matches the FHIR `uuid`
  primitive), Identifier.system must be `urn:ietf:rfc:3986`."
- `expression`:
  `value.exists() implies (value.matches('^urn:uuid:[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$') implies system = 'urn:ietf:rfc:3986')`

Apply across all maintained FHIR versions where `Identifier` is in
scope (R4 / R4B / R5 / R6 if the workgroup considers it a technical
correction). Update the Identifier prose section to reference the new
invariant and update the existing examples and the V3 mapping note
accordingly. Coordinate with the validator team so the new constraint
fires at validation time.

#### Justification

This codifies the interpretation Grahame stated in the Zulip thread
and removes ambiguity for implementers — the rule moves from prose
("naturally globally unique identifiers… SHALL be `urn:ietf:rfc:3986`")
to a machine-checkable invariant, which is the stated rationale of the
ticket. It also removes the gap exploited in the example
`{"system":"http://abc.com","value":"urn:uuid:…"}`, which today some
toolchains accept silently.

The justification is weak, however, on two grounds the workgroup will
need to weigh: (i) it directly contradicts the recently-applied
clarification in FHIR-45479 / FHIR-41478, which the workgroup adopted
specifically to **allow** custom systems with globally-unique values
including UUIDs; and (ii) it is a behavior change for normative R4 /
R5 content, so any production resource that today carries a non-RFC3986
system together with a `urn:uuid:` value would suddenly fail
validation — a non-trivial backwards-compatibility cost for a ticket
classified as a "Technical Correction".

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new invariant. Instead, treat FHIR-50164 as a
documentation/examples follow-up to FHIR-45479 by:

1. Adding an explicit non-normative example to the Identifier datatype
   page that shows a UUID value carried under a custom `system` (e.g.
   the company-product-ID case Morten Ernebjerg gave in FHIR-45479) and
   labels it as **conformant**, with a sentence explaining why.
2. Adding a second example showing the `urn:ietf:rfc:3986` +
   `urn:uuid:…` form and noting that this is the form to use **when no
   meaningful issuing system exists**.
3. Tightening the prose in the "naturally globally unique identifiers"
   paragraph so that the FHIR-45479 outcome is unmistakable: the URI
   form is required only when no custom system is being asserted, not
   whenever the value happens to be a UUID.
4. Cross-linking from MHD's `EntryUUID.Identifier` profile back to
   this section so IHE-style "system fixed to `urn:ietf:rfc:3986`"
   constraints are clearly profile-level decisions, not core-spec
   invariants.

#### Justification

This is consistent with the position the workgroup just adopted in
FHIR-45479 (Persuasive with Modification, Applied 2025-02-11), and with
Daniel Vazquez Rivera's and the reporter's own line in the Zulip
thread that custom systems issuing UUID-shaped values are legitimate.
It removes the genuine confusion that produced FHIR-50164 (Grahame's
strong reading vs. the FHIR-45479 outcome) without paying the
backwards-compatibility cost of a new error-level invariant on the
most widely-used datatype in the spec. It also keeps the
"`EntryUUID` / `uniqueId` disambiguation" problem where it actually
lives — in MHD's profiling — rather than punching a hole in core
that all profiles inherit.

---

### Disposition C: Decline

#### Proposal

Resolve **Not Persuasive** on the basis that the underlying question
has been recently addressed by FHIR-45479 (and its predecessor
FHIR-41478), which the workgroup applied as **Persuasive with
Modification** on 2025-02-11. The clarified prose explicitly permits
custom systems for globally-unique values including UUIDs, which is
incompatible with the requested invariant. Note in the resolution that
the disambiguation use case driving the request (`EntryUUID` vs
`uniqueId` in MHD) is properly an IHE MHD profiling concern and not a
core FHIR Identifier constraint.

#### Justification

A core-datatype invariant whose effect is to **forbid** a use that the
workgroup has very recently and deliberately documented as permitted
would be self-contradictory and would re-open closed ground. It is also
the most disruptive option for existing R4 / R5 deployments. The
ticket's own reporter is on record opposing the underlying rule on the
merits and only proposing the invariant as a way to force the
question; the workgroup is entitled to answer "we already did, in
FHIR-45479, and the answer was the other way." The ticket can point
implementers at FHIR-45479 for the rationale.

---

### Recommendation

**Recommended disposition:** B

Disposition C (decline outright) is defensible on the closed-ground
argument, but it leaves intact the very ambiguity that produced this
ticket: a senior figure (Grahame Grieve) is on record in `#ihe` saying
`{"system":"http://abc.com","value":"urn:uuid:…"}` is invalid, while
the FHIR-45479 outcome says custom systems for UUID-shaped values are
allowed. Implementers will keep filing variants of this ticket until
the spec clearly resolves the contradiction. Disposition A would
resolve it, but in the wrong direction — it contradicts the
workgroup's own February 2025 decision and breaks existing instances.
Disposition B settles the ambiguity in the direction the workgroup
already chose, in a backwards-compatible way (added prose and
examples, no new error-level invariant), and pushes the MHD-specific
"distinguish EntryUUID from uniqueId" concern back to IHE where
Grahame and John Moehrke had already indicated it belongs. That is
the cleanest path through the disagreement on the Zulip thread.
