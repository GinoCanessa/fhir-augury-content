## Summary

The reporter (Justin Ramm) is asking the FHIR Infrastructure work group to
clarify the relationship between the newly-defined `limited-filter`
`OperationOutcome.issue.code` value and the long-standing Bundle self-link
guidance at <https://build.fhir.org/search.html#selflink>. Both mechanisms
appear to let a server tell a client that the search it actually performed
differs from the search the client requested — `limited-filter` by raising
an informational/warning issue in an OperationOutcome, and the self link by
returning the literal URL of the search the server actually executed. The
ticket asks the work group to spell out which mechanism should be preferred,
how they differ, and when each is appropriate, so that server implementers
have unambiguous guidance and clients can rely on a consistent signal.

For background, `limited-filter` itself was added as part of
[FHIR-37993](https://jira.hl7.org/browse/FHIR-37993) ("Search parameters
that are only supported within a specific category", resolved Persuasive
with Modification) — that ticket added guidance to the Search page and
introduced the new issue-type code so a server could explicitly announce
that one or more requested search parameters were not (fully) honoured.
FHIR-54031 is the natural follow-on: now that the code exists, the spec
should explain how it composes with the existing self-link mechanism that
ostensibly covers similar ground.

## Details

## Proposed Dispositions

> Note: Because there are no GitHub cross-references and therefore no
> repo briefings to draw on, the dispositions below describe the change in
> spec terms (Search page narrative + the issue-type CodeSystem +
> OperationOutcome guidance). Implementers landing the change will, per the
> normal FHIR Core authoring path, edit the relevant `source/search.html`
> (or its `pages.xml`-driven equivalent) and the `issue-type` CodeSystem
> resource in the build.

### Disposition A: Accept As Requested

#### Proposal

Add a normative-style clarification block to `search.html` that explicitly
contrasts `OperationOutcome.issue.code = limited-filter` against the
Bundle `self` link, and add an aligned note to the definition of the
`limited-filter` concept in the `issue-type` CodeSystem. Concretely:

1. In the search page, in (or immediately following) the existing
   "Self Link" section (`search.html#selflink`), add a sub-section titled
   "Self link vs. `limited-filter`" stating:
   - The Bundle `self` link **MUST** always reflect the search the server
     actually executed (i.e., the URL a client could re-issue to obtain
     the same logical result set). This is the canonical, machine-actionable
     signal of what the server did.
   - When the executed search differs from the requested search because
     one or more requested parameters were not (fully) applied — for
     example the parameter is unsupported, is only supported for a subset
     of the resource's categories, or was silently ignored — the server
     **SHOULD** also include an `OperationOutcome` (in `Bundle.entry`
     of type `searchset`, with `search.mode = outcome`, per existing
     guidance) carrying one `issue` per affected parameter with
     `issue.code = limited-filter`, `issue.severity = warning`, and
     `issue.expression`/`issue.location` identifying the parameter(s)
     that were dropped or narrowed.
   - The two mechanisms are complementary, not alternatives: the self
     link is the authoritative *what was run*; the `limited-filter`
     issues are the human/diagnostic *why what you asked for and what we
     ran differ*. Clients that only inspect the self link remain
     conformant; clients that want to surface a warning to a user or to
     decide whether to retry should also inspect the OperationOutcome.
2. In the `issue-type` CodeSystem, extend the `definition` of
   `limited-filter` with a one-sentence pointer back to
   `search.html#selflink` and a note that issuing this code does not
   relieve the server of the obligation to return an accurate self link.

#### Justification

This is exactly what the reporter asked for and is the lowest-risk path:
it is purely editorial relative to the published artifacts, it does not
change any conformance bits, and it preserves the existing investment in
both mechanisms. It also aligns with the disposition of the originating
ticket [FHIR-37993](https://jira.hl7.org/browse/FHIR-37993), which
explicitly noted that self links alone cannot express
"this parameter applied to subset X but not to subset Y" — that gap is
the whole reason `limited-filter` exists, and the spec should say so. The
"self link is authoritative, OperationOutcome is diagnostic" framing is
already implicit in how `Bundle.entry.search.mode = outcome` is used
elsewhere, so the clarification only has to write down a relationship the
spec is already living with.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying need for clarification, but rather than only adding
prose to `search.html`, also strengthen the `OperationOutcome` profile /
guidance for searchset bundles so that the relationship is structurally
visible and tooling can validate it:

1. Apply the prose change from Disposition A.
2. Additionally, define (or update) the searchset-`OperationOutcome`
   guidance in `search.html` and on the `OperationOutcome` resource page
   so that a `limited-filter` issue **SHALL** populate
   `issue.expression` (FHIRPath) or `issue.location` (legacy) with the
   parameter name(s) that were dropped/narrowed, and **SHOULD** populate
   `issue.diagnostics` with a human-readable explanation. Add a worked
   example to `search.html` showing a request, the resulting self link
   (with the dropped parameter omitted or with its value normalised),
   and the accompanying OperationOutcome carrying one `limited-filter`
   issue per affected parameter.
3. Optionally, file a follow-on ticket to add a "non-normative best
   practice" invariant to the `OperationOutcome` profile guidance for
   searchset bundles (no new constraint on the base resource — any
   constraint would live in a search-bundle-scoped guidance block).

#### Justification

Pure prose (Disposition A) leaves the relationship ambiguous on the wire:
implementations may emit `limited-filter` without enough structured
information for a client to know *which* parameter was limited, which
defeats the purpose of having two complementary mechanisms. Requiring a
populated `issue.expression`/`issue.location` plus a worked example makes
the contrast machine-checkable and dramatically reduces the chance that
implementers re-litigate the same question. The cost is modest: it is
still editorial within FHIR Core and adds at most a `SHALL`/`SHOULD`
clause and an example, with no schema or CodeSystem-membership change.

---

### Disposition C: Decline

#### Proposal

Mark as "Not Persuasive — Already Specified". Take the position that
`Bundle.link` (`self`) and `OperationOutcome.issue.code = limited-filter`
serve clearly different purposes — the self link is a re-runnable URL,
the OperationOutcome is a diagnostic message — and that the existing
narrative on `search.html#selflink`, plus the published definition of
`limited-filter` in the `issue-type` CodeSystem, are jointly sufficient.
Recommend no spec change and close the ticket with an explanatory
comment pointing to those two locations.

#### Justification

A defensible decline rests on the argument that the two mechanisms are
already documented in their respective homes and that the FHIR spec does
not generally cross-reference every code in every CodeSystem against
every adjacent narrative. Declining keeps the spec text smaller and
avoids creating the impression that every new issue-type code needs a
bespoke contrast block.

The weakness of this disposition is that it does not actually answer
the reporter's question — "which should be preferred and/or how are
they different?" — and the originating ticket
([FHIR-37993](https://jira.hl7.org/browse/FHIR-37993)) explicitly
documents that the self link cannot express partial / category-scoped
parameter handling, which is *the* gap `limited-filter` was created to
fill. Closing this ticket without writing that down invites the same
question to be raised again.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the smallest correct fix and would unblock the
reporter, but it leaves a known structural gap: nothing in the spec
ties a `limited-filter` issue to the specific parameter that was
limited, which is precisely the information a self link cannot convey
and therefore the entire reason the new code exists. Disposition B is
only marginally more work — it is still editorial, lives entirely in
`search.html` and the `OperationOutcome`/`issue-type` guidance, and
requires no schema change — but it produces guidance that is both
human-readable *and* machine-checkable, with a worked example that
implementers can copy. Disposition C is not defensible: the originating
ticket [FHIR-37993](https://jira.hl7.org/browse/FHIR-37993) already
established that the self link is insufficient on its own, so the spec
owes implementers a concrete statement of how the two mechanisms compose.
The work group should adopt Disposition B, assign it to the same author
who landed `limited-filter` (Gino Canessa per FHIR-37993), and treat the
worked example as the acceptance criterion.
