## Summary

The ticket asks the FHIR Infrastructure work group to clean up an
inconsistent element-naming pattern that appears on participant-style
backbone elements in three definitional resources. Today
`ActivityDefinition.participant` carries a primary `type` element
(`code`, 0..1) plus two sibling elements `typeCanonical` (`canonical`,
0..1) and `typeReference` (`Reference`, 0..1). The shared `type*`
prefix makes those siblings *look* like the members of a `type[x]`
choice even though they are not, which can confuse implementations that
key off the `[x]` naming convention. The reporter proposes that for
R6 the work group remove `typeCanonical` and `typeReference` (both of
which were only introduced in R5, so the regression risk is limited to
R5 adopters) and replace them with a properly-named choice
`detail[x]` of `canonical | Reference`, carrying forward the existing
definitions and constraints. The same three-element pattern exists at
`PlanDefinition.actor.option.type`, `PlanDefinition.action.participant.type`,
and `RequestOrchestration.action.participant.type`, and the ticket asks
that the same fix be applied uniformly across all four locations. No
related Jira tickets were surfaced by cross-references, and there are
no linked GitHub items.

## Details

**Description:**

The ActivityDefinition resource defines the following elements:

- ActivityDefinition.participant.type: 0..1, code
- ActivityDefinition.participant.typeCanonical: 0..1 canonical
- ActivityDefinition.participant.typeReference: 0..1 Reference

The names of these elements are problematic in that `typeCanonical` and
`typeReference` appear to be choices of `type`, but they are not.
Although not technically incorrect, this pattern can cause issues with
implementations that make use of type name conventions as part of
element processing, and the pattern is discouraged.

To better align with recommended methodology, we should:

1. Remove `typeReference` and `typeCanonical` (they were introduced in R5).
2. Add `detail[x]` (canonical | Reference) (with the definition and
   constraints as specified in the `typeCanonical` and `typeReference`
   elements).

Note that this same pattern appears in:

- PlanDefinition.actor.option.type
- PlanDefinition.action.participant.type
- RequestOrchestration.action.participant.type

We should apply the same updates to each of those locations.

This is the conversation in which the change request originated. Gino
Canessa raised the issue on the FMG stream after spending time on
ActivityDefinition and being misled by the `type` / `typeCanonical` /
`typeReference` naming, and observed that the normative rules in
principle allow turning `participant.type` into a choice type later —

the work group both discourage the pattern and add a rule that
elements whose names already imply a type (e.g., `typeCanonical`)

uniformly to ActivityDefinition, PlanDefinition, and
RequestOrchestration. Bryn then filed FHIR-55632 capturing that
proposal. The thread reflects clear agreement that the pattern is
problematic; no objections were raised, and the only open question
implicit in the discussion is the broader methodology rule about
disallowing later promotion of suffixed elements to choice types,

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the change exactly as the reporter described, in R6, at all four
locations identified in the cross-references and description:

1. **ActivityDefinition.participant** — remove
   `participant.typeCanonical` (0..1 canonical) and
   `participant.typeReference` (0..1 Reference); add
   `participant.detail[x]` of types `canonical | Reference` (0..1),
   carrying over the existing element definition text, short
   descriptions, comments, and any invariants/binding from the
   removed elements. Retain `participant.type` (0..1, code) as-is.
2. **PlanDefinition.actor.option** — apply the same removal/add at
   `actor.option.typeCanonical`, `actor.option.typeReference`, adding
   `actor.option.detail[x]` (canonical | Reference). Retain
   `actor.option.type` (code).
3. **PlanDefinition.action.participant** — apply the same
   removal/add at `action.participant.typeCanonical`,
   `action.participant.typeReference`, adding
   `action.participant.detail[x]` (canonical | Reference). Retain
   `action.participant.type` (code).
4. **RequestOrchestration.action.participant** — apply the same
   removal/add at `action.participant.typeCanonical`,
   `action.participant.typeReference`, adding
   `action.participant.detail[x]` (canonical | Reference). Retain
   `action.participant.type` (code).

Tag the four removed elements with R5→R6 deletion notes in the
maturity / changes pages, and add a R5-to-R6 conversion note pointing
adopters to the new `detail[x]`. Confirm whether either of the
removed elements participates in any current invariant, search
parameter, or compartment definition; carry forward or rework as
needed.

#### Justification

The proposal is internally consistent, scoped to four sibling
locations that share one pattern, and limits the regression surface to
R5-only behaviour (these elements did not exist in R4). Replacing the
ad-hoc `typeCanonical` / `typeReference` siblings with a real
`detail[x]` choice removes a known foot-gun for tooling that keys off
the `[x]` naming convention (StructureDefinition snapshot generators,
profilers, code generators, FHIRPath/element walkers), aligns the
three resources with the FHIR methodology guidance the FMG
conversation referenced, and avoids the long-term hazard the reporter
flagged: that `participant.type` could in principle be promoted to a
choice later and silently collide with the existing siblings. The
symmetric application across ActivityDefinition, PlanDefinition, and
RequestOrchestration is required to keep the pattern coherent across
the definitional-workflow resource family.

---

### Disposition B: Alternative Approach

#### Proposal

Achieve the same de-naming objective without renaming the payload
elements, by **renaming only the suffixed elements** rather than
introducing a `detail[x]` choice. Concretely, in R6:

- Rename `participant.typeCanonical` → `participant.profile`
  (canonical, 0..1).
- Rename `participant.typeReference` → `participant.actor`
  (Reference, 0..1).
- Add an invariant at the `participant` backbone forbidding both
  `profile` and `actor` from being populated simultaneously, mirroring
  the de-facto exclusivity that exists between `typeCanonical` and
  `typeReference` today.
- Apply the same renames at the three matching sites in
  PlanDefinition (`actor.option`, `action.participant`) and
  RequestOrchestration (`action.participant`).

Element definition text, bindings, and constraints from the removed
elements would be carried forward to the renamed elements unchanged.

#### Justification

This alternative removes the offending `type*`-prefixed names — and
therefore removes the choice-type look-alike problem — without
introducing a new `[x]` choice on a backbone that historically had
none. Two scalar elements with a mutual-exclusion invariant are
usually easier for code generators and validators to handle than a
choice element, and the names `profile` (for the canonical pointer to
an `ActivityDefinition` / `PlanDefinition` / etc.) and `actor` (for
the direct reference to a participant) describe the *role* of each
element more clearly than `detail[x]` does. The migration cost for R5
adopters is comparable to Disposition A (two element renames vs. two
deletions plus one addition), but the resulting model is arguably
more readable. Trade-off: it diverges from the reporter's preferred
shape and gives up the closed `canonical | Reference` semantics that
a real choice type encodes in `StructureDefinition.snapshot`.

---

### Disposition C: Decline

#### Proposal

Do not change the resource shape. Instead, document the existing
`type` / `typeCanonical` / `typeReference` triplet on
ActivityDefinition, PlanDefinition, and RequestOrchestration as an
intentional historical pattern, add a methodology note discouraging
new uses of `type<Suffix>` element naming for non-choice elements,
and add an explicit rule (in the FHIR methodology / formats pages)
that an existing scalar element whose name *looks like* a choice
suffix may not later be promoted into a real `[x]` choice. No element
additions, removals, or renames are made.

#### Justification

The current shape, while ugly, is not technically incorrect, has
existed since R5, and is in active use. Removing `typeCanonical` and
`typeReference` is a breaking change for every R5-conformant
implementer and instance, and a `detail[x]` rename also forces
profile, IG, terminology, and tooling churn across all consumers of
these three resources. The underlying risk (a future promotion of
`participant.type` to a choice silently colliding with the siblings)
can be eliminated by a methodology rule alone, without disturbing
deployed artefacts. If the only concrete complaint is that the
naming is confusing, a strongly-worded methodology note plus a
warning paragraph on each affected backbone may be a more
proportionate response than a structural change.

---

### Recommendation

**Recommended disposition:** A

The ticket targets a small, bounded set of four sibling locations in
three definitional resources, all of which share an identical pattern
that the reporter, the FMG discussant who first raised it, and the
work-group commenter on Zulip independently agreed is problematic.
Disposition A removes that pattern in the only window where the cost
is contained — the R5→R6 boundary, before further adoption locks the
shape in — and replaces it with a properly-formed `detail[x]` choice
that matches the FHIR `[x]` convention the reporter cites and that
implementers' tooling already understands. Disposition B is a
defensible second choice and should be considered if the work group
prefers to avoid introducing a new `[x]` choice on these backbones,
but it loses the closed `canonical | Reference` semantics that the
choice form gives for free. Disposition C leaves a known, agreed-upon
methodology smell in three normative-track resources and relies on
prose to prevent a problem the structure itself can prevent; given
that the originating Zulip thread shows no defenders of the status
quo, the work group should take the structural fix while the R5→R6
window is open.
