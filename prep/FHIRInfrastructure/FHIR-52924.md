## Summary

The ticket asks the FHIR Infrastructure work group to update the `basic-example`
(referral) example for the `Basic` resource so that it stops using the
`basic-resource-type` CodeSystem from THO. The motivation is a parallel effort
(see [FHIR-52925](https://jira.hl7.org/browse/FHIR-52925), which has already
been resolved as Persuasive) to deprecate or outright retire the
`basic-resource-type` CodeSystem in THO, on the grounds that the code system —
originally a placeholder for "resources not yet supported by FHIR" — is
obsolete and should not continue to ship. Because the FHIR Core `Basic`
example currently references that code system, the THO retirement cannot be
cleanly executed until the example is changed; this ticket exists to handle
the FHIR Core side of that cleanup. A separate UTG ticket
([UP-734](https://jira.hl7.org/browse/UP-734)) tracks the THO/UTG-side change.

## Details

**Description:**

> We want to remove the basic-resource-type code system from THO with extreme

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FHIR Core build, modify the `basic-example` (the referral example
for the `Basic` resource) so that `Basic.code` no longer binds to the
`http://terminology.hl7.org/CodeSystem/basic-resource-type` code system.
Replace the existing `Coding` (system =
`http://terminology.hl7.org/CodeSystem/basic-resource-type`, code =
`referral`) with an example coding that is appropriate for a referral —
either a SNOMED CT code (e.g., `3457005 |Patient referral|` from
`http://snomed.info/sct`), an example-only local code in a clearly
example-scoped system, or a plain `text`-only `CodeableConcept` with
`text = "Referral"`. Regenerate the example artefacts (XML/JSON/Turtle)
and update any narrative or registry that references the example.
Coordinate with [FHIR-52925](https://jira.hl7.org/browse/FHIR-52925) /
[UP-734](https://jira.hl7.org/browse/UP-734) so the THO retirement of
`basic-resource-type` lands after this example has been updated.

#### Justification

This is exactly what the ticket and the Zulip thread ask for, and it is
the only outstanding blocker for the already-Persuasive
[FHIR-52925](https://jira.hl7.org/browse/FHIR-52925). Examples in the
FHIR Core spec should not bind to a CodeSystem that the same
organisation is actively retiring, both because it perpetuates a stale
artefact and because broken / dangling example bindings undermine
implementer trust. Replacing with a SNOMED CT referral concept (or a
narrative-only `CodeableConcept`) is consistent with how other
`Basic`-style examples illustrate "any concept can go here" without
recommending a particular code system. The change is local to one
example file and its generated outputs, so risk is minimal.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than just swapping the code in the existing referral example,
replace the example outright with a more illustrative use case for
`Basic` — one whose `Basic.code` draws from an external, real-world
terminology that the community would recognise (e.g., a SNOMED CT
clinical concept that genuinely has no dedicated FHIR resource), and
add a short narrative note in the example's introduction explaining
that `Basic` is intended for concepts that have no dedicated resource,
and that implementers should bind `Basic.code` to a domain-appropriate
external code system rather than to a FHIR-/THO-curated list. As part
of the same change, audit the rest of FHIR Core for any other
references to `basic-resource-type` (search examples, profiles,
narrative) and remove them in the same commit. The mechanical removal
of the offending coding is still done, but the opportunity is taken to
improve guidance.

#### Justification

The underlying problem is not just one bad example — it is that the
existence of `basic-resource-type` implies FHIR-I curates a list of
"things that should have been resources but aren't". Retiring the code
system without replacing the implicit guidance leaves implementers
asking "what should I bind `Basic.code` to?". A better example plus a
short narrative note answers that question and reduces the chance of
similar tickets recurring (e.g., implementers re-inventing the same
list as local code systems). The cost is small — a few extra lines of
narrative — and it maximises the benefit of touching the example at
all.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the basis that the example is harmless even if
the `basic-resource-type` CodeSystem is deprecated: example content is
illustrative, not normative, and a deprecated-but-still-resolvable
CodeSystem URL in an example does not break conformance for any real
implementer. Close FHIR-52924 as "Not Persuasive" and let
[FHIR-52925](https://jira.hl7.org/browse/FHIR-52925) deprecate (rather
than retire) the code system so historical references continue to
resolve.

#### Justification

This is defensible only under the narrow reading that examples are
non-normative and that THO can keep retired/deprecated code systems
resolvable indefinitely. It minimises FHIR Core churn at the cost of
leaving a stale, FHIR-I-owned code system in the published spec's
example surface. It also contradicts the consensus on the Zulip thread
(Grahame: "should be deleted ASAP"; Marc, Paul, Lloyd in agreement),
and it forces [FHIR-52925](https://jira.hl7.org/browse/FHIR-52925) into
"deprecate, don't retire" territory, which the resolver has explicitly
left open ("either deprecated or removed, as appropriate"). Listed for
completeness rather than as a serious option.

---

### Recommendation

**Recommended disposition:** A

The ticket is small, surgical, and directly enables an already-approved
companion change ([FHIR-52925](https://jira.hl7.org/browse/FHIR-52925),
resolved Persuasive) plus a queued UTG action
([UP-734](https://jira.hl7.org/browse/UP-734)). The Zulip discussion
shows clear FHIR-I consensus in favour of doing this, with no
dissenters. Disposition B is appealing but expands scope (narrative
guidance, repo-wide audit) beyond what the reporter actually asked for
and risks delaying the THO retirement that is genuinely waiting on this
change; the broader "what should `Basic.code` bind to" guidance can be
filed as its own ticket if FHIR-I wants to pursue it. Disposition C
contradicts the in-thread consensus and leaves a known-bad reference in
the example surface for no real benefit. Accept as requested, replace
the offending coding in `basic-example` (a SNOMED CT referral concept
or a `text`-only `CodeableConcept` are both acceptable), regenerate the
example outputs, and coordinate the merge with the THO/UTG side so the
code-system retirement can proceed without dangling references.
