## Summary

The ticket reports a metadata error in the FHIR Core specification: the
"Datatypes" page is currently attributed to the **Modeling and Methodology
(M&M)** work group, but according to the reporter the correct owner is
**FHIR Infrastructure (FHIR-I)**. It is filed as a Technical Correction
against FHIR Core under the FHIR Infrastructure work group, and is
intended to be a small, mechanical fix to bring the page's published
ownership into line with the work group that actually maintains the
data type infrastructure. No related Jira tickets, Zulip threads, or
GitHub items were surfaced via cross-references or content search, so
the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the work-group attribution of the FHIR Core "Data Types" page
from **Modeling and Methodology (M&M / fhir)** to **FHIR Infrastructure
(fhir-i)**. Concretely, in the FHIR Core authoring source update the
work-group metadata that drives the page's published "Work Group"
header / footer for `datatypes.html` (and any sibling pages that share
the same wg tag, e.g., the data types intro/index pages) so that the
page renders as owned by FHIR-I. No content changes; metadata only.

#### Justification

The reporter (Sarah Gaunt) is asserting a factual error about
ownership, and the ticket is filed as a Technical Correction against
FHIR Infrastructure — meaning FHIR-I itself is willing to take
ownership. The operational responsibility for the data types
infrastructure (datatypes machinery, primitive types, general-purpose
types) sits with FHIR-I in current practice, so aligning the page
attribution with that reality is straightforward and consistent. As a
metadata-only correction it carries no normative or behavioural risk
and has no implementer impact beyond the published "Work Group"
attribution on the page.

---

### Disposition B: Alternative Approach

#### Proposal

Before flipping the attribution, perform a short audit: identify every
page currently tagged to **M&M (`fhir`)** that is, in practice, owned
by **FHIR-I (`fhir-i`)** (and vice-versa), and change them all in a
single coordinated PR. At minimum, check the data types index, the
individual primitive type pages (`string.html`, `code.html`, etc.), the
general-purpose type pages (`Quantity`, `CodeableConcept`,
`Identifier`, …), and the data types extensibility / extensions pages.
Coordinate the resulting list with M&M co-chairs before merging so that
both work groups sign off on the new boundary.

#### Justification

A point fix to one page risks leaving inconsistent attribution across
pages that logically belong to the same owner — the data types
machinery is rarely a single page. A coordinated audit also avoids
re-opening the same question ticket-by-ticket if other data types
pages turn out to have the same problem, and it gives M&M a chance to
object to or refine the boundary rather than learning about it after a
silent metadata change. The trade-off vs. Disposition A is that it is
strictly more work and requires inter-WG coordination, but it produces
a stable answer rather than a one-off correction.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave the data types page attributed to M&M.
Rationale: ownership of cross-cutting foundation pages has historically
sat with M&M because the modelling rules for data types are normative
M&M output, even when day-to-day editorial maintenance happens
elsewhere.

#### Justification

If M&M considers the data types modelling rules to be part of its
remit (and not just FHIR-I's infrastructure), then the page header
correctly reflects the *normative* owner of the content, even if FHIR-I
provides editorial support. Changing the attribution without M&M
sign-off could be read as transferring normative responsibility, which
is more than a "technical correction." Declining keeps the status quo
until the two work groups jointly agree on the boundary.

---

### Recommendation

**Recommended disposition:** A

This is a Technical Correction filed *by* FHIR Infrastructure against
its own work-group attribution, the reporter is unambiguous about the
desired state, and the change is metadata-only with no implementer
impact. Disposition A delivers exactly that with the smallest possible
footprint. Disposition B's audit is a worthwhile follow-up but should
not block this fix — it can be opened as a separate ticket once A is
in. Disposition C only makes sense if M&M actively objects, and there
is no evidence of that on the ticket, in linked tickets, or in any
Zulip thread the search surfaced; absent such an objection, defaulting
to "leave it wrong" is the worst of the three options. The workgroup
should accept as requested, confirm that the change lands on the
correct authoring file in `HL7/fhir` (most likely the `datatypes`
page's wg metadata), and close the ticket as a Persuasive / Applied
technical correction.
