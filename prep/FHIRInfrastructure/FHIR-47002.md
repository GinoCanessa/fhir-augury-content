## Summary

Lloyd McKenzie is requesting that the FHIR Core specification provide
clearer, normative-quality guidance about how validators should populate
validation responses — specifically `OperationOutcome` issues returned
from validation (e.g., the `$validate` operation, the IG Publisher /
java validator, or any conformance tool). The core ask is to disambiguate
three things that today all surface through `OperationOutcome.issue.severity`:

1. **Issues with the instance** ("the resource is broken").
2. **Issues that the validator could not electronically check** (e.g.,
   missing terminology, unresolvable profile, code from an unknown
   `CodeSystem` against an extensible binding).
3. **Issues with the validation process itself** (the validator failed
   internally, so even the things it normally catches were not actually
   checked).

Lloyd proposes a precise reading of `error` / `warning` / `information`
in the validation context: **error** = definitely invalid (cardinality,
invariants, vocabulary against required bindings); **warning** =
possibly invalid but the validator cannot prove it (extensible-binding
mismatch with unknown system, unresolvable declared profile, etc.);
**information** = does not affect validity but may indicate something
worth a human's attention (best-practice notes, unsupported elements,
slicing diagnostics). For conformance testing he frames these as
"definite failure", "potential failure", and "ignorable but useful".

There is **one** related discussion: a Zulip thread in `#fmg`
("Do you remember...") opened by Lloyd searching for an old categorisation
spreadsheet of validator messages. That thread surfaced two important
counter-points from Grahame Grieve: (a) today's validator already
overloads `error` for unexpected validator-internal process failures
(server didn't respond, etc.) on top of true non-conformance, and uses
`warning` for SHOULD rules, security/implementation considerations,
warning-level invariants, and best-practice rules; and (b) the existing
formal definitions of `error`/`warning` in `IssueSeverity` are
operation-centric ("severe enough to cause the action to fail"), which
Lloyd points out is the wrong frame for *validation results*, where the
question is "is the instance valid", not "did the validator finish".
Grahame agreed that the lack of validation-specific definitions is
"evidently a problem". Josh Mandel separately asked (in FHIR-I) for
this to be paired with a more complete treatment of how the top-level
severities combine with specific `OperationOutcome.issue.code` /
`details` codes so the result is interoperable, not just human-readable.

There are **no** cross-referenced Jira tickets and **no** related
GitHub PRs/issues/commits in the FHIR Augury index for FHIR-47002.

## Details

**Description:**

> We should provide more guidance about expectations for validation
> results.  Specifically, the need to differentiate between "issues
> with the instance" and "issues with the validation process".
> Implementers need to understand when they see a result whether it
> means "this instance is definitely broken" as opposed to "there
> might be an issue here that the validator can't electronically
> check" vs. "the validation process outright failed and even issues
> that would normally be caught haven't been checked."
>
> In particular, we should be clear about the meaning of error vs.
> warning vs. information when we're talking about validation.
>
> My opinion:
>
> **Error** — An issue that shows that the instance is definitely
> invalid. (Not met cardinality, invariants, vocab validation)
> **Warning** — An issue that shows that the instance might be
> invalid but the validator can't tell for sure (non-met extensible
> binding, declared profile the validator can't find, declared code
> outside a binding where the system is unknown, etc.)
> **Information** — An issue that doesn't impact the validity of the
> issue, but may still indicate something that could be a problem or
> could be better, or that provides guidance to help understand other
> errors or warnings in the instance (e.g. elements in an instance
> that aren't expected to be supported, violated best practice
> recommendations (if the validator hasn't been set to treat those as
> errors), elements that didn't match any slices during open slicing,
> slicing information, etc.)
>
> For conformance testing, an error indicates a definite failure.  A
> warning indicates a potential failure.  Information messages can be
> ignored, but they're useful to authors of examples or when
> diagnosing systems.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new "Validation results" subsection to the FHIR Core validation
guidance (alongside the existing `$validate` operation page and the
`OperationOutcome` resource page) that gives **validation-specific**
definitions of `OperationOutcome.issue.severity` for results returned
by a validator (as opposed to operation-level errors):

- **`error`** — The validator has positive evidence that the instance
  does not conform: failed cardinality, failed invariant (severity
  `error`), failed required-binding terminology check, structural
  parse error, fixed/pattern mismatch, etc.
- **`warning`** — The validator could not prove conformance. Includes
  unmet extensible bindings, codes from an unknown `CodeSystem`,
  declared profiles or extensions the validator cannot resolve,
  warning-severity invariants, SHOULD rules, and best-practice checks
  that have not been escalated to error by configuration.
- **`information`** — Diagnostic content that does not bear on
  validity: open-slicing notes, unsupported-element notes, validator
  configuration echoes, performance hints, etc.
- Separately, **process failure of the validator itself** (terminology
  server unreachable, internal exception, validation skipped) MUST be
  reported as an `OperationOutcome` issue with `severity = error` *and*
  with a distinguishable `issue.code` (proposed: `incomplete` /
  `processing`) plus a clear `details` text — and the response MUST
  make explicit that "no errors of validity were found that were
  checked" is not equivalent to "the instance is valid". Pair this
  with a recommended `OperationOutcome.issue.code` value-set extension
  (or pattern) so the *kind* of issue is interoperable, not just the
  severity (per Josh Mandel's request).

Touch points in `HL7/fhir`:

- `source/operationoutcome/operationoutcome-notes.xml` (or the
  corresponding intro / notes page) — add the validation-results
  subsection.
- `source/operationoutcome/codesystem-issue-severity.xml` (or the
  matching `IssueSeverity` page) — cross-link to the new guidance and
  clarify that the existing operation-centric definitions still apply
  to non-validation uses.
- `source/resource/operation-resource-validate.xml` (the `$validate`
  operation definition page) — reference the new guidance and require
  conformant validators to follow it when populating
  `OperationOutcome`.
- A note on overrideability: implementers MAY expose parameters that
  promote/demote specific issues (preserving Grahame's "operational
  freedom" point), but the *default* mapping is normative-style
  guidance.

#### Justification

This is exactly the gap Lloyd named and Grahame conceded ("evidently a
problem"): we have operation-centric definitions of `IssueSeverity` but
no validation-specific ones, even though `OperationOutcome` is the
standard transport for validator results. Defining them in the spec
(rather than per-IG or per-tool) is the only way implementers can
reason about cross-validator results — today an `error` from the IG
Publisher and an `error` from a server-side `$validate` mean different
things. It also directly addresses Josh Mandel's FHIR-I request to
treat severity + issue-code as a coherent pair so consumers can
distinguish "instance is bad" from "validator couldn't check".

---

### Disposition B: Alternative Approach

#### Proposal

Instead of redefining what `severity` means in a validation context,
keep the existing operation-centric `IssueSeverity` definitions
unchanged and instead introduce a **structured, validator-result
classification** carried in `OperationOutcome.issue.code` and a new
standard extension on `OperationOutcome` (or on `issue`) — e.g.,
`http://hl7.org/fhir/StructureDefinition/operationoutcome-validation-result`
with a coded value set such as:

- `definitely-invalid` — proven non-conformance.
- `possibly-invalid` — conformance could not be determined.
- `process-incomplete` — validator failed; result is partial.
- `advisory` — best-practice / informational note.
- `clean` — no issues found.

Validators would still emit `severity = error|warning|information` per
the existing rules (preserving backward compatibility with every
existing consumer), but the new classification answers the *real*
question — "is the instance valid?" — unambiguously. Add guidance on
the `$validate` page and `OperationOutcome` notes describing how to
populate the new field, and recommend a top-level summary
`OperationOutcome.issue` (with `code = informational`) that carries the
overall verdict for the validation run.

#### Justification

Compared to Disposition A, this avoids changing the meaning of
`severity` — a concept consumed by far more than just validators
(REST responses, batch/transaction outcomes, terminology operations,
etc.) and where redefining the term invites cross-context confusion of
the kind Grahame warned about. It also makes the result *machine*
readable in the way Josh asked for: a consumer can branch on
`validation-result = definitely-invalid` without having to re-derive
that from heuristics over `severity` + `code`. Trade-off: it adds new
machinery (an extension or new code values), and any consumer that
only inspects `severity` will still see the legacy overloaded
behaviour — so it does not, on its own, fix the "is this validator
internal failure or real non-conformance?" ambiguity unless adopted
broadly.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Mod → not now / out of scope for the spec
text**, on the grounds that (a) `OperationOutcome.issue.severity` and
`issue.code` already exist and are technically sufficient to express
each of the three situations Lloyd lists, (b) the substantive complaint
is that the *reference validator* and the IG Publisher use those fields
inconsistently, which is a tooling-conformance concern best handled in
the validator's own documentation and bug tracker rather than by adding
prose to FHIR Core, and (c) any normative-style narrative added now
risks contradicting deployed validator behaviour that consumers already
depend on.

#### Justification

The strongest decline argument is backward compatibility: Grahame's
comment establishes that real-world validators today use `error` for
internal process failures and `warning` for SHOULD/best-practice rules.
Pinning new, stricter definitions into the core spec would either
declare the reference validator non-conformant overnight or be
toothless if it is published as non-binding guidance. Combined with the
fact that no Jira / GitHub follow-ups have appeared in 18+ months and
the only Zulip activity is the originating thread, a workgroup could
reasonably judge there is not yet enough implementer demand to justify
the spec churn.

---

### Recommendation

**Recommended disposition:** A (with the issue-code interoperability
work from B folded in)

The ticket and the Zulip thread together establish that the gap is
real and named ("we don't actually have formal definitions of error,
warning, and information in the context of validation responses" —
Lloyd; "evidently that's a problem" — Grahame). Disposition C
under-serves implementers: the reason cross-validator results are
unreliable today is precisely that the spec does not pin down what
the severities mean for *validation*, so kicking it back to "tool
documentation" institutionalises that fragmentation. Disposition B is
attractive because it does not perturb the existing `IssueSeverity`
semantics, but on its own it leaves the most-consumed field
(`severity`) just as ambiguous as it is today — most downstream tools
read severity, not extensions.

The recommended path is therefore A as the primary change — add a
validation-specific reading of `severity` to the `$validate` /
`OperationOutcome` guidance, with the explicit carve-out that
implementers can override individual classifications via parameters
(addressing Grahame's "operational freedom" constraint) — and pull in
B's structured classification for the **validator-process-failure**
case (a distinct `issue.code` such as `incomplete` plus a clear
"results are partial" marker), which is the one situation where
severity alone genuinely cannot disambiguate. This gives the
workgroup a concrete proposal that resolves the ambiguity Lloyd
raised, satisfies Josh's interoperability concern, and respects the
deployed-validator reality Grahame highlighted.
