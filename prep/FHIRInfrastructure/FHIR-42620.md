## Summary

The ticket asks that FHIR Shorthand (FSH) support sharable **RuleSet
libraries**. The reporter, Bas van den Heuvel, proposes that RuleSets
become first-class sharable elements — ideally distributed as part of an
NPM package (e.g. as a FHIR `Library` resource) and consumable in other
FSH projects, possibly in combination with an `Include:` rule. The
reporter explicitly notes the request is acceptable to carry forward as
"Deferred."

FHIR-I originally resolved this on 2023-11-06 as "Considered for Future
Use" (Canessa/Grieve, 12-0-0), agreeing the capability would be useful
but flagging several open design questions: whether RuleSets are the
only shareable unit, how FSH libraries should be packaged, how they
should be distributed, and how they should be scoped when imported into
a project. The resolution pointed at the FHIR `Library` resource and
[CRMI](https://hl7.org/fhir/uv/crmi/2023Sep/index.html) as candidate
mechanisms, and noted prior discussion of importing FSH definitions
directly from GitHub repositories. The ticket was re-opened by
lynn_laakso on 2024-08-19 "post N2 publication" and is currently
`Triaged` with `Highest` priority, which is why it is back on the
workgroup's plate for disposition.

No Jira tickets are explicitly linked to or cross-referenced with this
item; the only external reference is a Zulip ballot-block announcement
(see below). There are no GitHub cross-references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Commit FHIR Shorthand (specification + SUSHI) to a concrete RuleSet
library mechanism in a future FSH version:

1. **Packaging.** Define a FSH "library" artifact distributed inside a
   standard FHIR NPM package. The library declares one or more named
   RuleSets (and, optionally, their transitive RuleSet dependencies) in
   the package's FSH-source area so that it can be consumed by SUSHI
   without needing to re-publish the compiled output.
2. **Import surface.** Extend the FSH grammar with an explicit import
   construct (e.g. a project-level `Library:` declaration or an
   `Include:`-style rule) that names a package and an exported RuleSet.
   The existing `insert <RuleSet>` rule continues to be the *use* site;
   only *resolution* changes.
3. **Scoping.** Imports are project-scoped, not global; imported
   RuleSets live in a namespace derived from the package id to avoid
   collisions with local RuleSets of the same name.
4. **Representation in published IGs.** Record the shared FSH library
   in the published IG as a FHIR `Library` resource (per the
   2024-08-19 comment), aligning with
   [CRMI](https://hl7.org/fhir/uv/crmi/2023Sep/index.html) patterns for
   canonical/version/packageId so that downstream tooling can locate
   the source.

#### Justification

This is what the reporter actually asked for, and the workgroup has
already agreed (12-0-0) that the feature is valuable. Re-using the
FHIR package/NPM distribution channel avoids inventing a new transport
and gives IG authors the same dependency/version semantics they already
use for StructureDefinitions and ValueSets. Using `Library` +
CRMI-style metadata keeps shared FSH artifacts discoverable in
published IGs and lets tooling other than SUSHI (e.g. validators,
registries) see that the RuleSets exist. The trade-off is real
design-and-implementation work in both the FSH spec and SUSHI, which
is why the original vote deferred rather than rejected.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than standardizing a new "RuleSet library" artifact, lean on
mechanisms that largely exist today:

1. **SUSHI project dependencies.** Formally specify that any FSH file
   shipped inside a dependency package's `input/fsh/` (or equivalent)
   directory is *available* to the dependent project, and that
   RuleSets defined there can be `insert`ed by name — effectively
   blessing the "ship FSH sources in the package" pattern rather than
   adding new grammar.
2. **Namespacing by package id.** To avoid name collisions, require
   cross-package RuleSet references to be qualified (e.g.
   `insert MyRuleSet from hl7.fhir.uv.example`) instead of adding a
   separate `Library:` / `Include:` construct.
3. **No `Library` resource requirement.** Treat the FSH source files
   as the library; do not require authors to also produce a FHIR
   `Library` resource, since doing so is largely bookkeeping for
   content that is already represented elsewhere in the published IG.

#### Justification

This is strictly less new surface area than Disposition A: no new
first-class "FSH library" artifact, no new resource, no CRMI
entanglement. It answers three of the four open questions the
workgroup raised in 2023 (packaging = existing NPM package,
distribution = existing registry, scoping = package-id qualifier) and
lets implementers start sharing RuleSets almost immediately. It does,
however, leave the fourth question — "What if a RuleSet references
something else?" — partially unsolved, since cross-package references
within imported RuleSets would still need to resolve through the
normal package dependency graph. It also ties FSH semantics to SUSHI's
file-layout conventions more tightly than Disposition A does, which
some members of the workgroup have historically pushed back on.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — out of scope for the current FSH maintenance
cycle." State that authors who need to share RuleSets today should
either (a) copy them into the project that needs them, or (b) refactor
the shared content into profiles/extensions/valuesets, which FSH and
the IG Publisher already distribute through NPM packages.

#### Justification

The ticket acknowledges in its own text that being labelled "Deferred"
is acceptable, and the workgroup's 2023 vote (12-0-0) already disposed
it as "Considered for Future Use." No new use cases, tooling, or
implementer pressure have been surfaced in the interim — the only
movement since then is the 2024-08-19 re-open, which is procedural
rather than substantive. There is no cross-referenced Jira, no GitHub
PR exploring the design, and no Zulip debate about the feature itself.
Declining keeps FSH's scope tight and preserves workgroup bandwidth
for items with clearer design direction; the ticket can be raised
again when someone brings a concrete proposal.

---

### Recommendation

**Recommended disposition:** A

Disposition A is recommended. The workgroup has already publicly
agreed on the direction (12-0-0 in November 2023), a substantive
design sketch exists in the 2024-08-19 comment (NPM packaging, FHIR
`Library`, CRMI alignment, package-scoped imports), and the ticket has
been re-opened precisely so the workgroup can move past "Considered
for Future Use" and commit to a concrete mechanism in a future FSH
version. Disposition B is attractive as an interim pattern and should
probably be adopted *as a bridging behaviour* while A is being
specified, but adopting B as the permanent answer leaves the
"reference graph across shared RuleSets" question unsolved and ties
FSH semantics to SUSHI file layout. Disposition C is defensible only
if the workgroup decides FSH itself should stop growing — given the
`Highest` priority on this ticket and the fact it was deliberately
re-opened, that does not appear to be the case.
