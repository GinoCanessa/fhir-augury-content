## Summary

The reporter (James Jahns) flags an out-of-date sentence on the FHIR Core
search documentation page describing the `reference` search parameter type.
Historically that sentence noted that, in addition to elements of type
`Reference` and `canonical`, the `reference` search parameter type was also
used to search resource elements of type `Resource` — i.e. resources directly
nested in another resource — and cited `Bundle.entry` search parameters
`message` and `composition` as the canonical example.

Since R5 added a dedicated `resource` search parameter type, both of those
Bundle examples have been migrated to the `resource` type, leaving the
sentence both inaccurate and (per the reporter) misleading. The ticket is a
**Technical Correction** asking that the now-stale sentence simply be removed
from the description of the `reference` search parameter type. There are no
linked or cross-referenced Jira tickets, no Zulip discussions surfaced for
the ticket key, and no related GitHub PRs/issues/commits in the index.

## Details

**Description:**

> The addition of the `resource` search parameter type makes this line about
> the `reference` search parameter type no longer true:
>
> "The reference search parameter type is mostly used for resource elements
> of type `Reference` or `canonical`. However, it is also be used to search
> resource elements of type [Resource](https://build.fhir.org/resource.html)
> — i.e. where one resource is directly nested within another — see the
> [Bundle search parameters](https://build.fhir.org/bundle.html#search)
> 'message' and 'composition' as an example of this."
>
> So, this line should be removed — both these examples now use the
> `resource` parameter type.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the requested technical correction to the FHIR Core search
documentation page (`search.html`, "Search Parameter Types — reference"
section). Specifically:

- Remove the sentence beginning "However, it is also be used to search
  resource elements of type Resource — i.e. where one resource is directly
  nested within another — see the Bundle search parameters 'message' and
  'composition' as an example of this." (and the surrounding stray quote
  marks shown in the description).
- Leave the preceding sentence ("The reference search parameter type is
  mostly used for resource elements of type `Reference` or `canonical`.")
  in place, optionally tightening "mostly used" to "used" given that the
  carve-out it qualified is being removed.
- Do **not** alter the `Bundle.entry` search parameter definitions for
  `message` and `composition` — those are already on the `resource` search
  parameter type and serve as the new canonical example for that type.

#### Justification

This is a pure documentation fix that brings the prose into alignment with
the normative artifacts. Once `Bundle.message` and `Bundle.composition` were
moved to the dedicated `resource` search parameter type, the sentence
describing `reference` as covering "elements of type Resource" became both
factually wrong and a likely source of implementer confusion (it suggests
using `reference` semantics — `_include`, chained search, modifiers like
`:identifier` — against contained resources, which is no longer the
intended pattern). It is a low-risk, low-priority Technical Correction and
matches what the reporter explicitly asked for.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than simply deleting the sentence, **rewrite** the `reference`
search parameter type description so that:

1. It describes `reference` strictly in terms of elements of type
   `Reference` and `canonical` (the current, correct scope).
2. It adds a short cross-reference / "See also" pointer to the new
   `resource` search parameter type entry on the same page, explaining
   that searching directly-nested resources (the former `Bundle.message` /
   `Bundle.composition` case) is now handled by the `resource` type.
3. Mirror the change on the `resource` search parameter type entry by
   citing `Bundle.message` and `Bundle.composition` as the canonical
   examples, so the page documents the migration explicitly.

This keeps the same surface deletion the reporter asked for, but layers a
discovery aid on top so existing readers — and existing implementer notes
that linked to the old sentence — are not left guessing where the
"directly-nested resource" pattern went.

#### Justification

Implementers carrying R4/R4B muscle memory may search the page for "Resource"
under `reference` precisely because that is what they remember; quietly
removing the sentence loses that lookup path. A small bidirectional
cross-reference between the `reference` and `resource` SP-type entries
preserves discoverability with negligible additional editorial cost, and
also documents (rather than silently completes) the R5 split between the
two SP types. The downside vs. Disposition A is slightly more editorial
work and the need to keep both entries in sync going forward.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" / no change. Leave the sentence in
`search.html` describing the historical use of `reference` against
elements of type `Resource`, on the grounds that the prose is informative
about pre-R5 behaviour and that no normative artifact is incorrect.

#### Justification

Declining is defensible only on the narrow ground that the sentence does
not affect any normative content — the `reference` SP type definition
itself, the `Bundle.message` / `Bundle.composition` SearchParameter
resources, and the `resource` SP type definition are all correct
independently of this prose. However, this rationale is weak: the
sentence is now demonstrably false ("it is also be used to search resource
elements of type Resource" is no longer how the spec models that case in
R5+), and leaving false text in normative documentation is exactly what
Technical Corrections exist to fix. There is no implementation cost,
backward-compatibility risk, or scope concern that would justify keeping
inaccurate prose.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the literal ask and is clearly correct in direction, but
since the page is being edited anyway it is cheap and high-value to add a
short "See also" pointer between the `reference` and `resource` SP-type
entries (Disposition B). The R5 introduction of the `resource` SP type is
exactly the kind of change where a stale lookup path — readers searching
the page for "Resource" under `reference` because that is where it lived
historically — will mislead implementers for several ballot cycles.
Disposition C should be rejected: leaving prose that is now factually
wrong in a normative spec page is the case Technical Corrections are
designed to address, and the editorial cost is trivial. If FHIR
Infrastructure prefers the minimum-diff option, fall back to Disposition A;
Disposition C should not be selected.
