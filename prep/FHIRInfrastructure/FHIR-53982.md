## Summary

The reporter (DanielSchuster) has spotted a stale CQL example on the
**DataRequirement** page of the FHIR Core specification. The example
selects `Procedure` resources and filters with `where P.performedPeriod
during "Measurement Period"`. That path is no longer valid in R5: the
`Procedure.performed[x]` choice element was renamed to
`Procedure.occurrence[x]` (so the matching `Period` typed slice is now
`Procedure.occurrencePeriod`). The ticket is a documentation-only
technical correction — the CQL snippet on the DataRequirement page
should be updated so its element path matches the current Procedure
resource definition. No related Jira tickets, Zulip discussions, or
GitHub items were surfaced by cross-references for this issue; the only
cross-references found are to the FHIR canonical paths
`Procedure.performed` and `Procedure.performedPeriod`, which are exactly
the elements named in the description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CQL example on the **DataRequirement** narrative page so the
filter clause uses the current R5 element path. Concretely, change:

```
[Procedure: "Total Colectomy Value Set"] P
  where P.performedPeriod during "Measurement Period"
```

to:

```
[Procedure: "Total Colectomy Value Set"] P
  where P.occurrencePeriod during "Measurement Period"
```

No structural definition change is required — only the prose/example on
the DataRequirement page (and any duplicated copies of the same example
elsewhere in the spec, if grep surfaces them at edit time).

#### Justification

This is a pure documentation correction. `Procedure.performed[x]` was
renamed to `Procedure.occurrence[x]` in R5, and the CQL example in the
DataRequirement page must reflect the current resource. Leaving the old
path in place actively misleads implementers who copy the example as a
template for their own CQL criteria — they will either author CQL that
fails to compile against an R5 Procedure or, worse, silently treat
`P.performedPeriod` as an extension/unknown path and produce no results.
The fix carries essentially no risk: it does not change normative
content, does not affect bindings or constraints, and does not require
coordination with other resources.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the rename as in Disposition A, **and** rework the example to be
self-explanatory about the R4 → R5 path change so the same confusion
does not recur for readers who arrive from older copies of the spec or
from R4-vintage CQL libraries. Specifically:

1. Replace `P.performedPeriod` with `P.occurrencePeriod` in the
   DataRequirement page CQL example.
2. Add a one-line editorial note immediately under the example, e.g.
   "*Note: `Procedure.performed[x]` was renamed to
   `Procedure.occurrence[x]` in R5; the filter uses
   `occurrencePeriod` accordingly.*"
3. Sweep the FHIR Core spec for other occurrences of `performedPeriod`
   in narrative pages or examples (CQL, FHIRPath, JSON/XML samples) and
   update them in the same change so the correction is uniform.

#### Justification

This goes one step beyond the literal request and treats the ticket as
evidence of a class of stale content rather than a single typo. The
DataRequirement page is high-traffic for measure/decision-support
authors, so a brief note on the rename costs almost nothing and helps
readers who carry R4 mental models. The companion sweep is cheap to do
in the same PR and prevents follow-up tickets for the same issue
elsewhere in the spec. Trade-off vs. Disposition A: a slightly larger
diff, requiring a quick scan for other `performedPeriod` appearances.

---

### Disposition C: Decline

#### Proposal

Close the ticket without changing the spec, on the grounds that the
example is illustrative only and the surrounding DataRequirement
narrative already directs readers to the current Procedure resource
definition for authoritative element names.

#### Justification

This would be defensible only if the example were marked as
non-normative pedagogy unrelated to a specific FHIR version, or if the
page already disclaimed that element paths in examples may lag the
current resource. Neither is the case here — the example is presented
as a directly usable CQL snippet against the R5 `Procedure` resource,
so leaving an invalid element path in place is materially misleading.
Decline should therefore be considered the weakest of the three
options.

---

### Recommendation

**Recommended disposition:** **B**

The literal fix in Disposition A is correct and uncontroversial, but
the rename from `performed[x]` to `occurrence[x]` is exactly the kind
of cross-version trap that catches measure authors repeatedly,
especially those porting CQL libraries from R4. Doing the rename plus a
brief inline note plus a one-pass sweep for other stale
`performedPeriod` references in spec narrative is still a small,
low-risk PR for FHIR Infrastructure, but it removes a whole class of
"copy-pasted from the spec and it doesn't work" reports rather than
just the single instance the reporter happened to land on. Disposition
A is an acceptable fallback if the work group prefers the smallest
possible diff; Disposition C should not be adopted.
