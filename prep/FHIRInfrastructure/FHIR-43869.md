## Summary

The reporter notes that the section numbering shown for the "Brand"
section of the SMART on FHIR (SMART App Launch) specification is
incorrect: the first sub-heading is rendered as **8.0.1** rather than
the expected **8.1**. This is a presentation/numbering defect in the
published specification's table of contents / heading scheme rather
than a normative-content change. The ticket is filed as a
Technical Correction at Highest priority but contains no further
detail, no cross-references in FHIR Augury, no Zulip discussion, no
linked GitHub items, and no comments. Likely root cause is an extra
or mis-leveled heading at the top of the Brand chapter (or its intro
include) producing an empty parent section "8.0", which then numbers
its first child as "8.0.1" instead of promoting the children to
"8.1", "8.2", … under the chapter heading "8".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a publication bug in the SMART App Launch IG and fix
the heading structure of the Brand chapter so that its first
sub-section renders as **8.1** instead of **8.0.1**. Concretely, in
`HL7/smart-app-launch`:

1. Open the Brand chapter narrative source under `input/pages/`
   (most likely `brands.md` or `brands-intro.md`).
2. Remove the spurious empty parent heading (or promote the existing
   chapter heading by one level) so the file contains exactly one
   chapter-level heading followed by `##` sub-sections — eliminating
   the empty "8.0" container that is forcing children to "8.0.1".
3. Verify in `sushi-config.yaml` (the `pages:` block) that the Brand
   page is registered at the chapter level (depth 1) rather than as
   a child of an empty wrapper.
4. Re-build the IG locally with the IG Publisher and confirm the
   TOC renders "8 Brand" → "8.1 …", "8.2 …".
5. Open a PR against `HL7/smart-app-launch` referencing FHIR-43869,
   and re-publish the affected continuous-build / next ballot.

#### Justification

The ticket is filed as a Technical Correction with no normative
content change requested — only the rendered numbering. Heading-
level / TOC bugs in IGs built with the IG Publisher routinely
produce exactly this "X.0.1" symptom when an extra `#`-level heading
sits above the intended sub-sections. The fix is local to one or
two files in the spec authoring repo, has no impact on
implementations or conformance, and matches the reporter's stated
expectation. Resolving the cosmetic defect at Highest priority is
appropriate because section numbers are referenced from external
material (slides, ballot comments, conformance discussions) and a
visibly broken TOC undermines reader confidence in the spec.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve the visible numbering defect in the published Brand chapter,
**and** at the same time audit every other chapter of the SMART App
Launch IG for the same class of defect (empty parent section
producing `N.0.1` children, off-by-one chapter numbering, duplicated
headings). Add a small CI check to the repo's IG build (e.g., a
post-publish script that scans the generated `toc.html` for any
section number containing `.0.`) so that future contributions can't
re-introduce the same bug silently.

#### Justification

The reported symptom (`8.0.1` instead of `8.1`) is a class of bug,
not a one-off. SMART App Launch has been actively edited around the
Brand / User Access Brands chapters, which is exactly the kind of
churn that produces stray heading levels. A repo-wide sweep plus a
cheap regression check costs little beyond Disposition A and
prevents the same Technical Correction from being raised again
against a different chapter in the next ballot cycle. The trade-off
is a slightly larger PR and a one-time CI addition, which is
proportionate for a Highest-priority publication bug.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the report is unverifiable
as filed — it does not state which build (current published version,
continuous build, ballot snapshot, specific URL) exhibits the
`8.0.1` rendering — and close as "Not Persuasive — Insufficient
Information", inviting the reporter to re-open with a link to the
specific page and version where the numbering is wrong.

#### Justification

The ticket body is one sentence with no URL, no version, no
screenshot, and no comment thread, and Augury surfaces no Zulip
discussion or PR confirming the defect. SMART App Launch has been
republished multiple times since 2024-01-19, and the Brand chapter
itself has been restructured during that window, so the numbering
the reporter saw may already differ from any current build. Acting
on an unreproduced report risks "fixing" something that is no
longer broken or breaking a chapter whose current numbering is
deliberate.

---

### Recommendation

**Recommended disposition:** A

The defect described — a TOC entry numbered `8.0.1` where `8.1` is
expected — is the textbook symptom of a stray heading level in IG-
Publisher-generated narrative, is cheap to diagnose by inspection
of the Brand chapter source in `HL7/smart-app-launch`, and carries
no normative risk. The Highest priority assignment, the
Technical Correction issue type, and the FHIR Infrastructure work
group's standing practice of clearing publication defects between
ballots all point to a straightforward "fix the heading and
republish" outcome. Disposition B is attractive as a follow-up but
should not block A; the audit/CI work can be tracked as a separate
maintenance ticket so this Technical Correction can be cleared in
the next continuous build. Disposition C is defensible only if the
defect cannot be reproduced against the current build — the
workgroup should attempt to reproduce against the latest SMART App
Launch publication first; if the numbering is already correct,
close as "No Change Required" with a link to the verified TOC,
rather than declining for missing information.
