## Summary

The reporter (Bas van den Heuvel) asks the FHIR Shorthand (FSH) authors to
add a language construct that would auto-populate the `display` element of
a `Coding` / `CodeableConcept` when an FSH author writes only the system
and code (e.g., `http://snomed.info/sct#363346000`). The reporter explicitly
does **not** want the display to be populated implicitly by every
system+code expression — implicit population would prevent authors from
deliberately omitting `display` when needed — but rather a short, opt-in
syntax (illustrated with a trailing `+`) that tells the FSH compiler to
fetch and emit the canonical display string for that code.

There are no cross-referenced Jira tickets. The only cross-reference is a
Zulip thread, which is purely about a Jira workflow/status bug that
affected this ticket administratively (it does not add substantive
discussion of the proposal itself). The ticket has a voting history: it
was previously voted "Considered for Future Use" (Mark Kramer / Bas van
den Heuvel: 10-0-1), then accidentally moved to "Resolved - Change
Required" by a Jira workflow bug, then reopened to be re-deferred, and
later flagged for re-opening "post publication of Normative Shorthand
spec." It currently sits as `Triaged` / `Unresolved` with the
`block-vote-0` label.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the FHIR Shorthand grammar with an opt-in operator that requests
the FSH compiler to populate `Coding.display` (and, by extension, the
matching `CodeableConcept.coding[].display` and the optional
`CodeableConcept.text`) from the referenced `CodeSystem`. Concretely, add
a trailing marker on a code literal — adopting the reporter's `+` is one
option — so that:

```
* code = http://snomed.info/sct#363346000 +
```

is compiled by SUSHI as:

```json
{
  "system": "http://snomed.info/sct",
  "code":   "363346000",
  "display": "Procedure (procedure)"
}
```

Mechanics:

1. Update the FSH ANTLR grammar (FHIR Shorthand spec) to recognize the
   new operator on `concept`/`code` literals in any position that
   currently accepts a `Coding` shorthand.
2. Update the FHIR Shorthand specification's normative section on coded
   values to describe the operator, its semantics, and its failure modes
   (warning vs. error).
3. Define a configuration knob in `sushi-config.yaml` that lists the
   terminology source(s) to use for resolution — e.g., a terminology
   server URL plus an optional list of `CodeSystem` instances declared
   in the project — and the on-failure behavior (`error` | `warn` |
   `ignore`).
4. Implement resolution in SUSHI: prefer in-project `CodeSystem`
   resources, then fall back to the configured terminology server,
   honoring the version pinned by the FSH project.
5. Cache resolved displays in the SUSHI build output to keep builds
   reproducible offline once the cache is populated.

#### Justification

The proposal is opt-in, so it preserves today's behavior for authors who
deliberately omit `display`. It addresses a real authoring pain point —
copy/paste errors and stale display strings are common in IGs — and
mirrors patterns used by other FHIR tooling (the IG publisher already
auto-populates and validates displays via tx.fhir.org). FHIR Infrastructure
has progressively standardized on the terminology-service contract for
exactly this kind of resolution, so requiring a configured terminology
source is consistent with current practice. Bas van den Heuvel's own
comment on the ticket already maps the failure-mode design (warn-or-fail
when the `CodeSystem` is unavailable), which is the same pattern
implementers of this proposal would adopt.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding new grammar to FHIR Shorthand itself, address the
need entirely in **SUSHI** (and any other FSH compiler) as a build-time
linter / autofixer:

- Add a SUSHI configuration option (e.g., `autoDisplay: warn | fix | off`
  in `sushi-config.yaml`, default `off`).
- When set to `warn`, SUSHI emits a warning for every `Coding` /
  `CodeableConcept.coding` it produces with a missing `display`, and
  reports the canonical display it would have used.
- When set to `fix`, SUSHI populates the missing `display` during build
  using the same project-local + terminology-server resolution chain as
  Disposition A.
- The decision is made per-project rather than per-literal, and FSH
  source files remain unchanged.

This keeps the FHIR Shorthand language surface stable and confines the
behavior to tooling.

#### Justification

The original "Considered for Future Use" rationale (10-0-1, recorded by
Lloyd McKenzie on 2021-10-27) was specifically that *the FSH interpreter*
would need terminology access for every code system, which would make
plain FSH non-interpretable without a terminology server. Disposition B
sidesteps that concern entirely: the FSH **language** stays
self-contained, and only the **build tool** optionally enriches output.
This avoids a normative grammar change to a now-published Normative
specification (which is high-cost) and keeps the door open to evolve the
behavior in tooling without re-balloting the language. Disposition B also
offers richer ergonomics than the `+` operator — one project-wide switch
beats annotating every code site — and lets the same machinery surface
*incorrect* hand-typed displays (a bonus capability not provided by
Disposition A).

The trade-off vs. Disposition A is that authors lose the ability to
opt-in display population at a single site while leaving others alone;
they must commit to project-wide behavior. In practice, that is rarely a
real constraint because the project-wide warning still flags the
exceptions.

---

### Disposition C: Decline

#### Proposal

Maintain the existing FHIR-I disposition: do not adopt, on the grounds
that this requirement is already adequately addressed and the cost of
introducing it into normative FHIR Shorthand outweighs the benefit.
Recommend that authors who want display values either (a) write them
explicitly in FSH, or (b) run their published IG through the IG
publisher, which performs display validation/population against
tx.fhir.org as part of the normal build. Close the ticket as
"Considered for Future Use" / "Persuasive with Modification — not adopted".

#### Justification

The original 10-0-1 vote captured the core design problem: FSH was
intentionally designed to be interpretable without external dependencies,
and adding terminology resolution into the language semantics
fundamentally changes that contract. FHIR Shorthand has since gone
Normative, which raises the bar for any grammar change considerably.
Furthermore, the `display` element is — per FHIR conformance rules —
informational and never a basis for matching codes, so its absence is
not a correctness defect. Because the IG publisher already validates and
warns about missing/incorrect displays during the IG build, the
end-to-end author workflow already catches the failure modes Bas
described, just in a different tool. Spending normative-spec churn on a
convenience feature that downstream tooling already provides is hard to
justify.

---

### Recommendation

**Recommended disposition:** B

The substantive concern that drove the original deferral — that FSH must
not implicitly require a terminology server to be interpretable — was
correct and is even more important now that Shorthand is Normative.
Disposition A re-opens that concern by baking terminology resolution into
the language; Disposition C is defensible but leaves the real authoring
pain point (copy/paste errors and drift in `display`) unaddressed inside
the FSH ecosystem itself. Disposition B threads the needle: it gives Bas
the ergonomic benefit he is asking for, keeps the FSH language pure,
avoids a normative grammar change, and produces a richer feature
(detect-and-fix incorrect displays as well as missing ones). FMG / FHIR-I
should ask the SUSHI maintainers whether `autoDisplay: warn | fix` can
be scheduled into a near-term SUSHI release; if they accept, the ticket
can be closed as "Persuasive with Modification" and tracked against the
SUSHI repo. If they decline, fall back to Disposition C and document the
IG-publisher path as the recommended workflow.
