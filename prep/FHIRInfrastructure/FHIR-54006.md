## Summary

The ticket reports a documentation defect in the FHIR Core specification:
the long-form definition of `Bundle.total` still references the search-mode
value `"outcome"`, even though that mode has been deprecated. The reporter
asks that the mention of `"outcome"` be removed from the `Bundle.total`
definition so that the element's documentation aligns with the current
state of the `Bundle.entry.search.mode` value set / search behavior. No
related Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-references; the only cross-reference is to the FHIR artifact
`Bundle.total` itself, confirming the scope is purely a definition-text
fix on that single element.

## Details

Search mode "outcome" should be removed from the long description for

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the long-form definition text of `Bundle.total` in the FHIR Core
specification so that it no longer mentions the deprecated search mode
value `"outcome"`. Concretely: in the `Bundle.total` `ElementDefinition`
authored in the Bundle resource definition (the source that produces
both the structure definition and the rendered Bundle resource page),
strike the sentence/clause that calls out `"outcome"` as a search mode
contributing to (or excluded from) the count, and reword the surrounding
text so that the definition only references currently-supported search
modes (`match`, `include`). Regenerate the published artifacts so the
rendered HTML, the `StructureDefinition` JSON/XML for Bundle, and any
derived snapshots all carry the corrected text. No element cardinality,
type, binding, invariant, or value-set membership changes — this is a
definition-text-only update.

#### Justification

The reporter is documenting an internal inconsistency: the
`Bundle.entry.search.mode` value set no longer offers `"outcome"`, yet
`Bundle.total`'s definition still describes how `"outcome"` entries
interact with the count. Leaving the stale reference in place is
actively misleading to implementers — they may either look for an
`outcome` mode that no longer exists or assume the `total` semantics
still depend on it. Definition-text alignment with the current value
set is a routine, low-risk editorial correction that keeps the spec
internally consistent without affecting wire format, validation, or
backwards compatibility.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than silently deleting the `"outcome"` reference, rewrite the
`Bundle.total` definition to (a) describe the count strictly in terms
of the current `match` semantics, and (b) add a brief historical note
(or a changelog-style sentence) acknowledging that the previously
deprecated `"outcome"` mode is no longer counted. Optionally pair this
with a forward link to the `Bundle.entry.search.mode` element so that
readers tracing search-bundle semantics land in one canonical place.
Implementation is the same single source edit on the Bundle resource
definition, plus a one-line note.

#### Justification

A bare deletion loses provenance: implementers maintaining older code
that previously emitted or consumed `"outcome"` entries may benefit
from an explicit acknowledgement that the mode is gone and is not
counted by `Bundle.total`. A short historical note costs almost
nothing, prevents future "did this used to say something else?"
tickets, and keeps the editorial tone consistent with how other
deprecated FHIR concepts are handled in element definitions. The
trade-off vs. Disposition A is slightly more prose to maintain, but
the clarity benefit is meaningful for an element as widely consumed
as `Bundle.total`.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that the existing `Bundle.total`
definition is not technically incorrect — it merely references a mode
that some implementations may still encounter in legacy data — and
that touching the definition risks unintended interpretation
differences across versions. Direct the reporter to the
`Bundle.entry.search.mode` value set as the authoritative source of
which modes are currently valid.

#### Justification

This is the weakest of the three options and is included for
completeness. It would only be defensible if the workgroup believed
that retaining the `"outcome"` mention serves a back-compat
documentation purpose for implementations still processing historical
bundles. Even then, the better resolution is Disposition B (keep the
context, drop the misleading current-tense framing) rather than
leaving the definition as-is.

---

### Recommendation

**Recommended disposition:** B

This is fundamentally a small editorial correction, so doing nothing
(C) is hard to justify — the current text genuinely is misleading.
Between A and B, the marginal cost of a one-sentence historical note
is trivial compared to the recurring confusion it prevents, and it
matches the editorial pattern the spec already uses elsewhere when
elements outlive deprecated companions. Disposition B therefore gives
the workgroup the cleanest outcome: `Bundle.total`'s definition stops
asserting anything about a mode that no longer exists, while still
acknowledging the deprecation so that implementers reading the spec
alongside older code aren't left wondering whether something was
quietly removed. If the workgroup prefers maximum brevity,
Disposition A is an acceptable fallback.
