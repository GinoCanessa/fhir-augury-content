## Summary

Brian Postlethwaite is requesting a substantial cleanup and expansion of the
documentation for the FHIR-defined FHIRPath terminology extensions (the
`%terminology` factory functions and the related `subsumes` / `subsumedBy` /
`memberOf` / `weight` operations). The current spec text leaves several gaps
that make the functions hard to implement consistently:

1. There is no documentation of how the `params` parameters are processed —
   especially how to "URL-encode" complex inputs such as `Coding`,
   `CodeableConcept`, or `CodeSystem`. If a different part of the FHIR spec
   already explains this, the FHIRPath section should at least cross-link to
   it; if some `params` cannot be supported, the spec should call out which
   ones explicitly are.
2. The `%terminology` factory functions accept multiple parameters and use
   `{}` to skip unused positional slots, which is inconsistent with how other
   FHIRPath functions are described.
3. There are no worked examples showing a sensible call site for each
   function.
4. The relationship between `%terminology.subsumes` and the fluent
   `subsumes` / `subsumedBy` functions is not explained — if the only
   difference is that `%terminology.subsumes` exposes additional operation
   parameters, the request is to either fold those parameters into the fluent
   forms and remove `%terminology.subsumes`, or document why it is being
   kept.
5. The behaviour when a terminology server is unavailable needs guidance.
   Brian explicitly carved this out into a sibling ticket,
   [FHIR-53953](https://jira.hl7.org/browse/FHIR-53953) ("No documentation on
   remote interactions"), which asks for a new top-level "Handling Remote
   Data" section covering `resolve`, the terminology functions
   (`memberOf`, `subsumes`, `subsumedBy`, `weight`, `%terminology`), the
   profile-related functions (`conformsTo`, `slice`, `elementDefinition`),
   and the `%server` factory functions, including what should happen on
   failure (probably "return empty" rather than throw) and whether something
   like a `traceOnRemoteError()` hook is appropriate. FHIR-53953 supplies the
   broader framing for the per-function fixes requested here.

Michael Lawley's comment narrows one concrete pain point: there is no
URL-encoding for a `CodeableConcept`, so if these functions need to accept
genuinely complex inputs they likely need a `Parameters`-style mechanism
rather than encoded query string parameters.

exists at all and is therefore relevant to disposition (4) in the

### [implementers > FHIRPath Subsumes](https://chat.fhir.org/#narrow/stream/implementers/topic/FHIRPath%20Subsumes)

This 2018 thread (originating from GF#13451 / GF#15777) is where the
FHIRPath terminology service API was first proposed. Grahame Grieve
concluded that FHIRPath should not bake subsumption semantics into the
language itself but should instead "define a standard API to the terminology
service that is available in FHIRPath," giving the example
`iff(terminologyService().subsumes(concept1, concept2) = 'subsumes', …)`.
Bryn Rhodes pushed back, noting CQL had deliberately excluded subsumption
support to avoid spec-level variability, and suggested the same for
FHIRPath. Lloyd McKenzie and Rob Hausam debated the surrounding question of
when multiple codings of differing granularity belong in the same
`CodeableConcept`. The thread ended with Grahame committing to a task to
"document a standard terminology service API in FHIRPath" — i.e. the very
surface FHIR-54069 is now asking to be tightened up. The discussion is
useful when arguing whether `%terminology.subsumes` should be retained as a
distinct function (Disposition B) or collapsed into the fluent `subsumes` /
`subsumedBy` (Disposition A).

Participants: Grahame Grieve, Bryn Rhodes, Lloyd McKenzie, Rob Hausam,
Michael Lawley, Michelle (Moseman) Miller, Eric Haas.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Land a documentation-only update to the FHIRPath chapter of FHIR Core (the
`%terminology` / terminology-service-API section in `fhirpath.html`) that
addresses each of the five points raised:

1. Add a paragraph (or cross-reference to the existing search-parameter
   serialization rules) describing exactly how `params` values are encoded
   on the wire for each FHIR datatype that may appear, and explicitly list
   which datatypes are NOT supported as `params` inputs (notably
   `CodeableConcept`, per Michael Lawley's comment).
2. Replace the positional `{}`-as-skip pattern in the `%terminology`
   factory function signatures with named-parameter syntax consistent with
   the rest of FHIRPath, deprecating the positional form.
3. Add at least one worked, runnable example per published function
   (`%terminology.expand`, `%terminology.lookup`, `%terminology.validate`,
   `%terminology.translate`, `%terminology.subsumes`, plus `memberOf`,
   `subsumes`, `subsumedBy`, `weight`).
4. Add an explanatory note for `%terminology.subsumes` stating that it
   exists solely to expose the additional `$subsumes` operation parameters
   that the fluent `subsumes` / `subsumedBy` functions cannot carry, and
   either (a) fold those parameters into the fluent forms and remove
   `%terminology.subsumes`, or (b) keep both and document the boundary.
5. Defer the "terminology server unavailable" guidance to FHIR-53953 and
   add a forward link from this section to the new "Handling Remote Data"
   section that ticket creates.

#### Justification

Every point in the ticket is a legitimate documentation gap: implementers
today cannot deterministically encode complex parameters for a
`%terminology` call, the `{}` skip pattern is genuinely inconsistent with
how the rest of FHIRPath functions are documented, and the lack of examples
makes the section hard to consume. Pairing point (5) with FHIR-53953
preserves a single source of truth for remote-interaction semantics rather
than duplicating it in the terminology section. Michael Lawley's comment
already validates the substantive issue under (1) — there is no
URL-encoding for `CodeableConcept`, which the spec must either fix or
explicitly disallow.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of patching the existing `%terminology` factory functions in place,
restructure the section in two stages:

1. Replace ad-hoc `params` strings with a typed parameter object derived
   from the FHIR `Parameters` resource (per Michael Lawley's suggestion).
   `%terminology.expand`, `.lookup`, `.validate`, `.translate`, and
   `.subsumes` would then take a `Parameters`-shaped FHIRPath object
   literal as their input, eliminating the URL-encoding question entirely
   and the `{}` skip pattern with it.
2. Retain `%terminology.subsumes` (do not remove it as point 4 contemplates)
   because the fluent `subsumes` / `subsumedBy` operate on `Coding` /
   `CodeableConcept` values inside the resource, while `%terminology.*`
   operate against the terminology service with operation-level parameters
   (e.g. `system-version`, `displayLanguage`, `useSupplement`). Document the
   distinction explicitly rather than collapsing the surface.

Worked examples and the cross-link to FHIR-53953's "Handling Remote Data"
section would still ship as in Disposition A.

#### Justification

The 2018 implementers thread shows that the original intent was for
FHIRPath to expose the full terminology service API, not just the subset
expressible as fluent functions on existing `Coding` / `CodeableConcept`
values. Collapsing `%terminology.subsumes` into the fluent forms (as
Disposition A's point 4(a) allows) loses that capability for parameters the
fluent forms cannot carry. Switching from URL-encoded `params` to a
typed `Parameters` object also addresses the substantive blocker Michael
Lawley raised — that `CodeableConcept` cannot be URL-encoded — without
having to enumerate datatype-by-datatype which inputs are or are not
supported. The trade-off is a larger, breaking change to the function
signatures rather than a documentation-only patch, and it should ideally
land coordinated with FHIR-53953 so that both surfaces stabilise together
in the same ballot cycle.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the `%terminology` extensions are
already labelled as Incubator-status content and that the appropriate place
to harden their documentation is alongside the broader "Handling Remote
Data" overhaul scoped under [FHIR-53953](https://jira.hl7.org/browse/FHIR-53953).
Close FHIR-54069 as "duplicate / will be addressed by FHIR-53953" and roll
the five sub-points into that ticket's scope.

#### Justification

The ticket carries the `Incubator` label, indicating the surface is not yet
considered normative-ready and is expected to evolve. Several of the
sub-points (notably (5) on server-unavailable behaviour) are explicitly
already covered by FHIR-53953, and the remaining points (URL-encoding,
positional vs. named parameters, `%terminology.subsumes` vs. fluent forms,
worked examples) all require the same editor to be in the same section at
the same time. Treating them as a single editorial pass under FHIR-53953
avoids two ballots' worth of churn on the same paragraphs.

---

### Recommendation

**Recommended disposition:** A

The ticket is, at its core, a request for documentation that should already
be there. Disposition A delivers exactly that without committing the work
group to a breaking signature change before there is consensus that
`%terminology` should pivot to a `Parameters`-shaped input. It also
explicitly hands the server-unavailable question off to FHIR-53953, which
is where the broader remote-interaction policy lives, so the two tickets
stay coherent. Disposition B is the right long-term direction and the
2018 Zulip thread supports keeping `%terminology.subsumes` distinct from
the fluent forms, but the typed-`Parameters` rework is a meaningful
breaking change and is best raised as a follow-up Change Request once the
documentation under Disposition A makes the current behaviour
unambiguous. Disposition C under-serves implementers: the `Incubator`
label does not excuse leaving the section without examples or with
inconsistent function signatures, and rolling everything into FHIR-53953
risks both tickets stalling. Recommend acceptance with the explicit caveat
that point 4(b) (keep `%terminology.subsumes`, document the boundary) is
preferred over 4(a), pending work-group discussion.
