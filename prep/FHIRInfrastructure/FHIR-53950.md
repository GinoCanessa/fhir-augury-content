## Summary

The ticket asserts that the FHIR Core spec page that describes how FHIR `Quantity`
maps to / is used as a FHIRPath quantity is wrong on three counts and is also
poorly factored relative to the FHIRPath base specification:

1. **Unit systems are under‑stated.** The page implies that FHIR `Quantity` can
   only be used as a FHIRPath quantity when the units come from UCUM
   (`http://unitsofmeasure.org`). FHIRPath also recognises calendar duration
   units from `http://hl7.org/fhirpath/CodeSystem/calendar-units`, and the FHIR
   page should say so and link to the FHIRPath spec.
2. **Duplication of the FHIRPath time‑valued‑quantities table.** The same
   normative table (and its accompanying "dragon" warning) is reproduced on the
   FHIR page. The reporter wants the duplication removed and a pointer
   substituted; if the content must remain, it should at minimum cite the
   original FHIRPath section rather than silently repeating it (and the dragon
   should go — the FHIRPath source does not have one).
3. **Implicit code‑system conversion is wrong.** The page suggests / implies
   that FHIRPath engines convert between the UCUM and calendar‑units code
   systems implicitly. The reporter's position is that an engine must not
   convert code systems unless the caller has explicitly asked for it
   (e.g., via `toQuantity(unit)`).
4. **Behaviour for unsupported unit systems is unspecified.** When a `Quantity`
   uses neither UCUM nor calendar‑units, the page does not state what the
   FHIRPath engine should do. The reporter expects "return empty", but wants
   the spec to make this explicit (vs. raising an error to the caller).
5. **Missing back‑references.** The reporter asks the work group to consider
   whether the FHIR page should also link to three additional FHIRPath
   sections — `#datetime-arithmetic`, `#toquantityunit--string--quantity`, and
   `#quantity-equality` — rather than re‑explaining (or ignoring) that material.

The reporter explicitly flags this ticket as a sibling to the FHIRPath base
specification work happening under
[FHIR-53250](https://jira.hl7.org/browse/FHIR-53250) ("Calendar duration
equivalence has conflicts"; ITS, FHIRPath spec; resolution
*Persuasive with Modification*). FHIR‑53250 tightens the rules around when
calendar‑duration conversion factors may be used and where they should be
documented in the FHIRPath spec; FHIR‑53950 is the FHIR‑Core‑side
counterpart that wants the FHIR page realigned with whatever FHIRPath ends up
saying. No formal Jira "links" relation was returned by Augury — the
relationship is asserted in the reporter's comment and confirmed by the
parallel Zulip discussions captured below.

## Related Jira Tickets

### [FHIR-53250: Calendar duration equivalence has conflicts](https://jira.hl7.org/browse/FHIR-53250)

Lodged against the FHIRPath base spec (work group: Implementable Technology
Specifications; specification: FHIRPath (FHIR); resolution: *Persuasive with
Modification*). It points out that the existing FHIRPath text in §6.1.2.1
"Quantity Equivalence" — *"Implementations that do support units shall do so
as specified by [UCUM] as well as the calendar durations as defined in the
toQuantity function"* — combined with UCUM's own conversions, makes
`1 year ~ 1 'a' ~ 365.25 'd'` and `1 year ~ 365 days ~ 365 'd'` both true and
therefore conflicting. The accepted disposition (with modification) is to
remove that sentence and to make explicit in `toQuantity` that the calendar
duration conversion factors are *only* for explicit use through
`toQuantity()` — they cannot be used implicitly in equality, equivalence, or
comparison. This is the upstream change that FHIR‑53950 wants the FHIR Core
page to mirror.

Users involved: JingTang (reporter), bryn.rhodes (commenter / proposed
disposition author), brian.postlethwaite (referenced in adjacent Zulip
discussion).

## Related Zulip Discussions

### [fhirpath > Problems with calendar duration units](https://chat.fhir.org/#narrow/stream/fhirpath/topic/Problems%20with%20calendar%20duration%20units)

Yury Sedinkin opened the topic on 2026-01-09 (three days before FHIR‑53950
was filed) asking what `Observation.value.toQuantity('a')` should do when the
underlying `Quantity` uses UCUM `'a'`, and how implicit conversion between
`'a'` and `year` should behave. Brian Postlethwaite — the reporter of
FHIR‑53950 — answered that conversion only happens when needed and that the
data is *not* converted on read; he linked to the (then-current) build of the
FHIRPath `toQuantity` section and noted the spec had recently been updated
there. Yury accepted the explanation and flagged that `fhirpath.js`
incorrectly converts at reference time. Lukas Schmierer marked the topic
resolved, then re‑opened it shortly after. The thread is the proximate
trigger for FHIR‑53950: it confirms the reporter believes implicit
code‑system conversion is wrong and that the normative source for the rules
is the FHIRPath spec, not the FHIR Core page.

Participants: Yury Sedinkin, Brian Postlethwaite, Lukas Schmierer.

### [fhirpath > Definite durations above seconds in date time arithmetic](https://chat.fhir.org/#narrow/stream/fhirpath/topic/Definite%20durations%20above%20seconds%20in%20date%20time%20arithmetic)

Started by Jing Tang on 2025‑12‑16 about test cases that use `'d'`/`'h'` etc.
in date arithmetic despite the spec line *"If a definite-quantity duration
above seconds appears in a date/time arithmetic calculation, the evaluation
will end and signal an error to the calling environment."* Bryn Rhodes
clarifies that "above seconds" is stale wording — it should read "above days
(and weeks)" — and that this is being corrected as part of FHIR‑53250. The
discussion then expands into whether `1 year.toQuantity('hours')` should be
valid (Bryn: no, because no direct conversion factor is defined), how
calendar conversions should be referenced from the Date/Time Arithmetic and
Quantity sections (Brian wants the duplication minimised; Bryn prefers a
link from one section to the other rather than a move), and where
`toQuantity` semantics for unsupported units should be documented. Consensus
within the thread: calendar conversion factors are only for explicit use via
`toQuantity`; implicit conversion is forbidden; cross‑unit conversions that
span variable‑length periods (year ↔ hour) are invalid. Open question: where
exactly the rule about *unsupported* unit systems should live and exactly
what it should say. The thread directly informs every part of FHIR‑53950's
ask.

Participants: Jing Tang, Bryn Rhodes, Brian Postlethwaite, Yury Sedinkin,
Paul Lynch.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the "FHIR Quantity" / "Using FHIR Quantity in FHIRPath" section of the
FHIR Core spec page (`fhirpath.html`) so that:

1. It explicitly lists **both** allowed code systems for a FHIR `Quantity`
   used as a FHIRPath `Quantity`:
   - `http://unitsofmeasure.org` (UCUM), and
   - `http://hl7.org/fhirpath/CodeSystem/calendar-units` (FHIRPath calendar
     duration units).
2. It removes the locally‑duplicated time‑valued‑quantities table and the
   accompanying "dragon" warning, replacing them with a short paragraph
   ("FHIRPath defines how time‑valued quantities are handled; see …") that
   links into the FHIRPath spec at:
   - `#time-valued-quantities`,
   - `#datetime-arithmetic`,
   - `#toquantityunit--string--quantity`, and
   - `#quantity-equality`.
3. It states normatively that **implementations must not implicitly convert
   between the UCUM and calendar‑units code systems**; conversions only occur
   via explicit `toQuantity(unit)` and only in the directions FHIRPath
   permits.
4. It states that when a FHIR `Quantity` uses a code system other than the
   two above, FHIRPath operations on it return an empty collection (`{ }`)
   rather than raising an evaluation error.

#### Justification

This is a documentation‑only change to the FHIR Core page that mirrors what
the FHIRPath base spec is already converging on (FHIR‑53250 and the
*Definite durations above seconds in date time arithmetic* Zulip thread
both agree that calendar conversion factors are explicit‑only, and that
implementations differ on what to do with unsupported units precisely
because the spec is silent). The two‑code‑system claim is uncontested and
historically documented (Bryn Rhodes, 2019, *FHIR.Quantity versus
System.Quantity*; Bryn Rhodes, 2020, *Date Time operations*). Returning
empty for unknown unit systems is consistent with the rest of FHIRPath's
"unknown / not‑applicable → empty collection" pattern (and is what the
reporter expects). Replacing the duplicated table with a link removes a
known source of drift between the FHIR Core page and the FHIRPath spec —
exactly the drift FHIR‑53250 has had to clean up.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the substantive content of Disposition A, but only after FHIR‑53250 is
applied to the FHIRPath base spec, and frame the FHIR Core change as a
**back‑reference rewrite** rather than a re‑specification:

1. Reduce the FHIR Core "Using FHIR Quantity in FHIRPath" section to a brief
   summary plus deep links into the corresponding FHIRPath sections (one
   bullet per: allowed unit systems, time‑valued quantities, datetime
   arithmetic, `toQuantity`, and quantity equality/equivalence).
2. Move the explicit "no implicit cross‑codesystem conversion" and the
   "unsupported unit system → empty" statements into the FHIRPath spec (as a
   small addendum to FHIR‑53250 or as a new ticket against FHIRPath / ITS),
   and merely cite them from the FHIR Core page.
3. Keep the current "dragon" warning out of the FHIR Core page (as the
   reporter requests) regardless of whether it eventually lives on the
   FHIRPath side.

#### Justification

The trade‑off vs. Disposition A: Disposition A has the FHIR‑I work group
asserting normative behaviour ("must not implicitly convert"; "returns
empty") that arguably belongs on the FHIRPath side, where it can apply to
*every* host — not just FHIR‑Core resources. Putting the rules in two
specifications risks the same drift that produced this ticket in the first
place. Disposition B keeps a single source of truth (FHIRPath), defers the
substantive normative wording to the work group that owns it (ITS), and
reduces the FHIR Core page to a stable summary + links. The downside is
schedule coupling: FHIR‑53950 cannot complete until the matching FHIRPath
text is in place, which means it is unlikely to be resolved in the same
ballot cycle.

---

### Disposition C: Decline

#### Proposal

Decline the request as written. Rationale category: *the underlying spec
issues are already being addressed by FHIR‑53250 against the FHIRPath base
specification; FHIR‑I should not maintain a parallel set of normative
statements on `fhirpath.html`.*

#### Justification

Every substantive issue the ticket raises — implicit calendar‑unit
conversion, duplication of normative tables, missing/unclear behaviour for
unsupported units — is a FHIRPath‑spec issue, not a FHIR‑Core issue. They
are already on the ITS work group's plate via FHIR‑53250 and the
"Definite durations above seconds…" Zulip thread. A FHIR‑Core change is at
best premature and at worst will conflict with what ITS finalises. The right
disposition is to defer to FHIRPath and let the FHIR Core page absorb any
necessary editorial trim‑down through normal documentation maintenance after
FHIR‑53250 ships.

The weakness of this position: even if the normative rules belong in
FHIRPath, the FHIR Core page is currently *demonstrably wrong* on point #1
(it implies UCUM is the only allowed code system) and is the page most
implementers actually read first. Declining leaves that misleading wording
in place until the next maintenance pass.

---

### Recommendation

**Recommended disposition:** B

The ticket reporter is right on every individual point, but the natural home
for "no implicit code‑system conversion" and "unknown unit system → empty"
is the FHIRPath spec, not the FHIR Core page — and the FHIRPath work group
is already mid‑flight on the same conceptual cleanup via
[FHIR‑53250](https://jira.hl7.org/browse/FHIR-53250). Disposition A solves
the FHIR Core page in isolation but creates two parallel sources of
normative text on calendar/UCUM units, which is exactly how the current mess
arose. Disposition C undersells the ticket's value: the FHIR Core page is
genuinely misleading today (it omits the calendar‑units code system
entirely) and the FHIR‑I work group is the only group that can fix that
specific page. Disposition B keeps the FHIR‑I edits to what only FHIR‑I can
do — list the two allowed unit systems, drop the duplicated table and
dragon, and replace them with deep links into FHIRPath — while pushing the
remaining normative statements upstream to FHIRPath/ITS where they belong
and where FHIR‑53250 has already opened the door. Recommend that the work
group accept Disposition B and coordinate with ITS so the FHIR Core
rewrite ships in the same (or the immediately‑following) ballot cycle as
FHIR‑53250.
