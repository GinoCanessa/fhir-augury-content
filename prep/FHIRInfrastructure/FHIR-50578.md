## Summary

[FHIR-50578](https://jira.hl7.org/browse/FHIR-50578) is a Technical
Correction asking the FHIR Core specification to explicitly state that
cross-version extensions (XVer extensions) targeting **modifier
elements** in the source version SHALL themselves be represented as
**modifier extensions** when projected onto another version. The reporter
notes that this constraint is already encoded in the generated XVer
extension `StructureDefinition` resources (so the rule is in effect for
any tooling that consumes those `StructureDefinition`s), but it is not
called out in the narrative spec text. The ticket asks for the spec to
explicitly mention this rule so that implementers reading the
cross-version extension guidance see it stated normatively (or at least
descriptively) in the spec, rather than only being able to discover it by
inspecting the generated extensions.

No cross-referenced Jira tickets were surfaced by the cross-reference
service for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a short, normative-flavoured paragraph to the FHIR Core
extensibility / cross-version-extensions narrative (the `extensibility`
and `versions` page family in the FHIR Core spec) stating:

> When a cross-version extension represents an element whose source
> definition has `isModifier = true`, the cross-version extension SHALL
> itself be defined and used as a **modifier extension** (i.e., placed in
> `modifierExtension` rather than `extension`, with `isModifier = true`
> on its `StructureDefinition`). This mirrors the rule already applied
> when the cross-version extension `StructureDefinition`s are generated.

Cross-link the new sentence from both the cross-version extensions
section and the modifier-extension definition so readers arriving from
either direction find the rule.

#### Justification

The rule is already enforced in the generated extension artifacts, so
this change is purely a spec-narrative clarification — no behavioural
change for conformant tooling. Stating it explicitly closes a real
documentation gap: today an implementer reading only the spec text would
not learn this rule and could (a) hand-author a cross-version extension
on a modifier element as a plain extension or (b) consume one as if it
were non-modifying, both of which would silently violate FHIR's
modifier-element safety contract. The Zulip discussions (XVer Extension
Generation; Cross-Version code with required bindings) show that
implementers and editors actively reason about this case, so a clear
spec statement helps consensus and reduces re-litigation.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only asserting the rule in prose, add it as a documented
**invariant of the cross-version extension generation process** in the
versions / cross-version page, AND surface it in the rendered
`StructureDefinition` intros for cross-version extensions on modifier
elements (e.g., a "This extension is a modifier extension because it
represents a modifier element in `<source-version>`" callout in the
generated narrative). The narrative spec change is the same as in
Disposition A, but is paired with a generator change so the rule is
visibly justified in every affected extension's published page.

#### Justification

Stating the rule in one paragraph of the narrative is easy to miss; the
ticket itself is filed precisely because something true of the artifacts
isn't visible in the spec. Surfacing the rule in each generated
extension's narrative makes it self-documenting wherever an implementer
lands, and keeps the spec text and the artifacts in sync if the rule is
ever revisited. The trade-off is more work (a generator/template change
in addition to spec text), and it touches the cross-version extension
generation pipeline rather than only the narrative.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the basis that the rule is already correctly
encoded in the published cross-version extension `StructureDefinition`s,
and conformant tooling derives behaviour from the `StructureDefinition`s
rather than from prose. Close as "Persuasive with Modifier — no spec
change required; the artifacts are normative."

#### Justification

FHIR generally treats `StructureDefinition` as the source of truth for
extension semantics, and validators / generators consume those artifacts
directly. From a strict conformance standpoint, no implementer following
the artifacts can get this wrong. Declining keeps the spec text leaner
and avoids restating in prose what is already machine-checkable.

---

### Recommendation

**Recommended disposition:** A

The change requested is small, low-risk, and addresses a real
discoverability gap: the rule is true and already enforced by generated
artifacts, but a human reader of the spec narrative cannot see it. Both
Zulip threads above show that the modifier / non-modifier question for
cross-version extensions comes up repeatedly in design discussion, which
is exactly the sort of recurring confusion that a one-paragraph spec
clarification is designed to prevent. Disposition B is attractive but
expands the scope into generator/template work that isn't necessary to
satisfy the ticket's ask; it can be tracked as a follow-up if WG wants
the in-artifact callout. Disposition C ("artifacts are the source of
truth") is technically defensible but ignores that the spec narrative is
the on-ramp implementers actually read first, and the cost of one
sentence is much lower than the cost of the next implementer
rediscovering this the hard way.
