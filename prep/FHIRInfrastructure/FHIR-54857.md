## Summary

This ticket asks the FHIR Infrastructure work group to expand the
Subscriptions framework module page so that it explicitly addresses
authorization in two directions: (1) Server → Subscriber, i.e. how
notifications delivered out from a subscription server should be
authorized/authenticated to the subscriber's endpoint, and (2)
Subscriber → Server, i.e. how follow-up requests a subscriber makes
back to the server (typically to fetch the resources referenced in a
notification, or to manage the Subscription itself) should be
authorized. Today the framework page leaves authorization largely
implicit — concrete guidance is scattered across the
`notification-authorization-hint` extension, the R5 backport IG, and
historical Zulip threads — and implementers consistently report that
they cannot tell from the spec alone how to wire authorization end to
end. There are no cross-referenced Jira tickets, but the request is
strongly motivated by the long-running community discussions
referenced below.

## Details

**Description:**

> The page should include a section with information about authorization,
> specifically regarding the sending of notifications (Server → Subscriber)
> and follow-up requests (Subscriber → Server).

## Related Zulip Discussions

### [subscriptions > OAuth & subscriptions](https://chat.fhir.org/#narrow/stream/subscriptions/topic/OAuth%20%26%20subscriptions)

The original (2019) thread that surfaced the gap this ticket targets.
Participants debated whether the spec should mandate a full
bi-directional OAuth flow for subscriptions or leave it as an
implementation detail. Pascal Pfiffner and John Moehrke argued that
the strength of authorization should scale with payload sensitivity
(ping vs. id-only vs. full content) and that, ideally, the broader
HTTP/OAuth ecosystem — not FHIR — should solve the callback-auth
problem. Gino Canessa noted that the spec at the time only carried an
"Authorization header" slot and that the working assumption was a
server-side step where the Subscription server obtains/holds a token
to present on notification delivery. Jenni Syed pushed for the
authorization mechanism to be more explicit and "less easy to miss"
in the spec — the exact gap FHIR-54857 now targets.

Participants: Gino Canessa, John Moehrke, Pascal Pfiffner, Jenni Syed,
Mohammad Jafari, Isaac Vetter.

### [subscriptions > Subscription PHI leak](https://chat.fhir.org/#narrow/stream/subscriptions/topic/Subscription%20PHI%20leak)

A 2022 thread highlighting that authorization is not just about
transport — even minimal notifications can leak PHI by signalling
that a topic matched. Elliot Silver described an R4 implementation
that strips notifications down to id/type/version/lastUpdated; Gino
Canessa noted that topic-based subscriptions already provide guidance
about empty notifications and that the server is responsible for
checking auth before accepting a subscription, considering policy on
lifetime/renewal. John Moehrke reinforced that auth checks are
necessary but distinct from the cancellation flow. The thread shows
that authorization guidance needs to cover not just headers/tokens
but also when a server should accept or refuse a subscription based
on the subscriber's authorization to see the implied data.

Participants: Gino Canessa, John Moehrke, Elliot Silver.

### [Da Vinci PAS > Subscription authentication](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PAS/topic/Subscription%20authentication)

A 2025 thread showing the gap is still painful for implementers.
November Valentine asked how an EHR is supposed to authenticate that
an inbound notification really came from a given payer, given that
the spec offers no pre-notification OAuth handshake and that a
"secret URL" alone is a weak control. Gino Canessa replied that there
is no single standardized mechanism — jurisdictions/communities use
SMART, UDAP, mutual TLS pinning, or pre-shared keys in headers — and
that the Subscription resource includes a header element so common
patterns work, but standardization would require more community
appetite than has existed so far. Mathew Osmanski then asked the
inverse question (how to ensure only a particular organization can
modify a given Subscription when several orgs share a client), and
Gino pointed at the `notification-authorization-hint` extension and
the R5 backport guidance. Net effect: implementers consistently ask
for exactly what FHIR-54857 proposes — a single page on the framework
that explains both directions and points at the available
extensions/patterns.

Participants: November Valentine, Mathew Osmanski, Gino Canessa,
Eric Haas.

### [workflow > Topic for upcoming call](https://chat.fhir.org/#narrow/stream/workflow/topic/Topic%20for%20upcoming%20call/near/576323431)

Vassil Peytchev flagged "guidance on using the

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level "Authorization" section to the Subscriptions
framework narrative page covering the two directions called out in
the ticket:

1. **Server → Subscriber (notification delivery).** Document the
   `Subscription.channel.header` slot as the spec-supported carrier
   for credentials on rest-hook notifications, list the patterns in
   active use (pre-shared bearer token, SMART/UDAP-issued tokens,
   mutual TLS, signed payloads), and explicitly call out the
   `notification-authorization-hint` extension and how a server is
   expected to obtain/refresh the credentials it uses when delivering
   notifications. Cross-link to the R5 backport IG section that
   already discusses this, and to the channel-type-specific pages
   (rest-hook, websocket, email, message) for channel-specific
   considerations.
2. **Subscriber → Server (follow-up requests and subscription
   management).** Document how the subscriber should authorize
   follow-up `read`/`search` calls triggered by an `id-only` or
   `empty` notification (typically the subscriber's own existing
   authorization context against the server, not the notification
   credential), how the server should authorize creation /
   modification / deletion of a Subscription (including the guidance
   already in topic-based subscriptions that a server SHALL check the
   subscriber's authorization to see the data implied by the topic +
   filters before accepting the subscription), and how to handle
   tenant/organization scoping when multiple organizations share a
   client (the question raised in the PAS thread).

The section should be normative where existing guidance is normative
(e.g., "server SHALL check authorization before accepting a
subscription") and informative where the community has not converged
on a single mechanism (auth flavor on rest-hook delivery).

#### Justification

This is exactly what the reporter, the long-running OAuth &
subscriptions thread, the PHI-leak thread, and the recent Da Vinci
PAS thread all ask for. The information already exists, but it is
fragmented across three places (framework page, R5 backport IG,
authorization-hint extension definition) and an implementer cannot
discover it from the framework page itself. Consolidating it into a
clearly-named section on the framework page is consistent with how
the spec treats other cross-cutting concerns (e.g., security,
versioning) and does not require any resource or schema change, so
it is a low-risk, high-value addition. It is also the lowest-cost way
to honor Jenni Syed's 2019 ask to make the authorization story "more
explicit and less easy to miss."

---

### Disposition B: Alternative Approach

#### Proposal

Rather than building a single new "Authorization" section on the
Subscriptions framework page, refactor the spec's authorization story
across three coordinated touch points:

1. Add a short overview block at the top of the Subscriptions
   framework page that frames the two directions and **links out** to
   the canonical material instead of duplicating it.
2. Promote the substantive guidance into a dedicated
   `subscriptions-security.html` (or `subscriptions-authorization.html`)
   sub-page under the Subscriptions module, organized by channel type
   (rest-hook, websocket, email, message) so channel-specific
   nuances live with the channel.
3. Tighten the `notification-authorization-hint` extension's narrative
   so that it is the canonical reference for how a server signals to
   a subscriber what authorization it needs to present, and link to
   it from both the framework page and the per-channel pages.

This keeps the framework page itself short and discoverable while
giving authorization guidance room to grow as the community
standardizes on specific flows (SMART, UDAP, etc.) without forcing
the framework page to keep expanding.

#### Justification

The Da Vinci PAS thread and the Workflow "Topic for upcoming call"
thread both show that authorization guidance is going to grow — at
minimum to cover the authorization-hint extension's reuse outside
Subscriptions and any future standardized profile (SMART-on-FHIR
subscription delivery, UDAP-secured rest-hook, etc.). A dedicated
sub-page absorbs that growth without bloating the framework page,
and channel-type organization matches how implementers actually
think about the problem (the auth story is meaningfully different
for rest-hook vs. websocket vs. email). The trade-off vs.
Disposition A is more authoring effort up front and more
hyperlink-following for a casual reader; the upside is a structure
that scales.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for the framework page. Rationale category:
**already addressed by existing mechanisms**. Close the ticket with
a note that authorization for subscription notifications is covered
by (a) the `notification-authorization-hint` extension definition,
(b) the Subscriptions R5 backport IG's security guidance, and (c)
the broader FHIR Security page; and that channel-specific
authorization concerns belong on the channel-type pages, not on the
framework page.

#### Justification

A defensible decline argument is that the framework page is meant to
describe the abstract subscription mechanism (topics, filters,
notifications, lifecycle) and that security/authorization is a
cross-cutting concern handled by the FHIR Security page and IGs. The
2019 OAuth & subscriptions thread explicitly raised the option of
deferring to the broader OAuth ecosystem rather than inventing
FHIR-specific machinery, and Gino Canessa has noted in the 2025 PAS
thread that there is "not enough appetite" to standardize on a
single mechanism. Declining keeps the framework page narrow and
avoids documenting patterns that may be obsoleted by a future
standardized profile.

---

### Recommendation

**Recommended disposition:** A

The Zulip evidence over six years (2019 → 2025) is unusually
consistent: implementers do not know, from the framework page alone,
how to wire authorization in either direction, and they keep asking
the same questions. The information needed to answer them already
exists in the spec ecosystem; it just is not where readers look
first. Disposition A is the minimum change that closes that gap, is
consistent with how the spec handles other cross-cutting concerns,
introduces no new normative requirements that would require a
ballot-cycle redesign, and directly satisfies Jenni Syed's 2019 ask
to make the authorization story "more explicit and less easy to
miss." Disposition B is attractive longer-term and the workgroup
should keep it on the table if the section grows quickly, but it
should not block the immediate fix; the sub-page split can happen
later once content has accumulated. Disposition C should be rejected
because the empirical record shows the existing distribution of
guidance is failing implementers — declining preserves a known
problem.
