## Summary

The reporter (Adam Strickland) asks the SMART App Launch specification to add
explicit developer guidance to its worked example launches showing how the
already-required *session binding* of the OAuth `state` parameter (and the
PKCE `code_verifier`) should be implemented in practice. The current spec
already says: *"Apps SHALL generate an unpredictable state parameter for each
user session; SHALL include state with all authorization requests; and SHALL
validate the state value for any request sent to its redirect URL."* The
reporter's point is that, while the normative SHALL exists, the worked
examples do not show developers the mechanic — i.e., creating a *pre-login
session*, storing the `state` value and PKCE `code_verifier` in that session,
and then validating them on redirect. The proposed change is documentation
oriented: edit the example flows in the SMART App Launch IG to call out
session binding inline. The reporter's referenced
[PR HL7/smart-app-launch#392](https://github.com/HL7/smart-app-launch/pull/392)
contains the prototype edits. No directly cross-referenced Jira tickets were
found, but the related Zulip discussion (see below) ties this work to the
earlier PKCE/session-binding conversation that produced FHIR-36762 and
ultimately led the reporter to redirect his original PR toward state-binding
guidance.

## Details

**Description:**

More details in this [PR](https://github.com/HL7/smart-app-launch/pull/392),
but I'd like to add some guidance to developers on this requirement:

> *Apps SHALL generate an unpredictable state parameter for each user session;
> SHALL include state with all authorization requests; and SHALL validate the
> state value for any request sent to its redirect URL.*

The key here is to create a pre-login session, and to store the state
parameter and PKCE code verifier in that session to mitigate certain attacks.
In that PR above, I added explicit guidance to the example launches on how to
do this.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt PR HL7/smart-app-launch#392 (or a refreshed equivalent) and merge
example-level guidance into the SMART App Launch worked examples (standalone
launch and EHR launch). Specifically, in each example flow add an explicit
step showing the app:

1. Creating a server-side pre-login session before redirecting to
   `/authorize`.
2. Storing the freshly generated `state` and PKCE `code_verifier` in that
   session (keyed by the session cookie/identifier).
3. On the redirect to the app's `redirect_uri`, looking up the session,
   validating the returned `state` against the stored value, and only then
   exchanging the `code` (sending the stored `code_verifier`).

Add a short call-out box near the existing SHALL language linking to RFC 6819
§5.3.5 (binding state to user-agent session) and RFC 7636 §7 (PKCE security
considerations), so example readers see *why* the example is structured that
way. No normative text changes — this is purely worked-example and informative
prose.

#### Justification

The change is documentation-only and matches what the community converged on
in the Zulip thread: the SHALL is already present, but implementers are not
seeing the mechanic. Worked examples are the SMART IG's primary teaching
surface, and showing the session-binding step in line with the example is the
lowest-cost, highest-leverage way to reduce CSRF-class mistakes. It also
aligns the IG with the underlying OAuth security BCP and PKCE RFC, which
reviewers in the thread (Mandel) explicitly endorsed as the right anchors.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) editing each worked-example flow, add a
single, dedicated *Security worked example* / *Session binding* page in the
SMART App Launch IG that:

- Explains the session-binding requirement in one place.
- Provides one canonical pseudocode snippet showing the pre-login session,
  `state` storage, `code_verifier` storage, and post-redirect validation.
- Cites RFC 6819 §5.3.5 and RFC 7636 §7 normatively-as-informative.
- Is linked from each worked-example flow with a one-line "see also" pointer
  next to the existing SHALL.

This keeps the worked-example flows lean and avoids duplicating the same
pre-login-session walk-through across every launch variant.

#### Justification

The Zulip discussion surfaced a real concern (Josh Mandel) about *dividing
developers' attention* by sprinkling new prose around topics that are already
covered by SHALL requirements. A single dedicated section is easier to
maintain, easier to keep in sync with the OAuth BCP as it evolves, and lets
each example link out rather than carrying its own copy of the guidance.
Trade-off vs. A: readers who skim only one example may miss the dedicated
page, so the inline pointers must be impossible to overlook. This option also
has lower merge-conflict risk against unrelated example edits.

---

### Disposition C: Decline

#### Proposal

Do not adopt new example-level guidance. Close the ticket as *already
addressed* by the existing normative language (the SHALL on `state`
generation, inclusion, and validation) and by the upstream references
(RFC 6749, RFC 6819, RFC 7636), with a note pointing implementers to those
references.

#### Justification

This was Josh Mandel's initial position in the Zulip thread: PKCE's design
*requires* session-bound state, the SHALL is already in the spec, and
RFC 6819 §5.3.5 already prescribes the mechanic. Adding example-level
guidance risks (a) implying the SHALL was somehow optional before, and
(b) drifting out of sync with the OAuth Security BCP over time. A
maintenance-minimization argument can be made for keeping the IG thin and
deferring to the underlying RFCs.

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach)** — with the inline
example pointers from A.

The Zulip thread is unusually clear: both the reporter and the most active
reviewer (Mandel) ended up agreeing that *session binding of `state`* is the
right framing and that the gap is one of *developer guidance*, not normative
strength. That makes Disposition C (decline) defensible only on a "RTFRFC"
basis, which is not how the SMART IG has historically chosen to teach
implementers — the worked examples exist precisely because pointing at OAuth
RFCs has not been sufficient in practice. Between A and B, B better answers
Mandel's concern about not fragmenting developer attention: a single
authoritative security/session-binding page, cross-linked from every worked
example, gives the reporter the explicit mechanic he is asking for *and*
keeps the IG maintainable. The work in PR #392 should be reshaped as the
seed of that dedicated page rather than as edits scattered across each flow.
