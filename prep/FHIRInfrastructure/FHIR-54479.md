## Summary

The ticket targets the StructureMap specification's "Dependent Rules"
section, which currently describes only how variables passed to a
dependent group must match required parameters "in order" and how the
mode (source vs. target) must align. Brian Postlethwaite points out
that the grammar also allows **literal parameters** (e.g.,
`{ "valueId": "vvv" }` / `'literal'` in FML) to be passed to dependent
groups, but the prose neither specifies how literals are to be handled
nor shows an example. He also asks whether a parameter supplied as a
literal (or via a FHIRPath expression) is treated as a source, a target,
or either, and requests that the spec be explicit about whether a
FHIRPath expression used to populate a dependent-group parameter may
yield a collection (to be iterated by the group's rules in the normal
way) or whether it must be singleton. In short, this is a clarification
request on three points in the Dependent Rules normative text: (1)
literals as dependent parameters, (2) source-vs-target semantics of
dependent parameters, and (3) collection-vs-singleton semantics for
FHIRPath-valued dependent parameters.

No cross-referenced Jira tickets, GitHub PRs/issues, or prior
resolutions were surfaced by the Augury cross-reference lookup, and the
ticket has no comments yet, so the summary above reflects only what the
reporter wrote.

## Details

**Description:**

> In the Dependent Rules section this text:
>
> *"The parameters provided must match the parameters required by the
> dependent group, in order. In addition, the mode of the variable must
> match - inputs that are targets must be target variables. Note,
> though, that target variables can be treated as source for a group."*
>
> Doesn't state how literal parameters should be handled, nor is there
> an example of doing so. The grammar does support doing this.
>
> Are the source or target variables, or can they be used as either?
>
> If a fhirpath expression is used to populate the variable, be clear
> that it cannot produce a collection and pass that into the group, or
> state that it can be a collection, as groups process those in their
> rules normally.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Amend the StructureMap "Dependent Rules" section of the FHIR Mapping
Language page to explicitly cover all three gaps the reporter
identified:

1. **Literal parameters.** State that a dependent group parameter may
   be a literal value (matching the grammar's
   `valueInteger | valueString | valueBoolean | valueId` etc. in the
   JSON/XML form and the corresponding FML literal syntax), in addition
   to a variable name. Add a small worked example (both FML and JSON)
   showing a dependent group invocation that passes a literal as one
   parameter and a variable as another.
2. **Source vs. target semantics of a literal/FHIRPath parameter.**
   Clarify that a literal (or a value produced by a FHIRPath
   expression) is, by construction, a source — it has no identity in
   the caller's scope to be written back to — and therefore can only
   bind to a parameter declared as a source in the dependent group.
   Reinforce the existing rule that named target variables may be
   passed to a source-mode parameter but not vice versa.
3. **Collection semantics for FHIRPath-valued parameters.** State
   explicitly whether a FHIRPath expression used as a dependent
   parameter is required to yield a singleton or whether a collection
   is permitted and iterated by the dependent group in the same way as
   source rules. Pick the behavior that matches the reference
   implementation (likely: singleton required for dependent
   parameters, since the dependent group is invoked once per call
   site), and call it out as a rule with a short rationale.

#### Justification

This is a documentation / clarification change against a section that
already acknowledges parameter-passing rules; the grammar already
permits the constructs in question, so the normative prose is simply
incomplete relative to the grammar. Resolving all three sub-questions
in one edit is more useful to implementers than leaving any of them
open, and closing the gap between grammar and prose is a core editorial
responsibility for FHIR Infrastructure. The Zulip example in
`mapping-framework > How to find a mapping rule` demonstrates that
implementers are already using literal `valueId` parameters in
dependent invocations, which makes documenting the intended semantics
urgent rather than speculative.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding three free-form paragraphs, restructure the
"Dependent Rules" section around a small **parameter-kind table** that
enumerates the allowed parameter forms and their binding rules:

| Parameter form | Allowed parameter mode in dependent group | Collection allowed? | Notes |
|-|-|-|-|
| Named source variable | source or target | per source variable's cardinality | existing behavior |
| Named target variable | source or target | singleton (targets are singletons) | existing behavior |
| Literal (`valueX`) | source only | singleton | new: explicit |
| FHIRPath expression | source only | singleton (must not yield a collection) | new: explicit |

Pair the table with one worked example per row in FML and JSON. Keep
the existing prose about "parameters must match in order" and "mode
must match" but make the table the normative reference for the fine
points the ticket raises.

#### Justification

A table is easier for implementers to scan than additional prose,
reduces the chance of internal contradiction when future changes are
made, and directly mirrors the way the grammar enumerates parameter
kinds. It also produces a single, testable rule set rather than three
scattered clarifications, which is helpful for downstream validators of
`StructureMap.group.rule.dependent.parameter`. The trade-off vs.
Disposition A is purely editorial — the semantic conclusions are the
same — but the table form tends to age better.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the existing prose plus the grammar is
sufficient: dependent parameters inherit the source/target semantics of
the mapping as a whole, and a FHIRPath expression producing a
collection in a dependent-parameter position is already implicitly
disallowed by the general rule that a variable binding is a single
value unless explicitly iterated. No spec change; close as
Persuasive – Not Persuasive or Not Persuasive With Mod, with a comment
pointing at the relevant grammar productions.

#### Justification

The spec leaves a number of such details to the reference
implementation and the grammar; adding more prose risks over-specifying
behavior that implementers currently infer correctly. If no Zulip or
implementer discussion has produced actual interoperability failures
(and none were surfaced by Augury for this ticket), the cost of a
normative edit may exceed the benefit.

---

### Recommendation

**Recommended disposition:** B

Disposition B (table-based clarification) is recommended. The reporter
is pointing at three concrete questions that the normative prose does
not answer, and the grammar genuinely does permit literal and
FHIRPath-valued dependent parameters — implementers are already using
them in the wild (see the `mapping-framework` Zulip example). That
makes Disposition C (decline) hard to justify: the questions are real
and implementer-facing, not hypothetical. Between A and B, the table
form in B produces a tighter, more implementation-testable answer and
avoids the drift that comes from scattering three separate
clarifications across the same paragraph. It also provides a natural
place to document future parameter kinds without further restructuring.
FHIR Infrastructure should confirm with the StructureMap reference
implementation (HAPI / .NET / Matchbox) that the proposed rule
"literals and FHIRPath-valued dependent parameters are source-only and
singleton" matches actual behavior before finalizing the table; if any
implementation allows collections, the table's "Collection allowed?"
column should be adjusted accordingly and the change re-scoped.
