## Summary

The ticket reports that, as part of the R6 changes to the `Medication*`
resource family (notably `Medication`, `MedicationRequest`,
`MedicationDispense`, `MedicationAdministration`, `MedicationStatement`,
and related resources), the structure and element set of those resources
have been revised. Existing search examples in the FHIR Core specification
that reference these resources still use elements/paths from earlier
versions and are therefore no longer valid against the R6 definitions.
The reporter asks that those search examples be updated so that the search
parameters, element paths, and FHIRPath/expression usages they show are
consistent with the current R6 element set.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the cross-reference query for FHIR-49231, so the request is
treated as a stand-alone editorial / technical-correction task scoped to
the search documentation pages that show Medication-resource examples.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a technical correction against the FHIR Core
specification's search documentation. Concretely:

1. Inventory every search example on the search-related pages
   (e.g., `search.html`, the search examples block, and any
   per-resource search example sections) that references a
   `Medication*` resource — `Medication`, `MedicationRequest`,
   `MedicationDispense`, `MedicationAdministration`,
   `MedicationStatement`, and `MedicationKnowledge`.
2. For each example, validate the element paths and search parameter
   names used in the URL/`_query`/`_filter` expression against the
   current R6 element set for the relevant resource.
3. Replace any obsolete element references with the equivalent R6
   element (for example, paths affected by element renames, type
   changes such as `CodeableConcept` ↔ `CodeableReference`, or
   re-nesting of dosage/ingredient structures), and update the
   surrounding prose so that the example explanation matches the new
   path.
4. Where the previous example demonstrated a search that no longer
   maps cleanly onto an R6 element, replace it with an example that
   exercises the closest equivalent R6 search parameter and note the
   substitution in the change log.
5. Run the standard build of the search pages and any example
   validation to confirm the corrected examples parse and resolve
   against the R6 StructureDefinitions.

#### Justification

The ticket is filed as a Technical Correction with no scope ambiguity:
the examples are factually wrong against the current normative content
and need to match it. Search examples are reference material that
implementers copy, so leaving them invalid is actively misleading and
undermines confidence in the spec's accuracy. Correcting examples does
not change normative behavior or break conformance, so the change is
low-risk and well-aligned with FHIR Infrastructure's stewardship of
search documentation.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the corrections from Disposition A, **and** add a lightweight
generation/validation step so this class of drift cannot recur silently:

1. Replace hand-authored Medication search examples with examples
   generated from (or validated against) the R6 search-parameter and
   element definitions during the spec build, so any future rename or
   type change immediately fails the build instead of silently
   producing invalid documentation.
2. Where full generation is impractical, add a build-time check that
   parses the search example URLs, resolves each path/parameter
   against the current StructureDefinitions and SearchParameter
   resources, and errors on unknown elements or parameters.
3. Apply the same mechanism to the other "high-churn" resource
   families (e.g., the medication, medications-administration,
   nutrition, and biologically-derived-product groups) that have
   undergone large R6 reshaping.

#### Justification

The underlying problem — examples becoming silently invalid after a
resource is restructured — is structural, not specific to Medication.
Solving it at the build level prevents the ticket from being filed
again the next time a resource is reshaped, and reduces the long-term
maintenance burden on FHIR Infrastructure. The trade-off versus
Disposition A is up-front tooling work in the spec build, weighed
against repeated manual sweeps across releases.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — already addressed by the
ongoing R6 ballot reconciliation**, on the grounds that the resource
maintainers (Pharmacy and the per-resource owners) update their own
example bundles as part of resolving R6 ballot items, so a separate
sweep by FHIR Infrastructure is redundant and risks conflicting with
in-flight per-resource edits.

#### Justification

Defensible only if the workgroup is confident that the per-resource
owners are systematically validating every search example that
references their resource. In practice this is unlikely to be true for
generic search documentation pages owned by FHIR Infrastructure (as
opposed to per-resource example sets), so the rationale for declining
is thin. The argument would rest on avoiding duplicate edits rather
than on the request itself being wrong.

---

### Recommendation

**Recommended disposition:** A

The ticket is a straightforward, low-risk technical correction against
spec content owned by FHIR Infrastructure (the cross-resource search
documentation), filed by a knowledgeable reporter, and there is no
disagreement to weigh — no Zulip discussion, no linked tickets, no
competing GitHub work surfaced by the cross-reference query. Disposition
A unblocks implementers immediately and is the smallest change that
fully satisfies the request. Disposition B is attractive as a follow-up,
but bundling tooling work into a Triaged technical-correction ticket
risks delaying the fix; if the workgroup wants the validation
machinery, it should be tracked as a separate enhancement so the
example corrections can land on their own schedule.
