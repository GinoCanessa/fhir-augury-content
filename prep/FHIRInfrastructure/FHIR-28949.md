## Summary

Elliot Silver requests that FHIR Shorthand (FSH) / SUSHI provide a built-in
mechanism to include and reuse rules (notably `RuleSet` definitions, and by
extension other reusable FSH constructs such as aliases) that are defined
in other FSH projects, rather than forcing authors to copy or symlink
files between projects. The ticket is short and scoped to FSH itself; no
other Jira tickets were found to be cross-referenced with it. The only
external context is a later Zulip thread on `#shorthand > Reusable
RuleSets` where Jose Costa Teixeira independently raised the same need
and the SUSHI maintainers (Chris Moesel) acknowledged that no such
mechanism exists yet but sketched several possible designs (FSH inside
FHIR packages, FSH-only packages, GitHub-URL dependencies, simple
include directories).

## Related Zulip Discussions

### [shorthand > Reusable RuleSets](https://chat.fhir.org/#narrow/stream/shorthand/topic/Reusable%20RuleSets/near/268981065)

Jose Costa Teixeira asked in January 2022 whether there is a way to
reuse a single FSH file (rulesets, aliases, common conventions) across
multiple IGs. David Pyke and Chris Moesel confirmed that there is no
sophisticated mechanism today — the only options are copying files
into the `fsh/` directory, symlinks (Elliot Silver), or git submodules
(Chris Moesel, with the caveat that submodules are awkward to version
in the FHIR ecosystem). Elliot Silver then linked this Jira ticket
(FHIR-28949) into the thread, noting he had already filed it. Chris
Moesel laid out four possible directions for shareable FSH:
(1) include FSH source inside published FHIR packages,
(2) publish FSH-only packages (registry TBD),
(3) let SUSHI resolve GitHub project URLs as dependencies, or
(4) "something else" — and called out open design questions
(should `dependsOn` distinguish "FSH-only" dependencies so they
aren't republished as IG resources? do authors need to opt in to
consuming FSH from a dependency? what kinds of FSH entities are
shareable — RuleSets only, aliases, everything?). Ward Weistra
suggested simply putting an `/fsh` folder in a package. Jose Costa
Teixeira leaned toward the package + `dependsOn` route but flagged
that SUSHI would need to use those dependencies without re-publishing
them as IG artifacts. Elliot Silver pushed back at the end of the
thread, arguing the package route is over-engineering and that simple
"include directories" would solve the problem. No consensus was
reached; the maintainers explicitly parked the work as
"idea phase" pending bandwidth.

Participants: Elliot Silver, Jose Costa Teixeira, David Pyke, Chris
Moesel, Ward Weistra, Richard Stanley.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a first-class FSH mechanism for consuming reusable FSH definitions
(particularly `RuleSet`s, and also `Alias`es and `Invariant`s) defined
in another FSH project. Concretely, extend SUSHI / FSH so that an
entry in `sushi-config.yaml`'s `dependencies` (or a new sibling key
such as `fshDependencies`) may point at a FHIR package that ships its
FSH source under a conventional path (e.g. `package/fsh-source/`),
and update the FSH compiler to load those files into the symbol table
without re-emitting their generated FHIR resources into the dependent
IG. Update the FHIR Shorthand specification to define: (a) the
`RuleSet`/`Alias`/`Invariant` resolution order across local FSH and
imported FSH, (b) name-collision and shadowing semantics, and (c) the
package-layout contract that publishers must honour for their FSH to
be importable.

#### Justification

This is what the reporter literally asked for, what Jose Costa
Teixeira independently asked for in `#shorthand > Reusable RuleSets`,
and what Chris Moesel identified as the most natural fit with the
existing FHIR packaging story. Reusing the existing `dependsOn` /
package machinery keeps versioning, registry distribution and
provenance consistent with how every other FHIR artifact is shared,
and avoids inventing a parallel "FSH-only" distribution channel. It
also avoids the brittle workarounds in use today (copying files,
symlinks, git submodules), which Chris Moesel explicitly called out
as not appropriate for the broader FHIR ecosystem.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or as a smaller first step toward) full package-based
reuse, add a lightweight `include` / `import` directive to FSH that
points at one or more local or remote FSH source paths (e.g.
additional folders configured in `sushi-config.yaml` under a
`fshIncludes:` key, or a `Include: path/to/file.fsh` line at the top
of a `.fsh` file). SUSHI would parse those files into the same symbol
table as the project's own FSH but would not treat their definitions
as IG artifacts to be published. Document the include resolution
order, cycle handling, and the rule that included files cannot
themselves declare profiles/extensions intended for publication unless
explicitly opted in.

#### Justification

This is the position Elliot Silver took at the end of the Zulip
thread ("the package route is over-engineering ... go with some
simple include directories"), and it materially lowers the
implementation burden: no IG Publisher changes, no package-format
changes, no registry changes, and no need to decide today which FSH
entity kinds are "shareable". It also gives authors an immediate
escape from copy/symlink workarounds. The trade-off versus
Disposition A is weaker provenance and versioning — included files
travel as raw source rather than as versioned FHIR packages — but
that may be acceptable for the RuleSet/Alias use cases that motivated
the ticket, and Disposition A could be layered on later without
breaking the include mechanism.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Not Persuasive — workaround exists / out of scope
for the FSH spec". Note that authors today can already share FSH
across projects by (a) copying files into the `fsh/` directory,
(b) using OS-level symlinks, or (c) using git submodules to vendor a
shared FSH library, and that any richer mechanism is a SUSHI tooling
concern rather than a change to the FHIR Shorthand specification.

#### Justification

The ticket is two sentences with no concrete proposal, no design, no
required cardinalities, and no implementer use cases beyond
"reuse would be helpful". The Zulip thread shows that the SUSHI
maintainers themselves have not converged on a design and have
explicitly parked the work, which makes adopting a normative FSH
change premature. The existing workarounds, while clunky, are
functional, and standardising a half-formed mechanism risks locking
the spec into a design that will need to be revisited once the
package-vs-include question is properly answered. Declining keeps
the door open for a future, better-scoped change request once the
SUSHI team has prototyped one of the options.

---

### Recommendation

**Recommended disposition:** B

The Zulip thread makes clear that there is real demand for FSH reuse
but no community consensus on the package-based design proposed in
Disposition A, and that the SUSHI maintainers have deferred the
package work due to its scope (IG Publisher changes, package-format
changes, opt-in semantics, registry questions). Disposition B
delivers most of the practical value the reporter asked for, can be
implemented entirely within SUSHI/FSH without touching the IG
Publisher or the package format, and is the option Elliot Silver
himself ultimately argued for. It is also forward-compatible with
Disposition A: if and when shareable FSH-in-packages is properly
designed, the include directive can be retained for local reuse and
deprecated or repurposed for resolving package-sourced FSH.
Disposition C is defensible on procedural grounds but ignores the
fact that the same need has now been raised twice, two years apart,
by different people.
