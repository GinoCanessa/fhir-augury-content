## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Jira `FHIR` project's version/artifact configuration for the `Shorthand (FHIR)` specification component to include `1.0.0` in the `Affects Version/s` and `Fix Version/s` picklists. This is a Jira administrative task performed by the FHIR Infrastructure Jira admin(s) against the hl7.org Jira instance; it does not require any code or specification change. After the update, confirm that `1.0.0` is selectable on new change requests filed against Shorthand (FHIR), backfill the version on any in-flight tickets where applicable, and resolve this ticket with `Done`.

#### Justification

The request is a routine housekeeping action that keeps the tracker's version list aligned with the actual release history of FHIR Shorthand. Without the `1.0.0` entry, implementers cannot accurately report which version a bug affects or target a fix version, which degrades the usefulness of the tracker and obscures release-specific triage. The change has no spec-level, normative, or implementation impact, and the cost to perform it is trivial.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than handling this as a one-off manual Jira edit, take the opportunity to formalize a lightweight release-to-Jira process for the Shorthand (FHIR) specification: at each SUSHI / FHIR Shorthand release, the release manager (or an automated script run against the Jira REST API) adds the newly published version to the `Shorthand (FHIR)` component's version list and marks the previous version as "Released". As part of adopting the process, add `1.0.0` (and any other missing released versions) now, document the procedure in the FHIR Infrastructure / FSH release checklist, and then close this ticket.

#### Justification

FHIR-29400 is almost certainly not a one-time problem — the same request will recur with every Shorthand release unless the process is institutionalized. Building the version-list update into the release checklist (or automating it) eliminates an entire class of "please add version X" tickets and keeps the tracker in sync without relying on external reporters to notice the gap. The incremental burden over Disposition A is small and is amortized across all future releases.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that this is not a specification change request and therefore should not be tracked as a FHIR change request. Close the ticket as `Won't Do` / out of scope, and redirect the reporter to file a Jira administration request through the standard HL7 Jira admin channel (helpdesk / Confluence admin ticket) rather than through the FHIR change-tracker workflow.

#### Justification

The FHIR change tracker is intended to capture proposed changes to FHIR specifications, not edits to Jira's own configuration. Accepting administrative-only tickets here dilutes the tracker's signal and creates workflow ambiguity (e.g., who is the "assignee" for a Jira config change, and what does "Applied" mean?). Routing these to the Jira admin channel keeps the change tracker focused on specification work.

---

### Recommendation

**Recommended disposition:** A

The request is trivially small, clearly beneficial, and blocks no other work. Declining on process-purity grounds (Disposition C) would force the reporter to re-file the same request elsewhere for no net benefit, and Disposition B's process-improvement angle, while worthwhile, is scope creep for a ticket that has sat at `Triaged` since 2020. The pragmatic path is: apply the version-list update (Disposition A) now, and — if the work group wants — capture Disposition B's checklist improvement as a follow-up action item for FHIR Infrastructure / the FSH release manager, without holding this ticket hostage to it.
