## Summary

The reporter (Elliot Silver) flags what appears to be a typo in the FHIR
Shorthand (FSH) specification's worked example for *soft indexing* on
`HumanName`. The current example shows three rules — two assignments of
`given` and one of `family` — all of which use `name[+]` as the soft index
into the `name` repeating element. The reporter argues that the second
`given` assignment should re-use the *current* name instance via `name[=]`
rather than start a new one with `name[+]`; otherwise the example creates
two distinct `HumanName` entries, each with one given name, instead of
one `HumanName` with two given names ("Robert" and "David") and a family
name "Smith". This is a documentation correctness issue in the FSH
specification's soft-indexing tutorial, not a tooling/grammar change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the FHIR Shorthand specification's *Soft Indexing* page so that the
worked example for assigning two given names plus a family name to a
single `HumanName` reads:

```
* name[+].given[+] = "Robert"
* name[=].given[+] = "David"
* name[=].family   = "Smith"
```

That is, change the second rule's `name[+]` to `name[=]`. The first
rule advances the `name` soft index (creating a new `HumanName`
instance) and starts its `given` list; the second rule must stay on the
*same* `HumanName` (`name[=]`) while advancing the `given` index
(`given[+]`) so that "Robert" and "David" sit on one name. The third
rule already uses `name[=]` correctly. No grammar or tooling changes are
required — this is a pure documentation correction. (Per the workflow,
the actual file path lives in the FSH spec source repository, not yet
surfaced by FhirAugury for this ticket.)

#### Justification

The reporter is correct as a matter of FSH semantics: the `[+]` soft
index advances the slice/instance counter for the *element it is
attached to*, so `name[+].given[+]` on consecutive rules creates two
separate `HumanName` entries, each carrying a single given name. To
build the intended single `HumanName` with two given names, the second
rule must reuse the most-recent `name` index via `name[=]`. This
matches the well-established FSH semantics described elsewhere on the
same page and the third rule (`name[=].family = "Smith"`) already
exhibits the correct pattern. The fix is small, low-risk, internally
consistent, and directly improves the tutorial's usefulness for
implementers learning soft indexing.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the corrected rule line as in Disposition A, **and** expand the
section with a short callout that explicitly contrasts the two
behaviours, e.g.:

> Note: `name[+].given[+]` advances *both* indices, producing a new
> `HumanName` for each rule. Use `name[=].given[+]` to add another
> `given` to the *same* `HumanName`.

Optionally add a second worked example showing the "wrong" version and
its FHIR-instance result alongside the correct version, so future
readers see both shapes side-by-side rather than just the corrected
output.

#### Justification

The original mistake is exactly the kind of trap the soft-indexing
notation is designed to make easy — and easy to get wrong. The reporter
caught it; future readers may not. A targeted clarification at the
point of the example reduces the chance of the same confusion
recurring, without changing FSH itself. The cost is a few extra lines of
prose plus (optionally) a contrasting example. Trade-off vs.
Disposition A: slightly more editorial work and review surface, but
materially better as teaching material.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the ticket targets a single
illustrative line in tutorial prose for a specification (FHIR
Shorthand) that may have moved on substantially since 2021-01-29, and
ask the reporter to re-verify against the current published FSH spec
before any edit is made — closing the ticket as "Not Persuasive" /
stale unless the issue is reproduced against the current text.

#### Justification

The ticket has sat in `Triaged` since January 2021 with no comments,
no cross-references, no Zulip discussion, and no linked PRs or commits
in any indexed source. FSH has had multiple releases in the interim
and the page in question may already have been rewritten. Acting on a
four-year-old report of a single typo without re-verifying the current
text risks "fixing" something that no longer exists or, worse,
re-introducing a stale example structure on top of a newer one.

---

### Recommendation

**Recommended disposition:** A (with the option to escalate to B).

The reporter's analysis of FSH soft-indexing semantics is correct on
the face of the snippet quoted in the ticket: `name[+]` on consecutive
rules creates separate `HumanName` instances, which defeats the stated
goal of giving one person two given names. The fix is a one-character
change to a single rule and is fully consistent with the third rule of
the same example. There is no design controversy, no breaking-change
risk, and no implementation cost beyond the spec edit itself.
Disposition C's "stale ticket" concern is reasonable as a process
matter, but should be handled by *verifying the current text and then
applying the correction* rather than by declining outright; if the
typo has already been fixed, the ticket can be closed as "Already
Fixed" — but that is a verification step, not a reason to refuse the
correction. Disposition B is a sensible upgrade if the work group
wants to invest a little more editorial effort to harden the tutorial
against the same confusion in the future.
