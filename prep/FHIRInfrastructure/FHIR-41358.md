## Summary

The reporter is asking how to support **rich-text presentation** (bold, italic,
underline, colored text, embedded objects, etc.) for the prompt text rendered
to a user inside a `Questionnaire` — for `Questionnaire.item` of type `group`,
`display`, and standard question items — and, by extension, in the corresponding
`QuestionnaireResponse`. They observe that `Questionnaire.item.text` is typed
as `string`, which (as currently constrained) does not natively carry markup,
and they are unsure whether FHIR offers a sanctioned mechanism to attach
rich-text formatting to question/display prompts. The ticket is filed as a
Question (not a change request), seeking guidance on whether the requirement
can be met today and, if so, how.

There are no Jira tickets, Zulip threads, or GitHub items cross-referenced to
this ticket in the FHIR Augury index, so the request stands on its own.

## Proposed Dispositions

Because this is filed as a **Question**, and not as a change request against
the spec, the dispositions below are framed as the workgroup's response to the
reporter rather than as concrete spec edits. None of the dispositions imply
authoring changes in a specific repo / authoring root, so no per-repo briefing
gotchas apply.

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a *question answered* and respond with a precise pointer
to the existing FHIR mechanisms that already let a `Questionnaire` carry
formatted prompt text without changing the spec:

1. Keep `Questionnaire.item.text` as the plain-text fallback (its current
   `string` typing is intentional — every renderer must be able to display
   *something* even if it does not understand formatting).
2. Use the standard FHIR **rendering** extensions on `Questionnaire.item`
   (and on `Questionnaire.item._text` where applicable) to carry richer
   presentation:
   - `http://hl7.org/fhir/StructureDefinition/rendering-xhtml` —
     attaches an XHTML version of the text for renderers that can display HTML.
   - `http://hl7.org/fhir/StructureDefinition/rendering-markdown` —
     attaches a Markdown version of the text.
   - `http://hl7.org/fhir/StructureDefinition/rendering-style` and
     `rendering-styleSensitive` — attaches CSS-style hints (bold, italic,
     color, etc.) to the text.
3. Direct the reporter to the **Structured Data Capture (SDC) IG** for the
   end-to-end pattern (form-filler behavior, fallback rules, embedded
   images via `itemMedia`, etc.) and for guidance on what real-world
   form-fillers (e.g., LHC Forms, NLM Form Builder) actually render.
4. Close the ticket as **Answered – Question** and steer further discussion
   to the `#questionnaire` Zulip stream as the existing comment already
   suggests.

#### Justification

The capability the reporter wants already exists in normative FHIR via the
rendering extension family on `string`/`markdown` primitives, and the
SDC IG documents the implementation pattern in depth. There is therefore
nothing to change in the base spec; the right action is to give a clear,
linkable answer so future implementers asking the same question land on
the canonical guidance. This is also consistent with how FHIR Infrastructure
typically handles "How do I…?" tickets that have a complete answer in the
existing artifacts.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to answering the question (per Disposition A), open a
**follow-up documentation work item** against the Questionnaire resource
page (and/or the SDC IG) to add a short, named subsection — e.g.,
"Formatted prompt text" — that:

- States explicitly that `Questionnaire.item.text` is a plain-text fallback
  by design.
- Lists the supported rendering extensions (`rendering-xhtml`,
  `rendering-markdown`, `rendering-style`, `rendering-styleSensitive`) with
  a minimal worked example for an `item` of type `display` and one of type
  `group`.
- Cross-links to the SDC IG's form-filler rendering rules and to
  `itemMedia` for embedded images.

The Questionnaire resource itself would not change; only the
narrative/intro text would gain a discoverable answer to this exact
question.

#### Justification

This ticket is the third+ "can Questionnaire show bold text?" question to
land on FHIR Infrastructure, which is a signal that the answer, while
technically present, is not discoverable from the resource page alone —
implementers reach for `Questionnaire.item.text` first, see `string`, and
file a ticket. A named documentation block costs little, removes a
recurring class of questions, and improves the on-ramp for SDC without
introducing any new conformance surface.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive – out of scope for a tracker item**:
the request is a usage question rather than a defect or change proposal,
the existing comment already redirected the reporter to chat.fhir.org,
and the Jira tracker is not the right venue for implementation Q&A.
Take no spec or documentation action.

#### Justification

The ticket is filed as `Question`, has no proposed change, no use-case
beyond "user wants bold text", and no engagement from the reporter since
the redirect comment in July 2023. FHIR Infrastructure's tracker
discipline reserves work for actionable change requests; usage questions
are explicitly handled on Zulip. Declining is defensible on the grounds
that (a) the mechanism already exists, (b) the reporter has been pointed
at the right forum, and (c) leaving the ticket open consumes review
bandwidth without producing a spec change.

---

### Recommendation

**Recommended disposition:** B

Disposition C is technically correct on tracker hygiene, but it leaves the
underlying discoverability problem in place — implementers will keep
filing the same question because the Questionnaire resource page does not
visibly point them at the rendering extensions. Disposition A answers
this reporter but does the same. Disposition B costs only a small
narrative addition (no normative or schema change), gives FHIR
Infrastructure a concrete artifact to point at when the next "how do I
bold this?" question arrives, and aligns with how SDC is positioned as
the deep-dive companion to the base Questionnaire resource. The
workgroup should answer this ticket with the rendering-extension
guidance and spin off (or attach) a small documentation work item
against the Questionnaire resource page / SDC IG.
