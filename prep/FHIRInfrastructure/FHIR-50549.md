## Summary

The reporter notes that the "Search this specification" link rendered on
HL7 FHIR specification pages is broken. Following the link returns an
error page reading: "Oops! There appears to be an error! The webmaster
will be notified." The failure is reproducible across multiple published
versions of the spec, including the current build at
`https://hl7.org/fhir/search.cfm` and the R5 variant at
`https://hl7.org/fhir/R5/search.cfm`. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were found through the FhirAugury index,
so this appears to stand alone as an infrastructure / publication-site
issue rather than a specification content defect.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a site-operations defect and route it to the HL7
webmaster / publication-site team to restore the `search.cfm` endpoint
behind the "Search this specification" link. Concretely:

1. Confirm reproduction against `https://hl7.org/fhir/search.cfm`,
   `https://hl7.org/fhir/R5/search.cfm`, and a representative sample of
   other published versions (R4, R4B, R6 ballot/build, etc.).
2. Have the webmaster diagnose the ColdFusion `search.cfm` handler
   (server logs, search-index backing store) and restore service so
   that submitting a query returns a results page scoped to the
   spec version under which the link was clicked.
3. Once the endpoint is restored, FHIR-I closes the ticket as fixed,
   with no change to the published specification content itself.

No FHIR Core authoring change is required under this disposition; the
fix is entirely in the HL7 site infrastructure that serves
`*/search.cfm`.

#### Justification

The ticket is a "Technical Correction" filed at "Highest" priority
against the FHIR Core specification, but the actual failure mode is
purely operational: the link target HTML is correct, and the failure
occurs server-side at `search.cfm`. The least-invasive fix that fully
addresses the reported behaviour is to restore the existing endpoint
rather than redesign the search experience. This also preserves
deep links from older published versions that already point at
`*/search.cfm`.

---

### Disposition B: Alternative Approach

#### Proposal

Replace the legacy ColdFusion `search.cfm` mechanism with a
client-side or statically-hosted spec-search experience embedded in the
publication. Concretely:

1. Update the publication template (the footer / header partial that
   currently emits the "Search this specification" link) so that the
   link points at an in-page search UI shipped with each published
   version — e.g., a `search.html` page backed by a pre-built
   Lunr/MiniSearch index produced at publication time, or by an
   embedded Google Programmable Search Engine scoped to that version's
   URL prefix.
2. Have the IG publisher / spec publication tooling generate the
   search index (or emit the `search.html` shell) as part of the
   normal build, so every future version ships with a working,
   self-contained search.
3. Keep `search.cfm` as a thin redirect to the new `search.html`
   for the current and recent published versions, so existing deep
   links continue to work.

#### Justification

The current outage exposes a structural fragility: a single
server-side ColdFusion endpoint is in the critical path for "search
this specification" across every historical version of FHIR. A
client-side or statically-built search makes the spec self-hosting for
that capability, eliminates webmaster involvement when something
breaks, and ages with the published artefact rather than with HL7's
hosting platform. The trade-off is a one-off investment in the
publication tooling and a slightly larger published footprint per
version, in exchange for removing a recurring class of "site is down,
spec link is broken" tickets.

---

### Disposition C: Decline

#### Proposal

Decline the ticket against FHIR Infrastructure on the basis that it is
not a defect in the FHIR specification itself but an HL7 site / hosting
incident. Close as "Not Persuasive" or "Not Related" and ask the
reporter to refile with HL7 IT / the webmaster, who own the
`hl7.org/fhir/search.cfm` endpoint.

#### Justification

FHIR-I owns the content and structure of the FHIR specification, not
the HL7 web hosting environment that serves `*.cfm` endpoints. If the
WG's policy is to keep the Jira backlog focused on spec-level defects
and route operational issues through HL7 IT, then this ticket belongs
in that operational queue rather than in FHIR-I's tracker.

---

### Recommendation

**Recommended disposition:** A

This is a high-visibility, "Highest" priority defect against a link
that is rendered on essentially every page of the FHIR specification,
and the reporter has shown that it fails on both the current build and
R5. The right immediate action is to get `search.cfm` working again
(Disposition A) so the published spec stops showing a broken link to
every reader. Disposition B captures a real, longer-term resilience
improvement and should be considered as a follow-up — ideally tracked
as a separate publication-tooling ticket against the spec build —
rather than as a substitute for fixing the live outage. Disposition C
under-serves implementers given the impact and the fact that FHIR-I
is the natural escalation point for "the FHIR site is broken in a way
that affects all readers", even when the underlying fix lives with HL7
IT.
