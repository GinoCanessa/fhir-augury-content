## Summary

Brian Postlethwaite is asking the FHIR Infrastructure work group to make
the FHIR Core specification (specifically the FHIRPath page,
`fhirpath.html#variables`) explicit and unambiguous about how the
FHIRPath environment variables `%resource`, `%context`, `%rootResource`,
and the system variable `$this` are initialised when evaluating
expressions defined inside FHIR artifacts (most notably invariants on
`StructureDefinition.snapshot.element.constraint`, but also search
parameters and other FHIRPath-bearing elements).

The current spec text on these variables is short and has been read
inconsistently by implementers. Two separate Zulip threads on
`#fhirpath` show that even active fhirpath engine authors (Brian
Postlethwaite, Grahame Grieve, Lloyd McKenzie, Gino Canessa, Evgeny
Mukha, Paul Lynch) had to work out the intended semantics from first
principles, and several reference implementations
(java-validator/HAPI, Firely, fhirpath.js, fhirpath-lab static checker)
have historically disagreed on edge cases — particularly:

- `%resource` and `%rootResource` for invariants whose context path is
  `Bundle.entry.resource`, `Parameters.parameter.resource`,
  `Bundle.entry.resource.outcome`, etc. (i.e. where a resource is held
  inline by another resource, but **not** via `DomainResource.contained`).
- The relationship between `%context`, `%resource`, and `$this` for
  invariants defined deep inside an element path.
- Whether `%rootResource` is "always the outermost serialized resource"
  (Brian's prior assumption, also a common implementer assumption) or
  is "the same as `%resource` unless the focus is a `contained`
  resource" (Grahame's clarification, which is closer to the current
  spec wording).

The disagreement was resolved by direction in the 2023 thread
(`%rootResource` only differs from `%resource` for `DomainResource.contained`)
and reaffirmed in the March 2026 thread, after which Brian explicitly
said: *"I'll log ticket(s) to clear this up in both specs."* This
ticket is that follow-through for the FHIR Core spec; a parallel
ticket against the FHIRPath spec itself is also implied.

## Related Zulip Discussions

### [fhirpath > ✔ Context specific FHIRPath invariant](https://chat.fhir.org/#narrow/stream/fhirpath/topic/%E2%9C%94%20Context%20specific%20FHIRPath%20invariant)

This March 2026 thread is the direct precursor to FHIR-55378 — Brian
Postlethwaite's closing comment in the thread is *"I'll log ticket(s)
to clear this up in both specs."* Corey Spears asked how to write an
invariant on `Condition.subject` that is conditional on whether the
`Condition` is being validated standalone vs. as a `Bundle.entry`. The
group debated whether `%rootResource` could be used to detect "I am
inside a Bundle"; Brian initially asserted yes, Grahame stated
authoritatively that `%rootResource` only differs from `%resource` for
`DomainResource.contained`, so `%rootResource is Bundle` cannot be true
for a `Condition` carried in a `Bundle.entry`. Conclusion: invariants
cannot detect bundled context that way; the right mechanism is either
an invariant **on the Bundle** or `ElementDefinition.type.aggregation =
bundled` (Grahame ultimately agreed to make that aggregation check
retrospective in the IG Publisher). Brian's commitment to log spec
clarification tickets came out of this exchange.

Participants: Corey Spears, Paul Lynch, Brian Postlethwaite, fang jun,
Grahame Grieve, Eric Haas (referenced).

### [fhirpath > FHIRPath %resource variable](https://chat.fhir.org/#narrow/stream/fhirpath/topic/FHIRPath%20%25resource%20variable)

The original July 2023 deep-dive into exactly the variables FHIR-55378
calls out. Evgeny Mukha (writing a new FHIRPath engine) found that
`%resource`, `%rootResource`, and `%context` behave inconsistently
across reference engines and that the spec text is sparse. After back
and forth, Grahame Grieve gave the canonical answer: `%context`,
`%resource`, and `%rootResource` are fixed before evaluation starts and
never change during evaluation; `%resource` is the resource that
contains the element which is the focus of evaluation; `%rootResource`
equals `%resource` **unless** the focus is inside
`DomainResource.contained`, in which case `%rootResource` is the
container; for invariants whose context path traverses
`Bundle.entry.resource` or `Parameters.parameter.resource`, the
invariant runs *in the context of the outer resource*, so `%resource`
is the Bundle/Parameters, not the inner resource. The thread concluded
with Grahame opening
[FHIR/fhir-test-cases#140](https://github.com/FHIR/fhir-test-cases/pull/140)
and a fix in
[hapifhir/org.hl7.fhir.core#1353](https://github.com/hapifhir/org.hl7.fhir.core/pull/1353)
to align the Java validator. Importantly, this clarification was never
folded back into the FHIR Core spec page — that gap is what FHIR-55378
asks to close.

Participants: Evgeny Mukha, Gino Canessa, Lloyd McKenzie, Brian
Postlethwaite, Grahame Grieve, Josh Mandel, Paul Lynch.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the **Variables** section of `fhirpath.html`
(`source/fhirpath.xml` in the HL7/fhir spec authoring source) to give
each FHIRPath environment / system variable a complete, normative
definition, plus a worked example for every distinct evaluation
context FHIR uses. Specifically:

- `%context` — the node FHIR passes to the FHIRPath engine as the
  initial focus. For an invariant on
  `StructureDefinition.snapshot.element[X].constraint`, `%context` is
  the node identified by that element's path (e.g. `Patient`,
  `Patient.contact`, `Bundle.entry.resource`).
- `%resource` — the resource that contains the element which is the
  focus of evaluation. For invariants defined on a resource's own
  elements, `%resource` is that resource; for invariants whose context
  path crosses an inline resource boundary
  (`Bundle.entry.resource`, `Parameters.parameter.resource`,
  `Bundle.entry.resource.outcome`), `%resource` is the **outer**
  resource (Bundle, Parameters), not the inner one.
- `%rootResource` — equal to `%resource` **except** when the focus is
  inside `DomainResource.contained`, in which case `%rootResource` is
  the containing `DomainResource`. Explicitly call out that
  `%rootResource` does **not** "walk up Bundles or Parameters".
- `$this` — the current iteration value when an expression descends
  into a collection (e.g. inside `where(...)`, `select(...)`,
  `repeat(...)`); equal to `%context` at the start of evaluation.

Add a small table mapping the four canonical contexts (a) plain
resource invariant, (b) invariant on a path inside a resource, (c)
invariant whose path traverses `DomainResource.contained`, (d)
invariant whose path traverses `Bundle.entry.resource` /
`Parameters.parameter.resource`, to the values of `%context`,
`%resource`, `%rootResource`, and `$this`. Cite the worked
example(s) used to repair the Java validator in
[`FHIR/fhir-test-cases#140`](https://github.com/FHIR/fhir-test-cases/pull/140)
to ensure the spec text and the test suite agree. Repeat the
clarification on the Search Parameter expression page where applicable.

#### Justification

This is exactly the gap Brian named in the precursor Zulip thread, and
it was the working consensus reached by Grahame, Brian, Gino, Lloyd,
and Evgeny in 2023. Reference implementations (java-validator, Firely,
fhirpath.js, fhirpath-lab) have already been brought into alignment
with this model, so updating the prose is a low-risk, additive
clarification — there is no behavioural change for conformant engines.
Doing nothing perpetuates the situation where IG authors (e.g. Corey
Spears) reach for `%rootResource` to detect "am I in a Bundle" and
lose days to the resulting confusion.

---

### Disposition B: Alternative Approach

#### Proposal

Make the clarification jointly with the **FHIRPath specification**
rather than only in FHIR Core. Concretely:

1. Lift the formal definitions of `%resource` and `%rootResource` (and
   the constraint that they are immutable for the lifetime of an
   evaluation) into the FHIRPath spec itself, since these are the
   variables the FHIRPath engine sees. Open a parallel ticket against
   the FHIRPath spec (Brian's "in both specs" comment).
2. Keep the FHIR Core `fhirpath.html#variables` section as the place
   that defines **how FHIR sets** those variables for each FHIR
   evaluation surface (invariants, search parameters,
   GraphDefinition links, Questionnaire/SDC, mapping language, etc.).
3. Add a non-normative "FHIRPath context for invariants" worked-example
   page (or appendix) to the FHIR Core spec that walks through the
   four canonical context patterns above with real example resources,
   and link to it from `StructureDefinition`,
   `ElementDefinition.constraint`, and `SearchParameter.expression`.
4. As a complement to the prose, also propose a new
   `ElementDefinition.type.aggregation` code (e.g. `bundled-if-bundled`,
   per Grahame's comment) so authors no longer need to write
   conditional FHIRPath at all for the most common motivating use case.

#### Justification

The underlying problem Corey Spears actually had in the Zulip thread
was "how do I express *check this only when bundled*", and Brian/Grahame
agreed that no FHIRPath variable can answer that question — the right
fix is structural (aggregation type or invariant-on-Bundle), not
prose. Splitting the work across both specs also addresses the fact
that some of the affected text (immutability of `%resource` /
`%rootResource` during evaluation) really belongs to FHIRPath, not
FHIR. The trade-off vs. Disposition A is scope: this is a larger
piece of work touching two specifications and (optionally) introducing
a new aggregation code, but it produces a cleaner long-term factoring
and reduces future "how do I detect bundled?" questions.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Persuasive with Modifier — No Change*, on the basis
that the existing text at `fhirpath.html#variables` together with the
worked invariant test cases shipped in
[`FHIR/fhir-test-cases`](https://github.com/FHIR/fhir-test-cases) is
already sufficient, and that the apparent disagreement among
implementers is a transient implementation issue (validator bugs that
have since been fixed) rather than a spec defect. Direct future
questions to the FHIRPath chat stream and to the Java validator's
expected-results test fixtures.

#### Justification

Defensible only if the work group believes the current short
definitions are unambiguous when read carefully, and that the cost of
re-balloting the FHIRPath text outweighs the benefit. The 2023 thread
*did* eventually converge without any spec change. However, the very
existence of FHIR-55378 — raised by an experienced FHIRPath
implementer/work-group co-chair (Brian Postlethwaite) after the same
confusion recurred in 2026 — is strong evidence against this position.
This disposition should only be selected if the work group also
commits to a much more visible link from the spec to the test cases.

---

### Recommendation

**Recommended disposition:** A

The reporter is the person who has done most of the practical
exploration of these variables (he maintains fhirpath-lab and
co-authored several reference fixes); his ticket is explicitly a
follow-through on a public commitment to "clear this up in both
specs" after a recurring confusion in 2026 over text that was already
clarified in chat in 2023. Disposition A captures everything Brian
asked for, is additive (no breaking change), is consistent with the
behaviour reference engines were aligned to in
[`hapifhir/org.hl7.fhir.core#1353`](https://github.com/hapifhir/org.hl7.fhir.core/pull/1353)
and [`FHIR/fhir-test-cases#140`](https://github.com/FHIR/fhir-test-cases/pull/140),
and unblocks IG authors writing context-sensitive invariants.
Disposition B is the right *long-term* shape and the WG should
strongly consider opening the parallel FHIRPath-spec ticket and the
`bundled-if-bundled` aggregation request as separate follow-ups, but
those should not gate FHIR-55378 itself. Disposition C ignores
documented evidence that the current text is misread by careful
implementers and is not recommended.
