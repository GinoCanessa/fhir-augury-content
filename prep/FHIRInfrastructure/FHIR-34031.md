## Summary

The ticket asks the SMART on FHIR / FHIR Infrastructure work group to
define a standard, RFC 7591–conformant extension parameter for OAuth 2.0
Dynamic Client Registration so that a SMART app can convey one or more
fixed, fully-specified EHR-launch URLs as part of its registration request
to the authorization server. The need arises because SMART App Launch
1.0.3 (§ "Registering a SMART App with an EHR") already requires apps to
register zero or more launch URLs with the EHR authorization server, but
the SMART specification does not currently say *how* those launch URLs
should be transmitted when the registration is performed dynamically via
RFC 7591 instead of through a manual developer-portal flow. The ticket
was filed by Alexander Zautke (Firely) immediately after a Zulip
discussion in `#smart` ("Dynamic Client Registration Launch Url"), in
which Josh Mandel suggested registering an extension parameter such as
`smart_ehr_launch_uri` (an array of strings); Alexander notes in that
thread that, per RFC 7591 § 2, any such extension parameter name must
either be an absolute URI or be registered in the official IANA OAuth
Dynamic Client Registration Metadata registry.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative section to the SMART App Launch specification ("App
registration with the authorization server" / "Dynamic client
registration") that defines a new client-metadata field for use in RFC
7591 dynamic-registration requests and the corresponding `client_get`
responses. Concretely:

- **Parameter name:** register a SMART-namespaced absolute-URI extension
  metadata name, e.g.
  `http://fhir.org/argonaut/smart/ehr_launch_uri` (or, equivalently,
  request a short name such as `smart_ehr_launch_uri` from the IANA
  *OAuth Dynamic Client Registration Metadata* registry created by RFC
  7591). The spec must explicitly call out whichever of the two paths is
  chosen, because RFC 7591 § 2 forbids unregistered short names.
- **Type:** JSON array of strings, each a fully-qualified `https:` URI.
- **Cardinality:** 0..\* (mirrors the existing manual-registration
  language in § 1.0.3).
- **Semantics:** identical to the existing manually-registered
  fixed-launch-URL list — the authorization server MUST treat each value
  as an exact-match allowed value of the `launch_uri` parameter on a
  subsequent EHR-launch authorization request.
- **Validation rules:** server MUST reject any value that is not an
  absolute `https:` URI; SHOULD reject open-redirect-style wildcards.
- Add a non-normative example registration request and response, plus a
  conformance bullet that authorization servers supporting dynamic
  registration for EHR-launch apps MUST advertise support via a
  capability flag (e.g., add `ehr-launch-dynamic-registration` to the
  `capabilities` array returned by `.well-known/smart-configuration`).

#### Justification

This is the literal request in the ticket and is consistent with how
SMART has handled prior gaps (defining a small, RFC-conformant extension
parameter rather than overloading existing OAuth fields). The Zulip
thread surfaces a real interoperability gap: SMART App Launch already
*requires* registered launch URLs but does not say how to convey them
dynamically, so today every implementer fills the gap with a private
parameter. An IANA-registered (or absolute-URI-named) extension
preserves RFC 7591 conformance, and tying it to a
`.well-known/smart-configuration` capability flag lets clients discover
support without trial-and-error registration.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of defining a new SMART-specific dynamic-registration parameter,
**reuse `redirect_uris` plus a discovery flag**:

1. Document, in SMART App Launch, that for EHR-launch apps the
   `redirect_uris` array submitted under RFC 7591 carries the
   double duty of being both the OAuth redirect targets and the allowed
   `launch_uri` values, *unless* the app also supplies a separate
   per-deployment launch endpoint via the existing `launch_uri`
   request parameter. Authorization servers compare incoming
   `launch_uri` values against the registered `redirect_uris` set using
   exact match.
2. Add a single boolean capability — e.g.
   `ehr-launch-uri-from-redirect-uris` — to
   `.well-known/smart-configuration` so clients know whether a given
   server uses this convention.
3. If, and only if, the work group decides the two URI lists must be
   distinct, fall back to a registered absolute-URI extension as in
   Disposition A.

#### Justification

`redirect_uris` is already mandatory under RFC 7591 and is already the
URL set the authorization server treats as trustworthy for the client.
For the overwhelming majority of SMART EHR-launch apps the launch URL
*is* the redirect URL (the same SPA endpoint that handles the OAuth
callback). Reusing `redirect_uris` avoids minting a new IANA-registered
parameter, avoids the implementation cost of teaching every
authorization server to recognise yet another piece of client metadata,
and sidesteps the RFC 7591 § 2 requirement entirely. The trade-off
versus Disposition A is loss of separation between "URLs the AS may
redirect the user-agent to after authorization" and "URLs the EHR may
launch", which some deployments may genuinely need; the capability flag
gives those deployments a clean opt-out by negotiating Disposition A's
parameter instead.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Modification → No Change** on the
grounds that (a) RFC 7591 already permits implementers to add absolute-URI
extension parameters out-of-band without a SMART specification change,
(b) there are no implementation reports demonstrating an actual
interoperability failure since the ticket was filed in 2021, and (c) the
current set of EHRs offering dynamic client registration for SMART apps
is small enough that bilateral profiling between app and EHR is more
realistic than a one-size-fits-all extension. Recommend instead that the
work group document, as guidance only, that implementers SHOULD use
`redirect_uris` to also carry launch URLs and SHOULD coordinate with
their EHR partners via the existing SMART implementers' channel.

#### Justification

The ticket has sat in *Triaged* for four years without a single Jira
comment and with only the original two-message Zulip thread of community
discussion — there is no follow-on demand visible in the FhirAugury
cross-reference graph (no related Jira tickets, no related GitHub
items). RFC 7591 already gives implementers a conformant escape hatch
(absolute-URI extension names), so the cost of *not* standardising is
bounded. Standardising prematurely risks locking in a parameter shape
(name, cardinality, exact-match vs. prefix-match semantics) that does
not match how dynamic registration for EHR launch ends up being deployed
in the field.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the only option that closes the actual gap the ticket
identifies — SMART App Launch *requires* registered launch URLs but is
silent on how they reach the authorization server when registration is
dynamic — and it does so in a way that cleanly satisfies RFC 7591 § 2 by
either using an absolute SMART URI or by going through the IANA OAuth
Dynamic Client Registration Metadata registry. Disposition B is
attractive for its minimalism but conflates two security-relevant URL
sets (`redirect_uris` is bound to the OAuth user-agent redirect; launch
URIs are about who may *initiate* a session) and silently changes the
trust semantics of `redirect_uris` for any AS that adopts the
convention, which is a bigger spec change than it appears. Disposition C
ignores the fact that SMART App Launch already imposes the registered-
launch-URL requirement on manual registration; leaving the dynamic path
underspecified pushes every implementer to invent a private parameter,
which is the worst interoperability outcome. The work group should
adopt Disposition A, but should explicitly decide between the
absolute-URI form and an IANA-registered short name — that single
decision is the only substantive open question once the principle is
accepted, and it is exactly the question Alexander flagged in the Zulip
thread when he filed the ticket.
