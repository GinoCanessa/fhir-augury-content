## Summary

The ticket asks the FHIR Infrastructure work group to clarify how the
`_elements` search parameter applies to **secondary** resources (i.e.,
those returned via `_include` / `_revInclude`) when those resources are
*not* explicitly named in the `_elements` filter. The R5 text says that
when an entry in `_elements` is prefixed with a resource type (e.g.,
`_elements=Patient.gender`), servers SHOULD apply the restriction to all
matching resources in the bundle (not just the primary search matches).
What the spec does **not** say is what should happen to a secondary
resource type when *no* `Resource.element` entries for that type appear
in `_elements` at all — should the entire resource type be filtered out
of the bundle, should it be returned with only its mandatory elements,
or should it be returned in full? The reporter explicitly enumerates
those three options and asks the WG to pick one.

There are no cross-referenced Jira tickets. The only cross-reference is
the originating Zulip thread `#implementers > _include and _elements`,
in which the question was raised and Lloyd McKenzie suggested filing a
change request — which is this ticket.

## Details

**Description:**

> The text for `_elements` indicates that if the resource type is
> included in the element name, then those are the only elements
> returned for secondary resources in the search. However, it isn't
> clear what to do with those secondary resources if the resource name
> isn't included in the element name, or if resources match that aren't
> mentioned.
>
> Clarify whether entire secondary resources should be filtered out,
> whether only the mandatory content of secondary resource should be
> returned, or whether the full secondary resource should be returned if
> they aren't mentioned in an `_elements` filter.

**Resolution Description (already recorded on the ticket — Persuasive):**

> Suggest adding clarification that resources not specified in the
> `_elements` parameter are returned in full.
>
> Other default behaviors do not allow the full resource to be returned
> without specifying every single element in the resource.

**Comments:** No comments are recorded on the Jira ticket itself; the
substantive discussion happened in the linked Zulip thread (see below).

## Related Zulip Discussions

### [#implementers > _include and _elements](https://chat.fhir.org/#narrow/stream/implementers/topic/_include%20and%20_elements/near/341516916)

Elliot Silver opened the thread asking how `_include`/`_revInclude` and
`_elements` should interact and listed several possible behaviors. Gino
Canessa pointed out that R5 already says resource-prefixed `_elements`
values (e.g., `Patient.gender`) SHOULD be applied to all bundle entries
regardless of `search.mode`, and Lloyd McKenzie agreed a change request
was needed to clarify the gaps. The discussion then split into two
threads of opinion. **On the question of unmentioned secondary
resources** (the actual subject of this ticket), Gino proposed that "if
you do not list any elements for a resource that is in the response it
would not get filtered" — i.e., a request that filters Patient and
Encounter elements but says nothing about Observation should still
return full Observations. That is the position the recorded Persuasive
resolution adopts. **On the orthogonal question of whether `_include`
should fire when its referencing element has been stripped**, Lloyd
argued that floating includes (entries whose linking reference is not
present, whether stripped by `_elements`, by consent, or otherwise) are
confusing and possibly unsafe and should be omitted; Gino and Paul
Church pushed back, noting that mixing `_elements` filtering with
`_include` resolution forces servers to either re-query or post-filter,
and that authorization-driven filtering is a separate concern from
result shaping. Daniel Venton and John Moehrke leaned toward Lloyd's
view that an `_include` whose linking element is absent is effectively
useless. No formal consensus was reached on that secondary question; it
was explicitly noted that it would need its own ticket. The thread also
spawned tangential items (warning OperationOutcome for missing
references, whether `_elements` belongs in CapabilityStatement) that
are out of scope here.

Participants: Elliot Silver, Lloyd McKenzie, Gino Canessa, Paul Church,
John Moehrke, Daniel Venton, Ming Dunajick.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already recorded on the ticket: amend the
`_elements` description on the *Search — Modifying Search Results* page
of the FHIR Core specification to state explicitly that **secondary
resources whose type is not referenced anywhere in the `_elements`
parameter are returned in full** (i.e., they are neither filtered out
of the bundle nor reduced to their mandatory-only subset). Concretely,
add a sentence such as:

> If a resource type appears in the bundle (as a primary match or via
> `_include`/`_revInclude`) and the `_elements` parameter contains no
> entries that apply to that resource type — that is, neither bare
> element names that resolve against it nor any `Type.element` entries
> prefixed with that type — the server SHALL return instances of that
> resource type in full. `_elements` only restricts the resource types
> it explicitly mentions.

This complements the existing R5 text about resource-prefixed
`_elements` applying to all bundle entries and avoids ambiguity for
the unmentioned-type case.

#### Justification

This matches the reporter's request, the recorded "Persuasive"
resolution, and the consensus voiced in the Zulip thread by Gino
Canessa. It is also consistent with how every other FHIR result-shaping
mechanism behaves: full content is the default, and a client must
opt in to reduction (`_summary=text|data|count`, `_elements=...`,
profile-driven slicing, etc.). Treating an unmentioned type as
"reduce to mandatory" or "drop entirely" would be the only place in
the spec where omission of a directive silently *removes* data, which
would surprise implementers and break round-tripping for the common
case where a client only wants to trim the primary resource type.

---

### Disposition B: Alternative Approach

#### Proposal

Clarify the `_elements` text along the same lines as Disposition A but
make the behavior **negotiable** rather than prescriptive: state that
the default is "return unmentioned resource types in full", but allow
servers to advertise (via a `CapabilityStatement.rest.searchParam`
extension or a documented `Parameters`-style option on the search) that
they instead apply `_elements` globally — e.g., reducing every
unmentioned secondary type to its mandatory elements, or dropping it
entirely. Document the three possible modes the reporter listed
(`full`, `mandatory-only`, `omit`) and require the server to pick one
and disclose it.

#### Justification

The Zulip discussion exposed real implementer divergence: some servers
(per Paul Church) apply `_elements` very late in the pipeline and
naturally produce full secondary resources, while Lloyd's privacy/
"floating include" concerns hint that some deployments will want to
prune aggressively. A negotiable default acknowledges both camps and
makes interop predictable when a client talks to an unfamiliar server.
The cost is added complexity in `CapabilityStatement` and one more
knob for clients to inspect — non-trivial, especially given the
acknowledgement in the thread that there is no current mechanism to
advertise *result-shaping* parameters in `CapabilityStatement` at all.

---

### Disposition C: Decline

#### Proposal

Do not change the `_elements` text. Take the position that the existing
R5 wording — combined with the general FHIR principle that omitted
directives do not reduce content — already implies the correct
behavior, and that any further normative clarification risks
constraining valid existing server implementations. Close as
"Not Persuasive" with a note pointing readers to the R5 prefixed-
elements text and to general result-shaping defaults.

#### Justification

The R5 text the WG added in 2022 already says prefixed `_elements`
values SHOULD be applied to all matched and included resources; the
contrapositive (types not mentioned are not filtered) can be argued
to follow. Declining avoids prescribing behavior on which the Zulip
thread itself shows no full consensus (Lloyd's "no floating includes"
position would push toward stricter pruning rules that this
clarification does not address). It also keeps the door open for a
fuller treatment in the planned R6 work on advertising result-shaping
parameters in `CapabilityStatement`, where this question can be
answered alongside `_summary`, `_total`, etc.

---

### Recommendation

**Recommended disposition:** A

The ticket asks a narrow, concrete question, the recorded resolution
is already "Persuasive", and the answer it gives — "unmentioned
secondary types come back in full" — is the only one consistent with
how every other FHIR result-shaping parameter defaults. Disposition B's
negotiable-mode idea is interesting but answers a different (larger)
question about advertising result-shaping options, which the Zulip
thread itself flagged as a separate, deferred work item; bundling it
here would expand scope and likely delay the simple clarification the
reporter is asking for. Disposition C ("the spec already implies it")
is defensible but leaves implementers to keep guessing — exactly the
situation the ticket was filed to fix. A targeted one-sentence
clarification on the *Modifying Search Results* page is low-risk,
backward-compatible with the dominant implementation behavior, and
discharges the ticket cleanly. Lloyd's "floating include" concern and
the broader interaction between `_elements` and `_include` resolution
should be tracked as a separate change request, as already suggested
in the thread.
