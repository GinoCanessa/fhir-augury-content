## Summary

The reporter (Bas van den Heuvel) asks, in the context of FHIR Shorthand
(FSH), what fields of a `Reference` data type instance are populated when a
reference is assigned via the `Reference()` keyword (e.g.,
`Reference(EveAnyPerson)`). Specifically, is only `Reference.reference`
set, or are `Reference.type` and/or `Reference.display` also populated?
If not, how should those be populated, and should FSH offer a built-in
shortcut (e.g., extra positional arguments to `Reference()` or a RuleSet)
rather than requiring follow-up rules per element?

The cross-reference index returned **no related Jira tickets, Zulip
threads, or GitHub items** for this ticket, so the analysis is grounded
entirely in the ticket itself and the resolution recorded by the FSH
maintainers.

## Details

**Description:**

> When assigning a Reference using a instance indication (e.g.
> `Reference( EveAnyPerson )` ). What fields in the resulting Reference
> instance are populated?
>
> Only `Reference.reference`?
>
> Also `Reference.type` and/or `Reference.display`?
>
> If not, how do I populate these fields?
>
> Is there a way to indicate to do so in the reference without requiring
> an additional rule (e.g. additional parameters in the `Reference( )`
> indication or using a RuleSet? If not we should consider adding one.

**Resolution recorded on the ticket:**

> The FSH grammar already allows the display to be set by specifying a
> quoted string after the reference (e.g., `Reference(Amy) "Amy P"`).
> This is consistent with the syntax for setting display on a code. We
> will document this in the specification.
>
> The type and identifier can be set via normal rules. E.g.:
> ```
> * subject = Reference(Amy) "Amy P"
> * subject.type = "Patient"
> * subject.identifier.system = "http://mysystem.org"
> * subject.identifier.value = "abc123"
> ```
> Rules for type and identifier can go before or after the reference
> assignment. Assigning the reference via the `Reference` keyword does
> not overwrite type or identifier.
>
> We considered adding special syntax for setting type or identifier
> alongside the `Reference()` keyword, but felt that the frequency of
> setting these did not justify the additional complexity. Once a single
> syntactical statement contains more than 2 or 3 parts, it becomes
> difficult for users to remember the correct order of the parts.

**Comments:**

- **cmoesel (2023-10-12):** Confirmed that the argument inside
  `Reference()` is exported as `Reference.reference`. Pointed at the
  [FSH ANTLR grammar](https://github.com/FHIR/sushi/blob/master/antlr/src/main/antlr/FSH.g4)
  showing that a trailing quoted string after `Reference()` is exported
  as `Reference.display` (e.g., `subject = Reference(Amy) "Amy P"`
  yields `{ "reference": "Patient/Amy", "display": "Amy P" }`); this
  works in SUSHI but is not documented in the spec. Recommended either
  documenting it or removing it from the grammar — preference to keep
  it because it is consistent with the `displayName` shortcut for
  codings. Stated that `Reference.type` and `Reference.identifier` can
  be set via subsequent dotted rules (`subject.type = "Patient"`,
  `subject.identifier.system = "..."`, etc.) and argued against new
  positional syntax to keep FSH small and easy to remember. Pinged
  Mark Kramer for input.
- **cmoesel (2023-10-12):** Apology for typos — Jira does not allow
  editing comments.
- **esilver (2023-10-12):** Supported keeping (and documenting) the
  trailing-string display shortcut. Suggested that `Reference.type`
  could also be supported via shortcut syntax, possibly between the
  reference and the display using a `[CodeSystem]#code`-style form for
  consistency, but flagged uncertainty about precedence when fields are
  already set individually. Agreed `identifier` should remain set via
  explicit rules because it is used less often.
- **bvdh (2023-10-30):** Accepted the proposed approach. Noted it is
  confusing that the same syntax behaves differently in profiles vs.
  instances (per §3.6.2 it appears to be used as the `description` /
  `definition` field in profiles). Felt these elements are
  underspecified given the grammar is informative, and asked that the
  documented behavior be added explicitly to the spec (e.g., §3.3.13 or
  §3.5.5).
- **cmoesel (2023-11-01):** Agreed the trailing-string-as-display is
  consistent with codings, acknowledged Add Element rules are awkward.
  Committed to updating the spec; identified the 3.0 ballot
  (Sep 2023) sections
  [3.3.6 References](https://hl7.org/fhir/uv/shorthand/2023Sep/reference.html#references)
  and
  [3.6.3.6 Assignments Involving References](https://hl7.org/fhir/uv/shorthand/2023Sep/reference.html#assignments-involving-references)
  as the right home for the documentation.

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Shorthand specification (Shorthand IG, 3.0 ballot and
later) to **explicitly document** the behavior of the `Reference()`
keyword in instance assignments:

1. State that `Reference(<id>)` populates `Reference.reference` only.
2. Document the existing-but-undocumented FSH grammar feature that a
   trailing quoted string sets `Reference.display`
   (`* subject = Reference(Amy) "Amy P"` →
   `{ "reference": "Patient/Amy", "display": "Amy P" }`), framing it as
   the parallel of the `displayName` shortcut for codings.
3. State that `Reference.type` and `Reference.identifier` are **not**
   set by the `Reference()` keyword and must be assigned via normal
   dotted rules (`* subject.type = "Patient"`, etc.), and that those
   rules may appear before or after the `Reference()` assignment
   without being overwritten.
4. Place the description in §3.3.6 *References* and §3.6.3.6
   *Assignments Involving References* of the Shorthand IG (paths the
   committer already named in the ticket), and add the shortcut to the
   formal grammar/syntax summary section so the grammar is no longer
   informative-only on this point.

No code change to SUSHI is required — only documentation changes in the
Shorthand IG repository.

#### Justification

This is exactly what the ticket asks for and exactly what the FSH
maintainers (cmoesel) committed to in the final comment. The behavior
already works in SUSHI and is grounded in the published ANTLR grammar
(`FSH.g4`); the gap is purely documentary. Documenting in §3.3.6 and
§3.6.3.6 keeps the description next to the existing `Reference()`
description and the existing assignment semantics, which is where a
reader would look. The original reporter also explicitly endorsed this
outcome ("I'm ok with the proposed approach"). Treating the
trailing-display as the same idiom as coding `displayName` keeps FSH
internally consistent and reduces cognitive load for authors.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to the documentation fix in Disposition A, **extend the
`Reference()` syntax** to allow setting `Reference.type` inline, e.g.
admit a form like `Reference(Amy|Patient) "Amy P"` (or
`Reference(Amy)#Patient "Amy P"`, mirroring `[CodeSystem]#code` as
suggested by esilver), and provide a canonical RuleSet pattern in the
spec for the `identifier` case rather than baking it into the
positional syntax. Update both `FSH.g4` and the SUSHI exporter to
populate `Reference.type` from this token, and document the precedence
rule (explicit dotted rules win over the shortcut, regardless of order).

#### Justification

The reporter's underlying concern is ergonomic: today, populating
`type` and `display` requires three or four lines per reference, and
external consumers frequently rely on `Reference.type` to disambiguate
polymorphic targets and for FHIRPath type filtering. esilver supported
exactly this — keep the trailing display, add a type token. The
trade-offs vs. Disposition A are real: it grows the FSH grammar
surface, introduces another positional slot to remember, and creates a
new precedence question (what happens when both the shortcut and a
dotted `subject.type =` rule are present). cmoesel's pushback — "once
a single syntactical statement contains more than 2 or 3 parts, it
becomes difficult for users to remember the correct order" — is the
core counter-argument, and any acceptance of B should commit to a
fixed, documented order and SUSHI lint warnings on conflicts.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Modification — documentation
only**, declining the request to add new positional syntax to
`Reference()` for `type`/`identifier`, on the grounds that:

- The dotted-rule pattern (`* subject.type = "Patient"`,
  `* subject.identifier.system = "..."`, etc.) is the existing,
  general-purpose mechanism for setting subfields of any data type,
  including `Reference`, and it already works.
- Adding new positional arguments would expand the FSH grammar for a
  use case (`Reference.type` / `Reference.identifier`) that is not
  common enough to justify the additional cognitive load and
  precedence rules.

(Document the existing trailing-display behavior as part of accepting
the ticket's documentation question; only the *new syntax* portion of
the request is declined.)

#### Justification

This is essentially the position that won out on the ticket and is
defensible on the same grounds cmoesel articulated: FSH deliberately
keeps its surface small, dotted-rule assignment is the universal
mechanism, and adding more positional slots would erode the simplicity
that is one of FSH's selling points. The risk of declining the broader
request is small because users are not blocked — they can already set
`type` and `identifier` today with one extra line each — and because
the documentation update (the part actually being accepted) closes the
substantive complaint that the behavior is undiscoverable.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as "Considered - Question
answered," the original reporter accepted the proposed approach, and
the FSH maintainer (cmoesel) explicitly committed to documenting the
behavior in §3.3.6 and §3.6.3.6 of the Shorthand IG. The work
remaining is purely documentary, has clear scope, and resolves the
substantive complaint (that the trailing-display shortcut is
undocumented and that the `type`/`identifier` populating story is
unclear). Disposition B is intriguing and has a real ergonomic
benefit, but it conflicts with the FSH design principle of keeping
syntax small and would require coordinated changes in `FSH.g4`, SUSHI
codegen, and the spec — disproportionate effort for a use case the
maintainers judge to be infrequent. Disposition C overlaps with A on
the substantive outcome and is essentially A re-labeled; A is the
clearer way to record what the WG has already agreed to do.
