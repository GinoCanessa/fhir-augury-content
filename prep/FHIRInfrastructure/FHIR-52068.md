## Summary

This SDC change request proposes adding one sentence of normative guidance
to the paragraph in the SDC IG that introduces the `linkIdPrefix`
mechanism for modular Questionnaires. Today the spec describes how a
parent Questionnaire MAY declare a `linkIdPrefix` variable so that a
sub-questionnaire's `linkId`s (and any expressions that reference them)
can be deterministically prefixed when the sub-questionnaire is
assembled into the parent. The ticket asks the spec to also state, as a
SHALL, what an assembler must do when the parent does **not** declare
that variable but the sub-questionnaire's expressions depend on it: the
assembler SHALL itself synthesize a `linkIdPrefix` variable with a
unique value for that sub-questionnaire. The reporter is Lloyd McKenzie
(SDC). The ticket was originally voted "Considered for Future Use"
(Brian Postlethwaite / Brenin Rhodes, 4-0-0, 2025-08-28) on the grounds
that the behaviour should be tested against an implementation before
being normatively pinned, and was re-opened post the SDC STU4
publication (lynn_laakso, 2026-03-26) so the FHIR Infrastructure /
Questionnaire community can revisit it. There are no cross-referenced
Jira tickets, GitHub items, or other Augury-tracked relations, but
historical Zulip discussion in `#questionnaire > Problem with
prohibiting variable overlap` (2021) shows that the WG explicitly
contemplated the prefix mechanism when carving out the variable-shadowing
rules, and that implementer behaviour around variable scoping in
modular forms is not uniform.

## Details

**Description:**

At the end of the paragraph starting "In order to avoid duplicate
linkIds, a parent Questionnaire MAY declare a special variable with the
name linkIdPrefix.", add:

> "If the parent Questionnaire does not define a linkIdPrefix variable,
> and the sub-questionnaire includes expresssions that depend on the
> linkIdPrefix variable, the assembler SHALL create a linkIdPrefix
> variable with a unique value for the sub-questionnaire."

## Related Zulip Discussions

### [questionnaire > Problem with prohibiting variable overlap](https://chat.fhir.org/#narrow/stream/questionnaire/topic/Problem%20with%20prohibiting%20variable%20overlap/near/260595995)

This 2021 thread established the rule on the SDC "Using Expressions"
page that variables sharing a name within the same scope are an error,
but Lloyd McKenzie explicitly carved out the prefix-based modular
Questionnaire pattern (the same mechanism this ticket is amending) as
a case where a child must be able to override a parent's variable —
otherwise `linkIdPrefix` cannot work. Brian Postlethwaite pushed back
on parent/child shadowing in his implementation, while Paul Lynch
(LHC-Forms) said his implementation already lets child nodes override
parent variables and saw no need for the restriction. The thread did
not close out a single rule; it ended with agreement that same-item
duplicate variable declarations are forbidden but parent/child
override semantics remained implementation-divergent. This is directly
relevant: the SHALL proposed by FHIR-52068 assumes the assembler can
freely synthesize a `linkIdPrefix` variable on the sub-questionnaire
without colliding with the parent's scope, which depends on the
override semantics this thread debated.

Participants: Lloyd McKenzie, Brian Postlethwaite, Paul Lynch.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the exact text from the ticket as a SHALL appended to the existing
`linkIdPrefix` paragraph in the SDC IG (in the Modular Forms / assembler
guidance, the paragraph beginning "In order to avoid duplicate linkIds, a
parent Questionnaire MAY declare a special variable with the name
linkIdPrefix."). The added sentence:

> "If the parent Questionnaire does not define a linkIdPrefix variable,
> and the sub-questionnaire includes expressions that depend on the
> linkIdPrefix variable, the assembler SHALL create a linkIdPrefix
> variable with a unique value for the sub-questionnaire."

would be added verbatim (with a typo fix on `expresssions`) to the SDC
narrative page that documents the assembler's behaviour for prefix-based
modularized questionnaires. No structure-definition or extension change
is required — this is purely narrative guidance pinning assembler
behaviour as normative.

#### Justification

The change closes a real conformance gap: today the spec only describes
the *parent's* option to declare `linkIdPrefix`, leaving undefined what
an assembler must do when a sub-questionnaire references the variable
but the parent has not declared it. Without this SHALL, two conformant
assemblers could produce different (and silently incompatible) assembled
Questionnaires for the same input, defeating the determinism that the
prefix mechanism exists to provide. The wording is consistent with the
WG's own framing of the prefix mechanism in the 2021 `#questionnaire`
thread, where Lloyd noted explicitly that the prefix approach requires
descendant scopes to be able to introduce/override the variable.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent but rework the wording so the requirement is on the
**sub-questionnaire context** rather than the assembler, and broaden it
into a small "default behaviour" subsection on the SDC modular-forms /
assembler page. Specifically:

1. Append a subsection (e.g., "Default `linkIdPrefix` for assembled
   sub-questionnaires") that states the assembler SHALL ensure that, at
   assembly time, every sub-questionnaire is evaluated with a
   `linkIdPrefix` variable in scope: either the value inherited from the
   parent's declared `linkIdPrefix`, or a generated unique value if the
   parent has not declared one.
2. Specify the uniqueness contract (unique within the assembled output;
   stable for the duration of one assembly operation; opaque to
   consumers) and recommend a generation strategy (e.g., a short
   deterministic hash of the sub-questionnaire canonical URL plus
   inclusion site, or a UUID-derived prefix), with a non-normative
   example.
3. Keep the parent's `MAY` declaration unchanged.

#### Justification

The literal one-sentence SHALL leaves two questions open that the
"Considered for Future Use" vote was likely flagging: what does "unique"
mean (unique within the assembled instance? globally?), and how should
implementers generate it deterministically enough to be testable? A
slightly broader subsection answers those without changing the intent,
makes the conformance statement testable (which is exactly what the
2025-08-28 vote asked for), and aligns the prose with the variable
scoping rules that the 2021 Zulip thread carved out (descendant scopes
may introduce a `linkIdPrefix` even where the parent has not). It also
keeps the pressure off implementations that already declare
`linkIdPrefix` at the parent (no behaviour change for them).

---

### Disposition C: Decline

#### Proposal

Resolve as **Persuasive with Mod / Not Persuasive (No Change)** on the
basis that the underlying behaviour is already obtainable today:
sub-questionnaire authors who depend on `linkIdPrefix` can require, in
the sub-questionnaire's usage notes, that the consuming parent declare a
`linkIdPrefix` variable; assemblers that do not see one can refuse to
assemble or surface a warning, and implementers can document their own
defaulting behaviour. Defer any normative SHALL until at least two
independent assembler implementations have demonstrated interoperable
auto-generation behaviour against a shared test fixture.

#### Justification

This mirrors the original 2025-08-28 WG vote ("Want to test this before
putting it in the spec.") and the implementer-divergence visible in the
2021 `#questionnaire` thread, where Brian Postlethwaite, Paul Lynch and
Lloyd McKenzie did not converge on a single variable-scoping rule for
modular forms. Pinning a SHALL on assemblers without an interop test is
exactly the situation the WG was trying to avoid; declining (or at most
deferring) keeps the door open for the same change once an SDC test
fixture proves the behaviour is implementable consistently.

---

### Recommendation

**Recommended disposition:** B

The intent of the request is correct — leaving assembler behaviour
undefined when a sub-questionnaire depends on `linkIdPrefix` is a
real interoperability hazard, and the re-open comment makes clear the
WG no longer wants to defer it. But the bare one-sentence SHALL is
exactly the wording the WG previously balked at because "unique" and
"create" are not specified tightly enough to test, and the 2021
`#questionnaire` discussion shows that variable-scoping behaviour in
modular forms is implementer-divergent today. Disposition B keeps the
ticket's normative ask, but supplies the missing pieces (uniqueness
contract, recommended generation strategy, explicit alignment with the
existing parent-`MAY` rule) that turn the SHALL into something
implementers can write a conformance test against. Disposition A is
acceptable as a fallback if the WG wants to ship the minimum-viable
text first and tighten it in a subsequent ticket; Disposition C
should be reserved for the case where no implementer is willing to
prototype the auto-generation behaviour before SDC's next ballot.
