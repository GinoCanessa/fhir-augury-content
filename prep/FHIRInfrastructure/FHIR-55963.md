## Summary

Lloyd McKenzie raised this Change Request to make it possible to declare
that a particular value-set binding (or, more generally, a canonical
reference) is **not** subject to the IG-publisher's package-dependency
based version "pinning" of canonical references. Today, when FHIR Core
or an IG is published, every canonical reference (including the
`ValueSet` referenced by a binding) effectively resolves to the version
shipped in the package dependency tree — even when the binding was
written in a version-independent manner. There are well-known cases
(notably required bindings to THO value sets that the spec wants to
"float" so net-new codes added after publication remain valid) where
this is the wrong behaviour.

Grahame Grieve has already drafted candidate guidance at
[ig-guidance/pinning.html#specifying-not-to-use-package-dependency-based-resolution](https://build.fhir.org/ig/FHIR/ig-guidance/branches/master/pinning.html#specifying-not-to-use-package-dependency-based-resolution),
but (a) the FHIR Core text on canonical matching at
[references.html#matching](https://build.fhir.org/references.html#matching)
does not yet describe or even permit such an extension, and (b) the
tooling (Kindling, the IG publisher, the validator) does not yet honour
or render it. The ticket therefore asks for a new extension on canonical
references / bindings that flags them as exempt from package-based
pinning, plus the spec text that makes that extension legal and the
tooling work to honour and render it.

The ticket has been resolved as **Persuasive** with a concrete proposal
to define a new extension `version-resolution-method` (cardinality 0..1,
type `code`, context `canonical`) with the codes `package` (default,
package-based resolution), `latest` (use the highest version per the
resource's version algorithm), and `manifest` (resolution governed by an
explicitly supplied manifest). Grahame's follow-on comment notes that
the documentation describing how canonical resolution works must also
be updated to make the extension possible.

A closely related Terminology Change Request,
[FHIR-55991](https://jira.hl7.org/browse/FHIR-55991), was logged by Bryn
Rhodes against the canonical-version-matching language; FHIR-I should
coordinate with Terminology so the spec text in `references.html` and
the binding-pinning rules end up internally consistent.

**Resolution Description (Persuasive):**

> 1. We will define a new extension called: `version-resolution-method`,
>    0..1 `code`, with a context of `canonical`.
>
> Definition: "If a canonical reference does not declare a specific
> version, indicates the algorithm to use to determine which version
> should be used."
>
> Comment: "See discussion on canonical version regulation [here]()"
>
> Allowed codes:
>
> - **package / package-based** — "The canonical is determined based on
>   the version of the resource as defined in the package hierarchy in
>   which the referencing artifact is defined, where the 'nearest'
>   definition is the version used. See further details [here](). This
>   is the default when the referenced artifact is defined in a
>   package, though it will only come into play if the referenced
>   artifact has any versions declared in the package hierarchy of the
>   referencing artifact."
> - **latest / latest** — "Regardless of packaging of the referencing
>   artifact, resolution is to the version that is determined to be the
>   highest value according to the version algorithm for the resource.
>   (Note that this is **not** necessarily the one with the most recent
>   publication date.)"
> - **manifest / manifest-based** — "The determination of the version
>   is explicitly declared by a manifest made available in the scope of
>   the resolution. If a manifest is not available, or the referenced
>   resource does not have a manifest entry, then resolution is not
>   possible."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution as already drafted on the ticket. Specifically:

1. **Define a new extension** `http://hl7.org/fhir/StructureDefinition/version-resolution-method`,
   cardinality `0..1`, type `code`, **context = `canonical` (the
   primitive)**, with binding to a new required-bound value set whose
   codes are:
   - `package` ("package-based") — package-hierarchy nearest-version
     resolution; default when the referenced artifact lives in a
     package and any version is reachable in the dependency closure.
   - `latest` — use the highest version per the resource's version
     algorithm, irrespective of packaging.
   - `manifest` ("manifest-based") — version is dictated by an
     externally-supplied manifest; if no manifest entry exists,
     resolution fails.
2. **Update `references.html#matching` in FHIR Core** so the canonical
   resolution algorithm explicitly enumerates these three resolution
   methods, names this extension as the way to opt out of the default
   package-based behaviour, and states what implementations MUST do
   when the extension is absent vs. present (i.e., the default == the
   current package-based pinning behaviour).
3. **Reconcile with `ig-guidance/pinning.html`** so Grahame's draft
   "specifying-not-to-use-package-dependency-based-resolution" section
   uses the new extension by name and the two pages describe one
   coherent algorithm.
4. **Tooling work (separate but tracked)** — file/keep companion items
   for Kindling, the IG publisher, and the validator to honour the
   extension during canonical resolution, and to render it in the
   bindings table where bindings carry it. (Per Grahame: "the other of
   us should get around to making this work in the tooling.")
5. **Coordinate with Terminology on
   [FHIR-55991](https://jira.hl7.org/browse/FHIR-55991)** so the
   matching text revised here resolves the contradictory conformance
   requirements Bryn flagged in the same pass.

#### Justification

This is precisely what the originating Zulip thread agreed to and what
the ticket's own Persuasive resolution already proposes; both Lloyd
(reporter) and Grahame (author of the draft pinning text) endorsed this
shape. Defining the extension on `canonical` (rather than on
`ElementDefinition.binding.valueSet` only) generalises cleanly to
`StructureDefinition`, `ConceptMap`, etc. that need the same opt-out,
and the three-code value set captures all real-world cases that have
been raised: package-bound (today's default), float-to-latest (the THO
required-binding case Lloyd raised), and manifest-driven (the
deterministic-but-not-package case the IG-publisher already supports as
a short-cut). Putting the normative algorithm in `references.html` and
having `ig-guidance/pinning.html` refer to it eliminates the very
"two-pages-disagree" problem Bryn called out as FHIR-55991. The
extension being absent must mean "package-based" so existing artifacts
remain conformant — i.e., this is an additive change.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of inventing a new value-set-bound `code` extension, **reuse
the existing IG-publisher pinning short-cut vocabulary** by defining a
single boolean extension on `canonical` —
`http://hl7.org/fhir/StructureDefinition/canonical-resolution-floats`
(`0..1 boolean`, default `false`) — that simply means "do not apply
package-based pinning to this canonical; resolve to the latest version
per the target resource's version algorithm." Update
`references.html#matching` and `ig-guidance/pinning.html` to describe
exactly two behaviours (package-pinned default vs. floating) and reuse
the *manifest* mechanism that already exists at the IG-publisher level
rather than promoting it into a third coded value on every canonical
reference. Tooling work (Kindling, IG publisher, validator) is the
same as Disposition A.

#### Justification

Most of the demand in the Zulip thread — Lloyd's THO required-binding
case, Bryn's "latest indicator" question, AU Base's STU note — is
binary: "this binding should not be frozen by the package." A boolean
captures that with the lowest possible cognitive and migration cost,
keeps existing bindings conformant by default, and avoids forcing
authors to learn a new code system whose third value (`manifest`) is
already handled at the IG-publisher / package level rather than on the
canonical itself. Trade-off vs. A: it gives up the ability to declare
on the canonical that resolution must be manifest-driven (rather than
floating), which Grahame's draft text explicitly contemplates; if FHIR-I
believes the manifest case will realistically be selected per-canonical
(not just per-IG), Disposition A is the better fit. Note also that
late-binding has the well-known instance-breaking risks Gino raised and
Bryn confirmed in the Zulip thread; under either A or B the spec text
should warn authors that floating bindings can retroactively
invalidate previously-conformant instances, and should strongly
discourage floating on `extensible` bindings.

---

### Disposition C: Decline

#### Proposal

Decline to add a new extension, on the grounds that the existing
mechanisms — the IG publisher's `pinning.html#ig-publisher-short-cuts`
"latest" indicator, plus authoring-time guidance to keep `required`
bindings against THO out of IG package dependencies that would freeze
them — are sufficient to address the cases raised. Update
`references.html#matching` only to clarify the existing rules and add
non-normative guidance pointing at the IG-publisher short-cut.

#### Justification

A defensible "no" rests on the position Grahame articulated multiple
times in the originating thread: "if the content comes from a package,
then it's version-bound by the package references," and the right
answer for content that needs to float is to keep it out of the
package-pinned closure rather than to layer per-canonical opt-outs on
top. Adding a new extension that every tool (Kindling, IG publisher,
validator, downstream profile tooling) must learn to honour and render
is a non-trivial change to the canonical-resolution model and a
permanent surface-area increase; the late-binding semantics also
introduce real instance-breaking risk (Gino/Bryn in Zulip). If the
short-cut already covers the IG-publisher case and Terminology can
guide implementers on how to structure THO dependencies, the cost of
the new extension may not be justified. **However**, this position is
hard to sustain given Lloyd's concrete requirement that `required`
bindings to THO should accept net-new codes after publication — there
is no purely-packaging way to get that behaviour without the validator
treating the binding as floating, which is exactly what the extension
is for.

---

### Recommendation

**Recommended disposition:** A

The ticket's Persuasive resolution and the originating Zulip thread
already converged on this answer, and both of the principals (Lloyd as
reporter, Grahame as the author of the draft pinning section) endorsed
it. Defining `version-resolution-method` on `canonical` with the three
codes `package` / `latest` / `manifest` cleanly covers every case that
was raised, leaves existing artifacts conformant by default (absent =
`package`), gives FHIR-I a single normative home for the algorithm in
`references.html#matching` (which will dissolve the contradiction Bryn
flagged as [FHIR-55991](https://jira.hl7.org/browse/FHIR-55991)), and
gives the tooling team (Kindling, IG publisher, validator) one
extension to honour and render. The Justification of Disposition A
should be expanded with the warning Gino and Bryn raised about
late-binding's instance-breaking risk on `extensible` bindings — the
spec text should advise authors to use `latest` only on `required`
bindings against terminologies that are explicitly additive (the THO
case), and to be cautious otherwise. Disposition B is a reasonable
fallback if FHIR-I decides the `manifest` case does not need to be
expressible per-canonical, but it would force a second extension later
if that decision is revisited; Disposition C does not actually solve
Lloyd's concrete required-binding-to-THO problem and should be picked
only if the workgroup wants to push the entire concern back to package
authoring discipline.
