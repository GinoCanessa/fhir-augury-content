## Summary

Emma Jones reports that on the R5 CarePlan search-parameter page
(`https://hl7.org/fhir/R5/careplan-search.html`), several search parameter
rows render an "extra link" preceding the parameter description. The
affected parameters are `date`, `encounter`, `identifier`, and `patient` —
all of which are common (inherited) search parameters rather than
CarePlan-specific ones. The reporter has not (yet) explained why those
links are problematic; the only follow-up is a clarifying comment from
Michelle Miller noting that *all* common search parameters render in this
format and asking whether the issue is really a tooling/publisher concern
(label `FHIR-Tools-Core`) rather than something for Patient Care WG to
fix in CarePlan itself. No cross-referenced Jira tickets, GitHub items,
or Zulip threads were surfaced for this ticket by the augury cross-ref
index (the comment links to a Zulip thread in `#committers > Tooling
issue?`, but it could not be retrieved through the augury Zulip search
or thread query, so no third-party context is available beyond the
ticket itself).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a publisher/tooling defect and remove the leading link(s)
that appear before the description text for the common (inherited)
search parameters (`date`, `encounter`, `identifier`, `patient`) in the
generated `*-search.html` pages. Concretely, this would mean adjusting
the search-parameter row rendering in the FHIR core publisher so that
either (a) the inheritance "source" link is rendered *after* the
description, or (b) it is rendered as a small bracketed annotation
(e.g., `[from DomainResource]`) rather than as a hyperlink that
visually precedes and competes with the description text. The change
applies uniformly to all resources, not only CarePlan, since "all
common search parameters share this format" (per Michelle Miller's
comment).

#### Justification

The reporter labelled the ticket `FHIR-Tools-Core` and filed it under
FHIR Infrastructure, both of which point at the publisher rather than
the CarePlan resource definition. Because the same rendering is used
for every resource that inherits common search parameters, fixing it
once in the publisher template propagates the improvement everywhere
without touching any resource definitions or search-parameter
artifacts. This keeps the change scoped to documentation rendering, so
there is no normative or wire-format impact.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing link, but reduce its visual prominence and clarify
its semantics rather than removing it. For each inherited common search
parameter, render an explicit "Inherited from: [Resource/Pattern]"
prefix (or a small icon with hover text) so it is obvious that the
preceding link points at the *source* of the parameter definition, and
that the following description is the inherited description. No
parameter is removed; only the rendering of the row changes.

#### Justification

The link is genuinely useful — it lets a reader navigate from a
resource's search-parameter table to the canonical definition of the
inherited parameter. Stripping it (Disposition A) would be a regression
for implementers who use it to confirm that a parameter on CarePlan is
in fact the same parameter as on, e.g., DomainResource or the clinical
pattern. Re-skinning it as an explicit "Inherited from" marker
addresses the reporter's concern (the link no longer competes with the
description) while preserving the navigation affordance. Trade-off vs.
Disposition A: slightly more publisher-template work, but no loss of
information.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive / Working as Designed". The link that
precedes the description identifies the resource/pattern from which
the common search parameter is inherited and is the *intended*
rendering for inherited parameters across every resource in the
specification, not a CarePlan-specific defect. The reporter has not
(after ~9 months and a request for clarification) articulated a
concrete user-facing problem caused by the link.

#### Justification

Per Michelle Miller's comment, "All common search parameters share
this format" — i.e., the rendering is consistent across the spec, so
treating it as a defect on CarePlan (or even as a general defect) is
not supported by the ticket as written. Without a documented
implementer harm (e.g., readers being misled about what the parameter
applies to), removing or restyling the link risks losing useful
inheritance information for the sake of an aesthetic preference.
Declining is defensible until the reporter can supply a concrete
reason the existing rendering is misleading or wrong.

---

### Recommendation

**Recommended disposition:** B

The ticket points at a real readability concern — a link rendered
*before* the description does compete with the description for the
reader's attention — but the link itself is informative and removing
it (Disposition A) would degrade the published spec for the many
implementers who rely on the inheritance pointer when reconciling
search-parameter behaviour across resources. Disposition C is
tempting given the reporter has not justified the request, but the
underlying observation (common search-parameter rows look visually
"noisy" because of the leading link) is reproducible on the cited
page and worth fixing once for all resources rather than re-litigating
ticket-by-ticket. Disposition B fixes the rendering globally in the
publisher, preserves the inheritance information that the link
provides, and stays inside the FHIR-Tools-Core / FHIR Infrastructure
scope implied by the labels. The work group should confirm with the
reporter that an explicit "Inherited from …" treatment satisfies the
original concern before the change is implemented.
