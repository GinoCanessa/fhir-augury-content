## Summary

The ticket asks that `OperationDefinition.resource` be widened from the
current bound code (`VersionIndependentResourceTypesAll`) to a URI so
that an `OperationDefinition` can declare that the operation applies to
a logical model (or any non-core resource type) — not just to a FHIR
core / additional resource type. Grahame also notes the spec needs new
language stating that, at the *invocation* level, the meaning of an
operation declared against a logical model is not defined (i.e., the
typing is allowed, but how the operation is wired into the RESTful API
when the "type" is a logical model is left unspecified).

The need surfaced concretely on Zulip (`#IG creation > Define operation
for ViewDefinition`, message [543532264](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/Define%20operation%20for%20ViewDefinition/near/543532264)),
where Nikolai Ryzhikov tried to define a type-level `$viewdefinition-export`
operation against `ViewDefinition` (a logical model in sql-on-fhir-v2)
and was blocked because `OperationDefinition.resource` only accepts
codes from the all-resource-types value set. Grahame's interim
workaround was to set `resource = #CanonicalResource` and use the
`operationdefinition-allowed-type` extension to narrow the actual
allowed type — which works for `parameter.type` but is awkward (and
arguably incorrect) for `resource`. That awkwardness is what
[FHIR-53233](https://jira.hl7.org/browse/FHIR-53233) (sibling ticket
filed by Grahame against the FHIR Extensions Pack) addresses by
proposing the `allowed-type` extension be extended to apply to
`OperationDefinition.resource`. FHIR-51651 is the deeper, core-spec
fix: change the type of the element itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the type of `OperationDefinition.resource` in FHIR Core from a
`code` bound (required) to `VersionIndependentResourceTypesAll` to a
`uri`. The URI is expected to be the canonical URL of either a core
resource type (e.g., `http://hl7.org/fhir/StructureDefinition/Patient`),
an additional resource type, or a logical model
`StructureDefinition` (e.g., `http://sql-on-fhir.org/StructureDefinition/ViewDefinition`).

Add normative wording to the OperationDefinition narrative stating that:

1. When `resource` references a logical model (i.e., a
   `StructureDefinition` whose `kind = logical`), the *typing* of the
   operation's input/output is well-defined, but *invocation semantics*
   over the RESTful API for that operation are explicitly **not
   defined** by the base specification — implementers/IGs are
   responsible for specifying how (or whether) such an operation is
   invoked.
2. Existing core-resource use of `resource` continues to be valid — the
   URI form is a strict superset of the current code-typed values
   (each old code maps to its canonical StructureDefinition URL).

Also update `CapabilityStatement.rest.resource.operation` guidance and
any tooling that consumes `OperationDefinition.resource` to accept
URIs.

#### Justification

This is the cleanest fix to the underlying typing gap exposed in the
Zulip thread and it removes the need for the `allowed-type` workaround
entirely for the `resource` element. It is consistent with FHIR's
direction on additional-resource and logical-model support
(cf. FHIR-41356's discussion of `$validate` over logical models, and
the ongoing additional-resources work). Treating the value as a
canonical URL also aligns with how every other "what type does this
apply to" slot in FHIR (e.g., `targetProfile`, `Parameters` typing
extensions) is expressed.

The "invocation semantics not defined" sentence is the right
conservative scope: it lets IGs (sql-on-fhir, CDS Hooks-style payloads,
etc.) declare typed operations against logical models without forcing
core to invent a REST projection for logical models.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `OperationDefinition.resource` as a `code` bound to
`VersionIndependentResourceTypesAll`, and instead resolve the gap via
the FHIR Extensions Pack: adopt FHIR-53233 to widen the
`operationdefinition-allowed-type` extension so that it explicitly
applies to `OperationDefinition.resource` (as a *primitive extension*
on each `resource` code), with the documented pattern being:

- Set `resource = #CanonicalResource` (or another suitably broad code
  already in the value set) as a placeholder.
- Attach `operationdefinition-allowed-type` to that `resource` element
  with the canonical URL of the actual logical model.

Add a non-normative note in core's OperationDefinition page pointing
implementers at this extension when the target is a logical model, and
clarify that invocation semantics over a logical model are not defined.

#### Justification

This avoids a breaking type change to a normative element of a core
resource — `OperationDefinition` is FMM 5 / Trial Use and widely
implemented. Changing `.resource` from `code` to `uri` invalidates all
current parsers/validators and persisted instances (StructureMap,
toolchains, registries, IG publishers all special-case `code` here).
Trial-use → URI is still a breaking schema change.

The extension-based path is what Grahame already recommended on Zulip
as the working pattern; FHIR-53233 only formalizes it. It delivers the
ability-to-define-operations-on-logical-models outcome without
disturbing the core element type or its required binding, and it keeps
the core resource backward compatible.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-51651 as out of scope for the core OperationDefinition
typing model, with the rationale that operations are defined by FHIR
to operate on FHIR resources (including additional resources via
`VersionIndependentResourceTypesAll`), and operations on logical
models are an IG-level concern. Direct the requester to FHIR-53233
(extension-level fix) and to per-IG conventions for invocation
semantics. Optionally add a one-paragraph clarifying note to the
OperationDefinition page that operations targeting logical models
are out of scope for core invocation semantics.

#### Justification

The element has been a bound `code` since DSTU2; no implementer is
currently blocked from defining the operation logically (the
`allowed-type` workaround exists). The deeper problem the ticket
mentions — "what it means at the invocation level is not defined" —
is a hint that core may not be the right venue: if invocation
semantics are not specified, then the typing change buys nothing
testable at the core level and risks confusion ("you can declare it
but you can't call it"). Logical-model-typed operations are arguably
better governed by the IGs that introduce the logical models in the
first place.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach)

The Zulip thread shows that the immediate, real-world need
(sql-on-fhir-v2's `$viewdefinition-export`) is *already* satisfiable
today using `resource = #CanonicalResource` plus the
`operationdefinition-allowed-type` extension; the only thing missing is
the extension's official applicability to `.resource`, which is
exactly what FHIR-53233 fixes. That makes Disposition B the lowest-risk,
highest-leverage path: it unblocks the use case in this cycle without
touching the type of a normative element on a heavily-implemented
resource.

Disposition A is the technically cleaner long-term shape — and the
ticket's wording ("should be a URI") clearly favors it — but it is a
breaking change to the core schema that conflicts with
`OperationDefinition`'s maturity status and would invalidate existing
instances and tooling. It deserves a serious discussion, but ideally as
an R6+ planning item, paired with the additional-resources and
logical-model invocation work, rather than as a standalone change here.

Disposition C undersells the legitimate need the Zulip thread surfaced
and is harder to justify now that the requester has explicitly been
pointed at the `allowed-type` extension as the workaround.

If FHIR-I prefers A, the Justification on A still applies, and the
recommendation is to coordinate with the Extensions Pack team so that
FHIR-53233 is **withdrawn-as-superseded** rather than adopted in
parallel.
