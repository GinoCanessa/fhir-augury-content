## Summary

Grahame Grieve reports that the documentation for the `additional-binding`
extension is incomplete with respect to its use on `CodeSystem.property`.
Today the definition describes the extension's behaviour when it appears on
ElementDefinition.binding, but does not explain that — when attached to a
`CodeSystem.property` (specifically a property whose values are themselves
coded) — it applies to those property-coded values much the same way that a
normal terminology binding applies to an element. The ticket asks that the
extension's definition (now part of the Tooling Extensions IG, having been
moved out of FHIR Core during May 2024 WGM) be expanded to describe this
behaviour explicitly so authors know how to interpret and use it on
`CodeSystem.property`.

No related Jira tickets were surfaced by cross-references. The single Zulip
cross-reference ("Reference to Person") mentions `FHIR-40729` in passing but
the surrounding conversation is about an unrelated Person/Reference issue
(very likely a typo for `FHIR-40279`); it does not contribute substantive

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the prose definition of the `additional-binding` extension in the
**Tooling Extensions IG** (the spec named on the ticket; ownership of the
extension was moved to that IG at the May 2024 WGM) so that, in addition
to its existing behaviour on `ElementDefinition.binding`, it explicitly
documents use on `CodeSystem.property`. Concretely, add a new sub-section
to the extension's definition page covering:

- That `additional-binding` MAY appear on a `CodeSystem.property` whose
  `type` is `code`, `Coding`, or `CodeableConcept` (i.e., property values
  that are themselves drawn from a value set).
- That, when present in that location, each `additional-binding` declares
  an additional value set that the *property's coded values* SHALL/SHOULD/
  MAY (per `purpose`) be drawn from — i.e., it behaves analogously to an
  additional binding on an ElementDefinition, but the bound "element" is
  the property's value rather than an instance element.
- How `purpose` (`required`, `extensible`, `current`, `preferred`,
  `ui`, `starter`, `component`, `maximum`, `minimum`) is interpreted in
  this context, including any purposes that are not meaningful for a
  property and should be discouraged.
- A worked example showing a `CodeSystem` whose property is defined with
  a `valueSet` and one or more `additional-binding` extensions on the
  property definition.

The change is documentation-only on the extension StructureDefinition's
description / comments and the Tooling IG narrative page that introduces
the extension; no element, type, or cardinality change is needed.

#### Justification

This is exactly the change Grahame asked for and matches the pattern
already used for `additional-binding` on element bindings, so it
introduces no new concept — it merely closes a definitional gap. Because
`CodeSystem.property` already permits coded values and already optionally
carries a primary `valueSet`, modelling additional bindings on a property
is a natural extension of existing semantics. As a definition-only
clarification in the Tooling Extensions IG, it is non-breaking for
existing implementers and brings the documentation into line with how the
extension is already (or is being) used in practice.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding prose, also tighten the extension's
StructureDefinition so the dual usage is machine-discoverable:

- Update the extension's `context` to enumerate both
  `ElementDefinition.binding` (current usage) and `CodeSystem.property`
  (new usage), so validators recognise both as legal anchor points.
- Add a FHIRPath `constraint` on the extension that, when its context is
  `CodeSystem.property`, enforces that the parent property's `type` is one
  of `code | Coding | CodeableConcept` (it is meaningless on, e.g., a
  string or integer property).
- Update the extension's description, plus the Tooling IG narrative, with
  the same prose as Disposition A so humans and tooling agree.

#### Justification

A pure prose change (Disposition A) leaves the divergence between
"documented use" and "machine-checkable use" in place — implementers and
validators have to take the IG's word for the new usage. Encoding the
context and the type-of-property constraint in the StructureDefinition
prevents accidental misuse on non-coded properties, gives validators a
hook, and is still a small, contained change confined to the extension
definition in the Tooling Extensions IG. The trade-off is a slightly
larger surface area to review and the small risk that any in-the-wild use
on a non-coded property would suddenly become invalid; this is a feature,
not a regression, since such usage was always nonsensical.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that `CodeSystem.property` already supports a
single `valueSet` reference, and the existing `additional-binding`
documentation on element bindings — together with the analogy stated on
the Tooling IG's introduction to the extension — is sufficient guidance.
Close as "Persuasive with Mod / Not Persuasive" with a note pointing
implementers at the existing element-binding documentation.

#### Justification

The literal request is "explain it better", not "change behaviour". One
could argue that the extension's behaviour on a `CodeSystem.property` is
already implied by parallel construction with `ElementDefinition.binding`
and that the IG narrative is the appropriate venue for tutorial-level
explanation rather than the normative definition. Declining keeps the
extension definition stable and avoids any risk of subtle wording changes
introducing new ambiguity.

---

### Recommendation

**Recommended disposition:** B

The reporter (Grahame Grieve) is the original author of the extension and
is explicitly flagging that the definition is unclear about an
already-intended usage pattern, so doing nothing (Disposition C) leaves a
documented gap that he himself surfaced and is hard to defend at ballot.
Disposition A fixes the gap but stops at prose; given that this ticket is
already labelled `R6` and the Tooling Extensions IG is itself moving with
R6, this is a low-cost moment to also encode the second context and the
"property must be coded" constraint in the StructureDefinition. That
gives validators something to check, prevents misuse on non-coded
properties, and makes the extension's dual nature self-describing for
downstream tooling — at the cost of a slightly bigger but still
narrowly-scoped change in the Tooling Extensions IG, fully owned by FHIR
Infrastructure. Recommend B as the starting point, with A as an
acceptable fallback if the workgroup wants to limit scope to
documentation-only.
