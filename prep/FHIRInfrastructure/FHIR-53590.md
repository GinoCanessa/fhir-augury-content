## Summary

The submitter (Lloyd McKenzie) reports that the Clinical Document Architecture
(CDA) does not use SemVer; CDA versions are expressed as a four-node numeric
string (e.g., `major.minor.patch.build`). FHIR currently has no
version-comparison algorithm code that handles this 4-node form, so tooling
and resources that need to order or compare CDA versions cannot pick a
standard `versionAlgorithm` value. The ticket proposes adding a new code
— `semver4` — to the FHIR version-algorithm code system, defined as: "A
semantic versioning scheme where in addition to the major.minor.patch
version numbers, there's also a .build node. This 4th node takes the place
of the tags and build metadata supported by semver." No cross-referenced
Jira tickets, Zulip threads, or GitHub items were returned by FHIR Augury
for this key, so the surrounding discussion context could not be mined.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new code `semver4` to the FHIR `version-algorithm` code system (and
its companion value set used by `versionAlgorithm[x]` on canonical
resources such as `CodeSystem`, `ValueSet`, `StructureDefinition`,
`ConceptMap`, `NamingSystem`, `Library`, etc.). Use the submitter's
proposed definition verbatim:

> "semver4": "A semantic versioning scheme where in addition to the
> major.minor.patch version numbers, there's also a .build node. This
> 4th node takes the place of the tags and build metadata supported by
> semver."

Specify the comparison rules in the description: each of the four nodes
is a non-negative integer compared numerically left-to-right; absent
trailing nodes compare as if `0`; no pre-release / build-metadata
suffixes are recognised (a deliberate divergence from SemVer 2.0.0).
Add the code to the existing version-algorithm CodeSystem (no new
namespace), and surface CDA as the canonical example use-case in the
code's comments.

#### Justification

The `version-algorithm` element exists precisely so that authors can
declare which ordering rule applies to their `version` strings, and the
existing codes (`semver`, `integer`, `alpha`, `date`, `natural`) do not
correctly order four-node CDA versions: `semver` rejects the 4th node,
`integer` rejects dots, and `natural` ordering is lexicographic and not
guaranteed to match the intent of the CDA publishers. Adding a single
well-defined code is a low-risk, additive change that lets implementers
consuming CDA-derived canonicals (e.g., terminology and document
templates expressed as FHIR canonicals) compare versions deterministically.
The change is consistent with the way `semver` was originally added to
the same code system.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of minting a CDA-specific `semver4` code, generalise the
solution by adding a code such as `numeric-dotted` (or
`numeric-segments`) defined as: "A version expressed as one or more
non-negative integers separated by `.`, compared numerically
segment-by-segment from left to right, with missing trailing segments
treated as 0." Document CDA's four-node scheme as the motivating
example. Optionally retain the `semver4` label as an alias / synonym in
the description but do not pin the code to exactly four nodes.

#### Justification

The underlying comparison rule is not unique to four-node strings —
many legacy specifications (including some IHE profiles and OID-rooted
identifier registries) publish versions with two, three, or four+
numeric segments. A `numeric-dotted` code covers all of these without
needing a `semver5`, `semver6`, etc. in the future, and avoids
overloading the "semver" brand for a scheme that intentionally drops
SemVer's pre-release and build-metadata semantics (which could mislead
implementers into expecting SemVer-compatible behaviour). The trade-off
versus Disposition A is loss of self-documentation: the code no longer
calls out CDA explicitly, and the asker's exact requested code/label is
not adopted.

---

### Disposition C: Decline

#### Proposal

Decline to add a new code. Recommend that publishers of CDA-derived
canonical resources either (a) set `versionAlgorithm` to the existing
`natural` code (which is defined as "Sort versions in the order
implied by treating embedded numeric segments numerically and other
segments alphabetically") and rely on conformant `natural` comparators
to handle four numeric nodes correctly, or (b) supply a `Coding` for
`versionAlgorithmCoding` that points to a CDA-specific code system
maintained by Structured Documents.

#### Justification

`versionAlgorithm[x]` already supports a `Coding` choice precisely so
that communities with bespoke version schemes can name their own
algorithm without growing the core code system. Declining keeps the
core code system small and avoids precedent for adding one code per
external versioning convention. The downside is that interoperable
tooling has no standard, pre-baked comparator for CDA versions, which
re-pushes the burden onto every consuming implementation — exactly the
problem the ticket was filed to solve.

---

### Recommendation

**Recommended disposition:** B

The underlying need — a standard, deterministic, segment-numeric
comparison algorithm — is real and well-scoped, so declining
(Disposition C) leaves implementers without a usable interoperable
answer and forces ad-hoc `Coding` values that defeat the purpose of the
code system. Disposition A solves the immediate CDA case but bakes a
fixed segment count and a misleading "semver" label into the standard;
the rule has nothing to do with SemVer's pre-release / build-metadata
machinery, and CDA is unlikely to be the last specification with a
non-three-node numeric version scheme. A generalised `numeric-dotted`
code (Disposition B) closes the gap the submitter identified, costs the
same to add as `semver4`, and avoids future churn when the next
N-segment scheme appears. The workgroup should confirm the precise
code id and definition wording, but the shape of the change is clearly
"one new additive code in the existing version-algorithm CodeSystem."
