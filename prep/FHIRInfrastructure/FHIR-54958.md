## Summary

Brian Postlethwaite is asking whether the FHIR Mapping Language (FML)
specification's *General Syntax Notes* — the section that defines the
media types used to identify FML content — should also describe how the
language is versioned and how that versioning is conveyed alongside the
media type. The underlying concern is that as FML evolves (new syntax,
new functions, normative vs. trial-use changes), implementations need a
deterministic way to know which version of the language a given map was
authored against, and authors need guidance on how to declare that
intent. The ticket itself is short and exploratory; it raises the
question rather than proposing a specific syntax.

There are no cross-references on the ticket and no related Zulip
threads were surfaced, but the request lands in well-trodden territory.
A directly analogous decision was made for CQL/ELM in
[FHIR-32666](https://jira.hl7.org/browse/FHIR-32666), where a `version`
parameter on the media type (`text/cql; version=1.4`) was added — that
provides a strong precedent for how FHIR has solved the same problem
before. The ticket also sits alongside
[FHIR-54473](https://jira.hl7.org/browse/FHIR-54473) (FML not yet ready
for normative), which captures broader concerns about FML's stability,
round-trip behaviour with StructureMap, and whether FML should remain
in FHIR Core at all — those concerns directly affect what a "version"
means here (FHIR-coupled vs. independent FML versions).

### [FHIR-32666: Add version parameter for media types](https://jira.hl7.org/browse/FHIR-32666)

Bryn Rhodes' Persuasive/Published resolution adds documented use of a
`version` parameter on the CQL/ELM media types — e.g.,
`text/cql; version=1.4` — recording only major.minor of the CQL spec
version, marked trial-use, with `SHOULD` guidance for implementations
to respect it and degrade gracefully on unknown versions. The ticket
also confirms that BCP-13 still treats the parameterised string as a
valid code, so existing MIMEType value-set bindings continue to hold.
This is the closest existing precedent for what FHIR-54958 is asking
about and provides a near-ready template for FML.

Users involved: Bryn Rhodes (reporter / commenter).

### [FHIR-19715: Syntax improvements to the mapping language](https://jira.hl7.org/browse/FHIR-19715)

Grahame Grieve's R3→R4 syntax cleanup of FML (replacing `make` with
`=>`, statement-terminating `;`, Pascal-style group parameters, `{}`
group bodies, stereotype types). It is included here because it
illustrates exactly the kind of grammar-level change that breaks older
parsers and motivates a media-type-level version indicator: a map
written for the post-R3 syntax cannot be parsed by an R2-era engine,
and today there is no in-band signal of which dialect was intended.

Users involved: Grahame Grieve (reporter / committer).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the *General Syntax Notes* section of the FHIR Mapping Language
page to add an explicit "Versioning" subsection that:

1. Defines a `version` parameter on the FML media types
   (`text/fhir-mapping`, and any aliases such as `text/fml` if
   applicable), following the CQL precedent set by
   [FHIR-32666](https://jira.hl7.org/browse/FHIR-32666). The value is
   `major.minor` of the FML grammar version
   (e.g., `text/fhir-mapping; version=2.0`).
2. Specifies that the parameter is **trial-use** in R6.
3. States that implementations **SHOULD** honour the parameter and
   degrade gracefully when given a version they do not understand
   (parse-then-fail-clearly rather than silently mis-interpret).
4. Notes that the parameterised value remains a valid BCP-13 code and
   therefore does not break existing MIMEType value-set bindings.
5. Documents how FML grammar versions will be assigned going forward
   (each FHIR release that changes the grammar bumps the FML version;
   non-grammar changes do not).
6. Cross-links to the StructureMap resource so authors know how to
   carry the same version signal when an FML map is round-tripped to a
   StructureMap (e.g., via an existing or new extension on
   `StructureMap`).

#### Justification

This is a low-risk, high-clarity change that mirrors a resolution the
project already accepted for an analogous language (CQL). The CQL
resolution explicitly addressed the BCP-13 binding question, the
trial-use status, and the implementation-guidance phrasing — all of
which carry over verbatim. Adding it to the *General Syntax Notes* (the
section the ticket calls out) keeps the information where readers
already look for the media-type definition. Marking it trial-use
respects the open question about FML's normative readiness raised in
[FHIR-54473](https://jira.hl7.org/browse/FHIR-54473) without blocking
implementers who need a version signal today.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) a media-type `version` parameter,
introduce an in-content versioning convention for FML maps:

1. Define an optional first-line directive in FML, e.g.
   `// fml-version: 2.0` or a structured pragma such as
   `%fml-version 2.0;`, that authors include at the top of a map.
2. Require the FML→StructureMap round-trip to preserve this directive
   (capturing it on `StructureMap` via a standard extension, e.g.
   `http://hl7.org/fhir/StructureDefinition/structuremap-fml-version`).
3. Update the *General Syntax Notes* to describe the directive and to
   note that when both an in-content directive and a media-type
   parameter are present, the in-content directive wins (since it
   travels with the artifact).
4. Defer any media-type parameter to a follow-up ticket scoped under
   FHIR-54473's broader FML disposition.

#### Justification

The media-type parameter only helps when content moves through a
transport that preserves the full media-type string with parameters
(HTTP `Content-Type`, some packaging formats). FML maps are also
distributed inside `StructureMap.text` blocks, IG packages, NPM
tarballs, and source repositories where the media type is not attached
or is normalised away. An in-content directive survives those
transports and matches how FHIRPath/CQL communities have, in practice,
used header comments or pragmas. It also avoids prejudging the outcome
of [FHIR-54473](https://jira.hl7.org/browse/FHIR-54473): if FML moves
to an external IG with its own versioning cadence, the directive
mechanism is still valid; a media-type registration could become stale
or contested.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-54958 as **Not Persuasive — Out of Scope for R6** and defer
any FML versioning treatment until the disposition of
[FHIR-54473](https://jira.hl7.org/browse/FHIR-54473) is settled. Add at
most a one-line editorial note in *General Syntax Notes* acknowledging
that FML versioning is an open question being tracked under
FHIR-54473, with no normative or trial-use mechanism specified.

#### Justification

FHIR-54473 captures FHIR-I's WGM consensus that FML is "promising but
not quite finished" and questions whether FML belongs in FHIR Core at
all. Defining a versioning mechanism — whether a media-type parameter
or an in-content directive — implicitly commits to a versioning model
(grammar version coupled to FHIR releases? independent FML versions?
how do they relate to StructureMap, which stays normative?). Making
that commitment now risks locking in something that has to be
re-litigated when FHIR-54473 resolves. A short-term defer is
defensible: there is no evidence in this ticket of an implementer
blocked today, no Zulip thread of community demand, and no concrete
proposal attached. Document the question; defer the answer.

---

### Recommendation

**Recommended disposition:** A

The CQL precedent in
[FHIR-32666](https://jira.hl7.org/browse/FHIR-32666) is direct, recent,
already shipped, and explicitly addressed the same questions — BCP-13
binding compatibility, trial-use status, implementation expectations.
Re-using that pattern for FML costs little, gives implementers the
deterministic version signal the ticket is asking about, and makes the
*General Syntax Notes* section answer the question Brian raised
verbatim. Marking the FML `version` parameter trial-use cleanly defers
to FHIR-54473's broader discussion: if FML eventually moves to an
external IG, the parameter and its documented semantics travel with
it; if FML remains in core normative, the trial-use flag can be
promoted on its own merits. Disposition B's in-content directive is a
worthwhile follow-up but should not block A — the two are complementary
rather than mutually exclusive, and the media-type parameter is the
piece the ticket actually points at. Disposition C is the safest
short-term answer but underserves the implementer audience that has
already had to deal with FML grammar changes (e.g., the R3→R4 cleanup
in [FHIR-19715](https://jira.hl7.org/browse/FHIR-19715)) without any
in-band version signal.
