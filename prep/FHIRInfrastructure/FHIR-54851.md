## Summary

The ticket asks the FHIR Infrastructure work group to revisit the
"dragon" warning currently published on the FHIR search page concerning
timezones and offsets in date-typed search parameters. The existing
warning tells implementers that timezone handling in search is unsettled,
that future versions of the specification may impose additional rules,
and invites feedback via the tracker and chat.fhir.org. The reporter
asks that the committee review the present rules around timezone
handling in search, decide whether any normative or guidance changes are
warranted, and revise (or remove) the warning text accordingly. There
are no cross-referenced Jira tickets, Zulip threads, or GitHub items
linked to this issue, so it is presented here as a standalone editorial
and substantive review of the current Search page guidance.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Conduct the requested review of the current Search-page rules around
timezones and offsets, and revise the existing dragon warning to reflect
the outcome. Concretely:

1. Audit the Search page rules covering the `date` search parameter
   family (and the related semantics for `Period`, `instant`,
   `dateTime`, and `date`-typed elements), focusing on:
   - implicit precision / range semantics for partial dates,
   - behavior when a search value carries a timezone offset and the
     stored value does not (and vice-versa),
   - comparator semantics (`eq`, `ne`, `lt`, `gt`, `le`, `ge`, `sa`,
     `eb`, `ap`) when timezone information is absent on either side,
   - the server's freedom to assume a local timezone for timezone-less
     stored data.
2. Where the current normative text is ambiguous, tighten it (no new
   normative SHALLs without WG agreement); where it is already
   sufficient, leave the rule and rewrite only the warning.
3. Replace the existing dragon callout with a shorter, factual note
   that (a) summarises the resolved rules, (b) identifies the
   remaining areas where IGs are expected to add constraints (e.g.,
   required precision, required offset), and (c) drops the
   "future versions may impose rules" language unless the WG genuinely
   intends to reserve that latitude.

#### Justification

The ticket is explicitly an editorial-plus-review request from within
the responsible WG (FHIR Infrastructure). The current dragon warning is
several cycles old and signals unsettled design without pointing
implementers at the rules that *do* exist; that is a poor reader
experience on a high-traffic page. Performing the review now lets the
WG either confirm the status quo (and write a calmer note) or surface
real gaps for separate change requests, which is a normal cleanup
activity for the Search page.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-opening the underlying timezone/offset rules, treat this
as a pure editorial change: keep the current normative behaviour
unchanged and rewrite the dragon warning into a concise informative
note that:

- states plainly that timezone-less search values and timezone-less
  stored values are matched per the existing rules already documented
  on the page (with an in-page anchor link to those rules),
- notes that IGs MAY tighten requirements (e.g., require an offset on
  search values, require a specific precision), with a forward link to
  the IG-authoring guidance,
- drops the "future versions of this specification may impose rules"
  sentence and the "feedback welcome" boilerplate (the tracker and chat
  links exist site-wide).

No rule changes; only the call-out text is revised. File a separate
change request if, during drafting, the editor discovers a genuine rule
gap.

#### Justification

The ticket text is "review and revise this text", which can be read
narrowly as an editorial ask. A pure-editorial pass is low-risk, low-
cost, and avoids re-litigating timezone semantics that the WG has
historically found contentious. Splitting any newly-discovered
substantive issue into its own ticket keeps the change history clean
and lets reviewers ballot the editorial fix on its own merits.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Modification — No Change" (or
equivalent disposition): acknowledge that the dragon warning is still
appropriate because timezone handling in search remains an area where
the community is actively learning and where IGs are expected to add
local rules. Leave the existing text in place.

#### Justification

The current warning is honest: timezone behaviour in search genuinely
is one of the harder corners of FHIR, IGs do in fact add rules in this
area, and the WG has not (in this ticket) been presented with concrete
new rules to adopt. Removing or softening the warning without a
substantive rule change risks giving implementers false confidence
that the area is fully settled. Until a concrete proposal exists, the
status-quo guidance is defensible.

---

### Recommendation

**Recommended disposition:** B

The ticket is best read as an editorial review request, and Disposition
B delivers exactly that with the lowest risk: it rewrites the dragon
callout into a calmer, more useful note, removes stale "future versions
may…" language, and points implementers at the rules that already exist
on the page. Disposition A is appealing but converts a small
documentation cleanup into a potentially open-ended re-examination of
timezone semantics, which is unlikely to converge inside a single
ballot cycle absent a concrete proposal. Disposition C is defensible
but unsatisfying — the warning as written invites implementers to
provide feedback yet does not summarise the rules they should be
testing against, so leaving it untouched perpetuates the very ambiguity
it complains about. Disposition B should be the starting point; if the
editor doing the rewrite uncovers a genuine rule gap, that becomes a
follow-up ticket rather than scope creep on this one.
