## Summary

Reporter Isaac Vetter (Epic) observes that, while reviewing US Core's
new specification of SMART requirements, there is a clear gap in the
SMART on FHIR App Launch implementation guide: there is no defined
**capability set for backend services / system-only clients**. Today,
the published capability sets cover patient/user-facing apps but do not
provide a packaged set of capabilities a server can advertise (or an
implementer can target) for backend (system-to-system) deployments.

He proposes a new "backend system" capability set that would require
either `SHALL client-confidential-asymmetric` (or, more loosely, `SHOULD
client-confidential-asymmetric` paired with `MAY
client-confidential-symmetric`). In follow-up comments he clarifies
that the new set should also include the `permission-system` capability
and a newly-defined `system-user` capability. He further suggests that
all existing capability sets should reference `permission-v1` or
`permission-v2`, and that the addition of a backend capability set
should land in **SMART App Launch v2.0.0** (i.e., versioned as a 2.0.0
patch / point release of that IG).

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced by the FHIR Augury indexes for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next patch / point release of the **SMART App Launch IG (v2.0.0
line)**, add a new named **capability set for backend services**
alongside the existing patient-/user-facing sets. The set would, at
minimum:

- Require `client-confidential-asymmetric` as **SHALL** (the canonical
  asymmetric, jwt-based client authentication SMART defines for
  backend services), and optionally allow
  `client-confidential-symmetric` as **MAY** for servers that also
  support shared-secret authentication.
- Require the `permission-system` capability (system-level scopes such
  as `system/*.read`, `system/*.rs`, etc.).
- Define and require a new `system-user` capability (a backend-services
  analogue to the existing user-mode capability) so that servers can
  unambiguously advertise that backend principals are recognized as a
  distinct actor type.
- Reference the appropriate scope-syntax permission capability — i.e.,
  align with `permission-v2` going forward (and tolerate `permission-v1`
  as a downgrade path), and update the *existing* capability sets so
  they likewise pin a `permission-v1`/`permission-v2` capability rather
  than leaving scope syntax implicit.

The change is purely additive (new named set + capability) and so is
appropriate for a 2.0.x patch of the SMART App Launch IG, as the
reporter requested.

#### Justification

The SMART backend services profile is a first-class authentication
flow in SMART App Launch but is currently not surfaced as a
**packaged**, name-able conformance bundle the way the patient- and
user-mode capability sets are. That is a real gap for downstream IGs
(US Core, Da Vinci, bulk-data implementers) that today have to enumerate
the individual capabilities by hand to express "this server supports
SMART backend services." Defining the bundle in the IG itself centralizes
that contract and removes the inconsistency between IGs.

Tying every capability set — including the new one — to an explicit
`permission-v1` / `permission-v2` capability also closes a long-standing
ambiguity about which scope syntax a server is advertising, which
matches direction the WG has already taken with v2 scopes.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than defining a brand-new "backend system" capability *set* and a
new `system-user` capability inside the SMART App Launch IG, instead:

1. **Document a recommended composition** in the IG narrative: a
   non-normative table that says "for a backend-services deployment,
   advertise the following capabilities: `client-confidential-asymmetric`
   (SHALL), `permission-system` (SHALL), `permission-v2` (SHOULD), …"
   without minting a new named set.
2. **Tighten the existing capability sets** to explicitly require one of
   `permission-v1` / `permission-v2` (the part of the request that is
   uncontroversial and purely clarifying).
3. Defer the `system-user` capability and the named backend set to the
   next minor (2.1.0 / 3.0.0) release of SMART App Launch, where
   introducing a new capability identifier can go through normal
   ballot rather than a patch.

#### Justification

Named capability sets become part of the IG's normative surface — once
servers and downstream IGs reference `backend-system` by name, removing
or restructuring it is a breaking change. Introducing a brand-new
capability identifier (`system-user`) in a **patch** release is in
tension with the IG's own versioning policy, which generally reserves
patch releases for clarifications and errata, not new conformance
machinery. The alternative still gives implementers and US Core the
guidance they need (a documented, recommended composition + explicit
permission-v* references) without minting normative artifacts via a
patch.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the existing SMART App Launch
**Backend Services** profile already enumerates the required behaviors
(asymmetric client authentication, JWT assertion, system scopes), and
that downstream IGs (notably US Core and Bulk Data) are the appropriate
place to compose a "backend deployment" conformance expectation. Close
the ticket as **Not Persuasive — already addressed by existing
mechanism**, with a note pointing implementers to the Backend Services
section of SMART App Launch and to the system scope syntax in the
permission-v2 capability.

#### Justification

SMART already defines the backend-services flow normatively; capability
sets in the IG are primarily a packaging convenience for *user-facing*
launch contexts, where there is real combinatorial complexity
(EHR-launch vs. standalone, public vs. confidential, patient vs. user
scopes). Backend services have far less variation — essentially "do
client-confidential-asymmetric and use system scopes" — which makes a
dedicated named set arguably redundant. Adding it (and a new
`system-user` capability) in a **patch** release also conflicts with
typical IG versioning expectations.

---

### Recommendation

**Recommended disposition:** A (with the scope of changes scaled to
match what is actually safe in a 2.0.x patch).

The underlying gap the reporter identifies is real: SMART App Launch
publishes named capability sets for user-facing apps but leaves
implementers (and downstream IGs like US Core) to assemble the
backend-services conformance story by hand. Codifying that story as a
named capability set in the IG is exactly the kind of consolidation
SMART's capability-set machinery exists to do, and doing so makes US
Core's SMART requirements expressible by reference instead of by
ad-hoc enumeration.

That said, FHIR Infrastructure should be deliberate about what lands in
a **2.0.0 patch** versus what waits for the next minor: the new named
set and the explicit `permission-v1` / `permission-v2` references on
the existing sets are clarifying and purely additive and are good
candidates for the patch. Introducing a brand-new `system-user`
capability identifier is closer to new normative machinery and should
either (a) be confirmed as in-scope for the patch by the WG explicitly,
or (b) be deferred to the next minor release, with the patch shipping
only the named set + the `client-confidential-asymmetric` /
`permission-system` requirements that already exist as capabilities.
That split captures most of the reporter's value immediately while
keeping the IG's versioning discipline intact.
