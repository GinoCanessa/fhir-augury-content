## Summary

The ticket asks the FHIR core specification to clarify the textual definition of `Signature.data` for JWS-based digital signatures. Today the spec says `Signature.data` is "a base64 encoded JWS-Signature as specified in RFC 7515", but RFC 7515 defines two distinct serialization forms — **JWS Compact Serialization** (§3.1 / §7.1) and **JWS JSON Serialization** (§3.2) — and both internally rely on base64url encoding. The reporter (Bas van den Heuvel) and a security reviewer (Essien Ge, JIRAUSER26199) argue that "base64 encoded JWS" is ambiguous and suggest the spec explicitly name the intended serialization (the reporter initially suggested restricting to compact serialization per RFC 7515 §7.1).

In the comment thread, Eric Haas (ehaas) clarifies that FHIR does **not** restrict implementers to compact serialization — JSON serialization is also permitted (e.g., when unprotected headers or multiple signatures are required) — and that, regardless of the JWS serialization chosen, the result is then placed into a `Signature.data` element whose datatype is `base64Binary`, so the JWS is always wrapped in another base64 layer at the FHIR container level. He proposes a clarified sentence: "The Signature.data is a base64 encoding of a JWS-Signature specified in RFC 7515… the field contains the JWS signature, which is generated and serialized according to the JWS specification (e.g. compact serialization, JSON serialization) and then encoded in base64." The reviewer accepts this direction but raises a follow-up: if both serializations are allowed, FHIR currently has no element to declare which one is in use (`Signature.sigFormat` is a MIME type and does not differentiate compact vs. JSON serialization within the JWS family), so verifiers have to sniff the payload.

There are no Jira cross-references and no related GitHub PRs/issues. The ticket originates from a coordinated batch of Signature-related comments filed by Bas van den Heuvel (siblings include FHIR-52810 through FHIR-52820 and FHIR-52745, plus the older FHIR-47196 / FHIR-47197 from Essien Ge), all discussed in the Zulip thread "Signature comments" in `fhir/infrastructure-wg`.

## Details

**Description:**

> RFC 7515 defines 2 kinds of serialization for JWS Compact serialization and JSON serialization. Both are using based64. Just saying base64 encoded JWS is not enough to specify the one which is used here. Suggest making it clear this is compact serialization (compact serialization according to RFC7515).

**Comments:**

- **ehaas (2025-09-24, 00:53 UTC):** Proposes clarified wording — *"The Signature.data is a base64 encoded JWS-Signature as specified in RFC 7515… the field contains the JWS signature, which is generated according to the JWS specification and then encoded in base64."* Frames the JWS signature as being created per RFC 7515 and then base64-encoded into the FHIR container.
- **JIRAUSER26199 (Essien Ge) (2025-09-24, 02:20 UTC):** Counter-proposes that the term "Compact serialization" (RFC 7515 §7.1) be used directly, because both compact and JSON JWS serializations involve base64url somewhere, making "base64 encoded" ambiguous. Provides side-by-side examples of the compact form (`BASE64URL(header).BASE64URL(payload).BASE64URL(signature)`) and a multi-signature JSON serialization example.
- **ehaas (2025-09-24, 13:04 UTC):** Clarifies that FHIR does **not** restrict JWS to compact serialization; JSON serialization is required for use cases like unprotected headers or multiple signatures (RFC 7515 §3.2). Notes that the FHIR `Signature.data` datatype is `base64Binary`, so even a JWS compact serialization (which is itself base64url) gets wrapped in another base64 layer when stored in the resource — i.e., implementers are "base64 encoding base64 encoded strings."
- **JIRAUSER26199 (Essien Ge) (2025-09-25, 02:14 UTC):** Accepts ehaas's proposal but suggests the wording explicitly mention "serialization" and state that **both** compact and JSON serializations are supported. Raises an open question: do we need an element to distinguish which serialization is in use, since `Signature.sigFormat` (a MIME type) cannot differentiate compact from JSON JWS, or do we leave it for verifiers to sniff?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the clarified wording proposed by Eric Haas in the ticket comments, expanded per Essien Ge's feedback to explicitly call out both serializations. Specifically, in the FHIR Core specification's Signature datatype description (datatypes.html#Signature), replace the current sentence describing `Signature.data` with text along the lines of:

> *Signature.data* contains a JWS-Signature as specified in [RFC 7515: JSON Web Signature (JWS)](https://tools.ietf.org/html/rfc7515). The JWS may use either the **JWS Compact Serialization** (RFC 7515 §3.1 / §7.1) or the **JWS JSON Serialization** (RFC 7515 §3.2); the choice is determined by the implementation's needs (e.g., JSON Serialization is required when multiple signatures or unprotected headers are present). The serialized JWS is then encoded as `base64Binary` for transport in this element. Note that JWS Compact Serialization is itself base64url-encoded, so this results in a base64 wrapping of an already base64url-encoded value.

In addition, clarify the binding/usage of `Signature.sigFormat` to indicate that the MIME type identifies the JOSE family (e.g., `application/jose`, `application/jose+json`) and, where ambiguity exists, recommend `application/jose` for compact serialization and `application/jose+json` for JSON serialization (per RFC 7515 §9 media type registrations), so that verifiers do not need to sniff the payload.

#### Justification

This directly resolves the ambiguity the reporter identified, lands the wording the assigned commenter (Eric Haas) already drafted, and incorporates Essien Ge's "both are supported" clarification — meaning the change is largely editorial and reflects existing implementation behaviour. Reusing the RFC 7515 §9 MIME types in `Signature.sigFormat` is a natural extension of the already-present `sigFormat` element and avoids inventing a new FHIR element to carry serialization metadata, which preserves backward compatibility. The change is non-breaking: instances valid today (compact serialization wrapped in `base64Binary`) remain valid, and the additional `sigFormat` recommendation is guidance, not a new constraint.

---

### Disposition B: Alternative Approach

#### Proposal

Take a narrower, purely editorial fix and defer the JWS-vs-JSON disambiguation to a future, more comprehensive revision of the Signature page. Specifically:

1. Update the Signature datatype prose (datatypes.html#Signature) to add a single clarifying sentence: *"The JWS may be in either Compact Serialization (RFC 7515 §3.1) or JSON Serialization (RFC 7515 §3.2); in both cases the serialized JWS is then encoded as `base64Binary` to fit the `Signature.data` datatype."*
2. **Do not** change `Signature.sigFormat` guidance in this ticket. Instead, file (or link) a follow-up ticket — co-ordinated with FHIR-52818 / FHIR-52819 and the broader Signature redesign Essien Ge proposes in the Zulip thread — to decide whether `sigFormat` should distinguish compact vs. JSON JWS (e.g., via `application/jose` vs. `application/jose+json`) or whether a new element is needed.

#### Justification

The Zulip thread shows the FHIR Signature design is under broader review (integrity vs. non-repudiation, XAdES / JAdES support, what should be signed in a referenced model), and `sigFormat` semantics intersect that review. Bundling a `sigFormat` recommendation into this ticket risks pre-empting decisions that belong to the larger Signature redesign tracked by sibling tickets (FHIR-52810…FHIR-52820, FHIR-47196 / FHIR-47197). A minimal editorial patch resolves the reporter's stated concern (ambiguity in the description) while keeping the substantive design choice in front of the WG with full context.

---

### Disposition C: Decline

#### Proposal

Persuasive — Considered for Future Use. Close the ticket without changing the Signature datatype text on the grounds that (a) `Signature.data` is typed as `base64Binary`, so the FHIR-level encoding is unambiguous regardless of which JWS serialization is used inside, and (b) RFC 7515 itself is the authoritative reference for which serializations are valid; readers needing that detail can follow the existing link. Note in the resolution that implementers MAY use either compact or JSON serialization and that `Signature.sigFormat` carries the MIME type.

#### Justification

The current text already states `Signature.data` is "base64 encoded JWS-Signature as specified in RFC 7515," which — read with `Signature.data`'s `base64Binary` datatype — is technically complete: the base64 layer belongs to FHIR, and the JWS internals are governed by RFC 7515. There is real cost to expanding normative-adjacent prose late in a release cycle (translation, conformance test impact, possible reinterpretation by tooling). The reporter's concern is largely a documentation-clarity preference, and the substantive design questions (which serialization to use, how to signal it) are being addressed by the broader Signature redesign tickets, not by FHIR-52817.

---

### Recommendation

**Recommended disposition:** B

This ticket is genuinely an editorial-clarity request — the reporter's accepted reading after the comment exchange is "make it clearer that both serializations are allowed and that base64 wrapping happens at the FHIR layer," which Disposition B delivers in one sentence with no risk of pre-empting in-flight design work. Disposition A is attractive because it also resolves the open `sigFormat` question Essien Ge raised, but `sigFormat` semantics are entangled with the wider Signature redesign discussed in the `fhir/infrastructure-wg > Signature comments` Zulip thread (integrity vs. non-repudiation, XAdES / JAdES, what gets signed across references) and with sibling tickets FHIR-52818 / FHIR-52819 and FHIR-47196 / FHIR-47197. Resolving the MIME-type guidance here would commit the WG to an answer before the broader picture is settled. Disposition C understates the value of the requested clarification: the comment thread shows two informed reviewers actually disagreed on what the current sentence means, which is good evidence the wording is unclear in practice. Disposition B threads the needle: ship the clarification the reporter has effectively already agreed to, and keep the `sigFormat` decision with its natural cohort of Signature-redesign tickets.
