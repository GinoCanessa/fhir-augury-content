## Summary

This ticket responds to an explicit open question raised inside the SMART
Web Messaging specification about whether SMART Web Messaging should
follow the pattern used by FHIRcast and CDS Hooks — namely, defining the
exchange / infrastructure layer in one specification and each event,
hook, message type, or activity in its own separately‑lifecycled
document. The submitter (Paul Knapp) recommends adopting that pattern:
keep the message exchange specification as a single base specification
and break the individual message definitions out into separate documents
so that each set of messages can mature on its own schedule and new
messages can be added without forcing a re‑ballot of the underlying
infrastructure. The ticket was originally dispositioned as "Considered
for Future Use" / "Defer" on 2021‑03‑05 (vote 7‑0‑0, Anderson / van den
Heuvel) per the linked Confluence reconciliation page, then reverted /
re‑opened by Lynn Laakso in May 2022 with the instruction "re‑open
after publication of STU1", indicating the work group intends to
revisit the structural question now that STU1 has shipped.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure SMART Web Messaging along the same lines as FHIRcast and
CDS Hooks: keep one base "SMART Web Messaging" specification that
defines only the transport / exchange infrastructure (envelope, message
header semantics, security model, error handling, capability
negotiation), and move each individual message type / activity into its
own separately‑published, separately‑versioned document with its own
FMM (FHIR Maturity Model) level. Concretely: (a) remove the normative
definitions of individual message types from the base SMART Web
Messaging IG and replace them with a registry / index of message types
plus their hosting documents; (b) create one IG (or one
`MessageDefinition` / activity bundle per IG) per message family,
each with its own FMM and ballot lifecycle; (c) document the rules a
new message document must follow to claim conformance to the base
exchange spec (naming, JSON envelope shape, required metadata, security
profile binding, registration). Update the "Existing Wording" passage
in the spec to record the chosen direction.

#### Justification

This is the pattern the submitter recommends, the pattern explicitly
called out in the ticket as the FHIRcast / CDS Hooks precedent, and
the pattern that aligns with FHIR Infrastructure's general preference
for separating slow‑moving infrastructure from fast‑moving content. It
delivers the stated benefit — new message types can be added or
matured without re‑balloting the exchange infrastructure — and it
mirrors a model the work group already supports for two adjacent
specifications, so reviewers and implementers do not have to learn a
new structural idiom. The 2022 re‑open note ("re‑open after
publication of STU1") signals that the work group expected to revisit
this once the base spec was stable; STU1 publication is the natural
inflection point at which a structural split is least disruptive.

---

### Disposition B: Alternative Approach

#### Proposal

Keep SMART Web Messaging as a single specification document, but
introduce an internal modularization that achieves most of the
"separation of concerns" benefit without splitting into multiple
ballot artifacts: (a) carve the spec into clearly‑labelled
"Infrastructure" and "Message Catalog" sections, each with its own
section‑level FMM; (b) require every message type / activity to be
defined via a `MessageDefinition` (or equivalent conformance resource)
so message additions can be expressed as resource‑level changes rather
than narrative re‑writes; (c) commit, in the spec, to a publication
policy that allows additive message changes (new message types, new
optional fields) to ship as point releases without reopening the
infrastructure normative content. Optionally publish individual
message catalogs as companion non‑normative "module guides" linked
from the main IG.

#### Justification

This addresses the underlying concern — that adding a new message
should not force re‑ballot of the entire exchange infrastructure —
without the overhead of standing up and maintaining N separate ballot
artifacts, each with its own ballot cycle, project scope statement,
and publication request. For a young specification with a still‑small
catalog of message types, the operational cost of multi‑IG governance
(per the FHIRcast / CDS Hooks experience) may outweigh the benefit
until the catalog grows. Section‑level FMM and additive‑release
policies are mechanisms FHIR already uses elsewhere (e.g., resource
maturity within the core spec) and can deliver independent evolution
of messages while preserving a single, easier‑to‑navigate
specification for implementers.

---

### Disposition C: Decline

#### Proposal

Decline the request and close as "Not Persuasive" / leave the
specification monolithic. Keep all message types and activities in the
base SMART Web Messaging specification; revise the "Existing Wording"
passage simply to record the work group's decision that, given the
expected size of the SMART Web Messaging message catalog, separating
the infrastructure from the message definitions is unnecessary
overhead.

#### Justification

The 2021‑03‑05 reconciliation ("Considered for Future Use", 7‑0‑0)
already reflected a reasoned consensus to defer this restructuring.
Splitting into multiple specifications has real costs — additional
ballot artifacts, additional project scope statements, additional
publication coordination, and an extra layer of versioning that
implementers must reason about — and FHIRcast / CDS Hooks adopted that
pattern in part because their event / hook catalogs are expected to
grow large and to be authored by parties outside the core WG. If
SMART Web Messaging does not yet have evidence of comparable catalog
growth or external authorship demand, the decline preserves
implementer simplicity until that evidence exists.

---

### Recommendation

**Recommended disposition:** A

The submitter's request is structurally consistent with two adjacent
HL7 specifications (FHIRcast and CDS Hooks) that the same
Infrastructure work group already maintains, and the explicit 2022
re‑open instruction ("re‑open after publication of STU1") signals
that the work group itself anticipated revisiting the question once
the base spec stabilized — i.e., now. Disposition B is attractive on
short‑term overhead grounds, but section‑level FMM inside a single
IG is a weaker promise to implementers than a separately‑balloted
companion document, and history (CDS Hooks, FHIRcast) suggests
message / hook catalogs do grow once a base exchange spec is real.
Disposition C effectively re‑affirms the 2021 deferral, which the
work group has already chosen to revisit. Accepting and committing
to the FHIRcast / CDS Hooks pattern gives SMART Web Messaging a
predictable governance model for message growth, lets the
infrastructure mature on its own track, and is the least surprising
outcome for implementers familiar with the neighbouring specs. The
work group should, as part of accepting, scope the actual split (one
IG per message family vs. one IG per message vs. a registry +
companion catalog) and assign owners before the next ballot cycle.
