## Summary

Brian Postlethwaite is asking the FHIR Infrastructure work group to clarify the
behaviour of the FHIR Mapping Language (StructureMap) `list-option` source
selectors — specifically `first`, `not_first`, `last`, and `only_one` — and how
they relate both to the (undefined) rule evaluation order and to the now-supported
FHIRPath source expressions. The ticket title flags that "rule evaluation order
[is] undefined" and that today the spec leans on `first`/`last` to manage that
order, and asks how that actually helps in practice. The reporter then suggests
that, given FHIRPath source expressions are supported, the list-option values
could be described in terms of equivalent FHIRPath calls (e.g. `src.element.first()`,
`src.element.skip(1)`, `src.element.last()`), and asks for a precise definition
of `only_one` versus `first` — in particular whether `only_one` is intended to
behave like FHIRPath `single()` (i.e. assert that exactly one value is present
and otherwise raise an error/check condition) or whether it just means "the
mapper does not care which one of several values is used."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Mapping Language / StructureMap specification (the `list-option`
source-list-mode definitions and the surrounding `rules` description in
`structuremap.html` / the mapping-language narrative pages) to:

1. State explicitly that, **within a group**, rule evaluation order over a
   source collection is otherwise unspecified, and that `first` / `not_first`
   / `last` are the supported levers for callers that need order-sensitive
   selection behaviour.
2. For each `list-option` value (`first`, `not_first`, `last`, `only_one`,
   `share`, `collate` — whichever apply on the source side), add a normative
   "Equivalent in a FHIRPath source expression" note. At minimum:
   - `first` ↔ `<src>.first()`
   - `not_first` ↔ `<src>.skip(1)`
   - `last` ↔ `<src>.last()`
   - `only_one` ↔ `<src>.single()` (i.e. evaluate to the single value, and
     raise an error if the source has zero or more than one element)
3. Add an explicit, prose distinction between `first` and `only_one`:
   `first` silently picks the first element when more than one is present,
   while `only_one` asserts that exactly one element is present and is a
   check condition (failing the rule / raising an error if not).
4. Apply the same clarifications, with appropriate wording, to the target
   `list_mode` codes where the codes overlap.

#### Justification

The reporter is the StructureMap editor and is identifying real ambiguities
that implementers regularly trip over. The fix is purely editorial /
clarifying — it does not change wire format, schemas, or the `map-source-list-mode`
value set membership — and therefore is safe across versions. Anchoring each
list-option value to a concrete FHIRPath equivalent makes the semantics
testable: an implementation that supports FHIRPath source expressions can be
validated against the equivalence, and a conformant engine can be expected to
produce the same output for both forms. Calling out that `only_one` is the
`single()`-style check (not a synonym for `first`) closes the question the
ticket raises and aligns the mapping language with FHIRPath terminology that
implementers already know.

---

### Disposition B: Alternative Approach

#### Proposal

Take the same clarifications as Disposition A, but **deprecate** (soft —
"discouraged, retained for backward compatibility") the source `list-option`
values whose semantics are fully expressible in FHIRPath (`first`, `not_first`,
`last`, and `only_one`), and recommend that new maps use the FHIRPath source
expression form (`src.element.first()`, `…skip(1)`, `…last()`, `…single()`)
instead. Keep `share` and `collate` (and any target-side `list_mode` codes that
have no clean FHIRPath equivalent) untouched. Add an editorial note that the
list-option codes remain valid for legacy maps but that authoring tools should
prefer the FHIRPath form for new content. No codes are removed from the value
set or code system in this cycle.

#### Justification

Once FHIRPath source expressions are supported, `first` / `not_first` / `last`
/ `only_one` become a second way of saying something the mapping language can
already say more precisely. Carrying both indefinitely doubles the surface area
implementers have to support and the surface area the spec has to define, and
it is exactly what produces the ambiguity Brian is reporting (`first` vs
`only_one`). Soft deprecation captures the editorial intent ("prefer FHIRPath")
without breaking existing maps or removing codes from the value set, which
keeps the change non-breaking for FHIR R4/R5/R6 implementers and for tools
compiled against the existing `map-source-list-mode` code system.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive**: leave the StructureMap source `list-option` and
target `list_mode` definitions as they are, on the grounds that the existing
text ("first matching element", "last matching element", "exactly one
matching element expected") is sufficient and that introducing FHIRPath-equivalence
notes risks implying a normative equivalence that engines are not currently
required to implement.

#### Justification

A defensible decline rests on two points: (1) FHIRPath source expressions in
StructureMap are a relatively recent addition and not all engines support them,
so codifying `list-option = X ↔ FHIRPath function Y` could create a conformance
trap — engines that implement only the list-option form would now appear
non-conformant against the prose. (2) The semantic distinction between `first`
and `only_one` arguably already exists in the mapping language reference (one
selects, the other asserts cardinality), so the perceived ambiguity is an
editorial gap rather than a substantive defect.

---

### Recommendation

**Recommended disposition:** A

The ticket is a focused editorial request from the StructureMap editor about
two real ambiguities — undefined rule evaluation order and the `first` vs
`only_one` distinction — and proposes a concrete, low-risk way to resolve them
(equivalence with FHIRPath functions). Disposition A delivers exactly that
clarity without changing the wire format, the code systems, or any existing
map's behaviour, and it leverages FHIRPath terminology (`single()`) that
implementers already understand. Disposition B is attractive on aesthetic
grounds but combines an editorial fix with a deprecation policy decision that
deserves its own ticket and its own work-group discussion, and risks scope
creep that could stall the simple clarification Brian is asking for.
Disposition C is hard to defend: the reporter is the editor flagging that the
text as it stands does not answer a basic question implementers ask, and the
"conformance trap" concern in C is better addressed by phrasing the FHIRPath
equivalence notes as informative ("equivalent to") rather than dropping them.
The work group should adopt A now and, if it wants to pursue Disposition B's
deprecation direction, raise a separate change request that can be debated on
its own merits.
