## Summary

Brian Postlethwaite is requesting a documentation enhancement to the
FHIR specification's coverage of the FHIRPath `subsumes()` and
`subsumedBy()` functions. Today, the spec text for those functions
mentions that they require a terminology service and shows an example
that calls the `%terminologies.subsumes()` factory form, but it does
not show an example using the actual fluent function being defined.
The ticket asks that a side-by-side comparison example be added so
readers can clearly see the two equivalent invocation styles — fluent
(`Procedure.category.subsumes(Procedure.code)`) versus factory
(`%terminologies.subsumes(Procedure.category, Procedure.code) =
'subsumes'`) — and can choose the form that best suits their use. The
ticket also asks for a hyperlink between the two definitions and for
the same comparison example to be repeated in the `%terminologies`
section so it is discoverable from either entry point. No related
Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-references.

## Details

**Description:**

The **subsumes()** and **subsumedBy()** functions mention using a
terminology service, and show an example of calling using the
**%terminolgy.subsumes()** function, but not using the actual function
created.
This below example should be clearer to show the difference between
using the fluent or factory approach.

```
Procedure.category.subsumes(Procedure.code) // via fluent invocation
%terminologies.subsumes(Procedure.category, Procedure.code) = 'subsumes' // as a boolean test using the %terminology factory
```

And also include a link to the function that is also being used, and
repeat the example down in the %terminology section too so people are
aware of the 2 ways to call the functionality and be able to decide
which is more appropriate for them.

## Proposed Dispositions

> Note: Because no GitHub cross-references and therefore no
> `repo-analysis` briefing were available, the dispositions below
> describe the change at the spec-section level rather than at a
> specific authoring path. A reviewer with repo context should pin
> these to concrete files (e.g., the FHIRPath / terminology pages in
> the FHIR Core sources) before implementation.

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR specification's FHIRPath documentation for the
`subsumes()` and `subsumedBy()` functions so that each function's
description includes a worked example showing both invocation styles
side by side, using the reporter's `Procedure.category` /
`Procedure.code` example:

- Fluent form: `Procedure.category.subsumes(Procedure.code)` —
  returning the subsumption relationship code (or boolean, as currently
  specified).
- Factory form using the terminology environment variable:
  `%terminologies.subsumes(Procedure.category, Procedure.code) =
  'subsumes'` — wrapped as a boolean test against the relationship
  code.

Add a hyperlink from the `subsumes()` / `subsumedBy()` function
descriptions to the `%terminologies.subsumes` factory definition (and
vice versa). Repeat the same comparison example in the
`%terminologies` section so readers approaching the topic from the
factory direction also see the fluent equivalent and can pick the
form that fits their style.

#### Justification

The change is purely additive documentation: it does not alter the
behaviour, signatures, or semantics of any FHIRPath function. The
reporter is correct that the current text leaves a gap — the function
being defined is not actually demonstrated, and the two equivalent
ways of invoking the terminology service are not presented together,
which makes it harder for implementers to choose. Bidirectional
cross-linking between the two sections is consistent with how the
spec already cross-links related FHIRPath operators and environment
variables. The change is low risk, has no normative impact, and
directly addresses the reported clarity issue.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying need but consolidate rather than duplicate.
Rather than carrying the full worked example in two places (the
fluent function descriptions and the `%terminologies` section),
introduce a single short "Calling the terminology service from
FHIRPath" subsection that:

1. Presents the fluent / factory pair once, with both examples and a
   brief note on when each is more idiomatic (fluent for chaining,
   factory for explicit boolean tests against a specific relationship
   code).
2. Includes a short table mapping each fluent function (`subsumes`,
   `subsumedBy`, and any siblings such as `memberOf`, `validateCS`,
   `expand` if applicable) to its `%terminologies.*` counterpart.
3. Is linked from each individual function description and from the
   `%terminologies` section, instead of repeating the example body
   inline in each location.

#### Justification

This delivers the same clarity benefit as Disposition A while
avoiding two parallel copies of the example that can drift over
time. It also generalises the fix: today only `subsumes()` /
`subsumedBy()` are flagged, but the same fluent-vs-factory ambiguity
exists for the other terminology functions, and a consolidated
section future-proofs the docs against the same ticket being filed
again for `memberOf`, etc. The trade-off is one extra hop for the
reader (a click instead of inline text), which is a modest cost
against the maintenance benefit.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the existing text already names both the
fluent function and the `%terminologies` factory, and the FHIRPath
specification is the normative source for the function definition
itself. Implementers needing further example material should look to
the FHIRPath specification or to companion implementer guidance
rather than expanding the FHIR Core spec text.

#### Justification

A defensible decline rationale would be scope minimisation: the FHIR
Core spec aims to specify behaviour, not to teach idiomatic FHIRPath
style, and adding worked examples for every function variant could
balloon the spec. However, this rationale is weak in this specific
case because the request is small, the reporter has supplied the
example text, and the spec already chooses to give an example — the
ticket only asks that the example actually demonstrate the function
being defined. Decline is included for completeness rather than as a
strong option.

---

### Recommendation

**Recommended disposition:** A

The request is a small, additive documentation fix from an
experienced contributor who has supplied the exact example text and a
clear motivation. Accepting as requested is the lowest-friction path
to closing a real clarity gap (the current example does not
demonstrate the function it sits under) and to making the two
equivalent invocation styles discoverable from either entry point.
Disposition B's consolidation idea is attractive in principle and
worth keeping in mind if the same gap is later reported for other
terminology functions, but proposing it now risks expanding the
ticket's scope and delaying a fix that is otherwise ready to land.
Disposition C is not justified by the evidence: the cost of the
change is low, there is no normative risk, and the reporter's point
that the function being defined is not actually demonstrated is
correct on the face of the current text.
