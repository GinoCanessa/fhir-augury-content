## Summary

This ticket requests that the SDC (Structured Data Capture) IG define a
new mechanism for declaring a Questionnaire-specific Liquid template that
governs how a QuestionnaireResponse derived from that Questionnaire is
rendered (specifically, how its narrative is generated). The IG publisher
already supports generic, resource-level Liquid templates, but
QuestionnaireResponses are unusual in that the optimal presentation of
the captured data depends on the structure and intent of the specific
Questionnaire — the order, phrasing, and grouping of questions during
data capture is frequently different from the order/labels that make
sense when the captured data is consumed. Today the only response-side
control is the `enableWhen`-style "show only when displaying completed
form" toggle, which is too coarse.

The ticket proposes a new extension named `responseRenderingLiquid`,
typed `0..1 canonical(Library)`, with context `Questionnaire`, that
points to a Library carrying a [FHIR
Liquid](https://confluence.hl7.org/spaces/FHIR/pages/66938964/FHIR+Liquid+Profile)
template used to generate the narrative of QuestionnaireResponses for
that Questionnaire. The ticket originally resolved as "Considered for
Future Use" (Brian Postlethwaite / Brenin Rhodes, 3-0-1, 2025-07-10) on
the grounds that more implementation experience was needed; that
resolution was reverted on 2026-03-26 to be re-opened post-STU4. On
2026-04-16 the workgroup agreed in principle to proceed, with Lloyd to
draft a wiki/spec page including examples and to address how
Questionnaire variables interact and where Questionnaire-specific vs.
Questionnaire-agnostic Liquid each apply.

## Details

**Description:**

The IG publisher supports defining generic Liquid templates that govern
the rendering of a given resource. It would be useful to have an ability
to define a Liquid template that applies to QuestionnaireResponses from
a given Questionnaire.

Right now, the only control we have over rendering QuestionnaireResponses
is the ability to flag certain items as "only display this when showing
a completed form" or "don't display this when showing a completed form".
This is pretty limiting. As well, this doesn't necessarily drive how the
narrative is populated.

The reality is that the order in which questions are asked, and how the
questions are phrased is often quite different than the order, label,
and other organization you would want when presenting information for
actual use. However, making optimal decisions about this can't be done
on a generic basis. It requires knowledge of the specific Questionnaire
whose response is being rendered.

Proposal:

Define a new extension called "responseRenderingLiquid" with a type of
`0..1 canonical(Library)` and context of `Questionnaire`.

Definition: Points to a Library containing [FHIR
Liquid](https://confluence.hl7.org/spaces/FHIR/pages/66938964/FHIR+Liquid+Profile)
template that defines how the narrative for QuestionnaireResponses
created for this Questionnaire should be generated.

## Proposed Dispositions

> Note: with no GitHub cross-references, the dispositions below identify
> the *likely* affected repos by reasoning about the SDC specification
> (the work happens in the SDC IG, with a possible touch in the FHIR
> extensions registry / FHIR Liquid profile pages). Concrete file paths

> implementation.

### Disposition A: Accept As Requested

#### Proposal

Add a new extension to the SDC IG named `responseRenderingLiquid`:

- **Context:** `Questionnaire`
- **Cardinality:** `0..1`
- **Type:** `canonical(Library)`
- **Definition:** Points to a Library resource whose content conforms to
  the [FHIR Liquid Profile](https://confluence.hl7.org/spaces/FHIR/pages/66938964/FHIR+Liquid+Profile)
  and that defines how the narrative for any QuestionnaireResponse
  derived from this Questionnaire should be generated.
- **Bindings / dependencies:** The referenced Library SHALL be of type
  `liquid` (matching the FHIR Liquid Profile usage) and SHOULD be
  included in the IG's dependency closure when present.
- **Companion guidance page:** Add a new SDC IG page (per Lloyd's
  2026-04-16 commitment) that:
  1. Shows a worked example of a Questionnaire with a
     `responseRenderingLiquid` extension and the corresponding Library +
     Liquid template.
  2. Explains how Questionnaire-level variables (and item-level
     variables) are exposed to the Liquid context (e.g., `%resource` =
     QuestionnaireResponse, plus access to the parent Questionnaire and
     its variables).
  3. Distinguishes Questionnaire-specific Liquid (this extension) from
     existing Questionnaire-agnostic Liquid mechanisms used by the IG
     publisher, including precedence rules when both are present.
  4. Addresses fall-back behavior when the Library is unresolvable or
     the template fails to execute (recommend: emit the standard
     auto-generated narrative).

Likely-touched authoring locations (to be confirmed against an SDC IG
repo briefing): the SDC IG's extension definitions directory, the IG's
narrative on Questionnaire/QuestionnaireResponse rendering, and a new
guidance page under the SDC IG's input pages tree.

#### Justification

The proposal directly fulfills Lloyd's stated need and aligns with the
workgroup's 2026-04-16 in-principle agreement to proceed. The mechanism
is fully consistent with how SDC already attaches behavioral extensions
to Questionnaires (e.g., other `Questionnaire`-context extensions
pointing to `canonical(Library)`), so it does not introduce a new
modelling pattern. Reusing the existing FHIR Liquid Profile keeps the
authoring surface familiar to implementers who already use Liquid for
generic resource rendering in the IG publisher. Defining it as `0..1`
preserves backward compatibility — existing Questionnaires render
exactly as today; only authors who opt in get the customized narrative.

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) a single `responseRenderingLiquid`
extension on `Questionnaire`, define a small *family* of presentation
extensions that the IG publisher (and any consumer) can pick up:

- `Questionnaire.extension:responseRenderingLiquid` →
  `canonical(Library)` — narrative generation, as proposed.
- `Questionnaire.extension:responseRenderingStyle` →
  `code` (binding: a small value set such as `default | grouped |
  tabular | custom`) — coarse-grained, codeable rendering hints that do
  not require Liquid authoring.
- Optionally allow the Liquid extension at the `Questionnaire.item`
  level too (context `Questionnaire.item`), so authors can customize a
  sub-section without rewriting the full narrative template.

Document precedence: item-level Liquid > Questionnaire-level Liquid >
generic Questionnaire-agnostic Liquid > IG-publisher default narrative.

#### Justification

Many implementers will want better-than-default rendering without
authoring a full Liquid template. A codeable hint covers the common
cases cheaply; the Liquid extension covers the long tail. Allowing the
Liquid extension on `Questionnaire.item` matches the granularity at
which authors actually think about presentation (sections / groups)
and avoids forcing every customization through a single monolithic
template. The trade-off vs. Disposition A is more surface area and a
new value set to maintain, plus precedence rules to specify and test —
but the underlying need is broader than a single extension can serve,
and the workgroup discussion already flagged "how the various
questionnaire variables fit in" as a non-trivial concern that benefits
from finer-grained context.

### Disposition C: Decline

#### Proposal

Do not adopt the new extension. Re-resolve the ticket as **Considered
for Future Use** (or **Persuasive with Mod** to point implementers at
existing mechanisms), citing that:

- The IG publisher already supports generic Liquid templates keyed by
  resource type, which can branch on `QuestionnaireResponse.questionnaire`
  to apply per-Questionnaire logic without a new extension.
- Authors who need full control today can author the narrative directly
  on the QuestionnaireResponse (`text.div`) or supply a custom
  rendering tool outside the IG publisher.
- The original 2025-07-10 vote (3-0-1) explicitly cited "need more
  experience before including in the spec"; no concrete implementation
  experience or worked examples have been added to the ticket since
  re-opening.

#### Justification

Declining (or re-deferring) is defensible while no public examples,
prototype Liquid library, or implementer feedback are attached. Adding
a normative-context extension that depends on a Library carrying a
specific FHIR Liquid profile creates a tooling expectation (the IG
publisher and any rendering client must resolve the Library, execute
Liquid against the QR, and merge the result into narrative) that the
SDC community has not yet validated end-to-end. The original "need
more experience" rationale has not been retired by new evidence — only
by a procedural decision to re-open post-STU4.

### Recommendation

**Recommended disposition:** A

The workgroup has already agreed in principle (2026-04-16) to proceed,
with Lloyd committing to draft a worked example and address the open
questions (variable scoping, Questionnaire-specific vs. -agnostic
Liquid). The proposed mechanism is small, additive, optional
(`0..1`), and reuses the existing FHIR Liquid Profile rather than
inventing a new templating surface, so the implementation and
maintenance cost is low and the backward-compatibility risk is
essentially zero. Disposition B's broader surface (codeable hint +
item-level Liquid) is appealing and worth tracking as a follow-up
ticket once the base extension has been exercised, but bundling it
into this ticket would expand scope past what the workgroup actually
agreed to and would re-introduce the "need more experience" objection.
Disposition C ignores both the workgroup's stated intent to proceed and
the concrete implementer pain — generic Liquid keyed off
`QuestionnaireResponse.questionnaire` is technically possible but
forces every author to maintain a single shared template, which is
exactly the limitation the ticket is calling out. Accept A, with the
explicit dependency that the SDC IG page Lloyd will draft must land in
the same ballot cycle as the extension definition so reviewers see the
mechanism and a worked example together.
