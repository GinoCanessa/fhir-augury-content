## Summary

Martin von Siebenthal observed that at the end of the FHIR Resource List
page (`resourcelist.html`) the build emits lines such as
"Not listed enough: MolecularDefinition = 5" and
"Not listed enough: MolecularSequence = 5" with no in-spec explanation
of what that label means. He filed this as a Technical Correction
(rather than a Question) so that the spec itself could explain the
counter to all readers, since other implementers are likely to be
similarly confused. A second commenter (jwhitby, 2025‑09‑15) confirmed
that they had spotted the same wording and were about to file a
duplicate ticket, indicating the confusion is not an isolated case.

There are no Jira cross-references for this ticket, but a 2023
`#committers` Zulip thread ("Resources not listed enough?") covers the
same surface from the editor side: Jose Costa Teixeira hit the same
"Not listed enough: X = 5" output for resources he had recently added
(BiologicallyDerivedProductDispense, DeviceAssociation, InventoryItem,
TestPlan), and Gino Canessa explained that the file to edit is
`source/resourcelist.html` and that each new resource must be added to
each of the tables on that page in the correct places. In other words,
the counter is an authoring/QA signal saying "this resource appears in
fewer than the expected number of tables on the resource-list page",
and it is fixed by listing the resource in every category table — not
by changing anything in the resource definition itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a short, reader-facing explanation of the "Not listed enough:
&lt;Resource&gt; = N" line to the FHIR Core spec, so that readers of
the published Resource List page understand it. Concretely:

- In `source/resourcelist.html` (or the template that emits the
  counter), surround the existing counter output with an explanatory
  paragraph along the lines of:
  *"The following entries are diagnostics emitted by the publication
  tooling: each resource on this page is expected to appear in N of
  the categorisation tables above (Clinical, Administrative, Workflow,
  Financial, Specialised, Foundation, Base). A 'Not listed enough:
  &lt;Resource&gt; = K' message means the resource is currently listed
  in fewer than the expected number of tables and should be added to
  the missing ones by editors. End-users can ignore these messages;
  they exist to flag editorial gaps before publication."*
- As a small follow-on, ensure MolecularDefinition and
  MolecularSequence (the two resources Martin called out) are actually
  added to the appropriate categorisation tables on
  `resourcelist.html`, which removes the warnings for those two
  resources at the same time.

#### Justification

The ticket is explicitly framed as a documentation / clarity request,
and the second commenter confirms the message is opaque to more than
one reader. The 2023 `#committers` Zulip thread shows that even active
spec editors hit the same confusion until told what to do, which is a
strong signal that the inline message needs to be self-describing.
Adding a sentence or two next to the counter is a low-risk,
backward-compatible change that lives entirely on the resource-list
page. Fixing the two flagged resources (MolecularDefinition,
MolecularSequence) at the same time keeps the page diagnostic-clean
and gives a concrete worked example readers can compare against.

---

### Disposition B: Alternative Approach

#### Proposal

Treat "Not listed enough" as a build/editorial diagnostic rather than
end-user content, and *suppress it from the published page* while
surfacing it through tooling instead. Specifically:

- Modify the publication tooling (the generator that currently emits
  the "Not listed enough" lines into `resourcelist.html`) so that
  these messages are written to the build log / QA report and gated
  behind a build-warnings page (or the existing publication QA
  output), but are no longer rendered into the published HTML the
  community reads.
- Optionally, fail the build (or raise a publication QA warning) when
  any resource has `count < expected`, so editors are forced to fix
  the categorisation tables before publication rather than shipping
  the diagnostic.
- Still fix MolecularDefinition / MolecularSequence so the build is
  clean after the change.

#### Justification

The root cause of Martin's confusion is that an editorial diagnostic
intended for spec editors is being shown to all readers of the
specification. That is a layering problem more than a wording problem:
the spec page is being asked to do double duty as both reference
material and as an editor TODO list. Moving the message into the build
toolchain matches the way other publication QA signals are handled,
removes the need to teach every reader what the counter means, and
forces the underlying issue (resources missing from categorisation
tables) to be resolved at edit time instead of papered over with prose.
The trade-off versus Disposition A is that it touches the publication
tooling rather than just the page, so it is a slightly larger change
and needs a publisher-side owner.

---

### Disposition C: Decline

#### Proposal

Decline the request as **not actionable as a Technical Correction**:
the "Not listed enough" output is a build-tooling diagnostic, not
normative or informative spec content, and the appropriate remediation
is purely editorial — add MolecularDefinition and MolecularSequence to
each of the resource categorisation tables on
`source/resourcelist.html` so that the diagnostic disappears for those
two resources. Close the ticket as "Not Persuasive — editorial fix
applied" once those entries are in place, without adding any
explanatory prose to the spec.

#### Justification

If the diagnostic only shows up when a resource is missing from one
or more categorisation tables, then by definition a clean spec never
displays it. Documenting a message that should never appear on a
properly maintained page risks ossifying a piece of the build tooling
into the spec text and inviting future readers to treat it as a real
specification feature rather than a transient warning. Declining,
combined with the editorial fix, removes the immediate user-visible
symptom Martin reported without committing FHIR-I to maintaining
documentation about an internal QA counter.

---

### Recommendation

**Recommended disposition:** B (with the editorial fix from C as a
prerequisite).

The strongest signal in the evidence is that the same diagnostic has
confused at least three different people across two years (Jose in
2023, Martin in May 2025, jwhitby in September 2025), and in every
case the resolution required someone with insider knowledge to
explain that the message is a build-time editorial signal. That
pattern says the message is in the wrong place, not that it needs
better wording. Disposition B fixes the layering problem (move the
diagnostic out of the published page and into publication QA) and
will make this entire class of confusion go away for future readers,
not just the two resources Martin happened to notice. Disposition A
is a reasonable fallback if the publication tooling change is judged
too heavy for a Technical Correction — in that case, accept A and
make the wording explicit that this is an editorial diagnostic.
Disposition C alone is not recommended, because it leaves the
underlying mechanism (a confusing build-only message rendered to
end-users) in place for the next resource that gets added without
being categorised.
