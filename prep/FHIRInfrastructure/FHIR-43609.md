## Summary

The reporter (Cooper Thompson) observes that the `backport-related-query`
extension, as shown in the Encounter example of the R5 Subscriptions Backport
IG (`Basic-r4-encounter-complete`), appears to use a templating syntax inside
the query URL — specifically the token `[id]` in
`http://example.org/fhir/Encounter/[id]/$prescribed-medications`. The IG does
not currently describe what template language (if any) is supported, how
tokens are delimited, which tokens are legal, or how substitution is performed
at notification time. The ticket asks the WG to either (a) specify the
templating behavior if it is intended, or (b) rewrite the example to remove
the templating if it is not.

The reporter also notes a companion ticket,
[FHIR-43608](https://jira.hl7.org/browse/FHIR-43608) ("The query for notify
pull wouldn't be in the Topic, only the query 'name' would"), which proposes
making the actual query string optional on the `SubscriptionTopic` and
moving it onto the notification. FHIR-43608 has since been resolved as
**Persuasive** (assigned to Gino Canessa, labeled `block-vote-3`). If
FHIR-43608's resolution is applied — i.e., topics only carry the coded
query name and notifications carry the resolved query string — then the
templating question on the topic largely goes away, because the query
string in the notification would already be fully resolved for that specific
event. The reporter explicitly flags this dependency in the description.

Reporter's companion ticket on the same IG. It argues that the actual query

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Subscriptions Backport IG to explicitly document a minimal
template-token syntax for the URL carried by the `backport-related-query`
extension, and keep the Encounter example intact.

Concretely:

1. In the narrative that introduces `backport-related-query` (alongside the
   description that currently links to `Basic-r4-encounter-complete`), add
   a new normative sub-section titled "Template tokens in related-query
   URLs" that specifies:
   - Tokens are delimited with square brackets, e.g. `[id]`.
   - The only defined token in this IG is `[id]`, which resolves to the
     `id` element of the resource that triggered the notification (i.e.,
     the `focus` of the notification Bundle).
   - Substitution is performed by the Subscription server when it emits
     the notification; consumers SHALL receive a fully-resolved URL and
     MUST NOT be required to perform substitution themselves.
   - Any unknown `[name]` token in a related-query URL is an authoring
     error and SHALL cause the server to fail the notification with an
     `OperationOutcome` rather than emit a partially-resolved URL.
2. Keep `Basic-r4-encounter-complete.html` unchanged; add a margin note
   explaining that `[id]` in the example is a template token per the new
   section.
3. Cross-link the new section from the notification-side narrative
   (`notifications.html#adding-queries-to-notifications`) so that
   implementers of both sides see the contract.

#### Justification

This preserves a feature implementers clearly find useful (authors of
topics can share a template once and have the server substitute at
notification time). A tight, single-token vocabulary (`[id]` only in
this IG) is easy to implement and easy to validate, and square-bracket
placeholders are already the de-facto convention in the existing example,
so the change is additive rather than breaking for anyone who has already
implemented against the current Encounter example. It also aligns with
the community's general pattern of using URL-style operation parameters
rather than inventing a new expression language.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt FHIR-43608's resolution as the preferred path and make the
templating question moot: remove the templated URL from the topic
definition entirely, and specify that when the query string is runtime-
dependent, only the coded query `name` lives on the `SubscriptionTopic`
while the fully-resolved query URL is carried on the notification Bundle
via the `backport-related-query` extension.

Concretely:

1. Update the IG to state that a `SubscriptionTopic` MAY carry a query
   `name` (coded) without the query `string`; when the string would
   require runtime substitution, it SHALL be omitted from the topic and
   provided only on the notification.
2. Rewrite `Basic-r4-encounter-complete` so that the topic carries only
   the coded query name (e.g., `prescribed-medications`), and the
   notification Bundle carries a fully-resolved URL with no brackets:
   `http://example.org/fhir/Encounter/enc-123/$prescribed-medications`.
3. Explicitly state that no templating language is defined in this IG,
   and that any brackets in a query string are to be treated as literal
   URL characters.

#### Justification

FHIR-43608 was accepted as **Persuasive** by the same work group, which
means the IG is already on track to separate "name on topic" from
"resolved string on notification." Aligning FHIR-43609 with that
direction removes an entire class of interoperability ambiguity (what
counts as a token? what is the substitution algorithm? what about
literal `[` in a real URL?) and keeps the IG from standing up a small,
half-specified expression language. It is also the least-surprise option
for downstream R5-to-R4 backport implementers, who already expect the
notification to carry fully-resolved data.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive — Already Addressed**, on the grounds that
FHIR-43608 (Resolved / Persuasive) already drives the Encounter example
and the topic-vs-notification split to a state where templating in the
topic is neither required nor expected. Close FHIR-43609 with a pointer
to FHIR-43608's resolution and a note that the example will be
refreshed as part of that work.

#### Justification

FHIR-43608 has been accepted and carries the `block-vote-3` label,
meaning it is already scheduled for the Encounter example and related
narrative. Opening a second normative change (defining a template
language) on top of a resolution that likely removes the need for
templating risks specifying machinery that the IG then does not use.
Declining here keeps the IG smaller and avoids introducing a novel,
IG-scoped templating grammar that would otherwise need maintenance in
subsequent ballots.

---

### Recommendation

**Recommended disposition:** B

FHIR-43608 has already been resolved as Persuasive by this same work
group, and its resolution directly contradicts the premise of Disposition
A: if the query string does not need to live on the topic at all, then
defining a template language *on the topic* is solving a problem the IG
is about to eliminate. Disposition C is tempting for the same reason,
but simply closing FHIR-43609 as "already addressed" leaves the Encounter
example in the IG with an unexplained `[id]` placeholder, which is
exactly the defect the reporter flagged. Disposition B picks up the
cleanup work FHIR-43608 implies but does not explicitly schedule — fixing
the example and stating plainly that no template language is defined —
so the two tickets land together and the IG ships with a consistent,
implementable story: coded names on topics, fully-resolved URLs on
notifications, no hidden grammar.
