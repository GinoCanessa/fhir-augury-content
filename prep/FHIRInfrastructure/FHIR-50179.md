## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive — Editorial. Fix the source text of the FHIR home-page
introduction so that:

1. Each occurrence of the literal `â€"` (UTF-8 em dash decoded as
   Windows-1252) is replaced with a properly UTF-8–encoded em dash
   (`—`, `U+2014`) — or equivalently the HTML entity `&mdash;` — in the
   three identified spots:
   - `FHIR® — Fast Healthcare Interoperability Resources (hl7.org/fhir) — is a next generation standards framework …`
   - `… suitable for use in a wide variety of contexts — mobile phone apps …`
2. Restore the missing whitespace adjacent to **implementability.**
   so the sentence renders as `… applying a tight focus on
   **implementability**.` (with the period outside the bold and a
   single space preserved before the bolded word).
3. Save the edited source as UTF-8 (no BOM) and verify the build does
   not re-introduce mojibake (i.e., confirm the publisher's pipeline
   does not re-encode the file as Windows-1252 on write).

The change touches the front-matter narrative for the FHIR Core
specification (the page that becomes `index.html` in the published
site). No structural / definitional content is affected.

#### Justification

These are pure presentation defects in normative-spec narrative. The
mojibake string `â€"` is unambiguously a UTF-8/CP-1252 round-tripping
bug — there is no editorial judgement to exercise, only a mechanical
fix. Leaving it in place undermines the perceived quality of the
specification's landing page, which is the first content most newcomers
read. The missing-space typography defect is in the same paragraph and
should be fixed in the same edit to avoid a second round-trip through
review. The change is local, low-risk, and carries no implementer
impact.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification. Apply the textual fix from Disposition A,
**and** add a guard in the publisher / CI pipeline to detect this class
of regression going forward. Concretely:

- Replace the three `â€"` sequences with `—` and fix the
  `implementability.` spacing as in Disposition A.
- Add a lightweight build-time check (e.g., a grep/regex step in the
  publisher or a pre-commit hook) that fails the build if any of the
  common mojibake signatures (`â€"`, `â€œ`, `â€`, `Â`, `Ã©`, etc.)
  appear in published HTML or in the narrative source files.
- Confirm the source file's encoding declaration (`<?xml … encoding="UTF-8"?>`
  or the HTML `<meta charset="utf-8">`) and ensure the build tool reads
  and writes UTF-8 consistently.

#### Justification

The literal symptom is small, but the root cause — an encoding
mismatch somewhere in the authoring/publishing toolchain — has caused
the same family of defects elsewhere in HL7 publications historically.
A one-line fix without a guard means the next editor who opens the
file in a tool that re-saves as CP-1252 will silently reintroduce the
bug. A targeted lint costs almost nothing and protects every page in
the build, not just this paragraph. The trade-off vs. Disposition A is
a small amount of FHIR Infrastructure / publishing-toolchain work; the
defect is in their domain anyway.

---

### Disposition C: Decline

#### Proposal

Not Persuasive. Close the ticket without changing the source text, on
the grounds that the issue is a transient artifact of the 6.0.0-ballot2
publication run rather than a defect in the source content.

#### Justification

This disposition is only defensible if FHIR Infrastructure can confirm
two things: (a) the source file in the `hl7/fhir` repository already
contains a correctly-encoded em dash (i.e., the mojibake was injected
by a one-off step in the ballot2 publication and has since been fixed),
and (b) the next published build verifiably renders `—` rather than
`â€"`. Without that evidence the ticket should not be declined,
because the screenshot/text in the ticket clearly shows the corrupted
characters in a published HL7 artifact.

---

### Recommendation

**Recommended disposition:** B

The textual fix is uncontroversial and should ship as part of the next
build (Disposition A is a strict subset of B). However, this exact
class of bug — UTF-8 text round-tripped through CP-1252 — is a
recurring failure mode in HL7 publications, and fixing only the three
visible occurrences leaves the pipeline that produced them unchanged.
A small mojibake-signature check in the FHIR Core publisher (or as a
narrative-source pre-commit hook) is cheap, lives squarely in FHIR
Infrastructure's remit, and prevents the workgroup from re-litigating
the same defect on a future ballot. Disposition C should only be
considered if the ballot2 source is already shown to be clean and the
defect is purely a publication-run artifact.
