## Summary

Brian Postlethwaite raises a gap in the FHIR Mapping Language (FML) /
StructureMap design: there is no clearly defined way for a `group` (and
its rules) to produce a *collection* of outputs — e.g., more than one
instance of the same target resource type, or instances of multiple
target types — from a single invocation. The ticket asks the workgroup
to specify how groups can yield collections in a way that is consistent
with FHIRPath (where every value is implicitly a collection) and with
the existing rule-processing model (where rules already iterate over
collections implicitly). It also asks for explicit guidance on what an
engine should do when a map declares more than one output parameter —
specifically, how/when it should instantiate each declared output.

This question is not academic: the related Zulip thread *"One source
producing two target files?"* in the **FHIR Mapping Language** stream
shows Jean Duteau hitting the same gap in practice with `validator_cli.jar`
when trying to transform a single Observation (e.g., Blood Pressure) or
Patient into multiple target instances/rows, and finding that the CLI
surfaces only one output even though the map can declare several.
Grahame Grieve confirmed in that thread that the map *can* express
multiple outputs but that the validator does not currently surface them,
and Brian explicitly tagged FHIR-54478 there as "one of the already
identified gaps" with non-breaking fix ideas in mind.

Jean Duteau described a real use case where a single source instance

the targets in a Bundle and ignore the wrapper downstream, but the
underlying spec gap (how groups produce collections / multiple outputs)
was not resolved.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Mapping Language / StructureMap specification to state
explicitly that:

1. Any `group` rule output is conceptually a *collection*, mirroring
   FHIRPath's "everything is a collection" semantics. A rule that
   currently produces a single target instance is just the singleton
   case of producing N target instances of the declared type.
2. A rule MAY produce more than one instance of a declared target
   parameter, and MAY produce instances of more than one declared target
   parameter, in a single invocation. Engines MUST iterate / create as
   many target instances as the rule yields, in source order.
3. When a `StructureMap` (or invoked group) declares more than one
   `output` parameter, the engine MUST create a separate output instance
   per declared output parameter, using the parameter's declared type
   and mode (`target`/`produced`) to decide instantiation. The mapping
   between a rule's emitted values and the named output parameters is
   determined by the parameter the rule writes to (e.g., `tgt1`, `tgt2`
   in the `group` signature).
4. Clarify that this is a clarification of intent, not a change to the
   abstract syntax: existing single-output maps remain valid and
   single-output by virtue of their `group` declaring one `target`
   parameter.

Add normative-style examples to the FML page covering: (a) a single
group producing 1..* of one target type, (b) a single group producing
instances of two distinct target types (the Blood-Pressure / Patient
+ location / AllergyIntolerance + reaction shapes from the Zulip
thread), and (c) the `Bundle`-wrapper pattern as an existing-engine
workaround.

#### Justification

This directly answers the four bullets in the ticket and aligns FML
with the FHIRPath collection model that mapping authors already think
in. Brian — the original ticket author and one of the engine
implementers — has stated on the related Zulip thread that he believes
this can be done in a non-breaking way for existing maps. Grahame's
confirmation that multi-target maps are already expressible (just not
surfaced by the validator CLI) supports treating this as a
specification clarification rather than a new feature, which is the
lowest-risk path for implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-specifying group/rule semantics globally, address the
problem in two narrower pieces:

1. **Spec clarification (small):** Add a short "Multiple outputs" /
   "Collection outputs" section to the FML page that documents the
   *current* (already-legal) ability to declare multiple `target`
   parameters on a group/map and to write to each from rules, without
   changing rule-evaluation semantics. Explicitly state that producing
   "a collection" of one target type is achieved either by declaring
   that target with cardinality `*` in the receiving structure or by
   wrapping outputs in a `Bundle`.
2. **Tooling change (separate item):** Open a follow-up against the
   reference engine / `validator_cli.jar` to surface all declared
   outputs (e.g., a `-output` parameter that may be repeated, or a
   directory output mode that writes one file per declared output
   parameter). Track that as an implementation ticket against the
   validator, not as a normative spec change.

#### Justification

This minimizes spec churn and keeps the normative surface narrow: it
acknowledges that the *language* already permits what Jean was trying
to do, and that the actual blocker reported in Zulip is the CLI, not
the language. Splitting tooling from spec also avoids the risk of an
under-specified "implicit collection" rule (bullet 2 of the ticket)
becoming load-bearing for engine behavior before implementers have
agreed on it. Trade-off vs. Disposition A: it leaves the deeper
"FHIRPath-like everything-is-a-collection" question (bullets 2 and 3 of
the ticket) unanswered, which may force the workgroup to revisit it
later.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54478 as **already addressed by existing mechanisms**, on
the grounds that:

- A group can already declare multiple `target` parameters, so
  multi-type output is expressible today.
- Producing a collection of a single target type is already expressible
  by giving the receiving element cardinality `*` (or by wrapping in a
  `Bundle`, as Brian advised on the Zulip thread and Jean adopted).
- Engine/CLI ergonomics around emitting multiple files belong to the
  reference implementation, not to the spec.

Recommend the reporter and the Zulip thread participants raise the
validator-CLI ergonomics against the implementation tracker.

#### Justification

Defensible only if the workgroup judges that the existing FML text and
the Bundle-wrapper workaround are sufficient and that the open
questions in the ticket (collection semantics, multi-output disambiguation)
do not actually leave implementers free to diverge. In practice the
Zulip thread shows two experienced implementers reaching for different
mental models (multi-target vs. Bundle vs. CLI parameter), which
suggests the spec is currently under-specified — making a pure decline
hard to defend.

---

### Recommendation

**Recommended disposition:** A

The ticket asks four pointed clarification questions about
group/rule/output semantics, and the related Zulip thread shows two
implementers (Grahame and Brian) and at least one mapping author (Jean)
each reasoning about the same use case from a different angle —
multi-target groups, FHIRPath-style implicit collections, Bundle
wrappers, and CLI parameters. That is exactly the symptom of a
specification gap, not a tooling gap, and it will not be closed by a
validator change alone. Disposition A (clarify FML so that group
outputs are explicitly collections, and define how multiple declared
output parameters are instantiated) takes the smallest normative step
that actually answers the ticket, lines up with how rules already
behave, and — per Brian's own note on the Zulip thread — can be
expressed in a way that is non-breaking for existing maps and engines.
Disposition B's tooling follow-up (surfacing multiple outputs from
`validator_cli.jar`) is still worth doing, but it should be a
*consequence* of the clarified spec rather than a substitute for it.
