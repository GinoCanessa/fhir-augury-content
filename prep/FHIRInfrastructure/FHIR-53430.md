## Summary

The FHIR Documents page contains a sentence asserting that "The XML Tools
reference implementation includes a XSLT transform that converts an XML
document into browser-ready XHTML." Grahame Grieve reports that this
statement is no longer true and asks the workgroup to decide whether the
note should be removed outright or replaced with an accurate pointer to a
currently-supported equivalent. The ticket was originally filed against
FHIR Composition; a follow-up comment from gdolin established that the
offending sentence actually lives in the FHIR Documents page and
reassigned the ticket to FHIR Infrastructure. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were surfaced for this change.

## Proposed Dispositions

### Disposition A: Accept As Requested — Remove the obsolete sentence

#### Proposal

In the FHIR Documents page (FHIR Core spec source for `documents.html`),
delete the sentence "The XML Tools reference implementation includes a
XSLT transform that converts an XML document into browser-ready XHTML."
along with the inline link to `downloads.html#refimpl`. Leave the
surrounding paragraph intact, adjusting punctuation and connectives as
needed so the remaining text still reads cleanly. Make no other changes
to the page or to `downloads.html`.

#### Justification

The reporter has confirmed the statement is factually incorrect: the
XML Tools reference implementation no longer ships an XSLT that
converts a FHIR XML document into browser-ready XHTML. Leaving an
incorrect pointer in normative-adjacent guidance is worse than having
no pointer at all, because implementers who follow the link will waste
time looking for a transform that does not exist and may infer that
HL7 endorses an unmaintained tool. Outright removal is the lowest-risk
fix, requires no new artifact, and does not depend on identifying or
vetting a replacement.

---

### Disposition B: Alternative Approach — Replace with current guidance on rendering FHIR Documents

#### Proposal

Replace the sentence with a short paragraph that describes the
currently-supported way to render a FHIR Document for human display.
Concretely, point readers at (a) the per-resource `text.div` narrative
that every entry in a Document Bundle already carries, and (b) the
Composition's own narrative, which by the rules of FHIR Documents is
the authoritative human-readable rendering of the document. Where a
maintained tool or library exists that assembles those narratives into
a single browser-ready view, link to it from `downloads.html#refimpl`
(or from a more appropriate "Rendering" subsection) and update the
Documents page to reference that anchor. If no such tool is currently
maintained, state explicitly that the recommended rendering path is
the per-resource and Composition narratives and omit the tooling link.

#### Justification

The original sentence existed because readers reasonably ask "how do I
display a FHIR Document in a browser?" Simply deleting it leaves that
question unanswered on the page that introduces the Documents
paradigm. Replacing it with guidance that is grounded in the
already-normative narrative model gives implementers a correct and
durable answer that does not depend on any single tool's continued
existence. The trade-off versus Disposition A is that this requires
the workgroup to agree on the replacement wording (and potentially to
identify a maintained tool), which is more work than a straight
deletion.

---

### Disposition C: Decline — No change

#### Proposal

Close the ticket as "Not Persuasive" or "Already addressed elsewhere",
on the grounds that the sentence is incidental, the link is not
normative, and any reader who follows it will quickly discover that
the referenced transform is not present.

#### Justification

This is the weakest option. The reporter — who is also the spec
editor — has explicitly stated the text is incorrect, and there is no
countervailing community signal (no Zulip discussion, no related
tickets, no GitHub PRs) suggesting the sentence is still useful. The
only defensible reason to decline would be if the workgroup believed
a more comprehensive rewrite of the FHIR Documents rendering guidance
was already in flight and would supersede this fix; nothing in the
gathered evidence supports that.

---

### Recommendation

**Recommended disposition:** A, with a follow-on commitment to revisit B.

The cheapest correct action is to remove the demonstrably-false
sentence now (Disposition A). It eliminates the misinformation, can be
made by FHIR-I in a single editorial commit to the FHIR Documents page,
and carries no risk of breaking implementations because the sentence is
purely informative. Disposition B is the better long-term answer, but
it requires the workgroup to either bless a specific replacement tool
or write durable narrative-based guidance, neither of which is
available from the evidence gathered for this ticket. The pragmatic
path is therefore: take Disposition A immediately, and either fold the
"how do I render a FHIR Document?" guidance into a separate ticket or
hand it to the editor as a follow-up. Disposition C should not be
entertained — leaving an acknowledged-incorrect statement in published
spec text is not consistent with FHIR-I's normal editorial standards.
