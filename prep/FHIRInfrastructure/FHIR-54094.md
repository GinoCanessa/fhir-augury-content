## Summary

The reporter (Abderrazek Boufahja) flags ambiguous wording in the Short
display of `Reference.type` in the FHIR R6 ballot 4 publication. The
current text reads "must be a resource in resources", which is unclear:
it is not obvious whether "resources" refers to the `Resource Types`
value set, to resources actually present in some bundle/server, or to
something else. The reporter notes that if the intent really is "must
be a resource from [Resource Types]", then the binding to the
`Resource Types` value set should logically be `required` rather than
`extensible`, which would be unduly restrictive (it would forbid logical
models, custom resources, and out-of-scope resource types in
references). To preserve the extensible binding, the reporter
recommends softening the short text to "should be a resource from
[Resource Types](valueset-resource-types.html)".

There are no cross-referenced Jira tickets and no GitHub
cross-references, so this ticket stands on its own as a small editorial
/ binding-clarity correction against the `Reference` data type
definition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Short display of `Reference.type` in the `Reference`
data type definition (driving `references.html` and
`references-definitions.html#Reference.type`) from:

> "a resource (or, in `logical` models, the URL of the
> definition of a type of resource). If the type is `'Reference'` then
> the application **must be a resource in resources** ..."

(or whatever the current rendered short text is) to:

> "... the resource type **should be a resource from
> [Resource Types](valueset-resource-types.html)** ..."

Concretely: replace "must be a resource in resources" with
"should be a resource from [Resource Types](valueset-resource-types.html)"
in the Short text for `Reference.type` in the source definitions for the
`Reference` data type. Do not change the binding of `Reference.type` —
keep it bound to the `Resource Types` value set with strength
`extensible`. This preserves the option for logical models,
cross-version references, and custom resource types that fall outside
`Resource Types`, while making the textual guidance match the binding
strength.

#### Justification

The reporter's observation is correct: "must" in the short text reads
as a hard constraint, but the formal binding is `extensible`, so the
two are inconsistent. FHIR's normative pattern is that the short
description should align with the binding strength — `required` ⇒
"must / shall", `extensible` ⇒ "should / typically", `preferred` ⇒
"is encouraged to". Softening "must" to "should" brings the prose in
line with the binding without weakening the value set or breaking
implementations that already rely on `extensible` semantics
(logical-model references, XVer extensions, profile-specific resource
types). It is also a minimally-invasive technical correction
appropriate for the ballot stage.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only softening "must" → "should", rewrite the Short
display to describe both halves of `Reference.type` explicitly, so that
the value-set guidance, the URL-of-definition case for `logical`
models, and the absolute-URL case are all mentioned. For example:

> "The expected type of the target of the reference. SHOULD be a code
> from [Resource Types](valueset-resource-types.html); for `logical`
> models, MAY be the canonical URL of a logical model definition."

Keep the binding as `extensible` to `Resource Types`. Optionally add a
sentence to the formal Definition (not just the Short) repeating the
guidance and noting the relationship to `Reference.reference` and to
canonical URLs for logical models, so readers do not have to triangulate
between Short, Definition, and binding strength.

#### Justification

The current Short text is ambiguous beyond just the "must" vs "should"
issue — "a resource in resources" reads as a typo or copy-paste
artefact, and the Short does not mention the `logical`-model URL case
that the existing Definition already supports. Disposition A fixes the
binding-strength mismatch but leaves the Short text awkward and
incomplete. Disposition B addresses the underlying cause of confusion:
the Short tries to compress two distinct cases (resource type code vs.
logical-model canonical URL) into one phrase. This is consistent with
how other value-set-bound coded elements (e.g., `Coding.system`,
`code`-typed elements with `extensible` bindings) phrase their Short
displays. The cost is slightly longer text and a wider editorial change.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive — Editorial / Already Sufficient**: keep
the current Short display and the current `extensible` binding to
`Resource Types`. Rationale: the formal binding is the normative
artefact, and implementers are expected to read binding strength to
determine conformance; the prose "must be a resource in resources" is
informal Short text and is unambiguous in context with the binding.

#### Justification

This disposition is defensible only on the grounds of stability — the
`Reference` type is one of the most widely-implemented FHIR primitives
and any change ripples through tooling, validators, and downstream IGs.
However, the change requested is purely editorial in the Short text
and does not alter cardinality, type, or binding, so the stability
argument is weak. Decline is the least attractive option and is
included primarily to frame the trade-off.

---

### Recommendation

**Recommended disposition:** B

Disposition A correctly identifies and fixes the binding-strength
mismatch the reporter raised, and is the smallest acceptable change.
Disposition B does the same fix and additionally cleans up the
"resource in resources" phrasing and surfaces the `logical`-model URL
case, which is already permitted by the Definition and binding but is
hidden from anyone who reads only the Short. Because `Reference.type`
is read by virtually every implementer and is a frequent source of
"why is this binding extensible if the text says must?" questions, the
small additional editorial cost in B is worth paying once during the
R6 ballot cycle. The change is contained to the `Reference` data type
source definitions, does not touch the binding strength or value set,
and does not require updates outside the FHIR Core specification — so
the implementation footprint is the same as Disposition A.
