## Summary

The ticket reports that the Subscriptions R5 Backport IG's
`SubscriptionStatus` profile defines `notificationEvent.eventNumber` more
narrowly than the R5 base resource. The backport profile currently states the
element is "the sequential number of this event in this subscription
context," which implies a globally unique, monotonically-increasing value
across the subscription. R5 broadened the definition so that `eventNumber`
may also be a *relative* event index within a single notification when the
channel guarantees delivery (e.g., a notification carrying multiple events
numbered 1, 2, 3…), and only needs to be globally monotonic when delivery is
not guaranteed (e.g., REST-Hook). The reporter — prompted by a Zulip
discussion with the IG editor (Gino Canessa) — asks that the Backport IG be
brought into alignment with the R5 definition rather than carrying the older,
stricter constraint. The R5 change traces back to a prior R5 ticket
(FHIR-38920, referenced only in the Zulip thread; not a direct
cross-reference and not present in the local cache) which originally asked
for `eventNumber` to be optional but was resolved by allowing the alternative
"local within notification" interpretation while keeping the element
required.

**Resolution Description (from snapshot):**

Align the IG with the changes made for R5 and R6:

- remove any constraints built around the `eventNumber` element
- clarify that `eventNumber` can be used in a local-only context if
  described by the channel type and supported by the server
- document that channels SHOULD only allow deviation from global and
  monotonic `eventNumber` values if reliable / guaranteed delivery is
  used

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Subscriptions R5 Backport IG so that the
`backport-subscription-status-r4` profile's
`Parameters.parameter:notificationEvent.part:eventNumber.value[x]`
definition matches the R5 `SubscriptionStatus.notificationEvent.eventNumber`
definition. Concretely, in the IG's authoring source for that profile:

1. Replace the short/definition text with the R5 wording: "Either the
   sequential number of this event in this subscription context or a
   relative event number for this notification."
2. Add the R5 comment explaining the two interpretations: globally unique
   and monotonically-increasing where delivery is not guaranteed (e.g.,
   REST-Hook), and a per-notification relative index where delivery is
   guaranteed.
3. Remove any backport-specific constraint (invariant or differential
   binding/cardinality tightening) that forces the value to be globally
   sequential across the subscription. `eventNumber` remains required (per
   the FHIR-38920 resolution); only the *meaning* is relaxed.
4. Add narrative in the IG's SubscriptionStatus / Notifications page noting
   that channels SHOULD only allow deviation from global/monotonic values
   when they offer reliable / guaranteed delivery, mirroring the R5/R6
   guidance.

#### Justification

The reporter, the IG editor (Gino Canessa), and the prior R5 resolution
(FHIR-38920) all agree the backport should carry the same semantics as
R5/R6; the divergence is an oversight, not a deliberate stricter
backport. Aligning prevents implementer confusion when reading the
backport profile against the R5 base and avoids forcing R4 implementers
to track a more burdensome eventNumber model than R5 implementers. The
ticket has already been auto-resolved as **Persuasive** with a
resolution description that lays out exactly these three changes,
indicating Infrastructure WG has effectively pre-endorsed this path.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the R5 wording for `eventNumber` in the backport, **and**
additionally publish a short cross-version conformance note (in the IG's
"Notes for R5 backport implementers" page, or equivalent) explicitly
calling out:

- That R4 servers backporting R5 subscriptions MAY interpret `eventNumber`
  in either mode, but SHOULD declare which mode they use via the
  `channelType`-specific extension or a CapabilityStatement note.
- That clients consuming both R4-backport and R5-native notifications
  cannot assume globally-monotonic numbering across all sources.

The element-level definition change is identical to Disposition A; the
addition is implementer guidance to manage the ambiguity that the looser
definition introduces.

#### Justification

Disposition A faithfully copies R5 wording but doesn't address a real
operational concern that is more acute in the backport context: many
existing R4 backport implementations were almost certainly written
against the *strict* "globally sequential" interpretation, so simply
loosening the definition silently changes what conforming clients can
expect from existing servers. A dedicated implementer note preserves
backward compatibility for clients while still aligning the formal
definition with R5. Trade-off: more authoring work and one more page to
maintain; benefit: fewer surprises during the next backport publication
cycle.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — backport intentionally retains the stricter
R4-era semantics." Keep the current backport profile wording requiring a
sequential, subscription-scoped event number, and document in the IG that
this is a deliberate divergence from R5 to preserve deterministic event
ordering for R4 clients.

#### Justification

The only defensible reason to decline would be that the backport
audience (R4 implementers, often legacy) benefits from a single,
unambiguous interpretation and that loosening the constraint creates a
silent behavioral change for already-deployed servers and clients. This
is a real concern in principle. However, neither the Zulip discussion
nor the ticket comments surface any party arguing for this position;
the IG editor explicitly asked for the change to be filed, and the
ticket has already been resolved as Persuasive. Recording this option
mainly so the WG can confirm there is no implementer constituency that
would be harmed by the alignment.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as **Persuasive**, the IG editor on
the Zulip thread agreed that the backport divergence was an oversight
rather than a design choice, and the resolution description on the
ticket itself enumerates exactly the changes Disposition A proposes.
Disposition B's implementer-note addition has merit and could be folded
in as a friendly amendment if the WG wants to soften the impact on
existing R4 backport deployments, but it should not block the core
alignment. Disposition C has no advocate in the captured discussion and
would leave the backport profile permanently out of step with R5/R6 for
no documented benefit.
