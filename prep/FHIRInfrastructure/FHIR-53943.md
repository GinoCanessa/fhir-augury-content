## Summary

The `$validate` operation (defined on `Resource` and inherited/specialised across the FHIR Core resource set) declares a `graph` input parameter typed as `canonical(GraphDefinition)`. With GraphDefinition having been removed from the FHIR Core specification and relocated to an incubator, the `$validate` operation now holds a canonical reference to a resource type that is no longer part of Core. Brian Postlethwaite is asking the workgroup to decide whether the `graph` parameter should be removed entirely from `$validate`, retyped (or re‑pointed at the incubator artifact), or otherwise adjusted so that Core no longer normatively depends on a resource that lives outside it. As a secondary point, the reporter also raises a rendering question: whether the IG publisher's rendering of a canonical reference should additionally hyperlink the referenced *resource type* (so a reader of `canonical(GraphDefinition)` can click through to the GraphDefinition page even when that page is no longer in the immediate spec). No related Jira tickets, GitHub items, or Zulip discussions surfaced via cross-reference or keyword search at the time of preparation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the `graph` input parameter from the `Resource-validate` OperationDefinition (`$validate`) in FHIR Core. Concretely:

- Edit the `$validate` OperationDefinition source so that the `graph` parameter (currently typed `canonical` with `targetProfile` referencing `GraphDefinition`) is dropped from the `parameter` list.
- Add a release-note / change-note entry under the operation page noting that `graph` has been removed because GraphDefinition is no longer part of Core, and that any equivalent functionality is now defined by the GraphDefinition incubator.
- Update the operation's narrative / human description on the operation page to remove references to GraphDefinition-driven validation.
- Independently, address the rendering side-question by filing (or referencing) a publisher/IG-tooling improvement so that `canonical(<ResourceType>)` rendering hyperlinks the resource type as well as the canonical value, but treat that as a separate item rather than blocking this ticket.

#### Justification

FHIR Core OperationDefinitions should not contain normative parameters whose `targetProfile` points at a resource that is not part of Core. Leaving `graph: canonical(GraphDefinition)` in `$validate` creates a dangling normative dependency: implementers of Core `$validate` are nominally expected to honour a parameter whose contract is defined in an artifact that has been pulled out of the specification. Removing the parameter is the cleanest way to keep `$validate` self-contained in Core and avoids implying that Core servers must understand an incubator artifact. If/when GraphDefinition stabilises and re-enters Core (or a successor spec), the parameter can be re-introduced — additive changes to operations are well precedented and lower-risk than carrying a broken reference forward.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the `graph` parameter on `$validate` but decouple it from the now-removed Core GraphDefinition:

- Change the parameter's type from `canonical` (with `targetProfile = GraphDefinition`) to a plain `canonical` with **no** `targetProfile`, so it remains a generic canonical URL whose semantics are deferred to the validating server / referenced artifact.
- Update the parameter's documentation on the `$validate` operation page to explain that the canonical is expected to identify a graph-validation artifact (such as the GraphDefinition resource defined by the GraphDefinition incubator), and link out to the incubator IG.
- Mark the parameter as `Trial Use` (if not already) and add a publisher cross-version note so consumers know the `targetProfile` was relaxed in this release.
- Pair this with the rendering improvement called out by the reporter: extend OperationDefinition rendering so canonical-typed parameters render both the value link and a hyperlink to the resource type, mitigating the loss of the explicit `targetProfile` link.

#### Justification

This preserves the existing `$validate` API surface — implementers who already pass a `graph` canonical do not have to change their requests, and existing fixtures, tests, and tooling continue to round-trip. It also avoids a normative deletion in a widely-implemented operation, which is a higher-impact change than typically warranted by a "the target moved out of Core" cleanup. By relaxing `targetProfile` rather than removing the parameter, Core stops asserting that the target resource type is defined here, while still telling implementers what the parameter is for via narrative + a link to the incubator. The companion rendering change directly answers the reporter's parenthetical question and is generally useful for any canonical reference that points at a resource type not rendered alongside the operation.

---

### Disposition C: Decline

#### Proposal

Take no normative change to the `$validate` OperationDefinition. Instead:

- Leave the `graph` parameter and its `canonical(GraphDefinition)` typing as-is.
- Update only the surrounding narrative on the `$validate` operation page to note that GraphDefinition is currently published from the GraphDefinition incubator rather than from Core, with a link to the incubator.
- Optionally file a separate, narrowly-scoped publisher ticket for the canonical-rendering hyperlink request so it can be triaged independently of `$validate`.

#### Justification

Removing or retyping a parameter on `$validate` is a breaking change to one of the most widely implemented operations in FHIR. The reporter has not identified a concrete implementer harm — the question is framed as "should this be changed?" rather than "this is breaking implementations." Until there is evidence that the dangling `canonical(GraphDefinition)` is causing validators, code generators, or conformance tooling to fail, a documentation-only clarification is the lowest-risk response and preserves backwards compatibility with the substantial body of existing `$validate` clients/servers. A normative change can be revisited once the GraphDefinition incubator stabilises and its long-term home is known.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of correctness and stability. Disposition A is technically the cleanest, but silently removing a parameter from `$validate` — an operation that virtually every conformant FHIR server implements — is a notable break for a problem that is, today, primarily a spec-hygiene issue rather than an interoperability failure. Disposition C is the smallest change but leaves Core asserting (via `targetProfile`) that GraphDefinition is a Core resource, which is exactly the inconsistency the reporter is flagging. Disposition B fixes the actual defect (Core no longer referencing a non-Core resource via `targetProfile`) without removing the parameter, keeps existing clients working, and gives the workgroup a natural place to point implementers at the GraphDefinition incubator. It also lets the workgroup absorb the reporter's secondary rendering request as a paired publisher improvement, rather than treating it as an afterthought. If FHIR Infrastructure decides that the `$validate` `graph` parameter has no real-world usage and that a clean removal is preferable, falling back to Disposition A is reasonable — but that decision should be made on usage evidence, not on the canonical-target question alone.
