## Summary

Chris Moesel requests that FHIR Shorthand (FSH) provide a dedicated syntax for (a) adding a new item at the end of an array when the current array length is not known, and (b) referring to the last item in an existing array. The motivating use case is mixins and other composable constructs that need to append to arrays without knowing the resulting index, because the index depends on where the mixin is applied. The ticket illustrates the need with an example using `name[+].given` / `name[+].given` to add two successive `name` entries and suggests one possible syntax (redefining the omitted index to mean "last") as a starting point. The reporter subsequently clarified that any reasonable syntax is acceptable — `[+]` was only an example. The ticket was briefly resolved as "Considered for Future Use" in August 2020 (Kramer/Rhodes 13-0-0), then re-opened after STU1 publication to track the problem going forward. There are no cross-referenced Jira tickets, but a later Zulip thread in `#shorthand` (June 2022) shows that SUSHI has since implemented soft indexing (`[+]` to increment the last referenced index, `[=]` to re-use it), which partially addresses the ticket but surfaces unresolved interactions between `[+]` and named slices.

## Details

**Description:**

> Consider a syntax to allow authors to create a new item at the end of an array and/or reference the last item in an array. (Both are probably needed for a complete solution). This would be useful when the array length is not known (or may change). For example, mixins the add array elements but might not know the index to use (because it depends on where/how the mixin is applied).
>
> Perhaps redefining the omitted index to mean the **last** item in the array would make this more feasible. Then you could do something like:
>
> - `name[+].given = "John"`
> - `name.family = "Doe"`
> - `name[+].given = "Jean"`
> - `name.family = "Does"`

**Comments:**

- **Chris Moesel (2020-06-05):** Clarifies that he is fine with any reasonable syntax — `[+]` was just an example to demonstrate the point. A better syntax would be welcome.
- **Lynn Laakso (2020-10-12):** Reverted the prior "Considered for Future Use" resolution (Kramer/Rhodes 13-0-0, 2020-08-03). The original rationale was that the grammar doesn't work well for populating arrays because the index has to be numbered — inserting between elements 3 and 4 requires renumbering everything. After internal discussion and considerable Zulip input, the path forward still wasn't clear, so the issue was deferred and then re-opened for attention after STU1 publication.

## Related Zulip Discussions

### [shorthand > slice names -- plus](https://chat.fhir.org/#narrow/stream/shorthand/topic/slice%20names%20--%20plus/near/284622859)

Discussion from 2022-06-01 in which Chris Moesel confirms the current FSH spec wording — "`[+]` is used to increment the last referenced index by 1, and `[=]` is used to reference the same index that was last referenced; when an array is empty, `[+]` refers to the first element `[0]`" — and explains how this interacts (poorly) with slices. Mint Thompson had implemented a branch making `[+]` always append to the end, but it broke too many existing IGs that relied on the old behavior of `[+]` starting at `[0]`, so the fix was not merged; instead warnings were introduced when `[+]` overwrites an existing slice entry. John Moehrke argues the soft-break (warnings) just prolongs the problem and those relying on undocumented hacks should not be prioritized over authors who followed the spec; Moesel counters that many failures from changing the default are silent (wrong instances exported, not build errors), which is why the transition has to be cautious. Jose Costa Teixeira proposes either a new `[++]` syntax or a temporary config property to opt into the corrected behavior, with a future flip of the default. No consensus was reached; the thread ends with open questions about tracking which SUSHI version built an IG and how to manage the transition.

Participants: Chris Moesel, John Moehrke, Elliot Silver, Jose Costa Teixeira, Gino Canessa.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt dedicated FSH syntax that directly addresses both parts of the original request, building on the soft-indexing tokens that have since been added to the FSH grammar:

1. **Append-to-end**: Formally define `[+]` as "append a new element to the end of the array" whenever no prior index has been referenced in the current Instance/Profile rule block, rather than the current "starts at `[0]` if nothing has been referenced yet" behavior. For empty arrays this still means `[0]`; for non-empty arrays it means `[length]`.
2. **Reference-the-last**: Add a new token (e.g., `[$]` or `[last]`) in the FSH grammar's path indexing production that resolves to the highest existing index of the array at the point the rule is evaluated. This is distinct from `[=]`, which re-uses the most recently *referenced* index, not the actual last element.
3. Update the FSH specification (Shorthand `reference.html` path-and-index section and the Instance/slicing examples) to document both tokens, their interaction with named slices (named slices resolve first; `[+]` appends after them; required slices inserted by the tool go at the end when not referenced in FSH), and the migration/config story.

#### Justification

This is the only option that fully addresses both halves of the reporter's request ("add at end" *and* "reference the last"). The Zulip thread shows that the community has already converged on "`[+]` should append to the end" as the natural semantics; the only reason SUSHI has not switched is backward compatibility of IGs that exploited the old behavior. Providing a distinct new token for "last existing index" (rather than overloading `[=]`) avoids the ambiguity Moesel flagged in Zulip between "last referenced in FSH" and "last in the serialized array". A specification-level change in FSH is the correct locus because this is a grammar/semantics question, not a tool-specific one — the spec must dictate what SUSHI (and any future FSH processor) does.

---

### Disposition B: Alternative Approach

#### Proposal

Close FHIR-27282 as **substantially addressed by the existing `[+]` / `[=]` soft-indexing feature** added to the FSH specification since this ticket was filed, and open a narrower successor ticket covering only the two residual gaps:

1. Clarify in the Shorthand spec what `[+]` means on a non-empty array when *no* prior index has been referenced in the current rule block — the current wording ("refers to the first element `[0]`" is only stated for empty arrays) is silent on this case.
2. Specify the interaction between `[+]` and named slices (ordering of slice entries vs. `[+]`-appended entries, behavior of `[+]` after a `[slicename]` reference), with worked examples.

No new tokens are introduced. The "reference the last element" half of the original request is treated as satisfied by `[=]` for the common case of "the one I just added", which covers the mixin use case that motivated the ticket.

#### Justification

The soft-indexing feature that has been added since 2020 already covers the primary mixin use case (`[+]` to add, `[=]` to keep writing to the one you just added). Adding more tokens to the grammar increases learning cost and implementation burden for marginal benefit, and Moesel himself said in Zulip he'd "prefer that we don't introduce another syntax like `[++]`". The real pain points surfaced in the `slice names -- plus` thread are specification gaps and interaction rules, not missing tokens. This disposition preserves backward compatibility and keeps the grammar small while still fixing the concrete issues authors are hitting today.

---

### Disposition C: Decline

#### Proposal

Close FHIR-27282 as **Persuasive with Modification — already addressed**, on the grounds that `[+]` and `[=]` soft indexing, added to the FSH specification after this ticket was filed, are the community's agreed answer to the mixin-append use case. The "reference the last element" half is considered out of scope for FSH because FSH is an authoring language, not a query language; reading/pointing-to arbitrary elements of an existing array is not something FSH needs to do, since FSH rules always define what gets written.

#### Justification

FSH's mandate is to describe conformance resources and instances, not to traverse or query them. A `[last]`-style reader token implies semantics that don't fit the "define-only" model of FSH and would invite scope creep (why not `[first]`? `[where(x=y)]`?). The append-to-end half of the request is operationally handled by `[+]`/`[=]`, and the remaining friction documented in Zulip is about *transition policy* (when to change SUSHI's default, whether to add a config flag) rather than a missing spec feature. The ticket has been open and unactionable since 2020 because there is no clean way to change the default without breaking existing IGs; declining acknowledges that the spec itself is fine and the problem is one of tool migration, which is out of scope for a Shorthand spec change request.

---

### Recommendation

**Recommended disposition:** B

The soft-indexing tokens added since 2020 already solve the concrete mixin-append use case that motivated the ticket, and the reporter has been publicly relaxed about the exact syntax. The genuinely unresolved items — surfaced in the June 2022 `shorthand > slice names -- plus` thread — are the undefined behavior of `[+]` on a non-empty array when no index has been referenced, and the messy interaction between `[+]` and named slices (where SUSHI's current "slices first" ordering silently overwrites `[+]`-added entries). Those are spec-clarity problems, not missing-feature problems. Disposition A would add a second token for "reference the last element" that neither the reporter nor the Zulip participants are asking for, and Disposition C understates the real remaining work. Closing this ticket with a pointer to a focused successor that tightens the `[+]`-plus-slices specification gives the workgroup a tractable next step without re-opening the grammar.
