## Summary

Brian Postlethwaite is requesting a documentation-quality pass over the FHIR
Core spec's *Additional Functions* (the FHIRPath-in-FHIR functions and
"%-prefixed" environment trees) so that **every function/feature has worked
examples, every example states the expected result (including its type when
not obvious), and every function/parameter explicitly declares its return and
argument types/cardinality** — using either FHIR types or FHIRPath system
types as appropriate.

The reporter calls out a non-exhaustive first-pass list of items that need
this treatment: `elementDefinition`, `checkModifiers`, `subsumedBy`,
`htmlChecks`, `weight`, the entire `%terminology.*` tree, any remaining
`%factory.*` items, any remaining `%server.*` items, plus `hasValue` and
`toString` (the latter specifically needs more examples and a description of
behavior in the presence of extensions). He notes there are likely others;
this ticket is a global "make the spec internally complete and typed" change
request rather than a fix for any single function.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-53951 as a tracking/umbrella documentation ticket against the
FHIR Core *Additional Functions* page (`fhirpath.html` and any companion
`fhirpath-*` files) and commit to a documentation pass with two concrete
deliverables:

1. **Examples + expected results.** For each of the explicitly-named items —
   `elementDefinition`, `checkModifiers`, `subsumedBy`, `htmlChecks`,
   `weight`, every `%terminology.*` entry, every remaining `%factory.*`
   entry (subject to the outcome of FHIR-54078), every remaining `%server.*`
   entry (subject to FHIR-53953), `hasValue`, and `toString` — add at least
   one worked example and a description of the expected result, including
   the result's type when it is not obvious from context. `toString` must
   include an example showing behaviour when the source has only
   extensions/id (no value), per FHIR-53959.
2. **Explicit signatures.** For every function and `%`-tree property in the
   *Additional Functions* section, declare the return type and each
   argument's type and cardinality, using either the FHIR datatype name
   (`FHIR.ElementDefinition`, `FHIR.code`, …) or the FHIRPath system type
   name (`System.Boolean`, `System.String`, `System.Integer`, …) as
   appropriate, and use the same convention consistently across the page.

Sweep the page to identify other functions/features missing one of these
two things and add them to the same pass. Co-ordinate with FHIR-53960 so
that the typed signatures introduced here also pin down whether each
function returns empty or throws on bad input.

#### Justification

The request is a documentation-quality and typing fix on a normative-track
section. FHIR Core already documents the bulk of FHIRPath functions with
worked examples and explicit signatures; the items called out are the gaps,
not the rule, so applying the same pattern is consistent with the
surrounding spec rather than novel. Explicit return/argument typing is a
prerequisite for implementers (engines, validators, IDE tooling) to
correctly statically analyse FHIRPath expressions that use these functions,
and the FHIRPath spec itself uses an explicit type-and-cardinality
signature convention — aligning with that convention is the path of least
surprise. The named functions also overlap heavily with the cluster of
related ballot comments (FHIR-53959, FHIR-53960, FHIR-53961, FHIR-53953,
FHIR-54069, FHIR-54484); accepting FHIR-53951 as the umbrella keeps the
reporter's review pass cohesive instead of fragmenting it.

---

### Disposition B: Alternative Approach

#### Proposal

Decompose FHIR-53951 into per-function child tickets (or align it with the
already-filed sibling tickets) and adopt this ticket as the parent
"Additional Functions documentation hardening" tracker only. Concretely:

- Fold the `elementDefinition()` work into FHIR-53961 and close the
  `elementDefinition` bullet here as a duplicate.
- Fold `toString` / `hasValue` examples and the empty-collection-on-
  extensions-only behaviour into FHIR-53959.
- Make the `%terminology.*` typing/example work a follow-on to FHIR-54069
  (apply once the parameter/encoding refinements land).
- Make the `%factory.*` work conditional on FHIR-54078 (skip if `%factory`
  moves to an incubator IG).
- Make the `%server.*` work a follow-on to FHIR-53953 (so per-function
  docs reference the new "Handling Remote Data" section).
- File new, narrow tickets only for `checkModifiers`, `subsumedBy`,
  `htmlChecks`, and `weight` (the items not already covered elsewhere).

Adopt a single editorial convention for return/argument types and
cardinality across the page, but apply it via the per-function tickets
rather than as a single sweep here.

#### Justification

The literal request — "all functions/features should have examples and
explicit types" — is broad enough that it is hard to know when this
ticket is "done", which makes it a poor unit of work for a single
ballot-comment disposition and a poor unit of review for the work group.
Most of the named items already have a more specific sibling ticket from
the same reporter; doing the work there keeps each disposition reviewable
and lets each function's fate (e.g. `%factory` possibly moving to an
incubator) drive its own documentation effort. Treating FHIR-53951 as a
parent/tracker preserves the reporter's intent without doubling the
editorial work or risking divergent fixes between this ticket and its
siblings.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-53951 as filed on the grounds that it is **not actionable as
a single change request** — it asks for an open-ended editorial sweep over
an unbounded set of functions/features without nominating specific text
changes — and direct the reporter to file (or rely on) per-function
tickets for the items that genuinely need work, several of which already
exist (FHIR-53959, FHIR-53960, FHIR-53961, FHIR-53953, FHIR-54069,
FHIR-54078, FHIR-54484).

#### Justification

FHIR Jira convention is that ballot comments propose a concrete change
the work group can vote on. FHIR-53951 explicitly says "These are just
the ones I came across in the first pass, I'm sure there are likely
others" — i.e. the scope is acknowledged to be incomplete and unbounded.
The reporter has already filed narrowly-scoped tickets covering most of
the named items, so declining FHIR-53951 does not lose any of the
underlying asks; it only avoids carrying an open-ended tracking ticket
that would either duplicate the sibling work or be impossible to mark
"resolved" against any specific spec change.

---

### Recommendation

**Recommended disposition:** B

This is a real and reasonable ask — examples and explicit type/cardinality
signatures are baseline expectations for a normative section, and the items
the reporter names genuinely are gaps — so an outright decline (C) would
ignore good feedback and force the reporter to re-file work they have
already substantially scoped. At the same time, accepting FHIR-53951 as
filed (A) commits to a single-ticket sweep that overlaps with at least six
sibling tickets from the same reporter, several of which (FHIR-54078 on
`%factory`, FHIR-53953 on `%server.*`, FHIR-54069 on `%terminology.*`)
will materially change the surface area that needs documenting. Adopting
FHIR-53951 as the parent/tracker (B), routing each function to its
existing sibling ticket, and filing narrow new tickets only for the
genuinely-uncovered items (`checkModifiers`, `subsumedBy`, `htmlChecks`,
`weight`) gives the work group reviewable units, lets each function's
disposition follow its own structural decision, and still delivers exactly
the editorial outcome the reporter asked for.
