## Summary

FHIR-52912 is feedback from Patient Administration (PA) to FHIR Infrastructure
(FHIR-I) on the "Operations for Large Resources" page. PA wants to leverage the
`$filter` / `$add` / `$remove` operations defined there to manage repeating
elements that can grow unbounded — most importantly `Encounter.participant`
(and `Encounter.location`, `Account.coverage`, deeply nested
`CodeSystem.concept.concept.concept`, `Questionnaire.item.item.item`) — but it
identifies several functional gaps that block adoption, plus a set of
copy-paste typos in the page's "remove" examples.

The ticket originates from the PA-owned [FHIR-37648](https://jira.hl7.org/browse/FHIR-37648)
("Better approach for large lists of participants in Encounter.participants",
resolution **Persuasive**), which over multiple WGM cycles narrowed the
solution space toward "let the server detect the threshold, return a subset
with a flag, and allow the client to opt in to the full content". When PA
tried to map that resolution onto the existing Operations for Large Resources
spec they found three open questions (how to *detect* that a resource was
truncated, how to *demand* the full content anyway, and how to *update* the
non-list parts of the resource). The reporter created FHIR-52912 to ask FHIR-I
to extend the spec accordingly, and listed concrete proposals: a specific
`OperationOutcome` issue type (e.g. `too-long` / `too-costly`) and/or
auto-subsetting via the existing `SUBSETTED` meta tag plus a marker extension;
adding `property` and date-range parameters to `$filter`; allowing PUT on a
subset resource provided the missing backbone is flagged; and considering an
`$update` item operation. The ticket also lists copy-paste defects in the
`$remove` example block ("remove two items to a Group" header text is "add two
items to a List").

The cross-referenced Zulip threads show the question was carried into FHIR-I
(`fhir/infrastructure-wg > "Large Resources" - FHIR-52912`) and into the
broader implementer community (`implementers > Dealing with Large resources`).
FHIR-I's response — primarily from Grahame Grieve, with input from Gino
Canessa — was that the Operations for Large Resources work is moving into the
**API Incubator** project for further iteration, that today's answers are
"`HEAD` then `_elements`/`_summary` to discover/avoid the large list, standard
`GET` to demand the full content, standard `PATCH` for non-list updates", and
that no new core machinery is being added on the R6 timeline. PA pushed back
that this is impractical (clients don't `HEAD` first today; for facade servers
a `HEAD` is nearly as expensive as the full read; `PATCH` of a list element
requires knowing its index, which requires the full resource first). Grahame's
closing position on the implementers thread was that "dealing with these
issues is in the incubator space".

## Details

**Description:**

> While reviewing FHIR-37648…
>
> PA would like to use this functionality however we are still missing:
>
> - How to detect the large list in the first place — specific exception?
> - How to request the complete list anyway
> - Update the core properties outside the list
>
> Some examples of properties we expect may get large are:
>
> - Encounter.participants
> - Encounter.location
> - Account.coverage
>
> There are two options we could consider for allowing a client to detect
> large resources (or both/either options at server discretion):
>
> **a) Error/exception** — should be a specific error code (HTTP status?,
> and OperationOutcome issue type — `too-long` or `too-costly`?). The
> error/exception approach has implications on searching, as if one
> resource is too large that may need to abort the entire search *(an
> option, though not preferred)*.
>
> **b) Auto-subset** — if the server decides the resource is too large to
> return complete, then the server should return the resource with the
> `SUBSETTED` meta tag (and maybe another meta tag to indicate this
> specific type of subsetting?), and exclude ANY participants or
> locations; the client would then use `$filter` to request the content.
>   - How can the client know which property was "too large" — an
>     extension?
>
> Either of these still don't provide a way for the client to *get* the
> large resource. Recommend adding additional fields to the `$filter`
> operation to support these:
>   - **property** (e.g. `Encounter.participant`)
>   - **filter** — start/end date
>   - Extension indicating which properties have been filtered
>   - Path syntax such as `"Encounter.participant"` (at worst include an
>     indexer), `"CodeSystem.concept.concept.concept"`,
>     `"Questionnaire.item.item.item"`
>
> The remaining issue is how to update the large resource excluding the
> large set — is patch the only option (we thought that required the
> complete retrieval)? Maybe permit update with subset, provided the
> extension for the "missing" backbone is indicating it, and then require
> `$add` / `$remove` to update the backbone. `Group` currently has no way
> to update the core elements, or is that "use `$add` with nothing to do
> it"? Should there be an update-item option too *(or do we need to
> remove, then add back in)*?
>
> **Also noted these typos:** Remove sections have copy-paste issues
> from add. Heading "remove two items to a Group" — example invocation
> says "add two items to a List" (in the remove example).

## Linked Jira Tickets

### [FHIR-37648: Better approach for large lists of participants in Encounter.participants](https://jira.hl7.org/browse/FHIR-37648)

PA-owned, **Persuasive**-resolved ticket from 2022 that motivated FHIR-52912.
PA observed that long-stay encounters can produce very large
`Encounter.participant` lists dominated by historical participants, and worked
through several solution options (CareTeam re-anchoring, EncounterHistory,
`_summary=active` / `_activeAt`, custom operations, extracting participants to
a separate resource). The Jan 2024 PA WGM resolution was to adopt the new
Operations for Large Resources page once the listed gaps are addressed —
which is exactly what FHIR-52912 asks FHIR-I to do.

Users involved: Cooper Thompson (reporter, frequent commenter), Brian
Postlethwaite (commenter, opened FHIR-52912 as the FHIR-I-facing follow-up).

## Related Zulip Discussions

### [fhir/infrastructure-wg > "Large Resources" - FHIR-52912](https://chat.fhir.org/#narrow/stream/fhir%2Finfrastructure-wg/topic/%22Large%20Resources%22%20-%20FHIR-52912/near/544608474)

Cooper Thompson opened the topic on 2025-10-13 explicitly asking whether
FHIR-I plans to address FHIR-52912 ahead of normative. **Gino Canessa**
responded that the Operations for Large Resources work is moving into the
**API Incubator** project so it can continue to iterate. **Grahame Grieve**
answered the three PA gap questions with the existing toolbox: detection via
`HEAD` plus `_elements`/`_summary` (with the caveat that nothing tells the
client a list is "too large"); full retrieval is just a standard `GET`;
non-list updates are a standard `PATCH`. **Cooper** and **Brian** pushed
back: `_elements=participant` still returns a huge payload when the list has
10k+ entries; `<1%` of Encounters are oversize so requiring a `HEAD` round
trip on every read is a behaviour change clients haven't budgeted for and is
nearly as expensive as a full read on a facade server; and `PATCH` to a list
element requires knowing its index, which forces the client to retrieve the
full resource first. Grahame's closing stance was that the shift in client
behaviour is "driven by the problem, not [my] answer" and that clients can
keep getting big resources today. **No consensus reached, no FHIR-I
commitment to spec changes on the R6 timeline.**

Participants: Cooper Thompson, Gino Canessa, Grahame Grieve, Brian
Postlethwaite.

### [implementers > Dealing with Large resources](https://chat.fhir.org/#narrow/stream/implementers/topic/Dealing%20with%20Large%20resources/near/562791000)

Broader community thread (Nov–Dec 2025) on the same problem space.
**Aleksandr Kislitsyn** framed the cross-cutting impact: large resources
break HTTP CRUD without streaming, slow down validation, surface
unexpectedly in search results, and bloat subscription notifications;
solutions today are ad-hoc per resource type (id-only subscriptions for
some, streaming CRUD for big CodeSystems, etc.), and asked for a
generalised "this instance is large — handle differently" signal. **John
Moehrke** noted that vocabulary already addresses much of this with
operations rather than CRUD. **Gino Canessa** floated an unwritten Cascais
proposal for an NDJSON streaming wire-format: first row is the resource
minus its big arrays, subsequent rows are array elements. **Grahame
Grieve** pushed back on NDJSON ("not comfortable… because of nesting"),
pointed at the existing Operations for Large Resources page, and ultimately
concluded that "dealing with these issues is in the incubator space".
**Jim Steel** described his system's pragmatic answer: default search to
`_summary` (which strips `CodeSystem.concept`), and invest in faster
validation (FHIR Schema). **Guillermo Rodríguez** suggested a `meta.tag`
to indicate a redacted/summary form, modelled on the v3 ObservationValue
`REDACTED` tag (but noted that code is too security-specific). **Michael
Lawley** suggested HTTP `Range` headers (e.g. `Range:
CodeSystem.concept=0-1000`) plus `PATCH` for updates, with content
compression. **Cooper Thompson** linked back to the PA tickets, noting no
general solution yet. **Open consensus:** the underlying need is real and
broadly felt, but FHIR-I's R6-timeline answer is "use what we have; the
rest belongs in the API Incubator".

Participants: Aleksandr Kislitsyn, John Moehrke, Gino Canessa, Jim Steel,
Guillermo Rodríguez, Grahame Grieve, Michael Lawley, Cooper Thompson.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-52912 as two separable changes against the **Operations for
Large Resources** page (`source/operations-for-large-resources.xml` /
`.html` in `HL7/fhir`):

1. **Editorial / typo fix (uncontroversial):** correct the `$remove`
   example block. The heading "remove two items to a Group" currently
   carries an example invocation captioned "add two items to a List";
   replace with a Group-rooted `$remove` example that mirrors the
   structure of the corresponding `$add` example.
2. **Functional extensions to the operations + Bundle/meta machinery:**
   a. Add a defined detection path: a new `OperationOutcome` issue-type
      code (`too-costly`, with documented mapping to HTTP `413
      Content Too Large`) that a server MUST return when it refuses to
      materialise the full resource, plus an alternative auto-subset
      path that returns the resource with the existing `SUBSETTED` meta
      tag and a new meta tag (or marker extension on the truncated
      element) identifying which repeating element(s) were dropped.
   b. Extend the `$filter` operation Parameters with `property` (a
      FHIRPath / element path with optional indexers, e.g.
      `Encounter.participant`, `CodeSystem.concept.concept.concept`,
      `Questionnaire.item.item.item`) and `period` (`start` / `end`),
      and document a deterministic period-overlap matching rule.
   c. Allow `PUT` of a subset instance — provided the server-emitted
      "missing backbone" extension is preserved on the way back —
      treating untouched truncated elements as "unchanged", and require
      `$add` / `$remove` for actual list mutation. Document that this
      path SHOULD NOT use a version-specific update (since the client
      doesn't have the full version).
   d. Add an `$update-item` operation (or formally define
      `$remove`+`$add` as the supported idiom and call that out
      prominently) so callers can mutate a single existing list entry
      without the full-resource fetch + index discovery dance.

#### Justification

The PA use case is concrete (`Encounter.participant` for long-stay
admissions) and the ticket is the explicit outcome of the FHIR-37648
**Persuasive** resolution. The pieces are all additive: a new
issue-type code, optional `$filter` parameters, an additional allowed
shape for `PUT`, and a new operation. The typo fix is unambiguous and
should land regardless of the rest. The functional extensions also align
with the wider implementer thread's signal that a generalised "this
instance is large; here's how to deal with it" pattern is missing
today.

---

### Disposition B: Alternative Approach

#### Proposal

**Split the ticket and route the functional gaps through the API
Incubator (per the FHIR-I Zulip outcome), while landing the editorial
fix and a clarifying note in R6 core.** Concretely:

1. Land the typo fix in `source/operations-for-large-resources.xml`
   immediately as a pure editorial change.
2. Add a non-normative paragraph to that page that:
   - Acknowledges the three gaps PA raised (detection, full retrieval,
     non-list update).
   - Documents the *current* idiomatic answers from the FHIR-I Zulip
     thread (`HEAD` + `_elements` / `_summary` to discover and avoid;
     plain `GET` to demand the full content; `PATCH` for fields outside
     the large list) with their stated limitations.
   - Points at the API Incubator as the venue for evolving the richer
     mechanism (issue-type code, `$filter` extensions, subset-PUT,
     `$update-item`).
3. File the functional extensions as separate Incubator work items
   carrying FHIR-52912 as the source of requirements, so that the work
   is visible and tracked but not gated on the R6 ballot timeline.

#### Justification

This matches what FHIR-I actually said on Zulip (Gino: moving to API
Incubator; Grahame: "dealing with these issues is in the incubator
space") and what Cooper effectively offered as the fallback ("R6 doesn't
handle large resources well — figure it out on the implementation
side"). It still gives PA something concrete to cite (the typo fix plus
a guidance paragraph), unblocks the page from sitting with known
defects, and avoids designing four new pieces of machinery under R6
ballot pressure. Trade-off vs. A: PA does not get the elements of the
solution they're asking for in R6, so `Encounter` going normative still
inherits the open question; that risk should be weighed by FHIR-I and
PA jointly — e.g. via the dedicated joint call Brian flagged in
FHIR-37648.

---

### Disposition C: Decline

#### Proposal

Apply the typo fix as a trivial editorial correction, but decline the
functional extensions on the grounds that (i) the existing toolbox
(`HEAD`, `_summary`, `_elements`, `GET`, `PATCH`, `$filter`/`$add`/`$remove`
as already specified) is sufficient for the documented use case, and
(ii) generalised "large-resource" mechanics are explicitly out of
scope for the page as currently chartered and belong in the API
Incubator. Mark the ticket **Persuasive with Modification** for the
typo and **Not Persuasive** for the functional asks, with rationale
pointing to the FHIR-I Zulip discussion and the Incubator pathway.

#### Justification

Defensible on three grounds. (1) Adding an issue-type code, four new
`$filter` parameters, a subset-`PUT` semantics, and a new operation in
one ticket is a substantial scope expansion for what is currently a
narrowly scoped operations page. (2) Several PA-listed gaps already
have answers per Grahame's Zulip reply, even if those answers are
ergonomically unattractive. (3) The implementer-thread consensus is
that a *generalised* large-resource story is genuinely complex (compare
NDJSON streaming, `Range` headers, `meta.tag` redaction, FHIR Schema
validation effort) and is poorly suited to a single ticket on a
specific operations page; punting to the Incubator is the honest answer.
The cost is clear: PA stays blocked on FHIR-37648 for R6, and the page
ships with known unanswered questions that implementers will keep
re-asking.

---

### Recommendation

**Recommended disposition:** B

Disposition B reflects what the responsible parties have already said in
public — FHIR-I has signalled in writing that the Operations for Large
Resources work is moving into the API Incubator, and the implementers
thread closed with the same conclusion — while still delivering value in
R6 (the unambiguous typo fix plus an honest "here is what works today
and here is where the rest is being worked" paragraph). It avoids the
real risk in Disposition A, which is designing four interconnected new
mechanisms (issue-type code, `$filter` parameters, subset-`PUT`,
`$update-item`) under ballot pressure when the FHIR-I commenters are
not aligned on the need, and it avoids the costs of Disposition C,
which leaves the page with the typo *and* with PA's questions
unaddressed in any visible forum. The workgroup should also commit to
the dedicated PA / FHIR-I joint call Brian flagged in FHIR-37648 so
that the Incubator work has a clear set of acceptance criteria pulled
from the PA `Encounter.participant` use case, rather than drifting.
