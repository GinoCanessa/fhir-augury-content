## Summary

Chris Moesel ([FHIR-29550](https://jira.hl7.org/browse/FHIR-29550)) reports
that the FHIR Shorthand (FSH) specification does not currently document an
important behavior of FSH instance generation: when an `Instance` is declared
with `InstanceOf:` a profile (or any StructureDefinition), the SUSHI
implementation automatically applies any required `fixed[x]` and `pattern[x]`
values from that profile (including those bound through required slices) to
the generated instance. The example given is
`USCoreLaboratoryResultObservationProfile`, which fixes the `laboratory`
code on a required `category` slice — instances declared as
`InstanceOf: USCoreLaboratoryResultObservationProfile` therefore do not need
to set `category` explicitly because the value is auto-populated. The ticket
asks that this auto-application behavior be called out in the FSH
specification so that authors can rely on (and reason about) it. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the indexed sources for this ticket.

## Details

**Description:**

> One benefit of using FSH for examples is that any required fixed (or
> pattern) values are automatically assigned to the instance. For example,
> instances of USCoreLaboratoryResultObservationProfile do not have to
> assign the "laboratory" code to "category" because "category" is required
> (1..*) in the profile with a required slice that fixes the category code
> to "laboratory". Therefore, an instance that is
> "InstanceOf: USCoreLaboratoryResultObservationProfile" will automatically
> have that value applied.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative-style subsection to the FSH specification (in the
"Instances" / "Defining Instances" chapter, near the description of the
`InstanceOf` keyword) titled something like *"Automatic Application of
Fixed and Pattern Values"*. The subsection would state that when an
`Instance` is generated against a profile or StructureDefinition, the
implementation applies, on the generated resource:

1. Any element-level `fixed[x]` value defined on a required (min ≥ 1)
   element in the differential or snapshot.
2. Any element-level `pattern[x]` value defined on a required (min ≥ 1)
   element in the differential or snapshot.
3. The same, when those constraints are carried by a **required** slice
   discriminator (i.e., a slice with min ≥ 1 whose discriminator value is
   fixed/patterned), as in the
   `USCoreLaboratoryResultObservationProfile.category:us-core` example.

Include a short worked example using the US Core laboratory category slice
and explicitly note that authors do not need to (and should not need to)
re-state these fixed values in the FSH `Instance` definition. Also document
the boundary cases: optional (min = 0) fixed/pattern values are *not*
auto-applied, and `MustSupport` alone does not trigger auto-application.

#### Justification

The behavior already exists in SUSHI and is one of the principal
authoring affordances FSH provides for example resources, so documenting
it is purely a clarification — it does not change tooling behavior or
break existing IGs. Leaving it undocumented forces authors to learn it by
experiment or by reading SUSHI source, and creates ambiguity about which
fixed values are guaranteed to appear in the generated JSON/XML. The
reporter is the FSH lead and is describing observed, intended behavior,
so the spec is the gap, not the implementation.

---

### Disposition B: Alternative Approach

#### Proposal

Document the behavior, but frame it more carefully as a description of
**SUSHI's reference implementation** (and the conformance expectation
for any FSH compiler) rather than a guarantee about every conceivable
fixed/pattern situation. Specifically:

1. Add the same subsection as in Disposition A, but place it under a
   "FSH Compiler Behavior" / "Conformance" heading and use language such
   as "A conforming FSH implementation MUST apply ... " for the cases
   that are unambiguous (single-cardinality required elements with
   `fixed[x]` / `pattern[x]`), and "SHOULD apply ... " for the more
   nuanced cases (required slices, nested patterns inside complex
   patterns, choice-type `[x]` elements with patterns).
2. Add a short "Authoring Guidance" callout stating that authors *may*
   still write the fixed values explicitly in the `Instance` if they
   prefer literal/self-contained examples, and that doing so is not an
   error as long as the values match.
3. Add an explicit non-goal: FSH does not auto-populate values driven by
   invariants, terminology bindings, or `MustSupport` flags.

#### Justification

This addresses the same documentation gap but separates the *conformance
contract* (what every FSH compiler must do) from the *user-facing
convenience* (what an author can rely on when reading their own FSH).
That separation matters because at least one edge case — required slices
where the discriminator is `pattern` rather than `fixed`, and nested
fixed values inside complex patterns — is implementation-sensitive and
has historically tripped up tooling. Calling those out as SHOULD-level
behavior, with examples, gives implementers room to converge while still
giving authors the documentation they asked for.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** on the grounds that auto-application of
required fixed/pattern values is an implementation behavior of SUSHI
rather than a property of the FSH language itself, and that documenting
it in the FSH specification risks coupling the language to one
implementation's choices.

#### Justification

This rationale is technically defensible (FSH-the-language vs.
SUSHI-the-compiler is a real distinction), but it is weak in practice:
the FSH spec already describes other behaviors that are effectively
SUSHI-defined (e.g., how `Caret Rules` resolve, how slicing rules are
interpreted), and the auto-application behavior is something authors
materially depend on when writing `Instance` definitions. Declining
would leave the documented gap that the reporter identified and would
not improve interoperability between FSH compilers — it would simply
push the question into tribal knowledge.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures everything Disposition A asks for — the worked
example, the explicit statement that fixed/pattern values on required
elements (including required slices) are auto-applied, and the
US Core laboratory category illustration — but does so in a way that
draws a clean line between *conformance* (what every FSH compiler must
do) and *convenience* (what authors get to omit). That distinction
protects the spec from being over-specified by a single implementation's
behavior, gives other FSH tooling a clear conformance target, and still
delivers the documentation the reporter is asking for. The added
"non-goals" callout (no auto-application from invariants, bindings, or
`MustSupport`) prevents a predictable follow-up ticket. Disposition A is
a perfectly acceptable fallback if the workgroup feels B over-engineers
a documentation fix; Disposition C should be rejected because the
underlying gap is real and authors are already relying on the behavior.
