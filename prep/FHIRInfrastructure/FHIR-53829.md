## Summary

Eric Haas requests that the FHIR specification's rendering of
`SearchParameter` resources be enhanced so that, alongside the
human-readable description currently published, each SearchParameter
also surfaces (a) a link to its **formal (machine-readable) definition**
and (b) an **example URL** showing the SearchParameter in use. Today,
to obtain the formal definition of a SearchParameter a reader must
download either the FHIR package or the full source for
[searchparameter-registry.html](https://hl7.org/fhir/6.0.0-ballot4/searchparameter-registry.html);
the request is to bring SearchParameter rendering in line with the
treatment already given to other conformance artifacts (notably
OperationDefinition), where the formal definition is offered directly
from the spec page. The reporter has attached a mock-up showing the
proposed additions in red. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were found by Augury for this issue.

## Details

**Description:**

Enhance SearchParameter rendering to include a link to add a Formal

   <https://hl7.org/fhir/6.0.0-ballot4/searchparameter-registry.html>.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Core publication tooling so that every rendered
`SearchParameter` page (and the per-row entries in
`searchparameter-registry.html`) gains the same "formal definition" and
"example" affordances that OperationDefinition pages already expose.
Concretely:

- Add download links for the SearchParameter resource in each serialization
  the spec already publishes for other conformance resources (JSON, XML,
  Turtle, and any canonical form), pointing at the per-resource artifact
  files emitted by the publisher (e.g.,
  `SearchParameter-<id>.{json,xml,ttl}`).
- Add an "Example" section that renders a worked example URL using the
  SearchParameter (e.g., `GET [base]/Patient?<code>=<value>`), derived
  from the SearchParameter's `base`, `code`, and `type`. Where multiple
  bases apply, render one example per base.
- Apply the same additions to the rows in `searchparameter-registry.html`
  so the registry is navigable to both the formal definition file and a
  representative example without needing to download the full package.

The mock-up attached to the ticket
([26520_image-…png](https://jira.hl7.org/secure/attachment/26520/26520_image-2026-01-06-10-29-32-806.png))
should be treated as the visual contract for the change.

#### Justification

The change is small in spec semantics (no new SearchParameter elements
or constraints) but high in reader value: it brings SearchParameter to
parity with OperationDefinition, StructureDefinition, ValueSet, and the
other conformance artifacts the spec already renders with download links.
Consistency across conformance-artifact pages is a long-standing FHIR
publication principle, and the missing affordance is an irritant for
implementers who currently must pull the entire NPM package just to
inspect one SearchParameter's JSON. Generating the example URL is
mechanical from the existing SearchParameter content
(`base[]` + `code` + `type`).

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the intent of the request but split the two affordances and
narrow the scope:

1. **Formal definition links** — add download links (JSON/XML/Turtle) to
   each individual SearchParameter page and to each row of
   `searchparameter-registry.html`. This is the unambiguous parity fix.
2. **Examples** — instead of auto-generating a single example URL on the
   page, link each SearchParameter to a curated examples section (or to
   the `searchparameter-registry` examples table) so that examples can
   be reviewed, hand-tuned for realism (modifiers, chaining, composite
   parameters), and translated where the auto-generated form would be
   misleading (e.g., reference, composite, or special-type parameters
   such as `_filter`, `_has`, `_text`).

This keeps the high-value, low-risk piece (download links) on the
fast track while routing the more nuanced "what does a good example
look like" question through a curation process rather than codegen.

#### Justification

Auto-generating example URLs from `base` + `code` + `type` works
cleanly for `string`, `token`, `date`, and `number` parameters but is
materially misleading for `reference`, `composite`, `quantity`, and
`special` SearchParameters, and for parameters that interact with
modifiers or chaining. Mechanically rendered examples could entrench
incorrect usage patterns. Splitting the change lets the formal-definition
parity ship immediately while the examples portion gets the editorial
attention it needs.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the formal definitions are
already published — the FHIR NPM package and the
`definitions.json.zip` / `definitions.xml.zip` bundles linked from the
downloads page contain every SearchParameter resource — and that the
existing `searchparameter-registry.html` plus per-resource search pages
already document each SearchParameter's behaviour. The "Examples" piece
is best handled by the implementation guides that profile the parameters
in context.

#### Justification

A defensible decline rests on (a) the formal definitions being
non-novel and already redistributable via the package, and (b) concern
about expanding the publisher's per-artifact templates when registry
download links could be added more cheaply. This is the weakest of the
three positions because it does not address the reporter's actual pain
point (per-page friction).

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the entire reader-experience win the reporter
is asking for — direct download links to the formal definition on every
SearchParameter page and on the registry — while sidestepping a known
hazard with auto-generated example URLs for `reference`, `composite`,
`quantity`, and `special`-typed parameters where a naive `?<code>=<value>`
template would mislead implementers. It is also the disposition that
most cleanly mirrors how OperationDefinition pages are actually rendered
today (download links present; examples are curated and linked rather
than synthesised). If the work group decides the example-URL generation
is acceptable for the well-behaved subset of SearchParameter types, the
proposal naturally collapses back to Disposition A.
