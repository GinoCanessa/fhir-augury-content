## Summary

The reporter asks the FHIR Shorthand (FSH) specification to drop its use of
the full RFC 2119 conformance vocabulary and align with the smaller subset
that HL7 implementation guides conventionally use: **SHALL, SHALL NOT,
SHOULD, SHOULD NOT, MAY**. The argument is that the broader RFC 2119 set
(MUST / MUST NOT / REQUIRED / RECOMMENDED / OPTIONAL, in addition to the
HL7 set) introduces ambiguity for readers — e.g., it is not obvious whether
"SHALL" and "MUST" carry different normative weight — and that this
divergence creates inconsistency between the FSH IG and other HL7 guides
that already use only the HL7 subset. No cross-referenced Jira tickets,
linked GitHub items, or Zulip discussions were located in the FHIR Augury
indexes for this ticket, so the request stands on its own.

## Details

**Description:**

There is no benefit from allowing the wide range of conformance language
specified in RFC2119, as the distinctions between these words can be quite
ambiguous and is not standard for HL7 guides. (e.g., "SHALL vs. MUST vs.
REQUIRED?", etc.)

Utilizing this broad set of terms will actually introduce potential
confusion around the inconsistent use of conformance language between this
guide and others which follow the concise typical HL7 subset.

Recommend aligning this IG with the standard set set of conformance
language endorsed/used by HL7:

- SHALL
- SHALL NOT
- SHOULD
- SHOULD NOT
- MAY

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Shorthand IG so that all normative statements use only
the HL7-standard subset — **SHALL, SHALL NOT, SHOULD, SHOULD NOT, MAY** —
and remove (or rewrite) any prose currently using **MUST, MUST NOT,
REQUIRED, RECOMMENDED, NOT RECOMMENDED, OPTIONAL**. Concretely:

- Replace the existing RFC 2119 boilerplate (typically on the spec's
  conventions / normative-language page) with the standard HL7 wording
  that defines only the five-keyword set, citing the FHIR core spec's
  conformance-rules page rather than RFC 2119 directly.
- Sweep the FSH IG narrative pages and rewrite each occurrence:
  `MUST` / `REQUIRED` → `SHALL`; `MUST NOT` → `SHALL NOT`;
  `RECOMMENDED` → `SHOULD`; `NOT RECOMMENDED` → `SHOULD NOT`;
  `OPTIONAL` → `MAY`. Verify each rewrite preserves the intended
  normative force (no silent strength changes).
- Add an editor's note in the change log identifying this as an
  editorial alignment with HL7 house style and stating that no
  normative requirement of the IG has been weakened or strengthened.

#### Justification

This is consistent with HL7 publishing convention: the FHIR core spec
itself uses only the five-keyword subset and is the upstream authority
for FSH. Aligning the IG removes a real source of reader confusion
(SHALL vs. MUST is a recurring question on chat.fhir.org) and brings the
guide in line with sibling HL7 publications, with no implementer-facing
behavior change. Because the change is editorial — every replaced
keyword has a one-to-one HL7 equivalent — the risk of regression is low
provided each rewrite is reviewed in context.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the underlying RFC 2119 vocabulary in scope but constrain the IG's
*usage* via a conformance-language section rather than a wholesale
sweep:

- Add an explicit "Conformance Language" section to the FSH IG that
  states the IG uses the HL7 subset (SHALL / SHALL NOT / SHOULD /
  SHOULD NOT / MAY), and that any occurrence of MUST / REQUIRED /
  RECOMMENDED / OPTIONAL is to be read as the equivalent HL7 keyword
  (define the mapping table inline).
- Pair this with a more limited prose sweep that fixes only the most
  visible occurrences (the introduction, the conventions page, and any
  page that mixes both vocabularies in a way that is genuinely
  ambiguous), deferring a full rewrite.
- Optionally, add a CI / lint check (markdown or page-source linter) to
  flag new uses of the non-HL7 keywords in future PRs, so the IG
  converges on the HL7 subset over time without a single large diff.

#### Justification

This addresses the reporter's concern (reader confusion, divergence
from HL7 norms) while reducing the cost and risk of a full rewrite —
notably the risk of a careless replacement subtly changing a normative
statement's strength. It is also more friendly to outstanding PRs and
draft content that may already use mixed vocabulary. The trade-off is
that the IG remains technically permissive of the broader vocabulary
until the long-tail rewrite finishes, so the inconsistency the reporter
flagged persists for some pages until cleanup completes.

---

### Disposition C: Decline

#### Proposal

Decline the change as out of scope for normative impact: confirm in the
ticket resolution that the FSH IG considers any RFC 2119 keyword to
carry its standard RFC 2119 meaning, that the five-keyword HL7 subset
and the broader RFC 2119 set are not in conflict (the HL7 subset is a
strict subset of RFC 2119), and that no implementer is being misled in
a way that affects conformance testing. Close as "Not Persuasive" /
"Considered for Future Use" rather than rewriting prose.

#### Justification

RFC 2119 is the upstream authority both HL7 and FSH cite, and the
keywords the reporter wants removed are well-defined by RFC 2119 with
unambiguous semantics. No conformance ambiguity actually exists at the
testing level — `MUST` and `SHALL` are equivalent normative force.
House-style alignment is a legitimate goal but, absent a concrete
implementer harm, may not justify the editorial churn and review cost
across the IG. This disposition is the weakest of the three because it
ignores the consistency-with-other-HL7-IGs argument, which is the real
substance of the request.

---

### Recommendation

**Recommended disposition:** A

The request is essentially editorial and aligns the FSH IG with
established HL7 house style and with the conformance-language
conventions used by the FHIR core specification itself. The reporter's
underlying concern — that mixing vocabularies invites readers to assume
a meaningful difference where none exists — is a real and recurring
source of confusion in HL7 specifications. A clean sweep (Disposition
A) produces a single, reviewable change set and leaves the IG with a
single conformance vocabulary, which is easier to maintain and to
reason about than the staged approach in Disposition B. Disposition C
does not engage with the consistency argument and is not recommended.
The workgroup should ask the FSH editors to scope the rewrite as
strictly editorial, with explicit reviewer guidance to flag any
substitution that appears to change normative force.
