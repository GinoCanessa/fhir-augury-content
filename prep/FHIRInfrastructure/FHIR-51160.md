## Summary

Isaac Vetter reports that the element documentation ("Comments") on
`CapabilityStatement.rest.searchParam` (the system-level search-parameter
list, applied when a client searches across all resource types) appear to
have been partly copied from `CapabilityStatement.rest.resource.searchParam`
(the per-resource list) and are therefore incorrect at the system level.
Specifically, the comment text talks about parameters that "also apply to
this resource" and self-references `CapabilityStatement.rest.searchParam` as
the place where many control parameters "will be listed" — both phrasings
only make sense in the per-resource context, not at the system level.

The reporter asks that the last two sentences of the comments on
`CapabilityStatement.rest.searchParam` be stricken or corrected. The fix is
purely editorial / definitional — it changes the human-readable element
documentation in the CapabilityStatement StructureDefinition; it does not
change cardinality, datatypes, invariants, search-parameter behavior, or
any wire format.

There are no cross-referenced Jira tickets, no related GitHub items, and no
in-person requesters. The Augury cross-reference index links the ticket
only to the two FHIR artifacts above (the very elements being discussed).

### [implementers > CapabilityStatement.rest.searchParams for common params](https://chat.fhir.org/#narrow/stream/implementers/topic/CapabilityStatement.rest.searchParams%20%20for%20common%20params/near/225855226)

Cooper Thompson asked whether `CapabilityStatement.rest.searchParam` is the
right place to document parameters that are common across all the
resources a server lists (e.g., `_id`, `_count`, `_revInclude`, `_include`).
Initial responses from René Spronk and Eric Haas suggested that
system-level use is reasonable when the parameter behaves identically
across types. Grahame Grieve then clarified that
`rest.searchParam` is specifically for searches **without** a type
specified (i.e., `GET [base]?...`) — it is **not** a shorthand for "this
parameter is supported on every resource type listed below." Cooper
concluded he would file a tracker (FHIR#31051) to clarify the spec text,
and noted that Inferno was already enforcing the per-resource
interpretation. Eric Haas remained unconvinced, noting confusion about why
some control parameters seem appropriate at the system level but `_id`
would not be. The thread reached rough consensus on Grahame's
interpretation but explicitly flagged the spec wording as unclear.

Participants: Cooper Thompson, Lloyd McKenzie, René Spronk, Eric Haas,
Yunwei Wang, Grahame Grieve.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Strike the last two sentences of the Comments on
`CapabilityStatement.rest.searchParam` (the system-level element). The
revised Comments would read:

> Typically, the only search parameters supported for all searches are
> those that apply to all resources — tags, profiles, text search etc.

Concretely: in the FHIR Core source for the CapabilityStatement
StructureDefinition (the element-definition entry for
`CapabilityStatement.rest.searchParam`), remove the two erroneous
sentences from the `comment` field. No change to `definition`, `min`,
`max`, `type`, invariants, or any binding. Regenerate the published
`capabilitystatement-definitions.html` page from the revised source.

#### Justification

The flagged sentences are demonstrably inconsistent with Grahame Grieve's
own clarification on Zulip: `rest.searchParam` describes parameters
supported when the client searches **without** specifying a type
(`GET [base]?...`), not parameters "that also apply to this resource."
The phrase "this resource" has no referent at the system level, and the
self-link to `CapabilityStatement.rest.searchParam` as the place where
control parameters "will be listed" is an obvious copy-paste artifact
from the sibling `rest.resource.searchParam` element. Striking the two
sentences is the minimum change that removes the error without
introducing new substantive guidance that the workgroup would need to
re-debate.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only striking the offending text, **rewrite** the Comments on
`CapabilityStatement.rest.searchParam` so they describe what the element
is actually for, and add a parallel cross-reference to its sibling. Suggested
text:

> Typically, the only search parameters supported for all searches are
> those that apply to all resources — tags, profiles, text search, etc.
> These are the parameters honored when the client issues a search
> across the whole server without specifying a resource type
> (e.g., `GET [base]?...`). Search parameters that apply when the
> client constrains the query to a specific resource type (including
> control parameters such as `_sort`, `_count`, `_include`, and
> `_revInclude` that the server only honors per type) belong on
> [CapabilityStatement.rest.resource.searchParam](capabilitystatement-definitions.html#CapabilityStatement.rest.resource.searchParam).

In parallel, tighten the Comments on
`CapabilityStatement.rest.resource.searchParam` to remove its
self-referential "(though many will be listed at
CapabilityStatement.rest.searchParam)" parenthetical, replacing it with a
forward pointer to the system-level element only when the parameter is
truly server-wide.

#### Justification

The Zulip thread from 2021 shows that the underlying confusion was not
limited to one wrong sentence — implementers (Cooper, Eric Haas, Yunwei
Wang) and a co-chair (Grahame) all read the existing wording differently.
A bare strike (Disposition A) removes the wrong text but leaves the
element under-documented and does not protect future readers from making
the same conflation. Rewriting both sibling elements in tandem fixes the
root cause (parallel comments that drifted apart) and aligns the
normative intent that Grahame articulated. The cost is that the workgroup
must agree on the new wording, and the change is editorial-but-substantive
rather than purely editorial.

---

### Disposition C: Decline

#### Proposal

Reject the change. Leave the Comments on
`CapabilityStatement.rest.searchParam` as currently published.

#### Justification

The element's `definition` field (not quoted in the ticket) is the
normative description; the `comment` field is non-normative explanatory
text. An argument can be made that the workgroup should not spend cycles
on non-normative wording when no implementer behavior is at stake and no
breaking change is involved. This disposition is the weakest of the
three: the contested sentences are not merely awkward, they are
factually wrong (they refer to "this resource" at a system-level
element and self-link the element to itself), and leaving them in place
risks perpetuating the confusion documented on Zulip.

---

### Recommendation

**Recommended disposition:** B

The 2021 implementers thread is direct evidence that the current wording
misleads not just the reporter but multiple experienced implementers and
even prompted disagreement that required a co-chair to resolve. A pure
strike (Disposition A) is technically sufficient to remove the falsehood,
but it would leave the system-level element with a single sentence of
explanation and would not address the symmetric copy-paste contamination
on the sibling element's parenthetical. Disposition B does the
slightly-larger editorial job once, captures Grahame's clarification in
the spec itself (so future readers do not have to find the Zulip thread),
and remains a Comments-only change — no normative content, cardinality,
binding, or wire-format impact. Decline (Disposition C) is not defensible
given that the reporter and the prior thread both demonstrate active
implementer confusion. Workgroup discussion should focus on agreeing the
exact replacement wording for the two sibling elements.
