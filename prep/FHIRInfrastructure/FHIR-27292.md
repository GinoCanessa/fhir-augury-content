## Summary

The reporter argues that the FHIR Shorthand (FSH) specification reads adequately as a reference but
is insufficient as teaching material for newcomers. They specifically call out that the example
content is too narrow and request a more introductory, tutorial-style presentation of how to use
SUSHI for practical work — invoking the first half of Kernighan & Ritchie's *The C Programming
Language* as the model of the kind of pedagogical on-ramp they have in mind. The cross-reference
search surfaced no related Jira tickets, no Zulip discussions, and no GitHub items; the sole
cross-reference is a Confluence link to the *2020-05 FHIR Shorthand Track* tutorials page, which
the FHIR Shorthand work group offered (via Mark Kramer and Lynn Laakso) as the existing
tutorial material that partially addresses the request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Commission a dedicated pedagogical/tutorial section for the FHIR Shorthand (FSH) specification
that sits alongside (not in place of) the existing reference material. Concretely:

- Add a new top-level "Tutorial" or "Learn FSH" section to the FSH IG that walks a brand-new
  reader from "what is FSH and why would I use it" through building a small, complete IG with
  SUSHI end-to-end (install, init, author one Profile + one Extension + one ValueSet + one
  example Instance, run SUSHI, inspect the generated artifacts, publish).
- Expand the example portfolio: replace the single narrow example with a graduated sequence
  modeled on K&R — each chapter introduces one new FSH construct and shows the corresponding
  generated FHIR artifact diff.
- Promote the Confluence Connectathon tutorial material
  (https://confluence.hl7.org/display/FHIR/2020-05+FHIR+Shorthand+Track) into the published IG so
  that learners do not need to know about the Connectathon page to find it.
- Re-resolve the ticket as *Persuasive with Modification* once the new material lands in the
  next FSH publication.

#### Justification

The reporter's concern is concrete and aligned with prior work group sentiment: Mark Kramer and
Lynn Laakso both acknowledged the gap and pointed at compensating material (the *Overview* page
and the Connectathon tutorials). Pulling that tutorial content into the published IG closes the
discoverability gap that motivated the original ticket without invalidating any normative or
informative content already published in STU1. K&R-style "introduce one feature at a time with
a runnable example" is a well-proven format for tooling docs and matches how SUSHI is actually
used day-to-day. Because FHIR Infrastructure / FSH owns its own publication pipeline, this is
purely an authoring change — no normative impact, no breaking change for existing FSH authors.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the FSH IG itself focused on reference material, but publish the pedagogical content as a
*separate, linked* "FSH Tutorial" companion artifact (either a sibling IG, a standalone HL7
Confluence-hosted course, or a curated link bundle inside the existing *Overview* page).
Specifically:

- Leave the current FSH IG structure intact.
- Restructure the existing *Overview* page into a "Where to learn FSH" hub that points
  prominently at: (a) the Confluence tutorials, (b) any recorded Connectathon walkthroughs,
  (c) the SUSHI README/quick-start, and (d) a short worked example embedded in the IG.
- Optionally, package a small "fsh-tutorial" sample project (in a dedicated repository) that
  learners can clone and run against current SUSHI.

#### Justification

This honours the reporter's underlying need (a friendlier on-ramp for new users) without
expanding the scope or page count of the normative IG. The work group already signalled in
2020 that they wanted to ship a timely STU release and treat docs improvement as ongoing;
splitting tutorial material out keeps the IG release cadence decoupled from tutorial churn,
and makes it easier to keep the tutorial in sync with the rapidly evolving SUSHI tooling.
Trade-off vs. Disposition A: discoverability is weaker — readers who land on the IG first
still have to follow a link to find the tutorial — but the maintenance burden on the
specification editors is materially lower.

---

### Disposition C: Decline

#### Proposal

Resolve as *Not Persuasive* (or *Considered for Future Use*, restoring the 2020-07-06 vote
disposition that was later reverted): the request is too open-ended to action ("more
pedagogical", K&R-style) and the work group has already added the *Overview* page and
maintains Connectathon tutorial material that addresses the underlying need. Close with a
note pointing readers to the existing Overview page and the Confluence tutorial track.

#### Justification

The ticket does not name a specific section, example, or construct that is missing — it
asks for a stylistic re-framing of the IG. Stylistic rewrites are hard to bound, hard to
review, and hard to declare "done"; the original 2020-07-06 vote (11-0-1 Moesel/Rhodes,
deferring) reflected exactly that pragmatism. The compensating material called out in the
comments (Overview page + Connectathon tutorials) already exists, so the gap is one of
discoverability rather than absence. Declining preserves editor time for changes that have
clearly bounded scope and observable acceptance criteria.

---

### Recommendation

**Recommended disposition:** A

The reporter's complaint is concrete enough to action even though it is phrased aspirationally:
the existing IG has *one* narrow example and the tutorial material that the work group itself
treats as part of the package lives outside the published IG. Disposition A converts the
already-acknowledged compensating material (Overview page + Connectathon tutorials) into
first-class, discoverable parts of the FSH IG and adds the graduated worked-example sequence
the reporter is really asking for. It carries no normative risk, it builds on work the FSH
team has already done, and it directly addresses the reason the ticket was re-opened after
STU1 publication. Disposition B is a defensible fallback if editor capacity is the binding
constraint, but it leaves the discoverability problem largely intact. Disposition C recreates
the resolution that Lynn Laakso explicitly reverted in October 2020, so re-applying it without
new information would not be a satisfying outcome for the workgroup.
