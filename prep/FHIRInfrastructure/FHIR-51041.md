## Summary

Rik Smithies argues that the term "additional resources" — currently used
in FHIR for the recently-introduced category of resources that sit
alongside the established core set — is misleading. Read literally, it
suggests that anyone can author their own resources (which FHIR would
actually call "custom resources"), rather than describing a curated set
of HL7-stewarded core resources that are simply newer or in a different
maturity tranche. The reporter proposes renaming the category to either
"additional core resources" (preserving the existing short form
"additional resources" in informal use) or "interim resources" (framing
the distinction as one of timing rather than stewardship). The ticket is
purely a naming/communication concern and does not change normative
behaviour or content of any resource. No cross-referenced Jira tickets,
Zulip discussions, or GitHub items were surfaced for this ticket.

## Details

## Proposed Dispositions

Note: because there are no related GitHub items or repo briefings, the
dispositions below are scoped to FHIR Core specification prose and
navigational pages rather than to specific authoring roots. If a
disposition is adopted, the implementing change is expected to land in
the FHIR Core specification source (the build inputs that produce the
"additional resources" category page and its inbound links from
overview / TOC pages), rather than in any resource definition file.

### Disposition A: Accept As Requested

#### Proposal

Rename the category currently called "additional resources" to
"**additional core resources**" throughout the FHIR Core specification.
Concretely:

- Update the heading and prose of the dedicated category page (the page
  that today introduces and lists the "additional resources" tranche)
  to use "Additional Core Resources" as the primary name.
- Update inbound links from the resource list / overview / TOC pages
  and any "What's New" / maturity narrative that references the
  category.
- Add a one-sentence note clarifying the contrast with **custom
  resources** (i.e., user-defined resources outside the HL7 core set),
  and explicitly state that "additional resources" remains acceptable
  as an informal short form.
- Leave the underlying resource definitions, URLs, and conformance
  artefacts untouched — this is a documentation/terminology change
  only.

#### Justification

The reporter is a long-standing FHIR implementer and explicitly
describes being misled by the current name. "Additional core resources"
is the smallest possible change that removes the ambiguity, because it
adds the one word ("core") that disambiguates HL7-stewarded content
from implementer-defined content. It preserves the existing short form
for slides and conversation, so no community muscle memory is lost, and
the change is confined to specification prose with no impact on
generated resource definitions, URLs, or conformance behaviour.

---

### Disposition B: Alternative Approach

#### Proposal

Rename the category to "**interim resources**" (the reporter's second
suggestion), and re-frame the introductory prose so that the category
is defined by *maturity / timing* rather than by being "additional" to
anything. Concretely:

- Rename the category page to "Interim Resources" and rewrite its
  opening paragraph to explain that these are HL7-stewarded core
  resources whose lifecycle / balloting timing differs from the
  established core set, rather than resources that sit "outside" or
  "in addition to" FHIR.
- Update inbound links and the resource list / TOC.
- Add explicit cross-links to the definition of **custom resources** so
  readers cannot conflate the two categories.
- Optionally, retain "additional resources" only as a redirect / legacy
  anchor for one cycle to avoid breaking external links.

#### Justification

"Additional core resources" (Disposition A) removes the
implementer-authoring ambiguity but still implies an ordering
relationship ("additional *to* what?") that some readers will find
awkward. "Interim resources" reframes the distinction as a lifecycle
property, which is closer to what the category actually means in
practice and is harder to mis-read as inviting implementer-authored
resources. The trade-off is higher disruption: the new name is less
recognisable, materials and slide decks already in circulation will
need updating, and "interim" carries its own risk of being read as
"temporary / will be removed", which the prose change has to
neutralise.

---

### Disposition C: Decline

#### Proposal

Decline the rename and instead address the misleading-name concern
through a clarifying note on the existing "additional resources" page
and on the resource list / overview pages that link to it. The note
would explicitly contrast "additional resources" (HL7-stewarded core
resources in this tranche) with "custom resources"
(implementer-defined). No identifiers, page slugs, or category labels
would change.

#### Justification

The category name is already in use across FHIR Core prose, conference
material, and downstream community content; renaming has a non-trivial
churn cost for documentation, presentations, and external references.
If the only evidence of confusion is a single reporter's first
impression and there is no Zulip / GitHub trail of broader
implementer-community confusion (none was surfaced for this ticket),
then a clarifying paragraph is a proportionate response that preserves
existing references and search results.

---

### Recommendation

**Recommended disposition:** A

"Additional core resources" is the smallest change that directly fixes
the ambiguity the reporter identified — it adds one word to make HL7
stewardship explicit, and it preserves the existing short form
"additional resources" for informal use, so already-circulating slides
and conversational shorthand remain valid. Disposition B (interim
resources) reframes the category more cleanly but introduces its own
risk ("interim" reading as "temporary") and requires more prose
rework; it would be the right pick only if the work group concludes
that the *additional-to-what* framing is itself the problem, not just
the missing "core" qualifier. Disposition C (clarifying note only)
under-responds to a concern raised by an experienced implementer about
a category name that will appear in prominent navigational positions
for years; a small terminology fix now is cheaper than a clarifying
note that has to keep being re-issued. Recommend the work group adopt
A and treat the in-prose contrast with "custom resources" (which both
A and B include) as a required part of the change.
