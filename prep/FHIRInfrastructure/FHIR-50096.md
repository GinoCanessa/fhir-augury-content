## Summary

The ticket asks the SMART App Launch (STU2/v2.2) specification to clarify
whether a SMART **server** must reject `/authorize` requests that omit the
`code_challenge` (PKCE), `aud`, or `state` parameters — or whether the
server merely needs to "support" them. The reporter argues that the spec
text only obliges the **app** to send those parameters; that PKCE, `aud`,
and `state` exist primarily to protect the *app* (not the server); and
that there is no normative SHALL on the server to reject requests missing
them. He notes that the only practical way to enforce PKCE server-side
would be to actively reject non-PKCE requests, which would break every
SMARTv1 client sharing the same authorize endpoint, with no obvious
signal for the app to know when PKCE is mandatory. He therefore asks
FHIR-I to either (a) state explicitly that servers must reject such
requests and explain the security benefit, or (b) state explicitly that
"support" does not require rejection. The cross-referenced 2022 Zulip
discussion (`smart > ✔ SMARTv2 PKCE Requirements`) reached an informal
conclusion that v2 PKCE enforcement is hard to test on a mixed v1/v2
endpoint and pragmatically suggested keying enforcement off v2-style
scopes — but this was never written into the specification.

## Details

**Description:**

PKCE, `aud`, and `state` parameters are required to be provided by the
app to the SMART server's authorize endpoint, and are required to be
"supported" by the SMART server in SMARTv2. All three can meaningfully
be supported by the server without the server requiring the client to
include the specific parameter on an authorize request. Please clarify
if the server should reject requests to the authorize endpoint unless
these variables are provided.

SMART v2.2 clearly mandates that both apps and servers support PKCE:
*All SMART apps SHALL support Proof Key for Code Exchange (PKCE) ...
SMART servers SHALL support the `S256` `code_challenge_method` and SHALL
NOT support the `plain` method.*
(<https://hl7.org/fhir/smart-app-launch/STU2/app-launch.html#considerations-for-pkce-support>)

Additionally, it's clear that apps are required to at least somewhat use
PKCE by supplying the `code_challenge` and `code_challenge_method`
parameters as part of the `/authorize` request:
*The app supplies the following parameters ... required ...
code_challenge*
(<https://hl7.org/fhir/smart-app-launch/STU2/app-launch.html#request-4>)

The only way to validate that an app used PKCE would be to introduce a
failure condition of returning a bad code to the app. There's no
explicit requirement that a SMART server mandate that apps supply PKCE
parameters by rejecting authorize requests without one. Further, PKCE
protects the app, not the server, and PKCE is primarily relevant in
specific scenarios that the server isn't able to know (e.g. native
apps). If the app doesn't need or chooses not to use this additional
capability, it seems to violate a separation of concern for the server
to require it.

Josh and Stephen from MITRE had a somewhat similar discussion in 2022
(see `smart > ✔ SMARTv2 PKCE Requirements`) with a conclusion that
SMARTv2 servers are required to reject requests without a
`code_challenge`, but that Inferno couldn't test for that because it
would exclude SMARTv1 servers.

A more impactful reason is that requiring `code_challenge` would break
hundreds, thousands of apps. And it's not obvious how an app would even
know when they're required to supply a `code_challenge`, and when
they're required to ignore the absence of code.

I think that this is why `code_challenge` is only mandated for the app
actor.

~~~

Very similarly — the `aud` parameter:
*The app supplies the following parameters ... required ... aud*
(<https://hl7.org/fhir/smart-app-launch/STU2/app-launch.html#request-4>)

But, just as with PKCE, there's no explicit requirement that the server
reject requests without an `aud`, AND the `aud` exists to protect the
app and can only be used by the app. (Note that the language — "For the
current release, servers SHALL support the `aud` parameter and MAY
support a `resource` parameter" — didn't exist in v1 and was added to
explain the parameter name; the term *support* doesn't mean *require*.)

~~~

Lastly, `state` — here's the normative language:
*Apps SHALL generate an unpredictable `state` parameter for each user
session; SHALL include `state` with all authorization requests; and
SHALL validate the `state` value for any request sent to its redirect
URL.*
(<https://hl7.org/fhir/smart-app-launch/STU2/app-launch.html#app-protection>)

And:
*The app supplies the following parameters ... required ... state*
(<https://hl7.org/fhir/smart-app-launch/STU2/app-launch.html#request-4>)

`state` also protects the app, not the server, there's no requirement
on the server to reject requests without this parameter, and the server
can meaningfully support it without doing so.

For each of: PKCE, `aud`, and `state`, please clarify if the server
should or should not reject authorize requests missing these
parameters. At the very least, rejecting requests without PKCE will
break every v1 SMART app. So, if a server should reject, please explain
the security benefits for doing so.

## Related Zulip Discussions

### [smart > ✔ SMARTv2 PKCE Requirements](https://chat.fhir.org/#narrow/stream/smart/topic/%E2%9C%94%20SMARTv2%20PKCE%20Requirements)

Stephen MacVicar (MITRE/Inferno) opened the thread in August 2022
asking whether SMARTv2 truly mandates PKCE on every authorize request,
noting that the parameter chart on `/authorize` and `/token` colour-codes
PKCE the same as conditional parameters. Josh Mandel confirmed PKCE *is*
required in SMARTv2 and that the chart colours are inconsistent. The
hard question quickly became: if a single endpoint supports both v1 and
v2, how can a server enforce PKCE on v2 requests without breaking v1
clients (since nothing in an authorize request reliably identifies a
"v2 request" unless it uses v2-style scopes)? Josh suggested using
v2-style scopes (e.g. `patient/AllergyIntolerance.rs`) as the trigger
for enforcing v2 requirements like PKCE, acknowledging this is an
imperfect but pragmatic backward-compatibility trade-off and that a
purely negative test ("rejects requests when PKCE params are absent")
is hard to express. Stephen noted that the draft ONC g10 test procedure
only requires support for v1-style scopes, so Inferno was unlikely to
add v2-scope-based enforcement logic. The thread was marked **resolved**
by the Notification Bot on 2022-08-12, but the resolution was a shared
understanding rather than a normative spec change — exactly the
ambiguity FHIR-50096 is now asking the workgroup to close.

Participants: Stephen MacVicar, Josh Mandel, Notification Bot.

## Proposed Dispositions

(There are no GitHub cross-references and therefore no repo briefings;
all dispositions below would be authored against the SMART App Launch
specification source — the
[`HL7/smart-app-launch`](https://github.com/HL7/smart-app-launch)
repository — under its IG authoring root, but this skill has no cached
briefing for that repo, so concrete file paths are not asserted here.
The dispositions point at the affected spec sections instead.)

### Disposition A: Accept As Requested

#### Proposal

Add normative server-side language to **SMART App Launch STU2.x** that
explicitly states the server's behaviour for missing `code_challenge`,
`aud`, and `state` on the `/authorize` endpoint. Specifically, in the
`#request-4` section table and in `#considerations-for-pkce-support` /
`#app-protection`, add (or upgrade) the following SHALLs:

- **PKCE:** "SMART servers SHALL reject any `/authorize` request whose
  `response_type=code` AND whose request signals SMARTv2 (e.g., uses a
  v2-style scope such as `patient/Resource.cruds`, requests
  `fhirContext`, or asserts `code_challenge_method`) when the
  `code_challenge` and `code_challenge_method` parameters are not
  present. Servers SHALL NOT reject `/authorize` requests that signal
  SMARTv1 solely on the basis of missing PKCE parameters." Also add a
  short explanatory paragraph to
  `#considerations-for-pkce-support` describing the v1/v2 coexistence
  rule and pointing implementers at Inferno-style scope-based
  detection.
- **aud:** "SMART servers SHALL validate the `aud` parameter and SHALL
  reject `/authorize` requests where `aud` is absent or does not match
  the server's own FHIR base URL." (This matches OAuth 2.0
  audience-restriction guidance and aligns with how `aud` is already
  treated as required in the request table.)
- **state:** "SMART servers SHOULD echo `state` unchanged when
  redirecting back to the app, but SHALL NOT be required to reject
  requests that omit `state`; absence of `state` is an app-side
  protection failure, not a server-side authorization failure." (This
  preserves the existing app-protects-itself semantics for `state`
  while answering the question explicitly.)

Bundle the change as a clarification-and-correction trackable on the
SMART App Launch STU2 errata / next ballot, and update the parameter
table colours that originally caused the 2022 confusion.

#### Justification

The reporter is correct that the spec is ambiguous: "the app SHALL
supply X" plus "the server SHALL support X" does not, in OAuth 2.0
parlance, automatically imply "the server SHALL reject requests missing
X." Real-world EHRs running mixed v1/v2 endpoints have already had to
make ad-hoc decisions, and Inferno's own conclusion in the linked Zulip
thread was that the only testable interpretation is scope-keyed
enforcement. Writing that explicitly — *with different rules for the
three parameters* (mandatory rejection for `aud`, scope-keyed rejection
for PKCE, no rejection requirement for `state`) — closes the
interoperability gap and matches the underlying threat model: `aud`
prevents token-substitution attacks against the *server's* resource
base, PKCE protects the app's authorization-code-leg, and `state`
protects the app's session against CSRF. The disposition is consistent
with Josh Mandel's pragmatic resolution in the Zulip thread and with
OAuth 2.1 / RFC 7636 conventions.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding new SHALL-level server enforcement, **publish a
normative "Server Behaviour for Required App Parameters" subsection**
in SMART App Launch that does the following:

1. Defines two terms once and reuses them: *app-protective parameter*
   (PKCE, `state`) and *server-protective parameter* (`aud`).
2. States that **server-protective parameters SHALL be validated and
   missing/invalid values SHALL cause request rejection** — naming
   `aud` explicitly.
3. States that **app-protective parameters SHALL be supported by the
   server (accepted, validated when present, echoed where applicable)
   but their absence SHALL NOT, by itself, cause the server to reject
   the request** — naming `code_challenge`/`code_challenge_method` and
   `state` explicitly.
4. Adds a Security Considerations note recommending (SHOULD) that
   servers detect SMARTv2 intent (via v2 scopes, `fhirContext`, or a
   future explicit `version` hint) and apply stricter PKCE enforcement
   in those cases, but explicitly leaving this as a SHOULD so v1/v2
   coexistence is not broken.
5. Marks the parameter-table colour glitch as fixed in errata.

#### Justification

This disposition addresses the reporter's underlying ask — *"please
clarify whether the server should reject"* — without forcing a
breaking change on the wide install base of SMARTv1-only or mixed
v1/v2 endpoints. It reframes the question around the threat each
parameter mitigates, which is the principled way OAuth 2.0 / OAuth 2.1
treat the same trade-off (`aud` is server-protective in OAuth 2.0
audience-restriction; PKCE in OAuth 2.1 is mandatory for *public*
clients but not retroactively for confidential clients). It avoids the
worst-case outcome flagged by the second commenter — that server
operators implement aggressive PKCE rejection, breaking thousands of
v1 apps — while still committing the spec to a clear answer. The
SHOULD on scope-keyed PKCE enforcement gives Inferno and ONC g10 a
testable, bounded behaviour without a hard breaking change.

---

### Disposition C: Decline

#### Proposal

Decline to change normative text. Add only a non-normative
**implementer's note** to `#considerations-for-pkce-support` and to
`#app-protection` stating that the existing language is intentional:
"server SHALL support" means the server MUST accept and process the
parameter when present and MUST validate it when present, but does NOT
require the server to reject requests where the parameter is absent.
Direct readers to the SMARTv1↔v2 coexistence discussion in the
SMART App Launch FAQ. Close the ticket as *Persuasive — Not Persuasive
With Mod*.

#### Justification

A defensible reading is that this is a documentation problem, not a
spec defect: OAuth 2.0 already defines what "support" means, the
parameter-rejection question is fundamentally a deployment/policy
choice, and most real implementers have already converged on the
pragmatic behaviour described in the 2022 Zulip thread. Adding new
SHALLs about rejection (Disposition A) risks being a breaking change
to STU2 between minor releases, which the second commenter explicitly
warns against. A short non-normative note costs little, avoids
re-balloting, and lets the next major SMART revision (e.g. SMART
v3 / OAuth 2.1 alignment) handle the strict-enforcement question with
proper migration guidance.

---

### Recommendation

**Recommended disposition:** B

Disposition B gives the workgroup what the reporter actually asked for
— an unambiguous, normative answer for each of the three parameters —
without the disruption risk of Disposition A's blanket
rejection-on-missing rules and without the unsatisfying punt of
Disposition C. Crucially, it grounds the answer in the *threat model*
each parameter addresses (`aud` → server-protective → MUST reject;
PKCE/`state` → app-protective → MUST support but not MUST-reject),
which is the principle the reporter, Josh Mandel, and Stephen MacVicar
all converged on independently in 2022. The SHOULD on scope-keyed PKCE
enforcement preserves Inferno's testability story and lines SMART up
cleanly with OAuth 2.1's PKCE-for-public-clients direction without
forcing today's mixed v1/v2 endpoints to break thousands of legacy
apps. FHIR-I should take this to the General FHIR-I call (per Isaac's
2025-10-08 comment) and route the resulting normative text through the
SMART App Launch errata / next-ballot cycle.
