## Summary

This Change Request, raised by Bas van den Heuvel against FHIR Core, questions
whether the FHIRPath **Terminology Service API** — added to the FHIRPath
specification in R5 — has seen sufficient real-world implementation experience
to justify advancing it to **normative** status. The reporter explicitly asks
the FHIR Infrastructure work group to consider whether that section should
instead be **spun out into a separate Implementation Guide** rather than
locked down as part of the normative FHIRPath surface. The cross-reference
index returned no linked Jira tickets, no Zulip threads tagged to this issue,
and no GitHub items, so this ticket currently stands on its own as a ballot /
scope concern rather than a defect with downstream artifacts.

## Details

**Description:**

> The terminology page is a useful extension of FHIRpath that was added in R5.
> Is it used widely enough to justify moving it to normative? Please consider

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the **Terminology Service API** section from the normative scope of
the FHIRPath specification for the upcoming ballot and migrate the
existing content into a **new, trial-use Implementation Guide** owned by
FHIR Infrastructure (e.g., *FHIRPath Terminology Service Extension IG*).
The FHIRPath core specification would retain only a non-normative pointer
to that IG. The relocated material would include the
`%terminologies`-style functions (`expand`, `lookup`, `validateVS`,
`subsumes`, `translate`, etc.) and their parameter and return-value
contracts.

#### Justification

The reporter's concern is fundamentally about HL7 normative criteria:
normative content should reflect proven, widely-implemented behaviour and
be stable for the long term. The Terminology Service API was added in R5
and the available evidence (no cross-references, only sparse implementer
chatter on Zulip) does not demonstrate the breadth of independent
implementations typically expected for normative status. Splitting it out
to a trial-use IG preserves the design work, lets the API continue to
mature against connectathon and real-world feedback, and avoids locking
in a contract that may need breaking changes. It also matches the
established HL7 pattern of incubating cross-cutting service contracts in
their own IGs (cf. the standalone Terminology Service IG itself).

---

### Disposition B: Alternative Approach

#### Proposal

Keep the Terminology Service API **inside the FHIRPath specification**
but explicitly mark that section as **Trial Use / FMM lower than the
surrounding normative content**, while ballotting the rest of FHIRPath as
normative. Add a short "Maturity" callout at the top of the Terminology
Service API section identifying the exact functions that are TU, the
criteria that would have to be met for them to become normative, and a
target ballot cycle. No content is moved; only the maturity label and
introductory text change.

#### Justification

This addresses the reporter's substantive concern — that the API is not
yet ready to be locked down — without the editorial cost of standing up a
separate IG, re-homing canonical URLs, or fragmenting where implementers
look for FHIRPath features. Mixed-maturity normative specifications are
already established practice in FHIR Core (where individual resources
carry their own FMM levels independent of the surrounding infrastructure).
It keeps a single, discoverable home for FHIRPath while honestly
signalling that the terminology functions are still maturing. The risk is
that mixed-maturity FHIRPath is more confusing to implementers than a
clean split, and that "trial use within a normative spec" carries less
visibility than a separately-versioned IG.

---

### Disposition C: Decline

#### Proposal

Decline the change. Retain the FHIRPath Terminology Service API in the
normative ballot of FHIRPath as currently scoped, on the grounds that
(a) the API has been in the published FHIRPath specification since R5
without substantive change requests against its contract, (b) the
functions are thin, declarative wrappers over the existing normative
Terminology Service operations (`$expand`, `$lookup`, `$validate-code`,
`$subsumes`, `$translate`), and (c) advancing them to normative is a
prerequisite for downstream normative content (profiles, IGs, validators)
that already invoke them.

#### Justification

The Terminology Service API is largely a FHIRPath-flavoured surface over
operations that are themselves already normative. The semantic and wire
contracts those FHIRPath functions delegate to are stable, so the
incremental normative risk is limited to the FHIRPath binding itself.
Declining defends the predictability of the ballot timeline and avoids
re-litigating a feature that has been in published specifications for one
full release cycle without a concrete defect being filed. The weakness of
this position is that "no defects filed" is not the same as "broad
independent implementation evidence", which is the question the reporter
actually raised.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible middle ground. The reporter's
concern — insufficient implementation evidence to justify normative
lock-in — is real and is not refuted by the Zulip record (which shows
design activity but little implementer dialogue). At the same time, the
disruption of spinning up a new IG (Disposition A) is significant for a
feature that is conceptually a thin FHIRPath surface over already-normative
terminology operations, and would fragment where implementers look. A
mixed-maturity FHIRPath spec — normative core plus an explicitly Trial
Use Terminology Service API section with documented criteria for
promotion — preserves the editorial cohesion of FHIRPath, gives the
work group a clean answer to the "is this ready?" question, and leaves
the door open to either promote the section in a future cycle or, if
implementation evidence still does not materialise, to migrate it to a
standalone IG (Disposition A) at that point. The work group should be
asked specifically to confirm the FMM level and the promotion criteria
for the Terminology Service API section.
