## Summary

The ticket asks the FHIR Infrastructure work group to clarify the rules
governing how a "compliant" implementation may deviate from a published
`CompartmentDefinition`. The current narrative on the
`CompartmentDefinition` resource states that only HL7 International may
publish compartment definitions, and notes that servers "typically do not
support the full definition of a compartment, and are not required to,"
but does not specify the boundaries of that latitude. The reporter
documents two assertions raised in prior discussion: (1) partial
implementation is intended to let a server still claim a compartment when
it does not support every resource type referenced by the definition
(e.g., a Patient compartment without `Account`); and (2) some
implementations (e.g., façades) cannot realistically implement the full
definition even when all the underlying resource types are supported.
Neither rationale is stated in the spec, nor is it clear whether
implementations may legitimately do *more* than the published definition
(though the resource text appears to imply this is allowed).

The reporter requests explicit guidance on six specific deviation
scenarios:

1. Not implementing a compartment at all, even though the base
   (anchor) resource is supported.
2. Omitting a listed resource type from the compartment, even though
   that resource type is supported.
3. Not using a listed search parameter to determine compartment
   membership, when the parameter is supported on the resource directly.
4. Using a search parameter (or other mechanism) **not** listed in the
   compartment definition to determine membership.
5. Including a resource type **not** listed in the compartment
   definition.
6. Excluding a specific resource instance that would otherwise be in
   the compartment (for reasons other than security).

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with this ticket in FHIR Augury, so the ticket stands
on its own and the resolution must be driven from the existing
`CompartmentDefinition` narrative and the work group's collective
experience.

## Details

**Description:**

> - At present, compartment definitions can *only* be defined by HL7
>   International. This is because their existence creates significant
>   impact on the behavior of servers.
> - Although compartment definitions can be a useful foundation for
>   access control, they may not be sufficient for this purpose.
>   Resources outside the compartment might still be appropriate to be
>   made available or subject to compartment restrictions even though
>   they are not formally part of the compartment (see below about
>   access rules).
>
> ...
>
> The full definitions of these compartments are published as
> `CompartmentDefinition` resources. Servers typically do not support
> the full definition of a compartment, and are not required to. Systems
> may publish `CompartmentDefinition` resources so that other systems
> may make use of compartments properly.

The description of `CompartmentDefinition` indicates that while only HL7
may define `CompartmentDefinition`s, implementations are not bound to
that definition (although there is no computable way of expressing what
they do implement, since only HL7 may define `CompartmentDefinition`s).

In discussion, there was an assertion that the ability to implement
less than a full compartment definition is to allow a server to still
support a compartment even if a particular resource type is not
supported by the system. (E.g., a system that implements the Patient
compartment doesn't need to include `Account` in the compartment if it
doesn't generally support `Account`.) However, this intent is not clear
in the text, and it isn't clear if there may be other reasons for
supporting less than the full definition. It also isn't clear if
supporting more than the definition is permitted, although there is
text in the resource that implies that.

There was also an assertion that it may not be feasible for some
systems (e.g., façades) to implement the full definition, even if all
resources are supported outside the compartment.

Clarify under what circumstances a compliant implementation is
permitted:

1. To not implement a compartment when the base resource is supported.
2. To not include in a compartment a resource type listed in that
   compartment definition, when the resource type is supported.
3. To not use a search parameter listed in the definition to determine
   membership of a resource in a compartment, when the search
   parameter is supported on the resource directly.
4. To use a search parameter not included in the compartment
   definition, or another method, to determine membership of a
   resource in a compartment.
5. To include a resource type not included in the compartment
   definition, in a compartment.
6. To not include a resource instance in the compartment when it
   otherwise would be expected in the compartment (other than
   "security").

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new "Conformance to compartment definitions" subsection to the
`CompartmentDefinition` resource narrative (and a brief mirror note in
the compartments page) that addresses each of the six scenarios the
ticket lists, in order, with a clear MAY / SHALL NOT / SHOULD-style
verdict for every one. Concretely:

1. **Omitting a compartment entirely.** A server MAY choose not to
   support a published compartment even when the anchor resource type
   (e.g., `Patient`) is supported. Non-support is signalled by the
   absence of a `CapabilityStatement.rest.compartment` entry; clients
   MUST NOT infer compartment support from anchor-resource support.
2. **Omitting a listed resource type from a supported compartment.** A
   server MAY omit a resource type from its implementation of an
   otherwise supported compartment, regardless of whether that
   resource type is supported elsewhere by the server. The server
   SHALL document the omission (the recommended mechanism is an
   extension on `CapabilityStatement.rest.compartment` listing
   `excluded` and/or `included` resource types — to be defined as
   part of this change).
3. **Using a different search parameter than the one listed.** A
   server MAY determine membership using a different supported search
   parameter than the one named in the published definition, provided
   the resulting membership semantics match the definition's intent.
   This SHOULD be disclosed in the compartment-support extension.
4. **Using a non-listed mechanism (search parameter or otherwise).**
   Permitted under the same rule as (3). Servers SHOULD prefer
   listed parameters when supported.
5. **Including a non-listed resource type in the compartment.** A
   server MAY extend the compartment with additional resource types,
   provided the additions are signalled in
   `CapabilityStatement.rest.compartment` (via the same extension as
   (2)). Clients SHALL NOT assume such extensions are portable across
   servers.
6. **Excluding individual resource instances.** A server MAY exclude
   specific instances for reasons beyond security (e.g., business
   rules, lifecycle state). The presence of such exclusions SHOULD
   be flagged in the compartment-support extension as
   "instance-level filtering applies"; clients MUST treat
   compartment results as a server-defined view, not a complete one.

The change is a narrative + small `CapabilityStatement` extension
addition; no breaking change to `CompartmentDefinition` itself.

#### Justification

The ticket is explicitly a clarification request, and the six scenarios
are well-formed. Answering them in the spec rather than leaving them to
list-discussion folklore is consistent with FHIR's pattern of making
conformance verbs explicit. Tying each "MAY deviate" to a disclosure
extension on `CapabilityStatement.rest.compartment` addresses the
ticket's observation that "there is no computable way of expressing
what they do implement" while preserving HL7's exclusive authorship of
the definitions themselves.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than enumerate per-scenario MAY/SHOULD/SHALL rules in the
narrative, redefine the conformance contract in two layers:

- **Definitional layer (HL7-only):** the published
  `CompartmentDefinition` describes the *intended* membership rules.
  Wording is tightened to make clear it is a *target*, not a
  conformance baseline.
- **Implementation layer (per-server):** servers declare their
  *effective* compartment via a structured extension on
  `CapabilityStatement.rest.compartment` that contains a (possibly
  pruned, possibly augmented) list of resource type / parameter
  mappings, plus a free-text `deviationNote`. A server is "compliant
  with the Patient compartment" iff its declared effective compartment
  is a structurally valid `CompartmentDefinition`-shaped object
  derived from the published one (resource types may be subset or
  superset; parameters may be substituted; instance-level exclusions
  declared by a boolean flag).

This collapses scenarios 1–6 into a single rule: "compliance is
declarative — what you publish in your CapabilityStatement is what
you must honour, and the published HL7 definition is the recommended
starting point, not the bar."

#### Justification

This trades a longer narrative for a single, computable model and
neatly addresses the reporter's concern about the lack of a
computable way to express what an implementation actually does.
Façade servers and partial implementations get a first-class place to
declare reality. The cost is a slightly larger
`CapabilityStatement` surface and the need to design (and ballot) the
new extension; the benefit is that downstream tooling (validators,
test harnesses, conformance dashboards) gets something it can act on
instead of prose. Downside vs. A: heavier change, requires extension
design and review across IGs that already use compartments (e.g.,
US Core, IPS).

---

### Disposition C: Decline

#### Proposal

Resolve as "Not Persuasive — already addressed." Take the position
that the existing `CompartmentDefinition` narrative ("Servers
typically do not support the full definition of a compartment, and
are not required to") combined with the general FHIR conformance
principle that servers declare what they support in
`CapabilityStatement` already covers all six scenarios, and that the
appropriate fix is editorial only (e.g., a one-line cross-reference
from `CompartmentDefinition` to `CapabilityStatement.rest.compartment`).

#### Justification

The current text does, in a thin way, permit deviation. A defensible
reading is that any deviation is allowed so long as it is disclosed
via `CapabilityStatement`. Choosing this disposition keeps the
specification surface small and avoids re-litigating compartment
semantics in this cycle.

The weakness — and the reason this disposition is harder to justify
— is that the reporter is plainly correct that current text is silent
on five of the six scenarios, and an "it's already there if you read
between the lines" answer is exactly the kind of response that
generates more tickets, not fewer.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the lowest-risk path that actually answers the
question the ticket asks. It matches the ticket's structure
(scenario-by-scenario), produces something a server author can
unambiguously check their implementation against, and pairs each
"MAY deviate" with a disclosure obligation so that downstream
clients aren't left guessing. Disposition B is more elegant
long-term but carries a larger design and ballot burden — it should
be considered as a follow-up, not as the response to a clarification
ticket. Disposition C understates the real ambiguity in today's
text and is likely to produce more tickets of the same shape. Start
with A; if the work group is interested in the declarative model, B
can be raised as a separate ticket against the next ballot cycle.
