## Summary

This is a follow-up / meta-ticket raised by Bas van den Heuvel after the
R6 ballot, asserting that eleven previously-filed signature tickets
(FHIR-52810 through FHIR-52820) are either not yet resolved or — where
they *were* resolved by Security / FHIR-I — do not appear to have been
actually applied to the R6 ballot text. The description is a table
listing each source ticket, its Jira-recorded resolution, and the
reporter's field observation of whether the R6 ballot version reflects
that resolution; the reporter marks all eleven with "Updated? = N".

The underlying concerns cluster into four themes:

- **Fragment identifiers for signed Bundles** — [FHIR-52810](https://jira.hl7.org/browse/FHIR-52810) (Applied / Persuasive-with-Modification in Security: agreed to replace the magic `Bundle/[id]` fragment with pointer-based fragments `#/`), [FHIR-52811](https://jira.hl7.org/browse/FHIR-52811) (still unresolved; `#` semantics in Provenance inside a Bundle remain unclear), [FHIR-52813](https://jira.hl7.org/browse/FHIR-52813) (Applied / Persuasive: countersigning rules re-expressed in terms of `#/`), and [FHIR-52814](https://jira.hl7.org/browse/FHIR-52814) (unresolved; need canonical-rules pointer from document rules into §6.1.2.2.9).
- **Canonicalization / meta behavior** — [FHIR-52812](https://jira.hl7.org/browse/FHIR-52812) (unresolved; whether additional canonical rules are needed when signing the whole Bundle) and [FHIR-52815](https://jira.hl7.org/browse/FHIR-52815) (unresolved; `meta` behavior for `#static` signatures must change, i.e. strip the whole `meta` element).
- **Terminology / serialization clarity** — [FHIR-52816](https://jira.hl7.org/browse/FHIR-52816) (unresolved; the term "Electronic Signature" is confusing) and [FHIR-52817](https://jira.hl7.org/browse/FHIR-52817) (unresolved; calling the serialization "base64" is imprecise because the payload is a JWS/JAdES compact form that is itself base64 segmented before being base64-wrapped again).
- **Advanced-signature profile recommendations** — [FHIR-52818](https://jira.hl7.org/browse/FHIR-52818) (Applied / Persuasive-with-Modification: "MAY conform to JAdES-B-T … SHOULD conform to JAdES-B-LTA for long-term"), [FHIR-52819](https://jira.hl7.org/browse/FHIR-52819) (Applied / Persuasive-with-Modification: equivalent XAdES-B-T / XAdES-B-LTA language), and [FHIR-52820](https://jira.hl7.org/browse/FHIR-52820) (unresolved; add JAdES-B-T / JAdES-B-LT examples).

The ask is therefore two-fold: (1) close out the seven still-Triaged
tickets so R6 actually takes a position on each, and (2) verify that
the four tickets already marked Applied by Security have in fact landed
in the R6 ballot text, because the reporter's sampling of §6.1.2.2.x
suggests the agreed edits were never committed.

## Related Jira Tickets

### [FHIR-52810: Signature on Bundle should not use id](https://jira.hl7.org/browse/FHIR-52810)

Filed by Bas van den Heuvel against Security. Applied /
Persuasive-with-Modification: the WG agreed that a signed Bundle should
not rely on a "magic" `Bundle/[id]` fragment (because the document ID
can change on persistence) and should instead reference
pointer-based fragments — specifically the fragment `#/` — so the
target of the embedded signature is unambiguous. Page rules and worked
examples in the Bundle/Provenance signature section were to be updated.
FHIR-53810 alleges this edit never actually landed in the R6 ballot.

*Users involved:* Bas van den Heuvel (reporter), Security WG triagers.

### [FHIR-52811: The use of `#` in Provenance in Bundles is unclear](https://jira.hl7.org/browse/FHIR-52811)

Filed against FHIR Infrastructure. Still Triaged / Unresolved. The
reporter's reading of §6.1.2.2.9 is that a Provenance embedded in a
Bundle needs *two* `Provenance.target` entries — one to `Bundle/[id]`
and one to `#` — to express "signed both the Bundle and the
Provenance," and that this should be stated explicitly (and is
entangled with the FHIR-52810 pointer-fragment decision).

*Users involved:* Bas van den Heuvel (reporter), FHIR-I triagers.

### [FHIR-52812: Do we need additional canonical rules for signing the whole bundle](https://jira.hl7.org/browse/FHIR-52812)

Filed against FHIR Infrastructure. Still Triaged / Unresolved. Asks
whether a whole-Bundle signature requires canonicalization beyond the
Signature datatype's existing rules. The reporter notes its disposition
depends on FHIR-52811; if `#/` semantics are clarified there, this may
fall out.

*Users involved:* Bas van den Heuvel (reporter), FHIR-I triagers.

### [FHIR-52813: Countersigning not completely clear](https://jira.hl7.org/browse/FHIR-52813)

Filed against Security. Applied / Persuasive. Agreed wording:
"Provenance SHALL have a `Provenance.signature` element containing the
signature of the Bundle, with all entries that contain embedded signing
Provenance resources removed (i.e. any entry whose Provenance.target
references `#/`). Counter-signing signatures will include additional
targets, but one of the targets is always `#/`." The reporter asserts
the R6 ballot text still matches the original (pre-agreement) wording
and that `#/` does not appear.

*Users involved:* Bas van den Heuvel (reporter), Security WG triagers.

### [FHIR-52814: Update canonical rules for documents](https://jira.hl7.org/browse/FHIR-52814)

Filed against FHIR Infrastructure. Still Triaged / Unresolved. Asks
that the document/Bundle canonical rules cross-reference §6.1.2.2.9 (or
that §6.1.2.2.9 cross-reference them) so implementers signing a
Document Bundle are not surprised. The reporter's remark notes
§6.1.2.2.9 itself is unclear today.

*Users involved:* Bas van den Heuvel (reporter), FHIR-I triagers.

### [FHIR-52815: Change meta behavior for Signatures](https://jira.hl7.org/browse/FHIR-52815)

Filed against FHIR Infrastructure. Still Triaged / Unresolved. Asks to
remove the whole `meta` element (not just `meta.versionId`/`lastUpdated`)
from the `#static` canonical form in §6.1.2.2.4, because
implementation-controlled `meta` tags can vary between recipients and
break signature verification.

*Users involved:* Bas van den Heuvel (reporter), FHIR-I triagers.

### [FHIR-52816: The term "Electronic Signature" is confusing](https://jira.hl7.org/browse/FHIR-52816)

Filed against FHIR Infrastructure. Still Triaged / Unresolved. The
reporter and commenters note the phrase "electronic signature" is used
inconsistently (sometimes meaning any digital signature, sometimes a
legally-weighted eIDAS-style signature) and should be replaced or
qualified throughout the signature narrative.

*Users involved:* Bas van den Heuvel (reporter), FHIR-I triagers.

### [FHIR-52817: Defining serialization as base64 is imprecise](https://jira.hl7.org/browse/FHIR-52817)

Filed against FHIR Infrastructure. Still Triaged / Unresolved. Through
a comment exchange with Eric Haas, the reporter clarifies that the
intended serialization is a JWS/JAdES compact form (which is itself
three base64url-encoded segments joined by `.`), and that this compact
string is *then* base64-encoded into `Signature.data`. The spec
currently says only "base64", which hides the JWS layer and leaves
distinguishing JWS / JAdES / JAdES-B-T / JAdES-B-LTA ambiguous.

*Users involved:* Bas van den Heuvel (reporter), Eric Haas, FHIR-I
triagers.

### [FHIR-52818: Recommend JAdES-B-T for non-repudiation](https://jira.hl7.org/browse/FHIR-52818)

Filed against Security. Applied / Persuasive-with-Modification. Agreed
wording: "The Signature MAY conform to JAdES-B-T or any other profiling
of JOSE-Signature. If more than integrity is required, the JAdES
specification provides a set of profiles, and implementers should
consider the rules found in those profiles carefully. For long-term
signatures, implementers SHOULD conform to JAdES-B-LTA." Reporter says
this was not carried into the R6 ballot text.

*Users involved:* Bas van den Heuvel (reporter), Security WG triagers.

### [FHIR-52819: Recommending XAdES-T for non-repudiation](https://jira.hl7.org/browse/FHIR-52819)

Filed against Security. Applied / Persuasive-with-Modification. Direct
XML-signature parallel to FHIR-52818: MAY conform to XAdES-B-T or any
other profiling of XMLDigSig; SHOULD conform to XAdES-B-LTA for
long-term. Reporter says not carried into R6 ballot.

*Users involved:* Bas van den Heuvel (reporter), Security WG triagers.

### [FHIR-52820: Add additional example of JAdES-B-T (and JAdES-B-LT)](https://jira.hl7.org/browse/FHIR-52820)

Filed against FHIR Infrastructure. Still Triaged / Unresolved. Minor
request to add a worked JAdES-B-T (and ideally JAdES-B-LT) example to
the signature worked-examples page, paired with FHIR-52818.

*Users involved:* Bas van den Heuvel (reporter), FHIR-I triagers.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-53810 as the tracking umbrella for getting the signature
story correct in R6, and take the following concrete actions per
source ticket:

1. **Re-open and actually apply the four already-agreed changes** that
   were marked Applied but, per the reporter's sampling, are not
   visible in the R6 ballot:
   - FHIR-52810: rewrite the Bundle-signature page-rules and worked
     examples to use the pointer fragment `#/` (and equivalent JSON
     Pointer sub-paths for targeted sub-resources) instead of the
     `Bundle/[id]` form.
   - FHIR-52813: replace the countersigning bullet in §6.1.2.2.x with
     the `#/`-based wording agreed on the ticket; ensure the phrase
     "one of the targets is always `#/`" appears.
   - FHIR-52818: add the "MAY conform to JAdES-B-T … SHOULD conform to
     JAdES-B-LTA for long-term signatures" sentence to the JSON /
     JOSE-Signature narrative.
   - FHIR-52819: add the equivalent XAdES-B-T / XAdES-B-LTA sentence to
     the XML / XMLDigSig narrative.
2. **Resolve the seven still-Triaged tickets** before R6 normative
   publication:
   - FHIR-52811: state explicitly how many `Provenance.target` entries
     are required to sign both the Bundle and the embedded Provenance,
     and which value(s) of `#`/`#/` apply.
   - FHIR-52812: either add the specific additional canonical rules, or
     close as "handled by FHIR-52811 / Signature canonical rules."
   - FHIR-52814: add a forward/back cross-reference between the
     document canonical rules and §6.1.2.2.9.
   - FHIR-52815: remove the entire `meta` element from the `#static`
     canonical form in §6.1.2.2.4.
   - FHIR-52816: rename / disambiguate "electronic signature" uses
     (prefer "digital signature" for the cryptographic act and reserve
     "electronic signature" for the legal/eIDAS sense where it is
     actually meant).
   - FHIR-52817: describe the serialization as "a JWS or XML-DSig
     compact form that is base64-encoded into `Signature.data`", show
     the three-segment `header.payload.signature` structure, and name
     how to distinguish JWS / JAdES / JAdES-B-T / JAdES-B-LTA.
   - FHIR-52820: add a worked JAdES-B-T (and ideally JAdES-B-LT)
     example alongside the existing signature examples.
3. Close FHIR-53810 itself once the above are reflected in the CI
   build.

#### Justification

Each of the eleven source tickets was individually triaged and, in four
cases, explicitly agreed; the current state is simply that the agreed
wording was either not committed or not carried through from the R5
disposition into the R6 branch. Honoring prior WG decisions is a
baseline expectation for ballot-comment hygiene. The still-Triaged
tickets are all small, self-contained editorial/normative
clarifications that cluster around the same four pages
(Signatures.html, Bundle.html, Provenance, and the document signing
narrative) and are therefore cheap to batch. Declining or deferring
them to R7 would leave signed-Bundle semantics materially ambiguous in
the first normative FHIR.

---

### Disposition B: Alternative Approach — consolidate as a cross-WG signature work-package

#### Proposal

Rather than treating FHIR-53810 as a request to re-action eleven
individual tickets one-by-one, convert it into a coordinated
**signature work-package** owned jointly by FHIR Infrastructure and
Security, with a single tracking epic and a single ballot-comment
PR against `HL7/fhir`. Specifically:

- Nominate a single editor (the assignee of this ticket once chosen)
  responsible for the Bundle/Provenance/Signature sections in R6.
- Create a child-of-epic link from FHIR-53810 to each of FHIR-52810
  … FHIR-52820, re-opening the four "Applied" tickets so they visibly
  block R6 normative.
- Produce one PR that:
  1. Rewrites §6.1.2.2.4 (`#static` canonical form) to remove `meta`
     entirely (covers FHIR-52815).
  2. Rewrites §6.1.2.2.9 using pointer fragments `#/` and states the
     target-cardinality rules for embedded vs. counter-signing
     Provenance (covers FHIR-52810, FHIR-52811, FHIR-52813,
     FHIR-52814).
  3. Rewrites the JSON / XML Signature narrative to describe
     Signature.data as "a base64-encoded JWS / XML-DSig compact form"
     and names JAdES-B-T / JAdES-B-LTA / XAdES-B-T / XAdES-B-LTA with
     the previously-agreed MAY/SHOULD levels (covers FHIR-52817,
     FHIR-52818, FHIR-52819).
  4. Replaces ambiguous "electronic signature" prose (covers
     FHIR-52816) and adds worked JAdES examples (covers FHIR-52820).
- Defer FHIR-52812 into the "covered by FHIR-52811" bucket unless the
  editor finds residual canonicalization work.

#### Justification

Eleven separate tickets that all touch three adjacent pages tend to
produce inconsistent edits and repeated re-balloting churn when worked
independently. The Applied/Triaged split across Security and FHIR-I
is also precisely why the R5 dispositions did not make it into the R6
ballot — neither WG owned the last-mile commit. A single epic with one
editor and one PR is the smallest change that fixes the coordination
failure the reporter is actually flagging, while still giving each
source ticket a traceable outcome. This approach is also friendlier to
ballot reviewers, who will see one signature-section PR rather than
eleven overlapping ones.

---

### Disposition C: Decline (close as duplicate / track individually)

#### Proposal

Close FHIR-53810 as a duplicate of the eleven underlying tickets it
enumerates. Reopen the four "Applied" tickets (FHIR-52810, FHIR-52813,
FHIR-52818, FHIR-52819) with a comment that their agreed wording must
still be committed, and leave the seven Triaged ones in their existing
WG queues. Do not carry FHIR-53810 itself as a work item.

#### Justification

A ballot comment that merely lists other ballot comments does not,
strictly speaking, request a new change to the specification — it
requests that previously-dispositioned changes be honored. Jira already
has first-class ticket state for that (re-open / status="Applied but
not yet in build"), and a tracking meta-ticket risks drifting out of
sync with the individual dispositions. Declining this ticket and
handling the substantive work through the original eleven is also the
cleanest audit trail for ballot reconciliation, because each substantive
edit is attributed to the ticket that actually raised it.

---

### Recommendation

**Recommended disposition:** B

Disposition A is correct in substance but underestimates why the agreed
edits from R5 failed to land in R6: there was no single owner for the
signature sections. Re-instructing both WGs to "actually apply" the
agreed text without changing that ownership pattern is likely to
produce the same result at the next ballot. Disposition C is
administratively tidy but throws away the one thing FHIR-53810 uniquely
provides, namely a single place where the reporter has already
cross-walked eleven tickets into a coherent to-do list for the R6
editor. Disposition B preserves that to-do list, assigns it, and turns
it into a single PR scoped to §6.1.2.2.x plus the signature narrative,
which is the minimum intervention that actually closes the loop for R6
normative publication. The workgroup should pick an editor in the
same meeting the disposition is taken, re-open the four "Applied"
tickets, and link all eleven source tickets as children of FHIR-53810.
