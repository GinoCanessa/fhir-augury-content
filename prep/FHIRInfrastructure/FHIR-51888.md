## Summary

The ticket asks FHIR Infrastructure to change the FHIR Core specification so
that the `Bundle`, `Parameters`, and `Binary` resources can carry standard
`Resource.extension` (and, by implication, `modifierExtension`) elements like
every other resource. Today these three resources derive directly from
`Resource` rather than `DomainResource`, so the normal FHIR extension
mechanism is not available on them. The reporter cites two concrete pain
points: needing to attach metadata to a `Bundle` instance, and the Bulk Data
IG using `Parameters` for Bulk Import where child IGs have had to invent
"janky" workarounds to add their own metadata. The reporter notes a partial
workaround for `Parameters` (defining a single parameter whose
`Parameters.parameter.resource` is a `Basic` resource and putting extensions
there) and observes that R4 backport / preadoption extensions cannot fix this
because the constraint is structural rather than a missing element
definition.

There are no explicitly-linked Jira tickets and no GitHub items cross-
referenced to this ticket. The only cross-reference in the index is a FHIR
artifact mention of `Parameters.parameter.resource` (from the reporter's own
comment about the workaround).

## Details

**Description:**

> Bundle, Binary and Parameters resources don't allow extensions. This has been problematic in at least two cases:
>
> - I've personally had useful metadata I wanted to include in Bundle.
> - The Bulk Data IG uses the Parameters resource for Bulk Import, and we had to use janky workarounds to allow child IGs to add their own metadata.
>
> FHIR defines a super useful, consistent way to extend FHIR resources (extensions). We should allow that feature to be used on all the FHIR resources, including Bundle and Parameters. And Binary while we are at it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-base `Bundle`, `Parameters`, and `Binary` so they support
`Resource.extension` and `Resource.modifierExtension` directly. Concretely,
in FHIR Core (the next normative cycle for these resources):

- Either reparent each of `Bundle`, `Parameters`, and `Binary` to
  `DomainResource`, or
- Add `extension : Extension [0..*]` and `modifierExtension : Extension
  [0..*]` directly to each resource definition (preserving the
  intentional omission of `text` for these "non-narrative" resources).

Update the publication tooling and validator so that:

1. Existing `Resource.meta.extension` workarounds (e.g., the
   `resourcename` / `resourcedescription` extensions discussed in the IG
   creation thread) keep working but are documented as legacy.
2. The XML/JSON serializers and the FHIRPath/element model accept the new
   element on these three resources.
3. `modifierExtension` on `Bundle`/`Parameters` is called out in the
   resource page with explicit guidance about how it interacts with
   `Bundle.entry.resource` (it modifies the `Bundle`, not the contained
   resources) and with `Parameters.parameter` semantics.

The Bulk Data IG and any other IG that has been using
`Parameters.parameter.resource = Basic` purely to attach metadata can be
updated in a follow-on ticket to use first-class extensions.

#### Justification

The "no extensions on `Bundle`/`Parameters`/`Binary`" rule is a structural
historical artefact, not a use-case-driven design choice. The IG-creation
Zulip thread shows the workgroup already accepts that the workaround
(extensions on `Resource.meta`) is a workaround. The Bulk Data note in the
ticket and the bulk-data Zulip thread show the workaround creating real
specification awkwardness for an HL7-published IG. Allowing extensions
restores symmetry across the resource hierarchy, removes the need for the
`Basic`-resource trick, and aligns with FHIR's foundational principle that
extensibility is a first-class mechanism on every resource.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** change the type hierarchy of `Bundle`, `Parameters`, or
`Binary`. Instead, formally bless and document a single sanctioned
extensibility pattern per resource:

- For `Bundle` and `Parameters`: standardise on
  `Resource.meta.extension` as the supported extension surface, and
  publish a small set of HL7-managed extensions that cover the common
  needs (e.g., human-readable name/description for examples, page
  tokens / paging metadata for Bulk Data).
- For `Binary`: continue to discourage extensions and direct callers to
  `DocumentReference` (or a wrapping `Bundle` / `Parameters`) when
  metadata is needed, since `Binary` is intentionally a thin wrapper
  around an opaque payload.

Add explicit normative guidance pages on each of the three resource pages
explaining what is allowed, what is not, and why.

#### Justification

`Bundle`, `Parameters`, and `Binary` were deliberately kept off the
`DomainResource` branch because they are infrastructure containers rather
than clinical/administrative resources. Adding `extension` directly is
non-trivial in practice: validators, schemas (XSD/JSON Schema), reference
implementations, generated bindings, terminology tooling, and the
serializer test suites all assume the current shape, and `modifierExtension`
on a `Bundle` raises genuinely hard semantics around what "modifies"
means for a `searchset` or a transaction. `Resource.meta.extension`
already works today (per the IG creation thread, this is exactly what
HL7 tooling extensions like `resourcename` use), so the cost/benefit of
the structural change may not be justified if the same need can be met by
a documented and supported pattern.

---

### Disposition C: Decline

#### Proposal

Decline the request. Document the existing patterns:

- For `Parameters`: use `Parameters.parameter.resource = Basic` (or a
  named domain resource) carrying the desired extensions, as the
  reporter already notes works.
- For `Bundle`: use `Bundle.meta.extension` for bundle-level metadata
  and put richer metadata on individual `Bundle.entry.resource`
  instances.
- For `Binary`: wrap in `DocumentReference` or include in a `Bundle`
  with sibling resources that carry metadata.

Close the ticket as "Persuasive with mod" in favour of the existing
mechanisms, and capture the patterns above as new IG-publishing guidance.

#### Justification

The reporter himself enumerates a working `Basic`-based workaround for
`Parameters`, and `meta.extension` already covers the `Bundle` case. The
restriction on these three resources is long-standing; reversing it has
ripple effects across every reference implementation, schema, validator,
and conversion layer in the ecosystem. If the only motivation is two IG
authors finding the workaround inelegant, that may not clear the bar for
a normative shape change to three core infrastructure resources.

---

### Recommendation

**Recommended disposition:** A

The Zulip evidence consistently shows that the current restriction is
treated as a workaround in waiting rather than a deliberate design: the
IG Publisher already needs HL7-managed extensions stuck onto
`Resource.meta` precisely because they cannot be placed on the resource
itself, and Bulk Data implementers have had to bend `Parameters` into
shapes it was not designed for. The reporter is from a major
implementer organisation and is calling out a real, recurring problem on
an HL7-published IG (Bulk Data), which is the strongest signal FHIR-I
gets that the workaround tax has compounded. Disposition B preserves
the status quo without solving the underlying asymmetry, and
Disposition C amounts to declining a pain point that has already been
documented in two HL7 IGs. Disposition A is invasive — and the
implementation work on schemas, validators, and serializers should not
be under-counted — but it is the only option that actually delivers what
implementers have been quietly building around for years. The workgroup
should sequence this carefully (likely as part of the next normative
cycle for `Bundle` and `Parameters`, with explicit handling guidance for
`modifierExtension` on container resources), but the direction is
correct.
