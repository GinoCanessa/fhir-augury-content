## Summary

The ticket requests a new standard extension on the `Attachment` data type that
carries a thumbnail / preview / low-resolution rendering of the parent
attachment as a nested `Attachment`. Today the only standard mechanism for
flagging "this is a thumbnail" is the
[`documentreference-thumbnail`](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-documentreference-thumbnail.html)
extension, which is a boolean flag on `DocumentReference.content` and therefore
only usable when the parent resource is `DocumentReference`. The proposed
extension would let any element typed `Attachment` (e.g.
`Patient.photo`, `Practitioner.photo`, `Person.photo`,
`DiagnosticReport.presentedForm`, `Media.content`,
`Communication.payload.contentAttachment`, plus the many BackboneElements that
carry an `Attachment`) bundle an inline / referenced low-resolution preview
alongside the full asset. Proposed shape: `id` = `thumbnail` / `preview` /
`low-resolution`, `value[x]` = `Attachment` (1..1).

This request is the latest cycle of a long-running discussion. The closest
prior ticket is [FHIR-14532](https://jira.hl7.org/browse/FHIR-14532)
("Add media thumbnails (or similar)", Orders & Observations, 2018), which was
resolved Not Persuasive on the basis that
`DocumentReference.content` repetitions plus size/height/width metadata could
distinguish a thumbnail from a full asset â€” a resolution that several FHIR-I
participants (notably Lloyd McKenzie, Gino Canessa, Morten Ernebjerg) pushed
back on at the time, arguing that thumbnails are semantically *not*
interchangeable representations of the same asset and so should not be
encoded as just another `content` repetition. That earlier discussion led to
the boolean `documentreference-thumbnail` extension as a partial fix scoped
solely to `DocumentReference`. FHIR-56002 generalises the same need to the
underlying `Attachment` datatype so that the boolean-flag-on-DocumentReference
workaround stops being the only standard answer.

## Details

**Description:**

> When an Attachment is large, there can be the desire to include a thumbnail
> / preview version. Currently, the only standard representation of a
> thumbnail is a flag on `DocumentReference.content`
> (https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-documentreference-thumbnail.html),
> which signifies that content is a thumbnail.
>
> Request defining a standard extension on `Attachment`:
> - Id: `thumbnail`, `preview`, or `low-resolution`
> - value[x]: 1..1 `Attachment`

## Related Zulip Discussions

### [implementers > Thumbnails in Attachments](https://chat.fhir.org/#narrow/stream/implementers/topic/Thumbnails%20in%20Attachments)

Long (36-message) thread that is the canonical prior discussion for this
request. Morten Ernebjerg opened it asking how to attach thumbnails to
scanned documents and listed three options: multiple `DocumentReference.content`
entries, separate `DocumentReference` resources linked via `relatesTo`, or a
custom extension on `Attachment`. Grahame Grieve recommended an extension
(suggesting the URL `http://hl7.org/fhir/StructureDefinition/iso21090-ED-thumbnail`
typed `Attachment`). John Moehrke argued the existing `.content` repetitions
plus size metadata are sufficient. Lloyd McKenzie made the principled case
that thumbnails are a *different purpose* (summary/preview) rather than an
*alternate rendering* of the same content, so they should be a distinct
construct, not just another `.content` row. Gino Canessa agreed that
overloading `.content` repetitions with semantically different uses
(formats, pages, thumbnails) without a discriminator forces implementers
to make value judgements from byte-size heuristics, which is the wrong
default. Eric Haas suggested an extension now and a possible
`content.thumbnail` element later. Morten created **FHIR-26556** to track,
which subsequently shipped as the boolean `documentreference-thumbnail`
flag on `DocumentReference.content` rather than as the typed-Attachment
extension Grahame originally suggested. The thread ends with John Moehrke
in 2022 questioning whether the eventual boolean-flag implementation
matches the original `iso21090` alignment intent â€” the same gap
FHIR-56002 is now reopening.

Participants: Morten Ernebjerg, Lloyd McKenzie, Grahame Grieve, John Moehrke,
Eric Haas, Gino Canessa, Bapi Behera.

Short FHIR-I follow-up (Nov 2021) where John Moehrke flagged that he was

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new standard extension in **HL7/fhir-extensions** (FHIR Extensions
Pack) named `attachment-thumbnail` with the following profile:

- **Context:** `Attachment` (the datatype itself, so it applies wherever
  `Attachment` is used â€” `Patient.photo`, `Practitioner.photo`,
  `Person.photo`, `RelatedPerson.photo`, `DocumentReference.content.attachment`,
  `DiagnosticReport.presentedForm`, `Media.content`,
  `Communication.payload.contentAttachment`,
  `CommunicationRequest.payload.contentAttachment`,
  `Questionnaire.item.initial.valueAttachment`,
  `QuestionnaireResponse.item.answer.valueAttachment`, etc.).
- **Cardinality:** 0..* (allow multiple thumbnails â€” e.g. small + medium
  preview â€” as Eric Haas suggested in the implementers thread).
- **Type:** `Attachment` (1..1).
- **Description:** A reduced-fidelity preview of the parent `Attachment`,
  intended for display purposes only and not interchangeable with the full
  content. Implementers SHOULD populate `contentType`, `width`/`height`,
  and `size` on the inner `Attachment`.
- **Comparison-with-existing-mechanism note** in the extension's
  description that explicitly references `documentreference-thumbnail` and
  explains when to use which: the existing boolean continues to be valid
  for "this whole `DocumentReference.content` row is the thumbnail"
  patterns; the new `attachment-thumbnail` extension is for "this single
  `Attachment` carries its own thumbnail inline" patterns and is the
  recommended approach for non-`DocumentReference` resources.

#### Justification

This is exactly the design Grahame Grieve proposed in 2020 (an extension
typed `Attachment`, modeled on `iso21090-ED-thumbnail`) and that Lloyd
McKenzie defended on principled grounds in the same thread. The only
reason it didn't ship at the time was that FHIR-26556 was scoped to
`DocumentReference` and the simpler boolean flag was easier to land. The
need has since recurred outside `DocumentReference` (the recent
Questionnaire-stream threads are the clearest evidence), and continuing
to push implementers toward `DocumentReference.content` repetitions
re-creates exactly the implementer-side problem Gino Canessa flagged in
the original thread: systems are forced to make heuristic value
judgements from byte-size metadata to decide which row is the "real"
content and which is the preview. Putting the extension on the
`Attachment` datatype is the smallest change that fixes the recurring
need everywhere `Attachment` is used, while leaving the existing
`documentreference-thumbnail` extension and `DocumentReference.content`
repetition pattern untouched for backward compatibility.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a single new extension, ship **two** coordinated changes,
both still in `HL7/fhir-extensions`:

1. A `Coding`-valued extension on `Attachment` named
   `attachment-purpose`, bound (extensible) to a small new value set â€”
   e.g. `original`, `thumbnail`, `preview`, `low-resolution`,
   `summary`, `transcoded` â€” that lets the publisher *label* what role
   each `Attachment` is playing without forcing a parent-child
   relationship.
2. A second extension `attachment-relatesTo` (`Reference(Attachment)` or
   identifier-based) so two sibling `Attachment` instances can declare
   "B is a thumbnail of A" when they live in different `.content`
   repetitions of the same parent (the case the existing
   `documentreference-thumbnail` boolean partially addresses).

The pair lets implementers express "this Attachment is a thumbnail" both
in the inline-bundling pattern that Disposition A targets and in the
multi-content-row pattern that John Moehrke advocated, without
overloading either mechanism with the other's semantics.

#### Justification

This approach addresses the deeper concern Gino Canessa, John Moehrke,
and Lloyd McKenzie circled around in the original Zulip thread â€”
namely, that the real interoperability gap is the *missing
discriminator* on `.content` repetitions, not just the missing
thumbnail slot. Labeling each `Attachment` with a `purpose` solves
"which one is the preview?" for every consumer in one move and is
forward-compatible with future categories (e.g. machine-readable vs.
human-readable transcoding). The trade-off vs. Disposition A is added
spec surface and a new value set to govern, plus implementers having to
inspect two extensions instead of one to find a thumbnail. It is also a
larger ask of FHIR-I review time than Disposition A and may be harder
to land in a single ballot cycle.

---

### Disposition C: Decline / Persuasive with Mod

#### Proposal

Decline the request as written and instead document the existing
mechanism as the FHIR-recommended pattern:

- For `DocumentReference`-rooted use cases, continue to use
  `documentreference-thumbnail` on the relevant `.content` repetition.
- For non-`DocumentReference` use cases (`Patient.photo`,
  Questionnaire-attachment uploads, `Media.content`,
  `DiagnosticReport.presentedForm`), guidance is to publish the
  thumbnail bytes via `Attachment.data` / `Attachment.url` on a
  separate-but-equal element (e.g. `Patient.photo` already has
  cardinality `0..*`, so multiple `Attachment`s can cohabit), and to
  rely on `contentType` / `width` / `height` / `size` to disambiguate.

#### Justification

This is essentially the FHIR-14532 disposition reasserted: the
`Attachment` datatype already carries enough metadata
(`contentType`, `size`, `width`/`height`, `language`) for a
sophisticated consumer to identify a low-resolution sibling, and any
parent element typed `0..* Attachment` can already hold a thumbnail
without spec change. Declining keeps the spec surface small, avoids
overlap with `documentreference-thumbnail`, and avoids re-litigating
a topic that the workgroup has touched at least three times. The
counter-argument â€” and it is real â€” is exactly the one Lloyd McKenzie
and Gino Canessa made in 2020: this forces every consumer to implement
heuristic disambiguation from byte-size metadata, which is precisely
the kind of implementer hostility FHIR usually tries to avoid.

---

### Recommendation

**Recommended disposition:** A â€” Accept As Requested.

The community thread history shows a clear, repeated implementer need
that the current `documentreference-thumbnail` boolean only partially
addresses, and the proposed shape is a near-verbatim restatement of the
extension Grahame Grieve recommended in 2020. The change lands cleanly
in `HL7/fhir-extensions` (no FHIR-core normative impact, no
backward-compatibility break with the existing
`documentreference-thumbnail` extension), it is scoped to a single new
extension definition, and it directly serves use cases that have surfaced
multiple times since the original FHIR-26556 fix shipped (notably the
Questionnaire-stream attachment-upload scenarios from 2024â€“2025).
Disposition B is intellectually attractive but is materially larger
work, introduces a new bound value set, and risks stalling on
governance over the `purpose` codes; it should be considered as a
follow-on if implementers come back asking for richer labelling.
Disposition C is hard to defend given the implementer pushback against
the equivalent 2019 disposition of FHIR-14532 â€” the workgroup has
already tried "use existing `.content` metadata" once, and the
community kept asking. Recommend FHIR-I accept FHIR-56002 with the
extension named `attachment-thumbnail`, cardinality `0..*`, value type
`Attachment`, contexted on the `Attachment` datatype, with the
extension's description explicitly cross-linking
`documentreference-thumbnail` so implementers know which to reach for.
