## Summary

The reporter proposes that the SMART Web Messaging (SWM) specification add a
new message/event in which a *client* (a SMART app or other companion
application) requests the *EHR host* to launch an app. Today SWM defines
messages that flow from the embedded app to the host (e.g., `ui.done`,
`scratchpad.*`, `ehr.open`-style requests), but the request explicitly asks
the workgroup to consider the inverse direction — a client-initiated
"please launch app X" event — and points at an external Google Doc capturing
a discussion of the underlying handshake protocol. No formal cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced for this ticket;
the request currently stands on its own and is in `Triaged` state with no
assignee.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new client-initiated message to the SMART Web Messaging specification
that allows an embedded SMART app to ask the EHR host to launch another
SMART app. Concretely:

- Define a new message `messageType` (e.g., `ehr.launch-app` or
  `app.launch-request`) in the SWM message catalogue.
- Payload fields would include at minimum: a target identifier for the app
  to launch (e.g., a registered `client_id` or a launch URL), an optional
  `launchContext` (patient, encounter, intent), and an optional
  `accompanyingData` envelope for opaque handoff data referenced in the
  Google Doc handshake discussion.
- Define the response message: success (with launched-app handle, where
  applicable), denied (host policy / user choice), or unsupported.
- Update the SWM IG narrative to describe the new bidirectional flow and
  add a worked example to the IG's examples page.
- Add conformance language so hosts can advertise support for the new
  message, since not every EHR will expose programmatic app-launching.

#### Justification

This directly fulfills the reporter's request and fills a real gap: SWM today
is largely app-to-host, and a sanctioned client-to-host launch message would
let workflow-oriented apps (e.g., a sidebar app spawning a focused
documentation app) compose without each vendor inventing a proprietary
mechanism. Modelling it as a normal SWM message preserves the existing
envelope, security boundary, and `messagingHandle` flow, so implementers do
not need a new transport. Aligning with the handshake discussion in the
linked Google Doc keeps SWM consistent with the broader SMART App Launch
patterns rather than diverging.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a brand-new SWM message, define the client-initiated
launch as a profile/usage of the existing SMART App Launch flow surfaced
*through* SWM. Specifically:

- Add a section to the SWM IG titled "Requesting another app launch" that
  specifies the embedded app SHALL request the launch by sending an SWM
  message whose payload simply carries a SMART App Launch URL (i.e., the
  host's launch endpoint plus required `iss`/`launch` parameters), and
  the host SHALL treat receipt as equivalent to the user clicking that
  launch URL inside the EHR.
- Reuse an existing generic message type (e.g., a `ui.*` navigation
  message) rather than minting a new top-level message.
- Leave authorization, scopes, and context entirely to the existing SMART
  App Launch flow; SWM is only the conduit that says "please initiate this
  launch."

#### Justification

This addresses the underlying need ("from inside an embedded app, cause the
EHR to launch another app") without duplicating launch-context, scope, or
authorization semantics that SMART App Launch already specifies. It keeps
SWM small and focused on UI-coordination messages and avoids two competing
launch surfaces in the SMART family. It is also less risky to implement
because hosts mostly already know how to honour a SMART launch URL; the
only new behaviour is "accept that URL via SWM instead of via a click."
The trade-off vs. Disposition A is reduced expressiveness — there is no
typed response carrying the launched app's handle back into the SWM
session — but that capability can be added later if implementer demand
materialises.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped, on the grounds that (a) the
specification is *SMART Web Messaging*, whose charter is in-session
coordination between an already-launched app and its host, not host
control-plane operations, and (b) the existing SMART App Launch flow
already provides a well-understood mechanism for an EHR to launch an app.
Suggest that a SMART app needing to spawn a sibling app instead navigate
the user (or open a new window) to the host's published launch URL,
which exercises the existing, secured launch path. Close as
"Persuasive with Modifier — not adopting; existing mechanism applies"
unless concrete implementer demand and a fleshed-out proposal arrive.

#### Justification

The ticket is sparse: it links a Google Doc (which is not part of the
balloted record), has no use cases attached, no implementer support
expressed in comments, no Zulip discussion, and no associated GitHub
artifacts. SWM is a small, focused IG and adding a new control-plane
message — particularly one whose security model overlaps SMART App
Launch — risks both confusion and a non-trivial conformance burden on
hosts. Without evidence of demand or a concrete handshake design carried
into the ticket itself, the safer default is to lean on the existing
SMART App Launch flow.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the legitimate need raised by the reporter — letting
an embedded SMART app trigger the launch of another app — while avoiding
the cost and risk of duplicating SMART App Launch's authorization and
context-passing semantics inside SWM. It keeps SWM aligned with its
in-session-coordination charter, leverages a security flow that
implementers and hosts already understand, and is small enough to specify
with a section and one example. Disposition A is attractive if the
workgroup believes a typed, response-bearing SWM message materially
improves orchestration, but the current ticket does not carry the
evidence (use cases, implementer support, fleshed-out handshake) needed
to justify that scope. Disposition C is defensible given the thinness of
the ticket, but outright decline forfeits the small, low-risk
clarification that B offers and tends to push vendors toward proprietary
solutions. The workgroup should ask the reporter to confirm whether the
"handshake" in the Google Doc requires a typed response — if yes, escalate
to A; if no, B is sufficient.
