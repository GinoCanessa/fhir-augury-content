## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as an editorial sweep ticket against FHIR Core. Before the
next ballot/publication cut, do a full-text scan of the published
narrative pages for the literal markers `TODO`, `TBD`, and `???`, plus
section headings (`<h2>`/`<h3>`/`<h4>`) whose body is empty or
whitespace-only. For each hit, take one of three actions: (1) write the
missing content, (2) delete the heading/marker if it is no longer
relevant, or (3) replace it with an explicit "Reserved for a future
version" note. Track the resulting list of pages in a short tracker
(e.g., a checklist on this ticket or a child task per page) so progress
is visible, and gate publication on the tracker reaching zero open
items. Coordinate with FMG so that any non-trivial content additions
go through the appropriate work group rather than being silently
written by the editor.

#### Justification

This matches the literal request and is consistent with normal
publication-quality expectations for a normative spec: a reader
landing on a published page should never be confronted with an editor
note like `TODO` or an empty section. The sweep is mechanical (a grep
over the rendered/HTML output catches almost all instances) and the
remediation per hit is small. It is also consistent with FHIR's stated
publication discipline that "the build is the spec" — markers that
slip through are by definition bugs in the spec, not in the build.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than handle this as a one-off cleanup ticket, codify it as a
publication gate. Add (or extend) a publication-time check in the FHIR
Core publisher / QA tooling that fails the build — or at least raises
a high-severity warning surfaced in `qa.html` — when any narrative
page contains `TODO`, `TBD`, `???`, or an empty subsection in the
rendered output. Backfill an initial scrub of the existing pages so
the build passes once the check is enabled, and from then on treat
any future occurrence as a regression that must be fixed before merge.
File this ticket's specific cleanup as the first run of the new check.

#### Justification

A point-in-time cleanup will regress: editors routinely use `TODO` as
an in-progress marker, and without an automated gate the same content
will reappear in the next cycle. A publisher-level check makes the
problem self-policing and turns it from a recurring editorial chore
into a build invariant, which is the same pattern used elsewhere in
FHIR Infrastructure (e.g., link checking, FHIR-path validation,
example validation). The trade-off vs. Disposition A is up-front
tooling work in exchange for not having to re-open this ticket every
ballot cycle.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" or "Works as Designed" on the grounds that
the ticket as filed is not actionable — it identifies no specific
page, section, or marker — and that ad-hoc editorial polish is
already the implicit responsibility of every work group on its own
pages. Encourage the reporter to file targeted tickets per page or
per offending section so the right work group can resolve each one in
context.

#### Justification

The ticket as written is a generic quality complaint without a list
of affected pages or any pointer to where the problem is most acute.
Acting on it without that scope means an editor (or FHIR-I
specifically) has to either guess at intent or do the discovery work
themselves, which then risks editing pages owned by other work
groups. Declining and asking for targeted tickets routes the work to
the right owners. This disposition is defensible only if the
work group genuinely believes targeted tickets will be filed; if not,
the placeholder content stays in the spec.

---

### Recommendation

**Recommended disposition:** B

The literal request (Disposition A) is reasonable and should be done,
but on its own it is not durable: `TODO`/`TBD`/`???` markers and empty
sections are exactly the kind of artifact that creeps back into a
multi-author spec between every publication. Disposition B does the
same cleanup as a side effect (the check has to pass before the build
is green) and then prevents the regression by adding it to the
existing FHIR Core publication QA. That is a much better fit for FHIR
Infrastructure's role than committing the work group to repeating the
sweep every cycle. Disposition C should only be considered if FHIR-I
has no appetite to own either the sweep or the tooling, and even then
it leaves known-bad content in the published spec, which is hard to
defend.
