## Summary

This ticket is one of the auto-imported "spelling/abbreviation issues"
technical corrections raised by Lloyd McKenzie against the FHIR Core

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the editorial fix exactly as filed: in the narrative source for
the `compartmentdefinition` page in the FHIR Core spec, locate every
occurrence of the contraction `doesn't` and replace it with `does not`.
Re-publish the page so the published HTML at
`https://hl7.org/fhir/compartmentdefinition.html` no longer contains
the contraction. No structural, schema, or normative change is
involved; this is a pure prose correction. Mark the ticket
`Resolved – Persuasive` and close once the replacement is on the
build.

#### Justification

This matches the long-standing FHIR publication-style convention that
contractions and unexpanded abbreviations are avoided in normative
narrative. The fix is local to a single page, has no semantic impact,
and carries no implementation risk. The reporter is the editor and has
filed effectively-identical "spelling/abbreviation" corrections against
many other resource pages, all of which have been accepted as filed;
treating this one consistently keeps the corpus uniform.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the request but address it editorially as part of
a batch sweep rather than a one-off page edit. Concretely: (a) make the
prose change to `compartmentdefinition` as in Disposition A, **and** (b)
in the same editorial pass, run a repo-wide grep for the same forbidden
contractions (`doesn't`, `don't`, `won't`, `can't`, `isn't`, `aren't`,
`it's` used as a contraction, etc.) across all resource narrative pages
under the spec source tree, and fix any other hits in the same commit
or PR. Track the sweep in this ticket (or a parent "editorial sweep"
ticket) and close the page-specific imported tickets — including this
one — as duplicates of the sweep.

#### Justification

The auto-imported "page comment 125" tickets arrive one page at a time
but stem from a single editorial rule. Fixing them page-by-page
generates churn (one PR per page, one resolution per page) without
materially changing reviewer effort. A single sweep PR is cheaper to
review, easier to verify with a follow-up grep, and reduces the chance
that the same contraction is reintroduced elsewhere before the next
ballot. The trade-off is that it slightly delays closing this specific
ticket until the sweep lands, and it requires someone with commit
rights to the FHIR Core spec to run the grep — a low cost for the
consistency benefit.

---

### Disposition C: Decline

#### Proposal

Mark the ticket `Not Persuasive` on the grounds that a single
contraction (`doesn't`) on a non-normative narrative page is below the
threshold of an actionable "Technical Correction", and that imported
page-comment tickets of this granularity should be batched or
suppressed at import rather than triaged individually by FHIR
Infrastructure.

#### Justification

The defensible reason to decline is process-oriented, not editorial:
the change itself is harmless, but routing single-word style nits into
the workgroup's ballot queue at `Highest` priority consumes
disproportionate review bandwidth. Declining at the page level (with a
note pointing to a future editorial sweep) signals that the import
pipeline, not the workgroup, should aggregate these. This is the
weakest of the three options because it leaves the actual error in the
published spec.

---

### Recommendation

**Recommended disposition:** B

The underlying fix is uncontroversial and should ship — Disposition C
leaves a known style-guide violation in the published spec, which is
hard to defend for a "Highest" priority Technical Correction filed by
the editor himself. Between A and B, B delivers the same correction on
this page **and** prevents the workgroup from re-litigating the same
rule the next time the import pipeline surfaces another page with the
same contraction. The marginal cost (one repo-wide grep, one
consolidated commit) is small relative to the cost of triaging,
reviewing, and closing each imported page-comment ticket individually.
If the workgroup prefers minimal scope creep on a per-ticket basis,
Disposition A is a perfectly safe fallback and should be applied
without further discussion.
