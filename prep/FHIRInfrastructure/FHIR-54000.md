## Summary

The ticket reports a consistency issue with the value-set bindings on
`SearchParameter.base` and `SearchParameter.target`. Both elements are
typed `code` and are currently bound (required) to the
`version-independent-all-resource-types` value set, which enumerates only
resource types defined by the core FHIR specification at publish time. As
"Additional Resources" (extended resource types that may be defined in
downstream IGs or future versions) are not included in that value set, a
`SearchParameter` cannot legitimately reference them in its `base` or
`target`. The reporter proposes rebinding both elements to the
`extended-resource-types` value set, which is explicitly designed to
cover additional resource types in addition to the core resource types.

No other Jira tickets, Zulip threads, or GitHub items surfaced as
cross-references for this ticket, so the request stands alone in the
corpus gathered here.

## Details

**Description:**

`SearchParameter.base` and `SearchParameter.target` are of type `code`
and have required bindings to
<https://build.fhir.org/valueset-version-independent-all-resource-types.html>.

This ValueSet doesn't include future additional resources.

Presumably these bindings should be to
<https://build.fhir.org/valueset-extended-resource-types.html> instead.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the required bindings on `SearchParameter.base` and
`SearchParameter.target` in the FHIR core specification from the
`version-independent-all-resource-types` value set to the
`extended-resource-types` value set. No changes to cardinality, type
(`code`), or binding strength (remain `required`). This would allow
SearchParameter instances to legitimately name additional/extended
resource types defined by downstream IGs or future core versions as
their base or target.

#### Justification

The fix is a narrow terminology correction that brings SearchParameter's
metadata in line with the broader FHIR stance that non-core IGs can
define additional resource types. The current binding effectively
invalidates any SearchParameter that targets an extended resource type,
which is a real gap given that IGs already define such resources and
publish SearchParameters over them. Other metadata resources that
enumerate resource types have been (or are being) migrated to
`extended-resource-types` for the same reason, so this is a
consistency-preserving change rather than a design departure.

---

### Disposition B: Alternative Approach

#### Proposal

Weaken the binding on `SearchParameter.base` and
`SearchParameter.target` from `required` to `extensible` and keep the
bound value set as `version-independent-all-resource-types`. Implementers
needing to reference an extended resource type would supply a code from
`extended-resource-types` (or any other code designating a known resource
type) under extensible-binding rules, rather than being blocked by a
required binding that omits those codes. Optionally, document the
expected use of `extended-resource-types` codes in the element
definition.

#### Justification

This alternative accommodates the reporter's underlying need
(SearchParameters over extended resources) with the smallest possible
change to the normative shape of the resource. Because SearchParameter
is already published and widely implemented, moving the binding target
is a breakable change for tooling that validates against the old value
set URL; relaxing strength to `extensible` is less disruptive to
validators while still capturing the real-world practice of using codes
outside the core list. The trade-off is that this does not by itself
signal that extended resource types are fully first-class; it just
permits them.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the existing required binding to
`version-independent-all-resource-types`. Rationale: already addressed
by existing mechanisms — implementers who need to reference an extended
resource type on a SearchParameter can do so by defining the resource
in a way that is registered in the applicable version's resource-type
value set (or by using the extension mechanism around `base`/`target`
rather than changing the core binding).

#### Justification

A decline would be defensible if the working group concludes that
(a) the extended-resource-types pattern is still evolving and the core
SearchParameter bindings should wait until extended-resource semantics
are normatively stable, or (b) changing a required binding on a widely
implemented metadata resource carries a higher compatibility cost than
the benefit. Given that SearchParameter is central to server
capability advertising, any change to its value-set bindings flows
through to CapabilityStatement tooling and downstream validators; the
working group may prefer to hold that change until extended resource
types are broadly relied on.

---

### Recommendation

**Recommended disposition:** A

The ticket identifies a concrete consistency bug: a required binding
that excludes the very resource types the FHIR Infrastructure WG has
otherwise committed to supporting via `extended-resource-types`.
Disposition B preserves more backward-compat optics but leaves
SearchParameter semantically inconsistent with the rest of the
extended-resource-types story, and Disposition C effectively
invalidates a known implementer pattern. Rebinding to
`extended-resource-types` (Disposition A) is the change that most
directly matches the reporter's request and best aligns SearchParameter
with the broader direction of the specification; the compatibility
cost is limited to tooling that hard-codes the old value-set URL, which
the WG can call out in the release notes.
