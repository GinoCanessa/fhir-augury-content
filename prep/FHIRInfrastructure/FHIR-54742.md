## Summary

The reporter argues that the **General Service API** (the `%server`
factory in FHIRPath, which exposes search/read/create/update/delete and
related FHIR REST interactions inside FHIRPath expressions) is "powerful
and useful but under-used", and is therefore not yet ready to be
normative in R6. They request that the corresponding section of the
FHIRPath page be removed from the normative FHIR Core specification and
allowed to mature in a dedicated incubator IG before any future
normative ballot.

This ticket is part of a coordinated cluster of "Incubator"-labelled
trackers raised against the FHIRPath page in R6, all with the same
underlying theme — that several R5-introduced FHIRPath extensions
arrived at normative ballot with insufficient implementation experience.
Closely related tickets (surfaced via keyword similarity, not via
explicit Jira links) include
[FHIR-54074](https://jira.hl7.org/browse/FHIR-54074) and
[FHIR-53954](https://jira.hl7.org/browse/FHIR-53954), which make the
same incubator argument for `%server` with substantial technical
detail; [FHIR-54746](https://jira.hl7.org/browse/FHIR-54746) (same
request for the Terminology Service API `%terminologies`);
[FHIR-54745](https://jira.hl7.org/browse/FHIR-54745) and
[FHIR-54078](https://jira.hl7.org/browse/FHIR-54078) (same request for
the Type Factory `%factory`); and
[FHIR-54473](https://jira.hl7.org/browse/FHIR-54473) (same pattern for
FHIR Mapping Language). Of these, FHIR-54745 and FHIR-53954 have
already been resolved Persuasive in favour of moving the corresponding
content to an incubator, which is a strong precedent for how this
ticket is likely to be dispositioned.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Mark the **General Service API** (`%server`) section of the FHIRPath
page as **non-normative** for R6 and physically move its content out of
the normative FHIR Core FHIRPath page into a separate incubator IG
(e.g., a "FHIRPath Service APIs" incubator alongside, or co-located
with, the FHIRPath-in-FHIR content). Concretely:

- In the FHIRPath page section that currently documents `%server`
  (search / read / create / update / delete / patch / transform /
  validate / capabilities / everything), replace the body with a short
  pointer paragraph and a link to the new incubator IG.
- Move the full normative-style description of each `%server.*`
  function to the incubator IG, where it can carry an explicit FMM
  level (≤ 2) and a "trial-use, may change" notice.
- Retain the `%server` *name* as a reserved external constant in the
  FHIRPath page so that engines that already expose it do not collide
  with future re-use.
- Cross-reference FHIR-54745 and FHIR-53954 in the disposition text so
  the same editorial pattern is reused.

#### Justification

This mirrors exactly the resolution already accepted (Persuasive) on
[FHIR-53954](https://jira.hl7.org/browse/FHIR-53954) and
[FHIR-54745](https://jira.hl7.org/browse/FHIR-54745), which together
established the precedent that R5-draft FHIRPath extensions
(`%factory`, `%server`) should not graduate to normative without
demonstrated implementation experience. The technical backing in
[FHIR-54074](https://jira.hl7.org/browse/FHIR-54074) — undocumented
security model, side-effecting functions usable in invariants and
search parameters, inconsistent parameter design, near-zero engine
support — makes the "not ready for normative" judgement defensible on
its own merits, independent of the higher-level "under-used" argument
in this ticket. Moving to an incubator preserves the work and lets it
mature without locking the normative spec to an under-implemented API
surface.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the `%server` section in the FHIR Core FHIRPath page, but
explicitly demote it to **trial-use / non-normative** within an
otherwise-normative page (using the same banner/box mechanism the spec
already uses for mixed-maturity content), and add:

- A normative "Security Considerations" subsection forbidding use of
  side-effecting `%server.*` functions (`create`, `update`, `delete`,
  `patch`, `transform`) in any context where the FHIRPath expression
  comes from untrusted input — invariants, search parameters,
  questionnaire enableWhen / calculatedExpression, etc.
- A required "context declaration" mechanism (per
  [FHIR-54074](https://jira.hl7.org/browse/FHIR-54074)) so engines must
  opt-in per evaluation context before exposing the mutating subset.
- An explicit FMM rating (≤ 2) on the section, with a stated
  re-evaluation milestone for a future ballot.

#### Justification

This addresses the underlying concerns — under-implementation, security
risk, design churn — without fragmenting the FHIRPath specification
across two documents. Implementers (and the FHIRPath engine community)
benefit from having one canonical reference for the `%server` surface,
and the FHIR Core page already mixes maturity levels for individual
elements. The trade-off vs. Disposition A is editorial: it leaves the
content inside a normative document, which historically has caused
exactly the "under-cooked content escapes into normative" pattern this
cluster of tickets is reacting against, and it does not match how
[FHIR-53954](https://jira.hl7.org/browse/FHIR-53954) and
[FHIR-54745](https://jira.hl7.org/browse/FHIR-54745) were resolved.

---

### Disposition C: Decline

#### Proposal

Leave the General Service API (`%server`) section in the normative
FHIRPath page in FHIR Core R6 unchanged, on the grounds that the
section has been through ballot, the design has been stable since R5,
and "under-used" is not by itself a normative-blocking criterion.
Address specific defects via the more detailed tickets
([FHIR-54074](https://jira.hl7.org/browse/FHIR-54074) etc.) without
relocating the content.

#### Justification

A decline is defensible only if one believes (a) `%server` has enough
real-world engine implementations to satisfy the FHIR maturity model
for normative content, and (b) the security and design issues raised
in [FHIR-54074](https://jira.hl7.org/browse/FHIR-54074) can be fully
resolved within the R6 ballot cycle. Neither premise is well
supported: `%server` was maturity-0 in R5, the cluster of related
trackers shows the design is still in flux, and the same work group
has already accepted Persuasive resolutions to move the sibling
`%factory` content out of normative. Declining this ticket would also
create an inconsistency between `%factory` (moving out) and `%server`
(staying in) that would be hard to defend editorially.

---

### Recommendation

**Recommended disposition:** A

The work group has already, on FHIR-53954 and FHIR-54745, accepted the
exact pattern this ticket asks for — moving R5-draft FHIRPath
extensions into an incubator rather than graduating them to normative
in R6. Treating FHIR-54742 the same way is the consistent outcome and
aligns the disposition of `%server` with the disposition of `%factory`
(and very likely the parallel `%terminologies` request in
[FHIR-54746](https://jira.hl7.org/browse/FHIR-54746)). The deeper
technical concerns enumerated in
[FHIR-54074](https://jira.hl7.org/browse/FHIR-54074) — particularly
the security profile of side-effecting functions invokable from
invariants and search parameters — are not realistically resolvable
inside the R6 cycle, and trying to keep `%server` normative while
fixing them in flight would put the rest of the FHIRPath page at
risk. Disposition B is a reasonable fallback if the work group decides
the incubator path is operationally too heavy, but Disposition C
should not be entertained without first resolving the security and
implementation-experience gaps called out in the linked tickets.
