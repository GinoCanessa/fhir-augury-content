## Summary

The ticket reports that `List.mode` is defined as a 1..1 mandatory element with
a required binding to the [`list-mode`](http://hl7.org/fhir/ValueSet/list-mode)
value set, but the bound value set provides no `unknown`, `other`, or
equivalent "escape-valve" concept. This means there are real-world cases where
an instance of `List` cannot be validly populated — the producer does not know
which of `working`, `snapshot`, or `changes` actually applies, yet the element
must be present and the binding strength forbids any extension code. The
reporter (Lloyd McKenzie) asks FHIR-I to add an `unknown`/`other` concept (or
equivalent mechanism) to the value set.

The ticket is a direct sibling of [FHIR-55173](https://jira.hl7.org/browse/FHIR-55173)
("List.status has no escape valve"), which raises the identical issue against
`List.status`. Both tickets were filed by Lloyd McKenzie on the same day with
the same priority (Highest) and the same Triaged status, and the implementer
discussion in Zulip explicitly treats them as a pair — and broadens the
question to whether `List.status` and `List.mode` should be mandatory at all.
Any disposition for FHIR-55174 should be coordinated with FHIR-55173 so the two
elements end up with compatible semantics.

## Related Zulip Discussions

### [implementers > List.status and List.mode](https://chat.fhir.org/#narrow/stream/implementers/topic/List.status%20and%20List.mode/near/581209815)

Grahame Grieve opened the thread on 2026-03-23 to solicit implementer feedback
on FHIR-55173 and FHIR-55174 and, more broadly, on whether `List.status` and
`List.mode` are fit for purpose. Grahame raised three secondary concerns:
(1) `List.status`'s definition talks about "patient record" but most lists are
not patient-specific; (2) is there any use for a `proposed` list?; and (3) do
the `mode` values cover all the uses?

John Moehrke contributed the only substantive implementer experience: IHE
reused `List` for XDS SubmissionSet/Folder and found `List.mode` fixed at
`working` "too limited to be useful" — they had to add an extension alongside
`code`, effectively splitting "kind of list" from "clinical meaning of the
list" the way other resources separate `.category` from `.code`. `List.status`
was usable as-is for IHE.

When Grahame pinged the thread two weeks later, Nikolai Ryzhikov questioned
the underlying premise: are `List.status` and `List.mode` essential to the
semantics of a List at all, or could they be optional (analogous to
`Patient.active`)? Grahame replied that `mode` is necessary — "you can't work
with a list with confidence unless you know what the mode is" — but conceded
that the same argument does not necessarily apply to `status`, and that
`List.status` is probably mandatory only because "no one ever asked for it to
be" otherwise. Grahame closed by noting he sees no other tasks related to
this beyond FHIR-55174 (and FHIR-55173). Consensus was not reached.

Participants: Grahame Grieve, John Moehrke, Nikolai Ryzhikov.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an `unknown` concept (and optionally an `other`) to the
`http://hl7.org/fhir/list-mode` CodeSystem and surface it through the
`http://hl7.org/fhir/ValueSet/list-mode` ValueSet. The new concept's display
would be "Unknown" with a definition along the lines of "The mode of this
list is not known. Used when the producer cannot reliably classify the list
as `working`, `snapshot`, or `changes`." Keep `List.mode` 1..1 with a
required binding; the new code becomes the explicit escape valve. Coordinate
with FHIR-55173 so an analogous `unknown` is added to `list-status` for
symmetry.

#### Justification

This is the smallest change that fully addresses the reporter's complaint
and matches the standing FHIR pattern for required-bound mandatory codes
(many FHIR value sets bound to mandatory elements include `unknown` exactly
to give producers a defensible value when the real value is not known).
Because `List.mode` is required+mandatory, no extension is permitted in its
place, so the value set itself is the only place the escape valve can live.
Adding `unknown` is non-breaking for existing instances and existing systems
that consume the value set, and it cleanly resolves the "impossible to
populate" cases the ticket calls out. It also keeps List aligned with
FHIR-55173's resolution if both are accepted.

---

### Disposition B: Alternative Approach — Loosen the Binding / Cardinality Instead of Adding a Code

#### Proposal

Rather than (or in addition to) adding a code, address the underlying
"impossible to populate" problem structurally. Two sub-options, either of
which would be paired with FHIR-55173:

- **B1:** Relax the binding strength on `List.mode` from `required` to
  `extensible`, so that producers who genuinely cannot pick `working`,
  `snapshot`, or `changes` can supply a code from another system (e.g., a
  data-absent-reason code or a domain-specific code such as the IHE XDS
  extension John Moehrke described) without violating the binding.
- **B2:** Relax the cardinality of `List.mode` from `1..1` to `0..1`,
  allowing the element to be omitted when unknown, and rely on producers
  populating it whenever they can. This is the path Nikolai Ryzhikov
  questioned about in Zulip, and the path Grahame indicated he was less
  willing to accept for `mode` (versus `status`).

In either sub-option, the `list-mode` value set is left unchanged.

#### Justification

B1 acknowledges John Moehrke's IHE feedback: the existing three modes are
genuinely insufficient for some real deployments, and an `unknown` code does
not solve that — what those implementers actually need is the freedom to use
their own classification. An extensible binding lets them do that without
HL7 having to enumerate every domain-specific mode. B2 is a more aggressive
change but is the cleanest resolution if the workgroup accepts Nikolai's
framing that `List.mode` is not strictly essential to the resource's
semantics. The trade-off versus Disposition A is: A is small, safe, and
backwards-compatible but does not address the deeper "the modes don't cover
my use case" problem; B1 addresses both the escape-valve and the
extensibility complaint but is a normative-level change to a binding; B2 is
the most disruptive and Grahame already pushed back on it on Zulip for
`mode` specifically.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" / "Will Not Fix" on the grounds that
producers who genuinely do not know the mode of a list should not be
producing a `List` resource in the first place — `List.mode` is a core
semantic discriminator (working vs. snapshot vs. changes) that consumers
must rely on to interpret the entries safely, and a `List` whose mode is
unknown is not safely consumable. Direct any deeper "the modes don't cover
my use case" feedback into a separate, scoped ticket against the
`list-mode` value set.

#### Justification

This is the position implicit in Grahame Grieve's Zulip comment that "you
can't work with a list with confidence unless you know what the mode is, so
it should always be explicit." Adding `unknown` to a value set whose values
are intended to be mutually exclusive interpretation contracts undermines
the contract: a consumer cannot safely treat an `unknown`-mode list as
either a snapshot or a change set. The decline is defensible on
interoperability grounds and points implementers toward the right
remediation (either pick the closest mode, or model the data as something
other than a `List`).

---

### Recommendation

**Recommended disposition:** A (with an explicit follow-up ticket for the
broader `list-mode` extensibility concern raised by John Moehrke).

Disposition A is the smallest change that resolves the literal request, is
consistent with how FHIR routinely handles required-bound mandatory codes
elsewhere, and is non-breaking for existing implementations. It also pairs
naturally with FHIR-55173 so `List.mode` and `List.status` end up with
symmetric treatment. Disposition B1's underlying point is real — IHE's
experience shows the three modes are not exhaustive — but addressing that
correctly requires a deliberate review of the `list-mode` value set rather
than a binding-strength change bolted onto this ticket; that work belongs
in a separate, scoped ticket. Disposition C is defensible in principle but
ignores the concrete use cases the reporter and IHE have surfaced and
leaves implementers with no valid way to produce a `List` instance when the
mode is genuinely not known. The workgroup should accept A, file the
follow-up against the value set, and coordinate the wording with
FHIR-55173's resolution.
