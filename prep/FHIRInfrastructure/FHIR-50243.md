## Summary

The reporter (QingyangXie / Blake Xie) flags an inconsistency in the
**FHIR R5 Subscriptions Backport** IG. The example
[`Bundle-r4-notification-id-only`](https://hl7.org/fhir/uv/subscriptions-backport/Bundle-r4-notification-id-only.json.html)
is presented as a response to the Subscription `$status` operation
(its `Bundle.entry[0].request.url` references `$status`), but the
notification `Parameters` resource in that bundle carries
`SubscriptionNotificationType = "event-notification"` rather than
`"query-status"`. Per the IG, a `$status` response is supposed to
return a notification bundle whose notification type is
`"query-status"`, not tied to a specific event. A sibling example,
[`Parameters-r4-notification-status`](https://hl7.org/fhir/uv/subscriptions-backport/Parameters-r4-notification-status.json.html),
correctly uses `query-status`, which the reporter cites as the right
shape. The request is to fix the example so it agrees with the IG's
own normative text. There are no cross-referenced Jira tickets; the
only cross-reference is the originating Zulip thread in the
`Da Vinci PAS` stream where Gino Canessa confirmed the example is
incorrect and asked the reporter to file the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Fix the example `Bundle-r4-notification-id-only` in the Subscriptions
Backport IG so it agrees with the IG's normative `$status` shape:

- Change the embedded `Parameters` resource (the `SubscriptionStatus`
  payload) so `SubscriptionStatus.type` is `"query-status"` rather
  than `"event-notification"`.
- Drop or zero out event-specific machinery that is meaningless for a
  `$status` snapshot, namely `SubscriptionStatus.notificationEvent`
  (or, if events are kept to illustrate "most recent event included",
  retain at most one entry and explicitly call that out in the
  example's narrative).
- Re-check `Bundle.entry[0].request.url`/`fullUrl` so the bundle
  remains a coherent `$status` response (id-only flavor: the
  `focus`/notification entries should remain references only, not
  full resources).
- Regenerate the rendered `Bundle-r4-notification-id-only.json.html`
  page so the published example matches the corrected source.

If the IG also has matching `xml`/`turtle` renderings of the same
example, update those too to keep formats in sync.

#### Justification

The reporter and Gino Canessa already agree on Zulip that the example
is wrong: the IG's normative description of `$status` says the
operation returns a notification with `SubscriptionStatus.type =
"query-status"`, and the sibling example
`Parameters-r4-notification-status` already uses that type. The fix is
local to one example file and its narrative — it does not change any
profile, value set, or operation contract — so it carries minimal
downstream risk while removing a documented source of implementer
confusion. Accepting as requested is the most direct path to restoring
internal consistency in the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this as a broader **example-set audit and clarification** rather
than a single one-line fix:

1. Keep `Bundle-r4-notification-id-only` as an **`event-notification`
   id-only example** (its current form), but change its narrative and
   the `Bundle.entry.request.url` so it is no longer presented as a
   `$status` response — this preserves the existing id-only event
   illustration that implementers may already be referencing.
2. Add a **new** id-only example specifically for `$status` (e.g.,
   `Bundle-r4-notification-id-only-status`) with
   `SubscriptionStatus.type = "query-status"` and no
   `notificationEvent` entries, mirroring the shape of
   `Parameters-r4-notification-status` but in the id-only payload
   variant.
3. Sweep the other notification examples (`full-resource`,
   `empty`, `handshake`, `heartbeat`) and confirm each one's
   `SubscriptionStatus.type` matches the scenario it claims to
   illustrate; document the matrix (notification type × payload
   content) in the IG's narrative so the relationship is explicit.
4. Optionally add a short "How to read the notification examples"
   section that maps each example file to its
   `SubscriptionNotificationType` and the operation/event that would
   produce it.

#### Justification

The defect the reporter found is almost certainly not unique — the
IG carries several notification examples, and the same kind of
mismatch (URL/context implies one `SubscriptionNotificationType` while
the payload uses another) could plausibly exist elsewhere. A targeted
audit avoids "fix one example, ship the same class of bug in another"
and turns the ticket into a small but durable improvement to the IG's
example coverage. The trade-off versus Disposition A is scope: this is
more work for the editor and reviewers, and it touches additional
example files that may be referenced externally; backward
compatibility for implementers consuming the JSON examples by file
name is preserved by adding new examples rather than relabeling
existing ones in place.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing example, while
ambiguously presented, does not violate the IG's normative content:
the `Bundle.entry.request.url` is a hint, not a constraint, and the
authoritative shape of a `$status` response is fully specified in the
IG's operation definition and in
`Parameters-r4-notification-status`. Add an FAQ/clarification note in
the IG narrative pointing implementers at the correct example for
`$status`, but leave `Bundle-r4-notification-id-only` unchanged.

#### Justification

This stance is defensible only if the workgroup judges that changing
published example artifacts is more disruptive than living with a
single confusing example, given that the normative text is correct
and a clearer example (`Parameters-r4-notification-status`) already
exists. It is the weakest of the three options because the reporter
and the responding subject-matter expert have already agreed the
example is incorrect, and "the spec is right, the example is just
misleading" is exactly the failure mode IG examples are supposed to
prevent.

---

### Recommendation

**Recommended disposition:** A

The defect is small, the fix is local to one example file plus its
generated HTML, and the correct shape is already demonstrated by
`Parameters-r4-notification-status`, so there is no design question
left to resolve — only an editorial correction. Disposition B is
attractive as a follow-on but should not block this fix; if the
workgroup wants the broader example-matrix audit, it can be tracked
as a separate ticket so FHIR-50243 can close cleanly. Disposition C
leaves a known-incorrect published example in place after the
maintainer has already acknowledged it on Zulip, which is hard to
defend to implementers who hit the same confusion. Recommend Accept
As Requested, with the editor optionally retaining a single
`notificationEvent` entry in the corrected example and noting in the
example's narrative that events MAY be included in a `$status`
response per the IG's `$status` description.
