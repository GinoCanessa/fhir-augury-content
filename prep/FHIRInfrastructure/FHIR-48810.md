## Summary

The ticket reports a typo in an XML example block in the FHIR Core
specification. The example uses the malformed sequence `<--! rest of
resource omitted -->` to indicate elided XML, where the correct XML
comment opener is `<!--` (exclamation mark before the dashes, not
after the opening angle bracket). The reporter, Rik Smithies, asks
that the example be corrected so it is well-formed XML and copy/paste
safe. There are no cross-referenced Jira tickets, Zulip discussions, or
GitHub items associated with this ticket — it is a self-contained
editorial fix.

## Details

**Description:**

> XML comment is wrongly formatted:
>
> ```xml
> <resource>
>        <Observation>
>          <subject>
>             <reference value="Patient?identifier=12345"/>
>          </subject>
>          <--! rest of resource omitted -->
>        </Observation>
>      </resource>
> ```

The defect is the comment delimiter `<--!` which should be `<!--`. As
written, the snippet is not well-formed XML and would fail any
parser/validator a reader pasted it into.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive — Editorial. Apply the literal correction the reporter
asks for: replace the malformed comment delimiter `<--!` with the
well-formed XML comment opener `<!--` in the affected example. The
corrected snippet should read:

```xml
<resource>
  <Observation>
    <subject>
      <reference value="Patient?identifier=12345"/>
    </subject>
    <!-- rest of resource omitted -->
  </Observation>
</resource>
```

Because the ticket does not name the source file, the editor applying
the fix must locate the snippet in the FHIR Core narrative (it is the
canonical `Patient?identifier=…` conditional-reference example, most
likely on the references / conditional-reference page) and update it
in place. The fix is in narrative-only content and does not affect any
StructureDefinition, schema, or generated artifact.

#### Justification

The reported text is objectively malformed XML — `<--!` is not a valid
comment opener in any XML grammar — so a reader copying the example
verbatim would get a parse error. Correcting it to `<!--` aligns the
example with the XML 1.0 specification and with every other XML
comment used elsewhere in the FHIR narrative. There is no semantic
change, no normative impact, and no implementer disruption; it is a
pure editorial fix that improves the spec's copy/paste correctness.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive With Modification — Fix the typo *and* sweep adjacent
examples. While correcting `<--!` → `<!--` in the cited snippet, also
grep the FHIR Core narrative sources for any other occurrences of the
same malformed delimiter (and the mirror typo `<!- … -->` / `<-- …
-->`) and correct them in the same change. Optionally, normalise the
elision wording to a consistent phrase such as `<!-- snip -->` or
`<!-- … other elements omitted … -->` across examples so readers see a
uniform convention.

#### Justification

Typos in delimiter syntax tend to be copy-paste artefacts that
propagate — if one example has `<--!`, sibling examples authored at
the same time may have it too. A single sweep is cheap relative to
re-opening the spec for each new sighting and reduces the chance of a
follow-up ticket on the same defect. Normalising the elision phrasing
is a small additional polish that aids readers scanning many examples.
Because the change is still narrative-only, the risk profile is
identical to Disposition A.

---

### Disposition C: Decline

#### Proposal

Not Persuasive — Decline the ticket on the grounds that the snippet
is illustrative narrative rather than a conformance example, and the
intent (to indicate elided content) is obvious to a human reader.

#### Justification

This justification is weak and is included only for completeness. The
FHIR specification explicitly aims to be implementer-friendly and
copy/paste safe; shipping malformed XML in an example contradicts
that goal even when the intent is "obvious." There is no
backward-compatibility, scope, or cost argument that supports
declining a one-character editorial fix.

---

### Recommendation

**Recommended disposition:** B

The defect is real and trivially fixable, so declining (C) is not
defensible. Between A and B, the marginal cost of a repo-wide grep
for `<--!` (and the mirror typos) while the editor is already in the
narrative is negligible, and it forecloses near-duplicate tickets if
the same copy-paste error appears elsewhere. Recommend Persuasive
With Modification: apply the literal fix the reporter asks for and
sweep the surrounding examples for the same delimiter typo in the
same change.
