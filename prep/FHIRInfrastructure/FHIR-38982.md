## Summary

Cooper Thompson (Epic) proposes that FHIR Subscriptions support a new
notification payload option that conveys results as a Bulk-Data-style manifest
of NDJSON files, rather than packing all resources directly into the
subscription notification Bundle or forcing the receiver to perform follow-on
REST queries to assemble the full data set. The motivating use case is
genomic results, where a single notification may represent very large
collections of resources that exceed practical Bundle / message-size limits
and where the sender — not the receiver — knows the full set of resources
that constitute "the result." The request is effectively a hybrid between the
existing `id-only` notification (which makes the receiver "stumble through
the REST API") and the existing `full-resource` notification (which is
size-bound), borrowing the manifest pattern from Bulk Data Access. The ticket
discussion cross-references the `SubscriptionStatus.additionalContext` and
`SubscriptionStatus.notificationEvent.additionalContext` elements (both
`Reference(Any)`) as the natural attach point for such a manifest today via a
`Binary` resource. There are no linked or cross-referenced Jira tickets; the
sole external discussion is the `#social > Interesting discussion on using
Kafka with FHIR` Zulip thread that originated the idea, where message-size
limits of MQ/streaming channels (Kafka, Service Bus, etc.) were debated.
Since the original filing the reporter has indicated their genomics use case
was solved another way and is comfortable deferring or closing the ticket.

## Details

**Description:**

We are looking (early phase, lots can change) at using subscriptions for
genomic results, where we want the lab to send the result via
SubscriptionStatus. However, for some genomic results, the content can be
very large. One thought we had was adding an option to SubscriptionStatus to
specify the payload as a "Bulk-Data-like" manifest of NDJSON files.

This is basically a hybrid between an id-only notify with REST query and
sending everything in the notification payload. Instead of having to stumble
through the REST API to find everything, you get it all neatly packed up for
you in the manifest and files. It also avoids the size limitations of trying
to push very large content inside the notification bundle.

**Comments:**

  recipient is preferable. Frames the design tension as (1) which party
  knows what data is relevant and (2) what access the data owner is willing

- **ginocanessa (2022-10-13):** Clarifies how `SubscriptionTopic`

  servers have flexibility but should derive a new topic if they add

  `SubscriptionStatus.notificationEvent.additionalContext` is

- **cooper.thompson (2022-10-13):** Acknowledges this works but notes that

- **cooper.thompson (2022-10-14):** Realizes referenced resources can simply

## Related Zulip Discussions

### [social > Interesting discussion on using Kafka with FHIR](https://chat.fhir.org/#narrow/stream/social/topic/Interesting%20discussion%20on%20using%20Kafka%20with%20FHIR/near/303636922)

This is the originating thread for the ticket. The discussion starts from a
LinkedIn article on FHIR + Apache Kafka and quickly converges on the core
problem behind FHIR-38982: every major message-queue / streaming channel
(Kafka ~1 MB default, Azure Service Bus ~256 KB, etc.) imposes a maximum
message size that subscription-notification Bundles can easily exceed
(genomic data, large code systems, resources with embedded binary, etc.).
Participants explore several mitigations, with rough alignment that "the
spec needs to acknowledge channel size limits" but no consensus on
mechanism: (a) channel-defined automatic downgrade (e.g., serve
`full-resource` "unless size > X bytes"), with most preferring this be a
property of the channel definition rather than the core resource; (b) a
generic "ids-only" or "small-only" mode that could apply not just to
subscriptions but to search Bundles too (Elliot Silver, Paul Church, Grahame
Grieve found this attractive); (c) compression negotiation on the channel,
particularly for byte-oriented channels like Kafka (Mike Lohmeier — others
noted HTTP-level gzip already exists); (d) the existing `id-only`
notification + REST follow-up, which Cooper rejects as the framing for this
ticket. Felipe Soriano dissents, arguing message-broker concerns are
infrastructure / IaC and should not be in a FHIR spec. Grahame notes prior
HL7 attempts to standardize MQ/streaming channels concluded "not enough
interest," but acknowledges the resurfacing problem is "interesting."
Gino Canessa points out Topic-Based Subscriptions deliberately did not
pre-define a Kafka (or other MQ) channel and suggests an IG could carry
those channel definitions; he also flags that with the R5 ballot window
closing that week, anything more than an extension is unrealistic for R5.
Cooper's manifest idea (FHIR-38982) is dropped into the thread as one of
several candidate solutions but is not concluded; David Pyke suggests
moving deeper subscription discussion to `#subscriptions`.

Participants: Cooper Thompson (originator/author of FHIR-38982),

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a first-class "manifest payload" option to FHIR Subscriptions in R6,
parallel to the existing `full-resource`, `id-only`, and `empty` payload
modes. Concretely:

- Add a new value (e.g., `bulk-manifest`) to `Subscription.content` (R5
  `SubscriptionTopicCanonical.payload.content` — the
  `SubscriptionPayloadContent` value set) so a subscriber can request that
  notifications be delivered as a Bulk Data manifest reference rather than a
  Bundle of resources.
- Add a typed slot on `SubscriptionStatus` (or constrain
  `SubscriptionStatus.notificationEvent.additionalContext`) to carry either
  (i) an inline Bulk Data-style manifest object (transactionTime, request,
  output[].type/url, error[].type/url, optional access-token hint), or (ii)
  a typed reference to a fetchable manifest endpoint. Avoid forcing the
  manifest through a `Binary` wrapper, which is the friction point Cooper
  called out.
- Allow a small "header" set of resources to still ride in the notification
  Bundle alongside the manifest reference (e.g., the focus
  `DiagnosticReport`, `Patient`) so receivers can act before the bulk
  download completes (jmandel's question in comments).
- Document interaction with `SubscriptionTopic` traversal hints: a
  `bulk-manifest` payload's NDJSON files SHOULD include resources that
  satisfy the topic's include/revinclude graph.

#### Justification

The existing payload modes form a deliberate spectrum (`empty` → `id-only`
→ `full-resource`) and `bulk-manifest` is the natural fourth point that
covers "sender knows the full set, but the set is too large for a
notification Bundle." The receiving side already has a well-understood
processing model for Bulk Data manifests, so reuse minimizes new client
code. Cooper's framing — "which party knows what the relevant data is" and
"what access the data owner is willing to give" — argues for a
sender-packaged option whenever the sender is the authoritative grouper,
which is exactly the genomics / lab-result / ordering-workflow shape where
"stumbling through the REST API" is brittle. The Zulip thread shows
recurring demand (Paul Church's Google notification "payloadType",
Craig McClendon's homegrown approach) for exactly this hybrid; codifying it
in the core avoids vendor-specific divergence.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a new `Subscription.content` value or change `SubscriptionStatus`
in core. Instead, address the underlying need with a layered combination of
existing mechanisms plus a single extension and an IG:

1. Define a standard FHIR extension (e.g.,
   `http://hl7.org/fhir/StructureDefinition/subscription-bulk-manifest`)
   applicable to `SubscriptionStatus.notificationEvent.additionalContext`
   (or as a sibling element) that carries a Bulk Data-style manifest object
   without requiring a `Binary` wrapper. This is the path implementers can
   take in R5 today.
2. Encourage the Bulk Data IG (or a new "FHIR Subscriptions over Streaming
   Channels" IG, as suggested in Zulip) to define channel-typed payload
   profiles — including the MQ-channel size-limit concerns the Zulip thread
   raised — and to specify how a `bulk-manifest` payload negotiates and is
   downgraded.
3. Add a generic `bulk-manifest` value to `SubscriptionPayloadContent`
   (codes only — not new structural elements) so that a server's choice of
   payload mode can be communicated without inventing a new core element.

#### Justification

Most of the desired behavior is achievable today: `additionalContext` is
`Reference(Any)`, so a server can already deliver a focus resource plus
links to NDJSON files, either through a `Binary` (Gino's path), a
`Parameters` resource carrying URLs, or jmandel's `Binary`-as-focus pattern
(see his
[gist](https://gist.github.com/jmandel/bfe7c26d7df1e327282952083f744d4c)).
The only real gap is the awkwardness of the `Binary` hop and the lack of a
named, profileable manifest shape. An extension + code addition fills that
gap with minimal core surface change and lets implementer experience inform
whether to promote the mechanism into core in a later cycle. This also
respects Grahame's note in Zulip that prior attempts to standardize
queue/stream channel mechanics in core had "not enough interest," and
matches Gino's suggestion to "see it in practice before worrying about
adding more to R5/R6 at this point." It keeps the door open for the
broader, non-subscription use cases Elliot Silver raised (a generic
"ids-only / small-only" pattern for search Bundles) without coupling them to
the subscriptions resource.

---

### Disposition C: Decline

#### Proposal

Resolve as Not Persuasive (with optional "Already Addressed by Existing
Mechanism") and close. Rationale to record in the resolution: (a) the
reporter has stated on the ticket (2023-07-10) that their genomics use case
was solved by other means and the need has not recurred; (b) the existing
`SubscriptionStatus.notificationEvent.additionalContext` element, being
`Reference(Any)`, already permits any implementer to attach a Bulk Data-
style manifest today (e.g., via `Binary`, `Parameters`, or a focal manifest
resource), as documented in the discussion; (c) wider work on
queue/streaming channel constraints (Kafka, Service Bus, …) is properly
the subject of a separate channel-definition IG rather than a core
Subscription change.

#### Justification

The ticket itself signals it is no longer pressing — the originator
explicitly offered to defer or close. There is no other implementer on the
ticket asking for the feature, no linked Jira tickets, and no GitHub
implementation work tracking it. The Zulip discussion that triggered the
ticket converged on extensions and IGs as the right venue, not core
schema changes. Adding a fourth payload mode to a normative-track resource
without active demand or implementer experience risks introducing a
mechanism that bakes in assumptions (manifest shape, downgrade rules,
access-token hints) that the community has not yet validated; Gino's
guidance in the Zulip thread — "I would prefer to see it in practice before
worrying about adding more" — applies just as strongly to R6.

---

### Recommendation

**Recommended disposition:** B

The use case is real and recurs (Paul Church's Google "payloadType",
Craig McClendon's homegrown approach, Cooper's genomics scenario, the
broader Kafka thread), so simply declining (C) leaves a known interop pain
point unaddressed and ignores the structural friction Cooper identified —
having to wrap a manifest in a `Binary` just so a `Reference` can carry it.
But the design space is genuinely under-explored: manifest shape, downgrade
rules, access-token hints, channel-specific size negotiation, and the
relationship to `SubscriptionTopic` traversal hints have all been raised
but not settled. Going straight to a normative core change (A) risks baking
in choices the implementer community has not yet validated, exactly the
concern Gino Canessa and Grahame Grieve voiced in the Zulip thread. A
standard extension on `additionalContext` plus a `bulk-manifest` code value,
carried into a Subscriptions-over-streaming-channels IG, gives implementers
a named, profileable mechanism today, gathers real-world experience, and
keeps the door open to promote it into core in a later cycle — while
sidestepping the `Binary` wrapper that is the only real friction with the
status quo. The workgroup should also explicitly invite the originator
(Cooper) and the Zulip participants (Paul Church, Craig McClendon, Mike
Lohmeier, Elliot Silver) to validate the extension shape before any core
promotion.
