## Proposed Dispositions

Because this ticket exists purely to attach an external ballot vote
(BALLOT-55016) to a FHIR tracker, the dispositions below focus on **how
the work group should handle the unattached negative ballot comment**, not
on a code/spec change. There is no GitHub repo context to ground a
file-level change here; once the substantive comment text is retrieved
from BALLOT-55016, any spec change would land in the **Subscriptions
Backport IG** sources (`HL7/fhir-subscription-backport-ig`) and would need
its own follow-up Change Request to drive that edit.

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-43726 as the canonical FHIR tracker for the negative vote in
BALLOT-55016. Specifically:

1. Pull the actual negative-comment text and rationale directly from
   BALLOT-55016 in Jira (not available in the cached snapshot) and copy it
   into a comment on FHIR-43726 so the change request stands on its own.
2. Re-classify FHIR-43726 from generic "Change Request" to a more specific
   sub-type if the workgroup uses one for ballot comments (e.g. set the
   appropriate `Related Artifacts` / `Related Pages` once the comment
   subject is known).
3. Assign FHIR-43726 to a Subscriptions Backport maintainer for
   substantive disposition (Persuasive / Persuasive with Mod / Not
   Persuasive / Not Persuasive with Mod), and resolve BALLOT-55016 by
   reference to whatever resolution lands here.

#### Justification

This is the standard HL7 ballot-reconciliation pattern: every negative
ballot vote must be tracked by a FHIR-project change request that the
sponsoring work group dispositions and that the negative voter then either
withdraws or upholds. FHIR-43726 was created for exactly that purpose
(reporter Michael van der Zel, who manages Subscriptions Backport ballot
hygiene) and pointing it at BALLOT-55016 is the bare minimum needed to
preserve traceability between the ballot record and the substantive change
request. Doing nothing more leaves the negative comment effectively
unaddressed in the FHIR project tracker.

---

### Disposition B: Alternative Approach

#### Proposal

Split the work: keep FHIR-43726 strictly as a *linkage / metadata* record
for BALLOT-55016, and open one or more **new, narrowly-scoped Change
Requests** under the FHIR Infrastructure work group / Subscriptions
Backport specification — one per substantive issue raised in the negative
ballot comment. Close FHIR-43726 with resolution "Tracking" (or similar)
once those substantive tickets exist and are linked from FHIR-43726 via
"is related to" Jira links. Disposition of the negative vote then happens
on the new tickets, each of which can carry its own affected artifact,
related page, and proposed spec text.

#### Justification

The existing ticket has neither a description of the actual problem nor
an `Affects` artifact / page; it is purely a pointer. Trying to
disposition a negative ballot directly on a stub ticket forces all the
discussion, the proposed spec change, and the Persuasive/Not-Persuasive
vote into one record that has no real subject line. Splitting preserves
auditability (the original ballot link survives on FHIR-43726) while
giving each substantive issue its own discoverable ticket with
`Specification`, `Related Artifacts`, and a clear resolution path. This
also matches how the Subscriptions Backport team has historically handled
multi-issue negative ballots, where one ballot submission expands into
several FHIR tickets.

---

### Disposition C: Decline

#### Proposal

Close FHIR-43726 as **Not Related / Persuasive-No-Change** without
substantive action, on the grounds that BALLOT-55016 is a ballot
submission whose attached negative comment must already be reconciled
inside the BALLOT project (the HL7 ballot tooling) and does not require a
parallel record in the FHIR project. The negative voter (Niek van Galen)
would either withdraw the negative on BALLOT-55016 directly or escalate
through the standard ballot-appeals process, with no ongoing change
request needed in the FHIR tracker.

#### Justification

This is defensible only if the negative comment's substance is purely
procedural (e.g. concerns the ballot package, not the IG content) or has
already been addressed by another, already-tracked FHIR ticket in the
2024-Jan Subscriptions Backport reconciliation set. Because this dataset
exposes no Zulip discussion, no cross-references, and no body text on
either FHIR-43726 or BALLOT-55016, we cannot confirm those conditions —
so this disposition should only be selected if the workgroup, after
inspecting BALLOT-55016 in Jira, determines the negative comment was
withdrawn or duplicated.

---

### Recommendation

**Recommended disposition:** A

The ticket's sole purpose is to give a FHIR-project home to a negative
ballot vote on the 2024-Jan Subscriptions Backport STU. The minimum
correct action is to pull the actual negative-comment text out of
BALLOT-55016, paste it onto FHIR-43726, assign a Subscriptions Backport
maintainer, and disposition the comment on this ticket — exactly what
Disposition A describes. Disposition B (split into multiple tickets) is a
reasonable refinement, but should be applied only *after* the comment
content has been read and only if it actually raises more than one
distinct issue; doing it pre-emptively risks creating empty stub tickets.
Disposition C should be reserved for the case where the BALLOT-55016
comment turns out to be procedural or already-resolved, which the
available data does not support. Because the priority on FHIR-43726 is
**Highest** and the status is still **Triaged** more than two years after
the 2024-Jan ballot, the work group should treat this as overdue and
move it forward under Disposition A immediately.
