## Summary

The reporter asks FHIR Infrastructure to define a standard mechanism for asserting that a *profile implements / realizes a particular logical model* — i.e., a simple, machine-readable statement that "this profile implements logical model X". The request explicitly distinguishes this top-level "implements" relationship from the more granular *element-to-element* mappings that already have a home (ElementDefinition.mapping, ConceptMap, StructureMap). The reporter enumerates three candidate mechanisms — a new extension on the profile, a ConceptMap that maps the top elements, or a mapping carried inside the resource metadata — and asks the WG to pick one and document it on the Logical Models page (and define an extension if that is the chosen path).

The motivating use cases drawn out in the comments are: (1) IG documentation ("Profile B implements Logical Model A") so the IG publisher can render the relationship on profile pages; (2) cross-IG traceability — finding which profiles are affected when a logical model changes; and (3) optional richer relationship qualifiers (broader-than / narrower-than, "no map" vs. "maps to nowhere"). No external Jira tickets are linked; the only cross-references the augury index surfaces are one Zulip thread on `#ig publishing requirements` (where the question originated) and `HL7/fhir-ig-publisher#1220`, which the resolution adds as a follow-up tracker for rendering the cross-references the new guidance produces.

## Details

**Description:**

When defining a profile, we want to link it to the Logical Map that represents it.

The detailed mapping — at element level — between the LM and profile can be established with ConceptMap.

This request is for a simple indication that "this profile implements logical model X". This could be expressed by either/any of:

- a new extension on the profile
- a ConceptMap (mapping the top elements)
- mapping inside the resource metadata

If the preferred approach is a ConceptMap, this may be interesting to document in the logical models page. If the preferred approach is an extension, one should be defined.

**Resolution Description (proposed):**

Add a section in the "notes" area of StructureDefinition called **"Approaches for Mapping between models"** with the following decision-tree text:

> There are a number of different ways to express mappings between data elements in one StructureDefinition to those in a different StructureDefinition, each offering varying levels of complexity and capability.
>
> - If you want to be able to computably convert between two structures and check validity of those maps — **StructureMap**.
> - If you want an independently maintained assertion of how elements in one model correspond to another model without necessarily having the rigor to transform between them — **ConceptMap**, with each of the structural models expressed as *implicit code systems* (codesystem.html#implicit-code-systems).
> - If you're saying "this corresponds to concept x" (where x is a code from a code system or a data element from a library) — **ElementDefinition.code**.
> - If you want to maintain mappings inside one of the structures, but the target isn't a simple code or element id — **ElementDefinition.map**, ideally with a language of FHIRPath if you want rougher 'equivalence' mappings, using `map.comment` for guidance beyond what FHIRPath can express.

In addition, [HL7/fhir-ig-publisher#1220](https://github.com/HL7/fhir-ig-publisher/issues/1220) was opened to deal with the rendering of the cross-references this guidance encourages.

**Comments:**

- **corey_spears (2024-04-15):** Asked the reporter which approach he prefers and to articulate pros/cons of the listed options.
- **costateixeira (2024-04-15 / 2024-04-16 / 2024-04-29):** Re-stated the use case as "StructureDefinition B realizes/implements StructureDefinition A" for IG documentation, cross-IG traceability ("which profiles need to change if a logical model changes"), and possibly diagrams. No personal preference between extension, ConceptMap, or mapping-inside-the-resource; lets tooling drive the choice. Notes: extension is visible and easy to author; ConceptMap requires a new resource but allows richer relationship qualifiers; mapping-in-resource works as in core spec patterns provided tooling reads it across IGs. Whatever is chosen "should be easy to use in an IG" and the IG profile page should render it. Open question whether other resources have an analogous "implements" relationship to data elements / code-system concepts.
- **mckenzie (2024-05-06):** Asked why not use `ElementDefinition.mapping.map`, the historical mechanism.
- **costateixeira (2024-05-06):** OK with `ElementDefinition.mapping` if tooling (notably the IG publisher) processes it and surfaces "this profile implements logical model X" when a top-level mapping exists.
- **ahenket (2024-05-20):** Reports prior practice of putting mappings inside the profile (downside: every mapping change is a profile change, awkward for external profiles such as IPS, where they end up creating a derived profile just to add mappings). Mappings outside the profile have been considered before but were rejected because there was no good way to render the relationship. Notes that a derived profile holding only mappings is effectively the same as the proposed ConceptMap, but with more context and validation.
- **mckenzie (2024-05-20):** Two-part proposal — (1) add a decision tree in StructureDefinition R6 covering StructureMap (computable conversion + validity), `ElementDefinition.code` (correspond-to-concept), and `ElementDefinition.map` with FHIRPath / `map.comment` (with ConceptMap as alternative); (2) tooling should produce, per element, the list of models that map to it and the list of models it maps to, and validate the maps point to valid paths. ConceptMap *could* gain validation if elementIds were validatable as codes.
- **mckenzie (2024-05-20):** Notes ConceptMap could be validated if elementIds from a profile were treatable as codes from an implicit code system.
- **ginocanessa (2024-05-23):** Discussed at WGM; reporter volunteered to mock up each proposal for follow-up discussion.
- **costateixeira (2024-05-23):** Cross-product harmonization considerations: mappings may carry "broader than" / "narrower than" qualifiers; some elements may map to nowhere (is that distinct from "no map"?).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution exactly as drafted: in `HL7/fhir`, under the StructureDefinition resource page, add a new notes-area section titled **"Approaches for Mapping between models"** containing the four-bullet decision tree (StructureMap / ConceptMap-with-implicit-code-systems / `ElementDefinition.code` / `ElementDefinition.map` + FHIRPath + `map.comment`). Land the change in `source/structuredefinition/structuredefinition.html` (or the resource's `*-notes.html` companion if present) and add a cross-link from the Logical Models page (`source/logical.html`) so the guidance is discoverable from the place implementers actually start. Do **not** define a new "implements" extension and do **not** mint a new ConceptMap convention beyond pointing implementers at *implicit code systems* for structural models. Track IG-publisher rendering of the resulting cross-references in the already-opened [HL7/fhir-ig-publisher#1220](https://github.com/HL7/fhir-ig-publisher/issues/1220).

#### Justification

This disposition is consistent with the late-2024 / early-2025 WG discussion captured in the comments: McKenzie's two-part proposal (decision tree + tooling), Costa-Teixeira's acceptance of `ElementDefinition.mapping` as the practical carrier provided tooling renders it, and Henket's observation that an external "mapping-only derived profile" is functionally equivalent to the proposed ConceptMap convention. It uses mechanisms that already exist in the spec (`StructureMap`, `ConceptMap`, `ElementDefinition.code`, `ElementDefinition.mapping`) and avoids introducing a new normative artefact (extension) into FHIR Core for a problem the WG has settled is documentation-shaped rather than data-model-shaped. It also matches the briefing's *Update narrative / prose* recipe — a self-contained HTML edit in `source/` with no generated areas, no cross-repo coupling beyond the publisher follow-up.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the decision-tree text from Disposition A **and** define a first-class extension `http://hl7.org/fhir/StructureDefinition/structuredefinition-implements` (cardinality 0..\*, value `canonical(StructureDefinition)`) authored in **`HL7/fhir-extensions`**, not in core. Update the new "Approaches for Mapping between models" notes section in `HL7/fhir` to add a fifth bullet pointing implementers at this extension when their need is *only* the top-level "Profile B implements Logical Model A" assertion (no element-level mapping). The IG publisher (`HL7/fhir-ig-publisher`, follow-up #1220) would then have one canonical, easily-detected element to render the link from. Optionally allow `code` slots on the extension (e.g., `realizes` / `implements` / `derived-from`) so the relationship type is explicit, addressing Costa-Teixeira's "broader than / narrower than" comment without requiring a full ConceptMap.

#### Justification

The discussion shows broad agreement that the current mechanisms can carry the information but no agreement that they are *easy* — Henket's point that an external relationship "was unsolvable back in the day" because nothing rendered it, and Costa-Teixeira's repeated "whatever is the mechanism, it should be easy to use in an IG" both point at a discoverability problem more than a modelling gap. A dedicated extension is the lowest-friction mechanism: cheap to author, trivially indexable by the IG publisher, and avoids the *"every mapping change is a profile change"* drawback Henket flagged for inline `ElementDefinition.mapping` (the extension lives on the profile root and changes only when the *target* changes). Per the HL7/fhir briefing, new extensions on core resources are authored in `HL7/fhir-extensions`, not in `HL7/fhir`, so this disposition keeps core spec edits limited to the documentation change from Disposition A; the briefing's authoring-root rule is therefore not violated.

---

### Disposition C: Decline

#### Proposal

Decline as *already addressed by existing mechanisms*. Close the ticket with a pointer to `ElementDefinition.mapping` (and `ConceptMap` for the cross-structure case), and do not add new normative content to StructureDefinition. The IG-publisher rendering question becomes a pure tooling matter to be handled in `HL7/fhir-ig-publisher` (#797 / #1220).

#### Justification

A purely doctrinal reading of the comments — particularly McKenzie's "Is there a reason to not use `ElementDefinition.mapping.map`? That's what's been done historically." — supports the view that the spec already has the mechanism and the request is really about tooling. Declining keeps the StructureDefinition page from accumulating yet more "guidance about which mapping mechanism to use" prose. However, this position ignores that the WG discussion did not actually converge on "do nothing" — it converged on McKenzie's decision-tree proposal. Decline is the weakest of the three; it is included for completeness and to give the WG a clean "no-change" option to vote against.

---

### Recommendation

**Recommended disposition:** A

The resolution as drafted (Disposition A) directly reflects what the WG converged on in the ticket discussion and is already labelled `Ready-For-Vote` with `Persuasive with Modification`. It is a documentation-only change confined to a single specification page in `HL7/fhir` (`source/structuredefinition/...`), with no generated-area edits, no cross-repo coupling beyond the already-opened publisher follow-up (`HL7/fhir-ig-publisher#1220`), and no new normative artefacts. It addresses the reporter's actual asks — give implementers a clear answer to "how do I assert that profile B implements logical model A?" and surface the answer on the Logical Models page — while leaving room to escalate to Disposition B (a formal extension) in a follow-up if real-world implementer feedback shows the prose alone is not enough for the IG publisher to render the relationship reliably. The WG should adopt A now, monitor adoption via the publisher rendering work, and only revisit the extension proposal if the decision-tree-only approach proves insufficient.
