## Summary

FHIR-44403 is one of a large family of templated change requests filed by
Ron G. Parker (TSMG) in January 2024 asking individual HL7 publications to
re-anchor their locally-defined `CodeSystem` and `ValueSet` URLs in
`terminology.hl7.org` (THO) rather than in the publication's own canonical
namespace. The request is grounded in the TSC-approved
[Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy. As filed against the **FHIR R5 Subscriptions Backport (FHIR)** IG,
it asks the FHIR Infrastructure work group to either (a) initiate the UTG
process to move any IG-defined code systems / value sets into THO, or (b)
formally request an exception from TSMG when a code system is judged to be
too tightly bound to the IG to live in THO.

The cross-reference index returns no direct Jira links, no Zulip threads,
and no GitHub items for FHIR-44403, but related-by-keyword surfaces the
sibling bulk tickets — most notably the originating-style ticket
[FHIR-44228](https://jira.hl7.org/browse/FHIR-44228) (Clinical Genomics /
Genomics Reporting), where the same template has accumulated substantive
TSMG / work-group discussion (review against the Terminology Expectations
checklist, "Considered for Future Use", deferral, re-open after
publication, etc.). That sibling thread is the best available indication
of how comparable work groups have triaged this template; FHIR-44403 has
not yet received any of that discussion in-ticket.

## Proposed Dispositions

> Note: The ticket is filed against the **FHIR R5 Subscriptions Backport
> (FHIR)** IG. That IG itself does not introduce its own clinical
> terminology — its content is dominated by `SubscriptionTopic`
> definitions, `Subscription` channel-type extensions, and notification
> bundles. Any IG-local `CodeSystem` / `ValueSet` resources are
> infrastructure-level (e.g. backport-status, channel-type, payload-type
> kinds of vocabularies). The dispositions below are framed accordingly.
> No `repo-analysis/briefing.md` was loaded because no GitHub
> cross-references exist for this ticket; if work proceeds, a briefing
> for `HL7/fhir-subscription-backport-ig` should be generated first so
> the actual `input/resources/CodeSystem-*` / `ValueSet-*` files and
> their canonical URLs can be enumerated.

### Disposition A: Accept As Requested

#### Proposal

For each `CodeSystem` and `ValueSet` defined inside the R5 Subscriptions
Backport IG (canonical prefix
`http://hl7.org/fhir/uv/subscriptions-backport/CodeSystem/...` and
`.../ValueSet/...`), open a UTG submission to relocate the resource into
THO under `http://terminology.hl7.org/CodeSystem/...` /
`.../ValueSet/...`. Update the IG content to reference the new THO
canonicals. For any code system the WG judges to be too tightly coupled
to the IG (e.g. backport channel-type vocabularies whose semantics are
defined entirely by this IG's narrative), file an explicit TSMG
exception request via the TSMG listserv / Zulip / meeting, capturing
the rationale on this ticket before resolving.

#### Justification

This literally satisfies the TSC-approved Terminology Expectations for
IG Developers and aligns FHIR Infrastructure with how FHIR Core itself
treats THO-anchored vocabularies. It eliminates the canonical-URL
ambiguity for downstream implementers who already pull terminology from
THO via `tx.fhir.org`. Where a vocabulary genuinely belongs in the IG
(e.g., it is defined only by this IG's prose), the explicit-exception
path keeps the audit trail TSMG asks for.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-44403 as a coordination ticket rather than an item-by-item
work order. Inventory the IG's `CodeSystem` and `ValueSet` resources
(scan `input/resources/` in the IG repository), classify each into one
of three buckets:

1. **Move to THO** — anything that could plausibly be reused outside
   this IG, file a single batched UTG submission for the set.
2. **Keep in IG with documented exception** — vocabularies whose
   semantics are inseparable from a `SubscriptionTopic`/channel-extension
   definition in this IG; record the exception rationale as a comment
   on FHIR-44403 and notify TSMG once.
3. **Retire / replace** — anything already redundant with an
   R5-canonical or THO vocabulary; replace the reference instead of
   migrating.

Resolve FHIR-44403 once the inventory + dispositions are recorded; spawn
narrower follow-up tickets only for the bucket-1 items that actually
need UTG paperwork.

#### Justification

The ticket as filed is template-level and gives no indication of which
specific IG resources motivated it. Producing the inventory first
prevents the WG from doing UTG work on vocabularies that should simply
be retired, and avoids a per-resource explosion of follow-up tickets.
The bulk-filing pattern (FHIR-44228 / 44230–44237 / 44247 / 44382 /
44390 / 44392 / 44411 / 44412 / 44418 …) suggests other WGs would
benefit from the same inventory-first pattern, and the discussion on
FHIR-44228 shows that an answer-the-checklist approach is what TSMG
expects when an exception is claimed.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-44403 "Persuasive with Mod / Considered – Not Persuasive"
with the rationale that the R5 Subscriptions Backport IG's IG-local
vocabularies are tightly bound to the backport mechanism itself
(channel-type semantics, backport-status flags, etc.) and therefore
qualify for the exception path described in the Terminology
Expectations document. Document the exception rationale as a single
comment, notify TSMG via the listserv/Zulip channel referenced in the
ticket, and close. No source-of-truth migration to THO is performed.

#### Justification

The Subscriptions Backport IG exists specifically to back-port an R5
mechanism into R4/R4B; many of its IG-local vocabularies have no
meaningful life outside that backport context. Moving them into THO
would create maintenance overhead (UTG submissions tied to IG
publication cadence) without producing reuse value, which is exactly
the situation the Terminology Expectations document's exception clause
contemplates. FHIR-44228's history shows precedent for a "tightly
coupled to IG, keep in-IG" outcome being acceptable to the WG.

---

### Recommendation

**Recommended disposition:** B

The literal request (Disposition A) presupposes that every IG-local
vocabulary belongs in THO, which is rarely true for an
infrastructure-style IG like Subscriptions Backport, and Disposition C
risks blanket-declining the request without first checking whether any
of the IG's `CodeSystem`s actually do warrant a THO move (the kind of
mistake the FHIR-44228 discussion was unwinding). Disposition B —
inventory first, then split into move / exception / retire — is the
cheapest way for FHIR Infrastructure to give TSMG a defensible answer,
matches the question-by-question checklist approach Cerk Pavkovic walked
through on FHIR-44228, and yields the audit trail TSMG asks for in the
exception path. As a concrete first step, generating a
`repo-analysis/briefing.md` for the Subscriptions Backport IG repository
will let a follow-up `ticket-plan` invocation enumerate the exact
`input/resources/CodeSystem-*` / `ValueSet-*` files in scope.
