## Summary

The reporter asks the FHIR Infrastructure work group to publish the FHIR JSON
Schema artifacts as plain, per-resource JSON Schema documents (conforming to
the JSON Schema specification at <https://json-schema.org/specification>),
rather than embedding/bundling them inside an OpenAPI document. They observe
that the published schema currently advertises `"$schema":
"http://json-schema.org/draft-06/schema#"` while also using OpenAPI-only
constructs such as `discriminator` / `mapping`, which are not part of JSON
Schema draft-06 and therefore make the artifact non-conformant to the draft it
claims to follow. The proposal is either (a) to ship the per-resource JSON
Schemas individually inside a zip (as the spec already does for XSDs), or (b)
if an OpenAPI-flavoured Schema variant is intended, to make that a separately
labelled download so consumers using vanilla JSON Schema tooling get a clean
artifact. No cross-referenced Jira tickets were found, so this stands on its
own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the FHIR JSON Schema artifact as two distinct outputs of the publication
pipeline and ship both:

1. A pure JSON Schema variant that is conformant to the declared
   `$schema` (currently draft-06). Strip OpenAPI-only keywords
   (`discriminator`, `mapping`, and any other OAS-specific extensions) from
   this variant. Continue to define resource selection by `resourceType` using
   only JSON Schema mechanisms (e.g., `oneOf` / `anyOf` over the
   resource-type subschemas with `const` constraints on `resourceType`, which
   is already how draft-06 validators discriminate). Publish per-resource
   files inside a zip, mirroring how XSDs are already distributed.
2. The existing OpenAPI-flavoured combined schema, retained but relabelled as
   the "OpenAPI Schema" download on the downloads page so consumers can pick
   the variant matching their tooling.

#### Justification

The ticket is, on the face of it, correct: declaring `$schema:
http://json-schema.org/draft-06/schema#` while using OpenAPI-only keywords
makes the published artifact non-conformant to the draft it advertises, which
is a real interoperability concern for implementers using vanilla JSON Schema
validators (Ajv, jsonschema, NJsonSchema, etc.). Splitting the artifact into
two clearly-labelled variants honours the original publishing intent
(supporting both audiences) without forcing OpenAPI semantics on consumers
who only want JSON Schema. It also keeps the OpenAPI variant available for
tooling chains (Swagger/OpenAPI generators) that already depend on it.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a single combined `fhir.schema.json` artifact, but make it conformant by
either (i) upgrading the declared `$schema` to a JSON Schema draft that
natively supports the discrimination pattern being expressed (e.g.,
2019-09 / 2020-12, where the same intent can be expressed with `if`/`then`,
`oneOf`+`const`, or vocabulary extensions and where unknown keywords are
ignored rather than treated as semantically meaningful), and at the same time
(ii) restructuring the resource selection to use only standard JSON Schema
keywords (drop `discriminator`/`mapping`; replace with a `oneOf` over each
resource subschema with a `const` constraint on `resourceType`). The result
is one artifact that validates correctly under any modern JSON Schema
validator, and that no longer mis-declares its dialect.

#### Justification

The reporter's underlying complaint is "the schema doesn't conform to the
spec it points at", not "I need two files". A single, correctly-declared
JSON Schema avoids the maintenance overhead of producing and version-aligning
two parallel artifacts and avoids fragmenting the implementer ecosystem
between "JSON Schema FHIR" and "OpenAPI FHIR". The discrimination behaviour
the OpenAPI keywords were providing is already expressible with stock JSON
Schema constructs; OpenAPI tooling that consumed `discriminator`/`mapping`
generally also accepts the equivalent `oneOf`/`const` pattern. The trade-off
versus Disposition A is more invasive change to the schema generator and a
breaking change for consumers who specifically relied on the
`discriminator`/`mapping` keys, but the long-term result is cleaner and avoids
shipping two artifacts forever.

---

### Disposition C: Decline

#### Proposal

Close as "Working as intended": the FHIR JSON Schema is and has always been a
hybrid artifact intended to be consumed both by JSON Schema validators
(which ignore unknown keywords) and by OpenAPI tooling (which uses
`discriminator`/`mapping`). The existing pages at
<http://hl7.org/fhir/json.html#schema> document this. Implementers who want
strict draft-06 conformance can post-process the published file to strip the
OpenAPI keywords; HL7 should not duplicate publication artifacts to do so
on their behalf.

#### Justification

JSON Schema validators are required to ignore keywords they don't recognise,
so the presence of `discriminator`/`mapping` does not in practice cause
validation to fail in compliant validators — it just means those keywords are
inert. Producing and maintaining a second variant of the schema across every
release adds publication-pipeline complexity, doubles the surface for bugs
(divergence between variants), and creates a permanent versioning question
("which variant is canonical?") for downstream tooling and IGs. The change
request does not cite a concrete validator or use case that fails today; it is
framed as a conformance-purity concern.

---

### Recommendation

**Recommended disposition:** A

The reporter is identifying a real conformance defect (OpenAPI keywords
inside an artifact that declares itself as JSON Schema draft-06), and the
"validators ignore unknown keywords" defence in Disposition C is weaker than
it sounds — `discriminator` is a *semantic* keyword in OpenAPI, and shipping
it inside a file that advertises a JSON Schema dialect that does not define
it invites silent divergence between what the schema says and what
validators do. Disposition B (one corrected artifact) is technically the
cleanest answer, but it is a breaking change for the OpenAPI-tooling
audience that has been quietly relying on `discriminator`/`mapping` and it
would force a coordinated migration on every implementer in that camp.
Disposition A — publish two clearly-labelled variants and continue the
existing OpenAPI-flavoured download for backwards compatibility — preserves
both audiences, fixes the conformance problem the reporter raised, and is the
least disruptive path forward. The work group should also use this ticket as
the prompt to revisit the declared `$schema` URI on the OpenAPI variant so
that *it* is honestly labelled (e.g., as the OpenAPI Schema dialect) rather
than as draft-06.
