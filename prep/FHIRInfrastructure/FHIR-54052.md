## Summary

The reporter flagged a grammatical/typographical error in the `_summary`
section of the FHIR Search page (FHIR Core, ballot 6.0.0-ballot4). The
existing sentence — "Servers SHOULD mark the by populating `meta.tag`
resources with the code `SUBSETTED` to ensure that the incomplete
resource is not accidentally used to overwrite a complete resource" — is
broken: it is missing the object of "mark" and reads as if `meta.tag` is
qualifying "resources" rather than the element being populated. The
ticket has been Triaged with resolution **Persuasive** and proposes a
clear rewording that preserves the normative intent (SHOULD) while
correctly identifying the resource as potentially incomplete and
attaching the `SUBSETTED` tag to `meta.tag`. There are no
cross-referenced Jira tickets, Zulip threads, or GitHub items associated
with this ticket.

## Details

**Description:**

> "Servers SHOULD mark the by populating `meta.tag` resources with the
> code [`SUBSETTED`](https://terminology.hl7.org/CodeSystem-v3-ObservationValue.html#v3-ObservationValue-SUBSETTED)
> to ensure that the incomplete resource is not accidentally used to
> overwrite a complete resource".

The reporter notes this is a typo in the FHIR Search page's `_summary`
section.

**Resolution Description (already proposed in the ticket):**

Change the text from:

> Servers SHOULD mark the by populating `meta.tag` resources with the
> code [`SUBSETTED`](https://terminology.hl7.org/CodeSystem-v3-ObservationValue.html#v3-ObservationValue-SUBSETTED)
> to ensure that the incomplete resource is not accidentally used to
> overwrite a complete resource.

to:

> Servers SHOULD indicate the resource as potentially incomplete by
> populating `meta.tag` with the code
> [`SUBSETTED`](https://terminology.hl7.org/CodeSystem-v3-ObservationValue.html#v3-ObservationValue-SUBSETTED)
> to ensure that the incomplete resource is not accidentally used to
> overwrite a complete resource.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the editorial fix exactly as written in the ticket's resolution
description. In the FHIR Core specification source (`HL7/fhir`), edit
the Search page authoring file that defines the `_summary` section
(under `source/search/`) and replace the sentence:

> Servers SHOULD mark the by populating `meta.tag` resources with the
> code `SUBSETTED` to ensure that the incomplete resource is not
> accidentally used to overwrite a complete resource.

with:

> Servers SHOULD indicate the resource as potentially incomplete by
> populating `meta.tag` with the code `SUBSETTED` to ensure that the
> incomplete resource is not accidentally used to overwrite a complete
> resource.

No schema, StructureDefinition, ValueSet, or example artifacts are
affected; this is a pure prose change in the published Search page.

#### Justification

The current sentence is grammatically broken ("mark the by populating
…") and structurally misleading because it reads as though `meta.tag`
modifies "resources" rather than naming the element being populated. The
proposed rewording (a) restores grammatical correctness, (b) makes the
subject of the SHOULD explicit ("indicate the resource as potentially
incomplete"), and (c) preserves the normative force, the target element
(`meta.tag`), the code (`SUBSETTED` from v3-ObservationValue), and the
rationale verbatim. The ticket has already been triaged Persuasive,
which signals workgroup agreement that the existing wording is
defective.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the fix but tighten the wording further to align
with FHIR's standard phrasing for tagged-resource semantics. Replace
the sentence with:

> Servers SHOULD tag responses returned with `_summary` as potentially
> incomplete by adding the [`SUBSETTED`](https://terminology.hl7.org/CodeSystem-v3-ObservationValue.html#v3-ObservationValue-SUBSETTED)
> code to `Resource.meta.tag`, so that the incomplete resource is not
> accidentally used to overwrite a complete resource.

Optionally, also link from this sentence to the `meta.tag`
documentation in `resource.html` and to the SUBSETTED entry already
referenced in `terminologies-systems.html`, so readers can navigate
directly to the normative definitions.

#### Justification

Disposition A fixes the grammar but keeps the sentence's information
structure unchanged. Naming the element with its full path
(`Resource.meta.tag`) and tying the action explicitly to "responses
returned with `_summary`" removes the residual ambiguity about *which*
resources should be tagged (only those actually subset by the server in
response to `_summary`, not all resources globally). It also matches
the way other parts of the spec discuss `meta.tag`-based signaling
(e.g., `SUBSETTED`, `SECLABEL`). The cost is a slightly larger
editorial change than what was filed, but it remains a pure prose edit
to the same Search page section.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the change is editorial-only, the broken
sentence is parseable in context, and the workgroup may prefer to defer
prose touch-ups to the next ballot cycle's editorial pass rather than
opening individual technical-correction tickets for typos.

#### Justification

Declining is defensible only if the workgroup has an explicit policy of
batching editorial corrections. There is no spec-level harm caused by
the current wording — implementers familiar with `meta.tag` and
`SUBSETTED` will understand the intent. However, the sentence is
genuinely broken (missing object, mis-scoped modifier), the reporter
has supplied a drop-in replacement, and this is exactly what the
"Technical Correction" issue type is for, so the case for declining is
weak.

---

### Recommendation

**Recommended disposition:** A

The ticket is a low-risk, low-cost editorial fix on a single sentence
in the Search page's `_summary` section. The reporter has already
supplied a clean replacement, and the resolution is marked Persuasive.
Disposition A (accept as requested) lands the fix immediately with the
exact wording the reporter proposed, which minimizes review cycles and
avoids introducing additional editorial decisions that would need to
be re-confirmed by the workgroup. Disposition B is a reasonable
follow-on if the workgroup wants to tighten the phrasing, but it
should not block accepting A — it can be filed as a separate editorial
ticket if desired. Disposition C has no real upside given the trivial
implementation cost.
