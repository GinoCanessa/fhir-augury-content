## Summary

The FHIR Core `languages.html#term` page uses an example/reference that
points to the code `display` from CodeSystem
`http://terminology.hl7.org/CodeSystem/designation-usage`. The reporter
(Michael Lawley) notes that this CodeSystem only exists in HL7
Terminology (THO) version 1.0.0 — it was dropped in THO 2.0.0 and is
absent from the current 3.1.0 release — so the FHIR Spec is effectively
linking to an obsolete artifact. The reporter suggests the intended
forward-compatible code is `preferredForLanguage` from
`http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra`, and asks that
either (a) the spec be updated to point to that newer code with a note
on backward compatibility, or (b) some other forward-compatible
resolution be adopted.

No other Jira tickets are linked through Augury's cross-reference index,
but a 2025-08 Zulip thread on `#terminology > CodeSystem designation-use`
(the only cross-reference) re-opens the same problem from the
cross-version-package angle. In that thread Marc Duteau also points to
the historically-related tickets [FHIR-37440](https://jira.hl7.org/browse/FHIR-37440)
(R4B "claimed-as-THO" CodeSystems list) and
[UP-107](https://jira.hl7.org/browse/UP-107) as the most recent issues
about the CodeSystem itself versus its disappearance.

Gino Canessa opened the topic in August 2025 while cleaning up the
cross-version (XVer) packages: the R5 OperationOutcome CodeSystem uses

Grahame Grieve initially asked "what is the actual problem" and how

re-add the CodeSystem and re-open discussion on additional concepts.

XVer packages until THO republishes them. Grahame later reported that
he had switched his sources to use
`http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra#preferredForLanguage`
in preference to `…/designation-usage#display` and that the change
would propagate through R6 and the IG ecosystem. A follow-up sub-thread
on the right way to convey translations of definitions (designation,
extension, or property) was raised by Grahame and Michael Lawley but
left unresolved.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `languages.html#term` example in the FHIR Core specification
so that the designation `use` no longer references
`http://terminology.hl7.org/CodeSystem/designation-usage#display`. Replace
it with `http://terminology.hl7.org/CodeSystem/hl7TermMaintInfra#preferredForLanguage`
(the code Grahame Grieve has already switched the canonical sources to
use), and add an explanatory note on the page covering:

- the historical use of `designation-usage#display`,
- the fact that the older CodeSystem was only published in THO 1.0.0
  and is not part of current THO releases, and
- guidance for implementers handling legacy content (back-compat
  mapping from `display` to `preferredForLanguage`).

The same substitution should be applied to any other example in FHIR
Core that still references `designation-usage#display` (the ticket
specifically calls out `languages.html`, but a sweep is appropriate).

#### Justification

This is exactly what the reporter asks for, and Grahame Grieve has
independently arrived at the same conclusion in the August 2025 Zulip
thread, where he confirmed he has switched his sources to
`preferredForLanguage` and that the change will propagate through R6
and the IGs. Pointing FHIR Core at a code that exists in current THO
removes the dangling reference, eliminates the `UNKNOWN_CODESYSTEM`
errors that motivated the related XVer cleanup work, and aligns the
specification with the direction the publisher tooling is already
moving.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the root cause as a THO-publication bug rather than a FHIR Core
example bug: ask the UTG curator to re-add (and properly version)
`http://terminology.hl7.org/CodeSystem/designation-usage` in current
THO — Marc Duteau's stated next step in the Zulip thread — and update
the FHIR Core `languages.html#term` example only to pin to a specific
THO version (or to clarify provenance), without changing the code
itself. Optionally, retain `designation-usage#display` as a deprecated
alias and document its mapping to
`hl7TermMaintInfra#preferredForLanguage`.

#### Justification

`designation-usage#display` is in active use both in core definitions
(R5 OperationOutcome) and in publisher tooling (`ValueSetExpander`
addCode behaviour referenced in the Zulip thread), and the
disappearance from THO 2.0.0+ appears to have been unintentional. Fixing
the omission at the THO/UTG layer preserves backward compatibility for
all artifacts that already use the code, avoids forcing every
downstream IG and cross-version package to migrate, and keeps the FHIR
Core change minimal. The cost is that this disposition does not solve
the original semantic concern (whether `display` or
`preferredForLanguage` is the correct concept going forward).

---

### Disposition C: Decline

#### Proposal

Decline the request as written. Rationale: this is fundamentally a THO
curation problem, not a FHIR Core normative problem. Refer the issue to
the Vocabulary work group (or directly to UTG) to decide whether
`designation-usage` should be re-published, deprecated in favour of
`hl7TermMaintInfra#preferredForLanguage`, or replaced by a different
mechanism (extension or CodeSystem property — the open sub-thread that
Grahame and Michael did not resolve), and close FHIR-37991 as
"addressed elsewhere" once that decision lands.

#### Justification

FHIR Infrastructure does not own the `designation-usage` CodeSystem,
and the right answer to "what code should `designation.use` point at?"
depends on a Vocabulary/UTG decision that is still in flight (Marc's
"shrug and a 'who knows...'", plus Grahame's open question about the
correct way to convey translations of definitions). Making a unilateral
substitution in FHIR Core risks pre-empting that decision and forcing a
second migration if Vocabulary/UTG choose a different resolution. The
cost is that the broken example in `languages.html` lingers in the
meantime.

---

### Recommendation

**Recommended disposition:** A

The August 2025 Zulip thread effectively converges on Disposition A:
Grahame has already moved his sources to
`hl7TermMaintInfra#preferredForLanguage`, that change is propagating
through R6 and the IG ecosystem, and the older `designation-usage`
CodeSystem is not coming back in any controlled way. Updating the
`languages.html#term` example to match what the spec maintainers are
already publishing is low risk, eliminates a known broken reference,
and is consistent with both the reporter's request and the implementer
behaviour emerging on Zulip. Disposition B is attractive on
backward-compatibility grounds but depends on UTG action that has been
stalled since at least 2022; it can be pursued in parallel without
blocking the spec fix. Disposition C unnecessarily defers a fix that
the spec lead has already, in practice, made.
