## Summary

The reporter (Lloyd McKenzie) asks the FHIR Infrastructure work group to add
explicit normative language to the R6 RESTful API specification covering two
points that the community discovered are under-specified. First, when a server's
CapabilityStatement at `[base]/metadata` declares support for a resource type
`foo`, every `GET [base]/foo/[someString]` against that base SHALL behave in
conformance with FHIR's REST rules — i.e. a URL that merely *looks* like a
RESTful URL should not be treated as outside the contract once the server has
claimed conformance for that resource type. Second, the spec should enumerate
the limited circumstances in which HTTP redirects (e.g. 301/302/307/308) are
permitted on a read interaction, and where they are not. The two motivating
examples cited in the ticket are: (a) a canonical URL that happens to be served
from a FHIR endpoint redirecting to the instance representing the most recent
business version (and whether the canonical needing to match the id of the
original version makes a difference), and (b) a merged resource (e.g. a merged
Patient) redirecting to the surviving record. The ticket grew out of a long
`#terminology > VSAC Appending Version to ID` thread where VSAC began returning
resources whose `Resource.id` did not match the `[id]` in the request URL,
revealing that current spec text on REST conformance vs. canonical-URL
resolution vs. redirects is open to multiple readings. There are no
cross-referenced sibling Jira tickets and no linked GitHub issues/PRs.

## Details

**Description:**

From the discussion here:
https://chat.fhir.org/#narrow/stream/179202-terminology/topic/VSAC.20Appending.20Version.20to.20ID

Need to make explicit:

If the conformance statement hosted at `[base]/metadata` asserts support for
resource `foo`, then all GETs performed against `[base]/foo/[someString]` SHALL
behave in conformance with FHIR's REST rules, except that redirects are
permitted in limited circumstances.

Then we need to identify what those circumstances are. E.g.:

- can a canonical URL redirect to the instance that is the most current
  version? Does it matter if the canonical matches the id of the original
  version?
- can a merged resource redirect to the survivor?

**Comments:**

- **grahamegrieve (2023-01-30):** "I don't see how the first part isn't
  already explicit: if the capability statement says something is supported,
  it must be supported conformantly. Clarifying redirects is something we
  should do. I don't see how canonical URLs are relevant in this regard, it's
  a question of how the RESTful API behaves."
- **mckenzie (2023-01-31):** "If I see 'references' to resources, I'd expect
  to be able to perform a read on them. Same detail if I receive a resource
  in a search result. We've always said that servers are allowed to do
  'extra' things beyond what the conformance statement says. I don't think
  we've said anywhere that having a URL that mirrors the syntax of a RESTful
  URL but hasn't been shared in a way that tells you it's a RESTful URL must
  always behave as if it **were** a RESTful URL. If that is, indeed, the
  rule, we need to make that explicit."

## Related Zulip Discussions

### [terminology > VSAC Appending Version to ID](https://chat.fhir.org/#narrow/stream/terminology/topic/VSAC%20Appending%20Version%20to%20ID/near/324800363)

Chris Moesel opened the thread reporting that a `GET
https://cts.nlm.nih.gov/fhir/ValueSet/2.16.840.1.113762.1.4.1221.110` against
VSAC was returning a ValueSet whose `Resource.id` was the requested id with the
business version appended (`...-20210128`), and that VSAC search result Bundles
contained an `entry.fullUrl` of the version-less id while embedding a resource
whose `id` was the versioned one — both of which he read as violations of the
RESTful API spec and of the `Bundle.entry.fullUrl` rules. Robert McClure (VSAC
side) and Bryn Rhodes initially explained the change as needed so that prior
business versions of the resource remain individually addressable. Lloyd
McKenzie distinguished record version (`Resource.meta.version`) from canonical
business version (`Canonical.version`), then took the position that a canonical
URL — even one shaped like a RESTful URL — is not bound by RESTful read
semantics, so resolving it to the current business version (whether by 200 or
by 302) is acceptable. Grahame Grieve pushed back firmly: once the server's
CapabilityStatement at `/metadata` asserts conformance for that resource type,
**every** `GET [base]/[type]/[id]` under that base must conform to the REST
rules regardless of whether the URL also doubles as a canonical, and VSAC's
behaviour is non-conformant. Michael Lawley argued more broadly that
canonical URIs should never have been allowed to look like FHIR RESTful URLs
because exactly this confusion is the result, and that direct GET-resolution
of canonicals is not something tooling should rely on. Peter Jordan, Rob
Dingwell, and Chris Moesel all sided with Grahame's reading. The thread
converged on two open spec questions that became this ticket: (1) should the
spec explicitly state that `[base]/[type]/[id]` GETs are governed by REST
rules whenever the CapabilityStatement at `[base]/metadata` declares that
type, and (2) when (if ever) is a redirect (e.g. 302 to a versioned id, or
to a merge survivor) a conformant response to a read? Grahame indicated that
"we have said that redirects are acceptable on Zulip, but it doesn't appear
to be documented", and Lloyd asked whether explicit language on this should
be added to R5 (now R6 per the ticket label). The thread closed with Chris
reporting that VSAC fixed the GET behaviour (returned id now matches
requested id), with the `Bundle.entry.fullUrl` issue still under
investigation by NLM.

Participants: Chris Moesel, Robert McClure, Lloyd McKenzie, Peter Jordan, Rob
Dingwell, Patrick McLaughlin (NLM), Bryn Rhodes, Michael Lawley, Grahame
Grieve.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative paragraph to the R6 RESTful API page (`http.html`,
"read" interaction section, with a forward reference from the
CapabilityStatement conformance section) stating:

> When a server publishes a CapabilityStatement at `[base]/metadata` that
> declares support for resource type `R` (i.e. `rest.resource.type = R`),
> then every HTTP `GET` against any URL of the form `[base]/R/[someString]`
> SHALL behave in conformance with the rules defined for the corresponding
> FHIR RESTful interaction (read / vread / history-instance), regardless of
> whether the URL was learned from a Reference, a canonical, a Bundle
> entry.fullUrl, an out-of-band link, or any other source. In particular,
> on a successful read the returned `Resource.id` SHALL equal the
> `[someString]` segment of the request URL, as required by the read
> interaction.

Then add a companion subsection ("Redirects on read") enumerating when 3xx
responses are conformant. Concretely:

1. A server MAY respond to a read with `301`, `302`, `307`, or `308` whose
   `Location` header points to another `[base]/R/[id2]` URL on the same
   server, **provided that** the server treats the request as a read and
   the client following the redirect lands on a resource with
   `Resource.id = [id2]`.
2. Two specific reasons for such redirects are explicitly permitted:
   (a) Resolution of a canonical-shaped URL — when `[base]/R/[someString]`
   is also published as a canonical, the server MAY redirect to the
   `[base]/R/[id]` of the instance representing the current business
   version of that canonical (it is *not* required that `[someString]`
   match the id of the first/original version);
   (b) Resolution of a merged record — when the resource at
   `[base]/R/[someString]` has been merged into a surviving record, the
   server MAY redirect to the survivor's `[base]/R/[id]`. For Patient
   merges, the survivor SHALL also link back via `Patient.link` per the
   existing merge guidance.
3. A server SHALL NOT respond to a read with a `200` whose body contains a
   resource whose `Resource.id` differs from the `[someString]` segment of
   the request URL — that case must be expressed as a redirect.
4. Redirects SHALL NOT be used to circumvent search semantics
   (`[base]/R?...`), the `vread` interaction, or operations.

Cross-link this section from the `Bundle.entry.fullUrl` text (which already
constrains `fullUrl` to be RESTful and to agree with `Resource.id`) so that
implementers see a single coherent story.

#### Justification

This is exactly what the ticket asks for and exactly what the Zulip thread
converged on (with Grahame, Chris, Peter, Rob D., and ultimately Lloyd all
agreeing that the rules need to be in the spec rather than only on chat).
The REST-conformance clause restates an interpretation the WG has already
been giving on Zulip — making it normative removes the ambiguity that
allowed VSAC to ship a non-conformant API in good faith. The redirect
clause closes a real implementation gap: today the read interaction does
not list `3xx` among its expected responses, yet redirects are demonstrably
needed for the canonical-resolution and Patient-merge use cases. Tying the
permitted redirect targets to `[base]/R/[id2]` on the same server keeps the
behaviour testable and preserves the `Resource.id` ↔ URL invariant that
the spec already places on `Bundle.entry.fullUrl`.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the REST-conformance clause from Disposition A unchanged, but instead
of normatively defining specific redirect *use cases* in the read section,
take a narrower, more general approach:

1. Amend the read interaction to add `3xx` (`301`, `302`, `307`, `308`) to
   the list of permitted HTTP responses, with text saying: "A server MAY
   redirect a read to another URL on the **same** service base; the
   redirect target MUST itself be a conformant FHIR RESTful URL, and a
   client following the redirect MUST receive a resource whose
   `Resource.id` matches the `[id]` segment of the redirect target."
2. Add a non-normative "Common patterns" callout listing canonical
   resolution and merged-record resolution as recognised reasons to
   redirect, but without making either pattern mandatory or normative.
3. Cross-reference the existing per-resource merge guidance (e.g. Patient
   merge / `Patient.link`) and the canonical-URL resolution discussion
   under `references.html` rather than duplicating that semantics in
   `http.html`.
4. Optionally, introduce a new `CapabilityStatement.rest.resource.interaction`
   extension or documentation note allowing a server to advertise that it
   uses redirects on read so that clients can opt in to following them.

#### Justification

This addresses the underlying need — making the read contract crisp and
sanctioning redirects — without committing R6 to a closed list of
permitted redirect *reasons*. Grahame raised exactly this concern in the
thread: "we sure don't want to require the use of redirects when they're
mostly not necessary" and noted the controversy between "redirected to a
different location for the requested resource" vs. "redirected to a
different resource". A general rule plus non-normative examples is
backwards-compatible (existing servers that never redirect remain
conformant), reduces the surface area of new normative text in R6, and
leaves room for additional future redirect use cases (e.g., resource
relocation across server reorganisations) without spec churn. It also
avoids the awkwardness of having read-interaction text reference
Patient-specific merge semantics, which arguably belongs in the Patient
resource page rather than the cross-resource RESTful API page.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Modification → no spec change
needed** (or, if a change really is desired, restrict it to a single
non-normative clarifying note under `http.html`). The position is that the
existing spec already says what it needs to say:

- `CapabilityStatement` already binds the server's declared interactions
  to the FHIR RESTful API, and the read interaction already says "The
  returned resource SHALL have an id element with a value that is the
  `[id]`."
- `Bundle.entry.fullUrl` already constrains `fullUrl` to be a RESTful URL
  agreeing with `Resource.id`.
- Canonical URLs are explicitly *not* required to be RESTful URLs even
  when they look like one, so resolution behaviour for canonicals is
  out-of-band and need not be standardised in the REST section.
- HTTP semantics already permit `3xx` responses to a `GET`; FHIR does not
  need to re-state HTTP. The VSAC issue was a server bug, was acknowledged
  as a bug by the operator, and was fixed without any spec change.

#### Justification

This is defensible on the grounds that (a) the VSAC incident was resolved
operationally rather than normatively, (b) adding closed lists of
permitted redirect reasons risks excluding legitimate future cases, and
(c) every additional SHALL/MAY in `http.html` raises the conformance bar
for every existing FHIR server. Lloyd's first comment captures this
position: the spec has historically said servers may do "extra" things
beyond what the CapabilityStatement asserts, and importing a
strong "any URL matching the RESTful pattern under [base] SHALL be REST"
rule risks invalidating long-standing server behaviours (he flags one
merge-handling strategy that would become non-conformant).

---

### Recommendation

**Recommended disposition:** B

Disposition B captures everything the ticket actually needs to fix while
preserving the flexibility the work group will want for cases nobody has
thought of yet. The Zulip thread shows clear consensus on two narrow
points: (1) once a CapabilityStatement claims conformance for a resource
type, GETs at `[base]/type/[id]` under that base must obey the RESTful
read contract (this is currently only "obvious" to insiders and was
genuinely under-read by a major terminology server), and (2) redirects
need to be explicitly named as a permitted response so the
canonical-resolution and merge use cases stop being grey areas. Both of
those are addressed by B's normative changes. What B avoids is what
Grahame himself flagged as risky: enumerating *which* redirect reasons
are blessed and which are not, and dragging Patient-merge specifics into
the cross-resource REST page. Disposition A is more directly responsive
to the literal ticket text but bakes in policy choices (e.g., "merged
records MAY redirect to the survivor") that deserve their own focused
discussion at the resource level. Disposition C under-serves the ticket:
the VSAC episode demonstrated empirically that the current text *does*
admit conflicting readings even among long-time WG members, so closing
without any spec change leaves the next implementer to repeat the same
mistake. Recommend the WG adopt B for R6 and, if appetite exists, spawn
a follow-up ticket against the Patient resource page to standardise the
merge-redirect pattern separately.
