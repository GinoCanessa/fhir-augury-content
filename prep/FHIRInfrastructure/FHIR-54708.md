## Summary

The reporter (Jose Costa-Teixeira) asks the FHIR Infrastructure work group to
either **document** or **add** a mechanism on the StructureMap `$transform`
operation for passing arbitrary runtime parameters (key/value pairs) into a
map execution. The motivating use case is populating data such as a transform
timestamp into a `Provenance` (or other) resource produced by the map — i.e.,
information that is not available in the source content but is known by the
caller of `$transform`. The current `OperationDefinition` for
`StructureMap/$transform` defines only `source` and `content`, with no
discoverable channel for caller-supplied parameters, so implementers have no
interoperable way to feed timestamps, correlation IDs, user context, or
similar values into a map.

No cross-referenced Jira tickets, Zulip threads, GitHub items, or Confluence
pages were surfaced for this ticket key by the FhirAugury cross-reference,
search, or `links`/`related` indexes, so the review must rest on the ticket
itself and the two existing comments. Brian Postlethwaite (FHIR-I) notes that
some such values (e.g., evaluation time) can already be obtained inside a map
via FHIRPath (`now()`), and asks for concrete additional examples and a
design for how parameters would be bound to the first group of the map. Bryn
Rhodes points to the `parameters` parameter on `Library/$evaluate`
([CQL IG](https://hl7.org/fhir/uv/cql/OperationDefinition-cql-library-evaluate.html))
as a precedent worth copying — i.e., a repeating part with name/value where
the value can be any FHIR type.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new repeating input parameter `parameters` to the
`OperationDefinition/StructureMap-transform`
([structuremap-operation-transform](https://build.fhir.org/structuremap-operation-transform.html)),
following the shape used by `Library/$evaluate`'s `parameters` parameter:

- Name: `parameters`
- Use: `in`
- Min/Max: `0..1`
- Type: `Parameters`
- Documentation: each `parameter.name` in the supplied `Parameters` resource
  becomes a named input available to the *first group* of the map; each
  `parameter.value[x]` (any FHIR type) is the bound value. Names that do not
  correspond to an input on the first group are silently ignored (or, at
  most, generate an `OperationOutcome` warning) for forward compatibility.

In the StructureMap narrative, add a sub-section to the `$transform`
operation page explaining:

1. How values from the supplied `Parameters` resource are matched by `name`
   to the inputs declared on the first `group` of the map (those whose
   `mode = source` but which are not satisfied by the operation's existing
   `source`/`content` parameter).
2. The interaction with the existing `now()` / `today()` FHIRPath functions —
   recommend `now()` for "time of the transform" and reserve the new
   `parameters` channel for values the map cannot derive itself (e.g.,
   user identity, request-id, environment).
3. Behaviour when a named parameter is supplied multiple times, and when a
   declared first-group input has no supplied parameter (error vs. null).

#### Justification

- Directly satisfies the request and matches a precedent the FHIR community
  has already balloted (`Library/$evaluate.parameters` in the CQL IG, cited
  by Bryn Rhodes in a ticket comment), so reviewers and implementers have
  an existing pattern to copy.
- `Parameters` already supports `value[x]` of any FHIR type, so the
  mechanism is generic enough to cover timestamps, identifiers, codeable
  concepts, references, and primitive flags without further extension.
- Brian Postlethwaite's concern — "how are they passed to the first group?"
  — is answered explicitly by binding by `name` to first-group inputs, which
  is the same convention the language already uses inside the map for
  group-to-group calls.
- Backwards compatible: the new parameter is optional (`0..1`), so existing
  clients and servers continue to work unchanged.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new operation parameter. Instead:

1. **Documentation-only change.** Update the `$transform` operation page to
   spell out that the existing `content` parameter (`Resource`) is the
   correct channel for caller-supplied data, and that callers who need to
   pass *non-source* runtime values today should:
   - Use FHIRPath functions inside the map for environmental values
     (`now()`, `today()`).
   - Wrap their input in a `Bundle` or `Parameters` resource that the map
     itself unpacks (treating one named entry/parameter as the "real"
     source and others as inputs the map reads via FHIRPath).
2. **Define a named extension** — e.g.,
   `http://hl7.org/fhir/StructureDefinition/structuremap-transform-parameter`
   — that callers can add to a `Parameters` resource sent in the request
   body, scoped to the StructureMap mapping language IG (FHIRPath WG /
   FHIR-I), so that the convention can mature outside the core
   `OperationDefinition` before being promoted in a future release.

#### Justification

- Avoids a normative-track change to a widely-implemented operation while
  the use cases are still being enumerated (Brian Postlethwaite's open
  question in the ticket comments).
- Keeps the core `OperationDefinition` minimal; servers that do not
  understand the convention simply ignore an extra `Parameters` body, where
  Disposition A would force every conformant server to grow new
  parameter-binding logic on the first group.
- Preserves a clear migration path: if the extension sees real-world
  uptake, it can be promoted into the operation definition in a later
  cycle with concrete evidence of need.
- Trade-off vs. A: less interoperable in the short term — two
  implementations might pick different parameter-passing conventions — and
  it asks map authors to do un-marshalling work that the engine could do
  for them.

---

### Disposition C: Decline

#### Proposal

Decline the change request as **not yet justified**. Close with a
direction to the reporter to:

1. Use `now()` / `today()` inside the map for time values (per Brian
   Postlethwaite's comment).
2. Encode any other caller-supplied data into the `content` parameter (or
   into a wrapping `Parameters`/`Bundle` passed as `content`) and read it
   from inside the map.
3. Re-open with a concrete list of values that *cannot* be supplied this
   way, plus at least one implementer-confirmed use case, before the WG
   considers a normative change.

#### Justification

- The single concrete example in the ticket (transform timestamp for
  `Provenance`) is already solvable today with `now()`, as Brian
  Postlethwaite pointed out; no other examples have been provided in the
  ticket or in any cached Zulip / GitHub discussion.
- There is **no cached cross-reference evidence** — no related tickets, no
  Zulip threads, no PRs — that the broader community has hit this gap, so
  the request lacks the use-case breadth normally needed to justify a
  change to a normative operation.
- A premature change risks fragmenting first-group input semantics and
  conflicting with future work on the FHIR Mapping Language itself.

---

### Recommendation

**Recommended disposition:** A

The request is small, the design is well-understood, and the FHIR
community already has a balloted precedent (`Library/$evaluate.parameters`)
that one of the comment authors (Bryn Rhodes) explicitly invites the WG to
copy. Adding an optional `parameters : Parameters (0..1)` input on
`StructureMap/$transform` is backwards compatible, generic enough to cover
every value type implementers might want to pass in (timestamps, user
identity, correlation IDs, codes), and answers Brian Postlethwaite's
"how do they reach the first group?" question with the same name-binding
convention the mapping language already uses internally. Disposition B's
extension-first approach is reasonable but trades short-term
interoperability for caution that does not seem warranted given how
established the precedent is. Disposition C — decline — is hard to defend
publicly: while the *single* example in the ticket can be worked around
with `now()`, denying the underlying capability simply pushes implementers
toward bespoke wrappers, which is the opposite of what FHIR
Infrastructure normally aims for.

The work group should, however, take Brian Postlethwaite's prompt
seriously and ask the reporter (and the wider Mapping Language stream) for
two or three additional concrete examples *before* drafting the operation
change, so the new parameter's documentation can give implementers
worked-out examples beyond timestamps.
