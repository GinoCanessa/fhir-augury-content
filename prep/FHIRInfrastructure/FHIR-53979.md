## Summary

The reporter (Michael Donnelly) observes that R6 declares all FHIR resources
to have a maturity level of "normative" while explicitly disclaiming that this
says anything about real-world *adoption*. He agrees with the rationale but is
concerned that, in the process, the spec loses an important signal about how
broadly each resource has actually been adopted. He proposes that HL7 define a
mechanism to track adoption on a per-resource basis — for example, which
companies/organizations have implemented a given resource and the countries
where it is in production use. Because R6 itself will be frozen, the data
would necessarily live outside the spec, but each resource page should
**link** to the adoption record so that potential implementers can find it
easily and so the link reinforces that the data is an official companion to
the spec. The reporter notes that Chris Schaut is exploring whether HL7 staff
would be willing to build such a mechanism; if not, the reporter offers to
help build a workable solution using templated Confluence pages. No
cross-referenced Jira tickets, GitHub items, or Zulip threads were surfaced.

## Details

**Description:**

With the publication of R6, my understanding is that we are declaring all
resources in the spec to have a maturity level of "normative," while
explicitly stating that this implies nothing about the level of *adoption* of
any given resource. I understand and agree with the rationale for this
approach.

My concern is that, in so doing, we lose an important signal: a way to
understand how broadly a given resource has been adopted.

I propose that we define a mechanism to track adoption on a per-resource
basis. This could include (but need not be limited to) which companies or
organizations have implemented the resource and the countries in which it is
in use.

Because the R6 spec itself will be frozen, this tracking would necessarily
live outside the spec. However, I believe it is important that each resource
**link** to this information directly, both to make it easy for potential
implementers to find and to reinforce that this is an official companion to
the specification.

I have concrete ideas for how such an adoption-tracking mechanism could work,
but I do not want to dilute this comment by going into design details here.
At a high level, Chris Schaut is exploring whether HL7 staff would be
interested in developing something along these lines. If not, I believe we
could make a workable solution using templated Confluence pages, and I am
willing to help with that effort.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an "Adoption" link to each resource's landing page in the FHIR Core spec
(e.g., on the resource summary box / status block, alongside the existing
maturity / standards-status indicators). The link target would be a
standardized, per-resource page hosted *outside* the frozen R6 spec — most
practically a templated HL7 Confluence space (one page per resource), so
that:

- The link itself is part of the spec build (one entry per resource in the
  resource intro template / publisher metadata), satisfying the reporter's
  requirement that "each resource link to this information directly."
- The actual adoption content lives in Confluence, where it can be edited
  after R6 freeze without re-publishing the spec.
- The page template captures, at minimum: implementing organizations,
  countries of production deployment, and (optionally) link-outs to known
  IGs / profiles using the resource (cross-referencing
  <https://packages2.fhir.org/xig>).

Concretely, this is a publisher / spec-build change in the FHIR Core
publication tooling to emit a resource-adoption URL into each resource page
template, plus a Confluence space initialized with one templated page per
R6 resource. Population of the Confluence pages is self-reported by
implementers, gated by HL7 (or FHIR-I) editorial review.

#### Justification

This is the smallest change that delivers what the ticket asks for: a
direct, per-resource link from the spec to an adoption record, while
keeping mutable adoption data out of the frozen normative artifact. WGM Q2
discussion already reached general consensus that (a) there is value in
surfacing this information, (b) the data would be self-reported and (c) it
is *not* appropriate inside the spec, so "add a link, host the data
elsewhere" is the disposition that aligns with the room. Confluence
templates are a known, low-cost mechanism HL7 already operates and the
reporter has explicitly offered to help build them. It also preserves the
distinction WGM Q2 flagged — profiling != production use — by keeping XIG
(profiling) and the new adoption record (production use) as separate but
both linkable signals.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a bespoke per-resource Confluence page set, surface adoption
indirectly by:

1. Adding to each resource page a single "Known uses" link that points to a
   filtered XIG view (`https://packages2.fhir.org/xig?resource=<Name>`) for
   profiling/IG-based usage signals — this is data we already have and can
   be wired up immediately.
2. Asking HL7 staff (per Chris Schaut's exploration) to scope a proper
   adoption registry as a separate work item, with structured submissions
   (organization, country, contact, resource list, version, production /
   pilot, last-confirmed date), accessed from the same "Known uses" link
   once it exists.

The spec change is therefore *one* generic link per resource ("Known uses
of this resource"), pointing at a redirect that today resolves to the XIG
filter and tomorrow resolves to the staff-built registry, with no further
spec change required when the registry lights up.

#### Justification

WGM Q2 explicitly raised the data-quality concern: self-reported adoption
data is hard to keep accurate, and "what kind of grid?" was an open
question. Standing up a Confluence-templated page set per Disposition A
risks shipping pages that are mostly empty (or stale) for the long tail of
resources, which actively *undermines* the credibility benefit the ticket
is trying to create. Disposition B avoids that by (1) leaning on data we
already have (XIG / profiling) for the immediate spec-side change, and (2)
deferring the adoption registry until HL7 staff can scope it properly,
including governance for accuracy. It also keeps the spec-side commitment
to one stable URL pattern per resource, which is forward-compatible with
either mechanism. Trade-off vs. A: it does not give the reporter the
explicit per-resource adoption page on day one, and it conflates two
distinct signals (profiling and production) behind a single link unless
the destination distinguishes them.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for FHIR Core / R6 spec changes. Recommend the
reporter pursue the adoption registry directly with HL7 staff (the
Chris Schaut conversation already underway) and/or the HL7 Marketing /
Education work groups, and bring back a concrete proposal — including
governance, data model, and a worked example — once a host and steward are
identified. No change to the FHIR Core spec, the publisher, or the
resource page templates at this time.

#### Justification

WGM Q2 noted real concerns about budgeting, location, accuracy, and how to
surface the data usefully — none of which FHIR-I can answer alone. The
ticket itself acknowledges that the content "would not be appropriate to
include in the specification" and that the substantive design work is
happening elsewhere (Chris Schaut + HL7 staff). Adding spec-side links
before the destination exists risks dead links or links to a placeholder
that never gets populated, which damages spec credibility more than the
absence of an adoption signal does. Declining now is reversible: once a
funded, governed registry exists, a follow-up ticket can add the link in a
technical-correction or subsequent release. Risk: the ticket's underlying
need — discoverability of who actually uses each resource — remains
unmet, and the momentum from the WGM Q2 discussion may be lost.

---

### Recommendation

**Recommended disposition:** B

The ticket is asking for two things that are easy to conflate but should
be separated: (1) a stable, per-resource link from the spec to a "where is
this used?" page, and (2) the actual adoption data behind that link. The
WGM Q2 discussion gave clear consensus on (1) — adding a link is fine and
useful — and clear unease on (2) — accuracy, governance, and presentation
are unresolved, and profiling is explicitly *not* the same as production
use. Disposition B lets FHIR-I act on the part it controls (the spec-side
link, immediately backed by XIG which we already have) without
pre-committing to a Confluence-page model that the WGM has not endorsed
and that, if launched half-populated, would actively damage the
credibility benefit the ticket is trying to create. Disposition A is the
right *eventual* shape if and only if HL7 staff decline to build a real
registry; in that case the recommendation should be revisited. Disposition
C is too passive — there is consensus that *something* should be linked,
and shipping the generic "Known uses" link in R6's spec build is cheap
and forward-compatible with whatever the adoption-tracking mechanism ends
up being.
