## Summary

The ticket asks the FHIR Infrastructure work group to clarify how
`OperationOutcome.issue.expression` should be populated when the offending
input is an HTTP request parameter (specifically a search parameter), not an
element inside the request body. The reporter notes that this has become
ambiguous now that `OperationOutcome.issue.location` — the legacy element
that historically supported a "hokey" syntax for naming HTTP parameters — is
deprecated (resolved under [FHIR-17589](https://jira.hl7.org/browse/FHIR-17589)).
With `location` deprecated and `expression` defined as FHIRPath against the
request body, there is no documented, normative way for a server to point at a
search parameter (e.g. an unsupported modifier on `_lastUpdated`). The
reporter cites the canonical "illegal modifier" example
(`operationoutcome-example-searchfail`) as exactly the case that should be
updated to demonstrate the correct pattern. Related work in
[FHIR-45314](https://jira.hl7.org/browse/FHIR-45314) (the new FHIRPath
`pathname()` function) addresses pathing for body elements but does not
resolve the HTTP-parameter case, and
[FHIR-54483](https://jira.hl7.org/browse/FHIR-54483) raises the analogous
problem for FML/StructureMap source locations — together they suggest a
broader gap in how non-instance error locations are conveyed.

## Details

**Description:**

It is not clear how one can use OperationOutcome.issue.expression to express
request parameters (e.g. search parameters) now that OO.issue.location is
deprecated.

In particular, the "illegal modifier" example should be updated here:

https://build.fhir.org/operationoutcome-example-searchfail.json.html

## Related Jira Tickets

### [FHIR-17589: Drop OperationOutcome.issue.location](https://jira.hl7.org/browse/FHIR-17589)

The historical change request that deprecated `OperationOutcome.issue.location`
in favour of `OperationOutcome.issue.expression`. It was resolved
"Persuasive with Modification" with the resolution explicitly to mark
`location` as deprecated and "rework the documentation about the use of the
element." FHIR-54030 is essentially asking the work group to finish that
documentation work for the specific case of HTTP/search parameters, which
`location` used to handle and `expression` does not yet clearly cover.

Reporter: Lloyd McKenzie. Comments by: grahamegrieve.

### [FHIR-45314: Add a new function `pathname()` that returns the location(s) of the focused item(s)](https://jira.hl7.org/browse/FHIR-45314)

Adds a FHIRPath `pathname()` helper that returns the simplified path of a
focused element so engines can populate `OperationOutcome.issue.expression`
from arbitrary FHIRPath evaluations. Status "Applied" (FHIRPath spec, ITS
work group). It directly improves expression-based error pointing for
*body* elements but does not address the HTTP-parameter case raised in
FHIR-54030 — pointing at `?_lastUpdated:exact=…` cannot be expressed as a
FHIRPath against the request body alone. It is the natural reference point
for any disposition that proposes a FHIRPath-based convention.

Reporter: Brian Postlethwaite. Comments by: bryn.rhodes, brian.postlethwaite.

### [FHIR-54483: How should error messages be reported - FML or StructureMap location?](https://jira.hl7.org/browse/FHIR-54483)

Raises the analogous problem for FHIR Mapping Language: when an engine
reports parse/eval errors, where do the locations point — into the FML
source or into the compiled StructureMap — and how is that carried in
`OperationOutcome.issue.expression` (which is "in simple fhirpath format").
Same root issue as FHIR-54030: `expression` is FHIRPath against a FHIR
instance, but the input that actually failed is something else (search
parameters here, FML there). A consistent, documented convention would
benefit both tickets.

Reporter: Brian Postlethwaite.

The original 2018 thread that framed the deprecation of `location` in favour

the same thing we did for location or adding a separate element (allowing

favoured "a single location, a simple fhirpath, with a `%http` context to

re-opens.

Participants: Lloyd McKenzie, Grahame Grieve.

Mona O asked how to associate an OperationOutcome with a specific failed
include reference. Lloyd McKenzie observed that `issue.expression` "the best
that could do is point to the _include parameter" and that the per-reference
detail must go in `issue.diagnostics` text or a custom extension. The thread
implicitly assumes — but does not normatively define — a syntax for naming

the spec leaves implementers to invent their own conventions.

Participants: Mona O, Andy Stechishin, Lloyd McKenzie.

Dennis Patterson asked what to put in `expression` when the offending input
is a JSON-patch body, not a FHIR resource. Grahame Grieve answered that the
"least worst" answer is to write FHIRPath against the JSON request directly
(e.g. `$this[0].op`), and Brian Postlethwaite preferred Option 1 — pointing
at the resource element that ultimately failed an invariant. No consensus
on a single answer; the discussion confirms there is a recurring class of
"input that is not a FHIR instance" cases (search params, JSON-patch, FML)
where `expression`'s contract is ambiguous.

Participants: Dennis Patterson, Grahame Grieve, Brian Postlethwaite.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a documentation / example update against
`OperationOutcome.issue.expression` in FHIR Core. Specifically:

1. In `operationoutcome.html` (Detailed Descriptions for
   `OperationOutcome.issue.expression`), add a normative subsection
   "Pointing at request parameters" that defines a single convention. The
   recommended convention is the one Grahame proposed in 2018: a FHIRPath
   expression rooted at a `%http` (or `%request`) environment variable
   exposing the parsed request parameters, e.g.
   `%http.parameter.where(name = '_lastUpdated').modifier` for an illegal
   modifier on `_lastUpdated`. Document that when the failure is in an
   HTTP-level construct (search parameter, header, modifier) the engine
   SHOULD use this `%http`-rooted form; when the failure is in the request
   body it SHOULD use a normal FHIRPath against the resource (and MAY use
   `pathname()` per FHIR-45314 to simplify it).
2. Update the example `operationoutcome-example-searchfail` (referenced in
   the ticket) so its `issue.expression` uses the new convention instead of
   the deprecated `location`-style "hokey" syntax.
3. Cross-link from the deprecated `OperationOutcome.issue.location`
   description to the new subsection so readers migrating off `location`
   land on guidance, not silence.

#### Justification

This is the change the reporter literally asked for and it closes a
documentation gap that has been open since FHIR-17589 (2018). It is
consistent with the direction Lloyd and Grahame agreed to on the
implementers stream, aligns with how FHIR-45314 positions `pathname()` for
the body case, and gives implementers a single answer instead of forcing
each one to re-invent the convention (as evidenced by the recurring Zulip
threads on `_include` and JSON-patch). It is a clarification, not a
breaking change — `expression` remains a FHIRPath, and the `%http`
environment variable is the standard FHIRPath extension point for
non-instance context.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than overloading `expression` with a `%http` rooting convention,
add a sibling element (or standard extension) on `OperationOutcome.issue`
specifically for naming the offending request parameter — for example
`OperationOutcome.issue.parameter` (0..* string) or a standard extension
`http://hl7.org/fhir/StructureDefinition/operationoutcome-issue-parameter`
carrying the parameter name (and optionally modifier/value). Keep
`expression` as a pure FHIRPath against the request body, never against
HTTP. Update `operationoutcome-example-searchfail` to use the new element
for the bad modifier and leave `expression` empty for that issue.

Tie this to the FML/StructureMap case in
[FHIR-54483](https://jira.hl7.org/browse/FHIR-54483) by defining the
sibling element generically enough (or adding a parallel one) to also
identify a non-FHIRPath source location, so the spec gains a single,
consistent answer for the entire class of "input is not a FHIR instance"
problems.

#### Justification

Keeps `expression` honest as FHIRPath and avoids requiring every FHIRPath
engine that consumes an OperationOutcome to understand a `%http`
environment variable that is not currently defined in FHIRPath itself.
The 2018 thread shows that mechanism was proposed but never specified, so
shipping it normatively in R6 is heavier than it sounds — every validator,
viewer, and SDK would have to implement it. A first-class element / extension
is cheaper to implement, easier to validate, and more discoverable for
implementers reading an OperationOutcome on the wire. Trade-off vs. A: it
adds surface area (a new element / extension) and means `expression` alone
no longer fully describes "where the issue is" for HTTP failures.

---

### Disposition C: Decline

#### Proposal

Decline as already addressed. Position: `OperationOutcome.issue.expression`
is defined as FHIRPath against the request body; for HTTP-level failures
implementers should use `issue.diagnostics` (free-text) and/or
`issue.code` (e.g. `not-supported`, `invalid`) to convey the parameter
problem, and update the searchfail example to drop `expression` for the
bad-modifier issue rather than invent new syntax. No new normative
mechanism is added.

#### Justification

The combination of `issue.code`, `issue.details`, and `issue.diagnostics`
already lets a server tell a client "your `_lastUpdated:exact` modifier is
not supported" in a machine-readable code plus human-readable text. Adding
a new convention or element costs implementer effort across the whole
ecosystem for what is essentially a cosmetic improvement on the error
payload. Defensible if the work group judges that the value of a
machine-pointable parameter location does not justify R6-era spec churn on
a normative resource.

---

### Recommendation

**Recommended disposition:** A

The reporter is pointing at a real, long-standing documentation gap that
the work group itself acknowledged when it deprecated
`OperationOutcome.issue.location` (FHIR-17589) — the deprecation
resolution explicitly committed to "rework the documentation about the
use of the element," and FHIR-54030 is calling in that debt. Disposition A
delivers exactly that: a single documented convention plus a fixed
canonical example, with no new element and no behaviour change for engines
that don't need `%http`. It leverages the already-applied `pathname()` work
(FHIR-45314) for the body case and codifies the `%http`-rooted FHIRPath
that Grahame proposed on the original 2018 thread, so the work group is
ratifying a direction it already converged on rather than starting fresh.
Disposition B is a defensible fallback if the work group is uncomfortable
mandating a `%http` environment variable that FHIRPath itself has not yet
specified — in that case A and B can be merged by adopting the sibling
element *and* documenting that `expression` MAY remain empty for HTTP-only
failures. Disposition C is the least useful outcome: it leaves implementers
in exactly the situation that produced this ticket and the recurring Zulip
threads, and it wastes the deprecation of `location` by not finishing the
migration story.
