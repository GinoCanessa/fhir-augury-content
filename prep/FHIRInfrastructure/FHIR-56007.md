## Summary

The ticket asks the FHIR Infrastructure work group to clarify the
specification's expected behavior for an `OperationDefinition` that does not
declare any `out` parameters. Today the spec is silent on whether such an
operation is even legal, and—if it is—what HTTP response body (if any) the
server should return on success. The reporter encountered the gap in the
`#conformance > OperationDefinition without out-Parameter` Zulip thread,
where contributors offered conflicting expectations (HTTP 204 with no body,
HTTP 200 with an empty body, HTTP 200 with an empty `Parameters` resource,
or HTTP 200 with an informational `OperationOutcome`). Eric Haas also noted
that several existing operations such as `$submit-attachment` and
`$submit-data` are already defined with no `out` parameters and rely on a
plain HTTP 200, which underscores that this is a documentation gap rather
than a missing feature. No related Jira tickets were surfaced by the
cross-reference index. The ticket has been triaged with a preliminary
resolution of *Persuasive with Modification* that proposes to allow either
an empty body or an empty `Parameters` resource on success and to update
both `operations.html#executing` and `operations.html#response` accordingly,
plus a small typo correction ("ne or more" → "one or more").

## Details

**Description:**

It is unclear, how an Operation without specified out Parameters is supposed
to behave or whether having no out parameters is even legal.

See discussion: [#conformance > OperationDefinition without out-Parameter](https://chat.fhir.org/#narrow/channel/179177-conformance/topic/OperationDefinition.20without.20out-Parameter/with/576968618)

**Resolution Description (current draft, Persuasive with Modification):**

After reviewing https://build.fhir.org/operations.html#executing and
https://build.fhir.org/operations.html#response, it is clear that there are
two options for an operation that does not return an `out` parameter when
successfully completing:

- return an empty body
- return an empty `Parameters` resource

Both of the above sections will be updated to clarify the allowed
behaviors. A typo correction is also proposed: change "containing ne or
more output parameters" to "containing one or more output parameters".

## Related Zulip Discussions

### [conformance > OperationDefinition without out-Parameter](https://chat.fhir.org/#narrow/stream/conformance/topic/OperationDefinition%20without%20out-Parameter/near/577161307)

Simone Heckmann opened the thread asking how an operation with no declared
`out` parameter is expected to behave on success—HTTP 200 with an empty
body, an `OperationOutcome`, or something else—and noted that
`hl7.org/fhir/operations.html#response` is vague on the point. Patrick
Werner suggested HTTP 204 (no content) for success and the standard 4xx/5xx
+ `OperationOutcome` for failure. Gino Canessa walked through the
Operations Framework, the `OperationDefinition` resource, and the HTTP
page, observing that the spec only explicitly covers the cases of one
resource-typed `out`, one non-resource-typed `out`, multiple `out`
parameters, and zero `out` parameters as a Named Query (search), so there
is no documented contract for "return nothing" or for OperationDefinitions
that genuinely have no outputs, and he encouraged a Jira ticket. Lloyd
McKenzie said there are valid reasons not to return anything beyond HTTP
200. John Moehrke framed the underlying question as whether `out` should
have minimum cardinality 0 with an empty 200 explicitly allowed in that
case. Brian Postlethwaite was comfortable with HTTP 200 with or without an
`OperationOutcome` (e.g., to convey informational/warning messages). Eric
Haas pointed out that operations like `$submit-attachment` and
`$submit-data` are already defined with no `out` parameters and rely on
HTTP 200 in practice, treating the issue as primarily a documentation
clarification rather than a new behavior. No firm consensus was reached on
a single canonical response shape, but the majority view supports allowing
no `out` parameters and accepting either an empty body or a `Parameters` /
`OperationOutcome` payload on success.

Participants: Simone Heckmann, Patrick Werner, John Moehrke, Gino Canessa,
Lloyd McKenzie, Brian Postlethwaite, Eric Haas.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the current draft resolution verbatim. In `operations.html#executing`
and `operations.html#response`, add explicit normative text stating that an
`OperationDefinition` MAY declare zero `out` parameters, and that on
successful execution such an operation returns either (a) an HTTP 2xx with
an empty body, or (b) an HTTP 2xx with an empty `Parameters` resource. The
existing rules for error responses (HTTP 4xx/5xx with `OperationOutcome`)
remain unchanged. Apply the typo correction in the same edit: change
"containing ne or more output parameters" to "containing one or more
output parameters".

#### Justification

This matches what the spec already implies elsewhere — the Operations
Framework discusses `out` parameters as describing the response payload,
and the "no `out`" case is currently underspecified rather than forbidden.
It also matches deployed practice: Eric Haas confirmed in the Zulip thread
that `$submit-attachment` and `$submit-data` are already defined without
`out` parameters and have been returning a plain HTTP 200 in the field.
Accepting both "empty body" and "empty `Parameters`" on success preserves
the parsing model for clients that always expect a `Parameters` resource
back from `$operations` while not forcing servers to invent a payload they
have no semantics for. The change is purely clarifying and backward
compatible: no existing conformant server or client behaviour is
invalidated.

---

### Disposition B: Alternative Approach

#### Proposal

Same documentation edits as Disposition A, but pin the success contract
more tightly:

1. In `operations.html#response`, state that an operation with no `out`
   parameters returns **HTTP 204 No Content** with an empty body on
   success; an `OperationOutcome` MAY be returned with HTTP 200 if the
   server has informational or warning issues to surface (per Brian
   Postlethwaite's note).
2. Forbid the "empty `Parameters` resource" form, on the grounds that an
   empty `Parameters` carries no information and only adds wire/parse
   cost.
3. Add a sentence to the `OperationDefinition` resource page noting that
   `out` is `0..*` and that a definition with no `out` parameters is
   conformant.

#### Justification

This option produces a single canonical success shape per outcome type
(HTTP 204 for nothing-to-say, HTTP 200 + `OperationOutcome` for
informational/warning messages), which is easier for client implementers
to code against than "either empty body or empty `Parameters`". It is
consistent with general REST guidance — 204 specifically signals "success,
no body" — and with what Patrick Werner expected in the Zulip thread.
However it breaks at least one existing pattern: deployed operations such
as `$submit-attachment` / `$submit-data` are documented to return HTTP 200,
not 204, so this disposition would either need to be relaxed for those
operations or would force a backward-incompatible change to long-standing
guidance, which is the main argument against it.

---

### Disposition C: Decline

#### Proposal

Close the ticket as *Not Persuasive*. Reject the request to change the
specification on the grounds that the existing text in
`operations.html#executing` and `operations.html#response`, read together
with the cardinality of `OperationDefinition.parameter` (`0..*` with
`use = out`), already permits an operation with no `out` parameters and
already permits a server to return any HTTP 2xx response. No normative
change is required; the gap is a documentation comprehension issue best
addressed in implementer guidance or FAQ rather than in the core spec.

#### Justification

The Operations Framework page does not explicitly forbid the no-`out`
case, and `OperationDefinition.parameter` is already `0..*`. Existing
operations such as `$submit-attachment` are already shipping under that
reading, so arguably the spec is permissive enough today. Declining keeps
the normative footprint small. The weakness of this position is exactly
what Simone Heckmann and the Zulip thread demonstrated: experienced
implementers disagreed on the expected behaviour after reading the spec,
which is itself evidence that the current text is insufficient.

---

### Recommendation

**Recommended disposition:** A

Disposition A matches the resolution the work group has already drafted
(*Persuasive with Modification*), is backward compatible with deployed
operations like `$submit-attachment` / `$submit-data`, and resolves the
documented ambiguity that motivated the ticket and the Zulip discussion in
the first place. Disposition B produces a cleaner single contract but
would either contradict or force a churn of existing operation
definitions that already return HTTP 200, which is a poor trade for
marginal client-side simplification. Disposition C is hard to defend
given that multiple senior implementers in the Zulip thread reached
different conclusions from the same text, which is itself prima facie
evidence that the spec needs the clarification. Disposition A is also the
minimal-risk option: it explicitly authorises both response shapes
already seen in the wild, picks up the trivial typo fix, and does not
require touching the `OperationDefinition` resource itself.
