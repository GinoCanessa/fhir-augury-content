## Summary

The reporter notes that the worked example for the Group-level `$add`
operation, published on the *Operations for Large Resources* page
(§1.4.1.1) of the v6.0.0-ballot2 build, references an element named
`actual` in a position where no such element is defined. `Group.actual`
is a real boolean on the Group resource itself, but it is not defined on
the input parameters of the `$add` operation (nor on `Group.member`),
so its appearance in the example is either a stray field or a holdover
from a Group instance that was copy-pasted into the operation payload.
The ticket is logged as a **Technical Correction** and asks that the
example be fixed so it validates against the published `$add`
OperationDefinition and against the Group element list. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found
in the FhirAugury index for this ticket, so the request is being
considered on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the issue as a pure example-payload bug and edit the example shown
in *Operations for Large Resources* §1.4.1.1 (the Group-level `$add`
worked example) to remove the stray `actual` element. Concretely:

- Locate the example Parameters / Group fragment rendered into
  `operations-for-large-resources.html#1.4.1.1` in the FHIR Core build
  (HL7/fhir).
- Strip the `"actual": …` (or XML `<actual …/>`) line from the example
  so that what remains validates against the `$add` OperationDefinition
  and, where Group is shown, against the `Group` StructureDefinition.
- Re-run the publisher to confirm the example now validates clean.

No normative or schema content changes; this is a documentation/example
fix only, appropriate for a Technical Correction.

#### Justification

The reporter has correctly identified that `actual` is not part of the
`$add` operation's parameter set, and including it makes the example
either invalid or, at best, misleading to implementers who reverse-
engineer the operation from the worked example. Technical Corrections
are explicitly the lane for example/text fixes that do not change
behavior, and the change is localized, low-risk, and consistent with
how comparable example fixes are handled across `HL7/fhir`.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only deleting the offending field, audit the entire
Group-level `$add` example end-to-end and republish a corrected,
representative payload:

- Verify each input parameter against the `$add` OperationDefinition
  (parameter names, cardinalities, types).
- Verify any embedded Group fragment against the current `Group`
  StructureDefinition (including whether `Group.actual` should appear at
  the Group level — and, if so, place it there rather than on a member
  entry).
- Optionally add a one-line caption above the example reminding readers
  that the body is a `Parameters` resource and that member entries
  follow `Group.member`'s element list, not `Group`'s.
- Apply the same audit to the sibling `$remove` example on the same
  page so the two examples stay in sync.

#### Justification

If the wrong element slipped into one example, neighboring examples on
the same `operations-for-large-resources` page were likely produced
under the same assumptions and may carry subtler errors that the
reporter did not flag. A small audit is cheap relative to the cost of
implementers writing client code against a defective example, and it
keeps `$add` and `$remove` symmetric so they read as a pair. The cost
versus Disposition A is modest because the change still touches one
page and one publisher run.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the published example is
sufficiently obvious as a copy/paste artifact of a Group instance, that
implementers can be expected to ignore the unrecognized field, and that
no normative behavior is affected. Close as *Not Persuasive* and leave
the example in place until the next regularly scheduled rewrite of the
operations-for-large-resources page.

#### Justification

The defect is cosmetic, the page is on the v6.0.0-ballot2 build and
will be regenerated for the next ballot anyway, and FHIR Infrastructure
does not have unbounded review bandwidth for low-priority example
typos. Examples are explicitly non-normative, so a stray field cannot
make a conformant implementation non-conformant.

---

### Recommendation

**Recommended disposition:** A

This is a textbook Technical Correction: a single, narrowly-scoped
example-payload error on a build-only page, raised by an external
reporter who has cited the exact section and the exact resource page
that disagree. Disposition A fixes precisely what was reported with
minimal risk and minimal reviewer load. Disposition B's broader audit
is attractive but is better filed as a separate, scoped task against
the *Operations for Large Resources* page so the two pieces of work can
be reviewed independently — applying it inside this ticket risks
expanding scope past what the reporter raised. Disposition C should be
rejected: declining a clearly-correct, easy-to-fix example bug erodes
contributor trust on exactly the kind of report the Technical
Correction lane is designed to absorb.
