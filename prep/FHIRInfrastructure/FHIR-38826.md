## Summary

The reporter, Paul Lynch, asks the SDC IG to add a section providing
guidance on "translation" — or, per Brian Postlethwaite, more broadly
"international" — issues for `Questionnaire` and `QuestionnaireResponse`.
Seven specific points are proposed: how to enumerate the languages a
form supports (extensions on `Questionnaire.title` rather than scanning
the whole form); the language the response should be saved in when a
user switches language mid-fill; how `QuestionnaireResponse.language`,
`Questionnaire.language`, the `translation` extension on
`QuestionnaireResponse.item.text` / `answer.valueString` /
`answer.valueCoding.display`, and the preservation of the default
language (via a `translation` extension on the answer) should
interoperate.

The ticket has been the subject of multiple Zulip conversations on
`#questionnaire` and `#Multi-Language Implementation` (see *Related
Zulip Discussions* below) where the SDC project, Vocabulary, and
implementers debated whether all translations may be carried in
`Coding.display`, whether responses can mix languages, and whether the
default-language values must be preserved when the user answers in a
translation. Lloyd McKenzie posted a detailed proposed resolution
(2024-11-28) that would (a) define a new core
`Resource.language.availableTranslation` extension plus a nested
`translationMethod` extension; (b) define three search parameters
(`availableTranslation`, `availableLanguage`, `availableTranslationMethod`);
and (c) add narrative to the SDC profile clarifying the
`Questionnaire.language` / `QuestionnaireResponse.language` semantics
and the use of the `language` and `translation` extensions on
`QuestionnaireResponse.item.text` / `answer.*`. That resolution was
voted "Considered for Future Use" (4-0-0, Postlethwaite/Beda) on
2024-11-28, but was reverted by Lynn Laakso on 2026-03-26 with the note
that the team did not have time to test it before the next release and
Grahame Grieve's broader work on translation is still in progress. The
ticket is therefore re-opened post STU4 and back on the FHIR
Infrastructure / SDC docket.

There are no explicit Jira-to-Jira links and no related GitHub items in
the cross-reference index for this ticket.

## Details

**Description:**

A section should be added to the guide about "translation issues" or
more broadly (as Brian Postlethwaite suggested) "international issues".
In particular there are a number of choices when using the translation
extension on which recommendations and/or expectations should be set:

1. Options for translation should be obtainable by looking at the
   extensions on `Questionnaire.title`; one should not have to search
   all strings in the form looking for translation extensions to come
   up with a list of languages.
2. `QuestionnaireResponse`s should be saved in the language in which
   they were being viewed at the time of the save (except for free-text
   answers). If a user fills out question A in French, switches to
   Spanish, fills out question B, and saves, the saved language of
   both should be Spanish.
3. `QuestionnaireResponse.language` is used to indicate the language in
   which the response was saved.
4. Saving a response in a language involves writing
   `QuestionnaireResponse.item.text`, `item.answer.valueString`, and
   `item.answer.valueCoding.display` in that language (replacing the
   default language's values).
5. When saving answers which have translation extensions on them, the
   answer with all of its extensions are written into the
   `QuestionnaireResponse`.
6. In order to preserve the default language's value for the answers,
   so the `QuestionnaireResponse` can be viewed in the default
   language (again with the exception of free-text answers), the
   default language answer value should be appended in a `translation`
   extension to the answer's list of extensions.
7. `Questionnaire.language` should be used to indicate the default
   language of the `Questionnaire` when translations are provided via
   the translation extension.

## Related Zulip Discussions

### [questionnaire] > [valueCoding with extensions](https://chat.fhir.org/#narrow/stream/questionnaire/topic/valueCoding%20with%20extensions/near/302868113)

The original SDC discussion that motivated FHIR-38826 (started by Paul
Lynch on 2022-09-27). Paul asked what should happen when an
`item.answerOption.valueCoding.display` carries both `rendering-style`
and `translation` extensions and the user picks the option while a
French translation is rendered: (1) is it important to record that the
choice was made via the translation, and (2) does the entire
`Coding` (with extensions) flow into the `QuestionnaireResponse`?
Lloyd McKenzie reported the Vocabulary position: `Coding.display` SHALL
be one of the displays/designations defined by the CodeSystem (or one
of its supplements), so translations would ideally live in a CodeSystem
supplement — but that breaks down for inline `answerOption` lists with
no terminology server. Brian Postlethwaite and Lloyd both suggested
that "good practice" is to copy the whole `Coding` including extensions
into the QR. The thread then pivoted to whether the QR should be
saved in the default language or the language the user saw, with
Jose Costa Teixeira arguing for the latter (so that downstream
resources are also authored in that language). Paul raised the
mixed-language case (a user switches language mid-form) and
`QuestionnaireResponse.language` semantics; Lloyd's position was that
`QR.language` is whatever language the answers are stored in, and that
seeing the QR back in the "default" language is not super-necessary
because re-translating arbitrary string answers is impractical anyway.
Diane and Elliot Silver raised the additional concept of a "response
view language" distinct from the saved language. Paul concluded by
filing FHIR-38826 to capture the recommendations.
*Participants:* Paul Lynch, Lloyd McKenzie, Brian Postlethwaite,
Jose Costa Teixeira, Diane, Elliot Silver.

### [questionnaire] > [Storing default answer language in QR](https://chat.fhir.org/#narrow/stream/questionnaire/topic/Storing%20default%20answer%20language%20in%20QR/near/305232987)

Paul Lynch (2022-10-19) raised the specific problem behind point 6 of
FHIR-38826: if answers are stored in the user's chosen language, the
default-language form of the answer is overwritten and may not be
recoverable from the `Questionnaire` (e.g. when the answer came from
`answerExpression` or `answerValueSet`). His proposal was to append the
default-language value as an extra `translation` extension on the
answer. Lloyd McKenzie pushed back, arguing that for `valueString`
options the answer must literally match the option string, so a chosen
translation effectively becomes a new answer; if the questionnaire uses
`valueString` with translations and the workflow needs cross-language
preservation, the right answer is to translate the `Questionnaire`
itself rather than the QR. Diane warned about false cognates (e.g.
*embarazada* vs. "embarrassed") and supported per-item language
recording via an extension or hidden question. Ye Wang argued from a
programmer's perspective that the original-language string should be
preserved in the QR so that validation, `enableWhen`, and
`calculatedExpression` keep working consistently across languages.
Brian Postlethwaite countered that authors who base expression logic
on raw string content "get what they deserve" if they don't account
for translations, and that the issue is mostly moot for coded answers.
Paul closed the thread with options 1) store the default language in
the QR for `answerOption.valueString` or 2) disallow the `translation`
extension on `answerOption.valueString` (and codeless `Coding` answers)
— preferring (1).
*Participants:* Paul Lynch, Lloyd McKenzie, Diane, Ye Wang,
Brian Postlethwaite.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restore Lloyd McKenzie's 2024-11-28 proposed resolution as the next
candidate disposition for the SDC IG, scoped to the Structured Data
Capture (SDC) (FHIR) specification owned by the FHIR Infrastructure
work group. Concretely:

1. **New core extensions** (would need cross-WG agreement, since
   `Resource.language` is core, not SDC):
   - `availableTranslation` — 0..* extension on `Resource.language`,
     `value[x]` of type `code`, required binding to the FHIR languages
     value set.
   - `translationMethod` — 0..* extension nested inside
     `availableTranslation`, extensible binding with codes
     `translation-extension` and `translation-file`.
2. **New search parameters** on `Resource`:
   - `availableTranslation` (token) — match resources whose
     `availableTranslation` lists the given language.
   - `availableLanguage` (token) — match resources whose
     `Resource.language` *or* any `availableTranslation` matches.
   - `availableTranslationMethod` (combined `token|token`) — match
     resources with an `availableTranslation` of a given language and
     a given method.
3. **SDC IG narrative** — add a "Translation / Internationalisation"
   section that captures:
   - Point 1: enumerate available languages by reading the
     `translation` extensions on `Questionnaire.title` (plus the new
     `availableTranslation` once defined).
   - Points 2/4: when capturing answers, the stored
     `QuestionnaireResponse.item.text` and translatable
     `item.answer.value[x]` (string, `Coding.display`, etc.) SHOULD be
     what the user saw when answering. Per-question granularity is
     RECOMMENDED; systems that can only store one language MAY store
     the last-active rendering language for all items.
   - Points 3/5/6/7: define
     `Questionnaire.language` = majority authoring language;
     `QuestionnaireResponse.language` = primary language of the saved
     response (typically the active language at last save); allow
     systems to determine "primary" otherwise; require
     `Resource.language` when no typical language is implied by
     context.
4. **SDC profiles**:
   - Add the standard `language` extension to the SDC
     `QuestionnaireResponse` profile on `item.text`,
     `item.answer.valueString`, and `item.answer.valueCoding.display`
     (optional). Narrative: if a stored string is in a language other
     than `QuestionnaireResponse.language`, it SHOULD carry the
     `language` extension naming its language.
   - Add the standard `translation` extension to the same elements
     (optional). Narrative: if translations were available during
     capture (from the `Questionnaire`, the expanded `ValueSet`, an
     expression result, etc.) they SHOULD be carried into the QR. If
     the user picked a translation, that picked value SHOULD be the
     root value, with all other translations *and* the original
     authoring-language value carried as `translation` extensions —
     this directly addresses point 6 (preserving the default language
     for read-back).

Land the change in the next SDC publication after STU4 (which is the
window the 2026-03-26 re-open referenced).

#### Justification

This is the proposal the SDC project actually converged on after two
years of Zulip and ticket discussion (see *Related Zulip Discussions*).
Points 1/3/7 are non-controversial. Points 2/4 follow the
already-stated SDC position that QRs should reflect what the user saw.
Point 6 is the critical addition — carrying the default-language value
as a `translation` extension on the answer — and it directly answers
the *"Storing default answer language in QR"* thread without requiring
either side of the Lloyd-vs-Paul argument to give ground: the chosen
translation is the legal answer (so `valueString` validation still
works for Lloyd), and the default-language text is still recoverable
for downstream readers (so Paul's "review responses in another
language" use case still works). The new core
`availableTranslation` / `translationMethod` extensions and the three
search parameters provide a uniform way to ask "which languages does
this resource exist in?" — directly answered as "looking at the title
extensions" by point 1, but generalised so that resources translated
via files (e.g. the validator/IG-publisher pipeline that Lloyd cited
on 2024-10-04) can still be discovered.

### Disposition B: Alternative Approach — SDC-only narrative now, defer the core extensions

#### Proposal

Split the work and land in two stages:

1. **In the next SDC release** (the publication window referenced by
   Lynn Laakso's 2026-03-26 re-open), add only the SDC-scoped pieces
   of disposition A — i.e. the narrative section on translation, the
   semantics for `Questionnaire.language` and
   `QuestionnaireResponse.language`, and the use of the existing
   standard `language` and `translation` extensions on the SDC
   `QuestionnaireResponse` profile (`item.text`, `answer.valueString`,
   `answer.valueCoding.display`), including the rule that the
   default-language value is preserved as a `translation` extension on
   the answer. **No** new core extensions, **no** new search
   parameters, **no** changes outside the SDC IG.
2. **Defer** the proposed `availableTranslation` /
   `translationMethod` core extensions and the three new search
   parameters to a separate ticket against `Resource` /
   FHIR Infrastructure, to be coordinated with Grahame Grieve's
   ongoing translation work (the explicit reason given for reverting
   the 2024-11-28 resolution).

#### Justification

The 2026-03-26 revert called out two concerns: lack of testing time
and conflict with Grahame's in-progress translation work. Both apply
specifically to the *new core extensions and search parameters*; the
SDC-internal narrative does not depend on either and could land
independently. This keeps SDC's STU progress unblocked, gives
implementers the rules they need today (which is what the original
ticket actually asked for — "a section should be added to the guide"),
and avoids prejudging Grahame's design for resource-level translation
metadata. It also aligns with Jose Costa Teixeira's 2022 comment,
which agreed with point 1 in principle but asked to defer the
mechanism until the broader FHIR translation story is clearer.

### Disposition C: Decline — point to forthcoming core guidance instead

#### Proposal

Decline adding new normative material in SDC. Instead, add a short
non-normative paragraph in the SDC IG that:

- Notes that translation/internationalisation of `Questionnaire` and
  `QuestionnaireResponse` is governed by the cross-resource translation
  rules being developed in core FHIR (the work referenced by Lynn
  Laakso's 2026-03-26 comment and Lloyd's 2024-10-03 action item to
  Grahame Grieve).
- Points readers to `https://build.fhir.org/languages.html` and the
  Confluence "How to add translations to the FHIR Validator and IG
  publisher" page already cited by mckenzie on 2024-10-04.
- Defers the seven specific recommendations in the ticket until the
  core guidance is published, at which point SDC will revisit any
  SDC-specific gaps.

#### Justification

The reverting comment is explicit that Grahame's translation work is
"still in-progress" and that SDC will not have time to test the prior
proposal before the next release. Defining SDC-local rules — and,
worse, creating new core extensions and search parameters — that
might conflict with whatever core lands risks creating two parallel
mechanisms that implementers have to reconcile. Declining (with a
forward pointer) keeps SDC consistent with core, is the lowest-risk
option for the upcoming publication, and preserves the option to
reopen once the core guidance exists. The downside is real: the
underlying interoperability questions raised by Paul Lynch in the two
`#questionnaire` threads (point 6 in particular) remain unanswered for
implementers in the interim.

---

### Recommendation

**Recommended disposition: B (SDC-only narrative now, defer core
extensions).**

Disposition B is the proposal that most cleanly threads the needle
between the 2024-11-28 vote (which actually agreed with the substance
of the resolution, 4-0-0) and the 2026-03-26 revert (which only cited
testing time and a conflict with Grahame's in-progress core work).
Both of those concerns attach specifically to the new core
extensions / search parameters in disposition A, not to the SDC-level
narrative; splitting the work lets SDC ship the guidance the ticket
actually asked for ("add a section to the guide"), gives implementers
a concrete rule for the contentious "preserve the default-language
answer" question (Paul's point 6, the focus of the *Storing default
answer language in QR* thread), and avoids pre-empting whatever
resource-level translation metadata Grahame's work eventually lands.
Disposition A is the right *direction* but is exactly the proposal
the WG already declined to ship in this window. Disposition C
under-delivers: the original ticket and three Zulip threads
demonstrate that implementers do need SDC-level guidance now, and
SDC has the local authority to provide it without waiting on core.
