## Summary

The reporter notes that the `SearchParameter` invariant `spd-2` currently restricts use of `SearchParameter.chain` to search parameters whose `type` is `reference`:

> `chain.empty() or type = 'reference'`

However, FHIR has since introduced a new `resource` search parameter type (per the resolution of [FHIR-48363](https://jira.hl7.org/browse/FHIR-48363) — *Invalid search parameters on Bundle*) that is explicitly intended to enable chaining through directly-embedded resources (for example, `Bundle.composition` chaining into the embedded `Composition`). Because `spd-2` was not updated when the new `resource` type was added, every `SearchParameter` of type `resource` that uses chaining technically violates `spd-2`. The reporter requests that the invariant be relaxed to:

> `chain.empty() or (type in ('reference' | 'resource'))`

and that the long Definition text on `SearchParameter.chain` be updated accordingly. They also ask the WG to consider adding `'special'` for symmetry with `spd-3`, while noting they are unsure of a use case.

This ticket is a follow-on to [FHIR-48363](https://jira.hl7.org/browse/FHIR-48363) (Applied, Persuasive), which defined the `resource` parameter type and documented its use for chaining. The closely-related [FHIR-53761](https://jira.hl7.org/browse/FHIR-53761) (also from the same WGM-style review wave) requests narrative documentation improvements for the new `resource` type, of which the spd-2 fix is the formal/computable counterpart. The original `spd-2` invariant traces to [FHIR-3090](https://jira.hl7.org/browse/FHIR-3090) (Published), where the constraint that `chain` must be empty unless type = `reference` was first introduced.

## Details

**Description:**

For SearchParameter, invariant spd-2 requires:

`chain.empty() or type = 'reference'`

This should also include the "resource" data type (for example, Bundle's "composition" search parameter), which are described as using chaining in their definition (https://build.fhir.org/search.html#resource):

`chain.empty() or (type in ('reference' | 'resource'))`

The long Definition for SearchParameter.chain would also need to be updated accordingly.

For consistency with invariant spd-3, you could also consider adding 'special' as a valid type to this list, although I personally am not sure if there's a use case for that.

## Related Jira Tickets

### [FHIR-48363: Invalid search parameters on Bundle](https://jira.hl7.org/browse/FHIR-48363)

This is the parent ticket that motivated the introduction of the `resource` search parameter type. It was resolved Persuasive / Applied with a four-part resolution that (1) defined a new `"resource"` type of search parameter, (2) documented it as targeting directly-embedded resources such as `Composition` in a Document Bundle or `MessageHeader` in a Message Bundle, (3) explicitly documented that the new type is intended for *chaining* (`message.x=y`), and (4) noted that the prior use of `reference` for these parameters was incorrect. FHIR-53764 fixes the loose end this ticket left behind: the `spd-2` invariant was never relaxed to permit `chain` on the new `resource` type, so every `SearchParameter` produced by FHIR-48363's resolution currently fails `spd-2`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `SearchParameter` resource definition in FHIR Core (FHIR R6 / build) as follows:

1. **Invariant `spd-2`** on `SearchParameter`: change the FHIRPath expression from
   `chain.empty() or type = 'reference'`
   to
   `chain.empty() or type = 'reference' or type = 'resource'`
   and update the human-readable description to: *"Chain can only be used when type is `reference` or `resource`."*
2. **`SearchParameter.chain` element Definition / Comments**: extend the long-definition text to clarify that `chain` is also permitted (and meaningful) for parameters of type `resource`, where chaining traverses into a directly-embedded resource (e.g., chaining `Bundle.composition.subject` through the Composition embedded in a Document Bundle entry). Reference `search.html#resource`.
3. **Regenerate** the SearchParameter StructureDefinition / examples and re-run the publication to ensure no current `resource`-typed search parameters (notably `Bundle-composition` and `Bundle-message`) trip `spd-2`.
4. Defer the reporter's optional suggestion about `'special'` (see Disposition B for why).

#### Justification

FHIR-48363 was resolved Persuasive and Applied with the explicit four-point resolution that the `resource` search parameter type *is intended for chaining* (`message.x=y`). The current `spd-2` invariant directly contradicts that resolution: as written, it makes any `SearchParameter` of type `resource` with a non-empty `chain` invalid, which mechanically invalidates the very Bundle search parameters FHIR-48363 was created to fix. Accepting this change closes a known gap left by FHIR-48363 rather than introducing new design. It is also consistent with the historical intent of `spd-2` (FHIR-3090): the rule existed to gate `chain` to parameter types that *have* chaining semantics, and `resource` now has chaining semantics. The change is non-breaking — it relaxes a constraint, so no instance that currently passes `spd-2` can newly fail it.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the `spd-2` relaxation for `'reference' | 'resource'` exactly as in Disposition A, **but additionally** widen the invariant to align with `spd-3`'s expression style:

`chain.empty() or type in ('reference' | 'resource' | 'special')`

i.e., also permit `chain` on `special` parameters. Document in `SearchParameter.chain` that for `special` parameters the meaning of `chain` is implementation-defined (the spec does not assign chaining semantics to `special`, but it does not need to forbid the element either, since `special` exists precisely to cover behavior the core spec doesn't define). Pair this with FHIR-53761's narrative change so the spec text matches.

#### Justification

The reporter explicitly raised `'special'` as an open question for symmetry with `spd-3`. The argument *for* including it is consistency: `spd-3` already enumerates `('reference' | 'resource' | 'special')` for its own restriction, and treating `chain` the same way avoids a small inconsistency in the resource's invariants. The argument *against* — and the reason A was the primary recommendation — is that FHIR has no defined semantics for chaining a `special` parameter, so allowing `chain` there could create implementer confusion and raises a real "what does this mean?" question that today has no answer. Disposition B is appropriate only if the WG is comfortable saying "chain on `special` is implementer-defined" in the chaining narrative; otherwise A is safer.

---

### Disposition C: Decline

#### Proposal

Decline the change. Keep `spd-2` as `chain.empty() or type = 'reference'` and instead address the apparent conflict by either (a) reclassifying the `resource` type as a sub-mode of `reference` for invariant purposes, or (b) ruling that `Bundle.composition` / `Bundle.message` should not advertise chained children at all and treating the chaining behavior described in `search.html#resource` as informative rather than normative.

#### Justification

This disposition is hard to justify. The `resource` type was introduced specifically to support chaining (per FHIR-48363's resolution), so an invariant that forbids chain on it is internally inconsistent with the spec the WG already approved. Declining would either require unwinding part of FHIR-48363 or accepting a permanent gap between the prose at `search.html#resource` and the computable conformance rule on `SearchParameter`. The only reason to choose Decline is if the WG concludes that the `resource` type itself was a mistake and should be reverted — which would be a much larger, breaking change and out of scope for this ticket.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the minimal, non-breaking change that closes a concrete inconsistency between an Applied, Persuasive resolution (FHIR-48363) and a current normative invariant. Every search parameter produced by that resolution — `Bundle-composition`, `Bundle-message`, and any future `resource`-typed parameters — needs this fix to remain conformant, so leaving it unaddressed forces implementers (and our own publication tooling) to either suppress `spd-2` or ship known-failing instances. It should be applied alongside FHIR-53761 (narrative documentation of the `resource` type) so the spec text and the invariant land together. Disposition B's `'special'` extension is reasonable but introduces a real semantic question the WG has not answered; it is worth a brief WG poll, but should not block A. Disposition C is not defensible without reopening FHIR-48363.
