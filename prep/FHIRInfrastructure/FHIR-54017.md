## Summary

The ticket requests cleanup of the narrative in FHIR Core section 2.1.29.4.2.1 ("creating additional resources outside this specification") prior to the R6 ballot. The current prose contains a bare placeholder — "HL7 publishes the list of approved resource at **(somewhere?)**" — followed by a `todo` noting that the "functional" list is currently maintained in the [`FHIR/ig-registry/additional-resources.json`](https://raw.githubusercontent.com/FHIR/ig-registry/refs/heads/master/additional-resources.json) file and that new resources are proposed through the Zulip channel [#Additional-Resource-Proposals](https://chat.fhir.org/#narrow/channel/491058-Additional-Resource-Proposals). The single comment (JIRAUSER24240, 2026-01-12) extends the request to the neighbouring section 2.1.29.4.2.2 ("moving additional resources into core"), where the phrase "FMG needs to approve moving into core (criteria TBD by FMG)" still has unresolved criteria that need to be finalised by FMG before ballot. There are no cross-referenced Jira tickets, Zulip threads, or GitHub items — the scope is narrative/process documentation only.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rewrite the placeholder in section **2.1.29.4.2.1** so the "HL7 publishes the list of approved resource at **(somewhere?)**" sentence is replaced with a concrete, ballot-ready statement that:

1. Names the canonical location as the FHIR Additional Resources registry and links to the rendered registry page (e.g. `https://registry.fhir.org/additional-resources` or the equivalent build page), with a stable secondary reference to the machine-readable JSON at `https://raw.githubusercontent.com/FHIR/ig-registry/refs/heads/master/additional-resources.json`.
2. Points implementers to the [#Additional-Resource-Proposals](https://chat.fhir.org/#narrow/channel/491058-Additional-Resource-Proposals) Zulip channel as the starting point for registering a new additional resource, and links to any published proposal template.
3. Removes the `todo:` marker and the parenthetical "(somewhere?)".

In parallel, finalise the narrative in section **2.1.29.4.2.2** so the promotion-into-core process is concrete: ask FMG to supply the approval criteria (which the comment explicitly flags as "TBD by FMG"), and replace "FMG needs to approve moving into core (criteria TBD by FMG)" with the agreed criteria and the governance route (which WG requests it, what FMG review looks like, what ballot class the newly-promoted resource is introduced under).

#### Justification

The text is already identified as a `todo` in the current ballot source, and FHIR Infrastructure owns the "Resources" framework chapter. Publishing R6 with a literal "(somewhere?)" placeholder and an unresolved `todo:` inside a cited paragraph is not acceptable for a normative/ballot draft; it has to be replaced with something. The functional location (`FHIR/ig-registry`) and the proposal channel already exist and are in operational use, so the edit is documentation-only and does not introduce new machinery — it just codifies what is already happening. Resolving 2.1.29.4.2.2 in the same pass is a natural companion change because the two sections describe the two halves of the same life-cycle (creation outside core → promotion into core) and the comment on the ticket explicitly scopes both.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the editorial intent but split the work:

- **In the spec (2.1.29.4.2.1 / 2.1.29.4.2.2):** Remove the `todo:` and "(somewhere?)" text and replace them with a short, *process-level* description that delegates specifics to an external, versioned governance page. Explicitly:
  - Keep only stable, non-URL-sensitive language in the ballot HTML (e.g. "HL7 publishes the current list of approved additional resources in the FHIR Additional Resources Registry" and "Proposals follow the process described on the Additional Resources governance page").
  - Link to one **governance page** (e.g. `https://confluence.hl7.org/…/additional-resources-governance` or an equivalent `HL7/JIRA-Spec-Comments`-style page) that in turn carries the current registry URL, JSON source, Zulip channel, and FMG promotion criteria.
- **Outside the spec:** File a FMG action item (or companion ticket) to document the promotion-into-core criteria on that governance page, so 2.1.29.4.2.2 can reference it without being blocked on FMG turnaround before R6 ballot freeze.

#### Justification

The specific mechanisms named in the description (an `ig-registry` JSON file, a Zulip channel number) are implementation details of HL7's current tooling, not normative FHIR content. Hard-coding them into the R6 ballot text creates maintenance debt — if the registry moves or Zulip channel IDs change, the spec goes stale between ballots. Indirecting through a governance page keeps the ballot text durable across R6.x publications and isolates the ticket from the FMG "criteria TBD" blocker, which otherwise holds up 2.1.29.4.2.1 for reasons that are really about 2.1.29.4.2.2. It also matches how other FHIR Infrastructure governance topics (IG registration, resource approval) are described in the spec today.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as **Not Persuasive — editorial; already tracked in-source**: the `todo:` and `(somewhere?)` markers in 2.1.29.4.2.1 are existing editorial notes; they can be resolved by the editors during ballot preparation without a workgroup disposition. Close the ticket without a specific substantive change and rely on the R6 ballot editorial pass to remove the placeholders.

#### Justification

The request is explicitly an editorial cleanup of text that is self-flagged as TODO; there is no disputed FHIR semantics, no cross-spec impact, and no cross-referenced discussion. If the editorial pass catches it, there is nothing the workgroup needs to decide. The 2.1.29.4.2.2 half of the comment is only actionable once FMG publishes criteria — which is an FMG deliverable, not a FHIR Infrastructure one — so arguably that half doesn't belong on this ticket either.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the "right" answer in spirit but it pins the ballot narrative to URLs and a numeric Zulip channel ID that are operational tooling, not normative policy. Dispositon C understates the fact that `(somewhere?)` is sitting inside a *cited* paragraph in the ballot draft and will be visible to every R6 reviewer — it really does need to be fixed, not just noted. Disposition B captures the concrete cleanup FHIR Infrastructure can make unilaterally before R6 ballot (remove the `todo:`, point at a stable governance page) while isolating the one piece that genuinely requires FMG input (the promotion criteria in 2.1.29.4.2.2) from the editorial freeze. That keeps the ticket closable within the FHIR Infrastructure work queue, produces durable ballot prose, and still captures the comment's concern about section 2.1.29.4.2.2 via a companion FMG action.
