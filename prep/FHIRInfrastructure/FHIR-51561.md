## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a two-part editorial fix to the FHIR Overview narrative:

1. **Fill out section 2.16.6.3 "Security"** with a short, dedicated
   overview paragraph (a few sentences) covering: FHIR's stance that
   security is layered on top of the API (authentication, authorization,
   audit, consent, integrity/confidentiality), pointers to the relevant
   spec pages — Security, Security Labels, SMART/OAuth-based
   authorization, AuditEvent, Consent, Provenance — and a clear note that
   FHIR itself does not mandate a single security model. Remove the
   `(section to be filled out)` placeholder and the interim
   R4-security.html link in favor of the current-version link.
2. **Promote "Example Use Cases Using FHIR" to a heading** at the same
   structural level as the surrounding overview sub-sections, instead of
   leaving it as a stranded body line immediately under 2.16.6.3.

The text content for #1 should be reviewed/authored by FHIR Infrastructure
in coordination with the Security and Privacy WG, since the spec page it
summarizes is owned jointly.

#### Justification

The section as it stands is overtly broken — both the placeholder note
and the orphaned line are visible defects in published narrative. The
Overview is one of the first pages an implementer reads, so a placeholder
here is disproportionately damaging to first impressions of the spec. A
short summary that simply links into the existing Security page set is
the lowest-risk fix and matches how other Overview sub-sections behave
(short framing, then deep links).

---

### Disposition B: Alternative Approach

#### Proposal

Split into two tickets and resolve only the structural fix in this one:

- In FHIR-51561, **fix only the heading**: promote "Example Use Cases
  Using FHIR" to a proper section heading and, as the minimum fix to the
  placeholder, replace `(section to be filled out)` with a one-line
  pointer of the form *"For security considerations, see the
  [Security](security.html) page."* — using the current-version link, not
  the R4 link. Close FHIR-51561 with that change.
- **Open a follow-up ticket** under FHIR Infrastructure (and label it
  for Security and Privacy review) to author the substantive overview
  paragraph, since drafting it well requires WG sign-off on what the
  Overview should say about security.

#### Justification

Filling the section properly is a content question (what should the FHIR
Overview say about security?) and arguably belongs to the Security and
Privacy WG, not Infrastructure. Splitting it lets the visible structural
defect — the orphan heading and the embarrassing
"section to be filled out" string — be cleared immediately on the next
publication, while the substantive write-up moves through the right WG
without blocking the trivial fix. It also avoids a bikeshed cycle on the
overview wording delaying the heading correction.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-51561 as a duplicate of normal documentation maintenance.
Keep the placeholder pointing at the live Security page as a stop-gap and
treat the orphaned "Example Use Cases Using FHIR" as already-rendered
content; rely on the editorial pass that precedes each ballot to clean
both up.

#### Justification

This is purely editorial, has no normative impact, and there is no
implementer waiting on it. FHIR Infrastructure has limited bandwidth for
narrative rewrites, and editorial cleanup is routinely batched into the
pre-ballot pass without a ticket. There is no use-case argument or
community discussion attached to FHIR-51561 driving it.

---

### Recommendation

**Recommended disposition:** B

Disposition A is correct in spirit, but bundling a substantive
security-overview write-up with a structural heading fix risks the ticket
sitting in Triaged indefinitely while WGs negotiate wording — exactly the
state it has been in since July 2025 with no comments and no movement.
Disposition C ignores the fact that "(section to be filled out)" and an
orphaned line are visible in published narrative, which is a real
embarrassment in a chapter as prominent as the Overview.

Disposition B clears the visible defect immediately with a one-line
pointer plus the heading promotion — a change small enough for an
Infrastructure editor to land on their own — and routes the actual
content authoring to the Security and Privacy WG via a fresh, scoped
ticket where it belongs. It minimizes the time the published spec
displays placeholder text while still making space for a substantive
treatment of the topic.
