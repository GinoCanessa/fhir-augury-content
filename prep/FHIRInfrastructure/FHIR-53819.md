## Summary

The ticket reports a broken / duplicated hyperlink on the FHIR Core
specification home page (`https://hl7.org/fhir/6.0.0-ballot4/index.html`).
In the **Implementation Links** table on that page, the row labelled
*"Useful Tools for Implementers"* currently points to the same Confluence
URL as the *"Public Test Servers"* row — namely
`https://confluence.hl7.org/spaces/FHIR/pages/35718859/Public+Test+Servers`.
The reporter (Abderrazek Boufahja) asks whether this is intentional or a
bug. Effectively, the *Useful Tools for Implementers* entry has lost its
own target and instead duplicates the *Public Test Servers* entry, which
is almost certainly an authoring error in the home page source. There are
no cross-referenced Jira tickets, Zulip threads, or GitHub items; the
only cross-reference is the (incorrect) Confluence page being pointed at.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a confirmed authoring bug on the FHIR Core home page and
fix the *Useful Tools for Implementers* row in the **Implementation
Links** table on `index.html` so that it links to its own Confluence
page (the HL7 Confluence "Useful Tools for FHIR Implementers" / "Tooling"
page) instead of duplicating the *Public Test Servers* URL
(`https://confluence.hl7.org/spaces/FHIR/pages/35718859/Public+Test+Servers`).
The correct target should be the existing FHIR-tooling Confluence page
(e.g. `https://confluence.hl7.org/spaces/FHIR/pages/35718656/Useful+Tools+for+FHIR+Implementers`,
or whatever current page the FHIR Infrastructure WG maintains for that
purpose). The fix is a single-line change to the home-page source in the
FHIR Core publication and should be back-portable to the next ballot/CI
build of 6.0.0.

#### Justification

The reporter has identified a clear, reproducible defect: two distinct
rows in a navigational table point at the same URL, with one of them
clearly mismatched against its own label. There is no design rationale
for *Useful Tools for Implementers* and *Public Test Servers* to share a
target — they are deliberately listed as separate items. This is a
non-controversial technical correction in scope for FHIR Infrastructure,
costs nothing to ship, and removes a real navigation dead-end for
implementers reading the ballot.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying defect, but address it by removing the *Useful
Tools for Implementers* row from the Implementation Links table on the
home page rather than re-pointing it. Instead, expose the tools list in
a single, canonical location: either (a) fold the tools into the existing
"Public Test Servers" Confluence page (renamed or restructured as a
combined "Implementation Resources" page), or (b) replace the row with a
direct link to the FHIR Tooling section already present elsewhere in
the spec (e.g. the downloads / tooling page in the FHIR Core
publication), so the home page stops linking out to Confluence for this
content at all.

#### Justification

The home-page Implementation Links table currently mixes spec-controlled
content with externally-maintained Confluence pages. Confluence URLs
have proven fragile (as this ticket itself demonstrates), and the
"useful tools" content is largely duplicative of what already lives in
the spec's own tooling pages. Consolidating reduces the number of
hand-maintained external links the FHIR Infrastructure WG has to keep
in sync across ballot publications, and shrinks the surface area where
this exact class of bug can recur. The trade-off is that it is a
slightly larger edit than Disposition A and may need WG sign-off on
which page is canonical.

---

### Disposition C: Decline

#### Proposal

Close the ticket as *Not Persuasive* on the grounds that the
Implementation Links table is a curated convenience pointer to
externally-maintained HL7 Confluence content, that the linked
Confluence page itself can be updated to surface tooling information,
and that no normative spec content is affected. Direct the reporter to
raise the issue against the Confluence page rather than against the
FHIR Core ballot.

#### Justification

The defect is in a non-normative navigational table that links out to
HL7 Confluence; arguably the "fix" belongs on the Confluence side
rather than in the published ballot. Declining keeps the ballot text
stable and avoids spending FHIR Infrastructure cycles on a single-link
edit. This is the weakest position because the home page is part of
the published spec and *does* render an objectively wrong link to
implementers, but it is defensible if the WG wants to reserve ballot
edits for normative content only.

---

### Recommendation

**Recommended disposition:** A

This is a clear-cut, low-risk technical correction filed against an
authoring error in the FHIR Core home page. The reporter has done the
diagnostic work — both rows resolve to the same URL — and the fix is a
single hyperlink edit to `index.html` in the FHIR Core publication
source, with no normative impact and no implementer behaviour change.
Disposition B (consolidating away from Confluence) is a reasonable
follow-up for a future cycle but is out of proportion with what was
actually reported and would delay the obvious fix. Disposition C
understates the impact: the home page is part of the published ballot
and currently points implementers to the wrong destination. Accept as
requested, repoint *Useful Tools for Implementers* to its correct
Confluence page (FHIR Infrastructure WG to confirm the canonical URL),
and ship the fix in the next 6.0.0 publication build.
