## Summary

The reporter (Brian Postlethwaite) flags that the FHIR Mapping Language
(FML) mapping tutorial in FHIR Core contains multiple example rules whose
embedded FHIRPath expressions are written **without** the parentheses that
the FML grammar requires around a `where` clause's FHIRPath body. The
specific example called out is:

```
src.a2 as a where a2.length <= 20 -> tgt.a2 = a; // ignore it
```

which, per the reporter, should be:

```
src.a2 as a where (a2.length <= 20) -> tgt.a2 = a; // ignore it
```

The reporter notes "many cases" in the tutorial exhibit the same issue, so
this is a documentation/grammar-conformance bug across the FML mapping
tutorial pages, not a single typo. There are **no cross-referenced Jira
tickets, Zulip threads, or GitHub items** indexed against this ticket at
the time of preparation.

## Details

**Description:**

> There are many cases where the embedded fhirpath expressions are not
> contained in brackets as the language requires:
>
> *`src.a2 as a where a2.length <= 20 -> tgt.a2 = a; // ignore it`*
>
> should be
>
> `src.a2 as a where `**`(a2.length <= 20)`**` -> tgt.a2 = a; // ignore it`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation/grammar-conformance defect against the FML
mapping tutorial pages of FHIR Core (the `mapping-tutorial*.html` family,
e.g. `mapping-tutorial.html`, `mapping-language.html`, and the per-step
tutorial pages). Do an editorial pass over every code sample on those
pages and wrap the FHIRPath expression that follows a `where` keyword in
parentheses, e.g.:

- Change `where a2.length <= 20` → `where (a2.length <= 20)`
- Apply the same fix to every other `where <expr>` occurrence in the
  tutorial samples (and any analogous spots, e.g. `check (...)` /
  `condition (...)` clauses if they exhibit the same omission).

Verify each corrected sample still parses with the reference FML parser
(e.g. by round-tripping through HAPI / org.hl7.fhir.utilities StructureMap
parser, which is what the publisher uses). Add a regression check in the
build, if practical, that runs the tutorial snippets through the FML
parser so future drift is caught.

#### Justification

The reporter is the FML/StructureMap maintainer and is asserting that the
shown form is invalid per the FML grammar. Tutorials are the primary
on-ramp for new implementers; samples that don't actually parse are an
acute usability bug — readers copy them verbatim and then hit confusing
errors. Fixing the samples in place preserves the tutorial's pedagogical
flow and brings it back into conformance with the FML grammar definition.
This is a low-risk editorial change confined to narrative/tutorial pages;
it does not alter any normative artifact, schema, or resource definition.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (only) editing each example, fix the underlying *grammar*
ambiguity at its source. Two flavors:

1. **Relax the FML grammar** to make the parentheses optional around the
   FHIRPath expression in a `where` clause (and similarly for `check`),
   parsing the rest of the line up to `->` as the FHIRPath body. Update
   the FML grammar reference page accordingly, then leave the existing
   tutorial samples as-is. Implementations (HAPI, the FHIR validator,
   matchbox, etc.) would need to track the relaxed grammar.
2. Or, **invert it**: keep the grammar strict but add an explicit
   "FHIRPath in FML" callout box on the tutorial landing page that
   states the parenthesization rule once, then fix the examples (a
   subset of Disposition A scoped only to the most-trafficked pages).

#### Justification

Disposition A treats the symptom; Disposition B(1) treats the cause for
authors who find the parentheses ceremonial when the expression is
unambiguous. However, loosening a published grammar is a normative change
that affects every FML parser in the wild and any tooling that
round-trips StructureMaps — a much larger blast radius than fixing
documentation. Disposition B(2) is a lighter middle ground that improves
discoverability of the rule but still leaves the broken samples in place
until edited. Both options trade documentation churn for either
implementer churn (B1) or continued reader confusion (B2).

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" on the grounds that the FML grammar in the
specification already requires parentheses around the `where` clause's
FHIRPath, the production parsers enforce it, and the tutorial text
elsewhere demonstrates the parenthesized form — i.e., treat the cited
samples as isolated typos to be filed individually rather than as a
ticket warranting a sweep.

#### Justification

This would only be defensible if the omission were genuinely a one-off
typo. The reporter explicitly states there are *"many cases"* in the
tutorial, and the reporter is the maintainer of the relevant
infrastructure, so the assertion is credible. Declining would leave a
known set of non-parseable code samples in front of new implementers,
which conflicts with FHIR Infrastructure's stewardship of the FML
tutorial pages. Listed for completeness; not recommended.

---

### Recommendation

**Recommended disposition:** A

This is a documentation defect with an unambiguous, narrow fix. The
reporter has identified that a class of FML samples in the tutorial does
not conform to the FML grammar's `where (<fhirpath>)` requirement, and
the corrected form is already used elsewhere in the spec. Disposition A
brings the samples back into conformance with zero impact on normative
artifacts or downstream parsers, and it costs the reader nothing. The
grammar-relaxation path in Disposition B(1) is disproportionate to the
problem and would create real implementer cost across HAPI, the
validator, and other FML toolchains; B(2) is a half-measure that still
requires editing the worst examples. Disposition C ignores the
reporter's explicit statement that the issue is systemic in the
tutorial. Accept the change, sweep the FML tutorial pages, and — if
feasible — add a build-time parse check on tutorial snippets so this
class of drift is caught automatically going forward.
