## Summary

The ticket requests the addition of a new FHIR extension to represent the
content-encoding (transfer/compression encoding) applied to the payload of
an `Attachment`'s inline `data` element (or, equivalently, to values of
the `base64Binary` primitive). Today, `Attachment` carries a MIME
`contentType` but has no way to say that the bytes inside `data` are
compressed with `gzip`, `deflate`, or `zlib`, so implementers either
invent proprietary conventions, use an out-of-band `Content-Encoding`
HTTP header that is lost once the bytes land in a FHIR resource, or
abuse the `contentType`. The reporter proposes an extensible coding
seeded with at least `deflate`, `gzip`, and `zlib`, and asks that the
extension be publishable as part of the FHIR Extensions Pack so it can
apply to `Attachment` and/or `base64Binary` wherever they are used
(e.g., `DocumentReference.content.attachment`, `Binary.data`,
`Media.content`, `Communication.payload.contentAttachment`, etc.). No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced by Augury for this ticket at the time of preparation, so the
request currently stands on its own.

## Details

**Description:**

It is common to represent data in standard encodings other than 'raw',
specifically for compression.

`Attachment` has a `contentType`, but no equivalent of `contentEncoding`
when including `data`.

Request the addition of a `content-encoding` extension on either
`Attachment` or `base64Binary` to allow representation of encoding
types.

Suggest an extensible coding, with at least: `deflate`, `gzip`, `zlib`
as a starter set.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Publish a new standard extension in the FHIR Extensions Pack named
`content-encoding` (canonical URL of the form
`http://hl7.org/fhir/StructureDefinition/content-encoding`) with:

- **Context of use:** `Attachment.data` and the `base64Binary`
  primitive type. Defining the context on both covers the original use
  case on `Attachment` as well as `Binary.data` and any future element
  typed as `base64Binary`.
- **Value:** `code` (single cardinality 0..1).
- **Binding:** a new `ValueSet` (e.g., `content-encoding`) with an
  `extensible` strength, defined against a new `CodeSystem`
  `http://hl7.org/fhir/content-encoding` whose initial concepts are
  `deflate`, `gzip`, and `zlib` (matching the IANA HTTP Content Coding
  Registry so `br`, `compress`, `identity`, and `zstd` can be added
  later without a breaking change).
- **Semantics:** when present, the extension asserts that the bytes in
  the sibling `data` element are the result of applying the named
  encoding to the payload whose media type is given by `contentType`.
  Consumers MUST decode in the reverse order of encoding before
  interpreting `contentType`.

Ship supporting assets (StructureDefinition, ValueSet, CodeSystem,
examples for `DocumentReference`, `Binary`, `Media`) in the Extensions
Pack, and cross-link the extension from the `Attachment` and
`base64Binary` core-spec pages.

#### Justification

The request identifies a genuine gap: `Attachment.contentType` answers
"what kind of content is this?" but not "how were the bytes on the
wire transformed?", and FHIR currently has no primitive-level
convention for signalling compression. The extension mirrors HTTP's
long-standing `Content-Encoding` header, which is the de-facto
cross-industry vocabulary for this information, so implementers
already understand the semantics and failure modes. Delivering it as
an Extensions-Pack extension avoids a breaking change to the core
`Attachment` datatype while still giving every specification (US Core,
IPS, IHE profiles, etc.) a single canonical way to opt in. Declaring
the context on `base64Binary` — not just on `Attachment` — future-
proofs the mechanism for `Binary.data`, `Media.content.data`, and any
logical model that embeds raw bytes. An `extensible` binding aligned
with the IANA registry keeps the door open for `br` / `zstd` without a
new extension.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than a single `content-encoding` extension, define a small
**Attachment-level** extension (`attachment-encoding` on
`Attachment`) whose value is a `Coding` rather than a `code`. The
`Coding.system` would default to the IANA HTTP Content Coding
Registry (`https://www.iana.org/assignments/http-parameters`) with
`Coding.code` drawn from that registry (`gzip`, `deflate`,
`compress`, `br`, `zstd`, `identity`), and the extension would be
bound only where `Attachment` appears. Leave `base64Binary` untouched.
Optionally, standardise a second extension
`attachment-uncompressed-size` (`unsignedInt`) so receivers can
allocate buffers safely before decoding.

#### Justification

Scoping the extension to `Attachment` (not the raw `base64Binary`
primitive) keeps the change inside the "document-ish payload" type
where the need is concrete and avoids opening a debate about adding
metadata to every `base64Binary` in the specification (e.g.,
`Signature.data`, `ElementDefinition.defaultValueBase64Binary`,
where compression semantics do not make sense). Using a `Coding` with
the IANA registry as its system is more extensible than a local
`CodeSystem` and aligns one-to-one with what servers would set on the
HTTP `Content-Encoding` header if the bytes were streamed rather than
inlined, reducing round-trip conversion bugs. Adding the
`uncompressed-size` companion addresses a well-known denial-of-service
concern ("zip-bomb") that inline compressed payloads introduce and
that the reporter's request does not currently cover.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive**. Document, in the resolution
and on the `Attachment` page, that FHIR's guidance is to leave
compression to the transport layer (HTTP `Content-Encoding`,
`Accept-Encoding` negotiation, or server-level TLS/HTTP compression)
and that `Attachment.data` is defined to carry the *decoded* payload
whose media type is given by `Attachment.contentType`. Implementers
who need to ship pre-compressed bytes should either (a) use a
`contentType` whose media type already implies compression (e.g.,
`application/gzip`, `application/zip`), or (b) use
`Attachment.url` to point at a resource that is negotiated via HTTP.

#### Justification

Adding a `content-encoding` extension subtly changes the contract of
`Attachment.data` from "the bytes" to "the bytes plus a sidecar
instruction", which must then be respected by every validator,
search-parameter extractor, digital-signature routine, and receiver
in the ecosystem; in practice many will ignore it and silently
mis-interpret compressed payloads. Transport-level compression
(HTTP) is already ubiquitous, handled by web servers and proxies,
and correctly bypassed for signed bundles. Where the payload itself
is genuinely an archive (a gzipped CDA, a ZIP of images),
`contentType` already carries the right information. The cost of
adding one more extension that every serious FHIR stack must learn
to honour is, on this view, disproportionate to the problem that
transport-level mechanisms already solve.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the real implementer need — a first-class way
to say "this inline blob is gzip'd" — while keeping the blast radius
small and aligning with a vocabulary (the IANA HTTP Content Coding
Registry) that every HTTP stack already understands. Scoping to
`Attachment` rather than `base64Binary` sidesteps a much larger
review about what "encoded" means for unrelated base64-valued elements
such as `Signature.data`, and the companion `uncompressed-size`
extension addresses the zip-bomb concern that Disposition A leaves
unsolved. Disposition A is defensible and only marginally more work,
but its `code`-typed value and locally-defined `CodeSystem` would
almost certainly need to be reworked to track IANA over time.
Disposition C is a principled stance, but it ignores the fact that
inlined payloads *do* show up in FHIR (e.g.,
`DocumentReference.content.attachment.data`, `Binary.data` for
store-and-forward scenarios) where transport compression has already
been unwound by the time the resource is persisted — exactly the gap
the reporter is calling out.
