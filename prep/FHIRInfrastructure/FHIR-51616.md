## Summary

The reporter notes that the two canonical resources `SubscriptionTopic`
and `ObservationDefinition` appear to be missing the standard
descriptive statement on their `url` element — both in the current
build and in R5. Other canonical resources carry the boilerplate
"Canonical identifier for this structure definition, represented as a
URI (globally unique) + Warning: URL should not contain | or # - these
characters make processing canonical references problematic" on `url`,
but `SubscriptionTopic.url` and `ObservationDefinition.url` lack this
text. The ticket originated from a brief Zulip exchange in
`#implementers > Potential missing statements in spec`, where Jens
Villadsen flagged the omission and Grahame Grieve agreed a Jira ticket
should be filed. There are no cross-referenced Jira tickets and no
linked GitHub items.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a technical correction to the FHIR Core specification that
updates the `definition` (and matching `short`) text of the `url`
element on both `SubscriptionTopic` and `ObservationDefinition` to
match the standard wording used on other canonical resources (e.g.,
`StructureDefinition.url`, `ValueSet.url`, `CodeSystem.url`):

> Canonical identifier for this structure definition, represented as a
> URI (globally unique) + Warning: URL should not contain `|` or `#` -
> these characters make processing canonical references problematic.

This is purely a definitional / documentation change to the resource
StructureDefinitions for `SubscriptionTopic` and `ObservationDefinition`
in the build (and as a technical-correction note for R5 errata). No
cardinality, type, invariants, or wire-format changes are involved.

#### Justification

This is a Technical Correction to bring two canonical resources into
alignment with the boilerplate text already carried by every other
canonical resource. The `url` element semantics and warnings are
universal across canonical resources, so the missing text is an
omission, not a deliberate design choice. Adopting the standard
wording improves consistency, helps implementers understand the same
restrictions on `|` and `#` characters that apply to other canonical
URLs, and has no behavioural or schema impact. There is also explicit
agreement on Zulip from Grahame Grieve that the issue is worth a Jira
ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than patching the two resources individually, audit **all**
canonical resources (and any other resources whose `url` element is
typed as `uri` and intended as a canonical) for the same missing
boilerplate. Where the standard text is absent, apply it uniformly in
a single coordinated technical correction. Optionally, capture the
required `url` boilerplate as a reusable definition snippet (or as
part of the `CanonicalResource` pattern documentation) so future
canonical resources inherit the wording by construction rather than by
copy-paste.

#### Justification

The fact that two distinct canonical resources both drifted from the
standard wording suggests this is a process problem, not just two
isolated typos. A broader audit closes any remaining gaps in one pass
and avoids a follow-up ticket every time another instance is found. It
also addresses the underlying maintainability issue by making the
canonical `url` definition centrally specified, consistent with how
the `CanonicalResource` logical pattern is meant to drive shared
behaviour. The trade-off is slightly more work and review surface than
Disposition A, but the additional scope is mechanical and low-risk.

---

### Disposition C: Decline

#### Proposal

Decline the technical correction on the grounds that the missing text
is a non-normative descriptive convenience: the element type, name,
cardinality and the canonical-reference processing rules (including
the prohibition on `|` and `#` in canonical URLs) are already
specified normatively elsewhere in the specification. Implementers
relying on the formal definitions and the canonical-reference rules
would not behave differently with or without the boilerplate sentence
on these two resources.

#### Justification

If the workgroup judges that the descriptive text adds no normative
content beyond what is already stated in the canonical-resource
machinery, declining avoids editing R5 errata for what is essentially
prose tidying. This is the weakest of the three options because the
ticket is small, low-risk, and explicitly framed as a Technical
Correction, but it is included for completeness in case the workgroup
prefers to defer all R5 prose changes unless there is implementer
harm.

---

### Recommendation

**Recommended disposition:** B

Disposition A is clearly correct in substance and would resolve the
ticket as filed, but the existence of two independent canonical
resources with the same missing boilerplate is a strong hint that
others may also have drifted. Disposition B does everything
Disposition A does, plus a one-time audit across the canonical
resources to catch any remaining instances and (optionally) hardens
the process so the same omission cannot recur silently. The
incremental cost is modest, the risk is low (no schema or behavioural
change), and the value to implementers — consistent, predictable
descriptive text on every canonical `url` element — is concrete.
Disposition C is not recommended: this is exactly the kind of small,
consistency-improving prose fix the Technical Correction process
exists to handle.
