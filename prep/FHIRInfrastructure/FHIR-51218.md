## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the four requested concepts into
`http://hl7.org/fhir/ValueSet/identifier-type`:

1. Coordinate with the Vocabulary work group (UTG-708) to add the
   following concepts to `http://terminology.hl7.org/CodeSystem/v2-0203`
   (HL7 v2 Table 0203, "Identifier Type"):
   - `RFI` (Referral ID) — narrower than `PLAC` (Placer Identifier)
   - `ORI` (Order ID) — narrower than `PLAC` (Placer Identifier)
   - `SIUID` (Study Instance UID)
   - `ENID` (Encounter ID)
   (Codes shown are illustrative; the actual mnemonics are assigned by
   UTG / v2 Management Group.)
2. Once published in v2-0203, extend the FHIR Core `identifier-type`
   ValueSet so it continues to enumerate the relevant concepts
   (the ValueSet is composed by `include`-ing v2-0203 codes; verify that
   the new concepts are inside the existing filter — if not, add an
   explicit `concept` enumeration or relax the filter to pick them up).
3. For Referral ID and Order ID, document in the ValueSet narrative /
   identifier-type page that they are *more specific* alternatives to
   `PLAC` so implementers know when to use which.

#### Justification

The request fills real implementer needs. Referral and Order are common
business identifiers that today get squeezed into `PLAC` ("Placer"),
which obscures intent in messages and FHIR resources where both a
referral identifier and an order identifier may travel together
(e.g., `ServiceRequest.identifier`, `ImagingStudy.identifier`).
`Study Instance UID` is the universally-used DICOM identifier on
`ImagingStudy` / `ImagingSelection` and a dedicated type code makes
cross-system filtering reliable. `Encounter ID` is similarly common on
clinical and financial resources. Identifier type codes are intentionally
extensible, and FHIR Infrastructure does not need to invent a new
CodeSystem when v2-0203 is the canonical home and UTG governance already
exists. Doing this addition the right way — CodeSystem first, ValueSet
follows — is exactly the pattern FHIR-IM has used for previous
`identifier-type` extensions.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the FHIR Core `identifier-type` ValueSet itself,
treat these four concepts as *use-case-specific* extensions and resolve
them outside of FHIR Core:

- For **Study Instance UID**, do not add a generic identifier-type code;
  instead point implementers at the existing pattern of using
  `Identifier.system = "urn:dicom:uid"` (or the imaging IGs' published
  systems) on `ImagingStudy.identifier` and `ImagingSelection.identifier`,
  where the *system* — not the type — already carries the semantics.
- For **Referral ID** and **Order ID**, decline to subdivide `PLAC` in
  v2-0203 / `identifier-type`, and instead recommend the realm-specific
  IGs (e.g., US Core, IHE, regional ordering IGs) bind a profile-level
  ValueSet that includes their own type codes when finer granularity is
  required. Continue to recommend `PLAC` for the generic case.
- For **Encounter ID**, recommend that implementers reference the
  Encounter via `Identifier.assigner` / `Identifier.system` plus a
  resource reference, since identifier-type is a coarse classification,
  not a context label.

If, after triage, any of these are still considered FHIR-Core-worthy,
escalate only those to UTG-708.

#### Justification

The `identifier-type` ValueSet is a *general-purpose* classification.
Every concept added pulls in implementer expectations that the type
distinguishes a real semantic class across all resources, not just the
context where it was requested. `Study Instance UID` in particular is
already addressed by a well-known `Identifier.system`, and adding it as a
*type* code creates two equally valid ways to flag the same identifier,
which fragments interoperability. Referral / Order specialisations of
`PLAC` raise the same concern at a smaller scale: most exchanges already
disambiguate by resource type (`ServiceRequest` vs. order resources) or
by `Identifier.system`. Pushing the specialisation into IG-level
ValueSets keeps FHIR Core stable and lets the requesting communities
move at their own pace.

---

### Disposition C: Decline

#### Proposal

Close FHIR-51218 as **Persuasive with Modification → No FHIR-side change
required**, with the following rationale:

- The FHIR Core `identifier-type` ValueSet is mechanically composed from
  `http://terminology.hl7.org/CodeSystem/v2-0203` (and a small set of
  FHIR-specific additions). FHIR Infrastructure cannot meaningfully add
  `Referral ID`, `Order ID`, `Study Instance UID`, or `Encounter ID`
  unless and until those concepts exist in the source CodeSystem(s).
- The substantive request therefore lives in UTG-708 and belongs to the
  Vocabulary work group / v2 Management.
- Mark this ticket as dependent on UTG-708 and close it on the FHIR-IM
  side; once UTG publishes the codes the ValueSet will pick them up
  automatically (or a trivial follow-up ticket will be opened to update
  the include/concept list).

#### Justification

This is not "decline the idea" — it's "decline to act here". The FHIR
Infrastructure side has nothing to change while the codes do not exist
upstream, and tracking a no-op ticket against the FHIR Core spec creates
the false impression that FHIR-IM is the gating party. Closing with a
clear pointer to UTG-708 keeps governance honest and avoids parallel
debate of the same concepts in two work groups. If UTG-708 is rejected
on substantive grounds, the FHIR ValueSet does not (and cannot) change;
if UTG-708 is accepted, a small administrative ticket re-opens the
ValueSet update at that time.

---

### Recommendation

**Recommended disposition:** A (with the procedural framing of C)

The four concepts each address a concrete, repeatedly-encountered
modelling gap, and `identifier-type` is exactly the right vocabulary for
classifying a generic `Identifier.type`. Disposition B's "use the system
URL" argument is correct for Study Instance UID in isolation but does
not generalise: type codes and system URLs serve different roles
(human-readable classification vs. machine-resolvable namespace), and
implementers consistently want both. Splitting `PLAC` into Referral vs.
Order is a small, well-scoped clarification that has been requested
before and that aligns with how v2 has historically grown table 0203.

That said, the workgroup should adopt the Disposition C *workflow*
inside Disposition A: the substantive change must originate in UTG-708,
and FHIR-IM should record this ticket as **blocked on UTG-708**, not act
unilaterally on the FHIR Core ValueSet. Once UTG publishes the new
codes in v2-0203, FHIR-IM verifies the ValueSet expansion picks them up
(adjusting the `include` filter or adding explicit `concept`
enumerations as needed) and updates the `identifier-type` page narrative
to point at the new specialisations (especially the Placer → Referral /
Order relationship). This sequencing keeps governance with the right
work group while still landing the implementer-visible improvement the
reporter asked for.
