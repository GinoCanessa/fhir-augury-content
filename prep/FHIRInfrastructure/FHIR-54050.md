## Summary

The reporter (HaydenBader) has identified a mismatch between the human-readable
explanation text and the FHIRPath expression on the `cpb-4` invariant in the
`CapabilityStatement` resource. The explanation text reads "There *should* only
be one CapabilityStatement.rest per mode," using the lower-case "should," which
in IETF/normative drafting conventions implies a recommendation rather than a
strict requirement. The actual FHIRPath expression, however, is
`rest.mode.isDistinct()`, which is a hard, machine-checkable assertion: any
instance with two `rest` entries sharing the same `mode` will fail validation.
The ticket asks that the explanation text be reworded to use a normative verb
(SHALL / must) so that the prose matches what the constraint actually enforces.
There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
beyond the `CapabilityStatement.rest` artifact reference detected in the
ticket body itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the human-readable description ("explanation text") on invariant
`cpb-4` of the `CapabilityStatement` resource so that the verb matches the
strictness of the FHIRPath expression `rest.mode.isDistinct()`. Concretely,
change the text from:

> There *should* only be one CapabilityStatement.rest per mode.

to:

> There **SHALL** only be one CapabilityStatement.rest per mode.

(or, equivalently, "Each CapabilityStatement.rest entry SHALL have a distinct
`mode`.") The invariant `severity` remains `error` and the expression remains
`rest.mode.isDistinct()`; only the `human` text changes. No structural,
expression, or severity change is made, so no instances that previously
validated will newly fail.

#### Justification

This is a pure technical correction. The invariant already has
`severity = error`, which by the FHIR invariants framework means the
constraint is enforced as a hard rule — instances that fail it are invalid.
Using the lower-case "should" in the human description is misleading because
it suggests a SHOULD-level recommendation when in fact the validator will
reject non-conforming instances. Aligning the human text with the actual
enforcement (and with FHIR's RFC 2119-style conventions, where SHALL =
mandatory) removes ambiguity for implementers reading the constraint and is
consistent with how other error-severity invariants in the spec are phrased.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adjusting the verb, rephrase the `human` text so that it
mirrors the structure of the FHIRPath expression itself, e.g.:

> Each value of `CapabilityStatement.rest.mode` SHALL be distinct (i.e., a
> CapabilityStatement SHALL NOT contain more than one `rest` entry with the
> same `mode`).

Optionally, also normalize the `human` text on any sibling invariants in
`CapabilityStatement` whose human descriptions use weak verbs while their
expressions use hard predicates (`exists()`, `isDistinct()`, `empty()`, etc.).
Scope of the change: still authored on the `CapabilityStatement` source
definition; expression and severity unchanged.

#### Justification

A one-word swap (B → A) fixes the immediate complaint, but the underlying
issue — human text drifting from FHIRPath semantics — is a recurring source
of confusion. Phrasing the human text in terms of the predicate (`distinct`,
`exists`, etc.) makes the relationship between description and expression
self-evident and harder to drift apart in future edits. Doing a small sweep
of `CapabilityStatement`'s other invariants while the file is open is
low-risk and amortizes the editorial cost. Trade-off vs. A: slightly larger
diff and a small chance of touching wording that another reviewer prefers
left alone.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing wording, while
imperfect, is non-normative prose attached to an invariant whose
machine-checkable expression is the source of truth. Tools and validators
key off the expression and severity, not the English text, so no
implementer behavior changes regardless of the verb used.

#### Justification

Defensible only on minimalist grounds: the constraint already enforces
distinct `mode` values via `rest.mode.isDistinct()` at error severity, so
no validator output changes either way. Editorial-only changes carry a
small but non-zero cost (publication churn, translation impact, ballot
review noise). However, this rationale is weak because the FHIR spec
explicitly treats invariant `human` text as the human-facing contract, and
clarity for implementers is itself a goal of the spec.

---

### Recommendation

**Recommended disposition:** A

This is a textbook technical correction: the FHIRPath expression and the
severity already define the rule unambiguously, and the only thing being
corrected is the English verb so it matches what the constraint actually
enforces. The change is editorial, has no impact on existing instances or
tooling, carries no backward-compatibility risk, and removes a real source
of implementer confusion. Disposition B is reasonable if the work group
wants to take the opportunity to harmonize sibling invariants, but absent
evidence that other `CapabilityStatement` invariants suffer the same drift,
the smaller, surgical fix in A is the most defensible response to what the
ticket actually asks for. Disposition C should be rejected because invariant
`human` text is part of the published contract with implementers and
should not be allowed to contradict the expression.
