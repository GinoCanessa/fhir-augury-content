## Summary

Brian Postlethwaite argues that the new `%server` FHIRPath factory â€” the
collection of functions that let a FHIRPath expression invoke
server-side REST interactions (`read`, `create`, `update`, `delete`,
`search`, `patch`, `validate`, `transform`, `everything`, `apply`,
`capabilities`) â€” is not ready to ship as normative content in the
FHIR Core specification and should be moved into an incubator IG. He
notes the feature was only `maturity=0` (draft) in R5, has very limited
known implementation experience, and exposes a serious attack surface:
side-effecting functions injected via search parameters, invariants, or
Questionnaire expressions could create/modify resources or leak data.
He further lists numerous unresolved design issues (security,
authorization, error reporting, parameter ordering, inconsistent
signatures, unclear context-of-use, ambiguous return types, missing
versioning semantics for `update`/`delete`, `capabilities` returning
the wrong resource type, etc.) and offers to do detailed work on them
inside an incubator. The ticket is one of a cluster of related
"not-ready-for-normative" change requests Brian filed against the same
ballot family â€” see in particular [FHIR-54078](https://jira.hl7.org/browse/FHIR-54078)
(same argument applied to the `%factory` Type Factory functions),
[FHIR-53960](https://jira.hl7.org/browse/FHIR-53960) (inconsistent
error-handling patterns across the new functions, including the
`%server` family), [FHIR-53953](https://jira.hl7.org/browse/FHIR-53953)
(no documentation for remote-interaction concerns â€” timeouts, caching,
availability, security â€” that the `%server` functions inherently
introduce), and [FHIR-54473](https://jira.hl7.org/browse/FHIR-54473)
(parallel "move FML to its own incubator/spec" request). Together they
frame a coherent ask: pull immature normative-track features out of the
core normative package and mature them in incubators before reballoting.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the entire `%server` FHIRPath factory and all of its functions
(`read`, `create`, `update`, `delete`, `search`, `patch`, `validate`,
`transform`, `everything`, `apply`, `capabilities`) **out of the
normative FHIR Core specification** and into an incubator IG (e.g., a
`fhirpath-server-functions` incubator under `HL7/`, or as an extension
to the FHIRPath spec in `HL7/fhirpath`). Concretely:

- Strip the `%server` factory section from the FHIRPath usage chapter
  in FHIR Core for the current ballot, leaving a placeholder pointing
  at the incubator.
- Reset the maturity level on the moved content to draft / FMM 0.
- Open companion trackers (or carry forward FHIR-53953, FHIR-53960,
  the issues enumerated in this ticket's bullet list, and the security
  / authorization / context-of-use questions) against the incubator
  rather than blocking the ballot on them.
- Defer any normative ballot of `%server` to a future cycle once an
  incubator has demonstrated multi-vendor implementation experience.

#### Justification

The reporter's core factual claims are well grounded: `%server` was
`maturity=0` in R5, no public evidence of multiple independent engine
implementations exists, and the design has many open issues. HL7
ballot policy generally requires substantial implementation experience
before a feature can go normative. The security argument is also
serious â€” once `%server.create()` etc. are FHIRPath-callable from any
context that evaluates FHIRPath (search parameters, invariants,
Questionnaire expressions, GraphDefinition links, etc.) the attack
surface is broad and the existing spec text does not constrain it.
Moving the feature to an incubator is the standard FHIR mechanism for
exactly this situation (cf. the parallel ask in FHIR-54473 for FML)
and lets the listed design issues be worked openly without blocking
the rest of the FHIRPath additions.

---

### Disposition B: Alternative Approach

#### Proposal

**Keep `%server` in the normative package but demote its maturity and
fence its usage**, rather than physically relocating it. Specifically:

- Mark the entire `%server` factory as Trial Use / FMM 1 within an
  otherwise normative chapter (allowed for clearly-bounded sub-areas).
- Add a new mandatory "Context of Use" subsection that **enumerates
  the only contexts in which side-effecting `%server` functions
  (`create`, `update`, `delete`, `patch`, `apply`, `transform`) MAY be
  evaluated** â€” e.g., explicitly disallow them in SearchParameter
  expressions, invariants, and any context derived from
  client-supplied content â€” and define the conformance rule a server
  uses to refuse evaluation otherwise.
- Resolve the concrete API issues called out in the ticket inline
  (fix `search` parameter ordering so `doPost` is optional/second;
  align `patch` parameter shape with the rest of the family;
  collapse `transform` to `transformStructure(canonicalUrl)`; make
  `capabilities` return `CapabilityStatement`; rename `apply` /
  document its mapping to `$apply`; add versioning semantics to
  `update` / `delete`; align `validate` with `conformsTo`).
- Adopt the error-handling and remote-interaction sections proposed
  in [FHIR-53960](https://jira.hl7.org/browse/FHIR-53960) and
  [FHIR-53953](https://jira.hl7.org/browse/FHIR-53953) before publication.
- Add a `CapabilityStatement.rest.feature` (or new feature
  capabilities) entry by which a server advertises which `%server`
  functions it supports.

#### Justification

This addresses the substantive concerns â€” security, scoping, API
consistency, error semantics â€” without the publication-mechanics cost
of pulling content out of the normative package mid-ballot. It also
preserves the read-only subset (`read`, `search`, `capabilities`,
`everything`) that the reporter himself describes as "reasonable",
which would otherwise be lost to anyone consuming the normative
package. Trial-Use designation inside a normative chapter is an
established precedent and signals "stable enough to ship, not yet
locked" â€” appropriate given the lack of broad implementation
experience but the clear demand. The trade-off is that it leaves more
work in the current ballot cycle than Disposition A.

---

### Disposition C: Decline

#### Proposal

Reject the request to move `%server` to an incubator. Keep the feature
in the normative spec as currently drafted, with the open design
issues handled as ordinary in-cycle change requests against this
ballot. The detailed sub-points in the ticket (parameter ordering,
`patch` shape, `capabilities` return type, error reporting,
versioning) should be re-filed (or split out from this ticket) as
individual trackers to be dispositioned on their own merits.

#### Justification

A "Decline" disposition is defensible only if the WG concludes that
(a) implementation experience is sufficient â€” i.e., enough vendors
have demonstrated `%server` to claim normative readiness â€” and (b) the
security/context-of-use concerns can be fully addressed with
specification text alone in the current cycle. If those conditions
are not met, this disposition is hard to justify and would likely
draw negative ballot reconciliation later. The main argument for it
is that pulling content from a normative package this late is
disruptive to consumers who have already been integrating against the
draft, and that the listed design issues are individually fixable
without changing the feature's normative track.

---

### Recommendation

**Recommended disposition: B (Alternative Approach).**

The reporter's substantive concerns are correct and must be addressed
â€” the ticket cannot simply be declined (Disposition C) without
publishing a normative feature that has known security holes,
inconsistent error semantics, and unresolved API shape questions, all
of which would harden into a normative contract that is expensive to
fix later. However, fully relocating the feature to an incubator
(Disposition A) discards the read-only subset (`read`, `search`,
`capabilities`, `everything`) that the reporter himself accepts as
reasonable, and creates a downstream coordination problem with the
companion tickets (FHIR-54078, FHIR-53960, FHIR-53953) that would
also need to follow the content out of the normative package.
Disposition B keeps the work in one place, lets the WG ship the
read-only subset with appropriate guards, demotes the side-effecting
half to Trial Use with an explicit context-of-use fence, and folds
in the prerequisite remote-interaction and error-handling sections
already proposed in the related tickets. If the WG finds during
reconciliation that the API shape fixes plus the context-of-use
fence cannot be drafted in time for the ballot, the fallback is
Disposition A â€” but that fallback should be an explicit decision,
not the default.
