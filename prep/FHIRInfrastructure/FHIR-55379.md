## Summary

The ticket asks the FHIR Infrastructure work group to extend the existing FHIR
Mapping Language (FML) tutorial to cover more of FML's features. The reporter
(Brian Postlethwaite, who is one of the most active FML implementers and a
frequent responder on the `FHIR Mapping Language` Zulip stream) notes that the
current tutorial only covers the very basics, leaving substantial parts of the
language without a worked example or guided walk-through.

No Jira tickets are formally cross-referenced, but the request lands in the
middle of an active R6-ballot debate about FML's place in the specification.
On the `implementers` stream Brian opened the topic *"FHIR mapping language -
move to its own spec?"*, noting that several R6 ballot submitters asked to
remove FML and the FML tutorial from the core specification (leaving only
StructureMap). Day-to-day Zulip traffic in the `FHIR Mapping Language` stream
also routinely surfaces tutorial-shaped gaps — e.g., the *"Explicit name"*
thread where David Pyke explicitly offered "$10 to make the FML docs and
tutorial more clear", the *"resolve() references within a Bundle"* thread
where step 14 of the tutorial was found to be misleading about implicit
reference resolution, and older *"FML: why is this not compiling?"* threads
where users got stuck because steps were missing. This ticket is the
work-group-level expression of that recurring user feedback.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-55379 as a documentation-only change against `mapping-tutorial`
(the FHIR Mapping Language tutorial page in the FHIR core specification) and
extend it with worked examples for the FML features that are currently
unexplained or underexplained, drawn from real Zulip user-failure modes:

1. **Named / "explicit-name" complex rules** — a step that introduces the
   `... } "ruleName";` syntax after a rule body and explains the
   "Complex rules must have an explicit name" diagnostic (see the
   *Explicit name* Zulip thread).
2. **`group … extends …` and `<<types>>` / `<<type+types>>` modes** — a
   step that walks a small inheritance example and explains when the
   engine will auto-create dependent rules.
3. **Reference handling** — rewrite step 14 so that implicit reference
   following is described accurately, then add an explicit `resolve()`
   example for the Bundle case raised in the *resolve() references
   within a Bundle* thread.
4. **`where` / `check` / `log` clauses**, **`first` / `last` /
   `not_first` / `not_last` list modes**, and **`dependent` invocations
   between groups** — at least one minimal worked example each.
5. **Terminology mapping** — a step that shows `translate(...)` against a
   ConceptMap end-to-end, since this is the most-asked tutorial gap on
   the `mapping-framework` stream.
6. Cross-link each new step to the corresponding section of
   `mapping-language.html` and to the StructureMap example resources so
   readers can copy a runnable map.

Mechanically, this is a content edit on the tutorial page only — no
StructureMap schema, no FML grammar, and no engine behavior changes. It
sits squarely in FHIR Infrastructure's editorial scope.

#### Justification

The request is explicitly tutorial-scoped ("current steps only cover the
very basics") and is documentation, so it carries no normative risk and
no breaking-change surface. Each proposed extension is grounded in a
concrete Zulip thread where a real implementer got stuck because the
tutorial was silent or wrong on that feature, so the changes are
demand-driven rather than speculative. This also strengthens FHIR
Infrastructure's position in the parallel R6 debate about whether to
move FML out of the core spec — a fuller, more accurate tutorial is the
strongest defence of keeping it in.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding `mapping-tutorial.html` indefinitely as a single
linear walk-through, restructure the FML educational content into two
layers:

1. Keep `mapping-tutorial.html` as a tight, beginner-friendly
   "first 30 minutes with FML" — same shape as today, lightly fixed for
   the known errors (notably the step 14 reference-resolution wording).
2. Add a new **"FML Cookbook"** page (or a small set of
   `mapping-tutorial-<topic>.html` pages) under FHIR Infrastructure,
   each one a focused worked example for one feature: named complex
   rules, `group extends`, `dependent` invocations, list modes,
   `where`/`check`/`log`, terminology `translate`, Bundle / `resolve()`
   handling, and any new R6 features. Each cookbook entry is
   self-contained, links back to the main tutorial, and references the
   relevant section of `mapping-language.html`.

The cookbook pages are still pure documentation in FHIR Core; no
StructureMap or FML grammar changes are implied.

#### Justification

The Zulip evidence shows the problem is not just *quantity* of tutorial
content but *findability*: users currently scan a single linear page and
either bail out (David Pyke) or take a wrong inference from a single
step (the `resolve()` thread). A cookbook layout matches how users
actually arrive at FML — they hit a specific error message or feature
question and search for it — without bloating the introductory tutorial
to the point that beginners give up. It also gives the work group a
natural place to land R6-era additions (e.g., new FML syntax) without
each one re-opening the "how long should the tutorial be?" question.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-55379 as a tutorial-extension request and instead defer all
FML educational expansion to whatever output the *"FHIR mapping language
- move to its own spec?"* discussion produces. If FML is split out of
the core specification in R6+, the tutorial would move with it and the
right home for additional examples becomes the new FML specification /
IG, not `mapping-tutorial.html` in core. Close FHIR-55379 as
"Persuasive – With Mod" with a note that the substantive content
request is preserved and will be re-opened against the new home for FML
documentation.

#### Justification

There is non-trivial risk that significant editorial work invested in
`mapping-tutorial.html` is throwaway if FML is removed from the core
specification in R6. Declining (or at least pausing) until the
move-to-its-own-spec question lands avoids duplicated effort and avoids
shipping a substantially expanded tutorial in core that then has to be
deleted or migrated. This is the conservative editorial path.

---

### Recommendation

**Recommended disposition:** B

The ticket is a real, recurring user-experience problem — Zulip evidence
goes back years and includes implementers and users who actually paid
real money (or offered to) to get this fixed — so Disposition C is too
weak: declining sends the message that FHIR Infrastructure is
comfortable with a tutorial that demonstrably fails its readers, which
also undercuts the case for keeping FML in the core spec at all.
Disposition A solves the user problem but treats the tutorial as an
open-ended scroll; the same Zulip threads that motivate the ticket also
show that users don't read the tutorial linearly, they search it for a
specific error or feature. Disposition B keeps a small, friendly
on-ramp page for first-time users while giving the work group a clean
place to land per-feature worked examples driven by real Zulip
failure-modes — which is the actual ask once you read past the
ticket's one-line description. It also leaves a clean migration path if
the R6 "move FML to its own spec" discussion eventually lands: the
cookbook pages move as a unit.
