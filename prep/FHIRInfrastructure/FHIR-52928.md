## Summary

The reporter (Bryn Rhodes) observes that the FHIR signatures page defines
canonicalization methods for JSON, JSON-XML and XML, plus a small set of
variants, but does not define an *artifact* canonicalization. The ticket
requests that a new canonicalization method — identified by
`http://hl7.org/fhir/canonicalization/#artifact` — be added to the
signatures page. The semantic of this new method is that the resource's
`id` and `meta` (referred to in the ticket as "metadata") are omitted
prior to canonicalization/signing, so that an artifact (e.g., a
knowledge-artifact resource such as a Library, Measure, PlanDefinition,
ValueSet, etc.) can be copied from one server to another — where
`id`, `meta.versionId`, `meta.lastUpdated`, server-assigned tags, etc.
will inevitably differ — while keeping its signature valid. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
surfaced for this ticket, so the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new canonicalization method to the FHIR Core *Digital Signatures*
page (`source/signatures.html`, "Canonicalization" section) named
**Artifact Canonicalization**, identified by the URI
`http://hl7.org/fhir/canonicalization/artifact`. Specify it as a
*modifier* canonicalization that composes with the existing JSON
(`http://hl7.org/fhir/canonicalization/json`), JSON-XML
(`http://hl7.org/fhir/canonicalization/json-xml`) and XML
(`http://hl7.org/fhir/canonicalization/xml`) base methods, in the same
manner as the other entries in the existing "Canonicalization Variants"
table. Specifically:

- Define that, prior to applying the chosen base canonicalization,
  the signer/verifier SHALL remove `Resource.id` and `Resource.meta`
  in their entirety from the resource being signed.
- State that the variant URI MAY be supplied in
  `Signature.targetFormat` either standalone (e.g.
  `application/fhir+json;canonicalization=http://hl7.org/fhir/canonicalization/artifact`)
  or combined with a base method per the existing combination rules
  (clarify the combined-URI syntax in the same way as other variants).
- Add a one-paragraph rationale: this enables an artifact to be copied
  from server to server (where `id`, `meta.versionId`,
  `meta.lastUpdated`, and server-assigned `meta.tag` / `meta.security`
  values will differ) while leaving the signature valid.
- Update the "Canonicalization Variants" table on
  `signatures.html` with the new row.

#### Justification

This is a small, additive spec change that fills a real gap: the
existing canonicalization vocabulary already anticipates "variants" that
strip dynamic content (e.g., narrative-stripped, server-data-removed),
and the requested method is a logical sibling. Knowledge artifacts
(Library, Measure, PlanDefinition, Questionnaire, ValueSet,
StructureDefinition, etc.) are routinely cloned across servers, and
server-assigned `id`/`meta` are precisely the elements that change in
that flow, so signatures over the as-stored representation break for
reasons unrelated to artifact content. Defining an explicit URI lets
implementations declare intent in `Signature.targetFormat` instead of
inventing private conventions, and aligns with how the spec already
handles canonicalization composition.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than defining a single new "artifact" canonicalization URI,
generalise the mechanism: add a **"meta-stripping" canonicalization
variant** to the existing "Canonicalization Variants" table that names
exactly which top-level elements are removed prior to canonicalization
(at minimum `id` and `meta`), and define one or more URIs under
`http://hl7.org/fhir/canonicalization/` that select well-known strip
sets. Suggested concrete shape:

- `.../canonicalization/no-id` — removes `Resource.id` only.
- `.../canonicalization/no-meta` — removes `Resource.meta` only.
- `.../canonicalization/artifact` — removes both `Resource.id` and
  `Resource.meta` (i.e., what the ticket asks for, expressed as the
  composition of the two above).

Document that these are *variants* (per the existing variants section)
that SHALL be combined with a base canonicalization (`json`,
`json-xml`, `xml`) using the existing combination URI syntax. Note
explicitly whether `meta.security` and `meta.tag` are in scope (they
should be, since they are server-assigned).

#### Justification

The underlying need — "ignore parts of the resource that the signer
does not control" — is broader than just artifacts. Some signers will
want to keep `meta.profile` or `meta.security` in the signature scope
(they encode declared conformance / labels) while still stripping
`id` and `meta.versionId`. A single monolithic "artifact"
canonicalization couples a use-case label (artifact portability) with a
specific transformation (drop both `id` and `meta`), which makes future
refinement awkward. Naming the *transformation* and providing the
artifact-portability set as a documented composition keeps the
mechanism orthogonal and easier to extend, at modest extra spec cost.
Trade-off vs. Disposition A: more URIs to maintain and a slightly
larger surface for implementers to learn, but better long-term
hygiene.

---

### Disposition C: Decline

#### Proposal

Decline to add a new dedicated `artifact` canonicalization URI to
`signatures.html`. Rationale category: *already addressable by existing
mechanism*. The existing canonicalization framework already provides a
named slot (`Signature.targetFormat`) and an extension point ("further
URIs MAY be defined"); implementation communities that need to sign
artifacts portably can register and publish a community-specific URI
(e.g., under a CQF / cross-paradigm URI) that documents the same
transformation, without burdening the FHIR Core spec.

#### Justification

The signatures page intentionally defines a small, stable core of
canonicalization methods and explicitly anticipates further URIs. A
dedicated "artifact" method introduces a use-case-named canonicalization
into the core spec where every other entry is named after a
*serialization*, which sets an inconsistent precedent. If only a single
implementation community (e.g., CQF / knowledge artifact authors) needs
this today, it is lower-risk for them to publish a URI under their own
namespace and let usage prove the design before promoting it into FHIR
Core. This avoids freezing details (e.g., should `meta.security` be
stripped? what about `meta.profile`?) into normative-track text on
insufficient evidence.

---

### Recommendation

**Recommended disposition:** A

The change is small, well-scoped, and additive: it adds one URI plus a
short paragraph to `source/signatures.html` and an extra row in the
existing "Canonicalization Variants" table — no resource, element, or
constraint changes are required. The use case (signing knowledge
artifacts that move between servers, where `id` and `meta` are by
definition server-assigned) is real, recurring, and well understood,
and the proposed URI follows the existing pattern of variant URIs under
`http://hl7.org/fhir/canonicalization/`. Disposition B is the better
long-term shape and the workgroup should consider it if there is
appetite for a slightly broader normative refactor of the variants
table; if accepted as a follow-on, the artifact URI in Disposition A
becomes the documented composition. Disposition C unnecessarily pushes
a clearly-FHIR-Core concern (how to sign FHIR resources portably) into
implementation-community URIs and risks fragmentation. Before merging
under Disposition A, the workgroup should pin down two open points in
the spec text: (1) whether stripping covers all of `meta` or only the
server-assigned subset (`versionId`, `lastUpdated`, `source`, plus
server-managed `tag`/`security`), and (2) the exact combination
syntax with the base `json` / `json-xml` / `xml` URIs in
`Signature.targetFormat`.
