## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the four prose fixes verbatim to the List resource page in
HL7/fhir under the `source/` authoring root (expected file:
`source/list/list.html`, with possible spillover into sibling narrative
files in the same `source/list/` folder if the offending strings live
in introductory or boundary text rather than the main HTML page):

1. Replace each occurrence of `doesn't` with `does not`.
2. Replace each occurrence of `isn't` with `is not`.
3. Replace each occurrence of `wasn't` with `was not`.
4. Replace `processsing` with `processing`.

Edits are made directly to the HTML source (the HL7/fhir publisher
reads `source/*.html` as authored content). No StructureDefinition
changes, no schema regeneration, no terminology changes, and no
cross-repo work are required. Verify each replacement by case (do
not silently rewrite `Doesn't` at sentence start as `does not`
without re-capitalising to `Does not`), and confirm none of the
matches occur inside `<code>` / `<pre>` blocks where the original
spelling is intentional (e.g., quoted JSON, example payloads, or
prose attributed to a third party).

#### Justification

This is exactly what the reporter asks for, and it is consistent with
the long-standing HL7 normative-prose style of avoiding contractions
in published specification text. The change is editorial only — it
does not alter the List resource's definition, cardinality,
invariants, bindings, or examples — and therefore carries no
implementer impact and no risk of breaking downstream IGs, validators,
or examples. The HL7/fhir briefing confirms `source/<resource>/` as
the correct authoring location and confirms that direct HTML edits are
the right mechanism (no FSH, no `input/`, no generated `output/` to
worry about).

---

### Disposition B: Alternative Approach

#### Proposal

Accept the four fixes from Disposition A, but **scope the work as a
batch editorial sweep** rather than a single-page fix:

- Run a repo-wide search across the `source/` authoring tree for the
  four target strings (`doesn't`, `isn't`, `wasn't`, `processsing`)
  plus a broader contraction list (`won't`, `can't`, `don't`, `it's`
  used as "it is", `aren't`, `haven't`, `hasn't`, `couldn't`,
  `wouldn't`, `shouldn't`).
- Fix all hits that are clearly normative prose, leaving alone those
  inside `<code>`, `<pre>`, third-party-quoted text, JSON examples,
  and attributed comments.
- Resolve FHIR-55172 against this single sweep and reference the
  ticket in the commit / PR description so the comment-138 import is
  closed cleanly.

#### Justification

Comment-138-imported tickets typically signal a pattern: if "list"
has these four issues, peer resource pages almost certainly have the
same contractions. Doing a single sweep is materially cheaper than
opening 50+ near-identical tickets one per page, and it produces a
more consistent specification. The trade-off versus Disposition A is
that the PR diff becomes wider (touching many `source/*.html` files)
and the reviewer load is larger, but the per-edit risk is identical
(prose only, no semantics). The HL7/fhir briefing's warning about not
editing generated areas (`<snapshot>` blocks, `tools/`, `qa/`,
`schema/`) still applies to the wider sweep but does not bite for
HTML page text.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-55172 as **Not Persuasive — editorial-only**, on the
grounds that (a) "doesn't" / "isn't" / "wasn't" are universally
understood English contractions whose meaning is unambiguous to
implementers, (b) the cost of relitigating contraction style across
hundreds of pages exceeds the readability benefit, and (c) the
single genuine typo (`processsing`) can be folded into the next
unrelated edit to the page rather than spending a Technical
Correction slot on it.

#### Justification

This disposition is defensible only as a procedural pushback against
single-typo Technical Corrections that pre-empt scarce ballot-cycle
review bandwidth. There is no substantive FHIR design principle at
stake — the underlying request is unobjectionable on the merits. The
weakness of this disposition is that it leaves a known typo
(`processsing`) and a known style inconsistency on a published page,
which is exactly the kind of long-tail debt that erodes spec polish
over time. Use only if the workgroup has an explicit policy of not
processing single-page editorial tickets through the formal pipeline.

---

### Recommendation

**Recommended disposition:** B

The four issues are real, the fix is unambiguous, and the cost of
each individual edit is trivial. Accepting exactly the four
characters-in-a-page that the reporter happened to notice
(Disposition A) would close this ticket while leaving the same
contractions in place on every neighbouring resource page, which
guarantees a steady drip of identical tickets. Folding FHIR-55172
into a single, narrowly-scoped editorial sweep across `source/`
(Disposition B) closes this ticket and pre-empts that drip at
near-zero additional risk — it is still pure prose, still edits only
authoring HTML, and still touches none of the generated areas
flagged by the HL7/fhir briefing. Disposition C should be rejected:
`processsing` is a genuine misspelling and there is no reason to
ship a known typo. If the workgroup wants to limit blast radius for
a first pass, the practical compromise is to accept Disposition B
**but constrain the sweep to the four exact strings called out in
the ticket** (no broader contraction list) — that captures the spirit
of the request without expanding the review surface.
