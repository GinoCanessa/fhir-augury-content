## Summary

The ticket asks the FHIR Infrastructure work group to narrow the meta-element
stripping behaviour of the `#static` canonicalization profile
(http://hl7.org/fhir/canonicalization/) when it is used to canonicalize a
resource prior to signing. Today `#static` strips the entire `Resource.meta`
along with `Resource.id`. Bas van den Heuvel argues that only `Resource.id`,
`Resource.meta.versionId`, and `Resource.meta.lastUpdated` should be removed,
because the remaining `meta` sub-elements — most notably `meta.security`
(security labels), `meta.tag`, `meta.profile`, and `meta.source` — carry
information that the signer typically *does* intend to bind to the signature.

No related Jira tickets are cross-referenced, but the ticket is part of a
batch of signature-review change requests filed by the same reporter
(FHIR-52810 through FHIR-52820, plus FHIR-52745) discussed on the
`fhir/infrastructure-wg > Signature comments` Zulip thread, where John
Moehrke confirms FHIR-I has the lead on Signature updates for R6. The single
on-ticket comment from John Moehrke notes that canonicalization may be needed
"too" and questions whether core can dictate a universally accepted approach.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `#static` canonicalization profile at
http://hl7.org/fhir/canonicalization/ so that, when canonicalizing a
resource prior to signing, the canonicalizer removes **only** the following
elements:

- `Resource.id`
- `Resource.meta.versionId`
- `Resource.meta.lastUpdated`

All other `Resource.meta` sub-elements (`meta.source`, `meta.profile`,
`meta.security`, `meta.tag`, and any future siblings) are **retained** in
the canonical byte sequence and therefore covered by the signature. Update
the canonicalization page narrative and any worked examples to reflect the
narrower stripping rule, and call out explicitly that `meta.security` and
`meta.tag` are signature-bound under `#static`.

#### Justification

This matches the intent stated in the ticket and is consistent with the
common interpretation that server-managed identity fields (`id`,
`versionId`, `lastUpdated`) must be excluded so that the same logical
resource produces the same canonical bytes across servers and versions,
while semantically meaningful labels (security labels, tags, claimed
profiles, source) are part of what the signer is asserting. It also aligns
with the Zulip discussion (Essien Ge), which distinguishes server-managed
metadata from signer-asserted metadata, and with the general FHIR design
principle that `meta.security` is a normative, payload-bearing element.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-defining `#static` in place, introduce an additional named
canonicalization profile — for example `#static-meta-preserving` (or
extend the existing menu of profiles in
http://hl7.org/fhir/canonicalization/) — that strips only `Resource.id`,
`Resource.meta.versionId`, and `Resource.meta.lastUpdated`, and leave the
existing `#static` profile unchanged for backward compatibility. The
Signature page would then explicitly recommend the new profile for
non-repudiation use cases that need security labels covered, while
permitting the legacy `#static` profile for integrity-only use cases (per
Essien Ge's split between integrity and non-repudiation signatures on
Zulip). Document the trade-offs and which profile a signer SHOULD pick for
each use case.

#### Justification

The Zulip thread shows the FHIR-I working assumption is that more than one
signature use case needs first-class support in R6 (authentication out of
scope; integrity vs. non-repudiation in scope). A second profile honours
that split without breaking implementations that have already adopted the
current `#static` semantics, and avoids silently changing the signed-byte
output for any existing signer. It also addresses John Moehrke's concern
that "we can[not] tell at core which will be universally accepted" by
making the choice explicit at the profile-selection level rather than
baking one answer into `#static`.

---

### Disposition C: Decline

#### Proposal

Decline the change to `#static` and instead instruct signers who need
`meta.security` or `meta.tag` covered by their signature to use a
non-`#static` canonicalization profile (e.g., a profile with
`Resource.meta` left intact, modulo `versionId` / `lastUpdated`) or to
sign a `Bundle` whose `meta` they control. Treat `#static` as a stable,
already-shipped contract.

#### Justification

`#static` is one of the published canonicalization profiles and changing
which elements it strips is, by definition, a breaking change for any
existing signer or verifier — two implementations of "`#static`" would no
longer agree on the canonical bytes, silently invalidating previously
generated signatures. John Moehrke's on-ticket comment ("I am not sure we
can tell at core which will be universally accepted") and the unresolved
state of the Zulip discussion both suggest core should be conservative
about prescribing one answer. Implementations that need `meta.security` in
the signature can already select a different canonicalization profile or
sign at the Bundle level.

---

### Recommendation

**Recommended disposition:** B

Disposition B preserves the existing `#static` contract — and therefore
existing signatures — while giving the signature spec a clean way to
express the perfectly reasonable use case the ticket is pointing at:
binding `meta.security` / `meta.tag` to the signature. It also lines up
naturally with the integrity-vs-non-repudiation split that Essien Ge
articulated on Zulip and that John Moehrke confirmed FHIR-I owns for R6,
and it directly addresses Moehrke's concern that core cannot pick a single
universally accepted behaviour by making the choice explicit at the
canonicalization-profile selection point. Disposition A is the cleanest
outcome semantically but carries the real risk of silently invalidating
any signature already produced under today's `#static` definition;
Disposition C leaves a legitimate, security-relevant gap unaddressed.
