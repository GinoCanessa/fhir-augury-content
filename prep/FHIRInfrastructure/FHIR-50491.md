## Summary

The reporter notes that the FHIR R6 ballot 3 `search.html` page still
documents `Bundle.entry.search.mode = outcome` as the way for a server
to convey an `OperationOutcome` inline with a search-set Bundle, even
though the `outcome` code in the `search-entry-mode` value set has been
deprecated in favour of the new `Bundle.issues` element introduced in
R5/R6. The ticket asks the FHIR Infrastructure work group to bring the
`search.html` narrative into line with that deprecation by replacing
the `search.mode = outcome` guidance with guidance on using
`Bundle.issues`. A first follow-up comment on the ticket flags two
additional locations on `search.html` (the `#errors` section and the
`#_maxresults` section) that still mention `entry.search.mode=outcome`
and therefore need the same treatment, so the change is a cluster of
narrative edits rather than a single sentence fix.

There are no other Jira tickets cross-referenced to FHIR-50491 in the
augury index. The only direct cross-reference is to the FHIR artifact
`Bundle.issues`, which is precisely the replacement mechanism the
ticket points at, so no sibling tickets are pulling the discussion in
a different direction.

### [implementers > ✔ Conveying additional Search Info in R4 Bundles](https://chat.fhir.org/#narrow/stream/implementers/topic/%E2%9C%94%20Conveying%20additional%20Search%20Info%20in%20R4%20Bundles/near/532333168)

Alexander Kiel asked how to convey query-execution-plan information in
an R4 search-set Bundle. He explicitly noted that he could not use
`Bundle.issues` (R5 only) and was reluctant to fall back to
`Bundle.entry.search.mode = outcome`, observing that "I don't think
that the search mode is always handled properly by clients. That might
be also the reason why it was deprecated in R5 and replaced by
`Bundle.issues`." Paul Church agreed that "using outcome is likely to
confuse clients - it isn't widely used", and Gino Canessa confirmed
that `Bundle.issues` is **not** being backported as an R5 extension on
R4 because extensions cannot contain a resource. The thread closed
with rough consensus that, in R4, an `outcome` entry (or an extension
on `Bundle.meta`) is still the only practical option, and Lloyd
McKenzie raised the open R6 design question of whether `Bundle.issues`
should map to an `outcome` entry when serialised down-version — Gino
replied that the work group has not yet decided and may use an
`outcome` entry plus a flag extension. Net effect: the implementer
community already understands `outcome` as deprecated in R5+ but is
still exposed to it through R4 servers and through cross-version
mapping, which strengthens the case for clear, current narrative on
`search.html`.

Participants: Alexander Kiel, Paul Church, Lloyd McKenzie, Gino Canessa, Notification Bot.

### [implementers > Azure FHIR API - Limits On _include & _revinclude](https://chat.fhir.org/#narrow/stream/implementers/topic/Azure%20FHIR%20API%20-%20Limits%20On%20_include%20%26%20_revinclude/near/463915950)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a technical correction to `search.html` (FHIR Core, R6 ballot 3
and current build) that replaces every reference to
`Bundle.entry.search.mode = outcome` with guidance on `Bundle.issues`.
Concretely:

- In the description of `Bundle.entry.search.mode`, remove the
  paragraph that explains the `outcome` mode and replace it with a
  short paragraph that says: "OperationOutcomes related to the search
  itself (warnings, informational notes, processing details) are
  conveyed via `Bundle.issues`. The `outcome` code in
  `Bundle.entry.search.mode` is deprecated and is retained only for
  backwards compatibility with pre-R5 search responses." Link to the
  `Bundle.issues` definition page.
- In `search.html#errors`, rewrite the sentence "This is included in
  the search results as an entry with a search mode of `outcome`" to
  describe inclusion via `Bundle.issues` instead, with a footnote
  noting the deprecated `outcome` mode for legacy servers.
- In `search.html#_maxresults`, change "or operation outcomes
  (`entry.search.mode=outcome`)" to "or operation outcomes (conveyed
  via `Bundle.issues`)".
- Leave the `search-entry-mode` `outcome` code itself unchanged
  (already marked deprecated) and add a deprecation banner / link from
  the narrative to its value-set page so readers can see the
  status-quo for legacy implementations.

#### Justification

The `outcome` code was explicitly deprecated in R5 in favour of
`Bundle.issues`, and `search.html` is the canonical narrative for
search responses in FHIR Core. Leaving the page describing a
deprecated mechanism as the primary recommendation contradicts the
value-set definition the page itself links to and confuses
implementers — exactly the situation the Zulip threads describe, where
seasoned implementers (Paul Church, Alexander Kiel) treat `outcome` as
"deprecated and probably mishandled by clients" while still seeing it
as the documented path. This is a narrative-only correction, has no
schema or wire-format impact, and aligns the page with existing
normative artefacts. As a Technical Correction the change is in scope
for the work group between ballots.

---

### Disposition B: Alternative Approach

#### Proposal

Treat `search.html` as describing both R4-style and R5+/R6 servers
side-by-side rather than wholesale replacing the `outcome` text.
Concretely:

- Promote `Bundle.issues` to the primary mechanism in the description
  of `Bundle.entry.search.mode`, in `#errors`, and in `#_maxresults`,
  matching Disposition A.
- Retain a clearly labelled "Legacy (pre-R5) servers" call-out box at
  each of the three sites that still documents
  `Bundle.entry.search.mode = outcome`, links to the deprecated value
  in the `search-entry-mode` value set, and notes that R6 servers MUST
  use `Bundle.issues` and SHOULD NOT use `outcome` going forward.
- Add a single cross-referenced anchor (e.g. `#legacy-outcome-entry`)
  that the value-set page can also link back to, so the deprecation
  story is documented in one place.

#### Justification

The Zulip "Conveying additional Search Info in R4 Bundles" thread
shows that, in practice, R4 deployments are still in heavy use and
have no path to `Bundle.issues` (Gino Canessa confirmed it is not
being backported as an extension because extensions cannot contain a
resource). Implementers reading `search.html` are not always working
on greenfield R6 servers; many are maintaining R4 servers or building
cross-version clients (the Lloyd / Gino exchange about mapping
`Bundle.issues` to an `outcome` entry on down-conversion is exactly
this case). A pure deletion of the `outcome` guidance (Disposition A)
risks leaving R4-era implementers without a documented mechanism in
the canonical page. A side-by-side narrative gives the same modern
recommendation while preserving the legacy guidance behind a clear
deprecation label, with no extra schema burden.

---

### Disposition C: Decline

#### Proposal

Decline the request as a duplicate of an already-applied normative
signal: the `outcome` code is marked deprecated in
`valueset-search-entry-mode`, and `Bundle.issues` is documented in
`bundle.html`. Add at most a one-line deprecation note next to the
existing references on `search.html` and otherwise leave the page
alone. Reasoning category: "already addressed by existing normative
mechanism".

#### Justification

The `outcome` code's deprecation is already surfaced where it is
authoritative — in the `search-entry-mode` value set / code-system
pages — and `Bundle.issues` is documented in `bundle.html`. The
`search.html` text accurately describes how a server that does choose
to use the (still-defined-but-deprecated) `outcome` mode behaves,
which is useful for clients consuming such Bundles. Rewriting the
narrative duplicates information that already lives on the
authoritative pages and creates a maintenance burden if the
deprecation story changes again. Declining preserves the description
of legacy behaviour for clients, with a small belt-and-braces
deprecation pointer.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the correctness benefit of Disposition A —
making `Bundle.issues` the primary, recommended mechanism on
`search.html` and resolving the contradiction with the deprecated
value-set entry — while honouring the practical reality that came out
of the Zulip discussions: R4 servers are still in service, cannot use
`Bundle.issues` (no backport extension is possible), and their
implementers and clients still need a documented home for the
`outcome` mechanism. Disposition A risks leaving those readers without
guidance on the canonical search page, and Disposition C leaves the
page asserting a deprecated approach as the primary recommendation,
which is precisely what the reporter is complaining about. A
side-by-side rewrite with a clear deprecation banner is the smallest
change that fixes the reported defect without regressing the page's
usefulness for cross-version implementers, and it sets up a clean
hook for the open R6 design question Lloyd McKenzie raised about how
`Bundle.issues` should map to an `outcome` entry on down-conversion.
