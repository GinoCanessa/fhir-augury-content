## Summary

The ticket asks the FHIR Shorthand (FSH) language to automatically populate
`Instance` definitions with any `defaultValue[x]` declarations that come from
the `StructureDefinition` the Instance conforms to. Today, an FSH `Instance`
already inherits values that the underlying `StructureDefinition` marks as
required (e.g., assigned codes, assigned booleans), and authors layer
additional values via Assignment rules (the only rule types permitted on
Instances are Assignment, Insert, and Path). The reporter argues that
`defaultValue[x]` — values declared in the StructureDefinition as the value to
use when no explicit value is supplied — should be treated similarly and
materialised onto generated Instances. The reporter also flags that the change
is non-trivial because `defaultValue[x]` may apply to elements with cardinality
greater than one, where automatic insertion has to interact carefully with FSH
array indices. No related Jira tickets, Zulip threads, or GitHub items were
surfaced by the cross-reference index for this ticket.

## Details

**Description:**

> Instances inherit values from their StructureDefinition (i.e. assigned
> codes, assigned booleans) if those values are required. Assignment rules
> are used to set additional values.

Rules types that apply to Instances are: Assignment, Insert, and Path. No
other rule types are allowed.

What about defaultValues? I think these should be set as well.
Doing this is tricky as it may interfere with array indices. Suggested as
future work.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the FSH `Instance` materialisation logic in SUSHI so that, in addition
to walking the StructureDefinition for elements with `fixed[x]` /
`pattern[x]` / required-bound assignments, it also walks elements with
`defaultValue[x]` and emits the corresponding value into the generated
Instance whenever the author has not already supplied a value for that
element via an Assignment rule. Applied scope:

- Apply on every `Instance` whose `InstanceOf` ultimately inherits a
  StructureDefinition with `defaultValue[x]` populated.
- Honour element cardinality: for `0..1` elements, emit the default once
  if absent. For `0..*` / `1..*` elements, do **not** auto-create array
  entries; only populate `defaultValue[x]` when the author has explicitly
  added an entry at a given index but left that element unset, mirroring
  the way required `fixed[x]` is handled inside slices today.
- Document the new behaviour in the FSH Reference under the "Instance
  Inheritance" section, with examples that show interaction with slicing
  and arrays.
- Add explicit test fixtures in SUSHI's instance-generation test suite
  covering: scalar defaults, defaults on slice members, defaults on
  elements inside repeating BackboneElements, and the no-op case where
  the Instance already supplies a value.

#### Justification

The behaviour is consistent with how FSH already projects required
StructureDefinition values onto Instances; `defaultValue[x]` is, in spirit,
the StructureDefinition's stated intent for what an unsupplied element
should be, so omitting it from generated Instances surprises authors and
diverges from the declared profile. The original 2021-10-04 ballot vote
(11-0-5, "Considered for Future Use") and Lynn Laakso's 2022 reopen note
both record agreement that "it would make sense" to do this; the only
reason it was deferred was that it had to wait for a post-normative FSH
release. The cardinality / array-index hazard the reporter flagged is
contained by restricting auto-emission to elements where the author has
not already produced a sibling value, which is exactly the rule SUSHI
already applies for required values.

---

### Disposition B: Alternative Approach

#### Proposal

Add `defaultValue[x]` materialisation as **opt-in** behaviour rather than
making it the default for every Instance. Concretely:

- Introduce a per-`Instance` keyword such as `IncludeDefaults: true` (or
  the inverse `OmitDefaults`) that toggles the new logic for that
  Instance.
- Optionally surface the same toggle in `sushi-config.yaml` as a
  project-wide default (e.g., `instanceOptions.includeDefaults: true`)
  so IGs can adopt it wholesale once they have validated the impact on
  their existing examples.
- Behaviour, when enabled, is identical to Disposition A. When
  disabled (the default), SUSHI behaves as it does today.

#### Justification

There are real downside risks to silently changing what SUSHI emits for
existing IGs: every Instance that inherits a `defaultValue[x]` from a
core or external StructureDefinition would suddenly start carrying
extra properties, which can break golden-file diffs, downstream
validators that test for *minimal* JSON, and IG comparison tooling. An
opt-in flag preserves backward compatibility while still giving authors
the requested capability, and lets IG maintainers stage the rollout.
This pattern is already established in SUSHI for behaviour-changing
features (e.g., `instanceOptions.setMetaProfile`,
`instanceOptions.setId`).

---

### Disposition C: Decline

#### Proposal

Resolve as "Persuasive with Mod / Not Persuasive — use existing
mechanism" and close without changing SUSHI's Instance generation.
Authors who want a `defaultValue[x]` to appear on a generated Instance
can already do so today by writing an explicit Assignment rule on that
Instance (`* path = <defaultValue>`). The FSH Reference documentation
for Instances should be amended to call this out, with a note that
`defaultValue[x]` is intentionally **not** auto-projected because of
the array/slicing interaction risks.

#### Justification

`defaultValue[x]` is, by FHIR's own definition, the value to use *when
the element is missing* — i.e., it is computed at consumption time, not
authoring time. Materialising the default at IG-build time changes the
on-the-wire instance and erases the distinction between "author did not
say" and "author chose the default", which downstream consumers may
legitimately want to preserve. The reporter themselves flagged the
array-index interaction as "tricky" and "future work". Given that the
existing Assignment rule already gives authors a precise, explicit way
to set the value when they want it, the cost/benefit of adding implicit
behaviour (and the risk of perturbing every IG's generated examples)
does not clearly favour the change.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the intent the workgroup already expressed in
2021 and that Lynn Laakso reaffirmed on reopening — that defaults
*should* be projectable onto Instances — while respecting the very real
backward-compatibility hazard that Disposition C correctly identifies.
Making the behaviour an Instance-level (and project-level) opt-in lets
SUSHI ship the feature in a minor release without silently changing the
output of every IG currently in maintenance, and it gives the
implementation room to handle the array-index edge cases conservatively
(only emitting defaults where the author has clearly left a slot
empty). Disposition A is the cleanest semantically but is a
behaviour-changing release for every existing FSH project; Disposition
C ignores the workgroup's stated preference. B is the path that
delivers the feature without breaking installed bases.
