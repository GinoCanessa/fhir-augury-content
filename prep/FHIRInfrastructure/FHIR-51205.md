## Summary

Bryn Rhodes is asking the FHIR Infrastructure work group to clarify, in
the definition of the `MetadataResource` data type / pattern, the
distinction between the `author` element and the `publisher` element.
The reporter proposes a specific framing: an *author* is a contributor
to the actual content of a metadata-bearing resource, whereas the
*publisher* (or steward) is the entity that is primarily responsible
for ongoing support and maintenance of that content. The change is
purely descriptive — it is a request to tighten the prose definitions
of two existing elements so authoring teams (especially those producing
knowledge artefacts such as `Library`, `Measure`, `PlanDefinition`,
`ActivityDefinition`, etc., which inherit these elements via the
metadata pattern) apply them consistently.

No cross-referenced Jira tickets, Zulip threads, GitHub items, or
Confluence pages were surfaced for this ticket, so the request stands
on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the element definitions of `MetadataResource.author` and
`MetadataResource.publisher` (and, by inheritance, the corresponding
elements on `CanonicalResource` and on every concrete metadata-bearing
resource that surfaces these elements — `Library`, `Measure`,
`PlanDefinition`, `ActivityDefinition`, `ValueSet`, `CodeSystem`,
`StructureDefinition`, etc.) so that:

- `author` is described as **"An individual or organization that
  authored / contributed to the actual content of the resource."**
- `publisher` is described as **"The name of the individual or
  organization that is primarily responsible for the ongoing support
  and maintenance of the content (the steward), and who is the canonical
  point of contact for issues with the artefact."**

Add a short note (e.g., a `comment` on each element, or a paragraph on
the MetadataResource / CanonicalResource narrative page) explicitly
contrasting the two roles, and clarifying that the same party may
appear in both lists but the *roles* are distinct: `author` =
contributor to content; `publisher` = steward / maintainer.

#### Justification

The wording proposed by the reporter is consistent with how these
fields are already used in the wild for knowledge-artefact resources
(CQF / CDS Hooks ecosystem, where Bryn Rhodes is a long-standing
contributor). A pure prose / definition clarification has no impact on
the schema, no breaking-change implications, and is the lightest-touch
fix that resolves the ambiguity. Because the MetadataResource pattern
propagates to many resources, fixing the definition in one place yields
broad, consistent improvement.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) tightening the element definitions on
MetadataResource, formalise the role distinction in the
**CanonicalResource / MetadataResource narrative page** as a dedicated
"Roles in a Canonical Resource" section that covers the full set of
party-bearing elements (`author`, `editor`, `reviewer`, `endorser`,
`contact`, and `publisher`) and explains how they relate. Cross-link
that section from each element's `comment`. Optionally introduce a
machine-readable mapping (e.g., a "role" code list) that maps each of
these elements to a `ContactDetail` purpose, so tooling can render them
consistently.

#### Justification

`MetadataResource` already carries `editor`, `reviewer`, `endorser`,
and `contact` alongside `author` and `publisher`. Clarifying only
*two* of those elements risks leaving implementers confused about the
boundary between, e.g., `author` and `editor`, or between `publisher`
and `contact`. A holistic "roles" treatment is more durable and
addresses the underlying ambiguity (who plays what role on a canonical
artefact?) rather than just one slice of it. Trade-off vs. Disposition
A: more drafting work and broader review surface, but a single,
coherent answer that won't need to be revisited the next time a
similar ticket is filed against `editor` or `endorser`.

---

### Disposition C: Decline

#### Proposal

Decline the request as already addressed by the existing definitions on
`MetadataResource.author` (sourced from `ContactDetail`) and
`MetadataResource.publisher` (a `string` naming the publishing
organisation), and by the existing pattern guidance on the
CanonicalResource / MetadataResource narrative page. Close as "Not
Persuasive" with a pointer to the existing definitions.

#### Justification

The existing `publisher` definition in FHIR Core already states that it
is "The name of the individual or organization that published the
\[artefact\]" and is widely understood as the steward / point of
contact, while `author` is already defined as a `ContactDetail` for
"An individiual or organization primarily involved in the creation and
maintenance of the content." If the work group judges the existing
prose to be sufficient and does not see field evidence of widespread
mis-use, declining avoids ripple edits across every metadata-bearing
resource and avoids re-litigating wording that has been stable across
multiple ballot cycles.

---

### Recommendation

**Recommended disposition:** A

This is a low-risk, definition-only change with clear sponsor intent
and no apparent opposition (no Zulip discussion, no related tickets,
no PRs in flight). The reporter's proposed phrasing is unambiguous and
aligns with how the knowledge-artefact community already uses these
fields, so adopting it as written gives implementers an immediate
clarity win without any schema impact. Disposition B is attractive in
principle, but expanding scope to all of `editor`/`reviewer`/`endorser`
risks blocking this small, valuable fix on a much larger drafting
exercise — the work group can pursue the broader "roles" narrative as a
follow-up ticket if it sees value, while still landing this clarification
now. Disposition C is hard to defend given that an active contributor
to the relevant ecosystem felt the distinction was unclear enough to
file a ticket; treating that as evidence that the current prose is not
doing its job argues against simply closing it.
