## Summary

The ticket reports an editorial defect on the FHIR Core "mapping-tutorial"
page: the page uses one or more contracted forms ("doesn't", "doesn't:",
"isn't") that should be fully spelled out per HL7 editorial style, which
discourages contractions and unexpanded abbreviations in normative
specification text. The request is a pure technical/editorial correction
with no scope, semantic, or interoperability impact — only the prose on
the affected page needs to be updated. No related Jira tickets, Zulip
threads, or GitHub items were surfaced by the cross-reference index, so
this appears to be an isolated editorial fix raised by Lloyd McKenzie
under the standard "Spelling/abbreviation issues on page" template.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the prose of the `mapping-tutorial` page in the FHIR Core
specification source to expand the three flagged contractions:

- "doesn't"  → "does not"
- "doesn't:" → "does not:"
- "isn't"    → "is not"

Apply the change in the page's authoring source (the markdown/HTML
fragment under the FHIR Core publication root that produces
`mapping-tutorial.html`) and re-publish. Sweep the same page for any
other contractions ("won't", "can't", "it's", "we're", etc.) while in
the file, since the reporter's pattern is "expand all contractions on
this page, not just the three I happened to spot."

#### Justification

This is a Technical Correction at Highest priority filed against a
narrative tutorial page; the requested change is purely editorial and
has no impact on conformance, schemas, value sets, or downstream
implementations. HL7 editorial guidance for normative specification
prose discourages contractions, so the fix aligns the page with house
style and with how the rest of the spec is written. Doing the broader
contraction sweep in the same edit avoids the same ticket being
re-filed against the same page next ballot cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the editorial fix on `mapping-tutorial`, and additionally run a
repo-wide lint/sweep across all FHIR Core narrative pages
(`*.html`/`*.md` source under the publication's authoring root) to
expand contractions consistently, gated by a simple regex check
(`\b(can|don|won|isn|aren|wasn|weren|doesn|didn|hasn|haven|wouldn|couldn|shouldn|it|that|there|they|we|you|I)['’]\w+`).
Land the result as a single editorial PR (or a small series scoped per
chapter) so the style is enforced uniformly rather than ticket-by-ticket.

#### Justification

The reporter has historically filed the same template against many
individual pages; each one is cheap to fix in isolation but expensive in
aggregate (workgroup review overhead per ticket dwarfs the actual edit).
A one-time spec-wide sweep dramatically reduces future ticket volume and
yields a consistent reading experience. The trade-off is a larger,
noisier diff that needs careful review to avoid changing prose inside
quoted examples, code blocks, or third-party text where contractions
are intentional — that risk is mitigated by the lint scoping to
narrative blocks only and by reviewing the diff chapter-by-chapter.

---

### Disposition C: Decline

#### Proposal

Mark as "Persuasive with Modification" → no change, on the grounds that
common English contractions ("doesn't", "isn't") are widely accepted in
modern technical writing, do not impair comprehension, and are not
called out as forbidden in HL7's current editorial guidance for
tutorial-grade pages. Close with a note that future blanket
"contractions on page X" tickets will be batched rather than processed
individually.

#### Justification

A defensible position is that `mapping-tutorial` is explicitly a
*tutorial* page, not normative conformance text, and tutorial prose
benefits from a slightly more conversational register; expanding every
contraction makes the page read more stiffly without improving
correctness or interoperability. Spending workgroup cycles on this
class of edit also has an opportunity cost against substantive issues
in the same backlog.

---

### Recommendation

**Recommended disposition:** B

This ticket is a textbook example of editorial debt that's cheaper to
clear in bulk than one ticket at a time. Disposition A makes the
specific reporter happy on this specific page but guarantees the same
template will be re-filed against the next ten pages; Disposition C is
defensible in principle but ignores that the reporter (and historically
the rest of the editorial review) treats contractions as out-of-style
for the spec, so declining will likely be appealed. Disposition B
closes FHIR-55114 with a concrete change on `mapping-tutorial` *and*
removes the recurring source of these tickets in a single editorial
pass, which is the highest-leverage outcome for FHIR Infrastructure's
time. The work group should confirm the scope (publication-wide vs.
chapter-scoped) and whether the sweep is done against the current ballot
branch or a maintenance branch before merging.
