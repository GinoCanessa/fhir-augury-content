## Summary

The ticket asks the FHIR Extensions Pack to add `Ratio` to the `context` of the
`rendered-value` extension (`http://hl7.org/fhir/StructureDefinition/rendered-value`).
The argument is that a `Ratio`'s display is a property of the numerator and
denominator *together* — for example a `Ratio` of 33/99 is more usefully
rendered as `1/3`, but you cannot safely achieve that by attaching
`rendered-value` to the numerator alone (replacing 33 with 1) or to the
denominator alone (replacing 99 with 3) because either change destroys the
underlying mathematical value. Allowing `rendered-value` on the `Ratio`
itself lets implementers carry an authoritative pre-computed display string
for the pair without distorting the constituent `Quantity` values.

There are no Jira-side cross-references, no linked tickets, no GitHub items,
and no comments yet — this is a new, isolated request against the
extensions pack.

## Details

**Description:**

> Ratios may have specific renderings that are preferred and cannot be tagged individually on the numerator and denominator elements safely.
>
> E.g., a numerator of 33 and denominator of 99 can **together** be displayed as `1/3`, but just replacing 33 with 1 can be problematic.
>
> Request adding `Ratio` to the context of the rendered-value extension.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add `Ratio` to the `StructureDefinition.context` of the `rendered-value`
extension (`http://hl7.org/fhir/StructureDefinition/rendered-value`,
canonical published in the FHIR Extensions Pack).

Concretely, append a new `context` entry:

- `context.type` = `element`
- `context.expression` = `Ratio`

No change to the extension's cardinality (`0..1` on the parent element),
type (`string`), or definition is required. The publication note for the
extension should be expanded to explain that when the extension is placed
on a `Ratio`, the rendered string represents the ratio as a whole (and
therefore implementers MUST NOT attach a `rendered-value` to
`Ratio.numerator` or `Ratio.denominator` for the purpose of representing
the combined display, since that would mis-state the underlying numeric
value of either `Quantity`).

#### Justification

The `rendered-value` extension exists to carry a presentation string when
the rendering cannot be reconstructed from the raw value (the canonical
example is masked SSN on `Identifier.value`). `Ratio` has the same
property as `Identifier` for this purpose: the preferred display
(`1/3`) is information that cannot be derived without loss from the
stored values (`33` and `99`), and there is no element below `Ratio`
where the extension can be attached without corrupting the data — the
constituent `Quantity` values must remain mathematically accurate so
that downstream consumers can compute, compare, and convert. Adding
`Ratio` to the context is therefore aligned with the extension's
original intent and with how it is already applied to `Identifier`,
which is itself a complex type that is not strictly a "single
primitive."

### Disposition B: Alternative Approach

#### Proposal

Instead of expanding the `rendered-value` context to `Ratio`, define (or
re-use) a `Ratio`-specific narrative/display extension whose semantics
are explicitly "preferred display for the ratio as a whole." Two
options:

1. Add `Ratio` to the context of the more general
   `http://hl7.org/fhir/StructureDefinition/originalText` extension
   (which is already the de-facto carrier of "what the source system
   actually showed the user"), and document the same "do not place on
   `numerator`/`denominator` for combined display" guidance.
2. Mint a new sibling extension (e.g.,
   `http://hl7.org/fhir/StructureDefinition/ratio-display`) constrained
   to `Ratio` only, returning a `string` value, with a definition that
   spells out the combined-rendering semantics directly.

#### Justification

Lloyd's stated design boundary in the Zulip thread above was that
`rendered-value` is for "simple types... a single value, not a complex
type." Honoring that boundary by routing the use case through a
purpose-built extension preserves the existing extension's
narrowly-scoped semantics, avoids setting a precedent that pulls every
future complex-type rendering request into `rendered-value`, and makes
the "do not put this on the constituent `Quantity` parts" rule easier
to discover (it lives in the new extension's definition rather than a
note buried under `rendered-value`). The downside is implementer churn
— consumers that already process `rendered-value` would need to learn a
second extension — and the trade-off is essentially semantic purity vs.
implementer convenience.

### Disposition C: Decline

#### Proposal

Decline the request and direct implementers who need a combined ratio
display to use the existing narrative mechanisms — most commonly placing
the human-readable form in the surrounding resource's `text.div`
(narrative), or in a co-located `Annotation`/comment field on the parent
resource — rather than expanding the `rendered-value` extension's
context surface.

#### Justification

This is defensible if the working group wants to hold the line that
`rendered-value` is for primitive-shaped values where the stored value
*itself* has been deliberately altered for presentation (masked SSN,
abbreviated identifier). A `Ratio`'s display is genuinely a derived
view of two faithfully-stored numbers, not an alteration, so it is
arguably a narrative concern rather than an extension concern. Declining
also avoids opening the door to similar requests for every other
complex datatype (`Quantity`, `Range`, `Period`, `Address`, `HumanName`,
`Coding`…) where a "preferred combined display" argument can be made.
The cost is that implementers lose a structured place to carry a
machine-readable display hint at the field level, and have to
synthesize it at narrative-generation time.

### Recommendation

**Recommended disposition:** A

The request fits the extension's existing role: it carries a preferred
display string when the natural rendering would distort the underlying
data. `Identifier` is already in the `rendered-value` context list and
is itself a complex type, so the "simple types only" framing from the
2020 Zulip thread is more historical than current. The reporter's
specific argument — that you cannot push the combined display onto
`numerator` or `denominator` without corrupting the `Quantity` values —
is a real correctness problem with no clean alternative *inside* the
`Ratio` itself. Disposition B is defensible but introduces a new
extension and an implementer-migration cost for what is structurally
the same operation. Disposition C effectively tells implementers "use
narrative", which loses the field-level structure that `rendered-value`
is uniquely positioned to provide. Accepting the request is the
lowest-friction, highest-consistency outcome; the only follow-on work
is the publication note clarifying that `rendered-value` on a `Ratio`
governs the pair, and MUST NOT be paired with `rendered-value` on the
constituent `Quantity` elements for the purpose of combined rendering.
