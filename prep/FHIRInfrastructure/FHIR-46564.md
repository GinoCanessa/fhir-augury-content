## Summary

Thomas Zhou requests that section **3.5.4.1 "Asynchronous Messaging using the
RESTful API"** of the FHIR Core specification be expanded with additional
context describing the orchestration components needed to manage the
asynchronous operation lifecycle. He notes that HTTP and RESTful interactions
are intrinsically synchronous, so any "asynchronous messaging via REST"
pattern necessarily depends on additional mechanisms (e.g., polling or
subscription) that are only implicit in the current text. He asks the
specification to make that coupling explicit, and in a follow-up comment
suggests integrating a link to the existing async page
(`messaging.html#async`) at the phrase "low-volume asynchronous exchange."

The ticket has been triaged with resolution **"Persuasive with Modification."**
The proposed modification (recorded in the resolution) replaces the existing
section with a new **"Asynchronous Application Responses"** subsection that
clarifies:

1. Messaging intrinsically supports asynchronous processing, so the standard
   RESTful async-processing mechanism (`async.html`) is not required for
   message exchanges.
2. When the receiver cannot fully process the event before the transport
   timeout, two patterns are offered: (a) returning a MessageDefinition
   `allowedResponse` that conveys an "in progress / pended" business
   acknowledgement in the 200 OK body, or (b) returning a 200 OK with no body
   and delivering the eventual response as a new message from receiver back to
   sender, potentially as a multi-message exchange.
3. An **Implementation Note** explains that previous releases combined
   `Bundle.id` and `MessageHeader.id` to establish message identity, which
   broke down at messaging/REST boundaries; the current release uses
   `Bundle.identifier` exclusively.
4. A new **"Message Delivery without Operations"** paragraph documents the
   alternative pattern of `create`-ing message Bundles on a server and having
   recipients poll or subscribe to retrieve and delete them, noting that
   messages always carry an implicit "please process" semantic.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items;
the only cross-references surfaced are to the FHIR artifacts named in the
proposed Implementation Note (`Bundle.id`, `Bundle.identifier`,
`MessageHeader.id`).

## Details

**Description:**

> 3.5.4.1 Asynchronous Messaging using the RESTful API
> This section might need some additional context and content to specify
> orchestration components to manage the operation process chronologically.
> Polling and subscription are two examples of mechanisms.

**Resolution Description (Persuasive with Modification):**

The Asynchronous section will be rewritten as **"Asynchronous Application
Responses"** with the following content:

> Messaging intrinsically supports asynchronous processing. While in typical
> REST, if a response is not ready right away, there's a need to use special
> [asynchronous processing](async.html). However, with messaging, responses
> can always be handled as "just another message", this time from initial
> receiver to initial sender. As a result, there's no need to use RESTful
> asynchronous processing in situations where it's not possible to provide an
> immediate response.
>
> If the system responding to a message might not be able to fully process
> the event in the timeframe required by the transport technology (e.g.
> before an HTTP timeout), there are two possibilities:
>
> 1. In some cases the MessageDefinition for the original message might have
>    an `allowedResponse` that indicates the response is "in progress",
>    "pended", or some similar business concept. In that case, such a
>    response can be included as response body of the initial 200 OK.
> 2. In other cases, there won't be an appropriate "message-level" response
>    available. In that situation, the response to the initial
>    `$submit-message` would simply be an HTTP 200 with no body.
>
> In either case, the eventual response (if one is needed) will be handled by
> transmitting a new message, this time from initial recipient to initial
> requester. In some cases, there could be a series of back and forth
> messages.
>
> - **Implementation Note:** Previous releases used a combination of
>   `Bundle.id` and `MessageHeader.id` in an attempt to establish the message
>   identity. This posed problems when crossing the boundaries between
>   messaging and RESTful exchange. The current release uses
>   `Bundle.identifier` exclusively to establish and maintain the message
>   identity. The subsequent sections reflect this change.
>
> **Message Delivery without Operations**
>
> In a RESTful environment, rather than invoking a `$process-message`
> operation, it's also possible to perform a `create` of message Bundles on a
> server and for recipients to poll or subscribe to receive the message and
> delete it once processed. Unlike "Request" resources, messages always have
> an implicit "please process" for the recipient of the message.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution text verbatim — i.e., replace the current §3.5.4.1
"Asynchronous Messaging using the RESTful API" with the new **"Asynchronous
Application Responses"** + **"Message Delivery without Operations"**
content shown in the Resolution Description, including the Implementation
Note that establishes `Bundle.identifier` (rather than `Bundle.id` +
`MessageHeader.id`) as the canonical message identity. The change is
narrative-only in `messaging.html` and does **not** alter any
StructureDefinition, ValueSet, or operation definition.

This disposition matches the workgroup's existing "Persuasive with
Modification" resolution and addresses every point Thomas Zhou raised:
async semantics over a synchronous REST transport are made explicit, the
two response patterns (`allowedResponse` body vs. empty 200 + return
message) are named, polling/subscription via Bundle `create` is documented,
and the `Bundle.identifier`-based message identity rule is clarified at
the REST/messaging boundary.

#### Justification

The reporter's underlying complaint is that the section silently relies on
mechanisms it never names. The rewrite fixes that directly: it names the
mechanisms (intrinsic message responses, `allowedResponse`, REST async
processing fallback, polling/subscription on Bundle `create`) and explains
when each applies. It is consistent with the FHIR Messaging design (which
already treats responses as "just another message") and with the
Subscriptions framework's role for server-mediated delivery. Because the
change is editorial and located in a single narrative page, the
implementer impact is low; the only normative implication — that
`Bundle.identifier` is the message identity — is already the current
behaviour and is being documented, not introduced. This is the path of
least surprise for the workgroup.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the resolution text **plus** the small editorial improvements
Thomas Zhou requested in his January 2026 comments: integrate the
`messaging.html#async` link inline at the phrase **"low-volume
asynchronous exchange"** in the section's opening paragraph, and reword
that opening sentence so the link reads naturally rather than as a
parenthetical. In addition, add a short "See also" pointer at the top of
the new section to **Subscriptions** (R5 `Subscription` /
`SubscriptionTopic`) and to the **REST `Prefer: respond-async`**
asynchronous-processing pattern (`async.html`), so readers can see at a
glance which mechanism applies to which use case (intrinsic message
response vs. server-side polling/subscribe vs. REST async). No change
to the substantive resolution text.

#### Justification

This goes slightly beyond the literal resolution but stays well within
"editorial". It picks up the link-integration request from the
reporter's most recent comments (which the current resolution text
does not yet reflect) and pre-empts the most common reader confusion —
"which of the three async-shaped mechanisms in FHIR am I supposed to
use?" — by making the cross-references navigable instead of implicit.
The trade-off vs. Disposition A is a slightly larger editorial diff and
the need for the editor to confirm the exact link target wording, but
the burden is still low and no normative content moves.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** and leave §3.5.4.1 unchanged, on the
grounds that (a) the existing section already states that polling and
subscription are valid mechanisms, (b) the broader REST asynchronous
processing pattern is fully described on a dedicated page (`async.html`),
(c) the Subscriptions framework is normatively specified elsewhere, and
(d) the proposed Implementation Note about `Bundle.identifier` vs.
`Bundle.id`/`MessageHeader.id` already appears in adjacent sections.
Direct the reporter to those existing pages.

#### Justification

A defensible decline argument exists if the workgroup judges that the
ticket is essentially asking the spec to repeat content that is already
present elsewhere, and that adding the rewrite risks duplication or
drift between `messaging.html` and `async.html`. Declining keeps the
narrative surface minimal and avoids introducing a new section name
("Asynchronous Application Responses") that overlaps with the existing
"Asynchronous Interactions" terminology used by the REST chapter.
However, this position is weak: the ticket and the comment thread
demonstrate that even an experienced reader missed the implicit
coupling, which is exactly the gap a normative spec should close.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B carries the workgroup's existing "Persuasive with
Modification" resolution forward — which is already the right call,
because the reporter's clarifications and Lloyd's drafted rewrite show
that the current section genuinely under-specifies the orchestration
story — and folds in the small editorial follow-up that the reporter
made in January 2026 (the `messaging.html#async` inline link plus the
opening-sentence rewording). Adding the "see also" pointers to
Subscriptions and `async.html` costs almost nothing and prevents the
next reader from filing the same ticket against §3.5.4.1. Disposition A
is a perfectly safe fallback if the workgroup wants to ship the
resolution exactly as drafted and treat the link-integration request as
a separate editorial pass; Disposition C is not recommended because the
ticket history (a co-chair asking the reporter to clarify, and the
reporter then itemising four specific gaps) is itself evidence that the
current text is doing its job poorly.
