## Summary

Eric Haas is asking the FHIR Infrastructure work group to consider whether
`SubscriptionTopic` should grow a notion of "must-support" vs. optional
elements (filters, resource triggers, notification shape parameters, etc.).
Today, a `SubscriptionTopic` is an all-or-nothing contract between the
server and its clients: a server either supports the topic as authored or
it does not. In large, reuse-oriented domains — the ticket cites SDOH as
an example — this forces authors either to bloat a topic with filters
that many servers cannot support, or to fragment one conceptual topic
into many near-duplicates that differ only by a single filter or trigger.
The ticket frames this as a design question — is lack of optionality
intentional, or should `SubscriptionTopic` gain optionality semantics so
a single topic can be reused across implementations with varying
capability? — and explicitly asks the work group to decide whether the
added complexity is justified.

No cross-referenced tickets, Zulip threads, or GitHub items were
returned for this ticket key.

## Details

**Description:**

> Currently there is no notion of server optionality for a SubscriptionTopic
> item; in other words, it's an all-or-nothing thing. Therefore, if an
> SDOH topic has too many filters, etc., to be a minimum, a new
> SubscriptionTopic needs to be created. This limits reuse and leads to
> many SubscriptionTopics that could differ by a single filter, etc.
> Maybe this is by design and SubscriptionTopics are not widely reused.
> Otherwise we need to discuss whether a SubscriptionTopic should have
> a concept of "must-support" vs optional for further reuse across
> implementations and use cases and whether the added complexity is
> justified.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Introduce explicit optionality semantics on `SubscriptionTopic` elements
that today act as the authored contract — specifically `resourceTrigger`,
`eventTrigger`, `canFilterBy`, `canInclude`, and `notificationShape` — so
a topic author can mark each entry as either required (a server claiming
support for the topic MUST honour it) or optional (a server MAY honour
it, and if it does not, clients can still use the topic against a
reduced surface). The most natural mechanism is to add a `mustSupport`
(or equivalent `required`/`optional` code) flag on each of those
backbone elements, paired with normative text in the `SubscriptionTopic`
narrative describing the conformance expectation for servers that claim
the topic in `CapabilityStatement.rest.resource.supportedProfile` or in
the subscription-specific conformance block. Servers advertising a topic
would then be expected to support every required sub-element and MAY
omit optional ones; clients creating `Subscription` resources against
that topic would be required to confine `filterBy` / `include` choices
to the intersection of the topic's authored options and the server's
advertised capability.

#### Justification

This matches the pattern already used throughout FHIR for partial
conformance to a structural contract (`mustSupport` on
`StructureDefinition`, optional slices in profiles) and would let a
single topic — e.g., an SDOH notification topic — carry a rich optional
surface while still defining a hard minimum that all conformant servers
must honour. It directly addresses the reporter's stated pain point
(topic fragmentation driven purely by which filters a given server can
support) without inventing a new conformance primitive: servers and
clients already understand "must-support" semantics.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `SubscriptionTopic` as the invariant authored contract and instead
introduce a server-side capability-advertisement mechanism that
describes, per topic, which optional features the server actually
implements. Concretely: extend the subscription-related
`CapabilityStatement` extensions (or define a companion
`subscription-topic-capability` extension / operation) so that a server
publishing support for a topic URL also advertises which `canFilterBy`
entries, `canInclude` entries, and `notificationShape` params it honours,
and the expected behaviour when a client requests an unsupported one
(reject at `$subscribe`/create time vs. silently ignore). The topic
itself stays a single, stable artefact; optionality lives on the server,
not on the topic.

#### Justification

`SubscriptionTopic` is a shared, often cross-jurisdiction artefact —
multiple servers publish against the same topic URL, and changing the
topic to carry optionality semantics risks re-versioning topics every
time the set of optional features is refined. Putting optionality on
the server's capability advertisement keeps the topic stable, keeps
conformance checks client-visible (the client reads the server's
capability document before crafting a `Subscription`), and avoids adding
`mustSupport`-style flags to a resource whose semantics are already
subtly different from profiling. It also dovetails with the existing
R5 Backport pattern, where the subscription machinery leans heavily on
capability-statement extensions rather than on the topic resource.

---

### Disposition C: Decline

#### Proposal

Decline the request as "already addressed by existing mechanism." The
current design intentionally treats `SubscriptionTopic` as an atomic
conformance target: if a server cannot honour every authored
`resourceTrigger` / `canFilterBy` / `canInclude` entry for a topic, it
should publish a derived topic (with its own canonical URL) that
represents the subset it does support. Topic fragmentation is the
intended cost of a clear, verifiable client contract, and the R5
Backport IG and the core R5 Subscriptions chapter both assume this
model.

#### Justification

Introducing optionality on `SubscriptionTopic` creates a second,
subtler conformance axis on top of an already complex resource and
would require corresponding machinery in `Subscription` creation,
matching, and notification-shape resolution — every client would have
to reason about "which subset of this topic does this particular
server implement?" at runtime. The existing pattern (derive a narrower
topic, reference the parent via `derivedFrom`) already solves the reuse
problem without changing the resource contract, and avoids breaking
servers that today rely on an all-or-nothing conformance check. The
reporter himself flags that the added complexity may not be justified;
absent concrete implementer demand beyond the SDOH example, the cost
outweighs the benefit.

---

### Recommendation

**Recommended disposition:** B

Disposition B best balances the real reuse pain the reporter describes
against the cost of changing a widely-referenced conformance artefact.
`SubscriptionTopic` instances are shared across jurisdictions and IGs,
and baking `mustSupport` into the topic (Disposition A) would entangle
the topic's version lifecycle with every server's evolving capability
set — a `SubscriptionTopic` whose optionality flags change is a new
topic. Disposition C understates the reporter's point: the existing
`derivedFrom` workaround does produce the topic-fragmentation problem
Eric describes, especially in SDOH-style ecosystems with many
near-identical topics. Putting optionality on the server's capability
advertisement (Disposition B) keeps one canonical topic, makes per-server
variation discoverable by clients before they `$subscribe`, and reuses
an extension surface the Subscriptions R5 Backport already relies on.
The work group should decide whether to pursue this as a new extension
in the Backport IG first (lower-risk, faster) and feed it back into
core R5 on a later cycle.
