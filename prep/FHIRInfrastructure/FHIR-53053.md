## Summary

The reporter observes a long-standing inconsistency in the Bundle resource:
`Bundle.entry.response.outcome` is typed as the abstract `Resource` (so any
resource is structurally permitted), yet the element's narrative says
"OperationOutcome with hints and warnings (for batch/transaction)." The
request is that, if the declared type cannot be tightened to
`OperationOutcome`, the spec should be updated so the description makes the
intent explicit and adds a non-normative SHOULD that servers return an
`OperationOutcome` when conveying processing details or errors for a
batch/transaction entry.

This request explicitly revisits the historical attempt
[FHIR-14427](https://jira.hl7.org/browse/FHIR-14427) (2018), which proposed
changing the datatype outright and was Resolved – Not Persuasive after the
change was made and then reverted because of widespread fallout in
infrastructure and invariants. It is also closely aligned with two more
recent, accepted tickets that addressed the same pattern in adjacent
elements: [FHIR-46484](https://jira.hl7.org/browse/FHIR-46484) (Applied),
which clarified `Bundle.issues` via short/definition/comment updates noting
it is an OperationOutcome — explicitly modelled on what was done for
`Bundle.entry.response.outcome` — and
[FHIR-46561](https://jira.hl7.org/browse/FHIR-46561) (Applied), which
updated rendering to display the profile constraint on `Resource`-typed
elements (e.g. `Resource(OperationOutcome)`). Together these establish a
clear precedent for the textual / rendering remedy this ticket is asking
for. The originating
[Zulip thread](https://chat.fhir.org/#narrow/stream/implementers/topic/Should%20Bundle.entry.response.outcome%20be%20constrained%20to%20OO%20%3F)
is the immediate trigger: Gino Canessa explained the history, agreed the
text should be improved, and asked the reporter to file this ticket.

## Details

**Description:**

In R4, `Bundle.entry.response.outcome` is typed as `Resource`. The base
spec doesn't constrain it to `OperationOutcome`, yet the description says
"OperationOutcome with hints and warnings (for batch/transaction)." If the
type cannot be changed, please clarify in the spec that `OperationOutcome`
is the intended/typical resource, and add a non-normative SHOULD for
servers to return `OperationOutcome` when providing processing details or
errors.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a text-only update to `Bundle.entry.response.outcome` in the Bundle
StructureDefinition, mirroring what was done for `Bundle.issues` under
FHIR-46484:

1. **Short display** — change to make the OperationOutcome intent explicit,
   e.g. "OperationOutcome with hints and warnings (for batch/transaction)
   — typed as Resource for historical reasons; servers SHOULD use
   OperationOutcome."
2. **Definition / comments** — expand the comment to state that, although
   the declared type is the abstract `Resource`, the element is profiled
   to `OperationOutcome` and servers SHOULD return an `OperationOutcome`
   when conveying processing details, warnings, or errors for a batch or
   transaction entry; clients SHOULD be prepared for any resource for
   forward compatibility.
3. **Non-normative SHOULD** — add the SHOULD as conformance guidance in
   the surrounding Bundle narrative (the batch/transaction processing
   section), not as a new invariant, to avoid the "long sting in the tail"
   that derailed FHIR-14427.
4. **Cross-link to FHIR-46561 rendering** — confirm the published
   rendering shows `Resource(OperationOutcome)` so the textual claim and
   the rendered type line up.

No type change, no new invariant, no profile changes — purely descriptive.

#### Justification

This is exactly what the reporter asked for and what Gino Canessa
pre-agreed to on Zulip. There is a directly applicable, recently accepted
precedent: FHIR-46484 disposed of the structurally identical issue on
`Bundle.issues` with a description/short/comment update, explicitly modelled
on prior treatment of this very element. Text-only changes avoid the
infrastructure breakage that caused FHIR-14427 to be reverted (Grahame
Grieve's "long sting in the tail" comment is the standing rationale for
not touching the type or adding an invariant). Adding the SHOULD as
narrative rather than as a constraint preserves backwards compatibility
with any deployed implementations relying on the abstract `Resource`
typing.

---

### Disposition B: Alternative Approach

#### Proposal

Take a more limited "documentation + rendering" approach: do **not** add a
new SHOULD, but (a) update the element description and comment so the
profile constraint and intent are unambiguous (essentially step 2 of
Disposition A), and (b) verify and, if necessary, finish the FHIR-46561
rendering work so that `Bundle.entry.response.outcome` is rendered as
`Resource(OperationOutcome)` (matching how `Reference(Patient)` is
rendered). Coordinate the wording with FHIR-53947 (intro section general
improvements) so the prose change lands as part of a consistent pass over
all `Resource`-typed elements (`Parameters.parameter.resource`,
`Bundle.entry.resource`, `Bundle.issues`,
`Bundle.entry.response.outcome`).

#### Justification

The trade-off vs. Disposition A is conformance weight. A non-normative
SHOULD is still observable: implementers will read it, validators may
choose to flag deviations, and downstream IGs may quote it as guidance.
FHIR-46484's accepted resolution did **not** add a SHOULD; it stuck to
short/definition/comment updates. This option is more conservative and
aligns more tightly with that precedent. It also reduces risk of
inconsistency by deferring the prose phrasing into a coordinated pass with
FHIR-53947, which already enumerates this same set of elements. The cost
is that "servers SHOULD return OperationOutcome" — which is genuinely the
implementer-facing guidance the reporter wants — would be left implicit in
the description rather than called out as a SHOULD.

---

### Disposition C: Decline

#### Proposal

Resolve as Not Persuasive / Already Addressed: the underlying concern is
already covered by the combination of the existing element profile to
`OperationOutcome` plus the rendering correction from FHIR-46561 and the
text already updated under FHIR-46484's pattern. Direct the reporter to
those tickets and close.

#### Justification

If the rendering from FHIR-46561 ships visibly on the published spec, the
declared type for `Bundle.entry.response.outcome` will appear as
`Resource(OperationOutcome)` — directly answering the reporter's "any
resource is allowed" surprise. The FHIR-14427 history establishes that
even small attempts in this area have caused outsize breakage, and a
SHOULD added now risks being lifted into validators or IGs in ways that
disrupt deployed batch/transaction servers that legitimately return other
resource types (or none) in `outcome`.

---

### Recommendation

**Recommended disposition:** A

The combination of evidence makes Disposition A the right starting point.
The Zulip thread shows pre-existing FHIR Infrastructure agreement (Gino
Canessa explicitly invited the ticket and described the intended fix). The
most recent accepted precedent on the structurally identical sibling
element (FHIR-46484, Applied, Persuasive) demonstrates the workgroup's
preferred remedy is a textual clarification on `Resource`-typed elements
that are intended to carry `OperationOutcome`, not a type change and not
silence. The rendering work in FHIR-46561 is complementary, not a
substitute — readers of the element table see the type, but the
description is what carries the *why*, the *when* and the *should*.
Disposition C undersells the genuine implementer confusion documented in
both the 2018 and 2025 Zulip threads. Disposition B is a reasonable
fallback if the workgroup is uncomfortable promoting the SHOULD: drop the
SHOULD bullet from Disposition A and you are effectively in B, so the
choice between A and B is just whether to spell the guidance out as a
SHOULD or leave it implicit. Recommend going with A and coordinating the
exact wording with FHIR-53947 so all four affected elements get
consistent treatment in the same publication cycle.
