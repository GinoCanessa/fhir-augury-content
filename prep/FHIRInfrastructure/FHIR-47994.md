## Summary

FHIR-47994 is a "Various technical corrections" ticket raised by Bas van
den Heuvel during the September 2024 R6 ballot. It bundles a tranche of
small, non-substantive editorial fixes to the FHIR core specification —
typo, grammar, punctuation, and clarity corrections, plus a few HTML
formatting cleanups — into a single Jira item, with the actual change
set delivered as
[HL7/fhir#3163](https://github.com/HL7/fhir/pull/3163) ("Bvdh sept2024
ballot technical corrections"). The PR touches 23 files under `source/`
across multiple resources/pages (ServiceRequest, Task, Subscription,
SubscriptionStatus, Consent, Contract, async, security, versions, the
spelling dictionary, etc.) but every diff is editorial: expansions of
the abbreviation "PHI" to "PHI (Personal health information)",
"this/the async pattern", "informations" → "information", "have been"
→ "has been", "depend" → "depends", capitalisation of "task"/"Task",
removal of stray UTF-8 corruption strings such as `i+R[9]Cs`, addition
of missing periods/closing parens, and re-flowing of long paragraphs in
`servicerequest-introduction.xml` and `task-introduction.xml`. The PR
was merged on 2024-11-12 by Graham Grieve, and per the only ticket
comment Grieve has already accepted the change in his role as product
director and asks FHIR-I to "rubber stamp the task even though it
technically covers multiple committees." The disposition decision in
front of FHIR Infrastructure is therefore essentially a
post-hoc ratification rather than a design call.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the ticket as a technical correction and ratify
[HL7/fhir#3163](https://github.com/HL7/fhir/pull/3163) as already
merged. Concretely:

- Mark the ticket **Persuasive — Technical Correction**, resolution
  *Done*, applied in PR #3163 (merged into `HL7/fhir@master` on
  2024-11-12 by `grahamegrieve`).
- Take the change set as-is — all 23 `source/`-rooted edits in PR
  #3163, namely:
  - PHI expansions ("PHI" → "PHI (Personal health information)") in
    `source/security.html`, `source/consent/consent-example.xml`,
    `source/consent/consent-example-notAuthor.xml`,
    `source/contract/pcd-example-notAuthor.xml`,
    `source/subscription/subscription-notes.xml`,
    `source/subscriptionstatus/subscriptionstatus-notes.xml`.
  - Async-pattern wording: "this async pattern" → "the async pattern"
    in `source/async-bulk.html` and `source/async-bundle.html`; new
    intro sentence "FHIR supports two asynchronous patterns:" in
    `source/async.html`.
  - StructureDefinition `<differential>` text fixes: ServiceRequest.code
    "have been" → "has been"; Subscription.parameter
    `short` "Channel type" → "Channel type dependent information" and
    `comment` "depend" → "depends"; Task `<description>`
    "informations" → "information"; Task.intent
    `definition` removes the `i+R[9]Cs` corruption (also in
    `task-request-mapping-exceptions.xml` and `task.svg` `<title>`);
    Task.requestedPerformer `short` "Who should perform Task" → "Who
    should perform the Task" (also propagated to
    `task-event-mapping-exceptions.xml`).
  - ServiceRequest and Task introduction prose re-flow / capitalisation
    of "task" → "Task" where the resource is meant.
  - `source/versions.html` punctuation fixes (missing periods, missing
    closing parenthesis after "inter-version compatibility rules", and
    "fwe" → "few").
  - `source/fivews/fivews-spreadsheet-introduction.xml`: turn "Five
    Ws" into a hyperlink to the Wikipedia article.
  - `source/spelling/add.txt`: remove obsolete allow-list entries
    `excluded_structure`, `lifecycle`, `subject_state`,
    `included_structure`, `hrf`.
- No further edits required; close as resolved.

#### Justification

Every diff in PR #3163 is editorial: spelling, grammar, punctuation,
HTML/XML formatting, removal of an internal corruption artefact
(`i+R[9]Cs`) that was already present in normative text, and PHI
abbreviation expansion. None changes a resource's structure,
cardinality, binding, invariant, or semantics, so the change is by
definition a "Technical Correction" under the FHIR change classification
and does not require workgroup-by-workgroup re-review. The product
director has already vetted the PR and merged it (Grieve's 2025-03-07
comment), so declining here would create a divergence between Jira and
the master branch with no editorial benefit. The Repo Context confirms
the changes touch only the authoring surface (`source/...` HTML and
`<differential>` text), avoid generated areas (`<snapshot>`, `tools/`,
`qa/`, `schema/`, `publish/`), and stay within FhirCore — there is no
cross-repo (UTG, fhir-extensions, IG) coupling to coordinate. Rubber-
stamping is the disposition that matches what is actually in master.

### Disposition B: Alternative Approach — Accept with Targeted Follow-up

#### Proposal

Accept and ratify PR #3163 as in Disposition A, **but** create three
small follow-up tickets / actions to close the loose ends the PR
exposes, none of which block this disposition:

1. **Spelling allow-list QA sweep.** PR #3163 removes
   `excluded_structure`, `lifecycle`, `subject_state`,
   `included_structure`, and `hrf` from `source/spelling/add.txt`.
   File a follow-up to (a) run the publisher's spell-check QA against
   `master` and (b) for any of those tokens that still appear in
   authored pages, either restore the dictionary entry or fix the
   page. This is a one-off QA task, not a spec change.
2. **Owning-committee notice.** The PR edits content nominally owned by
   multiple committees (Orders & Observations for ServiceRequest;
   Patient Care/FHIR-I for Task; Security for security.html and
   PHI expansions; FMG for versions.html; FHIR-I for async). Per
   Grieve's comment FHIR-I is rubber-stamping on their behalf, so add
   a single note on the ticket listing the affected committees so the
   Jira record makes that delegation explicit and any future audit can
   confirm consent.
3. **Snapshot regeneration verification.** Confirm that the next
   publisher run regenerates the `<snapshot>` blocks in
   `structuredefinition-ServiceRequest.xml`,
   `structuredefinition-Subscription.xml`, and
   `structuredefinition-Task.xml` so that the corrected `short` /
   `definition` / `comment` text propagates everywhere it is rendered
   (resource summary tables, definitions pages, etc.). This is
   automatic per the briefing's "generated areas" rule but is worth a
   smoke check given the StructureDefinition edits.

#### Justification

This keeps the substantive answer the same as Disposition A — accept
and ratify the merged PR — while addressing the two items the PR set
up but does not itself finish (spelling-dictionary fall-out and
multi-committee provenance), and explicitly verifying that the
StructureDefinition `<differential>` edits flow through the publisher
into the rendered pages. It costs nothing in spec terms, gives the
audit trail FHIR-I would normally want when ratifying another
committee's text, and matches the briefing's Gotchas (do not edit
`<snapshot>`; the publisher regenerates them — but verify) and its
Recommended Change Recipe for narrative/prose updates. It is a
strictly additive variant of A.

### Disposition C: Decline

#### Proposal

Mark FHIR-47994 **Not Persuasive** and request that PR #3163 be split
into per-committee tickets so that each affected committee (Orders &
Observations for ServiceRequest, Patient Care for Task, Security for
security/PHI changes, FHIR-I for async/versions, etc.) reviews and
approves only its own surface. Until that is done, do not ratify the
change here.

#### Justification

The ticket is filed as a single FHIR-I item but the change set spans
content owned by other workgroups, which is normally a process
violation under HL7 governance (each workgroup approves its own
artefacts). Declining-to-split would force the normal process and
would make the audit trail per-committee.

This rationale is real but weak in this case: every change is
demonstrably editorial (typo, grammar, punctuation, abbreviation
expansion, paragraph reflow, removal of binary corruption), the
product director has already exercised the cross-committee technical-
correction prerogative and merged the PR, and re-opening
per-committee tickets would generate substantial committee load with
zero specification effect. Disposition C is included for completeness
— it is the position to take only if FHIR-I wants to push back on the
"product-director rubber-stamp" pattern as a matter of process.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. Substantively it is identical to A — the
PR is already merged, every change is editorial, and FHIR-I is being
asked by the product director to ratify on behalf of the multiple
committees whose pages were touched. Adding the three lightweight
follow-up actions (spelling-dictionary QA sweep, an explicit
multi-committee delegation note on the ticket, and a verification that
the StructureDefinition `<differential>` edits flow through the
regenerated `<snapshot>` blocks) costs almost nothing and addresses
the two real concerns the change set raises: a small QA debt from the
removed `add.txt` entries, and the cross-committee governance optics
flagged in Grieve's comment. Disposition C is not warranted — the
content is genuinely a Technical Correction and the change is already
in `master`, so declining would create a divergence between Jira and
the spec for no editorial benefit. Disposition A is acceptable if the
workgroup wants the absolute minimum overhead; B is preferred because
it leaves a cleaner audit trail.
