## Summary

The reporter (Weiyu Zhang) has identified a rendering defect in the FHIR
Core specification's `SubscriptionTopic` resource pages: several inline
HTML hyperlinks that point at `elementdefinition-definitions.html#ElementDefinition.type.code`
appear in the rendered output as **plain text** rather than as clickable
links. The affected text — "For more information, see
ElementDefinition.type.code." — appears in the detailed descriptions of
three sibling elements: `SubscriptionTopic.trigger.resource`,
`SubscriptionTopic.trigger.canFilterBy.resource`, and
`SubscriptionTopic.trigger.notificationShape.resource`. The expected
behaviour is that the anchor markup authored in the source descriptions
is preserved through the publisher and emitted as a real `<a>` tag in
the generated definitions page. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this ticket, so the report
relies solely on the ticket text and the FHIR Core repository
conventions.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a Technical Correction against `HL7/fhir`. In
`source/subscriptiontopic/` locate the StructureDefinition
authoring file for `SubscriptionTopic` (typically the bespoke
`StructureDefinition-SubscriptionTopic.xml` / `.fsh-equivalent`
authored differential) and inspect the `definition` text on the three
elements — `SubscriptionTopic.trigger.resource`,
`SubscriptionTopic.trigger.canFilterBy.resource`, and
`SubscriptionTopic.trigger.notificationShape.resource`. Replace the
literally-escaped anchor (currently rendering as
`<a href="elementdefinition-definitions.html#ElementDefinition.type.code">ElementDefinition.type.code</a>`
in the published view) with markup that the publisher will emit as a
real anchor tag in the definitions page. Concretely: ensure the
`definition` content uses the publisher's expected escaping for inline
HTML in element definitions (e.g., real `<` / `>` rather than
`&lt;` / `&gt;` in the source XML, or use the markdown-link form
`[ElementDefinition.type.code](elementdefinition-definitions.html#ElementDefinition.type.code)`
if the field is now markdown-typed). After the fix, regenerate the
`<snapshot>` via the publisher and verify
`subscriptiontopic-definitions.html` shows three clickable links.

#### Justification

This is a pure rendering / authoring-escape defect, not a normative
change. The reporter has narrowed the problem to three sibling
elements with identical boilerplate, so the fix is small, local, and
mechanical. The briefing for HL7/fhir confirms `<differential>` content
is hand-editable and that authored markup flows through the publisher,
which is exactly the surface that needs adjustment. No spec semantics
change, so there is no backward-compatibility risk — only the rendered
HTML improves.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than patching only the three SubscriptionTopic descriptions,
fix the problem at the **publisher / template level** so that any
authored anchor of the form
`<a href="elementdefinition-definitions.html#…">…</a>` in any element
`definition` is rendered as a hyperlink across all resource definition
pages. Audit the rendering pipeline for element `definition` text in
`*-definitions.html` to determine whether anchors are being
HTML-escaped by an over-zealous sanitiser, then either (a) whitelist
intra-spec links, or (b) replace the boilerplate everywhere with a
publisher macro / well-known token that always renders as a link
(e.g., a markdown reference to ElementDefinition.type.code that all
resource pages can share).

#### Justification

The reporter mentions "Some Links" — i.e., the underlying cause is
unlikely to be unique to SubscriptionTopic. Other resources almost
certainly contain the same "For more information, see
ElementDefinition.type.code." boilerplate, and a per-resource patch
will leave equivalent defects elsewhere. A publisher- or
boilerplate-level fix is durable, prevents regressions in future
resources, and aligns with the FhirCore briefing's note that
specification pages and element descriptions are emitted by a bespoke
publisher whose escaping rules can be tuned in one place. Trade-off:
larger blast radius — the fix touches the publisher / template layer,
requires regression-checking other resource pages, and may need
FHIR-I tooling sign-off rather than a SubscriptionTopic-only edit.

---

### Disposition C: Decline

#### Proposal

Decline as **insufficient evidence of a publisher defect**. Mark
the ticket as Not Persuasive on the grounds that the rendered HTML for
those elements may already be a correct (if visually plain) display of
escaped angle-brackets in the source — i.e., the original definition
text might intentionally embed literal `&lt;a …&gt;` sequences as
documentation about anchor markup, not as a live link. Ask the
reporter for a screenshot, the exact published page URL, and the
rendered HTML so the workgroup can confirm there is a real defect
before opening editing time on it.

#### Justification

The ticket has no attachments, no comments, no Zulip discussion, and
no cross-referenced PRs or issues. The description is a single
sentence plus a snippet. Declining (or at minimum returning for more
information) avoids spending publisher / FHIR-I cycles on a fix that
might mis-target the underlying cause. This is the weakest position
because the reporter's claim is plausible on its face, but it remains
defensible if the workgroup wants to enforce a standard of evidence
for rendering bugs.

---

### Recommendation

**Recommended disposition:** A (with a follow-on consideration of B).

This is exactly the kind of low-risk technical correction the
"Technical Correction" type exists for: a named reporter, three
specifically identified element paths, an obvious expected rendering,
and zero normative impact. Disposition A unblocks the immediate
defect on the SubscriptionTopic definitions page with a
small, local edit under `source/subscriptiontopic/`. Once the fix is
in, FHIR-I should open a separate ticket to investigate whether the
same boilerplate appears on other resource definitions pages — that
broader audit (Disposition B) is the right durable fix but is too
large in scope to bundle with this Technical Correction. Disposition
C is not recommended: the reporter has been precise enough that
asking for more evidence is more friction than just opening
`subscriptiontopic-definitions.html` and confirming the defect.
