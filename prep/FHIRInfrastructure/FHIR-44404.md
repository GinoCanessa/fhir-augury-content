## Summary

This change request, filed against the **FHIR R5 Subscriptions Backport
(FHIR)** specification, asserts that any code systems and value sets
defined or referenced by the IG should be anchored in
`terminology.hl7.org` (THO) rather than being defined inline within the
specification. The request invokes the TSC-approved policy
"Terminology Expectations for IG Developers" and directs that the
applicable code system(s) be moved through the **UTG (Unified
Terminology Governance) process** so they live in THO. The reporter
notes this is "Comment 2" of a series, suggesting a coordinated
sweep of similar tickets across multiple specifications.

The cross-reference query returned no related Jira tickets, Zulip
threads, or GitHub items for this ticket key, so the substance of the
request stands on its own without recorded community discussion in the
augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the request and start the UTG process to migrate the
Subscriptions Backport IG's locally-defined code system(s) and value
set(s) into `terminology.hl7.org`. Concretely:

1. Identify every `CodeSystem` and `ValueSet` resource currently
   defined inside the FHIR R5 Subscriptions Backport IG whose canonical
   URL is **not** under `http://terminology.hl7.org/...`.
2. For each such artifact, file a UTG proposal (per the THO submission
   workflow) to move the resource into THO under an appropriate
   `terminology.hl7.org/CodeSystem/...` /
   `terminology.hl7.org/ValueSet/...` canonical.
3. Once accepted into THO, replace the inline definitions in the
   Subscriptions Backport IG with references (by canonical URL) to the
   THO-hosted artifacts, and update any binding statements on profiles
   / extensions / SearchParameters in the IG accordingly.
4. Add release notes / migration guidance to the IG narrative noting
   the canonical URL change.

#### Justification

This is the literal action requested and aligns directly with the
TSC-approved [Terminology Expectations for IG
Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy, which states that HL7-published code systems and value sets
should be governed through THO/UTG rather than embedded in individual
IGs. Anchoring in THO yields:

- Centralized terminology governance and a single authoritative
  canonical URL.
- Reuse across other specifications (Core R5, R4 backport consumers,
  other IGs) without each restating the codes.
- Consistency with the Highest priority assigned to this ticket and
  with the broader "Comment 2" family of tickets the reporter is
  filing against multiple specifications.

### Disposition B: Alternative Approach

#### Proposal

Accept the *intent* of the request but stage it: in the **current**
publication of the Subscriptions Backport IG, leave the existing
canonical URLs in place to avoid breaking implementers that have
already coded against them, and instead:

1. Open the UTG submission(s) now so the THO-hosted equivalents are
   created and reviewed.
2. In THO, define the canonical(s) and add a `ConceptMap` (or simple
   `valueSet.compose.include.valueSet`) that aliases the IG-local
   canonical to the THO canonical.
3. Mark the IG-local definitions as **deprecated** in the next IG
   publication, and switch all bindings inside the IG to the THO
   canonicals while keeping the old artifacts as
   `status = retired` shells that point readers to the THO replacement.
4. Remove the deprecated shells in the publication after the
   deprecation window.

#### Justification

The Subscriptions Backport IG is already published and may be in
production use. A hard cut-over to new canonical URLs (Disposition A)
risks silently breaking implementers whose validators, terminology
caches, or hard-coded bindings reference the IG-local URLs. A staged
deprecation:

- Achieves the policy goal (THO-anchored governance going forward)
  without an immediate breaking change for current implementers.
- Gives downstream consumers time to repoint to the THO canonical(s).
- Preserves the audit trail by leaving the old artifacts retired
  rather than deleted.

This is the standard pattern HL7 uses when relocating canonical URLs
across publication boundaries.

### Disposition C: Decline

#### Proposal

Decline the request as filed against this specification, on the
grounds that the Subscriptions Backport IG may not actually define any
new code systems requiring THO migration — it is primarily a
*backport* of R5 Subscriptions machinery to R4/R4B and most of its
terminology dependencies are already either in THO or in FHIR core.
Close the ticket as **Not Persuasive — Not Applicable** with a
direction to refile against any specific code system that the
reporter believes needs to be moved, naming the artifact.

#### Justification

The ticket text is generic ("this code system") and does not name a
specific `CodeSystem` or `ValueSet` resource. No augury
cross-references, Zulip threads, or GitHub items were located that
identify the offending artifact(s). Without that specificity:

- A wholesale "move all terminology to THO" disposition cannot be
  scoped or estimated.
- The UTG submission(s) cannot be prepared, because UTG requires the
  source artifact(s).
- Acting on the ticket risks proposing a migration of artifacts that
  are already either THO-hosted or out of scope for the
  Subscriptions Backport IG.

Declining as filed (with an invitation to refile concretely) keeps
the workgroup's UTG bandwidth focused on actionable migrations.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of policy compliance and
implementer protection. Disposition A is correct in principle but is
likely to break existing Subscriptions Backport consumers if it is
applied in a single publication step, and the IG has been in the wild
long enough that a hard canonical change is a real risk. Disposition C
is defensible on the procedural ground that the ticket does not name
the artifact(s) — but Highest priority + a TSC-approved policy + a
coordinated multi-specification "Comment 2" sweep tell us the
workgroup's clear direction is to comply, not to push back on
specificity. Staging the change through THO + deprecation honours both
the policy and the implementers, and gives FHIR-I a clean answer to
return to the reporter and to the TSC.

Note for the workgroup: before this disposition is acted on, please
ask the reporter to confirm exactly which Subscriptions Backport
`CodeSystem` / `ValueSet` resources they intend to move. The augury
index has no cross-references identifying them, and the disposition
plan needs that list to file the UTG submissions.
