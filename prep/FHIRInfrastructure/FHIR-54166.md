### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-54166 as a tracking ticket for an editorial cleanup task
against FHIR Core: locate the specific TODO the reporter had in mind,
draft the missing content, and remove the TODO marker. Before any
content change is made, the ticket must first be augmented with the
concrete target — at minimum the resource (or general page), the
section heading, and ideally the source file path or a quoted snippet
of the existing TODO — by asking the reporter (Gino Canessa) to
update the description and populate the Related Artifacts / Related
Pages / Related Sections fields. Once that scope is captured, the
fix is a narrative/editorial change in FHIR Core authoring sources
(the resource intro, formal definition narrative, or page markdown
that contains the TODO), with no behavioral change to the resource
schema, invariants, search parameters, or operations.

#### Justification

The reporter has explicitly asked for the content to be finished and
the TODO removed, so accepting the request honors the original intent.
TODO markers in normative or trial-use spec content are a recognized
quality issue at ballot time, and the FHIR Infrastructure work group
routinely owns this kind of editorial cleanup. The pre-condition of
nailing down the specific TODO is consistent with normal FHIR ticket
hygiene — the work group cannot change spec text without knowing which
text is in scope — and it does not change the disposition itself, only
the order of operations.

---

### Disposition B: Alternative Approach

#### Proposal

Convert FHIR-54166 from a single change request into a tracking
umbrella for *all* outstanding TODO markers in the FHIR Core
specification. Concretely: (1) run a build-time scan over the FHIR
Core authoring sources for `TODO` / `TBD` / `XXX` markers, (2) record
the inventory (file, section, surrounding context) on this ticket as
a comment or attachment, (3) split each remaining TODO into its own
child change request (or sub-task) routed to the appropriate work
group, and (4) close FHIR-54166 once the inventory and child tickets
exist. Optionally, add a publication-tooling check that fails the
build (or emits a QA warning) when a TODO marker survives into a
ballot or release branch, so that this class of ticket does not
re-occur in the same shape.

#### Justification

The ticket's text ("Additional resource contents need to be
completed") and the absence of any specific scope strongly suggest
this is one instance of a general problem rather than a single
isolated TODO. Handling it as an umbrella avoids the work group
absorbing a stream of near-identical, low-context tickets and gives
each real TODO a dedicated, properly-scoped change request that the
right work group can disposition. Adding a tooling check addresses
the root cause — TODOs leaking into published spec content — which
the literal request (delete one TODO) does not. The trade-off vs.
Disposition A is a larger up-front effort, but the ongoing cost is
lower and the result is auditable.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54166 as **Not Enough Information** and close it without
making any spec change. Ask the reporter to file a new, properly
scoped change request that identifies the specific resource, page, or
section containing the TODO and (where possible) quotes the offending
text. If no such follow-up ticket arrives within the work group's
normal stale-ticket window, leave the issue closed.

#### Justification

The description does not identify any resource, page, section, or
file, the Related Artifacts / Related Pages / Related Sections
fields are empty, and there are no comments, linked tickets, Zulip
threads, or GitHub items in the Augury index that pin the request to
a concrete location. Under the work group's normal ticket-quality
expectations, a change request must be actionable on its face;
"finish the incomplete content" is not actionable without the target.
Declining is defensible because no spec text can be confidently
edited from the information present, and accepting the ticket in its
current shape risks an editor making a guess that the reporter did
not intend.

---

### Recommendation

**Recommended disposition:** A

The reporter is the work group's own member and the request is plainly
a real editorial issue (a stray TODO in FHIR Core), so outright
declining (C) is harsher than necessary and would likely just produce
the same ticket again with one extra round-trip. Promoting the ticket
to a global TODO-sweep umbrella (B) is attractive but is a much larger
piece of work than what is in front of the work group today, and
should arguably be its own infrastructure ticket rather than being
hung off this one. Disposition A — accept as an editorial cleanup,
but require the reporter to first identify the specific TODO before
any content change is made — keeps the original intent, keeps the
scope small, and gives the work group a clean record that the issue
was handled. If, while resolving FHIR-54166, the editor finds that
TODOs are widespread, that finding can be escalated into Disposition
B as a follow-on ticket.
