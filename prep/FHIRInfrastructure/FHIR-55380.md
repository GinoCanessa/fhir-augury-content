## Summary

The ticket reports that the FHIR Mapping Language tutorial contains numerous
technical errors in its example code and narrative. Specifically, the reporter
calls out incorrect variable names, FHIRPath expressions that are missing
the surrounding parentheses required by the Mapping Language grammar,
general grammar issues, `translate` parameters that are used without
explanation, references to content (e.g. the literal `null`) that does not
appear in the expression being discussed, inconsistent use of single vs.
double quotes in examples (e.g. `cast(a, "integer")` is cited as wrong
because Mapping Language string literals use single quotes), and missing /
inconsistent semicolons between statements. The ask is editorial: correct
the tutorial so its examples are syntactically valid, its narrative matches
the examples, and its style is consistent. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced for this ticket, so the scope
below is inferred solely from the description.

## Proposed Dispositions

Because no GitHub cross-references were surfaced, none of the dispositions
below can name a specific repo/path from a `repo-analysis` briefing. The
ticket is nonetheless clearly scoped to a single narrative page in
`HL7/fhir` — the FHIR Mapping Language tutorial (the page rendered as
`mapping-tutorial.html`) plus, where its examples reveal real defects, the
FHIR Mapping Language reference page (`mapping-language.html`). The
dispositions therefore speak in those terms; if accepted, the implementer
should confirm the authoring root for those pages against the current
`HL7/fhir` briefing before editing.

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-55380 as an editorial-fix ticket against the FHIR Mapping
Language tutorial. Walk the tutorial top to bottom and correct each class
of defect the reporter lists:

1. **Variable names** — reconcile every variable referenced in narrative
   with what is actually bound in the example's `source`/`target` clauses;
   rename in the narrative (not the examples) unless the example itself is
   wrong.
2. **FHIRPath parentheses** — wherever a `where`, `check`, or similar
   clause embeds a FHIRPath expression, ensure the expression is wrapped
   per the Mapping Language grammar (`where(<fhirpath>)` form). Add the
   parentheses that are missing.
3. **String literals** — change every double-quoted string literal in
   example code to single quotes (e.g. `cast(a, "integer")` →
   `cast(a, 'integer')`). Audit both tutorial and reference pages for
   stray double-quoted literals.
4. **`translate` parameters** — for each use of `translate(...)` in the
   tutorial, add a short inline explanation of its positional parameters
   (source code, concept-map URL, output mode) or a forward link to the
   `translate` section of the Mapping Language reference.
5. **Narrative vs. example mismatch ("null")** — locate the passage that
   refers to a `null` literal not present in the shown expression; either
   adjust the example to include the case being discussed, or rewrite the
   narrative to describe what the example actually does.
6. **Semicolons** — standardise on semicolon-terminated rules/statements
   across every example, fixing omissions so formatting is consistent
   throughout the tutorial and reference.

Land the fixes as a single editorial PR against `HL7/fhir` (tutorial page
plus any co-edited reference page), cross-linked to FHIR-55380, with no
normative/behavioural changes.

#### Justification

Every defect the reporter lists is reproducible from the Mapping Language
grammar: FHIRPath arguments are defined to be parenthesised, Mapping
Language string literals are single-quoted, and rules are semicolon-
terminated. The tutorial is a first-contact document for new implementers
of FML; examples that don't parse are actively misleading and erode trust
in the specification. Because the change is purely editorial and does not
touch the grammar, conformance rules, or any computable artifact, the risk
surface is small and the benefit is concentrated on exactly the audience
the page targets. Acceptance also honours the reporter's observation that
the inconsistencies are scattered ("Limitted places use the double
quotes") — a comprehensive pass is more useful than patching a single
example.

### Disposition B: Alternative Approach

#### Proposal

Accept the editorial fixes (as in A) **and additionally** add a build-time
syntactic check: round-trip every fenced Mapping Language example in the
tutorial and reference pages through the publication's FML parser (the
same one used to validate committed `.map` / StructureMap assets). Any
example that fails to parse fails the build. Treat the prose "null" /
variable-name inconsistencies noted in FHIR-55380 as content the automated
check *cannot* catch, and handle those manually; but let the automation
own the parenthesisation, quoting, and semicolon issues going forward so
that regressions cannot silently reappear.

#### Justification

The root cause the reporter is surfacing isn't any one typo — it's that
tutorial examples are authored as free text and therefore drift away from
the grammar whenever the grammar evolves or new contributors edit the
page. A one-time editorial pass (Disposition A) fixes today's state but
does not prevent tomorrow's regression. Parsing each example at build
time makes the tutorial self-policing for precisely the class of defects
FHIR-55380 enumerates. The trade-off is up-front tooling work — wiring a
parser invocation into the publication pipeline and annotating which
fenced blocks are FML vs. illustrative pseudo-code — which is why this is
offered as an alternative rather than the default. FHIR Infrastructure
already owns the FML parser, so the tooling sits within the group's
remit.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-55380 as filed on the grounds that it is not actionable as a
single change request: it enumerates a category of defects ("many
technical errors") rather than identifying the specific examples,
sections, and line-level changes needed. Close with a request that the
reporter re-file one ticket per concrete defect (or supply a patch /
pull-request against `HL7/fhir`) so each fix can be reviewed and
dispositioned on its own merits.

#### Justification

The ticket's body is a bullet list of symptom classes without pointers to
the offending examples. Acting on it blind risks either missing instances
the reporter had in mind or rewriting passages that the reporter
considered correct, and either outcome generates follow-up tickets. FHIR
Infrastructure's standing practice for editorial cleanups is to accept
targeted tickets (or PRs) rather than open-ended "sweep the page"
requests, because targeted items are auditable at disposition time. This
path is defensible but undesirable: the reporter is a domain expert whose
observations are almost certainly accurate, and forcing a refile imposes
friction on exactly the contributor the workgroup most wants to keep
engaged.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right answer because every defect class the reporter
names is grammar-level and therefore falsifiable from the Mapping Language
specification itself — an editor does not need further clarification from
the reporter to know that `cast(a, "integer")` must become
`cast(a, 'integer')`, that FHIRPath arguments must be parenthesised, and
that rules must end in semicolons. A single editorial PR against the
`HL7/fhir` Mapping Language tutorial (and the reference page where the
same patterns recur) closes the ticket cleanly and improves the first-
contact experience for FML implementers. Disposition B is attractive and
should be filed as a follow-up tooling ticket, but coupling it to this
editorial fix risks stalling a low-risk cleanup on a larger
infrastructure change. Disposition C understates the information value of
the reporter's classification — the categories he lists are precise
enough to drive a full editorial pass without a refile.
