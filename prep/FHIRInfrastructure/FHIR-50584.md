## Summary

The reporter observes that the "comments" paragraph shown on the root of
the `SampledData` datatype in the published FHIR specification
(datatypes-definitions.html#SampledData) is out-of-date relative to the
current definition of the type. Specifically, the paragraph:

1. States that `origin` and `dimensions` are optional, but both are now
   required elements on `SampledData`.
2. References a `period` element that has been removed from
   `SampledData` (replaced by the current interval representation).
3. References example datatypes/resources called "DeviceCapabilities"
   and "DeviceLog" which do not exist in the current specification.

The reporter confirms this outdated text appears both in the
6.0.0-ballot3 build and in the continuous-integration build
(build.fhir.org), so it is not specific to one branch. The single
cross-reference on the ticket is to `Observation.value[x]`, which is
referenced inside the quoted comment text itself (as an example of a
summary-view rendering of `SampledData`) rather than a separate related
change. No related Jira tickets, Zulip threads, or GitHub items were
surfaced by the search indexes for this ticket. The request is a
targeted editorial/technical correction to the `SampledData` root
comment to bring it back in line with the type's current structure and
the rest of the specification.

## Details

**Description:**

The "comments" paragraph on the root of SampledData is out-of-date.

It claims required fields (origin and dimensions) are optional,
references the now-removed period field, and references
"DeviceCapabilities" and "DeviceLog".

Current text:

> The data is not interpretable without at least origin, period, and
> dimensions, but these are optional to allow a separation between the
> template of measurement and the actual measurement, such as between
> DeviceCapabilities and DeviceLog. When providing a summary view (for
> example with Observation.value[x]) SampledData should be represented
> with a brief display text such as "Sampled Data".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the current root `SampledData` "comments" paragraph so that it
(a) describes `origin` and `dimensions` as required (matching the
current element cardinalities), (b) removes all mention of the
now-removed `period` element, (c) drops the example pair
"DeviceCapabilities / DeviceLog" (which are not real FHIR artifacts),
and (d) retains the still-valid guidance that when `SampledData`
appears in a summary view (e.g. in `Observation.value[x]`) it should be
rendered with a brief display such as "Sampled Data".

Suggested replacement text (illustrative, to be finalized by the work
group):

> SampledData carries a series of discrete measurements as a compact
> digital representation. The `origin`, `interval` / `intervalUnit`,
> and `dimensions` elements are required because the `data` string is
> not interpretable without them. When `SampledData` appears inside a
> summary-level element such as `Observation.value[x]`, it should be
> represented with a brief display text such as "Sampled Data".

The change is narrative only, limited to the comments paragraph of the
`SampledData` datatype definition in the core FHIR specification
(rendered on `datatypes-definitions.html#SampledData`). No schema,
StructureDefinition, or invariant changes are required.

#### Justification

This is a straightforward technical correction: the published text
materially misrepresents the current type by describing required
elements as optional, naming an element (`period`) that no longer
exists, and citing artifacts (`DeviceCapabilities`, `DeviceLog`) that
are not in the specification. Leaving the text in place risks
implementers coding to a nonexistent contract. The fix is editorial,
low-risk, does not alter conformance, and is consistent with how other
datatype comments describe required elements. The commenter has also
confirmed the defect is present in both the ballot branch and the CI
build, so a single corrective edit will flow through normally.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the correction, but take the opportunity to split the root
"comments" paragraph into two clearly-scoped parts:

1. A **definitional** sentence that states the required elements
   (`origin`, `interval`/`intervalUnit`, `dimensions`) and explains why
   the `data` string is uninterpretable without them.
2. A **rendering** sentence that preserves the existing "summary view"
   guidance (using `Observation.value[x]` as the example and the
   suggested short display text "Sampled Data").

At the same time, add a short cross-link from this paragraph to the
element-level descriptions of `origin`, `interval`, `intervalUnit`, and
`dimensions`, so readers landing on the datatype overview can jump to
the authoritative element detail rather than relying on the comment to
describe cardinality.

#### Justification

Disposition A fixes what is wrong; Disposition B additionally addresses
a structural weakness that contributed to the defect in the first
place — namely that a single prose paragraph was being used both to
describe required-element semantics and to give rendering guidance.
Splitting the two concerns makes the required-element claim easier to
keep in sync with the StructureDefinition (and easier to spot when a
required element is renamed or removed, as happened with `period`), and
the cross-links reduce the risk of the narrative drifting away from the
element table again. The cost is modest — a small amount of extra
editing over Disposition A, with no behavioural or conformance change.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "not persuasive" / decline to change, on the basis
that the affected text is non-normative narrative and that the
authoritative source for element cardinality and the list of defined
elements is the `SampledData` element table (and underlying
StructureDefinition), which is already correct.

#### Justification

The root "comments" paragraph is advisory narrative; readers who need
authoritative cardinality consult the element table, which correctly
shows `origin` and `dimensions` as required and does not list `period`.
There is no defect in the type definition or in any normative artifact,
and no implementation has been shown to be harmed by the stale
paragraph.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the minimum-viable fix and is clearly correct — there
is no defensible case for declining (Disposition C), because the
paragraph actively contradicts the element table by describing required
elements as optional and by naming a removed element (`period`) and two
non-existent artifacts (`DeviceCapabilities`, `DeviceLog`). However,
the defect itself — prose drifting out of step with the
StructureDefinition — is exactly the kind of problem that tends to
recur when required-element claims live inside a free-form comments
paragraph. Disposition B costs very little extra effort (a second
sentence and a few cross-links) and leaves the narrative in a shape
that is much easier to audit against the element table going forward.
Given that this is already flagged as a Technical Correction in FHIR
Infrastructure and that the ticket explicitly enumerates three separate
stale claims in one paragraph, B is the more robust landing. If the
work group wants to keep the change footprint as small as possible for
the ballot cycle, falling back to Disposition A is entirely reasonable.
