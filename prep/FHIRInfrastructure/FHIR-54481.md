## Summary

Brian Postlethwaite reports that the FHIR specification does not contain
an official, normative mapping (parser/grammar binding) from the FHIR
Mapping Language (FML) text format to the `StructureMap` resource — the
only complete implementation of that translation lives inside HAPI's
reference code. As a consequence, implementers wishing to consume `.fml`
or `.map` files have no spec-level definition that tells them how each
FML construct must be projected into the corresponding `StructureMap`
elements (groups, rules, sources, targets, dependent rules, defaults,
etc.); they must reverse-engineer HAPI's behaviour. This ticket is one
of a tightly-coupled tranche filed by the same reporter on
2026-01-16 against FHIR Core / FHIR Infrastructure as part of his
review of FML's normative readiness, all surfaced via shared-keyword
similarity rather than explicit cross-references:
[FHIR-54473](https://jira.hl7.org/browse/FHIR-54473) (umbrella: "FHIR
Mapping Language not ready for normative" — proposes moving FML out of
core into an incubator/separate spec while keeping StructureMap
normative; WGM Q4 on 2026-01-28 reached general consensus that FML
needs more time),
[FHIR-54480](https://jira.hl7.org/browse/FHIR-54480) (FML grammar
doesn't support the super-simple form),
[FHIR-54482](https://jira.hl7.org/browse/FHIR-54482) (default value
typing regression vs. R4),
[FHIR-54483](https://jira.hl7.org/browse/FHIR-54483) (error reporting
location ambiguity between FML and StructureMap),
[FHIR-54476](https://jira.hl7.org/browse/FHIR-54476)/[FHIR-54477](https://jira.hl7.org/browse/FHIR-54477)
(round-trip / chaining gaps), and
[FHIR-46284](https://jira.hl7.org/browse/FHIR-46284) (FML grammar for
target rules inconsistent with the spec). Together these argue that the
FML ↔ StructureMap projection is currently de facto specified by HAPI,
which is a normative blocker.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a normative-publishing gap and add an authoritative
**FML grammar specification** plus an **FML → StructureMap projection
table** to the FHIR Core specification, located alongside
`mapping-language.html` / `structuremap.html`. Concretely:

1. Publish the FML grammar in a normative form (ABNF or EBNF) on a
   `mapping-language-grammar.html` page, frozen against the same
   ballot cycle as `StructureMap`.
2. For every production in that grammar, define which `StructureMap`
   element(s) it projects to (e.g., `group` → `StructureMap.group`,
   `source/target` → `StructureMap.group.rule.{source,target}`,
   `dependent` → `StructureMap.group.rule.dependent`, default value
   typing per [FHIR-54482](https://jira.hl7.org/browse/FHIR-54482),
   error-location convention per
   [FHIR-54483](https://jira.hl7.org/browse/FHIR-54483)).
3. Require the projection to be loss-less in both directions, and
   carry a conformance statement that any FML↔StructureMap converter
   MUST round-trip the canonical examples in the spec.
4. Donate the relevant grammar/projection logic out of HAPI (`StructureMapUtilities`) into an HL7-owned reference implementation
   that is treated as informative, not normative.

#### Justification

The reporter's underlying observation is correct and well-evidenced by
the related tickets (54473, 54476, 54480, 54482, 54483, 46284): there
is no FHIR-spec text that tells an implementer how to translate `.fml`
into a `StructureMap` resource, so HAPI is the de facto standard. For
StructureMap to remain credibly normative — which both Brian
Postlethwaite (FHIR-54473) and Grahame Grieve (in the Zulip thread)
agree it should — the textual surface developers actually author in
must have a normative grammar and a normative projection. Without
this, every new FML feature creates the kind of silent divergence the
sibling tickets enumerate.

---

### Disposition B: Alternative Approach

#### Proposal

Don't add the FML↔StructureMap mapping to FHIR Core. Instead:

1. Move the FML language definition (grammar + projection to
   `StructureMap`) into a **separate, externally-balloted specification**
   — the path proposed in [FHIR-54473](https://jira.hl7.org/browse/FHIR-54473)
   and echoed by Lloyd McKenzie on the "Move FML to a new spec?" Zulip
   thread, paralleling FHIRPath and CQL.
2. Keep `StructureMap` normative in FHIR Core as the wire/exchange
   form, but make it spec-policy that it is the *target* artefact, not
   the authoring surface.
3. The FML spec then owns the grammar, the projection table, the
   round-trip conformance suite, and a versioning policy independent
   of FHIR Core releases.
4. Re-target this ticket and the sibling FML tickets (54476, 54477,
   54480, 54482, 54483) at the new FML spec's tracker.

#### Justification

The WGM Q4 minutes on 54473 explicitly note timing/attention concerns
relative to the rest of R6, and the Zulip thread shows senior
contributors (Grahame, Lloyd) actively considering exactly this split.
Keeping FML in core forces every grammar fix or projection
clarification through the FHIR Core ballot cycle, which is heavyweight
and slow — hence the years-long backlog of FML tickets. An external
spec lets FML iterate at the cadence its maturity actually warrants
without holding StructureMap (which is much closer to ready) hostage.
The trade-off versus Disposition A is that R6 ships without a
normative FML grammar in core; that is a real cost, but the WGM
already accepted it implicitly when it agreed FML is not ready for
normative.

---

### Disposition C: Decline

#### Proposal

Decline as **out of scope for this ticket as filed**: close FHIR-54481
in favour of [FHIR-54473](https://jira.hl7.org/browse/FHIR-54473),
which is the umbrella ticket where the FML normative-readiness
question is being decided. The specific technical sub-issues (grammar
gaps, default typing, error location, chaining, round-trip) are
already individually tracked by the sibling tickets and don't need a
separate "no official mapping exists" ticket on top.

#### Justification

This ticket overlaps almost entirely with FHIR-54473 and its tranche.
Resolving 54473 (whichever way the WG decides — keep FML in core, move
to incubator, or external spec) inherently resolves the question of
where the official FML→StructureMap mapping lives. Maintaining 54481
as a separate item risks a partial fix (e.g., a grammar appendix)
that doesn't cohere with the umbrella decision. Declining is
defensible only as bookkeeping; it is not defensible as a substantive
"no, we will continue to have no official mapping" answer, which is
why this is the weakest of the three options.

---

### Recommendation

**Recommended disposition:** B

The combination of the WGM Q4 consensus on FHIR-54473 ("promising but
not quite finished/ready for normative"), the active Zulip thread
where Grahame Grieve and Lloyd McKenzie are converging on moving FML
out of core, and the sheer size of the sibling backlog (FHIR-54476,
54477, 54480, 54482, 54483, 46284, plus 42995 and 54779) all point at
the same conclusion: FML's grammar and its projection to
`StructureMap` need to be specified somewhere normative, but FHIR Core
R6 is the wrong place and the wrong cadence to do it. Disposition B
gives the request what it actually asks for — an *official* mapping —
while honouring the WG's judgement that FML isn't core-normative-ready
and freeing the FML team to publish the grammar/projection on its own
schedule. Disposition A is the right answer in a world where FML was
already mature; it is not. Disposition C is fine as housekeeping but
should not be used to dismiss the underlying defect, which is real.
