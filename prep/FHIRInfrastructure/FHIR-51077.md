## Summary

FHIR-51077 is an editorial/clarity change request against the search-prefix
page of the FHIR Core specification. Specifically, it targets the "Three
ranges are identified" table on `search.html` that defines the three kinds
of range ("range of the value", "range below the value", "range above the
value") for Decimal and Date/Time values. The reporter (Elliot Silver)
identifies four distinct defects in that table:

1. The phrase *"any value less or equal to &lt;2.00000…"* is self-contradictory.
   Either `<` is a stray character that should be removed (but removing it
   conflicts with the row label "range **below** the value"), or the "less
   or equal to *something less than x*" reading reduces to simply "less
   than *x*".
2. *"any value greater or equal to &lt;2.00000000000000000000"* has the
   same problem in the "range above" row for decimals.
3. The Decimal Values grouping's "range above the value" cell mixes a
   date-time example into the numeric group (`2015-08-12T05:23:45 …`),
   which belongs in the Date/Time Values section below.
4. The Date/Time "range of the value" cell says *"The date 2015-08 has a
   range from 2015-08-12T00:00:00.0000…"* — the day component `-12` is
   clearly a copy/paste error and should be `2015-08-01T00:00:00.0000`
   (the first instant of August 2015).

The ticket is the editorial complement to the semantic discussion that is
being worked in the linked Zulip thread and in companion tickets
FHIR-51068, FHIR-40319, and FHIR-40304 (referenced only conversationally
in Zulip; none are currently linked as Jira cross-references). Those
sibling tickets concern the *meaning* of `lt` / `le` / `eq` on periods,
while this ticket is strictly about cleaning up the **existing** table so
that readers can parse what the spec currently says without
contradictions or miscategorised examples.

## Details

**Description:**

There are several issues with the following table.

- "any value less or equal to <2.00000..." - what does "less or equal to
  < *x*" mean? Do we mean a value which is less or equal to something
  which is less than *x* (which would be the same as less than *x*)? Or,
  is the < symbol an error which should be removed (which would then
  conflict with the first column of this row which indicates "range
  **below** the value".
- "any value greater or equal to <2.00000000000000000000" is similarly
  confusing.
- The first grouping of ranges is for numeric values, but the "range
  above the value" contains a date time example.
- The "range of the value" entry for dates includes "The date 2015-08
  has a range from 2015-08-12T00:00:00.0000..." I believe this should be
  "...a range from 2015-08-01...".

Quoted from the spec table (Decimal Values / Date-Time Values):

> Three ranges are identified:
>
> **Decimal Values**
> - range of the value — The limits implied by the precision of the value — The number 2.0 has a range of 1.95 to 2.05
> - range below the value — Up to the specified value — The range below 2.0 includes any value less or equal to &lt;2.00000000000000000000
> - range above the value — The specified value and up — The range above 2.0 includes any value greater or equal to &lt;2.00000000000000000000; The range after 2015-08-12T05:23:45 includes any time after 2015-08-12T05:23:45.000000000000000
>
> **Date / Time Values**
> - range of the value — The limits implied by the precision of the value — The date 2015-08-12 has a range from 2015-08-12T00:00:00.0000 inclusive to 2015-08-13T00:00:00.0000 exclusive; The date 2015-08 has a range from 2015-08-12T00:00:00.0000 inclusive to 2015-09-01T00:00:00.0000 exclusive
> - range below the value — Up to the specified value — The range before 2015-08-12T05:23:45 includes any time up to 2015-08-12T05:23:45.000000000000000; The range before 2015-08 includes any time up to 2015-08-00T00:00:00.000000000000000
> - range above the value — The specified value and up — The range after 2015-08-12T05:23:45 includes any time from 2015-08-12T05:23:46.000000000000000 and later; The range after 2015-08 includes any time from 2015-09-01T00:00:00.000000000000000 and later
>
> Additional details about how ranges are applied are specific to each
> data type and are documented there.

## Related Zulip Discussions

### [implementers > DateTime search with Period value](https://chat.fhir.org/#narrow/stream/implementers/topic/DateTime%20search%20with%20Period%20value/near/523610955)

The thread is a long-running (June–September 2025) debate about how
range-based prefixes — especially `le`, `lt`, and `eq` — should behave
when the target element is itself a range (e.g., a `Period`). Yunwei
Wang reports that hapi.fhir.org returns a `Procedure` whose
`performedPeriod = [2015-04-13T20:27:01, 2015-04-13T20:42:01)` for the
query `date=le2015-04-13T20:27:01-04:00`, even though neither the `lt`
nor the `eq` query by themselves return it. Daniel Venton, Stephen
MacVicar, Craig McClendon and Elliot Silver debate whether HAPI's
behaviour matches the spec's wording (*"the range below the search value
intersects… or the range of the search value fully contains the range of
the resource value"*); the consensus is that the current spec text is
ambiguous and under-specified. Gino Canessa (FHIR-I) joins and proposes
that the ambiguity is really about **ordering of logical composition**
(compose `lt ∪ eq` ranges first and intersect, vs. test `lt` and `eq`
separately and OR the results) and agrees the spec must be clarified.
FHIR-51068 was filed for the semantic question, FHIR-51077 was filed
for the editorial/table defects (explicitly "a simple one to throw onto
that pile"), and FHIR-40319 / FHIR-40304 are cited as possibly
overlapping tickets. A follow-up discussion in September 2025 discusses
consistency between `le` on dates vs. `le` on numbers and whether R6
should standardise around a single-range-intersection interpretation.
No final disposition is reached in the thread; the plan is to settle
the semantic question before or during the WGM and then dovetail the
editorial cleanup.

Daniel Venton, Keith Boone, Stephen MacVicar, Craig McClendon, Gino

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Fix the four defects in the "Three ranges are identified" table on
`search.html` (the FHIR Core search page, served by `build.fhir.org/search.html#prefix`
and pinned versions thereof) without changing the semantics of any
prefix. Concretely:

1. In the **Decimal Values → range below the value** row, change the
   example cell from *"The range below 2.0 includes any value less or
   equal to &lt;2.00000000000000000000"* to *"The range below 2.0
   includes any value less than 2.00000000000000000000"* (i.e., drop
   the redundant "or equal to" that is contradicted by the row label
   "**below** the value" and the stray `<`).
2. In the **Decimal Values → range above the value** row, change the
   example cell from *"… any value greater or equal to
   &lt;2.00000000000000000000"* to *"… any value greater than or equal
   to 2.00000000000000000000"* (the "range above the value" is "The
   specified value and up", so `>=` is correct; drop only the `<`).
3. Move the **date/time example** (*"The range after
   2015-08-12T05:23:45 includes any time after
   2015-08-12T05:23:45.000000000000000"*) out of the Decimal Values
   "range above the value" cell and into the Date/Time Values "range
   above the value" cell, where it belongs.
4. In the **Date/Time Values → range of the value** row, fix the
   copy-paste error *"The date 2015-08 has a range from
   2015-08-12T00:00:00.0000 inclusive…"* to *"The date 2015-08 has a
   range from 2015-08-01T00:00:00.0000 inclusive to
   2015-09-01T00:00:00.0000 exclusive"*. While fixing that, also
   correct the related *"range below the value"* example that uses
   `2015-08-00T00:00:00…` (`-00` is not a valid day) to
   `2015-08-01T00:00:00…` exclusive, so that the two partial-date
   examples are internally consistent.

No conformance language changes; all four edits are pure editorial
corrections.

#### Justification

All four issues are plainly wrong on their face — the `<` characters
are typographical artifacts, the day `-12` in *"The date 2015-08 has a
range from 2015-08-12"* is self-evidently wrong (August 2015 does not
start on the 12th), and a date/time example in the Decimal Values
section is miscategorised. These are the kind of defects that trip
first-time readers and provide ammunition for divergent implementation
choices like the ones debated in the companion Zulip thread. Accepting
as requested keeps the current semantics intact (which is important
because search is normative and implementations rely on it) while
removing a standing source of reader confusion. The reporter framed
this explicitly as "a simple one to throw onto that pile" — i.e., the
editorial layer of a larger semantic discussion — and the workgroup
has no reason to hold it up while FHIR-51068 / FHIR-40319 / FHIR-40304
are resolved.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the editorial fixes from Disposition A, and additionally rewrite
the "Three ranges are identified" table to use explicit **interval
notation** (e.g., `[a, b)`, `(−∞, a)`, `[a, +∞)`) rather than the
current narrative "less or equal to … up to …" prose. Specifically:

- Replace each example cell with a pair (half-open interval + prose),
  e.g. *"range of 2.0: `[1.95, 2.05)` — the limits implied by the
  precision of the value"*.
- Likewise for date/time: *"range of `2015-08`: `[2015-08-01T00:00:00,
  2015-09-01T00:00:00)`"*.
- Keep the "Additional details about how ranges are applied are
  specific to each data type and are documented there." sentence and
  add a cross-link to the prefix definitions (`eq` / `lt` / `le` /
  `gt` / `ge` / `sa` / `eb` / `ap`) so readers who need to reason
  about boundary behavior can follow through.

This is still scoped to the table, but it pre-emptively removes an
entire class of boundary-ambiguity bugs (the kind that drives
FHIR-51068) at the cost of a more invasive diff.

#### Justification

Interval notation is unambiguous about inclusive vs. exclusive
boundaries, which is exactly the axis on which the Zulip thread keeps
running aground (is `le2015-04-13T20:27:01` `[−∞, 20:27:01]` or `[−∞,
20:27:02)`? does `range below` include the boundary or not?). The
current prose hides that choice behind "up to the specified value" /
"and up", which different implementers read differently. The drawback
is that interval notation is denser and may feel less friendly to
business analysts scanning the page; however, the accompanying prose
is preserved, and the examples already contain explicit "inclusive" /
"exclusive" labels in some rows, so the register shift is small.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Mod — subsumed by FHIR-51068"
without independently editing the table. The rationale: the workgroup
intends to revisit the entire prefix-semantics section as part of
resolving FHIR-51068 (plus FHIR-40319 / FHIR-40304) before R6 goes
normative; any table edits made now will almost certainly be rewritten
by that larger change. Mark this ticket as related to FHIR-51068 and
track the four defects as part of that resolution's editorial
follow-up.

#### Justification

The table sits inside a section that is actively being rethought for
R6 normative behavior. Making a focused editorial fix now risks
generating churn: the same cells will be rewritten again when
FHIR-51068 lands, and reviewers of that larger change would have to
reconcile two distinct diffs against the same rows. Deferring keeps
the editorial effort paired with the semantic decision and ensures the
final table reflects one coherent pass.

---

### Recommendation

**Recommended disposition:** A

The four defects the ticket cites are uncontroversial — they are
typos, miscategorisation, and a copy/paste day error, none of which
depend on how FHIR-51068 / FHIR-40319 / FHIR-40304 are eventually
resolved. A reader today cannot parse the current cells without
guessing what was intended. Disposition C's "wait for FHIR-51068"
argument is real but weak: the semantic ticket is unlikely to delete
this table, only to refine the prefix definitions that reference it,
so the risk of rework is low. Disposition B's interval-notation pass
is attractive and would genuinely help the downstream semantic
discussion, but it crosses into the territory that FHIR-I explicitly
wants to settle in FHIR-51068 before rewriting — adopting interval
notation here would pre-commit to a particular reading of inclusive
vs. exclusive boundaries ahead of that decision. Disposition A is the
minimal, low-risk fix that makes the page readable now and leaves
every downstream option open. It also honours the reporter's framing
of this ticket as the "simple one on the pile", freeing work-group
cycles to focus the semantic debate on FHIR-51068 proper.
