## Summary

The ticket reports an interaction between the SMART App State scopes and the
generic SMART on FHIR scope semantics for the `Basic` resource. Because the
SMART App State feature stores app-state as `Basic` resources, a
`patient/Basic.rs?code=...` scope (which under the Patient Compartment also
matches `Basic` resources whose `author` references the patient) ends up
authorising more than the App State use case strictly intended. The reporter
asks the WG to confirm whether this overlap is intentional, and — if it is
not — to introduce App-State-specific scope semantics (e.g., scoping by
profile canonical rather than by resource type + code).

The companion ticket [FHIR-39699](https://jira.hl7.org/browse/FHIR-39699)
(also raised by Bas van den Heuvel and explicitly cross-referenced from
FHIR-39701's own comments) makes the analogous point on the practitioner /
user side: the App State `user/Basic.cuds?code=...` scope does not actually
force any link to the current Practitioner, so it overlaps with global app
scopes. Together the two tickets argue that App State has bolted onto the
generic Basic-scope mechanism without giving implementers a clean way to
isolate App-State Basics from "real" Basic resources or to bind scopes to
the appropriate compartment principal.

The workgroup's prior (subsequently reverted) resolution acknowledged the
overlap but argued the App State spec deliberately leverages existing
`patient/Basic.rs?code=...` semantics rather than defining new ones, and
that EHRs are free to apply additional restrictions. After STU 2.1
publication the ticket was re-opened for further consideration.

## Details

**Description:**

> When interpreting the SMART app state scopes according to the SMART on
> FHIR guidance, a Basic resource using the author field to refer to a
> patient also complies to the patient/code=... scope.
>
> Is this intended, if not, the scopes for appstate behave differently from
> other SMART scopes. This might cause issues when other Basic resources
> are used as well.
>
> If the behavior is specific for appstate, we should consider to create
> appstate specific rules (e.g. use the profile cannonical in the scope).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Amend the SMART App Launch IG (App State page,
`http://hl7.org/fhir/smart-app-launch/app-state.html`) to define
App-State-specific scope semantics that disambiguate App-State Basics from
other Basic resources. Concretely:

1. Define new scope token forms anchored to the App-State profile
   canonical (e.g.,
   `patient/Basic.rs?_profile=http://hl7.org/fhir/smart-app-launch/StructureDefinition/smart-app-state-basic`)
   as the **normative** way to request App-State access; deprecate plain
   `patient/Basic.rs?code=...` for App-State purposes.
2. Require servers to refuse access to non-App-State `Basic` resources
   under these App-State-specific scopes, and conversely refuse access to
   App-State Basics under generic `Basic` scopes that do not mention the
   profile.
3. Add normative text to the App State page clarifying that
   `Basic.author = Patient/x` does **not** by itself bring a resource
   into the patient's compartment for App-State scope evaluation —
   matching is by `Basic.subject` (or by App-State profile membership).
4. Mirror the change for the user-side scopes (per the companion
   FHIR-39699), so the two tickets are reconciled together.

#### Justification

This is the only approach that fully eliminates the overlap the reporter
identifies: today a generic `patient/Basic.rs?code=...` scope granted to a
non-App-State app can incidentally read App-State Basics whenever the
server uses Patient-Compartment semantics, and an App-State app holding
that scope can read non-App-State Basics that happen to share the code.
Anchoring App-State scopes on the profile canonical is consistent with
FHIR's existing use of `_profile` for resource discrimination and removes
the ambiguity at the protocol layer rather than relying on per-EHR
hardening. It also positions the spec to reuse the same mechanism the
moment additional Basic-based features are layered on top of SMART App
Launch.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing `patient/` and `user/` scope syntax for App State (no
breaking change) but add **non-normative implementation guidance** to the
App State page describing how servers SHOULD constrain authorization to
avoid the overlap. Specifically:

1. Add a "Scope evaluation guidance" subsection recommending that servers
   evaluating App-State scopes match on `Basic.subject` only, not on
   Patient-Compartment membership via `Basic.author`.
2. Document the advanced-scope pattern offered in the workgroup's
   reverted resolution to FHIR-39699 — e.g.,
   `user/Basic.rs?subject:missing=false&subject=Practitioner/123&code=...`
   — as the supported way for clients/servers that need stricter
   isolation today.
3. Recommend (SHOULD) that servers either store App-State Basics in a
   dedicated logical partition or distinguish them by `meta.profile` so
   App-State scopes can be enforced without changing the scope syntax.
4. Defer any new scope-prefix or profile-anchored syntax to a future
   major SMART App Launch revision (tracked alongside FHIR-39699 and
   FHIR-40015).

#### Justification

This matches the position the WG articulated in the (reverted) 2023-02-13
resolution and in the SMART App Launch 2.1 ballot-reconciliation thread:
the spec deliberately leans on existing FHIR scope semantics rather than
introducing new ones, and any change to scope syntax is a breaking change
for already-deployed SMART 2.x clients and servers. Documenting the
hardening pattern keeps the door open for stricter isolation today
without invalidating implementations that already shipped against the
STU 2.1 wording. It also keeps FHIR-39701 and FHIR-39699 aligned with
FHIR-40015's resolution path so the three tickets do not produce
contradictory guidance.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** (or restore the prior **Considered for
Future Use** disposition) on the grounds that the behaviour described is
the intended consequence of layering App State on top of the Basic
resource and standard SMART scopes. Note in the resolution that the
optional App-State scopes are not normative for App-State implementation
(servers may implement App State without exposing the SMART scopes at
all), and that existing FHIR mechanisms (`_profile`, advanced scope
parameters, server-side authorization policy) are sufficient for any
implementer that needs stricter isolation.

#### Justification

The spec already calls the App-State scopes optional, and the overlap
with generic Basic scopes is a direct consequence of the design choice
— made deliberately and documented in the SMART App State Zulip thread
— to use core `Basic` CRUDs rather than a bespoke endpoint. Introducing
profile-anchored or new-prefix scopes would be a breaking change for
SMART 2.1 implementers, and the same protective behaviour can be
achieved today via server policy. The WG previously voted 13-0-0 in
favour of this position, and re-opening the ticket post-publication does
not change the underlying design rationale.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible starting point for the workgroup.
The overlap the reporter describes is real and worth acknowledging in the
spec — declining outright (C) leaves implementers without guidance and
ignores the fact that the WG itself reverted the original "future use"
resolution, signalling appetite to revisit. At the same time, fully
accepting (A) introduces a breaking change to scope syntax for a feature
that just balloted as STU 2.1, which the ballot-reconciliation thread
explicitly tried to avoid for 2.1. Disposition B captures the workgroup's
existing position (existing FHIR machinery is sufficient) while turning
the previously-implicit hardening guidance into normative-flavoured
"SHOULD" text on the App State page, and it keeps FHIR-39701 aligned with
its sibling FHIR-39699 and the related FHIR-40015 so the three tickets
can be reconciled in one coherent change. If the WG later decides
profile-anchored scopes are required (Disposition A), that work is
naturally scoped to a future major SMART App Launch revision.
