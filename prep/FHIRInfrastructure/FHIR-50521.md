## Summary

The reporter has identified that the FHIRPath invariants `tst-1`, `tst-2`, and `tst-3`
on the `TestScript` resource do not match their natural-language descriptions. The
descriptions for `tst-1` and `tst-2` state that only one of three sibling fields may
be present, but the implementation uses chained `xor`, which evaluates to `true` when
all three operands are `true` (`TRUE xor TRUE xor TRUE = TRUE`) — so the constraint
incorrectly passes when all three fields are populated. `tst-3` is described the same
way ("only 1 of its 3 fields") but its expression only verifies that at least one of
the three is absent, which is a much weaker check. The ticket asks the FHIR
Infrastructure work group to correct these invariant expressions so they actually
enforce a strict "exactly one of N" / "at most one of N" semantic that matches the
documented intent. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items associated with this ticket in the indexed sources.

## Details

**Description:**

The expressions for the TestScript constraints `tst-1`, `tst-2`, and `tst-3` do not
seem to match their descriptions.

`tst-1` and `tst-2` claim that only 1 of its 3 fields can be present. But the
expression uses `xor` such that the expression will be true if all 3 fields are
present (`TRUE xor TRUE xor TRUE = TRUE`).

Similarly, `tst-3` claims that only 1 of its 3 fields can be present, but it is
actually only checking if at least 1 is absent.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Correct the FHIRPath expressions of the three `TestScript` invariants so each one
implements a true "exactly one of these three fields is present" check, matching
the existing human-readable descriptions:

- For `tst-1` and `tst-2` (whichever sibling sets these constrain — typically the
  three mutually-exclusive script/source/ref-style choices on a setup/test
  action), replace the chained `xor` with an idiomatic FHIRPath "exactly one"
  expression such as:

  ```fhirpath
  (a.exists().toInteger() + b.exists().toInteger() + c.exists().toInteger()) = 1
  ```

  or equivalently:

  ```fhirpath
  (a | b | c).count() = 1
  ```

  if the three are siblings of compatible types. Pick the form already used
  elsewhere in the spec for consistency.

- For `tst-3`, which is described as "only 1 of its 3 fields can be present" but
  currently only checks that at least one is absent, replace the expression with
  the same "exactly one" formulation as above (or "at most one" — `count() <= 1`
  — if review of the original intent shows `at-most-one` was meant). The
  description should also be re-checked and tightened to remove the ambiguity
  between "exactly one" and "at most one".

The change is purely a constraint-expression fix; no element definitions,
cardinalities, or resource shape changes are required. Bump the maturity-neutral
"technical correction" notes for `TestScript` and regenerate the constraint
table in the spec build.

#### Justification

The invariants exist precisely to enforce the documented "only one of" semantics.
As written, `tst-1`/`tst-2` silently accept clearly-invalid instances (all three
populated), and `tst-3` accepts instances with two of three populated. That
defeats the purpose of having the invariants at all and lets non-conformant
content through validators that rely on FHIRPath enforcement. Since this is a
technical correction to align expression with description, it carries minimal
risk: the only instances it newly rejects are ones the spec already considered
non-conformant. This matches HL7's standing approach to invariant bugs in
released specifications.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same fix as Disposition A, but additionally:

1. Re-examine all `TestScript` invariants (not only `tst-1`/`tst-2`/`tst-3`) for
   the same `xor`-as-mutual-exclusion anti-pattern, since the bug suggests a
   copy-paste pattern that may exist elsewhere on the resource (and possibly on
   other Infrastructure resources authored at the same time).
2. Add a dedicated FHIRPath helper / documented idiom in the FHIRPath
   specification or the "Using FHIRPath" page describing the correct way to
   express "exactly one of N" and "at most one of N", so future authors don't
   repeat the mistake.
3. Add a build-time lint (in the publication tooling) that flags chained `xor`
   over more than two boolean operands as a likely "exactly one" anti-pattern.

#### Justification

The reported defect is almost certainly a category bug, not a one-off: chained
`xor` is a common mis-implementation of "exactly one" by anyone who hasn't
worked through the truth table. Fixing only the three reported invariants leaves
the underlying authoring trap in place. A documented idiom plus a publication
lint is cheap and prevents regressions across all resources, not just
`TestScript`. The trade-off versus Disposition A is scope: this is no longer a
narrow technical correction but a small Infrastructure-wide cleanup, and the
audit + lint work needs to be tracked.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Persuasive – Will not fix" (or the equivalent disposition
for technical corrections that the work group decides not to apply). Document
that the invariant text is treated as the normative source-of-truth for human
readers, and that implementers who care about the constraint should rely on
profile-level or business-rule validation rather than on the FHIRPath
expression. Leave the existing expressions in place to avoid changing
validator behavior on already-published instances.

#### Justification

Declining is defensible only on grounds of stability: any tightening of an
invariant is a behavior change for validators, and instances that were
previously accepted (all-three-present, two-of-three-present) would begin to
fail. For an R5/published resource, that is a non-trivial backwards
compatibility consideration. However, the ticket reports a true bug, not a
design preference, so declining means knowingly leaving a broken invariant in
the spec — which is a weak position to defend at ballot.

---

### Recommendation

**Recommended disposition:** B

The reported defect is real, narrowly scoped to invariant expressions (no
element-shape changes, no breaking changes to instances that were *intended* to
be valid), and exactly the kind of bug HL7 fixes via Technical Correction.
Disposition A solves the immediate problem and should be the minimum bar.
Disposition B is recommended over A because the chained-`xor` anti-pattern is a
generalizable authoring mistake; a small audit and a publication-time lint are
cheap insurance against the same class of defect appearing on other
Infrastructure resources, and a documented "exactly one of N" FHIRPath idiom
gives the community a clear pattern to copy. Disposition C should be rejected:
preserving a knowingly-incorrect invariant trades implementer trust for a very
small stability benefit, because the only instances newly rejected are ones the
spec already labels non-conformant.
