## Summary

Lloyd McKenzie proposes that FHIR Core require all `required` and
`extensible` bindings that point at value sets owned by Terminology.HL7.org
(THO) to be **version-fixed** (i.e., the binding's value-set canonical must
include a `|version` pin) rather than left "current/floating". The
underlying conformance concern is that floating bindings introduce silent,
breaking change for implementers: a `required` binding is violated as soon
as a code that used to be in the value set leaves it (or content semantics
shift), and an `extensible` binding's "must use a code from the value set
if one exists" rule is impossible to satisfy reliably when the value set's
contents are dynamic. Pinning to a published THO version makes the
conformance contract deterministic and lets implementers reason about
upgrades explicitly. No cross-referenced Jira tickets, Zulip threads, or
GitHub items were found for this ticket — it is a freshly triaged change
request with no prior discussion linked in Augury's index.

## Details

**Description:**

There's an issue of conformance for both required and extensible bindings
if the value set isn't stable. With required, you're not allowed to send
anything not in the value set, so if the content changes, that breaks
stuff. With extensible you're required to map and not send local codes or
free text in place of the bound codes unless none exist - if what exists
is dynamic, that's impossible for implementers to cope with.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt a publication-rule (and a build-time check) that every
`ElementDefinition.binding` in the FHIR Core specification with
`strength = required` or `strength = extensible` whose `valueSet`
canonical resolves to a THO-owned value set (`http://terminology.hl7.org/...`)
**MUST** include an explicit version pin (`|<version>`). Concretely:

- Update the FHIR Core publisher / IG-publisher QA rule set to fail (or
  at minimum error-flag) any required/extensible binding to a THO value
  set whose canonical lacks a `|version` segment.
- Sweep the current R6 build: enumerate all
  `required`/`extensible` bindings to THO value sets, and pin each to
  the THO release that the current ballot/CI build is consuming
  (typically the latest published THO release at branch-cut).
- Add the rule and rationale to the "Using Codes in Resources" /
  "Terminologies" guidance pages, framed as a normative constraint on
  spec authoring (not on instance authors).
- Leave `preferred` and `example` bindings untouched, and explicitly
  exempt non-THO value sets (e.g., SNOMED/LOINC implicit value sets,
  externally maintained code systems) from the auto-pin rule, since
  their version model is owned outside HL7.

#### Justification

This addresses the conformance hazard exactly as Lloyd describes it: a
`required` binding is a hard contract, and an `extensible` binding's
"use a code if one fits" obligation both depend on the binding's value
set being a stable, knowable set at the time the conformant artefact is
authored. Pinning to a THO release version makes that set
deterministic and makes upgrades an explicit, reviewable change to the
spec rather than an invisible side-effect of a THO republish. THO
already ships dated, immutable releases, so the mechanism is available;
this change just makes the spec actually use it. It is also consistent
with how IG authors are already encouraged to pin dependencies in
`package.json` / `dependsOn`.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mandating version-pinning on every required/extensible THO
binding, introduce a **stability classification on the value set itself**
and only require pinning where the value set is not classified as
"stable for binding". Mechanically:

- Add (or surface, if already present) a `binding-stability` flag on
  THO `ValueSet` resources (e.g., `stable`, `evolving`, `experimental`).
- Update the FHIR Core publisher rule to require a `|version` pin on
  required/extensible bindings only when the target value set is not
  marked `stable`.
- For `stable` THO value sets, allow unversioned canonicals (current
  behaviour) — the THO governance commitment is that content of a
  `stable` value set is append-only and semantics-preserving, so
  `required`/`extensible` semantics remain safe across releases.
- Provide a one-time sweep that either pins or upgrades each existing
  binding to the right shape based on the new flag.

#### Justification

Disposition A is a blunt instrument: pinning every binding means every
THO content fix (e.g., a typo in a designation, an obviously-missing
synonym) requires a FHIR Core spec republish to be visible to
conformant systems, which inverts the reason THO exists as a
separately-governed terminology repo. Classifying value sets and
requiring pinning only for the unstable ones preserves the conformance
guarantee Lloyd is asking for while keeping the "fix in THO, propagate
without a Core release" workflow for the long tail of well-curated,
truly stable value sets. It also gives THO governance a concrete lever
to reason about: a value set that *can't* commit to stability is
flagged, and Core (and IGs) have to pin against it.

---

### Disposition C: Decline

#### Proposal

Decline the change as written. Maintain the current rule set
(authors choose whether to pin a binding's value-set canonical) and
instead document the existing risk in the "Terminologies" guidance,
recommending — but not enforcing — version pinning on
required/extensible THO bindings. Treat individual unstable bindings
as bugs to be filed and fixed case-by-case.

#### Justification

A blanket rule risks two side-effects the workgroup should weigh
before adopting it: (1) it couples FHIR Core releases to THO content
churn (every THO bump that an IG wants to consume needs a Core
republish or per-IG override), and (2) it creates a large mechanical
sweep over every required/extensible binding in the spec, with
follow-on churn in every IG that re-uses those bindings. If the real
problem is a small number of THO value sets that are genuinely
unstable, those can be addressed individually (either by stabilising
the value set in THO or by pinning that single binding) without a
spec-wide policy change. This disposition is the right one *only* if
the workgroup judges the unstable-binding population to be small and
already known.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible middle ground. Lloyd's
conformance argument is correct on its own terms — a `required`
binding to a moving target *is* an unfulfillable contract, and an
`extensible` binding to a moving target *is* an unanswerable question
for implementers — so simply declining (C) leaves a real conformance
hole that will keep showing up as new tickets. But Disposition A
buys that conformance guarantee at the price of coupling every THO
content fix to a FHIR Core release cycle, which is exactly the
coupling the THO/Core split was designed to avoid. Disposition B
keeps the conformance guarantee where it matters (value sets that
can't commit to stability *must* be pinned) while preserving the
THO-fixes-without-a-Core-release flow for the bulk of well-curated
value sets. It also gives THO governance a concrete, auditable
artefact (the per-value-set stability flag) instead of an unwritten
expectation. Workgroup should validate two things before adopting:
(a) THO is willing to maintain a stability flag with real semantics,
and (b) the publisher tooling can enforce the conditional pin rule at
build time.
