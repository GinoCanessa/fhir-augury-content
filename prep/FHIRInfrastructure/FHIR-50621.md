## Summary

The reporter (Weiyu Zhang) flags an internal inconsistency in the R6 Subscription
module's example workflow diagrams. The REST Hook, WebSockets, and Email workflow
diagrams each include arrows/actions that send notifications (handshakes,
heartbeats, and event-notifications) and label the payload `Bundle.type: history`
— a value carried over from R4. In R6, subscription notification payloads are
expected to use `Bundle.type: subscription-notification`, and the messaging
diagram already uses that value correctly. The request is therefore a
documentation / example correction to bring the REST Hook, WebSockets, and Email
workflow diagrams in line with the messaging diagram and with the current
Subscription specification's normative behavior. The only cross-reference picked
up by FhirAugury is the FHIR artifact mention `Bundle.type`; no related Jira
tickets, Zulip threads, or GitHub items were surfaced.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the example workflow diagrams in the Subscription module so that the
REST Hook, WebSockets, and Email channel diagrams show notification payload
arrows (handshake, heartbeat, and event-notification) carrying
`Bundle.type: subscription-notification` instead of `Bundle.type: history`.
Concretely:

- Edit each of the three diagram source files (e.g., the PlantUML / SVG /
  diagram-and-caption pair on the REST Hook, WebSockets, and Email channel
  pages of the R6 Subscriptions module) so that every arrow currently labeled
  `Bundle.type: history` (or `bundle: history`) is relabeled
  `Bundle.type: subscription-notification` (matching the messaging diagram's
  existing label).
- Re-render any generated artifacts (PNG/SVG) and refresh the figure captions
  / surrounding prose if they reiterate the bundle type.
- Leave the messaging diagram untouched — it already uses
  `subscription-notification`.

No normative content changes; this is an example/diagram correction only.

#### Justification

The Subscription R6 specification defines `subscription-notification` as the
`Bundle.type` for notification payloads, and the messaging diagram already
uses this value. The other three channel diagrams using `history` is a
holdover from R4 that contradicts the normative spec and the sibling diagram
within the same module — a textbook "Technical Correction." Accepting as
requested keeps the four channel diagrams internally consistent, removes a
clear source of implementer confusion, and carries no compatibility risk
because no normative artifact, profile, or value set is altered.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying correction, but broaden the fix beyond the three
diagrams:

1. Apply the same `history` → `subscription-notification` relabeling on the
   REST Hook, WebSocket, and Email diagrams (as in Disposition A).
2. Sweep the rest of the Subscription module pages (channel pages,
   notifications page, examples, and any narrative that still describes
   notification bundles as `history`) and update prose, table cells, and
   inline code samples to match.
3. Audit the Subscription example bundle instances (XML/JSON) shipped with
   the spec to confirm they all use `Bundle.type = subscription-notification`,
   and fix any stragglers in the same change.
4. Add a short "Migrating from R4" note next to the diagrams pointing out
   that R4-era examples used `history` and that R6 uses
   `subscription-notification`, to help readers who arrive via search engines
   pointing at older versions.

#### Justification

The reporter spotted the inconsistency in three diagrams, but the same R4
holdover pattern often shows up elsewhere in the same module (page text,
example bundles, and migration notes). Fixing only the diagrams risks leaving
a partial inconsistency: a reader sees a corrected diagram next to prose or
an example file that still says `history`. A small migration note also
proactively addresses the confusion that produced this ticket in the first
place. The trade-off versus Disposition A is a slightly larger editorial
diff and the need to verify that none of the touched examples are referenced
by tests/IGs that depend on the old label — but as a Technical Correction
this remains low-risk.

---

### Disposition C: Decline

#### Proposal

Decline the change and close as "Not Persuasive." Keep the diagrams as-is on
the grounds that (a) they are illustrative examples rather than normative
content, (b) the normative `Bundle.type` value is already specified in the
Subscriptions notifications page and on `Bundle.type` itself, and (c)
implementers should rely on the normative text rather than inferring values
from diagram labels.

#### Justification

This disposition is included for completeness; it is not a strong position.
It would be defensible only if the workgroup had a policy of freezing
diagrams once published, or if there were a broader plan to retire/replace
the channel diagrams in a near-term release. Otherwise, leaving a known,
reporter-confirmed contradiction between the messaging diagram (correct)
and the REST Hook / WebSocket / Email diagrams (incorrect) inside the same
module section is hard to justify when the fix is purely editorial.

---

### Recommendation

**Recommended disposition:** A

This is exactly the kind of error the "Technical Correction" type was created
for: a clearly identified, narrow, non-normative inconsistency between
sibling diagrams in the same module, with the messaging diagram already
demonstrating the correct value (`Bundle.type: subscription-notification`).
Accepting as requested resolves the ticket with minimal editorial risk and
no normative impact. Disposition B is also reasonable and worth executing if
the editor doing the work has cycles to sweep adjacent prose/examples for
the same R4 holdover; if that audit is in scope, it can be folded into the
same change set. Disposition C is included for completeness but is not
recommended — declining a confirmed, low-risk Technical Correction would
leave an avoidable contradiction inside the published Subscription module.
