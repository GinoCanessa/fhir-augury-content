## Summary

This ticket asks the SMART Web Messaging specification to define a standard
way for an activity to acknowledge a request as **received but not yet
completed** — i.e., an "in-progress" or pending response — in addition to the
current pattern of single or multiple final response messages. Today,
activities such as `ui.done` only carry success or failure outcomes, which
leaves no normative way to signal cases like "I have shown the save dialog
and am waiting for the user." The reporter notes that today this is left to
each activity to invent ad-hoc (e.g., extending `ui.done`), and proposes
that SMART Web Messaging instead establish a single shared pattern available
to **all** activities — either always-valid by default, or opt-in via a
parameter on the originating request that signals whether a pending response
is acceptable.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
for this issue in the indexed sources, so the ticket stands on its own.

## Details

**Description:**

> Currently, activities can respond to requests with either a single message
> or multiple messages. However, activities such as `ui.done` only have a
> way of indicating success or failure. There are use cases where an
> application may want to respond with "received, but not completed" and
> there is no way to communicate that (e.g., I am waiting for the user to
> respond to a "save" dialog).
>
> So far, it is left to each activity to define any functionality that
> could be used (e.g., `ui.done` itself could be modified to match). There
> should be an established pattern that can be made available to all
> activities so that "in-progress" is an option for any request. This could
> be either by default (e.g., it's always valid), or opt-in (e.g., a
> parameter on the request indicating if a pending response is allowed).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define, in the SMART Web Messaging specification, a normative
**"in-progress" / pending response pattern** that is available to **every
activity** rather than redefined per-activity. Concretely:

- Add a new shared **`responseStatus`** (or equivalent) field on the
  response envelope with at least the values `in-progress`, `success`, and
  `failure`. `in-progress` indicates the recipient has accepted the request
  and will produce a final `success` / `failure` response later, correlated
  by the original request `messageId`.
- Specify that an `in-progress` response **does not terminate** the
  exchange: the recipient remains obligated to send a terminal
  (`success` / `failure`) response, and senders MUST be prepared to receive
  one or more `in-progress` responses followed by exactly one terminal
  response.
- Make the behavior **opt-in per request** via a new request-level field
  (e.g., `allowInProgress: true`) so existing recipients that don't
  understand the pattern continue to behave as today; recipients MUST NOT
  emit `in-progress` unless the request opted in.
- Update the `ui.done` activity (and any other activity examples in the
  spec) to show the pattern in action, replacing the bespoke handling
  hinted at in the ticket.

#### Justification

This is exactly what the reporter asks for: a single, shared mechanism so
that every activity gets "received but not completed" semantics for free,
instead of each activity reinventing it. Putting the field on the
**response envelope** keeps the change orthogonal to any specific activity
verb, which is consistent with how SMART Web Messaging models its
request/response envelope. Making the behavior opt-in via the request side
preserves backward compatibility: hosts that don't set `allowInProgress`
will never see a new response shape, so existing apps continue to work
unchanged. Aligning `ui.done` with the new pattern in the same change
prevents the "two ways to do the same thing" problem the ticket warns
about.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding a normative `responseStatus = in-progress` to the
envelope, treat "in-progress" as a **progress notification** layered on
top of the existing multi-message response capability:

- Keep the response semantics as they are today (a response is either
  success or failure, terminal).
- Define a small, optional **progress message type** (e.g., a
  `messageType` of `progress` or a dedicated `progress` envelope) that a
  recipient MAY emit zero or more times before the terminal response,
  carrying a free-form `state` (e.g., `awaiting-user-input`, `working`)
  and an optional human-readable `detail`.
- Tie the progress messages to the original request via the existing
  `messageId` / correlation mechanism that multi-message responses
  already rely on.
- Add a single sentence to each activity's response section noting that
  progress messages MAY be emitted; do **not** modify `ui.done` itself.

#### Justification

This keeps the existing "a response is terminal" invariant intact, which
is simpler for implementers who only care about success/failure and never
have to learn a third response state. It piggybacks on the already-defined
multi-message capability, so the wire format change is minimal and
backward compatible by construction (older receivers that don't
understand `progress` messages can simply ignore them and wait for the
terminal response). It also keeps activity definitions like `ui.done`
untouched, which avoids opening up version-compatibility questions for
hosts that have already implemented them. The trade-off vs. Disposition A
is that "in-progress" is then a soft, advisory signal rather than a
first-class response state — implementers who want strict typing on
response status will still have to model it themselves.

---

### Disposition C: Decline

#### Proposal

Decline to add a generic in-progress response pattern to SMART Web
Messaging at this time. Document in the ticket resolution that activities
which genuinely need long-running semantics (e.g., `ui.done` waiting on a
save dialog) should handle that **inside the activity's own response
contract** — for example, by defining activity-specific intermediate
states — rather than introducing a cross-cutting envelope-level concept
without demonstrated multi-activity demand.

#### Justification

The ticket is reported by a single contributor, has sat at `Triaged` /
`Medium` since 2021 with no comments, no linked tickets, no Zulip
discussion, and no GitHub PRs or issues indexed against it. Adding a new
response state (or a new progress-message type) to the envelope is a
non-trivial spec change that all hosts and apps will need to handle, and
it should be justified by concrete cross-activity use cases and
implementer demand — neither of which is currently on the record.
Declining now does not foreclose the change: if more activities surface
the same need, the request can be re-opened with that evidence.

---

### Recommendation

**Recommended disposition:** A

The ticket identifies a real gap — there is currently no portable way for
an activity to say "received, working on it" — and the reporter explicitly
asks for a **single shared pattern** rather than per-activity invention.
Disposition A delivers exactly that: a small, opt-in addition to the
response envelope (`responseStatus = in-progress`, gated by an
`allowInProgress` request flag) plus an aligned update to `ui.done` so
the spec demonstrates the pattern. Backward compatibility is preserved
because nothing changes for requests that don't opt in, which addresses
the main risk that drives Disposition C. Disposition B is a reasonable
fallback if the workgroup wants to avoid introducing a new response state,
but it leaves "in-progress" as an advisory signal that each activity is
free to interpret differently — which is precisely the situation the
reporter is asking the workgroup to fix. Given the silence on the ticket
since 2021, the workgroup should also confirm there is at least one
concrete implementer use case before scheduling the change; if none
materializes, fall back to Disposition C.
