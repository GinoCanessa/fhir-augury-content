## Summary

The ticket asks the FHIR Infrastructure WG (SMART on FHIR / Brands) to add
a way for an EHR's Brands bundle to publish the *trust communities*
(e.g., TEFCA, Carequality) that each endpoint participates in, including
enough information for a client to find the trust community's dynamic
registration endpoint. The motivation is the proposed ONC HTI-2 rule
§(b)(2)(iv) "Trust community discovery for dynamic registration", which
would require EHRs to publish, for every service base URL, the trust
community's name, contact information, web address, and identifying URI
in a publicly available computable format. Cooper Thompson (the
reporter) suggests a new standard extension on Brands to carry this
information, while noting the data could alternatively (or additionally)
live in `.well-known/udap`; including it in Brands would let an app
decide *up-front* whether to register manually via the EHR portal or
dynamically via a trust community.

The related ticket [FHIR-40510](https://jira.hl7.org/browse/FHIR-40510)
("Discuss mechanism for determining trust community for a given resource
server", filed against the deprecated US UDAP Security IG and resolved
"Persuasive with Modification" by the Security WG) covers an adjacent
problem: when a resource server supports multiple trust communities,
clients need a way to discover which ones, which is preferred, and how
to negotiate. FHIR-40510 lives at the *server-discovery* layer (per-
endpoint metadata), whereas FHIR-46557 asks for the same information to
be denormalized into the *directory-style* Brands bundle one level above
that.

The Argonaut Zulip thread that the reporter started about this proposal
quickly converged on the view that the trust-community information
belongs primarily in the server's discovery document (UDAP / SMART
configuration), and that duplicating it into the Brands bundle is of
limited additional value because a client must hit the server's
discovery endpoint anyway to find the dynamic-registration URL. Cooper
himself ends the thread by saying the recommendation should be to ask
ONC *not* to require this information be published in duplicate, and
Josh Mandel agrees that the Jira can be quietly deferred until the final
HTI-2 rule lands.

## Details

**Description:**

> HTI-2 is proposing a requirement that EHRs publish trust community
> details (including dynamic registration endpoint) as part of the
> endpoint publication details.
>
> *(iv) Trust community discovery for dynamic registration.* Trust
> community details such as trust community name, contact information,
> web address, and identifying Uniform Resource Identifier (URI) must be
> publicly published in a computable format at no charge for each
> service base URL published in accordance with (b)(2)(iii) of this
> section.
>
> While this information probably best belongs in something like
> `.well-known/udap`, it isn't unreasonable to duplicate that into the
> brands bundle for ease of access.
>
> I'd propose we create an extension to represent this information in
> the Brands bundle. We might consider whether it makes more sense to
> include the name, contact info, and the identifier URL directly in
> the Brands bundle, and then a link to the `.well-known/udap` endpoint
> for the actual registration endpoint. Once you get into dynreg, you
> probably need to hit `.well-known/udap` anyway, so just pulling out
> only the URL doesn't seem like it would fully solve a problem. By
> putting name, contact info, and identifying URL in the Bundle, we
> give the app enough clues for them to decide if they need to hit the
> `.well-known/udap` endpoint for technical details. We might need to
> see if that would be a reasonable interpretation of ONC's proposed
> HTI-2 requirement.

## Related Jira Tickets

### [FHIR-40510: Discuss mechanism for determining trust community for a given resource server](https://jira.hl7.org/browse/FHIR-40510)

Filed against the (now deprecated) US UDAP Security IG, owned by the
Security WG, and resolved **Persuasive with Modification**. It explores
how a client should determine which trust communities a given resource
server supports, how a server should advertise a default or preferred
community, and how the SW-statement JWT trust-community claim should be
negotiated. This is the *server-side discovery* counterpart to
FHIR-46557's *directory-side publication* request — they ask the same
underlying question (how does a client know which trust community
applies?) at two different layers of the stack, and any solution adopted
in FHIR-46557 should remain consistent with the server-discovery
mechanism FHIR-40510 lands on.

Users involved: Joseph M. Lamy (reporter); Security WG (resolver,
unassigned individual).

## Related Zulip Discussions

### [argonaut > HTI-2 Endpoint Publishing of Trust Community Details](https://chat.fhir.org/#narrow/stream/argonaut/topic/HTI-2%20Endpoint%20Publishing%20of%20Trust%20Community%20Details)

Cooper Thompson opened the topic on 2024-08-06 by announcing that he'd
filed FHIR-46557 to define a Brands extension for the ONC HTI-2
trust-community publication requirement. Josh Mandel pushed back almost
immediately: the trust-framework list should live in the *server's*
discovery document (SMART configuration / UDAP signed metadata), the
same way FHIR version is, since a server can support trust communities
without supporting Brand discovery and vice versa. Andrew Fagan and
Cooper countered that even if the canonical home is the server, a
denormalized copy in the Brands endpoint listing reduces the number of
fetches an IAS app needs before it can render a chooser to the user.

Jason Vogt raised a governance objection from the SSRAA IG perspective —
broadcasting *which* trust communities a server participates in lets
clients shop between communities, which trust-network operators worry
will erode the trust fabric. Josh Mandel and Brendan Keeler challenged
that framing, arguing that if real differences exist between
communities, transparency is a feature, not a bug, and that a community
shouldn't "own" its members. Dan Cinnamon brought up prior connectathon
work where a national directory was expected to point clients at FHIR
base URLs and let them follow up at `.well-known/udap` for registration
details.

Josh Mandel then made the substantive argument that ultimately settled
the thread: to actually *use* "this server belongs to TEFCA" the client
still has to fetch the server-specific configuration to learn the
dynamic-registration endpoint, so the value of denormalizing the
trust-community list into the Brands bundle is limited; if anything,
trust-framework directories will define their own discovery
mechanisms. Cooper concluded (2024-08-08 21:46) that the recommendation
should be to ask ONC **not** to require duplicate publication —
publishing once in `.well-known` is enough — and that FHIR-46557 should
either be deferred or marked not-persuasive. Josh agreed and suggested
quietly deferring until ONC's final rule. A late note from Mark
Marciante-LP (2024-08-20) observed that the trust-framework work and the
FAST National Directory are tackling overlapping concerns from different
(legal vs. technical) angles and need harmonisation.

Participants: Cooper Thompson, Josh Mandel, Greg Thole, Andrew Fagan,
Jason Vogt, Brendan Keeler, Dan Cinnamon, Mark Marciante-LP.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new standard extension on the Endpoint resource (carried in the
Brands publication bundle) named something like
`smart-endpoint-trust-community` with a repeating complex value
containing:

- `name` (string) — human-readable trust community name (e.g., "TEFCA").
- `contact` (ContactDetail) — operator contact information.
- `webAddress` (url) — public web page describing the community.
- `identifier` (Identifier with system + value) — the URI that uniquely
  identifies the trust community (matches the URI that appears in UDAP
  signed metadata's `community` claim).
- `dynamicRegistrationEndpoint` (url, 0..1) — convenience pointer; if
  omitted, the client falls back to the server's `.well-known/udap`
  metadata.

Document the extension in the SMART App Launch / Brands publication
section, and mark it usable directly on `Endpoint` entries in the
publication bundle so HTI-2-conformant publishers can satisfy
§(b)(2)(iv) without changes to the base SMART configuration.

#### Justification

This is the literal ask in the ticket and aligns with how SMART Brands
already denormalises a small amount of server metadata (e.g., FHIR
version) into the Brand bundle to spare patient-facing IAS apps an extra
network round-trip before they can present a chooser. Andrew Fagan and
Cooper made exactly that "developer-burden" argument in the Argonaut
thread, and ONC HTI-2 §(b)(2)(iv) — if it lands as proposed — explicitly
requires the trust community details to be published "for each service
base URL published in accordance with (b)(2)(iii)", which is the SMART
Brands publication endpoint. Defining the extension via SMART (rather
than each trust framework defining its own) gives ONC a single, named
artifact to point at.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add trust-community fields to the Brands bundle. Instead:

1. Coordinate with the Security WG (owners of FHIR-40510 and the UDAP
   Security IG) to make the *server's* discovery document the canonical
   home for the trust-community list — extending the existing UDAP
   signed metadata `community` mechanism (see
   `https://build.fhir.org/ig/HL7/fhir-udap-security-ig/discovery.html#multiple-trust-communities`)
   and mirroring the same data into `.well-known/smart-configuration` so
   non-UDAP servers can also publish it.
2. In the Brands bundle, add only a *minimal* pointer: confirm that the
   existing `Endpoint.address` (the FHIR base URL) is sufficient to let
   a client locate `.well-known/smart-configuration` and
   `.well-known/udap`, and document this lookup pattern in the SMART
   Brands publication guidance.
3. For trust-community membership identifiers that an IAS app might want
   to filter on *before* hitting the server, recommend using
   `Organization.identifier` with the trust community's assigned
   organisational identifier (Josh Mandel's suggestion in the thread),
   reusing existing FHIR machinery.
4. Submit comments to ONC asking that HTI-2 §(b)(2)(iv) be satisfied by
   server-side discovery rather than mandating duplicate publication in
   the directory bundle.

#### Justification

This is the position the Argonaut thread converged on. Josh Mandel's
"every path of use requires resolving additional server information
anyway" argument makes the denormalisation in Disposition A largely
cosmetic: the client still has to fetch `.well-known/*` to get the
dynamic-registration URL, so it might as well learn the trust-community
list at the same time. It also avoids two real costs of Disposition A:

- **Drift / duplication risk** between the Brands-bundle copy and the
  authoritative server-discovery copy, and
- **Premature commitment** to a structure (extension on `Endpoint`)
  before FHIR-40510 / Security WG settle the server-side model the
  Brands extension would have to mirror. Doing the server-side work
  first avoids defining a Brands extension that has to be revised once
  the canonical model lands.

It also gives the workgroup a clean public-comment posture toward ONC
that doesn't commit the SMART spec to a duplicate-publication mandate.

---

### Disposition C: Decline (defer pending HTI-2 finalisation)

#### Proposal

Mark the ticket "Persuasive but not adopted in this cycle / deferred"
with a note that the WG will revisit only if and when ONC HTI-2
§(b)(2)(iv) is finalised in a form that materially requires Brands-level
publication of trust-community details. In the meantime, take no action
on the SMART Brands bundle or on a new extension. Direct any near-term
implementer questions to the existing UDAP signed-metadata
multiple-trust-communities mechanism.

#### Justification

This is what the reporter and the Argonaut participants explicitly
recommended: Cooper Thompson on 2024-08-08 — "we can either defer
FHIR-46557, or make it not persuasive" — and Josh Mandel — "quietly
deferring makes sense to me". The HTI-2 rule that motivates the request
is *proposed*, not final; defining a normative extension against a
proposal risks locking in structure that doesn't match what ONC
ultimately publishes. Declining now is reversible: if the final rule
preserves §(b)(2)(iv) substantially as written, the WG can re-open and
adopt Disposition A or B with full information; if the rule is dropped
or restructured (Brendan Keeler's read that ONC is using deliberately
vague language to avoid mandating TEFCA suggests this is plausible),
the WG saves itself from defining and then deprecating an extension.

---

### Recommendation

**Recommended disposition: C — Decline (defer pending HTI-2 finalisation).**

This is the disposition the ticket's own reporter and the most engaged
Argonaut participants converged on, and it has the strongest
risk/benefit profile for a proposal whose primary driver — ONC HTI-2
§(b)(2)(iv) — is still in proposed-rule form. Disposition A would commit
SMART Brands to a denormalised structure that, by Josh Mandel's
analysis, adds limited value (clients still must fetch
`.well-known/*` to act on the trust-community membership) and that may
need revision once the Security WG's adjacent work
([FHIR-40510](https://jira.hl7.org/browse/FHIR-40510)) defines the
canonical server-side model. Disposition B is the right *long-term*
direction but is best executed jointly with the Security WG and after
the final ONC rule clarifies what publishers are actually required to
expose; pursuing it now without that clarity risks the same
re-do problem. Deferring keeps the ticket alive as a tracking record,
positions the WG to submit consolidated public comment to ONC asking
that server-side discovery be accepted as conformant, and preserves the
option to adopt either A or B cleanly when the regulatory picture
settles.
