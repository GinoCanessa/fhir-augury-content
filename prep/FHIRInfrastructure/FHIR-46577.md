## Summary

Emma Jones reports that, in the Patient resource's Search Parameters
documentation, the entry for the common `goal` search parameter is rendered
with an extra hyperlinked "Goal" (the resource name) preceding the parameter's
description, and asks that the link be removed. Triage history (captured in
the Zulip thread referenced below) shows the ticket was originally assigned
to Patient Care, then re-routed: Michelle Miller and Grahame Grieve confirmed
this is a tooling rendering pattern, not a Patient-Care content issue, which
is why the ticket now sits with FHIR Infrastructure and carries the
`FHIR-Tools-Core` label. Companion ticket **FHIR-46576** raises the same
concern against another resource page; both tickets share the same root cause
(IG/publisher rendering of common search parameters that are sourced from
another resource). At present the submitter has not yet explained *why* the
link should be removed, and the reviewing committers have noted that all
common search parameters share this format intentionally (the link points the
reader at the source resource that owns the search parameter definition).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Suppress the leading resource-name hyperlink (the "Goal" link) for the `goal`
common search parameter when it is rendered in the Patient resource's Search
Parameters table on `patient.html`. Concretely, change the publisher's
search-parameter rendering so that, for entries where the search parameter
is sourced from another resource (i.e. a common / cross-resource search
parameter such as `Patient.goal` → `Goal.patient`), the inline "(SourceResource)"
hyperlink that currently precedes the description text is removed. The link
to the canonical SearchParameter definition (the existing column / cell
linking) is retained; only the redundant inline resource-name link in the
description cell is dropped. The same fix would also resolve the companion
ticket FHIR-46576.

#### Justification

This honours the submitter's literal request. It can be argued on grounds of
visual consistency with non-common search parameters (which do not show the
inline source-resource link), and on the basis that the link is partially
redundant with other navigation already present on the page. Because the
rendering is generated by tooling, the change is local to the publisher and
does not alter any normative artifact (the SearchParameter resource itself is
unchanged).

---

### Disposition B: Alternative Approach

#### Proposal

Keep the source-resource link (it is informative and intentional — Grahame
explicitly confirmed it as the designed behaviour) but improve its
*presentation* so it reads less like a duplicate of the parameter name.
Concrete options, any of which would address the underlying complaint
without losing information:

1. Move the source-resource indicator out of the inline description and into
   a dedicated "Source" column / badge on the Search Parameters table, so it
   stops appearing to "precede the description".
2. Re-style the inline reference (e.g. as a small "(from Goal)" suffix after
   the parameter name, or as a tooltip on the parameter name) instead of a
   leading hyperlink that visually competes with the description.
3. Document the convention with a one-line legend above the Search Parameters
   table on every resource page, so readers know the leading link denotes the
   *defining* resource for common search parameters.

Pick option (2) as the concrete proposal: render common search parameters as
"`goal` *(from [Goal](goal.html))* — description…" rather than
"[Goal](goal.html) `goal` — description…".

#### Justification

The Zulip discussion shows the link is deliberate and uniformly applied
across all common search parameters; outright removal (Disposition A) would
hide the fact that the parameter is defined by another resource, which is
genuinely useful navigational metadata for implementers writing search
queries. Re-styling addresses the submitter's apparent concern (that the link
visually intrudes on the description) while preserving the information.
Because the convention is global to the tooling, fixing it once benefits
every resource page, not just Patient. This also keeps FHIR-46576 in scope
with the same fix.

---

### Disposition C: Decline (Not Persuasive / As Designed)

#### Proposal

Mark **Not Persuasive** (or **Persuasive – With Mod = no change** pending the
submitter's clarification) on the basis that the leading source-resource
hyperlink is the designed rendering for common search parameters across the
entire FHIR specification. Resolution comment: the inline link denotes the
resource that *defines* the common search parameter (here, `Goal.patient`,
surfaced on the Patient page as `goal`), is applied uniformly to every common
search parameter on every resource page, and removing it would lose
information without providing a corresponding benefit.

#### Justification

Both reviewing committers in the Zulip thread (Michelle Miller, Grahame
Grieve) note the rendering is intentional and consistent. Michelle's comment
on the ticket explicitly states "All common search parameters share this
format". Grahame asked outright "Why would it be removed?" and the submitter
has not (as of last update) responded with a rationale. Removing the link in
isolation for `goal` on Patient would *break* the consistency the submitter
implicitly invokes; removing it everywhere would discard navigational
information used by implementers. Until a concrete harm or use case is
articulated, declining is defensible.

---

### Recommendation

**Recommended disposition:** C (with a graceful path to B if the submitter
provides a stronger rationale).

The Zulip discussion makes a clear case that the rendering is deliberate,
universal across resource pages, and informational rather than incidental,
and the submitter has not yet articulated *why* the link is undesirable. On
that record, declining ("not persuasive — as designed") is the
intellectually honest answer, and it has the side benefit of resolving
FHIR-46576 the same way without touching publisher rendering globally. The
workgroup should, however, leave the door open: if Emma Jones responds to
Michelle's clarifying question with a concrete usability or accessibility
concern (for example, screen-reader noise, or visual confusion in tables
with many common search parameters), the right follow-up is Disposition B
(re-style globally) rather than Disposition A (remove for Patient only),
because a one-resource fix would itself create the inconsistency the
submitter is implicitly objecting to.
