## Summary

Rene Spronk requests a documentation improvement to the description of the
new `resource` search parameter type in the FHIR Core specification. The
reporter observes that the current narrative is too terse for readers
encountering this parameter type for the first time and proposes a small
rewording that (a) adds an explicit cross-reference to section 3.2.1.5.16
where the mechanism is described in more depth, and (b) makes it clearer
that the chained-through element is itself a resource *embedded within an
overarching resource* (the canonical example being `Bundle.entry.resource`
inside a `Bundle`). The change is purely editorial — no schema, conformance
artifact, or behavioral change is implied. Augury surfaces only a single
cross-reference (the FHIR artifact `Bundle.entry.resource` mentioned in the
description); no other Jira tickets, Zulip threads, or GitHub items are
linked to this issue.

## Details

> [Bundle.entry.resource](https://build.fhir.org/bundle-definitions.html#Bundle.entry.resource)).",

> actual resources (e.g.,
> [Bundle.entry.resource](https://build.fhir.org/bundle-definitions.html#Bundle.entry.resource))
> embedded within an overarching resource (e.g. Bundle)"

## Proposed Dispositions

The cross-reference index ties this ticket only to the FHIR Core artifact
`Bundle.entry.resource`. There is no GitHub cross-reference and therefore
no repo briefing to anchor a code/spec-path proposal; however, the change
is editorial and lives in the FHIR Core search documentation page that
defines the `resource` search parameter type (the page that introduces the
SearchParameter `type` value and currently contains the wording quoted in
the ticket).

### Disposition A: Accept As Requested

#### Proposal

Apply the reporter's suggested wording verbatim to the FHIR Core
documentation page that defines the `resource` search parameter type (the
SearchParameter `type` table / narrative in the search documentation,
where the text quoted in the ticket currently appears). Specifically:

- Replace
  > "A `resource` type parameter is a special case parameter type that is
  > used to enable chaining through elements that are actual resources
  > (e.g., Bundle.entry.resource)."

  with
  > "A `resource` type parameter is a special case parameter type (see
  > section 3.2.1.5.16) that is used to enable chaining through elements
  > that are actual resources (e.g., Bundle.entry.resource) embedded
  > within an overarching resource (e.g. Bundle)."

- Render the section pointer as an in-page anchor link rather than a
  literal numbered reference, so the cross-reference survives section
  renumbering. (FHIR Core publication does not stabilise section numbers
  across versions, so the link target should be the anchor of the
  detailed-description section, not the literal "3.2.1.5.16" string.)

#### Justification

The change is editorial, scoped to a single sentence, and addresses a real
clarity problem: the current wording introduces a non-obvious parameter
type without telling the reader where the full description lives, and the
example (`Bundle.entry.resource`) only makes sense once the reader knows
the embedded-in-an-overarching-resource framing. Adding the cross-reference
matches the documentation pattern used elsewhere in the search chapter,
and the "embedded within an overarching resource (e.g. Bundle)" gloss
mirrors how the mechanism is explained in the longer narrative section the
reporter is pointing at. There is no conformance impact, no
SearchParameter resource changes, and no implementer-visible behavior
change, so the cost/risk is essentially zero.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the *intent* of the request but refine the wording and link
structure rather than copying the suggestion verbatim. Concretely:

1. Keep the introductory sentence short and factual, e.g.
   > "A `resource` type parameter is a special case parameter type used
   > to enable chaining through elements whose value is itself a resource
   > embedded inside an overarching resource (for example,
   > [`Bundle.entry.resource`](https://build.fhir.org/bundle-definitions.html#Bundle.entry.resource)
   > inside a `Bundle`). For full semantics, see [Searching the contents
   > of a Bundle (or other embedded resources)](#anchor-to-detailed-section)."

2. Replace the literal "section 3.2.1.5.16" pointer with a named anchor
   link (as in Disposition A), and add a parallel forward-reference from
   the SearchParameter `type` value table entry for `resource` so a
   reader landing on either page can navigate to the other.

3. Optionally add one sentence noting that the parameter type is *only*
   meaningful when the underlying element is typed as `Resource` (or a
   choice that includes `Resource`), which is the actual gating condition
   and is currently implicit.

#### Justification

The reporter's proposed wording solves the immediate clarity gap but
introduces two minor issues that are worth fixing in the same pass: a
literal section number ("3.2.1.5.16") is fragile across publication runs,
and the "embedded within an overarching resource (e.g. Bundle)" phrasing
reads as if Bundle is the only host, when in fact any element typed as
`Resource` (e.g., `Parameters.parameter.resource`,
`DomainResource.contained`) qualifies. Tightening the wording while we
have the page open avoids a follow-up ticket and produces documentation
that ages better. The trade-off vs. Disposition A is a small amount of
extra editorial work and a slight deviation from the literal suggestion,
which is appropriate for narrative documentation where the reporter's
intent — not their exact words — is what matters.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" / no change, on the grounds that the
existing one-sentence introduction is intentionally terse because the full
description of the `resource` search parameter type is given immediately
after in the same chapter, and adding a "see section X" pointer to text
that is only a few paragraphs above its detailed treatment is
unnecessary.

#### Justification

This is the weakest of the three options and is included for completeness.
It would be defensible only if the workgroup determines that the existing
narrative already provides the cross-context within a short scroll
distance and that adding pointers risks over-cluttering the search
chapter. The reporter has, however, identified a concrete reading
difficulty, the proposed change is local and editorial, and FHIR Core
generally favours small documentation improvements that lower the
on-ramp for new readers — so a decline would need stronger justification
than "the information is nearby."

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The reporter has correctly identified a real
clarity gap and proposed a sensible fix, so a flat decline (C) is not
warranted. However, accepting the suggestion verbatim (A) bakes in a
literal section-number cross-reference that will not survive
renumbering across FHIR Core publications, and the "embedded within an
overarching resource (e.g. Bundle)" wording understates the mechanism —
any element typed as `Resource` qualifies, not just `Bundle.entry.resource`.
Disposition B keeps the reporter's intent (add a forward link, add the
"embedded resource" framing) while replacing the brittle numeric pointer
with an anchor link, generalising the example, and optionally clarifying
the gating condition (the underlying element must be typed as `Resource`).
The change remains purely editorial, has no conformance impact, and can be
made directly in the FHIR Core search documentation page where the quoted
sentence currently lives.
