## Summary

The ticket reports that the FHIR Subscription specification under-specifies the
`handshake` notification exchange used to confirm a REST hook channel. The
current text says the server SHALL POST a `handshake` notification and that the
client SHALL return a "success HTTP code (e.g., `200`)", but it does not
clarify (a) which response codes count as success (only 200, the full 2xx
range, 3xx redirects, etc.), (b) whether the response is expected to carry a
body, and (c) whether an `OperationOutcome` is expected on a non-success
response — noting that a misdirected handshake may hit an endpoint that has
no FHIR awareness at all.

The reporter asks for two coordinated edits in FHIR Core: first, add the three
items above to the "Channel basics" checklist of things that channel
definitions must specify; and second, fill those items in for the existing
channel definitions, particularly REST hook. No companion Jira tickets were
linked, but the discussion on the `#subscriptions` Zulip stream
("Handshake request and response payload") is the direct origin of the
ticket and indicates that a parallel "mirror change" ticket would later be
needed against the Subscriptions Backport IG.

## Details

**Description:**

> The section on REST hook says:
>
> > After a successful `handshake` notification has been sent and accepted, the
> > server SHALL update the status to `active`. Any errors in the initial
> > `handshake` SHALL result in the status being changed to `error.`
>
> And:
>
> > 3. Server performs an HTTP POST to the requested endpoint with a `handshake` notification.
> >
> > 4. Client Endpoint accepts the POST and returns a success HTTP code (e.g., `200`).
>
> It isn't clear if there is an expected body in the response, or if any
> response code other than 200 is "success", eg. a 201 or a 3xx.
>
> [Channel basics](https://build.fhir.org/subscription.html#channel-extending-basics) says:
>
> > At a minimum, the following items should be defined: ...
> >
> > - Whether `handshake` notifications are used, and guidance on usage if they are
>
> Clarify in Channel Basics that the definition should include:
>
> - which handshake response codes indicate "success";
> - the content of a handshake response payload (or that none is expected);
> - whether an OperationOutcome payload is expected in a non-success response (this would need to be optional because a handshake with an incorrect endpoint wouldn't necessarily know anything about FHIR).
>
> Then update the definition of the existing channels (particularly REST hook)
> to provide these additional details.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Core Subscription specification (`subscription.html`) in two
coordinated edits:

1. In the **Channel basics** section
   (`#channel-extending-basics`), expand the bulleted list of items a
   channel definition must specify so that, in addition to "Whether
   `handshake` notifications are used, and guidance on usage if they are",
   a channel that uses handshakes MUST also define:
   - which HTTP response status codes from the client endpoint indicate a
     successful handshake (e.g., narrowing to `200` only, or admitting any
     `2xx`, and stating explicitly how `3xx` redirects are treated);
   - the expected content (if any) of the handshake response body — i.e.,
     whether the response payload is empty, an `OperationOutcome`, or some
     channel-specific structure; and
   - whether an `OperationOutcome` is expected (or merely permitted) in a
     non-success response, with a note that this MUST be optional because a
     handshake delivered to an incorrect endpoint cannot be assumed to be
     FHIR-aware.

2. In the **REST hook channel** section, fill in those three items
   concretely for REST hook: state that any `2xx` response is "success" (or
   pick `200` only — the workgroup should decide), that no response body is
   required and any body SHOULD be ignored by the server, and that an
   `OperationOutcome` MAY be returned on a non-success status to aid
   diagnostics but MUST NOT be relied upon.

#### Justification

This is a clarification, not a behavioural change: the spec already requires
the server to set the Subscription status to `active` on success and `error`
on failure, but leaves the success/failure boundary ambiguous. Tightening
"a success HTTP code (e.g., `200`)" into a concrete rule is consistent with
how FHIR generally treats HTTP semantics elsewhere (e.g., the RESTful API
documents the meaning of each 2xx and 3xx code). The reporter and the
responsible workgroup chair (Gino Canessa) explicitly agreed on the
`#subscriptions` Zulip thread that more specificity is needed, and Gino
recommended filing against Core precisely so the IG can later mirror the
change.

---

### Disposition B: Alternative Approach

#### Proposal

Make the same Channel basics checklist additions as Disposition A, but for
the REST hook concretisation, defer the response-body and
`OperationOutcome` rules to a follow-up ticket and limit the immediate Core
change to:

- accepting any `2xx` response as success and any other response (including
  `3xx`, `4xx`, `5xx`, and connection failures) as failure; and
- a SHOULD-level recommendation that handshake responses use `204 No
  Content` when no body is returned.

Couple this with a note that handshake-response semantics may be revisited
once the in-flight work on integrating auth into subscription workflows
(referenced by Gino on Zulip) lands, since auth challenges (e.g., 401/redirect
flows) interact directly with what counts as a "successful" handshake.

#### Justification

The Zulip thread surfaced two adjacent concerns that are not in the ticket:
auth integration into the subscription workflow, and Cooper's prior issues
about wanting a body in the response. Both could materially change what
"success" means for a handshake (e.g., a 401 with a `WWW-Authenticate`
header is not really a hard failure). Locking down the response-body and
`OperationOutcome` rules now risks having to revise them in the next cycle.
Splitting the work keeps the urgent, low-risk clarification (which codes are
success) on the current train while leaving room for the richer
response-payload design to be done deliberately.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that "a success HTTP
code (e.g., `200`)" is already the standard FHIR convention for success
across the RESTful API and that channel-definition authors are free, today,
to tighten the rules in their own channel definitions under the existing
Channel basics extension mechanism.

#### Justification

The Channel basics section is intentionally a *checklist for channel
definitions*, not a normative rule for any one channel. Implementers
designing a new channel can — and arguably should — already specify
response semantics. For REST hook specifically, the existing wording is the
same level of precision as comparable HTTP-callback specs in the wider
ecosystem.

---

### Recommendation

**Recommended disposition:** A

The change is a textual clarification with no schema or wire-format impact,
and it directly addresses interoperability ambiguity that the reporter hit in
practice and that the workgroup chair acknowledged on Zulip. Disposition C
under-serves implementers: the ambiguity is already producing questions on
the `#subscriptions` stream, and pushing it down to per-channel definitions
just moves the inconsistency. Disposition B is defensible if the workgroup
believes the auth-integration work is imminent enough that response-body
rules will need to change anyway, but the response-body clause in
Disposition A is permissive (no body required, `OperationOutcome` optional)
and should not need to be rewritten when auth lands. Once Disposition A is
applied to Core, a mirror change ticket should be filed against the
Subscriptions Backport IG, as Gino suggested.
