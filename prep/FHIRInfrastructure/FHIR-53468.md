## Summary

The ticket asks the SDC `Questionnaire/$populate` operation to gain a new
`initialResponse` parameter typed as `QuestionnaireResponse 0..1`, so that
populate logic can take a previously-provided (in-progress or
prior-instance) `QuestionnaireResponse` as input and use its answers to
inform what gets populated. Two driving use cases are called out: (1)
**DTR**, where a provider-side populate may run after a payer-side
pre-population has already produced a partial QR, and the provider step
needs to start from those values; and (2) **Adaptive Questionnaires**,
where answers already given to earlier questions should be available as
inputs (via FHIRPath/CQL `%context`-style expressions) when populating
initial values for newly-revealed questions. The discussion has been
explicitly broadened on the `questionnaire` Zulip stream into a
"re-populate" capability that the SDC IG already gestures at but does
not formally support; a Confluence page (*SDC RePopulate*,
[FHIRI/413044258](https://confluence.hl7.org/spaces/FHIRI/pages/413044258/SDC+RePopulate),
the only cross-reference reported by Augury) was started by Brian
Postlethwaite to collate the design questions. Comments on the ticket
flag two follow-on design needs that any accept-as-requested wording
will have to grapple with: a way to mark items as **"do not (re-)populate"**
(important for adaptive forms where a user has explicitly cleared a
populated answer), and clarifications around the existing `context`
parameter (`context.name` cardinality, and why `context` only allows
references rather than inline resources when `local = false`). The FHIR
Infrastructure call agreed the substantive work belongs on an SDC call
rather than FHIR-I.

## Details

**Description:**
> In order to allow population logic to refer to previously provided
> questionnaire responses, the operation should have a `initialResponse`
> `QuestionnaireResponse 0..1` parameter.
>
> Use cases for this include:
> 1. DTR where the questionnaire population may be happening by the
>    provider after payer pre-population has already occurred, the
>    provider may want to use values from the provider pre-population
>    step.
> 2. Adaptive, where other questions have already been answered and may
>    drive initial values for new questions.

## Related Zulip Discussions

### [questionnaire > $populate operation in re-populate](https://chat.fhir.org/#narrow/stream/questionnaire/topic/$populate%20operation%20in%20re-populate)

This is the canonical thread for the re-populate concept and the direct
parent of FHIR-53468. It opened in **July 2024** with Vitaly Banchenko
pointing out that the SDC IG's
[`populate-operations`](https://build.fhir.org/ig/HL7/sdc/populate.html#population-operations)
section claims `$populate` can be used for "re-populate", but the
operation has no parameter to receive an existing `QuestionnaireResponse`.
Lloyd McKenzie confirmed the gap and invited a change request (which
ultimately became FHIR-53468). Paul Lynch initially questioned whether
re-populate is even desirable. Through August 2024, Brian Postlethwaite
enumerated the hard design questions: when should re-populate fire,
what to do if source data has changed or been deleted, and what to do
if the user has edited a populated answer. In November 2024, Axel
Vanraes reinforced the use case (re-populating from voice transcription
on top of an existing QR) and asked whether the feature could also
support populating a caller-supplied empty-but-metadata-rich QR. In
January 2025 Lloyd clarified the semantic preference: a separate QR
per fill-in is normally desirable so there is a snapshot per session,
but re-population is a different mode. In **December 2025** Brian
created the [SDC RePopulate](https://confluence.hl7.org/spaces/FHIRI/pages/413044258/SDC+RePopulate)
Confluence page as a working canvas, and on **2026-01-05** Brenin
Rhodes added an explanation of the DTR use cases plus concrete
proposals. On **2026-02-13** Brian posted a sample questionnaire on
fhirpath-lab and a HAPI R4 test deployment so the design can be
prototyped. **Consensus is forming around adding a parameter that
carries a prior `QuestionnaireResponse` plus a parallel mechanism to
opt items out of (re-)population**; details (parameter name, item-level
control vocabulary, semantics for changed/deleted source data) are
still open.

Participants: Vitaly Banchenko, Lloyd McKenzie, Paul Lynch, Brian
Postlethwaite, Axel Vanraes, Forest, Brenin Rhodes.

### [questionnaire > Granular Repopulate button](https://chat.fhir.org/#narrow/stream/questionnaire/topic/Granular%20Repopulate%20button)

Started by **Sean Fong (CSIRO)** on 2025-08-11 about a renderer-side
"granular re-population" UX: a button per item with an
`initialExpression` so the user decides per-question whether to refresh
the populated value. The discussion is directly relevant because the
behaviour the button drives is conceptually an item-scoped re-populate
and the same `Questionnaire`/`QuestionnaireResponse` machinery has to
support it. Lloyd cautioned that historical positions on re-population
asked for more in-production experience before standardising. Brian
proposed naming alternatives (`questionnaire-repopulatable`) and
suggested an enum of behaviours (`manual`, `never`, …) rather than a
boolean show/hide flag — a design point that converges with the "do
not populate" flag mckenzie raised on FHIR-53468. Sean iterated on the
extension URL/structure (currently a CSIRO-namespaced extension
`questionnaire-initialExpression-showRepopulateButton`) but signalled
willingness to align with whatever SDC standardises. **No formal
consensus**, but there is clear appetite for an item-level
declarative control over (re-)populate behaviour.

Participants: Sean Fong, Lloyd McKenzie, Brian Postlethwaite.

## Proposed Dispositions

The dispositions below propose changes to the **SDC IG** (HL7
specification artifact, not a GitHub repo surfaced by Augury for this
ticket). Specific authoring locations within the SDC IG should be
confirmed against the SDC build before implementation; FHIR-53468 has
no GitHub cross-references and no repo briefing is available to point
at concrete file paths.

### Disposition A: Accept As Requested

#### Proposal

Extend the SDC `Questionnaire/$populate` operation definition to add a
new IN parameter:

- **`initialResponse`** : `QuestionnaireResponse`, cardinality `0..1`,
  use = `IN`. When present, the operation MUST treat the supplied
  `QuestionnaireResponse` as the starting state of the response being
  populated. Server behaviour:
  1. Items already containing an `answer` in `initialResponse` are
     **preserved** in the returned `QuestionnaireResponse` and are NOT
     re-populated, **unless** the corresponding `Questionnaire.item`
     carries an explicit "always re-populate" marker (see below).
  2. Items present in `initialResponse` without an `answer` (i.e., the
     user has explicitly cleared them, or the structure was generated
     by a previous populate) are likewise left empty — i.e. the
     presence of the item is treated as "do not populate this slot".
  3. Items absent from `initialResponse` are populated normally from
     `initialExpression` / `initial[x]` / `launchContext`.
  4. Answers in `initialResponse` are made available to populate
     expressions via a defined FHIRPath/CQL variable (e.g.
     `%initialResponse`) so adaptive populate logic can reference
     previously-given answers.

Add a companion item-level extension on `Questionnaire.item` (e.g.
`questionnaire-repopulate-behavior`) with a code-bound `code` value of
type `code` and a small ValueSet (suggested codes: `auto` —
unconditionally re-populate, `manual` — only on explicit per-item
trigger, `never` — never overwrite once present, default `never` for
backward compatibility). Document the precedence between this
extension and `initialResponse` in the `populate-operations` section
of the SDC IG, and update that section so its existing "re-populate"
language references the new parameter explicitly.

Also resolve, in the same edit pass, the two clarity items raised by
@jmandel: (a) review whether `context.name` should be `1..1` rather
than `0..*` (or document why the current cardinality is intentional);
(b) document — or extend — the rule that `context` parameters with
`local = false` must be `Reference`s.

#### Justification

This is the literal request and matches the direction the
`$populate operation in re-populate` Zulip thread is converging
toward, including Brenin Rhodes' DTR-specific proposals on the
*SDC RePopulate* Confluence page and Sean Fong's item-level granular
re-populate work. Coupling the parameter with an item-level opt-out
addresses Lloyd McKenzie's and mckenzie's concern (2025-11-20 /
2025-12-04) that "presence of an item" is not, by itself, a reliable
"don't populate" signal — particularly for items auto-generated by a
prior `$populate` invocation. A new `0..1` IN parameter is purely
additive and cannot break existing implementations; an item-level
extension is similarly additive and aligns with SDC's general pattern
of using extensions for behavioural hints on `Questionnaire.item`
(`initialExpression`, `launchContext`, `entryMode`, etc.). Folding in
the `context.name` / `context` reference clarifications uses the same
edit cycle to address adjacent friction surfaced while reading the
populate definitions.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding `initialResponse` as a new IN parameter, formalise
**re-populate as a distinct operation** — `$repopulate` — defined in
SDC and layered on top of `$populate`:

- `Questionnaire/$repopulate` (or `QuestionnaireResponse/$repopulate`,
  invoked on the in-progress instance) takes the existing
  `QuestionnaireResponse` as the operation target (or a required IN
  parameter `questionnaireResponse : QuestionnaireResponse`), plus the
  same `subject` / `context` / `launchContext` parameters as
  `$populate`, and returns an updated `QuestionnaireResponse`.
- Re-populate semantics (preserve user-edited answers, recompute
  derivable answers, honour an item-level `repopulate-behavior`
  extension) are defined once on this operation, leaving `$populate`
  unchanged for "fresh fill" use cases.
- Adaptive use cases are addressed by a smaller, separately-scoped
  change: define a FHIRPath/CQL variable (`%response`) inside
  `initialExpression` that always resolves to the in-progress
  `QuestionnaireResponse` being built, so adaptive populate has access
  to already-given answers without needing the caller to round-trip a
  full QR back into `$populate`.

#### Justification

Splitting the operation cleanly separates two semantically different
intents: "build me a fresh QR from clinical context" vs. "merge new
clinical context into an existing QR while respecting what's already
there". That separation makes server conformance statements clearer —
an implementer can claim `$populate` support without implicitly
inheriting the harder re-populate semantics (which Lloyd flagged as
needing more production experience in the *Granular Repopulate
button* thread). It also lets the merge/preserve/overwrite policy live
in one place rather than as a conditional branch inside `$populate`,
reducing the risk of subtle interop divergence. The trade-off vs.
Disposition A is more spec surface area (a new operation plus a new
extension) and a migration story for callers who today *say* they
re-populate by re-invoking `$populate` (the IG already implies this is
allowed); those callers would be steered toward `$repopulate` over
time.

---

### Disposition C: Decline

#### Proposal

Decline to add `initialResponse` (or a `$repopulate` operation) at this
time. Instead, document in the SDC IG's `populate-operations` section
that **re-population is currently out of scope for `$populate`** and
that callers wanting to merge a populate result into an existing
`QuestionnaireResponse` are responsible for performing that merge
client-side. Optionally, add a brief implementer note pointing at the
*SDC RePopulate* Confluence page as the location where a future
mechanism is being designed, and remove or soften the existing IG
language that says `$populate` "can be used in a re-populate manner"
— since that wording is precisely what triggered Vitaly Banchenko's
2024-07-01 confusion that started the Zulip thread.

#### Justification

This was Lloyd McKenzie's stated historical position in the *Granular
Repopulate button* thread (2025-08-11): the workgroup wanted "more
in-production experience before we decided what, if anything, we
wanted to do in that space." The open design questions Brian
Postlethwaite enumerated (what to do when source data changes, when
source data is deleted, when the user has edited a populated answer,
when items were auto-generated by an earlier populate) are non-trivial
and partially overlap with item-level UX concerns — codifying them
prematurely risks an interop point the workgroup will want to
re-litigate. Declining for now keeps `$populate` minimal and cleans up
the misleading IG sentence that implied a feature the operation does
not actually support, which is itself a meaningful improvement.

---

### Recommendation

**Recommended disposition:** **A**

The change is additive (`0..1` IN parameter on an existing operation,
plus a new optional item-level extension) so it cannot break existing
implementers. The thread history shows substantive demand (DTR
provider-side re-population, adaptive populate, voice-transcription
re-population, CSIRO's granular repopulate UI) and a designer
(Brenin Rhodes) already drafting concrete proposals on the *SDC
RePopulate* Confluence page, which makes "we don't have enough
experience yet" (Disposition C) harder to defend than it was in 2024.
Disposition B — splitting `$repopulate` out as a distinct operation —
is conceptually cleaner, but in practice the merge/preserve/overwrite
rules required for safe re-populate also have to be specified for
adaptive use of `$populate` (where new items appear over the lifetime
of one fill-in and need access to existing answers); putting both
behind one operation parameterised by `initialResponse` keeps the
specification surface smaller and avoids two near-duplicate operation
definitions. The recommendation is contingent on the SDC call also
landing the item-level "do not (re-)populate" extension in the same
ballot cycle — without it, the ambiguity Lloyd McKenzie raised about
"presence of an item" semantics will produce the very interop
divergence Disposition C is worried about.
