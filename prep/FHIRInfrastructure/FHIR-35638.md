## Summary

The reporter (Adam Strickland) asks the SMART on FHIR work group to relax the
conformance rule that obligates a SMART resource server to publish **both**
`jwks_uri` and `issuer` in its `smart-configuration` discovery document when it
advertises the `sso-openid-connect` capability. He proposes that only `issuer`
be required, on the grounds that the OpenID Connect well-known metadata
retrieved from `issuer` already contains the required `jwks_uri`. Mandating
both values, he argues, makes `jwks_uri` duplicative and forces resource
servers to mirror — and stay in sync with — a security-sensitive value owned
by their authorization server. His concrete failure scenario: the auth server
rotates its key-set URL, the resource server has not been updated, and clients
are misled by a stale `jwks_uri` advertised in `smart-configuration`.

There are **no cross-referenced Jira tickets** in the Augury index, so this
review is scoped to the original ticket and the on-record discussion.

## Details

**Description:**

> The current smart spec requires resource servers to broadcast both
> "jwks_uri" and "issuer" data points if it broadcasts the "sso-openid-connect"
> capability.
>
> I would argue that only "issuer" should be required, since the retrieving
> that issuer's Openid Connect metadata would contain the required "jwks_uri"
> field. Currently this is a duplicative field, and puts extra burden on
> resource servers to keep up with the security of their authorization
> servers.
>
> For example, if the authorization server changes their key set URL, the
> resource server may not know that for some period of time and falsely
> broadcast a jwks_uri to it's clients.

**Comments:**

- **JIRAUSER20113 (2022-01-11):** Notes that `smart-configuration` already
  contains several URLs in addition to `jwks_uri` (e.g. `authorize` /
  `token` endpoints). For consistency, if the work group drops `jwks_uri`,
  it should also consider dropping the other URLs and relying solely on the
  authorization server's well-known endpoints.
- **jmandel — Josh Mandel (2022-09-18):** Argues against the change. Servers
  MAY advertise `jwks_uri` even when they do not support OIDC, so a
  consistent property in the SMART discovery document lets clients find the
  key set directly without an extra dereference. Frames it as a design
  principle: simplify life for clients as long as the obligation on servers
  remains manageable.

## Proposed Dispositions

Note on repo grounding: because no GitHub cross-references were surfaced,
no briefing-backed file paths can be cited. The dispositions below name the
SMART App Launch publication artefact (`smart-configuration` discovery
document, defined on `conformance.html`) by content, which is the only
artefact the ticket touches.

### Disposition A: Accept As Requested

#### Proposal

Amend the SMART App Launch conformance rules for the `smart-configuration`
discovery document so that, when `sso-openid-connect` is advertised in
`capabilities`:

- `issuer` **remains REQUIRED**.
- `jwks_uri` is downgraded from REQUIRED to OPTIONAL, with normative text
  stating that, when `jwks_uri` is omitted, clients MUST resolve the
  authorization server's `issuer` + `/.well-known/openid-configuration` and
  use the `jwks_uri` advertised there.
- Add a SHOULD-level recommendation that, if a server does publish
  `jwks_uri` directly in `smart-configuration`, it MUST be the same value
  currently published by the OIDC discovery document and the server is
  responsible for keeping the two in sync.

#### Justification

This directly addresses the reporter's stated concern: a resource server
that mirrors the authorization server's `jwks_uri` accepts an availability
and freshness obligation it does not really own, and the failure mode
(stale key URL after auth-server rotation) is a real security/availability
risk. OIDC discovery already defines `jwks_uri` as REQUIRED in the
issuer's well-known metadata, so the data is never lost — only its
location is delegated to the canonical owner. This aligns with the general
FHIR principle of avoiding duplicate normative data.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `jwks_uri` REQUIRED in `smart-configuration` when `sso-openid-connect`
is advertised, but add normative guidance that addresses the reporter's
underlying concern without changing cardinality:

- Add a normative statement that, when `sso-openid-connect` is advertised,
  the `jwks_uri` published in `smart-configuration` MUST equal the
  `jwks_uri` published in the issuer's OIDC discovery document at the
  point of publication.
- Add a SHOULD that resource servers fetch the OIDC discovery document
  dynamically (or cache it with a short TTL) when constructing
  `smart-configuration`, rather than hard-coding the key URL.
- Add a non-normative implementer note describing the key-rotation failure
  mode and recommending operational practices (e.g., propagate auth-server
  metadata updates to all dependent resource servers; monitor for drift).

Optionally, in line with the comment from JIRAUSER20113, open a separate
follow-up ticket to evaluate whether the other auth-server-owned URLs in
`smart-configuration` (`authorization_endpoint`, `token_endpoint`, …)
deserve the same treatment, so the work group does not address `jwks_uri`
in isolation.

#### Justification

This preserves the client-side ergonomic that Josh Mandel calls out in his
comment — clients (including non-OIDC clients that still want to verify
JWTs) get `jwks_uri` from a single, consistent property in the SMART
discovery document and avoid a second well-known dereference. It still
removes the silent-divergence failure mode the reporter is worried about,
because the spec now obliges the resource server to source the value from
the canonical owner rather than hard-code it. The trade-off versus
Disposition A is that the operational burden on the resource server is
reduced but not eliminated; the upside is full backward compatibility for
existing clients.

---

### Disposition C: Decline

#### Proposal

Decline the change as out of scope for a Technical Correction and close
the ticket as "Persuasive with mod" of guidance only (or "Not Persuasive"
if no guidance is added). Rationale category: existing mechanism already
addresses the need.

#### Justification

`smart-configuration` is published intentionally as a self-contained
discovery document so a SMART client can obtain everything it needs in a
single request. Making `jwks_uri` optional would force every SMART client
that verifies ID tokens to either (a) implement OIDC discovery as a
fallback or (b) fail against compliant servers that omit the field —
shifting the burden Adam describes from the resource server onto every
client implementer, where the cost multiplies. Furthermore, OIDC discovery
permits servers to advertise `jwks_uri` even without OIDC support, which
would be impossible to express if the field were tied solely to the
`issuer` round-trip. A Technical Correction is intended for unambiguous
errors in the spec; this is a design trade-off and does not meet that
bar.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible compromise. It directly addresses the
reporter's failure scenario — `jwks_uri` drifting out of sync with the
authorization server after a key rotation — by making the source of truth
explicit and binding the SMART discovery document to the OIDC discovery
document. At the same time, it preserves the only on-record counter-argument
in the discussion (Josh Mandel's point that a single, consistent
`jwks_uri` property simplifies client implementations, including non-OIDC
clients that still verify JWTs) and avoids forcing every existing SMART
client to implement OIDC discovery as a fallback path. Disposition A is
cleaner from a "no duplicative data" standpoint but breaks that client
ergonomic and would require coordinated change across the implementer
ecosystem. Disposition C ignores a legitimate operational risk that the
reporter has documented with a concrete failure mode. The work group
should also seriously consider opening the follow-up ticket suggested in
the 2022-01-11 comment to look at the other auth-server-owned URLs in
`smart-configuration` together, so this conformance question is resolved
once for the whole document rather than per-field.
