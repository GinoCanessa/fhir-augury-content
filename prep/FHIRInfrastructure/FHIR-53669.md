## Summary

FHIR-53669 reports a small but real documentation defect in the FHIR Core
specification's description of the `id` primitive type. The reporter
points out an unmatched opening parenthesis in the ASCII-letter clause:
the description currently reads "Any combination of upper- or lower-case
ASCII letters ('A'..'Z', and 'a'..'z', numerals ('0'..'9'), '-' and '.', …"
— the parenthesis opened before `'A'..'Z'` is never closed, which makes
the sentence ambiguous on first read. The ticket proposes inserting the
missing closing parenthesis after `'a'..'z'`. The workgroup has already
flagged the resolution as **Persuasive**, indicating editorial agreement
that the text should be repaired. No related Jira tickets are linked;
the only keyword-similar historical item is
[FHIR-3088](https://jira.hl7.org/browse/FHIR-3088) ("Definition of id
data type"), an old change request that established the current wording
of the `id` description and provides background on how this clause came
to read the way it does today.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the exact edit specified in the ticket's Resolution Description.
In the FHIR Core specification source for the `id` primitive type
(rendered at `datatypes.html#id`, sourced from the `HL7/fhir` build
repository), replace:

> Any combination of upper- or lower-case ASCII letters ('A'..'Z', and
> 'a'..'z', numerals ('0'..'9'), '-' and '.', with a length limit of 64
> characters. (This might be an integer, an un-prefixed OID, UUID or any
> other identifier pattern that meets these constraints.)

with:

> Any combination of upper- or lower-case ASCII letters ('A'..'Z', and
> 'a'..'z'**)**, numerals ('0'..'9'), '-' and '.', with a length limit
> of 64 characters. (This might be an integer, an un-prefixed OID, UUID
> or any other identifier pattern that meets these constraints.)

i.e., insert the missing `)` after `'a'..'z'`. This is a
documentation-only edit; no schema, StructureDefinition, regex, or
profile change is implied or required.

#### Justification

This is a literal punctuation defect: the opening parenthesis before
`'A'..'Z'` is never closed, which makes the enumeration of allowed
character classes (letters, numerals, `-`, `.`) read as if "numerals
('0'..'9')" is parenthetically nested inside the letter clause. The
fix restores standard mathematical/grammatical balancing of
parentheses without changing the normative meaning of the description
in any way. The `id` regex itself (`[A-Za-z0-9\-\.]{1,64}`) and the
length limit are untouched, so there is zero downstream impact on
implementers, validators, or generated artifacts. The workgroup has
already marked the resolution as **Persuasive**, which signals
agreement with this approach.

---

### Disposition B: Alternative Approach — Rewrite the Sentence for Clarity

#### Proposal

Rather than only inserting the missing parenthesis, take the
opportunity to rewrite the sentence so the allowed character classes
are listed without nested parentheticals. For example:

> Any combination of upper- or lower-case ASCII letters (`A`–`Z`,
> `a`–`z`), digits (`0`–`9`), `-`, and `.`, with a length limit of 64
> characters. This may be an integer, an un-prefixed OID, UUID, or any
> other identifier pattern that meets these constraints.

This addresses the same defect, but also (a) eliminates the second
parenthetical clause that triggered the original confusion, (b) uses
en-dashes for character ranges (a more conventional notation), and (c)
promotes the examples ("This may be an integer …") to a separate
sentence rather than a nested parenthetical.

#### Justification

The reporter's complaint is technically about a missing `)`, but the
underlying issue is that the sentence stacks parentheticals (one for
letter ranges, one for digit ranges, and one for clarifying examples),
which is what made the imbalance hard to spot in the first place. A
light editorial pass costs almost nothing and reduces the chance that
a future reader files the same ticket again. The trade-off is that
this is slightly more invasive than a one-character insertion: it
changes wording that has been stable since FHIR-3088 was resolved in
2018, so reviewers may prefer the surgical Disposition A simply to
minimize churn against translations and downstream re-publications.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" / leave the description as-is on the
basis that the existing text, while parenthetically unbalanced, is not
ambiguous about the actual character set permitted in an `id` (which is
defined normatively by the regex `[A-Za-z0-9\-\.]{1,64}` and by the
explicit length statement, not by the prose). No spec change.

#### Justification

The normative content of the `id` definition lives in the regex and
the length constraint, not in the descriptive prose. A purely cosmetic
punctuation issue in non-normative descriptive text could be argued to
not justify a published-spec edit. This is the weakest of the three
dispositions because (1) the workgroup has already indicated the
resolution is Persuasive, (2) the cost of fixing is essentially zero,
and (3) doing nothing leaves a known confusing typo in a high-traffic
documentation page. Listed for completeness only.

---

### Recommendation

**Recommended disposition: A (Accept As Requested).**

The ticket is a clean, narrowly-scoped editorial bug report; the
reporter has even drafted the exact replacement text, and the workgroup
has already flagged the resolution as Persuasive. Disposition A is the
minimum-risk way to land the fix: a single-character insertion that
restores parenthesis balance without touching any normative content,
without affecting the `id` regex, and without disturbing translations
or downstream re-publications more than necessary. Disposition B is
attractive in principle but trades a ~zero-risk edit for a wording
change against text that has been stable since 2018 — that's not
worth the additional review burden for a documentation-only fix unless
the workgroup independently wants to revisit the prose. Disposition C
is not defensible given that the typo was confusing enough for an
external reader to file a ticket and the cost of correction is
negligible.
