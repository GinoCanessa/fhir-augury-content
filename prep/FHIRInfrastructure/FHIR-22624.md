## Summary

The ticket asks SDC to define a set of extensions on `Questionnaire.item`
that let a form author recommend how a renderer should lay out a question
(or group of questions) as a two-dimensional grid. The reporter's example
proposes nested extensions modelled on HTML tables —
`html-table` containing `html-table-column` and `html-table-row` with
`unsignedInt` values that pin a question to a specific row/column position.
The intent is that authors get explicit control over visual placement
beyond the existing top-to-bottom, one-question-per-line default.

The ticket originally landed in FHIR Core, was moved to SDC by Rob Hausam
in 2019, and was voted "Considered for Future Use" (7-0-0, Lynch /
Postlethwaite) on 2019-10-10 with the note that the WG liked the idea but
wanted connectathon experimentation around group-level declarations,
fixed vs dynamic row/column counts, and span behaviour before standardising.
Lynn Laakso re-opened the ticket post-STU4 (2026-03-26) so SDC can revisit
it for the next release. There are no explicitly-linked Jira tickets and no
GitHub cross-references; the only structured cross-reference is to the
`Questionnaire.item` element itself.

## Details

**Description:**

> To define a set of extensions to propose a recommended way to render a
> resource (e.g., Questionnaire).
>
> For example, a *Questionnaire.item* element may have the following
> extensions defined by the form creator to propose how questions may be
> rendered by the application and presented to the end-user:
>
> ```xml
> <extension url="http://hl7.org/fhir/StructureDefinition/html-table">
>   <extension url="http://hl7.org/fhir/StructureDefinition/html-table-column">
>     <valueUnsignedInt value="1"/>
>   </extension>
>   <extension url="http://hl7.org/fhir/StructureDefinition/html-table-row">
>     <valueUnsignedInt value="2"/>
>   </extension>
> </extension>
> ```

## Related Zulip Discussions

### [questionnaire > Implementing Grid Layout for Questions in Android FHIR SDK](https://chat.fhir.org/#narrow/stream/questionnaire/topic/Implementing%20Grid%20Layout%20for%20Questions%20in%20Android%20FHIR%20SDK)

A 2025-01 thread initiated by santosh pingle (Google / Android FHIR SDK)
asking how to render multiple Questionnaire items per line, with the
column count specified in the questionnaire. Paul Lynch suggested
extending the existing `column-count` extension (currently used on answer
lists) to be valid on group items; Fikri Milano countered with the
existing `orientation` extension applied to nested groups (horizontal
parent, vertical children) as a flexible alternative. Brian Postlethwaite
warned that whatever is chosen is "still just a hint" and must degrade
gracefully across phone, tablet and desktop, and predicted the inevitable
follow-on request for row/column span. Jing Tang clarified the canonical
column-count extension URL
(`http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-questionnaire-columnCount`)
and noted the use case is specifically tablets. santosh pingle posted a
worked example showing `columnCount` on a group-type item with three
child items, and Vitaly Banchenko described their internal
`questionnaire-itemColumnSize` extension that treats a questionnaire /
group as a CSS-style 12-column grid where each item declares a 0–12
size. Vitaly indicated they intend to file a feature request to the SDC
IG once their implementation matures. No formal consensus was reached,
but two converging directions emerged: (1) extend `columnCount` to apply
on `group` items and (2) optionally add a per-item column-size /
flow-control hint.

Participants: santosh pingle, Forest, Benjamin Mwalimu, Japheth
Kiprotich, Paul Lynch, Fikri Milano, Brian Postlethwaite, Jing Tang,

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define three new SDC extensions modelled on the reporter's HTML-table
analogy, scoped to `Questionnaire.item`:

- `sdc-questionnaire-grid` (complex extension, allowed on
  `Questionnaire.item` of `type = group`) — declares that the group is
  rendered as a fixed grid. Sub-extensions: `rowCount` (`positiveInt`,
  optional) and `columnCount` (`positiveInt`, required).
- `sdc-questionnaire-gridPosition` (complex extension, allowed on any
  `Questionnaire.item` whose parent group carries `sdc-questionnaire-grid`)
  with sub-extensions `row` (`positiveInt`) and `column` (`positiveInt`),
  both 1-based, identifying the cell the item occupies.
- `sdc-questionnaire-gridSpan` (complex extension, allowed alongside
  `gridPosition`) with sub-extensions `rowSpan` and `columnSpan`
  (`positiveInt`, default 1) to let a single item span multiple cells —
  the row/col-span feature Brian Postlethwaite predicted on Zulip.

Publish all three under the SDC IG canonical
(`http://hl7.org/fhir/uv/sdc/StructureDefinition/...`), not under the
core `http://hl7.org/fhir/StructureDefinition/` namespace the original
ticket showed (per Rob Hausam's 2019 comment). Document them as
*rendering hints* — renderers MAY ignore them or fall back to the
default vertical layout when the target form factor cannot honour them.

#### Justification

This delivers exactly what the ticket asks for and gives form authors
deterministic placement, which the existing `columnCount` /
`itemColumnSize` machinery does not. Modelling it on a fixed grid with
explicit `(row, column)` coordinates and an explicit span sub-extension
covers the connectathon questions raised in the 2019-10-10 vote (group-
level declaration, advance declaration of row/column counts, and
spanning) in a single coherent design. Re-homing the URLs under the SDC
canonical is consistent with how the WG already publishes
`columnCount`, `choiceColumn`, and `orientation`, and avoids polluting
core.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of introducing a new fixed-grid coordinate system, extend the
existing SDC presentation extensions to cover the use case:

1. Broaden the context of
   `http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-questionnaire-columnCount`
   so it is allowed on `Questionnaire.item` of `type = group` (today it
   is scoped to answer-option rendering).
2. Add a lightweight, flow-based companion extension
   `sdc-questionnaire-itemColumnSize` (`integer`, 1–12), defined on
   `Questionnaire.item`, that lets an item declare how many columns of
   its parent group's grid it occupies — borrowing directly from the
   `itemColumnSize` design Vitaly Banchenko described on Zulip and from
   the Bootstrap / CSS-grid 12-column convention. Items flow
   left-to-right, top-to-bottom into the parent's grid; `itemColumnSize`
   replaces explicit `(row, column)` coordinates.
3. Keep the existing `orientation` extension for the simpler "lay these
   children horizontally" case, and document the interaction.

#### Justification

Layout on a Questionnaire is, as Brian Postlethwaite observed, only ever
a hint — fixed row/column coordinates are brittle when the same form has
to render on a phone, tablet and desktop, and they make it hard for
renderers to gracefully reflow. A flow-based 12-column model degrades
naturally (a renderer with a narrow viewport can collapse `itemColumnSize`
to 12 and stack vertically) and is what the implementer community on
Zulip has converged on independently. It also reuses two extensions
(`columnCount`, `orientation`) that already exist and are understood,
reducing the standards surface the WG has to maintain.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — Existing Mechanism**. Direct form
authors to the existing SDC rendering extensions
(`sdc-questionnaire-columnCount`,
`sdc-questionnaire-choiceOrientation` /
`sdc-questionnaire-itemControl`) plus standard `text` /
`displayCategory` / Item Control coding for layout intent. If a
particular renderer needs richer placement, it can use a profile or its
own extensions until a clear cross-implementation requirement emerges.

#### Justification

The 2019-10-10 vote already concluded the WG liked the idea "in theory"
but wanted connectathon evidence before standardising — and seven years
later that evidence is still anecdotal: the only concrete implementer
work surfaced (Vitaly Banchenko's `itemColumnSize`) is private and the
author themselves said they will file a feature request once the design
proves out. Standardising a fixed-grid placement model now risks
locking SDC into the original ticket's HTML-table metaphor, which the
broader community has already moved away from in favour of
flow-and-column-size models. Declining preserves authoring freedom via
implementer-defined extensions until a more mature proposal arrives.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle the WG identified back in 2019 — it
addresses the underlying need (multi-column layout of Questionnaire
items) and the concrete connectathon questions (group-level declaration,
spanning, responsive degradation) without introducing a brand-new
coordinate-grid mechanism that overlaps with `columnCount`,
`orientation`, and `itemControl`. It also aligns with the only
real-world implementer signal we have: the Zulip thread shows the
Android FHIR SDK community already gravitating toward `columnCount` on
groups plus a per-item size hint, and Vitaly Banchenko has explicitly
flagged intent to file an SDC feature request along those lines.
Disposition A is more deterministic but harder for renderers to honour
across form factors and re-litigates ground the implementer community
has already covered. Disposition C is defensible but throws away seven
years of accumulated implementer interest and the open Zulip momentum.
B is the disposition most likely to ship in the next SDC release with
implementer buy-in.
