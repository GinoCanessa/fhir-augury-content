## Summary

The reporter asks that the SMART on FHIR AppState (patient-specific) scope
examples be updated so they exercise the resource-specific `patient` search
parameter in addition to the generic `Reference`-typed `subject` search
parameter. Today the example queries narrow AppState requests to a given
patient using `Basic?subject=Patient/a&code=…`; the ticket suggests showing
the equivalent `Basic?patient=Patient/a&code=…` form alongside (or in place
of) it, so implementers see both idioms and pick the one that matches how
they have parameterised the `Basic` (or other AppState-bearing) resource.

The ticket was previously voted "Considered for Future Use" (Bas van den
Heuvel / Yunwei Wang, 13-0-0) on the rationale that scope grants don't
typically include patient-scoping query parameters because the patient ID is
not always known at scope-request time. That resolution was **reverted and
the ticket re-opened post-STU2.1 publication**, so the work group now needs
to reconsider it. There are no Jira links, no Zulip cross-references, and
no GitHub items associated with the ticket in the augury index, so the
disposition rests on the ticket text and prior voting history alone.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the SMART on FHIR AppState section (the patient-specific scope
example block in the SMART App Launch IG, where AppState scopes are
described) so each example query is presented in **both** forms:

- `Basic?subject=Patient/a&code=https://app|1` *(generic Reference-typed
  search parameter)*
- `Basic?patient=Patient/a&code=https://app|1` *(resource-specific patient
  compartment search parameter)*

Add one sentence of narrative immediately above the example list noting
that AppState resources whose patient compartment is expressed via a
typed `patient` search parameter (e.g., conformant
`SearchParameter` definitions on the chosen AppState resource) MAY use
that parameter in scope-restricting query strings, and that the two
forms are equivalent for `Basic` because both resolve to the same
`Patient` reference. No normative statement is changed; only example
syntax and a clarifying note are added.

#### Justification

This is a documentation-only change to example queries that does not
alter SMART scope grammar, server obligations, or the meaning of any
existing scope. It addresses the reporter's concern that implementers
who model AppState with the resource-specific `patient` search parameter
have no example to copy from, while preserving the existing
`subject`-based form for parity with the generic Reference search
parameter pattern. Because both `subject` and `patient` are defined on
`Basic` (and on most patient-scoped resources), showing both forms
mirrors the dual-parameter convention used elsewhere in the FHIR core
spec and reduces the chance that an implementer infers `subject` is
the only valid form.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than duplicating every example, **rewrite the AppState scope
example to use a single canonical form plus an explanatory note**.
Concretely:

- Keep one example per scenario (preferring `patient=…` because it is
  the FHIR-canonical patient-compartment parameter) and remove the
  duplicate `subject=…` form.
- Add a short paragraph stating that any `SearchParameter` defined on
  the AppState resource that resolves the patient compartment MAY be
  used in the scope query string, and explicitly call out `subject`
  (for resources that use it generically) and `patient` (for resources
  with a typed patient parameter) as the two common choices.
- Optionally cross-link to the Patient Compartment definition so
  implementers can identify which parameter applies to a given
  AppState resource type.

#### Justification

This avoids visual clutter from listing equivalent forms for every
example and educates the implementer on the underlying rule (any
patient-compartment search parameter is valid) rather than enumerating
syntactic variants. It is more durable: if the work group later
recommends a different AppState resource (or implementers profile their
own), the guidance still applies without needing more examples. Trade-off
vs. Disposition A: implementers who copy/paste blindly may miss the
note; partly offset by keeping a worked example for the most common
parameter (`patient`).

---

### Disposition C: Decline (Not Persuasive — No Spec Change)

#### Proposal

Re-affirm the original *Considered for Future Use / no change required*
resolution. Close FHIR-39700 with a short statement that scope syntax
permits any valid search parameter on the named resource and that
example queries showing `subject=…` are illustrative only, not
restrictive. Capture the rationale from the prior 13-0-0 vote in the
resolution comment so the decision is traceable.

#### Justification

The original committee discussion concluded that the scope grammar
already allows any search parameter the AppState resource supports, so
no normative change is needed; the reverted resolution was driven by a
desire to revisit examples post-STU2.1, not by new technical evidence.
Declining is defensible if the work group judges that the existing
example is unambiguous and that an implementer reading the SMART scope
grammar (which permits arbitrary search-parameter constraints) would
already understand that `patient=…` is equally valid. The risk is that
implementers who only read the example — not the grammar — continue to
write `subject=…` queries that don't match their AppState
`SearchParameter` definitions.

---

### Recommendation

**Recommended disposition:** B

The ticket is fundamentally an examples / clarity issue, not a
grammar issue, and the prior 13-0-0 vote correctly identified that
nothing normative needs to change. However, the fact that the ticket
was re-opened post-STU2.1 indicates the work group accepts that the
examples as written are misleading — otherwise the simple "no change
required" close would have stood. Disposition B addresses the
underlying confusion (implementers don't realise any
patient-compartment search parameter is valid) without committing the
spec to maintaining parallel example pairs forever, and it scales to
future AppState resource choices. Disposition A is the safer, lower-risk
edit if the work group prefers minimal change; Disposition C should
only be selected if the group is willing to formally state that the
existing single-example form is sufficient and re-close with prejudice.
