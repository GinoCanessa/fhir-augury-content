## Summary

The ticket targets the prose in the FHIR Core CodeableConcept datatype
description that explains why a `CodeableConcept` may carry more than one
`coding`. The reporter (Carmela A. Couderc) flags the existing sentence —
"The codes in the codings may have different granularities, whether they
are from different code systems, or the same code system." — as potentially
misleading, on the grounds that the word "granularities" may be read as
implying a subsumption (broader/narrower) relationship between the codings
within a single `CodeableConcept`. The proposed replacement is a
normative-style statement that **"Each coding MUST be either equivalent,
narrower or broader than other codings in the codeableConcept,"** with the
stated rationale that this both removes the ambiguous "granularity" wording
and provides a smoother lead-in to the next paragraph in the datatype
description.

No Jira cross-references were surfaced for this ticket. There is, however,
a directly relevant prior community discussion on the FHIR Zulip
`#terminology` stream ("Multiple codes in CodeableConcept", April–June
2025) in which Grahame Grieve, Rob Hausam, Lloyd McKenzie, Jose Costa
Teixeira, Frank Oemig, Robert McClure and others debated exactly this
sentence; the consensus position from that thread is **less restrictive**
than what the ticket proposes (intersection-of-extents, not strict
subsumption — see Related Zulip Discussions below).

## Related Zulip Discussions

### [terminology > Multiple codes in CodeableConcept](https://chat.fhir.org/#narrow/stream/terminology/topic/Multiple%20codes%20in%20CodeableConcept)

A long, multi-month thread (earliest captured message 2025-04-02; latest
captured 2025-06-04; ~100 captured messages) that debates exactly the
sentence FHIR-53822 wants to change. Grahame Grieve — author of the
original wording — articulates the rule as "**the intersection of the sets
of things described by the codings must be non-null**", explicitly
stating: "legal is any amount of granularity difference as long as the
intersection is not empty; good practice is not to push the envelope
here." Rob Hausam agrees the description "around 'slightly different
granularity'" needs to be updated and that "slightly" causes confusion.
Several participants (Jose Costa Teixeira, Frank Oemig, Daniel Venton,
Robert McClure) push for tighter normative language (e.g., "the code
SHOULD be the most specific known, allowed by the value set"), but
Grahame is explicit that any such tightening "has to be a SHOULD or a
best practice, not a SHALL." The thread also surfaces a meaningful
counter-case — `doNotPerform` and other negation contexts, where the
*broadest*-scope code is the safest interpretation rather than the
narrowest — which complicates any blanket "narrower/broader/equivalent"
constraint. Lloyd McKenzie, David Pyke, Rik Smithies, Michael Lawley,
Joshua Wiedekopf and Uri Lerner contribute additional nuance about
mixing value-set-bound codes with translations and about partially
non-conformant data. **No firm consensus on replacement wording was
reached**, but the dominant position is to soften/clarify rather than
tighten the rule, and to explicitly avoid mandating subsumption.

Participants: Grahame Grieve, Lloyd McKenzie, Jose Costa Teixeira, Frank
Oemig, Robert McClure, Daniel Venton, Rik Smithies, David Pyke, Rob
Hausam, Uri Lerner, Michael Lawley, Rutt Lindström, Joshua Wiedekopf.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the current sentence in the CodeableConcept datatype narrative
(`datatypes.html#CodeableConcept`, paragraph that today reads "The codes
in the codings may have different granularities, whether they are from
different code systems, or the same code system.") with the reporter's
text: **"Each coding MUST be either equivalent, narrower or broader than
other codings in the codeableConcept."** Apply this in the FHIR Core
specification source for the CodeableConcept datatype description and
flow through to any cross-referenced terminology guidance. Add a brief
follow-up sentence noting that the codings may come from different code
systems or the same code system, to preserve the information that the
current sentence carries.

#### Justification

It directly answers the reporter's stated concern by removing the
"granularity" wording. It is short, declarative, and (as the reporter
notes) flows into the next paragraph. For implementers who have always
read the existing sentence as implying a subsumption-style relationship,
this codifies that reading.

### Disposition B: Alternative Approach

#### Proposal

Adopt clarifying — but **non-normative** — wording instead of an upper-case
MUST. Replace the sentence with something like:

> "Each `coding` represents the same concept; codings may differ in
> granularity (one being broader or narrower than another) or simply
> overlap. It is required that the sets of things described by the
> codings have a non-empty intersection — i.e., they all be
> simultaneously true of the concept being represented. Codings may come
> from different code systems or from the same code system. Where
> possible, the most specific code allowed by any applicable value-set
> binding SHOULD be sent."

This both eliminates the bare "granularity" phrasing the reporter objects
to, and aligns the description with the rule Grahame Grieve articulated
in the `#terminology` thread (intersection-of-extents) plus the
"most-specific-SHOULD" best-practice the thread coalesced around.

#### Justification

The reporter's proposed wording (Disposition A) is **stricter** than the
rule the editors describe in chat: "equivalent / narrower / broader" is a
subsumption requirement, but the FHIR rule (per Grahame) only requires
non-empty intersection of the codes' extents — two codings from different
classifications can validly overlap without one subsuming the other (e.g.,
a brand-name product code and a therapeutic-class code, or a SNOMED CT
finding and an ICD-10 code that share a non-empty patient population but
neither subsumes the other). Disposition B preserves that flexibility,
explicitly addresses negation/`doNotPerform` cases by talking about
"simultaneously true," avoids elevating prose to a SHALL/MUST that has
not been agreed in the workgroup, and incorporates the best-practice
guidance ("most specific known, allowed by the value set") that the
thread reached rough agreement on. It also avoids upgrading a long-
standing description to normative status (a change with implementer
impact) without a corresponding invariant or test artifact.

### Disposition C: Decline

#### Proposal

Decline the proposed text. Optionally, as a minimal editorial touch, drop
the word "slightly" if it appears in the surrounding paragraph (per Rob
Hausam's suggestion in the thread) but otherwise leave the existing
sentence about granularity in place.

#### Justification

The proposed wording would convert long-standing descriptive prose into
a normative MUST that is **more restrictive** than the rule the
specification editors and the `#terminology` workgroup discussion
actually intend (intersection-of-extents, not subsumption). Adopting it
risks invalidating real-world `CodeableConcept` instances that pair
overlapping-but-non-subsuming codes (e.g., a clinical code from one code
system plus a billing/classification code from another), and it conflicts
with the negation/`doNotPerform` semantics raised by Jose Costa Teixeira
and Grahame in the thread. Because no consensus emerged on alternative
normative wording, deferring is defensible: the existing prose has
shipped through multiple ballots without an invariant attached, and
re-opening normative status of CodeableConcept warrants more than a
prose edit.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern is valid — the word "granularity" in the
current sentence does invite a subsumption reading, and the FHIR
Infrastructure work group should clean that up. However, the literal text
the reporter proposes (Disposition A) accidentally enshrines the very
subsumption requirement it is meant to clarify away, and it does so as a
normative MUST. The directly relevant `#terminology` Zulip thread shows
the editors' intended rule is the more permissive "intersection of
extents must be non-empty," with "most specific code SHOULD be sent" as
a best practice (not a SHALL). Disposition B captures both points,
removes the problematic word, addresses the `doNotPerform`/negation
edge-case the community raised, and preserves backward compatibility for
implementations that legitimately mix codes from different
classifications. It also avoids the implementer-impact of converting a
description sentence into a hard normative constraint without a
corresponding invariant. The work group can refine the exact wording, but
the disposition direction should be "clarify, don't tighten."
