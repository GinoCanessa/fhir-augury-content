## Summary

The reporter argues that `Subscription.filterBy.comparator` is redundant
with `Subscription.filterBy.value` and should be removed. Two pieces of
evidence are offered: (1) the long description of `filterBy.value`
already shows comparator‑prefixed example values such as `le1950`,
indicating the value field is already expected to carry the search‑style
prefix; and (2) `filterBy.comparator` has cardinality 0..1, which cannot
express common "or logic" filters such as `lt50,gt100` that are routinely
needed (for example, when subscribing to extreme values). The proposed
fix is to drop the separate `comparator` element and let
`filterBy.value` carry the entire search‑parameter value, including any
prefix, exactly as it would be sent on a REST search query string. No
related Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-references; the only cross-references are to the two FHIR
artifacts directly named in the description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the `Subscription.filterBy.comparator` element entirely and
re‑specify `Subscription.filterBy.value` as the literal search‑parameter
value string, including any search prefix(es) and any
comma‑delimited "or" list — i.e., the exact value that would appear on
the right‑hand side of a search query string for the named
`filterParameter`. Update the element's short and definition text to say
so explicitly (e.g., "The value as it would appear on a REST search
query string, including any prefix such as `lt`, `gt`, `le`, `ge`, `eq`,
`ne`, `sa`, `eb`, or `ap`, and supporting comma‑delimited 'or'
values"). Mark the change as breaking for R6 / next normative ballot of
the Subscription resource and call it out in the FHIR Infrastructure
substantive change list.

#### Justification

The element's own long definition already documents prefix‑embedded
values (`le1950`), so today the spec is internally inconsistent: the
schema says "use `comparator` plus `value`" while the prose says "put
the prefix in `value`". Removing `comparator` resolves that
inconsistency in favour of the form that mirrors the REST search API,
which is the model implementers already understand. It also restores
parity with FHIR search semantics for "or" logic (`,`‑separated
alternatives), which a 0..1 `comparator` cannot express — the reporter's
`lt50,gt100` example is a legitimate clinical alerting pattern that the
current schema cannot carry without losing fidelity. Aligning
`filterBy.value` with the search‑parameter value grammar also makes
`SubscriptionTopic.canFilterBy.comparator` / `.modifier` the single
place where allowed prefixes/modifiers are advertised, rather than
duplicating that information across `Subscription` and
`SubscriptionTopic`.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Subscription.filterBy.comparator` but raise its cardinality to
`0..*`, and clarify that when `comparator` is populated the prefix
**must not** also appear inside `filterBy.value`. Tighten the
`filterBy.value` definition to remove the prefix‑embedded examples
(`le1950`) and instead say "the value only, without any search prefix —
prefixes are conveyed via `comparator`". Update the FHIRPath
invariant(s) on `Subscription.filterBy` to forbid mixing the two styles
(e.g., reject a `value` that starts with a recognised prefix when
`comparator` is also present). When multiple `comparator` values are
supplied, define their semantics as "or" against the same `value`
(matching the search grammar's `,`‑separated alternative semantics on
the prefix).

#### Justification

This addresses the reporter's two concerns (the inconsistency in the
definition and the inability to express "or" logic) without removing a
distinct element that some implementers may already be using. Keeping
`comparator` as a structured field has analytical value — servers and
auditors can inspect prefix usage without parsing free‑text values, and
profile authors can constrain allowed prefixes via `comparator`'s value
set without writing string‑pattern invariants on `value`. The trade‑off
is greater implementation surface (two ways to carry the same
information at the wire level even with the new invariant) and the need
for clear migration guidance, which is more disruptive than Disposition
A if no implementers have actually adopted `comparator` yet.

---

### Disposition C: Decline

#### Proposal

Decline to remove or restructure `filterBy.comparator`. Instead, treat
this as an editorial / clarification ticket: update the definition of
`filterBy.value` so its examples no longer include a prefix
(`le1950` → `1950`), and add a note to `filterBy` stating that prefixes
are carried by `comparator` and that the current model intentionally
supports only a single comparator per `filterBy` entry. Implementers
needing "or logic" across comparators are directed to use multiple
`filterBy` entries (one per comparator) under the same
`filterParameter`.

#### Justification

`Subscription.filterBy` is structured deliberately so that each
constraint is machine‑inspectable; collapsing comparator and value into
a single string re‑introduces parsing burden on every server. The
current model already supports the "extreme values" example via two
`filterBy` repetitions (`filterBy[0] = {param, lt, 50}` and
`filterBy[1] = {param, gt, 100}`), provided the topic semantics treat
multiple `filterBy` entries on the same parameter as a logical "or". If
that semantic is what is actually missing, the right fix is a
clarification of `Subscription.filterBy` cardinality semantics, not
removal of `comparator`. This disposition is defensible only if the
workgroup confirms no implementer is putting prefixes in
`filterBy.value` today, since the existing definition prose explicitly
shows them doing so.

---

### Recommendation

**Recommended disposition:** A

The strongest argument for Disposition A is that the spec is already
self‑contradictory: the normative definition of `filterBy.value` shows
`le1950` as a legal value, yet a separate `comparator` element exists
to carry exactly that prefix. Implementers reading the resource will
reasonably do whichever the prose tells them, producing wire‑format
divergence that no profile or invariant currently catches. Aligning
`filterBy.value` with the FHIR search‑parameter value grammar is also
the model that maximally reuses what implementers already know — the
search query string — and it is the only option that natively supports
the "or" pattern (`lt50,gt100`) that the reporter calls out without
forcing the workgroup to also redefine the semantics of repeated
`filterBy` entries (Disposition C) or to invent new mixing rules
between `comparator` and `value` (Disposition B). Disposition B should
be held in reserve in case the workgroup discovers material
implementer adoption of the structured `comparator` field that would
make a clean removal too disruptive; in that case, raising
`comparator` to `0..*` and forbidding mixed styles is the next best
step. Disposition C is the weakest because it leaves the
prefix‑in‑`value` examples in the spec or quietly removes them without
acknowledging that some implementers will already have followed them.
