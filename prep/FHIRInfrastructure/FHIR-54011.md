## Summary

The ticket reports that the human-readable text of Bundle invariant
`bdl-17` — *"Use and meaning of issues for documents has not been validated
because the content will not be rendered in the document."* — does not
read like a constraint at all. It states a status/observation about the
spec rather than a testable rule on instance data, leaving implementers
unsure what behaviour, if any, the constraint is supposed to forbid or
require. The reporter (HanhongLu) asks the work group to either (a)
demote the statement to informational prose and move it to the Bundle
*Notes* section (`2.4.4.3`), or (b), if it is genuinely intended to be a
rule, rewrite it so the expected behaviour is clearly stated.

There are no Jira cross-references and no GitHub cross-references on
this ticket. The only related Zulip context is the older `committers >
Testing Invariants` thread, which is general background on invariant
quality and unit-testing of constraints — including a sub-discussion
about Bundle `bdl-17` where Brian Postlethwaite discovered that the Java
validator could not even parse a Bundle containing `Bundle.issues`,
which is consistent with the reporter's observation that the rule is
under-specified and under-exercised.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat `bdl-17` as **informational, not normative**. Concretely:

- Remove the constraint element with `key = "bdl-17"` from the `Bundle`
  StructureDefinition (and from any element-level invariant block on
  `Bundle.issues`).
- Move the substantive content — the warning that *the use and meaning
  of `Bundle.issues` in a document Bundle has not been validated because
  the issues will not be rendered as part of the signed/displayed
  document* — into a new sub-bullet under section *2.4.4.3 Notes about
  Bundle* on `bundle.html`, alongside the existing prose about document
  Bundles and signatures.
- Cross-reference the note from the `Bundle.issues` element definition
  (definition / comment text) so implementers reading the data dictionary
  are pointed at the narrative.

#### Justification

Invariants in FHIR Core are expected to express a testable rule over
instance data via a FHIRPath expression. The current `bdl-17` text
("…has not been validated because the content will not be rendered…")
is a meta-statement about the spec's maturity, not a property of any
individual `Bundle`. There is no FHIRPath expression that can be true or
false on an instance for that meaning, and the Zulip *Testing
Invariants* thread shows that even getting tooling to evaluate anything
on `Bundle.issues` was non-trivial — i.e. the rule has effectively never
acted as a constraint in the wild. Demoting it to a Note preserves the
authoring-team's intent (warn implementers about a known under-specified
area) without leaving a malformed invariant in the resource definition.
This also aligns with the prevailing FMG/FHIR-I practice of fixing
poorly-worded invariants by Technical Correction rather than leaving
them in.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `bdl-17` as a **real constraint** but rewrite both the human text
and the FHIRPath expression so that the rule is unambiguous and
testable. A concrete tightening, scoped to what the original text seems
to be hinting at, would be:

- Human: *"For Bundles of `type = document`, `Bundle.issues` SHOULD NOT
  be populated, because the contents of `Bundle.issues` are not part of
  the document's signed narrative and will not be rendered with the
  document."*
- Severity: `warning` (matches "SHOULD NOT" semantics).
- Expression: `type = 'document' implies issues.exists().not()`
- Keep the constraint anchored at the `Bundle` root (where the other
  `bdl-*` invariants live), and add at least one passing and one failing
  invariant-test instance under
  `source/bundle/invariant-tests/bdl-17.*.json` per the convention
  agreed in the *Testing Invariants* thread (Grieve / Postlethwaite).
- Add a short companion note under *2.4.4.3 Notes about Bundle*
  explaining the rationale (signed narrative, non-rendered content).

#### Justification

If the work group's underlying concern really is "implementers should
not be smuggling operational issues into a document Bundle", that is a
genuine constraint and is best expressed as such, not buried in prose
that implementers and validators routinely skip. The reformulation
above (a) is a single FHIRPath expression that is trivial to evaluate
on every example in the build, (b) uses `warning` severity so it does
not break existing instances that happen to populate `Bundle.issues` on
documents, and (c) mirrors the pattern of the surrounding `bdl-*`
constraints that gate behaviour on `Bundle.type`. It also closes the
gap surfaced in the Zulip thread, where the rule was effectively
untested because no one had ever tried to validate `Bundle.issues` —
adding the standard `bdl-17.pass` / `bdl-17.fail` instances would
finally make the behaviour exercised by every core build.

---

### Disposition C: Decline

#### Proposal

Leave `bdl-17` exactly as it is, with no change to wording, severity,
expression, or location. Mark the ticket *Not Persuasive — Existing
Behaviour* on the basis that the current text, while inelegant, is
deliberately a published caveat about an under-validated area of the
spec and any change risks being read as a normative tightening that the
work group has not actually agreed to.

#### Justification

The defensible reason to decline is conservatism: `bdl-17` has been in
the spec for several releases, has not (according to the cross-reference
index) generated a stream of implementer complaints, and any rewrite —
even a "clarifying" one — will be read by some implementers as a
substantive change in what document Bundles are allowed to do. If the
work group is not prepared to commit to the scoped rule in Disposition
B (e.g., that document Bundles SHOULD NOT carry `Bundle.issues`), then
the safest course is to leave the existing soft caveat alone rather
than promote it to a rule by accident. This option is weak because it
does not address the reporter's actual concern (the text does not read
as a constraint), but it is on the table if the work group concludes
the wording is acceptable as-is.

---

### Recommendation

**Recommended disposition:** A

The reporter's diagnosis is correct on its face: `bdl-17` is currently
phrased as a status report on the spec ("…has not been validated…"),
not as a property of any Bundle instance, and there is no FHIRPath
expression that can meaningfully test it. The Zulip *Testing
Invariants* history reinforces this — the rule has never really been
exercised end-to-end by the validators, which is exactly the situation
the description is admitting to. Demoting the statement to a Note in
section *2.4.4.3* (Disposition A) preserves the original authoring
intent — warn implementers about a known under-specified area — without
leaving a malformed invariant in the `Bundle` StructureDefinition, and
it can be applied as a Technical Correction with no instance impact.
Disposition B (rewrite as a real `SHOULD NOT` constraint on document
Bundles) is the right escalation *if* the work group has the appetite
to assert that policy; absent that, going straight to B risks tightening
the spec beyond the ticket's ask. Disposition C should be rejected
because it ignores the substantive defect the reporter has correctly
identified.
