## Summary

The ticket asks FHIR Infrastructure to provide a discrete, machine-readable
way to declare which self-referential ("circular") `Reference` elements
correspond to a true *hierarchical* relationship — i.e., the kind that the
`:above` and `:below` search modifiers are designed to traverse transitively.
Today this distinction is described only in narrative on the
[Circular References](https://hl7.org/fhir/6.0.0-ballot4/references.html#circular)
page (with example codes such as `part-of`, `build-on`, `replaces`, etc.),
and there is no element-level metadata an implementer or profile author can
use to assert "this self-reference forms a hierarchy that `:above`/`:below`
should follow." The reporter argues this should be expressible discretely on
`ElementDefinition` (in the base spec where reliably knowable, and otherwise
at the profile level).

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FhirAugury for this key, so the request is being prepared for
workgroup discussion without prior community consensus on the mechanism.

## Details

**Description:**

> The concept of "hierarchical" resource references are defined on references
> to the same resource type, and are used by the :above and :below modifiers
> to search transitively through the hierarchy. The documentation points out
> that some circular references correspond with a hierarchical structure, and
> some do not.
>
> This seems like something that should be able to be defined through
> discrete element-level metadata. If it can't be reliably determined for the
> base spec, there should be a standardized way to define these relationships
> at the profile level.

**Comments:**

- **corey_spears (2026-01-17):** "Do you have a recommended approach?"
- **JIRAUSER27471 (2026-01-19):** Suggests adding a new multi-valued
  `CodeableConcept` element on `ElementDefinition` that lets an element
  declare the relationship code(s) listed on the
  [circular references](https://hl7.org/fhir/6.0.0-ballot4/references.html#circular)
  page (`part-of`, `build-on`, etc.) and additionally declare whether the
  element permits circularity. The base resource `StructureDefinition`s would
  then populate this for each self-typed reference element.

### Disposition A: Accept As Requested

#### Proposal

Add discrete element-level metadata to `ElementDefinition` that declares,
for any `Reference` element pointing back to the same resource type, both
(a) the kind of relationship the reference expresses and (b) whether the
relationship is hierarchical (and therefore traversable by `:above` /
`:below`). Concretely:

- Introduce a new optional, repeating element on `ElementDefinition` —
  e.g., `ElementDefinition.referenceRelationship` (a backbone element)
  with at least:
  - `code : CodeableConcept` bound to a new value set seeded from the
    codes already enumerated narratively on the Circular References page
    (`part-of`, `build-on`, `replaces`, `successor-of`, `member-of`,
    `composed-of`, `derived-from`, etc.).
  - `hierarchical : boolean` indicating whether `:above` / `:below`
    should treat the reference as a transitive hierarchy.
  - `direction : code` (e.g., `parent` | `child` | `peer`) so that
    `:above` and `:below` know which way is "up".
- Populate this metadata in the base `StructureDefinition`s for every
  self-typed `Reference` element where the relationship is reliably
  known (e.g., `Group.member.entity` is **not** hierarchical;
  `Observation.partOf`, `Procedure.partOf`, `Encounter.partOf`,
  `Location.partOf`, `Organization.partOf` **are** hierarchical with
  `direction = parent`).
- Update the search-modifier documentation for `:above` / `:below` to
  state that servers MUST/SHOULD honor `referenceRelationship.hierarchical`
  when deciding which reference elements participate in transitive
  traversal, and that profiles MAY add the metadata where the base
  spec does not.
- Replace the narrative table on the Circular References page with a
  generated table sourced from the new metadata.

#### Justification

This is the minimum-viable expression of what the reporter actually asked
for: it lifts a piece of currently-implicit, prose-only knowledge into the
machine-readable layer that tooling already consumes. Putting the metadata
on `ElementDefinition` (rather than on `StructureDefinition` as a whole)
matches the granularity of the underlying problem — `:above`/`:below`
operate per reference element, not per resource — and means profile
authors can declare hierarchy on extension-defined references without
forking the base resource. The proposed shape is consistent with how
`ElementDefinition` already carries other reference-shaping metadata
(`type.targetProfile`, `type.aggregation`, `type.versioning`).
JIRAUSER27471's suggestion in the ticket comments points in the same
direction; this disposition simply formalizes it and adds a `direction`
discriminator so the modifiers are unambiguous.

---

### Disposition B: Alternative Approach

#### Proposal

Address the underlying need with an **extension** plus a **dedicated
`SearchParameter` property** rather than a new core `ElementDefinition`
field, in two parts:

1. Define a standard extension (e.g.,
   `http://hl7.org/fhir/StructureDefinition/elementdefinition-reference-hierarchy`)
   carrying `{ relationshipCode, direction, hierarchical }`. Apply it to
   the relevant base-spec elements via the existing extension mechanism;
   profiles may apply it to their own constraints.
2. Add (or formalize) a property on `SearchParameter` for reference-typed
   search parameters that declares "this parameter participates in
   `:above`/`:below`" and points at the hierarchy-bearing element(s).
   Servers consult `SearchParameter`, not `ElementDefinition`, when
   resolving modifier semantics — which matches where modifier behavior
   is otherwise specified.

#### Justification

Putting the relationship metadata in an extension avoids a normative
change to `ElementDefinition` (which is heavily used and conservatively
evolved) and lets the feature ship without waiting for a core data-type
revision. Anchoring the modifier-traversal rule on `SearchParameter`
keeps `:above`/`:below` semantics in the same place as the rest of
search-modifier behavior, which is where implementers look first.
Trade-offs vs. Disposition A: discoverability is weaker (an extension is
easier to ignore than a first-class element), and the metadata is split
across two artifacts (extension on `ElementDefinition`, declaration on
`SearchParameter`), which raises the maintenance cost. However, it is
cheaper to roll out, easier to back-port to existing IGs, and reversible
if a future major revision wants to absorb the extension into core.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped. Keep the existing narrative
description on the [Circular References](https://hl7.org/fhir/6.0.0-ballot4/references.html#circular)
page, but tighten that text so that for every base-resource self-typed
`Reference` element it explicitly states whether `:above`/`:below` apply
and in which direction. Do not add new discrete metadata to
`ElementDefinition` or `SearchParameter` at this time; revisit if a
concrete implementer use case (server, validator, profile IG) demonstrates
that the narrative form is causing interoperability failures.

#### Justification

Defensible on three grounds: (1) **scope** — the request introduces a new
normative element on a foundational data type to solve a problem that, in
practice, affects a small fixed list of resources; (2) **existing
mechanism** — the set of self-typed references in the base spec is
enumerable and can be specified prescriptively in narrative without
machine-readable plumbing; (3) **lack of demonstrated demand** — no
cross-referenced Jira tickets, Zulip threads, or GitHub issues were found
for this key, suggesting the implementer pain has not yet been quantified.
Adding `ElementDefinition` machinery for a problem that may be solvable by
a clearer table is a poor cost/benefit trade.

---

### Recommendation

**Recommended disposition:** B

The underlying need — making `:above`/`:below` traversal rules
machine-checkable rather than prose-only — is real and worth addressing,
so a flat decline (C) leaves implementers reverse-engineering a narrative
table forever. However, jumping straight to a new normative element on
`ElementDefinition` (A) is heavier than this problem warrants for a first
pass: `ElementDefinition` changes ripple through validators, profile
tooling, snapshot generation, and every IG, and the workgroup would be
committing to a shape before any implementer has built against it.
Disposition B threads the needle: it ships the same semantics via a
standard extension and a `SearchParameter` declaration, gets the
information into the machine-readable layer where `:above`/`:below`
implementations can consume it, and preserves the option to promote the
extension into core in a later major revision once usage patterns are
clear. The workgroup should ask the reporter whether an extension-based
form would meet the use case before committing to a core change, and
should also commission a follow-up `repo-analysis` of `HL7/fhir` so the
specific files (`source/elementdefinition/`, `source/references.html`,
`source/searchparameter/`) can be named in the implementation plan.
