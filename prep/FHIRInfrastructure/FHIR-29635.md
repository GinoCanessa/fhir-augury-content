## Summary

The ticket asks the FHIR Shorthand (FSH) language to extend its existing
"quantity literal" shortcut so authors can also set the human-readable
`Quantity.unit` string in a single assignment. Today, FSH lets an author
write `* valueQuantity = 155.0 '[lb_av]'` to populate `Quantity.value`,
`Quantity.system` (UCUM) and `Quantity.code`, but provides no compact
syntax for `Quantity.unit`. The reporter (Nick Freiter) proposes
appending a trailing string literal — for example
`* valueQuantity = 155.0 '[lb_av]' "lb"` — that would set
`valueQuantity.unit` to `"lb"`. The reporter notes that there is ample
precedent in FSH for trailing string literals being used to convey
human-readable values, so the addition is unlikely to confuse users.
There are no cross-referenced Jira tickets recorded in FhirAugury for
this issue, but adjacent Zulip discussion in `#shorthand` shows
substantively the same proposal landed in SUSHI 1.1.0 (January 2021)
under the bullet "Extends Quantity syntax to include `unit` string
(proposed FSH STU2 feature)", indicating the SUSHI implementation
shipped while the FSH specification change itself remained open as this
Triaged ticket.

## Details

**Description:**

FSH supports an easy syntax for specifying a Quantity when the units
are taken from UCUM:

```
* valueQuantity = 155.0 '[lb_av]'
```

But there is no way to specify the `unit` field when doing this. It
might make sense to support a syntax like:

```
* valueQuantity = 155.0 '[lb_av]' "lb"
```

where the `valueQuantity.unit` would then get set to `"lb"`. The `unit`
is meant to be a human readable value, and there is plenty of precedent
for setting human readable values as strings at the end of assignments,
so I don't think users would find this a confusing addition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the FSH grammar's "quantity literal" production to accept an
optional trailing string literal representing the human-readable
`Quantity.unit`, exactly as the reporter proposed:

```
QuantityLiteral := NUMBER UCUM_CODE STRING?
```

Concretely:

- `* valueQuantity = 155.0 '[lb_av]'` — unchanged behaviour: sets
  `value`, `system` (`http://unitsofmeasure.org`) and `code`.
- `* valueQuantity = 155.0 '[lb_av]' "lb"` — new behaviour: sets the
  same three plus `unit = "lb"`.

This requires changes in `HL7/fhir-shorthand` (the FSH reference
specification — grammar section, assignment-with-Quantity reference,
worked examples) and in `FHIR/sushi` (parser / quantity-rule exporter).
The SUSHI side appears to have already been implemented as part of the
1.1.0 release (Jan 2021, "Extends Quantity syntax to include `unit`
string"), so for SUSHI this disposition is largely a matter of
ratifying / formalising the existing behaviour rather than building
something new.

#### Justification

The proposal is consistent with the existing "human-readable string at
the end of an assignment" precedent in FSH (e.g., the trailing display
string on `code` assignments). It is purely additive — the third
operand is optional, so all existing FSH source remains valid. It also
brings the FSH grammar back into alignment with what the reference
implementation (SUSHI ≥ 1.1.0) already accepts, eliminating a latent
spec-vs-implementation drift that has persisted since this ticket was
filed in 2020. The Zulip discussion shows the change was treated as
"proposed FSH STU2 feature" without recorded objection.

> _Caveat from the missing repo briefing:_ specific edits in
> `HL7/fhir-shorthand` (grammar file, reference page) and the SUSHI
> grammar / exporter cannot be named precisely until a `repo-analysis`
> briefing exists for those repos.

### Disposition B: Alternative Approach — Require `Quantity(...)` Constructor Form Instead

#### Proposal

Instead of overloading the existing two-operand quantity literal with
an optional third operand, introduce (or formalise) a more explicit
constructor-style assignment that names each populated element:

```
* valueQuantity = Quantity(155.0, '[lb_av]', "lb")
```

or, equivalently, recommend the existing per-element form for any case
where `unit` is needed:

```
* valueQuantity.value  = 155.0
* valueQuantity.system = "http://unitsofmeasure.org"
* valueQuantity.code   = #[lb_av]
* valueQuantity.unit   = "lb"
```

Under this disposition, the spec would *not* change the quantity
literal's grammar; instead, the reference page would explicitly call
out that `unit` requires either the per-element form or (if
introduced) a constructor form.

#### Justification

This disposition keeps the existing two-operand quantity literal
unchanged, avoiding any risk of confusion about whether the trailing
string is a `unit` (Quantity.unit) or a `display` (as it is in code
assignments — and trailing strings already mean "display" in the FSH
code-assignment grammar). That overload is the one substantive
trade-off Disposition A glosses over: a reader who knows
`* code = LOINC#1234-5 "Glucose"` ("Glucose" is `code.display`) will
now also have to internalise that the same trailing-string position
means `unit` for Quantity. A constructor form, or simple per-element
assignments, removes the overload at the cost of being more verbose.

The downside is that this contradicts what SUSHI 1.1.0 already ships,
which would mean either rolling back working SUSHI behaviour or
tolerating a spec-implementation gap.

---

### Disposition C: Decline — Existing Mechanism Already Suffices

#### Proposal

Decline the request on the grounds that authors who need to set
`Quantity.unit` already have a clear mechanism: a follow-up assignment
rule on the `.unit` element after the standard quantity literal:

```
* valueQuantity      = 155.0 '[lb_av]'
* valueQuantity.unit = "lb"
```

Mark the ticket as "Not Persuasive — existing mechanism" (or
equivalent) and update the FSH reference's "Assignments with the
Quantity Data Type" page to explicitly show the two-line idiom for
setting `unit`.

#### Justification

The ticket is a *convenience* request, not a capability gap — every
field on `Quantity` is already settable today. Extending the grammar
introduces a new grammar production that all FSH parsers/IDEs/linters
must learn, and (per Disposition B's analysis) overloads the meaning
of "trailing string" in an assignment rule between `display` and
`unit`. The marginal saving is one line of FSH per Quantity assignment
that needs `unit`. If the workgroup judges that the SUSHI 1.1.0
behaviour was implemented prematurely, Decline is the disposition that
lets the spec stay simple — at the cost of asking the SUSHI maintainers
to either deprecate or quarantine the 1.1.0 syntax.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested).

The change has already been implemented and shipped in SUSHI 1.1.0
(January 2021) and presented to the `#shorthand` Zulip stream as a
"proposed FSH STU2 feature" without recorded objection. Leaving
FHIR-29635 in `Triaged` indefinitely simply preserves a known
spec-vs-implementation drift: SUSHI accepts
`* valueQuantity = 155.0 '[lb_av]' "lb"` while the FSH reference does
not document it. The most honest outcome is to ratify the existing
behaviour in the FSH reference (grammar section + the "Assignments
with the Quantity Data Type" reference page), which is a small,
purely-additive spec edit. The trailing-string-vs-display overload
flagged under Disposition B is real and worth a sentence of
clarification in the reference page, but it is not severe enough to
override two factors: (1) the ticket reporter's original argument that
trailing human-readable strings are already a familiar FSH pattern,
and (2) the strong pragmatic pull of aligning the spec with shipping
tooling that implementers have been using for several years.

If the workgroup elects Disposition A, the implementation plan should
begin by running `repo-analysis HL7/fhir-shorthand` (and ideally
`FHIR/sushi`) so subsequent planning can name the precise grammar and
reference-page files to edit.
