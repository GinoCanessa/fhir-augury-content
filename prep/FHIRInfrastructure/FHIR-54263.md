## Summary

The reporter proposes adding a way to organize long lists of `Questionnaire.item.answerOption` choices into visually grouped sub-headings when rendered. Today, SDC lets authors control option order, column counts, and flow direction, but provides no mechanism to introduce in-line headings that cluster related choice values. The ticket proposes two new SDC extensions — `GroupingLabel` (applied per `answerOption`) and `GroupingProperty` (applied at the `Questionnaire.item` level, restricted to `choice`, `open-choice`, or `Coding`-typed items) — that together allow either explicit per-option labelling or a code-driven grouping derived from a coded property (typically supplied via a `CodeSystem` supplement). No related Jira tickets surfaced via cross-references; the only related discussion is a Zulip thread the reporter opened in `#questionnaire` to gauge implementer interest before progressing the proposal.

## Details

**Description:**

Sometimes when you have a long-ish list of options, it's useful to group them under sub-headings.  Proposal is to define 2 extensions:

"GroupingLabel" 0..1 string with context of `Questionnaire.item.answerOption`

Definition: "A label to nest this option under.  All options with the same label will be grouped under that one label.  This may change the order in which some options appear"

Comment: "Also see the GroupingProperty extension"

GroupingProperty 0..1 string with context of `Questionnaire.item`

contextconstraint - can only appear on items of type `choice`, `open-choice`, or `Coding`

Definition: "A code property that serves as the grouping label when presenting options.  All codes with the same grouping label will appear under that label."

Comment: "This only works for coded questions.  In many cases, the properties will be introduced via a code system supplement.  Also see the GroupingLabel extension"

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define two new extensions in the SDC IG exactly as the reporter described:

1. `http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-questionnaire-groupingLabel`
   - Context: `Questionnaire.item.answerOption`
   - Cardinality: 0..1
   - Value: `string`
   - Definition: "A label to nest this option under. All options with the same label will be grouped under that one label. This may change the order in which some options appear."
   - Comment: cross-references the `groupingProperty` extension.

2. `http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-questionnaire-groupingProperty`
   - Context: `Questionnaire.item`
   - Cardinality: 0..1
   - Value: `string` (the name of a `CodeSystem` property to use as the grouping key)
   - Context constraint / invariant: only allowed when `Questionnaire.item.type` is `choice`, `open-choice`, or `Coding`.
   - Definition and comment as quoted in the ticket, including the note that the property is typically introduced via a `CodeSystem` supplement and that `groupingLabel` is the per-option alternative.

Add narrative in the SDC IG's "Form Behavior and Calculation" / rendering chapter describing both extensions, give at least one Questionnaire example using each extension, and include them in the SDC extension index.

#### Justification

The proposal fills a real rendering gap that today forces implementers to either pre-process options into separate items (losing the single-question semantic) or invent vendor-specific extensions. Two extensions are warranted because the underlying need has two natural authoring styles: hand-curated lists (per-option `groupingLabel`) and `ValueSet`/`CodeSystem`-driven lists where a coded property already classifies the answers (`groupingProperty`). Modeling this as SDC extensions — rather than core `Questionnaire` elements — is consistent with how SDC has historically extended base `Questionnaire` rendering behaviour (e.g., `choiceOrientation`, `itemControl`) and avoids a base-resource change for what is purely a presentation concern. The CSIRO outreach noted in the comment is intended to confirm implementer demand before publication.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the underlying capability with a single, more general extension instead of two parallel ones:

- One extension `sdc-questionnaire-optionGrouping` on `Questionnaire.item` with a small structured payload (a `BackboneElement`-style complex extension) containing:
  - `source` (code, required): `inline` | `codeProperty`.
  - `property` (string, 0..1, required when `source = codeProperty`): the `CodeSystem` property name.
  - `order` (positiveInt, 0..*): optional explicit ordering of group labels.
  - A `contextconstraint` limiting use to items of type `choice`, `open-choice`, or `Coding`.
- For the `inline` case, reuse the existing pattern of a per-`answerOption` extension whose value is `Coding` (referencing a code declared in the parent extension's `order` list) rather than a free-text `string`, so groups are first-class and can be localized via the question's binding rather than relying on string equality across `answerOption` instances.

#### Justification

Two unrelated `string`-valued extensions whose values must match exactly across siblings are fragile: typos, casing, whitespace, and translation all silently break grouping. A single complex extension makes the relationship between "declare the groups" and "assign options to a group" explicit and enforceable, lets the IG state ordering rules once, and lets renderers validate the construct without comparing free-text strings. It also reduces the surface area implementers have to learn and test from two extensions to one, and makes future additions (e.g., per-group `itemControl`, collapse/expand defaults) extensible without inventing yet more parallel extensions. Trade-off vs. A: slightly more authoring ceremony for the simplest "just slap a label on these three options" case, and a complex extension is mildly harder to render in low-end tooling.

---

### Disposition C: Decline

#### Proposal

Do not adopt either extension. Recommend that authors who need grouped options model the groups as `Questionnaire.item` of `type = group` (or `display`-type child items used as headings) wrapping their own child items, or that they use `ValueSet` composition + `itemControl` patterns already available in SDC to render multi-section choice lists. Mark the ticket as "Persuasive with mod / Already addressed" and document the recommended pattern in the SDC IG's rendering guidance.

#### Justification

The base `Questionnaire` already supports nested `item`s, `display` items, and `itemControl`-driven rendering, and SDC already standardizes a number of presentation extensions. Introducing two more presentation extensions whose semantics are purely visual risks fragmenting how implementers render choice lists, and — given the open Zulip question about whether implementers will actually support the feature — the workgroup may not yet have evidence that adoption justifies the maintenance cost and conformance surface of two new extensions. The fragility of cross-`answerOption` string matching (see Disposition B's critique) is a further reason to be cautious about Accept-as-Requested specifically; declining and documenting an existing-mechanism workaround keeps the SDC extension catalogue smaller.

---

### Recommendation

**Recommended disposition:** B

The underlying capability is genuinely useful and the Zulip thread suggests the reporter is actively seeking implementer signal rather than asserting it exists, so an outright decline (C) is premature. However, accepting the two extensions exactly as written (A) bakes in a brittle string-match contract between sibling `answerOption` instances and offers no place to declare group order, localization, or future per-group rendering hints — all of which the workgroup will predictably be asked for next. A single complex `optionGrouping` extension that supports both authoring modes (inline `Coding`-valued group assignments and `CodeSystem`-property-driven grouping) keeps the conformance surface small, makes the construct validatable, and leaves room to grow. Recommend the workgroup pursue B, contingent on the CSIRO/implementer feedback Brian Postlethwaite is gathering confirming non-trivial demand; if that feedback comes back negative, fall back to C and document the existing `group`/`display`/`itemControl` workaround.
