## Summary

The ticket reports that the **AppState** scopes defined for SMART App
Launch reuse the same syntactic pattern as ordinary SMART on FHIR
scopes but behave inconsistently for non-patient subjects. Patient-bound
scopes are prefixed with `patient/`, which forces the launch context to
include a `Patient` reference. By contrast, both **practitioner-bound**
AppState and **global (app-wide) AppState** are prefixed with `user/`,
which leaves the two indistinguishable at the scope layer: a global
scope can be exercised against a practitioner reference and vice versa,
and a `user/Basic.cuds?code=<system>|<code>` scope does not require the
authorization server to bind the practitioner identity into the launch
context.

The reporter proposes introducing a new `practitioner/` scope prefix —
e.g. `practitioner/Basic.cuds?code=<system>|<code>` — that would oblige
clients to include the active `Practitioner` reference in the same way
`patient/` scopes oblige a `Patient` reference. The cross-reference
index does not surface any related Jira tickets, Zulip threads, or
GitHub PRs/issues; the only outbound references are the literal
`Basic.cuds` and `Basic.rs` artifact mentions in the scope-syntax
examples used in the description and comments.

The ticket was originally voted "Considered for Future Use"
(13-0-0, Bas van den Heuvel / Yunwei Wang, 2023-02-13), reverted, and
then re-opened post-STU2.1 publication so that the workgroup can revisit
whether a `practitioner/` prefix should be introduced in a future
(likely major-version) release of SMART App Launch.

## Details

**Description:**

The AppState scopes use the same pattern as the normal SMART on FHIR
scopes but seem to have a slightly different behavior.

The scopes that require patient references start with `patient` and
thereby force the inclusion of a patient reference.

The practitioner and global app scopes are prefixed with `user`. From a
scope perspective, there is no difference between them. This allows a
global scope to be used with a practitioner reference and vice versa. A
`user/Basic.cuds?code=<system>|<code>` scope also does not force the
inclusion of the current practitioner reference.

This could be solved by introducing a new prefix called `practitioner`.
A scope formatted as `practitioner/Basic.cuds?code=<system>|<code>`
would require the app to include its Practitioner reference similarly
to patient scopes.

### Disposition A: Accept As Requested

#### Proposal

Introduce a new scope prefix `practitioner/` in the SMART App Launch
AppState scope grammar, parallel to the existing `patient/` prefix.
Concretely:

- Extend the AppState scope BNF / regex in the SMART App Launch IG to
  accept `practitioner/<resource-type>[.<verb>][?<query>]` (e.g.
  `practitioner/Basic.cuds?code=<system>|<code>`,
  `practitioner/Basic.rs?code=<system>|<code>`).
- Define the launch-context obligation: when a `practitioner/` AppState
  scope is granted, the authorization server MUST bind a Practitioner
  reference into the launch context (mirroring the
  `need_patient_banner` / `patient` launch-context behavior for
  `patient/` scopes), and the resource server MUST scope reads/writes
  to the bound Practitioner.
- Clarify that `user/` AppState scopes remain "global" / cross-subject
  and that `practitioner/` is the SHALL way to express
  practitioner-subject AppState going forward.
- Update the AppState section of the SMART App Launch IG (scope
  syntax tables, examples, and conformance statements) and add a
  changelog entry flagging this as a SMART App Launch enhancement
  intended for the next major version (post STU2.1, per
  lynn_laakso 2023-04-18).

#### Justification

The asymmetry the reporter identifies is real: `patient/` is both a
subject filter **and** a launch-context obligation, while `user/`
collapses two distinct intents (the running practitioner vs. global,
non-subject AppState) into one prefix. Introducing `practitioner/`
preserves the existing scope grammar (parallel construction with
`patient/`), reuses the launch-context machinery already specified for
`patient/`, and gives implementers an unambiguous way to express
"AppState that belongs to the active practitioner." The original
2023-02-13 ballot acknowledged the request was reasonable but
out-of-cycle; the 2023-04-18 reversal explicitly invites the workgroup
to reconsider this for a future major release, which is the
appropriate vehicle for a new scope prefix.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a new prefix. Instead, formalise the
search-parameter-based pattern that lynn_laakso outlined in the
2023-04-18 comment, by:

- Adding normative guidance to the SMART App Launch IG AppState
  section that explains how to constrain `user/`-prefixed AppState
  scopes to the active practitioner using existing FHIR search
  parameters — e.g.
  `user/Basic.rs?subject:missing=false&subject=Practitioner/{id}&code=...`.
- Defining a server obligation that, when a `user/` AppState scope
  carries a `subject=Practitioner/{id}` (or equivalent reference)
  query parameter, the `{id}` MUST resolve to the launch-context
  practitioner (i.e. the server MUST reject mismatches).
- Providing worked examples for both the practitioner-restricted and
  global-app cases so that implementers can disambiguate intent
  without a new scope prefix.
- Optionally, in the same section, signal that a future major version
  may introduce a `practitioner/` prefix as a syntactic convenience —
  effectively a forward-compat note rather than a new normative
  construct now.

#### Justification

This route exploits SMART v2's existing "advanced scopes" / FHIR
search-parameter-restricted scope mechanism, which already provides
the expressivity needed to bind AppState to the active practitioner.
It avoids extending the scope grammar (which propagates to every
SMART-conformant client, server, AS, and conformance test), avoids a
breaking-style change to clients that currently parse only `patient/`
and `user/`, and keeps the change tractable inside a STU dot-release.
The trade-off is that it is more verbose, places the burden of
correctness on the AS/RS to validate the `subject` query parameter
against the launch context, and does not give the practitioner case
the same first-class ergonomics that `patient/` enjoys.

---

### Disposition C: Decline

#### Proposal

Re-resolve as **Considered for Future Use** (restoring the
2023-02-13 disposition) without committing to a normative change in
either STU2.1 or the next dot-release. Document in the AppState
section that `user/` AppState scopes do not, by themselves, bind a
Practitioner reference, and refer implementers to the SMART v2
advanced-scope / search-parameter mechanism for any
practitioner-scoped restriction they need today. Park the
`practitioner/` prefix proposal on the SMART App Launch backlog for a
future major-version cycle.

#### Justification

A new top-level scope prefix is a non-trivial change to a widely
deployed authorization grammar: every SMART AS, RS, client SDK, and
conformance test needs to learn it, and ecosystem rollout typically
lags the IG by years. The 2023-02-13 vote (13-0-0) and the
follow-up commentary both signalled that the workgroup considers this
a future-major-version concern rather than an STU2.1 defect. The
reporter's underlying need — preventing cross-subject AppState
leakage — is addressable today using SMART v2 advanced scopes, and
declining now keeps STU2.1 stable while leaving the door open for a
purpose-built design when the workgroup opens the next major version.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle that the workgroup itself drew on
2023-04-18: it acknowledges that the practitioner-vs-global ambiguity
in `user/` AppState scopes is a real gap, and it closes that gap using
machinery (SMART v2 advanced scopes, FHIR `subject` search-parameter
restriction) that is already normative in SMART App Launch STU2 and
already cited by lynn_laakso as the interim guidance. That makes it
implementable in a STU dot-release without forcing every SMART
implementer to upgrade their scope parser, while still giving the
reporter a first-class, conformance-checkable way to express
"AppState belonging to the active practitioner."

Disposition A (a new `practitioner/` prefix) is the cleaner long-term
design and should remain on the table for the next major version of
SMART App Launch — but the original 13-0-0 vote and the
2023-04-18 reopening commentary both flag it as out-of-cycle for
STU2.1, and a new prefix is exactly the kind of change that benefits
from a major-version ecosystem reset. Disposition C is defensible but
under-serves the reporter, because the asymmetry they identified is a
genuine interoperability hazard (silent cross-subject AppState
exchange) rather than a stylistic preference, and SMART has the tools
to address it now without waiting for a major version.

The workgroup should pick B for STU2.1/dot, and explicitly carry A
forward as a candidate for the next SMART App Launch major version.
