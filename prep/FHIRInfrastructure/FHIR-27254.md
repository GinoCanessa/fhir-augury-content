## Summary

The ticket requests that FHIR Shorthand (FSH) allow **inline definition of
invariants** on profile elements, in addition to the current syntax that
requires invariants to be declared as standalone entities and then referenced
from elements via `obeys`. The reporter (Bas van den Heuvel) argues that
because invariants are frequently used in only a single location, requiring
them to be defined separately bloats profiles unnecessarily, and an inline
form would make profile authoring more concise. No related Jira tickets,
Zulip threads, or GitHub items were surfaced via cross-references for this
ticket; the discussion captured here is limited to the ticket's own comments.

## Details

**Description:**

Invariants are often only used in one location, as such they should also be
allowed to be provided in line. This will make the profile more condensed for
invariants only applicable for a single profile.

## Proposed Dispositions

The dispositions below cannot be tied to a specific authoring root or repo
file path because no GitHub cross-references or repo briefings are available
for FHIR-27254. They instead reference the FHIR Shorthand grammar surfaces
that any change would touch (FSH spec language definition and the SUSHI
reference parser/compiler).

### Disposition A: Accept As Requested

#### Proposal

Extend the FSH grammar to permit defining an invariant inline on the rule
that constrains the element it applies to. Introduce a syntactic form that
co-locates the four `Invariant`-defining metadata items
(`Severity`, `Description`, optional `Expression`, optional `XPath`) with the
element rule, and emits the same artefact pair that the existing two-step
form does today: (a) a top-level `Invariant:` definition, and (b) an
`obeys` rule on the element. A strawman shape (subject to FHIR Infrastructure
WG bikeshedding) might be:

```
* status obeys us-core-status-1 ("status SHALL be active or completed",
                                  #error,
                                  "status = 'active' or status = 'completed'")
```

The compiler would synthesise the `Invariant:` definition under the cover so
that the produced StructureDefinition is byte-equivalent to the explicit
two-step form. The change would land in:

- The FHIR Shorthand specification (grammar and reference docs in HL7's FSH
  spec source) — new production rule plus normative text and examples.
- The SUSHI reference implementation — parser, AST, validation (key
  uniqueness across an inline invariant and any like-named top-level
  invariant), and serialiser.

#### Justification

This directly addresses the reporter's concern about profile verbosity for
single-use invariants and is consistent with FSH's design goal of being a
concise authoring language for FHIR conformance resources. The alternative
already cited in the ticket — caret syntax on
`^constraint[+]` — works but is widely considered an escape hatch and
defeats the readability argument. An inline invariant production is a pure
addition (the existing `Invariant:` form remains valid), so it is
non-breaking. The original 14-0-0 "Considered for Future Use" disposition
acknowledged the value of the request and only deferred it past STU1, which
has since shipped, so the original deferral rationale no longer applies.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing new inline-invariant grammar, document and bless the
existing **caret syntax** path as the supported mechanism for single-use
constraints, and ship a SUSHI lint/style guide entry that explains when to
use each form:

- **Top-level `Invariant:`** when the invariant is reused or has substantive
  metadata (description, expression, severity rationale).
- **Caret rules on `^constraint[+]`** for one-off, single-element
  constraints where the author wants to keep the constraint definition
  co-located with the element.

Optionally, add a SUSHI helper / snippet that expands a short pseudo-syntax
into the corresponding `^constraint[+].key`,
`^constraint[+].severity`, `^constraint[+].human`,
`^constraint[+].expression` rules at compile time, without changing the
normative FSH grammar.

#### Justification

This reuses an existing, already-implemented mechanism (caret syntax on
`ElementDefinition.constraint`) that the prior FHIR-I disposition explicitly
called out as a current solution. It avoids growing the FSH grammar, which
keeps the language smaller and reduces the test/regression surface for
SUSHI and any other FSH-aware tooling. The downside is that caret syntax
is widely perceived as awkward, so this disposition leans heavily on
documentation and tooling polish to make the existing path discoverable
and ergonomic.

---

### Disposition C: Decline

#### Proposal

Decline the request and reaffirm the prior committee position
("Considered for Future Use", Kramer/Rhodes 14-0-0). Close the ticket as
**Persuasive with Mod / Not Persuasive** with the rationale that the
existing caret syntax on `^constraint[+]` already provides an in-line
mechanism for single-use invariants, and that adding a second, parallel
syntactic form would (a) increase FSH surface area without adding new
expressive power, and (b) create two ways to do the same thing, which
hurts readability and tooling.

#### Justification

The prior FHIR-I vote was unanimous (14-0-0) on the substance of the
question, the existing caret-based path already works today, and no new
implementer pain points have been added to the ticket since it was
re-opened in 2020. Single-use invariants are common, but the marginal
keystroke cost of the `Invariant:` plus `obeys` pair is small and the
top-level form keeps invariant keys greppable across a project, which has
ongoing maintenance value when an invariant later needs to be reused or
referenced from issues / errata.

---

### Recommendation

**Recommended disposition:** A

The original 14-0-0 vote did not reject the request on its merits; it
deferred it past FSH 1.0 as a "potential post-1.0 extension." STU1 has
shipped, and Bas van den Heuvel reopened the ticket for exactly that
reason. Disposition C amounts to reaffirming a deferral whose stated
precondition (post-1.0) is no longer in force, and Disposition B leans on
caret syntax that is technically capable but is widely regarded as the
escape hatch FSH was meant to avoid for ergonomic authoring. Disposition
A delivers the conciseness benefit the ticket actually asks for, is purely
additive to the FSH grammar (no break to existing files), and produces an
identical StructureDefinition output to today's two-step form so downstream
tooling (publisher, validator) sees no difference. The remaining work is
genuine spec/grammar bikeshedding plus a SUSHI implementation, which is
appropriate scope for FHIR Infrastructure to take on now that FSH is past
its initial normative gate.
