## Summary

The ticket asks the FHIR Infrastructure work group to clarify the
specification's `number` search semantics around how *implicit precision*
ranges interact when the search-parameter value and the resource value
have different numbers of significant digits. The current
`search.html#number` table illustrates `[parameter]=100.00` as matching
"values that equal 100, to 5 significant figures precision … the range
[99.995 … 100.005)", but is silent on whether a stored target value of
`100.0` (implicit range `[99.95 … 100.05)`) is considered equal to the
search parameter `100.00` (implicit range `[99.995 … 100.005)`). The
reporter notes that some of the values encompassed by `100.0` fall into
the `100.00` range and others do not, and asks the analogous question of
whether `100.0` is equal to `100.04`.

This is one of a tightly clustered set of "number search precision"
tickets, all from or co-signed by the same author / reviewers:

- [FHIR-51077](https://jira.hl7.org/browse/FHIR-51077) (Triaged) —
  general errors in the number-search range table on the same page.
- [FHIR-43208](https://jira.hl7.org/browse/FHIR-43208)
  (Applied / Persuasive with Modification) and
  [FHIR-40319](https://jira.hl7.org/browse/FHIR-40319)
  (Applied / Persuasive with Modification) — earlier rounds of cleanup
  for the same range examples and comparison definitions, which the
  current ticket builds on.
- [FHIR-40029](https://jira.hl7.org/browse/FHIR-40029) (Published /
  Persuasive) — clarified which side of a comparison is the
  search-parameter value vs. the resource (target) value.
- [FHIR-30096](https://jira.hl7.org/browse/FHIR-30096) (Published /
  Persuasive) and [FHIR-7994](https://jira.hl7.org/browse/FHIR-7994)
  (Published / Persuasive) — corrected the per-row significant-figures
  / range arithmetic in the same table.
- [FHIR-55742](https://jira.hl7.org/browse/FHIR-55742) (Submitted) —
  flags that the `1e2` row in the same table is still wrong and that
  the `100` / `100.00` rows could be more explicit, which directly
  overlaps this ticket.

The aggregated picture is: the table is technically self-consistent for
each individual row (each row's range is the implicit range of *that
row's* search-parameter value), but the spec never spells out the
*two-sided* `eq` rule — namely that the implicit range of the
search-parameter value must fully contain the implicit range of the
target value — in a way that lets a reader reason about asymmetric
precision (`100.0` vs. `100.00`, `100.0` vs. `100.04`).

## Related Zulip Discussions

### [implementers > Search for Decimals respecting Ranges](https://chat.fhir.org/#narrow/stream/implementers/topic/Search%20for%20Decimals%20respecting%20Ranges)

A long-running, two-phase thread (2020-02 and 2021-04 → 2022-05) that
is the canonical community discussion of exactly the question
FHIR-51080 raises. Key points:

- **Alexander Kiel** opened the topic asking whether the implicit
  range applies to *both* the query value and the target value, and
  argued that returning more (rather than fewer) results was the safer
  default.
- **Lee Surprenant** initially interpreted only the query value as a
  range, then switched to the two-sided range interpretation, then
  later said the spec is genuinely ambiguous because there are two
  values involved.
- **Grahame Grieve** stated the design intent: "Searches are always
  performed on values that are implicitly or explicitly a range" is a
  general statement; the `gt/lt/ge/le/sa/eb` rule explicitly strips the
  implicit precision from the search value for those prefixes; for
  `eq`, the rule is "the range of the search value fully contains the
  range of the target value." He confirmed `gt2.2` *should* match a
  resource value of `2`.
- **Craig McClendon** carried the thread into 2021/2022 with the exact
  asymmetric example FHIR-51080 raises: instance `1.0`, query
  `eq1.00` → no match (because `[0.995, 1.005)` does not contain
  `[0.95, 1.05)`); instance `1.00`, query `eq1.0` → match. He argued
  this violates the symmetric property of equality and that "implicit
  precision should apply to the search parameter only" is the only
  interpretation that always returns "more stuff" for safety.
- **Josh Mandel** confirmed the asymmetric "no match" result is the
  intended behaviour ("consider equal if the value is *definitely* in
  range" rather than "if the value *might* be in range") and noted the
  same shape of surprise applies to date `eq2022-05-01` not matching a
  stored value of `2022`. He also flagged that `ap` is the prefix to
  use when "any overlap" is the desired semantic.
- **Vassil Peytchev** dissented strongly on patient-safety grounds,
  saying `eq2.0` not returning `2` is "never a good thing" and is a
  clear 80/20 violation.
- **ryan moehrke** noted that mixing range semantics and precision
  semantics into a single search value is the root cause of the
  confusion.
- **Mirjam Baltus**, **Paul Church**, and **Lloyd McKenzie**
  participated; the thread also references the (now closed) ticket
  [FHIR-26311](https://jira.hl7.org/browse/FHIR-26311) which fixed the
  `gt`/`lt` side but left the `eq` asymmetric-precision case open —
  which is exactly the gap FHIR-51080 wants closed.

Participants: Alexander Kiel, Lee Surprenant, Grahame Grieve, Craig
McClendon, Lloyd McKenzie, Mirjam Baltus, Paul Church, Josh Mandel,
Vassil Peytchev, ryan moehrke.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the "number" sub-section of `search.html` (in `HL7/fhir`,
authoring root `source/`, file `source/search.html`) to add an explicit
worked-example paragraph immediately after the existing
`[parameter]=100.00` row that spells out the asymmetric-precision case:

> When the search-parameter value and the stored target value have
> different numbers of significant figures, the implicit ranges of the
> two values are computed independently, and the `eq` test passes only
> when the search-parameter range *fully contains* the target-value
> range (consistent with the existing `eq` row of the comparator
> table). Worked examples:
>
> - Search `[parameter]=100.00` (implicit range `[99.995 … 100.005)`)
>   against a stored value of `100.0` (implicit range
>   `[99.95 … 100.05)`) does **not** match, because the target range is
>   wider than the search range.
> - Search `[parameter]=100.0` (implicit range `[99.95 … 100.05)`)
>   against a stored value of `100.00` (implicit range
>   `[99.995 … 100.005)`) **does** match.
> - Search `[parameter]=100.0` against a stored value of `100.04`
>   (implicit range `[100.035 … 100.045)`) **does** match, because
>   `[100.035 … 100.045)` ⊂ `[99.95 … 100.05)`.
> - To get "any overlapping range" semantics (which would, for example,
>   make `eq100.00` match a stored `100.0`), use the `ap` prefix
>   instead of `eq`.

Reuse the boundary-language vocabulary introduced by FHIR-40319 and
FHIR-40029 ("range of the search value", "range of the target value",
"upper / lower boundary") so the new prose is consistent with the rest
of the section.

#### Justification

This is the smallest change that closes the documented gap: it does
not alter conformance, it stays within the existing authored HTML page,
and it directly answers the two questions in the ticket
(`100.0` vs. `100.00`, `100.0` vs. `100.04`). It also matches what
Grahame Grieve and Josh Mandel stated as the design intent on Zulip
("consider equal if the value is *definitely* in range") and is
consistent with the `eq` rule already in the comparator table ("the
range of the search value fully contains the range of the target
value"). Pointing readers at `ap` for the alternative semantic
addresses the patient-safety concern Vassil Peytchev raised without
re-litigating the underlying rule. Bundling this edit with the closely
overlapping FHIR-55742 and FHIR-51077 in the same PR avoids touching
`source/search.html` three times.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only adding worked-example prose, restructure the
number-search table itself so that each `[parameter]=…` row carries
*two* derived columns:

1. The implicit range of the **search-parameter** value (already
   shown).
2. A short rule restating the `eq` containment test in symbolic form
   (e.g., `match iff target-range ⊆ search-range`), with one or two
   "matches" / "does not match" example resource values per row.

In addition, promote the long-form discussion ("how implicit precision
interacts with the comparator prefixes") to its own subsection
immediately above the comparator table in `source/search.html`, with
named worked examples for `eq`, `ap`, and the strict comparators
(`gt`/`lt`/`ge`/`le`), reusing the FHIR-40319 boundary vocabulary.
Cross-link the new subsection from the `number`, `quantity`, and `date`
sub-sections (the same surprise applies to date searches per Josh
Mandel's Zulip note about `eq2022-05-01` not matching a stored
`2022`).

#### Justification

Disposition A patches the immediate question but leaves the structural
problem the Zulip thread keeps surfacing — readers cannot tell from
the per-row prose alone whether the implicit range applies to the
query value, the target value, or both. A single, named "Implicit
precision and equality" subsection that the per-type tables link into
gives implementers one canonical place to read the rule, and reusing
the same subsection for `date` and `quantity` searches preempts the
analogous tickets that will otherwise be filed against those sections
(FHIR-43208 already showed this pattern for dates). The cost is a
larger spec edit that is more likely to need a second review pass and
that may interact with FHIR-40319 / FHIR-40029 wording the WG has
already agreed; it is also harder to keep "narrow scope" given the
number of co-authors who will want to weigh in.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed**: the comparator table on
`search.html` already says, for `eq`, that "the range of the search
value fully contains the range of the target value", and the
`[parameter]=100.00` row already states the implicit range of the
search-parameter value. Mechanically applying the containment rule to
both implicit ranges yields a single, deterministic answer to both of
the reporter's questions. Close as **Not Persuasive** (or
**Persuasive — No Change**) with a comment pointing the reporter at
the existing `eq` row and at the `ap` prefix for "any overlap"
semantics, and direct any further discussion to the open Zulip thread.

#### Justification

This is the cheapest defensible outcome and avoids re-opening a debate
the WG has resolved several times. It is, however, hard to defend on
substance: the same question has been asked repeatedly in Jira
(FHIR-7994, FHIR-30096, FHIR-43208, FHIR-40029, FHIR-40319,
FHIR-55742) and at length on Zulip across two phases (2020 and
2021–2022). Multiple senior implementers — including a co-chair and
multiple server authors — have stated the spec is genuinely ambiguous
or surprising. Declining a clarification request that is so cheap to
satisfy mostly serves to push the work onto the next reporter.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the best starting point for the work group. It
directly answers both of the reporter's questions, it costs a single
small edit to `source/search.html`, it is consistent with the design
intent stated by Grahame Grieve and Josh Mandel on Zulip and with the
vocabulary already accepted in FHIR-40319 / FHIR-40029, and it can be
bundled with FHIR-55742 (and ideally FHIR-51077) in a single PR
against the same page so the table is touched once. Disposition B
captures a real opportunity but is a meaningfully larger change and is
better filed as a follow-up improvement ticket against the whole
"implicit precision" discussion (it would also benefit from explicit
sign-off from the comparator-table co-authors). Disposition C is hard
to defend given how many times this exact question has been asked.
The work group should adopt A, and consider opening a separate
"restructure the implicit-precision discussion" ticket along the lines
of B if reviewers feel the per-row prose is still load-bearing.
