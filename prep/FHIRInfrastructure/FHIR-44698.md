## Summary

The reporter raises a documentation/navigation problem on the FHIR Core
specification: the Types page does not link out to the abstract
`Resource`, `DomainResource`, `MetadataResource`, and `CanonicalResource`
types except indirectly via the UML diagrams — the visible "Resources"
link instead jumps to the full resource list. The reporter also flags an
inconsistency between the UML page (§2.1.6.1.1, "Interfaces"), which
introduces abstract "interface" types, and the Patterns page (§1.2),
which discusses "interfaces" without listing any of those interface
types — implying possible conceptual duplication or unclear scope
between "interface types" and "interface patterns". Finally, the reporter
notes that the Patterns page itself does not appear in the documentation
index. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items recorded in FhirAugury for this ticket.

## Details

**Description:**

There are no links to Resource, DomainResource, MetadataResource or
CanonicalResource on the Types page, except in the diagrams. The
"Resources" link goes to the full list of resources.

Section 2.1.6.1.1 (https://build.fhir.org/uml.html#interfaces) indicates
that there are abstract "interface" types. However, section 1.2
(https://build.fhir.org/patterns.html#1.2) does not list any of the
interface types in its discussion of interfaces. (I suppose there may be
a distinction between interface types and interface patterns, but given
that both are fairly esoteric areas the duplicative language isn't
helpful.)

The Patterns page (https://build.fhir.org/patterns.html) does not show
up in the documentation index.

**Comments:**

No comments are recorded on the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a documentation defect and apply three coordinated
edits to the FHIR Core spec (HL7/fhir, source pages
`source/types.html`, `source/uml.html`, `source/patterns.html`, and the
documentation index/TOC source under `source/`):

1. On the Types page, change the prose so that the standalone words
   "Resource", "DomainResource", "MetadataResource", and
   "CanonicalResource" link to their respective definition pages
   (`resource.html`, `domainresource.html`, `metadataresource.html`,
   `canonicalresource.html`) in addition to the existing diagram
   hyperlinks. Leave the "Resources" link pointing at the resource list,
   but add adjacent inline links for the four abstract types so readers
   are not forced into the diagram to navigate.
2. On the UML page §2.1.6.1.1 (Interfaces), add a short cross-reference
   sentence pointing readers to Patterns §1.2 and explicitly clarify
   that "interface types" are the abstract base types listed there
   (Resource, DomainResource, MetadataResource, CanonicalResource), and
   that "interface patterns" on the Patterns page describe a related
   but distinct mechanism. Add the reciprocal cross-reference from
   Patterns §1.2 back to UML §2.1.6.1.1, and include the names of the
   interface types alongside the pattern discussion.
3. Add the Patterns page to the documentation index/TOC so it is
   discoverable from the spec navigation, not only via inbound links.

#### Justification

This is a low-risk, mechanical documentation fix that addresses the
exact issues the reporter raised. None of the changes alter normative
content; they only improve discoverability and remove ambiguity between
two adjacent concepts (interface types vs. interface patterns). The fix
is consistent with FHIR's general practice of cross-linking related
narrative pages and including authored pages in the documentation
index. Because the report cites three concrete, verifiable defects
(missing links, conceptual overlap, missing index entry), accepting as
requested gives implementers and authors a cleaner navigation surface
without requiring any community trade-off discussion.

---

### Disposition B: Alternative Approach

#### Proposal

Address the underlying need — "I cannot navigate to the abstract base
types and I cannot tell interface types from interface patterns" — but
do so via a small structural change rather than only inline edits:

- Introduce a new short narrative section on the Types page titled
  "Abstract base types" that lists Resource, DomainResource,
  MetadataResource, and CanonicalResource with one-line summaries and
  links to their definition pages. This becomes the canonical landing
  point for these abstract types and avoids relying on prose-internal
  links.
- Consolidate the discussion of "interfaces" by making one page
  (Patterns §1.2 is the more user-facing of the two) the single
  authoritative explanation, and reduce the UML §2.1.6.1.1 text to a
  pointer that says "the UML representation of the interface types
  defined in Patterns §1.2 is shown above". This eliminates the
  duplicative language the reporter complained about.
- Add Patterns to the documentation index as in Disposition A.

#### Justification

The reporter notes that "given that both [pages] are fairly esoteric
areas the duplicative language isn't helpful." Disposition A leaves the
duplication in place and only adds cross-links; Disposition B removes
the duplication by designating a single source of truth for the
"interface" concept and demoting the other site to a pointer. The new
"Abstract base types" section also gives implementers a stable URL to
cite, which is more durable than relying on inline prose links being
preserved through future edits. The trade-off is that this is a slightly
larger authoring change and requires a workgroup decision about which
page should own the interface narrative.

---

### Disposition C: Decline (Not Persuasive / Already Addressed)

#### Proposal

Close as Not Persuasive on the grounds that the abstract types are
already reachable through the UML diagrams on the Types page and
through the resource list, and that the Patterns page is reachable from
the pages that reference it. Treat the "interface types vs. interface
patterns" overlap as intentional layering (UML/structural view vs.
patterns/usage view) rather than a defect to be merged.

#### Justification

A defensible decline rests on two observations: (1) the information is
not missing — it is reachable, just not via the most obvious route;
and (2) the UML and Patterns pages are written for different audiences
(spec authors / implementers of the meta-model vs. resource designers),
so some apparent duplication is by design. Declining avoids spending
authoring cycles on what could be characterised as a polish issue, and
avoids the risk of a consolidation edit (Disposition B) accidentally
dropping nuance from one of the two pages.

---

### Recommendation

**Recommended disposition:** A

The ticket reports three concrete, low-controversy documentation
defects on FHIR Core authored pages: missing inline links to the four
abstract base types from the Types page, an unhelpful overlap between
UML §2.1.6.1.1 and Patterns §1.2, and a missing entry for the Patterns
page in the documentation index. Disposition A fixes all three with
minimal risk and no normative impact, which is exactly the kind of
change FHIR Infrastructure should be willing to land directly.
Disposition B is attractive in principle but requires the workgroup to
pick a "winner" between the UML and Patterns narratives, which expands
the scope beyond what the reporter actually asked for and is better
filed as a follow-up if reviewers feel strongly about the duplication.
Disposition C is hard to defend given the reporter's specific,
verifiable observations — the documentation index gap in particular is
a straightforward bug rather than a stylistic preference.
