## Summary

The ticket asks the FHIR Infrastructure work group to clarify how a future
mandate to support the `mode` parameter on the `capabilities` interaction
would be introduced if FHIR R6 ends up being balloted (and ultimately
published) as Normative. The current text in the HTTP page's
*capabilities* section says servers **MAY** ignore the `mode` parameter
and just return a `CapabilityStatement`, while also warning in a Note
that "servers might be required to support this parameter in further
versions of this specification." The reporter (Daniel Schuster) points
out that, under FHIR's normative-content rules, an open-ended promise
of a future tightening of conformance is awkward at best — once the
RESTful API and `CapabilityStatement` machinery are normative, changing
a `MAY` to a `SHALL` is a breaking change that cannot simply be slipped
in via a non-substantive technical correction. The ticket does not
propose a specific edit; it requests that the spec describe the
mechanism (versioning policy, mixed-normative status, profile-level
constraint, etc.) by which such a future tightening would actually be
introduced.

There were no Jira cross-references for this ticket in the cache, but
there is a directly relevant Zulip thread on
**fhir/infrastructure-wg > capabilities interaction** opened by Grahame
Grieve in August 2025 that flags exactly the same friction (a `SHALL`
to return a resource combined with a `MAY` to ignore the `mode`
parameter) and floats a different fix: tighten to `SHALL not ignore
mode` but allow servers to return an error for modes they don't
support. That thread is the most likely seed for the disposition
discussion.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an explanatory paragraph to the *capabilities* section of
`http.html` (immediately after the existing Note that warns the
parameter "might be required to support … in further versions") that
spells out the mechanism the work group intends to use if/when the
`mode` parameter is tightened. Concretely, state that:

1. Because the RESTful API and `CapabilityStatement` are intended to be
   normative in R6, the existing `MAY ignore the mode parameter`
   permission cannot be silently changed to a `SHALL` in a later
   technical correction.
2. Any future tightening will be introduced either (a) as a new
   conformance variant gated on `CapabilityStatement.fhirVersion` /
   instance metadata, (b) via a named conformance profile of the
   RESTful API (e.g., a "modal capabilities" capability) that servers
   declare support for, or (c) through the formal substantive-change
   process that reopens the relevant normative section in a future
   ballot.
3. The Note as currently worded should be reframed as an
   **informative forward-looking statement**, not a future conformance
   commitment — i.e., it warns implementers but does not pre-bind R6
   conformance.

No semantic change to the current `MAY ignore` rule; only the document
clarifies *how* the rule could change later.

#### Justification

This matches what the ticket literally asks for: clarification, not a
behavioural change. It also stays consistent with FHIR's normative
versioning rules (a `MAY` cannot become a `SHALL` without a
substantive-change ballot once the section is normative), with the
existing pattern of using `CapabilityStatement` itself to advertise
optional behaviour, and with the Zulip thread's observation that the
permissive `MAY ignore` is already widely deployed ("the horse has
probably bolted"). It does not pre-empt the more ambitious tightening
that Grahame floated — that conversation can continue independently —
and it lowers the risk that R6 ships with text that implies a future
breaking change without a documented path to it.

---

### Disposition B: Alternative Approach

#### Proposal

Take the Zulip thread's suggestion as the working draft and tighten the
*capabilities* section now, while it is still pre-normative in R6:

1. Replace "Servers MAY ignore the mode parameter and return a
   CapabilityStatement resource" with language along the lines of
   "Servers **SHALL** honour the `mode` parameter; a server that does
   not support a particular mode **SHALL** return an HTTP `4xx` error
   (e.g., `400 Bad Request` or `422 Unprocessable Entity`) with an
   `OperationOutcome` describing the unsupported mode."
2. Keep `mode=full` as the default (when `mode` is absent), so existing
   clients that never send `mode` continue to work unchanged.
3. Explicitly state that returning a `TerminologyCapabilities` resource
   is **only** required when the server has declared support for
   `mode=terminology`; servers that do not support it return an error
   for that mode rather than a `CapabilityStatement`.
4. Drop the "might be required in further versions" Note entirely,
   because the requirement is now in this version.

#### Justification

This addresses the underlying interoperability problem — a client today
cannot rely on `mode=terminology` returning a `TerminologyCapabilities`
because any server may legally ignore the parameter, which makes the
parameter close to useless on the open web — instead of just papering
over it with a clarification. Doing it pre-normative is the cleanest
window to make a `MAY → SHALL` change without invoking the substantive
ballot machinery later. The trade-off vs. Disposition A is real: it is
a **breaking change** for any server today that ignores `mode`. The
Zulip thread acknowledges this risk explicitly. Adopting B implies
willingness to absorb that breakage in exchange for a usable parameter;
choosing A defers the question.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" / decline, on the grounds that the
existing text is sufficient: the *capabilities* section already states
the current rule (`MAY ignore`), and the Note is intentionally
informative about a possible future direction. Any future tightening
will, as a matter of HL7 process, have to go through the normal ballot
and substantive-change rules — there is no need to bake a description
of that meta-process into the spec itself, because it is not specific
to the `mode` parameter.

#### Justification

A defensible position if the work group judges that (a) clients in
practice do not depend on `mode` being honoured, (b) the existing Note
is read by implementers as advisory only, and (c) every normative
section in FHIR carries the same implicit "future tightening goes via
ballot" rule, so calling it out for one parameter sets an awkward
precedent. The risk is that this leaves the same `SHALL`/`MAY` tension
flagged on Zulip unresolved, and ships R6 with an informative Note
that cannot actually be acted on as written.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the lowest-risk path that still gives the reporter
what was asked for. The ticket is explicitly a *clarification* request
("how this new requirement would be introduced"), not a request to
change conformance, so accepting a clarification edit is directly
responsive. Disposition B is technically more satisfying and aligns
with Grahame's own stated preference on Zulip, but it is a breaking
change against deployed servers and should not be slipped in under a
ticket whose title is "Clarify…" — if the work group wants to pursue
B, it should be tracked as its own substantive change request so that
implementers see it on the substantive-change list and have the
opportunity to comment in ballot. Disposition C does not address the
underlying drafting issue and leaves the Note pointing at a future
tightening with no documented mechanism, which is exactly the
unsatisfying state the ticket is calling out. Recommend the work
group adopt A now and, separately, decide whether to spawn a
follow-up ticket to take Grahame's "SHALL not ignore mode" proposal
through the substantive-change process.
