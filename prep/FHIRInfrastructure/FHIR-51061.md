## Summary

The reporter (Frank Oemig) notes that the FHIR Core specification does not
clearly demonstrate how to populate the `iso21090-ADXP-postBox` extension on
`Address.line`. Two plausible authoring patterns exist: (a) put a
human-readable, prefixed string such as `"Postbox 1234"` in the
`Address.line.value` while also carrying the structured post-box number in
the extension's `valueString`, or (b) put only the bare number `"1234"` in
`Address.line.value`. The ticket asks the work group to add a concrete
example to the Address examples on the *datatypes-examples* page so that
implementers have a single, normative reference for how to combine
`Address.line` text with the post-box ISO 21090 extension. The reporter
provides a worked sample using option (a):

```xml
<address>
  <line value="Postbox 1234">
    <extension url="http://hl7.org/fhir/StructureDefinition/iso21090-ADXP-postBox">
      <valueString value="1234"/>
    </extension>
  </line>
</address>
```

There are **no cross-references to other Jira tickets, GitHub items, or
Zulip threads** in the augury cross-reference index. A keyword search of
Zulip surfaces older (2016–2017) German-community discussions about
post-box handling in basis profiles, but no thread that directly addresses
this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a worked Address example to the FHIR Core *datatypes-examples* page
(`datatypes-examples.html`, in the Address subsection) that demonstrates
the `iso21090-ADXP-postBox` extension exactly as the reporter sketched:
the human-readable form `"Postbox 1234"` lives in `Address.line.value`,
and the structured value `"1234"` lives in the extension's
`valueString`. Mirror the example in JSON as well as XML for parity with
the surrounding examples on the page. Add a one- or two-sentence
narrative ahead of the example stating that `Address.line.value` carries
the rendered (display) form, while the ISO 21090 ADXP extensions carry
the parsed components, and that this same pattern applies to
`streetName`, `houseNumber`, `unitID`, etc.

#### Justification

This is the literal request from the ticket and is consistent with the
documented intent of the ISO 21090 ADXP extensions: the line.value is
the rendered/display form, while the ADXP extensions carry the parsed
parts. It also matches the older DACH community guidance and existing
real-world German base profile usage. The change is purely
example/narrative on a single page in FHIR Core, has no normative or
breaking impact, and gives implementers a single citable reference.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding only one example, add **two contrasting examples**
to the Address section of *datatypes-examples* and a short
"Authoring guidance" callout that explicitly states the recommended
convention. The two examples would be:

1. The reporter's pattern — `Address.line.value = "Postbox 1234"` plus
   the `iso21090-ADXP-postBox` extension with `valueString = "1234"`.
2. A multi-extension line — e.g., a street address that carries
   `iso21090-ADXP-streetName` and `iso21090-ADXP-houseNumber` extensions
   alongside a rendered `Address.line.value = "Hauptstraße 7"`.

The callout would state the rule in one sentence: "`Address.line.value`
SHOULD contain the human-readable rendered form; the ISO 21090 ADXP
extensions on that same `line` carry the structured components." Place
this guidance both on *datatypes-examples* (as a callout next to the
examples) and on the
`http://hl7.org/fhir/StructureDefinition/iso21090-ADXP-postBox`
extension definition page so implementers find it from either entry
point.

#### Justification

The ambiguity the reporter identified is not specific to `postBox` — it
applies to every ADXP line-component extension (`streetName`,
`houseNumber`, `unitID`, `postCode` is on Address itself, etc.). Showing
two examples and stating the rule once cures the whole family of
ambiguities for the same authoring cost as Disposition A, and putting
the callout on the extension page itself means implementers reading the
extension definition (which is how most will arrive at this question)
get the answer in place. The trade-off vs A is slightly more authoring
work and a small narrative addition to FHIR Core, but no normative
change.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for FHIR Core. Document that the ISO 21090
ADXP extensions are inherited semantics from ISO 21090 itself, point
implementers to the ISO 21090 specification and to existing
implementation guides (e.g., the German basisprofil-de) where worked
examples already exist, and close the ticket as "Persuasive with
Modification — see referenced material" without changing the FHIR Core
example pages.

#### Justification

This is the weakest position: the ticket asks for a tiny, non-normative
documentation improvement on an FHIR Core page that already exists for
exactly this purpose (`datatypes-examples`). The argument for declining
would be that ADXP semantics are ISO 21090's responsibility, but the
extensions live in the FHIR Core registry under
`http://hl7.org/fhir/StructureDefinition/iso21090-ADXP-*`, so passing
implementers to an external standard for their basic usage pattern is
unhelpful and inconsistent with how other examples on the page are
maintained.

---

### Recommendation

**Recommended disposition:** B

Disposition B costs essentially the same as A (it is still a
non-normative, examples-only change on `datatypes-examples.html` plus a
small note on the extension definition page) but it solves the whole
class of "what goes in `line.value` vs. in the ADXP extension" questions
instead of just the post-box case. The ticket's wording shows the
reporter has already decided that the rendered/structured split is the
intended pattern, and the older DACH Zulip thread shows the same
implicit convention in the wild — so codifying that rule once, with two
worked examples, is more useful to implementers than adding a single
post-box example. Disposition A is an acceptable fallback if the work
group prefers the minimum surface change. Disposition C should be
rejected because the extensions are FHIR-Core-published artifacts and
implementer confusion about their authoring is squarely something the
FHIR Infrastructure work group can and should address on the existing
examples page.
