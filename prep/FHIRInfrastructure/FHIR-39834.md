## Summary

FHIR-39834 reports a typo in one of the SMART on FHIR sequence diagrams: the
"access token request" arrow is drawn going to the FHIR Server, while the
"access token response" arrow originates from the EHR / authorization server.
The reporter (Frank Oemig) asks whether the request arrow should likewise
target the EHR / authorization server, since OAuth2 token endpoints live with
the authorization server, not the FHIR resource server.

This issue is effectively a **duplicate of two earlier, already-Published
"Persuasive" tickets** against the same SMART on FHIR sequence diagram:

- [FHIR-39655](https://jira.hl7.org/browse/FHIR-39655) — "Sequence diagram
  typo: App issues request to EHR authorization server for access token"
  (filed 2022-12-19 by Keith Carlson, ~2 weeks before FHIR-39834). Resolution
  Persuasive; fix delivered via
  [HL7/smart-app-launch PR #402](https://github.com/HL7/smart-app-launch/pull/402).
- [FHIR-41189](https://jira.hl7.org/browse/FHIR-41189) — "Wrong actor in
  sequence diagram" (filed 2023-05-04 by Josh Mandel, after FHIR-39834 was
  opened). Resolution Persuasive; pre-applied via commit
  [e1c36102](https://github.com/HL7/smart-app-launch/commit/e1c36102b5c07c49c2a6d05231ceff52f9f0c140).

All three tickets describe exactly the same diagram defect in the SMART App
Launch sequence; FHIR-39834 has simply not been swept up and dispositioned in
line with the other two.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Mark FHIR-39834 **Persuasive** as a duplicate of
[FHIR-39655](https://jira.hl7.org/browse/FHIR-39655) and
[FHIR-41189](https://jira.hl7.org/browse/FHIR-41189). Resolution text:
"The reported diagram defect (access-token *request* arrow drawn to the
FHIR Server rather than the EHR / authorization server) was already
corrected in `HL7/smart-app-launch` via PR #402 and commit e1c36102 in
response to FHIR-39655 and FHIR-41189. No further change is required.
Closing as a duplicate; the corrected diagrams are already Published in
the SMART App Launch IG that the SMART on FHIR (FHIR) section references."

No edits to spec sources are needed — the diagram fix is already in the
authoring repo (`HL7/smart-app-launch`) and Published.

#### Justification

The two related tickets describe the identical defect on the identical
diagram, were both resolved **Persuasive** and **Published**, and the
correcting PR / commit have been merged into the IG that the SMART on FHIR
(FHIR) section is built from. Re-doing the work would introduce churn with
no spec impact. Accepting-as-requested with a duplicate-pointer is the
honest, low-cost outcome and gives the reporter a clean trail back to the
already-applied fix.

---

### Disposition B: Alternative Approach

#### Proposal

Mark FHIR-39834 **Persuasive with Modification** and use the disposition as
the trigger to do a one-pass **diagram audit** across the SMART App Launch
sequence diagrams in `HL7/smart-app-launch` (EHR launch, Standalone launch,
backend services if applicable). The audit would (a) confirm that *every*
"access token request / response" pair correctly attributes the request to
the authorization server and the response from the authorization server,
(b) verify that the eight-step framing from
[FHIR-33271](https://jira.hl7.org/browse/FHIR-33271) is consistently
labeled across diagrams, and (c) check for the same mis-attribution on
introspection and refresh-token arrows. Any further corrections found are
folded into a single follow-up PR against `HL7/smart-app-launch`; the
ticket is closed once the audit is filed.

#### Justification

FHIR-41189 explicitly notes this was "a very long standing bug that has
survived multiple ballots." That history suggests reviewers tend to read
past these arrows. A one-pass audit catches the second-order risk that
analogous mis-labels exist in other sequence diagrams that nobody filed a
ticket about. Cost is bounded (one IG repo, diagram sources only). The
gotcha is that, without a current `repo-analysis` briefing for
`HL7/smart-app-launch`, the exact authoring paths and any
"do-not-edit / generated" boundaries cannot be cited here; that needs to
be resolved (run `repo-analysis HL7/smart-app-launch`) before the audit
PR is opened so the work doesn't accidentally edit a generated artifact.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-39834 **Not Persuasive – Duplicate** of FHIR-39655 / FHIR-41189
with no spec change and no further investigation. Close as a duplicate of
already-Published Persuasive tickets.

#### Justification

The defect has already been adjudicated twice (FHIR-39655, FHIR-41189),
both with Persuasive resolutions, and the corrective change has been
merged and Published. From a pure "should the spec change because of
this ticket?" perspective, the answer is "no — the spec already changed."
"Decline" is defensible on those grounds and removes the ticket from the
queue with minimum overhead. The downside is purely procedural: the
reporter receives a "Not Persuasive" label for what is in substance a
correct technical-correction request that the workgroup *did* accept,
just via different ticket numbers — which is unfriendly to outside
reporters.

---

### Recommendation

**Recommended disposition:** A

The defect is real, the workgroup has already agreed it is real (twice),
and the fix is already merged and Published in `HL7/smart-app-launch`. The
courteous, accurate disposition is **Persuasive – duplicate of FHIR-39655
and FHIR-41189**, with the resolution text pointing at PR #402 and commit
e1c36102 so the reporter can verify the corrected diagram. Disposition B
(broader diagram audit) is attractive given the "long-standing bug that
survived multiple ballots" comment from FHIR-41189, but it should be
opened as a *separate* ticket against the SMART on FHIR (FHIR)
specification rather than being smuggled into FHIR-39834's disposition,
and it should not happen until a current `repo-analysis` briefing exists
for `HL7/smart-app-launch`. Disposition C (Not Persuasive – Duplicate) is
procedurally cleanest but penalises a reporter who was, in fact,
correct; A delivers the same close-out with the right label.
