## Summary

Elliot Silver asks the FHIR Infrastructure work group to clarify what
`SearchParameter.target` actually *does* for `reference`-typed search
parameters: is it documentation only, or does it have a computational
filtering role on what the server is allowed to return? He also asks
what the meaning of the element is when it is missing.

He works through three concrete cases against a hypothetical
`SearchParameter` whose `target` lists `Patient | Practitioner | Group`
and whose expression covers `Encounter.subject`, `ServiceRequest.subject`,
and `Observation.subject`:

1. `Encounter.subject` permits a proper **subset** of the SP target list
   (`Patient | Group`). The current spec already covers this — listing
   more targets than can actually appear is explicitly allowed.
2. `ServiceRequest.subject` permits some types not in the SP target list
   and the SP target list permits some types not in
   `ServiceRequest.subject`. The current spec is silent on whether
   `Location` / `Device` may be returned in the absence of a narrowing
   FHIRPath.
3. `Observation.subject` permits a **superset** of the SP target list.
   Again the spec is silent on whether the result set is filtered down
   to the listed targets or whether the wider element-allowed set is
   returned.

The cross-referenced Zulip thread shows that the reporter and Gino
Canessa already discussed this on `#implementers > R5: missing
SearchParameter.target`; Gino's reading of the current normative text
is that incomplete `target` lists are explicitly allowed and that a
missing `target` is best treated as "any / Resource", but he agrees
nothing in the spec actually says so. The ticket asks the work group
to pick a position (filter vs. documentation), document the
meaning-when-missing, and update `SearchParameter.target` (and
related search guidance) accordingly. A side note in the description
observes that if `target` is treated as a filter, a number of
`somePath.where(resolve() is X)` expressions could collapse to
`somePath`, which might make it possible to disallow `resolve()` in
search expressions altogether.

There are no other Jira tickets cross-referenced to this issue.

## Details

**Description:**

> It isn't clear to me whether `SearchParameter.target` provides
> documentation of the behavior of the search param, or drives the
> behavior; or what the meaning is when missing.
>
> Consider a search parameter ("SP") of resource type that with target
> of Patient, Practitioner, Group and a context of `Encounter.subject`,
> `ServiceRequest.subject`, and `Observation.subject`.
>
> `Encounter.subject` allows a proper subset (Patient, Group) of the
> types that the `SP.target` lists. There is no question of returning a
> type not listed by `target`, and `SearchParameter.target`
> documentation makes clear that listing more types than may be returned
> is permitted.
>
> `ServiceRequest.subject` allows types (Patient, Group, Location,
> Device) not listed by the `SP.target` list, and the `SP.target` list
> includes types not allowed by `ServiceRequest.subject`. Again,
> according to the SearchParameter documentation, it isn't an issue
> that the `SP.target` allows types not included in
> `ServiceRequest.subject`. However, the SearchParameter documentation
> isn't clear on whether (in the absence of any FHIRPath to further
> narrow the returned content), search will return Location or Device
> resource or if the `target` element serves to filter them out.
>
> `Observation.subject` permits a superset of types listed in
> `SP.target`. It isn't clear (again, in the absence of further
> narrowing FHIRPath) of whether that search will only return types
> listed by `SP.target`, or whether it would return any of the types
> permitted by `Observation.subject`.
>
> If we accept that `SearchParameter.target` acts as a filter on the
> results, then any search that uses
> `someElement.where(resolve() is Patient)` could be simplified to
> `someElement` only. This might enable us to disallow use of
> `resolve()` completely in search.
>
> If the element has a filtering role (only types listed are permitted
> to be returned, but not all types listed need to be returned) that
> should be made clear. In this case, there seems to be an implied `*`
> / `Resource` / `Any` if the element is missing which should be
> documented.
>
> On the other hand, we could consider that `SearchParameter.target` is
> purely documentation, and actually has no role in what could be
> returned. Again there is an implication that any type could be
> returned if the element isn't present, but since it doesn't actually
> impact the result, we can be more relaxed about clarifying.
>
> Clarify if `SearchParameter.target` has a computational filtering
> role, or if it has a documentary role, and update documentation as
> needed.
>
> Linked: [#implementers > R5: missing SearchParameter.target](https://chat.fhir.org/#narrow/channel/179166-implementers/topic/R5.3A.20missing.20SearchParameter.2Etarget/with/544252819)

## Related Zulip Discussions

### [#implementers > R5: missing SearchParameter.target](https://chat.fhir.org/#narrow/stream/implementers/topic/R5%3A%20missing%20SearchParameter.target/near/544257939)

The thread originally opened in February 2023 with Eugene Blokhin
(Firely) reporting a long list of R5 reference-type SearchParameters
with no `target` element and asking whether `target` is required for
reference SPs. Grahame Grieve replied that this would be cleaned up in
the full R5 release; Louis Latour later checked the published R5
package and confirmed that effectively only
`http://hl7.org/fhir/SearchParameter/Resource-in` is left without a
`target`. Grahame and Alexander Zautke acknowledged that
`Resource-in` is "a bit weird" and that it probably should have been
typed as `special` rather than `reference`.

The thread was reopened in October 2025 by Elliot Silver, who pivoted
the question from "is `target` populated" to "what does `target`
actually mean": is it definitional (a filter on what the server may
return) or documentary, and what does an absent `target` imply? He
also asked whether a build/QA check should be added for R6. Gino
Canessa responded that he would *like* to argue `target` is
definitional but the current normative text says only that the list
SHOULD cover the possible targets and explicitly allows it to be
incomplete, and there is no defined meaning-when-missing — so in
practice he reads a missing `target` as equivalent to `* / Resource /
Any`. Elliot agreed the meaning-when-missing makes sense and pushed
on the harder question — when the listed `target` is a strict subset
of what the path's element type allows (`(A, B, C)` listed on an
element that allows `(A, C, D)`), is `D` ever returned? Gino's
reading was that yes, it can be: the SHOULD wording reads in both
directions, and `target` is therefore not a hard filter today.

No consensus was reached. Open questions left for the work group:
(1) is `SearchParameter.target` filtering or documentary;
(2) what is the meaning-when-missing; (3) should a build/QA check be
added for R6; (4) if `target` is treated as a filter, can common
`where(resolve() is X)` patterns (and possibly `resolve()` in search
expressions altogether) be simplified or deprecated.

Participants: Eugene Blokhin, Grahame Grieve, Louis Latour, Alexander
Zautke, Elliot Silver, Gino Canessa.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat `SearchParameter.target` as **documentary, not a hard filter**,
and tighten the normative text accordingly. Concretely, in the FHIR
Core specification:

- Update the element definition of `SearchParameter.target` (and the
  narrative on the SearchParameter resource page and the search page)
  to state explicitly that `target` documents the resource types a
  client can *expect* to see returned by the parameter, that the list
  MAY be a subset or superset of the types permitted by the
  parameter's expression, and that **servers are not required to
  filter results to the listed targets** — narrowing is done through
  the FHIRPath expression (e.g., `where(resolve() is Patient)`).
- Define meaning-when-missing for `SearchParameter.target` on
  reference-typed parameters as "equivalent to all resource types
  permitted by the expression" (i.e., `*` / `Resource` / `Any`,
  scoped by the path's allowed reference targets).
- Add a build/QA check (per Gino Canessa's note in the Zulip thread —
  "I'll add it to the list of things to check") that flags
  reference-typed `SearchParameter` resources without a `target` for
  manual review, rather than failing them outright. The check should
  whitelist the known-special case `Resource-in` (or, separately,
  retype `Resource-in` as `special` — see Disposition B).

#### Justification

This is the smallest change that resolves the ambiguity Elliot is
asking about, and it matches the current de-facto behavior: the
existing normative text already says the `target` list SHOULD cover
the possible types and explicitly permits it to be incomplete, which
is incompatible with treating `target` as a hard server-side filter.
Servers would otherwise need to break existing clients that rely on
incomplete target lists. Defining meaning-when-missing as "any"
matches Gino Canessa's reading in
[#implementers > R5: missing SearchParameter.target](https://chat.fhir.org/#narrow/stream/implementers/topic/R5%3A%20missing%20SearchParameter.target/near/544249521)
and aligns with how `Resource-in` and similar parameters appear to
work in practice today.

---

### Disposition B: Alternative Approach

#### Proposal

Make `SearchParameter.target` **prescriptive on the *upper bound* of
returnable types**: a server MUST NOT return references whose target
type is not listed in `SearchParameter.target` (when `target` is
populated), but the list MAY still be a subset of what the FHIRPath
expression can match (so `target` continues to be allowed to be
narrower than the element's permitted reference types). Update the
spec as follows:

- Reword `SearchParameter.target` to: "If present, results for this
  reference search parameter SHALL be limited to references whose
  target resource type is in this list. If absent, no
  type-based filter applies; results are bounded only by the
  expression."
- Define meaning-when-missing as "no target-based filter; equivalent
  to all resource types permitted by the expression".
- Add a non-normative note that, where `target` exhaustively covers
  the expression's reference targets, FHIRPath patterns of the form
  `someElement.where(resolve() is X)` can be replaced by `someElement`
  with an appropriately narrowed `target`. Do *not*, in this round,
  deprecate `resolve()` in search expressions — that is a larger
  question best handled in a follow-up ticket.
- Retype `http://hl7.org/fhir/SearchParameter/Resource-in` from
  `reference` to `special` to remove it from the population of
  reference parameters that "should" carry a `target`.
- Add the build/QA check from Disposition A, and tighten it: any
  *new* reference-typed SearchParameter introduced after R6 SHALL
  carry a non-empty `target`.

#### Justification

This is the option Elliot is implicitly steering toward in the
description ("If we accept that `SearchParameter.target` acts as a
filter…"). It gives implementers a single, computable contract
(`target` is the upper bound, expression narrows further), enables
the simplification of `where(resolve() is X)` patterns the ticket
calls out, and lines up with what most server implementations
intuitively do today. The trade-off vs. Disposition A is that any
existing SearchParameter whose `target` is *currently* incomplete
becomes potentially non-conformant in R6 — that is a real breaking
change, but it is bounded and locatable (Gino notes only six
reference SPs in the current CI build are missing `target`, and the
broader population of "incomplete but present" target lists can be
audited via the same build check). The retyping of `Resource-in`
matches the consensus on the Zulip thread (Alexander Zautke,
Grahame Grieve).

---

### Disposition C: Decline

#### Proposal

Decline the request as a normative change to FHIR Core. Instead,
mark FHIR-53110 as **Persuasive with Modification → Documentation**:
add a non-normative paragraph to the SearchParameter resource page
that walks through Elliot's three example cases (`target` is a
subset, an overlap, or a superset of the expression's permitted
types) and notes that the spec intentionally allows latitude here —
servers MAY treat `target` as advisory and rely on the FHIRPath
expression for filtering — without changing the conformance rules.

#### Justification

The current text already explicitly permits `target` to be
incomplete, which is incompatible with promoting it to a hard
filter (Disposition B), and "documentary only" (Disposition A) is
arguably already implied by the existing "SHOULD cover" wording. A
narrative clarification avoids any breaking change to existing
SearchParameter resources, IGs, and server implementations, and it
keeps the scope away from adjacent and contentious topics like
deprecating `resolve()` in search expressions. Defensible as "no
behavioral change is needed; the spec is just under-explained".

---

### Recommendation

**Recommended disposition:** B

Disposition B is the one that actually answers the question Elliot
is asking, and it is the only option that lets the work group act on
the secondary win the ticket calls out — collapsing
`someElement.where(resolve() is X)` patterns and tightening future
SearchParameter authoring. Disposition A leaves implementers in
substantively the same place they are today (an element whose
behavior is "documentary, kind of") and effectively concedes that
`target` will continue to be ignored by careful implementations,
which is a poor outcome for an element that exists in the resource
specifically to describe reference targeting. Disposition C is the
safest but it preserves exactly the ambiguity Elliot is reporting
and the work group will see this ticket again. The breaking-change
risk in B is real but small and locatable (per Gino Canessa's
audit, only six reference SPs in the current CI build are missing
`target`, and `Resource-in` is the recurring outlier that Alexander
Zautke and Grahame Grieve already agree should be retyped to
`special`); pairing the change with a build/QA check turns the risk
into a one-time cleanup rather than an ongoing source of
non-conformance. The work group should be prepared to push back on
the implicit follow-up ("should `resolve()` in search expressions be
deprecated?") and keep that scoped to a separate ticket.
