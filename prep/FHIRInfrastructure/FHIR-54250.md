## Summary

The reporter notes that on the FHIR R6 ballot 4 `CompartmentDefinition`
page (https://hl7.org/fhir/6.0.0-ballot4/compartmentdefinition.html),
section "5.7.2 Boundaries and Relationships" contains a hyperlink to
"GraphDefinition" that points at
`https://build.fhir.org/HL7/api-incubator/StructureDefinition-GraphDefinition`
and returns HTTP 404. The expected behavior is that the link resolve to
the published `GraphDefinition` resource page within the same
specification version. No cross-referenced Jira tickets, Zulip threads,
or GitHub items were found by the data layer, so the ticket stands on
its own as an isolated technical correction to the published HTML.

## Proposed Dispositions

Note: No GitHub cross-references were surfaced for this ticket, so no
repo briefing was loaded. The dispositions below identify the most
plausible authoring location based solely on the ticket content (the
`CompartmentDefinition` resource page in the core FHIR spec source). If
the workgroup adopts a code/spec change, the assignee should run the
`repo-analysis` skill against `HL7/fhir` (or the relevant publication
repo) to confirm the exact authoring root and file before making edits.

### Disposition A: Accept As Requested

#### Proposal

Fix the broken hyperlink on the `CompartmentDefinition` page so that the
"GraphDefinition" reference in section "5.7.2 Boundaries and
Relationships" resolves to the canonical, in-spec GraphDefinition
resource page (e.g., `graphdefinition.html` within the same publication,
which under the R6 ballot 4 build resolves to
`https://hl7.org/fhir/6.0.0-ballot4/graphdefinition.html`). The likely
authoring location is the `CompartmentDefinition` resource intro/notes
file in the core FHIR source repository (typically under
`source/compartmentdefinition/` in `HL7/fhir`); the current target,
`https://build.fhir.org/HL7/api-incubator/StructureDefinition-GraphDefinition`,
is an absolute pointer into an unrelated incubator publication and
should be replaced with a relative link (`graphdefinition.html` or
`[[GraphDefinition]]` per the publisher's link macro convention) so the
link tracks each published version of the spec.

#### Justification

This is a straightforward technical correction of the type the
"Technical Correction" issue type is designed for: the link target
returns 404, harming the published spec, and the fix is a one-line
source-file edit with no semantic impact on the normative content. Using
a relative / macro-expanded link (rather than an absolute URL into
`build.fhir.org/HL7/api-incubator`) is consistent with how every other
intra-spec resource cross-reference is authored on the
CompartmentDefinition page and across the rest of the core spec, and it
prevents the same class of breakage in future builds and ballots.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to fixing this single occurrence, treat the ticket as a
trigger for a broader sweep: grep the core spec source for any
hard-coded `https://build.fhir.org/HL7/api-incubator/...` URLs (and
similar absolute links to other branches such as
`build.fhir.org/branches/...`), replace them with relative or
macro-based links, and add a publisher / CI link-check step that fails
the build when a generated page contains an absolute URL pointing at an
incubator or branch build of another IG. The targeted CompartmentDefinition
fix from Disposition A is included as the first instance.

#### Justification

The specific target URL pattern (`build.fhir.org/HL7/api-incubator/...`)
strongly suggests this link was authored or auto-rewritten while the
GraphDefinition resource lived in an incubator IG and was not updated
when GraphDefinition moved/published into core. That class of bug almost
never occurs in isolation, and waiting for individual ballot comments to
surface each broken link is expensive for both reviewers and the
publisher. A one-time sweep plus a guard rule in the publication
toolchain is a higher-leverage fix and aligns with prior FHIR-I
preferences for systemic prevention over per-instance triage. The
trade-off versus Disposition A is scope: the fix touches more files and
likely the publisher tooling, so it has a longer review/merge cycle.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the broken target lives outside
the FHIR core publication (at `build.fhir.org/HL7/api-incubator/...`)
and is a transient artifact of an in-flight incubator branch rather
than a defect in the normative `CompartmentDefinition` content. Close as
"Not Persuasive" with a note that the link will self-heal once the
referenced incubator build is republished or the ballot is re-rendered.

#### Justification

Defensible only if the WG concludes that the offending URL is
auto-generated by the publisher from upstream metadata that has since
been corrected, in which case manual editing of the source would be
redundant. In practice this is the weakest line of argument, because
even if the upstream resolves, the *published* R6 ballot 4 HTML is
frozen and will continue to ship a 404 link to every reader of that
ballot snapshot.

---

### Recommendation

**Recommended disposition:** A

This is an unambiguous broken link in a published ballot snapshot, the
fix is a low-risk source edit with no normative impact, and the
"Technical Correction" type exists precisely for this scenario.
Disposition B is attractive as a follow-on quality improvement, but
bundling a publisher-tooling change and a source sweep into a single
ticket will slow down what should be a same-day fix; the sweep can be
filed as a separate FHIR Infrastructure tracker once A is committed.
Disposition C is not defensible because the broken link is baked into a
published ballot artifact that real reviewers are reading right now.
