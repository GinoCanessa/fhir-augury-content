## Summary

The ticket asks FHIR Infrastructure (FHIRPath maintainers) to tighten and
clarify the prose of the "Restricted Subset" section of the FHIRPath
specification (§2.1.9.2 of `fhirpath.html`, also known as the "simple"
subset). The reporter, Brian Postlethwaite, lists six concrete editorial
issues:

1. The opening sentence ("This page documents a restricted subset of the…")
   is ambiguous about scope — page vs. section.
2. The narrow grammar should be stated explicitly: only child element names
   (recursively) plus indexers (e.g. `[0]`) are permitted.
3. The note about `OperationOutcome.issue.expression` should be amended to
   say that, when used there, the language is further constrained to
   disallow **all** FHIRPath functions — only property navigation and
   indexers are allowed.
4. `DataRequirement.dateFilter` appears to use an even tighter grammar
   than the simple subset (`ofType` and `extension` are missing); the
   editor should confirm whether this is intentional.
5. A note should be added that other implementation guides MAY also adopt
   the restricted Simple subset.
6. Several worked examples should be added, e.g. `Patient.birthDate`,
   `Patient.name[0].family`, `Patient.name[2].given[1]`, and
   `QuestionnaireResponse.item[0].item[2].answer[0].value`.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items recorded for FHIR-53966. The cross-reference index returned only
the three FHIR-artifact mentions that appear inline in the example
expressions (`Patient.birthDate`, `Patient.name`,
`QuestionnaireResponse.item`). Background context, however, is well
documented in a recent FHIRPath stream discussion (see *Related Zulip*
below) in which the reporter himself articulates the design intent of
the simple subset: a property-path-and-indexer-only grammar that allows
implementations to navigate to a specific node in a known instance
without a full FHIRPath engine, e.g. for use in
`OperationOutcome.issue.expression`.

## Related Zulip Discussions

### [fhirpath > Observation.component.value for given component.code](https://chat.fhir.org/#narrow/stream/fhirpath/topic/Observation.component.value%20for%20given%20component.code/near/526148132)

Elliot Silver asks how to use the "restricted (simple)" subset of
FHIRPath to point at the value of a particular `Observation.component`
(e.g. the systolic component of a blood-pressure observation). Brian
Postlethwaite — the reporter of FHIR-53966 — replies that the simple
subset is intentionally an *explicit, direct path to nodes of a known
instance*, not a query language: "there are no operators (e.g. `=`) and
almost no functions, specifically no `where`". He notes that **indexers
are allowed**, and that this is exactly how the simple subset is used in
`OperationOutcome.issue.expression` (e.g. `Observation.component[1].value`).
Grahame Grieve and Bryn Rhodes branch into a side discussion about
whether the modelling problem is better solved with a `TriggerDefinition`
or a new optional element on `DeviceAlert`/`Device-alertDetection`. The
thread reaches consensus that, for instance-anchored references, the
simple subset is appropriate and that indexers are part of it; this
matches almost word-for-word the clarifications the ticket is asking the
spec to make explicit. Participants: Brian Postlethwaite, Elliot Silver,
Grahame Grieve, Bryn Rhodes.

### [implementers > slice list of References by values in referred resources?](https://chat.fhir.org/#narrow/stream/implementers/topic/slice%20list%20of%20References%20by%20values%20in%20referred%20resources%3F/near/527687433)

Chris Moesel reminds the asker that "slice discriminators use a
restricted subset of FHIRPath, but `resolve()` is one of the allowed
operators." This is a useful reminder that the *slicing-discriminator*
restricted subset is **not** the same restriction as the §2.1.9.2 simple
subset (the discriminator subset allows `resolve()`, `extension(url)`,
`ofType(...)`; the simple subset allows essentially only property
navigation and indexers). The ticket does not propose merging the two,
but the spec edit should be careful not to blur them. Participants:
Volker Wegert, Chris Moesel, Gino Canessa.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply all six clarifications to §2.1.9.2 "Restricted Subset" of
`fhirpath.html` in HL7/fhir, and to `OperationOutcome.issue.expression`
guidance:

1. Reword the opening sentence to refer to the **section** rather than
   the page (e.g., "This section documents a restricted subset of the
   FHIRPath language…").
2. Add a normative-style sentence stating that the simple subset
   permits only (a) child-element navigation (recursively), and
   (b) integer indexers of the form `name[n]` on collection-valued
   elements; no functions, no operators, no path expressions other
   than these two constructs are permitted.
3. Update the `OperationOutcome.issue.expression` element definition /
   usage note to call out that, when used in this element, FHIRPath
   is further constrained to disallow **all** functions — only child
   property navigation and integer indexers are allowed (matching
   Brian's Zulip explanation that this is how it is already used in
   practice).
4. Editorial review of `DataRequirement.dateFilter`: either explicitly
   document its narrower grammar (no `ofType`, no `extension`) as
   intentional, or relax it to align with the simple subset.
5. Add a sentence stating that other implementation guides MAY adopt
   the restricted Simple subset for their own elements when an
   instance-anchored, function-free path expression is appropriate.
6. Add the worked examples verbatim from the ticket
   (`Patient.birthDate`, `Patient.name[0].family`,
   `Patient.name[2].given[1]`,
   `QuestionnaireResponse.item[0].item[2].answer[0].value`).

#### Justification

These are all editorial/clarifying changes, not normative behaviour
changes — the underlying grammar is unchanged. Each item closes a
documented implementer pain point: indexer support is currently inferred
rather than stated (Elliot Silver explicitly admitted on Zulip he hadn't
realised indexing was allowed); the function-free constraint on
`OperationOutcome.issue.expression` is enforced in practice but not in
prose; the discriminator-subset vs. simple-subset confusion is recurring
across multiple Zulip streams (analytics-on-FHIR, shorthand,
implementers); and the IG-reuse note merely codifies what is already
happening informally. The reporter is a well-known FHIRPath SME and
authored the prevailing community explanation in Zulip, so the
proposed wording mirrors established community understanding rather
than introducing new design.

---

### Disposition B: Alternative Approach

#### Proposal

Accept items 1, 2, 3, 5, and 6 as editorial-with-clarification, but
**split out item 4 (`DataRequirement.dateFilter`)** into a separate
ticket targeted at the relevant work group (Clinical Quality
Information / Clinical Decision Support — owner of `DataRequirement`)
rather than handling it inside an FHIR-Infrastructure FHIRPath
clarification. In addition, replace item 2 with a more structured
re-write of the section: instead of one sentence, introduce a
mini-grammar block (e.g. ABNF or a numbered "permitted constructs"
list) so that the simple subset is unambiguous to implementers writing
lightweight parsers — the stated motivation Brian gave on Zulip ("permit
usage without a full fhirpath engine, just walking property names and
indexers").

#### Justification

`DataRequirement.dateFilter` belongs to a different module and a
different work group's editorial backlog; bundling it risks the whole
ticket stalling on cross-WG sign-off, and the answer is genuinely "ask
CQI/CDS whether the tighter restriction is intentional" rather than
something FHIR-I can decide unilaterally. Replacing the freeform note
in item 2 with a small grammar block addresses the recurring
implementer confusion shown across the Zulip evidence (multiple
implementers conflating the discriminator subset with the simple
subset) far more durably than another paragraph of prose. The remaining
items are uncontroversial and can land together.

---

### Disposition C: Decline

#### Proposal

Decline as "Not Persuasive — already documented" on the grounds that
the existing prose at `fhirpath.html#simple`, plus the
`OperationOutcome.issue.expression` element definition, plus the
discriminator section under profiling, collectively already specify the
permitted constructs, and that adding more prose risks introducing new
inconsistencies between sections (especially around the
`DataRequirement.dateFilter` and discriminator-subset edge cases).
Suggest instead that the reporter open targeted ballot comments on any
specific sentences he finds genuinely ambiguous.

#### Justification

This disposition is defensible only narrowly: it leans on the fact
that no normative behaviour is being changed and that the grammar is
discoverable from existing text. However, it is weak — the Zulip
evidence (Elliot Silver's "I hadn't realized that indexing was
allowed", Chris Moesel's repeated need to remind implementers that the
discriminator subset is *not* the simple subset, Grahame Grieve's note
that the definition is hard to find) directly contradicts the premise
that the existing prose is sufficient.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach).

The substance of the request is clearly correct — the Zulip record
shows real, repeated implementer confusion that these clarifications
would resolve, and the reporter is the same person whose informal
Zulip explanations the spec edit would essentially codify, so
Disposition C is hard to defend. Disposition A is the cleanest "say
yes" answer, but it has two soft spots: (i) item 4 is genuinely a
question for the `DataRequirement` owners and forcing FHIR-I to
adjudicate it inside this ticket invites a stall, and (ii) item 2 is
better served by a small, well-structured grammar block than another
paragraph of English prose, given the documented pattern of
implementers confusing the simple subset with the discriminator
subset. Splitting `dateFilter` out and tightening item 2 keeps the
useful editorial work moving while routing the one cross-WG question
to the right group.
