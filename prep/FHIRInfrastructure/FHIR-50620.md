## Summary

The reporter has identified a cluster of small editorial defects in the
narrative of the `Subscription` resource page (FHIR R5 / Core), in
sections 2.12.1, 2.12.7.3, 2.12.9.2, and 2.12.9.3. The defects are all
prose-level: a transposed clause that drops the reference to
`SubscriptionTopic` while garbling the filter-criteria sentence in
2.12.1; a stray duplicate "the" in 2.12.7.3; multiple mark-up and
wording problems in 2.12.9.2 (the literal `rest-hook` not formatted as
code, a duplicated "and and", inconsistent code/plain rendering of
"notification Bundle", and the malformed phrase "per the format
requests" which should read "per the format requested"); and an
apparently incorrect endpoint name in 2.12.9.3 step 12, where the
narrative says "Client Endpoint" but seemingly should say "Server
Websocket Endpoint" given the surrounding context about reconnecting
when `websocket-url` changes. No comments have been added to the ticket
since it was filed, no other Jira tickets cross-reference it, and no
Zulip threads or GitHub items mention it. The single discovered
cross-reference is the FHIR artifact `Subscription.endpoint`, which is
mentioned in the body of the ticket as part of the original text being
corrected, not as an element-level change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the four editorial corrections to the narrative of the
`Subscription` resource page exactly as the reporter suggests:

1. **Section 2.12.1** — Reword the sentence to:
   "Once a subscription is created, any triggering state change for the
   specified `SubscriptionTopic` that meets the filtering criteria
   specified in the `Subscription` will cause a notification to be sent
   using the provided channel." This restores the missing preposition
   and keeps the explicit reference to `SubscriptionTopic` while making
   it clear that the filter criteria live on the `Subscription`.
2. **Section 2.12.7.3** — Remove the stray leading "the" so the sentence
   reads: "For example, if the notification is for a topic about patient
   admission, the subscriber may require related `Observation`
   resources."
3. **Section 2.12.9.2** — Make four narrow markup/wording fixes:
   (a) format `rest-hook` as code in both prose occurrences;
   (b) collapse "and and" to a single "and";
   (c) render "notification Bundle" consistently with the rest of the
       page (treat "Bundle" as a code-rendered resource name and
       "notification" as plain prose, e.g. "notification `Bundle`");
   (d) change "per the format requests in the Subscription" to "per the
       format requested in the Subscription".
4. **Section 2.12.9.3, step 12** — Change "the Client establishes a new
   connection to the Client Endpoint" to "the Client establishes a new
   connection to the Server Websocket Endpoint", matching the
   `websocket-url` semantics described earlier in the same section.

These are pure narrative changes; no element definitions, profiles,
operations, or examples are affected.

#### Justification

Every item the reporter raised is a verifiable defect in the rendered
spec text — broken grammar, duplicated words, inconsistent mark-up, or
an internally contradictory endpoint name. Technical Correction is the
intended ticket type for exactly this class of editorial cleanup, and
fixing them in place keeps the ballot-quality bar consistent with the
surrounding Subscription/SubscriptionTopic narrative. There is no
behavioural change, no schema impact, and no compatibility risk, so the
cost/benefit favours simply applying the fixes as written.

---

### Disposition B: Alternative Approach

#### Proposal

Accept items 2.12.1, 2.12.7.3, and 2.12.9.2 verbatim, but for
2.12.9.3 step 12 take a slightly broader pass on the WebSocket
sub-section instead of just renaming "Client Endpoint" to "Server
Websocket Endpoint" in step 12. Concretely:

- Audit every use of "Client Endpoint", "Server Endpoint", and
  "Websocket Endpoint" in 2.12.9.3, choose one canonical phrase for the
  server-side WebSocket URL (almost certainly "Server Websocket
  Endpoint" given existing usage), and apply it consistently across the
  whole sub-section, not only step 12.
- While doing so, double-check that the surrounding steps that hand
  out / rotate `websocket-url` are unambiguous about which side opens
  the new connection.

The other three section fixes are applied identically to Disposition A.

#### Justification

The reporter's read of step 12 is almost certainly correct, but
"Client Endpoint" vs. "Server Websocket Endpoint" terminology drift in
the WebSocket section has bitten implementers before, so a one-line
swap risks leaving the rest of the sub-section internally inconsistent.
A tightly-scoped pass on the WebSocket terminology costs little extra
editor time, gives reviewers a single coherent diff, and reduces the
chance of a follow-up Technical Correction ticket against the same
sub-section. The trade-off vs. Disposition A is a slightly larger diff
that needs a second pair of eyes on the WebSocket flow; the trade-off
vs. C is obvious.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the changes
are purely editorial, the meaning of each affected sentence is still
recoverable by a careful reader, and FHIR Infrastructure has higher-
priority Subscription work in flight; ask the reporter to refile any
genuinely ambiguous item (most plausibly 2.12.9.3 step 12) as a
narrowly-scoped Technical Correction once that work has settled.

#### Justification

This disposition is only defensible if the WG is explicitly triaging
out all narrative-only Technical Corrections against the Subscription
page in the current cycle. Even then it is a weak position: items 2 and
3 in 2.12.9.2 ("and and", "the if the") are unambiguous defects, and
declining them invites the same ticket to be refiled by another
reviewer. The chief upside is editorial throughput — no spec edits, no
publisher rerun, no reviewer cycles — but in the FHIR Core context that
is a small saving.

---

### Recommendation

**Recommended disposition:** A

Every item in the ticket is a small, low-risk, narrative-only fix to
the `Subscription` resource page, and the reporter has done the work of
locating each defect, quoting the surrounding text, and proposing the
exact correction. Disposition A clears the entire ticket in a single
editorial pass with no behavioural impact and no risk of regression.
Disposition B is a reasonable refinement only if a reviewer is already
willing to spend extra time on 2.12.9.3, and can be folded into A at
edit time without changing the WG decision. Disposition C is hard to
defend given that several of the issues ("and and", "the if the") are
plain typos that the WG would otherwise have to re-litigate when the
next reader files them.
