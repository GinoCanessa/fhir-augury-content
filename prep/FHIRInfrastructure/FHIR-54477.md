## Summary

Brian Postlethwaite is reporting two related gaps in the FHIR Mapping
Language (FML) specification around the so‑called "simple identity
transform". First, the published FML grammar does not formally allow
the most compact identity form
`src -> tgt: type, subtype, action, recorded;`, in which the right‑hand
side is just a list of properties to be copied 1‑for‑1 from the source
to the target. Second, the surrounding prose in the FML spec does not
adequately explain that identity transforms are the implicit basis for
all property creation in a map — i.e. that simply naming a property as
a target is enough to instantiate it (except when the type is ambiguous,
such as for choice types). The reporter is asking that this simple form
be (a) made grammatically legal, (b) given its own numbered sub‑heading
so it can be referenced, (c) better explained in narrative, including
how it is round‑tripped to/from `StructureMap`, and (d) moved earlier in
the page so readers encounter the common case before the more advanced
forms.

There are no cross‑referenced Jira tickets, Zulip threads, or GitHub
items associated with FHIR‑54477 in the indexed sources at the time of
this prep, so the scope is bounded to the FML page and its grammar.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Mapping Language specification page (`mapping.html`)
and the FML grammar artifact in **HL7/fhir** to:

1. Extend the FML grammar so that the right-hand side of a rule may be
   a bare comma-separated list of property names that exist on **both**
   the source and target types, e.g.
   `src -> tgt: type, subtype, action, recorded;`. Define this as
   syntactic sugar for the equivalent set of explicit
   `src.type -> tgt.type` (etc.) identity rules. Update the ANTLR/PEG
   grammar definition, the railroad diagrams generated from it, and
   any FML parser test fixtures shipped with the spec.
2. Promote the existing "Simple Form" content (and the new
   property-list form) into a numbered sub-heading early in the
   `mapping.html` narrative — before the longer-form rule examples —
   so it can be cross-referenced (e.g., `mapping.html#simple-form`).
3. Add explanatory prose stating that an identity transform is the
   implicit creation mechanism for any target property: simply naming
   a property on the target side instantiates it, and an explicit
   `create` is only required when the type is ambiguous (e.g., a
   `value[x]` choice).
4. Document the canonical `StructureMap` representation of the simple
   form, showing the expanded `StructureMap.group.rule` JSON/XML so
   round-trip behaviour between FML text and StructureMap resources is
   unambiguous.

#### Justification

The change is purely additive at the syntax level (new sugar over
existing rule semantics) and additive at the documentation level, so
it is backward compatible: every map that is legal today remains
legal, and the expanded `StructureMap` form is unchanged. Codifying the
property-list shorthand in the grammar matches how implementers
already write maps in practice and removes a class of "spec-vs-tooling"
confusion. Re-ordering and numbering the simple form aligns the
narrative with the FHIR documentation pattern of leading with the
common case (used elsewhere on resource pages, search parameter pages,
etc.) and gives the workgroup a stable anchor to cite from other
specs and from Jira tickets.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the request as **two separable items** and accept only the
documentation half in this cycle:

- Do **not** extend the FML grammar. Keep the canonical form as
  individual rules per property, but add a new, numbered "Simple
  Form / Identity Transform" sub-section near the top of
  `mapping.html` that:
  - Explicitly states that naming a property as a target implicitly
    creates it, calling out the choice-type exception.
  - Shows a worked example of the existing legal short form and the
    `StructureMap` it produces.
  - Notes the property-list sugar as a **proposed future extension**
    tracked under this ticket, rather than landing it now.
- Defer the grammar change to a follow-up ticket against the next
  version, gated on (a) reference-implementation support
  (org.hl7.fhir.core, matchbox, HAPI mapping engine) and (b)
  agreement on the StructureMap serialisation.

#### Justification

The grammar change, however small, has downstream impact on every FML
parser, validator, and authoring tool in the ecosystem (org.hl7.fhir.core,
matchbox, HAPI, the IG publisher, etc.). Splitting the work lets the
documentation improvements — which are the higher-leverage, lower-risk
half of the ticket — ship immediately and be cited by other tickets,
while the grammar extension goes through a normal implementation /
review cycle without holding up the prose fix. It also avoids landing
syntax that tooling does not yet accept, which would otherwise create
maps that round-trip cleanly through the spec's examples but fail in
production engines.

---

### Disposition C: Decline

#### Proposal

Decline the grammar extension and decline the structural reorganisation
of `mapping.html`. Close the ticket as "already addressed" on the
basis that the existing per-property short form
(`src.x -> tgt.x;` repeated, or use of `for` / `then` blocks) already
covers the identity case, and the existing prose around implicit
creation, while terse, is technically present.

#### Justification

This is the weakest position but is defensible if the workgroup
considers FML grammar to be effectively frozen mid-cycle and prefers
to avoid any churn on a syntax that has multiple independent
implementations. Declining keeps the grammar stable and avoids forcing
parser updates across org.hl7.fhir.core, matchbox, HAPI, and the IG
publisher just to support sugar that does not add expressive power.

---

### Recommendation

**Recommended disposition:** B

The documentation half of this ticket is unambiguously valuable: a
numbered "Simple Form / Identity Transform" sub-heading early in
`mapping.html`, with an explicit statement that naming a target
property is enough to create it (choice-type exception aside) and a
worked `StructureMap` example, costs the workgroup very little and
gives every downstream IG and tutorial a stable anchor to cite. That
should land now. The grammar extension, by contrast, touches every
FML parser in the ecosystem and deserves its own implementation
agreement before it is normative; deferring it under a follow-up
ticket — explicitly noted in the new sub-section as a known proposed
extension — captures the reporter's intent without surprising tooling
maintainers. Disposition A is attractive but couples a low-risk doc
fix to a multi-implementation grammar change; Disposition C ignores a
real reader-experience problem that the reporter has surfaced from
authoring maps in practice.
