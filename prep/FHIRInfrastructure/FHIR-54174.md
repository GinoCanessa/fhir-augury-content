## Summary

The reporter (Brian Postlethwaite) observes that the FHIRPath specification
has been updated since the references currently embedded in the FHIR Core
specification were written, and asks that FHIR be updated so it refers to

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Core specification so that every reference to the
FHIRPath specification points to the current published version. Concretely:

- Bump the FHIRPath version string and citation in the FHIRPath usage
  page in FHIR Core (the page that introduces FHIRPath and explains how
  it is used in invariants, search parameters, and `$evaluateMeasure`-
  style operations).
- Update the external link(s) that resolve to `hl7.org/fhirpath` /
  `build.fhir.org/ig/HL7/FHIRPath` so they target the latest *published*
  release of the FHIRPath spec, rather than a pinned older version.
- Refresh any normative reference list / bibliography entry for FHIRPath
  in the FHIR Core publication so that the version, publication date,
  and URL match the current FHIRPath release.
- Verify that no inline narrative accidentally describes behaviour that
  has changed between the previously-referenced FHIRPath version and the
  new one (e.g., changes to `as`, `is`, type system, or
  `defineVariable`); if any wording is now inconsistent with the newer
  spec, correct it as part of the same change.

#### Justification

The request is editorial in spirit: FHIRPath is an external HL7
specification that FHIR cites normatively, and stale citations are a
documentation hygiene issue. Pointing readers and implementers at the
current FHIRPath release is consistent with how FHIR Core treats other
external HL7 references (terminology, data types specs), reduces the
risk that implementers code against an older FHIRPath grammar by
accident, and avoids implementers seeing two different "current"
FHIRPath versions depending on whether they read the FHIRPath site or
the FHIR Core spec. Because FHIRPath releases generally aim to be
backwards-compatible, bumping the citation is unlikely to introduce
behaviour changes for existing implementers — it mostly aligns the
documentation surface with reality.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) hard-coding a specific newer version,
change the way FHIR Core cites FHIRPath so it refers to "the latest
published version of FHIRPath" via a stable, version-agnostic URL (e.g.,
the canonical `hl7.org/fhirpath/` landing page that always redirects to
the current release), and call out an explicit *minimum* version of
FHIRPath that conformant FHIR implementations must support. The
bibliography entry would then carry both: the canonical URL plus the
minimum-version statement.

#### Justification

This addresses the same underlying need — keeping the reference fresh —
but avoids re-opening the same ticket every time FHIRPath cuts a new
release. It also makes the conformance contract clearer to implementers:
they know which features they may rely on (the minimum version) and
where to find the latest spec (the canonical URL). The trade-off vs.
Disposition A is that "latest" links can be ambiguous if a reader needs
to reproduce exactly what the FHIR ballot saw, so a minimum-version
clause is essential to keep the citation precise enough for normative
reference.

---

### Disposition C: Decline

#### Proposal

Do not change the FHIRPath reference at this time. Note on the ticket
that the FHIR Core specification deliberately pins to a specific
FHIRPath version per ballot cycle, that bumping the reference outside a
ballot cycle is out of scope for an editorial change, and that the
update should instead be picked up as part of the next FHIR Core
publication's planned refresh of normative external references.

#### Justification

A defensible reason to decline would be that changing a normative
external reference mid-cycle has substantive implications: implementers
and downstream IGs may have built tooling around the previously cited
FHIRPath grammar, and a quiet bump of the cited version blurs which
FHIRPath version a given FHIR publication was actually balloted against.
On this view, the right place to refresh the reference is the next
formal FHIR Core publication, where the bump is reviewed alongside any
behavioural deltas in FHIRPath. This rationale is weaker for a purely
editorial bump where FHIRPath is backwards-compatible, but it is the
honest case for declining now.

---

### Recommendation

**Recommended disposition:** B

The underlying problem the ticket is flagging — that FHIR Core's
FHIRPath citation lags the actual current FHIRPath spec — is real, and
Disposition A (just bump the version) fixes it once. But "the latest
version of the fhirpath specification" is, almost by definition, a
moving target: any future FHIRPath release will recreate the same
ticket. Disposition B addresses the same need *and* the recurrence: cite
the canonical FHIRPath URL plus an explicit minimum-version requirement,
and the FHIR Core spec stays correct across future FHIRPath releases
without further ballot-cycle work. The cost is small (a wording change
on the FHIRPath usage page and the bibliography entry, plus a one-time
agreement with the FHIRPath maintainers that the canonical URL is
stable). Disposition C is hard to defend for what is fundamentally an
editorial / link-hygiene fix on a backwards-compatible external
reference.
