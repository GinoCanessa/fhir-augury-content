## Summary

The ticket asks FHIR Infrastructure to add a new paragraph of guidance to
the Bundle batch processing rules in the core specification clarifying
that `batch` entries SHOULD NOT have interdependencies between entries
even when those interdependencies do not cause changes on the server
(i.e., even for read-only / operation-style entries). Today the spec
only forbids interdependencies that "cause change on the server" (the
classic SHALL about multiple updates/patches/deletes to the same
resource). That wording does not cover a newer class of problem:
operation-style batch entries (notably `$validate-code`) where one
entry's behaviour can be implicitly influenced by parameters supplied
on another entry (for example, sharing a single `tx-resource` ValueSet
across the rest of the batch to avoid >1 GB payloads). Because the
batch language is normative, the reporter explicitly proposes new
SHOULD-level guidance rather than a stronger SHALL, and warns clients
that they cannot rely on servers honouring such cross-entry
relationships. The request is closely related to
[FHIR-51050](https://jira.hl7.org/browse/FHIR-51050) (Terminology
Infrastructure), which proposes a brand-new `$validate-code-batch`
operation as the longer-term solution to the underlying terminology use
case that motivated this discussion.

## Details

**Description:**

The documentation about the relationships between batch entries says this:

> For a `batch`, there SHALL be no interdependencies between the
> different entries in the Bundle that cause change on the server. E.g.
> multiple entries making updates, patches, or deletes to the same
> resource with would be considered interdependencies. The success or
> failure of one change SHOULD not alter the success or failure or
> resulting content of another change. Servers SHOULD validate that
> this is the case. Note that it is considered that servers execute the
> batch in the same order as that specified below for transactions,
> though the order of execution should not matter given the previous
> rule.

That fails to deal with interdependencies between entries in regard to
operation parameters. Ideally, we would like to prohibit this, but it's
too late to add SHALL rules about this because of normative. So propose
to add a new paragraph:

> "Batches SHOULD not have interdependencies between entries such that
> they cannot be processed independently, irrespective of whether they
> cause change on the server. Note that prohibiting this was omitted in
> the past, so the rule cannot be stated more strongly, but clients are
> not able to rely on servers correctly processing explicit or implied
> relationships between e.g. parameters to operations on different
> entries in the batch."

## Related Jira Tickets

### [FHIR-51050: Define a new operation $validate-code-batch](https://jira.hl7.org/browse/FHIR-51050)

Filed by Grahame Grieve immediately after FHIR-51049 was opened, this
companion Terminology Infrastructure ticket proposes a new
`$validate-code-batch` operation that explicitly allows a single
`tx-resource`/ValueSet to be shared across many code validations in
one call. It is the long-term, semantically-clean replacement for the
batch-with-shared-tx-resource pattern that triggered FHIR-51049, so the
two tickets are deliberately complementary: FHIR-51049 strengthens the
guidance that batch entries must be independent, while FHIR-51050
provides the sanctioned alternative for the use case that the new
guidance forbids.

Reporter / Assignee: Grahame Grieve. Commenter: carmela_couderc (noted it
should be reclassified as a Change Request). Status: *Waiting for
Input*; Work Group: *Terminology Infrastructure*.

## Related Zulip Discussions

### [Terminology Service Test Cases > Batch](https://chat.fhir.org/#narrow/stream/Terminology%20Service%20Test%20Cases/topic/Batch/near/523159474)

Long thread (~130 messages, June–July 2025) that motivated the ticket.
Grahame Grieve explained that he had added a test case where a single
`tx-resource` parameter is supplied on the **first** entry of a
`$validate-code` batch and is then expected to be applied to every
subsequent entry. He acknowledged this conflicts with the spirit of
batch semantics but said it was operationally necessary because
repeating the ValueSet on every entry produced batches >1 GB that
servers reject under DoS limits. Jim Steel and Michael Lawley both
pushed back: batch entries are supposed to be independent and
re-orderable, allowing per-entry parameters to silently affect other
entries would surprise implementers and is "not done anywhere else".
Several alternative shapes were floated and rejected — a
`$validate-codes` plural form (no one liked it), reusing the existing
`cache-id` mechanism (Lawley historically declined it; Grahame
clarified how he uses session establishment via `/metadata`), and
Alexander Kiel's proposal of a generic `$terminology-batch` or a
temporary "bucket" of canonical resources that could replace both
`tx-resource` and `cache-id`. The conversation reached the conclusion
that a dedicated batch operation is the right path; that triggered the
FHIR-Infrastructure call, FHIR-51049 (this ticket — strengthen the
guidance), and FHIR-51050 (the new operation). A follow-up sub-thread
in July 2025 polled the wire-format choice for the new operation
(single `Parameters` with nested sub-parameters vs. nested `Parameters`
resources). Consensus on the guidance side was clear: batch entries
must remain independently processable, so the proposed SHOULD wording
is welcome.

Participants: Grahame Grieve, Jim Steel, Michael Lawley, Alexander Kiel,
Bart Decuypere, Reuben Daniels.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add the proposed paragraph verbatim (or with light editorial polish) to
the Bundle batch processing rules in `source/http.html` (or wherever the
batch/transaction processing rules currently live in `HL7/fhir`),
immediately after the existing SHALL paragraph about interdependencies
that cause change on the server. Concretely:

> "Batches SHOULD NOT have interdependencies between entries such that
> the entries cannot be processed independently, irrespective of
> whether the entries cause change on the server. Note that
> prohibiting this was omitted in the past, so the rule cannot be
> stated more strongly, but clients SHALL NOT rely on servers
> correctly processing explicit or implied relationships between (for
> example) parameters to operations on different entries in the
> batch."

Because the surrounding text is normative, the change should be flagged
as a clarifying SHOULD-level addition (not a substantive change to the
existing SHALL) and introduced via the standard core-spec change
process.

#### Justification

This is exactly the editorial reinforcement implementers and server
authors asked for on Zulip. It closes a real gap — operation-style
batch entries with cross-entry parameter dependencies are not covered
today, and the absence of guidance is being interpreted in conflicting
ways. Keeping the new rule at SHOULD respects the normative status of
the existing batch text while still giving server authors a citation
when they refuse to honour cross-entry dependencies. It also aligns
neatly with FHIR-51050, which provides the sanctioned operation-level
alternative for the original use case, so accepting this guidance and
the new operation together gives clients a clear migration path
("don't do this in batch — use `$validate-code-batch` instead").

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent but reshape the wording so that it (1) explicitly
calls out *operation-style* entries as the in-scope failure mode, (2)
is symmetric for clients and servers, and (3) cross-references the
forthcoming `$validate-code-batch` operation (FHIR-51050) and the
existing `tx-resource` / `cache-id` mechanisms as the right tools for
batched terminology work. For example:

> "In addition to the SHALL above, batch entries SHOULD be processable
> independently of each other in every respect — including operation
> parameters supplied to entries that do not change server state.
> Servers SHOULD NOT treat a parameter (such as `tx-resource`) on one
> batch entry as applying to other entries, and clients SHOULD NOT
> author batches that depend on such cross-entry semantics. Use cases
> that require shared inputs across many sub-requests (for example,
> validating many codes against a single supplied `ValueSet`) SHOULD
> use a purpose-built operation such as
> [`$validate-code-batch`](valueset-operation-validate-code-batch.html)
> rather than a `Bundle` of type `batch`."

#### Justification

Disposition A's wording is faithful to the reporter's text, but it is
abstract ("interdependencies … irrespective of whether they cause
change on the server"). The Zulip discussion shows the actual confusion
is concrete: people are sharing `tx-resource` across `$validate-code`
batch entries. Spelling that out in the spec, and pointing to the
correct alternative, will be far more useful to implementers than a
generic SHOULD. It also keeps FHIR-51049 and FHIR-51050 visibly linked
in the published specification, which mirrors how the workgroup
actually intends the two changes to be used together. The trade-off is
a slightly longer paragraph and a forward reference that must be kept
in sync with FHIR-51050's resolution.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-51049 on the grounds that the existing batch text already
implies entry independence (the SHALL says success/failure of one
change SHOULD NOT alter the success/failure or resulting content of
another, and notes that ordering should not matter), and that the real
remediation belongs in the operation-level fix (FHIR-51050 —
`$validate-code-batch`) rather than in additional Bundle-level prose.

#### Justification

A pure-prose SHOULD that says "do not depend on a thing the spec never
told you you could depend on" arguably doesn't add normative force; it
risks being read as a substantive change to a normative section by
careful reviewers, and any implementation that was already exploiting
cross-entry semantics will continue to do so regardless. Once
FHIR-51050 lands, the legitimate motivating use case has its own
operation, and any future server that quietly honours `tx-resource` on
a sibling entry can be told "that's a non-conformant extension; use
`$validate-code-batch`."

---

### Recommendation

**Recommended disposition:** B

Disposition A is the safe, minimal answer the reporter asked for, and
Disposition C is defensible on normative-stability grounds, but
Disposition B captures what the workgroup and the Zulip thread
actually want from this ticket. The Zulip discussion makes clear that
the confusion is not "what does interdependency mean in the abstract"
— it is specifically about operation parameters such as `tx-resource`
being silently shared across `$validate-code` batch entries, with
implementers (Steel, Lawley, Kiel) firmly opposed and the reporter
acknowledging the practice is questionable. A SHOULD-level addition
that names that pattern and points readers to `$validate-code-batch`
(FHIR-51050) gives implementers a single, citable place in the spec
that resolves the ambiguity, keeps the two related tickets visibly
coupled in the published HTML, and avoids re-litigating this on every
future ticket. The risk — keeping the forward reference in sync with
FHIR-51050 — is small and easily managed via the existing change
tracking. FHIR Infrastructure should accept the spirit of the request
but adopt the more concrete wording in Disposition B.
