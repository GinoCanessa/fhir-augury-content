## Summary

The reporter identified a small typographical error in the FHIR Core
specification's narrative explaining partial version matching for
canonical references. The current text reads "A partial **verion** was
specified e.g. `http://example.org/CodeSystem/abc|1.2` matches both
versions 1.2.1 and 1.2.3-draft" — the word "verion" is missing its `s`
and should read "version". The ticket asks that an `s` be added so the
sentence reads correctly. No cross-referenced Jira tickets, Zulip
discussions, or GitHub items were found for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a one-character editorial fix to the FHIR Core specification
narrative wherever the string "A partial **verion** was specified"
appears (most likely the page describing canonical references with
version pinning, e.g. the references / canonical-URLs narrative or the
CodeSystem discussion of partial version matching). Change "verion"
to "version" so the sentence reads: *"A partial version was specified
e.g. `http://example.org/CodeSystem/abc|1.2` matches both versions
1.2.1 and 1.2.3-draft."* No semantic change, no schema change, no
publisher-tooling change.

#### Justification

This is unambiguously a typo in narrative text. The surrounding sentence
already uses "version" / "versions" correctly twice; the bolded
"verion" is clearly an orphaned misspelling of the same word. Fixing
typos in published specification narrative is exactly the purpose of the
*Technical Correction* issue type and carries effectively zero risk:
no normative content, no examples, no profiles, and no test artifacts
are affected. Accepting matches the reporter's explicit request and
removes a small but distracting blemish from the spec.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the typo fix as in Disposition A, **and** while the page is
open, sweep the immediately surrounding paragraph(s) for any sibling
issues — e.g., the doubled smart quotes (`""…""`) the reporter quoted
verbatim, redundant `&nbsp;` runs, or any other near-by misspellings —
and clean those up in the same editorial commit. Keep the change
strictly editorial (no normative or example changes), and group it
under a single "spec narrative cleanup" commit referencing FHIR-50449.

#### Justification

A Technical Correction commit that touches a single page costs the
same to review and ballot as one that fixes the typo plus a few
neighbouring nits. Bundling avoids opening a second ticket later for
the same paragraph and is consistent with how editorial sweeps are
normally handled. The risk profile is identical to Disposition A as
long as changes remain confined to narrative text.

---

### Disposition C: Decline

#### Proposal

Decline as not worth a tracked correction: close the ticket with a
note that purely cosmetic single-character typos can be fixed
out-of-band by an editor without going through the Technical
Correction workflow.

#### Justification

The change has no normative impact and no implementer is misled by the
sentence — "verion" is obviously a misspelling of "version" in
context. Some projects prefer to keep the issue tracker focused on
substantive corrections and let editors fix typos directly. This is
defensible but weak: the ticket already exists, the reporter took the
time to file it, and resolving it costs less than declining and
explaining why.

---

### Recommendation

**Recommended disposition:** A

This is a textbook Technical Correction: a one-character fix to
narrative text with zero normative impact, zero downstream tooling
impact, and an unambiguous, reporter-supplied diff. Disposition B is
reasonable if an editor happens to be in the same paragraph anyway,
but it should not block accepting the fix, and the additional cleanup
should not be a precondition. Disposition C wastes the reporter's
goodwill for no gain. Accept the change, apply the `s`, and close the
ticket.
