## Summary

The reporter asks SDC to provide a standardized way for a Questionnaire renderer
to display a Questionnaire item (or a group of items) inside a *dialog* (a modal
pop-up surface). The motivation comes from the Android FHIR SDK, where the
client wants a dialog presentation even when the number of choices is small
enough that the SDK would otherwise pick radio buttons or a drop-down. Two
mechanisms are proposed: (1) a new `dialog` extension, separate from
`questionnaire-itemControl`, so a single item can declare both `dialog` and
another control such as `radio-button`; or (2) adding a `dialog` code to the
existing `questionnaire-item-control` CodeSystem (rejected by the reporter
because `itemControl` is 0..1 and so cannot be combined with another control
code on the same item). The use cases are radio buttons displayed in a dialog
and free-form content (analogous to the `using-contextexpression` SDC example)
displayed in a dialog.

There are no cross-referenced Jira tickets, so the request stands alone. It was
originally voted *Considered for Future Use* on 2024-11-21 (Brian
Postlethwaite/Ilya Beda, 7-0-0, "Will consider this when we have more details.
Not for inclusion in the upcoming ballot") and was re-opened by Lynn Laakso on
2026-03-26 "post STU4 publication".

## Details

**Description:**

> **Is your feature request related to a problem? Please describe.**
>
> In FHIR, it's currently not possible to specifically render a dialog. The
> dialog can have varying content, it can contain input type of widgets (radio
> button, text field, etc) or even simply displaying texts.
>
> **Describe the solution you'd like**
>
> Introduce the dialog extension (http://hl7.org/fhir/StructureDefinition/dialog)
> to FHIR.
>
> **Describe alternatives you've considered**
>
> Introduce the `dialog` type to FHIR itemControl
> (https://build.fhir.org/ig/HL7/fhir-extensions/CodeSystem-questionnaire-item-control.html).
> But the 0..1 cardinality makes it not possible to declare both `dialog` and
> `radio-button` on the same item (See use case 1 where we need both `dialog`
> and `radio button`).
>
> **Use cases (See attached example)**
>
> 1. Radio buttons that are displayed in a dialog
>    (https://m3.material.io/components/dialogs/overview)
> 2. Content that are displayed in a dialog
>    (https://build.fhir.org/ig/HL7/sdc/examples.html#using-contextexpression)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new `dialog` extension to the SDC IG (URL
`http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-questionnaire-dialog` or
similar in the SDC canonical space, **not** the core `http://hl7.org/fhir/`
namespace the reporter proposed). Context: `Questionnaire.item`. Cardinality
0..1. Type: complex extension with sub-extensions to answer the open design
questions raised by Brian Postlethwaite and Lloyd McKenzie:

- `trigger.label` (string, 0..1) — label of the button that opens the dialog.
- `submit.label` and `cancel.label` (string, 0..1 each) — button labels;
  presence of `cancel.label` implies a Cancel control exists.
- `cancelDiscards` (boolean, 0..1, default `true`) — whether closing with
  Cancel discards captured data.
- `evaluateExpressionsOn` (code, 0..1; values `submit` | `change`) — when
  `calculatedExpression`, `enableWhen`, etc. are re-evaluated for items inside
  the dialog.
- `summaryExpression` (Expression, 0..1) — FHIRPath/CQL expression yielding the
  string shown on the parent form to summarise what was captured inside the
  dialog.

Allow the extension to coexist with `questionnaire-itemControl` (since it lives
on a different extension URL, the existing 0..1 on `itemControl` is not a
blocker), so an item may declare both `dialog` and (e.g.) `radio-button`. Apply
to both `group` items (use case 2: dialog as a group container, aligning with
Ye Wang's framing) and question items with answerOptions/answerValueSet (use
case 1).

#### Justification

This is the path the Zulip discussion converged toward: Paul Lynch agreed that
a separate extension "sounds better" once the cardinality problem on
`itemControl` was raised, and the remaining objections (Brian, Lloyd) are
about *underspecification*, not about the feature itself. By specifying
trigger label, button labels, cancel semantics, expression evaluation timing,
and a summary expression up front, the proposal directly answers the four
considerations Brian listed in his 2024-10-24 comment and addresses Ye Wang's
"how do you reopen / is it modal" concerns. Living in the SDC IG (rather than
in core FHIR as the reporter literally requested) is consistent with where the
rest of the questionnaire rendering vocabulary already lives and avoids
elevating a UI hint into the base spec — which would directly contradict
Lloyd's "Dialog says pop-up, that's not what we're going for here" concern at
the *core* level while still making the feature available to SDC consumers.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new extension. Instead, add a single new code `dialog` to the
`questionnaire-item-control` CodeSystem (under the `Group` category alongside
`page`, `tab-container`, and `list`), and explicitly scope it to group items
only. To satisfy use case 1 (radio buttons in a dialog), require authors to
wrap the choice item in a single-child `group` whose `itemControl` is `dialog`;
the inner choice item retains its own `itemControl` (`radio-button`,
`drop-down`, etc.). The 0..1 cardinality on `itemControl` is then not a
blocker, because each level carries its own single code. Document the dialog's
behaviour (modal, opens on tap of the group's text/prefix, follows the host
platform's dialog conventions, all expressions inside the dialog evaluate
continuously as in the rest of the form) in the SDC rendering guidance rather
than introducing new sub-extensions.

#### Justification

This matches Ye Wang's framing — `dialog` is "essentially same as `page` and
`tab-container`": a group container, not an input control — and reuses the
existing extension point and code system that renderers already consume. It
avoids inventing a new complex extension and the surface area that comes with
it (sub-extensions, cancel semantics, expression-evaluation modes), which Brian
identified as the "awful lot of complexities." It also gracefully degrades: a
renderer that does not understand the `dialog` code falls back to rendering the
group inline, with no loss of data or interoperability. The trade-off versus
Disposition A is that the more nuanced behaviours the reporter wants (a Cancel
button that discards entries, a summary string shown on the parent form,
explicit OK semantics) are *not* expressible declaratively — they would be
purely renderer-defined — which is consistent with Brian's position that
itemControl is "a hint" and not a behavioural contract.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for SDC. Document in the resolution that the desired
behaviour is achievable today by the renderer alone: the Android FHIR SDK (and
any other renderer) is free to display a `choice` item in a modal dialog
whenever it judges that to be the right UX, exactly as it already does for the
"≥10 options" case. No spec change is required. If the workgroup wants to
codify guidance, it can add a non-normative note to the SDC rendering section
saying that renderers MAY present any item in a modal dialog when appropriate
to the platform.

#### Justification

This is the position Brian Postlethwaite advanced repeatedly on Zulip — "In a
phone you could make that decision anyway without having something in the item.
The UI hints are just that" — and Lloyd McKenzie's terse "Dialog, to me, says
'pop-up', and that's not what we're going for here" points the same way. The
ticket has been open since September 2024 with no concrete answers to the four
design questions Brian listed, and was originally voted Considered For Future
Use 7-0-0 precisely because the workgroup felt it lacked detail. Declining
keeps SDC out of the business of dictating modal-vs-inline rendering, avoids
inventing semantics (cancel-discards-data, when-do-expressions-fire) that would
have to be supported by every conformant renderer, and preserves the
established principle that itemControl values are hints — not commands — to
the renderer.

---

### Recommendation

**Recommended disposition:** B

The Zulip thread's strongest *technical* argument is Ye Wang's: a dialog is a
container, structurally equivalent to `page` and `tab-container`, and therefore
belongs in the existing `questionnaire-item-control` CodeSystem at the Group
level. Disposition B captures that insight, satisfies both stated use cases by
wrapping in a single-child group when needed, sidesteps the `itemControl` 0..1
cardinality complaint without introducing a new extension surface, and
gracefully degrades on renderers that don't recognise the code. It also avoids
committing the SDC IG to specifying cancel semantics and expression evaluation
timing — the exact "awful lot of complexities" Brian flagged as unresolved —
while still giving the reporter a standardized way to declare the intent. If
the workgroup later finds that the nuances in Disposition A (named buttons,
cancel-discards behaviour, summary expression) are essential for
interoperability between SDC implementers, those can be layered on as separate
small extensions targeted at the dialog group, without having to undo
Disposition B. Disposition A is acceptable but heavier than the evidence on
file justifies; Disposition C is defensible (and was effectively the original
2024-11-21 outcome) but the re-open in March 2026 signals the workgroup is
ready to revisit, so a concrete proposal is more useful than another decline.
