## Summary

This is a follow-on technical correction to [FHIR-54064](https://jira.hl7.org/browse/FHIR-54064)
("Clarify base64 encoding in Json Signature Requirements"), which was resolved
Persuasive and added the explanatory note: *"NOTE: Because JWS is already
base-64 encoded, this means that the signature will be double-encoded."* to the
description of the `Signature` data type's `Signature.data` element.

The reporter argues the new note is technically inaccurate: per RFC 7515, JWS
uses **base64url** (URL-safe, unpadded) encoding, not standard **base64**.
Because `Signature.data` is itself `base64Binary`, the wording is both
imprecise and potentially confusing — implementers who read "JWS is already
base-64 encoded" and "Signature.data is base64 encoded" may reasonably
question why double-encoding is required at all. The ticket asks for a small
edit to swap "base-64" for *base64url* in the note (and drop the italics
emphasis), so the rationale for double-encoding becomes obvious: the inner
encoding is base64url, the outer (FHIR `base64Binary`) is base64.

## Details

**Description:**

The resolution to [FHIR-54064](https://jira.hl7.org/browse/FHIR-54064) Clarify
base64 encoding in Json Signature Requirements - Jira says that the JWS is
base-64 encoded.

Can we change the note from:

> NOTE: Because JWS is already base-64 encoded, this means that the signature
> will be double-encoded.

To:

> NOTE: Because JWS is already *base64url* encoded, this means that the
> signature will be double-encoded.

Emphasis not needed.

Primary reason is that base64url and base64 are meaningfully different from
my understanding. And even though the note adds the technical clarification
needed (the data will be double encoded), I could see the current language
leading to folks not wanting to double encode because it says JWS is base-64
encoded and the data needs to be base64 encoded — which leads to the natural
question of why does it need to be double encoded if they're already encoded
the same way.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the exact textual edit requested by the reporter to the `Signature`
data type description (the spec page that documents `Signature.data`).
Replace the existing note:

> NOTE: Because JWS is already base-64 encoded, this means that the
> signature will be double-encoded.

with:

> NOTE: Because JWS is already base64url encoded, this means that the
> signature will be double-encoded.

Drop the italic emphasis on *base64url* — render in the same style as
surrounding prose. This is a Technical Correction touching only the
`Signature.data` element description; no schema, structure definition,
cardinality, binding, invariant, or example needs to change.

#### Justification

RFC 7515 §3 and §7.1 are unambiguous that the three components of a JWS
Compact Serialization (header, payload, signature) are each `BASE64URL(...)`
encoded — explicitly URL-safe and unpadded, distinct from standard `base64`
(RFC 4648 §4 vs §5). FHIR's own `base64Binary` primitive is standard base64.
Conflating the two in the very note that exists to *clarify* double-encoding
is internally contradictory and reintroduces the confusion FHIR-54064 was
meant to remove. The change is editorial, non-substantive, backward
compatible, costs nothing for implementers, and is consistent with how the
Security and Privacy WG (e.g., John Moehrke on Zulip) already speaks about
JWS encoding. Accepting as requested resolves the issue cleanly with the
smallest possible change.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substance of the request but expand the note slightly to remove
all residual ambiguity, rather than swap a single word. Replace the existing
note with something like:

> NOTE: A JWS Compact Serialization is itself `base64url`-encoded
> (RFC 7515 §3.1). Because `Signature.data` is FHIR `base64Binary` (standard
> base64 per RFC 4648 §4), the JWS string is base64-encoded a second time
> when it is placed into `Signature.data`. Implementations must perform this
> outer base64 step; they must not "re-use" the inner base64url bytes.

Edit the same `Signature` data type page; still no structural change.
Optionally add a short non-normative example showing an outer-encoded value
alongside its decoded JWS form.

#### Justification

The reporter's core concern is that the current wording invites the wrong
implementer behavior (skipping the outer base64). A one-word swap fixes the
factual error but still leaves a terse note that depends on the reader
spotting the difference between "base64" and "base64url". A slightly fuller
note explicitly names both RFCs, ties the inner encoding to JWS and the
outer encoding to FHIR's `base64Binary`, and states the implementation
obligation directly. The cost is a few extra sentences in the same place
already being edited; the benefit is that a future ticket like FHIR-55573
becomes much less likely. Trade-off vs. A: marginally larger text-diff and
slightly higher review effort, with better long-term clarity.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the note added by FHIR-54064 already
states the operative requirement ("the signature will be double-encoded") and
that the term "base-64" is colloquially used to cover the base64 family
(including base64url) in many specifications. Mark FHIR-55573 as
*Not Persuasive — Already Addressed*.

#### Justification

Defensible only on a minimum-disruption argument: FHIR-54064 was just
resolved, the note achieves its operational purpose (telling implementers to
double-encode), and reopening the wording invites bikeshedding. However,
this position is weak — RFC 7515 itself takes care to distinguish base64url
from base64, and the `Signature.data` page sits next to FHIR `base64Binary`
prose that uses "base64" in its strict sense. Declining preserves a
demonstrable inaccuracy in normative-adjacent text, which is hard to defend
for a Technical Correction.

---

### Recommendation

**Recommended disposition:** B

The reporter is technically correct, and Disposition A would already be a
clear improvement over the status quo. However, the underlying reason
FHIR-54064 was filed in the first place — and the reason FHIR-55573 now
exists — is that terse encoding notes around `Signature.data` keep tripping
implementers up. A one-word swap from "base-64" to "base64url" fixes the
present error but leaves the same shape of note that just needed a second
correction. Spending a few extra sentences to (a) name both RFCs, (b)
explicitly contrast the inner JWS base64url with the outer FHIR
`base64Binary`, and (c) state the implementation obligation directly closes
this loop for good. The change is still purely editorial within the
`Signature` data type page, carries no compatibility risk, and aligns with
how the Security and Privacy community already discusses JWS on Zulip. If
the workgroup prefers the lightest possible touch, fall back to Disposition
A — but the cost of the slightly fuller wording in B is small relative to
the cost of yet another follow-up ticket on the same sentence.
