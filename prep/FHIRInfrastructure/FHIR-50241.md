## Summary

The ticket asks the SMART App Launch / User Access Brands (UAB) authors to
relax the cardinality constraint on `Organization.telecom` in the UAB
profile from `1..1` to `1..*` (the description text mentions "0..*" in
the title but the body and the originating Zulip thread make clear the
real ask is to allow more than one telecom). The motivation is telehealth
providers: a UAB-published `Organization` for a telehealth-only facility
is most usefully reached via more than one contact channel (e.g., a
public website *plus* a scheduling phone number or patient portal URL),
which the current `1..1` constraint forbids. The reporter and other
participants on the originating Argonaut Zulip thread cannot recall a
documented design reason for limiting the slot to a single value, and
the discussion concluded with the reporter offering to file this Jira so
the constraint can be revisited. There are no other Jira tickets or
GitHub PRs cross-referenced to this issue; the only cross-reference is
the originating Zulip thread.

## Details

**Description:**

> We currently restrict Organization.telecom to be 1..1.  We should relax that to be 1..*.
>
> Based on [this chat.fhir thread](https://chat.fhir.org/#narrow/channel/179175-argonaut/topic/User.20Access.20Brands.20-.20telehealth.20locations.3F/with/511710162), we don't remember why we limited it to 1..1.  And for telehealth providers, including telecom details (e.g., phone number or portal to schedule telehealth visits) in addition to the primary public website is useful.

Cooper Thompson opened the thread asking how the UAB bundle should
represent purely-telehealth facilities — for which a physical address is

constraint of `Organization.telecom 1..1` blocks publishing both a

organizations should appear in the UAB bundle at all and, if so, what
shape they should take (e.g., "no address, scheduling details in
telecom") — which is adjacent to but separate from the cardinality fix
this ticket targets. Sam Ashby reported their implementation hadn't hit
the issue because they only publish a website per organization, and
Brett Marquard confirmed he could find no documented rationale for the
`1..1` choice in his notes from the IG's earlier "FHIR Endpoint and
Structure → Patient Access Brands → User Access Brands" lineage. No
opposition to relaxing the cardinality was voiced on the thread.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the UAB profile constraint on `Organization.telecom` from `1..1
MS` to `1..* MS` in the SMART App Launch / User Access Brands
publication (the IG that owns the `OrganizationBrand`-style profile
referenced by this ticket — work group: FHIR Infrastructure,
specification: SMART on FHIR). Concretely: edit the UAB Organization
profile differential so the `telecom` element's `min` stays at `1` and
`max` becomes `*`; keep the Must Support flag; do not change the slot's
type, binding, or any sub-element constraints. Add a short usage note in
the profile narrative explaining that publishers SHOULD include the
public website as one entry and MAY include additional contact channels
(e.g., scheduling phone, patient portal URL) — particularly for
telehealth-only organizations. No changes are required to base FHIR
Organization, which is already `0..*` on `telecom`.

#### Justification

The change is consistent with the base `Organization.telecom` cardinality
(`0..*`), so no implementer who already supports the base resource will
be surprised by a profile that allows multiple entries. No participant
on the originating Zulip thread could recall or document a design
rationale for the `1..1` limit, and the only speculative reason offered
(rendering simplicity) is a UI concern that clients can address by
choosing which entry to surface, not a data-model concern that requires
the profile to forbid additional entries. The change is non-breaking:
existing UAB publications that include exactly one telecom remain valid,
and consumers that today read `telecom[0]` continue to work. The use
case (telehealth providers needing both a website and a scheduling
contact) is concrete and was raised by an active UAB implementer.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Organization.telecom 1..1 MS` on the UAB profile but introduce a
slicing on `telecom` (discriminator: `system`) with two named slices:
`website` (system `url`, `1..1 MS`) and additional optional slices such
as `phone` (system `phone`, `0..*`) and `portal` (system `url` with a
purpose/use distinguishing it from the primary website, `0..*`). This
preserves the existing "exactly one canonical contact" guarantee for
the primary website while letting telehealth-only organizations layer in
scheduling endpoints. Alternatively, leave `telecom` at `1..1 MS` and
add a UAB-defined extension (e.g., `uab-scheduling-contact`) carrying
`ContactPoint`, used specifically for telehealth scheduling.

#### Justification

If the `1..1 MS` choice was, in fact, intentional — to give clients a
deterministic "primary contact" to render — sliced cardinality preserves
that guarantee while still admitting the new use case. The slicing
approach also makes the *meaning* of additional entries explicit
(scheduling vs. general phone vs. fax), which a flat `1..*` does not.
The trade-off is added profile complexity and additional implementation
burden for both publishers (must populate the website slice) and
consumers (must understand the discriminator). The extension variant is
even more conservative but pushes telehealth scheduling out of the
standard `telecom` discovery path, which is likely worse for consumer
ergonomics.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep `Organization.telecom 1..1 MS` on the UAB
profile, on the basis that the canonical "User Access Brands" contact
is intended to be the publisher's single primary public-facing channel
(the brand website), and that telehealth scheduling endpoints belong on
a different resource (e.g., `Endpoint`, `HealthcareService`, or
`Location.telecom`) rather than on the brand `Organization` itself.

#### Justification

UAB is explicitly a *brand* publication, not a service directory; mixing
operational scheduling contacts into the brand `Organization`'s
`telecom` slot risks blurring that distinction and may push consumers to
treat UAB as an authoritative scheduling-discovery surface, which it was
not designed to be. There is also a precedent for keeping brand-level
metadata small and predictable so that consumer UIs (e.g., patient app
brand pickers) can render uniformly. Declining would, however, leave
telehealth-only publishers without a standard place to surface their
scheduling endpoint within the UAB bundle today.

---

### Recommendation

**Recommended disposition:** A

The originating Zulip thread surfaces a real implementer pain point
(telehealth-only organizations) and finds no documented design rationale
for the current `1..1` constraint, with multiple participants —
including the IG's long-standing contributors — independently saying
`1..*` is what they would have expected. The change is non-breaking,
aligns the profile with base `Organization.telecom` (`0..*`), and is
the smallest edit that resolves the use case without inventing new
slicing or extension scaffolding that future UAB consumers would have
to learn. Disposition B's slicing model is defensible if the workgroup
decides on review that the `1..1 MS` was indeed a deliberate
"single-primary-contact" guarantee worth preserving, but absent any
record of that intent the simpler relaxation is the better default.
Disposition C should only be considered if the workgroup actively wants
to scope telehealth scheduling out of UAB entirely, which contradicts
the implementer feedback in the linked thread.
