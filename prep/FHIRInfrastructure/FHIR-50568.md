## Summary

The reporter identifies a typographical/markup error on the FHIR R6
ballot 3 page `domainresource.html` in the "Additional Resources"
section. An example XML snippet on that page closes with `</Bundle>`
when the resource being shown is a `Basic` resource and the closing
tag should therefore be `</Basic>`. This is a pure documentation /
narrative correction in the FHIR Core specification — no schema,
StructureDefinition, profile, or normative content is affected. There
are no cross-referenced Jira tickets, Zulip threads, or GitHub items
associated with this ticket in the indexed sources, so the request
stands on its own as a small editorial fix.

## Details

**Description:**

> Update XML closing tag from `</Bundle>` to `</Basic>`
> https://hl7.org/fhir/6.0.0-ballot3/domainresource.html#additional

The referenced page anchor (`#additional`) is the "Additional
Resources" sub-section of the `DomainResource` page, which discusses
how to convey resources that are not yet standardised (typically via
the `Basic` resource and an extension on `code`). An illustrative XML
example in that section opens a `<Basic …>` element but mistakenly
closes the element with `</Bundle>`, producing an XML snippet that is
both ill-formed and conceptually misleading to readers learning the
"Additional Resources" pattern.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the editorial fix to `domainresource.html` in the FHIR Core
specification by changing the example's closing element from
`</Bundle>` to `</Basic>` so that the XML snippet in the "Additional
Resources" section is well-formed and consistent with the opening
`<Basic …>` element. Treat this as a technical correction against
R6 ballot 3 to be carried into the next publication of the page (no
StructureDefinition, schema, or normative content is touched). No
balloter notification is required because the content of the
specification is unchanged in substance — only the example XML is
corrected.

#### Justification

The reporter has identified an objectively wrong character sequence:
an opening `<Basic>` element cannot legally be closed by `</Bundle>`
in XML, and the surrounding prose in the "Additional Resources"
section discusses the `Basic` resource pattern, not bundling. Fixing
the example removes a copy-paste error that would otherwise mislead
implementers reading the page and would fail any cut-and-paste XML
validation. Technical Correction is precisely the issue type FHIR
Infrastructure uses for these editorial fixes, and the change is
risk-free with respect to backwards compatibility, conformance, and
tooling.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to fixing the closing tag, replace the inline hand-typed
XML example in the "Additional Resources" section with a generated
example sourced from the publication tooling's example pipeline (the
same mechanism that produces validated examples elsewhere on the
specification). This eliminates the class of error rather than just
the specific instance — any future edit of the example will go
through the validator and will not be able to reintroduce a
mismatched closing tag. While in the area, scan the rest of the
`DomainResource` page for any other hand-authored XML/JSON snippets
that bypass the example pipeline and either migrate them or add a
linter rule that flags `<X …>…</Y>` mismatches in narrative HTML.

#### Justification

A Basic-resource example is small enough that a generated, validated
form is feasible and the maintenance overhead is negligible. The
underlying root cause — hand-edited XML in narrative pages — has
produced this defect and is likely to produce more in similar
sections (e.g., "Additional Resources", "Custom Resources", and
extension how-tos). A linter / generated-example approach is more
defensible than continuing to fix one-off typos as they are
reported, and is consistent with the way most other example blocks
on the spec are produced. The trade-off versus Disposition A is
modest extra work in the publication tooling, weighed against
preventing the next instance of the same issue.

---

### Disposition C: Decline

#### Proposal

Decline on the basis that R6 ballot 3 is a frozen ballot snapshot
and editorial corrections to that specific publication will not be
made; the fix would only ever land in the post-ballot continuous
build. Close the ticket with a note pointing the reporter at the
build branch where the page is regenerated.

#### Justification

If the workgroup's policy is that ballot snapshots are immutable
artefacts of record (so balloters can rely on a stable URL during
the comment period), then patching the published ballot 3 page is
not appropriate. The defect can be acknowledged and tracked in the
build, with the reader-facing remediation being the next ballot or
the continuous build, rather than republishing ballot 3.

---

### Recommendation

**Recommended disposition:** A

This is a textbook Technical Correction: an objectively wrong XML
closing tag in a narrative example, with no impact on conformance,
tooling, or balloted normative content. The fix should be made in
the source for `domainresource.html` and rolled into the next
publication of the page; the reporter has done the work of locating
the exact anchor and stating the exact change, so the cost of
acceptance is minimal. Disposition B is a worthwhile follow-up but
should not block this fix — file it as a separate publication-tooling
improvement rather than coupling it to this ticket. Disposition C is
hard to defend for a ballot-period editorial defect of this kind
(mismatched XML tags in an illustrative snippet); the reporter's
expectation that a stated typo gets corrected in the next refresh of
the page is reasonable.
