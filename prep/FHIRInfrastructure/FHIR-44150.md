## Summary

TSMG has flagged that the "Backported R5 Subscription Content Code System"
defined inside the *FHIR R5 Subscriptions Backport* IG appears to fall under
the TSC-approved policy that externally-useful code systems should live at
[terminology.hl7.org](http://terminology.hl7.org/) (THO) rather than inside an
individual IG. The reporter (Jessica Bota, on behalf of TSMG) is asking the
FHIR Infrastructure work group, as owners of the Subscriptions Backport IG,
to review the [Terminology Guidance for IG Authors](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
and either (a) move/anchor the code system in THO, or (b) formally request an
exemption from TSMG (via listserv, meeting attendance, or the relevant Zulip
thread) on the grounds that the code system is tightly bound to the IG and
is not intended for reuse outside it. The ticket itself notes the reporter's
expectation that the second option is likely the correct one.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
recorded for FHIR-44150 in the Augury index, so this prep is based solely on
the ticket text and the cited TSMG policy.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a request to migrate the Backported R5 Subscription Content
Code System into THO. Concretely:

- File a THO change request to add a new CodeSystem at
  `http://terminology.hl7.org/CodeSystem/subscription-backport-content`
  (exact id to be agreed with TSMG) with the same set of concepts, metadata,
  and `caseSensitive` / hierarchy semantics currently published inside the
  Subscriptions Backport IG.
- In the Subscriptions Backport IG source, replace the locally-defined
  CodeSystem with a `NamingSystem`/aliased reference to the THO URL, update
  any ValueSets that include it to point at the THO canonical, and remove
  the in-IG CodeSystem resource so it is not republished.
- Add a redirect/alias note in the IG so existing instances using the old
  IG-scoped canonical continue to resolve, and update the IG's narrative,
  history page, and CI build to consume the THO canonical.
- Coordinate with TSMG on stewardship (which work group owns the THO
  artifact going forward — typically the originating WG, FHIR-I).

#### Justification

This matches the literal ask of the ticket and the TSC-approved policy in
[Terminology Expectations for IG Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers):
code systems that *could* be reused, that are referenced from generated
artifacts, or that have any plausible cross-IG audience belong in THO so
that consumers do not have to depend on an IG release cadence to get
terminology updates. Moving the code system also gives it a stable
canonical that survives any future re-spin or deprecation of the
Subscriptions Backport IG itself.

---

### Disposition B: Alternative Approach

#### Proposal

Formally request a TSMG exemption to keep the code system inside the
Subscriptions Backport IG. Concretely:

- Draft a short exemption request that: (1) lists the affected CodeSystem
  canonical(s); (2) explains that the codes are a 1:1 backport of R5
  `Subscription`-related content codes whose meaning is defined entirely
  by the backport mechanics (i.e., they have no use outside an
  R4/R4B implementation that is emulating R5 Subscriptions); (3)
  commits that the codes will not be referenced from non-backport
  artifacts; and (4) notes that the codes will be retired when the
  backport IG is itself retired.
- Submit the request via the TSMG listserv and/or Zulip
  [#tsmg](https://chat.fhir.org/#narrow/stream/179202-terminology) thread
  named in the ticket, and attend a TSMG call to walk it through.
- On approval, add a note to the IG's terminology section recording the
  granted exemption (date, link to TSMG decision) so future reviewers
  do not re-open the same issue.
- If TSMG declines, fall back to Disposition A.

#### Justification

The reporter explicitly anticipates this path ("seems like this may be
the case"). The Subscriptions Backport content code system exists only
to give R4/R4B implementations a way to reference R5 `Subscription`
status/notification semantics; it has no independent terminology life
and is not expected to be consumed by other IGs. Moving it to THO would
add ongoing maintenance overhead (THO release cycle, separate change
control, separate ownership conversation) for a small, stable,
IG-scoped vocabulary, which is exactly the situation the exemption
mechanism was designed for. This also avoids any churn for current
implementers who already depend on the IG-scoped canonical.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as out of scope for further FHIR-I action: take the
position that the existing code system is already appropriately scoped
to the IG, that no migration or formal exemption is required, and close
the ticket as Not Persuasive.

#### Justification

This is the weakest of the three options because the TSC-approved policy
referenced in the ticket is normative for HL7 IGs and explicitly requires
*either* an anchoring in THO *or* a granted exemption — silently doing
neither leaves the IG out of compliance with TSMG governance and exposes
it to re-flagging at every TSMG/QA review. The only defensible reading
of "decline" here would be to argue that the policy does not apply to
backport-style IGs, which is a position that needs to be litigated with
TSMG rather than decided unilaterally by FHIR-I.

---

### Recommendation

**Recommended disposition:** B

The Backported R5 Subscription Content Code System is, by construction, a
shim that exists only so R4/R4B implementations can talk about R5
`Subscription` semantics; it has no independent terminology audience and
its lifecycle is bound to the backport IG itself. That is precisely the
fact pattern TSMG's exemption mechanism is designed for, and the ticket's
own wording signals that TSMG expects an exemption request to be a
plausible outcome. Disposition B discharges the FHIR-I obligation under
the TSC policy with the least disruption to existing implementers, keeps
ownership and change control with the work group that already maintains
the codes, and produces a durable artifact (a recorded TSMG decision
linked from the IG) that prevents this issue from being re-raised.
Disposition A should remain the documented fallback if TSMG declines the
exemption; Disposition C is not recommended because it leaves the IG out
of compliance with a TSC-approved policy.
