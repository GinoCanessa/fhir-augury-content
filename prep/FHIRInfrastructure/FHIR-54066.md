## Summary

The ticket flags an inconsistency between two normative descriptions of `Signature.targetFormat` in the FHIR core build. On the **Signatures** page (`signatures.html`), the spec says `Signature.targetFormat` SHOULD be populated with a canonicalization URL drawn from the list of canonicalization methods defined on that page (with the JOSE custom header `canon` as an alternative). On the **DataTypes** page entry for `Signature.targetFormat`, the element is described as "A mime type that indicates the technical format of the target resources signed by the signature", with comments indicating values such as `application/fhir+xml`, `application/fhir+json`, or `application/fhir+turtle` are expected. The reporter (Hayden Bader) asks the workgroup to align the two descriptions, expressing a preference for the canonicalization-URL framing.

The wider design context is captured in [FHIR-51098](https://jira.hl7.org/browse/FHIR-51098) — opened by Grahame Grieve directly out of the Security and Privacy > Signature Zulip thread — which proposes adding a **new dedicated element** on `Signature` to carry the canonicalization algorithm rather than overloading `targetFormat`. The two tickets are complementary: FHIR-54066 is about reconciling the existing prose, while FHIR-51098 proposes the structural change that would let `targetFormat` revert to a pure MIME type. Any disposition here should be coordinated with FHIR-51098 so the spec ends up internally consistent.

### [Security and Privacy > Signature](https://chat.fhir.org/#narrow/stream/Security%20and%20Privacy/topic/Signature/near/523945922)

This is the originating thread for both tickets. Grahame Grieve posts a worked example of a signed `Bundle` (JOSE/JWS) and notes it would be nice to make the canonicalization method visible alongside the MIME type — initially proposing a MIME-type parameter form like `application/fhir+json; canonicalization=http://hl7.org/fhir/canonicalization/json`. After back-and-forth with Gino Canessa, the group converges on the view that a **dedicated optional element on `Signature` is more durable** than a MIME parameter (which is fragile across reserializations and harder to manage when multiple signatures are present). There was also an aside (John) that `Signature` is targeted for normative in R6 but is short on community engagement, and a related desire for a Digital Signatures IG that has not found authors.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **DataTypes page** entry for `Signature.targetFormat` so that its short description and comments are consistent with the **Signatures page** guidance: change the description from "A mime type that indicates the technical format of the target resources signed by the signature" to language that says the value SHOULD be a canonicalization URL drawn from the canonicalization methods defined in the Signatures page, and remove/replace the existing examples (`application/fhir+xml`, `application/fhir+json`, `application/fhir+turtle`) with canonicalization URL examples (e.g. `http://hl7.org/fhir/canonicalization/json#document`). At the same time, change the binding of `Signature.targetFormat` away from the generic `mimetypes` value set to a value set that contains the FHIR canonicalization URLs (or at minimum widen / re-bind so the canonicalization URLs validate). This is the "in favor of the canonicalization option" path the reporter explicitly prefers.

#### Justification

This option fully resolves the prose inconsistency the ticket calls out and aligns the DataTypes page with what is already being produced in the wild (see the Provenance Signature examples and the IG-Publisher-generated AU PS Bundle in the FHIR Validator thread). It also explains, rather than ignores, the validator failures Xun Ma reported: the binding to `mimetypes` is the root cause, and rebinding fixes them without requiring implementers to change their payloads. The downside is that it doubles down on overloading `targetFormat` to mean two different things (technical format **and** canonicalization scope), which contradicts John Moehrke's stated design preference and complicates the path to making `Signature` normative in R6.

---

### Disposition B: Alternative Approach — align to MIME type and defer canonicalization to a new element

#### Proposal

Resolve FHIR-54066 in coordination with [FHIR-51098](https://jira.hl7.org/browse/FHIR-51098):

1. On the **Signatures page**, rewrite the bullet so `Signature.targetFormat` is described as the **MIME type of the resource being signed** (e.g. `application/fhir+json`), matching the DataTypes page, and move the canonicalization-URL guidance onto a **new optional element on `Signature`** (cardinality 0..1 per the Zulip consensus, with multi-canonicalization handled by repeating the `Signature` datatype itself). The new element should carry the canonicalization URL from the existing list at `signatures.html` and SHOULD be populated when `sigFormat` is `application/jose`.
2. Keep the existing JWS `canon` protected-header guidance unchanged, since the canonical algorithm must be inside the signed payload regardless.
3. Leave the `mimetypes` binding on `Signature.targetFormat` intact so existing implementations validate again.
4. Add a migration note to the Signatures page acknowledging that examples using `targetFormat: application/fhir+json;canonicalization=...` were intermediate guidance and should move the canonicalization URL to the new element.

#### Justification

This is the disposition most consistent with the Zulip consensus (Grahame, Gino, John all converged on a new element being more durable than a MIME parameter) and with the road-to-R6 goal of getting `Signature` to normative without semantic overloading. It keeps `targetFormat` doing one job — naming the format of the signed bytes — which is what implementers in the older `implementers` channel discussions assumed. It also fixes the validator errors in the IG Publisher thread (FHIR Validator > Errors on Bundle.signature) because `targetFormat` would once again carry a value the `mimetypes` value set actually contains. The tradeoff is that this is a structural change to a TU datatype and requires FHIR-51098 to be accepted in the same cycle; if FHIR-51098 stalls, FHIR-54066 stalls too.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54066 as "Persuasive with Modification – defer", on the grounds that the underlying design question (where the canonicalization algorithm lives on `Signature`) is already tracked by FHIR-51098 and that any prose alignment on the DataTypes page should follow that decision rather than precede it. Take no edit on the DataTypes page until FHIR-51098 is dispositioned.

#### Justification

Editing the DataTypes prose now — in either direction — risks immediately re-introducing inconsistency once FHIR-51098 lands, since FHIR-51098's likely outcome (a new dedicated element) implies `targetFormat` reverts to "pure MIME type". Declining-and-deferring avoids two churn cycles on the same prose block. The downside is that it leaves the documented inconsistency in place for at least one more publication window, and does nothing for implementers currently hitting the validator errors documented in the FHIR Validator thread.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the only option that resolves both the prose inconsistency the reporter raised **and** the live validator failures implementers are reporting, without adding new complexity to `Signature.targetFormat`. The Zulip consensus (Grahame, Gino, John) on the originating thread already lands on a new dedicated element for the canonicalization URL; Hayden's prose-alignment request is the logical follow-on to make the DataTypes and Signatures pages tell the same story once that element exists. It also keeps `Signature.targetFormat` clean enough to carry into a future R6 normative push, which is an explicit FHIR Infrastructure goal noted on the Zulip thread. The workgroup will need to coordinate FHIR-54066 and FHIR-51098 as a pair; if the workgroup is not ready to take the structural change in this cycle, fall back to Disposition A (accept as requested) rather than C, because the validator pain reported in the FHIR Validator thread is real and recurrent and warrants action this cycle even if it is later revisited.
