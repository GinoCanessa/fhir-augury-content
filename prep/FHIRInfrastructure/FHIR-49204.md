## Summary

The ticket proposes extending the SDC StructureMap-less extraction model so
that a single Questionnaire item profiled with `extractDefinition` /
`extractValue` can populate **multiple** target resource instances of the
same profile in parallel. The reporter (Lloyd McKenzie) suggests this be
done by adding an optional `sequence` or `alias` discriminator to the
existing `extractDefinition` / `extractValue` extensions, so authors who
do not need parallel extraction see no added complexity, while authors
who do can group extracted values into distinct target instances. The
ticket was originally voted "Considered for Future Use" (Postlethwaite /
Rhodes 4-0-0, 2025-07-31) on the grounds that the Work Group was
"amenable to introducing some sort of 'alias' ability, but we need to
experiment with it, and that's not going to happen before the release"
(STU4). It was reverted and re-opened by Lynn Laakso post-STU4
publication (2026-03-26) with the intent of revisiting it for the next
SDC ballot cycle. There are no Augury-tracked cross-referenced Jira
tickets, Zulip threads, or GitHub items linked to this issue.

### Disposition A: Accept As Requested

#### Proposal

Add an optional element — name it `instance` (preferred) or `alias` —
to both the `sdc-questionnaire-extractDefinition` and
`sdc-questionnaire-extractValue` extensions in the SDC IG. The new
sub-extension carries a single string token (or small integer) that
acts as a discriminator: all `extractDefinition` / `extractValue`
occurrences that share the same `(profile, instance)` tuple within a
given Questionnaire scope contribute to the **same** target resource
instance; distinct `instance` values produce **separate** target
instances of the same profile, populated in parallel from the same
QuestionnaireResponse. When `instance` is absent, behavior is identical
to today's (single instance per profile in scope), preserving full
backward compatibility for existing Questionnaires and SDC-conformant
extractors.

Update the SDC IG narrative pages that describe definition-based
extraction to:

- Describe the new sub-extension and its discriminator semantics.
- Show one worked example (e.g., a single questionnaire item that
  produces multiple `Observation` resources of the same profile, one
  per `instance` group).
- Specify extractor behavior when the same `instance` token appears on
  `extractValue` occurrences whose ancestor `extractDefinition`
  occurrences do **not** share the same `instance` token (treat as
  conformance error).

#### Justification

This is the minimal, additive change that delivers exactly what the
reporter asked for: it removes the current 1-Questionnaire-item-to-1-
instance-per-profile ceiling without requiring authors who don't need
parallel extraction to learn anything new. It matches the workgroup's
prior signal that they are "amenable to introducing some sort of
'alias' ability" and the only blocker recorded was timing relative to
STU4 publication. Putting the discriminator on both extensions
(definition and value) keeps grouping semantics local to the extraction
extensions themselves rather than introducing a new top-level grouping
extension, which keeps the surface area small and consistent with the
existing SDC pattern of layering small extensions on Questionnaire
items.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding a discriminator field to the existing extensions,
introduce a single new wrapper extension — e.g.,
`sdc-questionnaire-extractGroup` — that nests one or more
`extractDefinition` / `extractValue` occurrences under a named group.
Each `extractGroup` occurrence under a Questionnaire item produces one
target resource instance of the profile referenced by its child
`extractDefinition`; repeating the wrapper N times under the same item
yields N parallel target instances. Existing Questionnaires that don't
use the wrapper continue to behave exactly as today.

#### Justification

A wrapper extension makes the grouping boundary syntactically explicit:
reviewers see "this block of extractDefinition + extractValue belongs
to one instance" without having to mentally join occurrences by a
shared discriminator token. It also avoids any ambiguity about what
happens when authors type the discriminator inconsistently across
extensions, and it leaves the existing extensions untouched (no risk
of confusing tooling that already understands them). The trade-offs
vs. Disposition A are:

- **Cost:** a brand-new extension to define, register, and document,
  versus a one-line addition to two existing extensions.
- **Author ergonomics:** more nesting in the Questionnaire JSON / XML
  for the common single-instance case, even though authors can keep
  using the flat form when they don't need grouping.
- **Migration:** existing tooling needs to learn the new wrapper to
  emit multiple instances, but does not need to re-validate any
  existing Questionnaires.

This is worth weighing because the reporter's proposal solves the
problem with the least added syntax, while a wrapper solves it with
the least ambiguity. The workgroup will need to decide which axis
matters more for SDC implementers.

---

### Disposition C: Decline

#### Proposal

Decline the request and direct authors who need parallel extraction of
multiple instances of the same profile to use a full StructureMap (the
`sdc-questionnaire-targetStructureMap` extension) instead of the
definition-based extraction extensions. Document this guidance
explicitly on the SDC IG extraction page so the limitation of the
definition-based path is no longer a surprise.

#### Justification

The SDC IG already provides a fully-general extraction mechanism —
StructureMap-driven extraction — that handles arbitrarily complex
mappings, including parallel population of multiple target instances
from a single QuestionnaireResponse, with no spec change required.
The definition-based extensions were intentionally introduced as a
**simpler** path for the common case, and adding instance-grouping
semantics arguably re-introduces the very complexity the simpler path
was meant to avoid. There is no use-case evidence in the ticket itself
or in Augury-indexed Zulip / GitHub discussions that the demand is
broad enough to justify expanding the simpler path's surface area, and
declining preserves the clean two-tier model (simple → definition-based;
complex → StructureMap).

---

### Recommendation

**Recommended disposition:** A

The Work Group has already signaled, on the record (Postlethwaite /
Rhodes 4-0-0, 2025-07-31), that it is "amenable to introducing some
sort of 'alias' ability" and that the only reason it deferred was the
STU4 publication window. Re-opening the ticket post-STU4 explicitly
revives that intent for the next ballot cycle, so a "Decline"
disposition would contradict the Work Group's previously expressed
position. Between the two "accept" shapes, Disposition A (a
discriminator on the existing extensions) is the smaller, more
backward-compatible change: it costs one optional sub-extension on
each of two already-defined extensions, leaves every existing
Questionnaire and every existing extractor implementation strictly
conformant, and matches the reporter's stated design goal that the
mechanism "wouldn't add complexity to those that don't need it." A
wrapper extension (Disposition B) is defensible on clarity grounds
and is worth raising at the meeting, but it introduces a new
extension and more nesting for what is, in the end, a fairly narrow
extension to existing semantics. The recommended path is therefore
Disposition A, with Disposition B held in reserve if reviewers raise
concrete ambiguity concerns about discriminator-based grouping during
the discussion.
