## Summary

The reporter (Gino Canessa, FHIR Infrastructure) requests adding a coded
classifier element to the `RelatedArtifact` data type so that downstream
implementation guides can convey additional type/code information about
each related artifact entry without resorting to extensions. Concretely,
the ask is to add an element — name suggested as `code` or `class`,
typed as `Coding` or `CodeableConcept`, cardinality `0..1` or `0..*`,
short label "Kind of attachment" — that supplements the existing
`RelatedArtifact.type` enumeration with a more flexible, IG-extendable
classification axis.

No formal Jira or GitHub cross-references were returned by FhirAugury
for this ticket. However, the request lands squarely in a long-running
FHIR-I/CDS/CQI design conversation about whether `RelatedArtifact`
should be enriched in place or replaced with explicit per-resource
canonical relationships (deprecate-and-replace approach pursued under
[FHIR-50010](https://jira.hl7.org/browse/FHIR-50010), with related
trackers [FHIR-40920](https://jira.hl7.org/browse/FHIR-40920),
[FHIR-49782](https://jira.hl7.org/browse/FHIR-49782),
[FHIR-49783](https://jira.hl7.org/browse/FHIR-49783),
[FHIR-50011](https://jira.hl7.org/browse/FHIR-50011), and the
CitationInformation work in
[FHIR-49758](https://jira.hl7.org/browse/FHIR-49758)). Adding a coded
classifier directly to `RelatedArtifact` would be a strictly additive,
backward-compatible change that addresses the IG pain point without
restructuring the type.

## Related Zulip Discussions

### [implementers > RelatedArtifact Replacement Discussions](https://chat.fhir.org/#narrow/stream/implementers/topic/RelatedArtifact%20Replacement%20Discussions)

Bryn Rhodes opened the thread proposing to deprecate `RelatedArtifact`
on the knowledge-artifact resources (Measure, Library,
ActivityDefinition, PlanDefinition, Citation, ArtifactAssessment) and
replace it with explicit per-resource canonical elements (`dependency`,
`component`, `derivedFrom`, `replaces`, `citation`, `citeAs`,
`documentation`, `justification`). Lloyd McKenzie strongly supported
the replacement, arguing the data type can't be tightened per-resource
(its codes, types, and reference targets are fixed at the data-type
level), which is "an anti-pattern in FHIR" for a generic typed
relationship. Grahame Grieve pushed back, preferring to fix the design
of the data type in place rather than deprecate it ("adding 10 new
fields to these already fat resources? Not a fan"). Brian Alper
suggested a middle path: simplify to a single `target[x] 1..1` plus a
coded `type`, optionally backed by resource-specific `[Resource]
RelatedArtifactType` code systems, possibly with a `classifier 0..*
CodeableConcept`. Brian Kaney (implementer) called himself
"ambivalent" — filtering by `relatedArtifact.type` already works for
him; the bigger pain is resolving canonical references when the target
type is unknown. Stan Rankins raised an FMM concern (Measure/Library
are FMM 4 with production code behind them). Bryn Rhodes ultimately
filed [FHIR-50010](https://jira.hl7.org/browse/FHIR-50010) for the
simplification and updated dispositions on the related WG-specific
trackers (FHIR-40920, FHIR-49782, FHIR-49783, FHIR-50011). The thread
did not reach consensus on whether to enrich the type vs. replace it,
which is the open methodology question that FHIR-56300's "just add a
code" framing intersects with.

Participants: Bryn Rhodes, Lloyd McKenzie, Grahame Grieve, Brian Alper,
Brian Kaney, Stan Rankins, René Spronk.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new element `RelatedArtifact.classifier 0..* CodeableConcept`
(short: "Additional classification of the related artifact") to the
`RelatedArtifact` data type in FHIR Core R6. Bind it to an example-
strength value set seeded empty (e.g.,
`http://hl7.org/fhir/ValueSet/related-artifact-classifier`), so that
downstream IGs can supply their own coded values without defining
extensions. Keep the existing `RelatedArtifact.type` element unchanged
(it remains the required, enumerated relationship code) — `classifier`
is purely an additive, IG-facing axis. Use `CodeableConcept` (not
`Coding`) for parity with the other "classifier"-style elements in
core (e.g., `Citation.classifier`, `ArtifactAssessment.classifier`)
and to allow text fallback. Use `0..*` (not `0..1`) so an IG can layer
multiple classification axes (e.g., audience and content-type).

#### Justification

The change is strictly additive and backward-compatible (no existing
instance becomes invalid). It uses an established FHIR pattern —
`classifier 0..* CodeableConcept` already exists on `Citation`,
`ArtifactAssessment`, `EvidenceReport`, and several other knowledge
artifacts, so reviewers and implementers will recognize the shape and
cardinality. Because the binding is example-strength with an empty
seed value set, IGs (US Core, IPS, Da Vinci, EBM resources, etc.) can
classify related artifacts without burning extension URLs, which is
exactly the pain the reporter describes. It also sidesteps the
deeper, contested replacement debate captured in the Zulip thread:
nothing in this change forecloses the
[FHIR-50010](https://jira.hl7.org/browse/FHIR-50010) replacement
direction, and nothing in this change depends on it.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change `RelatedArtifact` itself. Instead, define a single,
standard FHIR Core extension
`http://hl7.org/fhir/StructureDefinition/relatedArtifact-classifier`
(`CodeableConcept`, `0..*`) that IGs can adopt to classify a
`RelatedArtifact` entry. Document the extension in the
`RelatedArtifact` data-type page as the recommended way to add an
IG-specific classification axis, and ship it via the FHIR Extensions
Pack so it does not require an R6 cycle.

#### Justification

This addresses the reporter's goal — "without requiring extensions [in
each downstream IG]" — by providing a single, well-known, HL7-owned
extension that all IGs can reuse, while leaving the core data type
untouched. It avoids growing a data type that the methodology
discussion in
[#implementers > RelatedArtifact Replacement Discussions](https://chat.fhir.org/#narrow/stream/implementers/topic/RelatedArtifact%20Replacement%20Discussions)
already characterizes as overloaded, and aligns with Grahame Grieve's
"fix it without making it bigger" preference and Lloyd McKenzie's
view that classification breadth is exactly the kind of variation the
80% rule pushes into extension space. It also has lower risk and
lower coordination cost than a core change: no R6 normative-impact
review for any resource that uses `RelatedArtifact` (Measure, Library,
ActivityDefinition, PlanDefinition, Citation, ArtifactAssessment,
ValueSet, CodeSystem, ConceptMap, NamingSystem, …), and the FMM-4+
artifacts Stan Rankins called out are not perturbed.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-56300 as written. Rationale: the underlying need (richer
per-IG classification of related artifacts) is already addressable
through an IG-defined extension on `RelatedArtifact`, and the broader
methodology question about whether to enrich vs. replace
`RelatedArtifact` is being tracked under
[FHIR-50010](https://jira.hl7.org/browse/FHIR-50010) and the
WG-specific trackers (FHIR-40920, FHIR-49782, FHIR-49783, FHIR-50011).
Adding a generic `code`/`class`/`classifier` to a data type whose
future is actively under debate risks committing core surface area
that may need to be reworked or deprecated alongside the
replacement-pattern outcome. Recommend the reporter raise the
specific use case on
[FHIR-50010](https://jira.hl7.org/browse/FHIR-50010) so the broader
proposal can absorb it.

#### Justification

The Zulip thread shows active disagreement among methodology leads
(Grahame Grieve, Lloyd McKenzie, Bryn Rhodes, Brian Alper) about
whether `RelatedArtifact` should be modified, replaced, or
augmented. Adding a coded classifier now bakes the "augment in place"
direction into core before that question is resolved, which is exactly
the kind of premature commitment the FHIR methodology cautions
against for a data type that is consumed by multiple FMM-4+ resources
(Measure, Library, ActivityDefinition, PlanDefinition, plus the EBM
and terminology resources in R5+). IG-level extensions are a
well-understood, low-risk fallback for the duration of the larger
debate.

---

### Recommendation

**Recommended disposition:** A

The reporter's underlying need is real and recurring: implementation
guides repeatedly want to attach a classification axis to
`RelatedArtifact` entries (audience, content-type, jurisdiction,
intended use, etc.) and the existing enumerated `type` element does
not stretch to cover it. Disposition A solves that directly, uses an
established and recognizable FHIR pattern (`classifier 0..*
CodeableConcept` already exists on neighboring knowledge-artifact
resources), is purely additive (no breaking change for any of the ~10
resources that carry `RelatedArtifact`), and is independent of the
larger
[FHIR-50010](https://jira.hl7.org/browse/FHIR-50010) replacement
debate — it neither blocks nor forecloses that direction. Disposition
B (single FHIR Core extension) is a reasonable fallback if FHIR-I
wants to defer touching the data type until the replacement debate
settles, but it leaves IG authors paying the small-but-real ergonomic
tax the reporter is trying to remove. Disposition C is the
methodologically purest answer, but in practice the
classification-axis need is broader than the replacement debate and
will outlive whatever R6 decides about deprecating `RelatedArtifact`,
so declining without giving IGs a path forward is unsatisfying. FHIR-I
should adopt A, and explicitly note in the resolution that
`classifier` is orthogonal to and does not prejudge the FHIR-50010
direction.
