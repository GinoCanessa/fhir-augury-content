## Summary

James Jahns reports that `Subscription.filterBy.event` is unclear in two
distinct ways. First, `Subscription.filterBy` requires both
`filterParameter` and `value`, so an implementer who wants to filter by
*just* an event (without also constraining a search-style filter
parameter) has no obvious way to model that — the cardinality forces
populating fields that have no meaningful value for an event-only
filter. Second, the long description for `filterBy.event` says that
"multiple values are or-joined, multiple codings within a value are
and-joined." The reporter notes that this contradicts standard
`CodeableConcept` semantics: the `coding` array is intended to carry
*translations of the same concept*, so AND-joining codings within a
single `CodeableConcept` is semantically odd. Without that explanatory
sentence, the reporter would have expected implementers to treat the
codings as alternative representations of one event and match if any
recognized coding matches.

There are no cross-referenced Jira tickets, no Zulip threads tagged to
this issue, and no PRs/issues linked to it. The only cross-reference is
the FHIR spec artifact `Subscription.filterBy.event` itself, plus the
authoring file `source/subscriptiontopic/subscriptiontopic-notes.xml`
in `HL7/fhir` which mentions the artifact in narrative.

## Details

**Description:**

Two parts of `Subscription.filterBy.event` are confusing to me:

1. If I want to filter by *just* an event (not also a parameter), it's
   unclear how to model this, because `filterBy.filterParameter` and
   `filterBy.value` are both required.
2. The long description for this element says:

   > An event filter to be applied to the topic - e.g., if a topic
   > defined multiple event triggers, this can be used to specify a
   > single one. Multiple values are or-joined, **multiple codings
   > within a value are and-joined.**

   This is confusing to me; `CodeableConcept.coding` should provide
   translations of a given singular concept, but "and-joining" implies
   that multiple different events would be supplied.

   (without this description, I'd have expected implementers to just
   find all recognized codings within the `CodeableConcept`, and match
   upon at least one of those)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Make two coordinated edits in `HL7/fhir` under `source/subscription/`:

1. **Cardinality**: in the Subscription StructureDefinition `<differential>`,
   change `Subscription.filterBy.filterParameter` from `1..1` to `0..1`
   and `Subscription.filterBy.value` from `1..1` to `0..1`, and add an
   invariant on `Subscription.filterBy` of the form
   `event.exists() or (filterParameter.exists() and value.exists())`
   so that a `filterBy` entry must populate *either* an event filter
   *or* a parameter+value pair (or both). This lets implementers
   express "filter by just an event" without inventing placeholder
   `filterParameter` / `value` strings.
2. **Narrative**: rewrite the long-form description of
   `Subscription.filterBy.event` to drop the "multiple codings within a
   value are and-joined" sentence and replace it with text aligned to
   normal `CodeableConcept` semantics — i.e., codings within a single
   `event` value are alternative representations of the same event, and
   multiple `filterBy.event` entries are OR-joined across the
   `filterBy` array. Mirror the wording in
   `source/subscriptiontopic/subscriptiontopic-notes.xml` so
   `SubscriptionTopic` and `Subscription` agree.

#### Justification

This addresses both ambiguities the reporter raises with concrete spec
text. Treating multiple codings within one `CodeableConcept` as
alternative representations is consistent with how
`CodeableConcept.coding` is defined everywhere else in FHIR; the
current AND-join wording is the actual outlier. Making
`filterParameter` / `value` optional, gated by an invariant, removes
the modeling dead-end without introducing a new element. The change is
contained in `source/subscription/` and `source/subscriptiontopic/`,
both standard FhirCore authoring roots, and uses the playbook's
*Update narrative* and *Edit `<differential>`* recipes. The briefing
gotcha about hand-editing only `<differential>` is honoured; snapshot
regeneration is left to the publisher.

---

### Disposition B: Alternative Approach — Narrative-Only Clarification

#### Proposal

Leave the cardinality of `Subscription.filterBy.filterParameter` and
`Subscription.filterBy.value` unchanged, but rewrite the long-form
descriptions of `Subscription.filterBy` and
`Subscription.filterBy.event` to (a) clarify that, when filtering
purely on an event, implementers should set `filterParameter` to the
canonical token `"event"` (or whatever search parameter name the
SubscriptionTopic exposes for events) with `value` set to the event
code as a string, with the `event` `CodeableConcept` carrying the
structured form; and (b) drop the AND-join sentence and replace it
with normal `CodeableConcept`-coding semantics ("codings within a
single value are alternative representations of the same event;
multiple `filterBy.event` entries OR-join"). Add a worked example to
`source/subscription/subscription-notes.xml` (or the equivalent
narrative page) showing an event-only filter.

#### Justification

This avoids any normative cardinality change to Subscription —
attractive given the resource's recent normative motion in R6 — while
still resolving the two specific points raised. The reporter's main
asks are clarity, not new structure: the issue title is "Usage of
Subscription.filterBy.event is *unclear*". A narrative fix is squarely
in the playbook's *Update narrative / prose* recipe and avoids the
gotcha around invariants/snapshots that Disposition A introduces.
Trade-off: it leaves implementers populating a token that has no
intrinsic meaning when only an event filter is intended, which is
arguably the deeper smell.

---

### Disposition C: Decline

#### Proposal

Close as Not Persuasive on the grounds that
`Subscription.filterBy.event` is intentionally paired with
`SubscriptionTopic.canFilterBy.event` and that the AND/OR-join
semantics for codings are deliberate (not an accident of
`CodeableConcept` reuse), and that the `filterParameter` / `value`
cardinality is part of the resource's normative shape that should not
be changed mid-cycle on a single Triaged request. Acknowledge the
ambiguity but defer to a future ticket that proposes specific normative
text rather than mixed prose+cardinality change.

#### Justification

Defensible only if the workgroup believes the AND-joining-within-a-CodeableConcept
behaviour is a designed feature (perhaps to allow code+system
disambiguation in environments where multiple codes have overlapping
display strings) and that the existing cardinality reflects deliberate
design. The brief duration since R6 publication and the absence of any
Zulip discussion about this exact question may make a "wait for more
implementer feedback" stance reasonable. The cost is leaving a known
spec-clarity bug in place; the reporter has explicitly stated that the
description contradicts their reading of `CodeableConcept`, which is a
strong signal that other implementers will hit the same confusion.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified two genuinely problematic elements of the
spec that compound: a cardinality that forces nonsense values when
only an event filter is intended, and a long-description sentence that
contradicts how `CodeableConcept.coding` is defined throughout the
rest of FHIR. Disposition B fixes only the narrative half and leaves
implementers populating placeholder strings; Disposition C effectively
documents that future implementers will hit the same confusion. The
edits in Disposition A are confined to `source/subscription/` and
`source/subscriptiontopic/` in `HL7/fhir`, follow the *Update
narrative* and *Edit `<differential>`* recipes from the FhirCore
playbook, and the new invariant
(`event.exists() or (filterParameter.exists() and value.exists())`)
is the FHIR-idiomatic way to express the "either/or" relationship
without losing any expressive power. The workgroup should still
confirm that no existing FHIRPath invariant on `Subscription.filterBy`
already assumes both `filterParameter` and `value` are populated; if
one does, it must be updated in the same change.
