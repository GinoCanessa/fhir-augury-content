## Summary

FHIR-40485 (reported by Michele Mottini against the SMART on FHIR
specification) asks the SMART working group to add an **optional `aud`
(audience) parameter to the token-introspection request** so that the
authorization server can disambiguate which FHIR endpoint a caller is
introspecting a token for.

The reporter highlights the case of "facade" servers that front several
FHIR endpoints. In that topology the same logical patient can have
**different FHIR `Patient.id` values per endpoint** (because each facade
performs its own internal-to-FHIR id mapping). Introspection today
returns a single patient identifier, but without knowing the audience
the authorization server cannot decide *which* endpoint-specific
mapping to apply. Letting clients supply an `aud` on the introspection
call would give the authorization server enough context to return the
correct, endpoint-scoped patient id (and, by extension, any other
endpoint-relative context values).

The closest companion ticket is
[FHIR-36179](https://jira.hl7.org/browse/FHIR-36179) — "Add an optional
`aud` parameter to the `/token` response" — which was resolved
**Persuasive with Modification** and addressed the *outbound* multi-audience
problem on the `/token` response. FHIR-40485 is the symmetrical request
for the *introspection* endpoint and was not picked up in the SMART 2.1
triage cycle (Josh Mandel acknowledged in the Zulip thread below that
the WG agreed the information was worth putting in introspection
responses but did not triage it in time).
[FHIR-43623](https://jira.hl7.org/browse/FHIR-43623) (Persuasive,
Published) overlaps in subject area — it clarified how access-control
on the introspection endpoint should work — but does not itself add an
`aud` input parameter.

## Details

**Description:**

> Token introspection returns the patient id for patient-scoped tokens,
> but in systems that have multiple FHIR endpoints the correct value of
> the FHIR patient id can be different depending on the endpoint (in
> 'facade' servers where the internal ids are converted in some way -
> that can be endpoint specific - to the FHIR ids).
>
> Adding an optional audience - specifying which FHIR endpoint the
> introspection is for - would enable the system to do the correct
> conversion.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the SMART App Launch token-introspection section to define an
**optional `aud` request parameter** on the `/introspect` endpoint
whose value is the FHIR base URL that the introspecting resource
server represents.

- Conformance: clients of `/introspect` (resource servers) **MAY**
  include `aud`; authorization servers **SHOULD** honor it when
  computing context values that are endpoint-relative (notably
  `patient`, and any other id whose value depends on the resource
  server's id-mapping rules).
- When `aud` is omitted, the authorization server's behavior is
  unchanged from current SMART (i.e., it returns whichever single
  value it has historically returned, typically scoped to the audience
  used at `/authorize`).
- When `aud` is supplied but does not match an audience the token is
  valid for, the authorization server **SHALL** treat the token as not
  active for that audience (`active: false`), mirroring RFC 7662
  semantics.
- Add an example illustrating a facade server with two FHIR endpoints
  returning different `patient` values for the same access token under
  different `aud` values.

#### Justification

This is the symmetrical mirror of FHIR-36179 ("aud on `/token`
response"), which the WG already accepted *Persuasive with
Modification*. Accepting the input-side counterpart is consistent with
the design pattern the WG has already endorsed and with Josh Mandel's
on-record agreement in the Zulip thread that "this is information
worth putting in introspection responses." It directly supports the
FHIR-43623 model in which independent resource servers introspect with
their own credentials, and it avoids inventing a new mechanism — `aud`
is already a well-understood OAuth 2.0 / OIDC concept.

---

### Disposition B: Alternative Approach — derive audience from the introspecting client's identity

#### Proposal

Instead of adding an `aud` request parameter, **clarify** in the
introspection section that the authorization server **SHOULD** scope
context values (e.g., `patient`) to the **registered audience(s) of
the authenticated client** that is calling `/introspect`. Resource
servers would already be authenticating to the introspection endpoint
(per FHIR-43623), so the auth server can look up which FHIR endpoint
that client represents and apply the correct id mapping without an
explicit parameter. If a client legitimately fronts more than one
audience, recommend that it register a separate introspection client
per audience (or document a `client_id`-to-audience map) rather than
parameterise each call.

#### Justification

This approach reuses the trust establishment that FHIR-43623 already
requires (the introspection endpoint is protected, callers
authenticate), avoids changing the wire shape of introspection
requests, and prevents the failure mode where a misbehaving client
supplies an arbitrary `aud` to coerce the auth server into producing a
specific id mapping. The downside is that it forces facade
deployments to provision one introspection client per audience, which
is operationally heavier than letting a single client supply `aud`,
and it pushes more configuration burden onto registration. It is also
slightly less aligned with FHIR-36179, which chose to surface
audiences explicitly rather than infer them from client identity.

---

### Disposition C: Decline

#### Proposal

Do not add an `aud` parameter to the introspection request. Document
that introspection responses are scoped to the audience under which the
token was originally issued (the `aud` from `/authorize`), and that
deployments needing endpoint-specific id values should either (a) use
distinct authorization servers per FHIR endpoint, or (b) ensure that
ids surfaced in launch context are valid across all audiences the
token covers.

#### Justification

This keeps the introspection contract simple and aligned with the
narrowest reading of RFC 7662, where introspection answers "is this
token active and what does it bear" rather than "what does it mean for
this *specific* resource server." The defensible argument is that
SMART-issued tokens are intended to bind to an audience at
authorization time (FHIR-36179 already provides the mechanism to
disclose multiple audiences then), so introspection per-audience
re-resolution is a deployment-architecture concern rather than a spec
gap. The cost of declining is real, however: the facade-server
topology described by Michele Mottini is documented as deployed in
production and is not reasonably solved by "use one auth server per
endpoint."

---

### Recommendation

**Recommended disposition: A — Accept As Requested.**

The combination of (1) FHIR-36179 already being accepted as
*Persuasive with Modification* for the *outbound* aud problem, (2)
FHIR-43623 establishing the multi-resource-server introspection model,
and (3) Josh Mandel's on-record acknowledgement in the Zulip thread
that this information belongs in introspection responses, makes
Disposition A the natural and least-surprising path. It is a small,
optional, backwards-compatible addition that uses an OAuth-native
parameter name with well-understood semantics, and it solves a
concrete production problem reported by an implementer (CareEvolution
facade servers). Disposition B is defensible but pushes operational
complexity onto registration and is inconsistent with the precedent
set by FHIR-36179. Disposition C should only be chosen if the WG is
prepared to formally state that facade-style deployments are out of
scope for SMART introspection, which would contradict the direction
of FHIR-43623. The drafting work would land in
`HL7/smart-app-launch` (note: no local repo-analysis briefing is
cached, so the exact source-file path should be confirmed at
implementation time).
