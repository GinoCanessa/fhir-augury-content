## Summary

The ticket asks the FHIR Infrastructure work group (SMART on FHIR
specification) to clarify what value the `scope` field is required to
contain in the JSON response when a refresh token is redeemed for a new
access token. The current SMART App Launch text (App Launch →
"Refresh Access Token") describes `scope` as required and says the
returned scope "will be the same as the scope of the original access
token, and it can be different from the scopes requested by the app."
Cooper Thompson reports that this language is ambiguous in practice:
some scopes from the original token (notably `openid` and `fhirUser`,
and arguably `launch`/`launch/*` and `offline_access`) do not naturally
"re-apply" to a refresh response, because the refresh exchange does
not re-authenticate the user and does not necessarily re-issue an
`id_token`. The issue surfaced in ONC (g)(10) certification through
Inferno, where the test kit was interpreting the spec strictly and
expecting an exact match of original scopes
(onc-healthit/onc-certification-g10-test-kit#39). No related Jira
tickets, no Jira issue links, and no GitHub cross-references are
recorded in the Augury index for this ticket; the only related
discussion captured is the `#smart` Zulip thread cited in the ticket.

## Details

**Description:**

http://www.hl7.org/fhir/smart-app-launch/app-launch.html#refresh-access-token

The response JSON object for redeeming a refresh token for a new access token has this language:

| | | |
|-|-|-|
| `scope` | required | Scope of access authorized. Note that this will be the same as the scope of the original access token, and it can be different from the scopes requested by the app. |

It is unclear what the scope content should be. The language implies that it should be identical to the initial token, but some of the scopes don't always make sense to re-send (`fhirUser` and `openid`).

## Related Zulip Discussions

### [#smart > Expected scopes on a "refreshed" access token](https://chat.fhir.org/#narrow/stream/179170-smart/topic/Expected.20scopes.20on.20a.20.22refreshed.22.20access.20token)

Cooper Thompson opened the thread quoting the App Launch refresh-token
response text and flagging that `openid` and `fhirUser` do not always
make sense to re-issue. Robert Scanlon (Inferno) framed the question
as whether the spec — or OAuth 2.0 — gives the client any guarantee
about the relationship between the original access token's scope and
the refreshed access token's scope, noting Inferno had been
interpreting the SMART text strictly. Josh Mandel responded that the
intent was that **FHIR data-access scopes** (`patient/*`, `user/*`,
`system/*`) from the original token should be present on the refreshed
token, but that the **authentication context** (`openid`, `fhirUser`,
and the associated `id_token`) need not be re-issued — citing the
OpenID Connect Core spec's refresh-token response section, which
allows the `id_token` to be omitted on refresh. He also acknowledged
that SMART has no conformance language nailing this down today, which
is exactly the gap the ticket asks the WG to close. No firm consensus
on normative wording was reached in the thread; the conversation
lands on a clear distinction between data-access scopes (must
persist) and authentication-context scopes (may be omitted).

Participants: Cooper Thompson, Robert Scanlon, Josh Mandel.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the SMART App Launch specification — specifically the
**App Launch → Refresh Access Token** section's response-object table
— to replace the current ambiguous `scope` description with explicit,
normative guidance that distinguishes data-access scopes from
authentication-context scopes. Concretely:

- `scope` (required): MUST include every **FHIR data-access scope**
  (`patient/*.*`, `user/*.*`, `system/*.*`) that was present in the
  original access token's `scope` and that the authorization server
  still considers granted; it MAY be a narrower subset if the server
  has revoked or down-scoped any data-access permission, but MUST NOT
  include data-access scopes that were not in the original grant.
- `openid` and `fhirUser` SHOULD NOT be re-asserted on a refresh
  response unless the server is also returning a fresh `id_token` in
  the same response (per OpenID Connect Core §12, which permits the
  `id_token` to be omitted on refresh).
- `offline_access` MAY be re-asserted; servers that rotate refresh
  tokens SHOULD re-assert it when a new refresh token is issued.
- `launch` and `launch/{context}` scopes from the original
  authorization SHOULD NOT be re-asserted on refresh, since launch
  context is established once at authorization time and is conveyed
  on the access-token response via the launch-context parameters
  (e.g., `patient`, `encounter`), not via re-issuing the launch
  scope.
- Add a short non-normative note that clients MUST NOT assume the
  refreshed `scope` is byte-identical to the original and SHOULD
  re-parse `scope` after every refresh.

This change is editorial-plus-conformance in the App Launch IG; it
does not touch capability statements, OperationDefinitions, or any
core FHIR resource.

#### Justification

This is exactly the clarification the ticket and the Zulip thread
ask for, and it matches the position Josh Mandel (a SMART editor)
already articulated on chat: data-access scopes persist,
authentication-context scopes do not. Aligning the spec with that
already-stated intent removes the implementer ambiguity that
triggered the Inferno (g)(10) issue and prevents servers from being
forced to re-issue scopes that have no operational meaning on a
refresh exchange. The change is consistent with OAuth 2.0 (RFC 6749
§6, which lets refresh responses narrow but not broaden scope) and
with OpenID Connect Core's refresh-response section (which makes
`id_token` optional, implying `openid`/`fhirUser` need not be
re-asserted). Because no implementer can reasonably claim the current
text is unambiguous, tightening it is unlikely to break any
conformant deployment.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than amending the response-object table inline, add a new
dedicated subsection to App Launch — e.g.,
**"Scope semantics across token refresh"** — directly under
"Refresh Access Token", and reduce the table cell for `scope` to a
one-line cross-reference into that subsection. The new subsection
would:

1. Define two scope categories normatively: **data-access scopes**
   (`patient/`, `user/`, `system/` plus `offline_access`) and
   **authentication/launch-context scopes**
   (`openid`, `fhirUser`, `launch`, `launch/*`).
2. State the persistence rules from Disposition A as a small
   conformance table (MUST persist / SHOULD NOT re-assert / MAY
   narrow).
3. Add explicit guidance for **down-scoping** on refresh (server
   MAY return a narrower `scope`; client MUST handle that) and for
   **scope absence** (if the server omits a previously-granted
   data-access scope, the client MUST treat it as revoked for the
   life of the new access token).
4. Cross-reference RFC 6749 §6 and OIDC Core §12 so test kits like
   Inferno can cite normative text rather than chat consensus.

This keeps the response-object table compact and gives Inferno /
other test harnesses a single anchor URL to cite.

#### Justification

The underlying problem is not just "the table cell is wrong" — it is
that scope semantics across the full token-lifecycle (initial grant,
refresh, down-scope, revoke) are scattered and only partly
specified. A dedicated subsection lets the WG fix all of those at
once with a single editorial change, and it gives certification
programs and conformance test kits a stable anchor to cite. It also
avoids the readability problem of jamming a multi-bullet conformance
list into a single table cell, which is the main risk with
Disposition A's inline edit.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as **Persuasive with Modification → editorial only**
or close as **Not Persuasive / Already Specified**, on the grounds
that the existing language ("the same as the scope of the original
access token, and it can be different from the scopes requested by
the app") together with the underlying OAuth 2.0 / OIDC
specifications already covers the case, and that any
implementer-facing clarification should live in non-normative
implementer guidance (e.g., a community FAQ or the SMART
implementers' page) rather than in the IG itself.

#### Justification

Defensible only on the narrow argument that SMART explicitly
profiles OAuth 2.0 and OIDC, and those base specifications already
permit a server to omit `id_token` on refresh and to narrow scope on
refresh. Under that reading, the current SMART text is not wrong,
just terse, and adding normative SMART-specific wording risks
diverging from the underlying OAuth/OIDC behavior. This
interpretation, however, is contradicted by the Zulip thread (where
even SMART editors agree the text is ambiguous) and by the
real-world impact captured in the Inferno (g)(10) issue, so this
disposition is included only as a completeness option.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The Zulip thread shows that the SMART
editors already agree on the substantive answer (data-access scopes
persist; `openid`/`fhirUser` need not be re-asserted), so the
question is purely how best to put that into the IG. Disposition A
solves the immediate ambiguity but bolts a multi-rule conformance
list onto a single table cell, which is hard to read and gives
test-kit authors no stable anchor to cite. Disposition B fixes the
same ambiguity, normatively addresses the adjacent issues
(down-scoping on refresh, treatment of omitted scopes,
launch-context scopes) that the Inferno (g)(10) issue and the chat
discussion both touch on, and produces a single subsection URL that
Inferno and ONC can cite directly. Given that the ticket has been
open since 2022 and has already caused a certification-program
issue, the small additional editorial cost of a dedicated subsection
over an inline table edit is well worth the durability and
test-kit-friendliness it buys. Disposition C should be rejected:
the chat thread itself demonstrates that the current text is not
self-explanatory to implementers, and "go read OAuth 2.0 and OIDC"
is precisely the answer that drove the original Inferno divergence.
