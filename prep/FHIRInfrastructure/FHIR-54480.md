## Summary

FHIR-54480 reports that the FHIR Mapping Language (FML) ANTLR grammar
(`mapping.g4`) does not support what the reporter calls the "super
simple identity form" of a mapping rule, written as:

```
src -> tgt: type, subtype, action, recorded;
```

In this form a single rule names a source (`src`) and a target (`tgt`)
and then lists the property names that should be implicitly copied from
source to target (each property becoming an identity copy of the same
named property on the source). The current grammar requires each of
those copies to be expressed as a separate `ruleTarget` (one
`ruleContext` per property, optionally followed by `= transform`), so
the comma-separated property-list shorthand is not parseable today. The
ticket also asks the secondary question of how this shorthand would
even be represented in the `StructureMap` resource — i.e., whether one
FML rule of this shape collapses into one `StructureMap.group.rule`
with multiple `target` entries, or expands into N rules.

This ticket is effectively a narrower restatement of
[FHIR-54477](https://jira.hl7.org/browse/FHIR-54477), filed by the same
reporter ~45 minutes earlier, which raises the same "simple identity
form" gap plus two adjacent asks (give the simple form its own
sub-heading so it is easy to reference, and better describe how
identity transforms implicitly create the target). FHIR-54480 should
almost certainly be triaged together with FHIR-54477. It also sits
inside the broader FML quality concerns captured in
[FHIR-54473](https://jira.hl7.org/browse/FHIR-54473) ("FHIR Mapping
Language not ready for normative", currently *Waiting for Input*) and
the related grammar-vs-spec inconsistency tracked in
[FHIR-46284](https://jira.hl7.org/browse/FHIR-46284) ("FML Grammar for
target rules is inconsistent with the spec"), both of which document a
pattern of the published FML grammar lagging behind both the prose and
the reference Java implementation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-54480 as the grammar half of FHIR-54477 and resolve them
together with two concrete changes in `HL7/fhir`:

1. **Grammar (`source/mapping.g4`)** — extend the `ruleTarget` /
   `rule` productions so that a single rule of the form
   `src -> tgt: prop1, prop2, prop3;` is parseable as a "property
   list" identity-copy form. Concretely add a new alternative such as
   `ruleTarget : ruleContext ':' identifier (',' identifier)* ;` (or a
   dedicated `propertyList` production wired in at the rule level),
   with the semantics that each listed identifier is treated exactly
   as if it had been written as `tgt.identifier = src.identifier as
   identifier "<auto-name>";`.
2. **Spec page (`source/mapping-language.html`)** — add a new numbered
   sub-heading "Simple identity / property-list form" under the rule
   syntax section, show the canonical example `src -> tgt: type,
   subtype, action, recorded;`, state that each listed property is an
   implicit identity copy from source to target, and explicitly
   document the equivalent `StructureMap` representation: one
   `StructureMap.group.rule` whose `source` references `src` and whose
   `target` array contains one entry per listed property with
   `transform = create` only when the type is ambiguous (e.g. choice
   types) and otherwise an implicit copy, mirroring how the existing
   identity transforms are described.

#### Justification

The FHIR Core spec already says identity transforms implicitly create
the target, and reference engines (HAPI Java, the Firely/Brian
serializer work referenced in the 2021 Zulip thread) tolerate
shorthand forms the grammar doesn't formally describe — the same
pattern FHIR-46284 documents for the `(fpExpression)` and
context-less-target shortcuts. Bringing `mapping.g4` into line with
the prose and with field practice is consistent with FHIR design
guidance that the published grammar should be authoritative for
implementations. The `StructureMap` representation can be specified
without any resource changes because every property-list rule maps
1:1 to a `group.rule` with N `target` entries, each of which is
already representable today.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve FHIR-54480 as **prose-only**: do **not** add a new production
to `mapping.g4`, but in `source/mapping-language.html` add an
explicit, numbered sub-heading that clarifies that today the shorthand
`src -> tgt: type, subtype, action, recorded;` is **not** valid FML
and must be expanded into N separate rules (one per property), each of
which is an identity copy. Include the `StructureMap` representation
(one `group.rule` per property, with `target.transform` either
implicit or `create`/`copy` as appropriate) and a worked example
showing the long form. Roll the request for the grammar change into
the broader FML cleanup tracked under FHIR-54473 / FHIR-46284, where
it can be sequenced with the other grammar-vs-spec corrections rather
than landing as a one-off addition.

#### Justification

FHIR-54473 ("FML not ready for normative", *Waiting for Input*) and
the WGM Wednesday Q4 discussion indicate FHIR-I is currently weighing
whether to keep iterating on FML inside core or move it to an
incubator/external IG. Adding a new shorthand production to
`mapping.g4` while that decision is pending risks either (a) extending
a grammar that may shortly move out of core or (b) creating a new
construct whose round-trip behaviour to/from `StructureMap` (already
flagged as lossy in FHIR-54473's WGM notes — extensions/comments
disappear) hasn't been worked through. A prose-only clarification
fully answers the second half of FHIR-54480 ("how is this represented
in the StructureMap resource?") and gives implementers an unambiguous
answer today, without committing to a grammar surface that the WG may
want to redesign holistically.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54480 as a duplicate of FHIR-54477 and close it with a
"Persuasive with mod / Not Persuasive — duplicate" disposition,
pointing the reporter to FHIR-54477 (which captures the same gap plus
the heading-restructure and identity-creation prose asks). The
substantive work — grammar update, prose update, and the StructureMap
representation question — would be tracked and resolved on FHIR-54477
only.

#### Justification

FHIR-54480 and FHIR-54477 were filed by the same reporter on the same
day, ~45 minutes apart, and the body of FHIR-54480 is verbatim a
subset of FHIR-54477's first bullet plus its third sub-bullet
("how is this represented in the StructureMap?"). Carrying both
through ballot disposition risks divergent resolutions and double
work; declining one as a duplicate is the standard FHIR-I practice in
this situation and keeps the audit trail clean.

---

### Recommendation

**Recommended disposition:** C, then resolve the underlying request on FHIR-54477 using the substance of Disposition A (or B, depending on FHIR-54473's outcome).

The cleanest path is to close FHIR-54480 as a duplicate of FHIR-54477
so the WG only has one ticket carrying this change, and then make the
real disposition on FHIR-54477 itself. On the substance, A is the
more useful long-term answer — the grammar genuinely doesn't describe
syntax that the spec's own narrative implies and that implementers
already informally support, and the `StructureMap` representation is
trivial (one `group.rule` per listed property), so we can document
both at once. However, if FHIR-I lands FHIR-54473 in the direction of
moving FML out of normative core into an incubator, B becomes the
more defensible answer because we'd be deferring a grammar surface
change to the body that ends up owning FML. Either way, the WG should
decide A-vs-B in the context of FHIR-54473 and apply the result on
FHIR-54477; FHIR-54480 itself should not carry a separate parallel
resolution.
