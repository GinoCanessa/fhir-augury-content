## Summary

The reporter (Brian Postlethwaite) raises two related defects in the formal
ANTLR grammar published in the FHIR Mapping Language (FML) specification
(`mapping-language.html` §7.8.0.8.2) where the grammar disagrees with both
the prose of the spec and with implementations that have shipped for years.

1. The **`ruleTarget`** production requires a `ruleContext` before the
   optional `'=' transform`, which makes a *context-less* target rule
   illegal. In practice, FML scripts (including HL7's own cross-version
   maps such as `HL7/fhir-cross-version/input/R3toR2/CarePlan.fml#L25`)
   write target rules whose only purpose is to evaluate a FHIRPath
   expression and bind the result to an alias for use by a dependent
   group — e.g.
   `... -> (src.careTeam.reference.resolve()) as ct then plnParticipant(ct, tgt) "...";`
2. The **`transform`** production does not list the `'(' fpExpression ')'`
   "evaluate FHIRPath" shortcut form, even though the surrounding spec
   prose describes it.

The HAPI Java reference implementation
(`StructureMapUtilities.java` line 1122 in `hapifhir/org.hl7.fhir.core`)
has supported both forms for some time, so the ticket is fundamentally a
spec-vs-implementation alignment request.

The reporter proposes concrete grammar replacements:

```
ruleTarget
  : ruleContext ('=' transform)? alias? targetListMode?
  | transform alias? targetListMode?
  | invocation alias? // alias not required when simply invoking a group
  ;

transform
  : literal           // trivial constant transform
  | ruleContext       // 'copy' transform
  | invocation        // other named transforms
  | '(' fpExpression ')' // fhirpath based expressions
  ;
```

and notes that the third `ruleTarget` alternative (`invocation alias?`) is
arguably redundant once `transform` covers it.

There are no other Jira tickets cross-referenced to FHIR-46284. A single
Zulip thread on `#FHIR Mapping Language > fhirpath target variable`
captures Brian raising the issue and Vadim Peretokin agreeing with the
proposed formalisation.

## Details

**Description:**

There are 2 main issues, the evaluate shortcut fhirpath mode isn't in the
grammar, and the context-less target is also not supported.

https://hl7.org/fhir/mapping-language.html#7.8.0.8.2

Existing:

```
ruleTarget
  : ruleContext ('=' transform)? alias? targetListMode?
  | invocation alias? // alias is not required when simply invoking a group
  ;
```

There are several examples where there is no ruleContext provided (and is
intending just to use a fhirpath expression to put in a variable (alias)
for use in a dependent group.

Suggest updating to this:

```
ruleTarget
  : ruleContext ('=' transform)? alias? targetListMode?
  | transform alias? targetListMode?
  | invocation alias? // alias is not required when simply invoking a group
  ;
```

We could even omit the last line, as that would be covered by the
`transform` anyway.

And also the `transform` rule updates to this (including the fhirpath
shortcut):

```
transform
  : literal           // trivial constant transform
  | ruleContext       // 'copy' transform
  | invocation        // other named transforms
  | '(' fpExpression ')' // fhirpath based expressions
  ;
```

All of this is already implemented in the HAPI Java Mapping engine (and
has been for some time):
`hapifhir/org.hl7.fhir.core .../StructureMapUtilities.java#L1122`.

The text for the concrete syntax for the *evaluate FHIRPath* shortcut is
already in the spec, but there isn't anything indicating that the context
is mandatory or optional (though the grammar suggests it is mandatory).

An example that uses this format is:

```
src.careTeam as vs where reference.resolve().exists()
  -> (src.careTeam.reference.resolve()) as ct
  then plnParticipant(ct, tgt) "CarePlan-participant";
```

Which is from the cross-version maps:
`HL7/fhir-cross-version/input/R3toR2/CarePlan.fml#L25`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the reporter's two grammar amendments verbatim to the FHIR Core spec:

1. In `source/mapping.g4` (and the mirrored grammar block in
   `source/mapping-language.html` §7.8.0.8.2 *Target rules*) replace

   ```
   ruleTarget
     : ruleContext ('=' transform)? alias? targetListMode?
     | invocation alias?
     ;
   ```

   with

   ```
   ruleTarget
     : ruleContext ('=' transform)? alias? targetListMode?
     | transform alias? targetListMode?
     | invocation alias?
     ;
   ```

2. In the same grammar source, replace

   ```
   transform
     : literal
     | ruleContext
     | invocation
     ;
   ```

   with

   ```
   transform
     : literal           // trivial constant transform
     | ruleContext       // 'copy' transform
     | invocation        // other named transforms
     | '(' fpExpression ')' // fhirpath based expressions
     ;
   ```

3. Update the surrounding prose in `source/mapping-language.html`
   §7.8.0.8.2 to (a) explicitly state that the `ruleContext` before
   `'=' transform` is *optional* when the `transform` is a FHIRPath
   `(fpExpression)` whose only purpose is to bind the result to the
   target alias, and (b) document the `(fpExpression)` evaluate
   shortcut as a first-class transform form.

4. Add a worked example using the cross-version maps form
   `... -> (src.careTeam.reference.resolve()) as ct then plnParticipant(ct, tgt) "CarePlan-participant";`
   so implementers can pattern-match against an HL7-published
   reference.

(Optional sub-step: drop the `| invocation alias?` alternative from
`ruleTarget`, since the new `transform` alternative subsumes it — this
matches the reporter's note. Keeping it preserves backward grammar
compatibility for parsers that already special-case `invocation`, so
it is recommended to leave it for now.)

#### Justification

The change brings the published spec into line with (a) the prose that
already describes the FHIRPath evaluate shortcut, (b) the HAPI Java
reference implementation that has accepted both forms for years
(`StructureMapUtilities.java#L1122`), and (c) HL7's own cross-version
maps under `HL7/fhir-cross-version` that depend on the context-less
form. Refusing the change leaves the normative grammar contradicted by
HL7's own published artifacts and by the only widely-used reference
engine — a worse outcome than any compatibility risk from broadening
the grammar. The fix is a pure addition of accepted productions
(grammar widening), so existing valid FML scripts remain valid. The
bespoke HL7 publisher (per the `HL7/fhir` briefing) makes the edit
mechanically simple: two source files in `source/` and no IG-publisher
pipeline to coordinate.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same grammar widening as in Disposition A, but in two
narrower parts and with a stricter framing:

1. **Grammar:** Add the `(fpExpression)` alternative to `transform`
   exactly as proposed (this is uncontroversial and already documented
   in prose).
2. **`ruleTarget`:** Instead of adding a free-standing
   `transform alias? targetListMode?` alternative — which legalises
   *any* transform as a context-less target — restrict the new
   alternative to the FHIRPath form only:

   ```
   ruleTarget
     : ruleContext ('=' transform)? alias? targetListMode?
     | '(' fpExpression ')' alias targetListMode? // context-less FHIRPath target; alias REQUIRED
     | invocation alias?
     ;
   ```

   The `alias` is made mandatory in this new alternative because a
   context-less target whose result is not bound to anything is
   meaningless.
3. Add normative prose stating that a context-less target rule is
   only legal when its transform is the FHIRPath evaluate shortcut and
   an alias is provided, mirroring the only pattern actually observed
   in HL7's cross-version maps.

#### Justification

Disposition A's broader form ("any `transform` may stand alone as a
target") is technically simpler but legalises shapes nobody currently
emits — e.g. a bare `literal` or a bare `ruleContext` (copy) target
with no LHS. That risks becoming a future surprise: implementers will
need to decide what those forms *mean*, even though the originating
use case (FHIRPath shortcut binding) is much narrower. Disposition B
keeps the grammar change tightly scoped to the actually-observed
pattern in `HL7/fhir-cross-version`, preserving room for future,
deliberate widening if other use cases emerge. The trade-off is a
slightly less elegant grammar and a subtle divergence from what HAPI
already accepts (HAPI is permissive in line with Disposition A), so
HAPI would in practice still parse a superset of what the spec
requires — which is the normal direction of permissiveness.

---

### Disposition C: Decline

#### Proposal

Decline the grammar change and instead resolve the inconsistency in
the *other* direction: amend the spec prose in
`source/mapping-language.html` §7.8.0.8.2 to make explicit that
context-less target rules and the `(fpExpression)` shortcut are **not**
part of the normative FML grammar, and treat HAPI's behaviour and the
`HL7/fhir-cross-version` examples that use them as bugs to be fixed
downstream (rewrite those FML scripts to use a `ruleContext` LHS, e.g.
`tgt.dummy = (src.careTeam.reference.resolve()) as ct`-style binding
through an explicit context).

#### Justification

This option is defensible only on the grounds of grammar minimality —
keeping the FML surface as small as possible eases implementation in
new engines and reduces the test matrix. However it requires actively
breaking real, HL7-published FML in `HL7/fhir-cross-version` and
demanding that HAPI tighten its parser, which both the reporter and
Vadim Peretokin have implicitly argued against on Zulip. There is no
community signal in favour of this direction, and the prose already
describes the shortcut as if it were supported, so this option is
listed for completeness rather than as a credible candidate.

---

### Recommendation

**Recommended disposition:** A

The grammar is currently provably wrong against HL7's own published
content (the `HL7/fhir-cross-version` `CarePlan.fml` example) and
against the only mature reference implementation (HAPI's
`StructureMapUtilities`). The Zulip thread shows community agreement
(Brian + Vadim) and no opposition. The change is a *widening* of an
existing grammar, so backward compatibility is preserved by
construction. Disposition B is a reasonable conservative fallback if
the workgroup is uncomfortable with a free-standing `transform`
alternative on `ruleTarget`, but it creates a deliberate spec/HAPI
divergence (HAPI accepts the broader form) that the workgroup would
have to live with going forward. Disposition C is not credible because
it would require breaking already-published HL7 FML. The work is
mechanically small per the `HL7/fhir` briefing — two files in
`source/` (`mapping-language.html` and `mapping.g4`), the bespoke
publisher will pick it up directly, and no cross-repo coordination is
needed.
