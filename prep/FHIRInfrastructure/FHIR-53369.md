## Summary

Bryn Rhodes (reporter) is asking the FHIR Infrastructure work group to
clarify the FHIR Core specification's treatment of *detached* JSON Web
Signatures and to consider adding a clearer worked example. In
`signatures.html` the JWS rules state (line 371) that "the signature is
a Detached Signature … and the payload SHALL be omitted", and the JWS
example (lines 540‑657) is in fact a detached JWS — the middle segment
of the `Signature.data` JOSE compact-serialization string is empty
(`..`) and the page explicitly notes "The payload is not provided,
since this is a detached signature." However, because the entire
`Signature.data` JWS string is embedded inside the very Bundle it
signs, readers find the framing confusing: at first read the signature
appears to be *attached* to the bundle, and only by decoding the
base64 JWS and noticing the empty middle segment does it become
obvious that the JWS itself is detached in the RFC 7515 Appendix F
sense (the JWS's payload, not the resource that carries the JWS, is
what is "detached"). The reporter offers a worked detached-signature
example that the CRMI IG team has prepared
([HL7/crmi-ig#94](https://github.com/HL7/crmi-ig/pull/94)) as
candidate material that could be borrowed or referenced to make the
distinction crisper.

There are no linked Jira tickets, no comments on the ticket, and no
Zulip discussions surfaced by either the cross-reference index or a
keyword search for the ticket key. The only cross-reference is the
inline mention of the CRMI PR.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit `source/signatures.html` in `HL7/fhir` to clarify the use of the
word "detached" and to make the existing JWS example unambiguously
read as a detached JWS:

1. In the JWS rules list (line 371), change the bullet from
   *"The signature is a [Detached](rfc7515#appendix-F) Signature
   (where the content that is signed is separate from the signature
   itself), and the payload SHALL be omitted"* to wording that
   explicitly anchors "detached" to the JWS payload, e.g.:
   *"The JWS SHALL be a Detached JWS as defined in
   [RFC 7515 Appendix F](https://tools.ietf.org/html/rfc7515#appendix-F):
   the JWS payload (the middle segment of the JOSE compact
   serialization carried in `Signature.data`) SHALL be empty. Note
   that the JWS string itself is still carried inside the FHIR
   resource via `Signature.data`; "detached" refers to the JWS's
   payload, not to the location of the JWS."*
   Apply the same wording fix to the equivalent XML Digital Signature
   bullet (line 398).
2. Add a short framing paragraph at the top of the
   `<h3>JWS example</h3>` section (just after line 545) that names
   the same distinction before the reader hits the bundle: e.g.,
   *"The example below is a detached JWS (RFC 7515 Appendix F): the
   JOSE compact serialization in `signature.data` has an empty middle
   segment (`..`), and the verifier reconstructs the payload from the
   resource's canonical form. The JWS itself is carried inline within
   the signed `Bundle.signature`, which is consistent with the
   detached-JWS pattern."*
3. Reinforce in the existing Payload subsection (around lines
   641–657) that the empty `..` segment in `Signature.data` is *the*
   marker of a detached JWS, and link forward to the verification
   subsection (line 719).

No new resources, elements, or constraints are introduced; the
`Signature` datatype, `sigFormat`, `targetFormat`, and existing JWS
rules are unchanged. The CRMI PR
([HL7/crmi-ig#94](https://github.com/HL7/crmi-ig/pull/94)) is added
as a non-normative external pointer (e.g., a "see also" link) rather
than imported wholesale, until that PR merges.

#### Justification

The reporter's confusion is a documentation defect, not a defect in
the normative rules. The page already requires payload omission and
already states that the example payload is not provided, but those
two facts are separated by ~280 lines and the word "detached" is
overloaded between the JWS-payload sense (RFC 7515 Appendix F) and
the colloquial "signature in a separate file" sense. Tightening the
wording where it first appears, and adding a framing paragraph
before the example, addresses the confusion at minimal cost and
without breaking any conformance behaviour. This is a pure
`source/signatures.html` edit and matches the FhirCore "Update
narrative / prose" recipe from the briefing; the bespoke publisher
will pick it up with no other plumbing changes.

### Disposition B: Alternative Approach

#### Proposal

In addition to the wording clarifications in Disposition A, add a
**second worked example** to `source/signatures.html` that shows a
detached JWS where the `Signature.data` JWS string is *transmitted
out-of-band* from the resource it signs (e.g., as a sidecar file
referenced from `Signature.who` / `Signature.when` plus an external
descriptor). Model the example on the CRMI PR
([HL7/crmi-ig#94](https://github.com/HL7/crmi-ig/pull/94))'s
end-to-end walkthrough (sign → canonicalize → produce JWS →
verify), but pared down to FHIR Core's pedagogical needs: a small
Bundle, the canonical JSON, the resulting detached JWS, and the
verification steps. Place it after the existing `<h3>JWS example</h3>`
section, with a heading like `<h3>JWS example — sidecar JWS</h3>`,
so readers see two complementary patterns:

- existing example: detached-JWS *carried inline* in
  `Bundle.signature.data`;
- new example: detached-JWS *carried out-of-band* alongside the
  signed bundle.

If and when the CRMI PR merges, this second example can be kept in
sync with the CRMI artifact-signing walkthrough so implementers see a
single coherent story across Core and CRMI.

#### Justification

Disposition A removes the linguistic confusion but leaves the page
showing only one shape of detached signature (JWS carried inside the
signed resource). Some implementers — including those building IG
distribution and provenance tooling, which is exactly the audience
the CRMI PR targets — will still wonder how to use FHIR's signature
machinery when the JWS is genuinely transmitted separately from the
bundle. A second worked example covers that case explicitly and
forecloses the next round of "but what about …" tickets, at the cost
of more page real estate and a second example to maintain. The
gotcha from the briefing about `<a name="…"></a>` anchors applies:
add a stable `#example-jws-sidecar` anchor and search `source/` for
inbound link impact before merging. No CRMI-side dependency is
created — the FHIR Core change is self-contained — but it stays
aligned with the CRMI direction surfaced in the cross-reference.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-53369 as *Not Persuasive* on the grounds that the existing
text in `source/signatures.html` is already technically correct: the
JWS rules (line 371) require the JWS to be a Detached JWS with the
payload omitted, the example's `Signature.data` does have an empty
middle segment, and the Payload subsection (lines 641–657) explicitly
explains that the payload is omitted because the signature is
detached. Close the ticket with a comment pointing the reporter at
RFC 7515 Appendix F (which the page already links) and at the
existing Payload subsection, and decline to import the CRMI PR's
example into FHIR Core on the basis that worked artifact-signing
walkthroughs belong in IGs (such as CRMI), not in the base
specification.

#### Justification

The normative rules are unambiguous when read against RFC 7515
Appendix F, and a reporter who already understands "detached JWS" in
the RFC sense can reconcile the example by inspecting the `..`
segment in `Signature.data`. Declining keeps the base specification
small and avoids competing example styles between FHIR Core and CRMI.

---

### Recommendation

**Recommended disposition:** B

Disposition C is hard to defend: the reporter is one of the more
experienced implementers in the community, and if the framing reads
as contradictory to *them*, it will read as contradictory to most
implementers, regardless of whether the underlying rules are
technically correct. "The text is already right if you decode the
base64" is not a serviceable response for a reference specification.

Disposition A fixes the immediate wording problem and is the minimum
defensible response. Disposition B does the same wording fix *and*
adds the missing pedagogical case (detached JWS carried out-of-band)
that motivated the reporter to point at the CRMI PR in the first
place. The marginal cost is one additional worked example in a
single HTML page (`source/signatures.html`); the marginal benefit is
that the FHIR Core signatures page becomes the canonical
explanation of both detached-JWS shapes, and downstream IGs (CRMI in
particular) can simply link back to it instead of re-deriving the
walkthrough. Given that the only cross-reference on this ticket is
exactly such a downstream walkthrough, B is the disposition that
most directly retires the underlying need.

Two caveats the work group should weigh before finalising B:

1. The CRMI repo's `repo-analysis` briefing is **not** present in
   the cache, so any commitment to "track CRMI's walkthrough"
   should be parked until `repo-analysis HL7/crmi-ig` has been run
   and the CRMI authoring layout (FSH vs. pages, IG-publisher
   conventions) has been confirmed.
2. The CRMI PR is still **Open**; FHIR Core should not import its
   example verbatim until it merges. A non-normative "see also"
   link is safe in the interim.
