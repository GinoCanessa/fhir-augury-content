## Summary

The ticket reports a typo in the human-readable text of an invariant on the
`SampledData` datatype in FHIR Core. The current text reads "A SampledData
SAHLL have either an interval and offsets but not both"; "SAHLL" should be
"SHALL". This is a pure narrative correction — the FHIRPath expression of the
invariant is not affected, and there are no semantic changes implied. No
related Jira tickets, Zulip threads, or GitHub items were surfaced by
cross-reference lookups, which is consistent with a small, isolated technical
correction.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Correct the typo in the SampledData invariant's human-readable text from
`A SampledData SAHLL have either an interval and offsets but not both` to
`A SampledData SHALL have either an interval and offsets but not both`.
The change is a single-word fix to the `human` attribute of the relevant
constraint element on the SampledData StructureDefinition's
`<differential>` in `HL7/fhir` (under `source/sampleddata/`). Leave the
constraint id, FHIRPath `expression`, severity, and key unchanged. Do
not edit `<snapshot>` content or `source/datatypes.html` directly — both
are regenerated by the publisher from the differential.

#### Justification

This is a pure technical correction with no semantic effect: the
machine-evaluated FHIRPath expression is unaffected and only the
human-facing description changes. Editing the `<differential>` is the
documented authoring path in the HL7/fhir briefing, and the publisher
will propagate the corrected text into the rendered datatypes page and
the regenerated snapshot. Accepting the fix as requested costs nothing,
removes an obvious embarrassment in a normative-track datatype, and
matches the ticket type (Technical Correction).

---

### Disposition B: Alternative Approach

#### Proposal

Apply the typo fix exactly as in Disposition A, and at the same time
audit the `human` text of the other SampledData invariants (and, more
broadly, the invariants on neighbouring primitive/complex datatypes) for
similar spelling or grammar errors using a quick `rg -n "SAHLL|MUSY|MSUT|SHOULDD"`
style sweep over `source/`. Roll any additional findings into the same
correction (or note them as follow-up tickets, per work-group preference).
The code change still lands only in `<differential>` `human` attributes
under `source/<datatype>/` in `HL7/fhir`.

#### Justification

The ticket's underlying need is "the rendered SampledData constraint
text is wrong"; the broader implementer-facing need is "invariant
narrative text should be free of obvious typos." Bundling a tightly
scoped grep-based audit with this fix is low cost (the briefing's
authoring-root guidance makes the search trivially safe — `source/` is
the only authoring tree) and reduces the chance that the next reader of
SampledData (or a sibling datatype) files yet another typo ticket. The
trade-off versus Disposition A is a marginally larger PR and one extra
review pass; the trade-off versus Disposition C is obvious — the typo
clearly should be fixed.

---

### Disposition C: Decline

#### Proposal

Do not adopt the change. Close as "Persuasive with Modification" or
similar, noting that the misspelling is in human-readable text only,
does not affect conformance or tooling, and can be left to a future
batch editorial pass.

#### Justification

This disposition is not defensible on the merits. The ticket type is
Technical Correction, the fix is a single character, the FHIRPath
expression and constraint key are untouched, and the briefing's
authoring rules make the edit unambiguously safe (`<differential>`
`human` text in `source/sampleddata/`). Declining would leave a visible
typo in a normative datatype's invariant narrative with no offsetting
benefit. Documented here only for completeness.

---

### Recommendation

**Recommended disposition:** A

Accept as requested. This is a one-character correction to the `human`
text of a SampledData invariant in `HL7/fhir`'s `source/sampleddata/`
StructureDefinition `<differential>`. There is no semantic impact, no
cross-repo touch points, and no Zulip or GitHub discussion suggesting
the wording itself is contested. Disposition B is a reasonable
opportunistic upgrade if the work group is comfortable with a slightly
broader audit, but it should not block the trivial fix; if B is chosen,
land A first and treat any additional findings as a follow-up. Disposition
C should not be selected — the briefing's gotchas are all satisfied by
the minimal edit, and there is no scenario in which leaving "SAHLL" in
the published spec is preferable.
