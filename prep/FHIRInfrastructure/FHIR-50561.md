## Summary

The ticket asks the FHIR Infrastructure work group to "Update Backbone Data
Type" by adding `RelativeTime` to the set of data types that derive from
`BackboneType`. `BackboneType` is the abstract base used by complex data
types that need their own `modifierExtension` slot (today: `Timing`,
`Dosage`, `ElementDefinition`, `ProductShelfLife`, `MarketingStatus`,
etc.). `RelativeTime` is a newer complex datatype (introduced in the R6
build) used to express timing relative to events — it is currently
referenced from R6 work in Dosage (`step.start` / `step.end`), Group
cohort definitions, and EvidenceVariable-related modeling. The reporter
appears to be requesting a structural change so that `RelativeTime`
inherits from `BackboneType` rather than the plain data type / element
base, which would let `RelativeTime` carry `modifierExtension` directly
(a capability already seen as desirable for relative-timing modeling
based on historical implementer threads about "negation" / conditional
timing).

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items linked to this ticket in the augury index, so the surrounding
context is inferred from related FHIR-I and committers conversations
about `RelativeTime` and `BackboneType`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FHIR R6 build, change the parent of the `RelativeTime` datatype
from `DataType` (or `Element`, whichever it currently inherits from) to
`BackboneType`. Concretely:

- Update the `RelativeTime` `StructureDefinition` so its `baseDefinition`
  is `http://hl7.org/fhir/StructureDefinition/BackboneType`.
- Ensure `RelativeTime` exposes the `modifierExtension` element
  (inherited from `BackboneType`) and that the diagram / page rendering
  shows the correct hierarchy (the same kind of rendering bug Brian
  Postlethwaite flagged for `Timing` in the 2022 BackboneType thread
  must be re-checked).
- Add `RelativeTime` to any list / matrix on the
  `datatypes.html` and `backboneelement.html` pages that enumerates
  BackboneType-derived datatypes (alongside `Timing`, `Dosage`,
  `ElementDefinition`, etc.).
- Re-run the spec build; verify downstream resources that already use
  `RelativeTime` (R6 `Dosage.step.start`/`step.end`, Group cohort
  modeling, `EventDefinition` references) still validate.

#### Justification

`RelativeTime` is structurally a complex, behaviour-bearing datatype
similar to `Timing` and `Dosage` — both of which are BackboneType — so
parent-class consistency argues for the same treatment. Implementer
discussion as far back as 2016 (the OrderSets thread) and the more
recent R6 Dosage / cohort discussions show real demand for being able to
attach modifying semantics (e.g., negation, conditional offsets) to a
`RelativeTime` instance, which `modifierExtension` is the standard FHIR
mechanism for. Because `RelativeTime` is new in R6 and has no published
release (per Bryn Rhodes in the IG-creation thread), this change has no
backward-compatibility cost.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `RelativeTime` parented to `DataType` and instead address the
underlying need for "modifying semantics on relative timing" via
explicit, narrowly-scoped standard extensions on `RelativeTime`. For
example, define a small set of extensions under
`http://hl7.org/fhir/StructureDefinition/relativetime-*` (e.g.,
`relativetime-negation`, `relativetime-condition`) that can be applied
where modifying behaviour is required, and document on the
`RelativeTime` page that consumers must support those extensions when
present. Optionally, schedule a follow-up ticket to revisit the parent
class once usage stabilises.

#### Justification

Promoting `RelativeTime` to `BackboneType` is a change that has ripple
effects in the data-type hierarchy, the published rendering, the IG
publisher (R4↔R5↔R6 cross-version mappings already do not handle
`RelativeTime`, per Bryn's R4/R5 IG threads), and tooling that
enumerates BackboneType-derived datatypes. Using targeted extensions
delivers the same expressive power for the small number of known cases
without locking R6 into a structural choice that is hard to revisit.
This is also more conservative for a Triaged Medium-priority Technical
Correction whose description is a single sentence and lacks documented
use cases. The trade-off versus Disposition A is that authors lose the
generic `modifierExtension` slot and must instead use the named
extensions, which is slightly less ergonomic and requires extension
publication.

---

### Disposition C: Decline

#### Proposal

Decline the request as written, on the grounds that the ticket does not
articulate the implementer problem the change is meant to solve. Ask
the reporter to either (a) supply concrete use cases where existing
`RelativeTime` modelling is insufficient and `modifierExtension` would
fix it, or (b) restate the request as a documentation correction (e.g.,
"the BackboneType page should list RelativeTime") if the intent is
purely cosmetic. Resolve as "Persuasive with Modification — Not
Persuasive without further input."

#### Justification

The description is a single sentence with no scenario, no comments, no
linked tickets, and no Zulip discussion attached. FHIR-I's prior
BackboneType thread (2022) shows that the parent-class question is
non-trivial and prone to rendering / hierarchy errors, and the 2023+
RelativeTime threads show active modelling work where a structural
change at the parent-class level could conflict with in-flight PRs
(e.g., the EventDefinition addition Bryn Rhodes mentioned in Nov 2025).
Without a documented use case, accepting this risks introducing
inconsistency in the datatype hierarchy for limited demonstrated value.

---

### Recommendation

**Recommended disposition:** A

`RelativeTime` is, in shape and in usage, in the same category as
`Timing` and `Dosage`: it is a complex, semantically-rich datatype that
is already being embedded in modifier-bearing BackboneType structures
(R6 `Dosage.step.start`/`step.end`, Group cohort modelling) and is
actively being extended by FHIR-I (the EventDefinition addition).
Inheriting from `BackboneType` brings `RelativeTime` in line with its
peers, gives implementers a standard mechanism to express modifying
semantics that the community has been asking about since the original
OrderSets discussions, and — because R6 is unpublished — incurs no
backward-compatibility cost. Disposition B's targeted-extension
approach is workable but spreads the burden across multiple
publications and is harder to align across the resources already using
`RelativeTime`. Disposition C is defensible only if the reporter cannot
supply the missing use-case detail; even then, the structural argument
for parent-class consistency stands on its own. The recommendation is
to accept, with a follow-up to ensure the rendering / hierarchy bugs
historically seen on `Timing` are not reintroduced for `RelativeTime`.
