## Summary

Brian Postlethwaite reports that the FHIR Core specification uses
"definitional blocks" (BNF-/grammar-like notations that show the shape of
a language construct) without ever defining the conventions used to read
them. Several of the offending blocks come from the FHIR Mapping
Language pages — examples include the marker syntax
`/// name{.property}* = {value}`, the `let [name] = [fhirpath
expression];` form, the `group [group-name] (inputs) (extends
[other-group]) (<<stereotype>>) {` group declaration, and the transform
rule `src_context.field as new_variable where (condition) ->
tgt_context.field = create([type]) as new_variable then [details]
"name";`. A reader cannot tell which characters are literal and which
indicate optionality, repetition, or a placeholder identifier, and
examples are visually indistinguishable from the definitional blocks
that precede them. The reporter points to the FHIRPath spec's
[Conventions section](https://build.fhir.org/ig/HL7/FHIRPath/#conventions)
as a model and asks whether using fragments of an ANTLR-style grammar
would remove the ambiguity. No cross-referenced Jira tickets, Zulip
threads, or GitHub items are recorded in the FHIR Augury index for this
ticket.

## Details

**Description:**

The style of representing the language's Definitional blocks, no conventions
are defined in the specification to explain how to read these blocks.

(see how fhirpath does it here https://build.fhir.org/ig/HL7/FHIRPath/#conventions)

Also Examples are not easily differentiated from the definitional blocks.

Some of the mixed formats that need definition (how can we understand what is
literal characters vs features such as optional/repeating etc):

- `/// name{.property}* = {value}`
- `let [name] = [fhirpath expression];`
- `group [group-name] (inputs) (extends [other-group]) (<<stereotype>>) {`
- `[mode] [name]( : [type])`

And blocks like this one in transform Rules, looks like it's defining the
entire block, but I don't think this is a definition, or an example,
especially when it is followed immediately by an example.

- `src_context.field as new_variable where (condition) -> tgt_context.field = create([type]) as new_variable then [details] "name";`

This appears to be implying that the type is optional, unless the square
brackets are meant to be documenting identifiers? If they are, then why
aren't `src_context` and `field` like it?

Would using fragments of an ANTLR style grammar be a better choice for
the definitional parts? That would take out the ambiguities.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an explicit "Conventions" section to the FHIR Core specification, modelled
on the FHIRPath spec's
[Conventions section](https://build.fhir.org/ig/HL7/FHIRPath/#conventions),
that defines the meta-syntax used in every definitional block in FHIR
Core narrative — at minimum:

- `[name]` — placeholder identifier (must be replaced by a concrete
  value).
- `{...}` — grouping, with `?`, `*`, `+` modifiers for optional,
  zero-or-more, and one-or-more repetition (or document whichever
  modifiers are actually in use).
- `(...)` — optional group (or whatever the spec actually intends —
  this needs to be reconciled, since the current Mapping Language
  pages use parentheses both as literal grammar punctuation and as
  optionality markers).
- `<<stereotype>>` — UML-derived stereotype marker.
- Quoted strings and other characters as literals.

Additionally, restyle definitional code blocks so they are visually
distinct from examples (e.g., a "Definition" caption, a different
border colour, or a CSS class such as `def-block` vs. the existing
example styling) and audit the FHIR Mapping Language pages and any
other narrative pages using the same notation to ensure each
definitional block is followed by, not interleaved with, its
example(s).

#### Justification

This directly answers what the reporter is asking for and follows a
precedent already established by HL7 itself in the FHIRPath
specification, which makes the change defensible and low-controversy.
The ambiguity the reporter calls out (e.g., "is `[type]` optional or is
it a placeholder?") is a real interoperability hazard for the FHIR
Mapping Language: implementers writing parsers from the spec narrative
have to guess at the meta-syntax. A single Conventions section plus a
visual distinction between definitions and examples is also the
cheapest way to fix the problem because it leaves the existing
notation in place — only the legend and the styling change.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than documenting the ad-hoc meta-syntax, replace the
definitional blocks on the FHIR Mapping Language pages (and any other
page that defines a sub-language, e.g., the FHIRPath grammar
references in Core) with **fragments of the actual ANTLR grammar**
that are already maintained in the spec source. Each definitional
block would render the relevant grammar production verbatim (e.g., the
`group`, `rule`, `letDeclaration` productions from the FHIR Mapping
Language `.g4`), with the block clearly labelled "Grammar
(ANTLR4)". Examples remain as-is but get a separate "Example" caption
and styling. Add a short pointer at the top of the affected pages
noting the meta-syntax used by ANTLR4 (`?`, `*`, `+`, `|`, `(...)`)
and link to the ANTLR4 reference for anything further.

#### Justification

The reporter explicitly raises this option ("Would using fragments of
an ANTLR style grammar be a better choice for the definitional
parts?"). Its appeal is that ANTLR4 has a published, stable
meta-syntax that implementers already need to understand to consume
the spec's grammar artefacts, so the spec stops maintaining a second,
informal notation that drifts. The trade-off is that ANTLR fragments
are more verbose and less skim-friendly than the current
prose-friendly stubs, and they couple the narrative tightly to the
grammar tooling — if the grammar is restructured for parser reasons,
narrative blocks have to follow. It also requires editorial work on
every affected page, not just the addition of a single Conventions
section.

---

### Disposition C: Decline

#### Proposal

Decline as out-of-scope of a substantive spec change: close the
ticket with a note that the issue is editorial and route it to the
next FHIR Mapping Language editorial pass, without committing to a
specific solution. Optionally, accept only the narrow improvement of
visually distinguishing examples from definitions (a CSS-only change)
and decline the broader request to formalise the meta-syntax.

#### Justification

The substantive content of the FHIR Mapping Language is unchanged by
this ticket — every implementation can already parse FML correctly
because the canonical definition is the ANTLR grammar, not the
narrative. Adding a Conventions section or rewriting the definitional
blocks is editorial polish that competes for limited FHIR
Infrastructure cycles against changes with conformance
implications. Declining (or scoping down to CSS only) is defensible if
no implementer has reported actually being misled in practice; the
ticket as written reads as a usability concern from a single reporter,
without Zulip discussion or other tickets indicating broader pain.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a real and concrete defect — the same
bracket and brace notation is used for at least two distinct meanings
in the FHIR Mapping Language pages (placeholder identifier vs.
optional group), and there is no legend anywhere in FHIR Core that
disambiguates them. FHIRPath has already solved this problem in HL7's
own house style, so Disposition A is both low-risk and precedent-
backed: a single new Conventions section plus a CSS-level distinction
between "Definition" and "Example" blocks resolves the ticket without
re-opening the grammar or touching the normative content. Disposition
B (ANTLR fragments) is intellectually cleaner but a much bigger
editorial undertaking and would couple narrative pages to grammar
maintenance in a way the spec has historically avoided; it is worth
keeping on the table only if the editor finds, while doing the audit,
that the existing notation cannot be salvaged consistently. Disposition
C undersells a real readability problem in a sub-language that
implementers are still actively adopting (FHIR Mapping Language) and
should be rejected unless the workgroup is sure no implementer has
been confused — a claim the lack of Zulip traffic does not actually
support.
