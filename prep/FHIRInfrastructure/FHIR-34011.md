## Summary

The reporter asks the FHIR Shorthand (FSH) language to gain a mechanism for
authoring `StructureMap` resources using the FHIR Mapping Language (FML) —
the concise, FHIR-native syntax that already exists for expressing
`StructureMap` logic. Today FSH can describe `StructureMap` instances only
through generic instance-style assignments on each `group`, `rule`, `source`,
and `target` element, which is verbose and largely unreadable compared with
FML. The reporter proposes either embedding FML directly in FSH (so that an
`Instance` of a `StructureMap` could be authored in mapping-language form
and compiled to a JSON `StructureMap` by SUSHI), or, as an alternative, not
supporting FML natively in FSH but providing some kind of language
translation plug-in / extension point so that an external FML compiler can
produce the `StructureMap` instance that FSH then references. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found
for this ticket in the available indexes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add first-class support in FHIR Shorthand for authoring `StructureMap`
resources via the FHIR Mapping Language. Concretely, introduce a new FSH
construct — for example a `Mapping` block specialised for `StructureMap`,
or an `Instance: ... InstanceOf: StructureMap` body that accepts an FML
literal — so that authors can write something like:

```fsh
Instance: PatientToPerson
InstanceOf: StructureMap
Usage: #definition
* url = "http://example.org/StructureMap/PatientToPerson"
* fml = """
    map "http://example.org/StructureMap/PatientToPerson" = "PatientToPerson"
    uses "http://hl7.org/fhir/StructureDefinition/Patient" as source
    uses "http://example.org/StructureDefinition/Person" as target
    group main(source src : Patient, target tgt : Person) {
      src.name -> tgt.name;
    }
  """
```

SUSHI would parse the embedded FML using the reference FML grammar and
emit the equivalent `StructureMap.group/rule/source/target` JSON, exactly
as the existing FML-to-StructureMap compilers do. The FSH spec would
define: (a) the new keyword/attribute, (b) the rule that the embedded
text must be valid FML and is compiled at build time, (c) error
reporting for FML syntax errors at the FSH source location, and (d) the
interaction with other FSH rules on the same `Instance` (e.g., metadata
overrides like `name`, `status`, `experimental`).

#### Justification

This is the lowest-friction outcome for IG authors who already use FSH
for everything else in their guide. `StructureMap` is the one resource in
FHIR with its own purpose-built compact syntax (FML); forcing authors to
hand-write the verbose JSON-shaped FSH instance for it is the largest
ergonomic gap in FSH today, and the request is well-scoped — it does not
ask FSH to invent anything, only to host a language that HL7 already
maintains. The FML grammar, parser, and JSON projection are stable and
implemented in `org.hl7.fhir.core` (Java) and in `fhir.js` /
`fhir-validator` flows, so SUSHI can either link to a JS port or shell
out to the validator. Native support also lets SUSHI emit source-mapped
errors, which a plug-in approach cannot easily provide.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add FML as a built-in FSH construct. Instead, formalise the
"language translation plug-in" path the reporter mentions as a fallback:
define a generic SUSHI extension point that lets an external tool emit a
JSON `StructureMap` (or any other resource) which SUSHI then ingests as
if it were a hand-authored `Instance`. In practice this means:

1. Document an `input/fml/` (or similar) convention where `.fml` files
   live alongside `.fsh` files.
2. Have SUSHI invoke a configured external compiler (defaulting to the
   HL7 FHIR validator's `-transform` / FML compile mode) on each `.fml`
   file during the build, writing the resulting `StructureMap` JSON
   into the IG's generated resources.
3. Allow FSH to *reference* the generated `StructureMap` by canonical
   URL the same way it already references externally-supplied
   resources, but not to *contain* the FML.

#### Justification

This keeps the FSH grammar small and stable: FSH does not become
responsible for the semantics of a second language whose specification
evolves on its own cadence (FML R5 already differs from R4B). It also
generalises — the same plug-in mechanism could later host CQL, ViewDefs,
or any future DSL — so the cost is amortised. The downside vs.
Disposition A is that authors must context-switch between two file types
and lose single-file authoring; error locations come from the external
compiler rather than from the FSH source. For organisations that already
treat FML as a separate authoring artefact (e.g., maintained by a
mapping team), this is closer to current practice.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the FHIR Shorthand specification.
The ticket should be closed with a `Not Persuasive` / `Not Related to
FSH` resolution, on the grounds that FML is already a complete and
standardised authoring syntax for `StructureMap`, that IG Publisher
already accepts `.fml` files alongside FSH input and produces the
corresponding `StructureMap` resources, and that no FSH-level change is
required to use FML in an IG today.

#### Justification

`StructureMap` is the only FHIR resource with a dedicated authoring
syntax precisely because the JSON form is too verbose to write by hand —
which is exactly why FSH exists for every *other* resource. Layering
FSH on top of FML therefore solves a problem that does not exist:
authors who want concise mapping syntax already have FML, and the IG
Publisher / Sushi pipeline already integrates `.fml` inputs into the
guide. Adding an FSH construct whose body is "verbatim FML" buys the
author nothing beyond the file extension and forces the FSH spec, the
FSH parser in SUSHI, and every downstream FSH-aware tool (linters,
language servers, formatters) to take a hard dependency on the FML
grammar and version strategy.

---

### Recommendation

**Recommended disposition:** B

The underlying need — that IG authors should be able to express
`StructureMap` in FML rather than as verbose FSH instance assignments —
is real, and Disposition C understates it: today the integration between
SUSHI and FML is a convention of the IG Publisher rather than a
documented part of the FSH/SUSHI surface, and contributors regularly
re-discover that `StructureMap` is "the resource FSH cannot really
author". However, Disposition A bakes a second standardised language
into the FSH grammar and creates a long-lived coupling between the FSH
specification and FML's release cadence (which has already diverged
between R4B and R5). Disposition B captures the ergonomic win the
reporter asks for — authors write FML, not JSON-shaped FSH — without
making FSH itself responsible for FML's semantics, and it generalises
to other DSLs the IG ecosystem may want next. The recommendation is
therefore to formalise and document the FML-alongside-FSH workflow as a
first-class SUSHI extension point and to close FHIR-34011 against that
work, leaving the door open to revisit Disposition A if the plug-in
path proves insufficient in practice.
