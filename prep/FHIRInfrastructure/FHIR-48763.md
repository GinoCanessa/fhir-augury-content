## Summary

The reporter (Robert Scanlon) observes that the **User-access Brands `Bundle-example1`** in the SMART App Launch IG (`https://build.fhir.org/ig/HL7/smart-app-launch/Bundle-example1.html`) now fails validation with the error: *"Unknown code 'laboratory' in the CodeSystem 'http://terminology.hl7.org/CodeSystem/organization-type' version '2.0.1'."* The example uses the code `laboratory` from the THO `organization-type` CodeSystem, but that code (along with several other concepts added under [UP-516](https://jira.hl7.org/browse/UP-516)) appears to have been silently dropped between THO v5.5.0 and THO v6.0.2 (specifically removed in THO v6.0.0). Both `validator.fhir.org` (R4 mode) and the HL7 FHIR Validator running locally via Inferno reproduce the failure.

The reporter is uncertain whether the root cause sits with (a) the SMART App Launch example, (b) the validator's terminology configuration, or (c) THO itself, but flags that working examples are critical to downstream testing tools (Inferno, etc.). The single comment notes that the codes were originally added by [UP-516](https://jira.hl7.org/browse/UP-516) ("Include additional concepts in Organization Types", Patient Administration, Resolution: Done, 2024-03), and asks whether their disappearance in the next major THO release indicates a terminology problem rather than a SMART issue.

## Details

**Description:**

I'm getting errors now validating User-access Brands [Example1](https://build.fhir.org/ig/HL7/smart-app-launch/Bundle-example1.html):

> "Unknown code 'laboratory' in the CodeSystem 'http://terminology.hl7.org/CodeSystem/organization-type' version '2.0.1'"

- v6.0.2: https://terminology.hl7.org/6.0.2/CodeSystem-organization-type.html
- v5.5.0: https://terminology.hl7.org/5.5.0/CodeSystem-organization-type.html

It is very helpful for testing tools if examples do not raise validation errors because examples serve as a great way to demonstrate tests running against known good implementations/data sets.

I'm not sure if this is a validator configuration thing, or an issue with the User-access Brands example (e.g. laboratory was decided to not be valid after all), or an issue with THO, or an issue with the Codesystem definition. It does seem odd that the code just disappeared between v5.5.0 and v6.0.2 without any obvious description why though, but I'm not an expert in terminology.

This happens both on https://validator.fhir.org/ (you need to select R4 to validate against in the options), and when running locally through Inferno's integration with the HL7 FHIR Validator as of today.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a **terminology/THO defect** and route the substantive fix to **Vocabulary / UTG**: re-instate the codes added under UP-516 (notably `laboratory`, plus the other four concepts approved by the PA motion of 2024-02-07) in `CodeSystem-organization-type` for the next THO publication, and bump the CodeSystem version accordingly. In parallel, in the **SMART App Launch IG**, leave `Bundle-example1` unchanged (it currently uses an intentionally-approved code) but pin the example's CodeSystem reference / IG dependency to a THO version that contains `laboratory`, so the example continues to validate during the window before THO is corrected. The disposition for FHIR-48763 itself is then "transferred to Vocabulary WG / UTG with a tracking note in SMART App Launch".

#### Justification

UP-516 is a fully-approved Patient Administration motion (5-0-0) that explicitly added these codes to the THO source of truth (`input/sourceOfTruth/fhir/codeSystems/CodeSystem-organization-type.xml`, per the build manifest in UP-516). A code that was added by an approved WG motion should not silently disappear in the next major THO release without a corresponding deprecation/withdrawal motion. THO's own change-management policy expects deprecated codes to be marked, not removed, to preserve referential integrity for prior IG versions and for examples like this one. Because Patient Administration owns `organization-type`, FHIR Infrastructure's role here is to recognize the upstream defect and forward the substantive change rather than patch the example.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the example in the **SMART App Launch IG** rather than (or in addition to) THO. Specifically, in `HL7/smart-app-launch`, edit `Bundle-example1` to replace `Organization.type = laboratory` with a code that is present in the THO version the validator uses (e.g., `prov` / `provider` / a more general organization type still in v6.0.2), and add a brief narrative note explaining that the example previously used `laboratory` (per UP-516) but was updated when that code was withdrawn in THO v6.0.0. Optionally, file a Vocabulary ticket separately to ask whether the removal was intentional.

#### Justification

This is the cheapest, lowest-blast-radius change: it requires only an example edit in one IG and an IG rebuild, and it restores the property the reporter actually cares about — *examples that validate cleanly so downstream tooling (Inferno, validator.fhir.org) demonstrates green*. It does not require a THO publication cycle, and it avoids the risk that the THO removal turns out to have been deliberate (in which case re-adding would itself be inappropriate). The downside is that it papers over the underlying terminology drift; if the removal was a mistake, multiple other IGs and implementations may still fail silently elsewhere.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive (in FHIR Infrastructure)** with a referral note rather than a substantive fix. Rationale: FHIR Infrastructure does not own either the affected example (SMART App Launch belongs to FHIR-I/Security but the change set is governed by the SMART project) nor the affected CodeSystem (`organization-type` is curated by Patient Administration / Vocabulary in THO). FHIR-I would record that the issue should be re-filed against the appropriate work group(s) and close FHIR-48763.

#### Justification

The defect is real but the routing is wrong. THO content errata are handled in the UTG / Vocabulary process, and example errata in `Bundle-example1` are handled by the SMART App Launch project. Resolving FHIR-48763 in FHIR-I risks (a) being a no-op (FHIR-I has no editorial control over THO codes) or (b) appearing to dispose of an issue that the responsible WG has not seen. Decline-with-referral is defensible if the WG concludes that triage to the right owner is more useful than substantive action here.

---

### Recommendation

**Recommended disposition:** A

The reporter has, in effect, already done the diagnostic work: the comment chain identifies UP-516 as the source of `laboratory` and identifies THO v6.0.0 as the point at which the code disappeared, and UP-516's audit trail (PA motion, 5-0-0, build manifest touching `CodeSystem-organization-type.xml`) confirms the addition was intentional and work-group-approved. A code that was added by an approved Patient Administration motion should not vanish from THO without a corresponding withdrawal motion, so the substantive defect is in THO, not in the SMART example. Disposition A routes the fix to the right owners (UTG/Vocabulary for the THO restore; SMART App Launch only for an interim version pin if needed) and preserves the example as an honest demonstration of the approved code set. Disposition B is tempting because it is fast, but it bakes the THO regression into the IG and would mask the same regression for any other implementer using `laboratory` in good faith. Disposition C correctly identifies that FHIR-I is not the substantive owner, but a bare decline without forwarding to UTG/PA leaves the underlying terminology defect untouched, which is precisely what the reporter is asking the project to avoid. The workgroup should adopt A, log the THO/UTG follow-up, and (optionally) instruct the SMART App Launch editors to pin the THO dependency in the interim per A's second clause.
