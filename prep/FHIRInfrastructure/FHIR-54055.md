## Summary

The reporter (Hayden Bader) identifies a small grammatical typo in the
narrative content of the `CapabilityStatement` resource page. In the section
"Using additional resources in CapabilityStatements" the link text reads
"additional resource" (singular) where the surrounding sentence calls for
"additional resources" (plural). The fix is purely editorial — adding a
trailing `s` inside the existing `<a href="resource.html#additional">…</a>`
link — with no normative impact on the `CapabilityStatement` resource, its
StructureDefinition, or any operation. There are no cross-referenced Jira
tickets, Zulip threads, or GitHub items, consistent with this being a
straightforward Technical Correction.

## Details

**Description:**

> Change:
>
> CapabilityStatements may represent the use of [additional resource](https://build.fhir.org/resource.html#additional).
>
> To:
>
> CapabilityStatements may represent the use of [additional **resources**](https://build.fhir.org/resource.html#additional).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the literal one-character edit requested by the reporter. In the FHIR
Core specification's narrative for the `CapabilityStatement` resource, in the
"Using additional resources in CapabilityStatements" subsection (anchor
`#additional`), change the link text from "additional resource" to
"additional resources" so the sentence reads:

> CapabilityStatements may represent the use of
> [additional resources](resource.html#additional).

The `href` target (`resource.html#additional`) is unchanged; only the
link text is pluralised. No StructureDefinition, ValueSet, search parameter,
operation, or example needs to be touched. The change is editorial and
does not require regeneration of derived artifacts beyond the standard
publisher run that produces the HTML page.

#### Justification

The surrounding paragraph and the next paragraph both speak of "additional
resources" in the plural, and the linked target section on `resource.html`
is itself titled in the plural ("Additional Resources defined Outside This
Specification"). The current singular form is a clear grammatical mismatch
with both the local sentence (the verb phrase "represent the use of" takes
a plural complement here) and the destination heading. Pluralising the link
text aligns the prose with the rest of the section, with the target page,
and with the way implementers already speak about Additional Resources in
the FHIR Infrastructure-managed registry. Risk is essentially zero: no
normative content, no tooling impact, no breaking change for implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the typo fix and, in the same edit, lightly tighten the surrounding
paragraph for clarity while it is open. Specifically, change the lead
sentence to something like:

> CapabilityStatements may represent the use of
> [additional resources](resource.html#additional) — i.e., resource types
> defined outside this specification and registered in the
> [HL7 registry](https://chat.fhir.org/#narrow/channel/491058-Additional-Resource-Proposals).

This still resolves FHIR-54055's literal request (link text becomes
"additional resources") but folds the existing follow-on explanation about
the HL7 registry into the same sentence so a reader who only scans the
first line still sees the registry context. No other paragraphs in the
section change.

#### Justification

The Triaged disposition is essentially "edit narrative prose"; the marginal
cost of touching one extra sentence in the same file is trivial, and the
reworded lead reduces ambiguity for first-time readers about what
"additional resources" means in the CapabilityStatement context. This is
the kind of opportunistic micro-improvement that FHIR Infrastructure
typically does accept when a page is already being opened for a typo fix,
and it keeps the Additional Resources concept explicit at the top of the
section.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — Editorial". Argue that the singular
"additional resource" is acceptable as a generic noun phrase referring to
the concept of an Additional Resource (a defined FHIR concept) rather than
a count, similar to "use of HTTP" or "use of FHIR resource", and that
opening a publisher run for a one-character cosmetic change is not
warranted on its own.

#### Justification

The defensible angle here is purely process-based: Technical Corrections
that are this small are sometimes batched against the next substantive edit
to the same file rather than landed individually. There is no semantic harm
from the current wording — a reader following the link still reaches the
correct anchor on `resource.html`, and no implementer behaviour is changed
either way.

---

### Recommendation

**Recommended disposition:** A

This is a textbook Technical Correction: a clear grammatical typo flagged
by an external reviewer, with an unambiguous fix that has no normative
impact and no risk to implementers. The reporter has done the editorial
work — diagnosed the wording, identified the exact location via the live
build URL, and supplied the before/after — so the FHIR Infrastructure cost
is essentially the publisher run. Disposition B's broader rewording is
tempting but expands scope beyond what the ticket asks for and risks
litigating sentence structure in a workgroup call that should take ten
seconds; if the workgroup wants that improvement it should be a separate
ticket. Disposition C is hard to justify when the change is this small,
this clearly correct, and this clearly aligned with the destination
heading's own pluralisation. Accept as requested.
