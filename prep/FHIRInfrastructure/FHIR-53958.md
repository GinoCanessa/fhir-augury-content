## Summary

Brian Postlethwaite proposes a focused cleanup of the **Additional Functions**
section of the FHIR core spec's `fhirpath.html` page. The request has three
parts: (1) remove the trailing sentence and bullet point that claimed "some
resources define their own functions" — this is no longer true now that
QuestionnaireResponse defines none; (2) remove the `comparable(quantity)`
function from FHIR core, because the companion ticket
[FHIR-53957](https://jira.hl7.org/browse/FHIR-53957) (Resolution: Persuasive,
Status: Applied; PR [HL7/FHIRPath#24](https://github.com/HL7/FHIRPath/pull/24))
moved that function into the FHIRPath specification proper, along with the
sentence saying that "some of these functions are candidates for elevation
to the base version of FHIRPath"; and (3) reorganize the section into two
clearer subheadings — **Additional Functions** (true FHIR-only additions) and
**Function Overrides** (functions like `toString`/`ofType` whose behaviour is
re-specified for FHIR types) — and rename "Changes to operators" to
**Operator Overrides** for parallelism, also adding a linkable section heading
for the equivalence operator. The ticket has already been triaged with a
**Persuasive** resolution and the resolution text confirms each of the
proposed edits will be made.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply all four edits to the FHIR core `fhirpath.html` page exactly as
described in the ticket and confirmed in the Persuasive resolution:

1. **Delete the `comparable` function definition** from the Additional
   Functions section (it is now defined in FHIRPath proper — see
   build.fhir.org/ig/HL7/FHIRPath/#fn-comparable, landed via
   [HL7/FHIRPath#24](https://github.com/HL7/FHIRPath/pull/24)).
2. **Delete the trailing sentence** *"Some of these functions are
   candidates for elevation to the base version of FHIRPath when the
   next version is released."* and the trailing bullet point about some
   resources defining their own functions (QuestionnaireResponse no
   longer does; no others remain).
3. **Restructure the section into two subheadings:**
   - **Additional Functions** — true FHIR-only additions remaining
     after `comparable`/`highBoundary`/`lowBoundary` left for FHIRPath.
   - **Function Overrides** — re-specifications of core FHIRPath
     functions for FHIR-specific types (e.g., `toString`, `ofType`).
4. **Rename "Changes to operators" → "Operator Overrides"** for parallel
   structure with Function Overrides, and **add a linkable subheading
   for the equivalence operator (`~`/`!~`)** so it can be deep-linked.

#### Justification

Every element of the request has already been validated: the
`comparable` move is a *fait accompli* in FHIRPath (FHIR-53957 status
Applied, PR merged); the resolution on this ticket is Persuasive; and
the Infrastructure WG Zulip thread captured a proxy "vote for making
this change" from Grahame with no objections raised. The reorganization
is purely editorial — it adds zero new normative content and removes
content that is either obsolete (the QuestionnaireResponse bullet) or
duplicative of FHIRPath proper (the `comparable` function). Adding a
heading to the equivalence operator is a low-cost authoring fix that
unblocks deep-linking from other specs and IGs. The change is
intra-page within `fhirpath.html` of the HL7/fhir core repo.

---

### Disposition B: Alternative Approach

#### Proposal

Apply parts (1), (2), and (4) — the deletions and the equivalence
heading — exactly as in Disposition A, but **defer the "Additional
Functions vs. Function Overrides" subheading split (part 3) into a
separate tracker ticket** to be batched with the related FHIRPath-page
cleanups already on the Infrastructure WG block-vote list (FHIR-55637,
FHIR-55417, FHIR-55495, FHIR-37861, FHIR-54601, FHIR-40271, FHIR-36040,
FHIR-53949). That follow-up ticket would also commit to a precise list
of functions in each new subheading (so the workgroup is approving the
classification, not just the heading names) — for example, deciding
whether `extension()`, `hasValue()`, `resolve()`, `getResourceKey()`,
`getReferenceKey()`, `htmlChecks()`, `memberOf()`, `subsumes()`,
`subsumedBy()`, `conformsTo()` belong under "Additional Functions" and
whether `toString()` / `ofType()` are the only "Function Overrides".

#### Justification

The deletion edits and the linkable equivalence heading are mechanical
and uncontroversial. The heading split, however, has a small
classification judgement call baked in (where does each existing
function land?), and once landed it changes anchor URLs that downstream
IGs and tooling may already deep-link to. Splitting the work lets the
mechanical cleanup ship immediately while the WG reviews a concrete
proposed mapping for the rename, and lets the rename ride with the
larger FHIRPath-page block vote where heading/anchor changes can be
announced together.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as no longer needed because the substantive concern —
the `comparable` function being defined in two places — was addressed
by FHIR-53957's merge into FHIRPath, and the rest of the request is
purely editorial polish that can be picked up opportunistically rather
than as a tracked change.

#### Justification

Strictly speaking, leaving the redundant `comparable` definition in
`fhirpath.html` is not normatively harmful (the FHIRPath spec is now
authoritative for the function), and the heading reorganization is
cosmetic. A workgroup that wanted to minimise churn in a Normative
section just before publication could argue that any avoidable edit
to `fhirpath.html` carries publication-risk and should wait. This is a
defensible-but-weak option included for completeness.

---

### Recommendation

**Recommended disposition:** A

The resolution on this ticket is already **Persuasive**, the
preconditions are all in place (FHIR-53957 Applied, PR merged), and the
Infrastructure WG Zulip thread shows explicit support and zero
objections, including a proxy vote from Grahame to "make this change".
The four edits are tightly scoped to a single page (`fhirpath.html`) in
the core repo, do not change normative behaviour, and actively reduce
contradiction with the FHIRPath spec. Disposition B is technically
reasonable but introduces unnecessary process for an editor's-pen
change the WG has already informally agreed to; Disposition C ignores
the real downside that leaving the duplicated `comparable` definition
in FHIR core invites future divergence with FHIRPath. The cleanest path
is to apply all four edits in one PR against `HL7/fhir`, calling out
the renamed/added anchors in the PR description so downstream
deep-linkers can update.
