## Summary

(`https://hl7.org/fhir/R4/profiling.html`). In the Glossary section, the

which no longer resolves to a valid US Core ValueSet page. The request is a
documentation/link-fix issue against the FHIR R4 narrative content rather
than a normative spec change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a Persuasive documentation fix against the FHIR R4 narrative.
In `profiling.html`'s Glossary section (sourced from the FHIR R4 narrative
in `HL7/fhir`, R4 branch / publication), replace the dead link to
`https://hl7.org/fhir/us/core/ValueSet-us-core-problem.html` with a
working, version-pinned US Core URL — for example the current published
US Core problems-related ValueSet (e.g.
`https://hl7.org/fhir/us/core/STU3.1.1/ValueSet-us-core-condition-code.html`
or the equivalent in the latest published US Core release that is the
intended successor of `us-core-problem`). The change is narrative/glossary
text only; no resource definitions, profiles, or value sets change.

#### Justification

R4 is published and effectively closed for normative changes, but
documentation/link errata are routinely corrected. The link as written
violates the intent of the Glossary, which is to point readers at a
concrete US Core example; a 404 there is purely confusing. Pinning the
replacement URL to a specific US Core version avoids re-introducing link
rot the next time US Core renames or restructures the value set.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than substituting a single replacement URL, restate the Glossary
example without an external IG-specific link. Replace the broken link with
either (a) a generic textual reference such as "see US Core's
problem/condition value sets at the current US Core publication
(`https://hl7.org/fhir/us/core/`)", or (b) a link to a stable FHIR-core
artifact (e.g. a core ValueSet or the Profiling page's own examples) so the
Glossary entry no longer depends on an external IG's URL stability. As part
of the same edit, sweep `profiling.html` for other US Core links and apply
the same treatment.

#### Justification

The original link broke because US Core renamed/retired
`ValueSet-us-core-problem`. Any direct replacement to a versioned US Core
URL will rot again the next time US Core restructures. A core-only or
landing-page reference removes that coupling and is more durable for an R4
page that will not see further normative work. It also matches FHIR
Infrastructure's general preference to keep core narrative loosely coupled
to specific IG artifact URLs.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Not Persuasive – Out of Scope" on the grounds that
FHIR R4 is a published, frozen normative release and the broken link is in
an external-IG reference whose lifecycle is owned by the US Core project.
Direct the reporter (and US Realm Steering) to raise the issue against
US Core to restore or redirect the legacy URL, and close FHIR-49656
without an R4 narrative edit.

#### Justification

The defect is link rot caused by US Core's own publication, not by the R4
core spec. Editing R4 narrative chases the symptom rather than the cause,
and any future US Core rename will reproduce the issue. Encouraging US
Core to maintain redirects for retired ValueSet pages addresses the root
cause for every consumer of those URLs, not just this Glossary entry.

---

### Recommendation

**Recommended disposition:** B

R4 narrative is still routinely patched for errata, so doing nothing
(Disposition C) leaves a confirmed 404 in published guidance and pushes the
fix onto a different work group with no guarantee of resolution.
Disposition A fixes the immediate link but re-creates the same fragility,
since US Core has already demonstrated it is willing to rename or retire
this ValueSet. Disposition B fixes the user-visible defect *and* removes
the cross-IG coupling that caused it, which is a better long-term fit for a
Glossary entry on a published, frozen spec. The edit is small, scoped to
`profiling.html` in `HL7/fhir`, and well within FHIR Infrastructure's
remit; the workgroup can still escalate to US Realm separately if it wants
the underlying US Core URLs restored.
