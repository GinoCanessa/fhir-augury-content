## Summary

The ticket reports that the FHIR Core specification's `fhirpath.html` page
inconsistently refers to FHIRPath *functions* as "operations." Because the
FHIRPath language specification uses the term **function** for these
constructs (e.g., `is()`, `getValue()`), using "operation" in the FHIR
narrative is misleading — it conflates FHIRPath functions with FHIR API
*operations* (`$something`-style endpoints). The reporter asks that prose
referring to FHIRPath functions as "operations" be reworded to "function,"
while leaving alone any genuine references to FHIR API operations
(e.g., terminology service operations invoked via a FHIRPath function).
The ticket has already been triaged with a resolution of **Persuasive**
and a resolution description of "Update the wording as requested." No
other Jira tickets are linked or cross-referenced; the only cross-reference
is a Zulip agenda post that bundles this ticket with a batch of other
small FHIRPath-page corrections proposed for a block vote.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the FHIR Core `fhirpath.html` narrative so that every prose
reference to a FHIRPath construct uses the word **function** rather than
**operation**. Concretely, walk every occurrence of "operation" /
"operations" on the page and reclassify each one:

- "the parameter to the `is()` operation" → "the parameter to the
  `is()` function"
- "The primary exception is the `is()` operation" → "The primary
  exception is the `is()` function"
- "all FHIR primitives have the operation `getValue()`" → "all FHIR
  primitives have the function `getValue()`"
- Apply the same substitution to any other in-prose mentions of FHIRPath
  built-ins (e.g., `as()`, `type()`, `extension()`, etc.) currently
  described as "operations."

Explicitly do **not** touch text where "operation" refers to a FHIR API
operation (`$expand`, `$validate-code`, terminology service operations
invoked from a FHIRPath function, etc.). Where the same paragraph mixes
both senses, keep "FHIR operation" / "terminology operation" wording for
the API side and switch only the FHIRPath-construct references.

#### Justification

This matches the resolution already recorded on the ticket
("Persuasive — Update the wording as requested") and aligns the FHIR
Core narrative with the FHIRPath specification's own vocabulary, which
consistently uses *function*. It also removes a real source of reader
confusion: in FHIR, "operation" is a loaded term that already denotes
RESTful `$`-prefixed endpoints, so using it for FHIRPath built-ins
overloads the word in the one place readers are most likely to be
cross-referencing the API. The change is purely editorial — no
schema, conformance, or behavioural impact — and the Zulip thread shows
explicit support (Grahame Grieve's proxy vote) and no opposition. The
FHIR-I co-chair has already proposed bundling this with similar
small FHIRPath-page corrections in a block vote, which is the natural
ballot path.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the wording change, but pair it with a one-sentence terminology
note near the top of `fhirpath.html` that explicitly distinguishes the
two senses of "operation" in the FHIR ecosystem — for example:

> In this page, *function* refers to a FHIRPath language function (such
> as `is()`, `as()`, or `getValue()`). The term *operation* is reserved
> for FHIR API operations (RESTful `$`-style endpoints, including those
> exposed by the terminology service).

Then perform the same find-and-replace as Disposition A.

#### Justification

The ticket itself draws attention to the fact that this conflation has
been hiding in the page for some time, and the reporter explicitly
warns the editor not to over-correct (don't rewrite genuine FHIR API
operation references). A short normative-style note up front prevents
future drift back toward the old wording, helps reviewers of unrelated
PRs spot the distinction, and makes the page self-documenting for
readers cross-referencing the FHIRPath spec. The cost is one extra
sentence; the benefit is a durable guard against the issue recurring.
Trade-off vs. Disposition A: this is slightly more than the reporter
asked for, but it stays within an editorial change and does not affect
conformance.

---

### Disposition C: Decline

#### Proposal

Close the ticket without making the change, on the grounds that "operation"
and "function" are colloquially interchangeable in implementation
discussions and that readers of `fhirpath.html` who reach the FHIRPath
sections are expected to consult the FHIRPath specification for
authoritative terminology.

#### Justification

A "decline" rationale would have to rest on the claim that the current
wording causes no real reader harm and that editorial churn on the FHIR
Core spec is itself a cost (additional ballot reconciliation, diff
noise, downstream re-publication). It would also lean on the fact that
nothing about implementer behaviour changes either way. This rationale
is weak in the present case because (a) the FHIR-I work group has
already triaged the ticket as **Persuasive**, (b) there is explicit
support and no opposition on the Zulip agenda thread, and (c) the
overload of "operation" with FHIR API operations is a genuine source of
ambiguity for readers — exactly the kind of small editorial defect
ballot block votes exist to clean up.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as **Persuasive** with the resolution
note "Update the wording as requested," the change is purely editorial,
the reporter has scoped the work precisely (and even called out the one
class of references that must *not* be changed), and the Zulip agenda
thread shows explicit pre-vote support (Grahame Grieve's proxy "for
making this change") with no dissent. Disposition A is the minimum
faithful execution of what FHIR-I has already triaged and is the natural
fit for the proposed block vote with the other FHIRPath-page
corrections (FHIR-55637, FHIR-55417, FHIR-55495, FHIR-37861, FHIR-54601,
FHIR-53958, FHIR-40271, FHIR-36040). Disposition B is a reasonable
enhancement if FHIR-I wants to harden the page against regression, but
it expands scope beyond the ticket and is better handled as a follow-up
once the wording sweep itself is in. Disposition C contradicts the
existing triage and the work-group sentiment on the Zulip thread.
