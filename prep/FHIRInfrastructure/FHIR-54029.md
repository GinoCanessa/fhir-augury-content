## Summary

The reporter argues that some values in the `OperationOutcome.issue.severity`
value set are ambiguous — specifically that the distinction between `fatal`
and `error` is not clearly defined in the specification. They also suggest
adding a constraint that ties `OperationOutcome.issue.severity` to
`OperationOutcome.issue.code` for the "success" case, so that a successful
outcome cannot be reported with a contradictory severity/code combination.

No related Jira tickets were surfaced via cross-reference, and no Zulip
threads or GitHub PRs/issues link to this ticket. Background community
chatter about `OperationOutcome.issue.severity` exists (mostly Death-on-FHIR
implementer threads using `fatal`/`error` interchangeably for invalid input),
but none of those threads explicitly debate the semantic distinction this
ticket asks to clarify.

## Details

**Description:**

> Certain values of severity of an OperationOutcome are ambiguous. In
> particular, it is not clear how a "fatal" severity differs from an "error"
> severity.

> In addition, it might be helpful to have a constraint on OO.severity ==
> OO.code on success

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Core specification's narrative for `OperationOutcome` so
that the `IssueSeverity` value set definitions explicitly distinguish
`fatal` from `error`. Concretely:

1. In the `OperationOutcome` resource page (and the `IssueSeverity` code
   system / value set definitions), tighten the definition of each code:
   - `fatal` — the issue caused the operation to fail and processing was
     aborted; subsequent issues in the same `OperationOutcome` may not
     have been evaluated.
   - `error` — a substantive problem was detected with the request or its
     processing, but the server was able to continue evaluating and may
     report additional issues.
   - `warning` / `information` — unchanged.
2. Add a normative invariant on `OperationOutcome` that constrains the
   `severity`/`code` pairing for a successful outcome — for example:
   `oop-1: When OperationOutcome.issue.code = 'informational' and the
   diagnostics indicate a successful operation, severity SHALL be
   'information'` (subject to redrafting once the workgroup picks the
   exact pairing it considers "success").

#### Justification

The current definitions of `fatal` and `error` in `IssueSeverity` are
narrative-only and read as near-synonyms, which is the source of the
implementer confusion the reporter is calling out. Tightening the
narrative is a non-breaking documentation change that helps both
implementers (who must choose a value) and consumers (who must branch on
it). Adding an invariant to forbid contradictory `severity`/`code`
combinations on success matches the general FHIR pattern of using
invariants to encode cross-element semantics that can't be expressed in
the type system.

---

### Disposition B: Alternative Approach

#### Proposal

Address only the documentation half of the request and decline the
invariant. Specifically:

1. Update the narrative definitions of `fatal` and `error` in
   `IssueSeverity` exactly as in Disposition A.
2. Add an implementer-facing usage note on the `OperationOutcome` page
   describing the recommended `severity`/`code` pairings for the common
   cases (success, validation failure, transient failure, fatal abort)
   without making them a SHALL-level invariant.
3. Defer any normative `severity == code on success` invariant to a
   future ticket once the workgroup has agreed on what "success" means
   in `OperationOutcome` terms (R5 introduced `success` in
   `IssueSeverity`; older versions did not, so an invariant has version
   sensitivity).

#### Justification

The clarity problem is real and worth fixing, but the proposed
invariant is under-specified in the ticket: `OperationOutcome.issue.code`
is drawn from `IssueType` (e.g., `invalid`, `structure`, `required`,
`informational`), which doesn't have a single "success" value, and the
correspondence to `severity` differs between R4/R4B (no `success`
severity) and R5 (which added `success`). Encoding a SHALL-level
invariant in the FHIR Core spec without working through that mapping
risks adding a constraint that breaks for valid existing payloads.
Splitting the work isolates the cheap-and-safe documentation fix from
the harder semantic question.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for a normative change. The existing
`IssueSeverity` definitions, combined with the `OperationOutcome` page's
existing description of how issues are reported, are sufficient guidance
for implementers; the ticket can be closed as "Not Persuasive" with a
note pointing to the current definitions and to `IssueType` for the
`code` axis.

#### Justification

`IssueSeverity` has been stable across multiple FHIR releases and is in
wide implementation use. The reporter does not cite a specific
interoperability failure caused by the ambiguity, only that the
definitions could be clearer. Tightening them risks invalidating
established server behavior where `fatal` and `error` have been used
somewhat interchangeably (as the Death-on-FHIR / implementers Zulip
traffic illustrates). The proposed invariant is also under-specified
and risks introducing a backwards-incompatible constraint.

---

### Recommendation

**Recommended disposition:** B

The ambiguity between `fatal` and `error` is a genuine documentation
gap and is worth closing with a narrative tightening — that part of the
ticket is low-risk and helps implementers. However, the proposed
`severity == code on success` invariant is under-specified in the
ticket: `OperationOutcome.issue.code` is drawn from `IssueType`, which
does not have a single "success" value, and the available severity
codes differ between R4/R4B and R5 (where `success` was added to
`IssueSeverity`). Adopting Disposition A wholesale risks shipping a
SHALL-level invariant that either contradicts existing valid payloads
or has to carve out version-specific exceptions. Disposition B captures
the safe half of the request now and asks the reporter to come back
with a concrete severity/code pairing if they want the invariant
pursued, which gives the workgroup a clean scope to act on without
prejudging the harder semantic question. Disposition C leaves a real
clarity problem on the books and should only be selected if the
workgroup judges the ambiguity to be already adequately addressed
elsewhere.
