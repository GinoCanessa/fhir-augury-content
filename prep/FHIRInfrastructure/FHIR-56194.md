## Summary

The ticket asks FHIR Infrastructure to bring the prose in `operations.html`
into line with the rule that already governs GET invocation of FHIR
operations. The "Executing an Operation" section currently says an
operation "MAY be invoked using GET" when `affectsState = false` and "the
parameters are all primitive datatypes with no extensions", and the
"Operation Request" section similarly refers to "primitive types". That
wording contradicts the immediately preceding bullet in the same page,
which already allows GET when "the only parameters being passed are ones
whose data type has a corresponding search type (i.e., datatypes like
Identifier or Reference which have a search type of 'token' are
permitted, while Address and HumanName are not)". Bryn Rhodes proposes
replacing the "primitive types" phrasing in both locations with
"types that have a corresponding search type" (or a construction like
"are all primitive types or have a corresponding search type"), and
additionally asks that the spec explicitly enumerate which FHIR
datatypes have a corresponding search type — ideally via a table — so
implementers do not have to reverse-engineer the mapping. No other
Jira tickets were cross-referenced.

## Details

**Description:**

The operations framework allows operations to be invoked with GET if:

> the only parameters being passed are ones whose data type has a
> corresponding search type (i.e., datatypes like Identifier or Reference
> which have a search type of 'token' are permitted, while Address and
> HumanName are not)

However, there are at least two places where the "primitive type"
characterization is used:

After the first example in the "Executing an Operation" section
(https://build.fhir.org/operations.html#executing):

> When an operation has affectsState = false, and the parameters are all
> primitive datatypes with no extensions (as is the case with the example
> above), it MAY be invoked using GET as well....

Suggest that be updated to:

> ... and the parameters are all types that have a corresponding search
> type...

Also here (https://build.fhir.org/operations.html#request):

> If all the parameters for the operation are primitive types and the
> operation has affectsState = false, the operation is also able to be
> invoked by performing an HTTP GET operation...

Suggest that this be clarified as well to:

> If all the parameters for the operation are primitive types (or have a
> corresponding search type...

Also suggest that we be explicit about what "has a corresponding search
type" means, is there a table somewhere that lists the corresponding
search type for each FHIR data type?

No comments on the ticket.

## Related Zulip Discussions

### [IG creation > targetProfile on string-valued operation parameters](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/targetProfile%20on%20string-valued%20operation%20parameters/near/582633316)

This is the thread that produced the ticket. Bryn Rhodes described an
IG that declares `targetProfile` on string-valued operation parameters
used as references; the IG Publisher renders it correctly but flags
`opd-3` ("`targetProfile` can only be specified for parameters of type
Reference or Canonical"). Lloyd McKenzie pushed back that `string` is
not a legal reference type and that only `Reference`/`canonical` (not
`uri`/`url`) should carry a `targetProfile`. Bryn explained the pattern
is used so the parameter can be passed in a URL during a GET
invocation — i.e., the string here is conceptually a search-style
token value. Gino Canessa reported that the CI-build spec already
relaxes the GET rule to any datatype "with a corresponding search
type" (citing the "Executing an Operation" section), and quoted the
updated bullet. Bryn noted two other places on the same page still use
the "primitive types" wording and filed FHIR-56194 to fix them, and
also asked whether the clarification should be back-ported to R4.
Grahame Grieve then pushed back harder: in his view you cannot pass a
string-valued reference in a GET using a search-style value because
that is a *search criterion*, not a value — search-typed parameters
only make sense for operations of type `query`; for actual references
the parameter type must be `Reference`. Gino later summarized FHIR-I
discussion: named-query operations use search semantics straight
through; for non-query operations, primitive types use their JSON/XML
string representation, `searchType` should not be used, comma
separation is unnecessary (use repetitions), some complex types have a
defined URL representation, and multiple parameters can be used for
things like coding (system/version/code). He also flagged an open gap:
there is currently no way to mark *some* parameters of a non-query
operation as search-style while leaving others as direct values.
Consensus on the narrow editorial request is cautiously positive;
consensus on whether the broader GET-semantics model is correct is
explicitly unresolved.

Participants: Bryn Rhodes, Lloyd McKenzie, Gino Canessa, Grahame
Grieve.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the FHIR Core `operations.html` page (CI build) in two places so
that the prose matches the already-normative bullet in "Executing an
Operation":

1. In the paragraph after the first example in the **Executing an
   Operation** section
   (<https://build.fhir.org/operations.html#executing>), replace:
   > When an operation has `affectsState = false`, and the parameters
   > are all primitive datatypes with no extensions (as is the case
   > with the example above), it MAY be invoked using GET as well...

   with wording equivalent to:
   > When an operation has `affectsState = false`, and every parameter's
   > datatype has a corresponding search type (including all primitive
   > datatypes, and complex types such as `Identifier`, `Reference`,
   > `Coding`, `CodeableConcept`, `Quantity`, `Period`, etc., which map
   > to search types like `token`, `quantity`, and `date`; `Address`
   > and `HumanName` do not qualify), and no parameter carries
   > extensions, the operation MAY be invoked using GET as well...

2. In the **Operation Request** section
   (<https://build.fhir.org/operations.html#request>), replace:
   > If all the parameters for the operation are primitive types and
   > the operation has `affectsState = false`, the operation is also
   > able to be invoked by performing an HTTP GET operation...

   with:
   > If the operation has `affectsState = false` and every parameter's
   > datatype has a corresponding search type (see the bullet in
   > Executing an Operation for the definition), the operation is
   > also able to be invoked by performing an HTTP GET operation...

3. Add an explicit "FHIR datatype → corresponding search type" table
   to `operations.html` (or link prominently to one in
   `search.html`), enumerating which primitive and complex datatypes
   have a search-type mapping and which do not. Cross-link both
   passages above to that table.

#### Justification

The ticket is an editorial consistency fix against text that already
exists in the same page. The bullet list in "Executing an Operation"
is the authoritative rule; the two paragraphs Bryn cites are older
descriptive prose that were not updated when the rule was relaxed to
"corresponding search type" (as Gino traced in the Zulip thread).
Aligning the three passages removes a direct internal contradiction
that is actively confusing IG authors — the ticket exists precisely
because a real IG author (Bryn) could not reconcile them. The
requested datatype table addresses a legitimate implementer need: the
"has a corresponding search type" phrase is currently only defined by
example (`Identifier`/`Reference` yes, `Address`/`HumanName` no), and
implementers should not have to reverse-engineer the full mapping
from `search.html`.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the editorial alignment in (1) and (2) above, but scope the
rewrite more narrowly and defer the broader question surfaced by
Grahame in the Zulip thread:

- Keep the phrase "types that have a corresponding search type", but
  add a sentence immediately after it in both passages saying:
  *"When a parameter's type has a search-type mapping, the GET-form
  value is the search-style expression of that value (e.g., a
  `token` system|code, a `quantity` value|unit). Operations that
  interpret these values as search criteria (rather than literal
  values) SHOULD use `searchType` and SHOULD be declared with
  `OperationDefinition.kind = query`."*
- Instead of inlining a new datatype/search-type table on
  `operations.html`, add a single cross-reference link to an existing
  normative section of `search.html` (or publish the table there and
  link to it). If no such table exists today, open a follow-on
  editorial ticket to add it in `search.html`, rather than in
  `operations.html`.
- Explicitly do **not** change the rule itself, and do **not** close
  the FHIR-I gap Gino flagged (mixed search-typed and
  direct-valued parameters within a single non-query operation) — log
  that as a separate tracker.

#### Justification

Disposition A treats the ticket as purely editorial, but the Zulip
thread makes clear that the "primitive vs. search-typed" distinction
is entangled with a semantic question: for non-query operations, is a
search-typed parameter passed as a *value* or as a *search criterion*?
Grahame's position is that the two should not be conflated.
Disposition B adopts the editorial fix so the page stops contradicting
itself, but adds one sentence that names the value-vs-criterion
distinction so implementers do not misread the relaxation as a
licence to pass search expressions into arbitrary non-query
operations. Keeping the datatype table in `search.html` (where the
search types themselves are defined) avoids duplicating mapping
information across two pages and reduces the risk of drift.
Deferring the mixed-parameter gap keeps the ticket bounded to what
Bryn actually asked for.

---

### Disposition C: Decline

#### Proposal

Do not amend `operations.html`. Close FHIR-56194 as "Not Persuasive —
Already Addressed" on the grounds that the bullet list at the top of
"Executing an Operation" already states the current rule in full, and
the subsequent paragraphs are illustrative prose describing the most
common case (all-primitive parameters). A reader who reaches the
bullet list has the correct rule; the paragraphs are not normative.

#### Justification

The bullet list is unambiguous and defines the condition precisely;
the paragraphs that use "primitive types" can be read as illustrative
examples rather than as restrictions. Rewriting them risks
introducing new normative language that has to be revisited alongside
the semantic question Grahame raised (whether search-typed GET
parameters are values or criteria), which in turn risks either
locking in a position FHIR-I has not fully resolved or producing
more confusion than the current state. A decline preserves the
current rule and lets FHIR-I tackle the larger value-vs-criterion
question on its own timeline.

---

### Recommendation

**Recommended disposition:** B

The ticket is a real bug in the spec — two paragraphs on the same page
directly contradict the authoritative bullet list, and a working group
co-chair (Bryn) could not reconcile them while authoring an IG. That
alone rules out Disposition C; "read the bullet list and ignore the
paragraphs" is not a defensible reader-experience answer, and closing
the ticket will leave implementers relying on Zulip archaeology to
understand the rule. Disposition A fixes the contradiction cleanly,
but the Zulip thread shows the group is not actually aligned on what
the relaxed rule *means* for non-query operations: Gino quoted the
current text approvingly, Grahame explicitly rejected the idea that
a string-valued search-typed parameter can carry a reference in a
GET, and FHIR-I has not yet decided how to represent operations with
mixed search-style and direct-valued parameters. Adopting A
wholesale — especially inlining a brand-new datatype/search-type
table in `operations.html` — would lock in a position on that
unresolved question by implication. Disposition B makes the minimal
editorial repair the ticket asks for, adds one sentence that names
the value-vs-criterion distinction so the relaxed wording is not
misread, and parks the deeper FHIR-I question on a separate tracker
where it can be discussed on its own merits. That is the safest
forward motion consistent with both what Bryn asked for and what the
Zulip conversation actually converged on.
