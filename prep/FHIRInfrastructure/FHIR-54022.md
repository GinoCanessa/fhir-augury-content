## Summary

This ticket asks the FHIR Infrastructure (FHIRI) work group to align all
FHIRI-owned content in FHIR R6 on the spelling **"payer"** rather than the
alternative **"payor"**. Financial Management (FM) previously made this
standardization on its own content, and the reporter would like the same
spelling convention applied uniformly across all R6 deliverables. This is one
of a coordinated set of work-group-scoped tickets driving the same
standardization; the companion ticket on the Patient Experience (PE) side,
[FHIR-54020](https://jira.hl7.org/browse/FHIR-54020), has already been
resolved Persuasive ("Agreed, will correct spelling of payer throughout") by
FHIR Mgmt Group / Grahame Grieve, establishing the precedent that this
ticket's disposition will most likely follow.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as an editorial / technical correction. Sweep all
FHIRI-owned R6 content for the string "payor" (case-insensitive,
whole-word) and replace with "payer", covering at minimum:

- Resource and DataType narrative pages, definitions, short labels, and
  element descriptions.
- Operation, search parameter, capability statement, and example
  narrative owned by FHIRI.
- ImplementationGuide, profile, extension, and value set / code system
  display strings owned by FHIRI (display-only — see Justification
  re: codes).
- Glossary, infrastructure / specification-internal pages, and
  introductory / overview prose.

Exclude (i.e., leave as-is):
- Code values and canonical URLs that contain "payor" — these are
  identifiers, not prose, and changing them is a breaking change. Only
  the human-readable display / definition text is updated.
- Quoted external normative text where "payor" appears verbatim.
- Content owned by other work groups (FM has already done its own;
  PE is being handled under [FHIR-54020](https://jira.hl7.org/browse/FHIR-54020)).

Open the change as a single coordinated PR (or small set of PRs grouped
by area) tagged with FHIR-54022 so reviewers can verify scope.

#### Justification

The reporter's request is purely editorial consistency — there is no
design trade-off to weigh, and the parallel PE ticket
[FHIR-54020](https://jira.hl7.org/browse/FHIR-54020) was already
resolved Persuasive with the explicit commitment "will correct
spelling of payer throughout." Accepting this ticket is the only
disposition that produces a consistent R6 spec; partial adoption would
leave FHIRI-owned pages diverging from FM's and PE's pages. Limiting
the sweep to display / prose (and explicitly carving out code values
and canonical URLs) preserves backwards compatibility for
implementers, which is the only real risk in an otherwise mechanical
change.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the standardization in principle, but route execution through a
**spec-wide editorial pass coordinated by FMG** rather than handling it
as a per-work-group ticket. Concretely:

1. Close FHIR-54022 with a pointer to a single tracking ticket that
   covers all remaining work groups (FHIRI, plus any others that have
   not yet adopted "payer").
2. Have the editorial / Mgmt Group own a single sweep across the R6
   build, with per-WG sign-off on the diff for their content.
3. Add a build-time or CI check (e.g., a lint rule in the publisher /
   IG-publisher tooling, or a repo-level grep check) that flags new
   occurrences of "payor" in prose so the standardization does not
   regress between R6 ballot cycles.

#### Justification

The substantive change is identical to Disposition A, but the
execution model is different. A single coordinated editorial pass
avoids the risk of inconsistent application across multiple
work-group-scoped tickets (FM done, PE done, FHIRI pending, others
unknown), and a lint check converts a one-time correction into a
durable convention. The trade-off is process overhead and a slower
turnaround vs. just applying the change directly under this ticket;
whether that overhead is justified depends on how many other WGs still
have "payor" in their prose. If the residual set is small, Disposition
A is cheaper.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** on the basis that (a) "payor" and "payer"
are both accepted English spellings in the U.S. healthcare-finance
vernacular, (b) FHIRI-owned content does not have substantial
payor/payer prose (the term is primarily an FM / financial-resources
concept), and (c) the editorial churn — touching narrative across many
files, re-reviewing for unintended changes, regenerating downstream
artifacts — is not justified by an out-of-WG style preference.

#### Justification

This is the weakest position given the precedent on
[FHIR-54020](https://jira.hl7.org/browse/FHIR-54020) (Persuasive, with
an explicit commitment to correct the spelling throughout). Declining
on FHIRI alone would actively re-introduce the inconsistency that the
overall standardization effort is trying to remove, and it would
contradict an FMG-level decision already on record. The only scenario
where this disposition is defensible is if FHIRI-owned content
genuinely contains zero or near-zero occurrences of "payor", in which
case the ticket can be closed as "no change required" rather than as
Not Persuasive.

---

### Recommendation

**Recommended disposition:** A

This is a low-risk technical correction with an established precedent.
FM has already done it, FMG agreed to do it across the spec via
[FHIR-54020](https://jira.hl7.org/browse/FHIR-54020), and the only
remaining question for FHIRI is whether to do the work under this
ticket or roll it into a coordinated editorial pass. Disposition A is
the cleanest answer: accept, scope the sweep to FHIRI-owned prose and
display strings, explicitly exclude code values / canonical URLs to
avoid any breaking impact on implementers, and land it as a single
PR. Disposition B is a reasonable upgrade if FMG is willing to own a
spec-wide pass and add a lint check, but it should not block accepting
the substantive change. Disposition C should not be entertained absent
strong evidence that FHIRI content has effectively no occurrences of
"payor" — and even then it should be closed as "no change required",
not Not Persuasive, to remain consistent with the FMG decision on
FHIR-54020.
