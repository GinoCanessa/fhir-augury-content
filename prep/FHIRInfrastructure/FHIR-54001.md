## Summary

The ticket asks the FHIR Infrastructure work group to clarify the wording
of constraint `bdl-16` on the `Bundle` resource. The current human-readable
description of `bdl-16` says: *"Issue.severity for all issues within the
OperationOutcome must be either 'information' or 'warning'."* The reporter
notes that this phrasing is ambiguous because `Bundle` does not contain an
`OperationOutcome` directly — the constraint actually scopes to the
`Bundle.issues` element (an inline `OperationOutcome`-shaped element added
to `Bundle` for transaction/batch responses). The request is editorial:
update the `bdl-16` description to make explicit that the rule applies to
issues carried in `Bundle.issues`, not to a general `OperationOutcome`.

There are no linked or cross-referenced Jira tickets — the only
cross-reference surfaced is the FHIR artifact `Bundle.issues` itself,
which is exactly the element the constraint targets.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the human-readable description of constraint `bdl-16` on the
`Bundle` resource so that it explicitly names `Bundle.issues` as the
scope. Concrete edit: change the current text from

> "Issue.severity for all issues within the OperationOutcome must be
> either 'information' or 'warning'."

to something like

> "Bundle.issues.issue.severity must be either 'information' or 'warning'
> for every issue carried in Bundle.issues (the inline OperationOutcome
> used in transaction/batch responses)."

The FHIRPath expression of `bdl-16` itself is not changed — only the
human description / `requirements` text on the `Bundle` StructureDefinition
is reworded. This is a purely editorial change with no behavioral or
schema impact.

#### Justification

The reporter is correct: `Bundle` does not contain a free-standing
`OperationOutcome`; it contains a `Bundle.issues` element typed as an
inline `OperationOutcome`. Naming the element on the resource the
constraint is attached to (rather than a different resource type) is
consistent with how other Bundle invariants (`bdl-1` … `bdl-15`) are
worded. The Zulip "Testing Invariants" example demonstrates that the
ambiguous wording does cause real confusion when readers / tooling parse
the constraint. As an editorial clarification on a `Triaged` / `Low`
ticket, this is the lowest-risk path and matches FHIR Infrastructure's
usual handling of wording-only tickets.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the editorial clarification (as in Disposition A) **and** also
align `bdl-16` with the broader pattern by tightening the FHIRPath to
explicitly anchor on `Bundle.issues`. For example, change the expression
to something like

```
issues.issue.all(severity = 'information' or severity = 'warning')
```

(or `Bundle.issues.issue.severity.all(...)`), and update the
`xpath`/`expression` and `human` text together. Optionally also clarify
in the `Bundle.issues` element definition itself that only
`information`/`warning` issues are permitted, with a pointer to `bdl-16`.

#### Justification

If the underlying intent is that this rule clearly belongs to
`Bundle.issues`, expressing that in the FHIRPath as well as the human
description prevents future drift between the two. It also makes the
constraint robust if someone later adds another `OperationOutcome`-shaped
element to `Bundle`. The trade-off is that touching the FHIRPath is no
longer purely editorial: it is a normative-text change that requires
re-running invariant tests and may surface differences between
publishing tooling. For a `Low`-priority clarification ticket this is
arguably more change than is needed, but it produces a cleaner end
state.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" / no change required, on the basis
that `Bundle.issues` is itself defined as an inline `OperationOutcome`,
so the existing wording — "all issues within the OperationOutcome" — is
technically accurate when read in the context of the `Bundle` resource.

#### Justification

A reader who already understands that `Bundle.issues` is typed as
`OperationOutcome` can reconcile the constraint as written. Declining
avoids spending normative-edit budget on a wording tweak. However, this
rationale is weak: the reporter is not the only reader who has hit this
ambiguity (the Zulip thread illustrates the same confusion), and FHIR
Infrastructure has historically accepted similar editorial sharpenings.

---

### Recommendation

**Recommended disposition:** A

This is a textbook editorial-clarification ticket: a single sentence in
the human description of one invariant on one resource, with a clearly
correct fix that makes the constraint match how every other Bundle
invariant is phrased (in terms of `Bundle.*` elements). Disposition A
fixes the actual confusion the reporter — and at least one Zulip
participant — ran into, with zero behavioral risk and no FHIRPath
churn. Disposition B is defensible but escalates a `Low`-priority
clarification into a normative-expression change that needs invariant
re-testing; the work group can layer that in later if drift between
human text and FHIRPath becomes a real problem. Disposition C should be
avoided: the ticket is small, correct, and low-cost, and declining it
preserves a known source of misreading in the spec.
