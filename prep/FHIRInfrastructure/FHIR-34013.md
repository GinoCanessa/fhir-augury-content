## Summary

The token introspection feature in SMART App Launch v2.0 is described as
being based on RFC 7662 (OAuth 2.0 Token Introspection). RFC 7662 §2.1
requires the `/introspect` endpoint to be protected by some form of
authentication (it states the endpoint **MUST** require some form of
authorization, e.g., client authentication or a separate bearer token,
to prevent token-scanning attacks). The SMART App Launch IG, however,
only states that EHRs **MAY** implement access control on the Token
Introspection endpoint and that, *if* access control is implemented,
authorized clients **SHOULD** be able to authenticate with their client
credentials.

The reporter — Dan Cinnamon — argues this leaves SMART in conflict with
the OAuth 2.0 specification it cites, and proposes tightening §6.3
("Authorization to Perform Token Introspection") so that authentication
is required (MUST) rather than optional (MAY). The issue surfaced during
Connectathon 28, when an RFC 7662–compliant OAuth 2.0 server expecting
authentication failed an Inferno-driven test that did not present any
credentials.

There are no formal Jira cross-references on this ticket, but the same
behavioural gap is being discussed in real-world testing today (see the
December 2025 Inferno Zulip thread under "Related Zulip Discussions"),
which shows the ambiguity is still actively confusing testers and
implementers four years after the ticket was filed.

## Details

**Description:**

> The token introspection feature of SMART v2.0 is stated to be based
> upon RFC7662 in the introductory paragraphs of the Token Introspection
> page.
>
> In RFC7662, it is stated in section 2.1
> (https://datatracker.ietf.org/doc/html/rfc7662#section-2.1) that the
> /introspection endpoint MUST have some sort of authentication to
> protect the endpoint against abuse.
>
> In section 6.3 of the SMART v2.0 introspection page however, it only
> states that authentication MAY be used to protect this same endpoint.
>
> The concern is that SMART v2.0 is now in conflict with the standard
> upon which it is based. It seems like section 6.3 should be updated to
> be MUST instead of MAY, to be in-line with RFC7662.
>
> I uncovered this during Connectathon 28, because i tested with an
> RFC7662 compliant OAuth2 server which was expecting authentication,
> and the test suite was not providing any- resulting in a failed test.

## Related Zulip Discussions

### [inferno > Inferno introspect test seems at odds w/ spec](https://chat.fhir.org/#narrow/stream/inferno/topic/Inferno%20introspect%20test%20seems%20at%20odds%20w%2F%20spec/near/562540038)

Rob Resendez opened the thread (Dec 8 2025) noting that the Inferno
introspection test sends only the `token=` parameter and no
authentication, which appears at odds with RFC 7662's MUST-authenticate
requirement, and quoted the SMART STU2 §"Authorization to Perform Token
Introspection" passage that softens this to MAY/SHOULD. Karl Naden
responded that the (g)(10) test kit Token Introspection tests (9.11 for
SMART 2.0, 9.20 for SMART 2.2) **do** allow testers to inject custom
headers via the "Custom HTTP Headers for Introspection Request"
(`custom_authorization_header`) input so that any server-specific access
requirements can be satisfied. Karl further explained that because the
OAuth and SMART specs do not prescribe a "standard" authentication
method for the introspection endpoint, Inferno deliberately leaves it
to the tester to supply whatever the server expects rather than
inferring credentials. Rob acknowledged the resolution; no normative
position was taken on whether SMART itself should be tightened, but the
exchange confirms the underspecification reported in FHIR-34013 still
catches implementers off guard.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update SMART App Launch §"Authorization to Perform Token Introspection"
(STU2 / current build) on the `token-introspection.html` page to require
authentication on the `/introspect` endpoint, aligning with RFC 7662
§2.1. Concretely:

- Change "SMART on FHIR EHRs **MAY** implement access control protecting
  the Token Introspection endpoint" to "SMART on FHIR EHRs **SHALL**
  protect the Token Introspection endpoint with some form of client
  authentication or access control, consistent with RFC 7662 §2.1."
- Strengthen the follow-on sentence so that any client authorized to
  call the introspection endpoint **SHALL** (not SHOULD) be able to
  authenticate using its registered SMART client credentials
  (`client_secret_basic` / `client_secret_post` for confidential
  symmetric clients, and `private_key_jwt` per the SMART backend
  services / asymmetric profile for confidential asymmetric clients).
- Add a non-normative note pointing implementers to RFC 6749 client
  authentication and RFC 6750 bearer-token usage as the two
  RFC-7662-sanctioned mechanisms, and explicitly call out that the
  `token` query/body parameter is **not** an authentication mechanism.
- Cite the new MUST in the page's security-considerations section and,
  if present, in the conformance/CapabilityStatement guidance for
  introspection.

This change would land in the SMART App Launch IG repository
(`HL7/smart-app-launch`) on the page sources backing
`token-introspection.html`.

#### Justification

This is the literal request and is well-grounded:

- RFC 7662 §2.1 is unambiguous that authentication is required; SMART's
  current MAY directly contradicts the spec it cites as its basis.
- The Inferno Zulip thread (Dec 2025) shows the ambiguity continues to
  confuse testers four years after the ticket was filed; tightening the
  text removes a recurring interoperability papercut.
- SMART already defines, in its core authentication sections, the
  client-authentication mechanisms a server would need to require here
  (`client_secret_basic`, `private_key_jwt`), so the normative tightening
  does not introduce new machinery — it simply requires use of mechanisms
  the IG already specifies.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the IG-level normative language as **SHALL protect, MAY choose
mechanism**, rather than mandating a specific authentication scheme:

- Change the §"Authorization to Perform Token Introspection" lead from
  "MAY implement access control" to "**SHALL** require authorization to
  call the Token Introspection endpoint, consistent with RFC 7662 §2.1.
  The specific mechanism (client authentication per RFC 6749, bearer
  token per RFC 6750, or an equivalent EHR-defined mechanism) is at the
  EHR's discretion and **SHALL** be advertised in the server's
  conformance / `.well-known/smart-configuration` documentation."
- Add a new key (e.g., `introspection_endpoint_auth_methods_supported`
  mirroring the OIDC discovery convention) to the SMART configuration
  document so that clients (and conformance tooling such as Inferno)
  can discover what the server expects rather than requiring
  out-of-band knowledge or per-tester header injection.
- Leave existing deployments room to use the bearer-token form
  (RFC 6750) if that is what they already do, since RFC 7662 explicitly
  allows that path.

#### Justification

This addresses the underlying defect — under-specification of
authentication on `/introspect` — without forcing a single auth scheme
onto deployments that have legitimately chosen bearer tokens or other
RFC 7662-conformant mechanisms. It also responds directly to the
Inferno discussion: Karl Naden's point was that there is no *standard*
way to express what an introspection endpoint requires, which is why
Inferno punts to a free-form custom-header input. Adding a discovery
metadata key fixes that root cause and lets conformance tools probe the
right scheme automatically.

Trade-offs vs. Disposition A:

- **+** Backward-compatible with existing RFC 7662–conformant
  deployments that don't use SMART client authentication.
- **+** Gives tooling (Inferno, the (g)(10) test kit) a deterministic
  discovery target, which is what's missing today.
- **−** Adds a new `.well-known/smart-configuration` field, which is a
  larger change than a simple MAY→MUST swap and would need IG ballot
  consideration.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep §"Authorization to Perform Token
Introspection" as currently written (MAY implement access control;
SHOULD authenticate with client credentials when access control is in
place). Optionally, add only a clarifying non-normative note that
"protection of the introspection endpoint per RFC 7662 §2.1 is the
responsibility of the OAuth 2.0 authorization server and is out of
scope of additional SMART-specific normative requirements."

#### Justification

A defensible decline rests on two points:

- RFC 7662's MUST already binds any conformant OAuth 2.0 authorization
  server. SMART builds on OAuth 2.0 and inherits that requirement
  transitively; restating it as a SMART-level MUST is arguably
  redundant.
- SMART historically scopes itself to the FHIR-facing additions over
  OAuth and tries to avoid re-normativising the underlying OAuth/OIDC
  specs; tightening this one clause invites the same scrutiny on every
  other OAuth requirement SMART references.

The Inferno discussion is also a counterweight: Karl Naden's resolution
shows that, in practice, conformance tooling can already accommodate
servers that require authentication via the custom-header input, so the
real-world interoperability harm is limited to documentation and test
ergonomics rather than a security gap.

---

### Recommendation

**Recommended disposition:** B

Disposition B fixes the actual interoperability problem the ticket and
the Inferno thread surface — clients/testers cannot reliably know what
authentication the introspection endpoint expects — without picking a
fight over which RFC 7662-allowed mechanism is "the" SMART one.
Disposition A is normatively cleanest but risks invalidating existing
RFC 7662-conformant deployments that authenticate via bearer tokens
rather than SMART client credentials, and it leaves the discoverability
gap (the root cause of the Inferno friction) unaddressed. Disposition C
is technically defensible on "OAuth already says MUST" grounds, but it
ignores four years of evidence that implementers and test-suite authors
read the SMART page and conclude authentication is optional. B asserts
the RFC 7662 requirement at the SMART layer, preserves implementer
choice of mechanism, and gives tooling a metadata hook to make the
behaviour deterministic — which is the combination most likely to close
the ticket without creating a new round of re-balloting churn.
