## Summary

Brian Postlethwaite is asking that the FHIR-specific FHIRPath specification
gain a new top-level section — at the same hierarchy level as the existing
"terminology service API" / "general service API" sections — that documents
the behaviour of FHIRPath functions that depend on **remote data**. A number
of FHIR-flavoured FHIRPath functions can reach out to external systems
(resolution of non-contained references, terminology operations, profile/
conformance lookup, the `%server` factory) and therefore are subject to
network-level concerns the spec is currently silent on: timeouts,
availability, caching, security, and error handling. The reporter also asks
the spec to make explicit what happens when the remote content is
unavailable — his expectation is that the function should return an empty
collection rather than throwing and aborting evaluation — and floats the
idea of a `traceOnRemoteError()` style hook (possibly more appropriate at
the FHIRPath base spec level than the FHIR usage layer).

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the augury index, so this review is based on the ticket text
alone.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level section to the **FHIR usage of FHIRPath** page in
FHIR Core, sibling to the existing terminology / general service API
sections, titled **"Handling Remote Data"**. The new section should:

1. Enumerate the FHIRPath functions whose evaluation can require an
   out-of-process call:
   - `resolve()` for non-contained references.
   - Terminology functions: `memberOf()`, `subsumes()`, `subsumedBy()`,
     `weight()`, and the `%terminology` factory.
   - Profile/conformance functions: `conformsTo()`, `slice()`,
     `elementDefinition()` (noting that these may be backed by a
     conformance server resolving `StructureDefinition`s).
   - The `%server` factory functions.
2. State normative expectations for each concern: timeout behaviour,
   availability/unreachability, caching policy (e.g., implementations MAY
   cache, with cache-invalidation guidance), security (auth propagation,
   trust boundaries), and performance trade-offs.
3. Specify the **unavailable-content rule**: when the engine cannot
   retrieve the remote artifact (network error, timeout, 404, auth
   failure, …), the function returns an empty collection rather than
   raising an exception that aborts evaluation. Document this as the
   default and, where applicable, allow engines to surface the failure
   via diagnostics without changing the value semantics.
4. Add a diagnostics hook — e.g., a FHIR-defined `traceOnRemoteError()`
   or a recommendation that engines route remote-call failures through
   the existing `trace()` channel — and flag that the FHIRPath base spec
   may be the more appropriate long-term home (file/coordinate with the
   FHIRPath WG accordingly).

#### Justification

The reporter is filling a real documentation gap: the FHIRPath usage
page already standardises terminology/service interactions but is silent
on the network semantics of `resolve()`, profile lookups, and the
`%server`/`%terminology` factories, which are the very functions most
likely to fail in deployed implementations. Specifying the
empty-collection-on-failure rule is consistent with FHIRPath's general
"propagate empty" philosophy and avoids forcing implementations to
choose incompatible error behaviours. Bundling these concerns into one
clearly named section (rather than scattering notes across each function
definition) makes the contract discoverable for both content authors
and engine implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than a single FHIR-only section, split the work along the
spec/engine boundary:

- File a companion item against the **FHIRPath base specification** to
  introduce the generic "remote data" framing — including the
  empty-on-failure rule and a `traceOnRemoteError()` (or equivalent
  diagnostic surfaces such as a structured `trace()` event) — so that
  any FHIRPath dialect benefits.
- In the FHIR Core "FHIR usage of FHIRPath" page, add only a short
  **"Remote Data"** subsection that (a) lists which FHIR-specific
  functions are remote-capable, (b) defers the generic semantics to the
  base spec, and (c) layers FHIR-specific guidance on caching policy,
  conformance-server resolution of `StructureDefinition`s, and security
  (e.g., honouring SMART scopes or server context).

#### Justification

The reporter himself notes the diagnostic hook may be "more appropriate
at the fhirpath level". Putting timeout/availability/error-handling
semantics in the base spec avoids three problems: (1) every FHIRPath
dialect re-inventing the same rules, (2) FHIR Core normatively defining
behaviour for `%terminology`/`%server` factories that the base spec also
governs, and (3) divergence between R5/R6 and future FHIR releases as
the rules evolve. The cost is coordination overhead with the FHIRPath
maintainers and a slower path to a normative answer; the FHIR Core
section can be merged independently as informative until the base-spec
work lands.

---

### Disposition C: Decline

#### Proposal

Decline as **out of scope** for the current ballot cycle / spec, on the
basis that remote-call semantics (timeouts, caching, retry, security)
are an **engine implementation concern** rather than a spec concern, and
that the empty-on-failure question is already implicitly answered by the
general FHIRPath rule that propagates empty collections from missing
inputs.

#### Justification

Defensible only if the WG concludes that the existing "missing input →
empty collection" semantics already cover the unavailable-content case,
and that timeout/caching/security choices are deployment-specific enough
that normative wording would either over-constrain implementations or
go stale quickly. This stance avoids new normative surface in a late
cycle and keeps the FHIRPath spec compact.

---

### Recommendation

**Recommended disposition:** B

The substantive content is needed — implementers today have no
authoritative answer to "what does `resolve()` do when the server is
down?" — so a flat decline (C) leaves a real interoperability gap.
However, accepting verbatim (A) risks codifying remote-data semantics in
FHIR Core that properly belong in the FHIRPath base specification, which
would create drift the moment the base spec catches up (especially for
`traceOnRemoteError()`-style diagnostics, which the reporter already
acknowledges may belong upstream). Disposition B captures everything A
delivers — a clearly-labelled "Remote Data" section enumerating the
remote-capable FHIR functions, the empty-collection contract, and
caching/security guidance — while routing the engine-level mechanics
(timeouts, error trace events) through the FHIRPath WG so the rules are
consistent across dialects. Implementers get the FHIR-specific list
immediately; the cross-cutting semantics get a durable home.
