## Summary

The reporter (Weiyu Zhang) questions the SubscriptionStatus guidance for
the `query-status` notification type. In the "Query Status" section
(2.13.8.4) of the SubscriptionStatus page, the table that lists which
elements are required vs. optional for each notification type currently
includes `SubscriptionStatus.notificationEvent` as a "Special" element
for `query-status`, with a note that "a `query-status` notification MAY
contain recent notifications." The ticket asks whether
`notificationEvent` should be listed at all for `query-status` â€” i.e.,
whether the `$status` operation response (which is what produces a
`query-status` SubscriptionStatus) should ever carry per-event
notification payloads, or whether that row should be removed from the
table to make clear that `notificationEvent` is reserved for the
delivery-side notification types (`event-notification`,
`heartbeat`-adjacent flows) and not for an on-demand status query.

There were no cross-referenced Jira tickets, Zulip threads, or GitHub
items returned for this issue; the only cross-references are mentions
of the SubscriptionStatus elements themselves
(`SubscriptionStatus.status`, `.type`, `.eventsSinceSubscriptionStart`,
`.notificationEvent`, `.topic`).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive with Modification â€” remove the `SubscriptionStatus.notificationEvent`
row from the Query Status table in section 2.13.8.4 of the
SubscriptionStatus page (`source/subscriptionstatus/subscriptionstatus.html`
narrative) and add an explicit prohibition: "A `query-status`
SubscriptionStatus SHALL NOT include `notificationEvent`." Optionally
strengthen this with an invariant on SubscriptionStatus along the lines
of `notificationEvent.empty() or type != 'query-status'` so that the
constraint is machine-checkable. The "Event Numbering" guidance and the
existing "this value SHALL NOT be incremented by sending a query-status
notification" note already imply that a `$status` response is a
state-snapshot, not a delivery vehicle for events; this change makes
that implication normative.

#### Justification

The `$status` operation is, by design, a synchronous client-initiated
query for the *current* state of a subscription. Conflating it with a
"send me recent events too" channel mixes two different interaction
patterns â€” pull-state vs. event-delivery â€” and makes server behaviour
ambiguous (which events would a server choose to include? would the
client be expected to ack them via `eventsSinceSubscriptionStart`?
already disallowed). Removing `notificationEvent` from the `query-status`
profile keeps the semantics of each notification type clean: delivery
types carry events, status types carry state. This is consistent with
the way other rows in the same table push back on cross-purposing
(e.g., the explicit "SHALL NOT be incremented" rule for
`eventsSinceSubscriptionStart`).

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification â€” keep the `notificationEvent` row but
tighten its wording so that its narrow, opt-in nature is unmistakable.
Replace the current note with: "A `query-status` notification MAY
contain `notificationEvent` entries only when the client has explicitly
requested event payloads (e.g., via a `$status`-side parameter or via
an `events` query parameter on the operation). When included, these
entries are informational only: they SHALL NOT advance
`eventsSinceSubscriptionStart`, SHALL NOT be considered delivered for
the purposes of guaranteed-delivery semantics, and the server SHALL
NOT rely on the client acknowledging them." Pair this with a
clarification on the `$status` operation page describing exactly when
event payloads are returned and how the client opts in.

#### Justification

There is a real implementer use case for "give me the current state and
also catch me up on the last N events in one round trip" â€” particularly
for clients that lost a websocket or REST-hook session and want to
recover without first fetching state and then separately pulling event
history. Removing `notificationEvent` outright (Disposition A) forecloses
that pattern. Tightening the spec text instead preserves the capability
but eliminates the ambiguity the reporter is reacting to: today the
spec only says "MAY contain recent notifications," with no guidance on
when, how the client signals interest, or how those events interact
with delivery accounting. A clearer opt-in framing matches how other
"special" elements in the same table are scoped (e.g.,
`eventsSinceSubscriptionStart` is "special" because its requiredness
depends on delivery semantics â€” a known, named condition).

---

### Disposition C: Decline

#### Proposal

Not Persuasive â€” leave the Query Status table unchanged.
`SubscriptionStatus.notificationEvent` remains a "Special" element for
`query-status` notifications, and the existing note ("A `query-status`
notification MAY contain recent notifications") stands as the normative
guidance.

#### Justification

The current text is already permissive, not prescriptive: `MAY` lets
servers choose whether to include events, and the "SHALL NOT be
incremented by sending a `query-status` notification" rule on
`eventsSinceSubscriptionStart` already prevents the most dangerous
side-effect (silently advancing the delivery counter). No implementer
report of harm has been attached to this ticket, and there is no Zulip
or GitHub discussion indicating the current text is causing
interoperability problems. Tightening or removing `notificationEvent`
mid-cycle would be a behaviour change for any server that already uses
it for state-plus-catchup responses.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance between the reporter's concern (the
current text is ambiguous and invites confusion about whether
`$status` is allowed to deliver events) and preserving a useful
implementation pattern (state-plus-recent-events in a single
round-trip). Disposition A is cleaner on paper but removes a capability
that some implementers likely depend on, and does so on the basis of a
single Triaged technical-correction ticket with no implementer
complaints attached. Disposition C ignores the legitimate critique:
"MAY contain recent notifications" with zero accompanying guidance on
opt-in, accounting, or acknowledgement semantics is genuinely
under-specified, and the ticket is correct to flag it. The workgroup
should treat this as a clarification opportunity rather than a deletion
opportunity, and tighten the wording to make the opt-in nature, the
informational status of the events, and the non-interaction with
delivery accounting explicit.
