## Summary

The reporter flags an editorial/grammar defect in the FHIRPath Patch
section of the FHIR Core specification. The closing normative sentence
currently reads *"Servers SHALL return an error if the outcome of the
patch operation is a not a valid resource"* — the phrase *"is a not a
valid resource"* contains a stray article ("a") that makes the sentence
ungrammatical. The intended wording is *"is not a valid resource"*.
This is a Technical Correction against FHIR Infrastructure and does not
change conformance behaviour; it is a textual fix to the published
normative requirement.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned for this ticket key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a persuasive-with-modification editorial fix to the FHIRPath
Patch page (`fhirpatch.html`, sourced from `HL7/fhir`'s
`source/fhirpatch/` authoring directory) by removing the stray article
in the closing normative sentence. Concretely, change the sentence at
the end of the FHIRPath Patch grammar/normative section from:

> Servers SHALL return an error if the outcome of the patch operation
> is **a** not a valid resource.

to:

> Servers SHALL return an error if the outcome of the patch operation
> is not a valid resource.

No conformance semantics change. The fix is text-only and does not
require touching schemas, examples, validators, or test cases. The
publication build of the page must be regenerated so the corrected
sentence appears in the next ballot/CI build.

#### Justification

This is a verbatim copy of the request and is unambiguous: the current
text contains a duplicated article ("is a not a") that makes the
sentence ungrammatical. Removing the stray "a" restores the obviously
intended reading and matches the symmetric phrasing used elsewhere in
the spec for similar SHALL statements ("…is not a valid…"). Because the
change is purely editorial, it carries the lowest possible risk to
implementers and to backwards compatibility.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this as an opportunity to rewrite the sentence for clarity in
addition to fixing the grammar. Replace it with something like:

> Servers SHALL return an error (e.g., an `OperationOutcome` with
> `issue.severity = error`) if applying the patch would produce a
> resource that does not conform to the resource's structural and
> invariant rules.

This makes the failure mode (`OperationOutcome`) explicit and clarifies
what "valid" means (structural + invariants), which matches the
guidance already given for other error-returning operations in the
spec.

#### Justification

Implementers periodically ask on `#implementers` what *kind* of error
is expected and what counts as "valid" (see the 2020 and 2023 Zulip
threads above). A small clarification at the same edit point removes a
recurring source of confusion at essentially zero cost. The downside
is that this expands the scope of a Technical Correction into a minor
editorial change, which some reviewers may prefer to handle as a
separate ticket so the grammar fix can be rolled in without further
debate.

---

### Disposition C: Not Persuasive

#### Proposal

Decline the change on the grounds that the existing sentence is
understandable in context and does not warrant a normative-text edit.

#### Justification

This disposition is hard to defend: the current text is objectively
ungrammatical and the proposed fix is a one-character deletion with no
semantic impact. The only argument for declining is "the meaning is
clear enough", which is rarely a reason to refuse an obvious editorial
correction in a normative SHALL sentence. Recording it here only for
completeness.

---

### Recommendation

**Recommended disposition:** A

Disposition A (Accept As Requested) is the right call. The defect is a
single stray article in a normative SHALL sentence, the fix is one
character, there are no conformance, schema, or example impacts, and
the change has no cross-repo or implementer-facing ripple effects.
Disposition B has merit on its own, but bundling a clarifying rewrite
into a Technical Correction risks re-opening discussion about *what*
"valid" means and stalls a fix that should be uncontroversial — better
to file a follow-up ticket for the clarification if the workgroup wants
it. Disposition C is included only to make the choice explicit; there
is no reasonable basis to decline a pure grammar correction.
