## Summary

The ticket asks whether FHIR Shorthand (FSH) Aliases should be allowed to
expand within other Alias definitions — i.e., when defining a new alias,
should references to previously-defined aliases (e.g., `$LOINC`) be
substituted into the new alias's value? Today, FSH treats alias values as
opaque URL/code tokens and does not perform recursive alias resolution
inside them. Keith Boone raised this against the FSH specification and
pointed to a parallel SUSHI issue (FHIR/sushi#356) where the same need
surfaced from an implementation perspective. There are no cross-referenced
Jira tickets for this item; the discussion has been carried in the
parallel SUSHI GitHub issue and (tangentially) in shorthand Zulip threads
about parser/grammar rules for where aliases may appear.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Shorthand specification to allow an alias name (e.g.,
`$BASE`) to appear inside another alias's *value* and to be expanded
during preprocessing, subject to Chris Moesel's prefix-only rule captured
in Mark Kramer's comment:

1. The alias reference **must be at the start** of the alias value being
   defined.
2. It **must be immediately followed by** `/`, `?`, or whitespace (i.e.,
   a URL path/query/segment boundary).
3. Expansion is performed once per alias definition during the same
   preprocessing pass that resolves alias usages elsewhere; cycles must
   be detected and reported as an error.
4. Add a normative example to the Aliases section of the Shorthand
   reference (e.g.,
   `Alias: $BASE = http://hl7.org/fhir/us/core` followed by
   `Alias: $USCorePatient = $BASE/StructureDefinition/us-core-patient`).
5. Add a corresponding implementation in SUSHI per FHIR/sushi#356.

#### Justification

The prefix + delimiter rule is the narrowest change that makes nested
aliases unambiguous: it preserves the existing "aliases only stand in for
URL tokens" invariant (consistent with the Zulip clarification from Chris
Moesel that aliases are not expanded inside arbitrary strings) and avoids
accidental substring matches in the middle of URLs or quoted text. It
also matches a common authoring pattern in real IGs, where many alias
URLs share a base canonical and currently must be repeated verbatim,
which is error-prone and harms readability.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of changing alias semantics, introduce an explicit "URL base"
mechanism — for example a top-level `Base:` keyword (or a reserved alias
name like `$BASE`) whose value is automatically prepended to relative
references in *other* alias definitions and FSH constructs. Aliases
themselves remain opaque, single-token substitutions; the
concatenation/composition need is met by a clearly-scoped, documented
construct rather than by allowing one alias to contain another. This
could also be deferred to RuleSet/parameterization mechanics, which
already provide controlled token substitution.

#### Justification

This addresses the underlying authoring pain (avoiding repetition of long
canonical URLs) without changing the long-standing rule that aliases
resolve to a single URL/code token. Keeping alias semantics opaque
preserves backward compatibility with all existing FSH files and avoids
the edge cases that motivated Chris Moesel's prefix-only constraint
(accidental substring matches, ordering of alias declarations, cycle
detection). It also separates two concerns — *naming* a URL vs.
*composing* URLs — which tends to age better as the spec evolves.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — Considered for Future Use" (restating the
prior 2020-06-22 vote of 14-0-0 by Kramer/Rhodes), with the rationale
that the existing alias mechanism, combined with RuleSets and
parameterized RuleSets, is sufficient for current authoring needs and
that the additional grammar/preprocessor complexity is not justified by
the volume of demand observed since STU1.

#### Justification

The change request has been open since 2020 with limited additional
community traction (no cross-referenced tickets, no substantive Zulip
thread devoted to nested aliases, and only one parallel SUSHI issue).
Even the narrow prefix-only proposal introduces new failure modes —
declaration order dependence, cycle detection, divergent behavior between
SUSHI and any other FSH tooling — that have to be specified, tested, and
maintained. Declining preserves the simplicity that has been a stated
strength of FSH and avoids opening a class of preprocessor semantics
questions that would otherwise need ongoing curation.

---

### Recommendation

**Recommended disposition:** A

The prefix-only rule already captured in the ticket's own comment is a
clean, minimal extension that respects the existing "alias = single URL
token" invariant called out in the shorthand Zulip discussion. The
authoring benefit (no more repeated canonical bases across an IG's alias
block) is concrete and matches the original requester's motivation, and
the constraint that the inner alias must appear at the start of the
value followed by `/`, `?`, or whitespace eliminates the substring-match
risk that motivated past hesitation. Disposition B is a reasonable
fallback if the workgroup decides the change should be expressed as a
new construct rather than as a tweak to alias semantics, and Disposition
C remains defensible only if the workgroup judges that demand has not
materialised — but the ticket's re-opening after STU1 publication and
the parallel SUSHI issue suggest demand has, in fact, persisted.
