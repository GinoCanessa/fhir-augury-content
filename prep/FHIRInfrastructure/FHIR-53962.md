## Summary

The reporter asks whether the FHIRPath `slice()` function — together with
the `%profile` variable and the `elementDefinition()` function — should
be restricted to (or at minimum documented as only meaningful inside)
invariant/constraint expressions on `ElementDefinition`, rather than
sitting in the general "Additional Functions" pile in the FHIRPath in
FHIR spec page. As a concrete remedy he suggests collating these
profile/invariant-aware features into a new sub-section, e.g.
"Additional Functions/Features for Invariants".

This ticket is one of a coordinated cluster Brian raised against the
FHIRPath in FHIR additional-functions docs:
[FHIR-53960](https://jira.hl7.org/browse/FHIR-53960) (inconsistent
error-handling patterns across the additional functions, with `slice`
and `elementDefinition` explicitly called out as "return empty on
error/invalid input"),
[FHIR-53961](https://jira.hl7.org/browse/FHIR-53961)
(`elementDefinition()` needs to be clearer about where it can be used
and what it returns — and notes the same `%profile` dependency that
implies invariant-only context), and (by topic, not cross-ref)
FHIR-53952 on `%profile` variable documentation. They should be
considered as a set: the implicit thesis across all four is that the
profile/invariant-aware capabilities of FHIRPath in FHIR are currently
under-scoped in their documentation.

Background context that the workgroup will want to bring to the
discussion: [FHIR-42983](https://jira.hl7.org/browse/FHIR-42983)
(Applied, Persuasive with Modification) was the previous round of
clarification on `slice()` and explicitly tied the function to the
`%profile` context variable, while
[FHIR-45455](https://jira.hl7.org/browse/FHIR-45455) (Applied,
Persuasive) added bookmarks/parameter clarity to the additional
functions section.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the editorial restructuring on the FHIRPath in FHIR page
(`fhirpath.html`). Specifically:

1. Introduce a new sub-section under "Additional Functions" titled
   **"Additional Functions and Variables for Invariants"** (or
   "…for Profile/Invariant Context").
2. Move into that sub-section the function definitions for `slice()`,
   `elementDefinition()`, and the documentation of the `%profile`
   variable.
3. At the head of the new sub-section, state explicitly that these
   functions/variables are only meaningful inside an
   `ElementDefinition.constraint.expression` (i.e., invariant
   definitions) because they require the `%profile` context, and that
   FHIRPath engines invoked outside that context are expected to treat
   them as returning empty (consistent with the error-handling pattern
   that FHIR-53960 catalogues).
4. Cross-link from the original locations of `slice()` /
   `elementDefinition()` / `%profile` so readers searching the prior
   layout still land in the right place.

This is a documentation-only change against the FHIR Core spec
authoring source (no normative API change).

#### Justification

The change crystallises what FHIR-42983 already established (that
`slice()` is `%profile`-scoped) and what FHIR-53961 confirms about
`elementDefinition()`. Grouping the three together matches how
implementers actually need to reason about them: as a coherent
profile-aware sub-language inside FHIRPath. It also lines up with the
editorial precedent set by FHIR-45455 (bookmarks/parameter clarity in
the same page) and FHIR-44774 (adding a context-oriented framing).
Resolving FHIR-53962 in tandem with FHIR-53961 (and
ideally FHIR-53960 + FHIR-53952) avoids three near-identical edits
landing piecemeal.

---

### Disposition B: Alternative Approach

#### Proposal

Do not move the function definitions, but **add a normative scoping
note in place** for each of `slice()`, `elementDefinition()`, and
`%profile`, in the form of a one-paragraph "Context" note attached to
each existing definition that says:

> This function/variable is only available when the FHIRPath
> expression is being evaluated in an `ElementDefinition.constraint`
> (i.e., an invariant). When evaluated in any other context (e.g.,
> general-purpose FHIRPath, search, mapping), it returns empty.

Additionally, add a bookmarked anchor list at the top of the
"Additional Functions" section labelled "Profile/Invariant-only
functions" that simply links to those existing definitions.

#### Justification

This addresses the underlying clarity problem (readers can't tell
which functions are profile-scoped) without re-flowing the page or
breaking inbound deep links and IG-Publisher anchors that already
point at the existing function names. It also avoids relitigating
*where* the spec authority for these functions lives — the previously
resolved FHIR-42983 text stays put. The trade-off vs. Disposition A
is that the related functions remain visually scattered and the
"these belong together" insight is weaker; but it is materially
lower risk for downstream tooling and external references.

---

### Disposition C: Decline

#### Proposal

Decline the request as primarily editorial preference and defer to
the in-place clarifications already being captured by the sibling
tickets. Recommend that FHIR-53961 (scope of `elementDefinition()`)
and a future ticket on `%profile` cover the "where can this be used?"
question per-function, without introducing a new section.

#### Justification

The substantive scoping question — that these features depend on
`%profile` and therefore on invariant context — is already being
addressed point-by-point. Restructuring the spec page risks breaking
external bookmarks (the Additional Functions anchors that FHIR-45455
explicitly added) and may pre-empt future additional functions that
also touch profile context but aren't strictly invariant-only (e.g.,
the `%factory` family, server-context functions in FHIR-53960's
catalogue). Declining keeps the section open-ended.

---

### Recommendation

**Recommended disposition:** A

The Zulip thread that seeded this ticket showed two careful FHIR
authors talking past each other on edge cases of `slice()`
specifically because the spec doesn't make the profile/invariant
context first-class — it's something readers have to reconstruct from
the `%profile` reference inside the function description. Brian has
filed three near-simultaneous tickets (53960/53961/53962) that all
point at the same gap from different angles; resolving them with a
single editorial restructuring is cheaper than three independent
in-place notes, and it gives the workgroup a natural home for the
`%factory` and server-context discussions that FHIR-53960 will force
soon anyway. The risk Disposition C raises about broken bookmarks is
real but mitigable by leaving HTML anchors at the original IDs and
inserting `<a id="…">` shims, which is standard practice in the FHIR
publication tooling.
