## Summary

The reporter requests that FHIR Shorthand (FSH) allow a `Quantity` value to be
written by supplying only the UCUM code, with the remaining UCUM-defined
elements (the canonical `system` URL and the human-readable `unit`/display)
auto-populated by the FSH compiler. Because UCUM formally defines both the
code and its display string, the ticket argues that requiring authors to type
all three is redundant and error-prone. The ticket is closely paired with
[FHIR-33559 (Autopopulate display)](https://jira.hl7.org/browse/FHIR-33559),
which makes the analogous request for `Coding`/`CodeableConcept` displays in
FSH; the reporter explicitly notes the two should be discussed together
because they share the same underlying constraint — whether the FSH compiler
should be allowed to depend on terminology knowledge (UCUM, SNOMED, etc.) at
compile time. Both tickets were originally voted "Considered for Future Use"
in 2021, then reopened and pulled from FSH Block Vote #1 by the reporter
before re-publication of the Normative Shorthand spec; they currently sit in
Triaged status awaiting fresh disposition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an FSH language affordance that lets an author write a `Quantity` (or
`SimpleQuantity`) by supplying only a UCUM code, and have SUSHI / the FSH
compiler auto-populate `Quantity.system` (fixed to
`http://unitsofmeasure.org`) and `Quantity.unit` (the UCUM-defined display
for that code). Concretely, extend the existing FSH `Quantity` literal /
assignment rule (currently of the form `<value> '<unit>'` with `system#code`
syntax for typed units) so that an author can write something like
`* valueQuantity = 5 '#mg'` or `* valueQuantity = 5 'mg' (UCUM)`, where the
leading `#` (or an analogous marker) instructs the compiler to look the code
up in UCUM and emit:

```
valueQuantity.value  = 5
valueQuantity.system = "http://unitsofmeasure.org"
valueQuantity.code   = "mg"
valueQuantity.unit   = "milligram"
```

Ship UCUM as a built-in dependency of SUSHI (UCUM is small, stable, and
already embedded in many FHIR tools), so that no terminology-server round
trip is required and the FSH spec can describe the lookup deterministically.
Document the new syntax in the Shorthand IG's "Assignment Rules" /
"Quantities" section and add a corresponding error ("Unknown UCUM unit
'XXX'") when the supplied code does not resolve.

#### Justification

UCUM is uniquely well-suited to this treatment among code systems: it is
algorithmically defined, redistributable, free, and already a hard dependency
of FHIR's `Quantity` validation. Embedding UCUM unit lookup in SUSHI does
not establish a general "FSH must talk to a terminology server" precedent
because it scopes the dependency to a single, bundled, deterministic code
system. The original 2021 rationale ("crosses into the territory of
validation, which is outside the scope of FHIR Shorthand") explicitly left
the door open ("we recognize the usefulness of this function ... possibly
incorporating other layers of the publishing process"), and the reporter's
follow-up comment correctly observes that this would meaningfully reduce
copy/paste errors in IGs that hand-author Quantity examples and slices.

---

### Disposition B: Alternative Approach

#### Proposal

Do not extend the FSH grammar. Instead, address the underlying need at the
publication-pipeline layer that the original WG response anticipated:

1. Add an opt-in IG Publisher / SUSHI post-processing step that, for any
   `Quantity` whose `system` is `http://unitsofmeasure.org` and whose `code`
   is set, fills in a missing `unit` from a bundled UCUM table (and
   conversely fills `system` if only `code` is set in a UCUM context). This
   step would emit an info-level message per substitution.
2. Provide an opt-in flag in `sushi-config.yaml` (e.g.
   `autoPopulateUcumDisplay: true`) so authors who want the convenience can
   enable it without changing the FSH file format, and authors who
   intentionally omit `unit` (e.g. for locale reasons) can leave it off.
3. Keep the FSH grammar unchanged; resolve FHIR-33559 in lockstep by either
   declining display auto-population for general code systems or scoping it
   to the same opt-in publishing step using only locally-cached
   `CodeSystem` resources.

#### Justification

This achieves the reporter's actual goal (less typing, fewer copy/paste
errors) without expanding the FSH language definition or re-opening the
broader "should the FSH compiler know about code systems?" question that
killed FHIR-33559 in 2021. Because the substitution happens after FSH has
been compiled to FHIR resources, FSH itself remains terminology-agnostic and
the Normative Shorthand grammar is undisturbed — an important consideration
given Shorthand is now a published normative artifact and grammar additions
have a high bar. It also generalises cleanly to FHIR-33559: a single
"display autofill" pass can handle both UCUM units and other locally-cached
code systems with consistent semantics, and IG authors who do not want
auto-population simply leave the flag off.

---

### Disposition C: Decline

#### Proposal

Persistent Considered-for-Future-Use / Not Persuasive. Do not change the FSH
language or tooling for this case; require IG authors to continue writing
the full `valueQuantity = 5 'mg' "milligram"` form (or the equivalent
explicit `system`/`code`/`unit` assignments). Recommend that authors who
find this verbose use FSH `RuleSets` / aliases or a local snippet library to
factor out commonly used Quantities.

#### Justification

The original 2021 vote (11-2-3) settled this on a clear principle: FSH is
deliberately a syntactic-sugar layer over FHIR resources and intentionally
avoids terminology-server-style behaviour, even for "easy" code systems.
Carving out a UCUM-only exception now would erode that principle and
predictably attract follow-on requests for SNOMED CT, LOINC, and others —
exactly the slope the WG declined to start down on FHIR-33559. The
`Quantity.unit` element is also explicitly defined as a *display* string
that "SHOULD be the same as ... but MAY be different from" the canonical
UCUM display, so silently populating it from UCUM may actively conflict
with author intent in localized or domain-specific contexts. Authors who
want brevity already have FSH `RuleSets`/aliases, which solve the
ergonomics complaint without changing language semantics.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the cleanest path forward because it threads the needle
between the two real constraints visible in the history. Disposition A
delivers exactly what the reporter asked for, but it has to defend a
language-level grammar change against a Normative spec and a closely
related ticket (FHIR-33559) that the WG previously declined for the same
reason; that is a tall order and likely to relitigate the 2021 vote.
Disposition C is internally consistent but ignores the WG's own caveat in
the original rationale ("we recognize the usefulness of this function ...
possibly incorporating other layers of the publishing process") and leaves
the reporter's concrete pain point (verbose, error-prone repetition of
UCUM display values) unaddressed. Disposition B captures the value the WG
already acknowledged, keeps FSH itself terminology-agnostic and Normative-
stable, and — critically — provides a single mechanism that can resolve
both FHIR-33560 and FHIR-33559 with consistent semantics. If the WG is
unwilling to add even an opt-in publisher step, fall back to Disposition C
and document FSH `RuleSets` as the supported workaround.
