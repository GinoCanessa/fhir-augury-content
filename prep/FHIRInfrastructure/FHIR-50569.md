## Summary

The reporter requests a typo correction on the FHIR R6 ballot 3
CapabilityStatement resource page. Specifically, the text uses the plural
form "CapabilityStatements" where it should refer to the resource by its
correct singular name, "CapabilityStatement". The pointer is to the
"Additional Capabilities" section of `capabilitystatement.html` in the
6.0.0-ballot3 publication. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the literal correction: in the "Additional Capabilities" section
of `capabilitystatement.html` (the page rendered at
`https://hl7.org/fhir/6.0.0-ballot3/capabilitystatement.html#additional`),
change the occurrence of "CapabilityStatements" to "CapabilityStatement"
so the prose refers to the resource by its canonical singular name.
Verify that the change is made in the authoring source (the resource
intro / narrative for `CapabilityStatement` in the FHIR core spec) and
not in a generated artifact, and re-publish so the change appears in the
next ballot snapshot.

#### Justification

The FHIR convention is that resource names are singular, capitalised
nouns (e.g., `Patient`, `Observation`, `CapabilityStatement`). Using the
plural "CapabilityStatements" in narrative prose that is referring to
the resource itself is inconsistent with that convention and with how
every other resource is named throughout the spec. As a Technical
Correction with no behavioural impact, this is exactly the class of
change that should be accepted as filed.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the ticket but broaden the scope: rather than
fixing only the single occurrence in the "Additional Capabilities"
section of `capabilitystatement.html`, run a spec-wide audit of the
authoring sources for accidental plurals of resource names
("CapabilityStatements", "Observations", "Patients", etc.) used in
contexts where the singular resource name is meant. Fix all instances
under this ticket (or spawn a tracking ticket if the audit surfaces
many), so the typo is corrected uniformly rather than spot-treated.

#### Justification

Single-instance typo fixes tend to recur when the underlying habit
(pluralising resource names in prose) is not addressed. A broader sweep
costs little extra effort once an editor is already in the file, keeps
the spec's terminology consistent, and avoids a stream of near-identical
tickets in future ballot cycles. The trade-off vs. Disposition A is
slightly more reviewer effort and a marginally larger diff, but no
behavioural risk because the change is purely editorial.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" on the grounds that
"CapabilityStatements" is a grammatically valid plural noun in English
prose (e.g., "servers may publish multiple CapabilityStatements") and
without a quoted snippet of the offending sentence the requested
correction may change correct text into incorrect text.

#### Justification

The FHIR style is to use the singular resource name when *naming the
resource type*, but the plural form is routinely used when *referring
to multiple instances of the resource* in narrative. Without confirming
which sense the affected sentence uses, blindly singularising could
introduce a grammatical error. Declining (or at minimum requesting more
detail before acting) would be defensible if the editor cannot identify
the intended fix from the section reference alone.

---

### Recommendation

**Recommended disposition:** A

This is a Medium-priority Technical Correction filed against a specific
section of a single page, with a clear before/after intent: the resource
is named `CapabilityStatement`, and the page refers to it by name. The
fix is editorial, has no normative or implementer impact, and is the
kind of low-cost cleanup the workgroup should land without ceremony.
Disposition B is reasonable as a follow-up but should not block this
ticket; Disposition C reads the request too narrowly given that the
reporter has pointed at the exact section and the fix is obvious once
the page is opened.
