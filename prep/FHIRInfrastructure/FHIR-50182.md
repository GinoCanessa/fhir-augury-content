## Summary

The reporter followed the **Support** link in §2.1.16 of the *FHIR Overview – Developer* page, which sends users to the HL7 Confluence "Implementer Support" page (`https://confluence.hl7.org/display/FHIR/Implementer+Support`). On that Confluence page, under "Asking Questions", the fourth bullet ("[FHIR Change Requests]: If you want to submit a change request, please read this page") still links to the legacy MediaWiki page `http://wiki.hl7.org/index.php?title=FHIR_Change_requests`. That wiki page now only contains a stub message stating that the wiki has migrated to Confluence, with a generic link back to the Confluence root — there is no longer a usable destination explaining how to submit a FHIR change request. The reporter is asking the workgroup to (a) decide what the link should point to and (b) update the Confluence page so the "Implementer Support" → "FHIR Change Requests" path lands on a live, accurate page about how to file a tracker item.

No related Jira tickets, Zulip discussions, or GitHub items were surfaced via cross-references or search; the issue is scoped to a documentation/link-rot fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a two-part documentation fix, owned jointly by FHIR-I and the Confluence page maintainer:

1. **Confluence – `Implementer Support` page, "Asking Questions" section, 4th bullet ("FHIR Change Requests"):** replace the dead `http://wiki.hl7.org/index.php?title=FHIR_Change_requests` link with a live target. The intended destination is the standing HL7 guidance on how to file FHIR change requests — i.e., a Confluence page (e.g., a "FHIR Change Requests" / "Submitting a Tracker Item" page under the FHIR space) plus the canonical Jira project URL `https://jira.hl7.org/projects/FHIR`. The bullet should read approximately: *"FHIR Change Requests: to propose a change to a FHIR specification, file a tracker item in the [HL7 FHIR Jira project](https://jira.hl7.org/projects/FHIR). See the [Submitting a Change Request](<confluence-page-URL>) page for guidance."*
2. **FHIR spec – `overview-dev` page, §2.1.16 ("Support"):** verify the "Support" link still resolves to the corrected Confluence page (it currently does), and add an inline reference to the Jira project URL so the spec page is no longer solely dependent on the Confluence redirect being correct. Edit performed in the `HL7/fhir` source page `overview-dev` (or its included fragment).
3. **Sweep:** as part of the fix, search the `HL7/fhir` source for any other occurrences of `wiki.hl7.org/index.php?title=FHIR_` and replace them with the equivalent Confluence/Jira URLs (this addresses the same link-rot class).

#### Justification

This is a *Technical Correction* against documentation, not a normative change — the request is a straightforward link-rot fix and is squarely in the maintainer's lane. Pointing users directly at the Jira project for change requests matches current HL7 practice (the legacy wiki has been retired and the Jira tracker is the authoritative entry point). Adding the Jira URL into the spec page itself makes the spec self-sufficient and reduces future exposure to Confluence reorganizations. The accompanying sweep is cheap and prevents the same complaint from being filed again against other chapters.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of patching only the broken bullet, restructure §2.1.16 of `overview-dev` so the spec links *primarily* to authoritative endpoints owned by HL7's process (Jira, hl7.org "How to participate" pages) and uses Confluence only as a secondary "background reading" link. Concretely:

- Replace the single "Support" link with a small bulleted list in §2.1.16: (a) **Ask a question** → Zulip `chat.fhir.org` `#implementers`; (b) **Report a problem / propose a change** → `https://jira.hl7.org/projects/FHIR`; (c) **Background / community process** → the Confluence Implementer Support page.
- File a follow-up ticket against the Confluence page maintainer to clean up the "FHIR Change Requests" bullet (i.e., still do the Confluence edit in Disposition A, but treat it as a separate work item so the spec change is not blocked on Confluence access).
- Add the same three-bullet block to any other "Support"/"Help" sections in `HL7/fhir` (e.g., the home page footer, the corresponding section in `overview-clinical` / `overview-arch` if present).

#### Justification

The root cause of this ticket is that the spec routes implementers to Confluence for an action (filing a change request) that has its own canonical system (Jira). Every additional hop is a chance for link rot; the reporter's "adventure" is the predictable result. This disposition costs slightly more editorial work but removes the failure mode entirely for the "report a change" path and aligns the spec with how implementers actually engage today (Zulip + Jira). It also decouples the spec fix from a Confluence edit cycle the workgroup may not directly control.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive – out of scope for the FHIR specification**. The broken link is on an HL7 *Confluence* page (`Implementer Support`), not in the FHIR specification itself. The §2.1.16 "Support" link in `overview-dev` correctly resolves to that Confluence page; the broken bullet ("FHIR Change Requests" → legacy wiki) is content owned by the Confluence page maintainers (HL7 community/process), not by the FHIR Infrastructure work group's normative content. Refer the reporter to the HL7 Confluence administrators / page owner to fix the broken bullet, and take no action against the FHIR spec.

#### Justification

A strict reading of scope limits FHIR-I's editorial responsibility to artifacts shipped as part of the FHIR specification (under `HL7/fhir`). Confluence pages, while linked from the spec, are maintained outside the spec's release cycle and have their own owners. Declining keeps the change boundary clean and avoids FHIR-I taking on long-term responsibility for content drift on Confluence. (This is the weakest option but is included for completeness — the spec itself does send users into the broken chain, so there is at least some FHIR-I responsibility in practice.)

---

### Recommendation

**Recommended disposition:** A

The reported defect is real, easy to reproduce, low-risk to fix, and squarely within the spirit of a *Technical Correction*: a user followed the spec's documented support path and hit a dead end. Disposition A is the smallest, fastest action that restores the broken path and adds a short defensive sweep for the same link-rot pattern. Disposition B is attractive longer-term and the workgroup may want to file a follow-up ticket to do the §2.1.16 restructure described there, but it should not block the immediate Confluence/spec link fix. Disposition C is not recommended: although the broken bullet is technically on Confluence, the FHIR spec page is what sends implementers down that path, so FHIR-I should at minimum coordinate the Confluence edit and update the spec to reference Jira directly so the failure mode cannot recur.
