## Summary

The reporter is requesting clarification — not a defect fix — about how
`SubscriptionTopic.trigger.notificationShape.relatedQuery` (and the
mirroring element on `SubscriptionStatus`) is intended to be used. Three
specific questions are raised: (1) whether related queries are
server-defined and fixed (since `Subscription` itself has no
`relatedQuery` analogue, subscribers cannot opt in or pick from a set);
(2) whether a standard token/placeholder vocabulary exists for
referencing the focal/notification resource(s) inside the query string,
analogous to CDS Hooks' `{context.patientId}`; and (3) whether there is
an example or recommended binding for `relatedQuery.queryType`, and what
shape of code is expected there.

The ticket is open against FHIR Infrastructure, classified as a
Question, currently Triaged with no assignee, no comments, and no
explicitly linked Jira tickets. The only cross-reference recorded by
Augury is to the FHIR artifact path itself
(`SubscriptionTopic.trigger.notificationShape.relatedQuery`), confirming
the request is purely about the existing core element rather than any
downstream IG.

## Details

**Description:**

> The SubscriptionTopic resource defines a relatedQuery element and has
> associated narrative in section 2.14.6.8.3 (and is similarly defined
> in SubscriptionStatus), but I'm unclear on how this element should be
> used:
>
> 1) RelatedQuery is defined on SubscriptionTopic and SubscriptionStatus,
> but not Subscription. Are RelatedQueries hardcoded by the server, and
> not something that each subscriber can ask for/pick between a set of
> options?
>
> 2) Are there standard tokens to use in the query itself, for referring
> to the focal resource? (e.g. requesting all conditions for a patient
> notification via `Condition?patient={focusResourceFHIRID}`, like
> `{context.patientId}` in CDS Hooks)
>
> 3) Is there an example binding for relatedQuery.queryType? What type of
> codings are expected there?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a documentation-only change against the
`SubscriptionTopic` (and, where parallel, `SubscriptionStatus`) narrative
in FHIR Core, and answer all three questions in the spec:

1. Add a clarifying paragraph in `SubscriptionTopic.trigger.notificationShape.relatedQuery`
   stating explicitly that related queries are **defined by the topic
   author / executed by the server**, are **not** negotiated per-
   `Subscription`, and explaining *why* `Subscription` has no
   `relatedQuery` element (the contract is fixed by the topic so that
   notification shape is deterministic across subscribers).
2. Define and publish an explicit, normative list of substitution
   tokens that may appear inside `relatedQuery.query` (for example
   `%focus`, `%previous`, `%resource.id`, or an explicit
   `{focusResourceFHIRID}`-style token), modeled on the CDS Hooks
   `context.*` pattern. Specify escaping/quoting rules and which tokens
   are valid in which `notificationShape` (`current` vs `previous`).
   Provide at least one worked example such as
   `Condition?patient={focus.id}`.
3. Add an example binding (preferred binding strength `example`) for
   `relatedQuery.queryType` pointing at a value set that enumerates the
   shapes the WG actually intends (e.g., `query-by-reference`,
   `query-by-search-params`, `graphql`, `fhirpath`), with at least one
   non-normative example invocation per code in the SubscriptionTopic
   examples folder.

#### Justification

The reporter's questions are legitimate ambiguities: the element exists
and is normatively present, but the spec gives implementers no answer
to "who chooses these queries?", "how do I reference the focal
resource?", and "what goes in `queryType`?". Closing those gaps is a
straightforward narrative + binding change, fully within FHIR-I's
authority over `SubscriptionTopic`. Defining the token vocabulary now
(rather than waiting for ad-hoc IG conventions) prevents the same
divergence the community already saw with CDS Hooks before its context
substitution rules were tightened. Specifying an example binding for
`queryType` follows the existing FHIR Core pattern for "we don't want
to constrain but we want to show implementers what we mean" coded
elements.

---

### Disposition B: Alternative Approach

#### Proposal

Answer questions (1) and (3) in the SubscriptionTopic narrative as in
Disposition A, but **defer** the substitution-token work (question 2)
to a separate, properly scoped tracker:

1. Add narrative confirming that `relatedQuery` is server- /
   topic-defined and explaining the design rationale for not surfacing
   it on `Subscription`.
3. Add an `example` binding to `relatedQuery.queryType` pointing at a
   small illustrative value set, with the explicit caveat that
   conformance is not required.
2. (Deferred.) Open a follow-up ticket — flagged for the FHIR
   Subscriptions sub-stream and connectathon — to design a
   substitution-token grammar properly (placeholder syntax,
   escaping, behavior when the focus is a bundle, interaction with
   `notificationShape.current` vs `previous`, security considerations
   such as injection into search parameters). Reference that ticket
   from `relatedQuery` so implementers know it is in flight.

#### Justification

Questions (1) and (3) are documentation gaps with a clear answer that
FHIR-I can settle in one pass. Question (2), in contrast, is *de facto*
a small specification design effort: any token grammar adopted in core
becomes part of every server's parse path and must reconcile with the
existing `_filter`, search-parameter, and FHIRPath surfaces, plus
GraphQL if that is ever a permitted `queryType`. Bundling a design
deliverable into a "Question"-typed ticket risks rushing the syntax;
splitting it lets FHIR-I close the easy clarifications quickly while
giving the token-substitution work the discussion it actually needs.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Answered – No Spec Change** with an inline
comment that:

- Confirms (Q1) `relatedQuery` is intentionally server- and topic-
  defined; subscribers do not negotiate it, by design.
- States (Q2) that no normative substitution-token syntax is defined
  and that implementations should treat `relatedQuery.query` as an
  opaque string the topic author has pre-bound to its focus, with any
  templating handled out-of-band by the server.
- Notes (Q3) that `queryType` deliberately has no binding and is open
  to implementer choice.

Make no changes to `SubscriptionTopic` or `SubscriptionStatus` content.

#### Justification

This is technically defensible: nothing in the spec is *wrong*, the
element is usable as-is in profiles and IGs, and the WG's historical
position on Subscription has been to keep core flexible and let
specific use-cases (Da Vinci PAS, etc.) constrain in IGs. Declining
keeps the core surface area unchanged and avoids locking in a token
grammar that future profiles may want to override.

---

### Recommendation

**Recommended disposition:** B

Disposition C is too thin: the reporter is asking exactly the questions
a careful implementer asks, and the fact that they had to file a
ticket — with no answer in the narrative — is itself the bug. Leaving
`queryType` unbound and `relatedQuery` ungrounded in subscriber-vs-
server semantics will simply produce more of the same tickets, plus
divergent IG-level interpretations.

Disposition A is the right *shape* of fix, but tries to do too much at
once. The substitution-token grammar (Q2) is not a clarification; it
is new normative syntax with security and FHIRPath/search-param
implications, and it deserves its own ticket, sub-stream discussion,
and ideally connectathon validation before landing in core.

Disposition B captures the easy, defensible wins now — narrative on
who owns related queries (Q1), example binding on `queryType` (Q3) —
and routes the genuinely design-loaded question (Q2) to a forum where
it can be designed instead of guessed at. That gives the reporter
concrete, citable answers in the next publication while protecting the
WG's ability to design the substitution syntax deliberately.
