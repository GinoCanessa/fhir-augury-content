## Summary

Hayden Bader reports two defects in the "Common uses of the Timing Datatype
criteria" example table on the Timing datatype documentation page. First, the
`endOffset` element (recently added alongside `startOffset`) has no example
row in the table, even though, per the reporter, it is "weird enough" that an
example is essential to make it understandable to implementers. Second, the
existing "Every second day" example is incorrect: the row sets
`frequency = 1`, `period = 1`, `periodUnit = d`, which actually expresses
"every day" rather than "every second day", and the companion row "Every
second day starting on Day 2" inherits the same error before adding
`startOffset = 1 d`. The request is purely documentation-level — fix the two
faulty rows so they actually express "every second day" (i.e., `period = 2 d`)
and add at least one new row exemplifying `endOffset`. This complements
[FHIR-54062](https://jira.hl7.org/browse/FHIR-54062) ("Rework
startOffset/endOffset constraint"), filed the same day by the same reporter,
which has already been resolved *Persuasive with Modification* to add a new
"Periods with start or end offsets" notes section, require `period` whenever
either offset is present, and (per its resolution) clarify that the offsets
shrink the effective period over which `frequency` applies. The two tickets
should be coordinated: any new `endOffset` example row in this ticket must be
consistent with the constraint and prose changes landing under FHIR-54062.

### [FHIR-54062: Rework startOffset/endOffset constraint](https://jira.hl7.org/browse/FHIR-54062)

Filed by HaydenBader the same day as FHIR-54058. Argues that the existing
`tim-11`/`tim-12` constraints (`startOffset.exists() implies (frequency > 1)`
and the equivalent for `endOffset`) are under-specified: `period` is not
required, yet the offsets only make sense relative to a `period`. Already
**Resolved – Persuasive with Modification** with this disposition: tighten
both invariants to also require `period`; add a notes subsection "Periods
with start or end offsets" pointed to by both elements' comments explaining
that offsets shrink the effective period over which `frequency` is counted
(e.g., `period = 28 d`, `endOffset = 7 d`, `frequency = 21` ⇒ once per day
across the 21-day on-window); and fix a typo in the short description
("step" → "stop"). Any new `endOffset` example added under FHIR-54058 must be
consistent with these tightened constraints — i.e., must include a `period`
and a `frequency > 1`, and the prose around the example should reuse the
"effective period" phrasing introduced by FHIR-54062.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Make a documentation-only change to `source/datatypes.html` in `HL7/fhir`,
covering both defects raised in the ticket:

1. **Fix the "every second day" rows.** In the example table at
   lines 1673–1674:
   - Line 1673 ("Every second day"): change `period = 1, periodUnit = d`
     to `period = 2, periodUnit = d` so the row actually expresses
     once per two-day window.
   - Line 1674 ("Every second day starting on Day 2"): change `period = 1,
     periodUnit = d` to `period = 2, periodUnit = d`. Leave the
     `startOffset = 1 d` cell as-is so the row still illustrates the
     intended phase shift onto Day 2.
2. **Add an `endOffset` column.** In the header row at line 1666, insert
   `<td><b>endOffset</b></td>` immediately after the `<b>startOffset</b>`
   cell. In each existing data row (lines 1667–1685, plus the changed
   1673–1674), insert a corresponding empty `<td></td>` cell in the same
   position to preserve table alignment.
3. **Add at least one new row exemplifying `endOffset`.** Use the same
   "21 days on, 7 days off, over a 28-day cycle" pattern that
   FHIR-54062's resolution canonicalises, e.g.:
   `description = "21 days on, 7 days off, repeating in a 28-day cycle"`,
   `frequency = 21`, `period = 28`, `periodUnit = d`, `endOffset = 7 d`.
   Optionally also add a mirror row exercising `startOffset` with the
   same `period`/`frequency` shape so the two offsets are illustrated
   symmetrically.
4. Verify cell counts across all rows after the edit (the table is wide
   and easy to leave ragged).
5. Coordinate the merge with the FHIR-54062 changes so the new row, the
   tightened `tim-11`/`tim-12` invariants (now also requiring `period`),
   and the new "Periods with start or end offsets" notes section all land
   in the same publish.

#### Justification

The ticket is a low-risk, narrowly-scoped documentation fix that addresses
two clearly demonstrated defects on a publicly-rendered spec page. The
"Every second day" example is, by inspection, wrong — `frequency = 1` over
a `period = 1 d` window is once per day, not once per two days. Leaving it
in place misleads implementers and undermines the table's stated purpose
("common uses of the Timing Datatype criteria"). Adding an `endOffset`
column and example is consistent with what the surrounding paragraph
already promises the reader (lines 1687–1692 explicitly call out that
"there are two offsets here … `startOffset` and `endOffset`"), so the
table is currently inconsistent with its own narrative. Anchoring the new
row on the 21-of-28-days example reuses the worked case already provided
in the FHIR-54062 resolution, which keeps the two tickets aligned and
gives the reader the same numbers in two places.

---

### Disposition B: Alternative Approach

#### Proposal

Same fix to the buggy "Every second day" rows as in Disposition A
(Item 1), **but** instead of widening the existing table with an
`endOffset` column, add a small **second** example table immediately
below the existing one specifically for offset-aware schedules. The
second table would have a narrower set of columns
(`description, frequency, period, periodUnit, startOffset, endOffset,
bounds[x], count`) and contain two or three worked examples: the
21-on/7-off cycle from FHIR-54062, an analogous `startOffset` example,
and (if useful) one combining both. Cross-link from the first table's
explanatory paragraph (lines 1687–1692) to this new sub-table.

#### Justification

The existing "common uses" table is already 14–15 columns wide and most
rows leave most cells blank; widening it further degrades readability for
the more common, simpler examples (which dominate the table) in service
of two specialist columns that are mostly empty. A focused second table
keeps the offset-aware examples legible side-by-side, gives room for
prose around them, and matches the structure used elsewhere in the spec
where a complex element gets its own example block. The trade-off is one
extra page section for the editor to maintain and a small loss of the
"single canonical table" property; balanced against the readability win
on a page that publishes to the public spec, the trade is reasonable.

---

### Disposition C: Decline

#### Proposal

Decline the request to add an `endOffset` example column to the table,
on the grounds that `startOffset`/`endOffset` are already worked
extensively on the dedicated **Dosage Examples** page
(`source/dosage.html`, lines ~1100–1440 in the cached clone, including
full JSON and XML fragments) and that the explanatory paragraph
immediately following the Timing table already directs readers there
("see [Dosage Examples](dosage.html#examples) for usage"). Accept only
the narrow correction to the two "Every second day" rows
(Item 1 of Disposition A).

#### Justification

The Timing example table is intentionally a quick-reference for the most
common, simple cases; deeper offset semantics — including the
shrunken-period model formalised under FHIR-54062 — are already
documented on the dosage page and now also in the new "Periods with
start or end offsets" notes section. Duplicating that material in the
Timing table risks drift between the two locations and adds visual noise
to a wide table. The bug in the "Every second day" rows is independent of
the `endOffset` request and should still be fixed.

---

### Recommendation

**Recommended disposition:** **A**

The ticket is a small, well-scoped documentation fix on a high-visibility
page, and the reporter's own argument — that `endOffset` is "weird enough
that an example really helps understand it" — is well-supported by the
prose immediately below the table, which already promises the reader that
both offsets exist. Disposition A discharges that promise inline, where a
reader looking at the example table will actually see it, rather than
forcing them to navigate to the Dosage Examples page (Disposition C).
Disposition B is a defensible compromise on readability grounds, but the
table already carries a `startOffset` column with one worked row, so the
precedent for representing offset semantics inline in this table is
established; adding a single symmetric `endOffset` column extends an
existing pattern rather than introducing a new one. Because FHIR-54062 is
already resolved with a worked 21-of-28-day example, Disposition A also
gets a "free" canonical row to use, which keeps the two tickets in sync
and minimises authoring overhead. The companion fix to the two faulty
rows is uncontroversial under any disposition and should be merged
regardless.
