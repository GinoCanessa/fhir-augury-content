## Summary

The ticket reports that the per-resource "Operations" tab in the FHIR build
(e.g., the Operations page for resources such as `Goal`, `CarePlan`,
`Provenance`, `Transport`, and `MedicationRequest`) does not display an
"Implementation Readiness" / Standards Status (maturity) column or value for
the listed operations, whereas the same tab on
[Observation](https://build.fhir.org/observation-operations.html) does show
implementation readiness. Emma Jones (reporter) asks that implementation
readiness be added to those operation tabs so reviewers and implementers can
see the maturity of each operation per resource.

The ticket was originally triaged to Patient Care, but the linked Zulip
thread `#committers > Tooling issue?` (driven by Michelle Miller) established
that the missing column applies to operations defined at the
"all resources" level (which Patient Care does not own) and that the gap is
in the IG publisher / tooling rendering rather than in any one resource
definition. Grahame Grieve confirmed in chat that this is a tooling concern,
so the ticket has been re-grouped to FHIR-Tools-Core under FHIR
Infrastructure. There are no other cross-referenced Jira tickets or GitHub
items recorded for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a publisher / tooling defect: update the FHIR build tooling
that renders the per-resource `<resource>-operations.html` page so that the
operations table includes an "Implementation Readiness" (Standards Status /
Maturity) column populated from each `OperationDefinition`'s standards
status / FMM extension, in the same way Observation's operations page
already renders it. The fix is in the rendering template/logic for the
operations tab, not in any resource definition. After the fix, all resources
(e.g., Goal, CarePlan, Provenance, Transport, MedicationRequest) will show
implementation readiness for both resource-specific and all-resource
operations. Verify on the build site that the column is populated for the
full set of resources before closing.

#### Justification

The Zulip discussion explicitly diagnosed this as tooling, and Grahame
Grieve confirmed. The data source already exists — every
`OperationDefinition` carries (or can carry) a standards-status /
maturity-level extension that Observation's operations page successfully
renders. The asymmetry between Observation and the other resource pages is
strong evidence the underlying data is fine and only the renderer differs
for the "operations applied to all resources" section. Fixing it once in the
publisher restores consistency across every resource without per-resource
changes, which is the lowest-risk, highest-leverage path.

---

### Disposition B: Alternative Approach

#### Proposal

Address the same underlying need but go further: rather than only adding
the Implementation Readiness column to the per-resource operations tab,
factor the rendering so that operations applied to all resources are
rendered from a single shared partial that includes Standards Status,
Maturity Level (FMM), Security Category, and any other status flags carried
on `OperationDefinition`. Apply that partial uniformly to every resource's
operations tab (and review the global
[operations summary page](https://build.fhir.org/operationslist.html) for
the same gap). Optionally back-fill any all-resource OperationDefinitions
that are missing a standards-status / FMM value so the new column never
renders blank.

#### Justification

The ticket describes a single missing column, but the screenshots and
Michelle's checks show the same symptom across many resources and several
status fields. A one-column fix risks repeating the asymmetry the next time
another status field is added. Consolidating the all-resource operations
render path also makes future maintenance cheaper and reduces the chance of
Observation drifting again from the rest. The trade-off versus Disposition
A is more publisher refactoring and possibly some
`OperationDefinition` data-quality follow-ups, but the result is more
durable.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the operations being rendered are
"operations applied to all resources" whose canonical authority is the
operation page in the operations module, not the per-resource page; direct
implementers there for maturity information and leave the per-resource
operations tab as-is.

#### Justification

This is a defensible-only-on-paper position. It would avoid any tooling
change and keep the per-resource page narrowly focused on resource-specific
operations. However, the precedent set by Observation already showing
implementation readiness — and committers (Lloyd, Jean, Grahame) explicitly
calling this a tooling issue — make a hard decline difficult to defend in
practice. It is included for completeness so the workgroup has the option
to push the implementer to the operations module page rather than fix the
renderer.

---

### Recommendation

**Recommended disposition:** A (with consideration of B if the publisher
refactor is small).

The Zulip thread reached clear consensus that this is a publisher/tooling
gap, not a content gap or a Patient Care concern, and Grahame confirmed the
re-routing to FHIR-Tools-Core under FHIR Infrastructure. The data needed
for the "Implementation Readiness" column already exists on
`OperationDefinition` and is already being rendered for Observation, so the
fix is mechanical and low-risk. Disposition A delivers exactly what the
reporter asked for and resolves the inconsistency Michelle Miller and Emma
Jones documented across Goal, CarePlan, Provenance, Transport, and
MedicationRequest. If, while implementing A, the engineer finds the
all-resource operations are rendered by a separable partial, it is worth
opting into Disposition B's small refactor at the same time so future
status fields propagate uniformly. Disposition C should be declined: the
asymmetry with Observation is the bug, and committers have already
identified the correct fix domain.
