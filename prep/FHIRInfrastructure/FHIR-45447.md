## Summary

The reporter asks the SMART App Launch specification (under FHIR Infrastructure)
to update its normative references for client authentication. Today, the
Client Authentication section cites only RFC 7519 (JSON Web Token, JWT) when
describing the JWT-based asymmetric and symmetric client authentication flows.
RFC 7519 has since been supplemented by RFC 8725 / BCP 225 ("JSON Web Token
Best Current Practices"), which captures community-agreed mitigations for the
known weaknesses in JWT processing. The reporter requests that SMART App
Launch be updated to reference and require RFC 8725 / BCP 225 alongside RFC
7519 for both front-end (EHR / standalone launch) and back-end services
(client_credentials / SMART Backend Services) launches.

The Augury cross-reference index returns no incoming or outgoing links for
FHIR-45447, and no Zulip thread, GitHub PR, or Confluence page mentions the
ticket key directly. Keyword-similar Jira tickets show that the workgroup has
been actively maintaining the SMART security guidance — see in particular
[FHIR-56134](https://jira.hl7.org/browse/FHIR-56134) (Applied — relax US Core
to "SMART App Launch 2.0.0 or later"),
[FHIR-56141](https://jira.hl7.org/browse/FHIR-56141) (Submitted — disambiguate
`iss` between SMART and RFC 9207),
[FHIR-37825](https://jira.hl7.org/browse/FHIR-37825) (Published — earlier
OAuth/RFC compatibility fix), and
[FHIR-38805](https://jira.hl7.org/browse/FHIR-38805) (Published — refresh of
the FHIR Security page after SMART/Bulk IG changes). These give context that
incremental hardening of SMART's RFC references is a recurring, accepted
pattern of work for FHIR-I / Security WG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the SMART App Launch IG (specification "SMART on FHIR (FHIR)") so that
RFC 8725 / BCP 225 ("JSON Web Token Best Current Practices") is added as a
**required** normative reference everywhere RFC 7519 is currently cited,
covering both front-end launches (EHR launch, standalone launch) and back-end
services (SMART Backend Services / `client_credentials`). Concretely:

- Edit the *Client Authentication* page(s) in the SMART App Launch IG —
  notably the asymmetric (`private_key_jwt`) and symmetric authentication
  pages, plus *Backend Services* — to add wording such as: "JWTs used in
  client authentication SHALL be constructed and validated in conformance
  with RFC 7519 and SHALL follow the Best Current Practices defined in
  RFC 8725 (BCP 225)."
- Add RFC 8725 to the IG's normative references list.
- Cross-link from the *Conformance* and *Security Considerations* sections
  so implementers can see that BCP 225 mitigations (algorithm allow-listing,
  rejection of `alg: none`, explicit `typ` checks, audience validation,
  cryptographic-agility guidance) are mandatory.

(No repo briefing was loaded for `HL7/smart-app-launch`, so the exact
authoring root and file paths are not asserted here. The change is content
only — IG narrative and references — not StructureDefinition / ValueSet
artifacts.)

#### Justification

This is the lowest-risk way to honor the literal request and is consistent
with how the workgroup has handled similar updates (FHIR-56141 for RFC 9207,
FHIR-38805's security-page refresh). RFC 8725 was published specifically
because RFC 7519 alone left enough flexibility for known JWT vulnerabilities
("alg: none" attacks, algorithm confusion, weak audience checks) to recur in
production OAuth deployments — which is the same threat surface SMART
clients face. Making BCP 225 a SHALL keeps SMART aligned with current
IETF/OAuth WG guidance and with downstream IGs (e.g., US Core via FHIR-56134)
that already say "SMART 2.0.0 or later", expecting newer SMART publications
to pull in newer security baselines.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt RFC 8725 / BCP 225 by reference, but use a **layered** conformance
statement instead of a blanket SHALL: cite BCP 225 as a SHOULD in the
Client Authentication and Backend Services pages, and split out the *subset*
of BCP 225 mitigations that SMART makes mandatory into an explicit
bulleted "JWT Validation Requirements" section. Mandatory items would
include at minimum:

- `alg: none` MUST be rejected;
- the `alg` header MUST be matched against an allow-list registered for the
  client (and the IG should state the allow-list, e.g., `RS384`, `ES384`
  as already required for asymmetric auth);
- `typ` (when present) MUST be validated;
- `aud` MUST equal the token endpoint URL (already required — restate to
  align wording with BCP 225 §3.11);
- replay protection via `jti` MUST be enforced for assertion JWTs.

Other BCP 225 items (e.g., key-rollover guidance, encrypted JWT handling)
would be SHOULD/MAY, with a normative pointer to BCP 225 for full text.

#### Justification

A blanket "conform to BCP 225" SHALL is broader than SMART actually needs —
BCP 225 covers JWE, nested JWTs, and several patterns SMART does not use
today, so a literal SHALL risks creating ambiguous conformance claims and
testing burden disproportionate to the threat. Naming the specific
mitigations SMART cares about (a) makes conformance testable, (b) makes
the relationship to existing SMART text auditable, and (c) avoids the
incompatibility-by-overreach pattern that FHIR-37825 documented. This
matches the structure SMART already uses for asymmetric authentication
(named algorithm allow-list rather than "any RFC 7518 alg").

---

### Disposition C: Decline

#### Proposal

Mark FHIR-45447 as Persuasive — Considered for Future Use (or Not Persuasive
With Mod), on the basis that SMART App Launch's existing JWT requirements
already encode the most important BCP 225 mitigations (asymmetric algorithm
allow-list `RS384`/`ES384`, `aud` = token endpoint, short JWT lifetime,
single-use `jti`), and that adding a normative reference to RFC 8725 without
also re-validating each BCP 225 §3 sub-item against current SMART text would
risk introducing the kind of inconsistency that FHIR-37825 had to repair.
Defer to the next substantive SMART security pass (e.g., the work being
scoped under FHIR-56141 / RFC 9207) and bundle the BCP 225 reference there.

#### Justification

Defensible if the workgroup's read is that the protective behaviour BCP 225
asks for is *already* in SMART, just not labelled as such. Declining avoids
a "reference but no implementer-visible change" edit and avoids opening
parallel security edits in the same IG that could conflict on ballot.

---

### Recommendation

**Recommended disposition:** B

Disposition A satisfies the letter of the request but invites the exact
problem FHIR-37825 documented: a broad SHALL pointing at a long BCP whose
sub-requirements have not been individually reconciled with SMART's existing
text. Disposition C is hard to defend publicly, since BCP 225 is widely
treated as the baseline for production JWT deployments and "we already do
most of it" is not testable. Disposition B preserves the reporter's intent —
SMART normatively recognises BCP 225 — while keeping conformance crisp and
testable, and it lines up cleanly with the in-flight RFC 9207 work
(FHIR-56141) so the workgroup can land a single, coherent "SMART security
RFC refresh" rather than two overlapping edits. It also gives downstream
IGs that already say "SMART 2.0.0 or later" (FHIR-56134) a concrete,
implementer-visible improvement to inherit.
