## Summary

The reporter (Paul Lynch) asks FHIR Infrastructure to stop overloading the
UCUM unit `mo` (and, by extension, `a`, `wk`, `d`) with calendar-month /
calendar-year semantics, and to instead expose the calendar-based duration
units that FHIRPath already defines (`year`, `month`, `week`, `day`, `hour`,
`minute`, `second`, `millisecond` — see
<https://hl7.org/fhirpath/#datetime-arithmetic>). The ticket proposes three
concrete steps:

1. Define a FHIR CodeSystem for the FHIRPath calendar duration units.
2. Update ValueSets like
   [`units-of-time`](http://build.fhir.org/valueset-units-of-time.html) and
   [`age-units`](http://hl7.org/fhir/valueset-age-units.html) to include
   (or replace UCUM with) these calendar units where calendar semantics are
   intended.
3. Replace the wording introduced by
   [FHIR-24857](https://jira.hl7.org/browse/FHIR-24857) — which advises
   readers to treat UCUM `mo` as a calendar month — with guidance that
   directs users to the new FHIRPath calendar `month` (and `year`) unit when
   calendar semantics are required. The known affected locations are
   `valueset-units-of-time.html`, `Timing.repeat.durationUnit`, and
   `Timing.repeat.periodUnit`.

The ticket arose out of a long Zulip thread on the `#fhirpath` stream
(topic `FHIR-24858`), where Paul Lynch argued that the UCUM license and
UCUM's purpose as a deterministic, convertible unit system forbid
re‑interpreting `mo`/`a`, while Bryn Rhodes and Gino Canessa argued the
existing wording is "implementation guidance" rather than redefinition.
Bryn explicitly closed the thread by suggesting the reporter file a
comment against R5 to define a CodeSystem for the FHIRPath calendar units
and recommend their use — which is what this ticket does.

The two related Jira tickets cited in the description are
[FHIR-24857](https://jira.hl7.org/browse/FHIR-24857) (the prior change
that introduced the "treat `mo` as a calendar month" wording) and
[FHIR-24858](https://jira.hl7.org/browse/FHIR-24858) (the FHIRPath-side
ticket about warning when UCUM units appear in `Timing`). Neither is
returned by the cross-references API as a structured link, but both are
substantively part of the same conversation.

The ticket was triaged for R5, deferred (R5 was published before
resolution), and per the comments needs to be reconsidered for R6.

## Details

**Description:**

FHIRPath defines a set of calendar based units
(<https://hl7.org/fhirpath/#datetime-arithmetic>) so that when adding 1
month, you get the same day of the next month, regardless of the number of
days. In many cases this is what systems would prefer over working with the
UCUM unit `mo`, which is specified as 1/12 of the UCUM `a_j` unit (a Julian
year). The proposal here is:

1. Create a CodeSystem for these units defined in FHIRPath.
2. Change ValueSets like
   <http://build.fhir.org/valueset-units-of-time.html> and
   <http://hl7.org/fhir/valueset-age-units.html> to use the new calendar
   based units (either adding them to the existing lists or replacing as
   appropriate for the ValueSet's use cases).
3. Replace wording introduced by
   [FHIR-24857](https://jira.hl7.org/browse/FHIR-24857) advocating treating
   `mo` as a calendar month, with wording recommending the use of the new
   FHIRPath month unit when calendar months are desired. (Known locations
   are: `valueset-units-of-time.html`,
   `datatypes-definitions.html#Timing.repeat.durationUnit`, and
   `datatypes-definitions.html#Timing.repeat.periodUnit`.)

For additional discussion, see:
<https://chat.fhir.org/#narrow/stream/179266-fhirpath/topic/FHIR-24858/near/296428394>

## Related Zulip Discussions

### [#fhirpath > FHIR-24858](https://chat.fhir.org/#narrow/stream/fhirpath/topic/FHIR-24858/near/296428394)

The thread begins with Grahame Grieve questioning whether FHIR-24858's
resolution belongs in FHIRPath or the FHIR wrapper. Bryn Rhodes clarifies
the warning is about the implicit conversion from FHIR Quantity to
FHIRPath Quantity, and that "in FHIR, the Quantity 1 `a` almost certainly
means 1 year." Paul Lynch then objects that both FHIR-24857 and FHIR-24858
amount to redefining the meaning of UCUM calendar units (`mo`, `a`) while
still claiming the system is `http://unitsofmeasure.org`, which violates
UCUM's license and undermines convertibility. Grahame and Bryn frame the
new wording as "implementation advice" / refinement rather than
redefinition; Gino Canessa adds that UCUM itself acknowledges that any
unit larger than `s` is necessarily an estimate and so "guidance, not
redefinition" is defensible. Paul counters that UCUM's deterministic
definitions are precisely what enable reliable conversion and that the
right answer is to introduce non‑UCUM calendar units (`year`, `month`)
into FHIR's value sets and point implementers at them. The thread ends
with Bryn agreeing that, now that the FHIRPath calendar units exist, "it
would be better for systems to indicate calendar durations using those
units" and suggesting Paul submit an R5 comment to (a) define a CodeSystem
for the FHIRPath calendar units and (b) recommend their use — which is
the genesis of FHIR-38064. No firm consensus was reached on whether the
FHIR-24857 wording should be reverted; the disagreement on
"refinement vs. redefinition" of UCUM `mo` remains open.

Richard Townley‑O'Neill.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Implement all three steps for R6 in the FHIR Core specification:

1. Define a new CodeSystem (e.g.,
   `http://hl7.org/fhir/CodeSystem/fhirpath-calendar-units`) whose codes
   are the FHIRPath calendar duration unit literals — `year`, `month`,
   `week`, `day`, `hour`, `minute`, `second`, `millisecond` — with
   definitions sourced from the FHIRPath
   [§ datetime-arithmetic](https://hl7.org/fhirpath/#datetime-arithmetic)
   table.
2. Update the affected value sets to include (and prefer) the new calendar
   units alongside the UCUM codes:
   - [`units-of-time`](http://build.fhir.org/valueset-units-of-time.html):
     compose the new CodeSystem in addition to the existing UCUM include.
   - [`age-units`](http://hl7.org/fhir/valueset-age-units.html): add `year`
     and `month` from the new CodeSystem; consider deprecating `a` and `mo`
     for ages where calendar semantics are intended.
3. In the affected narrative — `valueset-units-of-time` definition,
   `Timing.repeat.durationUnit`, and `Timing.repeat.periodUnit` — replace
   the FHIR-24857 wording ("Normal practice is to use the `mo` UCUM code
   as a calendar month…") with guidance recommending the FHIRPath calendar
   units when calendar semantics are required, and noting that UCUM `mo`
   and `a` retain their UCUM-defined deterministic meaning
   (`mo = a_j/12`).

#### Justification

This is what the ticket asks for, and it directly resolves the problem
flagged on Zulip by Paul Lynch: the current narrative attempts to
redefine UCUM `mo` in place, which conflicts with UCUM's own definition
and license and undermines the only semantic guarantee a UCUM unit
provides — deterministic convertibility. Defining a separate FHIR
CodeSystem for the FHIRPath calendar units lets implementers express
"calendar month" unambiguously without reinterpreting UCUM, and it
follows the precedent already set by `fhirpath-types`. Bryn Rhodes
explicitly endorsed this resolution path on the Zulip thread ("it would
be better for systems to indicate calendar durations using those units…
submit a comment against R5 to that effect"), so there is in‑principle
agreement from the original FHIR-24857 author. The ValueSet updates are
additive (codes are added; UCUM codes remain), so the change is largely
non‑breaking for systems that continue to send `mo`/`a`.

---

### Disposition B: Alternative Approach

#### Proposal

Take steps 1 and 2 (define the CodeSystem and extend `units-of-time` /
`age-units`) but soften step 3: keep the FHIR-24857 implementation
guidance for `mo` / `a` while clearly marking the FHIRPath calendar units
as the **preferred** way to express calendar semantics going forward.
Specifically:

- Add the new `fhirpath-calendar-units` CodeSystem and weave it into the
  affected value sets (as in Disposition A).
- Re‑word the `valueset-units-of-time`, `Timing.repeat.durationUnit`, and
  `Timing.repeat.periodUnit` narrative to (a) state that new
  implementations SHOULD use the FHIRPath calendar units when calendar
  semantics are intended, and (b) retain a backwards‑compatibility note
  that historical content using UCUM `mo` / `a` is commonly interpreted
  as the corresponding calendar duration, without describing that as
  "normal practice".
- Do not deprecate or remove `mo` / `a` from any value set.

#### Justification

This addresses Paul Lynch's correctness concern (the spec stops *advising*
people to overload UCUM `mo`) while protecting the very large installed
base of resources that already use UCUM `mo` / `a` with calendar intent.
A hard removal of the FHIR-24857 wording could leave implementers without
guidance for parsing legacy data, and Bryn / Gino's point on the Zulip
thread — that the existing wording reflects what implementers are *in
fact* doing — has practical force even if the wording is theoretically
problematic. This option respects the FMG's prior 6‑0‑1 vote to consider
the change for future use without committing to an outright reversal of
FHIR-24857.

---

### Disposition C: Decline

#### Proposal

Reject the change for R6 on the grounds that:

- The `Timing.repeat.durationUnit` / `periodUnit` value sets are bound
  `required`, so changing or replacing the codes is a breaking change for
  every implementer of `Timing`.
- The FHIR-24857 wording is already shipped in normative R4 / R4B / R5
  and reflects how implementers actually interpret `mo` / `a` in FHIR
  resources.
- FHIRPath's calendar units remain available to anyone who wants them
  (FHIRPath expressions can use `1 'month'` literals today); no spec
  change is needed for engines or authors who want calendar semantics in
  computation.

#### Justification

This is defensible primarily on stability grounds: any change to the
`required`-bound `units-of-time` value set or its sibling `age-units`
ripples through every Timing-using profile and every age expression in
the ecosystem. The Zulip thread does not show consensus that the
FHIR-24857 wording is wrong — only that it is contested. Declining
preserves backwards compatibility and avoids the implementer cost of a
new CodeSystem that overlaps an existing one.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the real win the ticket is asking for — giving
implementers a non‑UCUM, semantically unambiguous way to say "calendar
month" / "calendar year" inside FHIR resources — while avoiding the
disruption of stripping a normatively‑published interpretation of UCUM
`mo` that a substantial installed base now relies on. Bryn Rhodes's own
closing position on the Zulip thread (calendar units *should* be used
going forward, and a CodeSystem for them *should* be defined) maps
directly onto Disposition B's first two steps, and softening step 3 to
"prefer FHIRPath calendar units; legacy `mo`/`a` is commonly read as
calendar-equivalent" honours both Paul Lynch's correctness concern and
Gino Canessa's installed‑base concern. Disposition A is the cleaner
answer in principle but carries a meaningful narrative‑reversal cost on
text that has been normative since R4; Disposition C is the safest but
leaves the underlying UCUM‑license / convertibility problem unresolved
and effectively closes the door that Bryn opened on Zulip.
