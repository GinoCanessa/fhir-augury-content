## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply a purely editorial fix to the FHIR Core "resource" narrative page,
correcting each item enumerated in the ticket exactly as reported:

- Expand `FMG` → `FHIR Management Group (FMG)` on first occurrence,
  then use `FMG` thereafter; ensure the parenthesized form `FMG).`
  also reads naturally after the expansion (e.g.,
  `… FHIR Management Group (FMG).`).
- `recources` → `resources`
- `busines` → `business`
- `commited` → `committed`
- `doesn't` → `does not` (replace the contraction with the expanded
  form, consistent with the reporter's request to avoid abbreviated /
  contracted forms in normative narrative).
- `identiity` → `identity`

No normative text, conformance language, schema, invariants, examples,
or links are touched. The change is confined to the prose of the
`resource` page and produces a single editorial PR against the FHIR Core
spec source.

#### Justification

The reporter has provided the exact tokens to change, all of them are
unambiguous typos or unexpanded acronyms, and none of them carry
normative weight. HL7 publication quality conventions consistently
favour spelling-out acronyms on first use and avoiding contractions in
normative narrative, so each requested correction is internally
consistent with how the rest of the spec is written. As a Technical
Correction with `Highest` priority and no cross-references implying
wider impact, the lowest-risk path is to make exactly the changes asked
for and close the ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the ticket but broaden the scope slightly: in addition to the
six tokens called out by the reporter, run a one-off editorial sweep of
the same `resource` page for the same *classes* of issue and fix any
additional instances found:

- Other unexpanded HL7 / FHIR governance acronyms on the same page
  (e.g., FHIR-I, HSC, TSC, etc.) — expand on first use.
- Other contractions (`don't`, `won't`, `can't`, `it's` where it means
  "it is") — expand to their long forms.
- Other obvious typos surfaced by a spell-check pass against the
  page's prose blocks.

Document the broader sweep in the PR description and link it back to
FHIR-55187 so reviewers can see that the ticket's intent (publication
quality of the `resource` page) was applied uniformly rather than only
to the specific tokens listed.

#### Justification

Comment 153 was filed during ballot review and the reporter explicitly
treats `FMG`, contractions (`doesn't`), and typos as defects. If those
patterns appear elsewhere on the same page, fixing only the enumerated
instances will leave the page in an inconsistent state and is likely to
attract a follow-up ticket of the same shape. A page-scoped editorial
sweep is still very low risk — no normative change, no impact on
generated artefacts — and it amortises reviewer effort. The trade-off
versus Disposition A is a slightly larger diff that takes marginally
more reviewer time, but it eliminates the "whack-a-mole" failure mode.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as out of scope for normative ballot processing,
with the rationale that purely editorial corrections (typos, expansion
of well-known governance acronyms, contractions) do not require a
Technical Correction Jira and can be handled via the standard editorial
channel — i.e., a direct PR against the spec source by any committer,
without ticket overhead. Close the ticket as `Not Persuasive` with a
note pointing the reporter at the editorial PR workflow for future
typo-class issues.

#### Justification

Defensible only on process grounds: the FHIR-Infrastructure backlog is
finite, and routing single-page typo lists through Jira consumes
disposition cycles disproportionate to the substantive risk. The
practical cost of declining, however, is that the typos remain in the
published page until *someone* either re-files the change as a PR or
re-opens the ticket. This option is included for completeness; it is
not normally how the workgroup treats a `Highest`-priority Technical
Correction filed by a co-chair-class reporter.

---

### Recommendation

**Recommended disposition:** B

Disposition A discharges the literal request and is entirely safe, but
the ticket is a ballot-comment import (Comment 153) explicitly
complaining about *patterns* — unexpanded abbreviations, contractions,
and typos — on a single specific page. If the workgroup applies the
fix only to the six enumerated tokens, any other instance of the same
patterns on `resource.html` will still be there at publication and is
very likely to come back as a fresh ticket. Disposition B costs almost
nothing extra (the sweep is bounded to one page, touches no normative
text, and produces no ripple in generated artefacts), eliminates the
re-ticket risk, and is more aligned with the publication-quality
intent behind the original ballot comment. Disposition C is included
only for completeness; declining a `Highest`-priority Technical
Correction filed by Lloyd McKenzie purely on process grounds is not
warranted here.
