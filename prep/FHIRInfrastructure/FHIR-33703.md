## Summary

Carl Anderson reports that the SMART App Launch IG's Conformance /
`.well-known/smart-configuration` metadata section does not mention the
`token_endpoint_auth_signing_alg_values_supported` discovery field, even
though the companion SMART Backend Services guidance (published from
the Bulk Data IG) requires servers that support backend services to
advertise this list (for example, values such as `RS384` / `ES384`).
The ticket asks the workgroup to decide whether this field should be
added to the SMART App Launch Conformance metadata list — and, if so,
whether it should be optional or required — and to add it to the
metadata table at
`build.fhir.org/ig/HL7/smart-app-launch/conformance.html#metadata`. No
other Jira tickets are cross-referenced in Augury for this item, so the
request stands alone, but it is tightly coupled to the Bulk Data / SMART
Backend Services conformance story.

## Details

**Description:**

The SMART Backend Services IG mentions an advertised list of algorithms
supported for the token endpoint signing process:

`token_endpoint_auth_signing_alg_values_supported`

See Backend Services guidance:
https://hl7.org/fhir/uv/bulkdata/authorization/index.html#advertising-server-conformance-with-smart-backend-services

Should this be added as an optional/required advertisement within the
SMART Launch IG Conformance section?

If so, please add it here:
https://build.fhir.org/ig/HL7/smart-app-launch/conformance.html#metadata

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the SMART App Launch IG Conformance page
(`conformance.html`, Metadata section at
`build.fhir.org/ig/HL7/smart-app-launch/conformance.html#metadata`)
to add `token_endpoint_auth_signing_alg_values_supported` as an
**OPTIONAL / RECOMMENDED** metadata field in the
`/.well-known/smart-configuration` document. Specifically:

- Add a new row / bullet in the metadata table after
  `token_endpoint_auth_methods_supported` with:
  - **Name:** `token_endpoint_auth_signing_alg_values_supported`
  - **Optionality:** RECOMMENDED when the server supports asymmetric
    (JWT-based) client authentication methods such as `private_key_jwt`;
    otherwise OPTIONAL.
  - **Type:** array of JSON strings (JWA `alg` values).
  - **Description:** "Array containing a list of the JWS signing
    algorithms (`alg` values) supported by the token endpoint for the
    signature on the JWT used to authenticate the client at the token
    endpoint (e.g., for `private_key_jwt`). Servers SHOULD include at
    least one of `RS384`, `ES384` for alignment with SMART Backend
    Services."
- Cross-reference the SMART Backend Services "Advertising Server
  Conformance" section so the two IGs stay consistent.

#### Justification

The field is already part of the SMART Backend Services conformance
list and is also well-established in OAuth 2.0 / OIDC discovery
metadata (RFC 8414 §2 registers
`token_endpoint_auth_signing_alg_values_supported`). Adding it to SMART
App Launch's metadata table closes a documentation gap that the Bulk
Data community identified several years ago (see the #bulk data Zulip
threads summarized above). Marking it RECOMMENDED-when-applicable
rather than REQUIRED preserves backward compatibility with existing
SMART App Launch deployments that don't use asymmetric client
authentication, while giving Backend Services profiles a stable anchor
to point to. It also reduces the risk of spec drift between the two IGs
that share the same `/.well-known/smart-configuration` endpoint.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding the field to the SMART App Launch metadata table
directly, add a short **normative paragraph** to the Conformance page
stating that servers advertising support for any asymmetric client
authentication method (`private_key_jwt`, or any method whose registered
metadata lives in OAuth 2.0 Authorization Server Metadata, RFC 8414)
**SHALL** conform to RFC 8414 / OIDC Discovery for those fields,
including `token_endpoint_auth_signing_alg_values_supported`, and link
to the SMART Backend Services conformance section for the concrete
algorithm expectations (`RS384`, `ES384`). No new field-by-field table
entry is added; the IG instead defers to the external specs via a
normative reference.

#### Justification

SMART App Launch already defers to OIDC Discovery / RFC 8414 for
`.well-known/smart-configuration` shape. A defer-by-reference approach
avoids having to re-document every RFC 8414 field in SMART App Launch
and prevents future drift if new algorithms or fields are added at the
OAuth/OIDC layer. It also keeps the normative "must include these
algorithms" requirement localized in SMART Backend Services, where it
belongs, and leaves SMART App Launch (user-authorized apps) free to
omit the field when the server doesn't support asymmetric client auth.
The trade-off vs. Disposition A is that readers who only consult SMART
App Launch's metadata table won't see the field explicitly — they must
follow the normative reference.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that
`token_endpoint_auth_signing_alg_values_supported` is an **OAuth 2.0
Authorization Server Metadata** field (RFC 8414) that is already
implicitly available to any SMART App Launch server — there is nothing
SMART-specific about it — and that its normative presence in SMART
Backend Services is the correct place for an algorithm-list
requirement. Close the ticket as "Not Persuasive — already addressed
by external references" and leave the SMART App Launch conformance
page unchanged.

#### Justification

SMART App Launch is explicit that its `/.well-known/smart-configuration`
is modeled on OAuth 2.0 Authorization Server Metadata; any server is
free to publish RFC 8414 fields alongside the SMART-specific ones
without the IG enumerating each one. Adding a single RFC 8414 field
piecemeal to the SMART App Launch metadata table invites a long tail
of similar "please also mention X" tickets for every other OAuth/OIDC
discovery field (`jwks_uri`,
`code_challenge_methods_supported`, etc.), which would bloat the IG
without adding normative content. Declining is defensible if the
workgroup judges that the external reference already covers the need.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest resolution because the two IGs that share
`/.well-known/smart-configuration` (SMART App Launch and SMART Backend
Services via the Bulk Data IG) currently document its contents
differently, and the asymmetry creates real confusion — which is
exactly the confusion the reporter and the older #bulk data Zulip
threads flagged. Making
`token_endpoint_auth_signing_alg_values_supported` a first-class row in
the SMART App Launch metadata table (RECOMMENDED when asymmetric client
auth is supported, OPTIONAL otherwise) eliminates that asymmetry with a
minimal, backward-compatible change and gives Backend Services a
stable anchor to point to. Disposition B solves the same problem but
pushes readers into RFC 8414, which is a worse reader experience for an
IG whose audience is primarily FHIR implementers. Disposition C is
technically defensible but leaves the documented inconsistency between
the two IGs in place, which is what generated this ticket.
