## Summary

FHIR-55173 reports that `List.status` is a mandatory element with a required
binding to a small value set (`active | retired | entered-in-error`) that lacks
any "unknown", "other", or equivalent escape-valve concept. The reporter
(Lloyd McKenzie) argues that this combination — `1..1` cardinality plus a
required binding with no fallback code — can make `List` impossible to
populate in legitimate cases where the status of the list is genuinely not
known or does not fit the existing concepts. The ticket asks FHIR-I to add
`unknown`, `other`, or some equivalent concept to the bound value set /
underlying code system.

The ticket is closely related to its sibling [FHIR-55174](https://jira.hl7.org/browse/FHIR-55174)
("List.mode has no escape valve"), which raises the same concern against
`List.mode`. Both were discussed together by Grahame Grieve in the Zulip
thread `#implementers > List.status and List.mode`, where the conversation
broadened into whether `List.status` and `List.mode` should be mandatory at
all, what they actually mean for non-patient lists, and whether the existing
mode codes are sufficient for real-world list use (e.g., IHE's XDS
SubmissionSet/Folder profiles needed extensions on top of `List` and found
`mode = working` too limiting). The ticket is currently `Triaged` with
`Highest` priority and unassigned; the only Jira comment notes that Grahame
will start (and has now started) a Zulip conversation because "the current
set of codes seems inadequate and we're wondering why no one's complained
yet."

## Related Zulip Discussions

### [#implementers > List.status and List.mode](https://chat.fhir.org/#narrow/stream/implementers/topic/List.status%20and%20List.mode/near/581209815)

Grahame Grieve opened the thread on 2026-03-23 explicitly to gather
implementer feedback on FHIR-55173 and FHIR-55174, observing that the
existing definition of `List.status` is framed around the patient record
even though most lists are not patient-specific, questioning whether a
"proposed" list is meaningful, and asking whether the current `List.mode`
codes cover real uses. John Moehrke responded that IHE, when using `List`
to support XDS SubmissionSet and Folder, found the existing `code` element
insufficient (so they layered an extension on top to separate "kind of
list" from "clinical meaning"), and that `mode = "working"` was "too
limited to be useful" while `status` was used as-is. Nikolai Ryzhikov then
challenged the more fundamental premise — why are `List.status` and
`List.mode` mandatory at all, drawing an analogy to `Patient.active` which
is optional — to which Grahame agreed that `mode` is genuinely required
("you can't work with a list with confidence unless you know what the mode
is"), but that the same argument does not clearly apply to `status`,
suggesting it may have only been mandatory because "no one ever asked for
it to be" otherwise. As of 2026-04-06 Grahame noted no further responses
and observed the thread had only the two referenced tasks attached; no
consensus has been reached on whether to (a) extend the value set, (b)
relax cardinality on `List.status`, or (c) leave both elements as-is.

Participants: Grahame Grieve, John Moehrke, Nikolai Ryzhikov.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the `ListStatus` code system (`http://hl7.org/fhir/list-status`) and
its bound value set (`http://hl7.org/fhir/ValueSet/list-status`) by adding
an `unknown` concept (and, optionally, a separate `other` concept) with
definitions consistent with the FHIR-wide pattern for "unknown"
escape-valve codes (e.g., the `unknown` code in the data-absent-reason
value set and other resources' status value sets that already include
`unknown`). Keep `List.status` as `1..1` with a required binding — the
escape valve is provided by the new code rather than by relaxing the
binding. Coordinate with FHIR-55174 to add the analogous code(s) to the
`list-mode` code system in the same release.

#### Justification

This is the most conservative change that fully addresses the reporter's
concern. It preserves the mandatory + required-binding shape of
`List.status` (which is the design choice the workgroup originally made and
is consistent with how many other FHIR resources handle status), and it
brings `List` into line with the precedent already used elsewhere in the
spec where required-bound status value sets include an `unknown` member
(e.g., `Observation.status` indirectly via `unknown`, and the broader
pattern documented in the data-absent-reason guidance). It is a
non-breaking change for existing implementers (no removed codes, no
changed cardinality), and it answers the use case John Moehrke described
on Zulip where the IHE XDS profiling needed a way to express list state
without having to lie about the value.

---

### Disposition B: Alternative Approach

#### Proposal

Relax `List.status` from `1..1` to `0..1` (and weigh the same change for
`List.mode`, deferring to FHIR-55174). The required binding stays in place
for the cases where the element is populated, but the element becomes
omittable when the sender genuinely does not know or cannot meaningfully
assert the status. As a smaller variant, leave `List.status` mandatory but
weaken its binding strength from `required` to `extensible`, allowing
implementers to bind a local `unknown`/`other` code when the standard
codes don't fit.

#### Justification

This is the direction Nikolai Ryzhikov pushed on Zulip and that Grahame
appeared at least partially open to ("I don't think the same rules
[mandatoriness] apply to status"). It addresses the underlying problem
(the element is impossible to populate in some real cases) at the
structural level rather than papering over it with a single extra code,
and it brings `List.status` into line with comparable optional-status
elements such as `Patient.active`. The cardinality option is technically a
breaking change for consumers that assume the element is always present,
and the binding-strength option introduces interoperability risk because
local codes won't be understood by other systems — both trade-offs need
to be weighed against the simpler value-set extension in Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline both FHIR-55173 and FHIR-55174 on the basis that (a) no
implementer in over a decade of FHIR has filed a usage problem against
`List.status` until now, (b) the Zulip thread did not surface a concrete
production use case where the existing codes (`current | retired |
entered-in-error`) are inadequate, and (c) implementers who genuinely
cannot determine status can already use `entered-in-error` or omit the
`List` resource entirely in favour of a simpler container. Mark as "Not
Persuasive" with a note pointing to the Zulip discussion.

#### Justification

The Jira comment trail itself flags the surprise that "no one's
complained yet" — a strong signal that real-world demand is thin. The
only concrete implementer feedback on Zulip (John Moehrke's IHE
experience) was about `List.mode` and `List.code` insufficiency, not
about `List.status`. Adding codes to a required-bound terminology has a
non-trivial cost: every conformant system must now recognise `unknown` as
a valid status, validators must be updated, and downstream profiles
(including IGs that already constrain the `list-status` value set) must
react. Without a documented use case, declining is defensible.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right balance of solving the reporter's problem
without destabilising existing implementations. The reporter's concern
is real and self-consistent (mandatory + required binding with no
fallback is a known FHIR anti-pattern — it is precisely the situation the
data-absent-reason guidance was created to avoid), and adding `unknown`
to the `list-status` code system is a small, additive, non-breaking
change that follows established precedent in other FHIR status value
sets. Disposition B's structural change (relaxing cardinality or binding
strength) is appealing in principle and aligned with Grahame's own
hesitation about whether `List.status` truly needs to be mandatory, but
it is materially riskier: cardinality changes break consumers, binding
strength changes erode interoperability, and neither is necessary to
unblock the use cases described. Disposition C is hard to defend now
that the issue has been raised explicitly and Grahame has publicly
agreed the codes look inadequate. The recommendation should be
coordinated with FHIR-55174 so that `list-mode` receives the same
treatment in the same release; the workgroup may also want to explicitly
revisit the documentation of `List.status` (the patient-record framing
Grahame called out) at the same time.
