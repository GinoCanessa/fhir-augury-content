## Summary

The ticket asks the FHIR Infrastructure work group to broaden the
declared *context of use* of the existing `package-source` extension
(defined in the FHIR Extensions Pack at
`HL7/fhir-extensions/StructureDefinition-package-source`) so that it
may also be applied to a `RelatedArtifact` element, not only to a
resource (or `Meta`) directly. The motivation, per the description, is
that during a "dependency walk" — i.e., traversal of artifact-to-artifact
references such as those in a manifest `Library` — the consumer often
does not control, and cannot rely on, the source artifact carrying
its own `package-source` extension. Allowing the extension on the
`relatedArtifact` entry lets the walker record the package origin of
each referenced artifact in the manifest itself.

This request is tightly coupled to [FHIR-53268](https://jira.hl7.org/browse/FHIR-53268)
("Support excluding resources from a given package"), filed by the same
reporter on the same day against the Clinical Decision Support work
group / CRMI specification. FHIR-53268 was resolved as *Persuasive*
(change required) and its resolution explicitly relies on the same
package-source-on-`relatedArtifact` mechanism: step 1 of its
"unpackaged" provenance algorithm is "Looking for a `package-source`
extension on the `relatedArtifact` entry in the manifest library".
FHIR-53267 is, in effect, the prerequisite extension-context change
that the CRMI `$package` `excludePackageId` resolution depends on.

## Details

**Description:**

> The package source extension allows the source package of an artifact
> to be declared on that artifact. However, there are use cases where
> we do not have control over the source of the artifact and need to
> capture package source as part of a dependency walk, rather than
> relying on it being present in the resource.
>
> Expand the context of use to allow package-source to be specified on
> a relatedArtifact so that we can capture package-source during a
> dependency walk.

**Comments:**

No comments are recorded on FHIR-53267.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add `RelatedArtifact` to the `context` list of the
`http://hl7.org/fhir/StructureDefinition/package-source` extension in
the FHIR Extensions Pack (`HL7/fhir-extensions`,
`StructureDefinition-package-source`). Concretely, append a new
`context` element with `type = element` and `expression =
"RelatedArtifact"` alongside the existing `Resource` / `Meta`
contexts, and update the extension's narrative / "Context of Use"
section to describe the dependency-walk use case (e.g., "may also be
applied to a `RelatedArtifact` entry — typically inside a manifest
`Library` — to record the package origin of the artifact referenced
when that origin cannot be carried on the artifact itself"). No
changes to the extension's child structure (`packageId`,
`packageVersion`) are made; only the `context` is widened.

#### Justification

This is the minimal change that unblocks the resolution already
adopted for FHIR-53268, whose `$package` `excludePackageId` algorithm
explicitly looks for `package-source` on the `relatedArtifact` entry
in a manifest `Library`. The Zulip "Extension 'Package Source' -
datatype issue" and "FHIR Package Scope" threads both show
implementers (terminology server, UploadFIG, CRMI tooling) already
attaching, or wanting to attach, this extension in non-self-asserted
contexts; widening the formal `context` follows the FHIR design
principle that an extension's `context` should mirror real, defensible
use cases, and avoids implementers using the extension in places its
StructureDefinition declares illegal. Adding a `context` is additive
and non-breaking for any existing user of the extension.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than overload the existing `package-source` extension, define a
purpose-built complex extension on `RelatedArtifact` —
`relatedArtifact-package-source` (or reuse a single new extension
`http://hl7.org/fhir/StructureDefinition/relatedArtifact-package-source`)
— whose `context` is `element / RelatedArtifact` only, and whose
sub-elements (`packageId`, `packageVersion`, and optionally
`packageScope` per the Tooling-channel discussion) mirror
`package-source`. Cross-link the two extensions in their narratives so
that consumers can pick the one whose `context` they need without
ambiguity. Optionally, while editing the Extensions Pack for this
change, also resolve the long-standing question raised in the
infrastructure-wg thread by tightening `package-source`'s own
`packageId` slicing.

#### Justification

A separate extension keeps the semantics of "the resource declares its
own provenance" (current `package-source`) cleanly distinct from "a
manifest entry asserts the provenance of *another* artifact." That
distinction matters because the trust model is different — a
`relatedArtifact`-level assertion is third-party metadata, often added
by a dependency walker rather than the artifact's author. It also
sidesteps the open Zulip question about whether `package-source`
should sit on `Resource` vs `Meta` (Brian Postlethwaite / Grahame
Grieve, March 2025) by not piling another, structurally different,
context onto the same definition. The cost is one more extension URL
for implementers to learn and another StructureDefinition to maintain
in `HL7/fhir-extensions`.

---

### Disposition C: Decline

#### Proposal

Do not change the `context` of `package-source`. Instead, advise CRMI
(via FHIR-53268) to look up package provenance through
`Library.relatedArtifact.resource` → resolve the canonical → read
`package-source` (or `Meta.extension:package-source`) on the
*resolved* artifact, falling back to canonical-prefix matching against
`package.json` / `package-list.json` (steps 2–3 of the FHIR-53268
algorithm). Mark FHIR-53267 as "Persuasive with Modification — see
existing dependency-walk options" or "Not Persuasive" with that
guidance in the resolution.

#### Justification

This is defensible if the WG considers third-party assertion of
package provenance — i.e., one resource attaching `package-source` to
talk about a *different* resource — to be out of scope for this
extension. It also avoids the ambiguity of two competing places (the
artifact itself, and any manifest entry pointing at it) for the same
fact, which can disagree. The cost is real, however: it forces every
dependency walker to resolve every reference even when the origin is
already known to the manifest author, and it leaves FHIR-53268's
resolution leaning on a step (#1) that the extension definition does
not allow.

---

### Recommendation

**Recommended disposition:** A

Accepting as requested is the lowest-risk, highest-leverage option.
The change is purely additive to the extension's `context`, it is
already assumed by an adjacent ticket (FHIR-53268) that the wider
community has voted *Persuasive*, and there is documented community
demand from at least three independent contexts (CRMI dependency
tracing, UploadFIG package loading, IG-Publisher auto-population) for
exactly this use. Disposition B (a parallel extension) is a clean
design but adds maintenance and learning cost without solving a real
semantic problem — `package-source` already accommodates third-party
assertion in practice (e.g., UploadFIG injecting it on load), so
formalising that on `RelatedArtifact` is consistent rather than novel.
Disposition C should only be selected if the WG actually wants to
reverse FHIR-53268's resolution, which would create cross-WG churn
with CRMI. The WG should, while accepting A, also pick up the parked
questions from the Zulip threads — the `Meta` vs `Resource` ambiguity
and the missing `packageScope` sub-extension — as follow-up tickets
so the extension is hardened end-to-end rather than only at its
context list.
