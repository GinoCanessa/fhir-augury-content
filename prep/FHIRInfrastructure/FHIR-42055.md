## Summary

Cooper Thompson (Epic) is asking the Bulk Data specification to provide a
standardized way to communicate, per patient in a group export, *why* that
patient was included in the group — i.e., which specific resource(s)
(diagnosis, lab result, encounter, etc.) caused them to satisfy the group's
inclusion criteria. The `Group` resource can describe the general selection
logic, but today there is no way for the export output to attribute a given
patient's inclusion to the concrete clinical resource(s) that triggered it.
The reporter notes this matters most for criteria-based groups, including
server-defined groups (e.g., "patients currently admitted" or "patients
discharged in the last week"), where the requestor often actually cares
about the specific qualifying `Encounter` or `Condition`.

The reporter explicitly relates this to two other Bulk Data tickets:

- [FHIR-38075](https://jira.hl7.org/browse/FHIR-38075) (Persuasive with
  Modification) — adds an alternate file structure that groups all data for
  a patient together. The reporter suggests that whatever per-patient
  metadata mechanism is adopted should plug into that grouping (for example,
  via an extension on `Bundle.entry`), so that the metadata travels with
  the patient block.
- [FHIR-49837](https://jira.hl7.org/browse/FHIR-49837) (Persuasive with
  Modification) — defines the Bulk Cohort API and a Group profile for
  *defining* criteria-based groups. In a follow-up comment the reporter
  clarifies that FHIR-49837 only standardizes how the criteria are
  expressed; it does **not** annotate the export output to say which
  resource(s) actually matched those criteria for each patient. This
  ticket is about the output-side annotation, and it must also work for
  server-defined groups that were never created via the Bulk Cohort API.

The request is open-ended on mechanism — the reporter does not insist on a
Bulk-Data-specific solution and is open to a generic "metadata per patient"
extension in `Bundle.entry` that could also serve other use cases.

## Details

**Description:**

> We support group export, and have clients that want to know why each
> patient was included in the group.
>
> The Group be composed of patients that have specific diagnosis or lab
> results, and the Group resource itself can express that general logic,
> but we'd like to be able to communicate which specific diagnosis
> qualified the patient for inclusion in the Group.
>
> While we'd like this to be supported in Bulk FHIR, I'm not sure if
> necessarily needs to be a Bulk FHIR specific solution. But I don't see
> an obvious way to communicate this given the current structures. We
> might consider addressing this and
> [FHIR-38075](https://jira.hl7.org/browse/FHIR-38075) together in such a
> way that enables the communication of metadata about patients (e.g., an
> extension in `Bundle.entry`).
>
> If we have a way to communicate per-patient "inclusion criteria", that
> could be a way we could communicate encounter-level info too. One of
> the most common group requests we get is "patients that were discharged
> in the last week" or "patients that are currently admitted". While
> those criteria are framed in terms of patients, *usually,* the
> requestor is interested in the actual admission *Encounter* for that
> patient. If a patient is included in a "currently admitted" group, then
> this is how we can communicate which Encounter qualified them for
> inclusion.

## Linked Jira Tickets

### [FHIR-38075: Support a file structure for processing data patient-at-a-time](https://jira.hl7.org/browse/FHIR-38075)

Persuasive with Modification, Published. Adds an optional alternate
output structure for Bulk Data export so that all resources for a single
patient are grouped together in the output (rather than only being
partitioned by resource type). Cooper sees this ticket and FHIR-42055 as
naturally paired: once data is grouped per patient, that grouping is the
obvious place to carry per-patient metadata such as the qualifying
resource(s), e.g., via a new parameter on the patient block or an
extension on the enclosing `Bundle.entry`.

Reporter: Cooper Thompson; Discussants include the Bulk Data / FHIR-I
group.

### [FHIR-49837: API and profile to support criteria based groups for export](https://jira.hl7.org/browse/FHIR-49837)

Persuasive with Modification, Published. Standardizes the *Bulk Cohort
API* and a `Group` profile for criteria-based groups, originating from
the Argonaut Bulk Optimize project. It governs how the criteria are
defined and how a client can ask a server to materialize a group, but it
deliberately does not specify how the per-patient *match* is attributed
back in the export output. FHIR-42055 is the complementary downstream
problem on the export-output side, and (per Cooper's comment) it must
also cover server-defined groups that bypass the Bulk Cohort API.

Reporter: Daniel Gottlieb; Discussants include the Bulk Data / Argonaut
Bulk Optimize working group.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a standardized mechanism to the Bulk Data IG to communicate, per
patient in a group export, which resource(s) caused that patient to
satisfy the group's inclusion criteria. Concretely:

- Define a new standard extension (e.g.,
  `http://hl7.org/fhir/uv/bulkdata/StructureDefinition/group-inclusion-reason`)
  whose value is one or more `Reference` to the qualifying resources
  (e.g., `Condition`, `Observation`, `Encounter`, `MedicationRequest`).
- Carry this extension on `Bundle.entry` so it composes naturally with
  the patient-at-a-time file structure introduced by
  [FHIR-38075](https://jira.hl7.org/browse/FHIR-38075). When that
  structure is used, the extension lives on the `Patient` entry that
  heads each patient's block; when the classic per-resource-type
  structure is used, the extension lives on the `Patient` resource
  itself (carried in the Patient NDJSON).
- Add a corresponding optional `_includeInclusionReason` (or similarly
  named) parameter on `$export` / `/Group/[id]/$export` so clients can
  opt in, and so servers signal support via `CapabilityStatement`.
- Specify behavior for both Bulk-Cohort-API groups and server-defined
  criteria-based groups; the mechanism must not require that the group
  was created via the Bulk Cohort API
  ([FHIR-49837](https://jira.hl7.org/browse/FHIR-49837)).

#### Justification

This delivers exactly what the reporter and Epic need, and slots cleanly
into already-accepted Bulk Data work: FHIR-49837 standardizes the
*definition* of criteria-based groups, FHIR-38075 standardizes the
patient-at-a-time *output structure*, and this ticket fills the
remaining gap — *attribution* of group membership in the output.
Anchoring on `Bundle.entry`/`Patient` keeps the mechanism extension-based
(so it is non-breaking), reuses the established FHIR pattern of
`Reference`-typed extensions, and is symmetric with how other "why is
this here?" provenance signals are conveyed in FHIR.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a new bulk-export-specific extension, address the underlying
need by emitting `Provenance` resources alongside the patient data,
where each `Provenance` has:

- `Provenance.target` = the `Patient` (and optionally the `Group`),
- `Provenance.reason` = a code such as `GROUPINCL` (group inclusion),
- `Provenance.entity.what` = `Reference` to the qualifying clinical
  resource(s) (`Condition`, `Encounter`, `Observation`, …),
- `Provenance.activity` referencing the `Group` (and, when applicable,
  the Bulk Cohort API operation) so consumers can correlate the match
  back to the group definition.

The existing `_typeFilter` / Bulk Data `Provenance` handling already
gives implementations a path to include these resources in the export
output, including under the patient-at-a-time structure from
FHIR-38075.

#### Justification

`Provenance` is the FHIR-native way to record "why is this resource
here, and what caused it to be included?" Reusing it avoids minting yet
another extension, gives clients a uniform way to consume inclusion
attribution across both bulk and non-bulk contexts, and keeps the
attribution data structurally rich (timestamp, agent, activity) instead
of a single reference list. It is also more robust for server-defined
groups, where the attribution may need to record evaluation metadata
that an extension on `Bundle.entry` cannot easily express. The trade-off
versus Disposition A is heavier on the wire and a small change to client
processing pipelines, but it is in line with how FHIR already models
this kind of attribution.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for the Bulk Data IG: the existing `Group`
resource and the Bulk Cohort API ([FHIR-49837](https://jira.hl7.org/browse/FHIR-49837))
already let a server publish the criteria, and clients that need
per-patient attribution can re-evaluate the criteria themselves against
the exported data, or request the qualifying resource types via
`_type`/`_typeFilter`. Vendor-specific use cases (e.g., Epic's
short-term need) can be served by a vendor-specific extension on
`Bundle.entry` without changing the standard.

#### Justification

The argument for declining is that the request mixes two distinct
concerns — group *definition* (already addressed) and per-patient
*derivation traceability* (a general FHIR concern, not unique to bulk
export) — and that solving it in the Bulk Data IG risks a one-off
mechanism that doesn't generalize. If the underlying need is broad,
it should be solved in the FHIR core via `Provenance` rather than in
the Bulk Data IG; if it is narrow, it can stay vendor-specific.

---

### Recommendation

**Recommended disposition:** A

The reporter has articulated a concrete, recurring implementer need
(Epic), and the workgroup has already accepted the two adjacent pieces
of work (FHIR-38075 for output structure, FHIR-49837 for criteria-based
group definition). Leaving the attribution gap unsolved means each
vendor will fill it with an incompatible extension, exactly the
fragmentation the IG is meant to prevent. Disposition B is intellectually
the cleanest fit with FHIR provenance patterns, but `Provenance` is
heavyweight for what is in practice a small list of qualifying
references, and several Bulk Data servers do not export `Provenance` at
all today, so adoption would be slow and inconsistent. Disposition C
under-serves server-defined criteria-based groups, which Cooper
explicitly calls out as the harder case. A targeted extension on
`Bundle.entry` plus an opt-in `$export` parameter (Disposition A) is a
minimal, non-breaking, implementer-friendly solution that composes
naturally with the patient-at-a-time output already on the path to
publication, and it leaves the door open to a future `Provenance`-based
representation for richer use cases.
