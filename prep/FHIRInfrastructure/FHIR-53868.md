## Summary

Daniel Schuster reports that the in-spec links from text labelled
"Asynchronous Interaction Request Pattern" point to a broken target —
`https://build.fhir.org/ig/HL7/api-incubator/async-bundle.html` — and
asks that those links be corrected. The ticket is a one-line technical
correction filed against FHIR Core, scoped to FHIR Infrastructure, and
has not yet been triaged to an assignee. There are no Jira links and no
GitHub or Zulip cross-references registered against the ticket itself,
but contemporaneous Zulip activity (the August 2025 "Async interaction

`async-bundle` page has been undergoing relocation/redesign work between
the FHIR Core spec and the `HL7/api-incubator` IG, which is the
underlying cause of the broken cross-references the reporter is seeing.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a pure link-rot fix in the FHIR Core spec
(`HL7/fhir`). Locate every occurrence of the link text "Asynchronous
Interaction Request Pattern" (and any nearby variants such as "async
bundle pattern") in the Core authoring sources and repoint them to the
current canonical URL for that page. Based on the contemporaneous Zulip
references (Da Vinci PDex thread cites `hl7.org/fhir/R5/async-bundle.html`;
HAPI thread cites `hl7.org/fhir/async-bundle.html`), the intended
canonical target is the publication URL on `hl7.org/fhir/`, not the
`build.fhir.org/ig/HL7/api-incubator/...` incubator path the link
currently points at. Update the references to the published URL (or
its build-spec equivalent rooted under `build.fhir.org/`), republish,
and verify with the Core link checker.

#### Justification

The ticket is a Technical Correction with a single concrete defect:
the cross-link lands on a non-existent page. The publish URL of the
async pattern has not been removed from the Core spec, only its
location was changed during the api-incubator relocation work
discussed on Zulip. Pointing the in-spec text at the working
canonical URL is the smallest change that fully discharges the
reporter's request, preserves implementer-facing references already in
the wild (HAPI, PDex), and does not pre-judge the larger redesign
debate that is still active on `#implementers`.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the broken link **and** add a server-side redirect (or short
landing-page stub) at the `build.fhir.org/ig/HL7/api-incubator/async-bundle.html`
path that points at the canonical page. In the FHIR Core spec, change
the link target to the canonical async-bundle URL (as in Disposition
A); additionally, in the `HL7/api-incubator` IG, add either an HTML
redirect file or an `async-bundle.html` stub that 30x-redirects (or
client-side redirects) to the canonical page. Document on the stub
page that the async pattern lives in Core, with a back-link.

#### Justification

The Zulip evidence shows the async-bundle page has been the subject of
relocation between Core and the api-incubator IG, and other documents
already in the wild link to the incubator path. Killing only the Core
side of the broken link leaves any other consumer (other IGs, slide
decks, Confluence pages, blog posts) still pointing at a 404. Fixing
both sides closes the loop and makes the relocation discoverable
rather than silently broken. The cost is one additional file in the
incubator repo, which is cheap relative to the implementer confusion
demonstrated in the August 2025 "Async interaction pattern" thread.

---

### Disposition C: Decline

#### Proposal

Decline as "won't fix — superseded." Rationale category: the
`async-bundle` page is actively being reconsidered (see the August 2025
`#implementers` "Async interaction pattern" thread, where Josh Mandel
floated a simpler poll-then-GET pattern and Nikolai Ryzhikov proposed
HTTP-semantics changes such as `303 See Other` / `Content-Location`),
and the larger redesign will replace the page entirely. Patching the
link to a target that is itself slated to change risks doing the work
twice. Close FHIR-53868 with a pointer to the redesign track and
re-open / re-file once the destination is stable.

#### Justification

Defensible only if the work group is confident the redesign will land
in the next ballot cycle and that the current page will be superseded
or removed. It avoids spending editor time on a link target that is
known to be in flux. The downside — and the reason this is unlikely to
be the right call — is that implementers in the field today (HAPI,
PDex bulk member match) are following the existing page right now, so
leaving a 404 in Core for the duration of the redesign harms exactly
the audience the spec is supposed to serve.

---

### Recommendation

**Recommended disposition:** A

This is a one-line technical correction filed against published normative
content. The reporter is not asking the work group to relitigate the
async pattern — they are asking for a working hyperlink, and that
request is fully separable from the redesign discussion happening on
`#implementers`. Disposition A delivers exactly that, in the smallest
edit, and unblocks the reporter and any other reader hitting the same
404 today. Disposition B is a sensible amplification if (and only if)
the api-incubator maintainers are prepared to take the redirect stub —
the work group can adopt B as a friendly amendment if that owner is in
the room. Disposition C should be reserved for the redesign tickets
themselves; using it to close FHIR-53868 punishes implementers for an
editorial process they are not party to and leaves a known-broken link
in the published spec.
