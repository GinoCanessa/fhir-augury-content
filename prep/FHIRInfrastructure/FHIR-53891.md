## Summary

Eric Haas (FHIR Infrastructure / Security & Privacy) proposes a set of
editorial revisions to the **Canonicalization** section of the FHIR Core
**Signatures** page (`source/signatures.html`) for clarity and internal
consistency. The proposed text restructures the section so the
canonicalization-method URI is introduced once at the top
(`Signature.targetFormat` SHOULD carry it), then defines three sibling
methods — JSON, JSON-XML (XML-Safe JSON), and XML — each with their own
URI under `http://hl7.org/fhir/canonicalization/{json|json-xml|xml}` and
their own ordering and serialization rules. Notably, the rewrite makes
explicit that when `Provenance` has multiple signed targets, the JSON
canonicalization places them in a JSON Array that preserves
`Provenance.target` order (and is itself canonicalized per RFC 8785),
while the XML method canonicalizes each target separately and includes
them in the digital signature in `Provenance.target` order. The proposed
text is hosted on Eric's personal fork
(`Healthedata1/fhir`); no PR has been opened against `HL7/fhir` yet, and
the only system cross-reference attached to the ticket is the FHIR
artifact `Provenance.target`. There are no explicitly-linked Jira tickets,
but the change is the editorial follow-up to an extended Security &
Privacy discussion on Zulip ("Provenance Signature", Aug–Sep 2025) in
which Grahame Grieve published a related editor's draft on a working
branch (`2025-08-gg-signatures`).

## Details

**Description:**

> Edits to the Canonicalization section for clarity and consistency:
>
> Proposed Text: (Source at <https://github.com/Healthedata1/fhir/blob/master/source/signatures.html>)
>
> ### Canonicalization
>
> A consistent representation of resources in their wire formats (JSON
> and XML) is required to make signatures interoperable; this is called
> 'Canonicalizing' resources. Because of the complexities of the
> interplay between data exchange and human-level signatures, FHIR
> defines several ways to canonicalize resources, each suitable for
> different purposes and identified by a URI. The canonicalization
> method URI **SHOULD** be represented in the `Signature.targetFormat`
> to communicate the method used. Canonical methods for JSON, JSON-XML,
> and XML and their variants are defined in the following sections.
>
> #### JSON
>
> When JSON resources are canonicalized, they SHALL be canonicalized in
> accordance with the JSON Canonicalization Scheme defined in RFC 8785.
> In addition, the following rules and guidelines apply:
>
> - When a provenance has multiple target resources that are signed,
>   the resources are placed in a JSON Array preserving the
>   `Provenance.target` order, which itself is canonicalized according
>   to the rules of RFC 8785.
> - If the resource is represented as XML at some point in the life
>   cycle between signing and verification, see the JSON-XML
>   canonicalization method below.
>
> The base JSON canonicalization method canonicalizes the entire
> resource and is identified by the URI
> `http://hl7.org/fhir/canonicalization/json`.
>
> #### JSON-XML (XML Safe JSON Canonicalization)
>
> The simple JSON Canonicalization defined above does not guarantee the
> reconstruction of a standard payload for signature verification if
> the resource is represented as XML at some point in the life cycle
> between signing and verification. This is because, in XML, XHTML
> elements (i.e., Resource and Section Narratives) are part of the XML
> DOM, and whitespace and namespaces may be reorganised without
> changing the canonical XHTML. Therefore, the JSON-XML method is
> defined. The rules for this canonicalization are the same as for the
> JSON canonicalization, except that prior to serialization, the
> content of any xhtml in the narrative SHALL be canonicalized
> according to the XML canonicalization rules defined for XML below.
> The base JSON-XML canonicalization method is identified by the URI
> `http://hl7.org/fhir/canonicalization/json-xml`.
>
> #### XML
>
> Resources are canonicalized using XML canonical method Canonical XML
> 1.1 (`http://www.w3.org/2006/12/xml-c14n11`). In addition, the
> following rules apply:
>
> - The XHTML Narrative SHALL be canonicalized following these same rules
> - Whitespace in other primitive values SHALL not be altered
>   (including multiple spaces in markdown)
> - Resources are to contain no whitespace other than as found in
>   attribute values and in the XHTML in the narrative, and single
>   spaces between attributes in XML elements
> - Use default namespaces for the FHIR and XHTML namespaces
> - Omit all comments
> - Include the XML processing instruction
>   `<?xml version="1.0" encoding="UTF-8"?>`
> - When sets of resources are signed, each resource gets its own
>   `Reference` element inside the XML Digital Signature's `SignedInfo`
>   element.
> - When a provenance has multiple target resources that are signed,
>   the multiple canonicalized XMLs are contained in the digital
>   signature as separate signed content in the order in which the
>   targets are defined.
>
> The base XML canonicalization method is identified by the URI
> `http://hl7.org/fhir/canonicalization/xml`.

Moehrke raises the larger modeling question of whether the `Signature`

notes the duplication of `who`/`when`/`type` between `Signature` and
`Provenance`, but ultimately defers that to a separate ticket; consensus
on the editorial portion was effectively reached and Grahame indicated
"made the rest of the changes, thanks." A Security WG Q1 session was
scheduled to confirm.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply Eric Haas's proposed text to `source/signatures.html` in
`HL7/fhir`, replacing the current Canonicalization section
(introductory paragraph + JSON, JSON-XML, XML subsections) verbatim
with the prose linked from the Description (mirrored on
`Healthedata1/fhir@master:source/signatures.html`). Specifically:

1. Rewrite the Canonicalization intro to introduce the
   "method URI SHOULD be represented in `Signature.targetFormat`"
   sentence once, at the top, instead of repeating it in each
   subsection.
2. Replace the JSON subsection with the two-bullet form, making
   explicit that multi-target signing produces a JSON Array preserving
   `Provenance.target` order and that the array itself is canonicalized
   per RFC 8785.
3. Replace the JSON-XML subsection with the cleaned-up paragraph that
   reuses JSON canonicalization rules and adds the XHTML-narrative
   carve-out.
4. Replace the XML subsection with the new bullet list (predicate "When
   sets of resources are signed," restored, hanging colon removed,
   "multiple canonicalized XMLs … in the order in which the targets are
   defined" wording adopted).
5. Preserve the existing `<a name="…">` anchors (`Canonicalization`,
   `JSON`, `JSONXML(XMLSafeJSONCanonicalization)`, `XML`) so inbound
   deep links from other pages continue to resolve.

This is a documentation-only change; no StructureDefinition,
ValueSet, or example resource changes are required. Open the PR
against `HL7/fhir:master`.

#### Justification

The edits are editorial, not normative: they do not change the
canonicalization URIs, the conformance verbs (`SHALL`/`SHOULD`), or
the underlying algorithms (RFC 8785 for JSON, Canonical XML 1.1 for
XML). They eliminate concrete defects flagged on Zulip — the missing
"When sets of resources are signed," predicate, the hanging colon in
the whitespace bullet, the duplicated mention of
`Signature.targetFormat`, and the under-specified ordering for
multi-target signed resources. Grahame Grieve has already
incorporated equivalent edits on the working branch
`2025-08-gg-signatures`, demonstrating editor consensus that the
existing text is unclear and that this rewrite is acceptable. The
fhir-core playbook's "Update narrative / prose" recipe applies
cleanly: the change touches a single `source/*.html` page in the
authoring root with no generated-content or cross-repo implications.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the editorial cleanups in Disposition A **but split the
ordering/Provenance-target semantics out as a separate normative
clarification ticket**. Concretely:

1. Apply only the *editorial* edits in this ticket: anchor-preserving
   restructure, fixing the hanging colon, restoring the missing
   predicate, deduplicating the `Signature.targetFormat` sentence, and
   tightening prose. Preserve the existing wording on multi-target
   ordering as-is.
2. File a follow-up Change Request (linked to FHIR-53891 and to the
   Zulip "Provenance Signature" thread) that owns the substantive
   questions raised on Zulip but **not** resolved in the proposed text:
   - Whether the `Signature` datatype's `who`/`when`/`type` should be
     deprecated in favour of `Provenance` backbone equivalents (per
     John Moehrke).
   - Whether a single signature may legitimately cover a subset of
     `Provenance.target` (per Grahame's comment), and if so how
     implementers indicate the covered subset.
   - How mixed JSON+XML targets within a single Provenance are
     signed/canonicalized when no single `Signature.targetFormat` URI
     applies cleanly.
3. Land the editorial PR first; tackle the normative work on the
   separate ticket on the normal ballot cadence.

#### Justification

The proposed text bundles two distinct concerns: pure clarity edits
(low risk, no behaviour change) and clarifications about multi-target
ordering and JSON-Array wrapping that *do* nudge implementer behaviour
(specifically, the explicit "JSON Array preserving `Provenance.target`
order, which itself is canonicalized per RFC 8785" sentence is new
guidance, not just a rewording). Splitting the work lets Infrastructure
land the safe editorial pass quickly while Security & Privacy resolves
the harder modeling question without holding up the cleanup. It also
keeps the audit trail clean: the editorial PR can ship as
"non-substantive change" while the normative clarification gets the
review attention it warrants. The cost is modest — one extra ticket
and one extra round-trip.

### Disposition C: Decline

#### Proposal

Close FHIR-53891 as **Persuasive with Modification — already
addressed**, on the basis that Grahame Grieve's editor's draft on
branch `2025-08-gg-signatures`
(<https://build.fhir.org/branches/2025-08-gg-signatures/signatures.html>)
already incorporates equivalent edits and is on track to merge. Direct
the reporter to confirm the working branch matches their proposed
text and, if so, to comment on the relevant PR rather than opening a
parallel change. No further action on this ticket.

#### Justification

The Zulip thread shows Grahame explicitly stating "made the rest of
the changes, thanks" after working through Eric's bullet list, and
publishing the result on a named editor's branch. If that branch
captures the same intent, accepting FHIR-53891 as a separate change
risks duplicate or conflicting edits to a single section. Declining
in favour of the in-flight branch concentrates review attention on one
artifact and avoids merge churn on `source/signatures.html`. The risk
is real — without a verbatim diff against the working branch, the
workgroup cannot be certain Eric's text and Grahame's text are
identical, so this disposition should only be chosen if the reporter
confirms equivalence. If the texts diverge in any non-trivial way,
fall back to Disposition A or B.

### Recommendation

**Recommended disposition:** **A — Accept As Requested.**

The change is a single-file edit to `source/signatures.html` in
`HL7/fhir`, in the authoring root, with no generated-content or
cross-repo blast radius (per the fhir-core briefing). The proposed
text fixes concrete, agreed-upon defects (missing predicate, hanging
colon, duplicate sentence) and tightens the multi-target ordering
language in a way that the Zulip thread treated as the obvious
reading rather than a behavioural change — Grahame Grieve and John
Moehrke both engaged with the wording and Grahame already reflected
the substance of it on the working branch. Disposition B is the
defensible fallback if the workgroup wants to keep the
"JSON Array preserving `Provenance.target` order" sentence under a
separate normative-clarification banner, but splitting introduces
ticket overhead for what is, on net, a cleanup of wording the
authoring team has already converged on. Disposition C should only be
selected if the reporter confirms the Healthedata1 fork and the
`2025-08-gg-signatures` editor's draft are textually equivalent;
absent that confirmation, declining risks losing the explicit
`Provenance.target`-ordering sentence that Eric proposes and that the
current published wording lacks.
