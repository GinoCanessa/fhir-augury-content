## Summary

Shanai Thornton is requesting a wording tightening on the FHIR Core
"Resource" page (`resource.html#additional`), section *Additional Resources
defined Outside This Specification*. The third rule in that section currently
reads (paraphrased): *"Registered and approved resources are given a name by
HL7, and this is the name that **must** be used for the resource."* The
reporter argues that this rule is, in substance, an absolute conformance
requirement and should therefore use the RFC 2119 keyword **SHALL** rather
than the lower-case "must". The change is purely a conformance-language
clarification on a single specification page; no new behaviour is being
introduced and no resource definitions change. No related FHIR Jira tickets
were surfaced through cross-reference lookups, so the request stands on its
own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the third rule under *Additional Resources defined Outside This
Specification* on `resource.html` in the `HL7/fhir` repository
(authoring root `source/`, file `source/resource.html`) so that the
lowercase "must" is replaced with the RFC 2119 keyword **SHALL**, in
small caps as used elsewhere in the specification:

> Registered and approved resources are given a name by HL7, and this is
> the name that **SHALL** be used for the resource.

No other rules in the bullet list are altered and no surrounding prose
needs to change. After the edit, run a quick sanity check that no
inbound anchors (e.g., `resource.html#additional`) are affected (the
section anchor is unchanged), and confirm the rendered page picks up
the keyword styling consistent with other SHALL statements on the same
page.

#### Justification

The sentence describes a single, testable, absolute conformance
obligation: there is exactly one HL7-assigned name for a registered
additional resource, and an implementation that uses any other name is
non-conformant. Per the FHIR specification's own conformance rules
(`conformance-rules.html`) and the inherited RFC 2119 vocabulary, that
is the textbook definition of a SHALL statement. The other rules in
the same bullet list already use uppercase keywords (SHALL, SHOULD)
where they impose conformance requirements, so promoting "must" to
SHALL here improves intra-page consistency and removes a small but
genuine ambiguity for tooling vendors and HL7 reviewers who scan for
RFC 2119 keywords. The change is documentation-only, has no impact on
any StructureDefinition, and carries no migration cost.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only swapping "must" → "SHALL" in the third rule, perform a
small targeted audit of the *Additional Resources defined Outside This
Specification* section as a whole and normalise the conformance
vocabulary in **all** rules that express absolute requirements. In
particular, review each bullet rule for cases of "must", "must not",
"should", and "may" used in lowercase and either (a) promote them to
uppercase RFC 2119 keywords where they are genuinely normative, or (b)
rephrase them as descriptive prose where they are not. Land all of
these as a single edit to `source/resource.html` so reviewers can
weigh the wording changes together.

#### Justification

Single-keyword edits to specification prose tend to attract follow-up
tickets pointing at the next-most-glaring instance on the same page,
which then incur their own ballot/comment lifecycle. Doing the small
cluster in one pass is cheaper for the workgroup and produces a more
internally consistent section. The trade-off versus Disposition A is
scope creep: a one-word fix becomes a small editorial sweep that may
touch wording the original commenter did not raise, and reviewers must
agree on each promotion. It also delays resolution of the specific
issue the reporter raised, since the broader sweep needs more eyes.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave the third rule reading "must" as-is, on
the basis that lowercase "must" in FHIR specification prose is widely
understood as an obligation in context and that the *Resource* page
already documents elsewhere (in the conformance section) how RFC 2119
keywords are interpreted. The rule's force is not changed by the
keyword case, and FHIR Infrastructure has historically been
conservative about purely editorial keyword promotions on stable
pages.

#### Justification

A defensible "Decline" rests on three points: (1) the surrounding
sentence is unambiguous in plain English regardless of casing; (2)
mechanically promoting every lowercase "must" to SHALL across the
specification would be a large, unbounded editorial program that this
single ticket does not authorise; and (3) the change has no
implementer-visible behaviour, so the cost of opening the ballot
window for a one-word change is hard to justify if the WG is in a
late ballot freeze. This disposition is the weakest of the three
because the cost of the change is genuinely tiny, but it is the
appropriate fallback if the WG decides not to touch ballot prose for
non-substantive edits.

---

### Recommendation

**Recommended disposition:** A

The reporter is correct on the merits: the third rule expresses a
single, absolute, testable obligation on the name used for a
registered Additional Resource, which is exactly what RFC 2119 SHALL
exists to mark. The fix is a one-word edit to a single authored file
(`source/resource.html`) in `HL7/fhir`, with no impact on
StructureDefinitions, examples, generated artefacts, or downstream
IGs, and no migration burden for implementers. Disposition B is
attractive in principle but converts a trivial fix into an editorial
sweep that this ticket did not scope; FHIR Infrastructure can take
that on as a separate, broader hygiene ticket if it wants to. The
strongest argument against acting at all (Disposition C) is "it
doesn't change behaviour," but the same argument applies in reverse:
the change is so small and so well-aligned with FHIR's own conformance
vocabulary that declining sends a worse signal — that the spec is not
willing to tighten its own conformance language even when a community
member has done the work of identifying a clear case. Accept as
requested.
