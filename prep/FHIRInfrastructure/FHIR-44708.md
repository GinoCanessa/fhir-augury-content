## Summary

The reporter requests that the FHIR R5 Subscriptions Backport IG publish a set
of `SearchParameter` artifacts defined against the `Basic` resource so that
implementers who discover backported SubscriptionTopics via `Basic` (the
pattern used when a native `SubscriptionTopic` resource is not available in
R4) have at least the same discovery search capability that R4B offers on
`SubscriptionTopic` itself. The reporter points to the
[R4B SubscriptionTopic search parameters](https://hl7.org/fhir/R4B/subscriptiontopic.html#search)
as the minimum target set and cites the
[IHE DSUBm `SearchParameters.fsh`](https://github.com/IHE/ITI.DSUBm/blob/edit-post-pc/input/fsh/SearchParameters.fsh)
as a concrete example of what "the same parameters, retargeted at `Basic`"
looks like in FSH. No cross-referenced Jira tickets were returned by the
cross-reference query, so this change is scoped to the Subscriptions Backport
IG and is not entangled with other tracked tickets.

## Related Zulip Discussions

### [subscriptions > Subscription backport in R4](https://chat.fhir.org/#narrow/stream/subscriptions/topic/Subscription%20backport%20in%20R4/near/419922920)

Stefano Contento (the ticket reporter) opened this thread a few days before
filing FHIR-44708, asking how to profile a `SubscriptionTopic` in an R4 IG and
whether the R4 Subscriptions Backport provides a way to keep the R5
SubscriptionTopic search parameters when the topic is expressed as a `Basic`
resource in R4. Jean Duteau explained that `SubscriptionTopic` is not an R4
resource and therefore cannot be profiled in FSH via `Parent: SubscriptionTopic`,
but an instance can be authored in JSON/XML and the IG Publisher will handle
it; John Moehrke added that such instances belong in `input/resources`. Gino
Canessa (Subscriptions Backport maintainer) confirmed that search parameters
must be defined against `Basic` for them to work in R4 and that no tooling
automatically retargets SubscriptionTopic search parameters; he suggested the
extension can be referenced from a `Basic`-targeted `SearchParameter`.
Contento closed the exchange by stating the intent to define the search
parameters against `Basic` to emulate the SubscriptionTopic set — which is
exactly the action the ticket asks the IG itself to publish. No opposition was
voiced; the discussion reached a working consensus that `Basic`-targeted
`SearchParameter` artifacts are the correct mechanism.

Participants: Stefano Contento, Jean Duteau, Gino Canessa, John Moehrke,
Gianluca Pavan, Lloyd McKenzie, Josh Mandel, Grahame Grieve, Jens Villadsen,
David Otasek.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a set of `SearchParameter` resources to the FHIR R5 Subscriptions Backport
IG (`HL7/fhir-subscription-backport-ig`) whose `base` is `Basic` and whose
expressions mirror the R4B `SubscriptionTopic` search parameters listed at
<https://hl7.org/fhir/R4B/subscriptiontopic.html#search>. At a minimum, cover
the parameters the reporter highlights and that the IHE DSUBm IG already
implements against `Basic` (e.g., `url`, `version`, `title`, `date`, `status`,
`resource`, `trigger-description`, `derived-or-self`), using expressions that
walk the backport-defined extensions on `Basic` (the SubscriptionTopic
canonical extension, the R5 SubscriptionTopic backport extensions for status,
trigger, resource type, etc.). Publish them alongside the existing
SubscriptionTopic backport artifacts so IG-conformant servers can advertise
them in their CapabilityStatement and clients can discover topics uniformly
whether the server materializes them as `SubscriptionTopic` (R4B/R5) or as
`Basic` (R4).

#### Justification

This is the literal request and it aligns with the Zulip thread consensus:
`SearchParameter`s in R4 must be defined against `Basic` to function, no
tooling auto-retargets R5 SubscriptionTopic search parameters, and an existing
downstream IG (IHE DSUBm) has already proven the shape of the solution. The
backport IG's whole purpose is to make the R5 Subscriptions pattern usable in
R4; shipping the discovery search parameters is a natural completion of that
contract and avoids every implementer re-authoring the same set in their own
IGs. No backward-compatibility concern arises because these are additive
artifacts.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than publishing a full parallel set of `Basic`-targeted
`SearchParameter`s in the core Subscriptions Backport IG, add a single
**normative section plus a non-normative FSH snippet (or an included
capability example)** showing how to author `SearchParameter`s against `Basic`
that walk the backport extensions, and optionally ship a small curated subset
(e.g., `url`, `status`, `resource`) as examples. Direct implementers who need
the full R4B SubscriptionTopic search surface to copy/adapt the snippet or
reuse IHE DSUBm's `SearchParameters.fsh`. Cross-reference DSUBm from the
backport IG so the work is discoverable.

#### Justification

`SearchParameter`s bound to `Basic` are arguably out of band for a backport
whose primary job is to carry the R5 SubscriptionTopic/Subscription shapes
into R4 via extensions. Formally publishing them in the core IG risks pinning
expression syntax choices (the exact FHIRPath walking the backport extensions)
across all downstream profiles — if the extension layout ever shifts, every
server conforming to these parameters would be affected. A documentation +
example posture preserves flexibility while still closing the knowledge gap
that drove the Zulip thread and this ticket.

---

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope** for the FHIR R5 Subscriptions Backport
IG. Note in the resolution that `SearchParameter`s targeting `Basic` are the
implementer's responsibility in R4 because (a) `Basic` is the generic
escape-hatch resource, (b) different communities will profile their
`Basic`-as-SubscriptionTopic pattern differently, and (c) the R4B track
(which has native `SubscriptionTopic`) is the recommended path when R5-style
discovery search is required.

#### Justification

Publishing `Basic`-bound `SearchParameter`s in a core IG can entrench a single
interpretation of the backport pattern when `Basic` is deliberately a low-
semantics fallback. If implementers want R5-style discovery, moving to R4B is
the already-sanctioned answer, and the backport IG's README could be
strengthened to say so. This keeps the backport surface minimal and avoids
committing the WG to maintaining a parallel search-parameter set.

---

### Recommendation

**Recommended disposition:** A

The Subscriptions Backport IG already commits to making the R5 pattern usable
in R4; shipping discovery `SearchParameter`s against `Basic` is the last mile
of that commitment, and the Zulip thread shows an IG-side maintainer (Gino
Canessa) confirming that `Basic`-targeted `SearchParameter`s are the only
mechanism that actually works in R4. Option B under-delivers — every
implementer re-authoring the same eight-ish search parameters is exactly the
duplication an IG is supposed to eliminate, and IHE DSUBm has already shown
the artifacts are stable enough to ship. Option C effectively tells R4
implementers to migrate to R4B, which defeats the purpose of the backport.
Accepting the request, scoped to the search parameters enumerated at
<https://hl7.org/fhir/R4B/subscriptiontopic.html#search>, is the cleanest
outcome and gives the WG a bounded, reviewable change set.
