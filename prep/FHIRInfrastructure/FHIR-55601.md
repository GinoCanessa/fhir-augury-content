## Summary

The ticket asks the FHIR Infrastructure work group to add a general-purpose
`otherCode` extension to the FHIR Extensions Pack so that, whenever a
required value set binding includes an explicit "other" placeholder code,
implementers have a standard, well-known mechanism for conveying the
actual coded concept that "other" stands for. Today there is no general
extension for this purpose, so each specification or implementation that
uses an "other"-style required binding has to invent its own pattern,
which is inconsistent and hard for downstream consumers to interpret.
The reporter (Bryn Rhodes) proposes a single shared extension that can
be applied wherever this pattern occurs.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with this issue in the search index, so the request
stands alone. The resolution recorded on the ticket is **Persuasive**
and includes a concrete shape for the extension (see Details below).

## Details

**Description:**

When a required binding has an "other" code, an extension to be used to
provide the "other" code is suggested. However, there is not a general
extension for that purpose. Consider adding "otherCode" as an extension
that can be used routinely to specify the "other" code for required
bindings.

**Resolution Description:**

Define an `otherCode` extension that can be used when an "other" code
appears in a required binding to specify what the actual code is:

> `otherCode` 0..1 CodeableConcept, usable on `code`
> **Short:** Indicates the actual code for an "other"
> **Definition:** Used on code elements with a required binding that
> includes an "other" value to provide the actual coded concept.

No comments are recorded on the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new extension in the FHIR Extensions Pack with the following
shape, exactly as proposed in the resolution:

- **URL:** `http://hl7.org/fhir/StructureDefinition/otherCode`
- **Context:** element of type `code` (and, by extension, `Coding` /
  `CodeableConcept` elements that wrap a `code` whose binding includes
  an "other" value).
- **Cardinality on context:** 0..1.
- **Value type:** `CodeableConcept`.
- **Short:** "Indicates the actual code for an 'other'".
- **Definition:** "Used on code elements with a required binding that
  includes an 'other' value to provide the actual coded concept that
  the 'other' code stands for."
- **Comment / usage notes:** Only meaningful when the parent element's
  required binding contains an explicit "other" placeholder code; the
  `CodeableConcept` value SHOULD come from a value set or code system
  that is more specific than the parent binding.

Publish the extension in the FHIR Extensions Pack so that it is
available to all FHIR specifications and implementation guides without
requiring per-IG redefinition.

#### Justification

This is exactly what the workgroup has already marked **Persuasive**
and is the simplest path to consistency across the ecosystem. A single,
well-known extension URL means tooling (validators, terminology
servers, mapping engines) can recognize and special-case the "other →
actual code" pattern uniformly. Defining it in the Extensions Pack
(rather than in any single resource or IG) matches FHIR's pattern for
cross-cutting, optional metadata that augments existing required
bindings without forcing breaking changes to those bindings.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same `otherCode` extension shape, but broaden the design and
guidance in two ways before publishing:

1. **Permit a `code` value alongside `CodeableConcept`.** Define the
   extension's value as a choice of `code | Coding | CodeableConcept`
   (or publish a small extension family). Many "other"-style bindings
   live on `code`-typed elements where a full `CodeableConcept` is
   structurally awkward, and consumers often only need the literal code
   plus its system.
2. **Add explicit usage rules and an invariant** in the extension
   definition stating that the extension SHALL only appear when the
   parent element's actual code is the binding's designated "other"
   value, and that the value SHOULD NOT repeat that "other" code. Pair
   this with a short "How to use otherCode" page in the Extensions Pack
   and a brief reference from the binding-strength documentation in
   FHIR Infrastructure's terminology guidance.

#### Justification

The bare proposal in Disposition A is sound but under-specified for an
extension intended to be used "routinely". Allowing a `code` value
reduces friction on the most common authoring case (a single replacement
code from a known system) and avoids implementers wrapping a single code
in a `CodeableConcept` purely to satisfy the type. Adding an invariant
and guidance prevents the predictable misuse where authors attach
`otherCode` to elements whose binding has no "other" placeholder, which
would dilute the extension's semantics and complicate validator
behaviour. The cost is modest — one extra extension definition (or a
choice type) and a short guidance page — and the long-term
interpretability win is significant.

---

### Disposition C: Decline

#### Proposal

Do not add a new general-purpose `otherCode` extension to the FHIR
Extensions Pack. Instead, advise implementers to either (a) use the
existing `valueCodeableConcept` / `valueCoding` extension pattern
defined locally in the relevant IG, or (b) replace the required binding
that contains an "other" code with an extensible binding so that the
actual code can be sent in the element itself rather than via an
extension.

#### Justification

A generic `otherCode` extension formalizes — and arguably encourages —
the "required binding with an 'other' escape hatch" pattern, which is
already a workaround for value sets that should probably have been
extensible in the first place. Standardizing the workaround makes it
cheaper to keep using a design FHIR generally discourages, and it
shifts the actual coded value off the bound element into an extension
where validators, profiles, and mapping rules will not see it without
extra work. Declining keeps pressure on spec authors to choose the
right binding strength rather than papering over it with a global
extension.

---

### Recommendation

**Recommended disposition:** B

Disposition A delivers what the reporter asked for and matches the
**Persuasive** resolution already on the ticket, so the workgroup can
ship it quickly. However, an extension intended to be used "routinely"
across every required binding with an "other" code deserves slightly
more design care before it lands in the Extensions Pack. Disposition B
keeps the same shape and the same publication target, but tightens
two things that will matter once the extension is in the wild: a value
type that fits the most common `code`-typed parent elements without
forcing a `CodeableConcept` wrapper, and an invariant plus short
guidance page so validators and authors apply it consistently.
Disposition C raises a real design concern about "other"-style required
bindings, but that is a broader methodology debate; blocking this
ticket on it would leave implementers with the same ad-hoc patterns
they have today, which is strictly worse than a well-defined extension.
B therefore best balances the reporter's intent, the workgroup's
existing Persuasive call, and the long-term interpretability of FHIR
data.
