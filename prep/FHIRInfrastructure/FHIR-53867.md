## Summary

The ticket asks the FHIR Infrastructure work group to clarify the
expected server behavior when a `Create` request arrives carrying an
`If-None-Exist` header on a server that supports the standard `Create`
interaction but does **not** advertise support for the `Conditional
create` interaction. The current specification is silent on this case,
leaving it ambiguous whether the server should (a) silently ignore the
header and process the POST as a plain create, or (b) reject the request
with an error.

The reporter notes that although a client can in principle inspect the
server's `CapabilityStatement` to determine whether conditional create
is supported, an explicit error response would provide an additional
safety net for use cases where duplicate prevention has clinical
consequences (e.g. Medications, Patient matching). The request is for
the spec to state, normatively, what conformant behavior is in this
scenario.

This ticket sits alongside the recently-applied
[FHIR-53546](https://jira.hl7.org/browse/FHIR-53546), which added a
"dragon" warning that conditional create is not transactionally
enforceable and that race conditions can leak duplicates even when a
server supports it. FHIR-53867 is complementary: 53546 covers what can
go wrong when the feature **is** implemented; 53867 covers what should
happen when it is **not**. Together they tighten the conformance story
around `If-None-Exist`.

## Details

**Description:**

> If a server does not support the *Conditional Create* interaction, but
> does support the *Create* interaction, and receives a *Create* request
> with an `If-None-Exist` header, should the server ignore the header
> and process the request as if it were a normal *Create*, or should the
> server reject the request?
>
> While clients should be able to determine whether Conditional create
> is supported for a particular server via the `CapabilityStatement`,
> throwing an error might provide an extra layer of protection,
> particularly for use cases in which duplicate checking has clinical
> implications (e.g. Medications, Patient matching).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative paragraph to the `If-None-Exist` subsection of
`http.html#ccreate` in FHIR Core stating that:

> If a server receives a `POST` create request that includes an
> `If-None-Exist` header **and** the server does not declare support
> for the conditional-create interaction in its `CapabilityStatement`
> (i.e. `CapabilityStatement.rest.resource.conditionalCreate = false`
> or absent), the server **SHALL** reject the request with HTTP
> `412 Precondition Failed` (or `400 Bad Request` if the header is
> malformed) and **SHALL NOT** silently process the request as an
> unconditional create. The response **SHOULD** include an
> `OperationOutcome` whose `issue.code = not-supported` and whose
> `diagnostics` names the unsupported `If-None-Exist` header.

This mirrors the existing rule for unrecognized `Prefer` and `If-Match`
preconditions: when a precondition the client asked the server to
honor cannot be honored, the safe default is failure rather than
silent demotion. The change is editorial-plus-normative in the same
file FHIR-53546 just touched; no schema or resource changes are
required.

#### Justification

The client's intent in sending `If-None-Exist` is unambiguous: "do not
create a duplicate." Silently ignoring the header converts the request
into something semantically different from what the client asked for,
and in clinical contexts (Medication, Patient matching, Immunization
records) that demotion can cause patient-safety harm. RFC 7232
precedents (`If-Match`, `If-None-Match`) require servers to *honor or
fail* a conditional header, never to ignore it. Aligning HL7's
custom-but-analogous `If-None-Exist` with the same behavior keeps the
mental model consistent and gives clients a deterministic signal
without forcing them to pre-fetch the `CapabilityStatement` on every
request. It also dovetails with the dragon warning added in
FHIR-53546: 53546 says "even when supported, beware of races"; 53867
adds "when not supported, fail loudly."

---

### Disposition B: Alternative Approach

#### Proposal

Make the behavior **explicit but configurable**, mirroring how the
spec already treats other capability-driven behaviors. Add language
that:

1. By default, a server that does **not** support conditional create
   **SHOULD** reject a `Create` request bearing `If-None-Exist` with
   `412 Precondition Failed` plus an `OperationOutcome` (`code =
   not-supported`).
2. A server **MAY** instead silently process the request as an
   unconditional create *only if* this fall-through behavior is
   advertised in `CapabilityStatement` — for example via a new
   `CapabilityStatement.rest.resource.extension`
   `if-none-exist-fallback` (boolean) defined in the core
   extensions pack.
3. Clients **SHALL** treat absence of advertised conditional-create
   support as "expect rejection" unless the fallback extension is
   present.

The proposal is otherwise editorial in `http.html#ccreate` plus one
new core extension definition.

#### Justification

This preserves backward compatibility for servers that have, in
practice, been silently ignoring the header for years (a behavior the
Zulip "Duplicate Patients" thread suggests is widespread) while still
making the *interoperable* default the safe one. It also gives
clinical clients a single, machine-readable place to discover which
behavior to expect, rather than relying on probing. The trade-off vs.
A is implementation cost: a new extension, plus discovery logic in
clients. A also has the advantage of being simpler to test for
conformance.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive**, with the rationale that the existing
`CapabilityStatement.rest.resource.conditionalCreate` field is
already the normative mechanism by which clients determine whether
conditional create is supported, and that server behavior on
unrecognized HTTP headers is governed by general HTTP semantics
(servers MAY ignore headers they do not understand). A FHIR-specific
rule risks contradicting deployed implementations and creating new
conformance debt without addressing a demonstrated interoperability
failure.

#### Justification

The reporter has not cited a specific implementation that has been
harmed; the request is preventive. RFC 7230 §3.2.4 already permits
servers to ignore unrecognized headers, and clients that care about
duplicate prevention are already expected to inspect the
`CapabilityStatement`. Adding a new normative rule could break
existing servers that have shipped with the silent-ignore behavior
since DSTU2. If clarification is needed at all, it could be confined
to a non-normative editorial note rather than a new SHALL.

---

### Recommendation

**Recommended disposition:** A

The reporter's framing is correct: `If-None-Exist` is a *precondition*,
not a hint. Every other HTTP precondition header in widespread use
(`If-Match`, `If-None-Match`, `If-Unmodified-Since`) is required to be
either honored or to fail the request — silent demotion is not a
permitted option, precisely because demotion changes the semantics of
the call and undermines the safety guarantee the client was asking
for. Because HL7 invented `If-None-Exist` (FHIR-9682), HL7 is free to
mandate the same "honor or fail" rule, and doing so aligns the FHIR
custom header with the well-understood HTTP precondition pattern that
implementers already know. The recently-applied FHIR-53546 has just
touched the same paragraph and established that safety warnings about
conditional create belong inline in the `http.html` section, so the
editorial overhead is minimal. Disposition B is attractive on
backward-compatibility grounds, but introducing a new core extension
to legitimize an ambiguous behavior is heavier machinery than the
problem warrants and pushes complexity onto every client. Disposition
C leaves a real patient-safety gap unaddressed for the sake of
preserving undocumented current practice, which is the wrong trade
for an Infrastructure-level normative spec.
