## Summary

FHIR-53959 ("Additional Functions cleanup") is a documentation/clarification
change request from Brian Postlethwaite against the FHIRPath "Additional
Functions" section of the FHIR core specification. It collects a set of small
edits to the descriptions of the FHIR-specific FHIRPath functions
(`getValue`, `hasValue`, `toString`, `resolve`, `ofType`) so that their
behaviour with FHIR primitives, extensions, contained resources, canonical
references, and subtype hierarchies is described unambiguously. Specifically
the ticket asks to (a) move `getValue` next to `hasValue()`; (b) clarify
that `toString()` is the standard FHIRPath function and explicitly state
that a primitive with no value (only `id`/extensions, e.g. a data-absent
reason) returns an empty collection; (c) note that `resolve()` can resolve
contained resources and discuss whether/how it resolves canonical (and
versioned canonical) references, plus a note that an unresolvable reference
due to permissions is indistinguishable from "not found"; (d) clarify that
`hasValue()` returns true on a FHIRPath primitive (which always has a
value) and that there is no exposed `value` property — callers must use
`getValue()`; (e) make `getValue()` return the value itself when applied
to a FHIRPath primitive; and (f) replace the misleading `ofType` example
that implies `markdown` is not a subclass of `string` with a better
example such as `canonical`.

The ticket is a direct follow-on to the broader restructuring proposed in
[FHIR-53958](https://jira.hl7.org/browse/FHIR-53958) ("Additional Functions
section cleanup", Persuasive), which splits the section into "Additional
Functions" and "Function Overrides" and removes content that has migrated
to base FHIRPath (e.g. `comparable`, `highBoundary`/`lowBoundary`).
FHIR-53959 fills in the per-function clarifications that belong in the
restructured section.

## Details

**Description:**

- Move `getValue` to be next to `hasValue()`.
- The `toString()` function should be noted that this IS a standard
  FHIRPath function, and the documentation here is only clarifying how the
  function will behave with FHIR types. Specifically it should explicitly
  state that if there is no value, only extensions, and/or `id` then it
  will return an empty collection (cf. FHIR-53958 "Create Function
  Overloads section") — e.g. if there is a data-absent-reason extension.
  *Could be mentioned in the examples when they are added for this
  function.* Be explicit that a FHIR primitive that has no value, and
  extensions only, returns an empty result/collection.
- `resolve()` should also mention that the reference could resolve from
  a contained resource as with normal resource references.
  - Should it also resolve canonical resource references? This should be
    mentioned here, and also that they could be versioned.
- `resolve()` should include a note that if a resource is not resolved
  due to permissions, this is no different to not finding the resource
  (is this needed?).
- `hasValue()` — include additional text to indicate that if the item
  is a FHIRPath primitive, then it can also return true if it has a value
  (which is always).
  - Also note that there is no `value` property exposed, though you can
    use `getValue()` instead — the `value` property is hidden/internal.
- The `getValue()` function should return the value itself when
  evaluated on a FHIRPath primitive too.
- `ofType` erroneously indicates that `markdown` is not a subclass of
  `string`. A better example would be `canonical`.

No comments have been recorded on the ticket.

## Proposed Dispositions

Note: the ticket bundles seven small, largely independent edits. All
three dispositions below treat the ticket as a single unit; in
practice the workgroup may wish to accept some bullets and decline
others, or split the ticket. Where helpful, the bullet-level intent
is called out explicitly.

### Disposition A: Accept As Requested

#### Proposal

Apply all seven edits to the FHIR Core "Additional Functions" /
"Function Overrides" section of `fhirpath.html` (after the FHIR-53958
restructure has landed). Concretely:

1. Re-order the page so the description of `getValue()` immediately
   follows `hasValue()` (both in the new "Additional Functions"
   subsection).
2. In the `toString()` description (which moves into "Function
   Overrides"), add an opening sentence that this is the standard
   FHIRPath `toString()` function and that the FHIR text only
   clarifies behaviour against FHIR types. Add a normative-style
   sentence: "When invoked on a FHIR primitive that has no `value`
   (i.e. only `id` and/or extensions are populated, e.g. a
   data-absent-reason extension), `toString()` returns an empty
   collection." Plan an example illustrating this (deferred to the
   examples work item).
3. In `resolve()`: add a paragraph explicitly stating that the
   reference can resolve to a contained resource using the same rules
   as the FHIR Reference data type. Add a paragraph clarifying
   behaviour for canonical and versioned canonical references — that
   `resolve()` will attempt to resolve a `canonical` value by URL
   (and, when present, by `|version`), with the resolution context
   being the same as for any canonical resource reference.
4. In `resolve()`: add a short note that an inability to resolve due
   to access control is, from FHIRPath's perspective, indistinguishable
   from "not found" — both yield an empty collection.
5. In `hasValue()`: clarify that when invoked on a FHIRPath primitive
   the function returns `true` (a FHIRPath primitive always has a
   value), and explicitly note that there is no exposed `value`
   property on FHIR primitives — callers should use `getValue()`; the
   underlying `value` is internal.
6. Update `getValue()` to specify that when evaluated on a FHIRPath
   primitive it returns the primitive value itself (i.e. it is a
   no-op for already-primitive inputs).
7. In `ofType`, replace the misleading example that implies
   `markdown` is not a subclass of `string` with an example using
   `canonical` (which is a subtype of `uri` and clearly demonstrates
   subtype-vs-base-type filtering without the markdown/string
   ambiguity).

#### Justification

Each bullet is a documentation-only clarification of existing,
implemented behaviour: it does not change conformance, change a
FHIRPath grammar/semantics rule, or introduce new wire content. The
edits remove genuine ambiguities that have generated implementer
questions (FHIR primitive vs FHIRPath primitive, `value` property
visibility, contained-resource resolution, canonical resolution,
markdown/string subtyping). The wording is consistent with how
mature FHIRPath engines (HAPI, .NET, Firely) already behave, and
aligns with the restructuring already approved as Persuasive in
FHIR-53958.

### Disposition B: Alternative Approach

#### Proposal

Accept the documentation clarifications, but split the ticket and
narrow two of the bullets:

- Apply bullets 1, 2, 5, 6, and 7 (re-order, `toString` empty-on-no-
  value, `hasValue` on FHIRPath primitives + missing `value`
  property, `getValue` on FHIRPath primitives, `ofType` example) as
  documentation-only edits exactly as in Disposition A.
- Split bullets 3 and 4 (the `resolve()` clarifications) into a
  separate ticket and treat them as **substantive**, not editorial:
  - For contained-resource resolution, state explicitly that
    `resolve()` MUST follow the same fragment / contained-resource
    rules as the Reference type, with a normative reference back to
    the Reference page rather than restating the rules.
  - For canonical resolution, do not silently expand `resolve()` —
    instead document the current behaviour (resolution of canonical
    URLs is server/context-dependent, and `|version` selects a
    specific version when present), and *only* commit to the
    behaviour after a Zulip / WG discussion confirms which engines
    already do this and which do not.
  - For the access-control bullet, add the empty-collection
    clarification only if the WG agrees the parenthetical "(is this
    needed?)" should be answered yes; otherwise omit, on the grounds
    that FHIRPath does not generally describe authorization.

#### Justification

Bullets 1, 2, 5, 6, and 7 are pure documentation cleanups with no
implementer impact and should be merged quickly. The two
`resolve()` bullets are different in character: clarifying how
`resolve()` interacts with canonical references (and especially
versioned canonicals) is *not* purely editorial — it is a
behavioural statement that some FHIRPath engines may not currently
implement. Bundling it with editorial cleanups risks either (a) the
whole ticket stalling on the canonical-reference question, or (b)
the canonical clarification being adopted without the broader
discussion it merits. Splitting preserves the easy wins and gives
the harder questions their own visibility, including a Zulip thread
the workgroup can point implementers to.

---

### Disposition C: Decline

#### Proposal

Close FHIR-53959 as "Not Persuasive" / "Not Persistent" and absorb
the bullets that are still relevant directly into the work for
FHIR-53958, dropping the rest. Rationale categories:

- Bullets 1, 2, 5, 6 are stylistic/restructuring follow-ups that
  belong inside FHIR-53958's restructure work and do not need a
  separate change request.
- Bullet 3's canonical-reference question is out of scope for FHIR
  Core and should be raised against the FHIRPath specification
  itself (or the IG publisher / canonical-resolution machinery)
  rather than addressed in a per-function note here.
- Bullet 4 (permissions) is out of scope for FHIRPath, which does
  not model authorization.
- Bullet 7 is debatable — `markdown` is in fact a `string`-derived
  type in FHIR, so the existing example is arguably *correct* about
  the distinction `ofType` is illustrating; replacing it risks
  introducing a new inconsistency.

#### Justification

The ticket is essentially a list of editorial nits piggy-backing on
an already-approved restructure (FHIR-53958). Tracking it as a
separate change request creates ballot/review overhead disproportionate
to the substance, and the one genuinely substantive item
(canonical-reference resolution) is better handled in the FHIRPath
spec where it would apply uniformly to every FHIRPath consumer, not
just FHIR Core.

---

### Recommendation

**Recommended disposition:** B

The cleanest path forward is Disposition B. Most of the edits in
FHIR-53959 are unambiguous documentation improvements that should be
applied as-is when the FHIR-53958 restructure lands — there is no
reason to make implementers wait while the workgroup debates them.
However, the two `resolve()` bullets — particularly the canonical /
versioned-canonical behaviour — describe behaviour that several
FHIRPath engines may not consistently implement, and quietly
documenting that behaviour as if it were settled risks creating
non-conformance overnight. Splitting those bullets into their own
ticket (and pinging the implementer community on Zulip) gives that
question the airtime it deserves without holding up the rest. This
is preferable to Disposition A (which under-weights the canonical
question) and to Disposition C (which throws away genuinely useful
clarifications about FHIR-vs-FHIRPath primitives, the hidden `value`
property, and the misleading `ofType` example).
