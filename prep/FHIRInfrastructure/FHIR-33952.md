## Summary

FHIR-33952 was opened on 2021-09-17 by Isaac Vetter and is, per the reporter's
own comment, an **accidental clone** of [FHIR-32323](https://jira.hl7.org/browse/FHIR-32323)
("Add Background: Trade-offs in App Architecture"). The cloned content
proposes adding non-normative "Best Practices" guidance to SMART App Launch
discussing the trade-offs between confidential and public client
architectures, the role of refresh tokens in enabling persistent access, and
how the choice of architecture should be driven by context rather than by
the need to work around missing refresh-token support.

The substantive request that this ticket carries was already accepted and
applied under the original [FHIR-32323](https://jira.hl7.org/browse/FHIR-32323)
(resolved **Persuasive**, status **Published**), with the change landed in
commit [`65cc644`](https://github.com/microsoft-healthcare-madison/smart-app-launch/commit/65cc64452eb1b54639657bf8d52df59d161e48ed)
on the SMART App Launch repository. The reporter explicitly stated that
this issue was opened in error and asked that it be closed; he noted he
lacked the permissions to do so himself. The ticket therefore exists only
as an administrative duplicate; there is no new requirement for the work
group to consider on the merits.

## Details

**Description:**

> The non-normative page on best practices should include the following
>
> ---
>
> **App and Server developers should consider trade-offs associated with confidential vs public app architectures:**
>
> 1. Persistent access is important for providing a seamless consumer
>    experience, and Refresh Tokens are the mechanism SMART App Launch
>    defines for enabling persistent access. If an app is ineligible for
>    refresh tokens, the developer is likely to seek other means of
>    achieving this (e.g., saving a user's password and simulating login;
>    or moving to a cloud-based architecture even though there's no use
>    case for managing data off-device).
>
> 1. Client architectures where data pass through or are stored in a
>    secure backend server (e.g., many confidential clients) can offer
>    more secure {refresh token :: client} binding, but are open to
>    certain attacks that purely-on-device apps are not subject to (e.g.,
>    cloud server becomes compromised and tokens/secrets leak). A breach
>    in this context can be widespread, across many users.
> 1. Client architectures where data are managed exclusively on end-user
>    devices (e.g., many public clients including most native apps today,
>    where an app is only registered once with a given EHR) are open to
>    certain attacks that confidential clients can avoid (e.g., a
>    malicious app on your device might steal tokens from a valid app, or
>    might impersonate a valid app). A breach in this context is more
>    likely to be isolated to a given user or device.
>
> --> The choice of app architecture should be based based on context.
> Apps that already need to manage data in the cloud should consider a
> confidential client architecture; apps that don't should consider a
> purely-on-device architecture. But this decision only works if refresh
> tokens are available in either case; otherwise, app developers will
> switch architectures just to be able to maintain persistent access,
> even if the overall security posture is diminished.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-33952 as a substantive content request and re-apply the
proposed "App and Server developers should consider trade-offs
associated with confidential vs public app architectures" prose to the
SMART App Launch non-normative **Best Practices** page (in the current
HL7/smart-app-launch IG repository), even though the same prose has
already been published via FHIR-32323.

#### Justification

The recorded ticket resolution is **Persuasive with Modification**, which
on its face implies a content change. Accepting the ticket as requested
would, however, simply re-litigate guidance that the work group already
accepted as **Persuasive** in FHIR-32323 and that has been **Published**
in the SMART App Launch best-practices page since 2021. There is no new
substantive request beyond the original; the only "modification" worth
considering would be cosmetic editorial polish that the editors can
handle independently of this ticket.

---

### Disposition B: Close as Duplicate of FHIR-32323

#### Proposal

Resolve FHIR-33952 as **Not Persuasive — Duplicate** (or the work group's
equivalent administrative disposition) with a pointer to
[FHIR-32323](https://jira.hl7.org/browse/FHIR-32323) and to the
implementing commit
[`65cc644`](https://github.com/microsoft-healthcare-madison/smart-app-launch/commit/65cc64452eb1b54639657bf8d52df59d161e48ed).
Update the resolution from "Persuasive with Modification" to a duplicate
status, link the two issues if not already linked, and move the ticket
to a closed state. No specification text change is needed.

#### Justification

The reporter, Isaac Vetter, explicitly recorded in the ticket's only
comment that the issue was opened in error as an accidental clone of
FHIR-32323 and asked that it be closed; he noted he did not have
permissions to close it himself. FHIR-32323 was already resolved
**Persuasive** and is **Published**, so the substantive request is
already in the spec. Closing FHIR-33952 as a duplicate honors the
reporter's stated intent, removes a misleading "Triaged" item from the
work-group queue, and corrects the resolution metadata — which currently
suggests an unapplied modification that does not in fact exist.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-33952 as **Not Persuasive** on its own merits, on the basis
that the requested guidance — even if novel — should not be added to the
SMART App Launch best-practices page in the form proposed.

#### Justification

This disposition is hard to defend on the substance, because the same
text was already accepted as **Persuasive** under FHIR-32323. Declining
FHIR-33952 outright would create an internally inconsistent record
(accepting the guidance under one ticket while declining identical
guidance under its clone) and would not address the reporter's actual
ask, which was for the ticket to be closed administratively, not for
the underlying guidance to be reconsidered. Use this disposition only if
the work group wants to formally reject the duplicate without invoking a
"duplicate" resolution category.

---

### Recommendation

**Recommended disposition:** B (Close as Duplicate of FHIR-32323).

The ticket is unambiguously an accidental clone: the reporter said so in
the only comment, the title is literally prefixed with "CLONE - ", the
description is byte-identical to FHIR-32323's, and FHIR-32323 itself is
already Published with the requested change applied in commit
[`65cc644`](https://github.com/microsoft-healthcare-madison/smart-app-launch/commit/65cc64452eb1b54639657bf8d52df59d161e48ed).
Disposition A would re-do work the work group already completed and
would risk producing duplicated or conflicting prose on the
best-practices page. Disposition C would muddy the historical record by
formally declining content that the work group already accepted under
the original ticket. Disposition B is the only option that aligns with
the reporter's stated request, with the existing published content, and
with sensible administrative hygiene; the work group should close
FHIR-33952 as a duplicate of FHIR-32323 and correct the lingering
"Persuasive with Modification" resolution that currently misrepresents
the ticket's actual state.
