## Summary

The ticket reports a technical error in the HL7 v2 mapping table for the
`HumanName` data type on the FHIR Core specification. Specifically,
`HumanName.period` is mapped to `XPN.13 + XPN.14`, but per the v2.5.1
definition of the XPN data type the relevant fields are `XPN.12` (Effective
Date) and `XPN.13` (Expiration Date). The reporter indicates the mapping
should therefore be corrected to `XPN.12 + XPN.13`. The reporter also
clarified in a follow-up comment that this submission was intended as a
Change Request accompanied by a negative ballot vote.

No related Jira tickets, Zulip discussions, or GitHub items were surfaced
through cross-reference, search, or content lookups for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the technical correction to the HL7 v2 mapping for `HumanName` in the
FHIR Core specification: change the `period` row from
`XPN.13 + XPN.14` to `XPN.12 + XPN.13`, matching the v2.5.1 definition
where `XPN.12` is "Effective Date" and `XPN.13` is "Expiration Date". The
fix is confined to the v2 mapping content for the `HumanName` datatype
(the `HumanName` mappings table rendered in the FHIR Core datatypes page
and any companion mappings file).

#### Justification

The reporter cites the normative v2.5.1 definition of the XPN data type
directly. `XPN.14` does not correspond to a date/time component in
v2.5.1, so the existing `XPN.13 + XPN.14` mapping is demonstrably
incorrect and cannot reflect a `Period`. Correcting to `XPN.12 + XPN.13`
restores semantic accuracy (Effective Date → `period.start`, Expiration
Date → `period.end`) and is consistent with how other v2 ↔ FHIR date-pair
mappings are expressed throughout the spec. The change is non-breaking
for FHIR implementers because no resource, element, cardinality, or
binding is affected — only documentation in the v2 mappings table —
which fits the "Technical Correction" issue type.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substantive correction (the existing mapping is wrong) but
broaden the fix:

1. Update `HumanName.period` to `XPN.12 + XPN.13` as in Disposition A.
2. While editing the table, audit the entire `HumanName → XPN` mapping
   row-by-row against v2.5.1 (and note any divergence for v2.7+ where
   XPN was extended), and correct any other off-by-one or stale field
   references in the same change.
3. Add a short footnote/comment in the mappings source noting the v2
   version basis used for the table, so future corrections have an
   explicit anchor.

#### Justification

The reported error is an off-by-one against the v2.5.1 field numbers,
which is a common failure mode in long-lived hand-maintained mapping
tables. Fixing only the one row leaves adjacent rows unverified and
risks a follow-up ticket for the next implementer who notices the same
class of issue. Doing the audit while the section is already open
amortises reviewer effort and produces a more durable correction.
Recording the v2 version basis in source mitigates recurrence without
introducing any normative change.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave the existing `XPN.13 + XPN.14` mapping in
place, on the grounds that v2 mapping tables in the FHIR specification
are informative guidance rather than normative bindings, and that the
reporter has not supplied a real-world interoperability problem caused
by the discrepancy.

#### Justification

This is the weakest of the three options and is offered for completeness.
It is defensible only if the workgroup considers the v2 mapping section
purely illustrative and not worth the editorial cycle for a single-row
fix. The argument is fragile because the mapping is verifiably wrong
against the cited v2.5.1 specification, and consumers reading the table
have no way to distinguish "approximate" from "incorrect"; declining
sustains misinformation in a normative-looking artefact.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified an unambiguous factual error against a
published HL7 v2 specification, and the fix is a one-line edit to a
documentation table with no impact on resource structure, conformance,
or implementer behaviour beyond clarifying intent. Disposition A
delivers exactly what the ticket asks for, matches the "Technical
Correction" issue type, and addresses the reporter's stated intent of a
ballot-blocking change request without expanding scope. Disposition B is
attractive and the workgroup may opt to fold in a quick audit of the
adjacent rows opportunistically, but that should not be a precondition
for accepting the fix; the audit can be tracked as a follow-up if it
proves larger than expected. Disposition C is not recommended: the
mapping is wrong on the face of the cited v2 definition, and leaving it
uncorrected undermines confidence in the v2 mapping content generally.
