## Summary

The reporter (Sudhansu Mishra) is asking the FHIR Infrastructure work group to
clarify what server behavior is implied by `notificationEvent` being marked
MustSupport on the R4 backport `SubscriptionStatus` profile that is returned
from the `$status` operation. Specifically, when a client invokes
`Subscription/[id]/$status` (or the type-level `Subscription/$status`), does
the server have to return *every* `notificationEvent` it has ever generated
for the subscription, or is some bounded subset (e.g., last N events, last
24 hours, only the most recent) acceptable? The ticket originated in the
Zulip `#subscriptions > PAS Subscription` thread, where Gino Canessa
confirmed there was never an intention that servers return all historical
events in `$status`, that `notificationEvent` is intended as *conditional*
MustSupport (server generates when appropriate, client processes when
appropriate), and that the parameters-slice MustSupport guidance appears to
have been accidentally omitted from the IG's MustSupport description. He
asked the reporter to file this ticket so the IG can be clarified. The
ticket has already been triaged with a "Persuasive with Modification"
resolution: add the SubscriptionStatus parameter slices to the MustSupport
section of the Conformance page with explicit "conditional MustSupport"
language, and add guidance to the `$status` operation page about
expectations for the number of events returned (declaring a specific count
out-of-scope for this guide and leaving it to implementer / downstream-IG
discretion). No related Jira tickets were identified via cross-references.

## Related Zulip Discussions

### [subscriptions > PAS Subscription](https://chat.fhir.org/#narrow/stream/subscriptions/topic/PAS%20Subscription/near/542371141)

Sudhansu Mishra opened the thread asking PAS-flavored questions about the
`$status` operation: whether `SubscriptionStatus.type` is always
`query-status` for `$status` responses, whether the payer has to expose a
separate endpoint per client, and whether `notificationEvent` being
MustSupport means the server must return every event ever generated for a
subscription. Gino Canessa first redirected PAS-specific questions to
`#Da Vinci PAS` but answered the FHIR-Infrastructure-shaped pieces inline:
servers are not required to expose client-specific endpoints (a single
`[base]/r4/Subscription/$status` is fine, client-discriminated paths are
also allowed); there was never an intent for `$status` to return all
historical events; and `notificationEvent` was meant as *conditional*
MustSupport (cardinality 0..*, generate/process when appropriate). He
observed that the IG's MustSupport guidance appears to have accidentally
omitted the SubscriptionStatus parameter-slices and asked Sudhansu to file
this Jira ticket so the IG can be clarified. He also pointed at the
`$events` operation as the proper way to retrieve event history with
filters. On the type-level `Subscription/$status` question, Gino confirmed
the expectation is that the server returns the subscriptions the client has
access to (authorization-scoped, optionally narrowed with the `status`
filter to active subscriptions) — analogous to a `/Patient` or
`/Observation` query — and that the R4 response is the Parameters
equivalent of the R5 searchset Bundle (Subscriptions Backport v1.1).
Sudhansu acknowledged and closed out the thread thanking Gino.

Participants: Sudhansu Mishra (reporter), Gino Canessa (FHIR Infrastructure
co-chair / Subscriptions WG steward).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution that is already recorded on the ticket
("Persuasive with Modification"), making two narrowly-scoped editorial
changes to the **Subscriptions R5 Backport IG**:

1. **Conformance / MustSupport page.** Add an explicit subsection covering
   the SubscriptionStatus parameter-slices (`status`, `type`, `eventsSinceSubscriptionStart`,
   `notificationEvent`, `subscription`, `topic`, `error`). For each slice,
   state the server-side and client-side MustSupport expectations. For
   `notificationEvent` specifically, document it as *conditional MustSupport*:
   the server MUST be capable of populating it when appropriate (e.g., when
   responding to `$status` for an active subscription where the server has
   chosen to convey recent event metadata) and the client MUST be capable of
   processing zero-to-many `notificationEvent` entries when present. Make it
   explicit that absence of `notificationEvent` in a `$status` response is
   conformant.
2. **`$status` OperationDefinition page** (`OperationDefinition-backport-subscription-status.html`).
   Add a "Number of events returned" guidance block clarifying that:
   - Specifying a particular number of `notificationEvent` entries is
     **out-of-scope** for the Subscriptions Backport IG.
   - Downstream IGs (e.g., Da Vinci PAS) MAY constrain this further based
     on their use cases.
   - In the absence of downstream constraints it is at server discretion;
     reasonable strategies include "most recent event", "events since last
     error", "events in a server-defined recent window", or "none".
   - Cross-link to the `$events` operation as the supported mechanism for
     retrieving event history with filters.
   Also note explicitly that `Subscription/$status` (type-level, no `id`)
   is authorization-scoped — the server returns the subscriptions the
   client has access to, optionally further narrowed by the `status`
   parameter — analogous to any other type-level FHIR interaction.

No profile, OperationDefinition machine-readable artifact, or cardinality
changes — this is documentation/narrative on the IG's Conformance and
operation pages.

#### Justification

This matches the language Gino Canessa already used in the originating
Zulip thread, and matches the "Persuasive with Modification" resolution
already entered on the ticket. The reporter's confusion is genuinely
caused by an editorial gap (the parameter-slice MustSupport expectations
were accidentally omitted from the Conformance prose), so the right fix is
to close that gap rather than to alter conformance semantics. Treating
`notificationEvent` as *conditional* MustSupport is consistent with how
FHIR Infrastructure has historically described 0..* MustSupport elements
on response payloads (capability, not obligation-to-populate) and avoids
imposing a costly "return all events ever" obligation on every server.
Declaring the specific event-count out-of-scope respects the layered-IG
architecture (PAS, Da Vinci-Alerts, etc. can pick numbers that suit their
use cases) without forcing the base backport IG to over-specify. The fix
is narrative-only, so it is low risk and does not invalidate any existing
implementations.

---

### Disposition B: Alternative Approach

#### Proposal

Take the same editorial actions as Disposition A, **plus** modify the
backport-subscription-status R4 profile (and the equivalent R4B profile)
to demote `notificationEvent` from MustSupport to *not* MustSupport, and
instead introduce a CapabilityStatement-level extension (or a
`subscription-status-event-strategy` value-set + extension on the
SubscriptionTopic CapabilityStatement) that lets a server advertise its
chosen event-inclusion strategy (`none` / `most-recent` / `since-error`
/ `windowed` / `all`). Clients would consult that advertised strategy
rather than infer behavior from MustSupport.

#### Justification

This addresses the reporter's underlying concern — uncertainty about what
to expect when invoking `$status` — by making server behavior
**discoverable** at runtime rather than something the implementer has to
figure out by reading IG prose. It also avoids the awkward "MustSupport
but not really" pattern that conditional-MustSupport language always
generates. The trade-off is significant: this is a normative change to
the profile (downgrading MustSupport is technically a backwards-compatible
relaxation, but tooling that drives compliance off MustSupport flags will
behave differently), it adds a brand-new extension that downstream IGs
have to learn, and it widens the scope of a ticket that the reporter
framed as a clarification request. It also lands during the maintenance
phase of an already-published IG, where the bar for normative changes is
high.

---

### Disposition C: Decline

#### Proposal

Reject the change request as **already addressed by existing mechanisms**
and close as "Not Persuasive". Rationale: (a) the FHIR core definition of
MustSupport already permits 0..* MustSupport elements to be absent; (b)
the `$events` operation already exists explicitly to retrieve event
history with filters; (c) downstream IGs (PAS et al.) are the right
venue to specify the number of events that must accompany a `$status`
response in a given use case; and (d) the reporter received a complete
answer in the Zulip thread.

#### Justification

A pure-decline disposition is defensible because the underlying FHIR
semantics are already correct: nothing in the spec or the backport IG
actually requires servers to return all historical events in `$status`,
and the `$events` operation is designed for exactly the
event-history-retrieval use case. The decline keeps the IG narrative
unchanged, which is the lowest-risk option in maintenance mode.

---

### Recommendation

**Recommended disposition:** A

The Zulip thread itself is direct evidence that the current IG narrative
is *not* sufficient — a careful implementer reading
`StructureDefinition-backport-subscription-status-r4.html` arrived at the
wrong default ("server has to return everything ever") because the
Conformance / MustSupport page does not spell out what the parameter
slices' MustSupport expectations actually are, and the `$status`
OperationDefinition page is silent on event-count expectations. That gap
is exactly what FHIR-53032 asks the work group to close, and it is what
the resolution on the ticket already commits to. Disposition C ignores
that documented confusion and effectively bets that the next implementer
will read more carefully — an unsafe bet for a published IG. Disposition
B *would* improve the long-term story, but it converts a small editorial
fix into a normative profile change plus a new capability extension during
the IG's maintenance window — disproportionate to the problem and
disruptive to existing implementations and downstream guides (PAS in
particular). Disposition A delivers exactly what the reporter asked for,
matches the steward's already-stated answer, costs only IG-narrative
edits, and leaves the door open for Disposition-B-style refinements in a
future major version of the backport IG if event-strategy discoverability
becomes a felt need across multiple downstream guides.
