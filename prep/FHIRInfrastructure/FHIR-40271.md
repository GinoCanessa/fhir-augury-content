## Summary

The ticket reports that the `is()` examples on the FHIR FHIRPath page are
misleading because they apply `is()` to non-singleton inputs. Per the
FHIRPath specification, `is()` (like the `is`/`as` operators) throws an
error when its input collection contains more than one item, so the
worked example `Patient.name.given.is(FHIR.string)` will blow up in
practice on any patient with multiple given names. Grahame proposes that
the examples be rewritten to use singleton inputs (e.g. `family` instead
of `given`). A proposed resolution has been recorded on the ticket
("Not Persuasive with Modification") arguing that the page should
instead steer authors toward `ofType()` — which is the
collection-friendly replacement for `is()`/`as()` — and that any
remaining `is()` examples that perform a boolean type check should be
wrapped inside a `select()`/`where()` so the example is not itself a
demonstration of bad practice.

There are no other Jira tickets cross-referenced from FHIR-40271, but
the same issue is discussed at length in the
[#fhirpath](https://chat.fhir.org/#narrow/stream/fhirpath/topic/dom-3%2C%20descendants%28%29%20for%20%25resource%2C%20as%20type%20specifier)
thread that triggered the ticket, and it has been bundled (alongside
FHIR-55637, FHIR-55417, FHIR-55495, FHIR-37861, FHIR-54601, FHIR-53958,
FHIR-36040 and FHIR-53949) for a possible block vote on the FHIR-I call
of 2026-04-13.

## Details

**Description:**

> The is() examples on the FHIRPath page are misleading because they
> use non-singleton inputs, and so they are likely to blow up in
> practice. The examples should use singleton inputs (family in this
> case)

**Resolution Description (proposed):**

> While this request is correct, the **is()** function is present in
> FHIRPath for legacy support, we should be encouraging the use of the
> **ofType()** function, which is the plug-in replacement for `is()`
> that DOES handle collections.
>
> This should be changed for all uses of the `is()` function on the
> page (including `Patient.is(FHIR.Patient)` at the very start of the
> section).
>
> The examples that are using `is()` to perform a boolean type check,
> not a cast (or example of using the type style parameter) should be
> corrected to be inside a `select` so that errors don't come from
> having collections — don't show examples of bad practice.

**Comments:**

- **brian.postlethwaite (2023-01-18):** Suggests an alternative
  rewrite, `Patient.name.given.where($this.is(FHIR.string))`, to make
  the singleton-vs-collection trap explicit in any retained example.

Paul, Grahame concedes he had read the spec wrong in the Java

spec-conformant single-item semantics and just fix the misleading

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FHIR Core specification's FHIRPath page (`fhirpath.html`), in
the "Polymorphism in FHIR" / `is()` discussion, replace the
`Patient.name.given.is(FHIR.string)` style examples with
singleton-input examples (e.g. `Patient.name.family.is(FHIR.string)`,
or `Patient.name.first().given.first().is(FHIR.string)` where a
plural element must be retained). Make the same replacement for the
`Patient.is(FHIR.Patient)` example at the very start of the section —
either keep it as-is (a Patient resource is already a singleton) or
add a comment that the input must be a singleton. Do not touch the
normative definition of `is()`/`as()`/`ofType()`; this is purely an
editorial fix to the examples.

#### Justification

This is exactly what the reporter asked for and is the lowest-risk
change. It removes the demonstrably wrong examples without revisiting
the long-standing `as`/`is` vs `ofType` semantic debate (which the
Zulip thread shows is unresolved — Paul Lynch raised it, Grahame
explicitly declined it, and Brian agreed). It is consistent with the
FHIRPath specification's documented behaviour ("If there is more than
one item in the input collection, the evaluator will throw an error")
and with Grahame's own Java engine update plus the FHIRPath test cases
he added in
[FHIR/fhir-test-cases@84a07a6](https://github.com/FHIR/fhir-test-cases/commit/84a07a6a92bedd18072c1cb7f0da11fe693f7191).
It is small enough to bundle with the other editorial FHIRPath fixes
in Brian's proposed FHIR-I block vote.

---

### Disposition B: Alternative Approach (recommended by the resolution text)

#### Proposal

Adopt the resolution already drafted on the ticket: rewrite the FHIR
FHIRPath page's examples to **prefer `ofType()`** over `is()`/`as()`
wherever a cast is shown (so `Patient.name.given.ofType(FHIR.string)`
instead of `Patient.name.given.is(FHIR.string)`). Where an example
genuinely needs the boolean `is()` form (a true type-test, not a
cast), wrap it in `select()` or `where()` so the input to `is()` is a
singleton — e.g. Brian's
`Patient.name.given.where($this.is(FHIR.string))`. Treat
`Patient.is(FHIR.Patient)` the same way (single-item input is fine,
but call out the constraint inline). Do **not** redefine `is()`/`as()`
to operate over collections — keep the FHIRPath-spec semantics
intact, just steer the documentation away from the pitfall.

#### Justification

This addresses the same underlying problem as Disposition A but also
nudges implementers toward the idiom that actually scales —
`ofType()` — which is the FHIRPath construct designed for collection
inputs. It matches the resolution already drafted on the ticket
("Not Persuasive with Modification") and matches the position
Grahame and Brian converged on in the
[#fhirpath dom-3 thread](https://chat.fhir.org/#narrow/stream/fhirpath/topic/dom-3%2C%20descendants%28%29%20for%20%25resource%2C%20as%20type%20specifier/near/322132357):
keep `is()`/`as()` single-item, fix the docs to stop modelling bad
practice, and make the page educational about why the pattern exists.
It also keeps FHIR aligned with R5's `ofType()` rewrite of the dom-3
invariant, which is the same fix already applied at the invariant
level.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" on the basis that the FHIRPath
specification already states that `is()` errors on multi-item input,
that the example on the FHIR page is documenting the FHIRPath
function (not endorsing the call site), and that the issue is purely
cosmetic.

#### Justification

A purely defensive position: implementers who care about correctness
read the FHIRPath spec, and the example on the FHIR page is short
enough that the singleton constraint is implicit. Declining avoids
having to re-touch the FHIRPath section as part of FHIR-I's editorial
sweep. This is a weak justification — the Zulip thread shows that even
spec authors (Grahame for the Java engine; the dom-3 invariant in
shipped resources) got this wrong — so this disposition is offered
mainly for completeness.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the resolution already drafted on the ticket and is
the position the FHIRPath subject-matter experts converged on in the
related Zulip thread. It does the editorial fix the reporter asked
for (Disposition A), but it also moves the page's examples onto
`ofType()` — the construct that does handle collections — so readers
copy-pasting from the spec stop reproducing the dom-3-style mistake
that triggered this ticket in the first place. It is in scope for the
proposed FHIR-I block vote on 2026-04-13, has no normative impact on
the FHIRPath specification itself, and Grahame has already
proxy-voted in favour. Disposition C should be rejected: the ticket
exists precisely because the misleading examples are causing real
implementer confusion; declining would leave the trap in place.
