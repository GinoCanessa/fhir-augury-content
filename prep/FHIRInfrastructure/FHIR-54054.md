## Summary

The ticket reports a typo in the worked example that the FHIR Core
specification uses to illustrate the `_include:iterate` modifier on the
search page. The example is currently written as:

```
GET [base]/MedicationDispense?_include=Medication:prescription
     &_include:iterate=MedicationRequest:requester&criteria...
```

The first `_include` parameter should reference the resource being
searched (`MedicationDispense`), not `Medication`. `Medication` does not
have a search parameter named `prescription`; only `MedicationDispense`
does. The reporter therefore asks that the example be corrected to
`_include=MedicationDispense:prescription`. No cross-referenced Jira
tickets were returned by Augury, so this is a stand-alone technical
correction against the spec text.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a pure documentation/technical correction against
`search.html` in the FHIR Core specification. In the worked example
illustrating `_include:iterate` (the section that R5 publishes at
`hl7.org/fhir/R5/search.html#_revinclude`), change the first include
parameter from `_include=Medication:prescription` to
`_include=MedicationDispense:prescription` so that the example reads:

```
GET [base]/MedicationDispense?_include=MedicationDispense:prescription
     &_include:iterate=MedicationRequest:requester&criteria...
```

No other text in the example needs to change; the iterated include
(`MedicationRequest:requester`) is unaffected. The fix should be
applied in the current build (so it flows into the next R5 technical
correction publication and into R6).

#### Justification

`MedicationDispense` defines a `prescription` search parameter that
references `MedicationRequest`; `Medication` does not. As written, the
example is internally inconsistent: the iterated include only makes
sense if the first include actually pulls `MedicationRequest` instances
into the bundle, which requires `MedicationDispense:prescription`.
René Spronk already confirmed the typo on Zulip and asked for a Jira
ticket; this ticket is that ticket. As a Technical Correction against a
narrative example, the fix carries no normative impact, no schema
change, and no implementer burden.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the correction, but at the same time replace the brittle
abbreviated form with a fully-qualified, runnable example and add a
short sentence above it explaining why the source resource of the first
`_include` must be the resource being searched (i.e., the resource on
the left of `?`). For example:

```
# Return MedicationDispense resources, also include the referenced
# MedicationRequest (via MedicationDispense.prescription), and then
# iterate to also include the requester referenced from each included
# MedicationRequest.
GET [base]/MedicationDispense?_include=MedicationDispense:prescription
     &_include:iterate=MedicationRequest:requester
```

Optionally cross-link the example to the `_include` definition table so
readers can verify the source resource / search-parameter pairing.

#### Justification

The Zulip history (`About _include:iterate`,
`_include:recurse (_include:iterate) ambiguity`,
`Iterative include and revinclude`, `Multi-server environments`) shows
that this exact `MedicationDispense` / `MedicationRequest` example is
the one the community keeps reaching for when reasoning about iterative
includes, and that confusion about which side of the colon names which
resource is recurring. A small explanatory wrapper around the corrected
example would prevent future tickets of this same shape, at very low
editorial cost. The downside is that it is slightly broader than a
strict technical correction, so it may need a second pair of eyes from
FHIR-I before merging.

---

### Disposition C: Decline

#### Proposal

Close as `Not Persuasive` / `Won't Fix` on the grounds that the example
is non-normative and the syntax of `_include` is otherwise fully
defined elsewhere on the same page.

#### Justification

This is the weakest option. The text is objectively wrong (the named
search parameter does not exist on the named resource), Zulip already
reached consensus that it is a bug, and the cost of fixing a single
token in an example is negligible. Declining would leave a verifiably
incorrect example in the published spec, which is exactly the failure
mode the Technical Correction track exists to catch.

---

### Recommendation

**Recommended disposition:** A

The reported defect is unambiguous, externally confirmed on Zulip by a
FHIR-I-aligned voice (René Spronk), and isolated to a single token in a
non-normative example. Accepting As Requested fixes the bug with zero
risk to implementers and zero impact on the normative content of
`search.html`. Disposition B is attractive and could be folded in as a
follow-up editorial pass, but it should not block landing the
correction itself; if the workgroup wants the broader cleanup, the
cleanest path is to apply Disposition A now and open a separate
editorial ticket for the surrounding prose. Declining (Disposition C)
is not defensible given the Zulip evidence.
