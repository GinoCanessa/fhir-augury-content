## Summary

Nicolas Griffon is asking the SDC IG to provide guidance on how to perform
Questionnaire data extraction when a `QuestionnaireResponse` (QR) is revised
after it has already been validated and extracted (i.e., the QR moves to
`status = amended`). In that situation the extractor must locate the
resources that were previously created from the QR — without knowing their
ids — so they can be updated or deleted to reflect the revised answers, and
new resources created for any newly-added answers.

The reporter's current workaround uses the `itemExtractionContext` extension
with a `valueExpression` that issues a FHIR query keyed off `Provenance`, for
example
`Observation?_has:Provenance:target:entity={{'QuestionnaireResponse/' + %QuestionnaireResponseId}}&encounter={{'Encounter/' + %encounter.id}}&patient={{'Patient/' + %patient.id}}&code=http://loinc.org|55281-0`.
That pattern depends on the EHR (a) creating Provenance for extracted
resources and (b) supporting `_has` chained search — neither of which is
guaranteed.

The ticket originated from the `#questionnaire` Zulip thread "Extraction
after QuestionnaireResponse revision" where Lloyd McKenzie agreed the case
is messy (some resource types can be re-found by code, e.g.
`AllergyIntolerance`/`Condition`; others such as `Observation` need
`derivedFrom` or `_has:Provenance`) and explicitly asked the reporter to
file a change request so the spec can at least document guidance. Brian
Postlethwaite raised related questions about whether Provenance should be
included in the extract transaction bundle and whether a separate
"data-source" extension/search parameter on extracted resources would help.
The ticket was originally voted "Considered for Future Use" on 2024-11-21
(7-0-0, Beda/Postlethwaite) on the grounds that all four extraction
mechanisms could in principle support it but each would behave differently
and the WG did not want to publish untested guidance before STU4 ballot. On
2026-03-26 Lynn Laakso reverted that resolution and re-opened the ticket
post STU4 publication.

The cross-reference index surfaces no related Jira tickets and no GitHub
items; only the originating Zulip thread is linked.

## Details

**Description:**

> Use case :
>
> There is big questionnaire in my information system.
>
> For some item of these questionnaire I define some definition based form
> data extraction.
>
> The rule is to await that the questionnaire response is fully validated
> before launching the data extraction.
>
> However we all know that sometimes, a validated questionnaire response is
> changed after validation (status = ammended)
>
> In this case we have to found the resources extracted, without knowing
> their id, to change them.
>
> Solution used :
>
> We set the `itemExtractionContext` extension with a fhir query
> `valueExpression`. The query looks in provenance resources to find any
> resource created from the QR.
>
> If there is one or many answer they are updated/deleted depending on the
> final (sic) content of the QR.
>
> If there is no answer new resources are created (of the type querried in
> the fhir query).
>
> Example of valueExpression :
>
> `Observation?_has:Provenance:target:entity={{'QuestionnaireResponse/'`
> `+ %QuestionnaireResponseId}}&encounter={{'Encounter/' + %encounter.id}}`
> `&patient={{'Patient/' + %patient.id}}&code=http://loinc.org|55281-0`

### [questionnaire > Extraction after QuestionnaireResponse revision](https://chat.fhir.org/#narrow/stream/questionnaire/topic/Extraction%20after%20QuestionnaireResponse%20revision)

Nicolas Griffon described the use case (large Questionnaire, definition-based
extraction, QR later revised to `amended`) and shared the
`_has:Provenance:target:entity=…` query he uses to locate previously-extracted
`Observation` resources by QR id, encounter, patient, and LOINC code. Lloyd
McKenzie agreed the scenario is messy: for some resources (e.g.
`AllergyIntolerance`, `Condition`) you can re-find the existing instance by
code, but for `Observation` and similar you generally need either
`Observation.derivedFrom` pointing to the QR, or a `_has` chained search via
`Provenance`. He noted both depend on the EHR — Provenance has to be created
during extraction and `_has` is far from universally supported — and asked
Nicolas to open a change request so the spec can at least give guidance.
Brian Postlethwaite asked whether the Provenance resources should be
included in the extraction transaction bundle, floated (and then largely
walked back) the idea of a generic "data-source" extension plus search
parameter on extracted resources, and reported he had a draft definition-based
extract implementation on his test server (`fhir.forms-lab.com`) that he
intended to exercise at the connectathon, calling out edge cases around
fixed/pattern values, `min=1` properties, and collections along the
definition path. No formal consensus was reached; the participants converged
on "we should at least document guidance" rather than on a single
mechanism.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new "Re-extraction of revised QuestionnaireResponses" section to the
SDC IG Data Extraction page that:

1. States the problem: when a previously-extracted QR moves to `amended` (or
   any subsequent status change that alters answers), implementers SHOULD
   detect and reconcile previously-extracted resources rather than blindly
   creating duplicates.
2. Documents a recommended pattern per extraction mechanism (definition-based,
   template-based, observation-based, StructureMap-based), describing how
   each can identify previously-extracted resources. For definition-based
   (the mechanism the reporter is using), describe the
   `itemExtractionContext` `valueExpression` pattern with both options:
   - **`derivedFrom` form** where the target resource type carries a
     reference back to the QR (e.g. `Observation.derivedFrom`,
     `Procedure.basedOn`).
   - **`_has:Provenance` form** when the target resource type has no
     suitable reverse reference, with the explicit prerequisite that the
     extractor MUST also create a `Provenance` whose `Provenance.target`
     points to the extracted resource and whose `Provenance.entity.what`
     references the QR (with `entity.role = source`).
3. Adds a normative SHOULD that extractors include those `Provenance`
   resources in the same transaction Bundle as the extracted resources, so
   that the `_has` query is well-defined immediately after extraction.
4. Adds an implementation note that `_has` is not universally supported and
   that systems unable to rely on it SHOULD prefer the `derivedFrom`-style
   pattern or a domain-specific code/identifier match.
5. Adds caveats called out by Lynn Laakso's revert comment: derivations
   don't necessarily point to a single question, and which fields may
   legitimately change on `amended` is implementation-defined.

#### Justification

This is exactly what the reporter and Lloyd McKenzie agreed on Zulip the
spec is missing — guidance, not a new resource or extension. It does not
require any new conformance artifacts and respects the WG's prior concern
that they did not want to publish guidance before STU4 had been tested:
post-publication is now the right window. Naming `Provenance` as the
recommended linking mechanism is consistent with FHIR's existing pattern
for extraction lineage and avoids inventing yet another extension. The
gotcha called out in the revert comment ("derivations don't necessarily
point to a single question") is addressed by framing the guidance as
SHOULD-level patterns rather than a single mandated query template.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) prose guidance, define a small, mechanism-
neutral SDC extension on extracted resources — e.g.
`http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-extracted-from` — whose
value is a `Reference(QuestionnaireResponse)` plus an optional `linkId`
pointing at the source item, together with a companion search parameter
`extracted-from` (token or reference). Update the SDC extraction guidance
so each of the four extraction mechanisms either populates that extension
on the resources they create or, when the target resource type already has
a suitable element (e.g. `Observation.derivedFrom`), prefers that element
and otherwise falls back to the new extension. Re-extraction then becomes
a uniform query of the form
`Observation?extracted-from=QuestionnaireResponse/{id}` (optionally
`&extracted-from-linkId={linkId}`) regardless of resource type and without
requiring `_has` or Provenance support.

#### Justification

This addresses the underlying interoperability problem (extractors today
have no portable way to find their previous output) rather than only
papering over it with prose. It removes the dependency on `_has` and on
EHRs creating Provenance — both of which Lloyd flagged as fragile — and
gives a single recipe that works across all four extraction mechanisms,
which directly answers the WG's earlier concern that documenting four
different patterns would be confusing. It is the idea Brian Postlethwaite
floated on Zulip ("another extension … to indicate their data source (and
search parameter)"); he tentatively walked it back, but it deserves a
formal vote because it is the only option that works without server
features outside the SDC implementer's control. The trade-off vs.
Disposition A is scope: this is an IG-level normative addition (new
extension + search parameter) that consumers of extracted resources have
to opt into to gain the benefit, whereas A is purely guidance and can ship
in a smaller errata-style update.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — already addressable with existing FHIR
mechanisms". The committee's position would be that implementers can
already solve this on a case-by-case basis using `Observation.derivedFrom`
(and the equivalent reverse-reference elements on other resource types),
search-by-code matching for resources with intrinsically unique codes
(`AllergyIntolerance`, `Condition`), or `_has:Provenance` queries when
Provenance is created during extraction. Re-extraction semantics are
fundamentally a workflow concern that depends on what changes are allowed
on `amended` QRs, and that varies enough by deployment that the SDC IG
should not standardize a single approach.

#### Justification

This mirrors the substance of the original 2024-11-21 vote ("Considered
for Future Use", 7-0-0): the WG concluded all four extraction mechanisms
could in principle support re-extraction but each would behave differently
and the spec should not document untested guidance. If, post-STU4, the WG
still does not have implementation experience to back a recommendation,
declining and pointing implementers at the existing primitives is more
honest than publishing aspirational guidance. The downside is that the
ticket comes back the next time someone hits the case, and the
`#questionnaire` thread shows that it does come up.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly delivers what the originating Zulip discussion
asked for — documented guidance the spec currently lacks — and is the
cheapest concrete next step now that the ticket has been re-opened post
STU4 publication. It explicitly accommodates the gotcha the WG raised on
revert (derivations may not map 1:1 to a single question) by framing the
recommendations as SHOULD-level patterns rather than a single query
template, and it sets up the connectathon work Brian Postlethwaite
described to validate each mechanism's recipe before any normative tightening.
Disposition B is the right long-term answer if the WG concludes that
prose guidance is insufficient and that a portable, mechanism-neutral
extension is needed; it can be raised as a follow-up ticket once
Disposition A's guidance has been exercised at a connectathon. Disposition
C re-applies the original "Considered for Future Use" rationale but the
re-open comment from Lynn Laakso explicitly signals the WG no longer
considers that the right outcome, so declining now would be inconsistent
with the WG's most recent action on the ticket.
