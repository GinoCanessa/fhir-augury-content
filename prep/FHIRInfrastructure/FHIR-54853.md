## Summary

This ticket asks the FHIR Infrastructure work group to finalize the
behavior of the `:not` search modifier when combined with `,`-joined
("OR") parameter values. The current `search.html` page contains a
"dragon" call-out that flags `gender:not=male,female` as ambiguous:
under the spec's stated rule that modifiers apply per-value, the query
expands to the union of "patients whose gender is not male" and
"patients whose gender is not female", which collapses to "all
patients" for any scalar element — a clearly unintended result. The
text currently asks for implementer feedback and recommends `_filter`
or named queries as a workaround. The reporter requests that the
work group either (a) define normative semantics for `:not` with
OR-joined values, or (b) prohibit that combination, and in either
case remove the placeholder feedback text.

No cross-referenced Jira tickets, GitHub items, or Confluence pages
were surfaced for FHIR-54853 by the cross-reference index. However,
substantial implementer discussion exists on Zulip
([`#implementers > :not modifier`](https://chat.fhir.org/#narrow/stream/implementers/topic/%3Anot%20modifier)
and
[`#implementers > token search with :not modifier`](https://chat.fhir.org/#narrow/stream/implementers/topic/token%20search%20with%20%3Anot%20modifier)),
and that thread references an earlier predecessor ticket
**FHIR-40629** opened by Paul Lynch in March 2023 to track exactly
this question. FHIR-40629 is not present in the local Jira cache; the
work group should retrieve it from Jira directly to confirm prior
disposition history before resolving FHIR-54853.

## Related Zulip Discussions

### [`#implementers > :not modifier`](https://chat.fhir.org/#narrow/stream/implementers/topic/%3Anot%20modifier)

A long-running thread (2016 → 2023, 50+ messages) that is the canonical
implementer discussion of `:not` semantics. The 2016 portion
(Corina Ciocanea, Grahame Grieve, Bryn Rhodes, Stephen Royce)
established that `gender:not=male` includes resources with no
`gender` element ("safer to return the nulls — the client can filter
them out", per Grahame). The 2022 portion (Simone Heckmann, René
Spronk, Stefan Lang, Josh Mandel) re-opened the question for
OR-joined values: Heckmann observed that HAPI and Firely both
implement `gender:not=male,female` as "neither male nor female"
(logical AND of the per-value negations) even though the literal
spec rule (modifier applied per value, results unioned) would
return all records. Spronk, Lang, and Mandel argued the De-Morgan
("AND of negations") interpretation is what implementers expect, and
that it should be aligned with `:not-in=<valueset>`. The 2023
portion (Paul Lynch, Gino Canessa, René Spronk, John Silva, Josh
Mandel) confirmed HAPI's behavior, opened FHIR-40629, and noted that
the per-value rule was clarified for R5 but the `:not` corner case
was deliberately left open with a "dragon" because there was no time
to gather implementer feedback. Mandel's closing point is important:
adopting the De-Morgan special case for `,`-joined `:not` adds **no**
new expressive power (`code:not=A&code:not=B` already expresses
"neither A nor B"), but it **removes** the ability to express "any
resource that does not have code A or does not have code B" via the
shorthand. No consensus was reached.

Participants: Corina Ciocanea, Grahame Grieve, Bryn Rhodes, Stephen
Royce, Simone Heckmann, René Spronk, Stefan Lang, Josh Mandel, Paul
Lynch, Gino Canessa, Sujeito de Sorte, John Silva.

### [`#implementers > token search with :not modifier`](https://chat.fhir.org/#narrow/stream/implementers/topic/token%20search%20with%20%3Anot%20modifier)

A shorter 2022 thread in which Yunwei Wang asks whether
`?token:not=A,B,C` and `?token:not=A&token:not=B&token:not=C` are
equivalent. Josh Mandel explains that the first matches "any resource
that does not include a value of A, a value of B, or a value of C"
(notably including resources with no value at all), while the second
would also return resources that have all three values present at
once for a repeating element. Mandel explicitly flags that De Morgan's
laws do **not** apply because FHIR search parameter values are not
boolean propositions — they may match repeating fields, so
`:not=A,B,C` is "any value distinct from any member of {A, B, C}".

Participants: Yunwei Wang, Josh Mandel.

### [`#implementers > :not modifier for uri`](https://chat.fhir.org/#narrow/stream/implementers/topic/%3Anot%20modifier%20for%20uri)

Brief 2022 exchange confirming that `:not` is only defined on
search parameters of type `token` (Gino Canessa). Tangentially
relevant: it bounds the scope of any FHIR-54853 fix to the token
search type.

## Proposed Dispositions

### Disposition A: Accept As Requested — define De-Morgan semantics for `:not` with OR-joined values

#### Proposal

Update `search.html` (in `HL7/fhir`, the `source/search.html`
authoring file) to make the following normative:

> When the `:not` modifier is applied to a search parameter with
> comma-separated ("OR") values, the modifier is applied to the
> **value set** as a whole rather than per value. That is, for a
> token search parameter `p` and values `v1,v2,…,vn`,
> `p:not=v1,v2,…,vn` matches resources for which **no** instance of
> `p` has a code in `{v1, v2, …, vn}`. Equivalently, this is the
> negation of `p=v1,v2,…,vn`.
>
> This is consistent with `:not-in=<valueset>` where the value set
> contains exactly `{v1, v2, …, vn}`, and matches the behavior
> implemented by HAPI and Firely Server today.

Remove the existing "dragon" paragraph that requests implementer
feedback and recommends `_filter` / Named Queries as a workaround.
Add a worked example for `gender:not=male,female` showing it returns
patients whose gender is `other`, `unknown`, or absent. Add a note
that `p:not=A&p:not=B` retains its existing per-parameter AND
semantics ("resources for which no instance of `p` is `A` and no
instance of `p` is `B`") and is therefore equivalent to
`p:not=A,B` only when there is no other distinction; the two forms
behave identically for scalar elements but can differ for repeating
elements (call this out explicitly).

#### Justification

This aligns the spec with the de facto implementation in the two
most prominent open-source FHIR servers (HAPI, Firely), aligns
`:not` with `:not-in` so implementers can reason about them
together, and matches the interpretation favored by named
implementers in the Zulip thread (Spronk, Lang, Mandel as a
"reasonable special case", Lynch). It removes the only currently
nonsensical form of the query (`gender:not=male,female` returning
all records). The work group already accepted the principle that
modifier semantics for the `:not`/value-set combination needed
follow-up after R5 (per Gino Canessa's 2023-03-20 comment), and
this disposition discharges that follow-up. The trade-off Mandel
raised — losing the literal "OR of per-value negations" reading —
is acceptable because that reading produces a tautology for scalar
elements and is essentially never what an implementer wants for
repeating elements either; where it is wanted, `_filter` already
expresses it precisely.

---

### Disposition B: Alternative Approach — prohibit `:not` with OR-joined values

#### Proposal

Update `search.html` to make `:not` with comma-separated values an
**error** that servers SHALL reject (HTTP 400 with an
`OperationOutcome`). The text would read approximately:

> The `:not` modifier SHALL NOT be combined with comma-separated
> ("OR") values on a single search parameter. Servers SHALL reject
> such requests with HTTP 400 and an `OperationOutcome` of severity
> `error`, code `invalid`. Clients that need to exclude multiple
> values SHALL either repeat the parameter
> (`p:not=A&p:not=B`, which excludes resources containing any of
> the listed values), use `:not-in` against a value set
> (`p:not-in=<valueset>`), or express the query with `_filter`.

Remove the "dragon" paragraph and replace it with a short rationale
pointing to the ambiguity and to the safer alternatives.

#### Justification

This is the most conservative reading of the existing spec: it
preserves the per-value modifier rule that R5 clarified for all
other modifiers, avoids any hidden behavior change for servers that
currently implement the literal (tautological) reading, and forces
clients to be explicit. It also avoids Mandel's concern about
silently losing the "OR of negations" reading by simply taking that
question off the table for the comma form. The trade-off is real:
clients lose a compact shorthand that two major server
implementations already accept, and existing client code that uses
the shorthand would start receiving 400s. For a feature that has
sat under a "dragon" since R5 with no documented in-the-wild
reliance on the literal reading, that breakage is bounded.

---

### Disposition C: Decline — leave the dragon in place / defer

#### Proposal

Mark FHIR-54853 as "Persuasive with Mod / Defer" and leave the
existing dragon paragraph in `search.html` unchanged for this
cycle, on the basis that:

- the predecessor ticket FHIR-40629 may already carry a disposition
  that this work group should honor (the local cache cannot confirm
  one way or the other);
- implementers continue to disagree on Zulip about whether the
  De-Morgan reading or the literal per-value reading is preferable;
- the workaround (`_filter`, repeated `:not` parameters,
  `:not-in=<valueset>`) covers all real use cases; and
- changing the semantics now risks a silent behavior change for
  any server that implemented the literal reading.

The ticket would be reopened only after FHIR-40629's history is
reviewed and a fresh implementer survey is run.

#### Justification

Defensible if the work group concludes that the cost of a wrong
choice (silent client-visible behavior change in either direction)
outweighs the cost of leaving a known ambiguity documented. It also
avoids stepping on whatever FHIR-40629 already decided. The
weakness of this disposition is that the reporter's central
complaint — "the dragon should be resolved" — is exactly what is
not addressed; the same ticket will simply reappear next cycle.

---

### Recommendation

**Recommended disposition: A**

Disposition A resolves the long-standing dragon in the way that
already matches the two leading server implementations and the
intuition of every implementer who weighed in on the Zulip thread,
while preserving an unambiguous fallback for the rare case where
the per-value-negation reading is genuinely wanted (`_filter`).
Disposition B is technically clean but introduces a hard breaking
change for HAPI and Firely clients with no compensating gain, and
Disposition C punts on a question the reporter has explicitly asked
the work group to close. Before publishing the change, the work
group should pull FHIR-40629 from Jira to confirm there is no
contradictory prior disposition, and the worked example for
`gender:not=male,female` (returning `other`, `unknown`, and
absent) should be reviewed by Paul Lynch and Josh Mandel given
their depth on the thread.
