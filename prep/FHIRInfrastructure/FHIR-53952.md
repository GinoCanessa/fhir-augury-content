## Summary

Brian Postlethwaite asks FHIR Infrastructure to clean up the FHIR-specific
variables section of the FHIRPath page. The section currently introduces
"two" environment variables but actually defines three (`%resource`,
`%rootResource`, and `%context`) and also mentions `%profile`, even though
`%profile` is only available in profile invariants and is not documented in
the variables section itself. Brian asks the WG to: (a) restructure so the
always-available variables are listed together and the context-specific
ones (including `%profile`, plus references to the subscription-topic
variables `%previous`/`%current` and the Questionnaire variables
`%questionnaire`/`%qItem`) are called out as a bulleted list of
purpose-specific additions; (b) replace the current code block with a table;
(c) add a block of concrete, annotated examples of how each variable is used
in an expression, with the context it applies to; and (d) move the detailed
`%profile` documentation (how it supports profile-invariant validation) off
the FHIRPath page and onto ElementDefinition (or a more appropriate
validation/invariants page). Finally, add a note that downstream FHIR IGs
may define their own context-specific variables.

There are no cross-referenced Jira tickets. The request did not surface any
explicit Jira links, but the Zulip searches revealed the direct motivation:
on the fhir/infrastructure-wg thread *"fhirpath slice() definition"*
(2025-10-22/23) Brian and Grahame Grieve discussed how `%profile` is used
inside the normative definition of the new `slice()` function from
FHIR-42983. Brian noted that `%profile` is referenced but never introduced
in the variables section and asked to "Introduce `%profile` in the variables
section — and indicate when it is expected to be available — assume only
profile invariants." Grahame agreed ("good idea"). FHIR-53952 is the
follow-up ticket that captures that agreement plus the other editorial
cleanup Brian identified while reading the section.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply all four edits to the FHIR Core specification, as Brian describes:

1. **Restructure the "FHIR specific variables" section on `fhirpath.html`.**
   List the always-available variables (`%resource`, `%rootResource`,
   `%context`) as the primary set, and move the purpose-specific variables
   into a bulleted list noting the context in which each is defined:
   - `%profile` — available in `ElementDefinition.constraint.expression`
     (profile invariants) only.
   - `%previous`, `%current` — available in `SubscriptionTopic` FHIRPath
     criteria only.
   - `%questionnaire`, `%qItem` — available in Questionnaire-related
     FHIRPath expressions only.
   Add a closing note that implementation guides MAY define additional
   FHIR-specific variables for their own contexts.
2. **Convert the existing code block of variable definitions into a
   table** with columns *Variable / Type / Where available / Description*.
3. **Add a new worked-examples block** (separate from the definitional
   table) containing a small set of realistic FHIRPath expressions that
   use each variable, each annotated with what the example demonstrates
   and the context in which it runs (core invariant, profile invariant,
   subscription-topic criteria, questionnaire enableWhen/expression, etc.).
4. **Move the detailed `%profile` documentation** (how it enables
   invariants to reach back into the containing profile — including the
   interaction with `slice()` discussed on the Zulip thread — and when
   implementations must / must not expose it) out of `fhirpath.html` and
   onto `elementdefinition.html` (or a dedicated "Constraints and
   Invariants" sub-page) next to where `ElementDefinition.constraint.expression`
   is described, leaving a one-line stub + link on `fhirpath.html`.

#### Justification

The request is entirely editorial: it improves accuracy (the section
currently says "two" variables but defines three), aligns the spec with
the actual normative surface (`%profile` is already referenced
normatively by `slice()` per FHIR-42983 but is not introduced anywhere),
and fixes a documentation placement problem that Brian and Grahame
explicitly agreed on in the `fhir/infrastructure-wg` Zulip thread. The
subs-topic and questionnaire variables are already defined elsewhere in
the spec, so listing them (with links) in the FHIRPath variables section
costs little and dramatically improves discoverability. Moving the
`%profile` deep-dive next to `ElementDefinition.constraint.expression`
keeps the FHIRPath page focused on the language and puts the
invariant-specific behaviour where implementers looking at profiles will
actually find it. There is no API or wire-format impact, no breaking
change, and no ambiguity in the ask.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the editorial cleanup (items 1–3 of Disposition A: fix the
"2 vs. 3" wording, list the context-specific variables as bullets,
convert to a table, add worked examples), but **keep the primary
`%profile` documentation on `fhirpath.html`** in the variables section,
and add only a short *normative* cross-reference and a deeper
implementer-focused sub-section on `elementdefinition.html` rather than
*moving* the text. In parallel, open a small companion ticket to track
the IG-variable-extensibility note (item 5) so it can be worded
consistently with however R6 chooses to let IGs register new FHIR-specific
variables.

#### Justification

`%profile` is a FHIRPath environment variable. Readers coming to the
FHIRPath variables section will expect every FHIR-defined variable to be
listed and defined there, even if narrowly scoped; burying its full
definition on `elementdefinition.html` risks making it feel like an
ElementDefinition-only construct rather than a FHIR-wide FHIRPath variable
that simply happens to only be populated in constraint expressions today.
This approach still satisfies Brian's underlying need (the variable is
introduced, its availability is called out, examples exist) without
fragmenting the FHIRPath variables reference. It also avoids the risk of
readers of a profile-invariant expression having to page-jump from
FHIRPath → ElementDefinition → back to FHIRPath to understand a single
expression. The companion ticket for IG extensibility lets the WG decide
whether IGs registering new variables needs any formal machinery (naming
rules, collision avoidance) before the note is added.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as purely editorial polish that does not change
normative behaviour, and close as *Not Persuasive* with a note directing
the reporter to open a pull request against the FHIR source directly (or
handle it as editor's discretion during the R6 ballot-to-publication
editorial pass).

#### Justification

No normative change is being requested: every variable listed is already
defined somewhere in the spec, `%profile`'s semantics are already fixed
by FHIR-42983 and the existing invariants machinery, and nothing about
validator behaviour or implementer contracts would change. A formal WG
disposition, ballot note, and downstream implementer-review cycle is
heavy machinery for what amounts to restructuring a section and adding
examples. Declining with a "please submit editorially" note keeps the WG
agenda focused on substantive change requests.

---

### Recommendation

**Recommended disposition:** A

The ticket is clearly scoped, the reporter has already identified the
concrete edits, the underlying motivation (the `%profile` / `slice()`
interaction) is a *correctness* problem — a normatively-referenced
variable is effectively undocumented — not just cosmetic, and Grahame
already agreed to the core change on Zulip. Disposition B preserves a
reasonable editorial preference (keep all FHIRPath variables documented
on the FHIRPath page) but loses the ticket's strongest argument: the
detailed `%profile` behaviour is tangled up with invariant evaluation and
profile resolution, which is ElementDefinition territory, not FHIRPath
language territory. If the WG ends up preferring B, the compromise should
be "full stub + link on fhirpath.html, detailed behaviour on
elementdefinition.html" rather than splitting either page's text in half.
Disposition C undersells the request: a spec section that contradicts
itself ("introduces 2, defines 3") and fails to introduce a variable its
own normative `slice()` function depends on deserves a WG-blessed fix,
not editorial triage. Recommend accepting A and, as a small refinement,
explicitly cross-linking `fhirpath.html`'s new `%profile` bullet to the
relocated text on `elementdefinition.html` so neither reader path dead-ends.
