## Summary

Lloyd McKenzie reports that the codes used for `Group.membership`
(`definitional`, `conceptual`, `enumerated`) are described in a way that
is not clear and not disjoint: the definitions for `definitional` and
`conceptual` overlap, and the description of `enumerated` carries the
implication that *all* members must be listed — which is not the
intent. The ticket proposes replacement descriptive text for all three
codes that (a) makes the partition explicit, (b) clarifies which codes
imply that `Group.characteristics` are *necessary and sufficient* vs.
merely *necessary*, and (c) clarifies the role of `Group.member` for
each code (notably, that under `enumerated` membership can only be
confirmed by inspecting `Group.member`, but the list is not guaranteed
to be exhaustive).

Although there are no Jira links, the keyword-related ticket
[FHIR-45005](https://jira.hl7.org/browse/FHIR-45005) ("Group
characteristic comments are incorrect and combinationMethod defaults
need to be defined") is the most recent change in the same area and
already adjusted the comment on `Group.characteristic` and gave
`combinationMethod` a `meaningIfMissing`. The current ticket continues
that line of clarification work, this time on the membership-basis
codes themselves.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt Lloyd's proposed text verbatim as the descriptions of the three
codes in the `group-membership-basis` value set / code system (the
codes bound to `Group.membership`):

- **definitional** — "The `Group.characteristics` specified are both
  necessary and sufficient to determine membership. Members are
  represented as FHIR resources. All entities that meet the criteria
  are considered to be members of the group, whether referenced by the
  group or not. If members are present, they are individuals that
  happen to be known as meeting the `Group.characteristics`. The list
  cannot be presumed to be complete."
- **conceptual** — "Identical to definitional, except that members are
  entities that cannot be represented as FHIR resources."
- **enumerated** — "The `Group.characteristics` are necessary but not
  sufficient to determine membership. Membership can only be confirmed
  by checking whether the entity is listed as a `Group.member`, though
  it's possible not all members are listed."

In addition, update the `Group.membership` element comment in the
resource definition to point at these refined code descriptions and
note that the partition is intended to be disjoint along the
"resource-representable?" and "characteristics sufficient?" axes.

#### Justification

The proposal is a pure clarification: it does not change the value
set's codes, the element's cardinality, or any conformance constraint,
so there is no breaking impact on instances or implementers. It
removes a real ambiguity (definitional vs. conceptual) and corrects a
genuine misreading risk on `enumerated` (that "enumerated" forces an
exhaustive `Group.member` list). It also harmonises with FHIR-45005
(which already shifted the membership-criteria narrative onto
`combinationMethod`) by making it explicit which codes treat
`Group.characteristics` as sufficient and which do not. Editorial
fix-ups of value-set descriptions in this area have an established
precedent in FHIR Infrastructure (FHIR-45005, Persuasive).

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substance of the proposal but split the change in two:

1. Adopt the new descriptions for `definitional` and `conceptual`
   essentially as proposed, since those are the codes whose definitions
   actually overlap.
2. For `enumerated`, instead of just rewriting the description, also
   re-examine the modelling: introduce a boolean (or a code) on
   `Group` — e.g. `Group.actual` / `Group.complete` (or a new code
   inside `group-membership-basis` such as `enumerated-complete`) — to
   let an instance signal whether the listed `Group.member` entries
   are intended to be the *complete* set. The description of
   `enumerated` would then say "characteristics are necessary but not
   sufficient; whether the listed members are exhaustive is signalled
   by `Group.<flag>`" rather than baking "possibly not complete" into
   the code's definition.

#### Justification

The current proposal solves the description ambiguity but quietly
*locks in* the semantic that an `enumerated` group's `Group.member`
list is never guaranteed to be exhaustive. That is a useful default,
but several real-world use cases (closed cohorts, dispensing lists,
patient lists handed off between systems) genuinely *do* want to
assert "this is the complete membership". Surfacing that as a separate
signal is more expressive and matches the historical R4 `Group.actual`
boolean, which carried exactly this distinction. The trade-off vs.
Disposition A is heavier scope (a structural change instead of a
text-only change) and a longer review cycle, but it avoids re-litigating
"how do I assert a closed enumerated group?" in a future ticket.

---

### Disposition C: Decline

#### Proposal

Decline the change as out of scope for an editorial pass: keep the
existing descriptions of `definitional`, `conceptual`, and
`enumerated`, and instead direct the reporter to raise a more concrete
ticket *only* if a real interoperability problem (a misimplementation
or an actual disagreement between two implementers) can be cited.

#### Justification

The defensible reason to decline is that the proposed text introduces
a new normative-flavoured statement — "the list cannot be presumed to
be complete" for both `definitional` and `enumerated` — into what is
nominally a value-set description. That statement is reasonable but is
not currently constrained anywhere on the resource, so embedding it in
code descriptions risks people reading conformance into a code-system
display. Declining preserves the status quo while inviting a
better-scoped fix (either constraint text on the resource itself, or a
proper structural ticket like Disposition B).

---

### Recommendation

**Recommended disposition:** A

The ticket is exactly the kind of low-risk editorial clarification
that the FHIR Infrastructure work group has historically taken on
Persuasive (cf. FHIR-45005), and the proposed text materially improves
the value-set's clarity without changing any conformance rule, code,
or cardinality. Disposition B's structural concern about
"closed/complete enumerated groups" is real, but it is a separate
modelling discussion that should not block this clarification — and
nothing in Disposition A precludes opening that ticket later.
Disposition C is hard to defend: the overlap between `definitional`
and `conceptual` is genuine, the reporter has done the drafting work,
and waiting for a concrete misimplementation to surface is a poor use
of the ballot cycle.
