## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a Technical Correction in the FHIR Core publication:
locate the "+ see also Testing Resources" cross-reference block whose
anchor currently targets `https://build.fhir.org/ig/hl7/fhir-testing-ig/index.html`
and repoint it at the canonical, working URL for the Testing
Resources IG. Concretely, change the `href` to
`https://build.fhir.org/ig/HL7/fhir-testing-ig/` (case-sensitive `HL7`,
no trailing `index.html`). Apply the correction to whichever page in the
6.0.0-ballot4 publication source contains the block (the inline
`hl7.org/fhir/6.0.0-ballot4/external.png` icon in the ticket markup
indicates this lives in the core publication, most likely a
testing-related landing page such as `testing.html` / `testing-tools.html`
or an equivalent intro file in the publication source). No specification
content changes; the change is a single anchor's `href` (and, if the
project convention is to use root-relative or build-time constants for
IG cross-references, switch to that mechanism in the same edit).

#### Justification

This matches what the ticket literally asks for, addresses a Highest /
Technical Correction issue with no debatable design surface, and aligns
the link with the URL form the Testing IG actually publishes under (as
corroborated by the Zulip threads above, which reference
`build.fhir.org/HL7/fhir-testing-ig` with case-sensitive `HL7`). It is a
minimum-risk publication fix with no implementer impact beyond removing
a broken outbound link.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the immediate broken link **and** harden the publication against
recurring breakage of cross-IG links. Two parts:

1. Apply the same `href` correction described in Disposition A.
2. In the same change, replace any direct `build.fhir.org/ig/...`
   hard-coded links in FHIR Core's "see also" / additional-resources
   cross-references with an indirection (e.g., a publication variable,
   ig-registry lookup, or `{% include cross-ig-link ... %}`-style macro)
   that resolves to the IG's canonical URL at build time. Add a CI link
   check (or extend the existing one) so that the next time an IG moves
   or is renamed, the publication build fails fast instead of shipping a
   404 to readers.

#### Justification

The reported bug is a symptom of a broader fragility: cross-IG links
written as case-sensitive absolute URLs to `build.fhir.org/ig/...` will
break whenever the target IG renames its repo, changes its publication
layout (e.g., introduces an `/en/` localized landing), or moves between
`build.fhir.org` and `hl7.org/fhir/...` at publication time. The Zulip
context explicitly shows the Testing IG was a recent spin-off with a
landing path that has already shifted (`/en/`). Fixing only this link
guarantees the next ticket of this shape will land on FHIR-I again.
Trade-off vs. Disposition A: more work and a small risk of
out-of-scope-for-a-Technical-Correction churn, but it actually addresses
the underlying cause.

---

### Disposition C: Decline

#### Proposal

Decline as not a defect in the FHIR specification: the link target
exists and works at `https://build.fhir.org/ig/HL7/fhir-testing-ig/`, so
the URL the reporter encountered is functionally a publication-tooling
or content-authoring slip rather than a normative-spec problem.
Recommend the reporter file the issue against the relevant publication
source repository (or the IG itself) and close the Jira ticket with a
pointer to that repo.

#### Justification

A Jira ticket is heavyweight for what is, materially, a one-character
case fix in a single `href`. If the project's stated home for
publication-tooling and link-rot issues is GitHub on the publication
source repo (rather than Jira), funnelling these through Jira creates
process drag for the work group and obscures the real fix surface. Note,
however, that the ticket is explicitly typed as `Technical Correction`
at `Highest` priority and routed to FHIR Infrastructure — so this
disposition runs against the ticket's own classification and against the
reporter's evident expectation that Jira is the right venue.

---

### Recommendation

**Recommended disposition:** A

A is the right call: the ticket is a textbook Technical Correction
(broken outbound link, no normative content at stake), the fix surface
is a single anchor, and the canonical target URL is unambiguous from the
Zulip evidence (case-sensitive `HL7/fhir-testing-ig/`). Disposition B is
attractive in principle and the work group should keep it on the radar
for a follow-up infrastructure ticket, but bundling a publication-
tooling refactor and a CI link-checker rollout into a `Highest`-priority
Technical Correction risks blocking a small fix on a much larger
discussion. Disposition C is hard to defend given the ticket's own type
and routing — FHIR-I owns "FHIR Core" publication corrections, and
declining would push the reporter into a more confusing process for a
fix the work group is best positioned to land quickly.
