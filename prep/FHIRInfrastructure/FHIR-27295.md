## Summary

The ticket asks the FHIR Shorthand (FSH) language to provide a shorthand
syntax for binding an element to a small, ad-hoc value set without
requiring the author to declare a full `ValueSet` instance. The reporter
proposes inline enumerated bindings of the form
`status = #complete | #pending`, and suggests that when a `title` is
supplied, the rest of the resulting `ValueSet`'s metadata (`name`, `id`,
`description`) can be derived automatically from context — e.g.,
`name` from `title` with disallowed characters substituted, and
`description` synthesised as `"used to restrict values of fieldname in
profilename"`. The motivating observation is that hand-authoring a
trivial value set typically takes 3–5× more lines than the binding it
supports, which undercuts FSH's "shorthand" value proposition. The
proposed wording links to upstream tooling discussion at
[FHIR/sushi#321](https://github.com/FHIR/sushi/issues/321). Augury
returned no Jira, Zulip, Confluence, or GitHub cross-references for this
ticket, so there is no other linked work in the indexed corpus.

## Details

**Description:**

A simplified syntax would be appreciated for very simple restrictions.
It's so much easier to say:

- `status = #complete | #pending`

Then it is to craft a complete value set. It seems like might of the
value set operational definition could be automatically determined from
context. A `ValueSet` has a name, id, title and description in FSH. If
"title" is provided, everything else can be determined from context.

(If Title is specified, name can be Title with some special characters
removed, and others replaced with `_`), and description can be something
like: "used to restrict values of fieldname in profilename".

It's supposed to be shorthand, we should make it easy to do some of
these. Having to define the valueset is easily 3–5x more lines.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the FSH grammar with an inline value-set binding shorthand for
required (and possibly extensible) bindings, taking the literal form
proposed by the reporter:

```
* status = #complete | #pending
```

Semantics:

- The compiler (SUSHI) materialises an anonymous `ValueSet` whose
  `compose.include` lists each `#code` token. Where a `system` is not
  written explicitly (e.g., `system#code`), the system is taken from
  the bound element's existing required-system context if unambiguous,
  otherwise the author must qualify each code (`http://hl7.org/...|...`
  or `system#code`).
- The binding strength defaults to `required` (matching the `=`
  operator's existing FSH semantics for fixed values), with `from … (extensible)`
  / `(preferred)` / `(example)` continuing to be available via the
  longhand form. Optionally introduce `?=` or a trailing strength
  marker for non-required inline bindings.
- The synthesised `ValueSet.id`, `name`, `title`, and `description` are
  derived from context: `id` = `<Profile>-<element>-vs`, `name` =
  PascalCase form of the id, `title` = `"<Element> in <Profile>"`,
  `description` = `"Used to restrict values of <element> in <Profile>"`.
  If a `title` is supplied via an extended form, everything else is
  derived from the title as the reporter describes.
- The synthesised `ValueSet` is emitted as a real artifact in the IG so
  that it is reviewable, terminology-validated, and resolvable by
  consumers; it is not "hidden" inside the profile.

This change lives in the FSH language specification
(`HL7/fhir-shorthand`) and in the SUSHI compiler (`FHIR/sushi`,
specifically the work tracked by `FHIR/sushi#321`).

#### Justification

The shorthand is consistent with FSH's stated goal of being a concise
authoring language: today, a one-line conceptual binding to two codes
costs roughly five lines of FSH and a separately-named `ValueSet`
artifact, which is the exact pain point the reporter calls out. The
syntax mirrors patterns FSH authors already use for fixed coded values
(`* status = #complete`) and for `from <ValueSet>` bindings, so it is
discoverable. Anonymous, derived-name value sets are a well-precedented
pattern in IG tooling (e.g., StructureDefinition `binding.valueSet`
pointing at a small, profile-scoped set), and emitting a real `ValueSet`
keeps the result terminology-validatable rather than introducing a new
"binding-only" construct outside the FHIR information model. Both
2020-08 (Kramer/Rhodes 13-0-0) and the 2020-10 reopening reflect WG
support for the idea in principle.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the inline-codes shorthand, but treat it as syntactic sugar over
the existing `from <ValueSet>` binding rather than introducing
context-derived names. Concretely:

- Add a literal-`ValueSet` expression usable on the right-hand side of
  a `from` binding, e.g.
  `* status from #complete | #pending (required)`.
- The compiler synthesises an anonymous `ValueSet`, but its `id`/`name`
  are deterministic, content-addressed (e.g., a stable hash of the
  ordered code list, prefixed by the profile id), and its `title` /
  `description` default to the profile + element context. Authors who
  want a friendly name use the existing longhand `ValueSet:` block.
- Binding strength is always written explicitly using the existing
  `(required|extensible|preferred|example)` suffix; no overload of
  `=`.
- Restrict the shorthand to bindings on `code` / `Coding` /
  `CodeableConcept` elements; reject it elsewhere with a clear compiler
  error.

This change still lives in `HL7/fhir-shorthand` (grammar) and
`FHIR/sushi` (compiler) but avoids the new `=`-based binding operator.

#### Justification

Reusing `from` keeps a single mental model for "this element is bound
to this set of codes" and avoids overloading `=`, which today means
"fix the value of this element". Using a deterministic, content-derived
id sidesteps the open question Lynn flagged ("the source of the
ValueSet name, id and description will have to be figured out") without
relying on profile/element names that may change and silently break
downstream references. Requiring an explicit binding strength is more
defensive and matches FHIR's general posture that binding strength is
not a thing to default silently. The trade-off vs Disposition A is a
small loss in terseness in exchange for fewer surprising emergent
identifiers and no new operator semantics.

---

### Disposition C: Decline

#### Proposal

Decline to add an inline-value-set shorthand to the FSH language.
Document, instead, a recommended idiom in the FSH style guide for
small bindings: a one-line `ValueSet:` declaration immediately above
the profile rule that uses it, with a short conventional naming
pattern (e.g., `<Profile><Element>VS`). Direct further work on
ergonomic improvements to the SUSHI tooling layer
(`FHIR/sushi#321`) — for example, an editor snippet or a `sushi
generate vs` helper — rather than to the language.

#### Justification

The current FSH grammar already supports the use case unambiguously;
the complaint is line count, not capability. Adding a second way to
express bindings increases language surface area, complicates the
grammar, and creates a class of "anonymous" `ValueSet` artifacts whose
identity is derived from context — a fragile coupling that has caused
problems in other generated-artifact pipelines (renaming a profile or
element silently renames downstream value sets, breaking external
references and validators). Lynn's comment explicitly flags the
"figure out where the name/id come from" problem as unresolved, and
declining keeps the spec narrow until a concrete, tested proposal
exists upstream.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the real ergonomic win the reporter is asking
for — letting an author write the codes inline at the point of binding
— without inventing a new meaning for `=` and without committing the
spec to a context-derived naming scheme that the WG has already
identified as the hard, unresolved part of the problem (per Lynn
Laakso's 2020-10 comment). Reusing `from` keeps the language's mental
model consistent: `*element = #code` continues to mean "fix the value",
and `*element from <codes> (strength)` continues to mean "bind to this
set". A deterministic, content-addressed synthetic id keeps the
emitted `ValueSet` a first-class, validatable artifact (the main
benefit of A over C) while avoiding A's silent rename-on-refactor
hazard. Disposition C is defensible but underweights the reporter's
quantitative point that today's longhand is 3–5× the size of the
binding it supports, which is the kind of friction FSH exists to
remove. The WG should sponsor a joint spec/SUSHI proposal along the
lines of B, coordinated with `FHIR/sushi#321`, and revisit naming
conventions only after the syntax and validation behaviour have been
agreed.
