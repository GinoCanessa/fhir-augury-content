## Summary

The ticket reports that the SMART on FHIR conformance section currently
links to [RFC 5785](https://datatracker.ietf.org/doc/html/rfc5785) when
describing how clients locate the `/.well-known/smart-configuration`
document. RFC 5785 has been formally obsoleted by
[RFC 8615](https://datatracker.ietf.org/doc/html/rfc8615), so the
reporter asks that the link be updated to point at the current RFC.
In addition, the inline normative sentence that says *"Contrary to
RFC5785 Appendix B.4, the `.well-known` path component may be appended
even if the FHIR endpoint already contains a path component"* needs to
be re-pointed to the equivalent location in the new RFC, which is
**RFC 8615 Section 3**. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this issue, so the request
stands on its own as a documentation/reference correction in the SMART
App Launch conformance text.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the literal edits the reporter described to the SMART App Launch
conformance section (the page that defines the
`/.well-known/smart-configuration` requirement, in the
`HL7/smart-app-launch` IG):

1. Replace every link to `https://datatracker.ietf.org/doc/html/rfc5785`
   with a link to `https://datatracker.ietf.org/doc/html/rfc8615` and
   change the displayed text from "RFC 5785" to "RFC 8615".
2. Update the normative sentence from
   *"Contrary to RFC5785 Appendix B.4, the `.well-known` path component
   may be appended even if the FHIR endpoint already contains a path
   component."* to
   *"Contrary to RFC8615 Section 3, the `.well-known` path component may
   be appended even if the FHIR endpoint already contains a path
   component."*

No normative behaviour changes — only the citation and the section
pointer are updated to track the obsoletion of RFC 5785 by RFC 8615.

#### Justification

This is the most direct, lowest-risk fix and aligns the SMART App
Launch IG with current IETF status: RFC 5785 explicitly states it has
been obsoleted by RFC 8615, so any reader following the existing link
is already redirected to RFC 8615 in practice. The relocated text
("Contrary to ... Section 3") preserves the original normative intent —
that SMART servers may serve the well-known document even when the
FHIR base URL already has a path component — and re-anchors it to the
correct, current section number. Because the change is purely editorial
(a reference update, not a behavioural change), it is appropriate to
treat as a persuasive/editorial correction with no impact on
implementers' conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the same RFC 5785 → RFC 8615 retargeting **and** take the
opportunity to slightly tighten the surrounding language so the
conformance text no longer relies on a "Contrary to RFC X" formulation
at all. Rewrite the paragraph as something like:

> *"FHIR endpoints requiring authorization SHALL serve a JSON document
> at the location formed by appending `/.well-known/smart-configuration`
> to their base URL. The `.well-known` path component is appended
> directly to the FHIR base URL even when that base URL already
> contains a path component; this differs from the path-handling
> guidance in [RFC 8615 Section 3](https://datatracker.ietf.org/doc/html/rfc8615#section-3),
> and is intentional so that SMART discovery is deterministic for any
> FHIR base URL."*

Optionally add a short non-normative note explaining why SMART chose
this convention, and (if the WG agrees) add a callout that
implementations should not attempt to "promote" the well-known document
to the host root for FHIR endpoints whose base URL has a path.

#### Justification

The current sentence works, but framing the requirement as "Contrary
to RFC X Section Y" forces the reader to chase the IETF spec to
understand SMART's rule. Re-pointing the citation (Disposition A)
fixes the staleness, but doing the rewrite at the same time eliminates
a known reader-friction point, makes the SMART rule self-contained,
and avoids needing another ticket the next time the IETF document is
revised. The cost is a slightly larger edit and the need for the WG to
sign off on the new wording, which is heavier than the strictly
editorial change in Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the requested change on the grounds that RFC 5785 is
permanently archived by IETF with a clear "Obsoleted by 8615" header,
so any reader landing on the existing link is automatically informed
of, and pointed to, the superseding RFC. The conformance section's
citation is therefore not actually broken, and the SMART App Launch
publication can leave the existing reference in place until a larger
substantive update is made to that page.

#### Justification

A defensible-but-weak rationale: editorial-only changes to a published
IG carry a non-zero cost (publication run, downstream republishing of
the SMART App Launch package, IG-template build churn, and
re-validation of dependent IGs that pin to the page). If the working
group is in a strict "no new publications without a substantive
trigger" posture, batching this with the next normative change is
defensible. Note that this disposition still leaves the textual
reference to "RFC5785 Appendix B.4" technically incorrect — Section
B.4 no longer exists in the active document — so this option degrades
the precision of the spec text and is harder to justify than
Disposition A.

---

### Recommendation

**Recommended disposition:** A

This is a textbook editorial correction: the cited RFC has been
formally obsoleted, the appendix the SMART text refers to no longer
exists in the active document, and the requested fix is a one-for-one
substitution that preserves all normative behaviour. Disposition A
delivers the correctness improvement with the smallest possible diff
and the lowest risk of inadvertently changing conformance semantics.
Disposition B is attractive but expands scope into a wording
discussion that the WG may not want to take on for what is otherwise a
trivial reference update; it can always be revisited in a follow-up
ticket. Disposition C is the weakest option because it knowingly
leaves an inaccurate section pointer ("Appendix B.4") in a normative
conformance section, which is exactly the sort of paper cut SMART
implementers tend to file again later. Because no cross-referenced
Jira, Zulip, or GitHub discussion exists to suggest the WG has
weighed alternatives, the safe default is to accept the reporter's
literal request.
