## Summary

Karl Naden reports that the **"Enforce Authorization"** subsection on the
SMART App Launch *Backend Services* page (STU 2.2) appears to be located
in the wrong place in the document outline. Today it is nested under
section **3.6 (Access Token Request)**, alongside the description of how a
backend client obtains a token from the authorization server. The reporter
observes that the content of the subsection is not really about the token
request at all — it concerns how the **resource server** must inspect the
access token's scopes and other authorization context when the FHIR API
is later called, and how it must filter or reject responses accordingly.
That behavior is the subject of section **3.7 (FHIR API Access)** and
specifically **3.7.2 (FHIR API Access Response)**. The ticket asks
whether the "Enforce Authorization" subsection should be moved from 3.6
to 3.7.2 so that the document structure matches what the normative text
actually describes.

This is a documentation / structural-clarity ticket against
SMART App Launch; no normative requirements are being changed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the entire **"Enforce Authorization"** subsection
(`#enforce-authorization`) on `backend-services.html` out from under
section **3.6 (Access Token Request)** and into section
**3.7.2 (FHIR API Access Response)** in the SMART App Launch IG. Keep
the anchor id (`enforce-authorization`) so existing deep links continue
to resolve. Renumber the surrounding subsections of 3.6 and 3.7 as a
consequence. Optionally insert a brief forward-pointer at the end of
3.6 ("After the access token has been issued, the resource server must
enforce authorization on every FHIR API request — see §3.7.2") so a
reader following the request flow is not left wondering where token
enforcement is described.

#### Justification

The current placement is genuinely misleading: the "Enforce
Authorization" content describes obligations of the **resource server**
when responding to a FHIR API call (validating scopes, scoping returned
data, rejecting out-of-scope requests), which is exactly the topic of
3.7.2. The Backend Services flow narrative reads token-issuance →
API-access in numerical order, so moving the subsection aligns the
document outline with the lifecycle of a request and with the content
of the subsection itself. It is a low-risk, editorial / structural
change with no normative effect — no requirement is added, removed, or
weakened. Anchor preservation avoids breaking inbound links from the
ecosystem (referrers, Zulip, slide decks).

---

### Disposition B: Alternative Approach

#### Proposal

Leave the subsection where it is in the document tree, but address the
reader-confusion that motivated the ticket via two smaller edits:

1. Rename the heading from "Enforce Authorization" to something that
   makes it clear the requirement spans both the token-issuance step and
   subsequent API access — e.g., **"Authorization enforcement
   responsibilities"** — and rewrite the lead-in paragraph to call out
   that the requirements apply at FHIR API access time, not at token
   request time.
2. Add a short cross-reference at the top of **3.7.2 (FHIR API Access
   Response)** ("Authorization enforcement obligations are described in
   §3.6.x") so readers arriving via the API-access flow are pointed at
   the canonical text.

Optionally split the subsection in two: keep the
authorization-server-side text (e.g., scope grant decisions) under 3.6,
and lift only the resource-server-side obligations into a new short
subsection under 3.7.2.

#### Justification

This preserves stable section numbers in a published STU and avoids
re-numbering downstream references in implementation guides, test
suites, and certification programs that cite the Backend Services IG by
section number. It also preserves the existing anchor's containing
section, which can matter for tooling that uses heading hierarchy
(e.g., navigation panes, generated PDFs). The trade-off is that it
leaves the structural mismatch the reporter identified — the
subsection still lives under "Access Token Request" — and addresses
the real complaint indirectly via wording, not structure.

---

### Disposition C: Decline

#### Proposal

Decline as **not warranted** (editorial preference). Acknowledge that
the subsection touches on resource-server behavior, but argue that
authorization enforcement is logically tied to the access token that
3.6 has just described being issued, and that the current grouping
("how the token is obtained, and what it obligates the resource server
to do") is a defensible authorial choice. Close as "Persuasive with
Mod" only if the workgroup wants to add a one-line cross-reference,
otherwise "Not Persuasive".

#### Justification

The reorganization affects only the SMART App Launch IG's outline; the
normative requirements are unchanged. Section-number churn in a
published STU has real downstream cost (broken citations from external
specs, certification programs, and test plans). If no implementer has
reported being misled in practice — and the ticket's evidence is the
reporter's own reading, with no Zulip discussion or other community
support surfaced — the bar for restructuring a published STU is
arguably not met.

---

### Recommendation

**Recommended disposition:** A

The reporter's observation is correct: the "Enforce Authorization"
subsection describes resource-server behavior at FHIR API access time,
which is exactly the scope of §3.7.2, not §3.6. Backend Services is a
short, sequentially-read page, so a structural fix is more honest than
papering over the mismatch with a renamed heading and cross-references
(Disposition B). Decline (Disposition C) is hard to defend on the
merits — the reporter is pointing at a real outline bug, even if no one
else has complained on Zulip. The risk of section-number churn cited
against A is real but bounded: SMART App Launch already publishes
versioned URLs (`STU2.2/`), and preserving the `#enforce-authorization`
anchor keeps all deep links live. The workgroup should accept the move
to §3.7.2 and add a one-line forward pointer at the end of §3.6 so
readers tracing the request lifecycle still find the enforcement
requirements without surprise.
