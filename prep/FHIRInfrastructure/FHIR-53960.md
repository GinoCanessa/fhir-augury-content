## Summary

The ticket reports that the FHIR-specific "Additional Functions" layered
on top of FHIRPath use inconsistent error-handling semantics. Some
functions silently return empty on error / invalid input (e.g.
`hasValue`, `getValue`, `toString`, `memberOf`, `subsumes`, `subsumedBy`,
`htmlChecks`, `conformsTo`, `elementDefinition`, `slice`, `resolve`,
`ofType`, and every `%server` function — `read`, `create`, `update`,
`delete`, `search`, `patch`, `validate`, `transform`, `everything`,
`apply`), while others throw (e.g. `checkModifiers` throws when
modifiers are found, and the `%factory` primitive forms are described
as returning a value "or an error"). The reporter also calls out that
the FHIRPath core specification requires singleton-only functions to
throw when given multiple values, but FHIR's profile of FHIRPath
returns empty instead — a deliberate deviation that is currently not
explicitly documented or justified in the spec. Finally, the `weight`
function is flagged: its page does not declare whether its input is a
single-cardinality function, and the reporter expects it should be.
The ticket asks FHIR Infrastructure to bring these behaviours into a
coherent, documented pattern.

No Jira cross-references, Zulip discussions, or GitHub items were
surfaced for this ticket, so the request is considered on its own
narrative.

## Details

**Description:**

Inconsistent error handling patterns — some functions return empty,
others throw errors:

- **Return empty on error / invalid input:** `hasValue`, `getValue`,
  `toString`, `memberOf`, `subsumes`, `subsumedBy`, `htmlChecks`,
  `conformsTo`, `elementDefinition`, `slice`, `resolve`, `ofType`, and
  all `%server` functions *(read, create, update, delete, search,
  patch, validate, transform, everything, apply)*.
- **Throw errors:** `checkModifiers` *(throws if modifiers found)*,
  `%factory` functions *(noted as "or an error" for primitives)*.

***Note:*** FHIRPath core specification says singleton-only functions
should throw when given multiple values, but FHIR's implementation
returns empty instead — this deviation should be explicitly documented
and justified.

Also the **`weight`** function doesn't declare if it's a
single-cardinality input function. Expect that it should be.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIRPath "Additional Functions" section of the FHIR Core
specification (`fhirpath.html`) to (1) document a single, consistent
error-handling policy per function family, and (2) explicitly call out
and justify FHIR's deviation from FHIRPath core for singleton
evaluation. Concretely:

- Add an introductory subsection "Error handling" to the Additional
  Functions page that states the default policy: FHIR-defined
  functions return empty (`{}`) on invalid input, missing input, or
  failed resolution, and only throw when the function's semantics are
  inherently assertional (i.e., the function exists to signal a
  violation).
- Re-categorise the two current "thrower" functions under that policy:
  `checkModifiers` is retained as throwing because its purpose is
  assertional (it is used to enforce the "must understand" contract);
  `%factory` primitive forms are aligned by either (a) replacing "or
  an error" with "or empty" for consistency with the rest of the
  library, or (b) being kept as throwing with an explicit note that
  their purpose is to construct a valid primitive and that an invalid
  input is a programming error, not a data condition.
- Add an explicit "Deviation from FHIRPath core" note stating that
  FHIR's profile of FHIRPath chooses to return empty rather than throw
  when a singleton-only function is given a collection of size > 1,
  with a justification (typically: robustness in mixed-cardinality
  data graphs, and consistency with the "empty propagates" convention
  used throughout FHIR validation and invariants).
- For each function currently listed as "return empty", annotate the
  page entry with a short explicit phrase: "Returns empty on invalid
  input or when the input is not a singleton."
- For `weight`, explicitly declare the input cardinality (singleton)
  and its error-handling behaviour (return empty on non-singleton or
  non-coded input), matching the default policy above.

#### Justification

This is a spec-hygiene change — no behavioural break for implementers
who already align with the existing prose. It addresses the reporter's
core complaint (that the deviation from FHIRPath core is undocumented)
and removes ambiguity around `weight`. Aligning `%factory` primitive
forms with the default "return empty" policy removes the only
non-assertional thrower in the list and leaves `checkModifiers` as the
single principled exception, which is easy to explain (its job is to
fail loudly). Because these functions are FHIR-specific extensions of
FHIRPath, FHIR Infrastructure has the authority to set their
error-handling contract; documenting that contract explicitly is
consistent with existing FHIR design principles (prefer graceful
degradation in FHIRPath invariants to avoid masking other validation
errors).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than normalising to a single "return empty" policy plus a
narrow assertional exception, introduce a small, named typology on
the Additional Functions page and classify every function against it:

- **"Query" functions** (read-like, terminology-like, resolution-like:
  `hasValue`, `getValue`, `toString`, `memberOf`, `subsumes`,
  `subsumedBy`, `htmlChecks`, `conformsTo`, `elementDefinition`,
  `slice`, `resolve`, `ofType`, all `%server.*`) — contract: return
  empty on any failure, never throw, so they can be used safely inside
  invariants without masking the surrounding expression.
- **"Assertion" functions** (`checkModifiers`) — contract: throw on
  the condition the function is designed to detect; they are intended
  to be used as guard expressions, not as data accessors.
- **"Constructor" functions** (`%factory.*`) — contract: return a
  valid instance on success, and empty on invalid arguments (i.e.
  demote "or an error" to "or empty"). This keeps them safe inside
  invariant expressions and aligns with the Query family.

The Additional Functions page would then have an up-front table
mapping each function to its typology and its error-handling
contract, plus a "Deviation from FHIRPath core" note (same wording as
in Disposition A) justifying the singleton-collection-as-empty choice
for the Query family. `weight` is listed in the Query family with
singleton input and empty-on-non-singleton behaviour.

#### Justification

This approach produces the same end-state for most functions as
Disposition A, but gives implementers a named framework rather than a
list of exceptions. That is valuable because the Additional Functions
set has grown organically and will keep growing; having an explicit
typology makes it obvious where to slot new functions (e.g. future
`%server` operations, future terminology helpers). The cost is
slightly more editorial work on the page, and the need to re-review
each function's classification. Compared to Disposition A, it more
fully closes the door on future drift, but it does demote the
`%factory` "or an error" phrasing, which a reader might argue is a
subtle behaviour change; in practice current implementations vary and
"return empty" is the more defensible, safer-in-invariants behaviour.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive / Works as Designed**, on the
grounds that the mix of "return empty" vs. "throw" is deliberate: most
Additional Functions are designed to be safe inside invariants (and
therefore return empty), while `checkModifiers` and the `%factory`
primitives are deliberately assertional and meant to surface
programmer or data errors. Add at most a one-line editorial note on
the Additional Functions page acknowledging the split, but do not
restructure the section or reclassify any function. The `weight`
cardinality question could be handled as a separate, small editorial
ticket.

#### Justification

The current behaviour is already implemented by multiple FHIRPath
engines in the ecosystem, and changing the documented contract (even
"just" demoting `%factory` from "or an error" to "or empty") risks
disturbing downstream validation pipelines that rely on the current
shape. A decline-with-editorial-note avoids that risk. That said, this
disposition does not address the reporter's substantive concern —
that FHIR's deviation from FHIRPath core on singleton evaluation is
undocumented — and so is the weakest of the three options on its
merits.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The reporter's underlying point is
correct: the Additional Functions set has two unrelated issues — an
undocumented deviation from FHIRPath core on singleton evaluation, and
an ad-hoc split between "return empty" and "throw" functions — and
both are fundamentally documentation problems that the workgroup can
fix without breaking existing engines. Disposition A fixes them but
leaves the exception (`checkModifiers`) as a special case; Disposition
B generalises that into a small typology (Query / Assertion /
Constructor) that also tells the workgroup where to put *future*
Additional Functions, which is valuable because this list has only
grown over successive releases. Disposition C is defensible only if
the workgroup judges the `%factory` "or an error" phrasing to be load-
bearing; if it is, the workgroup can opt into the "A-minus" variant
(Disposition A without the `%factory` change) during review. Either
way, the `weight` cardinality clarification and the "Deviation from
FHIRPath core" note should land — those are unambiguous wins for
spec clarity.
