## Summary

The ticket reports that the `_outputFormat` parameter is overloaded as the
disambiguator between the two async patterns defined in FHIR R6 ballot —
**Async Bulk** (NDJSON file-set output) and **Async Interaction** (Bundle/
single-resource output). With the rules currently published in
`async-bulk.html`, omitting `_outputFormat` means *Async Bulk* on a
server that only supports Bulk, but means *Async Interaction* on a
server that supports both. The reporter, James Jahns, points out two
real-world consequences: (1) a working client request can silently
change pattern when a server later adds Async Interaction support, and
(2) a client interacting with arbitrary servers has no robust way to
*force* the Async Interaction pattern, because the only signal —
omitting `_outputFormat` — is interpreted differently by different
servers. He proposes (a) defining a way to unambiguously request Async
Interaction (e.g. dedicated `_outputFormat` values) and (b) making the
omitted-`_outputFormat` behaviour consistent regardless of server
capability.

This is the direct follow-up to **[FHIR-47271](https://jira.hl7.org/browse/FHIR-47271)**
("Make the trigger that decides the async pattern used more clear",
*Applied / Persuasive*, R6), whose accepted resolution introduced
exactly the rule James is now objecting to: *"For servers that support
both the Bulk Data and Interaction patterns, the `_outputFormat`
parameter is used to disambiguate the two; when present, the Bulk Data
pattern is used. Servers MAY only support the Bulk Data pattern, in
which case the `_outputFormat` parameter is optional."* In a comment on
FHIR-50598, Gino Canessa states the authors' intent was that an
operation is generally tied to *one* async style (declared in the
operation's narrative), and that the `_outputFormat` disambiguation
text was a secondary safety net that "appears to be more problematic
than it is worth" and should be removed; he also raises whether an
extension on `OperationDefinition` should be defined to declare which
async pattern an operation expects. Related work is being tracked in
**[FHIR-50861](https://jira.hl7.org/browse/FHIR-50861)** ("Async Pattern
return type not always a bundle"), and the **MovingToGuide** label on
both 50598 and 50861 indicates the eventual home for these changes is
the new API Incubator IG rather than core FHIR.

## Details

**Description:**

The behavior of `_outputFormat` in terms of distinguishing Async Bulk
from Async Interaction seems strange in practice.

According to <https://hl7.org/fhir/6.0.0-ballot3/async-bulk.html>:

- if a server *only* supports Async Bulk, `_outputFormat` is totally
  optional. Omitting it (or providing it) triggers Async Bulk, since
  `_outputFormat` "defaults" to `application/fhir+ndjson`.
- if a server supports *both* Async Bulk and Async Interaction,
  `_outputFormat` is still optional. However, omitting it now triggers
  Async Interaction instead; its "default" value is now functionally
  different.

This is especially awkward for a server that starts with only
supporting Async Bulk. In that case, a client may be triggering an
Async Bulk request while omitting the `_outputFormat` parameter. Then,
if the server later adds support for Async Interaction, that client's
request will totally change functionally, due to an unrelated
enhancement from the server.

On the other hand, a client has no way to force that Async Interaction
occurs over Async Bulk, when interacting with servers more generically.
They must omit `_outputFormat` to trigger Async Interaction, but for
some servers, omitting this parameter will actually trigger Async Bulk,
without any immediate error or warning.

The reporter suggests:

1. Providing a way to unambiguously request the Async Interaction
   pattern over the Async Bulk pattern. For example, defining
   `_outputFormat` values for this pattern.
2. Making the behavior for when `_outputFormat` is omitted be
   consistent regardless of a server's current capabilities.

**Comments:**

- **ginocanessa — 2025-05-12:** Tags Josh Mandel and Daniel Gottlieb
  for input, summarising the concern that bulk-export clients which
  rely on implicit ndjson will break if the server later adds the
  generic async pattern.
- **ginocanessa — 2025-06-18:** States the authors' actual intention:
  an operation is generally expected to use only one async style,
  which should be identified when the operation is defined (currently
  via narrative). The text using `_outputFormat` as a disambiguator
  was only meant as an additional safety mechanism, is written in only
  one direction, and "appears to be more problematic than it is
  worth"; it should be removed. Opens for discussion whether to
  define an extension on `OperationDefinition` to declare async
  expectations (e.g. `bulk-data` vs. `async-interaction`).

## Proposed Dispositions

The dispositions below reflect the absence of repo briefings (no GitHub
cross-refs were surfaced), and are scoped instead to the spec text in
`HL7/fhir`'s `async-bulk.html` / `async-interaction.html` (a.k.a.
`async-batch.html`) and to the new API Incubator IG referenced from the
#implementers Zulip thread and FHIR-50861. The MovingToGuide label
indicates that the spec home for any change is expected to migrate from
core to the Incubator IG.

### Disposition A: Accept As Requested

#### Proposal

Change the R6 `async-bulk.html` and `async-interaction.html` text so
that:

1. A new, explicit `_outputFormat` value (e.g. `application/fhir+bundle`
   or, more concretely, the literal sentinel `async-interaction`) is
   defined as the unambiguous request for the Async Interaction
   pattern, and a server that supports Async Interaction SHALL honor
   it regardless of whether it also supports Async Bulk.
2. The default behaviour when `_outputFormat` is omitted is fixed
   independent of server capability — recommended: omitted ⇒ Async
   Bulk (`application/fhir+ndjson`), preserving the long-standing Bulk
   Data IG default and the behavior FHIR-32062 codified for omitted
   headers. Servers that support Async Interaction MUST require an
   explicit `_outputFormat` (or other selector) to invoke it.
3. Capability advertising is tightened so a server's
   `CapabilityStatement` declares which `_outputFormat` values it
   recognises for which pattern.

#### Justification

This directly addresses both of James Jahns's failure modes: (i)
existing Bulk-only clients keep working byte-for-byte when the server
later adds Async Interaction (because omission still ⇒ Bulk), and (ii)
a client can deterministically request Async Interaction across
heterogeneous servers by sending the new explicit value. It preserves
the Bulk Data IG's documented default (`application/fhir+ndjson`) and
the "omitted ⇒ server-defined default with Bulk semantics" precedent
from FHIR-32062. The community thread *#bulk data > Parameter Names*
already accepted `_outputFormat` as the right knob for this kind of
infrastructure choice, so reusing it (rather than inventing a new
parameter) is consistent with existing patterns.

---

### Disposition B: Alternative Approach

#### Proposal

Take the resolution Gino Canessa proposed in the 2025-06-18 comment
and back out the disambiguation rule entirely:

1. In `async-bulk.html` and `async-interaction.html`, **remove** the
   text that uses `_outputFormat` to choose between Async Bulk and
   Async Interaction. `_outputFormat` reverts to its original Bulk
   Data role: selecting the file format *within* the Async Bulk
   pattern, with the existing `application/fhir+ndjson` default.
2. Define a new extension on `OperationDefinition` (e.g.
   `http://hl7.org/fhir/StructureDefinition/operationdefinition-async-pattern`)
   with a coded value set of `bulk-data | async-interaction | sync`,
   so each operation declares which async pattern (if any) it supports.
   Clients select the pattern by selecting the operation, not by
   shaping the request.
3. Move the consolidated rules into the new API Incubator IG (per the
   MovingToGuide label and the #implementers thread direction), citing
   FHIR-50861 as the companion cleanup.

#### Justification

This is what the original authors actually intended (per Gino's
comment) and removes the source of the ambiguity rather than papering
over it. It also aligns with the active redesign direction in the
#implementers > *Async interaction pattern* Zulip thread, where Josh
Mandel and Nikolai Ryzhikov are independently sketching unified
async-pattern proposals for the Incubator IG, and with FHIR-50861's
proposal to bring the simplified async pattern into the IG. Using an
extension on `OperationDefinition` is a non-breaking, declarative
mechanism — far cleaner than overloading a runtime parameter — and
it lets capability tooling reason about each operation statically.
The cost is that legacy clients keying off the (recently published)
"omit `_outputFormat` ⇒ Async Interaction" rule from FHIR-47271 must
re-key on the operation/extension instead; given that R6 is still in
ballot, that cost is acceptable.

---

### Disposition C: Decline

#### Proposal

Decline the change. Document in the resolution that the existing R6
ballot text from FHIR-47271 is the WG's accepted disambiguation
strategy: `_outputFormat` present ⇒ Async Bulk; omitted on a
dual-supporting server ⇒ Async Interaction; servers that support only
Bulk treat `_outputFormat` as optional. Mitigations for James's
concrete concerns are addressed at the *capability advertising* layer
rather than by changing the trigger: servers MUST advertise which
patterns they support in `CapabilityStatement`, and clients SHOULD
query that statement before relying on omitted-parameter defaults.

#### Justification

FHIR-47271 was applied as *Persuasive* and shipped via
[HL7/fhir#3192](https://github.com/HL7/fhir/pull/3192); reopening it
during ballot reverses a recently-agreed decision and risks churn for
implementers who have already updated to the R6-ballot text. The
breakage scenario James describes is real but solvable by clients
inspecting `CapabilityStatement` before assuming a default; that is
the standard FHIR pattern for capability-sensitive behaviour. Decline
is defensible *only* if no broader async redesign is on the table —
which, given FHIR-50861 and the active #implementers thread, is
unlikely to remain true through the next ballot cycle.

---

### Recommendation

**Recommended disposition:** **B**

The MovingToGuide label, the active #implementers > *Async interaction
pattern* thread, and the parallel FHIR-50861 cleanup all point in the
same direction: the WG is going to redesign the async pattern in the
API Incubator IG rather than micro-patch core. Disposition A keeps the
overloaded `_outputFormat` knob and only smooths its edges, leaving
the deeper design problem (Gino's "the disambiguation text was a
secondary safety net that is more problematic than it is worth") in
place. Disposition C reaffirms a freshly-applied resolution that
multiple senior contributors have already started to walk back. **B**
matches the original authoring intent, removes the ambiguity at its
source, and lands the durable fix in the same artifact (the API
Incubator IG) where the broader async redesign is converging — letting
FHIR-50598, FHIR-50861, and the simplified-async hackmds be resolved
together rather than as competing point fixes. Implementers lose the
recently-introduced "omit ⇒ Async Interaction" trigger, but gain a
declarative, statically-inspectable signal on `OperationDefinition`,
which is strictly more useful for tooling and capability reasoning.
