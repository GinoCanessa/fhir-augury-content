## Summary

FHIR-34040 requests a new SDC extension that conveys, per
`Questionnaire.item`, how the item's `text` should be rendered when a
completed `QuestionnaireResponse` is displayed. Diane Rodden (the
reporter) describes a real implementation that today carries an
out-of-band per-`linkId` table of render hints (e.g. `LabelWithData`,
`ShowHeader`, `DataNoLabel`, `suppressHeader`) and asks for a
standardised way to attach those hints to the `Questionnaire` so that
they are carried into any rendering of the corresponding
`QuestionnaireResponse`. After Zulip discussion in
`#questionnaire > instructions for rendering the responses by linkId`,
Lloyd McKenzie agreed the underlying need is narrower than Diane's
internal model and converges to roughly three values for the item
label/text: **"display always" / "display on first repetition" /
"don't display"**, asked for a change request, and that change request
is this ticket.

The ticket was originally voted "Considered for Future Use" on
2021-10-07 (Diane Rodden / Stoyan Halkaliev: 6-0-0) with the
guidance to use a custom extension in the meantime and revisit in the
next release. Lynn Laakso has since reverted that resolution and
re-opened the ticket post-STU4 publication of SDC, with a note that
the workgroup may instead pursue a more generic
"narrative generation" mechanism for `QuestionnaireResponse`. There
are no other Jira cross-references and no GitHub cross-references; the
only external touchpoint is the Zulip thread.

## Details

**Description:**

An enhancement is needed where there is an extension containing the
instructions for rendering the questionnaire response item text. I
think that the extension would need to be placed against the
questionnaire definitions when the questionnaire is being created and
then carried over to the questionnaire response so that any generic
questionnaire response rendering software would have access to it
during the rendering step.

Per Lloyd McKenzie, the extension would allow 3 values for the
rendering instructions: "display always, display on first repetition,
or don't display" for the item text. (Whether the middle one is
relevant depends somewhat on what the control mechanism is ā€” if you're
doing a grid view or something, the label would only be displayed once
regardless.)

**Comments:**

- **mckenzie (2021-09-30):** Notes this is different from
  `valueset-questionnaire-usage-mode` because `usageMode` disables the
  whole item (question + answer), whereas this proposal only governs
  the appearance of the item's `text`. If adopted, `item-control`
  would need to override it (e.g., a table layout would not repeat the
  label per row regardless of the new extension). Unsure how many
  questionnaires/implementations need this.
- **mckenzie (2021-09-30):** Pointer to the originating Zulip thread
  `#questionnaire > instructions for rendering the responses by linkId`.
- **lynn_laakso (2026-03-26):** Reverted previous resolution
  (`Considered for Future Use`, 2021-10-07, vote 6-0-0). Original
  guidance was "use a custom extension; we'll look at this in the next
  release; might pursue a more generic 'narrative generation'
  mechanism for QuestionnaireResponse."
- **lynn_laakso (2026-03-26):** Re-open post STU4 publication.

## Related Zulip Discussions

### [#questionnaire > instructions for rendering the responses by linkId](https://chat.fhir.org/#narrow/stream/questionnaire/topic/instructions%20for%20rendering%20the%20responses%20by%20linkId/near/254476392)

Diane Rodden opened the thread asking whether there is any SDC
extension that lets her annotate per-`linkId` rendering instructions
on a `Questionnaire` so that those hints flow through to renderers of
the corresponding `QuestionnaireResponse`. She showed an in-app table
keyed by `linkId` with values like `LabelWithData`, `ShowHeader`, and
`DataNoLabel`, and noted that
[`extension-designnote`](https://www.hl7.org/fhir/extension-designnote.html)
does not carry over to `QuestionnaireResponse`. Brian Postlethwaite
asked whether the request was about styling the *display* of completed
responses (PDF/HTML output), pointed to his own loose-interpretation
renderer, and argued that response rendering should generally reuse
the questionnaire's entry-time layout (tables/grids etc.) with labels
substituted for edit fields, since otherwise consumers of the
`QuestionnaireResponse` cannot tell what was hidden. Lloyd McKenzie
reframed the request as specifically "should `item.text` be displayed
when rendering a completed form" ā€” he noted SDC already has a way to
suppress an item as a whole but no one has previously asked to
suppress just the item text ā€” and converged with Diane on three values
("display always" / "display on first repetition" / "don't display"),
caveated by the fact that `item-control` choices like grids would
override it. He invited Diane to file a change request, which became
FHIR-34040. Tilo Christ asked whether the same hints would also be
honoured when generating a `QuestionnaireResponse.text` narrative;
that question was not answered in-thread. No firm consensus on
mechanism (new extension vs. broader narrative-generation mechanism)
was reached.

Participants: Diane Rodden, Brian Postlethwaite, Lloyd McKenzie, Tilo
Christ.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new SDC extension on `Questionnaire.item` (and have it carry
through, by reference to `linkId`, to renderers of
`QuestionnaireResponse.item`) called something like
`questionnaire-itemTextRendering` (canonical
`http://hl7.org/fhir/uv/sdc/StructureDefinition/questionnaire-itemTextRendering`).

- Cardinality: `0..1`.
- Type: `code`.
- Bound (required) to a new SDC ValueSet
  `questionnaire-item-text-rendering` with three concepts:
  - `display-always` (default if extension absent)
  - `display-first-repetition` (only show item.text on the first
    occurrence of a repeating item / repeating group)
  - `do-not-display`
- Place the extension on `Questionnaire.item`. Specify in the
  extension definition that the hint applies to the rendering of the
  item's `text` element only, not to the data/answer.
- Document precedence rules:
  - `Questionnaire.item.extension:itemControl` (e.g. `grid`, `table`,
    `htable`, `gtable`) overrides this extension where the chosen
    control inherently controls label placement.
  - This extension is *independent of* (not a replacement for)
    `valueset-questionnaire-usage-mode`, which suppresses the whole
    item.
  - Renderers MAY use this hint when generating
    `QuestionnaireResponse.text` narrative, in addition to interactive
    rendering.
- Add the new extension to the SDC IG (likely under
  `input/resources/` or the extensions section of `HL7/sdc`),
  cross-link it from the rendering page, and add an example.

#### Justification

This is exactly the shape Lloyd McKenzie proposed in the originating
Zulip thread (3 values, scoped to `item.text`, with `item-control`
taking precedence) and matches Diane Rodden's stated implementation
need without inventing a richer typology of rendering instructions
that the workgroup has not asked for. Restricting the extension to
`item.text` keeps it cleanly orthogonal to the existing
`questionnaire-usage-mode` ValueSet (which McKenzie explicitly
distinguished in comment 69103) and to `item-control`, so it does not
re-litigate either of those mechanisms. Anchoring the canonical to
`hl7.org/fhir/uv/sdc/...` keeps the change inside SDC's own publishing
surface and avoids touching FHIR core or the base
`Questionnaire`/`QuestionnaireResponse` resources, consistent with the
ticket's `SDC-IG` label and FHIR Infrastructure work group ownership.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** introduce a new per-item rendering-hint extension. Instead,
pursue the broader "narrative generation mechanism for
`QuestionnaireResponse`" that Lynn Laakso flagged in the
2026-03-26 reverted-resolution comment, treating FHIR-34040 as one
input into that larger design.

Concretely:

1. Open (or, if already exists in SDC backlog, link to) a parent
   change request for a `QuestionnaireResponse` narrative-generation
   mechanism in the SDC IG. Suggested shape:
   - A `Questionnaire`-level extension
     (`questionnaire-narrativeRenderingProfile`) that points to a
     reusable rendering profile (a `Parameters`-style or coded set of
     defaults: show/hide labels for non-repeating items, for repeating
     items, for groups, plus narrative template hooks).
   - Per-item override extension (effectively the Disposition-A
     extension, but defined as an override of the
     `Questionnaire`-level profile defaults).
   - Optional binding from
     `QuestionnaireResponse.text` generation rules to that profile.
2. In the meantime, document on the SDC rendering page that
   implementers MAY use a custom extension with the same three values
   (`display-always` / `display-first-repetition` / `do-not-display`)
   on `Questionnaire.item`, and that such custom extensions are
   expected to be subsumed by the future SDC mechanism.
3. Resolve FHIR-34040 as `Persuasive with Modification` /
   `Considered for Future Use` against the new parent ticket.

#### Justification

This is the direction Lynn Laakso explicitly preserved when she
re-opened the ticket ("we might pursue a more generic 'narrative
generation' mechanism for a QuestionnaireResponse"), and it is
consistent with the unresolved question Tilo Christ raised in the
Zulip thread about whether these hints should also drive
`QuestionnaireResponse.text` narrative. Lloyd McKenzie himself noted
in 2021 that "we're not sure how many questionnaires/implementations
will need this" ā€” that uncertainty argues for shipping a small,
extensible hook inside a larger framework rather than a stand-alone
extension that may later need to be retro-fitted into a profile-based
mechanism. The cost is one more design cycle before Diane's concrete
need is standardised, but Diane has been using a custom extension
since 2021, so the marginal delay is small relative to the risk of
publishing a stand-alone extension that the eventual narrative
mechanism contradicts.

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-34040 as **Not Persuasive** / **Considered ā€” No Action**
on the grounds that:

- `QuestionnaireResponse` rendering is intentionally an
  implementation concern and is already addressable today via
  `Questionnaire.item.extension:itemControl`, the base SDC rendering
  extensions, and `valueset-questionnaire-usage-mode` for whole-item
  suppression.
- Where label suppression is needed for a *form layout* (grids,
  tables), the existing `itemControl` codes already handle it, as
  Lloyd McKenzie observed in comment 69103.
- For all remaining cases, implementers can (and Diane already does)
  carry per-`linkId` rendering hints in a custom or local extension;
  the Zulip thread did not surface any second implementer with the
  same need.
- Keep the door open by directing implementers to the existing
  custom-extension pattern documented in the original
  2021-10-07 vote ("for now, use a custom extension").

#### Justification

The originally-voted resolution (`Considered for Future Use`, 6-0-0)
already reflected the workgroup's judgement that this need does not
clearly rise to the level of a normative SDC extension, and the
re-open note in 2026-03-26 did not introduce new
implementer demand. Adding an extension whose only required behaviour
is "render or don't render the label" risks fragmenting the SDC
rendering story between `usageMode`, `itemControl`, and a third axis
that will, per McKenzie, have to be defined as subordinate to
`itemControl` anyway. Declining cleanly is defensible until a second
independent implementer asks for the same hook or until the
narrative-generation work proposed under Disposition B firms up
enough to subsume it.

---

### Recommendation

**Recommended disposition:** **B ā€” Alternative Approach.**

Disposition B is the best fit for where the ticket actually sits today.
Lynn Laakso's 2026-03-26 re-open notes that the SDC workgroup is
already contemplating a more general narrative-generation mechanism
for `QuestionnaireResponse`, and Tilo Christ's unanswered Zulip
question ā€” "would this then also be used when I create a narrative
for inclusion in the questionnaire response?" ā€” is a strong signal
that any per-item label-rendering hint is going to interact with
`QuestionnaireResponse.text` generation. Shipping the Disposition-A
extension now would lock in a single-purpose hook against
`item.text` only, which (a) leaves Tilo's question unresolved, and
(b) risks being re-shaped or re-canonicalised once the broader
profile-driven mechanism is designed. Disposition C is unattractive
because Lynn explicitly reverted the earlier "Considered for Future
Use" outcome, signalling that the workgroup no longer wants to leave
the request entirely to custom extensions. Disposition B preserves
Diane Rodden's concrete need (the three values
`display-always` / `display-first-repetition` / `do-not-display`) as
a per-item override inside the larger framework, gives implementers
explicit interim guidance to keep using a custom extension with that
shape, and aligns the disposition with the work-group direction Lynn
already telegraphed in her re-open comments.
