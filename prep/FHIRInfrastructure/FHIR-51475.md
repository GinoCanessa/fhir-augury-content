## Summary

The reporter (JornDuwel) flags an inaccurate worked example on the FHIR Core
"Testing FHIR" narrative page (`testing.html`). The page describes how to use
the `TestScript` resource and gives an example of a search on `Patient`,
followed by an `<assert>` that purports to "verify that the search operation
returned the right resource type" by asserting `<resource value="Patient"/>`.
The reporter points out that a FHIR search interaction does not return a
`Patient` resource directly — it returns a `Bundle` (of type `searchset`)
that contains zero or more `Patient` resources — so the documented assert is
incorrect. The reporter also notes that the two subsequent examples on the
same page are similarly affected and additionally use `path` and
`compareToSourcePath`, which are slated to be deprecated in R6, suggesting
that the example block is overdue for a rewrite.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
linked to this issue in the Augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation/example bug on the FHIR Core "Testing FHIR"
narrative page (`testing.html`) and rewrite the offending example block so
the assert reflects what a FHIR search actually returns. Concretely:

1. In the first example, change the explanatory sentence to
   "Verify that the search operation returned a `Bundle`" and update the
   asserted resource accordingly:

   ```xml
   <action>
     <assert>
       <resource value="Bundle"/>
     </assert>
   </action>
   ```

2. Add a follow-on assert (or expand the example) showing how to confirm
   that the returned `Bundle` is a search result and contains `Patient`
   entries — e.g., assert `Bundle.type = 'searchset'` and use a
   FHIRPath/expression assert against
   `Bundle.entry.resource.ofType(Patient)` to confirm the contained
   resource type.

3. Apply the same correction to the two follow-on examples on the same
   page so they also assert `Bundle` (and `Bundle.entry.resource` where
   the original used `path` / `compareToSourcePath`). While doing so,
   replace the use of `path` and `compareToSourcePath` with the R6
   replacements (FHIRPath-based asserts) so the page does not teach a
   pattern that is being deprecated in the same release.

#### Justification

The reporter's reading is correct: a RESTful FHIR `search` interaction
returns a `Bundle` of type `searchset`, not the searched resource type
directly. As written, the example would teach implementers an assert
that would fail on any conformant server, undermining the page's didactic
purpose. Correcting the assert keeps the example aligned with the search
interaction defined in the RESTful API and with the `TestScript.assert`
semantics. Folding in the migration away from `path` /
`compareToSourcePath` at the same time avoids publishing fresh examples
in R6 that immediately advertise deprecated machinery. This is a low-risk
narrative-only change with no normative impact.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only fixing the prose, replace the inline XML snippets on
`testing.html` with a reference to a small, real `TestScript` example
instance that lives under the spec's example artifacts (e.g., a new
`testscript-example-search.xml`/`.json` pair, or extension of an existing
TestScript example). The narrative page would show a short excerpt and
link to the full, validating example, which would assert against a
`Bundle` of type `searchset` and use FHIRPath-based asserts to inspect
`Bundle.entry.resource`. Optionally, also add a brief inline note on
`testing.html` calling out that `path` and `compareToSourcePath` are
deprecated in R6 and pointing readers to the FHIRPath-based asserts.

#### Justification

The original defect is fundamentally that hand-authored XML in the
narrative drifted from the actual TestScript semantics and was not
exercised by any tooling. Replacing inline snippets with excerpts of a
real, published example instance ensures the snippet stays in sync with
the resource definition and with whatever mechanical validation the
publisher applies to example resources. This addresses the reporter's
concern (the example is wrong) while reducing the chance of the same
class of defect recurring on the page in future releases. The cost is
slightly higher than Disposition A because it requires creating or
extending an example artifact and wiring it into the narrative.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the surrounding paragraph is
illustrative only and that the broader rewrite of the testing examples
should be deferred to the larger R6 work that already plans to deprecate
`path` and `compareToSourcePath` on `TestScript.assert`. Close as
"Not Persuasive" or fold into whatever R6-era TestScript narrative
refresh ticket exists.

#### Justification

This is defensible only if the workgroup has an existing, scheduled
rewrite of the `testing.html` examples as part of the R6 TestScript
deprecations, in which case patching the assert in isolation could
create churn. Even then, leaving a known-incorrect example in place
during ballot is a poor outcome, so this disposition is weak unless
that larger rewrite is genuinely imminent.

---

### Recommendation

**Recommended disposition:** A

The reported defect is unambiguous: a search interaction returns a
`Bundle`, so the example assert is wrong as published, and the same
defect propagates through the two follow-on examples. The fix is
narrative-only, has no normative impact, and is small enough to land
without waiting on a larger TestScript refresh. Disposition B is
attractive in principle (sourcing the snippet from a validated example
instance is more durable), but it adds artifact and tooling work that is
disproportionate to a documentation correction and can be pursued
separately as a follow-up. Disposition C should be avoided: leaving an
incorrect assert in the published spec while pointing at a future
rewrite is exactly the kind of friction implementers hit when learning
TestScript. While applying Disposition A, the editor should also swap
the `path` / `compareToSourcePath` usage in the two follow-on examples
for the R6-aligned FHIRPath asserts, so the page does not introduce
fresh examples built on deprecated mechanics.
