## Summary

The ticket asks FHIR Infrastructure to define a supported mechanism that
lets a working group experiment with **trial-use elements on resources
that have become (or will become) normative in core**. The reporter
observes that as more core resources move to normative, future
experimentation in those resources will require extensions, and that
making such extensions later become first-class core elements creates a
backwards-compatibility problem (existing instances that used the
extension form remain incompatible with the new core form). The reporter
asks FHIR-I to investigate and specify a mechanism by which an
"Additional Resource"-style IG can introduce trial-use elements onto a
core resource within the scope of that IG.

The ticket was raised at the FHIR-I Q2 WGM session on 26 January 2026
and is tagged for WGM follow-up. There are no Jira-side cross-references
or explicit ticket links recorded in FhirAugury for FHIR-54783, so the
context here is drawn from the ticket itself, the on-ticket comments
from `bvdh` and `ginocanessa` sketching two solution shapes, and
recent Zulip discussion of the broader Additional Resources / normative
boundary work.

## Details

**Description:**

> As all resources in core will become normative, it will be harder to
> experiment with changes in those resources. Trial use changes in these
> resources require the use of extensions. This makes it harder to
> migrate these elements into core as elements as existing content using
> such features will no longer be compatible.
>
> Please investigate and specify a mechanism for core resources to be
> extended in Additional Resource IG's allows trial use element use on
> core resources within the scope of such IG's.

**Comments:**

- **bvdh (2026-01-26):** Notes that this ticket was requested by the
  FHIR-I Q2 WGM session on 26 Jan 2026 and that the resolution will
  require a change to the `resourceDefinition` definition (i.e., to
  `StructureDefinition` / the resource-definition machinery itself).

- **ginocanessa (2026-01-26):** Sketches two candidate approaches to
  capture in the disposition discussion.

  - *Option 1 — "Incubation IG" of normative resources.* Publish one
    incubation version of a normative resource that may carry new
    trial-use elements. Implementers should already be ignoring unknown
    elements, but consumers of the incubation form must be defensive:
    long-term storage of "normative" content currently carries no
    inter-version compatibility guarantee, and the danger of TU
    elements is amplified in this setting. Instances would be clearly
    marked (same as other Additional Resources) and the incubation
    version must be maintained by the same committee responsible for
    the underlying resource.

  - *Option 2 — Extensions on core resources, with an
    `ElementDefinition` flag.* Continue using extensions for trial-use
    additions, but add a flag in `ElementDefinition` that indicates the
    element represents content that was previously delivered via an
    extension. The flag would let tooling and validators bridge between
    the extension form and a future core element. Whether the flag can
    work in *both* directions (core → extension as well as extension →
    core) needs separate verification. The element gets added to core
    when it is ready for normative definition. Normal extension-breaking
    rules still apply, so the recommendation is to host the in-flight
    extension definitions in either the FHIR Extensions Pack or some
    incubation-style IG with similar discipline.

### [fhir/infrastructure-wg > Conformance Module](https://chat.fhir.org/#narrow/stream/fhir%2Finfrastructure-wg/topic/Conformance%20Module/near/548258827)

Grahame Grieve, responding to a Lloyd McKenzie review of his rewrite of
the Conformance Module narrative, walks through the current "stability"
language for each conformance resource. The R6 framing is that
`StructureDefinition`, `OperationDefinition`, `CapabilityStatement`,
`SearchParameter`, `ImplementationGuide`, and `CompartmentDefinition`
are intended to be normative and stable, while `StructureMap` and
`MessageDefinition` may still attract scope/feature/element growth, and
`GraphDefinition` is moving out to the API Incubator IG. The thread
captures the live tension that FHIR-54783 highlights: even on resources
described as "normative and stable", individual elements still carry a
trial-use marker (explicitly noted for `CapabilityStatement`), and the
plan to remove/deprecate or graduate those is still hand-waved rather
than mechanised. No formal consensus on a TU-in-normative mechanism
was reached in this thread.

Participants: Grahame Grieve, Lloyd McKenzie.

### [fhir/infrastructure-wg > Additional Resources](https://chat.fhir.org/#narrow/stream/fhir%2Finfrastructure-wg/topic/Additional%20Resources/near/541259342)

Josh Mandel summarises the FHIR-I "Additional Resources" programme: an
active development lane for resources, operations, and pages not yet
ready for core, published in IG packages but kept under the
`http://hl7.org/fhir` canonical namespace. The post describes the
governance flow (propose → incubate in an IG → prove maturity →
graduate to core or retire), the supporting machinery (an
"allowed-targets" extension to authorise references to Additional
Resources, a compartment-membership extension, package dependencies
resolved by tooling), and the destination IGs (API Incubator, Testing
IG, Subscriptions Backport, Feature Capabilities, topic IGs such as
SDC). It explicitly contemplates "Trial-Use narrative is removed from
core" — i.e., resources stay normative in core while the still-evolving
parts move out. This is the closest current-state articulation of the
container that FHIR-54783 wants to plug trial-use *elements* into;
notably, the post talks about *resources* / *operations* / *pages*
being incubated in IGs, not about new TU *elements* on existing
normative core resources, which is the gap FHIR-54783 highlights.

Participants: Josh Mandel.

### [fmg > Standards-status = draft](https://chat.fhir.org/#narrow/stream/fmg/topic/Standards-status%20%3D%20draft/near/542786591)

(a) for R6 specifically, every artifact in core will end up `normative`

should stay aligned with HL7 process rather than be made generic
(Grahame Grieve, John Moehrke confirming IHE doesn't use it at the

extension is HL7-process-specific). The relevance to FHIR-54783 is
direct: if everything in core must be normative or informative, then the
"trial-use elements on a normative resource" pattern needs an explicit
home — either via the Additional Resources / IG mechanism, or via a new
ElementDefinition signal — rather than being left as today's ad-hoc
mix.

Participants: Elliot Silver, Lloyd McKenzie, Gino Canessa, Grahame

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define and document a first-class **"Additional Resource overlay"**
mechanism that lets an Additional Resources IG declare *new trial-use
elements on a core (normative) resource* within the scope of that IG.
Concretely:

- In `StructureDefinition`, allow a constraint/extension form whose
  `baseDefinition` is a normative core resource, whose `derivation` is
  an existing value (constraint), and whose differential adds *new*
  elements (not just constraints on existing ones) that carry a
  `standards-status` of `trial-use` and an explicit
  `additionalResourceOverlay` marker. Today the spec disallows adding
  new logical elements in a constraint StructureDefinition; this
  disposition introduces the overlay form as the supported escape hatch.
- Require any such overlay element to be hosted by an Additional
  Resources IG owned by the same work group that owns the underlying
  core resource (mirroring the rule already discussed in the
  *Additional Resources* Zulip thread for incubated resources).
- On graduation to core, the overlay element's path/cardinality/type
  must be preserved one-for-one so that instances produced under the
  overlay remain valid against the core resource — i.e., the overlay
  *is* the proposed normative form, not a parallel extension that has
  to be re-mapped later.
- Update the publisher and validator to (a) recognise the overlay form
  as a legitimate StructureDefinition shape, (b) resolve overlays via
  the IG's package dependencies, and (c) preserve canonical-URL
  resolution (overlay element URLs live under the core resource so that
  graduation does not change canonicals).

#### Justification

This is the literal reading of the ticket: it asks for *trial-use
element use on core resources within the scope of an Additional
Resource IG*. The Additional Resources programme already establishes
the governance flow, the package machinery, the canonical namespace
discipline, and the WG-ownership rule (per Josh Mandel's summary in the
*Additional Resources* thread). What is missing is a sanctioned shape
for adding *new elements* — today that requires extensions, which
breaks the smooth-graduation property the ticket calls out. By
preserving paths and types from overlay to core, this disposition
directly fixes the migration discontinuity the reporter describes.
It also addresses bvdh's note that the resolution will require a change
to the `resourceDefinition` definition itself.

The biggest risk — that implementers consume overlay content and then
break when the overlay changes — is the same risk identified by Josh
Mandel in the *Agenda topic* and *Standards-status = draft* threads.
This disposition contains it by (i) confining overlays to Additional
Resource IGs that the implementer must opt into via package dependency,
(ii) requiring an explicit `standards-status = trial-use` marker on
overlay elements, and (iii) requiring same-WG ownership so the
governance signal matches the technical signal.

---

### Disposition B: Alternative Approach

#### Proposal

Do not introduce a new "overlay" `StructureDefinition` shape. Instead,
keep using **extensions** for trial-use additions on normative core
resources, but add two pieces of machinery to make
extension → core-element graduation lossless:

1. Add an `ElementDefinition` flag (per ginocanessa's on-ticket comment)
   — e.g., `ElementDefinition.extensionOrigin` or a
   `structuredefinition-extension-origin` extension — that records, on
   a future core element, the canonical URL of the extension that
   previously carried the same data. Validators and converters can use
   this to accept either the legacy extension form or the new core
   element form, and to round-trip between them.
2. Require that any trial-use extension targeting a normative core
   resource be hosted in either the **FHIR Extensions Pack** or in an
   Additional Resource IG owned by the same WG as the underlying
   resource, with the same governance rules (propose → incubate →
   graduate or retire). Verify in advance whether the
   `extensionOrigin` mapping needs to work in both directions (core →
   extension as well as extension → core); that question is called out
   in the on-ticket comment as separate from this resolution.

#### Justification

Compared to Disposition A, this keeps the core resource's
StructureDefinition strictly normative — no new shape of
StructureDefinition is introduced and the existing "constraint
profiles cannot add new elements" invariant is preserved. That
invariant is load-bearing for tooling and for the implementer mental
model that a normative resource's element set is fixed for the life of
the normative version.

The extension-graduation flag also covers a class of cases the overlay
mechanism does not: extensions that were defined *before* there was any
intent to graduate them, but that the WG later decides to absorb.
The cost is that consumers of the trial-use form do not get to write
the element using its eventual core path — they have to migrate from
`extension[url=…].value[x]` to `core.element` — which is exactly the
backwards-incompatibility the reporter is trying to eliminate. The
`extensionOrigin` mapping makes this *tooling-supported* rather than
*manual*, which is materially better than today, but does not make it
*invisible* to instances at rest.

This disposition is also more incremental: it does not require changes
to publisher/validator handling of constraint StructureDefinitions, and
the `standards-status` work-stream from FHIR-53045/FHIR-53046 already
provides the surrounding vocabulary.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54783. Take the position that **normative core resources
should not host trial-use elements at all**, in either the literal
("element lives in the core SD with a trial-use marker") or the overlay
sense. New behaviour that cannot yet be normative belongs in an
Additional Resource IG as a *new resource or operation* (per the
existing Additional Resources programme) or as a vanilla extension in
the FHIR Extensions Pack, with no special machinery for later
graduation — graduating an extension into a future major version
remains an explicit, instance-incompatible migration that implementers
opt into.

#### Justification

This is the position implied by the *Standards-status = draft* FMG
thread: for R6, every artifact in core is `normative` or `informative`,
so the "trial-use element on a normative resource" pattern is itself
the bug, not something we should design supporting infrastructure for.
Josh Mandel's *Agenda topic* note from 2023 makes the same point from
the implementer side — trial-use elements that change inside otherwise
normative resources are exactly the surface implementers cannot
tolerate.

Declining is defensible on three grounds: (i) the existing Additional
Resources mechanism already covers iteration that is not yet normative,
just at the resource/operation grain rather than the element grain;
(ii) every "make trial-use safer inside normative" mechanism dilutes
the meaning of "normative" that the work-stream is otherwise trying to
strengthen; and (iii) the migration cost the ticket is trying to avoid
(extension → core element) is a one-time cost per element and is the
same cost the rest of the spec already pays for trial-use → normative
graduation of resources and operations.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the recommended starting point for FHIR-I discussion.
It directly addresses the migration-discontinuity the reporter raises
(extensions today don't graduate cleanly into core elements) by adding
the `ElementDefinition` flag ginocanessa already sketched on the
ticket, while preserving the strong guarantee — defended in the
*Standards-status = draft* and *Conformance Module* threads — that a
normative core resource's element set is fixed and that a constraint
StructureDefinition cannot introduce new elements. It is also the
smallest change to the conformance machinery that still meaningfully
improves on today's state, which matters given that the resolution must
land in an R6 timeframe and that bvdh has flagged it will already
require a change to the resource-definition definition.

Disposition A is the more ambitious option and would arguably give the
cleanest implementer experience for the *next* trial-use element
graduation, but it introduces a new StructureDefinition shape with
non-trivial publisher/validator impact and weakens the
"normative-resource SDs are stable" invariant that is currently being
sold to implementers — a trade-off the workgroup may not want to make
this close to R6. Disposition C is the cleanest principled position
but does not actually answer the reporter's request and leaves the
extension → core migration cost unaddressed; the workgroup should only
land there if it is prepared to tell working groups to stop adding
trial-use behaviour to normative resources entirely.
