## Summary

Lloyd McKenzie reports that the FHIR Core specification for
`ElementDefinition.constraint.expression` does not currently state that
the FHIRPath expression is *expected* to evaluate to a boolean, and does
not say what an evaluator should do when an expression returns something
other than a boolean (e.g., an empty collection, a non-boolean
singleton, or a multi-valued collection). The reporter proposes a
specific fix: treat an empty collection as `false`, and treat any other
non-boolean result as an error.

The cross-reference index returned no items linking to FHIR-56154 — no
related Jira tickets, Zulip threads, or GitHub PRs/issues are recorded
against this key in the local store. Two historically related tickets
about constraint expressions whose value type was mishandled
([FHIR-43168](https://jira.hl7.org/browse/FHIR-43168) and
[FHIR-43169](https://jira.hl7.org/browse/FHIR-43169) — `nsd-3`
comparing the boolean `authoritative` to the string literal `'true'`)
illustrate the kind of authoring mistake that the missing rule allows
to slip through, and underscore why an explicit conformance statement
would help implementers.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt exactly what the reporter proposes. In `HL7/fhir` —
`source/elementdefinition.html` (and the matching `<differential>`
definition text on `ElementDefinition.constraint.expression` in the
`source/elementdefinition/` folder) — add a normative paragraph stating:

> The FHIRPath expression in `ElementDefinition.constraint.expression`
> SHALL evaluate to a boolean. If the expression evaluates to an empty
> collection, the constraint is considered to evaluate to `false` (i.e.,
> the constraint fails). If the expression evaluates to anything other
> than an empty collection or a single boolean (for example, a
> non-boolean singleton, or a multi-valued collection), the evaluator
> SHALL raise an error.

Add a one-sentence cross-link from `source/fhirpath.html` pointing back
to this rule so implementers reading the FHIRPath usage page see the
same statement. No StructureDefinition snapshot regeneration is
required — only the `<differential>` definition text on
`ElementDefinition.constraint.expression` and the narrative pages
change.

#### Justification

The proposal closes a real interoperability gap: today, two equally
spec-conformant validators can disagree on whether a constraint with an
empty result "passes" or "errors", and on whether a non-boolean result
should be coerced, ignored, or rejected. The empty-as-`false` rule is
consistent with how FHIRPath's `iif()` and other conditional contexts
treat empty as the absence of a true value, and matches the historical
behaviour of the reference validator. Treating non-boolean results as
errors is consistent with FHIRPath's strict-typing posture (cf.
[FHIR-43169](https://jira.hl7.org/browse/FHIR-43169) where the lack of
implicit `boolean`/`string` conversion silently broke `nsd-3`). The
change is narrative + a `<differential>` edit, both within the
authoring root identified in the `HL7/fhir` briefing, with no
cross-repo impact.

---

### Disposition B: Alternative Approach

#### Proposal

Make the rule explicit in the spec, but soften the non-boolean case
from "error" to "invalid invariant result; the validator SHALL report
the constraint as failed and SHOULD also issue a warning identifying
the expression as malformed." Concretely, the wording in
`source/elementdefinition.html` and the `<differential>` definition
becomes:

> The expression SHALL evaluate to a boolean. An empty collection is
> treated as `false`. Any other non-boolean result causes the
> constraint to be reported as failed; validators SHOULD additionally
> issue a warning indicating that the expression is malformed and does
> not return a boolean.

Optionally, add a publisher-time check to `tools/` that flags
hand-authored constraints whose static analysis suggests a non-boolean
return type, but defer that to a separate ticket (it lives outside the
authoring root).

#### Justification

The reporter's "non-boolean → error" rule is the cleanest answer, but
in practice many existing IG and core constraints have minor authoring
slips (string-vs-boolean comparison, accidental projection) that
currently fail silently. A hard error rule would, on the day it is
adopted, retroactively turn those quiet bugs into hard validator
failures across every published IG that consumes the new FHIR version
— a non-trivial blast radius. A "report as failed + warn" rule still
gets implementers the diagnostic they need (the constraint is flagged,
the warning points at the malformed expression) without breaking
existing publish pipelines on day one. It also stays consistent with
how the bespoke HL7 publisher in `HL7/fhir` already reports invariant
problems through its QA channel rather than aborting.

---

### Disposition C: Decline

#### Proposal

Resolve as *Not Persuasive* on the grounds that
`ElementDefinition.constraint.expression` is a FHIRPath expression and
the FHIRPath specification already governs evaluation semantics; layer
the answer at the FHIRPath spec level (or in
`source/fhirpath.html`) rather than at `ElementDefinition`. No edit
to `source/elementdefinition.html` is made.

#### Justification

This is defensible only if the workgroup believes that FHIR Core
should not duplicate semantics that are arguably the FHIRPath
specification's responsibility. In practice it is a weak position: the
FHIRPath spec defines how an expression evaluates, but it does not
define how the *consumer* of that expression (here,
`ElementDefinition.constraint`) interprets a non-boolean or empty
result. That interpretation is precisely what the reporter is asking
to pin down, and it cannot live in the FHIRPath spec without
overreach. Decline is included here for completeness rather than as a
serious option.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures everything the reporter is asking for — empty →
`false`, non-boolean → diagnostic — but routes the non-boolean case
through "report as failed + warn" instead of a hard evaluator error.
This is the right trade-off given the FhirCore briefing's reminder
that the FHIR publisher and downstream IGs already consume hundreds of
hand-authored invariants: a hard-error rule (Disposition A) would
convert latent authoring bugs like [FHIR-43169](https://jira.hl7.org/browse/FHIR-43169)
into immediate validation failures the moment implementers picked up
the new FHIR version, with no transition path. Disposition B closes
the same interoperability gap, gives validators clear and consistent
behaviour to implement, and gives IG authors a warning channel to
clean up offending expressions before any future ticket tightens the
rule to "error". Disposition C should be rejected — the question is
unambiguously about how `ElementDefinition.constraint` consumes a
FHIRPath result and belongs in the FHIR Core narrative.
