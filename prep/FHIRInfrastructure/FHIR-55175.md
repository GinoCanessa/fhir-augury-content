## Summary

FHIR-55175 is a Technical Correction against the FHIR Core specification's
`operationdefinition` page. The reporter (Lloyd McKenzie) flags that the page
contains one or more informal contractions that violate HL7's editorial style
rule that abbreviations must be fully spelled out in normative prose.
Specifically, the submission calls out `doesn't` and `wasn't`, which should be
rewritten as `does not` and `was not`. The ticket imports Comment 141 from a
larger editorial review pass, and carries no cross-references to other Jira
tickets, Zulip threads, or GitHub items — it is an isolated, narrowly scoped
editorial cleanup.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Make a pure editorial change to the `operationdefinition` page in the FHIR
Core specification: replace the contractions `doesn't` with `does not` and
`wasn't` with `was not` wherever they appear in the page's narrative prose.
No schema, structure definition, invariant, example, or normative semantic
content is modified. Apply the same pass across any closely related narrative
blocks on the page (e.g., operation notes, introduction, examples' prose)
to avoid leaving sibling contractions behind.

#### Justification

This is a Technical Correction that aligns the page with HL7's long-standing
editorial convention that abbreviations and contractions be spelled out in
normative prose. The change is fully backward-compatible (prose only),
carries no implementer impact, and the reporter has explicitly identified
the offending tokens. Accepting as requested is the lowest-risk, lowest-cost
path and keeps the spec internally consistent with pages that have already
been cleaned up by the same editorial pass.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent of the correction, but broaden the scope: instead of
fixing only `doesn't` and `wasn't` on the `operationdefinition` page, also
sweep for the full common set of English contractions on that page
(`can't`, `won't`, `isn't`, `aren't`, `hasn't`, `hadn't`, `wouldn't`,
`shouldn't`, `couldn't`, `didn't`, `don't`, `it's` where it means "it is",
etc.) and expand each to its full form. Record the expanded pattern list
in the ticket resolution so subsequent pages caught by the same editorial
pass can be handled consistently.

#### Justification

The reporter's note ("The page contains one or more misspelled or
abbreviated words … Specifics: …") implies the listed tokens are
examples, not an exhaustive enumeration. Fixing only the two cited
contractions risks leaving near-identical violations in place and
generating follow-on tickets on the same page. A single broader editorial
sweep is cheaper than repeated round-trips, and documenting the pattern
list makes the convention enforceable at review time. The trade-off
versus Disposition A is a slightly larger diff, but still zero semantic
impact.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" on the grounds that natural-language
contractions like `doesn't` and `wasn't` are widely understood, do not
impede implementer comprehension, and are not abbreviations in the
technical sense the style rule was written to address (e.g., `HTTP`,
`URL`, `e.g.`). Retain existing prose as-is.

#### Justification

This position is defensible only if the work group takes the view that
the editorial rule targets *acronyms and initialisms* rather than
*contractions*, and that enforcing contraction expansion across the
whole spec is disproportionate editorial churn relative to reader
benefit. It also avoids the risk of drift between pages touched by the
pass and pages not yet touched. However, it contradicts the editorial
pattern already being applied across the specification via the broader
Comment-141 import, so it would need an explicit work-group statement to
stand.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The ticket is a Highest-priority Technical
Correction from an editor whose Comment-141 import is clearly driving a
systematic editorial pass across multiple pages; the listed tokens
(`doesn't`, `wasn't`) are evidently illustrative rather than exhaustive.
Fixing exactly those two tokens (Disposition A) satisfies the ticket text
but almost guarantees a follow-up ticket the next time another
contraction on the same page is noticed. Declining (Disposition C)
swims against an editorial convention that has already been applied
elsewhere and would require work-group-level reversal of that
convention. A single broader sweep on the `operationdefinition` page —
expanding the full common contraction set and recording the pattern
list in the resolution — is low-risk, semantically neutral, and leaves
the page in a state where any future reviewer is unlikely to raise the
same category of issue again.
