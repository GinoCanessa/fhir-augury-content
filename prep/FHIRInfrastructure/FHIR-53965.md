## Summary

Brian Postlethwaite is requesting three clarifications to the
"Environment Variables" section of the FHIR FHIRPath page
(`fhirpath.html`) so that the distinction between FHIRPath
*environment variables* and FHIR-specific FHIRPath *variables* is made
explicit, and so that an implementer footgun around
`%[name]`-style lookups of profiles and value sets is documented.

Specifically the ticket asks the FHIR Infrastructure work group to:

1. Add a note explaining that **environment variables do not change
   between evaluations** of an expression, in contrast to other FHIR
   FHIRPath variables (such as `%resource`) which are scoped to and
   re-bound for each individual evaluation. This is positioned as
   *the* primary semantic difference between the two categories.
2. **Remove `%resource` from the Environment Variables section.**
   `%resource` is already documented as a FHIR-specific FHIRPath
   variable elsewhere in the spec, so listing it as an environment
   variable is inconsistent and confusing.
3. Add an **implementer note about `%[name]`** (the
   `%`-bracket-name syntax used to refer to extension profiles and
   value sets by id from FHIRPath). The note should warn that this
   form requires the referenced StructureDefinition / ValueSet to be
   resolvable by *the same id it was published with*, and that
   resolution may be ambiguous when multiple versions of the same
   artifact are in scope.

There are no cross-referenced Jira tickets and no cross-referenced
GitHub items. The ticket sits squarely in the FHIRPath documentation
on the FHIR Core spec.

## Details

**Description:**

Make the following clarifications to the Environment Variables section:

- Include a note that the environment variables don't change between
  uses where the other extended variables defined in parts of FHIR
  (such as `%resource`) are specific to an expression's specific
  evaluation. — This is the primary difference between env vars and
  FHIR-specific vars.
- `%resource` is already documented as a FHIR-Specific variable, it
  should not be defined in the environment variables section.
- Include an implementer note that the use of the `[name]` form in
  the environment variables requires that the extension profiles and
  value sets referred to have the same id as was published with —
  which may have issues where there are multiple possible versions in
  scope.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the **Environment Variables** section of `fhirpath.html` in the
FHIR Core specification to apply all three of Brian's clarifications:

1. Add a leading paragraph at the top of the Environment Variables
   section stating that environment variables are bound by the
   *host* before evaluation begins and **remain constant for the
   lifetime of an expression evaluation**, in explicit contrast to
   FHIR-specific FHIRPath variables (`%resource`, `%rootResource`,
   `%context`, `%ucum`, `%sct`, `%loinc`, etc.), whose values are
   re-bound by the FHIRPath engine per evaluation according to the
   element being evaluated. Cross-link to the FHIR-Specific
   Variables section.
2. **Remove `%resource` from the Environment Variables list / table.**
   It is already documented as a FHIR-specific variable, so the
   duplicate listing is removed and a sentence is added pointing
   readers to the FHIR-Specific Variables section for `%resource`.
3. Add an **implementer note** under the Environment Variables
   section describing the `%"name"` / `%`*name* form used to refer
   to a published StructureDefinition (extension / profile) or
   ValueSet by id. The note must call out that:
   - resolution is by **id only**, so the artifact must be available
     to the evaluation engine under the id under which it was
     published, and
   - when multiple versions of the same canonical are in scope (e.g.
     during package upgrades, IG composition, or multi-IG
     environments) the result is **implementation-defined / may be
     ambiguous**, and implementers SHOULD provide a deterministic
     resolution policy (latest version, package-pinned version,
     etc.).

All three changes are pure documentation edits to the FHIR Core
publication source for `fhirpath.html`; no schema, profile, or
validator change is required.

#### Justification

The ticket is a documentation clarification with no normative
behavioural change. It removes an internal contradiction in the
spec (`%resource` listed in two places with different binding
semantics), and it makes explicit a binding-lifetime distinction
that the FHIRPath stream has repeatedly relitigated in chat (see the
"Accessing variables from the FHIR variable extension" thread, where
Brian himself observes that `%resource` "feels more like a variable"
than an environment value). The implementer note about `%[name]`
captures a real, recurring gotcha — version-ambiguity for canonical
artifacts is one of the most common sources of resolver bugs — and
documenting it is consistent with the FHIRPath spec's existing
practice of carrying implementer guidance in callout boxes. Because
the change is editorial in scope, it carries minimal risk and can be
folded into the next FHIR Core editorial pass.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt items (1) and (2) from the request as straightforward editorial
fixes, but rather than adding the `%[name]` warning *only* under
Environment Variables, **promote it into a small dedicated subsection**
("Resolving named artifacts from FHIRPath") that:

- Defines the `%"name"` form once, in one canonical place.
- States that resolution is by id (not canonical URL or
  canonical+version), and is delegated to the host environment.
- Notes the multi-version ambiguity and SHOULDs a deterministic
  resolution policy.
- Cross-links from both the Environment Variables section and the
  FHIR-Specific Variables section.

Optionally, also flag (non-normatively) that a future version of the
spec could introduce a canonical-aware lookup form
(e.g. `%canonical("...|version")`) so the current id-only behaviour
is clearly framed as the *current* state.

#### Justification

The third bullet of the ticket is really about how named artifact
*resolution* works from FHIRPath, not about the lifetime semantics
of environment variables — those are two different concepts that the
ticket happens to bundle together because the `%[name]` form is
introduced under Environment Variables today. Pulling the resolution
note into its own subsection reduces the chance that future readers
miss it (because they were looking at the FHIR-Specific Variables
section instead), gives the topic a stable anchor that other specs
and IGs can link to, and leaves room to evolve the resolution rules
without further restructuring. The cost over Disposition A is one
extra heading and a couple of cross-links — modest editorial work
for noticeably better discoverability.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the existing Environment
Variables and FHIR-Specific Variables sections, read together,
already convey the intended semantics: `%resource` is listed in the
FHIR-Specific Variables section as an evaluation-scoped variable,
the Environment Variables section already says environment variables
are supplied by the host, and the `%[name]` form is described
alongside the existing examples. No normative change is needed and
implementers who read both sections will reach the correct
conclusion.

#### Justification

The change is editorial only and not strictly required for
conformance. There is no defect in implementations caused by the
current wording, and the work group's editorial bandwidth could be
spent on issues with implementer-visible impact. (This option is
included for completeness; see Recommendation below for why it is
not preferred.)

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. Items (1) and (2) of the ticket are
unambiguous editorial fixes — the duplicate listing of `%resource`
is a real internal contradiction and the binding-lifetime wording is
genuinely missing — so they should be accepted as Brian wrote them.
Item (3) is the more interesting one: the FHIRPath stream chat
history (e.g. the "Accessing variables from the FHIR variable
extension" thread) shows that implementers and even spec authors
have been confused about whether `%resource` is "an environment
variable" or "a FHIR variable," and the named-artifact lookup form
sits on the same fault line. Promoting the `%[name]` resolution
rules into their own short subsection (B) gives that guidance a
durable home, makes the eventual jump to a canonical-aware lookup
form much easier to introduce, and costs only marginally more than
the in-place note in Disposition A. Disposition C is not
recommended: the cost of the edit is small, the duplication of
`%resource` is a real bug in the prose, and FHIR Infrastructure has
generally accepted similar editorial cleanups to `fhirpath.html` in
prior cycles.
