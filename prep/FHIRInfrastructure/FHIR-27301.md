## Summary

The reporter notes that slicing is one of the most common profiling tasks but
that FHIR Shorthand (FSH) currently requires authors to use caret (`^`) syntax
to set slicing parameters (e.g., `^slicing.discriminator`, `^slicing.rules`).
The ticket requests that FSH define a native, first-class rule for declaring
slicing — analogous to other dedicated FSH rule types — instead of relying on
the lower-level caret mechanism. The reporter acknowledges that this is
already mentioned under "Future Considerations" in the FSH specification and
asks that it be promoted to a "Future Requirement" so it is not forgotten.
The cross-reference query returned no linked Jira tickets, Zulip threads, or
GitHub items, so this prep is grounded solely in the ticket itself and the
captured comment history.

## Proposed Dispositions

> Note: With no GitHub cross-references and no repo briefings to anchor
> the proposals to a specific authoring path, the dispositions below
> name the FSH specification (the `HL7/fhir-shorthand` IG) and the SUSHI
> compiler (`FHIR/sushi`) at the conceptual level only. Any actual
> implementation work should be re-grounded against fresh
> `repo-analysis` briefings for those repos before code changes are
> drafted.

### Disposition A: Accept As Requested

#### Proposal

Add a first-class **slicing rule** to the FSH grammar so that authors do
not need caret syntax to declare slicing on an element. Concretely:

- Define a new FSH rule production along the lines of
  `<path> contains <slice-name> [<min>..<max>] [and <slice-name> …]`
  paired with a sibling rule
  `<path> ^slicing` is replaced by a dedicated declaration such as
  `<path> sliced by <discriminator-type> on <discriminator-path>
   [rules: open|closed|openAtEnd] [ordered|unordered]`.
- Update the FSH specification's "Slicing" section (currently in the
  "Profiles" / "Defining Slices" chapter of the Shorthand IG) to make
  the new rule the recommended form, and demote caret syntax to an
  explicit fallback for cases the native rule does not cover (e.g.,
  setting unusual `slicing.*` sub-elements).
- Implement parser, AST, and SUSHI export support so that the new
  syntax round-trips to the same `ElementDefinition.slicing` /
  `ElementDefinition.sliceName` content as the equivalent caret form.
- Promote the existing "Future Considerations" entry on slicing to a
  "Future Requirement" / accepted backlog item, as the reporter
  explicitly asked.

#### Justification

Slicing is, as the reporter states, one of the most frequent profiling
tasks; requiring authors to drop into caret syntax for it is at odds
with FSH's design principle of providing concise, declarative rules for
common profiling operations (cardinality, fixed value, binding, contains
slices, etc.). A dedicated rule keeps the surface area uniform with
other slice-adjacent constructs (notably the existing
`contains <slice>` rule, which already implies slicing without making
the slicing parameters declarative). Mark Kramer's comment confirms
this option ("create a slicing rule with special grammar") was on the
table from the start, and the prior 11-0-1 vote in 2020 shows broad
support for at least carrying the work forward.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) introducing new FSH grammar, have **SUSHI
infer slicing parameters** from the existing `contains` rule plus a
small amount of metadata declared inline. Concretely:

- Extend the existing `<path> contains <slice> …` rule with optional
  `by <discriminator-type> on <discriminator-path>` and
  `rules <open|closed|openAtEnd>` clauses, e.g.
  `* component contains systolic 1..1 and diastolic 1..1 by pattern on code`.
- When those clauses are present, SUSHI emits the equivalent
  `ElementDefinition.slicing` content automatically; when they are
  absent and the path is being sliced for the first time, SUSHI uses
  sensible defaults (e.g., open, value-based discriminator on the path
  declared by an accompanying `^slicing` if any) and emits a warning
  pointing the author at the new clause.
- Caret syntax remains supported unchanged for advanced cases, but the
  IG documents the inferred / extended-`contains` form as the
  recommended path.

#### Justification

This addresses the same author pain point — eliminating boilerplate
caret invocations for the common case — with a smaller grammar change
and lower risk of fragmenting the rule surface. It also leverages the
existing mental model: authors already write `contains` to introduce
slices, so attaching the discriminator and rule kind to that same
statement keeps related declarations together. The trade-off vs.
Disposition A is expressiveness: a fully-separate `sliced by` rule may
be clearer for readers and easier to teach, whereas an
extended-`contains` form is more compact but slightly overloaded.
Either way, this option preserves backward compatibility with all
existing FSH that uses caret syntax for slicing.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep caret syntax as the documented mechanism
for setting slicing parameters in FSH. Update the "Future
Considerations" wording to make clear that no native slicing rule is
planned, and rely on tooling (editor snippets, SUSHI templates) to
reduce author friction.

#### Justification

Caret syntax is FSH's general-purpose escape hatch for setting any
`ElementDefinition` field, and slicing is already partially served by
the dedicated `contains` rule. Adding another slicing-specific rule
expands the language surface, increases the FSH parser/compiler
maintenance burden, and creates a second way to do something that
already has a working idiom — at the cost of teaching authors when to
use which form. Given that the original 2020 vote (Moesel/Rhodes
11-0-1) resolved this as "Considered for Future Use" and the ticket
has had no movement since being re-opened in October 2020, declining
is defensible if the workgroup judges that current tooling support
(SUSHI's `^slicing` shorthand handling, IDE assistance) is now
sufficient.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the practical benefit the reporter is asking for
— removing caret-syntax boilerplate for the common slicing case — with
the smallest grammar addition and the least risk of fragmenting FSH's
rule surface. It also honors Mark Kramer's "have SUSHI infer the
correct values" suggestion from the original comment thread, while
still giving authors an explicit, declarative way to state the
discriminator and rule kind when defaults are not appropriate.
Disposition A is a defensible alternative if the workgroup prefers a
visually distinct `sliced by` rule for readability, but it is more
disruptive and creates a third slicing-related construct alongside
`contains` and caret syntax. Disposition C is hard to justify given
that slicing is genuinely the most common reason FSH authors reach for
caret syntax and the original 11-0-1 vote signaled clear appetite for
improvement; the workgroup should at minimum re-affirm the "Future
Requirement" framing the reporter requested.
