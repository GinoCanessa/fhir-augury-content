## Summary

The reporter observes that on the FHIR Core specification pages for data
types that derive from `BackboneType` (specifically `Timing`,
`ElementDefinition`, `Dosage`, and `RelativeTime`), the **Structure** tab
of each type page does not indicate inheritance from `BackboneType`.
Instead, the Structure table shows these types as inheriting only from
`Element`. The **UML** tab on the same pages does render the inheritance
chain correctly, so the underlying type model is right; the defect is
confined to what the Structure tab view surfaces. The ticket is
classified as a Technical Correction against FHIR Core and is currently
Triaged with no assignee.

No related Jira tickets, Zulip threads, or GitHub items were surfaced
via cross-references or text search, so this ticket is being reviewed
purely on the information in the report.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Correct the Structure tab rendering for every datatype that derives
from `BackboneType` so that the inherited-type label matches what the
UML tab already shows. Concretely, for `Timing`, `ElementDefinition`,
`Dosage`, and `RelativeTime` (and any other type deriving from
`BackboneType`), the Structure table header row should display
`BackboneType` as the base type instead of `Element`. The fix is in the
spec-publication pipeline that produces the Structure tab — i.e., the
code that emits the "Elements defined in Ancestors" / base-type column
of the Structure table for complex datatypes — not in the underlying
StructureDefinition resources (the UML tab already resolves the chain
correctly, confirming the model is right).

#### Justification

The reporter has identified a genuine inconsistency between two views
of the same spec fact on the same page: the UML tab shows the correct
parent (`BackboneType`), while the Structure tab shows the grandparent
(`Element`). That is, by definition, a technical-correction-grade
rendering bug rather than a design question. Accepting the ticket as
requested aligns the Structure tab with the UML tab, with the
StructureDefinition resources, and with the class hierarchy documented
for `BackboneType` itself. Implementers who read the Structure tab to
understand a type's contract today are being told the type is an
`Element` (no `modifierExtension`) when it is in fact a `BackboneType`
(with `modifierExtension`); correcting that closes a small but real
source of confusion and risk.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only replacing `Element` with `BackboneType` in the
Structure table header, render the full inherited-type chain for any
datatype that inherits through more than one intermediate base — e.g.,
show `Timing` as `Element → BackboneType → Timing` (or an equivalent
breadcrumb/tooltip). Apply this generically to every type page so the
fix is not limited to the four types called out in the ticket, and so
future additions (e.g., any new `BackboneType`-derived datatype) do not
regress the same way. The change still lives in the Structure-tab
rendering code; the StructureDefinitions are untouched.

#### Justification

Treating this as a rendering-model fix instead of a string swap
eliminates the whole class of defects the ticket exemplifies. The
underlying cause is that the Structure tab picks one ancestor (the
ultimate root below the resource/type boundary) instead of the
immediate parent, and that heuristic will keep being wrong for any
multi-level inheritance. Showing the full chain — or at least the
immediate parent plus a tooltip of the full chain — is consistent with
what the UML tab already conveys visually, and matches how other parts
of the spec (e.g., the "inherits from" text on type narrative pages)
describe datatype lineage. The trade-off versus Disposition A is a
slightly larger code change and a modest visual change on type pages
that were previously rendering correctly, balanced against a more
durable fix.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" on the grounds that the authoritative views
(UML tab, underlying StructureDefinition, and type narrative) already
convey the correct inheritance, so the Structure tab's abbreviated
label is a presentation shorthand rather than an incorrect statement
about the model.

#### Justification

A reviewer could argue that the Structure tab is intentionally
simplified — it is meant to show *elements*, not the class hierarchy —
and that the "inherits from Element" line is a pragmatic shorthand for
"inherits the element-level machinery defined at the Element layer."
Under that reading, the UML tab is the correct place to consult type
lineage, and changing the Structure tab risks destabilising a widely
referenced visual without material benefit. This is, however, a weak
position: it requires every reader to know that the Structure tab's
inheritance label is not literal, which is exactly the confusion the
ticket documents.

---

### Recommendation

**Recommended disposition:** B

Disposition A is correct and low-risk, but Disposition B addresses the
root cause of the defect rather than patching the four datatypes the
reporter happened to notice. The Structure tab and UML tab are two
views of the same model, and users reasonably expect them to agree;
making the Structure-tab rendering walk (or at least surface) the
immediate parent type in all cases prevents the same bug from silently
returning for any future `BackboneType`-derived (or otherwise
multi-level) datatype. The work group should push back only if there
is a specific reason the Structure tab must keep collapsing the
hierarchy — none was argued in the ticket, and no Zulip or GitHub
discussion surfaced any such constraint. Disposition C should be
rejected: it asks the reader to treat one tab on a spec page as
non-literal, which is not a standard users should be expected to meet.
