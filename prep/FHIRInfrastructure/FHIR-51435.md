## Summary

Eric Haas requests that the cardinality of `ImplementationGuide.definition.resource.groupingId` be relaxed from `0..1` to `0..*` so that a single resource or example listed in an IG can belong to more than one grouping. His motivating example is an Implementation Guide where a `Questionnaire` example also serves as a `Signature` example; under the current `0..1` cardinality the IG author has to pick one grouping and add a manual note in the description ("(See the Signature examples as well)") to point readers at the other group. The ticket itself acknowledges a known downstream issue: the current HL7 artifact-list rendering does not support a resource appearing in multiple groupings, so a metadata change here would need a corresponding rendering change.

There are no other Jira tickets cross-referenced from FHIR-51435. The only other source-system reference is the originating Zulip thread in `#IG creation` (see "Related Zulip Discussions" below), where Lloyd McKenzie pushed back that grouping currently drives ToC order and section numbering and that polyhierarchy "isn't supported right now"; he suggested the question be revisited only after the artifacts page is reworked to be defined via `Composition`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the cardinality of `ImplementationGuide.definition.resource.groupingId` from `0..1` to `0..*` in the FHIR R6 build of the `ImplementationGuide` resource. Update the element definition and short/comment text to make explicit that a resource or example may appear in more than one grouping (polyhierarchy), that the order of appearance in the rendered artifact list / ToC is determined by the order of `groupingId` values, and that section numbering when a resource appears in multiple groupings is governed by the IG-publisher / rendering tooling rather than by the resource itself. Coordinate the change with the IG publisher / artifacts-page rendering owners so that the rendered Artifacts page can either duplicate the entry in each referenced grouping or render a single entry annotated with all its groupings.

#### Justification

The change is a non-breaking cardinality relaxation (`0..1` → `0..*`); existing IG instances with at most one `groupingId` remain valid. The use case Eric describes — a single example serving multiple semantic roles in an IG — is real and is independently confirmed in the Zulip thread by another implementer (Rutt Lindström) who hit the same wall on the Artifacts page and had to fall back to hand-written narrative cross-links. As Eric noted in the thread, the resource change would not ship until R6, which gives downstream tooling (IG publisher, HL7 artifacts page renderer) a full cycle to absorb the polyhierarchy. Doing the metadata change now also signals the direction of travel and lets the rendering work be scoped against a known target.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `ImplementationGuide.definition.resource.groupingId` at `0..1` for R6 and instead define a standard FHIR extension (e.g. `http://hl7.org/fhir/StructureDefinition/implementationguide-additional-grouping`, cardinality `0..*`, type `id`, context `ImplementationGuide.definition.resource`) that lets an IG author list any number of *secondary* groupings beyond the primary `groupingId`. The primary `groupingId` retains its current role of driving ToC order and section numbering for that resource; the extension is purely advisory and intended for tools (such as the Artifacts-page renderer) that want to surface the resource under additional groupings. As a companion, log a separate IG-publisher / artifacts-page rendering issue tracking the eventual move to a `Composition`-based artifacts page that would natively support polyhierarchy and at that point promote the extension into the core element with `0..*` cardinality.

#### Justification

This directly addresses Lloyd McKenzie's concern in the Zulip thread that `groupingId` currently drives ToC order and section numbering and that "polyhierarchy isn't supported right now." Keeping the primary grouping single-valued preserves the existing deterministic ordering and section-numbering semantics — and the existing rendering pipeline — while still giving Eric and Rutt's use case a first-class, machine-readable home rather than free-text "(see also …)" notes in the description. Extensions are the standard FHIR mechanism for adding optional, additive metadata without forcing every consumer / renderer to handle a new cardinality, and the path forward to eventually fold the extension into the base element once the `Composition`-based artifacts page lands matches the staging Lloyd suggested.

---

### Disposition C: Decline

#### Proposal

Decline the request and close as "Persuasive with mod" / "Not Persuasive — out of scope until artifacts-page rework," with the explicit rationale that the proposed cardinality change is a metadata change in name only: until the IG-publisher artifacts-page rendering pipeline is reworked to be `Composition`-driven, the resource cannot meaningfully express multi-grouping membership in any rendered IG, and the existing single-`groupingId` plus narrative cross-link pattern (which both Eric and Rutt are using today) remains the supported approach. Suggest the reporter raise (or follow) a separate ticket against the IG publisher / artifacts-page rendering owners to track the `Composition`-based rework, and revisit this Jira once that work is scheduled.

#### Justification

Lloyd McKenzie's objection in the Zulip thread is substantive: `groupingId` is not just a tag — it currently anchors ToC order and section numbering, and the rendering layer assumes a single grouping per resource. Eric and Jose both ended the thread acknowledging that the real blocker is the artifact-index / rendering rework, not the cardinality on the resource. Loosening the resource cardinality without that rework would publish a constraint that no current tool can render correctly, which risks divergent vendor-specific renderings of the Artifacts page across IGs and creates a migration problem when the canonical rendering does eventually ship. Declining now (with a clear pointer to the rendering ticket) keeps the resource and its tooling in lockstep.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between the two real positions in the Zulip thread. It accepts the user need that Eric raised and Rutt independently confirmed — a resource genuinely belonging to more than one grouping is a recurring IG-authoring pattern, and forcing that need into a free-text "(see also …)" note in the description is a clear loss of machine-readable IG metadata. At the same time it preserves Lloyd's invariant that the *primary* grouping continues to drive ToC order and section numbering, so existing IG-publisher rendering doesn't have to change in lockstep with the R6 metadata change. Routing the additional groupings through a standard extension is the conventional FHIR move for "we want this as structured metadata, but not every consumer must handle it yet," and it gives the IG-publisher / artifacts-page rework a concrete data shape to render against once that rework is scheduled. Disposition A is acceptable in principle but ships a `0..*` core element ahead of the rendering pipeline that is supposed to honour it; Disposition C is defensible but leaves real implementer pain (and free-text cross-links in IG descriptions) in place with no migration path.
