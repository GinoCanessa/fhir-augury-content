## Summary

The reporter, François Peverali, finds the following sentence in the FHIR
search specification ambiguous (from `search.html#reference`):

> "Chaining is not supported when using the :identifier modifier, nor are
> chaining, includes or reverse includes supported for reference elements
> that do not have a `reference` element."

As written, the sentence reads as if chaining and the `:identifier` modifier
must never be combined, which would (incorrectly) forbid queries such as
`GET [base]/DocumentReference?encounter.account:identifier=7567867`. The
reporter notes that other implementers have raised the same confusion and
proposes rewording to make clear that the prohibition is on chaining
*after* the `:identifier` modifier, and that the second clause applies to
**Reference type** elements that lack a `reference` element. The proposed
replacement text is:

> "When using the :identifier modifier, chaining after the modifier is
> not supported. Additionally, chaining, *includes*, or *reverse includes*
> are not supported for **Reference type** elements that do not have a
> reference element."

The closely related ticket [FHIR-45252](https://jira.hl7.org/browse/FHIR-45252)
("Expectations for non Reference references") was raised earlier by Gino
Canessa, has resolution **Persuasive**, status **Applied**, and rewrites
exactly this same passage (along with broader updates to chaining,
include/revInclude, the `[type]` modifier, and a new Canonical References
section). FHIR-45252's replacement text uses "is not defined … though
servers MAY choose to implement" rather than the reporter's "is not
supported". Because FHIR-45252 has already been applied, FHIR-51598 may
be redundant in substance — but the Zulip discussion (notably by Simone
Heckmann) suggests that even FHIR-45252's wording does not unambiguously
distinguish "chaining before the modifier" from "chaining after the
modifier," which is exactly the ambiguity FHIR-51598 is trying to fix.

## Details

**Description:**

Within the specification, the reporter found the following sentence
confusing:

> "Chaining is not supported when using the :identifier modifier, nor are
> chaining, includes or reverse includes supported for reference elements
> that do not have a `reference` element."
> ([source](https://hl7.org/fhir/R4/search.html#reference))

As written, it sounds as if, in general, chaining and the `:identifier`
modifier should not be combined. Therefore, a query like the following
should be forbidden — although this is not the case:
`GET [base]/DocumentReference?encounter.account:identifier=7567867`.

The reporter and other implementers have raised similar questions, and
propose the rewording quoted in the Summary above. A discussion thread is
available on the Zulip `implementers` stream under topic
"No chaining AFTER :identifier modifier".

### [FHIR-45252: Expectations for non Reference references](https://jira.hl7.org/browse/FHIR-45252)

Surfaced via the Zulip thread (Gino Canessa explicitly pointed at this
ticket). FHIR-45252 was filed in April 2024 by Gino Canessa, was
resolved **Persuasive** and is now **Applied** (label `AwaitingMerge`),
and rewrites the very passage at issue in FHIR-51598 — replacing
"Chaining is not supported when using the :identifier modifier…" with
"Chaining is not defined when using the `:identifier` modifier, nor are
chaining, includes, or reverse includes defined for reference search
parameters that target elements without an explicit reference, though
servers MAY choose to implement any or all of them." It also broadens
the chaining section, the `[type]` modifier text, and adds a new
Canonical References include section. This makes FHIR-45252 the direct
prior art for FHIR-51598, and the two should be reconciled rather than
both applied independently.

The reporter (François Peverali) opened the thread on 2025-07-08 with
exactly the wording-change proposal that became this ticket, asking
whether anyone had objections to filing a change request for R6.
Michele Mottini reacted with a thumbs-up. Lloyd McKenzie replied that a
"change request to clarify would be fine." Gino Canessa pointed out
that FHIR-45252 already covers this passage, summarising the intent as
"we will not prohibit use of the modifier, but we also do not want to
set the expectation that it works." Simone Heckmann pushed back: even
the proposed/already-resolved language ("Chaining is not defined when
using the :identifier modifier") is still ambiguous about whether it
restricts chaining *before* the modifier vs. *after* it, and noted that
the `DocumentReference.encounter.account:identifier` example is in fact
clearly defined chaining-before-modifier. Gino conceded the point and
quoted the related text from `Search:Chaining (chained parameters)` —
"chained search parameters are not intended to work with search
modifiers. While their use is not prohibited, [use] Named Queries or

neither prohibit nor explicitly define modifier+chain combinations,
because at least one implementation supports it but many implementers
pushed back hard on defining the behaviour. No final wording consensus
was reached in the thread.

Participants: Francois Peverali, Michele Mottini, Lloyd McKenzie, Gino
Canessa, Simone Heckmann.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's exact wording for the `Reference` section of
`search.html#reference`, replacing the current sentence:

> "Chaining is not supported when using the :identifier modifier, nor
> are chaining, includes or reverse includes supported for reference
> elements that do not have a `reference` element."

with:

> "When using the `:identifier` modifier, chaining after the modifier is
> not supported. Additionally, chaining, *includes*, or *reverse
> includes* are not supported for **Reference type** elements that do
> not have a `reference` element."

Because FHIR-45252 (Applied) has already rewritten this same passage,
"Accept As Requested" must be carried out as an *amendment on top of*
FHIR-45252's text, not against the R4 wording quoted in the ticket.
Practically, the FHIR-45252 sentence "Chaining is not defined when using
the `:identifier` modifier, nor are chaining, includes, or reverse
includes defined for reference search parameters that target elements
without an explicit reference, though servers MAY choose to implement
any or all of them." would be split into two sentences along the lines
the reporter proposed: one constraining chaining strictly to *after*
`:identifier`, and one covering reference elements that lack an explicit
`reference`. The second sentence in `search.html#chaining` ("chained
search parameters are not intended to work with search modifiers …")
would be cross-linked so the two passages stay consistent.

#### Justification

The community Zulip thread — including Lloyd McKenzie's "change request
to clarify would be fine" — confirms that the existing wording is
genuinely ambiguous and that a clarification is welcome. The reporter's
framing ("chaining *after* the modifier") matches the actual normative
intent and the `DocumentReference.encounter.account:identifier` example
that implementers already use in practice. Pure editorial clarifications
of this sort are within the remit of FHIR Infrastructure and carry no
implementer burden.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-51598 as a **follow-up clarification on FHIR-45252** rather
than a standalone wording change. Specifically:

1. Mark FHIR-51598's resolution as "Persuasive with Modification" and
   re-use FHIR-45252's "is not defined … servers MAY choose to
   implement" framing (which is intentionally weaker than "is not
   supported").
2. Address Simone Heckmann's objection by *explicitly* distinguishing
   the two cases in `search.html#reference`, e.g.:
   - "Chaining **after** the `:identifier` modifier is not defined,
     though servers MAY choose to implement it."
   - "Chaining, `_include`, and `_revinclude` are not defined for
     reference search parameters that target elements without an
     explicit `reference` element (for example, `canonical` or `uri`
     elements), though servers MAY choose to implement them — see
     [Chaining (chained parameters)](search.html#chaining)."
3. Confirm that the existing example
   `DocumentReference.encounter.account:identifier` (chaining *before*
   `:identifier`) is permitted, and add it (or an equivalent) as an
   illustrative example so future readers are not misled.
4. Cross-link from `search.html#chaining` back to the `:identifier`
   modifier rule so both sections stay in sync.

#### Justification

This preserves the deliberate compromise documented by Gino in the
Zulip thread — "neither prohibited nor explicitly defined" — while
fixing the specific ambiguity the reporter and Simone both flagged.
Using "is not defined / MAY implement" instead of "is not supported"
matches what was already accepted in FHIR-45252, avoiding a regression
in tone (FHIR-51598's literal text reverts "is not defined" back to "is
not supported", which is stricter than what the WG chose). Adding a
worked example is the most reliable way to disambiguate
chaining-before-modifier from chaining-after-modifier without re-litigating
the underlying behaviour.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-51598 as "Not Persuasive — Already Addressed", citing
FHIR-45252 (Applied, Persuasive). The R6 build text already reads:

> "Chaining is not defined when using the `:identifier` modifier, nor
> are chaining, includes, or reverse includes defined for reference
> search parameters that target elements without an explicit reference,
> though servers MAY choose to implement any or all of them."

…which is, in the WG's view, clearer than the R4 wording the reporter
quoted. Recommend the reporter validate the R6 build text and re-open
only if a residual ambiguity remains.

#### Justification

The reporter cited the R4 page as the source of the confusion, but
FHIR-45252 has already rewritten that exact passage in the build, and
the WG has explicitly chosen "is not defined / MAY implement" over the
reporter's stricter "is not supported" phrasing. Declining keeps the
spec consistent with the WG's prior decision and avoids a near-immediate
re-edit of recently applied normative text.

---

### Recommendation

**Recommended disposition:** B

Disposition C undersells the issue: the Zulip discussion shows that
even after FHIR-45252's rewrite, careful readers (Simone Heckmann)
still cannot tell whether the rule restricts chaining *before* or
*after* the `:identifier` modifier. That is precisely the bug
FHIR-51598 is reporting, and it should not be closed as "already
addressed". Disposition A overcorrects in the opposite direction —
the reporter's literal text reintroduces the absolute "is not
supported" wording that FHIR-45252 deliberately moved away from, which
would walk back the WG's compromise. Disposition B keeps FHIR-45252's
"is not defined / MAY implement" framing (preserving the
implementer-pushback compromise Gino documented), splits the sentence
to make the before/after distinction explicit, and adds a worked
example (`DocumentReference.encounter.account:identifier`) so future
implementers do not have to re-derive the rule from prose. This is
the smallest change that actually resolves the reported ambiguity.
