## Summary

Jens Villadsen requests that the published transaction example Bundle in
the FHIR Core specification be expanded so it includes a conditional
reference between two entries — specifically, an additional `Patient`
POST whose payload contains a `Patient.link.other.reference` pointing
to another `Patient` that is also being created in the same
transaction. The reporter argues this is a missing illustration in the
existing transaction example and notes that the same example
enhancement would be applicable to R4, R4B, and R5.

There are no other Jira tickets cross-referenced to FHIR-40498. The
single related discussion is the 2025 `#implementers > PartOf
resolution` thread on chat.fhir.org, where implementers debated whether
conditional references in transaction Bundles resolve against the
server, within the bundle, or both — exactly the kind of ambiguity a
worked example in the spec could help disambiguate. That thread also
spawned a follow-on clarification ticket, FHIR-51130, which is not
formally linked to FHIR-40498 but is closely related.

## Details

**Description:**

The reporter proposes amending the existing transaction Bundle example
shown in the spec so the final entry is a `Patient` POST whose body
contains a conditional reference (in `Patient.link.other.reference`)
that resolves to another `Patient` being created earlier in the same
Bundle. The submitted JSON sketch (badly mangled by the Jira macro
renderer in the source content) shows a transaction Bundle with:

- Several `Patient` POSTs using `urn:uuid:` `fullUrl` values.
- `Patient` PUT/conditional-update style entries against
  `http://example.org/fhir/Patient/123` and `Patient/123a`.
- A `$lookup`-style `Parameters` resource with a `valueCoding`.
- A trailing `Patient` POST whose `link.other.reference` is the
  conditional reference the reporter wants demonstrated.

The reporter explicitly calls out: *"where the addition is the last
Patient POST in the transaction that contains a reference to a Patient
within the bundle. I think this part is missing out."* The reporter
also notes the example improvement is applicable to R4, R4B, and R5.

## Related Zulip Discussions

### [implementers > PartOf resolution](https://chat.fhir.org/#narrow/stream/implementers/topic/PartOf%20resolution/near/524484339)

Implementer Robert Mayore asked whether a transaction Bundle posting
two `Location` resources, where the second carries
`partOf.identifier` pointing at the first, would resolve correctly
against a FHIR server. Michele Mottini's first response was direct:
no — the bundle needs `entry.fullUrl` populated and
`partOf.reference` to point at that `fullUrl` for in-bundle
resolution to work. The conversation then split. Gino Canessa quoted
the spec section *Resolving references in Bundles → Resolving
conditional references* and argued the rules are written so that the
search runs *on the server*, regardless of whether the target also
appears in the same bundle. Rik Smithies and Paul Church pushed back
with the (long-held) implementer reading that conditional references
should only resolve against pre-existing server state, that
`fullUrl` is the proper mechanism for in-bundle resolution, and that
allowing both opens unresolved precedence and ordering questions
(e.g., what if the same bundle also updates the candidate match so it
no longer matches?). Elliot Silver agreed that he has always expected
conditional references to resolve only against the server. Gino noted
that Synthea-style transactions assume in-bundle resolution at least
some of the time, and submitted FHIR-51130 to ask FHIR-I for an
explicit clarification, linking back to this thread. He explicitly
called out FHIR-40498 as *the* existing ticket asking for an example
in this area, while noting no other ticket asks to clarify the
underlying rules — i.e., FHIR-40498's worked example would partially
land FHIR-51130's resolution.

Participants: Robert Mayore, Michele Mottini, Gino Canessa, Rik
Smithies, Paul Church, Elliot Silver.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the published transaction example Bundle in the FHIR Core
specification (under `source/bundle/` in `HL7/fhir`, with the rendered
prose on the corresponding top-level `source/*.html` Bundle examples
page) by adding a final `Patient` POST entry whose resource carries a
`Patient.link` whose `other.reference` value targets *another* `Patient`
created earlier in the same transaction Bundle. Demonstrate the two
canonical in-bundle resolution mechanisms in the same example — at
minimum (a) a `urn:uuid:`-style `entry.fullUrl` reference, and ideally
also (b) a conditional reference (`Patient?identifier=…`) — so that
implementers can see both forms side by side. Mirror the change onto
the `R4` and `R4B` maintenance branches as the reporter requests.

#### Justification

The ticket is narrowly scoped, low-risk, and addresses a documented
pain point: the 2025 `PartOf resolution` Zulip thread shows that
implementers continue to disagree about what conditional references do
in a transaction Bundle, and even the participant who quoted the spec
(Gino Canessa) acknowledged the prose alone is not unambiguous. A
concrete worked example is a cheap and high-leverage clarification
that changes no normative content and breaks no existing implementer.
The HL7/fhir briefing confirms the change lives entirely in the
authoring root (`source/`) and matches the `Update narrative / prose`
recipe, with the only minor twist being that example data is also
edited (alongside the rendered narrative). No StructureDefinition
snapshot or generated content is touched.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding to the *existing* transaction example, add a **new,
purpose-built** example file under `source/bundle/` (e.g.,
`bundle-transaction-conditional-references.xml` / `.json`) that is
solely about reference resolution inside a transaction. The example
should illustrate, in clearly labelled entries:

1. An `entry.fullUrl` (`urn:uuid:`) reference resolved within the
   bundle (the canonical pattern).
2. A conditional reference of the form
   `Patient?identifier=…` and a discussion of the spec rule that the
   search runs on the server.
3. A side comment (in the rendered prose) about what happens when the
   conditional reference's target is also being created/updated in the
   same bundle — explicitly cross-linking to FHIR-51130's clarification
   request so readers understand which behaviour is normative and which
   is implementation-defined.

Add a new section, "Resolving references in transaction Bundles —
worked example", on the relevant `source/*.html` page that already
documents resolving references in Bundles, and link to the new
example file. Apply on `master`/R5 first; backport to R4B and R4
only if the working group agrees the prose is settled.

#### Justification

The Zulip discussion shows that the *underlying confusion* is not just
"how do I write the syntax" but "what does the server actually do" —
i.e., people are reading the same paragraph and walking away with
different mental models (in-bundle vs. server-only vs. both). Squeezing
this onto the end of the existing transaction example risks giving
implementers a single isolated snippet that *looks* authoritative
without any of the surrounding prose explaining the resolution rules,
which would arguably make the situation worse. A dedicated example
file plus a paired prose section keeps the explanation co-located with
the worked syntax, and is also more compatible with the briefing's
guidance to keep authoring under per-resource folders (`source/bundle/`)
and rendered prose on the matching top-level page. The briefing's
HTML-anchor gotcha applies here: any new fragment anchor introduced
should be repo-grep'd for inbound links before merging. Gating the R4
/ R4B backport on consensus also keeps a purely-editorial change from
becoming an unintentionally normative one on the maintenance branches.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-40498 as *Persuasive with Modification — duplicate-of /
subsumed-by FHIR-51130* and close as "not persuasive" on its own
merits. The rationale is that adding a worked example before the
underlying normative question (do conditional references in
transaction Bundles resolve against the server, against the bundle,
or both?) is settled would risk codifying one reading via example.
Once FHIR-51130 produces an unambiguous clarification of *Resolving
references in Bundles → Resolving conditional references*, the example
work proposed here can be done as part of that ticket's resolution.

#### Justification

The Zulip thread surfaces a real, current disagreement among
experienced implementers (Mottini, Canessa, Smithies, Church, Silver)
about what the spec actually requires. Until that disagreement is
resolved by FHIR-I, any example added to the spec will tacitly endorse
one interpretation, which is exactly the failure mode the existing
prose already exhibits. Closing FHIR-40498 in favour of FHIR-51130
also reflects the FHIR-I house style of pairing normative
clarifications with their illustrating examples in a single
disposition, rather than landing the example first and the
clarification later.

---

### Recommendation

**Recommended disposition:** B

The original request (Disposition A) is well-meaning and the change is
small, but the 2025 Zulip thread is strong evidence that the *existing*
transaction example doesn't suffer from "missing one more entry" — it
suffers from being the only on-page artefact for a topic where the
normative prose itself is contested. Tacking a conditional reference
onto the end of the current example would land an isolated snippet
with no explanatory framing, which is unlikely to actually settle the
implementer confusion that motivates the request.

Disposition C is defensible as a process matter, but it leaves
implementers without any worked example for an unbounded amount of
time while FHIR-51130 makes its way through FHIR-I. Given that the
*shape* of the example (`fullUrl` for in-bundle vs. conditional
reference for server-side) is uncontroversial — even the participants
who disagree about the precedence rule agree on what the two
mechanisms each *do* in isolation — a dedicated, clearly-framed example
plus accompanying prose (Disposition B) can be authored now without
prejudicing FHIR-51130's outcome, and the prose can simply note the
open clarification question and link to FHIR-51130. That gives
implementers immediate value, keeps the example out of the disputed
precedence territory, and aligns with the HL7/fhir briefing's guidance
to keep authoring under `source/bundle/` with rendered prose on the
matching `source/*.html` page.
