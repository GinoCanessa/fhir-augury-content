## Summary

Hanhong Lu observes that `Bundle.request` and `Bundle.response` in the core
FHIR Bundle resource only model a subset of the HTTP request/response headers
documented on the FHIR `http.html` page, and asks whether the resource should
be extended to cover more of them. Two concrete questions are posed:

1. For `Bundle.request`, should there be a generic `header` / `customHeader`
   element so that callers using batch / transaction / history bundles can
   convey the [custom HTTP headers](https://build.fhir.org/http.html#custom)
   defined in the spec (e.g., `X-Provenance`, `X-Request-Id`, `Prefer`,
   handling preferences) rather than only the few currently-modeled fields
   (`method`, `url`, `ifNoneMatch`, `ifModifiedSince`, `ifMatch`,
   `ifNoneExist`)?
2. For `Bundle.response`, should there be a `contentLocation` element to
   carry the `Content-Location` header that asynchronous request patterns
   rely on, alongside the currently-modeled `status`, `location`, `etag`,
   `lastModified`, and `outcome` fields?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir`, edit `source/bundle/`'s Bundle StructureDefinition
`<differential>` to add the two requested capabilities directly to the
core resource:

1. On `Bundle.entry.request`, add a repeating BackboneElement
   `Bundle.entry.request.header` with children `name : string [1..1]` and
   `value : string [1..1]`, cardinality `0..*`, with documentation
   restricting it to headers not already representable by the typed
   fields and pointing at `http.html#custom` for the catalog of named
   custom headers (`X-Provenance`, `X-Request-Id`, `Prefer`,
   `handling=...`, etc.).
2. On `Bundle.entry.response`, add `Bundle.entry.response.contentLocation
   : uri [0..1]` with documentation tying it to the `Content-Location`
   HTTP header used by async patterns and bulk-data style responses,
   parallel to the existing `Bundle.entry.response.location` element.

Update `source/bundle/bundle.html`, `bundle-definitions.html`, and
`source/http.html` to describe the new elements and update the
"headers carried in Bundle" table. Add at least one transaction example
under `source/bundle/bundle-examples-*.xml` that demonstrates a custom
header on a request entry and a `contentLocation` on a response entry.
Do **not** edit `<snapshot>` blocks — the publisher regenerates them.

#### Justification

The request is grounded in the spec's own `http.html` page: FHIR
explicitly enumerates custom headers and the asynchronous interaction
pattern's reliance on `Content-Location`, but `Bundle.request` /
`Bundle.response` only model a partial subset. Modeling them as
first-class elements means batch/transaction/history payloads are
self-describing without implementers having to juggle out-of-band HTTP
state or extensions. `contentLocation` in particular has a clear,
narrowly-scoped HTTP analog and naturally parallels the existing
`location` element, so the addition is symmetric and low-ambiguity.

### Disposition B: Alternative Approach

#### Proposal

Resolve the underlying need with **standard extensions in
`HL7/fhir-extensions`** rather than new core Bundle elements:

1. Define `http://hl7.org/fhir/StructureDefinition/bundle-request-header`
   (complex extension with `name` and `value` sub-extensions, applicable
   to `Bundle.entry.request`, cardinality `0..*`).
2. Define
   `http://hl7.org/fhir/StructureDefinition/bundle-response-contentLocation`
   (simple `uri` extension on `Bundle.entry.response`).

In `HL7/fhir`, leave the Bundle `<differential>` unchanged but update
`source/bundle/bundle.html` and `source/http.html` to point at the two
extensions and to clarify the design rule: only headers that have first-
class semantics on Bundle's wire contract (conditional headers, ETag,
Last-Modified, Location, status) are modeled as core elements; everything
else, including custom headers and async-only headers, is carried via
extension.

#### Justification

Bundle is a foundation resource touched by every server; adding even a
0..* optional element bumps the Bundle resource version and forces every
implementer to re-evaluate their parsers and validators for very narrow
use cases. Custom HTTP headers are by definition open-ended — modeling
them as a core element either invites unbounded growth or duplicates the
extension mechanism that already handles open-ended payload shape.
`Content-Location` for async is real but is also exactly the kind of
"only used by some interaction styles" header that the extension
mechanism was designed for. This keeps the core Bundle stable while
giving implementers a sanctioned, discoverable place to put the data,
and aligns with the FhirCore playbook's preference for using extensions
for genuinely optional capabilities. Trade-off vs. A: extensions are
slightly less discoverable in tooling and require the implementer to
opt in to the extension definitions, but they avoid a normative-
resource bump.

### Disposition C: Decline

#### Proposal

Decline as **already addressed by existing mechanisms**, with no change
to the Bundle resource and no new extensions published. Resolution
comment: HTTP headers that affect the wire-level behavior FHIR's REST
contract cares about (`If-None-Match`, `If-Modified-Since`, `If-Match`,
`If-None-Exist`, ETag, Last-Modified, Location, status) are already
modeled on `Bundle.entry.request` / `Bundle.entry.response`. Headers
that are not modeled — custom headers like `X-Provenance`,
`X-Request-Id`, `Prefer`, and `Content-Location` for async patterns —
can be carried today via the standard extension mechanism on
`Bundle.entry.request` and `Bundle.entry.response`, or, for async, via
the asynchronous interaction pattern documented in
`source/async.html` and the `Subscription` / bulk-data specifications
where appropriate.

#### Justification

Bundle deliberately models only the headers that materially change how
a server processes a transaction/batch entry (conditional create/update,
optimistic concurrency, redirect/location for create). Custom headers
and async-pattern headers fall outside that scope by design; FHIR's
extension mechanism is the canonical answer for "we want to carry an
additional optional value on this element," and the bulk-data /
subscription / async-interaction specs already define the wire contract
for `Content-Location`-based polling. Adding more headers to the core
resource invites scope creep — every implementer has a favorite header
they'd like first-class — without solving a problem the spec hasn't
already addressed.

### Recommendation

**Recommended disposition:** B

The ticket asks two distinct questions but both reduce to "should
optional, narrowly-used HTTP headers live on core Bundle?" Disposition C
underestimates the friction of telling implementers "use extensions" when
the spec doesn't actually publish those extensions — they end up either
inventing local ones or shipping the data out-of-band. Disposition A
overcorrects in the other direction: a generic `header`/`customHeader`
BackboneElement on a foundation resource is exactly the kind of open-
ended optional addition the FhirCore guidance steers away from, and a
`contentLocation` core element duplicates a value that the bulk-data /
async specs already describe at the wire level. Disposition B threads
the needle: it gives implementers a sanctioned, discoverable
representation for both custom request headers and async
`Content-Location` without bumping the Bundle resource version or
opening the door to a steady drip of "please add header X to Bundle"
tickets. The Bundle prose in `HL7/fhir` is updated to point at the
extensions and to articulate the design rule, which is itself useful
documentation. Workgroup should confirm with the implementer (Hanhong
Lu) that extensions adequately solve the underlying integration need
before finalizing.
