## Summary

The reporter (Sebastian Günther) flags that on the FHIR Core continuous
build (`build.fhir.org`) the "downloads" section advertises packages that
support "the FHIR specification" but both the link text and the link
targets still reference the R5 package IDs (`hl7.fhir.r5.core`,
`hl7.fhir.r5.expansions`, `hl7.fhir.r5.examples`, `hl7.fhir.r5.search`,
`hl7.fhir.r5.corexml`). Since the continuous build now represents R6,
the labels and `https://build.fhir.org/hl7.fhir.r5.*.tgz` URLs should
all be updated to `hl7.fhir.r6.*` so that readers actually download the
package matching the spec they are viewing. There are no cross-referenced
Jira tickets, Zulip threads, or GitHub items to add additional context.

## Details

**Description:**

The download links are incorrect:

> There are several packages that support the FHIR specification:
> - **`hl7.fhir.r5.core`** (https://build.fhir.org/hl7.fhir.r5.core.tgz): Contains all the resources needed to test conformance to the FHIR specification, and/or generate code
> - **`hl7.fhir.r5.expansions`** (https://build.fhir.org/hl7.fhir.r5.expansions.tgz): Contains the expansions of the subset of the value sets the have a required binding (other than those that can't be expanded at all)
> - **`hl7.fhir.r5.examples`** (https://build.fhir.org/hl7.fhir.r5.examples.tgz): All the resources that are defined as part of the FHIR specification
> - **`hl7.fhir.r5.search`** (https://build.fhir.org/hl7.fhir.r5.search.tgz): Just the search parameters. In this package, the combined search parameters are uncombined so that there is a set of search parameters for each resource (performance considerations)
> - **`hl7.fhir.r5.corexml`** (https://build.fhir.org/hl7.fhir.r5.corexml.tgz): The same content as hl7.fhir.r5.core, but with the resources in XML, not JSON

Link text and URLs point to R5, should be R6 in both.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Core downloads page so that, for the R6 build, every
occurrence of `hl7.fhir.r5.<x>` (both the rendered link text and the
target URL) is replaced with `hl7.fhir.r6.<x>` for the five packages
listed in the ticket: `core`, `expansions`, `examples`, `search`, and
`corexml`. Concretely, the five anchors that today read
`https://build.fhir.org/hl7.fhir.r5.<x>.tgz` (and display the matching
`hl7.fhir.r5.<x>` text) become `https://build.fhir.org/hl7.fhir.r6.<x>.tgz`
with `hl7.fhir.r6.<x>` text. No descriptive prose needs to change.

#### Justification

This is a Technical Correction, not a design change: the prose around
the links already says "the FHIR specification" (i.e., the version being
built), so leaving R5 IDs in an R6 build mismatches the surrounding
narrative and silently hands users the wrong tarball. Renaming version
identifiers in the downloads page is the standard housekeeping step at
each major release and should be in lock-step with the package publisher
(which already produces `hl7.fhir.r6.*` artifacts on `build.fhir.org`).

---

### Disposition B: Alternative Approach

#### Proposal

Replace the hard-coded `r5` segment in the downloads page with a
template/macro driven by the active publication's FHIR version (e.g.,
`hl7.fhir.{{fhir-version-id}}.<x>` where `fhir-version-id` is `r6` for
the current build and would be `r7` next). Apply the same templating to
the rendered link text. As part of this change, fix the current R5/R6
mismatch listed in the ticket as the first concrete instance.

#### Justification

The bug in this ticket is a recurrence of the same class of mistake
made at every major-version transition (the page was last touched for
the R4→R5 cut and missed the R5→R6 cut). Driving the version segment
from a single configured value removes the manual rename step from
future releases and keeps text and URL automatically consistent. The
trade-off versus Disposition A is a slightly larger change to the
downloads page authoring file plus a one-time wiring task in the
publisher, but it eliminates this entire defect class going forward.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive / Already Addressed" on the basis
that the R5 package IDs on this page are intentional historical links
to the R5 release tarballs, and users wanting R6 should navigate via
the R6-specific downloads page rather than via the build copy.

#### Justification

This argument would only hold if (a) the page in question were
explicitly the R5 downloads page, or (b) the surrounding prose
clarified that the listed packages are the R5 packages. Neither is the
case in the cited content — the prose generically references "the FHIR
specification" in the context of an R6 build. Declining would leave
users on `build.fhir.org` actively pointed at the wrong release
artifacts. There is no defensible reason to leave the page as-is.

---

### Recommendation

**Recommended disposition:** A (with a follow-up to consider B)

Disposition A is the minimum action required to honour a `Highest`
priority Technical Correction filed against an obviously incorrect set
of links on the FHIR Core build site, and it can be applied immediately
without coordinating any downstream change. Disposition B is the right
*long-term* fix and is worth filing as a separate ticket against the
publisher / downloads page templating, but bundling it into this
correction risks delaying the user-visible fix while the templating
work is scoped. Disposition C is not defensible: the existing links
demonstrably point at the wrong release.
