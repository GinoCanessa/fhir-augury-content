## Summary

Brian Postlethwaite is asking the FHIR Infrastructure work group to clarify
and expand the StructureMap / FHIR Mapping Language specification on **how
and when target properties (the `context.element` referenced on the
left-hand side of a target rule) are created** during a mapping execution.
The current specification does not state the cardinality of the `target`
components, leaving it ambiguous whether `context`/`element` are required
on a target rule or optional. The reporter wants a dedicated section that
explains:

- what "auto-created elements" means in the mapping engine,
- how the type of an auto-created element is determined,
- when an explicit `create()` call is required (the reporter suspects this
  is essentially limited to choice `[x]` types where the engine cannot
  infer which type to instantiate), and
- how auto-creation interacts with constraints declared on the underlying
  type.

A second, related ask is that the **Target Transform** definition does not
currently show its FHIRPath usage the way the Source Context definition
does. The reporter proposes adding an optional grammar line such as:

> *[context.element =] transform_code(parameters...) [as variable] {list_modes}*
> or
> *[context.element =] (FHIRPath) [as variable] {list_modes}*

and notes that even this draft "doesn't feel clear enough either," so the
work group is being asked to land on a clearer, normative form.

There were **no cross-referenced Jira tickets, Zulip threads, or GitHub
items** returned for this ticket through FHIR Augury, so this report is
grounded solely in the ticket's own description.

### Disposition A: Accept As Requested

#### Proposal

Update the StructureMap / FHIR Mapping Language pages (the `structuremap`
resource page and the dedicated mapping-language / mapping-tutorial pages
in the FHIR Core spec) as follows:

1. **Cardinality of `StructureMap.group.rule.target.context` and
   `…target.element`:** state the cardinality explicitly. Make
   `context` `0..1` and `element` `0..1` (matching today's de-facto
   behaviour where a target rule may be a pure side-effecting transform
   such as `evaluate(...)` with no element assignment), and document the
   condition under which each is required (i.e., `element` is required
   whenever the transform produces a value that needs to land on a
   named element of `context`).
2. **New "Auto-created target elements" section** in the Mapping Language
   page that normatively defines:
   - When the engine auto-creates the element named by
     `target.element` on `target.context`. The default rule: if the
     named element does not yet exist on `context` and the transform
     produces a value (or the rule has child rules that will populate
     it), the engine instantiates it.
   - How the **type** of the auto-created element is determined: from
     the StructureDefinition of `context` resolving `element`. For
     non-polymorphic elements the type is unambiguous; for polymorphic
     (`[x]`) elements the type cannot be inferred and must be supplied
     by an explicit `create('TypeName')` transform (or by a transform
     whose return type pins the choice, e.g. `cast`).
   - That **`create()` is only required for choice types** (and any
     other element where the engine cannot determine a single concrete
     type — e.g., abstract types or open extensions), and that for all
     other elements `create()` is optional and equivalent to relying on
     auto-creation.
   - **Interaction with type constraints**: auto-creation respects
     constraints from the underlying StructureDefinition (slicing,
     fixed values, pattern values, required bindings) — i.e., the
     engine instantiates the constrained type and applies fixed/pattern
     values where defined; if constraints conflict with the value the
     transform produces, the rule fails.
3. **Target Transform grammar:** add the optional `[context.element =]`
   prefix and the FHIRPath alternative to the canonical grammar block,
   matching the Source Context style. A clearer two-line form:

   ```
   target  ::= [context '.' element '='] (transform | '(' fhirpath ')')
                [ 'as' variable ] { listMode }*
   ```

   accompanied by a worked example for each shape (transform call,
   parenthesised FHIRPath, side-effect-only `evaluate`).

#### Justification

This is the minimum change needed to make the spec match what
implementations already do (the major mapping engines — Matchbox, the
HAPI/Java reference implementation, the .NET reference implementation,
Aidbox — already auto-create non-choice elements and require `create()`
only for `[x]` slots), and the change is editorial / normative-clarifying
rather than introducing new semantics. The reporter is the lead
implementer of one of those engines, so the proposal reflects observed
runtime behaviour. Pinning down `target.context`/`target.element`
cardinality is also a prerequisite for any conformance tooling that
validates StructureMap instances. The grammar tweak parallels the Source
Context entry the reporter cites, which keeps the page internally
consistent.

### Disposition B: Alternative Approach

#### Proposal

Adopt the same normative clarifications as Disposition A (cardinality,
auto-creation rules, choice-type handling, constraint interaction) **but
do not modify the canonical Target Transform grammar block**. Instead:

- Leave the existing one-line grammar in place.
- Add an explanatory subsection immediately beneath it titled
  "Optional `context.element =` prefix and FHIRPath form" that shows
  the same two shapes the reporter sketched, with examples, and
  explicitly notes that the FHIRPath form is shorthand for an
  `evaluate()` transform.
- Add a non-normative table mapping each transform_code (`copy`,
  `create`, `truncate`, `evaluate`, `cast`, …) to whether it requires
  `target.element`, whether it can be replaced by the `(FHIRPath)`
  shorthand, and whether it triggers auto-creation.

#### Justification

Touching the grammar line risks invalidating downstream parsers and
documentation generators that key off the existing wording, and the
reporter himself notes the proposed grammar "doesn't feel clear enough
either." Keeping the canonical grammar stable and putting the new
material in prose plus a transform-by-transform table addresses the
substantive ambiguity (when do I need `create()`? do I need
`context.element`?) without forcing every consumer of the grammar
fragment to re-parse it. The cost is that the grammar block continues to
under-describe the FHIRPath form, but it is mitigated by the new
subsection that sits directly under it.

---

### Disposition C: Decline

#### Proposal

Decline the request as scoped. Close as **Persuasive with Modifier** /
**Not Persuasive** with the rationale that the StructureMap mapping
language is already sufficiently described for current implementers, and
that finer-grained semantics (auto-creation triggers, choice-type
handling, FHIRPath-vs-transform equivalence) are appropriate for an
implementation guide / cookbook rather than the normative resource page.
Point the reporter at the existing tutorial and at the test suite as the
operational source of truth, and suggest that any further clarifications
be raised against the Mapping Language tutorial page rather than the
StructureMap resource.

#### Justification

The reporter's own draft grammar acknowledges the proposed text
"doesn't feel clear enough," which suggests the work group risks
churning the spec without converging. If the major engines already
agree on the behaviour, an out-of-band cookbook entry can capture it
without touching normative text and without the cross-engine review
burden a normative change carries. This disposition is defensible
purely on scope-control grounds; its weakness is that it leaves the
`target.context`/`target.element` cardinality genuinely undefined,
which is a real interoperability gap.

---

### Recommendation

**Recommended disposition:** A

The ambiguity the reporter is pointing at — particularly undefined
cardinality on `target.context` / `target.element` and the missing
normative description of when `create()` is required — is a real
interoperability gap, not a documentation polish item. Two independent
mapping engines can today produce different output for the same
StructureMap because the spec is silent on auto-creation and on
choice-type handling, and that is exactly the class of ambiguity the
work group should close. Disposition C leaves that gap open.
Disposition B fixes the semantics but ducks the grammar, which means
the Target Transform definition will continue to omit its FHIRPath
form and stay out of step with the Source Context entry that the
reporter cites — an inconsistency that future readers will keep
re-raising. Disposition A addresses both the semantic and the
presentation problems in one pass, aligns the spec with what major
implementations already do, and the grammar wording can be refined in
review (the reporter himself invites that). The work group should
adopt A and assign it to a spec editor familiar with the Mapping
Language pages.
