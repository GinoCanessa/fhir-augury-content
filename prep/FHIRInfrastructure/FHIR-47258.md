## Summary

The reporter (Weiyu Zhang) asks a clarifying question about the
**FHIR Application Feature Framework Implementation Guide**'s Feature
Query operation. Specifically, while the IG clearly defines the
*structure* of feature query inputs and outputs — i.e., how the
`feature + context + value` triples are conveyed on the operation —
the IG does not appear to define what concrete values `context` should
take. The reporter draws an analogy to `ValueSet.$expand`, where
context values are typically supplied by downstream IGs that depend on
the base specification, and asks whether the same delegation model is
intended here. The ticket has been triaged with the resolution
**"Considered - Question answered"**, directing the reporter to the
`FeatureDefinition` logical model
(<https://build.fhir.org/ig/HL7/capstmt/StructureDefinition-FeatureDefinition.html>),
which is the artifact intended to describe the legal context (and
value) values for any given feature.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items recorded for this issue in the augury index.

## Details

**Description:**

> While this IG has clearly defined how the context value should be
> specified (i.e. how to specify feature + context + value combos on
> the feature query operation input/output), I'm not sure what the
> exact value of context should be. Is the exact value for context
> supposed to be defined by subsequent IGs that will be dependent on
> this IG (like the context for ValueSet.$expand)?

**Resolution Description:**

> The context for any specific feature will be described using the
> FeatureDefinition logical model:
> <https://build.fhir.org/ig/HL7/capstmt/StructureDefinition-FeatureDefinition.html>

## Proposed Dispositions

> Note: this ticket is already resolved as **"Considered - Question
> answered"**. The dispositions below frame the choices the workgroup
> faces if it wants to revisit the ticket — i.e., whether the existing
> answer is sufficient, or whether the IG should do more to make the

### Disposition A: Accept As Requested

#### Proposal

Confirm and ratify the existing resolution: the legal values of the
`context` parameter on the Feature Query operation
(`$feature-query`) are *not* enumerated in the Application Feature
Framework IG itself. Instead, each concrete feature is described by a
`FeatureDefinition` logical model instance
(<https://build.fhir.org/ig/HL7/capstmt/StructureDefinition-FeatureDefinition.html>),
and that instance is the authoritative source for which `context`
values (and which `value` values) are valid for that feature.
Downstream IGs that define features are expected to publish
`FeatureDefinition` instances that pin the context vocabulary, in the
same way downstream IGs constrain `ValueSet.$expand` context.

To make the answer discoverable without reading the Jira resolution,
add a short narrative note to the Feature Query operation page in the
Application Feature Framework IG (in `HL7/capstmt`) saying:

- `context` values are not enumerated by this IG;
- each feature's permitted `context` values are described by the
  feature's `FeatureDefinition`;
- with a hyperlink to the `FeatureDefinition` logical model.

Close the ticket as **Persuasive** (or keep the current
"Considered - Question answered") with that small documentation
clarification.

#### Justification

This is consistent with how FHIR handles other open-vocabulary
operation parameters (e.g., `ValueSet.$expand` context), and matches
the answer that has already been given to the reporter. The framework
IG is intentionally a *framework* — it defines the *shape* of feature
queries, not the catalog of features — so delegating the context
vocabulary to per-feature `FeatureDefinition` resources is the correct
layering. The only gap is discoverability: a reader of the operation
page today does not learn this without finding the Jira ticket. A
sentence-level narrative addition closes that gap without changing
any normative content.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the delegation model from Disposition A, but go further by
**formalising the relationship between the operation parameter and
`FeatureDefinition`** in the IG. Specifically:

1. On the `$feature-query` `OperationDefinition`, update the binding /
   documentation for the `context` input and output parts to reference
   `FeatureDefinition` and explain that legal values are drawn from
   the `FeatureDefinition.context` element of the matching feature.
2. Add a normative paragraph to the operation page describing the
   resolution algorithm: given a `feature` URI, locate the
   `FeatureDefinition` resource, and use its declared context (and
   value) elements to validate / interpret the request.
3. Add at least one worked example in the IG showing a complete
   feature + context + value triple alongside the `FeatureDefinition`
   that defines its legal contexts.

#### Justification

Disposition A answers the reporter's literal question but leaves the
binding implicit. Implementers who do not read the Jira history will
still be uncertain about *how* to discover legal context values at
runtime. Making the link normative and example-driven removes that
ambiguity, raises the testability of the operation, and aligns the
IG with FHIR's general preference for machine-discoverable
conformance over prose-only conventions. The cost is modest — a few
paragraphs and one example in the framework IG — and it does not
change the underlying delegation model.

---

### Disposition C: Decline

#### Proposal

Decline to make any further change. The ticket is a **Question**, not
a defect or change request, and the workgroup has already answered it
in the resolution: legal `context` values are described by
`FeatureDefinition`. No spec change is required, and the existing
"Considered - Question answered" resolution stands.

#### Justification

The IG is explicitly a framework that does not own a catalog of
features; enumerating context values in the framework IG would
contradict its scope. The reporter's question has been answered with
a direct pointer to the `FeatureDefinition` logical model, which is
the canonical mechanism. Spending workgroup cycles on additional
narrative is low-value relative to other Application Feature
Framework work, and risks duplicating content that properly belongs
on the `FeatureDefinition` page itself.

---

### Recommendation

**Recommended disposition:** A

The reporter's underlying concern — "I don't know what to put in
`context`" — is a real discoverability problem even though the
*answer* (use `FeatureDefinition`) is correct as given. Disposition A
preserves the existing, correct delegation model and adds a single
narrative pointer on the operation page so the next implementer who
hits the same question finds the answer in the IG instead of in
Jira. Disposition B is attractive but is more change than this
specific ticket warrants and is better tracked as a separate
enhancement once `FeatureDefinition` itself is more stable.
Disposition C is technically defensible — the question *was*
answered — but leaves the documentation gap that produced the
question in the first place, and similar questions will recur. The
small narrative addition in A is the lowest-risk way to retire the
issue durably.
