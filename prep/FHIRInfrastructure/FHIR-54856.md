## Summary

The ticket requests that the `Subscription` resource populate `meaningWhenMissing` on four optional coded elements — `Subscription.filterBy.comparator`, `Subscription.filterBy.modifier`, `Subscription.contentType`, and `Subscription.content` — so that server and client behavior in the absence of a supplied value is consistent and unambiguous across implementations. Today these elements are optional and coded, but the resource definition does not state what a sender/receiver should assume when the element is omitted, leading to potential divergence between conformant implementations. No cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced by Augury for this ticket; the only cross-references recorded are mentions of the four Subscription element paths themselves.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a `meaningWhenMissing` string to the ElementDefinition for each of the four elements in the Subscription resource, stating the default behavior a server/client MUST assume when the element is absent. Concretely:

- **`Subscription.filterBy.comparator`** — "If no comparator is supplied, the filter is evaluated using the default comparator semantics of the referenced SearchParameter (equivalent to `eq` for most types)."
- **`Subscription.filterBy.modifier`** — "If no modifier is supplied, the filter is evaluated without any modifier; the match is a direct (non-modified) comparison of the SearchParameter value against `Subscription.filterBy.value`."
- **`Subscription.contentType`** — "If no contentType is supplied, the server selects the payload MIME type using normal content-negotiation rules (per the channel definition); receivers SHALL accept either `application/fhir+json` or `application/fhir+xml`."
- **`Subscription.content`** — "If no content level is supplied, the server uses `id-only` as the default notification payload level."

Apply these changes in the `Subscription` resource authoring source for the R6/current build of FHIR Core (i.e., the HL7 `fhir` build source for the Subscription resource StructureDefinition and its narrative invariant/description pages). Regenerate downstream artifacts (search parameters, examples, bindings documentation) as part of the normal publishing pipeline.

#### Justification

`meaningWhenMissing` is the mechanism FHIR Core uses for exactly this situation: optional elements whose absence has a defined semantic interpretation. Filling it in is low-risk, non-breaking (no structural change), and directly resolves the ambiguity the reporter flags. The chosen defaults align with how mature Subscription (R5) implementations already behave in practice — `id-only` is the most widely deployed payload level, `eq` is the idiomatic default for SearchParameter comparators, and content-negotiation is already the universal fallback for FHIR payload MIME types. Documenting them explicitly improves interop without forcing any server to change code.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) setting `meaningWhenMissing`, promote the default semantics into the Subscription resource's **narrative / notes section** and into the Subscriptions R5 module's **"Notifications" and "Filtering"** pages, and add an invariant (or explicit conformance statement) that a server SHALL behave as if `Subscription.content = "id-only"` and `Subscription.contentType = application/fhir+json` (or the channel's negotiated type) when those elements are omitted. Treat `filterBy.comparator`/`filterBy.modifier` as genuinely absent (i.e., "no comparator/modifier applied") rather than defaulting them to a specific code, and document that distinction in the filter evaluation section.

#### Justification

`meaningWhenMissing` is useful but is easy for implementers to overlook because it lives on the ElementDefinition rather than in the human-readable resource guidance pages. For `filterBy.comparator` and `filterBy.modifier`, picking a single default code (e.g., `eq`) could obscure a meaningful distinction: "no modifier" is semantically different from "modifier = (something)" in search semantics, and baking `eq` into `meaningWhenMissing` could inadvertently require servers to treat explicit `eq` and omitted comparator identically in audit/logging contexts where they are currently distinguishable. Surfacing the defaults in narrative plus a conformance statement captures the intent without over-constraining the element-level semantics.

---

### Disposition C: Decline

#### Proposal

Do not change the Subscription resource. Close the ticket as "Persuasive with Modifier — Not Persuasive" on the grounds that the default behavior is either (a) already implicitly defined by the referenced binding (e.g., the SearchComparator/SearchModifierCode value sets and their parent SearchParameter semantics), or (b) intentionally left to the channel definition and/or implementation, and should not be globally fixed in Subscription.

#### Justification

Declining is defensible only if the WG concludes that the four elements' default behavior is genuinely implementation-defined (e.g., `contentType` is naturally controlled by channel-level content negotiation and fixing a default in `meaningWhenMissing` could conflict with channel semantics). It is the weakest option: the reporter's concern — inconsistent behavior across implementations — is real, the cost of documenting defaults is low, and FHIR Core already uses `meaningWhenMissing` for many analogous optional elements across other resources. Absent a concrete conflict with channel-level content negotiation rules, declining would leave the interop gap in place.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the smallest, most direct fix that is fully consistent with how FHIR Core documents default behavior elsewhere, and it uses the mechanism (`meaningWhenMissing`) that was designed precisely for this case. It is non-breaking, requires only authoring-side edits to the Subscription resource definition, and produces machine-readable guidance that tooling (validators, IGs, implementation libraries) can surface automatically. Disposition B's narrative/invariant approach has merit for `Subscription.contentType` specifically (where channel-level negotiation is a real concern) and the WG may wish to **combine A with a narrative note for `contentType`** — i.e., accept A but word `contentType`'s `meaningWhenMissing` to defer to channel-level content negotiation rather than pinning a single MIME type. Disposition C should only be adopted if the WG identifies a concrete conflict with channel semantics that cannot be resolved by careful wording.
