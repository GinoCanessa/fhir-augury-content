## Summary

The reporter, James Jahns, flags a broken/incorrect cross-reference on the
FHIR Core "Extensibility" page (R6 ballot 3 publication). In the section that
explains the **Data Absent Reason** extension, the inline link labelled
"data missing code" points at `https://hl7.org/fhir/6.0.0-ballot3/general-extensions.html`,
which now resolves to an empty "general extensions" page — the consequence
of the general/common extensions having been moved out of FHIR Core into
the FHIR Extensions Pack. The reporter proposes that the link should
instead target the canonical Extensions Pack StructureDefinition page for
data-absent-reason: `https://hl7.org/fhir/extensions/StructureDefinition-data-absent-reason.html`.
There are no cross-referenced Jira tickets, Zulip threads, GitHub items,
or Confluence pages tied to this issue in the FHIR Augury index, so the
review hinges entirely on the editorial fix proposed by the reporter.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the "Data Absent Reason" subsection on the FHIR Core
**Extensibility** page so that the inline anchor currently labelled
"data missing code" — which today points at
`https://hl7.org/fhir/6.0.0-ballot3/general-extensions.html` — instead
points at the canonical Extensions Pack page:
`https://hl7.org/fhir/extensions/StructureDefinition-data-absent-reason.html`.
The change is a single-anchor edit on the Extensibility page in the
`HL7/fhir` repository (FHIR Core). The link should be authored as a
relative or canonical reference into the Extensions Pack publication
(consistent with other Core → Extensions Pack cross-links) so it
re-validates cleanly under future publishes. No other prose, examples,
or StructureDefinition content needs to change.

#### Justification

This is exactly the technical correction the ticket reports: a stale
cross-reference into a section of the spec that was emptied when the
general/common extensions were migrated to the FHIR Extensions Pack.
Replacing the URL with the Extensions Pack StructureDefinition page
restores the reader's path from the Extensibility narrative to the
authoritative definition of the very extension being discussed, which
is the obvious editorial intent. It is the lowest-risk fix, scoped to a
single anchor on a single page, and aligns with the broader convention
already in use elsewhere in the spec where Core narrative defers to
Extensions Pack pages for extensions that no longer ship in Core.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than fix only the one anchor, perform a small editorial sweep of
the Extensibility page (and any other Core specification pages that
discuss removed/migrated extensions) to (a) repoint every link that
still targets `…/general-extensions.html` or other now-empty Core
extension pages to the corresponding StructureDefinition page in the
Extensions Pack, and (b) add (or confirm) a short banner / inline note
in the Extensibility page explaining that common extensions formerly
documented in Core now live in the Extensions Pack, with a single
top-level link to the Extensions Pack landing page. The Data Absent
Reason link is fixed as part of this sweep.

#### Justification

The reporter found one broken link, but the underlying cause — the
move of general/common extensions out of Core — almost certainly
produced sibling broken links on the same page and on other Core
pages that historically referred to `general-extensions.html`. Fixing
them as a set is cheaper than re-litigating each instance as a separate
ticket, and giving the page a single, durable orientation note about
"extensions now live in the Extensions Pack" reduces the rate at which
similar tickets are filed in the future. The trade-off versus
Disposition A is scope: this requires a grep across the Core authoring
sources for stale `general-extensions.html` references and a small
amount of new prose, which carries a marginally higher review cost
but eliminates a class of tickets rather than one instance.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as already-addressed-by-pending-work or out of
scope, on the grounds that the broken-link condition is a transient
artefact of the in-flight R6 ballot 3 publication mechanics (the
general-extensions page being intentionally emptied while extensions
are migrated) and will be corrected as part of normal pre-publication
link checking before R6 final, without requiring a targeted edit driven
by this ticket.

#### Justification

This is defensible only if the workgroup believes (a) the publisher's
existing link-checking pass will catch and repoint this link before
final publication of R6, and (b) opening a one-line correction ticket
against the page is unnecessary churn for an issue already on the
publication QA list. In practice this rationale is weak: the reporter
verified the broken state on the published `6.0.0-ballot3` site, the
ticket has been Triaged but unresolved for over a week, and there is
no evidence in the FHIR Augury cross-reference index that any other
work item, PR, or Zulip thread is tracking the fix. Declining therefore
risks the link shipping broken into the next ballot snapshot.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the literal fix the reporter asked for and is clearly
correct, but the broken link is symptomatic of a structural change —
the migration of general/common extensions out of Core — that very
likely left more than one stale cross-reference on the Extensibility
page and on related Core pages. Disposition B costs only marginally
more (a grep over the Core authoring sources for `general-extensions.html`
references plus a small orienting note) and resolves a class of issues
in one editorial pass instead of forcing the workgroup to re-triage
each successive report from implementers reading the same page.
Disposition C should be rejected: there is no evidence the publisher
QA pass will silently fix this, and the ticket has already sat
Triaged-Unresolved without movement, which is precisely the failure
mode that produces broken links in published ballots.
