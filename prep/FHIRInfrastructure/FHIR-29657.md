## Summary

Mark Kramer is asking the FHIR Shorthand (FSH) documentation to be made
explicit on a rule that authors frequently get wrong: every top-level
extension (i.e., an extension defined as its own artifact) must be
authored using the **Standalone Extension** form. Inline extension
syntax should only be used for the children of a complex extension
(i.e., the nested extension elements *inside* an already-defined
parent extension). The change is purely a documentation
clarification — no language or compiler change is requested. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket, so the request stands on its own.

## Details

**Description:**

Documentation should clarify that all top-level extensions must be
defined as Standalone. Only complex extensions can use inline
extensions.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Shorthand specification's *Defining Extensions* section
(and the corresponding tutorial / cheatsheet entries) to add an
explicit rule: **all top-level extensions MUST be defined using the
Standalone Extension form** (i.e., introduced with the `Extension:`
keyword as their own artifact). State that the **inline extension**
syntax (`* extension contains foo 0..1 ...`) is permitted **only as
sub-extensions of a complex extension** — never as a way to introduce
a brand-new top-level extension. Add a short worked example showing
the wrong vs. right pattern (a top-level inline-only extension on a
profile, contrasted with the same extension authored as a Standalone
Extension and then bound onto the profile via `* extension[foo]`).
Cross-link the new note from the *Profiles → Adding Extensions*
section so readers hitting the topic from either direction land on
the same rule.

#### Justification

This is consistent with how SUSHI actually compiles FSH today and with
the FHIR core guidance that every extension is identified by a
canonical URL, which requires a standalone `StructureDefinition`.
Documenting the existing constraint costs nothing in tooling, removes
a recurring source of authoring confusion (writers attempting to
"declare" a new extension via inline syntax on a profile), and
matches the reporter's stated concern. It is the minimum-impact
disposition: a documentation-only change in the Shorthand spec under
the FHIR Infrastructure work group's stewardship.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to the documentation clarification in Disposition A,
strengthen the guidance into a **normative authoring rule** in the
Shorthand spec, and have **SUSHI emit a warning** (or an opt-in error)
when it encounters inline-extension syntax that would have the effect
of introducing a new top-level extension URL rather than refining a
sub-extension of a complex extension. Phrase the spec text as a SHALL
("A top-level extension SHALL be defined as a Standalone Extension;
the inline extension form SHALL be used only for sub-extensions of a
complex extension."), and reference the new diagnostic by ID so users
hitting the warning can find the rule that produced it.

#### Justification

Clarifying the docs (Disposition A) helps readers who actually consult
the spec, but the misuse pattern described in the ticket typically
shows up at authoring time, not at reading time. A SUSHI diagnostic
catches the mistake at the point it is made and points the user back
to the spec text, which is a more durable fix than prose alone. The
trade-off is that this requires a coordinated change between the
Shorthand spec and the SUSHI implementation (separate codebase,
separate release cadence) and may flag pre-existing FSH that, while
discouraged, currently compiles. That coordination cost is the main
reason to consider it as an alternative rather than as the primary
disposition.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Modification → Not Persuasive (Already
Documented)** on the grounds that the FHIR Shorthand specification's
*Defining Extensions* and *Extensions on Profiles* sections, together
with the existing tutorial examples, already model the Standalone
Extension as the canonical way to introduce a new extension and only
use the inline form within complex extensions. No spec change is
required; readers who follow the tutorial in order will arrive at the
correct pattern.

#### Justification

This is defensible only if a reviewer reads the existing FSH spec and
concludes that the rule is already conveyed by example, even if not
stated as an explicit constraint. The risk is that the reporter — an
experienced FHIR author — explicitly identified the documentation as
unclear, which is evidence that "implied by example" is not enough.
Decline should therefore be reserved for the case where the
work group judges that no FSH author of reasonable diligence would
actually be confused.

---

### Recommendation

**Recommended disposition:** A

The request is a small, well-scoped documentation clarification of a
rule that is already true in practice (every top-level extension
needs its own canonical URL, hence a Standalone Extension), reported
by a credible FHIR author who hit the ambiguity. Disposition A
delivers exactly that with no tooling coordination cost and no risk
of breaking existing FSH. Disposition B is a reasonable follow-on if
the FSH/SUSHI maintainers want to harden the rule with a diagnostic,
but it should not block the doc fix and can be tracked as a separate
SUSHI ticket. Disposition C is hard to justify given the reporter's
explicit statement that the current text is unclear.
