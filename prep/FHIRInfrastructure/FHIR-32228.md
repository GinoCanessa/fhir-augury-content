## Summary

The ticket asks the SMART on FHIR specification to define how access to FHIR
operations is authorized via SMART scopes. Today the SMART v2 scope grammar
covers CRUD-and-search interactions on resources (`c`, `r`, `u`, `d`, `s`)
but is silent on operation invocation, leaving implementers without a
standard way to grant permission to invoke `$everything`, `$export`,
`$validate`, custom operations, etc. The reporter proposes extending the
permission letters with an `o` (operation), and optionally allowing a
suffix to scope down to a specific operation name — e.g.,
`user/Patient.crudso` for all Patient interactions plus operations,
`user/Patient.crudso.everything` for `$everything` only,
`user/Resource.o` for all system-level operations, and
`user/Resource.o.export` for `$export`.

There are no Jira cross-references on this ticket; the only cross-reference
hits are mentions of FHIR resource paths (`Patient.crudso`,
`Patient.crudso.everything`) that appear inside the ticket's own
description. The ticket was once resolved "Considered for Future Use"
(vote 8-0-0, 2021-08-02) on the basis that connectathons had not been able
to drive testing or consensus on operation-based scopes, and several
alternative syntaxes (e.g., `user/$export.type(Patient)`) had been
discussed but not converged. Lynn Laakso reverted that resolution on
2021-11-29 with the note "re-open post publication of STU2", so the
question is being re-opened for the post-STU2 cycle. Community discussion
on `chat.fhir.org` (see Related Zulip Discussions below) shows that the
gap continues to cause real implementation confusion.

## Details

**Description:**

> We should add support for operation scopes. At least the specification
> should state what is required to access operations.
>
> Options to for adding operation support could be: add an 'o' for
> operation permission. E.g. as indicated below.
>
> - `user/Patient.crudso` → permission to use all Patient based instance
>   and resource permissions.
> - `user/Patient.crudso.everything` → permission to use the `$everything`
>   method on resource and instances.
> - `user/Resource.o` → permission to use all system operations.
> - `user/Resource.o.export` → permission to use the `$export` operation.

## Related Zulip Discussions

### [smart > SMART scopes for operations](https://chat.fhir.org/#narrow/stream/smart/topic/SMART%20scopes%20for%20operations)

Long-running thread (2022-09 → 2023-06) where implementers ask how to
authorize specific operations under SMART. Alexander Zautke asked
whether the operation could be identified by its canonical URL (so
servers don't have to maintain a hard-coded allow-list of operation
names). Igor Bossenko proposed scopes like
`user/MpiPatientDisability.s` and later preferred a dedicated `e`
(execute) suffix, e.g. `user/MpiPatientDisability.e`. Josh Mandel (a
SMART editor) responded that there is **no standardized way** to
describe operation-invocation scopes today, because the right
authorization decision often depends on the operation's *parameters*
rather than just its name, and suggested that operation logic should
make decisions based on the surrounding data scopes the caller already
holds. No consensus on a syntax was reached.

Participants: Alexander Zautke, Igor Bossenko, Josh Mandel.

### [smart > $everything operation and referenced patients](https://chat.fhir.org/#narrow/stream/smart/topic/%24everything%20operation%20and%20referenced%20patients)

Brief thread in which Josh Mandel acknowledges, while discussing
`Patient/$everything`, that "we don't even have operation-specific
scopes in smart today" — i.e., the gap this ticket targets is still
open as of 2023-04 and has direct user-visible consequences for
operation behavior.

Participants: Josh Mandel (and the wider `smart` stream).

### [smart > SMART scopes for delete-history?](https://chat.fhir.org/#narrow/stream/smart/topic/SMART%20scopes%20for%20delete-history%3F)

Discussion (2024-02) about authorizing `$delete-history`. Josh Mandel
notes that this question "relates to questions about generic operation
invocation" and argues that for things like pre-binding operation
parameters, SMART's deliberately-limited scope syntax is unlikely to be
sufficient — implementers will likely need a different tool such as the
IETF **OAuth 2.0 Rich Authorization Requests (RAR)** model. This is
the most explicit on-record signal from a SMART editor that *granular
scope syntax alone is the wrong layer* for full operation-permission
expressivity.

Participants: Josh Mandel.

### [smart > Using SMART scopes for authorizing Operations](https://chat.fhir.org/#narrow/stream/smart/topic/Using%20SMART%20scopes%20for%20authorizing%20Operations) and [implementers > ✔ SMART on FHIR Scopes for Operations](https://chat.fhir.org/#narrow/stream/implementers/topic/%E2%9C%94%20SMART%20on%20FHIR%20Scopes%20for%20Operations)

Same question, cross-posted by Chris Watts (2024-04). Using a custom
`POST [base]/Patient/$update-name` operation as the example, he asks
what scope should authorize it. The reply traffic on these threads
points implementers at the PDex Payer-to-Payer Bulk Exchange IG, which
defines a per-operation scope convention for `$MemberMatch`
([build.fhir.org](https://build.fhir.org/ig/HL7/davinci-epdx/payertopayerbulkexchange.html#scopes-for-operations)).
Alexander Zautke flags an ecosystem problem with that approach: it
encodes operation parameters after a `?`, and authorization servers
that assume "everything after `?` is a search parameter" reject the
scope. There is clear demand for a single, agreed mechanism.

Participants: Chris Watts, Alexander Zautke (and others on the
`smart`/`implementers` streams).

### [smart > SMART scopes v2](https://chat.fhir.org/#narrow/stream/smart/topic/SMART%20scopes%20v2)

Older (2019) historical thread from the SMART v2 design phase. Jim
Steel, Jenni Syed, Grahame Grieve, and Isaac Vetter discuss whether and
how an `OperationOutcome` should signal that an operation could not be
performed for authorization reasons. Provides background on why SMART
v2 ultimately shipped without operation-scope syntax.

Participants: Jim Steel, Jenni Syed, Grahame Grieve, Isaac Vetter
(among others).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's `o` ("operation") permission letter into the SMART
v2 scope grammar in the **SMART App Launch** specification
(`scopes-and-launch-context` page, "Scopes for requesting clinical
data" / permission-letter section). Concretely:

- Extend the permission-letter alphabet from `cruds` to `crudso`, where
  `o` grants permission to invoke FHIR operations on the resource type
  (or system, when used with `Resource`).
- Allow an optional dotted suffix that names a specific operation, with
  the `$` stripped — e.g., `user/Patient.o.everything` to grant only
  `Patient/$everything`, and `user/Resource.o.export` to grant only
  system-level `$export`.
- Define interaction with the existing search-parameter scope suffix
  syntax (`?param=value`): operation-name suffixes use `.name`, search
  constraints continue to use `?…`, and the two MAY be combined (e.g.,
  `user/Patient.o.everything?_since=...`) where the operation defines
  the parameter.
- Update the `scope` ABNF, examples, and the PDex Payer-to-Payer Bulk
  Exchange IG's `$MemberMatch` example to use the new syntax.

#### Justification

This is the syntactically-cheapest fix that closes the gap implementers
are repeatedly hitting (see the 2023-06 and 2024-04 Zulip threads). It
re-uses the existing dotted-suffix mechanism that SMART v2 already uses
for sub-resource constraints, so authorization servers and clients can
extend their existing parsers in a small, additive way. It directly
addresses Alexander Zautke's complaint that PDex's `?operation=...`
pattern collides with the search-parameter suffix grammar by giving
operation invocation its own first-class slot in the scope.

### Disposition B: Alternative Approach

#### Proposal

Rather than extending the permission-letter alphabet, define a new
**operation-centric scope production** in the SMART scope grammar:

```
scope = role "/" (resourceType | "Resource") "$" operation-name [ "?" params ]
```

so the canonical form becomes `user/Patient.$everything`,
`user/Resource.$export`, `system/Observation.$validate`. Pair this with
an explicit statement in the SMART spec that operation invocation is
**not** implied by `c|r|u|d|s` permission letters; servers MUST require
an explicit operation scope (or an `*` wildcard at an appropriate
level) to authorize `POST … /$op`. For operations whose authorization
depends on parameters, normatively defer to **OAuth 2.0 Rich
Authorization Requests (RFC 9396 / `authorization_details`)** as the
mechanism for parameter-bound permissions, and provide a non-normative
example showing how a SMART scope and an `authorization_details`
object combine.

#### Justification

This matches the direction Josh Mandel has consistently steered the
community toward in the SMART scopes for operations / delete-history
threads: granular scope syntax is deliberately limited, and operation
authorization is fundamentally about *parameters*, not just operation
names. Reusing the `$op` notation that FHIR already uses for operation
invocation makes scopes self-documenting, avoids overloading the
permission-letter slot with semantics that don't fit the CRUDS model,
and avoids painting SMART into a corner the next time an operation
needs richer parameter binding (a problem `crudso.everything` cannot
solve). The trade-off vs. Disposition A is more conceptual surface area
in the spec (a new production plus a normative pointer to RAR), but
much better long-term extensibility and a cleaner story for
implementers building authorization servers today.

### Disposition C: Decline

#### Proposal

Re-resolve the ticket as **Considered for Future Use** (restoring the
2021-08-02 disposition that Lynn reverted), with an updated rationale:
SMART scope syntax is intentionally limited and the community has not
converged on a single mechanism in the four years since the original
request. Any operation-permission grammar baked into SMART now risks
being superseded by IETF Rich Authorization Requests
(RFC 9396), which is already the direction SMART editors are
recommending for parameter-bound operation permissions. Add a
non-normative implementation note to SMART App Launch pointing
implementers at:

1. The **PDex Payer-to-Payer Bulk Exchange** IG's per-operation scope
   convention as a reference example, and
2. **OAuth 2.0 Rich Authorization Requests** for cases where parameter
   binding is required.

#### Justification

There is real implementation pain (multiple 2023–2024 Zulip threads),
but four years of discussion have not produced consensus on a syntax,
and the most experienced SMART voice on the threads has explicitly
argued that the right tool may live outside SMART scope syntax
altogether. Locking in `crudso` or `$op` now risks fragmenting the
ecosystem if RAR-style mechanisms become the de-facto norm. A documented
"don't standardize yet, here's what to do in the meantime" answer is
defensible and matches HL7's typical posture when a problem is
acknowledged but the design is unsettled.

---

### Recommendation

**Recommended disposition:** B

The ecosystem evidence is clear that the gap is real and is actively
biting implementers (`$everything`, `$export`, `$MemberMatch`,
`$update-name`, `$delete-history` all show up in the last two years of
Zulip traffic), so Disposition C — re-deferring — is hard to defend a
*second* time. Between A and B, Disposition B better matches the actual
shape of the problem: operation authorization is mostly about *which
operation* (and sometimes *with which parameters*), and a `$op`-shaped
production in the scope grammar (a) reuses notation FHIR already
defines, (b) avoids overloading the CRUDS letter alphabet with
semantics it was not designed for, (c) handles the existing `?param=…`
suffix collision Alexander Zautke flagged for PDex, and (d) gives the
spec a clean place to point at OAuth Rich Authorization Requests for
the parameter-binding cases that granular scopes will never cleanly
express. Disposition A would land faster, but it inherits the
"`.export` vs. search-parameter suffix" ambiguity that the 2021-08-02
discussion already called out and would need to be patched again the
first time someone needs parameter-bound operation permissions.
