## Summary

Jens Villadsen reports that the FHIR Core `graphql.html` specification page
contains an outdated description of GraphQL. Specifically, line 32 of
`source/graphql.html` (as of commit `4810bd7c`) characterises GraphQL in a
way that the reporter believes is no longer accurate: GraphQL's intellectual
property and stewardship are now held by the **GraphQL Foundation**
(<https://graphql.org/brand/>), and GraphQL is published as a formal
specification at <https://spec.graphql.org/> rather than only an industry
"standard". The reporter notes uncertainty about the precise distinction
between "standard" and "specification" but asks that the FHIR text be
brought into line with current GraphQL Foundation branding and the existence
of the published specification. No related Jira tickets or GitHub PRs were
surfaced by cross-reference; the only linked discussion is the originating
Zulip thread that triggered the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update `source/graphql.html` in `HL7/fhir` (authoring root `source/`,
recipe: *Update narrative / prose*) so that the introductory paragraph
around line 32:

1. Replaces any wording that implies GraphQL is owned/copyrighted by
   Facebook (or that treats GraphQL purely as an informal "standard") with
   wording that attributes stewardship and brand/copyright to the **GraphQL
   Foundation**, citing <https://graphql.org/brand/>.
2. Refers to GraphQL as a published **specification** and links the
   canonical specification at <https://spec.graphql.org/> (and, where
   useful, a versioned URL such as the current `October2021` draft listed
   on `spec.graphql.org`).
3. Retains the existing FHIR-side framing (that the FHIR GraphQL binding
   is HL7's mapping of FHIR onto GraphQL), changing only the descriptive
   characterisation of GraphQL itself.

No anchors (`<a name="…">`) on the page are removed; only prose and link
URLs change, so no inbound-link sweep is required beyond a normal
build-warning check.

#### Justification

The reporter has identified a factual inaccuracy in a normative-adjacent
narrative section. The fix is small, mechanical, and confined to a single
HTML page under the established authoring root (`source/`), exactly the
shape the FhirCore briefing's *Update narrative / prose* recipe describes.
Citing the GraphQL Foundation brand page and `spec.graphql.org` matches the
sources the reporter supplied and is consistent with how the FHIR
specification cites other external standards (e.g., HTTP, JSON, XML
specifications) by canonical specification URL. There is no design or
interoperability impact and no implementer burden.

---

### Disposition B: Alternative Approach

#### Proposal

Take the same edit to `source/graphql.html` as in Disposition A, **and**
add a short footnote/aside in the same paragraph clarifying HL7's intended
usage of "specification" vs. "standard" in the context of external
dependencies — e.g. a sentence stating that FHIR treats `spec.graphql.org`
as the authoritative published specification of GraphQL and that any
references in this page to the "GraphQL standard" should be read as
synonymous. Optionally, add a non-normative "External References" list
near the bottom of `graphql.html` that consolidates the GraphQL Foundation
brand page, the specification, and the GraphQL.js reference implementation,
so that future drift in external GraphQL URLs only has to be fixed in one
place.

#### Justification

The reporter explicitly flagged ambiguity ("where a standard deviates from
specification I do not know"). Disposition A fixes the immediate factual
problem but does not address that ambiguity, which is likely to resurface
when the page is next edited. Consolidating external GraphQL references
into a single block also reduces the maintenance surface for future
technical corrections of the same shape. Trade-off vs. A: slightly larger
edit, slightly more reviewer burden, and the footnote is editorial commentary
that some reviewers may consider out of scope for a "Technical Correction"
ticket.

---

### Disposition C: Decline

#### Proposal

Close the ticket as *Not Persuasive* / *No Change*, on the grounds that the
existing description of GraphQL is non-normative narrative and the cited
inaccuracies (Foundation stewardship, specification status) do not affect
conformance, the GraphQL binding semantics, or any implementer behaviour.
Direct the reporter to raise a new, scoped ticket only if specific normative
text becomes incorrect.

#### Justification

The page is informational; FHIR conformance does not turn on whether the
copyright holder is named correctly. A "decline" is defensible if the WG
wants to keep technical-correction tickets focused on normative content.
However, this rationale is weak in this case because the cost of the fix is
trivial and the benefit (accuracy of an externally-facing reference) is
real; declining mostly signals process discipline rather than substantive
disagreement.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a clear factual error in a piece of
externally-facing narrative, supplied authoritative sources for the
correction, and the cited maintainer (Grahame Grieve) already agreed in the
Zulip thread that a fix should be filed. The change is a textbook *Update
narrative / prose* edit on a single `source/*.html` page in `HL7/fhir` with
no cross-repo impact, no anchor churn, and no risk to generated artefacts.
Disposition B's editorial expansion is reasonable but exceeds the scope the
reporter and Grahame agreed to and risks turning a 5-minute fix into a
review thread about HL7's terminology preferences. Disposition C is hard to
defend given the maintainer's prior agreement that the text should be
corrected. Accept and apply the minimal accuracy fix; if the
"specification vs. standard" terminology question is worth pursuing it can
be raised as a separate, scoped ticket.
