## Summary

The ticket asks FHIR Infrastructure to fix and tighten the FHIR Core
specification's description of how multiple values are combined in a
single search parameter (the "Multiples" row that appears on every
resource-search page, plus the central
`search.html#multivalue` definition it references). The reporter raises
two concrete defects in `search.html#multivalue`:

1. The text says OR-values **can** be separated by commas, but does not
   make commas the **required** separator. The wording should be
   normative (`shall`).
2. There is no description of how to escape a literal comma when one
   appears inside a value.

The ticket is the escalation of two earlier Imaging Integration tickets
that were closed "Resolved - No Change / Not Persuasive":
[FHIR-54778](https://jira.hl7.org/browse/FHIR-54778) ("Correct and
extend description of Multiples") and
[FHIR-54806](https://jira.hl7.org/browse/FHIR-54806) ("Improve all
Multiples' multipleAnds"). Both raised the same questions against the
ImagingStudy / ImagingSelection search pages and were sent upstream to
FHIR Infrastructure on the grounds that the issue lives in the shared
`search.html` text and the per-resource generators that emit the
Multiples rows, not in any one Imaging artifact. The reporter also notes
that a corollary fix is needed in the tooling that generates each
resource's search page so the Multiples row text is regenerated
consistently.

A side observation in the ticket — that the first bullet of the
Multiples row does not say how values are separated — is explicitly
acknowledged by the reporter as a non-issue (the parameter itself
repeats, joined by `&` per HTTP), so it does not need a spec change.

## Details

**Description:**

This ticket was raised as a result of tickets
<https://jira.hl7.org/browse/FHIR-54778> and
<https://jira.hl7.org/browse/FHIR-54806>. Here, it was noted that in the
applicable search sections (ImagingStudy - FHIR v6.0.0-ballot4 and
ImagingSelection - FHIR v6.0.0-ballot4) every Multiples row of a search
parameter contains the same issues:

- The second bullet specifies how the multiples are to be separated,
  while it is undefined in the first bullet. *[This is actually not an
  issue, as the parameter (including value!) may repeat, being separated
  the HTTP-way, namely with an ampersand.]*
- It is not clear how the separator (supposedly a comma) is to be
  escaped when needed in a value.

While looking to resolve this issue, a reference to
<https://build.fhir.org/search.html#multivalue> was made. However, that
text has issues:

- It says that OR-values **can** be separated by commas, but not that
  they **shall** be.
- It does not tell how escaping the (comma) separator is to be done when
  a comma is required in a value. *[I suppose that the &-escaping
  required for AND-searching is already described somewhere.]*

Please correct both issues.

Note that the original comments probably apply to all searching, and
hence the tools to generate each resource are also to be adapted
according to corrections required for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update `search.html#multivalue` (the canonical "Multiple Values" text in
the FHIR Core search page) on two specific points, then propagate the
edits everywhere the Multiples row is generated:

1. **Make the comma separator normative.** Change the OR-values
   sentence from "values **can** be separated by commas" to "values
   **SHALL** be separated by commas (`,`)" and align the surrounding
   bullet with the existing `&`-separated AND-values wording so the two
   forms read in parallel.
2. **Specify the escape mechanism for a literal comma in a value.**
   Add a short paragraph (and example) immediately after the OR rule
   stating that a literal comma inside a value SHALL be escaped as
   `\,` (consistent with the existing "Escaping Search Parameters"
   convention used for `|`, `$`, and `\` in token / reference values),
   that the escape SHALL be applied before percent-encoding, and that
   servers SHALL un-escape `\,` before splitting the value list.
3. **Regenerate the per-resource Multiples row.** Where the
   resource-search pages (e.g., `imagingstudy-search.html`,
   `imagingselection-search.html`, and every other
   `<resource>-search.html`) emit the Multiples row from a template,
   update that template / generator so the second bullet links to the
   updated `search.html#multivalue` text and surfaces the `\,`
   escape rule. The first-bullet wording is left as is — the reporter
   explicitly withdrew that part of the complaint.

The change is editorial-plus-normative-clarification on the search
page, with a corresponding regeneration of every resource search page;
no resource definitions, search-parameter definitions, or operation
signatures change.

#### Justification

The two defects are real and isolated to spec text. The OR-separator
behaviour is already required by every conformant FHIR server in
practice (the `,` syntax is what tooling and test suites assume), so
upgrading "can" to "SHALL" documents existing reality rather than
introducing a new requirement. The escape mechanism also already exists
elsewhere in `search.html` for `|`, `$`, and `\`; extending it to `,`
keeps the search grammar internally consistent and removes a genuine
ambiguity that two separate Imaging tickets surfaced. Closing the loop
by regenerating the per-resource Multiples rows addresses the
reporter's correctly-noted concern that fixing only the central page
leaves dozens of resource pages still echoing the old wording.

---

### Disposition B: Alternative Approach

#### Proposal

Address the underlying need by **rewriting `search.html#multivalue` as a
self-contained, normative grammar** rather than tightening individual
sentences, and **removing the duplicated Multiples bullets** from the
per-resource search pages in favour of a single link.

Concretely:

- Replace the prose in `search.html#multivalue` with a short ABNF (or
  ABNF-style) production for a search-parameter value list that names
  `,` as the OR delimiter, names `&` (URL-level) as the AND delimiter,
  and names `\` as the escape character with `\,`, `\\`, `\|`, `\$` as
  the recognised escapes, plus a normative statement on
  escape-vs-percent-encode ordering.
- Adjust the resource-search page generator so the Multiples row no
  longer restates the separator/escape rules inline; instead it links
  to the new grammar block and only describes the resource-specific
  multiplicity (e.g., "AND-able and OR-able" vs. "OR-able only").

#### Justification

The root cause flagged by FHIR-54778, FHIR-54806, and now FHIR-56291 is
that the rules are scattered: `search.html` describes them in prose,
each resource page restates them in a Multiples row, and neither place
fully nails down separator-or-escape semantics. A single normative
grammar fixes the defect once and prevents the next round of "the
generated page disagrees with the central page" tickets. The trade-off
versus Disposition A is scope: this is a larger editorial change to
`search.html` and to the resource-search-page generator, and it could
land in the same ballot cycle but needs more reviewer time. It does not
change wire behaviour for any conformant client or server.

---

### Disposition C: Decline

#### Proposal

Resolve as "Not Persuasive — already addressed" with the explanation
that `search.html` already permits `,` as the OR separator and that the
escape mechanism (`\` as the escape character) is already defined in the
"Escaping Search Parameters" portion of `search.html`, so any
ambiguity is in the reader, not the spec. Close FHIR-56291 with a
pointer to those existing sections.

#### Justification

This is the disposition Imaging Integration effectively chose for
FHIR-54778 and FHIR-54806. It is defensible if the workgroup believes
the existing text is already normative-by-implication and that
implementers are not actually confused (no Zulip traffic exists on the
question, and no implementer-side bug reports are linked to this
ticket). It avoids touching `search.html` mid-ballot and avoids a
generator-level change to every resource-search page.

---

### Recommendation

**Recommended disposition:** A

The reporter has already shown that Disposition C does not hold up:
FHIR-54778 and FHIR-54806 were closed with exactly that rationale and
the same defect was immediately re-raised here, which is strong
evidence that the current text is genuinely ambiguous to a careful
reader. Disposition B is the cleaner long-term answer but is a larger
edit than the ticket asks for, and the underlying complaint (one
sentence non-normative, one rule undocumented) can be fixed surgically.
Disposition A delivers exactly the two changes the reporter asked for,
matches the existing escape convention used elsewhere in `search.html`,
and explicitly handles the reporter's correct observation that the
per-resource Multiples row is generated and therefore needs the
generator updated alongside the central text. If reviewer appetite
exists for a broader rewrite, Disposition B is a reasonable upgrade
path the workgroup can fold in during discussion.
