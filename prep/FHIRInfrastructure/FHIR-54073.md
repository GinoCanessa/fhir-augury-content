## Summary

Brian Postlethwaite proposes replacing the current `%terminology.*` factory-style FHIRPath functions with a fluent set of methods that operate directly on the input collection of `code`/`Coding`/`CodeableConcept`, mirroring the style of the existing fluent `memberOf()`, `subsumes()`, and `subsumedBy()` functions (and inspired by CSIRO's Pathling extensions). The proposed surface includes `useTerminologyServer(...)` (an evaluation-context modifier analogous to `defineVariable`, replacing `%terminology.at()`), `display([language])`, `designation(use, [language])`, `property(code, [language])`, `translate(conceptmapUrl, reverse, equivalence, targetSystem, [sourceSystem])`, `validateCode(system, [date], [language], [abstract])`, and `expandValueSet()`. The ticket also asks the spec to record *why* the factory approach was chosen if it is retained, and to enumerate any `%terminology.lookup` use cases not covered by the simpler fluent set. The closely related ticket [FHIR-54069](https://jira.hl7.org/browse/FHIR-54069) (also from Brian Postlethwaite) calls out that the factory functions take a `params` object whose URL-encoding rules for complex types (`Coding`, `CodeableConcept`, `CodeSystem`) are undocumented, and Michael Lawley has commented there that complex parameters cannot be URL-encoded at all and a `Parameters` object would be needed — that constraint is directly relevant here when deciding which of the proposed fluent signatures need a `params`/`Parameters` overload.

## Details

**Description:**

The set of `%terminology` factory functions are not like the rest of the FHIRPath functions which are fluent in nature, permitting easier usage. The existing `memberOf()`, `subsumes()`, and `subsumedBy()` functions are already fluent.

I'd like to suggest replacing the existing set of factory-style functions with this new set of fluent functions (inspired if not directly copied from CSIRO's Pathling extensions).

*(Also consider if any of these would require the "params" parameter that was included in the factory functions for some of the uncommon parameters — as noted in [FHIR-54069](https://jira.hl7.org/browse/FHIR-54069).)*

**`useTerminologyServer(...)`**
Modifies the evaluation engine's current configuration with the value for anything chained after it. This would also work for the `subsumes` function. Similar to how `defineVariable` works. *(This would/could replace the `%terminology.at()` function.)*
*e.g. `Patient.gender.useTerminologyServer('<TSserverURL>').translate('http://example.org/ConceptMap/local-gender-codes')` — translate the patient gender into a local coding based on a specific concept map on a specific terminology service.*
This could be set at the start of the expression so it does not need to be restated anywhere else; not required to be immediately before the terminology function call.

**`display([language: String]): collection<string>`**
Get the display value for each input code/coding. *(One of the possible uses of `lookup` — without all the extra result processing needed.)*
*e.g. `item.where(linkId='a1').answer.value.display('fr')` — retrieve the French display text for the coded response.*

**`designation(use: coding, [language: string]): collection<string>`**
Lookup/retrieve the designation for the input coding(s) — could use TS `lookup`. *(One of the possible uses of `lookup` — without the extra result processing.)*

**`property(code: string, [language: string])`**
Lookup/retrieve the property for the input coding(s) — could use TS `lookup`; output type will be whatever the type of that property is defined.
*Note: the `weight()` function could have partially been implemented using this function.*
*e.g. `item.where(linkId='a1').answer.value.property('http://hl7.org/fhir/StructureDefinition/itemWeight')`*

**`translate(conceptmapUrl, reverse, equivalence, targetSystem, [sourceSystem]): collection<coding>`**
Works on input collections of type `code` (needs `sourceSystem`), `Coding`, or `CodeableConcept`. Source and target systems are canonical URLs that can also contain the version.
*e.g. `Patient.gender.translate('http://example.org/ConceptMap/local-gender-codes')` — translate the patient gender into a local coding based on a specific concept map.*

**`validateCode(system: url, [date: dateTime], [language: string], [abstract: boolean])`**
Canonical (optionally versioned); input collection `code`, `Coding`, `CodeableConcept`; returns `OperationOutcome` or boolean. *Could have variations for VS/CS, i.e. `validateCodeInCS` / `validateCodeInVS`.*

**`expandValueSet()`**
Simple function, same params as the factory proposal.

Indicate why the factory approach is preferable, and include this justification/reasoning in the specification. What other use cases does `%terminology.lookup` provide that the above simpler-to-use functions do not?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FHIR Core specification's FHIRPath terminology-extensions page (the page that today defines `%terminology.translate`, `%terminology.lookup`, `%terminology.validateVS`/`validateCS`, `%terminology.expand`, and `%terminology.subsumes`), replace the factory-style API with the fluent surface listed in the ticket:

- `useTerminologyServer(url)` — evaluation-context modifier, scoped to the rest of the expression, analogous to `defineVariable`. Deprecate and eventually remove `%terminology.at()`.
- `<code|Coding|CodeableConcept>.display([language])` — collection<string>.
- `<code|Coding|CodeableConcept>.designation(use, [language])` — collection<string>.
- `<code|Coding|CodeableConcept>.property(code, [language])` — typed by the property definition.
- `<code|Coding|CodeableConcept>.translate(conceptMapUrl, reverse, equivalence, targetSystem, [sourceSystem])` — collection<Coding>.
- `<code|Coding|CodeableConcept>.validateCode(system, [date], [language], [abstract])`, with variants `validateCodeInCS` / `validateCodeInVS` if VS/CS disambiguation is required.
- `expandValueSet(...)` — same parameters as the factory form.

For each fluent function, define semantics on `code` (requires `sourceSystem` where applicable), `Coding`, and `CodeableConcept`, document the result type, and note the corresponding terminology-server operation (`$lookup`, `$translate`, `$validate-code`, `$expand`). Where FHIR-54069's `params` problem applies (uncommon optional parameters that can't be expressed positionally), add a single optional `Parameters`-resource overload rather than a URL-encoded `params` string. Leave the existing fluent `memberOf`/`subsumes`/`subsumedBy` functions in place. Mark the old `%terminology.*` factory functions as deprecated for one cycle, then remove.

#### Justification

The fluent style is already the established pattern in FHIRPath generally and in the existing `memberOf`/`subsumes`/`subsumedBy` terminology functions specifically; aligning the rest of the terminology surface eliminates an arbitrary inconsistency and removes the need for the awkward positional `{}` skip-token noted in FHIR-54069. `useTerminologyServer` as a context modifier mirrors the already-accepted `defineVariable` mechanism, so implementers have a precedent for scope handling. Adopting an explicit `Parameters`-resource overload for complex inputs directly addresses Michael Lawley's observation in FHIR-54069 that `CodeableConcept` cannot be URL-encoded, which means the cleaner fluent signatures do not lose expressive power. The ticket also asks the spec to record *why* the factory style was chosen if it is kept — accepting the fluent proposal makes that question moot.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the fluent surface, but **as an additive, non-breaking layer alongside** the existing `%terminology.*` factory functions for at least one normative cycle, rather than replacing them. Concretely:

- Add `useTerminologyServer`, `display`, `designation`, `property`, fluent `translate`, fluent `validateCode`, and fluent `expandValueSet` to the FHIRPath terminology-extensions page as new fluent functions, defined on `code`/`Coding`/`CodeableConcept` inputs.
- Keep `%terminology.translate`, `%terminology.lookup`, `%terminology.validateVS`/`validateCS`, `%terminology.expand`, `%terminology.subsumes`, and `%terminology.at` defined and supported, but mark them "legacy" with an explicit forward-pointer to the fluent equivalent and a note that the fluent forms are preferred for new content.
- For the `params` problem (FHIR-54069), add a single optional final `Parameters`-resource argument to whichever fluent functions need it, rather than a URL-encoded string.
- Resolve the ticket's open question — "what does `%terminology.lookup` provide that the fluent `display`/`designation`/`property` do not?" — by either (a) adding any missing capability to a fluent `lookup()` function, or (b) documenting in the spec the residual cases that justify keeping the factory `%terminology.lookup`.

#### Justification

The factory functions are already published in trial-use form and almost certainly have implementer uptake (engines, validators, IGs that embed FHIRPath expressions). Removing them outright, even with a deprecation cycle, is a breaking change for those consumers and risks invalidating expressions in published IGs. Layering the fluent surface additively gets the ergonomic benefit Brian is asking for while letting existing content keep working; the deprecation can become a removal in a later cycle once telemetry / community feedback shows the factory forms are no longer in use. This approach also lets the workgroup answer the ticket's `%terminology.lookup` question empirically — if `display`/`designation`/`property` cover every real use, deprecate `lookup`; if not, the residual reason becomes documented spec rationale, satisfying the ticket's request for justification without forcing a hard either/or.

---

### Disposition C: Decline

#### Proposal

Decline the request to replace the `%terminology.*` factory functions. Close as "not persuaded" with the rationale that the factory-style functions were chosen deliberately to make the terminology-service boundary explicit at the call site (the `%terminology` prefix signals "this hits a terminology server"), that they accommodate the full parameter surface of `$lookup`/`$translate`/`$validate-code`/`$expand` via the `params` object, and that the fluent ergonomics issue is better solved by improving documentation (the actual ask of FHIR-54069) than by introducing a parallel function set. Direct the reporter to FHIR-54069 for the documentation gaps and to FHIR-53953 for terminology-server availability/error semantics.

#### Justification

Two parallel APIs for the same operations is a real cost: implementers must support both, validators must understand both, conformance statements get more complex, and downstream tools (FHIRPath analyzers, IG publishers) must keep both code paths alive. The factory functions are not actually misshapen — they make the side-effect (a terminology-server call) syntactically visible, which is arguably *more* honest than fluent chaining that hides it. The ticket itself acknowledges that the new surface might still need a `params`-style escape hatch for uncommon parameters, which means the fluent forms do not cleanly eliminate the very thing the reporter dislikes. Declining keeps the spec stable and concentrates effort on FHIR-54069's concrete documentation gaps.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the ergonomic win that motivates this ticket — fluent `display`/`designation`/`property`/`translate`/`validateCode`/`expandValueSet` plus a `useTerminologyServer` context modifier, all consistent with the already-fluent `memberOf`/`subsumes`/`subsumedBy` — without making a breaking change to the trial-use `%terminology.*` surface that implementers and IGs may already depend on. It also gives the workgroup a clean way to discharge the ticket's two open questions: the `params` problem (FHIR-54069 + Lawley's comment) is solved by an explicit optional `Parameters` overload rather than URL-encoding hacks, and the "what does `%terminology.lookup` uniquely provide?" question becomes a concrete spec exercise — either fold the residual capability into a fluent `lookup()`, or document the residual factory-only use case. Disposition A is the right *eventual* destination but should follow a deprecation cycle rather than ship as a single breaking change; Disposition C undersells the consistency complaint, which is legitimate and shared with at least the FHIR-54069 author.
