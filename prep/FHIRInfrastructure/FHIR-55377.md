## Summary

Brian Postlethwaite raises a documentation gap in the FHIR Mapping Language (FML)
specification. The spec currently states, in multiple places, that "if a different
source epoch is required, this can be achieved by adding or subtracting the
difference in ticks prior to calling this transform" (referring to the
`unixToDate` / `unixToDateTime` transform functions) — but it never shows what
that actually looks like in FML. The reporter argues this is too easy to read
without internalising, and asks the workgroup to inline a concrete worked
example showing the offset technique using a variable plus a FHIRPath
expression on the source side, e.g.:

```
src.epoch as e -> (e + 45) as et, tgt.epoch2 = unixToDate(et)
```

The ticket is purely a request to add an explanatory example to the FML page;
no schema, grammar, or transform-function semantics change is requested. There
are **no explicit Jira cross-references** on this ticket. It does, however, sit
in a recognisable cluster of FML-quality tickets driven largely by the same
reporter (FHIR-54484, FHIR-54479, FHIR-54477, FHIR-54919, FHIR-54940,
FHIR-54952, FHIR-54955, FHIR-46284) and is a direct downstream consequence of
the older ticket FHIR-40645 ("Define parameters of dateOp"), whose resolution
("Persuasive with Modification", PR
[HL7/fhir#3967](https://github.com/HL7/fhir/pull/3967)) is what introduced /
clarified the `unixToDate` family in the first place.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive. In `HL7/fhir` (clone `db79dcd`), edit `source/mapping-language.html`
to inline a worked example immediately adjacent to the existing
"Note that if a different source epoch is required…" sentence(s) attached to
the `unixToDate` / `unixToDateTime` rows of the transform-functions table.
Concretely:

1. Locate each occurrence of the "different source epoch" note on the FML page
   (the reporter notes "there are multiple references to do this").
2. After the first such occurrence, insert a short example block (HTML `<pre>`
   in the same style as other FML examples on the page), using the reporter's
   own snippet adapted to a realistic offset and named target:

   ```
   src.epoch as e -> (e + 2208988800) as et, tgt.epoch2 = unixToDate(et)
   ```

   with one or two lines of prose explaining that `2208988800` is the seconds
   between the NTP epoch (1900-01-01) and the Unix epoch (1970-01-01), and
   that the same pattern applies in reverse for any other source epoch.
3. At subsequent occurrences of the same note, replace the prose with a short
   back-reference (`see the example above`) so the example lives in exactly
   one place and the page does not drift out of sync with itself.
4. No grammar (`source/mapping.g4`) change, no StructureDefinition change, no
   resource change.

#### Justification

This is exactly the change the reporter asks for and matches the FhirCore
playbook's "Update narrative / prose" recipe. The repo briefing confirms FML
page edits are authored directly in `source/mapping-language.html` and the
clone is current, so there is no infrastructural risk. The example is
self-contained, uses an existing supported pattern (FHIRPath on the source
plus an intermediate variable), and addresses the documented complaint that
the "ticks offset" workaround is described but never shown. Cost is minimal
and the educational payoff for implementers is high — FHIR-54484 already
records "provide examples for all the transform functions" as a standing
request, so this change moves the FML page in the direction the workgroup has
already endorsed.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification. Rather than (or in addition to) documenting the
offset workaround, extend `unixToDate` / `unixToDateTime` to accept an
optional second parameter that names the source epoch — either as an
ISO-8601 date literal (`unixToDate(e, '1900-01-01')`) or as a named token
drawn from a small fixed set (`'unix' | 'ntp' | 'dotnet' | 'oadate'`,
defaulting to `'unix'`). Document the new signature in the transform-functions
table on `source/mapping-language.html`, update `source/mapping.g4` if the
parameter list needs grammatical accommodation, and provide both the new
direct call form and (for backwards compatibility) the existing
"add-then-call" form as a worked example.

This proposal would be raised as a follow-up Change Request rather than
silently expanded inside FHIR-55377; this disposition would resolve
FHIR-55377 itself by accepting the documentation example (Disposition A) and
opening a sibling ticket for the parameter extension.

#### Justification

The reporter's underlying complaint — "users keep being told to add/subtract
ticks but never shown how" — has a deeper root cause: the transform forces
every caller to reinvent the offset, which makes maps less readable and
error-prone (off-by-one on epoch boundaries, signed/unsigned arithmetic,
millisecond-vs-second precision). A first-class `epoch` parameter would
remove that whole class of mistake and align `unixToDate` with how comparable
date-conversion APIs are designed elsewhere. The trade-offs versus
Disposition A are: larger surface change, requires grammar/handler/test work
across implementations (HAPI, .NET, Matchbox), and a longer ballot path. The
briefing explicitly warns that out-of-scope FML concerns should not be
silently rolled in, so this option deliberately splits: take A now, file the
extension separately.

---

### Disposition C: Decline

#### Proposal

Not Persuasive. Close FHIR-55377 with the rationale that the existing prose
("…can be achieved by adding or subtracting the difference in ticks prior to
calling this transform") is sufficient guidance, and that the broader request
to "provide examples for all the transform functions" is already tracked under
FHIR-54484 — so this ticket is a duplicate of that umbrella and should be
folded into it without a standalone fix.

#### Justification

Defensible only on the narrow ground that FHIR-54484 explicitly asks for
examples across the whole transform-functions section, and adding examples
one-at-a-time per ticket creates editorial churn on `source/mapping-language.html`.
The case is weak overall, however: FHIR-54484 has been open since 2025 with no
PR yet, and declining FHIR-55377 leaves a known, easily-fixed documentation
gap unresolved for an indeterminate period. The briefing's gotcha about not
silently rolling out-of-scope concerns into adjacent tickets cuts in the
opposite direction here — declining and pointing at FHIR-54484 risks the fix
being lost rather than batched.

---

### Recommendation

**Recommended disposition:** A

The ticket asks for a single, well-scoped, low-risk documentation
improvement on a single page (`source/mapping-language.html`) in a single
repo (`HL7/fhir`). The repo briefing confirms the file is current, the page
is hand-authored HTML, and the edit fits the standard "Update narrative /
prose" recipe with no grammar, schema, or cross-repo implications. The
reporter has already supplied a working example, so the editorial burden is
minutes, not hours. Disposition B identifies a real follow-up (a first-class
`epoch` parameter on the transform), but treating it as a *replacement* for A
would block a trivial fix on a much larger design discussion; the right move
is to accept A now and, if the workgroup agrees the underlying API is
awkward, to spin off B as its own Change Request. Disposition C is hard to
justify when the cost of fixing is so low and the existing FHIR-54484
umbrella has not produced movement on this specific gap.
