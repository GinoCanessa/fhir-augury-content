## Summary

The ticket reports a documentation inconsistency in FHIR R6 ballot 3.
The UML diagram on the `RelativeTime` datatype page
(`datatypes.html#RelativeTime`) shows that `RelativeTime` inherits from
`BackboneType`, but `RelativeTime` is not listed as a subtype of
`BackboneType` on the Type Framework page. A follow-up comment from
JIRAUSER18552 also notes that `RelativeTime` is missing from the
datatype index at the top of the Datatypes page
(`datatypes.html#2.1.28.0`). The request is therefore a narrow technical
correction to bring two spec narrative pages into agreement with the
already-published `RelativeTime` datatype definition. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
found.

## Details

**Description:**

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply two narrative-only edits in the FHIR Core specification:

1. On the **Type Framework** page (`types.html`), in the section that
   enumerates the concrete types deriving from `BackboneType`, add
   `RelativeTime` to the list (alongside the other R6 backbone-typed
   datatypes such as `Availability`, `MonetaryComponent`,
   `ExtendedContactDetail`, etc.), with a link to the `RelativeTime`
   datatype page.
2. On the **Datatypes** page (`datatypes.html`), add `RelativeTime` to
   the index/table of contents at the top of the page (the listing
   referenced by anchor `#2.1.28.0`), in alphabetical position, linking
   to the existing `#RelativeTime` section.

No structural change to the `RelativeTime` `StructureDefinition`,
`BackboneType`, or any other resource is required — only the two
authored narrative pages need updating.

#### Justification

The UML diagram on the `RelativeTime` page and the underlying
`StructureDefinition` already establish that `RelativeTime` derives from
`BackboneType`. The Type Framework page and the Datatypes index are
hand-curated narrative views of those facts, and they are simply
out-of-sync. Bringing them into agreement is the textbook definition of
a Technical Correction: zero semantic change, zero compatibility risk,
and it removes a real source of confusion for readers comparing the
narrative pages to the datatype definitions. The fix is small, local,
and reversible.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than maintaining two hand-curated lists, generate the
"BackboneType subtypes" list on the Type Framework page and the
datatype index at the top of the Datatypes page from the published
`StructureDefinition` resources at build time (for example, via the
existing IG-publisher / FHIR-build macros that already enumerate
resources elsewhere in the spec). As part of that change, regenerate
both pages so `RelativeTime` (and any future `BackboneType`-derived
datatype) appears automatically.

#### Justification

This addresses the root cause — the lists are hand-edited and therefore
drift from reality every time a new datatype is introduced. A
generated index would prevent the same class of ticket from recurring
for `RelativeTime`'s siblings and successors. The cost is higher than
Disposition A (publisher-side tooling change rather than a narrative
edit), and it is wider in scope than the ticket strictly asks for, but
it converts a recurring maintenance defect into a build-time invariant.
Trade-off: this is no longer a pure Technical Correction; it would
likely need to be re-classified and scheduled accordingly.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the authoritative artifact —
the `RelativeTime` `StructureDefinition` and its UML diagram — already
correctly states that `RelativeTime` derives from `BackboneType`, and
readers should rely on the formal definition rather than the narrative
indices.

#### Justification

Declining is defensible only in the narrowest sense: no implementer
behavior depends on the Type Framework or Datatypes-index pages, so
there is no conformance impact. However, the spec narrative is the
primary entry point for human readers, and inconsistency between
diagrams and indices erodes confidence in the document. The request is
trivially small and carries no downside, so there is no good reason to
decline it.

---

### Recommendation

**Recommended disposition:** A

This is a clean Technical Correction: two narrative pages are
demonstrably out-of-sync with the published `RelativeTime` datatype and
its UML diagram, and a second reviewer has independently confirmed a
related omission on the Datatypes index. The fix is bounded, has zero
conformance impact, and resolves the reported confusion immediately.
Disposition B (auto-generating the lists) is genuinely attractive as a
follow-up and should be considered as a separate Infrastructure work
item, but it should not block this correction. Disposition C has no
upside given the cost of A is negligible.
