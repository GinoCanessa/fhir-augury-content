## Summary

Eric Haas asks SDC to clarify how multiple signatures should be expressed on a `QuestionnaireResponse`. The SDC Questionnaire Response Common Elements (Abstract) Profile currently allows the `questionnaireresponse-signature` extension at cardinality `0..*`, leaving it ambiguous (a) which subset of the QuestionnaireResponse each `Signature` covers when more than one is present, and (b) how complex signing patterns (co-signature, sequential approval, countersignature) are represented. The reporter offers three options: (1) deprecate the extension and rely on `Provenance`-based signatures (mirroring the proposal he is championing in FHIR-Infrastructure for `Bundle`); (2) reduce the extension's max cardinality to `1` and document that multi-signature use cases must use `Provenance`; or (3) keep `0..*` but require the JAdES `sigD` header parameter inside each `Signature` to enumerate exactly what is covered. No other Jira tickets were surfaced as cross-references, so this ticket stands alone, although it is conceptually paired with the FHIR-Infrastructure work to move `Bundle` signatures onto `Provenance`.

## Details

**Description:**

> The HL7 FHIR Structured Data Capture (SDC) Questionnaire Response Common Elements (Abstract) Profile uses a signature extension with a maximum cardinality of `0..*`.
>
> This leads to several potential challenges:
>
> - **Verification Challenges**: Relying solely on the extension data makes it difficult to verify digital signatures in multi-signature cases, because the FHIR `Signature` datatype captures basic details like signer, timestamp, and type but lacks mechanisms to explicitly denote complex relationships or contexts between multiple signatures, leading to ambiguity when there is more than one signature.
> - **Diverse Signing Scenarios**: Healthcare workflows involve various signing scenarios, such as co-signature, sequential approvals, and countersignatures, which require representing inter-signature relationships and dependencies.
>
> Options include:
>
> 1. Deprecate the extension and use `Provenance`-based signatures instead (we are proposing this in FHIR-I for `Bundle`s.)
> 2. Reduce the maximum cardinality to one, to eliminate the ambiguity on what is signed, and document the need for `Provenance`-based signature for multiple signature use cases.
> 3. Document the use of the JAdES `sigD` header parameter (inside the signature) that lists what is covered by the signature.

## Related Zulip Discussions

### [questionnaire > Questionnaire Signatures](https://chat.fhir.org/#narrow/stream/questionnaire/topic/Questionnaire%20Signatures)

The thread spans two waves of discussion. In Feb 2024, Stas Buldakov asked how to require multiple signatures (e.g., patient + practitioner) on a `QuestionnaireResponse` using the standard SDC extensions, whether servers should validate empty signatures, and how to control signature placement. Brian Postlethwaite reported that in practice he built a custom item-control that captures a finger/pen signature into an `Attachment` image, and questioned the value of free-hand signatures when the user is already authenticated. Lloyd McKenzie drew a clear distinction between two flavors: the *author's* signature on the form (handled by the SDC extension) and *third-party reviewer* sign-offs (which should go on `Provenance`); he warned that digital signatures on FHIR resources are fragile because intermediaries may strip extensions, inject codings, or transform versions, and that any robust signature requires a tight custom canonicalization that signs only the critical elements (e.g., `linkId` plus answer code/system).

In Aug 2025, Eric Haas reopened the topic with the same text that became this ticket and proposed scrapping the extension in favor of `Provenance` (parallel to the FHIR-I `Bundle` proposal). Lloyd McKenzie pushed back: the multi-signature ambiguity is "really just a matter of the canonicalization algorithm excluding the signatures," there is no use-case for signatures signing other signatures, the WG lacks industry feedback to justify a deprecation, and with SDC R4 about to publish his inclination is to **leave things unchanged**. Eric softened to suggesting a warning that the extension should not be used for more than one digital signature unless what is signed is precisely defined; Lloyd noted that defining what is signed is mandatory for any extension-based signature regardless.

**Participants:** Stas Buldakov, Brian Postlethwaite, Kari Heinonen, Lloyd McKenzie, Eric Haas.

## Proposed Dispositions

> Note: Because no `HL7/sdc` briefing is available, the file paths and authoring roots below are inferred from the cached clone layout (`input/fsh/`, `input/resources/`, `input/pagecontent/`) rather than from a validated briefing. The Recommendation calls for a briefing refresh before implementation.

### Disposition A: Accept As Requested (Option 2 — tighten cardinality + document Provenance for multi-signature)

#### Proposal

In the SDC IG (`HL7/sdc`, authoring root `input/`), modify the **Questionnaire Response Common Elements (Abstract) Profile** so that the `questionnaireresponse-signature` extension slice is constrained from `0..*` to `0..1`. Concretely, update the FSH that defines the abstract profile under `input/fsh/` so the relevant `* extension contains questionnaireresponse-signature ...` rule sets the slice cardinality to `0..1`. Update the narrative on the corresponding page under `input/pagecontent/` (the SDC behavior page that discusses signatures) to (a) state explicitly that the extension is intended to carry the *author's* signature over a defined canonicalization, and (b) direct implementers to use `Provenance` (with `Provenance.signature` and `Provenance.target` pointing at the `QuestionnaireResponse`) for any additional signatures — co-signature, countersignature, sequential approvals, etc. Add an example `Provenance` instance under `input/fsh/` (or `input/resources/`) demonstrating a co-signature pattern. Do **not** modify the `Signature` datatype itself or the standalone `questionnaireresponse-signature` StructureDefinition; the change lives entirely in the abstract profile and accompanying narrative.

#### Justification

This is the minimal, behavior-clarifying change that addresses the verification ambiguity Eric raised: with at most one in-line `Signature`, the canonicalization scope is unambiguous, and the broader healthcare-workflow patterns (co-signature, sequential approval, countersignature) are pushed onto `Provenance`, which already models actor, role, sequence, and target — the very relationships the ticket calls out as missing. It is consistent with Lloyd's analysis on Zulip that two flavors exist (author signature vs. reviewer/approver signature) and aligns the SDC story with the parallel FHIR-Infrastructure direction for `Bundle.signature`. It is implementable inside the SDC IG without touching the core spec or the `Signature` datatype.

---

### Disposition B: Alternative Approach — keep `0..*`, add normative guidance and a JAdES `sigD` requirement

#### Proposal

Leave the slice cardinality on `questionnaireresponse-signature` at `0..*` in the abstract profile, but in `input/pagecontent/` add a normative subsection ("Multiple signatures on QuestionnaireResponse") that:

1. Reaffirms a single, IG-defined canonicalization algorithm for the `QuestionnaireResponse` payload that **excludes** the signature extension itself (echoing Lloyd's "really just a matter of the canonicalization algorithm excluding the signatures" point).
2. **Requires** that when more than one `questionnaireresponse-signature` extension is present, each `Signature.data` MUST be a JAdES signature whose `sigD` header parameter enumerates the exact JSON pointers / element paths it covers.
3. Recommends `Provenance` for non-author signatures (countersignature, sequential approval) but does not forbid in-line multi-signature where JAdES `sigD` is used.
4. Adds at least one example: a `QuestionnaireResponse` with two `questionnaireresponse-signature` extensions where each `Signature.data` is JAdES with `sigD` populated.

No StructureDefinition or `Signature` datatype changes are needed; this is purely IG narrative plus example artifacts under `input/pagecontent/` and `input/fsh/` (or `input/resources/`).

#### Justification

This option (Eric's option 3) preserves backward compatibility for any existing implementer using multiple in-line signatures and avoids a `0..*` → `0..1` cardinality narrowing that would technically be a breaking change to the abstract profile. By tying multi-signature use to JAdES `sigD`, the ambiguity ("what does each signature cover?") is resolved by the signature payload itself, not by FHIR cardinality. The trade-off vs. Disposition A is that JAdES is a heavier conformance bar and adds implementation burden; Disposition A pushes complexity onto `Provenance`, which is already in implementers' toolboxes.

---

### Disposition C: Decline (defer until SDC R4 is published; document a warning only)

#### Proposal

Decline to make any structural change to the abstract profile or `questionnaireresponse-signature` extension at this time. Instead, add a short non-normative warning in the SDC behavior narrative under `input/pagecontent/` stating: "Implementations SHOULD NOT use more than one `questionnaireresponse-signature` extension on a single `QuestionnaireResponse` unless the canonicalization algorithm and the precise scope of each signature are explicitly defined and agreed by trading partners; for co-signature, sequential approval, or countersignature workflows, use `Provenance` with `Provenance.target` referencing the `QuestionnaireResponse`." Re-evaluate in the post-R4 cycle once industry feedback on actual multi-signature deployments has been collected.

Rationale category: **insufficient use cases / timing** — SDC R4 is on the cusp of publication and the WG lacks deployment evidence to justify a normative cardinality change or a deprecation.

#### Justification

This is exactly Lloyd McKenzie's stated position on the Zulip thread ("Given that we're about to publish release 4 of the spec in the next month, my inclination is to leave things unchanged"). Declining now avoids: (a) a breaking change to the abstract profile right before publication; (b) committing to either Provenance-only or JAdES-required without confirmation from the (small) community that actually uses the extension; and (c) divergence from the parallel FHIR-Infrastructure `Bundle.signature` work, which can act as a forcing function for a coordinated SDC change in the next cycle. Adding a warning still gives implementers immediate guidance against the ambiguous multi-signature pattern.

---

### Recommendation

**Recommended disposition:** **A**

Disposition A is the cleanest answer to the actual problem the ticket identifies: ambiguity over what each signature covers when more than one is present. Tightening to `0..1` removes the ambiguity by construction, and pushing multi-actor workflows (co-signature, sequential approval, countersignature) onto `Provenance` is consistent with both the existing FHIR pattern and the in-flight FHIR-Infrastructure direction for `Bundle.signature`, giving implementers a uniform story across resources. It is also the option that best honors Lloyd's two-flavor framing on Zulip without requiring SDC to take on JAdES as a normative dependency (Disposition B). Disposition C is defensible if the WG concludes that a cardinality change is inappropriate this close to R4 publication, but it leaves the verification problem unsolved and effectively kicks the question into the next cycle. If R4's publication timeline cannot accommodate a profile change, falling back to Disposition C with the warning text is acceptable as an interim — but the WG should commit to revisiting Disposition A in the next SDC cycle and should align timing with the FHIR-I `Bundle.signature` work so the two specs move together.
