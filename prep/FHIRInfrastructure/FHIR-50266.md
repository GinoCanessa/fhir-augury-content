## Summary

Alexander Henket reports a typo in the FHIR Core search documentation
describing the `_sort` parameter on reference and token search
parameters with the `:text` modifier. The example sentence reads
"`_sort=code:text` or `_sort:-code:text`" — the second example uses a
colon (`:`) where it should use an equals sign (`=`), since `_sort` is a
search parameter whose value is provided after `=`. The intended example
is `_sort=-code:text` (descending sort on the text/display portion of
`code`). The ticket was filed as a Technical Correction against R6
(R5 is not being updated). No other Jira tickets are cross-referenced;
the issue originates in a single Zulip thread on `#implementers`, where
Lloyd McKenzie confirmed the typo and asked Alexander to file the
correction.

## Details

**Description:**

> When sorting on search parameters of reference and token types, the
> search parameter MAY include the text modifier to request sorting
> based on the text or display portions of the element values. For
> example, `_sort=code:text` or **`_sort:-code:text`**.

The very last bit of this sentence should have been `_sort=-code:text`
so with "=" instead of ":".

Source page: <https://hl7.org/fhir/search.html#_sort>

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the technical correction to the `_sort` section of
`hl7.org/fhir/search.html`. Change the example sentence from:

> For example, `_sort=code:text` or `_sort:-code:text`.

to:

> For example, `_sort=code:text` or `_sort=-code:text`.

This is a single-character edit (replace the second `:` after `_sort`
with `=`) on the FHIR Core search documentation page. No schema,
StructureDefinition, search parameter, or normative behavior changes
— the prose example is brought in line with the actual `_sort=...`
syntax already documented elsewhere on the page.

#### Justification

The corrected form (`_sort=-code:text`) is consistent with the rest of
the `_sort` documentation, which uniformly uses `_sort=<value>` with a
leading `-` to indicate descending order, and with the `:text`
modifier examples for other search parameter types on the same page.
The colon form `_sort:-code:text` is not valid FHIR search syntax —
`:` after `_sort` would be parsed as a modifier on `_sort` itself,
which is not defined. Lloyd McKenzie explicitly endorsed treating
this as a technical correction in the originating Zulip thread, and
the reporter has already filed accordingly. Technical corrections of
demonstrable typos in normative spec prose are exactly the case the
"Technical Correction" type exists for.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the correction, but go further: replace the single-line example
with a small bulleted list that shows ascending and descending sorts on
the text modifier together, e.g.:

- `_sort=code:text` — ascending sort by the textual representation of
  `code`.
- `_sort=-code:text` — descending sort by the textual representation
  of `code`.

This both fixes the typo and removes the ambiguity that allowed it to
slip in (the original sentence runs two examples together with "or",
which made the malformed second example look superficially plausible).

#### Justification

The underlying defect is partly editorial: the two examples differ in
two ways at once (sign and — incorrectly — separator), so a reader
scanning quickly may not catch which character was intended to vary.
Splitting them out makes the contrast (ascending vs. descending)
explicit and reduces the chance a future edit reintroduces a similar
mistake. Trade-off vs. Disposition A: this expands the scope of a
technical correction beyond a strict character swap, which some
workgroup members may prefer to handle as a separate editorial task
rather than under "Technical Correction".

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" on the grounds that the surrounding paragraph
already defines the `_sort=<param>[:modifier]` syntax and the leading
`-` for descending sort, so an attentive reader will recognise the
malformed `_sort:-code:text` as a typo and read it as
`_sort=-code:text`.

#### Justification

The most defensible reason to decline would be that the change is so
trivial it is not worth a normative-prose edit cycle, and that the
intent is unambiguous from context. This is weak: the reporter is an
experienced implementer who nonetheless had to ask whether the example
was wrong, and Lloyd McKenzie agreed it should be corrected. There is
no design implication, no breaking change, and no implementer would be
harmed by the edit — declining mainly preserves the existing typo for
no benefit.

---

### Recommendation

**Recommended disposition:** A

The ticket is a textbook technical correction: a single-character typo
in a normative example, confirmed in chat by a co-chair, with the
correct form already in active use elsewhere on the same page. There
is no design controversy, no implementer impact, and no scope risk.
Disposition A applies the minimum change needed to fix the defect,
which is the right shape for a Technical Correction. Disposition B has
editorial merit but should be considered as a follow-up if the
workgroup wants to clean up the surrounding paragraph; bundling it
risks expanding a trivial fix into a small editorial debate.
Disposition C has no upside — the typo is real, the fix is one
character, and leaving it in place only continues to confuse readers.
