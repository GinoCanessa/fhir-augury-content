## Summary

The ticket reports that the FHIR Core Search page still documents
`Bundle.entry.mode = outcome` as if it were a valid value, even though
`outcome` has been deprecated as an allowed value for `Bundle.entry.mode`.
The reporter asks that the Search page either drop those references entirely
or, at a minimum, add a note flagging the deprecation so that implementers
do not infer that `outcome` is still a recommended way to convey search
operation outcomes inside a Bundle. The cross-reference index links the
ticket to the FHIR artifact `Bundle.entry.mode`, confirming the scope is the
Search page's narrative use of that element rather than the element
definition itself.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Core Search page so that it no longer presents
`Bundle.entry.mode = outcome` as a current option. Concretely:

- Remove the prose and any inline examples on the Search page that show or
  describe `Bundle.entry.mode = outcome` as a way to mark a Bundle entry
  carrying an `OperationOutcome`.
- Where the Search page currently lists the modes used in search-result
  Bundles, retain only the still-valid values for that context
  (`match`, `include`) and drop `outcome` from the list.
- Add a short note on the Search page (e.g., a sidebar or callout near the
  removed text) stating that `outcome` is deprecated as a value for
  `Bundle.entry.mode`, and that an `OperationOutcome` resource carrying
  warnings/errors about the search should instead appear as a top-level
  entry in the result Bundle (per the current guidance for returning
  `OperationOutcome` in search responses), with a link to the
  `Bundle.entry.mode` element definition for the deprecation record.

#### Justification

This is the narrowest fix that makes the Search page consistent with the
current state of `Bundle.entry.mode`, where `outcome` is documented as
deprecated. Leaving the Search page describing a deprecated value as if it
were normative misleads implementers and contradicts the element
definition. As a Technical Correction the change is in-scope for the
FHIR Infrastructure work group and does not alter normative behaviour —
it brings the narrative into line with the existing artifact metadata.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the discussion of `Bundle.entry.mode = outcome` on the Search page
but rewrite it as an explicit "deprecated / historical" note rather than
removing it, and pair that note with positive guidance on the supported
pattern. Concretely:

- Move all `outcome`-related text into a clearly labelled "Deprecated"
  block on the Search page (e.g., a `<div class="warning">` or equivalent
  callout) that states `Bundle.entry.mode = outcome` is deprecated, must
  not be produced by new servers, and may still be encountered when
  consuming results from older servers.
- Immediately above or below that block, document the current recommended
  pattern for returning an `OperationOutcome` from a search interaction
  (top-level Bundle entry without a `search.mode` value, or with a mode
  value that remains valid for the context).
- Cross-link the deprecation note back to the `Bundle.entry.mode`
  element page so readers can see the deprecation in the element
  definition.

#### Justification

Search responses produced by older servers in the wild may still carry
`Bundle.entry.mode = outcome`, and consumers will continue to need to
recognise that value for the foreseeable future. Outright removal from
the Search page (Disposition A's stronger reading) risks erasing the
institutional explanation of what those legacy responses mean, which is
unhelpful to implementers maintaining interop with older systems. A
visible "deprecated, here's what to do instead" block both honours the
deprecation and preserves the documentation that downstream implementers
rely on, at the cost of slightly more page churn than a simple deletion.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the Search page is correct as
written and that the deprecation of `outcome` is already recorded on the
`Bundle.entry.mode` element page itself, so the Search page's mention of
the value is not actively wrong — only stale. Close as
"Not Persuasive / Documentation already adequate".

#### Justification

If the deprecation note already exists on the `Bundle.entry.mode`
element definition, a strict reader of the spec can reach the correct
conclusion by following the link from the Search page to the element.
Declining keeps the editorial surface area small and avoids re-litigating
the deprecation discussion every time a downstream page references the
value.

---

### Recommendation

**Recommended disposition:** B

Disposition B (rewrite as an explicit deprecation block plus positive
guidance) best serves implementers. The reporter's underlying concern is
that the Search page reads as though `Bundle.entry.mode = outcome` is
still a recommended pattern, which is misleading; that is a real defect
worth fixing, so Disposition C (decline) is too weak and leaves the
misleading text in place. Pure removal (Disposition A) fixes the
misleading-recommendation problem but throws away context that consumers
of legacy search responses still need — they have to be able to look up
what an older server meant by `mode = outcome`. A clearly labelled
"Deprecated" block with a pointer to the current pattern delivers both
goals: the Search page stops endorsing a deprecated value, and the
historical explanation remains discoverable. The change is editorial,
scoped to the Search page in FHIR Core, and matches the ticket's framing
as a Technical Correction.
