## Summary

The reporter (DanielSchuster) asks the FHIR Infrastructure work group to
clarify the practical use case for sending an `If-Match` header carrying a
weak ETag on the `conditional-delete-single` interaction. The
conditional-delete-single interaction lets a client delete a resource by
search criteria rather than by logical id, and the spec also permits an
`If-Match` ETag so the server can refuse the delete if the matched
resource has changed since the client last saw it. The reporter observes
that, in nearly every scenario they can imagine, a client that has the
correct version-specific ETag for a resource must already have read that
resource and therefore already knows its logical id — at which point a
plain (non-conditional) `DELETE [type]/[id]` with `If-Match` would do
the same job. They are asking the work group to either supply the
counter-examples that justify keeping the combination in the spec or to
acknowledge that the combination has no real use case. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found
for this ticket in the available sources.

## Details

**Description:**

The conditional-delete-single interaction allows a client to delete a
resource/s based on some selection criteria rather than by a specific
logical ID. It also defines an If-Match header containing a weak E-Tag
to ensure that the resource to be deleted is up to date.

In most scenarios, if a client has access to the correct ETag value,
they would also have access to the FHIR ID and could just do a regular
Create request. I can't think off the top of my head of what
counterexamples might be.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Acknowledge that allowing `If-Match` / weak ETag on
`conditional-delete-single` has no demonstrated standalone use case
beyond what plain `DELETE [type]/[id]` with `If-Match` already provides,
and remove the explicit allowance from the
`conditional-delete-single` description in `http.html` (FHIR Core,
RESTful API section). Concretely: edit the Conditional Delete section to
either (a) drop the sentence permitting `If-Match` on
conditional-delete-single, or (b) add a normative SHALL NOT / "servers
SHOULD reject" clause stating that `If-Match` is not meaningful when the
target is selected by search criteria. Cross-reference the existing
`If-Match` / version-aware update text so readers are pointed at plain
DELETE-by-id when they need optimistic concurrency.

#### Justification

The reporter's argument is grounded in how the REST API actually composes:
a weak ETag is version-specific, so a client cannot legitimately obtain it
without first reading the resource — and that read returns the logical id.
Once the id is in hand, `DELETE [type]/[id]` with `If-Match` already
provides the optimistic-concurrency guarantee. Having two ways to express
the same intent invites server-implementer divergence (does the server
re-search after locking? does it apply `If-Match` against the search
result or against the version the client originally read?) and increases
the surface area for interop bugs without adding capability. Removing
the allowance simplifies the spec without removing any capability that
implementers actually rely on.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the allowance but add explicit guidance in `http.html`'s
Conditional Delete section describing the **one** scenario where
`If-Match` on conditional-delete-single is meaningful — the cached /
indirect-knowledge case — and tighten the server behaviour. Specifically:
amend the spec to say that when a client previously read a resource by
search and cached its version id (e.g., via `_summary` or a search bundle
that returned `meta.versionId`), it may re-issue the same search criteria
as a conditional delete with `If-Match` to atomically assert "delete the
single match, but only if it is still at the version I last saw." Add a
note that servers SHALL match the `If-Match` against the resource
selected by the conditional criteria (not against any later/earlier
version), and SHALL return `412 Precondition Failed` if the matched
resource's `versionId` does not match. This formalises the rare but
legitimate "I have the ETag from a search bundle, not from a direct
read" pattern.

#### Justification

Search bundles do return `meta.versionId` per entry, so it is technically
possible for a client to know an ETag without having issued a
read-by-id, and a strict-but-rare workflow (e.g., long-running batch
clients, some FHIRPath-driven cleanup tools, or clients operating against
a search-only facade) might genuinely want to re-assert the search and
the version atomically in one round-trip. Preserving the capability and
documenting the contract is more conservative than removing a feature
that some implementers may already rely on, and it directly answers the
reporter's "what counterexample" question rather than just deleting the
sentence. The trade-off versus Disposition A is added spec text and a
clearer server-side obligation, but no behaviour change for clients that
do not use `If-Match` on conditional delete.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive / Already Addressed**. Close with a comment
explaining that `If-Match` on `conditional-delete-single` is permitted
deliberately as the conditional analogue of version-aware delete, and
that the absence of an obvious motivating workflow is not a sufficient
reason to remove an established RESTful capability that is consistent
with how `If-Match` is permitted on conditional update. Direct the
reporter to the Concurrency section of `http.html` for the general
contract.

#### Justification

The combination of conditional interaction + `If-Match` is symmetric
with `conditional-update` (which has long permitted `If-Match`), so
removing it only from conditional-delete-single would create an
inconsistency in the RESTful API. Removing the allowance is also a
breaking change for any server or client that has implemented it, even
if the use case is niche. A pure-question ticket of the form "can
someone justify this existing feature to me?" is generally a poor basis
for normative change — the burden of proof for removing established
behaviour should rest on demonstrating harm, not absence of obvious
benefit.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible path: it directly answers the
reporter's question (yes, there is a counter-example: cached ETags from
search bundles), it preserves symmetry with conditional-update +
`If-Match`, and it removes the real risk the reporter implicitly
identifies — that the spec is silent on *how* the server should evaluate
`If-Match` when the target is selected by search rather than by id.
Disposition A is cleaner but discards a capability that has a (narrow)
legitimate use and would be a normative-breaking change for implementers
that have already wired it up. Disposition C dismisses a fair question
without improving the spec. Adding a short clarifying paragraph in
`http.html`'s Conditional Delete section, plus a normative statement
that the server SHALL evaluate `If-Match` against the
conditionally-selected resource, costs little and closes a genuine
ambiguity that the reporter has surfaced.
