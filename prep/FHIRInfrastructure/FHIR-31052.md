## Summary

The reporter (Varghese "Matt" Mathew) argues that the SMART Web Messaging
specification's four message groups — `scratchpad.*`, `ui.*`, `fhir.*`, and
`status.*` — are arbitrary and add complexity without value. Specifically,
the `scratchpad.*` group bakes an EHR implementation detail (whether or not
the host maintains a scratchpad / shopping-cart of unsaved work) into a
wire-level interface, which the reporter contends is not appropriate for a
specification. The ticket proposes (1) removing the `scratchpad` group
entirely, (2) collapsing `status.*` (which currently contains only the
EHR-capability handshake) into `ui.*` (renamed `ui.handshake` / `host.handshake`),
and (3) keeping `fhir.*` only as the special case where the EHR host acts as
a FHIR proxy on behalf of the app. The end state would be three orthogonal
groups: `ui.*` / `host.*` (app → EHR), `app.*` / `guest.*` (EHR → app), and
`fhir.*` (EHR-as-FHIR-proxy).

The ticket is the formal write-up of an extended Zulip debate and is
explicitly cross-referenced from two related Jira issues raised by the same
reporter:

- [FHIR-29349 — *Scratchpad undefined*](https://jira.hl7.org/browse/FHIR-29349)
  (originally filed by Bas van den Heuvel; resolved *Persuasive with Modification*)
  notes that "scratchpad" is not clearly defined in the spec.
- [FHIR-29368 — *I'm struggling with the notion of scratchpad*](https://jira.hl7.org/browse/FHIR-29368)
  (resolved *Not Persuasive*) raises that whether a scratchpad exists is an
  EHR implementation detail and should not appear in the spec.

The reporter has already opened
[HL7/smart-web-messaging#41](https://github.com/HL7/smart-web-messaging/pull/41)
implementing the proposed change. The reporter explicitly acknowledges this
is a hefty breaking change and may not land in an immediate ballot.

## Details

**Description:**

The SMART Web Messaging spec introduces three or four types of message
groups (`scratchpad.*`, `ui.*`, `fhir.*`, `status.*`). The reporter argues
these groupings are arbitrary and complicate the specification rather than
simplifying it.

*Scratchpad.* The "scratchpad" group assumes the EHR has a scratchpad. Two
prior tickets (FHIR-29349, FHIR-29368) point out that the term is undefined
and that whether such a buffer exists is an implementation detail. SMART Web
Messaging originated from the PAMA / CDS Hooks use-case where a SMART app
needs to place, remove, or update orders *with UI integration* — i.e. the
physician sees the change and is prompted to sign or react. It is already
possible to write to a FHIR server directly; the value SMART Web Messaging
adds is the UI side-effect, not the persistence model. Therefore the reporter
argues the messages should be `ui.create`, `ui.delete`, `ui.update` and the
question of whether the EHR happens to store them in a scratchpad / shopping
cart should not appear in the wire format.

*Status.* The `status.*` group today contains a single message — an EHR
capability handshake. The reporter proposes folding this into `ui.handshake`
(or `host.handshake`). A handshake in the opposite direction (host → app) is
semantically different (a status check rather than registration) and should
be a separate message such as `app.status` / `guest.status`, not part of the
same group.

*FHIR.* The `fhir.*` group is the EHR-as-FHIR-proxy case. The reporter is
willing to keep `fhir.*` as a distinct group precisely because that role is
"special enough" to merit its own namespace — `fhir.*` MUST be semantically
equivalent to the app calling the FHIR server directly.

*Summary.* Remove `scratchpad`; keep three groups: `ui.*` / `host.*`
(app → EHR), `app.*` / `guest.*` (EHR → app), and `fhir.*` (FHIR proxy). The
reporter notes this is a hefty breaking change and asks the WG to prioritise
how to move forward.

## Related Zulip Discussions

### [smart > Web messaging, scratchpad](https://chat.fhir.org/#narrow/stream/smart/topic/Web%20messaging%2C%20scratchpad/near/226672219)

A focused thread (17 Feb 2021) immediately following the filing of
FHIR-31052. Matt Varghese restates the case for replacing `scratchpad.*`
with `ui.create` / `ui.delete` / `ui.update`, arguing the word "endpoint"
or "host" better captures the entity providing managed access to in-progress
work. Josh Mandel pushes back: he reads this as a *rename* that preserves
semantics (which he says he is open to discussing) and is wary of changes
to language the WG had voted on the previous week. He recommends Matt land
a concrete PR to drive the discussion concretely — the PR Matt then opened
is HL7/smart-web-messaging#41. Lloyd McKenzie reinforces a hard requirement:
content in the scratchpad MUST NOT be silently written through to the REST
endpoint without the user's intent, framing scratchpad as fundamentally
about access-control over unfinalized content rather than about a UI
metaphor. Process-wise the thread converges on "open the PR, document the
lineage from the prior vote, iterate before the next vote". No final
consensus on the naming was reached.

Participants: Matt Varghese, Josh Mandel, Lloyd McKenzie, Carl Anderson
(referenced).

### [implementers > FHIR, Scratchpad, SMART Web Messaging, and unfinalized in...](https://chat.fhir.org/#narrow/stream/implementers/topic/FHIR%2C%20Scratchpad%2C%20SMART%20Web%20Messaging%2C%20and%20unfinalized%20in...)

The longer pre-history thread (12–16 Feb 2021) that motivated FHIR-31052.
Matt argues that "unfinalized" data should be modelled with FHIR's existing
draft `status` codes on resources like `MedicationRequest` / `ServiceRequest`
rather than with a separate scratchpad concept; he uses the example of
pended orders that must still go through CDS to argue scratchpad and the
REST endpoint are not disjoint. Lloyd McKenzie counters that "scratchpad"
is content not yet persisted at all (or that the user has no obligation /
intent to persist) and that this is materially different from a draft
resource on a server — not all servers can or should accept draft writes
(e.g. tablet authoring, cloud-stored intermediate state). Josh Mandel
summarises: (1) the use-case for accessing not-yet-persisted, FHIR-shaped
data is real (CDS Hooks is precedent); (2) when surfacing such data, apps
need a clear contract that does not silently leak to the REST API. John
Moehrke characterises the underlying questions as systems-design / policy
beyond core FHIR. The thread ends without consensus, with Matt explicitly
pointing to FHIR-31052 as the formal venue to continue the discussion.

Participants: Matt Varghese, Josh Mandel, Lloyd McKenzie, Paul Church,
John Moehrke.

## Proposed Dispositions

> **Caveat:** repo-analysis briefing for `HL7/smart-web-messaging` is not
> available; specific authoring paths below are inferred from the spec name

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's three-group model in `HL7/smart-web-messaging`:

1. **Remove the `scratchpad.*` message group.** Delete the `scratchpad`
   message-group section from the SMART Web Messaging spec pages and the
   message catalogue, including any references in the worked examples.
2. **Replace the existing `scratchpad.create` / `scratchpad.update` /
   `scratchpad.delete` semantics with `ui.create` / `ui.update` /
   `ui.delete`** in the `ui.*` group. Document that these messages request
   the EHR host to perform the operation with appropriate UI integration;
   whether the host backs that with an internal scratchpad / shopping cart
   is left to the implementation.
3. **Collapse the `status.*` group into `ui.*`.** Rename the existing
   `status.handshake` to `ui.handshake` (or `host.handshake`). Reserve
   `app.*` / `guest.*` for the symmetric host→app direction (e.g. a future
   `app.status`).
4. **Retain `fhir.*` as its own group** for the EHR-as-FHIR-proxy case,
   with the conformance statement that any `fhir.*` message MUST be
   semantically equivalent to the app calling the FHIR server directly.
5. Land changes via the in-flight
   [HL7/smart-web-messaging#41](https://github.com/HL7/smart-web-messaging/pull/41),
   updating the PR to also delete (rather than deprecate) the `scratchpad.*`
   names since the spec has not yet been balloted.

#### Justification

The SMART Web Messaging spec is pre-ballot, so a hard rename / delete is
cheaper here than at any later point and avoids carrying deprecated names
into the first published version. The reporter's core argument is sound on
spec-design grounds: the wire format describes the *interaction* between
app and host; whether the host happens to buffer unsigned content is an
implementation strategy, not part of the contract. Three orthogonal groups
keyed on *direction* (`ui.*` / `app.*`) plus the *FHIR-proxy* special case
is a more legible mental model than four groups split partly by direction
and partly by storage assumption. FHIR-29349 (resolved Persuasive with
Modification) already concedes the term "scratchpad" needs work; cutting
the term resolves that ticket cleanly as a side effect.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the *renaming* of message groups but **preserve the access-control
semantics that Lloyd McKenzie raised on the Zulip thread**:

1. Rename `scratchpad.*` → `ui.*` (and the associated handshake to
   `ui.handshake`), keeping the three-group structure proposed in
   Disposition A: `ui.*`, `app.*` (or `guest.*`), `fhir.*`.
2. **Carry forward an explicit normative statement** in the `ui.*`
   section (or in a shared "Conformance" section) that:
   - Content delivered or modified via `ui.*` MUST NOT be silently
     persisted by the host to the FHIR REST endpoint without user intent.
   - Hosts MUST apply access control such that an app receiving content
     via `ui.*` does not gain authority to propagate it to the RESTful API.
3. Add a non-normative implementation note explaining that hosts are free
   to back `ui.*` with an internal scratchpad, shopping cart, or any other
   mechanism; the term "scratchpad" appears only as informative example.
4. Land via PR #41 with the additional conformance language layered on top
   of the rename.

#### Justification

This addresses the spec-design objection (the reporter's main concern)
without losing the substantive protection Lloyd flagged on Zulip — that
scratchpad-style content is fundamentally about *unsaved, access-controlled,
in-progress* data and that the SHALL-NOT-write-through requirement must
survive the rename. Disposition A as written deletes the term but is silent
on the underlying access-control rule, which would be a regression from the
intent of the language the WG voted on the week before this ticket was
filed. This option also preserves the work captured in FHIR-29349's
"Persuasive with Modification" resolution by re-expressing the definition
as conformance language rather than as a named message group. Trade-off
vs. A: slightly more spec text and one more normative statement to ballot;
benefit is that the implication of FHIR-29368's "Not Persuasive" decision
(that the access-control intent is real and worth keeping) is honoured.

---

### Disposition C: Decline

#### Proposal

Close FHIR-31052 as **Not Persuasive**, mirroring the disposition of the
reporter's earlier FHIR-29368. Keep the existing four groups (`scratchpad.*`,
`ui.*`, `fhir.*`, `status.*`) and instead resolve FHIR-29349 ("Scratchpad
undefined") with a clarifying definition of `scratchpad` in the spec,
explicitly contrasting it with the FHIR REST endpoint and with
draft-status resources.

#### Justification

The Zulip discussion shows the WG's senior voices (Josh Mandel, Lloyd
McKenzie) read the ticket as primarily a rename request that does not
materially improve the contract, and worry that reopening a vote held the
prior week is process churn for a spec that is still pre-publication. The
prior ticket on the same complaint (FHIR-29368) was already declined as
Not Persuasive. The breaking-change cost — even pre-ballot — is non-zero
because there are early implementers tracking the spec. If the substantive
issue is "scratchpad is undefined", the targeted fix is to define it
(closing FHIR-29349's modification commitment), not to delete the
abstraction.

---

### Recommendation

**Recommended disposition:** B

The reporter's spec-design argument is genuinely strong: hard-coding
"scratchpad" into the wire format conflates *what the app is asking the
host to do* (UI-mediated create / update / delete) with *how the host
chooses to buffer it*, and a four-group taxonomy split on inconsistent
axes (storage vs. direction) is harder to teach than three groups split
purely by direction plus a `fhir.*` proxy escape hatch. The
already-existing PR #41 means the cost of action is mostly review, not
authoring.

That said, Disposition A as written drops the substantive
access-control / no-silent-write-through rule that Lloyd McKenzie made the
WG aware of on Zulip and that the prior-week vote was about. Disposition C
preserves that rule but at the cost of carrying a name (`scratchpad`) that
two separate tickets — one of which the WG already accepted as Persuasive
with Modification — agree is unclear. Disposition B captures the
Disposition A rename *and* re-expresses the Disposition C / prior-vote
intent as normative conformance language attached to `ui.*`, which is the
combination most likely to satisfy both the reporter and the senior
reviewers. It also lets FHIR-29349 and FHIR-29368 be closed coherently as
side effects.
