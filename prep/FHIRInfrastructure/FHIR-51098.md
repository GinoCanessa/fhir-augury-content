## Summary

Grahame Grieve requests adding a new element to the FHIR `Signature`
datatype to explicitly carry the canonicalization algorithm that was
applied when the resource (typically a `Bundle`) was signed. Today the
canonicalization method is embedded inside the JWS protected header
(when `sigFormat = application/jose`) and is also signed there, but
there is no signature-level convenience element that lets a reader
discover the scope/algorithm of a signature without parsing the JWS.
The reporter argues that — like `Signature.type`, `.when`, and `.who` —
this is useful "alternative-selection" metadata even for parties that
are not verifying the signature: when multiple `Signature` instances
exist, knowing each one's canonicalization lets a recipient pick the
`Signature.data` they actually want. The new element is intentionally
not the trusted value; the trusted value remains the one inside the
signed payload.

The request is closely tied to [FHIR-51099](https://jira.hl7.org/browse/FHIR-51099)
("Bundle.signature should be 0..\*"), which proposed making
`Bundle.signature` repeating so a Bundle can carry multiple
signatures — different purposes, signers, target formats, or
canonicalizations. FHIR-51099 has since been resolved
**Persuasive with Modification** with a different remedy: rather than
relax `Bundle.signature` cardinality, FHIR Infrastructure plans to
deprecate `Bundle.signature` in favor of carrying multiple
signatures via `Bundle.entry` Provenance resources. That outcome
weakens (but does not remove) one of the Zulip rationales for FHIR-51098,
since multi-signature scenarios will increasingly live outside
`Bundle.signature`. The "alternative-selection" use case still applies
wherever multiple `Signature` instances are presented together (e.g.,
sibling Provenance signatures, or other resources that carry
`Signature` directly).

## Details

**Description:**

When creating a signature on a resource — typically, a Bundle — the
resource has to be canonicalized as part of the signature process. The
standard defines several different values for how a resource was
canonicalised that have to do with the scope of what's being signed.

The canonicalisation method has to be written into the JWS header, if
you're doing `sigFormat = application/jose`, and it needs to be there —
it needs to be signed.

But it's an interesting thing to know even if you're not verifying the
signature — what's the scope of the signature. The best way is to add
an element for this to `Signature`.

See [Signature on Zulip](https://chat.fhir.org/#narrow/channel/179247-Security-and-Privacy/topic/Signature/with/523945922).

**Comments:**

- **John Moehrke (2025-06-13):** "Where multiple signatures are
  available, knowing the canonicalization of each would also allow you
  to select the signature `.data` that is the one you want." Frames
  the new element as analogous to `.type`, `.when`, `.who` — convenient
  metadata, not a trusted value.
- **John Moehrke (2025-06-13):** Suggests the new element should be
  `0..*`, because both XML-Signature and JOSE permit multiple
  transforms in principle.
- **John Moehrke (2025-06-16):** "Discussed in Security WG. We agreed
  to elevate this to a CP. Discussed that we agree with the proposal.
  Ran out of time to complete the disposition and voting. Will discuss
  next call."
- **John Moehrke (2025-06-16):** Notes that to support multiple
  signatures with different purposes, signers, times, or
  canonicalizations on a Bundle, [FHIR-51099](https://jira.hl7.org/browse/FHIR-51099)
  is also needed (relax `Bundle.signature` to `0..*`). FHIR-51099 has
  since been resolved Persuasive with Modification — the workgroup
  intends to deprecate `Bundle.signature` in favor of Provenance
  entries, so this dependency is partially addressed but in a different
  direction than FHIR-51098 originally assumed.

Companion ticket filed by John Moehrke the same day. It originally

## Related Zulip Discussions

### [Security and Privacy > Signature](https://chat.fhir.org/#narrow/stream/Security%20and%20Privacy/topic/Signature)

The thread starts with Grahame Grieve sharing a signed Bundle and
walking through the JWS/JOSE workflow he used (canonicalize per
`http://hl7.org/fhir/canonicalization/json`, then sign). He proposes
encoding the canonicalization in `targetFormat` as a MIME parameter
(`application/fhir+json;canonicalization=...`), and Gino Canessa
independently arrives at the same syntax. John Moehrke initially
prefers keeping `targetFormat` purely a mime-type and adds that the
canonicalization must be signed (it already is, inside the JWS
header). The conversation then converges on adding a **separate
`Signature` element** for the canonicalization algorithm — Gino notes
that an element is more durable than a MIME parameter (parameters can
be dropped or lost across reserializations) and that an element makes
multi-signature selection cleaner; Grahame agrees. John initially
proposes `0..1`, then revises to `0..*` after looking at XML-Signature
(multiple transforms) and JWS JSON Serialization (multiple signatures
with distinct protected headers). Grahame is skeptical that anyone
actually emits multiple canonicalizations in one signature in practice,
but does not block `0..*`. The thread also surfaces adjacent issues
that are out of scope for this ticket: detached vs. enveloped JWS
mechanics (Steve Swinsburg's bundle was enveloped, not detached, which
Grahame later corrects with an apology and a worked detached example),
the gap that canonical JSON/XML pages do not yet describe stripping
`Bundle.signature` for re-verification (tracked in
[FHIR-45023](https://jira.hl7.org/browse/FHIR-45023), assigned to
ITS), and a recurring debate between Grahame and John about whether
`Signature.who` should be considered trustworthy when it is not itself
covered by the signature. John also pitches a future Signature IG;
Eric Haas expresses interest and points at the Da Vinci CDEX
signatures guidance as prior art.

Participants: Grahame Grieve, John Moehrke, Gino Canessa, Steve
Swinsburg, Eric Haas.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new element to the `Signature` datatype in FHIR Core to carry the
canonicalization algorithm(s) applied to the signed content:

- **Name:** `Signature.canonicalization`
- **Cardinality:** `0..*` (per the Zulip and ticket consensus that
  XML-Signature transforms and JWS JSON Serialization can in principle
  involve multiple canonicalizations)
- **Type:** `uri` (a canonical URL identifying the algorithm, e.g.
  `http://hl7.org/fhir/canonicalization/json`,
  `http://hl7.org/fhir/canonicalization/xml`, or one of the
  scope-qualified variants such as `…/json#data`)
- **Binding:** Bind to the existing canonicalization-method value set
  used by the FHIR canonicalization spec (extensible).
- **Definition text** must mirror the framing for `Signature.type`,
  `.when`, and `.who`: convenience metadata for recipients to select
  among alternative signatures; the trusted value is the algorithm
  asserted inside the signed payload (e.g., the JWS protected header's
  `canon`/`alg` claims, or the XML-Signature `<Transforms>` element).
  The element MUST NOT be relied upon for verification.
- Update the Signature page guidance to describe how an implementer
  should populate this element when producing JOSE or XML-Signature
  output, and to warn verifiers that the in-payload value wins on
  mismatch.

#### Justification

This is exactly what the Zulip thread converged on after Grahame, Gino,
and (eventually) John agreed an element was more durable than a MIME
parameter and cleaner for multi-signature selection. It is consistent
with the existing pattern of `Signature.type`/`.when`/`.who` —
convenience metadata that mirrors signed content but is not itself
trusted — and the Security WG has already minuted agreement with the
proposal pending formal disposition. `0..*` is defensible because both
XML-Signature `<Transforms>` and JWS JSON Serialization permit multiple
canonicalizations per signature, and the cost of `0..*` over `0..1` is
negligible while preempting a future breaking change. Because
`Signature` is currently Trial Use, adding an optional element is a
non-breaking change in R6.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a new core element. Instead:

1. Define a **MIME parameter** convention on `Signature.targetFormat`
   so the canonicalization can be expressed as
   `application/fhir+json;canonicalization=http://hl7.org/fhir/canonicalization/json`
   (this is the syntax Grahame and Gino both initially proposed and
   that Grahame's worked example in the thread already uses).
2. For implementers who need a structured (non-MIME-parameter) form,
   publish a standard **extension** on `Signature` —
   `http://hl7.org/fhir/StructureDefinition/signature-canonicalization`
   typed `uri`, `0..*`, bound to the canonicalization value set —
   rather than a core element.
3. Update the Signature page to document both the MIME-parameter
   convention and the extension, and reiterate that the trusted value
   lives inside the signed payload.

#### Justification

This keeps the `Signature` core surface area minimal during the push to
make `Signature` normative in R6 (John explicitly noted in-thread that
the WG is "struggling to get community engagement" on Signature).
`targetFormat` is already an element-typed `code` whose MIME value can
legitimately carry parameters, so this approach reuses existing
machinery. An extension provides a structured fallback for tooling that
cannot rely on MIME parameters. The trade-off vs. Disposition A is
durability: as Gino noted in the thread, MIME parameters can be
silently dropped on reserialization, and an extension is easy for
implementers to ignore — both of which weaken the
"alternative-selection" use case that motivated the ticket.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by existing mechanisms** and
**weakened in scope by the resolution of FHIR-51099**. Specifically:

- The canonicalization algorithm is already carried — and signed —
  inside the JWS protected header (for `application/jose`) and inside
  `<Transforms>` (for XML-Signature). Recipients can read it there
  without trusting it.
- The MIME-parameter convention on `Signature.targetFormat`
  (`application/fhir+json;canonicalization=…`) is already legal and
  is what Grahame's own worked example in the Zulip thread emits.
- Multi-signature scenarios on Bundles will, per the resolution of
  FHIR-51099, move to `Bundle.entry` + `Provenance` rather than
  multiple `Bundle.signature` instances, so the "pick which sibling
  signature to use" use case is no longer a `Signature`-level concern
  for Bundles.
- Adding a non-trusted element invites misuse: implementers will
  inevitably read it without verifying the signature, then act on a
  value the producer can lie about. The Zulip thread already shows
  this debate (Grahame vs. John on whether `.who` is trustworthy).

#### Justification

The information the ticket wants exposed is already present and signed
in every conformant `application/jose` signature (and in XML-Signature
`<Transforms>`). The only "convenience" gap is parsing the JWS header,
which any implementation handling `Signature.data` already has to do.
Coupled with FHIR-51099's resolution to deprecate `Bundle.signature`,
the marginal benefit of a new core element is small and the risk of
producing yet another untrusted-but-acted-upon field is real.

---

### Recommendation

**Recommended disposition:** A — Accept As Requested, with the
specifics in Disposition A's proposal (`Signature.canonicalization
0..*` typed `uri`, extensibly bound, documented as convenience
metadata).

The Security WG has already minuted agreement with the proposal
pending formal disposition, and the design fits the existing
`Signature` pattern of mirrored convenience metadata
(`type`/`when`/`who`). Disposition B's MIME-parameter / extension path
is technically workable but sacrifices durability — Gino's point in
the thread that MIME parameters get dropped on reserialization is the
strongest single argument against B and for an actual element.
Disposition C overweights the "already in the JWS header" argument:
the explicit motivation in the ticket is precisely that recipients
should be able to discover scope without parsing the signed payload,
which is consistent with how `Signature.who` and `.when` already work.
The resolution of FHIR-51099 narrows but does not eliminate the
multi-signature selection use case (sibling `Signature` instances on
non-Bundle resources, and Provenance-carried signatures still benefit
from per-instance canonicalization metadata). Because `Signature` is
Trial Use in R5, adding an optional `0..*` element is non-breaking and
should be done now, before the R6 normative push lands.
