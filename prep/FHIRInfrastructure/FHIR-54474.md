## Summary

The ticket asks that the `source` of a FHIR Mapping Language (FML) rule —
which today refers to a single property/element on the source object —
be allowed to be a full FHIRPath expression. The motivating use case is
splitting a single delimited input field into multiple values that can
then be projected into a repeating target (e.g. taking a comma‑separated
`codes` field on the source and producing one `Coding` per token in
`target.coding`). Brian Postlethwaite frames the request as filling a
gap where there is currently "no way to take a single property and split
it into multiple properties/values"; his suggested syntax is along the
lines of `(src.codes.split(',')) as scv -> tgt.coding as c, c.code = scv`.

The cross‑reference data surfaces no other Jira tickets that link to
FHIR-54474 directly. The two related conversations are both on Zulip:
the originating design thread `#FHIR Mapping Language > fhirpath
expressions as source` (where Brian and Michael Lawley work through the
delimiter‑split scenario), and a separate `#FHIR Mapping Language >
Move FML to a new spec?` thread, where Grahame Grieve cites this very
ticket as evidence that "many of the proposed fixes to FML require
updates to the StructureMap resource", which is why splitting FML out
into its own spec is being debated. So FHIR-54474 sits at the
intersection of two questions: (1) can FML express multi‑valued source
selections with FHIRPath, and (2) where does that change land —
StructureMap in core, an FML‑only spec, or both.

## Details

**Description:**

Currently there is no way to take a single property and split it into multiple properties/values.

To achieve this we could update the **source** to be able to be a fhirpath expression (so that you can split an input field into multiple values).

e.g.

```
(src.codes.split(',')) as scv -> tgt.coding as c, c.code = scv
// example that shows taking an input field that is comma separated, and populate that into a set of codings
```

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the FML `source` clause and the underlying
`StructureMap.group.rule.source` so that the element selector may be a
full FHIRPath expression rather than only a single navigable property
name. Concretely:

- Update the FML grammar so the production currently expecting
  `src.element` accepts a FHIRPath expression, parenthesised when it is
  not a simple path (e.g. `(src.codes.split(','))`).
- On the StructureMap resource, either (a) widen
  `StructureMap.group.rule.source.element` (currently `string`,
  intended as a single element name) to permit a FHIRPath expression,
  documenting the change in scope, or (b) add a sibling element such as
  `StructureMap.group.rule.source.expression` (FHIRPath) that, when
  present, supplants `element` and yields the rule's input collection.
  Option (b) is the safer wire‑level change because it is purely
  additive.
- Specify that the rule iterates once per item in the collection
  produced by the FHIRPath expression, with the `as` alias (e.g. `scv`)
  bound to each item — matching how `as` already works for navigated
  source elements today.
- Add the example from the ticket (`(src.codes.split(',')) as scv ->
  tgt.coding as c, c.code = scv`) to the FML page as a worked example
  of multi‑valued source projection.

#### Justification

The request fills a real expressive gap that today forces implementers
into either pre‑processing inputs outside FML or contorted multi‑rule
workarounds. FHIRPath is already the expression language of FHIR and
is already used elsewhere in StructureMap (e.g. on the target side via
`transform = evaluate`), so reusing it on the source keeps the spec
internally consistent rather than inventing a new sub‑language. The
Zulip design thread shows the use case is concrete (delimited inputs
to repeating `Coding`s) and not hypothetical. Choosing the additive
`source.expression` variant (option b above) means existing
StructureMaps and existing engines continue to validate and run
unchanged; only engines that opt in need to evaluate the FHIRPath
collection — which mirrors how FHIRPath was rolled into other
resources.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `StructureMap.group.rule.source.element` as a single‑element
selector on the wire, and instead introduce a small set of named FML
"source transforms" that cover the missing collection‑producing
operations — most importantly `split(delimiter)` — in the same family
as the existing `transform` codes used on the target side. The FML
syntax could remain close to what the ticket proposes (e.g. `src.codes
split ',' as scv -> tgt.coding as c, c.code = scv`) but compile down
to a structured representation in StructureMap (a new `transform`
value such as `split` carried on the source, with `parameter` for the
delimiter), rather than an opaque FHIRPath blob.

#### Justification

This addresses the underlying need (one input → many target
occurrences) without giving the source side the full power and
ambiguity of FHIRPath. It keeps StructureMap a relatively
deterministic, statically analysable structure — which matters for
tooling (mapbuilder, matchbox, the dotnet engine, MaLac‑HD) that does
not necessarily ship a full FHIRPath engine and for visualisation /
diffing of maps. It also avoids the question Grahame raises in the
"Move FML to a new spec?" thread, namely that any FHIRPath‑on‑source
change is simultaneously a StructureMap resource change in core; a
constrained `transform = split` is a smaller, more reviewable
StructureMap delta. The trade‑off is expressiveness: only the
operations the WG explicitly enumerates become available, so future
multi‑valued patterns require new tracker items rather than just
"write some FHIRPath".

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for the current FML/StructureMap normative
window. Document the workaround patterns (pre‑split into a transient
structure before invoking the map, or use multiple chained rules /
helper groups) in the FML page, and re‑file the request against the
prospective standalone FML specification once the
"Move FML to a new spec?" governance question (FHIR-54473) is settled.

#### Justification

The Zulip "Move FML to a new spec?" thread shows there is unresolved
disagreement about *where* FML lives and how aggressively its grammar
should evolve relative to StructureMap. Landing a non‑trivial
expressivity change to `source` while that question is open risks
committing the core spec to a syntax/semantics that the new home for
FML may then need to revise. This is also a backwards‑compatibility
hazard: engines that don't yet evaluate FHIRPath on the source will
silently fail to execute maps written against the new feature. If the
underlying capability (split a delimited field across repeating
targets) is needed urgently, implementers can already achieve it via
a helper group or via a `transform = evaluate` on the target side; the
absence of source‑side FHIRPath is not a hard blocker.

---

### Recommendation

**Recommended disposition:** A (with the additive
`source.expression` variant from option b in the proposal).

Disposition A delivers what implementers — including the engine
authors active on the Zulip thread — are asking for, and FHIRPath is
already the project's chosen expression language, so reusing it on the
source is the simplest, most internally consistent answer. Done as a
new optional `source.expression` element rather than by repurposing
`source.element`, the StructureMap wire change is additive and does
not invalidate any existing map or engine, which directly answers the
backwards‑compatibility concern that drives Disposition C and most of
the "Move FML to a new spec?" anxiety. Disposition B is an attractive
fallback if the WG decides FHIRPath on the source is too open‑ended,
but it pays for that with an open‑ended series of future
"please add transform X" tracker items, whereas Disposition A solves
the whole class. The recommendation is therefore A; Disposition C
should only be selected if the WG concludes that the FML spec home
must be settled (FHIR-54473) before any expressivity change can be
accepted.
