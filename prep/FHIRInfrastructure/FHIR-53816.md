## Summary

The ticket reports that a section of the FHIR Core specification contains
three hyperlinks pointing at the "API Incubator" Implementation Guide
(base URI `https://build.fhir.org/ig/HL7/api-incubator/`), all of which
return HTTP 404. The reporter does not name the specific FHIR Core page
that hosts the broken links, so triage will need to locate the
offending section before a fix can be applied. The same underlying
problem — outbound links from FHIR Core into the API Incubator IG that
no longer resolve — is also reported in
[FHIR-53868](https://jira.hl7.org/browse/FHIR-53868) (Broken links to
Asynchronous Interaction Request Pattern), which calls out the
specific URL `https://build.fhir.org/ig/HL7/api-incubator/async-bundle.html`.
Together, these tickets indicate that the move of incubated content
out of FHIR R6 core into the API Incubator IG (discussed extensively
on Zulip during late 2025 / early 2026) left dangling references
behind in the core spec because the published IG is not (yet) at the
URL the core spec points to.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a Technical Correction against FHIR Core. As part
of the fix:

1. Sweep all FHIR Core source pages (the publication source under
   `HL7/fhir`) for any anchor whose `href` starts with
   `https://build.fhir.org/ig/HL7/api-incubator` (and the build /
   `hl7.org/fhir/uv/api-incubator` variants), and fix the URLs to
   point at the canonical published location of the API Incubator
   IG. Confirmed instances to repair from the related tickets and
   Zulip context include:
   - `async-bundle.html` (the section reported in the sibling
     ticket FHIR-53868 — Asynchronous Interaction Request Pattern).
   - `operations-for-large-resources.html` (per the implementers
     thread on ConceptMap incremental updates and FHIR-52912).
   - GraphDefinition / `$graph` / `$current` / `$find` references
     from the Conformance module pages.
2. Update the "where moved to" note added for balloters (see fmg
   thread "API Incubator note to balloters") so the URLs it shows
   line up with the IG's actual publication path.
3. Where the incubator IG is not yet published at a stable URL,
   either gate the link behind a redirect from `build.fhir.org` or
   add an inline note that the target IG is still in the
   incubation phase, rather than linking to a 404.
4. Resolve FHIR-53868 as a duplicate / sibling once the sweep is
   complete, since it is one specific instance of the same defect.

#### Justification

The reporter's claim is verifiable (the URL pattern 404s today and is
independently reported in FHIR-53868), the fix is purely editorial
(no resource model change), and the FMG thread "API Incubator note
to balloters" makes clear that linking from R6 core into the API
Incubator IG is the agreed pattern — so deleting the links is not
the right answer. Doing the sweep once, instead of patching one
link per ticket, prevents an open-ended trickle of duplicate
Triaged corrections.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the link rot at the publication-infrastructure layer instead of
inside each FHIR Core source page:

1. Establish a permanent redirect on `build.fhir.org` (and on the
   eventual `hl7.org/fhir/uv/...` publication path) from
   `/ig/HL7/api-incubator/*` to wherever the API Incubator IG is
   actually published — e.g., `/ig/HL7/api-incubator-ig/*` per the
   FMG "FHIR IG Incubator Creation" thread that introduced the
   `HL7/api-incubator-ig` repo.
2. Add a build-time link checker step (or extend the existing one)
   that fails the FHIR Core publication run when an outbound link
   into `build.fhir.org/ig/HL7/api-incubator*` returns non-2xx, so
   future renames cannot silently introduce 404s.
3. Leave the in-source URLs in FHIR Core alone for this ticket; only
   patch the small number of pages that the link checker subsequently
   flags.

#### Justification

This addresses the underlying cause — the API Incubator IG was
renamed / relocated after FHIR Core was authored — rather than the
symptom. It scales to future incubator restructurings (which the
"FHIR IG Incubator Creation" thread shows are still ongoing), and
adding the link-checker gate prevents the same class of bug from
re-appearing each ballot cycle. Trade-off vs. Disposition A: it
requires coordination with the publishing infrastructure
(`build.fhir.org` redirect rules) and with the API Incubator IG
maintainers, which is more work for FHIR Infrastructure than a
pure source-text edit. It also leaves the FHIR Core source with
URLs that only resolve via redirect, which some reviewers may
dislike.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Not Persuasive" / Decline on the grounds that
the API Incubator IG is, by definition, an evolving incubator whose
URLs will continue to change, and that fixing individual outbound
links into it from FHIR Core creates ongoing churn without lasting
value. Recommend instead that the reporter open an issue against
the API Incubator IG itself once it is stably published, and
remove (rather than re-target) the offending links from FHIR Core
in a future technical correction.

#### Justification

The "FHIR IG Incubator Creation" thread shows the incubator repo
and its publication URL are still being shaped, so any URL fixed
today is a candidate to break again. The FMG "API Incubator note
to balloters" thread explicitly frames the API Incubator linkage
as something to comment on during ballot — i.e., the right venue
is ballot, not a Triaged technical correction. Declining keeps
FHIR Core source clean of speculative outbound links.

---

### Recommendation

**Recommended disposition:** A (with the link-checker idea from B
folded in as a preventative follow-up)

The ticket is a straightforward editorial defect: the cited URL
pattern 404s today, a sibling ticket (FHIR-53868) confirms the same
problem against a specific page, and Zulip discussion across FMG,
FHIR Infrastructure and implementers makes clear that linking from
R6 core into the API Incubator IG is intentional — so these links
need to work, not disappear. Disposition C ignores the fact that
implementers (e.g., on the `ConceptMap incremental updates` thread)
are already hitting the 404s in real use; Disposition B is the
right long-term hygiene play but is more disruptive than this
single Triaged correction warrants on its own. Accepting the
ticket as a sweep across `HL7/fhir` for `api-incubator` links —
folding in FHIR-53868 as a duplicate, and opening a follow-up
ticket for the build-time link-checker — is the cleanest path
forward and the easiest for the WG to action this cycle.
