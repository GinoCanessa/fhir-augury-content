## Summary

Brian Postlethwaite asks the FHIR Infrastructure work group to extend the
`checkModifiers()` FHIRPath function in two ways. First, the `modifier`
argument (the caller-supplied list of known modifier-extension URLs) should
become optional: when it is omitted, `checkModifiers()` should not raise an
error and should instead succeed only when there are *no* modifier extensions
present at all — letting authors write invariants that detect any unknown
modifier extension without having to enumerate a known list. Second, an
optional `recursive` flag should be added so the function walks the input
node's descendants, not just its immediate modifier extensions, allowing a
single invariant to assert "no unknown modifier extensions anywhere in this
subtree." The ticket also asks that the spec's worked examples cover both
the with-parameter and parameter-less invocations. No cross-referenced Jira
tickets, Zulip discussions, or GitHub items were surfaced for this issue.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `checkModifiers()` FHIRPath function definition in the FHIR Core
spec (FHIRPath supplements / "Additional functions" page,
`fhirpath.html#functions`) to:

1. Change the signature so `modifier` is optional:
   `collection.checkModifiers([modifier : List<canonical>])`.
   When `modifier` is omitted (or empty), the function returns `true` only
   if every element in the input collection has *no* modifier extensions at
   all; if any modifier extension is present, it returns `false` (i.e.
   "there are unknown modifier extensions, because none are known"). When
   `modifier` is supplied, semantics are unchanged: `true` iff every
   modifier extension's URL is in the supplied list.
2. Add an optional second parameter, `recursive : Boolean` (default
   `false`). When `true`, the function recurses into all descendant
   elements of each input node and applies the same check at every level,
   returning `true` only if every modifier extension found anywhere in the
   subtree is either known (per `modifier`) or absent (when `modifier` is
   omitted).
3. Specify that calling `checkModifiers()` with no arguments must not
   raise an error (this is a behavior change from the current spec, where
   a missing `modifier` may be treated as a parse / evaluation error).
4. Add two worked examples to the function's spec page: one with an
   explicit `modifier` list, and one with no parameter, each shown against
   a sample resource fragment with and without modifier extensions.
5. Bump the page change-tracker / publish a focused balloted change note,
   and ensure validator implementations (HAPI, .NET / firely-net-sdk,
   pathling, fhirpath.js) get an issue raised against the published
   change.

#### Justification

The two behaviors requested are consistent with how `checkModifiers()` is
already used in core invariants (e.g. the `ele-1` family of checks and
profile-level constraints that assert "no unknown modifier extensions"):
authors today have to either enumerate every URL they know about or fall
back to a hand-rolled FHIRPath expression. Making `modifier` optional
collapses the common "I know of no modifier extensions for this profile"
case into a single, declarative call, and a `recursive` flag matches the
existing pattern in other tree-walking FHIRPath helpers (e.g. `descendants()`,
`children()`). Both additions are backwards-compatible for the existing
two-argument call site and only relax behavior, so they should not break
any conforming implementation today.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the "make `modifier` optional" half of the request, but **decline
the new `recursive` flag** and instead document the canonical recursive
idiom using existing FHIRPath:

```
descendants().checkModifiers(%knownModifiers)
```

and, for the parameter-less variant:

```
descendants().checkModifiers()
```

Add the worked examples Brian asks for (with-parameter and without), and
in the function's spec page add a short note: "To check modifier
extensions recursively, combine `checkModifiers()` with `descendants()` or
`repeat(children())`." Also add a single non-normative example invariant
on a base resource page demonstrating the recursive idiom, so authors can
copy it.

#### Justification

`checkModifiers()` is a FHIRPath function, and FHIRPath already has
first-class machinery for tree traversal (`descendants()`, `children()`,
`repeat()`). Adding a `recursive` parameter to a single function
duplicates that machinery and risks inconsistency with the rest of the
FHIRPath surface (none of the other FHIR-specific FHIRPath functions
take a recursion flag). Keeping the function shape minimal — input
collection × optional known-list — is easier for FHIRPath engines to
implement uniformly and easier for spec readers to reason about, while
still meeting the underlying need ("detect unknown modifier extensions
anywhere in this resource") via a documented composition. The
optional-`modifier` change is preserved because it is a genuine
expressivity gap that cannot be papered over with composition.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Mod" → declined: do not change the
signature of `checkModifiers()`. Instead, clarify in the function's spec
page that:

- An empty `modifier` list is already a legal call (`checkModifiers({})`),
  and is equivalent to "assert that there are no modifier extensions";
  fix any wording that suggests this is an error.
- Recursive checking is out of scope for this single function and should
  be expressed using standard FHIRPath traversal (`descendants()` /
  `repeat(children())`) by the invariant author.

Optionally add the two worked examples Brian requests, but framed against
the existing signature.

#### Justification

`checkModifiers()` is widely implemented by FHIRPath engines and
referenced from invariants in published IGs; changing its arity (even in
a backwards-compatible way) forces every FHIRPath implementation to ship
a coordinated update and risks divergent behavior in the field during the
rollout window. The "no list, just check there are none" use case is
already expressible as `checkModifiers({})`, and the recursive use case is
already expressible by composition, so the spec change adds surface area
without enabling anything that cannot be expressed today. A
documentation-only clarification achieves the same author-facing benefit
with no implementer cost.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of the three. The ticket actually
identifies two separate problems, and they have very different cost
profiles. Making `modifier` optional is a real expressivity gap — the
"detect any unknown modifier extension without a known list" pattern is
common, and the current behavior (error or awkward `{}` workaround) is
hostile to invariant authors; that change is small, backwards-compatible,
and worth shipping. Adding a `recursive` flag, on the other hand, is
solving a tree-traversal problem inside a function that should be doing
modifier-extension checking, when FHIRPath already has `descendants()`
and `repeat()` for exactly that purpose; bolting recursion onto one
function sets a precedent the rest of the FHIR-specific FHIRPath surface
does not follow and increases the per-engine implementation burden
without adding expressive power. Accepting the optional-parameter half
and documenting the recursive idiom (with a worked example, as Brian
asks) gives invariant authors the ergonomic win they want while keeping
the FHIRPath surface coherent. If the work group disagrees on the
recursion half, falling back to Disposition A is reasonable; Disposition
C should be reserved for the case where implementer coordination cost is
deemed prohibitive.
