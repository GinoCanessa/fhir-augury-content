## Summary

Rene Spronk reports that a hyperlink to the `resource` search-parameter-type
section of the FHIR Core specification — intended to point at
`https://build.fhir.org/search.html#resource` — does not work correctly.
Inspection of the ticket shows that the rendered anchor element targets
`https://build.fhir.org/search.html#resource)` (with a trailing closing
parenthesis included inside the URL). This is a classic Markdown/wiki
auto-link defect where a parenthesis surrounding the URL in source text was
absorbed into the link target, producing an invalid fragment identifier.
There are no cross-referenced Jira tickets, Zulip threads, GitHub items, or
prior community discussions associated with this issue; it is a single,
narrowly-scoped technical correction.

## Details

**Description:**

> Link to the 'resource' search parameter type (which refers to
> https://build.fhir.org/search.html#resource\) doesn't appear to work
> correctly.

The HTML emitted in the ticket body is:

```html
<a href="https://build.fhir.org/search.html#resource)" ...>
  https://build.fhir.org/search.html#resource)
</a>
```

i.e. the trailing `)` from the surrounding parenthetical text was captured
into the URL. The intended target is the `resource` row of the "Search
Parameter Types" table on `search.html` (anchor `#resource`).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation defect and fix the malformed hyperlink at its
source. Concretely:

1. Locate the page in the FHIR Core specification (almost certainly an
   authored page under `source/` in `HL7/fhir`) whose rendered output
   contains an anchor element whose `href` ends in
   `search.html#resource)`. The most efficient way is a repo-wide grep for
   the literal strings `search.html#resource)` and
   `(search.html#resource)` across `source/**/*.html` and `source/**/*.md`.
2. In the source, change the link target from `search.html#resource)` to
   `search.html#resource`, and ensure the surrounding closing parenthesis
   stays as visible prose **outside** the link (e.g., move the `)` past the
   closing `</a>` tag, or — if the source uses Markdown-style auto-linking
   — wrap the URL in angle brackets / backticks so the `)` is not absorbed).
3. While in the area, verify that the linked target — the `resource` row in
   the "Search Parameter Types" table on `search.html` — still exists and
   that its `id`/`name` anchor is `resource`. If it has been renamed,
   update the link target accordingly.
4. Also run a grep across `source/` for any other links of the form
   `search.html#<type>)` to catch sibling defects (`#string)`,
   `#token)`, `#reference)`, `#date)`, `#number)`, `#quantity)`,
   `#uri)`, `#composite)`, `#special)`).

#### Justification

This is the literal request, and it is a low-risk, high-clarity technical
correction. The defect is objectively a bug: the rendered URL is not a valid
fragment identifier and resolves to nothing on `search.html`. There is no
design trade-off, no community disagreement, and no implementer impact
beyond restoring the intended cross-reference. A repo-wide sweep for the
sibling pattern is cheap insurance against the same authoring mistake
having been made elsewhere on the same page or in nearby narrative.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the immediate broken link as in Disposition A, **and** address the root
cause by adjusting the authoring convention so the class of defect cannot
recur. Two complementary sub-actions:

1. Where the FHIR Core spec uses Markdown-style auto-links inside
   parenthetical prose (`(see https://example.org/x.html#frag)`), convert
   them to explicit `<a>` tags or `[label](url)` Markdown links so the
   surrounding `)` cannot be absorbed.
2. Add a lightweight QA check to the publisher / CI pipeline that flags
   any anchor element whose `href` contains an unbalanced `)`, `,`, `.`,
   or `;` immediately before the closing quote — these almost always
   indicate punctuation absorbed from prose. The check can be a single
   regex over the generated HTML in `publish/` (or an equivalent pre-render
   check over `source/*.html`).

#### Justification

The reported broken link is the visible symptom of an authoring pattern that
is easy to repeat: any FHIR contributor who writes
`(see https://build.fhir.org/foo.html#bar)` in Markdown-flavoured prose
risks the same defect. Catching it once in CI is far cheaper than
discovering it ticket-by-ticket. Trade-off vs. Disposition A: more work
(touching the publisher / QA harness, not just one page), and the QA rule
will need a small allow-list for legitimate cases — but the long-term
maintenance saving is real, and the change is purely additive (no breaking
behaviour for implementers).

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" / decline to fix. Rationale category: the broken
link is on `build.fhir.org` (the continuous-integration build of FHIR R6),
not on a published normative version, and the surrounding text still
conveys the intended target URL in plain text, so a reader can copy the
correct fragment manually.

#### Justification

A defensible-but-weak position: `build.fhir.org` is by definition a
work-in-progress build, the visible link text still includes the correct
URL string, and no implementer behaviour depends on a documentation
hyperlink. The workgroup could argue that scarce editor cycles are better
spent on substantive corrections. This justification is documented for
completeness; it would be unusual to invoke it for a defect this small and
this cheap to fix.

---

### Recommendation

**Recommended disposition:** A

This is a textbook Technical Correction: the defect is unambiguous, the
fix is a one-line edit in a single source page (plus a quick grep for
sibling cases), and there is no design judgement involved. Disposition B's
longer-term hardening is a good idea on its merits but is materially
out of scope for a ticket that simply reports one broken anchor — it
should be raised as its own infrastructure ticket so it can be sized and
prioritised independently rather than blocking a 30-second fix.
Disposition C is not credible for a defect this trivial: declining a
clearly-broken hyperlink in the published spec sends the wrong signal to
the community about the workgroup's willingness to absorb low-cost
maintenance reports. Apply Disposition A, optionally accompanied by a new
ticket capturing Disposition B's CI hardening idea.
