## Summary

FHIR-50575 is a low-priority Technical Correction filed against FHIR Core
by Juhi Patel. The ticket reports that an example using LOINC code
`10160-0` displays the code text as "History of medication", which does
not match LOINC's official Long Common Name for that code. The reporter
notes that other LOINC-coded examples in the specification do use the
official Long Common Name and asks that this example be brought into line
with the rest by changing the display text to "History of Medication use
Narrative". No related Jira tickets, GitHub items, or Confluence pages
were found by the cross-reference index for FHIR-50575; relevant context
exists only in adjacent Zulip discussions (in the IPS and IG creation
streams) where the same code/display pairing is used and the Long Common
Name "History of Medication use Narrative" is the form in active use.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIR Core example(s) that currently render LOINC `10160-0` as
"History of medication" so that the display text matches the official
LOINC Long Common Name "History of Medication use Narrative". This is a
display-text-only change to one or more example resources in `HL7/fhir`
(typically a `List` or `MedicationStatement` example, or a `Composition`
section coding). The LOINC code (`10160-0`) and code system
(`http://loinc.org`) are unchanged; only the `display` element of the
relevant `Coding` is corrected. No structural, cardinality, or binding
changes are involved.

#### Justification

Examples in the specification are expected to be self-consistent and to
reflect the official terminology. Adjacent FHIR community usage — the
IPS Medication Summary section, the AU Patient Summary, and IG-creation
guidance from Chris Moesel and Grahame Grieve in Zulip — uniformly uses
"History of Medication use Narrative" for `10160-0`. Bringing the FHIR
Core example into line with the LOINC Long Common Name and with how the
broader community profiles this code is exactly the kind of low-risk,
no-downside fix the "Technical Correction" issue type was created for,
and it removes a small but real source of confusion for implementers
copying example text verbatim.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the corrected display ("History of Medication use Narrative")
*and* take the opportunity to sweep all FHIR Core examples that
reference LOINC codes for stale or non-canonical display text — not just
`10160-0`. Concretely: (1) make the `10160-0` correction now in the
identified example(s); (2) open a follow-up housekeeping task (or fold
into the next publication's QA pass) to programmatically check every
`Coding.display` whose system is `http://loinc.org` against the current
LOINC Long Common Name, and file corrections in bulk for any other
drift discovered. Tooling-side, this is naturally a job for the IG
publisher's terminology validation step or a one-off script over
`source/**/*example*.{xml,json}`.

#### Justification

The reporter explicitly framed this as "unlike the other examples,"
implying that the rest of FHIR Core *does* track LOINC's Long Common
Name. If that assumption is correct, the sweep finds nothing more and
costs little. If it is not — i.e., other examples have also drifted —
catching them in one pass is much cheaper than receiving them as a
trickle of one-off Technical Correction tickets over the next several
ballot cycles. The marginal cost over Disposition A is small (one
script + a follow-up task), and it converts an ad-hoc fix into a
durable invariant about example quality.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as Not Persuasive on the grounds that `Coding.display`
is informative, not normative — implementers MUST use the LOINC code to
identify the concept and SHOULD treat the human-readable display as
secondary. The example is therefore not "wrong" in any conformance sense,
and the cost of churning published examples for a cosmetic display
change outweighs the benefit.

#### Justification

There is a defensible read of the FHIR spec under which `Coding.display`
need only be a reasonable label for the code, not the code system's
canonical name. Editors sometimes deliberately shorten displays in
examples to keep narrative readable. Declining preserves the editor's
discretion and avoids re-touching long-stable example fixtures.

---

### Recommendation

**Recommended disposition:** A

This is a textbook Technical Correction: a single, factual,
display-only mismatch with the code system's official Long Common Name,
with zero conformance impact, no breaking change, and corroborating
evidence from multiple Zulip threads (IPS, IG creation, AU Patient
Summary) that "History of Medication use Narrative" is the form the
broader community has standardised on. Disposition C reads the spec
correctly about `display` being informative but spends political
capital defending a stale example string that no implementer is asking
us to keep. Disposition B is attractive but is really a separate piece
of work — it should not block this ticket. Accept as requested now,
and if the sweep idea has support, raise it as its own follow-up.
