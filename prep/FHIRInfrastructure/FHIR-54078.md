## Summary

Brian Postlethwaite proposes that the FHIR-defined `%factory` type-factory
functionality on the FHIRPath page is **not mature enough to go normative**
and should be moved to an incubator IG. The functions were only `draft /
maturity 0` in R5 and have very limited implementation experience. The
ticket overlaps substantively with the constructor-syntax work tracked by
[FHIR-33044](https://jira.hl7.org/browse/FHIR-33044) ("Support constructor
syntax", Applied / Persuasive in the FHIRPath spec, ITS work group), which
introduces a CQL-style instance selector (`Coding { system: …, code: … }`)
into the base FHIRPath grammar. Brian's argument is that the new FHIRPath
`instance-selector` already covers `%factory.create` / `%factory.{primitive}`,
making large parts of `%factory` redundant.

In addition to the maturity/redundancy concern, the ticket bundles a long
list of editorial and design problems with the existing `%factory` text
(missing summary list, inconsistent example formatting versus
`%terminologies` / `%server`, confusing `{primitive}` notation, conflicting
statements about parameter optionality, weak example expressions, the
`%factory.Extension` example using wrong case, etc.). If the workgroup
elects to keep the functions in normative content, Brian proposes
**replacing the `%factory.xxx` style with plain fluent functions** —
`createCoding(system, code, display, version)`, `createCodableConcept(coding,
text)`, `createIdentifier(...)`, `createContactPoint(...)`,
`createQuantity(...)`, plus fluent forms `withExtension(url, value)` and
`withProperty(identifier, value)` — and adding a `toCoding(codedString)` /
`convertsToCoding(codedString)` pair patterned after Pathling and CQL.

The single related Zulip thread (`#fhirpath > Object creation`) confirms
that `%factory` has essentially one implementation (fhirpath.js), that ITS
voted in early 2026 to adopt the CQL-style instance selector in the base
FHIRPath grammar (without anonymous types), and that the community is
divided: Bryn Rhodes and Steve Munini favour the new instance-selector;
Grahame Grieve dislikes anonymous types and is suspicious of new syntax;
Ilya Beda, Yury Sedinkin and Vasilii Kupriakov see continuing value in
fluent `withProperty` / `withExtension` helpers because the instance
selector cannot easily mutate an existing instance.

>
> (FML also has a similar capability)
>
> Other general notes:
> - Include some guidance in the header for this section to indicate what
>   contexts these functions would be used in — such as converting data during
>   SDC pre-population or extraction, …?
> - Remove the generic `%factory.primitive` and `%factory.create` — now
>   directly covered by the new fhirpath functionality (as noted in the intro
>   to the section — which can now be removed).
> - Missing summary list of functions (as is in other factories: `%terminology`
>   and `%server`).
> - Inconsistent examples formatting compared to all other functions in
>   fhirpath.
> - Additional examples that would be more typical examples of usage; none of
>   the existing examples make sense as valid uses anywhere a fhirpath
>   expression would be found — they don't refer to data from an input
>   resource, which masks how awkward the functions are in practice.
> - The functions are not "fluent" in style as most (almost all) fhirpath
>   functions are. The `%factory` prefix is not required.
> - `%factory.Extension` example invalid — wrong case for function name in the
>   example.
> - Inconsistent usage / definition style with fluent usage of `%terminologies`
>   and `%server`.
> - `%factory.{primitive}` formatting weird and not explained (outside the
>   example). Should this work for all primitive types? What about
>   `base64Binary`?
>   - The `{primitive}` notation can be confused with the empty-collection
>     syntax `{}`. Suggest using guillemets in documentation only:
>     `%factory.«primitive»(value, extensions) : «primitive»`. Actual usage
>     would still be `%factory.code('final')` or `%factory.string('test')`.
> - The two statements about parameter optionality are confusing:
>   - "For the specific type constructors, all the parameters are optional"
>   - "For the general type constructors, all the parameters are mandatory"
>   - Needs clearer separation. Be explicit per function which parameters are
>     required versus mandatory using the standard `[]` notation. Add a note
>     that `{}` may be used as an empty placeholder when supplying a parameter
>     after omitted optionals.
>
> The factory functions should just be regular functions; the `%factory.xxx`
> format is not required. Suggested replacements:
> - `create(type: identifier): Any` *(suggest remove — use new fhirpath
>   syntax)*
> - `create(type: identifier, [value: {primitive | string}]): Any` *(suggest
>   remove — use new fhirpath syntax)*
> - `createCoding(sys, code, display, version)`
> - `createCodableConcept(coding, text)` (can embed `createCoding()` for the
>   coding parameter)
> - `createIdentifier(system, value, [use], [type])`
> - `createContactPoint(system, value, [use])`
> - `createQuantity(system, code, value, unit)`
> - `createHumanName(...)` *(do we really need this one? The generic one is
>   probably clearer; too many parameters.)*
> - `createAddress(...)` *(same comment.)*
> - `withExtension(url, value)` — fluent form returning a copy of the input
>   object with the extension added.
> - `withProperty(identifier, value)` — fluent form returning a copy of the
>   input object with the property added (need to decide collection
>   semantics: append vs replace).
> - `toCoding(codedString string): FHIR.Coding` — parses a pipe-separated
>   coding string (system|version|code) into a Coding (per
>   https://pathling.csiro.au/docs/fhirpath#coding; thanks John Grimes).
> - `convertsToCoding(codedString): System.Boolean` — companion to
>   `toCoding`.

## Related Jira Tickets

### [FHIR-33044: Support constructor syntax](https://jira.hl7.org/browse/FHIR-33044)

Reported by Bryn Rhodes against the FHIRPath specification (work group:
Implementable Technology Specifications). Status **Applied / Persuasive**.
The ticket introduces the CQL-style instance-selector grammar
(`namedTypeSpecifier '{' (':' | (instanceElementSelector (',' instanceElementSelector)*)) '}'`)
into FHIRPath itself — i.e. `Coding { system: …, code: … }` — so that
arbitrary objects can be constructed in FHIRPath without needing
type-specific functions. This is the core "duplicates `%factory`" claim
behind FHIR-54078: if FHIR-33044 is published, `%factory.create` and
`%factory.{primitive}` become redundant. The ITS-approved wording is in
the `BP-FHIR-44774` working branch referenced in FHIR-54078's
description.

Users involved: Bryn Rhodes (reporter), Brian Postlethwaite, Steve
Munini, Paul Lynch, Bryn Rhodes, Grahame Grieve, Ilya Beda, Nikolai
Ryzhikov, Yury Sedinkin, Vasilii Kupriakov (via the Zulip discussion
that drove the change).

## Related Zulip Discussions

### [#fhirpath > Object creation](https://chat.fhir.org/#narrow/stream/fhirpath/topic/Object%20creation/near/569124392)

Long-running thread (Nov 2025 – Feb 2026) covering both the FHIRPath
constructor syntax (FHIR-33044) and the FHIR-level `%factory` tweaks
(FHIR-54078). Brian Postlethwaite opens by noting that creation of
arbitrary FHIR objects is currently awkward and contrasts three styles:
the CQL-style instance selector `Coding { system: …, code: …, display: … }`,
the FHIR `%factory.Coding(system, code, display)` factory call, and a
proposed fluent form `create(Coding).withProperty(system, …).withProperty(code, …)`.
Steve Munini, Bryn Rhodes and Ilya Beda all favour the CQL-style
instance selector (Bryn notes every CQL engine already supports it).
Grahame Grieve pushes back: he is "not a fan of anonymous types" and
"suspicious of introducing new syntax at this point", though he
acknowledges `.withProperty()` is an improvement over the factory
approach. Paul Lynch (fhirpath.js, the only `%factory` implementer)
finds the new syntax less FHIRPath-like but more concise where no
type-specific factory exists. Nikolai Ryzhikov suggests a Postgres-style
`build_object(Coding, 'system', …, 'code', …)` variadic form. In
January 2026 Brian reports that ITS voted to approve the CQL-style
constructor (without anonymous types) and links the wording in the
`BP-FHIR-44774` branch — this is the change that motivates FHIR-54078's
"move `%factory` to incubator" recommendation. The closing sub-thread
(Vasilii Kupriakov, Yury Sedinkin, Brian, Bryn, Feb 2026) explores
whether the instance selector can express FHIR primitive `_value`
extensions and concludes a fluent `withProperty` / `withExtension`
helper still has independent value because the instance selector
constructs new objects rather than mutating existing ones.

Participants: Brian Postlethwaite, Steve Munini, Bryn Rhodes, Ilya
Beda, Paul Lynch, Grahame Grieve, Nikolai Ryzhikov, Yury Sedinkin,
Vasilii Kupriakov.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the entire `%factory` section out of the normative FHIR Core
specification and into a new (or existing) incubator IG. Concretely:

- Delete the `%factory` subsection from `fhirpath.html` in the FHIR Core
  build (`HL7/fhir`, FHIRPath chapter).
- Remove `%factory` from any normative conformance language and the
  FHIRPath Environment Variables registry entries that ship with FHIR
  Core.
- Stand up an incubator IG (e.g. `HL7/fhir-tools-ig` or a new
  `HL7/fhir-factory-ig`) containing the current `%factory.create`,
  `%factory.{primitive}`, `%factory.Coding`, `%factory.CodeableConcept`,
  `%factory.Identifier`, `%factory.ContactPoint`, `%factory.Quantity`,
  `%factory.HumanName`, `%factory.Address`, `%factory.Extension`,
  `%factory.withProperty`, `%factory.withExtension` definitions, marked
  draft / FMM 0.
- Add a forward reference from the FHIRPath chapter to the incubator
  IG, plus a note that arbitrary instance construction is now handled
  by the FHIRPath instance selector approved under FHIR-33044.

#### Justification

`%factory` was draft (FMM 0) in R5, has only one known implementation
(fhirpath.js, per Paul Lynch on Zulip), and overlaps substantially with
the CQL-style instance selector that ITS approved into the base
FHIRPath grammar (FHIR-33044 — Applied / Persuasive). Promoting
`%factory` to normative now would lock in syntax that the community has
not yet agreed on (Grahame is opposed, the fluent `.withProperty` form
has not been voted on, parameter-order conventions for `createHumanName`
/ `createAddress` are explicitly questioned in the ticket) and that
duplicates a separately normative mechanism. The HL7 normative bar
requires demonstrated implementation experience; one implementer plus
extensive editorial issues (inconsistent examples, miscased
`%factory.Extension`, contradictory parameter-optionality statements,
ambiguous `{primitive}` notation, no summary list) does not clear that
bar. Incubation lets the community converge with FHIR-33044 before
committing to backwards-incompatible normative content.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `%factory` in the FHIR Core specification but **demote it from
normative back to STU/draft** and apply Brian's editorial fixes in
place, rather than physically relocating it to an incubator IG.
Concretely:

- Mark the `%factory` section as STU (or trial-use / FMM 1) in the
  FHIRPath chapter of `HL7/fhir`.
- Remove `%factory.create` and `%factory.{primitive}` outright, with a
  pointer to the FHIR-33044 instance selector for those use cases.
- Refactor the remaining functions into a fluent, non-`%factory` style:
  `createCoding(system, code, display, version)`,
  `createCodableConcept(coding, text)`,
  `createIdentifier(system, value, [use], [type])`,
  `createContactPoint(system, value, [use])`,
  `createQuantity(system, code, value, unit)`, plus fluent
  `withExtension(url, value)` / `withProperty(identifier, value)` and
  the `toCoding(codedString)` / `convertsToCoding(codedString)` pair
  modelled on Pathling.
- Drop `createHumanName` / `createAddress` (Brian's own "do we really
  need this one?" call) in favour of `withProperty` chains.
- Fix the editorial issues called out in the ticket: add a summary
  table aligned with `%terminologies` / `%server`, replace `{primitive}`
  with guillemets in documentation, separate the optional/mandatory
  parameter rules per-function, correct the `%factory.Extension` case,
  rewrite the examples against realistic resource-driven inputs (SDC
  pre-population / extraction is the example Brian suggests).

#### Justification

This addresses the substance of Brian's complaint (immature for
normative, redundant with FHIR-33044, editorially weak, awkward syntax)
without the disruption of carving the content out into a separate IG
and rewriting cross-references. STU status preserves the existing
implementer's (`fhirpath.js`) investment, signals that breaking changes
are still possible, and lets fluent `withProperty` / `withExtension`
mature alongside the FHIRPath instance selector — which Yury and
Vasilii pointed out on Zulip cannot easily mutate existing instances,
so the helpers retain genuine value. The risk Brian highlights —
"limited implementation experience" — is captured by the FMM/STU
classification system, which is exactly what it exists for.

---

### Disposition C: Decline

#### Proposal

Reject the request. Keep `%factory` in the normative FHIR Core
specification with its current `%factory.xxx` calling convention,
applying only the uncontroversial editorial corrections (summary list,
miscased `%factory.Extension` example, clarified parameter-optionality
text, footnote on `{primitive}` notation).

#### Justification

A defensible argument for declining is that `%factory` and the
FHIR-33044 instance selector occupy different layers: FHIR-33044 is
generic FHIRPath grammar (any model), whereas `%factory` is a
FHIR-specific environment variable optimised for the most common FHIR
construction shapes (Coding, CodeableConcept, Identifier, …) and
already has a shipping implementation. Removing it now would create a
gap during the period when only fhirpath.js supports the instance
selector against FHIR models. However, this rationale is weak: the
ticket reporter is one of the principal FHIR Infrastructure
implementers, the Zulip discussion did not produce a constituency that
defends `%factory.xxx` on its merits, and the editorial defects (case
errors, contradictory parameter rules) themselves indicate the content
has not received normative-grade review. Declining is therefore the
weakest of the three options and is offered primarily as a baseline
against which Dispositions A and B should be weighed.

---

### Recommendation

**Recommended disposition:** B

Disposition B (demote in place to STU and refactor) gives Brian
essentially everything he is asking for — the normative claim is
withdrawn, `%factory.create` / `%factory.primitive` are removed in
favour of FHIR-33044, the functions are refactored into the fluent
style he proposes, and the editorial defects are fixed — without the
overhead of standing up a new incubator IG and migrating
cross-references. The Zulip thread is consistent with this: the
community wants the new instance selector for object construction
(Bryn, Steve, Ilya), and several participants (Yury, Vasilii) have
identified continuing value in fluent `withProperty` / `withExtension`
helpers that the instance selector does not cover. Pushing the content
to an incubator IG (Disposition A) is justifiable but would split the
fluent helpers from the FHIRPath chapter where implementers actually
look for them, and would slow down convergence with FHIR-33044.
Disposition C — keeping `%factory` as-is in normative — is not
defensible given one implementer, FMM 0 in R5, and the editorial
defects the ticket itself enumerates. The workgroup should treat
Disposition B's refactor list (drop `%factory.create` /
`%factory.primitive`, drop `createHumanName` / `createAddress`, adopt
fluent `createCoding` / `createCodableConcept` / `createIdentifier` /
`createContactPoint` / `createQuantity` / `withExtension` /
`withProperty`, add `toCoding` / `convertsToCoding`) as the starting
point for the actual change, and coordinate with ITS so the FHIR Core
text references the FHIR-33044 instance selector consistently.
