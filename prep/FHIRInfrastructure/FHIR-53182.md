## Summary

John Moehrke is questioning a "Note" that appears in the FHIR Core
specification's Security Labels section (2.1.27.5.3.9.3) on
`Resource.meta.security`. The note currently asserts that it is valid to
create two security labels with the same `system`+`code` but different
`display` or `version`, and that `display` and `version` are not taken
into account when evaluating security labels. The reporter argues this
note is historical — it dates back to the change made under
[FHIR-20632](https://jira.hl7.org/browse/FHIR-20632) (when `.security`
was migrated from `uri` to `Coding`) — and that it is now problematic
given increasingly strict code-system version processing in the
ecosystem, and uncertainty about whether the validator actually permits
the described deviations (for example, varying `display` for the same
`system`+`code`). The ask is to remove (or at minimum rework) the
special-case rule so that `.meta.security` behaves like any other
`Coding` and inherits the general rules for code-system version and
display handling.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the highlighted "Note" paragraph from section 2.1.27.5.3.9.3
"Security Labels" in FHIR Core. Specifically, delete the sentence pair:

> Note, it is valid to create two security labels with the same
> (`system`+`code`) but different `.display` or `.version`.
> Furthermore, `.display` or `.version` are not taken into account when
> evaluating security labels.

After removal, `Resource.meta.security` would rely on the general rules
that apply to any `Coding`: `system`+`code` is the canonical identity
of the concept, `display` is a human-readable rendering governed by the
usual code-system display rules, and `version` is governed by the
normal code-system versioning semantics. No change is made to the
structure definition of `Resource.meta.security` itself; the edit is
narrative-only.

#### Justification

The note encodes a special-case evaluation rule for `.meta.security`
that diverges from the general `Coding` semantics used everywhere else
in FHIR. As tooling (validators, terminology servers, profile
constraints) has become more rigorous about code-system version
handling and `display` validation, that divergence is increasingly at
odds with how implementers and validators actually process `Coding`
values. The reporter — a long-standing contributor on FHIR security —
notes that it is not clear the reference validator even implements the
described leniency. Removing the note aligns `.meta.security` with the
rest of FHIR and eliminates a rule that is likely neither enforced nor
depended upon.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a note at section 2.1.27.5.3.9.3, but rewrite it so that it (a)
does not create special evaluation semantics for `.meta.security` and
(b) instead gives implementers practical guidance consistent with
general `Coding` rules. Proposed replacement wording:

> Note: security labels are `Coding` values and follow the general
> rules for `Coding`. The identity of a security label is given by its
> `system` and `code`; `display` is informative and is not used for
> matching, and `version` is interpreted according to the rules of the
> referenced code system. Access-control engines comparing security
> labels SHOULD compare on `system` and `code`, and SHOULD consider
> `version` only when the code system's versioning rules make it
> semantically significant.

This keeps a purpose-built hint for implementers of access-control
logic (which was arguably the original intent of the note) without
asserting a special FHIR-wide evaluation rule for this element.

#### Justification

The original note was presumably written to reassure implementers of
security-policy engines that they did not need to worry about
`display` drift or minor `version` differences when matching labels
against policies. That reassurance is still useful, but it can be
delivered as guidance ("SHOULD compare on `system` and `code`") rather
than as a normative statement that changes how `Coding` equality is
defined for this path. This preserves continuity for readers who
currently rely on the note while bringing it into line with how the
rest of the spec (and validators) treat `Coding`.

---

### Disposition C: Decline

#### Proposal

Retain the note unchanged. Close the ticket as "not persuasive",
noting that the special-case rule has been in place since
FHIR-20632, is load-bearing for security-policy implementations that
need to tolerate minor `display` and `version` drift across systems,
and that no concrete interoperability break has been reported.

#### Justification

Declining is defensible if the workgroup judges that the note
represents intentional, long-standing policy about how access-control
engines should compare labels, and that changing it risks breaking
those engines' conformance claims. The reporter acknowledges
uncertainty about validator behavior rather than citing a concrete
failure; absent a concrete interoperability report, removing
established normative-sounding text could itself cause confusion.

---

### Recommendation

**Recommended disposition:** B

The reporter's core concern is sound: the current note asserts a
special evaluation rule for `.meta.security` that diverges from the
general `Coding` semantics and sits uneasily with modern validators and
terminology servers. That argues against Disposition C. However,
simply deleting the note (Disposition A) throws away genuinely useful
guidance for implementers of access-control engines, who do in practice
need to know that they should match on `system`+`code` and treat
`display` as informative. Disposition B preserves that implementer
guidance while removing the problematic "special rule" framing and
aligning the behavior with the rest of FHIR. It is also the
lowest-risk option from a backward-compatibility perspective: anyone
currently relying on the note's intent (match on `system`+`code`, don't
rely on `display`) will find their behavior still endorsed; anyone
relying on the divergent special-case semantics likely has no
validator or tooling enforcing them today anyway. The workgroup should
treat the exact replacement wording as open for refinement during the
discussion.
