## Summary

The submitter (Emma Jones) reports that on the Goal resource page, the
common search parameter `identifier` is rendered with an extra "Goal"
hyperlink preceding its description text, and asks for the extra link
to be removed. The attached screenshot shows the unwanted "Goal" link
appearing immediately before the identifier search parameter's
definition. The ticket originated under Patient Care but discussion on
Zulip (`#committers > Tooling issue?`) concluded that this is a
publisher tooling rendering behaviour that affects all common search
parameters (e.g. `code`, `identifier`, `patient`, `date`, `encounter`),
not just Goal — common search parameters are rendered with a leading
resource-name hyperlink before the parameter's description, while
non-common parameters are not. The ticket has therefore been routed to
FHIR Infrastructure with the `FHIR-Tools-Core` grouping. The reporter
has not yet elaborated on *why* the link should be removed; Grahame
Grieve has acknowledged that this is the current tooling behaviour but
has explicitly asked "Why would it be removed?", and Michelle Miller
(2025-01-15) followed up on the ticket asking the submitter to justify
the request. A sibling ticket [FHIR-46577](https://jira.hl7.org/browse/FHIR-46577)
was raised at the same time about the same rendering behaviour and
should be considered together with this one.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the publisher tooling that renders the per-resource Search
Parameters table so that common search parameters (those inherited
from the common-search-parameter set: `_id`, `identifier`, `patient`,
`code`, `date`, `encounter`, etc.) no longer emit the leading
resource-name hyperlink before the parameter's description on the
resource page (e.g. on `goal.html`'s Search Parameters section). The
fix is a templating change in the FHIR build/IG-publisher rendering
of common search parameters; it would apply uniformly to all
resources, not just Goal — so this single ticket would close
FHIR-46576, FHIR-46577, and any sibling tickets reporting the same
artefact.

#### Justification

Removing the extra link gives a consistent rendering between common
and non-common search parameters on the resource page (today only
common SPs get the leading resource link, which is what the submitter
flagged as visually surprising). The change is purely cosmetic and
limited to publisher templates, with no impact on the normative
content of search parameter definitions. The Zulip thread confirmed
that the link is produced by the publisher rather than authored
content, so the fix is feasible in tooling.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the leading resource-name link but make its purpose explicit and
consistent: render it on **all** search parameters in the table (not
just the common ones) as a standard "defined on Goal" / "inherited
from Resource" affordance, styled as a small badge or prefix rather
than as part of the prose. Concretely, change the publisher template
so every row in the Search Parameters table starts with the same
"defined on \<Resource\>" indicator, then the parameter description,
so the row layout is uniform.

#### Justification

The current rendering already conveys useful information — that a
search parameter is inherited from the common set and is anchored on
this resource — and Grahame's comment ("Why would it be removed?")
suggests committer reluctance to drop information rather than
normalise it. Making the affordance uniform across all rows addresses
the submitter's underlying complaint (visual inconsistency) without
losing the provenance signal, and it lines up with how Search
Parameters are documented elsewhere in the spec (where the resource
on which the SP is defined is explicit). Trade-off vs. Disposition A:
slightly more tooling work, and a small risk of cluttering rows for
SPs that are obviously local to the resource.

---

### Disposition C: Decline

#### Proposal

Decline the request as **Not Persuasive** (or **Works as Designed**):
the leading resource-name hyperlink before common search parameter
descriptions is the intentional, long-standing rendering behaviour
applied uniformly to every resource page in FHIR Core. Without a
specific use-case harm (broken link, accessibility issue, ambiguity
in interpretation), the cosmetic preference of a single submitter
does not warrant a publisher change that would touch every resource
page in the spec.

#### Justification

The Zulip discussion is the closest the community has come to
adjudicating this and the prevailing position is "this is how it
works"; Grahame Grieve explicitly asked why it would be removed and
Michelle Miller has asked the submitter on the JIRA ticket itself
(2025-01-15) to elaborate, which has not yet happened. Per FHIR-I
practice, a tooling/rendering change that propagates to every
resource page should be backed by a concrete problem statement, not
just a visual preference. If the submitter does not provide a
rationale, declining is the defensible default. The same disposition
should be applied to the sibling ticket FHIR-46577.

---

### Recommendation

**Recommended disposition:** C

The submitter has not (as of 2025-01-15) supplied a rationale for the
removal, the cross-referenced Zulip discussion explicitly questions
the request ("Why would it be removed?"), and the rendering is
applied uniformly to every resource page for every common search
parameter — so any change is necessarily spec-wide tooling work.
Without a use-case-grounded justification, FHIR-I should decline (or
at minimum hold the ticket pending a response from Emma Jones), and
should treat FHIR-46576 and FHIR-46577 as a single decision. If the
submitter does come back with a concrete usability/accessibility
argument, Disposition B (normalise the affordance across all rows) is
a better path than Disposition A, because it addresses the
inconsistency the submitter likely actually saw without throwing away
the provenance information committers consider useful.
