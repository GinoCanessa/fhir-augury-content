## Summary

This ticket reports documentation/specification defects in the description of
the `evaluate()` target transform for `StructureMap` (FHIR Mapping Language).
Specifically:

1. The `transform` table on the StructureMap page lists `evaluate` as taking a
   single `resource` parameter, but the surrounding prose describes **two**
   parameters (a context resource and a FHIRPath expression). The two
   descriptions are inconsistent, and the parameter table is the one that's
   wrong (engines and existing FML usage all expect two parameters).
2. The semantics of `$this` inside the supplied FHIRPath expression are not
   defined. This is particularly ambiguous when a rule has more than one input
   source — the spec does not say which source `$this` binds to.
3. The type of the FHIRPath expression parameter is not stated. It is a
   plain `string` containing FHIRPath, *not* a parenthesised expression
   `(...)` as is used elsewhere in the FML concrete syntax. The spec text
   should make this explicit while still keeping the existing description of
   the `(...)` shorthand that the FML compiler rewrites into an `evaluate`
   call.

The ticket originates from a recent Zulip thread in **#FHIR Mapping
Language > evaluate transform?** where Jean Duteau (working on the
HL7/fhir-omop-ig Patient→Person map) hit `Transform evaluate takes 1
parameter(s) but 2 were found` while writing the canonical two-arg form, and
where the thread participants (Brian Postlethwaite, Grahame Grieve)
confirmed the spec text needed to be tightened. There are no linked or
cross-referenced Jira tickets, no GitHub items, and no related repositories
in the FHIR Augury index.

`evaluate` call). After several iterations, the working form turned out to
be `tgt.year_of_birth = (birthDate.toString().substring(0,4))` once the

runtime even though the publisher complained about it. Brian later
confirmed in fhirpath-lab that the form was correct and worked across all
three engines, then logged this ticket to fix the spec wording. The thread
also drifted into separate but related FML topics (publisher support for

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the StructureMap specification page (`structuremap.html` /
`mapping.html`) to fix the description of the `evaluate` target transform:

1. **Parameter table:** change the row for `evaluate` from "1 parameter
   (resource)" to two parameters:
   - `context` — a node/resource in scope (the value the FHIRPath is
     evaluated against), and
   - `expression` — a `string` containing a FHIRPath expression.
2. **`$this` semantics:** add an explicit sentence stating that, inside the
   supplied FHIRPath expression, `$this` is bound to the value passed as
   the first parameter (the `context`). When the enclosing rule has more
   than one input source, the author must pass the desired source
   explicitly as the first argument; `$this` is not implicitly the rule
   itself.
3. **Parameter type:** state that the second parameter is of FHIR type
   `string` whose value is a FHIRPath expression, **not** a
   parenthesised expression literal as appears elsewhere in the FML
   concrete syntax. Keep the existing paragraph that documents the
   `(...)` concrete-shorthand the FML compiler rewrites into an
   `evaluate` invocation, but mark it clearly as a shorthand for the
   string-valued form.

No structural change to the `StructureMap` resource is needed — both
`StructureMap.group.rule.target.parameter` and the `evaluate` transform
code already accommodate the two-parameter form; only the prose and the
transform-parameters table are wrong.

#### Justification

The reporter is the maintainer who routinely answers FML questions on
Zulip, and the originating thread shows real-world author confusion caused
directly by the inconsistency: the table says one parameter, the text and
all working engines expect two. Tightening the prose costs nothing in the
normative model and unblocks IG authors (e.g. fhir-omop-ig) who currently
have to reverse-engineer the correct form by trial and error. Defining
`$this` explicitly closes a real ambiguity for multi-source rules; today
the answer is "whatever the engine happens to pick", which is exactly the
sort of under-specification that produces inter-engine divergence.

---

### Disposition B: Alternative Approach

#### Proposal

Same documentation fixes as Disposition A, but additionally:

- Promote the `evaluate` transform to a small dedicated subsection on the
  StructureMap page (parallel to other transforms that have their own
  detailed sub-page in the FHIR Mapping Language documentation), with:
  - a formal parameter signature block,
  - a worked example showing both the `evaluate(context, 'expression')`
    form and the `(expression)` concrete-shorthand,
  - a worked example with multiple input sources to make the `$this`
    binding unambiguous, and
- Add a non-normative note saying that publishers/validators MUST accept
  the two-argument form and MUST NOT continue to emit
  "Transform evaluate takes 1 parameter(s) but 2 were found", which is the
  concrete defect that triggered this ticket.

#### Justification

The Zulip thread shows that the confusion is not purely textual — the
publisher's FML parser actively rejects the documented form, and authors
need an authoritative, example-driven section they can point implementers
at. A dedicated subsection plus an explicit conformance note pre-empts
both author confusion and validator divergence, at the cost of a slightly
larger spec edit and the small risk of being read as a normative change to
the transform itself (which it isn't — the underlying behaviour is
unchanged).

---

### Disposition C: Decline

#### Proposal

Decline as already addressed: the FML compiler already accepts the
two-argument form in practice and the parenthesised shorthand exists for
authors who find the function form awkward. Leave the spec text as-is and
direct readers to the existing `(...)` shorthand and the FML examples.

#### Justification

Defensible only if the workgroup judges that the existing prose plus the
shorthand together are sufficient, and that the parameter table is a
known-and-tolerable simplification. This is weak: the originating Zulip
thread is concrete evidence that competent users *do* read the table,
*do* try the documented form, and *do* get blocked. Recommended only as
an explicit "no spec edit this cycle" punt.

---

### Recommendation

**Recommended disposition:** A

The ticket is a narrowly-scoped documentation defect raised by the FML
maintainer himself, and the fix has no impact on the StructureMap resource
shape, the FML grammar, or any conformance footprint — it just makes the
spec match the engines and resolves a genuine ambiguity around `$this`
under multi-source rules. Disposition A delivers exactly what's asked for
with minimal risk; Disposition B's extra worked-example subsection is a
nice-to-have that can be folded into the same edit if reviewers want it,
but should not gate accepting the core fix. Disposition C is hard to
defend given a fresh Zulip thread showing real user friction caused by
the current wording.
