## Summary

When a user navigates to a resource's "Search Parameters" tab on the FHIR
Core build site (e.g., `build.fhir.org/<resource>-search.html`), the
breadcrumb / top menu displays "Mappings" rather than "Search
Parameters". The reporter (Ana Kostadinovska) identified the issue on a
single resource's search page; in triage, jwhitby confirmed it
reproduces across multiple resources — at minimum ImagingStudy,
ImagingSelection, Observation, and Patient — and re-assigned the ticket
from the originating work group to FHIR Infrastructure on the basis
that this is a generic publishing-template bug, not a
resource-specific authoring error. No related Jira tickets, Zulip
threads, or GitHub items were surfaced through cross-references or
search.

## Proposed Dispositions

> Note: no GitHub cross-references were surfaced for this ticket, so no
> `repo-analysis` briefing is loaded and no specific file path is
> named. Dispositions describe the change in terms of the FHIR Core
> publishing template/page-header generator that produces
> `<resource>-search.html` on `build.fhir.org`; the implementer will
> need to identify the exact template asset in the FHIR Core publishing
> tooling.

### Disposition A: Accept As Requested

#### Proposal

Fix the FHIR Core publishing template so that the top menu /
breadcrumb on a resource's Search Parameters page (`<resource>-search.html`)
reads "Search Parameters" instead of "Mappings". Concretely: when the
page-header / tab-bar component is rendered for the search-parameters
tab, the active tab label (and the breadcrumb segment derived from it)
must come from the search-parameters tab definition rather than
falling through to the "Mappings" label. Apply the fix once in the
shared template so that all `*-search.html` pages — including
`imagingstudy-search.html`, `imagingselection-search.html`,
`observation-search.html`, `patient-search.html`, and every other
resource's search page — render correctly. Add a regression check (or
a build-time spot-check on a couple of resources) so the breadcrumb
label matches the active tab.

#### Justification

This is a Technical Correction with an obvious user-visible defect: a
mislabeled tab/breadcrumb misleads readers about where they are in
the spec and undermines confidence in the publication. The reporter
and the FHIR-I triage comment both confirm the bug reproduces across
multiple unrelated resources, so the cause is in the shared template
rather than in any one resource's authored content. Fixing it at the
template level is the smallest change with the broadest correctness
benefit, costs no spec semantics, has no backward-compatibility
implications (it is a label string in generated HTML), and aligns
with how other resource-tab pages (e.g., the actual mappings tab)
are already labelled.

---

### Disposition B: Alternative Approach

#### Proposal

Same root fix as Disposition A (correct the template so the
breadcrumb reflects the active tab), **plus** generalise the fix:
audit the page-header generator for every resource sub-page tab
(`*-definitions.html`, `*-examples.html`, `*-mappings.html`,
`*-profiles.html`, `*-search.html`, etc.) and drive the breadcrumb
label from a single source of truth — the tab's own
`title`/`label` field — rather than letting it default to a
sibling tab's label. Add a publication-time validation that asserts,
for every generated `<resource>-<tab>.html`, the breadcrumb segment
equals the tab's declared title.

#### Justification

The reported defect is symptomatic of a template that treats the
breadcrumb label as decoupled from the tab definition. A point-fix
(Disposition A) eliminates the visible "Mappings ↔ Search Parameters"
swap but leaves the underlying coupling in place, so a similar
mismatch can recur on any other tab the next time the template is
edited. Driving the label from the tab definition, plus a cheap
build-time assertion, prevents an entire class of regressions for a
modest one-time investment in the publishing tooling. The trade-off
versus A is scope: this disposition touches more of the template /
generator and warrants a slightly broader review by FHIR-I before
merging.

---

### Disposition C: Decline

#### Proposal

Close as Not Persuasive / No Action — treat the breadcrumb label as
cosmetic and leave the template as-is, on the basis that the active
tab is already highlighted in the tab strip and the page URL
unambiguously identifies the current sub-page (e.g.,
`observation-search.html`).

#### Justification

The functional content of each search-parameters page is correct;
only the breadcrumb / top-menu label is wrong. A reader can still
reach and read the search parameters, and the URL plus the active
tab indicator both convey the correct location. In a strict
"semantic content vs. presentation" framing, this could be deprioritised
relative to spec-bearing tickets.

---

### Recommendation

**Recommended disposition:** A

This is a clear, low-risk Technical Correction with a confirmed
template-level cause and a known reproduction across at least four
resources. The fix is a label/data-flow correction in the FHIR Core
publishing template, with no spec-semantics impact and no backward-
compatibility cost. Disposition C understates the harm: a wrong
breadcrumb on multiple resource pages erodes reader trust in the
published spec and is exactly the kind of polish a Technical
Correction exists to address. Disposition B is attractive and
arguably the "right" engineering answer, but it expands scope into
the publishing tooling and should be tracked as a follow-up
hardening item rather than gating this fix; FHIR-I can adopt the
broader audit if the implementer reports that the underlying
template makes a one-line fix awkward. Start with A, and if the
implementer finds the same coupling elsewhere while making the fix,
escalate in-flight to B.
