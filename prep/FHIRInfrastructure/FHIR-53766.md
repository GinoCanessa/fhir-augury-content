## Summary

FHIR-53766 reports a residual editorial inconsistency left over from
[FHIR-48492](https://jira.hl7.org/browse/FHIR-48492) ("Un-deprecate
jurisdiction element in various resources", Applied / Persuasive with
Modification). FHIR-48492 removed the deprecation flag from the
`jurisdiction` element on the FHIR-I conformance resources (and updated
its definition), but the element's `comment` field still contains a
"DEPRECATION NOTE" paragraph telling implementers to migrate to
`useContext` with the `jurisdiction` usage-context code. James Jahns
points out that, since the element is no longer deprecated, this comment
is contradictory and should be removed or rephrased. He notes that the
element's Description has already been updated to explain when to use
`jurisdiction` vs. `useContext`, so the deprecation paragraph is
redundant as well as misleading. The fix is a narrow technical
correction to the element comment text on every `*.jurisdiction` element
in scope.

## Details

**Description:**

[FHIR-48492](https://jira.hl7.org/browse/FHIR-48492) removed the
deprecation flag from the "jurisdiction" element of conformance
resources.

However, these elements still have a "DEPRECATION NOTE" in their
comment field that wasn't removed. For example:
https://build.fhir.org/operationdefinition-definitions.html#OperationDefinition.jurisdiction

> DEPRECATION NOTE: For consistency, implementations are encouraged to
> migrate to using the new 'jurisdiction' code in the useContext
> element. (I.e. useContext.code indicating
> http://terminology.hl7.org/CodeSystem/usage-context-type#jurisdiction
> and useContext.valueCodeableConcept indicating the jurisdiction.)

This sentence should be removed or rephrased, since these elements are
no longer deprecated. Note that these elements already have an updated
Description to indicate when to use this element vs. useContext.

## Linked Jira Tickets

### [FHIR-48492: Un-deprecate jurisdiction element in various resources](https://jira.hl7.org/browse/FHIR-48492)

The parent ticket that FHIR-53766 directly follows up on. FHIR-48492
(Applied, Persuasive with Modification, voted at the May 2025 WGM) ruled
that the `jurisdiction` element on the FHIR-I conformance resources
should be undeprecated, that its definition should change to "A legal
or geographic region in which the authority that maintains the resource
is operating…", and that all jurisdiction value-set bindings should be
switched to `http://terminology.hl7.org/ValueSet/jurisdiction`. The
resolution explicitly says "Undeprecate the jurisdiction element in all
locations it appears **and remove any deprecation notes**" — exactly
the cleanup that FHIR-53766 says was missed for the element-level
comment text. Work-group scoping discussion in the comments confirms
that FHIR-I owns CanonicalResource, CapabilityStatement, GraphDefinition,
ImplementationGuide, OperationDefinition, SearchParameter,
StructureDefinition and StructureMap, while MessageDefinition (I&M) and
the TI resources (CodeSystem, ConceptMap, ValueSet, NamingSystem) are
out of scope for this ticket.

Users involved: Elliot Silver (reporter), Reuben Daniels, Marc Duteau,
Grahame Grieve.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-53766 as a Technical Correction follow-up to FHIR-48492 and
remove the residual "DEPRECATION NOTE" paragraph from the element
`comment` of every `*.jurisdiction` element on the FHIR-I conformance
resources that FHIR-48492 covered: `CanonicalResource.jurisdiction`,
`CapabilityStatement.jurisdiction`, `GraphDefinition.jurisdiction`,
`ImplementationGuide.jurisdiction`, `OperationDefinition.jurisdiction`,
`SearchParameter.jurisdiction`, `StructureDefinition.jurisdiction`, and
`StructureMap.jurisdiction`. Concretely, locate each element's comment
text in the resource's source definition (the conformance-resource
spreadsheet/`*.fsh` or per-element notes that feed
`*-definitions.html`) and delete the "DEPRECATION NOTE: For
consistency, implementations are encouraged to migrate to using the new
'jurisdiction' code in the useContext element …" paragraph. Leave the
already-updated element Description (per FHIR-48492's resolution)
untouched. No changes to bindings, cardinalities, or invariants.

#### Justification

This is the literal and obvious cleanup: FHIR-48492's resolution text
explicitly required removing "any deprecation notes" alongside the
deprecation flag, so the surviving comment paragraph is leftover work
from that resolution rather than a new design decision. It is a pure
technical correction — no semantic change, no impact on instances or
implementers beyond removing a contradictory note that currently tells
readers to migrate away from an element the spec has formally
un-deprecated. Leaving the note in place is actively harmful to
implementers reading the published HTML, because the element's
Description and its surviving comment now disagree.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than deleting the paragraph outright, rephrase it into a
non-deprecation editorial note that describes the relationship between
`jurisdiction` and `useContext`, e.g.: "Note: `useContext` may be used
in addition to `jurisdiction` to express jurisdictions of intended use
that differ from the defining jurisdiction. Use `useContext.code` =
`http://terminology.hl7.org/CodeSystem/usage-context-type#jurisdiction`
with `useContext.valueCodeableConcept` set to the jurisdiction." Apply
the same rewritten note to all eight FHIR-I `*.jurisdiction` elements
listed in Disposition A, and drop the "DEPRECATION NOTE:" prefix and
the migration framing.

#### Justification

FHIR-48492's resolution introduced an explicit distinction between the
defining jurisdiction (now `*.jurisdiction`) and additional
jurisdictions of use (carried in `useContext`), and Grahame's comment on
that ticket emphasised that this distinction matters for IG authors and
the package ecosystem. A short retained note — minus the deprecation
language — preserves the useful pointer to `useContext` for readers who
arrive at the comment looking for the migration guidance they remember
from R5, while complying with the un-deprecation. The cost is more
editorial wordsmithing than Disposition A and a small risk of
re-introducing the "two ways to do the same thing" confusion that
FHIR-48492 was trying to settle.

---

### Disposition C: Decline

#### Proposal

Close as Not Persuasive on the grounds that the Description of each
`*.jurisdiction` element was already updated under FHIR-48492 to
clarify the `jurisdiction` vs. `useContext` distinction, so the
surviving comment paragraph is harmless background context rather than
an active deprecation instruction, and that any reader needing
migration guidance from R5 still benefits from seeing it.

#### Justification

The ticket itself acknowledges that the Description has been updated.
If FHIR-I judged that the value of leaving migration guidance visible
to existing R5 implementers outweighs the editorial inconsistency, this
is a defensible (if unusual) outcome. The argument is weak, however,
because the surviving paragraph is literally headed "DEPRECATION NOTE"
and explicitly tells implementers to "migrate to using the new
'jurisdiction' code in the useContext element" — wording that directly
contradicts the resolution of FHIR-48492.

---

### Recommendation

**Recommended disposition:** A

FHIR-48492's adopted resolution already mandated removing "any
deprecation notes" from these elements; FHIR-53766 is simply pointing
out that one such note survived the implementation pass. Disposition A
finishes the job FHIR-48492 voted on, requires no new design decision,
and removes a published contradiction between the element's Description
and its comment. Disposition B is plausible but adds editorial work
without changing the substantive outcome — and the migration framing it
would preserve is exactly what FHIR-48492 ruled against. Disposition C
is hard to defend because the ticket is documenting an objective
inconsistency in the spec text, not a design preference. Recommend
treating this as a routine Technical Correction, applying it across all
eight FHIR-I `*.jurisdiction` elements in one editorial pass to keep
the eight resources aligned.
