## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a pure editorial fix on the FHIR Core `Group`
resource page. Locate every occurrence of the token `exceptiont` on
the page (and in any narrative text that the page generation pulls in,
such as the Group resource intro/notes file and any element
definitions/comments referenced by the Group `StructureDefinition`),
correct each to the intended word — `exception` if the surrounding
sentence is talking about an exceptional condition, or `exclude` if
the sentence is referring to `Group.characteristic.exclude`. While
making the change, do a single editorial pass over the same page for
any other obvious typos or unspelled abbreviations and fold those into
the same fix so the "one or more" wording in the ticket is satisfied.
Apply the corresponding fix wherever the same source text is reused
(e.g., element definitions/comments shared between the resource
narrative and the structure definition), so the next publication run
regenerates a clean page. No structural, schema, or semantic change is
required.

#### Justification

The defect is a documentation typo; the ticket explicitly identifies
itself as a Technical Correction and provides the offending token. The
priority is "Highest", which for editorial defects in FHIR typically
means the fix is expected in the next technical-correction window
without further design work. There is no Zulip discussion, no linked
ticket, and no implementer impact beyond display of the page —
correcting the spelling is consistent with prior handling of similar
typo-class Technical Corrections and carries effectively no risk of
breaking implementations. Rolling adjacent typos into the same fix is
the conventional, low-overhead way to honour the ticket's "one or
more" phrasing without proliferating tickets.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the ticket but generalize the work: open a small
companion editorial sweep covering the `Group` resource page **and**
the closely related per-element narrative sources, running an
automated spellcheck (and the existing build's text linters, where
present) across the Group page artefacts. Track any additional
findings either as additional points under this same ticket or as a
follow-up grouped editorial ticket so each individual typo doesn't
spawn its own Technical Correction. Fix `exceptiont` as part of that
sweep and capture the corrected token list in the resolution
comment for traceability.

#### Justification

The ticket's wording ("one or more misspelled or abbreviated words")
suggests the reporter suspects more than just `exceptiont` but only
recorded the specific one they noticed. A scoped sweep against the
single page is cheap, prevents a trickle of near-duplicate tickets,
and keeps the workgroup's Technical-Correction queue tidy. The
trade-off vs. Disposition A is slightly more reviewer time up front in
exchange for one consolidated change instead of N follow-on tickets;
the risk profile is unchanged because no normative content is being
altered.

---

### Disposition C: Decline

#### Proposal

Decline as **insufficient detail** / **not a defect at the asserted
location**: ask the submitter to identify the exact line or sentence
in which `exceptiont` occurs (or supply a snippet) and resubmit, on
the basis that a current-source search for the literal token does not
clearly localize the defect.

#### Justification

This is only defensible if a search of the current `Group` page
sources does **not** find the literal token `exceptiont` (e.g., it was
already fixed, or the original comment was against an older page
version). In that case, declining-with-clarification avoids a guessed
fix that touches the wrong word. It is otherwise a poor fit, because
the ticket is a Highest-priority Technical Correction with a clearly
named token; declining a documented typo in normative narrative is
hard to defend to implementers and to the original commenter.

---

### Recommendation

**Recommended disposition:** A

The ticket is a textbook Technical Correction: a clearly identified
typo (`exceptiont`) in a normative resource page, no semantic impact,
no community disagreement, and no linked discussion. The cheapest path
that fully satisfies the reporter — and that matches how the
workgroup has historically resolved similar editorial defects — is to
correct `exceptiont` to its intended word in the `Group` page sources
and apply the same fix anywhere that text is reused for generation.
Disposition B is a reasonable upgrade if a reviewer wants to bundle
adjacent editorial defects, but it should not block the immediate fix;
treat it as an optional "while you're in there" expansion of A rather
than an alternative path. Disposition C should be reserved for the
case where the literal token cannot be located in current sources, in
which case the workgroup should bounce the ticket back to the
submitter for line-level specifics rather than guess.
