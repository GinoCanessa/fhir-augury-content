## Summary

Brian Postlethwaite (FHIR-I) requests that the FHIR Mapping Language (FML)
tutorial be removed from the core FHIR specification on the grounds that it is
the only tutorial in the entire core spec and is therefore stylistically
out-of-place. He suggests the tutorial would be more appropriately hosted
either in an implementation guide or on Confluence, and notes that the issue
is "easily resolved by having the FML content in a separate specification."

The cross-reference graph in FHIR Augury surfaces no other Jira tickets or
GitHub items linked to this ticket, but the surrounding Zulip discussion
makes clear that FHIR-54486 is part of a broader R6-ballot conversation about
whether FML (and possibly StructureMap) should be split out of core into its
own specification, with parallel ballot tickets such as FHIR-54474 cited by
Grahame Grieve as needing FML/StructureMap changes that argue against a clean
split. The narrower request in this ticket — just remove the tutorial — is
therefore best read against that backdrop: it is one of several tickets
nudging FML content out of the core book.

This is the live design debate that contextualises FHIR-54486. Brian
Postlethwaite argues that FML has outgrown StructureMap — it is "way in
front" — and that, like FHIRPath, it should be allowed to evolve faster than
the core release cadence; he sees moving the FML content (including the
tutorial) into a separate, normative-ish specification as the right
mechanism. Grahame Grieve pushes back: FML is described as a textual
representation of StructureMap, several proposed FML fixes (e.g.
FHIR-54474) require StructureMap changes anyway, and the
`$transform` operation is intended to live in core FHIR — so a clean split
costs more than it gains. Lloyd McKenzie sketches a middle path of yanking
StructureMap from core into an incubator while pursuing a normative ballot of
FML, and observes that the same publishability question applies to *any*
core-spec page that we want to advance independently. Bas van den Heuvel and
Oliver Egger note that production users would tolerate an own-spec move so
long as StructureMap+FML stay coherent. No consensus is reached in the
captured window; the open questions are (a) whether the current arrangement
is so broken that current implementations must be disturbed, and (b) how to
handle the R6 publication mechanics if FML is split.

Participants: Brian Postlethwaite, Grahame Grieve, Lloyd McKenzie,
Oliver Egger, Bas van den Heuvel, Reinhard Egelkraut.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the FML tutorial page (`mapping-tutorial.html` and its supporting
example assets) from the core FHIR specification build for R6, and re-host
the tutorial content either:

- in a dedicated FML implementation guide (e.g. an `hl7.fhir.uv.fml-tutorial`
  or similar IG owned by FHIR-I), or
- on the HL7 Confluence space alongside other educational FML material.

In core, replace the tutorial page with a short stub that links to the new
home. Keep the FML language reference and the StructureMap resource
documentation in core for now (this ticket explicitly scopes itself to "the
tutorial").

#### Justification

The tutorial really is the only long-form tutorial in the core book, and its
"step 1 / step 2 / …" pedagogical style sits awkwardly next to the
reference-style content around it. Moving tutorial content to an IG or
Confluence is consistent with how other FHIR tutorials (FHIRPath walkthroughs,
SUSHI/Shorthand tutorials, terminology tutorials) are already published, and
it lets the tutorial be revised on a faster cadence than the core ballot.
None of the Zulip pushback against splitting *FML itself* applies to
splitting just the tutorial — even Grahame Grieve's main objection (that FML
fixes drag StructureMap with them) is unaffected by where the tutorial
lives, because the tutorial is documentation, not normative content.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the tutorial-removal decision and bundle it into the larger "Move FML
to a new spec?" question being worked through in the FML Zulip topic. Rather
than carve out the tutorial in isolation, target an R6 (or R6.x) outcome in
which the FML language reference *and* the tutorial both move to a dedicated
specification (e.g. an FML spec alongside the FHIRPath spec), while
StructureMap and `$transform` remain in core FHIR. Mark FHIR-54486 as
"Persuasive with mod – will be addressed by the FML-split work" and link it
to the umbrella ticket(s) coming out of that discussion.

#### Justification

The substantive Zulip debate (Brian, Grahame, Lloyd, Oliver, Bas) is not
really about the tutorial — it is about whether FML has outgrown StructureMap
and whether it should evolve on its own clock. Acting on the tutorial alone
risks (a) a confusing R6 in which the tutorial lives in one place, the
language reference in another, and StructureMap in a third, and (b)
double-rework if the larger split lands in R6.x. Resolving FHIR-54486 inside
the umbrella decision keeps the FML documentation set coherent and avoids
prejudicing the larger architectural question. The cost is that the tutorial
stays in core for at least one more publication cycle.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep the FML tutorial in the core specification.
Rationale category: editorial/style preference — the tutorial provides
substantial implementer value in the location where readers already discover
StructureMap and FML, and "it is the only tutorial in core" is not in itself
a defect.

#### Justification

The tutorial is heavily linked from the StructureMap and FML pages, and
Zulip evidence (e.g. the `resolve()` and `Translating Observation Codes`
threads) shows implementers routinely pointing newcomers at specific steps
of the tutorial. Splitting it out of core risks link-rot, version-drift
between the tutorial's step numbering and the live FML/StructureMap text,
and the loss of the integrated build-time validation the core publication
provides. If the broader "move FML to a new spec" effort proceeds, the
tutorial will move with it; if it does not, removing only the tutorial
leaves implementers worse off than today.

---

### Recommendation

**Recommended disposition:** B

The narrow request in FHIR-54486 is reasonable on its face, but the Zulip
record makes clear it is a fragment of a much larger, still-open question
about whether FML should leave core entirely. Acting on the tutorial alone
(Disposition A) optimises locally — the tutorial does sit awkwardly in core —
but it pre-decides part of the architectural question and risks an
incoherent R6 with FML language, FML tutorial, and StructureMap in three
different homes. Declining outright (Disposition C) ignores both the
reporter's stylistic point and the active design conversation that already
includes the tutorial in scope. Disposition B keeps FHIR-54486 alive,
acknowledges the reporter, and binds it to the umbrella decision so the
tutorial moves *with* the rest of the FML content (or stays put with it),
preserving coherence either way. The FHIR-I workgroup should triage this
ticket against whichever umbrella ticket(s) emerge from the
`#FHIR Mapping Language > Move FML to a new spec?` thread and resolve it
together.
