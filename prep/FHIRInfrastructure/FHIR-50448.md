## Summary

The reporter notes that, in the FHIR Core specification, a short bulleted
list discussing how to search on versioned references to canonical URLs
ends with an extra, empty bullet point that has no text after it. The
reporter believes this is a typo — either the trailing bullet should be
removed, or, if it was intended, the missing list item content should be
supplied. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items associated with this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a pure editorial fix in the FHIR Core narrative. Locate
the bulleted list about searching on versioned references to canonical
URLs (most likely on the search page covering canonical reference
search semantics) and delete the trailing empty `<li>` (or stray bullet
in the source format used by the page, e.g., a hanging `*` in a
list block) so the list ends with its last meaningful item. No
normative content changes; no resource, search parameter, or
constraint definitions are modified. Land the change as a typo-class
technical correction.

#### Justification

The ticket is explicitly typed as a Technical Correction and the
reporter describes a visible defect with no semantic impact. Removing a
stray empty bullet aligns with the spec's general editorial standard of
not shipping empty list items, and it has no implementation impact for
any consumer. Filing it as a corrected/applied technical correction is
the lowest-risk, highest-signal outcome for a one-line typo fix.

### Disposition B: Alternative Approach

#### Proposal

Before deleting the trailing bullet, ask the original page author /
work group to confirm whether the empty bullet was a placeholder for an
intended (but never added) list item — for example, an additional
caveat about how versioned canonical references interact with `_history`
or with chained / reverse-chained search. If so, supply the missing
text rather than deleting the bullet. If the author confirms it was
purely accidental, fall back to Disposition A and remove it.

#### Justification

The reporter themselves flags this possibility ("unless there was meant
to be additional content"). Trailing-empty-bullet patterns in
hand-authored narrative sometimes mark a TODO that was lost in a merge.
A short author check costs little and avoids silently dropping a
guidance item that implementers might have benefited from. Trade-off
vs. A: small additional latency and coordination, in exchange for not
losing potentially-intended normative-adjacent guidance.

### Disposition C: Decline

#### Proposal

Mark the ticket as "Not Persuasive" / decline on the grounds that
trailing whitespace in a rendered bulleted list is a cosmetic rendering
artifact, not a defect in the specification's normative content, and
defer any cleanup to the next routine editorial pass over the affected
page rather than tracking it as its own technical correction.

#### Justification

Defensible only in the narrow sense that the issue has zero
implementation impact and zero ambiguity for readers — they can see the
list ends and move on. Citing this as a reason to decline would be
consistent with declining other pure-cosmetic narrative tickets when WG
bandwidth is constrained. However, this rationale is weak in practice:
HL7 publishes the spec as the canonical artifact, and visible defects in
that artifact harm its perceived quality even when they are not
normatively significant.

---

### Recommendation

**Recommended disposition:** A

This is a textbook one-line editorial fix on a Technical Correction
ticket, raised by the reporter as "probably just a typo." There are no
cross-referenced tickets, Zulip threads, or GitHub items suggesting any
deeper substance, and the original priority was intended to be Low. The
work group should accept and apply the deletion of the trailing bullet
in the affected list on the FHIR Core page describing search on
versioned references to canonical URLs. Disposition B is only worth
escalating to if the page's recent git history shows the bullet was
introduced alongside surrounding text that hints at a missing item;
absent that signal, the cost of a coordination round-trip outweighs the
small risk of losing intended content. Disposition C should not be
chosen — leaving a known visible defect in the published specification
to "the next pass" is exactly the kind of paper cut that erodes
implementer trust in the spec's editorial quality.
