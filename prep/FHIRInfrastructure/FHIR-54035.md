## Summary

Eric Haas requests that a new **"CMS"** subsection be added to the FHIR
Signatures page (`source/signatures.html`), placed before the existing
**JWS** subsection, so that the page documents how to populate the FHIR
`Signature` datatype when the underlying digital signature is an
[RFC 5652 Cryptographic Message Syntax (CMS)](https://datatracker.ietf.org/doc/html/rfc5652)
signature or a CMS-based profile such as **CAdES (CMS Advanced
Electronic Signature)**. The proposal is deliberately styled to mirror
his companion JWS rewrite proposal in
[FHIR-54025](https://jira.hl7.org/browse/FHIR-54025): it focuses on
*how the FHIR `Signature` element is filled in* (i.e., `type`, `when`,
`who`, `onBehalfOf`, `targetFormat`, `sigFormat`, `data`) rather than
on prescribing CMS internals, so that profiles and IGs remain free to
make CMS-specific choices.

This ticket is one of a coordinated series Eric Haas filed to refactor
`signatures.html` end-to-end: the introduction
([FHIR-53835](https://jira.hl7.org/browse/FHIR-53835)), the Digital
Signature Methods overview
([FHIR-53955](https://jira.hl7.org/browse/FHIR-53955)), the JWS section
([FHIR-54025](https://jira.hl7.org/browse/FHIR-54025)), the XML/XAdES
section ([FHIR-54080](https://jira.hl7.org/browse/FHIR-54080)), and the
Signing Bundles section
([FHIR-53914](https://jira.hl7.org/browse/FHIR-53914)). It is also
aligned with [FHIR-53810](https://jira.hl7.org/browse/FHIR-53810),
which asks R6 to clear out long-standing referred signature comments.
A consistent theme across the series is that current `signatures.html`
content "focus[es] on JWS and XML, ignoring the widely used CMS,"
echoing John Moehrke's Zulip observation that "there is actually a
strong number of uses of CMS signatures (which we didn't profile)."

## Details

**Description:**

Add CMS signatures section to support the existing infrastructure.

- like my JWS proposal, focus on the FHIR content rather than the signature content.

Proposed Text before the JWS section (source at
<https://github.com/Healthedata1/fhir/blob/master/source/signatures.html>):

### CMS

When the digital signature is an
[RFC 5652: Cryptographic Message Syntax (CMS)](https://datatracker.ietf.org/doc/html/rfc5652)
signature or a profile built upon it, such as
[CAdES (CMS Advanced Electronic Signature)](https://www.etsi.org/deliver/etsi_ts/101700_101799/101733/01.17.01_60/ts_101733v011701p.pdf),
the following guidelines and rules apply:

- `Signature.type`: This element SHOULD be populated with the codes
  from [Signature Type Codes](https://hl7.org/fhir/6.0.0-ballot4/valueset-signature-type.html).
  - The Purpose can be the action being attested to or the role
    associated with the signature.
  - It SHOULD agree with any "purpose" attribute in the CMS signature
    (for example, the "commitment-type-indication" attribute defined
    in the CAdES profiles).
- `Signature.when`: This element SHOULD be populated with the
  signature timestamp.
  - It SHOULD agree with the "signingTime" or "signatureTimeStamp"
    attributes in the CMS signature.
- `Signature.who`: This element SHOULD be populated with a
  `Reference.reference` or `Reference.identifier` of who signed the
  content.
  - It SHOULD match the signer's identity found in the digital
    certificate (for example, the Subject Name (SN), or one of the
    Subject Alternative Names (SAN) in an X.509 certificate).
- `Signature.onBehalfOf`: When applicable, this element SHOULD be
  populated with a `Reference.reference` or `Reference.identifier` of
  the party represented by the signer.
- `Signature.targetFormat`: This element SHOULD be populated, and the
  value SHOULD contain a canonicalization URL from one of those
  defined above or defined by other specifications.
- `Signature.sigFormat`: This element SHOULD be populated with the
  value `application/pkcs7-signature` for CMS and CMS profiles.
- `Signature.data`: The base64 encoded DER or PEM formatted CMS
  signature.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new **"CMS"** subsection to `source/signatures.html`,
positioned immediately before the existing **JWS** subsection, with
exactly the bullet structure proposed in the ticket. Concretely:

- Open with the RFC 5652 / CAdES framing paragraph.
- For each of `Signature.type`, `Signature.when`, `Signature.who`,
  `Signature.onBehalfOf`, `Signature.targetFormat`, `Signature.sigFormat`,
  and `Signature.data`, add a SHOULD-level guideline that mirrors
  the JWS section's structure.
- Bind `Signature.type` to the existing
  [Signature Type Codes](https://terminology.hl7.org/ValueSet/v3-SignatureCode)
  value set used elsewhere on the page.
- Set `Signature.sigFormat` guidance to `application/pkcs7-signature`
  for CMS and CMS profiles, and `Signature.data` guidance to "base64
  encoded DER or PEM formatted CMS signature."
- Cross-reference CAdES attributes (`commitment-type-indication`,
  `signingTime`, `signatureTimeStamp`) and X.509 SN/SAN as the
  source of `who` identity.

This change is editorial-additive: it does not modify the `Signature`
datatype, the `valueset-signature-type` value set, or any existing
JWS/XMLDSig guidance.

#### Justification

The proposal is symmetric with the existing JWS subsection and with
Eric Haas's companion rewrites in FHIR-54025 (JWS) and FHIR-54080
(XMLDSig), so the page becomes internally consistent across the three
signature families. It directly addresses John Moehrke's stated gap on
the S&P > Signature Zulip thread ("a strong number of uses of CMS
signatures … which we didn't profile") and the FHIR-53835 critique
that the current page "focus[es] on JWS and XML, ignoring the widely
used CMS." Because every guideline is a SHOULD against the existing
`Signature` datatype — and the only new MIME type asserted
(`application/pkcs7-signature`) is the IANA-registered media type for
CMS — there is no breaking change, no new resource, and no new value
set. It is the minimum credible R6 response to the long-standing
referred signature comments tracked under FHIR-53810.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the *intent* of FHIR-54035 but restructure the page so that the
three signature-family subsections (CMS, JWS, XMLDSig) are generated
from a single shared template/table whose rows are the `Signature.*`
elements and whose columns are CMS / JWS / XMLDSig. Concretely:

- Define a single normative "Populating `Signature` for digital
  signature methods" table in `signatures.html` that has one row per
  `Signature` element (`type`, `when`, `who`, `onBehalfOf`,
  `targetFormat`, `sigFormat`, `data`) and one column per family
  (CMS/CAdES, JWS/JAdES, XMLDSig/XAdES), with the family-specific
  attribute names ("commitment-type-indication" vs "srCms" vs
  "CommitmentTypeId", `signingTime` vs `iat`/`sigT`, etc.) inline in
  each cell.
- Keep short narrative subsections per family for the framing
  paragraph and the `sigFormat` MIME type, but stop duplicating the
  bullet list across three sections.
- Treat FHIR-54025, FHIR-54035, and FHIR-54080 as a single editorial
  change set delivered together.

#### Justification

The literal proposal in FHIR-54035 — combined with the parallel
proposals in FHIR-54025 and FHIR-54080 — produces three near-identical
bullet lists that differ only in family-specific footnotes and the
`sigFormat`/`data` MIME details. That is exactly the kind of
copy/paste structure that produced the "`Signature.type` SHALL contain
the same values as the srCms element" copy/paste error FHIR-54080 is
fixing. A shared table makes the parallelism visible to the reader,
guarantees editorial consistency, removes one obvious source of future
drift, and makes it trivially easy to add a fourth family later (e.g.,
COSE/CWT) without restructuring the page. The trade-off versus
Disposition A is a slightly larger editorial scope and the need to
land FHIR-54025/54035/54080 as a coordinated set rather than
independently.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54035 on the grounds that CMS signature usage with FHIR
is already covered by the existing generic `Signature` datatype plus
the page's general "Other signature methods can be used, and their
use with FHIR can be extrapolated from the content on this page"
escape hatch. Refer implementers who need CMS-specific guidance to a
domain-specific IG (e.g., a future Security/IHE-DSG IG) rather than to
core FHIR.

#### Justification

This disposition is defensible only as a "scope" argument: core FHIR
historically resists profiling specific cryptographic standards
(Grahame Grieve repeatedly pushed the Zulip "Signing FHIR artifacts"
thread back toward signed-Bundle + Provenance as the spec-supplied
mechanism, and asked what per-resource signatures prove beyond TLS).
A decline keeps `signatures.html` smaller and avoids the appearance
that core FHIR endorses CMS over the alternatives. However, this
position is hard to sustain given that the page already singles out
JWS and XMLDSig with the exact same level of detail FHIR-54035
proposes for CMS — declining CMS specifically would leave the page
visibly asymmetric and contradict FHIR-53835's separately-proposed
intro that already names CMS alongside JWS and XMLDSig. There is no
backwards-compatibility or implementer-burden argument for declining;
the change is purely editorial-additive.

---

### Recommendation

**Recommended disposition:** B

Disposition A delivers exactly what the ticket asks for and would be a
clear improvement over the status quo, but the ticket cannot
realistically be evaluated in isolation — Eric Haas filed
FHIR-54025 (JWS), FHIR-54035 (CMS), and FHIR-54080 (XMLDSig) as a
deliberately parallel set, all editing the same `signatures.html` page
under the same intro rewrite (FHIR-53835) and methods rewrite
(FHIR-53955). Adopting them as three independent bullet lists locks
in the same kind of copy/paste structure that already produced the
`srCms`-in-XML defect FHIR-54080 is having to fix. Folding the three
into a single shared element-by-family table (Disposition B) preserves
every SHOULD-level guideline Eric proposes, captures the exact CMS /
CAdES detail in this ticket (commitment-type-indication, signingTime,
`application/pkcs7-signature`, DER/PEM), and answers the John Moehrke
"we didn't profile CMS" gap on the S&P Zulip thread — while leaving
the page editorially robust for any future signature family. If the
WG prefers minimum scope for R6, falling back to Disposition A is
acceptable; Disposition C should not be taken because the page's
existing JWS/XMLDSig coverage already commits the spec to this level
of detail.
