## Summary

Reporter Corey Spears observes that on the FHIR Core search page, the
worked example that is supposed to appear immediately after the
explanatory sentence — *"will return records that have BOTH a match for
the given name of `valueGivenA` AND a match for the given name of
`valueGivenB`. Note that this will match both patient records that have
a single name containing multiple given elements as well as records that
have a multiple name elements that, in total, contain the values:"* —
is not rendered. The trailing colon at the end of the sentence indicates
that an example block (URL / request, or sample resources, or both) was
intended to follow but does not appear in the rendered specification.
This is filed as a Technical Correction against FHIR Infrastructure for
the FHIR Core specification. No cross-referenced Jira tickets, Zulip
discussions, or GitHub items were surfaced for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation defect in the FHIR Core specification's
Search page (`search.html`), in the "Combining results from multiple
parameters" subsection that discusses repeating the same search
parameter to express AND semantics across multiple `given` values.
Restore the missing worked example so that the trailing colon in the
prose actually leads into a concrete example. The restored block should
contain at minimum:

- The example search URL using the documented AND-by-repetition form,
  e.g. `GET [base]/Patient?given=valueGivenA&given=valueGivenB`.
- A short illustration of the matching semantics described in the
  prose: a Patient with one `name` containing two `given` values
  (`valueGivenA`, `valueGivenB`) **and** a Patient whose two `name`
  entries collectively contain those two `given` values both match,
  while a Patient with only one of the two given values does not.
- Confirmation that this is the AND form, contrasted (by reference)
  with the comma-separated OR form (`given=valueGivenA,valueGivenB`)
  documented elsewhere on the page.

Investigate whether the example was lost via a stray markup/include
issue (e.g., a broken `{% include %}` / IG-publisher fragment, a
mis-nested `<div class="example">`, or a removed snippet during a prior
edit) and fix the underlying source so the example renders, rather than
only patching the rendered output.

#### Justification

The ticket is explicitly a Technical Correction, the reporter has
identified a concrete rendering gap, and the surrounding prose is
authored as if an example follows (the sentence terminates in a colon).
Restoring the example aligns the rendered spec with its evident editorial
intent and matches the established pattern on the Search page of pairing
each combining-rule explanation with a worked URL example. There is no
normative change implied — this is purely a documentation fidelity fix,
which is exactly what the Technical Correction type is for.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than restoring a single inline example, refactor the
"Combining search parameters" subsection so that all four combinations
are presented in a single, consistent comparison table:

1. Same parameter, repeated → AND
   (`given=valueGivenA&given=valueGivenB`)
2. Same parameter, comma-separated values → OR
   (`given=valueGivenA,valueGivenB`)
3. Different parameters → AND across parameters
   (`given=valueGivenA&family=Smith`)
4. Composite parameters → AND within a single value (where applicable).

Each row would carry a one-line semantics description and the worked
URL. Replace the now-missing prose-plus-colon-plus-example pattern with
the table reference, and keep the Patient `name`/`given` matching note
as a callout below the table.

#### Justification

The current page interleaves several closely related rules in long
prose, and the bug under review is partially a symptom of that style:
when one example block goes missing, the reader is left with dangling
prose. A comparison table keeps the rules visually adjacent, makes it
much harder for a single missing snippet to silently break the section,
and gives implementers a single canonical reference for the
AND-vs-OR-vs-multi-parameter distinctions that consistently confuse
new adopters. This is still a Technical Correction in spirit (no
normative change), but it improves the long-term maintainability of the
page rather than only patching the current symptom.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive / No Change** on the basis that the
explanatory prose surrounding the missing block already states the
matching rule in full ("BOTH a match for the given name of `valueGivenA`
AND a match for the given name of `valueGivenB`…"), so the omitted
example is not strictly required for a reader to understand the
behaviour, and the AND-by-repetition form is documented and exemplified
elsewhere on the same page.

#### Justification

This rationale is genuinely weak and is included only for completeness:
the reporter has identified a real rendering gap in normative
documentation, and the trailing colon in the source prose makes the
absence visible to every reader of the page. Declining a clear
documentation defect risks leaving implementers to infer what the
missing example was supposed to show, which directly undermines the
purpose of the Search page. There is no breaking-change risk, no scope
concern, and no competing mechanism that justifies leaving the page in
its current broken state.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right call because the ticket is a narrowly scoped
Technical Correction that matches an obvious editorial defect: the
surrounding prose is authored to lead into an example, and the example
is missing. Restoring it (and fixing the underlying source so it
renders, not just the rendered HTML) delivers exactly what the reporter
asked for, requires no normative change, and can be implemented and
verified entirely within the FHIR Core spec source for `search.html`.
Disposition B is attractive as a longer-term improvement, but bundling
a section refactor into a Technical Correction expands the change
surface, invites editorial debate, and risks delaying the simple fix
that the reporter actually needs; it would be better filed as a
follow-up improvement ticket once the immediate rendering gap is
restored. Disposition C should not be entertained — leaving normative
documentation visibly broken when the fix is straightforward is not
defensible. Before implementation, a `repo-analysis` run against
`HL7/fhir` should be performed to identify the exact authoring source
file (likely under the `source/` tree) and confirm the IG-publisher
fragment or include that produced the empty example slot.
