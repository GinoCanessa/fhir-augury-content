## Summary

Brian Postlethwaite has filed a "collection" change request against the
FHIR Mapping Language (FML) documentation, specifically the **Transforms**
section that lists the built-in transform functions used in
StructureMap rules (`escape`, `create`, `cast`, `reference`, `pointer`,
`translate`, `truncate`, etc.). The ticket raises both a documentation
quality issue — the current table format is hard to read and lacks
parameter definitions, examples, and links — and several semantic
questions about individual transforms: whether some of them duplicate
functionality already available as FHIRPath functions or operators
(`truncate`, `translate`, `cast`, `escape` vs. encode/decode,
`create` vs. the `%factory` / new CQL-style `create` syntax), how
multi-version FHIR scenarios resolve which type provider applies (source
vs. target) for `create` and `cast`, the relationship between
`reference` and `pointer`, and a duplicate `translate` row in the
table. There are no explicit Jira links and the cross-reference index
returned no related Jira, GitHub, or Zulip items, but recent FHIR
Mapping Language Zulip threads (notably `evaluate transform?` and
`Passing a string as parameter`) corroborate that real implementers
are tripping over exactly these documentation gaps.

### [FHIR Mapping Language > ✔ evaluate transform?](https://chat.fhir.org/#narrow/stream/FHIR%20Mapping%20Language/topic/%E2%9C%94%20evaluate%20transform%3F)

Jean Duteau attempted to use `evaluate(bdSrc, substring(0,4))` on a
`Patient.birthDate` and got "Transform evaluate takes 1 parameter(s)
but 2 were found", because the documented signature contradicts the
described 2-parameter (context, expression) behaviour. Brian
Postlethwaite recommended dropping `evaluate` and using the inline
FHIRPath shortcut `tgt.year_of_birth = (bdSrc.substring(0,4))`
instead — exactly the "use FHIRPath instead" point the ticket raises
about `truncate`/`translate`/`cast`. After several rounds of context
and conversion errors (`bdSrc` not bound; needing
`birthDate.toString().substring(...)`; needing the `src.` prefix even
though the publisher errors on it), the thread was resolved. It also
exposes that the FML lab/publisher disagree on what is legal, and that
the `evaluate` transform's parameter documentation is wrong or
incomplete. Participants: Jean Duteau, Brian Postlethwaite.

### [FHIR Mapping Language > Passing a string as parameter](https://chat.fhir.org/#narrow/stream/FHIR%20Mapping%20Language/topic/Passing%20a%20string%20as%20parameter)

Ritika Rawlani tried to pass a literal string as a group parameter
and used `create('String') as text, text = "unk"` as a workaround,
which failed. Oliver Egger confirmed that **FML does not currently
support constants as group parameters** ("there is a proposal for
creating variables, but not implemented yet") and recommended a
FHIRPath-expression workaround `vax -> ('unknown') as occ ...`. Jose
Costa Teixeira separately asked whether `$transform` can accept
additional named parameters; Oliver replied "not according to
[structuremap-operation-transform.html](https://build.fhir.org/structuremap-operation-transform.html)".
Both points reinforce the ticket's request that the `create`
transform and the `$transform` operation be documented precisely
(parameters, type provider semantics, FHIRPath equivalents).
Participants: Ritika Rawlani, Jose Costa Teixeira, Oliver Egger.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure the **Transforms** table on the FHIR Mapping Language
page (`mapping.html` in HL7/fhir, plus the `structuremap.html` cross-
references) into a per-transform subsection. For every built-in
transform (`create`, `copy`, `truncate`, `escape`, `cast`, `append`,
`translate`, `reference`, `dateOp`, `uuid`, `pointer`, `evaluate`,
`cc`, `c`, `qty`, `id`, `cp`), add:

1. A heading with a stable anchor.
2. A normative parameter list (name, type, cardinality, whether
   constant or variable, source vs. target type provider).
3. A worked example in FML and a sample input/output snippet.
4. Where a FHIRPath equivalent exists, an explicit
   "FHIRPath alternative" callout — e.g. `cast(X)` ↔ `ofType(X)` /
   `as X`; `truncate(s,n)` ↔ `s.substring(0, n)`; `translate(...)` ↔
   `%terminologies.translate(...)`; `escape` ↔ FHIRPath
   `encode`/`decode`.
5. Remove the duplicate `translate` row from the table.
6. Add a paragraph clarifying the type-provider rule for `create` and
   `cast` in multi-version contexts (per the ticket's hypothesis,
   document it as the **target** type provider unless the rule
   explicitly names a source type).
7. Either remove `reference` from the spec or document precisely what
   it produces and how it differs from `pointer` (e.g. literal
   `Reference` resource vs. a string pointer).

#### Justification

These are documentation defects, not normative changes. FHIR
Infrastructure has consistently treated "documented as a function but
no parameter list, no example, no FHIRPath cross-link" as a defect for
FHIRPath itself, and the same standard should apply to FML. The Zulip
threads above demonstrate that real implementers (Duteau, Rawlani,
Costa Teixeira) are deriving the parameter shape and type-provider
rules empirically, which is exactly the failure mode this disposition
fixes. The duplicate `translate` row is unambiguously a bug. None of
the proposed wording changes a normative behaviour, so backward
compatibility is preserved.

---

### Disposition B: Alternative Approach

#### Proposal

Split FHIR-54484 into one parent tracking ticket plus per-transform
child tickets, and adopt a tiered fix:

1. **Immediate (R6 ballot window, doc-only):** fix the duplicate
   `translate` row, add parameter tables for every transform, and add
   one canonical example per transform — without yet committing to
   FHIRPath-equivalence statements or to the type-provider rule.
2. **Follow-up tickets (per transform):** for `cast`, `truncate`,
   `escape`, `translate`, and `create`, raise individual tickets that
   propose **deprecating** the transform in favour of the FHIRPath
   equivalent, marking the transform as "retained for backward
   compatibility, prefer the FHIRPath alternative shown".
3. **Separate ticket on `reference` vs. `pointer`:** treat as a
   normative clarification (what each one returns and when each
   should be used) rather than just a doc fix, since the difference
   is substantive.
4. **Separate ticket on multi-version type provider:** make this a
   normative addition to the StructureMap page, because it changes
   how engines must resolve `create('Patient')` when both a source
   and target StructureDefinition declare `Patient`.

#### Justification

The ticket itself is explicitly a "collection of issues" and mixes
pure documentation defects with semantic questions that touch the
FML execution model. Splitting them lets the workgroup land the
non-controversial doc fixes immediately (helping the implementers in
the Zulip threads now), while routing the deprecation-style questions
through the normal change-control flow with proper migration impact
analysis. It also avoids one large omnibus PR that's hard to ballot.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54484 as **Not Persuasive — out of scope as filed** and
ask the reporter to refile each concern as a focused, testable
change request. Rationale category: a single ticket cannot resolve a
mix of "rewrite this section as headings", "add examples for ~16
functions", "document parameter types", "clarify type-provider rules
in multi-version contexts", "reconcile transforms with FHIRPath
equivalents", and "explain `reference` vs. `pointer`" — each of
those needs its own disposition and, in some cases, its own
implementer impact assessment.

#### Justification

The ticket is a useful checklist but is not actionable as a single
unit of work; any committer would have to break it down before
implementing, and the workgroup vote would have to bundle disparate
decisions. Declining-with-guidance is consistent with how FHIR-I has
handled prior "umbrella" tickets, and it preserves the substance of
each concern by inviting individual tickets that can be balloted on
their own merits.

---

### Recommendation

**Recommended disposition:** B

The reporter is correct on every point — the transforms table really
is too compact to document parameter types and examples, `translate`
really is duplicated, and the `reference`/`pointer`/`cast`/`create`
semantics really are under-specified — so an outright decline (C) is
not justified. But A bundles low-risk doc cleanup with two genuinely
normative questions (the multi-version type provider rule and the
`reference` vs. `pointer` distinction) that each deserve a focused
disposition and a clear implementer-impact note. B captures the best
of both: ship the doc cleanup and duplicate-row fix in the next
publication so implementers like those in the
`evaluate transform?` and `Passing a string as parameter` threads
get immediate relief, while routing the FHIRPath-equivalence
deprecations and the type-provider rule through their own tickets
where they can be discussed on their own merits and won't block the
documentation fix.
