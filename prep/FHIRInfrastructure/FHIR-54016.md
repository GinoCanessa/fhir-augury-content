## Summary

The submitter (Lloyd McKenzie) requests that `Requirements.statement.category`
be given a coded binding — specifically, a **`preferred`**-strength binding to
a UTG-managed value set derived from the one already used by the
**Coverage Requirements Discovery (CRD)** Implementation Guide. Today the
`category` element on `Requirements.statement` exists but is unbound, so
implementers categorising statements within a `Requirements` resource have no
shared vocabulary to draw from. Promoting CRD's existing categorisation list
to a UTG value set and binding it preferentially would give all consumers of
`Requirements` a common starting point while still allowing other code systems
where appropriate. There are no Jira cross-references on file; the only
related discussion located is a Zulip thread the same reporter started about
pre-adopting `Requirements.statement.category` (and sibling R6 elements) via
inter-version extensions in R5 tooling, which establishes that the element is
already being exercised in the wild ahead of R6 stabilisation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a `preferred`-strength binding on `Requirements.statement.category` to a
new UTG-managed value set (e.g.
`http://hl7.org/fhir/ValueSet/requirements-statement-category`) whose initial
content is lifted from the CRD IG's existing statement-category list, expanded
into a UTG code system if the CRD codes are not already in one. Concretely:
(1) author the code system and value set under the `terminology/` authoring
area in `HL7/UTG`, mirroring the CRD codes and definitions; (2) update the
`Requirements` resource definition in `HL7/fhir` to reference that value set
on `statement.category` with `strength = preferred`; (3) cross-check the CRD
IG to ensure its current usage continues to validate against (or, ideally,
adopt) the new core value set.

#### Justification

`Requirements.statement.category` is already being pre-adopted in the wild
(see the IG-creation Zulip thread), so leaving it un-bound forces every
consumer to invent local categorisations. CRD has already done the design
work for a reasonable categorisation, so reusing it avoids divergence and
gives implementers a curated starting point. A `preferred` strength is the
right vehicle because (a) `Requirements` is intended to be very broadly
applicable and a hard binding would over-constrain it, and (b) `preferred`
matches the submitter's stated intent and is consistent with how core resource
elements with non-universal categorisations (e.g. `Encounter.class` historical
peers, `Goal.category`) are typically bound in FHIR core.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than creating a new core-owned value set, define the binding as
`example` strength pointing at the existing CRD value set URL directly, *or*
keep the binding `preferred` but make the value set **extensible** off a small
core-curated seed (general-purpose categories such as `business`, `functional`,
`non-functional`, `regulatory`, `safety`) and let CRD continue to maintain
its own narrower list as a derived value set. Authoring would still be in
`HL7/UTG` for the seed value set/code system, but the CRD-derived list would
remain owned by the CRD IG.

#### Justification

A core-owned value set seeded from a single domain IG (CRD) risks bending
"Requirements categorisation" toward one use case (US payer coverage
requirements). An `example` binding to the CRD value set communicates that
the categorisation is illustrative without baking CRD vocabulary into the base
spec, and a `preferred + extensible-seed` variant gives implementers a small,
domain-neutral vocabulary while still allowing CRD-style specialisation. This
also reduces UTG governance load: only the small seed lives in UTG, and the
heavy lifting stays in the IG that already maintains it.

---

### Disposition C: Decline

#### Proposal

Do not add a binding on `Requirements.statement.category` at this time. Note
in the resource boilerplate that implementers SHOULD use a locally-agreed
value set (and may consult the CRD list as one example), and revisit the
binding only after R6 stabilises and we have evidence of categorisation
convergence across multiple IGs.

#### Justification

`Requirements` is still a relatively new resource being pre-adopted ahead of
R6 (per the Zulip thread). Adding a binding — even `preferred` — into core
now will cascade through the inter-version extension story Lloyd is already
wrestling with: any pre-adopting tooling will then also need to convey the
binding via inter-version mechanisms, and any future change to the value set
becomes a core-level breaking change. Without cross-IG evidence of
convergence on CRD's specific categorisation, declining (or deferring)
preserves flexibility at low cost — implementers can still reference the CRD
value set today.

---

### Recommendation

**Recommended disposition:** A

The submitter is the resource's primary author and is already pre-adopting
`Requirements.statement.category` in IG publisher tooling, so the absence of
a binding is an actual implementer pain point, not a hypothetical one. CRD
has done the categorisation design work and that list is, on its face,
applicable beyond the payer-coverage use case (categories such as functional
vs. non-functional vs. regulatory generalise well). A `preferred` strength —
not `extensible` and not `required` — keeps the door open for IGs that need
domain-specific categorisations while giving the majority of consumers a
curated default, which is exactly the role `preferred` was designed for.
Disposition B is reasonable if FHIR-I objects to seeding a core value set
from a single IG, but it adds governance complexity without clear benefit;
Disposition C effectively ratifies the status quo and pushes the same problem
back to the workgroup in the next cycle.
