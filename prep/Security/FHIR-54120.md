## Summary

Hans Buitendijk asks the Scalable Consent Management (FHIR) IG to address, in a
future version, how a consent registry decides which subset of a patient's
consent rules a given data source is allowed to see when that data source is
itself only authorized to receive a subset of the patient's data. The current
guide effectively treats access to a patient's consent rules as all-or-nothing,
which does not reflect realistic scenarios where (a) a data source receiving
some patient data should not see consents that govern data it is not entitled
to receive, or (b) different providers for the same patient may share with a
particular party while others may not. He notes that while the consent rule
content can encode such logic, distributing many fine-grained rules through a
consent registry quickly becomes unwieldy. The Security WG triaged the ticket
as **Persuasive** and resolved it by adding an implementer note (rather than
new normative machinery) that explicitly calls out this limitation: a data
source that holds some of a patient's data does not automatically receive all
of that patient's consents, and registries should constrain the consents
shared to those relevant to the data the recipient is concerned with. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were found.

## Details

**Description:**

> For the next version we should address how to manage at scale whether a data
> source that has a patient's data can access only some of a patient's consent
> rules to determine who they can or cannot further share that patient's
> information.
>
> Scenarios to consider are whether a data source that can receive some
> information on a patient can see consents that involve the data that they
> cannot receive.  Currently the focus is more on accessing all consent rules
> or none.  Other scenarios are where one or more providers for a patient can
> share information with a particular party, but other providers cannot share
> with that particular party.  While the rule itself can include that logic,
> particularly when a patient's consent rules are managed in a distributed
> fashion sharing too many consent rules becomes more challenging.

**Resolution Description:**

> We will add a section that details a note to implementers that a concern that
> we haven't addressed is that a data source that has a patient's data does not
> automatically get access to all of the patient's consents. They should only
> receive consents that deal with the data that the source is concerned with.
> Since the guide is concerned with the sharing of consent instances across
> consent registries, the actual rules around sharing of those can get tricky.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative or near-normative section to the Scalable Consent Management
(FHIR) IG that defines how a consent registry filters the set of `Consent`
resources returned to a requesting data source so that the source only sees
consents that govern data it is itself authorized to receive. Concretely:

- Define a "consent administrator access" concept (likely an actor /
  capability in the IG's actor model) that is distinct from a data source's
  data-access role.
- Specify a filtering algorithm keyed on the intersection of
  `Consent.provision.data`, `Consent.provision.class`,
  `Consent.provision.actor`, and `Consent.provision.purpose` with the
  requester's authorized scope (e.g., the data categories / classes the
  registry knows the requester is permitted to receive).
- Add a SearchParameter or operation (e.g., `$consents-for-source`) on
  `Consent` that returns only the in-scope subset, plus guidance on how
  registries should authorize the call (e.g., SMART scopes, mTLS-bound
  registry agreements).
- Address the multi-provider variant by allowing the filter to also constrain
  on `Consent.performer` / `Consent.grantor` so that a recipient sees only
  consents involving providers whose data they are entitled to receive.

#### Justification

This directly addresses the underlying problem the reporter raises: today
the IG implicitly treats consent-rule access as all-or-nothing, which forces
registries either to over-share (leaking metadata about data the requester
cannot receive) or under-share (breaking rule evaluation). Specifying the
filter in the IG avoids each implementer reinventing it incompatibly, which
is the central scalability concern. It also keeps the policy logic in the
registry where the authoritative knowledge of source authorization lives,
rather than pushing it into individual `Consent.provision` rules — which
the description explicitly flags as unworkable when consents are managed in
a distributed fashion.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the resolution that has already been recorded on the ticket: add an
**implementer note** (not new conformance machinery) to the Scalable Consent
Management IG that explicitly says:

- A data source holding some of a patient's data does **not** automatically
  receive all of that patient's consents.
- Consent registries are expected to share only those consents that pertain
  to data the recipient is concerned with / authorized to receive.
- The mechanism for that filtering, including the multi-provider /
  multi-recipient variants Hans describes, is out of scope for this version
  of the IG and is acknowledged as an open issue for a future version.

Optionally, accompany the note with the diagram annotation david.pyke offered
to accept ("writeups against the diagram from Hans to explain the issue") so
that the registry interaction diagram visually marks the filtered-consent
boundary as an unresolved policy point.

#### Justification

The change request is explicitly framed as "for the next version" and the
reporter himself stops short of proposing a concrete mechanism, noting that
even encoding the logic in `Consent.provision` itself becomes unwieldy at
scale. A formal filtering API (Disposition A) would require defining the
authorization scope of a "data source" in registry-neutral terms — work the
IG has not yet done — and risks locking in a design before implementer
experience exists. An implementer note preserves the integrity of the IG,
warns implementers off the naive all-or-nothing pattern, and creates an
anchor (with Hans's diagram writeup) for the next ballot cycle to attach a
concrete proposal to. This is also the path the WG has already converged on
in the ticket's recorded resolution.

---

### Disposition C: Decline

#### Proposal

Close the ticket without changing the IG, on the grounds that selective
disclosure of consent instances is a registry-side authorization concern
rather than a FHIR consent-modeling concern, and is already addressable with
existing mechanisms: registries can enforce per-requester filters using the
same SMART scopes / OAuth claims they already use to authorize access to the
underlying clinical data, and `Consent` search can be constrained with
existing search parameters (e.g., `category`, `actor`, `data`).

#### Justification

A decline is defensible because the IG is about how consents are *shared
between registries*, not about how a registry authorizes an individual
requesting client. Implementers already build per-client authorization
filters for clinical data; reusing that infrastructure for `Consent` keeps
the IG focused. However, this rationale ignores that the reporter's concern
is precisely that there is no *interoperable* way to know which consents are
in-scope for a given source — every registry inventing its own filter is
the scalability failure he is pointing at — so this disposition is the
weakest of the three and should only be selected if the WG decides the
problem is genuinely outside the IG's remit.

---

### Recommendation

**Recommended disposition:** B

The WG has already resolved the ticket as **Persuasive** and the recorded
resolution maps cleanly to Disposition B: an implementer note that names the
limitation, points registries away from the all-or-nothing pattern, and
defers a concrete mechanism to a future version. Disposition A is the right
*eventual* destination but is premature — the IG does not yet define the
registry-side authorization vocabulary an interoperable filter would need,
and Hans's own framing flags that even the rule-embedded variant is
unwieldy, so committing to a specific filter API now risks rework. Disposition
C understates the interoperability gap. Accept B, attach Hans's diagram
writeup once received (per david.pyke's comment), and log a follow-up
tracker for the Disposition-A work in the next ballot cycle.
