## Summary

Craig Newman (Security WG) is asking the Scalable Consent Management (FHIR) IG
to clarify what `AuditEvent.outcome` actually means in the context of
consent-driven disclosure decisions. The IG appears to assume that whether or
not information was shared is recorded in `AuditEvent.outcome`, but the
required value set (success / minor failure / serious failure / major failure)
does not obviously map onto a "shared vs. not shared" dichotomy. The reporter
wants explicit guidance on (a) whether `success` (0) means data *was* shared or
that privacy *was* protected and data was *not* shared, and (b) what the three
graded failure codes (minor, serious, major) are intended to convey for a
consent-mediated disclosure or authorization decision. There are no linked
Jira tickets and no Zulip cross-references; the only cross-references the
indexer surfaces are to the FHIR artifacts the description itself mentions
(`AuditEvent.outcome`, `AuditEvent.outcomeDesc`).

## Details

**Description:**

> Assuming that the outcome of whether or not information was shared is
> recorded in AuditEvent.outcome, please clarify how the values in the required
> value set reflect sharing or not sharing. Is "success" to indicate that data
> was shared or that privacy was protected and information was not shared?
> What is the intended meaning of the grades of failure (minor, serious and
> major)?

**Resolution Description (already entered on the ticket as "Persuasive"):**

> We will add explicit guidance clarifying the meaning of `AuditEvent.outcome`
> codes in the context of consent-related audit events. "Success" (0) indicates
> the audited system operation completed without error — it does not indicate
> whether data was shared or withheld. For disclosure audit events, success
> means the disclosure operation executed as authorized by the applicable
> consent. For authorization decision audit events, success means a valid
> decision was reached. The actual permit/deny outcome of an authorization
> decision should be captured using `AuditEvent.outcomeDesc` or through the
> appropriate structured elements defined in the IHE-BALP profiles, not
> through the outcome severity code alone.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the Persuasive resolution already drafted on the ticket: add an explicit
narrative section to the Scalable Consent Management (FHIR) IG (e.g., a
"Recording audit outcomes for consent-mediated disclosures" subsection on the
audit / AuditEvent guidance page) stating that:

- `AuditEvent.outcome` is an *operation severity* code, not a "shared vs.
  withheld" indicator. `0` (success) means the audited system operation
  completed without error and says nothing on its own about whether PHI was
  released.
- For disclosure audit events, `outcome = 0` means the disclosure executed as
  authorized by the applicable Consent (including the legitimate case where
  the Consent denied the disclosure and zero data was released — that is still
  a successful execution of the consent decision).
- For authorization-decision audit events, `outcome = 0` means a valid
  permit/deny decision was reached.
- The actual permit / deny outcome of a consent decision MUST be carried in
  `AuditEvent.outcomeDesc` (free text rationale) and/or in the structured
  IHE-BALP profile elements the IG already references, not inferred from the
  severity code.
- Add at least one worked example for each of the four severity codes in a
  consent context so implementers can see what `minor`, `serious`, and `major`
  failures look like (e.g., minor = transient retry-able error in policy
  evaluation; serious = decision could not be reached / fail-closed; major =
  unauthorized disclosure or PDP outage with potential data exposure).

#### Justification

The reporter's confusion is real and predictable: `AuditEvent.outcome` is
defined in the FHIR base spec as an operation-level severity bound to a
required value set (`http://hl7.org/fhir/ValueSet/audit-event-outcome`), and
the IG language reads as if it conflates that severity with the
*authorization* outcome. Clarifying the semantics in IG narrative — without
changing the underlying element binding or the IHE-BALP-aligned structure the
IG already mandates — is the lowest-risk way to remove that confusion. It is
also exactly the answer the WG has already agreed to in the resolution field
(`Persuasive`), so accepting locks in editorial work that is already scoped
and uncontroversial.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of relying on narrative guidance alone, add a *structured* mechanism
to the IG to carry the consent decision outcome explicitly, so that the
permit/deny outcome is never read off `AuditEvent.outcome`. Concretely:

- Profile `AuditEvent` in the IG with a `consentDecision` extension (or a
  required slice on `AuditEvent.entity` that points at the evaluated
  `Consent`) carrying a coded value from a small new ValueSet:
  `permit | deny | not-applicable | indeterminate` (XACML-aligned).
- Keep `AuditEvent.outcome` bound to the standard required value set, but in
  the profile add an invariant requiring the consent-decision extension to be
  present whenever `AuditEvent.code` indicates a consent / authorization
  decision event.
- Update `AuditEvent.outcomeDesc` guidance to be a *human-readable* rationale
  only, not the canonical decision carrier.
- Add the same narrative clarification proposed in Disposition A as
  supporting prose.

#### Justification

The advantage over A is machine-readability: any audit consumer (BALP,
SIEM, breach-reporting tooling) can filter on `permit` vs. `deny` directly
without parsing free text in `outcomeDesc` or inferring intent from severity.
It also makes the IG enforceable via FHIRPath / SHALL invariants rather than
"please read the prose".

The trade-offs are real: (1) it introduces a new IG-specific extension when
IHE-BALP already defines structured carriers for the same information, which
risks forking from the broader audit ecosystem; (2) it widens the scope of a
ticket that is fundamentally a *clarification* request into a *normative
profile change*; (3) implementers who have already coded against the current
IG would need to re-tool. The Persuasive resolution explicitly says the
permit/deny outcome should be captured via IHE-BALP elements, so duplicating
that with a new extension would need WG sign-off that the BALP path is
insufficient.

---

### Disposition C: Decline

#### Proposal

Decline as "Not Persuasive — addressed by base spec." Take the position that
`AuditEvent.outcome` is unambiguously defined in the FHIR base specification
as an operation-level severity (the value set is named
`audit-event-outcome` and the codes are explicitly `Success`, `Minor failure`,
`Serious failure`, `Major failure` describing the *audited operation*, not the
*business decision*), and that the Scalable Consent Management IG inherits
that definition without needing to restate it. Direct the reporter to the
base FHIR R5/R6 documentation on `AuditEvent.outcome` and to IHE-BALP for
the consent-decision carrier.

#### Justification

The defensible angle is that we should not duplicate base-spec definitions in
every IG that uses `AuditEvent`; doing so risks IG-specific drift if the base
binding ever changes. The honest weakness, however, is that the reporter is
not a casual reader — Craig Newman is an experienced Security WG contributor
— and if *he* finds the IG ambiguous, downstream implementers certainly
will. "Read the base spec" is rarely a satisfying disposition for a
clarification ticket and is hard to justify when the WG has already agreed
the request is Persuasive.

---

### Recommendation

**Recommended disposition:** A

The reporter is asking for clarification, not a normative change, and the
ticket's existing resolution field already commits to a Persuasive answer
that maps cleanly onto Disposition A. Disposition B (structured
`consentDecision` extension) is technically attractive but expands a
clarification ticket into a profile change that competes with the IHE-BALP
elements the IG already references — that conversation is worth having, but
it should be a separate, scoped ticket so it does not delay this fix.
Disposition C ("read the base spec") understates the real implementer
confusion: `AuditEvent.outcome` *severity* and a *consent permit/deny
decision* are easy to conflate, especially in an IG whose entire purpose is
consent enforcement, and the WG should own that clarification rather than
push it back to the base spec. Accept as proposed, and if the WG also wants
the structured-extension path, file a follow-on ticket to evaluate it
against the IHE-BALP carriers without blocking this one.
