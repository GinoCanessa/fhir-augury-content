## Summary

FHIR-54653 questions the need for the IG-defined `fast-auditevent-consent`
search parameter in the Scalable Consent Management IG. The reporter
(John Moehrke) observes that the same query — finding `AuditEvent`
resources whose `entity.what` references a particular `Consent` —
can already be expressed using the FHIR core `entity` search parameter
on `AuditEvent`. Defining a custom, Consent‑scoped search parameter
adds complexity without enabling any new query, since the receiver is
already the one constraining the reference target.

This concern is not isolated. [FHIR-53781](https://jira.hl7.org/browse/FHIR-53781)
("consent search parameter for AuditEvent", reporter Abderrazek Boufahja,
Community-Based Care and Privacy [deprecated]) raised the same point
two weeks earlier, citing the published artifact pages and asking whether
the IG could "rollback to using the native FHIR search parameter
`entity`". A third related ticket,
[FHIR-54652](https://jira.hl7.org/browse/FHIR-54652) ("nonsense sentence
in FASTAuditEventConsent"), flags that the description of the same
artifact ("The Record Disclosure AuditEvent consent reference which is
defined by the AuditEvent entity") is incoherent — a defect that
disappears entirely if the artifact is removed.

The Security WG has already disposed FHIR-54653, FHIR-53781, and
FHIR-54652 as **Persuasive** with the resolution: *"We will remove the
consent search parameter and just mandate the use of the entity search
parameter."* This prep packet documents the rationale and the change
surface so the workgroup can confirm the disposition at review.

## Details

**Description:**

> Why is this search parameter defined? This search can be done with
> FHIR core search parameter `entity`. If you pass a Consent to this
> parameter it will match where entity.what is that value. There is no
> need to add complexity to the search to also make sure that it is a
> Consent resource.

**Resolution Description (recorded on the ticket):**

> We will remove the consent search parameter and just mandate the use
> of the entity search parameter.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the IG-defined `fast-auditevent-consent` SearchParameter from
the Scalable Consent Management IG and update the technical guidance
to use the FHIR core `AuditEvent.entity` search parameter for the
"searching for disclosures" pattern. Concretely:

- Delete the `SearchParameter/fast-auditevent-consent` artifact (the
  resource backing
  `SearchParameter-fast-auditevent-consent.html`) from the IG source.
- Remove or hide its registry entry from the IG's artifact list and
  any CapabilityStatement entry that advertises it on `AuditEvent`.
- Rewrite the *Searching for Disclosures* section of
  `technical.html` to instruct implementers to use
  `GET [base]/AuditEvent?entity={Consent/<id>}` (or the equivalent
  `entity=Consent/<id>`) and to show one canonical example.
- Add a brief migration note for implementers who consumed the
  2026Jan publication, calling out that `_search?consent=...` is
  withdrawn and that `entity=` is the equivalent query.
- Resolves FHIR-54652 transitively, since the artifact whose
  description is incoherent no longer exists.

#### Justification

This is exactly the action recorded in the resolution description for
all three sibling tickets (FHIR-54653, FHIR-53781, FHIR-54652) and
the path the Security WG has already converged on (Persuasive). It
honors a core FHIR design principle — *do not re-define what core
already provides* — and removes a redundant artifact that adds spec
surface area without enabling any new query. `AuditEvent.entity` is
a `Reference` search parameter against `entity.what`, and any
implementation that supports it can already filter to a specific
`Consent` instance via `entity=Consent/<id>`; restricting by resource
type happens implicitly because the reference target is fully
qualified. The two reporters (Boufahja and Moehrke, from independent
implementation contexts) raising the same concern within two weeks
of January publication is strong signal that implementers are seeing
the redundancy too. FHIR-54652's narrative defect is also resolved
"for free", which is the desirable bundled outcome.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `fast-auditevent-consent` defined but **deprecate** it instead
of removing it, and add explicit guidance that `entity` is the
preferred form. Concretely:

- Mark the SearchParameter `status = retired` (or add a deprecation
  banner if `retired` would break existing CapabilityStatements).
- Leave the artifact page in place with a prominent
  "Use `AuditEvent.entity` instead" callout linking to the rewritten
  *Searching for Disclosures* section.
- Update CapabilityStatements that currently list it as a supported
  search parameter on `AuditEvent` to drop it from `SHALL`/`SHOULD`
  expectations but keep it permissible.
- Fix the description defect identified in FHIR-54652 in the
  retained artifact so the deprecated page is at least coherent.

#### Justification

This approach protects implementers who have already coded against
the 2026Jan publication. The IG was published in January 2026 and
this prep is in April 2026 — the deprecation window is short, but
non-zero, and a strict removal could break early-adopter clients
that hard-coded `?consent=`. Deprecation also leaves a trail for
implementers who Google the search-parameter URL: they land on a
page that explicitly redirects them to `entity`, rather than a 404.
The cost is carrying a redundant artifact in the IG and continuing
to maintain its description; given the IG is still pre-normative,
that cost is small. The trade-off vs. Disposition A is preserving
an unnecessary artifact in exchange for a softer migration story.

---

### Disposition C: Decline

#### Proposal

Decline the request. Keep `fast-auditevent-consent` as defined,
fix only the description text raised in FHIR-54652, and document in
the *Searching for Disclosures* section that the Consent-scoped
search parameter is the IG's preferred query form for the
disclosure-search use case.

#### Justification

The defensible argument for declining is *server affordance
discoverability*: a named, IG-scoped SearchParameter advertises in a
server's CapabilityStatement that disclosure search is a first-class
operation in this IG, whereas relying on the generic core `entity`
parameter requires implementers to know the IG-specific convention.
However, this argument is weak because (a) the IG can mandate
`AuditEvent.entity` support in its CapabilityStatement directly,
which provides the same affordance signal, and (b) two independent
reviewers have already pointed out the redundancy. Listed for
completeness so the WG has a no-change option, but not recommended.

---

### Recommendation

**Recommended disposition:** A

The Security WG has already converged on this outcome — the
resolution text on FHIR-54653, FHIR-53781, and FHIR-54652 all read
*"We will remove the consent search parameter and just mandate the
use of the entity search parameter"*, and all three carry resolution
**Persuasive**. The reporters represent two distinct implementation
viewpoints (FHIR-53781 from Abderrazek Boufahja, FHIR-54653 /
FHIR-54652 from John Moehrke) reaching the same conclusion within a
two‑week window of the January publication, which is a clear signal
that the redundancy is hurting implementers, not helping them.
Disposition B's deprecation softening would be defensible if there
were known production deployments depending on `?consent=`, but
none have surfaced in the cross‑reference index, on Zulip, or on
GitHub; the FAST Consent IG is also still pre‑normative, which is
the correct point in the lifecycle to delete redundant artifacts
rather than carry them forward. Disposition C contradicts the
already-recorded WG position. Recommend the WG ratify Disposition A
and bundle the FHIR-54652 description fix into the same removal
edit.
