## Summary

FHIR-53750 raises a conceptual question against the Scalable Consent
Management (FHIR) IG: does "consent", as described in the guide, also
cover the opt-out workflows that some public-health jurisdictions use
for certain reporting flows? Concretely, the reporter (Craig Newman)
asks whether opting out should be modeled as a **withdrawal of an
implied (default) consent**, or as a **distinct de novo consent**
expressing refusal. The ticket is scoped to clarifying the IG's stance
on implied vs. explicit consent and on Permit vs. Deny semantics, rather
than proposing a specific element-level change. It has been triaged as
**Persuasive**, with the resolution noting that the IG will add scope
language explaining that Consent (in this guide's terms) can be either
a Permit or a Deny, and that the broader determination of *implied*
vs. *explicit* consent for a patient is out of scope of this guide.
There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items in the indexed sources for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new subsection to the Scalable Consent Management (FHIR) IG
**Scope** page that explicitly addresses public-health opt-out
workflows. The section would:

1. State that, within this guide, a Consent instance carries a
   permit/deny decision (`Consent.provision.type`), and that an
   "opt-out" from a public-health jurisdiction's default-permit posture
   is represented as a Consent with a Deny provision against the
   relevant purpose-of-use / activity / data category, rather than as
   the absence of a Consent record.
2. Distinguish **implied / default consent** (the jurisdictional
   baseline) from **explicit consent** (a recorded patient decision),
   and note that determining which baseline applies is the
   jurisdiction's responsibility and **out of scope** for this guide.
3. Clarify that "withdrawal" is modeled by superseding the prior
   Consent (using `Consent.status` lifecycle and the existing
   supersedes/replaces linkage), not by a new resource type.

#### Justification

This matches the resolution already captured on the ticket
("Persuasive") and the reporter's intent: the IG today does not say
anything about opt-out semantics, and implementers in public-health
contexts cannot tell whether they should be authoring Consent resources
at all. Aligning the scope statement with the existing `Consent`
resource semantics (Permit/Deny on `provision.type`, status lifecycle
for withdrawal) keeps the guide consistent with the base spec and
avoids inventing new machinery. Treating implied vs. explicit consent
as a jurisdictional/policy concern rather than an IG concern is
defensible because the guide is about *scaling* consent processing, not
about establishing the default policy itself.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) scope-only clarifying text, publish a
short **Use Cases / Patterns** page in the IG that walks through the
opt-out workflow concretely, with one or two example `Consent`
instances:

- Example 1: a jurisdiction's default-permit baseline plus a
  patient-initiated opt-out, modeled as a Consent with
  `provision.type = deny` and a purpose-of-use code for the relevant
  reporting flow.
- Example 2: a subsequent withdrawal of that opt-out, modeled by
  superseding the prior Consent (`Consent.status = inactive` on the
  prior, new active Consent with `relatesTo` / supersedes pointer).

The scope page would still note that establishing the *default*
(implied) consent is out of scope, but the IG would carry worked
examples so implementers do not have to derive the pattern themselves.

#### Justification

A scope-only clarification (Disposition A) tells implementers what is
*not* covered but leaves them to invent the wire-level representation
on their own, which is exactly the situation that produced this
ticket. A worked example anchors the pattern in real `Consent`
instances and reduces divergence between implementations. The cost is
modest (a single narrative page plus 1–2 example resources) and it
does not require any conformance changes, so backward compatibility is
unaffected. Trade-off vs. A: more authoring work and a small ongoing
maintenance burden if the example codes drift.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive / Out of Scope** on the grounds
that opt-out semantics are already representable using the base FHIR
`Consent` resource (Permit/Deny via `provision.type`, lifecycle via
`Consent.status`, supersession via `relatesTo`), and that
jurisdictional default-consent policy is outside the remit of a
scalable-consent processing guide. Make no changes to the IG.

#### Justification

Defensible on the basis that the base `Consent` resource already
provides every mechanism needed to express an opt-out, and that the IG
should not be in the business of encoding jurisdictional policy
defaults. However, the ticket is *asking a clarifying question*, not
proposing a new mechanism, and the WG has already triaged it as
**Persuasive** — declining would leave the documented confusion in
place and would not match the WG's stated direction. Included here for
completeness.

---

### Recommendation

**Recommended disposition:** A (with the worked-example payload from B
folded in as a follow-up if authoring bandwidth allows).

The ticket has already been triaged as Persuasive, and the resolution
text on the ticket itself describes exactly Disposition A: add a scope
section that explains Permit/Deny and clarifies that
implied-vs-explicit consent is the jurisdiction's call, not the IG's.
Disposition A is the smallest change that resolves the reporter's
question, requires no conformance impact on existing implementers, and
keeps the IG aligned with the base `Consent` resource semantics.
Disposition B is strictly additive and is worth pursuing as a
follow-up because a worked opt-out example would prevent this same
question from recurring; it can be tracked as a separate ticket once
the scope text lands. Disposition C is not recommended because it
contradicts the WG's existing triage and leaves implementers without
guidance on a real-world public-health flow.
