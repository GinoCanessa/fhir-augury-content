## Summary

The reporter asks the FHIR Data Segmentation for Privacy (DS4P) IG editors
to add a worked example illustrating one of the potential use cases
referenced by the existing `#implementation` paragraph, which states that
"by defining profiles and maturity models for the security labeling
system, subsets of the security labeling systems can be selected in a way
that is tailored to suit specific application areas and use-cases." The
existing wording promises tailoring via profiles and maturity models, but
no concrete example is provided in the IG. The ticket is scoped to the
DS4P IG narrative — it does not request a new resource, element, or
constraint. The ticket was previously resolved as **Considered for Future
Use** (Connor / Gunter, 3-0-1, 2022-08-29) on the basis that no
qualifying profile-based example existed in the IG yet, and the reporter
accepted that resolution. lynn_laakso then **reverted the resolution and
re-opened the ticket post-STU1 publication** on 2023-04-17, which is why
it is back in the Security work group's queue at status `Triaged` with
`Highest` priority.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a worked example to the DS4P IG `#implementation` section that
illustrates how a profile of `Meta.security` (or a profiled clinical
resource carrying security labels) selects a subset of the HL7
Healthcare Privacy and Security Classification System (HCS) for a
specific application area. Concretely:

1. Author a small profile — e.g., `DS4PSubstanceUseLabeled<Resource>` or
   `DS4PMentalHealthLabeled<Resource>` — that constrains
   `Meta.security` to a named ValueSet drawn from HCS sensitivity,
   confidentiality, obligation, and refrain code systems appropriate to
   that domain (e.g., 42 CFR Part 2 substance-use disclosure).
2. Provide a paired example instance demonstrating the labeled
   resource.
3. Embed both in the `#implementation` page immediately after the
   existing paragraph the reporter quoted, with prose explaining how
   the profile + ValueSet pair "tailors the security labeling system to
   a specific application area and use-case," and how a maturity-model
   tier (e.g., minimum-necessary vs. full HCS) would scale the same
   pattern.

#### Justification

This is the smallest change that makes the existing DS4P narrative
self-contained: today the IG promises tailoring via profiles + maturity
models but never demonstrates it, which is exactly the gap the reporter
flagged and which the original *Considered for Future Use* resolution
acknowledged ("we don't have a current example other than this IG").
Adding the example post-STU1 also matches lynn_laakso's stated reason
for re-opening the ticket — STU1 publication is the natural point at
which downstream profiles begin to appear and an exemplar can be drawn
from real implementer activity rather than invented in the abstract.
Reporter has previously signalled they are willing to accept whatever
the work group decides, so the disposition risk is low.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of authoring a new profile + example purely to populate the IG,
**reference an external profile that already exemplifies the pattern**
(e.g., a Da Vinci, US Core, or 42 CFR Part 2-related IG that constrains
`Meta.security`) and add a short narrative block in the DS4P
`#implementation` page that:

1. Names the external IG / profile.
2. Quotes or links its `Meta.security` constraint and ValueSet binding.
3. Explains in 2–3 sentences how the constraint embodies "tailoring the
   security labeling system to a specific application area / use case"
   and where that profile sits on the maturity continuum.

If no suitable external profile exists at the time the ticket is
worked, reduce the change to a non-normative editorial note pointing
implementers at the HCS code systems and listing the design choices a
profile author must make (sensitivity vocabulary, confidentiality
floor, obligation/refrain policies, conformance level), without
authoring a new profile.

#### Justification

Inventing a profile solely for narrative purposes risks creating a
"DS4P-blessed" example that downstream implementers feel obliged to
follow, which is heavier weight than the editorial gap the reporter
identified. Pointing at an existing real-world profile (or, failing
that, a structured editorial checklist) addresses the same pedagogical
need with lower normative footprint and less long-term maintenance for
the Security WG. This also avoids coupling the DS4P IG's release
cadence to the maturity of any single example domain (e.g.,
substance-use vs. mental-health vs. genomic).

---

### Disposition C: Decline

#### Proposal

Re-affirm the prior **Considered for Future Use** disposition: leave
the `#implementation` paragraph as-is, do not author a worked example
in this ballot cycle, and close the ticket with a note that an example
will be added once a qualifying downstream profile is available to
reference.

#### Justification

The reporter explicitly accepted the earlier *Considered for Future
Use* resolution ("I accept this resolution. No need to discuss further.
Feel free to close out the ticket"), and the original work-group vote
(3-0-1) was unanimous-in-favor. The ticket was re-opened on procedural
grounds ("post STU1 publication") rather than because new use-case
material has surfaced — none of the comments after re-open identify a
specific profile or maturity-model artifact that would now anchor the
example. Declining is defensible if the Security WG still does not have
a concrete profile to point at, since synthesizing one purely to fill
narrative space is out of proportion to the originally-acknowledged
editorial nature of the gap.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between the reviewer's legitimate
editorial concern and the original work-group concern that motivated
*Considered for Future Use*. Disposition A has merit but commits the
Security WG to authoring (and maintaining) a brand-new profile purely
for illustration, which is a larger lift than the ticket strictly
demands and risks elevating one application area (substance use, mental
health, etc.) over others by example alone. Disposition C is the
lowest-effort path but ignores the reason the ticket was re-opened
post-STU1 — STU1 should make it possible to point at *some* concrete
artifact, even if only a community profile, and leaving the gap a
second time invites the same comment again next cycle. B keeps the
change editorial, is bounded in scope, leverages whatever real
profiling activity now exists in the ecosystem, and gracefully
degrades to a structured editorial checklist if no suitable external
profile is available — a clear improvement over the current `#implementation`
paragraph either way.
