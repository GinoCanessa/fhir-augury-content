## Summary

The reporter (John Moehrke) asks why `Consent.identifier` is constrained to a
maximum cardinality of 1 and argues that this is too restrictive — `Consent`
resources should be allowed to carry more than one business identifier. The
ticket is filed against the Scalable Consent Management (FHIR) specification
and has been triaged as Persuasive with a stated resolution to relax the
cardinality of `Consent.identifier` from `0..1` to `0..*` ("1..*" in the
resolution wording is treated here as the intended `0..*` form, since the
existing element is optional). No other Jira tickets, Zulip threads, or
GitHub items were surfaced as cross-references; the only cross-reference is
to the FHIR artifact `Consent.identifier` itself.

## Details

**Description:**

Consent.identifier must be allowed to hold more than one identifier. What is
the justification for max 1?

**Resolution Description:**

We will change identifier to be 1..* (i.e., relax the upper bound on
`Consent.identifier` so that more than one identifier is permitted).

**Comments:**

No comments are recorded on the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the cardinality of `Consent.identifier` from `0..1` to `0..*` in the
base `Consent` resource for the Scalable Consent Management (FHIR) release.
Concretely: change the `max` on the `identifier` element of the `Consent`
StructureDefinition (and corresponding XML/JSON examples / generated
artifacts) from `1` to `*`. No change to the element's datatype
(`Identifier`), short description, definition, or requirements text is
required beyond noting that multiple business identifiers are permitted.
This matches the stated resolution on the ticket.

#### Justification

Aligning `Consent.identifier` with `0..*` is consistent with the broad FHIR
convention for the `identifier` element on most DomainResources (e.g.,
`Patient.identifier`, `Observation.identifier`, `DocumentReference.identifier`,
`Encounter.identifier` are all `0..*`). A `0..1` cap on `Consent` is an
outlier that prevents legitimate scenarios where the same consent instance
is known by multiple business identifiers — for example, an internal
consent-management system identifier plus a payer-assigned or
exchange-assigned identifier, or a paper-record identifier paired with an
electronic identifier during transition. The change is backward-compatible
at the wire level: every existing instance with zero or one identifier
remains valid against the new `0..*` constraint, and no implementer needs
to change code to keep working. The Security WG has already triaged the
issue as Persuasive, signalling agreement with the underlying need.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the base `Consent.identifier` element at `0..1` but add a standard
extension (e.g., `consent-additionalIdentifier` with type `Identifier`,
cardinality `0..*`) to carry secondary business identifiers, and document
the extension in the Scalable Consent Management IG. Implementers needing
multiple identifiers would populate `Consent.identifier` with the primary
identifier and use the extension for any additional ones.

#### Justification

This option preserves strict backward compatibility for tooling and
profiles that have hard-coded the assumption of at most one
`Consent.identifier` (validators, mappers, some FHIR Path queries, search
parameter result handling, and generated client classes that may expose a
scalar property rather than a collection). It also avoids a base-resource
schema change just before / during a ballot cycle. The trade-off is that
implementers must know about and use a non-standard-shaped extension for
what is conceptually a normal business-identifier scenario, which raises
the implementation burden and partitions data across two locations
(element + extension) that downstream systems must merge.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep `Consent.identifier` at `0..1`, on the basis
that the resource owner intends a single canonical business identifier per
`Consent` instance, and that systems needing to track additional
identifiers should manage that mapping outside the resource (e.g., in a
local cross-reference table or via the containing `Bundle`'s
`identifier` / `entry.fullUrl`).

#### Justification

This position is defensible only if the Security WG can articulate a
concrete reason `Consent` is intentionally different from peer resources
that allow `0..*` identifiers. No such rationale is recorded in the ticket
or in any cross-referenced material, and the WG has already marked the
issue Persuasive, which directly contradicts a Decline. Declining would
also leave implementers without a clean, in-resource mechanism to express
the very common pattern of "this consent is known by more than one id"
(e.g., source-system id + exchange-assigned id), pushing them toward
out-of-band conventions or the extension in Disposition B.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been triaged as Persuasive with an explicit
resolution to widen `Consent.identifier` to allow more than one identifier,
which is exactly Disposition A. The change is consistent with the
near-universal FHIR convention that `Resource.identifier` is `0..*`, it
removes an unjustified outlier in the Consent resource, and it is
wire-compatible — every instance valid under the old `0..1` constraint
remains valid under `0..*`, so no existing data or implementation needs
to change. Disposition B (extension-based workaround) adds permanent
implementation burden to avoid a one-time, backward-compatible schema
relaxation, and Disposition C contradicts both the WG's own triage outcome
and the established pattern across the rest of the spec. The Security WG
should ratify Disposition A and route the change to the HL7/fhir authoring
team for the Consent StructureDefinition update in the next publication.
