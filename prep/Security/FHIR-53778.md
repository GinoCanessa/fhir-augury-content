## Summary

The ticket reports a defect in the FAST Scalable Consent Management IG's
`fast-consent-organizationId` SearchParameter. The reporter observes that the
SearchParameter's expression does not always resolve to an organization
identifier — the Consent grantee can also be, for example, a `HealthcareService`,
so naming and typing the parameter as an "organizationId" is misleading and
incorrect. The reporter further notes that this parameter is not listed among
the Consent search parameters enumerated in the IG's "Searching for Consents"
section (technical.html), which adds implementation complexity. The recommended
fix is to drop the cross-resource `organizationId` parameter and instead reuse
the standard Consent search parameters from the "Searching for Consent" section,
suggesting that the underlying need can be met by enhancing search via
`controller:identifier` and `manager:identifier`. The ticket has been resolved
as **Persuasive**, and the resolution note indicates the work group will remove
both the organization-ID and patient-ID search parameters (FHIR-I has indicated
that defining search parameters that cross resource boundaries is not allowed)
and will instead document that searching for Consents by patient identifier and
the various organization identifiers is required, without defining an improper
SearchParameter to do so.

No cross-referenced Jira tickets were surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify the `fast-consent-organizationId` SearchParameter so that, instead of
being defined as a cross-resource SearchParameter resolving to an
`Organization.identifier`, the IG provides guidance (in the "Searching for
Consents" section of `technical.html`) on using the existing Consent reference
search parameters with `:identifier` modifiers — specifically
`Consent:controller:identifier` and `Consent:manager:identifier` — to search
for Consents by organization identifier. Update the SearchParameter page (or
remove the artifact entirely if it cannot be reframed) and add a worked example
under "Searching for Consents" showing identifier-based search against the
controller and manager grantee references.

#### Justification

This matches the reporter's specific recommendation and directly removes the
incorrect assumption that the Consent grantee always resolves to
`Organization`. Reusing `controller:identifier` / `manager:identifier`
leverages well-understood R4/R5 search semantics, avoids inventing an IG-local
SearchParameter whose expression does not match the actual reference type
options (Organization, HealthcareService, etc.), and keeps the IG's search
guidance consolidated in the "Searching for Consents" section that
implementers already consult.

---

### Disposition B: Alternative Approach (Resolution as Recorded — Remove the Cross-Resource SearchParameters Entirely)

#### Proposal

Remove `fast-consent-organizationId` and the analogous patient-ID
SearchParameter from the IG. In `technical.html#searching-for-consents`, add
normative text stating that servers SHALL support searching for Consents by
patient identifier and by the various organization identifiers (controller,
manager, and any other in-scope grantee role), and show this is achieved via
the standard FHIR `:identifier` modifier on the existing reference search
parameters (`patient:identifier`, `controller:identifier`,
`manager:identifier`). Do not declare an IG-local SearchParameter whose
expression spans resource types.

#### Justification

This is the resolution already captured on the ticket, and it is grounded in
explicit FHIR-I direction that SearchParameters defining expressions that
cross resource boundaries are not allowed. Removing the offending parameters
eliminates the type-mismatch bug the reporter identified (grantee may be
`HealthcareService`, not `Organization`) without trying to salvage an artifact
that is structurally incompatible with FHIR search semantics. It still meets
the underlying functional need by mandating identifier-based search using
modifiers that every conformant FHIR server already supports, so implementers
get a clear contract without having to special-case an IG-local parameter.

---

### Disposition C: Decline

#### Proposal

Leave `fast-consent-organizationId` (and the patient-ID counterpart) as
published, on the grounds that the expression — though imperfect — communicates
intent and that implementers can interpret it as "where the grantee resolves
to an Organization." Document the limitation as a known issue in the IG and
defer any structural change to a later ballot cycle.

#### Justification

A defensible declination would rest on minimizing churn between ballot
versions and avoiding removal of a published artifact that downstream
implementers may already be coding against. However, this rationale is weak
here because the SearchParameter is structurally non-conformant per FHIR-I
guidance (cross-resource expression) and incorrect for the actual grantee
typing, so leaving it in place would knowingly publish a defective artifact.

---

### Recommendation

**Recommended disposition:** B

Disposition B aligns with the work group's already-recorded "Persuasive"
resolution and with explicit FHIR-I direction that cross-resource
SearchParameters are not allowed — that is a hard constraint, not a
preference, so Disposition C is not viable. Disposition A is close in spirit
to the reporter's suggestion, but it stops short of removing the
structurally-defective artifact; B goes further by deleting both the
organization-ID and patient-ID SearchParameters and replacing them with
normative search-conformance text in `technical.html#searching-for-consents`
that points implementers at the standard `:identifier` modifier on
`controller`, `manager`, and `patient`. This preserves the functional capability
the reporter wants (identifier-based search, including for
`HealthcareService`-typed grantees via `controller`/`manager`), keeps the IG
inside FHIR-I's rules, and consolidates the search guidance in the one section
implementers already use as the contract for Consent search.
