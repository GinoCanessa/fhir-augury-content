## Summary

The reporter (John Moehrke) observes that the **Scalable Consent Management
(FHIR)** specification points readers at a "Testing Guide" that is **not
hosted on an HL7 site**. Because the guide lives outside HL7's governance
boundary, it is unclear under whose authority the testing guidance is
maintained, what its change-control process is, and how implementers
should treat it relative to normative HL7 content. The ticket presumes
that the guide will eventually move under HL7 governance and suggests
that — until that happens — the specification should at least make the
non-HL7 governance status explicit. The ticket has been triaged as
**Persuasive**, with an agreed-on resolution to clarify the governance
status of the Testing Guide reference and update the link once the guide
is hosted on an HL7-governed site.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the FhirAugury index for `FHIR-54690`, so the request stands
on its own.

## Details

**Description:**
The Testing Guide pointed to is not on HL7 site. Therefore the
governance is unclear. I presume this guide will eventually be HL7, that
might be all that needed to be said until it is HL7 property.

**Resolution Description (already recorded on the ticket):**
We will update the reference to the Testing Guide to clearly indicate
its current governance status. If it has not yet been published under
HL7 governance at the time of publication of this version, we will add a
note stating that it is a working draft pending HL7 publication, and we
will update the link once it is hosted on an HL7-governed site.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **Scalable Consent Management (FHIR) IG** wherever it links to
the external Testing Guide to:

1. Add an inline editorial note immediately adjacent to the link stating
   that the Testing Guide is **not currently HL7-governed** and is being
   maintained externally as a **working draft pending HL7 publication**.
2. Mark the link as informative (not normative) and exclude it from any
   conformance language ("SHALL implement…", etc.).
3. Add a publication-time checklist item for the Security WG: before each
   ballot/publication, re-verify whether the guide has moved to an
   HL7-governed location and, if so, replace the URL and remove the
   governance disclaimer.

This matches the resolution already negotiated on the ticket
("Persuasive").

#### Justification

The change is purely editorial, addresses the reporter's stated concern
exactly, and has zero impact on conformance, interoperability, or the
shape of any FHIR artifact. Calling out non-HL7 governance is consistent
with how HL7 IGs treat other external/working references and removes the
ambiguity that prompted the ticket. The publication-time checklist item
hardens the fix against link rot once the guide does move to HL7.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than embedding a per-link disclaimer in the IG body, **remove the
direct link to the external Testing Guide** from the normative-adjacent
sections of the IG and instead point to a dedicated **"Testing"
sub-page** within the IG. That sub-page would:

- Describe what the Testing Guide is and what it covers, in HL7's own
  voice.
- Note explicitly that the guide is currently maintained outside HL7
  and is a working draft.
- Provide the external link in a clearly-labeled "External resources"
  section.

When the Testing Guide moves to HL7 governance, the sub-page is updated
in place (URL swap + remove the disclaimer); no other IG sections need
to change.

#### Justification

Centralising the external reference in one IG-owned page reduces the
risk of stale or inconsistent disclaimers scattered through the spec and
gives the Security WG a single, reviewable surface for the governance
caveat. It also makes the boundary between HL7-governed content and the
external guide visually obvious to implementers, which is the underlying
concern the reporter raised. The trade-off vs. Disposition A is a
slightly larger editorial footprint now, in exchange for lower
maintenance cost later.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the link's target is itself
expected to migrate under HL7 governance, and that a generic
"References" section disclaimer (already present in most HL7 IGs)
adequately covers the case where third-party material is cited. Take no
further action beyond standard publication-time link review.

#### Justification

If the Testing Guide is genuinely on a path to becoming HL7 property,
adding ticket-specific disclaimers risks creating editorial debt that
must be cleaned up shortly after. Most HL7 IGs already include boilerplate
acknowledging that external links are informative and may change; relying
on that boilerplate avoids one-off language for a transient situation.

---

### Recommendation

**Recommended disposition:** A

The reporter's concern is concrete and editorial, the resolution has
already been negotiated on the ticket as **Persuasive**, and Disposition A
implements that resolution exactly with the smallest possible footprint.
Disposition B is a reasonable refinement if the Security WG finds that
the Testing Guide is referenced from many places in the IG (in which
case centralising on a single "Testing" sub-page becomes attractive),
but it is a strictly larger edit than the ticket asks for. Disposition C
under-serves the reporter — relying on generic boilerplate is exactly
what produced the ambiguity in the first place. Recommend proceeding
with A, and upgrading to B only if the Security WG identifies more than
two or three distinct IG locations linking to the external guide.
