## Summary

FHIR-55945 is a follow-up to [FHIR-51052](https://jira.hl7.org/browse/FHIR-51052), which
established that URIs appearing in JWT claims (`iss`, `sub`, `aud`) and in
certificate Subject Alternative Name (SAN) values should be normalized at the
producer side, because comparison-time normalization is forbidden. The reporter
(Joseph M. Lamy) raises a specific edge case the parent ticket's resolution did
not fully address: in the FHIR-Backed FAST Security Discovery flow, the
certificate used to sign the `signed_metadata` JWT must include a SAN URI
matching the server's FHIR Base URL. Base FHIR explicitly says the FHIR base URL
does **not** carry a trailing slash, but RFC 3986 §6.2.3 and RFC 7230's URI
comparison rules normalize an `http(s)` URI with an empty path to a path of
`"/"`. The two rules are in direct tension for this exact comparison, and the
specification needs to clarify which form is required (or that both are
acceptable) for the SAN URI used in `signed_metadata` certificates — and whether
the same clarification should extend to certificates used to sign other JWTs in
FAST Security (e.g., UDAP-style client authentication).

### [FHIR-51052: Please clarify case-sensitivity / normalization for URI comparisons in iss, sub, aud, SAN](https://jira.hl7.org/browse/FHIR-51052)

The originating ticket. Status **Published**, resolution **Persuasive with
Modification** (resolved 2025-07-22), filed against the now-deprecated US UDAP
Security spec, labels `STU2, block-vote-4`. The agreed resolution was, "subject
to editor discretion, we will add guidance that URIs in JWT and certificates
should be normalized by the producer." Comments from the reporter flag that the
example "Paths normalize to end with a slash" is overstated (it should be "Some
paths..."), and explicitly call out RFC 3986 §6.2.3 / §2.4: an `http(s)` URI
that uses the generic syntax for authority with an empty path should be
normalized to a path of `"/"`. FHIR-55945 is the carve-out request for the case
where producer-side normalization (adding `/`) collides with the base FHIR rule
that the FHIR base URL has no trailing slash.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FAST Security IG (the successor to US UDAP Security), add a normative
clarification in the Discovery section covering the `signed_metadata` certificate
SAN URI. Specifically:

- State that the SAN URI **MUST** match the server's FHIR Base URL exactly as
  published in the Discovery document (i.e., **without** a trailing slash, per
  base FHIR), regardless of the RFC 3986 §6.2.3 / §2.4 normalization that would
  otherwise add `/` to a URI with an authority and empty path.
- Add a non-normative note explaining the conflict between the FHIR base URL
  rule and the generic URI normalization rule, and clarifying that this is an
  intentional exception to the producer-side normalization guidance from
  FHIR-51052.
- Apply the same exception to **any** certificate SAN URI in FAST Security that
  is required to match a FHIR Base URL (e.g., certificates used to sign other
  JWTs whose `iss` is the server's FHIR base). For SAN URIs that are *not* FHIR
  base URLs (e.g., client identifiers, audience URIs that point at token
  endpoints), the FHIR-51052 guidance to normalize per RFC 3986 still applies
  unchanged.

#### Justification

This keeps the FHIR base URL identifier stable and consistent with the rest of
the FHIR ecosystem (CapabilityStatement.implementation.url, conformance URLs,
service base URLs in implementation guides), all of which omit the trailing
slash. Forcing a `/`-suffixed form into the SAN would create a mismatch between
the certificate identifier and the literal string a client would derive from the
server's published base URL, defeating the comparison purpose. Since RFC 3986
§6.2.3 explicitly allows producers to perform syntax-based normalization but
does not require it for hierarchical paths whose semantics are known, scoping
the exception to the FHIR-base-URL-as-SAN case is consistent with both RFCs and
with the intent of FHIR-51052.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of carving out an exception, redefine the comparison rule for this one
case: state that when comparing the `signed_metadata` certificate SAN URI to the
server's FHIR Base URL, **both** the bare-authority form (no trailing slash) and
the RFC 3986-normalized form (with `/`) MUST be treated as equivalent. Producers
MAY emit either form, and validators MUST accept either. Document a SHOULD
recommending the bare-authority form (matching the FHIR base URL) for new
certificates, to avoid future churn. Apply the same dual-acceptance rule to any
other FAST Security cert SAN that mirrors a FHIR base URL.

#### Justification

This preserves the spirit of FHIR-51052 (normalization at production is
permitted and recommended) while accommodating already-issued certificates and
existing implementations that may have followed RFC 3986 strictly. It avoids
breaking deployed UDAP/FAST clients that currently match SANs against the
slash-suffixed form, and it sidesteps the philosophical question of which RFC
"wins" by saying both are acceptable. The downside is increased validator
complexity and a small interop risk if implementers read only the SHOULD and
ignore the MUST-accept-either side.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive**: take no action in the spec. Rely on the FHIR-51052
resolution (producer-side normalization per RFC 3986) as written, accepting that
SAN URIs in `signed_metadata` certificates will carry the `/`-suffixed form even
though the FHIR base URL does not. Implementers can be expected to handle the
single-character difference via the FHIR-51052 normalization guidance applied at
the validator side as well.

#### Justification

The conflict is essentially a one-character cosmetic difference between two
representations of the same resource, and RFC 3986's guidance is unambiguous for
URIs with empty paths. Adding a special case risks confusing implementers and
opens the door to other "but FHIR says…" carve-outs every time a FHIR
identifier rule differs from a generic URI normalization rule. However, this is
the weakest position because the parent ticket explicitly states that
comparison-side normalization is forbidden — meaning a strict validator that
sees `https://example.org/fhir` in the published metadata and `https://example.org/fhir/`
(or vice versa) in the SAN will *fail* the match, which is exactly the
interoperability problem FHIR-51052 sought to prevent.

---

### Recommendation

**Recommended disposition:** A

The collision the reporter identifies is real and stems from FHIR's own
deliberate choice that the base URL has no trailing slash — a choice that
predates and overrides generic URI normalization for the purpose of FHIR
identity comparison. Disposition A keeps the SAN URI literally identical to the
published FHIR base URL, which is the form every FHIR client already computes
and stores, and avoids forcing every validator to maintain a parallel
slash-tolerant comparison path that would itself violate the FHIR-51052
prohibition on comparison-time normalization. Disposition B is a defensible
fallback if the workgroup learns that significant deployed certificates already
carry the slash-suffixed form (in which case dual-acceptance buys a transition
window), but absent that evidence the cleaner spec text in A is preferable.
Disposition C should be rejected because it leaves the very interop break that
motivated FHIR-51052 in place for exactly the certificate that anchors the
entire Discovery trust chain.
