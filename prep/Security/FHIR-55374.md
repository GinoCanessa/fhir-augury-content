## Summary

The ticket questions the scoping of disclosure recording in the Scalable
Consent Management (FHIR) IG. The IG currently requires that, when a
system accesses a Consent instance, it invoke a `$record-disclosure`
operation. Because Consent instances are propagated and accessed across
multiple cooperating systems, the reporter (Joseph M. Lamy) asks which
system(s) the disclosing system is required to record disclosures
against — i.e., where the resulting audit/disclosure records must
land, and how that obligation is partitioned across the systems
holding or serving the Consent.

No related Jira tickets, Zulip threads, or GitHub items were surfaced
via cross-references or search for `FHIR-55374`, so the request is
addressed in isolation by the Security work group. The ticket has
already been Triaged with a resolution of "Persuasive with
Modification".

## Details

**Description:**

> There is a requirement to use the Record Disclosure Operation when a
> system accesses a Consent instance, but given systems are accessing
> Consent instances propagated across multiple systems, which system(s)
> is the disclosing system required to record disclosures to?

**Resolution Description (per Jira):**

> We are removing the Record Disclosure operation and relying on RESTful
> actions for the creation and searching of audit events. We will be
> updating the section on record disclosure to indicate how audit events
> are recorded and retrieved.

## Proposed Dispositions

The cross-reference and search calls did not surface any GitHub repo
context for this ticket. The Scalable Consent Management (FHIR) IG repo
is not currently registered as an Augury-tracked GitHub source, so the
dispositions below name the IG and its Record Disclosure section
generically rather than citing specific authoring paths from a
briefing. Any disposition the work group accepts should still be
implemented against that IG's authoring root (typically `input/pages/`
narrative pages plus any `OperationDefinition` / `CapabilityStatement`
artifacts under `input/resources/`).

### Disposition A: Accept As Requested

#### Proposal

Clarify within the Scalable Consent Management IG's "Record Disclosure"
narrative which system(s) MUST invoke `$record-disclosure` when a
Consent instance is accessed across propagated copies. Specifically:

- Define the **disclosing system** as the system that returned the
  Consent instance (or PHI gated by it) to the requester — i.e., the
  system that physically performed the disclosure.
- Require that the disclosing system record the disclosure against
  **its own** AuditEvent / disclosure log, not against upstream
  systems that originated or propagated the Consent.
- Add a SHOULD that, where the disclosing system also notifies the
  Consent's authoring/source system, the recorded disclosure carries
  the source Consent's canonical identifier so that downstream
  reconciliation is possible.
- Keep the `$record-disclosure` operation in place; tighten its
  `OperationDefinition` documentation to state the recording-system
  obligation explicitly.

#### Justification

This is the literal reading of the ticket: it asks "which system(s)?",
not "should the operation exist?". Pinning the obligation to the
disclosing system aligns with the broader audit principle that the
party performing an action records it. It avoids the multi-write
ambiguity (every propagator recording every other propagator's
accesses) that the reporter implicitly flags. It also preserves the
existing operation surface, minimizing churn for any implementer who
has already built against `$record-disclosure`.

---

### Disposition B: Alternative Approach (matches the recorded resolution)

#### Proposal

Remove the `$record-disclosure` operation from the Scalable Consent
Management IG and rely on standard RESTful interactions against
`AuditEvent` for both creation (POST/PUT) and discovery (search) of
disclosure records. Then rewrite the "Record Disclosure" section of
the IG to:

- Describe the disclosing system as the system that served the
  Consent / served PHI under the Consent.
- Specify the AuditEvent profile (or constraints) implementers should
  conform to when recording a disclosure.
- Define the search parameters and access pattern other parties
  (auditors, consent authors, patients) should use to retrieve
  disclosure records — including how to scope a search by Consent
  identifier / subject.
- Note that each system is responsible for its own audit store; there
  is no obligation to write back into upstream systems' audit stores.

This is the disposition recorded in the Jira resolution
("Persuasive with Modification").

#### Justification

Replacing a bespoke operation with RESTful AuditEvent interactions
keeps the IG aligned with the FHIR design principle of preferring
existing resources and interactions over new operations when the
operation does not provide additional semantics. It collapses two
questions into one: the operation question goes away, and the
"which system?" question becomes a clear narrative rule about who
owns the AuditEvent. It is also consistent with broader Security WG
direction that audit data flow over standard REST + AuditEvent rather
than verb-style operations. The trade-off vs. Disposition A is some
churn for early adopters, but the IG is still in
Scalable-Consent-Management-level maturity, so the cost is contained.

---

### Disposition C: Decline

#### Proposal

Decline to change the IG. Treat the question as already answered by the
existing audit/disclosure pattern: every system that performs an action
on a Consent (or on PHI gated by a Consent) records that action in its
own audit store using whatever mechanism the IG already specifies,
without further normative guidance on multi-system scope.

#### Justification

This would be defensible only if the work group concluded the existing
text was already unambiguous on its face. It is the lowest-effort
outcome but does not address the reporter's concern — the ticket
explicitly says the current text leaves the multi-system scope
unclear, so declining without clarification will likely re-surface as
a follow-up ticket.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because it both (a) answers the reporter's
"which system?" question by clearly assigning recording responsibility
to the disclosing system, and (b) removes a bespoke operation in
favor of RESTful AuditEvent interactions, which is the direction the
Security WG has been moving for audit and disclosure data. It matches
the resolution already entered on the ticket ("Persuasive with
Modification" — "removing the Record Disclosure operation and relying
on RESTful actions for the creation and searching of audit events"),
and gives implementers a single, conventional place to write and
search disclosure records without introducing IG-specific verbs.
Disposition A is a viable fallback if the work group decides the
operation surface is worth preserving for clients that have already
adopted it; Disposition C should be rejected because the reporter's
ambiguity concern is real.
