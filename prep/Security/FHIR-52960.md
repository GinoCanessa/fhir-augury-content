## Summary

The reporter asks whether the SSRAA (Security for Scalable Registration,
Authentication, and Authorization) IG should require authorization
servers to support HTTP `POST` to the authorization endpoint, rather
than only the OAuth 2.0 baseline of "MUST support GET, MAY support
POST." Section 4.1 of SSRAA currently delegates authorization-request
construction to RFC 6749, and Section 7 does not strengthen that
behaviour. By contrast, SMART App Launch (`app-launch.html#request-4`)
already requires `SHALL support GET and POST` at the authorization
endpoint and lets clients pick either method. Because many deployed
authorization endpoints already comply with SMART, the reporter
observes that SSRAA clients still cannot reliably depend on `POST`
support, and asks whether SSRAA should be tightened to match SMART.
The single comment cross-links to [FHIR-30588](https://jira.hl7.org/browse/FHIR-30588),
the persuasive SMART ticket that introduced POST-based authorization to
limit URL size; that history is the precedent being invoked here.

## Details

**Description:**

> Section 4.1 leaves the authorization request construction to RFC 6749,
> as well as requirements in Section 7.
>
> RFC 6749 says "The authorization server MUST support the use of the
> HTTP 'GET' method [RFC2616] for the authorization endpoint and **MAY
> support the use of the 'POST'** method as well", and section 7 does
> not modify this.
>
> Comparatively, [SMART requires](https://hl7.org/fhir/smart-app-launch/app-launch.html#request-4)
> servers support a POST authorize request: "Authorization **Servers
> SHALL support the use of the HTTP GET and POST** methods at the
> Authorization Endpoint... Clients SHALL use either the HTTP GET or
> the HTTP POST method".
>
> Many authorization endpoints already comply with SMART, yet in SSRAA
> clients cannot reliably assume support for POSTed authorization
> requests.
>
> Should authorization endpoints be required to support POST authorize
> requests in SSRAA?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the SSRAA IG so that conformant authorization servers **SHALL**
support HTTP `POST` at the authorization endpoint, matching SMART App
Launch's "Request" section. Concretely, update **Section 4.1
(Authorization Request)** of the SSRAA IG to add a normative
sentence such as:

> Authorization servers SHALL support the use of both the HTTP `GET`
> and HTTP `POST` methods at the authorization endpoint. When `POST`
> is used, request parameters SHALL be encoded in the request body
> using `application/x-www-form-urlencoded`, per RFC 6749 §3.1 and
> consistent with SMART App Launch.

Add a corresponding client-side statement: clients SHALL use either
GET or POST and MAY choose POST when expected request size (e.g.,
many granular scopes, JAR/`request` objects, or long state values)
risks exceeding URL length limits. Cross-reference Section 7 so that
no Section 7 hardening contradicts the new requirement (e.g., logging
and TLS guidance applies equally to POST bodies).

#### Justification

SSRAA already targets the same FHIR-on-OAuth ecosystem as SMART, and
the SMART community resolved the URL-length problem by mandating POST
support (FHIR-30588, persuasive). Implementations that already meet
SMART will incur zero additional work, and clients with long
authorization requests (granular scopes, request objects, multiple
`launch/*` scopes) gain a reliable transport. Aligning with SMART
also avoids a foot-gun where an SSRAA-only server would fail SMART
clients by silently rejecting POST.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than upgrading `MAY` to `SHALL` for `POST` at the legacy
authorization endpoint, require SSRAA authorization servers to
support **Pushed Authorization Requests (PAR, RFC 9126)** and
recommend (`SHOULD`) — but do not mandate — POST at the authorization
endpoint. Update Section 4.1 to:

1. Add a normative requirement that SSRAA servers SHALL implement the
   PAR endpoint and advertise `pushed_authorization_request_endpoint`
   in their authorization-server metadata.
2. Mark POST at the authorization endpoint as `SHOULD` for
   backward-compatibility with SMART clients.
3. Recommend clients use PAR when request size or
   confidentiality/integrity of request parameters matters.

#### Justification

PAR was designed precisely to solve the long-URL and
parameter-tampering problems that motivated SMART's POST mandate, and
it is the direction the broader OAuth 2.1 / FAPI ecosystem is moving.
SSRAA is positioned as the more security-forward profile relative to
SMART, so leveling up to PAR rather than re-litigating the GET-vs-POST
question is more consistent with that positioning. The trade-off vs.
Disposition A is higher implementation burden for servers that have
not yet adopted PAR, balanced against a cleaner long-term story and
better auditability of authorization requests.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep Section 4.1's delegation to RFC 6749 as
written (`MUST` GET, `MAY` POST). Optionally add a non-normative
note pointing implementers at SMART App Launch's stronger requirement
for ecosystems that need it.

#### Justification

SSRAA's stated layering is on top of OAuth 2.0 / OAuth 2.1; deviating
from RFC 6749's defaults on a per-method basis sets a precedent for
re-stating baseline OAuth semantics inside SSRAA, which the IG has
otherwise tried to avoid. Servers that need to interoperate with
SMART are already required to support POST by the SMART IG; SSRAA
clients targeting those servers can negotiate accordingly. Forcing
POST support on every SSRAA server (including non-SMART deployments
such as backend-services-only or B2B trust frameworks where request
size is not an issue) imposes work without a clear use case.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the smallest, lowest-risk change that closes the
real interoperability gap the reporter raises: SSRAA clients today
cannot rely on POST even when the underlying server already supports
it for SMART. Disposition B (PAR) is genuinely better long-term, but
it is a substantially larger normative change, will need its own
discovery / metadata work, and almost certainly merits a separate
ticket and ballot cycle rather than being folded in here. Disposition
C is hard to defend given that the SMART workgroup has already
resolved the same question persuasively (FHIR-30588) and that the
delta for any SMART-compliant server is zero. Recommend adopting A
now and tracking PAR adoption (Disposition B) as a follow-on enhancement.
