## Summary

FHIR-54652 reports that the description text on the FAST `fast-auditevent-consent`
SearchParameter in the Scalable Consent Management IG is incoherent. The
offending sentence — "The Record Disclosure AuditEvent consent reference which is
defined by the AuditEvent entity." — reads as nonsense and does not explain what
the search parameter matches or why an implementer would use it. The ticket is
tightly coupled to two sibling change requests against the same artifact:
[FHIR-54653](https://jira.hl7.org/browse/FHIR-54653) questions why the custom
`fast-auditevent-consent` search parameter exists at all given that native
AuditEvent `entity` search already covers the use case, and
[FHIR-53781](https://jira.hl7.org/browse/FHIR-53781) asks whether the IG can
roll back to using the native `entity` parameter. The workgroup resolution text
for FHIR-54652 notes that, rather than re-wording the description, the search
parameter itself is being removed in favour of directing implementers to the
core FHIR `entity` search parameter — making the description defect moot.

## Details

**Description:**

> The description is nonsense. "The Record Disclosure AuditEvent consent
> reference which is defined by the AuditEvent entity."

**Comments:**

None recorded on the ticket.

**Resolution (Persuasive):**

The description "The Record Disclosure AuditEvent consent reference which is
defined by the AuditEvent entity" is not meaningful. As noted in related
issues [FHIR-54653](https://jira.hl7.org/browse/FHIR-54653) and
[FHIR-53781](https://jira.hl7.org/browse/FHIR-53781), this custom search
parameter is being removed in favor of directing implementers to use the
native FHIR `entity` search parameter. The description problem will therefore
be resolved by removing the search parameter artifact entirely.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rewrite the `description` element on
`SearchParameter-fast-auditevent-consent` so that it clearly states what the
parameter matches. A concrete replacement would be: "Search AuditEvents whose
`entity.what` references a specific Consent resource. Used to find Record
Disclosure AuditEvents associated with a given patient consent." Leave the
expression and target resources unchanged. Apply the edit in the Scalable
Consent Management IG source and regenerate the published SearchParameter
page.

#### Justification

This literally addresses the defect reported: the current sentence is
grammatically broken and does not describe the parameter's behaviour. A
corrected description keeps the artifact in place for any implementer
already relying on the custom parameter name, and avoids a breaking change to
published IG artifacts mid-cycle. However, it ignores the overlap with core
`entity`-based search raised in FHIR-54653 and FHIR-53781.

---

### Disposition B: Alternative Approach

#### Proposal

Remove the `fast-auditevent-consent` SearchParameter artifact from the
Scalable Consent Management IG entirely, and update the "Searching for
Disclosures" section of `technical.html` to direct implementers to use the
native FHIR AuditEvent `entity` search parameter with a Consent reference
(e.g., `GET AuditEvent?entity=Consent/{id}`). Add a short migration note
stating that `fast-auditevent-consent` has been withdrawn because it was
functionally equivalent to the core `entity` parameter. The nonsense
description in FHIR-54652 is resolved as a side effect of the artifact being
deleted.

#### Justification

This matches the actual workgroup resolution recorded on FHIR-54652,
FHIR-54653, and FHIR-53781 (all Persuasive, all pointing to the same
remedy). It eliminates duplicate machinery in the IG, reduces the surface
area implementers must learn, and keeps the IG aligned with core FHIR
search semantics. The trade-off is a small breaking change for any early
adopter that has already coded against `fast-auditevent-consent`, mitigated
by the fact that the semantically equivalent `entity` parameter is a
drop-in replacement.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as a documentation nit that does not warrant change,
leaving the current description in place and closing the issue as "not
persuasive."

#### Justification

This is not defensible. The reported sentence genuinely is nonsensical and
would confuse implementers reading the published IG. Declining would also
conflict with the already-agreed Persuasive resolutions on the two sibling
tickets (FHIR-54653, FHIR-53781) that mandate removing the artifact. Listed
here only to document that "do nothing" has been considered and rejected.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right answer. The workgroup has already reached
Persuasive consensus on FHIR-54653 and FHIR-53781 that the
`fast-auditevent-consent` search parameter should be withdrawn in favour of
the native AuditEvent `entity` parameter, and the resolution text on
FHIR-54652 itself points to the same outcome. Rewriting the description
(Disposition A) would leave a redundant artifact in the IG and would be
immediately undone by the resolution of FHIR-54653. Declining (Disposition C)
contradicts those sibling resolutions and leaves a genuinely unreadable
description in the published spec. Acting on B resolves all three tickets
with a single, coherent change: remove the SearchParameter, update the
"Searching for Disclosures" narrative to use `entity`, and add a short
migration note for implementers.
