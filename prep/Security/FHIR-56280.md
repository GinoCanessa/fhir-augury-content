## Summary

Lloyd McKenzie reports that the UDAP-based "Security for Scalable
Registration, Authentication, and Authorization (FHIR)" specification
defines its discovery endpoint relative to `{baseURL}`, which the spec
describes as "the base FHIR URL for the FHIR server." He observes that
the protocol is likely to be reused in contexts where the host is not,
strictly speaking, a FHIR server — for example, if CDS Hooks adopts
UDAP in place of its current security process, the CDS Hooks service
endpoint is not a FHIR base URL even though discovery still needs to
work. He suspects similar non-FHIR-server reuse cases will arise and
asks the workgroup to clarify how discovery should be defined in those
situations. No cross-referenced Jira tickets, Zulip threads, or GitHub
items were found for this ticket, so the request stands on its own.

## Details

**Description:**

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Generalize the UDAP "Security for Scalable Registration, Authentication,
and Authorization (FHIR)" IG so that `{baseURL}` is no longer defined
as "the base FHIR URL for the FHIR server." Replace the wording in the
Discovery section (and any other section that uses the same phrasing)
with a definition such as "`{baseURL}` represents the base URL of the
service whose UDAP metadata is being discovered. For FHIR servers this
is the FHIR base URL; for non-FHIR services (e.g., a CDS Hooks
discovery endpoint) it is the equivalent service base URL against which
the `/.well-known/udap` document is published." Update related
narrative, examples, and any constraints/conformance statements that
currently presuppose a FHIR base URL so that the discovery, dynamic
client registration, and token-endpoint discovery flows are explicitly
applicable to non-FHIR services. Keep the existing FHIR-specific
guidance (e.g., capability-statement linkage) but scope it as
"applicable when the server is a FHIR server."

#### Justification

The discovery mechanism (`/.well-known/udap` published relative to a
service base URL) is technically agnostic to whether the service speaks
FHIR — the FHIR-specific framing is purely editorial. Loosening the
definition matches how UDAP is used in the broader ecosystem (UDAP.org
profiles already target non-FHIR endpoints) and aligns the IG with
imminent reuse such as CDS Hooks adopting UDAP. It is a low-risk
clarification: existing FHIR implementations are unaffected because
their base URL is still a valid `{baseURL}` under the new definition.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than redefining `{baseURL}` inline, add a new short normative
section — e.g., "Applicability beyond FHIR servers" — that:

1. Retains the current FHIR-centric wording in Discovery as the primary
   case.
2. Explicitly states that the same discovery, registration, and token
   flows MAY be applied to non-FHIR services, in which case `{baseURL}`
   is interpreted as the service base URL.
3. Calls out which conformance requirements that depend on FHIR
   semantics (e.g., references to CapabilityStatement, FHIR scopes)
   either do not apply or require an equivalent contract from the
   hosting specification (e.g., a CDS Hooks discovery document).
4. Optionally publishes companion guidance via the Security WG site
   (non-normative) showing the CDS Hooks worked example.

#### Justification

This keeps the IG's core, FHIR-server narrative stable for the large
existing implementer base while making the cross-domain reuse explicit
and bounded. It also forces the IG to enumerate which FHIR-specific
conformance hooks need substitutes when the host isn't a FHIR server,
which is information CDS Hooks (and other adopters) will need anyway.
The trade-off vs. Disposition A is a slightly more verbose spec in
exchange for clearer separation between "what UDAP requires" and "what
FHIR adds on top of UDAP."

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that this IG is explicitly the
*FHIR* binding of UDAP, and that non-FHIR adopters such as CDS Hooks
should profile UDAP themselves (or reference upstream UDAP.org
profiles) rather than expanding the scope of the FHIR Security IG.
Mark the ticket "Persuasive with Mod" only to the extent of adding a
short non-normative note pointing readers at upstream UDAP for
non-FHIR-server use cases.

#### Justification

The IG's scope, name, and conformance statements are oriented around
FHIR servers; broadening `{baseURL}` risks dragging FHIR-specific
requirements (scopes, CapabilityStatement linkage, SMART
interoperability) into contexts where they don't apply, and creates a
maintenance burden for the Security WG to keep parity with non-FHIR
hosts. Other specifications (CDS Hooks, etc.) are free to reference
UDAP independently and define their own discovery base URL semantics.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of reach and risk. The reporter's
core observation is correct — discovery, registration, and token
endpoints are mechanically independent of FHIR — and CDS Hooks adopting
UDAP is a near-term, concrete case the IG should be ready for, which
argues against Disposition C. At the same time, several conformance
points in the IG are genuinely FHIR-specific (FHIR scopes, capability
statement linkage, resource-server semantics), and a one-line redefine
of `{baseURL}` (Disposition A) would leave those statements ambiguous
in non-FHIR contexts. Adding a small, explicit "applicability beyond
FHIR servers" section preserves the existing FHIR-server guidance for
current implementers while enumerating the substitutions a non-FHIR
host must provide, which is exactly the structure CDS Hooks and any
"others" Lloyd anticipates will need.
