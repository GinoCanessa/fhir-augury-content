## Summary

The reporter observed during connectathon testing that JWTs used for UDAP
client authentication are very large, and that the dominant contributor to
the JWT size is the client certificate carried in the `x5c` header. The
ticket asks that UDAP allow clients, at registration time, to register a
`jku` (JWK Set URL) that points to their certificate. Servers that trust
the registered `jku` could then resolve and cache the key set per the
relevant RFC caching rules, and all subsequent client-authentication JWTs
could carry a `jku` header in place of the bulkier `x5c`, significantly
reducing on-the-wire JWT size while preserving the trust chain established
by UDAP. There are no cross-referenced Jira tickets in the knowledge base
for FHIR-50148, so the request stands on its own as a Security/UDAP
optimisation aimed at lowering authentication payload size.

## Details

**Description:**

> During connectathon testing, we've encountered the largest JWTs. The
> majority of the content in the JWT is attributed to the cert, a
> relatively static piece of content.
>
> Using X5Cs for all authentication requests bloats JWT sizes.
>
> Enabling clients to register their JKU that points to their cert at
> registration time, enables servers to trust the JKU provided by the
> client. Servers may cache the JKU response for the allowed time per
> RFC requirements.
>
> All subsequent authentication requests by the client would use JKUs,
> significantly reducing JWT sizes sent over the wire while maintaining
> the security and trust enabled by UDAP.

### [smart > TLS cipher suites for JKUs for asymmetric client auth](https://chat.fhir.org/#narrow/stream/smart/topic/TLS.20cipher.20suites.20for.20JKUs.20for.20asymmetric.20client.20auth)

Cooper Thompson opened a thread (2025-06-20, the same week as the UDAP
call where this ticket was first discussed) asking how implementers
handle TLS cipher-suite negotiation when an OAuth2 server must dereference
a `jku` to verify a client assertion. The thread highlights that JKU-based
authentication shifts the trust burden onto the server's outbound TLS
connection to the JKU host, including which cipher suites are acceptable
and how key rotation/caching is handled. Directly relevant to the
feasibility-and-burden side of FHIR-50148: any "use `jku` instead of
`x5c`" change adds a network and TLS-policy dimension that does not exist
with the self-contained `x5c` header.

Participants: Cooper Thompson (and other SMART/UDAP implementers in the
thread).

ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the UDAP IG (Security WG, "Security for Scalable Registration,
Authentication, and Authorization") so that client-authentication JWTs
**MAY** carry a `jku` header instead of `x5c`, conditional on the client
having registered that `jku` (and an associated certificate / JWK Set)
during UDAP Dynamic Client Registration. Concretely:

- Extend the UDAP DCR client metadata to allow a registered `jku`
  (HTTPS URL) and require it to be served over TLS with a certificate
  whose chain matches the cert the client would otherwise have
  presented in `x5c`.
- Update the UDAP JWT-Based Client Authentication section to state that
  a server **SHALL** accept either an `x5c` header or a `jku` header
  whose value matches the client's registered `jku`, and that servers
  **MAY** cache the JKU response for the period allowed by the
  HTTP `Cache-Control` and the relevant JWS/JWK RFCs (RFC 7515 §4.1.2,
  RFC 7517).
- Add normative requirements on `jku` resolution: HTTPS only, the host
  must match the registered URL exactly, the resolved JWK Set must
  contain the cert chain previously registered, and key-rotation rules.

#### Justification

This is exactly what the reporter is asking for and aligns UDAP with the
JWS-native key-discovery mechanism already used by CDS Hooks (see the
*CRD Security Guide* Zulip thread). It reduces wire size for every
authentication request — a real cost at connectathons and in production
fan-out scenarios — without weakening trust, because the `jku` is bound
to the cert at registration time. The 2025-04-22 motion from Isaac Vetter
("allow servers to authenticate clients with either `x5c` or a `jku`
containing an `x5c`") shows there is implementer support for the dual
mechanism. Making the new path **MAY** for clients (but **SHALL** for
servers to accept) preserves backwards compatibility with every existing
UDAP client that uses `x5c`.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing `jku` as a peer mechanism to `x5c`, take the
narrower step of allowing clients to send a JWT whose `x5c` value has
been replaced by an `x5t` / `x5t#S256` thumbprint of the previously
registered cert. The server resolves the thumbprint to the cert it
already has on file from UDAP DCR. Specify:

- During UDAP DCR, the server stores the client's full cert chain.
- For client authentication, clients **MAY** omit `x5c` and instead
  include `x5t#S256` (and optionally `kid`) referring to the registered
  cert; the server **SHALL** accept this when the thumbprint matches a
  registered, non-revoked cert.
- `jku` is **explicitly out of scope** in this disposition.

#### Justification

This addresses the underlying complaint (JWT size) without introducing
a new outbound HTTP dependency on the OAuth2 server, which is the
concern raised in the *TLS cipher suites for JKUs* SMART thread.
Avoiding `jku` keeps the server's authentication path
self-contained — no JKU host availability, no TLS-cipher negotiation,
no caching-policy disagreements, no SSRF surface — while still cutting
the JWT down to roughly the size of a header, signature, and a small
thumbprint. Trade-off versus Disposition A: it does not align with the
CDS Hooks `jku` pattern, so it does not buy the cross-IG convergence
that Disposition A would.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" (or restore the "Considered for Future
Use" disposition) on the grounds that:

- UDAP's reliance on `x5c` is intentional: the cert chain is
  self-contained in the JWT and the server does not need any side
  channel to validate the assertion.
- JWT size, while real, is bounded and acceptable for the
  authentication step (which is comparatively rare versus resource
  calls).
- Implementers who want a smaller authentication JWT can already
  shorten the `x5c` chain by registering an intermediate-rooted leaf
  cert.

#### Justification

The original 2025-07-08 vote (14-0-0, Pyke / Stringham) deferred this
ticket precisely because the change is "substantial" and was not safe to
land before the upcoming Connectathon / STU2 publication. The TLS,
caching, key-rotation, and SSRF concerns surfaced in the SMART Zulip
thread are non-trivial and would require careful normative text. If the
WG concludes the operational complexity outweighs the wire-size win for
an infrequent operation, declining is defensible — especially given that
no implementer has yet provided concrete production data showing the
`x5c` size is causing real interoperability problems (the 2025-06-24
note explicitly asks Rashid to come back with that data).

---

### Recommendation

**Recommended disposition:** A

The ticket has been re-opened post-STU2 by Lynn Laakso specifically
because the WG wanted to revisit it without the Connectathon time
pressure that drove the original deferral. Accepting the request as a
**MAY** for clients (Disposition A) preserves backwards compatibility
with every existing UDAP deployment (so nobody is forced to stand up a
JKU host), gives implementers a real wire-size reduction where it
matters, and aligns UDAP with the JWS-native `jku` pattern already used
by CDS Hooks — directly closing the UDAP↔CDS-Hooks gap that Isaac
Vetter and Joseph Shook flagged in the *CRD Security Guide* Zulip
thread. The legitimate concerns from the SMART *TLS cipher suites for
JKUs* thread (TLS policy, caching, SSRF) are real but are mechanical
problems with known solutions; they should be addressed in the
normative text of the change rather than used as a reason to block it.
Disposition B is a reasonable fallback if the WG cannot reach
consensus on `jku`, but it gives up the cross-IG convergence benefit.
Disposition C is hard to defend now that the ticket has been
explicitly re-opened.
