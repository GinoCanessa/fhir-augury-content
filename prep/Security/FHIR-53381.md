## Summary

The reporter objects to a normative `SHALL` statement on the FHIR Security
page that requires every access-control *deny* decision to be recorded
(with the AuditEvent resource recommended as the carrier). They argue
this is too strong a conformance bar for FHIR as a whole: there are
legitimate FHIR deployments — e.g., test servers, lightweight
implementations, environments without any audit/log infrastructure —
where such recording is impractical or unnecessary. Because failure to
log denied access would, under the current wording, render an
implementation non-conformant with FHIR, the reporter asks that the
requirement be downgraded to a `SHOULD` (or made conditional on
environment / logging capability), and that the Security page's other
`SHALL` statements be reviewed for the same overreach. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the augury index, so the request stands on its own. The
ticket is currently `Triaged` with resolution `Persuasive` and the
recorded resolution description is "Will change the SHALL to a SHOULD."

## Details

**Description:**

The security page includes a number of SHALL statements, including:

> Audit of the Access Control Decision: Any Access Control decision to
> deny an activity SHALL be recorded, and Access Control decisions to
> permit access SHOULD be recorded. The record SHOULD use the FHIR
> AuditEvent Resource.

This requirement to record any deny decision is too strong for all use
cases for FHIR and an absence of this capability would prevent an
implementation from claiming conformance with FHIR.

While it is certainly a best practice, and highly recommended, I don't
think we can mandate that failure to log denied access means your
implementation is non-compliant.

Please change this to a SHOULD, or make the language conditional on
factors such as environment, logging capabilities, etc.

Review other SHALL statements to ensure that they are appropriate for
all implementations in all environments.

**Comments:**

- **john_moehrke (2026-02-19):** Pushes back on the premise — argues
  that not recording access-control denies is itself bad policy because
  denied attempts are exactly the signal that shows where policy edges
  are being probed and where users are trying to act beyond their
  scope.
- **mckenzie (Lloyd McKenzie, 2026-03-23):** Sides with the reporter on
  conformance grounds. Whether or not it's good policy, FHIR can't say
  "any system that doesn't do this isn't FHIR-conformant." Cites test
  servers as an obvious counter-example, and notes that some systems
  legitimately maintain no audit log of *anything*. Concludes that
  because there are valid use cases without audit, and because it isn't
  HL7's place to set general audit expectations, the requirement
  shouldn't be stronger than `SHOULD`.
- **john_moehrke (2026-03-30):** Records that Elliot Silver agreed on
  the call to the SHALL→SHOULD change, and explicitly asks Lloyd
  McKenzie whether changing to `SHOULD` would be acceptable to him.

## Proposed Dispositions

> Note: Dispositions below are framed against the FHIR Core security

> conventionally `source/security.xml` (or the corresponding markdown

### Disposition A: Accept As Requested

#### Proposal

Edit the FHIR Core Security narrative page so that the bullet currently
reading:

> Audit of the Access Control Decision: Any Access Control decision to
> deny an activity SHALL be recorded, and Access Control decisions to
> permit access SHOULD be recorded. The record SHOULD use the FHIR
> AuditEvent Resource.

becomes:

> Audit of the Access Control Decision: Any Access Control decision to
> deny an activity SHOULD be recorded, and Access Control decisions to
> permit access SHOULD be recorded. The record SHOULD use the FHIR
> AuditEvent Resource.

In addition, perform a sweep of the remaining `SHALL` statements on the
Security page (and closely-linked pages such as `security-labels.html`
and the AuditEvent narrative) to confirm each is genuinely an
interoperability/conformance requirement that holds for every FHIR
implementation, and downgrade or qualify those that don't (e.g., make
them conditional on environment or logging capability, or move them
into best-practice / SecurityLabel guidance). This matches the
resolution description already recorded on the ticket ("Will change the
SHALL to a SHOULD").

#### Justification

This is the change the reporter asked for, the change Lloyd McKenzie
endorsed on conformance grounds, and the change the Security WG
recorded as "Persuasive" with Elliot Silver's verbal agreement on the
call. The conformance argument is sound: FHIR Core normative wording
binds every conformant implementation, and there are legitimate FHIR
deployments (test servers, sandboxes, lightweight components, systems
that intentionally maintain no persistent log) for which mandatory
auditing of denials is either impossible or contrary to design. A
`SHOULD` preserves the strong recommendation — and the existing
guidance that AuditEvent be the recording mechanism — without making
the absence of an audit pipeline a non-conformance. The follow-on sweep
of other Security-page `SHALL`s is consistent with the reporter's
explicit second ask and prevents the same issue recurring elsewhere on
the page.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the strong "MUST log denies" expectation, but move it out of FHIR
Core normative narrative and express it as a conformance rule that only
applies when a server *claims* an auditing/security capability. Concretely:

1. On the Security page, soften the base statement to `SHOULD`, as in
   Disposition A.
2. Add a conditional clause: *"Where an implementation claims support
   for access-control auditing (e.g., asserts the relevant
   `CapabilityStatement.security` element or conforms to a security
   IG), denied access decisions SHALL be recorded, and the record
   SHALL use the FHIR AuditEvent resource."*
3. Push the unconditional `SHALL` into the Security IG / SMART /
   relevant downstream profile(s) where the deployment context actually
   supports it, rather than leaving it on FHIR Core.

#### Justification

This addresses Moehrke's substantive concern — that detecting probing
of policy edges is exactly *why* deny-logging matters — without
overreaching FHIR Core's conformance scope. The strong requirement
survives, but only binds implementations that have voluntarily entered
the "I do access-control auditing" contract. Trade-off vs. Disposition
A: it's more text, more nuance, and slightly more work for the editor
and for downstream IG authors; it also defers to capability-statement
machinery and IGs that may not yet exist, so there's a transition
window where the practical effect is similar to a plain `SHOULD`.
Trade-off vs. Disposition C: it preserves the security signal the
reporter was *not* trying to weaken.

---

### Disposition C: Decline

#### Proposal

Reject the change. Keep the existing wording: deny decisions `SHALL` be
recorded, with AuditEvent as the recommended record, on the basis that
auditing of denied access is a foundational security expectation and
that systems unwilling or unable to meet it should not be claiming
unrestricted FHIR conformance. Optionally add a non-normative note
clarifying that the requirement applies "where access control is
performed" so that systems with no access-control layer are simply
out-of-scope rather than non-conformant.

#### Justification

This is the position Moehrke staked out in the comments: deny logs are
the signal that surfaces probing and policy-edge abuse, and weakening
them risks normalizing "secure-but-silent" implementations. A
clarifying scoping sentence (only applies where access control is
performed) addresses the test-server objection without surrendering the
normative force of the requirement. The defensibility of decline is
weaker here than in many tickets, however, because (a) the conformance
argument cuts cleanly — FHIR Core normative `SHALL`s really do bind
every implementation, including ones with no auditing infrastructure —
and (b) the reporter, the chair-equivalent commenter (McKenzie), and
the WG have all already converged on a different answer.

---

### Recommendation

**Recommended disposition:** A

The Security WG has effectively already landed on Disposition A: the
ticket carries resolution `Persuasive` with the resolution description
"Will change the SHALL to a SHOULD," Elliot Silver verbally agreed on
the call, and Lloyd McKenzie's comment makes the FHIR-Core conformance
case crisply — `SHALL` here would mean *every* FHIR-conformant system,
including test servers and audit-less components, must record deny
decisions, which isn't a tenable bar. Disposition B (capability-gated
`SHALL`) is intellectually attractive and preserves Moehrke's security
intent, but it adds normative machinery that doesn't exist on the
Security page today and can be picked up later by a Security IG without
holding this fix. Disposition C is hard to defend once the conformance
framing is on the table. Recommend going with A and coupling it with
the second ask in the ticket — a quick editorial sweep of the other
Security-page `SHALL`s — so the same problem doesn't resurface in
adjacent bullets. The Block-Vote-2 label is consistent with treating
this as a low-controversy editorial fix.
