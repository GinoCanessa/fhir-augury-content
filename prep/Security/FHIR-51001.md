## Summary

The reporter observes that the published history page for the Data
Segmentation for Privacy implementation guide
(`https://hl7.org/fhir/uv/security-label-ds4p/history.html`) exposes a
"CI build" link that points to
`https://hl7.org/fhir/ig/HL7/security-label-ds4p`, which is not the
active CI build location for this IG. The correct continuous-integration
build URL is
`https://build.fhir.org/ig/HL7/fhir-security-label-ds4p/`, reflecting
the repository name `HL7/fhir-security-label-ds4p`. The ticket is a
cosmetic / navigation fix against the history-page generator (or its
source-of-truth metadata for this IG), not a change to the IG's
publication source. No related Jira tickets, Zulip threads, or GitHub
cross-references were found for this ticket in the cache.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Correct the "CI build" pointer for the DS4P IG so that the entry on
`https://hl7.org/fhir/uv/security-label-ds4p/history.html` links to
`https://build.fhir.org/ig/HL7/fhir-security-label-ds4p/` instead of
`https://hl7.org/fhir/ig/HL7/security-label-ds4p`. The authoritative
source for this mapping is typically the IG's `package-list.json`
(the `ci-build` URL entry) and/or the HL7 IG registry entry that
drives the shared history-template. Update whichever of those sources
is out of date for `hl7.fhir.uv.security-label-ds4p` so the rebuilt
history page (and the IG registry) resolve to the correct
`HL7/fhir-security-label-ds4p` CI build, then republish the history
page.

#### Justification

The reported URL is plainly inconsistent with the actual GitHub
repository hosting the DS4P IG (`HL7/fhir-security-label-ds4p`), so
the current link is a broken/misleading navigation aid for anyone
trying to review the in-progress version of the IG. Fixing it is a
metadata change only — no normative content is affected — and it
aligns the publicly advertised CI build with the repository where
continuous integration actually runs. The reporter's own comment
confirms this is a tooling/metadata fix, consistent with how the FHIR
publication toolchain treats history-page links.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) patching the `ci-build` URL for this
one IG, treat this as a symptom of a broader drift between IG
repository names and the CI-build URLs advertised on published
history pages. Accept the specific correction for DS4P as in
Disposition A, and also ask HL7 publishing/Tooling to add an
automated check — for example, a sanity pass in the IG registry or
history-template generator that verifies the `ci-build` URL for each
IG resolves and matches the IG's declared GitHub repo — so similar
divergences (legacy `HL7/security-label-ds4p`-style paths vs current
`HL7/fhir-security-label-ds4p`-style paths) are flagged automatically.

#### Justification

The root cause is a common one: when an IG's GitHub repo is renamed
or created under a new naming convention, the history page's
`ci-build` pointer can be left behind. A one-off correction fixes
this ticket but does not protect against the same class of bug on
other IGs. Pushing a systemic check into the publication pipeline is
higher leverage; the DS4P fix itself is trivial and can be done now
while the broader check is scoped. This approach is slightly heavier
than the minimal fix but keeps the Security WG's ticket resolved
while nudging HL7 Tooling toward a durable solution.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "out of scope for the Security Work Group /
forward to HL7 Publications & Tooling," on the basis that the fix is
a publication-infrastructure concern rather than a normative change
to the DS4P IG itself. The Security WG would document that the
corrected CI-build URL is `https://build.fhir.org/ig/HL7/fhir-security-label-ds4p/`
and hand the ticket off to the appropriate HL7 staff / FHIR-I or
Tooling WG to action against the IG registry / history template.

#### Justification

Declining here is purely a routing decision, not a rejection of the
reporter's observation. The Security WG owns the DS4P specification
content but not the history-page generator or the IG registry
metadata that drives the `ci-build` link, so the Security WG cannot
close the issue by editing its own sources. This is also consistent
with the reporter's explicit question in-comment about whether the
fix "falls to staff." The downside is that an unrouted ticket can
linger; this disposition is only defensible if it is paired with an
explicit handoff, not simply closed.

---

### Recommendation

**Recommended disposition:** A

The request is narrow, well-scoped, and uncontroversial: the
published `ci-build` URL for the DS4P IG is demonstrably wrong and
should point at `https://build.fhir.org/ig/HL7/fhir-security-label-ds4p/`.
The fix is a metadata / tooling change — most plausibly in
`package-list.json` for `hl7.fhir.uv.security-label-ds4p` and/or the
IG registry entry — and carries no normative risk. Disposition A
resolves the reporter's concern directly. Disposition B's systemic
check is a good idea in principle but is better raised as a separate
item against HL7 Tooling than held as a blocker on this ticket.
Disposition C would route the work correctly but, without an
explicit handoff, risks leaving a visible broken link on the
published history page; Security WG should accept the fix and
coordinate the actual metadata change with HL7 staff as part of
executing Disposition A.
