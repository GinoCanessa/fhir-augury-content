## Summary

The reporter requests that the UDAP-based "Security for Scalable Registration,
Authentication, and Authorization" (SSRA) Implementation Guide add explicit
guidance addressing **federated client registration** — i.e., scenarios in
which a single registered UDAP client record is recognized (or shared) across
multiple authorization servers within a trust community. The ticket frames the
benefits (consolidated registration workflows, consistent client identities,
trust-community ability to detect misconfigurations or malicious actors,
reduced maintenance for client developers) and proposes two concrete additions
brainstormed at the September 2024 connectathon/WGM: (1) signaling in the
authorization server's metadata endpoint whether a federated client record is
shared with other auth servers, and (2) encouraging trust communities to
define a minimum (and possibly maximum) set of scopes required for clients to
participate in data exchange for a given purpose of use. Comments on the
ticket capture the Security WG's prior discussion: building a new technical
mechanism is widely viewed as out of scope, but adding *guidance* to the
registration page — particularly clarifying that a registration with one
authorization server should not silently affect registrations on other
authorization servers — was considered feasible and useful. The ticket was
originally resolved as "Considered for Future Use" (15-0-0 vote, 2024-11-26)
and was re-opened post STU2 publication for further consideration. There are
no cross-referenced Jira tickets, Zulip threads, or GitHub items in the
augury index for this issue.

## Details

**Description:**

Federating a client record across multiple authorization servers offers
several advantages:

- Consolidates registration workflows and associated business logic for a
  group of authorization servers.
- Minimizes discrepancies in client identities across different servers
  running the same software, improving integration consistency.
- Enables trust community participants to detect and mitigate
  misconfigurations or malicious actors by tracking shared client records.
- Lowers the maintenance load on client developers by reducing the need to
  support multiple versions of a client for different servers.

In the Sept connectathon/WGM we brainstormed several ideas in which we can
support a federated client approach by:

- Introducing guidance that informs clients on whether the authorization
  server is sharing a federated client record with other auth servers in the
  auth server's metadata endpoint.
- Encouraging trust communities to define a minimum (and possibly maximum)
  set of scopes required for clients to participate in data exchange for
  specific purposes of use.

We propose further exploration of these concepts within the UDAP SSRA Guide
to provide actionable guidance for both clients and authorization servers.

## Proposed Dispositions

> Note on Repo Context: No related GitHub repository was surfaced via
> cross-references and no `repo-analysis` briefing has been loaded for the
> UDAP SSRA IG. The dispositions below name the *publication target* (the
> SSRA IG's dynamic client registration / metadata pages) but do not cite

### Disposition A: Accept As Requested

#### Proposal

Add a new normative-but-non-breaking "Federated Client Registration"
subsection to the UDAP SSRA IG's Dynamic Client Registration page. The
subsection would:

1. Define the term *federated client record* (a client registration that is
   recognized by, or shared across, more than one authorization server
   within a trust community).
2. Introduce an OPTIONAL boolean indicator in the authorization server's
   UDAP metadata document (e.g., `udap_federated_client_registration_supported`,
   exposed via the `.well-known/udap` endpoint) signaling whether this server
   participates in a federated client arrangement, with a companion field
   identifying the federation/trust community.
3. State explicitly (per the 10/25 WG discussion) that, by default, a client
   registration with one authorization server SHALL NOT have side effects on
   any other authorization server's registration state — and that any
   federated arrangement that diverges from this default MUST be advertised
   via the new metadata indicator and documented by the trust community.
4. Recommend that trust communities define a baseline (minimum, and
   optionally maximum) set of scopes required for clients to participate in
   data exchange for each declared purpose of use.

#### Justification

This proposal directly delivers the two artifacts the reporter brainstormed
at the September connectathon/WGM (metadata signaling + trust-community
scope guidance) without inventing a new federation protocol. The metadata
flag is the smallest interoperable hook that lets clients reason about
federation without trial-and-error. Stating the "no cross-server side
effects" default in the IG addresses the concrete concern Luis Maas raised
(registration overwrites across endpoints) and aligns with the WG's
10/22 and 10/25 conclusions that *guidance* — not a new technical
mechanism — is the right vehicle. Re-opening the ticket post STU2
publication (per Lynn Laakso's 2025-12-09 comment) implies the WG is now
ready to accept additive guidance into a future SSRA revision.

### Disposition B: Alternative Approach

#### Proposal

Take the lighter-weight path the WG appeared to converge on during the
10/22 and 10/25 discussions: add only **clarifying narrative** to the
existing Dynamic Client Registration page in the UDAP SSRA IG, without
introducing any new metadata field. Specifically:

1. Add an editorial note that a UDAP client registration is, by default,
   scoped to the specific authorization server where it was performed and
   MUST NOT be assumed to register the client with any other authorization
   server — even servers operated by the same organization or running the
   same software.
2. Acknowledge federated client registration as a legitimate trust-community
   pattern that is **out of scope** for the SSRA IG technical mechanisms
   (per John Moehrke's 11/20 question), and point implementers at the trust
   community's documentation for any federation-specific behavior.
3. Add a non-normative recommendation that trust communities document any
   minimum scope set required for participation in their data-exchange
   purposes of use.

This disposition deliberately defers any UDAP metadata changes to a future
ticket once a concrete trust community has demand for them.

#### Justification

This matches the WG's prior consensus (per the 10/25 transcript summary):
"adding this to the guidance would be sufficient for now, and other
potential changes were out of scope." It also satisfies John Moehrke's
explicit ask that the IG distinguish legitimate-but-out-of-scope use cases
from supported ones. The trade-off versus Disposition A is reduced
interoperability — clients can't programmatically discover federation
behavior — but it avoids minting a metadata field that no auth server is
yet committed to publishing, which carries real ecosystem cost: once a
field is in the IG, implementers have to handle its absence, its presence
with a `false` value, and its presence with a `true` value, even if no
trust community ever uses it. Joseph Lamy's comment that the capability
may already be partially in practice and "simply need formalization"
supports formalizing the behavior in narrative first and adding machinery
later.

---

### Disposition C: Decline

#### Proposal

Restore the prior resolution of "Considered for Future Use" (the
2024-11-26 disposition that was reverted on 2025-12-09 to re-open the
ticket post-STU2). Decline to add either a metadata signal or new
narrative guidance to the SSRA IG at this time, on the basis that:

1. The WG has previously concluded (10/22) that "there are ways to
   accomplish this without creating a new technical solution" — i.e., the
   gap is not in the IG.
2. No concrete trust community has yet brought a use case that requires
   federated client registration to be discoverable via UDAP metadata, and
   no Zulip / GitHub discussion is recorded against this ticket to
   demonstrate active demand.
3. Adding even non-normative federation guidance to a published IG risks
   implementers reading it as encouragement to build divergent federation
   schemes before a trust community has codified what "federated" means in
   their context.

#### Justification

Declining is defensible primarily because the underlying need is currently
hypothetical at the IG level: the augury index surfaces no cross-references,
no Zulip threads, and no GitHub items tied to FHIR-48550. The original
15-0-0 vote to defer reflects strong WG consensus, and the 2025-12-09
re-open did not (per the visible comments) introduce new evidence that the
deferral was wrong — it simply reopened the question post-STU2. Declining
keeps the IG focused on STU2 errata while inviting the reporter to bring a
concrete trust-community proposal back as a future ticket.

---

### Recommendation

**Recommended disposition:** B

Disposition B best matches the WG's documented position. Two independent
threads in the comment history (10/22 and 10/25) converged on the same
conclusion: a new technical mechanism is out of scope, but clarifying
guidance — particularly the "no cross-server side effects" expectation
that Luis Maas articulated, and the legitimate-but-out-of-scope framing
that John Moehrke asked for — is feasible and useful. Disposition B
delivers exactly that and nothing more, which is appropriate given that
the augury index shows zero downstream Zulip discussion, zero GitHub
activity, and no cross-referenced Jira tickets demonstrating implementer
demand for a normative metadata signal. Disposition A is the right next
step *if* a trust community subsequently commits to publishing the
federation indicator — but it is premature to standardize that hook now.
Disposition C undersells the value of the editorial clarification and
leaves the registration-side-effects ambiguity (a real implementer concern,
not a hypothetical one) unresolved in the IG. The Security WG should
adopt B for the next SSRA revision and explicitly file a follow-up ticket
to reconsider A once a trust community is ready.
