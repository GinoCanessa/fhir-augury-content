## Summary

The ticket reports that the published examples for the VhDir `vhdir-restriction`
profile (a profile on the FHIR `Consent` resource in the Validated Healthcare
Directory IG) do not use the terminology defined by the core FHIR specification.
The reporter, Eric Haas, supplies a concrete replacement example sourced from
the VhDir demo server (`http://vhdir-demo.fhir.org/.../vhdir-miscellaneous-healthcareservice-9994427431`)
that exercises the proper code systems (`consentscope`, `v3-ActCode`,
`v3-ParticipationType`, `v3-ActReason`) in the appropriate `Consent` elements
(`scope`, `category`, `provision.actor.role`, `provision.action`,
`provision.securityLabel`, `provision.purpose`). A follow-up comment from
"mckenzie" points to a parallel HealthcareService example
(`HealthcareService-hcs-ws.html`) that should likewise be reviewed for
terminology alignment. No cross-referenced Jira tickets, Zulip discussions, or
GitHub items were surfaced for this ticket by the augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the existing example(s) for the `vhdir-restriction` profile in the
`HL7/VhDir` IG with the JSON instance supplied in the ticket description, which
correctly populates:

- `Consent.scope` with `consentscope#patient-privacy`
- `Consent.category` with `v3-ActCode#INFA`
- `Consent.provision.actor.role` with `v3-ParticipationType#IRCP`
- `Consent.provision.action` with `v3-ActReason#use`
- `Consent.provision.securityLabel` with `v3-ActCode#SDV`
- `Consent.provision.purpose` with `v3-ActReason#TREAT`

Also revisit the HealthcareService example called out in the comment
(`HealthcareService-hcs-ws`) in the same IG and align any miscoded codings to
the canonical core terminology before republishing the IG.

#### Justification

The current examples appear to use placeholder or non-canonical codings, which
is exactly the kind of defect a Change Request against an example is meant to
fix: examples are normative-by-demonstration for implementers, and getting the
binding-correct codings in front of them is the simplest, lowest-risk fix.
Eric Haas has provided a complete, drop-in replacement instance derived from a
running VhDir demo server, so the editorial cost is small and the change does
not touch the profile definition, value sets, or any conformance constraint.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than replacing the example wholesale, treat this as a small editorial
sweep across the VhDir IG: (1) audit every example bound to a profile on
`Consent`, `HealthcareService`, and other Restriction-adjacent resources for
correct use of the bound terminology (`consentscope`, `v3-ActCode`,
`v3-ParticipationType`, `v3-ActReason`); (2) where the supplied example
exercises additional elements not covered by the current example, retain the
existing example and add a second, richer example so implementers see both a
minimal and a fuller variant; and (3) add a short narrative on the profile
page explaining which code systems are expected at each `provision.*` element.

#### Justification

The reporter and the commenter both surface examples as the symptom, not the
root cause: the underlying need is that implementers of the VhDir IG can't
infer the correct terminology from the published examples. A single replacement
example fixes the most obvious instance, but a directed sweep plus narrative
reduces the chance that the same defect is filed again against a sibling
profile. The cost is modestly higher than Disposition A, but is still
example/IG-narrative work — no profile definition or value-set bindings would
change.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for the current cycle on the grounds that the
Validated Healthcare Directory IG has had limited active maintenance since the
ticket was filed (2019) and the Patient Administration WG may not have
publishing capacity to re-issue the IG solely for an example fix. Recommend
either deferring until the next planned VhDir publication, or transferring
ownership/closing if VhDir is not on the WG roadmap.

#### Justification

The defect is in published examples, not in the conformance surface
(StructureDefinitions, ValueSets, search parameters, operations), so no
implementer is blocked by an incorrect example in the same way they would be
by a broken constraint. If there is no near-term republication of VhDir, the
realistic outcome of accepting is that the fix sits in the build but does not
reach implementers, and the WG carries an open issue indefinitely.

---

### Recommendation

**Recommended disposition:** A

The ticket is narrow, the defect is real, and the reporter has supplied a
ready-to-merge example that uses the canonical terminology. The change is
example-only — no profile, value set, or constraint is touched — so the risk
of regression is essentially zero, and the fix directly improves the signal
implementers get from the IG. The HealthcareService example flagged in the
comment can ride along as part of the same editorial pass without expanding
scope materially. Disposition B is attractive if the WG has bandwidth for a
broader sweep, but it should not block the straightforward fix in A;
Disposition C is hard to defend on technical grounds given that a correct
replacement is already in hand.
