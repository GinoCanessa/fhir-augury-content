## Summary

[FHIR-51455](https://jira.hl7.org/browse/FHIR-51455) asks Patient
Administration to add normative-style guidance to the FHIR specification that
clarifies *when* and *how* implementers should use the `EncounterHistory`
resource versus modelling status-segment changes as child `Encounter`
instances. The request originated from FMG pushback during review: FMG
questioned whether `EncounterHistory` was actually needed given that nested /
child encounters could in principle be used to represent the same lifecycle
information. Patient Administration agreed the request is persuasive and a
clarification draft is being authored on the Confluence page
[Encounter History (PA space, page 358880497)](https://confluence.hl7.org/spaces/PA/pages/358880497/Encounter+History);
the deliverable for this ticket is to land that guidance in the published
spec (Encounter and EncounterHistory resource pages).

There are no cross-referenced Jira tickets and no cross-referenced GitHub
items returned by the augury index; the only formal cross-reference is the
Confluence draft page named in the resolution.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new "When to use EncounterHistory" subsection to both the
`Encounter` and `EncounterHistory` resource notes (FHIR Core spec,
Patient Administration boundary). Concretely:

- On the `EncounterHistory` resource page, add a "Boundaries and
  Relationships → Encounter vs EncounterHistory vs child Encounter"
  block that states:
  - `EncounterHistory` represents the **status-and-location lifecycle
    of a single Encounter** (an immutable audit-style projection of
    `Encounter.status` and `Encounter.location` over time). It is not a
    clinical context object and does not own clinical references.
  - A child `Encounter` (linked via `Encounter.partOf`) represents a
    **separate clinical interaction** that occurred within or as part
    of a parent encounter (e.g., a consult inside an inpatient stay).
    It carries its own participants, diagnoses, reasons, and clinical
    references.
  - Therefore `EncounterHistory` is **not** substitutable by child
    `Encounter` resources: child Encounters describe sub-interactions,
    not state transitions; conversely, `EncounterHistory` cannot
    represent a sub-interaction.
- On the `Encounter` resource page, add a short pointer in
  "Boundaries and Relationships" that says lifecycle/status/location
  history belongs on `EncounterHistory`, and that `Encounter.partOf`
  is for sub-interactions, not status segments.
- Add a usage example to `EncounterHistory` showing two history rows
  for the same `Encounter` (admit → discharge) and a contrasting
  example showing `Encounter.partOf` for a consult within a stay.

The text basis is the draft Patient Administration is preparing on
[Confluence page 358880497](https://confluence.hl7.org/spaces/PA/pages/358880497/Encounter+History).

#### Justification

This is exactly what the ticket asks for and what FMG asked PA to
provide. The distinction (history vs sub-interaction) is consistent
with how other FHIR resources separate audit/state projections from
clinical content (e.g., `Provenance` vs the resources it targets;
`Observation` history vs related Observations). Implementer threads
on Zulip (`implementers > Epic use of Encounter.class`,
`argonaut > Encounter-end`) show people *find* `EncounterHistory`
and want to use it but are unsure when — narrative guidance is the
lowest-cost, highest-value intervention. The resolution is already
marked **Persuasive** by PA, and a draft already exists, so the
remaining work is editorial.

---

### Disposition B: Alternative Approach

#### Proposal

Land the same clarifying narrative as Disposition A, **plus** add a
constraint and an explicit modelling rule to make the boundary
machine-checkable rather than purely narrative:

- Add an `EncounterHistory` invariant requiring
  `EncounterHistory.encounter` to be populated and stating that an
  `EncounterHistory` instance MUST refer to exactly one `Encounter`
  (no chaining of histories of histories).
- Add a SHOULD-level rule on `Encounter` that `Encounter.partOf`
  SHALL NOT be used to represent status or location transitions of
  the parent encounter — those belong on `EncounterHistory`.
- Add a one-paragraph "Common modelling mistakes" block with a
  do/don't pair (don't model an admit→discharge as two child
  encounters; do create one Encounter with two EncounterHistory
  rows).
- Optionally, expose a search parameter or add an example
  `Bundle` showing `Encounter` + ordered `EncounterHistory` retrieval.

#### Justification

The ticket's underlying problem is that two valid-looking modelling
choices (child Encounter vs EncounterHistory) coexist with no
spec-level rule preventing the wrong one. Narrative alone (Disposition
A) relies on implementers reading the prose; a SHOULD/SHALL rule plus
an invariant gives validators something to flag and gives reviewers a
citation. This costs more editorial cycles and a small number of
ballot comments, but it durably resolves the FMG-style "why not child
encounters?" question by encoding the answer in the resource
definition rather than only in narrative.

---

### Disposition C: Decline

#### Proposal

Close the ticket without making spec changes on the grounds that the
FMG concern was answered in the original `EncounterHistory` resource
proposal review (referenced in the FMG `Agenda topic` and `Trial-Use`
threads, and the OO-WGM Q3 reference to the approved FHIR-14157), and
that the existing resource definitions are sufficient. Add at most a
pointer from each resource page to the existing approved proposal.

#### Justification

`EncounterHistory` was reviewed and approved by FMG; further narrative
risks re-litigating a settled design. If implementers misuse
`Encounter.partOf` for status transitions, that is an implementer
education issue rather than a spec defect.

---

### Recommendation

**Recommended disposition:** A

The ticket is small, scoped, and explicitly editorial: PA has already
marked it **Persuasive** and a Confluence draft is in flight, so the
work to land Disposition A is mostly mechanical (move the draft text
into the `Encounter` and `EncounterHistory` resource notes). It
directly answers the FMG question that motivated the ticket and
addresses the demonstrated implementer confusion seen in the
`implementers` and `argonaut` Zulip threads. Disposition B is
technically stronger but introduces a new invariant and a
SHOULD-level constraint, which expands ballot-comment surface and
delivery risk for what is fundamentally a "how to use" clarification;
B is a good follow-up ticket if narrative alone proves insufficient
in the next ballot cycle. Disposition C is not defensible: even
though the resource was approved, the FMG pushback and implementer
confusion captured in Zulip show that *approval* did not produce
clarity, and the ticket's reporter (Cooper Thompson, PA member close
to this work) explicitly asked for clarification.
