## Summary

The Validated Healthcare Directory (VhDir) IG defines an `identifier-status`
extension whose value set includes concepts such as `active`, `inactive`,
`issued-in-error`, `revoked`, and `pending`. The reporter (on behalf of
FHIR-I) raises three concerns and asks the Patient Administration WG to
discuss them — ideally at a Working Group Meeting — together with FHIR
Infrastructure:

1. **Overlap with `Identifier.use`** — Several proposed status codes appear
   to duplicate existing `Identifier.use` semantics (e.g.,
   `inactive`/`revoked` ≈ `old`, `pending` ≈ `temp`).
2. **Modifier-ness** — Some of these codes change the meaning of the
   identifier (an `inactive`, `revoked`, or `issued-in-error` identifier
   must not be treated like an active one), which means they may need to
   be carried via a modifier extension rather than a regular extension.
3. **Scope** — If a status concept is genuinely needed for identifiers,
   the responsibility for defining it sits with FHIR-I as a generic
   extension on the `Identifier` datatype, not with VhDir as an
   IG-specific artifact.

There are **no Jira links, cross-references, or Zulip threads currently
indexed against FHIR-43338**, so this ticket is the sole carrier of the
discussion to date.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Schedule a joint FHIR-I / Patient Administration discussion (WGM agenda
item) on the future of the VhDir `identifier-status` extension. The
expected outcome is to **deprecate the IG-local extension in VhDir** and
**replace it with a generic, FHIR-I-owned extension** on the `Identifier`
datatype, with the value set narrowed to those concepts that are not
already expressible via `Identifier.use`. Specifically:

- Drop `inactive` and `pending` from the proposed value set; instruct
  implementers to use `Identifier.use = old` and `Identifier.use = temp`
  respectively.
- Retain only the concepts that genuinely add information beyond
  `Identifier.use` — primarily `issued-in-error` and `revoked` — and
  publish them on a generic `Identifier.status` extension defined in the
  base FHIR build.
- Mark the extension as a **modifier extension** (`isModifier = true`,
  with a `modifierReason`), because an `issued-in-error` or `revoked`
  identifier must not be acted on as if it were valid.
- Update VhDir to reference the new generic extension and retire the
  IG-specific one, leaving a redirect / deprecation note for existing
  implementers.

#### Justification

This is precisely what the ticket asks for and matches well-established
FHIR design rules:

- FHIR generally avoids two mechanisms with overlapping semantics; codes
  duplicating `Identifier.use` should be removed to prevent ambiguity in
  conformance and search.
- Status codes that change how the identifier may be used are textbook
  modifier extensions — the published "When to use a modifier extension"
  guidance applies directly.
- Cross-cutting datatype extensions belong in the core build under
  FHIR-I stewardship so every IG (not only VhDir) benefits and there is
  one canonical URL for tooling and validators.

### Disposition B: Alternative Approach

#### Proposal

Keep the `identifier-status` extension in VhDir for now, but **constrain
it to only the non-overlapping concepts** (`issued-in-error`, `revoked`,
and any directory-specific lifecycle states the WG can justify), mark
the extension as a modifier extension, and add an explicit usage note
that `Identifier.use` MUST be used for `old`/`temp` semantics. Open a
follow-up tracker against FHIR-I (separate ticket) to consider promoting
the cleaned-up extension to the base spec in a later cycle, but do not
make that promotion a blocker for the current VhDir publication.

#### Justification

This addresses the substantive concerns (overlap with `Identifier.use`
and modifier-ness) without taking a hard dependency on a base-spec
change that FHIR-I would need to schedule and ballot. It allows VhDir
to ship on its current timeline while removing the ambiguous codes and
flagging the extension correctly. The trade-off versus Disposition A is
that the extension URL stays IG-scoped in the short term, so other IGs
that want the same concepts will either reuse VhDir's URL (awkward) or
wait for a generic extension — but this is a manageable transition cost
compared with blocking the IG.

### Disposition C: Decline

#### Proposal

Decline the request to alter or relocate the `identifier-status`
extension. Keep the extension as currently defined in VhDir, with its
existing value set and non-modifier classification, on the grounds that
directory-specific identifier lifecycle is an IG-level concern and the
overlap with `Identifier.use` is acceptable because the two elements
serve different audiences (publication/use vs. lifecycle).

#### Justification

The defensible argument here is scope and stability: VhDir is in
active publication, implementers may already be relying on the current
extension, and a redesign could cause downstream breakage. However,
this disposition does not actually answer the modifier-ness concern,
which is a correctness issue (an `issued-in-error` identifier treated
as valid is a safety problem), so this option is the weakest.

### Recommendation

**Recommended disposition:** A

The reporter is FHIR-I's perspective speaking, and the three concerns
they raise are all well grounded in standing FHIR design rules:
duplicate semantics with `Identifier.use`, the need for
modifier-extension treatment of life-cycle codes, and the
"datatype extensions live in core" convention. Disposition A is the
clean, durable answer that fixes all three at once and gets the
extension out of the wrong stewardship layer. Disposition B is a
sensible fallback if the WGs cannot align on a base-spec change in
time for VhDir's next publication, but it should be treated as a
short-term mitigation that explicitly schedules the move to A. The
key actionable next step in either case is to **get this on a joint
FHIR-I / PA WGM agenda**, which is exactly what the reporter
requested.
