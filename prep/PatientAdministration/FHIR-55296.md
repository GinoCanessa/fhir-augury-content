## Summary

The submitter (Lloyd McKenzie, comment 260 imported into Jira) reports that
`EpisodeOfCare.statusHistory.status` is a mandatory element with a required
binding to the `EpisodeOfCare` status value set, but the bound value set
contains no "unknown", "other", or equivalent escape-valve concept. As a
result, when an implementer needs to record that an episode of care had a
historical status that does not map cleanly to any of the bound concepts,
the `statusHistory` element becomes impossible to populate at all (because
both the wrapper element and the inner `status` are required). The request
is to add an "unknown", "other", or equivalent concept to the value set so
that `statusHistory` entries can always be expressed.

The ticket was originally filed against another work group; Michelle Miller
re-routed it to Patient Administration on 2026-01-29 because PA owns
`EpisodeOfCare`. There are no Jira tickets cross-referenced from this item
in the current index, and no GitHub PRs/issues link to it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the value set bound to `EpisodeOfCare.statusHistory.status` (the
EpisodeOfCare status value set, which is shared with `EpisodeOfCare.status`)
by adding an explicit escape-valve concept. The most consistent option with
the rest of FHIR Core is to add the existing data-absent code
`unknown` (e.g., from the `request-status` / `EncounterStatus`-style
pattern, or via `http://terminology.hl7.org/CodeSystem/data-absent-reason#unknown`
included into the value set) so a `statusHistory` entry can always be
populated when the historical status is genuinely not known. Keep the
binding strength at `required`, leave both `EpisodeOfCare.status` and
`EpisodeOfCare.statusHistory.status` mandatory, and update the value-set
description and the resource narrative on the `EpisodeOfCare` page to
explain when `unknown` is appropriate.

#### Justification

This is exactly what the submitter asked for and matches the established
"escape valve in required value sets" pattern used widely across FHIR Core
(e.g., `Encounter.status` includes `unknown`, several workflow status value
sets include `unknown`/`other`). Adding a concept to the value set is
backward-compatible for readers (existing codes remain valid) and only
mildly disruptive for writers and validators, which is consistent with how
analogous tickets have been handled in the past. Because the same value set
backs `EpisodeOfCare.status`, the addition is symmetric across the resource
and avoids introducing two divergent status vocabularies.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding the required value set, relax the cardinality of
`EpisodeOfCare.statusHistory.status` from `1..1` to `0..1` (or, equivalently,
keep it `1..1` but change the binding strength to `extensible` so that
implementers can supply a code outside the core value set when needed).
Pair this with a constraint or guidance note stating that if the historical
`status` is not recorded, the `statusHistory` entry SHOULD still record at
least the `period`, and add a `dataAbsentReason` extension example for the
"truly unknown" case. No change is made to the value set itself.

#### Justification

This avoids adding semantically weak codes ("unknown", "other") to a
clinically-oriented status value set, which some PA members have historically
resisted because it weakens validator usefulness and can become a dumping
ground. The `dataAbsentReason` extension is FHIR's general mechanism for
"value is missing and here's why", and reusing it preserves the integrity of
the status code system. The trade-off vs. Disposition A is increased
implementer burden (extensions instead of inline codes) and the loss of a
single guaranteed code at every history entry, but it keeps the terminology
clean.

---

### Disposition C: Decline

#### Proposal

Decline the request and document the existing path. Mark FHIR-55296 as "Not
Persuasive" with a rationale that (a) `EpisodeOfCare.statusHistory` is
itself optional (`0..*`), so an implementer who genuinely cannot determine
the historical status simply omits the `statusHistory` entry rather than
inventing one with `unknown`; and (b) where some history is known but a
single transition is unknown, the `dataAbsentReason` extension on
`status` already provides a standards-based escape valve without modifying
the value set.

#### Justification

The premise that "the containing element [becomes] impossible to populate"
is only true if implementers feel obliged to emit a `statusHistory` entry
they cannot fully describe. Because `statusHistory` is `0..*`, omission is
always legal. Adding `unknown`/`other` to a required value set has a real
cost: it weakens the meaning of the binding for everyone, and it tends to
get used as a dumping ground that downstream consumers must then handle.
Existing FHIR mechanisms (omission, `dataAbsentReason` extension,
Provenance for reason-of-change context as Lloyd noted in the 2016
implementers thread) cover the legitimate use cases without changing core
terminology.

---

### Recommendation

**Recommended disposition:** A

The value-set escape-valve pattern is well-established across FHIR Core and
PA already follows it for `Encounter.status` (which includes `unknown`).
Disposition A keeps `EpisodeOfCare.status` and
`EpisodeOfCare.statusHistory.status` symmetric, is the minimum-surprise
change for implementers, and is backward-compatible. Disposition B is
technically defensible but pushes implementers to extensions for a case
that PA has chosen to handle inline elsewhere, which is inconsistent.
Disposition C is true on the strict reading of cardinality (you can omit
the entry), but the submitter's lived complaint — that they have *some*
history they want to record but can't pick a code — is real, and PA has
honoured similar requests on neighbouring resources. Given that the
reporter is Lloyd McKenzie and the ticket is filed at Highest priority,
the expectation is a concrete remedy, and the escape-valve addition is the
cheapest, most consistent one.
