## Summary

The ticket asks Patient Administration to consolidate the two cancellation-related
elements on the `Appointment` resource — `cancellationReason` and
`cancellationDate` — by grouping them under a new `cancellation` BackboneElement
(0..1) with two child elements (`cancellationDate : dateTime 0..1` and
`cancellationReason : CodeableConcept 0..1`). The reporter notes that the
existing invariants `app-4` and `app-7` would need to be updated to reference
the new path, and that the narrative in §15.13.5.3 (the "reject-flow"
description on the Appointment page) would also need to be reviewed.

The motivation is structural cleanliness: `cancellationReason` currently sits
alongside `status` (mirroring the `statusReason` pattern used in other
resources), while `cancellationDate` sits next to the resource's creation date
near the bottom of the resource. The reporter would prefer those two pieces of
information to live together. No cross-referenced Jira tickets, Zulip threads,
or GitHub items were surfaced for this ticket via `fhir-augury-cli`
(`cross-referenced` returned 0 hits, source-wide `search` for `FHIR-53971`
returned 0 results, and `links` returned an empty list).

The ticket has already been discussed at the January 2026 WGM (PA Wed Q1) and a
modified resolution has been recorded — see the Comments section below.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Introduce a new `Appointment.cancellation` BackboneElement (cardinality 0..1)
on the `Appointment` resource and move the existing `cancellationReason` and
`cancellationDate` elements underneath it as children:

- `Appointment.cancellation` : BackboneElement 0..1
  - `Appointment.cancellation.date` : dateTime 0..1 (formerly
    `Appointment.cancellationDate`)
  - `Appointment.cancellation.reason` : CodeableConcept 0..1 (formerly
    `Appointment.cancellationReason`)

Rewrite invariants `app-4` and `app-7` to reference the new paths
(`cancellation.date` / `cancellation.reason` rather than the flat
`cancellationDate` / `cancellationReason`). Update the narrative in
§15.13.5.3 ("reject-flow") on the Appointment resource page to use the new
element paths. Update Appointment examples that populate either field, and
add changelog / R4-to-R6 conversion notes flagging this as a breaking change
to the element paths.

#### Justification

Grouping closely-related cancellation metadata under a single BackboneElement
is a common pattern in FHIR (e.g., the various `*.performed[x]` /
`*.note` style groupings). It is structurally cleaner than two parallel
top-level fields and makes it easier to add future cancellation metadata
(e.g., who cancelled, cancellation channel) without continuing to widen the
top-level resource surface. The reporter has named the affected invariants
and narrative, so the work is well-scoped on the spec side.

### Disposition B: Alternative Approach

#### Proposal

Keep both elements at the top level of `Appointment` (no BackboneElement,
no path changes, no invariant changes, no narrative changes), but reorder
the resource so that `cancellationDate` appears immediately after
`cancellationReason` (i.e., move `cancellationDate` up out of its current
position next to the creation date and place it directly below
`cancellationReason`, which itself sits under `status`). Leave `app-4` and
`app-7` and the §15.13.5.3 narrative untouched. This is the disposition
captured in the WGM minutes recorded by `egelkraut` on 2026-01-28
("Persuasive with Modification").

#### Justification

This addresses the underlying complaint — the two cancellation fields are
visually and conceptually separated on the resource page — without the
backwards-compatibility cost of changing element paths between R4/R5 and
R6. Promoting the existing fields into a BackboneElement is a breaking
change: every existing Appointment instance, profile, search parameter,
mapping, and code generator that references `Appointment.cancellationDate`
or `Appointment.cancellationReason` would have to be updated. Co-locating
them in the differential preserves compatibility with R4 and R5, keeps
existing instances and invariants valid, and matches Brian Postlethwaite's
observation that `cancellationReason` is intentionally placed next to
`status` to mirror the `statusReason` pattern (so leaving it under `status`,
with `cancellationDate` moved next to it, gives the best of both worlds).

### Disposition C: Decline

#### Proposal

Take no action. Retain the current `Appointment` structure and ordering as
published in 6.0.0-ballot4, including the existing positions of
`cancellationReason` (under `status`) and `cancellationDate` (next to the
creation date), and the existing wording of `app-4`, `app-7`, and §15.13.5.3.

#### Justification

The current layout is a deliberate consequence of two cross-cutting
patterns: keeping `*Reason` next to its corresponding status (like
`statusReason` elsewhere in FHIR), and grouping creation/cancellation
timestamps together near the bottom of the resource alongside other "audit
style" patterns. Reordering or restructuring purely for cosmetic clustering
risks breaking either of those patterns. This rationale is the substance of
Brian Postlethwaite's 2026-02-01 comment ("So not an easy answer here").
The ticket is `Medium` priority, the affected fields are already discoverable
in the resource definition, and no implementer pain has been reported.

### Recommendation

**Recommended disposition:** B

The WGM has already converged on this outcome ("Persuasive with
Modification"), and it is the disposition that best balances the reporter's
concern (cancellation fields should sit together) against Brian
Postlethwaite's concern (the existing positions follow established
patterns). Promoting the elements into a BackboneElement (Disposition A)
would be a hard breaking change to two elements that have been in the
resource since R4 — every R4/R5 Appointment instance and every profile or
mapping that references `Appointment.cancellationDate` or
`Appointment.cancellationReason` would need to be migrated, which is
disproportionate to the cosmetic improvement gained. Declining outright
(Disposition C) ignores a reasonable usability complaint that can be fixed
without any normative impact. Disposition B costs only a re-ordering in the
differential, leaves `app-4`, `app-7`, the existing narrative, and all
existing examples and instances unchanged, and delivers the reporter's
intent.
