## Summary

Robert Dieterle requests two related changes to the `OrganizationAffiliation`
resource in FHIR Core. First, the resource description should be updated so
that it explicitly allows `OrganizationAffiliation` to express the
relationship between a single organization and one or more **networks,
locations, and healthcare services**, without requiring a relationship
between two distinct organizations. Second, `OrganizationAffiliation` should
be usable in **any resource that references `Organization`** — i.e., it
should be a valid target wherever a reference to `Organization` is currently
permitted. The cross-reference index returned no related Jira tickets, so
the request stands on its own description; however, the PA WG has confirmed
on Zulip that `OrganizationAffiliation` is now on the normative track,
which raises the stakes for any description/usage clarifications adopted at
this stage.

## Details

**Description:**

> Update description to allow use of Organization Affiliation to define
> relationships between an organization and one or more networks,
> locations, healthcare services, not dependent on defining a relationship
> between two organizations.
>
> Allow OrganizatonAffiliation to be used in any resource that references
> organization

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply both changes literally to the FHIR Core `OrganizationAffiliation`
resource:

1. **Description / Scope text:** Revise the resource definition (the
   `<short>` and `<definition>` strings on `OrganizationAffiliation` and
   the resource introduction page) so that the relationship can be
   between one `Organization` and one or more of `Network`,
   `Location`, or `HealthcareService`, without requiring a
   `participatingOrganization`. Concretely, relax the wording on the
   `participatingOrganization` element so it is clearly optional and
   document that an `OrganizationAffiliation` instance is valid when only
   `organization` plus at least one of `network`, `location`, or
   `healthcareService` is populated. Add an invariant such as
   `organizationaffiliation-1`: "An OrganizationAffiliation SHALL
   reference an organization, and SHALL also reference at least one of
   participatingOrganization, network, location, or healthcareService."
2. **Allowed reference target:** Add `OrganizationAffiliation` to the
   `targetProfile` list of every `Reference(Organization)` element across
   FHIR Core resources (e.g., `Patient.managingOrganization`,
   `Encounter.serviceProvider`, `HealthcareService.providedBy`,
   `Location.managingOrganization`, `Device.owner`,
   `Practitioner`/`PractitionerRole`/`Account`/`Coverage` org
   references, etc.), so that any place that may reference an
   `Organization` may instead reference an `OrganizationAffiliation`.

#### Justification

Part 1 aligns the resource definition with how implementers (NDH, Da
Vinci PDex Plan-Net) already use `OrganizationAffiliation` in the field
to relate a single organization to its networks, locations, and
services. Tightening the scope through an invariant rather than a
required cardinality avoids breaking existing instances. Part 1 is also
timely: PA has confirmed on Zulip that `OrganizationAffiliation` is on
the normative track, so the description should be authoritative before
lock-down. Part 2 directly delivers the second sentence of the request.

### Disposition B: Alternative Approach

#### Proposal

Adopt the **description / scope clarification (Part 1) only**, and
**decline the wholesale reference-target expansion (Part 2)**. Instead
of adding `OrganizationAffiliation` to every `Reference(Organization)`
in core, document the intended pattern explicitly in the
`OrganizationAffiliation` resource page and the `Organization` page:

- Update `OrganizationAffiliation.participatingOrganization` to
  cardinality `0..1` (already the case in current core; reaffirm in
  prose) and add the invariant from Disposition A so that
  network/location/healthcareService alone are sufficient.
- Add a "Boundaries and Relationships" note on
  `OrganizationAffiliation` and on `Organization` describing when to use
  one versus the other, and clarifying that
  `OrganizationAffiliation` is **not** itself a kind of `Organization`
  and therefore is referenced through dedicated affiliation searches
  (`_has`, `_include`) rather than by substituting it into existing
  `Reference(Organization)` slots.
- Provide a small number of canonical examples in
  `source/organizationaffiliation/` covering the
  Organization+Network, Organization+Location, and
  Organization+HealthcareService patterns.

#### Justification

The second half of the request — making `OrganizationAffiliation` a
valid target *anywhere* `Organization` is referenced — is a much larger
change than the description tweak. `Reference(Organization)` is used
across dozens of resources whose semantics assume an `Organization`
(e.g., `Patient.managingOrganization` is the org that manages the
patient record; substituting an `OrganizationAffiliation` muddles
ownership semantics). It would also be a normative-impacting change to
many resources outside PA's stewardship, requiring sign-off from
multiple work groups and breaking a large number of existing
constraints, profiles (US Core, IPA, IPS), validators, and server
implementations. The Plan-Net and NDH precedent shows that the
existing pattern (point at `Organization`, navigate to affiliation via
`OrganizationAffiliation:participating-organization` or
`OrganizationAffiliation:primary-organization` with `_include`/`_has`)
already works without changing reference targets.

### Disposition C: Decline

#### Proposal

Decline both parts of the request, with the recommendation that the
submitter file separate, more narrowly scoped tickets if there is a
specific implementation use case that is blocked by the current
definition. Rationale category: **insufficiently specified**.

#### Justification

The ticket is two sentences long, has no use-case narrative, no
example, no comments, no cross-references, and no linked
implementation guide tickets. Part 1 (description change) is
ambiguous — "allow use" can be read either as a normative widening or
a documentation clarification — and Part 2 (allow as a reference
target everywhere) is a sweeping change that, taken literally, would
touch nearly every FHIR Core resource and break implementer
expectations. Acting on either part without a concrete use case and
without the originating work group(s) for affected resources risks
introducing more confusion than it removes, particularly while
`OrganizationAffiliation` is being moved to normative.

### Recommendation

**Recommended disposition:** B

Disposition B accepts the legitimate, low-risk part of the request —
the description/scope clarification, which simply codifies how NDH and
Da Vinci PDex Plan-Net already use `OrganizationAffiliation` and is
timely given the resource's move to normative — while declining the
disruptive, under-specified part: redefining what every
`Reference(Organization)` in core may point at. That second change
crosses many work-group boundaries, would force re-balloting of
unrelated resources, and is not justified by any concrete blocker
captured in the ticket. PA can deliver real value here by tightening
the prose plus adding an invariant and worked examples; if a
downstream IG genuinely needs `OrganizationAffiliation` as a reference
target on a specific element, it can profile that element or open a
targeted ticket against the owning work group.
