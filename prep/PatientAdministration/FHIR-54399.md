## Summary

The reporter asks Patient Administration to add explicit guidance on how
to identify the "original" (already-booked) appointment that a
waitlisted appointment is intended to replace if the waitlist
opportunity is taken. `Appointment.replaces` already exists, but its
definition currently frames it as replacing a *cancelled* appointment,
not as a forward-looking link from a waitlisted candidate to a still-
booked appointment that may be cancelled later. The requested outcome
is normative/spec guidance: when a patient is offered a less-desirable
slot and also placed on a waitlist, the waitlisted `Appointment` should
be able to point at the booked one so that, upon acceptance of the
waitlist slot, the original can be cancelled (or flagged for manual
follow-up).

The only cross-reference recorded against this ticket is the FHIR
artifact `Appointment.replaces` itself; no other Jira tickets, GitHub
items, or Zulip threads are directly linked. Adjacent Zulip activity in
the `implementers` stream (e.g., the *automatic outbound calls* and
*AppointmentResponse* topics) shows the community has previously used
`status = waitlist` together with `requestedPeriod` to model this
pattern, but without a documented link back to the booked appointment —
which is exactly the gap this ticket targets.

## Details

**Description:**

For waitlisted appointments, there should be a way to identify an
appointment to replace in case this one occurs. There is a replaces
element, that could be used for this, but there is no guidance about it
being used for this purpose. In fact the description says it is
replacing a cancelled appointment. Some guidance on how to do specify
this link should be added so that if the waitlisted appointment is
accepted, the original (less desirable) appointment can be cancelled or
somehow flagged for manual follow-up.

**Resolution Description (Persuasive — drafted by Cooper Thompson,
2026-02-24):**

- Update the workflow table to update the `Appointment.replaces`
  attribute usage.
- Update the `Appointment.replaces` definition to include verbiage about
  being used for a *potentially* cancelled appointment due to a
  waitlist appointment being created.
- Add another workflow table for the use case of a stand-alone waitlist
  appointment in cases where an appointment can't originally be booked
  first.

The draft language proposed in the resolution describes two flows:

1. **No available slot:** create the appointment with `status =
   waitlist` and an optional `requestedPeriod`. Allocation is then a
   triage decision by the scheduling organisation.
2. **Booked + waitlisted in parallel:** book an available (but less
   desirable) slot, and *also* create a second `Appointment` with
   `status = waitlist` whose `replaces` points at the booked
   appointment, with a `requestedPeriod` covering the preferred window.
   If a better slot opens during that window, the patient is offered
   it; on acceptance, the waitlisted appointment is booked into the
   newly opened slot and the previously booked appointment is
   cancelled.

The draft also flags an open design question (a "ToDo" left for the
work group):

> Determine if we want to use `replaces` for two meanings: 1) one
> appointment **has actually** replaced another appointment (resolved
> waitlist) and 2) one appointment **is a potential** replacement for
> another appointment (unresolved waitlist).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the drafted resolution as written:

1. Extend the `Appointment.replaces` ElementDefinition.definition /
   .comment text in the FHIR Core Appointment resource to explicitly
   permit forward-looking use: a waitlisted appointment MAY point at a
   still-booked appointment that the waitlisted one would replace if
   the waitlist offer is taken. Make clear that, until that happens,
   the referenced appointment is not yet cancelled.
2. Update the existing Appointment scheduling workflow table on the
   Appointment narrative page to show the
   booked-plus-parallel-waitlist flow, including the `replaces` link
   from the waitlisted `Appointment` to the booked one and the
   subsequent cancel-on-accept transition.
3. Add a second workflow table (or section) for the stand-alone
   waitlist case (`status = waitlist` with a `requestedPeriod`, no
   prior booked appointment), to make explicit that this is a
   first-class scenario, not just a degenerate case of rebooking.
4. Resolve the open "ToDo" by stating that `replaces` MAY be used for
   *both* "has actually replaced" and "is a potential replacement"
   semantics, with the distinction conveyed by the referencing
   appointment's `status` (`booked` vs. `waitlist`/`proposed`). Add a
   short note to that effect alongside the element definition.

#### Justification

The mechanism the reporter wants already exists structurally
(`Appointment.replaces`, `status = waitlist`, `requestedPeriod`); the
only gap is documentation. A definition tweak plus workflow-table
additions is the lowest-risk way to close that gap and is consistent
with how Patient Administration has historically handled scheduling
guidance in past cycles. Making the dual semantics of `replaces`
explicit (resolved vs. potential, disambiguated by the referencer's
`status`) is also non-breaking — existing implementations that only
use `replaces` post-cancellation continue to be conformant.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `Appointment.replaces` strictly retrospective ("has actually
replaced") and instead introduce a dedicated forward-looking link via a
standard FHIR Extension on `Appointment` — e.g.,
`http://hl7.org/fhir/StructureDefinition/appointment-replacement-candidate`
of type `Reference(Appointment)` — published from the FHIR Core
`fhir-extensions` pack. Update the Appointment narrative and workflow
table(s) to describe the same waitlist-with-fallback-booked flow as
Disposition A, but using the new extension on the waitlisted
appointment instead of overloading `replaces`.

#### Justification

This avoids overloading a single element with two temporally distinct
meanings (potential vs. actual replacement), which the resolution's own
"ToDo" flags as an open concern. Implementations that today scan
`Appointment.replaces` to drive cancellation/audit logic do not have to
re-evaluate whether each hit is "real" or "speculative", reducing the
risk of accidental cancellations triggered by stale waitlist links.
Trade-off versus A: it costs an extension and a small amount of
implementer churn, and it does not benefit from `replaces` already
being core. The PA work group will need to weigh "minimal spec change"
(A) against "cleaner separation of intent" (B).

---

### Disposition C: Decline

#### Proposal

Decline the request. Document in the resolution that the existing
combination of `Appointment.status = waitlist` plus `requestedPeriod`,
together with implementer-side correlation (e.g., a local identifier or
an out-of-band scheduling-system link), is sufficient, and that adding
spec-level guidance on a forward-looking `replaces` would over-specify
a workflow that varies materially between scheduling systems.

#### Justification

This is defensible only on the grounds that (a) the structural pieces
already exist, (b) no downstream Jira tickets, GitHub PRs, or Zulip
threads are pressuring for the change, and (c) the resolution itself
is still a draft with an unresolved semantic question. However, the
core complaint — that the *current* `replaces` definition explicitly
ties itself to a *cancelled* appointment, leaving the waitlist case
ambiguous — is a real documentation defect, so a pure decline leaves
implementers with conflicting signals.

---

### Recommendation

**Recommended disposition:** A

The change is small, the structure is already in place, and the
reporter's pain point is genuinely a documentation gap rather than a
missing capability. Disposition A delivers the requested clarity with
two narrative-page edits and a definition refinement, and it resolves
the dual-semantics "ToDo" by leaning on `Appointment.status` (which
implementers already inspect) to disambiguate "potential" from "actual"
replacement. Disposition B is the right answer *if* PA decides the
overloading concern outweighs minimality — it is worth a brief
discussion at the call, especially with input from implementers who
already mine `Appointment.replaces`. Disposition C is not recommended:
the current `replaces` definition is actively misleading for the
waitlist case, and leaving it unchanged would perpetuate that.
