## Summary

FredTrotter requests that the "CMS Enrollment In Good Standing" extension
recently added to the NDH IG (as resolution of
[FHIR-55990](https://jira.hl7.org/browse/FHIR-55990), applied via
[HL7/fhir-us-ndh#10](https://github.com/HL7/fhir-us-ndh/pull/10)) be broken
apart into more granular, single-concern boolean extensions, and that the
extension names carry **agency prefixes** reflecting the responsible federal
data steward (CMS for Medicare enrollment and identity verification; HHS for
the exclusion list, which is an HHS-maintained data asset). Specifically, the
single "in good standing" concept would be replaced by four booleans:

- `cms_identity_verified` — has the practitioner completed CMS identity
  proofing (e.g., IAL2 login)?
- `cms_medicare_enrollment` — is the practitioner currently enrolled in
  Medicare?
- `hhs_in_exclusion_list` — is the entity on the HHS OIG exclusion list?
- `cms_aligned_with_data_network` — is the entity aligned with the CMS data
  network? (replaces/renames the existing `ndh-AlignedWithCMSDataNetwork`
  extension)

This ticket is a direct follow-up and refinement to
[FHIR-55990](https://jira.hl7.org/browse/FHIR-55990) ("How to show practicioner
verification status in NDH"), whose "Persuasive with Modification" resolution
added two boolean extensions ("NDH CMS Enrollment In Good Standing" and "NDH
Aligned with CMS Data Network") to PractitionerRole, Organization, and (for
the first) Verification. It also overlaps conceptually with
[FHIR-56337](https://jira.hl7.org/browse/FHIR-56337), which argues the same
extensions should not be on the Verification profile because they represent
attributes of the entity, not provenance of the verification act. Taken
together, FHIR-56331 proposes a rename + split, while FHIR-56337 proposes a
placement correction — both need to be reconciled.

## Details

**Description:**

After some internal debate, we need to break apart "in good standing" further.
We also wanted to use agency prefixes since the exclusion list is actually
maintained as an HHS data asset. So the new extensions would be:

- `cms_identity_verified`: true | false
- `cms_medicare_enrollment`: true | false
- `hhs_in_exclusion_list`: true | false
- `cms_aligned_with_data_network`: true | false

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Retire the single `ndh-CMSEnrollmentInGoodStanding` boolean extension
introduced by FHIR-55990 / PR #10 and replace it with four new boolean
extensions, named with agency prefixes reflecting the authoritative data
steward:

- `cms-identity-verified` (Boolean) — Practitioner, PractitionerRole.
  Indicates that CMS-level identity proofing (e.g., IAL2) has been
  completed for the individual.
- `cms-medicare-enrollment` (Boolean) — Practitioner, PractitionerRole,
  Organization. Indicates current Medicare enrollment.
- `hhs-in-exclusion-list` (Boolean) — Practitioner, PractitionerRole,
  Organization. Indicates presence on the HHS OIG List of Excluded
  Individuals/Entities (LEIE).
- `cms-aligned-with-data-network` (Boolean) — Practitioner, PractitionerRole,
  Organization (rename of the existing `ndh-AlignedWithCMSDataNetwork` to use
  the explicit `cms-` agency prefix for consistency).

Remove all four from the Verification profile's context-of-use per the
concern raised in [FHIR-56337](https://jira.hl7.org/browse/FHIR-56337) —
they are attributes of the entity, not of a verification event. Provide a
short migration note in the IG change log pointing implementers from the
old extension URL(s) to the new ones, and regenerate the two affected
profiles (NDH Practitioner / PractitionerRole / Organization).

#### Justification

- Each of the four concepts is independently observable by a different
  workflow / data source (IAL2 proofing, PECOS enrollment, OIG LEIE, NDH
  network alignment). Collapsing them into a single "good standing"
  boolean loses information and forces implementers to answer "why
  false?" out of band.
- Agency-prefixed names (`cms_*`, `hhs_*`) make the data steward
  explicit, which is important because the LEIE is maintained by HHS OIG
  (not CMS) and future additions from other agencies (e.g., DEA, SAMHSA)
  will follow the same pattern cleanly.
- Moving these off the Verification profile aligns with the argument in
  [FHIR-56337](https://jira.hl7.org/browse/FHIR-56337) and with the
  standard FHIR distinction between entity state and verification
  provenance.
- The change is contained to the NDH IG, pre-publication, and thus
  carries minimal downstream breakage — the prior PR #10 artifacts have
  not yet been balloted at the current state.

*Caveats:* Snake_case names (`cms_identity_verified`) as literally
requested do not match FHIR's extension-ID convention, which is
kebab-case. Accept the intent but normalise IDs to `cms-identity-verified`,
`cms-medicare-enrollment`, `hhs-in-exclusion-list`, and
`cms-aligned-with-data-network`. Extension URLs in the NDH canonical base
would follow the same pattern.

---

### Disposition B: Alternative Approach

#### Proposal

Replace the four proposed single-purpose boolean extensions with **one
coded extension** on Practitioner, PractitionerRole, and Organization whose
value is a `CodeableConcept` bound (Extensible) to a new NDH value set
`ndh-regulatory-status-flags` with codes such as:

- `cms-identity-verified`
- `cms-medicare-enrollment`
- `hhs-oig-excluded`
- `cms-aligned-with-data-network`

The extension would be repeating (0..*), each occurrence carrying one code
plus an optional `asserted` boolean and `period` sub-extension. Like
Disposition A, remove the extension from the Verification profile's
context-of-use (FHIR-56337).

#### Justification

- A single extensible coded extension scales better than a growing list of
  hand-rolled boolean extensions. If the next request adds DEA
  registration status, SAMHSA waiver status, or a state-level medical
  board action, the value set grows — no new StructureDefinitions, no new
  extension URLs, no new profile edits.
- Each flag naturally wants an assertion timestamp and an asserter; a
  bare boolean can't carry those without another parallel extension.
- It keeps the NDH extension surface smaller, which lowers the
  maintenance burden for the WG and for implementers generating/validating
  profiles.
- Trade-off vs. A: implementers lose the ergonomic "is this boolean true?"
  access pattern and must iterate the repeating extension. For simple
  directory-query use cases that is modest additional work.

---

### Disposition C: Decline

#### Proposal

Decline the rename/split. Keep the two extensions introduced by
FHIR-55990 (`ndh-CMSEnrollmentInGoodStanding`,
`ndh-AlignedWithCMSDataNetwork`) as-is in name and shape, and address
only the Verification-profile placement concern from
[FHIR-56337](https://jira.hl7.org/browse/FHIR-56337) separately.
Rationale category: **premature re-litigation of a recently-applied
decision**.

#### Justification

- FHIR-55990 resolved just days earlier (applied via PR #10) after WG
  discussion; rapidly reversing that decision fragments the historical
  record and creates churn for implementers already prototyping against
  the current extensions.
- "Good standing" as a single aggregate flag is actually useful to
  consumers who only want the yes/no answer and do not want to reason
  about the component checks.
- The HHS-vs-CMS stewardship concern can be addressed in the extension
  *definition* text (who maintains the underlying data) without a
  rename.
- The FHIR-56337 scope (remove from Verification profile) is an
  independent correction and can stand on its own.

---

### Recommendation

**Recommended disposition:** A

FHIR-55990 was resolved quickly and is still pre-publication, so correcting
it now is far cheaper than doing so after the IG ballots. The four concepts
the submitter names really are distinct — identity proofing (an NDH/IDP
event), Medicare enrollment (a PECOS state), OIG exclusion (an HHS data
asset), and data-network alignment (an NDH relationship) — and they have
different authoritative stewards, different refresh cadences, and different
possible future extensions (e.g., dates, assertion sources). A single
"in good standing" boolean loses that signal and, more importantly, makes
stewardship ambiguous at exactly the moment the NDH IG is trying to
establish trust semantics. Agency prefixing (`cms-*`, `hhs-*`) is a small,
cheap convention that pays off the first time a non-CMS flag is added.
Disposition B is architecturally cleaner but is a larger change than the
reporter asked for and introduces a new value-set lifecycle; that is a
reasonable next step if future additions prove the boolean approach
unsustainable, but it is not necessary to adopt now. Disposition C
underweights that FHIR-56337 independently flags the Verification-profile
placement as wrong — any cleanup pass should carry both corrections at
once, and Disposition A does that in a single coordinated edit to the NDH
IG. Normalise the names to kebab-case per FHIR convention when applying.
