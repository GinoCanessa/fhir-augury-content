## Summary

The reporter notes that the NDH `Group` profile fixes `Group.type` to
`practitioner`, and argues this is overly restrictive given that NDH is
positioning the resource (in conjunction with `CareTeam`) to represent
**non–patient-focused care coordination teams** for CMS initiatives such as
Accountable Care Organizations and Pay-for-Performance programs. They point
out that R5 expanded the `Group.type` value set to include codes such as
`careteam`, `healthcareservice`, and `organization`, and ask whether NDH
should likewise broaden the allowed types on its Group profile rather than
fixing the value to `practitioner`.

No cross-referenced Jira tickets, Zulip threads, or GitHub PRs/issues were
surfaced for this ticket; the only cross-reference is a content mention of
the FHIR element `Group.type`.

## Details

**Description:**

> It may be overly restrictive to require `Group.type` to be fixed to
> **practitioner**, especially since we are substituting it with **CareTeam**.
> Our intent is to represent CareTeams within the directory to support
> **non–patient-focused care coordination teams**, particularly those needed
> for CMS initiatives such as **Accountable Organizations, Pay for
> performance.**
>
> Look at R5 group.type, it has expended to include codes such as careteam,
> healthcareservice, organization. Should we take those codes into

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Relax the NDH `Group` profile so that `Group.type` is no longer fixed to
`practitioner`. Replace the fixed value with a **required** (or
**extensible**) binding to a value set that mirrors the R5 `Group.type`
expansion — specifically allowing at least `practitioner`, `careteam`,
`healthcareservice`, and `organization` (and, as appropriate, `device`
and `careteam`). The profile would still constrain `Group.member.entity`
to the corresponding directory profiles for each allowed type, so that a
`type = careteam` Group references NDH `PractitionerRole`/`CareTeam`
members, a `type = organization` Group references NDH `Organization`
members, etc.

#### Justification

This directly addresses the reporter's stated need: representing
non–patient-focused care coordination teams (ACOs, P4P arrangements) as
groupings of organizations, care teams, or services, rather than
shoehorning them into a `practitioner`-typed Group. Aligning the binding
with the broader R5 `Group.type` codes keeps NDH consistent with the
direction of the base resource and with FHIR-wide consistency principles
for value-set bindings. It also avoids the modeling smell of using
`Group.type = practitioner` for collections whose members semantically
are not practitioners.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the `Group` profile as-is (fixed `type = practitioner`) and instead
**introduce dedicated NDH Group sibling profiles** for each additional
needed grouping — e.g., `ndh-OrganizationGroup` (fixed
`type = organization`), `ndh-CareTeamGroup` (fixed `type = careteam`),
and `ndh-HealthcareServiceGroup` (fixed `type = healthcareservice`) —
each with their own member-entity constraints. The current profile
remains the canonical "group of practitioners" artifact; new profiles
cover the new use cases. NDH narrative and capability statement are
updated to point implementers at the right profile for each scenario.

#### Justification

This preserves backward compatibility for any existing implementer that
relies on `Group.type` being fixed to `practitioner` on the current
profile, and produces highly typed profiles whose constraints can be
validated mechanically without conditional logic on `Group.type`. It
also makes capability statements and search-parameter sets clearer per
use case. The trade-off is profile proliferation and the need to teach
implementers to choose among siblings — but it avoids touching the
existing profile's contract.

---

### Disposition C: Decline

#### Proposal

Decline the change. NDH already provides `CareTeam` (and the supporting
`Organization`, `HealthcareService`, etc. profiles) for representing
non-practitioner aggregations; the `Group` profile in NDH is
intentionally scoped to "groups of practitioners" and should remain
fixed to `type = practitioner`. The reporter's use cases (ACOs, P4P
collectives) should be modeled with `Organization` (with appropriate
`type` codes) and `CareTeam` rather than by broadening `Group`.

#### Justification

`Group` overlaps semantically with `Organization`, `CareTeam`, and
`HealthcareService`, and FHIR guidance generally cautions against using
`Group` when a more specific resource fits. ACOs are organizations and
should be represented as `Organization`; care coordination teams are
`CareTeam`s. Broadening `Group.type` invites duplicate modeling and
ambiguity in the directory ("is this ACO an Organization or a Group of
Organizations?"). Declining keeps the resource boundaries clean and
avoids a breaking change to the existing profile.

---

### Recommendation

**Recommended disposition:** A

The ticket identifies a real gap: NDH has already accepted `CareTeam` as
the substitute for practitioner-only Group semantics, which means the
fixed `type = practitioner` constraint on the `Group` profile no longer
matches NDH's own modeling intent. R5 has explicitly broadened
`Group.type` precisely to cover the cases the reporter cites
(`careteam`, `healthcareservice`, `organization`), so Disposition A is
the most consistent with both the base resource direction and NDH's
stated scope (CMS ACO / P4P aggregations).

Disposition B (sibling profiles) is defensible if the workgroup wants to
preserve the existing profile's `practitioner`-only contract for
backward compatibility, and the workgroup should weigh that explicitly —
but it pays a real cost in profile proliferation for what is
fundamentally a binding-strength change. Disposition C understates the
reporter's modeling problem: ACOs and P4P collectives are not always
cleanly an `Organization` or a single `CareTeam`, and the directory
needs a way to represent the **grouping** itself.

The workgroup should confirm (a) the exact NDH `Group` profile
StructureDefinition to amend, (b) the target value set (R5 `Group.type`
or an NDH-curated subset), and (c) whether existing implementers
depend on `Group.type` being fixed to `practitioner` before finalizing
Disposition A.
