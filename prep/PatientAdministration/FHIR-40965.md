## Summary

The reporter (Bas van den Heuvel) asks that the Real Time Location Services
(RTLS) FHIR specification be extended with additional profiles and examples so
that its conformance artifacts better reflect the scope described in the
narrative. Specifically, the request is to (1) add an RTLS subscription
notification profile showing the structure of a subscription-based
notification, (2) add profiles for each notification type defined in
section 3 of the spec, covering both message-based and subscription-based
notifications, and (3) add examples for each event type in both message and
subscription format. The ticket has been triaged with a proposed resolution
of "Not Persuasive" because the workgroup has decided to remove Subscriptions
from the RTLS specification, eliminating the need for the
subscription-oriented profiles and examples called for here. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items surfaced for
this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the RTLS IG with the full set of conformance artifacts the reporter
requested:

1. Define a `RTLSSubscriptionNotification` profile on `Bundle` (or the
   appropriate Subscription topic-based notification Bundle shape) that
   fixes the slice structure expected for an RTLS subscription-based
   notification payload.
2. For each notification type enumerated in section 3 of the RTLS spec,
   author two profiles — one message-based (a `MessageHeader` + content
   profile pair) and one subscription-based (a profile on the
   notification Bundle constraining the focus/notification-event data).
3. For every event type, publish at least one worked example in both the
   message-based and the subscription-based serialisation, wired to the
   relevant profiles so they validate.

#### Justification

The reporter's underlying concern — that the narrative describes a set of
events and notification modes that the conformance artifacts do not fully
cover — is a legitimate IG quality observation. Accepting would give
implementers concrete, validatable shapes for every notification type and
eliminate ambiguity between the two transport modes. It would also bring
RTLS closer in line with the "profile + example per event" pattern used by
other notification-oriented IGs.

---

### Disposition B: Alternative Approach

#### Proposal

Close this ticket as addressed by the architectural decision to drop
Subscriptions from RTLS, but partially honour the request by ensuring the
remaining (message-based) side of the spec is well-covered:

- Do **not** author any subscription-notification profiles or
  subscription-format examples.
- Audit the message-based notification types in section 3 of the RTLS
  spec and, where a notification type lacks either a profile or a worked
  example, add one so that every retained event type has at least one
  profile and one example in message format.
- Note in the RTLS IG narrative that subscription-based delivery has been
  removed from the spec, so implementers are not left expecting the
  artifacts requested here.

#### Justification

This preserves the useful half of the reporter's feedback (better
example/profile coverage of message-based events) while respecting the
workgroup's decision to remove Subscriptions. It avoids investing authoring
effort in artifacts for a delivery mode that is being withdrawn, yet
prevents the ticket from being closed without any improvement to the
conformance coverage that motivated it.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** (as currently proposed on the ticket)
with the stated rationale: the RTLS specification is being revised to
remove Subscription-based notification entirely, so adding subscription
notification profiles and subscription-format examples is no longer in
scope. Any remaining message-based profile/example gaps should be raised
as separate, narrower tickets rather than carried under this broad
request.

#### Justification

The ticket's scope is anchored on subscription-based notification
artifacts ("RTLS subscription notification profile", "subscription based
notifications", "both message and subscription format"). With
Subscriptions being removed from the RTLS spec, a majority of what was
asked for cannot be built, and the remaining message-only slice is not
what this ticket was filed to request. The unresolved 2025-06-19 comment
from the reporter ("What kind of input do you require?") also suggests
the original author has not re-scoped the request, making it cleaner to
decline this one and open a focused follow-up if gaps remain.

---

### Recommendation

**Recommended disposition:** C

The workgroup has already decided to remove Subscriptions from the RTLS
specification, which invalidates the bulk of what this ticket asks for (a
subscription notification profile, subscription-format profiles for every
event type, and subscription-format examples). Trying to partially honour
the ticket under its current wording would blur the scope of the ticket
and the scope of the spec change. Declining as Not Persuasive — matching
the resolution already recorded on the ticket — is the cleanest outcome.
If, after the Subscription removal lands, the workgroup still sees
message-based profile/example gaps in section 3, those should be filed as
new, narrowly-scoped tickets so the work can be tracked and balloted on
its own merits.
