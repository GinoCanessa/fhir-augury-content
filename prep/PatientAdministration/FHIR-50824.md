## Summary

Lloyd McKenzie reports that the value set bound to `Encounter.diagnosis.use`
is missing concepts that describe the diagnosis's relationship to the
encounter. He cites three motivating examples — "present on admission",
"primary complaint" (chief complaint), and "developed/exacerbated during
the encounter" (hospital-acquired) — and notes there are likely more. He
further argues that the existing value set actually conflates two
different ideas: the codes currently in it (e.g., `confirmed`,
`provisional`, `differential`) describe a property of the Condition
itself, not of the Condition's relationship to the Encounter, and so are
arguably mis-located. The request is to revise the value set so that the
codes it carries actually describe encounter-diagnosis roles.

The ticket is tightly coupled to its sibling
[FHIR-50827](https://jira.hl7.org/browse/FHIR-50827) (also Patient
Administration, Highest, terminology), which raises essentially the same
gap from the German Medical Informatics Initiative perspective and points
to `http://terminology.hl7.org/CodeSystem/diagnosis-role` as a candidate
source of codes. FHIR-50827 has been resolved **Persuasive with
Modification**: in R6 the chief complaint and admission diagnosis move to
`Encounter.reason` (bound to the broader `encounter-reason-use` value
set), billing-oriented "present on admission" lives on
`Account.diagnosis.onAdmission`, and "Discharge Diagnosis" is added to
the existing value set. PA's own resolution note flags TODO items still
open — co-morbidities, pre/post-op diagnoses (to be discussed with
Patient Care), and a draft narrative section in Encounter explaining
which diagnosis concept goes where. FHIR-50824 is referenced in the same
Zulip thread by Cooper Thompson as "we'll be addressing it as part of
those (FHIR-50827 and FHIR-50824)", so it is best treated as the
companion ticket whose resolution should mirror — and finish off — what
50827 started.

## Details

**Description:**

There are a number of other concepts that need to be captured about a diagnosis:

- is this an issue that was existing at the time the patient was admitted?
- is this the patient's primary complaint?
- is this an issue that developed or was exacerbated during the encounter?

There may be others.

The existing value set doesn't make clear the 'use' element is intended
to cover this breadth. And, quite frankly, the existing value set is
more a characteristic of the Condition than of the condition's
relationship with the encounter — it's getting at
"unconfirmed/provisional/differential" vs. "confirmed" which has nothing
to do with the encounter.

Suggest revising the value set to include concepts that show the
relationship of the diagnosis to the encounter, such as those I've
listed above.

## Related Zulip Discussions

### [implementers > Diagnosis Present on Admission not used?](https://chat.fhir.org/#narrow/channel/179166-implementers/topic/Diagnosis.20Present.20on.20Admission.20not.20used.3F)

The thread that motivated FHIR-50824 (and FHIR-50827). Lloyd McKenzie
opens by noting that the QI Core "Diagnosis Present on Admission"
extension is defined but never actually used by any QICore profile, and
asks whether "presence on admission" is only relevant in claim
situations. Bryn Rhodes confirms the QICore extension is a leftover —
QICore 6.0.0 switched to deriving PoA, principal procedure, and
principal diagnosis from Claim information instead, and the extension
was dropped entirely in QICore 7.0.0. Brian Postlethwaite suggests this
is precisely what `Encounter.diagnosis.use` is supposed to cover and
agrees the value set "could use some love". Cooper Thompson (Epic)
explains how Epic actually represents PoA — initially driven by CMS
claim submission, tracked on the Condition for encounter-diagnosis and
later flowed onto the Account/Claim — and notes interactions with the
CMS 72-hour rule. Lloyd argues a clinical use case also exists ("did
they have this rash when they were admitted?"); Cooper and Lloyd debate
whether `Condition.onsetDateTime` is sufficient. Cooper later issues a
correction that PoA is genuinely a clinical concept too, not just
billing, raising the question of whether a new
extension/element is needed. Cooper closes the thread by saying PA is
addressing the gap via FHIR-50827 and FHIR-50824, and points to a draft
Confluence narrative section that explains where each diagnosis concept
should live in FHIR. Consensus by the end: don't add a new top-level
flag yet, fix the bindings/value sets and the cross-resource story
first.

Lloyd McKenzie, Bryn Rhodes, Brian Postlethwaite, Cooper Thompson.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revise `ValueSet/encounter-diagnosis-use` to be a true "role of the
diagnosis within this encounter" set. Concretely:

- Remove (or move to a separate `condition-verification-status`-style
  binding) the codes that describe the Condition itself rather than its
  encounter role (`confirmed`, `provisional`, `differential`,
  `unconfirmed`, etc.).
- Repopulate the value set from
  `http://terminology.hl7.org/CodeSystem/diagnosis-role` (plus any
  additional UTG-managed codes needed) so it includes at least:
  `present-on-admission`, `chief-complaint` / primary, `discharge`,
  `hospital-acquired` / "developed or exacerbated during encounter",
  `comorbidity`, `pre-op`, `post-op`. Submit any missing codes to UTG.
- Update the narrative on the `Encounter.diagnosis.use` element
  definition to explicitly state that this element conveys the role of
  the diagnosis with respect to the encounter, not the verification
  status of the underlying Condition.
- Cross-link to `Encounter.reason.use` /
  `ValueSet/encounter-reason-use` and to
  `Account.diagnosis.onAdmission` so implementers know which slot to
  use for which concept.

#### Justification

This is the literal, minimum-surprise reading of Lloyd's request and it
aligns the value set with the resolution already adopted on FHIR-50827
(broader `encounter-reason-use`, "Discharge Diagnosis" added,
`Account.diagnosis.onAdmission` for billing PoA). The Zulip thread
established that `diagnosis-role` is already an HL7 Terminology code
system intended for this purpose and is in active use by national
projects (MII Kerndatensatz Fall). The existing codes really are about
the Condition rather than the encounter role, so removing them
eliminates a long-standing mis-binding rather than introducing new
breakage.

---

### Disposition B: Alternative Approach

#### Proposal

Mark FHIR-50824 as a duplicate of FHIR-50827 and roll its remaining
asks into that ticket's open TODO list rather than carrying two parallel
tickets:

- Close FHIR-50824 with resolution `Persuasive with Modification` and
  resolution text "Addressed by FHIR-50827."
- On FHIR-50827, expand the open TODO ("identify where to put
  co-morbidities, and pre/post-op diagnoses") to also cover Lloyd's
  three explicit cases (PoA, primary complaint, hospital-acquired) and
  the higher-level critique that the current value set describes the
  Condition rather than its encounter role.
- Defer the actual value-set rework to the FHIR-50827 implementation,
  ensuring the planned narrative section in the Encounter resource
  documents `Encounter.diagnosis.use` vs. `Encounter.reason.use` vs.
  `Account.diagnosis.onAdmission` vs. `Condition.onsetDateTime` so the
  conceptual carve-out from the Zulip thread is preserved.

#### Justification

sonja_ziegler's own comment on this ticket says "Both issues deal with
the value-set for `Encounter.diagnosis.use`. Resolution will be found on
the linked issue", and Cooper Thompson said the same thing in
#implementers. Carrying two open Highest-priority PA tickets that
demand a resolution to the same value set risks divergent edits and
makes it harder for the WG to track the remaining work. Using FHIR-50827
as the single seat of record consolidates the discussion, the
co-morbidity / pre-op / post-op TODOs, and the narrative-section work
into one ticket, while still committing to deliver the substance Lloyd
asked for.

---

### Disposition C: Decline

#### Proposal

Resolve as `Not Persuasive`: the encounter-diagnosis use case the ticket
points at is already addressed by R6 changes, namely

- admission diagnosis and chief complaint are expressed via
  `Encounter.reason` / `encounter-reason-use`,
- billing-oriented "present on admission" is captured on
  `Account.diagnosis.onAdmission`,
- "Discharge Diagnosis" is being added to the existing value set under
  FHIR-50827, and
- clinical onset can already be carried on `Condition.onsetDateTime`.

Do not change the existing `encounter-diagnosis-use` value set further;
direct implementers to the relevant element on each resource via
implementation guidance only.

#### Justification

The Zulip thread does explore an "is this fully solved already?" line
of argument: Bryn Rhodes notes QICore stopped using its PoA extension
because Claim already carries the information; Cooper Thompson
explained Epic's existing pattern (Condition + Account + Claim); and
the explicit R6 carve-out on FHIR-50827 already moves the
Lloyd-mentioned concepts (admission diagnosis, chief complaint) out of
`Encounter.diagnosis` entirely. Under that reading, no rework of
`encounter-diagnosis-use` is required and the ticket is asking the
value set to do work that has been redistributed across other elements
or other resources.

---

### Recommendation

**Recommended disposition:** B

The Zulip thread, the explicit linkage comment from sonja_ziegler, and
Cooper Thompson's "addressing it as part of FHIR-50827 and FHIR-50824"
all point to the same conclusion: this is the same problem as
FHIR-50827, which has already been resolved Persuasive with Modification
and which already carries open TODOs for exactly the items Lloyd raises
(co-morbidities, pre/post-op diagnoses, narrative section explaining
where each diagnosis concept lives). Disposition A is what the workgroup
should ultimately *do*, but doing it on this ticket in parallel with
FHIR-50827 risks duplicate edits and divergent value-set churn.
Disposition C understates Lloyd's "the existing value set is mis-scoped"
critique and would leave the residual gaps (hospital-acquired,
co-morbidity, pre/post-op) unaddressed. Folding FHIR-50824 into the
existing FHIR-50827 work plan (Disposition B) keeps the substantive
fix on the books while removing a redundant Highest-priority ticket
from PA's queue.
