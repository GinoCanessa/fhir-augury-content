## Summary

The Patient page in FHIR R6 ballot 4 (sections `#merge` and `#linking`) still
reads as if the `$merge` operation and the `Linkage` resource are part of the
FHIR Core specification, even though both artifacts have been moved to the
"Additional Resource" incubators. The hyperlinks were updated when those
artifacts were relocated (and the Linkage link is currently broken — see
[FHIR-53725](https://jira.hl7.org/browse/FHIR-53725)), but the surrounding
narrative was not adjusted. The reporter asks that explanatory text be added
to make clear that `merge` and `Linkage` are now being developed in
Additional Resource incubators and are expected to be re-integrated into the
core specification at a later point in time.

This ticket is closely related to [FHIR-53725](https://jira.hl7.org/browse/FHIR-53725)
("Remove reference and wording regarding the Linkage Resource, which is no
longer part of FHIR Core R6"), which was disposed by Patient Administration as
*Not Persuasive with Modification* — PA decided not to delete the comparison
text outright, but to wait for the incubator linking mechanism to become
operational. FHIR-54090 therefore picks up the gap: instead of removing the
content (FHIR-53725) or leaving it untouched, it asks for an interim narrative
clarification that the artifacts have moved to incubators.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Patient resource page (`patient.html`) in FHIR Core R6 in two
places:

1. **`#merge` section** — Add a leading paragraph (or callout box) along the
   lines of: *"The Patient `$merge` operation is currently being developed
   in the Patient Administration Additional Resource incubator and is not
   normatively part of FHIR Core R6. The description below summarises the
   intended behaviour; the operation is expected to be re-integrated into a
   future version of FHIR Core when the incubator work matures."* Link to
   the incubator IG location for `$merge`.
2. **`#linking` (section 8.1.8 — Patient vs. Person vs. Patient.Link vs.
   Linkage)** — Add the equivalent callout for the `Linkage` resource,
   stating that Linkage has moved to the Additional Resource incubator,
   that the comparison text is retained for context, and that the resource
   is expected to return to core in a later version. Fix the broken
   `Linkage` link to point at the current incubator URL (the issue raised
   in FHIR-53725).

This is purely a narrative / linking change in the Patient page; no
StructureDefinition, operation definition, or example is altered.

#### Justification

The text the reporter is concerned about is genuinely misleading as it
stands — readers of R6 ballot 4 cannot tell from the Patient narrative that
`$merge` and `Linkage` are no longer normative parts of core. Adding a
short incubator callout is the minimum change that removes the
misinformation while preserving the comparison text, which Grahame
explicitly endorsed in the `#implementers > Linking to Additional
Resources` thread ("you can link to it as work in progress in
documentation"). It is also consistent with PA's own disposition on
FHIR-53725, where the work group chose *not* to delete the comparison text
but deferred to "once the mechanism for linking to that content is
operational" — this disposition supplies that interim mechanism. Fixing
the broken Linkage link addresses an explicit defect noted in the ticket
and closes the gap left by FHIR-53725.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this as a cross-cutting documentation problem rather than a
Patient-page-only fix:

1. Define a single, reusable "Additional Resource / Incubator" callout
   include (e.g., a Jekyll/Liquid include or a styled `<blockquote
   class="incubator-note">`) that pages can drop in wherever they reference
   an artifact that has moved to an incubator. The include carries
   standard wording ("This artifact is currently developed in the X
   incubator, is not part of FHIR Core R<n>, and is expected to be
   re-integrated in a future version") and a link slot for the incubator
   IG.
2. Apply the include to the Patient `#merge` and `#linking` sections (the
   FHIR-54090 ask) **and** to the other already-known cases — at minimum
   the Security page's reference to `Permission` flagged by John Moehrke
   in the Zulip thread. Open follow-up tickets / PRs against the affected
   pages so the work is tracked rather than scattered.
3. Move the bulk comparison prose for Linkage and `$merge` into the
   respective incubator IG (per Grahame's "move comparison text to where
   it's being incubated"), leaving only a short pointer + callout in the
   core Patient page.

#### Justification

The Zulip thread makes clear that this is not a one-off Patient-page
problem — `Permission` on the Security page has the same defect, and there
are likely more. A reusable callout plus a co-ordinated sweep is more
durable than touching one section at a time and produces consistent
wording across the spec, which helps implementers calibrate how seriously
to take incubator references. It also aligns the spec with Grahame's
guidance that detailed comparison text should ideally live with the
artifact in its incubator IG, leaving core lighter and easier to keep
accurate as artifacts move in and out of the incubator pipeline. The
trade-off versus Disposition A is more co-ordination work and broader
scope (likely an FMG / methodology-touching change rather than a
PA-internal edit), but the resulting fix is structural rather than
spot-patched.

---

### Disposition C: Decline

#### Proposal

Close as *Not Persuasive*, on the basis that the existing links from the
Patient page already point to the incubator IG locations for `$merge` and
`Linkage`, and that the surrounding text is conceptual/comparison content
that remains accurate regardless of where the referenced artifacts live.
The only concrete defect (the broken Linkage link) is already in scope
under FHIR-53725 and should be handled there.

#### Justification

The narrative the reporter objects to describes how Patient compares with
related artifacts (Person, Patient.link, Linkage, `$merge`); that
comparison is still useful to implementers irrespective of whether the
compared artifact is currently in core or in an incubator. PA already
considered the closely related FHIR-53725 and explicitly chose not to
delete or rewrite the comparison text until the incubator linking
mechanism is operational. Adding interim callouts now risks having to
reword them again as soon as that mechanism lands, and could create
inconsistency with other pages (Security/Permission, etc.) that have not
yet been updated.

---

### Recommendation

**Recommended disposition:** A (with a strong nudge toward B as a
follow-up).

Disposition A is the right size for FHIR-54090 specifically. The reporter
has identified a real and narrow defect — the Patient page narrative
reads as if `$merge` and `Linkage` are still normative core artifacts —
and a small, surgical callout plus a fix to the broken Linkage link
removes that misinformation without re-opening the broader question PA
already settled in FHIR-53725. It is also the disposition most consistent
with the `#implementers > Linking to Additional Resources` thread, where
Grahame explicitly blessed "link to it as work in progress in
documentation". Disposition C understates the problem: the current text
genuinely misleads first-time readers, and PA's own resolution on
FHIR-53725 implicitly assumed *some* interim treatment would land before
the incubator linking mechanism does. Disposition B is the better
long-term answer (reusable callout, sweep across the spec, migration of
comparison text into incubator IGs), but it is realistically an FMG /
methodology conversation and a multi-page coordination effort; PA should
adopt A now to fix the immediate Patient-page issue and raise a separate
methodology ticket to pursue the structural fix described in B.
