## Summary

The reporter, Rick Lisseveld, identifies two related gaps in the **National
Directory of Healthcare Providers and Services (NDH) IG** that surfaced
during testing at the US Realm Connectathon. First, the IG ships **no
examples for the `Group` profile(s)**, leaving implementers without a
reference instance for that resource. Second, several **search parameters
defined by the IG cannot be exercised against the existing examples**
because the elements those search parameters target are not populated in
the example instances. The reporter intends to compile and attach the
specific list of un-exercisable search parameters as an amendment to the
description or as a follow-up comment. The cross-reference search returned
no related Jira tickets, Zulip threads, or GitHub items at the time of
this prep, so this report is grounded solely in the ticket itself.

### Disposition A: Accept As Requested

#### Proposal

Add the missing example coverage to the NDH IG in two coordinated edits:

1. **Add at least one `Group` example instance** for every NDH `Group`
   profile that currently lacks one. Each example must validate against
   the corresponding profile and be referenced from the profile's
   "Examples" tab and from the IG's example index.
2. **Audit every SearchParameter defined or referenced by the NDH IG**
   against the current example set. For any search parameter whose
   target element is not populated in any existing example, either
   (a) update an existing example to populate that element, or
   (b) add a new example whose populated elements demonstrate the
   parameter. The list compiled by the reporter from Connectathon
   testing is the authoritative starting set; the audit should extend
   it so every IG-defined search parameter has at least one example
   that exercises it.

Document the example-to-search-parameter mapping in the IG (e.g., in a
"Testing the IG" page or appendix) so future Connectathons can rely on it.

#### Justification

The ticket describes a concrete gap that blocked implementers at a US
Realm Connectathon — the IG is shipping `SearchParameter` definitions
that cannot be exercised against the IG's own example data, and a
profile (`Group`) with no example at all. Examples that exercise every
declared search parameter are a baseline expectation for a published IG
and are typically required for Connectathon and validator testing.
Accepting as requested also matches normal IG hygiene: examples are
authoritative test fixtures, and gaps in them propagate to every
implementer who tries to validate against the IG. The change is purely
additive (new examples, populated optional elements on existing
examples), so it carries no breaking-change risk for conformant servers
or clients.

---

### Disposition B: Alternative Approach

#### Proposal

Split the work into two tracked deliverables instead of treating the
ticket as a single change:

1. **Examples gap (immediate, non-normative):** Add the missing `Group`
   example(s) and update existing examples so the populated elements
   cover every IG-defined search parameter. Treat this as an
   errata/example-only update that can ship without a full
   re-balloting cycle.
2. **Search parameter / profile review (next cycle):** Use the audit
   list compiled by the reporter as input to a deeper review of the
   IG's `SearchParameter` set. For each parameter that no realistic
   instance would ever populate, evaluate whether the parameter is
   actually warranted, whether its `expression` targets the right
   element, or whether the underlying profile should mark the target
   element as Must Support. Capture any resulting changes as separate
   tickets so the example fix is not blocked on profile re-design.

#### Justification

Bundling "add examples" with "rethink which search parameters belong in
the IG" risks delaying the example fix that implementers need now, and
it may also obscure the fact that some of the failing parameters might
not actually be useful in production. Separating the deliverables lets
the WG ship the example update quickly (low risk, high implementer
value) while giving the parameter audit the room to produce real design
decisions rather than minimum-viable example padding. It also gives the
WG a defensible audit trail: the SearchParameter set was reviewed
against actual instance data, not just spec-level intent.

---

### Disposition C: Decline

#### Proposal

Decline as written, on the grounds that "missing examples" is a
maintenance task rather than a Change Request, and that the actionable
content (the list of un-exercisable search parameters) has not yet been
attached to the ticket. Close as **Not Persuasive — Insufficient
Information** and ask the reporter to either (a) re-file individual
tickets per missing example or per problematic search parameter, or
(b) submit a pull request against the NDH IG with the example additions.

#### Justification

The ticket currently contains a forward-looking promise ("I will compile
a list…") rather than the list itself, so the WG cannot triage scope or
estimate effort. Treating example gaps as a Change Request also conflates
editorial maintenance with normative spec changes, which makes
prioritization harder. If the missing examples are purely additive, a
PR is faster than a ticket; if specific search parameters are wrong,
each warrants its own ticket so it can be discussed on its merits.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the right outcome substantively — the IG should ship
examples that cover every declared search parameter and at least one
example per profile, including `Group`. But folding everything into a
single change loses important signal: some of the search parameters that
fail to test against current examples may themselves be the bug, not the
examples. Disposition B preserves the user-visible fix (Disposition A's
example additions) while opening a separate, narrower conversation about
the SearchParameter set, which is where the WG's design judgment is most
valuable. Disposition C is too dismissive — the reporter has identified
a real Connectathon-blocking gap, even if the supporting list is still
pending. The WG should accept the work, ask the reporter to attach the
compiled list as a comment, and then split execution along the lines in
Disposition B.
