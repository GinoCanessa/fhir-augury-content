## Summary

This ticket is part of a systematic sweep (originally filed by Lloyd
McKenzie as imported "Comment 258") flagging value sets used by the
EpisodeOfCare resource that are candidates for migration from the FHIR
Core specification to Terminology.HL7.Org (THO / UTG). The ticket
identifies two specific bindings on EpisodeOfCare whose backing
CodeSystems and ValueSets should be considered for THO migration:

- `EpisodeOfCare.reason.use` — bound to `ValueSet/encounter-reason-use`,
  which draws from `CodeSystem/encounter-reason-use` (EncounterReasonUse).
- `EpisodeOfCare.diagnosis.use` — bound to
  `ValueSet/encounter-diagnosis-use`, which draws from
  `CodeSystem/encounter-diagnosis-use` (EncounterDiagnosisUse).

The ticket explicitly notes that migration is **not required**, but
should be considered when (a) the value-set definition is realistic to
need to change in the coming years, or (b) the value sets might be
useful in CDA, V2, or other specifications outside FHIR. The only
substantive update so far is from Michelle Miller, reassigning the
ticket from PC to PA on the basis that EpisodeOfCare is owned by
Patient Administration (note: the same `encounter-*` code systems are
also bound by the Encounter resource, which is also PA-owned, so the
two resources should be considered together when deciding on
migration).

No Jira links, Zulip threads, or GitHub PRs/issues have been
cross-referenced to this ticket; the only cross-references found are to
the FHIR elements named in the description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate both code systems and their bound value sets out of the FHIR
Core specification and into Terminology.HL7.Org (UTG / THO):

- Move `CodeSystem/encounter-reason-use` (EncounterReasonUse) and
  `ValueSet/encounter-reason-use` to THO under the standard
  `http://terminology.hl7.org/CodeSystem/...` and
  `http://terminology.hl7.org/ValueSet/...` URIs.
- Move `CodeSystem/encounter-diagnosis-use` (EncounterDiagnosisUse) and
  `ValueSet/encounter-diagnosis-use` to THO on the same pattern.
- In FHIR Core, replace the in-spec definitions with references to the
  THO canonical URLs and update the bindings on
  `EpisodeOfCare.reason.use` and `EpisodeOfCare.diagnosis.use` (and, for
  consistency, the equivalent bindings on `Encounter.reason.use` and
  `Encounter.diagnosis.use`, since they share the same artifacts) to
  point at the new THO ValueSet canonicals.
- Add `oldCanonical`/equivalence notes so that existing implementers
  using the FHIR Core URIs can map forward.

#### Justification

This is exactly what the THO migration program asks of resource
stewards when the criteria are met. The "encounter-" prefix on these
artifacts (rather than "episodeofcare-") is itself evidence that the
value sets are not EpisodeOfCare-specific — they are already shared
with Encounter, which is precisely the kind of cross-resource reuse
that the THO program is designed to support. Putting them in THO also
opens them to use from CDA and V2 mappings of the same concepts
(reason vs. diagnosis "use" is a long-standing concept in HL7 messaging
that pre-dates FHIR), which is the second criterion the ticket itself
calls out.

---

### Disposition B: Alternative Approach

#### Proposal

Migrate **only the CodeSystems** (`encounter-reason-use` and
`encounter-diagnosis-use`) to THO, but leave the bound ValueSets in
FHIR Core. The FHIR Core value sets would be redefined to include all
codes from the now-THO-hosted code systems (i.e., the ValueSets become
thin wrappers around the THO CodeSystems). EpisodeOfCare's and
Encounter's bindings remain pointing at the FHIR Core ValueSet
canonicals, so no resource-level binding changes are needed.

#### Justification

The two stated criteria for THO migration apply most strongly to the
CodeSystems (they are the artifacts most likely to be reused by V2/CDA
mappings, and most likely to have codes added/refined over time).
ValueSets that simply enumerate "all codes from this CodeSystem" are
cheap to keep in Core and avoid binding-canonical churn for every
resource that consumes them. This alternative captures most of the
benefit of Disposition A with less specification disturbance for
implementers who already pin to the Core ValueSet canonicals — a
relevant trade-off given how widely Encounter is implemented.

---

### Disposition C: Decline

#### Proposal

Do not migrate either the CodeSystems or the ValueSets to THO. Leave
`encounter-reason-use`, `encounter-diagnosis-use`, and their backing
CodeSystems where they are, in the FHIR Core specification, with the
existing bindings on `EpisodeOfCare.reason.use`,
`EpisodeOfCare.diagnosis.use`, and the matching Encounter elements
unchanged. Resolve as **Persuasive with Modification — Not Persuasive**
on the grounds that the criteria for migration ("realistic to change",
"useful in CDA/V2") are not currently demonstrated for these specific
artifacts.

#### Justification

The ticket itself states migration is "not required" and is conditional
on either expected churn or cross-paradigm reuse. Both code systems are
small (a handful of codes each) and have been stable across recent FHIR
versions; Patient Administration has not received concrete requests
from CDA or V2 stewards to reuse them. Migration imposes a non-trivial
coordination cost (THO release cadence, dual-canonical handling, change
notes for implementers) and that cost is not justified absent a
specific consumer or proposed change. PA can revisit if and when such a
consumer or change request appears.

---

### Recommendation

**Recommended disposition:** A

The two value sets in question are already named with the
resource-neutral `encounter-` prefix and are already bound from **two**
PA-owned resources (EpisodeOfCare *and* Encounter), which is itself an
existing cross-resource reuse pattern — the first criterion in the
ticket is therefore arguably already met today, not just hypothetically
in the future. The "use" concept (admitting/discharge/billing/etc.)
also has direct analogues in V2 (PV1/PV2/DG1) and CDA encounter
templates, so the second criterion is plausible too. Moving both the
CodeSystems and their thin ValueSets together (Disposition A rather
than B) keeps the canonical URLs aligned and avoids a confusing split
where a CodeSystem lives at terminology.hl7.org but its companion
ValueSet still lives at hl7.org/fhir; THO migrations have generally
done both together for exactly this reason. PA should coordinate the
binding update on EpisodeOfCare and Encounter in the same ballot cycle
to keep the two resources consistent.
