## Summary

The reporter proposes adding a dedicated **"Conformance" tab** to the
*Interoperable Digital Identity and Patient Matching (FHIR)* IG. The
purpose is to give implementers a single, navigable place that spells
out exactly what it means to be "in conformance" with the IG, framing
those technical requirements by implementer role / use case (B2B, B2C,
IdP, etc.) rather than leaving readers to assemble the obligations
themselves from the rest of the guide.

No tickets are explicitly linked to FHIR-44594, and the cross-reference
index returns no Jira/Zulip/GitHub items. However, two siblings in the
same IG and reporter cluster are clearly germane:

- [FHIR-45271](https://jira.hl7.org/browse/FHIR-45271) — *Create Use
  Cases Tab and Flesh out Use Cases in more detail.* Same reporter,
  same IG, same "add a new tab to organise actor-facing content"
  pattern. It was resolved **Persuasive** and is now **Published**, so
  the IG already has a precedent (and a navigational slot) for
  per-actor tabs.
- [FHIR-48052](https://jira.hl7.org/browse/FHIR-48052) — *Patient B2C
  cases do not make it clear how all entities … come to shared
  consensus about identity of user.* Resolved **Persuasive with
  Modification** by adding diagrams under the **Digital Identity** tab.
  This shows the WG's recent pattern of expanding the existing
  Digital Identity / Use Cases navigation rather than spinning up new
  top-level tabs.

The ticket itself was triaged with resolution **"Considered for Future
Use"**, meaning the WG has already decided not to implement it in the
current cycle but wants to keep the idea on file.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level **"Conformance"** tab to the IG navigation
(alongside Home, Use Cases, and Digital Identity). Populate it with:

1. A short conformance overview that defines what "in conformance with
   this IG" means at the IG level (which capability statements, which
   profiles, which `$match` operation behaviour are normative).
2. An actor-segmented table of obligations, with one column per
   implementer role (**B2B Requester**, **B2B Responder**, **B2C
   Patient App**, **B2C Responder**, **IdP**, **EHR/PHR**), listing
   the SHALL/SHOULD/MAY requirements that apply to that actor (search
   parameters supported, `$match` inputs/outputs, identity assurance
   level, token handling, etc.).
3. Cross-links from each row back to the existing Use Cases section
   (FHIR-45271's tab) and to the relevant CapabilityStatement /
   profile pages so the tab is a navigation hub rather than a
   duplicate of normative content.

#### Justification

This is exactly what FHIR-44594 requests, and it is consistent with
the WG's already-demonstrated willingness to add top-level tabs for
implementer-facing material (FHIR-45271 added the Use Cases tab as
**Persuasive**). A per-actor conformance roll-up is a recognised IG
pattern (e.g., the "Conformance" / "Implementer Guidance" pages in
many HL7 IGs) and directly addresses the implementer pain that the
reporter describes: today, conformance obligations are scattered
across the CapabilityStatement, the profiles, and the use-case
narrative, and a B2C app developer has no easy way to enumerate just
their own SHALLs.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** introduce a new tab. Instead, add an **"Actor Conformance"
section under the existing Digital Identity (or Use Cases) tab** —
mirroring the precedent set by FHIR-48052, which added the Tiered
OAuth diagram under the Digital Identity tab rather than spawning new
navigation. The section would contain the same per-actor (B2B / B2C /
IdP / EHR / PHR) obligations table proposed in Disposition A, and the
existing CapabilityStatement page would be cross-linked from each
actor row. Optionally, also tag each requirement in the
CapabilityStatement / profile differentials with the applicable actor
so the table can be regenerated mechanically.

#### Justification

The IG is small enough that adding a fourth top-level tab risks
diluting navigation, and the WG's recent precedent (FHIR-48052) is to
extend an existing tab rather than create a new one. This keeps the
conformance information co-located with the use-case and identity
material it depends on, reduces the maintenance surface (one tab to
keep in sync with the CapabilityStatement instead of two), and still
delivers the actor-segmented view the reporter is asking for. It is
also less invasive to the IG template/menu, lowering the risk of
publication-tooling regressions.

---

### Disposition C: Decline

#### Proposal

Close as **"Considered for Future Use"** (the resolution already
recorded on the ticket): no change in the current publication cycle.
The CapabilityStatement plus the existing Use Cases tab (delivered by
FHIR-45271) already enumerate the IG's normative obligations, and
FHIR-48052 has just expanded the Digital Identity tab to clarify the
B2C flows. Re-evaluate after STU2 ballot feedback to determine
whether implementers actually report the navigational gap the ticket
anticipates.

#### Justification

The WG has already triaged the ticket with **Considered for Future
Use** and there are no Zulip threads, GitHub items, or follow-up Jira
comments indicating sustained implementer demand. FHIR profiles and
CapabilityStatements are themselves the normative conformance
mechanism in FHIR; adding a parallel narrative tab risks drift between
the prose tab and the machine-readable conformance resources, and
duplicates information that publication tooling already renders. The
recently-added Use Cases tab (FHIR-45271) plus Digital Identity tab
updates (FHIR-48052) materially close the same gap from a different
angle, so the marginal value of a new tab is modest.

---

### Recommendation

**Recommended disposition:** **B**

The underlying need — a single, actor-segmented view of who must do
what to be "in conformance" — is real and aligns with the reporter's
intent in FHIR-44594, so an outright decline (C) under-serves
implementers even though the ticket is currently resolved as
"Considered for Future Use". At the same time, the WG's most recent
analogous decision (FHIR-48052) explicitly chose to **extend an
existing tab** rather than create a new one, and FHIR-45271 already
gave the IG a Use Cases tab that is the natural home for per-actor
content. Disposition B captures everything Disposition A would, costs
less navigation real-estate and template churn, and keeps the
conformance narrative anchored to the CapabilityStatement / profiles
that actually carry the normative weight. If implementer feedback
during STU2 ballot shows the section becomes large enough to deserve
its own tab, the WG can promote it to a top-level "Conformance" tab
later — i.e., B is a low-regret step toward A.
