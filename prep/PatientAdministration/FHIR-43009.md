## Summary

The reporter (Vadim Peretokin) raised that the definition of
`ChargeItemDefinition.factor` (now backed by the shared `MonetaryComponent.factor`
metadata type) is ambiguous: implementers cannot tell from the spec text whether
a 25% tax should be encoded as `0.25` or `1.25`. The published informative
examples use values like `1.19` and `1.07` for German tax rates, which makes
the "factor as multiplier of the base price" reading look correct, but Simone
Heckmann clarified on Zulip that the factor is intended as the multiplier used
to compute that one component's amount (so a 25% tax line has factor `0.25`
and amount = base × 0.25), with the total derived by summing all components
rather than applying any factor to the base. The ticket asks PA / FM to make
this explicit in the spec, refine the examples, and generally clean up the
guidance around `MonetaryComponent`-based pricing in `ChargeItemDefinition`,
`ChargeItem`, and `Invoice`. PA disposed it Persuasive at the January 2024
WGM (Tue Q1) and recorded a concrete resolution that (a) drops the
`deduction` price-component code in favour of `discount`, (b) adds a worked
example and explanatory comment on `MonetaryComponent.factor` clarifying the
"factor × base = amount" semantics and the type-driven add/subtract behaviour,
(c) tightens the code-system definitions for the price-component types, and
(d) widens `ChargeItem.unitPriceComponent` and `ChargeItem.totalPriceComponent`
to `0..*`. There are no explicitly linked Jira tickets; the only cross-source
references are the originating Zulip thread on `#implementers` and a PA
conference-call agenda mention.

This is the originating thread (Oct 2023). Vadim asked how to encode 25% tax
in `ChargeItemDefinition.factor`, given the published examples use `1.19` /
`1.07` (which read as price-multipliers). Frank Oemig initially answered both
ways — `1.25` if the factor is a price multiplier, `0.25` if it is the tax

gave the authoritative reading: `factor` is the decimal used to compute that
single component's amount, so a 25% tax component has factor `0.25` and amount

`Invoice` line is built for an `Account`/`Patient`). Vadim agreed and said he

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution recorded on the ticket essentially verbatim, scoped to
the `MonetaryComponent` data type and its consuming resources owned by PA /
FM:

1. **`MonetaryComponent.factor`** — add a comment stating: "The factor is
   multiplied with the base value to produce the value stored in `amount`
   for this component (e.g., a 25% discount has factor `0.25` and amount =
   base × 0.25). The component's contribution to a total is then determined
   by `MonetaryComponent.type` — surcharges/taxes add, discounts subtract."
2. **`MonetaryComponent` (data type narrative)** — add the worked-example
   table from the resolution and a note that the element typically has
   multiple cardinality, ordering matters, and the slot should be flagged
   as ordered wherever it is used.
3. **Price-component code system** — remove the `deduction` code (it
   duplicates `discount`); use `code` and ordering to distinguish kinds of
   discount. Tighten the definitions of `base`, `discount`, `surcharge`,
   `tax`, and `informational` so each one explicitly says whether it adds
   to or subtracts from the running total.
4. **`ChargeItem`** — change `ChargeItem.unitPriceComponent` and
   `ChargeItem.totalPriceComponent` cardinality from `0..1` (current) to
   `0..*`, mirroring how `MonetaryComponent` is used elsewhere.
5. **`Invoice`** — keep `Invoice.totalNet` as the dedicated total, and add
   a note that aggregated discounts/surcharges can be represented as a
   single `Invoice.totalPriceComponent` whose `type` describes the
   aggregate.
6. **`ChargeItemDefinition`** — keep the existing structure but make sure
   the resource-level narrative cross-references the new
   `MonetaryComponent` guidance instead of re-explaining the semantics
   inline.

#### Justification

This is exactly what the reporter asked for, and Patient Administration
already adopted it at the January 2024 WGM as Persuasive. The Zulip thread
shows real implementer confusion (a co-chair-grade contributor and a
domain-expert vendor disagreed in public on a basic encoding question), so
clarification is not optional. Putting the guidance on `MonetaryComponent`
itself rather than on `ChargeItemDefinition` is the right scoping — the
ambiguity is in the data type, and the same fix benefits every resource that
embeds it (`ChargeItem`, `Invoice`, `ChargeItemDefinition`). The cardinality
relaxation on `ChargeItem` is consistent with the multi-component pricing
model the resolution narrative describes (base + multiple discounts + multiple
taxes) and avoids forcing implementers into awkward 1..1 workarounds.

---

### Disposition B: Alternative Approach

#### Proposal

Take the documentation parts of Disposition A (factor comment, worked example,
narrative guidance, tightened code-system definitions) but **defer the
structural changes** until they have an explicit Financial Management
co-sponsor sign-off:

- Apply items 1–3 (factor comment, narrative + table on `MonetaryComponent`,
  removal of `deduction` and tightening of code definitions) in this ballot
  cycle.
- **Hold** items 4 (`ChargeItem` cardinality changes to `0..*`) and the
  `Invoice` narrative tweak until FM has reviewed the change in a joint
  session, capturing the outcome on a follow-up tracker so it is not lost.
- Optionally introduce an extension on `MonetaryComponent` (e.g.,
  `monetary-component-rounding` or `monetary-component-precision`) only if
  FM identifies a concrete implementer need uncovered during review;
  otherwise leave structure unchanged.

#### Justification

The Zulip evidence (Brian's "I'm going to recommend that we hold this open
for wider review before we close out the issue.") shows PA itself flagged
that the structural changes — particularly cardinality changes on
`ChargeItem` — touch FM territory and should not land without FM review.
Splitting the work lets the unambiguous documentation fix ship immediately
(which is the core of what the reporter actually asked for), while the
breaking-shaped structural changes (cardinality widening is technically
non-breaking on read, but does change generated code and validators) get
proper cross-WG vetting. This trades some delivery completeness for lower
risk of needing a follow-up correction ballot.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing `MonetaryComponent.factor`
definition ("Factor used for calculating this component"), read together with
the per-type semantics, already encodes the answer Simone Heckmann gave on
Zulip; add only a single-sentence FAQ-style clarification in the
`MonetaryComponent` formal-comments and otherwise leave the spec unchanged.
Refresh the published examples to use values < 1.0 so the misleading 1.19 /
1.07 examples are removed, but make no schema or code-system changes.

#### Justification

A defensible argument for declining is that the current text is technically
correct — `factor` clearly says it is the factor for *this* component — and
the confusion was driven mainly by misleading examples. Swapping the
examples plus a one-line clarification fixes the proximate complaint without
touching the data-type narrative, the `MonetaryComponent` code system, or
`ChargeItem` cardinality. This minimises churn for implementers who have
already shipped against the current shape.

---

### Recommendation

**Recommended disposition:** A

PA already disposed this Persuasive at the January 2024 WGM and drafted the
resolution text recorded on the ticket; the workgroup's job at this point is
to land that resolution, not relitigate it. The Zulip thread makes it clear
the documentation-only path (Disposition C) is insufficient: two
domain-fluent contributors read the current text in opposite directions, and
the worked-example table plus the explicit "factor × base = amount, sign
depends on `type`" comment is what actually disambiguates the semantics.
Removing the `deduction` price-component code and tightening the rest of the
code-system definitions falls out of the same investigation and is cheap to
ship together. The cardinality widening on `ChargeItem.unitPriceComponent` /
`ChargeItem.totalPriceComponent` is the only piece that genuinely benefits
from FM review, and Disposition B's "split the change" framing is reasonable
*if* PA hears FM concerns at first reading; absent that, holding back the
cardinality change just to be cautious would create a second-pass
implementation burden for the same resources. Recommend taking Disposition A
as the working position and confirming the structural items with FM in a
joint Q before locking the ballot wording.
