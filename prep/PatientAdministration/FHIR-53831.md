## Summary

Linda Michaelsen reports that the element name `Account.balance.aggregate`
is misleading. In ordinary English, "aggregate" applied to a balance
suggests an *aggregated total of an account*, but in the Account resource
the element is actually used to identify **who** the balance is owed
to/attributed to (e.g., the patient vs. an insurer). She proposes
renaming the element to `responsibleParty`, which better conveys its
meaning as the party responsible for the balance.

This ticket is closely related to two other Patient Administration items
about the same `Account.balance` backbone:

- **[FHIR-53832](https://jira.hl7.org/browse/FHIR-53832)** — also from
  Linda Michaelsen, filed minutes after this one. It observes that
  `Account.balance.aggregate` lets you say "insurance" but does not let
  you say *which* insurance, and proposes adding an explicit reference
  to the `Coverage` resource. Together the two tickets argue that the
  current `aggregate` element is both poorly named *and* under-specified
  for real billing scenarios where multiple coverages are in play.
- **[FHIR-55284](https://jira.hl7.org/browse/FHIR-55284)** — Lloyd
  McKenzie's THO-migration roll-up for the Account resource. It flags
  the `account-aggregate` CodeSystem/ValueSet bound to
  `Account.balance.aggregate` as a candidate for migration to THO.
  Whatever rename is decided here, the bound value set is in active
  scope for terminology relocation, so the change should be coordinated
  with that work.

The two changes (rename and Coverage link) are not mutually exclusive
and would naturally land in the same Account.balance refactor. The
broader TSMG conversation surfaced in Zulip (see below) provides
context on the value set's terminology trajectory but is not a direct
discussion of the rename.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rename `Account.balance.aggregate` to `Account.balance.responsibleParty`
in the FHIR R6 Account resource. Carry the existing 0..1
`CodeableConcept` type and the existing binding to
`http://hl7.org/fhir/ValueSet/account-aggregate` (rename the value set
and code system to `account-responsible-party` to stay in lock-step
with the element name, or leave the binding URLs unchanged but update
their human descriptions). Update the element definition and short
text to read along the lines of "The party that is financially
responsible for the corresponding balance amount (e.g., the patient,
an insurer, a guarantor)." Add a renamed-from note in the
`Account.balance.aggregate` change-history so implementers tracking
R5→R6 see the rename. Coordinate with FHIR-55284 so that whichever
canonical the value set/code system end up with reflects this rename.

#### Justification

The reporter's reading is consistent with general use of "aggregate"
in finance and accounting, where it almost always means a sum or
roll-up rather than an attribution. The current binding values
(`patient`, `insurance`, etc.) are clearly party labels, not totals,
so `responsibleParty` matches the *actual* semantic of the element and
the values it carries. Element renames are a standard R-version
change (the resource is still pre-normative for this branch) and the
cost is bounded — only the element label and its bound terminology
descriptions need to change, mappings can be added with a single
"renamed from" note, and there are no schema-level breaking effects
for `code`-typed bindings (the binding is `CodeableConcept`).

### Disposition B: Alternative Approach

#### Proposal

Keep the element name `aggregate` but **restructure** `Account.balance`
so that the *party* aspect is split from the *aggregation* aspect.
Specifically: add a new `Account.balance.responsibleParty` element
typed as `Reference(Patient | RelatedPerson | Organization | Coverage)`
that names the actual party owing the balance, and either (a) retain
`Account.balance.aggregate` as a coarse-grained categorization
(`patient` / `insurance` / `total`) with its existing CodeableConcept
binding, or (b) replace it with a `boolean Account.balance.isAggregate`
flag indicating "this row is the rolled-up total across the other
rows". This simultaneously addresses FHIR-53831 (the misleading name
problem disappears once `responsibleParty` exists) and FHIR-53832
(implementers can finally point at *which* Coverage owns a given
balance) in one coherent change.

#### Justification

The two tickets from the same reporter on the same day strongly
suggest the underlying issue is structural, not just naming:
`Account.balance.aggregate` is being asked to do two jobs (categorize
the party *and* identify the specific party), and the current single
`CodeableConcept` cannot do both. A clean split — categorical
`aggregate` (or boolean rollup flag) plus a typed
`responsibleParty` reference — solves both complaints, is consistent
with the FHIR pattern of separating "category" CodeableConcepts from
typed References (e.g., `Encounter.participant.type` +
`Encounter.participant.individual`), and avoids a second round-trip
when FHIR-53832 comes back. Trade-off vs. Disposition A: more design
work and broader review surface, and renaming-plus-restructuring
should be socialized with implementers before R6 freezes the Account
resource shape.

### Disposition C: Decline

#### Proposal

Decline the rename. Keep `Account.balance.aggregate` as-is and instead
**clarify the element definition and short text** so that "aggregate"
is unambiguously documented as "the party against whose share of the
account this balance line aggregates" (with the value set examples
called out: `patient`, `insurance`, etc.). Mark the ticket as
"Persuasive with Mod" or "Not Persuasive with Mod" depending on
whether documentation tightening is offered.

#### Justification

`Account.balance.aggregate` exists in published R5 and is in the wild
in implementations and trial-use IGs; renaming it imposes migration
cost on every implementer of Account for what is, at root, a
documentation problem. There is no community Zulip thread escalating
the naming issue, only the reporter's own statement, and the bound
value set name (`account-aggregate`) is itself in flux as part of
FHIR-55284's THO-migration work — making one more concurrent change
to the same area amplifies churn. A precise definition and short text
can resolve the confusion without breaking anyone.

### Recommendation

**Recommended disposition:** B

The reporter has effectively filed the same complaint twice:
FHIR-53831 says the *name* of `Account.balance.aggregate` doesn't
describe what it carries, and FHIR-53832 says the *content* of
`Account.balance.aggregate` cannot identify the specific party.
Disposition A fixes the first complaint but leaves the second one
unaddressed; Disposition C fixes neither and just papers over the
naming with documentation. Disposition B treats the two tickets as
the single design issue they really are, follows an existing FHIR
idiom (categorical CodeableConcept + typed Reference, as in
`Encounter.participant`), and lets PA make one coordinated change to
`Account.balance` while the resource is still mutable for R6 and
while the bound terminology is already being touched under FHIR-55284.
The workgroup should weigh implementer impact of restructuring vs.
renaming, but B is the disposition that produces a coherent
`Account.balance` shape going into R6 ballot.
