## Summary

The ticket asks the National Directory of Healthcare Providers and Services
(NDH) IG to add a standardized "FHIR API Onboarding Metadata" data set so that
NDH publishers can declare the operational onboarding facts that are not
reliably auto-discoverable today via `.well-known/smart-configuration` or
`/metadata`. Specifically, the request targets information needed by app
developers and B2B counterparties to actually go from "I have a base URL" to
"I am a registered, trusted client": manual registration portal URL (when
`/register` is not supported), onboarding/support contact, whether Dynamic
Client Registration (DCR) and "trusted DCR" are offered (and under which
trust communities), how to register a JWKS URI for backend-services / bulk
data, and where to obtain digital certificates and trust artifacts for
FAST/UDAP/TEFCA-style trust ecosystems. The proposed shape is extensions
and/or profile elements on `Endpoint` (with optional inheritance/sharing at
the `Organization` level) plus attestation and freshness fields to control
staleness.

The request originated with Julie Maas during the FAST Implementer Support
call on 2026-02-03 and was filed by Ming Dunajick. In review comments, John
Moehrke pushes back by pointing at [FHIR-48541](https://jira.hl7.org/browse/FHIR-48541)
(NDH PA, *Applied/Persuasive*, 2024), in which the work group already removed
`Endpoint.connection-type-version` from NDH on the rationale that
endpoint-specific runtime metadata SHALL be maintained in the
`CapabilityStatement` / `.well-known` documents rather than in the directory
— a precedent that argues against putting onboarding facts in NDH at all.
Julie Maas responds that her original suggestion was not necessarily NDH but
rather "alongside the Service Base URL List" or in the endpoint's own
`CapabilityStatement`, and also notes that a related FAST Identity ticket,
[FHIR-44920](https://jira.hl7.org/browse/FHIR-44920) (*Published/Persuasive*),
established that responders should provide organizational contact information
(via `.identifier.assigner` → `Organization`) for matching follow-ups —
contact infrastructure that could be reused as the "B2B onboarding contact"
channel rather than minted anew. The ticket is therefore as much a placement
question (NDH vs. CapabilityStatement vs. Service Base URL List) as it is a
content question, and no concrete model has yet been agreed.

## Details

**Description:**

NDH currently supports discovery of FHIR endpoints (via `Endpoint` and
related content), but does not provide a standardized way to publish
operational onboarding details that are not reliably auto-discoverable
(e.g., how to obtain a client ID, whether DCR/trusted DCR is supported, how
to register JWKS URI for backend services/bulk, and where to obtain
certificates/trust artifacts for FAST/UDAP/TEFCA-like workflows). This gap
forces FHIR client app developers and counterparties to do manual
"last-mile" legwork even when Service Base URL lists and standard
SMART/FHIR/FAST security specs exist.

**Problem Statement / Background**
Production integration requires more than base URLs and protocol-level
metadata:

- `.well-known/smart-configuration` and `/metadata` enable runtime discovery
  of authorization endpoints and server capabilities, but do **not**
  reliably provide:
  - manual registration portal links (when not `/register`)
  - support contact for onboarding/approval
  - whether **Dynamic Client Registration (DCR)** is supported and for which
    communities
  - how to register **JWKS URI** (backend services/bulk data)
  - where to obtain a **digital certificate** and what trust anchors/policies
    apply for FAST/UDAP-type trust communities (including TEFCA-aligned
    ecosystems)

A National Directory is an appropriate place to publish these operational
onboarding facts, with attestation/verification and freshness controls, to
enable scalable interoperability across organizations and endpoints.

**Use Cases (Production)**

1. **Bulk Data / Backend Services app** needs to determine how to register
   its JWKS URI and keys for use with one or more provider
   organizations/endpoints (including whether there is a portal/endpoint and
   who to contact).
2. **Consumer-facing SMART app** needs to obtain a Client ID and determine:
   - the correct manual registration portal URL (if different from
     `/register`)
   - whether DCR is supported, and whether "trusted DCR" is supported (and
     under which trust communities/policies)
3. **FAST/UDAP-style client** needs to determine:
   - where to obtain a certificate accepted by the endpoint/operator
   - which trust communities are recognized (e.g., TEFCA-aligned trust,
     private trust frameworks)
   - where to find trust policies and (optionally) trust anchor bundles

**Proposed Change (High-level)**
Add a standardized "Onboarding Metadata" data element set to NDH (preferably
as extensions and/or profile elements) allowing an NDH publisher to declare
operational onboarding requirements for an `Endpoint` (and optionally
inherit/share at the `Organization` level). The model must include
attestation/freshness fields to avoid staleness.

This ticket is based on a request from Julie Maas and the discussion during
the FAST Implementer Support call on February 3, 2026.

## Related Jira Tickets

### [FHIR-48541: Endpoint connection-type-version should be removed from the extensions](https://jira.hl7.org/browse/FHIR-48541)

NDH change request from October 2024 (status **Applied**, resolution
**Persuasive**, label `block-vote-1`) that removed the
`connection-type-version` extension from `NDH-Endpoint`. The argued
rationale, attributed to Jason Vogt (MEDITECH), was that endpoint-specific
runtime data SHALL be maintained and gathered from the `CapabilityStatement`
and `.well-known/smart-configuration` rather than duplicated in the
directory, because servers change frequently and cannot be kept in sync in
multiple places. This is the precedent John Moehrke invokes against
FHIR-55595 — the work group has already established that runtime/operational
endpoint metadata belongs at the endpoint, not the directory. Resolution
delivered via [HL7/fhir-us-ndh PR #9](https://github.com/HL7/fhir-us-ndh/pull/9).

Reporter: Sheljina Ibrahim. Commenters: john_moehrke. Decision attributed in
the ticket text to Jason Vogt.

### [FHIR-44920: Reporting Issues in Matching](https://jira.hl7.org/browse/FHIR-44920)

FAST Identity / Patient Matching IG ticket from March 2024 (status
**Published**, resolution **Persuasive**, labels `STU2, block-vote-6`). The
original concern was that the Patient Matching IG was silent on how to
report match errors or suspected identity issues, and noted that the
responder has a responsibility to return relevant organization and contact
information. The applied resolution requires `$match` responders to include
`identifier.assigner` referencing an `Organization` in the returned Bundle.
Julie Maas cites this in FHIR-55595 because it establishes a precedent that
responders should expose organizational contact information at the FHIR
level — a contact channel that could be repurposed as the "who to call to
get onboarded for B2B access" channel that FHIR-55595 is asking for.

Reporter: Aaron Nusstein. Commenters: dmarcelonis, JIRAUSER24103,

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new "FHIR API Onboarding Metadata" extension bundle to the NDH IG
(repo `HL7/fhir-us-ndh`), applied primarily on the `NDH-Endpoint` profile
with an optional, inheritable application on `NDH-Organization`. Concretely:

- New complex extension `ndh-endpoint-onboarding` on `Endpoint` with
  sub-extensions:
  - `manualRegistrationUrl` (`url`, 0..1) — portal where humans register
    when `/register` is not offered.
  - `supportContact` (`ContactPoint`, 0..*) — onboarding/approval contact
    (email, phone, URL).
  - `dynamicClientRegistration` (CodeableConcept from a new ValueSet:
    `not-supported | open-dcr | udap-trusted-dcr | other-trusted-dcr`,
    0..*) — repeatable so multiple modes can be advertised.
  - `jwksRegistration` (complex; sub-extensions `mode`
    (`self-service-portal | api | manual-contact`), `url`, `instructionsUrl`,
    0..1).
  - `trustCommunity` (complex; sub-extensions `community` (Coding from a new
    ValueSet seeded with `tefca`, `udap-tier-X`, `private`), `policyUrl`,
    `trustAnchorBundleUrl`, `certificateIssuerUrl`, 0..*).
  - `attestation` (complex; sub-extensions `attestedBy` (`Reference(Organization|PractitionerRole)`),
    `attestedDate` (`dateTime`), `nextReviewDate` (`dateTime`), `source`
    (`code`: `publisher | self-attested | verified`), 1..1 when the
    onboarding bundle is present).
- Mirror extension `ndh-organization-onboarding` on `Organization` carrying
  the same sub-extensions, with an invariant that values on `Endpoint` win
  when both are present (i.e., `Organization` provides defaults).
- New ValueSets / CodeSystems for `DCR-Mode`, `JWKS-Registration-Mode`, and
  `Trust-Community` in the NDH IG terminology folder.
- Narrative updates in the NDH IG explaining the freshness model and the
  expectation that publishers refresh at least every N months (configurable;
  default 12).

#### Justification

This directly delivers what Julie Maas and Ming Dunajick described as a
real-world gap: production integration today requires manual API-doc
spelunking even after Service Base URL lists make endpoints discoverable.
NDH is, in principle, the right venue because it already publishes the
`Endpoint`/`Organization` graph that consumers consult before they ever hit
`/metadata`, and the proposed `attestation`/`nextReviewDate` model addresses
the staleness objection raised by FHIR-48541 head-on (the previous
`connection-type-version` extension had no attestation/freshness story).
Extending rather than altering core elements preserves backward
compatibility, and grounding it on `Endpoint` (with optional inheritance
from `Organization`) matches how implementers already consume NDH.

### Disposition B: Alternative Approach

#### Proposal

Do **not** add new structured elements to NDH. Instead, deliver the
onboarding-metadata gap as a two-part, lighter-weight change:

1. In `HL7/fhir-us-ndh`, add a single narrative `Endpoint` extension
   `ndh-endpoint-onboarding-info` of type `markdown` (0..1) that lets an NDH
   publisher describe onboarding in free text, with an accompanying
   `ndh-endpoint-onboarding-info-link` of type `url` (0..*) for portal /
   policy / JWKS-registration URLs, plus a single `attestationDate`
   sub-extension (`dateTime`, 1..1) to address staleness. This is the
   "simply a Markdown element" shape John Moehrke floated.
2. Refer the structured-metadata work upstream to FHIR-I / SMART /
   FHIR-Security so that DCR mode, JWKS registration mode, and
   trust-community advertisement are added to `.well-known/smart-configuration`
   (and/or `CapabilityStatement.rest.security.extension`), where
   FHIR-48541's precedent says runtime/operational data belongs. NDH would
   then reference the endpoint-side data rather than duplicate it.

#### Justification

This honours the FHIR-48541 precedent (operational endpoint data lives at
the endpoint, not in the directory) while still solving the human-facing
last-mile problem the ticket describes — which is dominated by *narrative*
("here's our portal", "email this team", "we accept these trust
communities") rather than codes that machines branch on. A single markdown
extension plus an attestation date is small, easy to populate, hard to get
out of sync (because it's narrative), and avoids minting NDH-only ValueSets
for `DCR-Mode` and trust communities that will inevitably need to be kept
in sync with the security specs that own those concepts. It also matches
Julie Maas's own walked-back framing in the comments — that the information
should travel with the endpoint or the Service Base URL List, not
necessarily live in the directory model.

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope for the NDH IG / not persuasive**,
with a written rationale that:

- The work group has already established (FHIR-48541, *Persuasive*,
  *Applied*) that endpoint-specific operational metadata SHALL be maintained
  in `CapabilityStatement` and `.well-known/smart-configuration`, not in
  NDH, because of the synchronization/staleness problem. FHIR-55595 would
  reverse that precedent within the same IG.
- The structured pieces (DCR mode, JWKS registration, trust-community
  advertisement, certificate issuance) are owned by SMART, UDAP, and the
  FAST security work, not by Patient Administration / NDH. The correct path
  is a ticket against those specifications.
- The contact / "who to call to get onboarded" piece is partially addressed
  by the FHIR-44920 pattern (`identifier.assigner` → `Organization` with
  contact information) and by `Endpoint.contact` itself; a clarifying
  narrative note in NDH about reusing those is sufficient.

Recommend the requester refile against the SMART/`.well-known` and
FAST/UDAP specs, and (optionally) against Service Base URL List
publishers, with NDH limited to a documentation note pointing at those
mechanisms.

#### Justification

Defensible because (a) it is internally consistent with the very recent
PA/NDH decision in FHIR-48541, (b) it avoids creating a parallel,
NDH-specific terminology surface for DCR/UDAP/TEFCA concepts that already
have or should have homes in the security specs, and (c) it avoids
attestation/freshness machinery in NDH that the IG does not currently
provide for any other endpoint-runtime field. The risk is real: directory
data goes stale, and a DCR/JWKS/trust-community fact that is wrong is
arguably worse than missing because clients will act on it.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible starting point because it
simultaneously honours both sides of the comment thread. It accepts the
ticket's underlying signal — that there is a real human-onboarding
last-mile gap, and Julie Maas's framing that it should be exposed as
standardly as the Service Base URL Lists themselves — by giving NDH a
small, attested, markdown-shaped place to publish "how to onboard with
this endpoint", which is what the production use cases in the ticket
overwhelmingly need. It also honours the FHIR-48541 precedent (and John
Moehrke's invocation of it) by not minting NDH-side structured codes for
DCR mode, JWKS registration mode, and trust communities, which legitimately
belong with SMART/UDAP/FAST. Disposition A is attractive in completeness
but is essentially asking PA/NDH to publish a parallel security-config
surface that other work groups own; the maintenance and synchronization
cost is exactly the failure mode FHIR-48541 was reacting to. Disposition C
is internally consistent but leaves the original requesters with nothing
actionable in NDH and effectively asks them to chase three other work
groups, which the comment thread already shows is part of the problem.
Disposition B threads that needle, gives the workgroup a concrete proposal
to debate (markdown + url + attestationDate), and creates a clean referral
path for the structured pieces.
