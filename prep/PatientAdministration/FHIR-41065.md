## Summary

FHIR-41065 asks whether the official (canonical) URL for the affected RTLS
artifact should be anchored under `terminology.hl7.org` (THO) instead of
`hl7.org/fhir`. The ticket was filed by Ron G. Parker as one of a small
series imported from review comments on the RTLS IG; this entry is
"Comment 3" of that series. Its sibling, [FHIR-41063](https://jira.hl7.org/browse/FHIR-41063)
("Comment 1"), poses the same canonical-URL question against the RTLS
**code system** (the list of RTLS "events"), while FHIR-41065 addresses
the same question for the **value set** that draws from that code
system. PA's draft resolution on FHIR-41063 is to migrate the code
system to THO after the 2.0 ballot; FHIR-41065's draft resolution
(captured as "Not Persuasive with Modification") is to **keep the value
set in the RTLS IG** because it is RTLS-specific, while noting that the
value set's CLD (content logical definition) will be updated once the
code system moves to THO. The ticket carries the
`rtls-ready-for-block-vote` label, indicating PA intends to dispatch it
in a block vote at the upcoming RTLS ballot reconciliation.

## Proposed Dispositions

> Note: PA has already drafted a "Not Persuasive with Modification"
> resolution on this ticket. The dispositions below restate the choice
> set explicitly so the workgroup can confirm or override that draft.

### Disposition A: Accept As Requested

#### Proposal

Move the RTLS value set's canonical URL from
`http://hl7.org/fhir/uv/rtls/ValueSet/<id>` (or whatever the current
RTLS IG anchor is) to a `http://terminology.hl7.org/ValueSet/<id>`
canonical and republish/host the value set as a THO artifact, in lock
step with the proposed move of the underlying code system in
[FHIR-41063](https://jira.hl7.org/browse/FHIR-41063). The RTLS IG would
reference the THO-hosted value set rather than defining it locally.

#### Justification

This is the most consistent with HL7's stated preference for hosting
shared, reusable terminology in THO, and it matches the direction
already accepted for the RTLS code system in FHIR-41063. Co-locating the
code system and the value set built directly from it avoids a split
where consumers must look in two places (RTLS IG + THO) for closely
related artifacts and avoids future canonical-URL churn if the value
set is later judged reusable.

---

### Disposition B: Alternative Approach (PA's drafted resolution — keep VS in IG, retarget the CLD)

#### Proposal

Keep the value set in the RTLS IG under its existing
`hl7.org/fhir/uv/rtls/...` canonical, but commit to updating the value
set's CLD (compose) so that it `include`s the THO-hosted code system
once [FHIR-41063](https://jira.hl7.org/browse/FHIR-41063) lands. No URL
move for the value set itself; only the underlying `system` reference
inside the value set's compose changes. Record this commitment in the
RTLS IG narrative (and ideally in a release-note / migration entry) so
implementers know the canonical of the VS is intentionally stable while
the system URI shifts.

#### Justification

The value set, as PA notes in the drafted resolution, is "very RTLS
specific" — its membership and intent are scoped to the RTLS IG's use
cases, even if the underlying code system is generally useful. Leaving
the value set canonical stable preserves any existing references from
RTLS profiles, examples, and CapabilityStatements; only the internal
`system` URI in the compose has to track the code system's eventual
move to THO. This is the smallest change consistent with the FHIR-41063
direction and aligns with the long-standing pattern of "code system in
THO, IG-specific value set in the IG."

---

### Disposition C: Decline

#### Proposal

Reject the request: keep both the code system and the value set hosted
under `hl7.org/fhir/uv/rtls/...` and make no change to the canonical
URLs. Close FHIR-41065 (and, by extension, decline FHIR-41063) as "Not
Persuasive."

#### Justification

This is defensible only if PA judges the RTLS terminology to be tightly
bound to this IG's scope and unlikely to be reused elsewhere — in which
case THO hosting buys little and a URL change costs implementer churn.
However, this contradicts PA's already-recorded direction on
FHIR-41063, so it should be selected only if the workgroup wants to
revisit that decision as well.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the recommendation, and it matches the resolution PA
has already drafted in Jira ("Not Persuasive with Modification"). It
threads the needle between the two siblings: the **code system** is
genuinely reusable and belongs in THO (per FHIR-41063), but the
**value set** carries RTLS-specific intent and would not benefit from
being lifted out of the IG that defines its purpose. Keeping the value
set's canonical stable also avoids breaking any RTLS profiles or
examples that already reference it, while the commitment to update the
CLD once the code system moves to THO ensures the two artifacts stay
coherent. Disposition A buys uniformity at the cost of a canonical-URL
move with little practical reuse benefit; Disposition C is inconsistent
with the FHIR-41063 direction and should only be revisited as a
package. Because the ticket carries `rtls-ready-for-block-vote`,
PA should confirm B and dispatch FHIR-41063 / FHIR-41065 together so
the code-system move and the value-set CLD update are scheduled as one
unit of work for post-2.0-ballot.
