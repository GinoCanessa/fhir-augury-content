## Summary

The reporter notes an inconsistency in the National Directory of Healthcare
Providers and Services (NDH) IG: the search parameters that the IG's
`ndh-server` CapabilityStatement advertises for the `HealthcareService`
resource do not match the set of search parameters listed in the
"Search Parameters" section of the `ndh-HealthcareService` profile page.
Implementers reading the profile page see search parameters that the
server does not formally claim to support, while the CapabilityStatement
omits parameters that the profile page implies are available. The
reporter asks the work group either to add the missing parameters to the
CapabilityStatement (with an explicit conformance expectation — `SHALL`,
`SHOULD`, or `MAY`) or to remove them from the IG so that the two views
agree and implementers are not left guessing which set is normative.

No related Jira tickets were surfaced via cross-references, and the
ticket has no comments yet.

## Details

**Description:**

The search parameters defined in the [NDH CapabilityStatement](https://build.fhir.org/ig/HL7/fhir-us-ndh/CapabilityStatement-ndh-server.html#HealthcareService1-2)
for HealthcareService do not include all of the search parameters listed
in the search parameters section of the [HealthcareService profile page](https://build.fhir.org/ig/HL7/fhir-us-ndh/StructureDefinition-ndh-HealthcareService.html#search-parameters).

The search parameters need to be added to the CapabilityStatement with
their expected conformance or removed from the IG to avoid implementer
confusion.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reconcile the `ndh-server` CapabilityStatement's `HealthcareService`
resource block in
`HL7/fhir-us-ndh:input/fsh/Ndh-CapabilityStatement.fsh` so it lists
every SearchParameter that appears in the rendered Search Parameters
table on `StructureDefinition-ndh-HealthcareService.html`. For each
parameter currently shown on the profile page but absent from the
CapabilityStatement, add the corresponding `SearchParam(...)` or
`SearchParamNdh(...)` insert with an explicit conformance code
(`#SHALL`, `#SHOULD`, or `#MAY`) chosen by the work group. Re-enable or
remove the commented-out `HealthcareService-location` line so its state
is intentional. Re-publish the IG and verify that the
HealthcareService section of `CapabilityStatement-ndh-server.html` and
the Search Parameters section of `StructureDefinition-ndh-HealthcareService.html`
list the same set of parameters in the same order.

#### Justification

The reporter's request is a documentation-consistency fix in a single
IG. CapabilityStatements are the conformance contract implementers
test against; if the profile page advertises a search parameter the
CapabilityStatement does not, implementers cannot rely on it being
supported, which defeats the purpose of publishing the SearchParameter.
Aligning the two surfaces is consistent with how NDH already declares
HealthcareService search parameters (each insert carries an explicit
conformance code) and requires no changes outside the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the CapabilityStatement as the single source of truth and prune
the IG instead of expanding the CapabilityStatement. Specifically, in
`HL7/fhir-us-ndh:input/fsh/Base-SearchPrameters.fsh` (and any other
SearchParameter authoring files), remove or move out of the IG any
`HealthcareService` SearchParameter that the work group does not intend
to require servers to support. For SearchParameters that should remain
defined but are optional, add them to the CapabilityStatement with
`#MAY` rather than dropping them, so the page table stops showing
parameters that have no advertised conformance level. Add a short
editorial note to the HealthcareService section of
`StructureDefinition-ndh-HealthcareService.html` (via `intro-notes` or
`pagecontent`) explaining that the CapabilityStatement is authoritative.

#### Justification

The current mismatch may signal that the IG is publishing
SearchParameters the work group never intended to mandate. Removing
those SearchParameters (rather than promoting them into the
CapabilityStatement at `#SHALL`) avoids inadvertently raising the
conformance bar on existing implementers and keeps the published
artifact set lean. It also removes the ongoing risk that future edits
to one file (CapabilityStatement vs SearchParameter inventory) silently
re-introduce the same mismatch. The trade-off vs Disposition A is that
some parameters useful to query authors get dropped, so this only fits
if the work group is confident the missing parameters are not in
production use.

---

### Disposition C: Decline

#### Proposal

Decline the change with a Persuasive / Not Persuasive disposition on
the grounds that the apparent mismatch is a rendering artifact of the
IG Publisher rather than a normative defect: the profile page's "Search
Parameters" table is auto-generated from every SearchParameter whose
`base` includes `HealthcareService`, while the CapabilityStatement
deliberately scopes the parameters that conforming `ndh-server`
implementations are required to support. Close the ticket with a note
clarifying that the CapabilityStatement is normative and the profile
page table is informational.

#### Justification

This disposition is defensible only if the work group has previously
made a deliberate decision that the two surfaces serve different
audiences (catalog of available parameters vs server contract). It
avoids re-opening conformance debates and avoids any breaking change
to existing implementations. The downsides are significant: it leaves
the implementer-confusion problem the reporter actually raised
unaddressed, and it is at odds with the existing partial cleanup in
`Ndh-CapabilityStatement.fsh` (the commented-out `location` insert),
which suggests the project itself does treat the CapabilityStatement
as the place to enumerate search parameters.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers the reporter's concern, keeps the NDH
IG's existing convention of declaring each HealthcareService search
parameter in the CapabilityStatement with an explicit conformance code,
and is a low-risk, IG-local change. Disposition B is a reasonable
follow-on if the work group concludes that some of the listed
SearchParameters were never intended to be required, but it should be
done on top of A (first align, then prune) rather than instead of it.
Disposition C is hard to defend given that the reporter's complaint —
implementer confusion — is exactly the failure mode the
CapabilityStatement is supposed to prevent, and the IG already uses
explicit conformance codes for every other search parameter on
HealthcareService. As a follow-up, the work group should also request
that `repo-analysis` be run for `HL7/fhir-us-ndh` so that future
NDH-related ticket prep and planning can be grounded in a persisted
briefing instead of re-deriving the layout each time.
