## Summary

Reinhard Egelkraut reports that the Administration Module landing page
(chapter 8.0 of FHIR Core R6) does not adequately reflect the recent
movement of several Administration-domain artifacts (resources, operations,
etc.) out of FHIR Core and into the new HL7 *AdditionalResource*
Administrative Incubator. Two concrete improvements are requested:
(1) add a notice in either section 8.0.1 *Introduction* or 8.0.2 *Index*
explaining that some artifacts previously listed on this page have been
moved to the Administrative Incubator, with a pointer to where readers
can find them; and (2) update the module's overview diagrams so that
readers can visually distinguish artifacts that remain part of the core
specification from those that now live in the Administrative Incubator.

This ticket is one of a small cluster filed by the same reporter on
2026-01-13 about the "Incubator effect" on the administration module:
[FHIR-54090](https://jira.hl7.org/browse/FHIR-54090) covers the analogous
problem on the Patient resource page (narratives still imply `merge` and
`Linkage` are core), and
[FHIR-54092](https://jira.hl7.org/browse/FHIR-54092) asks for the
Development Roadmap section (8.0.5) of the same module page to be
refreshed, optionally calling out the upcoming Incubator work. A related
Encounter-page ticket
[FHIR-54493](https://jira.hl7.org/browse/FHIR-54493) reports the same
class of staleness for `EncounterHistory` (now in the Administrative
Incubator). Reporter also references
[FHIR-53725](https://jira.hl7.org/browse/FHIR-53725) (Linkage cleanup) as
the precedent for the kind of rewording needed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the Administration Module landing page in FHIR Core
(`HL7/fhir`, authoring root `source/`, page
`source/administration-module.html`, sections 8.0.1 *Introduction* and
8.0.2 *Index*) to:

1. Add a short, prominently placed notice — in 8.0.1 *Introduction*
   (preferred) and a one-line cross-reference at the top of 8.0.2
   *Index* — that lists the Administration-domain artifacts that have
   moved out of FHIR Core into the AdditionalResource Administrative
   Incubator (currently published at
   `https://build.fhir.org/ig/HL7/admin-incubator/`), with a hyperlink
   to that incubator IG and to each affected artifact (e.g.,
   `EncounterHistory`, `Linkage`, the Patient `$merge` operation). The
   notice should make explicit that these artifacts may be integrated
   back into FHIR Core in a later release.
2. Regenerate (or hand-edit) the module overview diagrams on the same
   page so that artifacts no longer in FHIR Core are visually
   distinguished from core artifacts (e.g., a different colour,
   dashed outline, or an "Incubator" overlay), with the legend updated
   to match. The diagrams' source assets live under `images/` (or the
   per-page diagram source); the referenced PNG/SVG and any generator
   input must be updated together so the published page rebuilds
   cleanly.
3. Coordinate this edit with FHIR-54092 (Development Roadmap update on
   the same page) so that 8.0.1, 8.0.2, and 8.0.5 are all internally
   consistent about which artifacts are core vs. Incubator.

#### Justification

The ticket's request is a documentation accuracy fix on a single,
hand-authored module landing page — exactly the kind of "narrative
update" the FhirCore playbook treats as a low-risk in-place edit. The
problem is real and is independently confirmed at the resource-page
level by FHIR-54090 (Patient) and FHIR-54493 (Encounter): readers
currently cannot tell from any single page which Administration
artifacts are still in FHIR Core. Adding the notice on the module
landing page is the highest-leverage place to fix it, because it
catches readers who arrive at any resource in chapter 8 via the module
index. The diagram update is necessary because the textual notice
alone will not satisfy readers who navigate visually from the module
overview.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the underlying problem as a *spec-wide* Incubator-disclosure
pattern rather than a one-page edit:

- Author a single small reusable snippet (in the FHIR Core publisher's
  include / template mechanism under `source/`) that renders a
  standard "This artifact / section is currently maintained in the
  HL7 *AdditionalResource* Incubator — see <link>" callout box.
- Apply that snippet on the Administration Module landing page (8.0.1
  Introduction) **and** in-place at every resource/section that
  currently still describes a moved artifact (Patient `#merge` and
  `#linking` per FHIR-54090, Encounter §8.11.1.1 per FHIR-54493, and
  any others surfaced by a sweep of `source/<resource>/`).
- For diagrams, instead of redrawing the existing module diagrams,
  add a clearly labelled supplementary diagram immediately under the
  existing one showing only the moved artifacts and their target
  Incubator, leaving the historical diagram unchanged for ballot
  diff-ability. Mark the original diagram's caption with a short
  pointer to the supplementary one.
- Cross-link FHIR-54090, FHIR-54091, FHIR-54092, and FHIR-54493 as a
  single editorial work item so they are dispositioned together.

#### Justification

The "Incubator effect" the reporter describes is not unique to the
module landing page; FHIR-54090 and FHIR-54493 show it recurs
verbatim on individual resource pages. A reusable callout snippet
guarantees that wording, link targets, and disclosure level stay
consistent as more artifacts move to (or return from) the Incubator,
and avoids the maintenance trap of having to edit prose in many
places when the Incubator's content moves again. Keeping the
historical diagram intact and adding a delta diagram is friendlier
for ballot-comment review than a redraw, which can otherwise look
like an unrelated change. The trade-off vs. Disposition A is that it
is more work up front and slightly out of scope for a single ticket;
that is mitigated by closing the related tickets in the same pass.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed / out of scope at the module-page
level**, and ask the reporter to re-target the request at each
affected resource/operation page (i.e., consolidate behind the
existing FHIR-54090 / FHIR-54493 pattern), with FHIR-54092 carrying
the roadmap refresh. Rationale category: redundant with sibling
tickets that are more precisely scoped, plus the Administrative
Incubator IG itself already publishes the canonical list of moved
artifacts.

#### Justification

A defensible argument for declining is that the *authoritative* source
of truth for "what is in the Administrative Incubator" is the
Incubator IG itself, and that listing or diagramming Incubator
artifacts on the FHIR Core module page risks (a) drift between the
two specs as the Incubator evolves, and (b) confusing readers about
the conformance status of artifacts shown on a Core page. The
narrative gap the reporter identifies is real but is arguably better
fixed by per-page disclosures (FHIR-54090, FHIR-54493) plus a
prominent link from the Administrative Incubator IG back into FHIR
Core, rather than by re-mirroring Incubator content into Core.

---

### Recommendation

**Recommended disposition:** B

The reporter has effectively filed the same complaint four times in
one week (FHIR-54090, FHIR-54091, FHIR-54092, FHIR-54493), which is a
strong signal that the Patient Administration domain needs a *single
editorial pattern* for disclosing Incubator-resident artifacts inside
FHIR Core, not four bespoke prose edits. Disposition B delivers
exactly that: a reusable callout snippet plus a non-destructive
supplementary diagram on the module page, applied in one coordinated
pass across the related tickets. It satisfies the literal requests in
FHIR-54091 (a notice in 8.0.1/8.0.2 and a diagram update) while also
closing FHIR-54090, FHIR-54092, and FHIR-54493, and it leaves a
maintainable mechanism for the next artifact that moves into or out
of the Incubator. Disposition A is acceptable if the workgroup wants
the narrowest possible R6 ballot change and is willing to accept that
the same prose problem will keep being filed; Disposition C
understates the navigation problem (today, a reader on the module
page truly cannot tell what is core vs. Incubator) and should be
rejected.
