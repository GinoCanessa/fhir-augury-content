## Summary

Lloyd McKenzie reports that `EpisodeOfCare.status` is a mandatory (1..1)
element with a required binding to the `EpisodeOfCareStatus` value set,
but that value set contains no "unknown", "other", or equivalent
"escape valve" concept. As a result, in real-world cases where a system
must instantiate an `EpisodeOfCare` but cannot determine or assert one
of the defined statuses (planned, waitlist, active, onhold, finished,
cancelled, entered-in-error), there is no conformant code to use, which
can make the resource impossible to populate. The ticket asks Patient
Administration to add an `unknown`, `other`, or equivalent concept to
the value set so that implementers always have a valid way to express
"I have to populate this element but I do not have a defined status."
The ticket was originally filed against Patient Care; Michelle Miller
re-routed it to Patient Administration (2026-01-29) because PA owns
EpisodeOfCare. The cross-reference index links this ticket only to the
FHIR artifact `EpisodeOfCare.status`; no related Jira tickets, Zulip
threads, or GitHub items were surfaced.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an `unknown` concept to the `EpisodeOfCareStatus` CodeSystem and
include it in the `EpisodeOfCareStatus` value set bound to
`EpisodeOfCare.status`. Suggested concept:

- **code:** `unknown`
- **display:** `Unknown`
- **definition:** "The status of this episode of care is not known.
  Note: this concept is not to be used for 'no episode' — use a
  data-absent-reason extension at the resource level if the episode
  itself is not known to exist."

Update the element definition of `EpisodeOfCare.status` to note the
intended use of the new `unknown` value (i.e., that it is the escape
valve when the source system cannot determine a defined status, and is
not equivalent to "no episode"). Keep the binding strength as
`required`.

#### Justification

This is the minimal, targeted change that resolves the reporter's
concern without changing binding strength or invalidating existing
instances. There is precedent in FHIR for adding `unknown` to required
status value sets where the element is mandatory — e.g.,
`AllergyIntolerance.clinicalStatus` and other workflow resources have
explicit handling for missing/unknown state. Because `EpisodeOfCare`
is frequently populated from upstream systems whose internal status
codes do not always map cleanly to the FHIR status wheel, providing
`unknown` lets implementers remain conformant rather than choosing a
semantically wrong code (e.g., misusing `active` or `entered-in-error`)
or omitting the resource entirely.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the value set. Instead, document on
`EpisodeOfCare.status` (in the element definition / comments) that when
the status is genuinely unknown, implementers should use the standard
`data-absent-reason` extension on the `status` element with no value,
following the FHIR pattern for "Missing Data" on required-bound coded
elements:

```
"_status": {
  "extension": [{
    "url": "http://hl7.org/fhir/StructureDefinition/data-absent-reason",
    "valueCode": "unknown"
  }]
}
```

Add a short worked example in the EpisodeOfCare resource page showing
the pattern, and explicitly call it out as the supported escape valve.

#### Justification

FHIR already defines a generic, cross-resource mechanism
(`data-absent-reason`) for exactly this situation: a 1..1 element with
a required binding whose value cannot be determined. Using it here is
consistent with how other mandatory coded elements handle missing data
(e.g., `Observation.status` guidance, primitive-extension pattern) and
avoids polluting a workflow status enumeration with a non-workflow
concept. Adding `unknown` directly to `EpisodeOfCareStatus` muddies the
state machine — implementers may legitimately ask "can the status
transition to/from unknown?", "is unknown billable?", "does
unknown imply active or finished?" — questions that the
data-absent-reason pattern sidesteps cleanly. This option also avoids
any risk of breaking validators or terminology services that have
already cached the existing required value set.

---

### Disposition C: Decline

#### Proposal

Decline the request. The existing `EpisodeOfCareStatus` value set is
intentionally a closed workflow state machine, and the FHIR
infrastructure already provides the `data-absent-reason` extension as
the canonical mechanism for required-bound coded elements that cannot
be populated. Document in the resolution comment that implementers
unable to determine a status should use the `data-absent-reason`
extension on `_status`, and that an `EpisodeOfCare` resource should
not be created at all if its status is so unclear that even "unknown"
would be misleading.

#### Justification

`EpisodeOfCare.status` drives clinical and administrative behaviour
(active care management, billing windows, care team responsibility);
introducing `unknown` as a first-class status risks normalising
underspecified data in a workflow that downstream consumers expect to
be deterministic. The escape-valve gap the reporter describes is
already covered by the resource-wide `data-absent-reason` pattern, and
adding `unknown` would create overlap and ambiguity with that
mechanism. Where the source system genuinely cannot supply any
status, the right architectural answer is usually "do not create the
EpisodeOfCare", not "create it with an unknown status".

---

### Recommendation

**Recommended disposition:** A

The reporter (Lloyd McKenzie) is one of the most experienced filers of
this category of issue, and the underlying observation is correct:
`EpisodeOfCare.status` is mandatory with a required binding and lacks
an escape valve, which is a real interoperability hazard for
implementations that must surface episodes received from upstream
systems with non-aligned state models. While Disposition B
(data-absent-reason) is technically conformant, in practice many
implementers, validators, and downstream consumers do not robustly
handle primitive extensions on required-bound status codes, and the
"empty value + extension" pattern is poorly understood outside of
infrastructure circles. Adding an explicit `unknown` concept is the
lower-friction, higher-discoverability fix and is consistent with how
other workflow value sets in FHIR have been hardened over time.
Disposition C overstates the strictness of `EpisodeOfCare` as a state
machine — episodes are routinely created from claims and registry
feeds where status fidelity is genuinely unknown, and refusing to
represent them at all is worse for interoperability than admitting an
`unknown` value. Accept as requested, with the definition explicitly
scoping `unknown` to "status of an existing episode is unknown" (not
"existence of episode is unknown") to keep it from drifting into a
generic absence marker.
