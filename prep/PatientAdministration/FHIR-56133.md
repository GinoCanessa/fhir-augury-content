## Summary

The reporter notes a documentation/conformance inconsistency in the HL7 US
National Directory of Healthcare Providers and Services (NDH) FHIR IG: the
NDH `CapabilityStatement` (`CapabilityStatement-ndh-server`, Location entry)
declares support for three core FHIR search parameters on `Location` —
`name` (`http://hl7.org/fhir/SearchParameter/Location-name`), `near`
(`http://hl7.org/fhir/SearchParameter/Location-near`), and `partof`
(`http://hl7.org/fhir/SearchParameter/Location-partof`) — but those
search parameters are not listed in the "Search Parameters" section of
the `ndh-Location` profile page (`StructureDefinition-ndh-Location`).
Implementers reading the profile page therefore cannot tell that these
searches are expected to be supported by an NDH-conformant server. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the index for this issue, so the report relies solely on the
ticket itself.

## Details

**Description:**

The [NDH CapabilityStatement](https://build.fhir.org/ig/HL7/fhir-us-ndh/CapabilityStatement-ndh-server.html#Location1-4) and the Location Profile page are not aligned on the SearchParameters. The CapabilityStatement lists:

- `"name"` — http://hl7.org/fhir/SearchParameter/Location-name
- `"near"` — http://hl7.org/fhir/SearchParameter/Location-near
- `"partof"` — `[http://hl7.org/fhir/SearchParameter/Location-partof` *(URL is malformed in the ticket — opening bracket but no closing; intended target is `http://hl7.org/fhir/SearchParameter/Location-partof`)*

…but those SearchParameters are not included on the [Location Profile Page under Search Parameters](https://build.fhir.org/ig/HL7/fhir-us-ndh/StructureDefinition-ndh-Location.html#search-parameters).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as **Persuasive — IG content fix** against the NDH IG. Bring
the `ndh-Location` profile page's "Search Parameters" section into
alignment with what `CapabilityStatement-ndh-server` already advertises
for `Location` (interaction `Location1-4`):

- `name` → `http://hl7.org/fhir/SearchParameter/Location-name`
- `near` → `http://hl7.org/fhir/SearchParameter/Location-near`
- `partof` → `http://hl7.org/fhir/SearchParameter/Location-partof`

Concretely, ensure the `ndh-Location` `StructureDefinition` declares
those three core FHIR `SearchParameter`s as expected/supported (e.g.,
via the IG's standard mechanism for binding search parameters to a
profile — typically a `SearchParameter` reference list in the IG
configuration / FSH or a `Conformance`-style block on the profile page)
so that the IG publisher renders them in the profile's
"Search Parameters" section. Also fix the malformed `partof` URL in any
narrative that copies the CapabilityStatement (the ticket itself shows
`[http://...` without a closing bracket).

#### Justification

The CapabilityStatement is normative for what an NDH server must
support; the profile page is the primary reference an implementer reads
when building against `ndh-Location`. Having the two disagree is a pure
conformance-documentation defect — there is no design question, only a
publishing gap. Aligning the profile page to the CapabilityStatement
(rather than the other way around) preserves existing server
expectations and matches the pattern used elsewhere in the NDH IG and
in US Core, where each profile page enumerates the search parameters
its CapabilityStatement requires. The three parameters in question are
all base-spec `Location` search parameters, so no new artifacts need
authoring — only inclusion/listing.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the alignment intent, but resolve it by **publishing
NDH-specific `SearchParameter` resources** that derive from the three
core FHIR `Location` search parameters (`Location-name`,
`Location-near`, `Location-partof`) and reference `ndh-Location` in
their `base`. Bind those NDH `SearchParameter` instances to the profile
(so the IG publisher renders them on the profile's "Search Parameters"
tab) and update `CapabilityStatement-ndh-server` to point at the
NDH-scoped canonicals instead of the base FHIR canonicals. Document any
NDH-specific expressions, modifiers, or comparators (e.g., distance
units for `near`, partial-match semantics for `name`) on those new
`SearchParameter` resources.

#### Justification

NDH has stricter expectations than the base spec in several areas
(e.g., geographic distance handling for `near`, organization hierarchy
semantics for `partof`). Owning NDH-scoped `SearchParameter`s gives the
WG a place to record those expectations rather than relying on the
base-FHIR definitions, and it makes the profile page self-contained for
implementers who only read the IG. Trade-off: it is more work than
Disposition A and adds NDH-owned artifacts that must be maintained as
the base spec evolves; it is only worth doing if the WG actually
intends to constrain or extend the base search behavior. If
NDH-specific search semantics are *not* intended, Disposition A is
strictly cheaper and lower risk.

---

### Disposition C: Decline

#### Proposal

Mark as **Not Persuasive** on the grounds that the apparent gap is a
rendering artifact of the IG publisher rather than a substantive
conformance issue: the `CapabilityStatement` is the authoritative
declaration of supported search, and downstream tooling (and most
implementers' validators) will key off `CapabilityStatement-ndh-server`,
not the profile page's "Search Parameters" section. Close with a
pointer to the CapabilityStatement as the source of truth.

#### Justification

This is defensible only if the WG views the profile page's
"Search Parameters" section as informative-only and is unwilling to
spend cycles on IG-presentation polish. It avoids any FSH/IG config
churn. However, it is a weak position: the profile page is the most
discoverable surface for implementers of `ndh-Location`, and leaving it
visibly inconsistent with the CapabilityStatement invites repeat
tickets and erodes confidence in the IG's editorial quality.

---

### Recommendation

**Recommended disposition:** A

This is a low-risk, high-clarity editorial alignment of two artifacts
in the same IG that already agree in intent. There is no design
disagreement to resolve and no community discussion (Zulip / linked
tickets) suggesting NDH wants to *constrain* `Location` search semantics
beyond what base FHIR provides — which is what Disposition B would
require to justify the extra artifacts. Disposition C papers over a
real implementer-facing inconsistency and is likely to attract repeat
reports. Disposition A — list the three base `Location` search
parameters (`name`, `near`, `partof`) on the `ndh-Location` profile
page so it matches `CapabilityStatement-ndh-server`, and fix the
malformed `partof` URL in any copied narrative — is the smallest change
that closes the ticket. If, during implementation, the WG decides
NDH-specific search semantics are warranted (e.g., for `near` distance
units), that work can be tracked as a separate follow-up ticket and
implemented as Disposition B without blocking this fix.
