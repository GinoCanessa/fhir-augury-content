## Summary

The reporter notes a documentation/consistency gap on the Encounter resource page in the FHIR core specification. The `EncounterHistory` resource was removed from FHIR core and relocated to the HL7 Administrative Incubator as an `AdditionalResource`, but the narrative text in section 8.11.1.1 of the Encounter page still reads as if `EncounterHistory` is a first-class FHIR core resource. The ticket asks Patient Administration to (a) update the Encounter narrative to make clear that `EncounterHistory` is **not** currently a FHIR core resource — it lives in the Administrative Incubator and may return to core at a future date — and (b) consider deleting the legacy "note to implementers" that only describes the R4 → R5 change, since the surrounding context has since changed.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were returned by the augury index for this ticket; the request stands on its own.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Encounter resource page in the FHIR core build (`HL7/fhir`, the Encounter narrative source under the `source/encounter/` authoring area, specifically the content rendered as section 8.11.1.1 of `encounter.html`) to:

1. Replace any prose that implies `EncounterHistory` is a current FHIR core resource with an explicit statement that, as of this ballot cycle, `EncounterHistory` is **not** a FHIR core resource — it has been relocated to the [HL7 Administrative Incubator](https://build.fhir.org/ig/HL7/admin-incubator/en/StructureDefinition-EncounterHistory.html) as an `AdditionalResource`, with the possibility of reintegration into core in a future release.
2. Add a single canonical hyperlink to the Administrative Incubator `StructureDefinition-EncounterHistory.html` page so implementers can find the current artifact.
3. Remove the existing "note to implementers" in 8.11.1.1 whose sole content is the R4 → R5 history of how status tracking moved out of `Encounter` into a separate resource — that context is no longer actionable for current implementers and now causes more confusion than it resolves.

#### Justification

This is a pure documentation correction that brings the Encounter narrative in line with the actual structure of the spec. The status quo is misleading: a reader of `encounter.html` today will conclude that `EncounterHistory` is a sibling core resource, which is factually wrong and undermines trust in the narrative. FHIR's editorial conventions favour pointing at the canonical home of an artifact (in this case, the incubator), and dropping a stale R4→R5 changelog from inline narrative is consistent with how other resources have pruned legacy migration notes once the migration is far enough behind us. There are no normative or schema impacts, no breaking changes, and no downstream conformance fallout.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the substance of the request but soften the second bullet: rather than **removing** the R4→R5 note to implementers outright, **rewrite** it as a short historical sidebar that (a) explicitly flags `EncounterHistory` as living in the Administrative Incubator, (b) preserves a one-sentence pointer to the R4→R5 redesign for implementers still migrating long-lived R4 deployments, and (c) links to both the Administrative Incubator page and the relevant section of the R4→R5 migration documentation. The first bullet (clarifying that `EncounterHistory` is not in core) is implemented exactly as requested.

#### Justification

R4 is still widely deployed, and Patient Administration regularly hears from implementers who are mid-migration. Wholesale deletion of the only inline pointer to the R4→R5 redesign of encounter status tracking risks losing a useful breadcrumb for that audience, especially given that `EncounterHistory` is now one indirection further away (in the incubator, not in core). A rewritten sidebar achieves the reporter's primary goal — eliminating the false implication that `EncounterHistory` is a core resource — without removing a piece of historical context that still has practical value. This is also a lower-risk edit at ballot time, since reviewers tend to scrutinise deletions of implementer-facing notes more than additions or rewrites.

---

### Disposition C: Decline

#### Proposal

Decline the request as written and leave the Encounter narrative unchanged for this cycle, on the basis that the ambiguity is editorial rather than normative, and that wider work on the relationship between FHIR core and the Administrative Incubator is expected before changes here would stick.

#### Justification

The Administrative Incubator is itself an evolving artifact, and the long-term story of whether `EncounterHistory` returns to core or stays in the incubator is not yet settled at the WG level. Editing the Encounter narrative now to firmly assert "not in core" risks a second corrective edit if/when the resource is promoted back into core in a future ballot, churning normative-adjacent text twice. Declining at this point and revisiting once the incubator-to-core promotion policy is clearer would minimise editorial churn. (This is the weakest of the three options because it leaves a known incorrect statement in a published page.)

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the reporter's primary, uncontroversial fix — removing the false impression that `EncounterHistory` is a current FHIR core resource and pointing implementers at the Administrative Incubator — while keeping a slim, rewritten implementer note that still serves the substantial population of R4 deployments mid-migration to R5+. Disposition A is also defensible and only marginally more aggressive; the workgroup should feel free to fall back to A if it concludes the R4→R5 breadcrumb has outlived its usefulness. Disposition C should be rejected: leaving a known-incorrect statement on a high-traffic resource page to avoid a possible future re-edit is poor stewardship of the narrative, and the corrective edit if `EncounterHistory` later returns to core would itself be trivial.
