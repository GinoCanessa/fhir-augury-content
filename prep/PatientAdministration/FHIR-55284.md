## Summary

FHIR-55284 (Patient Administration, Change Request, Highest priority,
Triaged) flags four value sets bound to elements of the `Account`
resource as **candidates for migration to Terminology.HL7.org (THO)**.
The ticket is an "imported comment 248" filed by Lloyd McKenzie that
identifies the following code-system / value-set / binding triples:

- `account-billing-status` (CodeSystem `AccountBillingStatus`) →
  ValueSet `account-billing-status` → bound to **`Account.billingStatus`**.
- `encounter-diagnosis-use` (CodeSystem `EncounterDiagnosisUse`) →
  ValueSet `encounter-diagnosis-use` → bound to
  **`Account.diagnosis.type`**.
- `account-aggregate` (CodeSystem `AccountAggregate`) → ValueSet
  `account-aggregate` → bound to **`Account.balance.aggregate`**.
- `account-balance-term` (CodeSystem `AccountBalanceTerm`) → ValueSet
  `account-balance-term` → bound to **`Account.balance.term`**.

The ticket explicitly notes that THO migration is **not required** but
should be considered when (a) the value-set definition is realistically
expected to evolve in coming years, or (b) the value set could be
useful in CDA, V2 or other HL7 specs. The single comment from
@egelkraut (2026-01-26) defers the decision on the
`encounter-diagnosis-use` binding to the outcome of
[FHIR-50827](https://jira.hl7.org/browse/FHIR-50827) — a related PA
ticket (already resolved Persuasive-with-Modification) that re-routes
several diagnosis "uses" out of `Encounter.diagnosis` and into
`Encounter.reason`/`Account.diagnosis`, which directly affects what the
correct binding for `Account.diagnosis.type` should be going forward.

### [FHIR-50827: Encounter.diagnosis.use binding VS is not helpful for differentiating several Condition roles](https://jira.hl7.org/browse/FHIR-50827)

PA / Change Request, **Resolved Persuasive with Modification**
(2025-09-09). The reporter argued the existing `encounter-diagnosis-use`
value set is too narrow for hospitalisation workflows and proposed
incorporating codes from `http://terminology.hl7.org/CodeSystem/diagnosis-role`

"present-on-admission", and adds "Discharge Diagnosis" to the
`encounter-diagnosis-use` value set. The PA TODO carried forward
includes deciding where co-morbidities and pre/post-op diagnoses live
(joint with PC). This ticket is the direct upstream of FHIR-55284's
question about `Account.diagnosis.type` — its resolution reshapes the
content of `encounter-diagnosis-use` and therefore changes the cost /
benefit of migrating that VS to THO.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate **all four** Account-bound value sets and their backing code
systems from the FHIR core spec to THO:

1. `CodeSystem/account-billing-status` + `ValueSet/account-billing-status`
   (bound to `Account.billingStatus`).
2. `CodeSystem/account-aggregate` + `ValueSet/account-aggregate` (bound
   to `Account.balance.aggregate`).
3. `CodeSystem/account-balance-term` + `ValueSet/account-balance-term`
   (bound to `Account.balance.term`).
4. `CodeSystem/encounter-diagnosis-use` + `ValueSet/encounter-diagnosis-use`
   (bound to `Account.diagnosis.type` *and* used by Encounter, which is
   already being reshaped by FHIR-50827).

For each pair, move the source artifact from the FHIR core authoring
location (in `HL7/fhir`, the Account / Encounter source trees, alongside
`account-*.xml` and `encounter-diagnosis-use.xml`) into the appropriate
`HL7/UTG` authoring tree, retain the existing
`http://hl7.org/fhir/CodeSystem/...` URLs as the canonical identifiers
(or define an HL7-wide-canonical alias under
`http://terminology.hl7.org/CodeSystem/...` per the established THO
migration playbook), update the bindings on `Account.billingStatus`,
`Account.diagnosis.type`, `Account.balance.aggregate`,
`Account.balance.term` to point at the THO-hosted value sets, and update
the FHIR core build to consume them as external dependencies.

#### Justification

The four value sets are all narrow, Account-scoped, billing/finance
terminologies with realistic future evolution (billing-status states,
balance aggregation buckets, balance term buckets) and clear cross-spec
relevance — V2 financial messages and CDA billing sections cover
substantially the same concepts. Moving them to THO matches the
ticket's stated criteria ("realistic for the value set definition to
need to change", "useful in CDA, V2 or other specs") and aligns with
the broader PA pattern of relocating content code systems to THO so
they can version independently of the FHIR core release cadence.
Doing all four together amortises the migration mechanics (URL
aliasing, build-time package dependency, cross-references) and avoids a
piecemeal terminology layout for a single resource.

---

### Disposition B: Alternative Approach (migrate three, defer the fourth)

#### Proposal

Migrate the three **billing/finance** value sets immediately, but
**defer** the `encounter-diagnosis-use` migration until FHIR-50827's
resolution lands and stabilises:

- Migrate now: `account-billing-status`, `account-aggregate`,
  `account-balance-term` (and their backing code systems) to THO,
  rebinding the corresponding Account elements to the THO-hosted value
  sets.
- Defer: `encounter-diagnosis-use`. Instead, after FHIR-50827's R6
  changes are applied (admission/chief-complaint moved to
  `Encounter.reason`, "Discharge Diagnosis" added to
  `encounter-diagnosis-use`, billing diagnosis kept on
  `Account.diagnosis`), open a follow-up ticket scoped specifically to
  the THO migration of the now-stabilised
  `encounter-diagnosis-use` (or its successor) and to deciding whether
  `Account.diagnosis.type` should remain bound to that VS or to
  something narrower.

#### Justification

This honours the comment from @egelkraut: the
`Account.diagnosis.type` decision is genuinely entangled with
FHIR-50827 (Persuasive with Modification, still has open PA TODOs for
co-morbidities and pre/post-op diagnoses). Migrating
`encounter-diagnosis-use` to THO **before** its content and binding
locations have settled risks doing the migration twice (once now,
again after FHIR-50827 is implemented) and risks publishing a THO VS
that is immediately superseded. Splitting also reduces blast radius:
the three billing value sets are uncontroversially Account-scoped and
have no open upstream questions, so they can be migrated in the next
THO publication window without waiting on Encounter rework. The
trade-off vs. Disposition A is a minor coordination overhead (two PA
items instead of one) for materially lower rework risk.

---

### Disposition C: Decline

#### Proposal

Decline the migration request. Leave all four value sets and code
systems in the FHIR core spec, in their current authoring locations
under the Account / Encounter source trees, with their current
canonical URLs (`http://hl7.org/fhir/...`). Capture the rationale on
the ticket so future PA reviewers don't re-open it, and link to
FHIR-50827 as the controlling work for `encounter-diagnosis-use`.

#### Justification

The ticket itself states migration is **not required** and is justified
only when the VS is likely to evolve or to be re-used outside FHIR.
For three of the four value sets (`account-aggregate`,
`account-balance-term`, `account-billing-status`), the codes are tightly
coupled to the FHIR Account information model and to the ticket index
this ticket draws from there is no concrete cross-spec demand
(no V2/CDA tickets surfaced, no Zulip discussion, no GitHub
cross-reference). For `encounter-diagnosis-use`, the active work is
already being done under FHIR-50827, which is a structural change to
Encounter rather than a terminology-hosting question, so the THO
migration is at best premature and at worst a distraction.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B is the strongest fit for this ticket as written. Lloyd's
own framing — "consider migration when the VS may evolve or be reused
across CDA/V2" — applies cleanly to the three billing value sets
(`account-billing-status`, `account-aggregate`, `account-balance-term`):
they are narrow finance terminologies with obvious cross-spec analogues
in V2 financial segments, and migrating them now lets THO own their
release cadence. For `encounter-diagnosis-use`, however,
@egelkraut's comment is correct: FHIR-50827 (resolved Persuasive with
Modification) is actively re-shaping where diagnosis "uses" live, what
codes belong in the value set ("Discharge Diagnosis" being added), and
whether `Account.diagnosis.type` should even keep this binding.
Migrating it before that work is complete would almost certainly mean
republishing the THO artifact within one release cycle. Splitting the
work — migrate three now, decide on `encounter-diagnosis-use` after
FHIR-50827 lands — is a small process cost that avoids real rework and
keeps the PA / Vocab coordination clean. Disposition A is acceptable if
the workgroup wants to bundle the migration; Disposition C should only
be chosen if PA explicitly concludes none of these value sets meet the
"likely to evolve / reusable" bar, which is hard to defend for at least
`account-billing-status`.
