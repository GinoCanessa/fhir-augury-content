## Summary

Elliot Silver reports that the default behavior of the `Patient/$everything`
operation is effectively *unpaged*: a single Bundle is returned regardless of
how much data the patient chart contains. He argues that this is inconsistent
with the rest of FHIR (search results are paged by default) and that for
realistic charts — particularly when `Binary` resources are pulled in — the
single-Bundle default is unworkable for consumers and intermediate gateways.
He requests that paged results become the default, with unpaged behavior
available only on explicit request.

The Patient Administration work group has resolved the ticket as
**Persuasive with Modification**. Rather than simply switching the default
to Bundle paging, PA proposes to collapse the three currently-documented
options ("one big Bundle", searchset-style paging, async bulk) down to a
**single recommended mechanism: the Asynchronous Bulk Data Request Pattern
(`async-bulk.html`)**. The resolution also notes — and the related Zulip
threads confirm — that PA is moving `$everything` (Patient, Encounter,
EpisodeOfCare) and `$merge` out of the FHIR Core R6 spec into the Admin
Incubator, so the long-term direction may be to retire `Patient/$everything`
in favor of `Patient/$export`. There are no cross-referenced Jira tickets,
but several Zulip threads (implementers, committers, Da Vinci PDex,
german (d-a-ch)) discuss exactly these trade-offs.

## Details

**Description:**

> The default behavior of $everything appears to be unpaged. This is concerning,
> since this operation is likely to return more content than most standard
> searches (which are paged), even in the context of a single patient,
> especially if Binary resources are included. While I fully understand the
> usefulness of unpaged $everything from a sender's performance point of view,
> the lack of paging may make it difficult for consumers (as well as
> intermediate gateways, etc.).
>
> This approach is also counter to the standard behavior for search.
>
> Make paged results the default and unpaged only by request.

**Resolution Description (Persuasive with Modification):**

The current `$everything` operation paragraph reads (paraphrased): *"When this
operation is used to access multiple patient records at once, the return
bundle could be rather a lot of data; servers may choose to require that such
requests are made [asynchronously](https://build.fhir.org/async.html), and
associated with [bulk data formats](https://build.fhir.org/resource-formats.html#bulk).
Alternatively, clients may choose to page through the result set… using the
[`_count`](https://build.fhir.org/search.html#count) parameter and Bundle
links. There is no defined order for the results of `$everything`. Servers
MAY consider sorting the returned resources in descending order of last
record update."*

PA proposes replacing it with: *"When this operation is used, it is possible
that it would return large amounts of data, either for multiple patient
records at once, or for a single patient with a large chart. To efficiently
exchange this possibly large data set, the
[Asynchronous Bulk Data Request Pattern](https://build.fhir.org/async-bulk.html)
is used. The Bulk Data manifest allows clients to predict how much data
they'll need to download ahead of time. It also provides for a well-defined
and workable reference resolution approach."*

WG notes (not part of the proposal):
- PA wants to define **one way** instead of preserving today's optionality.
- Points in favor of **Async Bulk**: predictable download size; bundle
  reference resolution doesn't work well across paged Bundles.
- Points in favor of **Bundle paging**: backwards compatibility with R4.
- Aside: PA wants to discuss with FMG whether `$everything` could be deleted
  outright in favor of `Patient/$export`.

**Comments:**

- **esilver (2025-09-10):** Walks through the three options (one Bundle vs.
  Async Bulk vs. searchset paging). He calls Async Bulk "probably the best
  approach… but incompatible with existing behavior." On searchset paging he
  warns that having an "almost-searchset" Bundle whose references span pages
  forces implementers to write a custom Bundle handler; he argues PA should
  either commit to a fully standard searchset Bundle (with self/next/prev
  links and standard reference resolution) or use a clearly different Bundle
  type (e.g., `collection`) so implementers don't assume standard semantics.
- **cooper.thompson (2025-09-09):** Posts that PA is asking the implementer
  community to weigh in via the
  [#implementers > Patient/$everything paging](https://chat.fhir.org/#narrow/channel/179166-implementers/topic/Patient.2F.24everything.20paging)
  Zulip topic.
- **cooper.thompson (2025-09-09):** Pings Elliot Silver directly asking
  whether he prefers Bundle paging or async bulk as the single retained
  mechanism for R6.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the `Patient/$everything` operation page (and the parallel
`Encounter/$everything` and `EpisodeOfCare/$everything` operation pages —
now in the Admin Incubator), change the documented default behavior so
that **paged results are the default** and unpaged behavior is only
available on explicit request. Concretely:

- Make `_count` (search-style paging) the default response shape, returning
  a `Bundle.type = searchset` with `self`, `next`, and `prev` links,
  matching the rules in `http.html#paging` and `search.html#count`.
- Add a new operation parameter (e.g., `Parameters.parameter[name="_unpaged"]`
  with `valueBoolean = true`, or alternatively allow `_count=*`) that a
  client can set to opt back into the current single-Bundle behavior.
- Document explicitly that, because referenced resources may now be
  distributed across Bundle pages, clients MUST be prepared to resolve
  references across pages of the same `searchset` Bundle.

#### Justification

This directly addresses the inconsistency Elliot raised: every other
reads-many endpoint in FHIR is paged, and `$everything` — which is
strictly larger on average — should not be the exception. It is the
smallest semantic change to existing implementations: clients that don't
pass `_unpaged` get a paged response; servers that today only emit one
Bundle still legally satisfy the operation by emitting a `searchset` with
no `next` link. The implementer thread (Cooper, David Pyke) supports the
intuition that the current single-Bundle default cannot survive realistic
chart sizes. Elliot's own caveat — that "almost-searchset" Bundles with
cross-page references are awkward — is mitigated by *explicitly* declaring
the result a `searchset`, with a normative statement about cross-page
reference resolution, rather than leaving it implicit.

---

### Disposition B: Alternative Approach (PA's current Persuasive-with-Modification)

#### Proposal

Adopt PA's drafted resolution: collapse the three documented options down
to **one** — the **Asynchronous Bulk Data Request Pattern**
(`https://build.fhir.org/async-bulk.html`). Rewrite the relevant paragraph
on the `$everything` operation page (and the parallel
`Encounter/$everything` / `EpisodeOfCare/$everything` pages, now in the
Admin Incubator) to read approximately:

> *"When this operation is used, it is possible that it would return large
> amounts of data, either for multiple patient records at once, or for a
> single patient with a large chart. To efficiently exchange this possibly
> large data set, the Asynchronous Bulk Data Request Pattern is used. The
> Bulk Data manifest allows clients to predict how much data they'll need
> to download ahead of time. It also provides for a well-defined and
> workable reference resolution approach."*

Remove the `_count`/Bundle-paging language and the unstructured
"server-may-require-async" language. Keep the `$everything` *invocation*
shape unchanged; the response is always returned via the async/bulk
manifest pattern.

#### Justification

This is the WG's working resolution and it directly answers Elliot's
underlying concern (unbounded responses) without inheriting the awkwardness
he flagged about "almost-searchset" Bundles. It also reflects the
implementer consensus on the Zulip thread: Daniel Venton and Cooper
Thompson independently argued that synchronous, paged Bundles do not
realistically work for full-chart pulls; David Pyke agreed bulk is the
correct long-term mechanism. It uses an *already-defined* mechanism
(`async-bulk.html`) and gives implementers predictable download sizing and
defined reference resolution semantics — the two things Bundle paging
cannot offer for `$everything`. Trade-off vs. A: this is a bigger break
from R4/R5 behavior (clients must implement the async polling flow), and
it does not satisfy implementers who want a small, sync, "give me this one
small patient" call. PA should be explicit that `$everything` is no longer
intended for trivial sync usage.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-52121 on the grounds that it is being **superseded** by a
broader PA action: removing `Patient/$everything`,
`Encounter/$everything`, and `EpisodeOfCare/$everything` from FHIR Core R6
and migrating them to the
[Admin Incubator](https://build.fhir.org/ig/HL7/admin-incubator/branches/main/en/index.html),
while pointing implementers at `Patient/$export` (Bulk Data Access IG) for
real-world large-chart retrieval. Resolve as **Not Persuasive –
Superseded** (or "Considered – No Action") with a note that paging
semantics for `$everything` will be defined in the Admin Incubator and
that the production-grade alternative is `Patient/$export`. Do not modify
the FHIR Core R6 `$everything` page beyond the wording needed to flag the
move.

#### Justification

Per the [#committers > operations that move out](https://chat.fhir.org/#narrow/stream/committers/topic/operations%20that%20move%20out/near/554421302)
thread, PA has already moved these operations out of FHIR Core; Grahame
Grieve has agreed to remove the operation-definition pages. Per the
[#Da Vinci PDex](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20PDex/topic/Payer%20to%20Payer%20Single%20Member%20-%20Scoping%20to%20one%20Patient/near/576127444)
thread, Cooper Thompson has explicitly said implementers in regulated
exchange flows should not be relying on `$everything` ("$everything really
only works for tiny toy demo patients"); the production answer is
`Patient/$export` / `Group/{id}/$export`. The submitter's underlying
concern — that consumers can't safely consume an unbounded
`$everything` response — is *fully* addressed by directing those use cases
to the existing Bulk Data Access mechanism, which already requires the
async manifest pattern. Spending normative effort to redefine paging for
an operation that PA is removing from core is poor return on the WG's
time.

---

### Recommendation

**Recommended disposition:** B (PA's Persuasive-with-Modification).

Disposition B is the recommended path because it is what PA has already
drafted and what the Zulip implementer thread broadly supports, and it
genuinely addresses Elliot's concern: a consumer or gateway no longer has
to choose between "all in one Bundle" and "almost-searchset paging with
broken reference resolution." It also avoids creating a *new* normative
problem (cross-page reference resolution in `searchset` Bundles), which
Elliot himself warned about and which Disposition A would have to solve.

Disposition A is a viable fallback if the workgroup decides backward
compatibility with R4/R5 client code matters more than a clean single
mechanism — but it leaves the underlying reference-resolution problem in
the spec rather than fixing it.

Disposition C is tempting given that PA has moved `$everything` to the
Admin Incubator and there is real momentum behind `Patient/$export` as
*the* large-chart pattern. However, declining outright would leave the
Admin Incubator version of `$everything` with the same unbounded default
that Elliot flagged, just in a different document. Better to apply the
fix (B) **in whichever spec now hosts `$everything`** — Admin Incubator
for R6 onward, and optionally a clarifying note in R5 errata — and use
the disposition discussion to decide *where* the normative wording lands,
not whether to write it.
