## Summary

The ticket is a STU1 ballot comment against the Validated Healthcare
Directory (VHDir) IG (originally GF#18809 / VHD #17) asking that the
generated narrative for an example be made friendlier to readers. The
specific suggestion is to elevate the existing `note` value
("This care team was founded to optimize the management of complex
cardiac cases") so that it serves as the human-readable description of
the example as a whole, rather than being buried inside the generated
narrative table. The original VHD ballot disposition (2018‑10‑17, Alex
Kontur / Iryna Roy, 6‑0‑0) was *Considered for Future Use* on the basis
that narrative generation is performed by the HL7 IG publishing tooling
and the request belonged against the tooling rather than the IG; on
2025‑02‑18 Lynn Laakso reverted that resolution and re-opened the
ticket post‑STU1 publication, returning it to Triaged for the workgroup
to redispose. There are no other Jira or GitHub cross-references to the
ticket; the only related discussion is the original VHD block-vote
roll-up on Zulip.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the VHDir example that prompted the ballot comment (the CareTeam
example whose `note` reads "This care team was founded to optimize the
management of complex cardiac cases") so that the explanatory sentence
is promoted from a `CareTeam.note.text` line buried in the generated
narrative table into a top-of-page description for the example. In
practice this means authoring an introductory sentence/paragraph
adjacent to the example (e.g., in the example's intro/markdown content
or an `Resource.text` div pre-amble) so that the IG publisher renders
"This care team was founded to optimize the management of complex
cardiac cases." as the lead description of the example rather than as
an entry in the auto-generated narrative table. The underlying
`CareTeam.note` would remain unchanged so that data semantics are
preserved.

#### Justification

This is exactly what the commenter asked for and is a low-risk,
example-only edit: it changes only the human-readable framing of an
example resource, not any profile, value set, or conformance
constraint. The 2025-02-18 re-opening was explicit that the prior
"tooling is responsible" disposition is no longer the workgroup's
position post-STU1, so a substantive disposition is now expected.
Promoting the most descriptive sentence to the example's intro
materially improves reader comprehension of why the example exists,
which is the primary value of bundled examples in an IG.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) changing the example's narrative
framing, refresh the *set* of VHDir examples so that each one has a
short authored intro paragraph that explains the scenario, with the
auto-generated FHIR-publisher narrative table left as the structured
detail underneath. Apply this as a small editorial pattern across the
VHDir IG examples (CareTeam, HealthcareService, Practitioner,
PractitionerRole, etc.) rather than touching only the single CareTeam
example called out in this ticket.

#### Justification

The same readability concern raised here applies broadly to VHDir's
generated examples — VHD #17 is one symptom of a wider issue. A
consistent "short authored intro + generated narrative" pattern is
cheaper to maintain than ad-hoc per-example edits and produces a more
uniform reader experience. It also keeps the change firmly in the IG
authoring layer (IG content, not tooling), addressing the original
2018 concern that this should not block on changes to the FHIR
publisher itself.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive* (or restore the prior *Considered for
Future Use* posture) on the basis that VHDir STU1 has already been
published, that narrative quality of generated examples is a
publisher-tooling concern outside the IG's scope, and that the
commenter retains the option to file a separate tracker against the IG
publisher for a friendlier default narrative output.

#### Justification

This mirrors the original 2018‑10‑17 vote (6‑0‑0) and the rationale
recorded at that time — the IG publisher generates the narrative, so
durable improvements belong in the tooling layer, not in per-example
hand-edits that will need to be re-applied as examples are
regenerated. With STU1 already out the door, the cost/benefit of
re-opening the IG for a cosmetic example tweak is low. There are no
implementer reports of confusion, no Zulip discussion arguing the
ticket is blocking adoption, and no cross-referenced Jira items
suggesting this is part of a wider corrective package.

---

### Recommendation

**Recommended disposition:** A

The 2025-02-18 reversal of the prior disposition is a strong signal
that the workgroup no longer wants to defer to the publisher-tooling
argument that drove the original close, so Disposition C would re-do
the work that has just been undone. Disposition B's IG-wide editorial
sweep is attractive in principle but is materially out of scope for a
single ballot comment and would itself require a fresh disposition
across multiple examples. Disposition A delivers exactly the change
the commenter asked for, is confined to one example in a published
IG, carries no profile or conformance impact, and leaves the door open
to apply the same pattern to other VHDir examples later if the
workgroup decides to generalise it.
