## Summary

The reporter identifies a conformance inconsistency in the U.S. National
Directory of Healthcare Providers and Services (NDH) IG. The NDH server
CapabilityStatement marks the canonical search parameter
`http://hl7.org/fhir/SearchParameter/individual-gender` as **SHALL** for
the `Practitioner` resource (NDH CapabilityStatement, Practitioner
section), yet the underlying element `Practitioner.gender` is neither
required (cardinality `1..1`) nor flagged **Must Support** in the
`ndh-Practitioner` StructureDefinition — and it does not inherit either
treatment from the base R4 `Practitioner` resource. The reporter argues
that whenever a search parameter is mandatory in the CapabilityStatement,
the element it targets should be at minimum Must Support so that
implementers can reasonably populate and search on it. The single
comment on the ticket, from John Moehrke (Patient Administration / NDH
co-chair), proposes the alternative path of dropping the search
parameter conformance from SHALL to SHOULD instead of strengthening
the element. The ticket has been queued for the 2026 January WGM
discussion under the NDH agenda.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the NDH IG, modify the `ndh-Practitioner` StructureDefinition so
that the `Practitioner.gender` element is flagged **Must Support**
(`mustSupport = true`) on the profile. Cardinality remains `0..1` (the
underlying R4 cardinality), but Must Support obligates conformant NDH
servers to be able to populate and consumers to be able to handle the
element. Leave the CapabilityStatement entry for the
`individual-gender` SearchParameter at `SHALL`. Update the IG's
narrative for `Practitioner` to call out that the `gender` element is
Must Support specifically because the search parameter is mandatory.

#### Justification

This is the change the reporter literally asked for and aligns with
the established FHIR design principle that any element a server is
required to expose for search must, at minimum, be Must Support so
implementers know they need to populate it and consumers know they
must accept it. Marking the element MS while leaving the search
parameter SHALL keeps NDH's directory query surface intact (gender is
a routine practitioner directory filter) and avoids weakening the
contract that querying clients depend on.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt John Moehrke's comment: in the NDH server CapabilityStatement,
**drop the conformance of the `individual-gender` SearchParameter on
`Practitioner` from SHALL to SHOULD**. Do not add Must Support to
`Practitioner.gender`. Optionally add a narrative note explaining that
gender is supported as a search parameter where data is available but
is not required, given the privacy and data-quality concerns around
practitioner gender data in directory contexts.

#### Justification

Practitioner gender is frequently sparse, optional, or deliberately
omitted in real-world provider directories (privacy, data-collection
practices, jurisdictional rules). Mandating a SHALL search on an
element that is neither required nor MS sets up implementers to fail
conformance trivially. Lowering the search parameter to SHOULD keeps
the query surface available for servers that do hold the data without
forcing all NDH servers to expose a query that may have no
populatable backing data. This resolves the inconsistency by
adjusting the weaker side of the constraint pair, which is the
lighter-weight change and matches the NDH co-chair's stated
preference.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as Not Persuasive. Take no change to either the
`ndh-Practitioner` profile or the `ndh-server` CapabilityStatement.
Document in the resolution that the inheritance behaviour is intended:
search parameter conformance and element MS are independent layers,
and a SHALL on a search parameter does not, in the IG authors' view,
imply that the targeted element must be Must Support.

#### Justification

This position would have to argue that NDH consciously decoupled
search-parameter conformance from element MS, perhaps because the
directory is a query-oriented surface and absence of a value for a
given practitioner is acceptable. However, this is hard to defend
against the reporter's plain reading: a SHALL search whose target
element has no MS or required cardinality leaves implementers no
guidance about populating the element, which is exactly the
ambiguity FHIR conformance flags exist to remove. Declining is
defensible only if the WG can point to explicit prior intent, which
nothing in the gathered material supports.

---

### Recommendation

**Recommended disposition:** B

The inconsistency is real and should be resolved, so Disposition C is
weak. Between A and B, Disposition B (lower the search parameter to
SHOULD) is the better fit for NDH for two reasons. First, practitioner
gender is a notoriously sparse and policy-sensitive field in real
directories; mandating it as Must Support across all NDH servers via
Disposition A imposes a population/handling obligation that may not be
operationally realistic for all jurisdictions and may push servers to
emit low-quality or unknown values just to satisfy MS. Second, the
NDH co-chair (John Moehrke) has already publicly recommended this
path on the ticket, which signals the WG is likely to converge here
quickly. Disposition A should remain on the table as a fallback if
the WG concludes that gender-based search is genuinely a core
directory query and SHOULD is too weak — in which case the inverse
(MS on the element) is the right way to reconcile.
