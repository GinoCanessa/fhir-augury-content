## Summary

The reporter, Joseph M. Lamy, asks the Patient Administration work group
to add explicit guidance to the Interoperable Digital Identity and
Patient Matching IG on how identifiers should be **normalized for
comparison** during Patient `$match` operations. The motivating example
is a `$match` call that failed because the request supplied an SSN with
dashes while the server stored the same SSN without dashes — a
difference that should not, in practice, prevent a match. The reporter
notes that the `terminology.hl7.org` SSN NamingSystem already says SSNs
are stored without dashes, that base FHIR `Identifier.value` allows
normalization for comparison ("removing non-significant whitespace,
dashes, etc.") with case-sensitive treatment as the default, and that
FHIR R4 search guidance on `token` actually leans the other way ("when
in doubt, treat tokens case-insensitively"). The IG currently provides
no equivalent guidance, which leaves matching engines to invent their
own behavior. The reporter suggests the IG (1) recommend
case-insensitive comparison for codes and identifiers except in known
case-sensitive systems like UCUM, (2) prescribe stripping non-numeric
characters (dashes, spaces, asterisks) for SSN comparisons, and (3)
prescribe stripping non-alphanumeric characters for other government
IDs such as passports and driver's licenses. No cross-referenced Jira
tickets were surfaced for this issue.

## Details

**Description:**

> I recently ran a Patient $match test against a server and got no
> match when I passed an SSN with dashes and the server had the same
> SSN stored without.
>
> I see where the [guidance](https://terminology.hl7.org/NamingSystem-ssn.html)
> for this identifier says "The SSN is represented in resources with
> dashes removed", but I wonder if matching logic for this and other
> identifiers should be more forgiving in this IG (Postel's Law).
>
> The base FHIR requirements for
> [Identifier.value](https://www.hl7.org/fhir/R4/datatypes-definitions.html#Identifier.value)
> state "...it may be normalized for comparison purposes (e.g. removing
> non-significant whitespace, dashes, etc.)....Identifier.value is to
> be treated as case sensitive unless knowledge of the Identifier.system
> allows the processer to be confident that non-case-sensitive
> processing is safe."
>
> Interestingly, the FHIR search page (here for discussion only; it is
> not directly related to $match) only mentions case sensitivity in
> comparisons, not normalizing in other ways. And interestingly, it
> has the [opposite recommendation](https://www.hl7.org/fhir/R4/search.html#token):
> "When in doubt, servers SHOULD treat tokens in a case-insensitive
> manner, on the grounds that including undesired data has less safety
> implications than excluding desired behavior."
>
> Surprisingly, I could find little more in Jira and Zulip.
>
> It seems that this IG could provide some helpful guidance on
> normalizing identifiers for comparison, both in general and with
> respect to specific identifiers. For example:
>
> - Consider (maybe suggest) case-insensitive comparisons for codes and
>   identifiers, except in known case-sensitive code systems like UCUM.
> - For SSN, normalize by removing non-numeric characters (dashes,
>   spaces, asterisks)
> - For other IDs like passports and driver licenses, normalize by
>   removing non-alphanumeric characters
>
> What do other systems do?

**Resolution (per ticket snapshot):**

The triage resolution proposes adding text to **section 3.8 Attribute
Applicability** that mirrors the base-FHIR `Identifier.value`
language — "it may be normalized for comparison purposes (e.g. removing
non-significant whitespace, dashes, etc.)....Identifier.value is to be
treated as case sensitive unless knowledge of the Identifier.system
allows the processer to be confident that non-case-sensitive processing
is safe." — followed by an SSN-specific clause that appears to be
truncated in the captured snapshot (`"SSN sh"`). The resolution is
recorded as **Persuasive**, indicating the work group accepted the
underlying need but the exact normative text still needs to be
finalized.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the Identity Matching IG, add a new normative subsection (under
**3.8 Attribute Applicability**, as proposed in the captured
resolution) titled "Identifier Normalization for Comparison". The
subsection would:

1. Restate, normatively, the base-FHIR `Identifier.value` rule —
   "Identifier.value MAY be normalized for comparison purposes (e.g.
   removing non-significant whitespace, dashes); Identifier.value
   SHALL be treated as case-sensitive unless knowledge of
   `Identifier.system` allows the processor to be confident that
   case-insensitive processing is safe."
2. Recommend that servers performing `$match` apply normalization
   **symmetrically** to both the inbound `Patient.identifier.value`
   and the stored value before comparison, scoped by
   `Identifier.system`.
3. Provide a table of system-specific normalization rules for the
   identifier systems already named in the IG, e.g.:
   - `http://hl7.org/fhir/sid/us-ssn` → strip all non-digit characters
     (dashes, spaces, `*`); compare digit-for-digit.
   - `urn:oid:2.16.840.1.113883.4.330` (US passport) and similar
     government IDs → strip non-alphanumerics; case-insensitive.
   - `urn:ietf:rfc:3986` UUIDs → comparison MAY ignore dashes; MUST
     remain case-insensitive.
4. State the UCUM-style exception explicitly: code systems flagged as
   case-sensitive in their `CodeSystem.caseSensitive` (or recorded as
   such in the IG's identifier table) MUST NOT be normalized.

The change is purely additive narrative (no StructureDefinition or
profile changes) and lands in the IG's `input/pagecontent/` markdown
under the section that renders as 3.8.

#### Justification

This is the most direct response to the reporter's concrete failure
mode (an SSN-with-dashes vs. SSN-without-dashes mismatch that
suppresses a Patient `$match`). The proposed text is anchored in
already-normative base-FHIR language, so it does not invent new
semantics — it operationalizes them for the matching context. Listing
per-system rules removes the ambiguity that today forces every
implementer to reinvent the same logic, and the UCUM-style exception
preserves safety where case or punctuation is meaningful. The Patient
Administration work group has already triaged the ticket as
**Persuasive**, indicating in-principle agreement with this direction.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adopting a normative per-system normalization table in the
IG itself, address the gap with a layered, less-prescriptive
approach:

1. In the IG, add a short, **non-normative** subsection in 3.8 that
   re-states the base-FHIR `Identifier.value` normalization clause
   verbatim and explicitly says the IG **does not mandate** a
   particular normalization algorithm for `$match`.
2. Add a SHOULD-level recommendation that servers document their
   identifier-normalization behavior (per `Identifier.system`) in
   their `CapabilityStatement` (or in the
   `Patient-match` `OperationDefinition` documentation), so that
   clients can know what to expect.
3. Move the per-system normalization rules (SSN, UUID, passport,
   etc.) into a **companion implementation-guidance document /
   "living FAQ"** maintained alongside the IG (as juliemaas
   suggested), curated by the FAST Implementers group, and link to
   it from 3.8.
4. Optionally, file a separate ticket to track the companion document
   and its initial entries (SSN dash-stripping, UUID dash-handling).

#### Justification

This honors the work group's stated preference (per juliemaas's
comment) of **not being prescriptive about the matching algorithm**
while still closing the discoverability gap that the reporter ran
into. It avoids the risk that a normative table baked into the IG
becomes wrong or incomplete as new identifier systems appear (each
update would require a republish of the IG). Documenting normalization
behavior in `CapabilityStatement` shifts the burden from "every
implementer must reverse-engineer matching rules" to "every server
must declare them", which scales better and supports interop testing.
The companion doc gives the community a place to add SSN/UUID/etc.
rules at FAST cadence rather than IG cadence.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" (or "Not Persuasive with
Modifier") on the grounds that base FHIR `Identifier.value` already
permits normalization for comparison and the SSN NamingSystem at
`terminology.hl7.org` already specifies dash-free representation;
implementers experiencing the reported mismatch should look to those
existing artifacts and to server-side configuration rather than to
new IG text. Optionally, add a one-line cross-reference from 3.8 to
the base-FHIR `Identifier.value` definition and to the SSN
NamingSystem, but otherwise make no changes.

#### Justification

The matching algorithm is intentionally left to implementers in this
IG, and prescribing per-system normalization rules risks codifying
behavior that conflicts with regional, regulatory, or vendor-specific
requirements (e.g., some jurisdictions disallow storing SSNs at all,
some require fixed-format storage with dashes). The reported
interoperability failure is arguably an implementation defect on the
server side — it stored a normalized SSN but did not normalize the
inbound query, which is a bug the server should fix regardless of IG
text. Adding normative normalization rules also creates a maintenance
liability: the IG would have to track every identifier system anyone
uses with `Patient.identifier`.

---

### Recommendation

**Recommended disposition:** B

Disposition A directly solves the reporter's problem but commits the
IG to maintaining a normative table of per-system normalization rules
forever — and the Patient Administration work group has already
signaled (via juliemaas's comment) that it deliberately avoids being
prescriptive about matching algorithms. Disposition C respects that
posture but leaves the underlying problem (implementers reinventing
normalization, asymmetrically) entirely unaddressed, which is exactly
what produced the reporter's failed `$match`. Disposition B threads
the needle: it adds a small amount of non-normative IG text that
points implementers at the right base-FHIR clause, makes
normalization behavior **discoverable** by requiring servers to
document it (CapabilityStatement / OperationDefinition), and pushes
the volatile per-system rules (SSN, UUID, passport, driver's licence)
into a living companion document the FAST Implementers can curate.
That preserves the work group's algorithm-agnostic stance, gives
implementers a concrete remedy, and keeps the IG itself
maintenance-light. The triage resolution of "Persuasive" is consistent
with adopting the lighter-touch B variant rather than the full
normative table in A.
