## Summary

The ticket — titled *"Removal of NDH Qualification Extension"* — proposes
that the National Directory of Healthcare Providers and Services (NDH) IG
remove its custom qualification-related extension on `PractitionerRole`
in the current ballot cycle. The reporter (David Pyke) argues that while
the value of carrying role-specific qualification information may be real
in a future version, it currently adds complexity to the specification
without an immediate supplier or consumer. A 12/9/2024 HL7 FAST National
Directory Public Call (recorded in a comment by the assignee) confirmed
this direction: feedback from the group (Benji, Ron and others) was that
qualifications and state licensure introduce implementation complexity
that is not justified for STU2, and that NDH should restrict provider
characterization to **specialty using NUCC codes** for STU2 — NUCC being
the value set already supported by PlanNet and NPPES — with the option to
re-introduce additional qualification mechanisms in a future release.

A later technical comment pushes back, arguing on principle that all three
`PractitionerRole` "layers of provider identity" — `qualification`
(legal authorization / license), `code` (job function performed in the
organization, e.g. PCP, Hospitalist), and `specialty` (clinical specialty
practiced in the role, e.g. Cardiology) — are needed to satisfy CMS /
No Surprises Act directory accuracy rules, network adequacy calculations,
delegated credentialing, and provider search by function/specialty/license.
John Moehrke also asks how the workgroup can determine whether the
qualification structure is actually needed in practice rather than in
theory.

No cross-referenced Jira tickets, Zulip threads, or GitHub PRs/issues
were surfaced for this ticket; the only cross-references in the index are
mentions of the three `PractitionerRole` elements above.

**Comments:**

- **john_moehrke** (2026-01-19): Asks David Pyke "how can we determine
  the reality on if these are needed or if they are just justified by
  theory?" — i.e. requests evidence of real implementer demand before
  acting.
- **JIRAUSER20919** (2025-11-22): Long technical comment laying out the
  three distinct layers of provider identity in `PractitionerRole` and
  why NDH needs all three:
  - `PractitionerRole.qualification` — the *person's* credentials used
    in this role (MD, DO, RN, LCSW, board certification, state license
    number). Represents legal authorization. Needed for credential
    verification, delegated credentialing, and directory accuracy
    audits.
  - `PractitionerRole.code` — the *job function or role* performed in
    the organization/network (Primary Care Physician, Attending
    Cardiologist, Telehealth Provider, Hospitalist, On-call provider).
    Drives directory search ("Find a PCP", "Find a counselor").
  - `PractitionerRole.specialty` — the *clinical specialty* practised
    in this role (Cardiology, Endocrinology, Oncology, Orthopedics,
    Pediatric Medicine, Psychiatry). Drives network adequacy,
    directory search by specialty, plan-network validation, and
    QHP / MA directory rules.

  The author concludes that NDH's use cases — CMS / No Surprises Act
  directory accuracy, network adequacy, delegated credentialing,
  multi-organization relationships, and combined search — "cannot be
  expressed correctly without maintaining all three levels."
- **JIRAUSER17037** (2025-01-03): Records the 12/9/2024 FAST National
  Directory Public Call discussion. Group feedback (Benji, Ron, others)
  was that the qualification extension as drafted adds implementation
  complexity (qualifications, state licensure, etc.) that is not
  warranted for STU2. **Direction was to change NDH to only include
  specialty using NUCC codes for STU2** (NUCC is supported by PlanNet
  and NPPES). Additional qualifications could be added in a future
  release. Open follow-up: check whether specialties are also being
  carried under the `qualifications` element.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the NDH-defined qualification extension on `PractitionerRole`
from the NDH IG for STU2. Concretely, in the NDH IG profile of
`PractitionerRole`:

- Drop the IG's custom `qualification` extension definition and remove
  its `extension` slice from the NDH `PractitionerRole` profile
  (StructureDefinition).
- Where qualification information is currently described in the NDH
  narrative pages, replace it with guidance that role characterization
  for STU2 is conveyed via `PractitionerRole.specialty` bound to
  **NUCC Health Care Provider Taxonomy** codes (the value set already
  used by PlanNet and NPPES), retaining the FHIR-base
  `PractitionerRole.qualification` element as-is (no IG-specific
  constraint, no requirement to populate).
- Update conformance statements, examples, search-parameter
  documentation, and IG narrative to remove references to the
  removed extension.
- Add a forward-looking note indicating that a richer qualification
  mechanism may be re-introduced in a future NDH release once
  implementer demand is demonstrated.
- Action the open to-do from the 1/3/2025 comment: confirm that
  specialty values are not also being carried inside the
  `qualifications` slice in any NDH example or profile, and clean up
  if they are.

#### Justification

This is exactly the direction recorded by the FAST National Directory
Public Call on 12/9/2024 (Benji, Ron, others) and aligns with the
ticket reporter's stated rationale: the extension adds implementation
complexity without a confirmed supplier or consumer in STU2.
Constraining STU2 to `specialty` with NUCC re-uses a value set already
in production with PlanNet and NPPES, which lowers the migration cost
for the dominant existing US provider-directory implementer base.
It also directly answers John Moehrke's "theory vs. reality"
question — until real implementations need the additional layers,
the IG should not normatively require them.

---

### Disposition B: Alternative Approach

#### Proposal

Retain the structural distinction described in JIRAUSER20919's comment
but make it **optional and lighter-weight** rather than removing the
extension outright. Specifically:

- Keep the IG-level acknowledgement of the three layers
  (`PractitionerRole.qualification` for legal authorization,
  `PractitionerRole.code` for job function, `PractitionerRole.specialty`
  for clinical specialty) in the NDH narrative.
- Drop the NDH-defined extension *as drafted* (so we are not shipping a
  bespoke complex extension in STU2), and instead profile the
  base-spec `PractitionerRole.qualification` element with `0..*` (no
  `mustSupport`) plus a non-binding example value-set slot, leaving
  the door open for implementations that already track licensure to
  send it without forcing those that don't.
- Require `PractitionerRole.specialty` bound to NUCC for STU2 (same as
  Disposition A).
- Add a clearly marked "future direction" section flagging that
  delegated credentialing, license-level search, and No Surprises
  Act-driven directory accuracy reporting are expected to mature in a
  future release and may add `mustSupport` on `qualification` then.

#### Justification

This preserves the conceptual model that the technical commenter
argues is necessary for CMS / No Surprises Act compliance and
delegated credentialing, without forcing every STU2 implementer to
build the full qualification machinery now. It is a smaller break
from the current draft (no full removal of the concept), keeps the
upgrade path to a richer STU3 cleaner, and avoids a future
breaking re-introduction of an extension. The cost is a slightly
larger STU2 surface than Disposition A, and the risk that "optional
but defined" extensions get inconsistently populated by early
implementers.

---

### Disposition C: Decline

#### Proposal

Decline the request and **keep** the NDH qualification extension as
currently drafted in the IG. Rationale category: *premature removal —
underlying use cases (credential verification, delegated credentialing,
directory accuracy under No Surprises Act / CMS rules) are real and
already documented in the ticket itself.*

#### Justification

The technical comment from JIRAUSER20919 enumerates concrete US
regulatory and operational drivers (CMS provider directory accuracy,
No Surprises Act, network adequacy, delegated credentialing, combined
search) that depend on distinguishing legal authorization from job
function from clinical specialty. If the IG removes the qualification
extension for STU2 and then needs to re-introduce equivalent
machinery in STU3, implementers face a churn cost and the IG ships
a known-incomplete provider model in the interim. Declining preserves
the current capability and shifts the burden to the workgroup to
gather real implementer-balloter feedback during STU2 ballot rather
than pre-emptively cutting scope.

---

### Recommendation

**Recommended disposition:** A

The 12/9/2024 FAST National Directory Public Call gave explicit
group-level direction to drop the qualification extension and
constrain STU2 to specialty-via-NUCC, and the ticket reporter (David
Pyke) endorses that direction. NUCC is already the value set used by
the two largest US-relevant provider directories (PlanNet, NPPES), so
Disposition A minimizes implementer disruption for STU2 while
explicitly leaving room to re-introduce richer qualification handling
when a concrete supplier/consumer pair appears — directly answering
John Moehrke's "theory vs. reality" challenge. Disposition B is a
reasonable fallback if the workgroup hears, during ballot, from
implementers that already populate `PractitionerRole.qualification`
and don't want to lose round-tripping; in that case a non-mustSupport
profile of the base element is the cheapest hedge. Disposition C
should be reserved for the case where ballot comments produce concrete
evidence that the full qualification model is in active use today —
the comment record on this ticket does not yet show that evidence.
