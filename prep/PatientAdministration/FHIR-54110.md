## Summary

The reporter notes that many patient-monitoring vendors and customers
routinely use a "VisitNumber" as a patient-facing identifier, but FHIR
does not surface a "VisitNumber" code in the
[`identifier-type`](https://build.fhir.org/valueset-identifier-type.html)
value set used by `Patient.identifier.type` (or any other
`Identifier.type`). The only natural FHIR home for the concept is
`Encounter.identifier`, but the spec text does not call out
"VisitNumber" by name, which leaves implementers uncertain whether they
should (a) place the value on `Encounter.identifier` and label it some
other way, or (b) place it on `Patient.identifier` with a custom type.

The reporter requests one of two changes:

1. Expand the description of `Encounter.identifier` to explicitly call
   out "VisitNumber" so customers stop trying to model it as a Patient
   identifier; **or**
2. Add a "VisitNumber" code to the FHIR `identifier-type` value set so
   that it can be used on `Patient.identifier.type` (and elsewhere)
   alongside codes such as MR (medical record number).

No cross-referenced Jira tickets were returned by the augury index, so
the request stands on its own. However, the underlying ask has come up
repeatedly on `chat.fhir.org` over many years (see Related Zulip
Discussions), where the consistent answer has been "use
`Encounter.identifier` with `type` coded as `VN` from the HL7 v2-0203
CodeSystem"  — a pattern that is technically already legal under the
extensible binding but is invisible from the published value set.

Elliot Silver (Feb 2019) opens the same question — `Identifier.type`
has an extensible binding that draws from v2-0203, which defines `VN`,
yet `VN` is not in the default `identifier-type` ValueSet. Lloyd
McKenzie responds that "Encounter id" / admission id belongs in
`Encounter.identifier` and that there is no standard scheme for
assigning identifiers, since FHIR does not try to standardise business
practices. Kevin O'Donnell and René Spronk note that PV1-19 maps to
different hierarchical levels (organisational encounter, sub-encounter,
billing-level stay) in different countries. The thread reaches rough
consensus that the right home is `Encounter.identifier`, and that
`Identifier.type` can be coded with v2-0203 `VN` even though it is not
in the published value set, but Lloyd cautions that not all systems
will disambiguate by `Identifier.type`.

Participants: Elliot Silver, Kevin O'Donnell, Lloyd McKenzie, René
Spronk.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt **option 2** from the ticket: add `VN` ("Visit number", from
`http://terminology.hl7.org/CodeSystem/v2-0203`) to the FHIR
`identifier-type` ValueSet
([`http://hl7.org/fhir/ValueSet/identifier-type`](https://build.fhir.org/valueset-identifier-type.html)).
The existing extensible binding on `Identifier.type` would then surface
`VN` in the published expansion, eliminating the validator warnings
seen by implementers like frank cao and matching the behaviour of
sibling codes such as `MR` (Medical record number). In parallel,
tighten the descriptive prose on `Encounter.identifier` so it
explicitly states that v2 PV1-19 / "VisitNumber" / admission id /
encounter id all belong here (option 1 from the ticket), while making
clear that placing a visit number on `Patient.identifier` is **not**
the recommended pattern — `Encounter.identifier` remains the canonical
location.

#### Justification

`VN` is already a legitimate code under the extensible binding (it
lives in the bound CodeSystem v2-0203), so adding it to the default
ValueSet expansion is a non-breaking, low-risk change that simply makes
existing legal usage discoverable. It directly addresses the validator
friction reported in the implementers Zulip thread "Adding 'VN' to
identifer type value set" and the years-old "Admission ID" thread.
Pairing the ValueSet change with clearer guidance on
`Encounter.identifier` answers the reporter's underlying confusion
(customers reaching for `Patient.identifier` because the Encounter
docs do not mention VisitNumber) without endorsing visit number as a
patient-level identifier — which Lloyd McKenzie's responses
consistently push back on.

---

### Disposition B: Alternative Approach

#### Proposal

Decline to add `VN` to the `identifier-type` ValueSet, but **clarify
the spec** so the existing pattern is unambiguous:

1. Update `Encounter.identifier` short/definition text to explicitly
   call out "Visit number / admission identifier (HL7 v2 PV1-19)" as
   the intended use, and to recommend `Identifier.type.coding` =
   `{system: http://terminology.hl7.org/CodeSystem/v2-0203, code: VN}`.
2. Add a sentence to `Identifier.type` (or to the `identifier-type`
   ValueSet description) reminding implementers that the binding is
   **extensible** and that any code from v2-0203 — including `VN` —
   may be used without producing a conformance error.
3. Add a worked example to `Encounter` (or to the Identifier datatype
   page) showing an `Encounter.identifier` with `type.coding` = `VN`.
4. Explicitly note in the Patient page that visit-level identifiers
   belong on `Encounter.identifier`, not `Patient.identifier`.

#### Justification

The Zulip history shows repeated, confident answers from PA leadership
(Lloyd McKenzie, René Spronk) that `Encounter.identifier` is the
correct home and that the v2-0203 `VN` code is already usable via the
extensible binding. The real problem is **discoverability and
documentation**, not a missing code. A documentation-only fix avoids
ValueSet churn, keeps the curated default expansion small (which has
historically been the policy for `identifier-type` — only the most
broadly applicable codes are surfaced), and still resolves the
reporter's concern about customer confusion. It also acknowledges
Lloyd's caution from the 2019 thread that not all systems will
disambiguate by `Identifier.type`, so promoting `VN` could create a
false impression that downstream tooling will treat it specially.

---

### Disposition C: Decline

#### Proposal

Mark the request as **Persuasive with mod – No change required**, on
the grounds that the underlying need is already fully satisfied by
existing FHIR mechanisms:

- The binding on `Identifier.type` is **extensible**, so implementers
  may already supply `Identifier.type.coding` =
  `{system: http://terminology.hl7.org/CodeSystem/v2-0203, code: VN}`
  on `Encounter.identifier` without conformance error.
- `Encounter.identifier` is the modelled home for visit/admission
  identifiers; placing a visit number on `Patient.identifier` would
  be a category error, since a visit number identifies an encounter,
  not a patient (a patient typically has many).
- "VisitNumber" semantics vary internationally (PV1-19 can mean an
  admission, A&E attendance, community contact, or outpatient
  appointment, per the Kevin Mayfield Zulip thread), so promoting it
  to a globally curated identifier-type code risks implying a
  uniformity that does not exist.

Provide a short Jira response pointing reporters at this guidance.

#### Justification

This option is defensible because every part of the request is
technically already achievable today: `VN` is a legal `Identifier.type`
code under the extensible binding, and `Encounter.identifier` is the
documented home for visit identifiers. Adding `VN` to the default
ValueSet would not unlock any new behaviour — it would only reduce a
validator warning that some toolchains emit. However, declining
without **any** documentation change leaves the original confusion
intact: the reporter went looking on the Patient page and could not
find guidance, and the implementers Zulip threads show this is a
recurring pattern. So while declining is defensible on a strict
"no change required" reading, it is the weakest option for
implementer experience.

---

### Recommendation

**Recommended disposition:** A

The community evidence in Zulip (three independent threads spanning
2019–2025) shows the same pain point recurring: implementers see that
`VN` is conceptually valid, are told to use `Encounter.identifier` with
`type=VN`, and then either get validator warnings or end up modelling
visit number on `Patient.identifier` because it is not visible in the
default expansion. Disposition A removes the validator friction
(which is the concrete cost frank cao reported) by adding `VN` to the
ValueSet, **and** fixes the discoverability problem the reporter
identified by clarifying `Encounter.identifier` and explicitly steering
implementers away from putting visit number on `Patient.identifier`.
Disposition B preserves the status quo of the ValueSet but does
nothing to silence validators on conformant data, which is a real
day-to-day implementer cost. Disposition C is technically correct but
does not improve the implementer experience that drove the ticket.
The combined ValueSet + documentation change in A is non-breaking,
matches existing usage already permitted by the extensible binding,
and answers both halves of the reporter's "either/or" request.
