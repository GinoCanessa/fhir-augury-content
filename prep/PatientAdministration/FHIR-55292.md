## Summary

This ticket flags terminology artifacts associated with the FHIR `Endpoint`
resource as candidates for migration from the FHIR Core specification to the
Terminology.HL7.Org (THO) repository. Specifically, it identifies the
`EndpointEnvironment` CodeSystem and the `endpoint-environment` ValueSet
(bound to `Endpoint.environmentType`) as content whose long-term governance,
re-usability across HL7 product families (CDA, V2, FHIR), and likely future
maintenance argue for living in THO rather than in the FHIR core build. The
request originates from a broader value-set-triage exercise (imported as
"Comment 256" by the reporter), and it is one of several per-resource
tickets generated from that pass. No cross-referenced Jira tickets, Zulip
discussions, or GitHub items are linked in Augury at this time.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the `EndpointEnvironment` CodeSystem and the
`endpoint-environment` ValueSet from the FHIR Core build into THO
(`terminology.hl7.org`). In FHIR Core, replace the locally-defined
artifacts with references to their new THO canonical URLs (e.g.,
`http://terminology.hl7.org/CodeSystem/endpoint-environment` and
`http://terminology.hl7.org/ValueSet/endpoint-environment`), keep the
binding on `Endpoint.environmentType` unchanged in strength, and add a
ValueSet/CodeSystem alias / redirect note so that the existing canonical
URLs continue to resolve. Coordinate the change with the Vocabulary WG
and bump the relevant THO publication to include the migrated artifacts
before the next FHIR Core ballot/publication that drops the local copies.

#### Justification

The PA WG has already endorsed this direction at the Jan 2026 WGM
("Good candidate for moving to THO"). Endpoint connection environments
are inherently a cross-product concept — the same enumeration of
"production / test / development / training" semantics is useful in V2
messaging, CDA implementation guides, and non-FHIR endpoint catalogs,
which is exactly the criterion the value-set-triage rubric calls out.
Hosting the codes in THO lets them evolve on the THO release cadence
(independently of FHIR Core ballots) and avoids the long-standing
problem of FHIR-Core-versioned terminology being awkward to consume from
other product families.

---

### Disposition B: Alternative Approach

#### Proposal

Migrate the `EndpointEnvironment` CodeSystem to THO (because the codes
themselves are reusable across product families), but **leave the
`endpoint-environment` ValueSet in FHIR Core** as a thin wrapper that
includes-all from the THO CodeSystem. The binding on
`Endpoint.environmentType` is then a FHIR-Core ValueSet that points at
THO codes. This keeps FHIR Core in control of the binding-shaped artifact
(useful if PA later wants to constrain or extend the value set for FHIR
specifically), while still making the underlying codes reusable from
CDA, V2, and other specs.

#### Justification

ValueSet bindings are often spec-specific even when the underlying codes
are universal — a CDA template may want a different subset, a profile
may want to extend the list, etc. Migrating only the CodeSystem
preserves THO as the authoritative source of truth for the codes
themselves while leaving FHIR Core able to iterate the binding without
a THO release. This is also a smaller, lower-risk migration: only one
artifact moves, and consumers of the FHIR ValueSet canonical URL see no
change at all.

---

### Disposition C: Decline

#### Proposal

Leave both the `EndpointEnvironment` CodeSystem and the
`endpoint-environment` ValueSet in FHIR Core unchanged. Mark the ticket
"Persuasive with mod / Not Persuasive" with a rationale that the codes
are tightly coupled to the FHIR `Endpoint` resource, that there is no
demonstrated CDA/V2 consumer asking for them, and that the migration
cost (THO publication coordination, canonical URL redirects, downstream
implementer churn) is not justified by speculative future reuse.

#### Justification

Value-set migration to THO is explicitly described in the ticket itself
as *not required*. If no concrete cross-product consumer has surfaced,
moving the artifact incurs real coordination and breaking-change risk
(canonical URLs, IG dependencies, tooling caches) for hypothetical
benefit. Declining keeps the artifact owned by the WG that defines the
resource it binds to, which simplifies future edits.

---

### Recommendation

**Recommended disposition:** A

PA has already signalled at the Jan 2026 WGM that this is a "good
candidate for moving to THO," and the artifact in question
(environment classification: production / test / development / etc.) is
exactly the kind of cross-product, slowly-changing terminology that THO
exists to host. Disposition B is a defensible compromise if PA later
discovers it wants FHIR-specific freedom over the binding shape, but in
the absence of any current FHIR-specific constraint on the value set,
splitting the CodeSystem from the ValueSet adds indirection without
benefit. Disposition C is hard to justify given the WG's own stated
position on the ticket. The implementation work is mechanical
(coordinate a THO publication, add redirects, update the binding's
canonical URL) and well-trodden — multiple PA artifacts have made the
same trip. Recommend accepting and routing the change to Vocabulary for
THO publication coordination.
