## Summary

The ticket reports that many `SearchParameter` resources in the
Validated Healthcare Directory (VhDir) Implementation Guide use an
incorrect FHIRPath form to navigate into extensions. Specifically, the
existing `SearchParameter.expression` values use the shorthand
`Resource.extension('url')` invocation pattern (e.g.,
`Endpoint.extension('http://hl7.org/fhir/uv/vhdir/StructureDefinition/endpoint-usecase').extension('standard')`),
which is not valid FHIRPath. The reporter (Saul Kravitz) asks that all
extension-traversing search parameters in the IG be corrected to the
canonical filter form `Resource.extension.where(url='...')` — for
example,
`Endpoint.extension.where(url='http://hl7.org/fhir/uv/vhdir/StructureDefinition/endpoint-usecase').extension.where(url='standard')`
— and that every search parameter be re-validated using a FHIRPath
engine. No other Jira tickets, Zulip threads, or GitHub items are
linked through the cross-reference index.

## Details

**Description:**

> Many of the searchParameters involving extensions are incorrectly coded.
> All of them should be checked and tested using FHIRPath.
>
> For example: http://build.fhir.org/ig/HL7/VhDir/SearchParameter-endpoint-usecase-standard.json.html
>
> The expression included is:
>
> `"expression" : "Endpoint.extension('http://hl7.org/fhir/uv/vhdir/StructureDefinition/endpoint-usecase').extension('standard')"`
>
> It should be:
>
> `"expression" : "Endpoint.extension.where(url='http://hl7.org/fhir/uv/vhdir/StructureDefinition/endpoint-usecase').extension.where(url='standard')"`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Audit every `SearchParameter` resource published by the VhDir IG (the
authoring set under the IG's `input/resources/` SearchParameter folder,
including but not limited to `SearchParameter-endpoint-usecase-standard`)
and rewrite any `expression` that uses the invalid
`Resource.extension('url')` shorthand to use the canonical
`Resource.extension.where(url='...')` filter form. After the rewrite,
run each expression through a FHIRPath engine (e.g., the HAPI or
firely-net FHIRPath evaluator) against representative example
instances to confirm it parses and resolves to the intended target
node(s). Republish the IG with the corrected expressions.

#### Justification

The FHIRPath specification does not define a function-style
`extension(url)` invocation on `Resource.extension`; the correct
pattern for navigating to an extension by URL is
`extension.where(url='...')`. This pattern is what the FHIR core
specification, the FHIR-IG-Publisher's QA, and downstream FHIRPath
engines all expect, and it is what core-spec search parameters use
when they traverse extensions. Adopting the corrected form makes the
VhDir search parameters actually executable by conformant servers and
brings the IG into line with the rest of the FHIR ecosystem. The
ticket itself supplies a worked example of the fix, so the change is
mechanical once each affected SearchParameter is identified.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to the syntactic correction in Disposition A, take the
opportunity to add automated FHIRPath validation of all
`SearchParameter.expression` values to the VhDir IG's build pipeline
(for example, a small validation step in the IG's CI that loads each
SearchParameter, parses the expression with a FHIRPath engine, and
optionally evaluates it against the IG's example instances). This
prevents the same class of defect from recurring as new search
parameters are added or as expressions are edited.

#### Justification

The reporter explicitly asks that all expressions "be checked and
tested using FHIRPath", which signals that the underlying problem is
not just the few expressions cited in the ticket but the lack of any
mechanism that would have caught them. A one-time fix leaves the IG
exposed to regression the next time a SearchParameter is authored.
Wiring FHIRPath parsing/evaluation into the IG build is a small,
contained investment that converts this ticket from a point fix into
a structural improvement in IG quality, while still delivering
everything Disposition A promises.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — Out of Scope" on the grounds
that the cited URL is the auto-build (`build.fhir.org/ig/HL7/VhDir`)
rather than a published VhDir release, and ask the reporter to
re-file against a specific published version of the IG if the same
defect remains there.

#### Justification

This rationale is technically defensible only in the narrow sense
that the example link points at the continuous-build IG output. It is
weak on the merits because the defect is in the source SearchParameter
resources, not in the published rendering, and because the FHIRPath
syntax used is independently incorrect regardless of which build of
the IG it appears in. Decline is included for completeness so the
workgroup can compare it against the substantive options.

---

### Recommendation

**Recommended disposition:** B

The substantive defect — invalid FHIRPath in `SearchParameter.expression`
— is real, well-isolated, and fully described in the ticket, so some
form of acceptance (A or B) is clearly warranted; declining (C) would
leave the IG shipping search parameters that no conformant server can
execute. Between A and B, B is preferred because the reporter's own
phrasing ("All of them should be checked and tested using FHIRPath")
treats the audit as the primary deliverable and the example as one
instance of a class. Adding a FHIRPath validation step to the VhDir
build is a small incremental cost on top of the rewrite and protects
the IG from re-introducing the same defect, which is a much better
return on workgroup attention than fixing the same expressions a
second time in a future ballot cycle.
