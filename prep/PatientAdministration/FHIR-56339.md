## Summary

Lisa R. Nelson is requesting that the NDH (National Directory of Healthcare
Providers and Services) IG define one or more search parameters on
`PractitionerRole` that allow a client to retrieve **all endpoints for a given
Practitioner together with the context** in which those endpoints are used —
specifically the `Organization` the Practitioner is working for and the role
they are playing when they send or receive information through that endpoint.

The ticket is filed as a companion to [FHIR-56338](https://jira.hl7.org/browse/FHIR-56338)
("Remove endpoint extension from Practitioner Resource"), which argues that the
NDH-defined extension allowing a direct `Practitioner → Endpoint` reference is
unsafe because it produces "contextless" endpoint information. FHIR-56339 is
the constructive counterpart: if the Practitioner-level shortcut is removed,
NDH still needs a first-class, queryable path that returns the same
information *with* the role/organization context attached. Together the two
tickets express a single design intent — "endpoints for a practitioner must
always be retrieved through `PractitionerRole`, never directly through
`Practitioner`."

The reporter does not yet propose a specific search-parameter name, type, or
`_include` chain; the ticket is a request for the work group to design and
publish the appropriate query mechanism in NDH.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add (or formalize) a set of search parameters on `PractitionerRole` in the
NDH IG that lets a client retrieve every endpoint a given Practitioner uses,
together with the role and organization context, in a single round-trip.
Concretely:

- Confirm/define the canonical search:
  `GET [base]/PractitionerRole?practitioner=Practitioner/{id}&_include=PractitionerRole:endpoint&_include=PractitionerRole:organization`
  as the **normative** way for clients to discover a Practitioner's endpoints
  in an NDH directory.
- Document this pattern in an NDH "endpoint discovery" page, with worked
  examples showing the bundle returned (PractitionerRole + included
  Endpoint(s) + included Organization + included Practitioner) and an
  explicit note that querying `Endpoint` directly from `Practitioner` is
  out of scope / not supported.
- Mark `PractitionerRole.endpoint`, `PractitionerRole.organization`,
  `PractitionerRole.practitioner`, `PractitionerRole.code`, and
  `PractitionerRole.specialty` as **Must Support** on the NDH
  `PractitionerRole` profile so that the contextual elements the search is
  meant to surface are actually present in conformant servers.
- Add a `CapabilityStatement` requirement that NDH servers SHALL support
  `_include=PractitionerRole:endpoint`, `_include=PractitionerRole:organization`,
  and `_include=PractitionerRole:practitioner`.
- Cross-reference this search guidance from the `Endpoint` profile page and
  from any narrative that today suggests a Practitioner-direct endpoint
  lookup.

#### Justification

This is the most direct response to Lisa's request and aligns with the
core FHIR design intent for `PractitionerRole` (see the `PractitionerRole`
resource notes that explicitly position it as the contextual binding
between Practitioner, Organization, and Endpoint). It also dovetails with
the disposition Lisa proposes in the companion ticket FHIR-56338: if the
Practitioner-level endpoint extension is removed, NDH still owes
implementers a clearly documented, queryable path to the same data.
`_include` chaining is already idiomatic in US Core / NDH-style directory
queries (e.g., the existing `practitionerrole-network` and
`new-patient-from-network` search parameters), so adopting it here keeps
the IG internally consistent.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than relying on `_include` chains, define a **named NDH operation**
(or a custom composite search parameter) that wraps the multi-resource
fetch into a single, semantically explicit call — for example
`$practitioner-endpoints` invoked on `Practitioner/{id}` (or on
`PractitionerRole`) that returns a Bundle of `{ PractitionerRole,
Organization, Endpoint }` triples for the practitioner.

Implementation sketch:

- Define `OperationDefinition/practitioner-endpoints` in the NDH IG.
- Input: `Practitioner` reference (and optional filters such as
  `connection-type`, `network`, `active-only`).
- Output: searchset Bundle with one `PractitionerRole` per role/org
  combination, each accompanied by its referenced `Endpoint(s)` and
  `Organization`.
- Keep the `PractitionerRole`-based `_include` search from Disposition A
  as the underlying primitive but make the operation the recommended
  client API.

#### Justification

A named operation makes the *intent* of the query explicit ("get this
practitioner's endpoints, with context") instead of leaving it to clients
to discover the right `_include` incantation. It also gives the work group
a stable extension point for future filtering needs (e.g., filtering by
connection type, active period, in-network status) without growing the
search-parameter surface on `PractitionerRole`. The downside vs.
Disposition A is added implementation cost for servers and one more
NDH-specific surface that off-the-shelf FHIR clients won't auto-discover —
which is why this should be additive to, not a replacement for, the
underlying `PractitionerRole` search.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the requested capability is
**already achievable today** with base-FHIR search semantics:
`GET PractitionerRole?practitioner=Practitioner/{id}&_include=PractitionerRole:endpoint&_include=PractitionerRole:organization`
works against any conformant FHIR server, and `PractitionerRole` already
carries the `practitioner`, `organization`, `endpoint`, and role/specialty
elements needed for context. Resolve the ticket as "Not Persuasive — Already
Supported" and instead address the underlying problem entirely through
FHIR-56338 (removing the contextless `Practitioner.endpoint` extension)
plus a documentation note pointing implementers at the existing search.

#### Justification

`PractitionerRole` was designed precisely for this contextual binding, and
no new search parameter is technically required to satisfy the stated use
case. Adding NDH-specific search parameters or operations risks divergence
from base FHIR search idioms and creates additional conformance burden on
NDH server implementers without a clear capability gap. The risk is that
"already supported" is *technically* true but **operationally insufficient**
in an IG context — implementers tend to do what the IG explicitly tells
them to do, so silence on the discovery pattern is what allowed the
Practitioner-extension shortcut to appear in the first place. That is the
main argument against this disposition.

---

### Recommendation

**Recommended disposition:** A

The work group should accept the ticket and formalize a `PractitionerRole`-
based endpoint-discovery search in the NDH IG. The technical capability
exists in base FHIR (Disposition C), but the very existence of the
Practitioner-level endpoint extension that prompted FHIR-56338 demonstrates
that "available in base FHIR" has not been enough — implementers reach for
shortcuts when the IG does not spell out the correct query pattern.
Disposition A closes that gap with minimal new machinery: a documented
search, Must Support on the contextual elements, and a `CapabilityStatement`
requirement on the relevant `_include` parameters. It also pairs cleanly
with the proposed resolution of FHIR-56338, presenting the work group with
a coherent "remove the unsafe shortcut, document the safe path" story
rather than two independent decisions. Disposition B (a named operation)
is attractive as a future enhancement and can be revisited once the
baseline search guidance is in place, but it should not block adoption of
the simpler, idiomatic search pattern now.
