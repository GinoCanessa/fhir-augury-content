## Summary

The reporter asks the Patient Administration work group to add a
`udiCarrier` value to the `Device` example used in the RTLS (Real Time
Location Services) FHIR specification — i.e., to populate UDI carrier
information on the example RTLS Tag Device resource so that the example
demonstrates how UDI data would appear in practice.

A companion ticket, [FHIR-40970](https://jira.hl7.org/browse/FHIR-40970)
("Add udiCarrier in the Location Update Event example"), makes the same
request against the Location Update Event example. The two tickets were
filed together by the same reporter on the same day and have been
triaged together. The work group's draft resolution distinguishes the
two: the RTLS badge itself (the device represented in this ticket's
example) is not, to the work group's knowledge, a regulated medical
device and therefore does not carry a UDI, so this ticket is proposed
**Not Persuasive**. The companion FHIR-40970 is being treated as
**Persuasive with Modification** because UDI data can legitimately
appear when a medical device (e.g., an infusion pump) is *enrolled*
with an RTLS tag — the work group will instead add a UDI-bearing
example to the *Enroll Tag* message rather than the Location Update
example.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify the RTLS Tag Device example in the RTLS (FHIR) specification to
populate `Device.udiCarrier`. Concretely, add a `udiCarrier` element
with at least a representative `deviceIdentifier` (and ideally
`carrierHRF` or `carrierAIDC`, plus an `issuer` such as the GS1 OID),
treating the RTLS badge itself as if it were a UDI-bearing device for
illustrative purposes.

#### Justification

This literally fulfils what the ticket asked for and would make
`Device.udiCarrier` visible in the most widely-read RTLS Device
example, which is helpful for implementers who want to see the element
populated in context. It is also the simplest mechanical change to the
spec — a single example file edit with no schema or constraint impact.

### Disposition B: Alternative Approach (work group's drafted resolution for the companion ticket)

#### Proposal

Decline to add `udiCarrier` on the *RTLS Tag* Device example, but
satisfy the underlying need by creating (or augmenting, under
FHIR-40970) an *Enroll Tag* message example that shows an RTLS badge
being associated with a separate, UDI-bearing medical Device (e.g., an
infusion pump). The badge Device remains UDI-free; the associated
medical Device carries the `udiCarrier`. The new example would live
alongside the existing RTLS examples in the Real Time Location
Services (FHIR) specification.

#### Justification

This is the approach the Patient Administration work group has already
adopted on the companion ticket FHIR-40970 ("Persuasive with
Modification"). It addresses the implementer-visibility concern behind
the ticket while keeping the examples clinically truthful: an RTLS
badge is generally not an FDA/regulated medical device and would not
have a UDI, whereas the medical device it tracks does. Putting the
`udiCarrier` on the wrong Device would mis-teach implementers about
when UDI data legitimately appears in an RTLS exchange. It also keeps
the two tickets' resolutions consistent with each other.

### Disposition C: Decline (work group's current proposed resolution)

#### Proposal

Resolve **Not Persuasive**, on the rationale that the RTLS Tag Device
in this example represents an RTLS badge, which to the work group's
knowledge is not a regulated medical device and therefore does not
have a UDI. The UDI-on-RTLS-example need is satisfied via the
Enroll Tag example added under FHIR-40970; no edit to the RTLS Tag
Device example is required.

#### Justification

Adding `udiCarrier` to a Device that wouldn't have one in the real
world would be misleading and could be cited by implementers as
evidence that the spec expects RTLS badges to carry UDIs. The companion
ticket FHIR-40970 already gives the work group a place to show
`udiCarrier` correctly (on the *associated* medical device, in an
Enroll Tag example), so the implementer-education value the reporter
was after is preserved without compromising the accuracy of the RTLS
Tag Device example. The ticket is already labelled
`rtls-ready-for-block-vote` with this resolution drafted.

---

### Recommendation

**Recommended disposition:** C (with explicit cross-reference to B / FHIR-40970)

The work group has already drafted Resolution C and queued the ticket
for block vote alongside FHIR-40970, and the rationale stands up well:
the RTLS badge represented in the RTLS Tag Device example is not a
UDI-bearing device, and demonstrating `udiCarrier` on it would
mis-model real-world UDI semantics. The implementer-facing concern
that motivated the ticket — "show me UDI in an RTLS exchange" — is
already being satisfied under FHIR-40970 (Disposition B's approach) by
adding a UDI-bearing medical Device to the Enroll Tag example. The
recommendation to the work group is therefore to confirm the **Not
Persuasive** disposition for FHIR-40969 on block vote, while making
sure the resolution text explicitly points readers to the new Enroll
Tag example produced by FHIR-40970 so that anyone hitting this ticket
later can see how UDI is in fact represented in the RTLS spec.
