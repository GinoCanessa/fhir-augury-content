## Summary

`ChargeItemDefinition.instance` is currently typed as `0..* canonical(Medication | Substance | SubstanceDefinition | Device | DeviceDefinition | ActivityDefinition | PlanDefinition | HealthcareService)`. The reporter (Rik Smithies) observes that the element name `instance` is misleading because the type list mixes two very different categories of resource: real-world *instances* (Medication, Substance, Device, HealthcareService) and *definitions* (SubstanceDefinition, DeviceDefinition, ActivityDefinition, PlanDefinition). SubstanceDefinition was added recently — alongside Substance becoming an instance-only resource — for consistency, but the result is that one element semantically straddles two roles.

The ticket asks PA to either remove the definition resources from `instance`, or to clarify the intent of the element so that "instance" no longer refers to definitions. The resolution recorded on 2025-11-04 (Cooper Thompson) marks the request **Persuasive** with a concrete plan: split the property into two — `ChargeItemDefinition.instanceOf` for the definition resources and a tightened `ChargeItemDefinition.instance` for true instances — though the final names are still under discussion (candidates: `.appliesTo`, `.instanceOf`, `.definesBillingFor`).

There are no cross-referenced Jira tickets in Augury's index, but Zulip search surfaces prior community discussions of exactly this gap (notably 2021 and 2024 threads in `#implementers`) where implementers needed to relate `ChargeItemDefinition` to `HealthcareService`/services and ran into the same conflated type list.

## Details

> this list includes several resources that are specifically definitions rather than instances: Medication|Substance|
> SubstanceDefinition|Device|DeviceDefinition|ActivityDefinition|
> PlanDefinition|HealthcareService
>
> Needs a review or clarification. SubstanceDefinition was added recently, for consistency, because of the change to Substance resource (to become instance only) and because this element seems to cover both instances and definitions. But it probably should not cover both of those. DeviceDefinition, SubstanceDefinition, ActivityDefinition and
> PlanDefinition do not belong here as "instances".

**Resolution (Cooper Thompson, 2025-11-04):**

> - `.appliesTo`
> - `.instanceOf`
> - `.definesBillingFor`
>
> ---
>
> We will split the `ChargeItemDefinition.instance` property into two separate properties:
>
> - `ChargeItemDefinition.instanceOf` `0..*` `canonical(DeviceDefinition | ActivityDefinition | PlanDefinition | SubstanceDefinition)`
> - `ChargeItemDefinition.instance` `0..*` `canonical(Medication | Substance | Device | HealthcareService)`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution recorded on the ticket exactly: split `ChargeItemDefinition.instance` into two siblings.

- `ChargeItemDefinition.instanceOf` `0..*` `canonical(DeviceDefinition | ActivityDefinition | PlanDefinition | SubstanceDefinition)` — the *definition*-typed targets that are being removed from `.instance`.
- `ChargeItemDefinition.instance` `0..*` `canonical(Medication | Substance | Device | HealthcareService)` — true real-world instances (and `HealthcareService`, treated as the implementer-recognised instance-shaped service representation).

This is a breaking change to `ChargeItemDefinition`: any current use of `.instance` that points at `DeviceDefinition`, `ActivityDefinition`, `PlanDefinition`, or `SubstanceDefinition` must move to `.instanceOf`. Add a backwards-compatibility note and a maturity / changes-since-last-version entry on the resource page. Pick one of the candidate names already on the table (`.instanceOf`, `.appliesTo`, `.definesBillingFor`); the resolution text already lists `.instanceOf` as the leading candidate, so commit to that name.

#### Justification

This is what the workgroup already decided on the ticket (resolution = Persuasive, plan documented by Cooper Thompson 2025-11-04). It directly addresses the reporter's concern that one element conflates two semantically different cardinalities — definitions vs. real-world instances — and aligns the element name with the type list. It also matches Lloyd McKenzie's long-standing community guidance (2021 and 2024 Zulip threads above) that pricing-by-definition and pricing-by-instance are distinct linkages and benefit from being modelled separately. The breaking-change cost is bounded because all the current "definition" targets simply move to a clearly-named neighbour element.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a single `ChargeItemDefinition.instance` element but rename it (e.g., `.appliesTo` or `.subject`) and keep the union of all current target types. Add a `Slicing`-style invariant or narrative that documents which targets are "definitions" and which are "instances", but do not split the property. Provide a Forge/snapshot-friendly rename so that R5 → R6 conversion is mechanical (rename only).

#### Justification

This is less disruptive: one element stays, one rename, no new element to slot in. It addresses the surface complaint (the name `instance` is wrong) without forcing implementers to choose between two siblings. The downside is that it does *not* address the reporter's structural concern — the element still conflates two semantically different roles, and downstream profiles still cannot tighten `.appliesTo` to "only definitions" without slicing on the canonical's profile, which is fragile. It also leaves Lloyd's 2021 critique (definition-typed and instance-typed linkages should be modellable separately) unresolved.

---

### Disposition C: Decline

#### Proposal

Reject the change. Document on the resource page that `ChargeItemDefinition.instance` is intentionally a single union element covering both definitions and instances, with a note explaining that "instance" here is shorthand for "the thing being priced", regardless of whether that thing is itself an instance or a definition. Add an example for each target type. Do not change types or names in this cycle.

#### Justification

`ChargeItemDefinition` is in active use (R4 and R5 implementations exist, including the implementer scenarios discussed in the 2024 Zulip thread). Splitting `.instance` is a breaking change that will require every existing IG/profile that constrains `.instance` to be re-sliced and re-published, and every server with persisted ChargeItemDefinition resources to migrate data. If the workgroup judges that the conceptual cleanup is not worth that ecosystem cost in this cycle, declining and documenting the existing meaning is defensible. (The trade-off is that the misnomer persists and future implementers will keep tripping over the same Zulip-thread question.)

---

### Recommendation

**Recommended disposition:** A

The workgroup has already converged on the split (resolution = Persuasive on the ticket itself, plan written down by Cooper Thompson), and the Zulip evidence shows this is a recurring implementer pain point — both in 2021, when Rodrigo Sendin had to file FHIR-34621 to extend `.instance`, and in 2024, when Scott Carter ran into the same conflation. Disposition B keeps the bug just renamed; Disposition C accepts the misnomer and the recurring implementer questions for the sake of avoiding a one-time breaking change. Disposition A is the only option that resolves the underlying modelling problem the reporter raised. The remaining open decision — the *name* of the new element — should be settled in the disposition discussion; `instanceOf` is already the leading candidate in the resolution and matches FHIR's use of `instantiates*` / `*Of` patterns elsewhere, so the recommendation is to commit to `instanceOf` unless the workgroup surfaces a stronger contender.
