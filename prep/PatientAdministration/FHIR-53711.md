## Summary

`Endpoint.connectionType` today is a single `CodeableConcept`, which the
reporter argues is insufficient to convey what an endpoint actually supports.
Real deployments need to identify three things together: the **Implementation
Guide** an endpoint conforms to, the **actor** within that IG, and any
**named options/configuration** for that actor. Today implementers either
pre-coordinate all three into bespoke connection-type codes (as IHE-mCSD
does) or invent IG-specific extensions (as the FAST National Directory of
Healthcare Providers and Services (NDH) does). The reporter asks the spec
to provide a first-class way to express *(canonical or CodeableConcept,
version, actor, options)* on `Endpoint`.

The ticket has been triaged with a "Persuasive with Modification" resolution
proposing a new standard complex extension on `Endpoint` (or `Endpoint.payload`)
named `actor`, with parts `implementationGuide` (canonical | string, 1..1),
`version` (string, 0..1), `actor` (canonical | string | uri, 0..1), and
`options` (string, 0..*), plus optionally a generic placeholder code in the
example connectionType code system. The closely related NDH ticket
[FHIR-54130](https://jira.hl7.org/browse/FHIR-54130) ("ig-publication" naming
in the NDH `implementation-guide` extension) provides the IG-side experience
this work is generalising — it has already been Applied via
[HL7/fhir-us-ndh#9](https://github.com/HL7/fhir-us-ndh/pull/9) and adds an
`ig-usecase` part bound to a use-case value set.

## Details

**Description:**

Today Endpoint has one codeableConcept for the .connectionType. This element
is not sufficient to indicate an Implementation Guide + the Actor within that
implementation guide + the options within that implementation guide. Even for
non-IHE IGs, there are at least two actors (client vs server), but many IGs
have multiple Actors.

The solution today is either (a) define lots of codes representing the triplet
(see IHE-mCSD), or (b) define a new extension to carry the detail (see NDH).

Best would allow:

- canonical URI or CodeableConcept
- Version
- Actor
- Options

**Comments:**

- **egelkraut (2026-01-29)** — Captured at January WGM 2026, PA Thu Q2: asked
  whether `CapabilityStatement` would also provide the needed information,
  and noted that PA is currently leaning towards an extension rather than
  additional native properties; invited the reporter to discuss in upcoming
  PA calls.
- **john_moehrke (2026-01-29)** — `CapabilityStatement` only works for FHIR
  endpoints, while `Endpoint` is for endpoints including FHIR but beyond it
  (IHE and FAST NDH both hit this). Standard extensions are acceptable,
  preferable to one-off IG-specific extensions; if a standard extension is
  adopted the existing `connectionType` element should be removed because it
  is not sufficient to describe an endpoint. The need is mature enough to
  justify first-class elements.
- **esilver (2026-02-02)** — Broadly supportive of the proposed shape, but:
  (1) uncomfortable with a `connectionType` code that means "look in the
  extension" — would rather encode the IG in `connectionType` and put the
  rest in the extension, so consumers that don't understand the extension
  still get something useful; (2) suggests `options` should be coded
  (code/Coding array), and the wording corrected to "named options" since
  the cardinality is 0..*; (3) urges unification across the spec with
  Device/DeviceDefinition's `conformsTo` approach (cc Marti Velezis); (4)
  asks whether search params will be created.
- **lisarnelson (2026-02-02)** — The IG Publication extension also needs a
  coded "use case" concept so directory consumers can pick the right endpoint
  for the function they need. Cross-references
  [FHIR-54130](https://jira.hl7.org/browse/FHIR-54130) which clarifies the
  need for an `ig-use-case` component in the complex extension.
- **john_moehrke (2026-02-02)** — Unclear why a user can't follow the link to
  the IG to understand what the IG solves; expects the typical use to be
  "I know I want to communicate using IG ABC, actor 123 — which endpoints
  support that?".
- **brian.postlethwaite (2026-02-03)** — Reinforces that placing this in
  extension space (not normative) is deliberate so it can be worked through;
  the intent is exactly as costateixeira described — even if a consumer
  doesn't know the specific extensions they at least have a type they can
  show/find.
- **costateixeira (2026-02-03)** — Suggests the semantic is really
  "conformsTo(Actor)" / "implements(Actor)" and asks: (a) why pin to an
  ImplementationGuide (which itself contains many actors) rather than
  directly to an `ActorDefinition` canonical (which would already encode
  actor and version)? (b) `CapabilityStatement` of type `requirements` is
  used in his work to represent an actor; was it considered? Also notes that
  "options" is really configuration ("config A" vs "config B") and should be
  normalised, and questions repeating it on `Endpoint` if the endpoint can
  simply say "implements" or "conforms to".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new standard complex extension on the `Endpoint` resource — context
`Endpoint | Endpoint.payload` — named `actor` (URL e.g.
`http://hl7.org/fhir/StructureDefinition/endpoint-actor`) with the parts:

- `implementationGuide` : `canonical(ImplementationGuide) | string` — 1..1
- `version` : `string` — 0..1 (only used when `implementationGuide` is not a versioned canonical)
- `actor` : `canonical(ActorDefinition) | string | uri` — 0..1
- `options` : `string` — 0..* (named options/configurations defined by the IG/actor)

Repeatable on `Endpoint` (so an endpoint can advertise multiple IG/actor
conformances). Add a single placeholder code to the example-bound
`endpoint-connection-type` code system (e.g. `actor-extension`) signalling
that the connection details live in the `actor` extension. Provide at least
one IHE example (e.g. mCSD client) and one NDH example, and update the
Endpoint resource page narrative to reference the new extension. Leave
`Endpoint.connectionType` in place (do not break existing consumers).

#### Justification

Matches the reporter's request literally and is consistent with PA's stated
preference (egelkraut, brian.postlethwaite) to keep the new structure in
extension space where it can mature outside normative content. Generalising
the NDH/IHE pattern into a single, standard, HL7-published extension prevents
the proliferation of one-off IG extensions that john_moehrke flags. Keeping
`connectionType` and adding only an *example*-bound placeholder code preserves
backward compatibility and avoids the "look in the extension" anti-pattern in
required content (esilver's concern is partially mitigated because the code
remains optional and example-bound). Repeating cardinality on `Endpoint`
matches real deployments where one technical endpoint serves multiple
IG/actor pairs.

---

### Disposition B: Alternative Approach — `conformsTo`-style design unified across resources

#### Proposal

Instead of a single complex extension wrapping the IG, model conformance the
way `Device.conformsTo` / `DeviceDefinition.conformsTo` already do, and apply
the same idiom to `Endpoint`. Concretely:

- Add a standard extension `endpoint-conformsTo` on `Endpoint` (and
  `Endpoint.payload`), repeatable, with parts:
  - `system` : `CodeableConcept` (extensible to a system-of-conformance
    value set, e.g. IHE, IG, profile)
  - `specification` : `canonical(ImplementationGuide | ActorDefinition |
    CapabilityStatement) | uri | string` — 1..1
  - `version` : `string` — 0..1
  - `option` : `Coding` — 0..* (coded, not free string, per esilver)
- Drop a separate `actor` part where `specification` already points to an
  `ActorDefinition` canonical (which encodes actor identity and version),
  per costateixeira's question. Keep an explicit `actor` slot only when the
  specification target is an `ImplementationGuide` and the actor is identified
  by name.
- Define a corresponding search parameter
  (`Endpoint.extension('endpoint-conformsTo').specification` →
  `_conformsTo`) so directory queries like
  "give me endpoints that conform to ActorDefinition X" become first-class.
- Do **not** add any placeholder code to the connectionType code system; let
  consumers that don't understand the extension fall back to
  `connectionType` as today.

#### Justification

Directly addresses costateixeira's and esilver's points: it (a) reuses the
existing `conformsTo` pattern already in the spec (avoiding a Patient
Administration-specific idiom for the same concept), (b) lets implementers
point straight at an `ActorDefinition` canonical when one exists (collapsing
IG+actor+version into a single canonical), (c) makes `option` coded so it
can be exchanged interoperably rather than as opaque strings, and (d) avoids
the awkward "this code means look elsewhere" semantics in `connectionType`.
A search parameter is what actually makes this useful in a directory
context, which lisarnelson's use-case argument depends on. The trade-off vs
A is more cross-WG coordination (Orders & Observations / Devices for
`conformsTo` parity) and a slightly more abstract model that NDH would have
to migrate FHIR-54130 onto.

---

### Disposition C: Decline

#### Proposal

Decline the request as scoped. Rationale: extensions for this purpose
already exist in the IGs that need them (NDH `implementation-guide` plus
FHIR-54130's `ig-usecase` refinement; IHE-mCSD's pre-coordinated
`connectionType` codes), and `CapabilityStatement` (with `kind=requirements`
or `kind=instance`) plus `ActorDefinition` already cover FHIR-based
endpoints. Direct the reporter and FAST NDH to continue using IG-local
extensions, and ask IHE to publish their pre-coordinated codes in a shared
code system if cross-IG harmonisation is desired.

#### Justification

`Endpoint.connectionType` is example-bound, so any consumer is already
expected to handle codes it doesn't recognise. Adding a normative-track
core extension imposes maintenance burden across the spec and risks two
overlapping mechanisms (the new `actor` extension *and* existing IG
extensions like NDH's), which is exactly the situation john_moehrke wants
to avoid. It would also be defensible to argue the actual problem is
inadequate publication of `ActorDefinition` canonicals by IGs, not a gap
in `Endpoint`.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach)

Disposition A is what the ticket literally asks for and is the path of least
resistance, but it bakes in three problems the comment thread already raised:
the `connectionType` placeholder code is awkward (esilver), `options` as
free strings is not interoperable (esilver), and modelling IG + actor as
two separate parts duplicates information that an `ActorDefinition`
canonical already encodes (costateixeira). Disposition C ignores the
reporter's substantiated point — confirmed by both IHE and FAST NDH
experience — that *something* standard is needed; declining will simply
push more IGs into inventing their own incompatible extensions, which is
the harm the ticket is trying to prevent. Disposition B keeps the win of A
(a single HL7-published extension on `Endpoint`) while harmonising with
the existing `conformsTo` pattern on `Device`/`DeviceDefinition`, making
`option` coded, and adding the search parameter that directory use cases
actually need. The cost is a short cross-WG conversation with O&O / Devices
and a migration story for NDH (FHIR-54130) — both manageable, and worth
doing once at extension maturity rather than after the pattern has shipped.
