## Summary

The reporter requests that the cardinality of `InsurancePlan.ownedBy` and
`InsurancePlan.administeredBy` in the **US National Directory (NDH)** profile of
`InsurancePlan` be relaxed from `1..1` to `0..1`. The motivation is that the NDH
`InsurancePlan` profile is increasingly being adopted as a basis for directories
beyond NDH itself — including source directories that feed the NDH and adjacent
use cases such as **Formulary**. In several of those scenarios, the directory
publisher *is* the owning/administering organization for every plan in the
directory, so requiring those references on each `InsurancePlan` instance is
redundant data; in others (notably Formulary), the `Organization` resource is
not otherwise used, so making `ownedBy`/`administeredBy` mandatory forces
authors to materialise an `Organization` purely to satisfy cardinality.

No companion Jira tickets, Zulip threads, or GitHub items were surfaced through
cross-references for this ticket, so the analysis below is grounded in the
ticket text and the NDH `InsurancePlan` profile context.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **US National Directory (NDH)** IG (`HL7/fhir-us-ndh`), relax the
cardinality of both `InsurancePlan.ownedBy` and `InsurancePlan.administeredBy`
on the NDH `InsurancePlan` profile from `1..1` to `0..1`. Concretely:

- Update the FSH for the `ndh-InsurancePlan` (a.k.a. NDH `InsurancePlan`)
  profile under `input/fsh/` so that the two element rules read
  `* ownedBy 0..1` and `* administeredBy 0..1` (instead of `1..1`).
- Keep both elements as **Must Support** so that producers that *do* know the
  owning/administering organization continue to be required to populate them.
- Update the example(s) (e.g. `input/fsh/Example-InsurancePlan.fsh`) only if
  needed to demonstrate the optional case; existing examples that populate
  both references remain valid.
- Add narrative on the `InsurancePlan` page explaining when these references
  may be omitted (source directories where the publisher is unambiguously the
  owner/administrator; directories such as Formulary that do not otherwise use
  `Organization`) and when they must still be supplied (aggregated/
  multi-publisher directories).
- File a tracking note for downstream IGs that derive from or constrain the
  NDH `InsurancePlan` profile (e.g. Da Vinci PDex Plan-Net `plannet-
  InsurancePlan`, Formulary) so they can decide whether to re-tighten to
  `1..1` in their own profiles.

#### Justification

Cardinality changes from `1..1` to `0..1` are a backward-compatible relaxation
in FHIR profile semantics: every existing instance that conforms today (which
must have `ownedBy` and `administeredBy` populated) continues to conform after
the change. The change directly addresses the reporter's two concrete pain
points — source directories where the references are tautological, and
Formulary-style usage where `Organization` is otherwise unused. Keeping the
elements **Must Support** preserves the original intent for implementers that
have the data, while the narrative guidance prevents the looser cardinality
from being read as "owner/administrator are unimportant". Downstream IGs that
need stricter behaviour (e.g. consumer-facing payer directories) retain the
ability to re-constrain to `1..1` in their own derived profiles, so use cases
that depend on the stronger guarantee are not lost.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the NDH `InsurancePlan` profile's `ownedBy` and `administeredBy` at
`1..1`, but introduce an explicit **profile-layering strategy** in the NDH IG
to handle the source-directory and Formulary use cases:

- Define (or document) an **abstract / base** NDH `InsurancePlan` profile
  with `ownedBy 0..1` and `administeredBy 0..1` and **Must Support**.
- Re-derive the existing **published** NDH `InsurancePlan` profile from that
  base with `ownedBy 1..1` and `administeredBy 1..1`, preserving today's
  conformance contract for primary NDH usage.
- Recommend that source-directory publishers and Formulary authors target the
  looser base profile (or their own derivation of it), and document the
  organization-identification expectation via a directory-level
  `Organization` (or `Bundle.entry[0]` publisher) instead of per-`InsurancePlan`
  references.
- Optionally define a `InsurancePlan` extension or invariant that captures
  "publisher is owner/administrator for all plans in this directory" so the
  information is not lost when the references are omitted.

#### Justification

This avoids any change to the conformance contract that current NDH
implementers have already built against, which is important given the IG's
position as the basis for federal-scale provider/plan directory work. The
trade-off is significantly higher IG complexity (two profiles instead of one,
plus an optional extension/invariant) and a heavier authoring burden on
source-directory and Formulary implementers, who must now choose the right
profile and convey the publisher-as-owner relationship through a separate
mechanism. It is the more defensible path **only** if downstream consumers of
NDH instances are known to rely on the `1..1` guarantee in ways that cannot
tolerate optional references — evidence of which is not present in this
ticket or its (empty) cross-references.

---

### Disposition C: Decline

#### Proposal

Decline the request. Keep `InsurancePlan.ownedBy 1..1` and
`InsurancePlan.administeredBy 1..1` on the NDH `InsurancePlan` profile, on
the grounds that the owning and administering organizations are core to the
NDH directory model and that the alternate use cases cited (source
directories, Formulary) should either constrain or replace the NDH
`InsurancePlan` profile in their own IGs rather than weaken the NDH
contract for everyone.

#### Justification

The NDH `InsurancePlan` profile is intentionally tight to make consumer
queries deterministic: "which organization owns / administers this plan?"
should be answerable from a single resource without follow-on lookups or
out-of-band knowledge of the directory publisher. Other IGs (Da Vinci
Plan-Net, Formulary) routinely **further constrain** base/NDH profiles
rather than push relaxations upstream. If Formulary specifically does not
need `Organization`, that is a Formulary-profile decision and can be
expressed by Formulary defining its own `InsurancePlan` profile (or by NDH
publishing a sibling abstract profile, per Disposition B) — without
loosening NDH's primary deliverable.

---

### Recommendation

**Recommended disposition:** A

The change is a low-risk, backward-compatible relaxation that resolves a real
authoring pain point without removing any guarantees that today's NDH
producers actually provide (they will continue to populate the references
because Must Support is retained, and existing instances remain conformant).
The "source directory" and Formulary cases the reporter cites are exactly the
kinds of adjacent uses NDH is being positioned to support, and forcing them
to either invent dummy `Organization` resources or fork the profile is
disproportionate to the benefit of a `1..1` floor that is, in those
scenarios, tautological. Disposition B preserves more strictness but at a
real cost in IG complexity and implementer overhead, and the ticket surfaces
no consumer-side requirement that would justify that complexity. Disposition
C is the weakest option here because the reporter has identified concrete
in-scope use cases that the current cardinality actively impedes; declining
without offering an alternative path would push those use cases out of NDH
alignment entirely. The work group should still confirm with consumers of
NDH `InsurancePlan` data that none of them rely on the `1..1` guarantee
beyond Must Support — if such a consumer exists, fall back to Disposition B.
