## Summary

Linda Michaelsen reports that `Account.balance.aggregate` currently lets an
implementer flag a balance line as `patient | insurance | total`, but does
not let the implementer say *which* insurance the line refers to. In real
patient encounters more than one Coverage is frequently in force (e.g.,
primary commercial + Medicare secondary), and each payer carries its own
expected portion of the balance. Without a way to bind a balance line to a
specific Coverage, downstream systems cannot tell which insurer is on the
hook for which amount. The reporter proposes adding an element on
`Account.balance` that references the `Coverage` resource so balance lines
can be attributed to a specific payer.

This ticket is the direct sequel to [FHIR-20972](https://jira.hl7.org/browse/FHIR-20972)
(Persuasive with Modification, Published), which introduced the
`Account.balance` backbone with the `aggregate / term / estimate / amount /
calculatedAt` shape that this ticket now wants to extend. It is also paired
with [FHIR-53831](https://jira.hl7.org/browse/FHIR-53831), filed the same
day by the same reporter, which complains that the *name* "aggregate" is
confusing for the responsible-party concept and suggests `responsibleParty`
— so PA may want to consider both tickets together when reshaping
`Account.balance`. Related discussions on Coverage semantics
([FHIR-11367](https://jira.hl7.org/browse/FHIR-11367),
[FHIR-43028](https://jira.hl7.org/browse/FHIR-43028),
[FHIR-53993](https://jira.hl7.org/browse/FHIR-53993)) reinforce that
"insurance" alone is not a sufficient identifier when multiple coverages
apply.

## Details

**Description:**

> In Balance.aggregate, you can specify "insurance" but not which insurance.
> Often times, more than one is involved and each would have their own

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an optional `Reference(Coverage)` element to the `Account.balance`
backbone in FHIR Core so each balance line can be attributed to a specific
payer. Concrete shape:

- `Account.balance.coverage` 0..1 `Reference(Coverage)` — *"The specific
  Coverage to which this balance line applies. Only meaningful when
  `aggregate` indicates an insurance / payer responsibility."*

Keep `Account.balance.aggregate` as-is (CodeableConcept,
`patient | insurance | total`) so existing instances remain valid; the new
element only refines the `insurance` case. Cardinality stays `0..1`
because (a) a single balance line is, by definition, one party's expected
share, and (b) `Account.balance` itself is `0..*`, so multiple insurers
are represented by repeating the backbone — one entry per coverage.

Add an invariant on `Account.balance`:

> `acc-bal-1`: If `coverage` is present, `aggregate` SHALL include a code
> from the "insurance" branch of the value set
> (e.g., `Account-aggregate#insurance`).

Update the `Account` page narrative and the existing `aggregate` element
description to reference the new element.

#### Justification

The change is additive and optional, so it preserves backward
compatibility with the FHIR-20972 design that just shipped. It uses an
already-defined linkage (`Account.coverage` already references the same
`Coverage` resource on the parent Account, so implementers know how to
populate it) and matches the long-standing PA position from FHIR-11367
that multiple Coverages may each contribute to an Account. The
multi-payer scenarios documented in FHIR-43028's discussion (Cooper
Thompson's Medicare + Black Lung + commercial example) are exactly the
ones today's `aggregate` cannot disambiguate. Tying the reference to a
specific Coverage — rather than, say, a free-text payer name — keeps the
data structured and machine-actionable for COB workflows.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than add a typed `Reference(Coverage)` to the base spec,
**generalise `Account.balance.aggregate` from a CodeableConcept to a
choice / pair**: keep the coded responsibility category, and add a sibling
*reference* slot that the implementer fills in when the responsible party
is a known FHIR resource. Concretely:

- Rename / re-cut as part of dispositioning **FHIR-53831 together with
  FHIR-53832**:
  - `Account.balance.responsibleParty` 0..1 `CodeableConcept` (renames
    `aggregate`; same value set `patient | insurance | total`).
  - `Account.balance.responsiblePartyDetail` 0..1
    `Reference(Coverage | Patient | RelatedPerson | Organization)` — the
    specific party (Coverage when `responsibleParty = insurance`; Patient
    or RelatedPerson when `= patient`; absent when `= total`).
- Mark `aggregate` as renamed in the change log; provide a transform note
  for FHIR-20972 implementers.
- Add an invariant aligning the chosen reference type with the
  responsibleParty code.

#### Justification

This addresses both of Linda Michaelsen's same-day tickets in one pass
and avoids a second breaking-ish edit to `Account.balance` six months
later. It also fixes the "patient" half of the same problem — when a
balance is owed by the patient, today there is no way to point at the
specific guarantor (`Patient` or `RelatedPerson`) when more than one
person is on the Account. The cost is real: it churns an element name
that just stabilised in FHIR-20972, and it forces implementers who only
care about the Coverage case to think about a polymorphic reference. PA
should weigh that against the savings of dispositioning FHIR-53831 and
FHIR-53832 together.

---

### Disposition C: Decline

#### Proposal

Decline the change and instead document an existing-mechanism resolution:
implementers who need per-Coverage balance attribution should either (a)
publish multiple `Account.balance` entries — one per payer — and rely on
contextual ordering against `Account.coverage[*].coverage` /
`Account.coverage.priority`, or (b) carry the per-Coverage breakdown on
`ExplanationOfBenefit` / `ClaimResponse` from Financial Management, where
payer-specific allocation is the native concern. Add a note to the
`Account.balance` narrative pointing readers at those resources.

#### Justification

`Account.balance` is documented as a volatile, snapshot-style summary
(per the FHIR-20972 PA/FM discussion of September 2019). Per-payer
financial detail is the explicit job of the FM `ExplanationOfBenefit`,
`ClaimResponse`, and `PaymentReconciliation` resources, and duplicating
that linkage on `Account` invites inconsistency between the snapshot and
the authoritative billing artefacts. Declining keeps `Account` lean and
preserves the FHIR-11367 description that Coverages "may contribute" —
without dictating *how much*. The cost is that simple Account-only
consumers (the audience FHIR-20972 was written for) still cannot answer
"which insurer owes this $250?" from `Account` alone.

---

### Recommendation

**Recommended disposition:** B

The reporter has filed two tickets against `Account.balance.aggregate` on
the same day (FHIR-53832 and FHIR-53831) and a related FM ticket on
self-pay attribution (FHIR-53993) is in the same neighbourhood. Taking
the change in two surgical edits — first add `coverage`, then later
rename `aggregate` — guarantees PA will revisit the same backbone twice
and produce a less coherent shape than if it solved the naming and
attribution problems together. Disposition B also rounds out the
"patient" half of the responsibility model, which Disposition A leaves
under-specified, and it keeps the spec aligned with the long-standing PA
position from FHIR-11367 that multiple parties contribute to an Account.
The cost is one churn on a recently-stabilised element; that is worth
paying once, in a single ballot, rather than scheduling a second change
later. If PA judges the renaming risk to FHIR-20972 implementers as too
high, fall back to Disposition A and explicitly defer FHIR-53831.
