## Summary

This ticket asks the Patient Administration work group (as stewards of the
Interoperable Digital Identity and Patient Matching IG) to clarify the
specification of the **B2B with User Authorization Extension Object**
(`hl7-b2b-user`) relative to the existing **B2B Authorization Extension
Object** (`hl7-b2b`) defined in UDAP Security. The reporter raises three
concrete ambiguities: (1) which "requirements for use of `hl7-b2b`" carry
over to `hl7-b2b-user` (he suspects only those in §5.2.1.2 on identifier/code
formats), (2) what "as per UDAP Security 5.2.1.1" actually means when the
text says the object is included in the JWT extensions object as per that
section (analogously? in addition? instead?), and (3) whether `hl7-b2b-user`
is intended to be sent **in addition to** or **instead of** `hl7-b2b`,
noting that UDAP Security 5.2.1 currently mandates `hl7-b2b` for B2B clients
using `client_credentials` and that the Identity IG never makes the new
extension explicitly mandatory in any case. A separate question raised in
the comments is why the extension is documented on the `$match` page when
it is not specific to the patient match operation.

The reporter and the WG eventually arrived at working assumptions
(§5.2.1.2 applies, "as per" means "analogously", `hl7-b2b-user` replaces
`hl7-b2b` for the consumer-match flow) and the broader "how can other IGs
extend B2B?" question was split off into the related Security ticket
[FHIR-51161](https://jira.hl7.org/browse/FHIR-51161), which has since been
applied to the FAST Security IG. The current ticket is presently marked
**Resolution: Not Persuasive** with the resolution description "Discussed
and incorporated as part of other changes."

## Details

**Description:**
Comparing the B2B with User Authorization Extension Object to the B2B
Authorization Extension Object, the following need to be addressed:

- "The same requirements for use of `hl7-b2b` apply in the use of
  `hl7-b2b-user`." What does this mean? Which requirements specifically?
  I'm guessing those in section 5.2.1.2; any others?
- It says "includes this object in the extensions object of the
  Authentication JSON Web Token (JWT), as per UDAP Security 5.2.1.1". What
  does "as per" imply here? Analogously? Similarly to? It can't mean
  "follow 5.2.1.1 as well", can it?
- Is this intended to be included in addition to or instead of `hl7-b2b`?
  Given the overlap of fields, it appears the latter. But:
  - 5.2.1 requires `hl7-b2b` for B2B client apps using the
    `client_credentials` flow. This will need to be rewritten to allow
    for either.
  - There is no equivalent requirement in the Identity IG saying it
    SHALL be included in &lt;specific cases&gt;.

**Comments:**

- **jlamy (2023-12-04):** Asks why this extension is on the patient
  matching page given it has nothing to do with `$match`.
- **juliemaas (2024-03-28):** Explains that this is for person matching
  when a B2B use case has a user present and the requestor wants to
  provide details about the user — i.e., not an authorization-code-flow
  transaction. Asks whether the reporter is suggesting more explicit
  requirements about authorized representatives stored by responders.
- **juliemaas (2024-06-18):** Notes that the page is not unique to the
  FHIR `$match` operation but to person matching more generally.
- **jlamy (2025-06-19, comment 54589):** Now that Consumer Match has been
  separated as matching for the purpose of authentication, that part is
  clearer. Records working assumptions for testing: §5.2.1.2 is what
  applies, "as per" means analogously, and `hl7-b2b-user` is included
  **instead of** `hl7-b2b`. Commits to opening a ticket against the FAST
  Security IG to permit a different extension.
- **jlamy (2025-06-19, comment 54588):** Cross-links the new Security
  ticket [FHIR-51161](https://jira.hl7.org/browse/FHIR-51161) ("Clarify
  how B2B may be extended by other IGs").
- **juliemaas (2025-06-19):** Counters that there are use cases where
  **both** extensions could be needed (e.g., Facilitated FHIR for IAS, or
  other JWT-Based Auth B2B claiming to also have a user present but not
  "on the wire" as it would be in JWT-Based Auth B2C).
- **jlamy (2025-06-20):** Proposes a rubric: if only `hl7-b2b-user` is
  present, authenticate the user; if `hl7-b2b` is present, authenticate
  the user/org and treat any accompanying `hl7-b2b-user` as optional
  scope-narrowing context that may also be ignored.
- **juliemaas (2025-06-20):** Agrees this is a reasonable basis for test
  cases absent other context (purpose-of-use, trust-validation results),
  but flags a policy concern: telling implementers they may simply ignore
  `hl7-b2b-user` is worrisome, since including that information signals
  data may flow back to an individual and the IG would not recommend
  ignoring that user identity in policy decisions or audit logs. Asks
  whether existing text suggests this.

### [FHIR-51161: Clarify how B2B may be extended by other IGs](https://jira.hl7.org/browse/FHIR-51161)

A companion ticket (Work Group: **Security**, Specification: US UDAP
Security, **Resolution: Persuasive**, **Status: Published**) opened by the
same reporter on 2025-06-19 to ask UDAP Security to add language to §5.2.1
making clear that B2B can be extended (e.g., by providing an alternate B2B
Authorization Extension Object such as `hl7-b2b-user`). The applied
resolution text is: *"Other IG's MAY define additional AEO's in addition to
the HL7-B2B-object."* This ticket directly enables — and partially
discharges — the underlying spec gap that FHIR-43284 was originally
written against, by establishing in UDAP Security that an alternate AEO
like `hl7-b2b-user` is permitted at all.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the Identity IG's description of the B2B with User Authorization
Extension Object to remove the three ambiguities. Concretely, on the
Consumer Match / `hl7-b2b-user` description page in the Interoperable
Digital Identity and Patient Matching IG (the page that currently lives
under `patient-matching.html#consumer-match`):

1. Replace "The same requirements for use of `hl7-b2b` apply in the use
   of `hl7-b2b-user`." with an explicit cross-reference: "The
   identifier/code formatting requirements in **UDAP Security §5.2.1.2**
   apply to `hl7-b2b-user` in the same way they apply to `hl7-b2b`. No
   other §5.2.1.x requirements are inherited."
2. Replace "as per UDAP Security 5.2.1.1" with "**analogously to** UDAP
   Security §5.2.1.1" and add a one-sentence note clarifying that the
   client follows the *placement* (extensions object of the Authentication
   JWT) described in §5.2.1.1 but does **not** also need to send
   `hl7-b2b` in that JWT when sending `hl7-b2b-user`.
3. State explicitly when each AEO is used. Adopt the rubric the reporter
   and Julie Maas converged on in the June 2025 comments, with the
   policy concern Julie raised baked in:
   - When the Consumer Match flow is used, `hl7-b2b-user` is sent
     **instead of** `hl7-b2b`.
   - When a B2B client_credentials flow is used and a user identity is
     **also** carried (Facilitated FHIR for IAS, JWT-Based Auth B2B with
     a user present), both `hl7-b2b` and `hl7-b2b-user` MAY be sent;
     when both are present, responders SHALL NOT silently discard
     `hl7-b2b-user` from policy decisions and audit logs (Julie's
     concern), even if it is treated only as scope-narrowing context.
4. Add a forward link to FHIR-51161's resolution text in UDAP Security
   ("Other IGs MAY define additional AEOs in addition to the
   HL7-B2B-object") so that implementers see the upstream permission
   that authorizes `hl7-b2b-user` to exist.
5. Address the "why is this on the `$match` page?" concern by either
   moving the `hl7-b2b-user` description to its own page under the
   Consumer Match section, or by adding a one-line preamble noting that
   the extension is about person-matching authentication and not about
   the FHIR `$match` operation per se.

#### Justification

This is what the ticket literally asks for: each of the three description
ambiguities is closed, and the page-placement concern raised in the
December 2023 comment is addressed. The proposal is consistent with the
direction the WG already took on the related Security ticket FHIR-51161
(now Published, Persuasive), which establishes that alternate AEOs are
permitted; FHIR-43284 is the natural downstream cleanup in the Identity
IG. Importantly, item (3) preserves Julie Maas's policy point that
`hl7-b2b-user` should not be ignorable when present, which the reporter's
original "may also be ignored" rubric did not.

---

### Disposition B: Alternative Approach

#### Proposal

Mark FHIR-43284 as **Persuasive with Modification** but discharge it
with a *narrower* edit than Disposition A: only fix items (1) and (2)
(the textual ambiguities about which §5.2.1.x requirements apply and
what "as per" means), and defer the larger "in addition to vs. instead
of" question — together with the policy guidance about not ignoring
`hl7-b2b-user` — to a follow-up ticket co-owned by Patient
Administration and Security. That follow-up would land on the FAST
Security IG (where the AEO mechanism lives) and the Identity IG would
simply point at it.

In code/spec terms this means a single editorial PR against the Identity
IG's Consumer Match page making the two clarifications, plus opening a
new tracker ticket linked to both FHIR-43284 and FHIR-51161 for the
combined-AEO behavior.

#### Justification

The "in addition to vs. instead of" question is genuinely cross-WG: it
constrains UDAP Security's §5.2.1 mandate as much as it constrains the
Identity IG, and it touches at least two real flows (Consumer Match;
Facilitated FHIR / IAS with a B2B+user combination). Resolving it inside
the Identity IG alone risks contradicting whatever Security publishes
next on top of FHIR-51161. This option still gets the cheap, clearly
correct clarifications into the Identity IG immediately, while letting
the harder normative question be answered once, in one place, with both
WGs co-signing.

---

### Disposition C: Decline

#### Proposal

Confirm the existing **Not Persuasive** resolution, with the resolution
description kept as "Discussed and incorporated as part of other
changes." The position is that the substantive concerns raised in
FHIR-43284 have already been absorbed:

- The "how may B2B be extended?" question was answered upstream in
  FHIR-51161 (UDAP Security, Resolution: Persuasive, Status: Published).
- The "is this on the right page?" concern was answered in 2024 by Julie
  Maas's comments noting that the page is about person matching
  generally, not about the `$match` operation, and by the subsequent
  separation of Consumer Match.
- The remaining ambiguities are minor enough to be handled as ordinary
  editorial fixes during the next Identity IG publication cycle without
  needing a tracker.

No spec change is opened off this ticket.

#### Justification

This matches the present state of the ticket. It is defensible if the WG
believes that (a) FHIR-51161 plus the existing Consumer Match
restructuring already cover the substantive issue, and (b) the remaining
wording cleanups are too small to deserve a tracked disposition. The
risk is that the original three ambiguities still appear in the
published Identity IG text, so implementers who read the page in
isolation will not see the WG's clarifications captured in the Jira
discussion — that context only lives in this ticket and in FHIR-51161.

---

### Recommendation

**Recommended disposition:** A

The discussion on this ticket is mature: the reporter and Julie Maas
worked through all three ambiguities in June 2025 and arrived at a
workable rubric, and the upstream "may a different AEO exist at all?"
question has already been published as Persuasive in
[FHIR-51161](https://jira.hl7.org/browse/FHIR-51161). What is missing is
the corresponding text change in the Identity IG itself — the page
implementers will actually read still contains the original ambiguous
"as per UDAP Security 5.2.1.1" / "same requirements apply" / unstated
"in addition to vs. instead of" wording, which is exactly what
FHIR-43284 was opened to fix. Disposition A captures the converged
consensus (including Julie's correction that `hl7-b2b-user` SHOULD NOT
be silently ignored when both AEOs are present) and locks it into the
spec rather than leaving it as Jira folklore. Disposition B is a
reasonable fallback if the WG wants Security co-sign on the
combined-AEO behavior before publishing it; Disposition C leaves the
implementer-visible text unchanged and is hard to defend now that
FHIR-51161 has landed.
