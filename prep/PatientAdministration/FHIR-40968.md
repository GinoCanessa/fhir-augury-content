## Summary

The ticket asks Patient Administration (Real Time Location Services FHIR IG
authors) to consider using the `Device.udiCarrier` element in the RTLS Tag
Device profile. The reporter's one-line suggestion is that the RTLS Tag
Device profile should surface a UDI (Unique Device Identification) carrier,
presumably so that the tag itself — or the device it is affixed to — can be
identified against its UDI-DI/UDI-PI record. The ticket is linked thematically
to [FHIR-40970](https://jira.hl7.org/browse/FHIR-40970) ("Add udiCarrier in
the Location Update Event example"), which was filed by the same reporter on
the same day and resolved Persuasive with Modification. The WG's draft
resolution on FHIR-40968 distinguishes the two cases: the RTLS badge itself
is not regarded as a medical device and therefore would not carry a UDI,
whereas a medical device to which an RTLS badge is affixed (e.g., an infusion
pump) would, and that latter case is being addressed via FHIR-40970 by
updating the Enroll Tag example rather than by changing the RTLS Tag Device
profile.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the RTLS Tag Device profile in the Real Time Location Services IG to
surface `Device.udiCarrier` — either by setting `mustSupport = true` on
`Device.udiCarrier` (and typically on `udiCarrier.deviceIdentifier`,
`udiCarrier.issuer`, and `udiCarrier.carrierHRF`/`carrierAIDC`) or by adding
a `0..*` slice permitting a UDI to be conveyed with the tag instance. This
would allow an RTLS tag, where a regulatory or vendor-supplied UDI exists,
to carry that identifier directly on the profile used to represent the
badge.

#### Justification

This literally fulfils the reporter's request and keeps UDI modeling
consistent across Device-typed resources in FHIR. If any class of RTLS tag
hardware is ever classified as a medical device under a UDI regime (or if an
implementer chooses to assign a non-regulatory UDI-style identifier via a
private issuer), mustSupport/slice guidance would give them a canonical
place to put it rather than forcing an extension or identifier overload.
`Device.udiCarrier` is already 0..* in the base resource, so no breaking
change is introduced — only guidance/mustSupport is strengthened.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the RTLS Tag Device profile, but add non-normative guidance
(and an example) to the IG narrative explaining the two-device pattern: the
RTLS badge Device (no UDI), optionally related via `Device.parent` or an
association message to a second Device representing the medical device
that is being tracked, and the UDI lives on that second Device's
`udiCarrier`. Point implementers from the RTLS Tag Device profile page to
the Enroll Tag example being built under FHIR-40970 so they see where UDI
belongs.

#### Justification

This captures the reporter's underlying concern (that UDI needs a home in
the IG) without overloading the RTLS Tag Device profile with an element that
doesn't semantically apply to the badge hardware. Keeping `udiCarrier` on
the medical-device Device preserves the FHIR design intent that
`Device.udiCarrier` identifies *that* device under UDI regulation — not a
tag attached to it. It also reuses the Enroll Tag example investment
already committed to in FHIR-40970, so the IG gains traceable UDI coverage
without a second profile change.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Resolve as Not Persuasive with the language drafted in the ticket: the RTLS
Tag Device profile represents the RTLS badge itself, which is not a medical
device subject to UDI, so `udiCarrier` is not applicable on that profile.
UDI needs for the device the badge is affixed to are addressed separately
via the Enroll Tag example work tracked under
[FHIR-40970](https://jira.hl7.org/browse/FHIR-40970).

#### Justification

The request is predicated on the assumption that an RTLS badge would
itself be UDI-bearing, which is not generally true: UDI is a regulatory
identifier for medical devices (21 CFR 830 in the US and equivalent
regimes elsewhere), and RTLS badges are typically not regulated as medical
devices. Promoting `udiCarrier` on a profile whose instances will almost
never carry a UDI encourages implementers to either leave the element
empty or misuse it for non-UDI identifiers, weakening the meaning of
`udiCarrier` across FHIR. The companion ticket FHIR-40970 already creates
the right place for UDI — the Enroll Tag example that links the badge to
the tracked medical device — so the underlying need is met without
changing the Tag Device profile.

---

### Recommendation

**Recommended disposition:** C

The WG's drafted resolution (Not Persuasive) is the right answer and should
be confirmed. `Device.udiCarrier` has a specific regulatory meaning that
matches the medical device being tracked, not the RTLS badge doing the
tracking; surfacing it on the Tag Device profile would either mislead
implementers or leave them with an always-empty mustSupport element. The
substantive concern — that UDI must be representable somewhere in the RTLS
flows — is already being addressed under
[FHIR-40970](https://jira.hl7.org/browse/FHIR-40970) by adding an Enroll Tag
example that associates the badge with a medical device whose `udiCarrier`
is populated, which is the semantically correct home for the UDI. If the
workgroup wants to soften the message, Disposition B (keep the profile as
is, but add narrative and cross-link to the FHIR-40970 example) is a
reasonable belt-and-braces variation; Disposition A should be avoided
because it overloads `udiCarrier` onto a non-UDI-bearing profile.
