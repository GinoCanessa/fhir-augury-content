## Summary

During US Realm Connectathon testing in Touchstone, the NDH example
`HealthcareService/HartfordOrthopedicServices` failed terminology validation on
its `payment-accepted` extension. The Terminology Engine reported that the
`CASH` code from `http://terminology.hl7.org/CodeSystem/v2-0570` is not in
`http://terminology.hl7.org/ValueSet/v2-0570|2.0.0`, even though that value set
is required by the binding. Investigation showed the value set the NDH IG
binds to is anchored to THO **6.3** (which publishes `v2-0570` value set
version `2.0.0`), while the IG's declared dependency is on THO **7.0.1**
(which publishes `v2-0570` value set version `3.0.0`). The reporter is asking
the Patient Administration work group to align the NDH `payment-accepted`
extension's value-set binding with the THO version the IG actually depends
on, so that valid HL7 v2 Table 0570 codes such as `CASH` validate successfully.
No related Jira tickets, Zulip threads, or GitHub items were surfaced by
cross-reference or content search.

## Details

**Description:**

During USRealm Connectathon testing the example for
HealthcareService/HartfordOrthopedicServices in Touchstone we found the

```
ERROR: (TerminologyEngine) None of the codings provided are in the value set
'hl7VS-paymentMethodCode' (http://terminology.hl7.org/ValueSet/v2-0570|2.0.0),
and a coding from this value set is required) (codes =
http://terminology.hl7.org/CodeSystem/v2-0570#CASH).
Location: HealthcareService.extension[1].value.ofType(CodeableConcept)
(line 35, col 6).
```

After further investigation the CodeSystem source is anchored to THO 6.3, but
the IG claims a dependency on THO 7.0.1
(https://build.fhir.org/ig/HL7/fhir-us-ndh/index.html#dependencies).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the NDH `payment-accepted` extension (the `StructureDefinition` whose
`context` is `HealthcareService` and whose `value[x]` is bound to
`http://terminology.hl7.org/ValueSet/v2-0570`) to bind to the version of the
`hl7VS-paymentMethodCode` value set that ships in THO 7.0.1 (value set version
`3.0.0`). Concretely, change the `valueSet` reference on the
`Extension.value[x]` element binding from
`http://terminology.hl7.org/ValueSet/v2-0570|2.0.0` to
`http://terminology.hl7.org/ValueSet/v2-0570|3.0.0` (keeping the existing
`required` binding strength), regenerate the `payment-accepted` example on
`HealthcareService/HartfordOrthopedicServices`, and re-run the Touchstone
validation to confirm `CASH` (and the other v2-0570 codes) now resolve
against the bound value set. Document the change in the NDH change log.

#### Justification

This is the literal fix the reporter is asking for, and it restores internal
consistency between the NDH IG's declared THO 7.0.1 dependency and the value
set version actually referenced by its `payment-accepted` extension. Pinning
the binding to the value set version that matches the IG's THO dependency is
the standard FHIR pattern for IGs that need deterministic validation
behaviour, and it matches what the Touchstone Terminology Engine resolves
against today. It is also the lowest-risk change: only the version qualifier
on a single binding moves, and the `CASH` code (and the rest of v2-0570) are
preserved across THO 6.3 → 7.0.1.

---

### Disposition B: Alternative Approach

#### Proposal

Remove the explicit version pin from the `payment-accepted` extension's
binding so it references `http://terminology.hl7.org/ValueSet/v2-0570`
(unversioned) with `required` strength, and rely on the IG's declared THO
dependency (currently 7.0.1) to determine which value set version the
publisher and validators load. As part of the same change, audit the rest of
the NDH FSH (Base-VS, Base-Extensions, Base-Profiles) for other
`v2-0570|2.0.0`-style version-pinned bindings that were anchored to THO 6.3
and remove those pins as well, so a future THO bump only requires updating
`sushi-config.yaml` rather than every binding.

#### Justification

The root cause is a version-pin that drifted out of sync with the IG's THO
dependency; left as-is, the same class of bug will recur every time THO
publishes a new version. Dropping the explicit version qualifier lets the IG
publisher and downstream validators resolve the value set against the THO
version the IG actually depends on, eliminating an entire category of
"wrong-version" terminology errors going forward. The trade-off versus
Disposition A is that NDH gives up deterministic pinning and inherits any
future changes to v2-0570 silently — but for an HL7 v2 table value set that
is essentially stable across THO releases, that exposure is small, and the
maintenance win across the IG is significant.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the basis that this is a deployment / dependency
configuration issue rather than a defect in the NDH IG, and direct the
reporter to ensure their Touchstone validation environment loads THO 7.0.1
(matching the IG's declared dependency) so that
`hl7VS-paymentMethodCode|3.0.0` resolves correctly.

#### Justification

This is the weakest of the three options. The ticket's own evidence — the
Terminology Engine reporting the binding as
`v2-0570|2.0.0` — shows that the version pin lives inside the IG, not in the
validator's environment, so re-pointing Touchstone to THO 7.0.1 would not
change which value-set version the binding resolves against. Declining would
leave the IG internally inconsistent (THO 7.0.1 declared, THO 6.3-era value
set bound) and would push the same Connectathon failure onto every other
implementer. The only real reason to choose this disposition would be if PA
decides the inconsistency is intentional (e.g., NDH wants to freeze on
v2-0570|2.0.0 even after upgrading THO), in which case the IG's THO
dependency itself would need to be re-examined.

---

### Recommendation

**Recommended disposition:** B

Disposition B fixes the reported failure exactly the same way Disposition A
does — `CASH` and the rest of v2-0570 will validate against the value set
version that ships in THO 7.0.1 — but it also removes the underlying source
of the bug (a hand-maintained version pin that has to be remembered every
time THO is bumped). The ticket itself frames the issue as "the binding is
anchored to an older THO than the IG depends on", which is precisely the
pattern that unversioned bindings + a single declared THO dependency are
designed to prevent. Disposition A is an acceptable fallback if PA has a
specific reason to keep deterministic version pins on terminology bindings;
Disposition C should be rejected because the evidence in the ticket clearly
locates the defect inside the IG, not in the validator's environment.
