## Summary

FHIR-53853 asks Patient Administration to remove the `network` element from
the `PractitionerRole` and `OrganizationAffiliation` resources before those
resources go Normative in R6. The reporter argues that `network` was not
present in R4 or R5, that its semantics are still contested across realms and
domains, and that it may not yet have the implementation maturity required
for Normative status. The ticket proposes deferring the element — keeping
the capability available only as a profiled extension in experimental/STU
domains (e.g., NDH), and revisiting a Normative version of `network` when
the insurance-oriented resources (`InsuranceProduct`, `InsurancePlan`) are
ready to advance together.

The ticket explicitly cites two prior Jira items as evidence of ongoing
contention around the element's shape and semantics:
[FHIR-38193](https://jira.hl7.org/browse/FHIR-38193), which argued the
element's cardinality should be `0..*` so a role can belong to multiple
networks and was ultimately resolved in the NDH IG by requiring a separate
`PractitionerRole` per network; and
[FHIR-52859](https://jira.hl7.org/browse/FHIR-52859), which raised the same
`0..1` cardinality as a major workflow problem for payer/provider data
alignment and was resolved by changing the NDH `PractitionerRole.extension[network]`
from `0..1` to `0..*` (matching R6's cardinality). Together these tickets
show that both the cardinality and the conceptual placement of `network`
are still being negotiated inside the downstream IG that uses the element
most heavily, which is the reporter's core argument for not locking the
element into Normative.

### [FHIR-52859: Clarify PractitionerRole-Network Relationship Between Payer/Provider Data](https://jira.hl7.org/browse/FHIR-52859)

Resolved – change required · Persuasive with Modification · NDH (FHIR)
· Patient Administration. Reopens the same cardinality concern from the
opposite direction: `0..1` forces either duplicated `PractitionerRole`

contradicts the prior FHIR-38193 outcome and shows the WG has been
iterating on the element as recently as March 2026 — squarely supporting
the reporter's maturity argument.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the `network` element from both `PractitionerRole` and
`OrganizationAffiliation` in the R6 base resources before they go
Normative. Specifically:

- Drop `PractitionerRole.network` (Reference(Organization)) from the
  StructureDefinition of `PractitionerRole`.
- Drop `OrganizationAffiliation.network` (Reference(Organization)) from
  the StructureDefinition of `OrganizationAffiliation`.
- Migrate the current definition into a standard extension
  (e.g., `http://hl7.org/fhir/StructureDefinition/practitionerrole-network`
  and `…/organizationaffiliation-network`) with cardinality `0..*` so
  that NDH and other STU IGs can continue to profile it without
  requiring the element in the Normative base.
- Add a change note to both resources explaining the deferral and the
  intent to revisit the element for Normative once `InsuranceProduct` /
  `InsurancePlan` advance.
- Ensure the US Core / NDH IGs that currently rely on the base element
  are migrated to the extension (NDH already uses
  `PractitionerRole.extension[network]`, so this is largely a cleanup).

#### Justification

The element was not present in R4 or R5, so removing it from R6 core does
not break conformance against any existing Normative baseline — the
Normative bar is about mature consensus, not new additions. Both cited
tickets (FHIR-38193 → "use separate `PractitionerRole` per network";
FHIR-52859 → "change the NDH extension from `0..1` to `0..*`") were
resolved in opposite directions within roughly three years, and
FHIR-52859 was still being re-resolved in March 2026. That pattern is
exactly what the Normative gate is designed to screen out. Moving the
definition to an extension preserves every downstream use case while
keeping the Normative boundary clean, and it lines up semantically with
the "network" concept being insurance-adjacent — i.e., a companion to
the not-yet-Normative `InsuranceProduct`/`InsurancePlan` track.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `PractitionerRole.network` and `OrganizationAffiliation.network`
defined in the R6 base resources, but explicitly **exclude these two
elements from the Normative portion** of each resource and mark them
Trial Use (FMM lowered, or `fmm-no-warnings="false"`), while the rest of
each resource goes Normative. Concretely:

- Leave both `network` elements on the base `StructureDefinition`.
- Annotate each with a Trial-Use / lower-FMM marker and a prominent
  comment in the resource intro calling out that `network` is not
  Normative and that its cardinality and semantics may still change
  alongside `InsuranceProduct` / `InsurancePlan`.
- Publish a parallel extension so projects that want Normative-grade
  stability for the "network" concept can opt in to the extension rather
  than the base element — in effect, mirroring the NDH pattern upstream.

#### Justification

Precedent exists for shipping a Normative resource with specific
elements carved out as Trial Use (the same mechanism is used on
`Observation`, `Patient`, and others). This preserves implementer
ergonomics — NDH, EHR vendors, and DIRECT Trust profilers keep using a
first-class element — while capturing the reporter's concern that the
`network` semantics are still moving. It also avoids the downstream
churn of migrating NDH away from the extension it just finished
stabilising in FHIR-52859, then potentially re-migrating back once
`InsuranceProduct` is ready. The trade-off versus A is that "an element
on a Normative resource that isn't Normative" is a well-known source of
implementer confusion, so the carve-out must be signposted very clearly.

---

### Disposition C: Decline

#### Proposal

Keep both `PractitionerRole.network` and `OrganizationAffiliation.network`
in the R6 base resources and allow them to go Normative alongside the
rest of those resources, with no structural change. Treat the remaining
semantic questions (cardinality, relationship to `InsuranceProduct`,
payer-vs-provider modelling) as routine post-Normative errata/extension
work.

#### Justification

The `network` element is already in active use in at least one
production-track IG (NDH) and has been exercised by payer-directory
implementations. The two cited tickets did not invalidate the element
itself — they argued over its cardinality and over whether a parallel
`NetworkAffiliation` resource should exist, both of which can be
addressed through extensions, profiles, and future resources without
removing the base element. Removing a widely-implemented element
immediately before Normative publication is itself disruptive and can be
read as signalling that the resource isn't ready, which could cascade
into broader Normative-track delays for PA.

---

### Recommendation

**Recommended disposition:** A

The Normative bar is consensus + stability, and the cited history
makes the stability claim hard to defend right now: FHIR-38193 and
FHIR-52859 reach opposite conclusions about the same element, and the
latter was still being re-resolved in March 2026. FHIR-38193's resolution
(use a separate `PractitionerRole` per network) and FHIR-52859's
resolution (raise cardinality to `0..*`) cannot both be right, and the
WG has not yet reconciled them in the base resource. Because `network`
was not in R4 or R5, removing it from R6 core costs nothing in
Normative-baseline breakage, and migrating to a standard extension keeps
NDH and DIRECT-Trust-style profiles working with only a light lift (NDH
already uses the extension shape). Disposition B is a reasonable
fallback if the WG judges the implementer cost of dropping a first-class
element too high, but the "Normative resource with a non-Normative
element" pattern is precisely the kind of subtle Normative footnote that
implementers miss, and the reporter's framing — "revisit when
`InsuranceProduct`/`InsurancePlan` advance" — fits A more cleanly.
Disposition C should only be preferred if the WG has new evidence that
the cardinality and cross-domain semantics are in fact settled; the
ticket's own citations argue the opposite.
