## Summary

The reporter (Carmela A. Couderc) asks Patient Administration to change the
example binding on `Location.code` so that it points at the THO value set
[`http://terminology.hl7.org/ValueSet/v3-Country2`](https://terminology.hl7.org/ValueSet-v3-Country2.html)
(the THO 2-letter ISO 3166-1 country code value set). The current example
binding on `Location.code` resolves to the FHIR core value set
`valueset-iso3166-1-2` (the same conceptual list, but defined in core), which
is being retired as the agreed resolution of
[FHIR-51120](https://jira.hl7.org/browse/FHIR-51120) ("Table shows 3-letter
codes"). FHIR-51120 was discussed on the 2025-12-01 TI tracker call and
resolved as *Persuasive — Resolved, change required*: the duplicate
`valueset-iso3166-1-2` in core is to be retired in favor of the existing THO
`v3-Country2` value set, and Carmela was tasked with filing a follow-up
ticket against Patient Administration to repoint the `Location.code` example
binding. FHIR-56307 is that follow-up. Once applied, `Location.code` will
continue to have an example binding to a 2-letter ISO 3166-1 country code
value set, but sourced from THO instead of core.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the example binding on `Location.code` from the core
`http://hl7.org/fhir/ValueSet/iso3166-1-2` to the THO value set
`http://terminology.hl7.org/ValueSet/v3-Country2`. The binding strength
remains `example`. No change to `Location.code` cardinality, type
(`CodeableConcept`), definition, or other constraints. Concretely:

- In the Location resource definition, update the `binding.valueSet` on
  `Location.code` to `http://terminology.hl7.org/ValueSet/v3-Country2`
  (no version pin, consistent with how other THO references are made in
  example bindings).
- Refresh any narrative / rendered binding tables on the Location page
  to reference the THO value set.
- Confirm that the Location examples / example instances do not encode
  codes that were valid only in the retired core value set; the THO
  `v3-Country2` value set is the same 2-letter ISO 3166-1 set, so
  existing 2-letter examples remain valid.

#### Justification

This is exactly the disposition contemplated and pre-agreed by the
2025-12-01 TI tracker call that resolved FHIR-51120. The core
`valueset-iso3166-1-2` is being retired because it is a duplicate of THO
`v3-Country2` (and the core copy has rendering/regex inconsistencies);
leaving `Location.code` bound to the retired core value set would create a
dangling reference. THO is the correct home for cross-paradigm code
systems like ISO 3166-1, and an example-strength binding does not impose
any conformance burden on implementers. This change is purely a pointer
update and aligns Patient Administration's deliverable with the
Terminology Infrastructure decision.

---

### Disposition B: Alternative Approach

#### Proposal

Repoint `Location.code` to the THO `v3-Country2` value set as in
Disposition A, but at the same time strengthen the binding from `example`
to `preferred` (or add a `code` binding-name + maxValueSet of the same
THO VS) to give implementers a clearer steer that ISO 3166-1 alpha-2 is
the intended vocabulary for "Location code that conveys a country", while
still allowing other coding systems where Location is used for non-country
purposes (rooms, beds, etc.).

#### Justification

`Location.code` is a fairly free-form `CodeableConcept` whose intent
varies by context (country, region, building, room, bed). The current
example binding to a country code value set is arguably misleading: it
hints that countries are *the* expected vocabulary, when in practice
Location codes are dominated by physical-location codings like
`v3-RoleCode`. If the workgroup believes the country example is genuinely
representative, strengthening to `preferred` makes that intent explicit;
if not, the workgroup might consider Disposition A while opening a
separate discussion about whether a different example value set
(e.g., `v3-RoleCode` or a service-delivery-location subset) would better
illustrate Location.code. This option keeps FHIR-56307 small (still just
a binding update) but signals the broader question.

---

### Disposition C: Decline

#### Proposal

Do not change the example binding on `Location.code`, leaving it pointed
at the core `iso3166-1-2` value set (and rely on the retirement of that
value set in FHIR-51120 to leave a redirect / tombstone, or treat the
broken pointer as an acceptable cost of the R6 cycle).

#### Justification

Declining is hard to justify here. The only defensible rationale would be
that Patient Administration prefers to remove the example binding
entirely (because country codes are not a representative example for
`Location.code`, see Disposition B's analysis). However, that would
itself be a different change, not a "leave as-is", because the current
binding will become a dangling reference once FHIR-51120 is applied.
Pure decline therefore knowingly leaves a broken value-set pointer in the
published Location definition.

---

### Recommendation

**Recommended disposition:** A

This is the disposition the cross-workgroup agreement on the 2025-12-01 TI
tracker call already anticipated: Patient Administration follow-up to the
Terminology Infrastructure resolution of FHIR-51120. The change is a
one-line binding-target update at `example` strength against an
equivalent THO value set, with no implementer impact and no risk of
breaking existing instances. Disposition B raises a fair editorial
question about whether a country code value set is the best example for
`Location.code`, but that is a larger conversation that should not block
the housekeeping required to keep the binding resolvable; if the
workgroup wants to pursue it, it should be filed as a separate ticket.
Disposition C is not viable because it would leave a dangling reference
once FHIR-51120 is applied.
