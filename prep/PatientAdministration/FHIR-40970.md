## Summary

The ticket asks the Real Time Location Services (RTLS) FHIR IG (owned by
Patient Administration) to add a `udiCarrier` to the example used to
illustrate the **Location Update** event. During triage the work group
concluded that the Location Update message only carries information about
the RTLS badge itself, which would not normally have a UDI, so adding
`udiCarrier` directly to that example would be misleading. Instead, the
work group proposes a "Persuasive with Modification" disposition: rather
than modifying the Location Update example, create a new example for the
**Enroll Tag** message that demonstrates how an RTLS badge is associated
with a real medical device (e.g., an infusion pump), where the medical
device's `Device.udiCarrier` is the natural place for UDI data. No
cross-referenced Jira tickets, Zulip discussions, GitHub items, or
Confluence pages were found by FhirAugury for this ticket, so the request
appears to be self-contained within the RTLS IG.

## Details

**Description:**

Add udiCarrier in the Location Update Event example

**Resolution Description (proposed by the work group):**

There are two different Devices that could be relevant:

1. The RTLS badge, which I don't think would have a UDI.
2. A medical device, like an infusion pump, that has the RTLS badge
   affixed, so staff can track where that infusion pump is located.

The Location Update message type is only communicating information about
the RTLS badge, so as far as we know, that wouldn't ever include a UDI.

However, the Enroll Tag message is what associates the RTLS badge with
the medical device, so there is a good example we can create for that
situation.

*We will create an example Enroll Tag message that shows how to
associate the RTLS badge with a medical device like an infusion pump.*

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the existing **Location Update** event example in the Real Time
Location Services FHIR IG to include a `udiCarrier` element on the
`Device` resource that represents the RTLS badge. The example would
populate `Device.udiCarrier.deviceIdentifier` (and optionally
`carrierHRF`/`carrierAIDC`) with a representative value for the badge.

#### Justification

It directly fulfills the literal request in the ticket and gives
implementers a concrete example of `udiCarrier` populated alongside a
location update. If a vendor *does* assign UDIs to their tracking badges
(some asset-tracking vendors do, especially when badges are themselves
regulated medical devices), having an authoritative example reduces
ambiguity about where to put that data on the badge `Device`.

---

### Disposition B: Alternative Approach (Persuasive with Modification)

#### Proposal

Leave the existing **Location Update** event example unchanged (it
intentionally communicates only badge-level information, which typically
has no UDI), and instead author a **new example for the Enroll Tag
message** in the RTLS IG. The new Enroll Tag example would show:

- A `Device` for the RTLS badge (no `udiCarrier`).
- A `Device` for a real medical device — e.g., an infusion pump — with
  `Device.udiCarrier` populated (`deviceIdentifier`, `issuer`,
  `jurisdiction`, `carrierHRF` or `carrierAIDC`, `entryType`).
- The Enroll Tag association linking the badge `Device` to the medical
  device `Device` so subsequent Location Updates for the badge can be
  interpreted as the location of that medical device.

The example would be added under the RTLS IG's examples folder and
referenced from the Enroll Tag message page.

#### Justification

This is the work group's already-recorded "Persuasive with
Modification" position and matches how the RTLS message set is
designed: Location Update carries badge state, and Enroll Tag is the
specific point at which a badge becomes meaningful for tracking a
piece of equipment. Putting `udiCarrier` on the medical device inside
an Enroll Tag example teaches implementers the *correct* modeling
without overloading the Location Update example with data it would not
realistically carry. It also avoids implying that badges generally have
UDIs, which would set a misleading expectation.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive": do not change either the Location Update
example or add a new Enroll Tag example. Document that the Location
Update event intentionally describes only badge state and that
implementers needing to convey UDI information for the tracked medical
device should follow general FHIR `Device.udiCarrier` guidance from the
core specification.

#### Justification

This avoids any change to the IG and keeps the existing examples small
and focused. It is defensible if the work group judges that the
modeling is already obvious from the base FHIR Device documentation and
that an additional example would not materially help implementers.

---

### Recommendation

**Recommended disposition:** B

The work group has already labeled this ticket
`rtls-ready-for-block-vote` with a resolution of "Persuasive with
Modification" and a written rationale that closely matches Disposition
B. That rationale is technically sound: the Location Update message
genuinely is about badge state, badges typically have no UDI, and the
Enroll Tag message is the natural place to demonstrate badge-to-device
association — which is exactly where `udiCarrier` belongs. Disposition
A would put `udiCarrier` in a misleading location and risks codifying a
pattern (UDI-on-badge) that is not generally true. Disposition C
ignores a real implementer pain point (how do I tie a badge's location
updates to the actual piece of equipment being tracked?). Adding a new
Enroll Tag example is a small, contained change to the RTLS IG that
delivers the educational value the requester was asking for, without
distorting the Location Update example.
