## Summary

The reporter requests that the Identity Assurance / Patient Matching IG
("Interoperable Digital Identity and Patient Matching (FHIR)") expand and
clarify the Use Cases currently presented on the STU1 Home page. The stated
problem is that the existing use case descriptions are difficult to parse
and need clarification. The proposed change is to author tangible, worked
examples for each use case — illustrating the nuances between them and
including graphics that depict the workflows — and to surface this material
in a dedicated new "Use Cases" tab in the IG. No specific resources,
profiles, or conformance constraints are proposed; the request is purely
narrative/IG-page guidance content.

The ticket has no cross-referenced Jira tickets, Zulip threads, or GitHub
items in the Augury index, and a free-text Zulip search for the ticket key
and topic ("Identity Assurance use case examples") returned no hits. The
ticket has already been resolved as **Not Persuasive** on 2025-06-19 with
the resolution note "Covered by other changes."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level "Use Cases" tab to the Interoperable Digital Identity
and Patient Matching (IDI) IG navigation. Move the existing Home-page Use
Cases content into that tab and expand it with one worked example per use
case (at minimum: Patient-mediated / Patient Access scenarios and B2B
identity-assurance scenarios). For each example, author:

- A short narrative scenario (actors, trigger, pre/post-conditions).
- A sequence-style workflow diagram (PlantUML/SVG asset checked into the
  IG sources).
- The relevant FHIR exchanges (sample `Patient`, identifier slices,
  identity-assurance level extension(s), and any supporting
  `Verification`/match-quality artefacts the IG defines).
- A "what makes this use case different" callout that distinguishes it
  from neighbouring use cases.

Cross-link the new Use Cases tab from the Home page in place of the
current dense Use Cases section. Because no GitHub repository was
surfaced as cross-referenced for this ticket, the specific authoring
root would need to be confirmed against the IDI IG repo before the work
is scheduled.

#### Justification

This directly addresses the reporter's stated problem (Home-page use cases
are hard to parse) using the mechanism they explicitly proposed (a
dedicated Use Cases tab with worked examples and graphics). Concrete,
diagrammed scenarios are a well-established pattern in HL7 IGs (e.g.,
US Core, SMART App Launch) and materially help implementers distinguish
between similar identity-proofing flows. The change is purely narrative —
it does not alter any conformance artefacts — so the implementer impact is
strictly positive (better docs, no breaking changes).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than creating a new tab, **augment the existing Home-page Use Cases
section in place** with:

- A short table that names each use case, its primary actor(s), and the
  one-line distinguishing characteristic.
- An inline mini-diagram per use case (one SVG each).
- A "See also" link from each row into the relevant IG section
  (conformance / actor / profile pages).

If a worked example is genuinely needed for a particular use case, author
it as an `Example` instance attached to the relevant profile page rather
than as standalone narrative on a new tab.

#### Justification

A new tab carries navigation and maintenance cost (additional menu real
estate, more pages to keep in sync, cross-links to update). The reporter's
underlying need — readers being unable to tell the use cases apart — can
be met by a comparison table plus per-use-case diagrams without
restructuring the IG's table of contents. Anchoring worked examples to the
profile pages they exercise also keeps narrative and conformance artefacts
co-located, which reduces drift over time. This approach also responds to
juliemaas's comment that the workgroup considers the existing use cases
already aligned with commercial Patient Access and B2B workflows: the
problem may be presentation density, not missing content.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as **Not Persuasive** on the basis that (a) the request
does not identify any specific use case that is unclear or any specific
content that is missing, (b) the workgroup has previously deliberated and
articulated these use cases against established commercial workflows
(Patient Access, B2B), and (c) the requested artefacts (worked examples,
diagrams, new IG tab) are substantial editorial work that should be
prioritised against concrete reader feedback rather than a general
"difficult to parse" assertion. Invite the reporter to reopen with
specific examples of which use case is unclear and what they expected to
see, or to submit a PR with a concrete proposed Use Cases tab.

#### Justification

This matches the disposition the workgroup actually reached on 2025-06-19
("Not Persuasive — Covered by other changes"). juliemaas's only comment on
the thread asks precisely for the missing specificity, and the reporter
did not respond. Without a concrete list of which use cases are unclear or
how the proposed examples should differ from the current text, accepting
the ticket would commit the workgroup to open-ended editorial work whose
acceptance criteria cannot be verified. Declining is also defensible
because subsequent IG work has reportedly already addressed the affected
material ("Covered by other changes" per the resolution note).

---

### Recommendation

**Recommended disposition:** C

The workgroup has already resolved this ticket as Not Persuasive on the
record, and that disposition is well supported by the file: the only
substantive comment (juliemaas, 2024-02-13) asked the reporter to provide
specifics about which use cases are unclear, and no response was ever
recorded. Without that specificity, neither Disposition A (full new tab)
nor Disposition B (in-place augmentation) has actionable acceptance
criteria, and either would commit the workgroup to open-ended editorial
work. The resolution note ("Covered by other changes") further indicates
that subsequent IG updates have already addressed the underlying readability
concern. If concrete examples of unclear use cases surface from
implementers, a follow-up ticket framed around those specific gaps would
be a better vehicle than reopening this one.
