## Summary

The reporter notes that the Real Time Location Services (RTLS) Implementation
Guide page at `http://hl7.org/fhir/uv/rtls/2023May/spec.html` does not describe
the transactions/interactions that an implementer would need in order to build
to the IG. He requests that the IG add (a) sequence diagrams that show the
actors involved and the interactions between them, (b) an indication of which
FHIR resources are exchanged in each interaction, and (c) worked examples that
narrate a realistic scenario (e.g., a patient being admitted, receiving a
locator tag, and moving between rooms while the RTLS notifies the EMR). The
ticket has been triaged by Patient Administration; the recorded resolution is
"Considered for Future Use" with the rationale that the RTLS work is presently
focused on the Staff Duress scenario for the 2.0 ballot, and broader
transaction/interaction documentation will be addressed in a later cycle. There
are no cross-referenced Jira tickets, Zulip threads, or GitHub items associated
with this issue in Augury.

## Details

**Description:**

The page http://hl7.org/fhir/uv/rtls/2023May/spec.html is missing details
related to the transactions/interactions. This makes difficult to implement
the IG.

My suggestion is to include sequence diagrams with the actors and
interactions. Also with an indication of which FHIR resources are/may be used
in each interaction.

Examples connected to the transactions would definitely help in the
implementation. For example: *"Patient John was admitted in Hospital Good Life.
He received a tag... When he moved to the medication room, the RTLS sends a
notification to the EMR..."*

**Resolution Description (recorded by the work group):**

This would be useful, but since we are focusing on Staff Duress initially, we
want to defer this until after the 2.0 ballot.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the Real Time Location Services IG home page (and/or a new
`interactions.html` page reachable from the IG table of contents) to include,
at minimum:

1. A short narrative section per supported use case (Staff Duress for 2.0,
   plus stubs for Patient Tracking, Asset Tracking, Hand-Hygiene, etc.)
   describing the actors (e.g., RTLS Sensor Network, RTLS Gateway, EMR,
   Clinical Workflow System, Security Operations Console).
2. A PlantUML/Mermaid sequence diagram per use case showing the
   request/response (or notification) flow between those actors, annotated
   with the FHIR resource(s) exchanged on each arrow (e.g.,
   `Location`, `Device`, `DeviceMetric`, `Observation`,
   `Encounter`, `Communication`, `Subscription`/`SubscriptionTopic`,
   `Provenance`).
3. One worked example per use case using realistic (synthetic) instance data,
   modelled after the reporter's "Patient John … medication room" narrative,
   linked from the IG examples folder.
4. A "Conformance for Implementers" subsection that calls out which actor
   plays which role and which interactions are mandatory vs. optional.

Implementation lives in the RTLS IG authoring repository (HL7-published IG
source — confirm `owner/name` against the work group's published IG list
before editing). Likely-touched paths once the repo is identified:
`input/pagecontent/*.md` for narrative and diagrams, `input/examples/` for
worked instance data, and the IG ToC (`sushi-config.yaml` or
`ig.ini`/`ImplementationGuide-*.json`) to register the new page.

#### Justification

This is what the reporter actually asked for. Sequence diagrams plus
resource-mapped interactions are a long-standing pattern in HL7 IGs (compare
SMART App Launch, IPS, Bulk Data, IHE PIX/PDQm-on-FHIR), and they are widely
regarded as the single most valuable artifact for implementers picking up a
workflow-oriented IG. The reporter's narrative-style example shows that a
non-trivial barrier exists today — the page describes resources but not
*flow*, which is the dimension implementers care about. RTLS is a
workflow/eventing IG by nature, so the absence of interaction diagrams is a
genuine gap rather than a stylistic preference.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring fresh diagrams in narrative pages, formalise the
RTLS interactions as **CapabilityStatements + SubscriptionTopic + ExampleScenario**
artifacts:

1. Define one `CapabilityStatement` per actor (e.g., `RTLS-Server`,
   `RTLS-Consumer-EMR`) listing the resources, interactions, and search
   parameters each actor supports.
2. Define `SubscriptionTopic` resources for the location-change /
   duress-event triggers, so the "RTLS notifies the EMR" flow is described
   as a conformance artifact, not just prose.
3. Author one `ExampleScenario` per use case (Staff Duress for 2.0, others
   stubbed) — `ExampleScenario` was added to FHIR specifically to describe
   actor/step/instance flows, and the IG publisher renders it as a sequence
   diagram automatically, eliminating the need to hand-maintain PlantUML.
4. Cross-link a thin narrative page that points to those artifacts and walks
   through the rendered scenario.

#### Justification

This approach addresses the same implementer pain (no visible interactions)
while replacing hand-drawn diagrams with first-class, machine-readable FHIR
conformance resources. `ExampleScenario`-driven docs stay in sync with the
IG's resources because the publisher generates the rendered diagram from the
artifact; hand-authored sequence diagrams routinely drift from the
profiles they describe. It also gives implementers something they can
**test against** (a `CapabilityStatement` and a `SubscriptionTopic`), not
just something to read. Trade-offs vs. A: higher up-front authoring cost,
and `ExampleScenario` rendering is still uneven across IG publisher
versions, so the diagrams may need a fallback. Lower cost long-term.

---

### Disposition C: Decline (defer per current resolution)

#### Proposal

Decline to act on FHIR-40879 in the 2.0 ballot cycle and keep the recorded
resolution of **"Considered for Future Use"**. The IG editor would add a
single explicit note on the RTLS home page (and on the Staff Duress section)
saying that the 2.0 ballot is intentionally scoped to Staff Duress and that
broader interaction documentation, sequence diagrams, and worked examples
across other RTLS use cases will be added in a subsequent ballot cycle once
those use cases are themselves in scope. The ticket is then carried forward
on the work-group backlog (already labelled `rtls-ready-for-block-vote`) so
it is re-considered when scope expands.

#### Justification

This matches the work-group's existing recorded position: the 2.0 ballot is
narrowly scoped to Staff Duress, and authoring full
actor/interaction/example coverage for use cases that are *not* in 2.0 risks
publishing diagrams that will need to be reworked once those use cases are
formally specified. Declining (with a visible scope note) is also defensible
because the reporter's underlying concern — implementer confusion — is at
least partially mitigated by being explicit about scope on the page itself,
rather than leaving readers to infer it. The change request is not a
correctness defect; it is a documentation expansion request that depends on
specification work that has not happened yet.

---

### Recommendation

**Recommended disposition:** C (with a small carve-out from A for Staff Duress)

The work group has already triaged this as "Considered for Future Use" with
the explicit reasoning that 2.0 is scoped to Staff Duress, and that recorded
resolution should be honoured for the block vote. However, declining purely
on scope without giving the in-scope use case the diagram/example treatment
the reporter is asking for would leave 2.0 readers in the same position the
reporter complained about. The pragmatic path is therefore: take Disposition
C as the formal disposition for FHIR-40879 (defer broad coverage to a future
ballot, add a visible scope note on the IG home page), and *separately* —
not under this ticket — ensure that the Staff Duress section in the 2.0
ballot ships with at least one sequence diagram and one worked example,
which is the minimum needed to make the in-scope content implementable.
Disposition A remains the right long-term answer once additional RTLS use
cases are themselves in scope; Disposition B is worth revisiting at that
point if `ExampleScenario` tooling has matured.
