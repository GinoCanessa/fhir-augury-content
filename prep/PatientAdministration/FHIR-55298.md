## Summary

The ticket reports that the **InsurancePlan** resource is still part of the
balloted FHIR Core specification even though Patient Administration (PA) and
the FHIR Management Group (FMG) had previously agreed it would be moved out
of the core spec into an implementation guide. Lloyd McKenzie (the reporter)
asks PA to actually carry out the agreed removal **before the next ballot**,
so that an out-of-scope resource is not balloted again. No cross-referenced
Jira tickets, Zulip threads, or GitHub items are recorded in the Augury
index for FHIR-55298 itself, so the request stands on its own; supplementary
search confirms that an `insuranceplan/` resource folder still exists under
`HL7/fhir` and that a parallel `insuranceplan/` resource folder already
exists in `HL7/admin-incubator`, which is the most plausible target IG for
the move.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Carry out the previously-agreed move of **InsurancePlan** out of FHIR Core
and into an implementation guide before the next ballot freeze. Concretely:

1. In `HL7/fhir`, retire the `source/insuranceplan/` folder: remove the
   StructureDefinition, search-parameter bundle, examples, value sets,
   introduction page, and any per-resource indexes; remove InsurancePlan
   from the resource list, the admin-module page diagrams, the
   CapabilityStatement (`source/profiles-others.html` /
   `source/conformance/`), and any cross-resource references (e.g.,
   `Coverage`, `Organization`, `Patient`, `EpisodeOfCare`, `Account`)
   that point at InsurancePlan in core text or examples — replace with
   prose that says the resource lives in the admin IG and link to the
   IG's canonical URL.
2. Confirm/finish the receiving copy in `HL7/admin-incubator` under
   `input/resources/insuranceplan/` (StructureDefinition, examples,
   value sets `valueset-insuranceplan-applicability` and
   `valueset-insuranceplan-type`, the
   `bundle-InsurancePlan-search-params.xml`, and
   `list-InsurancePlan-examples` / `list-InsurancePlan-operations`),
   then republish the IG.
3. Add a redirect / "this resource has moved" note in core so existing
   `http://hl7.org/fhir/StructureDefinition/InsurancePlan` consumers know
   where to find it; coordinate canonical-URL handling with FMG.
4. Re-target any unresolved InsurancePlan PA tickets (e.g., FHIR-50876,
   FHIR-54445, FHIR-44965/44966, FHIR-26040, FHIR-18242) at the IG so
   they don't strand against a deleted core resource.

#### Justification

The ticket is procedural: the work group and FMG have already made the
substantive decision; this is execution. Doing it before the next ballot
ensures the published spec and the balloted scope agree, which is exactly
what FMG asked for and what the reporter is escalating. It also unblocks
the receiving IG (`HL7/admin-incubator`) — the resource skeleton is
already in place there, so the carve-out is mostly a delete-in-core /
finalise-in-IG operation, not new design work. The cost is mostly
mechanical (cross-reference cleanup in core, CapabilityStatement edits,
canonical-URL coordination) and is well-scoped.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than removing InsurancePlan from this ballot cycle, **mark it
"Withdrawn" / "Not balloted"** in core for this cycle (e.g., flag the
resource as not part of the balloted scope, suppress it from the printed
ballot package and from the resource list / module diagrams, and add an
explicit notice on `source/insuranceplan/insuranceplan-introduction.xml`
that it is being moved to `HL7/admin-incubator` and is not subject to
ballot in this cycle). Defer the physical delete from `source/` to the
*following* cycle, once the IG copy in `admin-incubator` has been
balloted and published at FMM ≥ current core FMM.

#### Justification

This addresses the underlying concern raised by the reporter — that an
out-of-scope resource gets balloted — without taking on the full
multi-repo carve-out under ballot pressure. It avoids two real risks of
Disposition A done in a hurry: (a) breaking inbound references from other
core resources / IGs that have not yet been retargeted, and (b) leaving
the IG copy at a lower maturity than the resource enjoyed in core, which
would be a real regression for implementers. The trade-off is that the
core source tree still carries the file for one more cycle, which is
strictly weaker than what the reporter asked for; it is, however, faithful
to the FMG decision because the resource is no longer *balloted* as core.

---

### Disposition C: Decline

#### Proposal

Decline the request and **leave InsurancePlan in core for the upcoming
ballot**, with a note in the ticket that the move-to-IG decision is
deferred to a future cycle and that this ticket should be closed as "Not
Persuasive — already in flight / out of cycle".

#### Justification

A defensible decline rests on (a) the move-to-IG work not yet being
credibly ready in `admin-incubator` (the receiving canonical URLs,
search-parameter behaviour, examples, and downstream IG references must
all be in place before deletion from core, otherwise the ballot ships a
broken cross-reference graph), and (b) ballot freeze being too close to
absorb a multi-repo carve-out safely. Declining preserves the status quo
for one more cycle and avoids the risk of shipping a partially-moved
resource. The cost is that the reporter's complaint — that the spec is
balloting an explicitly out-of-scope resource — remains true, and the
PA/FMG agreement is publicly broken for another cycle.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach)

The reporter and the FMG agreement both want InsurancePlan out of the
*balloted* scope; what they actually need before ballot is for the
resource to stop appearing as a balloted core artifact. Disposition B
delivers that — by marking the resource as withdrawn / not-balloted in
this cycle and pointing readers at the admin incubator IG — while leaving
the more invasive cross-reference cleanup, canonical-URL redirection,
and IG-side maturity work to land cleanly in the following cycle as
Disposition A. Disposition A is the right end-state but is risky to
land under ballot pressure given the cross-resource references from
Coverage, Organization, Patient, EpisodeOfCare, Account, and the absence
of a current `repo-analysis` briefing for either `HL7/fhir` (briefing is
scoped to a different ticket) or `HL7/admin-incubator` (no briefing) —
both of which should be refreshed before any code-level change is
attempted. Disposition C is the weakest option because it leaves the
ballot in the very state the reporter is escalating against.
