## Summary

The ticket asks NDH to stop carrying its IHE-specific "actor and transaction
indication" mechanism as an NDH-private extension and instead make the
equivalent functionality available more broadly — the reporter argues this
capability is generally useful and should live in the FHIR core extension
pack rather than being scoped to NDH. The reporter (Bas van den Heuvel)
flagged this together with [FHIR-42638](https://jira.hl7.org/browse/FHIR-42638),
the sister ticket that makes the same argument for the rest of NDH's general-
purpose extensions (Logo, Rank, Access Control, Associated Servers); both
were originally voted "Considered for Future Use" in 2023 and re-opened
post-STU publication in 2025 by Lynn Laakso for proper disposition.

While this ticket has been pending, the wider Endpoint conversation has
moved on. [FHIR-48540](https://jira.hl7.org/browse/FHIR-48540) (Applied)
consolidated NDH's `ihe-specific-connection-type`, `endpoint-usecase` and
`endpoint-nonfhir-usecase` extensions into a single renamed
`implementation-guide` extension on the NDH `Endpoint` profile. In parallel,
John Moehrke filed [FHIR-53711](https://jira.hl7.org/browse/FHIR-53711)
against **FHIR Core (R6)** asking for the same idea — a complex
`Endpoint.actor` extension with `implementationGuide`, `version`, `actor`
and `options` — which has been resolved Persuasive with Modification by
PA. The proposed disposition on FHIR-42639 leans on those: NDH's role is
not to define core extensions directly; the path is a separate Jira
ticket against the extension pack (now effectively FHIR-53711) followed
by an NDH adoption ticket once that core extension exists.

## Details

**Description:**

These extension indicate actors and transactions that are supported by an
endpoint. This is very interesting functionality that should not be NDH
specific but more readily available as a FHIR core extension.

**Resolution Description (proposed):**

The extension needs to be added to the extension pack by a Jira ticket for
the global PA committee. Once that extension exists then NDH can consider
that extension in a new Jira ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the NDH `ihe-specific-connection-type` extension (and the
companion structure for indicating the IHE actor + transaction +
options on `Endpoint`) out of the NDH IG and into the FHIR core
extension pack as a standard, IG-agnostic extension on `Endpoint` /
`Endpoint.payload`. Concretely, define a complex extension with
`implementationGuide` (canonical|string 1..1), `version` (string 0..1),
`actor` (canonical 0..1 to `ActorDefinition` | string | uri) and
`options` (string 0..*), and add a single placeholder code to the
example-bound `connectionType` code system that signals "see the
extension". Update the NDH `Endpoint` profile to reference the new
core extension and retire the NDH-local one.

#### Justification

The reporter's underlying point — that "indicate which IG/actor/
transaction this endpoint speaks" is not IHE- or NDH-specific — has
been independently confirmed by John Moehrke's
[FHIR-53711](https://jira.hl7.org/browse/FHIR-53711), which describes
the same mechanism in core terms and was resolved Persuasive with
Modification by PA. Doing it once in the extension pack avoids a
proliferation of one-off IG extensions (a concern Moehrke explicitly
called out) and lets non-FHIR endpoints (where `CapabilityStatement`
is not applicable) describe themselves consistently.

---

### Disposition B: Alternative Approach — Resolve via the FHIR Core / Extension-Pack Ticket and a Follow-On NDH Adoption Ticket (Persuasive with Modification)

#### Proposal

Resolve FHIR-42639 as **Persuasive with Modification**: agree with the
intent, but explicitly delegate execution to
[FHIR-53711](https://jira.hl7.org/browse/FHIR-53711) (the FHIR Core
ticket already accepted by PA at the January 2026 WGM) for defining
the new standard `Endpoint` actor extension. Add a new follow-on NDH
ticket (or reuse the open `implementation-guide` cluster — FHIR-53707,
FHIR-53708, FHIR-54130) that, once the core/extension-pack extension
is published, replaces the NDH `implementation-guide` /
`ihe-specific-connection-type` constructs with the core extension and
drops the NDH-local versions. This mirrors the same workflow the
proposed disposition on FHIR-42638 uses.

#### Justification

This is the path the workgroup has already started walking: FHIR-48540
consolidated the NDH extensions and renamed `fhir-ig` to
`implementation-guide`, FHIR-53711 has been accepted in core, and
several NDH tickets (FHIR-53707/53708/54130) are tracking the further
shaping of the same extension. NDH does not own the FHIR core
extension pack, so the cleanest, governance-correct resolution is to
mark FHIR-42639 as accepted-in-principle and explicitly point to the
core ticket as the implementation vehicle, rather than closing it as
"Not Persuasive" (which understates the agreement). It also gives
implementers a clear breadcrumb trail from FHIR-42639 to the actual
delivered change.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Resolve as **Not Persuasive** with the resolution text already drafted
on the ticket: "The extension needs to be added to the extension pack
by a Jira ticket for the global PA committee. Once that extension
exists then NDH can consider that extension in a new Jira ticket."
Take no action against NDH itself — the IHE-specific extension stays
in NDH for now, and any move is contingent on a separately filed
extension-pack ticket plus a future NDH adoption ticket.

#### Justification

NDH is not in a position to publish FHIR-core extensions, and the IG
is the appropriate venue for an extension to demonstrate maturity
before promotion to the extension pack (the same rationale used for
FHIR-42638 and consistent with general HL7 extension-promotion
practice). The reporter has not filed a separate extension-pack
ticket, so there is no concrete proposal in front of the workgroup
that NDH itself can act on. The reporter requested in-person but did
not respond on-ticket to the proposed disposition (Moehrke prompted
him on 2026-01-19), so resolving Not Persuasive is procedurally
defensible.

---

### Recommendation

**Recommended disposition:** B

Disposition C (the currently drafted "Not Persuasive") is technically
defensible but sends the wrong signal: PA *has* in fact accepted the
underlying request via FHIR-53711, and FHIR-48540 and the open
`implementation-guide` cluster show NDH is actively reshaping these
extensions in line with what the reporter asked for. Closing as Not
Persuasive will read, to the reporter and to anyone tracing back from
FHIR-53711, as a contradiction.

Disposition A overreaches for an NDH-scoped ticket — NDH cannot
unilaterally publish a FHIR core extension, and several substantive
design questions raised on FHIR-53711 (Eric Silver on whether
`options` should be coded; Costa Teixeira on whether
`ActorDefinition` alone could carry IG+version+actor; whether to
modify `connectionType` itself rather than add a "look-in-extension"
code) are not yet settled. Driving that work from this ticket would
duplicate FHIR-53711.

Disposition B captures the actual state of play: agree with the
intent, point at FHIR-53711 as the delivery mechanism, and queue an
NDH-side adoption ticket for when the core extension lands. This
keeps the audit trail clean, honours the reporter's original ask, and
matches the workflow already adopted for FHIR-42638's children.
