## Summary

This ticket, filed by Ron G. Parker against the Real Time Location Services
(RTLS) FHIR IG, asks whether the official URL for a terminology artifact
(specifically a Subscription Events value set / code system referenced by the
RTLS IG) should be anchored under `terminology.hl7.org` rather than under
`hl7.org/fhir`. The title carries a "2" suffix and the body is annotated as
"Comment 2 - imported by: Ron G. Parker", indicating it is a follow-up
comment imported as its own change request alongside related RTLS feedback.
No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the augury index for this ticket key, so the request stands on
its own without a linked discussion trail. The ticket has already been
resolved by the Patient Administration work group as **Not Persuasive**, on
the basis that the underlying Subscription Events value set and code system
are being removed from the IG entirely as part of other, more substantive
spec changes — making the THO-vs-IG hosting question moot.

## Proposed Dispositions

> Note: The work group has already resolved this ticket as "Not Persuasive"
> with the rationale that the underlying Subscription Events value set and
> code system are being removed. The dispositions below restate the
> available options to support a confirming block vote.

### Disposition A: Accept As Requested

#### Proposal

Move the canonical URL of the RTLS IG's Subscription Events value set and
code system from the IG namespace (`http://hl7.org/fhir/uv/rtls/...` or the
equivalent IG-rooted URL) to a `http://terminology.hl7.org/...` namespace,
and submit the artifacts to HL7 Terminology (THO) for hosting and stewardship
by the Vocabulary work group. Update all uses inside the RTLS IG to
reference the new THO canonical URL, and add deprecation/redirect notes for
the old IG-rooted URL.

#### Justification

HL7 governance and the Vocabulary work group's URL guidance generally
prefer that reusable code systems and value sets — particularly those that
might be referenced from outside the originating IG — live under
`terminology.hl7.org` rather than inside an IG-specific namespace. Moving
the artifacts to THO would align RTLS with that pattern, avoid future
confusion about ownership, and make the artifacts discoverable to non-RTLS
implementers who might want to reuse them (e.g., other Subscription-based
IGs).

---

### Disposition B: Alternative Approach

#### Proposal

Keep the artifacts inside the RTLS IG namespace (under `hl7.org/fhir/uv/rtls`
or equivalent) but add explicit notes in the IG's terminology section
clarifying (a) why these artifacts live in the IG rather than THO, (b) that
they are scoped to RTLS Subscription Events use cases, and (c) the
conditions under which they would be promoted to THO in a future version.
This preserves the current URLs (avoiding a breaking canonical change) while
addressing the underlying concern about discoverability and governance
clarity.

#### Justification

Changing a published canonical URL is itself a breaking change for any
implementer who has bound to it. If the Subscription Events terminology is
narrow in scope and unlikely to be reused outside RTLS, IG-local hosting is
defensible and consistent with how many "purpose-built" IG terminologies
are published. A documentation-level fix addresses the substance of the
reporter's concern (governance/discoverability) without forcing a URL
migration.

---

### Disposition C: Decline (Not Persuasive — current resolution)

#### Proposal

Decline the request as **Not Persuasive** on the grounds that the affected
artifacts (the Subscription Events value set and code system) are being
removed from the RTLS IG as part of other, more substantive changes. With
the artifacts gone, the question of whether they should be anchored at THO
or at `hl7.org/fhir` no longer has a target and need not be decided. Carry
this ticket through the RTLS block vote with the
`rtls-ready-for-block-vote` label.

#### Justification

This matches the current work-group resolution and is the lowest-risk path:
the Subscription Events value set and code system being removed eliminates
the artifact in question, so neither relocating it nor documenting its
hosting is meaningful work. The reporter's concern is rendered moot by the
broader spec change rather than dismissed on its merits, which is the
honest characterization to record in the resolution. The
`rtls-ready-for-block-vote` label indicates this disposition has already
been queued for confirmation.

---

### Recommendation

**Recommended disposition:** C

The ticket is already resolved as Not Persuasive, and the rationale is
sound: the Subscription Events value set and code system are being deleted
from the RTLS IG as part of other changes, so there is no longer a URL to
relocate. Disposition A (move to THO) and Disposition B (keep but document)
both presuppose the artifact's continued existence, which is no longer the
case. Confirming Disposition C via the queued block vote closes this
ticket cleanly without doing nugatory work, while preserving the option for
a future RTLS publication to revisit THO hosting if a successor
Subscription Events terminology is reintroduced. Patient Administration
should proceed with the block vote on the existing resolution.
