## Summary

The reporter is working with the FHIR `Group` resource as the basis for
representing **non‑patient‑centric care teams** in the NDH IG — i.e., a
`Group` whose members collectively deliver a defined bundle of services.
They observe that, as currently profiled (in NDH), `Group.code` is being
used to convey the **category of services** the group offers, but they
argue `Group.code` is semantically meant to describe the **type of the
group (or the type of its members)**, not the services rendered. They
propose two coordinated changes: (1) add an **extension** on `Group` to
carry the services the group provides, bound to the **HealthcareService
category** value set; and (2) bind `Group.code` to the
**NDHCareTeamCategoryVS** value set
(`http://hl7.org/fhir/us/ndh/ValueSet/NDHCareTeamCategoryVS`) so that
`Group.code` indicates the type/category of the group members. No other
Jira tickets are cross-referenced to this request, so the scope is
self-contained to NDH's `Group`-based care-team modelling.

## Details

**Description:**

> We intend to use the **Group** resource for non‑patient‑centric care
> teams, where the Group represents a set of individuals who collectively
> deliver a defined set of services. Currently, the value set for
> **Group.code** represents the category of services that the group
> provides. However, **Group.code** is more appropriately used to
> indicate the *type* of group or group member rather than the specific
> services they deliver.
>
> **Suggested changes:**
>
> 1. **Add an extension** to represent the specific services the group
>    provides, using the **HealthcareService category value set**.
> 2. Use `http://hl7.org/fhir/us/ndh/ValueSet/NDHCareTeamCategoryVS` to
>    indicate the type or category of group members.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the NDH IG, retarget the `Group` profile used for non‑patient‑centric
care teams as follows:

- **Rebind `Group.code`** on the NDH care-team `Group` profile to
  `http://hl7.org/fhir/us/ndh/ValueSet/NDHCareTeamCategoryVS`, so that
  `Group.code` conveys the **type/category of the group (and its
  members)** rather than the services delivered. Choose a binding
  strength consistent with neighboring NDH profiles (likely `extensible`
  to allow jurisdictional concepts).
- **Define a new NDH extension** (e.g.,
  `http://hl7.org/fhir/us/ndh/StructureDefinition/group-services-provided`)
  on `Group` of type `CodeableConcept` (cardinality `0..*`), bound to the
  **HealthcareService category value set**
  (`http://hl7.org/fhir/ValueSet/service-category`, the same VS used by
  `HealthcareService.category`). Document its meaning as "the categories
  of services that this group of providers collectively delivers."
- Update the NDH care-team `Group` profile, narrative, examples, and any
  CapabilityStatement entries to reference the new binding and the new
  extension; update profile guidance text to clarify the split between
  *what the group is* (`Group.code`) and *what the group does*
  (extension).

#### Justification

The reporter's semantic analysis is consistent with the base FHIR
intent: in core FHIR, `Group.code` describes the kind of group/members
(its example binding is a small "group type" set). Categorising the
*services* the group delivers is properly the domain of
`HealthcareService.category`, so reusing that value set behind a
purpose-built extension on `Group` keeps NDH aligned with mainstream
FHIR patterns and avoids overloading `Group.code`. Putting the
NDH-specific concepts behind `NDHCareTeamCategoryVS` also localises the
NDH vocabulary commitments without changing core FHIR.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a new extension on `Group`, model the "services
delivered by the care team" relationship through **existing
HealthcareService linkage** instead:

- Keep `Group.code` rebound to `NDHCareTeamCategoryVS` (per the
  reporter's intent for type/category of the group).
- Represent the services the group delivers by **referencing
  `HealthcareService` instances** whose `providedBy`/`HealthcareService`
  associations point at the providers in the group, or by introducing a
  small extension that simply **references one or more
  `HealthcareService` resources** rather than carrying coded categories
  inline. NDH already models `HealthcareService` first-class, so this
  reuses an existing artifact instead of duplicating its category coding
  on `Group`.
- Provide IG guidance describing the canonical join (Group ↔ members ↔
  `HealthcareService.providedBy`/location) so implementers can answer
  "what services does this group deliver?" via a query against
  `HealthcareService` rather than via a code on the group.

#### Justification

NDH already invests heavily in `HealthcareService` for service modelling.
Adding a coded "service category" on `Group` risks **drift** between the
group's coarse category list and the precise `HealthcareService`
records, and creates two competing answers to "what does this group do?"
A reference-based extension (or pure linkage) keeps the source of truth
in `HealthcareService` and limits new artifacts to the rebinding the
reporter is really asking for. The trade-off is that a quick "what kind
of services does this group deliver?" lookup now requires resolving
`HealthcareService` references rather than reading a code directly on
the `Group`, which may complicate simple consumers.

---

### Disposition C: Decline

#### Proposal

Decline both halves of the request and keep the current NDH `Group`
profile binding for `Group.code` unchanged. Document in the IG narrative
that, in NDH, `Group.code` is being used (intentionally) to convey the
**service category** of a non-patient-centric care team, and direct
implementers seeking "type of group" semantics to existing mechanisms
(e.g., `Group.type`, member typing, or profile selection).

#### Justification

`Group.code` is `0..1` and binding-flexible; NDH may have deliberately
overloaded it for service categorisation to avoid an extension. Changing
both the binding **and** introducing a new extension is a breaking
change for any early NDH implementers and any test data already aligned
with the current value set. If the working group judges that the
existing pattern is "good enough" — and that semantic purity does not
outweigh implementer churn — declining preserves stability. (This is a
weak position given the reporter's argument is technically sound; it
would mainly be defensible if there is evidence of significant
deployment against the current binding.)

---

### Recommendation

**Recommended disposition:** A

The reporter's modelling argument is correct: in core FHIR `Group.code`
is about *what kind of group this is*, not *what services it delivers*,
and the current NDH binding overloads the element. Disposition A keeps
NDH semantically aligned with base FHIR, localises NDH's
care-team-typing vocabulary in `NDHCareTeamCategoryVS`, and uses the
established `HealthcareService` category value set for the
"services-delivered" facet — all without requiring consumers to chase
references for a simple coarse classification. Disposition B is more
parsimonious but pushes a non-trivial query burden onto every consumer
that just wants to know what a care team broadly does, and Disposition C
is hard to defend on FHIR-modelling grounds. Given that the ticket is
still **Triaged** with no comments and no recorded Zulip/GitHub
activity, PA should confirm with NDH authors whether existing
implementations are bound tightly to the current value set before
finalising the binding strength on the rebinding.
