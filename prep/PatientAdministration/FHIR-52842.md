## Summary

Isaac Vetter (Epic) requests that the *Interoperable Digital Identity
and Patient Matching (FHIR)* IG strengthen its Recommended Best
Practices for patient identity matching so that they more closely
match the long-deployed weighted/fuzzy matching rules Epic uses across
XCPD, v2 and FHIR interfaces. Specifically, Epic considers two patient
records a confident match only when **Exact (or near-exact) Name +
Exact Sex/Gender + Exact DOB + at least one additional identifier**
(full address, exact phone, exact email, SSN/last‑4, or MRN/other
unique ID). The IG today requires one fewer mandatory demographic
(notably it does not require sex/gender) and gives MRN/identifier
heavier weight than Epic's deployed algorithms do. The reporter also
raises a related concern — current production matching has no discrete
notion of "verified demographics", so even an asserted certainty flag
on demographics would not, by itself, allow Epic to relax its default
matcher. The ask is to (a) add Sex/Gender to the minimum demographic
set in the Best Practices guidance and (b) re-tune the weighting in
the Patient Weighted Input Information table to align with widely
deployed practice.

A workgroup-side technical correction to the same Best Practices /
Weighted Inputs tables is already in flight as
[FHIR-52857](https://jira.hl7.org/browse/FHIR-52857) (referenced from
the comments of this ticket); that ticket only fixes a narrative
inconsistency in the first row of the Recommended Best Practices table
and the wording of the Score‑4 row in §3.5, so it does not, on its
own, satisfy this request.

## Details

**Description:**

At Epic, we've been refining standard weighted, fuzzy matching and
related functionality for the use of demographics to do patient
matching for decades at scale across a wide variety of interop
mechanisms — XCPD, v2 and other interfaces, and now FHIR as well.

The best practices recommended in the Identity IG are less than our
minimum requirements that we default and in some (many) cases require
for health system integrations.

Specifically, Epic's general patient matching guidelines are that a
pair of patients can be confidently considered a match if **all** of
the following are met:

- Exact Name or Near Exact Name match
- Exact Sex/Gender match
- Exact DOB match
- At least one other thing, which might be any of the following:
  - Full Address match
  - Exact Phone match
  - Exact Email match
  - SSN match or last 4 SSN match
  - MRN or other unique identifier, like member ID

Whereas the Identity IG requires one fewer mandatory demographic (not
sex/gender) and places a greater weight on mrn/id.

Additionally (and as we discussed during the FAST connectathon
breakout during the Sept connectathon!) — current state production
patient matching doesn't have a discrete concept of certainty of
demographics verification. Does this certainty enable the patient
matching to use fewer discrete pieces of demographics?

I'm not sure what or if there are formal expectations for demographic
certainty in, for example, XCPD.

It would be a struggle for Epic to relax our default patient matching
algorithms merely because there was now a discrete assertion that the
demographics were current as of the collection date per a driver's
license, for example (shouldn't that already be the case today?).

Please consider adding sex/gender and look for opportunities to align
with the above very widely deployed patient matching algorithm.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the *Interoperable Digital Identity and Patient Matching (FHIR)*
IG so that the Recommended Best Practices in §3.4 require **Sex/Gender**
as part of the minimum demographic set, alongside Name and DOB, and
re-tune the §3.5 Patient Weighted Input Information table so that
identifier-class inputs (MRN/Enterprise Identifier) do not, on their
own, dominate the weighted score relative to demographic agreement.
Concretely:

- In §3.4, change the Minimum Included Attributes for a "confident
  match" row to require First Name, Last Name, **Sex/Gender**, DOB,
  **plus at least one** of: full address, exact phone, exact email,
  SSN (or last‑4 SSN), or unique enterprise/member identifier — i.e.,
  the Epic rule set quoted in the description.
- In §3.5, lower the weight assigned to identifier-only matches
  (Score‑4 row clarified by FHIR-52857) so that an identifier match
  alone, without supporting demographics, does not produce a confident
  match score; and add a non-zero weighted input row for Sex/Gender
  agreement.
- Cross-link from the §3.4 narrative to the verification requirements
  in §3.5 so it is clear that the additional demographics still must
  be "verified" per the IG's existing best-practice criteria.

#### Justification

Aligning the IG with a widely deployed weighted matching algorithm
(Epic, XCPD, v2 patient demographics queries) reduces the gap between
"what the standard recommends" and "what production systems actually
do", which is the stated purpose of a Best Practices section. Industry
alignment is a recognized FHIR design principle for IGs whose audience
is implementers running at health-system scale. Sex/Gender is one of
the most reliably collected and exchanged patient demographics in
existing US clinical data, so adding it to the minimum set is low
implementation cost and high signal.

---

### Disposition B: Alternative Approach — add Sex/Gender as a *bonus*
weighted input and add a "verified demographics" qualifier without
changing the minimum set

#### Proposal

Keep the current Best Practices minimum demographic set unchanged
(Name + DOB + Enterprise Identifier, per FHIR-52857), but:

1. Add **Sex/Gender agreement** as an additional row in the §3.5
   Patient Weighted Input Information table with a small positive
   weight (lower than DOB or identifier), so that systems whose
   matchers — like Epic's — treat sex as material can score it
   without the IG mandating it for systems that do not.
2. Introduce a normative concept of **"verified demographic"** in
   §3.5 (e.g., demographics whose source is an authoritative identity
   document such as a driver's license / passport / payer ID), and
   give a verified demographic a higher weighted contribution than the
   same demographic when unverified. Explicitly state that, even with
   verified demographics, the §3.4 minimum demographic set is still
   required — i.e., verification *increases* confidence but does not
   *substitute* for additional matching attributes. This addresses
   Isaac's concern that "I'd struggle to relax matching just because
   the demographics were asserted current".
3. Leave FHIR-52857's narrative fix in place to harmonize the §3.4
   table with §3.5.

#### Justification

This avoids a hard breaking change to the Best Practices minimum
(which has been ratified by identity SMEs as recently as the special
session referenced by juliemaas) while still moving the IG toward
Epic's expectations on two of the three substantive concerns: it lets
implementations that rely on Sex/Gender express that in IG-conformant
scoring, and it pre-empts any future "verified demographics" loophole
by making it additive to — not a substitute for — the minimum set. It
also keeps the Best Practices guidance defensible against the SME
position that Sex/Gender is not generally identity-evidence verifiable
and so should not be *required*. Trade-off vs. Disposition A: Epic and
similar implementers will still see a gap between IG and their
deployed minimum, but the gap is closed in scoring rather than in
mandatory inputs.

---

### Disposition C: Decline

#### Proposal

Decline the request to add Sex/Gender to the minimum demographic set
or to materially re-weight identifier inputs. Rationale category:
**already addressed by existing mechanism + insufficient new
evidence** — the IG's match-weighting guidance was finalized in a
dedicated SME session whose consensus explicitly considered and
rejected Sex/Gender as instrumental for identity resolution and not
generally verifiable with identity evidence; the §3.4 narrative
inconsistency the reporter is reacting to is being fixed by
FHIR-52857. Recommend closing FHIR-52842 as *Persuasive with
Modification → no IG change*, with the modification being the
FHIR-52857 narrative correction.

#### Justification

Two of juliemaas's points carry significant weight in an Identity IG:
(1) Sex/Gender is not generally verifiable from identity evidence
(driver's license, passport, payer ID), and the IG's whole rubric is
built on verifiable inputs; and (2) MRN scoring already requires a
documented assignment process and verification, which the IG calls
out. Without new connectathon data or implementer evidence beyond the
single reporter's stance, restructuring the minimum set risks
de-stabilizing guidance that other implementers have already aligned
to. The reporter's "verified demographics shouldn't let you relax
matching" concern is, on this reading, already implicit in the IG
because §3.4 already encourages including all known demographics.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible middle path for a Best Practices
section in an interoperability IG. Disposition A would force a
mandatory-input change that the WG's identity SMEs have already
deliberated on and rejected (per juliemaas's comments), so it is
likely to be re-litigated and rejected again on the same grounds —
"Sex/Gender is not identity-evidence verifiable" — without resolving
Epic's underlying concern that production matchers will not be able to
follow the IG to the letter. Disposition C closes the ticket without
giving Epic or similarly-positioned implementers any IG-conformant way
to express their existing Sex/Gender weighting, which is exactly the
"IG ≠ deployed reality" gap the ticket is calling out. Disposition B
keeps the SME-approved minimum, but (i) gives Sex/Gender a real,
optional weighted-input row so deployments that score it can still be
IG-conformant, and (ii) introduces an explicit "verified demographics
do not substitute for additional matching attributes" statement in
§3.5, which directly answers Isaac's connectathon-breakout question
and prevents a future loophole. It also composes cleanly with the
FHIR-52857 technical correction rather than colliding with it.
