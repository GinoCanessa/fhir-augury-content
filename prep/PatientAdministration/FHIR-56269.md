## Summary

The reporter has reviewed the dependency list declared by the National
Directory of Healthcare Providers and Services (NDH) IG and is asking the
Patient Administration work group to confirm which of the eight declared
dependencies are still actually required by the IG. Two dependencies are
called out as suspect:

- `hl7.fhir.us.davinci-cdex` (2.1.0) — while building validation support
  for the IG, the reporter could not find any artifact in NDH that
  actually references a CDex artifact, suggesting the dependency may be
  unused.
- `hl7.fhir.us.davinci-pdex-plan-net` (1.2.0) — NDH was historically
  aligned with PDex Plan-Net but is described as no longer aligning. The
  reporter assumes the dependency is currently retained to support
  mapping references in NDH StructureDefinitions and asks whether that is
  still a sufficient reason to keep it.

The remaining six dependencies (`hl7.fhir.r4.core` 4.0.1,
`hl7.terminology.r4` 7.1.0, `hl7.fhir.us.core` 6.1.0,
`hl7.fhir.uv.extensions.r4` 5.2.0,
`hl7.fhir.uv.subscriptions-backport.r4` 1.1.0, and `ihe.iti.mcsd` 4.0.x)
are presented for context but not flagged. The desired outcome is a
publisher-time `dependencies` block in the NDH IG that contains only the
packages NDH actually uses, removing dead weight that slows IG builds,
inflates the package's transitive surface, and confuses implementers
about which other IGs they must implement to be NDH-conformant.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a confirmation request and act on it: in the NDH IG's
`sushi-config.yaml` (or equivalent IG configuration), remove the two
dependencies the reporter flagged as unused —
`hl7.fhir.us.davinci-cdex` and `hl7.fhir.us.davinci-pdex-plan-net` —
after re-running an IG build to verify there are no broken references
(no canonical URLs from those packages are resolved by NDH profiles,
extensions, ValueSets, examples, narratives, or `mapping` blocks). The
remaining six dependencies (`hl7.fhir.r4.core` 4.0.1,
`hl7.terminology.r4` 7.1.0, `hl7.fhir.us.core` 6.1.0,
`hl7.fhir.uv.extensions.r4` 5.2.0,
`hl7.fhir.uv.subscriptions-backport.r4` 1.1.0, `ihe.iti.mcsd` 4.0.x) are
kept as-is. Update the NDH change-log / dependencies page to record the
removals and any conformance impact.

#### Justification

The reporter's analysis (validation-build trace shows no NDH artifact
referencing a CDex artifact, and NDH no longer aligns with PDex
Plan-Net) is the kind of evidence an IG editor needs to drop a
dependency safely. A leaner dependency list reduces IG-publisher build
time, shrinks the transitive package set implementers have to load
into validators, and removes the implicit "you must also read these
IGs" signal that an unused dependency sends to the reader. Acting on a
documented review at Triaged status is exactly the maintenance posture
that keeps an IG's package metadata trustworthy across publications.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `hl7.fhir.us.davinci-pdex-plan-net` as a declared dependency
because the NDH StructureDefinitions carry `mapping` elements that
reference Plan-Net canonicals; remove only `hl7.fhir.us.davinci-cdex`.
Additionally, document the policy explicitly in the NDH IG: a
dependency is retained when (a) any NDH conformance artifact resolves
a canonical from that package, **or** (b) NDH publishes
`StructureDefinition.mapping` entries that target that package's
identity URI and the editors want the IG publisher to validate those
mapping URIs. Where the only purpose of a dependency is to anchor
mapping URIs, add a comment in `sushi-config.yaml` next to the entry
recording that fact so future reviewers don't repeat this
investigation.

#### Justification

The reporter's own framing ("I am assuming the dependency is to support
the mappings in the structure definitions") suggests Plan-Net is not
provably dead — it may still be load-bearing for the IG publisher's
mapping resolution and for human readers who follow the mapping links.
Dropping it could silently break those cross-walks even if no
validation error is produced. CDex, by contrast, has a concrete
"no artifact reference found" finding behind it and is the safer of
the two to remove. This split disposition matches the evidence we
have on each dependency rather than treating them as one decision, and
the documented policy prevents the same review having to be redone in
the next ballot cycle.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as **Persuasive with Modification — Not Persuasive
to Remove**: leave the current dependency list intact for this
publication and instead annotate the IG's dependencies page describing
why each dependency is present (validation, terminology, mapping
target, cross-IG narrative reference, etc.). Defer any actual removal
to a future, scoped "dependency cleanup" ticket once a full reference
audit (canonical resolution + mapping URI usage + narrative links) has
been performed across all NDH artifacts.

#### Justification

Removing a published IG dependency is a non-trivial change: it can
break downstream IGs that transitively relied on NDH pulling those
packages in, and it can invalidate examples and validator tooling
that currently load the larger dependency closure. Without a complete
audit (not just a validation-build trace) the work group risks
discovering after publication that an artifact, mapping, or example
*did* reference a removed package. Documenting the rationale for each
dependency now is low-risk, addresses the reporter's underlying
concern (transparency about why each package is listed), and lets the
removal happen later under a dedicated ticket with explicit before/
after artifact lists.

---

### Recommendation

**Recommended disposition:** B

The reporter's investigation cleanly distinguishes the two suspect
dependencies. CDex has a concrete negative finding ("no artifact from
CDex is referenced") that justifies removal, while Plan-Net is
explicitly hedged ("I am assuming the dependency is to support the
mappings"). Treating them as one yes/no decision (Disposition A) over-
applies the CDex evidence to Plan-Net; refusing to act at all
(Disposition C) ignores a credible, low-risk cleanup of a dependency
the IG demonstrably does not use. Disposition B captures the actual
state of the evidence, removes the dependency that is provably unused,
and pairs the retained Plan-Net dependency with a documented
"why it's here" policy so the next reviewer doesn't re-open this
question. The work group should ask the editors to (1) confirm there
are no remaining CDex canonical references in NDH artifacts before
removing CDex, and (2) record the mapping-target rationale for
Plan-Net in `sushi-config.yaml` and on the dependencies page.
