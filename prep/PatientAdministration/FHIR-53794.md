## Summary

The reporter notes that the NDH (National Directory of Healthcare Providers
and Services) IG's [Security Conformance section](https://build.fhir.org/ig/HL7/fhir-us-ndh/conformance.html#security-conformance)
states a SHALL-level requirement for security (specifically SMART Backend
Services Authorization, and additionally FAST Security / SSRAA), but the
NDH Server and Client CapabilityStatements do not reflect those security
requirements at the designated location, [`CapabilityStatement.rest.security`](https://hl7.org/fhir/R4/capabilitystatement-definitions.html#CapabilityStatement.rest.security).
The ticket asks whether the NDH CapabilityStatements should formally
declare those SHALL-level security expectations and, if so, how. The
underlying issue is a more general one — there is no settled FHIR
convention for how an IG-level "Requirements" CapabilityStatement
expresses a dependency on SMART (especially SMART Backend Services) and
on FAST/UDAP — and that ambiguity is what is currently being debated on
Zulip. The cross-references for this ticket point at exactly that
question: the FHIR element `CapabilityStatement.rest.security`, and the
`smart > other IGs using SMART` Zulip thread that John Moehrke opened
to chase the same concern.

John Moehrke opened the thread (2026-01-19) asking how an IG that calls
upon SMART for security should declare that requirement in a

`SMART-on-FHIR` code and does not differentiate SMART Backend Services

capabilities — authentication mechanisms, scopes — through the SMART

at the UDAP Security IG's "Required UDAP Metadata" section as a model.
John Moehrke (2026-02-12) clarified that those answers address the
*Instance* CapabilityStatement question ("what security does THIS
server implement") and not his actual question, which is how an IG
declares the *requirement* of conformance — i.e., how a Requirements
CapabilityStatement expresses "implementers SHALL support SMART Backend
Services". The thread therefore ends without consensus on a concrete
mechanism, with the open question explicitly distinguishing
Requirements vs. Instance CapabilityStatements.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the NDH IG so that both the NDH Server and NDH Client
CapabilityStatements populate `CapabilityStatement.rest.security` to
reflect the IG's SHALL-level security requirements:

- Set `rest.security.cors` according to the IG's existing CORS guidance
  (true if required by the security section).
- Add `rest.security.service` codings for each declared security
  mechanism. Use `http://hl7.org/fhir/restful-security-service#SMART-on-FHIR`
  for the SMART dependency (matching the precedent set by
  SDOHC-clinicalcare), and add an additional coding for FAST/UDAP using
  the appropriate code from the FAST IG (or a locally defined code in an
  NDH-owned CodeSystem if no normative code exists yet) so that the FAST
  Security (SSRAA) SHALL is also visible at the canonical location.
- Populate `rest.security.description` with narrative that explicitly
  cross-links to the IG's [Security Conformance](https://build.fhir.org/ig/HL7/fhir-us-ndh/conformance.html#security-conformance)
  section and names SMART Backend Services Authorization and FAST
  Security as the bound requirements.
- On each entry, attach the `capabilitystatement-expectation` extension
  with value `SHALL` so the requirements semantics are unambiguous in
  the Requirements CapabilityStatement.

#### Justification

`CapabilityStatement.rest.security` is the canonical location for
declaring security expectations of a conforming endpoint, and the IG
already states the requirement at SHALL level in prose. Mirroring that
SHALL in the machine-readable artifact is consistent with FHIR's
"narrative + structured data must agree" principle, supports tooling
(IG validators, conformance dashboards) that machine-checks
CapabilityStatements, and makes the NDH artifacts internally
consistent — which is exactly the consistency concern John Moehrke
raised in the ticket comments. SDOHC-clinicalcare provides direct
precedent for using `rest.security.service = #SMART-on-FHIR`, and the
`capabilitystatement-expectation` extension is the standard mechanism
for marking SHALL/SHOULD on Requirements CapabilityStatement entries.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) populating
`CapabilityStatement.rest.security.service`, declare the IG dependency
itself: list the SMART App Launch IG (and, where relevant, a SMART
Backend Services profile or the FAST/UDAP Security IG) on the
CapabilityStatement using the `CapabilityStatement.implementationGuide`
element, with the `capabilitystatement-expectation` extension set to
`SHALL` on each entry — following the davinci-epdx precedent that John
Moehrke called out. Use `rest.security.description` as narrative pointing
at the IG's Security Conformance section, and leave
`rest.security.service` either empty or set only to the generic
`SMART-on-FHIR` code (since no published code distinguishes SMART
Backend Services from SMART App Launch). Pair the change with a Zulip
note back to `smart > other IGs using SMART` flagging that NDH took
this approach and asking the SMART WG to define a more granular
`restful-security-service` code for SMART Backend Services so that
Disposition A becomes feasible later.

#### Justification

This addresses the *known gap* identified in the cross-referenced
Zulip thread: there is no SMART-WG-blessed way today to say "SMART
Backend Services SHALL" via `rest.security.service`, because the
existing code only says "SMART-on-FHIR". Forcing a locally-coined
coding into `rest.security.service` (Disposition A's FAST coding) risks
embedding NDH-specific terminology in a slot meant for cross-IG
interoperability and would need to be re-worked once the SMART/FAST
WGs agree on canonical codes. Declaring the dependency at the
`implementationGuide` level is already an established FHIR pattern
(davinci-epdx), is unambiguous about the SHALL, and avoids creating
NDH-specific security service codes. The trade-off vs. Disposition A
is reduced granularity (a tool that only inspects
`rest.security.service` won't see the requirement), but this is offset
by the narrative pointer in `rest.security.description` and by the
fact that Disposition A doesn't actually achieve full granularity
either (no code distinguishes Backend Services).

---

### Disposition C: Decline

#### Proposal

Decline to add structured security requirements to the NDH
CapabilityStatements at this time, on the grounds that the request is
blocked on cross-WG infrastructure that does not yet exist (no
`rest.security.service` code for SMART Backend Services; no settled
convention for representing FAST/UDAP requirements in a Requirements
CapabilityStatement). Resolve the ticket as "Persuasive with
Modification — Not Persuasive for now / Pending Dependent Change" and
record the underlying need against the cross-reference thread
`smart > other IGs using SMART`, with a follow-up tracker against the
SMART and Security WGs to define the missing
`restful-security-service` codes and a canonical
`capabilitystatement-expectation` pattern. In the meantime, the IG's
prose Security Conformance section continues to carry the SHALL.

#### Justification

The Zulip thread shows that even SMART-aware contributors (Josh Mandel,
Grant Smith) do not have a single agreed answer to John Moehrke's
question, and John has explicitly noted that the available tooling
answers a different question (Instance vs. Requirements
CapabilityStatement). Encoding NDH-local conventions before the
upstream pattern stabilises risks creating artifacts that have to be
re-worked once SMART/FAST/UDAP WGs publish guidance, and risks
divergence from any future US Core / Da Vinci / SDOH approach. The
ticket reporter is asking a genuine question, but the right venue for
the answer may be the SMART WG, not NDH.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because it is the only option that both
(a) actually closes the consistency gap the reporter raised — the
NDH CapabilityStatement will visibly declare its SHALL-level dependence
on SMART (and FAST), and (b) does so without minting NDH-local codes
into `rest.security.service`, which the cross-referenced
`smart > other IGs using SMART` thread shows is a known unresolved
problem at the FHIR-core level. Using `CapabilityStatement.implementationGuide`
plus `capabilitystatement-expectation` follows an established Da Vinci
precedent, requires no new code systems, and is forward-compatible: if
and when SMART WG defines a `SMART-Backend-Services` security service
code, the NDH IG can additionally populate `rest.security.service` in
a future version without invalidating today's choice. Disposition A is
attractive in principle but is partly speculative on the FAST coding
and over-claims granularity that the underlying terminology does not
yet support; Disposition C accurately describes the upstream gap but
leaves the original NDH internal inconsistency (prose SHALL vs. silent
CapabilityStatement) unresolved, which is exactly what the reporter
asked the WG to fix.
