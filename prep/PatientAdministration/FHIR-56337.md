## Summary

The reporter argues that two NDH-defined extensions currently slotted on the
NDH Verification profile — **"CMS Enrollment in Good Standing"** and
**"NDH Aligned with CMS Data Network"** — are misplaced. They describe
attributes of the *entity being verified* (a Practitioner or Organization),
not metadata about the *verification act* (who verified, when, and how).
Because the Verification resource is intended to capture provenance and trust
information about a verification event, carrying business or regulatory
status of the underlying entity on Verification conflates two different
modelling concerns. The ticket therefore asks that these two extensions be
removed from the Verification profile and instead be made applicable to the
Practitioner and Organization profiles in NDH.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by the FHIR Augury index for `FHIR-56337`, so the request stands on
its own as filed.

## Details

**Description:**

> The "CMS Enrollment in Good Standing" and "NDH Aligned with CMS Data
> Network" extensions represent attributes of the underlying entity (e.g.,
> Practitioner or Organization) rather than metadata about the verification
> process.
>
> The Verification resource is intended to capture provenance and trust
> information (who verified, when, how), not to carry business or regulatory
> status of the entity itself.
>
> Therefore, these extensions appear more appropriately applied to
> Practitioner and Organization, rather than to Verification.

## Proposed Dispositions

> Note: Because no `repo-analysis` briefing was loaded for `HL7/fhir-us-ndh`,
> the dispositions below name the conceptual artifacts the ticket calls out
> (the NDH Verification, Practitioner, and Organization profiles, and the two
> extension definitions) without asserting specific authoring file paths.

### Disposition A: Accept As Requested

#### Proposal

Remove the two extensions from the NDH Verification profile and re-scope
them so they apply to the entity being described:

1. In the Verification profile, drop the references to the
   `CMS Enrollment in Good Standing` extension and the
   `NDH Aligned with CMS Data Network` extension (no `obeys`/`contains`
   slice on Verification's extension element for these URLs).
2. In each extension's `StructureDefinition`, change the `context` from
   Verification to the entity types that actually carry these attributes —
   `Practitioner` and `Organization` (and, if NDH also profiles
   `PractitionerRole` / `OrganizationAffiliation` for these statuses,
   include those as additional contexts).
3. In the NDH Practitioner and Organization profiles, add the two
   extensions as optional (0..1) elements so implementers have a defined
   slot to populate them.
4. Update the IG narrative pages that describe Verification, Practitioner,
   and Organization to reflect the relocation, and add a migration note in
   the change-log / "what's new" section explaining that these extensions
   moved off Verification.

#### Justification

This matches the FHIR modelling convention that a *resource describes the
thing it is about* and that *extensions live on the resource whose instance
data they characterise*. Verification (in the family of provenance-style
resources) is meant to carry "who attested to what, when, and on what basis"
— not to restate properties of the subject. "Enrollment in Good Standing"
and "Aligned with CMS Data Network" are durable status facts of a
Practitioner/Organization that exist independently of any single
verification event, so they belong on the entity. Putting them on
Verification also creates a query/aggregation problem: a consumer asking
"is this provider in good standing?" would otherwise have to chain through
Verification rather than read the entity directly. The ticket's framing is
consistent with how analogous status flags are handled elsewhere in NDH and
in US Core (entity attributes on the entity, provenance on Provenance /
Verification). The change is also low-risk relative to NDH's current ballot
maturity — the extensions are NDH-defined, so HL7 Patient Administration /
NDH owns the context list and can re-target them without coordinating with
an outside extension publisher.

---

### Disposition B: Alternative Approach

#### Proposal

Allow the extensions to be used on **all three** contexts — Verification,
Practitioner, and Organization — rather than relocating them outright.
Concretely:

1. Update each extension's `StructureDefinition` `context` to list
   `Practitioner`, `Organization`, **and** `Verification` (and, if
   applicable, `PractitionerRole` / `OrganizationAffiliation`).
2. Add the extensions as optional slots on the NDH Practitioner and
   Organization profiles.
3. Leave the existing usage on the Verification profile in place but
   downgrade it to optional (0..1) and add narrative guidance: when used on
   Verification, the extension records the value *as observed at the moment
   of verification* (a point-in-time snapshot); when used on Practitioner /
   Organization, it records the *current* value.
4. Add an invariant or guidance note that consumers should prefer the
   value on the entity for "current state" queries.

#### Justification

This preserves backward compatibility with any early implementers who have
already populated the extensions on Verification, while still giving
Practitioner and Organization a first-class home for the data. It also
acknowledges a real, narrower use case: a verification record may want to
freeze "what CMS said at the time we checked" so that later changes to the
entity don't rewrite history. The trade-off is that having the same
extension live in two contexts with subtly different semantics is a known
source of implementer confusion, and it complicates conformance testing and
any future Subscription / change-detection logic. Disposition B is
preferable to A only if the workgroup hears from implementers that the
point-in-time use on Verification is actually being relied on; otherwise the
modelling cost outweighs the compatibility benefit.

---

### Disposition C: Decline

#### Proposal

Decline the request. Keep the two extensions defined with `Verification` as
their context, on the basis that within NDH the act of *verifying* a
provider's CMS enrollment status and CMS-data-network alignment *is* the
business event being recorded, and the extension values are meaningful only
in the context of that verification event.

#### Justification

A defence of the current design rests on reading the two extensions
narrowly: they aren't claiming "this Practitioner is in good standing" in
the abstract; they're claiming "as part of this verification activity, we
checked CMS and recorded this result." Under that reading, the extensions
are attestation outputs, not entity attributes, and Verification is exactly
the right host. This disposition is the weakest of the three because the
extension *names* read as entity attributes ("CMS Enrollment in Good
Standing"), not as verification outcomes ("Result of CMS enrollment
check"), and because consumers asking "is provider X in good standing right
now?" would still have no good place to look. It is included for
completeness so the workgroup can explicitly weigh and reject the
"do nothing" path.

---

### Recommendation

**Recommended disposition:** A

The ticket identifies a real modelling defect: the two extensions name and
describe attributes of the Practitioner / Organization, not properties of a
verification event. Leaving them on Verification forces every consumer
that wants the *current* status to traverse Verification instances and
reason about which one is "latest", which is exactly the pattern FHIR
asks IGs to avoid for stable entity attributes. Disposition B's
backward-compatibility argument is weak at NDH's current maturity and
introduces dual-context semantics that will create more implementer
confusion than they solve; if anyone has populated these extensions on
Verification today, a one-time migration note is a fair price for a
cleaner long-term model. Disposition C is hard to defend given the
extensions' own naming. The workgroup should accept the request, relocate
the extension contexts to Practitioner and Organization (and any relevant
role/affiliation profiles), and add a clear change-log entry so early
implementers can update.
