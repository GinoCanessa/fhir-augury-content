## Summary

The ticket asks the Patient Administration WG to clarify how the
`Patient/$match` operation behaves — and how the returned `count`
parameter should be interpreted — when a server holds **multiple
`Patient` resource instances that all represent the same real-world
human**. The reporter (John Moehrke) calls out two cases: (a) servers
where the duplicate `Patient` resources are joined via `Patient.link`,
and (b) servers where they are not. The motivating concrete scenario is
an HIE that sits in front of multiple EHRs, each of which has its own
`Patient` for the same person; a `$match` against the HIE can therefore
legitimately return one `Patient` per backing EHR for a single human.

The ticket does not propose specific spec text; it raises a definitional
question about the semantics of `$match` results in this duplicate-human
situation, and implicitly whether the `count` (`MatchGrade`-weighted
score) and the result Bundle composition need to be clarified to handle
"N Patient resources, 1 human" correctly.

This ticket is a companion to **[FHIR-55789](https://jira.hl7.org/browse/FHIR-55789)**
("multiple instances of Patient for one Patient"), filed by the same
reporter on the same day against the IDI / Patient Matching
specification. FHIR-55789 targets the `Patient.link` narrative on
`patient.html#links`, arguing the existing prose pushes one perspective
(linked Patients) and that the spec should be clearer that returning
more than one Patient from `$match` may simply mean the server found
one human represented by multiple Patient instances. Together, the two
tickets ask the WG to align the `Patient.link` guidance and the
`$match` operation definition around the same concept: a "match" is on
the **human**, not necessarily on a single `Patient` resource. The
related-by-keyword corpus also surfaces several adjacent open questions
about `$match` (FHIR-55759 "Is IDI Match an extension of `$match` or
not", FHIR-9539 "Make patient-mpi-match StructureDefinition more
generic", FHIR-24678 inactive-patient guidance, FHIR-51538 IDI-match
input/output definitions), suggesting the WG should consider this
ticket as part of a broader `$match` clarification thread rather than
as an isolated edit.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an "Interpretation when multiple Patient resources represent the
same human" subsection to the `Patient/$match` operation definition
page (`patient-operation-match.html`), and add a matching forward
reference from `patient.html#links`. The new subsection should
state, normatively where possible:

1. Each entry in the result Bundle is one **candidate `Patient`
   resource**; servers SHALL NOT silently collapse multiple `Patient`
   instances that represent the same human into one entry.
2. When a server knows two or more returned `Patient` entries
   represent the same human, it SHOULD link them — either via
   `Patient.link` between the returned Patients themselves, or via a
   `Person` resource (also returned in the Bundle, with `Person.link`
   targets pointing at the candidate Patients) — so the client can
   reconstruct the "one human, N Patient" relationship.
3. The output `count`/`MatchGrade` semantics: `count` is the number
   of `Patient` **resources** returned, not the number of distinct
   humans the server believes it found. Implementers SHOULD use
   `MatchGrade.certain` per matched human (typically once per linked
   group) and `MatchGrade.probable`/`possible` for the additional
   members of that group, or use `Person`-based grouping for an
   explicit signal. (Exact rule to be picked by the WG; the point is
   to specify *something*.)
4. Add a non-normative example showing the HIE-in-front-of-many-EHRs
   scenario from the ticket description, with a result Bundle
   containing N Patient entries plus one Person entry, mirroring the
   pattern Steve Atwood proposed in the Zulip thread.

#### Justification

The reporter has flagged a definitional gap, not a bug. The
Zulip thread shows the question has been live in the implementer
community since 2020 without authoritative guidance, and a senior WG
voice (René Spronk) has already reasoned by analogy from FHIR Search
that "include extra resources" is in the spirit of the spec but is
not actually written down. Writing it down — together with a clear
rule for `count` and a worked HIE example — directly resolves
FHIR-55791, gives FHIR-55789 a place to point at for the matching
clarification on `Patient.link`, and creates a stable target that
FHIR-55759 (IDI-match relationship) and FHIR-51538 (IDI-match output
definition) can be re-litigated against. It costs only operation-page
prose and one example; no schema, no profile, no breaking change.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this as a **modelling clarification** rather than an
operation-page edit, and resolve it by amending `Patient.link`
narrative (the change FHIR-55789 is already proposing) plus a
**single sentence cross-link** from `patient-operation-match.html` to
that narrative. Concretely:

- In `patient.html#links`, soften the current text and add an
  explicit statement that "a query or operation MAY return more than
  one `Patient` resource for what the server believes is a single
  human; `Patient.link` (or a returned `Person`) is the mechanism to
  signal that grouping."
- In `patient-operation-match.html`, add only: "When the server
  knows that two or more returned candidates represent the same
  human, it SHOULD signal that grouping using `Patient.link` or a
  `Person` resource included in the result Bundle. See
  [Patient: Links](patient.html#links)."
- Do **not** change `count` semantics or define a new
  `MatchGrade` convention; leave grading to existing implementer
  practice and to the IDI-Match work.

#### Justification

This keeps the normative weight in the model where it belongs, avoids
front-running the open IDI-match scoping work
([FHIR-55759](https://jira.hl7.org/browse/FHIR-55759),
[FHIR-51538](https://jira.hl7.org/browse/FHIR-51538)), and aligns the
two Moehrke tickets (FHIR-55791 and FHIR-55789) into a single
coherent edit instead of two parallel ones. It is also lower risk:
re-defining `count` or introducing per-group `MatchGrade` rules under
Disposition A could conflict with existing implementations that
already interpret `count` as "number of Patient resources returned"
(which is the natural reading and is what Disposition A would
codify, but only after a debate). Trade-off vs. A: clients still have
to infer the "N Patient, 1 human" case themselves when the server
omits links, but that ambiguity already exists today and is
arguably the implementer's contract with the upstream MPI, not
FHIR's.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive — already addressed**. The current
`Patient/$match` operation definition specifies a Bundle of
`Patient` entries with per-entry `MatchGrade`; nothing in that
definition prevents a server from returning multiple `Patient`
entries that happen to represent the same human, and `Patient.link`
and `Person` already provide the modelling primitives to convey the
grouping. No spec change is required; instead, point implementers to
existing `Patient.link` and `Person` guidance and to the IDI Patient
Matching IG for richer semantics.

#### Justification

Strictly read, the existing spec is permissive: a server MAY return
multiple Patient entries, MAY include additional resources (per the
generally-accepted "spirit of FHIR" position René Spronk articulated
on Zulip), and MAY use `Patient.link`/`Person` to express grouping.
Declining preserves implementation freedom and avoids creating new
SHOULD/SHALL constraints that current MPI-backed servers might
already violate. The risk is that the same question keeps coming back
(it has been open since 2020 per Zulip, and the reporter's 2026-03-16
comment indicates he does not regard the current text as sufficient),
which is why this is the weakest of the three.

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach)**

Disposition B is the right balance for this ticket. The substantive
ask — "tell implementers how to interpret multiple Patient results for
one human" — is fundamentally a **`Patient.link` / `Person` modelling
question**, not a new `$match` capability, and the companion ticket
[FHIR-55789](https://jira.hl7.org/browse/FHIR-55789) is already
queued against exactly that text. Resolving FHIR-55791 with a cross-
reference into the FHIR-55789 edit gives the WG one coherent change
across both tickets and lets the operation page stay narrow. It also
deliberately avoids re-defining `count` or introducing new
`MatchGrade` conventions, which would collide with the still-open
IDI-Match scoping work
([FHIR-55759](https://jira.hl7.org/browse/FHIR-55759),
[FHIR-51538](https://jira.hl7.org/browse/FHIR-51538)) and the
long-standing FHIR-9539 generalisation request — those are better
treated as a single follow-on tranche once the WG has decided whether
IDI-Match is layered on, or a sibling of, core `$match`. Disposition
A is attractive but front-runs that decision and risks a normative
`count` rule that some MPI-backed implementations will not match.
Disposition C is defensible on a strict reading of the spec but
ignores the empirical evidence (a 2020 Zulip thread that never reached
consensus, plus the reporter explicitly declining the triage
information request) that the current text is not in fact answering
the question implementers are asking.
