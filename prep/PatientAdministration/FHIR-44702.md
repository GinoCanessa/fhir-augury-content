## Summary

The submitter (Aaron Nusstein) requests that the *Interoperable Digital
Identity and Patient Matching (IDI Patient Matching)* IG add a "grammar"
that lets a sender or responder declare, on a per-element basis, which
demographic attributes used for identity matching have been verified
under Identity Assurance Level (IAL) guidance. Today, identity assurance
is expressed at the transaction level only, so the receiver cannot
distinguish a verified `Patient.name` or `Patient.birthDate` from one
that is merely asserted. The proposal is to introduce (1) a Patient
profile that places per-element identifiers (e.g., on `Patient.name`,
`.birthDate`, `.telecom`, `.address`, `.identifier`, …) so individual
attributes can be referenced, (2) a Provenance profile that uses the
`targetElement` extension to point at those per-element ids and assert
the verification claim, and (3) supporting examples and narrative
guidance in the IG. There are no cross-referenced Jira tickets,
Zulip threads, or GitHub items associated with this ticket in the
indexed sources; the only recorded discussion lives in the comments on
the ticket itself.

## Details

**Description:**

There is a need to assert when a demographic used in identity matching
has been verified by a requestor/responder so as to inform the party in
the transaction that the information provided has been verified per IAL
guidance. The proposed change to meet this need is to:

1. Create a Patient profile that creates an ID tag for each matching
   demographic that requires verification (`.name`, `.birthDate`,
   `.telecom`, `.address`, `.identifier`, etc.)
2. Create a Provenance profile that uses `targetElement` extensions in
   order to reference the ID tag in our Patient resource.
3. Create FHIR examples for this workflow.
4. Create narrative guidance for this workflow.

**Resolution:** Not Persuasive — "No industry requirement."

**Comments:**

- **juliemaas (2024-04-07):** Reports back from a workgroup discussion
  that, when the topic was first raised, stakeholders showed no interest

  identity-verification event at that level) was deemed sufficient for
  the time being.
- **juliemaas (2024-04-25):** Counter-argues for the request:
  some transaction types have no on-the-wire mechanism to express

  IAL1.5 for B2B, IAL1.8 + AAL2 for B2C) when higher than the use
  case requires — would be helpful in matching scenarios where some
  attributes are verified and others are merely asserted.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an "element-level verification grammar" to the Interoperable Digital
Identity and Patient Matching (IDI) IG:

1. Author an `IDIPatient`-derived profile (e.g.,
   `IDIPatientVerifiedAttributes`) that requires an `id` value on each
   element commonly used as a matching demographic — at minimum
   `Patient.name`, `Patient.birthDate`, `Patient.telecom`,
   `Patient.address`, and `Patient.identifier` — so each instance can
   be addressed individually.
2. Author a companion `Provenance` profile
   (e.g., `IDIProvenanceVerifiedAttribute`) that:
   - Targets the `Patient` instance via `Provenance.target`.
   - Uses the `http://hl7.org/fhir/StructureDefinition/targetElement`
     extension on `Provenance.target` to point at the verified element
     by its element `id`.
   - Carries the IAL/AAL claim (e.g., via a coded
     `Provenance.activity` or a dedicated extension bound to a value
     set covering IAL1, IAL1.5, IAL1.8, IAL2, AAL1, AAL2, AAL3).
3. Provide worked examples that show a `Patient` whose `.name` and
   `.birthDate` are verified at IAL1.8/AAL2 while `.telecom` is only
   asserted, with one or more `Provenance` instances referencing the
   verified elements.
4. Add an IG narrative page describing the workflow, the relationship
   to transaction-level identity assurance, and conformance
   expectations for senders and receivers.

#### Justification

The mechanism the submitter proposes is the FHIR-native pattern for
fine-grained provenance — `Provenance.target` plus the `targetElement`
extension is exactly designed to make assertions about a sub-element of
a target resource. Element-level claims address the on-the-wire gap
that `juliemaas` acknowledged in the 2024-04-25 comment: in
transaction types that do not carry transaction-level IAL, receivers
otherwise have no way to know which demographics were verified. Putting
the work in profiles and examples (rather than changing base FHIR or
inventing a new resource) keeps the change scoped to the IDI IG and
follows the existing precedent of the IG layering verification rules on
top of the base `Patient` resource.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of stamping ids on every matchable element of a profiled
`Patient` and pairing it with a `Provenance` profile, define a single
**`verified-attribute` extension** that can be applied directly to the
relevant `Patient` elements (`Patient.name`, `.birthDate`, `.telecom`,
`.address`, `.identifier`). The extension would carry:

- a coded `level` (IAL/AAL) bound to a value set, and
- optional `verifier` (Reference(Organization|Practitioner)) and
  `verified` (dateTime) sub-extensions.

For consumers that need full provenance semantics (who verified what,
when, by what method, with full audit trail), retain the
`Provenance` + `targetElement` pattern from Disposition A as an
*optional*, additive layer, but do not require per-element ids on the
Patient profile.

#### Justification

This keeps the verification claim co-located with the data it
qualifies, which is simpler for consumers that only need to know "was
this attribute verified, and to what level?" without dereferencing a
separate `Provenance` resource and walking a `targetElement` extension.
It also avoids forcing every conformant `Patient` instance to carry
synthetic element ids, which is awkward for systems that generate
`Patient` resources from existing demographic stores. The trade-off
versus Disposition A is that an inline extension cannot fully replace
`Provenance`'s audit semantics (signature, agent, activity history),
so this disposition keeps the `Provenance` pattern available for
implementations that need it. This addresses the submitter's underlying
need (per-attribute IAL on the wire) while reducing the conformance
burden on profile authors and implementers.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that:

- Transaction-level identity assurance, as currently expressed in the
  IDI IG (including the L0/L1 invariants the reporter and `juliemaas`
  reference), already conveys the assurance level for the set of
  attributes consistent with that verification event.
- When the topic was raised at workgroup, stakeholders showed no
  appetite for element-by-element identity assurance (per
  `juliemaas`'s 2024-04-07 comment summarizing the workgroup
  discussion).
- The submitter has not produced concrete implementer use cases or
  ballot/connectathon evidence demonstrating that the gap is
  encountered in practice, only the conceptual argument that some
  transaction types lack a wire-level mechanism.
- Implementations that genuinely need per-element verification can
  already use base-FHIR `Provenance` with the `targetElement`
  extension today without an IG-defined profile.

This is consistent with the resolution already recorded on the ticket
("Not Persuasive — No industry requirement").

#### Justification

The IG should not introduce a new normative grammar (Patient profile
with mandated element ids + a tied Provenance profile + IAL value sets
+ narrative + examples) absent demonstrated implementer demand. Adding
the grammar imposes a non-trivial conformance cost on every IDI
implementer — even those who do not need element-level claims — and
risks fragmenting how identity assurance is conveyed (transaction-level
vs. element-level) in a space where consistent matching semantics
matter. Existing `Provenance` + `targetElement` already covers the rare
case where an implementation must convey per-element verification, so
the IG can revisit a profiled grammar later if real-world traffic
emerges.

---

### Recommendation

**Recommended disposition:** C

The workgroup has already arrived at this conclusion (resolution
"Not Persuasive — No industry requirement"), and the on-ticket
discussion supports it: the only workgroup-level data point is that
stakeholders explicitly did not want element-by-element identity
assurance, and the counter-argument from `juliemaas` (2024-04-25) is
framed as conditional ("would be helpful…"), not as a concrete
implementer requirement. There are no related Jira tickets, no Zulip
threads, and no GitHub PRs/issues that suggest the use case has gained
traction since the original triage. Disposition A imposes a sizeable
conformance burden on every IDI implementer, and Disposition B —
while lighter — still introduces normative IG content for a need that
has not been substantiated. Declining now keeps the door open: if
implementer demand materializes (e.g., via a connectathon track or a
ballot comment from a deploying organization), the workgroup can
revisit with Disposition B as the natural starting point, since plain
`Provenance` + `targetElement` is already available to early adopters
without IG action.
