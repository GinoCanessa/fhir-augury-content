terminology used by the Real Time Location Services (RTLS) FHIR IG should
be anchored under `terminology.hl7.org` (THO) rather than under
`hl7.org/fhir`. The ticket title carries a trailing "4" and the body is

artifact in question is the IG's **Subscription Events** value set and
its companion code system, both currently published from the RTLS IG
namespace rather than from THO.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-host the RTLS IG's `SubscriptionEvents` value set and code system
under `terminology.hl7.org` (THO) rather than under the IG's own
`hl7.org/fhir/uv/rtls/...` (or equivalent) namespace. Concretely:

- Move the source definitions of the `SubscriptionEvents` ValueSet and
  CodeSystem out of the RTLS IG repository and into the THO repository
  (`HL7/UTG`), assigning canonicals of the form
  `http://terminology.hl7.org/ValueSet/subscription-events` and
  `http://terminology.hl7.org/CodeSystem/subscription-events` (final
  IDs to be agreed with Vocab/THO governance).
- Update the RTLS IG to reference the THO-hosted canonicals from any
  bindings on Subscription / SubscriptionTopic profiles or examples.
- Coordinate the move with Vocabulary WG via the standard THO intake
  process so the artifact is governed under THO's lifecycle rules
  rather than the IG's.

#### Justification

THO is the designated home for terminology that is general-purpose,
cross-IG, or that the community is expected to reuse outside the
originating IG. Anchoring shared subscription-event codes under THO
gives them a stable, IG-independent canonical and avoids the
versioning surprises that occur when consumers depend on a value set
that ships only with a single IG version. This is also consistent
with HL7's general direction of pushing reusable code systems and
value sets into THO.

---

### Disposition B: Alternative Approach

#### Proposal

Leave the artifact under the RTLS IG namespace, but explicitly document
the hosting decision and the criteria used:

- Keep `SubscriptionEvents` published from the RTLS IG (current
  canonical), because it is conceptually scoped to RTLS subscription
  semantics rather than to general-purpose terminology.
- Add a short note in the IG's terminology section explaining why this
  artifact is IG-local and not in THO (narrow scope, no cross-IG
  reuse expected, lifecycle tied to RTLS).
- If, in a future ballot cycle, another IG begins to depend on these
  codes, promote the artifact to THO at that point following the
  standard Vocab process.

#### Justification

THO is intended for terminology that is reused across the FHIR
ecosystem; locally-scoped codes that exist only to support a single
IG's behaviour can reasonably stay with that IG. Promoting every
IG-local value set to THO inflates THO's surface and creates
maintenance coupling between THO release trains and individual IG
release trains. Deferring promotion until cross-IG reuse is actually
demonstrated keeps the THO/IG boundary defensible.

---

### Disposition C: Decline (Moot — artifact removed)

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the question
is moot: as part of other, more substantive changes to the RTLS IG, the
`SubscriptionEvents` value set and code system are being removed
entirely. Once removed, neither hosting location applies, so no further
action against this comment is required.

#### Justification

The ticket asks where a specific artifact should be hosted. If that
artifact no longer exists in the next published version of the IG, the
hosting question has no concrete subject and there is nothing the WG
can act on. Recording the rationale on the ticket (artifact removed)
preserves the audit trail for the imported review comment without
creating spec churn.

---

### Recommendation

**Recommended disposition:** C

The ticket is already resolved as **Not Persuasive** with exactly this
rationale: the Subscription Events value set and code system are being
removed by other RTLS IG changes, which renders the THO-vs-IG hosting
question moot. Disposition C simply ratifies the resolution that the
WG (and the reporter, who is the same person managing the imported
RTLS comments under the `rtls-ready-for-block-vote` label) have already
arrived at, and aligns with the ticket being staged for block vote.
Dispositions A and B both presuppose an artifact that the next
version of the IG will not contain, and would force the WG to relitigate
a hosting decision that no longer has a target. If, after the artifact
is removed, a future version of RTLS reintroduces shared subscription
event codes, the Disposition A vs. B trade-off can be revisited at
that time on its own merits.
