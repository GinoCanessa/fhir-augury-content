## Summary

The reporter notes that the introduction of `Encounter.businessStatus` (added
under [FHIR-47297](https://jira.hl7.org/browse/FHIR-47297) — "Add
businessStatus(s)? to Encounter") created a semantic overlap with the
long-standing `Encounter.class` element. Both can now be used to describe
whether a patient encounter is inpatient, outpatient, or emergency, raising
the risk of contradictory data (e.g., `businessStatus.type = inpatient` while
`class = outpatient`). The ticket asks Patient Administration to either
harmonize the two elements or document clear guidance on when to use each,
and — if both remain — to consider invariants that prevent inconsistent
combinations.

Patient Administration has already drafted a disposition (visible in the
ticket's Resolution Description) that resolves the request as **Persuasive**:
keep both elements, move the existing `Encounter.class` note out of the
buried Notes section into the more prominent §8.11.1.1 *Status Management*
section, and add new prose there explaining how `status`, `businessStatus`,
and `subjectStatus` interrelate, with explicit acknowledgement that
`businessStatus.type` is "expected to line up with `Encounter.class` values
in most cases." No invariant is proposed in the draft. The cross-referenced
parent ticket FHIR-47297 (Applied) confirms the original design intent:
`businessStatus` is a typed backbone element specifically because Encounter
lifecycles are more complex than the simple `Task.businessStatus` pattern.

## Details

**Description:**

> With FHIR-47297 the backbone element for `Encounter.businessStatus` was
> introduced. While the purpose is clear, there is now an overlap between
> `Encounter.businessStatus` and `Encounter.class`. A patient having an
> inpatient, outpatient or emergency encounter can now be defined in both
> elements.
>
> The request of this ticket is to either harmonize the elements or add a
> clarification when to use what. If both elements remain separately,
> please consider adding invariants so that the values fit together and no
> mix up can occur (like `Encounter.businessStatus` states inpatient but
> `Encounter.class` states outpatient etc.).

**Resolution Description (drafted by PA, not yet Applied):**

- WG was happy with the drafted prose.
- Other tickets cover EncounterHistory topics; those are deliberately left out of scope here.
- Still TODO: physically move the existing `Encounter.class` note out of the Notes section into the new location.

The drafted §8.11.1.1 *Status Management* prose enumerates the three
status-bearing elements (`status`, `businessStatus`, `subjectStatus`),
states that `businessStatus` value sets vary by workflow (e.g., check-in
/ check-out for outpatient vs. admit / discharge for inpatient), and
explicitly says: *"The workflows tracked in `businessStatus.type` are
expected to line up with `Encounter.class` values in most cases."* It also
adds illustrative examples (ED `subjectStatus` flow of arrived → triaged,
on-leave/on-hold semantics, departed-without-being-seen, day-procedure
admissions) and ties the resource into the Clinical Workflow Process Life
Cycle pattern.

**Comments:** No comments are recorded on FHIR-54491 in the Augury
snapshot. (The drafted disposition prose lives in the Resolution
Description rather than as a comment.)

## Linked Jira Tickets

### [FHIR-47297: Add businessStatus(s)? to Encounter](https://jira.hl7.org/browse/FHIR-47297)

introduced `Encounter.businessStatus` as a 0..* backbone element with

FHIR-54491 reports.

## Proposed Dispositions

### Disposition A: Accept As Requested (Clarification Path — matches PA's drafted resolution)

#### Proposal

Adopt the Patient Administration drafted disposition essentially as
written:

1. In `HL7/fhir`, edit the Encounter narrative page under `source/encounter/`
   (the §8.11.1.1 *Status Management* section of `encounter.html`) to
   insert the drafted prose enumerating `status` / `businessStatus` /
   `subjectStatus`, including the explicit statement that
   "the workflows tracked in `businessStatus.type` are expected to line
   up with `Encounter.class` values in most cases" and the worked
   examples for ED, inpatient, on-leave, day-procedure, and
   departed-without-being-seen flows.
2. Physically relocate the existing `Encounter.class` note out of the
   Encounter Notes section into the new §8.11.1.1 location (the
   outstanding TODO recorded in the resolution).
3. Audit other `source/*.html` pages for anchors into the old
   `Encounter.class` note location and update or remove them so no
   inbound link becomes dangling (per the FhirCore briefing's anchor-
   removal gotcha).
4. Do **not** add a FHIRPath invariant constraining `businessStatus.type`
   ↔ `class` consistency in this ticket.

No StructureDefinition differential change, no new value set, no new
search parameter, no terminology repo edits.

#### Justification

This is exactly what the WG already drafted and labelled "we were happy
with the draft below," so adopting it preserves the design intent of
FHIR-47297 (typed backbone for complex Encounter lifecycles, deliberately
distinct from `Task.businessStatus`) while addressing the reporter's
core complaint — that the relationship between `class` and
`businessStatus` is undocumented and the existing note is buried. It is
a documentation-only change confined to a single authoring page, so it
carries minimal implementer risk and lines up cleanly with the FhirCore
briefing's *Update narrative / prose* recipe. Declining to add an
invariant is defensible because both `Encounter.class` and
`Encounter.businessStatus.code`/`.type` use **example**-strength bindings
(the additionalBindings for inpatient/outpatient/emergency are also
Example), so an invariant comparing the two would have to operate on
arbitrary local codes and could not be expressed reliably in FHIRPath
without forcing a tighter binding. The relevant gotcha is purely
mechanical: the anchor audit before merging.

---

### Disposition B: Alternative Approach — Clarify Plus Add a Soft Best-Practice Warning Invariant

#### Proposal

Adopt items 1–3 of Disposition A unchanged, then additionally add a
**warning-severity** invariant on the `Encounter` StructureDefinition
that fires when `Encounter.businessStatus.type` is populated and its
display/code clearly contradicts `Encounter.class`. Concrete shape:

- Edit the `<differential>` of the Encounter StructureDefinition (under
  `source/encounter/` in `HL7/fhir`) to add an invariant such as
  `enc-bs-class` with `severity = warning` and an English description
  like *"When Encounter.businessStatus.type is one of inpatient,
  outpatient, or emergency, it should be consistent with Encounter.class."*
- Express the FHIRPath as a best-effort comparison against the example
  additionalBindings codes only (e.g., `businessStatus.type.where(code in
  ('inpatient'|'outpatient'|'emergency')).empty() or class.where(...)`).
  Severity remains `warning` — never `error` — precisely because the
  bindings are Example.
- Update the new §8.11.1.1 prose to point at the warning invariant as
  the formal check.

#### Justification

This goes one step beyond Disposition A by giving the reporter the
"invariant so values fit together" they explicitly asked for, while
respecting the design constraint that both elements are example-bound.
A `warning` invariant signals intent without breaking instances that
legitimately use locally-defined codes (e.g., a community-clinic
workflow whose `class` doesn't map cleanly to the example
inpatient/outpatient/emergency triad). Trade-offs vs. Disposition A:
adds non-trivial editorial work to author and test the invariant,
broadens the change from documentation-only to a StructureDefinition
differential edit (still in `HL7/fhir`, but now within the
publisher-validated `<differential>` boundary called out in the
briefing), and may invite implementers to misread a `warning` as a hard
constraint. It also embeds knowledge of the *current* example
additionalBindings into a structural artifact — any change to those
bindings later requires the invariant to be revisited.

---

### Disposition C: Decline (Already Addressed by FHIR-47297 / No Real Overlap)

#### Proposal

Mark FHIR-54491 as **Not Persuasive** on the grounds that there is no
real semantic overlap once readers understand the design intent recorded
in [FHIR-47297](https://jira.hl7.org/browse/FHIR-47297) (the typed
`businessStatus` backbone is a *granular, current* business state,
whereas `Encounter.class` is the high-level kind/setting of the
encounter). Respond by pointing the reporter at the existing
`Encounter.class` note and the FHIR-47297 resolution; make no spec
changes.

#### Justification

Defensible on the narrow argument that the two elements have distinct
roles already documented in the resolution of FHIR-47297, and that
adding more prose risks duplicating definitions and introducing further
divergence between the Encounter narrative and the StructureDefinition
element comments. Declining also avoids the anchor-rewrite churn called
out in the briefing's gotchas.

The weakness — and reason this is recorded only for completeness — is
that the reporter's core observation is factually correct: the existing
`Encounter.class` note *is* buried in the Notes section, and the new
`businessStatus.type` example bindings (inpatient/outpatient/emergency)
read identically to common `Encounter.class` codes, which will keep
generating the same confusion this ticket reports. Declining therefore
predictably re-creates the same ticket from a future implementer.

---

### Recommendation

**Recommended disposition: A.**

Disposition A is the cleanest answer to the reporter's request and is
already pre-approved in substance by the WG's drafted resolution — the
only outstanding TODO is the physical relocation of the existing
`Encounter.class` note, which is a small editorial move within
`source/encounter/encounter.html`. It addresses the real defect
(buried note + undocumented interaction with the new `businessStatus`)
without inflating scope into StructureDefinition changes, value-set
work, or new search parameters, and it stays inside the *Update
narrative / prose* recipe that the FhirCore briefing flags as the
low-risk path. Disposition B is tempting because it literally answers
"please consider adding invariants," but example-bound elements cannot
support a meaningful machine-checkable invariant without first
strengthening the bindings, which is a much bigger conversation that
should not be folded into this ticket. Disposition C understates a real
documentation gap and predictably recurs. Recommend the WG ratify the
drafted prose, complete the note relocation, run the anchor audit
flagged in Repo Context, and close FHIR-54491 as **Persuasive**.
