## Summary

Eric Haas reports an inconsistency in the Validated Healthcare Directory (VhDir)
`NewPatients` extension `StructureDefinition`: the human-readable description
states the extension is `0..*` on `Location` and `PractitionerRole`, but the
formal element cardinality in the SD does not match. The ticket asks to fix the
SD so that the extension is `0..*` on both `Location` and `PractitionerRole`,
aligning the machine-readable definition with the description that already
states it. Eric characterises this as an error in the SD rather than a design
question.

For broader context, the closely-related downstream issue
[FHIR-43796](https://jira.hl7.org/browse/FHIR-43796) — filed by Tushar Nair
against the Da Vinci PDex Plan-Net IG, which clones the VhDir `NewPatients`
extension — explicitly questions whether `newpatients` being `0..*` while
`PractitionerRole.network` is `0..1` is internally consistent. That ticket was
resolved as **Not Persuasive with Modification** in early 2024 by the Financial
Management work group, who concluded the `0..*` cardinality is appropriate and
chose to add explanatory text rather than tighten the cardinality. That
disposition is directly relevant evidence that the `0..*` target Eric is asking
for is the intended design.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the VhDir `NewPatients` extension `StructureDefinition` (canonical
`http://hl7.org/fhir/uv/vhdir/StructureDefinition/newpatients`), change the
extension context cardinality so that it is `0..*` on both `Location` and
`PractitionerRole`, matching the wording already present in the extension's
human-readable description. Concretely, in the SD's `Extension.extension` /
context entries (or the slice on `extension` where the extension is embedded),
ensure the upper bound is `*` for the `Location` and `PractitionerRole`
contexts. No semantic change to sub-elements (`acceptingPatients`,
`fromNetwork`, `characteristics`) is required — this is purely a card fix to
align the SD with the prose.

#### Justification

This is the disposition Eric explicitly requested and is internally consistent
with: (1) the existing description in the same SD; (2) the downstream Plan-Net
clone, where the same extension is `0..*` and was confirmed appropriate by
Financial Management in FHIR-43796 ("the cardinality is appropriate"); and
(3) the use cases on Zulip (Scott Robertson's per-network /
per-InsurancePlan scenarios) which require multiple repetitions per
PractitionerRole. Since the prose already documents `0..*`, implementers have
been operating against the documented behaviour; the SD fix removes a latent
trap where a strict validator could reject conformant instances.

---

### Disposition B: Alternative Approach

#### Proposal

Make the SD/description alignment fix as in Disposition A, **and** at the same
time port the clarifying narrative that Financial Management adopted in
FHIR-43796 into the VhDir `NewPatients` extension's description. Specifically,
add wording explaining that the `0..*` repetition is intended to allow a
practitioner role to communicate accepting-patients status per network (and,
where relevant, per other characteristics), and cross-reference the
relationship between `newpatients` `0..*` and any `*.network` reference
extension that is `0..1`. This avoids re-litigating the same consistency
question for VhDir that was already settled for Plan-Net.

#### Justification

The cardinality fix on its own does not address the underlying confusion that
caused FHIR-43796 to be filed years after FHIR-19742 — namely, "if
PractitionerRole.network is `0..1`, why is `newpatients` `0..*`?". Importing
the Financial Management narrative into VhDir keeps the source-of-truth SD
self-explaining, helps future cloners (Plan-Net is not the only candidate),
and aligns the two specifications' guidance. The marginal cost is small (a few
sentences of narrative added alongside a card change that is already being
made).

---

### Disposition C: Decline

#### Proposal

Reject the change request as already addressed: leave the SD as it stands and
direct the requester to the existing description and to the FHIR-43796
clarification adopted in the Plan-Net clone of the same extension. Mark
"Not Persuasive" with a note pointing at FHIR-43796.

#### Justification

This is only defensible if Patient Administration determines that the current
SD is in fact already `0..*` on `Location` and `PractitionerRole` and that
Eric's report is based on a stale build, or if VhDir is considered frozen at
its 2018 ballot snapshot and not eligible for any further normative changes.
Both conditions are unlikely: Eric is a long-time VhDir contributor and the
ticket has remained Triaged since 2018, suggesting the SD/description mismatch
is real, and the IG is still maintained on `build.fhir.org`.

---

### Recommendation

**Recommended disposition:** B

This is unambiguously a description-vs-SD mismatch that should be fixed (so
some form of Disposition A is required), but doing only the cardinality fix
leaves in place the same ambiguity that drove a separate IG to file
FHIR-43796 six years later. Folding in the clarifying narrative Financial
Management already adopted — that `newpatients` is `0..*` to support per-network
accepting-patients statements while role-level `network` is `0..1` —
costs little, eliminates the recurring confusion, and keeps the VhDir SD and
its Plan-Net clone aligned in both letter and explanation. Patient
Administration should coordinate with FM/Plan-Net so the wording is
genuinely consistent across the two IGs.
