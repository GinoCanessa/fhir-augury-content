## Summary

The VHDir (Validated Healthcare Directory) CareTeam profile was originally
authored against FHIR STU3, where `CareTeam` had no `encounter` element. When
the VHDir IG was migrated to R4, the newly-added `CareTeam.encounter` element
began appearing in the continuous build's rendering of the VHDir CareTeam
profile. Because a provider/healthcare directory is not an operational system
of record for clinical encounters, the reporter argues that
`CareTeam.encounter` is meaningless in a VHDir context and should be explicitly
removed (constrained out) in the VHDir CareTeam profile. No cross-referenced
Jira tickets were found via the Augury cross-reference index.

## Details

**Description:**

> A reference to an encounter was added to the CareTeam resource in R4. As the
> VHDir IG was originally built in STU3, the encounter element did not exist.
> However, because the IG is based on R4, the element now appears in the
> continuous build. It should be removed in the VHDir CareTeam profile, as a
> VHDir would not be expected to store "local" data about encounters.

### [implementers > CareTeam without patient and encounter](https://chat.fhir.org/#narrow/stream/implementers/topic/CareTeam%20without%20patient%20and%20encounter/near/182979465)

Josh Haskins asked how to model a standing/permanent CareTeam that is not tied
to a specific patient or encounter (e.g., a "Code Blue" team). Lloyd McKenzie
confirmed that neither `subject` nor `encounter` is mandatory on `CareTeam` and
that this use case is in scope for the resource (citing the resource
definition's "prior to a subject being identified" language). Dave Carlson
then pointed to the VHDir CareTeam profile as an example of the pattern, noting
that it specifically constrains `subject` to `0..0`. Yunwei Wang subsequently
flagged an inconsistency in the VHDir profile: the background text says the
profile removes `careTeam.subject`, `careTeam.context`, and
`careTeam.reasonReference`, but the snapshot table still shows `context` as
`0..1` and `reasonReference` as `0..*`. This observation directly supports
the premise of FHIR-20674 — that the VHDir CareTeam profile's R4 rendering
has not been kept in sync with its stated intent.

Participants: Josh Haskins, Yunwei Wang, Lloyd McKenzie, Michele Mottini, Dave
Carlson.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the VHDir CareTeam profile (`StructureDefinition/vhdir-careteam`), add a
differential element that constrains `CareTeam.encounter` to `0..0`, mirroring
the existing `0..0` constraint on `CareTeam.subject`. At the same time,
reconcile the profile's Background / narrative with the snapshot so the list
of removed elements is accurate — per the Zulip thread, `CareTeam.context`
(STU3 name, now `encounter` in R4) and `CareTeam.reasonReference` are
described as removed in the narrative but still appear with their base
cardinalities in the snapshot. The authoring change is in the VHDir IG's
StructureDefinition source for `vhdir-careteam` (ElementDefinition
differential); the published build will then regenerate the snapshot with
`encounter 0..0`.

#### Justification

VHDir is a provider/healthcare directory profile intended to describe
standing, directory-level relationships (a CareTeam that exists as a
professional entity), not operational, patient-specific or encounter-specific
occurrences. `CareTeam.encounter` binds an instance to a single Encounter
within a clinical system, which is definitionally out of scope for a
directory. This also preserves consistency with the existing `subject 0..0`
constraint (the profile already takes the position that VHDir CareTeam
instances are not patient-specific), and aligns the profile with Lloyd
McKenzie's guidance in the Zulip thread that "standing team" CareTeams
should not carry subject or encounter references. Finally, fixing the
narrative/snapshot mismatch identified by Yunwei Wang in the same thread
removes an existing source of implementer confusion.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than constraining `CareTeam.encounter` to `0..0`, add a `mustSupport =
false` marker plus an invariant or profile-level guidance stating that
`CareTeam.encounter` SHALL NOT be populated by VHDir producers and MAY be
ignored by VHDir consumers, while leaving the cardinality at the base `0..1`.
Document the rationale in the profile's Background section and align the
narrative's "removed elements" list with whatever the snapshot actually
contains (same narrative/snapshot reconciliation as Disposition A, but
without the `0..0` slicing). Consider the same pattern for
`CareTeam.reasonReference` if the workgroup wants to remain conservative
about `0..0` constraints.

#### Justification

`0..0` constraints are maximally restrictive and prevent any downstream
profile or affiliate IG (e.g., a national profile built on top of VHDir)
from re-introducing the element even when a local use case would justify
it. A guidance-plus-`mustSupport=false` pattern achieves the same practical
outcome (directory implementers won't populate it; consumers won't rely on
it) while leaving the door open for derived profiles. It also avoids having
to later relax a `0..0` constraint if real-world directory deployments
discover a legitimate need to link a CareTeam to a founding encounter
(e.g., a research protocol kickoff). The cost is a weaker conformance
signal than Disposition A.

---

### Disposition C: Decline

#### Proposal

Do not change the VHDir CareTeam profile. Close FHIR-20674 as "Not Persuasive
– Work Group Unable to Progress" or redirect it to the successor
`HL7/fhir-us-ndh` (National Directory of Healthcare) repository, on the
grounds that VHDir is no longer actively maintained as a standalone ballot
artifact and that any directory-profile changes should be made in its R4/R5
successor instead.

#### Justification

VHDir was a STU1 ballot and has effectively been superseded by US NDH for
active directory work in the HL7 ecosystem. Investing in a structural change
to the VHDir CareTeam profile six years after the ticket was filed may not
produce implementer value proportional to the work, especially if no VHDir
implementers have since raised the issue. If the same concern applies to US
NDH's CareTeam profile, it can be tracked as a new ticket against that IG.

---

### Recommendation

**Recommended disposition:** A

The reporter's analysis is correct and the change is small, local, and
editorial in nature — it aligns the VHDir CareTeam profile with the stance
the profile has already taken for `subject` (`0..0`) and with the already-
written (but inconsistent) Background narrative flagged in the Zulip
thread. Directory CareTeams are by definition standing,
non-encounter-specific entities, and leaving `CareTeam.encounter` visible in
the profile's snapshot creates exactly the implementer confusion seen in the
`implementers > CareTeam without patient and encounter` thread. Disposition
B preserves flexibility but at the cost of a weaker conformance signal that
does not match the profile's existing pattern. Disposition C punts on a
well-scoped defect in a published IG; even if the workgroup's forward focus
is US NDH, it costs little to apply this correction (and the narrative
reconciliation) to the VHDir artifact so the continuous build is not
actively misleading readers who arrive from search engines or the Zulip
archive. If the workgroup confirms VHDir is frozen, a reasonable variant is
Accept-As-Requested *and* open a parallel ticket against `HL7/fhir-us-ndh`
to ensure the same constraint is present there.
