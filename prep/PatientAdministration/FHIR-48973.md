## Summary

The ticket, titled "Remove NDH Verification Status Extension", proposes
revisiting (and the title suggests removing) the NDH `verification-status`
extension that is currently applied to directory entries in the National
Directory of Healthcare Providers and Services (NDH) IG. The reporter (David
Pyke) notes there is high stakeholder interest in improving directory entry
quality and is inclined to test the existing approach with implementers, while
flagging an open question about how to determine *when* a verification status
was set (i.e., whether the current extension carries enough provenance).
Workgroup discussion on the 12/9/2024 HL7 FAST National Directory Public Call
agreed that knowing the validity / verification status of an entry is a
significant directory-data challenge, but recommended that this value be
re-evaluated as part of the broader STU2 rework of the V&V section, with a
suggestion to group it and make it easier to use rather than removed outright.
A separate comment from John Moehrke questions the extension's utility and
suggests the same functionality could be provided by `Provenance`. There are no
cross-referenced Jira tickets, GitHub items, or Zulip threads recorded for this
ticket in the local index.

### Disposition A: Accept As Requested

#### Proposal

Remove the `base-ext-verification-status` extension (canonical
`http://hl7.org/fhir/us/ndh/StructureDefinition/base-ext-verification-status`)
from the NDH IG (`HL7/fhir-us-ndh`, package `hl7.fhir.us.ndh`) for STU2.
Concretely:

- Delete (or retire / mark as deprecated in STU2) the
  `base-ext-verification-status` `StructureDefinition` from the NDH authoring
  source.
- Remove `extension` slices that bind to this URL on every NDH profile and
  base profile that currently uses it (the extension is applied broadly across
  NDH base profiles — e.g., the `CareTeam`-style usage seen in the indexed
  Zulip discussion of `CareTeam.extension.where(url = '…/base-ext-verification-status')`).
- Update the V&V narrative pages to remove references to the extension and
  point implementers to the replacement V&V approach (see Disposition B for
  the recommended replacement).
- Add a STU2 migration note describing how STU1 implementers should map any
  existing `base-ext-verification-status` content forward.

#### Justification

This is what the ticket title literally asks for, and the comment from John
Moehrke — that the extension feels like overhead and that `Provenance` can
already capture verification activity and timing — supports outright removal.
It also directly answers the reporter's open question ("is there a way to tell
when this status was set?") by deferring to `Provenance`, which carries
`recorded`, `occurred[x]`, `agent`, and `signature` and is purpose-built for
exactly this kind of attestation. Removing a heavyweight, broadly-applied
extension simplifies the NDH profile surface and reduces the cross-profile
maintenance burden flagged tangentially by the IG-publisher / FHIRPath
discussions in Zulip.

---

### Disposition B: Alternative Approach

#### Proposal

Do not remove the verification-status concept; instead **rework** it as part of
the STU2 V&V section refresh, per the explicit feedback from the 12/9/2024
FAST National Directory Public Call. Specifically:

- Replace the current single `base-ext-verification-status` extension with a
  **grouped complex extension** (e.g., `verification`) that carries, at
  minimum, `status`, `statusDate` (or `period`), `verifier` (Reference to
  `Organization`/`Practitioner`/`PractitionerRole`), and `method`/`source`
  sub-extensions. Element paths to be authored under the NDH IG's
  `input/resources` (or equivalent authoring root in `HL7/fhir-us-ndh`).
- Where the underlying base resource already supports verification natively
  (e.g., `Location.status`, `HealthcareService.active`, role/affiliation
  `period`), prefer the native element and *only* slot the new grouped
  extension where there is no native equivalent.
- Co-publish a normative pattern (narrative + example) that shows how a
  receiving system can derive "when the status was set" from a companion
  `Provenance` resource targeting the entry, so the grouped extension and
  `Provenance` together answer the reporter's question without duplicating
  data.
- Mark STU1's `base-ext-verification-status` as deprecated in STU2 with a
  clear forward-mapping rule into the new grouped extension.

#### Justification

This matches the workgroup feedback recorded in the comments verbatim ("As
part of STU2 updates the V&V sections will be revised … this value should be
revisited with the re-work of the V&V section. The suggestion was to group
this value, and make it easier to use."). It also answers the reporter's
"when was this set?" question structurally (via a `statusDate`/`period`
sub-extension) instead of forcing every implementer to bolt on `Provenance`,
while still allowing `Provenance` for full attestation. Compared to
Disposition A, it preserves a known, useful concept that stakeholders have
flagged as important to directory data quality, and avoids a hard breaking
change in STU2 that would force every NDH implementer to rewrite their
verification tracking logic.

---

### Disposition C: Decline

#### Proposal

Decline the request to remove `base-ext-verification-status` in its current
form, on the grounds that (a) the meeting feedback explicitly directs the
workgroup to *revise* — not remove — the V&V machinery during STU2, and
(b) the underlying need (knowing whether and when a directory entry was
verified) is a stated stakeholder priority. Close FHIR-48973 as "Not
Persuasive" with a pointer to the STU2 V&V rework workstream, and track the
grouping/usability improvements there rather than as a stand-alone removal.

#### Justification

Removal-as-such has not been justified in the ticket: the only argument for
removal is John Moehrke's comment that `Provenance` could cover the same
ground, but no concrete migration design has been proposed and no
implementers have reported that the extension is unusable. Meanwhile, the
recorded community feedback is the opposite — verification status is "an
important detail" that should be *easier* to use, not absent. Declining and
folding the work into the planned STU2 V&V section revision keeps STU1
implementations stable and avoids creating a breaking change with no
replacement story.

---

### Recommendation

**Recommended disposition:** B

Disposition B aligns precisely with the documented workgroup direction from
the 12/9/2024 FAST National Directory Public Call ("revisited with the
re-work of the V&V section … grouped … easier to use") while still answering
the reporter's substantive open question about *when* the status was set.
It captures the legitimate concern behind Disposition A — that the current
extension is awkward overhead and lacks provenance — without throwing away a
concept stakeholders have explicitly identified as important to directory
data quality, and it is more actionable than Disposition C because it ties
the change to concrete STU2 deliverables (a grouped extension, deprecation of
the existing one, and an explicit `Provenance` companion pattern). Before
acting, the workgroup should request a `repo-analysis` of `HL7/fhir-us-ndh`
so the exact authoring paths for `base-ext-verification-status` and any
profile slices that bind it can be enumerated as part of the STU2 V&V
work item.
