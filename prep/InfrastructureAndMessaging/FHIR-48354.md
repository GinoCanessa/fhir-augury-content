## Summary

This FHIRcast ticket, raised out of the Connectathon 37 II track, asks the
specification to clarify whether a subscriber/application can *bootstrap* a
content sharing session for a brand-new resource that does not yet exist on
any server. Concretely: after a `Patient-open` has established patient
context, may an application send a `DiagnosticReport-open` for a
DiagnosticReport that the application is in the process of creating (i.e.
no persisted DR yet)? If the hub permits this, the reporter wants the
specification to state (a) whether the just-introduced DiagnosticReport is
implicitly the initial member of the content collection associated with
that DR's session, and (b) whether the existing anchor resources from
parent contexts (Patient, Encounter, etc.) also appear in that content
collection. The reporter notes the asymmetric authority pattern that
already exists in FHIRcast: a hub may refuse the creation of a new anchor
context, but subscribers cannot — they can only refuse to follow a
context change.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items were
found for this ticket via FhirAugury, so no additional related-ticket
context is incorporated here.

Application connects to the FHIRcast hub.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative "Bootstrapping content sharing for a new resource"
sub-section under the FHIRcast content sharing chapter that explicitly
permits an application to initiate a content sharing session on a
not-yet-persisted anchor resource by sending the corresponding
`{resource}-open` request (e.g., `DiagnosticReport-open`) when an
ambient context for the parent anchor (e.g., `Patient`) is already
established. The text would specify:

1. The hub MAY refuse the creation of the new anchor context (consistent
   with current `*-open` semantics); subscribers SHALL NOT refuse but
   MAY decline to switch context.
2. On acceptance, the newly-introduced anchor resource is **implicitly
   the initial sole member** of the content collection associated with
   that anchor's content sharing session. In other words, the resource
   identified in the `*-open` request is treated as already shared
   content for that session, without requiring an explicit
   `*-update`/content event to seed it.
3. Anchor resources of *parent* contexts already in scope (Patient,
   Encounter, ImagingStudy, ...) are **not** implicitly added to the
   new resource's content collection. Their relationship is captured
   by the FHIRcast context graph (anchor hierarchy), not by content
   collection membership; if subscribers need them as content they
   must be added explicitly via the appropriate content event.
4. A clarifying note that an application may use this pattern to
   collaboratively author a brand-new DiagnosticReport (or other anchor
   type) without first creating it on a FHIR server — the resource may
   exist only in the FHIRcast session until persisted by one of the
   participants.

#### Justification

The Connectathon 37 II use case is a concrete, realistic
collaborative-authoring scenario (a clinician begins dictating /
drafting a report inside one application while peers want to view or
contribute via FHIRcast). FHIRcast already supports `*-open` for new
contexts, and the asymmetric refuse semantics the reporter cites
(hub can refuse creation, subscribers can only refuse to follow) are
already part of the spec; this disposition simply removes the current
ambiguity rather than introducing new mechanism. Treating the
newly-opened resource as the implicit seed of its own content
collection mirrors how human users mentally model "open this report"
(the report itself is the first thing being shared) and avoids forcing
an immediate redundant content event after every `*-open`. Keeping
parent anchor resources out of the child's content collection
preserves the existing separation of concerns between the *context
graph* and the *content collection* and avoids surprising duplication.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than encoding implicit content collection seeding into normative
text, address the underlying need by adding a small, optional
`bootstrap` (or `seedContent`) hint to the `*-open` request body whose
value is the canonical reference to the new anchor resource (or its
inline `Bundle`/`resource` payload when not yet persisted). The
specification would then say:

1. An application MAY initiate a content sharing session on a
   not-yet-persisted resource by sending `*-open` with the new
   anchor-resource reference (or inline body) and an explicit
   `bootstrap: true` flag.
2. When `bootstrap` is set, the hub treats the named resource as the
   initial member of that anchor's content collection; otherwise the
   content collection starts empty and is populated by subsequent
   content events.
3. Parent anchor resources are never implicitly copied into the new
   collection; if a subscriber wants them as content it issues the
   appropriate content event.
4. The hub's existing right to refuse creation of new anchor contexts
   continues to apply; a hub that does not implement bootstrap
   semantics MUST refuse `bootstrap: true` requests with a documented
   error code.

#### Justification

This keeps the more invasive "implicit seeding" behaviour opt-in and
discoverable by inspection of the request, which is friendlier to
existing hubs and subscribers that have not implemented the new
behaviour and to conformance testing. It also gives implementers a
clean way to distinguish "open this existing DR for shared viewing"
(no bootstrap) from "open this brand-new DR I am drafting" (bootstrap
true), which reduces the risk that a subscriber surprises its user by
treating a freshly-opened existing report as draft content. The
trade-off vs. Disposition A is added surface area on the `*-open`
event and a longer adoption path before all hubs support the flag.

---

### Disposition C: Decline

#### Proposal

Do not change the FHIRcast specification. Document that the question is
out of scope for the current FHIRcast normative content because the
existing `*-open` and content sharing event sequences are already
sufficient: an application that wants to collaborate on a new
DiagnosticReport can persist a draft DR (e.g., with `status = draft` /
`partial`) on a FHIR server first, then issue a normal
`DiagnosticReport-open` against the persisted reference, and use
existing content events to add or update content. Close the ticket as
*Persuasive – No Change* with that guidance.

#### Justification

Declining is defensible if the workgroup judges that the current
mechanism (persist first, then open) is the intended pattern and that
introducing implicit-seeding behaviour or a `bootstrap` flag risks
creating inconsistent implementations during the FTR/STU period. It
also avoids re-litigating the relationship between the context graph
and the content collection at a point in the spec's maturity where
implementer feedback is still arriving via Connectathons. The cost is
that the original Connectathon use case (drafting before persistence)
remains awkward and round-tripped through FHIR server writes that some
collaborative-authoring scenarios would prefer to defer.

---

### Recommendation

**Recommended disposition:** A

The reporter's question is fundamentally a *clarification* question:
the FHIRcast spec today does not say what happens when `*-open` carries
a not-yet-persisted resource, and that ambiguity will be answered one
way or another by every implementer who hits the Connectathon scenario.
Leaving it unwritten (Disposition C) defers the work and guarantees
divergent implementations. A bootstrap flag (Disposition B) is cleaner
in principle but introduces new surface area and a parallel
"with-flag / without-flag" code path for hubs and subscribers, which
adds conformance complexity without buying much beyond what a single
unambiguous normative paragraph delivers. Disposition A — explicitly
permit `*-open` to bootstrap a session on a new resource, treat the
named resource as the implicit initial member of *its own* content
collection, and explicitly do **not** propagate parent anchor
resources into that collection — is the smallest, most consistent
extension of existing FHIRcast semantics and directly answers all
three sub-questions the reporter raised. It also preserves the hub's
existing right to refuse creation, which the reporter explicitly
called out. The workgroup should still consider whether to mark the
new sub-section as *Trial Use* maturity given the limited
Connectathon evidence.
