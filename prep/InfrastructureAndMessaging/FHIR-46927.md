## Summary

The reporter notes an inconsistency in the FHIRcast STU3 publication: §2.3.4.4 marks the generic *Selection events* as **experimental**, but the corresponding `DiagnosticReport-select` event in the Event Catalog is not similarly flagged. The ticket asks that the `DiagnosticReport-select` event page be updated to carry the same "experimental" callout so the catalog entry matches the spec narrative. The request is a direct follow-on to [FHIR-36981](https://jira.hl7.org/browse/FHIR-36981) ("Consider *-select event for other resources"), which was applied for STU3 with the resolution that select events would be documented as a generic event category and **explicitly marked experimental for the STU3 publication**, with no additional `*-select` events being defined without implementation experience. FHIR-46927 simply closes the gap that the experimental callout, which was added to the generic event description, never made it onto the `DiagnosticReport-select` event page itself. The recorded resolution on the ticket already states: *"We will update the DxReport-select event to be labeled as experimental."*

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution exactly as already recorded on the ticket: edit the `DiagnosticReport-select` event page in the FHIRcast Event Catalog so that it carries the same **Experimental** maturity callout that §2.3.4.4 (Selection events) uses for the generic select category. Concretely, add a leading admonition / banner on the `DiagnosticReport-select` event page that states the event is experimental for the STU3 publication, mirroring the wording in §2.3.4.4, and (if FHIRcast uses a per-event maturity field) set that field to `Experimental` so it renders consistently with the section narrative.

#### Justification

This is precisely what the reporter requested and matches the resolution already entered on the ticket ("We will update the DxReport-select event to be labeled as experimental."). It is also fully consistent with the resolution applied for [FHIR-36981](https://jira.hl7.org/browse/FHIR-36981), which explicitly decided that select events would be marked experimental for STU3. The change is editorial, has no normative impact on event payload or behavior, and removes a real inconsistency between the prose in §2.3.4.4 and the catalog entry that an implementer would land on when reading about the `DiagnosticReport-select` event in isolation.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the inconsistency as a symptom of a missing single source of truth and update the FHIRcast page templates / event catalog generator so the "experimental" maturity is asserted **once** in §2.3.4.4 (or in event metadata) and is **rendered automatically** on every per-event page that belongs to the Selection events category — including `DiagnosticReport-select` and any future `*-select` events. As part of this change, also audit the other `*-select` event pages (if/when they exist) and ensure the rendered callout matches.

#### Justification

The root cause here is duplicated wording between a section-level callout and a per-event page; manually patching only `DiagnosticReport-select` will leave the same drift risk for any future select event. A template/metadata-driven approach respects FHIRcast's stated direction in [FHIR-36981](https://jira.hl7.org/browse/FHIR-36981) of treating select events as a *generic event category* rather than a per-resource feature. Trade-off: it is more work than Disposition A and may be out of scope for a block-vote-4 ballot disposition that the reporter framed as a small editorial fix; it would also delay closing this ticket if the templating change cannot land in the same publication window.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed by existing mechanism**: §2.3.4.4 already calls out the entire Selection events category — which includes `DiagnosticReport-select` — as experimental, so any reader following the spec from §2.3 down to the event will encounter the callout. No further per-event annotation is required.

#### Justification

A defensible argument for declining is that the experimental status is a *category-level* property and is already documented at the category level. Adding a duplicate banner on each event page risks future drift in the opposite direction (page says experimental, section later says otherwise) and adds editorial burden. Trade-off: this argument is weak in practice because implementers often deep-link directly to a specific event page (e.g. via search) and may never see the §2.3.4.4 callout, so the inconsistency the reporter identified is real and user-visible.

---

### Recommendation

**Recommended disposition:** A

The ticket's recorded resolution is already "Persuasive" with the action "We will update the DxReport-select event to be labeled as experimental.", and that aligns directly with the applied resolution of the parent ticket [FHIR-36981](https://jira.hl7.org/browse/FHIR-36981), which committed FHIRcast to marking select events as experimental for STU3. Disposition A closes the inconsistency the reporter identified with a small, low-risk editorial change in the same publication window as the block-vote-4 batch. Disposition B is a worthwhile follow-up if the FHIRcast tooling supports it, but should not block this fix; it can be filed as a separate tracker. Disposition C is not defensible because deep-linking to a single event page is a real implementer workflow and the experimental status materially affects how implementers should treat the event.
