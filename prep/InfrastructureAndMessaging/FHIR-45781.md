## Summary

The reporter argues that FHIRcast should not require a `Patient` to be present
on context resources (such as `Encounter`) in generic FHIRcast deployments.
Because the base FHIR `Encounter` allows `subject` to be either a `Patient` or
a `Group`, an FHIRcast profile or event definition that requires `Patient` is
narrower than the underlying FHIR specification and therefore excludes
legitimate use cases (e.g., group encounters, population-level workflows). The
ticket proposes two related changes: (a) loosen FHIRcast so that `Patient` is
**optional** rather than required as a context anchor, and (b) **add `Group`**
as an allowed context resource alongside `Patient` so that group-oriented
encounters can be exchanged.

This ticket is functionally identical to [FHIR-45780](https://jira.hl7.org/browse/FHIR-45780),
filed by the same reporter at the same minute with the same title and body —
almost certainly a duplicate submission. The closely related context-modeling
tickets [FHIR-37089](https://jira.hl7.org/browse/FHIR-37089) (handling of
secondary anchors when the upstream resource lacks them) and
[FHIR-48678](https://jira.hl7.org/browse/FHIR-48678) (broadening the SDC
launch-context valueset, also reported by Bas van den Heuvel) show a
consistent theme from this reporter: FHIRcast and SDC should align with the
flexibility of base FHIR rather than narrowing it through over-constrained
profiles.

## Proposed Dispositions

Note: with no GitHub cross-references and no surfaced repo briefings, the
dispositions below name the FHIRcast specification artifacts at the
conceptual level (events, profiles, context valuesets, anchor-resource
constraints) rather than specific authoring file paths.

### Disposition A: Accept As Requested

#### Proposal

Update the FHIRcast specification (current FHIRcast IG, `HL7/fhircast-docs`)
so that:

1. The `Encounter`-bearing context anchor in FHIRcast events whose subject
   is an `Encounter` (e.g., `Encounter-open`, `Encounter-close`, and any
   `*-open` event that carries an `encounter` context element) no longer
   requires the `Encounter` to have a `Patient` subject. Allow
   `Encounter.subject` to be either `Patient` or `Group`, mirroring the
   base R4/R5 `Encounter` resource.
2. Add `Group` to the set of allowed FHIRcast top-level context resources
   (and to any context-type valueset that enumerates allowed anchors), so
   subscribers can be told "the current context is a `Group`" for
   group-oriented encounters and population workflows.
3. Make `Patient` **optional** (rather than required) on events / context
   slots where it was previously mandatory but where the underlying
   resource (e.g., `Encounter`) does not require it. Where a downstream
   subscriber needs to know that no `Patient` is in context, recommend the
   `data-absent-reason` extension (consistent with the FHIR-37089 comment).

Resolve [FHIR-45780](https://jira.hl7.org/browse/FHIR-45780) as a duplicate
of FHIR-45781 (or vice versa) so a single change is tracked end-to-end.

#### Justification

FHIRcast profiles must not be more restrictive than the base FHIR
resources they wrap unless there is a deliberate, documented reason. The
base `Encounter` already supports `Group` as a subject; constraining
FHIRcast to `Patient`-only excludes legitimate clinical workflows
(group visits, family encounters, population/cohort activities) without
any FHIRcast-specific rationale. The same reporter's
[FHIR-48678](https://jira.hl7.org/browse/FHIR-48678) was accepted with
modification on the parallel principle that context typing should not
narrow the underlying FHIR model. Aligning FHIRcast with `Encounter`'s
real cardinality is a small, defensible normative move.

---

### Disposition B: Alternative Approach

#### Proposal

Keep FHIRcast's existing `Patient`-as-anchor model intact for the
default/most-common path, but introduce a profile- or capability-level
**switch** rather than a normative loosening of the core events:

1. Define a separate event family or capability flag for "patient-less"
   contexts — e.g., a `group-open` / `group-close` event pair (and/or a
   profile variant of `Encounter-open` that targets `Group`-subject
   encounters). Existing `Patient`-anchored events stay as they are.
2. For events where `Patient` remains the primary anchor but the
   underlying resource may legitimately not carry one (the FHIR-37089
   case), require subscribers/hubs to use the `data-absent-reason`
   extension on the missing anchor rather than dropping the slot.
3. Document explicitly in the FHIRcast IG that hubs are not required to
   support `Group`-anchored events and that subscribers must declare
   support via capability negotiation.

#### Justification

A wholesale loosening of `Patient` from required to optional changes the
contract every existing FHIRcast hub and subscriber relies on. Many
deployed hubs assume that every context update carries a `Patient` and
key off `Patient.id` for session routing, audit, and consent checks.
Splitting the new behavior into a distinct event/profile family
preserves backward compatibility, lets implementers opt in, and keeps
the common imaging/EHR use cases (which are overwhelmingly
patient-anchored) on the well-trodden path. It also addresses the
FHIR-37089-style "anchor is sometimes absent" case with the same
mechanism (`data-absent-reason`) the reporter has previously endorsed.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped, on the grounds that (a) the
ticket is an accidental duplicate of
[FHIR-45780](https://jira.hl7.org/browse/FHIR-45780) — only one should
remain open — and (b) the underlying need is already addressable today
without a normative change: a deployment that needs `Group`-anchored or
patient-less context can profile FHIRcast locally (or in a domain IG)
to describe its behavior, and use the `data-absent-reason` extension
(per the reporter's own recommendation on
[FHIR-37089](https://jira.hl7.org/browse/FHIR-37089)) when an expected
anchor is missing. Close as "Not Persuasive — already addressable;
duplicate of FHIR-45780", with a pointer to the existing extension
mechanism and to FHIR-48678 as the precedent for downstream profiles
broadening context typing.

#### Justification

FHIRcast's required `Patient` anchor is load-bearing for a large fraction
of real deployments (imaging launch, EHR-driven patient steering, audit).
The ticket presents the requirement abstractly, with no concrete user
story, no community discussion (no Zulip thread cross-references the
ticket, and no GitHub issue or PR was opened against
`HL7/fhircast-docs`), and no implementation experience report. Without
that evidence, a normative loosening risks fragmenting hub/subscriber
behavior to solve a hypothetical case already solvable through
profiling and `data-absent-reason`.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach), paired with
administratively closing [FHIR-45780](https://jira.hl7.org/browse/FHIR-45780)
as a duplicate of FHIR-45781.

The reporter's underlying point is sound and consistent with how the
workgroup has resolved his related tickets (FHIR-48678 was accepted with
modification on the same "don't narrow the base resource" principle).
But a blanket "Patient becomes optional everywhere" change (Disposition A)
silently rewrites the contract every deployed hub relies on for routing,
audit, and consent, and there is no community signal — no Zulip thread,
no GitHub issue/PR — that implementers are ready for that. Disposition B
captures the legitimate need (group-anchored encounters, patient-less
contexts) by adding a discrete, opt-in surface (a `Group`-targeted event
family or a profile variant) and standardizes the
"anchor-missing" case on `data-absent-reason`, which the reporter himself
proposed on FHIR-37089. That gives the work group a forward path that
honors the request without forcing every existing FHIRcast deployment to
re-validate its assumptions, and it lines up cleanly with the FHIR-48678
precedent of broadening context typing through additive, well-scoped
changes.
