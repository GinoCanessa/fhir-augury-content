## Summary

The ticket asks the FHIRcast specification to relax (or remove) the requirement
that a Hub advertise the exact set of supported events in its well-known
conformance statement. The reporter, Evan Schnell, argues that requiring a Hub
to enumerate its supported events undercuts event customization and prevents
useful pass-through behavior between applications that share a Hub. The
motivating use case is an integration where an EMR/RIS acts as the Hub and
mediates between an Image Viewer and a Dictation/Reporting system: those two
apps may want to exchange custom (or "content"-style) events that the EMR/RIS
itself does not understand, yet today the Hub's conformance statement would
block subscriptions to events it has not declared. The reporter wants the Hub
to be able to forward / pass on custom events regardless of whether the EMR
that hosts it actually consumes them.

There are no other Jira tickets cross-referenced to FHIR-36994; the only
related discussion is a Zulip thread ([FHIRcast > Hubs and Apps](https://chat.fhir.org/#narrow/stream/FHIRcast/topic/Hubs%20and%20Apps/near/284941341))
in which Evan expanded on the same concern and Isaac Vetter probed what a
"pass-through Hub" would actually be responsible for.

## Details

**Description:**

Currently, the [Hub conformance statement](http://hl7.org/fhir/uv/fhircast/2022May/2-7-Conformance.html#wellknown-endpoint)
specifies which events are supported. However, that is not so nice for a Hub,
if event customization is a feature. So can't we have a Hub just pass on
custom events regardless from if the EMR that hosts the Hub listens to those
events?

Consider integration between an ImageViewer and Dictation system with EMR/RIS
as hub. The EMR/RIS should propagate any events it does not recognize, right?

## Related Zulip Discussions

### [FHIRcast > Hubs and Apps](https://chat.fhir.org/#narrow/stream/FHIRcast/topic/Hubs%20and%20Apps/near/284941341)

Evan Schnell opened the topic to flag the same concern as FHIR-36994: apps
often assume "their" Hub will be the only Hub, but in real deployments an EMR
Hub and a departmental Hub may both be in play, each advertising a different
event set. Evan argued that constraining a Hub's conformance to only the
events its own driving app understands blocks all "following" apps from
expanding FHIRcast functionality, and he sketched complex multi-Hub / multi-WSS
proxy topologies that would otherwise be needed. Isaac Vetter pushed back by
asking what the responsibility of a "pass-through Hub" actually is — for
example whether `DiagnosticReport-update` events should derive
`ImagingStudy-update` or `patient-update`, and when such a Hub should ever
decline a subscription. Evan replied that the most elegant model is a Hub that
maintains an abstract graph of `Patient`/`Encounter` references, cascades
`*-close` events along that graph, and otherwise behaves as a primitive
forwarding middleware that never declines a subscription based on event type;
he also conceded that the broader "content" event design is still unsettled.
No consensus or formal disposition was reached in the thread.

Participants: Evan Schnell, Isaac Vetter.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Amend the FHIRcast Conformance section (currently §2.7 in the 2022May build,
the equivalent "Hub Conformance Statement / well-known endpoint" section in
the current build) to make event enumeration in the well-known
`fhircast-configuration` document non-exhaustive. Concretely:

- Change the wording around `eventsSupported` so that it advertises only the
  events the Hub itself understands and acts on, but explicitly states that a
  Hub MAY also accept subscriptions to, and route, events that are not listed
  ("pass-through events").
- Add a new boolean capability (e.g. `allowsUnknownEvents` or
  `passThroughEvents`) in the well-known configuration so that Apps can
  discover whether a given Hub will forward events it does not itself
  originate or interpret.
- Update the subscription-handling text so that a Hub conforming to this
  capability MUST NOT reject a subscription request solely because the
  requested event is not in `eventsSupported`.
- Add non-normative guidance describing the EMR-as-Hub + Viewer + Reporting
  integration as the motivating example, and clarify that pass-through events
  carry no semantic guarantees from the Hub (the Hub does not derive,
  cascade, or transform them).

#### Justification

This is exactly what the reporter asks for and what Evan elaborated on in the
Zulip thread: today the well-known conformance contract makes the Hub the
gatekeeper for the entire event vocabulary of every connected App, which is
inconsistent with FHIRcast's stated support for custom / experimental events
and with realistic multi-vendor deployments where an EMR/RIS Hub mediates
between specialty apps. Making event enumeration non-exhaustive — while
introducing an explicit `allowsUnknownEvents` capability — preserves the
current behavior for Hubs that want strict gating, but unblocks pass-through
Hubs without forcing every implementer to deploy a separate departmental Hub
and the proxy topology Evan sketched. It also keeps the change additive:
existing Hubs that do not advertise the new capability remain conformant.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the well-known `eventsSupported` list authoritative for events the Hub
**originates or validates**, but address the underlying need by introducing a
distinct, optional **"event relay" / "pass-through" Hub profile** rather than
loosening the base conformance rules. Specifically:

- Define a new conformance profile (e.g. "FHIRcast Relay Hub") whose
  well-known document advertises a `hubMode` of `relay` (or sets
  `passThroughEvents = true`).
- A Relay Hub MUST accept subscriptions to any event name that matches the
  FHIRcast event-name syntax, MUST forward matching `event-notification`
  payloads to subscribers without semantic interpretation, and MUST NOT be
  required to maintain anchor context for unknown events.
- Apps discover relay capability via the well-known document; Apps that
  require strict gating can refuse to use a Relay Hub.
- Leave the existing Hub conformance statement and `eventsSupported`
  semantics unchanged for non-relay Hubs.

#### Justification

This addresses the same use case (Viewer ↔ Reporting custom events through an
EMR/RIS Hub) without weakening the contract that today's Apps rely on when
they read `eventsSupported`. It also gives implementers and certifiers
something concrete to test against — "is this Hub a Relay Hub?" — instead of
the more fuzzy "this Hub may or may not accept events it does not list." It
directly answers Isaac Vetter's question in the Zulip thread about what a
pass-through Hub's responsibilities actually are: a Relay Hub's contract is
explicit and narrow (forward syntactically valid events; no semantic
guarantees; never decline a subscription on event-type grounds). Trade-off
versus Disposition A: it is a bigger spec change (new profile, new
discovery), but it avoids changing the meaning of `eventsSupported` for
existing deployments and keeps stricter Hubs first-class.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the current rule that the Hub's well-known
configuration enumerates the events the Hub supports, and that Hubs MAY
reject subscription requests for unlisted events. Close the ticket as "Not
Persuasive" with a note pointing implementers at:

- Per-installation `eventsSupported` configuration (the EMR/RIS Hub vendor
  can extend its supported events to include the custom events used by
  partner apps), and
- The existing FHIRcast extension/custom-event mechanism, which already lets
  vendors define custom event names without changing the spec.

#### Justification

The current model gives Apps a clear, discoverable contract: if an event is
not in `eventsSupported`, do not assume the Hub will route it. Loosening that
contract — or requiring Hubs to act as opaque relays for events they do not
understand — introduces real risks: safety (the Hub cannot enforce context
invariants such as `*-close` cascade for events it has never seen), security
(unbounded event names increase attack surface on the WSS channel), and
interop (Apps can no longer rely on the well-known document to negotiate
behavior). The reporter's motivating scenario can already be handled by
configuring the EMR/RIS Hub to declare the custom events the partner apps
need, which is a deployment-time concern rather than a spec change. Evan
himself acknowledged in the Zulip thread that the broader "content" event
design is still unsettled, so changing the conformance contract now would be
premature.

---

### Recommendation

**Recommended disposition:** B

Disposition B best balances the legitimate concern raised by the reporter
against the obligations a Hub takes on when it advertises an event. The
ticket and the Zulip thread together show that the real problem is not the
existence of `eventsSupported` per se — it is that today there is no way for
a Hub to honestly say "I will forward this, but I make no claims about it."
A new opt-in Relay Hub profile gives implementers exactly that knob, answers
Isaac's "what is the contract of a pass-through Hub?" question explicitly,
and avoids silently changing the meaning of `eventsSupported` for the many
existing strict Hubs (Disposition A's main risk). Disposition C is too
restrictive: requiring every EMR/RIS vendor to enumerate every partner app's
custom events as a deployment-time chore does not scale, and Evan's
multi-Hub/proxy workaround sketch shows what implementers will actually do
if the spec refuses to help them. The workgroup should treat B as the
starting point and use the discussion to nail down the Relay Hub's exact
forwarding rules (event-name syntax, anchor context handling, subscription
lifecycle) before balloting.
