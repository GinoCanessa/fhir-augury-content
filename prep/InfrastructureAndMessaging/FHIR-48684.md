## Summary

The FHIRcast `*-select` events (e.g., `DiagnosticReport-select`) carry a
context entry that identifies the items the user has selected within a
parent resource. The current FHIRcast specification of the context-entry
datatype rules does not list `resources` (an embedded array of FHIR
resources) or `references` (an array of `Reference`s) as an allowed
datatype for a select-event context entry, even though selecting more
than one item is the normal case. The reporter, Bas van den Heuvel,
asks the workgroup to close that gap by either (a) explicitly extending
the allowed context datatypes to permit `resources` / `references`
collections in select events, or (b) replacing the multi-item context
entry with a single `List` resource that itself enumerates the selected
items. A working branch on `HL7/fhircast-docs`
(`select-event-using-list`) demonstrates option (b) for
`DiagnosticReport-select` and the other `-select` events; the ballot
discussion on Zulip ties this to the parallel `FHIR-45719-Advanced-Reference`
branch which already moved `DiagnosticReport-update` from embedded
resources to `references`. No formally cross-referenced Jira tickets
were surfaced for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested — Extend the allowed context datatypes

#### Proposal

Update the FHIRcast specification's context-entry datatype rules
(currently used to constrain what may appear in the `context[].resource`
position of an event) so that `*-select` events explicitly permit a
multi-valued context entry of either:

- a collection of FHIR `Reference` objects (`references`), or
- a collection of inline FHIR resources (`resources`).

Concretely, in `3.6.4 DiagnosticReport-select` (and the analogous
sections for any other `*-select` events) define a `selections` (or
equivalently named) context key whose value is an array. State that
`Reference` is the recommended form, and that an inline-resource form
is permitted for compatibility with the existing single-resource
context pattern. Update the worked example, the schema/JSON examples,
and the conformance prose accordingly. No new resource type is
introduced; the change is localized to FHIRcast's own datatype rules.

#### Justification

This is the smallest spec change that closes the gap the reporter
identified: the underlying need is simply to allow more than one item
in a `-select` context entry, and the existing FHIRcast pattern already
uses `Reference` for everything else after FHIR-45719's "advanced
reference" work landed on `DiagnosticReport-update`. Extending the
datatype rules to allow a collection of references is a natural,
minimally-invasive continuation of that direction and is consistent
with how multi-valued context is handled elsewhere in messaging-style
FHIR specs. It also avoids forcing implementers to construct and
maintain a transient `List` resource just to convey a selection.

---

### Disposition B: Alternative Approach — Use a `List` resource

#### Proposal

Adopt the approach prototyped on the
[`select-event-using-list`](https://github.com/HL7/fhircast-docs/tree/select-event-using-list)
branch: keep the context-entry datatype rules unchanged (single
resource only) and instead specify that the `*-select` events convey a
single `List` resource as the context entry. The `List.entry[].item`
references then enumerate the selected items, and `List.subject` ties
the selection back to its parent resource (e.g., the parent
`DiagnosticReport`). Update `3.6.4 DiagnosticReport-select` and any
sibling `-select` event sections to require a `List` with a defined
profile (recommended `List.mode = working`, `List.status = current`,
and constrained `List.code` indicating "FHIRcast selection") and
publish the worked example already drafted in the branch.

#### Justification

This option keeps the FHIRcast context-datatype rules invariant and
re-uses an existing FHIR resource that was designed for exactly this
job (an ordered/unordered collection of references with provenance).
It centralises selection semantics (ordering, removed items via
`List.entry.deleted`, `List.date`, `List.source`) instead of inventing
those semantics inside FHIRcast. The reporter has already produced a
working preview, so the implementation cost is bounded and the
trade-off is clear: implementers gain expressive power at the cost of
constructing a `List` for every selection event. This option is most
attractive if the WG anticipates needing ordering, attribution, or
"item removed from selection" semantics — none of which fall out
naturally from a bare `references` array.

---

### Disposition C: Decline

#### Proposal

Close FHIR-48684 as "not persuasive" on the grounds that the existing
`-select` event pattern, combined with the
`FHIR-45719-Advanced-Reference` change already adopted for
`DiagnosticReport-update`, is sufficient: implementers can repeat the
single-context-entry pattern by sending one `-select` event per
selected item, or by selecting the parent resource and letting the
subscriber inspect it. No spec change.

#### Justification

The strongest argument for declining is parsimony: FHIRcast STU3 is
already in block-vote-3+ and adding a new datatype shape (Disposition
A) or a required `List` profile (Disposition B) at this stage risks
disturbing the ballot. If the WG judges that multi-item selection is
rare enough in practice to be handled by repeated single-item
`-select` events, the spec gap is more theoretical than operational
and can be deferred to a future release.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the most surgical fix that actually addresses the
reporter's complaint, and it lines up directly with the precedent the
WG already set in `FHIR-45719-Advanced-Reference`: prefer `Reference`
over embedded resources, and let the receiver resolve. The Zulip
thread shows no participant arguing against that direction — the
discussion mostly focused on block-vote logistics, and the reporter
himself extended the references-based PR to cover the `-select`
events. Disposition B (List) is defensible and worth taking seriously
**if** the WG concludes it needs ordering / "item removed" semantics
on selections, but absent a stated requirement for those features it
adds resource-construction overhead for every event without clear
payoff. Disposition C (decline) is hard to justify given that the
reporter has already produced a preview branch demonstrating the
problem is real and the fix is small. If the WG is ballot-skittish,
the safest path is to accept Disposition A but mark it for the next
FHIRcast release rather than block-voting it into STU3 mid-cycle.
