### Change Request

#### Topic: Patient/Group context anchor flexibility

Two near-identical tickets (filed by the same reporter at the same minute) asking that FHIRcast not require a `Patient` anchor on event contexts whose underlying FHIR resource (e.g., `Encounter`) already permits `Group` as its subject. They jointly request making `Patient` optional and adding `Group` as an allowed context resource so generic / group-oriented deployments can synchronize via FHIRcast. FHIR-45781 explicitly notes FHIR-45780 as an accidental duplicate.

### Change Request

#### Topic: Da Vinci Notifications messaging-paradigm extensions

Two requests to extend the Da Vinci Notifications (US Da Vinci Alerts) IG so it can be used in richer FHIR messaging scenarios beyond single, fire-and-forget notification POSTs. FHIR-26164 asks for guidance on batching multiple notifications (transaction/batch `Bundle`) and on bulk delivery via an `$import`-style asynchronous manifest. FHIR-50801 asks the IG's base `MessageHeader` profile to permit `MessageHeader.response` (currently `0..0`) so implementers can correlate ingestion-outcome responses (`OperationOutcome`) back to the originating notification.
