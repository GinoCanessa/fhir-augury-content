## Summary

The ticket, filed by Eric Haas against the Da Vinci Alerts (later Da Vinci
Notifications) FHIR IG, asks the authors to add a dedicated section to the
guide that describes how implementers should use FHIR **batch / transaction
Bundles** and **bulk data exchange** when sending alerts/notifications. The
underlying concern is that the IG, as balloted, does not give guidance for
delivering more than a single notification at a time, nor for situations where
the payload is too large to push synchronously inside one Bundle. In the
ticket's own comments, the reporter suggests that very large payloads could be
handled with an asynchronous `$import`-style operation, where the server
prepares a notification and posts a manifest to a bulk-import endpoint
(referencing the SMART `bulk-import/import-manifest.md` proposal).

The ticket has had a winding disposition history: an earlier vote (2020-05-07,
Newman/Knapp 3-0-1) marked it *Considered for Future Use* with a note that it
should be reconsidered "in a future version of this IG if there is demand".
That resolution was reverted, and the ticket eventually reached its current
resolution of **Persuasive with Modification** (resolved 2022-10-05) with the
direction: *"Batching is on the Notifications roadmap for the next version.
Determine the need for bulk (see comments for how to do bulk)"*. It carries
the `Block-Vote-20221220` label, indicating it was disposed as part of an INM
block vote on 2022-12-20. No cross-referenced Jira tickets were found and no
GitHub PR/issue/commit links to it; the only external context is two Da Vinci
ballot/block-vote summary threads on Zulip.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new normative-text section to the Da Vinci Notifications IG (the
successor of the balloted Da Vinci Alerts IG) titled **"Batch transactions and
bulk data"**, located alongside the existing transport / messaging guidance
pages. The section should:

1. Describe how senders MAY package multiple notifications in a single
   `Bundle` of `type = batch` (or `transaction` where atomic processing is
   required), one `MessageHeader`/notification per entry, with explicit
   rules on entry ordering, `fullUrl` resolution, and conformance of each
   entry to the IG's notification profile(s).
2. Describe an asynchronous bulk path for cases where the combined payload
   is too large for a synchronous Bundle POST: server prepares a manifest of
   notification artifacts and POSTs the manifest to a `$import` endpoint on
   the receiver, following the SMART bulk-import manifest proposal
   (`https://github.com/smart-on-fhir/bulk-import/blob/master/import-manifest.md`).
   Document required manifest fields, security expectations
   (SMART Backend Services), and acknowledgement / error semantics.
3. Add conformance language stating that batch and bulk delivery are
   OPTIONAL capabilities and, when claimed, MUST behave as described.

#### Justification

This is the literal request of the ticket and is consistent with the resolved
text that "Batching is on the Notifications roadmap for the next version".
Eric Haas's own comment lays out exactly the bulk approach (manifest +
`$import`) that has subsequently matured in the SMART ecosystem, so the
author has a concrete reference implementation to base the text on. Both
batch transport (via `Bundle`) and bulk transport (via `$import`-style
async manifests) are well-established FHIR patterns, and giving implementers
a single normative section avoids ad-hoc decisions about how to send
high-volume notifications.

---

### Disposition B: Alternative Approach

#### Proposal

Split the request into two distinct artefacts:

1. **In the IG body**, add a *short* "Sending multiple notifications" section
   that simply states notifications MAY be batched in a `Bundle` of `type =
   batch`, with a worked example, and references the FHIR core specification
   for transaction/batch semantics. Do **not** invent IG-specific transaction
   rules.
2. **Defer the bulk path** to a separate, narrowly-scoped companion guide (or
   a new ticket) that profiles SMART bulk-import for notifications. Include
   only a non-normative pointer in the IG saying "for bulk delivery of
   notifications, see [companion]" until that companion exists.

#### Justification

The ticket's resolution explicitly says *"Determine the need for bulk"*,
which signals that workgroup consensus on the bulk path is not yet there. The
comment trail also shows that this issue has bounced between *Considered for
Future Use* and *Persuasive with Modification* over multiple years, which is
typically a sign that the requirements are not stable enough to commit to
normative IG text. Splitting the work lets the IG ship a small, low-risk
batch clarification immediately while keeping the more controversial bulk /
`$import` design out of the critical path. It also avoids the IG re-defining
transport mechanics that arguably belong in a horizontal Da Vinci or HL7
infrastructure guide rather than in an alerts-specific IG.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **already addressed** by core FHIR mechanisms and out of
scope for the Da Vinci Notifications IG. Rationale: any FHIR server that
accepts a notification interaction can already accept it inside a `Bundle` of
`type = batch` or `transaction` per the FHIR core spec, and bulk delivery is
already covered by the FHIR Bulk Data Access IG and the SMART bulk-import
proposal. The Da Vinci Notifications IG would only need to re-state these,
which adds maintenance burden without adding interoperability value.

#### Justification

Declining is defensible because (a) the original request is very generic
("Add Section on batch transactions and bulk data") and the workgroup
explicitly noted in 2020 that the change should only be made *"if there is
demand"*; (b) the only substantive bulk proposal in the comments points to
an external SMART project rather than to a Da Vinci-specific need; (c)
duplicating core/Bulk-Data guidance inside an alerts IG risks drift from the
canonical specs and creates a forking point that implementers will have to
reconcile later. If a concrete, in-the-wild use case appears, a fresh,
narrowly-scoped ticket can be filed against the next IG cycle.

---

### Recommendation

**Recommended disposition:** B

The current resolution on the ticket (*Persuasive with Modification —
batching is on the next-version roadmap; determine the need for bulk*) is, in
substance, Disposition B: commit to the batch piece, but treat the bulk
piece as an open question rather than a deliverable. The history of this
ticket — vote, revert, re-disposition over more than two years, no concrete
implementer demand surfaced in Zulip beyond the original reporter — makes a
full Accept-As-Requested (Disposition A) premature: the IG would be writing
normative bulk-import text on the strength of one SMART project link and one
comment thread. At the same time, an outright Decline (Disposition C)
ignores the legitimate gap that nothing in the IG today tells implementers
how to send more than one notification at once, which is a reasonable
question for any production deployment. Disposition B captures both
realities: a small, low-risk batch clarification in the IG now, and a
deferred, ticket-tracked decision on bulk once a real implementer use case
exists. This also aligns with the I&M block-vote framing the ticket carries
(`Block-Vote-20221220`), where the workgroup has historically preferred to
ship the modest, well-understood part of a request and re-open the
controversial part.
