## Summary

The ticket asks the FHIRcast specification to clarify what "current context"
means for the `Get Current Context` operation, particularly in non-trivial
session histories. The submitter points out that the rule "the context
established in the most recent open operation is the current context" needs
to be stated not only on the `Get Current Context` page but also in the
Multi-tab Considerations and Multiple Anchor Considerations sections, since
the concept is broader than a single operation. The submitter further asks
what the current context should be when a series of opens and closes leaves
ambiguity (e.g., three opens followed by a close of the most recent one) and
asks for precise wording for the response when no context has yet been
established. No other Jira tickets were returned as direct or cross-referenced
items; one Zulip cross-reference points to a GitHub notification bundle that
also referenced FHIR-37008 and FHIR-37083 as being addressed by the same PR
effort.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the FHIRcast specification to state, consistently across the `Get
Current Context` page, the Multi-tab Considerations section, and the
Multiple Anchor Considerations section, that:

1. The *current context* is the context established by the most recent
   successful `*-open` request that has not been superseded by a subsequent
   `*-open` or invalidated by a matching `*-close`.
2. When a `*-close` removes the most recently opened context, the *current
   context* becomes the next-most-recent still-open context (i.e., context
   is managed as a LIFO stack of open contexts per anchor / per tab).
3. If no context has ever been established, or all opened contexts have
   been closed, the response to `Get Current Context` is a well-formed
   empty context notification (an event of the configured `*-open` type
   with an empty `context` array, or the equivalent "no context"
   representation defined for the anchor type), not an HTTP error.

Work this normative clarification into all three pages so that Multi-tab
and Multiple Anchor consumers see the same rule, and add the worked
example from the submitter's comment (A/B/C/B, then close B → current is
C) as a non-normative illustration.

#### Justification

This is what the reporter asked for and it matches the resolution already
recorded on the ticket ("Persuasive with Modification — Included with
PR486"). The LIFO-stack interpretation is consistent with the existing
"most recent open" rule the reporter quotes, and defining an empty-context
response removes an interoperability failure mode (implementers returning
404/400/empty body in incompatible ways). Placing the rule in Multi-tab
and Multiple Anchor Considerations — not only on `Get Current Context` —
reflects the reporter's observation that the concept is broader than a
single operation.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the normative rule narrowly on the `Get Current Context` page, but
add a short shared definition — "current context" — to the FHIRcast
glossary / core concepts section, and have Multi-tab Considerations and
Multiple Anchor Considerations link to that glossary entry rather than
restating the rule. In that glossary entry, define current context as
"the context established by the most recent successful `*-open` that has
not been closed or superseded, tracked per anchor; empty when no such
context exists." Also specify a single canonical empty-context payload
shape and require `Get Current Context` to return it (rather than an
error) when the stack is empty.

#### Justification

This achieves the same clarifying effect without duplicating normative
prose in three places, which reduces the risk of the three locations
drifting out of sync over future ballot cycles. The glossary approach is
consistent with how FHIRcast already centralizes cross-cutting concepts
(subscription, anchor, context) and keeps the Multi-tab and Multiple
Anchor pages focused on their specific considerations rather than
re-deriving a definition that is truly spec-wide.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the existing statement on the `Get Current
Context` page — "the context established in the most recent open
operation is the current context" — is already sufficient, and that
Multi-tab and Multiple Anchor Considerations are deliberately
non-normative discussion sections that should not repeat normative
language. The open/close sequencing question (A/B/C/B + close B) is
implicitly answered by the existing rule if implementers treat
"most recent open" as the most recent *still-open* open. Clarify the
no-context response informally in a worked example rather than changing
normative text.

#### Justification

Declining is defensible only if the workgroup judges that the current
wording is unambiguous to a careful reader and that further normative
text risks over-specifying implementation internals (e.g., mandating a
stack data structure). It also avoids re-opening the pages for what is
fundamentally an editorial clarification.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been resolved as *Persuasive with Modification*
with the change rolled into PR486 (and bundled into the `HL7/fhircast-docs`
PR #436 workflow with FHIR-37008 and FHIR-37083 per the Zulip
notification), so Disposition A aligns with the resolution of record and
with the submitter's concrete scenario. The reporter's close-of-current
scenario demonstrates that the existing one-liner on `Get Current Context`
is genuinely ambiguous to implementers (Disposition C's premise does not
hold up), and while Disposition B's glossary-centralization is attractive
as a long-term editorial strategy, it is a larger refactor than this
ticket scopes and would leave Multi-tab and Multiple Anchor readers
without the worked example that the submitter specifically asked for.
Disposition A delivers the clarification the workgroup already accepted,
in the three places the reporter identified, plus a normative
empty-context response that closes an interop gap.
