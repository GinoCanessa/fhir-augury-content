## Summary

The reporter requests that FHIRcast not require a `Patient` context for
generic FHIRcast deployments. The argument is that the FHIR `Encounter`
resource permits its `subject` to be either `Patient` or `Group`, and
therefore an `Encounter` may legitimately exist without a `Patient`.
Because FHIRcast currently mandates a `Patient` anchor (e.g., as part of
its workflow event contexts that include `Encounter`), it is judged to
narrow the underlying FHIR specification more than necessary. The
specific request is to (a) make `Patient` optional in the relevant
FHIRcast event contexts, and (b) add `Group` as an allowable subject so
that `Encounter`-only or `Group`-only sessions (e.g., veterinary,
multi-patient teaching, or population-oriented workflows) can be
synchronized via FHIRcast.

No cross-referenced Jira tickets were returned for FHIR-45780, so the
summary above is built from this ticket alone.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify the FHIRcast specification so that, in event contexts that
currently mandate a `Patient` anchor (notably the `*-open` /
`*-close` events whose context array lists `Patient` with cardinality
`1..1`), the `Patient` entry is relaxed to `0..1`, and a new context
entry of type `Group` is added (also `0..1`). Add a normative
constraint that at least one of `Patient` or `Group` SHALL be present
when an `Encounter` is in the context (mirroring `Encounter.subject`'s
own `Patient | Group` choice), and that for events that already
require an anchor resource (e.g., `ImagingStudy-open`) the existing
anchor remains the authoritative subject. Update the FHIRcast event
catalog, the context-resource table, and the Security & Authorization
section to reflect that the subscriber may be authorized for either a
`Patient` or `Group` scope.

#### Justification

`Encounter.subject` in core FHIR is `Reference(Patient | Group)`, so
FHIRcast's current `Patient`-only constraint is strictly narrower than
the spec it synchronizes. Aligning FHIRcast with the underlying
resource model removes a profile-level restriction the workgroup did
not intend and unblocks legitimate generic deployments (veterinary
imaging, group-oriented teaching/population workflows, situations
where patient identity is intentionally deferred). The change is
additive at the wire level — existing `Patient`-bearing senders and
subscribers continue to work — and only consumers that today reject
events lacking `Patient` would need to update their validation.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the base FHIRcast event contexts as-is (with `Patient` required
in the general case), and instead define a small set of
**generic-deployment event variants** — e.g., `Encounter-open` /
`Encounter-close` and a `Group-open` / `Group-close` pair — whose
context anchor is `Encounter` or `Group` rather than `Patient`. These
variants would be advertised through capability negotiation
(subscription `events`/`hub.events`), so a Hub or subscriber that does
not support the generic variants is unaffected. The "classic"
`Patient`-anchored events remain unchanged for the dominant clinical
case.

#### Justification

The vast majority of deployed FHIRcast hubs assume a `Patient` anchor
for security scoping, audit, and UI synchronization, and many
implementations have hard-coded that assumption. Relaxing
`Patient` to `0..1` on the existing events (Disposition A) silently
changes the contract for every existing subscriber, even those that
will never see a `Patient`-less event. Introducing parallel events
keeps the change opt-in: a generic deployment opts into the new event
names, and traditional patient-centric vendors are not forced to
re-validate their event handlers against a newly nullable field. It
also makes the security boundary explicit — a `Group`-anchored
session is plainly a different scope than a `Patient`-anchored one.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the current FHIRcast normative
surface. Document in the FHIRcast specification that `Patient` is
intentionally required as the anchoring subject of a context session,
and direct generic / non-Patient deployments to use a non-FHIRcast
synchronization mechanism (e.g., a custom WebSocket protocol, or a
profile of FHIRcast that the deploying organization controls).

#### Justification

FHIRcast's design centers on patient-context synchronization across
clinical applications; the `Patient` anchor underpins authorization
scopes (`patient/*.read`-style scopes), audit, and the user mental
model of "the apps are showing the same patient." Removing or
weakening that anchor risks ambiguous events that subscribers cannot
safely act on, complicates security analysis, and pushes FHIRcast
toward a generic pub/sub it was not designed to be. Without
implementer evidence of demand beyond the single reporter, declining
preserves spec coherence.

---

### Recommendation

**Recommended disposition:** B

Disposition B (parallel `Encounter`- / `Group`-anchored event
variants gated by capability negotiation) is the recommended starting
point. It honors the reporter's underlying need — FHIRcast should not
artificially narrow `Encounter.subject`'s `Patient | Group` choice —
without silently changing the contract for every existing
`Patient`-anchored deployment, which Disposition A would do. It also
makes the security and authorization story explicit: a session
anchored on `Group` is a meaningfully different scope from a session
anchored on `Patient`, and naming the events differently surfaces
that distinction in subscription requests rather than burying it in
nullable context fields. Disposition C is too restrictive given that
the request maps cleanly onto an existing FHIR cardinality choice.
The workgroup should weigh whether the additional event names are
worth the spec surface, and whether equivalent capability negotiation
could instead be expressed as a flag on the existing events; either
refinement is compatible with the spirit of B.
