## Summary

The reporter (Ulrike "Riki" Merrick) asks the Da Vinci Notifications IG
authoring team to revisit the base DaVinci Notifications `MessageHeader`
profile, which currently constrains `MessageHeader.response` to `0..0`.
That constraint forbids the profile from being used on response
messages, which is incompatible with allowing implementers to use the
IG in the FHIR messaging paradigm where a notification triggers a
correlated response message carrying an `OperationOutcome`. The ticket
proposes two alternative remediations: (a) relax / remove the `0..0`
slicing on `MessageHeader.response` in the base notification profile so
the same profile can carry a `response.identifier` plus
`response.details`, or (b) keep the request-side profile as-is and add
a sibling response-side `MessageHeader` profile in which `.response`
is `1..1`. The motivating use case is documented in the cross-referenced
Zulip thread "Feedback loop for ADT notifications" on the Da Vinci
Alerts stream, where an implementer (athenahealth / Siddharth Tarey)
described an asynchronous feedback workflow that needs exactly this
shape of response message.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Da Vinci Alerts (DaVinci Notifications) IG, **relax the
`MessageHeader.response` constraint on the base notifications
`MessageHeader` profile from `0..0` to `0..1`** (the FHIR core
cardinality), so that the same profile may be used for both the
outbound notification message and the correlated response message. Keep
all existing required elements on the request side
(`eventCoding`, `source.endpoint`, `focus`, etc.) unchanged. Add
profile-level guidance that:

- when `response` is absent, the `MessageHeader` represents the
  initiating notification;
- when `response` is present, `response.identifier` SHALL carry the URI
  of the previously received notification message and
  `response.details` SHALL reference an `OperationOutcome` describing
  the ingestion result.

Update the IG narrative pages on messaging usage, and add an example
response Bundle (MessageHeader + OperationOutcome) mirroring the
payload Siddharth Tarey described in the Zulip thread.

#### Justification

This matches what the reporter literally asked for and is the smallest
mechanical change: a cardinality relaxation on a single element plus
narrative. It aligns the IG with the FHIR core messaging paradigm
(`MessageHeader.response` is the standard correlation mechanism in
core), so implementers using the IG in messaging mode do not have to
deviate from the profile to carry a correlated `OperationOutcome`.
Because the change is from `0..0` to `0..1`, all currently-conformant
notification messages remain conformant — there is no breaking impact
on existing senders.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing notification `MessageHeader` profile strictly for
**outbound notification** messages (leave `.response` at `0..0`), and
introduce a **sibling profile** — e.g.
`DaVinciNotificationsResponseMessageHeader` — in which:

- `response` is `1..1`,
- `response.identifier` is `1..1` and SHALL carry the URI of the
  originally received notification,
- `response.details` is `1..1` and SHALL reference an
  `OperationOutcome`,
- `response.code` is `1..1` (constrained to `ok` | `transient-error` |
  `fatal-error` per core),
- and elements that only make sense on the request side (e.g. specific
  `eventCoding` slices, mandatory `destination` semantics) are relaxed
  or replaced with response-appropriate constraints.

Publish a paired example Bundle and link the two profiles from the
messaging-usage page so implementers can pick the right one by
direction.

#### Justification

A directional split makes conformance unambiguous: validators can flag
a "response Bundle" that lacks `response.identifier` / `response.details`,
which is impossible if both directions share a profile that merely
permits `response` to be present. It also matches how other FHIR IGs
that use messaging (e.g., several IHE / Da Vinci request-response IGs)
model request and response separately. The trade-off is more authoring
surface: two profiles to maintain, two examples, and additional IG
narrative explaining when to use which — plus the need to decide which
elements on the request profile are actively wrong (vs. merely
unnecessary) for responses.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the DaVinci Notifications IG is **scoped to
fire-and-forget ADT-style notifications**, not to bidirectional FHIR
messaging. Implementers who need to confirm ingestion of a notification
should use the underlying transport (HTTP status + `OperationOutcome`
returned synchronously to the POST) or an out-of-band channel, rather
than a correlated response `MessageHeader`. Recommend closing the
ticket as "Not Persuasive — out of scope".

#### Justification

This is defensible if the WG considers the IG's intended interaction
pattern to be REST-style POST of a notification Bundle, where the
synchronous HTTP response (with optional `OperationOutcome`) already
provides ingestion feedback at the transport layer. Adding a response-
side `MessageHeader` would pull the IG into the messaging paradigm
end-to-end (asynchronous correlation, message routing, idempotency
expectations on `response.identifier`) which is a larger conformance
surface than the IG currently commits to.

---

### Recommendation

**Recommended disposition:** A

The reporter is the IG editor / FMG-experienced contributor who already
fielded the implementer question on Zulip; she filed the ticket against
her own profile precisely because real implementers (athenahealth) are
running into the `0..0` constraint while building exactly the kind of
correlated response the FHIR core messaging paradigm is designed for.
Disposition A is the lowest-risk way to unblock them: relaxing
`0..0` → `0..1` is non-breaking for existing senders and does not
commit the IG to publishing or maintaining a second profile. Disposition
B is attractive in the long run for validator strictness, but can be
adopted later as a compatible refinement once the WG sees how
implementers actually populate the response side. Disposition C
contradicts the reporter's stated goal ("if we want to allow people to
use the IG in the messaging paradigm") and would leave athenahealth-
class implementers without a profile-conformant way to send the
response Bundle they have already designed.
