## Summary

The reporter (Brian Pech) asks the FHIRCast editors to expand the narrative
in section 3.6.3 ("sharing discrete data between applications"). The
existing example uses a `DiagnosticReport` whose Impression Narrative
contains a "simple cyst" finding, but the spec does not explain how a
downstream subscriber (e.g., an EHR/medical-record app) is expected to
consume that discrete content, nor what semantics the timing of the
arrival of such an event carries — for example, whether receipt of the
event should update an in-progress report on the receiver. The request is
purely editorial/clarifying: add more context to the prose around the
existing example so subscribers understand the consumption model and the
update semantics. No cross-referenced Jira tickets, Zulip threads, or
GitHub items were surfaced for this ticket via FHIR Augury.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the prose in FHIRCast §3.6.3 ("sharing discrete data between
applications") around the existing `DiagnosticReport` "simple cyst"
example to explicitly cover:

1. **Who is expected to consume the discrete payload.** Add a sentence
   clarifying that any subscribed application — including downstream
   record systems such as an EHR — that has subscribed to the relevant
   FHIRCast event (e.g., `DiagnosticReport-update`) MAY consume the
   embedded discrete content according to its own use case, and is not
   limited to the originating reporting application.
2. **What event arrival means semantically.** Add explicit narrative
   stating that a FHIRCast event is a notification of context/state at
   a point in time and is not, by itself, a directive to persist or
   modify the receiver's stored copy of the report. Whether a
   subscriber updates an in-progress or stored report on receipt is a
   subscriber-side policy decision, not a FHIRCast guarantee.
3. **Timing / ordering caveats.** Add a short note that FHIRCast does
   not guarantee delivery ordering or exactly-once semantics across
   subscribers, so subscribers MUST be prepared to reconcile late,
   duplicate, or out-of-order events when deciding whether the
   contained discrete data should overwrite, merge with, or be ignored
   relative to data they already hold.
4. **Pointer to the authoritative source.** Reinforce the existing
   FHIRCast principle that the resource referenced by the context is
   the authoritative state; the discrete payload in the event is a
   convenience snapshot, and consumers that need a guaranteed-current
   view SHOULD re-read from the FHIR server.

#### Justification

The change is purely editorial and is exactly what the reporter asked
for: the existing example shows discrete data being carried but does
not say what a subscriber is supposed to do with it. The four points
above mirror language already used elsewhere in FHIRCast (subscriber
autonomy, the "context is a notification, not a command" model, and
the authoritative-resource principle), so the addition is consistent
with the rest of the specification and does not introduce new
normative behaviour. A Technical Correction at "Highest" priority is
the appropriate vehicle because the gap is in the prose, not in any
event or schema definition. Repo target: the FHIRCast spec source
(typically `HL7/fhircast-docs`); no FHIR core or IG content is
affected.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) extending §3.6.3 prose only, refactor
the discrete-data discussion into a small dedicated subsection — e.g.,
"3.6.3.x Subscriber consumption of discrete content" — and:

- Move the existing `DiagnosticReport` "simple cyst" example into that
  subsection.
- Add a normative-style table that lists, for each common subscriber
  role (originating reporting app, peer reporting app, EHR / record
  system, analytics/secondary use), the expected consumption pattern
  (display only, merge into local state, defer to authoritative
  resource, ignore, etc.) as **non-binding guidance**.
- Add an explicit "Event arrival is not an update directive" callout
  box near the example, plus a forward-pointer from §3.6.3 to the new
  subsection.

#### Justification

The reporter's questions ("would this be consumable by a downstream
app?", "would the timing of the arrival update an existing report?")
suggest the underlying issue is structural: the consumption model is
implicit and scattered, not just under-described in one paragraph.
Pulling it into its own subsection makes the answer discoverable for
future readers who land on §3.6.3 from search or from the
`DiagnosticReport-update` event description, and the role-keyed table
gives implementers a concrete artefact to point at during interop
testing. Trade-off vs. Disposition A: more editorial work and a small
section-numbering churn in the FHIRCast TOC, but a materially clearer
result for implementers and a better foundation if future tickets
extend the same topic.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Persuasive with Modification" → "no normative
change required", on the basis that FHIRCast already documents (a) the
event-driven, notification-only semantics of context updates and (b)
the principle that the FHIR resource referenced by the context is the
authoritative state. Close with a short comment pointing the reporter
at the existing FHIRCast sections that cover subscriber autonomy and
authoritative-resource semantics, and note that subscriber-side
behaviour (whether to update an in-progress report on event arrival)
is intentionally out of scope for FHIRCast.

#### Justification

A defensible decline rests on the position that the ticket describes
an editorial gap, not a specification defect, and that the answers the
reporter is looking for are already implicit in the FHIRCast
event/notification model. This is the weakest of the three
dispositions because the reporter explicitly says "just a little more
context would help" — i.e., they are flagging exactly the kind of
clarity issue that warrants a small editorial fix rather than a
principled decline. It is included for completeness so the workgroup
can weigh it, e.g., if there is a deliberate desire to keep §3.6.3
short and to push consumption-model guidance to a future companion
implementation guide.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers the reporter's request, is small in
scope, fits the "Technical Correction" classification, and does not
introduce new normative behaviour — it just makes existing FHIRCast
principles (subscriber autonomy, authoritative-resource, event
ordering caveats) explicit at the point in §3.6.3 where new readers
encounter the discrete-data example. Disposition B is attractive if
the workgroup believes the consumption model deserves its own
subsection, but that is a larger editorial effort and can be done as
a follow-up once the immediate clarity gap is closed. Disposition C
should be rejected: the reporter is pointing at a real ambiguity, and
declining a low-cost clarification on a "Highest" priority Technical
Correction would be hard to defend at ballot. If the workgroup wants
to hedge, accept A now and open a follow-up ticket to consider the
B-style restructuring during the next FHIRCast cycle.
