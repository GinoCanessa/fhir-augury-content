## Summary

The reporter questions why the FHIR `messageheader-response-request` value set
(used by the corresponding extension in the FHIR Extensions Pack, which mimics
the V2 acknowledgment-request construct) defines its own set of codes
(`always`, `on-error`, `never`, `on-success`) instead of reusing the
pre-existing V2 codes from
[v2-0155](https://terminology.hl7.org/ValueSet-v2-0155.html)
(`AL` = always, `ER` = on-error, `NE` = never, `SU` = on-success). The
semantic mapping between the two sets is one-to-one. The request is to
align the extension's binding with the V2 code system rather than maintain a
parallel FHIR-specific set. There are no cross-referenced Jira tickets that
add further context — the only cross-reference is a Zulip tracking list of
ballot items for FMG and does not contain substantive discussion of this
issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-bind the `messageheader-response-request` extension's value to the V2
code system `http://terminology.hl7.org/CodeSystem/v2-0155` (codes `AL`,
`ER`, `NE`, `SU`), and either retire or deprecate the FHIR-specific
`http://hl7.org/fhir/messageheader-response-request` CodeSystem and
ValueSet in the Extensions Pack. Provide a ConceptMap mapping the legacy
codes (`always` → `AL`, `on-error` → `ER`, `never` → `NE`, `on-success`
→ `SU`) so existing instances can be translated.

#### Justification

The reporter's observation is correct: the extension explicitly mimics a
V2 construct (MSH-16 / acknowledgment-request) and the four codes are a
1:1 semantic match for v2-0155. Aligning bindings with the existing V2
code system reduces duplication, supports cross-paradigm interoperability
between V2 and FHIR messaging, and is consistent with the broader
HL7 effort to harmonize V2 and FHIR terminology where the underlying
concepts are identical. THO already curates v2-0155, removing a
maintenance burden from the Extensions Pack.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing FHIR `messageheader-response-request` CodeSystem and
ValueSet (preserving backward compatibility), but publish a normative
ConceptMap between it and v2-0155, and update the extension's
documentation to explicitly note the equivalence. Optionally, add the V2
codes as additional designations / equivalent codes on the FHIR
CodeSystem entries (e.g., via property or `concept.designation`) so
implementers translating from V2 have a clear, machine-readable bridge
without changing the binding.

#### Justification

This addresses the reporter's underlying concern — discoverable, formal
alignment between the two code sets — without introducing the breaking
change Grahame flagged in his comment. Existing FHIR instances using
`always`/`on-error`/`never`/`on-success` continue to validate. V2 ⇄ FHIR
gateways gain a normative mapping artifact. This is consistent with how
HL7 has handled other "FHIR re-coined a V2 concept" situations
(AdministrativeGender, MaritalStatus, etc.) where the FHIR codes were
retained but mappings were published.

---

### Disposition C: Decline (Persuasive with Modifier — "Not Persuasive: Breaking Change")

#### Proposal

Decline the request to re-bind the extension to v2-0155 because doing so
is a breaking change for every existing implementer of the
`messageheader-response-request` extension and the analogous R5
MessageHeader.response.details element. Mark the ticket as Not
Persuasive (or Persuasive with Modifier if combined with Disposition B's
ConceptMap) on the basis that the existing FHIR codes are stable, in
active use, and semantically sufficient.

#### Justification

The codes have been in the FHIR specification across multiple normative-
adjacent versions and in production use. Changing the binding — even to
a semantically equivalent code system — invalidates instances, validators,
profiles, and downstream code that pattern-matches on the FHIR code
strings. As Grahame Grieve noted on the ticket, the merits of the
alignment argument do not outweigh the cost of a breaking terminology
change, especially in the Extensions Pack where stability of long-lived
extensions is a core promise.

---

### Recommendation

**Recommended disposition:** B

The reporter raises a legitimate harmonization concern, and the lone
substantive comment on the ticket (Grahame Grieve) correctly identifies
the dominant constraint: re-binding is a breaking change. Disposition B
threads that needle — it preserves backward compatibility for every
existing implementer (avoiding the harm Disposition C is rightly worried
about) while delivering the practical benefit the reporter is asking for:
a clear, formal, machine-readable bridge between the FHIR codes and
v2-0155. A published ConceptMap and explicit documentation of the
equivalence give V2/FHIR gateway implementers everything they need
without forcing a churn event on the rest of the community. Disposition
A is the "right" answer in a greenfield world but is not justified given
the deployed footprint; Disposition C alone leaves the harmonization gap
unaddressed and is harder to defend on its own merits.
