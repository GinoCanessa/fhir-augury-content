## Summary

The submitter is reviewing a FHIRCast specification page that defines an
extension mechanism scoped narrowly to the events documented on that
page. They argue that this *event-context-specific* extension approach is
the wrong layer at which to define extensibility, and propose two
options: (1) replace it with a **generic extension support** mechanism —
ideally adopting the existing FHIR extension model — so that all
FHIRCast events share one consistent way to carry additional data, or
(2) move the discussion of extensions out of the page-specific section
and into the cross-cutting **section 2.3 "Event Format"**, which already
defines the shape of every FHIRCast event notification. The request is
filed against the FHIRCast specification under Infrastructure &
Messaging, marked Highest priority but currently Triaged with no
assignee.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items recorded in FhirAugury for this ticket, so the analysis below is
based solely on the ticket's own description and the structure of the
FHIRCast specification.

## Proposed Dispositions

The dispositions below cannot be grounded in a `repo-analysis` briefing
because no GitHub cross-references are recorded for this ticket. They
are anchored instead in the FHIRCast specification structure that the
ticket itself names (the per-event page and section 2.3 "Event Format").
If/when the FHIRCast spec repository is associated with this ticket via
a cross-reference, the dispositions should be tightened to name the
specific authoring file(s).

### Disposition A: Accept As Requested

#### Proposal

Remove the event-page-local extension mechanism from the affected
FHIRCast event page and replace it, in section 2.3 "Event Format", with
a single generic extension mechanism that applies to every FHIRCast
event. Concretely:

- Add a normative subsection under 2.3 (e.g., "2.3.x Extensions")
  stating that any FHIRCast `event.context` entry MAY carry a FHIR
  `extension` array on its `resource`, following standard FHIR
  extensibility rules (URL-identified extensions, modifierExtension
  semantics, slicing/profiling considerations).
- For non-resource context elements that FHIRCast adds (e.g., the
  envelope's `event` object), define a parallel `event.extension`
  array, again keyed by absolute URL, with the same modifier-vs-non-modifier
  distinction FHIR uses.
- Delete the page-local extension language on the affected event page
  and replace it with a single sentence pointing to 2.3.
- Update any other event pages that have copied the same page-local
  pattern, so that the spec ends up with exactly one extension story.

#### Justification

This is the option the reporter explicitly asks for, and it aligns
FHIRCast with how the rest of the FHIR ecosystem handles open-world
extensibility. A single mechanism in section 2.3 is easier to test,
profile, and reason about: subscribers and hubs need to implement
extension handling (including modifier-extension fail-safe behavior)
once, not per event. Adopting the FHIR extension model also lets the
community reuse existing extension URLs and tooling instead of inventing
FHIRCast-only equivalents.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the extensibility text close to the event it documents, but
*generalize* it by extracting the common rules into a reusable
"Extensibility Pattern" subsection of 2.3 "Event Format" and have each
event page reference that pattern instead of redefining it. Specifically:

- Add a short, normative "Extensibility" subsection to 2.3 that defines
  *how* FHIRCast events may be extended (URL-identified extensions on
  resource-typed context elements via FHIR `extension`, plus a clearly
  named slot — e.g., `event.context-extension` — for non-resource event
  metadata), but does not enumerate the actual extensions.
- Leave each event page free to *list* the concrete extensions it
  recognizes, but require those lists to conform to the 2.3 pattern and
  to use the FHIR extension model rather than ad-hoc JSON properties.
- Provide one worked example per event family in 2.3, then cross-link
  from the event pages.

#### Justification

This preserves the editorial value of having an event page that
describes everything an implementer needs for that event in one place,
which is one of FHIRCast's stated readability goals, while still solving
the underlying problem the reporter raises: that extensibility is
currently defined inconsistently, page-by-page. It is less of a
breaking change than Disposition A because existing event-specific
extensions remain documented where implementers already look for them,
but they become instances of one shared pattern rather than parallel
inventions. This is also lower-risk if the FHIRCast WG is not yet ready
to commit to a single spec-wide JSON shape for non-resource extensions.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently filed and ask the reporter to point at
the specific page they consider problematic and the specific
event-context-specific extension text they want removed. Without that,
the ticket cannot be acted on without guessing which of FHIRCast's event
pages it targets, and the change risks being scoped wrong.

#### Justification

The ticket is a single sentence that says "this page is event context
specific" without identifying *which* page, and it offers two materially
different remedies (adopt FHIR's extension model vs. relocate the text
to 2.3) without choosing. Acting on it as-is risks editing the wrong
page or producing a generic mechanism that does not actually replace the
specific behavior the affected event needed. A clarification round will
either (a) produce a focused change that can land cleanly, or (b)
demonstrate that the request is already addressed by current spec text,
in which case it can be closed as Not Persuasive.

---

### Recommendation

**Recommended disposition:** A

The reporter is raising a real consistency problem that will only get
worse as FHIRCast adds more events: every event page that defines its
own extension slot is a new bespoke extensibility surface that
implementers, profilers, and downstream IGs have to learn separately.
Disposition A solves that once, in the place the spec already reserves
for cross-cutting event mechanics (section 2.3 "Event Format"), and
borrows the FHIR extension model that the WG's audience already knows.
Disposition B is a safer half-step but leaves the door open for the
same drift to recur on the next event added to the spec, and Disposition
C punts on a request whose intent is reasonably clear from context. The
WG should adopt A and use the implementation review to flush out any
event whose current "page-local extension" slot needs a migration note
in the change log.
