## Summary

The reporter asks the Infrastructure & Messaging work group to extend the
FHIRcast Event Maturity Model so that an event definition can be flagged
with a "Deprecated" maturity title in addition to the existing levels
(Draft, Submitted, Tested, Considered, Documented, Mature, Normative).
The intent is to give event authors and Hub/Subscriber implementers a
formal, in-spec way to signal that a previously-published FHIRcast event
should no longer be used in new implementations, while leaving the
event's prior definition discoverable for legacy clients. No cross-
referenced Jira tickets were surfaced for this request, so the scope is
limited to what the ticket itself describes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new row, **"Deprecated"**, to the FHIRcast Event Maturity Model
table (currently published as the "eventmaturitymodel" page of the
FHIRcast specification, which today lists levels 0 Draft through 6
Normative). Treat "Deprecated" as a *terminal* maturity title rather
than a numeric level — i.e., add it as a labeled row outside the 0-6
sequence — with the following minimum semantics in the Requirements
column:

- The event was previously published at a non-Draft maturity level and
  is being retired.
- Hubs and Subscribers SHOULD NOT implement the event in new
  deployments. Existing implementations MAY continue to support it for
  backward compatibility.
- The event definition page SHALL retain the previous content and SHALL
  add a deprecation notice that names the replacing event (if any),
  the version in which deprecation took effect, and the rationale.
- The event's Change Log SHALL record the deprecation as a versioned
  entry.

Update the event template page (`3-1-1-template.md`) to allow the
`eventMaturity` line to reference "Deprecated", and update the
maturity-model page (`3-1-2-eventmaturitymodel.md`) to add the new row
and a short narrative paragraph that explains how Deprecated relates to
the numbered levels.

#### Justification

The current maturity model only describes a one-way progression toward
Normative and provides no in-spec mechanism for retiring an event that
has been superseded or found unworkable. Without a formal "Deprecated"
title, event authors have to invent ad-hoc wording on each event page,
which is inconsistent for implementers and invisible to tooling that
keys off the maturity title. Adding "Deprecated" as a sibling title
(rather than a numbered level) aligns with how the broader FHIR core
spec already treats deprecated artifacts — a status orthogonal to the
maturity continuum — and it is a small, additive, backward-compatible
change to the FHIRcast IG that requires no normative reballot of
existing events.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding "Deprecated" as a row in the maturity-title column,
introduce a dedicated **status indicator** on the event definition that
is independent of the Event Maturity Model. Concretely:

1. Leave the existing 0-6 maturity titles untouched.
2. Extend the event template (`3-1-1-template.md`) to add an optional
   second metadata line beneath `eventMaturity`, e.g.
   `eventStatus | active | deprecated | withdrawn`, with default
   `active`.
3. Define the three status values in a new short subsection of the
   maturity-model page, including the required deprecation notice
   (replacing event, deprecating version, rationale) and the
   implementer guidance ("SHOULD NOT use in new implementations; MAY
   retain for backward compatibility").
4. Provide a small CodeSystem or value-set entry (in the existing
   `FHIRcastCodeSystem.fsh`) for the lifecycle status if tooling
   discoverability is desired.

#### Justification

"Deprecated" is conceptually orthogonal to maturity: a Mature event can
be deprecated, and so (in principle) can a Draft one. Conflating the
two columns risks ambiguity — for example, an event whose maturity has
regressed from Mature to Deprecated loses the visible signal that it
was once Mature, and tooling that ranks events by maturity level no
longer has a numeric anchor. Splitting lifecycle status from maturity
preserves the maturity ladder, gives implementers two independent and
machine-checkable signals, and lines up better with how FHIR core
distinguishes `status` (active/retired) from FMM. The cost is slightly
more spec text and one additional metadata line per event.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the current FHIRcast cycle on
the grounds that no event in the FHIRcast IG is currently a candidate
for deprecation, and that the request can be revisited when the first
real deprecation case arises so the model can be designed against a
concrete example.

#### Justification

The ticket as written is a one-line ask with no cross-references, no
Zulip discussion, no linked PR, and no nominated event to deprecate.
Adding a maturity title (or a parallel status axis) without a worked
example carries a non-trivial risk of getting the semantics wrong — for
instance, conflating "deprecated" with "withdrawn" or failing to
specify implementer obligations during the transition period. Until
there is at least one event the work group actually intends to
deprecate, an existing mechanism (a per-event change-log entry plus a
narrative deprecation notice on the event page) is sufficient and
avoids locking in spec text that may need to be reworked.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers what the reporter asked for, is the
smallest possible change to the FHIRcast IG, and gives the work group
a stable, in-spec hook to use the next time an event must be retired.
Disposition B is the more architecturally clean answer, but it expands
the scope of a one-line request into a new status axis (and probably a
CodeSystem entry) without any cross-referenced discussion to back that
design choice — that is better raised as a follow-up ticket once
Disposition A is in place. Disposition C is defensible only if the work
group is confident no FHIRcast event will need deprecation in this
cycle; given that FHIRcast is actively evolving (multiple Open/Close
event pairs, Patient-, Encounter-, ImagingStudy-, DiagnosticReport- and
related events), it is more prudent to land the mechanism now than to
rediscover the gap during a future ballot. The work group should still
ratify the precise wording of the deprecation row and decide whether
to add a parallel status code in a future ticket.
