## Summary

Grahame Grieve has raised that the AU Base FHIR specification is not
explicit about how to represent the Samoan cultural gender identity of
*fa'afafine* (a recognised "third gender" in Samoan tradition). The
ticket asks the working group to (a) make the specification's guidance
on this case explicit and (b) considers adding *fa'afafine* as an
additional code in the gender-identity codes used by the AU Base
gender-identity extension. The issue was originally raised against AU
Core, and the FHIR WG co-chairs accepted it for handling here under
HL7 Australia FHIR / AU Base. There are no cross-referenced Jira
tickets, GitHub items, or Zulip threads specifically discussing
fa'afafine in the indexed corpus, but the broader topic of AU Base
gender-identity terminology has been an active area of work (see
"Related Zulip Discussions").

## Details

**Description:**

> The specification doesn't make clear what to do about the samoan
> tradition of Fa'afafine (see
> https://en.wikipedia.org/wiki/Fa%CA%BBafafine). It should be at
> least clear, and I think it should be yet another gender in the
> codes for the extension.

**Comments:**

- **brettesler — 2025-09-05** — Pointed at the existing AU Base
  guidance at
  <https://build.fhir.org/ig/hl7au/au-fhir-base/sexgender.html#gender-identity-gi>,
  noting that in the absence of a specific code, the published
  guidance is that *"[I/They] use a different term (please specify)"*
  may be represented by sending only `text` and no `coding`.
- **brettesler — 2025-09-05** — Expanded that gender identity is
  self-identified and that the value set bound on the AU Base
  gender-identity extension is the NCTS
  [`gender-identity-response-1`](https://www.healthterminologies.gov.au/integration/R4/fhir/ValueSet/gender-identity-response-1)
  value set, which is aligned with the ABS
  *Standard for Sex, Gender, Variations of Sex Characteristics and
  Sexual Orientation Variables* and AIHW METeOR 741825. The "Different
  term" case is supported today via no `coding` and `text` only, or by
  alternative `coding`s. There is no specific code for *fa'afafine*
  defined; it can be represented as text only or via an alternative
  coding. Adding common gender-identity terms to the NCTS set requires
  a request to NCTS to change the
  `gender-identity-response-1` value set.
- **brettesler — 2025-09-22** — Recorded that the FHIRWG co-chairs
  accepted the issue referred from the AU Core TDG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a **two-part change against AU Base (FHIR)**:

1. **Terminology change request to NCTS** to add a code for
   *fa'afafine* (and, by symmetry, consider whether other recognised
   third-gender terms — e.g., Tongan *fakaleitī*, Māori *takatāpui*,
   Hawaiian *māhū* — warrant codes) to the
   [`gender-identity-response-1`](https://www.healthterminologies.gov.au/integration/R4/fhir/ValueSet/gender-identity-response-1)
   value set. This is the binding NCTS publishes, so the change has
   to be made there; AU Base then re-binds against the next release of
   that value set.
2. **Editorial change in AU Base** in the
   [Sex and Gender → Gender Identity (GI)](https://build.fhir.org/ig/hl7au/au-fhir-base/sexgender.html#gender-identity-gi)
   page to:
   - Explicitly call out fa'afafine (and similar Pacific/Indigenous
     third-gender identities) as recognised cases the spec is aware of,
     so readers do not have to guess.
   - Show the explicit exchange pattern for these identities **once
     coded** (the new NCTS code in `coding`, plus a free-text `text`),
     and the **current** pattern (text-only with no coding, or an
     alternative coding) for use until the NCTS update is published.

#### Justification

This is what the reporter literally asked for ("at least clear, and I
think it should be yet another gender in the codes for the
extension"). It is also consistent with the existing AU Base model:
the value set is already bound *extensibly*, so adding a code at NCTS
is a non-breaking addition rather than a structural change. It
respects the chain of authority — NCTS owns
`gender-identity-response-1`, AU Base owns the spec page and the
binding — which Brett Esler's comments in this ticket explicitly
identify as the correct path. Adding a worked example in the IG page
also addresses the "make it clear" half of the request without waiting
on the NCTS cycle.

---

### Disposition B: Alternative Approach

#### Proposal

**Decline the value-set addition; resolve only via guidance.** Keep the
NCTS `gender-identity-response-1` value set unchanged and instead
publish a normative example block in the AU Base
[Gender Identity (GI)](https://build.fhir.org/ig/hl7au/au-fhir-base/sexgender.html#gender-identity-gi)
section that demonstrates representing fa'afafine — and culturally
analogous identities — using the **already-supported "Different term"
pattern**:

- `Extension.valueCodeableConcept.text = "Fa'afafine"` (no `coding`),
  **or**
- `Extension.valueCodeableConcept.coding` carrying an alternative
  CodeSystem (e.g., a community-maintained or implementer-local code
  system) with a parallel `text` element.

Add a paragraph naming fa'afafine, fakaleitī, takatāpui, māhū as
worked examples and explicitly state that these are conformant uses
of the existing extensible binding.

#### Justification

The ticket conflates two issues: clarity of the spec, and presence in
the value set. Brett Esler's responses in the ticket already note that
the existing extensible binding plus text/alternative-coding pattern
*does* support fa'afafine today; the gap is only that this is not
sign-posted in the IG. Resolving via guidance:

- avoids dragging AU Base's release cycle behind NCTS's;
- keeps NCTS as the sole curator of the canonical Australian
  gender-identity code set, rather than AU Base lobbying NCTS on
  behalf of every individual community-raised term;
- is a faster, lower-risk change (IG markdown only, no rebinding, no
  validator implications);
- still gives implementers a deterministic pattern they can ship
  today.

The trade-off vs. Disposition A is that fa'afafine remains text-only
in the canonical Australian set, which the reporter explicitly said is
not their preferred outcome.

---

### Disposition C: Decline

#### Proposal

Mark as **Persuasive with Modification — No Change**: the underlying
need is already addressed by the existing extensible binding to NCTS
`gender-identity-response-1`, where any term not in the value set —
fa'afafine included — is representable via `text` only or an
alternative `coding`, as documented in the AU Base
[Gender Identity (GI)](https://build.fhir.org/ig/hl7au/au-fhir-base/sexgender.html#gender-identity-gi)
section. Refer the requester to that guidance and, for code-set
additions, redirect to NCTS's value-set change-request process for
`gender-identity-response-1` as the correct standards owner.

#### Justification

This is defensible because (a) AU Base does not own the bound value
set — NCTS does — so AU Base is not the right venue for adding codes;
(b) the current spec is functionally complete: text-only is a
conformant exchange pattern for any non-binary-but-not-coded identity,
including fa'afafine; and (c) accepting the ticket as a code-list
change risks AU Base creating a parallel, AU-specific code system for
gender identities that diverges from the NCTS-curated set, which is
exactly the fragmentation NCTS exists to prevent. The harm of
declining is essentially nil — the reporter can immediately raise the
NCTS change request, and implementers already have a defined
representation.

---

### Recommendation

**Recommended disposition: A.**

Disposition A is the only one that fully discharges *both* halves of
the request — the explicit "make it clear" ask *and* the explicit
"another gender in the codes" ask — and it does so along the
governance lines that Brett Esler already flagged in the ticket
comments (NCTS owns the code set, AU Base owns the binding and the
narrative). Disposition C is defensible on jurisdictional grounds but
will read to the reporter as deflection, given that the comment thread
has already acknowledged the gap and the FHIRWG co-chairs accepted
the issue. Disposition B captures the editorial half cheaply but
permanently leaves fa'afafine as text-only in the Australian
canonical set, which (a) is what the reporter explicitly said was not
sufficient and (b) creates implementer friction where exchanging a
code is operationally simpler than parsing free text. The Zulip "AU
Base R7 scope" recap already lists "considering the gender identity
terminology change requests" as in-scope, so adopting Disposition A
also aligns with the announced AU Base R7 work programme. The
working group should pair the Jira disposition with a tracked NCTS
change request so the AU Base re-binding lands in the same R7 cycle.
