## Summary

The reporter asks the HL7 Australia FHIR work group to revisit the AU Base
Organization profile and represent the value sets currently described in the
profile's narrative implementation guidance using the
[`additionalBindings` extension](https://build.fhir.org/ig/FHIR/fhir-tools-ig/StructureDefinition-additional-binding.html)
from the FHIR Tools IG, rather than leaving them only as prose. A working
proposal already exists on the
[`additionalBindings` branch of `hl7au/au-fhir-base`](https://build.fhir.org/ig/hl7au/au-fhir-base/branches/additionalBindings/StructureDefinition-au-organization.html),
so the request is essentially: "review and apply the prepared branch to the
AU Base Organization profile." This ticket is one of a coordinated family of
AU Base "additionalBindings clarification / refactor" tickets opened by the
same reporter against the same `additionalBindings` working branch, which
together moved AU Base from prose-only or slice-based binding hints to
machine-readable `additionalBinding` extensions. The closely-related
[FHIR-44823](https://jira.hl7.org/browse/FHIR-44823) (Medication profiles),
[FHIR-44821](https://jira.hl7.org/browse/FHIR-44821) (Immunization.vaccineCode),
and [FHIR-46080](https://jira.hl7.org/browse/FHIR-46080) (Pathology Result)
have all been resolved Persuasive (or Persuasive with Modification) and
Applied. The sibling
[FHIR-46081](https://jira.hl7.org/browse/FHIR-46081) (PractitionerRole) is
still Triaged and is the closest analogue to this ticket, since both target
"clarification" of existing guidance rather than removing slices.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the prepared branch
[`additionalBindings` of `hl7au/au-fhir-base`](https://build.fhir.org/ig/hl7au/au-fhir-base/branches/additionalBindings/StructureDefinition-au-organization.html)
to the AU Base Organization profile (`StructureDefinition/au-organization`).
Concretely: identify each value set that is currently named only in
narrative "implementation guidance" on the Organization profile (typically
on `Organization.identifier` and/or `Organization.type`) and add an
`additionalBindings` extension entry on the corresponding ElementDefinition
binding, with an appropriate `purpose` (default to `preferred`, mirroring
the resolved sister tickets FHIR-44821 and FHIR-44823, unless the existing
narrative clearly indicates a different purpose such as `component`,
`required`, `extensible`, `maximum`, `ui`, or `starter`). Inherit binding
strength from the base FHIR R4/R4B Organization profile — do **not** change
the main conformance binding. Once the profile is updated, remove the now
redundant prose from the implementation-guidance narrative (or rewrite it
to point at the new `additionalBindings` block). The change is internal to
`hl7au/au-fhir-base`; no other AU IGs need to move in lockstep.

#### Justification

This is exactly the pattern the AU FHIR WG has already approved three times
in the same campaign (FHIR-44821, FHIR-44823, FHIR-46080), all reported by
the same author against the same working branch and all resolved
Persuasive. The reporter has already done the modelling work on the named
branch, so the WG's job is essentially review + apply, which keeps cost
low. Replacing prose-only guidance with the `additionalBindings` extension
makes the binding intent machine-readable and consumable by validators,
profile renderers, and downstream IGs — directly aligned with the FHIR
Tools IG's stated purpose for the extension. Crucially, because the main
conformance binding is left untouched, the change is non-breaking for
existing AU Base implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent but harmonise it with FHIR-46081 (PractitionerRole) and
process them together as a single "AU Base profile-clarification rollup":
both this ticket and FHIR-46081 describe identical "lift narrative bindings
into `additionalBindings`" work on a single profile, and both still point at
the same `additionalBindings` branch. As part of the rollup, also (a)
explicitly enumerate each `additionalBinding` (URL, purpose, documentation)
in the ticket resolution before merging, mirroring the table style used in
FHIR-46080's resolution, and (b) add a short "How AU Base uses
`additionalBindings`" subsection to the AU Base implementation-guidance page
so that the policy ("we use `additionalBindings` rather than narrative
guidance for non-conformance terminology hints, with `preferred` as the
default purpose") is documented once for implementers rather than
re-derived per profile.

#### Justification

Disposition A applies the change but does not capture the policy that has
emerged from the campaign. Treating FHIR-44822 and FHIR-46081 as one
review-and-apply unit avoids relitigating the same design question twice in
WG meetings, ensures consistency between the Organization and
PractitionerRole profile renderings, and gives the publisher a chance to
specify exact value-set URLs and purposes in the resolution — which the
Pathology Result ticket (FHIR-46080) demonstrated is valuable when the
purpose is non-default. Adding a short narrative subsection costs little
and turns an implicit convention into discoverable guidance, which helps
downstream IG authors who derive from AU Base.

---

### Disposition C: Decline

#### Proposal

Close the ticket with resolution "Not Persuasive" (or "Not Persuasive with
Reason: superseded") on the basis that the existing prose implementation
guidance is already adequate for AU Base implementers, that the
`additionalBindings` extension is supplied by the FHIR Tools IG (which is
itself still under active development), and that no specific implementer
problem has been demonstrated by the ticket itself.

#### Justification

This is the defensible "do nothing" option if the WG concludes that the
campaign has already produced enough machine-readable binding metadata
elsewhere (Medication, Immunization, Pathology Result) to validate the
pattern, and that the Organization profile in particular does not carry
enough additional terminology guidance to justify a structural edit. It
would also be the right disposition if the WG is uncomfortable taking on a
dependency on the FHIR Tools IG `additional-binding` extension while that
IG remains a work-in-progress.

---

### Recommendation

**Recommended disposition:** B

The bare "accept" (A) is technically correct and consistent with the three
resolved sister tickets, but it leaves two avoidable risks on the table:
(1) FHIR-44822 and FHIR-46081 are essentially the same request against two
profiles and should be approved together so the Organization and
PractitionerRole renderings stay consistent, and (2) when the WG previously
took a "clarification" ticket of this shape (FHIR-46080) it had to come
back with a "Persuasive with Modification" to nail down purpose semantics
mid-stream — capturing the value-set URLs, purposes, and short
documentation strings up front in this ticket avoids that. Disposition C
is hard to defend given that the WG has already accepted the same change
shape three times in the same campaign with no reported implementer harm.
Disposition B preserves the technical change in A, harmonises it with the
still-pending FHIR-46081, and writes the now-implicit "AU Base uses
`additionalBindings` for non-conformance terminology hints" policy down so
the next ticket of this shape doesn't need to be re-argued.
