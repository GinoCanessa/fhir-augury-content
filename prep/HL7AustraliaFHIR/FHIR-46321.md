## Summary

The ticket asks AU Base (FHIR) to define how a private health insurer registered in Australia is unambiguously identified for use in `Coverage.payor.identifier` and `Coverage.identifier.assigner` (i.e., the payor organisation itself, not the patient's individual private health insurance member number, which is already supported via the existing AU Insurance Member Number profile). The originating discussion (mirrored from `hl7au/au-fhir-base#791`) was surfaced by the Draft Diagnostic Requests work, where the eOrders Coverage profile needed a way to identify the private insurer for billing/ordering scenarios. Two candidate code lists are in scope — the ATO-published list of complying private health insurers (e.g., `HCF`, `NTF`, `BUP`) shown at `https://privatehealth.gov.au/dynamic/insurer`, and a parallel Services Australia list (e.g., `DHF` vs ATO `AHB` for Defence Health Limited) — and the central open question is which list to standardise on and what canonical identifier `system` URL to mint for it. dbojicic-agency (Agency) has proposed an ADHA-issued namespace `http://ns.electronichealth.net.au/id/private-health-insurer-id` (with an OID `1.2.36.1.2001.1005.xx` to be progressed inside Agency). The cross-reference graph in FhirAugury surfaces no related Jira tickets, Zulip threads, or GitHub items beyond the FHIR element mentions; the discussion lives almost entirely in the AU FHIR Base GitHub issue mirrored into the ticket body.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new Identifier profile in AU Base (FHIR), e.g., `au-privatehealthinsurer-identifier` (NamingSystem + StructureDefinition), bound to the ADHA-issued canonical `system` URL `http://ns.electronichealth.net.au/id/private-health-insurer-id` (OID `1.2.36.1.2001.1005.xx` once progressed inside Agency), with `value` drawn from the ATO published code list at `https://privatehealth.gov.au/dynamic/insurer` (e.g., `HCF`, `NTF`, `BUP`). Add an `Identifier.type` of `$v2-0203#NII` (National Insurance Identifier) by default. Update the existing `Coverage/private` example so `Coverage.payor.identifier` (and, where the insurer is the assigner, `Coverage.identifier.assigner.identifier`) uses this new identifier. Document in the profile that the chosen list is ATO-issued (and explicitly out-of-scope for non-complying funds tracked only by Services Australia).

#### Justification

This is what the ticket asks for: an unambiguous, canonical `system` URL for the payor organisation, separate from the already-supported AU Insurance Member Number. The ADHA-issued namespace pattern aligns with prior precedent in AU (`http://ns.electronichealth.net.au/id/abn-scoped/medicalrecord/1.0`) and was concretely proposed by dbojicic-agency on behalf of the Agency. Choosing the ATO list keeps the value set anchored to a single, publicly-published authoritative source and avoids the cross-list collisions called out in the discussion (e.g., Defence Health = `AHB` in ATO vs `DHF` in Services Australia). Defining the identifier in AU Base — rather than only in AU Core or in the eOrders IG — lets Coverage, Claim, ExplanationOfBenefit and downstream IGs reuse it consistently for both ordering/billing and theatre-booking scenarios.

---

### Disposition B: Alternative Approach

#### Proposal

Take the same shape as A, but make the AU Base Identifier profile **list-agnostic** by binding `system` to a small ValueSet of permitted canonicals (initially the ADHA-issued `http://ns.electronichealth.net.au/id/private-health-insurer-id` as the *preferred* canonical, plus an explicitly-published canonical for the Services Australia list, e.g., `http://ns.electronichealth.net.au/id/services-australia/private-health-insurer-id`). Require `Identifier.type` to be `$v2-0203#NII` and require the value to come from one of the published lists. Capture the same identifier shape in AU Core for the AU Core Coverage profile so implementers using AU Core (not the full AU Base) get the constraint too. Defer minting the OID until ADHA confirms `1.2.36.1.2001.1005.xx`, and ship the canonical URL ahead of the OID.

#### Justification

The discussion in the ticket is not yet settled on a single list: heathfrankel and Genie use ATO IDs; the ACT implementer (Timothy Panoho via brettesler-ext) uses Services Australia's Eclipse 3 acronyms; dbojicic-agency notes substantive divergences between the two lists for the same real-world insurers, and explicitly proposes the ADHA-issued namespace because the ATO list is restricted to "complying" funds. Hard-binding AU Base to the ATO list (Disposition A) leaves real Australian implementations — including state public-health EMRs — out of the core conformance story, and forces them to extend immediately. A list-agnostic profile with two named canonicals lets both communities conform now and gives ADHA a clean deprecation path if it ever consolidates the lists. Aligning AU Core in the same change avoids a second round of IG churn (the Genie comment explicitly asks for this in AU Core).

---

### Disposition C: Decline

#### Proposal

Decline to define a new private-health-insurer Identifier in AU Base on the grounds that (a) `Coverage.payor` already supports `Reference(Organization)` with an arbitrary `Organization.identifier` and `Coverage.identifier.assigner` already supports the same Identifier datatype, and (b) the canonical identifier `system` URL itself is owned by ADHA / the ATO, not HL7 AU. Recommend instead that implementers continue to use `https://privatehealth.gov.au/dynamic/insurer` (or a Services Australia equivalent) directly as the `system` URL until the Agency formally publishes an ADHA-issued canonical, at which point a follow-up ticket can add it as a NamingSystem in AU Base.

#### Justification

The base FHIR Identifier datatype is already adequate to carry an insurer code in `Coverage.payor.identifier` and `Coverage.identifier.assigner`; nothing in FHIR is technically blocking the use case today. The unresolved question is purely the canonical `system` URL, and as dt-r notes the consumer-facing privatehealth.gov.au URL is not a real identifier-system canonical — choosing one prematurely (especially the OID `1.2.36.1.2001.1005.xx`, which is still "to be progressed within Agency") risks burning a canonical that Agency later replaces. Declining now and waiting for ADHA confirmation avoids minting a canonical AU Base will then have to deprecate.

---

### Recommendation

**Recommended disposition:** B

The ticket has been open and discussed since mid-2023 with concrete implementer demand from at least Genie, Sonic/Healius, and ACT public health, and the only blocker is a canonical `system` URL. Disposition C punts on the question that AU implementers actually need answered, and Disposition A's hard binding to the ATO list ignores a substantive concern raised by both the Agency (dbojicic-agency) and the ACT EMR feedback that the ATO list is too narrow. Disposition B keeps the profile shape simple (a single AU Base Identifier profile + matching AU Core profile, anchored on `Identifier.type = v2-0203#NII`), adopts the ADHA-issued canonical that the Agency has already proposed as the *preferred* `system`, but explicitly admits the Services Australia list as a co-equal canonical so existing implementations remain conformant. Shipping the canonical URL ahead of the OID also addresses dt-r's concern about not burning a premature OID. The workgroup should confirm with dbojicic-agency / ADHA that `http://ns.electronichealth.net.au/id/private-health-insurer-id` is ready to publish, and decide explicitly whether to mirror the profile into AU Core in the same release.
