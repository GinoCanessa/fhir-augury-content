## Summary

This ticket is a follow-on to [FHIR-47148](https://jira.hl7.org/browse/FHIR-47148)
and [FHIR-47149](https://jira.hl7.org/browse/FHIR-47149), both of which were
applied to AU Base and re-pointed every published `CodeSystem` and `ValueSet`
resource away from the HL7 international `shareablecodesystem` / `shareablevalueset`
profiles and toward the National Clinical Terminology Service (NCTS) profiles
(`complete-code-system-4` and `composed-value-set-4`, respectively, on
`healthterminologies.gov.au`).

Once those changes were applied, a tooling gap surfaced: the FHIR IG Publisher
cannot validate the AU Base terminology resources against the new conformance
claims because the NCTS profile `StructureDefinition` resources are not
distributed as a FHIR package that AU Base can declare as a dependency. The
request here is purely a *dependency-management* change: when NCTS publishes
its profiles in a consumable FHIR package, declare that package as a dependency
of AU Base so the IG Publisher can resolve and validate against the NCTS
profiles. A blocking external action has been raised against the NCTS
publishing pipeline at
[aehrc/fhirpit#120](https://github.com/aehrc/fhirpit/issues/120). The
December 2024 triage comment explicitly flags this as a change request (not a
technical correction) and asks the work group to decide whether AU Base should
take a hard dependency on NCTS at all.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Once the NCTS publishes a FHIR package containing the
`complete-code-system-4` and `composed-value-set-4` `StructureDefinition`
resources (tracked by [aehrc/fhirpit#120](https://github.com/aehrc/fhirpit/issues/120)),
add it as a hard dependency of AU Base in the IG manifest:

- Update `sushi-config.yaml` (and any equivalent `package.json` /
  `ig.ini` / `ImplementationGuide.dependsOn` declarations) in the
  `hl7au/au-fhir-base` repo to add a `dependsOn` entry for the published
  NCTS package, pinned to a specific version (e.g.
  `au.gov.healthterminologies.fhir.r4#<version>` once the package ID and
  version are confirmed).
- Re-run the IG Publisher; confirm that `meta.profile` claims on every
  AU Base `CodeSystem` and `ValueSet` resolve and validate cleanly.
- Remove the workaround note that current QA was performed via local
  validation only.

#### Justification

The AU Base CodeSystem/ValueSet resources already claim conformance to NCTS
profiles (FHIR-47148 / FHIR-47149 are Applied), so without this dependency
the conformance claims are unverifiable by standard tooling and effectively
unenforceable for downstream implementers. Declaring a proper
`dependsOn` on the NCTS package is the standard FHIR mechanism for making
those profiles resolvable to the IG Publisher and is fully aligned with how
AU Base already depends on `hl7.fhir.uv.*` packages today. The change is
small, surgical, and reversible by version-pinning.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than wait for and then take a hard package dependency on NCTS,
*conditionally* declare the dependency and front it with a clear policy
statement in AU Base:

1. Author an "AU Terminology Conformance Policy" page (under the same
   2025 conventions project flagged in the FHIR-47148 / FHIR-47149
   resolutions and already scheduled with FHIR-47166 / FHIR-47174) that
   describes when and how AU Base profiles, CodeSystems and ValueSets
   reference NCTS artefacts, and what implementers should do if the NCTS
   profiles are unavailable in their environment.
2. Add the NCTS package as a `dependsOn` only when (a) NCTS publishes a
   versioned, package.tgz-compliant artifact, and (b) that package is
   mirrored to (or proxied by) `packages.fhir.org` / `packages2.fhir.org`
   so non-Australian validators can also resolve it.
3. As an interim step, vendor only the two needed `StructureDefinition`
   resources into AU Base under an `input/resources/` (or equivalent
   non-conformance) folder so the IG Publisher can resolve them locally,
   clearly marked as "mirrored from NCTS, authoritative source remains
   NCTS".

#### Justification

A package dependency on a non-`packages.fhir.org` registry creates a real
implementation burden for downstream consumers (notably international
implementers and CI pipelines that whitelist FHIR registries). This
alternative keeps the conformance story intact while (a) treating the
dependency decision as a *policy* decision — which is exactly what the
2024-12-09 triage comment asked for — and (b) giving implementers a
working IG today via an interim vendored copy. It also avoids the AU Base
release calendar being blocked indefinitely by the cadence of the
upstream `fhirpit` issue.

---

### Disposition C: Decline

#### Proposal

Decline as duplicative / out of scope. Close FHIR-48933 with a note that
the underlying issue is purely an upstream NCTS *tooling* gap tracked in
[aehrc/fhirpit#120](https://github.com/aehrc/fhirpit/issues/120) and not
something AU Base can act on. Re-open or re-raise once NCTS announces a
published package version.

#### Justification

There is no AU Base spec change to make until NCTS publishes a consumable
package; until then any dependency declaration would point at a
non-existent canonical. A separate Jira tracker for "add the dependency
when the package exists" arguably duplicates fhirpit#120 plus the work
already implied by FHIR-47148 / FHIR-47149's Applied state. The work
group can carry the action item in meeting minutes rather than as a
long-lived ticket on AU Base.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach).

Disposition A is the textbook outcome but it leaves AU Base validation
broken until an external repository (`aehrc/fhirpit`) ships and an
external registry adopts the package — neither of which is on the AU Base
work group's calendar. Disposition C is honest about scope but throws
away the only lever AU Base actually controls: the policy decision the
2024-12-09 triage comment asked for. Disposition B turns FHIR-48933 into
the trigger for the AU Terminology Conformance Policy work that
FHIR-47148 / FHIR-47149 already promised for 2025, restores IG Publisher
validation in the short term via a small vendored mirror of the two
needed `StructureDefinition` resources, and still lands at the
hard-dependency end state once NCTS distribution is in place. It also
explicitly addresses the implementer-friction concern that a non-standard
registry dependency would otherwise create.
