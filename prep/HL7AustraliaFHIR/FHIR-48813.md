## Summary

Heath Frankel asks the AU FHIR work group to revisit the **definition text** of the AU eRequesting Server CapabilityStatement in light of the changes adopted under [FHIR-47141](https://jira.hl7.org/browse/FHIR-47141). FHIR-47141 (Resolution: *Persuasive with Modification*, Status: *Applied*) replaced the originally-proposed separate "eRequesting Responder" actor with a relaxation of the Server CapabilityStatement's `ServiceRequest` `create`/`update` interactions from **SHALL** to **SHOULD**, so a placer (or intermediary) system that merely exposes its own `ServiceRequest` resources via `read`/`search-type` is now also conformant against the AU eRequesting Server CapabilityStatement. As a result, the existing prose definition of the Server actor — which presumes the implementer is a system that *accepts* incoming diagnostic service requests — is no longer accurate for the new placer-exposing-own-requests scenario. The reporter offers two candidate replacements: a deliberately vague form aligned with existing actor definitions ("a system that makes diagnostic service requests accessible and can accept diagnostic service requests"), and a more descriptive form that explicitly names both filler/patient-record systems and placer systems ("makes diagnostic service requests accessible filler and patient record systems and can accept diagnostic service requests from placer systems"). The reporter notes a personal preference for the more descriptive wording but concedes that the vaguer form is more in line with the project's house style for actor definitions.

## Details

**Description:**

> As a result of [FHIR-47141](https://jira.hl7.org/browse/FHIR-47141) change, the AU eRequesting Server CapabilityStatement definition needs to be reviewed as the current definition is no longer representative of the scenario when a Placer system implements the AU eRequesting Server CapabilityStatement.
>
> Suggest the following definition:
>
> - This CapabilityStatement describes the basic rules for the AU eRequesting Server actor that is is a system that makes diagnostic service requests accessible and can accept diagnostic service requests.
>
> Personally, I feel this definition is fairly vague and unhelpful but has been suggested to align with existing actor definitions.
>
> A more useful definition may be something like:
>
> - This CapabilityStatement describes the basic rules for the AU eRequesting Server actor that is is a system that makes diagnostic service requests accessible filler and patient record systems and can accept diagnostic service requests from placer systems.

### [FHIR-47141: Provide eRequesting Responder Actor and Capability Statement](https://jira.hl7.org/browse/FHIR-47141)

The originating ticket explicitly cited in this request's description. It originally proposed a separate "eRequesting Responder" actor so a placer could expose its own `ServiceRequest`s via `read`/`search` without implementing the full `create` flow. The work group resolved it *Persuasive with Modification* (Applied 2024-10-22) by **not** introducing a new actor and instead relaxing the AU eRequesting Server CapabilityStatement so `ServiceRequest` `create`/`update` are **SHOULD** (formerly **SHALL**) while `read`/`search-type` remain **SHALL**. That conformance change is precisely what makes the current Server-actor definition prose feel mis-fitted, motivating FHIR-48813.

## Proposed Dispositions

Note: there are no GitHub cross-references for this ticket and therefore no `repo-analysis` briefing context to anchor file paths against. The dispositions below name the artifact (the AU eRequesting Server CapabilityStatement) and the source IG (`hl7.fhir.au.ereq`, published at <https://hl7.org.au/fhir/ereq/>) but cannot cite specific authoring file paths from a briefing. If the work group accepts a wording change, the implementer should locate the CapabilityStatement source under the `hl7.au/ereq` IG repository (e.g., the `input/resources/CapabilityStatement-*.json|xml` for the Server actor) before editing.

### Disposition A: Accept As Requested

#### Proposal

Update the `description` element of the AU eRequesting **Server** CapabilityStatement (`CapabilityStatement.description`, and any equivalent narrative `<text>` block) to the more descriptive wording suggested by the reporter:

> *"This CapabilityStatement describes the basic rules for the AU eRequesting Server actor that is a system that makes diagnostic service requests accessible to filler and patient record systems, and can accept diagnostic service requests from placer systems."*

(Repair the duplicated "is is" and the missing "to" found in the ticket's prose so the published text is grammatical.) Apply the change in the `hl7.fhir.au.ereq` IG source for the next preview/ballot release. Leave conformance bits (the SHALL/SHOULD interactions established by FHIR-47141) untouched.

#### Justification

The wording explicitly enumerates the two implementer populations that FHIR-47141's resolution legitimised — (a) filler / patient-record systems that *accept* requests via `create` (SHOULD) and serve them back via `read`/`search-type` (SHALL), and (b) placer systems that simply *expose* their own `ServiceRequest` resources via `read`/`search-type` (SHALL) without implementing `create`. The current description, by referring only to "accepting" diagnostic service requests, mis-describes the placer-exposing-own-requests scenario and could lead implementers to incorrectly believe they fall outside the Server actor's scope. The reporter — also the originator of FHIR-47141 — is unambiguously the subject-matter authority on the intent of that change.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the *vaguer* candidate wording the reporter cites as having been suggested for cross-IG consistency:

> *"This CapabilityStatement describes the basic rules for the AU eRequesting Server actor that is a system that makes diagnostic service requests accessible and can accept diagnostic service requests."*

(Again, fix the "is is" typo.) In addition, retain the placer / filler / patient-record explanation but move it to the CapabilityStatement's `purpose` element (or to the IG narrative page that introduces the actor), rather than `description`. This keeps `description` short and stylistically consistent with peer AU IG actor definitions, while still capturing the implementer-population guidance somewhere first-class.

#### Justification

FHIR convention treats `CapabilityStatement.description` as a brief identity statement and `CapabilityStatement.purpose` (and accompanying IG narrative) as the place for "why and for whom". Splitting the content this way preserves the house-style alignment the reporter alluded to ("aligns with existing actor definitions") without losing the substantive guidance. It is also more resilient if the actor's responsibilities evolve again — narrative pages are easier to amend without churning the conformance artifact's `description`. The trade-off vs. Disposition A is that an implementer reading the CapabilityStatement in isolation (e.g., from a registry view) would not immediately see the placer-vs-filler distinction; mitigating this depends on the IG narrative being prominent and discoverable.

---

### Disposition C: Decline

#### Proposal

Decline the change as **not persuasive at this time**: leave the AU eRequesting Server CapabilityStatement `description` text as it currently stands, and instead cover the placer-exposing-own-requests scenario in the AU eRequesting IG narrative pages (the actor / interaction overview) rather than in the conformance artifact itself.

#### Justification

The reporter himself characterises the requested wording as "fairly vague and unhelpful" (the conformist option) or potentially over-specified (the descriptive option). The substantive conformance question — whether placers can be Server-actor implementers — was already settled by FHIR-47141 via SHALL→SHOULD on `create`/`update`, which is the part implementers actually validate against. Re-litigating prose in the absence of evidence that implementers are being misled (no implementer comments on this ticket; the December 2024 connectathon thread surfaced server-config and resource-coverage issues, not actor-scope confusion) risks bikeshedding. This disposition is hard to defend if any work-group member reports actual implementer confusion, however.

---

### Recommendation

**Recommended disposition:** A

The reporter is the originator of the underlying conformance change (FHIR-47141) and is best placed to judge whether the artifact's current prose accurately reflects that change's intent. He has stated it does not, and offers a concrete fix. The descriptive form names both implementer populations explicitly, which is the cheapest and most discoverable way to prevent placer-system implementers from wrongly concluding that the Server CapabilityStatement is "not for them". Disposition B is defensible on house-style grounds but introduces an indirection (narrative page vs. artifact description) that registry- and tooling-driven readers may miss. Disposition C is hard to justify when the cost of the change is a single sentence in a single artifact and the change has the explicit endorsement of the original requester. The work group should accept Disposition A, applying minor copy-edits ("is is" → "is", insert "to") during implementation.
