## Summary

The ticket, raised by Queensland Health and migrated from
[hl7au/au-fhir-core#69](https://github.com/hl7au/au-fhir-core/issues/69)
(originally part of the AU Core Condition Consensus
[discussion #52](https://github.com/hl7au/au-fhir-core/discussions/52)),
proposes that the `Condition.code` binding in the AU Core Condition
profile use an **AU Core–specific value set** rather than the current
extensible binding to the NCTS value set
[`https://healthterminologies.gov.au/fhir/ValueSet/clinical-condition-1`](https://healthterminologies.gov.au/fhir/ValueSet/clinical-condition-1).
Two reasons are given: (1) an AU Core-owned value set could be curated
through the HL7 Australia standards development process independent of
the NCTS (while still potentially being co-published by the NCTS); and
(2) the existing NCTS value set is defined as the entire
Problem/Diagnosis SNOMED CT-AU reference set, which was not necessarily
designed against AU Core's goals and whose membership can change (e.g.
NRC/SNOMED International concept retirements) in ways that affect the
binding's expansion.

The cross-reference graph for this ticket only surfaces a single
`fhir`-typed mention of `Condition.code`; no related Jira tickets,
Zulip threads, or GitHub items were returned by the augury index
(although the ticket body itself directly cites the migrated GitHub
issue and discussion above).

## Details

**Description:**

From Queensland Health:

...

3. Code for the Condition/Problem or Diagnosis is currently extensibly
bound to the
[`https://healthterminologies.gov.au/fhir/ValueSet/clinical-condition-1`](https://healthterminologies.gov.au/fhir/ValueSet/clinical-condition-1)
NCTS value set. It is suggested that an AU Core specific value set is
defined for this binding for the following reasons:

- An AU Core value set can be maintained/curated through the HL7
  Australia standards development process independent of the NCTS.
  Note: an AU Core value set may still be published in the NCTS if
  HL7 Australia and the NCTS operator are open to this.

- The NCTS value set is itself defined as including all the members of
  the Problem/Diagnosis SNOMED CT-AU reference set. While many of these
  concepts are likely to be appropriate for this binding, the reference
  set may not have been developed with the same goals as AU Core.
  Further, changes to the reference set (e.g. retirement of member
  concepts by the Australian NRC or SNOMED International) would impact
  this value set's expansion and thus the binding.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define an **AU Core-specific value set** for the AU Core Condition
profile's `Condition.code` element and rebind the element to it.
Concretely:

- Author a new value set under the AU Core canonical namespace (e.g.
  `http://hl7.org.au/fhir/core/ValueSet/au-core-condition-code`) in
  the `hl7au/au-fhir-core` IG (the AU Core authoring repo cited by the
  ticket).
- Define its content compose by referencing the same Problem/Diagnosis
  SNOMED CT-AU reference set used by
  `clinical-condition-1` (so the initial expansion semantically equals
  the current binding), but with AU Core retaining editorial control
  to add/remove inclusions or exclusions over time.
- Change the `Condition.code` binding on the AU Core Condition profile
  to point at the new AU Core value set (preserving the current
  binding strength of `extensible`), and demote the NCTS
  `clinical-condition-1` value set to an Additional Binding (e.g.
  `preferred` purpose) so existing implementers retain a known
  reference.
- Coordinate with the NCTS operator on whether the new AU Core value
  set should also be co-published in the NCTS, as the ticket explicitly
  contemplates.

#### Justification

This directly addresses the reporter's two stated concerns: it gives
HL7 Australia editorial control over the binding (decoupling the
profile's stability from changes to the SNOMED CT-AU
Problem/Diagnosis reference set) without sacrificing the existing
implementer experience (the value set still expands to substantively
the same SNOMED CT-AU concepts on day one). It also aligns with the
general FHIR practice of an IG owning bindings for its profiles where
the IG's scope and the upstream terminology authority's scope are not
identical.

---

### Disposition B: Alternative Approach — Keep NCTS as the primary binding, add an AU Core Additional Binding

#### Proposal

Leave the `Condition.code` primary binding on the AU Core Condition
profile pointing at the NCTS
`https://healthterminologies.gov.au/fhir/ValueSet/clinical-condition-1`
value set, but add an **Additional Binding** entry on `Condition.code`
that names an AU Core-curated subset (or, conversely, a recommended
"preferred" / "starter" subset) of conditions AU Core wants to
emphasise — analogous to the treatment of `AllergyIntolerance.code`
that JIRAUSER22800 calls out in the comments. The AU Core subset would
be authored in `hl7au/au-fhir-core` and may also be co-published via
the NCTS so it remains discoverable through Australian terminology
infrastructure.

#### Justification

This approach honours the existing PA-WG / Agency arrangement that
SNOMED CT-AU and LOINC value sets used in HL7 AU artefacts are
authored and published in the NCTS (raised by Danielle Tavares-Rixon
in the 2024-01-19 comment), while still giving AU Core a controllable
lever to express its own emphasis on `Condition.code`. It is also less
invasive than rebinding the primary value set: existing AU Core
implementers continue to validate against the same expansion they do
today, and AU Core's editorial input is added on top via Additional
Bindings — a mechanism the comment thread already proposes by
analogy.

---

### Disposition C: Decline

#### Proposal

Resolve as **No Change**. Retain the current extensible binding on
`Condition.code` to
`https://healthterminologies.gov.au/fhir/ValueSet/clinical-condition-1`
and continue to rely on the NCTS to maintain the AU
Problem/Diagnosis content per the long-standing PA-WG / Agency
arrangement.

#### Justification

A defensible decline rests on three points: (a) the established
practice — re-stated in the comments — is that HL7 AU value sets based
on SNOMED CT-AU are authored and published in the NCTS, and unilaterally
forking that arrangement for one element risks fragmenting AU Core's
terminology footprint; (b) the binding is already `extensible`, so
implementers are not blocked when a clinically appropriate concept is
absent from the NCTS expansion; and (c) the ticket does not present a
concrete content-driven gap (e.g. a list of concepts that AU Core
needs and the NCTS reference set does not contain) — the concerns are
governance- and risk-shaped rather than content-driven, and Danielle
Tavares-Rixon's question in the first comment ("is this a governance
question or is this driven by content?") was never answered on the
ticket.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between the reporter's legitimate
concerns and the established Australian terminology governance model.
The reporter's underlying worry — that AU Core's Condition binding can
be perturbed by changes to the SNOMED CT-AU Problem/Diagnosis
reference set, and that AU Core has no editorial seat at the table —
is real and is not fully answered by the status quo. But Disposition A
(rebind primary to an AU Core-owned value set) takes a unilateral step
away from the long-standing PA-WG / Agency arrangement that
SNOMED CT-AU value sets are NCTS-published, which the comment thread
flags is a non-trivial governance change to make in a single ticket
without the NCTS at the table. Disposition B is also the option
explicitly suggested in JIRAUSER22800's October 2024 comment by
analogy with `AllergyIntolerance.code`, which is the closest existing
precedent within AU Core itself. It also lines up well with the AU
Core R1 TDG decision (recorded in the most recent comment) to leave
this open and revisit at R3 — Additional Bindings can be added at R3
without breaking the R1/R2 primary binding contract for existing
implementers. If, when AU Core R3 takes this up, NCTS engagement
proves impossible, the workgroup can escalate from B to A; the
converse migration (A → B) would be a breaking change to the primary
binding and is harder to walk back.
