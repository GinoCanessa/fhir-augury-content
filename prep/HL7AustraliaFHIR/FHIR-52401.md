## Summary

The reporter argues that the AU Core Smoking Status profile, as it
currently stands, is clinically too vague to support meaningful risk
assessment. Three concerns are raised: (1) the value set bound to the
smoking-status observation value (Occasional tobacco smoker, Former
smoker, Lifetime non-smoker, Current smoker) does not capture the
clinically actionable variables — duration of smoking, duration since
cessation, pack-years, and quit attempts — that drive cardiovascular,
neurovascular, respiratory and cancer risk stratification; (2) the
extensible binding to `ObservationInterpretationCodes` is generic (e.g.,
"significant change up/down", "worse", "off scale low") and not
meaningful for smoking status; and (3) the preferred binding to
`ObservationReferenceRangeMeaningCodes` (e.g., "normal", "recommended",
"therapeutic", "pre", "post") is simply inapplicable to smoking status.
The reporter suggests curating values that are appropriate for smoking
status on all three elements. No other Jira tickets are formally
cross-referenced, though the request overlaps conceptually with the
AUCDI R2 Tobacco smoking summary model and the US Core Smoking Status
profile (both cited in the resolution). The ticket has been marked
"Triaged" with resolution "Not Persuasive" and was scheduled for
discussion at the 2025-11-27 AU Core TDG call.

## Details

**Description:**

(1) AU Core Smoking — `Observation.status` [sic; the reporter is
referring to the Observation value binding on the Smoking Status
profile]: values are clinically imprecise to be useful for risk
assessment. The value set includes:

- 428041000124106 Occasional tobacco smoker
- 8517006 Former smoker
- 266919005 Lifetime non-smoker
- 77176002 Current smoker

The key purpose of smoking status information is to inform risk
assessment. Clinical evidence indicates that duration of smoking and
duration of cessation are two key variables for patient health risk
assessment (CVS, neurovascular, respiratory and other cancer risks), in
addition to pack-year. See
https://www.webmd.com/smoking-cessation/ss/slideshow-effects-of-quitting-smoking
(also American Cancer Society and CDC infographics).

**Suggestion:** Consider updating the value set to include clinically
more precise smoking status values, in particular about duration since
cessation and quit attempts.

(2) `ObservationInterpretationCodes` (extensible) — value set is
generic / non-specific to smoking (e.g., significant change up/down,
worse, off scale low…).

(3) `ObservationReferenceRangeMeaningCodes` (preferred) — value set is
not applicable to smoking status (e.g., normal, recommended, treatment,
therapeutic, pre, post). These two value sets are generic and may be
useful for diagnostic test results, but are completely non-specific for
use in smoking status observations. Suggestion: curate values that are
appropriate for smoking status observation.

**Resolution (Not Persuasive):**

1. The concept of overall status and value set is aligned with AUCDI
   and also aligned with the US Core Smoking Status Observation profile
   ([us-core-smokingstatus](https://build.fhir.org/ig/HL7/US-Core/en/StructureDefinition-us-core-smokingstatus.html)).
   The suggested clinically more precise codings are not directly
   aligned with the [AUCDI R2](https://sparked.csiro.au/index.php/aucdi-release-2/)
   roadmap for Tobacco Use (3.35 Tobacco smoking summary). AUCDI R2
   defines related concepts — Overall years of smoking (duration),
   Overall quit date (cessation date), Typical amount (units/mass) —
   but the Sparked CDG has designated these as "Chronic condition
   management" use cases rather than Core, so they are not
   automatically included in AU Core. Their inclusion would require
   individual AU Core change requests, aligned with AUCDI, and backed
   by implementers.
2. & 3. `Observation.interpretation` and `Observation.referenceRange.type`
   are not supported in AU Core Smoking Status. Making these part of
   the minimum requirements would need specific change requests raised
   in AU Core and agreed by CDG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

On the AU Core Smoking Status profile:

1. **Observation.value[x] binding** — replace the current
   (Occasional tobacco smoker / Former smoker / Lifetime non-smoker /
   Current smoker) value set with a curated AU-specific value set that
   also captures clinically actionable gradations of status (e.g.,
   recent former smoker vs. long-term former smoker) and cessation
   milestones, aligned to AUCDI wherever possible.
2. **Observation.interpretation** — define an AU Core Smoking Status
   Interpretation value set (e.g., increasing consumption, decreasing
   consumption, cessation attempt in progress, relapse, sustained
   cessation) and bind it (`extensible`) on the profile, replacing the
   generic `ObservationInterpretationCodes`.
3. **Observation.referenceRange.type** — either curate an AU Core
   Smoking Status ReferenceRange Meaning value set or formally mark
   the element Must Not Support on the profile because the generic
   binding is clinically meaningless here.

In parallel, raise companion change requests to add the AUCDI R2
concepts — Overall years of smoking, Overall quit date, Overall pack
years, Typical amount — as extensions or slices on Smoking Status.

#### Justification

The underlying clinical argument — that the current value sets do not
support smoking-related risk stratification — is sound and well
documented in the literature cited. Accepting the request realigns AU
Core Smoking Status with its stated clinical purpose (risk assessment)
rather than a bare presence/absence indicator. Curating
observation-specific interpretation and reference-range-meaning value
sets is consistent with the FHIR guidance that generic code systems
should be replaced when they are not meaningful in context.

---

### Disposition B: Alternative Approach (recommended)

#### Proposal

Keep the AU Core Smoking Status core binding unchanged (aligned with
AUCDI R2 status concept and US Core Smoking Status) and address the
underlying need through **targeted, per-concept change requests**
driven by AUCDI R2 Tobacco smoking summary:

- Open individual AU Core change requests for each AUCDI R2 concept
  (Overall years of smoking, Overall quit date, Typical amount, Overall
  pack years, Overall comment), each with implementer support, so that
  risk-relevant variables are carried as first-class profile elements
  or extensions rather than crammed into a status value set.
- For `Observation.interpretation` and `Observation.referenceRange.type`,
  explicitly mark them as out-of-scope (not supported) on AU Core
  Smoking Status in the profile narrative, eliminating the misleading
  extensible/preferred bindings for implementers of this profile,
  rather than attempting to curate replacement value sets.
- Route the discussion through the Sparked CDG so that any AU Core
  change is anchored in AUCDI rather than diverging from it.

#### Justification

The resolution author correctly observed that the core smoking-status
concept is already aligned with AUCDI and US Core; changing the base
status value set in AU Core would fork AU Core away from both, which is
more disruptive than the clinical benefit warrants. The reporter's
clinical requirements (duration, cessation, pack-years, quit attempts)
are not really about the *status* coding at all — they are additional
observables that AUCDI R2 already models. Channelling the work into
per-concept AU Core change requests keeps AU Core narrow, preserves
alignment with AUCDI and US Core, and gets the reporter the data points
they actually want. Suppressing `Observation.interpretation` and
`Observation.referenceRange.type` on the profile (rather than rebinding
them) directly addresses concerns (2) and (3) without forcing AU Core
to maintain smoking-specific interpretation terminologies.

---

### Disposition C: Decline

#### Proposal

Close FHIR-52401 as **Not Persuasive** (the current state), on the
grounds that:

- The requested status value-set changes conflict with AUCDI R2
  alignment and with US Core Smoking Status alignment;
- The richer clinical data points (duration, cessation date,
  pack-years) belong to the AUCDI R2 "Chronic condition management" use
  case, which the Sparked CDG has explicitly scoped out of AU Core; and
- `Observation.interpretation` and `Observation.referenceRange.type`
  are not declared Must Support in AU Core Smoking Status, so the
  generic bindings are inherited from the base Observation resource
  and not within AU Core's remit to rebind.

Direct the reporter to raise specific AU Core change requests per
AUCDI concept, with implementer evidence.

#### Justification

This is the disposition implied by the current resolution text. It is
defensible because AU Core deliberately minimises divergence from
AUCDI, and the reporter's concerns about generic interpretation /
reference-range bindings are inherited from the base Observation
resource — addressing them in AU Core would set a precedent of rebinding
every non-meaningful base binding across every AU Core profile.

---

### Recommendation

**Recommended disposition:** B

Disposition B directly addresses the clinically valid core of the
reporter's argument — that AU Core Smoking Status does not today carry
the data needed for smoking-related risk stratification — while
preserving the AU Core ↔ AUCDI ↔ US Core alignment that the current
"Not Persuasive" resolution is defending. Disposition A, while
sympathetic to the reporter, would fork the base status value set away
from US Core/AUCDI and would oblige AU Core to maintain bespoke
interpretation and reference-range terminologies, which is more
maintenance than the benefit justifies. Disposition C, as currently
drafted, leaves the reporter with no constructive path and does not
acknowledge the legitimate clinical gap. Disposition B channels the
work through per-concept AUCDI-aligned change requests (the mechanism
already called out in the resolution) and additionally removes the
misleading `Observation.interpretation` and
`Observation.referenceRange.type` bindings from the profile narrative
so that concerns (2) and (3) are concretely resolved without new
terminology work. This is the outcome most likely to be acceptable to
both the Sparked CDG and implementers at the 2025-11-27 AU Core TDG
call.
