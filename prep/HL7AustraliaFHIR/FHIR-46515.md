## Summary

The ticket asks the AU eRequesting (FHIR) IG to "disable" the
`ServiceRequest.doNotPerform` element on the basis that the negation
semantics carried by `doNotPerform` are not relevant to the AU
eRequesting use cases (which centre on positively requested
investigations, referrals, and other services). In practical terms, the
reporter wants the AU eRequesting profile(s) of `ServiceRequest` to
prevent senders from populating `doNotPerform` (and receivers from
having to interpret it). No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this ticket, so the request
stands on its own and the disposition needs to be reasoned from the
base FHIR semantics of `ServiceRequest.doNotPerform` and standard IG
profiling mechanics.

## Proposed Dispositions

> Note: All dispositions below would be applied in the AU eRequesting
> (FHIR) IG. Because no GitHub cross-references are recorded, the
> specific source repo and authoring root for that IG have not been
> resolved here; the implementer should confirm the path to the
> `ServiceRequest` profile StructureDefinition(s) in the AU eRequesting
> IG source before making changes. `doNotPerform` is a **modifier
> element** on `ServiceRequest` (`isModifier = true`), which is a
> material consideration when constraining or removing it.

### Disposition A: Accept As Requested

#### Proposal

In the AU eRequesting `ServiceRequest` profile(s) (e.g., the AU
eRequesting ServiceRequest profile and any derived order/referral
profiles), constrain `ServiceRequest.doNotPerform` to cardinality
`0..0`. Concretely:

- Add a differential element for `ServiceRequest.doNotPerform` with
  `min = 0`, `max = "0"`.
- Add a short rationale to the element definition: e.g., "AU
  eRequesting use cases are positive requests; negation via
  `doNotPerform` is out of scope for this IG."
- Add a publication note in the IG's ServiceRequest profile narrative
  stating that `doNotPerform` is prohibited in AU eRequesting and that
  systems receiving an AU eRequesting `ServiceRequest` may safely
  ignore the element (it must not be present).
- Verify there are no invariants, slices, or examples in the IG that
  reference `doNotPerform`; remove or update as needed.

#### Justification

This is the most direct expression of the reporter's intent and is the
standard FHIR profiling mechanism for "this element is not used in this
context". `doNotPerform` is a boolean modifier element on
`ServiceRequest`; constraining it to `0..0` removes the modifier from
the AU eRequesting wire format entirely, which is the only safe way to
signal "ignore this concept" — leaving it optional but unsupported
would still require receivers to defensively check it because it is an
`isModifier` element. Profiling an unused element to `0..0` is an
established and well-understood pattern in published HL7 IGs.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than prohibiting the element outright, mark
`ServiceRequest.doNotPerform` as **not supported** in the AU
eRequesting `ServiceRequest` profile(s):

- Set `mustSupport = false` on `ServiceRequest.doNotPerform` (or simply
  do not list it as mustSupport) and add an explicit profile-level note
  that "AU eRequesting conformant senders SHALL NOT populate
  `doNotPerform`; receivers MAY ignore it. The element is left at its
  base cardinality to preserve compatibility with downstream profiles
  and tooling that derive from this IG."
- Optionally add a FHIRPath invariant such as
  `doNotPerform.exists().not()` with severity `error` and human text
  "AU eRequesting does not use doNotPerform" — this enforces the
  prohibition at validation time without changing the element's
  cardinality in the differential.

#### Justification

This achieves the same operational outcome (AU eRequesting messages do
not carry `doNotPerform`) while being marginally less disruptive to
downstream profiling: derived profiles that need to re-enable
`doNotPerform` for a different use case can lift the invariant or
re-profile, which is harder to do once a parent profile has set
`max = 0`. It also keeps the element visible in the StructureDefinition
narrative, which can be preferable for implementer education ("we
considered this and chose not to use it"). The trade-off is that
`isModifier` elements left in the structure place a slightly higher
defensive burden on receivers.

---

### Disposition C: Decline

#### Proposal

Decline the change and leave `ServiceRequest.doNotPerform` at its base
FHIR cardinality (`0..1`) in the AU eRequesting `ServiceRequest`
profile(s). The IG would continue to inherit base FHIR semantics for
the element without further constraint or commentary, on the grounds
that the request is a scope reduction without a documented harm: the
element is already optional, and senders that have no use for it
simply omit it.

#### Justification

`doNotPerform` is `0..1` in the base resource; a sender with no
negation use case can already comply with the AU eRequesting profile
without ever populating it. Removing or prohibiting an element that
costs nothing when unused is generally only justified when there is a
concrete interoperability harm (e.g., an `isModifier` element being
populated by mistake and changing the meaning of an order). The ticket
description does not articulate such a harm or cite implementation
evidence; declining keeps the AU eRequesting profile aligned with the
base resource and avoids closing the door on future AU use cases
(e.g., explicit "do not refer", "do not investigate" advance directives
expressed as a `ServiceRequest`).

---

### Recommendation

**Recommended disposition:** A

`ServiceRequest.doNotPerform` is an `isModifier` boolean: when present
and `true`, it inverts the meaning of the entire request from "please
perform X" to "please do not perform X". For an eRequesting IG whose
business purpose is exclusively positive ordering of services, that is
exactly the kind of element that should not be left optionally
populatable — a single mis-set value materially changes clinical
intent, and receivers conformant to the AU eRequesting profile
otherwise have to carry defensive logic for a concept the IG has
declared out of scope. Constraining the element to `0..0` (Disposition
A) makes that scope decision explicit on the wire and at validation
time, which is the strongest signal HL7 IGs have for "this element is
not used here". Disposition B is a reasonable softer option if the
work group anticipates near-term derivative profiles that genuinely
need negation semantics, but in the absence of such evidence the
modifier-element risk argues for the cleaner, validator-enforced
prohibition. Disposition C is the right call only if the work group
believes a future AU eRequesting use case for negation is plausible
enough to outweigh the modifier-element risk today.
