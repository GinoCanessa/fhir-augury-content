## Summary

This ticket asks the HL7 Australia FHIR work group to add migration / implementation
guidance to the AU Base **Encounter** profile to help implementers who had been using
the codes `EMAIL`, `SMS`, `PHONE`, and `VIDEO` from the AU-extended `ActEncounterCode`
value set bound to `Encounter.class`. Those codes were removed by the resolution of
[FHIR-47120](https://jira.hl7.org/browse/FHIR-47120) ("AU Encounter class codes not
appropriate", Persuasive with Modification, Applied), which is a breaking change for
any AU Base 4.2.2+ implementer that had relied on those values; the resolution
directs them to use the more general `VR` (virtual) code instead, and notes that the
finer-grained distinctions only become representable on R5 `Encounter.class`
(0..* CodeableConcept). The reporter cites QA feedback by Danielle Tavares-Rixon on
[hl7au/au-fhir-base#906](https://github.com/hl7au/au-fhir-base/issues/906)
recommending that AU Base call out this change and tell affected implementers what
to do (and explicitly raise the question of whether the removed concepts should
instead surface on `Encounter.type`, which is suggested as a possible follow-up
ticket). The first comment thread on the ticket (Osborne / a second AU FHIR member)
debates whether this is best handled as a STU note / change-notes entry or as
narrative implementation guidance on the profile.

## Related Jira Tickets

### [FHIR-47120: AU Encounter class codes not appropriate](https://jira.hl7.org/browse/FHIR-47120)

The originating change request that this ticket exists to follow up on. Heath
Frankel argued that PHONE / VIDEO / EMAIL / SMS in AU's extended `ActEncounterCode`
were modes rather than classes of encounter. The disposition (Persuasive with
Modification, Applied 2024-10-30 by Michael Osborne) removed those four codes and
kept `RACF`, with the rationale that the extensible binding requires use of the
bound term `VR` for these cases on R4 (1..1 Coding) and that finer-grained coding
only becomes possible on R5 `Encounter.class` (0..* CodeableConcept). FHIR-49151
exists to add the migration / "what implementers using the removed codes should do
now" narrative that FHIR-47120's resolution implies.

Reporter: Heath Frankel; Assignee: Michael Osborne; Commenters: brettesler (Brett
Esler), JIRAUSER25123.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add narrative migration guidance to the **AU Base Encounter** profile in
`hl7au/au-fhir-base` covering the FHIR-47120 removal of `EMAIL`, `SMS`, `PHONE`,
and `VIDEO` from the AU-extended `ActEncounterCode` value set bound to
`Encounter.class`. Specifically:

1. Add a STU / informative note on the AU Base Encounter profile page that
   (a) lists the four removed codes, (b) names the AU Base version in which they
   were removed, (c) directs implementers to use `VR` (virtual) on R4
   `Encounter.class`, and (d) notes that finer-grained coding becomes possible on
   R5 via `Encounter.class` 0..* CodeableConcept.
2. Add a corresponding entry to the AU Base change-log / release-notes page,
   cross-linking FHIR-47120 and this ticket.
3. Do not change any structural artifact (no profile, value set, or binding edit).

This matches both the literal request in the ticket and the lighter "STU note /
change-notes entry" preference expressed by JIRAUSER25123 in the ticket comments.

#### Justification

The underlying breaking change has already been Applied and is shipping to
implementers; the only outstanding harm is the absence of in-spec migration
guidance, which Heath Frankel flagged on Zulip as something that needs to be
resolved before AU Core R1 publishes. A narrative-only change is the lowest-risk
intervention: it has no value-set or binding impact, requires no FHIR-Infrastructure
review, and can be authored entirely within AU Base. It is also consistent with how
AU Base has historically announced breaking changes (the "au-base progress" topic
shows a long history of PRs adding narrative + change-log entries for similar
ballot fixes).

---

### Disposition B: Alternative Approach

#### Proposal

Do everything in Disposition A, **and additionally** open and resolve a sibling
ticket (as suggested in JIRAUSER25409's quoted GitHub comment) that completes the
modelling story for AU users by surfacing the removed concepts on
`Encounter.type` rather than `Encounter.class`. Concretely:

1. Apply the narrative changes from Disposition A.
2. Either (a) extend the AU Base `Encounter` profile's `Encounter.type` binding
   (or `additionalBinding`) to include an AU value set that contains `EMAIL`,
   `SMS`, `PHONE`, `VIDEO` (and any other AU-specific encounter modes worth
   preserving), or (b) define an `Encounter.type` slice/extension that explicitly
   carries those concepts, so that AU implementers who genuinely need to
   distinguish "telephone consultation" from generic "virtual" have a conformant
   place to record it.
3. Coordinate the change with `hl7au/au-fhir-core` so the AU Core Encounter and
   the AUCDI "modality" mapping reference the new `Encounter.type` location
   instead of the now-empty `Encounter.class` slot.

#### Justification

Disposition A only documents the loss; Brett Esler's comment on FHIR-47120 records
that PHONE/VIDEO were originally added because GP systems explicitly distinguish
those modes, and EMAIL/SMS were used to record practitioner time spent on those
contacts as standalone billable events. Pushing implementers to collapse all of
those onto `VR` discards real signal. The FHIR-47120 resolution itself
acknowledges the modelling tension and explicitly suggests CodeableConcept on R5
`Encounter.class` as the eventual home; Disposition B provides the equivalent
landing spot in R4 via `Encounter.type`, which is the AU FHIR convention for "kind
of encounter" beyond ActEncounterCode. The trade-off is wider scope (a profile
edit, a new value set, AU Core coordination) and longer turnaround. The
`hl7au/au-fhir-base` briefing is missing from the local cache, so the exact
authoring file (FSH vs. JSON profile) and any binding-style gotchas
(`additionalBinding` vs. slicing) need to be confirmed against the briefing
before estimating effort.

---

### Disposition C: Decline

#### Proposal

Close FHIR-49151 as **Not Persuasive** on the basis that FHIR-47120 already
documents both the removal and the migration path (use `VR`) in its Resolution
Description, that the change has been Applied and announced via the AU Zulip
"au-base progress" topic and the explicit usage survey, and that adding additional
narrative to the AU Base Encounter profile is therefore redundant.

#### Justification

A defensible "decline" rests on the position that breaking changes are documented
in the FHIR-47120 ticket itself and that downstream IGs (AU Core, AUCDI) are the
correct place to fix consumer-facing mappings, not AU Base. However, this is the
weakest of the three dispositions: Heath Frankel explicitly stated on Zulip that
the AUCDI mapping breakage needs resolving before AU Core R1, the JIRAUSER25409
comment captures real implementer demand for the guidance, and FHIR-47120's
resolution text is not visible from the AU Base profile pages where implementers
actually read. There is no strong harm-reduction argument for declining, only a
"minimum viable spec" argument.

---

### Recommendation

**Recommended disposition:** A (with B kept as an explicit follow-up).

Disposition A delivers exactly what the ticket asks for, satisfies the AUCDI / AU
Core dependency Heath Frankel raised, is narrative-only (so it can land quickly
without further FHIR-I / value-set review), and matches the lighter "STU note +
change-notes entry" approach preferred in the ticket's own comment thread.
Disposition B's additional `Encounter.type` modelling work is genuinely valuable —
it preserves the implementer signal (telephone vs. video vs. email vs. SMS) that
the FHIR-47120 resolution acknowledges is real — but it is a separate change in
scope and should be tracked under its own ticket as the JIRAUSER25409 comment
suggests, so that this ticket does not block on it. Disposition C should not be
chosen: there is documented downstream breakage (AUCDI mapping) and direct
implementer feedback (au-fhir-base#906) that argue against doing nothing.

When work begins, run `repo-analysis hl7au/au-fhir-base if-stale` first so the
briefing is available; the implementation plan should then identify the exact
Encounter profile and change-log files to edit.
