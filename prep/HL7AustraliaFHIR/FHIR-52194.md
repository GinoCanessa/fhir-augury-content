## Summary

Grahame Grieve raises a definitional problem with the **extensible binding**
that AU Base applies to its gender (gender-identity) coded element. The bound
value set offers only `male`, `female`, and `nonbinary`. Because an
extensible binding only permits "other" codes when none of the listed codes
covers the meaning, and the supplied codes already partition the conceptual
space (male, female, and "not male or female"), there is no semantic room
for additional codes. Grahame argues the binding therefore behaves as a
required binding in practice, and proposes that AU Base either (a) replace
the codes with a richer set whose definitions actually leave room for
extension, or (b) honestly mark the binding as **required**.

The ticket was originally filed against AU Core. Triage notes (FHIR WG
co-chairs and AU Core TDG co-chairs) confirm the binding is inherited from
AU Base and have referred the issue to the AU Base project for resolution
(comments by `JIRAUSER25123` 2025-09-02 and `brettesler` 2025-09-22).
Brett Esler's substantive comment (2025-09-05) clarifies that the bound
value set is the NCTS-published set aligned with the ABS *Standard for Sex,
Gender, Variations of Sex Characteristics and Sexual Orientation Variables*
and AIHW METeOR 741825, and that AU Base supports a "Different term (please
specify)" pattern via text-only or alternative codings.

No cross-referenced Jira tickets, GitHub items, or Confluence pages were
returned by the cross-reference index for FHIR-52194. A long-running Zulip
discussion in `#australia > AU Base Patient - genderIdentity binding`

## Details

**Description:**

> The binding is extensible, and the codes are 'male', 'female', and
> 'nonbinary'. Extensible means that other codes cannot be used unless they
> don't mean one of the codes. but this covers male, female, and not male
> or female. So what else is there? We should either use other codes with
> better definitions and values, so that we know what extensible means, or

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the inconsistency in AU Base as a real binding-strength/value-set
mismatch and resolve it in AU Base. Concretely, in AU Base Patient
(`au-fhir-base` IG), change the **`patient-genderIdentity`** extension's
binding from **extensible** to **required**, and keep the existing NCTS
value set (the ABS/AIHW-aligned "Gender Identity Response" set) unchanged.
Document explicitly that implementers who need a value not covered by the
three codes must use the existing "Different term" pattern (text-only /
alternative coding via `Coding.display` or a sibling `Extension`), not a
fourth coded value. The change would land in the AU Base authoring
sources for the genderIdentity extension definition and be accompanied by
release-note text explaining the rationale.

#### Justification

Grahame's underlying point is correct as a matter of FHIR binding
semantics: an extensible binding only permits "other" codes when no listed
code covers the meaning, and once the value set already partitions the
space into `male`, `female`, and "not male/female" (nonbinary), there is
no conceptual room left for an additional code — which is the operational
definition of a *required* binding. Brett's comment that the value set is
locked to the NCTS / ABS standard reinforces this: the value set is
intended as a normative national standard, not a starter set, and AU Base
already has an explicit text-/alternative-coding fallback for "Different
term". Marking the binding as required matches the actual implementation
contract and avoids misleading implementers into thinking they may add
codes that they cannot, in practice, add.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the extensible binding strength but **replace the bound value set**
with a richer NCTS set that genuinely leaves room for additional codes,
and tighten the documentation. Specifically:

1. Bind `patient-genderIdentity` to a value set whose definitions allow
   distinguishable additional concepts beyond `male`/`female`/`nonbinary`
   — for example by including the "I/They use a different term"
   placeholder and any further NCTS-aligned codes (e.g., agender,
   genderqueer, prefer-not-to-say, unknown) where NCTS / ABS publish
   them. The value set composition would be authored in the AU Base IG
   alongside the extension definition.
2. Retain the **extensible** strength, which would now be defensible
   because the value set no longer exhausts the conceptual space.
3. Add invariant or guidance text describing when "different term" coding
   is appropriate vs. when an implementer should propose a new code to
   NCTS.

#### Justification

This addresses Grahame's complaint at its root cause — the codes, not the
binding strength — and preserves AU Base's stated design goal of
*describing how a person identifies* (per Brett's comment) rather than
forcing implementers into a closed three-value choice. It also avoids a
breaking change in semantics for implementers who have built UI/data flows
expecting that "Different term" is a legitimate, non-listed coded
extension; under Disposition A those flows would technically need to drop
the coding and use text only. The trade-off is more work for the AU Base
editors and for NCTS — the value set has to be expanded coherently, and
guidance about when to use "Different term" coded vs. text becomes more
nuanced than under Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing AU Base design is
intentional and operationally sufficient: the three-code NCTS value set
plus the documented "Different term" text/alternative-coding pattern is
the AU jurisdictional answer, and Australian implementers and data
authorities (NCTS, ABS, AIHW) have aligned on it. Close the issue with
an editorial clarification — either in AU Base's binding usage notes or
in a sidebar — making explicit that the *only* permitted "other" usage
of `patient-genderIdentity` under the current extensible binding is the
"Different term" pattern (no further coded values).

#### Justification

The binding has been in production since AU Base shipped genderIdentity,
is built on a national standard (NCTS / ABS / AIHW), and is already
supplemented by a documented "Different term" pattern (Brett's 2025-09-05
comment). Changing strength to *required* risks invalidating existing
content that relies on the documented "alternative codings" pathway, and
expanding the value set risks fragmenting alignment with NCTS unless NCTS
publishes the expanded set first. A purely editorial clarification
preserves the existing implementation base and the national-standard
alignment while addressing the *interpretive* ambiguity Grahame raises.
The risk is that the editorial clarification papers over a real
specification incoherence rather than fixing it.

---

### Recommendation

**Recommended disposition:** A

The ticket is correct on the substance. An extensible binding to a value
set of `male`/`female`/`nonbinary` does not, in any meaningful sense,
leave conceptual room for additional codes — "nonbinary" is by
construction the residual category, and any further coded entry would
either (i) restate one of the three or (ii) be a sub-type that, by FHIR
extensible-binding semantics, the implementer is *not* permitted to use
because a listed code already covers the meaning. AU Base's documented
"Different term" pattern is also already a *non-coded* fallback, which is
exactly what a **required** binding allows (text + extension), so
adopting Disposition A formalises what AU Base already does in practice
without breaking implementers. Disposition B is the more ambitious answer
and is worth keeping on the table if the AU Base editors and NCTS are
prepared to coordinate a value-set expansion, but it is materially more
work and depends on an external party (NCTS) publishing additional codes;
it should not be the default. Disposition C is hard to defend: it
explicitly leaves a known specification incoherence in place and asks
implementers to read sidebar text to discover that the binding does not
mean what its strength says it means.
