## Summary

The reporter asks AU Base FHIR to tighten and split a single overloaded
Identifier.type code. Today, AU Base's `NOI` ("National Organisation
Identifier") code is used both for HPI-O (Healthcare Provider Identifier
– Organisation) values and, by way of the AU Base PAI-O datatype
identifier profile, for PAI-O (PCEHR Assigned Identity for
Organisations) values. The proposal is to:

1. Re-scope the existing `NOI` code so its definition refers exclusively
   to HPI-O — i.e., remove its applicability to PAI-O.
2. Add a new `PAIO` code with display "My Health Record (PCEHR)
   Assigned Identity for Organisations" describing identifiers issued
   by the Australian Healthcare Identifiers Service to organisations
   that are not eligible for an HPI-O (e.g., the MHR System Operator
   and the MHR National Consumer Portal).
3. Update the AU Base PAI-O datatype identifier profile so its
   `Identifier.type` is fixed to the new `PAIO` code instead of `NOI`.

The change emerged from the March 2026 Sydney Connectathon breakout on
Identifier.type values (see Confluence page HA/413057528) and was
scheduled for discussion at the 2026-03-25 AU FHIR WG meeting. A
draft implementation already exists as
[hl7au/au-fhir-base PR #1064](https://github.com/hl7au/au-fhir-base/pull/1064).
Note that no separate Jira tickets are cross-referenced — there are
only Confluence (the breakout investigation page) and Zulip (the
"Possible breaking changes to PAI-O profile" thread) cross-references —
so the related-ticket context is captured through those discussions
rather than linked tickets.

Change definition of NOI to reflect only HPI-O use - remove reference

Update PAI-O datatype identifier Identifier.type to use PAIO code.

Identifier type

| code | display | description |

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change in AU Base FHIR exactly as proposed in the ticket
and as drafted in [hl7au/au-fhir-base PR
#1064](https://github.com/hl7au/au-fhir-base/pull/1064):

1. In the AU Base local Identifier-type CodeSystem (the AU extension
   to `http://terminology.hl7.org/CodeSystem/v2-0203`, commonly
   referenced as `au-hl7v2-0203`), narrow the **definition** of the
   existing `NOI` code so it refers exclusively to HPI-O (Healthcare
   Provider Identifier – Organisation), with display "National
   Organisation Identifier" unchanged. Remove all reference to PAI-O
   from the `NOI` definition.
2. Add a new code `PAIO` to the same CodeSystem, with display "My
   Health Record (PCEHR) Assigned Identity for Organisations" and the
   description text from the ticket (identifiers issued by the HI
   Service to organisations that are not eligible for another type of
   HI Service identifier, e.g. the MHR System Operator and the MHR
   National Consumer Portal). Add `PAIO` to the corresponding
   Identifier-type ValueSet that AU Base uses for `Identifier.type`
   bindings.
3. Update the AU Base PAI-O datatype identifier profile so its
   `Identifier.type.coding` `slicing`/fixed value points at `PAIO`
   in the AU CodeSystem, replacing the current fixed `NOI` value.
4. Coordinate with AU Core: as Brett Esler flags in the comments, AU
   Core has a constraint that depends on the current `NOI` semantics
   for Organization identifiers. Update that AU Core constraint (and
   any AU Core narrative referencing `NOI` for PAI-O) so it is
   consistent with the narrowed `NOI` definition and recognises
   `PAIO` as the correct type for PAI-O values.

#### Justification

The current overloading of `NOI` is genuinely incorrect: `NOI` is
defined as the "National Organisation Identifier" but AU Base uses it
as the fixed type for PAI-O, which is a distinct identifier issued by
the HI Service to organisations that are explicitly *not* eligible
for an HPI-O. That mismatch makes implementer behaviour ambiguous
(systems cannot tell from `Identifier.type` alone whether the value
is an HPI-O or a PAI-O) and undermines the value of `Identifier.type`
as a discriminator. Splitting the codes restores a one-to-one mapping
between code and identifier scheme, aligns AU Base with the way HI
Service actually labels these identifiers, and matches the consensus
reached at the March 2026 Sydney Connectathon Identifier.type
breakout (Confluence HA/413057528). The proposed code, display, and
description text are already drafted by the reporter, the
implementation already exists as PR #1064, and the AU Core impact has
been explicitly identified — so the change is well scoped. The
breaking-change cost (re-fixing `Identifier.type` on PAI-O instances,
and the known downstream impact on ADHA's
`dh-organization-core-1` profile in the My Health Record IG) is real
but limited to a small, identifiable set of organisational-identifier
usages, and is a one-time migration that AU Base implementers can
absorb in the next AU Base release cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Split the codes as proposed, **but** treat the change as a
deprecation rather than a hard re-fix on the PAI-O profile, in order
to soften the breaking impact:

1. Add the new `PAIO` code to the AU CodeSystem and ValueSet exactly
   as in Disposition A.
2. Re-define `NOI` more carefully: instead of "remove all reference
   to PAI-O", mark `NOI` as deprecated for use with PAI-O values
   while leaving the code itself active so existing `Identifier.type
   = NOI` instances on PAI-O profiles do not become invalid
   overnight. Document the new HPI-O-only intent in the code
   description and/or via a `status` / deprecation extension.
3. On the AU Base PAI-O datatype identifier profile, change the
   `Identifier.type` binding from a single fixed code to a binding
   that **prefers** `PAIO` (e.g., a slice or `mustSupport` pattern
   that requires `PAIO` in new instances) but still accepts `NOI`
   for one ballot cycle, with narrative deprecation notice.
4. Coordinate with AU Core to relax (rather than re-write) the
   affected constraint in the same transitional window, then make
   the cutover firm in the following AU Base release.

#### Justification

Disposition B addresses the same underlying need (correct semantics
for PAI-O vs. HPI-O via distinct codes) while explicitly recognising
the breaking-change concern Brett Esler raised on Zulip and the
concrete downstream implementer Reuben Daniels identified
(`dh-organization-core-1` in the My Health Record IG). FHIR's
guidance for fixed/slicing changes on widely-implemented datatype
profiles is generally to provide a transition path rather than to
flip the fixed value in a single release. A two-step
deprecate-then-remove pattern keeps existing PAI-O instances valid
for one cycle, gives ADHA and other implementers a predictable
window to migrate, and still arrives at the same end state as
Disposition A. The trade-off is a slightly more complex CodeSystem
and profile (with deprecation metadata) and a longer overall
timeline, but it materially reduces the risk of orphaning conformant
instances at the moment AU Base publishes.

---

### Disposition C: Decline

#### Proposal

Decline the change as a **breaking modification to a published AU
Base datatype profile that is not justified by a corresponding
interoperability defect**, and instead document the HPI-O vs. PAI-O
distinction in narrative on the existing AU Base PAI-O datatype
identifier profile (and on the `NOI` code description) without
introducing a new `PAIO` code or re-fixing `Identifier.type`. The
existing `Identifier.system` already disambiguates HPI-O values
(`http://ns.electronichealth.net.au/id/hi/hpio/1.0`) from PAI-O
values (`http://ns.electronichealth.net.au/id/pcehr/paio/1.0`), so
implementers already have a deterministic way to tell the two apart.

#### Justification

`Identifier.type` is a hint, not the discriminator: in FHIR, the
authoritative discriminator between two identifier schemes is
`Identifier.system`. Because AU Base already assigns distinct,
well-known systems to HPI-O and PAI-O, no consumer that handles
these identifiers correctly is actually being misled by the shared
`NOI` type code — they should be switching on `system`. Introducing
a new `PAIO` code, narrowing `NOI`, re-fixing the PAI-O datatype
profile, and updating an AU Core constraint is a non-trivial
breaking change that invalidates known production usage (the ADHA
`dh-organization-core-1` profile) for what is essentially a
labelling refinement. If declined, the same outcome — clarity for
implementers — could be achieved with a narrative note on the AU
Base profile and a clarification in the `NOI` code description,
without breaking conformance.

---

### Recommendation

**Recommended disposition:** A

The shared `NOI` code is genuinely incorrect (PAI-O is explicitly
*not* an HPI-O), and the workgroup itself reached this conclusion at
the March 2026 Sydney Connectathon breakout (Confluence
HA/413057528). The reporter has done the work: the new `PAIO` code,
display, and description are drafted; the implementation exists as
[hl7au/au-fhir-base PR #1064](https://github.com/hl7au/au-fhir-base/pull/1064);
and the AU Core impact has already been flagged in comments by Brett
Esler. Disposition C under-weights the value of `Identifier.type` —
even though `Identifier.system` is the authoritative discriminator,
many implementers, downstream profiles, and human readers do rely on
the type code as a label, and continuing to label PAI-O values as
"National Organisation Identifier (NOI)" is actively misleading.
Disposition B is a reasonable mitigation if the workgroup judges the
ADHA `dh-organization-core-1` impact to be too disruptive, but it
adds permanent CodeSystem and profile complexity (deprecated codes,
transitional slices) for a single release of breathing room, and the
known impacted implementer is closely connected to the reporter's
own organisation. Disposition A — accept as requested, land PR
#1064, and coordinate the AU Core constraint update in the same AU
Base ballot cycle — gives implementers a single, clearly-announced
breaking change in a well-publicised release, which is the cheaper
total-cost-of-ownership path.
