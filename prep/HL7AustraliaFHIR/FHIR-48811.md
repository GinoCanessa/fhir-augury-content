## Summary

Heath Frankel asks the AU eRequesting work group to revisit the **AU eRequester Server actor obligations** on the AU eRequesting ServiceRequest profile in light of the resolution of [FHIR-47141](https://jira.hl7.org/browse/FHIR-47141). FHIR-47141 was applied (resolution: *Persuasive with Modification*) and downgraded the AU eRequester Server CapabilityStatement requirement for ServiceRequest `create`/`update` from SHALL to SHOULD, which means a Placer system can now legitimately conform to the Server CapabilityStatement by only providing `read` and `search-type` over its own ServiceRequests. Because of that change, several `SHALL:able-to-populate` obligations currently sitting on the *Server* actor are no longer reasonable for a Placer-acting-as-Server, since the Placer is the source of these elements and either knows them or it does not.

Concretely, the ticket proposes two adjustments to the AU eRequester Server obligations on the ServiceRequest profile:

- Demote `SHALL:able-to-populate` → `SHALL:populate-if-known` on the *optional / context-dependent* elements: `identifier`, `priority`, `encounter`, `reasonCode`, `insurance`, `bodySite`, `note`.
- Promote `SHALL:able-to-populate` → `SHALL:populate` on the *core / always-present* elements: `requisition`, `status`, `intent`, `category`, `code`, `subject`, `authoredOn`, `requester`.

Both adjustments are framed as bringing the Server actor obligations into alignment with the existing AU eRequesting Placer actor obligations, on the basis that the same physical system may now perform either role.

## Details

**Description:**

As a result of [FHIR-47141](https://jira.hl7.org/browse/FHIR-47141), the AU eRequester Server CapabilityStatement can now be implemented by a Placer system where they provide Read and Search interactions. As a result, the AU eRequester Server Actor obligations specified in the AU eRequesting ServiceRequest should be reviewed as it is no longer reasonable to require a placer system that implements the AU eRequester Server CapabilityStatement to `SHALL:able-to-populate` on elements such as `identifier`, `priority`, `encounter`, `reasonCode`, `insurance`, `bodySite` and `note` given that the placer system may not have a known value for these elements. These should be changed to `SHALL:populate-if-known` to align with the AU eRequesting Placer Actor obligations.

Similarly, elements such as `requisition`, `status`, `intent`, `category`, `code`, `subject`, `authoredOn` and `requester` should have the AU eRequesting Server obligations changed to `SHALL:populate` rather than `SHALL:able-to-populate` to align with AU eRequesting Placer Actor obligations.

### [FHIR-47141: Provide eRequesting Responder Actor and Capability Statement](https://jira.hl7.org/browse/FHIR-47141)

Heath Frankel's earlier change request that triggered this one. Originally proposed adding a new "eRequesting Responder" actor with a Capability Statement supporting only `read` and `search-type` on ServiceRequest, so that a placer/intermediary could expose its own service requests without having to implement the full create flow. The applied resolution did *not* add a new actor; instead it relaxed the existing AU eRequester Server CapabilityStatement so that `create` and `update` for ServiceRequest are SHOULD (not SHALL), while `read` and `search-type` remain SHALL. That CapabilityStatement-level relaxation is exactly what makes the obligation language on the Server actor in the AU eRequesting ServiceRequest profile inconsistent today, and it is the direct motivation for FHIR-48811.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the AU eRequesting ServiceRequest profile (`StructureDefinition-au-erequesting-servicerequest`), update the obligation extensions on the **AU eRequester Server** actor exactly as Heath specified:

- Change the AU eRequester Server obligation from `SHALL:able-to-populate` to **`SHALL:populate-if-known`** on: `identifier`, `priority`, `encounter`, `reasonCode`, `insurance`, `bodySite`, `note`.
- Change the AU eRequester Server obligation from `SHALL:able-to-populate` to **`SHALL:populate`** on: `requisition`, `status`, `intent`, `category`, `code`, `subject`, `authoredOn`, `requester`.

Leave the AU eRequester Placer actor obligations unchanged. Re-run the IG build and update any obligation summary table in the profile narrative so the Server and Placer actor columns now agree on these elements.

#### Justification

This is a minimal, mechanical follow-on to FHIR-47141, which established that the same physical Placer system can satisfy the AU eRequester Server CapabilityStatement using only `read` + `search-type`. Once that is true, requiring a Placer-acting-as-Server to be *able* to produce values it never had (e.g. `encounter`, `insurance`, `bodySite`) is incoherent — `able-to-populate` only makes sense for systems that authoritatively own the data. Demoting context-dependent elements to `populate-if-known` correctly expresses "if you have it, send it"; promoting the always-present elements to `populate` correctly expresses "you must always send a real value", which matches what the existing Placer obligations already say. The change is internally consistent (Server and Placer actors then carry the same obligations on the same elements), it does not relax the wire-level requirements on a true Placer (whose Placer-actor obligations are unchanged), and it does not affect validation since obligations describe sender/receiver behaviour rather than instance constraints — a point Lloyd McKenzie made explicitly in the related `#conformance` thread.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the same element-by-element re-labelling as Disposition A, but generalise the framing so the IG no longer maintains two parallel obligation sets that have to be kept aligned by hand. Specifically:

1. In the AU eRequesting ServiceRequest profile, define **a single shared obligation set** on each element that applies to *both* the Placer and Server actors (or define an "AU eRequesting ServiceRequest source" actor that the Placer and Server both inherit from for these obligations), with the values from Disposition A: `populate` for `requisition`, `status`, `intent`, `category`, `code`, `subject`, `authoredOn`, `requester`; `populate-if-known` for `identifier`, `priority`, `encounter`, `reasonCode`, `insurance`, `bodySite`, `note`.
2. Reserve actor-specific obligations on the Server actor only for elements that genuinely behave differently between Placer and Server roles (none, on this profile, after the alignment above).
3. Add a short narrative section to the profile explaining that — post FHIR-47141 — the Server CapabilityStatement may be satisfied by a Placer using only `read`/`search-type`, and that the obligations therefore describe what the *source of the ServiceRequest* must produce, regardless of which CapabilityStatement they conform to.

#### Justification

Disposition A fixes the immediate inconsistency but leaves the IG carrying two near-identical obligation sets (Placer vs Server) that must be hand-edited together every time an element changes; in practice this is exactly how the current divergence arose. Collapsing the duplication — either via a shared obligation set or via actor inheritance — removes the future drift risk and makes the post-FHIR-47141 model explicit: the obligations follow the *role of producing the ServiceRequest*, not the CapabilityStatement label. The trade-off is implementation effort (the IG editors have to refactor the obligation extensions and check tooling support for shared/inherited obligations) and a slightly heavier ballot impact, since the change touches the structural model of the profile rather than just element values.

---

### Disposition C: Decline

#### Proposal

Decline the change. Keep the existing AU eRequester Server obligations on the AU eRequesting ServiceRequest profile (`SHALL:able-to-populate` on the listed elements), and instead update the profile narrative to clarify that a Placer system implementing only the `read`+`search-type` half of the AU eRequester Server CapabilityStatement is not expected to satisfy the Server-actor *obligations* on every element — the obligations apply only to systems that authored the ServiceRequest.

#### Justification

The defensible decline argument is that obligations describe behaviour of the *system that owns the data*, and a Placer-acting-as-Server is, semantically, still acting as a Placer when it returns its own ServiceRequests on a `read`/`search-type` — so the existing Placer-actor obligations already cover that case and the Server-actor column is irrelevant for that deployment. On this view, the inconsistency reported by FHIR-48811 is documentary, not normative, and a clarifying note costs less than re-editing every Server obligation. This argument is weakest where the Server actor obligations are likely to be tooled (validators, conformance dashboards) against systems that present the AU eRequester Server CapabilityStatement, since those tools have no way to know which underlying actor a given system "really" is.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right starting point: it directly fixes the inconsistency that FHIR-47141 introduced, it touches only obligation values on a single profile, and the end state is exactly what the AU eRequesting Placer actor already says — so there is no new design to litigate and no new normative content to ballot. Disposition B is the *better long-term* shape (and the work group should consider whether the obligation refactor is feasible inside the current ballot window), but it carries refactor and tooling risk that does not need to block fixing the immediate Server/Placer divergence. Disposition C is the weakest option: it leaves an obligation table that disagrees with itself for any reader who does not already know the FHIR-47141 backstory, and it asks every downstream conformance tool to special-case "is this Server actually a Placer?" — which the obligation extension was specifically designed to avoid.
