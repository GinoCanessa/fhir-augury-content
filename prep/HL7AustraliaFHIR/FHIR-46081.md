## Summary

The reporter asks the HL7 Australia FHIR work group to use the new
`additional-binding` extension (defined in the FHIR Tools IG) to
formalise the value-set guidance that currently lives only as
implementation narrative on the AU Base PractitionerRole profile. The
ticket points at the `additionalBindings` branch build of AU Base as
the in-flight place where this is being prototyped, and frames the work
as a clarification of how value sets are bound to elements on
`au-practitionerrole` rather than a change to which value sets apply.
There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items recorded in FHIR Augury for this ticket, so the request stands on
its own narrative.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

On `StructureDefinition-au-practitionerrole`, replace the
prose-only value-set guidance currently carried in the profile's
implementation narrative with formal `additionalBinding` extension
entries on the affected coded elements (typically `PractitionerRole.code`,
`PractitionerRole.specialty`, and any other elements where AU Base today
documents "preferred", "candidate", or "consider also" value sets in
text). For each documented value set, add an `additionalBinding` with
the appropriate `purpose` (e.g., `preferred`, `extensible`, `candidate`,
`current`, `ui`) and a `valueSet` pointing at the canonical AU value
set, and reduce the prose to a one-line pointer into the formal
binding. Mirror the pattern shown on the `additionalBindings` branch
build at
`https://build.fhir.org/ig/hl7au/au-fhir-base/branches/additionalBindings/StructureDefinition-au-practitionerrole.html`,
and merge that branch into the next AU Base release once the bindings
have been reviewed.

#### Justification

The `additional-binding` extension exists precisely to express
secondary, non-normative, or purpose-specific bindings that previously
had to live in narrative — promoting AU Base's existing prose into
machine-readable bindings makes the guidance enforceable by validators
and discoverable through tooling such as IG-publisher snapshots and
profile-rendering. It also aligns AU Base with the direction
international IGs (e.g., IPS, US Core) are taking once the FHIR Tools
extension stabilises, and the reporter has already produced a working
prototype on the `additionalBindings` branch, so the spec impact is
demonstrable rather than hypothetical.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt `additionalBinding` on `au-practitionerrole` only for purposes
where the current AU Base prose maps cleanly onto a defined `purpose`
code (e.g., `preferred`, `extensible`), and keep the remaining prose
guidance in the profile narrative until the FHIR Tools `additional-binding`
extension is published in a stable release of the FHIR Extensions Pack
or absorbed into the core spec. Concretely: pick the subset of
PractitionerRole bindings the work group is most confident about (most
likely `PractitionerRole.code` and `PractitionerRole.specialty`),
formalise those via `additionalBinding`, and explicitly note in AU
Base's "About this IG" / dependencies page that the extension is being
used in a draft capacity and may be replaced when the extension's URL
or shape stabilises.

#### Justification

The `additional-binding` extension is currently defined in the FHIR
Tools IG, which is itself in continuous integration. Locking the entire
PractitionerRole profile to that extension up-front exposes AU Base
implementers to churn if the extension's canonical URL, element names,
or `purpose` value set change before publication. A staged adoption
captures most of the validator/tooling benefit for the highest-value
bindings while keeping the rest as prose that can be migrated in a
later AU Base release without forcing a breaking re-publication of the
profile. This also gives the work group a chance to socialise the
pattern with AU implementers before mandating it across every coded
element.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Persuasive with Modification — defer" or
"Not Persuasive": leave the AU Base PractitionerRole bindings as they
stand today (a single primary binding plus prose guidance), and revisit
once the FHIR Tools `additional-binding` extension has shipped in a
published release rather than a CI build.

#### Justification

The extension being relied on (`StructureDefinition/additional-binding`
in the FHIR Tools IG) is sourced from `build.fhir.org`, i.e., the CI
build, and is not yet part of a published FHIR Extensions Pack release.
Adopting an unpublished extension in a national base IG creates a
dependency on an artefact that can change shape without notice and that
downstream AU implementers may not yet have terminology or validator
support for. The current prose guidance, while not machine-enforceable,
is not actively wrong — declining (or deferring) avoids re-publishing
the profile twice in quick succession if the extension's contract
shifts before its first stable release.

---

### Recommendation

**Recommended disposition:** B

The underlying request is sound and the prototype branch already
demonstrates feasibility, so a flat decline (Disposition C) leaves real
implementer value on the table — the AU Base prose around
PractitionerRole bindings is exactly the kind of guidance the
`additional-binding` extension was designed to formalise. However,
adopting the extension wholesale across `au-practitionerrole` in one
pass (Disposition A) couples AU Base to an IG that is still iterating
in CI, which is awkward for a national base specification that other
AU IGs depend on. Disposition B captures the high-confidence wins now
(typically `PractitionerRole.code` and `PractitionerRole.specialty`),
explicitly flags the dependency as draft, and leaves room to extend the
pattern to the rest of the profile — and ultimately to the other AU
Base profiles the reporter alludes to — once the extension stabilises.
The work group should also use the discussion to decide whether the
same staged rollout should be tracked as follow-up tickets per AU Base
profile rather than re-opening this ticket repeatedly.
