## Summary

The ticket requests a new AU Base profile of `Identifier` for the
Australian *Healthcare Administration Entity Organisation* (HAE)
identifier, which is issued by Services Australia and is structurally
similar to existing HPI-O / PAI-O organisation identifiers already
profiled in AU Base. The reporter has produced a draft profile
(`StructureDefinition/au-hae`) on the `cr-FHIR-54928-HAE` branch of
`hl7au/au-fhir-base` modelled on `au-hpio`, and points to the ADHA
OID register and the assigned namespace
`http://ns.electronichealth.net.au/id/hi/hae/1.0` as the canonical
sources. The ticket is paired with sibling change request
[FHIR-54923](https://jira.hl7.org/browse/FHIR-54923) (HSP-O profile of
Identifier), which has the same shape and the same open issues —
together they cover the two new HI service organisation roles that are
not yet represented in AU Base. An open question raised on both
tickets is what `Identifier.type` to use: the draft uses `NOI`, but
AU Core Organization currently constrains `NOI` to HPI-O and PAI-O
only, so accepting either ticket as drafted requires an aligned change
to AU Core Organization (or a new identifier type code). The reporter
notes the FHIRWG agreed on 2026-02-18 to revisit the type allocation
at the 2026-03 HL7 Australia Sydney FHIR Connectathon, where a new
type code may be allocated to HAE.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new profile `StructureDefinition/au-hae` to AU Base
(`hl7au/au-fhir-base`), promoting the existing draft on the
`cr-FHIR-54928-HAE` branch. The profile constrains `Identifier` with:

- `Identifier.system` fixed to
  `http://ns.electronichealth.net.au/id/hi/hae/1.0` (the ADHA-asserted
  namespace, OID-rooted per the HI OID register).
- `Identifier.value` constrained to the format Services Australia uses
  for HAE identifiers (length / Luhn check / regex per the HI OID
  register entry).
- `Identifier.type` populated with `NOI` from
  `http://terminology.hl7.org/CodeSystem/v2-0203` (matching the
  approach used by `au-hpio`/`au-paio`) **and** at the same time make
  the parallel change in AU Core to allow `NOI` for HAE on
  `Organization.identifier` (the AU Core Organization invariant
  currently restricts `NOI` to HPI-O and PAI-O — see the reporter's
  2026-01-20 comment).
- Take FHIR-54923 (HSP-O) along for the ride as a parallel profile,
  given the two are structurally identical and share the AU Core
  Organization change.

Add a draft example once Services Australia publishes permitted test
values (per the reporter's note).

#### Justification

The HAE identifier is a real, externally-governed identifier scheme
(Healthcare Identifiers Act 2010, ADHA OID register, Services
Australia issuing authority) with a stable namespace already asserted
at `http://ns.electronichealth.net.au/id/hi/hae/1.0`. AU Base already
profiles the analogous HPI-O / PAI-O identifiers, so adding HAE keeps
AU Base aligned with the full set of HI-service organisation roles
implementers will encounter on the wire. The FHIRWG already
acknowledged the work on 2026-01-14 and 2026-02-18 and the reporter
has a published draft, so the spec change is concrete and low-risk.
The dependent AU Core change (broadening `NOI` to HAE — and HSP-O via
FHIR-54923) is small and additive: it extends an existing invariant
rather than reshaping the slice.

---

### Disposition B: Alternative Approach

#### Proposal

Same `StructureDefinition/au-hae` profile in AU Base, but **do not**
reuse `NOI`. Instead:

1. Define a new identifier-type code (e.g., `HAEI`) — either as a new
   AU-managed CodeSystem entry or as a request into v2-0203 — and use
   it on the new profile.
2. Apply the same change for HSP-O (FHIR-54923 — code `HSPI` or
   similar).
3. Leave AU Core's existing `Organization.identifier` invariant
   intact; the new types are simply added to the allowed set when
   AU Core is next versioned.

This is the "review the type at the 2026-03 Sydney connectathon"
direction the reporter flagged on 2026-02-18.

#### Justification

`NOI` ("National Provider Identifier", v2-0203) is generic and
overloaded: AU Base / AU Core have so far used it sparingly (HPI-O,
PAI-O) and the AU Core Organization invariant deliberately enumerates
which national-org identifier each `NOI`-typed slice represents. That
enumeration only works because the set is small and curated; quietly
expanding `NOI` to also mean HAE (and HSP-O) erodes the
discriminator's signal — a downstream consumer that sees
`type=NOI, system=…hae/1.0` already has to read the `system` to know
what kind of organisation it is, so the `type` adds no value. A
dedicated code per HI-service role keeps the typing precise, avoids
chained changes to AU Core's invariant every time a new HI role
appears, and matches the pattern AU Base already uses for other
purpose-built identifier types. Trade-off vs. A: a small amount of
extra terminology work (and a v2-0203 / AU CodeSystem decision) in
exchange for cleaner typing and no AU Core ripple effect.

---

### Disposition C: Decline

#### Proposal

Do not add `StructureDefinition/au-hae` to AU Base. Implementers
needing to exchange a HAE identifier should populate a generic
`Identifier` with `system =
http://ns.electronichealth.net.au/id/hi/hae/1.0` and the appropriate
`value`, relying on the existing namespace assertion rather than a
named profile. Close the ticket as "addressed by existing mechanism"
and (consistently) decline FHIR-54923.

#### Justification

The only thing a HAE-specific profile actually constrains beyond a
plain `Identifier` is `system`, an example, and possibly a `type`
code — `system` alone is already a discriminator, and the namespace
URL is already published by ADHA. A profile per HI-service-role does
add some implementer ergonomics (canonical URL to slice on, a
documented place to attach guidance) but it also commits AU Base to
maintaining one profile per Services-Australia role indefinitely, and
each new one (as this ticket itself demonstrates) drags a coupled
AU Core invariant change with it. If the workgroup wants to slow that
proliferation down, declining is defensible.

---

### Recommendation

**Recommended disposition:** B

The use case is real and AU Base should publish a profile — declining
(C) loses the implementer-facing benefits AU Base already invests in
for HPI-O / PAI-O and would be inconsistent with how the WG has
treated previous HI-service identifiers. Between A and B, B is
preferable because the reporter has *already* surfaced the AU Core
ripple as an "Issue" on the ticket and the FHIRWG has *already*
agreed (2026-02-18) to revisit the type allocation at the Sydney
connectathon — that is the workgroup signalling it is uncomfortable
overloading `NOI` further. Adopting B now lets the connectathon focus
on naming the new type(s) and folds FHIR-54923 (HSP-O) into the same
decision, which avoids a second round of the same AU Core invariant
edit. The cost is one terminology decision plus a small CodeSystem
edit; the benefit is a typing model that scales as new HI-service
organisation roles emerge. If the connectathon explicitly decides to
keep using `NOI`, fall back to A — but the change to AU Core
Organization must then be tracked and merged in lock-step with this
ticket and FHIR-54923.
