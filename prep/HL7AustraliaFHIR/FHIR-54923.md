## Summary

The reporter requests a new AU Base profile of `Identifier` for the
**HSP-O** (Healthcare Support Service Provider for Organisation)
identifier issued by Services Australia. HSP-O is a healthcare
identifier analogous to HPI-O, defined in the *Healthcare Identifiers
Act 2010* and the ADHA OID register, with namespace
`http://ns.electronichealth.net.au/id/hi/hspo/1.0`. The request is to
add an `au-hspo` profile parallel to the existing
[`au-hpio`](https://hl7.org.au/fhir/6.0.0/StructureDefinition-au-hpio.html)
profile in AU Base, so that implementers have a canonical way to
represent HSP-O values on `Organization.identifier`. A draft
implementation is already published on the CI build at
`StructureDefinition-au-hspo.html` on branch
`cr-FHIR-54923-HSP-O`. A second related identifier introduced by the
same legislation update — **HAE** (Healthcare Authorised Entity) — has
been raised in parallel discussions but is out of scope for this
ticket.

A cross-cutting issue surfaced by the reporter is the choice of
`Identifier.type`: the draft uses the V2-derived code **NOI** (National
Organisation Identifier), but [AU Core
Organization](https://hl7.org.au/fhir/core/2.0.0-preview/StructureDefinition-au-core-organization-definitions.html#key_Organization.identifier)
constraint `au-core-org-01` currently restricts NOI to HPI-O and PAI-O
only. Adopting NOI for HSP-O would therefore require a coordinated
change to the AU Core invariant (or use of a different/new type code).
This question was discussed at the 2026-01-14, 2026-02-18 and 2026-02-25
HL7 AU FHIRWG meetings and at the 2026-03 Sydney Connectathon
Identifier breakout, where the topic was marked resolved.

No related Jira tickets were returned by the cross-reference index;
the GitHub issue
[`hl7au/au-fhir-base#1037`](https://github.com/hl7au/au-fhir-base/issues/1037)
referenced in the description is noted by the reporter as the
superseded GitHub-based version of this request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new `au-hspo` profile of `Identifier` to AU Base, parallel in
shape to the existing `au-hpio` profile. Specifically:

- Profile URL `http://hl7.org.au/fhir/StructureDefinition/au-hspo`,
  base `Identifier`.
- Fix `Identifier.system` to
  `http://ns.electronichealth.net.au/id/hi/hspo/1.0` (per the ADHA
  namespace assertion).
- Constrain `Identifier.value` to a 16-digit numeric pattern matching
  the HI service rules (mirroring `au-hpio`'s value constraint, with
  any HSP-O-specific prefix once Services Australia confirms it).
- Set `Identifier.type` using the same V2-derived approach as
  `au-hpio`: bind to `hl7VS-identifierType` with the type code **NOI**
  (`http://terminology.hl7.org/CodeSystem/v2-0203`), as already used
  in the CI draft on branch `cr-FHIR-54923-HSP-O`.
- Adopt the draft already at
  [`StructureDefinition-au-hspo.html`](https://build.fhir.org/ig/hl7au/au-fhir-base/branches/cr-FHIR-54923-HSP-O/StructureDefinition-au-hspo.html)
  as the basis, plus a worked example once Services Australia
  publishes a permitted example HSP-O value.
- Coordinate a companion change to AU Core Organization invariant
  `au-core-org-01` to add HSP-O to the set of profiles for which NOI
  may be asserted (otherwise an HSP-O identifier with `type = NOI`
  would fail AU Core validation on `Organization`).

#### Justification

This is the path the reporter has already drafted and that the
2026-03 Sydney Connectathon Identifier breakout converged on
("HSP-O and HAE have distinctive prefixes, they can follow the
pattern of HPI-O and PAI-O"). It maximises consistency with the
existing identifier profile family in AU Base, keeps V2↔FHIR
translation tooling on a single pattern, and matches Services
Australia's framing of HSP-O as a sibling identifier to HPI-O under
the same Healthcare Identifiers Service. The known gotcha — the AU
Core `au-core-org-01` invariant — is addressed explicitly in the
companion change so this profile does not become unusable on
`Organization` resources validated against AU Core.

---

### Disposition B: Alternative Approach

#### Proposal

Add the `au-hspo` profile as in Disposition A, but **do not reuse the
NOI type code**. Instead either (i) request a new V2 0203 code for
HSP-O (e.g. `HSPO`) and bind `Identifier.type` to that, or (ii) leave
`Identifier.type` optional / unconstrained on `au-hspo` and rely on
`Identifier.system` alone to identify the namespace. AU Core
Organization `au-core-org-01` would then be left untouched in (i) and
(ii); under (i) the constraint could be tightened over time to also
permit the new HSPO code.

#### Justification

The Zulip thread *Use of Identifier.type data element* established
that `Identifier.type` is not sufficient to determine `system` even
for NOI today (NOI maps to either HPI-O or PAI-O namespaces), and
several participants questioned whether `Identifier.type` carries
enough semantics to be worth constraining at all on profiled
identifiers. A dedicated `HSPO` type code would remove the structural
ambiguity that NOI introduces (one code, three or more namespaces)
and would not require relaxing an AU Core invariant that exists for a
reason. Leaving type optional is the lighter-weight variant for IGs
that don't need round-trip V2 mapping. The trade-off vs. Disposition A
is dependency on an external (V2) code addition for path (i) and a
loss of consistency with the `au-hpio`/`au-paio` family for path
(ii).

---

### Disposition C: Decline

#### Proposal

Do not add `au-hspo` to AU Base in this release. Defer until Services
Australia publishes (a) the production HI Service rules for HSP-O
including a permitted example value, and (b) a written position on
the appropriate `Identifier.type` code, and until the FHIRWG decides
whether `Identifier.type` should remain mandatory across AU Base
identifier profiles at all.

#### Justification

Two open dependencies are noted in the ticket itself: Services
Australia has not yet provided permitted example values for HSP-O,
and the question of which `Identifier.type` is correct (NOI vs. a new
code) is the subject of a still-active design discussion that also
re-opens whether `Identifier.type` belongs in these profiles in the
first place. Publishing a profile now risks a breaking change in the
next AU Base release when those decisions land. However, this option
should be weighed against the fact that the underlying need is
real and immediate — Services Australia is issuing HSP-O identifiers
now, so implementers will create their own divergent representations
in the absence of an AU Base profile.

---

### Recommendation

**Recommended disposition:** A

The reporter has already published a working CI build of the profile,
the Identifier breakout at the 2026-03 Sydney Connectathon explicitly
endorsed the "follow the HPI-O/PAI-O pattern" approach, and the
companion `au-core-org-01` adjustment is a small, well-understood
change. Disposition B's structural arguments (NOI is ambiguous;
`Identifier.type` may be over-constrained) are real but are a
broader design conversation that should be tracked separately rather
than blocking publication of an HSP-O profile that implementers need
now. Disposition C defers a real and immediate need without resolving
any of the open questions — Services Australia's example values can
be added in a subsequent patch once published. Accept as drafted,
publish in the next AU Base release alongside the matching AU Core
constraint update, and keep the broader `Identifier.type` policy
discussion open as its own work item.
