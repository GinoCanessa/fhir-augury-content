## Summary

FHIR-56190 is a sweeping editorial and semantic clean-up of the FHIRPath
specification's handling of `Quantity` values, with particular focus on
unit conversions (UCUM and calendar units) and the operators that consume
them (equality `=`, equivalence `~`, comparison `<`/`<=`/`>`/`>=`, math
`+`/`-`/`*`/`/`, and date/time arithmetic). Brian Postlethwaite has
already drafted the proposed text in the preview build at
`https://build.fhir.org/ig/HL7/FHIRPath/branches/BP-2026-03-quantity-preview/`
and is asking the workgroup to accept the consolidated change so it
supersedes several previously-discussed but not-yet-applied tickets.

The ticket is explicitly framed as a roll-up that should collapse three
existing related tickets into a single resolution:
[FHIR-49525](https://jira.hl7.org/browse/FHIR-49525) (Quantity math and
unit conversion — left-operand-unit rule, `toQuantity` for explicit unit
selection),
[FHIR-53249](https://jira.hl7.org/browse/FHIR-53249) (ambiguity in
quantity equivalence — "most granular unit" vs. decimal-equivalence
precision rules), and
[FHIR-53250](https://jira.hl7.org/browse/FHIR-53250) (calendar-duration
equivalence transitivity conflicts between `1 year ~ 1 'a' ~ 365.25 'd'`
and `1 year ~ 365 days ~ 365 'd'`). Brian's intent is that those three
tickets be dispositioned as redundant once FHIR-56190 lands.

Concretely the proposal touches: the Quantity Literal section
(case-sensitivity, cross-links, removing duplicated narrative);
Time-valued Quantities (label UCUM `'a'`/`'mo'` as *mean* Julian
year/month, name the two code systems, distinguish UCUM-internal vs.
cross-system conversions); Conversion / `toQuantity` /
`convertsToQuantity` (add implicit `long → Quantity`, default unit `'1'`,
`45.toQuantity('g')` form, central "Unit conversions" section with a
Time-valued sub-section that includes the UCUM conversion factors and
"shortest path" rule for `'a'`/`'mo'`); Equality and Equivalence (require
UCUM-commensurable units, drop "uses the most granular unit", route
through the new Unit-conversions section, allow cross-code-system
comparison of time-valued quantities under `~`); Comparison and
`comparable()` (parallel cleanup, fix the `iff` example); Math
(left-operand unit for `+`/`-`, UCUM rules for `*`/`/`, calendar
multiplicand handling, `long` support, empty for unsupported cases);
and Date/Time arithmetic (collapse the bullet list and table into a
single table covering all units and input types, codify partial-date
behaviour, decimal-portion handling and warnings, cyclic Time wrap-around).

The ticket has been resolved **Persuasive** and is on the ITS Conference
Call agenda for ratification.

## Details

**Description:**

This is a VERY large set of changes, and also covers several existing
tickets (linked). The actual changes have been applied in a branch for
review at
`https://build.fhir.org/ig/HL7/FHIRPath/branches/BP-2026-03-quantity-preview/`
(green highlights mark the diffs).

The description is organised as a checklist by FHIRPath section. The
salient points (full text retrieved via `fhir-augury-cli get`):

- **Quantity Literal** — surface case-sensitivity earlier; link from the
  literal to equality, equivalence, comparison and arithmetic operations;
  refactor duplicated concepts; note that precision/rounding are *not*
  done as part of conversion but in the operations/functions themselves;
  point at implicit conversion or `toQuantity` for explicit conversion.
- **Time-valued Quantities** — label UCUM `'a'` and `'mo'` as the *mean*
  Julian year/month (not exact); move the table caption under the table
  with an inline example; clarify that UCUM handles UCUM conversions and
  FHIRPath defines calendar conversions and the cross-system mapping;
  list the system URIs of the two code systems used in FHIR.
- **Conversion** — add implicit `long → Quantity`; note that
  `decimal/integer/long → Quantity` produces unit `'1'`.
- **toQuantity** — when a string has no unit, the result has unit `'1'`;
  reorder rules vs. examples; add the `45.toQuantity('g')` form for
  supplying a unit on a numeric input; if input is non-convertible,
  return empty; add many examples; introduce a central "Unit
  conversions" section referenced from everywhere unit conversion is
  performed; under it, a "Time-values unit conversions" sub-section
  containing the existing table augmented with UCUM conversion factors,
  noting that all factors except year/month are identical and so
  inter-changeable; for `'a'`/`'mo'` cross-system conversions take the
  shortest path; spell out implicit-conversion behaviour (so as not to
  break equivalence semantics); note that conversion is generally not
  required for date/time operations except for partial-date handling.
- **convertsToQuantity** — remove duplication with `toQuantity`; add
  examples (e.g., `CEL` → `degF` returning `false` when no valid
  conversion).
- **Equality `=`** — link out to date/time equality; add Quantity to the
  bulleted list with a back-reference to Quantity Equality and the new
  conversion section; add many reasoned examples. **Quantity Equality**:
  require UCUM-commensurable units; drop "uses the most granular unit";
  refer to the new Unit-conversions section; expand on time-valued
  cases; cross-code-system comparison at year/month level requires
  explicit `toQuantity`. **Date/Time Equality**: refine examples.
- **Equivalence `~`** — Quantity in the bullet list, link to Quantity
  Equivalence; add reasoned examples. **Quantity Equivalence**: same
  commensurability requirement; route through Unit-conversions; explain
  precision/rounding as part of equivalence (not conversion); after
  conversion, decimal-equivalence rules apply (trailing zeros ignored,
  etc.); time-valued quantities *can* be compared across code systems
  under `~` (with reasoned examples).
- **Comparison** — include `long` (implicit); note decimal trailing-zero
  handling matches equality; require commensurability; drop "most
  granular unit"; route through Unit-conversions; year/month time-valued
  comparison across code systems is not comparable without explicit
  `toQuantity`. **comparable()**: cross-link Unit-conversions and the
  equality/comparison sections; refine the UCUM-commensurable wording;
  add examples covering supported / unsupported conversions; fix the
  `iff` example to return a more sensible default.
- **Math** — clarify "implicit conversion"; cross-link the date/time
  math section; remove the generic "operating on quantities with units"
  paragraph (the per-operation sections cover it, especially since
  `*`/`/` don't need the same units); state that engines without UCUM
  must return empty when they cannot convert.
  - **Multiplication**: implicit conversion; `long`; empty result for
    calendar-quantity multiplication except where the unit is `'1'`;
    empty when types are wrong; explain how to multiply quantities;
    reasoned examples.
  - **Division**: `long`; result is decimal except where quantities are
    involved; resulting units follow UCUM rules; calendar-quantity
    division empty except for unit `'1'`; reasoned examples
    (e.g., `60 / 1 's' => 60 '/s'`); mixed numeric/quantity arguments
    implicitly convert (numeric → unit `'1'`).
  - **Addition**: `long`; output type via implicit conversion; reference
    the date/time arithmetic section; document implicit-conversion
    behaviour, especially with unit conversions; **resulting unit is the
    unit of the left argument** (consistent with current engine
    behaviour); many reasoned examples including time-valued and
    cross-system.
  - **Subtraction**: `long`; output type; reference date/time
    arithmetic; treat as addition with a negated value plus extra
    examples.
- **Date/Time arithmetic** — merge the bullet list and the table into a
  single table (removing redundancy); fold the `'a'`/`'mo'` "signals
  error" rule into the table; include all supported units (UCUM and
  calendar) and input datatypes; note leap-second omission; require
  partial-input unit conversion to the precision actually present;
  ignore decimal portions except for seconds/millisecond precision (and
  warn — authors can use `round()`/`floor()`/`truncate()`/`ceiling()`);
  note cyclic wrap-around for the `Time` datatype. **Addition** and
  **Subtraction**: strip the duplicate definitions now in the parent
  section; add many more reasoned examples, with partial-date examples
  in their own sub-section.

## Linked Jira Tickets

No explicit Jira issue links are recorded on FHIR-56190
(`Jira.items.get-links` returns an empty array). The tickets below are

### [FHIR-49525: Quantity math and unit conversion](https://jira.hl7.org/browse/FHIR-49525)

Reports that the spec is silent on the unit of the *result* of quantity
arithmetic (e.g., `1 'kg' + 100 'g'` could legitimately be `1.1 'kg'` or
`1100 'g'`) and asks that the spec either pick a rule or explicitly let
implementations choose, with `toQuantity` as the escape hatch for
selecting a specific output unit. FHIR-56190 closes this by codifying
the **left-operand-unit rule** for `+`/`-` and routing unit choice
through `toQuantity`. Resolution currently *Not Persuasive with
Modification*.

Reporter: Bas van den Heuvel.

### [FHIR-53249: Quantity equivalence is ambiguous](https://jira.hl7.org/browse/FHIR-53249)

Points out that Quantity Equivalence does not actually say *how*
equivalence is computed (only how units differ), and that the existing
"comparison will be made using the most granular unit of either input"
contradicts the decimal-equivalence "least precise operand" rule —
converting to the more granular unit only to round back to the less
precise one is incoherent. FHIR-56190 fixes this by removing
"most-granular-unit" wording and re-defining quantity equivalence as
*convert to a commensurable unit, then apply decimal equivalence*.
Resolution *Persuasive*.

Reporter: JingTang.

### [FHIR-53250: Calendar duration equivalence has conflicts](https://jira.hl7.org/browse/FHIR-53250)

Demonstrates a transitivity conflict between
`1 year ~ 1 'a' ~ 365.25 'd'` (the UCUM definition) and
`1 year ~ 365 days ~ 365 'd'` (the calendar definition); proposes
removing the sentence binding implementations to the UCUM/calendar
duration definitions inside equivalence, and limiting the calendar
duration conversion factors to *explicit* `toQuantity` use. FHIR-56190
implements this — calendar-unit factors live only inside `toQuantity`
and a dedicated Time-values sub-section, and equivalence between
year/month time-valued quantities across the two code systems is
explicitly allowed under `~` while equality / comparison require
explicit conversion. Resolution *Persuasive with Modification*.

Reporter: JingTang. Bryn Rhodes is referenced in the discussion.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the FHIRPath spec edits exactly as drafted in
`https://build.fhir.org/ig/HL7/FHIRPath/branches/BP-2026-03-quantity-preview/`
and merge them via PR against `HL7/FHIRPath`. Specifically:

- **Quantity Literal**, **Time-valued Quantities**, **Conversion**,
  **`toQuantity`**, **`convertsToQuantity`** sections rewritten as in
  the description: introduce a single canonical "Unit conversions"
  section with a "Time-values unit conversions" sub-section containing
  the existing calendar-duration table augmented with UCUM conversion
  factors and the "shortest path" rule for `'a'`/`'mo'` across the two
  code systems; add implicit `long → Quantity`; specify that
  `decimal`/`integer`/`long → Quantity` produces unit `'1'`; add the
  `45.toQuantity('g')` overload; return empty for non-convertible
  inputs.
- **Quantity Equality (`=`)**, **Quantity Equivalence (`~`)**, **Comparison**,
  **`comparable()`**: require UCUM-commensurable units; remove
  "uses the most granular unit"; route all conversions through the new
  Unit-conversions section; under `~` allow cross-code-system
  comparison of time-valued quantities (resolves FHIR-53250); under `=`
  and comparison require explicit `toQuantity` for cross-system
  year/month operands.
- **Math**: codify the **left-operand-unit rule** for `+`/`-` (resolves
  FHIR-49525); add `long` support and empty-result rules for
  calendar-quantity `*`/`/`; specify that division of mixed numeric and
  Quantity arguments implicitly elevates the numeric to a unit-`'1'`
  Quantity; spell out that systems without UCUM must return empty when
  they cannot convert.
- **Date/Time arithmetic**: collapse the bullet list + table into one
  table covering all units and input datatypes, fold in the `'a'`/`'mo'`
  error rule, codify partial-date conversion-to-precision, decimal-portion
  handling with warnings, leap-second omission, and Time-datatype cyclic
  wrap-around.
- Disposition the three rolled-up tickets — FHIR-49525, FHIR-53249, and
  FHIR-53250 — as **redundant**, pointing each at FHIR-56190's
  resolution.

#### Justification

The proposal directly resolves three independently-filed defects whose
existing dispositions (Bryn's commentary on FHIR-49525/53249/53250) are
already converging on the same answers, and consolidates them into a
single coherent narrative. The "left-operand unit" rule for arithmetic
matches the **observed behaviour of every existing engine** — the
Zulip-thread implementer report from Brian Kaney (`rh-fhirpath`,
msg 583499734) explicitly confirms this — so the change is a
ratification of de-facto behaviour, not a breaking change. Removing
"most granular unit" eliminates the demonstrated incoherence in
FHIR-53249. Restricting calendar-duration factors to explicit
`toQuantity` (and a documented Time-values sub-section) resolves the
transitivity conflict in FHIR-53250 without breaking UCUM compliance.
Brian has already produced the rendered preview, lowering review cost
substantially. Resolution is *Persuasive* and the ticket is queued
"Fit-For-Teleconference"; the workgroup signal so far is favourable.

---

### Disposition B: Alternative Approach — Accept In Principle, Land Incrementally

#### Proposal

Approve the *substance* of the changes but split the PR into a small
number of reviewable, mergeable units, dispositioned and merged in
sequence rather than as a single change:

1. **Foundational text move** — introduce the new "Unit conversions"
   section and "Time-values unit conversions" sub-section; relocate the
   calendar-duration table; add UCUM factors. (No semantic change.)
2. **`toQuantity` / `convertsToQuantity` enhancements** — add the
   `n.toQuantity('unit')` form, the `long → Quantity` implicit, the
   default-unit `'1'` rule, and updated examples.
3. **Equality / Equivalence / Comparison / `comparable()` semantics** —
   the actual normative changes (left-operand unit, removal of "most
   granular unit", cross-system `~` allowance, commensurability
   requirement). This is the disposition vehicle for FHIR-49525,
   FHIR-53249, FHIR-53250.
4. **Math operators + Date/Time arithmetic table merge** — the
   operator-by-operator clean-up.

Each step is its own PR against `HL7/FHIRPath`, and FHIR-49525 /
FHIR-53249 / FHIR-53250 are dispositioned against step 3 specifically
rather than against FHIR-56190 wholesale.

#### Justification

The change set is unusually wide and crosses the boundary between
purely-editorial refactoring (sections 1, 2, parts of 4) and *normative*
changes to operator semantics (`=`/`~`/`<`, left-operand unit rule,
calendar-unit equivalence). A single mega-PR makes it hard for engine
authors to attribute behavioural deltas to a specific decision, and it
makes it harder for the workgroup to walk back any one piece if a
defect is found post-ballot. Splitting also lets the editorial moves
land before the ITS call closes the normative discussion, reducing the
diff the workgroup needs to read in flight. Disposition B accepts
everything in Disposition A — there is no semantic disagreement — only
the *delivery* differs. The downside is overhead and a longer overall
landing time, which matters because FHIR-56190 is the headline ITS-call
agenda item for closing the quantity discussion.

### Disposition C: Decline (Defer / Persuasive-with-Mod for Scope)

#### Proposal

Decline the ticket *as filed* on the grounds that it bundles too many
distinct normative decisions into one disposition vehicle, and instead
require Brian to refile the genuinely contentious normative pieces as
individual tickets that explicitly track the open implementer questions
called out on Zulip — in particular:

- the **`'wk'` vs. `week` equivalence** behaviour (still false in
  `rh-fhirpath` per msg 583499734);
- whether `~` and `=` should remain distinguishable in normative text
  given that several engines collapse them today;
- whether `Quantity * Quantity` (full unit algebra) is in scope for
  FHIRPath at all, or remains implementation-defined.

The **purely editorial** refactors (Quantity Literal cross-links,
table caption move, removing duplicated narrative, merging the
Date/Time arithmetic table) are accepted as a separate, low-risk
editorial pass.

#### Justification

The existing dispositions on FHIR-49525 ("Not Persuasive with
Modification"), FHIR-53249 ("Persuasive"), and FHIR-53250 ("Persuasive
with Modification") show that the workgroup has *already* settled most
of the core questions; bundling them under FHIR-56190 risks reopening
those decisions and potentially weakening their resolution narrative
(workgroup minutes, ballot history) by burying them inside a
"clarification" ticket. The Zulip implementer report exposes at least
two areas where engines genuinely diverge today (`'wk'`/`week`, `~` vs.
`=`); resolving those deserves dedicated tickets with their own
discussion record rather than appearing as one bullet in a 50-bullet
list. Declining is defensible if the workgroup considers traceability
of normative changes more important than landing the cleanup in one
go.

### Recommendation

**Recommended disposition: A — Accept As Requested.**

The ticket is large but is overwhelmingly editorial: the genuinely
normative decisions inside it (left-operand unit for arithmetic,
removal of "most granular unit", restricting calendar-duration factors
to explicit `toQuantity`, allowing cross-system `~` for time-valued
quantities) all align with implementer expectations and with the
existing dispositions on FHIR-49525, FHIR-53249, and FHIR-53250. The
Zulip thread (`Quanity equivalence unit granularity`) shows
convergence rather than disagreement; the only divergent implementer
behaviour reported is in directions the proposal already corrects
(`'wk'` ↔ `week` and the lack of `~`/`=` distinction in some
engines), so adopting FHIR-56190 gives those engines the unambiguous
target Brian Kaney explicitly asked for. The preview branch is already
written, so review cost is bounded and the workgroup can review the
rendered text rather than chasing a paragraph diff.

Disposition B (incremental landing) is a reasonable fall-back if any
ITS member cannot review the consolidated diff in one sitting, and it
preserves the same outcome; recommend it only if the call surfaces
material concern with bundling. Disposition C is not recommended —
the underlying tickets it would refile to are *already* dispositioned,
and re-litigating them would delay the FHIRPath quantity story without
producing different answers.
