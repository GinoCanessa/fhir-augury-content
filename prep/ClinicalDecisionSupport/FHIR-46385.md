## Summary

The ticket reports a gap in the CRMI `$clone` operation: a cloned canonical
resource must be assigned a new `url` (canonicals must be unique), but the
operation as currently defined provides no input parameter for the caller to
supply that new `url`. The reporter asks that an input parameter be added so
the caller can specify the canonical URL of the newly cloned artifact. A
follow-up comment expands the scope: because `$clone` also clones any
composed/child resources, each of those cloned children also needs a way to
receive a new `url`. The ticket has been triaged with a **Persuasive**
resolution that agrees with the request and proposes a concrete shape: a
mandatory `url` parameter plus a repeating `childUrl` parameter with
`current`/`new` sub-parts to remap each child artifact's canonical. No
cross-referenced Jira tickets, Zulip threads, GitHub items, or Confluence
pages were surfaced for this ticket.

## Details

**Description:**

> The new cloned resource created by the $clone operation needs to have a
> url element that has a different value than the resource being cloned,
> but the operation as currently defined does not provide an input
> parameter to provide the new url.  Add an input parameter to add the
> url for the new resource created by the $clone operation.

**Resolution Description (already attached to the ticket):**

> Agreed, add a url parameter to the clone operation:
>
> - `url` 1..1 `canonical` — The url that will be used for the newly cloned artifact resource.
> - `childUrl` 0..*
>   - `childUrl.current` 1..1 `canonical` — The url of the current child artifact.
>   - `childUrl.new` 1..1 `canonical` — The url that will be used for the newly cloned child artifact resource.

**Comments:**

- **mitrep9g (2024-07-16):** "Since the operation also clones all resources
  it is composed of, all of these other resources also need new urls passed
  in to the operation, or some way to handle the new urls." — This comment
  is what motivated the `childUrl` part-tuple in the resolution; without
  it, callers would have no way to control the canonical URLs of child
  artifacts that `$clone` creates as a side effect.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add input parameters to the CRMI `$clone` `OperationDefinition` exactly as
spelled out in the ticket's existing resolution text:

- `url` — `canonical`, cardinality `1..1`, **in** parameter. The canonical
  URL to assign to the top-level cloned artifact.
- `childUrl` — complex, cardinality `0..*`, **in** parameter, with parts:
  - `childUrl.current` — `canonical`, `1..1`. The canonical URL of the
    existing child artifact being cloned.
  - `childUrl.new` — `canonical`, `1..1`. The canonical URL to assign to
    the newly cloned child artifact.

Update the operation's narrative to (a) state that `url` is required and
must differ from the source artifact's `url`; (b) describe `childUrl` as
the mechanism for remapping each composed child artifact's canonical;
(c) define server behavior when a `childUrl.current` does not match any
artifact `$clone` would otherwise create (error vs. ignore); and (d)
clarify behavior when no `childUrl` entry is supplied for a child that is
being cloned (error, because every clone needs a unique canonical, vs.
server-assigned default).

#### Justification

This is exactly the change the ticket asks for and the change the work
group has already marked **Persuasive** with concrete parameter
signatures, so adopting it minimizes back-and-forth and locks in the
shape implementers have already been told to expect. Canonical URLs must
be globally unique to be addressable, so any `$clone` that materializes a
new canonical resource without letting the caller choose its `url` is
either unusable in practice or forces servers to invent identifiers the
caller cannot predict — both poor outcomes. The repeating `childUrl`
structure with `current`/`new` parts mirrors the established pattern used
by other CRMI/FHIR remapping inputs (a list of `(from, to)` tuples) and
directly addresses the second-level concern raised in the comment thread,
so accepting as written closes the issue completely rather than in two
rounds.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same intent (let the caller control the new canonical URLs) but
restructure the inputs so that the top-level artifact is just the
zero-index entry of a single repeating remap parameter, e.g.:

- `urlMap` — complex, `1..*`, **in** parameter, with parts:
  - `urlMap.current` — `canonical`, `0..1`. The current canonical URL to
    remap. When omitted on the first entry, that entry is interpreted as
    the top-level artifact being cloned.
  - `urlMap.new` — `canonical`, `1..1`. The canonical URL to assign to
    that artifact's clone.

Equivalently, keep `url` for the top-level artifact (as in Disposition A)
but rename `childUrl` to something more general such as
`componentUrl` / `referencedUrl` / `dependentUrl` to reflect that
"composed children" in CRMI terms are not always literal children in a
containment sense — they are referenced canonical artifacts that
`$clone` is also materializing.

Either variant would also tighten the spec by defining: (a) failure
behavior when `current` does not match any artifact actually being
cloned, (b) failure behavior when a clone target has no entry supplied,
and (c) whether `new` URLs are required to be distinct from each other
and from the originals.

#### Justification

The trade-off vs. Disposition A is uniformity of mechanism: a single
repeating `urlMap` (or a renamed `componentUrl`) treats every cloned
artifact the same way and avoids the structural asymmetry of having one
parameter for the root and a separate, similarly-shaped parameter for
its dependents. That symmetry makes client code simpler (one loop, one
schema) and reduces the chance that implementers handle the root and the
children inconsistently. The naming alternative (`componentUrl` rather
than `childUrl`) addresses a real ambiguity: in CRMI, the artifacts that
get cloned alongside the root are usually canonical *dependencies*, not
contained children, and the "child" wording risks confusion with FHIR's
contained-resource semantics. The cost is divergence from the resolution
text already attached to the ticket and a small amount of additional
discussion to settle the naming, so this option is only attractive if the
work group is willing to revisit the parameter shape before ballot.

---

### Disposition C: Decline

#### Proposal

Decline the request to add new input parameters and instead document that
callers who need to control the cloned artifact's canonical URL should
either (a) call `$clone` and then immediately `update` the resulting
resource(s) to set the desired `url`, or (b) use existing
authoring/repository workflows (create a new artifact and copy contents)
rather than `$clone`.

#### Justification

The defensible argument for declining is "already addressable by existing
mechanisms" — a server that returns the cloned resources lets the client
PUT them back with the desired `url` values, so no new operation surface
is strictly required. This option is only mentioned for completeness; it
is materially weaker than A or B because (1) it forces a multi-step
client workflow for what is conceptually a single operation, (2) it
creates a window in which the cloned canonical exists with the *source*
artifact's `url` (which violates canonical-URL uniqueness in any
publishing context that observes the intermediate state), and (3) the
work group has already signaled Persuasive on this ticket, so declining
would contradict the existing disposition without new evidence.

---

### Recommendation

**Recommended disposition:** A

Accept the change exactly as the existing Persuasive resolution describes
it. The work group has already converged on the parameter names and
cardinalities (`url` 1..1 plus a repeating `childUrl` with
`current`/`new` parts), the comment thread's concern about composed
resources is addressed by the `childUrl` structure, and the ticket
carries the `Block-Vote-3` label — a strong signal that the workgroup
intends this to move through as a routine, low-controversy change.
Disposition B's structural cleanup has merit and is worth tabling as a
follow-up if implementer feedback during ballot finds the
root-vs-children split awkward, but reopening the parameter shape now
would delay a change the group has already agreed is needed and risks
losing the block-vote slot. Disposition C is not viable: the
multi-step "clone then PUT" workaround reintroduces canonical-uniqueness
problems the new parameters are specifically designed to prevent. The
one item that should *not* be left implicit even under Disposition A is
server behavior when `childUrl` entries are missing, extra, or
conflicting; the workgroup should adopt error-on-mismatch wording
alongside the parameter additions so implementers do not diverge.
