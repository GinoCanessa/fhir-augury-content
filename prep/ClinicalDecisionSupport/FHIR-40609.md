## Summary

The reporter (Bryn Rhodes) requests that the FHIR Clinical Guidelines
(CPG-on-FHIR) specification add an activity definition for documenting a
clinical assessment using the `ClinicalImpression` resource. The motivating
workflow is conditional/contextual guidance — the example given is an
immunization recommendation phrased as "immunize if there is a compelling
reason," where the recommended activity is to *capture and document* the
compelling reason as a structured clinical impression so that downstream
guidance and audit can act on it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new `ActivityDefinition` to the Clinical Guidelines (FHIR) IG that
represents "Document a clinical impression," targeting the
`ClinicalImpression` resource (`kind = ClinicalImpression`). Provide a
canonical URL under the CPG IG's standard `ActivityDefinition` namespace,
populate `code` with a SNOMED/LOINC concept for "clinical assessment
documentation" (or a CPG-defined code if no suitable external one
exists), and document the recommended bindings for
`ClinicalImpression.status`, `ClinicalImpression.subject`,
`ClinicalImpression.encounter`, and a CPG-specific extension (or use of
`ClinicalImpression.note` / `summary`) that captures the *rationale* /
"compelling reason" text. Wire the new activity into the IG's
"Recommendation → Action" guidance pages so that authors can reference it
from a `PlanDefinition.action` when the recommendation requires recording
a justification (e.g., the immunization "compelling reason" example in
the ticket). Add a worked example using the immunization scenario.

#### Justification

This is the lowest-friction interpretation of what Bryn asked for and it
fills a real gap: CPG-on-FHIR already defines activities that produce
`MedicationRequest`, `ServiceRequest`, `Task`, `CommunicationRequest`,
etc., but does not define a canonical activity that *records an
assessment*. `ClinicalImpression` is the FHIR-blessed resource for
"clinician's record of evaluation" and is therefore the natural target.
Providing a named, reusable `ActivityDefinition` improves consistency
across guidelines that need to capture rationale, supports the
`Block-Vote-1 / Ready-For-Vote` disposition the WG has already staged,
and aligns with the "more input is needed" disposition that Lynn flagged
for post-STU2 follow-up.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of introducing a new `ActivityDefinition` for
`ClinicalImpression`, model the underlying need as an extension on the
existing recommendation/action machinery so the rationale travels *with*
the request rather than as a separate clinical impression. Concretely:
(1) define a CPG extension — e.g., `cpg-recommendation-rationale` — on
`PlanDefinition.action` and on the produced request resource (e.g.,
`ImmunizationRecommendation`, `MedicationRequest`) that carries the
"compelling reason" as a coded + textual reason, optionally referencing a
`DocumentReference` or `Observation`; and (2) provide IG guidance that
when a clinician *does* want a free-form structured assessment, they
should use the existing FHIR `ClinicalImpression` resource directly
without a CPG-defined `ActivityDefinition` wrapper. Document the
immunization scenario as the worked example.

#### Justification

The "compelling reason" that motivated the ticket is fundamentally a
*rationale on a recommendation*, not a standalone clinical evaluation.
Coupling the rationale to the action that triggered it keeps the
provenance tight, avoids creating a second resource (and a second write)
for what is effectively a justification field, and dodges some of the
known interoperability friction with `ClinicalImpression` (sparse
deployment, overlap with `Observation` / `Condition` for assessment
findings, status-machine ambiguity). It also keeps the CPG IG from
appearing to *promote* `ClinicalImpression` as the canonical assessment
resource — a position the broader FHIR community has not consistently
endorsed.

---

### Disposition C: Decline

#### Proposal

Decline the request as out-of-scope for the CPG IG. Authors who need to
recommend documenting a clinical impression can already construct an
ad-hoc `ActivityDefinition` with `kind = ClinicalImpression` in their own
implementation guide; CPG does not need to bless a canonical one.
Recommend the reporter close as "Considered — Persuasive with Mod" by
adding only a non-normative example to the IG (no new artifact).

#### Justification

`ClinicalImpression` is a low-maturity, infrequently-implemented resource,
and CPG has historically preferred to remain neutral about which
clinical-record resource implementers should use to capture assessments.
Defining a canonical CPG `ActivityDefinition` for it could be read as a
stronger endorsement than the WG actually wants to make, and would add
maintenance load (examples, tests, narrative, downstream IGs) for a
single use case that is already expressible by composing existing
`PlanDefinition` / `ActivityDefinition` machinery. This was effectively
the Committee's first instinct — the original 2024-07-10 vote disposed of
the ticket as "Considered for Future Use" 20-0-0 — and the
"more input is needed" reversal does not by itself create a mandate to
ship the artifact.

---

### Recommendation

**Recommended disposition:** A

The ticket is small, narrowly-scoped, and aligned with how the rest of
the Clinical Guidelines IG is structured: every other recommended
clinical action that CPG wants implementers to standardize on is
expressed as a named `ActivityDefinition`, and "document the rationale
for a conditional recommendation" is a recurring need that today every
guideline author has to invent locally. Disposition B is intellectually
appealing but pushes the work into an extension whose semantics will
overlap with the existing `reasonCode` / `reasonReference` fields on
request resources, and it does not give guideline authors the reusable
hand-off point they actually wanted. Disposition C ignores the explicit
"re-open post STU2 publication" instruction Lynn captured on
2024-11-26, which signals the WG already moved off the original "Future
Use" outcome. Recommend the WG accept (A), with the explicit caveat that
the new `ActivityDefinition` should remain non-normative for STU3-era
publication and that the `ClinicalImpression` element bindings be
reviewed against Patient Care WG conventions before lock-down to avoid
future divergence.
