## Summary

The reporter responds to a question posed by the CDS Hooks specification (or
its supporting survey/discussion) about whether IG-defined extensions that let
the CDS Client dynamically alter the behavior of a CDS Service at runtime —
exemplified by the Da Vinci Coverage Requirements Discovery (CRD)
[`configuration-options-extension`](https://hl7.org/fhir/us/davinci-crd/STU2.1/deviations.html#configuration-options-extension)
— are useful, versus achieving the same flexibility through design-time
configuration (e.g. publishing distinct services per workflow). The reporter's
position is that runtime configurability **is** useful and that, rather than
leaving each downstream IG to invent its own mechanism, CDS Hooks core should
define an optional, standardized way for a client to declare configuration
choices on a per-invocation basis.

No related Jira tickets, GitHub items, or formal cross-references were
returned by the Augury index for this ticket. The closest community context
is a long Da Vinci Burden Reduction Zulip thread (CRD Cards) where Lloyd
McKenzie explains how the CRD `configuration-options` extension is meant to
work, confirms that CRD servers SHALL offer configuration options for each
card type they support, and that the extensible code list lets payers (and
clients) adjust what comes back at runtime — illustrating both the demand
for, and the IG-specific shape of, the mechanism the ticket wants
standardized.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an optional, normatively-defined configuration mechanism to CDS Hooks
core, modeled directly on Da Vinci CRD's `configuration-options-extension`,
with two coordinated changes:

1. In the **CDS Service Discovery Response** (each entry of `services[]`),
   add an optional `configurationOptions` element. Each entry has:
   `code` (required, string; extensible value set), `type` (required;
   `boolean | integer | decimal | string | code`), `name` (required,
   string), `description` (required, string), `default` (required, typed
   per `type`), and an optional `valueSet` for `code`-typed options.
2. In the **CDS Hooks Request**, add an optional
   `configuration` object whose keys are `code` values declared by the
   service in Discovery and whose values match the declared `type`. CDS
   Services SHALL ignore unknown codes and SHALL apply declared `default`s
   when an option is not supplied.

Mark the feature OPTIONAL for both client and service, but require that a
service which advertises options SHALL honor any client-supplied value that
matches a declared option, and a client MAY supply only declared options.

#### Justification

This is exactly what the reporter asks for and exactly what the existing CRD
mechanism does, just promoted into core so downstream IGs (CRD, DTR, PAS,
clinical IGs) reuse one shape instead of inventing their own. The Zulip
thread confirms the mechanism is workable and well-understood by at least
one major IG community, and Lloyd McKenzie explicitly notes the core
extensibility points (Discovery Response object, request) that this
proposal would simply lift up. Standardizing here reduces implementer
burden across the CDS Hooks ecosystem and gives the spec a single, testable
configurability story — the explicit harm the ticket cites ("a multitude
of different alternative mechanisms") is best prevented by a first-class
element rather than yet another extension.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a new first-class element to the Discovery Response and
Request, formally **define and register a single CDS Hooks extension URL**
(e.g. `http://hl7.org/fhir/cdshooks/StructureDefinition/configuration-options`
on the service entry, and a peer extension on the request) that captures
exactly the same shape proposed in Disposition A (`code`, `type`, `name`,
`description`, `default`, optional `valueSet`). Document it in the CDS
Hooks specification as the **recommended** mechanism for runtime
configurability and explicitly request that downstream IGs (starting with
Da Vinci CRD) deprecate their bespoke versions in favor of this one.

In parallel, add non-normative guidance distinguishing when implementers
should use **runtime configuration** (cross-cutting toggles like
`max-cards`, card-type filters, debug verbosity) versus **design-time
configuration** (workflow-specific services, separate hook IDs,
authorization scopes).

#### Justification

CDS Hooks is intentionally a small, stable surface; adding a new
top-level element to `services[]` and to the request body is a wider blast
radius than is strictly necessary to prevent fragmentation. A registered,
referenced extension achieves the *real* goal of the ticket — one canonical
shape that all IGs can adopt — without breaking any existing client or
server that already conforms to today's Discovery/request schemas, and
without forcing a deeper version bump. It also matches how CDS Hooks has
historically governed cross-IG features: extension at `services[].extension`
is already an explicitly supported extension point (the very point CRD uses
today). The Zulip thread shows that CRD's mechanism is understood and
implemented; rebadging it as a core-blessed extension lets that
implementation experience carry forward verbatim, while the Disposition A
shape would force every existing CRD server and client to migrate to a
second, parallel mechanism.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by an existing mechanism**.
Note in the resolution that CDS Hooks already supports IG-defined
extensions on the Discovery Response service entry and on the request, and
that Da Vinci CRD's `configuration-options-extension` is the canonical
worked example. Recommend that IGs needing runtime configurability adopt
the CRD pattern (cite the URL) rather than inventing their own, and close
the ticket as Persuasive with no spec change.

#### Justification

CDS Hooks is deliberately minimal; promoting an IG-specific feature into
core risks scope creep and competing patterns (boolean toggles vs. coded
parameters vs. profile-driven services) that the workgroup hasn't yet
canvassed broadly. The ticket itself is a single-voice request without
demonstrated cross-IG demand in the Augury index — no cross-referenced
Jira tickets, no GitHub PRs/issues, and only one tangential Zulip thread.
Declining keeps the core spec stable and pushes the work to a community
proof-point (multi-IG adoption) before standardization, while still
giving Bas a clear, citable mechanism (the existing extension) to use
today.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the substantive intent of the ticket — one
canonical, spec-blessed shape so downstream IGs stop reinventing the
wheel — while honoring CDS Hooks' historical preference for a small,
stable core and avoiding a breaking restructure of `services[]` and the
request body. It also has the lowest migration cost: Da Vinci CRD already
implements the exact shape, the Zulip thread shows the mechanism is
already understood by implementers, and re-publishing it under an
HL7-owned, CDS-Hooks-namespaced extension URL turns existing CRD
implementations into reference implementations rather than legacy ones.
Disposition A is the right destination if/when the mechanism graduates
after broader uptake; Disposition C understates the real cost of the
status quo, which is precisely the fragmentation the reporter is trying
to prevent.
