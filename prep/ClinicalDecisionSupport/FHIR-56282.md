## Summary

This ticket asks how a DTR client should discover the endpoint for the
`$log-questionnaire-errors` operation. HRex already provides a discovery
mechanism for `$questionnaire-package`, and the per-Questionnaire endpoints for
`$next-question` and `$expand` are conveyed on the Questionnaire resource
itself. No equivalent convention has been defined for
`$log-questionnaire-errors`, so implementers are left guessing where to send
CQL/error reports back to the payer. Lloyd enumerates three options in the
description: (1) make the endpoint implicit/relative to the
`$questionnaire-package` operation that delivered the form, (2) extend HRex
endpoint discovery to include `$log-questionnaire-errors`, or (3) carry the
endpoint as a per-Questionnaire parameter (e.g., a new extension on
Questionnaire) so each form can route its errors to the responsible
organization. The trade-off is convenience (#1) and centralization (#2)
versus per-form routing accuracy when questionnaires originate from multiple
organizations (#3). Closely related published tickets give context:
[FHIR-50013](https://jira.hl7.org/browse/FHIR-50013) tightens the requirement
that clients SHALL report CQL errors via `$log-questionnaire-errors` (making
discovery non-optional in practice);
[FHIR-55875](https://jira.hl7.org/browse/FHIR-55875) and
[FHIR-55876](https://jira.hl7.org/browse/FHIR-55876) wire the operation into
the DTR capability statements at the Questionnaire-resource level; and
[FHIR-56281](https://jira.hl7.org/browse/FHIR-56281) (sibling Lloyd ticket,
already Applied) corrects the operation's input parameters so that the
operation is now properly described as taking a Questionnaire canonical and
an OperationOutcome.

## Details

**Description:**

HRex allows discovery of the $q-package operation, and the $next-question and
$expand endpoints are defined by the Questionnaire. However, we haven't
defined any convention or discovery mechanism for the
$log-questionnaire-errors endpoint. There are three options:

1. **Make it relative to the $q-package operation.** Pro: we don't need to
   change the data passed anywhere, and it's simple. It's also most likely
   what most implementers have presumed. Con: The questionnaires might come
   from multiple organizations which creates an issue on the payer side of
   getting the errors to the right organization.
2. **Add it to HRex endpoint discovery.** Pro: Allows error handling to be
   managed. If implementers were using approach #1, this still works as they
   can just specify that same endpoint here. Con: Still requires routing
   errors to the organization responsible, and means we need to update the
   list of endpoints.
3. **Pass it as a parameter in the Questionnaire.** Pro: allows different
   endpoints per form, ensuring that errors are reported to the appropriate
   place based on who's responsible for the form. Con: Requires updating the
   DTR spec to define a new extension which prior implementations would need
   to pre-adopt.

**Comments:**

No comments recorded on this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt **option #2** from the description: extend HRex endpoint discovery to
include `$log-questionnaire-errors`. Concretely, add
`log-questionnaire-errors` (or a comparably named entry) to the
`hrex-endpoint` extension's set of endpoint types (or the equivalent
HRex `Endpoint`/discovery mechanism that already conveys the
`questionnaire-package` endpoint), and update the DTR Formal Specification
§9.10.x and the affected capability statements
(`dtr-payer-service`, `full-dtr-ehr-311`, `full-dtr-ehr-700`,
`smart-dtr-client`) to state that clients SHALL discover the endpoint via
HRex. Define a fallback rule: if HRex does not advertise an explicit
`$log-questionnaire-errors` endpoint, the client SHALL invoke it on the
same base URL it used for `$questionnaire-package` (i.e., option #1 as a
default), preserving compatibility with existing implementations. Note
that this disposition crosses two repos: changes in `HL7/davinci-dtr`
(spec narrative + capability statements) and in `HL7/davinci-ehrx` (HRex
endpoint type registration).

#### Justification

This is the option Lloyd describes as "preserving" any implementation
that has been doing option #1: those implementations can simply advertise
the same URL via HRex and be conformant. It also addresses the central
con of option #1 — that questionnaires can come from multiple
organizations — because each payer/intermediary advertises the endpoint
that errors should be routed to for the forms it serves. Anchoring
discovery in HRex is consistent with how DTR already discovers
`$questionnaire-package`, so it doesn't introduce a new discovery
paradigm. FHIR-50013 makes invocation SHALL, FHIR-55875/55876 already
declare support in the capability statements; deterministic discovery is
the missing piece.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt **option #3**: define a Questionnaire-level extension (e.g.,
`http://hl7.org/fhir/us/davinci-dtr/StructureDefinition/log-questionnaire-errors-endpoint`)
of type `url` (or `Reference(Endpoint)`) that conveys the per-form
`$log-questionnaire-errors` endpoint. The extension would be authored by
whichever organization produced the Questionnaire (which is also the
organization best positioned to receive the error reports about it).
Update the DTR Formal Specification to require clients to (a) use the
extension when present, and (b) fall back to the `$questionnaire-package`
base URL when it is not — providing a clean migration path. The spec
change is in `HL7/davinci-dtr` only (new extension StructureDefinition
under `input/fsh/`, narrative update under `input/pagecontent/`, plus
capability statement updates noting the extension).

#### Justification

Option #3 is the only option that solves the multi-organization routing
problem cleanly: when a payer aggregates questionnaires from multiple
specialty bodies or downstream organizations, errors flow back to whoever
authored the form, without requiring a separate HRex endpoint per
authoring organization. Trade-offs vs. Disposition A: it requires an
extension, and existing implementations would need to pre-adopt it (the
con Lloyd called out); however, the spec is still in early adoption for
the operation (FHIR-56281 just corrected the parameters as recently as
2026-04-06), so the cost of adding an extension now is much lower than it
will be later. The fallback rule preserves option #1 behavior for
implementers who haven't adopted yet.

---

### Disposition C: Decline

#### Proposal

Adopt **option #1** as a normative rule and explicitly reject the need
for a separate discovery mechanism. State in the DTR Formal Specification
that the `$log-questionnaire-errors` endpoint SHALL be invoked at the
same base URL the client used for `$questionnaire-package`, full stop.
No HRex changes, no Questionnaire extension. Update the capability
statements' narrative to reflect the rule.

#### Justification

This is what most implementers have presumed (per the description), it
requires zero schema changes, and it ships a deterministic answer
immediately. It is defensible on the grounds that DTR's primary error
recipient *is* the entity that packaged the Questionnaire — i.e., the
party that invoked `$questionnaire-package` is also the party most able
to act on errors found while filling it. The multi-organization concern
is real but arguably handled at the payer/aggregator layer (the entity
serving `$questionnaire-package` can fan errors out to the originating
organization internally) and does not need to be solved by the DTR wire
protocol.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the best balance of correctness, backward compatibility,
and consistency with how DTR already does discovery. HRex is the
established place where DTR clients look for payer-specific endpoints
(`$questionnaire-package` is already there), so adding
`$log-questionnaire-errors` to the same mechanism is the smallest
conceptual change for implementers. Crucially, A subsumes C: the spec
can still say "if HRex does not advertise the endpoint, default to the
`$questionnaire-package` base URL", which preserves every implementation
that quietly assumed option #1 — exactly the compatibility property
Lloyd called out as a pro for option #2. Disposition B (the
Questionnaire extension) genuinely solves the multi-organization routing
case better than A, but at the cost of a new extension that prior
implementations have to pre-adopt; given FHIR-50013 has already promoted
invocation to SHALL and the capability statements (FHIR-55875,
FHIR-55876) just landed, the workgroup should prefer the lighter-touch
discovery fix now and revisit per-form routing via extension only if
real-world deployments show the multi-organization case is common
enough to justify it. C is the cheapest answer but locks in a single
routing model and gives the workgroup nothing to point at when
implementers ask "what if the form came from somewhere else?".
