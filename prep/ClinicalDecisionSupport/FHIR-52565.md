## Summary

The reporter notes that, in the current ballot of the Clinical Quality
Language (CQL) specification, new features added in versions 1.5 and 2.0
are highlighted in blue text in Chapter 3, but it is not clear whether
that highlighting is a complete enumeration of the changes since the
previous balloted version. The request is to add — for this ballot and
for future versions — an explicit, complete list of changes from the
previous balloted specification, placed in the introduction either
in-line or via reference. The intent is to give ballot reviewers and
future implementers a single, authoritative changelog to consult rather
than having to scan the entire specification for blue-highlighted text.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items
were surfaced for this ticket, so the request stands on its own.

## Proposed Dispositions

Note: The cross-reference service surfaced no GitHub repositories for
this ticket, so no `repo-analysis` briefings are available to anchor
file-path-level proposals. The CQL specification is authored outside
the FHIR-core repos that FhirAugury currently indexes; dispositions
below name the documentation areas of CQL by their conventional
location in the spec rather than by repo path.

### Disposition A: Accept As Requested

#### Proposal

Add a new "Summary of Changes" section to the introduction (Chapter 1)
of the CQL 2.0 specification — in line with the existing structure used
by other HL7 specifications — that enumerates, at minimum:

- Each new language feature added in 1.5 and in 2.0 (with anchors to
  the formal definition in Chapter 3 and elsewhere).
- Each backward-incompatible change, called out separately.
- Each clarification or corrigendum that changes conformant behaviour.
- Each deprecation or removal.

The section should be authored as the canonical, ballot-time list. The
existing blue highlighting in Chapter 3 is retained but is explicitly
documented (in the new section's preamble) as a visual aid that points
back to entries in this list, not as the authoritative source. For
future versions, make this section a required deliverable of the
ballot package, populated incrementally as STU comments are resolved.

#### Justification

A consolidated changelog at the top of the specification is a long-
standing convention across HL7 publications (e.g., the FHIR
"Changes since the last version" pages, IG history pages) and is one
of the first things ballot reviewers and implementers look for when
deciding what to re-read. Relying on inline visual highlighting to
communicate the scope of change forces every reviewer to scan the full
chapter to reconstruct the list, which is exactly the gap the reporter
identifies. The cost is editorial only; no normative content changes.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than maintaining a hand-authored "Summary of Changes" section,
generate the change list mechanically from the specification's source.
Concretely:

- Tag each change in the source with a structured marker (e.g., a
  `data-change-version="2.0"` attribute, or a small XML/AsciiDoc macro)
  at the point of change, replacing the ad-hoc "blue text" convention.
- Have the build produce a generated "Changes since CQL 1.4" appendix
  (or introduction section) that aggregates every tagged change, with
  deep links back to the section where the change is defined.
- Publish both the rendered appendix and a machine-readable
  `cql-changes-2.0.json` (or similar) alongside the spec for
  implementer tooling.

#### Justification

The reporter's underlying concern is *completeness* — they cannot tell
whether the blue highlighting is exhaustive. A generated list is
exhaustive by construction: if a change is highlighted, it is in the
list, and vice versa. This also pays dividends in future cycles: every
subsequent version gets a free, accurate "what changed" view without
relying on an editor to remember every edit. The trade-off is a one-
time tooling investment in the CQL build, which is heavier than
Disposition A and may not be feasible inside this ballot window.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for this ballot, on the grounds that (a) the
in-text blue highlighting in Chapter 3 already identifies new material,
(b) HL7's standard ballot package includes a Jira-derived change
report that lists every applied disposition since the previous ballot,
and (c) duplicating that information inside the spec risks the two
sources drifting out of sync. Direct reviewers to the existing ballot
change report for the authoritative change list.

#### Justification

There is a defensible argument that the canonical "what changed"
artifact for an HL7 ballot is the disposition log, not the
specification body, and that the specification should focus on the
normative content of the current version. Embedding a parallel
changelog creates a maintenance burden and a future drift risk. This
is the weakest of the three positions because it does not address the
reporter's actual experience — that they could not, while reading the
spec, tell whether they had seen all of the changes — but it is
defensible if the workgroup judges the editorial cost of A or the
tooling cost of B too high for this cycle.

---

### Recommendation

**Recommended disposition:** A

A hand-authored "Summary of Changes" section in the introduction is
the lowest-cost intervention that directly resolves the reporter's
complaint, and it aligns CQL with the documentation conventions used
by FHIR core and most HL7 IGs. Disposition B is strictly better in
the long run but is unlikely to land inside the current ballot
window, and adopting B without the interim section in A leaves the
ballot reviewers in exactly the situation the ticket describes.
Disposition C dismisses a real reviewer-experience problem and is
hard to defend given the ticket's "Highest" priority. The pragmatic
path is to accept A for this ballot, while logging a follow-up to
explore B for the next cycle so the changelog can become a generated
artifact rather than an editorial obligation.
