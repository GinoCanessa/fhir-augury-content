## Summary

Bryn Rhodes asks that the `artifactEndpointConfiguration` resolution
path in the Canonical Resource Management Infrastructure (CRMI)
specification be made aware of the `authoritativeSource` extension. As
written, endpoint configuration matching keys off the artifact's
URL/canonical; the request is for the resolver to prefer the
`authoritativeSource` extension when present, so that endpoint matching
reflects where the artifact is actually authoritative rather than only
its declared canonical. The ticket has been triaged as **Persuasive
with Modification**: the workgroup agreed in principle, but determined
that the correct extension to consult is `web-source` (which conveys
*where* an artifact is published/accessible when that location differs
from the canonical) rather than `authoritativeSource`. The agreed
modification has two parts: (1) add `web-source` as an optional element
to the Shareable pattern and to all Shareable profiles, and (2) update
the `artifactEndpointConfigurable` resolution logic so that
`web-source` and `trusted-expansion` extensions are consulted as a
final fallback step when no other resolved endpoint can successfully
retrieve the content or perform the requested operation. No
cross-referenced Jira tickets were found by the FHIR Augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CRMI specification's `artifactEndpointConfiguration`
resolution algorithm so that, when an artifact carries the
`authoritativeSource` extension, that extension's value is used as the
key for endpoint-configuration matching in place of the artifact's
declared URL/canonical. Document this as the first step of the
resolution algorithm. No changes are made to the Shareable pattern or
to any other extension; only the resolver's matching logic is amended.

#### Justification

This is the literal interpretation of the request and is a minimally
invasive change: it touches only the resolution algorithm narrative in
the CRMI IG and does not alter any structure definitions, profiles, or
extension definitions. It honors the reporter's intuition that
`authoritativeSource`, when present, expresses the artifact's intended
authoritative location and so should drive endpoint selection.

---

### Disposition B: Alternative Approach (matches the recorded resolution)

#### Proposal

Adopt the modification recorded on the ticket:

1. Add `web-source` as an optional element to the **Shareable pattern**
   and propagate it to all Shareable profiles defined by CRMI
   (Shareable, Computable, Publishable, Executable variants), so that
   any shareable artifact may carry a `web-source` value.
2. Update the `artifactEndpointConfigurable` resolution logic so that,
   after all other configured endpoints have been tried and have failed
   to retrieve the content or perform the requested operation, the
   resolver consults the artifact's `web-source` extension and the
   `trusted-expansion` extension as a **final fallback step**.
3. Leave `authoritativeSource` semantics unchanged — `web-source`
   (publication/access location) is the correct hook for endpoint
   resolution, while `authoritativeSource` continues to express
   provenance/authority.
4. Address the reporter's follow-up questions in the IG narrative:
   document explicitly that `web-source` is consulted for `$expand`
   (covering the trusted-expansion case) and that the fallback applies
   uniformly across the shareable/computable/publishable/executable
   capability tiers.

#### Justification

This is the disposition the workgroup has effectively already taken
("Persuasive with Modification"). It separates two distinct concepts
the ticket conflates — *who authored / is the source of truth for*
the artifact (`authoritativeSource`) versus *where the artifact is
published and can be fetched* (`web-source`) — and uses the
publication-location extension to drive endpoint resolution, which is
semantically correct. Making `web-source` an optional element of the
Shareable pattern (rather than only available via the extension on
specific resources) addresses Bryn's first follow-up question by
ensuring the mechanism is uniformly available to every artifact that
follows the pattern. Treating it as a *final* fallback rather than
the first step preserves existing deployments' configured endpoint
behavior and avoids regressing repositories that already resolve
correctly via canonical-based matching.

---

### Disposition C: Decline

#### Proposal

Decline to change the `artifactEndpointConfiguration` resolution
algorithm or the Shareable pattern. Document, instead, that
implementers who need provenance- or publication-aware endpoint
selection should configure their `artifactEndpointConfiguration`
entries explicitly using the canonical(s) they wish to route, and refer
them to the existing `authoritativeSource` and `web-source`
extensions as informational metadata only.

#### Justification

The current algorithm is deterministic and well-understood: endpoints
are matched against the artifact's canonical. Adding extension-driven
fallback paths increases resolver complexity and creates ambiguity when
multiple extensions disagree (e.g., `authoritativeSource` and
`web-source` point to different hosts). Implementers can already
achieve the requested behavior by configuring additional
`artifactEndpointConfiguration` entries keyed off the alternative URL.
Declining keeps the resolver narrow and predictable.

---

### Recommendation

**Recommended disposition:** B

Disposition B is what the workgroup has already converged on and what
is recorded in the ticket's resolution; the recommendation simply
formalizes it. It correctly distinguishes `authoritativeSource`
(provenance) from `web-source` (publication location) and uses the
right extension for the resolver hook, which avoids the semantic drift
that Disposition A would introduce. Promoting `web-source` into the
Shareable pattern answers Bryn's first follow-up question (uniform
availability across artifact types) at the cost of a small, additive
change to the pattern and its profiles — non-breaking for existing
instances because the element is optional. Treating the extension as a
*final* fallback (rather than the first step) preserves existing
endpoint-configuration behavior and means deployments that already
resolve successfully will see no change, while those that currently
fail to resolve gain a documented escape hatch. Disposition C is
defensible from a "keep the resolver simple" standpoint but ignores the
real implementer pain that motivated the ticket; Disposition A is
minimally invasive but pins endpoint resolution to the wrong extension
and would have to be revisited.
