## Summary

The reporter notes that the definition of `Library.version` in FHIR Core
references a "Decision Support Service specification" (DSS) without making
clear what that specification is — i.e., whether it means CDS Hooks, the
Clinical Reasoning module, or something else — and asks for clarification.
The accepted resolution (marked Persuasive) is to stop pointing at the
defunct DSS reference and instead point readers at the Canonical Resource
Management Infrastructure (CRMI) IG, which is now the canonical home for
versioning guidance on knowledge artifacts. The cross-reference index
shows only one related entity, the FHIR element `Library.version` itself
(the artifact whose definition is being corrected); there are no related
Jira tickets, Zulip threads, or GitHub items associated with this ticket.

## Details

**Description:**

> The definition of the Library.version element mentions a Decision Support
> Service specification? Is this CDS Hooks? Is the clinical reasoning
> module? Please clarify.
>
> "*There is also no expectation that versions can be placed in a
> lexicographical sequence. To provide a version consistent with the
> **Decision Support Service specification**,*"
>
> "

**Resolution Description (from Jira snapshot):**

> Agreed, update the language from Decision Support Service Specification
> (DSS) to reference CRMI, where this guidance is now provided:
>
> Update the sentence from:
>
> *To provide a version consistent with the Decision Support Service
> specification, use the format Major.Minor.Revision (e.g. 1.0.0). For
> more information on versioning knowledge artifacts, refer to the
> Decision Support Service specification. Note that a version is required
> for non-experimental active artifacts.*
>
> To:
>
> *For more information on versioning knowledge artifacts, refer to the
> Canonical Resource Management Infrastructure implementation guide
> versioning discussion (https://hl7.org/fhir/uv/crmi/artifact-lifecycle.html#artifact-versioning)*
>
> Apply this change to all the Knowledge Artifact version descriptions.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution exactly as written on the ticket. In the FHIR Core
definition for `Library.version`, replace the existing sentence:

> *To provide a version consistent with the Decision Support Service
> specification, use the format Major.Minor.Revision (e.g. 1.0.0). For
> more information on versioning knowledge artifacts, refer to the
> Decision Support Service specification. Note that a version is required
> for non-experimental active artifacts.*

with:

> *For more information on versioning knowledge artifacts, refer to the
> Canonical Resource Management Infrastructure implementation guide
> versioning discussion
> (https://hl7.org/fhir/uv/crmi/artifact-lifecycle.html#artifact-versioning).*

Then propagate the same edit to the `version` element definition on every
other Knowledge Artifact resource that carries the same boilerplate —
typically `PlanDefinition.version`, `ActivityDefinition.version`,
`Measure.version`, `Questionnaire.version`, `ValueSet.version`,
`CodeSystem.version`, `ConceptMap.version`, `GraphDefinition.version`,
`StructureDefinition.version`, `OperationDefinition.version`,
`SearchParameter.version`, `MessageDefinition.version`,
`CapabilityStatement.version`, `CompartmentDefinition.version`,
`NamingSystem.version`, `TerminologyCapabilities.version`,
`ImplementationGuide.version`, `ChargeItemDefinition.version`,
`EventDefinition.version`, `ExampleScenario.version`,
`ResearchDefinition.version`, `ResearchElementDefinition.version`,
`SubscriptionTopic.version`, etc. (the actual list is the set of
canonical/knowledge-artifact resources whose `version` element currently
carries the DSS sentence — verify against the source of truth before
editing).

#### Justification

The change is editorial and strictly clarifying. The current text
references the "Decision Support Service specification," which is not a
live HL7 deliverable that readers can navigate to; reporters and
implementers cannot tell whether DSS means CDS Hooks, the Clinical
Reasoning module, or something else, which is exactly the confusion that
generated this ticket. CRMI is the active HL7 IG that now owns guidance
on canonical/knowledge-artifact lifecycle, including versioning, so
redirecting readers to
`https://hl7.org/fhir/uv/crmi/artifact-lifecycle.html#artifact-versioning`
gives them a maintained, normative reference. The ticket is already
marked **Persuasive** by the work group, the request requires no schema
or behavior change, and the reporter and the work group converge on the
same fix.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same intent as Disposition A — drop the DSS reference and
point readers to CRMI — but make two adjustments:

1. **Keep the version-format guidance** that Disposition A drops. Rewrite
   the sentence as:

   > *Versions are not required to be lexicographically orderable. A
   > recommended format is `Major.Minor.Revision` (e.g. `1.0.0`); for
   > full guidance on versioning knowledge artifacts, see the Canonical
   > Resource Management Infrastructure implementation guide's
   > [artifact versioning discussion](https://hl7.org/fhir/uv/crmi/artifact-lifecycle.html#artifact-versioning).
   > A version is required for non-experimental active artifacts.*

2. **Centralize the boilerplate.** Rather than maintaining the same
   paragraph on N+ knowledge-artifact resources, factor the text into a
   single shared definition snippet (the same mechanism FHIR Core uses
   for other repeated element definitions) and reference it from each
   resource's `version` element. This avoids the historical drift that
   produced the DSS reference in the first place.

#### Justification

The ticket's literal resolution removes useful information that is
unrelated to the DSS confusion: the `Major.Minor.Revision` recommendation
and the "required for non-experimental active artifacts" reminder are
both still correct and helpful, and were not what the reporter complained
about. Preserving them keeps the element definition self-contained for
implementers who don't immediately click through to CRMI. Centralizing
the snippet also addresses the root cause — duplicated prose across many
resources is what allowed the stale DSS reference to persist undetected
for so long, and it ensures future updates land in one place.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — already addressed**, on the
grounds that CRMI is already the de facto reference for canonical
resource versioning and that the DSS phrasing in the existing element
definitions is widely understood by implementers as a vestigial pointer
to the now-superseded Decision Support Service work. Make no edits to
`Library.version` or any other knowledge-artifact `version` definition.

#### Justification

This option is included for completeness; it is hard to defend on the
merits. The reporter has demonstrated, by filing the ticket, that the DSS
reference is genuinely confusing to at least one implementer; "everyone
already knows" is not a viable position when the cited specification is
not navigable from the FHIR documentation. The only honest reason to
decline would be scope/cost (touching every knowledge-artifact resource),
but that cost is small and the work group has already judged the request
**Persuasive**.

---

### Recommendation

**Recommended disposition:** B

Disposition A is exactly what the work group already agreed to and is
defensible as-is, but the literal resolution drops two sentences
(`Major.Minor.Revision` recommendation and the
"required-for-non-experimental-active" rule) that have nothing to do
with the DSS confusion the reporter raised; that information is useful
on the element definition itself and shouldn't be lost just because the
neighboring sentence is being repaired. Disposition B preserves that
guidance, redirects readers to CRMI as agreed, and additionally proposes
factoring the shared paragraph into a single snippet so the next stale
cross-spec reference doesn't have to be hunted down on a dozen
resources. Disposition C is not recommended; the reporter's confusion
is real and the work group has already accepted the change.
