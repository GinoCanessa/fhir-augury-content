## Summary

Lynn Perrine is asking whether the Clinical Quality Language (CQL)
specification should remove its references and links to the **Atom**
editor. Atom (the GitHub-developed text editor that hosted the
`language-cql` authoring plugin) was sunsetted by GitHub and the
repository was archived in **December 2022**, so any CQL spec text or
hyperlinks that point readers to Atom as a recommended CQL authoring
environment are now pointing at an unmaintained tool. The ticket is a
short, narrowly-scoped editorial / tooling-guidance question with no
linked Jira tickets, no GitHub cross-references, and no comments to
date. No alternative editor (e.g., the VS Code CQL extension at
`cqframework/vscode-cql`) is named in the ticket itself, but historical
Zulip discussions in the `#cql` stream make clear that VS Code is the
de facto successor.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the Clinical Quality Language (FHIR) specification to **remove
all references and hyperlinks to the Atom editor and the Atom CQL
plugin**. Concretely:

- Locate the authoring-tools / editor sections in the CQL FHIR spec
  source (e.g., the "Authoring CQL" / "Tooling" pages in the
  `cqframework`-maintained CQL FHIR IG sources that publish to
  cql.hl7.org).
- Remove sentences and bullet points that list Atom as a recommended
  CQL authoring environment.
- Remove the outbound links to `https://atom.io/packages/language-cql`
  and to the `cqframework/atom_cql_support` repository (or replace
  them with a one-line historical note that the Atom plugin is
  archived because Atom itself was sunsetted by GitHub on 2022-12-15).
- Where Atom was previously offered as the primary worked example,
  re-point the example to the VS Code CQL extension
  (`cqframework/vscode-cql`).
- Update any screenshots or step-by-step instructions that show the
  Atom UI.

#### Justification

Atom and the `atom_cql_support` plugin are end-of-life. Pointing
implementers at an archived editor hurts new adopters of CQL: the
linked download pages are gone or broken, the plugin has not received
compatibility updates, and the Zulip `#cql` history (e.g., the *Editor
packages* and *CQL Plugins for VS Code (0.1.2) and Atom (2.11.0)
Published* threads) shows the maintainers have been steering users
toward VS Code for years. Removing the references is a small,
low-risk editorial change that aligns the spec with the current
tooling reality and with the maintainers' own behavior.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of deleting the Atom references outright, **rewrite the
authoring-tools section to be tool-agnostic** and add a clearly
labelled "Historical tooling" note. Concretely:

- Replace the Atom-centric guidance with a generic "CQL Authoring
  Tools" section that names the **VS Code CQL extension** as the
  currently maintained authoring environment, plus any other
  community-maintained editors / language servers that the CDS WG
  wants to acknowledge.
- Move the Atom mention into a short "Historical / archived tooling"
  paragraph that explains Atom was sunsetted by GitHub on
  2022-12-15, links to the archived `cqframework/atom_cql_support`
  repository for posterity, and warns that the plugin is no longer
  receiving updates.
- Add normative-style guidance that the spec does not endorse any
  single editor and that tools are listed informatively.

#### Justification

A flat removal (Disposition A) loses institutional memory: a number
of existing measure/CDS authoring guides, training materials, and
older Zulip threads still reference the Atom plugin, and readers who
follow those breadcrumbs into the CQL spec benefit from a brief
explanation of *why* Atom guidance is gone. Treating the
authoring-tools list as informative and tool-agnostic also future-
proofs the spec against the next editor sunset (e.g., if VS Code
support were ever to lapse) and avoids implying HL7 endorsement of a
particular vendor's IDE.

---

### Disposition C: Decline

#### Proposal

Decline the change as **out of scope for the CQL FHIR specification
itself**, on the grounds that authoring-tool recommendations are
informative and that maintenance of those informative pointers should
be handled in the tooling repos (e.g., `cqframework/atom_cql_support`
README and the CQF documentation site) rather than via a CQL spec
change request.

#### Justification

The CQL FHIR specification is normative on the language and its FHIR
binding; the editor recommendation is incidental. The `cqframework`
org has already archived the Atom plugin's repository in lockstep
with GitHub's Atom sunset, so users who follow the link land on an
archived repo with a clear sunset notice. Spending CDS WG ballot
cycles on an editorial pointer change carries opportunity cost
against more substantive CQL ballot work.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the reporter's clearly correct underlying
concern — Atom is dead and the spec should not be silently steering
new readers toward an archived editor — while preserving a small
amount of historical context for users coming from older training
material. It is only marginally more work than Disposition A
(essentially one extra paragraph), and unlike Disposition C it
actually fixes the broken-guidance problem that prompted Lynn Perrine
to file the ticket. Disposition C would also leave the CQL spec in
the awkward position of recommending a tool whose maintainers
themselves have moved on, which is hard to defend at ballot. The
work group should ask the CQF tooling maintainers (via the `#cql`
stream) to confirm the preferred wording for the VS Code-centric
replacement before the editorial change is made.
