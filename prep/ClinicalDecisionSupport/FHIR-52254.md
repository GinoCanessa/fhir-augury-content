## Summary

`ResearchStudySaeReportingMethod` extension: specifically, whether it is
linked to / used by the Vulcan Adverse Events Clinical Research profile.

## Proposed Dispositions

Because the ticket is a Question (no spec change requested) and a
resolution has already been drafted by the work group ("Considered -
Question answered"), the dispositions below are framed around how to
respond to the submitter rather than how to modify a normative artifact.
No GitHub repo briefings apply, so no spec/code paths are named.

### Disposition A: Accept As Requested

#### Proposal

Answer the question as the submitter posed it: confirm in a Jira reply
that the `ResearchStudySaeReportingMethod` extension is **not directly
bound** to the Vulcan Adverse Events Clinical Research profile, but is
defined on `ResearchStudy` and used by the EBM IG's `ResearchStudySite`
and `StudyDesign` profiles. Note that the Vulcan UDP IG is expected to
derive from the `StudyDesign` profile, so the extension is reachable by
Vulcan UDP transitively. Mark the ticket "Considered - Question
answered" (matching the resolution already on the ticket) and close.

#### Justification

This matches the existing resolution text already drafted on the ticket
and gives the submitter exactly what was asked for: a yes/no plus a
pointer to where the extension actually lives. There is no normative
change to evaluate, so accepting the question-as-asked carries no
implementation risk and avoids inventing scope the submitter did not
request.

---

### Disposition B: Alternative Approach

#### Proposal

Answer the question, **and** add a short narrative note (in the EBM IG's
`ResearchStudy`/extension intro page) that explicitly enumerates which
profiles consume `ResearchStudySaeReportingMethod` and calls out the
expected Vulcan UDP / Vulcan Adverse Events Clinical Research reuse
pattern. Optionally, file a follow-up tracker against the relevant
Vulcan IG to add a conformance example showing the extension on a
`StudyDesign`-derived profile.

#### Justification

The fact that this question was asked at all is evidence that the
linkage between the EBM IG extension and downstream Vulcan profiles is
not obvious from current narrative. A small documentation patch costs
little and prevents the same question from recurring from other Vulcan
implementers. This is heavier than Disposition A (it implies an
authoring change to the EBM IG and possibly a Vulcan-side follow-up),
but it converts a one-off Q&A into durable guidance.

---

### Disposition C: Decline

#### Proposal

Decline to take any action beyond closing the ticket: rule that the
question is out of scope for the EBM IG work group because the binding
of the extension into a Vulcan profile is a Vulcan IG editorial
decision, not an EBM IG one. Refer the submitter to the Vulcan
Accelerator / Adverse Events Clinical Research IG maintainers.

#### Justification

The extension is defined in the EBM IG, but whether a *Vulcan* profile
chooses to slot it in is a question for the Vulcan project. Declining
keeps the EBM work group out of authoring choices for downstream IGs it
does not own and avoids implying that the EBM IG endorses any particular
Vulcan profile composition.

---

### Recommendation

**Recommended disposition:** A

The ticket type is "Question," the submitter asked a narrow factual
question, and the work group has already drafted a resolution
("Considered - Question answered") that directly answers it. Disposition
A delivers exactly what was requested with no extra scope and matches
the in-flight resolution. Disposition B is reasonable as a secondary
follow-up if reviewers feel the linkage deserves explicit narrative
in the EBM IG, but it should not block closing this ticket. Disposition
C over-reads the question — the submitter is asking where the EBM IG
extension is used, which the EBM work group is the right body to
answer.
