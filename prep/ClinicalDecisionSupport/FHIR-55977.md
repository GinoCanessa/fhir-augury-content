## Summary

The ticket asks the Clinical Decision Support work group to extend the FHIR Type
Mapping in the *Using CQL With FHIR* specification to cover the previously
unspecified case of `Interval<System.Time>`. Today the mapping table tells
implementers how `System.Date`, `System.DateTime`, and `System.Time` map to
FHIR primitives and how `Interval<System.DateTime>` and `Interval<System.Date>`
map to `Period`, but it does not say what to do when an Interval whose point
type is `System.Time` (a time-of-day) needs to be conveyed back through a FHIR
type whose only interval-like representation is `Period`, which itself is
restricted to `dateTime`. The reporter proposes that `Interval<System.Time>`
be mapped to `FHIR.Period`, with the time values serialized as `dateTime`
anchored on the minimum date `@0001-01-01` (e.g., CQL `T10` becomes the FHIR
`dateTime` `@0001-01-01T10`); on the way back, the `cqlType` extension on the
parameter result is used to recognize that the value is really a `System.Time`
and the date component is dropped. The ticket has been triaged as
**Persuasive**, and the resolution text already drafts the table entry and
asks for a time-interval-valued element to be added to the complete mapping
example. There are no cross-referenced Jira tickets — only the originating
Zulip thread on `#cql` (see below).

## Details

**Description:**

The current FHIR Type Mapping does not call out the specific case of how an
Interval of `System.Time` is mapped. Because `Period` only supports
`dateTime`, we need to specify how `dateTime` is used to round-trip a
`System.Time` value.

Propose that we use the minimum date value and round-trip using that. For
example:

`T10` would be represented in FHIR as `@0001-01-01T10`

and then when coming back to a `System.Time`, `@0001-01-01T10` would be
converted to a `System.Time` based on the `cqlType` specifier indicating that
the value is a `System.Time`, rather than a `DateTime`.

See the Zulip discussion here:
<https://chat.fhir.org/#narrow/channel/179220-cql/topic/FHIR.20Representation.20of.20CqlInterval.3CCqlTime.3E/with/574482132>

**Resolution Description (already drafted on the ticket):**

> Add an entry to the table to indicate how `Interval<System.Time>` is
> mapped: `Interval<System.Time>` maps to `FHIR.Period`, with the `Time`
> values serialized as FHIR `dateTime` with a `Date` component of the minimum
> date (`@0001-01-01`). Add a time-interval-valued element to the complete
> mapping example.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already drafted on the ticket. In the *Using CQL With
FHIR* specification's **FHIR Type Mapping** table, add a row for
`Interval<System.Time>` whose mapping target is `FHIR.Period`, with the
following normative semantics:

- Each `System.Time` endpoint is serialized as a FHIR `dateTime` whose date
  component is the minimum date `@0001-01-01` and whose time component is
  the original `System.Time` value (e.g., CQL `T10` ⇒ FHIR `dateTime`
  `@0001-01-01T10`).
- On the inbound direction, when a parameter / result carries the `cqlType`
  extension identifying the logical type as `Interval<System.Time>`,
  consumers MUST drop the date component and reconstruct a `System.Time`
  from the time portion of each `Period` boundary.
- Add a time-interval-valued element to the *complete mapping example* that
  exercises this round-trip (using something analogous to Alexander's
  `expand … per hour` output).
- Cross-link the new row from the existing `Interval<System.DateTime>` /
  `Period` row so implementers find it from the obvious starting point.

#### Justification

This is exactly what was negotiated on `#cql` and what the reporter (the
spec's lead author) proposed; the only opposition would be silence, and the
thread shows alignment from the implementer who raised the issue. It reuses
existing datatypes (no new extension, no new resource), keeps `Period` as
the single canonical representation for any kind of interval-of-time, and
relies on the already-defined `cqlType` extension to disambiguate on the
inbound trip. The spec gap is real (mapping is currently silent on the
case) and Persuasive triage already commits CDS to filling it; "Accept As
Requested" simply ratifies the agreed wording.

---

### Disposition B: Alternative Approach

#### Proposal

Same end goal, but rather than overloading `Period` with a sentinel base
date, define a small **`time-period`** extension (in the CQL-on-FHIR or
CDS-supplied extension namespace) that carries two `time` primitives —
`start` and `end` — and map `Interval<System.Time>` to that extension on the
parameter / result element. The FHIR Type Mapping table would then read:

> `Interval<System.Time>` ⇒ extension `time-period` (with `start` / `end`
> of type `time`).

The `cqlType` extension would still be present, but disambiguation would no
longer depend on agreeing to ignore a sentinel date value.

#### Justification

The base-date convention works, but it deliberately produces FHIR
`dateTime` instances that are *factually wrong* (they assert a year of 1)
and rely on every consumer to know to ignore the date. Anything in the
pipeline that lacks `cqlType` awareness — a generic FHIR validator, a UI
that renders `Period.start`, an analytics extract — will display "year 1"
data, which is at best confusing and at worst a data-quality regression.
A purpose-built extension makes the time-only nature self-describing,
sidesteps the `@0001-01-01` magic value, and is internally consistent with
how CQL-on-FHIR already uses extensions (e.g., `cqlType`) to convey
type-system information that FHIR's primitive types cannot carry. The
trade-off is one more extension to author and the loss of `Period`'s
existing tooling, which Disposition A keeps.

---

### Disposition C: Decline

#### Proposal

Decline the request and document, in a non-normative note in the FHIR Type
Mapping section, that `Interval<System.Time>` has no FHIR representation
and that implementations needing to convey such results should either (a)
project the time interval onto a known date supplied by the caller before
returning, or (b) return the value as a serialized CQL literal in a
`string` parameter result.

#### Justification

`System.Time` intervals are an edge case generated mostly by
`expand … per hour`-style operations against time-only inputs; clinical
data is overwhelmingly date-or-instant valued, and forcing a normative
mapping risks ossifying a sentinel-date convention that we may regret if a
future FHIR R-version introduces a real `time-period` datatype. Declining
keeps the mapping table honest about the gap and pushes the workaround
into implementer-side code where it belongs until a first-class FHIR
representation exists.

---

### Recommendation

**Recommended disposition:** A

The CDS work group has already triaged this Persuasive, the spec's lead
author drafted the resolution, and the only implementer who hit the gap
agreed with the proposal on Zulip. The base-date convention is ugly but
it is **bounded**: the `cqlType` extension on the parameter result already
carries the disambiguating information that consumers need, so the
sentinel `@0001-01-01` only ever appears inside a payload that announces
itself as carrying a `System.Time`. Inventing a new extension
(Disposition B) is technically cleaner but introduces a second
representation that the rest of the *Using CQL With FHIR* mapping table
would have to learn about and that existing CQL-on-FHIR engines would
have to ship support for, with no implementer asking for it. Declining
(Disposition C) leaves a real, demonstrated interoperability hole open
and contradicts the Persuasive triage. Accept the proposal, but in
drafting the table row make the round-trip rules explicit and reinforce
that the `@0001-01-01` date is a sentinel that consumers MUST ignore when
`cqlType` indicates `Interval<System.Time>`.
