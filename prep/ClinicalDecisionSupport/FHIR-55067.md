## Summary

This ticket flags that the `GuidanceResponse` resource currently defines and
binds value sets and code systems whose home arguably belongs in
Terminology.HL7.org (THO) rather than the FHIR core specification. Specifically,
the `CodeSystem/guidance-module-code` (`GuidanceModuleCode`) and the
corresponding `ValueSet/guidance-module-code` — bound to
`GuidanceResponse.module[x]` — are identified as candidates for migration to
THO so that they can evolve independently of the FHIR release cadence and be
re-used across CDA, V2, and other HL7 product families. Migration is described
as not strictly required, but the workgroup is being asked to evaluate it. The
ticket originates from Comment 33 in a broader THO-migration sweep imported by
Lloyd McKenzie. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items associated with this issue in the cached corpus.

## Details

**Description:**

The GuidanceResponse references several value sets that are potential
candidates for migration to THO. While value set migration is not **required**,
if it's realistic for the value set definition to need to change in the coming
years, or the value sets might be useful in CDA, V2 or other specs, migration
should be considered. Specifics: CodeSystem: `guidance-module-code`
(`GuidanceModuleCode`) referenced by ValueSet `guidance-module-code` bound to
element `GuidanceResponse.module[x]`.

(Comment 33 - imported by: Lloyd McKenzie)

  `guidance-module-code` content stays in core but is flagged as experimental
  given that the binding strength to `GuidanceResponse.module[x]` is already

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate `CodeSystem/guidance-module-code` and `ValueSet/guidance-module-code`
from the FHIR core specification to Terminology.HL7.org (THO). Replace the
in-spec definitions with THO-hosted canonical URLs (e.g.,
`http://terminology.hl7.org/CodeSystem/guidance-module-code` and the matching
ValueSet canonical), update `GuidanceResponse.module[x]`'s binding to point at
the THO ValueSet canonical, retain the `example` binding strength, and add the
standard "moved to THO" redirect/note in the FHIR build for consumers of the
old canonical.

#### Justification

Aligns with HL7's broader practice of housing reusable, slowly evolving
terminologies in THO so they can be referenced from CDA, V2, CIMI, and other
product families without being pinned to a FHIR release. It also lets the
content be revised between FHIR releases. This is the pattern the import
comment is steering all candidate value sets toward, and it produces the
cleanest cross-product story.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `guidance-module-code` (both CodeSystem and ValueSet) in the FHIR core
spec, but mark both resources as `experimental = true` and leave the binding
on `GuidanceResponse.module[x]` at `example`. Add a short note in the
GuidanceResponse narrative explaining that the codes are illustrative only
and that implementers are expected to supply their own module identifiers
(typically URLs/canonicals to the actual decision-support module).

#### Justification

This is the path proposed by bryn.rhodes in the lone comment. Because the
binding is already `example`, no implementer is expected to use these codes in
production — the value set exists primarily as documentation. Migrating to THO
adds maintenance overhead (separate publication cadence, redirects,
cross-references) for content that, per the WG comment, isn't expected to be
authoritative or to grow. Marking it experimental signals "do not rely on
these codes" without paying the cost of a full THO migration, and matches how
CDS WG has historically treated other illustrative bindings on
GuidanceResponse and PlanDefinition.

---

### Disposition C: Decline

#### Proposal

Take no action. Leave `CodeSystem/guidance-module-code` and
`ValueSet/guidance-module-code` in the FHIR core spec as-is, with the existing
`example` binding on `GuidanceResponse.module[x]`, and close the ticket as
"Not Persuasive" on the grounds that an `example`-strength binding to an
illustrative value set already conveys that the codes are not normative and
need no further treatment.

#### Justification

`GuidanceResponse.module[x]` is bound at `example` strength, meaning the
codes are explicitly non-binding. Implementers are already expected to point
at their own module identifiers (frequently a canonical URL to a
PlanDefinition/Library), not to pick from this list. Neither THO migration nor
an `experimental` flag changes conformance behaviour for an example binding,
so the change has minimal practical value for implementers and incurs
publication / cross-spec maintenance cost.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the substantive concern raised in the ticket (that the
codes are not authoritative and should not be relied upon) at the lowest cost
and matches the only WG comment on file (bryn.rhodes). A full THO migration
(Disposition A) is defensible in principle but is hard to justify for a value
set bound at `example` strength that has no signalled cross-product demand and
no Zulip/GitHub discussion driving reuse. Outright declining (Disposition C)
ignores the clear request from the import comment and leaves a value set that
the workgroup itself characterises as illustrative without the
`experimental` marker that would warn implementers off. Marking both the
CodeSystem and ValueSet `experimental` while keeping the binding at `example`
gives consumers an unambiguous signal and preserves the option of a future
THO migration if real cross-product use cases emerge.
