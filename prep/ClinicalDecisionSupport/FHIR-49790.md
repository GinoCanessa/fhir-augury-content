## Summary

The `activity-example-recordinference-ad` ActivityDefinition in the CPG
IG demonstrates how a `record-inference` Task should be produced, but
its current `dynamicValue` list is wrong in two ways. First, it writes
the inferred Observation directly into `Task.input.value` via a
cql-expression that constructs an Observation inline — this is invalid
because `Task.input.value` has no Observation datatype and an
Observation must be referenced, not embedded as a value. Second, the
example does not set the `cpg-caseFeatureType` extension on the
inferred Observation, even though the IG's methodology page explicitly
calls out the `inferred` vs `asserted` distinction for case features
and recommends that extension for traceability in the CPG workflow.

The reporter (Taylor Le) proposes correcting the AD so that it builds
the Observation into `Task.contained` (id `inference`) and points
`Task.input[0].valueReference` at `#inference`, while also carrying the
`cpg-caseFeatureType = inferred` extension on the contained
Observation. The test-case Task and Observation instances rendered as
`Task-request-inf-scenario2`, `Observation-inf-scenario4`, and
`Task-event-inf-scenario4` must be updated in parallel so the
walk-through stays internally consistent. A PR implementing this
change is already up at
[HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51).
There are no cross-referenced Jira tickets.

## Details

**Description:**

The record inference activity may be used to propose a recommendation
to record an inferred observation. The current example sets the target
resource `Task.input.value` via a dynamic value of type
`cql-expression` where the expression returns an observation. In order
to appropriately reference the observation, the example should instead
set a contained observation on `Task` with a reference to the contained
resource from `Task.input.valueReference`.

In addition, the methodology section of the CPG IG provides guidance on
inferred vs asserted case features
(<https://hl7.org/fhir/uv/cpg/methodology.html#profiles>). It is
recommended to include the
[`cpg-caseFeatureType`](https://hl7.org/fhir/uv/cpg/StructureDefinition-cpg-caseFeatureType.html)
extension as a part of the CPG workflow for traceability. In the
example, this should be included as an extension on the inferred
observation. The test case request and event resources should be
updated to reflect the extension as well:

- <https://hl7.org/fhir/uv/cpg/Task-request-inf-scenario2.html>
- <https://hl7.org/fhir/uv/cpg/Observation-inf-scenario4.html>
- <https://hl7.org/fhir/uv/cpg/Task-event-inf-scenario4.html>

The ticket proposes a concrete corrected ActivityDefinition JSON whose
`dynamicValue` array has three entries: (1) `path = contained` builds a
`FHIR.Observation` with `id = 'inference'`, an `extension` carrying
`cpg-caseFeatureType = inferred`, `status = final`, a coded `code`, a
`subject` reference to the patient, and `issued = Now()`; (2)
`path = input[0].type` bound to `%resource.code`; (3)
`path = input[0].valueReference` bound to a `FHIR.Reference` whose
`reference.value` is `'#inference'`.

## Related Zulip Discussions

### [cql > Returning Observation after $apply and CQL execution](https://chat.fhir.org/#narrow/stream/cql/topic/Returning%20Observation%20after%20%24apply%20and%20CQL%20execution/near/499335643)

Ignacio Jauregui opened the thread trying to get
`ActivityDefinition/$apply` on the CPG record-inference example to
return a Task that carries a CQL-generated Observation, first hitting
"could not resolve the library" errors in both CQF Ruler and Reason.
Brenin Rhodes initially reminded the group that an ActivityDefinition
cannot *create* a free-standing Observation in R4 (Observations aren't
in the request-resource-types value set) and suggested a
Questionnaire/QR loop instead. Taylor Le then observed that the
CPG IG's existing record-inference examples are actively misleading —
they write Observations into `Task.input.value` (which has no
Observation datatype) rather than referencing them — and filed
**FHIR-49790** to drive an example correction in the next CPG ballot.
Brenin Rhodes followed up with a concrete corrected AD (matching the
contained-Observation + `input.valueReference = '#inference'` pattern
the ticket now proposes) and reported that it executes cleanly against
HAPI JPA Server Starter with the CR module enabled; Ignacio confirmed
it works with the `cds.application.yaml` alternate config. Taylor Le
raised whether the pattern should be adopted in the IG and Brenin
agreed to bring it to the CPG call. The thread then pivoted to a
library-based variant (Ignacio wrapping the Observation-building
expression in a `Library` resource); Forest and Brenin diagnosed a
`Library.content` encoding gap and a Library-name/URL mismatch, which
Ignacio resolved. Net consensus: the current example is wrong, the
contained-Observation + `valueReference` pattern works, and the IG
should adopt it.

Participants: Ignacio Jauregui, Brenin Rhodes, Taylor Le, Derek Ritz,
Forest.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the correction exactly as the reporter specified, in
`HL7/cqf-recommendations`:

1. In
   `input/fsh/examples/activity-examples/resources/activitydefinition-activity-example-recordinference.fsh`,
   replace the current `dynamicValue` rules with three entries:
   - `dynamicValue[contained]` → a CQL expression producing a
     `FHIR.Observation` with `id = 'inference'`, an `extension`
     carrying
     `http://hl7.org/fhir/uv/cpg/StructureDefinition/cpg-caseFeatureType = inferred`,
     `status = final`, coded `code = { code: 'Inference' }`,
     `subject.reference = Patient.id.value`, and `issued = Now()`.
   - `dynamicValue[input[0].type]` → cql-expression `%resource.code`.
   - `dynamicValue[input[0].valueReference]` → cql-expression
     `FHIR.Reference { reference: FHIR.string { value: '#inference' } }`.
2. Mirror the `cpg-caseFeatureType = inferred` extension onto the
   inferred Observation in
   `input/fsh/examples/activity-examples/test-cases/inf-scenario4.fsh`.
3. Ensure
   `input/fsh/examples/activity-examples/test-cases/request-inf-scenario2.fsh`
   and `.../event-inf-scenario4.fsh` reference the updated Observation
   (via contained + `input.valueReference` where they echo the AD's
   output) so the rendered
   `Task-request-inf-scenario2.html`,
   `Observation-inf-scenario4.html`, and
   `Task-event-inf-scenario4.html` pages tell a consistent story.
4. Sync the canonical
   `input/fsh/resources/activity-definitions/cpg-recordinference-activitydefinition.fsh`
   to the same pattern so the canonical AD and the example do not
   drift. Validate the result with sushi + IG-publisher and execute the
   AD against HAPI JPA + CR module (using `cds.application.yaml`) to
   confirm it `$apply`s cleanly, per the engine proven out in the
   Zulip thread. Close the ticket via merge of
   [HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51)
   once reviewed, updating that PR as needed to cover all three
   test-case files.

#### Justification

This matches the reporter's proposal one-for-one and is already
prototyped in a draft PR. The underlying bug is real: `Task.input.value`
has no `Observation` datatype, so the current example is invalid FHIR
regardless of whether any runtime tolerates it — the only correct way
to attach a generated Observation to a Task in R4 is via
`contained[]` + `input.valueReference = '#<id>'`, which is exactly what
the corrected example produces. Adding `cpg-caseFeatureType = inferred`
aligns the example with the IG's own methodology
(`methodology.html#profiles`), improving traceability for downstream
implementers who use case-feature distinctions. Brenin Rhodes confirmed
in the linked Zulip thread that this pattern `$apply`s successfully in
HAPI JPA with the CR module, so feasibility is not speculative. Because
the change is confined to example instances within the CPG IG, there is
no cross-repo blast radius and no FhirCore impact — the repo briefing
confirms the change is wholly internal.

### Disposition B: Alternative Approach

#### Proposal

Adopt the contained-Observation + `valueReference` fix from
Disposition A, **and in addition** refactor the in-line CQL expression
into a named CQL library so the example demonstrates the
production-style pattern rather than an inline literal. Concretely:

1. Move the Observation-building CQL out of the AD's `dynamicValue`
   entry for `contained` and into
   `input/cql/RecordInference.cql` (or a sibling
   `ObservationInference.cql`) as a top-level `define`, e.g.
   `define "Inferred Observation": FHIR.Observation { ... }` with the
   `cpg-caseFeatureType = inferred` extension baked in.
2. Set `ActivityDefinition.library` to the corresponding
   `Library/RecordInference` canonical (already represented by
   `input/fsh/examples/activity-examples/resources/library-RecordInference.fsh`).
3. Change the `dynamicValue[contained]` rule to
   `{ language: 'text/cql-identifier', expression: 'Inferred Observation',
   reference: 'http://hl7.org/fhir/uv/cpg/Library/RecordInference' }` —
   identical in shape to the working pattern
   Ignacio Jauregui arrived at in the Zulip thread once
   Brenin Rhodes and Forest resolved the Library name / content
   issues.
4. Add the `cpg-caseFeatureType = inferred` extension to the
   corresponding test-case Observation and to the request / event Task
   pages named in the ticket, as in Disposition A.
5. Ensure the CQL library name matches the `Library.name` /
   `Library.url` per
   <https://build.fhir.org/ig/HL7/cql-ig/conformance.html#library-name-and-url>
   (a pitfall the Zulip thread explicitly flagged).

#### Justification

The CPG IG's whole premise is that CQL-backed clinical logic is
shareable and executable; leaving the Observation-construction CQL as
an *inline* `text/cql-expression` in the AD hides that logic from
validation, versioning, and reuse. Disposition A cures the
immediate FHIR error but keeps the example as a demonstration of the
weakest variant of the pattern. Disposition B costs a little more
authoring (one extra FSH library, one extra CQL file) but teaches the
right shape — library-referenced CQL driving a contained-Observation
Task — and it's the shape Brenin Rhodes and Ignacio Jauregui
*actually got running end-to-end* in the Zulip discussion. The
gotcha the briefing calls out (CQL-expression `dynamicValue` surfaces
are fragile) also argues for this: moving the expression into a Library
gets it into the normal CQL toolchain where compile errors surface at
publish time, not at `$apply` time. The trade-off vs. A is a slightly
larger diff and a stronger implicit recommendation; the risk is that
some tools historically had trouble resolving Library references from
`dynamicValue`, but that's now known-working on HAPI JPA + CR.

### Disposition C: Decline

#### Proposal

Do not change the current example. Mark FHIR-49790 as "Persuasive with
Modification — deferred to next CPG major" or "Not Persuasive" on the
grounds that the example is already flagged as `experimental: true`,
that the methodology page already describes the inferred-vs-asserted
distinction, and that implementers should rely on narrative guidance
rather than the in-line cql-expression example. Close the ticket
without merging
[HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51).

#### Justification

This is defensible only if the WG decides the example is too deep in
the weeds for a point release and that fixing it now would imply
endorsement of an `$apply` pattern that is still evolving (the Zulip
thread shows the pattern is engine-sensitive). The WG could argue that
since Observations aren't producible by an ActivityDefinition in
standard R4 anyway (Brenin Rhodes' original point), leaving the example
as a cautionary artifact until CPG moves to an R5-aligned substrate is
less misleading than publishing a new "recommended" shape that only
works on certain engines. This position is weak, however: the current
example is actively broken FHIR (`Task.input.value` typed as an
Observation), so keeping it in the IG as `experimental` still ships
invalid example content and continues to mislead readers — exactly the
problem the reporter describes.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right call for this ticket. The bug is narrow,
well-understood, and already has a working fix in
[PR #51](https://github.com/HL7/cqf-recommendations/pull/51). The
reporter's proposal is faithful to the CPG methodology page, adds the
`cpg-caseFeatureType` extension the WG already recommends for
traceability, and has been validated end-to-end in HAPI JPA + CR per
Brenin Rhodes' confirmation in the Zulip thread. Disposition B is
technically preferable — a library-referenced CQL pattern is the shape
CPG should be modelling — but it's a larger change that the WG may
reasonably want to drive via a separate ticket so the fix for the
invalid `Task.input.value` / missing-extension bug isn't held up. The
WG should accept Disposition A as-filed, merge PR #51 with the
follow-up test-case updates for
`Task-request-inf-scenario2`, `Observation-inf-scenario4`, and
`Task-event-inf-scenario4`, and (optionally) spin a new ticket to
promote the example to the Library-referenced variant captured in
Disposition B.
