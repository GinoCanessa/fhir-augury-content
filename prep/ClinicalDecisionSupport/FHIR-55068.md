## Summary

The ticket reports that `GuidanceResponse.status` is defined as a 1..1
mandatory element with a required binding to the `GuidanceResponseStatus`
value set, but that value set offers no concept that can act as an
"escape valve" — there is no `unknown`, `other`, or equivalent code.
As a result, when an implementer needs to record a `GuidanceResponse`
whose status truly cannot be classified into the existing concepts
(`in-progress`, `success`, `data-requested`, `data-required`, `failure`,
`entered-in-error`), the resource becomes impossible to populate
validly. The reporter (Lloyd McKenzie) asks that an `unknown`, `other`,
or equivalent concept be added to the value set so the required-bound,
mandatory element always has a usable code. The single comment on the
ticket (from Bryn Rhodes) suggests an alternative approach: relax the
cardinality of `status` and/or introduce an `other` concept. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the data sources for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an escape-valve concept to the `GuidanceResponseStatus` value set
(and the underlying `guidance-response-status` code system) so that the
required-bound, 1..1 `GuidanceResponse.status` element can always be
populated. Concretely:

- Add a new code `unknown` to the `guidance-response-status` code
  system, with a definition along the lines of "The status of the
  guidance response is not known." This mirrors the pattern used by
  `Observation.status = unknown`, `MedicationRequest.status = unknown`,
  and other Clinical resources that already provide an `unknown` escape
  valve under a required binding.
- Include the new code in the `GuidanceResponseStatus` value set so the
  required binding continues to enclose every legal code.
- Update the element definition narrative for `GuidanceResponse.status`
  to clarify when `unknown` is appropriate (e.g., the response was
  generated by an external/legacy system that did not communicate a
  status, or the responding service truly cannot determine state).

#### Justification

This is the smallest, most targeted change that resolves the reported
defect without altering the resource's structural shape. The pattern of
adding an `unknown` code to required-bound status value sets is
well-established across FHIR (Observation, Encounter, MedicationRequest,
ServiceRequest, etc.), so it is consistent with the rest of the spec
and creates no new design precedent. It preserves the contract that
every `GuidanceResponse` instance has an explicit, machine-processable
status, which downstream CDS consumers rely on.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the alternative suggested in Bryn Rhodes' comment: relax the
cardinality of `GuidanceResponse.status` from `1..1` to `0..1` **and**
add an `other` (or `unknown`) concept to `GuidanceResponseStatus`. The
relaxed cardinality lets a sender omit `status` entirely when it is
genuinely unavailable, while the new code gives them a way to assert
"explicitly unknown / other" when the distinction matters. Optionally,
weaken the binding strength from `required` to `extensible` so
implementers can introduce a project-specific status code (e.g.,
"cancelled", "superseded") without waiting on a base-spec change.

#### Justification

This addresses the underlying need more completely than Disposition A.
There are two distinct failure modes behind "impossible to populate":
(1) the status is conceptually outside the current code list, and (2)
the status is genuinely unknown or unavailable at the time of
serialization. A single new `unknown` code only fixes case (1)
imperfectly and does not help case (2) at all. Loosening cardinality
removes the structural blocker, the new code communicates intent when
the sender does know it is unknown, and `extensible` binding leaves
room for legitimate domain-specific extensions. The trade-off is that
existing consumers must now handle a missing `status` and possibly
unrecognized codes — but those are already obligations under FHIR's
general "must support" / extensibility rules, and the change is
backward-compatible at the wire level (existing instances remain
valid).

---

### Disposition C: Decline

#### Proposal

Decline the change. Position the resolution as: instances that
genuinely cannot determine a status should use the
`data-absent-reason` extension on `GuidanceResponse.status` with a
reason of `unknown`, which is the standard FHIR mechanism for
"required element whose value is not known." No change to the value
set, code system, or cardinality is required.

#### Justification

FHIR already provides a uniform escape valve for required primitive
elements via the `data-absent-reason` extension, and that mechanism is
specifically designed to handle the "I must send this element but I do
not know its value" case without polluting every status value set with
sentinel codes. Adding `unknown` / `other` to `GuidanceResponseStatus`
risks weakening the semantic precision of the value set (consumers
would have to distinguish "actually unknown" from "the responder chose
not to classify"), and a one-off relaxation of cardinality is
inconsistent with the general FHIR pattern that workflow-status
elements are mandatory.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the minimum viable fix for the reported defect, is
consistent with the well-established pattern used by every other
required-bound status value set in FHIR Core (Observation,
MedicationRequest, ServiceRequest, Encounter, etc.), and avoids the
larger structural and binding-strength debates that Disposition B would
open. It also matches the first half of Bryn Rhodes' comment ("introduce
other") without taking on the cardinality change, which is a more
disruptive edit that deserves its own ticket and impact analysis if the
work group still wants it after seeing how widely Disposition A is
adopted. Disposition C is technically defensible via
`data-absent-reason`, but in practice CDS implementers consistently
report friction with that pattern for status elements, and the rest of
FHIR has chosen to provide an in-value-set escape valve rather than
rely on the extension — `GuidanceResponse` should follow suit. If the
work group later concludes that case (2) ("status genuinely
unavailable") is common enough to warrant cardinality relaxation, that
can be raised as a follow-up CR; nothing in Disposition A blocks it.
