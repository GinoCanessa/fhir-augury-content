## Summary

The ticket asks the Clinical Quality Language (FHIR) specification to add
an explicit statement and a worked example showing how to reference a
data model using an **unqualified** model name in the "Data Models" intro
of *Using CQL*. Today the intro jumps directly into fully qualified
model references (e.g., `FHIR.Patient`) and only implies that the
qualifier is optional via the line "Data model references may include
any number of qualifiers." Chris Moesel notes that unqualified
references have historically been the norm and are still common in the
field, so the spec should document that style explicitly — even if it
also recommends fully qualified names as the preferred form going
forward.

There are no formal Jira links on the ticket, but two sibling tickets
filed by the same reporter (and a closely-related ticket from Hans
Buitendijk) inform the broader picture:

- [FHIR-52460](https://jira.hl7.org/browse/FHIR-52460) — asks for a
  parallel clarification in the same *Multiple Data Models* area: how
  to reach the System type (`System.Quantity`) when an unqualified
  identifier resolves to a model type (`FHIR.Quantity`).
- [FHIR-52512](https://jira.hl7.org/browse/FHIR-52512) — flags the
  rule "the local identifier for a referenced model must be unqualified
  and unique within the artifact" as confusing and asks for
  clarification of when/why a model alias would be used.

Together these three tickets indicate that the *Data Models* / *Multiple
Data Models* section of the CQL-on-FHIR intro needs a small editorial
pass that walks readers through unqualified usage, the local-identifier
rule, qualifier disambiguation, and the System-vs-FHIR `Quantity` case.

## Details

**Description:**

> The intro to data models jumps right into using fully qualified data
> models. While it implies that you can use an unqualified model name
> (e.g., "Data model references may include any number of qualifiers"),
> it does not say so explicitly, nor does it provide an example. Since
> unqualified data models have been the norm up until now, it probably
> makes sense to continue to document that approach explicitly (even if
> it includes a caveat that using fully qualified model names is
> preferred).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the *Using CQL* page of the CQL-on-FHIR specification, expand the
introductory paragraph of the **Data Models** section so that it
explicitly states that a `using` declaration's model identifier may be
written as an *unqualified* name (e.g., `using FHIR version '4.0.1'`
followed by retrieves like `[Patient]` and `[Observation: "LOINC code"]`)
in addition to the fully qualified form (`[FHIR.Patient]`). Add a short
worked example immediately after the existing fully qualified example
that shows the same retrieve written without the `FHIR.` qualifier, and
follow it with a one-line caveat noting that fully qualified names are
the preferred style when more than one data model is in scope (linking
forward to the *Multiple Data Models* subsection). Keep the existing
"Data model references may include any number of qualifiers" sentence
but reposition it so it directly precedes the new unqualified/qualified
contrast.

#### Justification

This is a documentation-only change that exactly matches the reporter's
ask and carries no normative risk. Unqualified retrieves are how
existing measure and decision-support libraries are written in the
field today (see context in [FHIR-26592](https://jira.hl7.org/browse/FHIR-26592)
and [FHIR-31019](https://jira.hl7.org/browse/FHIR-31019)), so
documenting the pattern explicitly makes the spec match observed
implementer behavior rather than introducing a new style. Adding the
"prefer fully qualified" caveat preserves the spec's current direction
of travel without invalidating the large body of single-model CQL
already in production.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-52458 as part of a coordinated editorial pass on the
*Data Models* / *Multiple Data Models* intro and resolve it together
with [FHIR-52460](https://jira.hl7.org/browse/FHIR-52460) and
[FHIR-52512](https://jira.hl7.org/browse/FHIR-52512). Restructure the
intro into three short, explicitly-titled subsections:

1. **Single-model usage (unqualified identifiers)** — covers the case
   FHIR-52458 asks for, with a worked `[Patient]` / `[Observation]`
   example and a forward pointer to the qualified form.
2. **Multiple data models and qualifier disambiguation** — keeps the
   existing fully qualified example and adds the System-vs-model note
   requested by FHIR-52460 (`System.Quantity` vs `FHIR.Quantity`).
3. **Local identifiers and aliases** — explicitly defines the
   "unqualified and unique within the artifact" rule that FHIR-52512
   flags, with an example of when an alias is useful (e.g., importing
   two QI-Core versions into the same library).

Mark all three tickets as Persuasive-with-Mod referencing this combined
edit.

#### Justification

The three tickets all target the same handful of paragraphs and the
same conceptual material (qualified vs unqualified model identifiers
and the role of local aliases). Resolving them in a single editorial
pass produces a more coherent intro than three drive-by edits, avoids
churn for implementers reading the spec, and removes the risk that the
fix for FHIR-52458 inadvertently re-confuses the points raised in
FHIR-52460 and FHIR-52512. The cost over Disposition A is small
(slightly larger PR scope in `HL7/cql-ig`, no normative change), and
the benefit is a self-consistent narrative for new CQL-on-FHIR
authors.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive**, noting that the existing intro already
states "Data model references may include any number of qualifiers,"
which logically permits zero qualifiers (i.e., the unqualified form),
and that the rest of the *Using CQL* page contains numerous examples
of unqualified retrieves. Resolve with a comment pointing the reporter
to those downstream examples rather than expanding the intro.

#### Justification

The information the reporter asks for is technically already in the
spec, and CQL-on-FHIR is in a stable maintenance phase where
purely-editorial churn carries some review cost. If the working group
believes the intro is already adequate and that the existing examples
elsewhere in *Using CQL* are sufficient, declining keeps the intro
short and avoids re-opening editorial debate about which style is
"preferred." This disposition is the weakest of the three because the
reporter's underlying observation — that the current wording is
implicit rather than explicit — is fair, and ignoring it leaves
FHIR-52460 and FHIR-52512 as unresolved adjacent issues.

---

### Recommendation

**Recommended disposition:** **B (Alternative Approach)**

Disposition A is a perfectly defensible single-ticket fix, but
FHIR-52458, [FHIR-52460](https://jira.hl7.org/browse/FHIR-52460), and
[FHIR-52512](https://jira.hl7.org/browse/FHIR-52512) all land in the
same one or two paragraphs of the *Using CQL → Data Models* intro and
all reflect the same root cause: the intro mixes the qualified-name
syntax, the unqualified-name convention, and the local-identifier
rule without separating them. Rolling all three into a single
restructured edit gives reviewers one coherent change to weigh, makes
sure the FHIR-52458 example doesn't conflict with the FHIR-52460
System-type clarification, and resolves the FHIR-52512 confusion as a
side effect. The work remains documentation-only in `HL7/cql-ig`
(once that repo's `repo-analysis` briefing is generated and the exact
file paths are confirmed), so the implementation cost is modest
relative to the clarity gained for new CQL-on-FHIR authors. Disposition
C is not recommended: the reporter's observation about implicitness is
accurate and Disposition C leaves the two adjacent tickets unresolved.
