## Summary

The ticket asks the Clinical Quality Language (CQL) specification to make
explicit that the millisecond component of `DateTime` and `Time` literals is
written as a *fractional second*, not as a three-digit integer count of
milliseconds. As reported by Bryn Rhodes, the current text describes the `fff`
token as "A fractional millisecond value (0..999)", which is internally
inconsistent: a fraction must be less than one, yet the stated range goes up
to 999. Readers can plausibly conclude that `T12:00:00.1` means 1 ms when in
fact it represents 100 ms (one tenth of a second). The proposed fix is to
replace the `fff` description with "A millisecond value, represented as a
fractional second (e.g. `.1` = 100 ms)" and to add a worked example to the
`DateTime` and `Time` selector function pages showing that 10 ms renders as
`.01` in literal form.

There are no formal Jira, Zulip, or GitHub cross-references for this ticket,
but the request fits a multi-year cluster of CQL clarifications about
second/millisecond precision led by the same reporter — notably
[FHIR-20309](https://jira.hl7.org/browse/FHIR-20309) ("Define seconds as a
decimal …", Applied, ITS/FHIRPath),
[FHIR-48482](https://jira.hl7.org/browse/FHIR-48482) ("Clarify seconds and
milliseconds are a single precision …", Persuasive, CDS), and
[FHIR-48867](https://jira.hl7.org/browse/FHIR-48867) ("Clarify when seconds
and milliseconds should be combined …", Applied, CDS). The current ticket
extends that clean-up from comparison/duration semantics into the literal
syntax description itself.

## Details

**Description:**

> Clarify that the decimal representation of milliseconds is fractions of a
> second.
>
> So: `T12:00:00.1` corresponds to 100 milliseconds, not 1 millisecond.
>
> Update this language:
>
> `fff` - A fractional millisecond value (0..999)
>
> to
>
> `fff` - A millisecond value, represented as a fractional second
> (e.g. `.1` = 100 ms)
>
> Add an example in DateTime and Time selector functions to illustrate 10
> milliseconds results in 0.01 in a DateTime or Time literal.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the CQL specification source for the Author's Guide DateTime/Time literal
section (the table that defines the `yyyy`, `mm`, `dd`, `hh`, `mm`, `ss`,
`fff` tokens), replace the `fff` row:

- **Old:** `fff` — A fractional millisecond value (0..999)
- **New:** `fff` — A millisecond value, represented as a fractional second
  (e.g. `.1` = 100 ms, `.01` = 10 ms, `.001` = 1 ms; range `.000`..`.999`)

In the CQL Reference page for the `DateTime` and `Time` selector functions
(09-b), add a worked example block immediately after the parameter list
showing that:

```cql
DateTime(2024, 11, 15, 12, 30, 0, 10)
  // literal form: @2024-11-15T12:30:00.010
Time(12, 30, 0, 10)
  // literal form: @T12:30:00.010
```

with a one-sentence note that the integer millisecond argument to the
selector function is rendered in the literal as a three-digit fraction of a
second, so 10 ms appears as `.010` (i.e. `0.01` of a second), not as `.10`
or `.1`.

#### Justification

This is a pure editorial / clarification change with no normative impact:
the runtime semantics (seconds-as-decimal, second/millisecond combined as a
single precision) are already established by the resolved
[FHIR-20309](https://jira.hl7.org/browse/FHIR-20309),
[FHIR-48482](https://jira.hl7.org/browse/FHIR-48482), and
[FHIR-48867](https://jira.hl7.org/browse/FHIR-48867). The current `fff`
description ("A fractional millisecond value (0..999)") is internally
inconsistent — a value labelled "fractional" cannot range up to 999 — and is
demonstrably misleading enough that the reporter (an editor of the spec)
felt a ticket was warranted. Worked examples on the selector function pages
follow the existing CQL Reference convention of pairing constructor calls
with their literal forms.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the spirit of the request but tighten the grammar text further so
that the description matches the EBNF/regex actually used by parsers, rather
than introducing yet another prose phrasing. Specifically:

1. Drop the standalone `fff` token from the DateTime/Time literal table and
   replace the seconds row with a single `ss[.fff]` (or `ss('.' DIGIT DIGIT
   DIGIT)?`) entry described as: *"Seconds as a decimal value in the range
   `00.000`..`59.999`. The fractional part, when present, expresses
   sub-second precision; the convention is that three fractional digits give
   millisecond resolution."*
2. Cross-link this entry from the DateTime/Time selector function pages and
   from the Comparison Operators "decimal semantics" notes already added by
   FHIR-48867, so a reader landing on either page is taken to one
   authoritative definition.
3. Keep the worked example from Disposition A on the selector function
   pages, but additionally show one round-trip case
   (`Time(12, 30, 0, 10) = @T12:30:00.010` → `true`) to anchor the
   "10 ms ⇒ `.010`" rule against the comparison rules.

#### Justification

The literal grammar is the canonical source of truth and is what parser
authors implement against; restating "milliseconds, but actually a
fraction" in prose perpetuates the very confusion the ticket is trying to
fix. Treating seconds as `Decimal` in the grammar (consistent with
FHIR-20309) and explaining sub-second precision once, in the seconds row,
removes the need to explain that the "millisecond" token isn't really a
millisecond count. The trade-off versus Disposition A is a slightly larger
diff and a (minor) re-wording that affects every page that quotes the
DateTime/Time grammar, but it leaves the spec internally consistent rather
than papering over the inconsistency with a clearer comment.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as Not Persuasive on the grounds that the literal grammar
already shows the dot in the BNF (`hh:mm:ss.fff`) and that the runtime
semantics are unambiguous and tested. No spec text would change.

#### Justification

A purely defensive option: every example in the CQL specification already
uses `.fff` after a literal dot, and the underlying decimal-seconds model is
defined elsewhere. One could argue the existing prose, read in context, is
adequate. This option is included for completeness only.

---

### Recommendation

**Recommended disposition:** A

The ticket is small, narrowly scoped, editorial, and exposes a real
internal contradiction in the `fff` description ("fractional" but ranging
to 999). It rounds out a series of related, already-resolved CQL
clarifications around second/millisecond precision
([FHIR-20309](https://jira.hl7.org/browse/FHIR-20309) Applied,
[FHIR-48482](https://jira.hl7.org/browse/FHIR-48482) Persuasive,
[FHIR-48867](https://jira.hl7.org/browse/FHIR-48867) Applied), and the
reporter has even supplied the replacement text and the requested example.
Disposition B is attractive in principle but expands the scope from
"clarify two sentences and add an example" to a grammar-table refactor that
would touch the Author's Guide, the CQL Reference, and any quoted copies of
the DateTime/Time grammar — work the workgroup has not signed up for under
this ticket. Recommend taking Disposition A now and, if the workgroup wants
to pursue the deeper grammar consolidation in B, opening a separate ticket
that supersedes the relevant portions of this one.
