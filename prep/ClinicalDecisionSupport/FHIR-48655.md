## Summary

Bryn Rhodes requests that the CQL-related operations defined in the *Using
CQL With FHIR* IG be aligned with the operation pattern profiles published
by the Clinical Reasoning Module Infrastructure (CRMI) IG. Specifically,
the `$cql` operation should declare conformance to the CRMI
`Parameterizable`, `Data Configurable`, `Artifact Version Bindable`,
`Manifestable`, and `Artifact Configurable` operation patterns; and the
`Library/$evaluate` operation should declare conformance to the CRMI
`Artifact`, `Parameterizable`, `Data Configurable`, `Artifact Version
Bindable`, `Manifestable`, and `Artifact Endpoint Configurable` operation
patterns. The intent is to make the parameter signatures and behavior of
these CQL operations consistent with the broader knowledge-artifact
operation conventions adopted by CRMI, so that implementers building on
both IGs see a single, predictable contract. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were located in FHIR Augury for
this issue, so the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the *Using CQL With FHIR* IG to declare CRMI operation pattern
conformance on the two `OperationDefinition` resources:

- `OperationDefinition/Library-evaluate` (the `Library/$evaluate`
  operation): add the CRMI canonical URLs for the `Artifact`,
  `Parameterizable`, `Data Configurable`, `Artifact Version Bindable`,
  `Manifestable`, and `Artifact Endpoint Configurable` operation pattern
  profiles to `OperationDefinition.meta.profile`. Reconcile the
  parameter set (names, cardinalities, types, bindings) with each named
  pattern; add any missing required parameters (e.g., the artifact
  version / manifest / endpoint configuration parameters mandated by the
  patterns) and document any FHIR-specific parameters that go beyond the
  pattern.
- `OperationDefinition/cql` (the `$cql` operation): add the CRMI
  canonical URLs for the `Parameterizable`, `Data Configurable`,
  `Artifact Version Bindable`, `Manifestable`, and `Artifact
  Configurable` operation pattern profiles to
  `OperationDefinition.meta.profile`, and likewise reconcile the
  parameter list with the parameters required by those patterns. Update
  the operation narrative pages to describe the CRMI alignment, and add
  IG-publisher validation so that the operations are validated against
  the named CRMI profiles at build time. Tag the change in the IG's
  change log as a substantive change requiring re-ballot.

#### Justification

The CRMI IG was specifically designed to factor out the common shape of
knowledge-artifact operations so that downstream IGs (cql-ig, measure
evaluation, PlanDefinition `$apply`, etc.) all expose a consistent
parameter contract. Because *Using CQL With FHIR* is one of the canonical
consumers of CRMI, declaring conformance closes the obvious gap and
gives implementers a single, profile-driven definition of what these
operations accept. It also makes the operations testable against the
CRMI conformance suite, which lowers the cost of interoperability between
CQL evaluation engines.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than asserting conformance directly on the existing
`OperationDefinition` resources in *Using CQL With FHIR*, replace those
operations with a thin profile/derivation pattern: keep the base
operations in cql-ig as they are today, and publish *new* derived
`OperationDefinition` resources (e.g., `Library-evaluate-crmi` and
`cql-crmi`) that explicitly inherit from the corresponding CRMI base
operations and constrain them for CQL use. Mark the original operations
as informative or deprecated in a future release. This keeps the existing
canonical URLs stable for current implementers while still publishing a
fully CRMI-conformant variant for those who need pattern-level
guarantees.

#### Justification

Adding CRMI pattern profiles to the existing operations is technically a
substantive change: the patterns may require parameters that the current
operations do not declare, or impose cardinalities tighter than what
deployed servers implement. Splitting the new conformance into a derived
operation avoids breaking those servers while still making the
CRMI-aligned contract first-class. It also aligns with the way other
clinical-reasoning IGs (e.g., measure-ig, plan-definition apply) layer
CRMI conformance on top of base FHIR operations.

---

### Disposition C: Decline

#### Proposal

Do not modify the *Using CQL With FHIR* operations to declare CRMI
operation pattern conformance. Instead, document the relationship
informatively (a cross-reference from the operation pages to the CRMI
patterns) and defer any normative alignment to the next major version of
either the CQL IG or CRMI, once the pattern profiles themselves have
stabilized.

#### Justification

The CRMI operation pattern profiles are still maturing, and several of
them (notably `Artifact Endpoint Configurable` and `Artifact Version
Bindable`) are themselves being refined ticket-by-ticket. Locking
*Using CQL With FHIR* to conform to a moving target risks repeated
breaking changes to the CQL operations every time CRMI tightens a
pattern. A purely informative cross-reference preserves the design
intent without coupling the two specifications' release cycles.

---

### Recommendation

**Recommended disposition:** A

The CRMI operation pattern profiles exist precisely to standardize the
contract for knowledge-artifact operations, and *Using CQL With FHIR* is
one of their primary intended consumers. Declining (Disposition C) leaves
implementers to reverse-engineer the alignment, and the derived-operation
approach (Disposition B) doubles the surface area implementers must
support without a clear migration story for the original operations.
