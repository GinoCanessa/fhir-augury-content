## Summary

The reporter (Ignacio Jauregui) is exercising the CPG IG example
ActivityDefinition `activity-example-recordinference-ad` (canonical
`http://hl7.org/fhir/uv/cpg/ActivityDefinition/activity-example-recordinference-ad|2.0.0`)
through the `ActivityDefinition/$apply` operation against the CQF Ruler.
Instead of a clean Task being produced, `$apply` returns a Task containing an
`OperationOutcome` with two CQL evaluation errors:

1. `DynamicValue resolution for path input.type encountered exception: library expression loaded, but had errors: Could not resolve identifier code in the current library.`
2. `DynamicValue resolution for path input.value encountered exception: library expression loaded, but had errors: Expected an expression of type 'id', but found an expression of type 'System.String'.`

The reporter notes that the analogous instance shipped in the Connectathon-38
branch of [`cqframework/pd-apply`](https://github.com/cqframework/pd-apply/blob/connectathon-38/connectathon/output/ActivityDefinition-RecordInferenceActivity.json)
also fails to evaluate. The implication is that the canonical "record
inference" example published in the Clinical Guidelines (FHIR) IG is not a
working, conforming example — its dynamic values do not type-check against the
target Task profile (`cpg-recordinferencetask`) and the bound CQL.

This ticket overlaps strongly with the follow-up [FHIR-49790 — Record Inference
Activity Example Correction](https://jira.hl7.org/browse/FHIR-49790), filed by
Taylor Le on 2025-02-12 with a proposed corrected `ActivityDefinition`
(setting `Task.contained` to a CQL-built Observation, then pointing
`Task.input[0].valueReference` at `#inference`), an explicit
`cpg-caseFeatureType=inferred` extension on the inferred Observation, and a
note that the matching `Task-request-inf-scenario2`,
`Observation-inf-scenario4`, and `Task-event-inf-scenario4` test instances
should be updated for traceability. PR
[HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51)
is the in-flight correction for that work.

## Related Jira Tickets

### [FHIR-49790: Record Inference Activity Example Correction](https://jira.hl7.org/browse/FHIR-49790)

Filed by Taylor Le on 2025-02-12 (Triaged, Technical Correction, Clinical
Decision Support, Clinical Guidelines (FHIR)) directly in response to the
Zulip discussion that this ticket spawned. It states that the current record
inference activity example sets `Task.input.value` via a `cql-expression`
whose return is an `Observation` (an invalid datatype for that element);
instead, the example should set a `Task.contained` Observation and reference
it from `Task.input[0].valueReference`. It also calls for adding the
`cpg-caseFeatureType=inferred` extension to the inferred Observation per the
CPG methodology, and for updating the `Task-request-inf-scenario2`,
`Observation-inf-scenario4`, and `Task-event-inf-scenario4` test instances to
match. A full proposed corrected ActivityDefinition JSON is included, with
three `dynamicValue` entries: (1) build a contained Observation via CQL,
(2) set `input[0].type = %resource.code`, (3) set `input[0].valueReference =
#inference`. PR
[HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51)
is referenced as the implementation. This ticket is essentially the
"correction" half of the bug FHIR-49380 reports.

Reporter: Taylor Le. Assignee: Unassigned. Linked PR author: Bryn Rhodes
(per Zulip).

## Related Zulip Discussions

### [#cql > Returning Observation after $apply and CQL execution](https://chat.fhir.org/#narrow/stream/cql/topic/Returning%20Observation%20after%20%24apply%20and%20CQL%20execution)

24-message thread (Jan 23 – Feb 18, 2025) directly motivating this ticket.
Ignacio opens by trying to use an ActivityDefinition + a CQL Library to
generate an Observation and surface it on a Task via `$apply`, and reports
that both CQF Ruler and "reason framework" fail with library-resolution
errors. Brenin Rhodes initially clarifies that the R4
`request-resource-types` value set does not let an ActivityDefinition produce
an Observation directly, and recommends the SDC pattern (PlanDefinition →
Questionnaire → QR → Observation). Ignacio explains the actual use case is a
CDS that *computes* an Observation (e.g., BMI from existing weight/height) and
wraps it in a Task for the user to accept — i.e., the
`cpg-recordinferencetask` workflow. Taylor Le agrees that the published CPG
record-inference examples are misleading/incorrect (returning a new
Observation directly into `task.input.value`/`valueReference` is invalid) and
commits to a CPG WG correction. Brenin then posts a revised
ActivityDefinition that uses three `dynamicValue` entries — `contained` (CQL
returns the new Observation), `input[0].type`, and
`input[0].valueReference = #inference` — and reports it works against HAPI
JPA Server Starter (master) with the CR module enabled. Ignacio confirms it
fails on CQF Ruler and on default HAPI, but works on HAPI when started with
the alternate `cds.application.yaml` profile config. Taylor follows up with
[FHIR-49790](https://jira.hl7.org/browse/FHIR-49790) and
[HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51).
A late sub-thread (Forest, Brenin) walks Ignacio through correctly packaging
the inline CQL into a Library (base64-encoded `Library.content`, and matching
`Library.name` to the CQL library identifier per the CQL IG conformance
rules). Consensus: the published example is wrong, the recipe Brenin
demonstrated is the correct one, CQF Ruler is not the recommended runtime,
and the WG should land the corrected example via the `cqf-recommendations`
PR.

Participants: Ignacio Jauregui (reporter), Brenin Rhodes, Taylor Le, Derek
Ritz, Forest.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-49380 as a duplicate of (or a forcing-function for)
[FHIR-49790](https://jira.hl7.org/browse/FHIR-49790) and resolve it by landing
the corrected `activity-example-recordinference-ad` example already proposed
in [HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51).
Concretely, in the next CPG IG (Clinical Guidelines FHIR) publication:

1. Replace the published `ActivityDefinition/activity-example-recordinference-ad`
   (currently version `2.0.0`) with the corrected JSON drafted in FHIR-49790:
   - `dynamicValue[0].path = contained` with a `text/cql-expression` that
     constructs a `FHIR.Observation` (id `inference`, status `final`,
     `cpg-caseFeatureType=inferred` extension, code `Inference`,
     `subject = Patient`, `issued = Now()`).
   - `dynamicValue[1].path = input[0].type` with expression `%resource.code`.
   - `dynamicValue[2].path = input[0].valueReference` with expression
     `FHIR.Reference { reference: '#inference' }`.
2. Update the matching test/scenario instances called out in FHIR-49790 so
   they continue to round-trip cleanly:
   `Task-request-inf-scenario2`, `Observation-inf-scenario4`,
   `Task-event-inf-scenario4` — each updated to carry the
   `cpg-caseFeatureType=inferred` extension on the inferred Observation.
3. Add a short "Running this example" note under the record-inference
   methodology section pointing implementers at HAPI JPA Server Starter
   master with the CR module enabled and the alternate `cds.application.yaml`
   configuration (the only configuration the WG members confirmed
   reproduces a clean apply); explicitly note that CQF Ruler in the
   reporter's configuration does not currently execute this example.
4. Close FHIR-49380 as resolved-by FHIR-49790 once #51 is merged and the IG
   is republished.

#### Justification

This is what the ticket actually evidences: the published example is not a
conforming, executable example. The Zulip thread reaches clear WG consensus
on (a) the existing example being incorrect, (b) the corrected `dynamicValue`
shape, and (c) `cpg-caseFeatureType` being the right traceability hook.
FHIR-49790 already captures the fix and the PR is open against
`HL7/cqf-recommendations`, so the marginal cost of accepting is just merging
and republishing — there is no design question still in dispute. Closing
FHIR-49380 as a duplicate keeps the bug history visible to future
implementers searching for the original error strings.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the example correction as in Disposition A, but additionally
strengthen the published artifacts so this class of error becomes
machine-detectable rather than discovered at runtime by implementers:

1. Land the corrected `ActivityDefinition` and Library content from
   [HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51)
   as in Disposition A.
2. Add a CPG-IG-side validation/QA step (FHIR Validator + CQL translator
   invocation as part of the IG build) that mechanically `$apply`s every
   published example ActivityDefinition / PlanDefinition against its
   sample input bundle and fails the build if the resulting Task carries an
   `OperationOutcome` with severity ≥ `error`. This would be wired into
   whatever build pipeline `HL7/cqf-recommendations` uses today.
3. Tighten `StructureDefinition/cpg-recordinferencetask` (or its companion
   Library/StructureDefinition material) so that an
   `ActivityDefinition.dynamicValue` whose `expression` returns a value of
   the wrong CQL type for `Task.input.value[x]` is flagged at publish time
   — for example, by documenting in the profile narrative that
   `input[0].valueReference` MUST be sourced from a contained-resource
   reference and that `input[0].type` MUST be sourced from `%resource.code`,
   then asserting that pattern on the example.
4. Add a "Running CPG examples" section to the IG that names the supported
   reference runtime (HAPI JPA Server Starter + CR + `cds.application.yaml`)
   and explicitly scopes out other engines, to set implementer
   expectations.

#### Justification

Disposition A fixes the immediate ballot defect. It does not prevent the
next implementer from filing the same bug against a different example —
which is plausible given that the Zulip thread also surfaced the
Connectathon-38 `pd-apply` instance failing for the same reason, and
Taylor explicitly flagged "these examples are misleading" in the plural.
Wiring `$apply`-as-CI for every CPG example raises the cost of writing a
broken example in the IG to ~zero and would have caught FHIR-49380
before publication. The trade-off is real: adding a CQL execution step to
the IG build is non-trivial infrastructure work for the CPG WG, and may
need engine-version pinning to be reproducible. If the WG cannot afford
that work in this cycle, they can take A now and B as a follow-up Jira.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-49380 as already-resolved-elsewhere: the substantive defect is
captured in [FHIR-49790](https://jira.hl7.org/browse/FHIR-49790) with a
proposed correction and an open PR
([HL7/cqf-recommendations#51](https://github.com/HL7/cqf-recommendations/pull/51)),
so FHIR-49380 itself need not produce any spec/IG action and can be closed
as "Duplicate" pointing at FHIR-49790.

#### Justification

Defensible only as a workflow tidy-up. The two tickets are about the same
defect; the technical-correction ticket is the more actionable of the pair
and already has an implementation in flight. Closing the bug report ticket
as a duplicate avoids the WG processing the same fix twice. The risk is
that closing FHIR-49380 without a visible cross-link from the corrected
example back to the original error strings makes it harder for future
implementers to find the fix when they hit the same `DynamicValue
resolution for path input.type encountered exception: ... Could not
resolve identifier code` message on a search engine — so any "Decline as
duplicate" should still link FHIR-49790 prominently in the resolution
note.

---

### Recommendation

**Recommended disposition:** A

The reporter has accurately diagnosed a published, "shareable / computable
/ publishable" example in the Clinical Guidelines (FHIR) IG that does not
in fact apply cleanly against any of the engines an implementer is likely
to try first. The CPG WG has already done the hard part — Brenin posted
the correct `dynamicValue` recipe in the Zulip thread, Taylor opened
FHIR-49790 with a fully-drafted corrected `ActivityDefinition`, and PR
HL7/cqf-recommendations#51 is in flight. The only remaining decision is
whether to land that fix and close out FHIR-49380, which is exactly
Disposition A. Disposition B (CI-validate every CPG example via `$apply`)
is the right long-term answer and should become a follow-up CPG ticket,
but it should not block the immediate correction. Disposition C
(close-as-duplicate) is administratively reasonable but loses the bug
trail that future implementers will use to find the corrected example, so
A is the cleaner outcome: keep FHIR-49380 open just long enough to land
the FHIR-49790 fix, then close it with a pointer to the corrected example
and to the WG-confirmed runtime configuration (HAPI JPA Server Starter
master + CR module + `cds.application.yaml`).
