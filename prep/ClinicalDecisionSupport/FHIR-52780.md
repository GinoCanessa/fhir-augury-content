## Summary

The ticket reports an inconsistency between an example and the formal
grammar in the **Clinical Quality Language (CQL) FHIR** specification's
section on defining contexts. The example shows a class-type
declaration that declares a relationship to the `Patient` context with:

```
related to Patient by { subject }
```

The corresponding `contextRelationship` rule in the CQL grammar
(`cql.g4`), however, requires the literal keyword `context` between the
qualified identifier and `by`:

```
'related' 'to' qualifiedIdentifier 'context' 'by' '{' identifier (',' identifier)* '}'
```

So either the example is missing the `context` keyword (`related to
Patient context by { subject }`) or the grammar should be amended to
drop the keyword so the documented example parses. The reporter
(spec editor Bryn Rhodes) flags both options. A community contributor
has already submitted [HL7/cql#89](https://github.com/HL7/cql/pull/89)
which takes the second option (remove `'context'` from the grammar), so
the documented form `related to Patient by { subject }` becomes
syntactically valid.

There are no other Jira cross-references and no Zulip discussion of
this ticket key was found.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the *example only* in `HL7/cql:input/pages/03-developersguide.md`
(Defining Contexts subsection, currently around line 1822) so it
matches the existing grammar:

```
related to Patient context by { subject }
```

Leave `input/images/cql/cql.g4` unchanged (the
`contextRelationship` rule keeps the literal `'context'` token), and do
not regenerate downstream parser artefacts. Optionally add a sentence
in the surrounding narrative explaining that `context` is required
because `Patient` here refers to a context name, not a type name.

#### Justification

The author's literal first request is to align the example with the
grammar. Keeping the `context` keyword preserves an explicit syntactic
marker that disambiguates a context name from a (qualified) type
identifier, which is consistent with the rest of CQL's preference for
explicit, self-describing constructs (e.g., `define context`,
`context Patient`, `define type ... extends ...`). It is also the
zero-impact change for downstream tooling: no `.g4` edit means no
parser/translator regeneration, no railroad-diagram refresh, and no
change to any artefact already published from `cql.g4`. The "Defining
Contexts" feature is marked trial-use (CQL 2.0), so a small example fix
during the trial period is appropriate and low risk.

---

### Disposition B: Alternative Approach — Drop `'context'` from the grammar (the PR)

#### Proposal

Adopt the change already proposed in
[HL7/cql#89](https://github.com/HL7/cql/pull/89): in
`HL7/cql:input/images/cql/cql.g4`, change the `contextRelationship`
rule from

```
: 'related' 'to' qualifiedIdentifier 'context' 'by' '{' identifier (',' identifier)* '}'
```

to

```
: 'related' 'to' qualifiedIdentifier 'by' '{' identifier (',' identifier)* '}'
```

Pair this with:

1. Re-running the railroad-diagram / syntax-diagram generator so the
   rendered grammar artefacts under `docs/rrd/` (and any
   `19-l-cqlsyntaxdiagrams.html` content referenced from
   `03-developersguide.md` line ~1805) match.
2. Leaving the existing example in `03-developersguide.md` line ~1822
   (`related to Patient by { subject }`) **unchanged** — it now parses.
3. A short note in the changelog (`v2.0-changelog.md` already mentions
   the `related to` clause; extend that entry, or add a new one) noting
   the simplification.
4. Coordinating with downstream CQL translator/parser projects (e.g.,
   `cqframework/clinical_quality_language`) that regenerate from
   `cql.g4` so they pick up the relaxed rule on their next release.

#### Justification

The reporter offers this as the explicit alternative, and a community
PR is already in place implementing it. The `contextRelationship`
production sits inside `define type`, where the left-hand operand is
syntactically distinguishable from a type by the surrounding `related
to ... by { ... }` skeleton, so the `context` keyword is arguably
redundant; removing it makes the language a touch more concise and
matches what most readers would naturally write (the reporter wrote it
that way in the example, and a contributor wrote a PR for it without
prompting — both are evidence that the keyword is unintuitive). Because
"Defining Contexts" is trial-use, a minor relaxation of the grammar is
permissible. The trade-off vs. Disposition A is a larger blast radius:
parser regeneration in `HL7/cql` and any downstream translator that
consumes `cql.g4`.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Mod" → no spec change, on the
grounds that the example is informative and a careful reader can
reconcile it against the railroad diagram / formal grammar in
`08-a-cqlsyntax.md`.

#### Justification

This is defensible only on the narrowest reading — the inconsistency
is in narrative example text rather than in any normative artefact —
but it is weak. The whole point of an example in the Developer's
Guide is to be copy-pasteable; a documented snippet that does not
parse against the published grammar is a real defect for any
implementer following the guide. Decline is therefore not a strong
position here.

---

### Recommendation

**Recommended disposition:** B — adopt PR
[HL7/cql#89](https://github.com/HL7/cql/pull/89) (drop `'context'`
from the `contextRelationship` rule).

The ticket reporter (Bryn Rhodes, the spec editor) wrote the example
without the `context` keyword, and an external contributor independently
proposed the same simplification — that is two pieces of evidence that
the keyword is non-obvious and adds friction without disambiguation
value, since the surrounding `related to ... by { ... }` skeleton
already pins down the production. The change is a single-line grammar
edit, has a ready PR, and the "Defining Contexts" feature is trial-use
in CQL 2.0, which is exactly the time to absorb this kind of refinement
before stabilization. Disposition A is the safe fallback if CDS judges
the parser-regeneration / downstream-translator coordination cost too
high to take during this cycle. Disposition C should not be selected;
shipping a non-parseable example in the Developer's Guide is a defect
the workgroup should fix one way or the other.
