## Summary

The reporter (Isaac Vetter) suggests that the Clinical Decision Support
"module" — i.e., the term used to label the CDS work group's slice of
the FHIR Core specification (the page commonly rendered as the
"Clinical Decision Support Module") — be renamed to "topic". The
stated rationale is twofold: (a) "topic" is, in his view, the same
term used by [CDS Hooks](https://cds-hooks.org/), so adopting it would
align FHIR Core's terminology with the sibling CDS Hooks
specification; and (b) the word "module" is unintuitive on its own.
This is a pure naming / editorial change request scoped to the CDS
work group's portion of FHIR Core; the underlying content, resources,
and conformance behavior would not be affected. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were found that would
broaden or refine this scope.

## Proposed Dispositions

The proposals below describe what each disposition would entail for
the FHIR Core publication, even though no specific repository was
surfaced via cross-references. The CDS work group should treat the
named locations as illustrative anchors — the actual edits live in
whichever FHIR Core publication source repo is in use for the
in-flight cycle (commonly `HL7/fhir`).

### Disposition A: Accept As Requested

#### Proposal

Rename the Clinical Decision Support **module** to **topic**
throughout the CDS slice of FHIR Core. Concretely, this means:

- Update the page title and heading on the CDS module index page
  (today rendered with a title along the lines of "Clinical Decision
  Support Module") to read "Clinical Decision Support Topic", and
  rename references to the page in the FHIR table of contents and
  module index.
- Replace the word "module" with "topic" in the CDS work group's
  intro / module-level prose (e.g., on the CDS landing page and in
  the introductory paragraphs that frame the page as "this module
  contains…").
- Adjust any cross-page links and breadcrumb labels in the FHIR Core
  spec that name this page as a "module" so they refer to it as a
  "topic" instead.
- Leave the page slug / URL unchanged where possible, or add HTTP
  redirects if a slug rename is unavoidable, to preserve external
  links.
- Keep the change scoped to the CDS portion of FHIR Core — do not
  rename the global FHIR concept of "module" used by other work
  groups under this ticket.

#### Justification

This implements the request literally, and it is a low-cost editorial
change. Aligning the CDS section's framing word with the term used by
CDS Hooks (where the registry / catalog of available services is
talked about in topical terms) makes the relationship between the two
specifications easier to explain to newcomers, and removes a
self-described "unintuitive" label. Because the change is purely
nominal — no resource, profile, operation, value-set, or invariant is
touched — there is no normative impact and no implementer-visible
behavior change.

---

### Disposition B: Alternative Approach

#### Proposal

Do not rename the page, but **clarify the term** instead. Keep
"Clinical Decision Support Module" as the page title, and add a short
introductory note (one or two sentences near the top of the CDS
module page) that explains what "module" means in the FHIR Core
publication structure (i.e., the work group's curated slice of the
specification), and that explicitly relates it to the way CDS Hooks
talks about topics / catalogs. Optionally add a glossary entry for
"module" on the FHIR definitions / glossary page so the term is
defined consistently across all work groups' module pages.

#### Justification

The word "module" is used by **every** FHIR Core work group's page
(e.g., the financial module, the administration module, the clinical
module, the diagnostics module, the workflow module). Renaming only
the CDS page to "topic" would make CDS the odd one out in the FHIR
Core table of contents and would create a navigation inconsistency
that is itself unintuitive. Adding a clarification keeps the
publication-wide terminology coherent, addresses the "unintuitive"
concern with prose rather than a rename, and still creates the
explicit bridge to CDS Hooks that the reporter is asking for. It is
also reversible and cheap — a paragraph edit and possibly a glossary
entry.

---

### Disposition C: Decline

#### Proposal

Decline the rename. Keep "Clinical Decision Support Module" as the
page title and the established term throughout FHIR Core. Reason
category: **already addressed by existing publication-wide
convention** (every work group's page is a "module"), and the
proposed alignment with CDS Hooks is **not actually a 1:1 mapping** —
CDS Hooks does not use "topic" as a structural label for its
specification organization, so the renaming would not create the
terminology alignment the ticket implies.

#### Justification

The FHIR Core publication uses "module" consistently to denote the
work-group-curated section of the spec; renaming a single work
group's page breaks that consistency for every reader of the FHIR
Core table of contents. The CDS Hooks specification's own structural
units are called "services" and "hooks" (with "topics" appearing only
informally in some catalogs), so the asserted alignment with CDS
Hooks does not hold up: renaming "module" to "topic" would not
actually mirror CDS Hooks' structural vocabulary. Declining preserves
publication-wide consistency and avoids creating a precedent for
per-work-group renames of the "module" concept.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern — that "module" is unintuitive to
readers, especially those arriving from the CDS Hooks side — is
legitimate and worth addressing. However, "module" is a
publication-wide structural label across every FHIR Core work group,
so a one-off rename for CDS only (Disposition A) would create a
table-of-contents inconsistency that is at least as unintuitive as
the original term. Outright declining (Disposition C) leaves the
reporter's confusion unaddressed. Disposition B threads the needle:
it adds a short, targeted clarification on the CDS module page (and
optionally a glossary entry) that explains what "module" means in
FHIR Core and explicitly bridges to CDS Hooks terminology, at very
low editorial cost and with no risk of breaking external links or
diverging from the rest of FHIR Core. If the work group ultimately
wants the rename across the whole publication, that should be raised
as a separate, FMG-level proposal rather than handled inside a
single CDS-scoped ticket.
