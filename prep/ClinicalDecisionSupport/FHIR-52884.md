## Summary

The reporter requests reconsideration of the `Slice` function added to
Clinical Quality Language (CQL). They find the *exclusive* `endIndex`
semantics ("ending just before `endIndex`") counterintuitive and
demonstrate that `Slice({1, 2, 3, 4, 5}, 1, 3)` returns `{2, 3}` rather
than the `{2, 3, 4}` they expected. They highlight that extracting a
single element at index 1 requires `Slice(..., 1, 2)`, which they find
awkward. The ticket is filed as a Change Request against the FHIR
binding of CQL and asks the working group to consider whether the
semantics, signature, or documentation should be revised. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced by FhirAugury for this ticket.

## Details

**Description:**

> The definition of `endIndex` in the new `Slice` function feels
> unintuitive, as it is defined to be *exclusive*.
>
> According to the specification, the Slice operator returns a portion
> of the elements in a list, beginning at `startIndex` and ending just
> **before** `endIndex`. The function signature is:
>
> `Slice(argument List<T>, startIndex Integer, endIndex Integer)`
>
> Given this definition, I would expect the following call:
>
> `Slice({1, 2, 3, 4, 5}, 1, 3)`
>
> to return:
>
> `{2, 3, 4}  // starting from index 1 and including index 3`
>
> However, it actually returns:
>
> `{2, 3}  // starting from index 1 and ending just before index 3`
>
> This "just before" behavior feels counterintuitive, especially for use
> cases where I want to extract a single value. For example, to extract
> only the element at index 1, I need to call:
>
> `Slice({1, 2, 3, 4, 5}, 1, 2)  // returns {2}`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the semantics of the `Slice(argument List<T>, startIndex Integer,
endIndex Integer)` function in the CQL specification so that `endIndex`
is **inclusive** — i.e., the returned list contains elements from
position `startIndex` through and including position `endIndex`. Update
the normative description in the CQL Language Specification (Section 9
List Operators / Slice), the worked examples, and any reference
implementation translation rules so that
`Slice({1, 2, 3, 4, 5}, 1, 3) = {2, 3, 4}`. Add a clear non-normative
note that this differs from the half-open `[start, end)` convention
used by many host languages (e.g., Python, JavaScript Array.slice).

#### Justification

The reporter's reading is a real usability concern: a single-element
extraction requires `Slice(list, i, i + 1)`, which is unintuitive for
authors who do not come from a Python/JavaScript background. CQL
authors are typically clinicians and informaticists, not general-purpose
programmers, so favoring intuitive readability over alignment with
host-language conventions has precedent in CQL design.

#### Risks / Caveats

This is a **breaking change** to a published function. Any deployed CQL
logic, knowledge artifacts, or measure expressions that already use
`Slice` would silently shift behavior (one extra element returned),
which is the most dangerous class of breakage in clinical decision
support. The CDS / CQL work group would need to gate this on a major
version bump and provide migration tooling or a hard error for old
usages.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing exclusive-`endIndex` semantics of `Slice` (preserving
alignment with Python's `list[start:stop]`, JavaScript
`Array.prototype.slice`, and similar half-open conventions), but address
the underlying usability problem in two non-breaking ways:

1. **Documentation:** Strengthen the CQL spec's Slice section with an
   explicit "Note on indexing convention" callout that names the
   half-open `[startIndex, endIndex)` semantics, gives the
   `Slice(list, i, i + 1)` single-element pattern, and contrasts it
   with `IndexOf` / direct indexer access for single-element use
   cases.
2. **Convenience operator(s):** Introduce one or more thin convenience
   functions that cover the common cases without re-litigating Slice
   semantics — e.g., `Take(argument List<T>, count Integer)`,
   `Skip(argument List<T>, count Integer)`, and/or
   `Sublist(argument List<T>, startIndex Integer, length Integer)`.
   These are additive and unambiguous.

#### Justification

The half-open convention has well-known ergonomic advantages: the
length of the slice is `endIndex - startIndex`; concatenating
`Slice(l, 0, k)` and `Slice(l, k, Count(l))` reconstructs the list with
no off-by-one; and it matches the behavior CQL implementations
inherited from common host languages. Changing the semantics of an
already-published function in a clinical specification is unusually
costly because deployed measures and CDS rules cannot easily be
regression-tested for a silent off-by-one. Documentation plus
additive helpers gets the reporter's ergonomics without breaking any
existing artifacts.

---

### Disposition C: Decline

#### Proposal

Decline the request and close as "Working as Designed." Confirm in the
resolution that `Slice(argument List<T>, startIndex Integer,
endIndex Integer)` is intentionally defined with an exclusive
`endIndex` (half-open `[startIndex, endIndex)`) for consistency with
the conventions used by the host languages CQL is most commonly
implemented in, and that the published behavior will not be changed.

#### Justification

The reported behavior matches the published normative definition, and
the exclusive-end convention is the dominant industry standard for
slicing operators. Changing it would break deployed CQL artifacts in a
silent, hard-to-detect way, which is unacceptable in a clinical
context. The reporter's underlying single-element use case is already
served by direct indexer syntax and by `IndexOf`/`First`/`Last` style
operators, so there is no gap in expressive power to justify a breaking
change.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B preserves the existing half-open semantics (avoiding
silent breakage of deployed CQL logic and CDS artifacts, which would be
unsafe in a clinical setting) while taking the reporter's usability
feedback seriously. A clearer normative note plus additive helpers
(`Take`, `Skip`, and/or `Sublist(start, length)`) eliminates the
ergonomic complaint at its source for the next reporter who hits this,
without requiring the work group to choose between "break everything"
(A) and "tell the reporter no" (C). It is also the disposition most
likely to converge quickly in ballot, since neither side has to
concede on the underlying semantics question.
