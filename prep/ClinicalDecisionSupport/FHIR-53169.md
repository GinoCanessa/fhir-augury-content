## Summary

The ticket asks the Common CQL Assets for FHIR (US-Based) IG to document a
standardized way to pass DTR launch-context parameters into CQL Libraries
that back Questionnaires used in the Prior Authorization (PA) use case.
Specifically, it proposes that Library declarations consistently use two
parameter names — `Coverage` (typed `FHIR.Coverage`) and `Request`
(typed as a `Choice` of `FHIR.NutritionOrder`, `FHIR.ServiceRequest`,
`FHIR.DeviceRequest`, `FHIR.MedicationRequest`, `FHIR.VisionPrescription`,
`FHIR.Appointment`, `FHIR.Encounter`) — so that DTR launch context codes
map cleanly onto the parameters the pre-population logic expects.

The request is a direct follow-on to the cross-referenced ticket
[FHIR-53168](https://jira.hl7.org/browse/FHIR-53168) (US Da Vinci DTR),
which proposed defining `coverage` and `request` launch-context codes for
SDC forms in DTR. FHIR-53168 was resolved **Not Persuasive** on the
grounds that `Coverage` and `Request` resources are already inputs to
`$questionnaire-package` and that any data the form needs from them
should be pre-populated by the payer into the returned
`QuestionnaireResponse` rather than re-passed via launch context.
FHIR-53169 takes the complementary editorial path inside the US CQL IG:
rather than introduce DTR-side launch-context codes, it documents the
parameter-naming and -typing convention CQL Libraries should use so the
PA flow is unambiguous and consistent across implementers. The ticket
has been resolved **Persuasive**, with an associated draft PR
[HL7/us-cql-ig#40](https://github.com/HL7/us-cql-ig/pull/40) and a
preview branch under `use-case-pa.html`. Reporter comments also flag
follow-up work: refresh the example artifacts, update the Postman test
collections, and establish a unit-testing pattern that does **not**
require test-only scaffolding (e.g., no `RequestId` parameters) bleeding
into downstream infrastructure.

## Linked Jira Tickets

### [FHIR-53168: Define launch context codes for DTR-specific parameters](https://jira.hl7.org/browse/FHIR-53168)

The originating DTR-side proposal: define `coverage` and `request`
launch-context codes (for the same set of `Request`-typed resources) and
specify how they would render as `launchContext` extensions on a
DTR-launched Questionnaire. Resolved **Not Persuasive** — DTR concluded
that the information is already available to the form via
`$questionnaire-package` inputs and the payer-populated initial
`QuestionnaireResponse` (with an extension indicating which Request(s)
the response pertains to), so no new launch-context codes are needed.
FHIR-53169 is the explicit complementary action: instead of new DTR
codes, standardize the **CQL Library parameter contract** the IG
recommends so implementations remain consistent.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/us-cql-ig`, land [PR #40](https://github.com/HL7/us-cql-ig/pull/40)
substantially as drafted. Concretely:

1. In `input/pagecontent/use-case-pa.md` (the "CQL processing to
   pre-populate the QuestionnaireResponse" section), document the
   standard CQL Library parameter contract for PA-flow Questionnaires:

   ```cql
   parameter "Coverage" FHIR.Coverage
   parameter "Request" Choice<FHIR.NutritionOrder, FHIR.ServiceRequest,
                              FHIR.DeviceRequest, FHIR.MedicationRequest,
                              FHIR.VisionPrescription, FHIR.Appointment,
                              FHIR.Encounter>
   ```

   Explain how DTR launch context (and/or the payer-populated initial
   `QuestionnaireResponse`, per the FHIR-53168 resolution) maps onto
   these parameters, and prescribe these names/types as the recommended
   convention for IG-published examples.
2. Update the example CQL Libraries under `input/cql/` to use exactly
   these parameter names and types.
3. Refresh the corresponding Questionnaire / QuestionnairePackage
   example bundles (`bundles/questionnaire/`,
   `bundles/questionnare-package-response/`) and the Postman test
   collection(s) under `postman/` so they exercise the new parameter
   contract end-to-end.
4. Establish a unit-test pattern in the IG that exercises these
   parameters **without** introducing test-only scaffolding parameters
   (no `RequestId` etc.) that downstream infrastructure would inherit.

#### Justification

This is the path the work group has already endorsed: the ticket is
resolved **Persuasive**, the editorial change is contained to a
single IG (`HL7/us-cql-ig`), and it cleanly complements the
DTR-side **Not Persuasive** resolution of FHIR-53168 — together they
say "DTR doesn't need new launch-context codes; the US CQL IG just
documents the parameter convention." Standardizing on `Coverage` and
`Request` (with the explicit `Choice` of order-bearing resource
types) gives implementers a single, predictable contract for
pre-population logic and removes the ambiguity that motivated the
original DTR proposal. The reporter's twin caveats — refresh
examples/Postman *and* avoid leaking test-only scaffolding into the
parameter shape — are both addressable inside the same PR and protect
downstream tooling.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the parameter-naming guidance, but tighten its presentation in
two ways before merging:

1. Publish the contract as a **named, citable convention** in the IG
   (e.g., a dedicated subsection or a small CodeSystem of
   well-known CQL parameter identifiers) rather than only as inline
   prose in the PA use-case page, so other CQL-on-FHIR IGs can
   reference it normatively.
2. Express `Request` as a typed reference list (or a small,
   IG-defined `RequestChoice` alias) and explicitly cite the
   QuestionnaireResponse extension noted in FHIR-53168's
   resolution that identifies which Request(s) the response
   pertains to. Worked examples in `bundles/` would show both
   the launch-context path *and* the payer-populated
   QuestionnaireResponse path so implementers can see the two are
   equivalent inputs to the same parameter.

#### Justification

The substance of the ticket is correct, but its current framing is
PA-specific while the underlying convention (well-known `Coverage`
and `Request` parameter names typed against a defined Choice) is
useful anywhere a CQL Library is launched from a Questionnaire.
Promoting the convention to a named artifact improves
discoverability for non-PA consumers (other Da Vinci IGs, ePA-Lite,
non-payer CQL libraries) and avoids each IG re-deriving the same
parameter list. Aligning the worked examples with the FHIR-53168
resolution (payer-populated initial QuestionnaireResponse) keeps the
two CDS-side resolutions internally consistent and reduces the risk
that implementers conclude they need DTR launch-context codes after
all.

---

### Disposition C: Decline

#### Proposal

Decline to add any new prescriptive guidance to the IG. Treat the
parameter naming as an implementation detail of each Library author
and rely on the FHIR-53168 resolution alone — i.e., information
needed by the form is provided through `$questionnaire-package`
inputs and the payer-populated initial `QuestionnaireResponse`, and
CQL parameter names are out of scope for the IG to standardize.

#### Justification

A defensible "Decline" rests on the observation that FHIR-53168
already tells implementers where the data lives (in the
`QuestionnairePackage` inputs / returned `QuestionnaireResponse`)
and that prescribing CQL parameter names risks ossifying a
convention before broader CQL-on-FHIR IGs (outside PA) have
weighed in. Locking the `Request` Choice to a fixed list of
resource types could also create churn the next time another
order-bearing resource (e.g., a future `SupplyRequest` use case)
needs to participate.

---

### Recommendation

**Recommended disposition: A — with the small generalization from B.**

The work group has already adjudicated this **Persuasive**, a draft
PR exists, and the change is editorially confined to a single IG
that the CDS WG owns. Disposition C would re-open a question the WG
has effectively closed across the FHIR-53168 / FHIR-53169 pair and
would leave implementers without the very convention the ticket was
filed to establish. Disposition A delivers exactly that. The one
non-trivial risk Disposition B flags — that `Coverage` / `Request`
are useful well beyond PA and deserve a more durable home than the
PA use-case narrative — is cheap to mitigate inside the same PR by
giving the convention its own subsection (or a small named artifact)
in the IG, plus a paragraph that points at the FHIR-53168 mechanism
(payer-populated initial QuestionnaireResponse + extension) as the
canonical alternative input path. The reporter's two caveats
(refresh examples and Postman collections; do not introduce
test-only scaffolding parameters) should be enforced as merge gates
on PR #40.
