## Summary

The ticket asks the Clinical Decision Support work group to extend the
"Using CQL With FHIR" IG so that it directly addresses **version management
of `ModelInfo` files** and the **explicit mapping between a FHIR model
version and what CQL implementations can rely on**. The reporter (Bryn
Rhodes) notes that the current discussion is silent on how to handle a
change to a model's version (e.g., DSTU3 → R4 → R6), and that for IGs
that bind "with a FHIR model" there is no clear, enumerated statement of
which FHIR resources/elements are mapped into CQL types and what a CQL
`retrieve` can therefore expect to find. The Section 4 (Conformance)
example only shows a single mapped resource (`FHIR.Observation`) without
saying what else is — or is not — in scope for the model.

The ticket also raises the implementation-side problem of cross-version
behaviour: when an EHR returns a DSTU3 resource and another returns R4,
authors today have no specified mechanism to detect the model version a
retrieve produced, branch on it, or migrate libraries forward when the
binding is upgraded (e.g., to R6). The request is for the IG to make
both the **enumeration of the model contents per FHIR version** and the
**version-evolution rules for `ModelInfo`** explicit and normative.

There are no linked or cross-referenced Jira tickets, GitHub items, or
Confluence pages indexed for FHIR-50990. The most relevant prior
discussion lives in two `#cql` Zulip topics (see below): *ModelInfo
Links* (2024) — where Bryn and Gino Canessa worked through how `ModelInfo`
is generated, how dependencies between `FHIR` / `USCore` / `QICore`
flatten, and whether a per-IG `ModelInfo` should be the unit of
publication — and *Official FHIR Package for ModelInfo?* (2022) — where
implementers hit canonical/version mismatches between the CQF-published
`FHIR-ModelInfo` library and what the IG Publisher / CQF Tooling expects.

## Related Zulip Discussions

### [cql > ModelInfo Links](https://chat.fhir.org/#narrow/stream/cql/topic/ModelInfo%20Links)

A 2024 working thread between Bryn Rhodes and Gino Canessa about where
the canonical sources of `ModelInfo` live (the translator's
`quick/src/main/resources/org/hl7/fhir` tree and the
`cql-lm/schema/model/modelinfo.xsd` schema in
`cqframework/clinical_quality_language`, plus the per-class
configuration in `cqf-tooling`'s `FHIRClassInfoSettings.java`,
`USCoreClassInfoSettings.java`, and `QICoreClassInfoSettings.java`).
The conversation surfaces several issues that are squarely in scope
for FHIR-50990: (a) there is no published source-of-truth for
`primaryCodePath` per FHIR/IG version — it lives in a Java HashMap;
(b) when the same resource (e.g., `Communication`) has different
primary code paths across R4, US Core, and QI Core, the resolution
rule today is "build for one IG and flatten", not a declarative
dependency-aware lookup; (c) Bryn states the long-term direction is
that `ModelInfo` should be **published with each IG** and follow that
IG's dependency list (so the QICore `ModelInfo` would be shipped with
QICore), and (d) `conversionInfo` entries are only present in the
core-FHIR `ModelInfo` files and are inferred for derived IGs, which
Bryn agrees is a gap that should be made explicit input to the
generator. Gino prototyped representing the cross-version type maps as
`ConceptMap` resources (`cql-fhir-types-r4`, etc.) and the per-package
property overrides as `Parameters` (`cql-fhir-properties-us-core`,
etc.) — directly relevant prior art for any disposition that proposes
a normative artefact form. *Participants: Bryn Rhodes, Gino Canessa,
Mark Marciante-LP.*

### [cql > Official FHIR Package for ModelInfo?](https://chat.fhir.org/#narrow/stream/cql/topic/Official%20FHIR%20Package%20for%20ModelInfo%3F)

A 2022 implementer thread that documents the user-facing pain the
ticket is reacting to. Brian Kaney hit a publisher QA error because
his CQL `using FHIR version '4.0.1'` caused the IG Publisher to emit a
related-artifact reference to `http://hl7.org/fhir/Library/FHIR-ModelInfo`
version `4.0.1`, but the only published `FHIR-ModelInfo` lives at
`http://fhir.org/guides/cqf/common/Library/FHIR-ModelInfo` — different
canonical, different versioning. Bryn confirmed the canonical
mismatch is real and that the in-publisher translator was outdated;
the workaround John Silva mentioned ("we changed the CQL `using`
version to 4.0.0") illustrates that there is no specified contract
between the `using FHIR version 'X'` declaration and the published
`ModelInfo` artefact at that version — exactly the gap FHIR-50990
asks the IG to close. *Participants: Brian Kaney, Grahame Grieve, Paul
Denning, Bryn Rhodes, John Silva.*

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new normative section to the *Using CQL With FHIR* IG titled
"ModelInfo Versioning and Model Binding" that does three things:

1. **Bind a CQL `using FHIR version 'X'` to a single, named
   `ModelInfo` artefact.** Specify that conformant `ModelInfo`
   libraries SHALL be published as a FHIR `Library` resource whose
   `useContext` includes the FHIR version and whose canonical URL
   follows a fixed pattern (e.g., `http://hl7.org/fhir/Library/FHIR-ModelInfo|<fhir-version>`),
   and that the IG Publisher / CQF Tooling SHALL resolve the `using`
   declaration to that canonical at that version.
2. **Enumerate model contents per FHIR version.** Replace the
   single-`Observation` example in Section 4 (Conformance) with a
   normative table (or generated appendix) listing every FHIR resource
   that is in scope for the `FHIR` model at each supported FHIR
   version, plus the `primaryCodePath` for each — so a CQL author can
   see exactly what a `[Resource: …]` retrieve is allowed to assume.
3. **Specify version-evolution rules.** Add SHALL-level rules that
   (a) a new FHIR version MUST be accompanied by a new `ModelInfo`
   `Library` (versioned, immutable); (b) breaking changes in the
   underlying FHIR resource (renamed elements, cardinality changes)
   MUST be enumerated in a "ModelInfo migration notes" section per
   version; and (c) cross-version code MAY use a documented pattern
   (e.g., a CQL helper exposing the bound model version) to branch on
   the active model version.

#### Justification

This is what the reporter literally asks for, and it directly fixes
both pain points documented in the Zulip threads. The 2022 *Official
FHIR Package for ModelInfo?* thread shows that the binding from
`using FHIR version 'X'` to a publishable `Library` is currently
under-specified — Bryn explicitly acknowledged the canonical/version
mismatch and that the publisher needed to be updated. The 2024
*ModelInfo Links* thread shows that even Bryn and the tooling
maintainer treat the per-resource configuration (`primaryCodePath`,
`conversionInfo`) as configuration that "ought to be input" rather
than something the IG specifies — making the enumeration normative
closes that gap. Tying the rules to FHIR versioning conventions
(immutable canonical at a version, additive evolution between
versions) is consistent with how FHIR already manages ValueSet,
StructureDefinition, and `ConceptMap` evolution.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than enumerating the entire model in the *Using CQL With FHIR*
IG itself, specify a **machine-readable contract** and require each
FHIR / model-binding IG to publish its own `ModelInfo` against that
contract. Concretely:

1. Define a small set of conformance resources in the IG: a
   `Library` (the `ModelInfo` itself, FHIR-resource-form), a
   `ConceptMap` for the FHIR-type-to-CQL-type mapping (one per
   FHIR version, in the shape Gino prototyped: `cql-fhir-types-r4`,
   `-r5`, etc.), and a `Parameters` resource for per-package
   property overrides (`primaryCodePath`, `conversionInfo`,
   re-export/flattening behaviour) keyed by `packageId` /
   `packageVersion`.
2. Make the *Using CQL With FHIR* IG specify only **how these
   resources are produced, named, and resolved** — the canonical
   URL pattern, the `supersedes` / dependency semantics, and how a
   CQL `using <Model> version 'X'` clause is resolved against the
   published artefacts (per the IG dependency tree, with explicit
   tie-breaking).
3. Publish reference instances for the FHIR core models (R4, R5,
   R6) from this IG, and require IGs that bind to a model
   (US Core, QI Core, etc.) to publish their own `ModelInfo` +
   `Parameters` overrides, with a SHALL on declaring an explicit
   "base" package when an author builds against multiple bound IGs.

#### Justification

This matches the long-term direction Bryn articulated in the 2024
thread ("ideally the `ModelInfo` would be produced with and part of
the IG that it's for, so the QICore `ModelInfo` is published with
QICore"), and it leverages FHIR's own dependency machinery (IG
manifests, package versioning, `ConceptMap` for type mappings)
instead of inventing a parallel system in the IG prose. It also
absorbs Gino's already-prototyped artefact shapes, which means the
proposal is not speculative — the design has been exercised. The
trade-off vs. Disposition A is that the *Using CQL With FHIR* IG
itself stays smaller and stable, but downstream IG editors take on
new publishing obligations; given that QI Core / US Core already
carry `ModelInfo`-like configuration (in `cqf-tooling` Java today),
the net new burden is moving that config from a code repo into the
IG package, which is the right place for it.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive** with the rationale that the existing
mechanism — the CQF-published `FHIRModelInfo` and `FHIRHelpers`
libraries at `http://fhir.org/guides/cqf/common/` plus the per-IG
`ModelInfo` files generated by `cqf-tooling` — already provides a
versioned, package-scoped solution, and that the gap is
documentation/tooling, not a missing IG specification. Direct the
reporter to the existing CQF Common library and the `cqf-tooling`
configuration files, and note that any further version-evolution
rules can be handled as part of the standard CQF release cadence.

#### Justification

Defensible only on the narrow read that the published artefacts
already exist and the failure modes (canonical mismatch, missing
`primaryCodePath` source-of-truth, undocumented dependency
resolution) are tooling bugs rather than spec gaps. The 2022 Zulip
thread is consistent with this framing — the immediate fix Bryn
proposed there was a publisher PR to update the bundled translator,
not an IG change. Declining keeps the IG focused and avoids
committing the work group to enumerating model contents per FHIR
version, which is non-trivial maintenance.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the substance of what FHIR-50990 asks for —
explicit, version-scoped contracts for `ModelInfo` and model
bindings — without committing the *Using CQL With FHIR* IG to
maintaining a hand-written enumeration of every resource at every
FHIR version (which Disposition A would require, and which is where
that IG would silently drift out of date). It also matches the
direction Bryn already endorsed on Zulip ("`ModelInfo` published with
each IG, dependency resolution follows the IG dependency list") and
reuses the resource shapes (`Library`, `ConceptMap`, `Parameters`)
that Gino has already prototyped, so the work group is ratifying
existing prior art rather than designing in the abstract.
Disposition C is too narrow — the 2022 thread shows the canonical /
version contract genuinely is under-specified at the IG level, not
just at the tooling level. Disposition A is acceptable as a
fallback but risks turning the IG into a moving catalogue; the work
group should prefer specifying the *contract* and letting the bound
IGs publish the *content*.
