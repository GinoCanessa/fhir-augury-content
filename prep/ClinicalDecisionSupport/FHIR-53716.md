## Summary

Isaac Vetter is asking the Clinical Decision Support work group to revisit
the FHIR Core security categorization assigned to the `Library` resource.
Today `Library` is classified as **Anonymous** in the per-resource Security
Categorization table on `security.html`, which signals that instances are
not expected to carry sensitive content. The ticket points out that, in
practice, a substantial industry produces and sells knowledge artifacts
encoded as FHIR `Library` resources — prior-authorization rule sets,
medical-necessity criteria, copyrighted clinical guideline content from
academic societies, proprietary CDS algorithms, etc. From a business /
intellectual-property standpoint, those payloads are typically the
opposite of "anonymous" in the data-protection sense: they are
proprietary, copyrighted, or confidential. The reporter argues that the
current classification is the exception rather than the rule for `Library`
and asks the WG to consider moving it to a category that better reflects
its routine sensitivity.

No other Jira tickets were cross-referenced to this issue, no Zulip
threads were linked, and no GitHub PRs/commits/issues are associated.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reclassify `Library` in the FHIR Core Security Categorization table from
**Anonymous** to **Business**. Concretely:

- In `HL7/fhir`, edit the per-resource category metadata for `Library`
  in `source/library/` (the descriptor that the publisher uses to render
  the `security.html` table — typically the `Security Category` cell of
  the resource's `*-spreadsheet.xml` or equivalent metadata file) so
  that it emits `Business` instead of `Anonymous`.
- Refresh the `security.html` Security Categorization table so the
  `Library` row reads `Business`.
- Add a one-paragraph note on `source/security.html` (and a parallel
  cross-reference on `source/library/library.html` /
  `library-introduction.html`) explaining the change: that `Library`
  routinely carries proprietary, copyrighted, or otherwise
  business-sensitive knowledge artifacts (rule sets, guideline content,
  proprietary CDS logic, prior-authorization criteria), and that
  servers and clients should default to treating `Library` resources as
  business-sensitive unless an explicit `Library.meta.security` label
  indicates otherwise.

#### Justification

The `Business` category exists precisely for resources whose payload is
sensitive from a commercial / IP standpoint rather than a patient-privacy
standpoint, which matches the reporter's characterisation. Other
knowledge-asset resources in the same family (`Measure`, `PlanDefinition`,
`ActivityDefinition`) sit in adjacent categories that already acknowledge
business sensitivity, so promoting `Library` aligns it with how the
clinical-reasoning resources are actually deployed. The default-secure
direction also matches general HL7 security guidance: it is safer to
classify a resource as more sensitive and let implementers down-classify
specific instances via `meta.security` than to default to `Anonymous` and
have implementers belatedly discover a leak of proprietary content.

---

### Disposition B: Alternative Approach

#### Proposal

Leave the per-resource Security Categorization for `Library` at its
current value, but make the *instance-level* sensitivity guidance
explicit and authoritative:

- On `source/library/library.html` (or
  `library-introduction.html` / `library-security.html`), add a
  dedicated **Security Considerations for Library** section that
  enumerates the high-IP / business-sensitive use cases the reporter
  lists (prior-auth criteria, copyrighted guidelines, proprietary CDS
  logic) and *requires* implementers to evaluate `Library.meta.security`
  on every instance rather than relying on the resource-level category.
- Add a `SHOULD` statement: when `Library.content` carries
  copyrighted, licensed, or otherwise proprietary material,
  `Library.meta.security` SHALL include an appropriate confidentiality
  / IP-sensitivity label (e.g., `R` / `restricted`, or a
  `v3-Confidentiality` code) and `Library.copyright` /
  `Library.copyrightLabel` SHALL be populated.
- Optionally introduce or recommend a value set / extension that
  signals "knowledge artifact is proprietary" so downstream servers
  can apply consistent access policies.

#### Justification

The per-resource Security Categorization is a coarse, resource-wide
hint; it cannot distinguish a freely redistributable public-health
guideline (`Library` containing CDC opioid CDS released under public
domain) from a proprietary payer rule set. Pushing the discrimination
to the instance level, anchored on `meta.security` and
`copyright`/`copyrightLabel`, is more faithful to how `Library` is
actually used and avoids over-classifying genuinely open content.
This alternative is also lower-risk for downstream IGs and trial-use
implementers that rely on the current `Anonymous` classification in
their security narratives. It does, however, place a heavier burden on
publishers and on implementers' policy engines.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the current `Anonymous` security
categorization for `Library`. Document the rationale in a brief note on
`source/security.html` (or a tracker comment) clarifying that the
per-resource category in FHIR Core describes the *kind of data the
resource model carries*, not the IP / business sensitivity of any
particular instance, and that instance-level sensitivity is already
addressed by `Library.meta.security`, `Library.copyright`,
`Library.copyrightLabel`, and the general Security Labels framework.

#### Justification

The Security Categorization table on `security.html` is intentionally a
coarse classification of *patient-privacy* exposure; `Anonymous` means
"the resource model itself does not require a patient context to be
meaningful," not "the payload is freely shareable." Reclassifying
`Library` as `Business` could mislead implementers into thinking every
`Library` instance must be access-controlled like a payer artifact,
which is not true of the many openly published clinical-reasoning
libraries in HL7 IGs (`HL7/cqf-recommendations`, `HL7/cql-ig`, etc.).
The existing `meta.security` + `copyright` mechanism already gives
publishers a precise way to mark proprietary instances, so a Core
metadata change is unnecessary and would propagate into a large number
of downstream IGs and tooling tables for limited net benefit.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle between the legitimate concern Isaac
raises and the meaning the Security Categorization table actually
carries today. The reporter is correct that a large fraction of real
`Library` instances are proprietary or copyrighted; he is also correct
that defaulting them to `Anonymous` undersells their business
sensitivity. But the per-resource category in `security.html` is a
model-level, patient-privacy hint, not an IP-sensitivity flag, and
flipping it to `Business` (Disposition A) risks over-classifying the
substantial body of openly published CDS / measure libraries while
still not capturing the IP nuance for proprietary instances. Disposition
C (decline) leaves the documentation gap that motivated the ticket
unfilled.

A targeted **Library**-specific Security Considerations section that
calls out the proprietary-content use cases, mandates `meta.security`
labelling, and ties them to `copyright` / `copyrightLabel` gives
implementers the precise, instance-level guidance they need without
forcing a resource-wide reclassification cascade. If the work group
nevertheless concludes that the resource-level signal must change, the
fallback should be Disposition A with an accompanying narrative that
explicitly notes "Business" applies to typical commercial knowledge
artifacts and that openly published libraries can down-classify via
`meta.security`. Either way, the wording change should be coordinated
with the FHIR-I security editors before merging into `HL7/fhir`.
