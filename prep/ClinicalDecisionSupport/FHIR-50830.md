## Summary

This ticket asks the Clinical Decision Support work group to consider defining
a new operation, `PlanDefinition/$feedback`, that lets a system report what
actions a provider actually took in response to a previously-issued decision
support recommendation. The intent is to mirror, in PlanDefinition / Clinical
Reasoning terms, what the CDS Hooks "feedback" service does for hook-based
recommendations — specifically tying the feedback back to the response
returned by `PlanDefinition/$apply`. The reporter further suggests the feature
should land in CPG (Clinical Practice Guidelines) for R4-era support, and a
later comment proposes that the home for the operation is the in-progress
"Using Clinical Reasoning with CDS Hooks" IG. No cross-referenced Jira
tickets were surfaced for additional context.

## Details

**Description:**

Need to be able to indicate what actions were taken by providers in response
to decision support requests. This is similar to the feedback service defined
in CDS Hooks, but needs to be representable directly in terms of a
`PlanDefinition/$apply` response.

Consider this in CPG to support for R4.

**Comments:**

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new operation `PlanDefinition/$feedback` in the Clinical Reasoning
module of FHIR Core (and/or in the CPG IG for R4). The operation accepts the
`PlanDefinition` that was previously applied (by canonical reference) plus a
correlation handle to the original `$apply` response (e.g., the
`RequestOrchestration` / `CarePlan` returned, or the
`GuidanceResponse.identifier` / `requestIdentifier` used as the in/out
contract on `$apply`). The body of the operation carries one feedback entry
per recommended action, each entry indicating the action canonical (matching
`PlanDefinition.action.id`), an outcome code drawn from the value set already
used by the CDS Hooks feedback service (`accepted`, `overridden`, etc., as
mapped to a FHIR CodeSystem), an optional reason / override-reason
(`CodeableConcept`), an optional reference to the resulting clinical resource
(`MedicationRequest`, `ServiceRequest`, …), and a timestamp. Modeled as a
defined operation with a parameters profile (rather than a free-form payload)
so that engines can validate and persist feedback consistently. Land it in the
Clinical Reasoning module in the next FHIR cycle, with a back-port profile in
CPG for R4 implementers, per the reporter's note. Per the 2025-10-15 comment,
mirror the operation definition into the in-progress "Using Clinical
Reasoning with CDS Hooks" IG so the alignment with CDS Hooks `feedback` is
explicit.

#### Justification

`PlanDefinition/$apply` is the canonical Clinical Reasoning entry point for
"give me the recommended actions"; closing the loop with a parallel
`$feedback` operation is the smallest, most discoverable extension of that
contract and matches an already-validated pattern (CDS Hooks `feedback`).
Reusing the CDS Hooks outcome value set keeps the two surfaces semantically
interchangeable, which is the explicit intent in the ticket and in the
follow-up comment about the "Using Clinical Reasoning with CDS Hooks" IG.
Defining it as a typed operation (rather than expecting implementers to POST
ad-hoc resources) gives servers a stable place to enforce correlation back to
the original `$apply` response and to drive analytics / quality measurement.

---

### Disposition B: Alternative Approach

#### Proposal

Do not introduce a new core operation. Instead, document a normative pattern
for capturing provider response using existing resources: the `$apply` server
returns a `RequestOrchestration` (or `CarePlan`) whose `action.id` values are
stable correlation handles; clients then `POST` (or `PATCH`) standard
clinical resources (`MedicationRequest`, `ServiceRequest`, `Task`,
`ClinicalImpression`) carrying a `Provenance` resource that points back to
the originating `RequestOrchestration.action` via
`Provenance.basedOn` / `entity.what` and uses the existing CDS Hooks feedback
codes in `Provenance.activity`. Optionally, define a small extension
(`http://hl7.org/fhir/StructureDefinition/cdshooks-feedback-outcome`) on
`Provenance` to carry the override reason. Publish this pattern in the "Using
Clinical Reasoning with CDS Hooks" IG (and, for R4, in CPG) rather than
shipping a new core operation.

#### Justification

A `$feedback` operation duplicates information that FHIR already represents
natively: what was done is `MedicationRequest` / `ServiceRequest` / `Task`,
why it was done (and what triggered it) is `Provenance` with `basedOn`. A new
operation creates a parallel write surface that servers must implement and
that analytics tools must learn, with no additional information content over
"the resulting resources plus a `Provenance` chain". This approach also
back-ports cleanly to R4 without new operation definitions, which directly
addresses the reporter's R4 / CPG concern, and it avoids the value-set and
profile churn that comes with a new normative operation.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for FHIR Core, on the basis that the CDS
Hooks `feedback` service already covers this need for hook-based decision
support, and that any clinical-reasoning–specific feedback shape should be
incubated in the "Using Clinical Reasoning with CDS Hooks" IG (the home the
reporter himself proposed in the 2025-10-15 comment) rather than added to
Core or CPG. Close the tracker as "Persuasive with Mod / Move to IG" and
re-open against that IG when its repo is established.

#### Justification

The ticket itself is exploratory ("Consider …"), there is no recorded
implementer demand outside the originator (a single Zulip ping with no
discussion), and the only proposed home in the comment is an IG that does not
yet exist. Adding a new core operation on the strength of one tracker and one
chat mention sets a precedent that bypasses the IG-incubation path that CDS /
Clinical Reasoning normally follows for new operations. Declining-and-redirect
keeps Core stable while preserving the work for the right venue.

---

### Recommendation

**Recommended disposition:** A (with the explicit caveat that the operation
is incubated in the "Using Clinical Reasoning with CDS Hooks" IG first, per
the reporter's 2025-10-15 comment).

`$feedback` is a natural pair to `$apply`: any system that can apply a
PlanDefinition has, by construction, the correlation context needed to
report back what happened, and the CDS Hooks `feedback` service has already
proven this exact pattern is what implementers want. Disposition B
(Provenance + native resources) is intellectually clean but pushes a
significant integration burden onto every CDS engine and every analytics
consumer to re-derive correlation that the operation gives you for free; in
practice, sites end up inventing per-vendor APIs that look exactly like
`$feedback` but are not interoperable. Disposition C is too conservative
given the explicit alignment intent with CDS Hooks. The pragmatic path is to
ship the operation, but ship it where the reporter says it belongs — in the
Clinical Reasoning + CDS Hooks IG, with a CPG R4 profile to satisfy the R4
backport requirement called out in the description.
