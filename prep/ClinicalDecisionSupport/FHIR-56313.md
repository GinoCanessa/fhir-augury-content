## Summary

The ticket asks the Da Vinci DTR specification to clarify and correct
its current "single endpoint" conformance language. The reporter
(Lloyd McKenzie) flags that the current text — "payers SHALL have a
single endpoint (managed by themselves or a delegate) that can handle
responding to all DTR service calls" — is technically inaccurate (every
FHIR operation has its own endpoint URL beneath a base) and also
inadvertently disallows the coverage-specific endpoint discovery
mechanism that Da Vinci HRex was explicitly designed to support. He
also notes that DTR's questionnaire profiles already include
mustSupport elements that name a per-form endpoint and per-question
terminology service — those elements are pointless if the base must be
fixed. The proposed replacement text reframes the SHALL around a
single **base URL** for all DTR operations, while permitting multiple
coverage-specific bases when the payer implements HRex endpoint
discovery, and preserving the existing "managed by payer or a
delegate" allowance.

## Details

**Description:**

Current text:
"However, for the purpose of DTR conformance, payers **SHALL** have a
single endpoint (managed by themselves or a delegate) that can handle
responding to all DTR service calls."

This is problematic in two ways:

1. Every operation, by definition, has a distinct endpoint. I.e. you're
   going to have `[somebase]/$questionnaire-package` vs.
   `[somebase]/$next-question` or `[somebase]/$expand`.
2. When we defined the endpoint discovery mechanism in HRex, one of our
   objectives was to support having coverage-specific endpoints, so
   that a payer could say "for plan x, go to endpoint 1; for plan y,
   go to endpoint 2". It was not our intention to remove this ability
   for DTR.

As well, we have support in the DTR questionnaire profiles an ability
to identify the specific endpoint to call for each adaptive form and,
potentially, specific terminology service endpoints for any question.
The only reason to support these is if the endpoint bases are allowed
to vary. If the endpoint base is fixed, there's no need to have these
elements as mustSupport.

However, based on discussion, there's significant complexity with
supporting multiple base URLs as every additional base requires
security configuration and trust arrangements.

Recommendation:
Change the text to:
"However, for the purpose of DTR conformance, payers **SHALL** use the
same base URL for all DTR operations. If the payer supports [HRex
endpoint discovery], there may be distinct base endpoints for
different coverages. Otherwise the payer **SHALL** be only one base
endpoint for its DTR functionality. In either case, DTR endpoints may
be managed directly by the payer or a delegate."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the current DTR conformance sentence verbatim with the
reporter's proposed text, with a minor grammatical fix on the second
SHALL clause:

> "However, for the purpose of DTR conformance, payers **SHALL** use
> the same base URL for all DTR operations. If the payer supports
> [HRex endpoint discovery], there may be distinct base URLs for
> different coverages. Otherwise the payer **SHALL** expose only one
> base URL for its DTR functionality. In either case, DTR endpoints
> may be managed directly by the payer or a delegate."

The change lives in the DTR IG narrative page that currently carries
the "single endpoint" sentence (the conformance / capability section
of the DTR spec) in the `HL7/davinci-dtr` IG sources. Add (or confirm)
a cross-link to the HRex endpoint-discovery page so the conditional
clause resolves. No StructureDefinition changes are required, and the
existing mustSupport flags on the DTR questionnaire profile elements
that name per-form / per-question endpoints can remain as-is — the
revised text is now consistent with their existence.

#### Justification

The current wording is demonstrably inaccurate at the FHIR mechanics
level (operations have distinct endpoint URLs beneath a base) and
silently overrides an HRex capability that the Da Vinci stack has
already invested in. Adopting the reporter's text fixes the
inaccuracy, restores HRex alignment, and preserves the practical
intent (one logical service surface per coverage) without forcing any
schema or conformance-resource change. It also restores internal
consistency with the DTR questionnaire profiles' per-form and
per-question endpoint elements, which would otherwise be hard to
defend as mustSupport.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the substance of Disposition A but split the conformance
statement into two explicit, separately testable rules and surface
them as a small conformance subsection rather than a single sentence:

1. **Base-URL rule:** "A payer SHALL expose all DTR operations
   (`$questionnaire-package`, `$next-question`, terminology
   `$expand`, etc.) under a single FHIR service base URL per coverage
   context. The base URL MAY be operated by the payer or by a
   delegate."
2. **Multiple-coverage rule:** "When a payer supports HRex endpoint
   discovery, it MAY advertise different base URLs for different
   coverages. Clients SHALL resolve the base URL via HRex endpoint
   discovery before invoking DTR operations when the payer publishes
   that capability."

Add a short non-normative note explaining the operational cost of
multiple base URLs (security and trust arrangements per base) so
implementers understand why the simple case is the default.

#### Justification

The reporter's single-sentence rewrite mixes two independent
conformance concerns (base-URL uniformity per coverage, and
multi-coverage routing via HRex). Splitting them is friendlier to
testers and to implementers who skim conformance language, and it
gives HRex endpoint discovery a first-class normative hook from the
DTR spec rather than a parenthetical aside. The trade-off is more
narrative real estate and a slightly larger ballot delta, but the
underlying obligations are identical to Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline the proposed wording change. Keep the existing "single
endpoint" sentence and instead address the reporter's concern via an
editorial errata note clarifying that "single endpoint" is shorthand
for "single base URL" and that nothing in DTR is intended to override
HRex endpoint discovery.

#### Justification

Defensible only if the workgroup judges the current wording to be
"close enough" and is concerned that any normative re-wording requires
re-balloting. The cost is that the inaccuracy and the apparent
HRex conflict remain in the normative text, and the DTR questionnaire
profile mustSupport elements that name per-form / per-question
endpoints continue to look inconsistent with the spec. Given that the
ticket has the `Ready-To-Address-DTR` label, the bar for declining is
high.

---

### Recommendation

**Recommended disposition:** A

The reporter's proposed text is small, surgical, technically correct,
and restores alignment with HRex endpoint discovery and with DTR's own
profile mustSupport flags — all without touching any
StructureDefinition or operation definition. Disposition B captures a
real readability win and is a reasonable amendment if the workgroup
wants to invest the editorial effort, but it does not change the
substantive obligations. Disposition C leaves a known inaccuracy in
normative text and is hard to justify given the ticket is already
flagged Ready-To-Address-DTR. Recommend accepting A, with a friendly
amendment to fix the "SHALL be only one base endpoint" grammar
("SHALL expose only one base URL") and to make the HRex link explicit.
