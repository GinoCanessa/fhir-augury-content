## Summary

The ticket asks the Clinical Decision Support work group to add explicit
conformance requirements that govern how CQL evaluation **context** is
declared in CQL libraries that are used with FHIR. Concretely, it
proposes that (a) a CQL library used in FHIR may declare one and only
one `context`, and (b) the `Library.subject` element on the FHIR
`Library` resource SHALL correspond to that context declaration. It
also asks the group to define how an "unfiltered" context (a CQL
construct meaning the expressions do not filter data access by a
particular context) is represented on `Library.subject`, proposing a
new `cql-context-declaration-codes` CodeSystem and ValueSet whose first
member is `unfiltered`. The resolution text on the ticket already
sketches the proposed normative wording and terminology artifacts. No
other Jira tickets are linked or cross-referenced; the only external
reference is a single Zulip mention by the reporter announcing the
tracker during Connectathon 39.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change exactly as outlined in the ticket's resolution text on
the *Using CQL With FHIR* specification:

1. **Using CQL section** — add normative wording: *"A CQL library used
   in FHIR SHALL declare one and only one `context`."*
2. **Conformance section** — add normative wording: *"The
   `Library.subject` element on a FHIR `Library` resource that carries
   a CQL library SHALL correspond to the `context` declared in that
   library. When the CQL `context` is unfiltered, `Library.subject`
   SHALL be bound to the `unfiltered` code from
   `CQLContextDeclarationCodes`."*
3. **Terminology** — define a new CodeSystem
   `http://hl7.org/fhir/uv/cql/CodeSystem/cql-context-declaration-codes`
   and a matching ValueSet
   `http://hl7.org/fhir/uv/cql/ValueSet/cql-context-declaration-codes`,
   each containing the single concept `unfiltered` with the
   description proposed in the ticket and a link back to the CQL
   specification's context declaration section. Bind `Library.subject`
   (when used to convey CQL context) to that ValueSet using an
   extensible binding so that future CQL contexts not present in the
   FHIR resource model (e.g., novel non-clinical contexts) can also be
   represented.

#### Justification

This is exactly what the reporter — who is also the editor of the
*Using CQL With FHIR* specification — has asked for, and the
resolution wording is already drafted. Requiring a single `context` per
library aligns with how CQL libraries are intended to be packaged and
published: a `Library` resource has a single `subject`, so allowing
multiple `context` declarations inside one CQL library that is wrapped
by a single `Library` resource creates an ambiguity that implementers
have to resolve by convention. Tying `Library.subject` to the CQL
`context` removes that ambiguity at the conformance level and makes
authoring tools, evaluators, and content packagers behave the same
way. Defining `unfiltered` in a CodeSystem solves the long-standing
representation gap where `Library.subject` (a Reference/CodeableConcept
choice typically populated with a resource-type code such as `Patient`
or `Practitioner`) had no canonical way to indicate "no filtering
context" without overloading existing codes.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the conformance requirements but make the single-context
constraint a **SHOULD** (with a documented exception path) rather than
a **SHALL**, and represent `unfiltered` via an existing FHIR mechanism
instead of minting a new CodeSystem. Specifically:

1. Change the Using CQL conformance statement to: *"A CQL library used
   in FHIR SHOULD declare one and only one `context`. Libraries that
   declare more than one `context` SHALL be wrapped as multiple
   `Library` resources, one per `context`, related via
   `Library.relatedArtifact` of type `composed-of`."*
2. Keep the *"`Library.subject` SHALL correspond to the CQL
   `context`"* conformance requirement.
3. Instead of a brand-new CodeSystem, represent `unfiltered` by either
   (a) the absence of `Library.subject` entirely (with a normative note
   that absent `subject` means "unfiltered"), or (b) a single concept
   added to an already-published HL7 terminology
   (e.g., extending the existing CQF/CQL terminology in
   `terminology.hl7.org`) rather than a new CodeSystem in the *Using
   CQL With FHIR* IG itself.

#### Justification

The single-context-per-library constraint is reasonable, but a hard
SHALL retroactively makes a class of historically valid CQL libraries
non-conformant for FHIR usage, and several published measure / decision
support content packages do contain libraries with more than one
context declaration. A SHOULD with a documented "split into multiple
`Library` resources" path preserves the design intent — one `subject`
per `Library` — without invalidating existing content overnight, and
gives tooling a deterministic transformation. Avoiding a brand-new
single-code CodeSystem in the IG keeps the terminology footprint small
and matches HL7's general preference to reuse terminology assets;
either using absence-of-`subject` or piggy-backing on existing CQF
terminology is cheaper to maintain than a new CodeSystem/ValueSet pair
that today only carries a single concept. The trade-off vs.
Disposition A is that an absent-`subject` convention is easy to
overlook in validators and that downgrading SHALL to SHOULD weakens
interoperability guarantees.

---

### Disposition C: Decline

#### Proposal

Do not add the proposed conformance requirements. Document the
intended one-context-per-library / `subject`-equals-`context`
convention as **non-normative guidance** in the *Using CQL With FHIR*
specification, and do not introduce a new CodeSystem or ValueSet for
context declaration codes.

#### Justification

The CQL grammar already permits multiple `context` declarations in a
single library, and FHIR's *Using CQL With FHIR* specification is
intended to describe how CQL is packaged in FHIR rather than to
constrain CQL itself. Promoting these expectations to SHALL-level
conformance arguably puts FHIR in the position of constraining the
underlying CQL specification, which is governed elsewhere. The
representation of `unfiltered` could equally be left to implementer
convention (e.g., omitting `Library.subject`), avoiding a new
single-code terminology artifact whose only consumer is this one
binding. This option is the most defensible only if the work group
believes the multi-context use case is real enough to warrant
preserving, or that the conformance burden on existing content
outweighs the interoperability benefit.

---

### Recommendation

**Recommended disposition:** A

The ticket is filed by the editor of the spec, has been triaged with a
resolution of *Persuasive*, and the proposed wording is already
drafted; there is no countervailing community pushback in the linked
Zulip thread or in any cross-referenced ticket. The single-context
constraint reflects how `Library.subject` is actually used in published
CQL/CQF content (eCQM, QI-Core, US-CQL-IG, sample-content-ig), and
formalising it at conformance level closes a long-standing
representation gap rather than creating a new burden. Disposition B's
SHOULD-with-split path is a reasonable softening if the work group
hears at ballot that real-world libraries routinely declare multiple
contexts, but absent that evidence the SHALL is cleaner for tooling
and validators. Disposition C is hard to defend given that the editor
has explicitly asked for the requirement and that the *Using CQL With
FHIR* specification's whole purpose is to nail down conformance
expectations like this. Recommend accepting as requested, with a
ballot-ready note inviting comment specifically on whether SHALL vs.
SHOULD is appropriate for the single-context rule.
