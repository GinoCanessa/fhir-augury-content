## Summary

The ticket asks that `EventDefinition` gain a first-class `library` element so
that EventDefinitions whose triggering / matching logic is expressed in CQL (or
another formal logic language) can reference a `Library` resource directly,
instead of relying on the `cqf-library` extension. The reporter (Bryn Rhodes)
notes that this is already the established pattern for other knowledge-artifact
resources in the Clinical Reasoning module (e.g., `Measure.library`,
`PlanDefinition.library`), and the resolution text proposes adding
`library 0..* canonical(Library)` with the definition "Logic used by the event
definition / A reference to a Library resource containing any formal logic used
by the event definition." The ticket has been triaged Persuasive but is not
yet applied. No cross-referenced Jira tickets, Zulip threads, or GitHub items
were surfaced by the augury index for this key.

## Details

**Description:**

EventDefinition does not currently have a Library element. This requires
EventDefinitions that make use of logic to use the cqf-library extension. To
support this use case directly, library should be an element of
EventDefinition.

**Resolution Description (from snapshot):**

Add a library element to EventDefinition:

> library 0..* canonical(Library) — Logic used by the event definition
> definition: A reference to a Library resource containing any formal logic
> used by the event definition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir` (FHIR Core), add a new element to the `EventDefinition`
StructureDefinition `differential`:

- Name: `library`
- Cardinality: `0..*`
- Type: `canonical(Library)`
- Short: `Logic used by the event definition`
- Definition: `A reference to a Library resource containing any formal logic
  used by the event definition.`

Place the element alongside the other knowledge-artifact metadata, mirroring
the position and styling used for `Measure.library` and `PlanDefinition.library`.
Update the `EventDefinition` narrative / intro page to mention the new
element, and add a note to the `cqf-library` extension definition (in
`HL7/fhir-extensions`) that for `EventDefinition` the core element should now
be preferred and the extension is retained only for backward compatibility on
prior FHIR versions.

#### Justification

This is the standard Clinical Reasoning pattern: `Measure`, `PlanDefinition`,
and `ActivityDefinition` already expose `library 0..* canonical(Library)` as a
core element, and `EventDefinition` is the lone outlier that forces consumers
to reach for `cqf-library`. Promoting the extension to a core element on a
resource where logic references are clearly in-scope for the resource's
defined purpose is consistent with FHIR's general guidance ("if an extension
is widely used and clearly belongs to the resource, fold it into the core").
The cardinality and type match the sibling resources, which keeps tooling
(CPG-on-FHIR, CQF, $apply-style implementations) uniform across knowledge
artifacts. The ticket has already been triaged **Persuasive**, indicating
substantive committee agreement.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the `EventDefinition` resource. Instead, formally bind the
existing `cqf-library` extension to `EventDefinition` as a "modifier-equivalent"
companion via the published extension definition in `HL7/fhir-extensions`,
ensure it appears on the `EventDefinition` resource's "Extensions" tab in the
spec, and add explicit prose to the EventDefinition narrative explaining how
to reference logic via `cqf-library`. Optionally, also publish a small
"clinical-reasoning EventDefinition" profile in the CPG / CRMI IG that slices
the extension to `0..*` and makes it the recommended representation.

#### Justification

This keeps the base `EventDefinition` resource stable (no schema change, no
maturity reset, no impact on existing instances or generated SDKs) while still
giving implementers a documented, conformant path for attaching logic
libraries. Because `EventDefinition` is generally less widely deployed than
`Measure` or `PlanDefinition`, the cost / benefit of a core-element change
versus a well-documented extension is closer than it appears. This option
also avoids divergence between FHIR R5 and any in-flight R6 ballot windows
and lets the CPG / CRMI IGs absorb the recommendation without requiring a
core spec republish.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by an existing mechanism**: the
`cqf-library` extension exists precisely to attach `Library` references to
knowledge-artifact resources that don't have a native `library` element, and
it is already used by the Clinical Reasoning community for this purpose on
`EventDefinition`. Close the ticket with a pointer to that extension and a
note in the EventDefinition narrative.

#### Justification

Adding a new element to a normative-track resource has non-trivial cost:
maturity-level implications, downstream SDK regeneration, profile and IG
revalidation across CPG, CRMI, ECR, Da Vinci, and other ecosystems that
already author `EventDefinition` instances. If the existing extension is
functionally sufficient and is not a barrier to interoperability (the same
canonical(Library) value flows through either way), then a breaking-shaped
change to the resource is hard to justify on parity grounds alone.

---

### Recommendation

**Recommended disposition:** A

The ticket is a straightforward parity fix: every other knowledge-artifact
resource in the Clinical Reasoning family (`Measure`, `PlanDefinition`,
`ActivityDefinition`) already has `library 0..* canonical(Library)` as a core
element, and `EventDefinition` is the only resource in that family that
forces implementers into the `cqf-library` extension for a use case that is
unambiguously in-scope for the resource. Disposition B preserves stability
but locks in an inconsistency that implementers and validators have to keep
working around, and Disposition C is hard to defend given that CDS already
triaged the ticket Persuasive and the requested cardinality/type exactly
matches the sibling pattern. The change is small (a single new element with
a well-established type and definition), it is additive (`0..*`, no
breaking change to existing instances), and it lets `cqf-library` be
deprecated for `EventDefinition` going forward while still functioning for
back-compatibility. The workgroup should accept as requested and schedule
the addition of `EventDefinition.library` for the next applicable FHIR
release, with a coordinated note in `HL7/fhir-extensions` marking the
`cqf-library` use on `EventDefinition` as superseded by the new core element.
