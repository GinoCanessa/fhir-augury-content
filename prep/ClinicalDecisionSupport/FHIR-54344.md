## Summary

The reporter (Erik Moll) observes that the Overview section of the CDS Hooks
(FHIR) specification cites a Wikipedia article on
["Hooking"](http://en.wikipedia.org/wiki/Hooking) when introducing the
"hook"-based pattern that underpins the specification. He argues that
Wikipedia pages are volatile and not generally accepted as a reliable
citation in scientific or normative literature, and asks (a) whether HL7 has
a policy on what may be referenced from a published standard and (b) why a
more stable source — such as a peer-reviewed computer science paper or a
canonical textbook — is not used instead. Bryn Rhodes has responded asking
the reporter to suggest a preferred authoritative source, noting that the
Wikipedia link is intended to be informative rather than evidentiary.

No other Jira tickets, Zulip threads, or GitHub items cross-reference this
issue, so the request stands on its own as a small editorial / citation
hygiene change to the CDS Hooks Overview.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the Wikipedia citation in the CDS Hooks Overview with a stable,
citable source for the "hook" software pattern. Concretely:

- In the Overview prose
  ("It describes a 'hook'-based pattern for invoking decision support
  from within a clinician's workflow"), drop the inline link to
  `http://en.wikipedia.org/wiki/Hooking`.
- Replace it with a citation to a durable source for the hook /
  callback pattern, for example:
  - Gamma, Helm, Johnson, Vlissides, *Design Patterns: Elements of
    Reusable Object-Oriented Software* (Addison-Wesley, 1994) for the
    Template Method / callback discussion that underlies "hooks", **or**
  - the IEEE/ACM term entry for "callback / hook function" in a
    standards-grade glossary, **or**
  - an HL7-published informative reference (e.g., a CDS Hooks
    background document) if one is available.
- Move the citation into the specification's References section (if the
  Overview currently uses inline links, add a numbered/bracketed
  reference and list the full citation in References) so it follows
  HL7's normal citation style and is easier to update if the source
  moves.

This is an editorial / Technical Correction-level change in the CDS
Hooks (FHIR) specification's Overview page only; no normative content,
hook definitions, or schemas are touched.

#### Justification

Wikipedia URLs are explicitly volatile (article titles can be renamed,
content can be rewritten, and articles can be deleted), which is exactly
the reporter's concern, and HL7 published standards generally prefer
stable, attributable sources for both normative and informative
citations. Replacing the link with a textbook / standards-grade
reference preserves the pedagogical intent that Bryn Rhodes described
("informative, not authoritative") while removing the volatility risk.
The change is low-risk, fully backward-compatible (no implementer-visible
behavior changes), and can be made in a single editorial pass on the
Overview page.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the informative concept link, but stop citing Wikipedia
specifically. Instead:

- Rewrite the Overview sentence so the term "hook" is defined in-line by
  the CDS Hooks specification itself — e.g., "CDS Hooks defines a
  *hook* as a well-defined point in a clinician's workflow at which a
  CDS Service may be invoked" — and remove the external link entirely.
- Optionally add a short footnote or "see also" pointing at one or more
  durable secondary sources (textbook, ACM/IEEE entry), mirroring
  Disposition A's reference list, but make the specification
  self-contained for the term.
- Apply the same pattern wherever else the Overview / introductory
  material reaches for Wikipedia to define a software-engineering term.

#### Justification

The reporter's underlying concern is that a normative HL7 publication
should not depend on a non-curated external page to convey its core
concept. The strongest remedy for that concern is to define the term
inside the specification itself, since CDS Hooks is the document that
*operationalises* the concept of a "hook" in clinical workflow — its
definition is more authoritative for this context than any general CS
reference. This also avoids debates about which external source is
"the" canonical citation for hooks (callbacks, plug-in points, AOP
join-points are all defensible), and aligns with how other FHIR
specifications generally introduce key terms (definition first, see-also
references second). The trade-off vs. Disposition A is slightly more
editorial work and a small Overview rewrite rather than a one-line
citation swap.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-54344 as "Persuasive with Modification → no spec change",
i.e., acknowledge the reporter's policy question but do not change the
Overview text. Rationale category: *informative reference, no normative
impact*. Optionally pair with a follow-up note that any future
republication of CDS Hooks may revisit citation style as part of an
HL7-wide editorial policy, rather than per-IG.

#### Justification

The Wikipedia link is informative, not normative — it neither defines
conformance behavior nor is referenced from any conformance language in
CDS Hooks. As Bryn Rhodes noted in the comments, it is intended only as
a reader's gloss on the term "hook". HL7 does not currently have a
project-wide ban on Wikipedia in informative prose, and changing one
specification in isolation creates an inconsistency without resolving
the underlying policy question the reporter raised. Declining here
keeps the Technical Correction queue focused on conformance-affecting
issues and defers the policy question to TSC / Publishing.

---

### Recommendation

**Recommended disposition:** A

The cost of fixing this is essentially nil — a one-line citation swap on
the Overview page of CDS Hooks (FHIR), with no normative or
implementer-visible impact — and the benefit is real: the link is
already at a `http://` (non-HTTPS) Wikipedia URL whose target article
("Hooking") could be retitled or merged at any time, which is precisely
the failure mode the reporter is flagging. Disposition B is attractive
in principle (define the term inside the spec) and would be the right
answer if the workgroup is willing to spend a slightly larger editorial
pass; it could in fact be folded into Disposition A by adopting both
("define in-line, then cite a durable source"). Disposition C is
defensible — the link is informative — but it leaves the volatility
issue in the published artifact and gives the reporter no concrete
remedy for a Highest-priority Technical Correction. The Clinical
Decision Support workgroup should accept the request, ask the reporter
(or Bryn Rhodes) to nominate a preferred durable source per the open
question already in the comments, and apply it as an editorial fix
ahead of the next CDS Hooks publication.
