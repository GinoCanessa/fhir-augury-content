## Summary

The reporter notes that FHIR JSON is not the same as generic JSON — the FHIR
specification imposes additional serialization requirements on FHIR elements
(as documented at https://fhir.hl7.org/fhir/json.html) that are necessary for
correct clinical interpretation of resources. The ticket asks that the CDS
Hooks (FHIR) specification explicitly require FHIR JSON (i.e., the
`application/fhir+json` serialization) for any embedded FHIR resources
exchanged via CDS Hooks — either for embedded FHIR resources specifically, for
all payloads, or at minimum by stating the serialization rules that apply. The
ticket has been triaged with resolution "Persuasive," and the proposed fix
updates the "Use of JSON" section of the CDS Hooks spec to clarify that all
embedded FHIR resources use the standard FHIR serialization, with a link to
the FHIR JSON page and explicit wording that production RESTful CDS Hooks
APIs MUST send/receive JSON with embedded FHIR resources as `fhir+json`.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
for this ticket.

## Details

**Description:**

FHIR has additional requirements on serialization of FHIR elements in JSON
(see [Json - FHIR v5.0.0](https://fhir.hl7.org/fhir/json.html)) which are
required for proper clinical interpretation of the resources. Please require
FHIR json for any embedded FHIR resources (or for all) (or state the
serialization is FHIR json).

**Resolution Description:**

Will update the "Use of JSON" section to clarify that all embedded FHIR
resources use the standard FHIR serialization with a link to the
[https://hl7.org/fhir/R4/json.html](https://hl7.org/fhir/R4/json.html) page.

> "All data exchanged through production RESTful APIs MUST be sent and
> received as JSON (JavaScript Object Notation) structures and embedded FHIR
> resources as fhir+json."

**Comments:**

No comments are recorded on the ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CDS Hooks (FHIR) specification's "Use of JSON" section to state
that all embedded FHIR resources in CDS Hooks payloads (e.g., `context`
resources, `prefetch` response entries, and any FHIR resources carried in
cards or suggestions) MUST be serialized as FHIR JSON (`application/fhir+json`)
per the FHIR JSON specification, with a link to
https://hl7.org/fhir/R4/json.html (and equivalent pages for other FHIR
versions). Adopt the wording from the resolution:

> "All data exchanged through production RESTful APIs MUST be sent and
> received as JSON (JavaScript Object Notation) structures and embedded FHIR
> resources as fhir+json."

This normative clarification closes the ambiguity that generic JSON might be
accepted for embedded FHIR content.

#### Justification

FHIR JSON differs from generic JSON in several clinically meaningful ways
(e.g., primitive extensions via underscore-prefixed siblings, decimal
preservation rules, canonical property ordering allowances, and element
identity rules). Serializing a FHIR resource as generic JSON can silently
drop or mis-represent clinical content (notably primitive extensions and
precision-sensitive decimals), which matches the reporter's "required for
proper clinical interpretation" concern. The change is editorial/normative
clarification of existing intent rather than a behavioral break, and it
aligns CDS Hooks with the FHIR core specification's own serialization
contract. The workgroup has already marked the ticket "Persuasive" with a
concrete edit in mind, so Disposition A matches both the request and the
triaged resolution.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a spec statement alone, require `fhir+json` via the HTTP
content negotiation layer: update "Use of JSON" to SHOULD-level prose about
FHIR JSON, and additionally specify that CDS Services and CDS Clients MUST
advertise and accept the `application/fhir+json` media type on any request
or response that embeds FHIR resources (prefetch responses, card/suggestion
FHIR payloads, context resources). Keep `application/json` permitted for the
non-FHIR envelope (hook request/response JSON structure) but require
`application/fhir+json` for the embedded resource bodies where they appear
as first-class HTTP payloads. Add a conformance note that servers sending
embedded FHIR resources inside a JSON envelope are serializing them per the
FHIR JSON rules even when the envelope's Content-Type is `application/json`.

#### Justification

Disposition B addresses the same clinical-correctness concern but leverages
the existing FHIR media-type machinery instead of prose alone, which gives
implementers a testable, wire-observable conformance hook (Content-Type /
Accept). It also explicitly handles the awkward case that the CDS Hooks
envelope itself is plain JSON while the *embedded* FHIR payloads are
expected to be FHIR JSON — a distinction the prose-only edit in Disposition
A leaves implicit. The trade-off is more spec surface area and potential
interop friction with existing CDS Hooks clients that only advertise
`application/json`, so the benefit over Disposition A is modest if the
workgroup judges the prose clarification sufficient.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that FHIR resources, wherever they appear,
are already defined to serialize per the FHIR JSON rules by the FHIR core
specification, and the CDS Hooks specification's existing references to
FHIR resources transitively inherit that contract. No CDS Hooks-side edit is
required.

#### Justification

FHIR's serialization rules are defined once in the FHIR core spec and apply
to any FHIR resource instance in any context; a downstream specification
that says "a FHIR Patient resource" is already saying "serialized per FHIR
JSON." Re-stating that in every consuming IG risks drift if the core rules
evolve. However, this argument is weak in practice because implementers
have demonstrably been unsure enough to raise the ticket, and the cost of
a one-paragraph clarification is minimal — so Decline is defensible only in
principle, not in practice.

---

### Recommendation

**Recommended disposition:** A

Disposition A is recommended. The ticket has already been triaged as
"Persuasive" with a concrete editorial change prepared, the reporter's
concern is a real clinical-safety issue (FHIR JSON's primitive-extension
and decimal-precision rules are not satisfied by generic JSON parsers),
and the fix is a low-risk clarification rather than a behavior change.
Disposition B is a reasonable extension if the workgroup wants a
wire-level conformance hook via `application/fhir+json`, but it
introduces scope (media-type conformance for embedded vs. envelope) that
can be addressed in a follow-up ticket if implementers need it.
Disposition C understates how much ambiguity implementers perceive in
the current text and is not recommended.
