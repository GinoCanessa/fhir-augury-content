## Summary

Lloyd McKenzie requests that the CDS Hooks Hooks Library specification
define a normative code system that enumerates all "standard" CDS Hook
types. The motivation is that downstream specifications — most concretely
Da Vinci CRD, which currently maintains a temporary code system of hook
codes for use in a logical model that captures CRD usage metrics, and
which is also referenced in CapabilityStatements and potentially
Provenance — need a stable, authoritative coded vocabulary they can bind
value sets to. Today there is no such code system: each downstream IG
either rolls its own temporary codes or refers to hook types only as
free strings, and CRD's temporary list is incomplete (it covers only the
hooks CRD cares about). Locating the code system in the Hooks Library
IG keeps the codes in lockstep with the IG that defines the hook types
themselves, and avoids creating a THO code system that would have to be
manually kept in sync. The reporter notes that a TSMG exemption will be
needed for an IG-resident code system, but expects approval to be
straightforward because the code system content is intrinsically tied to
the IG content. A draft implementation is already proposed in
[HL7/cds-hooks-library#25](https://github.com/HL7/cds-hooks-library/pull/25).

No directly-linked Jira tickets were returned by the cross-reference
query, so the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new `CodeSystem` resource to the CDS Hooks Hooks Library IG that
enumerates every hook type defined in that IG (one concept per
"standard" hook, e.g., `patient-view`, `order-select`,
`order-sign`, `appointment-book`, `encounter-start`,
`encounter-discharge`, etc.). Use a stable canonical URL under the
Hooks Library IG's namespace (e.g.,
`http://hl7.org/fhir/uv/cds-hooks-library/CodeSystem/hook-types`), make
it `complete`, `experimental=false`, and `caseSensitive=true`, and
populate display names and definitions from each hook's narrative
section. Publish a companion all-hooks `ValueSet` bound to the same
canonical so downstream IGs (CRD, Da Vinci PAS, etc.) can either bind
directly or define narrower value sets that include only the hooks
they need. Add IG narrative documenting the publication policy
(every new hook added to the IG must add a concept in the same
release) and submit a TSMG exemption request citing the
intrinsic IG ↔ code-system coupling. Land the change via the
already-open
[HL7/cds-hooks-library#25](https://github.com/HL7/cds-hooks-library/pull/25),
amended as needed to match the agreed canonical URL and concept set.

#### Justification

The request is narrowly scoped, has a concrete consumer (Da Vinci CRD's
metrics logical model and CapabilityStatement bindings), and is
endorsed without dissent by the only Zulip participants from the work
group. Co-locating the code system with the IG that defines the hook
types is the standard FHIR pattern for tightly-coupled vocabularies and
sidesteps the chronic THO sync problem the reporter explicitly called
out. The TSMG exemption path is well-trodden for this exact situation,
and a working PR already exists, so the implementation cost to the work
group is minimal.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a brand-new code system, formalize hook identifiers by
adding a `CodeSystem` shell whose codes are sourced from the existing
hook page slugs in the Hooks Library IG (i.e., generate the
`CodeSystem` from the same source-of-truth that produces the hook
documentation pages, rather than maintaining a separate hand-edited
resource). Bind downstream usage to a single all-hooks `ValueSet`
with a stable canonical, but mark the `CodeSystem` as
`content=fragment` plus `supplements` so that third-party / vendor
hooks (which the IG already permits) can be modelled as supplements
without the core IG having to enumerate them. Alternatively, if the
work group prefers to avoid TSMG entirely, host the code system in
THO but auto-generate it from the Hooks Library publication pipeline
on each release, so the sync burden is mechanical rather than manual.

#### Justification

This addresses the same downstream need (a stable coded vocabulary for
hook types) while reducing two risks: (1) drift between the IG's
narrative and the code system's enumerated codes, by deriving both
from one source; and (2) the implicit assumption that the IG
maintainers will keep manually adding concepts for every new hook.
The supplements model also gracefully accommodates the vendor-defined
hooks the spec already allows, which a pure "all standard hooks" code
system does not. The THO-with-automation variant preserves the
"terminology lives in THO" invariant that some governance reviewers
prefer, at the cost of a slightly more complex publication pipeline.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that hook types are already
unambiguously identified by their string identifiers in the Hooks
Library IG, and downstream specifications (including CRD) can refer
to those identifiers directly via the IG's hook pages without a
formal `CodeSystem`. Recommend that CRD continue to maintain its
own temporary code system, or that the affected downstream IGs
coordinate a shared THO code system, rather than expanding the
scope of the Hooks Library IG.

#### Justification

A defensible-but-weak rationale: the CDS Hooks specification already
defines hook identifiers as strings with a clear naming convention,
and FHIR conformance machinery (CapabilityStatement, ValueSet
binding) does not strictly require those identifiers to be expressed
as a `CodeSystem`. Avoiding a new IG-resident code system also
sidesteps the TSMG exemption process and the obligation to keep the
code system synchronized with the IG narrative on every publication.

---

### Recommendation

**Recommended disposition:** A

The request is small, well-motivated, has a concrete downstream
consumer, and arrives with a draft PR already in flight. Zulip
discussion shows uniform support and no competing proposals. The TSMG
exemption is foreseeable and the reporter has already flagged it. The
alternative in Disposition B raises legitimate maintenance concerns,
but those are better addressed as follow-up improvements to the IG's
publication tooling than as blockers to landing the code system itself
— the work group can adopt a "generate from hook pages" pipeline in a
later cycle without changing the published canonical. Disposition C
preserves the status quo at the cost of forcing every downstream IG to
either invent its own codes or coordinate informally, which is exactly
the pain point the ticket is trying to remove. Accepting as requested
is the right starting point; the work group should agree on the
canonical URL, the initial concept list, and the publication-time
policy for adding new hooks before merging
[HL7/cds-hooks-library#25](https://github.com/HL7/cds-hooks-library/pull/25).
