## Summary

The reporter (Bryn Rhodes) asks the Clinical Quality Language (CQL)
specification to recommend that implementations emit a warning when a
choice-type resolution silently eliminates one or more of the possible
representations of a choice-typed value. In the supplied example, a
function `Normalize` is declared with a parameter of type
`Choice<Integer, String>`, but the body's `case` arms only handle
`DateTime` and `String`. The `Integer` choice is silently dropped, and
because the actual argument (`ChoiceResult`, a `Choice<Boolean, String>`)
contains a `Boolean`, the resolved call returns `null` with no
diagnostic. The request is to make this a translator/engine warning so
that authors are alerted when type resolution narrows the set of
representations the function can handle.

No related Jira tickets, Zulip threads, or GitHub items were surfaced by
the cross-reference, search, or related-item lookups, so the request
stands on its own merits.

## Details

**Description:**

Consider the following definitions:

```cql
library SilentNullidity

define function Normalize(value Choice<Integer, String>):
  case
    when value is DateTime then ToString(value)
    when value is String then value as String
    else null
  end

define ChoiceResult:
  First(List<Choice<Boolean, String>> { true })

define SilentNullidity:
  Normalize(ChoiceResult)
```

Consider recommending that implementations provide a warning if a choice
type function resolution eliminates choices of argument types.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative or guidance text to the CQL specification (Author's Guide
section on functions / type resolution, and the corresponding Language
Reference section on choice type matching) requiring implementations to
emit a **warning** when, during resolution of a function invocation, the
argument's set of choice representations is not fully covered by the
parameter's declared choice representations. Specifically:

- When an argument has type `Choice<A1, A2, ..., An>` and the resolved
  parameter has type `Choice<P1, P2, ..., Pm>`, the implementation must
  warn for each `Ai` that has no implicit conversion path to any `Pj`.
- The warning should identify the function, the argument position, the
  dropped choice type(s), and the source location.
- Update the Translator conformance / diagnostics section to register
  the new warning code (e.g., `CQL-W-CHOICE-NARROW`) and add an
  illustrative example mirroring the `SilentNullidity` example from the
  ticket.

No change is proposed to the type-resolution algorithm itself — calls
that resolve today continue to resolve, the change is purely diagnostic.

#### Justification

CQL's choice-type matching combined with `is`/`as` filtering inside
function bodies is a known sharp edge: a function declared over
`Choice<Integer, String>` will accept a `Choice<Boolean, String>`
argument because `String` is a common representation, but at runtime any
`Boolean` value flows through the `else null` branch with no signal to
the author. Emitting a warning is consistent with how CQL treats other
"likely-author-error" situations (e.g., implicit conversions across
quantity units, ambiguous overload resolution). It preserves backward
compatibility — existing libraries continue to compile and run — while
giving authors the diagnostic they need to catch silent-null bugs of the
kind shown in the ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of a single broad warning on any narrowing, define a more
nuanced diagnostic policy in the Translator conformance section:

1. **Error** when the argument's choice set has *no* overlap with the
   parameter's choice set after implicit conversions (today this is
   typically already an error; codify it explicitly as
   `CQL-E-CHOICE-DISJOINT`).
2. **Warning** when there is partial overlap — i.e., at least one
   argument representation is droppable — but only when the function
   body cannot be statically shown to handle the dropped
   representations. If the body's `case`/`if` analysis demonstrably
   covers the dropped types (e.g., via an `else` arm that returns a
   non-null value, or via an explicit `is`-guard chain), suppress the
   warning.
3. **Info** (suppressible) when the narrowing is fully covered.

Pair this with an opt-in stricter mode (compiler flag
`--strict-choice-resolution`) that promotes the warning to an error for
libraries that want hard guarantees, and document an annotation (e.g.,
`@SuppressWarning("CQL-W-CHOICE-NARROW")` on the call site or function
declaration) so library authors can acknowledge intentional narrowing.

#### Justification

A blanket warning on any narrowing risks noise for library authors who
write deliberately permissive functions (e.g., a `Normalize` that
intentionally returns `null` for representations it cannot meaningfully
normalize). A graded diagnostic policy keeps the signal high and aligns
with how mature compilers (e.g., C#'s nullable-reference analysis,
TypeScript's exhaustiveness checking) handle similar
"silent-narrowing" cases. The opt-in strict mode and per-call
suppression give Quality Measure authors and content IGs the choice
between safety and quietness without forcing one trade-off on the whole
community. The downside is added specification surface (three diagnostic
categories, a flag, a suppression mechanism) and translator
implementation cost.

---

### Disposition C: Decline

#### Proposal

Decline the request. Document in the resolution that authors are
expected to verify their function bodies cover all declared parameter
choice representations, and that the existing `is`/`as` operator
semantics (returning `null` when no branch matches) are by design.
Optionally, add a non-normative authoring note pointing out the pattern
shown in the ticket as a common pitfall.

#### Justification

CQL's null-propagating semantics are foundational and intentionally
quiet — many constructs return `null` rather than raising diagnostics.
Adding a warning specifically for choice-type narrowing risks
inconsistency with the broader "null is a value" stance of the language,
and the same class of silent-null bug exists for many other constructs
(e.g., a `case` with no matching `when` and no `else`, arithmetic on
`null`, etc.). The cost of specifying, implementing, and tuning the
warning across the translator and engine ecosystem may not be
proportionate to the benefit when an authoring-guidance note plus
existing static analysis tooling could address the same pitfall.

---

### Recommendation

**Recommended disposition:** B

The example in the ticket is a real and easily-overlooked authoring
hazard, and silently returning `null` is the worst possible outcome for
a clinical quality artifact — declining (Disposition C) leaves a known
trap in place. Disposition A (a single broad warning) is the right
direction but will likely produce false positives on intentionally
permissive normalization helpers, which CDS authors write often. The
graded policy in Disposition B preserves the diagnostic value (the
`SilentNullidity` example still warns), distinguishes the genuinely
broken case (disjoint choice sets) from the "narrowed but covered" case,
and gives content authors a documented suppression path so the warning
remains trusted signal rather than noise. It is more spec work than A,
but it lines CQL up with the diagnostic ergonomics implementers and
authors already expect from other strongly-typed languages, and it
leaves room to ratchet to strict-by-default in a later normative
release.
