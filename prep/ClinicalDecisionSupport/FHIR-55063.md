## Summary

The ticket asks the Clinical Decision Support work group to consider
migrating three value sets that are currently defined inside FHIR Core
and bound to elements on the `DetectedIssue` resource over to
Terminology.HL7.org (THO). The candidates are
`detectedissue-severity` (bound to `DetectedIssue.severity`),
`detectedissue-example-medication-class` (bound to
`DetectedIssue.medicationClass`), and `detectedissue-management-code`
(bound to `DetectedIssue.managementCode`), each backed by a
correspondingly-named CodeSystem. Migration is not strictly required,
but is recommended where the value-set definition is likely to evolve
between FHIR releases or where the same vocabulary could be reused by
CDA, V2, or other HL7 specifications. There are no cross-referenced
related Jira tickets surfaced by the augury index — this ticket stands
on its own as part of a broader audit of THO-candidate value sets.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate all three CodeSystem/ValueSet pairs from FHIR Core to THO:

1. `detectedissue-severity` (CS + VS) → THO, retaining its current
   canonical-URL redirection so that the existing required/extensible
   binding on `DetectedIssue.severity` keeps resolving.
2. `detectedissue-example-medication-class` (CS + VS) → THO, with the
   `DetectedIssue.medicationClass` example binding repointed to the
   migrated VS.
3. `detectedissue-management-code` (CS + VS) → THO, with the
   `DetectedIssue.managementCode` example binding repointed to the
   migrated VS.

For each migration, follow the standard THO migration recipe: publish
the artifacts in THO, add canonical-URL redirects from the core URLs,
update the binding in DetectedIssue's structure definition to reference
the THO canonical, and remove the now-duplicated artifacts from FHIR
Core.

#### Justification

This fulfills the literal request and is internally consistent — three
value sets attached to the same resource get the same governance
treatment. THO is the right home for vocabulary that may be reused in
CDA, V2, or other HL7 specifications, and putting the example value
sets there as well leaves the door open for that reuse without needing
another migration cycle later. Lloyd's audit (the source of this
ticket) flags exactly this pattern; following the recommendation
uniformly avoids the appearance of cherry-picking.

---

### Disposition B: Alternative Approach (Hybrid — Sever the Severity)

#### Proposal

Migrate only `detectedissue-severity` (CodeSystem + ValueSet) to THO,
since that binding is the most likely to be reused outside FHIR (CDA /
V2 alerts and clinical-decision artifacts routinely carry a severity
concept) and is the most likely to evolve independently of a FHIR
release cycle. For the two example bindings —
`detectedissue-example-medication-class` (bound to
`DetectedIssue.medicationClass`) and `detectedissue-management-code`
(bound to `DetectedIssue.managementCode`) — leave the artifacts in
FHIR Core but explicitly mark both the CodeSystem and ValueSet as
`experimental = true`, and confirm the element bindings remain
`example` strength. Document in the resource notes that implementers
are expected to substitute their own vocabulary for these elements.

#### Justification

This mirrors Bryn Rhodes' working-group comment exactly and reflects
the FHIR governance convention that "example" value sets are
illustrative rather than normative — they don't carry the kind of
stable, cross-spec semantic that THO is intended to curate. Migrating
example-only vocabulary to THO would impose maintenance overhead on
the terminology editorial group without a corresponding reuse benefit,
and could mislead implementers into thinking the example codes are
normative. Marking them `experimental` makes the "do not rely on
these" expectation explicit while leaving the migration option open
for the future.

---

### Disposition C: Decline

#### Proposal

Decline the migration request and keep all three CodeSystem/ValueSet
pairs in FHIR Core as they are today. Optionally, document in the
DetectedIssue resource notes that the bindings are stable for the
upcoming release and that any future migration will be tracked under a
separate ticket.

#### Justification

Defensible if the work group concludes that (a) none of the three
value sets has demonstrated cross-spec reuse demand, (b) the editorial
overhead of two simultaneous publication locations (Core + THO) is not
justified by the speculative benefit, and (c) the existing core-only
location has not caused any implementer pain to date. Migration is
explicitly described in the ticket as "not required," so declining is
within the work group's prerogative — particularly if the upcoming
release cycle is already over-committed.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended because it aligns with the only piece of
work-group input on the record (Bryn's comment) and respects the
distinction FHIR already draws between normative-ish bindings and
illustrative ones. `DetectedIssue.severity` carries a real, reusable
concept that benefits from THO's release cadence and cross-spec
visibility; the medication-class and management-code value sets are,
by their own naming and binding strength, examples whose real-world
use will almost certainly be implementer-supplied vocabulary rather
than these placeholder codes. Migrating the examples to THO would
spend terminology-governance effort on artifacts that no one is
expected to use as published, while leaving them in core with an
explicit `experimental` flag clearly signals their status to
implementers. The work group can revisit migration for either example
value set later if cross-spec demand actually materializes.
