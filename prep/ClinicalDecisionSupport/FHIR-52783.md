## Summary

The reporter (Bryn Rhodes) identifies two related grammar gaps in the
proposed CQL conversion-definition syntax. First, the new grammar for
`define implicit conversion ... from <type> to <type> using <function>`
does not allow a *qualifier* in front of a parameterized generic type
such as `Interval<...>`. As a result, the natural form
`from FHIR.Period to System.Interval<System.DateTime>` cannot be parsed,
and the author must drop the `System.` qualifier from the target type
before the parser will accept it. The same restriction exists in the
current published grammar for ordinary type references in function
parameter lists — e.g., `define function y(x System.Interval<Integer>): 5`
fails for the same reason. Second, the `using` clause inside a
conversion definition is currently restricted to a *bare* (unqualified)
identifier, so a fully-qualified helper reference such as
`FHIRHelpers.ToInterval` cannot be written; only the unqualified
`using ToInterval` parses. The ticket asks the CQL grammar to permit
qualified identifiers in both places: in front of generic/parameterized
types (notably `Interval<...>`, but by implication `List<...>` and
`Tuple{...}` as well), and after `using` in a conversion definition.

## Details

**Description:**

The proposed grammar for defining conversions doesn't support qualifiers
in front of `Interval<...>` (even in the current version), so something
like `define function y(x System.Interval<Integer>): 5` breaks and
`System.` needs to be dropped for it to work.

This conversion definition example cannot be parsed unless `System.` is
removed:

```
define implicit conversion
  from FHIR.Period to System.Interval<System.DateTime>
  using FHIRHelpers.ToInterval
```

Another problem this example is highlighting is that `using` can only be
followed by an unqualified identifier, so it'll only work with
`using ToInterval`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CQL grammar (ANTLR `cql.g4` in `HL7/cql`) to allow qualified
identifiers in two specific positions:

1. **Generic/parameterized type specifiers.** Permit an optional
   `qualifier '.'` (model identifier, e.g., `System.`, `FHIR.`) in
   front of `Interval<...>`, `List<...>`, and `Tuple{...}` in every
   place a `typeSpecifier` is accepted — at minimum
   `intervalTypeSpecifier`, `listTypeSpecifier`, and
   `tupleTypeSpecifier`, and consequently in function parameter type
   declarations (`functionDefinition` parameters), conversion
   definition `from` and `to` clauses, cast expressions, and
   `is`/`as` operators. In the current grammar these productions
   accept only the base keyword (`Interval`, `List`, `Tuple`); add the
   same `qualifier?` prefix that `namedTypeSpecifier` already supports.

2. **`using` clause in `define [implicit|explicit] conversion`.** Replace
   the unqualified-identifier production with a `qualifiedIdentifier`
   (or `qualifiedFunctionInvocation`) so that helper references such
   as `FHIRHelpers.ToInterval` parse. Update the corresponding
   conversion-definition AST node to carry an optional library
   qualifier and update name resolution to look the identifier up in
   the qualifying library's exported function set.

Carry the change through to the CQL specification narrative (the
"Conversions" and "Type Specifiers" sections) and to the parser
reference implementation in `cqframework/clinical_quality_language` so
that the published grammar and the canonical parser stay in sync.

#### Justification

The reporter's example
(`from FHIR.Period to System.Interval<System.DateTime> using FHIRHelpers.ToInterval`)
is exactly the form the rest of CQL already encourages: model-qualified
type names and library-qualified function calls. Forcing authors to
drop `System.`/`FHIR.` qualifiers in some positions but not others
breaks the language's own consistency rules, makes generated CQL (e.g.,
from translators that always emit fully-qualified names) silently
invalid, and conflicts with the documented guidance that qualifiers
disambiguate types defined in multiple data models. Allowing
`qualifiedIdentifier` after `using` mirrors how every other function
invocation in CQL is written and removes an artificial restriction that
forces helper libraries to expose globally-unique function names.

---

### Disposition B: Alternative Approach — Targeted Fix Plus Spec Note

#### Proposal

Make a narrower grammar change that addresses only the two surfaces the
reporter explicitly hit, instead of generalizing across every
type-specifier production:

1. Allow an optional qualifier in front of `Interval<...>` *only* in
   the `from` and `to` clauses of `define implicit conversion` /
   `define explicit conversion`, by introducing a dedicated
   `conversionTypeSpecifier` rule in `cql.g4` rather than touching
   `intervalTypeSpecifier`/`listTypeSpecifier`/`tupleTypeSpecifier`
   globally.
2. Allow `qualifiedIdentifier` after `using` in conversion
   definitions, exactly as in Disposition A.
3. Add a non-normative note to the CQL specification clarifying that,
   for *function parameter* type specifiers, the bare form
   (`Interval<Integer>`, `List<FHIR.Period>`) is canonical and the
   `System.`-qualified form is intentionally not supported in this
   release; provide a migration guideline for translator authors that
   over-qualify generic types.

#### Justification

Generic type specifiers appear in many places in the grammar (function
parameters, casts, `is`/`as`, query `let`/`with`, etc.). Touching the
shared productions risks subtle ambiguities — particularly with
`namedTypeSpecifier` already accepting a qualifier, the union of
`qualifier? Interval '<' typeSpecifier '>'` may interact with
expression-level identifier resolution and existing translator
implementations. A scoped fix limited to the conversion-definition
clause solves the reporter's "I can't even write the example" problem
in this release while deferring the broader normalization until the
CQL editors' group has reviewed parser-implementation impact across
`cqframework/clinical_quality_language` and downstream tools.

---

### Disposition C: Decline

#### Proposal

Decline the change and document the existing requirement that, in
positions where a generic/parameterized `typeSpecifier` is expected
(`Interval<...>`, `List<...>`, `Tuple{...}`), no model qualifier is
permitted, and that the `using` clause of a conversion definition
takes a bare function identifier resolved against the current library's
imported function set. Close the ticket as "Persuasive with Mod —
documentation only", adding a clarifying paragraph to the "Conversions"
section of the CQL specification with an explicit example showing the
required form (`from FHIR.Period to Interval<DateTime> using ToInterval`).

#### Justification

The current grammar is internally consistent — `Interval`, `List`, and
`Tuple` are reserved type keywords in CQL, not user-defined named
types, so qualifying them with a model namespace is arguably a category
error. Likewise, the `using` clause historically refers to a function
within the conversion-definition's enclosing library scope, which is
why an unqualified identifier suffices. If the only problem is that
authors are *writing* `System.Interval<...>` and `FHIRHelpers.ToInterval`
out of habit, a documentation fix and a translator lint may be a
cheaper, lower-risk remedy than a grammar change that has to be picked
up by every CQL parser implementation.

---

### Recommendation

**Recommended disposition:** A

The reporter's request is small, internally consistent, and aligned
with how CQL is already written everywhere else: model-qualified type
names and library-qualified function calls. Disposition C
(documentation only) leaves a real authoring trap in place — code
generators and humans both naturally write `System.Interval<...>` and
`FHIRHelpers.ToInterval`, and the parser silently rejecting them is a
poor experience for an HL7 normative language. Disposition B reduces
risk by scoping the grammar change to conversion definitions only, but
the underlying inconsistency (function parameter types reject the same
qualifiers) is exactly the second example the reporter cites; fixing
only one of the two surfaces would force a follow-up ticket. The
parser-impact concerns that motivate B are real but tractable: the
change is additive (`qualifier?` prefix on three existing productions
plus a `qualifiedIdentifier` in one place), no existing valid CQL is
invalidated, and the reference implementation in
`cqframework/clinical_quality_language` can adopt the updated grammar
in lockstep with the spec. Recommend the workgroup direct the CQL
editors to make the changes described in Disposition A, with the
implementation plan to be drafted after `repo-analysis` briefings have
been refreshed for `HL7/cql` (and any in-scope reference-implementation
repos).
