## Summary

The reporter (Isaac Vetter) asks whether `GuidanceResponse` — and specifically
the `cdshooksguidanceresponse` profile in the HL7 *CDS Hooks Clinical
Reasoning* IG — should be extended to carry more of the CDS Hooks
`card.source` sub-elements, naming `icon` and `label` explicitly. The CDS
Hooks v2.0.1 specification defines `card.source` as an object with
`label` (required string), `url` (optional URL), `icon` (optional URL to a
100×100 PNG), and `topic` (optional Coding describing the card's subject).
Today the `cdshooksguidanceresponse` profile only constrains the existing
`GuidanceResponse` backbone and does not surface `icon` or `label`
directly, so a CDS service that wants to round-trip a card through a
FHIR-shaped representation cannot fully reconstruct `card.source` from the
profile.

There are no cross-referenced Jira tickets, GitHub items, or comments on
this ticket. A FhirAugury cross-reference query for `FHIR-53713` returned
zero hits. A keyword search across Zulip surfaces older `cds hooks`
discussions about `card.source.topic`, card identification, and card
metadata; none of those threads explicitly cite this Jira ticket, but
together they document the long-running history that produced today's
`card.source` shape.

### [\[cds hooks\] `card.source.topic` proposal](https://chat.fhir.org/#narrow/stream/cds%20hooks/topic/%60card.source.topic%60%20proposal/near/183174823)

Isaac Vetter opened this thread in December 2019 to discuss
[cds-hooks/docs#505](https://github.com/cds-hooks/docs/pull/505), which
introduced `card.source.topic`. Bryn Rhodes explained how Clinical
Reasoning already addresses identification (canonical URL + version),
topical organisation (`topic`), and applicability (`useContext`) on
metadata resources such as `PlanDefinition`, and supported giving cards
both a way to identify the underlying guidance (canonical URL) and a way
to describe what the guidance is about (a topic Coding). The thread
landed on supporting both use cases distinctly rather than overloading a
single element — i.e., `card.source.url` for canonical/source pointer
and `card.source.topic` for categorisation.

Participants: Isaac Vetter, Bryn Rhodes, Dennis Patterson.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `cdshooksguidanceresponse` profile in
`HL7/cds-hooks-clinical-reasoning` (the IG that publishes the page the
reporter cites) to expose all of the CDS Hooks `card.source`
sub-elements that are not already round-trippable from
`GuidanceResponse`:

- `card.source.label` → reuse `GuidanceResponse.note` poorly fits;
  introduce an extension `cdshooksGuidanceResponse-source-label`
  (string, 0..1, mandatory if a card.source is present, mirroring CDS
  Hooks).
- `card.source.url` → already mappable to a canonical pointer; ensure
  the profile pins it to the existing `subjectOf` / extension slot or
  add `cdshooksGuidanceResponse-source-url` (url, 0..1).
- `card.source.icon` → add `cdshooksGuidanceResponse-source-icon`
  extension (url, 0..1) with the CDS-Hooks-defined 100×100 PNG
  invariant restated as an `extension`-level constraint.
- `card.source.topic` → add `cdshooksGuidanceResponse-source-topic`
  extension (Coding, 0..1) and document its alignment with
  `PlanDefinition.topic` per Bryn Rhodes' 2019 guidance.

Apply the changes only inside the IG's authoring root (no FHIR Core
spec change is required because `GuidanceResponse` itself does not
need new core elements). Add a Mappings table entry on the profile
page that explicitly cross-walks `card.source.{label,url,icon,topic}`
to the new extensions, plus a worked example.

#### Justification

The reporter's underlying need — full round-trip between a CDS Hooks
card and the FHIR-shaped `GuidanceResponse` profile — is real and is
already documented in the cited threads as something implementers do
in production (suppression rules driven by `card.source.url` and
`card.source.topic`, attribution rendering driven by `label`/`icon`).
Doing this as profile-level extensions on the existing
`cdshooksguidanceresponse` profile (rather than on the base resource)
keeps the cost localised to the CDS Hooks Clinical Reasoning IG, where
the mapping already lives, and avoids changing FHIR Core for an
IG-specific concern. It also follows the "extensions on profile,
elevate later if used widely" pattern that Lloyd McKenzie advocated in
the *CDS Hooks feature request* thread.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mint per-`card.source.*` extensions on the
`cdshooksguidanceresponse` profile, define **one** complex extension
`cdshooksGuidanceResponse-card-source` whose sub-extensions exactly
mirror the CDS Hooks `card.source` object (`label` 1..1 string, `url`
0..1 url, `icon` 0..1 url, `topic` 0..1 Coding). Slice it 0..1 on
`GuidanceResponse` in the profile, and provide a normative mapping
table so a serialiser can round-trip the entire `card.source` JSON
object into and out of the extension without ambiguity. Keep
`GuidanceResponse` core untouched.

#### Justification

CDS Hooks treats `card.source` as a single object with co-varying
sub-fields (you cannot meaningfully have an `icon` without a `label`,
and CDS Hooks already requires `label` whenever `source` is present).
A single complex extension preserves that invariant directly via
required sub-extensions, makes it impossible to produce a profile
instance that omits the mandatory `label`, and gives implementers one
extension URL to look for instead of four. It is also easier to
version: when CDS Hooks adds another `card.source.*` field, only one
extension definition needs to be revised. Trade-off vs. Disposition A:
slightly more verbose JSON in the FHIR representation, and tooling
that flattens single-element extensions has to special-case the
complex extension.

---

### Disposition C: Decline

#### Proposal

Decline the change as out of scope for FHIR Core (the specification
this ticket is filed against). Note in the resolution that
`GuidanceResponse` core is not the right place for CDS-Hooks-specific
card metadata; redirect any profile-level work to
`HL7/cds-hooks-clinical-reasoning` and ask the reporter to file the
profile change request against that IG's tracker (or re-file with the
specification changed to the CDS Hooks Clinical Reasoning IG).

#### Justification

The ticket is logged against *FHIR Core (FHIR)* but the cited artifact
(`cdshooksguidanceresponse`) and the gap (CDS-Hooks-specific source
metadata) are intrinsically IG concerns. Adding `icon`, `label`, etc.
into the base `GuidanceResponse` would couple FHIR Core to a
particular consumer protocol (CDS Hooks) and would not benefit any
`GuidanceResponse` user outside CDS Hooks — exactly the kind of
overload Bryn Rhodes argued against in the *card.source.topic*
discussion when reasoning about why Clinical Reasoning kept
identification, topic, and useContext on the metadata resources rather
than baking them into every consumer. Declining at the Core level
keeps the responsibility where it belongs and avoids a breaking-change
risk for `GuidanceResponse` implementers who do not use CDS Hooks.

---

### Recommendation

**Recommended disposition:** B (with a Core-level decline of the form
in C: do the work in the IG, not in FHIR Core).

The reporter's need is concrete and the implementer-facing Zulip
history (`Deferring cards`, `Batch mode`, `Coding data type`,
`card.source.topic` proposal) makes a clear case that more of
`card.source` should be representable in the `cdshooksguidanceresponse`
profile. But the work belongs in `HL7/cds-hooks-clinical-reasoning`,
not in FHIR Core, so the ticket should be re-scoped from
"Specification: FHIR Core (FHIR)" to that IG. Within the IG, a single
complex extension (`card-source` with required `label` and optional
`url`, `icon`, `topic` sub-extensions) is preferable to four loose
extensions because it preserves CDS Hooks' co-variance rules, reduces
extension-URL churn as CDS Hooks evolves, and gives implementers an
unambiguous round-trip mapping. The CDS WG should also confirm whether
`card.source.topic` should be aligned with `PlanDefinition.topic`'s
ValueSet or left as a free Coding, picking up the unresolved 2019
thread.
