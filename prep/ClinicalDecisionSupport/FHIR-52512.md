## Summary

FHIR-52512 questions a sentence that has been added to the CQL (FHIR)
specification regarding model aliases: *"The local identifier for a referenced
model must be unqualified and unique within the artifact."* The reporter
(Hans Buitendijk) finds it unclear how or when this model alias would be used
and notes that introducing a model alias could itself be confusing to authors
who are working with multiple models in the same CQL library. The ticket is in
**Triaged** state with no comments yet.

The wording in question was introduced by [FHIR-48762](https://jira.hl7.org/browse/FHIR-48762)
("Add the ability to alias a model declaration"), which was resolved
**Persuasive with Modification** and is now **Applied** — that change extended
the model declaration to support a local alias the way `include` already
supported one for libraries. A companion request,
[FHIR-50214](https://jira.hl7.org/browse/FHIR-50214) ("Define an extension to
allow an alias to be specified"), defined a `cqf-libraryAlias` extension on
the FHIR Extensions Pack so the alias mechanism can also be carried on
artifact-level library references (`cqf-library`, `ActivityDefinition.library`,
`PlanDefinition.library`, `Measure.library`). Together those tickets establish
a model-alias / library-alias pattern, but FHIR-52512 is essentially asking the
CDS WG to clarify (or reconsider) the model-alias half of that pattern in the
spec text itself. A related, still-open ticket
[FHIR-52458](https://jira.hl7.org/browse/FHIR-52458) asks for an example of
using an *unqualified* model in the same data model intro section, which
suggests there is broader confusion in the surrounding prose that this
ticket's resolution should ideally be coordinated with.

### Disposition A: Accept As Requested

#### Proposal

Treat the existing sentence — *"The local identifier for a referenced model
must be unqualified and unique within the artifact."* — as a clarity defect
introduced by FHIR-48762 and rewrite it. Specifically, in the CQL (FHIR) Data
Model section that describes the `using` declaration's optional alias:

1. Replace the bare constraint sentence with a short paragraph that:
   - Names the construct ("model alias on a `using` declaration"), and
   - States the two normative constraints separately and explicitly:
     "(a) the alias **SHALL** be an unqualified identifier (no `.`-qualified
     names), and (b) each model alias **SHALL** be unique within the
     artifact."
2. Add a one-sentence rationale immediately after, in non-normative prose,
   explaining *why* the alias exists — disambiguating multiple `using`
   declarations whose qualified model names share an unqualified suffix,
   parallel to library `include … called` — citing the same motivation
   recorded in FHIR-48762 and the library-alias pattern referenced in
   FHIR-21996.
3. Add a minimal CQL example showing two `using` declarations of models with
   colliding unqualified names disambiguated via aliases (and used in a
   `[Alias.Resource]` retrieve).

#### Justification

The reporter's complaint is not with the *constraint* but with its lack of
context: the spec asserts a rule about an alias before motivating the alias
itself. Splitting the normative SHALLs from a non-normative "why" paragraph,
and adding an example, follows the documentation pattern already used for
`include … called` in the CQL (and CQF) specs and aligns with the QM
guidance accepted as Persuasive in FHIR-21996. It also leaves the
already-applied FHIR-48762 mechanism intact, which avoids reopening a
resolved change. Coordinating this with FHIR-52458's request for an
unqualified-model example in the same intro lets the section read coherently
end-to-end.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the alias *capability* introduced by FHIR-48762 but narrow its
recommended use to the disambiguation case, by changing the spec's stance
from "may be used" to "SHOULD be used only when needed." Concretely:

1. In the same Data Model section, restructure the alias discussion as
   guidance with two bullets:
   - *Default*: Authors **SHOULD** use the unqualified model name and omit
     the alias when the artifact references only one model with that
     unqualified name. (This is the case FHIR-52458 asks be documented.)
   - *Disambiguation*: Authors **SHALL** declare a model alias when two or
     more `using` declarations in the same artifact would otherwise resolve
     to the same unqualified local identifier; in that case the alias
     **SHALL** be unqualified and unique within the artifact.
2. Add the same example as in Disposition A, but explicitly contrast a
   single-model artifact (no alias) with a multi-model artifact (aliases
   required).
3. Cross-link the section from the `cqf-libraryAlias` extension definition
   (FHIR-50214) so readers connect the CQL-level construct with the
   FHIR-level extension.

#### Justification

This disposition addresses the reporter's actual concern — that introducing a
model alias *gratuitously* may itself be confusing — by making "no alias" the
recommended default and reserving aliases for the case they were designed
for. It harmonizes naturally with FHIR-52458 (which wants the unqualified
case explicitly documented) and with FHIR-21996 (which established the
alias-on-collision precedent for libraries). It is also cheaper to implement
than a normative change to FHIR-48762: it tightens guidance without removing
the mechanism, so existing CQL processors that already implement model
aliases per FHIR-48762 remain conformant.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** on the grounds that (a) the wording, while
terse, is normatively correct and consistent with the parallel
`include … called` construct that has been in CQL since the original
specification, and (b) the underlying mechanism was deliberately introduced
by the recently-applied FHIR-48762, with that ticket's discussion serving as
the authoritative motivation. Direct the reporter to FHIR-48762 for the
rationale and close FHIR-52512 without a spec change. Optionally fold any
clarifying example into FHIR-52458 instead.

#### Justification

The constraint sentence the reporter quotes is short but unambiguous, and
the alias mechanism it governs was vetted through ballot via FHIR-48762
(Persuasive with Modification, Applied). Reopening prose around a
just-applied change, without new technical content, risks churn for
implementers who have already updated to support model aliases. If the only
substantive ask is "show an example," that is exactly what FHIR-52458 is
already tracking and can be handled there without a second editorial pass on
the same paragraph.

---

### Recommendation

**Recommended disposition:** B

Disposition B best honors the reporter's actual concern. The reporter is not
disputing FHIR-48762's mechanism; they are flagging that the spec, as
worded, lands a constraint on a feature whose purpose isn't motivated and
whose default usage isn't characterized. Disposition A fixes the prose but
leaves the door open to confused authors aliasing every `using` declaration
"because the spec says they can." Disposition B both fixes the prose *and*
sets the right default — alias only when disambiguation requires it — which
also aligns with the related, open FHIR-52458 request to document the
unqualified case in the same section. It preserves the FHIR-48762 mechanism
(no reopen) and matches the precedent set by FHIR-21996 and the
`cqf-libraryAlias` work in FHIR-50214. The CDS WG can then close FHIR-52512
and FHIR-52458 with a single coordinated edit to the Data Model intro.
