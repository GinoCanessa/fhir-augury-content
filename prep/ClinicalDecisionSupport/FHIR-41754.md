## Summary

The ticket asks the CQL specification to clarify the semantics of the
"compile-time" requirement that currently governs default values for CQL
parameters. The current text says, in effect, that the default value of a
parameter "must be able to be calculated at compile-time," but it does not
state the stronger, intended invariant: the value computed at compile-time
must be the same as the value that would be computed at run-time.

The motivating concern is built-in date/time functions such as `Today()`,
`Now()`, and `TimeOfDay()`. These are technically evaluable at compile-time
(the compiler could substitute a literal), but in general the compile-time
value will not equal the run-time value — `Now()` is required by CQL to be
constant only within a single execution, not across compilations. Using
them as parameter defaults would therefore produce defaults that drift from
the intuitive run-time semantics of those functions, especially for
distributed/precompiled ELM artifacts.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub items
for this issue in Augury, so the request stands alone.

## Details

**Description:**

> Current language states:
>
> > In other words, the value for the default of a parameter must be able to
> > be calculated at compile-time.
>
> But this does not make clear the intended semantics that the value that is
> calculated at compile-time must be the same as the value that would be
> calculated at run-time.
>
> The specific case in question is the use of `Today()`, `Now()`, and
> `TimeOfDay()`, because although they could in theory be calculated at
> compile-time, they would in general not be the same value as what would
> be calculated at run-time, at least per run.

**Comments:**

- **JIRAUSER23970 (2023-08-14)** — Notes that `Now()` (and derived
  functions) is required by the spec to remain constant *during the period
  of execution* of the CQL (i.e., at run-time). Argues that fixing `Now()`
  at compile-time would be surprising and would significantly reduce the
  usability of ELM: for example, a rolling window like "encounters during
  the last 2 days" would require continuous recompilation for the value to
  update, and distribution of pre-compiled ELM artifacts would become much
  harder. Proposes that the cleanest resolution is to clarify that
  "compile-time" here means anything that could be calculated at
  compile-time *and* would also remain constant at run-time. That
  formulation forbids built-ins like `Now()` in default parameter values
  while leaving all other behavior unchanged.

- **JIRAUSER25154 (2023-10-31)** — Observes that most languages have a
  clear concept of what constitutes a constant (linking the C# `const`
  documentation) and suggests CQL stay aligned with prior art by simply
  declaring that parameter default values can only be *literal
  expressions*.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Clarify the CQL specification's parameter-declaration text so that the
"compile-time" requirement on default values reads (substantively):

> The default value of a parameter must be an expression that can be
> evaluated at compile-time *and* whose compile-time value is the same as
> the value that would be produced by evaluating the same expression at
> run-time.

Add a non-normative note giving `Now()`, `Today()`, and `TimeOfDay()` as
the canonical examples of expressions that are *not* permitted as
parameter defaults, even though they appear evaluable at compile-time,
because their run-time value is not guaranteed to equal their
compile-time value. Translators should be expected to surface this as a
compile error when such an expression is used as a parameter default.

#### Justification

This matches the semantic Bryn Rhodes describes in the second comment and
is the smallest, most surgical change that closes the actual ambiguity
the ticket flags. It preserves the existing CQL contract that `Now()`
and friends are constant *within* an execution but free to change
*between* executions, which is essential for distributing pre-compiled
ELM artifacts (e.g., measure logic). It also preserves all currently
legal default expressions (literals, arithmetic over literals, code/code
system references, etc.).

---

### Disposition B: Alternative Approach

#### Proposal

Replace the "compile-time" wording entirely with a stricter,
language-aligned rule: a parameter default must be a **literal
expression** (in the sense already defined by the CQL grammar — numeric,
string, boolean, date/time, quantity, ratio, code, concept, interval,
list, and tuple literals composed only of other literal expressions).
Built-in function invocations, including `Now()`/`Today()`/`TimeOfDay()`,
would be disallowed, as would arithmetic and any other operator
applications.

#### Justification

This is the position taken by the first commenter (JIRAUSER25154) and is
attractive because it is trivially specifiable, trivially implementable,
and consistent with how mainstream languages (e.g., C# `const`) define
compile-time constants. It removes any judgment call about whether a
given expression is "the same value at compile-time and run-time." The
trade-off is that it would be a behavior change rather than a pure
clarification: any conformant translator that today accepts non-literal
defaults (e.g., `1 + 1`, `Interval[1, 10]`, code/concept references built
from named codes) would now be required to reject them, potentially
breaking existing CQL libraries. That is a heavier lift than the ticket
asks for and warrants its own ticket and migration story.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive" / "Already Specified" on the basis
that the intent is already clear from CQL's broader semantic model:
parameter defaults are part of the artifact's compile-time form, and
`Now()`/`Today()`/`TimeOfDay()` are defined to be constant only for the
duration of a single execution, so by construction they cannot satisfy
the requirement to be "calculated at compile-time" with the same value
across runs. Implementers should already be reading the spec this way.

#### Justification

This is defensible only if the workgroup believes the existing wording
is unambiguous to a careful reader and that no normative change is
warranted. It minimizes spec churn and avoids any risk of inadvertently
constraining legal default expressions. However, the ticket exists
precisely because the wording was *not* clear to a careful reader, and
declining without a clarifying note would likely see the same question
raised again.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly addresses the ambiguity the reporter (and the
discussion in the comments) identified, costs essentially nothing in
terms of spec impact, and preserves the existing semantics of `Now()`
and pre-compiled ELM distribution that the ecosystem depends on. It
explicitly rules out the problematic case (`Now()` / `Today()` /
`TimeOfDay()` as defaults) without forbidding existing legal patterns
such as arithmetic-over-literals, interval/list/tuple literal
constructions, and named code/concept references — patterns that
Disposition B would silently break and that the ticket does not call
out as problematic. Disposition B is worth revisiting as a follow-on
ticket if the workgroup decides CQL should adopt a strict
literal-expression rule for defaults, but it should not be bundled into
this technical correction. Disposition C is unattractive because the
ambiguity is real and a small editorial change resolves it cleanly.
