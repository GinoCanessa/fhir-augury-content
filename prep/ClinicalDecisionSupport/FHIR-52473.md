## Summary

The reporter, Chris Moesel, points out that the CQL (FHIR) specification's
documentation of the `EquivalentContains` (`~contains`) operator currently
illustrates only one null edge case — a non-null list tested against a `null`
right-hand side, yielding `false`:

```
define "EquivalentContainsNullIsFalse": { 'A', 'B', 'C' } ~contains null
```

This example does not exercise the more interesting (and more likely to
confuse implementers) edge case: when `null` itself is a member of the list
on the left-hand side. Under equivalence semantics, `~contains` treats two
nulls as equivalent, so the result becomes `true`. The ticket therefore asks
the spec to add a companion example such as:

```
define "EquivalentContainsNullIsTrue": { 'A', 'B', 'C', null } ~contains null
```

A follow-up comment by the reporter clarifies that the relevant location is
*Appendix B - Reference* (rather than the page originally noted on the
ticket). The change is purely documentary/illustrative — no normative
behavior change is implied; it just makes the asymmetry between
`equal`/`contains` (propagates null) and `equivalent`/`~contains` (treats
nulls as equivalent) easier to see at a glance. There are no
cross-referenced Jira tickets, Zulip threads, or GitHub items associated
with this ticket in Augury.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the CQL (FHIR) specification's *Appendix B - Reference* section for the
`EquivalentContains` (`~contains`) operator, add a second worked example
immediately adjacent to the existing `EquivalentContainsNullIsFalse`
example, exactly as suggested in the ticket:

```
define "EquivalentContainsNullIsTrue": { 'A', 'B', 'C', null } ~contains null
```

Include a one-sentence note that this evaluates to `true` because, under
equivalence semantics, two `null` values are considered equivalent — in
contrast to the existing `…NullIsFalse` example, where the left-hand list
contains no `null` member. No grammar, normative semantics, or
implementation conformance text changes; this is an additive
documentation-only change to the operator's example block.

#### Justification

The change is small, additive, and entirely consistent with how the rest of
Appendix B documents operator edge cases (paired examples that contrast
truthy and falsy outcomes). The two examples together make the distinction
between `contains` (null-propagating) and `~contains` (null-equivalence)
immediately visible without forcing the reader to reason about it from the
prose. There is no risk of breaking implementations because nothing
normative changes — only an illustrative example is added. The reporter is
a long-standing CQL implementer/editor (Chris Moesel), and the requested
example reflects a real edge case that implementers and authors regularly
encounter.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than just appending one more example, restructure the
`EquivalentContains` reference entry to present a **small truth-table-style
block** that covers the matrix of null cases the operator is sensitive to,
e.g.:

| Left list             | Right value | `~contains` result |
|-----------------------|-------------|--------------------|
| `{ 'A', 'B', 'C' }`   | `'A'`       | `true`             |
| `{ 'A', 'B', 'C' }`   | `null`      | `false`            |
| `{ 'A', 'B', 'C', null }` | `null`  | `true`             |
| `{ }`                 | `null`      | `false`            |

Keep the existing two `define` examples (the current one plus the one
requested in the ticket) as worked CQL forms below the table. Optionally
add a parallel "compare with `contains`" callout that shows the same rows
under standard equality semantics, so the contrast between the two
operators is explicit on the page. Apply the same pattern to other
equivalence-vs-equality operator pairs in Appendix B over time, but only
for `EquivalentContains` in this ticket.

#### Justification

The underlying need the reporter raised is that the *single* current
example under-specifies the operator's null behavior. A second example
fixes the immediate gap, but a small matrix fixes the whole class of "what
does it return when…?" questions in one place and removes the need for
implementers to mentally interpolate between examples. The cost is modest
(a handful of additional rows in the appendix, no normative change), and
it sets a reusable documentation pattern for the other equivalence
operators that have the same null-handling subtlety. Trade-off vs.
Disposition A: more editorial work and a slightly larger diff to
*Appendix B - Reference*, in exchange for clearer, more
self-contained reference material.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that the operator's null semantics are
already specified normatively elsewhere in Appendix B (and in the operator
description prose), and the additional example does not change conformance
or interoperability. Close as **Persuasive with Modification → Not
Persuasive** (or equivalent CDS workgroup outcome) with a note that
implementers should consult the normative text and that further worked
examples are out of scope for the current cycle.

#### Justification

A defensible argument for declining is purely scope/cost: editing
*Appendix B - Reference* to add example blocks accumulates pages of
illustrative material that has to be maintained alongside the normative
grammar, and every example becomes a potential source of drift if the
operator semantics are later refined. If the workgroup is trying to keep
Appendix B compact and is already comfortable that the normative text
unambiguously defines `~contains` behavior over null, a "no change
required" disposition is internally consistent. This is the weakest of the
three options because the request is small, additive, doesn't change
behavior, and is requested by an active implementer — but it is a
legitimate disposition the workgroup may want to consider for completeness.

---

### Recommendation

**Recommended disposition:** A

This is a small, additive, documentation-only change requested by an
experienced CQL implementer to address a real and frequently-encountered
edge case (null on both sides of `~contains`). Disposition A delivers
exactly what was asked for, with essentially zero risk to implementers and
no normative impact, and is consistent with the existing paired-example
style already used elsewhere in Appendix B. Disposition B is attractive as
a longer-term editorial improvement, but bundling a truth-table refactor
into a ticket that asked for a single example risks scope creep and
delays a fix that the reporter has already framed precisely. Disposition C
is hard to defend on cost grounds for a one-line example. Accept as
requested, and consider the broader Appendix B harmonization (Disposition
B) as a separate editorial ticket if the workgroup wants it.
