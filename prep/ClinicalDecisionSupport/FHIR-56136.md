## Summary

The reporter (Bryn Rhodes) asks the CQL specification to call out, explicitly,
how its rules for handling absent timezone offsets relate to — and differ from —
the corresponding FHIRPath rules. FHIRPath (per the
[datetime-equality section](https://build.fhir.org/ig/HL7/FHIRPath/#datetime-equality))
treats the use of a default timezone offset as a per-implementation **policy
decision**: a system may, but is not required to, supply a default offset when a
value carries none. CQL is stricter: when a DateTime has no timezone offset and
that value participates in a comparison or calculation involving time
components, the offset **SHALL** default to the offset of the evaluation
request. The ticket is asking the CQL spec to add a brief, explicit note that
this CQL behavior is *consistent with* FHIRPath's allowed policy space but
*tighter than* what FHIRPath itself mandates, so that readers don't read CQL's
SHALL as contradicting FHIRPath's discretionary wording.

Closely related context (no formal cross-references exist for this ticket, but
the related-by-keyword search surfaces these prior tickets):
- [FHIR-48530](https://jira.hl7.org/browse/FHIR-48530) — already added the
  defaulting clarification to the CQL DateTime type description (Published,
  Persuasive). FHIR-56136 builds on that by asking for an explicit pointer
  to the FHIRPath policy that CQL is constraining.
- [FHIR-20311](https://jira.hl7.org/browse/FHIR-20311) — the original FHIRPath
  ticket that turned defaulting into a policy decision (Applied, Persuasive).
- [FHIR-18607](https://jira.hl7.org/browse/FHIR-18607) — established the
  FHIRPath datetime equality / "respecting timezone offset" semantics.
- [FHIR-56046](https://jira.hl7.org/browse/FHIR-56046) — companion ticket on
  the Using-CQL-With-FHIR IG side asking for evaluation-time timezone guidance
  (e.g., `client-timezone` header, `evaluationRequestTimestamp` parameter).

## Details

**Description:**

> The FHIRPath specification indicates that systems may choose policies for timezone offset handling:
>
> https://build.fhir.org/ig/HL7/FHIRPath/#datetime-equality
>
> However, the CQL specification is clear that if no timezone offset is present, the offset SHALL be defaulted to the timezone offset of the evaluation request when the value is used in a comparison or calculation that involves time components.
>
> Clarify that this is a requirement of the CQL specification that is consistent with FHIRPath, but tighter than the constraints established by FHIRPath.

The FHIRPath ticket (Applied, Persuasive with Modification) that established

the CQL spec is valuable to implementers consuming both.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a short normative-style clarifying note to the CQL specification's
DateTime reference (the same section updated under FHIR-48530, in the HL7/cql
authoring root that produced commit `a9a03f5`) that explicitly contrasts
CQL's timezone-offset defaulting requirement with FHIRPath's policy-based
allowance. Suggested wording:

> Note: FHIRPath treats supplying a default timezone offset for DateTime
> values that lack one as a per-implementation policy decision (see
> [FHIRPath §DateTime equality](https://build.fhir.org/ig/HL7/FHIRPath/#datetime-equality)).
> CQL constrains this allowed policy space: when a DateTime value lacks a
> timezone offset and is used in a comparison or calculation involving time
> components, the offset **SHALL** default to the timezone offset of the
> evaluation request. CQL's behavior is therefore consistent with FHIRPath
> but tighter than what FHIRPath itself requires.

The note should be placed adjacent to the paragraph added under FHIR-48530,
and a parallel cross-reference should be added in the Author's Guide
(`02-authorsguide.html#simple-values`) so both reader paths see it.

#### Justification

This is precisely what the reporter requested, and it is a low-risk editorial
clarification rather than a normative behavior change — the SHALL itself is
already in CQL (codified by FHIR-48530), and FHIRPath's policy-decision wording
is already in FHIRPath (codified by FHIR-20311). Making the relationship
explicit in CQL prevents the very common implementer reading where a
side-by-side reading of the two specs looks contradictory ("FHIRPath says I
*may* default, CQL says I *SHALL* default — which is right?"). The implementation
survey captured under FHIR-18607 (four reference implementations producing
three different answers for the same equality survey) is direct evidence that
ambiguity in this area causes real interoperability divergence; an explicit
"tighter than" framing reduces the cognitive load on every CQL implementer who
also implements FHIRPath. As an editorial-only change, it is also publish-stream
safe (Persuasive, no implementation impact).

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) adding a prose note in the CQL DateTime
reference, introduce a dedicated **"Relationship to FHIRPath"** subsection
in the CQL specification (or expand the existing FHIRPath-relationship
section if one exists) that catalogues *all* places where CQL tightens, mirrors,
or extends FHIRPath semantics. The timezone-offset clarification requested by
FHIR-56136 becomes the first row of that table; rows for the related
clarifications already covered by FHIR-48530 (DateTime defaulting) and
FHIR-18607 (offset-respecting equality) — and the duration/UCUM clarifications
captured by tickets like FHIR-14089 — would follow. This work would happen in
the same HL7/cql authoring root, but in a structural section rather than as
an inline note.

#### Justification

The reporter's underlying concern — that readers can't tell whether CQL's
SHALL contradicts FHIRPath's MAY — is not unique to timezone offsets. CQL
tightens or constrains FHIRPath in several places, and historically each of
those tightenings has surfaced as its own ballot ticket only after some
implementer was confused. A consolidated relationship section would address
the timezone case (satisfying FHIR-56136) while pre-empting the next
N analogous tickets, and it would give readers a single canonical place to
look. The trade-off is scope: this is more editorial work than the reporter
asked for, and a cataloguing exercise tends to expose new questions ("does
CQL really tighten X, or just present it differently?"). It also delays the
narrow timezone clarification while the larger section is drafted, which is
a regression for the implementer pain point that motivated the ticket.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** on the grounds that the relationship is
already adequately covered: FHIR-48530 added the explicit CQL defaulting rule,
FHIR-20311 established the FHIRPath policy stance, and the CQL specification
already declares itself a profile/superset of FHIRPath in its introduction.
A reader who consults both specs and the existing CQL profile-of-FHIRPath
language can derive the "consistent with but tighter than" conclusion without
additional spec text. Recommend that the reporter raise it instead as a
documentation/explanatory note in the *Using CQL With FHIR* IG (where
FHIR-56046 is already collecting evaluation-time guidance), keeping the base
CQL spec free of meta-commentary about FHIRPath.

#### Justification

There is a genuine argument that adding "this is consistent with X but
tighter than X" prose every time CQL tightens FHIRPath risks turning the
spec into running commentary on a sibling spec, which can age badly when the
sibling spec changes. Operational/implementer-facing clarifications often
belong in IGs and implementer guides rather than the base spec. However,
this disposition is the weakest of the three because (a) the reporter is the
CQL spec lead and is requesting it directly, (b) FHIR-18607's implementation
survey shows that the ambiguity does cause real divergence, and (c) the cost
of the addition is two or three sentences of editorial text — not a
maintenance burden of any consequence.

---

### Recommendation

**Recommended disposition:** A

The request is narrow, the wording change is small, the change is purely
editorial/clarifying (the underlying SHALL is already normative in CQL via
FHIR-48530), and it directly addresses an interoperability ambiguity for
which there is documented historical evidence (FHIR-18607's four-implementation
survey, three different answers). Disposition B's consolidated
"Relationship to FHIRPath" section is attractive in principle but is
disproportionate to what the ticket is actually asking for and risks delaying
the narrow fix; if the work group wants that broader cataloguing, it should
be a separate ticket so FHIR-56136 can land cleanly. Disposition C
under-weights the practical value of an explicit "tighter than" framing
inside the CQL spec itself, where CQL implementers actually look. The
disposition should be coordinated with [FHIR-56046](https://jira.hl7.org/browse/FHIR-56046)
so the IG-level evaluation-time guidance and the spec-level clarification
land with consistent terminology (`evaluation request timestamp`,
`client-timezone`, etc.). Pending availability of an HL7/cql `repo-analysis`
briefing, the implementing change should be made in the same authoring root
and section that FHIR-48530 touched (HL7/cql, commit `a9a03f5` provides the
precedent path).
