## Summary

FHIR-52460 is a documentation-only change request against the **Clinical
Quality Language (CQL) FHIR** specification, in the *Multiple Data Models*
section. The existing example explains why an unqualified identifier
`Quantity` resolves unambiguously to `FHIR.Quantity` when only the FHIR
model is explicitly declared (because the CQL `System` model is implicit).
The reporter (Chris Moesel) observes that newcomers to CQL are likely to
also need the inverse guidance: how to refer to the **CQL `System`
`Quantity`** type when that is what they actually want. He proposes adding
one sentence indicating that the fully qualified identifier
`System.Quantity` should be used.

This ticket is a direct follow-on to **[FHIR-46435](https://jira.hl7.org/browse/FHIR-46435)**
("Clarify implicit system model resolution semantics", Published, reported
by Bryn Rhodes), which introduced the very example FHIR-52460 wants to
augment — that ticket established the rule that the implicit `System`
declaration is not considered when determining ambiguity. FHIR-52460 simply
asks to round out the same example with the disambiguation pattern for the
opposite case. It is also one of a small cluster of editorial CQL spec
tickets filed by the same reporter at the same time
([FHIR-52461](https://jira.hl7.org/browse/FHIR-52461) — typo in *Defining
Models*, Resolved – change required;
[FHIR-52462](https://jira.hl7.org/browse/FHIR-52462) — broader
clarification needs in *Defining Models*).

## Details

**Description:**

> In the last example of the section on *Multiple Data Models*, it shows
> CQL that references the `Quantity` type and says the following:
>
> > The identifier Quantity in this function declaration resolves to
> > FHIR.Quantity unambiguously because only the FHIR model is explicitly
> > declared.
>
> People who are new to CQL might appreciate guidance regarding how to
> address this situation when they *want* the System type. In that case,
> I suggest adding a sentence like the following:
>
> > In order to refer to the CQL `System` `Quantity` type instead, use
> > the fully qualified identifier `System.Quantity`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's wording essentially verbatim. In the *Multiple Data
Models* section of the Clinical Quality Language (FHIR) IG, immediately
after the existing sentence

> The identifier `Quantity` in this function declaration resolves to
> `FHIR.Quantity` unambiguously because only the FHIR model is explicitly
> declared.

append a new sentence:

> In order to refer to the CQL `System` `Quantity` type instead, use the
> fully qualified identifier `System.Quantity`.

This is a documentation-only, non-normative change scoped to a single
example block. No CQL grammar, model info, or conformance behavior is
altered.

#### Justification

The change is symmetric with the existing guidance: the section already
explains *why* `Quantity` resolves to `FHIR.Quantity`; explaining *how* to
opt into `System.Quantity` completes the explanation for a reader who
encounters this ambiguity in practice. It re-uses the same example, which
keeps cognitive load low. It is consistent with the rule established by
FHIR-46435 (the implicit `System` model is not used for ambiguity
resolution) — the qualified identifier `System.Quantity` is the only
reliable way to reach the system type once a non-System model has been
explicitly declared. The change has no normative impact and aligns with
the workgroup's recently-demonstrated willingness (FHIR-52461 is already
"Resolved – change required") to apply small editorial improvements in
this area.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent, but expand the edit slightly so the guidance generalises
beyond `Quantity`. Instead of adding a single `System.Quantity`-specific
sentence, add a short note in the same example block such as:

> More generally, when an identifier names a type that exists in both the
> implicit `System` model and an explicitly-declared model (for example,
> `FHIR`), the `System` type can be referenced using its fully qualified
> identifier (e.g., `System.Quantity`, `System.Code`, `System.Concept`).

Optionally, add a forward reference from this paragraph to the *Defining
Models* / model-aliasing material introduced by FHIR-48762, so readers see
how qualified-name disambiguation interacts with model aliases.

#### Justification

`Quantity` is the most visible collision but not the only one — `Code`,
`Concept`, `Ratio`, `String`, `Integer`, etc. exhibit the same potential
shadowing once any non-System data model is in scope. A one-off sentence
about `System.Quantity` answers the immediate question but invites the
next ticket asking for `System.Code`/`System.Concept` guidance. Stating
the general rule once, with `Quantity` as the worked example, addresses
the underlying need with effectively the same edit footprint. The trade-off
versus Disposition A is minor scope creep in wording; the trade-off versus
Disposition C is a small amount of additional editorial review. Pairing
this with FHIR-52462 (the broader *Defining Models* clarification ticket
from the same reporter) is also natural, since both touch the same
chapter.

---

### Disposition C: Decline

#### Proposal

Close as "not persuasive" / declined on the grounds that the rule is
already implicit in the surrounding text: the section explains qualified
type identifiers (`<model>.<Type>`) elsewhere, and an attentive reader can
infer that `System.Quantity` is the qualified form for the implicit
`System` model. No change to the spec.

#### Justification

The CQL FHIR IG is a dense normative document and every example carries
maintenance cost. The existing example was added specifically to
illustrate the resolution rule established by FHIR-46435; expanding it
risks blurring its focus. A reader who needs `System.Quantity` can deduce
the syntax from the general qualified-identifier rules already documented
in CQL itself.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach).

The reporter is correct that the existing example leaves an obvious
follow-up question unanswered, and the workgroup has already shown
willingness to make exactly this kind of editorial fix (see FHIR-52461,
already Resolved – change required, from the same reporter on the
adjacent section). Disposition C undersells the cost to first-time
readers — qualified vs. unqualified type resolution is a known
stumbling block in CQL, not just for `Quantity`. Disposition A solves
the immediate problem but invites a stream of follow-on tickets for
`System.Code`, `System.Concept`, etc. Disposition B closes the gap with
a single generalising sentence at essentially no extra editorial cost,
keeps the `Quantity` worked example intact, and composes cleanly with
FHIR-52462 if the WG wants to handle the *Multiple Data Models* /
*Defining Models* clarifications in one editorial pass.
