## Summary

The reporter is asking whether the curly-bracket fragment `{color:#FF0000}}`
that appears in a sentence in the Clinical Quality Language (FHIR)
specification carries any technical meaning. The sentence in question reads:

> "None of the features it uses are changed in R4B, so it can be used as is
> with R4B`{color:#FF0000}}` systems."

This is a documentation/editorial question rather than a technical change
request: the reporter wants to know whether the curly-bracket text is
intentional notation (e.g., a placeholder, FHIRPath fragment, or CQL syntax)
or an error in the published spec. There are no cross-referenced Jira
tickets, Zulip threads, or GitHub items associated with this ticket — it
stands alone.

The string `{color:#FF0000}` is Atlassian Jira / Confluence wiki markup that
renders the enclosed text in red. The most likely interpretation is that an
author drafted the sentence in a tool that supports Jira/Confluence color
markup, intended to highlight a word in red, and the markup leaked through
to the published HTML of the CQL-on-FHIR spec instead of being converted to
formatting (and the trailing `}}` plus the empty `<font color="">` tags
suggest the closing tag was also mangled).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation defect in the Clinical Quality Language
(FHIR) specification. In the affected narrative passage of the CQL-on-FHIR
IG (the sentence beginning "None of the features it uses are changed in
R4B…"), remove the stray Jira/Confluence wiki markup so the sentence reads
cleanly:

> "None of the features it uses are changed in R4B, so it can be used as is
> with R4B systems."

Concretely:

- Strip the literal `{color:#FF0000}}` token.
- Remove the empty `<font color=""></font>` tag that follows it (also a
  rendering artifact, not valid styling).
- If the original author's intent was to emphasise "R4B" (or "systems"),
  replace the markup with a supported emphasis mechanism for the IG
  source (e.g., `<em>` / `<strong>` in HTML, or the Markdown/AsciiDoc
  equivalent used by the CQL-on-FHIR IG sources), rather than re-introducing
  Jira-flavoured markup.
- Reply on the ticket confirming that the curly-bracket fragment has **no**
  technical significance — it was unintentional Jira wiki markup that
  leaked through to the rendered spec.

#### Justification

The fragment `{color:#FF0000}` is unambiguously Atlassian wiki markup for
red-coloured text; it has no defined meaning in CQL, FHIRPath, or the FHIR
specification narrative grammar. Leaving it in place is misleading to
readers who may assume it is normative syntax (which is exactly the
confusion the reporter has surfaced). Fixing the narrative is low-risk,
non-breaking, and directly answers the reporter's question.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the spirit of the request (it is a documentation defect) but use it
as the trigger for a small, generalised editorial pass rather than a single
inline fix. Specifically:

- Search the published CQL-on-FHIR IG output (and, where feasible, its
  source files) for other occurrences of leaked Jira/Confluence markup
  patterns: `{color:`, `{panel`, `{noformat`, `{code:`, `{quote}`, stray
  `||` table separators, and empty `<font color="">` shells.
- Replace each occurrence with the IG-native equivalent (or remove it if
  the highlight is no longer relevant).
- Add a short note to the IG's authoring/contributor guidance reminding
  contributors that Jira-flavoured wiki syntax does not render in the
  published IG and must be converted before merge.
- Reply on the ticket pointing the reporter at the corrected passage and
  the broader cleanup.

#### Justification

The fact that this artifact reached publication suggests the authoring
workflow does not catch leaked wiki markup. A one-line fix resolves the
immediate confusion but leaves a class of similar defects in place. A
narrow sweep is still cheap (the patterns are highly distinctive and
greppable) and prevents repeat tickets of the same shape. The trade-off
versus Disposition A is a slightly larger change footprint and a bit more
reviewer time, in exchange for higher confidence that the spec narrative
is clean.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** / **Question — Answered** without a
spec change: respond on the ticket explaining that the curly-bracket string
is leftover Jira wiki markup with no technical meaning, and direct the
reporter to ignore it. No edit is made to the published CQL-on-FHIR IG.

#### Justification

The strongest defensible reason to decline is that the underlying
specification semantics are unaffected — the sentence is still
intelligible and the surrounding paragraph conveys the intended R4B
compatibility statement. CDS WG time is finite and clearly-cosmetic
narrative defects can reasonably be deferred. However, this rationale is
weak: the reporter explicitly raised the fragment because it *looked*
significant, which is direct evidence that the artifact harms reader
comprehension. Decline is included for completeness but is the least
defensible of the three options.

---

### Recommendation

**Recommended disposition:** A

The ticket is a small, well-scoped editorial defect with a known root cause
(leaked Jira wiki markup) and a low-risk fix. Disposition A directly
answers the reporter, removes the misleading characters from the published
CQL-on-FHIR narrative, and costs essentially nothing in implementer or
reviewer impact. Disposition B is attractive as a follow-on housekeeping
task but is broader than what this single ticket asks for and risks
scope-creep; the workgroup can spawn a separate tracker for the wider
sweep if it wants the cleanup. Disposition C should be avoided: the
ticket itself is proof that the stray markup confuses readers, which is
exactly the kind of low-cost narrative bug the spec process is meant to
catch.
