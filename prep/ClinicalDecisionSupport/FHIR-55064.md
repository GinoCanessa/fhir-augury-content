## Summary

This ticket flags that the `Evidence` resource references value sets which are
plausible candidates for migration into the HL7 Terminology (THO) repository.
Specifically, it calls out the `variable-handling-example` ValueSet (backed by
the `EvidenceVariableHandlingExample` CodeSystem) bound to
`Evidence.statistic.modelCharacteristic.variable.handling`. While such migration
is not strictly required, the ticket asks the work group to consider it because
the value-set definition is expected to evolve and may have utility beyond FHIR
(e.g., CDA, V2). The triggering context is [FHIR-51456](https://jira.hl7.org/browse/FHIR-51456),
which changed `EvidenceVariable.handling` from a `code` with a required binding
to a `CodeableConcept` with an extensible binding — a change that strengthens
the case for hosting the underlying value set in THO so it can grow over time.
A comment notes that the necessary terms have already been approved for
inclusion in SEVCO and will be available with SEVCO Release 3.0, at which point
a corresponding ValueSet should be created and added to THO. Bryn Rhodes
suggests variable-handling may already be in THO and intends to verify.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the `variable-handling-example` ValueSet (and, as appropriate, its
backing `EvidenceVariableHandlingExample` CodeSystem) from the FHIR Core build
into HL7 Terminology (THO). Concretely:

1. Coordinate with the SEVCO maintainers and Vocab WG to confirm the timing of
   SEVCO Release 3.0 (per JIRAUSER18251's comment) which will include the
   approved terms.
2. Once SEVCO 3.0 is published, define a THO-hosted ValueSet that composes from
   the SEVCO code system for the variable-handling concepts (boolean,
   continuous, dichotomous, ordinal, polychotomous, time-to-event,
   not-specified, plus any additional SEVCO terms agreed for this binding).
3. Update the binding on `Evidence.statistic.modelCharacteristic.variable.handling`
   (and the parallel `EvidenceVariable.handling` binding established in
   FHIR-51456) to point at the new THO ValueSet URL, retiring the FHIR-Core-only
   `variable-handling-example` ValueSet/CodeSystem definitions in favor of THO
   stewardship.
4. Confirm Bryn's follow-up: if a variable-handling value set already exists in
   THO, reuse / extend it rather than creating a parallel artifact.

#### Justification

The ticket itself is a "consider migration" call rather than a forced change,
but the conditions THO migration is meant to address are clearly met here:
(a) the value set is bound `extensible`, not `required`, so it is explicitly
expected to grow; (b) FHIR-51456's resolution already flagged the eventual
landing place as an open question; (c) a comment confirms the SEVCO approval
path is in flight, which is the natural home for these scientific-evidence
terms; and (d) variable-handling concepts (boolean / continuous / dichotomous /
time-to-event …) are reusable outside of FHIR Evidence — they apply to
statistical modeling generally and could be referenced by V2, CDA, or other IGs.
Hosting the value set in THO avoids each downstream spec re-defining its own
parallel binding and gives Vocab WG a single point of stewardship.

---

### Disposition B: Alternative Approach

#### Proposal

Defer THO migration of the value set itself, but make the binding "THO-ready"
in this cycle by:

1. Keeping the value set in the FHIR Core build for now, but renaming /
   re-canonicalizing it so it does not carry the `-example` suffix that signals
   it is a placeholder (e.g., `variable-handling`), and documenting in the
   ValueSet narrative that it is a candidate for migration to THO once SEVCO
   Release 3.0 lands.
2. Adding a tracking note in the Evidence / EvidenceVariable spec pages that
   the canonical home of this value set is expected to move to THO, so that
   implementers can plan for the URL change.
3. Filing a paired ticket (or re-purposing this one) to perform the actual
   migration after SEVCO 3.0 is published, rather than blocking on it now.

#### Justification

The hard dependency in Disposition A is SEVCO Release 3.0, which is outside
the FHIR Core ballot timeline. Attempting the migration before SEVCO 3.0 ships
risks either (a) defining a THO ValueSet that has to be respelled once the
SEVCO codes land, or (b) blocking a Highest-priority ticket on an external
publication. Disposition B preserves the migration intent, signals it
explicitly to implementers, and avoids creating a near-term canonical URL that
will have to be deprecated. It is also consistent with the CDS WG's pattern of
handling THO migrations as separately-tracked terminology work after the
underlying code system is stable.

---

### Disposition C: Decline

#### Proposal

Close this ticket as "Not Persuasive" / out of scope on the grounds that
(a) FHIR-51456 already changed the binding strength to `extensible`, which is
the mechanism FHIR provides for value sets that are expected to evolve;
(b) the existing `variable-handling-example` ValueSet is functioning as a
suggested binding and is not blocking implementers; and (c) any future
migration to THO is a Vocab WG / terminology lifecycle decision that does not
require a Core change request.

#### Justification

The ticket itself acknowledges the migration is "not required". Until SEVCO
Release 3.0 publishes the approved terms, there is no concrete artifact to
move, and the existing extensible binding already permits implementers to use
codes outside the example set. Declining avoids speculative restructuring and
keeps the canonical URL stable for current implementers of the R6 ballot. The
work can be revisited by Vocab WG out-of-cycle once SEVCO 3.0 is published.

---

### Recommendation

**Recommended disposition:** B

The intent of the ticket — get this value set into THO so it can evolve and be
reused — is correct and well-supported by the FHIR-51456 history, the
extensible binding, and the cross-domain reusability of variable-handling
concepts (Disposition C undersells this). However, Disposition A's hard
dependency on SEVCO Release 3.0 is outside CDS WG's control and is not a safe
thing to commit to within a single ballot cycle: defining a THO ValueSet now
that has to be respelled when SEVCO 3.0 lands creates churn for implementers
and a deprecated canonical URL. Disposition B captures the migration intent in
the spec, removes the misleading `-example` framing of an extensible binding
that we actually expect to be normative-ish, and queues the real migration as
a follow-up tied to the SEVCO release. It also leaves room for Bryn's
follow-up: if variable-handling already exists in THO, the eventual migration
is just a re-binding rather than new artifact authoring. The work group should
endorse Disposition B and explicitly assign the follow-up THO migration ticket
to a named owner (likely jointly with Vocab) so it does not slip after SEVCO
3.0 ships.
