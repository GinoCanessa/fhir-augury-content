## Summary

The reporter (Christopher Schuler) observes that the flow diagrams used in
the CDS Hooks Library hook pages render too wide for the page and visually
overflow their container, making elements clip or require horizontal
scrolling. He requests that the diagrams be reformatted from a horizontal
(landscape) orientation to a vertical (portrait) orientation so that all
nodes/labels fit neatly within the page width. This is a documentation /
presentation-only change against the CDS Hooks Library spec; it does not
alter any hook semantics, context fields, or response structures.

No related Jira tickets, GitHub issues/PRs, or substantive Zulip threads
were surfaced via the FhirAugury cross-reference and search indexes
(see sections below), so this ticket appears to stand alone as an
editorial/layout fix.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-author the CDS Hooks Library flow diagrams in vertical (portrait)
orientation. Concretely:

1. Open `diagrams/cds-hooks-workflow.pptx` in `cds-hooks/docs` and
   re-lay out the offending hook flow slide(s) with a top-to-bottom
   (vertical) node arrangement instead of left-to-right.
2. Re-export the affected SVGs (and matching PNG fallbacks) into
   `docs/images/`, replacing `orderingflow.svg`, `orderingflow.png`,
   and `patient-view-hook-launch_spec.png` (and any other landscape
   diagram surfaced during the audit).
3. Verify each `docs/hooks/*.md` page that embeds these images renders
   without horizontal overflow at the standard mkdocs content width
   (and on a typical mobile viewport).
4. Commit on a topic branch in `cds-hooks/docs` and open a PR
   referencing FHIR-50076.

#### Justification

This is exactly what the reporter asked for and is the lowest-risk
change: it is a documentation/asset edit with no impact on the CDS
Hooks specification or any conformance surface. Vertical orientation
is the standard mitigation for diagrams that exceed a documentation
site's content column, and re-exporting from the canonical PPTX keeps
the SVG/PNG pair in sync with the editable source-of-truth. The CDS
Hooks Library is at hookMaturity 5 ("Mature") for several hooks — a
purely editorial fix is appropriate and uncontroversial at that
maturity.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the overflow with a CSS / layout change instead of re-authoring the
diagrams. Concretely:

1. In `cds-hooks/docs`, add (or extend) an `extra.css` referenced from
   `mkdocs.yml` with a rule such as:

   ```css
   .md-content img { max-width: 100%; height: auto; }
   ```

   so any oversized SVG/PNG scales to the content column.
2. Where a diagram becomes illegible at the scaled width (likely
   `orderingflow.svg`), wrap the `<img>` in an anchor that links to the
   full-resolution SVG so readers can open it in a new tab.
3. Optionally add a "click to enlarge" affordance using a lightweight
   mkdocs-material lightbox plugin.
4. Leave the source PPTX and exported diagrams in their current
   landscape orientation.

#### Justification

The underlying need is "diagrams fit on the page", not "diagrams must be
vertical". A CSS-based fix is faster, requires no graphic-design work,
and preserves the authors' chosen orientation (which often reads better
for sequence-style flows where actors are columns). The trade-off is
legibility: at narrow viewports a scaled-down landscape diagram can
become hard to read, which is why an enlarge / click-through path is
included. This approach also generalizes — any future diagram that
overflows is auto-handled.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for the CDS Hooks Library specification proper.
The CDS Hooks Library deliverable is the set of hook *definitions*
(context, prefetch, change log) — the ancillary workflow diagrams are
illustrative documentation, not normative. Close FHIR-50076 as
"Persuasive with mod / Not Persuasive" with a note directing the
reporter to file the issue in the `cds-hooks/docs` repository's
GitHub issue tracker as a documentation-rendering bug.

#### Justification

This treats the request as a site / theming concern rather than a
spec-change-request. It is defensible because no normative content is
affected and the CDS Hooks Library uses the FHIR Jira tracker primarily
for hook semantics. The downside is that it adds friction for a small,
clearly-actionable editorial fix that the WG can simply do — and
shifting it to GitHub may delay resolution further given the very
lightweight nature of the change.

---

### Recommendation

**Recommended disposition:** A

This is a tiny, low-risk editorial change against an asset already in
the repository, with a canonical editable source (`cds-hooks-workflow.pptx`)
that makes vertical re-layout straightforward. It directly delivers what
the reporter asked for and removes the symptom (overflow) at its source
rather than masking it with CSS. Disposition B is a reasonable
companion (a `max-width: 100%` rule is generally good hygiene for a
docs site and can be added in the same PR), but on its own it would
likely produce an unreadable downscaled diagram for `orderingflow.svg`
and would not address the reporter's specific concern. Disposition C is
not warranted: the WG owns the CDS Hooks Library docs, the change is
trivial, and routing the reporter elsewhere would be poor stewardship.
The WG should accept FHIR-50076, take the asset edit in `cds-hooks/docs`,
and optionally fold in the CSS hardening from B as a defensive measure.
