## Summary

The submitter argues that the CQL `@constraint` annotation is opaque: the
guide defines a constraint as firing only when its expression evaluates to
`false` (and explicitly *not* when it evaluates to `null`), and the behavior
on non-Boolean datatypes is undefined. They propose making the trigger
condition explicit by adding a companion tag — either `@when: <operator>
<value>` (positive form, e.g. `@when: >= 200 mg/dL`) or `@expect: <operator>
<value>` (inverse form, e.g. `@expect: >= 10 mg/dL`) — paired with the
existing `@constraint:` severity and `@message:` text. The intent is to
let constraint authors state the trigger condition declaratively rather
than encode it inside an arbitrary expression whose null/false semantics
are surprising.

## Details

**Description:**

> The conditions of the constraint are naturally unclear. Further text
> in the guide explains that it gives the error only if the definition
> evaluates to False but does not generate error on null. This behavior
> seems arbitrary and unclear and the behavior of constraints on other
> datatypes seems undefined. It seems this would be better generalized
> with a second tag `@when: <operator> <value>` to define the
> constraint trigger more clearly. For example
> `@constraint:warning @when: is True @message:The function evaluates True`.
> Or, `@constraint:error @when: >= 200 mg/dL @message: max value of 200 exceeded`.
> Alternatively a tag `@expect: <operator> <value>` could be used to
> define an expected value has not been met therefore the warning/error
> displayed (the inverse logic). For example
> `@constraint: error @expect: >= 10 mg/dL @message: value is null or minimum value not reached`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the CQL `@constraint` annotation grammar in the Clinical Quality
Language specification to recognize an additional companion tag pair
(choose one of `@when:` or `@expect:`, not both, to avoid two ways to
say the same thing). The chosen tag accepts a comparison expression of
the form `<operator> <value>` (e.g., `is True`, `>= 200 mg/dL`). When
present, the tag *replaces* the rule that the constraint fires when the
annotated expression evaluates to `false`: instead, the constraint
fires when the operator/value comparison against the annotated
expression's result evaluates to `true` (for `@when:`) or `false` (for
`@expect:`). When absent, today's behavior is preserved verbatim
(constraint fires on `false`, never on `null`). This requires updates
to: (i) the CQL specification's "Annotations / Constraints" section to
define the new tag, its grammar, and its precedence over the default
trigger; (ii) worked examples mirroring the submitter's `200 mg/dL`
and `10 mg/dL` cases including explicit null-handling for each form;
and (iii) a normative statement of behavior on non-Boolean datatypes
when the new tag is present.

#### Justification

The submitter's core observation is real: the current rule ("fires on
`false`, silent on `null`, undefined on non-Boolean") is asymmetric,
and the asymmetry is precisely what makes constraint authoring
error-prone in clinical contexts where missing data is the normal
case. A declarative trigger lets the author state intent in one place
and removes the need to defensively wrap expressions with
`Coalesce`/`IsNull` guards just to get the desired severity behavior.
Adopting `@expect:` (inverse form) is preferable to `@when:` because it
matches how clinicians describe rules ("we *expect* glucose ≥ 10
mg/dL") and naturally absorbs the null case into a single failure
state, which is the submitter's stated motivation.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the `@constraint` grammar. Instead, add a non-normative
"Constraint Authoring Patterns" section to the CQL spec that (a) makes
the existing semantics explicit ("fires iff the expression evaluates
to `false`; `null` is *not* a failure"), (b) gives the canonical
recipe for the submitter's two examples using existing CQL — wrapping
the predicate so that null is converted to the desired truth value
(e.g., `Coalesce(Glucose < 200 'mg/dL', false)` for "fail on missing"
or `... true` for "pass on missing"), and (c) explicitly notes that
constraints whose annotated expression returns a non-Boolean type are
ill-formed and should be flagged by tooling. Optionally add a
linter/validator rule (in the CQL translator) that warns when a
`@constraint`-annotated expression's static type is not `Boolean`.

#### Justification

This is the path implied by the 9/17 WGM comment: every example the
submitter cites is constructible today, the cost is borne by the spec
text and tooling rather than by every CQL implementation, and it
preserves the single-trigger model that makes static analysis of
constraints tractable. Tooling implementers carry real cost from a new
trigger grammar — the existing model lets a translator emit a single
Boolean-evaluation site per constraint; introducing operator/value
pairs forces the translator to materialize a comparison node and
re-derive type rules for each datatype the value could take (Quantity,
String, Code, Interval, …). The alternative also removes the
ambiguity of having two redundant tags (`@when` *and* `@expect`) that
would otherwise tempt different communities to standardize on
different conventions.

---

### Disposition C: Decline

#### Proposal

Mark the ticket Persuasive Not Persuasive — the request to add
`@when:` / `@expect:` companion tags is declined as already addressed
by the existing language. Record in the resolution that (1) the spec
already prescribes `false`-only firing with `null` ignored, (2) every
example the submitter provided can be expressed today with a wrapping
expression, and (3) constraint annotations whose expression is
non-Boolean are out of scope of `@constraint` and should be authored
as ordinary CQL expressions or validation rules. Close without spec
change.

#### Justification

The 9/17 WGM disposition is on this trajectory: the workgroup found
the documentation clear and the proposal "unnecessarily complicated,
especially on tooling," with the explicit observation that all
examples are constructible today. Declining is defensible because the
ask is ergonomic ("make the trigger more declarative") rather than
capability-extending — there is no clinical scenario the submitter
cites that current CQL cannot express. A breaking-or-near-breaking
grammar extension to a normative annotation, taken on for ergonomics
alone, is hard to justify against the cost to every conformant
translator and authoring tool.

---

### Recommendation

**Recommended disposition:** B

The 9/17 WGM comment is right that the existing language is
*sufficient*, but the submitter's frustration is also legitimate: the
"`false` triggers, `null` doesn't, non-Boolean is undefined" rule is
genuinely surprising and is not stated symmetrically in the current
spec text. Disposition C closes the ticket without addressing that
underlying documentation gap, and the submitter is likely to re-file.
Disposition A solves the ergonomics complaint but at a real and
recurring cost to every CQL translator and authoring tool — a cost
the workgroup has already flagged. Disposition B captures the value
of A (clearer authoring guidance, an explicit null story, and a
machine-checkable rule for non-Boolean expressions) without changing
the constraint grammar, and aligns with the next step the WGM
recorded ("talk with LawrenceMcKnight to understand the comment
further"). It also leaves the door open: if implementer experience
later shows the wrapping idiom is too verbose at scale, the workgroup
can revisit a grammar extension on stronger evidence than a single
ticket.
