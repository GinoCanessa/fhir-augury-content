## Summary

This ticket asks the CQL (FHIR) specification to add a `Comparable`
function that takes two `Quantity` values and returns `true` when the
engine can determine an order/equivalence relationship between them
(i.e., they share a comparable unit system). The request is a
straight-through alignment with the parent FHIRPath tracker
[FHIR-53957](https://jira.hl7.org/browse/FHIR-53957) ("FHIR is proposing
a function comparable, move that to fhirpath"), which has already been
**Applied / Persuasive** under the Implementable Technology
Specifications WG and merged into FHIRPath via
[HL7/FHIRPath#24](https://github.com/HL7/FHIRPath/pull/24). Because CQL
is layered on top of FHIRPath but maintains its own catalog of
operators and functions, an explicit decision is needed for whether and
how CQL exposes the new FHIRPath `comparable()` function in the CQL
language surface and translator. The ticket itself contains no further
detail beyond the pointer to FHIR-53957.

Brian Postlethwaite opened the thread (2026-02-01) asking how
`comparable()` should behave when one of the arguments is an `Integer`
literal rather than a `Quantity` (e.g., `1 '1'.comparable(2)` vs.
`1 '1'.comparable(2.toQuantity())`). The conclusion, reached with Steve
Munini, was that FHIRPath's existing implicit conversion from `Integer`
to `Quantity` (and the singleton-evaluation-of-collections rule) covers
these cases, so no extra signature is needed — but several engines
(including HAPI and Steve's implementation) had not yet been performing
the implicit conversion. Steve subsequently fixed his implementation in
multiple places and suggested the spec could state more explicitly that
implicit conversion is expected; Brian pointed at the existing
"Conversion" and "Singleton evaluation of collections" sections, which

it confirms the expected runtime semantics that a CQL adoption would
need to inherit.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a `Comparable` function to the CQL (FHIR) specification, declared
as:

```
define function Comparable(left Quantity, right Quantity) returns Boolean
```

with semantics that mirror FHIRPath `comparable()` exactly as merged in
[HL7/FHIRPath#24](https://github.com/HL7/FHIRPath/pull/24): returns
`true` iff both operands are non-null, both have a value, and either
(a) both have the same unit code regardless of system, or (b) both have
a `code` + `system` whose system is recognized by the engine and whose
codes are comparable within that system (so e.g. `mg` and `g`,
`d` and `h`, `[in_i]` and `cm` all return `true`). Inputs of CQL types
that implicitly convert to `Quantity` (e.g., `Integer`, `Decimal`)
follow CQL's existing implicit-conversion rules so that
`Comparable(1, 1 '1')` works without explicit `ToQuantity`. The
specification update lands in the CQL operator reference (the
`Comparison Operators` / `Quantity Operators` chapter) with examples
analogous to those in the FHIRPath thread, and the function is added
to the System library so the CQL-to-ELM translator can lower it to the
new FHIRPath operator when the target runtime is FHIRPath-based.

#### Justification

The parent ticket FHIR-53957 is already **Persuasive / Applied** in
FHIRPath, which CQL builds on; declining to mirror it would leave CQL
authors writing FHIR queries without a portable way to test
unit-comparability before invoking comparison operators that will
otherwise throw or return `null`. Adding a thin wrapper keeps the CQL
surface aligned with FHIRPath (a long-standing CQL design goal) and
matches the precedent set by other Quantity-related functions
(`ConvertQuantity`, `CanConvertQuantity`). The Zulip thread confirmed
the implicit `Integer → Quantity` conversion behavior is already
specified in FHIRPath, so CQL can reuse the same rules without inventing
new conversion semantics.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of defining a new top-level CQL function, expose the FHIRPath
`comparable()` operation **only via the translator**, so that any CQL
expression that targets a FHIRPath runtime can call it with
`Quantity.comparable(other)` style invocation through CQL's
FHIRPath-passthrough mechanism, and document a recommended idiom using
the existing `CanConvertQuantity(q, unit)` plus a unit-code check for
authors who want pure-CQL portability. No new entry is added to the
CQL System library; the operator reference instead gets a short
informative section describing that, for FHIR-based runtimes, the
`comparable()` FHIRPath function is available and should be used when
unit-comparability needs to be tested without raising errors.

#### Justification

CQL has historically been conservative about adding operators that
duplicate underlying FHIRPath/FHIR functionality, and the
unit-comparability check is arguably already expressible with
`CanConvertQuantity`. This avoids growing the CQL System library for a
function whose semantics are entirely engine-defined ("system is
recognized by the FHIRPath implementation" is non-deterministic across
engines), which the Zulip discussion flagged as a real interop gap
between implementations (HAPI and Steve Munini's engine had divergent
behavior until recently). Keeping the function as a FHIRPath-only
construct contains that variability rather than exporting it into the
CQL surface, where measure authors expect deterministic behavior across
engines.

---

### Disposition C: Decline

#### Proposal

Decline to add a `Comparable` function to CQL. Close as "Not Persuasive"
(or "Persuasive with Modification" pointing at existing
`CanConvertQuantity`) on the grounds that CQL already provides
`ConvertQuantity` and `CanConvertQuantity`, which give authors a
deterministic way to check whether two quantities are unit-compatible
under UCUM (the only quantity system CQL formally requires). The new
FHIRPath `comparable()` function is engine-defined for non-UCUM systems
and would introduce non-portable behavior into CQL libraries.

#### Justification

CQL libraries are intended to be portable across CQL engines; a
function whose return value depends on which code systems the engine
"recognizes" violates that portability expectation. The Zulip thread
on the FHIRPath side documented that real engines disagreed on the
behavior even for the basic `Integer → Quantity` case. Until the
underlying FHIRPath operator stabilizes across engines and the CQL
community articulates concrete measure-authoring use cases that
`CanConvertQuantity` cannot satisfy, adding it to CQL is premature.

---

### Recommendation

**Recommended disposition:** A

The parent FHIRPath ticket has already been resolved Persuasive and
applied via PR #24, and Bryn Rhodes (the CDS WG co-chair and CQL
spec lead, who is also the reporter here) filed FHIR-55384 specifically
to keep CQL aligned with that decision. Disposition A is a low-risk,
mechanical alignment that preserves CQL's long-standing rule of
matching FHIRPath's operator surface where feasible, and the Zulip
discussion has already settled the most common semantic ambiguity
(implicit `Integer → Quantity` conversion). Disposition B's portability
concern is real but is best addressed by a normative note in the CQL
operator reference (which Disposition A can incorporate) rather than by
withholding the function entirely; Disposition C would create a gap
where CQL-on-FHIRPath authors lose access to a function their runtime
already supports. The implementation work is small — a System library
entry plus a translator mapping to the FHIRPath operator — and falls
under the same authoring conventions used for the existing
`CanConvertQuantity` family.
