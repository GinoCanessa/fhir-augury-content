## Summary

The CQL specification's String overload of the `in(ValueSet)` membership
operator currently states that a run-time error must be thrown whenever
the resolved value set contains codes from more than one code system,
because the bare string code being tested cannot disambiguate which
system applies. Bryn Rhodes proposes relaxing this rule so the operator
only errors when the *same* string code actually appears in *more than
one* code system inside the value set — i.e., when there is a genuine
ambiguity for the value being tested. The motivation is consistency with
the FHIR terminology service `$validate-code` operation when invoked
with `inferSystem=true`, whose semantics (per Grahame Grieve and Michael
Lawley on Zulip) only treat the duplicate-code case as an error, not
the multi-system case.

Concretely:

- `'A' in V` where `V = { "A|http://codesystem.x", "B|http://codesystem.y" }`
  would become **legal** (currently an error).
- `'C' in V` where `V = { "C|http://codesystem.x", "C|http://codesystem.y" }`
  would **remain an error** (the string "C" matches in both systems).

In addition, the ticket asks that the CQL translator emit a warning
whenever the String overload of `in(ValueSet)` is used at all, because
it is inherently ambiguous compared to the `Code` and `Concept`
overloads.

There are no cross-referenced Jira tickets and no related GitHub items;
the only direct cross-reference is the Zulip thread that motivated the
ticket.

## Details

**Description:**

The String overload of the `in(ValueSet)` operator includes the
following language:

> Note that for this overload, because the code being tested cannot
> specify code system information, if the resolved value set contains
> codes from multiple code systems, a run-time error is thrown because
> the operation is ambiguous.

However, this was discussed in the context of the `inferSystem`
parameter of the `$validate-code` operation in a FHIR terminology
service, and for consistency with that specification, we propose that
the operation behavior should be changed to only throw an error if the
string value being tested actually matches the same code in different
systems in the value set, rather than just if the value set has multiple
code systems.

Explicitly, in the current behavior, this is an error:

```
ValueSet V { "A|http://codesystem.x", "B|http://codesystem.y" }
'A' in V
```

But we propose that that shouldn't be an error, but that this still
would be:

```
ValueSet V { "C|http://codesystem.x", "C|http://codesystem.y" }
'C' in V
```

It should also be the case that the translator issues a warning when
the string overload of `in(ValueSet)` is used at all, since it is
inherently ambiguous.

For further details, refer to the Zulip thread:
<https://chat.fhir.org/#narrow/channel/179202-terminology/topic/ValueSet.20.24validate-code.20with.20inferSystem>

**Comments:**

## Related Zulip Discussions

### [terminology > ValueSet $validate-code with inferSystem](https://chat.fhir.org/#narrow/stream/terminology/topic/ValueSet%20%24validate-code%20with%20inferSystem)

The thread starts with Alexander Kiel asking what
`ValueSet/$validate-code` with `inferSystem=true` should return when the
value set contains two different code systems with the same code, and
notes that CQL's `in(ValueSet)` mandates an error in any multi-system
case. Grahame Grieve responds that `$validate-code` only treats the
duplicate-code case as an error — having multiple code systems in a
value set is fine; the error condition is when the bare string code
*itself* appears in more than one system. Michael Lawley confirms the
`$validate-code` semantics distinguish the "M" (same code, two systems)
case from the "A/B" (different codes, two systems) case. Robert McClure
agrees with the underlying principle (a same-spelling string in two
systems can mean two different concepts, so it must not silently
validate). Bryn Rhodes traces the original CQL position back to DSTU
comment #1556, then concedes the looser rule is the right one and
proposes the change captured in this ticket. Gino Canessa raises a
secondary concern that even with the relaxed rule, the String/Code
overloads remain fragile because a non-breaking value-set update (adding
a system that introduces a duplicate string code) can silently turn
previously-valid CQL into a runtime error; he suggests a "dragon" /
implementer note. Grahame agrees the `Coding`-based overload is the
only fully safe option but acknowledges that is not how authors work in
practice. Alexander Kiel closes by noting the duplicate-code-on-add
problem actually argues for keeping the strict CQL behavior unless the
warning/dragon is added, and asks whether `code`-typed elements always
bind to single-system value sets — Grahame replies that they can and do
come from multiple systems. Consensus on the core proposal (relax the
error to the duplicate-code case) is broad among the terminology
participants; the open question is whether to also add a translator
warning / implementer note about the residual fragility of the String
overload.

Participants: Alexander Kiel, Grahame Grieve, Bryn Rhodes, Robert McClure, Michael Lawley, Gino Canessa.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Amend the CQL Reference, §9.B "Terminology Operators", entry for the
String overload of `in(ValueSet)`
(<https://cql.hl7.org/09-b-cqlreference.html#in-valueset>), as follows:

1. Replace the current normative note —

   > "if the resolved value set contains codes from multiple code
   > systems, a run-time error is thrown because the operation is
   > ambiguous"

   — with language stating that a run-time error is thrown only when
   the string value being tested matches **the same code in more than
   one code system** in the resolved value set. Multiple code systems
   in the value set are no longer, by themselves, an error condition.

2. Add a normative requirement to the CQL translator to emit a
   **warning** whenever the String overload of `in(ValueSet)` is used,
   noting that the operation is inherently ambiguous and that the
   `Code` or `Concept` overloads are preferred when the system is
   known.

3. Add a non-normative implementer note (a "dragon" / caveat) below the
   operator definition explaining that, even with the relaxed rule, a
   later addition of a duplicate string code from a second code system
   to the bound value set can turn previously-valid CQL into a runtime
   error — i.e., the String overload is fragile against value-set
   evolution.

This realigns CQL's behavior with FHIR's `$validate-code` /
`inferSystem=true` semantics (only the duplicate-code case is an error),
exactly as worked through on the Zulip thread.

#### Justification

This is the change Bryn Rhodes filed and is the position that
terminology subject-matter experts (Grahame Grieve, Michael Lawley,
Robert McClure) all converged on in the Zulip discussion. It removes a
spurious source of run-time errors in CQL libraries that bind to
real-world value sets containing multiple code systems with non-
overlapping codes, while preserving the safety property that
genuinely-ambiguous lookups still fail loudly. It also brings CQL into
alignment with the FHIR terminology service contract — important
because most CQL implementations resolve value-set membership through a
terminology service, and divergent semantics force adapter logic.
Item 3 directly addresses Gino Canessa's concern that the relaxation
introduces a value-set-evolution hazard the author has no good way to
detect; Item 2 (the translator warning) gives implementers and authors
a uniform signal that the overload is best avoided when the system is
in fact knowable.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt only items (1) and (2) of Disposition A — relax the runtime error
to the duplicate-code case **and** make the translator warning on the
String overload **mandatory and elevated** (e.g., a strongly-worded
warning that calls out the value-set-evolution hazard inline) — but do
**not** add a separate implementer "dragon" note. Optionally, also add
a normative recommendation in the CQL Reference that authors prefer
the `Code` or `Concept` overload of `in(ValueSet)` whenever the system
is known at authoring time, with the String overload reserved for cases
where the system is genuinely intended to be inferred from the value
set.

#### Justification

This keeps the spec change minimal and self-consistent: a single
normative behavior change plus a single translator-level signal,
without introducing prose that some implementations may render
inconsistently. It still captures the substantive Zulip consensus
(the multi-system-only case is no longer an error) and still gives
authors a visible signal at translation time. The trade-off vs.
Disposition A is that the residual fragility (a value-set update
introducing a duplicate string code) is communicated only through the
translator warning, not through the published spec text — implementers
who only read the spec may miss the hazard until it bites them. This
disposition is preferable if the workgroup judges the "dragon" note
to be redundant once the warning exists, or if there is concern about
adding non-normative caveats to the operator definition.

---

### Disposition C: Decline

#### Proposal

Reject the change and retain the current CQL Reference language: the
String overload of `in(ValueSet)` continues to throw a run-time error
whenever the resolved value set contains codes from more than one code
system, regardless of whether the specific string being tested is
duplicated across systems. Document that authors who need to test
membership in a multi-system value set must use the `Code` or
`Concept` overload of `in(ValueSet)`.

#### Justification

This is the position originally taken in CQL (per DSTU comment #1556)
and re-articulated by Alexander Kiel late in the Zulip thread: because
a "non-breaking" addition of a new code system to a bound value set can
silently introduce a duplicate string code and turn previously-valid
CQL into a runtime error, the only authorially-safe choice is to make
the String overload error **whenever** the value set is multi-system,
forcing the author to switch to the `Code` or `Concept` overload up
front. Declining preserves a stricter "fail closed" stance that
protects authors from value-set evolution hazards they cannot see.
The cost is divergence from FHIR `$validate-code` / `inferSystem=true`
semantics, which several terminology participants flagged as the
primary problem; that divergence is what Disposition A and B are
designed to fix.

---

### Recommendation

**Recommended disposition:** A

The Zulip discussion shows clear and explicit alignment among the
terminology experts (Grahame Grieve, Michael Lawley, Robert McClure)
and the CQL author (Bryn Rhodes) on the substantive change: the
multi-system-only case should not be an error; only the
duplicate-code case should be. Keeping CQL's stricter rule
(Disposition C) creates an avoidable inconsistency with the FHIR
terminology service contract that CQL implementations must talk to,
and forces adapter logic that has already been called out as a
problem. Between A and B, the additional implementer-note ("dragon")
in Disposition A directly responds to the residual fragility that
Gino Canessa raised and that Alexander Kiel used to argue for the
strict rule — addressing that concern in spec prose, not just in a
translator warning, is the most defensible way to relax the rule
without leaving authors exposed to a hazard they did not previously
have to think about. The translator warning is cheap to implement,
gives a uniform authoring-time signal, and matches the intent
expressed in the ticket itself.
