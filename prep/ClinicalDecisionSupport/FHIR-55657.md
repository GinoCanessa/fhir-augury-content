## Summary

Bryn Rhodes is asking the CQL specification to state explicitly how the
`is` and `as` type-testing operators behave when the operand is `null`.
Because `null` is the absence of a value (not itself a value of any type)
and `is` is evaluated at run time, the expression `null is Integer` must
evaluate to `false`. Likewise, since `as` is purely a compile-time type
assertion, the composite expression `(null as Integer) is Integer` is
semantically equivalent to `null is Integer` and must therefore also
return `false`. The ticket asks that this clarification be added to the
CQL reference for `is`, accompanied by these worked examples. No related
Jira tickets, Zulip threads, or GitHub items were surfaced by Augury for
this ticket.

## Details

**Description:**

> Clarify that, since `null` is not a value, and `is` is a strictly

> addition, `as` is strictly a compile-time consideration, so
> `(null as Integer) is Integer` is equivalent to `null is Integer`, and

>
> Add these as examples to the `is` reference documentation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CQL Reference page describing the `is` operator to add an
explicit normative sentence stating: "If the operand of `is` is `null`,
the result is `false`, regardless of the target type, because `null`
denotes the absence of a value rather than a value of any type."
Immediately follow it with two worked examples in the same `is`
reference section:

- `null is Integer` → `false`
- `(null as Integer) is Integer` → `false` — explained by the fact that
  `as` is a compile-time type assertion that does not introduce a value,
  so the operand of the outer `is` is still `null` at run time.

Add a forward reference from the `as` operator section to the new
examples so a reader landing on `as` finds the same clarification.

#### Justification

The change is purely editorial: it documents behavior that is already
implied by CQL's three-valued logic and by the existing definitions of
`is` (run-time test) and `as` (compile-time assertion). Making it
explicit removes a known ambiguity for implementers, aligns the spec
with the way conformant engines already behave, and is consistent with
the broader CQL convention that operators on `null` propagate or return
a defined null-aware result rather than an error. The reporter (the CQL
specification lead) is also the requester, which strongly indicates
this is the intended interpretation.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the clarification, but rather than localizing the fix to the
`is` reference, introduce (or expand) a single "Type Testing and
Casting with Null" subsection in the CQL operator chapter that covers
`is`, `as`, and `convertsTo`-family operators together. State the
general rule once ("type-testing and casting operators applied to `null`
yield `false` for `is`/`convertsTo*` and `null` for `as`/`convert*`,
because `null` is not a value of any type"), then list the specific
worked examples from this ticket plus the analogous `null is String`,
`null as String`, and `convertsToInteger(null)` cases. Cross-link from
each individual operator's reference page to the consolidated
subsection.

#### Justification

The same ambiguity exists for every type-testing and casting operator,
not just `is`. Fixing it only on the `is` page risks the next ticket
asking the same question about `convertsToInteger`, `as`, or
`convertsToString`. A single consolidated treatment is easier to keep
internally consistent, reduces duplication across operator pages, and
gives implementers one canonical place to cite. The trade-off versus
Disposition A is a slightly larger editorial scope and the need to
re-verify each operator's null behavior against the existing prose, but
the marginal cost is small relative to the durability of the fix.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — Already Specified", on the
grounds that CQL's three-valued logic and the existing definitions of
`is` (run-time) and `as` (compile-time) already determine that
`null is Integer` is `false` and `(null as Integer) is Integer` is
`false`. No spec change is required; conformant implementations should
derive the behavior from the existing rules.

#### Justification

The behavior is arguably derivable from first principles in the current
text, so a strict reading of the spec already gives the requested
answer. Declining keeps the reference text smaller and avoids the
maintenance cost of additional examples. The downside — and it is a
substantial one — is that the reporter is the CQL specification lead
explicitly flagging the prose as ambiguous, which is strong evidence
that derivation-from-first-principles is not in fact obvious to
implementers.

---

### Recommendation

**Recommended disposition:** B

Disposition A is correct and low-risk, but Disposition B answers the
same question once for the entire family of type-testing and casting
operators, which is where this ambiguity recurs in practice. Since the
reporter is the CQL specification lead and the request is editorial
rather than behavioral, the work group should take the small extra
effort to consolidate the null-handling rule for `is`, `as`, and the
`convertsTo*`/`convert*` operators in one place and back-link from each
operator's reference page. Disposition C should be rejected: the very
fact that the spec lead is asking for clarification is evidence that
the current prose is not clear enough to be left alone.
