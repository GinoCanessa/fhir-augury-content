## Summary

The ticket asks the Clinical Quality Language (CQL, FHIR edition)
specification to add explicit language for what should happen when math
operations are performed on UCUM "special" units — units that require a
non-linear conversion function rather than a simple scalar (e.g., `Cel`
for Celsius, `[degF]` for Fahrenheit, `B` for bel). These units cannot
participate in algebraic operations in the usual way, so the result is
not defined.

The reporter (Bryn Rhodes) explicitly points to the parallel FHIRPath
tracker [FHIR-41343](https://jira.hl7.org/browse/FHIR-41343) ("Return
empty or error when doing math with 'special' UCUM units"), which has
already been resolved Persuasive and **Applied** by ITS for FHIRPath
§6.6 (Math). The applied FHIRPath resolution clarifies that operations
on quantities with differing "special" UCUM units on non-ratio scales
are not supported and result in empty (`{ }`). FHIR-52039 is the CDS
mirror of that change: bring the equivalent language into the CQL
specification so CQL's quantity arithmetic semantics align with
FHIRPath.

Paul Lynch opened the thread (2023-05-25) explaining that UCUM forbids
"special" units like `B`, `[degF]`, and `Cel` from algebraic
operations, and that FHIRPath was silent on what should happen, asking
whether implementations raised errors. Gino Canessa illustrated the
problem with a temperature example (`0 [degC] + 0 [degC]`). Paul Lynch

error was preferable for this case. Grahame Grieve initially leaned

the expression itself is not malformed — the inputs simply cannot be
combined. Consensus on the thread therefore landed on **empty (`{ }`)**,
which is the resolution that was ultimately applied to FHIRPath §6.6
and which FHIR-52039 should mirror in CQL.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CQL (FHIR) specification's quantity arithmetic section
(the CQL counterpart to FHIRPath §6.6 — typically the "Quantity
Operators" / "Arithmetic Operators" section of the CQL Author's /
Reference guide that defines `+`, `-`, `*`, `/`, `mod`, `truncate`,
unary `-`, etc., on quantities) to add language closely paralleling
the applied FHIRPath text:

> *When operating on quantities, the dimensions of each quantity
> must be the same, but not necessarily the unit … However this is
> not supported with differing ["Special"](link to UCUM spec) units
> on non-ratio scales in UCUM (e.g. Fahrenheit (`[degF]`) or
> Celsius (`Cel`)) where a function is required to transform the
> unit into base units. Attempting to operate on quantities with
> invalid or "special" units will result in null.*

In CQL specifically, "empty (`{ }`)" in FHIRPath maps to **`null`**
(CQL is three-valued and uses `null` rather than the empty
collection for an undefined scalar result). The wording must
therefore be adapted: "result in `null`" rather than "result in
empty (`{ }`)". Apply the same clarification to the comparison
operators if they currently inherit the "differing units" language,
so behavior is uniformly defined for special units across `+`, `-`,
`<`, `<=`, `>`, `>=`, and equality.

#### Justification

This is exactly what the reporter is asking for and exactly what the
FHIRPath ITS work group has already agreed to and applied via
[FHIR-41343](https://jira.hl7.org/browse/FHIR-41343). CQL's quantity
semantics are layered on top of FHIRPath's quantity semantics; any
divergence between the two on what `1 'Cel' + 1 '[degF]'` returns
would be a footgun for implementers (most CQL engines reuse a
FHIRPath quantity engine underneath). The Zulip discussion behind
FHIR-41343 reached clean consensus (Grahame Grieve, Paul Lynch) on
returning the "no-value" sentinel rather than raising an error,
which fits CQL's existing convention of using `null` for undefined
arithmetic results (overflow, mismatched dimensions, etc.). This
disposition keeps CQL aligned with its base specification and
codifies the only behavior UCUM actually permits.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than restating the rule in CQL, **normatively defer to
FHIRPath**: add a single short paragraph to CQL's quantity-operators
section that says quantity arithmetic in CQL inherits the semantics
of FHIRPath §6.6, including the handling of UCUM "special" units on
non-ratio scales, and that — as in FHIRPath — such operations yield
the CQL `null` value (the CQL equivalent of FHIRPath empty). Add a
non-normative note listing `Cel`, `[degF]`, and `B` as the most
commonly encountered special units, and a single worked example
showing `1 'Cel' + 1 '[degF]' returns null`.

#### Justification

CQL already inherits a great deal of arithmetic semantics from
FHIRPath. Replicating the full UCUM-special-units paragraph in two
specifications creates a maintenance burden: the next time UCUM's
position on this evolves, both specs must be updated in lockstep,
and divergence is the predictable failure mode. A defer-and-summarize
approach gives implementers exactly the answer they need
(`null`) without committing CQL to maintain a parallel rule
statement. The downside is that CQL readers must also have FHIRPath
§6.6 open to see the full rule; the worked example mitigates that.
This is a real trade-off vs. Disposition A, but it scales better
across future UCUM/FHIRPath alignment tickets.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-52039 as **Not Persuasive / Already Specified** on the
grounds that CQL quantity arithmetic is already defined to inherit
FHIRPath semantics and FHIR-41343 already applied the special-units
rule to FHIRPath §6.6. No change to the CQL specification is
required because CQL implementations following the spec will
already produce `null` for `1 'Cel' + 1 '[degF]'` via the
FHIRPath inheritance chain. Close with a pointer to FHIR-41343 and
its applied resolution.

#### Justification

If CQL's spec text already says (somewhere — in the operators
section or a general "this builds on FHIRPath" preamble) that
quantity semantics follow FHIRPath, then strictly speaking no
normative change is needed. Declining avoids spec churn and avoids
the risk of subtly mis-stating the rule in CQL terms. The downside
is that this is only defensible if the inheritance is currently
explicit and unambiguous; in practice CQL spells out a number of
quantity operators directly with their own normative text, so a
plain decline likely leaves the question open for readers who don't
chase the FHIRPath cross-reference. The CDS work group should
verify the current CQL text before choosing this path.

---

### Recommendation

**Recommended disposition:** A

The change being asked for is small, precisely scoped, and already
agreed-upon in the upstream specification. FHIRPath §6.6 has been
updated and applied via FHIR-41343 with clean Zulip consensus that
the result should be the no-value sentinel rather than an error.
CQL has historically *spelled out* its quantity operators in its
own normative text (rather than relying purely on FHIRPath
inheritance), so an explicit CQL-side statement will be clearer to
implementers than either a defer-to-FHIRPath note (Disposition B)
or silence (Disposition C). The translation cost is minor — the
only substantive edit is "empty (`{ }`)" → "`null`" to match CQL's
value model. Disposition B is a reasonable fallback if CDS prefers
to minimize duplication and is willing to add the inheritance
clause across CQL's quantity section uniformly. Disposition C
should be avoided unless the work group can point to existing CQL
text that already makes the inheritance fully unambiguous.
