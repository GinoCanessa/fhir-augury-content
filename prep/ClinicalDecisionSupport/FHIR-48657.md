## Summary

Bryn Rhodes is requesting that the `PlanDefinition/$apply` operation defined in
FHIR Core be brought into alignment with the **CRMI (Canonical Resource
Management Infrastructure) Operation Pattern Profiles**. Specifically, the
operation should conform to the following CRMI patterns: **Artifact**,
**Artifact Version Bindable**, **Manifestable**, **Parameterizable**, **Data
Configurable**, and **Artifact Endpoint Configurable**. In practical terms,
this means extending the `$apply` operation signature so that callers can pass
canonical artifact references (with version binding), supply a manifest, pass
typed input parameters that are bound to the PlanDefinition's declared
parameters, supply a data bundle (or pointer to data) for evaluation, and
identify endpoints from which referenced artifacts and data should be
resolved. No related Jira tickets, Zulip discussions, or GitHub items were
surfaced through cross-reference or search lookups for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revise the `OperationDefinition/PlanDefinition-apply` definition in FHIR Core
(clinical reasoning module) so that its in/out parameters and conformance
statements explicitly conform to the CRMI Operation Pattern Profiles cited in
the ticket. Concretely:

- **Artifact** — formalize that `$apply` operates on a single canonical
  Artifact (the PlanDefinition), accepting `canonical` plus `id`/`url` +
  `version` parameters consistent with the CRMI Artifact pattern.
- **Artifact Version Bindable** — add/clarify a `version` (or
  `artifactVersion`) input parameter that pins the PlanDefinition to a
  specific version when resolving canonicals.
- **Manifestable** — add a `manifest` input (typed as `canonical(Library)`
  or equivalent per CRMI) so callers can supply a version manifest used to
  resolve dependent canonicals (Libraries, ValueSets, ActivityDefinitions,
  Questionnaires, etc.).
- **Parameterizable** — add a `parameters` input (`Parameters`) whose parts
  are bound by name/type to the PlanDefinition's declared `parameter`
  elements, consistent with the CRMI Parameterizable pattern.
- **Data Configurable** — add `data` (Bundle), `dataEndpoint` (Endpoint),
  and `useServerData` (boolean, default true) inputs so the data context
  for evaluation can be supplied explicitly or resolved from a remote FHIR
  endpoint, instead of always evaluating against the server's own store.
- **Artifact Endpoint Configurable** — add `contentEndpoint` and
  `terminologyEndpoint` (Endpoint) inputs so the resolution of referenced
  knowledge artifacts and value sets can be redirected to remote endpoints.

Update the operation's narrative and parameter documentation in
`source/plandefinition/operation-plandefinition-apply.xml` (and the
corresponding markdown), and add the CRMI pattern profiles to the
operation's `conformsTo` (or equivalent) so tooling can validate
conformance. Mirror the changes (where applicable) on
`ActivityDefinition/$apply` to keep the two siblings aligned.

#### Justification

This is exactly the alignment the CRMI IG was designed to drive: the CRMI
Operation Pattern Profiles exist precisely so that core operations like
`PlanDefinition/$apply` have a consistent, machine-checkable shape across
implementations. Accepting the request improves interoperability between
core FHIR servers and CRMI-conformant knowledge servers, removes
ambiguity about how callers pass parameters / data / endpoint context,
and matches the direction CDS WG has already taken with related
operations (`Library/$evaluate`, `Measure/$evaluate-measure`, etc.).
Bryn Rhodes, the ticket reporter, is the lead author of CRMI and of the
clinical reasoning module, so the proposed alignment reflects an
informed editorial intent rather than a speculative request.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than restructuring the core `PlanDefinition/$apply` parameter list
in FHIR Core, leave the core operation's existing signature stable and
instead **profile** `$apply` in the CRMI IG itself: publish a CRMI-flavored
OperationDefinition (e.g. `crmi-plandefinition-apply`) that `conformsTo`
the six pattern profiles and adds the parameterization / data / endpoint
inputs there. Core FHIR would then add a non-normative note on
`PlanDefinition/$apply` pointing implementers at the CRMI profile when
they need parameterization, manifesting, or endpoint configurability.
Optionally, add only the lowest-risk inputs (`parameters`,
`contentEndpoint`, `terminologyEndpoint`, `dataEndpoint`, `data`,
`useServerData`) to core to avoid duplicating common shapes, and keep
`manifest` and version-binding semantics in CRMI.

#### Justification

`PlanDefinition/$apply` has been in the spec across multiple normative-
adjacent releases and has many existing implementations; expanding its
parameter set in core risks churn for implementers who already have a
working operation surface and who are not consuming CRMI. Profiling the
operation in CRMI keeps the new conformance contract opt-in, lets CRMI
evolve at its own cadence, and avoids forcing the CR module's release
schedule to chase CRMI's. It also keeps the core operation simpler,
which is beneficial for implementers who only need the basic "apply this
plan to this subject" shape.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that conformance to the CRMI
Operation Pattern Profiles is properly the responsibility of the CRMI
IG, not FHIR Core. Close the ticket as "Not Persuasive — Already
addressed by CRMI" and direct implementers who need parameterization,
manifesting, data configuration, or endpoint configuration on `$apply`
to use the CRMI IG's profiled operation rather than altering the core
operation.

#### Justification

FHIR Core operations are intentionally minimal contracts; richer
configuration patterns (manifests, endpoint redirection, parameter
binding) are exactly the kind of cross-cutting infrastructure CRMI was
chartered to standardize. Pulling all six pattern profiles into core
expands the surface area of an operation that already has an established
implementation base, with corresponding regression risk. CRMI can — and
already does — express these patterns without requiring a core change,
so the request can be satisfied without modifying FHIR Core.

---

### Recommendation

**Recommended disposition:** A

Accepting the request is the right call. The CRMI Operation Pattern
Profiles were designed to be the canonical shape for knowledge-artifact
operations like `PlanDefinition/$apply`, and `$apply` is the flagship
operation of the clinical reasoning module — having it diverge from the
patterns CDS WG itself publishes in CRMI is an unnecessary inconsistency
that causes real implementer friction (most notably around parameter
binding and endpoint configuration, which today are handled
inconsistently across servers). Disposition B preserves stability but
institutionalizes a permanent split between the "core" and "CRMI" shapes
of the same operation, which the WG would eventually have to reconcile
anyway. Disposition C ignores that the very inputs CRMI standardizes
(parameters, data, content/terminology endpoints) are the inputs real
`$apply` implementations already need and are inventing ad hoc.
Disposition A should be paired with a clear migration note: the new
inputs are additive and optional, existing callers continue to work, and
servers gain the ability to advertise CRMI conformance via the
operation's `conformsTo` declarations.
