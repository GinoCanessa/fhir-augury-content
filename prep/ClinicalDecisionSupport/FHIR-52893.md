## Summary

The ticket proposes adding a first-class `let` *expression* to the Clinical
Quality Language (CQL). CQL today supports a `let` *clause* only inside
queries, where it lets authors break a complex expression into named
sub-expressions. The reporter (Ewout Kramer) argues this restriction is
arbitrary: because CQL is a fully expression-based language, a `let`
expression would compose naturally everywhere expressions are allowed —
parameter defaults, tuple constructors, function bodies, and so on —
without introducing a new syntactic category. The proposed surface
syntax borrows from the ML family: `let name = expression+ in
subexpression`, where bindings are visible inside the trailing
sub‑expression. A follow‑up comment from Bryn Rhodes sketches how the
existing query `let` clause and a new expression‑level `let` would
interact (including a deliberately convoluted nested example), and
Ewout's reply shows that the new construct could plausibly *replace* the
current query‑level `let` clause over time, unifying the two.

No related Jira tickets, Zulip threads, or GitHub items were surfaced by
cross‑reference lookups for this key, so the request currently stands on
its own as a CQL grammar / semantics enhancement.

## Details

**Description:**

> Currently, the CQL spec support a `let` clause in queries. This is
> useful to simplify expressions, by allowing you to break up complex
> expressions into subexpressions, giving each of the subexpressions a
> name. It is not clear why this construct only exists in queries, and
> I'd like to propose that we add a `let` expression to CQL.
>
> Note that I propose a `let` *expression*, as this means it will fit in
> seamlessly into the fully expression-based nature of CQL itself, and
> also this will allow it to appear anywhere we normally allow
> expressions, from parameters to tuple constructions and function
> bodies. It would also not introduce new constructs this way.
>
> There is a lot of prior art for using let expressions, and we can
> directly borrow it from the ML family of languages where the let
> expressions introduce a new scope explicitly. Roughly:
>
> ```
> let [name = expression]+ in [subexpression]
> ```
>
> The result of the evaluation is the result of evaluating
> `subexpression` where subexpression can refer to variables bound in
> the first part of the let. Note that later descendants of ML
> simplified the syntax somewhat and allow leaving out the `in`, but
> that seems slightly incompatible with the structure of CQL.
>
> Some examples:
>
> ```cql
> define "Patient16To23AndFemale":
>   let olderThan16 = AgeInYearsAt(start of MeasurementPeriod) >= 16,
>       youngerThan24 = AgeInYearsAt(start of MeasurementPeriod) < 24,
>       female = Patient.gender in "Female Administrative Sex"
>   in olderThan16 and youngerThan25 and female
> ```

**Comments:**

- **bryn.rhodes (2025-09-16):** Sketches several variant forms,
  including `let x: 2 + ({1, 2}) y where y > 1 return x`, a
  "ConvolutedLet" example that nests a `let` inside an `exists` over an
  `Encounter` retrieve, and a re-formulation of the
  `Patient16To23AndFemale` example that uses `:` (colon, matching the
  existing query `let` clause's binding syntax) instead of `=` and
  `return` instead of `in`. The comment is exploring how a new
  expression-level `let` would interact lexically with the existing
  query-level `let` clause and what binding/terminator tokens would feel
  consistent.

- **ewout (2025-09-16):** Notes that formatting was lost in his original
  description and re-posts the `Patient16To23AndFemale` example using
  `:` for bindings and `in` for the body. He then shows a more
  elaborate, realistic example — a medication / daily‑dose / MME
  pipeline that nests two `let` expressions inside a query's `return`
  clause — to argue that the expression form could in fact subsume the
  current query-level `let` clause entirely.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new `let` *expression* production to the CQL grammar, in addition
to the existing query-scoped `let` clause. Concretely:

- **Grammar:** introduce `letExpression : 'let' letBinding (',' letBinding)* 'in' expression ;`
  with `letBinding : identifier ('=' | ':') expression ;`. Make
  `letExpression` an alternative under the top-level `expression`
  production, so it is legal anywhere an expression is legal —
  parameter defaults, tuple element values, function bodies, query
  `where`/`return`/`sort` sub-clauses, etc. Settle on **one** binding
  separator (`=` matches Ewout's original proposal and ML tradition; `:`
  matches the existing query `let` clause and Bryn's variant) — see
  Recommendation.
- **Scoping semantics:** each `letBinding` introduces a name into a new
  lexical scope that is visible (a) in the body following `in`, and (b)
  in *subsequent* bindings within the same `let` (sequential, not
  mutually recursive — matching `let*` rather than `letrec` from ML/
  Scheme). Names shadow outer scopes. The expression's value is the
  value of the body.
- **ELM mapping:** translate `let x = E1 in B` into the existing ELM
  `Query` with a `let` clause over a single-tuple source, or — better —
  add a dedicated `Let` ELM expression node so engines that consume ELM
  can implement it without going through query machinery. Update the
  CQL→ELM translator and the ELM XSD/JSON schema accordingly.
- **Spec touch points:** update the CQL Reference grammar (Appendix B),
  the language tour ("Author's Guide") section on queries to
  cross-reference the new expression form, and the ELM specification's
  expression catalog. Provide normative scoping examples and at least
  one informative example for each context (parameter, tuple, function
  body, nested `let`).
- **Conformance:** existing query `let` clauses remain syntactically
  valid; no breaking change. Engines that do not yet implement the new
  node MUST reject CQL that uses it (rather than silently mis-evaluate)
  via a translator-emitted error keyed off the published CQL version.

#### Justification

The request is a straightforward generalization of an existing,
well‑understood construct. CQL is already an expression language and
already has `let` semantics — only the *placement* is restricted, and
that restriction has no apparent semantic motivation. Allowing `let`
anywhere an expression is allowed eliminates today's awkward workarounds
(synthetic single-row queries used purely to introduce a name, or
duplicated sub-expressions that hurt readability and cost extra
evaluations). The construct has decades of prior art in ML, OCaml,
F#, Haskell, Scala, Rust, and Scheme, so the design space is
well-understood and reviewers can borrow established scoping rules
rather than invent new ones. Finally, the change is additive: existing
CQL is unaffected, so the upgrade path for libraries and authoring
tools is purely opt-in.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a brand-new `let` *expression* alongside the existing
query-level `let` clause, **unify the two** by promoting `let` to be an
expression form and then *re-defining* the query-level `let` clause as
sugar for the expression form. Two concrete sub-options:

- **B1 (expression-only, query clause becomes sugar).** Introduce
  `letExpression` exactly as in Disposition A, then specify the existing
  query `let` clause as desugaring to a `letExpression` wrapping the
  query's `where`/`return`/`sort` body. Authors may continue to write
  the query-level form for readability; the spec describes only one set
  of scoping rules, and ELM can collapse to a single `Let` node.
- **B2 (function-call sugar, no new grammar).** Do not extend the
  grammar at all. Instead, document the idiom of defining a small
  inline `define function` (or use an immediately-invoked function
  expression if/when CQL gains lambdas) as the canonical way to name a
  sub-expression outside a query. This addresses the underlying
  readability need without adding a new construct.

#### Justification

B1 captures the same expressive power as Disposition A while giving the
specification a single, unified scoping story — Ewout's own follow-up
comment explicitly observes that the expression form "could even
replace the current let clause", which is a strong hint that the
language designer sees the unification as desirable. The cost is a
slightly larger spec edit (rewriting the query `let` clause prose as
desugaring) and a one-time clarification in the CQL→ELM translator,
which is offset by the long-term simplification.

B2 is the most conservative option: it adds zero grammar, zero ELM
nodes, and zero implementer work, at the cost of being noticeably more
verbose for the parameter / tuple / function-body cases the reporter
calls out, and at the cost of polluting the function namespace with
single-use helpers.

---

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope for the current ballot cycle**
and/or **insufficiently motivated by implementer demand**. Recommend
that the reporter (a) collect concrete authoring examples from
production CQL libraries (e.g., eCQM content, CDS Hooks libraries)
where the lack of an expression-level `let` causes measurable
duplication or readability problems, and (b) bring those examples back
as a follow-up ticket targeted at a future CQL minor version. Direct
the reporter to the existing query-level `let` clause and to the
`define`/`define function` mechanisms as the workarounds for now.

#### Justification

CQL grammar changes are not free: every new production has to be
implemented by every conformant translator and engine, documented in
authoring tools and IDE plugins, taught in training material, and kept
backward-compatible forever. The ticket as filed is largely a design
intuition ("it is not clear why this construct only exists in queries")
plus syntactic sketches; it does not enumerate concrete authoring pain
points or quantify how often the absence of a `let` expression bites
real CQL libraries. Without that evidence, the workgroup may
reasonably prefer to spend ballot bandwidth on changes with documented
implementer demand.

---

### Recommendation

**Recommended disposition:** B (specifically B1 — accept the `let`
expression and respec the query `let` clause as sugar for it).

The underlying request is sound and well-motivated by language-design
first principles: CQL is an expression language, `let` is already in
the language, and the current restriction to query scope is an
accident of history rather than a deliberate design choice. Disposition
A delivers the value but leaves the spec carrying two parallel
descriptions of `let` semantics indefinitely. B1 achieves the same user-
visible outcome while collapsing the spec onto one scoping story, and
the ticket reporter himself signalled in the comment thread that the
new expression form "could even replace the current let clause" —
suggesting the workgroup will not face strong resistance to the
unification. B1 is also kinder to ELM consumers, who would otherwise
have to handle two ways of expressing the same binding. The remaining
open design questions — `=` vs `:` as the binding separator, and
whether bindings are sequential (`let*`) or mutually recursive
(`letrec`) — should be decided in the ballot discussion; the
recommendation is `:` (consistent with the existing query clause and
with CQL's general use of `:` in `define`) and sequential (`let*`),
which is what every example in the ticket and comments implicitly
assumes. Disposition C is not recommended: the reporter is the CQL
specification's editor and his judgement about language coherence
should be given substantial weight, and the change is additive and
cheap to opt out of for engines that defer adoption.
