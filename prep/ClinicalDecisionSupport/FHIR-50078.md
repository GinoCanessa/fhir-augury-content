## Summary

The ticket reports that on the CDS Hooks Hook Library pages, the documentation for the `userId` context field links the `Practitioner` and `PractitionerRole` resource types to FHIR resource pages that resolve to the FHIR R5 version (i.e., versionless `https://www.hl7.org/fhir/practitioner.html` and `practitionerrole.html` URLs that redirect to the current published FHIR — currently R5). Because CDS Hooks is intentionally **FHIR-version-agnostic** — the FHIR version exposed by the implementing CDS service is what governs resource shapes — pinning readers to a specific FHIR version (visually or via the redirect target) is misleading. The reporter recommends removing the resource links and adding a note that the FHIR version should match the CDS service's implementation, and notes that the same problem exists on multiple Hook Library pages (not just one). No cross-referenced Jira tickets, Zulip threads, or GitHub PRs/issues were found in FhirAugury for this ticket; it stands on its own.

## Details

**Description:**

> The Practitioner and PractitionerRole links currently point to FHIR R5 versions, which may cause confusion since CDS Hooks is intended to be version-agnostic regarding FHIR. It is recommended to remove these links and include a note indicating that the FHIR version should match the CDS service's implementation. This issue is also present on other Hook Library pages.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit every Hook Library page (CDS Hooks specification) where the `userId`
context-field row links FHIR resource types — `Practitioner`,
`PractitionerRole`, and similarly `Patient` and `RelatedPerson` where
they appear — to:

1. Remove the inline hyperlinks to `https://www.hl7.org/fhir/<resource>.html`,
   leaving the resource type names as plain text (or as inline code such as
   `Practitioner`).
2. Add (or extend the existing) editorial note near the `userId` row, e.g.:
   > The FHIR resource types referenced for `userId` are version-agnostic.
   > The resource definitions in force are those of the FHIR version
   > implemented by the CDS service.

Apply the same edit to the **Hook page template** so that future hook
pages do not reintroduce versioned links.

#### Justification

This matches the reporter's request literally and aligns with CDS Hooks'
long-standing version-agnostic stance on FHIR — the spec itself does not
bind to an R-version, and any link that resolves to a single R-version
(today the unversioned `hl7.org/fhir/...` redirects land on R5) implies
otherwise to readers. Removing the links is also the lowest-risk option:
no link rot, no surprise behaviour change when HL7 publishes a new FHIR
version (R6, R7, …) that would silently re-target the same URLs.
Updating the template at the same time prevents the same defect from
re-emerging in new hooks.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the links to FHIR resource types, but change the URL form so the
target version is **explicit and stable per CDS Hooks release** rather
than implicitly tracking the latest published FHIR. Two flavours:

- B1 — Link to the canonical resource page that is *not* version-pinned
  (e.g., `https://hl7.org/fhir/uv/<...>` or the FHIR registry landing
  page for `Practitioner`) and add the editorial note that the
  authoritative shape is determined by the CDS service's FHIR version.
- B2 — Pin all references to the lowest commonly-supported FHIR version
  for CDS Hooks (FHIR R4, currently the most widely deployed FHIR
  version in CDS-Hooks-capable EHRs), with the same editorial note that
  the implementer's actual FHIR version takes precedence.

In either variant, also update the Hook page template to enforce the new
convention.

#### Justification

Implementers commonly want a one-click jump to the resource definition
while reading a hook spec, especially for newcomers. Removing the links
entirely (Disposition A) costs that affordance. A version-stable link
plus an editorial note preserves discoverability while still
communicating that the CDS service's FHIR version is normative. B2 has
the additional benefit of matching the FHIR version most CDS Hooks
deployments are written against today, but accepts the cost that the
links will look "out of date" once R4 references are obviously older
than the deployed FHIR version of new readers.

---

### Disposition C: Decline

#### Proposal

Decline as a documentation-only nit and instead add a single, prominent
note to the Hook Library overview page (one location, not per-hook)
clarifying that all FHIR resource references in CDS Hooks are
version-agnostic and that the FHIR version of the CDS service governs
the actual resource shape. Leave the existing per-hook links unchanged.

#### Justification

The links themselves are not technically incorrect — `Practitioner` and
`PractitionerRole` exist across all current FHIR versions, and the
`userId` semantics described do not depend on version-specific elements.
The risk of confusion is real but bounded; a single editorial note at
the top of the Hook Library covers the same readers without a sweeping
edit across many pages. Mass-editing every hook page also burns review
cycles for what amounts to cosmetic clarification.

---

### Recommendation

**Recommended disposition:** A

The reporter has correctly identified that the unversioned
`hl7.org/fhir/<resource>.html` links resolve to whichever FHIR version
HL7 has currently published as the "core" FHIR (R5 today, R6 tomorrow),
which actively misleads readers about CDS Hooks' deliberate
version-agnostic posture. Disposition A is the lowest-risk, smallest
ongoing-maintenance fix: removing the link plus a short editorial note
will not silently break or change meaning when a new FHIR version is
published. Disposition B's "stable link" variants either depend on a
URL pattern that doesn't really exist for an unversioned FHIR resource
(B1) or hard-codes one version (B2), which contradicts the very
principle the ticket is asking the spec to clarify. Disposition C
under-corrects: the reporter explicitly notes the problem appears on
*multiple* pages and is asking for a per-page fix, and a single overview
note is unlikely to be seen by a reader who lands directly on an
individual hook page from a search engine. Bundling the page-template
edit into the change (per Disposition A's second bullet) is what makes
this a one-time fix rather than a recurring janitorial task.
