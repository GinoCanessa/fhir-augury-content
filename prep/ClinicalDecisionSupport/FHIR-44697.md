## Summary

FHIR-44697 asks the Clinical Decision Support work group to add a worked
example to the Clinical Practice Guidelines on FHIR (CPG-on-FHIR) IG that
illustrates how a `Questionnaire` is used to gather information that is
missing from the patient record but is required by a CPG. The reporter
(Bryn Rhodes) gives a concrete motivating scenario: a CPG that needs to
calculate BMI and eGFR but is missing a `Weight` data point — the example
should show how the CPG engine recognizes the gap, produces or references
a `Questionnaire` to solicit the missing observation, and then resumes
the calculation once the response is available. The ticket has no
explicit links to other Jira tickets, no Zulip references, and no GitHub
items in the cross-reference index, so the request stands on its own as
an example/illustrative-content addition rather than a normative spec
change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new self-contained worked example to `HL7/cqf-recommendations`
(the CPG-on-FHIR IG) demonstrating the "missing data → Questionnaire →
resume calculation" loop end-to-end. Concretely:

- Author a new scenario folder under
  `input/fsh/examples/activity-examples/test-cases/` (e.g.,
  `bmi-egfr-missing-weight/`) containing:
  - A `Patient` instance missing a recent `Weight` `Observation`.
  - A `PlanDefinition` action whose CQL data requirement (`Weight`)
    fails to resolve, and whose `dynamicValue` / `action.input`
    surfaces the missing element.
  - A `Questionnaire` (or referenced `Questionnaire` canonical) shaped
    to collect the missing `Weight` (and any directly dependent items
    needed for BMI/eGFR — height for BMI, age/sex/serum creatinine for
    eGFR if those are also gappy in the scenario).
  - A `QuestionnaireResponse` example showing the captured `Weight`,
    plus the post-response `Observation` derived from it.
  - The two BMI / eGFR result `Observation` instances that the CPG
    produces once the gap is filled.
- Add a new narrative page under `input/pagecontent/` (e.g.,
  `example-missing-data-questionnaire.md`) walking the reader through
  the loop, with links to the FSH-generated artifacts, and register
  the page in `sushi-config.yaml` / `input/includes/menu.xml`.
- Add (or reuse) a small `Library` CQL file under `input/cql/` for the
  BMI/eGFR calculation logic so the example exercises a real CQL data
  requirement, and wire the `Library` in via the example
  `ActivityDefinition` / `PlanDefinition`.
- No new profiles or extensions are introduced; the example uses the
  existing CPG-on-FHIR profiles and standard SDC `Questionnaire` /
  `QuestionnaireResponse` shapes. CPG-on-FHIR depends on
  `hl7.fhir.extensions.r5` already; no new IG dependencies are
  required.

#### Justification

This is precisely what the ticket requests and it stays within the
authoring conventions of the target IG (FSH under `input/fsh/`,
narrative under `input/pagecontent/`, CQL under `input/cql/`). The
"missing data drives a Questionnaire" pattern is a recurring CPG-on-
FHIR teaching point for which no end-to-end worked example currently
exists, and BMI/eGFR are widely-recognized motivating calculations
that make the example concrete without inventing a new clinical
domain. Because the change is purely additive (new test-case folder +
new narrative page) it carries no backward-compatibility risk and
does not touch normative profiles or extensions; it can land in a
patch release of the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a new bespoke example to `HL7/cqf-recommendations`,
collaborate with the SDC and CRMI work groups to host the example in
an SDC-aligned location (or in a dedicated examples sub-IG /
companion guide) and link to it from the CPG-on-FHIR narrative. Two
sub-options are reasonable:

1. **SDC-hosted example, CPG cross-link.** Add the worked example to
   the SDC IG's example set (since the heart of the demonstration is
   `Questionnaire` / `QuestionnaireResponse` mechanics) and add a
   short narrative page in CPG-on-FHIR that points at it and explains
   the CPG-side wiring (`PlanDefinition` action emitting / referencing
   the `Questionnaire`).
2. **CRMI-aligned data-requirements example.** Frame the gap-detection
   step using the CRMI data-requirements / `$data-requirements`
   surface so the example doubles as a CRMI illustration of how a
   `Library`'s data requirements drive `Questionnaire` synthesis.

In either sub-option, the BMI/eGFR-needs-Weight scenario and its
artifact set are still produced; the difference is *where* they live
and *which* IG owns the narrative.

#### Justification

The mechanics that make this example interesting — `Questionnaire`
adaptive shaping, `QuestionnaireResponse` extraction into discrete
`Observation`s, and CQL data-requirement gap detection — are owned in
large part by SDC and CRMI rather than CPG-on-FHIR proper. Hosting
the example where the underlying machinery is normatively defined
keeps it maintained against the source-of-truth specs and reduces the
risk of CPG-on-FHIR drifting from SDC patterns. The trade-off versus
Disposition A is added coordination cost (two work groups, two PR
trains) and a slower path to a published example, in exchange for
reuse beyond CPG.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the basis that the requested illustrative
content overlaps with material already shipping (or planned) in
adjacent IGs and does not warrant a CPG-on-FHIR-owned example. The
ticket would be closed as *Persuasive with mod / Not Persuasive* with
a pointer to (a) the existing CPG-on-FHIR `RecordInference` example
family under `input/fsh/examples/activity-examples/` for the
"derive a missing observation" pattern, and (b) the SDC IG's
`Questionnaire` / `QuestionnaireResponse` extraction examples for the
data-collection half of the loop. Implementers needing the combined
flow would compose it from those building blocks.

#### Justification

CPG-on-FHIR already demonstrates inference / record-derivation
patterns (the `RecordInference` activity example family is the
canonical example of "fill in a missing data point"), and SDC owns
the worked examples for `Questionnaire` extraction. Adding a third,
overlapping example raises maintenance cost (every IG-publisher run
has to keep the cross-IG terminology and profile bindings in sync)
without expanding the conceptual surface that the IG teaches. This
disposition is the weakest of the three because the *combination*
("missing data → Questionnaire → resume calculation") is genuinely
not shown end-to-end anywhere today; it is offered for completeness
and to surface the maintenance argument explicitly.

---

### Recommendation

**Recommended disposition:** A

This is a low-risk, additive request whose value is exactly its
end-to-end character: today a reader can find inference examples in
CPG-on-FHIR and `Questionnaire` extraction examples in SDC, but no
single artifact set walks through "the CPG hit a data gap, asked the
user via a `Questionnaire`, and resumed". Adding that example in
CPG-on-FHIR — the IG whose `PlanDefinition` / `ActivityDefinition`
machinery owns the gap-detection step — is the smallest change that
delivers the requested teaching value. Disposition B is attractive in
principle but its coordination cost is disproportionate for an
illustrative example, and Disposition C undersells the genuine gap
the ticket identifies. The implementer should refresh the
`HL7/cqf-recommendations` repo briefing scoped to FHIR-44697 before
authoring the change so that any IG-publisher gotchas (e.g., the
existing CQL-`dynamicValue` brittleness flagged in the repo briefing's
Warnings section) are accounted for in the example's `Library` /
`PlanDefinition` wiring.
