## Summary

This ticket asks the Canonical Resource Management Infrastructure (CRMI)
specification to clarify how the `id` element of artifacts placed into the
output Bundle of the `$package` operation should be set. Today the
specification is silent on this point, and different implementations
produce inconsistent ids for the same conformance artifacts. The reporter
proposes that the `id` of each artifact in the package output be normalized
to the **tail of the canonical URL** (i.e., the segment after the last
`/`), which aligns with how conformance resources are distributed in NPM
FHIR packages. The Jira resolution has already been recorded as
**Persuasive** with text saying that the `$package` description should
state that artifacts in the bundle result SHOULD have an `id` equivalent
to the tail of the canonical URL (or the artifact URL for non-canonical
artifacts). No cross-referenced Jira tickets, Zulip threads, or GitHub
items were found by Augury for this ticket.

## Details

**Description:**

> The $package operation specification does not specify what should be
> done with the "id" element of the artifacts included in the bundle, but
> this results in inconsistent output.
>
> We should specify that the id is normalized to the tail of the
> canonical, consistent with the way that conformance resources are
> distributed in NPM packages.

**Resolution Description (already recorded on the ticket):**

> Agreed, add to the description of the package operation processing
> semantics that artifacts included in the bundle result SHOULD have an
> 'id' equivalent to the tail of the canonical url (or the artifact url
> for non-canonical artifacts).

**Comments:**

- **bryn.rhodes (2025-11-04):** Including the version in the id would
  partially address [collisions], though there is still the potential for
  collision. The only complete solution is to use the full canonical url
  and version as the id, with a non-lossy mapping for characters that
  cannot appear in ids.
- **bryn.rhodes (2025-11-04):** Note that this will in general result in
  clashes in the output if systems in different canonical bases use the
  same tail. This happens today with `condition-category` — one is in THO,
  one is in US Core.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CRMI `$package` operation specification (operation
description / processing semantics) to add a normative statement that
each artifact included in the result Bundle SHOULD have its `Resource.id`
set to the **tail of the canonical URL** of the artifact (the segment
after the final `/` in the canonical), and for non-canonical artifacts
to the tail of the artifact URL. Add a brief example showing, e.g., a
`ValueSet` with canonical `http://example.org/fhir/ValueSet/foo`
appearing in the output Bundle with `id = "foo"`. This is a
specification-text-only change to the `$package` operation page; it adds
a SHOULD-level conformance expectation but does not change the operation
signature or input/output structure.

#### Justification

This is exactly the resolution already recorded on the ticket as
**Persuasive**, and it directly aligns the `$package` output with the
established convention used by NPM FHIR packages, where conformance
resource files are named for the tail of the canonical and the
contained resource's `id` matches that tail. Choosing SHOULD (not
SHALL) preserves implementer flexibility for the edge cases the
reporter himself flagged in the comments — namely that two artifacts in
different canonical bases (e.g., `condition-category` in THO vs. US
Core) can share the same tail and therefore collide if naively merged
into one Bundle. SHOULD lets implementations deviate when collision
avoidance demands it, while still giving consumers a predictable
default.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the SHOULD "tail of the canonical" rule from Disposition A, but
also add explicit guidance for the **collision case** the reporter
raised in the comments. Specifically, extend the `$package` operation
text to say:

1. The default `id` SHOULD be the tail of the canonical (per A).
2. When two or more artifacts in the result Bundle would otherwise share
   the same `id`, implementations SHALL disambiguate — for example by
   appending the business version (`{tail}-{version}`) or by deriving a
   deterministic, non-lossy encoding of the full canonical+version into
   the `id` (using only characters allowed in `Resource.id`).
3. Optionally allow servers to advertise (e.g., via an extension on the
   returned Bundle) which id-normalization scheme was used, so consumers
   can round-trip.

#### Justification

Disposition A solves the consistency problem the ticket actually filed,
but Bryn's own comments document a real, currently-occurring collision
(`condition-category` in THO and US Core). Codifying only the tail rule
without addressing collisions risks "consistently wrong" output in the
exact scenario that motivated the comments. Adding a SHALL-level
disambiguation rule keeps the simple, NPM-aligned default for the 99%
case while preventing the package from producing two Bundle entries
with the same `Resource.id` — which is itself problematic for many
downstream tools and validators. The trade-off is a slightly larger
spec change and a small implementation burden for servers that
currently emit colliding ids silently.

---

### Disposition C: Decline

#### Proposal

Decline to mandate any normalization of `Resource.id` in the `$package`
output Bundle. Rationale category: **already addressed by existing
mechanism / out of scope of `$package`**. Consumers that need a stable
identifier for a packaged artifact already have `Resource.url` (and
`Resource.version`) which are the canonical identity of conformance
resources; `Resource.id` is, by FHIR design, a server-/transport-local
handle and not intended to carry canonical identity. If CRMI says
anything at all, it should be a non-normative note pointing consumers
at `url` + `version`.

#### Justification

In core FHIR, `Resource.id` is explicitly the logical id within a
server's resource scope, while canonical identity for conformance
resources is carried by `url` + `version`. Standardizing `id` to the
tail of the canonical conflates these two concepts and can mislead
implementers into treating the `id` in a packaged Bundle as a stable
cross-server identifier — which it is not. It also bakes in the very
collision problem the reporter flagged: two artifacts from different
canonical bases with the same tail will produce duplicate `id`s in one
Bundle, which is worse than today's "inconsistent but at least
distinct" behavior. Declining keeps the `$package` operation aligned
with core FHIR's identity model.

---

### Recommendation

**Recommended disposition:** B

The ticket has already been ruled Persuasive, so a "do nothing" outcome
(C) is unlikely to fly with the work group, and the NPM-alignment
argument behind A is genuinely strong — consumers and tooling already
expect the tail-of-canonical convention from FHIR NPM packages.
However, A as written silently inherits the collision case that the
reporter himself flagged in two follow-up comments (`condition-category`
in THO vs. US Core). Disposition B keeps A's NPM-aligned default and
its simple SHOULD-level statement, but pairs it with a small,
implementer-friendly SHALL-level rule for the collision case so that a
single `$package` result Bundle can never produce duplicate
`Resource.id`s. That converts the existing resolution text into a
specification that is both consistent *and* safe, with minimal extra
authoring effort on the CRMI `$package` operation page.
