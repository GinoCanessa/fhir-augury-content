## Summary

FHIR-51681 ("Surface Manifest Documentation") is a Clinical Decision
Support change request against the Canonical Resource Management
Infrastructure (CRMI) FHIR IG. The reporter (Bryn Rhodes) observes that
"Artifact Manifest" is a substantial topic in CRMI but is currently only
reachable via an inconspicuous link from the *Artifact Lifecycle* page,
making it hard to discover. The ticket asks the work group to (a)
promote *Artifact Manifest* to a top-level entry in the CRMI IG main
menu so it is discoverable on its own, and (b) reorder the page body so
the manifest specification content is presented first and the
background/justification discussion is secondary.

The ticket has already been triaged with a **Persuasive** resolution
("Agreed, add Manifest as a menu topic and rearrange the content so
that the background discussion is secondary and the specification
content is the focus"), and is carrying the `Block-Vote-3` label,
indicating the work group has slated it for a block-vote disposition
during a current ballot cycle. No cross-referenced Jira tickets, GitHub
items, or Zulip threads were surfaced for this key by the
cross-reference index. Keyword-similarity search did surface adjacent
but distinct CRMI/IG navigation tickets — FHIR-36349 "Split menu into
background and spec" and FHIR-34785 "simplify menu navigation" — which
share the same editorial pattern (separating normative spec content
from background discussion in menu structure and page ordering).

## Proposed Dispositions

> Note: FHIR-51681 has already been triaged as **Persuasive** with a
> resolution narrative that maps cleanly onto Disposition A below
> ("add Manifest as a menu topic and rearrange the content so that the
> background discussion is secondary and the specification content is
> the focus"). The dispositions below are written for the block-vote
> review as if the question were still open, so the work group has the
> alternatives explicit on the record.

### Disposition A: Accept As Requested

#### Proposal

Promote *Artifact Manifest* to a first-class entry in the CRMI IG main
navigation, and rewrite the page so the normative specification of the
Artifact Manifest is presented first, with background and design
rationale moved to a later section of the same page.

Specifically:

1. **Menu surfacing.** Add an *Artifact Manifest* entry to the CRMI IG
   menu structure so it appears at the same level as other top-level
   conceptual topics (e.g., *Artifact Lifecycle*), rather than only
   being reachable via an in-body link from the *Artifact Lifecycle*
   page.
2. **Page reorganisation.** Restructure the *Artifact Manifest* page
   body so that the leading sections describe what an Artifact Manifest
   *is* — its FHIR representation, the elements/extensions involved,
   how it is referenced from `$package` (e.g., the `manifest`
   parameter), and how it interacts with versioned/unversioned
   canonicals — and so that the existing motivational/background
   discussion (use cases, problem framing, "why we need this") is moved
   to a clearly-labelled later section on the same page.
3. **Cross-link cleanup.** Update the *Artifact Lifecycle* page so the
   in-body reference to *Artifact Manifest* remains, but is no longer
   the only practical entry point.

#### Justification

This is exactly what the reporter asked for, what the resolution
narrative records the work group has agreed to, and what the related
Zulip discussion in `cql > unversioned Measure.library reference`
indicates implementers need — Bryn Rhodes is recommending the
Artifact Manifest mechanism to implementers, and implementers are
reporting they cannot easily locate it. It is also consistent with the
editorial pattern being applied in adjacent tickets FHIR-36349 ("Split
menu into background and spec") and FHIR-34785 ("simplify menu
navigation"), so adopting the same shape here strengthens IG-wide
consistency rather than creating a one-off layout. The change is
purely editorial/navigational — it does not alter any FHIR resource,
profile, or operation definition — so the implementation cost is low
and there is no normative risk.

---

### Disposition B: Alternative Approach

#### Proposal

Surface *Artifact Manifest* navigationally (Disposition A's menu and
cross-link work), but instead of restructuring the existing page,
**split** it into two pages: a new top-level *Artifact Manifest*
specification page that contains only normative content (definition,
elements, package/manifest parameter behavior, conformance), and a
sibling *Artifact Manifest — Background and Rationale* page (or a
section under a CRMI "Background" parent) that retains the
motivational discussion. The current page would redirect to the new
specification page.

#### Justification

A page split — rather than a within-page reorder — has two advantages
over Disposition A. First, it gives the *background* material a stable
URL of its own, so any external citations (Zulip messages, Jira
tickets, slide decks) that currently link into specific anchors on the
combined page do not silently change meaning when the spec content
moves above them. Second, it matches what FHIR-36349 explicitly
proposes at the menu level ("split menu into background and spec"),
so the *Artifact Manifest* topic would model the eventual end state
the work group is moving toward, instead of partially reorganising
within a single page only to be reorganised again later.

The trade-off is more editorial work (a new page, redirect
configuration, link updates) and a marginally larger change set
relative to the simple reorder Disposition A proposes.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for a normative change request: the menu
structure and intra-page ordering of the CRMI IG are an editorial
matter, and changes of this kind should be handled as a documentation
issue against the IG repository (or a publication-time editorial pass)
rather than as a tracked Jira change request that consumes block-vote
bandwidth.

#### Justification

Strict reading: nothing in FHIR-51681 changes a resource, profile,
extension, value set, operation, or invariant. The menu link to
*Artifact Manifest* already exists; the page already exists; the
content already exists. The request is purely about discoverability
and ordering. A workgroup could reasonably argue that block-vote
bandwidth should be reserved for normative content and that editorial
adjustments be made by the IG editor as housekeeping, with this Jira
ticket closed as "Not Persuasive — editorial".

This is the weakest of the three options because the reporter has
identified a genuine implementer-visible problem (corroborated by the
`cql > unversioned Measure.library reference` thread), and because the
work group has *already* agreed in the resolution that the change is
warranted; declining now would reverse that consensus without new
information.

---

### Recommendation

**Recommended disposition:** A

Disposition A matches the reporter's request, the work group's
already-recorded *Persuasive* resolution, and the implementer evidence
from the `cql` Zulip thread that the Artifact Manifest mechanism is
being recommended faster than its documentation can be found. It is
the lowest-risk change (editorial-only, no normative impact) and it is
consistent with the editorial direction signalled by FHIR-36349 and
FHIR-34785. Disposition B is a defensible refinement and the work
group may wish to fold its "background lives on its own page" idea
into the implementation of A — but committing to a full page split
here would expand the scope beyond what the ticket asked for and
beyond what the resolution captured. Disposition C would walk back an
agreement the work group already made and is not recommended. Adopt
A, with an explicit option to escalate to B's page split if the
restructured page becomes unwieldy in editorial review.
