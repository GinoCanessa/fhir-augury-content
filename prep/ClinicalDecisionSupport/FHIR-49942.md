## Summary

The reporter, working with the Clinical Practice Guidelines (CPG-on-FHIR)
implementation guide, observes that the CPG profile of `Patient` makes
`Patient.name` mandatory (1..1). They argue this content-level constraint
is not appropriate in CPG, for two reasons: (1) implementations may legitimately
need to handle anonymised patients (e.g., research, public-health, or de-identified
analytics scenarios where a name is suppressed), and (2) CPG is intended to be
infrastructural guidance — guidance about how to represent and execute clinical
logic — rather than a source of clinical data-content rules. Mandating `name`
constrains *usage* of the framework without serving any infrastructural
requirement of CPG itself. The reporter therefore asks the workgroup to remove
the `Patient.name` 1..1 constraint and to review the CPG profiles for any
other content-level constraints that are not justified by CPG infrastructure
and remove those as well.

The single cross-reference returned by Augury is to the FHIR element
`Patient.name`, confirming that the substantive scope of the change is the
CPG `Patient` profile's cardinality on `name` (and, by extension, an audit
of comparable cardinality / required-binding constraints elsewhere in the
CPG profiles).

## Proposed Dispositions

> Note: because Augury surfaced no GitHub cross-references, no

### Disposition A: Accept As Requested

#### Proposal

Relax the cardinality of `Patient.name` on the CPG `Patient` profile from
`1..1` back to the base FHIR cardinality (`0..*`). Concretely:

- In the CPG `Patient` `StructureDefinition`, set `min = 0` on the
  `Patient.name` element differential (and remove any `mustSupport=true`
  unless the workgroup wishes to retain a "support if present" signal).
- Sweep the rest of the CPG profile set (`Patient`, `Encounter`,
  `Observation`, `Condition`, `MedicationRequest`, `Procedure`, etc.) for
  other content-level tightenings — increased cardinality (`0..1` →
  `1..1`), required value-set bindings added on top of the base spec,
  and `mustSupport` flags applied to data-content elements — and remove
  those that do not serve a documented CPG infrastructural need. Record
  any constraints retained, with a one-sentence rationale tying each to
  CPG executability (e.g., a quality-measure denominator query, a
  PlanDefinition input requirement).
- Add a short note to the CPG profiling narrative stating the policy:
  CPG profiles constrain only what is required for the CPG knowledge /
  execution infrastructure; jurisdictional and clinical content
  constraints are deferred to derived profiles (e.g., national IGs,
  measure-specific profiles).

#### Justification

This directly addresses the reporter's concern and aligns with the
explicit positioning of CPG-on-FHIR as infrastructural guidance.
Mandating `Patient.name` blocks legitimate use cases — de-identified
analytics, research cohorts, public-health line lists — without enabling
any CPG execution behaviour, since none of the CPG knowledge artifacts
(PlanDefinition, ActivityDefinition, Library, Measure) need to read
`Patient.name` to function. It is also consistent with the broader FHIR
profiling principle that an infrastructural / framework IG should
constrain only what its mechanics require and let downstream profiles
add jurisdictional or clinical-policy constraints. Pairing the
cardinality change with a documented constraint-review policy prevents
the same drift from recurring.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the element flagged as important but do not require it: relax
`Patient.name` to `0..1` (or `0..*`, matching base) **and** mark it
`mustSupport = true` with an `obligation` (or invariant / narrative)
that says "systems exchanging CPG-eligible patient data SHALL be capable
of producing or consuming `Patient.name` when it is known, but MAY omit
it for anonymised subjects." Apply the same "must-support, not required"
pattern to other content elements currently mandated in CPG profiles.
Document the de-identified / anonymised use case explicitly in the
profile narrative and in the CPG implementation guidance.

#### Justification

This preserves the original authors' likely intent — signalling that
`Patient.name` is a first-class element that conformant CPG systems
should handle — while removing the hard block on anonymised use. It
keeps tooling support (must-support drives validators, IG publisher
output, capability statements) without forcing every instance to carry
a name. It is also a smaller blast radius than option A: existing
implementations that already populate `Patient.name` remain conformant,
and downstream IGs can still tighten back to `1..1` where their
jurisdictional or clinical context demands it.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain `Patient.name 1..1` (and the comparable
content constraints) on the CPG profiles. Rationale: existing CPG
implementations and downstream profiles depend on the current
cardinality; relaxing it is a breaking change for conformance and would
require a new major version of the IG. Anonymised patients can be
represented by populating `Patient.name` with a `data-absent-reason`
extension or with a placeholder name, which is the established FHIR
pattern for "value not present for a regulatory/privacy reason."

#### Justification

This is defensible if (a) the workgroup judges that `Patient.name` is in
fact relied on by current CPG knowledge artifacts or downstream
consumers, and (b) the existing FHIR pattern for absent values
(`data-absent-reason` on a required element) is considered an adequate
answer to the anonymised-patient case. The trade-off is real:
implementers in research / public-health contexts do find the
"placeholder name + extension" pattern awkward, and it does not address
the reporter's broader point that CPG, as an infrastructural IG, should
not be the layer enforcing clinical-content rules.

---

### Recommendation

**Recommended disposition:** A

The reporter's framing is sound and matches how CPG-on-FHIR is described
in its own scope: it is an infrastructural IG for representing and
executing clinical knowledge, not a clinical-content profile set. The
`Patient.name 1..1` constraint serves no CPG execution requirement —
no PlanDefinition, ActivityDefinition, Library, or Measure logic needs
to read a patient's name — and it actively excludes valid use cases
(anonymised cohorts, de-identified analytics, public-health line lists).
Disposition A also lets the workgroup do the housekeeping the reporter
implicitly asks for: a one-time sweep of the CPG profiles to remove
content constraints that were inherited from copy-paste of clinical
profiles rather than driven by infrastructural need, and a documented
policy so the same drift does not recur. Disposition B is a reasonable
fallback if the workgroup wants to preserve a "must-support" signal for
tooling, but it carries the risk of being interpreted as "still
required" by implementers who conflate `mustSupport` with
`min = 1`. Disposition C should only be selected if the workgroup can
point to a concrete CPG infrastructural dependency on `Patient.name`,
which the ticket and the available cross-references do not reveal.
