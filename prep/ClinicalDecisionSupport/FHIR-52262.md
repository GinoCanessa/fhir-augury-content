## Summary

Lynn Perrine (Clinical Quality Language IG) reports a layout/presentation
issue in the rendered Clinical Quality Language (FHIR) specification:
section 1.2.1 (and its subsections) shows excessive whitespace because a
table begins at the bottom edge of the right-hand navigation pane. The
reporter asks whether the content in section 1.2.1 and its subsections
could be wrapped (in a container/section) so that the table no longer
"falls off" the right-nav region — citing section 1.9 Appendix B as an
example of a page that does not exhibit the problem. The request is
purely about the publication/presentation of the IG and does not change
any normative content, profiles, value sets, or CQL semantics. There are
no cross-referenced Jira tickets, Zulip threads, or GitHub items
associated with this issue in the Augury index, so no broader context
is available beyond the description itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the markdown/HTML source for section 1.2.1 and each of its
subsections in the CQL IG so that the section body (intro paragraphs +
the offending table) is wrapped in a single block-level container
(e.g., a `<div class="section-body">` or equivalent IG-publisher
construct) before the table begins. Mirror the pattern used by section
1.9 Appendix B, which the reporter calls out as rendering correctly.
The expected effect is that the right-hand navigation pane and the
table are vertically aligned with the section body rather than the
table starting flush against the bottom of the side-nav region, which
eliminates the visible whitespace gap. No normative content, CQL
grammar, profiles, value sets, examples, or conformance statements are
modified.

#### Justification

The reporter has identified a concrete, reproducible visual defect and
pointed to an existing in-spec example (1.9 Appendix B) that already
uses the desired layout — so the fix has a known-good reference and
carries essentially zero risk of regressing normative content. CDS WG
ballot guidance generally favours accepting "Persuasive" non-substantive
publication fixes that improve readability of an IG, especially during
ballot reconciliation, because they reduce reader friction without
changing what implementers must build. Because this is an IG-page
(intro/narrative) edit only, it does not require IG-publisher tooling
changes and can ship in the next CQL IG publication cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the underlying issue as a publication-template / CSS concern
rather than a per-page wrapper edit. Instead of (or in addition to)
wrapping the section bodies, adjust the IG's shared layout/styling so
that any section whose first child is a wide `<table>` gets the same
containing behaviour automatically. Concretely: add a CSS rule (or
template tweak) that ensures `.content > table:first-child` (and
analogous selectors used by the IG publisher template the CQL IG
inherits) is rendered inside a flow container that sits below the
right-nav float, and apply it once at the site level. Optionally still
make the local 1.2.1 wrapper change as a belt-and-braces measure.

#### Justification

The reporter's diagnosis ("table starting at the end of the right
navigation pane") suggests the root cause is the interaction between a
floated/sticky side-nav and a wide first-child table — which is a
template-level concern, not unique to section 1.2.1. Fixing it once in
shared layout/CSS prevents the same problem recurring in future intro
pages or other CQL-family IGs that share the template, and avoids a
pattern of one-off wrappers being added every time a section starts
with a table. The trade-off vs. Disposition A is higher up-front
investigation cost (and possibly coordination with the IG-publisher /
template maintainers) in exchange for a more durable fix.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Not Persuasive" / "Won't Fix" on the grounds that the
described whitespace is a cosmetic side-effect of the standard IG
publisher template, is not interfering with the readability or
machine-consumability of any normative content, and that any layout
change risks divergence from the publisher's standard output that other
CQL-family IGs rely on for visual consistency.

#### Justification

This is a defensible-but-weak position: it could be argued that
publication cosmetics are owned by the IG publisher rather than the CQL
IG content, and that the CDS WG should focus its limited reconciliation
bandwidth on substantive ballot comments. However, it would leave a
real, reader-visible defect in the published IG and ignores the fact
that the reporter has already provided an in-spec proof-of-fix
(Appendix B), which makes a "Won't Fix" disposition harder to defend
in reconciliation.

---

### Recommendation

**Recommended disposition:** A

This is a small, self-contained, narrative-only fix to one intro
section of the CQL IG, and the reporter has helpfully named a
sister-section that already renders the way she wants. Accepting as
requested (Disposition A) is the lowest-risk, fastest path to closing
the comment, and it can be implemented in `HL7/cql-ig` against the
intro pages (likely under the IG's `input/pagecontent/` area, mirroring
the markup used for Appendix B) without touching CQL grammar or
conformance artifacts. Disposition B (template/CSS fix) is a strictly
better long-term answer if the CDS WG has appetite for it, and it is
worth raising with the IG-publisher / template maintainers — but it
should not block this ticket. Disposition C is not recommended: the
defect is real, the fix is cheap, and a working example already exists
in the same document.
