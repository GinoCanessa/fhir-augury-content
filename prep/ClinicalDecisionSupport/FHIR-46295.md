## Summary

The ticket asks the Clinical Decision Support (CDS) work group, in the
context of the Canonical Resource Management Infrastructure (CRMI) IG,
to recommend a normative way to mark FHIR resources that constitute
test data for a particular knowledge artifact (Measure, Library,
PlanDefinition, etc.). At minimum, Bryn Rhodes proposes that every test
resource carry a `Meta.tag` whose code is the canonical reference
(possibly versioned) of the artifact under test. The ticket also asks
whether an extension would be more appropriate — noting that the
existing `cqf-testArtifact` extension already exists for that purpose
but is constrained to `Library` and `Group`, so it cannot be applied to
the heterogeneous mix of Patient/Encounter/Observation/etc. resources
that typically make up a measure test bundle.

The motivation, captured in the cross-referenced Zulip discussion on
`#cql > Problem with resource.id in CMS publications`, is operational:
implementers loading multiple eCQM publications (and multiple versions
of the same Measure) into a single FHIR evaluation server need a
reliable, searchable way to (a) identify the set of resources belonging
to a given test case and (b) bulk-delete that set before reloading an
updated test bundle. Today, with logical ids like `encounter-25` and no
artifact-level marker on test data, neither of those operations is
safe at scale.

The cross-references on this ticket are limited to FHIR element mentions
(`List.subject`, `List.entry`, `List.code`) that come from the
discussion of an alternative `List`-based design, plus the originating
Zulip message. There are no linked or related Jira tickets and no
related GitHub items.

## Details

**Description:**

Consider recommending that all resources related to testing a particular artifact be tagged with the canonical reference to the artifact.

At a minimum, a Tag with a code that is the canonical reference (possibly versioned)

But we could also consider an extension that is expected to be present? There is an extension that identifies the test artifact, (cqf-testArtifact) but it is only useable on Library and Group.

**Comments:**

### rene.spronk (2024-07-04)

A meta tag has the advantage that it is searchable on FHIR servers
(e.g. "GET all example resources associated with a certain test or
measure") and lets a client iterate the result set to perform DELETEs.
Using an extension would not support either search or deletion.

### bryn.rhodes (2024-07-10)

Reporting back from a CRMI call: because `Meta.tag` is only a `Coding`,
there is no way to indicate that the canonical reference carried in the
tag is there *because the resource is test data for that canonical*
versus for some unrelated reason. Asks the community for ideas on how
to disambiguate the semantics of such a tag.

### rene.spronk (2024-07-10)

A `Meta.tag` is a system+code pair. To carry a versioned Measure name
(or versioned canonical URL) as the code, an implementer would need to
mint a dedicated `CodeSystem` whose URL effectively means "this resource
serves as test data for the artifact identified by the following code"
— compliant but roundabout. Stresses that searchability is critical:
without it, one cannot retrieve or bulk-delete a measure's test
resources. Offers an alternative using the `List` resource:
`List.subject` = the Measure (canonical reference), `List.entry[*].item`
= each test resource, `List.code` = `'test data'`; `List.subject` is
searchable. Reiterates that an extension-only solution is unattractive
because it has no search support.

## Related Zulip Discussions

### [cql > Problem with resource.id in CMS publications](https://chat.fhir.org/#narrow/stream/cql/topic/Problem%20with%20resource.id%20in%20CMS%20publications/near/448883267)

The thread is the direct origin of this ticket. René Spronk reports
that across CMS eCQM publications the canonical artifacts (Library,
Measure) reuse the same logical `id` (e.g. `FHIRhelpers`) across
versions, so loading multiple publications into one evaluation engine
causes id collisions. He further notes that test bundles use
human-friendly ids like `encounter-25` / `observation-x` that do not
scale across measures or versions. Discussion converges on two
operational needs: (1) for canonical resources, addressing collisions
via either `id = name+version` or conditional `PUT` / conditional `POST`
(`?url=canonical|version`) — Bryn notes that CRMI already recommends
conditional `POST` in the packaging guidance
(`https://hl7.org/fhir/uv/crmi/packaging.html#fhir-bundle`), while René
prefers conditional `PUT` to guarantee server state on open/public
servers; (2) for **test data** specifically, the need to identify and
bulk-delete the resources belonging to a given measure / publication.
Bryn explicitly endorses tagging the test data ("I like the idea of
tagging the test data") and files this ticket. No firm consensus yet
on the exact tagging mechanism (Meta.tag vs. extension vs. List).

Participants: Bryn Rhodes, René Spronk.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative guidance to the CRMI IG (in the testing/packaging
section, alongside the existing conditional-create guidance at
`packaging.html#fhir-bundle`) that **every resource in a test bundle
for a knowledge artifact SHALL carry a `Meta.tag` whose `system` is a
new CRMI-defined CodeSystem (e.g.
`http://hl7.org/fhir/uv/crmi/CodeSystem/artifact-test-data`) and whose
`code` is the versioned canonical URL of the artifact under test
(e.g. `http://example.org/Measure/ColonCancerScreening|2024.1.0`)**.
Define the new CodeSystem and a companion `SearchParameter` (or rely on
the existing `_tag` search) so clients can issue
`GET [base]/?_tag=<system>|<canonical|version>` to retrieve the full
test set and `DELETE [base]/?_tag=...` to remove it. Cross-link the
guidance to the existing `cqf-testArtifact` extension and explain that
the tag complements (not replaces) that extension for Library/Group,
and provides the only mechanism for the heterogeneous resources
(Patient, Encounter, Observation, …) that make up a measure test bundle.

#### Justification

This is exactly what the reporter asked for and directly addresses the
operational problem documented in the Zulip thread: searchability and
bulk-delete of test data across eCQM publications. `Meta.tag` is
already a first-class, searchable, server-supported feature
(`_tag` is a standard search parameter; bulk-delete via
`?_tag=` is widely implemented). Defining a dedicated CRMI CodeSystem
addresses Bryn's own concern from the 2024-07-10 comment that a bare
`Coding` cannot disambiguate *why* a canonical appears in the tag —
the `system` URL itself encodes the semantic ("this resource is test
data for the canonical in `code`"), so any consumer can recognise the
tag unambiguously. The approach is consistent with existing CRMI
packaging guidance and avoids introducing new resource-level invariants.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) tagging individual resources, recommend
that test data be packaged using a `List` resource per artifact-under-
test: `List.status = current`, `List.code` = a new CRMI-defined code
(e.g. `test-data` from
`http://hl7.org/fhir/uv/crmi/CodeSystem/list-purpose`),
`List.subject` = a `Reference` to the artifact (or a contained
canonical), and `List.entry[*].item` = each test resource. Publish the
`List` alongside the test bundle (and inside the same Bundle when
loading), and update the CRMI testing guidance to describe the
expected search pattern (`GET List?subject=<artifact>&code=test-data`,
then resolve `entry.item`). For the heterogeneous-resource case where
even the `List` enumeration is undesirable, *also* extend
`cqf-testArtifact` (currently usable only on `Library` and `Group`) to
allow it on `Resource` so it can be applied uniformly — making the
`List` the recommended primary mechanism and the extension a fallback.

#### Justification

The `List` approach was raised by René Spronk in both the Zulip thread
and the ticket comments and is reflected in the only cross-references
on the ticket (`List.subject`, `List.entry`, `List.code`). It has two
real advantages over a tag: (1) it gives the test set first-class
identity — it can be versioned, signed, retrieved, and referenced as
a coherent unit, which matters for a "publication" abstraction CRMI
already cares about; (2) it sidesteps the "what does this canonical in
my tag mean?" disambiguation problem entirely, because the relationship
is expressed structurally via `List.subject`. The trade-off is that
bulk-delete becomes a two-step operation (resolve the List, then
delete its entries), and loaders must publish the List alongside the
data — both reasonable asks within a controlled CRMI packaging
workflow.

---

### Disposition C: Decline

#### Proposal

Do not adopt either a normative tagging requirement or a `List`-based
mechanism in CRMI. Instead, document the existing options in a
non-normative "Test Data Management" appendix: implementers MAY use
`Meta.tag` with an implementer-chosen system, MAY use a `List`, or
MAY rely on the `cqf-testArtifact` extension where applicable. Defer
any normative action until at least one connectathon cycle has
exercised the candidate approaches and produced concrete implementer
feedback.

#### Justification

The problem space is real but the design space is unsettled: the
ticket itself surfaces three plausible mechanisms (tag, extension,
List) and the Zulip thread shows that even the adjacent question
(conditional `POST` vs. conditional `PUT` for canonicals) is
contested. Locking in a normative recommendation in CRMI before
implementer experience has converged risks publishing guidance that
test-tooling vendors will quietly ignore. CRMI packaging already
addresses canonical-resource collisions via conditional create
(`packaging.html#fhir-bundle`); test-resource grouping is adjacent
but separable, and a non-normative appendix preserves implementer
freedom while still raising visibility.

---

### Recommendation

**Recommended disposition:** A

The reporter is the CRMI editor, the request is narrowly scoped, and
the Zulip discussion converged explicitly on tagging ("I like the
idea of tagging the test data"). Disposition A as written addresses
Bryn's own 2024-07-10 concern about Coding-level disambiguation by
defining a dedicated CRMI `CodeSystem` whose `system` URL carries the
"this is test data" semantics, leaving the `code` free to identify the
versioned canonical. It builds on a feature (`Meta.tag` + `_tag`
search) that every conformant FHIR server already supports, which
directly unlocks the operational use case driving the ticket — bulk
retrieval and deletion of a measure's test set across eCQM
publications. Disposition B (List) is the better answer if CRMI wants
to model "test publications" as first-class artifacts, and the WG
should consider it as a follow-up; but it is heavier-weight, requires
loaders to publish and maintain an additional resource, and does not
remove the need to label individual resources when they are loaded
outside their originating Bundle. Disposition C under-serves a
concrete, already-painful implementer workflow and risks the CRMI IG
shipping without guidance on a problem its own community has flagged.
