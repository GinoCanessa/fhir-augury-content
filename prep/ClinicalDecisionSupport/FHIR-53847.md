## Summary

The reporter notes that the CDS Hooks (FHIR) specification uses
"non-standard" conformance language — specifically RFC 2119-style terms
such as "MUST" and "MUST NOT" — which does not align with HL7's
official guidance on conformance language. The ticket requests that
every occurrence of such non-HL7 conformance terminology be removed
from the specification and replaced with the HL7-sanctioned terms:
SHALL / SHALL NOT, SHOULD / SHOULD NOT, and MAY. The change is
editorial in nature (terminology substitution) but applies wherever
conformance statements are made in the IG, including narrative text,
profile descriptions, and any normative requirements. The ticket has
been resolved as **Persuasive**, indicating the work group accepts the
underlying request; the resolution states the specification will be
updated so all conformance language matches HL7 guidance. No
cross-referenced Jira tickets, Zulip discussions, or GitHub items
surfaced from the FHIR Augury index for this ticket.

## Details

**Description:**

> The use of 'non standard' conformance language for an HL7 IG (e.g.,
> "MUST", "MUST NOT", etc.) is problematic and does not align with
> standard HL7 guidance on the use of Conformance Language.
>
> Request the removal of any conformance language outside of what HL7
> officially recognizes. The incorrect conformance language should be
> replaced by the corresponding HL7:
>
> - SHALL, SHALL NOT
> - SHOULD, SHOULD NOT
> - MAY

**Resolution:** Persuasive

**Resolution Description:**

> We will update the specification and ensure that all conformance
> language is aligned with HL7 guidance.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Perform a global editorial pass across the CDS Hooks (FHIR) IG and
replace every occurrence of RFC 2119 conformance verbs with the
HL7-sanctioned equivalents:

- `MUST` → `SHALL`
- `MUST NOT` → `SHALL NOT`
- `SHOULD` / `SHOULD NOT` → unchanged (already HL7-sanctioned)
- `REQUIRED` (used as a normative keyword) → `SHALL`
- `RECOMMENDED` (used as a normative keyword) → `SHOULD`
- `OPTIONAL` (used as a normative keyword) → `MAY`
- `MAY` → unchanged

Scope the pass to all narrative pages, profile descriptions, hook
definitions, the security/privacy chapter, and any normative
appendices. Update the IG's introduction/conformance section to state
explicitly that the IG follows HL7 conformance-language guidance and
to remove any RFC 2119 boilerplate. Verify that the substitution does
not change the semantic strength of any requirement (e.g., RFC 2119
"MUST" → HL7 "SHALL" preserves "absolute requirement"; "SHOULD"
maps cleanly; "MAY" maps cleanly). Run the IG publisher and confirm
no QA warnings reference the previous RFC 2119 terms.

#### Justification

This is exactly what the reporter asked for, and the work group has
already marked the resolution as **Persuasive**, so accepting the
request as written is the path of least friction. HL7 publishing
policy expects HL7-balloted artifacts to use SHALL/SHOULD/MAY, and
mixing in RFC 2119 vocabulary creates ambiguity for implementers and
reviewers who must reason about which keyword set governs a given
sentence. The change is purely editorial, has no on-the-wire impact,
and does not alter any conformance obligation when the substitutions
are done as a one-to-one semantic mapping.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the substitution above, **but** add a short, explicit
"Conformance Language" subsection to the IG's introduction that:

1. States the IG uses HL7 conformance keywords (SHALL, SHALL NOT,
   SHOULD, SHOULD NOT, MAY) and links to the HL7 governance page that
   defines them.
2. Provides a one-time mapping table from RFC 2119 keywords to HL7
   keywords for readers (especially CDS Hooks implementers coming from
   the original [non-HL7] CDS Hooks spec, which uses RFC 2119) so the
   transition is unambiguous.
3. Calls out that any lowercase use of "must", "should", "may" in
   the narrative is non-normative.

Then perform the same global substitution as Disposition A.

#### Justification

CDS Hooks originated outside HL7 and the broader CDS Hooks community
is accustomed to RFC 2119 terminology. A bare substitution (Disposition
A) silently changes the vocabulary and may confuse implementers
cross-referencing the upstream CDS Hooks spec with the HL7 (FHIR) IG.
Adding an explicit conformance-language section and mapping table costs
very little, makes the rationale visible, and reduces the chance that
future contributors reintroduce RFC 2119 language by reflex. It also
creates a stable anchor that other CDS Hooks tickets can reference if
similar wording slips back in.

---

### Disposition C: Decline

#### Proposal

Close the ticket without making the substitution, on the grounds that
RFC 2119 keywords are widely understood, semantically equivalent to
the HL7 set for the requirements they express, and the upstream
(non-HL7) CDS Hooks specification uses RFC 2119 — keeping the IG
aligned with the upstream vocabulary aids cross-reading.

#### Justification

This option is defensible only on a "consistency with the upstream
specification" argument. It is contradicted by HL7 publishing policy,
which expects HL7-balloted artifacts to use SHALL/SHOULD/MAY, and by
the work group's own resolution (Persuasive). Listed for completeness.

---

### Recommendation

**Recommended disposition:** B

Disposition B delivers the editorial cleanup the reporter asked for
(and that the work group has already accepted as Persuasive) while
addressing the most plausible source of future regressions: CDS Hooks
implementers and contributors arriving from the upstream RFC 2119
world. The added "Conformance Language" subsection plus a one-time
mapping table costs almost nothing in publication effort, gives
reviewers an explicit anchor for catching reintroductions of MUST /
MUST NOT in future PRs, and avoids the silent-vocabulary-swap surprise
that a bare Disposition A pass would create. Disposition C is not
viable given HL7 policy and the existing Persuasive resolution.
