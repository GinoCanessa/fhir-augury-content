## Summary

The reporter is asking the Clinical Decision Support work group to clarify the
value and purpose of the "equivalent contains" capability inside a CQL retrieve
expression. The ticket text is terse — "Equivalent contains within a retrieve
- it is unclear what this change enhances." — and reads as a ballot/review
question against the most recent CQL-on-FHIR change in this area rather than a
proposal of its own.

The change being questioned is the one introduced by
[FHIR-31392](https://jira.hl7.org/browse/FHIR-31392) ("Support contains as a
terminology comparator", Applied / Persuasive on 2025-07-09, committed to
HL7/cql at f1a2d0d). That change added a new **terminology `contains`**
comparator to the retrieve syntax — the inverse of the existing terminology
`in` membership — so that retrieves whose terminology element is multi-valued
(e.g., `Condition.category`) can be filtered against a single direct-reference
code or concept using equivalence semantics, e.g.
`[Condition: category ~ "Condition Problem List Item"]`. Because that
comparator is defined to use terminology *equivalent* (`~`) rather than strict
equality, FHIR-52513 is effectively asking the WG to spell out, in the spec,
why "equivalent contains" is needed (vs. plain `contains`, vs. expanding the
existing `in` overloads, vs. the long-hand `where` form), and what the user's
intuition should be when reading or writing it.

No automated cross-references were found between FHIR-52513 and other
artifacts; the linkage to FHIR-31392 was discovered via keyword search and is
recorded under "Related Jira Tickets" below.

## Related Zulip Discussions

### [cql > Fancy Translations of \[MedicationRequest: "MyValueSet"\]](https://chat.fhir.org/#narrow/stream/cql/topic/Fancy%20Translations%20of%20%5BMedicationRequest%3A%20%22MyValueSet%22%5D/near/457206958)

Chris Moesel observed that a plain CQL retrieve such as
`[MedicationRequest: "MyValueSet"]` can be silently rewritten by the
translator into a union that joins to `[Medication]` (an unbounded query in
some systems), and asked whether there is a way to opt out. Bryn Rhodes
clarified that the translator only applies that rewrite when the default
terminology code path is a choice type and that an *explicit* terminology path
in the retrieve (e.g., `[MedicationRequest: medication in "MyValueSet"]`)
suppresses or steers the rewrite by collapsing to the choice that resolves
against the supplied operator. The thread illustrates why authors care about
the precise semantics and choice-type behavior of terminology operators in the
retrieve — exactly the surface area FHIR-52513 wants the spec to clarify.
Participants: Chris Moesel, Bryn Rhodes, Michael Riley.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a clarifying explanatory paragraph (and a worked example) to the CQL-on-FHIR
retrieve section that introduced the terminology `contains` comparator under
FHIR-31392. The clarification should:

1. State explicitly that the new comparator implements a **terminology
   contains** — i.e., "the singular terminology argument is *equivalent* (`~`)
   to at least one of the codes/concepts in the multi-cardinality terminology
   element of the retrieved resource" — and is the inverse of the existing
   terminology `in` overload that supports multi-cardinality left-hand sides.
2. Show paired worked examples:
   - `[Condition: category in "Condition Category Codes"]` (existing
     multi-cardinality `in`), and
   - `[Condition: category ~ "Condition Problem List Item"]` (new singular
     equivalent contains),
   together with their long-hand `where` forms so authors can see the
   equivalence.
3. Call out that the operator is **terminology** equivalence (codeable-concept
   / coding equivalence using `~`), *not* string equality, and is therefore
   sensitive to translations and code-system rules — paralleling the existing
   wording around `in`.

This is a documentation / spec-prose change to CQL-on-FHIR; no syntactic or
semantic change to the retrieve grammar is required.

#### Justification

The ticket is a clarity/ballot-readability complaint, not a request to change
behavior. FHIR-31392's resolution text already names the construct ("support
contains as a terminology membership operator") and asserts it must use
terminology comparison rather than equality, but a reader coming to the spec
cold — as Hans did — does not get that explanation in the normative section.
Mirroring the existing `in` exposition (which the WG already approved) keeps
the prose consistent with how multi-cardinality terminology targets are
introduced today, satisfies the reporter's literal ask, and avoids relitigating
a recently-applied Persuasive resolution.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-52513 as the trigger for a small **focused review of the retrieve's
terminology-operator surface** rather than a one-paragraph addition. Produce
(in the CQL-on-FHIR spec):

- A single normative subsection titled "Terminology Comparators in Retrieves"
  that enumerates `=`, `~` (equivalent), `in`, and `contains`, with one row
  per comparator describing left-hand cardinality, right-hand kind
  (direct-reference code / concept / value set / code system), evaluation
  semantics (equality vs. equivalence vs. membership), and the long-hand `where`
  form each desugars to.
- An explicit non-normative note that the new `contains` comparator is
  intentionally defined with terminology-equivalence semantics so that it
  symmetrises the multi-cardinality `in` overload introduced earlier, and a
  forward pointer to the choice-type / fancy-translation behavior discussed on
  the `cql` Zulip stream so that authors can predict translator behavior.
- A short conformance/examples block that runs a single Condition example
  through all four comparators side-by-side.

#### Justification

The reporter's confusion almost certainly stems from the fact that the
retrieve's terminology operators have grown organically, with each addition
documented in its own paragraph at the point of change. A consolidated
operator table would make the *purpose* of each addition self-evident on first
read and would head off future "what does this enhance?" tickets when further
operators (e.g., subsumption-style comparators raised on Zulip in
"CQL equivalent of ELM SubsumedBy Subsumes") are added. The cost is modestly
larger than Disposition A, but the artefact is durable and it directly
addresses the underlying readability problem rather than only patching the
specific paragraph the reporter happened to land on.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-52513 as **Not Persuasive** on the grounds that the change being
questioned (terminology `contains` via FHIR-31392) is already adequately
described by the resolution text and the spec change committed at
[HL7/cql@f1a2d0d](https://github.com/HL7/cql/commit/f1a2d0dd1ef77625284b21b701b94ecb7d9487fa),
and that the rationale ("symmetrise the multi-cardinality `in` overload for
the direct-reference singular case") is recoverable from the FHIR-31392 ticket
record without additional spec prose.

#### Justification

If the WG considers the existing exposition self-sufficient — i.e., a CQL
author who understands `in` and `~` should infer `contains` by analogy — then
adding clarifying prose is busywork that risks de-syncing from the implemented
behavior. Declining keeps the spec terse and avoids re-opening a recently
Applied change. The defensible rationale category here is "request is for
clarification of an already-resolved item; the existing resolution and commit
record provide sufficient explanation."

---

### Recommendation

**Recommended disposition:** A

The ticket's text is short and unambiguous: a knowledgeable reviewer (Hans
Buitendijk) read the new "equivalent contains" capability in the retrieve and
could not tell *why* it was added or what it improves. That is precisely the
kind of feedback ballot review exists to surface, and it is satisfied cheaply
by a paragraph and a worked example that mirror how the WG already explains
the multi-cardinality `in` overload. Disposition B is attractive long-term but
is larger than what this ticket asks for and risks scope-creeping a simple
clarification into a refactor of an entire spec section; if the WG wants the
consolidated operator table, it should be tracked as a separate vNext editorial
ticket. Disposition C is hard to defend given that a senior implementer
explicitly asked the question — declining a clarification request for being
"obvious" tends to generate the same ticket again from the next reviewer.
Accept as requested, scope the edit narrowly to the retrieve subsection
introduced under FHIR-31392, and reuse the example pair already in
FHIR-31392's description as the worked example.
