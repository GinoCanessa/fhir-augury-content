## Summary

The reporter (Mitra Rocca) has flagged a typo / wording issue on the
Introduction page of the **Potential Drug-Drug Interaction (PDDI) (FHIR)**
implementation guide. The ticket itself carries no description text; the
substantive content is in the reporter's own comment, which proposes
replacement wording for the IG's discussion of CDS Hooks prefetch
templates: that prefetch data may span many resources and time periods,

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the editorial correction to the **Introduction** page of the
*Potential Drug-Drug Interaction (PDDI) (FHIR)* IG, using the wording
proposed by the reporter (or a close paraphrase that fits the
surrounding paragraph):

> "Depending on the patient and service, prefetch data may encompass a
> variety of resources captured during various time periods, so it is
> crucial that implementors and clinicians refine prefetch template
> parameters to obtain only data that is clinically relevant."

Then locate the PR/commit in the PDDI IG repository that already
contains this change (per boycerd's "This was fixed" comment), link it
on the Jira ticket, and mark the ticket **Applied** as Bryn Rhodes
offered to do.

#### Justification

This is a Technical Correction against IG narrative — the lowest-risk
class of change. The reporter has supplied the replacement wording, a
co-editor has already stated the change was made, and the only
remaining gate is providing the GitHub provenance Bryn requested. The
substituted wording is clearer about prefetch scope (resources × time
window) and reinforces guidance that is consistent with CDS Hooks
prefetch best practice, so it is editorially defensible on its own
merits independent of the original typo.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the ticket as a prompt for a slightly broader editorial pass on
the PDDI Introduction's prefetch discussion rather than a single
sentence swap. Specifically:

1. Apply the reporter's wording as in Disposition A.
2. Add (or hyperlink to) a one-line cross-reference to the CDS Hooks
   prefetch documentation so readers can follow the underlying
   normative guidance.
3. Confirm that the surrounding paragraphs on the Introduction page
   consistently use "prefetch template" (vs. ad-hoc terms like
   "prefetch query" or "prefetch request") and normalize if not.

Bundle these into the same Applied PR/commit that Disposition A is
trying to surface.

#### Justification

The reporter's comment frames the issue not just as a typo but as a
clarity concern about how implementers interpret prefetch scope. A
narrowly-scoped wording swap satisfies the letter of the ticket but
leaves the surrounding context unaligned. Doing a small consistency
pass at the same time costs little (the file is already being touched)
and reduces the chance of a second editorial ticket against the same
section later. The trade-off vs. A is a marginally larger diff for
review.

---

### Disposition C: Decline (Close as Already Applied without further change)

#### Proposal

Close the ticket as **Persuasive – Already Applied** on the strength of
boycerd's "This was fixed" comment, without waiting for the GitHub
reference Bryn asked for. Document in the resolution note that the
correction is believed to be in the published IG and that any
subsequent discrepancy should be raised as a new ticket against the
specific built page.

#### Justification

This is defensible only if the workgroup is confident the change is
already live on the rendered IG and is willing to accept the lack of a
linked PR. It avoids leaving a stale-looking ticket open against an IG
that is otherwise quiet. The risk is exactly what Bryn flagged: without
a PR/commit link, Augury / future reviewers cannot verify the change
actually landed, and a regression in a future republish would be
invisible. This is the weakest option and is included mainly to make
the trade-off explicit.

---

### Recommendation

**Recommended disposition:** A

This is a Medium-priority Technical Correction against IG narrative on
a small, low-traffic specification, with the replacement wording
already supplied by the reporter and a co-editor asserting the change
has been made. The only blocker to closing the ticket cleanly is the
provenance request Bryn Rhodes made in April 2025: identify the PR or
commit in the PDDI repository that carries the wording change, attach
the link to the Jira comment thread, and mark the ticket Applied.
Disposition B is a reasonable upsell if an editor is already in the
file, but it shouldn't gate closing this ticket. Disposition C should
be avoided — closing without the GitHub provenance ignores Bryn's
stated condition and leaves the audit trail incomplete.
