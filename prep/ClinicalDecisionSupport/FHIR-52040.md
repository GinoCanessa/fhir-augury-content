## Summary

FHIR-52040 asks the Clinical Decision Support work group to add support
for an order-preserving variant of the `combine` operation in the
Clinical Quality Language (CQL) specification, mirroring the change
already adopted in FHIRPath via [FHIR-49155](https://jira.hl7.org/browse/FHIR-49155).
That FHIRPath ticket was resolved Persuasive with the agreement to add
a `preserveOrder` boolean parameter to `combine` (defaulting to false)
that, when true, preserves the order of the elements in the input
collections — semantically the "append" operation from list algebras —
and that change has been applied to the FHIRPath spec
(see [HL7/FHIRPath#19](https://github.com/HL7/FHIRPath/pull/19)).
This ticket is the CQL-side follow-on so that CQL's translation/runtime
semantics for `combine` (and any `Combine`-style ELM operator) stay
aligned with the upstream FHIRPath behaviour.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CQL specification to add a `preserveOrder` boolean parameter
to the `Combine` operator (and the corresponding FHIRPath-flavoured
`combine` function exposed through CQL), matching the FHIRPath
resolution from FHIR-49155 exactly:

- Signature: `Combine(source List<T>, other List<T>, preserveOrder Boolean) returns List<T>`
  (with the existing two-argument form retained as `preserveOrder = false`).
- Semantics when `preserveOrder = false`: unchanged from today —
  duplicates are kept, order is unspecified.
- Semantics when `preserveOrder = true`: the result is the elements of
  `source` in their original order followed by the elements of `other`
  in their original order; duplicates are preserved (i.e., true
  list-algebra append).
- Mirror the change in the ELM schema (add the optional `preserveOrder`
  child to the `Combine` operator), in the CQL-to-ELM translator, and
  in the reference engine; add the example provided in FHIRPath-19 to
  the CQL spec to illustrate the behaviour.

#### Justification

This is the minimal, faithful CQL counterpart of the already-Applied
FHIRPath change. CQL's `Combine` is intentionally aligned with
FHIRPath's `combine`; diverging here would leave authors with no way to
express order-preserving concatenation in CQL even though the
underlying FHIRPath layer supports it, and it would create a confusing
semantic split between CQL-on-FHIRPath and pure FHIRPath expressions.
The default (`false`) preserves backward compatibility for every
existing artifact.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of overloading `Combine` with a `preserveOrder` flag, introduce
a dedicated `Append(source List<T>, other List<T>) returns List<T>`
operator in CQL that always preserves order and duplicates, and define
it as syntactic sugar for `Combine(source, other, true)` once the
FHIRPath-aligned form is in place. The two-argument `Combine` keeps its
current order-unspecified semantics.

#### Justification

A named `Append` operator is more discoverable for CQL authors (who
generally prefer named operators over boolean flags), reads naturally
in clinical logic ("append the new observations to the existing list"),
and matches the list-algebra terminology FHIRPath itself acknowledges
in its resolution text. The trade-off is that CQL's surface no longer
maps 1:1 to FHIRPath's parameter list, which the translator would have
to bridge — but the translator already performs this kind of mapping
for several operators (e.g., `Coalesce`, `First`/`Last`).

---

### Disposition C: Decline

#### Proposal

Decline FHIR-52040 on the grounds that authors who need an
order-preserving concatenation can already express it in CQL using
`source union other` with explicit ordering, or via list-comprehension
patterns (`(source) | (other)` followed by an `OrderBy`, or a
`flatten({source, other})` construct). Recommend documenting the
idiom in the CQL spec rather than introducing a new operator surface.

#### Justification

Adding a parameter or a new operator carries non-trivial cost: ELM
schema bump, translator changes, reference-engine changes, conformance
tests, and downstream tooling updates across the CDS ecosystem. If the
underlying need is rare in clinical-logic authoring, documenting the
existing-idiom workaround is cheaper and avoids fragmenting `Combine`'s
semantics with a flag whose default differs from what some authors
would expect.

---

### Recommendation

**Recommended disposition:** A

FHIR-49155 has already been resolved Persuasive and Applied in the
FHIRPath specification, which CQL composes over directly. Refusing or
diverging from that resolution in CQL would either leave a real
expressiveness gap (Disposition C — `union` is set-based and the
suggested workarounds either drop duplicates or impose an ordering the
author may not want) or introduce a CQL-only operator surface that
must then be reconciled with FHIRPath at translation time (Disposition
B). The cleanest, lowest-risk outcome is to mirror the FHIRPath
resolution exactly: add the `preserveOrder` boolean to `Combine` with
a `false` default so every existing artifact behaves identically, and
document the order-preserving form with the same example FHIRPath
adopted. A named `Append` alias can be revisited as a follow-on if
authoring feedback shows demand. Implementation will touch `HL7/cql`
(spec, ELM schema, translator, engine, tests); a `ticket-plan` run
against that repo should be commissioned once the work group
disposes the ticket.
