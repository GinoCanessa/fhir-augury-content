## Summary

The ticket asks the CDS Hooks specification to expand the allow-list of
constructs that a *valid prefetch template* may use so that it explicitly
includes the `_maxresults` search parameter alongside `_count`. The
reporter notes that `_count` only governs the page size of the returned
`Bundle`, whereas `_maxresults` is intended to bound the total number of
matching resources the server should return — a different (and, for
prefetch, arguably more useful) limit. Without `_maxresults` in the
allow-list, a CDS Service author cannot legitimately constrain the
total result-set size in a prefetch template even though the underlying
FHIR query mechanics support it. The ticket is already marked
**Persuasive** and the workgroup's resolution note states "This is a
reasonable change and we will add a callout for the _maxresults
parameter."

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket; the only Zulip/Jira/GitHub hit for either the
ticket key or the phrase "prefetch _maxresults" is the ticket itself.

## Details

**Description:**

> valid prefetch templates MUST only make use of:
> - *instance* level [read](https://www.hl7.org/fhir/http.html#read)
>   interactions (for resources with known ids such as `Patient`,
>   `Practitioner`, or `Encounter`)
> - *type* level [search](https://www.hl7.org/fhir/http.html#search)
>   interactions; e.g. `patient={{context.patientId}}`
> - Resource references (e.g. `patient={{context.patientId}}`)
> - *token* search parameters using equality (e.g. `code=4548-4`) and
>   optionally the `:in` modifier (no other modifiers for token
>   parameters)
> - *date* search parameters on `date`, `dateTime`, `instant`, or
>   `Period` types only, and using only the prefixes `eq`, `lt`, `gt`,
>   `ge`, `le`
> - the `_count` parameter to limit the number of results returned
> - the `_sort` parameter to allow for *most recent* and *first*
>   queries

This list should include `_maxresults` as well, as it is intended to
signal the number of resources that should be returned, while `_count`
only lists the size of the Bundle.

**Resolution Description (already recorded by WG):**

> This is a reasonable change and we will add a callout for the
> _maxresults parameter.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CDS Hooks specification's *Prefetch Template* prose so that
the allow-list of permitted constructs in a valid prefetch template
includes a new bullet item:

- the `_maxresults` parameter, to limit the total number of resources
  the server returns for the query (distinct from `_count`, which only
  controls Bundle page size).

In the same callout, briefly contrast `_maxresults` against `_count`
(page size of an individual Bundle) and `_sort`/`_count`-based
"_most recent_ / _first_" patterns, and clarify that `_maxresults`
applies only to type-level search interactions (consistent with the
existing rules for `_count` and `_sort`). No structural / schema
changes are needed — this is a normative-prose update to the prefetch
template section, matching the WG's already-recorded resolution
("we will add a callout for the _maxresults parameter").

#### Justification

This matches the existing WG resolution of **Persuasive** verbatim and
keeps the prefetch allow-list in step with the underlying FHIR search
mechanics. `_count` and `_maxresults` answer different questions —
"how big is each page?" vs. "how many matches in total?" — and prefetch
is precisely the place where a CDS Service author wants to cap total
result-set size to keep payloads bounded. The change is additive,
backward-compatible (a server that ignored `_maxresults` previously
will continue to behave the same), and consistent with the existing
treatment of `_count` and `_sort` in the same bullet list.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than enumerating `_maxresults` as a single new bullet, restructure
the prefetch allow-list to call out **result-limiting parameters** as a
group: `_count` (page size), `_maxresults` (total results), and `_sort`
(ordering for "most recent / first" patterns), with a short paragraph
explaining when each is appropriate and noting that other
result-shaping parameters (e.g., `_summary`, `_elements`, `_include`,
`_revinclude`, `_total`) remain **out of scope** for prefetch templates
unless explicitly added in a future ticket. Also state explicitly that
`_maxresults` is only meaningful at the type-search level, mirroring
the constraints already placed on `_count` and `_sort`.

#### Justification

The current bullet list grew organically and does not make the
distinction between "what you can filter on" (token/date) and "what
you can do to the result set" (count/sort) particularly clear. Adding
`_maxresults` is a natural opportunity to tighten that section so the
next person who asks "what about `_summary`?" or "what about
`_include`?" finds the answer in the spec rather than as another Jira
ticket. The trade-off vs. Disposition A is slightly more editorial
work and a marginally larger diff, against materially better
long-term clarity for implementers and reviewers.

---

### Disposition C: Decline

#### Proposal

Reject the request and leave the prefetch template allow-list
unchanged. Optionally, add a short note in the prefetch section
clarifying that any parameter not in the allow-list (including
`_maxresults`) is *not* permitted in a valid prefetch template and
that authors who need a hard cap on result-set size should rely on
`_count` plus appropriate query design.

#### Justification

`_maxresults` is not a FHIR-defined search parameter in the FHIR core
specification — its semantics and server support are uneven across
implementations. Standardising it in CDS Hooks could create
interoperability issues if CDS Services emit prefetch templates
containing `_maxresults` against servers that do not implement it,
producing either errors or silently ignored caps. Declining keeps the
prefetch surface small, well-defined, and aligned to FHIR core, and
defers the question until `_maxresults` is itself standardised at the
FHIR core level.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already triaged this as **Persuasive** and recorded a
resolution note committing to "add a callout for the _maxresults
parameter" — Disposition A is the literal execution of that decision
and avoids re-litigating a question the WG has already answered. The
change is a small, additive, backward-compatible prose edit to the
prefetch template section of the CDS Hooks (FHIR) specification; it
materially helps prefetch authors who today have to choose between
oversized payloads and a non-conformant template. Disposition B is
attractive editorially but expands scope beyond what the ticket asked
for and the WG resolved; it would be a sensible follow-up ticket
rather than a reason to delay this one. Disposition C contradicts the
recorded resolution and the implementer need expressed in the ticket;
the interoperability concern it raises is real but is better addressed
by the callout itself (e.g., noting server-support expectations)
than by withholding the change.
