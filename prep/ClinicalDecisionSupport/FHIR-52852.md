## Summary

This Change Request asks the Clinical Guidelines (CPG-on-FHIR) IG to add
first-class support for a "metric type" classification on the `CPGMetric`
profile (a profile of `Measure`) and to carry that classification through to
the corresponding `CPGMetricReport` profile (a profile of `MeasureReport`).
Concretely, the request defines a small CodeSystem/ValueSet of CPG metric
types (`goal`, `process-event`, `process-request`, `adherence`), binds it
extensibly alongside the existing measure-type ValueSet on `Measure.type`,
and creates an extension on `CPGMetricReport` so the metric type flows from
the definition into the report. The resolution (Persuasive with
Modification) expands the scope: in addition to the original asks, it adds
group-level support — an extension on `Measure.group.type` plus extensions
on the group level of both `CPGMetric` and `CPGMetricReport` for the
computable elements `scoring`, `compositeScoring`, `improvementNotation`
(and `type` on the report side) — and documents a SHALL that the metric
type be brought forward from `CPGMetric` to `CPGMetricReport` when a report
is produced. There were no Jira, Zulip, or GitHub cross-references in the
augury index for this ticket beyond the FHIR-element mentions
(`Measure.type`, `Measure.group.type`, `MeasureReport.group`) extracted
from the description itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution exactly as written:

1. Define `CodeSystem/cpg-metric-type` and the matching
   `ValueSet/cpg-metric-type` with codes `goal`, `process-event`,
   `process-request`, `adherence` (displays as in the ticket).
2. On the `CPGMetric` profile (profile of `Measure`):
   - Add an additional **extensible** binding on `Measure.type` referencing
     the new `cpg-metric-type` ValueSet alongside the existing measure-type
     ValueSet.
   - Add an extension on `Measure.group.type` to allow specifying metric
     type at the group level.
   - Add group-level extensions exposing the computational elements
     `scoring`, `compositeScoring`, and `improvementNotation` (so a single
     CPGMetric can describe heterogeneous group-level computations).
3. On the `CPGMetricReport` profile (profile of `MeasureReport`):
   - Add an extension expressing the metric `type`, allowed at both root
     and `MeasureReport.group`.
   - Add group-level extensions for `scoring`, `compositeScoring`,
     `improvementNotation`, and `type`.
4. Add narrative + a SHALL-level conformance statement that the metric
   type from the `CPGMetric` definition is carried forward into the
   produced `CPGMetricReport`.

#### Justification

This is the change CQI has already vetted and approved 15-0-0 (per the
2025-10-24 motion), which is the workgroup that owns `Measure` and
`MeasureReport`. The use of an *additional* extensible binding on
`Measure.type` is the standard FHIR pattern for layering domain-specific
codes on top of a base ValueSet without breaking existing implementers,
and group-level extensions for `scoring`/`compositeScoring`/
`improvementNotation` mirror what already exists at the resource root —
making the CPG profiles internally consistent and lining them up with how
implementers already think about heterogeneous-group measures. Promoting
"bring forward the metric type" to a SHALL keeps the CPGMetric →
CPGMetricReport contract checkable by validators.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substance of the resolution but tighten the scope and the
binding strength:

- Keep the new `cpg-metric-type` CodeSystem/ValueSet as proposed.
- On `CPGMetric.Measure.type`, bind the new ValueSet with strength
  **preferred** (rather than extensible) layered alongside the existing
  measure-type ValueSet, so CPG-aware implementers are guided to the new
  codes but non-CPG callers writing through the same profile aren't forced
  to add a CPG-specific code when their existing measure-type code already
  fits.
- Defer the group-level computational extensions (`scoring`,
  `compositeScoring`, `improvementNotation`) to a follow-up ticket scoped
  to "heterogeneous-group computation in CPG measures", and ship only the
  metric-type extension at the group level in this round.
- Keep the SHALL on bringing the metric type forward from `CPGMetric` to
  `CPGMetricReport`.

#### Justification

The original ticket asked for the metric-type classification only; the
group-level computable-element extensions were added during resolution and
broaden the surface area substantially (four new extensions on the report
side, three on the definition side, all touching authoring + report
generation). Splitting that work out lets the metric-type change ship now
with a clean review, while the group-level computational story gets its
own ticket where implementers can weigh in on whether they actually
populate per-group `scoring`/`compositeScoring`/`improvementNotation`
today. Choosing **preferred** over **extensible** for the metric-type
binding also avoids the "must come from one of these two ValueSets" trap
when the existing measure-type code is already a perfectly good fit.

---

### Disposition C: Decline

#### Proposal

Do not adopt this change. Recommend that CPG implementations expressing a
"metric type" use the existing `Measure.type` element with a
locally-defined CodeSystem (or a CPG-published companion ValueSet
referenced via guidance, not via a profile binding), and that
classification-on-report be handled by re-reading the linked `Measure`
rather than by adding a new extension on `MeasureReport`.

#### Justification

`Measure.type` already exists with a documented binding, and
`MeasureReport.measure` already lets a consumer dereference the source
`Measure` to recover any classification. Adding a second bound ValueSet
plus a parallel extension on `MeasureReport` introduces two ways to say
the same thing, and the SHALL "bring forward the metric type" obligation
risks divergence between the report extension and the underlying
`Measure.type` if the report is regenerated from a re-versioned
definition.

---

### Recommendation

**Recommended disposition:** A

CQI — the workgroup that actually owns `Measure` and `MeasureReport` — has
already approved this usage 15-0-0, and the ticket carries the
`Ready-For-Vote` label, so the CDS workgroup is being asked to ratify
something the upstream owners have already blessed. Disposition C is hard
to defend after that vote. Disposition B's instinct to split the
group-level computable-element work is reasonable, but those extensions
are tightly coupled to the same "describe heterogeneous metrics
per-group" goal the metric-type extension serves, and shipping them
together avoids a second round of profile churn for CPG implementers.
Accept the resolution as written; revisit binding strength only if CDS
review surfaces concrete implementer pushback.
