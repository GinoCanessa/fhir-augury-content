## Summary

Reporter Emma Jones requests that the CDS Hooks Library hook context be
extended so callers can supply an `EpisodeOfCare` identifier in addition
to (or as an alternative to) the existing `Encounter.id` context. Per
Isaac Vetter's 2025-03-05 comment, the request is scoped specifically to
the **`order-sign`** hook in the CDS Hooks Library (originally raised
when the ticket was framed against `patient-view`). The existing
context wording the reporter is targeting is: "The FHIR Encounter.id of
the current encounter in context." The proposal would add an optional
EpisodeOfCare reference so a hook caller working at an episode level
(e.g., longitudinal/PHR scenarios) can convey that context to the CDS
service.

The cross-reference index returns no related Jira tickets, Zulip
threads, or GitHub items — only a structural mention back to the FHIR
`Encounter.id` element this ticket cites.

## Proposed Dispositions

*Note: no GitHub cross-references were surfaced for this ticket, so the
following dispositions are framed against the CDS Hooks Library
specification authored by the CDS WG (the actual repo and authoring
root would be confirmed via a `repo-analysis` briefing if/when a
disposition advances to implementation).*

### Disposition A: Accept As Requested

#### Proposal

Add an **optional** `episodeOfCare` context parameter to the
`order-sign` hook definition in the CDS Hooks Library. Specifically:

- In the `order-sign` hook's Context table, add a new row:
  - **Field:** `episodeOfCare`
  - **Optionality:** OPTIONAL
  - **Type:** `string` (id of an EpisodeOfCare resource)
  - **Description:** "The id of the EpisodeOfCare resource the workflow
    is scoped to, when applicable."
- Keep `encounter` as-is (still OPTIONAL/REQUIRED per current
  definition).
- Update the example context JSON to show how `episodeOfCare` and
  `encounter` may co-exist or be used independently.
- Add prefetch examples that resolve `EpisodeOfCare/{{context.episodeOfCare}}`
  for services that need the resource body.

#### Justification

This directly satisfies the reporter's ask and is additive/optional, so
it does not break existing callers. It accommodates longitudinal /
episode-scoped workflows (e.g., disease-management programs, post-acute
care, PHR-driven `order-sign`) where the clinically relevant grouping
is the EpisodeOfCare rather than a single Encounter — a scenario Isaac
himself acknowledged in the 2020-04-01 comment. Adding it as OPTIONAL
keeps the bar low for existing implementers while giving advanced
implementers a standardised slot rather than forcing them to invent a
non-standard extension.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new top-level `episodeOfCare` context field. Instead,
clarify in the `order-sign` hook's Context section (and in shared CDS
Hooks Library guidance) that:

1. When an Encounter is supplied via `context.encounter`, CDS services
   that need EpisodeOfCare SHOULD resolve it via `Encounter.episodeOfCare`
   (which is already discoverable, per Bryn Rhodes' note in the
   2020-03-04 comment).
2. Define a recommended prefetch template
   (e.g., `episodeOfCare: "EpisodeOfCare?_id={{context.encounter.episodeOfCare}}"`
   or equivalent chained query) so services have a uniform way to obtain
   it.
3. Defer a dedicated context field until a future version when
   EpisodeOfCare maturity and implementer demand are demonstrated
   (e.g., via a CDS Hooks connectathon track or implementer survey).

#### Justification

Trades a smaller spec change for slightly more implementer work on the
service side. Avoids defining a context field whose value is, in
practice, almost always derivable from `context.encounter` —
addressing Isaac's 2020-03-04 and 2025-06-25 concerns about
EpisodeOfCare maturity and prevalence. It also sidesteps the awkward
edge case Isaac raised (EpisodeOfCare without an Encounter) by leaving
that scenario to be specified later, with evidence, rather than locked
into the `order-sign` hook now. Because guidance/prefetch examples are
non-normative, this option preserves backward compatibility and is
cheap to revisit.

---

### Disposition C: Decline

#### Proposal

Decline the request and close as **Considered — Not Persuasive (Insufficient
Implementer Demand)**, restoring the spirit of the original 2020-04-01
disposition (Rob McClure / Floyd Eisenberg, 19-0-0). Document in the
resolution that:

- EpisodeOfCare is reachable via `Encounter.episodeOfCare` whenever an
  Encounter context is supplied.
- The CDS WG has not seen sufficient `order-sign` implementer demand or
  EpisodeOfCare maturity to justify a new context field.
- The ticket may be reopened if implementer evidence emerges.

#### Justification

Multiple comments over five years (2020-03-04, 2020-04-01, 2025-06-25)
converge on the same underlying concern: low EpisodeOfCare maturity and
no demonstrated implementer demand for an episode-scoped `order-sign`
context. The original WG vote (19-0-0) reached the same conclusion. In
the absence of new evidence (e.g., a documented implementer use case or
connectathon results), adding optional context fields still imposes
spec-surface and conformance-testing cost on every implementer. Bryn
Rhodes' observation that EpisodeOfCare is already discoverable from
Encounter further weakens the case for a dedicated field.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle: it acknowledges the reporter's
underlying need (services may want EpisodeOfCare for `order-sign`
reasoning) without committing the CDS Hooks Library to a new normative
context field whose value is, today, almost always derivable from
`Encounter.episodeOfCare`. Adding non-normative guidance and a
prefetch template gives implementers a uniform pattern now, defers the
maturity question Isaac repeatedly raised, and keeps the option open to
upgrade to Disposition A in a future version if connectathon or
implementer-survey evidence appears. Disposition A is defensible if the
WG concludes the additive risk is negligible and wants to make episode
scoping a first-class concept; Disposition C is defensible but is
effectively the previous WG vote, which has already been reverted once
(lynn_laakso, 2021-06-16) and is unlikely to satisfy the reporter
without giving implementers any pattern to follow.
