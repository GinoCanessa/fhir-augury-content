## Summary

The reporter (BrianAlper) observes that the previous `Dosage` datatype has been
refactored in the current ballot into a family of types — `DosageDetails`,
`Dosage`, `DosageSafety`, and `DosageCondition`. As a knock-on effect of that
refactor, `ActivityDefinition.dosage` is currently typed as `0..* Dosage`. The
reporter argues that, to preserve the previous functional behavior of
`ActivityDefinition.dosage` (which under the prior model carried the full
dosing intent including timing, route, dose, and conditions), the element
should instead be `0..1 DosageDetails`, since `DosageDetails` is the new
container type that aggregates the constituent dosage information. The ticket
is filed as a "Highest" priority Change Request against FHIR Core and
acknowledges that the broader split-Dosage structure may itself attract
substantial ballot feedback; this ticket is narrowly scoped to fixing
`ActivityDefinition.dosage` so it is consistent with that new structure
regardless of the wider outcome.

The cross-reference lookup returned no related Jira tickets, no Zulip threads,
and no GitHub items — only an internal mention of the `ActivityDefinition.dosage`
FHIR element path itself.

## Proposed Dispositions

> Note: Because no GitHub cross-references were surfaced, the dispositions
> below cannot name a specific repo path. They are framed in terms of the
> normative spec change to `ActivityDefinition` and the new `DosageDetails`
> family of datatypes; whichever HL7/fhir authoring path currently defines
> `ActivityDefinition.dosage` is the path that would be edited.

### Disposition A: Accept As Requested

#### Proposal

Change `ActivityDefinition.dosage` from `0..* Dosage` to `0..1 DosageDetails`
in the FHIR Core specification. Concretely:

- Update the `ActivityDefinition` StructureDefinition so the `dosage` element
  has cardinality `0..1` and `type.code = DosageDetails` (replacing the
  current `Dosage`).
- Update `ActivityDefinition`'s element definition / short / comments to
  describe a single, aggregated `DosageDetails` instance (which itself
  composes timing, dose, route, `Dosage`, `DosageSafety`, `DosageCondition`,
  etc.) rather than a list of `Dosage` repetitions.
- Add a backwards-compatibility / migration note explaining that callers who
  previously populated multiple `Dosage` entries should now express the same
  intent as a single `DosageDetails` (with its repeating substructures where
  applicable).
- Sweep sibling resources that retained `0..* Dosage` typing after the
  Dosage refactor (e.g., `MedicationRequest`, `MedicationDispense`,
  `MedicationStatement`, `MedicationAdministration`, `NutritionOrder`-family
  references) to confirm whether the same correction applies; if so, file or
  link companion tickets so the change set is consistent across the
  specification.

#### Justification

The Dosage-family refactor introduced `DosageDetails` precisely as the
aggregating container for the granular `Dosage`, `DosageSafety`, and
`DosageCondition` datatypes. `ActivityDefinition.dosage` is a *definitional*
element that describes the intended dosing for an activity once, not a list
of independently-applied dose orders, so a single `DosageDetails` matches the
modeling intent and preserves the prior functional result. Repeating a
post-refactor `Dosage` (`0..*`) would actually *lose* expressive power
relative to the prior model, because timing/route/dose/safety/conditions were
previously co-located on a single `Dosage` and are now distributed across the
family — only `DosageDetails` reassembles them. Accepting the change as
requested keeps `ActivityDefinition` faithful to the new datatype design and
avoids a silent semantic regression for CDS implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Change `ActivityDefinition.dosage` to `0..* DosageDetails` (i.e., adopt
`DosageDetails` as requested, but keep the upper cardinality as many).

Optionally rename the element in invariants/short descriptions to `dosage`
(singular sense per occurrence) and add an invariant or guidance note
describing when multiple `DosageDetails` entries are appropriate (e.g.,
alternative regimens, taper schedules expressed as discrete phases, or
populations that share a single `ActivityDefinition` but require distinct
dosing variants).

#### Justification

`ActivityDefinition` is a reusable, definitional artifact that is sometimes
authored to express more than one acceptable dosing pattern (e.g., an adult
vs. pediatric regimen, or a step-down taper expressed as ordered phases).
Collapsing to `0..1` would force authors to either pick one regimen and lose
the others or move the variation outside `ActivityDefinition`, which can
reduce the artifact's reusability for CDS. `DosageDetails` already aggregates
the granular pieces, so keeping it repeating preserves both the post-refactor
typing the reporter is asking for *and* the prior `0..*` flexibility. The
trade-off vs. Disposition A is a slightly looser model and the need for
guidance on when multiplicity is meaningful, in exchange for not removing an
existing degree of expressivity.

---

### Disposition C: Decline

#### Proposal

Do not change `ActivityDefinition.dosage` at this time. Leave it as
`0..* Dosage` pending the resolution of the broader ballot feedback on the
overall Dosage / DosageDetails / DosageSafety / DosageCondition split, and
revisit `ActivityDefinition.dosage` (and any sibling elements on other
resources) as part of that consolidated decision.

#### Justification

The reporter themselves notes that the underlying refactor is likely to draw
substantial ballot feedback. Making a targeted cardinality + type change on
`ActivityDefinition.dosage` in isolation risks (a) immediately needing to be
re-reverted if the wider split is rolled back or restructured, and (b)
diverging `ActivityDefinition` from `MedicationRequest` and other resources
that still carry `0..* Dosage`, creating a temporary inconsistency that is
worse for implementers than the current uniform-but-imperfect state. A
"decline-for-now, re-open after the structural decision" disposition keeps
the spec internally consistent during the ballot cycle.

---

### Recommendation

**Recommended disposition:** A

The reporter's analysis is correct on the post-refactor semantics:
`DosageDetails` is the type that recovers the functional content of the
pre-refactor `Dosage`, and an `ActivityDefinition`'s `dosage` element is
naturally a single aggregated dosing intent rather than a collection of
independent `Dosage` instances. Accepting the change as requested
(`0..1 DosageDetails`) is the smallest, most semantically faithful fix and
prevents a silent loss of expressivity for CDS authors, which is
disproportionately serious for a "Highest" priority change. Disposition B is
a reasonable fallback if the workgroup believes multi-regimen authoring on a
single `ActivityDefinition` is a real and supported use case, but absent
evidence of that usage the simpler `0..1` shape is preferable. Disposition C
defers a clear, low-risk correction in exchange for cross-resource symmetry
that is itself a bug — better to fix `ActivityDefinition.dosage` now and
file companion tickets to bring the medication-family resources into line
than to leave a known regression in place during the ballot.
