## Summary

Bryn Rhodes requests that the Clinical Practice Guidelines (CPG-on-FHIR)
implementation guide explicitly accommodate `ObservationDefinition` within
the existing `PlanDefinition` / `ActivityDefinition` patterns, so that a
guideline activity can describe an expected observation (e.g., "take
blood pressure measurement") by pointing at — or generating — an
`ObservationDefinition`. The ticket was originally dispositioned
"Considered for Future Use" by CDS on 2024-07-10 (vote 20-0-0,
Greg White / Juliet K Rubini) on the rationale that observation taking
is normally expressed as a `ServiceRequest` driven by an
`ActivityDefinition`, and the workgroup wanted clearer use cases for
`PlanDefinition.action.definition` referring directly to an
`ObservationDefinition`. On 2024-11-26 Lynn Laakso reverted that
resolution and re-opened the ticket to be pursued after STU2
publication of the CPG IG. WHO (litlfred) commented in support, citing
SMART Guidelines content (e.g., ANC DT.04 "Conduct physical exams") as
a concrete consumer of an ObservationDefinition-aware CPG pattern.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the CPG-on-FHIR (Clinical Guidelines) IG so that
`ObservationDefinition` is a first-class definitional target for CPG
activities, in parallel with `ActivityDefinition`. Concretely:

1. Add an "Observation" CPG activity pattern (e.g.,
   `cpg-observationactivity` profile of `ObservationDefinition`)
   describing a guideline-recommended observation, mirroring the
   existing CPG activity profiles on `ActivityDefinition`.
2. Update CPG `PlanDefinition` guidance so that
   `PlanDefinition.action.definition` (`canonical`) is documented as
   resolvable to `ObservationDefinition` in addition to the currently
   documented targets, citing the base FHIR canonical-target set
   (`ActivityDefinition | MessageDefinition | ObservationDefinition |
   PlanDefinition | Questionnaire | SpecimenDefinition`) — no base
   spec change required.
3. Document the apply / `$apply` semantics for an
   `ObservationDefinition`-typed action target: how it is realized at
   run time (typically into an `Observation` resource with the
   permitted-data-type / quantitative-details propagated from the
   `ObservationDefinition`, optionally wrapped by a `ServiceRequest`
   when the observation must be ordered before being recorded).
4. Provide at least one worked example drawn from the WHO SMART
   Guidelines ANC content (DT.04 "Conduct physical exams", e.g.
   blood-pressure measurement).

#### Justification

The base FHIR `PlanDefinition.action.definition` already permits
`canonical(ObservationDefinition)`, and the workflow / order-catalog
communities (see Zulip threads above) already treat
`ObservationDefinition` as a peer of `ActivityDefinition` in
definitional graphs. The remaining gap is purely in the CPG IG, which
does not yet describe how to author or execute such a reference.
WHO/SMART Guidelines (litlfred's comment) is a concrete, named
implementer that needs this pattern, which directly answers the
question raised in the prior CDS vote ("what is the use case?"). The
re-open note by Lynn Laakso explicitly schedules the work for
post-STU2, so accepting fits the WG's stated intent.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of introducing a new CPG "Observation activity" profile on
`ObservationDefinition`, keep the activity pattern centered on
`ActivityDefinition` and add normative CPG guidance that, when an
activity describes "take an observation," the `ActivityDefinition`
SHALL/SHOULD reference the relevant `ObservationDefinition` via
`ActivityDefinition.observationResultRequirement` (and/or
`observationRequirement`), and the produced `ServiceRequest` /
`Observation` SHALL conform to that `ObservationDefinition`. Provide a
worked WHO ANC example using this composition rather than a
`PlanDefinition.action.definition → ObservationDefinition` direct
canonical link.

#### Justification

This preserves the original CDS rationale that observation taking is
fundamentally a workflow operation expressed via `ServiceRequest` /
`ActivityDefinition`, while still honoring the implementer need to
bind an `ObservationDefinition` to a guideline step. It avoids
multiplying CPG activity profiles, keeps the apply/execution semantics
in one place (`ActivityDefinition`), and uses a slot
(`observationResultRequirement` / `observationRequirement`) that was
designed specifically to point at `ObservationDefinition`. Trade-off
vs. A: it is slightly less direct from a `PlanDefinition` reader's
perspective (you must hop through an `ActivityDefinition` to see the
`ObservationDefinition`), and it is harder to express
"definitional-only" guidelines that have no operational/order
component.

---

### Disposition C: Decline

#### Proposal

Decline as "Not Persuasive — Already Supported." Document in the CPG
IG that `PlanDefinition.action.definition` is a `canonical` whose
permitted targets in base FHIR already include `ObservationDefinition`
(per the FHIR core definition cited in the cross-references), and that
guideline authors who need to reference an `ObservationDefinition` can
already do so today. No new CPG profile, element, or extension is
required.

#### Justification

The base `PlanDefinition.action.definition` element type set already
includes `canonical(ObservationDefinition)`. If the CPG IG adds no new
constraint preventing it, implementers (including WHO) can already
author the reference, and consumers can already resolve it (the
"General way to resolve a canonical resource" Zulip thread shows
implementers are actively building this resolution today). The earlier
CDS vote (2024-07-10, 20-0-0) reflected this view, framing
observation-taking as an `ActivityDefinition` / `ServiceRequest`
concern. Declining avoids adding a parallel activity profile family
solely for `ObservationDefinition` and the maintenance cost that comes
with it.

---

### Recommendation

**Recommended disposition:** A

The 2024-11-26 re-open note explicitly scheduled this work for after
STU2 publication, which signals the WG's intent is to accept rather
than re-decline. WHO/SMART Guidelines provides a named, large-scale
implementer use case (ANC DT.04 and similar physical-exam decision
tables) that directly answers the question raised in the prior
"Considered for Future Use" disposition. Disposition C is technically
correct that the canonical link already type-checks, but it leaves
authors and tooling without normative CPG guidance — which is exactly
what implementers are asking for and what the IG is meant to provide.
Disposition B is a credible fallback if the WG wants to minimize new
profiles, but it forces every observation-only guideline step through
an `ActivityDefinition` indirection that doesn't add semantic value
when the activity is purely "record an observation per this
`ObservationDefinition`." Disposition A is the most direct, makes the
WHO use case trivially expressible, and aligns with how the workflow
and order-catalog communities are already using `ObservationDefinition`
in definitional graphs.
