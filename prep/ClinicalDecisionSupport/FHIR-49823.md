## Summary

Bryn Rhodes reports that downstream R4 Implementation Guides which derive
profiles from the CRMI ShareableProfiles family (e.g. `crmi-shareablemeasure`)
fail snapshot generation with an unresolvable reference to the R5
`version-algorithm` ValueSet/CodeSystem. Although CRMI is published against
R4, its ShareableProfiles snapshots include the R5-only `versionAlgorithm`
extension, whose binding points at an R5-only value set that does not exist
in the R4 core. When an IG inherits the snapshot, the publisher cannot
resolve the binding URL, treats it as a local reference
(`version-algorithm.html`), and emits an error. Bryn frames this as a
publication defect in CRMI itself: the ShareableProfiles for R4 should not
be exposing R5 extensions/value sets to derived profiles. No related Jira
tickets are cross-referenced, but a Zulip thread on `#IG creation >
ShareableMeasure QA error` captures the original report (Jose Costa
Teixeira), Grahame Grieve's diagnosis, and a follow-up from Yan Heras
showing the same failure was still occurring against CRMI STU1 in
September 2025.

## Related Zulip Discussions

### [IG creation > ShareableMeasure QA error](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/ShareableMeasure%20QA%20error)

Jose Costa Teixeira opened the thread reporting that any FSH profile whose
parent is `http://hl7.org/fhir/uv/crmi/StructureDefinition/crmi-shareablemeasure`
(example repo: `costateixeira/shareable-measure`) immediately fails QA with
"The link 'version-algorithm.html' for 'version-algorithm' cannot be
resolved", traced to a relative link that appears in the inherited
snapshot. Grahame Grieve diagnosed it as a cross-version problem: the
`versionAlgorithm` extension is defined in R5 and its binding points at an
R5 ValueSet, but CRMI publishes against R4 where that value set does not
exist; the publisher's logical-to-literal URL algorithm cannot resolve the
binding, so it falls back to treating it as a local page reference, which
is never valid. Grahame noted he could reproduce locally but not on the
ci-build, and admitted he "doesn't know how to solve it"; Jose's
suggestion of minting an R4-flavoured copy of the extension was rejected
as "won't help". Yan Heras revived the thread on 2025-09-08 confirming the
same `http://hl7.org/fhir/ValueSet/version-algorithm` resolution error
still occurs against CRMI STU1, both locally and on the ci-build, with no
known workaround.

Participants: Jose Costa Teixeira, Grahame Grieve, Yan Heras.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a CRMI (FHIR) publication defect and fix the
ShareableProfiles family — primarily `crmi-shareablemeasure` and any
sibling `crmi-shareable*` profiles in the CRMI IG — so they no longer
inherit or surface the R5 `versionAlgorithm` extension when published
against R4. Concretely, in the CRMI IG sources: (a) on each ShareableX
StructureDefinition's differential, explicitly prohibit/zero-out
`extension:versionAlgorithm` (or remove the inherited differential entry)
so the published snapshot does not carry it, and (b) regenerate snapshots
so derived profiles see no reference to
`http://hl7.org/fhir/ValueSet/version-algorithm`. Republish CRMI (STU
update / technical correction) so downstream IGs deriving from the
Shareable* profiles stop hitting the unresolved-reference QA error.

#### Justification

Bryn (the CRMI editor) explicitly identifies the root cause as a CRMI
publication problem: an R4 IG should not be exposing R5-only extensions
and bindings to its consumers. The Zulip diagnosis confirms this — the
`versionAlgorithm` extension and its bound value set are R5 artifacts that
have no R4 equivalent, so the IG Publisher's logical-to-literal URL
resolver cannot do anything sensible with them. Yan Heras's September 2025
report shows that without a fix downstream implementers are still blocked,
which is the strongest signal that this needs to be addressed in CRMI
itself rather than worked around per IG. Removing the R5 extension from
the R4 ShareableProfiles snapshots is the most direct way to honour the
ticket's literal request.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than stripping the extension from CRMI's ShareableProfiles, push
the fix into the IG Publisher and/or the cross-version machinery so that
when an R4 publication encounters an R5-only extension whose binding
target does not exist in the active version, the publisher resolves the
canonical against the cross-version extension pack (e.g. publishes a
binding to `http://hl7.org/fhir/5.0/ValueSet/version-algorithm`, or treats
the binding as informational), instead of degrading to a relative
`version-algorithm.html` link and failing QA. CRMI would coordinate with
FHIR-I / the Tooling stream on the publisher behaviour and keep the
`versionAlgorithm` extension available on Shareable* profiles for
implementers who want it, while documenting the cross-version expectation
in the CRMI narrative.

#### Justification

The underlying class of problem — R4 IGs legitimately wanting to use R5
metadata extensions via the cross-version extension pack — is broader
than CRMI alone, and removing the extension from CRMI just shifts the
problem to the next IG that imports it. Grahame's note that "there's no
circumstance where [the local-reference fallback] is valid any more"
suggests the publisher itself has a latent bug that should be addressed
once, centrally. This option preserves the design intent of using
versionAlgorithm on shareable resources (a feature that materially helps
canonical-management consumers) and avoids a CRMI-specific patch that
other IGs reusing R5 extensions in R4 would still need to redo.

---

### Disposition C: Decline

#### Proposal

Decline to change CRMI on the basis that this is an IG Publisher /
cross-version-pack issue, not a defect in the CRMI specification itself.
Close as "Not a CRMI Issue" with a pointer to file the bug against the
HL7 FHIR IG Publisher (or the FHIR Extensions Pack) and add a CRMI
implementation note describing the current limitation and any per-IG
workaround (e.g. constraining the inherited extension to 0..0 in the
derived profile, or pinning the extension URL to the cross-version form).

#### Justification

CDS / CRMI editors could reasonably argue that the binding value set is
sourced from the FHIR core registry, not from CRMI, and that the failure
mode lives entirely in the publisher's URL-resolution algorithm. Grahame
confirmed in-thread that the publisher behaves inconsistently between
local and ci-build runs, which is publisher behaviour, not spec content.
Declining keeps the CRMI artifact stable and avoids what may be a churny
republication for what is ultimately a tooling defect. The cost is that
downstream IG authors continue to hit the error until the publisher / pack
is fixed elsewhere.

---

### Recommendation

**Recommended disposition:** A

The ticket reporter is the CRMI editor, and he has already characterised
this as a CRMI publication problem rather than a tooling complaint;
accepting that framing keeps responsibility where the affected
specification lives. Disposition A is also the only option that
demonstrably unblocks the implementers reporting the failure today
(Jose Costa Teixeira in February, Yan Heras in September) without waiting
on a coordinated publisher / extensions-pack change whose timeline CRMI
does not control. Disposition B is the more architecturally satisfying
long-term fix and is worth pursuing in parallel with FHIR-I / Tooling, but
it should not gate a near-term CRMI republication that removes the broken
inherited binding from `crmi-shareablemeasure` and its siblings. Disposition
C punts the problem and is not justified given an active in-the-wild
failure with a known, narrow CRMI-side fix.
