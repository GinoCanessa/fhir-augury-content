## Summary

This ticket asks the CQL specification to align the behavior of the
`LastPositionOf` string operator with the corresponding fix made in
FHIRPath: when the search pattern is an empty string, `LastPositionOf`
should return the **length of the source string** (i.e., the position
just past the last character), **not `0`**. The change is explicitly
modeled on the resolution of [FHIR-49227](https://jira.hl7.org/browse/FHIR-49227)
("LastIndexOf with an empty string should return the length of the
string"), which was applied to FHIRPath via [HL7/FHIRPath#19](https://github.com/HL7/FHIRPath/pull/19)
with resolution **Persuasive / Update as described**. The intent is to
keep CQL's string operators semantically consistent with their FHIRPath
counterparts, treating `LastPositionOf("0123", "")` as `4`,
`LastPositionOf("0", "")` as `1`, and `LastPositionOf("", "")` as `0`,
rather than the current trivial `0` for all cases.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CQL specification's definition of `LastPositionOf` so that
when the search argument is an empty string, the operator returns the
**length of the source string** (0-based "just past the last position")
rather than `0`. Concretely, in the `LastPositionOf(argument String,
string String)` operator definition (CQL string operators chapter):

- Add a normative sentence: "If `argument` is the empty string, the
  result is the length of `string` (i.e., the position immediately
  after the last character). If `string` is also empty, the result is
  `0`."
- Add worked examples mirroring FHIR-49227:
  - `LastPositionOf('', '0123') = 4`
  - `LastPositionOf('', '0') = 1`
  - `LastPositionOf('', '') = 0`
- Add a cross-reference note to FHIRPath's `lastIndexOf` (as updated
  per [FHIR-49227](https://jira.hl7.org/browse/FHIR-49227) /
  [HL7/FHIRPath#19](https://github.com/HL7/FHIRPath/pull/19))
  stating that the two operators are intentionally aligned.
- Mirror the change in the equivalent definition for any other CQL
  surface that re-exposes this operator (e.g., FHIR-mapped CQL
  operator catalog, if applicable).

#### Justification

The request is a direct, narrow alignment between two specifications
that already share a long history of operator parity (CQL borrows much
of its string-operator semantics from FHIRPath). FHIR-49227 was
resolved **Persuasive** by ITS and is **Applied** to FHIRPath; leaving
CQL's `LastPositionOf` returning `0` for the empty-search-string case
creates a silent semantic divergence that will surface as
implementation bugs and translation errors when CQL expressions are
lowered to FHIRPath (or vice versa). The Java/JavaScript
`String.lastIndexOf("")` precedent cited in FHIR-49227 supports the
same return-the-length convention, and the change is non-breaking for
any conforming engine that today returns the documented `0`, because
no realistic CQL logic relies on the degenerate "empty needle ⇒ 0"
result.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the semantic change but, rather than silently updating the
result for empty-string input, **define `LastPositionOf` with an empty
search argument as returning `null` (the empty result)** in CQL — and
add a normative note that engines following FHIRPath's
`lastIndexOf` may return the length of the source string. Update the
operator definition to:

- Specify: "If `argument` is the empty string, the result is `null`."
- Add an Implementation Note pointing to FHIR-49227 / HL7/FHIRPath#19
  and explaining that FHIRPath returns the length of the source
  string in this case; CQL authors who need that value should call
  `Length(string)` explicitly.

#### Justification

CQL is generally more null-propagating than FHIRPath and routinely
treats degenerate inputs as "no meaningful answer" rather than
manufacturing a sentinel position. Returning `null` for an empty
search argument keeps `LastPositionOf` aligned with how `PositionOf`,
`IndexOf`, and similar operators behave on null inputs in CQL today,
and avoids encoding a corner-case result that is rarely useful in
clinical logic. The trade-off vs. Disposition A is loss of strict
parity with FHIRPath; the Implementation Note mitigates that by making
the divergence explicit and offering a one-call workaround
(`Length(string)`).

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the current CQL definition of
`LastPositionOf` (empty-string argument ⇒ `0`). Resolve the ticket as
**Not Persuasive** with the rationale: "CQL's `LastPositionOf` has had
the documented behavior of returning `0` for an empty search argument
since first publication; changing it would be a breaking semantic
change for conforming engines and authored logic, and the use case is
degenerate. Authors who want the length of the source string should
call `Length(string)` explicitly."

#### Justification

The change, while small, alters the value returned by a normative
operator for a specific input class — by definition a breaking change
for any engine, test fixture, or authored expression that depends on
the documented `0` result. The use case ("what position does the empty
string occupy at the end of a non-empty string?") has no clinical
motivation; the value can already be obtained directly with
`Length(string)`. Declining preserves backward compatibility and
defers to the FHIRPath layer for any cross-spec semantic wrinkles.

---

### Recommendation

**Recommended disposition:** A

FHIR-52042's entire premise is alignment with FHIR-49227, which the
ITS work group already resolved **Persuasive** and which has been
**Applied** to FHIRPath via [HL7/FHIRPath#19](https://github.com/HL7/FHIRPath/pull/19).
CQL has a strong, long-standing convention of inheriting string-
operator semantics from FHIRPath, and silent divergence on a
specific operator's edge case is exactly the kind of trap that
produces mis-translated logic between the two specs. Disposition B's
"return null" instinct is defensible in the abstract but introduces a
*new* divergence at exactly the moment the rest of HL7 is closing one,
and Disposition C's backward-compatibility argument is weak: no real
CQL author is depending on `LastPositionOf(s, '') = 0`, and the
behavior change is trivially encodable in test suites. Accepting as
requested — with worked examples and an explicit cross-reference to
FHIR-49227 / HL7/FHIRPath#19 — keeps the two specifications in lock-
step and gives implementers an unambiguous target.
