## Summary

Bryn Rhodes is asking the Clinical Decision Support work group to add
implementation guidance — most likely in the Clinical Guidelines (CPG)
IG, with a possible touch-point in core ActivityDefinition — describing
how authors should specify, and how systems should resolve, the
participants declared on `ActivityDefinition.participant` when an
activity definition is realized into a concrete request resource.

The request is grounded in a concrete example: when an `ActivityDefinition`
is applied to produce a `Task`, the resulting `Task.requestedPerformer`
should be populated from the `ActivityDefinition.participant` entries.
The "Recording an Inference" use case is given to illustrate this — the
author's intent ("the system of record should record this inference")
should round-trip through `ActivityDefinition.participant` into
`Task.requestedPerformer` as the "system of record" performer.

There are no cross-referenced Jira tickets or Zulip threads for this
issue; the only cross-reference is the implicit mention of the FHIR
element `ActivityDefinition.participant`.

## Details

**Description:**

> As part of realizing an activity definition, provide guidance to
> support authors specifying and systems determining the requested
> participants.
>
> Specifically for a Task, the resulting task should have
> requestedPerformer set. Consider the example of Recording an
> Inference, where the author is just saying "the system of record
> should record this inference", the requestedPerformer should be the
> "system of record".
>
> Generally, we're providing guidance for how to resolve participants
> specified in the ActivityDefinition.participant.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new "Resolving ActivityDefinition.participant" guidance section to
the Clinical Guidelines (CPG) IG, alongside the existing `$apply`
guidance, that:

1. Describes how an authoring tool should populate
   `ActivityDefinition.participant.type` (and `.role`, `.function`) to
   express the requested performer of the activity, including how to
   express non-human performers such as a "system of record" (e.g.
   `participant.type = device` with a role/function code that names the
   system role, or `type = practitioner` with a role code drawn from the
   relevant value set).
2. Specifies the realization mapping when the target request is `Task`:
   each `ActivityDefinition.participant` SHALL contribute one entry to
   `Task.requestedPerformer`, with `type` mapping to
   `requestedPerformer.concept`/role and `role`/`function` carried
   through to the corresponding `Task.requestedPerformer.concept`
   coding(s). Walk through the "Recording an Inference" example end to
   end to illustrate the mapping.
3. Provides an analogous mapping table for the other request resources
   `$apply` can produce (ServiceRequest.performer/performerType,
   MedicationRequest.performer, CommunicationRequest.recipient,
   etc.) so the rule is uniform rather than Task-specific.
4. Adds a short cross-reference paragraph in core ActivityDefinition's
   element-level guidance for `participant` pointing to the new CPG
   section as the normative resolution algorithm.

#### Justification

The ticket explicitly asks for guidance both for authors (how to
specify) and for systems (how to resolve), and the CPG IG is where
`$apply` semantics already live, so that is the natural home for the
algorithm. Calling out `Task.requestedPerformer` specifically — as the
ticket does — closes a known gap: today, multiple implementations
generate Tasks from ActivityDefinitions but disagree on whether
`participant` flows into `owner`, `requester`, `performerType`, or the
newer `requestedPerformer`, and the ticket pins down the correct
target. Providing analogous mappings for the other request resources
prevents the same ambiguity from re-emerging the next time someone
applies an ActivityDefinition to a non-Task target. This is purely
additive guidance and does not change any element definitions, so it
carries no compatibility risk.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding only prose guidance, also publish a small,
machine-checkable artifact set in the CPG IG:

- A **CPG ActivityDefinition profile** that constrains
  `participant.type`/`.role` to the CPG-recommended value sets and adds
  invariants (e.g. "if the realized request is a Task, at least one
  participant SHALL be present").
- A **realization mapping** expressed as a ConceptMap (or a short
  StructureMap) from `ActivityDefinition.participant.type` codes to
  `Task.requestedPerformer` concept codings, plus a parallel mapping for
  ServiceRequest/MedicationRequest/CommunicationRequest performer
  fields.
- A worked **example pair** (an `ActivityDefinition` for "Record an
  Inference" and the realized `Task` it produces) referenced from the
  prose so implementers have a concrete conformance target.

The prose section from Disposition A is still added, but it points at
the profile, the maps, and the example as the source of truth.

#### Justification

CPG-IG already publishes profiles and maps for other realization
behaviors, so an artifact-driven answer is consistent with how the IG
treats `$apply`. A ConceptMap/StructureMap is testable by tooling
(validators, the CDS reference implementation), which avoids the
"two implementers, two interpretations" problem that arises when the
spec only ships prose. The cost is the additional authoring and ballot
review burden for the new artifacts, and the ConceptMap will need
maintenance as the participant value sets evolve — but the long-term
clarity for downstream IGs (e.g. CPG-on-FHIR-based recommendation IGs)
is worth that overhead.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Not Persuasive" on the grounds that
`ActivityDefinition.participant` and `Task.requestedPerformer` are
already defined element-by-element in core FHIR, and that mapping
participants to the realized request is an implementation concern best
left to applying tools and/or to downstream IGs that profile
ActivityDefinition for their own use cases.

#### Justification

Core FHIR is intentionally light on `$apply` realization mechanics —
that is the CPG IG's job, not the base spec's. If declined, this could
be re-filed against any IG that needs the mapping (e.g. a specific
recommendation IG), where the participant types are narrower and the
guidance can be more prescriptive. Declining keeps the CPG IG from
codifying a single mapping that may not fit every realization scenario
(e.g. cases where `ActivityDefinition.participant` should flow into
`Task.owner` or `requester` rather than `requestedPerformer`).

---

### Recommendation

**Recommended disposition:** A

The ticket asks a narrow, well-scoped question — "where does
`ActivityDefinition.participant` go when I realize the activity?" — and
the reporter (Bryn Rhodes) has already named the answer for the Task
case. Disposition A delivers exactly that, in the IG that owns `$apply`,
without committing the work group to authoring and balloting new
profiles, ConceptMaps, and StructureMaps. Disposition B is attractive
long-term but is materially more work and risks the ticket sitting in
"Triaged" for another cycle while the artifacts are produced; the work
group can always layer the machine-checkable artifacts on top of the
prose in a follow-on ticket once the algorithm is agreed. Disposition C
is hard to defend given that the reporter is the CPG editor and is
explicitly asking for the guidance to live somewhere — declining would
just push the same question to every downstream IG. Accept as requested,
prioritise the Task / `requestedPerformer` mapping in the first cut, and
file a follow-on for the ConceptMap/profile work.
