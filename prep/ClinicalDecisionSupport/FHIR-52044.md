## Summary

FHIR-52044 asks the CQL specification to incorporate the new FHIRPath
`repeatAll()` function — a variant of `repeat()` that omits the
uniqueness check on intermediate iteration outputs — into the CQL
Logical Specification, along with a mapping from `FHIRPath.repeatAll` to
the equivalent ELM construct. The ticket text mirrors the resolution of
the predecessor FHIRPath ticket
[FHIR-36708](https://jira.hl7.org/browse/FHIR-36708) ("add
repeatNonUnique"), which was already accepted as Persuasive against the
FHIRPath specification (resolved 2025-05-08, status Applied) and named
the new function `repeatAll` based on Brian Postlethwaite's suggestion.
Because `repeat()` deduplicates on each step, it can be substantially
slower than `descendants()` on large trees; `repeatAll()` skips that
check at the cost of inviting infinite loops, so the request also asks
that the CQL spec carry the matching warning text and "dragon" indicator
about careless expressions (e.g. `repeat('item')`), document the
existing `repeat()` function in the Reference (it is currently only in
the Logical Specification), and define the ELM mapping for the new
function.

## Details

**Description:**

> Include a new function repeatAll which is similar to repeat() except
> that it does not contain the uniqueness check. (This will call the
> input expression on the new output from each iteration)
>
> Include a warning that implementations should ensure that there are
> safety catches to ensure that infinite loops are not inadvertantly
> created.
>
> Include a dragon indicator to note that careless expressions could
> result in an infinite loop. Such as `repeat('item')`. An
> implementation could impose a max limit to the number of iterations to
> prevent infinite loops, though it could also statically check that the
> expression argument contains a property accessor, and could return an
> error in some cases.
>
> In addition, document the existing Repeat function in the Reference
> (it's only in the Logical Specification currently).
>
> Add the RepeatAll to the Logical Specification, and a mapping from
> FHIRPath.repeatAll to the ELM equivalent.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-52044 as Persuasive and apply the CQL-side companion to
FHIR-36708:

1. **CQL Logical Specification** — Add a new operator `RepeatAll`
   alongside the existing `Repeat`, defined identically to `Repeat`
   except that intermediate results are not deduplicated (i.e., the
   per-iteration source expression is evaluated against every element of
   the previous step's output, including duplicates). Carry the same
   signature/return-type pattern used by `Repeat`.
2. **CQL Reference** — Document the existing `Repeat` function (which
   currently appears only in the Logical Specification) and add the new
   `RepeatAll` function next to it. Include the safety guidance: a
   warning that implementations must guard against unbounded iteration,
   and a "dragon" callout that careless expressions such as
   `repeat('item')` can loop forever; suggest implementation strategies
   (max-iteration cap, or static analysis that the argument is a
   property accessor).
3. **FHIRPath ↔ ELM mapping** — Add a row mapping `FHIRPath.repeatAll`
   to the new `RepeatAll` ELM operator, mirroring the existing
   `FHIRPath.repeat` → `Repeat` mapping. Align prose with FHIRPath's
   navigation-model vocabulary ("node") so the two specs read
   consistently.
4. **Scoped-functions alignment** — Explicitly state, consistent with
   the FHIRPath FHIR-44774 thread, that `RepeatAll` does not set
   `$index` during iteration.

#### Justification

The underlying FHIRPath change has already been accepted, applied, and
shipped in [HL7/FHIRPath#19](https://github.com/HL7/FHIRPath/pull/19);
the CQL spec's stated policy is to track the FHIRPath grammar/library so
that the FHIRPath subset of CQL stays aligned. Without this work, CQL
authors get a function in FHIRPath with no defined ELM target, breaking
round-tripping between CQL→ELM compilers and any FHIRPath engine that
implements the new function. The warning/dragon text mirrors what the
FHIRPath spec already carries, so risk-of-infinite-loop is communicated
identically to authors regardless of which surface they use.
Documenting the existing `Repeat` in the Reference at the same time
closes a long-standing gap noted in the ticket itself.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent but apply a narrower change:

- Add `RepeatAll` to the **Logical Specification** and the
  **FHIRPath→ELM mapping table** only, defining it strictly as the
  no-uniqueness sibling of `Repeat`.
- **Do not** introduce a new "dragon" indicator in CQL; instead, add a
  single short non-normative note next to `Repeat`/`RepeatAll` pointing
  readers to the FHIRPath specification for the safety guidance, and
  defer documenting the existing `Repeat` in the Reference to a separate
  editorial ticket.
- Optionally, define `RepeatAll(source, element)` in ELM with an
  implementation-defined iteration cap as an authoring concern (rather
  than a normative requirement), so engines can choose between a hard
  cap and static analysis.

#### Justification

CQL historically defers FHIRPath-grammar semantics to the FHIRPath
specification rather than restating them. Re-stating the warning and the
"dragon" callout risks the two documents drifting if FHIRPath later
refines the wording. Splitting the Reference documentation of the
existing `Repeat` into a separate editorial ticket keeps the scope of
this ticket tight (one new operator + one mapping row) and avoids
bundling a doc-debt cleanup with a normative addition. The trade-off vs.
Disposition A is reduced visibility of the infinite-loop hazard for
CQL-only readers who do not consult the FHIRPath spec.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-52044 on the grounds that the request is already addressed
at the FHIRPath layer and that CQL does not need to carry a parallel
operator: CQL authors who require non-uniquing recursion can either (a)
call FHIRPath's `repeatAll()` directly through CQL's FHIRPath
pass-through, or (b) express the same traversal using existing CQL
constructs (`Descendants`, recursive function definitions, or `Repeat`
followed by `distinct`-aware logic where uniqueness was actually
desired).

#### Justification

This would only be defensible if CQL were prepared to treat FHIRPath
library extensions as out-of-scope for ELM mapping. In practice CQL has
historically chosen to map every FHIRPath function it intends to support
to an ELM operator, so leaving `repeatAll` unmapped creates a real
authoring/portability hole. This option is included for completeness,
but the rationale is weak.

---

### Recommendation

**Recommended disposition:** A

The predecessor ticket FHIR-36708 was already accepted as Persuasive and
applied to FHIRPath via HL7/FHIRPath#19; FHIR-52044 is the
straightforward CQL-side follow-through and was filed by the CQL
specification editor (Bryn Rhodes) himself. Splitting the safety prose
out (Disposition B) trades a small editorial-debt reduction for ongoing
risk that CQL readers miss the infinite-loop hazard, and declining
(Disposition C) leaves an unmapped FHIRPath function which directly
breaks the CQL/FHIRPath alignment policy. Adopting all four pieces of
Disposition A in one go — operator, Reference doc (including the
already-existing `Repeat`), ELM mapping, and the `$index` /
navigation-model alignment surfaced in the FHIRPath Zulip threads —
keeps the two specifications synchronized and resolves the documentation
gap the ticket explicitly calls out.
