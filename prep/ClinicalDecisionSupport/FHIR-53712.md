## Summary

The reporter (Isaac Vetter) asks an open, exploratory question: does the
core FHIR specification have — or should it have — an element analogous
to the CDS Hooks `Card.indicator` (the small enumerated severity / call-
to-action signal: `info | warning | critical | success`)? The ticket is
a single-sentence Change Request filed against FHIR Core under the
Clinical Decision Support work group, with no description body beyond
the title, no comments, and no explicit links. No cross-referenced Jira
tickets, Zulip threads, or GitHub items were surfaced for this key, so
the ticket should be treated as a scoping / discovery question rather
than a concrete change proposal. The work group needs to decide
*whether* such an indicator concept belongs in core FHIR, and if so,
*where* it should live (which resource(s) and under what name) and how
it should relate to the existing `Card.indicator` defined in the CDS
Hooks specification.

### [cds hooks > CDS Hooks Card Indicator Element](https://chat.fhir.org/#narrow/stream/cds%20hooks/topic/CDS%20Hooks%20Card%20Indicator%20Element)

In June 2021, Christa Wan asked whether the CDS Hooks Card structure
could be extended so that a service consuming the response (e.g., to
update a `CarePlan`) could programmatically tell *what* the indicator
meant, rather than relying on the free-text `detail` and the small
`indicator` enumeration. Bryn Rhodes responded that the spec already
supports two service-side mechanisms — returning a "proposed diagnosis"
via the `suggestions` element (e.g., a `Condition` resource), or
defining custom extensions such as `guidelineEligible` — but noted that
either approach requires client/service coordination. Alex Goel
followed up that for `CarePlan` update use cases, the existing
`indicator` element did not carry enough structure on its own. The
thread did not reach a formal conclusion; it illustrates that the
community has previously found `Card.indicator` to be useful for
display priority but insufficient as a structured signal — context the
work group should weigh when deciding what (if anything) a core-FHIR
indicator element would add.

Participants: Christa Wan, Bryn Rhodes, Alex Goel.

## Proposed Dispositions

Because no cross-referenced GitHub work has been identified, the
dispositions below intentionally avoid naming specific authoring paths
or files; any concrete spec change driven by Disposition A or B would
land in the FHIR Core build (`HL7/fhir`, source/<resource>/) and the
work group should request a `repo-analysis` refresh before drafting a
PR.

### Disposition A: Accept As Requested

#### Proposal

Add an `indicator` element (or equivalently-named element) to a
clinical-reasoning-oriented core resource that today carries the result
of a CDS evaluation. The natural candidate is `GuidanceResponse`, with
a new `0..1` element `GuidanceResponse.indicator` bound to a value set
that mirrors the CDS Hooks `Card.indicator` enumeration —
`info | warning | critical` plus `success` — drawn from (or aligned to)
the value set already published with the CDS Hooks specification.
Define the element as the "intended call-to-action level" of the
guidance, distinct from existing `status` and from
`OperationOutcome.issue.severity` which carries a different semantic
(processing severity, not clinical urgency). Document the relationship
to `Card.indicator` in the CDS Hooks IG so the two stay aligned across
ballots.

#### Justification

`GuidanceResponse` is FHIR's existing carrier for "the result of
applying a decision support module"; it already overlaps semantically
with a CDS Hooks Card payload but lacks the at-a-glance severity
signal that drives renderers. Reusing the CDS Hooks indicator
enumeration keeps a single shared vocabulary across CDS Hooks Cards,
PlanDefinition/RequestOrchestration evaluation results, and
GuidanceResponse, which lowers integration burden for vendors that
already render Cards. The June 2021 Zulip thread ("CDS Hooks Card
Indicator Element") shows implementer interest in a *programmatic*
indicator beyond free text; promoting it to a typed element on a
clinical-reasoning resource addresses that need without forcing
extensions on every implementer.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a new core element. Instead, publish a standard
**extension** — e.g., `http://hl7.org/fhir/StructureDefinition/cds-indicator`
— typed as `code`, bound to the same enumeration used by CDS Hooks
`Card.indicator`, and explicitly applicable to `GuidanceResponse`,
`Communication`, `CommunicationRequest`, `DetectedIssue`,
`RequestOrchestration`, and any future CDS-result-bearing resource.
Document the extension in the Clinical Reasoning module (or the CDS
Hooks IG itself) with mapping guidance to `Card.indicator`. Optionally,
bring the extension forward as a candidate core element in a future
ballot once usage data is in.

#### Justification

The ticket as written is a scoping question, not a concrete change
request, and the use cases for "CDS-style indicator on a FHIR resource"
are still emergent. A standard extension lets the community get
implementation experience across multiple resources without committing
core spec real estate or risking a poorly-placed element that has to
be deprecated later. It also avoids picking a single "right" host
resource prematurely — Disposition A presupposes `GuidanceResponse`,
but real-world CDS workflows also surface indicators via
`Communication`, `DetectedIssue`, and `RequestOrchestration`. An
extension is symmetric across all of these, and is the path FHIR
Methodology generally prefers for cross-cutting clinical signaling.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by existing mechanisms**.
The CDS Hooks specification (an HL7 standard in its own right) already
defines `Card.indicator` for exactly the call-to-action signaling the
ticket asks about. Where a similar signal is needed inside a FHIR
resource, implementers can use existing typed elements —
`OperationOutcome.issue.severity`, `Communication.priority` /
`CommunicationRequest.priority`, `DetectedIssue.severity`, and
`Flag.status` / `Flag.category` — combined with bound terminologies, to
convey severity or urgency without introducing a new core element.
Close as Persuasive with No Change, pointing the reporter to the CDS
Hooks `Card.indicator` definition and the existing severity / priority
elements on the resources above.

#### Justification

The ticket provides no specific use case, no target resource, and no
example payload; without that, adding a new element risks duplication
with `OperationOutcome.severity`, `Communication.priority`, and
`DetectedIssue.severity`, all of which already encode something close
to "how seriously should the recipient treat this?". CDS Hooks itself
is the appropriate home for `Card.indicator`, and any FHIR-resource-
side need can be met today via the priority/severity elements that
already exist. Adding another enumerated severity-like element to core
without a concrete gap risks confusion about which element to populate
and burdens implementers with yet another vocabulary to map.

---

### Recommendation

**Recommended disposition:** B

The ticket is a one-line scoping question with no description, no
comments, no linked tickets, and no GitHub work — it is not yet a
concrete proposal. Disposition A commits core FHIR to a specific
element on a specific resource (`GuidanceResponse`) before the work
group has agreed *which* resource is the right host, and the 2021
Zulip thread shows that real implementer needs span multiple
resources (CarePlan-update, eligibility evaluation, suggestions). At
the same time, Disposition C understates the gap: existing
priority/severity elements were not designed for the CDS Hooks
"call-to-action" semantic and do not align cleanly with
`Card.indicator`. Disposition B threads the needle: publish a standard
extension aligned to `Card.indicator`'s value set, applicable across
`GuidanceResponse`, `Communication[Request]`, `DetectedIssue`, and
`RequestOrchestration`, and revisit promotion to a core element once
implementer feedback shows where it actually gets used. The work group
should also ask the reporter for a concrete use case to attach to the
ticket before finalizing — that input may shift the recommendation
toward A on a specific resource.
