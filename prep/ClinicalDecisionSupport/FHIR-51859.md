## Summary

Bryn Rhodes proposes defining a new `GraphDefinition/$apply` operation,
modeled on the existing `PlanDefinition/$apply` operation but operating
over a `GraphDefinition` resource. When invoked, the operation would
return — per subject — a `Bundle` whose first entry is a `Composition`
representing the result of applying the `GraphDefinition` to that
subject, with subsequent entries holding the resources referenced by the
`Composition` (i.e., a FHIR Document). The `Composition` would be built
by walking the GraphDefinition's links, following the most pertinent
"case feature" relationships specified in the GraphDefinition. Bryn
notes that the operative `GraphDefinition` itself can often be inferred
from a starting pathway, strategy, or recommendation definition by
collecting its related case features, and suggests internally separating
that into its own operation (and possibly a `PlanDefinition/$infer-graph`
operation). The cross-reference and Zulip indices contain no related
discussions or implementing artifacts for this ticket; it appears to be
a fresh, scoped proposal for the Clinical Guidelines (CPG-on-FHIR)
specification.

## Details

**Description:**

Define a GraphDefinition/$apply operation with parameters analogous to
PlanDefinition/$apply, but for a GraphDefinition.

The result of the operation is a Bundle per subject, where the first
entry in the Bundle is the Composition that is the result of applying
the GraphDefinition to the subject, and subsequent entries in the
Bundle are the resources referenced by the Composition (i.e. each
entry is a FHIR Document).

The Composition is constructed by following the most pertinent case
feature relationships, as specified in the GraphDefinition.

Note that the GraphDefinition can be inferred by starting from a
specific pathway, strategy, or recommendation definition and retrieving
all the related case features.

Internally, these should be separate operations, and we could also
consider defining a $infer-graph operation on a PlanDefinition that
would infer the GraphDefinition resource to be used.

**Comments:**

- **bryn.rhodes (2025-12-03):** Consider the `$infer-graph` aspect as a
  separate ticket.
- **bryn.rhodes (2025-12-03):** The operation would walk the links in
  the GraphDefinition, creating a section for each link with the
  resources that conform to the profile identified in the link. If the
  link is itself a GraphDefinition, whether to recurse would be
  governed by a `recursive` parameter. If `recursive=true`, the section
  would itself be a `Composition`; if `false`, it would instead be a
  reference giving the consumer a way to invoke the operation for that
  graph definition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new `GraphDefinition/$apply` operation in the Clinical
Guidelines (CPG-on-FHIR) specification, modeled on
`PlanDefinition/$apply`. The operation:

- **Inputs:** `subject` (Reference, required, repeating to match
  PlanDefinition/$apply's per-subject semantics), `encounter`,
  `practitioner`, `organization`, `userType`, `userLanguage`,
  `userTaskContext`, `setting`, `settingContext`, plus `data` /
  `dataEndpoint` / `contentEndpoint` / `terminologyEndpoint` parameters
  mirroring `PlanDefinition/$apply`. Add a `recursive` (boolean,
  default `false`) parameter as described in Bryn's second comment to
  control whether nested `GraphDefinition` links are themselves applied
  inline (producing a nested `Composition` section) or surfaced as a
  reference back to a separate `$apply` invocation.
- **Behavior:** For each subject, walk
  `GraphDefinition.link`. For each link, create a `Composition.section`
  populated with resources that conform to the profile identified by
  the link's target. When the link's target is itself a
  `GraphDefinition`, honor `recursive`.
- **Output:** A `Bundle` of `type = document` per subject, with the
  generated `Composition` as the first entry and the referenced
  resources following.
- Defer the `$infer-graph` companion operation to a separate ticket
  per Bryn's first comment.

The operation would be authored alongside the other CPG-on-FHIR
operation definitions (the standard location is the
`OperationDefinition/cpg-*` artifacts and the operation page in the
`cpg` IG); this ticket should be wired into the IG's resource list and
its narrative pages.

#### Justification

The proposal is well-scoped and internally consistent with the
`PlanDefinition/$apply` pattern that CPG already standardizes, so it
minimizes new design surface for implementers. Following `GraphDefinition.link`
to assemble a `Composition` aligns with the
existing intent of `GraphDefinition` (declaring how related resources
hang off a root) and with `Composition`'s role as the entry point of a
FHIR Document `Bundle`, so both pieces are being used the way the core
spec intends. The `recursive` parameter is the standard way FHIR
operations bound the cost of walking arbitrarily deep graphs and gives
servers a knob to refuse expensive expansions. Splitting `$infer-graph`
into a separate ticket keeps the semantics of `$apply` crisp.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introduce a stand-alone `GraphDefinition/$apply`, extend
`PlanDefinition/$apply` (and/or its existing `$apply` family) with an
optional `graphDefinition` input parameter (a canonical or contained
`GraphDefinition`). When supplied, `$apply` would return a document
`Bundle` whose `Composition` is built by walking the supplied
`GraphDefinition`'s links over the resources produced by the plan.
Equivalently, this could be exposed as `PlanDefinition/$apply-graph`
on `PlanDefinition` only. `GraphDefinition` itself would not gain a new
`$apply` endpoint; instead the existing `$infer-graph` proposal in
Bryn's comment becomes the way to obtain the graph for a pathway.

#### Justification

In CPG-on-FHIR practice the GraphDefinition is rarely authored directly
by humans — it's typically inferred from a pathway/strategy/recommendation
definition (Bryn explicitly notes this). Putting the new behavior on
`PlanDefinition/$apply` keeps a single, familiar entry point for
clients and avoids splitting the CPG operation surface across two
resources. It also sidesteps the FMM/maturity asymmetry between
`GraphDefinition` (lower maturity, sparsely implemented operationally)
and `PlanDefinition` (broadly implemented in CPG tooling). The cost is
lost orthogonality: clients that already have a `GraphDefinition`
canonical and just want to "compose a document" must go through a
`PlanDefinition` shell, which is awkward.

---

### Disposition C: Decline

#### Proposal

Decline to define `GraphDefinition/$apply` in the base CPG-on-FHIR
specification. Document the desired behavior as an
implementation-guide-level pattern: clients that want a per-subject
document can already invoke `PlanDefinition/$apply` to produce the
content `Bundle` and then construct a `Composition` themselves by
walking a `GraphDefinition` they hold, using existing FHIR search and
`Composition.section.entry` semantics. Mark the ticket as
"persuasive — not adopted, no normative change" with a pointer to the
documented pattern.

#### Justification

`GraphDefinition` is currently low-maturity and has limited
server-side implementation; standardizing a new operation on it raises
the implementation cost across the ecosystem without a demonstrated
multi-implementer demand (the cross-reference and Zulip indices show
no community discussion threads, no PRs, and no related tickets driving
the request). The composition behavior described can be assembled today
from existing operations (`PlanDefinition/$apply` plus a client-side
walk of `GraphDefinition.link`), so the marginal benefit is convenience
rather than capability. Declining keeps the CPG operation surface small
and avoids locking semantics in before more than one implementer has
exercised the pattern in production.

---

### Recommendation

**Recommended disposition:** A

This is a well-shaped proposal from the resource's primary steward
that fills a real CPG-on-FHIR gap: today, "give me a per-subject FHIR
Document assembled from a knowledge graph" requires every client to
re-implement the link-walking and `Composition`-building logic. The
operation as described re-uses `PlanDefinition/$apply`'s parameter
shape, keeps the deeper `$infer-graph` design out of scope (per the
reporter's own comment), and includes a `recursive` knob that bounds
server cost. Disposition B's "fold it into `PlanDefinition/$apply`"
saves a binding but conflates two operations — composing a document
from a graph versus applying a knowledge artifact — and worsens the
ergonomics for clients that already hold a `GraphDefinition`.
Disposition C is defensible on maturity grounds, but the workgroup is
the natural place to incubate this; deferring it keeps the convenience
gap open indefinitely. Recommend accepting and tracking `$infer-graph`
as the follow-on ticket Bryn called out.
