## Summary

The ticket flags that the `ActivityDefinition` resource binds at least one
value set — `action-participant-function`, backed by the
`ActionParticipantFunction` CodeSystem — that is a candidate for migration
out of the FHIR Core specification into the Terminology HL7.org (THO / UTG)
publication stream. The reporter (Lloyd McKenzie) notes that migration is
not strictly required, but should be considered when the value set's
definition is likely to evolve over the coming years, or when the value set
might be reusable by other product families (CDA, V2, etc.). The ticket is
the result of an automated/templated import ("Comment 27 - imported by:
Lloyd McKenzie") and lists `ActivityDefinition.participant.function` as the
specific binding to evaluate.

The ticket was generated for the Clinical Decision Support work group as
the steward of `ActivityDefinition`. There are **no cross-referenced Jira
tickets, Zulip threads, or GitHub items** linked to FHIR-55061 in the
augury index — the only cross-reference detected is the self-mention of
the FHIR element `ActivityDefinition.participant.function`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the `ActionParticipantFunction` CodeSystem and the
`action-participant-function` ValueSet out of FHIR Core and into the
Terminology HL7.org (THO/UTG) publication. After migration:

- Update the binding on `ActivityDefinition.participant.function` to
  reference the THO-hosted canonical URLs for the CodeSystem and ValueSet.
- Replace the in-spec definitions in FHIR Core with stub references (or
  remove them entirely) so the THO copies are the single source of truth.
- Coordinate with the Vocabulary work group to schedule the move in the
  next THO publication cycle, and verify the binding strength
  (`example`/`extensible`/etc.) is preserved on the new canonical URL.

#### Justification

The CodeSystem is small, narrowly used (a single binding on a
plan/workflow resource), and exactly the kind of artifact THO is designed
to host so it can evolve on its own cadence and be referenced by CDA, V2,
and IGs without dragging the FHIR Core release calendar. It directly
implements Lloyd McKenzie's stated criteria — "value set definition needs
to change in coming years" and "useful in other specs" — and aligns with
the broader programme that produced the templated ticket. CDS owns the
binding but Vocabulary owns the publication mechanics, so this is a
scheduling exercise rather than a design change.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than migrating the existing CodeSystem to THO, **rebind
`ActivityDefinition.participant.function` as an `example` binding pointing
at an externally-maintained terminology** (per bryn.rhodes' comment). The
investigation step would:

1. Survey existing role/function code systems (e.g., HL7 v3
   `ParticipationFunction`, SNOMED CT roles, NUCC provider taxonomy) for
   coverage of the participant-function concept space.
2. If a suitable external code system exists, change the binding to an
   `example` binding to a curated value set drawn from that source and
   retire `action-participant-function` (or keep it only as a fallback
   example).
3. If no suitable external code system exists, fall back to Disposition A
   (THO migration of the existing artifact).

#### Justification

Bryn explicitly raised this option in the only reply on the ticket and
asked Rob McClure to weigh in. If a community-maintained terminology
already covers the concept space, leaning on it reduces the long-term
maintenance burden for both CDS and Vocabulary, gives implementers a
larger and more familiar code pool, and avoids creating yet another
HL7-published code system whose only use is a single binding on
`ActivityDefinition`. The trade-off vs. Disposition A is that it requires
real terminology research up front and may produce no usable result, in
which case THO migration is still the answer.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Persuasive with mod / Not Persuasive — no change in this
cycle*. Document that:

- `ActivityDefinition.participant.function` already has a workable binding
  to `action-participant-function`.
- The CodeSystem is stable, has not received change requests, and has no
  current cross-product (CDA/V2) consumers asking for it.
- Migration to THO is a non-trivial coordination cost across CDS,
  Vocabulary, and the FHIR Core publication, and there is no concrete
  driver for the move beyond the templated review.

#### Justification

The ticket is one of an automated batch ("Comment 27 - imported by Lloyd
McKenzie") rather than a stakeholder-driven request, the CodeSystem is
narrowly scoped and small, and there is no evidence in the augury index
of any external party asking to consume this terminology from THO, CDA,
or V2. Declining preserves CDS bandwidth for changes with concrete
implementer demand and avoids a cross-work-group migration whose only
benefit is theoretical reusability.

---

### Recommendation

**Recommended disposition:** B

The only substantive comment on the ticket — bryn.rhodes asking whether
existing terminologies could be reused — is exactly the question
Disposition B answers, and the answer determines whether Disposition A is
even needed. Doing the terminology survey first costs little, gives Rob
McClure a concrete deliverable to respond to, and produces one of two
clean outcomes: either CDS rebinds to a reusable external code system
(best-of-both-worlds: less HL7-maintained content *and* the cross-product
reusability the original ticket is asking about), or the survey confirms
no external option exists and the ticket folds neatly back into
Disposition A's THO migration. Disposition C is hard to defend while the
only on-ticket discussion is an open question that has not been answered.
