## Summary

Bryn Rhodes is asking the Clinical Decision Support work group to extend the
"Using CQL With FHIR" implementation guide so that the mapping section
explicitly covers how CQL values are represented inside FHIR `Extension`
elements — not just on `ParameterDefinition` and `Parameters`. The driving
use case is `cqf-supportingEvidence`, where the result of a CQL expression is
returned as an extension and therefore needs the same `cqf-isEmptyList`,
`cqf-isEmptyTuple`, and `cqf-cqlType` markers that are already used to round-
trip CQL semantics through FHIR.

The companion FHIR Infrastructure / Extensions Pack ticket
[FHIR-56173](https://jira.hl7.org/browse/FHIR-56173) (resolved Persuasive,
Applied via [HL7/fhir-extensions#239](https://github.com/HL7/fhir-extensions/pull/239))
adds `Extension` to the allowed Context of Use for those three extensions.
This ticket is the CDS-side companion: once those extensions can legally
appear on `Extension`, the "Using CQL With FHIR" IG must document that
mapping pattern alongside the existing ParameterDefinition / Parameters
mappings, so implementers know how and when to attach `cqf-isEmptyList`,
`cqf-isEmptyTuple`, and `cqf-cqlType` to extension values (e.g., empty lists
or tuples returned from a CQL expression embedded in a MeasureReport).

## Details

**Description:**

In addition to the mapping to and from ParameterDefinition and Parameters,
CQL values need to be able to be mapped in to Extensions in order to support
use cases such as cqf-supportingEvidence, where the result of a CQL
expression is represented in an extension.

Extensions pack tracker https://jira.hl7.org/browse/FHIR-56173 extends the
context of use for these extensions to support this use case, but it should
also be documented in the mapping section in this implementation guide.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the "Using CQL With FHIR" IG (`HL7/cql-ig`), extend the mapping section
that today describes the CQL ↔ `ParameterDefinition` and CQL ↔ `Parameters`
round-trip to add a parallel sub-section "CQL values in Extension". That
sub-section should:

1. State that any FHIR `Extension` may carry a CQL-typed value, and that the
   CQL semantics that cannot be expressed in the FHIR primitive value alone
   are conveyed via sibling extensions:
   - `cqf-cqlType` — the CQL type name (e.g., `List<System.Any>`,
     `Tuple{...}`, `Interval<DateTime>`).
   - `cqf-isEmptyList` — present on a List-typed value to indicate the empty
     list (since FHIR cannot represent an empty repeating element).
   - `cqf-isEmptyTuple` — present on a Tuple-typed value to indicate the
     empty tuple.
2. Reuse the same value-mapping table that already governs
   `ParameterDefinition` / `Parameters` (CQL primitive → FHIR primitive,
   List → repeated element + `cqf-isEmptyList` marker, Tuple → grouping
   extension + `cqf-isEmptyTuple` marker, etc.) and reference it from the
   new section rather than restating it.
3. Provide at least one worked example aligned with `cqf-supportingEvidence`
   (the use case Bryn cites) and a second MeasureReport-style example that
   mirrors the JSON in FHIR-56173's description, so implementers can see
   `cqf-isEmptyList`, `cqf-isEmptyTuple`, and `cqf-cqlType` used as siblings
   of the value extension.
4. Cross-link to FHIR-56173 / PR #239 so readers know the Extensions-Pack
   change is what makes this legal.

#### Justification

This is exactly what the reporter asked for, it is purely additive
documentation in the CDS-owned IG, and it tracks an Extensions-Pack change
that has already been resolved Persuasive and merged. Without it, the
extensions are legal on `Extension` (per FHIR-56173) but the canonical "how
to encode a CQL value in FHIR" guide in CDS is silent on the pattern,
guaranteeing inconsistent implementations. Mirroring the existing
ParameterDefinition / Parameters mapping keeps the IG internally consistent
and avoids inventing a parallel vocabulary.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than treating "CQL value in Extension" as a third top-level mapping
target alongside ParameterDefinition and Parameters, refactor the mapping
section into a single normative "CQL Value Encoding in FHIR" sub-section
that defines the encoding once (CQL type → FHIR value + the
`cqf-cqlType` / `cqf-isEmptyList` / `cqf-isEmptyTuple` markers) and then
explicitly enumerates the carriers it may appear in: `ParameterDefinition`,
`Parameters.parameter`, and `Extension`. The Extension-specific guidance is
then a short "carrier rules" paragraph (the value goes on the bare
`Extension`, the markers are sibling extensions on the same parent, with the
example from Disposition A) rather than a near-duplicate mapping table.

#### Justification

The current structure already risks drift between the ParameterDefinition
and Parameters subsections; adding a third near-clone for Extension would
compound that. Factoring the encoding out once and listing carriers is more
maintainable, makes future carriers (e.g., logical models, other resources)
trivial to add, and reduces the chance that an implementer reads the
"Extension" subsection in isolation and misses a rule that is stated only in
the "ParameterDefinition" subsection. The trade-off vs. Disposition A is a
larger editorial change (more sections moved) and a slight loss of
"ctrl-F to Extension and find everything" navigability, which can be
mitigated with a clear table of carriers at the top.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for the "Using CQL With FHIR" IG: the Extensions-
Pack change (FHIR-56173, PR #239) already widens the Context of Use, and the
generic FHIR rules for extensions on extensions plus the existing CQL ↔
ParameterDefinition / Parameters mappings are sufficient for an implementer
to derive the encoding. No additional normative text is needed; if anything
is added it should live in the `cqf-supportingEvidence` extension's own
documentation (in `HL7/fhir-extensions`) rather than in this IG.

#### Justification

This avoids duplicating in CDS what FHIR Infrastructure has already
documented on the extensions themselves, keeps the "Using CQL With FHIR"
mapping section narrowly focused on parameter passing (its original
purpose), and prevents a maintenance link between two specs maintained by
two different work groups.

---

### Recommendation

**Recommended disposition:** A

The reporter is the same author as the Extensions-Pack ticket and is
explicitly asking for the CDS-side documentation that pairs with a change he
has already shepherded through FHIR Infrastructure and merged. The omission
he is pointing at is real: the IG's current mapping section talks only about
ParameterDefinition and Parameters, so an implementer building
`cqf-supportingEvidence` payloads has no canonical reference for how to
attach `cqf-isEmptyList` / `cqf-isEmptyTuple` / `cqf-cqlType` to an
extension value. Disposition A fixes that with the smallest, lowest-risk
edit and keeps wording aligned with the existing patterns. Disposition B is
arguably cleaner long-term and the workgroup may prefer it, but it expands
the scope of the change beyond what the ticket asks for and should be
considered as a follow-up editorial cleanup rather than the response to this
ticket. Disposition C understates the ergonomic cost to implementers and
should be reserved for the case where the workgroup decides this guidance
belongs entirely on the extension definitions themselves.
