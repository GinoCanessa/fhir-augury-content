## Summary

Lloyd McKenzie (on behalf of the Da Vinci CRD work) reports that CRD has
encountered a need that is not currently met by the base CDS Hooks
specification: a CRD server may simultaneously support multiple versions of the
CRD guide (which differ in what response types are allowed), and a CRD client
needs a standard way to (a) discover which CRD/CDS Hooks versions a service
supports for each hook, and (b) tell the service which version it wants to
invoke on a given call. CRD currently solves this with two IG-defined
extensions — `davinci-crd.version` on each `services[].hook` entry in the
discovery response, and `davinci-crd.requestedVersion` in the request
`extensions` object — but the request asks the CDS WG to consider whether this
is really CRD-specific, or whether a general-purpose mechanism belongs in CDS
Hooks itself.

This ticket sits squarely in the lineage of two earlier, already-Applied CDS
Hooks tickets that the comments cite explicitly:

- [FHIR-28650](https://jira.hl7.org/browse/FHIR-28650) — *Deal with STU1 and
  STU2 version differences.* Resolved Persuasive in 2025-07; STU3 added a
  recommended `version` element to each service in the discovery response so a
  service can advertise which CDS Hooks specification version it implements.
  Isaac Vetter notes in the comments that this is the element that already
  partially addresses point (1) of FHIR-54673.
- [FHIR-28684](https://jira.hl7.org/browse/FHIR-28684) — *CDS Client
  capability detection.* Resolved Persuasive with Modification in 2025-07; the
  WG decided **not** to introduce computable capability negotiation, but rather
  to rely on human-readable documentation of CDS client / service
  capabilities, on the grounds that capabilities are mostly a design-time
  concern and there are not yet discrete enough concepts to model them
  computably.

Bryn Rhodes additionally notes (comment, 2026-02-25) that any solution should
also let a server advertise its discovery endpoint via a FHIR
CapabilityStatement, and let clients advertise the hooks they support. Isaac
Vetter's analysis prefers the "decorate discovery, don't change the request
contract" direction: he argues that a single `cds-services` base URL should
ideally be a single CDS Hooks version (like FHIR), that the existing
`services[].version` element holds only one value because the design
expectation is "new versions get new services," and that dynamically switching
behavior on a per-request `requestedVersion` adds complexity and risks breaking
existing integrations. He therefore reads CRD's request-side
`requestedVersion` extension as not strictly required — a clarification in the
spec may be sufficient.

## Details

**Description:**

> In CRD, we've identified a need to be able to communicate to a payer that
> might support multiple versions of CRD (which have differing expectations
> for allowed response types) what versions they support, and for a client to
> communicate what version they wish to invoke. For now, this requirement is
> being handled with extensions. However, it seems this is likely not a
> CRD-specific requirement and might benefit from a general-purpose solution.

**Comments:**

- **isaac.vetter (2026-03-18)** — Decomposes the request into two features:
  (1) standard documentation of the CDS Hooks version a CDS Service supports —
  notes that STU3 already adds a recommended `version` element to the
  discovery endpoint; observes that today a single `cds-services` base URL can
  host services of multiple CDS Hooks versions, and that it would be cleaner
  for a base URL to always be a single version (FHIR-style); (2) a new
  mandatory request element when a discovery endpoint advertises multiple
  versions for a single service — argues this is unnecessary because the
  balloted `version` element holds only a single value, and the expectation is
  "new versions ⇒ new services" or services are compatible across listed
  versions. Recommends a succinct clarification in the spec rather than a new
  request-side element.
- **isaac.vetter (2026-03-18)** — Cites that
  [FHIR-28650](https://jira.hl7.org/browse/FHIR-28650) already added a service
  `version` element in the discovery endpoint (STU3 ballot, 2026-Jan).
- **isaac.vetter (2026-03-05)** — Documents the two CRD-defined extensions
  (`davinci-crd.version`, `davinci-crd.requestedVersion`) and the deviation
  page that describes them; argues that decorating discovery is a
  straightforward enhancement, but dynamically changing service behavior based
  on a request-side `requestedVersion` increases complexity and that the
  cleaner approach is to define new services for new CRD/CDS Hooks versions to
  preserve existing integrations.
- **isaac.vetter (2026-03-05)** —
  [FHIR-28684](https://jira.hl7.org/browse/FHIR-28684) (CDS Client capability
  detection) is related.
- **bryn.rhodes (2026-02-25)** — Adds two adjacent requirements to consider:
  (a) a way for a server to advertise its CDS Hooks discovery endpoint via a
  FHIR CapabilityStatement, and (b) a way for clients to advertise which
  hooks they support.

### [FHIR-28684: CDS Client capability detection](https://jira.hl7.org/browse/FHIR-28684)

Reporter Bas van den Heuvel asked how a CDS Service can know whether the
client supports features such as feedback and system actions. Resolved
Persuasive with Modification (2025-07-09, Applied): the WG explicitly chose
**not** to introduce computable capability negotiation, on the grounds that
client/service capabilities are primarily design-time concerns and not yet
modellable as discrete fields, and instead encouraged richer human-readable
documentation. This is directly relevant to FHIR-54673 because the same
"design-time vs runtime" tension applies to version negotiation, and any new
runtime mechanism would need to justify why version is different from
capabilities.

Reporter: Bas van den Heuvel; Commenters: isaac.vetter, lynn_laakso.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a general-purpose, two-part version-negotiation mechanism to base CDS
Hooks, modeled directly on the CRD extensions:

1. In the **discovery response**, change `services[].version` from a single
   string to also allow an array of supported CDS Hooks / hook versions per
   service (e.g., reuse the FHIR-28650 `version` slot but permit either a
   string or an array of `{ cds, hook }` objects, mirroring cds-hooks/docs
   Issue #154 and CRD's `davinci-crd.version`).
2. In the **CDS Hooks request**, add an optional top-level
   `requestedVersion` field (or a normatively-defined extension on the
   request `extensions` object) that the client uses to select among
   advertised versions. The field becomes **required** on requests sent to a
   service whose discovery entry advertises more than one version, and is
   otherwise optional. Mark the CRD extensions as deprecated in favor of the
   base mechanism on the next CRD republication.

#### Justification

This is what CRD has already implemented and exercised in the field, so the
shape of the solution is proven. Generalizing it removes a per-IG extension
duplication burden for any future CDS Hooks IG that needs multi-version
coexistence (Da Vinci has several candidates). It also directly answers
Lloyd's framing — that this is not a CRD-specific concern — and aligns with
the historical Issue #154 design proposal for per-hook version arrays.

The justification must address the gotchas Isaac raised in the comments:
adding `requestedVersion` to the request contract makes service behavior
request-dependent, which the WG has historically resisted (the FHIR-28684
resolution chose human-readable documentation over computable capability
negotiation for related reasons). Mitigation here is that `requestedVersion`
only changes behavior when the *service* opted in by advertising multiple
versions, so single-version services are unaffected and existing integrations
do not break.

---

### Disposition B: Alternative Approach (Clarify-Only — "one base URL, one version")

#### Proposal

Do **not** add a request-side `requestedVersion` to base CDS Hooks. Instead:

1. Keep the existing scalar `services[].version` element from FHIR-28650 as
   the only version-advertising mechanism in the base spec.
2. Add a **normative clarification** to the CDS Hooks specification stating
   that a single `cds-services` base URL is expected to host a single CDS
   Hooks specification version, and that servers needing to expose multiple
   versions SHOULD do so via distinct base URLs (e.g.,
   `{base}/v1/cds-services`, `{base}/v2/cds-services`) — analogous to how
   FHIR versions its endpoints.
3. Optionally (per Bryn's comment) define a CDS-Hooks-defined extension on
   FHIR `CapabilityStatement.rest.resource` (or `rest.security`/
   `implementation.custom`) that points to the discovery endpoint URL, so a
   FHIR server can advertise its CDS Hooks discovery location, and let CDS
   clients similarly publish the list of hooks they support in their own
   metadata.
4. Treat CRD's `davinci-crd.version` /`davinci-crd.requestedVersion` as a
   permissible IG-local pattern but do not promote them; document in the
   guidance that IGs which absolutely need per-request version selection MAY
   define such extensions as CRD did.

#### Justification

This is the position Isaac argued for in his 2026-03-18 and 2026-03-05
comments, and it is internally consistent with the WG's FHIR-28684 decision
to avoid runtime capability negotiation in favor of design-time / discovery
mechanisms. It minimizes the surface area of breaking changes (no new
required request field, no schema change to the existing `version` element)
and pushes the multi-version concern into deployment topology, where FHIR has
shown it is well-handled. It also addresses Bryn's CapabilityStatement /
client-hooks-advertisement requests without introducing a new runtime
contract.

The trade-off vs Disposition A is that CRD continues to carry its own
extensions, and any future IG with the same need will likely re-invent them
rather than reuse a base mechanism. The Justification accepts that cost in
exchange for keeping the base spec simple and stable.

---

### Disposition C: Decline (Already Addressed by FHIR-28650)

#### Proposal

Decline FHIR-54673 as **Not Persuasive — already addressed**: the
discovery-side need (point 1 of Isaac's decomposition) was solved by
FHIR-28650 in STU3 via the recommended `services[].version` element, and the
request-side need (point 2) is not needed because the balloted `version`
element holds a single value and the design expectation is that new versions
are surfaced as new services. CRD is free to continue using its
`davinci-crd.version` / `davinci-crd.requestedVersion` extensions for its
specific multi-version-per-hook deployment model; the base spec does not
need to encode that pattern.

#### Justification

Strict reading of the existing resolutions: FHIR-28650 is Applied, the
discovery-side `version` element is in STU3, and FHIR-28684 already
established WG precedent against runtime capability/version negotiation in
the base spec. Declining preserves consistency with both prior resolutions
and avoids re-litigating settled scope.

The downside is that it does not engage with the substantive new evidence
Lloyd is bringing — that CRD is now a working real-world example of needing
multi-version per hook — and it pushes any future IG with the same problem
to re-invent the same extensions. Declining is defensible only if the WG is
confident that CRD's situation is genuinely IG-specific and not a leading
indicator of a general CDS Hooks gap.

---

### Recommendation

**Recommended disposition:** B (Clarify-only)

Disposition B threads the needle between the two prior, already-Applied
tickets and the new evidence from CRD. FHIR-28650 already gave us the
discovery-side hook (`services[].version`); the unresolved question is
really *governance*, not mechanism — does each CDS Hooks server endpoint
correspond to one spec version, or to many? The cleanest answer, supported
by Isaac's analysis and by FHIR's own precedent, is "one endpoint, one
version," and a normative clarification plus the `CapabilityStatement`
advertisement piece Bryn raised gives implementers everything they need
without mutating the request contract or contradicting the FHIR-28684
"no computable capability negotiation" decision.

Disposition A is a credible runner-up and the WG should consider it if there
is appetite to formally adopt CRD's pattern, but it should be entered into
with eyes open about the precedent it sets — once `requestedVersion` is in
the base request, future IGs will reach for analogous request-side
negotiation knobs (capabilities, feature flags) and the WG will be back in
the FHIR-28684 debate with weaker ground to stand on.

Disposition C is too dismissive of Lloyd's point that CRD is a real
counterexample; even if the answer is "no spec change to mechanism," the
spec deserves an explicit clarification so the next IG hits the rule rather
than the gap.
