## Summary

The ticket asks the CDS Hooks (FHIR) specification to add an explicit
"callout" — a visible, unambiguous statement — of which version of FHIR
each example uses. The reporter (Graham Stigler) notes that CDS Hooks
does not itself define FHIR artifacts, so the IG-level FHIR version
metadata is not sufficient: the spec could legitimately host examples
for more than one FHIR version, and currently it appears that all
examples are R4 but this is implicit rather than declared. The
requested change is editorial/documentation in nature: every example
(or example section) should clearly state the FHIR version of the
resources it uses, alongside the existing IG-wide FHIR version
declaration.

The cross-reference index returned no linked Jira tickets, Zulip
threads, or GitHub items for FHIR-53840 (the ticket appears to be
freshly triaged with no implementation activity yet). However,
keyword-similar tickets and a long-running CDS Hooks Zulip thread show
that "what FHIR version are these examples?" has been a recurring
concern in CDS Hooks for years, which strengthens the case for an
explicit callout convention.

## Details

**Description:**

> In order to avoid confusion, we should make a clear callout of the
> version of FHIR used in examples throughout the specification. This
> callout should be in addition to the IG FHIR version, since we don't

> examples for multiple versions. Currently, it looks like all examples

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt an explicit per-example FHIR-version callout convention in the
CDS Hooks (FHIR) specification:

1. Add a normative paragraph in the spec's "Conformance" or
   "Examples" front-matter stating that every example of a CDS Hooks
   request, response, or embedded resource MUST be accompanied by a
   visible label naming the FHIR release the example is written
   against (e.g., "FHIR R4", "FHIR R5"), in addition to the IG-wide
   FHIR version declaration.
2. Establish a single rendering convention (e.g., a small badge or a
   leading line `*FHIR version:* R4` immediately above each fenced
   code block) and apply it to every existing example throughout the
   spec — at present, mark all existing examples as **R4**, matching
   the reporter's observation.
3. Document this convention in the spec's editorial / contribution
   guidance so that newly contributed examples carry the callout from
   day one.

#### Justification

The change is purely editorial and additive: it does not alter any
wire format, hook contract, prefetch semantics, or service
discovery behaviour, so it is backward-compatible by construction. It
directly addresses a confusion vector that has been raised on the CDS
Hooks Zulip stream as far back as 2018 (Shekleton, Grieve, McKenzie)
and was never resolved. It also aligns CDS Hooks with the precedent
set by FHIR-42634 in FHIR Shorthand — another non-artifact-defining
FHIR-adjacent specification — where being explicit about the FHIR
version of examples was accepted as **Persuasive with Modification**.
Because CDS Hooks does not define FHIR artifacts, the IG-level FHIR
version field is genuinely insufficient signal: a per-example callout
is the smallest change that closes the gap.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) a free-text callout, encode the FHIR
version inside the example payloads themselves and standardise a
machine-readable convention:

1. Require every embedded FHIR resource in an example to include
   `meta.profile` referencing the appropriate version-specific
   StructureDefinition canonical URL (e.g.,
   `http://hl7.org/fhir/4.0/StructureDefinition/Patient`), so the FHIR
   version is unambiguous from the payload and parseable by tooling.
2. Where examples are full CDS Hooks request/response envelopes,
   recommend (non-normative) that the example file or fence carry a
   `fhirVersion` annotation in a comment or in the surrounding prose.
3. Render the FHIR version in the published HTML by extracting it
   from `meta.profile` (or from the annotation) so the human-visible
   callout from Disposition A is generated automatically rather than
   maintained by hand.

#### Justification

This addresses the same underlying need but produces a single source
of truth: the example payload itself declares its FHIR version, and
the human-visible label is derived from it. That eliminates the
classic editorial drift problem (prose says R4, JSON is actually R5),
which is exactly the failure mode Kevin Shekleton flagged in the 2018
Zulip thread ("examples where it appears the FHIR version of the
contained resources are outdated"). It also gives downstream
implementers, validators, and CDS Hooks sandboxes a deterministic
hook to validate examples against the declared FHIR release. The
trade-off is higher up-front editorial cost (every existing example
needs its `meta.profile` set) and a small risk of cluttering examples
with implementation detail that some readers will treat as
prescriptive even though it is illustrative.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that the IG-wide FHIR version declaration
already states the version against which the specification — and by
default its examples — is written. The reporter's observation that
"all examples are R4" matches the declared IG version, so the
documentation is internally consistent without per-example callouts.
If and when CDS Hooks publishes examples for multiple FHIR versions,
the per-example callout can be introduced at that time as part of the
multi-version publication work.

#### Justification

A decline avoids editing every example block in the spec for what is
arguably a redundancy under the current single-version posture. The
SDC work group set a similar precedent in FHIR-49534, where the
proposal to extend the spec to a second FHIR version was resolved
**Not Persuasive** with the reasoning that the IG is anchored to one
release and inter-version mechanisms suffice. By analogy, CDS Hooks
could argue that the IG-level FHIR version is sufficient signal
today, and a per-example callout is premature optimisation for a
multi-version scenario that does not yet exist.

---

### Recommendation

**Recommended disposition:** A

Accepting as requested is the right call. The change is small,
purely editorial, breaks nothing on the wire, and resolves a confusion
that the CDS Hooks community has explicitly raised (Zulip 2018,
unresolved) and which the reporter has now formally re-raised. The
SDC precedent cited in Disposition C does **not** support a decline:
SDC declined to broaden the IG to a new FHIR version, which is a
much larger change than what FHIR-53840 actually asks for —
FHIR-53840 only asks for a *label*, not for multi-version content.
The closer precedent is FHIR-42634 (FHIR Shorthand), which accepted
exactly this kind of "be explicit about the FHIR version of your
examples" change. Disposition B (machine-readable encoding via
`meta.profile`) is attractive and would be a worthwhile follow-up,
but it adds editorial burden and risks reading prescriptive intent
into illustrative payloads; it is best raised as a separate ticket
once the human-visible callout from A is in place. The work group
should adopt A now, and consider B as a follow-on tracked separately.
