## Summary

The reporter (Celine Lefebvre) requests a normative-keyword change in the CDS Hooks specification's "Hook Definition Format" guidance. Specifically, she calls out the sentence in the *Workflow* sub-section of a hook's definition template — "Describe when this hook occurs in a workflow. Hook creators SHOULD include as much detail and clarity as possible to minimize any ambiguity or confusion among implementers." — and asks that the "SHOULD" be tightened to "SHALL". The intent is to make it a mandatory authoring requirement for hook definitions to spell out the workflow context fully, eliminating ambiguity for EHR/CDS service implementers consuming new hooks. No cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced by FhirAugury for this ticket key, so the request stands on its own without recorded community discussion to anchor it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `cds-hooks/docs`, edit `docs/specification/current.md` (around line 1168, *Hook Definition Format → Workflow*) to change:

> Describe when this hook occurs in a workflow. Hook creators **SHOULD** include as much detail and clarity as possible to minimize any ambiguity or confusion among implementers.

to:

> Describe when this hook occurs in a workflow. Hook creators **SHALL** include as much detail and clarity as possible to minimize any ambiguity or confusion among implementers.

No other text changes; the surrounding "Hook Definition Format" template guidance is unaffected. Apply the same change to any in-flight branch of the spec (`current.md` and any `1.0`/`2.0` versioned copies if present) so the authoring rule is consistent across editions. Persist the change through the normal CDS Hooks publication pipeline (the `cds-hooks/docs` repo is the authoring source for the published spec at `cds-hooks.hl7.org`; no separate generated artifact needs hand-editing).

#### Justification

The sentence is *authoring guidance to hook creators*, not a runtime conformance rule on EHRs or CDS Services, so promoting "SHOULD" to "SHALL" raises the bar on what a hook proposal must contain without imposing any new burden on implementers of existing hooks. The Workflow description is the single most important piece of context an EHR vendor uses to decide *when* to invoke a hook; ambiguity there directly causes interoperability defects that downstream tickets then have to chase. Making it normatively required for hook definitions to describe workflow fully is consistent with how CDS Hooks already uses SHALL elsewhere in the same section (e.g., reverse-domain naming for organization-specific hooks).

---

### Disposition B: Alternative Approach

#### Proposal

Keep the keyword as `SHOULD` but expand the guidance to enumerate the minimum content a Workflow description must include, and add a SHALL-level checklist immediately under the sentence. For example, in `cds-hooks/docs/docs/specification/current.md` around line 1168, replace the single sentence with:

> Describe when this hook occurs in a workflow. Hook creators SHOULD include as much detail and clarity as possible to minimize any ambiguity or confusion among implementers. At a minimum, the Workflow description SHALL identify (a) the user-facing action or system event that triggers the hook, (b) the point in the EHR workflow at which the hook fires (before/after the triggering action), and (c) any preconditions on the EHR state (e.g., a chart open, an order signed but not yet committed).

This keeps the aspirational "as much detail as possible" wording (which is inherently subjective and therefore not testable as SHALL) while introducing testable SHALL-level requirements on the *specific* facts a Workflow description must cover.

#### Justification

"As much detail and clarity as possible" is unmeasurable — it cannot be conformance-tested, so promoting it to SHALL creates a normative requirement no reviewer can objectively check. The alternative addresses the underlying problem the reporter is trying to solve (vague Workflow descriptions causing implementer confusion) by attaching the SHALL to a concrete, enumerable set of facts that *can* be checked during hook intake review. It also provides hook authors with a usable template, reducing review-cycle churn.

---

### Disposition C: Decline

#### Proposal

Leave the text as `SHOULD`. Decline on the basis that the sentence is non-testable aspirational guidance ("as much detail and clarity as possible") and that promoting non-testable language to SHALL is contrary to RFC 2119 / HL7 conformance-language conventions. Note in the resolution that hook authors are already expected to provide a useful Workflow description as part of the standard hook proposal/intake review, which is the appropriate venue for enforcing thoroughness.

#### Justification

RFC 2119's SHALL/MUST is reserved for absolute requirements that can be unambiguously tested for conformance. "Include as much detail and clarity as possible" is by construction unbounded and unmeasurable, so changing the keyword does not change behavior — it only makes the spec internally inconsistent with how it uses normative keywords elsewhere. The real lever for raising hook-description quality is the CDS Hooks intake/review process, not the keyword. Declining preserves the spec's conformance language hygiene and avoids a change that cannot be enforced.

---

### Recommendation

**Recommended disposition:** B

Disposition A grants the reporter's literal request, but it attaches SHALL to an unmeasurable phrase ("as much detail and clarity as possible"), which weakens the spec's overall conformance language and gives reviewers no objective bar to apply. Disposition C is technically the cleanest from a conformance-language standpoint, but it leaves the underlying problem — vague Workflow descriptions causing implementer confusion — entirely to out-of-band review process, which the reporter clearly feels is not sufficient. Disposition B threads both needles: it preserves the aspirational "should be detailed" framing where SHALL would be untestable, while introducing concrete, checkable SHALL-level requirements (trigger, workflow position, preconditions) that match exactly the facts implementers complain are missing. It directly addresses the reporter's concern, gives hook-intake reviewers a checklist to enforce, and keeps CDS Hooks' use of normative keywords consistent with RFC 2119. The Clinical Decision Support work group should treat this as the starting point and refine the enumerated minimum-content list during discussion.
