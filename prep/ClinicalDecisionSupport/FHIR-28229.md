## Summary

The reporter (Bryn Rhodes) asks the Clinical Decision Support work group, in
the context of the **Clinical Guidelines (FHIR)** specification (a.k.a.
CPG-on-FHIR), to *consider* introducing two new characterizations of
recommendation/plan artifacts:

1. **Enablement / usability** — a way to characterize how readily an artifact
   can be implemented and used by an end consumer (e.g., a recommendation that
   only requires a printable patient handout vs. one requiring full CDS Hooks
   integration with an EHR).
2. **Impact / outcome** — a way to characterize the clinical impact or
   intervention "weight" of an artifact, of which the suggested label
   *"High Touch"* is an example (i.e., low-burden educational nudge vs.
   high-burden clinical workflow change).

The ticket is exploratory — the description does not yet propose a concrete
mechanism (extension, code system, structural element). It was **resolved
"Considered for Future Use" on 2024-07-10 (vote 20-0-0, Greg White / Juliet
K Rubini)** with the note "Will pursue after STU2 publication", then
**re-opened on 2024-11-26 by lynn_laakso** so the work group could revisit it
post-STU2.

There are no cross-referenced Jira tickets, Zulip threads, GitHub items, or
Confluence pages recorded for this key in FhirAugury (the Confluence source

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Introduce two new, optional knowledge-artifact-level characterizations in the
Clinical Guidelines (FHIR) IG, applied to `PlanDefinition` (and, where
appropriate, `ActivityDefinition` / `ActivityDefinition`-derived
recommendation artifacts):

1. **`cpg-enablement-rating`** extension on `PlanDefinition` (and on
   `ActivityDefinition`) carrying a `CodeableConcept` bound to a new
   `cpg-enablement-rating` CodeSystem/ValueSet (example codes: `narrative`,
   `data-only`, `decision-support`, `workflow-integrated`) describing the
   level of implementation effort required to enable the artifact in a
   consuming system. This aligns conceptually with the existing CPG
   knowledge-level pyramid (L1 narrative → L4 executable).
2. **`cpg-impact-rating`** extension carrying a `CodeableConcept` bound to a
   new `cpg-impact-rating` CodeSystem/ValueSet (example codes: `informational`,
   `low-touch`, `moderate-touch`, `high-touch`) describing the expected
   clinical/operational footprint of acting on the recommendation.

Both extensions would live in the CPG-Common profile pack alongside existing
characterization extensions, with cardinality `0..1`, and would be referenced
from the IG narrative pages that describe recommendation metadata.

#### Justification

This is what the ticket literally asks for, and the CPG IG already
establishes a pattern of layering optional metadata characterizations on
`PlanDefinition`/`ActivityDefinition` (e.g., strength of recommendation,
quality of evidence, intent, knowledge capability levels). Adding two more
small, optional, code-bound extensions is consistent with that pattern,
keeps the change additive and non-breaking, and gives knowledge-artifact
authors a standardized vocabulary for two distinctions that implementers
already make informally ("how hard is this to deploy?" and "how invasive is
this for the clinician/patient?").

---

### Disposition B: Alternative Approach

#### Proposal

Rather than minting two new bespoke extensions, **reuse and slightly extend
the IG's existing characterization machinery**:

- For **enablement**, reuse the IG's existing **knowledge-artifact capability
  level** (L1 Narrative → L2 Semi-structured → L3 Computable → L4
  Executable). Document explicitly in the IG narrative that this level
  *is* the enablement/usability characterization, and clarify how
  implementers should populate it. If gaps exist, fill them by extending
  the existing CodeSystem rather than introducing a parallel one.
- For **impact**, model "High Touch" / clinical footprint as a
  **`UsageContext`** entry on `PlanDefinition`, with a new (or
  extended) UsageContext-type code (`workflow-impact`) and a small CPG
  ValueSet of impact magnitudes (`informational`, `low-touch`,
  `moderate-touch`, `high-touch`). `UsageContext` is already designed to
  carry artifact-applicability characterizations and is filterable by
  registries today.

Pair this with a single IG narrative page titled something like
*"Characterizing knowledge artifacts — enablement and impact"* that points
implementers at both mechanisms.

#### Justification

CPG already invests heavily in L1–L4 capability levels and in
`UsageContext`-driven discoverability; adding two new parallel extensions
risks fragmenting the metadata story and forcing tooling to look in three
places for very similar information. Reusing existing slots is
backward-compatible with current authoring tooling, costs less to publish,
and avoids the "yet another extension" perception when the WG re-opens the
ticket post-STU2. It also lets the WG ship the conceptual clarification (the
narrative page) quickly, even if the underlying value sets need more
iteration.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive — already addressed by existing mechanisms*.
Specifically: enablement is already expressed by the CPG knowledge-artifact
**capability level** (L1–L4); clinical impact / "high touch" is
implementation-specific judgment that varies by deploying organization, care
setting, and patient population, and is not stable enough to standardize as
a coded characterization at the IG level. Encourage the original requestor
to capture this as **published guidance / implementation notes** rather
than a normative artifact characterization.

#### Justification

The ticket has sat for over four years (2020 → 2024) with only an
exploratory description and was previously voted *Considered for Future
Use*. There is no documented use case, no implementer demand visible in
Zulip or the CPG GitHub repos, and no proposed value set. "High Touch" in
particular is highly contextual — what is high-touch for a small ambulatory
clinic is routine for a large IDN — which makes a normative coded
characterization brittle. Standardizing it risks freezing a judgment call
into the spec without sufficient real-world feedback.

---

### Recommendation

**Recommended disposition: B**

The work group clearly wants to do *something* here — the ticket was
deliberately re-opened after the STU2 vote rather than being left as
"Considered for Future Use" — so a flat decline (C) does not match the
expressed intent. At the same time, jumping straight to two brand-new
extensions and two brand-new code systems (A) is heavier than the ticket
warrants given that no concrete proposal, use case, or vocabulary has been
contributed in four years, and the CPG IG already has well-known mechanisms
(capability levels, `UsageContext`, GRADE-style strength/quality) that
overlap substantially with what is being asked for. Disposition B threads
that needle: it commits the WG to publishing a clear narrative
characterization page (cheap, fast, low-risk), it formalizes "High Touch"
as a `UsageContext` value (additive and non-breaking), and it leaves the
door open to promote the ValueSet to a dedicated extension in a later cycle
if implementer feedback warrants it. That is a defensible starting point
for the post-STU2 discussion this ticket was re-opened to support.
