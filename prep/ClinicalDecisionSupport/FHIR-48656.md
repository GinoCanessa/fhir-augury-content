## Summary

The ticket asks the Clinical Decision Support work group to align the
Clinical Guidelines IG (CPG) `PlanDefinition/$apply` OperationDefinition
with the Canonical Resource Management Infrastructure (CRMI) Operation
Pattern Profiles, specifically: **Artifact**, **Artifact Version
Bindable**, **Manifestable**, **Parameterizable**, **Data Configurable**,
and **Artifact Endpoint Configurable**. The request is a continuation of
the broader CRMI-alignment program already underway in CDS (see
[FHIR-42973](https://jira.hl7.org/browse/FHIR-42973) "Refactor to use
CRMI profiles", resolved Persuasive/Published) and is part of a coherent
set of three sibling tickets filed together on 2024-10-16:
[FHIR-48654](https://jira.hl7.org/browse/FHIR-48654) (define the missing
Data Configurable / Parameterizable pattern profiles in CRMI),
[FHIR-48655](https://jira.hl7.org/browse/FHIR-48655) (apply the same
pattern profiles to the CQL `$cql` and Library `$evaluate` operations),
and [FHIR-48657](https://jira.hl7.org/browse/FHIR-48657) (add the
parameterization and data-configuration parameters that enable the
conformance to the **base** R6 PlanDefinition `$apply`). The ticket also
asks the IG to document that the CPG `$apply` is consistent with the R6
PlanDefinition `$apply` semantics being introduced in parallel — the
semantics-alignment companion work is tracked by
[FHIR-50004](https://jira.hl7.org/browse/FHIR-50004) (Applied/Persuasive)
and [FHIR-46314](https://jira.hl7.org/browse/FHIR-46314)
(Applied/Persuasive). Net effect: the CPG `$apply` operation should
formally claim conformance to the listed CRMI operation pattern profiles
and the IG narrative should explicitly cross-reference the R6
PlanDefinition `$apply` so implementers can reason about the two
operations as a single semantic family.

## Details

**Description:**

> The PlanDefinition $apply operation should conform to the CRMI Operation Pattern Profiles:
>
> - Artifact
> - Artifact Version Bindable
> - Manifestable
> - Parameterizable
> - Data Configurable
> - Artifact Endpoint Configurable
>
> Also document that the CPG PlanDefinition $apply is consistent with the semantics for the R6 PlanDefinition apply, and that these changes are also being made in the R6 specification

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the Clinical Guidelines (CPG) IG, update the `OperationDefinition`
for `cpg-plandefinition-apply` so it formally declares conformance
(via `OperationDefinition.profile` / IG narrative as appropriate) to
all six CRMI Operation Pattern Profiles named in the ticket:

1. CRMI Artifact Operation
2. CRMI Artifact Version Bindable Operation
3. CRMI Manifestable Operation
4. CRMI Parameterizable Operation
5. CRMI Data Configurable Operation
6. CRMI Artifact Endpoint Configurable Operation

This entails (a) adding the parameters required by each pattern profile
that the operation does not already expose (the same set being added to
base R6 `PlanDefinition/$apply` under [FHIR-48657](https://jira.hl7.org/browse/FHIR-48657));
(b) tightening parameter cardinalities/types to match the pattern
profile constraints; and (c) adding an IG narrative section on the CPG
`$apply` page that states the operation is consistent with R6
`PlanDefinition/$apply` and points at the R6 OperationDefinition (using
the same narrative pattern adopted under
[FHIR-46314](https://jira.hl7.org/browse/FHIR-46314) for `$apply`
clarifications). Mark this ticket as gated on
[FHIR-48654](https://jira.hl7.org/browse/FHIR-48654) (which must
publish the *Data Configurable* and *Parameterizable* pattern profiles
in CRMI before this conformance claim can be made).

#### Justification

This is the CDS WG's stated direction for CPG (see
[FHIR-42973](https://jira.hl7.org/browse/FHIR-42973), Persuasive /
Published), and other IGs are doing the same on the resource side
([FHIR-52769](https://jira.hl7.org/browse/FHIR-52769)). The conformance
claim is what gives implementers a machine-checkable contract for
parameterization, data sourcing, and artifact-version binding when
invoking `$apply` — the same contract that
[FHIR-50004](https://jira.hl7.org/browse/FHIR-50004) is bringing into
the R6 base. Doing this in CPG keeps the CPG and R6 operations
interchangeable from an implementer perspective and prevents drift.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than declare conformance to each pattern profile inside the CPG
IG today, **defer the formal conformance claim to the R6 ballot cycle**
and, in the interim, ship a CPG IG narrative-only update that:

1. Lists the six CRMI pattern profiles the operation is *intended* to
   conform to;
2. Documents which parameters of `cpg-plandefinition-apply` correspond
   to each pattern profile element;
3. Cross-links to [FHIR-48657](https://jira.hl7.org/browse/FHIR-48657)
   and the in-progress R6 `PlanDefinition/$apply` so readers can see
   the alignment plan; and
4. Lands the actual `OperationDefinition.profile` references in CPG
   only **after** [FHIR-48654](https://jira.hl7.org/browse/FHIR-48654)
   publishes the *Data Configurable* and *Parameterizable* profiles in
   CRMI and [FHIR-48657](https://jira.hl7.org/browse/FHIR-48657) lands
   the matching parameters in R6.

#### Justification

Two of the six named pattern profiles (*Data Configurable*,
*Parameterizable*) do not yet exist in CRMI — they are roadmap items
under FHIR-48654. Declaring conformance to profiles that are not yet
published is fragile and creates a build-time dependency on an
unreleased CRMI version. Sequencing the conformance claim behind the
prerequisite work avoids that and lets CPG ship the
documentation-of-intent now, which already delivers most of the
implementer value. It also removes the risk that the parameter shapes
in the eventual CRMI profiles diverge from what CPG ends up exposing.

---

### Disposition C: Decline

#### Proposal

Decline to declare formal conformance of CPG `cpg-plandefinition-apply`
to the CRMI Operation Pattern Profiles. Rationale: the CPG `$apply`
operation is deliberately specialized (it always returns a
`RequestOrchestration` of `option`-intent proposals — see
[FHIR-46314](https://jira.hl7.org/browse/FHIR-46314)), and stacking six
CRMI pattern-profile claims on top of it adds conformance surface
without changing observable behavior for CPG implementers. Treat the
CRMI pattern profiles as guidance the operation already follows in
spirit and rely on the R6 base `PlanDefinition/$apply` work
([FHIR-48657](https://jira.hl7.org/browse/FHIR-48657),
[FHIR-50004](https://jira.hl7.org/browse/FHIR-50004)) to carry the
formal conformance.

#### Justification

If the R6 base operation conforms to the pattern profiles and CPG
`$apply` is documented as semantically consistent with R6 `$apply`
(per [FHIR-46314](https://jira.hl7.org/browse/FHIR-46314) and
[FHIR-50004](https://jira.hl7.org/browse/FHIR-50004)), implementers
already get the contract transitively. Adding a second conformance
claim risks the two operations drifting out of lock-step at every
future CRMI revision and forces CPG into the CRMI release cadence.
This is the most defensible "do less" path if the WG concludes the
narrative-consistency statement is enough.

---

### Recommendation

**Recommended disposition:** B

CRMI-alignment is clearly the WG's direction (FHIR-42973 is already
Persuasive/Published and FHIR-52769 extends the pattern to downstream
IGs), so Disposition C — declining outright — would walk away from
work the WG has already committed to and would leave CPG `$apply` as
the only operation in the CDS family that doesn't carry the
conformance claim. But Disposition A as a single step is over-eager:
two of the six pattern profiles named in the ticket
(*Data Configurable*, *Parameterizable*) don't yet exist in CRMI
([FHIR-48654](https://jira.hl7.org/browse/FHIR-48654)) and the
parameters needed to honor the conformance are still being added to
the base operation in [FHIR-48657](https://jira.hl7.org/browse/FHIR-48657).
Disposition B captures the WG's intent now (the documentation-of-intent
and the R6-consistency narrative the ticket explicitly asks for), lets
implementers see the alignment plan, and sequences the formal
`OperationDefinition.profile` declaration behind its real
prerequisites. It is the lowest-risk path that still delivers what the
reporter asked for, and it explicitly addresses the gating
relationships that surfaced from the related-tickets analysis.
