## Summary

The reporter requests that the CDS Hooks specification be enhanced to
explicitly support **pseudonymized user identity** in the hook `context.user`
field and in associated security tokens, in order to reduce exposure of
clinical-user PII to externally hosted CDS Services and to better align with
data-minimization regimes (GDPR, PIPL, HIPAA). The proposal has three parts:
(1) allow the `user` field to be either a real `Practitioner`/`PractitionerRole`
reference **or** an opaque pseudonymized identifier scoped per EHR–CDS
relationship; (2) add optional "user attribute" fields (role, specialty,
organization, permissions) so CDS logic can act on role/workflow without an
identifying reference; and (3) clarify that JWT `sub` MAY be a pseudonymized
value and that CDS Services MUST NOT assume `sub` or `context.user` is
dereferenceable to a FHIR resource unless the EHR documents otherwise. The
reporter argues this is fully backward compatible and reflects zero-trust /
least-privilege design. CDS WG (Isaac Vetter) has proposed **Not Persuasive**
on the grounds that the current spec already permits unresolvable references
and that `PractitionerRole` already conveys role-type information without
exposing the underlying `Practitioner` identity. A follow-up comment from
Alan Zhang (JIRAUSER30404) partly agrees but suggests the CDS Hooks IG could
still provide guidance on how individual hook specs apply pseudonymization,
noting that the same concern extends to `Patient`, `RelatedPerson`, and to
prefetch FHIR Resources delivered to the CDS Service. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were found by the tooling.

## Details

**Description:**

The current CDS Hooks specification requires the `user` field in hook
context to contain a direct FHIR reference such as `Practitioner/{id}` or
`Patient/{id}`. While this enables the CDS Service to understand the actor
involved in the workflow event, this design mandates disclosure of personally
identifiable information (PII) about clinical users to external CDS Services,
which may be hosted outside the healthcare organization's trust boundary.

Given the increasing regulatory emphasis on data minimization (e.g., GDPR,
PIPL, HIPAA) and the growing use of cloud-hosted or vendor-operated CDS
Services, this mandatory exposure of user identity introduces avoidable
privacy and compliance risks.

To address this, the reporter proposes enhancing the specification to allow
**pseudonymized user identifiers** in the hook context, while maintaining
full compatibility with existing CDS workflows and audit requirements.

**Proposed Change**

1. **Modify the definition of the `user` field** to allow either:
   - an actual FHIR reference to the user resource, *or*
   - a **pseudonymized opaque identifier**, unique per EHR–CDS Service
     relationship, not resolvable outside the EHR environment.
2. **Add optional user-attribute fields** that allow the EHR to supply
   relevant non-identifying information (e.g., role, specialty, organization,
   permissions) when needed for CDS logic, without requiring user identity
   exposure.
3. **Clarify how pseudonymized identifiers interact with security tokens**:
   - JWT claims MAY include a pseudonymized subject (`sub`) value when
     compliant with local policy.
   - CDS Services **must not** assume that `sub` or `context.user` can be
     dereferenced to a FHIR resource unless explicitly documented by the EHR.
4. **Add privacy guidance** encouraging implementers to use the
   pseudonymization option when clinically feasible, especially for
   cross-organizational or cloud-hosted CDS integrations.

**Rationale**

- The identity of the practitioner is not clinically required in most CDS
  logic; what is usually needed is the user's *role* or *workflow position*,
  not the actual identity.
- Pseudonymization reduces privacy and compliance risks while preserving
  functional equivalence for CDS purposes.
- Aligns with contemporary security models (zero-trust, least-privilege)
  and ongoing HL7 work on tokenization and prefetch.
- Supports deployment in jurisdictions with strict data protection rules.

**Benefits**

- Stronger privacy guarantees for clinical users.
- Broader adoption of CDS Hooks in regulated markets.
- Improved architectural flexibility without structural changes to existing
  hook definitions.
- Fully backward compatible: existing systems may continue to use real
  identifiers.

**Resolution (proposed by CDS WG):** Not Persuasive — capability being
requested by the author already exists. See details in comments.

### isaac.vetter — 2026-03-18

Big picture — what the reporter is asking for is already entirely possible.
Pseudonymized user identifiers are already allowed because the FHIR
`Practitioner`/`PractitionerRole` reference is not required to be resolvable.
Proposes Not Persuasive because the capability already exists. Notes:

- The user field in hook context being a `Practitioner`/`PractitionerRole`
  reference is set by the individual hook specs (e.g., `order-select`), and
  those references need not be resolvable — even within the EHR. Resolvability
  is governed by OAuth scopes the EHR grants to the CDS Service and by whether
  the EHR exposes a FHIR server at all.
- The CDS Hooks JWT does not currently specify a `sub` claim
  (https://build.fhir.org/ig/HL7/cds-hooks/en/index.html#trusting-cds-clients).
- `PractitionerRole` is exactly the resource type intended to carry role-type
  information; whether it links to a `Practitioner` (and thus exposes
  identifying info) is in the EHR's control.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Amend CDS Hooks (and the hook-definition templates used by individual hooks
such as `order-select`, `order-sign`, `patient-view`, `appointment-book`)
so that:

1. The `context.user` field's data type is broadened from "FHIR reference to
   `Practitioner` or `PractitionerRole`" to a choice of (a) such a FHIR
   reference, or (b) a string containing an **opaque pseudonymized
   identifier** scoped to the EHR–CDS Service relationship and explicitly
   declared non-dereferenceable.
2. A new optional `context.userAttributes` (or equivalent) object is added
   to the hook context schema, with named fields for `role`, `specialty`,
   `organization`, and `permissions` — populated by the EHR when the CDS
   Service needs role/workflow signals but not identity. The same pattern
   is offered for `patient`/`relatedPerson` references called out by Alan
   Zhang.
3. The "Trusting CDS Clients" section of the IG is amended to state that
   JWT `sub`, when present, MAY be a pseudonymized value and that CDS
   Services MUST NOT assume `sub` or `context.user` can be dereferenced to
   a FHIR resource unless the EHR documents otherwise.
4. A new "Privacy and Pseudonymization" section is added to the IG narrative,
   recommending pseudonymization for cross-organizational and cloud-hosted
   CDS Services, with worked examples and implementer guidance.

#### Justification

This squarely addresses the reporter's data-minimization goal and aligns
CDS Hooks with current zero-trust / least-privilege security models and with
GDPR/PIPL/HIPAA-style data-minimization requirements. It is presented as
backward compatible (existing systems may continue to use real identifiers).
Alan Zhang's comment supports the value of explicit IG guidance on applying
pseudonymization to individual hooks and to prefetched FHIR Resources, which
is what this disposition delivers.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** change the data type of `context.user` or add a new
`userAttributes` object. Instead, add a new normative section to the CDS
Hooks IG titled "Pseudonymization and User Privacy" that:

1. Explicitly clarifies — citing the existing semantics — that `context.user`
   is a FHIR reference whose target is **not required to be resolvable**, and
   that EHRs MAY supply opaque, locally-scoped IDs in the reference (e.g.,
   `Practitioner/opaque-abc123`) where a real identifier would otherwise leak
   PII. Similar wording applies to `Patient`, `RelatedPerson`, and prefetched
   resources.
2. Recommends `PractitionerRole` as the preferred reference target when the
   CDS Service needs role/specialty/organization signals without identity,
   and notes that the EHR controls whether the `PractitionerRole.practitioner`
   chain is populated or omitted.
3. Adds non-normative guidance on JWT `sub`: when implementations introduce
   `sub`, it MAY be pseudonymized, and CDS Services MUST NOT dereference it
   without prior agreement.
4. Adds an implementer note that prefetch responses can be pseudonymized
   (e.g., `Patient.identifier`, names, addresses removed or replaced) when
   the workflow does not require identifying data.

#### Justification

This addresses the underlying privacy concern without introducing a new
context shape that every CDS Service implementation must learn to parse,
and without changing the contract of currently-defined hooks. As Isaac
Vetter notes, FHIR references in `context.user` already need not be
resolvable, so the capability the reporter wants exists today; the gap is
that this is not *visible* in the IG. A focused IG section captures the
intent, gives implementers a recognized pseudonymization pattern, and
satisfies Alan Zhang's request that the CDS Hooks IG actively guide how
individual hooks should treat pseudonymization — all without the
backward-compatibility risk and per-hook editing cost of a structural
change.

---

### Disposition C: Decline (Not Persuasive — author's request already met)

#### Proposal

Resolve as **Not Persuasive**, on the basis that the requested capability is
already supported by the current specification:

- `context.user` is a FHIR reference to `Practitioner`/`PractitionerRole`,
  and these references are not required to be resolvable, in or out of the
  EHR boundary. EHRs may therefore supply opaque IDs today.
- `PractitionerRole` is the FHIR resource designed to convey role-type
  information; whether it links to a `Practitioner` (and so exposes
  identifying data) is at the EHR's discretion.
- The CDS Hooks JWT does not currently define a `sub` claim, so there is no
  spec text mandating that any `sub` value be a real subject identifier.
- Backward-compatible enhancement of `context` to add optional
  user-attribute fields would be reconsidered if implementers demonstrate
  concrete CDS use cases that cannot be served by `PractitionerRole` plus
  unresolvable references.

This is the disposition Isaac Vetter has already proposed and that the
reporter is being asked to confirm by removing the in-person tag.

#### Justification

Per the CDS WG response, no spec change is required to permit pseudonymized
user identities in CDS Hooks: the existing FHIR-reference contract does not
mandate resolvability, and `PractitionerRole` covers the role-type need.
Adding a parallel "opaque ID" data type and a new `userAttributes` object
would split the spec into two ways of saying the same thing, multiply the
combinations CDS Service implementations must validate, and complicate every
existing hook definition. Alan Zhang's follow-up — that IG-level guidance
would still be useful — is a real point but does not by itself justify
Accepting this ticket as written; it could be picked up as a separate
documentation/IG ticket without changing the hook context schema.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach)

Disposition C is technically correct — Isaac Vetter is right that the
current spec already permits unresolvable references and that
`PractitionerRole` already provides role-type information without leaking
identity — but resolving as Not Persuasive leaves the CDS Hooks IG silent
on a question that is clearly being asked by implementers in regulated
markets, and Alan Zhang's comment explicitly asks for IG guidance even
while agreeing with the technical analysis. Disposition A is heavier than
necessary: it changes the data type of `context.user`, adds a new
`userAttributes` object that overlaps with `PractitionerRole`, and ripples
into every hook definition for a need the existing reference shape can
already meet. Disposition B captures the value the reporter is after —
explicit, discoverable, normative guidance on pseudonymization in
`context.user`, prefetched resources, and any future JWT `sub` — without
breaking the current hook contract or duplicating `PractitionerRole`'s
purpose. It also sets up a clean home for the broader pseudonymization
guidance Alan Zhang asked for (Patient, RelatedPerson, prefetch). The CDS
WG should consider re-categorizing this ticket to "Persuasive with
Modification" along these lines, or splitting off a sibling ticket against
the IG narrative.
