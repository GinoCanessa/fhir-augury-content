## Summary

The reporter (Brian Kaney) requests that a new code, `prompt-library`, be
added to the value set bound to `Library.type` so that FHIR `Library`
resources can formally represent collections of generative-AI / LLM
prompts as a recognised library category. The request is filed against
the Canonical Resource Management Infrastructure (CRMI) specification
under the Clinical Decision Support work group, on the basis that
`Library` is the canonical wrapper for computable knowledge artifacts
(logic-library, model-definition, asset-collection, etc.). The reporter
points to a gist that demonstrates an example `Library` resource shaped
as a "prompt library", and to the `reason-healthcare/fhir-gpt`
repository as a reference implementation that uses this pattern. No
prior community discussion (Zulip), related Jira tickets, or related
GitHub items were surfaced through the FHIR Augury cross-reference
index — only the artifact mention `Library.type` itself was found.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new code `prompt-library` to the value set bound to
`Library.type` (`http://hl7.org/fhir/ValueSet/library-type`, backed by
`http://terminology.hl7.org/CodeSystem/library-type`), with a
definition along the lines of:

> A collection of prompts (e.g., for generative AI / large language
> model interactions) that are intended to be invoked or composed by a
> consuming system. The Library's `content` (or referenced
> `Attachment`s) carry the prompt text and any associated metadata
> required to use the prompts.

Because `Library.type` is bound on the core resource (not in CRMI), the
core code system / value set would need the new concept and an updated
binding-version note. CRMI would then publish an accompanying narrative
describing how a `Library` of `prompt-library` type is expected to be
authored (recommended `useContext`, `relatedArtifact`, `parameter` and
`dataRequirement` patterns), mirroring the way it documents
`logic-library` today. No structural change to the `Library` resource is
proposed — only a new code in the existing extensible value set plus
CRMI guidance.

#### Justification

`Library.type` is explicitly extensible and was designed to grow as new
classes of computable knowledge emerge (it already accommodates
`logic-library`, `model-definition`, `asset-collection`, and
`module-definition`). Prompt collections for generative-AI workflows
are a clearly distinct class of computable knowledge that the existing
codes do not describe well: they are neither executable logic
(`logic-library`), nor statistical/ML model definitions
(`model-definition`), nor undifferentiated asset bundles
(`asset-collection`). Adding the code at the core level — rather than
asking every implementer to mint a local code — improves
interoperability and discoverability, supports CRMI tooling that filters
libraries by `type`, and aligns with the reporter's working
implementation pattern in `reason-healthcare/fhir-gpt`.

---

### Disposition B: Alternative Approach

#### Proposal

Do not modify the core `library-type` value set. Instead, address the
need entirely within CRMI by:

1. Defining a CRMI-published code system / value set (e.g.,
   `http://hl7.org/fhir/uv/crmi/CodeSystem/crmi-library-type`)
   containing the new concept `prompt-library`, with a clear
   definition.
2. Publishing a CRMI `LibraryPromptLibrary` profile on `Library` that:
   - Slices `Library.type` so the profile-level binding includes the
     CRMI-defined `prompt-library` code (leveraging the extensible
     binding on the base resource);
   - Documents required/expected `relatedArtifact`, `useContext`,
     `parameter`, and `content`/`Attachment` shapes for prompt payloads
     (including model targeting, prompt language, and versioning).
3. Optionally requesting that the new code be promoted into the core
   `library-type` value set in a later FHIR release once usage and
   feedback have stabilised.

#### Justification

The base `Library.type` binding is extensible, which means implementers
are *already permitted* to use codes from other systems when no core
code applies — so the immediate functional need can be met without a
core normative-track change. Keeping the new concept inside CRMI lets
the work group iterate on the definition, the profile, and the
companion guidance (which is the part that will actually drive
interoperability) without being gated by core release cycles, and
avoids minting a core code before the community has converged on what
"prompt library" should structurally mean (e.g., are prompts
`Attachment`s? `parameter`s? referenced `Binary`s? Is model targeting
in scope?). The trade-off vs. Disposition A is reduced
out-of-the-box discoverability — clients filtering on the core value
set won't see `prompt-library` until/unless it is later promoted — but
this is mitigated by the extensible binding and by CRMI tooling that
already understands CRMI-defined types.

---

### Disposition C: Decline

#### Proposal

Decline the change request. Recommend that implementers continue to use
the existing `Library.type` extensible binding with a locally-defined
code (or with `asset-collection` plus a discriminating
`relatedArtifact`/`useContext`) until the FHIR community has
accumulated enough deployed experience with prompt-library use cases
to justify a standardised code. Close the ticket as
"Persuasive with Mod / Not Persuaded — premature standardisation",
inviting the reporter to refile once a CRMI profile or IG has
demonstrated stable usage.

#### Justification

The request is supported by a single example gist and a single
reference repository (`reason-healthcare/fhir-gpt`), with no surfaced
Zulip discussion, no linked tickets, and no cross-implementation
demand. Generative-AI prompt management is a fast-moving area where
the very definition of "prompt library" (chat-completion templates?
parameterised prompts? agent tool descriptions? RAG context
collections?) is not yet stable. Adding a core code now risks locking
in a meaning that the community will outgrow within a release or two,
or that conflicts with terminology adopted elsewhere (e.g., a future
HL7 AI/ML accelerator). Because `Library.type` is extensibly bound,
nothing in the spec prevents the reporter from using the pattern today
under a local code — declining costs no functionality, only the
HL7-blessed label.

---

### Recommendation

**Recommended disposition:** B

Disposition B (CRMI-scoped code + profile) is the strongest balance of
the three. The underlying use case is real and well-formed — a prompt
collection genuinely is a kind of computable knowledge artifact and
`Library` is the right resource for it — so a flat decline
(Disposition C) understates the request. At the same time, the
*shape* of a prompt library (model targeting, parameter binding,
prompt format, versioning, evaluation metadata) is not yet
community-tested, and the ticket itself rests on a single reference
implementation with no surfaced Zulip or cross-ticket discussion. That
makes a core code addition (Disposition A) premature: once a code is
in the core value set, the structural expectations behind it become
hard to refine without breaking implementers. CRMI is precisely the
specification designed to define and govern new categories of
canonical/computable resources — declaring `prompt-library` there,
together with a profile that pins down the expected payload and
metadata, lets the work group ship value to the reporter quickly,
gather real-world usage, and then promote the code to core in a later
cycle from a position of evidence rather than speculation. CDS should
plan to coordinate with the Vocabulary work group (owner of core
`library-type`) on a future promotion path so that Disposition A
remains an explicit, deferred outcome rather than a road not taken.
