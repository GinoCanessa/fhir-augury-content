## Summary

The CQL specification's Language Semantics section (§05) currently
enumerates the language's capabilities only as narrative prose. Bryn
Rhodes (CDS WG) is asking for a *structured* representation of those
capabilities — a machine-readable artifact (CodeSystem + ValueSet, or
similar) that can be referenced from other artifacts so that tools can
answer questions such as: which language feature does a given
diagnostic message correspond to, which feature is a particular test
exercising, what is the minimum set of features required to execute a
given CQL library, and which features does a given engine claim to
implement. No related Jira tickets, Zulip threads, or repo briefings
were surfaced (see notes below); the only cross-reference in Augury is
to PR [HL7/cql#90](https://github.com/HL7/cql/pull/90), which the
reporter notes is a first-pass implementation that adds two
CodeSystems (`cql-language-capabilities` and
`cql-capability-characteristics-codes`) and their backing ValueSets to
the CQL FHIR IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt PR [HL7/cql#90](https://github.com/HL7/cql/pull/90) as drafted:
add a normative `CodeSystem/cql-language-capabilities` (one code per
language feature currently enumerated in
[§ 05 Language Capabilities](https://cql.hl7.org/2025Sep/05-languagesemantics.html#language-capabilities))
plus a same-named ValueSet that includes the entire CodeSystem;
similarly add `CodeSystem/cql-capability-characteristics-codes` (codes
classifying each capability — e.g., required-vs-optional, profile
membership, evaluation-time-vs-translation-time, etc., as defined in
the PR) plus a backing ValueSet. Update § 05 of the CQL specification
to reference the new CodeSystem from the prose list, and document the
intended consumers (diagnostic messages, test metadata, library
`requires`-style declarations, engine conformance statements).

#### Justification

The reporter's four use cases (locate the feature behind a message,
the feature exercised by a test, the feature set a library needs, and
the feature set an engine implements) all require a *stable, shared
vocabulary* of feature codes. A pair of FHIR CodeSystem+ValueSet
artifacts is the lowest-friction way to publish that vocabulary inside
the CQL FHIR IG without inventing a new artifact type, and PR #90
already proposes exactly that shape. The "characteristics" CodeSystem
gives downstream tools a way to slice capabilities (e.g., "all
optional features", "all features valid in CQL 1.5") without enlarging
the main capability list. Reuse of the FHIR terminology stack also
gives implementers free tooling: validation, expansion, and
binding from CapabilityStatement-like artifacts.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the CodeSystem from PR #90, but instead of a parallel
"characteristics" CodeSystem, model capability metadata as a
**StructureDefinition / Logical Model** (e.g.,
`CqlLanguageCapability`) whose instances carry: `code` (bound to
`cql-language-capabilities`), `version` (CQL version the feature was
introduced in), `optionality` (required | optional | deprecated),
`category`, and `definitionReference` (URL into § 05). Engines and
tests then point at the *instance* (or its `code`) rather than
combining two CodeSystems at the consumer side. Optionally, define a
profile on `CapabilityStatement` (or a new `CqlCapabilityStatement`
logical model) so that an engine can publish its claimed feature set
as a single instance.

#### Justification

A single CodeSystem of feature codes plus a logical model for
metadata is more extensible than two parallel CodeSystems whose codes
must be paired by the consumer. It avoids burning code values in the
"characteristics" CodeSystem for facets that are really *attributes
of* a capability (introduction version, optionality, category) and
that may evolve faster than the capability list itself. It also gives
engine vendors and the CQL test suite an obvious, structured place to
publish/consume conformance — matching the reporter's "what does this
engine implement?" use case more directly than two loose ValueSets.
Trade-off vs. A: more design work up-front, and a logical model is a
heavier publication artifact than a pair of CodeSystems.

---

### Disposition C: Decline

#### Proposal

Do not introduce structured capability artifacts in this ballot cycle.
Instead, keep § 05 Language Capabilities as narrative and add anchor
IDs / a stable numbering scheme so messages, tests, and engines can
reference capabilities by URL fragment (e.g.,
`…/05-languagesemantics.html#cap-retrieve-with-codes`) until concrete
consumers (engines, test runners, message catalogs) demonstrate they
need a coded vocabulary.

#### Justification

The four motivating use cases are real but currently aspirational —
none of them have shipping consumers in the CQL ecosystem that are
blocked on a coded representation today. Publishing CodeSystems is a
long-term commitment: codes, once balloted, are awkward to rename or
withdraw, and the capability list is still being refined in § 05.
Deferring lets the WG see at least one consumer (an engine, a test
runner, or a diagnostic catalog) prove out the shape of the metadata
before locking codes in normative content.

---

### Recommendation

**Recommended disposition:** A

PR [HL7/cql#90](https://github.com/HL7/cql/pull/90) already exists,
the reporter is the WG lead for CQL and authored both the prose and
the PR, and the proposed artifacts (CodeSystem + ValueSet pairs) are
the conventional FHIR mechanism for exactly this need. Disposition C
defers a real and now-actionable need; Disposition B's logical-model
refinement is attractive but can be layered on top of the CodeSystems
later (a `CqlLanguageCapability` logical model can bind its `code`
element to the same CodeSystem published under A) without requiring
A's artifacts to be redesigned. The recommendation, however, is
contingent on review of PR #90 against an up-to-date `repo-analysis`
briefing for HL7/cql — please run `repo-analysis HL7/cql` so the
authoring paths, generated areas, and any "do-not-edit" warnings can
be reflected in the final disposition before the WG meeting.
