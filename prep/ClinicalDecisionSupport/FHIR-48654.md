## Summary

Bryn Rhodes is requesting that the Canonical Resource Management Infrastructure (CRMI) IG add two new operation pattern profiles that are currently listed as roadmap items: a **Parameterizable** pattern (defining a `parameters` input) and a **Data Configurable** pattern (defining `useServerData`, `data`, `prefetchData` with its `key`/`descriptor`/`data` sub-elements, and `dataEndpoint`). The profiles should follow the pattern already established by the CQL IG's `OperationDefinition-cql-library-evaluate`, which composes these parameters into a concrete operation. The ticket also asks the profiles to document two mutual-exclusion rules: implementations can't use both `data` and `prefetchData` simultaneously, and they can't use both `useServerData` and `dataEndpoint` simultaneously (but may use neither). No other Jira tickets are cross-referenced, and only one Zulip thread ([cql > Data Bundle for $evaluate-measure](https://chat.fhir.org/#narrow/stream/cql/topic/Data%20Bundle%20for%20%24evaluate-measure/near/478567670)) touches the request — that discussion confirms there is active implementer interest in a standardized `data` parameter for measure/library evaluation across operations, which is precisely what these pattern profiles would formalize.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add two new `OperationDefinition` pattern profiles to the CRMI IG, authored directly after the model established by `OperationDefinition-cql-library-evaluate` in the CQL IG:

1. **`OperationDefinition-crmi-parameterizable`** — a pattern profile that fixes the shape of a single `parameters` input parameter (typed as `Parameters`, cardinality `0..1`, documenting its role as the container for operation-specific typed parameters). Profile the `parameter` slice by name (`parameters`), set `use = in`, and require documentation of how the contained `Parameters` resource is interpreted by the derived operation.

2. **`OperationDefinition-crmi-data-configurable`** — a pattern profile that fixes the shape of the full data-configuration surface: `useServerData` (`boolean`, `0..1`), `data` (`Bundle`, `0..1`), `prefetchData` (`0..*`) with its nested parts `prefetchData.key` (`string`, `1..1`), `prefetchData.descriptor` (`DataRequirement`, `1..1`), and `prefetchData.data` (`Bundle`, `1..1`), plus `dataEndpoint` (`Endpoint`/`Reference(Endpoint)`, `0..1`). Each slice is named (closed slicing on `parameter.name`) so derived operations inherit them by name.

3. Encode the two mutual-exclusion rules as FHIRPath `invariant`s on the data-configurable profile:
   - `dataVsPrefetch`: `parameter.where(name = 'data').empty() or parameter.where(name = 'prefetchData').empty()` — "`data` and `prefetchData` cannot both be supplied."
   - `serverVsEndpoint`: `parameter.where(name = 'useServerData').empty() or parameter.where(name = 'dataEndpoint').empty()` — "`useServerData` and `dataEndpoint` cannot both be supplied; neither is allowed."

4. Publish narrative in each profile's intro page explaining the pattern, citing `cql-library-evaluate` as the reference composition, and describing how a concrete operation derives from (and may combine) the two patterns.

#### Justification

This is exactly what the reporter asked for, follows a pattern already proven in the CQL IG, and is the natural home for these shapes: CRMI is where cross-operation canonical patterns live, so publishing them there lets Measure, Library, PlanDefinition, and ValueSet operations all derive from a single source of truth. The Zulip discussion confirms live implementer demand (Alexander Zautke was about to ship a divergent `data` parameter on `$evaluate-measure` because nothing standardized existed yet) — formalizing the pattern now prevents that divergence. Mutual-exclusion rules expressed as `invariant`s (rather than prose-only) are testable by validators, consistent with how CRMI encodes cross-parameter constraints elsewhere.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than two separate pattern profiles, publish a **single composite pattern profile** (`OperationDefinition-crmi-data-configurable-parameterizable`) that combines all seven data-configuration parameters with the `parameters` input, and split it into two behavioral "capability tags" (CapabilityStatement extensions or narrative conformance statements) that tell implementers which subset their derived operation supports. The mutual-exclusion rules are expressed as prose conformance statements plus OperationOutcome guidance for servers rejecting conflicting inputs, rather than as `invariant`s on the OperationDefinition.

#### Justification

In practice, every operation Bryn has cited (library evaluate, measure evaluate, activity definition apply, plan definition apply) uses **both** patterns together, so the two-profile split creates composition work (every derived operation inherits from both) without adding expressive power. A single profile is simpler to consume, and behavioral variation can be captured downstream. The trade-off vs. Disposition A: we lose the ability for a future operation to be "parameterizable but not data-configurable" (e.g., a pure computation operation). That loss is small compared to the clarity gained — and if such an operation ever appears, it can still deviate by simply not slicing those parameters. Expressing mutual-exclusion as prose rather than invariants is weaker for validators, but matches how several existing CRMI operations handle cross-parameter constraints today.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that `OperationDefinition-cql-library-evaluate` already serves as the de-facto reference implementation of both patterns, and any future operation that needs the same shape can be authored by copy-adapting from it. Leave Data Configurable and Parameterizable on the roadmap as informative guidance rather than publishing them as normative pattern profiles.

#### Justification

Defensible only if the WG judges that the cost of maintaining additional pattern profiles outweighs the benefit. The reference operation already exists and is discoverable; copy-paste is a legitimate (if suboptimal) reuse pattern in FHIR. Concretely, however, the Zulip thread shows an implementer was already on the verge of shipping a divergent `data` parameter because no normative pattern existed — which is the exact failure mode pattern profiles are meant to prevent. Decline is listed for completeness but is hard to justify given the evidence of real implementer demand.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right call. The request comes from the CQL/CRMI author who established the reference pattern, the target IG (CRMI) is specifically chartered to host cross-operation canonical patterns, and the Zulip thread provides concrete, recent evidence that the absence of these profiles is already causing implementer divergence on `$evaluate-measure`. Disposition B's single-profile composition is tempting for simplicity, but the two-profile split is cheap (CRMI already hosts several small pattern profiles) and preserves the option for "parameterizable-only" operations such as pure computation utilities. Encoding the two mutual-exclusion rules as `invariant`s rather than prose is a modest but real quality win — validators will enforce them automatically, and implementers currently have no machine-readable signal that `data` + `prefetchData` or `useServerData` + `dataEndpoint` are invalid combinations. The workgroup should accept and assign; the main open question worth raising in discussion is the exact type of `dataEndpoint` (direct `Endpoint` vs. `Reference(Endpoint)`) — CQL IG uses `Endpoint` directly and the new profile should match unless there's a reason to diverge.
