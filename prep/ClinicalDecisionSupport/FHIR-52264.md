## Summary

Lynn Perrine reports a layout/whitespace issue in the Clinical Quality
Language (CQL) FHIR specification. Specifically, in section 1.3.1 and its
subsections, a table begins at the bottom of the right-hand navigation
pane, leaving a large block of empty vertical whitespace above the table
in the main content area. The reporter notes that section 1.9 Appendix B
already renders without this problem and suggests using a similar
treatment — wrapping the content of 1.3.1 (and its subsections) so the
table flows immediately under the surrounding prose rather than being
pushed down by the navigation pane height. No cross-referenced Jira
tickets, Zulip discussions, or GitHub items were found for this
ticket, so the request stands on its own as a presentation-only
correction to the CQL IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adjust the CQL IG page source for section 1.3.1 and its subsections so
the table that currently begins at the bottom of the right-hand
navigation pane is moved up under the preceding prose. Concretely:
wrap the surrounding narrative + table in a container (e.g., a
`<div>` with the same class used in 1.9 Appendix B) so that the table
is treated as part of the same flow block as the heading and prose,
rather than being pushed down by the floated/sticky right-nav. No
normative content changes — text, table contents, and IDs remain
unchanged. The change is a presentation-only edit in the IG's page
source (likely the `input/pagecontent/*.md` or `*.xml` file backing
section 1.3.1) and possibly a small tweak to the IG's CSS or template.

#### Justification

This is a Technical Correction in the strict sense: it improves
readability without altering specification semantics. Section 1.9
Appendix B is cited as an existing, working example, so the change
amounts to applying an already-validated layout pattern to a section
that did not receive it. Implementer impact is zero (no schema,
profile, or terminology change); reader impact is positive (less
scrolling past blank space to reach the table). FHIR's IG publication
guidance favors consistent layout across sections, which this change
restores.

---

### Disposition B: Alternative Approach

#### Proposal

Address the underlying cause rather than patching one section: update
the IG's shared template / CSS so that any section whose first
content block is a table is automatically laid out the same way as
1.9 Appendix B. This could be done by adjusting the right-nav's
`position`/`float` rules, adding a `clear` rule before tables, or
introducing a Jekyll/IG-publisher include (e.g., a
`{% include table-section %}` wrapper) that 1.3.1 and any other
similar sections call. As part of this work, audit the rest of the
CQL IG for sibling sections that exhibit the same gap and apply the
fix uniformly.

#### Justification

Disposition A fixes the symptom in 1.3.1 only. If the same layout
quirk exists elsewhere in the CQL IG (or could appear when new
sections are added), a one-section patch will need to be repeated.
Centralizing the fix in the template/CSS keeps future authoring cost
low and prevents regressions. The trade-off is higher up-front work
(template edit + visual regression check across the whole IG) and a
slightly larger blast radius — any rule change to the shared
template must be validated to ensure it doesn't disturb sections
that currently render correctly.

---

### Disposition C: Decline

#### Proposal

Decline as **Not Persuasive — cosmetic only**. Mark the ticket
resolved without code/spec change, on the grounds that the visual
gap in 1.3.1 does not impair comprehension, the content is
reachable by scrolling, and the IG publisher template is shared
across many HL7 specifications where similar whitespace artifacts
are tolerated.

#### Justification

The request is a pure rendering nicety; nothing in the
specification's normative content, examples, or conformance
expectations is affected. Touching the IG template carries some
risk of disturbing the rendering of other sections, and editorial
resources for the CQL IG are limited. If the issue were
widespread or actually obscured information, that would tip the
balance — but as described it is a single-section cosmetic
imperfection.

---

### Recommendation

**Recommended disposition:** A

This is a small, low-risk, presentation-only fix in a single
section that the reporter has already pre-diagnosed (with a
working reference: 1.9 Appendix B). Disposition A delivers the
improvement immediately with minimal review surface. Disposition B
is the "right" engineering answer in principle, but absent
evidence that the same gap appears elsewhere in the CQL IG, it
risks scope creep on what is fundamentally a Technical
Correction; the workgroup can always escalate to B later if the
same ticket pattern recurs. Disposition C is hard to defend given
that the reporter has supplied both a clear description and an
in-spec exemplar of the desired result, making the fix cheap. The
implementer should, however, glance at the surrounding CQL IG
sections while making the change and bring any other obviously
affected sections forward in the same PR so this doesn't have to
be re-litigated.
