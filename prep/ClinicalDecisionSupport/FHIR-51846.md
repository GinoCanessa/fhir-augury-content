## Summary

The reporter (Brian Kaney) asks the CRMI specification to (1) explicitly
document that every FHIR (NPM) Package must contain exactly one
`ImplementationGuide` resource that *represents* that package, and (2)
publish a `PackageImplementationGuide` profile (derived from
`CRMIImplementationGuide`) so that a) conformance to the
"package-representation" role can be enforced and b) servers can return
only package-representing IGs in search responses, distinguishing them
from example or other non-package `ImplementationGuide` instances.

The reporter illustrates the proposed profile with pseudo-FSH that
slices `useContext` to require a `packagePurpose` slice using a CRMI use
context code (`purpose` / `package-identification`), tightens
`useContext` to `1..*` and `version` to `1..1`, and shows a sample
search such as
`/ImplementationGuide?context-type-value=purpose$package-identification`
to retrieve only package-representing IGs. A follow-up comment from the
reporter retracts the `manifest 1..1` constraint from the original
sketch, and notes that the WG is also weighing a paired
`SharablePackageImplementationGuide`. Bryn Rhodes proposes an
alternative that would add a tag to identify the primary
implementation guide for the package, instead of (or in addition to) a
new profile.

The recorded resolution declines the profile per se but accepts the
underlying need with modification: the WG plans to (1) document that
FHIR NPM packages are expected to carry an `ImplementationGuide`
resource whose `packageId` matches the NPM package id, (2) add a
`packageId` SearchParameter on `ImplementationGuide` with a usage
example, and (3) extend the Artifact Repository CapabilityStatement to
require `packageId` search support for `ImplementationGuide`
artifacts.

No cross-references, GitHub items, or Zulip threads were surfaced for
this ticket by FhirAugury.

## Details

**Description:**

> This ticket is to request that:
>
> 1. Document that each FHIR Package must have one ImplementationGuide
>    resource that represents that package.
> 2. Consider a PackageImplementationGuide profile so we can a) enforce
>    conformance and b) enable package searching using FHIR search
>    where examples and other non-package-representation
>    ImplementationGuide resources are not considered.
>
> Here is an example pseudo-FSH for #2:
>
> ```fsh
> Profile: PackageImplementationGuide
> Parent: CRMIImplementationGuide
>
> * useContext ^slicing.discriminator[0].type = #pattern
> * useContext ^slicing.discriminator[0].path = "code"
> * useContext ^slicing.rules = #open
> * useContext contains packagePurpose 1..*
> * useContext[packagePurpose].code = CRMIUseContextCodes#purpose
> * useContext[packagePurpose].valueCodeableConcept = CRMIUseContextCodes#package-identification "Package identification"
>
> * useContext 1..*
> * manifest 1..1
> * version 1..1
> ```
>
> Allows me to search for package-representing ImplementationGuides with
> something like:
>
> ```
> /ImplementationGuide?context-type-value=purpose$package-identification
> ```

**Resolution Description (recorded on the ticket):**

> It is already the case that an implementation guide resource
> identifies the package with the `packageId` element, so it's not
> clear that a profile is needed here, but we will:
>
> 1. Document that FHIR NPM Packages are expected to have an
>    ImplementationGuide resource with a `packageId` that matches the
>    id of the package in which they appear.
> 2. Add a `packageId` search parameter for ImplementationGuide with an
>    example of use.
> 3. Add `packageId` search support to the Artifact Repository
>    capability statement for implementation guide artifacts.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Author the requested `PackageImplementationGuide` profile in the CRMI
spec, derived from `CRMIImplementationGuide`, with these constraints
(reflecting the reporter's corrected FSH from the 2025-09-03 comment):

- Slice `useContext` by pattern on `code`, open slicing, and require a
  `packagePurpose` slice (`1..*`) whose `code` is
  `CRMIUseContextCodes#purpose` and whose `valueCodeableConcept` is
  `CRMIUseContextCodes#package-identification "Package identification"`.
- Tighten `useContext` to `1..*` and `version` to `1..1`.
- Do **not** constrain `manifest` (per the reporter's correction).

Additionally, document in the CRMI narrative that every FHIR NPM
package containing an IG MUST publish one `ImplementationGuide`
instance conforming to `PackageImplementationGuide`, and update the
Artifact Repository CapabilityStatement to require support for the
search
`/ImplementationGuide?context-type-value=purpose$package-identification`.

If/when ready, mirror the same shape as
`SharablePackageImplementationGuide` (per bkaney's 2025-09-03
follow-up) so authors can pick the more constrained variant when their
package warrants it.

#### Justification

This is the most precise, machine-checkable expression of "this
`ImplementationGuide` represents an NPM package." A profile lets
validators reject non-package IGs that claim the role and lets servers
support the `purpose=package-identification` search out of the box,
which is what the reporter actually needs operationally. CRMI has
already established the pattern of layering role-specific profiles
(e.g., `Shareable*`, `Computable*`, `Publishable*`) on the canonical
resource bases, so a `PackageImplementationGuide` (and matching
`SharablePackageImplementationGuide`) sits comfortably in that pattern
and reuses `CRMIUseContextCodes` rather than introducing a parallel
mechanism.

### Disposition B: Alternative Approach (Recorded Resolution — `packageId` SearchParameter + Documentation)

#### Proposal

Adopt the modification already recorded in the ticket's resolution
instead of authoring the profile:

1. Document, in the CRMI narrative, that FHIR NPM Packages are
   expected to publish exactly one `ImplementationGuide` resource
   whose `packageId` matches the NPM package id (i.e., reuse the
   existing `ImplementationGuide.packageId` element as the canonical
   "is this the package representation?" signal).
2. Define a `packageId` `SearchParameter` for `ImplementationGuide`
   (token, against `ImplementationGuide.packageId`), with at least one
   worked example: `/ImplementationGuide?packageId=<id>`.
3. Update the Artifact Repository CapabilityStatement to require
   `packageId` search support on `ImplementationGuide` artifacts.

Optionally adopt Bryn Rhodes' suggestion of a tag identifying the
primary IG for the package as a lightweight server-side hint, but do
not gate conformance on it.

#### Justification

`ImplementationGuide.packageId` already exists in core FHIR and already
identifies the NPM package. The reporter's underlying needs —
"identify which IG represents the package" and "search for it" — are
fully satisfied by exposing that element as a SearchParameter and
codifying the 1:1 expectation in CRMI prose, without minting a new
profile, a new use-context slice, and a new search shape that
implementers must learn. It also avoids the open question of whether
to constrain `version`, `useContext`, or `manifest`, which the
reporter himself revised mid-thread. This is the path the WG has
already recorded as the resolution, so it is the lowest-friction next
step.

---

### Disposition C: Decline

#### Proposal

Decline both the documentation change and the new profile. Treat the
existing `ImplementationGuide.packageId` element plus standard FHIR
search (e.g., querying by `name`, `version`, `url`) as sufficient.
Recommend that callers who need to distinguish package-representing
IGs from examples filter client-side on `packageId` presence.

#### Justification

Defensible only on a strict "no new normative material absent broader
demand" footing. The ticket is reported by a single implementer, has
no cross-references, no Zulip discussion, no linked GitHub PRs, and
the only ancillary comment proposes yet another mechanism (a tag),
suggesting the design is not yet settled. That said, the operational
gap is real and acknowledged in the recorded resolution, so a flat
decline ignores a problem the WG has already agreed needs at least
documentation. This option is included for completeness.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B aligns with the resolution already attached to the
ticket (`Not Persuasive with Modification`) and gives implementers
everything they actually asked for at materially lower spec-surface
cost. `ImplementationGuide.packageId` is the canonical identifier the
NPM/IG ecosystem already relies on, so making it searchable and
documenting the 1:1 expectation between an NPM package and its
representing IG closes the gap without inventing a new profile, a new
slicing of `useContext`, or a new code system pattern that downstream
profiles (US Core, Da Vinci, etc.) would then need to opt into. It
also sidesteps the unresolved design questions visible in the
comments — `manifest` cardinality, the `Shareable*` variant, the
tag-vs-profile choice — by leaving each of them open for a follow-up
ticket if real demand materialises. Disposition A remains the right
move *later* if multiple IG authors converge on needing
profile-grade enforcement and a use-context-based search; Disposition
C should not be adopted because it under-delivers against the WG's
own recorded resolution.
