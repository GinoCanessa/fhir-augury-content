## Summary

The ticket asks the CQL-with-FHIR Implementation Guide (`HL7/cql-ig`) to add a
documented best-practices section for working with `Date`, `DateTime`, and
`Time` values in CQL. Specifically, it calls out (a) the implicit conversion
of `Date` to `DateTime` with `null` time components and the resulting
mismatched-precision comparisons that yield `null` (and are typically
interpreted as `false`); (b) a recommendation to use `day of` precision when
comparing within periods such as the measurement period where date precision
is clinically appropriate; (c) `minute of` / `second of` precision when time
precision is clinically appropriate; (d) a pointer to the new "default
comparison precision" capability introduced in CQL 2.0; and (e) a correction
to the existing date/time arithmetic example that subtracts a UCUM annum
(`'a'`) from a `DateTime` — the current narrative claims it produces a
result, but in CQL that operation should now raise a run-time error because
the UCUM annum is not a calendar duration. No related Jira tickets or Zulip
threads were surfaced by the indexer; the only cross-reference is the GitHub
commit that already implements the change.

## Details

**Description:**

> Add a section on best-practices for Date and DateTime values:
>
> Note that comparing Date and DateTime values results in implicit conversion
> of the Date to a DateTime with null components, and this can lead to
> unexpected partial comparison cases. Be explicit about precision when
> comparing Date and DateTime, as comparing with mismatched precision can
> yield null results (which are often interpreted as false). In most cases,
> using 'day of' precision is appropriate when comparing a Date and DateTime.
>
> Also recommend using day of when comparing between events within the
> measurement period and date precision is clinically appropriate (e.g.
> comparing medication order dates or procedure dates against encounter
> periods).
>
> Also recommend using minute of or second of when comparing between events
> where time values are clinically appropriate.
>
> In addition, reference the new "default comparison precision" capability of
> 2.0 as a way to facilitate these best practices.
>
> In addition, correct the current example of date/time arithmetic that uses
> an annum, this example is incorrect, it will result in a run-time error.
> Update the discussion to say why it results in a run-time error.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt commit
[HL7/cql-ig@87434893](https://github.com/HL7/cql-ig/commit/87434893a869ff689c1916607379abdaa1a7178d)
as-is. Concretely, in `HL7/cql-ig`, file `input/pages/patterns.md`:

1. Insert a new H3 section **"Date, Time, and DateTime Values"** immediately
   before "Time-Valued Quantities" that:
   - Explains the implicit `Date` → `DateTime` conversion (with `null` time
     components) and the resulting mismatched-precision comparison hazard.
   - Recommends `day of` precision when comparing a `Date` with a
     `DateTime`, or when comparing events within a period such as the
     measurement period where date precision is clinically appropriate.
   - Recommends `minute of` / `second of` precision when comparing events
     where time precision is clinically appropriate.
   - Links to the CQL Author's Guide section on
     [Comparing Dates and Times](https://cql.hl7.org/02-authorsguide.html#comparing-dates-and-times).
   - Includes a callout pointing readers at the new
     [_default comparison precision_](https://cql.hl7.org/2025Sep/03-developersguide.html#defaultcomparisonprecision)
     feature in CQL R2 as a mechanism that supports this best practice.
2. Rewrite the existing UCUM-annum example so it states that
   `@2019-01-01T05:00:00 - 1 'a'` is now a **run-time error** (because a
   UCUM annum is 365.25 days and is not a calendar duration), and show the
   correct workaround using `convert 1 'a' to 's'`.
3. Add the paragraph clarifying that FHIR `Duration` values with `mo` and
   `a` units convert to **calendar** months and years on import into CQL
   (because that is virtually always the authorial intent in FHIR), with
   the `Patient.birthDate + (Condition.onset as Age)` example.
4. Update the trailing arithmetic-discussion link from the dated
   `2020May` Author's Guide URL to the undated current URL.

Land it as a single PR against `HL7/cql-ig` `master`, in the CQL-with-FHIR
2.0 / CQL R2 publication line. Mark FHIR-53575 **Persuasive — Applied** when
merged.

#### Justification

The request is narrowly scoped narrative guidance for a known authoring
hazard, and the implementing commit is already on `master`. The new section
is materially identical to what the description asks for: the `null`-on-
mismatched-precision behaviour is faithfully described, the `day of` /
`minute of` / `second of` recommendations are present, the new R2
"default comparison precision" capability is referenced, and the broken
UCUM-annum example is corrected with both the explanation of *why* it
errors and the supported workaround. The bonus paragraph about FHIR
`Duration` (`mo`, `a` → calendar) is a defensible expansion because it
addresses the same authoring hazard the ticket is flagging — implicit
conversions silently doing surprising things at the CQL boundary.
Resolution is already recorded as *Persuasive*, which is consistent with
"accept the request, change applied".

### Disposition B: Alternative Approach

#### Proposal

Accept the substance of the change but split it into two PRs against
`HL7/cql-ig/input/pages/patterns.md` and tighten the wording before
publication:

1. **PR-1 (this ticket):** keep only items (1) and (2) above — the new
   "Date, Time, and DateTime Values" section and the UCUM-annum
   correction. Fix the typo "intepreted" → "interpreted" in the new
   paragraph. Soften "this can lead to unexpected partial comparison case"
   → "…unexpected partial-comparison cases". Pin the
   _default comparison precision_ link to the chosen CQL R2 snapshot URL
   the IG is publishing against, rather than the floating `2025Sep`
   preview, so a stale link doesn't appear after the next CQL spec
   ballot.
2. **PR-2 (follow-up, separate ticket):** the FHIR `Duration` (`mo`, `a`
   → calendar) paragraph and the `Patient.birthDate + (Condition.onset as Age)`
   example. This is closely related but is really about the FHIR-to-CQL
   type-mapping rules for `Duration`, not about comparison precision —
   readers will look for it under FHIR-data-model guidance, not under
   "patterns". Tracking it in its own ticket also lets reviewers debate
   whether the implicit-conversion rule for `mo`/`a` belongs in
   `cql-ig/patterns.md`, in `cql-ig/using-cql.md`, or upstream in the
   `HL7/cql` Author's Guide itself.

#### Justification

The ticket explicitly enumerates four asks; the implementing commit
delivers all four plus a fifth (the FHIR `Duration` paragraph). Splitting
that fifth concern out keeps the ballot reconciliation tight — reviewers
can disposition the precise text the ticket asked for without also
weighing in on a related-but-separate FHIR-side conversion rule. It also
reduces the risk that a workgroup objection to the `Duration` paragraph
blocks the rest of the guidance from landing. Pinning the
_default comparison precision_ link and fixing the typo are zero-risk
quality improvements that are easier to apply now than after the section
ships.

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope for this IG**. Move the entire
discussion into the `HL7/cql` Author's Guide (the CQL specification
itself), under
[Comparing Dates and Times](https://cql.hl7.org/02-authorsguide.html#comparing-dates-and-times),
on the grounds that the precision-and-`null` behaviour and the
UCUM-annum arithmetic semantics are properties of the CQL language, not
of its FHIR binding, and replicating language-level guidance in
`HL7/cql-ig` risks the two documents drifting apart. From `cql-ig`,
add only a one-line pointer to the upstream section.

#### Justification

The IG's role is to document patterns specific to using CQL **with FHIR**
data — for example, comparing FHIR's `dateTime` against CQL's `DateTime`,
or interpreting FHIR `Period` boundaries. The mechanics of mismatched-
precision comparison and the UCUM-annum run-time error are pure CQL
behaviour and apply equally to non-FHIR data models. Putting the
canonical text in the language spec keeps a single source of truth and
avoids the IG having to re-version this content every time the CQL
specification clarifies the rules.

### Recommendation

**Recommended disposition:** A

The ticket is already resolved *Persuasive* with an implementing commit on
`HL7/cql-ig` `master` that faithfully covers every bullet in the
description; rolling that commit back to chase B's split-PR refinement or
C's "move it upstream" ideal would impose more reconciliation cost than
either provides in value, especially given that the Patterns page already
houses neighbouring CQL-flavour guidance (e.g., Time-Valued Quantities,
Code Comparison) that lives in this IG for exactly the same readability
reasons. Disposition C's "single source of truth" point is real but is
better served by a separate, lower-priority ticket against `HL7/cql` to
ensure the Author's Guide section is the authoritative one and that
`cql-ig/patterns.md` consistently cross-links to it (which the
implementing commit already does). Disposition B's polish items (typo,
softened wording, pinned link) are worth raising as PR-review comments on
the same merged commit or in a follow-up housekeeping PR, but they don't
warrant blocking acceptance. The workgroup should confirm A and, if
desired, capture B's editorial fixes as a tracked follow-up.
