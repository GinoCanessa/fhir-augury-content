## Summary

Isaac Vetter requests that FHIR add SearchParameters covering the
`module` and `occurrenceDateTime` elements of `GuidanceResponse` — the
CDS resource that conveys the result of evaluating a decision-support
module. Both elements are currently defined on `GuidanceResponse` (R5),
but neither is exposed as a built-in SearchParameter, so a client cannot
ask a server for "the GuidanceResponse(s) produced by *this*
PlanDefinition / Library" or "GuidanceResponses produced on/around *this*
date" without resorting to `_filter`, custom search parameters, or
client-side post-filtering. The ticket asserts that both queries are
"obviously useful." There are no cross-referenced Jira tickets, Zulip
threads, or GitHub items linked to this issue, so the request stands on
its own narrow merits.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add two new core SearchParameters to `GuidanceResponse`:

1. **`GuidanceResponse-module`**
   - **type:** `reference` (with companion `token` semantics for
     `CodeableConcept` form)
   - **expression:** `GuidanceResponse.module`
   - **target:** `PlanDefinition` (canonical / reference forms)
   - Because `module[x]` is a choice of `uri | canonical |
     CodeableConcept`, define the parameter as `reference` aimed at the
     `canonical` form (mirroring how other `module`-style canonicals are
     searched on, e.g. `Measure`-`derived-from`), and document that the
     `uri` and `CodeableConcept` forms are searchable via `_filter` or a
     companion `module-code` token parameter if the workgroup wants
     parity for the CodeableConcept form.
2. **`GuidanceResponse-occurrence`**
   - **type:** `date`
   - **expression:** `GuidanceResponse.occurrenceDateTime`
   - Name aligned with the existing FHIR convention of using
     `occurrence` for `occurrence[x]` date search parameters
     (e.g. `MedicationAdministration-effective-time`,
     `ServiceRequest-occurrence`).

Both parameters land in the CDS-owned `GuidanceResponse` definition and
its generated `searchparameter-*.xml/json` artifacts in the FHIR core
build.

#### Justification

The two queries are first-order CDS workflow needs: "show me every
GuidanceResponse that came from PlanDefinition X" is the basic
audit/trace query for any decision-support deployment, and filtering
GuidanceResponses by when they were produced is the standard temporal
slice every clinical resource supports. Sibling CDS resources already
expose comparable parameters — `Measure`/`MeasureReport` are searchable
by their canonical references and by date, and most event-style
resources expose `occurrence` as a date SearchParameter — so adding
these brings `GuidanceResponse` in line with established FHIR patterns
rather than introducing anything novel. The change is additive and does
not alter existing conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent but tighten and broaden the design before publishing:

- Define `GuidanceResponse-module` as a **`reference`** SearchParameter
  whose expression resolves the canonical form
  (`GuidanceResponse.module.ofType(canonical)`), **and** define a
  companion **`GuidanceResponse-module-code`** `token` parameter for the
  `CodeableConcept` form. This avoids ambiguity about which choice the
  parameter targets and matches how FHIR has handled prior
  canonical/CodeableConcept choice elements.
- Define `GuidanceResponse-occurrence` as `date` over
  `GuidanceResponse.occurrenceDateTime` (same as Disposition A).
- Additionally consider adding `GuidanceResponse-based-on`,
  `GuidanceResponse-encounter`, and similar low-hanging parameters in
  the same change so `GuidanceResponse` is not revisited every cycle for
  one parameter at a time. Scope of the extra parameters is at the WG's
  discretion.

#### Justification

`module[x]` is a choice type, and a single `reference` SearchParameter
silently ignores the `uri` and `CodeableConcept` forms, which can
mislead implementers who populate those forms in production. Splitting
into `module` (reference/canonical) plus `module-code` (token) is the
pattern FHIR has settled on for canonical/CodeableConcept choices and
keeps both populations queryable without `_filter`. Bundling adjacent
obvious parameters in the same ballot cycle is also more efficient for
implementers than dribbling them in one ticket at a time, given that
`GuidanceResponse`'s SearchParameter set is currently very thin.

---

### Disposition C: Decline

#### Proposal

Decline as written and direct the requester to use the `_filter` search
expression (or server-defined custom SearchParameters) for both queries
until a concrete, multi-implementer use-case is presented. The
rationale category is "insufficient evidence of broad implementer
need."

#### Justification

The ticket is a one-line assertion that the parameters "seem obviously
useful," with no implementer survey, reference implementation, or Zulip
discussion behind it. FHIR core SearchParameters are part of the
conformance surface every server must support or explicitly opt out of,
so adding them on aesthetic grounds increases the burden on every
GuidanceResponse implementer. `_filter` already covers both queries
today, and any deployment with a real need can ship a custom
SearchParameter without core action. Declining keeps the
`GuidanceResponse` SearchParameter surface minimal and demands a real
use-case before expanding it.

---

### Recommendation

**Recommended disposition:** B

The underlying request is sound — `module` and `occurrenceDateTime` are
the two most natural axes to query `GuidanceResponse` on, and the
absence of either in core is a real gap relative to peer CDS resources.
Disposition A solves the immediate ticket but leaves an avoidable
ambiguity around `module[x]`'s `CodeableConcept` form, which the
workgroup will almost certainly have to revisit. Disposition B solves
the ticket *and* closes that ambiguity with the established
`canonical` / `code` parameter-pair pattern, at very little additional
cost. Disposition C is defensible on process grounds but hard to
justify on substance: the parameters are conventional, additive, and
match how every other event-style resource is searched. CDS should
adopt B, optionally trimming the "while we're here" extra parameters if
scope is a concern.
