## Summary

The ticket asks the Canonical Resource Management Infrastructure (CRMI)
specification to disentangle two concerns that the current `$release`
operation conflates: releasing an individual canonical artifact (setting
status, version, and date and recursively releasing children per
`versionBehavior`) and managing a program-level manifest (pinning
versions of all direct and transitive references). The reporter argues
that manifest production should be its own operation — for example
`$build-manifest` or `$manifest` — and that the manifest concept should
not be tied to a specific "program." Once that separation is in place,
`$release` can be reduced to status/version/date updates and recursive
release of child artifacts, and manifests can be produced independently
of release timing or program scope. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced for this ticket, so the
report relies solely on the ticket's own description and resolution.

## Details

**Description:**

> The current description of the release operation includes "pins
> versions of all direct and transitive references and records them in
> the program's manifest."
>
> Manifest management should be separated from individual artifact
> release.
>
> One potential solution is to separate manifest management into a new
> operation (such as `$build-manifest` or `$manifest`) that will pin
> versions, update references, etc. for the manifest library.
> Individual artifact release would then be simplified to updating the
> status/version/date and recursively releasing child artifacts
> according to `versionBehavior`.
>
> Manifest management should also not be tied to a specific "program."

**Resolution (Persuasive):**

> Agreed, separate the aspects of
>
> 1. Determining dependencies
> 2. Establishing the current manifest based on those dependencies
> 3. Releasing the manifest
>
> Document that when creating a specification, the `$data-requirements`
> operation can be used to determine the set of dependencies referenced
> by the artifact. This set of dependencies can then be used to infer
> manifest parameters, using a new `$infer-manifest-parameters`
> operation, and then the release process consists only of determining
> latest version for any unversioned references in the artifact.
>
> Note that the release process must still be used to establish the
> version for any unversioned references in the artifact because it is
> a transactional operation (i.e. the resulting manifest must contain
> all the unversioned references in the artifact or any of its
> dependencies at the time of release).

## Proposed Dispositions

(There are no GitHub cross-references for this ticket and therefore no
repo-analysis briefings to anchor the dispositions in. The dispositions
below name the affected specification — Canonical Resource Management
Infrastructure (CRMI) — and the OperationDefinitions that would be
edited there, but no specific authoring file path is asserted because
no briefing is available.)

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution as written and refactor the CRMI release model
into three explicitly separated operations:

1. **Keep `$release` narrow.** Redefine the existing `$release`
   OperationDefinition so its only behavior is to (a) set
   `status = active`, set `version`, and set `date` on the target
   artifact; (b) resolve any unversioned references it directly owns to
   concrete versions (the transactional aspect called out in the
   resolution); and (c) recursively release child artifacts per the
   artifact's `versionBehavior`. Strip the manifest- and
   program-related language ("pins versions of all direct and
   transitive references and records them in the program's manifest")
   from the operation description.
2. **Introduce `$infer-manifest-parameters`.** Add a new
   OperationDefinition that, given an artifact, calls (or composes
   with) `$data-requirements` to compute the full set of direct and
   transitive dependencies and returns the parameter set needed to
   construct a manifest Library — independent of any "program" concept.
3. **Use a manifest-release flow built from existing pieces.** Document
   the recommended pipeline: `$data-requirements` →
   `$infer-manifest-parameters` → construct/update the manifest Library
   → `$release` the manifest Library itself. Update the CRMI narrative
   pages that currently describe release-as-manifest-pinning to reflect
   this separation, and remove any normative language that ties
   manifests to a "program."

#### Justification

This is exactly the path the workgroup has already endorsed
("Persuasive"). It honors the reporter's core observation that
`$release` is overloaded: a per-artifact lifecycle transition is being
mixed with a cross-artifact dependency-graph computation, which makes
`$release` hard to specify, hard to implement, and impossible to use
when the caller wants one behavior but not the other (e.g., releasing
an artifact without rebuilding a program manifest, or rebuilding a
manifest without re-releasing artifacts). Splitting on the natural
seam — dependency discovery (`$data-requirements`), manifest parameter
inference (`$infer-manifest-parameters`), and lifecycle transition
(`$release`) — also lets each operation be reused independently
(tooling, CI, authoring environments) and removes the "program"
coupling, which is not a first-class concept everywhere CRMI is used.
The transactional caveat in the resolution is preserved: `$release`
still pins unversioned references at release time, so callers don't
lose the atomicity guarantee they rely on today.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a single `$release` OperationDefinition but parameterize it so
manifest behavior is opt-in rather than implicit. Concretely:

- Add a `manifestMode` (or similarly named) `code` parameter to
  `$release` with values such as `none` (default — only update
  status/version/date and recurse per `versionBehavior`),
  `infer-only` (compute and return manifest parameters via the same
  logic as `$infer-manifest-parameters` but do not write a manifest),
  and `build` (compute parameters and produce/update a manifest
  Library).
- Remove all references to "program" from the operation's prose; the
  manifest, when produced, is associated with the released artifact (or
  a caller-supplied target Library), not with a program.
- Document `$data-requirements` as the underlying dependency-resolution
  primitive and note that the `infer-only` / `build` modes are exactly
  the composition the resolution describes — just exposed through one
  operation rather than three.

#### Justification

This addresses the same underlying concern — that today's `$release`
silently does manifest work and assumes a program — without introducing
two new OperationDefinitions and without forcing every existing
implementer to rewire their release pipeline. Tooling that wants the
old "release plus manifest" behavior can pass `manifestMode = build`;
tooling that only wants the lightweight lifecycle transition the
reporter is asking for gets it via the default. The trade-off vs.
Disposition A is composability: Disposition A's small operations can be
reused outside `$release` (e.g., a CI job that just wants to refresh a
manifest without releasing anything), whereas this approach keeps the
manifest path inside `$release` and may push callers to invoke
`$release` purely for its side effects. The cost saved is spec surface
area and migration churn for existing CRMI implementers.

---

### Disposition C: Decline

#### Proposal

Decline the request and instead clarify the existing `$release`
operation's prose: explicitly document that "manifest" in the current
text refers to the manifest Library produced as part of release, and
that "program" is a non-normative organizing concept used only when the
caller chooses to model one. Add a non-normative note that callers who
need manifest production independently of release can already achieve
it by invoking `$data-requirements` themselves and constructing the
manifest Library outside the operation.

#### Justification

Defensible only on the grounds that (a) `$release` already implicitly
composes dependency resolution and lifecycle change, and (b) splitting
it risks fragmenting a flow that today is atomic. Declining would
preserve backwards compatibility for existing CRMI implementers and
avoid adding two new OperationDefinitions to the spec. However, it
leaves the reporter's central complaint — that "manifest management is
tied to release and tied to a program" — unresolved, and the workgroup
has already labeled the request Persuasive. This option is included for
completeness rather than as a serious recommendation.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already marked the resolution Persuasive, and the
proposed three-way split (`$data-requirements` →
`$infer-manifest-parameters` → `$release`) cleanly matches the natural
seam in the current operation: dependency discovery, manifest parameter
construction, and lifecycle transition are genuinely separable
concerns, and each is independently useful to tooling. Disposition B
preserves a single entry point and reduces spec churn, but it keeps the
fundamental coupling the reporter is objecting to and forces callers
who only want a manifest to invoke `$release`. Disposition C does not
address the substantive concern. Accepting Disposition A — including
the explicit reminder in the resolution that `$release` must remain
transactional for unversioned references — gives implementers a
coherent pipeline, removes the program coupling, and aligns the spec
with how CRMI is already being used in practice.
