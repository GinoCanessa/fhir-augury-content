## Summary

The Artifact Operation profile in the Canonical Resource Management
Infrastructure (CRMI) IG defines a reusable parameter pattern for
identifying the focus artifact of an operation (by id, url, identifier,
or resource). Bryn Rhodes notes that the current pattern only
accommodates a *single* focus artifact and therefore cannot express
multi-artifact operation invocations such as `$evaluate-measure` against
several Measures at once. The ticket lays out three candidate shapes for
adding multi-artifact support and weighs the trade-off between expressive
power and the simplicity of GET-style invocation:

- **Option 1 — repeating single-parameter form.** Promote each artifact
  selector (`measureId`, `measureUrl`, `measureIdentifier`, `measure`) to
  `0..*`, with token-style version encoding for url/identifier so a
  single GET can carry multiple artifacts (e.g.,
  `?measureUrl=…/Measure/123|1.0.0;…/Measure/234|1.0.0`).
- **Option 2 — multi-part `measure` parameter.** Define a `measure 0..*`
  Parameters part with sub-parts `id`, `url`, `version`, `identifier`,
  `resource`, `topic`, requiring exactly one selector per occurrence.
  Reserve type-level `$evaluate-measure` for the multi-part form and keep
  instance-level invocation single-measure only.
- **Option 3 — generic `artifact 0..*` multi-part parameter.** Same shape
  as Option 2 but generalised to any artifact resource (`reference`,
  `url`, `version`, `identifier`, `resource`, `topic`), pushing the
  pattern up into the Artifact Operation profile itself so all
  artifact-bearing operations benefit.

The ticket explicitly asks the workgroup to update the Artifact
Operation **profile pattern** so multiple focus artifacts are
representable. There are no cross-referenced Jira tickets, Zulip threads,
or GitHub items to pull additional context from; the ticket carries no
comments. CDS has already triaged this as **Persuasive with
Modification**, with a resolution direction (see Details) that adopts a
variant of Option 1 generalised to `artifact*` naming.

## Details

**Description:**

The Artifact Operation profile defines a pattern for the various ways to
identify the focus artifact for the operation. However, it does not
support the ability to identify multiple artifacts (such as
`$evaluate-measure` with multiple measures).

However, there is a tension between supporting multiple artifacts and the
simplicity of GET invocation. Options include:

**Option 1 (single-parameter):**

- `measureId` id 0..*
- `measureUrl` canonical 0..* (this would be inclusive of a canonical reference)
- `measureIdentifier` string 0..* (would need an encoding to support version, similar to token encoding for identifiers)
- `measure` Measure 0..*

```
GET [base]/Measure/$evaluate-measure?measureId=123;234
GET [base]/Measure/$evaluate-measure?measureUrl=http://hl7.org/fhir/Measure/123|1.0.0;http://hl7.org/fhir/Measure/234|1.0.0
GET [base]/Measure/$evaluate-measure?measureIdentifier=http://example.org/fhir/Identifiers/measure-identifier#123|1.0.0
POST (for body)
```

Keep instance-level `$evaluate-measure` as only single-measure
evaluation. Define the type-level `$evaluate` with multi-part parameters.

**Option 2 (multi-part parameters):**

- `measure` 0..* (for each `measure` parameter, one and only one of
  (`id` | `version`/`url` | `identifier`/`url` | `resource` | `topic`))
  - `measure.id` 0..1
  - `measure.url` canonical 0..1
  - `measure.version` string 0..1
  - `measure.identifier` Identifier 0..1
  - `measure.resource` Measure 0..1
  - `measure.topic`

Define a server-level `$evaluate-measure` with multi-part parameters
(artifact can only be Measure or Library here).

**Option 3 (multi-part parameters):**

- `artifact` 0..* (for each `artifact` parameter, one and only one of
  (`id` | `version`/`url` | `identifier`/`url` | `resource` | `topic`))
  - `artifact.reference` string 0..1 (required to be a relative reference, i.e. `Measure/123`, `Library/123`)
  - `artifact.url` canonical 0..1
  - `artifact.version` string 0..1
  - `artifact.identifier` Identifier 0..1
  - `artifact.resource` Measure | Library 0..1
  - `artifact.topic`

The artifact profile pattern should be updated to support the possibility
of multiple focus artifacts.

**Resolution direction (per snapshot, "Persuasive with Modification"):**

Update the artifact operation profile pattern to allow specification of
multiple artifacts using the first pattern (generalised to `artifact*`
naming):

- `artifactId` id 0..*
- `artifactIdentifier` Identifier 0..* (version not allowed; up to server)
- `artifactUrl` canonical 0..* (version optional)
- `artifact` Resource 0..*

### Disposition A: Accept As Requested

#### Proposal

Adopt the **resolution direction already captured on the ticket** (a
variant of Option 1 generalised to `artifact*` naming). Update the
Artifact Operation profile pattern (the OperationDefinition template
in the CRMI IG that other artifact operations such as
`Measure/$evaluate-measure`, `PlanDefinition/$apply`,
`Library/$evaluate`, etc. inherit from) so the focus-artifact selectors
become repeating:

- `artifactId` : id, cardinality `0..*`
- `artifactIdentifier` : Identifier, cardinality `0..*` — version **not**
  allowed in the parameter (server picks the version)
- `artifactUrl` : canonical, cardinality `0..*` — version **optional**
  (carried via the canonical's `|version` suffix)
- `artifact` : Resource, cardinality `0..*`

Keep the single-artifact GET ergonomics by allowing the same parameter
name to repeat across the query string (one `?artifactUrl=…` per
artifact) rather than introducing a token-list separator. Re-derive each
concrete artifact operation (`$evaluate-measure`, `$apply`, `$package`,
`$data-requirements`, etc.) from the updated pattern, renaming the
parameter on each operation to its domain term (`measureId`,
`planDefinitionUrl`, …) but keeping the new `0..*` cardinality and the
"identifier carries no version, url may carry version" rules. Document
that instance-level invocation continues to identify exactly one
artifact (the path-bound resource), while type-level and system-level
invocations may carry multiple.

#### Justification

This is the disposition the CDS workgroup has already signalled in the
recorded resolution ("Persuasive with Modification" with the `artifact*`
0..* shape). It is the smallest change to the existing pattern: it does
not introduce new Parameters parts, preserves the simple GET form, and
keeps every existing single-artifact invocation valid (a `0..1`
parameter naturally widens to `0..*`). Splitting "version on `url`,
no version on `identifier`" matches existing FHIR conventions for
canonical references vs. business identifiers and avoids inventing a
new token-with-version encoding. Because no Zulip thread or
cross-referenced ticket pushes back on this shape, and the reporter
themselves listed it as Option 1, accepting it carries low community
risk.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt **Option 3** instead: extend the Artifact Operation profile with a
single repeating, multi-part `artifact 0..*` Parameters part rather than
a parallel set of repeating top-level parameters. Each occurrence
carries exactly one of `id`, `url` (with optional `|version`),
`identifier`, `resource`, `reference`, or `topic`, enforced by an
invariant on the OperationDefinition pattern. Restrict GET-style
invocation to the single-artifact case and require POST with a
`Parameters` body for multi-artifact invocation. Concrete operations
(`$evaluate-measure`, `$apply`, …) inherit the `artifact` part as-is
and may further constrain `artifact.resource` (e.g., `Measure | Library`
for `$evaluate-measure`).

#### Justification

The multi-part shape keeps the per-artifact selectors *grouped*, which
matters as soon as you need per-artifact context (e.g., per-measure
period, per-measure subject, per-artifact parameter overrides) — a
natural next request once multi-artifact invocation lands. With the
flat repeating shape in Disposition A, callers passing
`artifactUrl=A,B,C` and `artifactId=X,Y` have no way to associate
auxiliary parameters with a specific artifact, and a future ticket will
have to retrofit a grouping mechanism. Option 3 also pushes the
"exactly one selector per artifact" rule into the profile via an
invariant, removing ambiguity that Disposition A leaves implicit.
The cost is loss of the clean GET form for the multi-artifact case,
but the reporter already acknowledges that POST is required for the
multi-part variant, and instance-level GET (the most common case)
remains unchanged.

---

### Disposition C: Decline

#### Proposal

Do not change the Artifact Operation profile pattern. Leave each
artifact operation single-focus, and direct callers who need to
evaluate or apply multiple artifacts to either (a) issue one
operation invocation per artifact, optionally inside a FHIR
`transaction`/`batch` Bundle, or (b) use existing batching mechanisms
such as `$evaluate-measure` chained from a search/`Group` or a
caller-side fan-out. Document this expectation explicitly in the
Artifact Operation profile narrative so future tickets do not re-raise
it.

#### Justification

A defensible "decline" rests on the observation that multi-artifact
invocation is largely a *convenience* over batch/transaction Bundles,
which already exist and already give per-call isolation, error
reporting, and result correlation. Widening every artifact operation's
selectors to `0..*` adds combinatorial complexity (what if both
`artifactId` and `artifactUrl` are populated with mismatched lengths?
what does `instance-level` invocation do with `0..*`?) and forces every
downstream IG that derives from the pattern to re-validate its
operations. Given there is no Zulip discussion and no cross-referenced
implementer demand on file, the workgroup could legitimately argue the
use case has not been demonstrated strongly enough to justify a profile
change.

---

### Recommendation

**Recommended disposition:** A

The CDS workgroup has already triaged the ticket as Persuasive with
Modification along exactly the `artifact*` 0..* lines in Disposition A,
and that shape is the lowest-friction landing for the change: it widens
existing cardinalities rather than introducing a new Parameters
sub-structure, preserves backward compatibility for every existing
single-artifact invocation, and keeps the GET-friendly call shape that
the reporter explicitly flags as valuable. Disposition B (Option 3) is
the more *future-proof* design and deserves a follow-up ticket if/when
per-artifact auxiliary parameters become a real requirement, but
adopting it now would be a larger spec change than the evidence on this
ticket supports. Disposition C understates the demand: the reporter is
pointing to a concrete pattern gap (`$evaluate-measure` against multiple
Measures) that batch Bundles do not cleanly solve at the operation
signature level. Recommend accepting A and explicitly noting in the
profile narrative that grouped per-artifact parameterisation is a
deliberate non-goal for this revision, to keep the door open for a
future Option-3-style extension without re-litigating Disposition A.
