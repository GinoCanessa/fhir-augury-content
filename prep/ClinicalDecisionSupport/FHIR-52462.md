## Summary

The reporter (Chris Moesel) requests substantive clarification of the
*Defining Models* section of the Clinical Quality Language (FHIR)
specification. As currently written, the section claims to describe how
models are defined in CQL but in fact only describes how to define the
*components* of a model (types, conversions, contexts) — leaving
implementers without guidance on whether (or how) a complete model can
actually be authored in CQL. The ticket asks the spec to either (a)
clearly state that full model definitions are out of scope and limit the
language accordingly, or (b) describe end-to-end how the individual
component definitions are organized into a model: scoping to a CQL
library, the mechanism by which a library declares it is defining a model
(so other libraries can `using` it), whether non-model expressions may
co-exist in such a library, the mapping from library `name`/`version`/
namespace to model `name`/`version`/`url`, and how other ModelInfo
attributes (`targetUrl`, `targetQualifier`, `patientClassName`,
`patientBirthDatePropertyName`, etc.) are expressed. It also asks how
the additional Class Type attributes that normally live in ModelInfo
(`identifier`, `label`, `retrievable`, `primaryCodePath`, …) are
represented — or, if they cannot be, that the spec call out the
limitation. Two more focused questions are raised: how the `related to
… by` clause on contexts is intended to be interpreted given that CQL
has no native reference type, and how class types defined this way
participate in the resolution algorithm in the *Multiple Data Models*
section when type definitions are co-mingled with library logic.

There were no Jira cross-references on this ticket and no GitHub
cross-references. A related Zulip thread on `#cql` ("Feedback developing
a new CQL-to-ELM translator") is the most directly relevant piece of
public discussion: in it, JP (Jonathan Percival) explicitly notes that
CQL 2.0 adds *new keywords for defining models directly in CQL*, and
sketches a strategy of representing existing ModelInfos as CQL 2.0
libraries shipped as embedded shims — i.e., the very mechanism whose
under-specification the ticket complains about.

## Details

**Description:**

The Defining Models section needs additional clarification around the following areas:

- It says it is about how to define models, but it only describes how to define the components of a model. If it is not possible to define full models, then update the language to clarify.
- If it is possible to define full models, it needs to describe how the individual definitions are organized into a model. E.g.
  - Should a model be fully contained in a single CQL library?
  - How does that library declare that it is defining a model (so other libraries can target it for `using`)?
  - Is it permissible to have expressions of other types in the library, or only expressions that define components needed for the model (types, conversions, functions used by the conversions, and contexts)?
  - Do the library name, version, and namespace correspond to the model name, version, and url?
  - What about the other model attributes like `targetUrl`, `targetQualifier`, `patientClassName`, `patientBirthDatePropertyName`, etc.?
- How do you define other attributes of Class Types that are typically used in ModelInfos (e.g., `identifier`, `label`, `retrievable`, `primaryCodePath`, etc.)?
  - If that's not possible, then the spec should clearly indicate the limitations.
- When contexts specify `related to ... by`, how should that be interpreted? Since CQL doesn't have a native reference type, how do implementations know how to resolve the RHS (`subject`, in the example) to the LHS (`Patient`, in the example)?
- How do class types defined in this way fit into the resolution process described in the *Multiple Data Models* section? (This mainly applies if type definitions can be co-mingled with library logic.)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the *Defining Models* section of the CQL (FHIR) specification to
provide a complete, normative description of how a model is defined in
CQL 2.0, addressing every bullet in the ticket:

1. State explicitly that a model is fully contained in a single CQL
   library (the recommended convention) and define the marker by which
   such a library declares itself as a model definition (e.g., a
   `model` declaration analogous to `library`, or a reserved
   `using`-able shape) so that other libraries can `using` it.
2. State the policy on co-mingling: whether non-component expressions
   (regular `define` statements, functions unrelated to conversions)
   are permitted inside a model-defining library, and the implications
   for `using`/import.
3. Specify the mapping from library identity (name, version,
   namespace) to model identity (`name`, `version`, `url`), including
   how `targetUrl`, `targetQualifier`, `patientClassName`,
   `patientBirthDatePropertyName`, and the remaining ModelInfo-level
   attributes are expressed in CQL syntax (new keywords, annotations,
   or library-level metadata clauses).
4. Add syntax (or document the existing syntax) for the per-Class-Type
   attributes `identifier`, `label`, `retrievable`, and
   `primaryCodePath` — or, if they are intentionally not supported in
   CQL-defined models, call that out as an explicit limitation with
   guidance on when authors must fall back to a ModelInfo XML.
5. Define the semantics of `related to ... by` in the context of CQL's
   lack of a native reference type — i.e., specify how implementations
   are expected to resolve the RHS path against the LHS class (e.g.,
   structural path resolution against the model's own type definitions)
   and what error behavior is expected when resolution fails.
6. Update *Multiple Data Models* to describe how class types defined in
   a CQL library (potentially co-mingled with logic) participate in
   model lookup, name resolution, and conflict resolution.

#### Justification

The ticket is raised by an active CQL reference-implementation
contributor and is corroborated by the linked Zulip thread, in which JP
explicitly states that CQL 2.0 adds keywords for defining models
directly in CQL but that there is *no public example* of how to do so
end-to-end. The current text therefore over-promises ("Defining Models")
relative to what it actually documents. Filling in items (1)–(6) is the
minimum needed for an independent implementer (such as the Rust
translator project on the Zulip thread) to produce a CQL 2.0 model
without reverse-engineering the Java/Kotlin reference implementations.
Doing this in the spec — rather than in tooling — preserves CQL's
implementer-independence design principle.

---

### Disposition B: Alternative Approach

#### Proposal

Split the work into two coordinated edits:

- **In CQL (FHIR):** Replace the current *Defining Models* prose with a
  scoped statement that the FHIR-binding spec documents only the
  *components* of a model that can be declared in CQL (types,
  conversions, contexts) and that the *organization* of those
  components into a complete model — including library/model identity
  mapping, model-level attributes, Class-Type attributes, and
  `related to ... by` semantics — is normatively defined in the CQL
  base specification. Add a forward reference to the relevant CQL base
  section.
- **In CQL base (CQL 2.0):** Open (or attach this ticket to) a
  companion change against the base CQL specification to actually do
  the work in items (1)–(6) of Disposition A there, where the new CQL
  2.0 keywords for in-CQL model definition naturally live.

#### Justification

Per JP on the Zulip thread, the in-CQL model-definition mechanism is a
CQL 2.0 base-language feature, not a FHIR-binding concern. Documenting
it in the FHIR-binding spec risks duplicating (and eventually
contradicting) the base spec, and ties FHIR-binding ballot cycles to
base-language editorial work. This split preserves separation of
concerns, keeps the FHIR-binding spec accurate (no over-claiming) in
the short term, and channels the substantive editorial work to the spec
that owns the construct. Trade-off vs. Disposition A: it requires
coordination with the CQL base editors and leaves the FHIR-binding spec
temporarily thinner, but it avoids encoding model-definition semantics
in the wrong specification.

---

### Disposition C: Decline

#### Proposal

Decline on the grounds that full in-CQL model definition is a CQL 2.0
base-language feature still under active design (per JP: "we've not got
that far in the ref imp support for 2.0, and it's not certain that's the
path we'll take"), and that the FHIR-binding spec should not normatively
describe a mechanism whose base-language form is not yet stable. Mark
the ticket *Persuasive with Mod* / *Not Persuasive* and instead amend
*Defining Models* with a single sentence: "This section describes the
components of a model that may be declared in CQL; complete in-CQL
model definitions are deferred to a future version."

#### Justification

Specifying half of an unstable design risks locking the FHIR binding
into syntax or semantics that the base CQL 2.0 work later changes,
forcing a breaking change in a subsequent FHIR ballot. A single
clarifying sentence eliminates the over-promise that motivated the
ticket without committing the spec to details that may shift.

---

### Recommendation

**Recommended disposition:** B

Disposition B best reflects the actual division of labor between the
CQL base specification (where the new "define a model directly in CQL"
keywords live in CQL 2.0) and the FHIR binding (which inherits and
binds those constructs to FHIR). It directly addresses the reporter's
core complaint — that *Defining Models* over-promises relative to what
it documents — by scoping the section honestly and forwarding readers
to the spec that actually defines the mechanism, while still committing
the project to do the substantive editorial work that Disposition A
calls for, just in the right place. Disposition A risks producing
duplicate (and likely divergent) normative text in the FHIR binding;
Disposition C is too light a touch given that, per the linked Zulip
thread, real implementers are already trying to build against this
mechanism today. The workgroup should confirm that a coordinated
change against the CQL base spec is feasible on the current ballot
timeline; if it is not, fall back to the editorial scope-narrowing
half of Disposition B as an interim fix.
