## Summary

The ticket asks the CQL (FHIR) specification authors to add an example to
the *Parameter Constraints* section that demonstrates how to write a
parameter constraint that **rejects null** values, in addition to the
existing examples. The current section uses a `define IsYearly: start of
"Measurement Period" same year as end of "Measurement Period"` example
together with a note that constraints are considered satisfied when the
constraint expression evaluates to `null`. It then shows an `is null`
pattern for checking *required* parameters. The reporter (Chris Moesel)
points out that authors who copy the original `IsYearly` example may
inadvertently allow null `Measurement Period` values to pass the
constraint, and proposes a small companion example that wraps the
boolean expression with `... is true` so that null evaluates to false
and the error is raised. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this issue, so the request
stands on its own as a documentation/example clarification.

## Details

**Description:**

> The Parameter Constraints section uses the following example:
>
> ```
> /*
> @constraint: error
> @message: Measurement period must be a year
> */
> define IsYearly: start of "Measurement Period" same year as end of "Measurement Period"
> ```
>
> It also indicates in the description that constraints are considered
> satisfied if the expression evaluates to `null`. While it provides an
> example of using `is null` to check for *required* parameters, I think
> it may be helpful to first provide an example based on the original
> example showing how you can indicate that null values should be
> rejected, e.g.
>
> ```
> /*
> @constraint: error
> @message: Measurement period must be a year
> */
> define IsYearly: (start of "Measurement Period" same year as end of "Measurement Period") is true
> ```

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new example to the *Parameter Constraints* section of the CQL
(FHIR) specification, immediately after the existing `IsYearly` example
and before the `is null` example for required parameters. The new
example reuses the same constraint metadata but wraps the boolean
expression in `(...) is true`, exactly as the reporter proposes:

```
/*
@constraint: error
@message: Measurement period must be a year
*/
define IsYearly: (start of "Measurement Period" same year as end of "Measurement Period") is true
```

Accompany the example with a short note explaining that, because CQL
uses three-valued logic and a constraint is considered satisfied when
its expression evaluates to `null`, authors who want a missing
`Measurement Period` (or any null sub-expression) to actively *fail*
the constraint should compare the boolean expression to `true` (or
otherwise eliminate `null` from the result), as shown.

#### Justification

The change is purely additive documentation/example work in the
specification narrative — no changes to the CQL grammar, the FHIR CQL
IG conformance resources, or any tooling. It directly addresses a real
authoring pitfall created by CQL's three-valued logic combined with the
"null = satisfied" rule for parameter constraints, and it does so by
mirroring the section's existing example so readers can see the
contrast at a glance. The proposal is minimal, low-risk, and faithful
to the reporter's request.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) showing `(... ) is true`, expand the
*Parameter Constraints* section into a small "Null handling in
constraints" subsection that:

1. Restates the rule that constraints whose expression returns `null`
   are treated as satisfied.
2. Lists the idiomatic patterns for forcing null to fail:
   - `(<boolean expr>) is true` (the reporter's suggestion),
   - `Coalesce(<boolean expr>, false)`,
   - explicit `not (<param> is null) and <boolean expr>`,
   and notes when each is preferable.
3. Keeps the existing `IsYearly` and `is null` examples, but adds the
   `(... ) is true` variant in the new subsection rather than inline.

#### Justification

The underlying issue is broader than the single `IsYearly` example: any
constraint whose body can yield `null` is exposed to the same trap, and
CQL gives authors several equivalent ways to defend against it. A short
named subsection would give implementers a single place to look up
"how do I make this constraint fail on null?" and would make the spec
self-documenting on a known three-valued-logic gotcha. The trade-off is
a slightly larger doc edit and a small editorial decision about which
patterns to recommend; the reporter's exact one-liner becomes one
example among a few, rather than the only one.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing *Parameter
Constraints* section already (a) states the null-is-satisfied rule
explicitly and (b) shows an `is null` pattern, so a reader who has read
the section has the information needed to derive a null-rejecting
constraint. Close as "not persuasive — existing text sufficient".

#### Justification

This is a defensible-but-weak position. It avoids any spec churn and
relies on the principle that the spec defines semantics rather than
enumerating every authoring pattern. The cost is leaving in place a
known foot-gun that an experienced CQL contributor (Chris Moesel) has
flagged as confusing; it puts the burden of avoiding the trap on every
content author. There is no implementation cost or backward-
compatibility concern that motivates declining.

---

### Recommendation

**Recommended disposition:** A

The request is a small, additive, low-risk documentation improvement
proposed by an experienced CQL contributor, addressing a genuine
authoring pitfall that arises from CQL's three-valued logic combined
with the "null is satisfied" rule for parameter constraints. Disposition
A delivers exactly that improvement with minimal editorial overhead and
preserves the section's current shape. Disposition B is a reasonable
expansion and the workgroup may choose to scope-up to it during review,
but it is not necessary to act on this ticket; the workgroup can adopt
A now and open a follow-up if a fuller "null handling in constraints"
subsection is desired. Disposition C is hard to defend given that the
change has essentially no downside.
