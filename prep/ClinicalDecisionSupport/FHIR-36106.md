## Summary

The ticket asks that the `DataRequirement.profile` element (currently
typed as `canonical(StructureDefinition)`) be expanded so that it can
also reference a `Questionnaire` — and, as a follow-on suggestion,
possibly an `ObservationDefinition`. The motivating use case is a
PlanDefinition-driven workflow where the answers to one Questionnaire
(captured via a QuestionnaireResponse) are intended to trigger the
requirement that a second Questionnaire be completed. Because a
QuestionnaireResponse is shaped by its source Questionnaire rather than
by a StructureDefinition profile, today there is no clean way to express
"the data we need is a QuestionnaireResponse for *this specific*
Questionnaire" inside a DataRequirement. Brian Postlethwaite is asking
CDS/CPG-on-FHIR to consider broadening DataRequirement so it can carry a
canonical reference to definitional resources beyond StructureDefinition.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FHIR Augury for this key, so this summary is grounded solely
in the ticket itself and its three comments.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Broaden the type of `DataRequirement.profile` from
`canonical(StructureDefinition)` to a choice / open-target canonical
that also accepts `Questionnaire` and `ObservationDefinition` (and, by
the same argument, `SpecimenDefinition`). Update the element definition
and binding documentation so that when `DataRequirement.type =
"QuestionnaireResponse"` the `profile` slot can carry the canonical of
the Questionnaire that the response must conform to; when `type =
"Observation"` it can carry the canonical of an ObservationDefinition;
and when `type` points at a regular resource, the existing
StructureDefinition behaviour is preserved. Add invariants tying the
permitted target resource type to `DataRequirement.type` and update
PlanDefinition / Library narrative + examples to demonstrate the
QuestionnaireResponse-by-Questionnaire pattern.

#### Justification

This is the literal request and it cleanly enables the described
PlanDefinition-driven workflow where one QuestionnaireResponse triggers
the requirement for another Questionnaire. There is precedent inside
the CDS/CPG family already: ActivityDefinition currently carries
references to definitional resources of this kind, and bryn.rhodes
explicitly noted that handling the relationship inside DataRequirement
"would support that use case more cleanly." Centralising the concept on
DataRequirement keeps PlanDefinition.action.input/output consistent with
Library.dataRequirement and avoids forcing implementers to invent
parallel extensions per use case.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `DataRequirement.profile` typed as
`canonical(StructureDefinition)` and instead introduce a small,
purpose-built extension (e.g. a CDS/CPG extension on DataRequirement
named something like `definition-canonical`) that carries a
`canonical(Questionnaire | ObservationDefinition | SpecimenDefinition)`
and is constrained to be valid only when `DataRequirement.type` is the
matching response/observation/specimen resource. Document the extension
in the CPG-on-FHIR IG with worked examples for the
QuestionnaireResponse-trigger case the reporter described, and keep the
core DataRequirement element semantics unchanged.

#### Justification

Bryn explicitly characterises the literal change as "a fairly
significant expansion of what a data requirement fundamentally is."
Doing this through an IG-owned extension lets the CPG/CDS community
prove the pattern with concrete examples (which Bryn asked the reporter
to produce in the 2022-08-03 comment) without altering the core type of
`DataRequirement.profile` and without forcing every other
DataRequirement consumer to handle a wider canonical target set. If
real-world usage in the IG demonstrates the value, the extension can be
promoted into the base resource in a future normative cycle with
collected evidence in hand. This is also the lower-risk path for
existing implementations of `$data-requirements`,
`$apply`, and Library evaluation, which today assume profile means
StructureDefinition.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by existing mechanisms** and
keep `DataRequirement.profile` typed as `canonical(StructureDefinition)`
only. Direct implementers who need to pin a QuestionnaireResponse to a
specific Questionnaire to use either (a) a StructureDefinition profile
on QuestionnaireResponse that fixes
`QuestionnaireResponse.questionnaire` to the desired canonical, then
reference that profile from `DataRequirement.profile` in the normal
way, or (b) the existing `DataRequirement.codeFilter` /
trigger-condition machinery (e.g. a named-event trigger on
QuestionnaireResponse plus a FHIRPath/CQL guard on
`questionnaire`).

#### Justification

The use case can be expressed today with no spec change: a profile on
QuestionnaireResponse with a fixed `questionnaire` value is exactly the
StructureDefinition shape DataRequirement is built around, and CQL/CDS
expression languages can already inspect `questionnaire` to gate
PlanDefinition actions. Expanding `DataRequirement.profile` to permit
multiple definitional resource targets enlarges the conceptual surface
of a heavily-used type and forces every DataRequirement consumer
(evaluators, validators, $data-requirements implementations) to handle
a polymorphic target. Without a worked-through example — which Bryn
asked the reporter to provide and which has not, per the comment trail,
materialised — the cost/benefit does not yet justify the change.

---

### Recommendation

**Recommended disposition:** B

Disposition B (introduce a CPG-on-FHIR extension on DataRequirement that
carries a `canonical(Questionnaire | ObservationDefinition |
SpecimenDefinition)`) best fits the actual state of the discussion on
this ticket. The reporter has identified a real and recurring need;
bryn.rhodes has confirmed in two CPG-on-FHIR comments that the use case
is interesting, that DataRequirement is the conceptually right home,
and that a concrete example is still owed before the workgroup can
evaluate impact. An IG-level extension lets CDS/CPG-on-FHIR build and
publish that worked example immediately, gather implementer feedback,
and avoid making a structural change to a widely-implemented core type
ahead of evidence. Disposition A remains the right long-term target
once the extension has proved itself; Disposition C understates the
real ergonomic gap (profiling QuestionnaireResponse by fixing
`questionnaire` works but is awkward for the trigger use case the
reporter described). Recommend that CDS take this back to CPG-on-FHIR,
draft the extension and a PlanDefinition example built around the
QuestionnaireResponse-triggers-next-Questionnaire scenario, and revisit
core-resource promotion in a future cycle.
