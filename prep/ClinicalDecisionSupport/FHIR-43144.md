## Summary

Lloyd McKenzie (reporter) flags that two extensions defined in the
**US PACIO Cognitive Status (FHIR)** implementation guide declare an overly
permissive context — namely `Element` — even though the semantics described
in those extensions' definitions clearly do not apply "anywhere" in a FHIR
instance. The reporter asks the Clinical Decision Support work group (acting
as the sponsoring WG for this PACIO-derived guide) to constrain each
extension's `StructureDefinition.context` to the specific resource(s) and
element(s) where the extension is actually meaningful.

The ticket does not name the two extensions. Augury surfaced **no
cross-referenced Jira tickets, Zulip threads, or GitHub items**, and a
direct keyword search for `FHIR-43144` and for "PACIO Cognitive Status
extension context" returned zero hits across the indexed sources. The
ticket therefore stands on its own: the request is a generic but
well-known modeling-correctness concern (`context = Element` is rarely
appropriate for an extension that describes a domain-specific
attribute), and the disposition work consists of (a) identifying the two
offending extensions in the published IG and (b) replacing the
`Element`-typed context entries with one or more `fhirpath`/`element`
contexts that match the extension's intended use.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Constrain the `context` of each of the two flagged extensions in the
US PACIO Cognitive Status (FHIR) IG so that
`StructureDefinition.context.type = element` only when paired with a
specific `expression` that names the resource and element where the
extension legitimately applies. Concretely:

1. Identify the two `StructureDefinition` resources in the IG whose
   `context[].expression = "Element"` (i.e., the literal type `Element`,
   not a domain element). These are the extensions called out in the
   ticket.
2. For each, replace the single `{type: "element", expression:
   "Element"}` entry with one or more entries that name the actual
   target — e.g., `{type: "element", expression: "Observation"}`,
   `{type: "element", expression: "Observation.component"}`, or
   `{type: "fhirpath", expression: "..."}` as appropriate to the
   extension's documented intent.
3. Regenerate the IG (Sushi/Publisher), update narrative, and add a
   short note in the IG change log referencing FHIR-43144.

#### Justification

`context.expression = "Element"` is the broadest legal context an
extension can declare and effectively says "this extension may appear
on any element of any resource". For a domain-specific PACIO Cognitive
Status extension, that is almost certainly wrong: it weakens validation,
makes downstream profiling harder, and invites misuse by implementers
who match on the loose context. The fix is mechanical, low-risk, and
aligns with general IG hygiene guidance the methodology group has
issued for years (extensions should be scoped to their meaningful
targets). The reporter (Lloyd McKenzie) has long-standing expertise in
extension design and is requesting exactly the standard correction.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (or in addition to) tightening the two extensions in place,
review the IG's full extension catalogue and apply a consistent
context-scoping policy:

- Audit every `StructureDefinition` of `kind = complex-type` and
  `type = Extension` defined by the PACIO Cognitive Status IG.
- For any extension whose `context` resolves to `Element` or to a base
  type broader than its semantics warrant, narrow the context (as in
  Disposition A) **and** add a `comment` on each `context` entry
  explaining why the targets were chosen.
- Where two or more extensions share the same intended target set,
  consider replacing them with a single extension that has a richer
  `value[x]` (e.g., a `CodeableConcept` discriminator) plus a slicing
  rule on the parent, rather than carrying parallel narrowly-scoped
  extensions.

#### Justification

Treating FHIR-43144 as one instance of a class of issues catches the
two specific extensions the reporter noticed *and* surfaces any other
`context = Element` entries that would otherwise be reported in a
follow-up ticket. The cost is a one-time review pass; the benefit is
that the IG arrives at its next ballot with a coherent extension story
rather than a mix of tightly- and loosely-scoped contexts. The
trade-off vs. A is scope: B requires WG bandwidth to do the audit and
may touch more `StructureDefinition`s than the reporter strictly asked
for, which can slow publication if the IG is already on a tight
release cadence.

---

### Disposition C: Decline (Persuasive — wrong WG / route to PACIO)

#### Proposal

Decline the change in CDS as a *persuasive* — the substantive request
is sound, but the affected artifacts live in the PACIO project's
US PACIO Cognitive Status IG, which is maintained outside the HL7
Clinical Decision Support work group's directly-owned source tree.
Re-route the ticket to the PACIO Cognitive Status IG editors (or the
appropriate sponsoring HL7 work group for that IG) with a
recommendation that they apply the fix described in Disposition A.

#### Justification

The ticket landed against CDS because the IG's tracker is configured
that way, but the corrective action — editing
`StructureDefinition.context` on two specific extensions — must be
made in the PACIO IG repository, not in any artifact CDS publishes
directly. Routing to the actual maintainers shortens the path to a
fix and avoids a CDS-owned disposition that cannot be implemented from
within CDS. Decline-as-not-persuasive would not be appropriate, since
the technical request is correct; this is purely about ownership.

---

### Recommendation

**Recommended disposition:** **A (Accept As Requested)**, executed by
the PACIO Cognitive Status IG editors with CDS confirming the fix at
the next reconciliation meeting.

The request is small, technically uncontroversial, and aligns with
long-standing FHIR extension-design guidance: `context = Element` is
almost never appropriate for a domain-specific extension, and PACIO
Cognitive Status extensions are by definition domain-specific. The
two-step Disposition B audit is attractive in principle but should not
block the narrow fix the reporter actually asked for; if the IG
editors have time, B can run as a follow-up housekeeping pass without
re-opening this ticket. Disposition C correctly identifies that the
edits happen outside the CDS-owned tree, but in practice CDS is the
sponsoring WG for record on this Jira project and should still own
the disposition decision while delegating the mechanical edit.

The main caveat — and the reason the workgroup will need a second
pass before this is truly closed — is that **the two affected
extensions are not named in the ticket and the PACIO Cognitive Status
IG is not indexed in FHIR Augury**. Before voting Persuasive, ask the
reporter (or an IG editor) to identify the two `StructureDefinition`
URLs so the fix can be verified against the published IG.
