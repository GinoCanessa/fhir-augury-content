## Summary

The ticket requests the addition of an `applicableBehavior` (resolved as
`cqf-applicabilityBehavior`) extension on `PlanDefinition.action` in the
Clinical Guidelines (CPG / cqf-extensions) specification. Today, `$apply`
semantics in PlanDefinition evaluate *every* applicable action
independently, which forces authors to encode mutually-exclusive branches
by expressing both `A` and `not A` in each child action's applicability
condition. The proposed extension lets an action declare that its child
actions should be evaluated in order and that the **first** child whose
applicability evaluates to `true` wins (an `any` / "else-if" behavior),
while the current "all applicable children fire" behavior remains the
default (`all`).

This is the **R4 extension counterpart** of
[FHIR-50490](https://jira.hl7.org/browse/FHIR-50490), which adds the
same capability natively to `PlanDefinition.action.applicabilityBehavior`
in FHIR R6 core. The reporter explicitly notes that a cross-version
extension is *not* appropriate because the feature must be usable in R4
before R6 publishes. The resolution proposes a `0..1 code` extension
bound required to a new code system at
`http://hl7.org/fhir/action-applicability-behavior` with the values
`all` and `any`.

## Details

**Description:**

> Current PlanDefinition $apply semantics are that all actions that are
> "applicable" are evaluated. However, there is no current mechanism for
> specifying an "else", so that requires PlanDefinitions to express A
> and not A for all decisions. Add an "applicableBehavior" extension
> that can be used on an "action" to indicate that the first "action"
> that returns true is the "applicable sub-action". Current behavior is
> "all", new behavior would be characterized by "any".

**Resolution (Persuasive):**

> Agreed, add
>
> `cqf-applicabilityBehavior 0..1 code`
> - short: How applicability should be determined
> - definition: Indicates how applicability of child actions should be
>   interpreted. `All`, the default behavior, indicates that all actions
>   where the applicability logic evaluates to true should be applied,
>   whereas `Any` indicates that actions should be evaluated in order,
>   with the first action where the applicability logic evaluates to
>   true being applied. Once an action has been determined applicable,
>   processing stops.
>
> The extension would be bound required to a terminology defined in the
> extensions pack:
> `http://hl7.org/fhir/action-applicability-behavior`
> - `all` — All: applicability of each child action is evaluated
>   independently, and each action that returns true is applied.
> - `any` — Any: child actions are evaluated in order, and the first
>   action that returns an applicability of true is applied; processing
>   of the parent action stops.
>
> Note that this is the R4 implementation of the R6 feature from
> FHIR-50490. We are explicitly not using a cross-version extension
> here because the feature must be available in R4 prior to the
> publication of R6.

**Comments:**

### bryn.rhodes (2025-03-31)

Supplies a concrete illustration. Under the proposed `any` behavior, a
measles-immunization PlanDefinition can be authored as a nested tree of
sibling actions ordered from most to least specific:

```
PlanDefinition applicableBehavior = 'any'
"Measles Routine Immunization Schedule Incomplete"
    "No Primary Series Doses Administered"
        "Client Age Less Than 12 Months"
        "Last Live Vaccine Administered Within 4 Weeks"
        "Client Is Due For MCV12"
    "One Primary Series Dose Administered"
        "Client Age Less Than 15 Months"
        "Last Live Vaccine Administered Within 4 Weeks"
        "Client Is Due For MCV2"
    "Two Primary Series Doses Administered"
```

versus the equivalent logic today (with `all` semantics), which requires
each branch to explicitly re-assert the negation of its predecessors'
applicability conditions (`if not "Client Age Less Than 12 Months" then
...`). The comment is the core motivating example for why the extension
materially reduces authoring complexity.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution text verbatim. Concretely, in the Clinical
Guidelines / cqf-extensions extension pack:

1. Define a new extension **`cqf-applicabilityBehavior`**:
   - Context: `PlanDefinition.action` (and
     `ActivityDefinition`/`RequestOrchestration`-style action elements
     where CPG already layers parallel extensions, if consistent with
     the pack's existing pattern).
   - Cardinality: `0..1`.
   - Value type: `code`.
   - Binding: **required** to a new value set backed by a new code
     system `http://hl7.org/fhir/action-applicability-behavior` with
     codes `all` (default; evaluate every applicable child
     independently) and `any` (evaluate children in order, apply the
     first whose applicability is true, then stop processing the parent
     action).
   - Short: "How applicability should be determined."
   - Definition: as quoted in the resolution above.
2. Publish the code system and value set alongside the extension in the
   cqf-extensions IG.
3. Add narrative in the CPG $apply operation guidance describing how
   engines MUST honor `any` semantics (short-circuit on first true
   child, respect document order of sibling actions) and how absence of
   the extension preserves today's `all` semantics (fully backward
   compatible).
4. Cross-link the extension page to **FHIR-50490** / R6
   `PlanDefinition.action.applicabilityBehavior` so implementers know
   the extension is a forward-compatible stand-in for the R6 element.

#### Justification

The request is design-consistent with the broader CPG pattern of
exposing behavioral controls on `PlanDefinition.action` via extensions
(e.g., `groupingBehavior`, `selectionBehavior`, `requiredBehavior`,
`precheckBehavior`, `cardinalityBehavior`). It closes a concrete
authoring pain point (expressing `else` branches requires duplicating
and negating predecessor applicability expressions), which the
reporter's measles-immunization example demonstrates. The workgroup
already agreed to the identical semantics in R6 core via FHIR-50490
(status *Applied*), so adopting the R4 extension avoids a specification
gap for the 18–24 months before R6 publishes and keeps R4 CPG content
forward-compatible with R6. Because `all` remains the default, the
change is non-breaking for existing PlanDefinitions and existing $apply
implementations.

---

### Disposition B: Alternative Approach

#### Proposal

Publish the extension exactly as in Disposition A, but tighten the
spec contract to mitigate the "ordering" risk that was flagged in the
sibling R6 ticket (FHIR-50490) rather than inheriting its silence on
order:

1. Bind `cqf-applicabilityBehavior` required to `{all, any}` (same as
   A).
2. **Additionally** normatively state in the extension definition that
   when an `action` carries `applicabilityBehavior = any`, child
   actions MUST be evaluated in document order and engines MUST treat
   child order as semantically significant *for that action*. Leave
   ordering unspecified when the extension is absent or `all` (status
   quo).
3. Optionally, pair the extension with a companion extension
   `cqf-orderMeaning` on `PlanDefinition.action` (or include
   `orderMeaning` guidance in the extension narrative) that mirrors the
   `orderMeaning` clarification FHIR-50490 added to R6 core — so CPG R4
   authors have a single place to assert "these siblings are ordered".
4. Provide a worked example (the measles example from the reporter) in
   the extension page.

#### Justification

This preserves full backward compatibility (silent `action`s are
unchanged) while avoiding a trap: FHIR-50490 acknowledges that asserting
action order is *theoretically* a breaking change in R6 core. In R4, we
don't have the luxury of revising `PlanDefinition.action` normative
text, but we can scope the ordering assertion to only those actions that
explicitly opt into `any`. That keeps the extension purely additive for
existing content while giving authors and engines a crisp contract where
it matters. It also reduces the risk that an implementer silently
re-orders siblings during transformation (e.g., canonical JSON, XSLT,
FHIRPath-driven tooling) for actions that rely on `any` semantics.

---

### Disposition C: Decline

#### Proposal

Do not add `cqf-applicabilityBehavior`. Direct authors to continue
expressing mutually-exclusive branches by encoding negated
applicability conditions (`A` and `not A`) in sibling actions today,
and to wait for FHIR-50490's native
`PlanDefinition.action.applicabilityBehavior` in R6 core. Rationale
category: **already addressed by R6 core**.

#### Justification

FHIR-50490 is `Applied` in R6 core, so the capability will exist in the
next major release without introducing another CPG extension that must
be retired or aliased once R6 ships. Adding a parallel R4 extension now
creates a migration footprint (content authored against
`cqf-applicabilityBehavior` must be rewritten to the native R6 element,
or a cross-version mapping must be maintained), and the existing
workaround (explicit negated applicability) is verbose but functionally
complete.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the working-group's already-expressed agreement
(Resolution: Persuasive), and declining (C) is not credible given the
concrete authoring burden the reporter documents and the fact that R6
will ship the feature natively — R4 CPG content cannot wait 18–24
months for that to be usable. Between A and B, B costs very little
(one extra normative paragraph plus an optional worked example) and
eliminates the single most predictable interoperability failure mode:
engines or transformation tooling that do not preserve sibling-action
order silently corrupting `any`-semantics PlanDefinitions. Scoping the
"actions are ordered" assertion to the `any` case keeps the change
surgically additive, which is exactly the property the CPG pack
requires from an R4 extension, and it aligns neatly with the
`orderMeaning` clarification FHIR-50490 is adding to R6 core so that
the R4-to-R6 migration story is a straight structural lift rather than
a semantic renegotiation.
