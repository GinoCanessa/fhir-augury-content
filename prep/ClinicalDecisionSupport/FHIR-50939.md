## Summary

The reporter, writing against the *Using CQL With FHIR* specification,
asks the Clinical Decision Support work group to add documentation
clarifying the precision mismatch between CQL system types and the
FHIR primitive date/time types. Specifically, the request is to
document that millisecond values that can legitimately be carried in a
CQL `System.DateTime` or `System.Time` value cannot be represented in
the corresponding FHIR `dateTime` / `time` primitive types when those
values flow back from CQL into a FHIR resource. The single comment on
the ticket — left by the reporter himself — notes that a follow-up
discussion on Zulip clarified that the round-trip *is* in fact
possible, and asks for the ticket to be closed. No related Jira,
Zulip, or GitHub items were surfaced by the cross-reference index or
by direct search, so the Zulip clarification referenced by the
reporter could not be located through `fhir-augury-cli`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a short clarifying note to the *Using CQL With FHIR* IG, in the
section that describes the mapping between CQL system types and FHIR
primitive types (typically the "Type Mapping" / model-info discussion
covering `System.DateTime` ↔ `FHIR.dateTime` and `System.Time` ↔
`FHIR.time`). The note should:

- State the precision capabilities of `System.DateTime` and
  `System.Time` (sub-second / fractional-second precision).
- State the precision capabilities of FHIR `dateTime` and FHIR `time`
  (which permit fractional seconds via the `time` regex
  `([01][0-9]|2[0-3]):[0-5][0-9]:[0-5][0-9](\.[0-9]+)?` and the
  analogous `dateTime` regex), and explicitly call out that values
  flow without loss when both sides honor the fractional-second
  component.
- Add a worked example showing a `System.DateTime` value with
  millisecond precision being serialized into a FHIR `dateTime`
  primitive, demonstrating the round-trip preserves the milliseconds.

This treats the ticket as a documentation-only enhancement targeted
at the IG narrative, with no normative or schema change.

#### Justification

The reporter's own follow-up comment indicates that the underlying
*technical* concern was resolved on Zulip (i.e., FHIR `dateTime` and
`time` *can* carry fractional seconds, so the round-trip from CQL is
possible). The remaining value of the ticket is therefore purely
documentary: making the precision story explicit prevents the next
implementer from reaching the same incorrect conclusion that prompted
this ticket. Authoring a small, targeted clarification in the type
mapping section is consistent with the IG's role as the canonical
reference for the CQL ↔ FHIR boundary, and is the lowest-risk way to
prevent recurrence.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding free-text narrative, extend the IG's mapping table
between CQL system types and FHIR primitive types with an explicit
"precision" column (or footnote) for each row, documenting:

- The precision granularity supported by the CQL type.
- The precision granularity supported by the corresponding FHIR type.
- Whether values round-trip without loss, and any edge cases (e.g.,
  timezone offsets, partial dates, leap-second handling).

Apply this uniformly across all date/time-shaped mappings
(`System.Date` ↔ `FHIR.date`, `System.DateTime` ↔ `FHIR.dateTime`,
`System.Time` ↔ `FHIR.time`, `System.Quantity` of UCUM time units
where relevant) so that the table itself answers precision questions
without requiring readers to consult prose.

#### Justification

A structured column generalises beyond the specific milliseconds
question and pre-empts the same class of question for the other
date/time primitives (date-only, partial dates, timezone-bearing
values), which historically generate similar confusion. It also
formats the answer in a way implementers are likely to look at first
(the mapping table) rather than buried in a paragraph. The cost is a
slightly larger doc edit and a new column to maintain, but precision
data is stable and unlikely to churn.

---

### Disposition C: Decline (Close as "Not Persuasive — No Change" /
"Already Documented")

#### Proposal

Close the ticket without changing the IG, on the basis that:

- The reporter himself has stated on the ticket that the Zulip
  discussion clarified the situation and asked for closure.
- The FHIR primitive `dateTime` and `time` regexes already explicitly
  permit a fractional-second component, so the IG's existing
  reference to FHIR primitives is, strictly speaking, sufficient to
  reach the right answer.

If the work group agrees, resolve as "Not Persuasive With Mod" /
"Persuasive With Mod" only to the extent of adding a one-line link
from the type mapping section to the FHIR primitive type definitions,
or simply close as "Not Persuasive" honoring the reporter's request.

#### Justification

The reporter has explicitly asked for the ticket to be closed once
the Zulip clarification landed, and the underlying spec text is
already correct. Spending IG-editor and balloter time on a
documentation tweak that the reporter no longer needs has limited
return, particularly when the broader CDS / CQL backlog contains
other open items. Declining preserves work-group bandwidth without
introducing any technical risk.

---

### Recommendation

**Recommended disposition:** A

Although the reporter has asked for the ticket to be closed, the
underlying concern — that an implementer could reasonably look at the
*Using CQL With FHIR* IG and conclude that millisecond precision is
lost across the boundary — is real, and the cheapest durable fix is a
small, targeted clarification in the type mapping section
(Disposition A). Closing without any documentation change
(Disposition C) leaves the IG in exactly the state that produced the
ticket, inviting a future reporter to file the same issue. The
broader, table-shaped overhaul in Disposition B is attractive but is
larger in scope than the ticket warrants and should arguably be
opened as a separate enhancement; doing it now risks bundling
unrelated editorial decisions into a single change. Disposition A
captures the value the reporter implicitly identified (the gap in
narrative that drove him to Zulip in the first place) at minimal
cost, and leaves the door open to adopt B later if the work group
decides a structural overhaul is warranted.
