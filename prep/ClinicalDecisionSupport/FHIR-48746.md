## Summary

The ticket asks the CRMI specification to use "the new capability statement
features" (i.e. the R5+ CapabilityStatement machinery for advertising
operation-level features and expectations) to let a CRMI Artifact
Repository declare which **packaging capabilities** it supports — most
naturally, which values of the `$package` operation's `capability`
parameter (e.g. `shareable`, `computable`, `publishable`, `executable`)
the server can honour.

This sits alongside two earlier, now-applied CRMI tickets that frame the
problem space:

- [FHIR-32643](https://jira.hl7.org/browse/FHIR-32643) introduced the
  family of artifact-repository capability statements
  (`ShareableArtifactRepository`, `PublishableArtifactRepository`,
  `AuthoringArtifactRepository`) into CRMI.
- [FHIR-48745](https://jira.hl7.org/browse/FHIR-48745) clarified the
  default behaviour of `$package` when **no** `capability` parameter
  values are supplied (server-decided). FHIR-48746 is the natural
  follow-on: if the server gets to decide, clients need a discoverable
  way to find out what it actually supports.

No companion Zulip discussion, GitHub PR, or formally-linked Jira issue
was recorded against this ticket in the index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **HL7/crmi-ig** Implementation Guide, extend the existing
artifact-repository CapabilityStatements
(`ShareableArtifactRepository`, `PublishableArtifactRepository`, and —
where applicable — `AuthoringArtifactRepository`) to formally advertise
the set of `$package` `capability` values the server supports. Use the
"new capability statement features" the ticket alludes to:

- On each `CapabilityStatement.rest.resource.operation` entry that
  references the `$package` `OperationDefinition`, attach
  `extension[capabilitystatement-expectation]` (`SHALL` / `SHOULD` /
  `MAY`) to indicate that supporting the `capability` parameter is
  required, and
- Attach a new (or reused) `capabilitystatement-supported-feature`-style
  extension whose value enumerates the supported `capability` codes
  (`shareable`, `computable`, `publishable`, `executable`, `extensible`,
  …) drawn from the existing `crmi-artifact-capability` value set used
  by the `$package` parameter.
- Update the `$package` `OperationDefinition` narrative and the
  CRMI capability-statements page to point at the new advertisement
  mechanism, mirroring the pattern adopted for FHIR-43716 (manifest
  versioning as MAY).

#### Justification

This is precisely what the reporter requested, fits the precedent set by
FHIR-43716, and closes the discoverability gap opened by FHIR-48745
(when `capability` is omitted, the server decides — clients need a way
to know what that means in practice). Reusing the
`capabilitystatement-expectation` and feature-advertisement extensions
keeps CRMI consistent with how the rest of the FHIR Core ecosystem
declares optional-but-discoverable behaviour on operations, and it
avoids inventing a CRMI-specific mechanism for something Core already
solves.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than extending each existing artifact-repository
CapabilityStatement, define a dedicated **`crmi-package-capabilities`**
CapabilityStatement extension (or a small profile on
`CapabilityStatement.rest.resource.operation`) that declares, for any
operation that takes a `capability` parameter, the supported capability
codes. Reference this extension from the `$package`
`OperationDefinition` and from the CRMI capability-statements page,
and apply it to the existing repository CapabilityStatements as a
secondary step. Keep the value set (`crmi-artifact-capability`) as the
single source of truth for the codes themselves.

#### Justification

The `$package` operation is also defined for terminology services and
may be reused by future CRMI repository flavours; pinning the
advertisement mechanism to a reusable extension rather than ad-hoc
edits to three (or more) CapabilityStatements reduces duplication and
keeps the authoring surface in one place. It also avoids piling
additional R5-only `expectation`/feature extensions onto Shareable
profiles that may be implemented against R4 servers — the dedicated
extension can carry a clear FHIR-version conformance note. Trade-off
versus Disposition A: one extra extension to author, define, and
publish, in exchange for a single canonical place to look up "what
capabilities does this server support on `$package`?".

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the grounds that the discoverability need is
already satisfied by the combination of (a) the existing CRMI
artifact-repository CapabilityStatements, which declare which
operations the server supports, (b) the `crmi-artifact-capability`
value set, which already enumerates the capability codes, and (c) the
clarification from FHIR-48745 that the server may decide when the
caller does not specify capabilities. Clients that need to know
specific behaviour can negotiate at runtime by inspecting the
`Parameters` resource returned by `$package`.

#### Justification

This is defensible only if the CDS work group concludes that the
existing combination is "good enough" and that adding a formal
advertisement mechanism imposes more burden on implementers than the
benefit of static discoverability is worth. It avoids any churn in the
CapabilityStatements and operation definition.

---

### Recommendation

**Recommended disposition:** A

The reporter's intent is narrowly scoped — surface server-supported
`$package` capability values on the existing repository
CapabilityStatements — and CRMI already has the precedent
(FHIR-43716) for using `capabilitystatement-expectation` extensions to
mark optional CRMI features. FHIR-48745 has just made the gap material:
clients now know that the server decides when `capability` is omitted,
but have no static way to tell *what* it will decide. Disposition A
closes that loop with the smallest, most idiomatic change and reuses
infrastructure (expectation extensions + the existing
`crmi-artifact-capability` value set) that downstream tooling already
understands. Disposition B is worth raising in discussion if the work
group already foresees other operations needing the same mechanism, in
which case it is the cleaner long-term shape. Disposition C should
only carry the day if the work group judges that static advertisement
is not worth the authoring cost — but given the trajectory established
by FHIR-32643, FHIR-43716, and FHIR-48745, that seems unlikely.
