## Summary

The ticket asks the Clinical Decision Support work group to extend the
"Using FHIR Clinical Reasoning with CDS Hooks" guidance so that the
request/response mappings cover features introduced in **CDS Hooks 1.1**.
Specifically, the description names two new card-level concepts that need
FHIR Clinical Reasoning mappings:

- `isRecommended` → proposed mapping to `PlanDefinition.precheckBehavior`
- `card.topic` → proposed mapping to `PlanDefinition.topic` and
  `ActivityDefinition.topic`, with the open question of how to express an
  equivalent at the *Request* level (the reporter suggests a
  request-level extension for `topic`).

A reverted earlier resolution (originally Persuasive 16-0-0 on 2021-03-24,
re-voted 20-0-0 by Gail Winters / Chris Schuler in November 2025) widens
the scope to the full CDS Hooks 1.1 feature delta, including:

- System actions
- Feedback support and override reasons
- `isRecommended` and `topic` on cards
- Permitting `userId` to be a `PractitionerRole`
- Adding a `Coding` type to simplify use from object-oriented languages

The ticket has been re-opened and labelled `CDS-IG`, indicating the work
should land in the CDS-IG implementation guide rather than core FHIR.
The cross-reference index surfaced only FHIR artifact mentions
(`PlanDefinition.precheckBehavior`, `PlanDefinition.topic`,
`ActivityDefinition.topic`); no other Jira tickets, Zulip threads, or
GitHub items were linked to this ticket via the augury cross-reference
data.

### Disposition A: Accept As Requested

#### Proposal

Update the CDS Hooks ⇄ FHIR Clinical Reasoning request/response mapping
section of the CDS-IG to add the full CDS Hooks 1.1 delta agreed in the
2025-11-05 (20-0-0) re-vote:

1. **Card-level additions**
   - Map `card.isRecommended` (boolean) → `PlanDefinition.action.precheckBehavior`
     (specifically the `yes` value when `isRecommended = true`; absence/`false`
     ⇒ default behavior). Document the round-trip in both the
     "Request → PlanDefinition" and "PlanDefinition → Card" tables.
   - Map `card.topic` (Coding) → `PlanDefinition.topic` /
     `ActivityDefinition.topic`. For the Request side (where there is
     no native `topic` element), define a CDS-IG–owned extension
     `http://hl7.org/fhir/uv/cds-hooks/StructureDefinition/topic`
     (cardinality 0..*, value `Coding`) and apply it to the
     RequestGroup / Task / ServiceRequest profiles used by the IG so the
     topic survives the round trip.
2. **System actions** — add a new "System actions" subsection mapping
   `systemActions[]` → `RequestGroup.action` (or the IG's
   already-defined Action profile) with `intent = order` and
   `doNotPerform` semantics matching the hook's `delete`/`update`
   action types.
3. **Feedback / override reasons** — add a "Feedback service" mapping
   table that links the CDS Hooks 1.1 Feedback object
   (`card.uuid`, `outcome`, `acceptedSuggestions`, `overrideReason`) to
   `Task.statusReason` / `Provenance.reason` and to the existing
   GuidanceResponse mapping, with override-reason `Coding` carried as
   `GuidanceResponse.reasonCode`.
4. **UserId Practitioner Role** — relax the existing `context.userId`
   mapping table so that `userId` may resolve to either `Practitioner`
   *or* `PractitionerRole`; update the example payloads accordingly.
5. **Coding convenience type** — note in the Coding section that CDS
   Hooks 1.1 introduces a top-level `Coding` type that is structurally
   identical to FHIR `Coding`, and document that no mapping change is
   required beyond pass-through.

#### Justification

This is the disposition the workgroup already voted Persuasive twice
(16-0-0 in 2021, 20-0-0 in 2025), so the substance has community
backing. The mappings reuse existing Clinical Reasoning machinery
(`PlanDefinition.action.precheckBehavior`, `PlanDefinition.topic`,
`RequestGroup.action`, `GuidanceResponse.reasonCode`) which keeps the
IG internally consistent and avoids inventing parallel concepts. The
ticket's `CDS-IG` label and the "Move to CDS-IG" comment confirm that
the changes belong in the IG and are non-normative for core FHIR, which
keeps the change Compatible/substantive (the impact category recorded
on the original disposition).

### Disposition B: Alternative Approach

#### Proposal

Accept the *intent* of the ticket but split the work:

1. **Land now (Accept narrow):** the two items literally in the
   description — `isRecommended` → `PlanDefinition.precheckBehavior`
   and `card.topic` → `PlanDefinition.topic` / `ActivityDefinition.topic`
   plus a single new CDS-IG extension for request-level `topic`.
2. **Defer to follow-up tickets:** open separate tracking tickets (one
   each) for system actions, feedback / override reasons, the
   `userId = PractitionerRole` relaxation, and the `Coding` type note,
   referencing FHIR-28592 as the parent.

Each follow-up ticket would carry its own example payloads and round-trip
test, and the parent ticket would close when (1) lands.

#### Justification

The 2025-11-05 re-vote bundled five independently-scoped changes onto a
ticket whose original description names only two. Splitting them keeps
the audit trail honest, lets each sub-feature be reviewed and balloted
on its own merits, and avoids holding the well-understood
`isRecommended`/`topic` mappings hostage to the more nuanced design
work for system-action and feedback round-tripping (both of which need
profile decisions, not just mapping table entries). The trade-off vs.
Disposition A is loss of velocity — five tickets must each be
shepherded — but the published guidance lands incrementally and is
easier for implementers to consume.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on scope grounds: close as **Not Persuasive — out
of scope for core FHIR**, and ask the CDS Hooks specification (which
already publishes its own `PlanDefinition`/`RequestGroup` mapping
guidance for newer features) to maintain the request/response mapping
in its own publication rather than duplicating it in HL7's CDS-IG.

#### Justification

CDS Hooks is published by HL7's CDS WG outside the core FHIR
specification and ships its own mapping guidance for each release.
Mirroring that mapping in a separate FHIR-side IG creates two sources
of truth that drift over time (this very ticket has been open for five
years across CDS Hooks 1.1, 1.2, and 2.0 cycles). Declining and
pointing implementers to the canonical CDS Hooks docs avoids that
drift. The downside is that FHIR Clinical Reasoning loses an
authoritative bridge document, which is why this is the weakest of the
three options.

---

### Recommendation

**Recommended disposition:** B

The 2025-11-05 re-vote (20-0-0) makes clear the WG wants the broader
1.1 mapping, so straight Decline (C) goes against the room. But the
gap between the ticket's two-item description and the five-item agreed
scope is large enough that a single ticket / single PR makes review
unwieldy and history hard to read. Disposition B preserves the
workgroup's intent — every agreed item still lands — while giving each
sub-feature a clean ticket, a clean ballot row, and (importantly) a
clean place to capture the not-yet-resolved design questions
(request-level `topic` extension shape, system-action profiling,
feedback/override-reason round-trip target). Disposition A is a fine
fallback if the WG prefers a single combined PR; in that case, treat
the five bullets in this ticket's description as a checklist on the
implementing PR.
