## Summary

Bryn Rhodes (Clinical Decision Support WG) is requesting a small, purely
editorial change to the CQL grammar: move the `library` block to be the
first rule in the ANTLR grammar file (`CQL.g4`). The request is motivated
by tooling rather than by any change to language semantics — the ANTLR
plugin treats the *first* rule in a grammar as the start rule, so making
`library` the first rule lets the plugin operate naturally on CQL
artifacts. The resolution recorded on the ticket is "Persuasive — agreed,
change the `library` rule to be the first rule in the CQL.g4 file," and a
fix has already been published as
[HL7/cql#88](https://github.com/HL7/cql/pull/88). There are no
cross-referenced Jira tickets — the only cross-reference returned by
FhirAugury is the PR linked above. Because the change is local to the
grammar file and does not affect the generated parser's behavior, the
ticket is essentially a low-risk maintenance item awaiting workgroup
ratification of the already-prepared PR.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the change exactly as proposed and as already prepared in
[HL7/cql#88](https://github.com/HL7/cql/pull/88): in the canonical CQL
grammar file (`CQL.g4` in the `HL7/cql` repository), move the `library`
rule definition so that it is the *first* rule in the file. No other
grammar rules, semantics, or downstream code are modified. Regenerate any
derived artifacts (railroad diagrams, images) that depend on rule
ordering and confirm the grammar still compiles cleanly under ANTLR.

#### Justification

The author of the request (Bryn Rhodes) is also the steward of the CQL
grammar, the change has already been recorded on the ticket as
"Persuasive," and a fix PR is in hand. The change is purely editorial:
ANTLR treats the first rule as the default start rule, so making
`library` first aligns the canonical grammar with how CQL artifacts are
actually parsed in practice (a CQL artifact begins with a `library`
declaration). Generated parser code is unaffected by rule ordering, so
there is no behavioral regression risk for downstream implementers. This
matches the FHIR design principle of preferring small, low-risk
editorial fixes when they improve developer ergonomics without changing
semantics.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the intent of the request but, instead of relying on rule
ordering to declare the start rule, add an explicit grammar-level note
(comment block at the top of `CQL.g4`) stating that `library` is the
intended start rule for a CQL artifact, and document the ANTLR plugin's
"first rule = start rule" convention in the CQL spec's
"Grammar / Tooling" section. Optionally combine this with reordering
(as in Disposition A) so that both human readers and tooling get the
same answer, but make the documentation — not the ordering — the
normative statement.

#### Justification

Rule ordering inside an ANTLR grammar is not a normative concept in CQL;
the semantics are unaffected. Encoding "what the start rule is" as a
side effect of source ordering is fragile — a future editorial reshuffle
could silently change tool behavior. Documenting `library` as the
intended start rule makes the requirement explicit and survives any
future grammar refactor. The cost is small (a comment block + a short
spec paragraph) and compatible with also doing the reorder in
Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline to change the canonical `CQL.g4` for this purpose, and instead
recommend that ANTLR-plugin users configure the start rule explicitly
(e.g., via the plugin's "start rule" setting) when working with the CQL
grammar, leaving the grammar source untouched.

#### Justification

CQL is a normative HL7 specification; even editorial changes to the
canonical grammar carry process overhead (re-publishing artifacts,
notifying implementers, regenerating derived diagrams). If the only
motivation is convenience inside one tool — a tool whose default
behavior can be overridden in configuration — the most defensible
position is to leave the spec alone and configure the tool. This is the
"avoid touching the standard for tool ergonomics" stance.

---

### Recommendation

**Recommended disposition:** A

The request is small, semantically inert, has already been resolved as
"Persuasive" by the workgroup, and the implementing PR
([HL7/cql#88](https://github.com/HL7/cql/pull/88)) already exists. There
is no implementer impact (generated parsers are unchanged), and the
change brings the canonical grammar into alignment with how CQL
artifacts are actually structured (a CQL document starts with a
`library` declaration). Disposition B's documentation improvement is
worth keeping in mind as a follow-on editorial item but should not
block this change. Disposition C is hard to justify when the grammar's
own author is requesting the reorder and a clean PR is on the table.
*Note:* before merging, the workgroup should confirm via the (still to
be produced) `HL7/cql` repo-analysis briefing that no derived artifacts
under `docs/rrd/` or `input/images/cql/` need to be regenerated.
