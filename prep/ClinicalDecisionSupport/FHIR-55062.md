## Summary

This ticket reports that the `ArtifactAssessment` resource page contains
language implying missing or incomplete content — specifically, the marker
"TODO" — and asks that such indications be removed or replaced with a proper
implementation note before publication. The submission appears to have been
generated automatically from a tooling report run by Gino Canessa prior to the
ballot. On review during the CDS WGM call, Isaac Vetter could not find the
literal string "TODO" on either the balloted version
(`hl7.org/fhir/6.0.0-ballot4/artifactassessment.html`) or the current CI build
(`build.fhir.org/artifactassessment.html`). The reporter (Lloyd McKenzie)
acknowledged the issue may already have been resolved between report
generation and ballot submission, and offered that the ticket may be marked
non-persuasive (or that he would retract it) if a search of the source
confirms no remaining "TODO" markers. There are no cross-referenced Jira
tickets, Zulip discussions, or GitHub items recorded in the cache for this
ticket — it is a self-contained editorial cleanup request.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Search the FHIR Core source tree (the authoring sources backing
`artifactassessment.html`, including the resource definition, narrative
description page, intro/notes pages, and any embedded examples) for the
literal token "TODO" (case-insensitive). For each occurrence found, either
delete the placeholder, replace it with a properly written implementation
note, or convert it into a tracked editorial issue. After the sweep, regenerate
the published page and confirm no "TODO" tokens remain in the rendered output.
Update the ticket with the list of locations changed (or "no occurrences
found") so the resolution is auditable.

#### Justification

The reporter's concern — that ballot content should not contain placeholder
markers implying unfinished work — is consistent with HL7 ballot quality
expectations. Even though both reviewers searched the rendered HTML and found
nothing, the original report was generated from source artifacts (likely
including XML/JSON definition fragments and narrative templates) that may not
all surface in the rendered page text. A source-level sweep is cheap, gives a
definitive answer, and either resolves the ticket as "Persuasive with
Modification" (if anything is found) or provides the evidence base needed to
close it out.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the ticket as already addressed and resolve it as **Persuasive with
Modification – Already Applied**. Document in the resolution that a manual
review of the rendered ArtifactAssessment page (both the 6.0.0-ballot4
publication and the current CI build) confirms no "TODO" markers are present,
and that any prior occurrences captured in Gino's pre-ballot report were
remediated before ballot publication. Optionally pair this with a lightweight
follow-up: add an automated check (e.g., a build-time grep for "TODO" /
"FIXME" / "XXX" tokens in published HTML) so future ballots cannot regress.

#### Justification

The two CDS reviewers have already verified the rendered output and find no
issue. The reporter has explicitly stated he is comfortable with a
non-persuasive / retraction outcome if the source is clean. Closing the ticket
on the evidence already gathered avoids spending workgroup cycles on a
likely-already-resolved editorial matter. Adding a build-time check
generalizes the value: it prevents the same class of issue from recurring in
any resource page, not just `ArtifactAssessment`, which is more useful than a
one-off source sweep.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive**. The reporter's claim cannot be reproduced on
either the balloted page or the CI build, the reporter himself has
acknowledged the report may be stale, and no specific location of an offending
"TODO" has been provided. Without a reproducible defect, there is nothing
concrete for the workgroup to act on.

#### Justification

Ballot tickets require a verifiable defect; the substantive review by the CDS
workgroup found none, and the reporter has effectively conceded the point.
Holding the ticket open or investing in a source-tree search on the basis of
a stale auto-generated report does not respect workgroup time. A
"Not Persuasive" disposition is defensible, fully consistent with the
reporter's stated position, and leaves the door open for a fresh, specific
ticket if a real occurrence is later identified.

---

### Recommendation

**Recommended disposition:** B

Disposition B best reflects the actual state of the artifact and the
conversation on the ticket. The CDS workgroup has already verified that the
rendered ArtifactAssessment page is clean on both the balloted and CI-build
versions, and the reporter has stated he is comfortable with a non-persuasive
outcome if the source is clean — so resolving the ticket as "already applied"
captures the truth on the ground without requiring the reporter to formally
retract. Pairing the resolution with a build-time check for stray "TODO" /
"FIXME" tokens turns a one-off editorial complaint into a durable quality
guard, which is more valuable than either a one-time source sweep
(Disposition A) or a flat decline (Disposition C). Disposition A is a
reasonable fallback if the workgroup wants explicit source-level evidence
before closing; Disposition C is technically defensible but slightly under-
sells the reviewers' work and forgoes the chance to add the regression
guard.
