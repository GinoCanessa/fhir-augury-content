## Summary

Chris Moesel is requesting that, **if** CQL 2.0 retains the
`DefaultComparisonPrecision` option (and specifically retains the ability
for that default precision to flow down into included libraries),
additional normative guidance and/or warnings be added to the
specification to alert authors to the risks of doing so. Two distinct
concerns are raised: (1) a "flow-down" concern that an inclusion
behavior which propagates a non-standard default precision into
included libraries can silently change the semantics of expressions in
those libraries that the author did not write or review; and (2) a
behavioral concern that even within a single library, switching the
default precision affects ostensibly-precise comparison operators such
as `before` and `after`, producing results that authors will likely not
expect (the reporter's example `@2000-01-01T00:00:00.0 before
@2000-01-01T00:00:30.0` evaluates to `true` under standard precision
but to `false` under a `minutes` default).

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with FHIR-52456 in the augury index, so the request
stands on its own and is conditional on a separate (upstream) decision
about whether `DefaultComparisonPrecision` and its include-library
flow-down behavior survive in CQL 2.0.

## Details

**Description:**

> If CQL 2.0 keeps the option to flow default precision down to included libraries, then consider adding guidance or a warning similar to the following:
>
> > When specifying an inclusion behavior for included libraries, authors should carefully review those libraries to ensure that the new default precision does not alter the intent or logical correctness of expressions within those libraries.
>
> Also consider if it might be worth warning how this option might affect operators like `before` and `after`. For example, consider the following CQL:
>
> `define isBefore: @2000-01-01T00:00:00.0 before @2000-01-01T00:00:30.0`
>
> With standard default precision, this evaluates to `true` (as expected). If default precision is changed to `minutes` (however), this would evaluate to `false` (which might not be expected).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persist as not-yet-applied against CQL 2.0. When the CQL 2.0 ballot
publication of the Clinical Quality Language (FHIR) specification is
prepared, and if the `DefaultComparisonPrecision` option (with
include-library flow-down semantics) is retained, add two new pieces of
non-normative guidance to the section that defines
`DefaultComparisonPrecision`:

1. **Include-library flow-down warning** — verbatim or near-verbatim
   adoption of the reporter's suggested wording: a callout/warning
   advising that, when an inclusion behavior propagates a non-standard
   default precision into included libraries, library authors must
   review the included libraries to ensure the propagated precision
   does not alter the intent or logical correctness of expressions in
   those libraries.
2. **Operator-impact warning** — a worked example showing that
   `before`/`after` (and by extension `same as`, `same or before`,
   `same or after`, and the `meets`/`overlaps` family) are silently
   re-interpreted at the new default precision, using the reporter's
   `@2000-01-01T00:00:00.0 before @2000-01-01T00:00:30.0` example to
   contrast the standard-precision result (`true`) with the
   `minutes`-precision result (`false`).

If `DefaultComparisonPrecision` (or its flow-down semantics) is
**removed** from CQL 2.0, this ticket is moot and should be closed as
"Persuasive — addressed by removal of the underlying option."

#### Justification

The reporter's request is purely additive, non-normative author-facing
guidance — it does not change CQL semantics, the grammar, the type
system, or any executable behavior, so there is no implementation
burden on translators or engines. The two examples chosen are exactly
the kind of subtle, hard-to-debug behavior change that is most likely
to bite implementers and library authors in practice: precision is
applied implicitly across operators that authors typically reason about
as "literal" comparisons. Documenting these gotchas at the point the
option is defined is consistent with how CQL already calls out other
precision-sensitive constructs (e.g., the existing notes on uncertain
comparisons). Because the request is explicitly conditioned on CQL 2.0
retaining the feature, accepting it costs nothing if the feature is
dropped.

---

### Disposition B: Alternative Approach

#### Proposal

Address the underlying concern by **reducing the surface area of
`DefaultComparisonPrecision` rather than (or in addition to)
documenting it**. Specifically, in CQL 2.0:

1. Restrict `DefaultComparisonPrecision` so that it applies **only to
   the library in which it is declared** and is *not* propagated into
   included libraries — eliminating concern (1) entirely. The
   spec change is to the section defining the include behavior /
   library-options inheritance: declare that comparison-precision
   options are non-inherited.
2. For concern (2), require that translators emit a **warning
   diagnostic** when a `DefaultComparisonPrecision` other than the
   standard is in scope for any expression that uses `before`, `after`,
   `same as`, `same or before`, `same or after`, `meets`, or `overlaps`
   without an explicit precision qualifier. (This is a translator
   conformance statement, not a runtime behavior change.)
3. Keep a much shorter version of the prose guidance from
   Disposition A as a single "Implementation note" pointing readers at
   the new diagnostic.

#### Justification

The reporter's suggested guidance is good, but it is mitigation against
a sharp edge that the spec itself created. Authors routinely don't
read warnings in non-normative callouts — particularly in libraries
they include transitively — so a documentation-only fix tends to
under-perform in practice. Cutting off the flow-down at the language
level removes the worst of the two failure modes (silent semantic
change in code the author did not write) without sacrificing the
within-library use case for `DefaultComparisonPrecision`. The
translator-warning requirement is a well-precedented pattern in CQL
(authors already lean on translator diagnostics for ambiguous unit
conversions and implicit casts) and gives implementers a uniform,
machine-checkable surface for the second concern. Trade-off vs.
Disposition A: this is a normative change that requires translator
implementations to update, but the change is small and bounded, and
the resulting safety properties are substantially stronger.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive — author responsibility / sufficiently
documented by existing precision semantics**. The current CQL
specification already documents that comparison operators honor a
default precision when one is in effect, and that authors using
non-default precision are responsible for understanding its effect on
expressions in scope. No additional warnings are required; the
reporter's examples are direct consequences of behavior that is
already specified.

#### Justification

This disposition is defensible only if the workgroup takes the view
that adding more warnings to the specification is "noise" relative to
the existing precision documentation, and that the responsibility for
reviewing included libraries already lies with the author who chooses
to enable a non-default precision. It also avoids any precedent of
adding worked-example callouts every time a configuration option has
non-obvious downstream effects — a slope the workgroup may not want to
start down. The downside is that the reporter is a long-time CQL
implementer flagging behavior he believes is genuinely surprising,
which is itself evidence that the existing documentation is not
sufficient.

---

### Recommendation

**Recommended disposition:** A

The request is additive, low-risk, and conditional — it costs nothing
if `DefaultComparisonPrecision` is removed in CQL 2.0, and it costs
only an editorial pass plus one worked example if it is retained. The
reporter has already drafted the substantive wording for the
flow-down warning and provided a sharp, easy-to-reason-about example
for the operator-impact warning, so the editorial lift is minimal.
Disposition B is more protective and worth raising in workgroup
discussion, but it is a normative change that touches translator
conformance and library-include semantics; pursuing it would be better
tracked as a separate ticket so this one can be resolved
straightforwardly. Disposition C is hard to defend given that the
reporter — an experienced CQL implementer — explicitly identified
these as surprises in practice. Accept A now; if the workgroup wants
the stronger language-level mitigations from B, spin those out as a
follow-up against the same CQL 2.0 cycle.
