## Summary

Bryn Rhodes is asking the Using CQL With FHIR IG to forbid published CQL
`Library` resources from carrying a base canonical URL that differs from the
IG that publishes them. The HL7 IG publisher today permits any conformance
resource — `Library` included — to be published under an alternate base
canonical via the `special-url` parameter. For most conformance resources
this is harmless, but for CQL libraries it directly breaks CQL namespace
resolution: the CQL engine resolves library identifiers through the IG's
canonical, so a mismatched base canonical leaves the library unresolvable
within the namespace the IG declares. Bryn observes that, in practice, every
known published package already aligns the base canonical of its CQL
libraries with the IG, and is asking the workgroup to lock that practice in
as a normative publication requirement going forward. No related Jira
tickets, Zulip threads, Confluence pages, or GitHub items were surfaced by
cross-reference, search, or the local repo-analysis cache.

## Details

**Description:**

The publisher currently allows conformance resources to have a different base
canonical than the IG in which they are published (using the `special-url`
parameter). However, using this with CQL libraries introduces a significant
challenge into resolving CQL namespaces.

It is currently the case that all CQL content libraries in known published
packages do match with the base canonical of the IG, and we should ensure
this continues to be the case by requiring this for all published CQL
libraries.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative publication requirement to the *Using CQL With FHIR* IG
stating that any `Library` resource of type `logic-library` (CQL) published
within an IG MUST have a `Library.url` whose base canonical matches the IG's
declared base canonical (i.e., the same prefix as `ImplementationGuide.url`
up to the resource segment). Equivalently, CQL libraries MUST NOT be
published using the IG publisher's `special-url` mechanism. Express this in
the IG's "Conformance Requirements for Libraries" / "Publishing CQL
Libraries" section as a SHALL-level statement, and add a corresponding
FHIRPath invariant on the published Library profile (or an
`obligation`/conformance rule scoped to CQL libraries) so that the IG
publisher and validators can enforce it. Where appropriate, file a
companion enhancement against the IG publisher to surface a hard error (not
just a warning) when a CQL `Library` declares a `url` that does not share
the IG base canonical.

#### Justification

This is exactly what the reporter requests and matches existing practice
across all known published packages, so the normative cost to current
implementers is effectively zero. The CQL specification resolves library
identifiers through the namespace tied to the IG's canonical; allowing a
mismatched base canonical produces libraries that cannot be referenced
correctly by their declared identifier under the IG's namespace, which is
a silent correctness bug that only surfaces at evaluation time. Codifying
the rule in the *Using CQL With FHIR* IG (rather than leaving it as an
implicit convention) gives the publisher and validators a place to anchor
enforcement, and prevents future CQL-bearing IGs from accidentally adopting
`special-url` as a convenience and breaking namespace resolution downstream.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than forbidding `special-url` for CQL libraries outright in the
*Using CQL With FHIR* IG, generalise the change in two pieces: (1) update
the IG publisher to detect `Library` resources that carry CQL content
(`Library.type = logic-library` with a CQL `Library.content`) and emit a
publisher error — independent of any single IG — when the resolved
`Library.url` base does not match the IG's canonical, with `special-url`
explicitly excluded for these resources; and (2) document the rule once,
in the *Using CQL With FHIR* IG, as a SHOULD with a clearly stated rationale
and a single permitted exception: CQL libraries that declare an explicit
`identifier`/version pair that is reachable through a registered package
namespace alias (so the namespace can still be resolved without reading
the canonical's base). This places the enforcement in the tool that the
problem actually originates in (the publisher), keeps the spec text
descriptive rather than prescriptive, and leaves a documented escape hatch
for the rare legitimate case (e.g., a CQL library that is intentionally
shared across IGs and lives at its own canonical root).

#### Justification

The root cause of the failure mode is publisher behaviour, not author
behaviour, so fixing it in the publisher catches every offender — including
ones that never read the *Using CQL With FHIR* IG. A SHALL in the IG plus a
publisher check is belt-and-braces; a publisher error plus a SHOULD avoids
locking the spec into a position that may need to be relaxed later (for
example, if the cross-IG shared-library pattern that the CQL community has
discussed periodically gains traction). It also separates the "what's
allowed" question (spec) from the "what's enforced" question (tool), which
is the layering the broader FHIR conformance ecosystem prefers.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the existing situation — every
known published package already aligns CQL `Library` canonicals with the
IG's canonical — demonstrates that no normative spec change is required.
Address the underlying publisher concern by adding non-normative guidance
to the *Using CQL With FHIR* IG explaining why authors should not use
`special-url` for CQL libraries, and rely on package authors and reviewers
to continue the established practice without a SHALL-level constraint.

#### Justification

The reporter explicitly states that all known published packages already
do the right thing. A normative SHALL imposes a backwards-compatibility
ratchet that the community has not yet bumped into, and any future
legitimate exception (for example, an IG that intentionally republishes a
CQL library defined elsewhere) would require a follow-up errata or version
bump to relax. Where a problem has not actually manifested in the wild,
documentation plus a publisher warning is often a less costly remedy than
a normative constraint.

---

### Recommendation

**Recommended disposition:** A

The reporter is the CQL-on-FHIR specification lead, the failure mode is
silent (mismatched base canonical → CQL namespace cannot resolve the
library by its declared identifier), and the cost to implementers is
effectively nil because all known published packages already comply. That
combination — high correctness payoff, zero observed migration cost,
authoritative reporter — is the ideal profile for promoting an existing
convention to a SHALL. Disposition B's publisher-side enforcement is
desirable as a follow-up, but it should be layered *on top of* a normative
rule in the IG, not used as an excuse to leave the spec silent; the
workgroup can capture B's publisher work as a companion ticket against
HL7/cql-ig (and the IG publisher) once A is accepted. Disposition C
underestimates how easily the current good behaviour could erode the next
time an author reaches for `special-url` to work around an unrelated
publishing wrinkle, and accepts a silent-failure surface for no real gain.
