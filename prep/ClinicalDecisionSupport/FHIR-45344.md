## Summary

Bryn Rhodes (CDS WG) requests that the CRMI Publishable profile family
standardise a common set of `useContext` slices to make publishable artifacts
(measures, libraries, value sets, plan definitions, etc.) more easily
discoverable and filterable. The original ask listed three slices —
**focus**, **setting**, and **age range** — as standard `useContext` slices
on every Publishable profile.

The ticket has been triaged with resolution **Persuasive**, and the agreed
resolution expands the original list to seven `useContext` slices on every
Publishable profile, each `0..*` and bound *extensible* to a recommended
value set:

- `gender` → `http://hl7.org/fhir/ValueSet/administrative-gender`
- `age` → `http://terminology.hl7.org/ValueSet/v3-AgeGroupObservationValue` (`Range|code`)
- `focus` → `https://hl7.org/fhir/R4/valueset-condition-code.html`
- `user` → `http://hl7.org/fhir/ValueSet/provider-taxonomy`
- `workflow` → `http://terminology.hl7.org/ValueSet/v3-ActEncounterCode`
- `task` → `http://terminology.hl7.org/ValueSet/v3-ActTaskCode`
- `venue` → `http://terminology.hl7.org/ValueSet/v3-ServiceDeliveryLocationRoleType`

No explicit cross-references (Jira, Zulip, Confluence, GitHub) were returned
for this ticket by the augury index. Several keyword-related tickets (e.g.,
[FHIR-48533](https://jira.hl7.org/browse/FHIR-48533),
[FHIR-43320](https://jira.hl7.org/browse/FHIR-43320),
[FHIR-36776](https://jira.hl7.org/browse/FHIR-36776),
[FHIR-48493](https://jira.hl7.org/browse/FHIR-48493)) confirm there is an
active pattern across the CRMI / CQF-Measures / eCR ecosystems of slicing
`useContext` for venue, focus, and quality-program purposes — this CR
attempts to consolidate that pattern at the CRMI Publishable layer.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Implement the resolution as written in the CRMI IG (`HL7/crmi-ig`):
add seven new `useContext` slices to **every** profile in the CRMI
Publishable profile family — at minimum `PublishableMeasure`,
`PublishableLibrary`, `PublishablePlanDefinition`, `PublishableActivityDefinition`,
`PublishableValueSet`, `PublishableCodeSystem`, `PublishableConceptMap`,
`PublishableQuestionnaire`, `PublishableStructureDefinition`,
`PublishableNamingSystem`. Each slice is `0..*` and bound *extensible*
to its recommended value set:

| Slice (`useContext.code`) | Value (`Range` for `age`, `CodeableConcept` otherwise) | Binding strength | Value set |
|---|---|---|---|
| `gender` | CodeableConcept | extensible | `http://hl7.org/fhir/ValueSet/administrative-gender` |
| `age` | Range \| code | extensible | `http://terminology.hl7.org/ValueSet/v3-AgeGroupObservationValue` |
| `focus` | CodeableConcept | extensible | `http://hl7.org/fhir/ValueSet/condition-code` |
| `user` | CodeableConcept | extensible | `http://hl7.org/fhir/ValueSet/provider-taxonomy` |
| `workflow` | CodeableConcept | extensible | `http://terminology.hl7.org/ValueSet/v3-ActEncounterCode` |
| `task` | CodeableConcept | extensible | `http://terminology.hl7.org/ValueSet/v3-ActTaskCode` |
| `venue` | CodeableConcept | extensible | `http://terminology.hl7.org/ValueSet/v3-ServiceDeliveryLocationRoleType` |

Discriminator on each slice is `useContext.code` against the
`usage-context-type` code system. Update CRMI narrative pages on
publishable artifacts and discoverability/search to describe the standard
slices. Note that `code` is already a defined `usage-context-type` code
but `gender` is *not* in the existing `usage-context-type` code system
and would have to be added (it currently lives only as the example
`gender` in the `usage-context` value set), so a TSMG-coordinated update
to `usage-context-type` (or use of an alternative code) is required.

#### Justification

The set of slices directly maps to the dimensions implementers most often
filter publishable artifacts on (population age/gender, clinical focus,
intended user, workflow point, task, and care venue). Standardising them
at the CRMI Publishable layer means downstream IGs (cqf-measures, eRSD,
US Public Health, etc.) inherit a consistent search facet contract
instead of each re-inventing it — exactly the pattern already seen in
[FHIR-36776](https://jira.hl7.org/browse/FHIR-36776) (focus on US-PH
ValueSet) and [FHIR-48533](https://jira.hl7.org/browse/FHIR-48533)
(venue on PublishableMeasure). Extensible binding preserves the ability
of IGs to bind their own narrower code systems (as cqf-measures did
with `intended-venue`).

### Disposition B: Alternative Approach

#### Proposal

Adopt the slices but with two modifications:

1. **Define the slices on a single abstract `Publishable` parent profile**
   (or a CRMI `MetadataResource`-shaped mixin) and have the concrete
   `Publishable*` profiles derive from it, rather than duplicating seven
   slice definitions across ten-plus profiles. This matches the existing
   CRMI strategy of layering `Knowledge`/`Publishable`/`Executable`
   shapes.
2. **Defer the `gender`, `age`, `focus`, `user`, `workflow`, `task`,
   and `venue` value-set recommendations to the base `UsageContext`
   datatype** via the *preferred*-purpose additional bindings already
   agreed in [FHIR-48493](https://jira.hl7.org/browse/FHIR-48493), and
   in CRMI just declare the slice + discriminator (no binding restated
   at the profile level). Where CRMI needs to *strengthen* a binding
   above what Core declares (e.g., to extensible), do so explicitly
   only for that slice.

Cardinality stays `0..*`; mustSupport is **not** added (none of the
slices is required for any concrete use case yet).

#### Justification

This avoids two foreseeable problems with the literal resolution:

- **Maintenance drift.** Restating seven extensible bindings on every
  Publishable profile means that when the recommended value set
  changes (or when Core picks up the FHIR-48493 additional bindings),
  every CRMI profile must be re-published in lock-step. Sliced
  inheritance from a parent profile or via base-datatype bindings
  removes that duplication risk.
- **Binding conflict with Core.** Once Core/`UsageContext` declares
  preferred additional bindings (per FHIR-48493), restating them at
  extensible at the profile level is harmless but redundant; restating
  them at a *different* value set (e.g., the literal resolution uses
  R4's `valueset-condition-code` for `focus`, not the version-current
  one) risks inconsistency. Centralising the recommendation reduces
  that risk.

It also addresses the [FHIR-43320](https://jira.hl7.org/browse/FHIR-43320)
precedent, where a useContext slice (`qualityProgram`) was deemed
inappropriate to keep on Publishable Measure/Library and pushed to a
specific profile — the parent-profile design lets a derived profile opt
out of a slice if needed.

---

### Disposition C: Decline

#### Proposal

Do not add the seven `useContext` slices to the CRMI Publishable
profiles. Instead, document the recommended `useContext.code` values
(focus, setting, age, gender, user, workflow, task, venue) as
**non-normative guidance** in the CRMI narrative pages on publishable
artifacts and discoverability, and let individual implementation guides
slice `useContext` themselves where they have a concrete need (as
cqf-measures, eRSD, and US Public Health already do).

#### Justification

Defensible primarily because:

- **`UsageContext` is already extensible by design.** The `useContext`
  element accepts arbitrary `usage-context-type` codes plus value-bound
  `value[x]`. Adding seven slices does not increase what implementers
  *can* express — it only adds eleven-plus profile-level constraints
  to maintain.
- **WG precedent of removal.** [FHIR-43320](https://jira.hl7.org/browse/FHIR-43320)
  shows the WG actively *removed* a useContext slice from Publishable
  Measure/Library after deciding it was at the wrong level. A blanket
  "add seven everywhere" risks the same outcome at scale.
- **Pending Core resolution.** [FHIR-48493](https://jira.hl7.org/browse/FHIR-48493)
  will add additional bindings to `UsageContext` itself; doing so first
  in CRMI risks divergence from whatever Core ultimately ships.

---

### Recommendation

**Recommended disposition:** B

The intent of FHIR-45344 — give implementers a consistent, discoverable
metadata vocabulary for publishable artifacts — is well-motivated and
already validated by the parallel work in
[FHIR-48533](https://jira.hl7.org/browse/FHIR-48533) and
[FHIR-36776](https://jira.hl7.org/browse/FHIR-36776), so outright
declining (Disposition C) leaves the same searchability gap that
prompted the CR. However, the literal resolution (Disposition A)
duplicates seven extensible bindings across the entire CRMI Publishable
family and overlaps materially with
[FHIR-48493](https://jira.hl7.org/browse/FHIR-48493)'s
`UsageContext`-level work, creating a real maintenance burden and a
non-trivial risk of binding drift between CRMI and Core. Disposition B
keeps the Persuasive resolution's substantive content (the seven slices
and their recommended value sets) but delivers it via a single
parent-shape definition and defers binding recommendations to Core
where they are already being added — giving downstream IGs a clean
search-facet contract without locking CRMI into restating seven Core
bindings at extensible strength on every concrete profile. It also
preserves the WG's ability (per the FHIR-43320 precedent) to opt
specific profiles out of slices that turn out to be misplaced.
