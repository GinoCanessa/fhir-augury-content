## Summary

Bryn Rhodes asks the Clinical Decision Support work group to clean up the
"Using CQL With FHIR" IG so that its conformance constraints clearly
distinguish requirements that are *FHIR-specific* (and therefore properly
normative in this IG) from those that are general *CQL best-practices*
(which should either be promoted upstream into base CQL or relegated to
non-normative guidance). The submitter cites three concrete inconsistencies
in the current authoring section:

1. The IG SHALL include a `library` statement and SHOULD include a version —
   the submitter questions why this is FHIR-specific at all.
2. The `using` clause SHALL have a version, but only SHOULD use a FHIR data
   model — yet the entire IG is scoped to "CQL with FHIR", so the FHIR data
   model arguably ought to be SHALL.
3. Element/attribute naming SHOULD be camelCase — the submitter argues the
   *FHIR-specific* requirement is simply that names match the corresponding
   FHIR element name (case-sensitive), and that camelCase is a CQL style
   note, not a FHIR conformance requirement.

The cross-reference, Zulip, and GitHub indexes returned no related items
for this ticket key, so this Summary reflects only the ticket text itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the authoring conformance section of the *Using CQL With FHIR* IG to
audit each conformance statement and re-class it as either:

- **FHIR-specific (kept here, normative).** Tighten language where the IG's
  scope makes a SHOULD redundant. Specifically:
  - Change the `using` clause rule from "SHOULD use a FHIR data model" to
    **"SHALL use a FHIR data model"**, since the IG is scoped to CQL with
    FHIR by definition.
  - Promote the element/attribute naming rule from a generic "SHOULD be
    camelCase" to **"SHALL match the corresponding FHIR element name,
    including case"**, with a non-normative note that FHIR element names
    are themselves camelCase.
- **Generic CQL best practice (removed or demoted).** Move statements that
  are not FHIR-specific (e.g., the bare `library` statement / version
  guidance) to a non-normative "Authoring guidance" callout, and file a
  companion ticket against base CQL to consider promotion.

Add a short sub-section header at the top of the authoring conformance
list that explicitly states the rubric used for inclusion ("a constraint
appears here only if violating it would break interop between a CQL
artifact and a FHIR data model / FHIR-based engine").

#### Justification

This matches the submitter's intent literally and produces an IG whose
normative statements are defensible on FHIR-interop grounds rather than on
style grounds. Tightening the `using`-clause rule to SHALL is consistent
with the IG's own scoping (it would be incoherent for a "Using CQL With
FHIR" IG to allow non-FHIR data models). Demoting style-only rules
clarifies what implementers are actually being measured against in
conformance testing, and reduces the risk that a tool flags an artifact as
non-conformant for a stylistic preference.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-classifying constraints in place, restructure the
conformance section into **two clearly labeled subsections** without
changing any individual SHALL/SHOULD verb in this cycle:

1. **FHIR Conformance Requirements** — constraints whose violation breaks
   CQL ↔ FHIR interop (typing against a FHIR data model, element-name
   matching, FHIRPath/Expression integration points).
2. **Recommended CQL Authoring Practices (informative)** — items like
   camelCase preference, library naming style, version-pinning style.

Each item retains its current verb, but moves under the appropriate
heading. Open a follow-up ticket (or a small set) to revisit specific
SHALL/SHOULD strengths once the rubric is in place, so that verb changes
get individual review instead of being bundled.

#### Justification

This addresses the underlying confusion (which constraints are normative
*because of FHIR* vs. which are general style) without making any
backwards-incompatible verb changes in a single sweep. Strengthening
"SHOULD use a FHIR data model" to SHALL is a behavior change for any
existing artifact that relies on the SHOULD; surfacing the rubric first
lets the work group decide each verb change on its merits with implementer
input. It also creates a clean home for items that should ultimately be
promoted to base CQL, without losing them from the current IG text.

---

### Disposition C: Decline

#### Proposal

Decline as **editorial / no normative change required**. The current
constraints reflect intentional choices: SHOULD-level guidance for
camelCase and for the FHIR data model in the `using` clause was selected to
accommodate edge cases (e.g., mixed-model artifacts, legacy library naming)
without forcing non-conformance. Close the ticket with a note pointing the
submitter at the existing rationale (or, if that rationale is not written
down, ask them to file a more specific ticket per constraint rather than a
blanket clarification request).

#### Justification

Defensible only if the work group can produce concrete examples of
artifacts that legitimately need to (a) use a non-FHIR data model under
this IG, or (b) deviate from FHIR's case-sensitive element names. Without
such examples, the submitter's critique stands, so this disposition is the
weakest of the three on the merits.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right starting point because the submitter's core
complaint is *taxonomic* (the reader cannot tell which constraints are
normative because of FHIR), not primarily about specific verb strengths.
Restructuring the conformance section into "FHIR Conformance Requirements"
vs. "Recommended CQL Authoring Practices" addresses that complaint
immediately, with no risk of inadvertently breaking existing artifacts in
the field. It also creates the framework that makes Disposition A's verb
changes (SHALL FHIR data model, SHALL match FHIR element name with case)
straightforward to revisit on individual follow-up tickets — where each
strengthening can be reviewed by implementers who would be affected, and
where genuinely CQL-general items can be filed for promotion to base CQL.
Disposition C should be off the table unless the work group surfaces
concrete counter-examples that justify the current SHOULDs.
