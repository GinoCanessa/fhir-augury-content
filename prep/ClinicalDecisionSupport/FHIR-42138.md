## Summary

The reporter requests that the CDS Hooks specification add explicit guidance,
in the `order-select` hook section, warning implementers that this hook can
contribute to physician alert fatigue and ordering burden if a CDS Service
returns cards every time the user adds an element to an in-progress order.
The request is documentation/guidance only — it does not propose any change
to the hook's payload, context, or invocation contract. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced for FHIR-42138,
so the scope is bounded to the wording in the `order-select` hook page of
the CDS Hooks Library.

## Details

**Description:**

> Suggest adding verbiage recognizing that the order select hook could
> increase physician alert fatigue and burden if there is a ping each time
> the physician adds an element to the order. Perhaps we could add something
> about implementers being mindful of this when building out.

## Proposed Dispositions

The dispositions below are wording-only changes to the CDS Hooks Library
specification, scoped to the `order-select` hook page. Because no GitHub
cross-references were surfaced, no specific repo path can be cited from a
persisted briefing; the change target is whichever repository hosts the
CDS Hooks Library (FHIR) authoring source for the `order-select` hook
page (commonly the `cds-hooks/docs`-style spec source identified by the
CDS WG).

### Disposition A: Accept As Requested

#### Proposal

Add a short non-normative "Implementer Guidance" callout to the
`order-select` hook section of the CDS Hooks Library, with wording along
these lines:

> *Implementer guidance:* Because `order-select` fires each time the
> user selects (adds, removes, or modifies) an order in the in-progress
> order list, naive use of this hook can generate frequent CDS Service
> invocations and a corresponding stream of cards in the UI. CDS Service
> authors and CDS Clients SHOULD be mindful of clinician alert fatigue
> and ordering burden — for example by debouncing or coalescing
> responses, suppressing duplicate cards across successive selections,
> deferring non-urgent advice until `order-sign`, and only returning
> cards whose information clearly justifies an interruption at this
> stage of the workflow.

Place the callout in the descriptive prose of the `order-select` hook
page (not in the workflow / context / example payload sections), keep it
non-normative ("SHOULD"/"may"), and cross-link to the
`order-sign` hook so readers see the contrast between
per-selection feedback and per-signing feedback.

#### Justification

The ticket explicitly asks for wording, not for a contract change, and
the concern is well-grounded: `order-select` is defined to fire on every
change to the in-progress order list, which is exactly the pattern that
produces alert fatigue when paired with chatty CDS Services. Adding a
non-normative guidance paragraph is the lowest-risk way to address the
concern: it is backward compatible, requires no version bump, does not
change conformance, and matches the precedent already set elsewhere in
CDS Hooks where implementer guidance is interleaved with the hook
definition. It also gives reviewers a concrete artifact to point future
implementers at.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) inline guidance on the `order-select`
page, lift the concern into a single, reusable section — e.g.
"Designing CDS Services to Minimize Alert Fatigue" — at the CDS Hooks
specification level (or in the existing "Best Practices" / "Security &
Safety" area, depending on where the CDS WG prefers cross-cutting
guidance). The section would:

- Name the hooks most prone to high-frequency invocation
  (`order-select`, `order-dispatch`, `appointment-book`, etc.).
- Give a checklist for CDS Service authors (debounce, coalesce,
  suppress duplicates, raise the threshold for what justifies a card,
  prefer `order-sign` for final-state advice).
- Give a parallel checklist for CDS Client implementers (rate
  limiting, card de-duplication, user controls to mute a service).
- From the `order-select` hook page, add a one-line pointer to this
  section rather than duplicating the prose.

#### Justification

The underlying problem — chatty CDS Services causing fatigue — is not
unique to `order-select`. `order-dispatch` and several appointment /
encounter hooks share the same risk profile, and implementers have to
make the same design decisions for each. Centralizing the guidance
avoids inconsistent advice drifting across hook pages over time, gives
the workgroup one place to evolve the recommendations, and is a more
durable answer to the reporter's concern. The trade-off vs. Disposition
A is that readers who land directly on the `order-select` page may miss
the guidance unless we also include the pointer; that pointer is cheap
and addresses the trade-off.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" / "Won't Fix" on the basis that alert fatigue
is a generic CDS / EHR usability concern that applies to virtually every
CDS Hook (and indeed to any clinical decision support system), and is
already well-covered in the broader CDS literature and in the existing
CDS Hooks "card" guidance, which makes clear that cards should be
returned only when there is actionable, decision-relevant information.
No spec text change.

#### Justification

A defensible decline rests on two points: (1) the spec already implies
the right behavior — the card model is opt-in per response, so a CDS
Service author who returns no cards is the contractual answer to "don't
ping the user every time," and (2) usability guidance can balloon if the
spec starts cataloging every workflow risk. Declining keeps the spec
tightly scoped to the hook contract and leaves implementer education to
companion artifacts (implementation guides, conference talks, the CDS
Hooks sandbox). If the WG already has a written policy of keeping
non-normative usability guidance out of the core spec, this is the
consistent answer.

---

### Recommendation

**Recommended disposition:** B

The reporter's concern is real and the cost of writing the guidance is
low, so a flat decline (C) under-serves a "Highest" priority ticket that
has been open since 2023 with no pushback. Between A and B, B is the
better long-term answer: alert fatigue is a cross-hook concern, not an
`order-select`-specific one, and the CDS WG will almost certainly want
to apply the same advice to `order-dispatch` and the appointment hooks.
Centralizing the guidance once and pointing to it from `order-select`
delivers exactly what the reporter asked for at the `order-select` page
(a visible, in-context warning) while avoiding the maintenance debt of
copy-pasting the same paragraph into every chatty hook in future
ballots. If the WG prefers a faster turnaround, Disposition A is a
reasonable interim step that does not preclude promoting the wording to
a shared section later.
