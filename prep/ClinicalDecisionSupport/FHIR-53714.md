## Summary

Isaac Vetter is asking how `GuidanceResponse` represents the *inputs* that
were fed into a CDS process — for example, the proposed (not-yet-prescribed)
opioid medication that an MME/opioid risk algorithm evaluates, or the
"what-if" inputs a clinical calculator operates on. He calls these out as
the `inputParameters` and argues they are essential for understanding the
guidance produced. He further suggests that `GuidanceResponse.reason` feels
conceptually closer to a CDS Hooks **hook** (the trigger) than to the
CDS Hooks **context** (the data passed in), implying that the current
modelling/naming may be confusing for implementers coming from a CDS Hooks
mental model.

The ticket is framed as a clarification / design question rather than a
concrete change proposal: the reporter wants the workgroup to confirm
where these inputs live on the resource (`GuidanceResponse.inputParameters`,
which is a `Reference(Parameters)`), and to consider whether the
documentation, naming, or modelling should be tightened so the analogy to
CDS Hooks is clearer. There are no cross-referenced Jira tickets, Zulip
threads, or GitHub items in the augury index for this issue, so the
disposition discussion will rest on the existing `GuidanceResponse`
definition and CDS Hooks alignment principles.

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a documentation / clarification request and update
`GuidanceResponse` accordingly:

1. Expand the **element definition and comments on `GuidanceResponse.inputParameters`**
   to explicitly state that this is where the inputs to the CDS computation
   are recorded — including hypothetical / "what-if" inputs that are not
   present elsewhere in the patient record (e.g., a proposed but
   not-yet-ordered medication for an opioid MME calculation). Call out
   that the referenced `Parameters` resource is the right place to capture
   those candidate resources by-value.
2. Add an **explanatory paragraph to the `GuidanceResponse` resource notes
   page** that maps the resource onto the CDS Hooks model:
   - `GuidanceResponse.module[x]` ↔ the *service / hook definition*
   - `GuidanceResponse.reason` ↔ the *triggering reason* the module was
     invoked (the "why"), analogous to a CDS Hooks **hook**
   - `GuidanceResponse.inputParameters` (+ referenced `Parameters` /
     `dataRequirement` snapshots) ↔ the CDS Hooks **context** and
     **prefetch** payload (the "with what data")
   - `GuidanceResponse.result` / `outputParameters` / `evaluationMessage`
     ↔ the CDS Hooks response (cards, system actions, suggestions)
3. Add a **worked example** (an opioid MME calculator or similar
   what-if calculator) to the resource's Examples tab showing a
   `GuidanceResponse` with an `inputParameters` reference to a
   contained / referenced `Parameters` resource that carries a proposed
   `MedicationRequest` that does not exist anywhere else.

No structural change (no new elements, no cardinality changes, no
invariants) — purely definitional text and an example.

Authoring location: `HL7/fhir` core spec, `source/guidanceresponse/`
(structure definition narrative + `examples-*.xml/json`). Confirm via a
`repo-analysis HL7/fhir` briefing before opening a PR.

#### Justification

The ticket is fundamentally a "where do the inputs live?" question, and the
answer already exists in the resource (`inputParameters : Reference(Parameters)`
plus `dataRequirement` for the declared needs). What is missing is a clear,
prose-level mapping from the CDS Hooks vocabulary that most CDS implementers
already think in (hook / context / prefetch / cards) onto the resource
elements. The reporter explicitly frames the question that way ("my naive
expectation here is that reason is more similar to CDS Hooks' hook than
its context"). A definitional + example-level fix resolves the confusion
without disturbing any conformant implementation, and it aligns with the
CDS WG's general direction of keeping CDS Hooks and the clinical reasoning
resources mutually intelligible.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying need but address it with a **light structural
clarification** rather than docs alone:

1. Keep `GuidanceResponse.inputParameters` as-is, but add a **standard
   extension** (e.g., `guidanceresponse-inputContext`) on `GuidanceResponse`
   that explicitly carries the CDS Hooks *context* payload (or a pointer
   to the contextual resources) when the response was produced by a
   hook-style invocation. This separates the "context that fired the hook"
   from "all parameters/data the engine consumed", matching the CDS Hooks
   distinction the reporter is drawing.
2. Add an FHIRPath-level **invariant or guidance note** on
   `GuidanceResponse.reason` clarifying that `reason` describes *why* the
   guidance was sought (closer to a CDS Hooks hook semantic), not the
   data that was used to compute it.
3. Same documentation/example work as Disposition A, but anchored on the
   new extension so the CDS Hooks ↔ FHIR mapping is mechanically
   round-trippable, not just narrative.

Authoring location: `HL7/fhir` core spec for the resource notes
(`source/guidanceresponse/`) plus the standard extensions registry
(`source/guidanceresponse/` extension definitions). Confirm exact
authoring paths via a `repo-analysis HL7/fhir` briefing before
implementation.

#### Justification

`inputParameters` today is a single `Reference(Parameters)` and is
expected to capture *everything* the engine saw. Implementers coming from
CDS Hooks naturally separate "context" (small, hook-defined, often
hypothetical resources like a draft `MedicationRequest`) from "prefetch /
additional data" (real record content). A dedicated extension preserves
that distinction without reshaping the base resource and without breaking
existing implementations. The trade-off vs. Disposition A is added
spec surface area and a new extension to maintain; the upside is that
tools bridging CDS Hooks ↔ `GuidanceResponse` can mechanically place the
hook context where consumers will reliably find it, rather than relying on
each producer to encode their own convention inside `Parameters`.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive — already addressed**. Respond on the ticket
that `GuidanceResponse` already supports the use case:

- Inputs (including hypothetical / "what-if" resources) are carried via
  `GuidanceResponse.inputParameters : Reference(Parameters)`; the
  referenced `Parameters` resource can hold any FHIR resource by value,
  including a proposed `MedicationRequest` that has no independent
  existence in the record.
- The declared inputs the module *required* are described by
  `GuidanceResponse.dataRequirement`.
- `GuidanceResponse.reason` is intentionally the *clinical reason for the
  guidance request*, not the data feed; the CDS Hooks "hook" is more
  closely represented by `GuidanceResponse.module[x]` (the invoked
  artifact) plus the triggering context the caller passes in via
  `inputParameters`.

No spec change; the ticket is resolved by a comment explaining the
existing model.

#### Justification

The mechanism the reporter is asking about already exists, and the
analogy to CDS Hooks is workable today: `module[x]` ≈ the service being
invoked, `inputParameters` ≈ context+prefetch, `result/outputParameters`
≈ cards/system actions, `reason` ≈ the clinical motivation. Declining is
defensible because adopting either A or B implies spec churn for what is,
strictly speaking, a documentation/communication gap rather than a
modelling defect. If no other implementers raise the same confusion, the
cost of the change may not be justified.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers the question the reporter is asking, costs
the workgroup very little (resource notes + one example), and removes a
real source of confusion for the substantial overlap between CDS Hooks
implementers and `GuidanceResponse` implementers. Disposition C is
technically correct but leaves the next implementer to rediscover the
same question; given that the reporter is an experienced CDS implementer
and still found the mapping unclear, that is good evidence the docs need
to be sharper. Disposition B (a new extension) is plausible but heavier
than the evidence currently warrants — it should be reconsidered only if,
after the documentation/example clarification, implementers continue to
report that they cannot reliably distinguish "hook context" from "all
inputs" inside the `Parameters` payload. Starting with A keeps the door
open to B without committing to new spec surface area now.
