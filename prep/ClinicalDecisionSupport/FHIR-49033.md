## Summary

Reporter LinZhang flags a typo in the Clinical Practice Guidelines (CPG-on-FHIR)
IG, in the narrative section labelled **8.2.2 Example Case**. The example walks
through a CHF (congestive heart failure) patient whose CaseFeatures and Goals
are listed inline. For the **O2 Sat** CaseFeature, the rendered text shows a
`Goal:` value that does not correspond to oxygen saturation — the reporter
asks that the **Goal of CaseFeature O2 Sat** be corrected to **`>94%`**, the
clinically appropriate target value. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were surfaced for this issue, so the request is a
self-contained editorial / technical correction against the IG narrative.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **Clinical Guidelines (FHIR)** IG (`HL7/cqf-recommendations`), edit the
authored narrative `input/pagecontent/examples-chf.md` (the page that renders
as "Example Case" / 8.2.2) so that the `O2 Sat` CaseFeature block reads:

```
O2 Sat:
CaseFeatureDefinition:
Observation: 87% @2019-02-03T05:00 (Critical Low)
Observation: 93% @2019-02-03T09:41
Goal: > 94%
Proposed: @2019-02-03T13:00
```

Specifically, change the existing `Goal: < 8.0cmH2O` line — which is clearly
the JVP goal accidentally pasted under O2 Sat — to `Goal: > 94%`. No FSH,
CQL, ActivityDefinition, PlanDefinition, or terminology artifact is touched;
this is a pure markdown narrative fix and does not require regenerating any
example resources. The change ships with the next IG-publisher build of
`hl7.fhir.uv.cpg`.

#### Justification

The reporter is correct: an O2 saturation Goal expressed in cmH2O is
clinically nonsensical (cmH2O is a pressure unit, used in this same example
for the Jugular Venous Pressure CaseFeature directly above). `> 94%` is the
standard target for adult patients on room air and is consistent with the
two preceding Observations (87% Critical Low, 93%) in the same block, which
are clearly below target. Fixing the narrative to `> 94%` aligns the example
internally and removes a confusing clinical statement from a published HL7
IG. The change is editorial-only, has no normative impact, and carries no
backward-compatibility risk.

---

### Disposition B: Alternative Approach

#### Proposal

Apply the same correction as Disposition A, but additionally:

1. Express the goal using a structured comparator + UCUM unit, e.g.
   `Goal: >= 94 % (SpO2)` or `Goal: SpO2 >= 94 %` so the unit is explicit and
   matches how a downstream `CaseFeatureDefinition` Goal element would be
   computably encoded (Quantity with comparator `>=`, code `%`, system
   `http://unitsofmeasure.org`).
2. Audit the surrounding CaseFeature Goal lines on the same page for the
   same class of copy-paste errors (e.g., verify the JVP, Potassium,
   Creatinine, and Weight goals all use units that match their feature) and
   correct any others found in the same PR.

#### Justification

The narrative example is meant to teach implementers how CaseFeatures map
onto computable artifacts. Using `> 94%` is correct but informal; a slightly
more structured rendering (`SpO2 >= 94 %`) better mirrors the
Quantity-with-comparator pattern that a real `Goal` element would use, and
makes the example a better teaching tool. Bundling a one-pass audit of the
adjacent Goal lines is cheap insurance against shipping a second, similar
typo in the same section in the next ballot cycle, and keeps the
"Technical Correction" lifecycle to a single round-trip.

---

### Disposition C: Decline

#### Proposal

Decline the change as written and close as "Not Persuasive" on the grounds
that the section is illustrative narrative only and the existing text, while
imprecise, does not affect any computable artifact, profile, or conformance
statement in the IG.

#### Justification

This is the weakest of the three positions and is offered only for
completeness. The IG explicitly markets the methodology case study as a
teaching example, and a Goal expressed in pressure units (cmH2O) for an
oxygen-saturation feature is internally inconsistent and clinically
misleading regardless of whether it drives a profile or a CQL library.
Declining would leave a published HL7 specification with an obvious clinical
error that the reporter has already pinpointed, with essentially no upside.

---

### Recommendation

**Recommended disposition:** B

Disposition A fully satisfies the reporter's literal request and is trivially
safe. Disposition B is recommended because it costs almost nothing extra: the
fix is in a single markdown file, the contributor is already in that file,
and using a comparator + UCUM-unit rendering (`SpO2 >= 94 %`) makes the
example a better illustration of how a `Goal` would be encoded as a Quantity
in a real CaseFeatureDefinition. Bundling a quick audit of the adjacent Goal
lines (JVP, Potassium, Creatinine, Weight) on the same page also protects
against a second copy-paste typo surviving into the next publication. If the
work group prefers to keep the narrative exactly mirroring the reporter's
wording, falling back to Disposition A is acceptable; Disposition C should
not be adopted.
