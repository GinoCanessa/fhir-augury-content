## Summary

The ticket requests that the Clinical Quality Language (CQL) grammar
add direct support for the `not in` membership-test construct, so that
authors can write `1 not in { 1, 2, 3, 4, 5 }` instead of having to
wrap the expression in `not (1 in { 1, 2, 3, 4, 5 })`. The reporter
(Bryn Rhodes, the CQL spec lead) frames this as common syntactic
sugar that the grammar currently does not accept, and notes that a
companion change has already been opened in the reference translator
(`cqframework/clinical_quality_language` PR #1517) so a parser/AST
implementation can land alongside the spec change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Extend the CQL grammar (the ANTLR-defined `cql.g4` in the CQL
specification repo `HL7/cql`) to add `not in` as a membership-test
operator, alongside the existing `in` operator. Concretely:

- Add a `'not' 'in'` alternative to the `membershipExpression`
  production so that `<expression> not in <expression>` parses
  directly.
- Define `A not in B` as exact, semantically-identical sugar for
  `not (A in B)` (same return type, same null-propagation behaviour,
  same handling of the optional `during`/precision modifier already
  supported on `in`).
- Mirror the change in the prose grammar tables and the operator
  reference chapter of the CQL spec, including a worked example
  matching the one in the ticket.
- Coordinate the merge with the reference-implementation work already
  in flight at `cqframework/clinical_quality_language#1517` so the
  spec ships with a known-conformant parser.

#### Justification

The reporter is the CQL spec lead and has already committed
implementation work in the reference translator, so the request is
aligned with existing toolchain direction. `not in` is a near-universal
pattern across SQL-family and general-purpose languages (SQL, Python,
Kotlin, etc.) and is a pure syntactic-sugar addition: it introduces no
new semantics and no new types. As sugar, it is fully back-compatible —
existing CQL libraries continue to parse unchanged, and any new library
using `not in` can be losslessly rewritten to today's `not (... in ...)`
form for older parsers.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same surface syntax as Disposition A, but generalise the
grammar change to also cover the related membership-style operators
that suffer from the same readability problem — specifically `not
contains` (so `B not contains A` is sugar for `not (B contains A)`)
and the precision-qualified variants `not in <precision>` /
`not during`. Specify all of them as a single coordinated grammar
delta with a shared "negated membership" section in the spec, and
require translator implementations to lower them to the existing
`Not(In(...))` / `Not(Contains(...))` / `Not(IncludedIn(...))`
ELM nodes (no new ELM operators).

#### Justification

CQL today already has the asymmetric pair `in`/`contains` and the
precision-qualified `in <precision>` / `during` forms. Adding only
`not in` leaves authors with the same readability problem the moment
they reach for `contains` or a precision modifier, and risks a future
ticket asking for each negated form one at a time. Bundling them
keeps the grammar change atomic, gives implementers a single
translator pass to write, and produces a more consistent author
experience. The cost over Disposition A is one additional grammar
alternative per operator and a slightly larger spec edit; the ELM
surface is unchanged because everything still lowers to the existing
`Not(...)` wrapping.

---

### Disposition C: Decline

#### Proposal

Decline the grammar change and document `not (X in Y)` as the
canonical way to express negated membership. Optionally add a single
non-normative note in the operator reference chapter calling out the
pattern so authors can find it via search.

#### Justification

Defensible only on a strict "no syntactic sugar in the base grammar"
posture: every sugar form is one more production in the parser, one
more thing every conforming implementation must accept, and one more
shape that downstream tooling (formatters, linters, IDE refactors,
the reference translator's pretty-printer, third-party parsers) must
round-trip. The functional capability already exists via `not (... in
...)`, so no author is blocked. The argument against this disposition
is that the cost is genuinely tiny — one ANTLR alternative that
lowers to an existing ELM node — and the readability win is real,
which is why this is the weakest of the three options.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right call. The reporter is the spec lead, the
companion implementation PR (`cqframework/clinical_quality_language`
#1517) already exists, the change is pure back-compatible sugar with
no new semantics or ELM operators, and the readability benefit is
material for a construct that appears constantly in quality-measure
and decision-support logic. Disposition B's broader sweep (`not
contains`, precision-qualified negations) is attractive in principle,
but the workgroup has not yet seen demand for those forms and
expanding scope risks delaying the narrowly-scoped, already-implemented
`not in` change; B should be considered as a follow-up ticket if and
when authors request the symmetric forms. Disposition C is
unattractive: the implementation cost is trivial, the change is
non-breaking, and the author-ergonomics improvement is concrete. The
workgroup should accept A and coordinate the spec merge with the
landing of PR #1517.
