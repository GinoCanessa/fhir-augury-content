## Summary

The ticket asks the Extensions Registry maintainers to publish a new
extension, `artifact-extended-contact-detail`, that carries the
`ExtendedContactDetail` datatype and that can be applied on **any**
Resource. The reporter (Brian Alper, EBMonFHIR) notes there is currently
no registered extension whose `Type` is `ExtendedContactDetail`, leaving
implementers without a standard way to attach contact-with-purpose
information (copyright contact, correspondence, technical support, etc.)
to resources that do not already nest an `ExtendedContactDetail` element
in their core definition. The triggering use case is a guideline
registry built on EBMonFHIR, but the comment thread explicitly broadens
the scope: the extension should be usable on Composition, definitional
resources, metadata-pattern resources, and even on any other resource
where the author wants to expose contact detail with a stated purpose.
This ticket is the contact-detail half of a small group of "artifact
metadata" extensions Brian filed at the same time (see the sibling
ticket [FHIR-51512](https://jira.hl7.org/browse/FHIR-51512) for the
`artifact-publication-date` extension), and the disposition discussion
on Zulip confirms it has already been pre-applied via PR
[HL7/fhir-extensions#181](https://github.com/HL7/fhir-extensions/pull/181)
(merged 2025-07-19).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Publish a new extension in the FHIR Extensions Pack:

- **URL:** `http://hl7.org/fhir/StructureDefinition/artifact-extended-contact-detail`
- **Status:** active
- **Code:** `artifact-extended-contact-detail`
- **Context (FHIRPath):** `Resource` (i.e., applicable on any Resource).
- **Cardinality on the host element:** `0..*`
- **Type:** `ExtendedContactDetail` (single value of the datatype).
- **Short:** "Contact detail (with purpose) for the resource."
- **Definition:** "An ExtendedContactDetail value that can be attached
  to any Resource to identify a person, organization, role, or
  endpoint that should be contacted for a stated purpose related to
  the resource (e.g., copyright permission, correspondence, technical
  support, clinical follow-up)."
- **Comment:** "Use this extension when the resource being contacted
  about does not already provide a native ExtendedContactDetail-typed
  element."

This is exactly what PR
[HL7/fhir-extensions#181](https://github.com/HL7/fhir-extensions/pull/181)
already merged on 2025-07-19, so the disposition is to ratify the
pre-applied change, set Status=Applied, and close the ticket.

#### Justification

- The reporter's own clarification (2025-10-14 comment) demonstrates
  cross-domain need: guideline registries (EBMonFHIR), Composition,
  definitional resources, metadata-pattern resources, and even regular
  clinical resources like Observation can benefit. Defining this in the
  central Extensions Pack — rather than inside EBMonFHIR — avoids
  forcing every dependent IG to take an EBMonFHIR dependency just to
  attach contact-with-purpose information.
- It is consistent with the precedent set by
  [FHIR-41560](https://jira.hl7.org/browse/FHIR-41560), which
  explicitly relegates simple `ContactPoint`-purpose to "use only when
  not nested inside ExtendedContactDetail." That carve-out only makes
  sense if `ExtendedContactDetail` is the preferred vehicle —
  publishing this extension fills the missing piece so the carve-out
  is actually achievable on resources that do not natively expose
  `ExtendedContactDetail`.
- It mirrors the disposition of the sibling ticket
  [FHIR-51512](https://jira.hl7.org/browse/FHIR-51512)
  (`artifact-publication-date`), which has already moved to **Applied**
  with the same Persuasive resolution.
- The merged PR is small (+70 lines, 1 file) and contained — adding a
  StructureDefinition to the Extensions Pack does not perturb any
  existing artifact.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying need but constrain the publication footprint:
publish the extension in the Extensions Pack but with a **restricted
context list** rather than `Resource`. Concretely, set the context to:

- `Composition`
- `MetadataResource` (i.e., the metadata-pattern descendants:
  `ActivityDefinition`, `PlanDefinition`, `Library`, `Measure`,
  `EvidenceVariable`, `Evidence`, `EvidenceReport`, `Citation`,
  `ChargeItemDefinition`, `ConditionDefinition`, `Questionnaire`,
  `ResearchDefinition`, `ResearchElementDefinition`,
  `SubscriptionTopic`, etc.)
- All `*Definition` resources not already covered above.

The extension URL, type (`ExtendedContactDetail`), short, and definition
remain the same as Disposition A. Resources outside the listed contexts
that genuinely need contact-with-purpose would be expected to either (a)
file a follow-up to widen the context list, or (b) use the existing
`ContactPoint-purpose` extension on individual contact points.

#### Justification

- Most of the use cases the reporter actually names (guideline
  registries, Composition, definitional resources, metadata-pattern
  resources) are concentrated in the artifact / definitional domain;
  the "Observation needing a follow-up contact" example is a stretch
  that could be served by other means (e.g., per-`ContactPoint`
  purpose, or a domain-specific extension).
- A narrower context discourages drift in which `ExtendedContactDetail`
  starts appearing on every resource type via this extension and
  becomes a de-facto core element. Once an extension is published
  on `Resource`, it is hard to claw back.
- It still satisfies the reporter's primary use case (the EBMonFHIR
  guideline registry) and the sibling ticket FHIR-51512 disposition
  precedent, without committing the registry to a maximally-broad
  context.

Trade-offs vs Disposition A: callers who need this extension on a
not-yet-listed resource have to come back and ask, which adds friction
and ticket churn. It also moves against the explicit reporter request
that the extension "can be used on any Resource."

---

### Disposition C: Decline

#### Proposal

Do not publish a new `artifact-extended-contact-detail` extension.
Direct callers to either (a) profile the resource(s) they care about
with a domain-specific extension, or (b) use the existing
`ContactPoint-purpose` extension
([FHIR-41560](https://jira.hl7.org/browse/FHIR-41560)) on the contact
points they already have, or (c) carry the contact-with-purpose
information inside a `Composition` or definitional resource that
already exposes `ExtendedContactDetail`-typed elements.

Reason category: **already addressed by existing mechanism**.

#### Justification

- The Extensions Pack already carries `ContactPoint-purpose`
  (FHIR-41560), so per-contact purpose is solvable without introducing
  another wrapper extension.
- Most resources that would plausibly need this either already have a
  native `contact` / `ExtendedContactDetail` element (Composition,
  the metadata-pattern descendants, definitional resources) or have a
  more domain-specific way to express contact-for-follow-up.
- Adding a Resource-scoped extension creates a permanent surface that
  is hard to retract, and risks duplicating concepts that ought to be
  modeled in profiles.

This is the weakest disposition because it asks every dependent IG to
re-invent the same wrapper, but it is on the table for completeness.

---

### Recommendation

**Recommended disposition:** A

The pre-applied PR
[HL7/fhir-extensions#181](https://github.com/HL7/fhir-extensions/pull/181)
already implements exactly what the ticket requests, the work group
committers were notified on Zulip with no recorded objection, the
sibling ticket [FHIR-51512](https://jira.hl7.org/browse/FHIR-51512) has
been disposed the same way, and the resolution is already marked
**Persuasive**. Disposition B would require the workgroup to (i) revert
or constrain a merged PR and (ii) push back on a reporter who has
already articulated cross-resource use cases — both are net-negative
moves for very little long-term benefit, since broadening the context
later is straightforward and narrowing later is the harder direction.
Disposition C is essentially a no-op that contradicts the
[FHIR-41560](https://jira.hl7.org/browse/FHIR-41560) precedent. The
workgroup should ratify the pre-applied change, set the ticket to
**Applied**, and close it. As a follow-up housekeeping item, confirm
that the merged extension definition matches the contract spelled out
in the Disposition A "Proposal" section (URL, context = `Resource`,
single `ExtendedContactDetail` value, active status), and link the
ticket to the merged PR for traceability.
