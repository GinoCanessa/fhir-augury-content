## Summary

The ticket asks the Clinical Decision Support work group to change four
text-bearing fields in the CDS Hooks specification from the plain `string`
type to the FHIR-style `markdown` type so that authors and consumers may
unambiguously render formatted text in those fields. The fields named are
`CDSService.title`, `CDSService.description`, `Card.summary`, and
`OverrideReason.userComment`. The reporter argues that introducing an
explicit `markdown` type (analogous to FHIR's `markdown` datatype) would
make the intent — that these values may be rendered with markdown
formatting — clear to implementers, rather than leaving it to convention or
client-side guesswork. No related Jira tickets, Zulip threads, or GitHub
items were surfaced for this request.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the declared type of the following four fields in the CDS Hooks
specification from `string` to `markdown`:

- `CDSService.title` (Discovery endpoint response — service metadata)
- `CDSService.description` (Discovery endpoint response — service metadata)
- `Card.summary` (Card returned by a CDS service to an EHR)
- `OverrideReason.userComment` (Free-text user comment captured when a
  user overrides a card's suggestion)

In each field's table row, update the type column from `string` to
`markdown`, add a one-sentence note that the value MAY contain
CommonMark-formatted text, and note that EHRs SHOULD render the value
using a markdown renderer and SHOULD sanitise rendered output (no script
execution, no embedded raw HTML beyond what the EHR's markdown profile
permits). Add a brief explanation of the `markdown` type to the
specification's data types / conventions section, aligned with FHIR's
definition of the `markdown` datatype (a string subject to the same
length/character rules, intended for CommonMark rendering, no embedded
xhtml). Update the JSON Schema / examples accordingly so that the values
remain JSON strings on the wire.

#### Justification

This is exactly what the reporter asked for and it aligns CDS Hooks with
FHIR's existing convention: FHIR distinguishes `string` from `markdown`
specifically so that implementers know when rendering is expected.
`Card.summary` is already, in practice, often rendered with formatting
by EHRs; making the type explicit removes ambiguity, makes the
SHALL/SHOULD around rendering testable, and lets tooling (form builders,
validators, doc generators) treat the field appropriately. Because
`markdown` is, on the wire, still a JSON string, this change is
backward-compatible for producers that emit plain text — existing
content remains valid markdown.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the fields typed as `string` on the wire, but add normative prose
to each field's description stating that the value MAY contain
CommonMark and that EHRs SHOULD render it with a markdown renderer.
Additionally, distinguish the four fields by intent:

- For service-metadata fields (`CDSService.title`,
  `CDSService.description`) and `Card.summary`, allow markdown formally.
- For `OverrideReason.userComment`, retain plain `string` on the
  grounds that this value is end-user-entered free text captured by the
  EHR at click-time, so accepting markdown semantics increases the
  EHR-side rendering / sanitisation burden without a clear authoring
  benefit (most EHRs will not give end users a markdown editor for an
  override reason).

Optionally introduce a non-normative "CDS Hooks data types" appendix
defining the rendering and sanitisation expectations, so the change
lands without inventing a new typed primitive in the JSON schema.

#### Justification

This addresses the underlying need — clarity for implementers about
rendering — without changing the declared JSON schema type. It also
recognises that the four named fields are not symmetric: three are
authored by the CDS service (where markdown is genuinely useful), and
one is captured from an end user (where markdown is rarely authored and
where unsanitised rendering is a real XSS vector). It also avoids
implying that CDS Hooks is adopting the FHIR `markdown` datatype
verbatim — CDS Hooks is JSON-only and does not depend on FHIR
primitives, so naming the type `markdown` could mislead implementers
into expecting full FHIR datatype semantics (length limits, FHIRPath
checks, the FHIR rule against embedded xhtml, etc.).

---

### Disposition C: Decline

#### Proposal

Decline the change and document, in the existing field descriptions,
that the values are plain strings whose rendering is implementation-
defined. Rationale: already-addressed-by-existing-mechanism /
insufficient-justification. Implementers that wish to render formatted
text can already do so by convention; the spec does not forbid markdown,
and adding a new primitive type to a JSON-only specification has a wider
blast radius than the request acknowledges (tooling, schema, validators,
test suites, examples, and existing implementations).

#### Justification

The ticket does not cite implementer pain, conformance failures, or
interoperability problems caused by the current `string` typing — it is
a clarity / hygiene request. CDS Hooks is JSON, not FHIR-on-the-wire, so
a `markdown` "type" would be purely descriptive: it would not change the
JSON Schema, would not be validated by JSON Schema validators, and would
risk being interpreted as importing FHIR `markdown` semantics that CDS
Hooks does not actually enforce. For `OverrideReason.userComment` in
particular, declaring it `markdown` could pressure EHRs to render
end-user-entered text through a markdown engine, which is a net
security/UX regression unless paired with a sanitisation requirement the
spec does not currently have.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the reporter's actual goal — making it explicit
that these fields are intended for formatted rendering — while avoiding
the two real costs of Disposition A: (1) introducing a "type" to a
JSON-only specification that doesn't otherwise have typed primitives,
and (2) sweeping `OverrideReason.userComment` into the same bucket as
service-authored fields, when end-user-entered free text has different
authoring and sanitisation properties. The work group's discussion
should specifically decide whether `OverrideReason.userComment` is
authored content (treat like `Card.summary`) or captured content (leave
as plain string with sanitisation guidance), since that single decision
is what separates A from B. If the work group prefers to mirror FHIR's
type vocabulary one-for-one and is willing to accept the spec-wide
implications, Disposition A is a defensible second choice; Disposition C
should only be chosen if the work group concludes the request is purely
cosmetic and not worth a normative change.
