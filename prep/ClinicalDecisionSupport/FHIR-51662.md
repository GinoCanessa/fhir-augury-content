## Summary

The ticket asks the Clinical Decision Support work group to extend the
"mapping RequestOrchestration to CDS Hooks response" guidance in the
*Using FHIR Clinical Reasoning with CDS Hooks* implementation guide so
that it explicitly describes how a CDS Hooks **system-action** is
produced from a `RequestOrchestration.action`. CDS Hooks distinguishes
system-actions (executed immediately by the system, no user
confirmation) from suggestions (user-mediated). The reporter (Bryn
Rhodes) proposes that the FHIR `action.participant.type` carry that
distinction: `participant.type = device` ⇒ system-action;
`participant.type = user` ⇒ user-mediated action. The action's `type`
remains `create` (or whatever the underlying action verb is); only the
participant signals the execution mode. For R4 implementations, where
`RequestGroup.action.participant` does not have a `type` element, the
ticket calls for a cross-version extension on `RequestGroup.action.participant`
to carry the same value. A prior committee disposition resolved the
ticket "Persuasive with Modification" on 2025-10-15 (14-0-0), which was
then reverted on 2025-11-05 (Jeff Brown / Daryl Chertcoff: 20-0-0) so
the work group can re-vote — likely because the modification also
relocated the content into the *Using Clinical Reasoning with FHIR*
implementation guide and that scope decision warrants a fresh look.
There are no cross-referenced Jira tickets or Zulip threads recorded for
this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the *Using FHIR Clinical Reasoning with CDS Hooks* IG, in the
"mapping RequestOrchestration to CDS Hooks response" section, to add
explicit guidance that:

1. A `RequestOrchestration.action` whose `participant.type = device`
   maps to a CDS Hooks **system-action** (CDS service expects the
   client to apply the action without user confirmation).
2. A `RequestOrchestration.action` whose `participant.type = user`
   maps to a CDS Hooks **suggestion** (user-mediated action requiring
   confirmation).
3. The action's `type` (e.g., `create`, `update`, `delete`) is
   independent of the participant-driven execution mode and continues
   to map per the existing guidance.
4. For the R4 use case, define and reference a cross-version
   extension on `RequestGroup.action.participant` that carries the
   `ActionParticipantType` value (`device` | `user`) so R4 producers
   can convey the same distinction to a CDS Hooks consumer.

Add a worked example for each of the two participant types and a
mapping table row covering system-action.

#### Justification

This is exactly what the ticket asks for and aligns with the existing
FHIR design: `RequestOrchestration.action.participant.type` already
carries `ActionParticipantType` (including `device` and `user`), so the
mapping reuses an existing, bound element rather than introducing a new
flag. CDS Hooks 1.1+ defines system-actions specifically for actions a
client should apply automatically, which fits the semantic of a
`device` participant. Defining a cross-version extension for R4
preserves implementer support for the still-widely-deployed R4 stack
without requiring R4 producers to upgrade to RequestOrchestration. The
change is documentation-only in the IG and can be flagged
non-substantive (correction), matching the previously voted impact
classification.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the substance of Disposition A (use `participant.type = device`
vs. `user` to drive the system-action vs. suggestion mapping, with an
R4 cross-version extension), **but place the new normative mapping
guidance and the cross-version extension definition in the *Using
Clinical Reasoning with FHIR* implementation guide** rather than (or in
addition to) the *Using FHIR Clinical Reasoning with CDS Hooks* IG.
The CDS-Hooks-focused IG would retain a short pointer/cross-reference
to the canonical mapping guidance in the Clinical Reasoning IG.

Concretely:

- Add the mapping section (RequestOrchestration → CDS Hooks
  system-action / suggestion) to the *Using Clinical Reasoning with
  FHIR* IG.
- Define the R4 cross-version extension for
  `RequestGroup.action.participant.type` in the same IG so it lives
  alongside the rest of the Clinical Reasoning extension set.
- Update the *Using FHIR Clinical Reasoning with CDS Hooks* IG to link
  to the new section instead of duplicating it.

#### Justification

The reverted resolution explicitly included relocating the content to
the *Using Clinical Reasoning with FHIR* IG, which suggests the work
group sees the mapping as belonging in the broader Clinical Reasoning
guidance, not solely in the CDS-Hooks-specific IG. Centralizing the
RequestOrchestration ↔ CDS Hooks mapping (and the supporting R4
cross-version extension) in one place avoids duplicate, drift-prone
guidance across two IGs and gives implementers a single canonical
reference. The trade-off vs. Disposition A is scope: this disposition
touches two IGs instead of one and is the change that the work group
voted to re-evaluate, so it deserves an explicit decision rather than
being folded silently into the "Persuasive" path.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped. Rationale: the existing IG
already permits implementations to convey execution-mode intent through
extensions on the action, and CDS Hooks system-actions are a CDS
Hooks-side concept that consumers can derive from
`RequestOrchestration.action.type` and the presence of contributing
extensions. No core change to the mapping guidance is required, and
overloading `participant.type` to carry execution semantics conflates
*who/what performs the action* with *how the receiver should treat the
suggestion*.

#### Justification

`ActionParticipantType` is defined to describe the kind of participant
involved in the action (practitioner, patient, related person, device,
care team, group, healthcare service), not to signal whether a CDS
Hooks consumer should auto-apply the action. Reusing `device` for
"system-action" risks ambiguity where the action genuinely involves a
device participant (e.g., an infusion pump) but is still meant to be
user-mediated. A purpose-built extension on the action (or a CDS
Hooks-side flag in the response itself) would express the intent more
precisely and would not constrain the participant taxonomy. Declining
this specific proposal preserves the option to introduce a cleaner,
purpose-built mechanism in a future ticket.

---

### Recommendation

**Recommended disposition:** B

The ticket's substantive proposal — using
`RequestOrchestration.action.participant.type` (`device` vs. `user`) to
drive the CDS Hooks system-action vs. suggestion mapping, with an R4
cross-version extension on `RequestGroup.action.participant` — is
sound, low-risk, reuses an existing bound element, and was already
acceptable to the work group on its merits (Persuasive with
Modification, 14-0-0). The reason the prior resolution was reverted
(20-0-0 on 2025-11-05) was almost certainly the *location* of the
content, not the technical mapping, given the modification text
explicitly added "Move this content to the Using Clinical Reasoning
with FHIR implementation guide." Disposition B accepts the technical
proposal and makes the IG-placement decision explicit: put the
canonical mapping and the cross-version extension in the *Using
Clinical Reasoning with FHIR* IG and have the CDS-Hooks-focused IG
cross-reference it. This matches the modification the work group was
already comfortable with, avoids duplicated guidance across two IGs,
and gives implementers one place to look. Disposition A is acceptable
but leaves the placement question implicitly answered. Disposition C's
concern about overloading `participant.type` is real but, in practice,
the `device | user` distinction in `ActionParticipantType` already
aligns naturally with the CDS Hooks system/user execution modes and is
the least-friction option for both R5 and (via the cross-version
extension) R4 implementers.
