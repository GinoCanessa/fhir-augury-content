## Summary

The reporter asks the CDS work group (in the context of the Clinical
Guidelines / CPG IG) to clarify how implementers should interpret two
superficially similar ways of pulling a `Questionnaire` into a
PlanDefinition-driven workflow:

1. A "Collect Information" `ActivityDefinition` whose execution produces
   a `Task` carrying a `Questionnaire` reference — which the reporter
   reads as "the questionnaire may be filled out later, asynchronously".
2. A `Questionnaire` referenced **directly** from
   `PlanDefinition.action.definitionCanonical` — which the reporter
   reads as "the questionnaire should be presented and completed
   immediately by the implementing system".

The ticket is essentially a request for the CPG IG to confirm (or
correct) this immediate-vs-deferred semantic distinction, and ideally
to document it normatively so that authors and engines treat the two
patterns consistently. The cross-reference index links the ticket only
to the FHIR element `PlanDefinition.definitionCanonical`; no related
Jira tickets, GitHub items, or formally cross-referenced Zulip threads
were found.

## Details

**Description:**

It is not clear how a Collect Information ActivityDefinition, which
generates a task should be handled versus a Questionnaire in a
PlanDefinition.definitionCanonical element should be handled.

If the Collect Information ActivityDefinition is used to generate a
task with a Questionnaire it can be inferred that the task may be done
later.

If the Questionnaire is directly in the PlanDefinition.definitionCanonical
element it can be inferred that it should be done immediately in the
implementing system. Is this distinction correct for handling of a
Questionnaire versus the collect information ActivityDefinition?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **CPG IG** narrative for `PlanDefinition.action` (and the
matching guidance in the "Collect Information" ActivityDefinition
profile) to explicitly document the two patterns and their intended
runtime semantics:

- **Direct `definitionCanonical` → `Questionnaire`**: the engine
  evaluating the PlanDefinition is expected to surface the Questionnaire
  to the user *synchronously* as part of executing this action
  (subject to the action's `trigger`, `condition`, and `timing`
  constraints). No `Task` is produced by default.
- **`definitionCanonical` → "Collect Information" `ActivityDefinition`
  that yields a `Task` referencing a `Questionnaire`**: the engine
  produces a `Task` (typically `intent = order`, `status = requested`,
  with the Questionnaire as `Task.focus` / `Task.input`) that may be
  fulfilled *asynchronously* by a separate actor at a later time.

Add a short side-by-side table to the CPG IG's PlanDefinition page
illustrating the two patterns, and tighten the narrative on
`PlanDefinition.action.definitionCanonical` so the choice is made on
the basis of "do I need this answered right now in this evaluation, or
do I need durable, assignable work?" rather than implementer guesswork.

#### Justification

The reporter's reading is consistent with how the CPG IG's `$apply`
operation is generally implemented in the wild and with the intent
flagged by Lloyd in the *PlanDefinition activity references* Zulip
thread (canonicals are for stable, definitional artifacts; Tasks carry
deferred work). The Elliot Silver / Brian Postlethwaite exchange in
*\_include PlanDefinition:definition* shows implementers *already*
struggle to justify why `Questionnaire` is allowed at all under
`definitionCanonical`, which is exactly the gap a documentation change
would close. This is a low-risk, narrative-only change in the IG and
matches the ticket's `Fit-For-Teleconference` label — the work group
can ratify the wording in a single call.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than just narrative guidance, **promote the distinction into a
formal pattern** in the CPG IG: define (or reuse) a pair of
`ActivityDefinition`-flavored profiles — e.g.,
`cpg-collectinformationactivity` (already in CPG) and a complementary
`cpg-immediatequestionnaireactivity` — and recommend that
`PlanDefinition.action.definitionCanonical` reference one of these
profiles rather than a bare `Questionnaire`. Then:

- Direct `Questionnaire` references in `definitionCanonical` are
  marked **discouraged** (or warned-on by the reference implementation),
  with conformance guidance saying "wrap a Questionnaire in an
  ActivityDefinition that declares execution semantics".
- `$apply` behavior is keyed off the ActivityDefinition's profile /
  `kind` rather than off whether the canonical happens to resolve to a
  `Questionnaire` vs. an `ActivityDefinition`.

Where appropriate, capture the immediate-vs-deferred choice on the
ActivityDefinition itself (e.g., via an extension or a constrained
`kind`/`code`) so engines have a single, machine-checkable signal.

#### Justification

Disposition A leaves the semantic difference encoded in *which kind of
canonical you happen to point at*, which is precisely the source of
confusion the ticket is reporting. A profile-based approach makes the
intent explicit at authoring time, makes it validatable, and keeps the
PlanDefinition execution engine from having to special-case "is this
canonical a Questionnaire?". It also aligns with the CPG IG's existing
pattern of providing opinionated ActivityDefinition profiles. The
trade-off is non-trivial scope: this would touch multiple CPG IG
profile pages and the `$apply` documentation, and it changes guidance
for content authors who already use the bare-Questionnaire form. That
makes it a larger discussion than the ticket as written calls for.

---

### Disposition C: Decline (Not Persuasive / Already Addressed)

#### Proposal

Close as *Persuasive with Modification — point to existing CPG
guidance* (or *Not Persuasive* if the work group concludes the
distinction is already adequately covered). Specifically, decline to
add new normative semantics in core FHIR; if anything is added, it is
a single sentence in the CPG IG cross-referencing the existing
`$apply` operation's handling of `Task`-producing activities.

#### Justification

The base FHIR `PlanDefinition` resource is intentionally agnostic about
whether actions are executed immediately or deferred — that decision
belongs to the workflow engine and to IGs (CPG, SDC) that layer
execution semantics on top. The reporter's distinction is a reasonable
default but is **not the only valid interpretation**: an engine may
legitimately turn even a directly-referenced `Questionnaire` into a
`Task` (e.g., when the user is offline, when the action needs routing
to another role, or when the PlanDefinition is being applied
server-side without a UI). Locking the synchronous-vs-asynchronous
distinction to the *form* of the canonical reference would over-
constrain implementers without solving the underlying problem.

---

### Recommendation

**Recommended disposition: A**

Disposition A is the smallest change that actually addresses the
reporter's question and aligns with the ticket's `Fit-For-Teleconference`
label and Triaged status. The Zulip evidence — particularly Elliot
Silver's "why is `Questionnaire` even allowed here?" question and
Lloyd's redirect to Bryn Rhodes for an authoritative answer — shows
this is a recurring implementer confusion that the CPG IG can resolve
with narrative + a small example, without committing to the larger
profile-engineering work in Disposition B. Disposition B is worth
keeping on the work group's longer-term backlog (and is a reasonable
follow-up if the narrative fix proves insufficient), but it is broader
than what FHIR-36652 asks for and would benefit from its own ticket so
content authors and the SDC work group can weigh in. Disposition C
under-serves the reporter: even if the base resource should remain
agnostic, the CPG IG is exactly the right place to publish a
recommended interpretation, and silence here will keep producing the
same questions in `#implementers` and `#cql`.
