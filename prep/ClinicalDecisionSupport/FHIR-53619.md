## Summary

The reporter, Igor Bossenko, asserts that the value set bound to
`RiskAssessment.code` in the FHIR core specification is incorrect. Today the
element is bound (with strength `example`) to the
[Risk Assessment Method](https://build.fhir.org/valueset-risk-assessment-method.html)
value set. That value set enumerates the *method or instrument* used to perform
a risk assessment (e.g., scoring tools), whereas `RiskAssessment.code` is the
"what is being assessed" element — i.e., the type of risk being evaluated
(suicide risk, fall risk, cardiovascular risk, etc.). The ticket asks the
binding to be changed to "the separate VS" — i.e., a value set whose contents
describe the kind/type of risk assessment rather than the method employed.

No related Jira tickets are explicitly linked. Two historical tickets surfaced
by keyword/cross-reference scan are relevant for context: the now-closed
[FHIR-7869](https://jira.hl7.org/browse/FHIR-7869) ("2015May core #1189 — Add
Risk Assessment type", *Persuasive*, *Published*) which originally introduced
the *type-of-risk-assessment* concept on the resource and proposed an example
value set such as the SNOMED descendants of 225338004 |Risk Assessment|, and
its duplicate [FHIR-8537](https://jira.hl7.org/browse/FHIR-8537) ("RiskAssessment
needs a type code", *Duplicate*) which made the same proposal. Together they
establish the original intent that `RiskAssessment.code` (or its predecessor
`type`) should be bound to a value set of *risk-assessment types* (e.g.,
descendants of SNOMED 225338004), not to a methods value set — exactly the gap
this ticket is calling out.

## Related Jira Tickets

### [FHIR-7869: 2015May core #1189 - Add Risk Assessment type](https://jira.hl7.org/browse/FHIR-7869)

This 2015 change request proposed adding a `type` element on `RiskAssessment`
because not all risk assessments pertain to a specific condition (e.g.,
generic safety or pediatric risk). It was resolved *Persuasive* / *Published*
with the resolution that a `type` element of `CodeableConcept` (0..1) be
added. This is the historical origin of what is today `RiskAssessment.code`,
and the resolution implies the intended value set is one describing the
*kind* of risk being assessed — directly relevant to FHIR-53619's complaint
that the current binding instead points at *methods*.

Reporter: Michelle Miller. Assignee: Unassigned. Resolution authored within
Clinical Decision Support work group.

### [FHIR-8537: RiskAssessment needs a type code](https://jira.hl7.org/browse/FHIR-8537)

A near-duplicate of FHIR-7869, asking to add `RiskAssessment.type` or
`RiskAssessment.code` as a `CodeableConcept`, and explicitly suggesting an
example value set built from SNOMED CT descendants of 225338004 |Risk
Assessment|. Closed as *Duplicate*. It establishes the originally intended
content of the value set bound to this element — i.e., a *type-of-risk*
enumeration, not the *method* enumeration that is currently bound.

Reporter: Mark Kramer. Assignee: Unassigned.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the binding on `RiskAssessment.code` in FHIR Core. Specifically:

- Detach `RiskAssessment.code` from
  `http://hl7.org/fhir/ValueSet/risk-assessment-method` (which describes the
  *instrument/method* of the assessment, e.g., specific scoring tools).
- Bind `RiskAssessment.code` to a new (or repurposed) value set whose
  contents describe the *type of risk being assessed* — e.g., descendants of
  SNOMED CT 225338004 |Risk Assessment| (suicide risk assessment, fall risk
  assessment, cardiovascular risk assessment, etc.), as originally proposed in
  FHIR-7869 / FHIR-8537.
- Keep the binding strength as `example` (no normative tightening), matching
  current practice for this element and consistent with the
  [Example](https://build.fhir.org/terminologies.html#example) strength.
- Reserve `risk-assessment-method` for use on a *method* element, either on a
  future `RiskAssessment.method` (if added) or as an extension; alternatively
  retire the value set if no other element will reference it.

#### Justification

This restores the original design intent recorded in FHIR-7869 (resolved
*Persuasive* / *Published*) and FHIR-8537, both of which framed the element as
a *type-of-risk* code drawn from a SNOMED-based value set. It aligns with the
shared understanding voiced by Patient Care leadership on Zulip
(`RiskAssessment.code` = "what risk was assessed"). The current binding to a
*methods* value set is a category error: it conflates the subject of the
assessment with the technique used to perform it, which is misleading to
implementers and incompatible with the element's definition. Keeping the
strength at `example` minimises implementation impact while correcting the
content.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-binding `RiskAssessment.code` to a new value set, address the
underlying confusion by:

1. Adding a new `RiskAssessment.method` element (`CodeableConcept`, 0..1)
   bound to the existing `risk-assessment-method` value set with strength
   `example`, so the methods VS continues to have a home.
2. Either (a) leaving `RiskAssessment.code` with **no** value set binding (a
   plain `CodeableConcept` with no example VS, since "type of risk" is too
   broad to enumerate well) and tightening only the element definition to
   make clear that it is the *subject* of the assessment, or (b) binding
   `RiskAssessment.code` to a small example VS of canonical risk-assessment
   types (e.g., SNOMED 225338004 descendants) as in Disposition A.
3. Updating examples and the resource narrative to demonstrate the
   `code` / `method` split.

#### Justification

This trades a slightly larger structural change (adding `method`) for a
clearer long-term model. It separates the two concerns (what is assessed vs.
how it is assessed) at the schema level, not just at the binding level, which
is more durable than relying on implementers to read element definitions
carefully. It also preserves the existing `risk-assessment-method` VS rather
than orphaning it. The cost is a non-trivial structural change to a
maturity-stable resource and the need to update existing implementations and
profiles; if Disposition A is sufficient, this added structure may not be
warranted.

---

### Disposition C: Decline

#### Proposal

Decline the change. Retain the current binding of `RiskAssessment.code` to
`risk-assessment-method` (strength `example`). Optionally clarify the
element definition to explain that, where the value set does not contain a
suitable code, implementers should supply a code describing the type of risk
being assessed from any appropriate code system (e.g., SNOMED CT descendants
of 225338004).

#### Justification

The binding strength is `example`, which is explicitly non-normative and
imposes no conformance burden on implementers — they may use any code from
any code system. From a strict conformance standpoint the "wrong" VS does no
harm, and changing it could invalidate downstream profiles, IGs, and tooling
that reference the URL `risk-assessment-method` against `RiskAssessment.code`.
Declining avoids churn on a published, mature resource.

---

### Recommendation

**Recommended disposition:** A

The reporter's complaint is substantively correct: `RiskAssessment.code`
denotes *what* risk is being assessed, and the current example binding points
at a value set that enumerates *how* an assessment is performed. The Patient
Care Zulip discussion and the original Persuasive resolution of FHIR-7869
both confirm that the intended semantics for this element are
"type/subject of risk assessment", not "method". Disposition A is the
minimum necessary correction: it restores design intent with no
schema-shape change and no conformance impact (the strength remains
`example`). Disposition B is attractive in the long run but is heavier than
needed to resolve this ticket and can be pursued separately if CDS later
wants to model `RiskAssessment.method` first-class. Disposition C is
defensible only on the grounds that `example` bindings are advisory, but
leaving a known-incorrect example binding in place is a poor signal to
implementers and to authors of profiles/IGs that copy the binding forward;
the marginal cost of fixing it now is small relative to the ongoing
confusion.
