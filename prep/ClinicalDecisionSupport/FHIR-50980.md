## Summary

The reporter (Bryn Rhodes) requests that the Clinical Quality Language (CQL)
specification add support for **language translation** of two distinct surfaces:
(A) human-readable review of CQL logic (i.e., the source / annotated form that
clinicians and reviewers read), and (B) the textual output produced when CQL
expressions are evaluated (e.g., narrative messages and labels emitted by
expressions). The ticket points at GNU gettext's `ngettext` plural-aware
translation function as a reference for how plural forms should be handled,
and asks the workgroup to align whatever mechanism is adopted with the
existing FHIR IG language-support guidance at
`https://build.fhir.org/ig/FHIR/ig-guidance/languages.html`. A single comment
from the reporter points at the prototype repository
[FHIR/translation-utils](https://github.com/FHIR/translation-utils/tree/main/translation-utils)
as related prior art. No Augury cross-referenced Jira tickets were found, so
the request stands alone in the Augury index, although the historical
multi-language implementation discussion in Zulip (see below) provides the
broader FHIR context.

## Related Zulip Discussions

### [Multi-Language Implementation > Translation file format](https://chat.fhir.org/#narrow/stream/Multi-Language%20Implementation/topic/Translation%20file%20format/near/304469210)

A foundational 2022 discussion (moved from `#Multi-Language Implementation >
multi-lingual CQL`) on which translation file format FHIR tooling should adopt.
Participants converged on two main candidates — **XLIFF** (OASIS) and **GNU
gettext / `.po` files** — and weighed maturity, plural-form handling, markup
support, online tooling (Transifex, Launchpad, translatewiki), and round-trip
extraction/rendering. Carl Leitner argued plural forms are best represented in
the gettext form (which is also reusable inside XLIFF, as Transifex does);
Grahame Grieve tentatively suggested using XLIFF for resource/IG translations
and possibly gettext/PO for code-level messages, while noting that real-world
XLIFF tooling (including Transifex) is not strictly conformant. David Otasek
reported that the core Java libraries currently use `MessageFormat` and
explored ICU4J's `PluralRules` as a cleaner way to manage plural-aware
property-file keys (`some.key.plural_form.one/few/many/other`). No final
decision was reached, but the thread establishes that plural handling
(precisely what `ngettext` provides) is a cross-cutting concern that any CQL
translation mechanism should align with rather than reinvent.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a normative section to the CQL (FHIR) specification, "Language and
Translation", that defines:

1. **A `language` metadata declaration** at the CQL library level (e.g.,
   `library Foo version '1.0.0' language 'en'`) and a corresponding `language`
   element on the generated `Library` resource, mirroring how IGs declare a
   primary language per `ig-guidance/languages.html`.
2. **Translation of CQL "review" surfaces** — descriptions, `@description`,
   `@comment`, and `@display` annotation tags, and any human-readable labels
   on declarations — by attaching ordered `translation` extensions
   (`http://hl7.org/fhir/StructureDefinition/translation`) to the relevant
   string fields on the published `Library`/`PlanDefinition` resources, so
   tooling can render the CQL logic in the reader's locale without altering
   the executable text.
3. **Translation of expression output** by adding a normative `Message()`
   convention that recognises plural forms. Concretely, define a
   `System.Message` (or extend the existing `Message()` operator) so that the
   message identifier resolves through a translation catalog keyed on the
   library's language, and standardise an `ngettext`-style two-form (singular
   / plural) and ICU-style `n`-form (`one`, `few`, `many`, `other`) signature
   for plural-aware messages.
4. **File format alignment with IGs:** specify that translation catalogs
   accompanying a CQL library are published in the same format(s) the IG
   publisher uses (today: XLIFF for IG content), and recommend gettext/PO or
   ICU4J property files for code-level / expression-output strings, leaving
   the binding mechanism (sidecar file vs. embedded `Library.content`)
   normative.

#### Justification

This directly addresses both surfaces (A) and (B) called out in the ticket and
explicitly aligns with `ig-guidance/languages.html` as the reporter requested.
Reusing the existing `translation` extension for review-surface strings keeps
CQL consistent with how the rest of FHIR localises human-readable content,
and standardising a plural-aware `Message()` convention answers the
`ngettext` reference. The 2022 Multi-Language Implementation thread shows the
community has already converged on XLIFF for IG content and on plural handling
as a first-class concern, so adopting that stack avoids inventing a
CQL-specific format. The `FHIR/translation-utils` prototype is available as a
starting reference implementation for the catalog format and lookup behaviour.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add translation machinery to the CQL specification itself. Instead,
publish a companion FHIR Implementation Guide — "CQL Translation IG" (or fold
this into the existing CPG / CRMI guidance) — that defines:

- A profile on `Library` requiring `language` and permitting one or more
  `Library.relatedArtifact` entries of type `depends-on` that point to
  translation catalog `Binary` or `Attachment` resources (XLIFF for narrative
  /review content, gettext-PO or ICU property files for runtime messages).
- A small extension pack for the `translation` extension on CQL annotation
  tags rather than a base-spec change.
- A non-normative `Message()` usage pattern showing how engines should resolve
  message identifiers through the catalog, without changing the CQL grammar.

The CQL base spec would gain at most an informative cross-reference to the
new IG.

#### Justification

CQL is a cross-domain expression language used well beyond FHIR, and baking
specific translation mechanics (catalog formats, plural-form syntax) into the
normative spec risks coupling it to choices — XLIFF vs. PO vs. ICU4J — that
the broader community is still debating, as the 2022 Zulip thread makes clear
("transifex is not strictly conformant", "horrifying syntax", etc.). Doing
the work in an IG keeps CQL itself stable, lets implementers iterate on
catalog format without revving the language, and mirrors how FHIR has handled
other cross-cutting localisation needs (the IG-level `languages.html`
guidance is itself outside the base spec). It also avoids forcing every CQL
engine to implement a plural-rules library before they can be conformant.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the CQL (FHIR) specification.
Translation of CQL "review" surfaces is already achievable today by attaching
the standard `http://hl7.org/fhir/StructureDefinition/translation` extension
to the relevant string elements on the published `Library` resource, and
translation of CQL expression *output* is the responsibility of the consuming
application (which already controls locale, formatting, and plural rules in
its own UI stack). Recommend the workgroup close the ticket as
"Persuasive with Mod – Use existing translation extension" and add a
non-normative note to the CQL spec pointing at
`ig-guidance/languages.html` and the existing `translation` extension.

#### Justification

FHIR already has a general-purpose translation extension that has been used
for narratives, code displays, and other localisable strings; CQL libraries
publish through `Library` resources that can carry it without any spec
change. For runtime expression output, every consuming engine sits inside a
host application that has its own i18n stack (Java `MessageFormat` / ICU4J,
.NET `ResourceManager`, JavaScript `Intl`), and pushing plural-form
resolution into CQL itself — as the 2022 Zulip thread illustrates — opens a
bottomless pit of catalog-format and tooling choices that the workgroup has
not yet resolved after three years of discussion. Declining keeps the CQL
spec focused on its core concern (computable expressions) and lets the
broader Multi-Language Implementation effort settle on a single FHIR-wide
answer that CQL can adopt later by reference.

---

### Recommendation

**Recommended disposition:** B

The ticket raises a real and well-motivated need, but the 2022
Multi-Language Implementation thread shows that the FHIR community has not
yet settled the foundational questions — XLIFF vs. gettext/PO vs. ICU4J,
where plural rules live, what tooling is actually round-trippable — that any
normative CQL change would have to commit to. Disposition A would force the
CQL spec to make those choices unilaterally and risk drifting from whatever
the IG-side guidance ultimately standardises; Disposition C under-delivers
because the reporter's "review of CQL logic" surface (annotation tags,
`@description`, library-level metadata) genuinely is a CQL concern that the
generic `translation` extension only partially covers, and the
`ngettext`-style plural support for expression output is not addressable
purely in the consumer. A companion IG (Disposition B) is the right vehicle:
it lets the workgroup deliver the `language` declaration, the annotation
translation profile, and a recommended catalog format now, while leaving room
to harmonise with the wider Multi-Language Implementation effort and the
`FHIR/translation-utils` prototype without re-balloting the CQL base spec.
