## Summary

Bryn Rhodes is asking the CQL-on-FHIR specification (specifically the
ModelInfo / retrieve semantics used by the CQL-to-ELM translator) to
natively understand FHIR's `CodeableReference` choice pattern so that a
CQL retrieve with a terminology filter can be pushed through the
retrieve even when the targeted element is a choice between an
inline-codeable concept and a `Reference` to a separately-coded
resource. Today, against an element such as `MedicationRequest.medication`
(which in modern FHIR is `CodeableReference(Medication)` — and was
historically `medication[x]: CodeableConcept | Reference(Medication)`),
authors must hand-write a union: one branch retrieves rows where the
inline code matches, and a second branch joins to the referenced
resource (e.g., `Medication`) and filters on its code. The ticket
proposes that the translator/ModelInfo recognize this pattern and emit
the equivalent expanded retrieve automatically, so an author can write
`[MedicationRequest: "Aspirin"]` and have it transparently match both
branches. No cross-referenced Jira tickets, Zulip threads, or GitHub
items were located via the data sources, so the request stands on its
own description.

## Details

**Description:**

> A common pattern in FHIR is the CodeableReference, which allows for
> representation of a concept through either a terminology or a
> reference to a resource which has a terminology. When retrieves are
> invoked against elements of this type, the current retrieve cannot
> push the terminology filter through the retrieve, forcing authors to
> separate the terminology filter outside the retrieve. The ModelInfo
> should be aware of and support this pattern so that authors do not
> have to deal with this complexity.
>
> An example of this is the MedicationRequest resource medication
> element. When retrieving a MedicationRequest, the translator could
> expand the retrieve to:
>
> ```cql
> define TestChoiceMedicationRequest:
>   [MedicationRequest: "Aspirin"]
>
> // If the terminology target is a choice of a terminology-valued
> // element and a reference
>
> define TestChoiceMedicationRequestExpanded:
>   [MedicationRequest: medication in "Aspirin"]
>     union (
>       [MedicationRequest] MR
>         with [Medication] M
>           such that M.id = Last(Split(MR.medication.reference, '/'))
>             and M.code in "Aspirin"
>     )
> ```

## Proposed Dispositions

Note: the dispositions below reflect the absence of any cross-referenced
GitHub repo briefings. They name the conceptual artifacts (CQL ModelInfo,
retrieve semantics in the CQL-on-FHIR spec) but cannot be tied to a
specific authoring root in a cached briefing. If the workgroup decides
to act, the implementation owner should run `repo-analysis` against
the relevant CQL-on-FHIR / cqframework repos before drafting concrete
code/spec changes.

### Disposition A: Accept As Requested

#### Proposal

Update the CQL-on-FHIR ModelInfo and the retrieve-translation rules so
that when a retrieve's code path resolves to either (a) a
`CodeableReference(X)` element, or (b) a legacy `xxx[x]` choice between
a coded type (`CodeableConcept` / `code` / `Coding`) and a
`Reference(X)`, the translator automatically emits the union of:

1. A direct retrieve filtered on the inline coded branch
   (`[MedicationRequest: medication in "Aspirin"]`), and
2. A retrieve of the source resource joined (`with`) to the referenced
   target resource, filtered on the target's primary code element
   (e.g., `[Medication] M such that M.id = …MR.medication.reference… and
   M.code in "Aspirin"`).

Concretely:

- In ModelInfo, mark the choice/CodeableReference elements with the
  pair `(localCodePath, referencedResourceCodePath)` so the translator
  knows which element on the referenced resource carries the code
  (e.g., `Medication.code`, `Substance.code`, `Device.type`).
- In the translator, when a retrieve's `codeProperty` resolves to such
  an annotated element, generate the two-branch union shown in the
  ticket (using the standard `Reference` resolution helper, not raw
  string splitting, so the implementation is engine-agnostic).
- Document the new behavior in the CQL-on-FHIR specification's
  retrieve semantics section, with `MedicationRequest.medication` as
  the worked example.

#### Justification

The pattern is exactly what authors write by hand today, and the
two-branch union is the only correct interpretation of "retrieve
MedicationRequests for Aspirin" when `medication` may be either inline
or a reference. Pushing this into the translator (a) eliminates a
well-known foot-gun where authors silently miss the reference branch,
(b) keeps existing simple `[Resource: code]` syntax working, and (c)
is consistent with how the ModelInfo already abstracts other FHIR
quirks (choice types, primitive extensions) away from the author.
`CodeableReference` was added precisely to formalize this dual
representation, so the ModelInfo should track that formalization.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than auto-expanding the retrieve, define the support at the
*model* level via a new "code path with reference fallback" annotation
in ModelInfo, and expose a deterministic helper function (e.g.,
`ResolveCodeableReference(CodeableReference) returns List<Code>`) that
authors — and the translator's existing equality-style filters — can
use without changing retrieve semantics. The translator would then
optimize a retrieve whose code path resolves to such an element by
calling the helper inline rather than emitting a `union`-with-`with`
join. Optionally, gate the auto-union behavior behind a translator
option (`enableCodeableReferenceExpansion`) defaulted off for one
release cycle.

#### Justification

The auto-expanded `union` in Disposition A changes the runtime
semantics of `[MedicationRequest: "Aspirin"]` in subtle ways: it now
requires a join to a second resource type, which changes the data
that must be available to the engine, may change cardinality, and may
surprise authors who are used to retrieve being a single-resource
operation. A helper-based approach keeps the retrieve a
single-resource operation and makes the reference-branch behavior
*explicit* in the author's CQL, while still giving the ModelInfo the
metadata it needs. A feature flag during the transition limits
regressions in deployed CQL libraries.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the CQL-on-FHIR retrieve
semantics, on the grounds that (i) the existing `union` + `with`
pattern is already expressible and well-understood, (ii) the proposed
expansion bakes a FHIR-resource-specific assumption
(`MedicationRequest.medication` ↔ `Medication.code`) into translator
behavior, and (iii) implementations differ in whether the referenced
resource is even available in the data store at retrieve time, so the
optimization may not fire uniformly. Recommend instead that authoring
guides document the canonical hand-written union pattern.

#### Justification

CQL has historically kept retrieve narrow and predictable: one
resource type, one code path. Pushing a cross-resource join into
retrieve breaks that contract and creates engine-portability hazards
— a server that can serve `MedicationRequest` may not be able to
serve `Medication` in the same context, in which case the
auto-expansion silently returns the wrong (smaller) result set. If the
real complaint is authoring complexity, that is better addressed by
shared library functions (e.g., a published `CommonHelpers` library)
than by changing core retrieve semantics.

---

### Recommendation

**Recommended disposition:** A

The `CodeableReference` pattern is now pervasive in modern FHIR
resources (`MedicationRequest.medication`, `Condition.evidence`,
`Procedure.reason`, etc.), and forcing every CQL author to hand-roll
the same two-branch union for each one is precisely the kind of
boilerplate the ModelInfo abstraction exists to remove. Accepting the
request — with explicit ModelInfo annotations naming the referenced
resource's code path, and a worked specification example — gives
implementers a clear, single contract to satisfy. The portability and
"is the referenced resource available?" concerns raised in
Disposition C are real, but they apply equally to the hand-written
union authors already write today; codifying the expansion in the
translator at least makes the assumption uniform and inspectable.
Disposition B's helper-function path is a reasonable fallback if the
workgroup is uncomfortable changing retrieve semantics in a single
step, and could be folded into the Disposition A rollout as the
underlying mechanism (i.e., the translator's auto-union is implemented
*in terms of* the new helper), giving authors both the convenience
of A and the explicitness of B.
