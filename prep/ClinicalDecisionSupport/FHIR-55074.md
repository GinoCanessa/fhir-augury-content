## Summary

The submitter (Lloyd McKenzie, comment-imported) reports that
`RequestOrchestration.intent` is a 1..1 element with a required binding to
the `request-intent` value set, but that value set provides no `unknown`,
`other`, or equivalent "escape valve" concept. As a result, the element —
and therefore any `RequestOrchestration` instance — is impossible to
populate when the source system does not actually know which intent applies.
The reporter asks that an escape-valve concept be added to the bound value
set.

This ticket is one of a family of identically-worded change requests filed
against every resource that binds an `intent` element to the shared
`request-intent` value set. The umbrella tracker for the underlying pattern
issue is [FHIR-56142](https://jira.hl7.org/browse/FHIR-56142) (Request.intent
has no escape valve), which was filed by Michelle Miller against FHIR
Infrastructure and has since been **resolved as Persuasive**. Its agreed
resolution is to (1) add an `Other` code to the `request-intent` code system
with a definition restricting it to cases where no existing intent code
applies (typically future-version concepts back-ported via extension), and
(2) add a "Choosing an intent code" guidance section to the value set's
notes telling implementers to send the most specific code possible and to
fall back to `proposal` (not `Other`) for legacy data with unknown intent.
[FHIR-55336](https://jira.hl7.org/browse/FHIR-55336) (MedicationRequest.intent
has no escape valve) is the corresponding sibling tracker filed against
Pharmacy and is still Triaged. Bryn Rhodes' comment on this ticket
explicitly asks whether a generic Request-pattern tracker exists so the
group can "get consistent resolution to all the trackers that then derive
from `Request.intent`"; Lloyd confirms one has been raised. That generic
tracker is FHIR-56142, and because `RequestOrchestration.intent` binds to
the same `request-intent` value set, applying FHIR-56142's resolution
should resolve FHIR-55074 automatically — but the resolution has not yet
been propagated/closed against this CDS-owned ticket.

## Details

**Description:**

The element RequestOrchestration.intent is a mandatory element with a
required binding, but appears to have neither an 'unknown' or 'other'
concept, which can make the containing element impossible to populate in
some cases. Please add 'unknown', 'other', or some equivalent escape valve
concept to the value set.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an escape-valve concept to the `request-intent` value set / code system
that `RequestOrchestration.intent` binds to, mirroring the resolution
already adopted on the parent ticket [FHIR-56142](https://jira.hl7.org/browse/FHIR-56142).
Specifically:

1. Add an `Other` concept to the `request-intent` code system with the
   definition: *"Indicates that the intent value being communicated does
   not fall within any of the allowed intent codes for the version of FHIR
   being communicated. The specific intent concept SHOULD be communicated
   in an extension. Systems will not be able to safely process (and might
   not accept) instances with this code unless they are familiar with the
   extension communicating the actual intent."*
2. Add the FHIR-56142 "Choosing an intent code" guidance block to the
   notes section of `RequestOrchestration` (or rely on the value-set-level
   guidance once it lands), reinforcing that:
   - Senders SHOULD transmit the most specific intent code applicable.
   - Where intent is genuinely unknown for legacy/external data, systems
     SHOULD fall back to `proposal` (the safest, broadest concept), not
     `Other`.
   - `Other` is reserved for future-version concepts back-ported via an
     extension, and must not be used as a generic "unknown".
3. Resolve FHIR-55074 with disposition **Persuasive — Resolved by
   FHIR-56142**, adding a comment that no resource-specific change to
   `RequestOrchestration` is required because the value-set fix
   propagates automatically through the required binding.

#### Justification

The substantive design decision has already been made and voted by FHIR-I
on the parent tracker FHIR-56142 (Persuasive, 4-0-0). Because
`RequestOrchestration.intent` shares the `request-intent` value set with
the other Request-pattern intent elements, the FHIR-I change closes this
gap with no additional content change needed in CDS-owned files. Adopting
Disposition A keeps all `*.intent` tickets aligned, honors Bryn Rhodes'
explicit request for a consistent cross-resource resolution, and matches
the consensus reached on the "Handling 'escape' valve JIRAs" Zulip thread
where Patient Care, FHIR-I, and OO all converged on a single
pattern-level fix.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding an `Other` code to the bound value set, **relax the
`RequestOrchestration.intent` binding strength from `required` to
`extensible`** (or, equivalently, recommend implementers use the standard
`data-absent-reason` extension on the element to signal `unknown`). Under
this approach the value set itself stays clean of escape-valve codes, but
implementers gain a sanctioned mechanism to express "intent not known" via
either an out-of-VS code or a DAR extension on the element.

#### Justification

This alternative localises the change to `RequestOrchestration` and avoids
overloading the shared `request-intent` value set with a code (`Other`)
whose definition is narrow and easily misused as a generic "unknown".
However, it has real trade-offs the workgroup should consider:
binding-strength changes are substantive, divergent from the FHIR-56142
resolution (so the `*.intent` family would no longer be uniform), and
DAR-on-`intent` raises the question Lloyd already flagged in FHIR-56142
comments — *"Data absent reason doesn't have 'Other'"* — which is the
reason FHIR-I rejected a DAR-based approach there. Disposition B is
therefore primarily useful as a fallback if CDS objects to the
value-set change FHIR-I has already adopted.

---

### Disposition C: Decline

#### Proposal

Close FHIR-55074 as **Not Persuasive — Already Addressed**, on the
grounds that (a) the substantive change has been resolved upstream on
FHIR-56142, and (b) implementers needing an escape valve today already
have one: the spec's existing guidance permits `proposal` as the
broadest, safest fall-back when the more specific intent is unknown,
which is exactly the rationale documented in FHIR-56142's "Choosing an
intent code" guidance.

#### Justification

This avoids a CDS-side change entirely and treats the resource-level
ticket as duplicative of the pattern-level resolution. The downside is
that it leaves the ticket without a positive linkage to FHIR-56142 in
Jira, and the reporter's underlying complaint (no `unknown`/`other`
concept exists in the value set) remains literally true until the
FHIR-56142 build artifacts ship. If declined this way, the comment
should at minimum cite FHIR-56142 so the reporter and downstream
ticket-watchers know where the resolution actually lives.

---

### Recommendation

**Recommended disposition:** A

The pattern-level decision has already been taken by FHIR-I on FHIR-56142
(Persuasive, 4-0-0) and its resolution mechanically resolves this ticket
because `RequestOrchestration.intent` shares the `request-intent` value
set. Disposition A keeps CDS aligned with FHIR-I and with the sibling
Request-pattern resolutions, satisfies Bryn Rhodes' explicit ask for
cross-tracker consistency, and matches the Zulip-thread consensus that a
single value-set fix should subsume all the escape-valve trackers. CDS
should formally close FHIR-55074 as **Persuasive — Resolved by
FHIR-56142**, link the two tickets in Jira, and confirm at next
Workflow-pattern review that the `Other` code and guidance text actually
ship in the value set / `RequestOrchestration` notes section. Disposition
B should only be considered if the workgroup has a substantive objection
to the FHIR-I value-set change (none has surfaced in the available data),
and Disposition C is sub-optimal because it discards the explicit Jira
linkage to FHIR-56142 that future implementers will need to find the
actual fix.
