## Summary

The reporter (Dan Murphy) is asking the Clinical Decision Support work group to
add normative or implementation guidance to the US Da Vinci DTR IG (and
potentially the CRD IG) describing how a DTR client should behave when a CRD
server returns multiple `coverage-information` extensions on a single order
resource and more than one of them carries `doc-needed` while pointing at
different `questionnaire` URLs. The CRD IG explicitly permits this pattern in
its General Strategies / Cards guidance, but neither CRD nor DTR currently
tells the DTR client (Light or Full) how to disambiguate the resulting set of
required forms — leaving open whether the EHR should collapse the repetitions
before launching DTR, or whether the DTR app should surface all candidate
questionnaires and let the user choose. The ticket draws on a Da Vinci CRD
Zulip thread in which Lloyd McKenzie indicated that responsibility most
naturally sits on the DTR client side: the client should recognise that some
questionnaires are conditional (e.g., on performing provider, location, date
range, or other qualifiers carried in `coverage-information`) and present
those conditions so the user can choose to fill out all potentially-applicable
forms, only the ones they know are needed, or defer until the order is more
fully specified. The cross-reference index returned no linked Jira tickets,
GitHub items, or Confluence pages for FHIR-56023, so the request is not yet
tied to a concrete edit location or PR.

## Details

**Description:**

> The [CRD IG](https://build.fhir.org/ig/HL7/davinci-crd/en/cards.html#:~:text=This%20specification%20allows,make%20a%20decision.)
> permits a CRD server to return multiple coverage-information extensions on a
> single order resource, each potentially carrying doc-needed and referencing
> a different questionnaire URL. When this occurs, there is currently no
> guidance on how the DTR client should handle the resulting ambiguity.
>
> Additional guidance is warranted describing the expected DTR client
> behavior when multiple coverage-information extensions with questionnaires
> are present on the same order resource.
>
> Based on a [Zulip thread](https://chat.fhir.org/#narrow/channel/180803-Da-Vinci-CRD/topic/Multiple.20coverage.20information.20extensions/with/577595593),
> some initial thoughts:
>
> - The DTR client bears responsibility for recognizing that questionnaires
>   may be conditional (based on qualifiers such as performing provider,
>   location, or timeframe) and surfacing those conditions to the user.
> - The user should be given the choice to fill out all potentially
>   applicable forms, only the forms they know are needed, or defer until
>   the decision process is further along.

## Related Zulip Discussions

### [Da Vinci CRD > Multiple coverage information extensions](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20CRD/topic/Multiple%20coverage%20information%20extensions)

Dan Murphy opened the thread asking, in the context of the CRD IG's General
Strategies section, what an EHR / Light DTR workflow should do when a payer
returns multiple `coverage-information` extensions on the same order with more
than one carrying `doc-needed` — specifically whether the EHR should collapse
them to a single coverage-information before launching DTR, or whether DTR
must be able to consume multiple `questionnaire` references in its launch
context (and if so, what conformance language is needed; he also floats
DTR-specific Request profiles that limit `coverage-information` cardinality
to 1). Lloyd McKenzie responded that while an EHR *could* reduce the
repetitions exposed to DTR, that would be sophisticated and potentially
inappropriate (the suppressed extension might still matter if the order is
revisited); he placed responsibility on the DTR client (Light or Full) to
detect that some questionnaires are conditional on qualifiers (timeframe,
performing provider, location, etc.) and to expose those conditions so the
user can choose to fill out every potentially-needed form, only the forms
they know are required, or to defer until the order is further specified.
Lloyd then explicitly invited a change request to add this guidance to the
spec or the implementation-guidance Confluence page — which is what
FHIR-56023 captures. No formal consensus was reached, but no participant
disagreed with Lloyd's framing.

Participants: Dan Murphy, Lloyd McKenzie.

### [Da Vinci CRD > Use cases for multiple coverage-information extensions](https://chat.fhir.org/#narrow/stream/Da%20Vinci%20CRD/topic/Use%20cases%20for%20multiple%20coverage-information%20extensions)

Sean O'Quinn asked for real-world examples that justify the existing
cardinality (allowing multiple `coverage-information` objects in an
order-sign `systemAction`), offering the case of a single CT Brain order
that maps to a range of HCPCS codes (70450/70460/70470) as one possibility.
Lloyd McKenzie confirmed that's a valid example but emphasised that the
underlying ServiceRequest needn't carry an explicit `billingOptions` array;
multiple `coverage-information` repetitions are needed any time the payer's
answer is "depending on X, the answer could be 1, 2, or 3" — e.g., different
billing codes, different rules for different date ranges, or inpatient vs.
outpatient. This thread is directly relevant because it grounds the use
cases the DTR-side guidance proposed in FHIR-56023 needs to handle:
condition-bearing repetitions whose applicability depends on facts the
DTR user (not the payer) is best positioned to confirm.

Participants: Sean O'Quinn, Lloyd McKenzie.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new normative/implementer-guidance subsection to the DTR IG —
positioned alongside the existing DTR client launch / Questionnaire
selection guidance — titled along the lines of "Handling multiple
coverage-information extensions with doc-needed". The subsection would
state:

1. The DTR client (both Light and Full) **SHALL** be able to consume a
   launch context whose underlying order carries more than one
   `http://hl7.org/fhir/us/davinci-crd/StructureDefinition/ext-coverage-information`
   extension with `doc-needed = true` and distinct `questionnaire` URLs.
2. The DTR client **SHALL NOT** silently pick one questionnaire when
   multiple are returned; it **SHALL** present all candidate
   `questionnaire` references to the user.
3. For each candidate, the DTR client **SHALL** surface any qualifying
   conditions carried on the corresponding `coverage-information`
   repetition — including (but not limited to) `performingProvider`,
   `location`, `date`/timeframe, and any billing-code qualifiers — so
   the user can determine which questionnaires actually apply.
4. The DTR client **SHALL** allow the user to (a) complete every
   potentially-applicable questionnaire, (b) complete only the
   questionnaires they confirm apply, or (c) defer completion until the
   order is further specified.
5. The DTR client **SHALL NOT** modify or remove `coverage-information`
   repetitions on the underlying Request resource as a precondition to
   launch (this is consistent with Lloyd's caution that suppressed
   repetitions may still be needed if the ordering provider revisits
   the order).
6. Add a short cross-reference paragraph in the CRD IG's "General
   Strategies" section noting that when CRD returns multiple
   `coverage-information` extensions with `doc-needed`, DTR client
   behaviour is specified in the corresponding DTR IG section, and
   restating that EHRs should not collapse repetitions when surfacing
   the order to DTR.

#### Justification

This is the disposition Lloyd McKenzie effectively asked for in the
originating Zulip thread, and it is consistent with how DTR already
treats the `coverage-information` extension as the authoritative carrier
of payer rule context (qualifiers, doc-needed, satisfied-pa-id). Putting
the responsibility on the DTR client — rather than on the EHR's pre-launch
data shaping or on a new DTR-only Request profile — preserves the
information needed if the ordering provider later revisits the order,
matches how CRD already permits multi-valued responses, and aligns with
the broader Da Vinci pattern that user-facing decision support tooling
(not the EHR transport layer) is responsible for explaining payer
ambiguity to clinicians. The accompanying CRD-side cross-reference keeps
implementers from having to discover the rule by reading both IGs in
isolation.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of placing all of the burden on the DTR client at runtime, split
the response into two coordinated changes:

1. **CRD-side disambiguation (preferred carrier of meaning).** Update
   the CRD IG to require that when a server returns multiple
   `coverage-information` extensions with `doc-needed` on the same order,
   each repetition **SHALL** populate at least one machine-readable
   qualifier (e.g., `performingProvider`, `location`, `billingCode`,
   `date` / coverage period, or a documented "default / catch-all"
   marker), and **SHALL** carry a short human-readable label (extension
   or coding `display`) describing the condition under which that
   repetition's `questionnaire` applies.
2. **DTR-side rendering rules (lighter than Disposition A).** Update
   the DTR IG to say that when launched with such an order, the DTR
   client **SHALL** render the qualifier(s) and label for each candidate
   `questionnaire` and let the user select which to fill out, but does
   **not** prescribe a particular UX; in particular, it does not require
   the "fill all / fill some / defer" tri-state, leaving that to
   implementations.

#### Justification

The root cause of the ambiguity is on the CRD side — the spec currently
allows multiple `coverage-information` repetitions without requiring
machine-readable disambiguators that a generic DTR client could rely on.
Pushing the requirement upstream into CRD makes every DTR client's job
mechanical (display qualifier + label, let the user pick), rather than
forcing each DTR vendor to invent heuristics for inferring conditions
from sparse extensions. This is consistent with the use cases Lloyd
described in the "Use cases for multiple coverage-information
extensions" thread (different billing codes, different date ranges,
inpatient vs. outpatient): all of those are payer-side facts that the
payer already knows when emitting the response. The trade-off vs.
Disposition A is that it requires CRD-side normative tightening (which
some payer implementers may push back on as a breaking change for
servers already returning bare repetitions), but it gives DTR clients
a much simpler conformance target and reduces inter-implementation
variation in how clinicians experience the multi-questionnaire case.

---

### Disposition C: Decline

#### Proposal

Decline as out-of-scope for the current DTR ballot cycle and close as
"covered by existing guidance + Light DTR EHR responsibility". Rationale:
(a) the CRD IG already permits multiple `coverage-information`
extensions and the DTR IG already requires the DTR client to honour
`doc-needed`; (b) when the EHR is acting as a Light DTR client, it
already controls the launch UX and can decide locally which
questionnaire(s) to surface; (c) for Full DTR, the SMART app is free
to render multiple questionnaires from its launch context using
existing Questionnaire selection mechanisms, and prescribing a specific
UX flow risks over-constraining vendor implementations. If implementers
hit interoperability problems in the field, revisit with concrete
failure reports.

#### Justification

A defensible "decline" rests on the argument that no spec text today
*forbids* a DTR client from doing what the ticket asks for — it just
doesn't *require* it — and that the workgroup has historically left
DTR client UX largely unspecified. Adding new SHALL-level conformance
without field evidence of divergent implementations risks codifying
one vendor's UX as normative. However, this position is weakened by
the Zulip thread itself: a real implementer (Dan Murphy) is asking
the question, and the workgroup's most senior voice (Lloyd McKenzie)
explicitly invited the change request, indicating consensus that some
guidance is warranted.

---

### Recommendation

**Recommended disposition:** A

Disposition A directly answers the question Dan Murphy raised and
codifies the behaviour Lloyd McKenzie articulated on Zulip without any
participant pushing back, which is the closest thing to existing
work-group consensus on this issue. It is also additive: it tightens
DTR client expectations without breaking any current CRD server
behaviour, and it leaves the existing CRD `coverage-information`
cardinality and content untouched (so payer-side servers already in
production keep working). Disposition B is conceptually cleaner and
worth raising in committee discussion, but it requires CRD-side
normative changes that may surface as breaking for payer implementers
and would extend the scope to two IGs and (likely) two ballot cycles;
the work group can always layer the CRD-side qualifier requirements on
top of Disposition A in a follow-up if field experience shows DTR
clients can't reliably interpret bare repetitions. Disposition C is
not recommended: the originating Zulip discussion already establishes
that the current spec text leaves a real implementer guessing, and
declining would push the same question back at the next implementer
who hits the same scenario.
