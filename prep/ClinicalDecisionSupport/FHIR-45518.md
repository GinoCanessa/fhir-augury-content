## Summary

Bryn Rhodes reports that two example libraries in the Using CQL With FHIR IG —
`Library-Example` and `Library-module-example` (the module-definition example
that exposes `dataRequirements`) — naturally reference profiles and value sets
from IPS and QI-Core. Because those references exist only inside *examples*
(no conformance artifact in the CQL IG depends on them), the IG does not
declare IPS or QI-Core as dependencies; instead, the example payloads have
been rewritten to use `http://example.org/...` URLs in place of the real
`http://hl7.org/...` URLs so the publisher does not raise unresolved-URL QA
errors.

The ticket asks for two things, which are intentionally split across
specifications:

1. **In the CQL IG itself (this ticket / CDS work group):** once 1.0.0 is
   published, restore the real IPS/QI-Core URLs in those example libraries,
   declare the IPS/QI-Core dependencies, and document each one with an
   `ImplementationGuide.dependsOn.reason` explaining that the dependency
   exists only to support examples.
2. **As a companion request (filed by the reporter as
   [FHIR-45519](https://jira.hl7.org/browse/FHIR-45519), FHIR Infrastructure /
   Tooling Extensions):** define a computable extension on
   `ImplementationGuide.dependsOn` (and a corresponding `devDependencies` key
   in `package.json`) so a dependency can be flagged as an "example" /
   dev-time-only dependency, allowing downstream IGs and runtime tooling to
   skip it.

FHIR-45519 has already been resolved as **Persuasive with Modification**
("dev dependency" extension + `package.json` `devDependencies`), so the
remaining open work belongs to CDS on FHIR-45518: roll back the
`example.org` workaround in the CQL IG examples, restore real IPS/QI-Core
references, declare those dependencies (using the new dev/example marker once
the Tooling Extensions release exposes it), and populate `dependsOn.reason`.

## Related Zulip Discussions

### [fmg > Examples using undeclared dependencies](https://chat.fhir.org/#narrow/stream/fmg/topic/Examples%20using%20undeclared%20dependencies/near/437048796)

Bryn Rhodes opened the thread describing the exact CQL IG situation: example
resources reference IPS / QI-Core profiles, producing unresolved-reference
QA errors that FMG was generally willing to accept. Grahame Grieve first
suggested `ig-link-dependency`, but agreed it only covers HTML/markdown
links and not profile references, then pressed on *why* the dependency
should not just be declared. Bryn argued declaring would push unnecessary
~1.3 MB downloads onto downstream IGs and packaged deployments (notably
Android), and Brian Postlethwaite, Ward Weistra, and Bart Decuypere (with a
similar Belgian use case) supported the idea of a "dev"/"example"
dependency category. Josh Mandel framed it as: publishing examples should
not force a real dependency. Grahame's eventual landing position was that
if Bryn really wants this distinction, he should use
`ImplementationGuide.dependsOn.reason` and that an extension could be
defined to make "validation-only / example-only dependency" computable —
which is exactly what FHIR-45518 asks the CQL IG to adopt and what
FHIR-45519 was filed to standardize. Elliot Silver and John Moehrke also
flagged that adding a normal dependency drags in must-support and
obligation semantics, reinforcing that an "example-only" marker is needed.

Bryn Rhodes, Grahame Grieve, Ward Weistra, Josh Mandel, Brian Postlethwaite,
Bart Decuypere, John Moehrke, Elliot Silver.

### [tooling > Dev Dependencies](https://chat.fhir.org/#narrow/stream/tooling/topic/Dev%20Dependencies/near/440359054)

Josh Mandel cross-posted the proposed disposition for FHIR-45519 to the
tooling stream: model "dev dependencies" both as an extension on
`ImplementationGuide.dependsOn` (boolean `Extension-devDependency`) and as
a `devDependencies` key in `package.json`. Grahame Grieve pushed back on
naming and definition — npm's "devDependencies" means "needed for local
development and testing", whereas these are "needed only to validate
examples and not needed by implementers consuming the profiles" — and
worried implementers would not know whether to load them. Chris Moesel
reframed examples themselves as primarily a development/testing aid, so
calling their supporting packages `devDependencies` is defensible; if
someone integrates examples into production they can rescope the
dependency. Bryn pointed at concrete CQL IG content
(`input/examples/Library-moduledefinition-example.json`) referencing QI-Core
profiles and US Core extensions. Paul Denning added a DEQM use case where
"dev dependencies" would let DEQM keep using QI-Core Patient examples while
relaxing its conformance constraints, decoupling QI-Core / DEQM / QM IG
publishing. No firm objection emerged; the conversation broadly converged
on the dev-dependency mechanism, with Grahame's terminology concern noted
but not blocking.

Josh Mandel, Grahame Grieve, Bryn Rhodes, Chris Moesel, Paul Denning.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the change in the Using CQL With FHIR IG (CDS work group's
spec, `Using CQL With FHIR (FHIR)`) exactly as the ticket asks:

1. In the example libraries that currently use `http://example.org/...`
   placeholders — `Library-Example` and `Library-module-example`
   (the module-definition example with `dataRequirements`) — restore the
   real IPS and QI-Core canonical URLs for the referenced profiles and
   value sets.
2. In the IG definition (`ImplementationGuide` / sushi-config),
   re-declare IPS and QI-Core under `dependsOn`, and on each of those
   `dependsOn` elements set `dependsOn.reason` to a clear human-readable
   note such as *"Example-only dependency: needed solely to validate
   example resources that illustrate use of QI-Core / IPS profiles. Not
   required to implement the conformance content of this IG."*
3. Once the FHIR-45519 deliverable ships in the Tooling Extensions release
   (the new dev/example-dependency extension on
   `ImplementationGuide.dependsOn`, plus the `devDependencies` slot in
   `package.json`), apply that extension to the IPS and QI-Core entries and
   mirror the entries into `package.json.devDependencies` so downstream
   consumers and the IG publisher can skip them at runtime.
4. Defer execution until after CQL IG 1.0.0 publishes (per the ticket text
   and Bryn's commitment in the fmg Zulip thread).

#### Justification

This is the plan the reporter explicitly proposed, and FMG already
accepted the example.org workaround as a temporary measure until 1.0.0
shipped. The companion ticket FHIR-45519 has been resolved Persuasive
with Modification, so the dev/example-dependency mechanism the CQL IG
needs will exist; consuming it from the CQL IG is the natural follow-up.
Using the real canonicals also resolves the concerns Josh Mandel and
Bart Decuypere raised about realistic examples being penalized, and the
`dependsOn.reason` text directly addresses Grahame Grieve's concern that
implementers should be able to tell *why* a dependency is present and
whether they need to load it. It is also consistent with how DEQM (per
Paul Denning) wants to operate.

---

### Disposition B: Alternative Approach

#### Proposal

Achieve the same end state without inventing or relying on a new
"example dependency" marker, by using the **existing**
`http://hl7.org/fhir/tools/StructureDefinition/ig-internal-dependency`
extension that Grahame Grieve points to in the FHIR-45519 comment thread:

1. In the CQL IG, restore the real IPS / QI-Core URLs in
   `Library-Example` and `Library-module-example`.
2. Declare IPS and QI-Core under `ImplementationGuide.dependsOn`, populate
   `dependsOn.reason` with the example-only rationale, and apply the
   `ig-internal-dependency` tooling extension to those `dependsOn`
   entries.
3. Leave it to FHIR Infrastructure / FHIR-45519 implementers to decide
   whether `ig-internal-dependency` is renamed/clarified or whether a new
   `devDependency` extension supersedes it; the CQL IG only commits to
   the *semantic* (dependency exists for examples only) and migrates to
   whichever extension URL the Tooling Extensions release ultimately
   publishes.
4. Skip the `package.json` `devDependencies` change in this ticket, since
   that is a publisher/tooling concern owned by FHIR-45519 and the IG
   Publisher, not by the CQL IG content.

#### Justification

If the existing tooling extension already covers this case, defining and
adopting a parallel marker is wasted churn; this disposition lets CDS
unblock the CQL IG immediately without waiting on the final shape of the
FHIR-45519 deliverable. It is also lower risk: the CQL IG only commits
to the declarative semantics (real URLs + reason + a tooling extension
flag), and any later migration is a mechanical URL swap. The downside,
relative to A, is that until FHIR-45519's chosen mechanism is adopted by
publishers/validators, the `ig-internal-dependency` extension may not
actually cause downstream tooling to skip the IPS/QI-Core packages,
preserving the download-size concern Bryn raised.

---

### Disposition C: Decline

#### Proposal

Decline to change the CQL IG. Keep the current `http://example.org/...`
placeholders in `Library-Example` and `Library-module-example`, do not
declare IPS or QI-Core as `dependsOn`, and do not introduce
`dependsOn.reason` text or any tooling-extension flag for example-only
dependencies. Track the underlying mechanism question solely under
FHIR-45519 in FHIR Infrastructure, and leave the CQL IG examples in
their current "pseudo-canonical" form indefinitely.

#### Justification

A decline is defensible only on the narrow argument that FMG already
accepted the `example.org` workaround as a clean publication outcome and
that the broader "how do we represent example-only dependencies"
question belongs to FHIR Infrastructure (FHIR-45519), not CDS. It also
avoids re-litigating Grahame Grieve's position in the fmg thread — that
referencing a profile *is* a real dependency and ought to be declared
plainly. The cost is real: the example libraries no longer faithfully
illustrate use against IPS/QI-Core (Josh Mandel, Bart Decuypere, and
Paul Denning have all flagged this kind of "examples should reflect
reality" concern), and the workaround silently masks the very pattern
the CQL IG is trying to teach.

---

### Recommendation

**Recommended disposition:** A

FHIR-45519 has already landed Persuasive with Modification, with both an
extension on `ImplementationGuide.dependsOn` and a `package.json`
`devDependencies` mechanism on the table, and it is exactly the
infrastructure piece the reporter said this ticket was waiting on. CDS
should now do the corresponding CQL IG content work: restore the real IPS
and QI-Core canonicals in `Library-Example` and `Library-module-example`,
declare the dependencies, populate `dependsOn.reason`, and adopt the
new dev/example-dependency marker as soon as the Tooling Extensions
release exposes it. This honors the reporter's plan, gives implementers
faithful examples (the explicit ask from Josh Mandel, Bart Decuypere, and
Paul Denning), and gives Grahame Grieve's concern about implementer
clarity a concrete answer via the `reason` text and the computable flag.
Disposition B is the right fallback only if FHIR-45519's chosen mechanism
slips beyond the CQL IG's next publication window — in which case use
`ig-internal-dependency` now and migrate later. Disposition C should be
rejected because it locks in the `example.org` workaround the work group
already characterized as temporary and leaves the example libraries
permanently misleading.
