## Summary

The CDS Hooks specification defines `Card.detail` as "(GitHub Flavored)
Markdown" but does not say whether inline HTML is permitted inside that
markdown. The FHIR core `markdown` datatype is also based on GFM/CommonMark
but explicitly disallows inline HTML — processors SHALL treat embedded
XML/HTML tags as string content. Because CDS Hooks references GFM directly
rather than the FHIR `markdown` datatype, implementers are left without a
clear answer: some are emitting and rendering HTML (e.g., bulleted lists
inside guideline recommendations) on the assumption that GFM's permissive
HTML behaviour applies, while others may be sanitising it away. The
reporter asks that the CDS Hooks specification state explicitly whether
HTML is allowed in `Card.detail` (and, by implication, in any other CDS
Hooks markdown-typed field) so that producers and CDS Clients converge on
a single, interoperable behaviour.

There are no cross-referenced Jira tickets — only a Zulip thread that
prompted the issue (see below).

## Details

**Description:**

> The FHIR markdown data type is based on GitHub Flavored Markdown, but doesn't allow HTML.
>
> The description of the card detail field in the CDS Hooks specification states that the field is GitHub Flavored Markdown. It doesn't refer to the FHIR markdown data type.
>
> Because of this difference, the CDS Hooks specification should state explicitly if HTML is allowed in card details.

**Comments:**

- **JIRAUSER24667 — 2026-02-04:** Quotes the relevant rules from the FHIR
  core specification's `markdown` datatype:
  - Markdown is a string subject to the same rules (length, valid characters).
  - The specification requires GFM extensions on CommonMark format **with
    the exception of support for inline HTML, which is not supported.**
  - Processors SHALL treat embedded XML tags as string content (e.g., by
    escaping `>` to `\>` before processing). HTML cannot be embedded in
    FHIR markdown to influence rendering.

- **isaac.vetter — 2026-02-04:** Adds background by quoting the current
  CDS Hooks `Card.detail` definition: *"Optional detailed information to
  display; if provided MUST be represented in (GitHub Flavored) Markdown.
  (For non-urgent cards, the CDS Client MAY hide these details until the
  user clicks a link like 'view more details…')."* Notes the link to
  `#card-attributes` and includes the FHIR `markdown` datatype tables
  from R5 (build) and R4, both of which describe `markdown` as a FHIR
  `string` carrying GFM-on-CommonMark content (with the FHIR-level "no
  inline HTML" rule applying in both releases). Highlights that CDS
  Hooks's reference to GFM is currently bare — there is no link back to
  the FHIR `markdown` datatype rules.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an explicit statement in the CDS Hooks "Card Attributes" section
(`#card-attributes`) — wherever `Card.detail` (and any other
markdown-typed field, e.g., `Card.summary`-related text and
`Card.source.description`) is defined — that says inline HTML **is not
allowed** inside the GFM content, mirroring the FHIR `markdown` datatype
rule. Concretely:

- Update the prose definition of `Card.detail` to read approximately:
  *"Optional detailed information to display; if provided MUST be
  represented in (GitHub Flavored) Markdown. As with the FHIR `markdown`
  datatype, inline HTML is not supported: CDS Clients SHALL treat
  embedded HTML/XML tags as literal string content and MUST NOT render
  them as markup."*
- Add a normative bullet (SHALL) to the same section: *"CDS Services
  SHALL NOT rely on inline HTML being rendered. CDS Clients SHALL
  sanitise or escape any HTML/XML tags found inside markdown-typed
  fields before display."*
- Add a forward link from the CDS Hooks markdown wording to the FHIR
  `markdown` datatype page (`https://hl7.org/fhir/datatypes.html#markdown`)
  so the rules cannot drift in the future.

#### Justification

This is the most defensible default for a healthcare interoperability
specification. The FHIR core already prohibits inline HTML in `markdown`
and the documented reason — preventing CDS Clients from being forced to
implement a full HTML sanitiser, and avoiding the XSS / cross-origin
content risks that arise when a CDS Service can inject arbitrary HTML
into a clinician-facing UI — applies at least as strongly inside CDS
Hooks, where the CDS Service is often a third party. Aligning CDS Hooks
with the FHIR `markdown` datatype keeps the two specifications mutually
consistent (called out by the JIRAUSER24667 and isaac.vetter comments)
and gives implementers a single, well-known rule to follow. The Zulip
evidence that some implementers are already emitting HTML is a reason
to clarify *now*, before the divergent behaviour becomes entrenched.

---

### Disposition B: Alternative Approach

#### Proposal

Allow a constrained subset of inline HTML in `Card.detail`, explicitly
enumerated in the spec, and require CDS Clients to sanitise everything
else. Suggested approach:

- State that `Card.detail` is GFM and that CDS Services MAY include a
  small, explicitly-listed safe subset of HTML inline tags (e.g., `<br>`,
  `<b>`, `<i>`, `<em>`, `<strong>`, `<u>`, `<ul>`, `<ol>`, `<li>`,
  `<p>`, `<a href>`), with attribute restrictions (`href` limited to
  `http(s):` / `mailto:` schemes; no `style`, `script`, `on*`,
  `iframe`, `object`, `embed`, `form`, `img`, etc.).
- Require CDS Clients to apply an allow-list sanitiser conforming to
  that subset; tags or attributes outside the allow-list SHALL be
  stripped or escaped.
- Add a security-considerations paragraph spelling out the XSS surface
  and the responsibility split between CDS Service and CDS Client.
- Note explicitly that this is an intentional divergence from the FHIR
  core `markdown` datatype, justified by CDS Hooks's UI-rendering
  context (where CDS Services need richer formatting than plain GFM
  delivers — e.g., bulleted clinical-guideline summaries, as Mustafa
  Yuksel described).

#### Justification

This approach matches what real-world CDS Hooks implementations are
already doing (per the Zulip thread, both Ádám Z. Kövér and Mustafa
Yuksel emit HTML in `Card.detail`) and what GFM itself permits when it
is parsed by browser-grade renderers. Plain GFM is genuinely thin for
clinical-guideline content — line breaks and rich list nesting are the
common pain points — and forcing implementers to fall back on plain
markdown may simply push them to break the rule quietly. The cost is
higher than Disposition A: every CDS Client now needs an allow-list
HTML sanitiser, the spec carries a security-considerations burden, and
CDS Hooks formally diverges from the FHIR `markdown` datatype, which
weakens the "FHIR-family of specifications" story. A note in the spec
that this is an intentional, scoped exception is necessary to manage
that divergence.

---

### Disposition C: Decline

#### Proposal

Decline the request as already addressed. Resolve as "Persuasive with
Modifier — Not Persuasive" or "Already Specified", with the explanation
that CDS Hooks's reference to GFM (which itself follows CommonMark) is
sufficient: CommonMark/GFM defines exactly how raw HTML inside markdown
is to be parsed, and CDS Hooks does not need to restate it. Implementers
who want consistency with FHIR `markdown` are free to apply that
stricter rule in their own deployments, but the base CDS Hooks
specification should remain neutral.

#### Justification

The CDS Hooks `Card.detail` definition does point at the GFM spec, and
GFM/CommonMark *does* specify HTML handling — so technically the
behaviour is defined. Declining keeps the spec small and avoids a
normative change late in the cycle. However, this position is hard to
defend in light of the evidence: the entire reason the ticket exists is
that two independent implementers (Ádám Z. Kövér, Mustafa Yuksel)
arrived at the same uncertain reading of the spec, and the FHIR family
disagrees with CommonMark on this exact point. Declining leaves the
inconsistency with the FHIR `markdown` datatype unresolved and tells
the CDS Client community that the security/sanitisation question is
their problem, which is not where CDS Hooks should land.

---

### Recommendation

**Recommended disposition:** A

Disposition A is recommended. The ticket is fundamentally a request for
clarity, and the "no inline HTML" rule already in the FHIR `markdown`
datatype is the only answer that is (a) consistent with the rest of
FHIR, (b) safe by default for CDS Clients (no mandatory HTML sanitiser,
no XSS surface from third-party CDS Services), and (c) cheap to
implement — escape-on-render is trivial compared to maintaining an
allow-list sanitiser. Disposition B reflects what some implementers are
doing today and has real ergonomic merit for clinical-guideline content,
but it imposes a non-trivial security burden on every CDS Client and
formally splits CDS Hooks from FHIR core on a datatype interpretation;
that is a much bigger commitment than this ticket alone warrants. If
the workgroup feels the GFM-only restriction is too thin in practice,
the right next step is a follow-up ticket scoping out a richer
"renderable text" mechanism for cards (e.g., a structured
content-block list) rather than smuggling raw HTML through `Card.detail`.
Disposition C is not recommended — the Zulip thread shows the
specification text is genuinely ambiguous to careful readers, and
declining would entrench that ambiguity.
