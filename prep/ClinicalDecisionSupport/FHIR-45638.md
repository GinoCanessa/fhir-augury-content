## Summary

The CRMI (Canonical Resource Management Infrastructure) IG is published
against FHIR R4, but its shareable profiles want to slice on the
`cqf-knowledgeRepresentationLevel` extension, whose binding ultimately
points at the `KnowledgeRepresentationLevel` CodeSystem/ValueSet that
only exists in R5. From an R4 IG, that becomes a broken cross-version
reference for profile rendering, validation, and downstream tooling.
For ballot, CRMI worked around the issue by inlining a copy of the R5
code system and value set into the IG; for publication, the
`knowledgeRepresentationLevel` slice was instead removed from every
shareable profile until the publication and dependency chain can
properly handle the cross-version reference. The reporter (Bryn Rhodes)
proposes either (a) waiting for cross-version terminology support so
the slice can be restored as-is, or (b) relocating the
`KnowledgeRepresentationLevel` terminology into the FHIR extensions
pack — mirroring how `knowledgeCapability` is already handled there —
so that R4 IGs can reference it natively. The committee resolved the
ticket as **Persuasive with Modification**, picking the second option:
move the `knowledgeRepresentationLevel` CodeSystem and ValueSet into
the extensions pack and let CRMI re-add the slice against that
canonical home.

There are no cross-referenced Jira tickets; the only related external
discussion is the FMG Zulip thread "Using extensions in R4 with R5
terminology?" where this exact problem was first raised.

## Details

**Description:**

The Knowledge Representation Level code system is defined in the FHIR
R5 specification, but the `cqf-knowledgeRepresentationLevel` extension
references that code system (through the whole-codesystem value set).
When this extension is used by profiles in an R4 IG, this results in a
reference to the R5 value set, and is basically a broken link from the
profile/validation/tooling perspective.

For ballot, we copied the definition of this value set and code system
into the CRMI specification, but for publication, we removed the
`knowledgeRepresentationLevel` slice from all shareable profiles as a
work-around until the publication and dependency chain can support the
cross-version reference to the R5 terminology.

Alternatively, we could move the `knowledgeRepresentationLevel`
terminology into the extensions pack (as is the case with the
`knowledgeCapability` code system and value set).

**Resolution Description:**

Move the `knowledgeRepresentationLevel` terminology into the extensions
pack so that it can be referenced there, rather than in the base
specification (same approach as `knowledgeCapability`).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restore the `knowledgeRepresentationLevel` slice on every CRMI
shareable profile (`crmi-shareablemeasure`,
`crmi-shareablelibrary`, `crmi-shareableplandefinition`,
`crmi-shareableactivitydefinition`, `crmi-shareablequestionnaire`,
`crmi-shareablevalueset`, `crmi-shareablecodesystem`, etc.) exactly as
it was at ballot, slicing `extension` on
`http://hl7.org/fhir/StructureDefinition/cqf-knowledgeRepresentationLevel`
and binding to the R5
`http://hl7.org/fhir/ValueSet/knowledge-representation-level` value
set. Keep the duplicated copy of the `KnowledgeRepresentationLevel`
CodeSystem and ValueSet inside the CRMI IG (as was done for ballot) so
the cross-version dependency resolves locally during publication, and
add a publisher note that the duplicate will be retired once R4 IGs
can natively reference R5 terminology.

#### Justification

This is what the reporter literally asks for as the first option, and
it is the cleanest from a CRMI authoring perspective: the slice is
already designed and tested, the duplicate terminology is already in
the ballot artefacts, and no other IG has to change. It is also
consistent with a long-standing FHIR pattern of inlining the few R5
artefacts an R4 IG needs while waiting for proper cross-version
support. However, Grahame Grieve's response on the FMG thread is
explicit that there is no short-term cross-version fix coming, so this
disposition effectively bakes the duplicate into the published CRMI
spec for the foreseeable future, which the FMG conversation already
flagged as "wrong for publication."

---

### Disposition B: Alternative Approach (Persuasive with Modification — adopted)

#### Proposal

Move the `KnowledgeRepresentationLevel` CodeSystem
(`http://terminology.hl7.org/CodeSystem/knowledge-representation-level`
or its equivalent canonical) and its companion whole-codesystem
ValueSet from the base FHIR R5 specification into the **FHIR
extensions pack**, alongside the `cqf-knowledgeRepresentationLevel`
extension that already lives there — exactly mirroring how
`knowledgeCapability` is structured today. Once the terminology is
republished from the extensions pack, restore the
`knowledgeRepresentationLevel` slice on the CRMI shareable profiles
listed in Disposition A, binding to the new extensions-pack canonical.
Drop the temporary duplicate CodeSystem/ValueSet from the CRMI IG and
add a release note pointing implementers at the extensions-pack
canonical. Coordinate the move with the extensions-pack maintainers so
that the R5 base specification redirects/retains the existing
canonical for backward compatibility.

#### Justification

This is the resolution actually recorded on the ticket ("Persuasive
with Modification") and the precedent Bryn himself surfaced on the FMG
thread: `knowledgeCapability` already lives in the extensions pack and
"works fine" from R4 IGs. Hosting the terminology in the
version-neutral extensions pack removes the cross-version dependency
entirely — every IG that uses `cqf-knowledgeRepresentationLevel`
(CRMI today, others tomorrow) can resolve the binding without
duplicating artefacts or waiting on cross-version tooling. The main
trade-offs are (i) coordinating the canonical move with the
extensions-pack and base R5 publication so existing references do not
break, and (ii) ensuring the R5 spec text that currently introduces
the code system either redirects to the extensions-pack home or is
removed; both are addressable through the normal extensions-pack
release process and align with how the related artefact already ships.

---

### Disposition C: Decline

#### Proposal

Leave the publication state of CRMI as-is: the
`knowledgeRepresentationLevel` slice stays removed from all shareable
profiles, the duplicate terminology in the CRMI IG is dropped, and
implementers are directed to the `cqf-knowledgeRepresentationLevel`
extension and R5 terminology directly if they need it. Re-open the
issue only when a general cross-version terminology mechanism (e.g.,
the work Grahame referenced on the FMG thread) is available.

#### Justification

This is defensible only on the grounds that no movement of artefacts
between specifications is risk-free, and that "no slice" is less
broken than "slice that points at an unresolvable canonical." But it
loses the conformance value the slice was added for in the first place
(making knowledge representation level a first-class, validatable
expectation on shareable artefacts), it leaves the `cqf-` extension
under-specified for the entire R4 ecosystem, and it ignores that the
exact pattern requested already works for `knowledgeCapability`. It
should only be selected if the workgroup decides moving terminology
between the base spec and the extensions pack is itself out of scope
for CDS/CRMI.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the approach the workgroup already recorded as the
resolution, and it is the only option that resolves the underlying
problem rather than papering over it. The FMG Zulip thread establishes
both the constraint (no near-term cross-version terminology mechanism)
and the precedent (`knowledgeCapability` lives in the extensions pack
and works), and FHIR-45638's reporter explicitly offers this as the
preferred alternative. It restores the conformance expectation on CRMI
shareable profiles, lets every other R4 IG that uses
`cqf-knowledgeRepresentationLevel` resolve cleanly, and removes the
"duplicate terminology in CRMI" workaround that the FMG thread agreed
should not survive publication. The coordination cost (moving the
canonical between the base R5 spec and the extensions pack, and
re-binding the CRMI slices) is real but bounded and follows an
existing template.
