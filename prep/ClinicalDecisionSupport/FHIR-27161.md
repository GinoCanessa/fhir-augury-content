## Summary

This Change Request asks the Clinical Decision Support work group to extend
`CPGWorkflowDefinition` (a `PlanDefinition` profile in the Clinical
Practice Guidelines (CPG) IG) so that authors can model **data flow
between actions** within a workflow, not just control flow. The reporter,
Bryn Rhodes, proposes introducing extensions on the `input` element (the
ticket refers to it as a "custom type" wrapping `DataRequirement`) that
add two new pieces of structure:

1. An **`id` on the input** (e.g., `data_2`) so that a specific input of
   an action can be uniquely referenced.
2. A **`relatedRequirement`** child carrying a `requirementId` (e.g.,
   `data_1`) so that one action's input can be declared to depend on
   another action's input/output by id.

The illustrative example pairs a `QuestionnaireResponse`-typed
`dataRequirement` (filtered to a CPG common-identity Questionnaire) with
a `relatedRequirement` pointing at `data_1`, expressing "this input is
satisfied by/derives from the input identified as `data_1`." Together
these allow CPG workflow definitions to express a directed data graph
across actions, similar in spirit to BPMN data associations.

The ticket carries the `Block-Vote-1` label and is currently `Triaged`.
Per the comments, CDS originally voted **Considered for Future Use**
(20-0-0, 2024-07-10) on the basis that "more work is needed to make this
actionable," and Lynn Laakso then reverted that resolution on
2024-11-26 to **re-open post STU2 publication** of the CPG IG. There
are no cross-referenced Jira tickets, Zulip discussions, or GitHub
items associated with this ticket in the augury index.

### Disposition A: Accept As Requested

#### Proposal

Define two new standard extensions in the CPG IG and apply them in the
`CPGWorkflowDefinition` profile (the CPG profile of `PlanDefinition`)
on `PlanDefinition.action.input` (and, symmetrically, on
`PlanDefinition.action.output`):

1. **`cpg-inputId`** (and `cpg-outputId`) â€” `id` (`string`/`id`-typed)
   extension, cardinality `0..1`, that gives the input/output a workflow-
   local identifier (e.g., `data_2`) usable from elsewhere in the same
   `CPGWorkflowDefinition` instance.
2. **`cpg-relatedRequirement`** â€” `complex` extension with one nested
   `requirementId` sub-extension (`id`/`string`, `1..1`), cardinality
   `0..*`, declaring that this input depends on (is satisfied by /
   derived from) another action's input or output identified by its
   `cpg-inputId` / `cpg-outputId`.

In the IG narrative, describe the resulting data-flow graph semantics
(directed edges from `relatedRequirement.requirementId` â†’ the input/
output that bears that id), uniqueness/scoping rules for the id
(unique within a single `CPGWorkflowDefinition` instance), and provide
the QuestionnaireResponse / `cpg-common-identity` example from the
ticket as an example. Add invariants that `requirementId` MUST resolve
to an `id` defined elsewhere in the same definition.

#### Justification

The request is concrete, has an explicit example, and fills a real
expressivity gap: today a `CPGWorkflowDefinition` can describe what
inputs an action needs (`PlanDefinition.action.input.dataRequirement`)
and what triggers/precedes an action (`relatedAction`), but it cannot
say "this input is the same data item as that input/output of another
action." Modeling data flow as an extension on `input`/`output`
preserves backward compatibility (extensions are optional), avoids
disturbing the base `PlanDefinition` model, and is consistent with how
CPG already augments `PlanDefinition` (e.g., `cpg-knowledgeCapability`,
`cpg-knowledgeRepresentationLevel`). The pattern of a stable local id
plus an id-reference for relationships matches `relatedAction.actionId`
already in `PlanDefinition`, so implementers will recognise it.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing two new extensions on `input`/`output`, reuse
and generalise the existing `PlanDefinition.action.relatedAction`
machinery for data flow:

1. Confirm that input/output identification can be handled by an
   existing or lightweight extension that re-purposes
   `DataRequirement.id` (the standard `Element.id`) as the referenced
   token, eliminating the need for a new `cpg-inputId` extension.
2. Define **one** complex extension `cpg-dataDependency` on
   `PlanDefinition.action` (not on `input`) with sub-extensions
   `source` (action id + input/output id) and `target` (input/output
   id on this action). This expresses data flow at the action level,
   parallel to how `relatedAction` expresses control flow.
3. Provide an IG-level narrative + worked CPG example showing both
   styles (control flow via `relatedAction`, data flow via
   `cpg-dataDependency`).

#### Justification

The 2024-07-10 vote ("more work is needed to make this actionable")
suggests the original proposal needs sharper semantics before being
locked into the standard. Putting the dependency declaration on
`action` rather than on each `input` makes the data-flow graph easier
to enumerate, matches the existing `relatedAction` pattern (which is
also on `action`, not on a sub-element), and avoids inventing two
extensions when one will do. Using `Element.id` instead of a new
`cpg-inputId` extension reduces extension surface area and leverages
machinery FHIR already requires implementers to support. The trade-off
is that the syntax diverges slightly from the example in the ticket,
but the expressive power and the reviewable graph are equivalent.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the CPG IG at this time and
direct the use case to either (a) a downstream profile/IG that
specifically targets executable workflow orchestration, or (b) a
companion artefact (e.g., a CQL library or a `GraphDefinition`-style
companion) that captures inter-action data wiring outside of
`PlanDefinition`. Recommend closing as **Persuasive with Mod /
Considered for Future Use** depending on whether the work group wants
to keep it on the backlog.

#### Justification

`PlanDefinition` is intentionally a *declarative* description of a
plan, not an executable workflow language; full data-flow plumbing
between actions blurs that boundary toward BPMN/orchestration
territory and risks duplicating existing FHIR mechanisms (CQL
expressions can reference any prior data; `GuidanceResponse` /
`Task` chains carry runtime data flow). The 2024-07-10 vote already
recorded that the proposal is not yet actionable, and the ticket has
been open since 2020 with no cross-referenced Zulip discussion,
linked tickets, or GitHub work to demonstrate implementation demand.
Without concrete consumers, adding two new normative-track extensions
risks publishing untested machinery.

---

### Recommendation

**Recommended disposition:** A

The ticket reporter (Bryn Rhodes, the CPG IG lead) has provided a
specific, syntactically well-formed example that maps cleanly onto
extensions, and Lynn Laakso explicitly re-opened the issue on
2024-11-26 to be pursued post STU2 publication â€” i.e., the work group
has already signalled intent to do this work, just not in the STU2
window. Disposition A respects that signal, ships exactly what the
reporter described, and uses the standard "stable local id +
id-reference" pattern that already exists for `relatedAction`, so it
will feel familiar to CPG implementers. Disposition B's instinct to
move the declaration up to `action` is worth raising in discussion as
a refinement of A (and may well be folded in during implementation),
but it should not block accepting the underlying request. Disposition
C is hard to justify given that the reporter is the IG owner and the
work group has explicitly chosen to keep the issue alive past STU2.
The work group should accept the request and resolve any A-vs-B
shaping question during the post-STU2 implementation cycle.
