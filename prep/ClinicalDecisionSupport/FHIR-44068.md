## Summary

FHIR-44068 asks the Clinical Guidelines (CPG) IG to provide first-class
support for **site-specific (client-specific) configuration of CDS Hooks
responses** that are produced by evaluating a `PlanDefinition`. The
requested mechanism has three coordinated parts:

1. A `cdsHooksExtension` extension carried on `RequestGroup` (the
   resource the CPG `$apply` operation typically returns) so that a
   guideline author can attach arbitrary extensions that should appear
   on the resulting CDS Hooks response card.
2. Authoring guidance in `PlanDefinition` so that `action.dynamicValue`
   can populate that `cdsHooksExtension` value at `$apply` time.
3. Site-specific override of those `dynamicValue` expressions through
   `moduleConfiguration`, so that a deploying client (e.g., a specific
   payer or provider organization) can change the resulting extension
   without forking the published `PlanDefinition`.

The ticket sits in the broader CDS configuration discussion that has run
through the CRD/CDS Hooks community for years (see Related Jira Tickets
below). It was originally dispositioned **Considered for Future Use**
(Greg White / Juliet K Rubini, 20-0-0, 2024-07-10) with the comment
"More input is needed, will pursue after STU2 publication", and then
**re-opened post-STU2 publication** by Lynn Laakso on 2024-11-26. It is
currently flagged `Ready-For-Vote` and `Block-Vote-1`, indicating the
work group is staging it for a fresh disposition vote.

The cross-reference index returned no Jira/Zulip/GitHub items directly
linked to FHIR-44068, and no Zulip thread mentions the key. The keyword

PlanDefinition) is summarized in **Related Jira Tickets**.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **Clinical Guidelines (CPG) IG**, define a complete,
three-layer site-specific response-configuration mechanism exactly as
the ticket describes:

1. **`cdsHooksExtension` extension on `RequestGroup`** — define a new
   IG-owned complex extension (e.g.,
   `http://hl7.org/fhir/uv/cpg/StructureDefinition/cpg-cdsHooksExtension`)
   that may appear 0..* times on `RequestGroup` and on each
   `RequestGroup.action`. Its value is a nested extension whose `url`
   and `value[x]` are copied verbatim onto the corresponding CDS Hooks
   response card / suggestion at `$apply` time. Specify the projection
   rules from `RequestGroup`/`action` to card/suggestion explicitly.
2. **Authoring in `PlanDefinition`** — document the pattern of using
   `PlanDefinition.action.dynamicValue` with a path targeting the new
   extension (e.g., `action.extension('…cpg-cdsHooksExtension').value`)
   so guideline authors populate the extension at `$apply` time from
   CQL or FHIRPath. Provide a worked CQL example.
3. **`moduleConfiguration` override** — extend the existing CPG
   `moduleConfiguration` mechanism so that a deployment can replace or
   add a `dynamicValue` entry that targets a `cdsHooksExtension` slot,
   without modifying the published `PlanDefinition`. Specify precedence
   (site override wins over module default) and validation rules
   (override expression must be type-compatible with the extension's
   `value[x]`).

Add a normative example PlanDefinition + ModuleConfiguration pair and a
resulting CDS Hooks card in the IG.

#### Justification

The proposal directly fills a gap that CRD has worked around for years
via `davinci-crd.configuration` (see FHIR-22159, FHIR-18947) and that
prior tickets (FHIR-36134) explicitly demanded worked examples for. It
re-uses three mechanisms the IG already publishes (`RequestGroup`
extensions, `PlanDefinition.action.dynamicValue`, `moduleConfiguration`)
rather than inventing parallel infrastructure, and it stays consistent
with FHIR-34265 which aligned `RequestGroup.action` with
`PlanDefinition.action`. The reporter is Bryn Rhodes (CPG IG editor),
which signals that the design has been thought through against the rest
of the IG's machinery. The original "Considered for Future Use" vote
was unanimous (20-0-0) only because the WG wanted to defer past STU2
publication, not because of a substantive objection — the re-open puts
the same proposal back in front of the WG with that publication now
behind it.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the **same intent** but split the work and align with CDS Hooks'
own configuration model rather than introducing a CPG-specific
extension:

1. Do **not** define a new `cpg-cdsHooksExtension`. Instead, document
   in the IG that any extension already present on `RequestGroup` /
   `RequestGroup.action` is projected onto the corresponding CDS Hooks
   card/suggestion `extension` array at `$apply` time. Specify the
   projection in one short normative section and an example.
2. For authoring, point at the existing `dynamicValue` mechanism on
   `PlanDefinition.action` plus the FHIRPath-into-extension idiom that
   is already used elsewhere in the IG; add a worked example only.
3. For site-specific override, **defer to CDS Hooks core's existing
   `configuration` mechanism on the discovery / hook request** (the
   same surface CRD already uses, FHIR-22159), and define a single CPG
   convention: a hook configuration option named (e.g.)
   `moduleConfiguration` whose value is a contained
   `Parameters`-shaped object enumerating
   `(planDefinitionCanonical, dynamicValue)` overrides. The CPG `$apply`
   implementation reads that configuration from the hook request and
   applies the overrides before evaluation.

This keeps the spec change small (one projection rule, one
configuration convention, examples) and leans on already-balloted
machinery on both sides.

#### Justification

Disposition A introduces a new typed extension and bespoke override
syntax that overlap with existing CDS Hooks configuration plumbing; the
critique in FHIR-36134 ("can we provide examples?") and FHIR-28513
("is configuration plumbing actually needed?") will land harder against
a mechanism that is *only* CPG-shaped. Disposition B reduces the IG's
new normative surface to (a) the projection rule from `RequestGroup`
extensions to card extensions and (b) one named hook-configuration
convention, keeping the design extensible without minting a new
StructureDefinition. It also keeps a single configuration story across
CRD-style and CPG-style implementers, which is the long-standing
direction of WG discussion in the related tickets. Trade-off: site
overrides must be supplied by the *invoking client* in the hook
request, not authored once at deploy time on the server — so it is
slightly less convenient for installations that prefer a server-side
overlay file, but it is more interoperable.

---

### Disposition C: Decline

#### Proposal

Resolve as **Considered - No action required**. State that:

- Item (1), site-specific extensions on the response, can already be
  carried by attaching any extension to `RequestGroup` /
  `RequestGroup.action`; no new IG-defined extension is required to
  enable the pattern.
- Item (2), populating extensions via `dynamicValue`, is already
  permitted by `PlanDefinition.action.dynamicValue` and needs at most a
  documentation note, not a normative change.
- Item (3), site-specific override of a published `PlanDefinition`'s
  `dynamicValue`, is properly handled by either (a) profiling /
  publishing a derived `PlanDefinition` for the site, or (b) the
  existing CDS Hooks `configuration` extension; introducing a new
  `moduleConfiguration` override path risks two competing configuration
  surfaces in the same response.

Keep the ticket as a documentation enhancement (worked example only).

#### Justification

The ticket can be read as asking the IG to bless mechanisms that the
underlying resources already permit, plus an override mechanism whose
need has been actively questioned by the WG before (FHIR-28513). Adding
a parallel "module configuration" surface alongside CDS Hooks'
`configuration` extension creates two ways to do the same thing and
risks the FHIR-36134 critique (no clear interop story). Declining the
normative change while committing to documentation preserves
implementer freedom without locking in a design that has not yet been
validated by connectathon evidence.

---

### Recommendation

**Recommended disposition:** B

The underlying need — letting a deploying site shape the extensions
that flow into a CDS Hooks response from a CPG-evaluated
`PlanDefinition` — is real, recurring, and supported by adjacent work
(FHIR-34265, FHIR-22159, FHIR-18947). Disposition C is too thin: it
leaves authors guessing how response-shaping is supposed to work and
repeats the FHIR-36134 problem of "no examples, no adoption".
Disposition A is the most direct, but it stamps a CPG-specific
StructureDefinition and override syntax onto a problem that CDS Hooks
is already addressing in its own configuration model, and the WG has
historically been wary of multiplying configuration surfaces (see
FHIR-28513). Disposition B threads the needle: one normative
projection rule from `RequestGroup` extensions to CDS Hooks card
extensions, one named hook-configuration convention for site
overrides, and a worked example. It is the smallest spec change that
still gives the reporter the end-to-end story they asked for, and it
keeps the CRD and CPG configuration stories aligned. If the WG wants a
stronger commitment, A remains a reasonable fallback once
connectathon evidence justifies the new IG-owned extension.
