## Summary

The ticket asks the CDS work group to add formal guidance and new/extended
operations to support an "interactive CDS" pattern in the Clinical Practice
Guidelines (CPG) IG. Today, when `PlanDefinition/$apply` runs, the data
required to evaluate applicability conditions on a `PlanDefinition.action`
may be missing or incomplete, and there is no standardised way to fall back
on asking the user. The reporter notes that the existing core
`StructureDefinition/$questionnaire` operation can produce a Questionnaire
from a StructureDefinition and could be combined with SDC's
definition-based extract pattern to populate / extract data via
QuestionnaireResponse — but its single `supportedOnly` filter parameter is
too coarse for CPG case-feature extraction, and CPG needs *one* assembled
Questionnaire spanning all `PlanDefinition.action.input` items rather than
many per-input questionnaires.

The reporter therefore proposes three deliverables: (1) extend
`StructureDefinition/$questionnaire` with an additional parameter to
further limit which elements are mapped to questionnaire items;
(2) add a new `PlanDefinition/$questionnaire` operation that walks the
PlanDefinition's `action.input` set and assembles a single SDC
definition-based-extract-conformant Questionnaire; (3) add CPG IG
documentation covering the questionnaire-processing semantics and the use
of `$questionnaire` together with `$apply` for the interactive CDS model.
A draft PR (HL7/cqf-recommendations#52) and a draft IG branch
(`FHIR-50165-support-interactive-cds`) already exist. There are no
cross-referenced sibling Jira tickets; the only related discussion is a
single Zulip thread in `implementers` where Bryn Rhodes points
implementers at the same draft branch as the canonical "latest thinking"
on this pattern.

## Details

**Description:**

> Data requirements needed to evaluate applicability conditions on
> PlanDefinition/$apply may be missing or incomplete. There is an existing
> [StructureDefinition/$questionnaire](https://hl7.org/fhir/structuredefinition-operation-questionnaire.html)
> operation that returns a questionnaire based on the elements of
> StructureDefinition. This operation can be leveraged with $apply and the
> SDC specification to generate, populate, and extract
> questionnaire/questionnaireResponse based on `PlanDefinition.action.input`.
> However, the core $questionnaire operation specifies only 1 parameter to
> limit which elements are mapped to questionnaire items (`supportedOnly`),
> which is more robust than that required for data extraction of CPG case
> features. In addition, a single questionnaire for all Plan action inputs
> is required to support this process as opposed to individual
> questionnaires for each action input.
>
> Propose:
> 1. Extended `$questionnaire` operation for interactive CDS
>    (`StructureDefinition/$questionnaire`) with additional parameter to
>    further limit the elements that are processed.
> 2. New `$questionnaire` operation for interactive CDS
>    (`PlanDefinition/$questionnaire`) that returns a Questionnaire that
>    conforms to SDC definition-based extract by generating questionnaires
>    for each `action.input` and assembling into a single questionnaire.
> 3. Interactive CDS Documentation:
>    1. Questionnaire processing semantics.
>    2. Use of `$questionnaire` with `PlanDefinition/$apply` to support the
>       interactive CDS model.

## Related Zulip Discussions

### [implementers > An ActivityDefinition that requires a "app" to be used](https://chat.fhir.org/#narrow/stream/implementers/topic/An%20ActivityDefinition%20that%20requires%20a%20%22app%22%20to%20be%20used/near/513615769)

An implementer (Raheel Sayeed) described a real iOS-app CDS scenario in
which a PlanDefinition's ActivityDefinition needs cholesterol data, and
asked how to model "if the app cannot fetch the value, ask the user via a
Questionnaire instead". Bryn Rhodes confirmed that the explicit modelling
today is two coordinated actions (one with applicability `exists
(Cholesterol)`, one with applicability `not exists (Cholesterol)` that
hands off to a Questionnaire), but agreed this "ask the user when data is
missing" pattern is common enough that CPG has been working on first-class
support. He pointed to the draft branch
`FHIR-50165-support-interactive-cds` of cqf-recommendations
(`interactive-cds.html`) as the latest thinking — i.e. the very work this
ticket tracks. There was no dissent; the thread effectively endorses the
direction of the ticket and treats the draft IG branch as the canonical
forward path. Open question on the implementer side: whether it is ever
appropriate to use `relatedArtifact.depends-on` (or `UsageContext`) to
bind an ActivityDefinition to a specific app/device — left unresolved.

Participants: Bryn Rhodes, Raheel Sayeed, Lloyd McKenzie.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/cqf-recommendations` (authoring root `input/`), land the work
already drafted on branch `FHIR-50165-support-interactive-cds` /
PR #52 substantively as proposed:

1. **Extended `$questionnaire` operation (CPG-scoped).** Author a new
   `OperationDefinition` FSH instance under `input/fsh/operations/`
   (e.g. `OperationDefinition-cpg-structuredefinition-questionnaire.fsh`)
   that mirrors core `StructureDefinition/$questionnaire` and adds a new
   IN parameter — for example `elementFilter` (a repeating string of
   element ids/paths) or `profile` (a canonical to constrain the element
   set against) — so callers can narrow the produced Questionnaire to
   exactly the case-feature elements CPG cares about, beyond what
   `supportedOnly` alone permits. Define the parameter precisely (name,
   cardinality, type, semantics, interaction with `supportedOnly`).
2. **New `PlanDefinition/$questionnaire` operation.** Author a second
   FSH `OperationDefinition` (e.g.
   `OperationDefinition-cpg-plandefinition-questionnaire.fsh`) bound to
   `PlanDefinition`, that for a given PlanDefinition (and optional
   `subject`, `encounter`, `parameters`, `data`, terminology context)
   walks `PlanDefinition.action.input`, calls the CPG-extended
   `StructureDefinition/$questionnaire` per input, and assembles the
   results into a single SDC-definition-based-extract-conformant
   Questionnaire (one Questionnaire, with Questionnaire.item groups per
   originating `action.input`), suitable for SDC populate / extract
   round-tripping into the FHIR resources `$apply` then consumes.
3. **Interactive CDS guidance page.** Add `input/pagecontent/interactive-cds.md`
   (the same page already rendered on the draft branch as
   `interactive-cds.html`) covering: the interactive-CDS pattern,
   questionnaire processing semantics (when to call which operation,
   how `action.input` maps to Questionnaire.item, how the
   QuestionnaireResponse is fed back into `$apply`), and worked examples
   tying `$questionnaire` to `PlanDefinition/$apply`. Cross-link from
   `methodology.md` and register the page in `sushi-config.yaml`.
4. Register both operations on the relevant CPG CapabilityStatement(s)
   under `input/fsh/capability-statement/` and add per-SD intro notes
   under `input/intro-notes/` for any new StructureDefinitions /
   extensions introduced to support assembly.

#### Justification

This is the path the reporter, the implementer in the linked
`implementers` Zulip thread, and the existing draft branch + PR all
already converge on. Doing it as a CPG-scoped extended operation (rather
than mutating core `StructureDefinition/$questionnaire`) keeps the
change wholly inside `HL7/cqf-recommendations` per the briefing's
cross-repo notes — no R4 core change is implied, and the IG is at
`fhirVersion: 4.0.1` so it cannot ship a core mutation anyway. SDC's
definition-based extract already provides the round-trip semantics
needed; the contribution here is "assemble one Questionnaire from the
PlanDefinition.action.input set" plus narrative — both of which are
natural homes for the CPG IG. Anchoring deliverable (3) in
`input/pagecontent/` follows the briefing's recommended recipe for
narrative work and uses the same authoring root (`input/`) the rest of
the IG already uses.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the *intent* of the ticket but split the work across the right
scopes and lean harder on existing mechanisms before defining new
operations:

1. **Do not extend core `StructureDefinition/$questionnaire` from
   inside CPG.** Instead, log a separate FHIR-Core ticket against the
   FHIR-I work group requesting the additional element-filter parameter
   on the *core* operation, so future versions of FHIR carry it
   uniformly. Until then, ship a CPG-scoped *profile* (not a new
   operation) of the core operation that narrows usage and documents
   the workaround (call core `$questionnaire` once per relevant
   StructureDefinition and post-prune the result with a documented
   FHIRPath / element-path filter).
2. **Define `PlanDefinition/$questionnaire` as the only genuinely new
   operation**, but specify it as a thin orchestration layer: it MUST
   call SDC's existing populate / extract surface and the core
   `$questionnaire` operation; it MUST NOT redefine element-to-item
   mapping rules. Place it in `input/fsh/operations/` exactly as in
   Disposition A.
3. **Strengthen the modelling alternative for "ask the user when data
   is missing"** in `input/pagecontent/interactive-cds.md` by
   *first* describing the existing two-action pattern Bryn Rhodes
   sketched on Zulip ("retrieve cholesterol" with applicability
   `exists(Cholesterol)` + "Questionnaire-driven action" with
   applicability `not exists(Cholesterol)`) as the conformant R4 baseline,
   and *then* introducing the assembled-Questionnaire optimisation as
   the recommended path once an engine implements it. Keep the page,
   intro notes, and CapabilityStatement updates from Disposition A.

#### Justification

This trades a small amount of surface area for two real wins. First, it
respects the cross-repo touch point flagged by the briefing: deliverable
(1) as worded would silently expand the contract of a *core* FHIR
operation from inside an IG, which the IG is not authorised to do at
`fhirVersion: 4.0.1`; turning it into "CPG profile of the core op +
core ticket upstream" is honest and avoids the CPG IG advertising a
parameter implementers will not find on conformant FHIR servers.
Second, it grounds deliverable (3) in a pattern that *already works*
today (the two-action `exists` / `not exists` model from the Zulip
thread), so guideline authors are not blocked waiting for CR / Reason /
HAPI CR to implement a brand-new operation — the briefing's gotchas
explicitly warn that CDS engines reject novel `dynamicValue` /
operation surfaces. The `PlanDefinition/$questionnaire` assembly
operation is still introduced as the strategic target, but as an
optimisation over a working baseline rather than as the only path.

---

### Disposition C: Decline

#### Proposal

Decline to add either operation and decline to publish a new
"Interactive CDS" guidance page in this cycle. Instead, document under
existing CPG narrative (`input/pagecontent/methodology.md` or a short
appendix) that the supported interactive pattern is the two-action
`PlanDefinition` model already endorsed on Zulip — one action with
applicability `exists(<feature>)` that retrieves data, one action with
applicability `not exists(<feature>)` that points at a Questionnaire —
combined with SDC's existing definition-based extract and the core
`StructureDefinition/$questionnaire` operation as-is. Recommend that
implementers needing finer element selection use SDC's own profiling
mechanisms on the resulting Questionnaire rather than a new operation
parameter.

#### Justification

The deliverables overlap heavily with mechanisms that already exist:
SDC defines populate / extract; FHIR core defines
`StructureDefinition/$questionnaire`; the two-action applicability
pattern is conformant R4 today and was confirmed on Zulip as the
modelling answer for the implementer-driven "ask the user if missing"
case. The briefing's gotchas note that operations introduced in this IG
have a track record of being rejected by multiple engines (CQF Ruler,
Reason, HAPI CR), which raises the bar for adding *two* new operation
surfaces at once. Decline keeps the IG aligned with shipping engines
and avoids embedding a CPG-scoped `StructureDefinition/$questionnaire`
parameter that could mislead implementers about the core operation's
contract.

---

### Recommendation

**Recommended disposition:** B

The work the ticket asks for is genuinely useful and is already in
flight on PR #52, so straight Decline (C) understates the value the WG
has invested and the implementer demand surfaced on the `implementers`
Zulip thread. But Accept-as-requested (A) overreaches in one specific
way that the repo briefing and the cross-repo notes both flag:
"extending `StructureDefinition/$questionnaire`" cannot be done from
inside an IG pinned at `fhirVersion: 4.0.1` without either silently
diverging from the core operation or pretending to speak for FHIR-I.
Disposition B keeps everything Accept-as-requested actually delivers —
the new `PlanDefinition/$questionnaire` operation, the assembly
semantics, the new `interactive-cds.md` guidance page, the
CapabilityStatement registration, the per-SD intro notes — but
re-routes the core-operation extension into a (a) CPG-scoped profile of
the existing operation and (b) an upstream FHIR-Core ticket, and grounds
the narrative in the two-action `exists`/`not exists` pattern Bryn
already endorses on Zulip so authors have a conformant fallback while
engines catch up to the new operation. That gives CDS a clear
recommendation to bring forward, and gives the WG something they can
actually merge into `HL7/cqf-recommendations` without waiting on
HL7/fhir.
