## Summary

The CQL specification currently states that when extracting the `Date` or
`Time` from a `DateTime` value, "the value is normalized to the timezone
offset of the evaluation request timestamp." Bryn Rhodes reports that
real-world CQL implementations of the date-extraction operator do **not**
perform this normalization — they ignore the timezone offset and simply pull
the year/month/day components from the `DateTime` as authored. He proposes
amending the specification text so the normative language matches actual
implementation behavior.

The ticket explicitly aligns with the disposition reached in
[FHIR-49156](https://jira.hl7.org/browse/FHIR-49156) ("Refine how the toDate
operation is evaluated on a datetime") in the FHIRPath specification. That
companion ticket — raised by Brian Postlethwaite and applied as Persuasive
on 2025-10-15 — concluded that `DateTime.toDate()` in FHIRPath should
extract the year, month, and day directly **without timezone
conversion/normalization** (e.g.,
`@2024-01-15T23:30:00-05:00.toDate() = @2024-01-15`), and was implemented
via [HL7/FHIRPath#21](https://github.com/HL7/FHIRPath/pull/21). FHIR-53152
is the corresponding change request to bring the CQL spec into alignment
with that FHIRPath disposition and with implementation reality.

## Details

**Description:**

> The specification for `date from` has this language:
>
> > When extracting the Date or Time from a DateTime value, the value is
> > normalized to the timezone offset of the evaluation request timestamp.
>
> However, implementations for Date extraction have ignored the timezone
> offset. Suggest that this language be corrected to align with
> implementations.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the CQL (FHIR) specification's date/time-operator section that
currently reads:

> When extracting the Date or Time from a DateTime value, the value is
> normalized to the timezone offset of the evaluation request timestamp.

to specify, instead, that **no timezone normalization is performed**: the
year/month/day (for `date from`) or hour/minute/second/millisecond (for
`time from`) components are taken directly from the `DateTime` value as
authored, and the timezone offset of the source `DateTime` and of the
evaluation request are ignored for the purposes of the extraction.
Mirror the wording adopted in FHIR-49156 for FHIRPath, e.g.:

> When the operand is a `DateTime`, the year, month, and day components
> (for `date from`) — or the hour, minute, second, and millisecond
> components (for `time from`) — are extracted directly from the value
> without timezone conversion or normalization. For example,
> `date from @2024-01-15T23:30:00-05:00 = @2024-01-15`.

Add a non-normative note pointing to FHIR-49156 / HL7/FHIRPath#21 so that
implementers can see the alignment, and call out the behavior change
from the prior text in the spec's Change Log / Errata section for the
affected ballot cycle.

#### Justification

This is the change the reporter is asking for and it directly mirrors
the disposition already applied to the FHIRPath specification in
FHIR-49156. The FHIRPath disposition was supported by an explicit survey
of mainstream date/time libraries (Java, C#, Python, JavaScript,
Moment.js, Rust, Go), all of which extract date components without
applying a timezone shift, so the new language is consistent with what
implementers actually do. Because CQL inherits its date/time semantics
from FHIRPath wherever possible, leaving the CQL text in the older
"normalize to evaluation timezone" form would create a real divergence
between the two specs and would force CQL engines that share a FHIRPath
runtime to behave differently from `toDate()` — exactly the kind of
inconsistency Bryn Rhodes flagged on FHIR-49156 ("inconsistent with all
the other extraction operations"). It also restores authoring use cases
such as comparing `date from someDateTime` against `Patient.birthDate`
without producing surprising off-by-one-day results depending on the
evaluator's timezone.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substantive change (extract components directly, no timezone
normalization) but, in addition to correcting the existing prose, also:

1. Add an explicit worked example for both `date from` and `time from`
   on a `DateTime` with a non-UTC offset (e.g.
   `date from @2024-01-15T23:30:00-05:00 = @2024-01-15` and
   `time from @2024-01-15T23:30:00-05:00 = @T23:30:00`).
2. Add an explicit non-normative paragraph describing how to obtain the
   *previous* "normalize to a target timezone first" behavior when an
   author actually wants it — e.g. by first converting the `DateTime` to
   the desired timezone (paralleling Brian Postlethwaite's
   `toTimeZone(zone)` / `toUtc()` suggestion on FHIR-49156) and then
   extracting. If CQL does not yet expose a timezone-conversion
   operator, document the explicit string/component-based workaround
   already in use (`ToString()` + substring + `ToDate()`).
3. Re-open FHIR-28112 (the original CQL/FHIRPath ticket whose
   disposition introduced the now-removed normalization language) and
   close it with this same disposition, mirroring what the FHIR-49156
   resolution committed to do for the FHIRPath side, so the historical
   record on both specs is consistent.

#### Justification

The substantive direction is identical to Disposition A — there is no
reasonable alternative that preserves the current "normalize to
evaluation timezone" rule, since that rule is what implementations have
demonstrably ignored. The added value of Disposition B is preventing
the *next* round of confusion: the original CQL text exists precisely
because the previous disposition (FHIR-28112) chose normalization, so
silently flipping the rule without examples and without a documented
escape hatch risks the same churn re-occurring. Worked examples cost
little and remove ambiguity for every implementer; flagging the
companion FHIR-28112 cleanup keeps the audit trail across CDS, ITS, and
the CQL/FHIRPath specs aligned.

---

### Disposition C: Decline

#### Proposal

Reject the request and retain the current CQL text that says the
`DateTime` is normalized to the timezone offset of the evaluation
request timestamp before extraction. Direct implementers to fix their
engines to match the spec rather than amending the spec to match
non-conforming engines.

#### Justification

The defensible reason to decline would be that CQL is a normative
specification and its authored semantics should not be redefined to
match incorrect implementations; doing so risks rewarding
non-conformance and could break CQL evaluations that have been
deliberately written to rely on the documented normalization rule
(e.g., quality measures whose authors assumed a deterministic
"normalize to evaluator timezone" semantic). In practice, however, this
position is hard to sustain: the FHIRPath specification — which CQL
explicitly draws its date/time semantics from — has already accepted
the opposite resolution in FHIR-49156, so retaining the old CQL text
would put the two specs out of step and would not match either the
FHIRPath normative behavior or any mainstream library. This disposition
is included for completeness rather than as a serious option.

---

### Recommendation

**Recommended disposition:** B

Disposition B is recommended. The substantive change (Disposition A) is
clearly the right call: the FHIRPath spec has already adopted it under
FHIR-49156, every mainstream date/time library Brian Postlethwaite
surveyed extracts date components without applying a timezone shift,
and current CQL engines are already behaving this way. Keeping the
existing CQL text (Disposition C) would leave CQL out of step with
FHIRPath and with reality. The reason to prefer B over a bare A is that
the original CQL language exists *because* an earlier ticket
(FHIR-28112) deliberately chose normalization; if the workgroup just
deletes that language without adding worked examples, without
documenting how an author who genuinely wants normalization should now
express it, and without explicitly closing out FHIR-28112 with the new
disposition, the same question is highly likely to come back as a third
tracker. Disposition B costs only a paragraph or two of additional spec
text and a Jira housekeeping action, and in return it leaves the
CQL/FHIRPath/FHIR-28112/FHIR-49156/FHIR-53152 chain internally
consistent and self-explanatory for future implementers.
