## Summary

Bryn Rhodes is requesting that the *Using CQL With FHIR* implementation
guide define a new **CQL Expression Evaluation Service** capability
statement as a constrained subset of the existing CQL Evaluation Service.
The existing CQL Evaluation Service capability statement covers servers
that support both the system-level `$cql` operation and the
`Library/$evaluate` operation. The reporter observes that there is a
useful, narrower conformance profile for servers that only support the
expression-level `$cql` operation (without `Library/$evaluate`), and asks
that this subset be formalized as its own named capability statement so
that implementers can claim and test against it directly.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items were
returned by the data sources for this ticket.

### Disposition A: Accept As Requested

#### Proposal

Add a new capability statement to the *Using CQL With FHIR* IG named
`cql-expression-evaluation-service` (alongside the existing
`cql-evaluation-service`). The new CapabilityStatement shall:

- Declare the system-level `$cql` operation as `SHALL` support, by
  referencing the existing `OperationDefinition` for `$cql`.
- Explicitly **omit** `Library/$evaluate` (i.e., not list it under the
  `Library` resource's `operation` array).
- Inherit / mirror the same paging, parameter, terminology, and data
  access conformance language used by the existing CQL Evaluation
  Service capability statement, scoped to expression evaluation.
- Be referenced from the IG's narrative section that introduces the
  service tiers, with prose describing the relationship: "CQL
  Expression Evaluation Service" is a strict subset of "CQL Evaluation
  Service", such that any server conformant to the latter is
  automatically conformant to the former.

#### Justification

This is exactly what the reporter (the IG's primary author) is asking
for, and it follows established FHIR practice of layering capability
statements from minimal to full (see, e.g., the SMART App Launch IG's
tiered capability statements, and the way US Core layers on top of base
FHIR). Naming the smaller tier explicitly gives implementers — for
example, services that wrap a CQL engine but do not host a Library
repository — a conformance target they can publish and test against,
rather than having to claim partial conformance against the larger
statement. It also enables tooling (validators, IG publisher, conformance
dashboards) to cleanly distinguish the two server profiles.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of publishing a second standalone CapabilityStatement, model the
distinction as a single CapabilityStatement using the
`CapabilityStatement.imports` / `instantiates` mechanism, plus an
`expectation` extension on the `Library/$evaluate` operation:

- Keep the current `cql-evaluation-service` CapabilityStatement as the
  full profile, marking `$cql` with `expectation = SHALL` and
  `Library/$evaluate` with `expectation = SHALL`.
- Introduce `cql-expression-evaluation-service` as a thin
  CapabilityStatement that lists `$cql` as `SHALL` and is `instantiated`
  by the full service, so the relationship between the two tiers is
  machine-readable.
- Document the subset relationship in the IG narrative and add a
  conformance test to the CI that asserts the full service's
  CapabilityStatement is a superset of the expression-only one.

#### Justification

This addresses the same implementer need but makes the layering
explicit in the artifacts themselves rather than relying solely on prose,
which reduces the risk of the two statements drifting apart over time
(e.g., a future change to the full service's `$cql` parameters that is
not mirrored into the subset). The `instantiates` chain is the
FHIR-native way to express "service B is a strict superset of service
A". The trade-off vs. Disposition A is more authoring complexity and a
small amount of extra publisher tooling work, in exchange for stronger
guarantees that the two tiers stay aligned.

---

### Disposition C: Decline

#### Proposal

Decline the request and instead extend the existing
`cql-evaluation-service` CapabilityStatement narrative to note that
servers are free to support only the `$cql` operation, marking
`Library/$evaluate` with `expectation = MAY` (or `SHOULD`) rather than
`SHALL`. No new CapabilityStatement is published.

#### Justification

Defensible on the grounds that capability statements proliferate easily
and each new named tier carries a long-tail maintenance cost (cross-IG
references, conformance tooling, registry entries). If the existing
statement can be relaxed to express the optionality of
`Library/$evaluate`, implementers who only host expression evaluation
can still claim conformance against a single, authoritative statement,
and the spec avoids creating a second artifact whose only purpose is to
omit one operation. The downside — and the reason this is the weakest
of the three options — is that "claim conformance to the full service
but ignore half of it" is a poor signaling mechanism for consumers
shopping for a service implementation.

---

### Recommendation

**Recommended disposition:** A

The reporter is the IG's principal author, the request is narrow, and
the use case (CQL engine wrappers that do not host Library content) is
real and well understood in the CDS community. Disposition A is the
simplest change that gives implementers a clean conformance target and
matches established FHIR practice for tiered server capability
statements. Disposition B is technically more elegant but is more
authoring overhead than this change warrants on first introduction; the
`instantiates` linkage can be added later if drift between the two
statements becomes a real problem. Disposition C under-serves
implementers by making "expression-only" servers indistinguishable in
their published conformance from full-service servers, which defeats the
point of the request.
