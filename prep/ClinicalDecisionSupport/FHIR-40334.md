## Summary

The reporter asks the Clinical Practice Guidelines (CPG-on-FHIR) IG team to
add explicit guidance describing **how to attach an expression language —
specifically CQL — to a DMN (Decision Model and Notation) decision model**
when DMN is used as the representation of clinical decision logic inside a
CPG knowledge artifact. Today the IG describes how CQL is used directly in
FHIR resources (Library, PlanDefinition, Measure, etc.) and references DMN
as one of several allowed knowledge representations, but it does not
explain the integration pattern between the two: how a CQL expression
should be referenced from a DMN element (decision, input, business
knowledge model, literal expression, …), what the `expressionLanguage`
attribute should be set to, how CQL libraries are bound, and how runtime
evaluators should resolve the reference. The request is for normative or
informative IG text that closes that gap so DMN-based CPG content is
portable across engines.

There are **no cross-referenced Jira, Zulip, Confluence, or GitHub items**
in the knowledge base for this ticket. Keyword-similar CPG/CQL tickets
(listed below under *Related Jira Tickets*) provide adjacent context — most
notably FHIR-51771 (alternate expression languages for criteria Groups),
FHIR-49740 (expressions referencing library content), FHIR-49976 (use of
the `reference` element in Expression-valued elements), and FHIR-29551
(`text/cql.expression` MIME type) — all of which inform how CQL is
currently surfaced and bound in CPG, and therefore how a DMN binding
should be specified for consistency.

The ticket history is significant for disposition: it was first resolved
**Considered for Future Use** on 2024-07-10 by a 20-0-0 vote with the note
*"Will pursue after STU2 publication"*, and then **re-opened on
2024-11-26** by lynn_laakso post-STU2 so the work group can pursue it now.
The labels `Block-Vote-1` and `Ready-For-Vote` indicate it is queued for
the next block vote and a substantive disposition is expected.

### [FHIR-32672: Add support for specifying CQL options in CQL/ELM libraries and projects](https://jira.hl7.org/browse/FHIR-32672)

Concerns how CQL compilation / translation options are conveyed alongside

explicitly inherit) the same options as Library-hosted CQL for the

### [FHIR-40486: Add guidance on when available ELM can be used](https://jira.hl7.org/browse/FHIR-40486)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new informative section to the **CPG-on-FHIR IG** (Clinical
Guidelines) titled *"Using CQL inside DMN decision models"* that
specifies, normatively where possible:

1. **Expression-language identifiers.** When a DMN element
   (`decision`, `businessKnowledgeModel`, `inputData`, `literalExpression`,
   `context`, etc.) carries CQL, its DMN `expressionLanguage` attribute
   SHALL be set to one of the FHIR-registered CQL MIME-type URIs:
   - `text/cql` for a CQL library snippet,
   - `text/cql-identifier` for a reference to a named `define` in an
     external Library,
   - `text/cql.expression` for an inline CQL expression (per FHIR-29551).
2. **Library binding.** A DMN model that uses CQL SHALL declare the FHIR
   `Library` resources it depends on by URL (canonical + version) using a
   DMN `import` element with `importType` set to a CPG-defined URI (e.g.,
   `http://hl7.org/fhir/uv/cpg/StructureDefinition/cqf-library`) and
   `locationURI` pointing at the Library's canonical URL. Provide an
   example.
3. **Identifier resolution.** Inside a DMN literal expression with
   `expressionLanguage = text/cql-identifier`, the body is a single CQL
   identifier resolved against the imported Libraries using the standard
   CQL library-qualified-name rules.
4. **Data model & context.** State that the CQL evaluation context
   (Patient, Practitioner, …) is supplied to the DMN engine through DMN
   `inputData`, and that the CQL `using` clause SHALL match the FHIR
   model version used by the surrounding PlanDefinition / CPG artifact.
5. **Worked example.** Include a small end-to-end example: a DMN
   decision table whose input expressions are CQL identifiers backed by
   a CPG `Library`, packaged as a `PlanDefinition` action, with the
   expected evaluator behaviour.
6. **Conformance.** State conformance requirements for CPG engines that
   claim DMN+CQL support (MUST honour the registered MIME types, MUST
   resolve Library references via the FHIR canonical mechanism).

#### Justification

This is exactly what the reporter asked for and what implementers need
in order to author portable DMN+CQL content. The mechanism re-uses
existing FHIR primitives — registered CQL MIME types (FHIR-29551), the
`Expression`/`Library` reference pattern being clarified by FHIR-49740
and FHIR-49976, and CPG's existing Library canonical-URL conventions —
so it adds guidance without inventing new machinery. The 20-0-0 vote in
July 2024 to defer the work until after STU2 indicates the work group
already considers this in scope; the November 2024 re-open puts it
firmly on the STU3 cycle. Block-Vote-1 / Ready-For-Vote labels
corroborate that a substantive accept is expected.

### Disposition B: Alternative Approach

#### Proposal

Instead of writing DMN-specific binding text, **publish a small
companion *"Expression Language Binding"* section in the CPG IG that
generalises across all non-CQL knowledge representations** (DMN today,
BPMN, GDL, native rules, etc.) and treat DMN as the first concrete
example. Concretely:

1. Define a single CPG pattern: any non-CQL knowledge representation
   that wants to embed or reference CQL SHALL (a) identify the CQL
   payload using one of the registered CQL MIME types, and (b) bind to
   FHIR Library resources by canonical URL.
2. Provide a normative table mapping each registered CQL MIME type to
   the kind of payload it carries (snippet / identifier / inline
   expression).
3. Add a *non-normative* "DMN profile of the binding" subsection
   showing how the generic rule is realised in DMN's
   `expressionLanguage` and `import` constructs (i.e., the content of
   Disposition A, but framed as one example of the general pattern).
4. Cross-link to the resolution of FHIR-51771 (alternate languages for
   criteria Groups) so the two tickets share one binding mechanism.

#### Justification

DMN is not the only non-CQL representation the CPG community uses, and
FHIR-51771 is actively asking the same question for criteria Groups.
Solving FHIR-40334 in isolation risks producing a DMN-shaped answer
that has to be retrofitted when the next non-CQL language (BPMN, GDL, a
rules engine DSL) appears. A generalised binding pattern is a small
amount of additional text but materially reduces future churn, keeps
the CPG IG's expression-language story coherent, and is consistent with
FHIR design principles favouring one binding mechanism over many. The
trade-off versus Disposition A is scope: this disposition is broader
and may be objected to by reviewers who want a tightly-scoped DMN-only
fix in this ticket, in which case it can be split into a parent
"binding" section plus a DMN child section authored under FHIR-40334.

---

### Disposition C: Decline

#### Proposal

Mark the ticket *Not Persuasive — Out of Scope*: the CPG IG references
DMN as an allowed knowledge representation but does not own the DMN
specification, and the OMG DMN spec already defines `expressionLanguage`
as the standard mechanism for naming any embedded language. A reader
who knows DMN can simply set `expressionLanguage` to the appropriate
CQL MIME type (`text/cql`, `text/cql-identifier`,
`text/cql.expression`) without IG-specific text.

#### Justification

Defensible only on a strict reading of scope: DMN is an OMG standard,
the CQL MIME types are registered by HL7 elsewhere, and the
combination is "obvious" to anyone fluent in both. Declining avoids
adding IG text the work group would need to maintain. However, this
position is weak in light of (a) the explicit re-open by lynn_laakso
post-STU2, (b) the 20-0-0 prior vote that committed to pursuing the
work, and (c) the absence of any worked example anywhere in HL7
material — a freshly arriving implementer has to assemble the answer
from three separate specs, which is exactly the gap the IG exists to
fill. Declining should only be selected if the work group judges that
no implementer is actually authoring DMN-based CPG content today.

---

### Recommendation

**Recommended disposition: B (Alternative Approach).**

Disposition A correctly answers the literal request and would be a
good outcome on its own. The reason to prefer B is that the *same
question* is open for at least one other non-CQL representation
(FHIR-51771, alternate languages for criteria Groups), and the
mechanism that has to be specified — "identify the embedded language by
its registered MIME type and bind referenced content via FHIR Library
canonical URLs" — is identical in both cases. Writing the binding
pattern once, then giving DMN a worked example under it, satisfies
FHIR-40334 in full while creating the substrate that FHIR-51771 (and
future tickets) can reuse without re-litigating the design. It also
aligns with the in-flight clarifications in FHIR-49740 and FHIR-49976
about Expression/Library references, which are themselves moving
toward a single, consistent binding story. The history on this ticket
(20-0-0 prior vote, post-STU2 re-open, Block-Vote-1 / Ready-For-Vote)
strongly argues against Disposition C; the work group has already
signalled intent to act. If the work group prefers a tighter scope for
this specific block vote, fall back to Disposition A and open a
follow-up ticket to generalise the pattern.
