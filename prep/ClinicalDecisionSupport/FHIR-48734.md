## Summary

The ticket requests that the `cqf-messages` extension — used to convey warning
and informational messages produced by a CQL/logic engine during guideline
evaluation — be explicitly permitted on the Clinical Practice Guidelines (CPG)
result profiles: `CPGPathway`, `CPGStrategy`, `CPGRecommendation`, and all of
the Recommendation proposal profiles. It also asks that guidance be added to
the `PlanDefinition/$apply` and `ActivityDefinition/$apply` operation
definitions stating that this extension is the expected mechanism for
returning evaluation-engine messages, with all messages aggregated on the
topmost resulting resource of a given `$apply` invocation. The ticket has no
cross-referenced Jira items; the only cross-reference is the Zulip thread
(`cql` > `$apply cql Message operation warnings`) in which the gap the ticket
addresses was first surfaced by implementers.

## Details

**Description:**

The cqf-messages extension is used to return warning and information messages
generated by the logic engine as part of evaluating guideline logic, but this
extension is not specified in any of the CPG profiles.

Allow this extension to be used on:

- CPGPathway
- CPGStrategy
- CPGRecommendation
- All the Recommendation proposal profiles

In addition, provide guidance in the PlanDefinition/$apply and
ActivityDefinition/$apply operations that this extension should be used to
communicate warning and information messages coming back from logic
evaluation. Within any given operation, all messages should be reported at
the topmost resulting resource.

the extension approach but flagged the gap that it is neither in R4 nor in
the CPG profiles; no disagreement was recorded and consensus pointed toward
closing the gap on the CPG side rather than inventing a new mechanism.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the CPG IG (Clinical Guidelines / `cpg-on-fhir`) to add the
`cqf-messages` extension to the allowed-extensions set on `CPGPathway`,
`CPGStrategy`, `CPGRecommendation`, and all Recommendation proposal profiles
(the CPG `*Recommendation` proposal family) at cardinality `0..*`, binding
to the published extension at
`http://hl7.org/fhir/extensions/5.1.0/StructureDefinition-cqf-messages.html`.
In parallel, update the narrative and OperationDefinitions for
`PlanDefinition/$apply` and `ActivityDefinition/$apply` to specify that the
`cqf-messages` extension is the canonical mechanism for returning
evaluation-engine warnings and information messages, and that when multiple
resources are produced by a single `$apply` invocation, all messages MUST be
carried on the topmost resulting resource (the root `CarePlan` /
`RequestGroup` / recommendation resource) rather than being scattered across
contained resources.

#### Justification

This is exactly what the reporter asks for, and it closes the gap raised on
the `cql` stream by implementers who had no conformant place for `$apply`
diagnostics on R4/CPG outputs. The `cqf-messages` extension is already
published in the extensions pack and is already allowed on R5 `CarePlan`, so
permitting it on the CPG result profiles is an extension-pack-scoped change
that does not alter the base spec. Consolidating messages on the topmost
result resource matches the de facto practice reported in the Zulip thread
(JP's contained-`OperationOutcome`-on-the-root pattern) while standardising
on a single, typed extension rather than ad-hoc `OperationOutcome`
containment.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) allowing `cqf-messages` on the CPG profiles,
specify that `$apply` returns a `Bundle` (or `Parameters`) whose first entry
is the produced plan/recommendation and whose additional entries include an
`OperationOutcome` carrying the engine's warning/information `issue`s, and
cross-reference that `OperationOutcome` from the root resource via a single
`cqf-messages` extension instance. The CPG profiles would only need to be
touched to permit the extension on the root result types, leaving the
message payloads in an `OperationOutcome` — which is the FHIR-native
container for engine diagnostics and what JP's implementation already
produces in practice.

#### Justification

`OperationOutcome` is the standards-aligned carrier for engine-produced
diagnostics and is already widely supported by clients. Layering
`cqf-messages` on top of a real `OperationOutcome` (rather than carrying
free-text messages directly on the result resource) preserves severity/code
structure and avoids encouraging profiles to sprout parallel message
vocabularies. The downside versus Disposition A is a slightly more complex
`$apply` response shape, but it avoids a recurring problem where every
guideline-flavoured result profile ends up re-declaring where messages may
appear.

---

### Disposition C: Decline

#### Proposal

Decline the request on the grounds that the `cqf-messages` extension is
already usable on any resource per the FHIR extensibility rules and that
`$apply` implementations can already return engine diagnostics via
`GuidanceResponse.evaluationMessage` or a contained `OperationOutcome`, so
no normative change to the CPG profiles or the `$apply` operation
definitions is required.

#### Justification

FHIR extensions are, by default, usable on any resource unless a profile
forbids them; "allowing" `cqf-messages` on CPG profiles is arguably a
clarification rather than a substantive change, and the existing
`GuidanceResponse.evaluationMessage` slot plus the contained-
`OperationOutcome` pattern already covers the implementer need described in
the Zulip thread. Declining avoids a wave of coupled profile edits across
the CPG IG and keeps the message-carrying story centralised in the
extensions pack.

---

### Recommendation

**Recommended disposition:** A

The Zulip thread demonstrates a concrete implementer gap — there is no
agreed-upon place on CPG-flavoured `$apply` outputs to hang CQL `Message`
operation diagnostics — and the community (including the extension's own
author, Bryn Rhodes) has already converged on `cqf-messages` as the right
mechanism. Disposition A closes that gap with the minimum set of targeted
edits (extension allowance on the CPG result profiles plus narrative in the
two `$apply` operations) and aligns the CPG IG with the R5 `CarePlan` usage
that is already permitted. Disposition B has real merit if the work group
wants to standardise the `$apply` response shape as well, and can be folded
into A as a follow-on clarification. Disposition C under-serves the
implementer need: the "extensions are allowed anywhere by default" argument
is technically true but practically unhelpful — validators and profile
authors expect CPG result profiles to enumerate their supported extensions,
which is the whole point of the ticket.
