## Summary

The ticket flags four code systems / value sets bound on the `EvidenceVariable`
resource as candidates for migration to the Terminology HL7 Org (THO):
`variable-handling-example` (bound to `EvidenceVariable.handling`),
`evidence-variable-classifier` (bound to `EvidenceVariable.classifier`),
`evidence-variable-constraint` (bound to `EvidenceVariable.constraint.code`),
and `evidence-variable-unacceptable-data-handling` (bound to
`EvidenceVariable.unacceptableDataHandling`). Migration is not strictly
required, but is suggested where the value set definitions are likely to
evolve over time or where they may be useful outside FHIR (e.g., CDA, V2,
or other specifications). The reporter (Lloyd McKenzie) appears to have
filed this as part of a broader sweep of FHIR Core value sets reviewed
against THO migration criteria.

The discussion in the ticket links the `variable-handling` work back to
[FHIR-51456](https://jira.hl7.org/browse/FHIR-51456) ("Change
EvidenceVariable.handling from code to CodeableConcept"), which has
already been Applied/Persuasive and which explicitly anticipated that the
relevant codes would land in the Scientific Evidence Code System (SEVCO)
and the resulting value set would later move to THO. CDS triage has
already proposed a per–value-set disposition: wait for SEVCO Release 3.0
for `variable-handling`; keep `evidence-variable-classifier` as an
example binding only (no community appetite for a shared value set);
move `evidence-variable-constraint` and
`evidence-variable-unacceptable-data-handling` to THO, where future
community work can refine them.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the per–value-set plan already proposed in the triage comment:

1. **`variable-handling-example` → migrate to THO.** Once SEVCO Release
   3.0 is published with the approved codes, define a new ValueSet
   referencing SEVCO and host it in THO. Re-point
   `EvidenceVariable.handling`'s binding to the THO ValueSet
   canonical (extensible binding, consistent with FHIR-51456). Retire
   the in-spec example CodeSystem/ValueSet `variable-handling-example`
   from the FHIR Core build once the THO ValueSet is live. Coordinate
   with the SEVCO maintainers (Scientific Evidence Code System) to
   confirm publication timing.
2. **`evidence-variable-classifier` → no migration.** Leave the
   CodeSystem and ValueSet in the FHIR Core build; keep the binding
   strength as **example** on `EvidenceVariable.classifier`. Add a
   short note in the value set/element description that no shared
   community value set is currently anticipated.
3. **`evidence-variable-constraint` → migrate to THO.** Move the
   CodeSystem and ValueSet to THO with the existing canonical
   identifiers preserved (or with explicit `predecessor` mappings if
   the canonical changes), and re-point `EvidenceVariable.constraint.code`
   to the THO ValueSet. Keep the existing binding strength.
4. **`evidence-variable-unacceptable-data-handling` → migrate to THO.**
   Same treatment as constraint above; re-point
   `EvidenceVariable.unacceptableDataHandling`.

For all migrated artifacts, mark the FHIR Core copies as deprecated /
retired in the next ballot cycle, and add the new THO canonicals to the
EvidenceVariable resource page's terminology table.

#### Justification

This plan is the minimum-surprise outcome for the community: it is the
recommendation Lloyd's reviewer (JIRAUSER18251) made on the ticket and
that bryn.rhodes (the EBMonFHIR / CDS lead) explicitly agreed with. It
also follows the precedent set by FHIR-51456, which committed in
principle to moving the `handling` value set to THO once SEVCO held the
codes. THO is the right home for value sets that are likely to evolve
faster than the FHIR Core ballot cycle and may be reused in CDA/V2,
which is exactly the rationale Lloyd cites in the ticket. Keeping
`evidence-variable-classifier` in-spec as an example binding aligns
with the "no community appetite" finding and avoids creating a THO
artifact that nobody will maintain.

---

### Disposition B: Alternative Approach

#### Proposal

Migrate **only** `variable-handling-example` (the
FHIR-51456-driven artifact) to THO/SEVCO as part of the SEVCO 3.0 cut,
and **defer** the other three (`evidence-variable-classifier`,
`evidence-variable-constraint`,
`evidence-variable-unacceptable-data-handling`) to a later cycle,
gated on a concrete community use case being raised for shared
maintenance. In the meantime, leave all three as in-spec value sets
with their current binding strengths and add a sentence to each
ValueSet description noting that THO migration will be reconsidered
once a shared use case is identified.

#### Justification

Two of the four value sets are described in the triage comment as
having only a *possibility* of future shared community interest. THO
migration carries non-trivial overhead: namespace coordination,
downstream dependency updates, redirection of canonicals, and ongoing
maintenance ownership outside the resource's own work group. Deferring
those two until a real driver appears avoids creating "orphaned" THO
content that nobody actively maintains, while still doing the part of
the work that is concretely justified (the `handling` migration tied
to SEVCO 3.0). This approach is more defensible to implementers who
have already pinned to the in-spec canonicals.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — no migration**. Keep all four
CodeSystems and ValueSets in the FHIR Core build at their current
canonical URLs and binding strengths. Document on the EvidenceVariable
resource page that THO migration was considered and not pursued at
this time.

#### Justification

The ticket itself acknowledges that THO migration "is not required."
The triage comment indicates that for two of the four value sets there
is no current community desire for a shared value set, and for one
more the migration is gated on an external dependency (SEVCO 3.0)
that is not yet published. Declining preserves the existing canonical
URLs, avoids any churn for implementers and IGs that already reference
the FHIR Core artifacts, and keeps maintenance authority with the CDS
work group that knows the resource best. This is the least disruptive
option if the work group's priority is stability over cross-spec
reuse.

---

### Recommendation

**Recommended disposition:** A

The ticket is essentially a tracking item for a decision that has
already been broadly agreed: the triage commenter laid out a clean
per–value-set plan and bryn.rhodes (the EBMonFHIR / CDS sponsor)
explicitly agreed with it. The `variable-handling` migration is also
the natural completion of FHIR-51456, which the work group has already
applied and which only deferred the THO step pending SEVCO 3.0.
Disposition A captures all of that decision in one place — including
the explicit *non*-migration of `evidence-variable-classifier` — which
is exactly what the workgroup needs documented for future implementer
questions. Disposition B is a reasonable fallback if the work group
wants to wait for harder evidence on the constraint /
unacceptable-data-handling value sets, but it leaves the same
question open for the next ballot cycle. Disposition C is hard to
defend given that the originating ticket (FHIR-51456) and its
applied resolution already pre-committed to the THO direction for at
least the `handling` value set.
