## Summary

The ticket asks the Clinical Decision Support (CDS) work group to publish an
R4 extension in the FHIR Extensions Pack that mirrors the
`PlanDefinition.action.applicabilityBehavior` element being added to FHIR
Core (R6) under [FHIR-50490](https://jira.hl7.org/browse/FHIR-50490). The
extension is needed so that R4 implementers can express ordered, mutually
exclusive ("else if") evaluation of `action` children — the `all` (current
default) versus `any` (first-applicable-wins) semantics — without waiting
for R6. The submitter also explicitly requires that the extension be usable
on `RequestOrchestration` in addition to `PlanDefinition`, since both
resources carry the affected `action` backbone. A directly related earlier
CDS ticket, [FHIR-50150](https://jira.hl7.org/browse/FHIR-50150), already
proposed this R4 extension and was resolved Persuasive with a concrete
design (`cqf-applicabilityBehavior`, bound to
`http://hl7.org/fhir/action-applicability-behavior`); FHIR-52885 effectively
re-files that work and adds the `RequestOrchestration` usage requirement.

## Related Jira Tickets

A near-duplicate CDS Change Request from Bryn Rhodes (Mar 2025) that asks

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the design already approved on
[FHIR-50150](https://jira.hl7.org/browse/FHIR-50150) and publish a single
new extension in the FHIR Extensions Pack:

- **Extension URL:** `http://hl7.org/fhir/StructureDefinition/cqf-applicabilityBehavior`
  (matches the `cqf-` naming chosen in FHIR-50150's resolution).
- **Cardinality / type:** `0..1 code`.
- **Context:** allow on `PlanDefinition.action` **and**
  `RequestOrchestration.action` (this is the addition FHIR-52885 makes
  on top of FHIR-50150).
- **Binding:** `required` to the value set
  `http://hl7.org/fhir/ValueSet/action-applicability-behavior`, backed
  by code system `http://hl7.org/fhir/action-applicability-behavior`
  with codes:
  - `all` — current behavior; each child action's applicability is
    evaluated independently and every applicable action is applied.
  - `any` — child actions are evaluated in document order; the first
    action whose applicability evaluates to true is applied and
    processing of the parent action stops.
- **Short / definition:** mirror the wording adopted for the R6 Core
  element in FHIR-50490 so the R4 extension and the R6 element behave
  identically.
- **Documentation:** include an example on both `PlanDefinition` and
  `RequestOrchestration`, and note that the extension is intentionally
  *not* a cross-version extension because the feature must be available
  before R6 publishes. Cross-reference FHIR-50490 (R6 element) and close
  FHIR-50150 as a duplicate of FHIR-52885 (or vice versa).

Implementation work would land in the **FHIR Extensions Pack** repo
(`HL7/fhir-extensions`); a `repo-analysis` briefing for that repo should
be generated before opening a PR so authoring roots, terminology
publishing conventions, and IG-publisher gotchas are confirmed.

#### Justification

This is exactly what FHIR-52885 asks for, and the design has already
been agreed Persuasive on FHIR-50150 by the same work group. Reusing
the FHIR-50150 design keeps the R4 extension semantically aligned with
the R6 Core element added under FHIR-50490, which minimizes implementer
churn when migrating from R4 to R6. Allowing the extension on
`RequestOrchestration.action` is a low-cost addition that follows the
same precedent as other action-level extensions and matches the user's
explicit requirement; both resources share the same `action` backbone
pattern, so a single extension definition with two contexts is simpler
than two parallel extensions. Publishing in the Extensions Pack (rather
than a CPG/CQF IG) is consistent with FHIR-50150's resolution and makes
the extension broadly available without forcing implementers to adopt a
specific IG.

### Disposition B: Alternative Approach

#### Proposal

Resolve FHIR-52885 as a duplicate of
[FHIR-50150](https://jira.hl7.org/browse/FHIR-50150) and re-open / amend
FHIR-50150 with two adjustments before implementation:

1. Broaden the documented context from
   `PlanDefinition.action` to include `RequestOrchestration.action`
   (the only substantive addition FHIR-52885 introduces).
2. Reconsider the extension URL: instead of the `cqf-` prefix proposed
   in FHIR-50150 (which historically signals CPG/CQF-IG ownership), use
   a neutral name such as
   `http://hl7.org/fhir/StructureDefinition/applicabilityBehavior` so
   the extension's name matches the R6 Core element exactly and does
   not imply CQF-only scope.

Track the work on FHIR-50150 (which already has the agreed design and a
Persuasive resolution) and link FHIR-52885 to it as the source of the
`RequestOrchestration` requirement.

#### Justification

FHIR-52885 and FHIR-50150 are functionally the same request from the
same work group, with overlapping reporters/owners (Bryn Rhodes on
50150; Brian Kaney pointing back to the same R6 ticket on 52885).
Carrying two open tickets risks divergent resolutions and double
implementation work. Consolidating on FHIR-50150 — which already has a
fully specified design and a Persuasive resolution — preserves the
prior work-group decision while letting the small `RequestOrchestration`
delta be folded in cleanly. Renaming away from the `cqf-` prefix avoids
implementer confusion that the extension is CQF-IG-only and aligns the
R4 name with the R6 element name added in FHIR-50490, easing future
migration. Trade-off vs. Disposition A: one additional process step
(close-as-duplicate plus a small amendment), but stronger long-term
alignment between the R4 extension and the R6 Core element.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-52885 as **already addressed** and close it as a duplicate
of [FHIR-50150](https://jira.hl7.org/browse/FHIR-50150) with no further
change. Direct any remaining R4 implementers needing the
`RequestOrchestration` use case to add their requirement as a comment on
FHIR-50150 so it is handled within that ticket's implementation.

#### Justification

The substantive design work has already been done and accepted under
FHIR-50150 (Resolution: Persuasive). Maintaining a second open ticket
that re-files the same change risks divergent implementations and
duplicates work-group review effort. The only new content in
FHIR-52885 — usage on `RequestOrchestration.action` — is small enough
to be captured as a clarification on FHIR-50150 rather than as a
separate change request. Declining here is defensible because nothing
is being lost: the extension is still going to ship via FHIR-50150, and
the `RequestOrchestration` requirement can be carried forward as a
comment.

---

### Recommendation

**Recommended disposition:** B

FHIR-52885 should be resolved as a duplicate of FHIR-50150, with
FHIR-50150 amended to (a) explicitly include
`RequestOrchestration.action` in the extension's context and (b) align
the extension's URL/name with the R6 Core element added in FHIR-50490.
This captures everything FHIR-52885 actually adds (the
`RequestOrchestration` reuse) without leaving two parallel tickets open
in the same work group, and it tightens the R4↔R6 naming so the
extension is clearly the back-port of the R6 element rather than a
CPG/CQF-flavored variant. Pure Disposition A would also work, but it
ignores that FHIR-50150 already exists with an approved design and risks
divergent decisions; pure Disposition C drops the small but real
`RequestOrchestration` requirement on the floor. Disposition B preserves
the prior work-group decision, folds in the new requirement, and keeps
the R4 extension and the R6 Core element conceptually identical.
