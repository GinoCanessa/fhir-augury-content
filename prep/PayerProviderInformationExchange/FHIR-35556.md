## Summary

The ticket asserts that a CodeSystem defined inside the US CARIN Digital
Insurance Card (DIC) IG should not have its canonical URL anchored under
`http://hl7.org/fhir/...` (i.e., the IG's own canonical base) but should
instead be anchored under `http://terminology.hl7.org/...` (THO). The
reporter further requests that a corresponding UTG (Unified Terminology
Governance) ticket be opened to register the Code System inside THO so
that it has a permanent, governed home outside the IG. The body of the
ticket attributes the original wording to "Comment 6 - imported by:
Ron G. Parker", indicating it was migrated from a comment on an earlier
review. There are no Jira, Zulip, or GitHub cross-references for this
ticket in FhirAugury, so the review must rest on the ticket text itself
and standard HL7 practice for CodeSystem / THO governance.

## Proposed Dispositions

The dispositions below are constrained by the absence of any GitHub
cross-reference or persisted repo briefing — no specific authoring root
or file path can be cited from a briefing. Where a code/spec change is
proposed, it is described against the standard layout of an HL7 IG
publication source (CodeSystem resources under the IG's `input/` tree)
and the standard UTG/THO submission workflow, both of which are
process-level facts independent of any single repo briefing.

### Disposition A: Accept As Requested

#### Proposal

Move the CARIN DIC-defined CodeSystem(s) currently published with a
`CodeSystem.url` of the form `http://hl7.org/fhir/us/insurance-card/...`
to a THO-anchored canonical of the form
`http://terminology.hl7.org/CodeSystem/<code-system-id>`. The change
has two coordinated parts:

1. **In the CARIN DIC IG**: update each affected CodeSystem resource so
   that `CodeSystem.url` (and any `valueSet` reference, FSH alias, or
   markdown citation that hard-codes the old URL) points at the new
   THO canonical. Update bindings on profiles/extensions that reference
   the CodeSystem so they continue to resolve. Add an entry to the IG's
   change log calling out the canonical-URL migration and any
   compatibility note for STU1.1 implementers.
2. **In UTG / THO**: open the requested UTG ticket to register the
   CodeSystem under terminology.hl7.org with the new canonical, the
   CARIN DIC IG as the source, and the appropriate steward (Payer/
   Provider Information Exchange WG or, if delegated, Vocabulary WG).
   Once UTG approves and publishes the CodeSystem in THO, the IG's
   future builds resolve the bindings against the THO copy.

#### Justification

This is the canonical HL7 pattern: vocabulary that is reusable across
specifications, or that warrants permanent governance independent of
its originating IG, lives in THO under
`http://terminology.hl7.org/CodeSystem/...`, not under an IG's own
`hl7.org/fhir/us/...` namespace. The reporter's request mirrors this
pattern exactly, and the WG's own comment trail acknowledges the
change "should get added to THO" and only deferred on process grounds.
Re-opening "post STU1.1 publication" indicates the WG considers this
ticket eligible to move now that the deferral rationale (Vocabulary WG
process not yet formalized) is materially weaker than it was in 2022.

### Disposition B: Alternative Approach

#### Proposal

Keep the CARIN DIC CodeSystem(s) where they are for the current ballot
cycle but file the UTG submission now and add forward-pointing language
to the IG: a `CodeSystem.identifier` carrying the future THO canonical,
plus narrative in the CodeSystem and binding pages stating that the
authoritative location will move to THO in a future release. Once UTG
publishes the CodeSystem in THO, supersede the IG-internal copy in the
following publication and redirect bindings.

#### Justification

This trades immediate compliance with the THO anchoring rule for
publication-cycle stability. Changing `CodeSystem.url` is
binding-affecting: every profile binding, every conformance check, and
every implementer's local copy that pinned the old canonical needs to
move. Doing it in two phases (signal now, switch on a published THO
artifact later) avoids shipping a CodeSystem whose declared canonical
is not yet resolvable on terminology.hl7.org, which has historically
been an implementer pain point for CARIN-family IGs. It also lets the
Vocabulary WG-led UTG submission run on its own clock without blocking
the DIC publication train.

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Mod" → "no change to canonical
URL", on the grounds that the affected CodeSystem(s) are
DIC-specific (e.g., enumerations only meaningful inside the Digital
Insurance Card domain), that no other HL7 specification has shown a
need to reuse them, and that the existing `hl7.org/fhir/us/...`
canonical is already a published, stable, governed namespace owned by
the same WG. Note in the resolution that the THO move can be
reconsidered if and when reuse outside DIC emerges.

#### Justification

The IG-anchored canonical is not malformed or unstable — it is the
default canonical pattern for any HL7 IG, and HL7 has many published
CodeSystems that legitimately live under `hl7.org/fhir/...` because
they are scoped to their defining IG. The cost of the move (binding
churn, implementer migration, FSH/profile updates, an STU dot-release
to fix references) is real, while the benefit (reuse + central
governance) is speculative for a CodeSystem that today has exactly one
consumer. Declining is defensible if the WG concludes the CodeSystem in
question is genuinely DIC-local.

### Recommendation

**Recommended disposition:** A

This ticket has been on the WG's queue since 2022, has been
*re-opened twice* after publication, and the WG's own comments
explicitly state the CodeSystem(s) "should get added to THO" — the only
recorded objection has been process-level (Vocabulary WG had not
formalized the THO submission process). That process is now mature, and
the ticket carries `Block-Vote-6` plus `Highest` priority, signalling
the WG already intends to dispatch it. Disposition A delivers what the
reporter asked for and matches consistent HL7 practice for moving
IG-defined terminology to THO once it has stabilised. Disposition B is
the right fallback if, on discussion, the WG determines that the
binding/implementer churn cannot be absorbed in the next publication
window — in which case it should be paired with a firm target release
for the canonical switch, not left open-ended. Disposition C should
only be selected if the WG concludes the specific CodeSystem(s) at
issue are unambiguously DIC-local with no plausible reuse, which is
inconsistent with the WG's own prior comments.
