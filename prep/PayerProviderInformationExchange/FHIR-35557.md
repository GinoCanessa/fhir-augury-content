## Summary

FHIR-35557 is one item in a block-vote set (Block-Vote-6) raised against
the **US CARIN Digital Insurance Card (DIC)** IG. The reporter (Joan
Harper, conveying ballot Comment 7 originally authored by Ron G. Parker)
asserts that one of the IG's locally-defined `CodeSystem` resources is
currently published with a canonical URL anchored under `hl7.org/fhir/...`,
and that — per HL7 vocabulary policy — the defining (canonical) URL for
that `CodeSystem` should instead be anchored under
`terminology.hl7.org` (THO). The request has two parts: (a) re-anchor
the `CodeSystem.url` (and any references to it) to the THO namespace,
and (b) file a corresponding UTG ticket so the `CodeSystem` is formally
registered/maintained in THO rather than living only inside the CARIN
DIC IG.

The ticket itself does not name the specific `CodeSystem` — the snapshot
carries no `Related Artifacts`, `Related Pages`, `Related URLs`, or
`Related Sections` values, and the ticket has no recorded
cross-references in Augury (Jira, Zulip, or GitHub). The original ballot
context lives in "Comment 7" of the imported CARIN DIC ballot, which is
not attached to this Jira issue.

This ticket is one of a family of nearly-identical tickets filed against
CARIN DIC and other CARIN/Da Vinci IGs at the same time — the closest
sibling, [FHIR-35556](https://jira.hl7.org/browse/FHIR-35556) ("…hl7.org/fhir.1",
imported from Comment 6), is structurally identical and was deferred via
the same vote. Both were originally resolved "Considered for Future Use"
in March 2022 (Roberts/Kissel 11-0-1), reverted, deferred again in
January 2024, then re-opened post-STU 1.1 publication in April 2024 and
remain Triaged.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

For the specific CARIN DIC `CodeSystem` referenced by ballot Comment 7
(to be identified by the workgroup from the original ballot record):

1. In the next CARIN DIC publication, change `CodeSystem.url` from
   `http://hl7.org/fhir/us/carin-bb/...` (or whichever
   `hl7.org/fhir/...` namespace it currently uses) to the corresponding
   `http://terminology.hl7.org/CodeSystem/...` URL.
2. Update every internal reference to that URL inside the IG —
   `ValueSet.compose.include.system`, profile `binding.valueSet`
   targets that resolve through the value set, examples, and narrative
   — to the new canonical.
3. File the corresponding **UTG** ticket against the Vocabulary WG so
   the `CodeSystem` is formally registered, owned, and versioned in
   THO; remove the local copy from the IG once the THO publication is
   in place, and mirror it as `instantiates` only if needed.
4. Apply the same treatment to the sibling tickets in this batch
   (FHIR-35556, -35558, -35552, etc.) so CARIN DIC ships one consistent
   migration rather than a partial one.

#### Justification

This is exactly the canonical-URL discipline the HL7 Vocabulary WG asks
of all IGs: code systems that are not part of FHIR core's defined
namespace belong in THO, both so they can be consumed by IGs other than
the one that authored them and so they have a single authoritative
maintenance point. The 2024-04 comment from `lynn_laakso` explicitly
acknowledges that "there are some codeSystems/ValueSets that we agree
should get added to THO and we will work with Vocabulary WG to get
these addressed" — Disposition A is the materialisation of that
agreement once Vocabulary's THO process is ready.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the cut-off that ticket [FHIR-35552](https://jira.hl7.org/browse/FHIR-35552)
articulates and apply it to FHIR-35557:

- Migrate the `CodeSystem` to THO **only if** it is *not* the target of
  a `required` binding to a `code` data type element. In other words:
  - If the code system is bound `required` to a `code`-typed element
    (where the code values are part of the wire format), keep it
    anchored under `hl7.org/fhir/...` and mark this ticket
    "Persuasive with Mod" with that rationale.
  - Otherwise (the common case — bindings against `Coding`/`CodeableConcept`,
    or non-required strengths), perform the THO migration described in
    Disposition A and file the UTG ticket.

Document the chosen cut-off in the CARIN DIC IG's "Terminology" page so
implementers understand the difference between THO-anchored and
fhir.org-anchored code systems in the IG.

#### Justification

Disposition A treats every CARIN DIC code system the same; in practice
the FHIR/Vocabulary community has converged on the rule above because
`code`-typed required bindings encode the symbol literally on the wire
and changing the system URL has materially different implications than
for `Coding`-typed bindings. Aligning FHIR-35557 with the principled
cut-off in FHIR-35552 lets the workgroup dispose of the entire
Block-Vote-6 batch with one consistent rule rather than ticket-by-ticket
judgement calls, and avoids inadvertently breaking implementers who
treat `code` values as opaque tokens. It also gives Vocabulary WG a
clean, narrowed scope of code systems to onboard into THO.

---

### Disposition C: Decline

#### Proposal

Resolve **Not Persuasive** (or, if procedurally easier, "Considered for
Future Use") with rationale: the ticket as filed does not name the
specific `CodeSystem` to be migrated, the same migration request has
already been deferred twice (2022, 2024) on the explicit basis that
Vocabulary WG has not formalised the THO onboarding process, and the
underlying migration would be a breaking change to the CARIN DIC
canonical surface that is not justified inside an STU update. Close
this ticket and roll the work into the next CARIN DIC normative cycle,
where canonical URL changes are expected.

#### Justification

The defer-and-revert history on this ticket is itself the evidence: two
separate workgroup votes (Roberts/Kissel 11-0-1, twice) have already
concluded that the change is desirable in principle but not actionable
under current Vocabulary WG process. Re-anchoring a published
`CodeSystem.url` is, by FHIR's canonical-identity rules, a breaking
change for implementers who pinned to the original URL — making it an
inappropriate STU-level change without a coordinated migration plan.
Declining (or formally deferring to the normative cycle) acknowledges
that reality and stops the ticket from cycling through the queue
indefinitely.

---

### Recommendation

**Recommended disposition:** B

The two-times-deferred history makes pure Disposition A risky to commit
to inside this ballot cycle — the workgroup has already discovered that
the gating dependency is on Vocabulary WG / UTG process, not on CARIN
DIC's willingness to make the change. Disposition C, however, throws
away the genuine architectural improvement and ignores the 2024-04
acknowledgement that some of these code systems really do belong in THO.

Disposition B is the smallest defensible step forward: it borrows the
principled cut-off already articulated by [FHIR-35552](https://jira.hl7.org/browse/FHIR-35552)
(don't migrate code systems backing `required` bindings to `code`,
migrate the rest), lets the workgroup dispose of the entire Block-Vote-6
batch consistently, and produces a concrete, narrowed set of UTG
tickets that Vocabulary WG can absorb on their formalised timeline.
The first action item under Disposition B should be for the workgroup
to identify which specific `CodeSystem` ballot Comment 7 was referring
to, so the disposition can be written against a named artifact rather
than the abstract template wording on the ticket.
