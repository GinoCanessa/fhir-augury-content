## Summary

The ticket asserts that the canonical (defining) URL for a Value Set in the
US CARIN Digital Insurance Card (DIC) IG should be anchored at
`terminology.hl7.org` (THO) rather than `hl7.org/fhir`, with one exception:
Value Sets that are bound (with required binding) to a `code` data type
should remain anchored at `hl7.org/fhir`. The reporter requests that a
companion UTG ticket be filed to register the affected Value Set(s) in THO
so the canonical URL can be transitioned. The ticket title further notes
"4" suggesting it was one of a numbered series of related comments imported
into Jira.

The ticket was originally resolved as "Considered for Future Use" in
March 2022 (vote 11-0-1) on the basis that the Vocabulary WG had not yet
formalized the process for migrating IG-defined value sets into THO. It
has been re-opened twice (post-STU publication in 2022, and again post
STU 1.1 publication in 2024) and is currently `Triaged` / `Deferred`,
flagged for Block-Vote-6. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were found in the augury index for this
ticket.

## Proposed Dispositions

> Note: Because no GitHub cross-references were surfaced, no repo
> briefings were loaded; the dispositions below are written against the
> CARIN DIC IG and the THO repository as the natural targets for a
> canonical-URL re-anchoring change. If the WG accepts a code-bearing
> disposition, the implementer should request a `repo-analysis`
> briefing for `HL7/carin-digital-insurance-card` and
> `HL7/UTG` before opening PRs.

### Disposition A: Accept As Requested

#### Proposal

For each Value Set defined by the CARIN DIC IG that is **not** bound
(required) to a `code` data type, change the canonical URL from
`http://hl7.org/fhir/us/...` to `http://terminology.hl7.org/ValueSet/...`,
and file a companion UTG ticket against `HL7/UTG` to register each such
Value Set in THO. In the IG, this means updating the affected
`ValueSet-*.json` resources' `url` element, updating any `binding.valueSet`
references in profile StructureDefinitions and search-parameter / capability
artifacts, and adding redirect / deprecation guidance in the IG narrative
that calls out the canonical change for STU2. Value Sets backing
`required` bindings on `code`-typed elements (i.e., enumerations the
parser is expected to know up front) remain at `hl7.org/fhir/us/...` per
the ticket's stated exception.

#### Justification

This matches HL7 governance: terminology artifacts intended to be reused
across IGs belong in THO, while `code`-typed required bindings are part
of the structural contract of the IG and are conventionally co-located
with the spec. Doing both halves together (re-anchor + UTG registration)
avoids dangling canonicals and keeps validators happy. The reporter,
the 2022 motion, and lynn_laakso's revert notes all explicitly
acknowledge that this is the right destination for these Value Sets;
the only previous blocker was Vocabulary WG process maturity.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the canonical-URL change for STU2 and instead publish the affected
Value Sets in THO **with new canonicals**, while leaving the existing
IG-anchored canonicals in place as deprecated aliases for one publication
cycle. Concretely: in `HL7/UTG`, add the Value Sets at
`http://terminology.hl7.org/ValueSet/...`; in the CARIN DIC IG, retain
the existing `http://hl7.org/fhir/us/...` canonicals but mark them
deprecated, add `ValueSet.relatedArtifact` (or narrative) pointing at
the THO equivalent, and switch all internal `binding.valueSet`
references in profiles to the THO URLs. A subsequent STU release would
then remove the deprecated IG-anchored copies.

#### Justification

This honors the ticket's intent without creating a hard breaking change
for existing implementers and validators that have already cached the
`hl7.org/fhir/us/...` canonicals from STU1 / STU1.1. It also de-risks
the UTG migration: the THO publication can land first and be exercised
by the IG before the old canonicals are retired. The trade-off vs.
Disposition A is one extra publication cycle of duplicate value-set
identity, which is a known cost the Vocabulary WG has accepted for
similar migrations.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Modification → Not for this
Release**, on the basis that the IG is past STU publication, the
original 2022 disposition ("Considered for Future Use") still reflects
the reality that the affected Value Sets are not currently used outside
the CARIN DIC IG, and the Vocabulary WG process for accepting
IG-originated Value Sets into THO is the gating dependency rather than
anything the PIE WG can resolve. Recommend the reporter re-file against
the next major version of the IG once the THO intake process is
confirmed.

#### Justification

The Block-Vote-6 / Deferred status, the repeated re-opens without
forward progress, and the absence of any cross-referenced UTG ticket
or Zulip discussion in the augury index all point to this being
blocked on cross-WG process rather than on a substantive content
question. Declining now keeps the IG's STU surface stable and avoids
churning canonicals on a schedule the Vocabulary WG has not signed off
on.

---

### Recommendation

**Recommended disposition:** B

The original ticket is correct in principle — IG-private Value Sets
that aren't backing `code`-typed required bindings really do belong in
THO, and lynn_laakso's revert notes show the WG already agrees on the
destination. But Disposition A as written is a hard canonical break
against artifacts that have been published through STU1.1, and the
ticket has been sitting in `Deferred` precisely because the Vocabulary
WG hadn't formalized intake. Disposition B threads that needle: it
gets the THO registration done now (which is the part the reporter
actually asked for), it re-points the IG's internal bindings so new
implementers see the right canonical, and it keeps the old IG
canonicals alive for one cycle as deprecated aliases so existing
implementers don't trip validators. Disposition C is defensible but
amounts to a third defer on a ticket that has now been re-opened
twice; the WG should either land a concrete migration or acknowledge
that the answer is "never" — B is the concrete migration.
