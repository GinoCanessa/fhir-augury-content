## Summary

The ticket asks the Payer/Provider Information Exchange (PIE) work group to
untangle the QuestionnaireResponse profile dependency chain across four IGs:
US Da Vinci CDex, US Da Vinci DTR, HL7 US Core, and HL7 SDC. Today the CDex
SDC QuestionnaireResponse profile (which only constrains the `Signature`
datatype) is derived from the US Core QuestionnaireResponse profile, which
is in turn derived from SDC QuestionnaireResponse. That chain is broken
because US Core 3.1.1 has no QuestionnaireResponse profile, so the CDex
profile as published is invalid against any IG that pins US Core 3.1.1. In
parallel, DTR derives its `dtr-questionnaireresponse` and
`dtr-questionnaireresponse-adapt` profiles directly from SDC
QuestionnaireResponse and adds CDex-style references and extensions on top.
The result is two divergent dependency paths for the same Burden Reduction
use case and an inability for downstream IGs to depend cleanly on more than
one US Core version.

The reporter proposes (1) re-pointing the CDex QuestionnaireResponse
profile chain so that CDex depends on the DTR QuestionnaireResponse
profiles (rather than on US Core directly), and (2) having DTR formally
record a variance against US Core QuestionnaireResponse 6.1.0 / 7.0.0
before publication. The reporter also flags that the `Signature` datatype
is in flux for FHIR R6, which will affect both SDC and CDex independently
of how this dependency restructuring is resolved.

No related Jira tickets surfaced via cross-reference, but the ticket has
been actively tracked in Da Vinci CDex Zulip vote-tracker threads through
mid-2025; it has been carried as "disposition pending offline discussion
with Burden Reduction team" and has a recorded Jira resolution of
**Persuasive** ("change as proposed; this is a forward breaking change",
resolved 2025-01-22) but is still in **Triaged** status while the change
is staged for a future CDex ballot.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the reporter's two-part proposal to the US Da Vinci CDex IG:

1. **Re-parent the CDex QuestionnaireResponse profile chain onto DTR.**
   In the davinci-ecdx IG, change
   `StructureDefinition/cdex-sdc-questionnaireresponse` so its
   `baseDefinition` is the DTR `dtr-questionnaireresponse` profile (and,
   if a separate adaptive variant is needed, derive a second CDex profile
   from `dtr-questionnaireresponse-adapt`). Drop the direct `US Core
   QuestionnaireResponse` baseDefinition entirely from CDex. Preserve the
   existing CDex constraint on the `signature` element (which is the only
   thing the current CDex profile actually adds). Update the CDex IG
   `dependencies` block to add `hl7.fhir.us.davinci-dtr` (pinned to a
   ballot-aligned version) and remove the unnecessary `hl7.fhir.us.core`
   pin if no other CDex artifact still requires it. Update
   `signatures.html#cdex-signature-profiles` to describe the new
   inheritance and call out that the profile no longer transits US Core.
2. **Have DTR record a variance against US Core QuestionnaireResponse
   6.1.0 and 7.0.0** before its next publication, documenting that DTR's
   QuestionnaireResponse profiles intentionally derive from SDC rather
   than US Core, and that downstream IGs (CDex) inherit through DTR.

Mark the change as forward-breaking in the CDex change log (the Jira
resolution note already records this), and stage it for the same January
ballot cycle that PIE has already accepted CDex into (per the 2025-07-01
Zulip).

#### Justification

This is exactly what the reporter requested, what the Jira resolution has
already recorded ("change as proposed"), and what the Zulip vote-tracker
posts have been waiting on. The current CDex profile is demonstrably
invalid in the US Core 3.1.1 dependency context, so leaving it as-is
exports a known broken artifact. Re-parenting onto DTR yields a single
QuestionnaireResponse lineage for both Da Vinci IGs that share the
Burden Reduction use case (DTR â‡’ SDC, CDex â‡’ DTR â‡’ SDC), removing CDex's
direct coupling to US Core versioning. DTR formally noting a variance
against US Core QuestionnaireResponse 6.1.0/7.0.0 makes the deliberate
divergence visible to implementers reviewing the IG dependency graph
rather than implicit. The forward-breaking nature is acceptable because
PIE has already voted to take CDex into a substantive ballot for exactly
this class of change.

---

### Disposition B: Alternative Approach

#### Proposal

Address the broken dependency without binding CDex's QuestionnaireResponse
profile to DTR. Two coordinated actions:

1. **Re-parent the CDex QuestionnaireResponse profile directly onto SDC
   QuestionnaireResponse** (not DTR). In `cdex-sdc-questionnaireresponse`,
   set `baseDefinition` to
   `http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-questionnaireresponse`
   and drop the US Core ancestor entirely. Keep the existing constraint on
   the `signature` element. This restores validity (SDC actually defines
   the profile, unlike US Core 3.1.1) without making CDex's IG dependency
   graph coupled to DTR's release cadence.
2. **Coordinate (do not enforce) alignment with DTR** by publishing CDex
   guidance that says implementers building Burden-Reduction stacks
   SHOULD use the DTR QuestionnaireResponse profiles where DTR also
   applies, and reference DTR's profiles by URL in the CDex narrative
   rather than by `baseDefinition`. DTR still records its variance
   against US Core 6.1.0/7.0.0 as in Disposition A.

#### Justification

Disposition A buys profile-tree alignment at the cost of a hard CDex â†’
DTR IG dependency. CDex is consumed by use cases that don't involve DTR
at all (general clinical-data exchange via Task / `$submit-attachment`),
so coupling its QuestionnaireResponse profile to DTR pulls the entire
DTR package â€” including unrelated CQL, CDS Hooks, and PAS dependencies â€”
into every CDex implementer's IG resolver. Going directly to SDC keeps
the dependency surface minimal, eliminates the US Core 3.1.1 invalidity
just as effectively, and still leaves room for a DTR-aligned
implementation pattern documented in narrative. It also avoids creating a
release-ordering constraint where every CDex publication must wait on a
compatible DTR publication. The trade-off is that derived CDex resources
will not automatically inherit DTR's reference- and extension-level
constraints, which is the exact thing Disposition A is trying to
preserve.

---

### Disposition C: Decline

#### Proposal

Do not change the CDex QuestionnaireResponse profile dependency chain in
this cycle. Mark the ticket as deferred until after the FHIR R6 Signature
datatype change settles, on the grounds that any restructuring done now
will need to be revisited as soon as R6 lands.

#### Justification

The ticket itself flags that "Signature datatype is in flux in FHIR R6
and may affect both SDC and CDex." The only thing the current CDex
QuestionnaireResponse profile actually adds is a constraint on
`Signature`. Restructuring the profile inheritance now, only to have to
restructure it again to absorb the R6 Signature changes, doubles the
breaking-change cost imposed on implementers. Deferring also matches the
reporter's own October 2024 comment ("defer until after STU 2.1 is
published") and the Zulip status on 2025-07-08 ("â€¦disposition pending
offline discussion with Burden reduction team"), both of which suggest
the work group has not actually converged on the restructuring. Declining
in this cycle and revisiting once R6 Signature stabilises is the lower-risk
option.

### Recommendation

**Recommended disposition:** A

The Jira resolution is already recorded as "Persuasive â€” change as
proposed", PIE has already voted to take CDex into a substantive,
breaking ballot, and the most recent Zulip update confirms that the only
outstanding gate is an offline alignment conversation with Burden
Reduction (Jeff, Lloyd) â€” not an unresolved technical disagreement. The
status quo ships an objectively invalid profile (US Core 3.1.1 has no
QuestionnaireResponse for `cdex-sdc-questionnaireresponse` to derive
from), so doing nothing this cycle (Disposition C) leaves a known
defect in a published IG. Disposition B is technically cleaner in
isolation, but it discards the stated goal of harmonising the Burden
Reduction stack between CDex and DTR â€” which is precisely the problem

a known follow-up item in the CDex change log, gives implementers a
single coherent QuestionnaireResponse lineage now and leaves the
Signature datatype change as a separately scoped future ticket against
both SDC and CDex.
