## Summary

The ticket asks the CARIN Digital Insurance Card (DIC) IG authors to relocate
the canonical URL of (at least one) ValueSet from the IG's own `hl7.org/fhir/...`
namespace to the central HL7 terminology authority at
`terminology.hl7.org` (THO), and to file a corresponding UTG (Unified Terminology
Governance) request so the ValueSet is registered and maintained in THO. The
title further qualifies the scope: only ValueSets that are bound to a `code`
data type with a *required* binding should be anchored under
`hl7.fhir.org` / `hl7.org/fhir`; everything else should live in THO. The
request was originally deferred to "Considered for Future Use" pending the
Vocabulary WG formalising the THO contribution process, and was subsequently
re-opened post-STU 1.1 publication so the work group could revisit it on a
future ballot. There are no cross-referenced Jira tickets, Zulip threads, or
GitHub items linked to this ticket in Augury, so the request stands on its own
text.

## Proposed Dispositions

> Note: the following dispositions cannot reference specific repo paths
> because no GitHub cross-references exist for this ticket and the CARIN
> Digital Insurance Card IG is not surfaced in any repo briefing in this
> workspace. The dispositions are framed at the spec/governance level and
> name CARIN DIC and THO as the affected artifacts; if the work group
> wants file-level direction, a follow-up `repo-analysis` of the CARIN
> DIC IG repo (and any related UTG repo) would be needed first.

### Disposition A: Accept As Requested

#### Proposal

For the next published version of the CARIN Digital Insurance Card IG, audit
every ValueSet (and supporting CodeSystem) currently anchored under the IG's
own `hl7.org/fhir/us/insurance-card/...` canonical namespace and apply the
following rule:

- Keep the canonical under the IG's namespace **only** when the ValueSet is
  bound with `strength = required` to an element whose data type is `code`.
- For every other ValueSet (and any CodeSystem the IG defines that is not
  trivially scoped to a single `code` element), open a UTG ticket against
  Vocabulary WG to register the artifact in `terminology.hl7.org`, then update
  the IG to reference the THO canonical (`http://terminology.hl7.org/...`)
  instead. Retire the IG-local copy or leave a deprecation note pointing at
  the THO canonical.

Track the audit in a single child ticket per moved ValueSet so each move is
balloted explicitly.

#### Justification

This is the alignment the reporter is asking for and it matches the long-
standing HL7 convention that terminology shared (or potentially shared)
across IGs lives in THO under Vocabulary WG stewardship rather than being
"trapped" inside an IG's canonical namespace. The Vocabulary WG has, since
the original 2022-03-08 deferral, formalised the UTG submission process, so
the rationale that originally blocked the change ("they have formalized the
process") no longer holds — that is precisely what the two re-opens by
lynn_laakso were anticipating. Anchoring shared terminology in THO also
avoids duplicate canonical URLs if other Payer/Provider IGs (Da Vinci, CARIN
BB, etc.) later need the same value set.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the THO-anchoring rule prospectively rather than retroactively.
Concretely:

- Leave existing CARIN DIC ValueSet canonicals **as-is** in the IG's own
  namespace to preserve URL stability for STU 1.x implementers.
- For each existing ValueSet that should logically live in THO, register a
  THO ValueSet that mirrors the IG-local one, and add a non-normative note in
  the IG documentation pointing implementers at the THO canonical as the
  preferred reference going forward.
- Apply the "THO unless required-bound to `code`" rule strictly to any *new*
  ValueSets added in CARIN DIC 2.0 / future versions.
- Plan the canonical re-pointing for the next major (breaking) release.

#### Justification

Moving a published ValueSet's canonical URL is, in practice, a breaking
change for downstream Payer/Provider implementations: validators, CapabilityStatements,
and Questionnaire / SearchParameter references that hard-code the IG canonical
will silently stop matching when the URL changes. STU 1.1 has already shipped
and is being deployed by carriers; PIE has historically been protective of
that stability. This disposition honours the reporter's intent (terminology
should live in THO) without breaking the URLs implementers have wired into
production today, and it matches the deferred / future-use stance that the
work group has twice approved (and twice administratively reverted only to
keep the ticket alive). The trade-off versus Disposition A is some
near-term duplication between the IG-local and THO canonicals.

---

### Disposition C: Decline

#### Proposal

Resolve the ticket as *Not Persuasive* (or *Not Persuasive with Mod*) on the
grounds that the request is a generic governance/style observation rather
than a specific defect in CARIN DIC, and the IG already follows the HL7
authoring guidance in effect when STU 1.1 was balloted. Encourage the
reporter to file a single, scoped ticket per specific ValueSet that should
move to THO, with the candidate THO canonical and a UTG ticket reference
attached.

#### Justification

The ticket as written does not identify which ValueSet(s) should move; the
description was imported from a single ballot comment ("Comment 3") and
omits the actual artifact name(s). Re-balloting URL changes for an entire IG
on the basis of a generic principle, twice deferred by the work group, risks
breaking implementer integrations without a concrete scope. The procedural
mechanism the reporter is asking the WG to invoke — opening a UTG ticket —
is something Vocabulary WG already accepts directly; PIE does not need to
hold a ticket open to track that. Declining and asking for one
ValueSet-specific ticket per move keeps the change auditable and lets
each move be balloted on its own merits.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying point is correct and aligns with HL7's terminology
governance direction (THO is the right home for shareable ValueSets, and the
Vocabulary WG has, since the original 2022 deferral, formalised the UTG
submission process — which is the gating condition the two prior PIE motions
explicitly called out). At the same time, CARIN DIC STU 1.1 is in real-world
deployment by US payers, and silently changing the canonical URL of a
published ValueSet is a breaking change that downstream tools (validators,
CapabilityStatements, Questionnaires) cannot detect from the wire. Disposition
B preserves the existing canonicals for STU 1.x stability, registers the
THO-side canonicals so other IGs can adopt them now, and commits the work
group to applying the "THO unless required-bound to a `code`" rule from the
next major release forward. Disposition A is the right *destination* but
would be better landed at a major-version boundary; Disposition C
under-serves the reporter's substantive point and ignores that the
Vocabulary process objection from 2022 has since been resolved.
