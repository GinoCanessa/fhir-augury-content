## Summary

The ticket asks that the canonical defining URLs for value sets in the
US CARIN Digital Insurance Card (DIC) IG be anchored under
`terminology.hl7.org` (THO) rather than `hl7.org/fhir`, with a
companion UTG ticket created to register the value set in THO. The
title further qualifies the intent: only value sets bound (`required`)
to a `code` data type should remain anchored in
`hl7.fhir.org`/`hl7.org/fhir` namespaces; everything else belongs in
THO. The ticket is Comment 5 of a multi-comment block that was imported
by Ron G. Parker, and it is one of a numbered family of near-identical
tickets (`.1` through `.5` and beyond) raised against multiple PPIE /
Da Vinci / CARIN IGs at roughly the same time.

The cross-reference index in FHIR Augury reports **no items** linking
to `FHIR-35555` (no other Jira issues, no Zulip threads, no GitHub PRs
or commits, no Confluence pages — Confluence is not configured in this
environment). Keyword similarity does identify the parallel tickets
(`FHIR-35551`, `FHIR-35552`, `FHIR-35553`, `FHIR-35554`,
`FHIR-35404`, `FHIR-35490`, `FHIR-35423`, `FHIR-35612`, `FHIR-38529`,
`FHIR-38530`) as essentially the same change request applied to other
value sets / IGs, so any disposition here should be coordinated with
that batch.

## Proposed Dispositions

Because there are no concrete artifact references in the ticket body
(it only describes "the Value Set" generically as "Comment 5"), the
dispositions below treat the request at the policy level — what to do
about the URL anchoring of CARIN DIC value sets that fit the stated
criterion (not bound `required` to a `code` data type) — and call out
that the implementation step requires identifying the specific
ValueSet artifact(s) intended by Comment 5 before any change can land.

### Disposition A: Accept As Requested

#### Proposal

Re-anchor the in-scope CARIN DIC ValueSet's canonical URL from
`http://hl7.org/fhir/us/insurance-card/ValueSet/<id>` to
`http://terminology.hl7.org/ValueSet/<id>`. Concretely:

1. PPIE identifies the specific ValueSet (the artifact intended by
   "Comment 5" in the imported source) — most likely a DIC value set
   that is **not** bound `required` to a `code` data type.
2. File a companion UTG ticket against `HL7/UTG` requesting registry
   creation of that value set under `terminology.hl7.org`, including
   the canonical URL, version, copyright, steward (PPIE), and the
   underlying CodeSystem(s) it composes.
3. Once UTG accepts, update the DIC IG sushi-config / value set
   resource to use the new THO canonical, add a redirect note in the
   IG, and record the prior canonical for traceability.
4. Update any binding references in DIC profiles to point at the new
   canonical.

#### Justification

This is the literal ask in the description and matches HL7's stated
direction that terminology artifacts should live in THO so they can be
reused across IGs and version-managed independently of any single IG
release. Keeping the value set in the IG namespace makes cross-IG
reuse awkward and forces every IG that needs it to either copy the
definition or take a hard dependency on the DIC IG. The `Highest`
priority and the ticket's repeated re-opening after each STU
publication indicate this has remained material to PPIE for several
years.

---

### Disposition B: Alternative Approach

#### Proposal

**Defer the per-ticket relocation in favor of a coordinated PPIE batch
move.** Rather than acting on `FHIR-35555` in isolation, PPIE block-
votes the entire `Block-Vote-6` family (this ticket plus
`FHIR-35551`–`FHIR-35554`, `FHIR-35404`, `FHIR-35423`, `FHIR-35490`,
`FHIR-35612`, `FHIR-38529`, `FHIR-38530`, and any others Vocabulary /
PPIE find against the DIC and C4BB IGs) as a single change to
"Persuasive with Modifier", with the modifier stating:

- PPIE will produce a complete inventory of DIC ValueSets that meet the
  "not bound `required` to a `code` data type" criterion, with current
  and proposed canonicals.
- PPIE will file one consolidated UTG ticket per CodeSystem/ValueSet
  cluster (not one per FHIR-* ticket), so Vocabulary processes the
  family as a batch under whatever onboarding process they have
  formalized.
- DIC IG will only re-anchor in its **next** ballot cycle, with a
  documented redirect from the old IG-anchored URL.

#### Justification

The comment history shows the underlying blocker is not technical
disagreement; it is that Vocabulary WG had not formalized a THO
onboarding process at the time of the prior deferrals. Acting on a
single value set in isolation will recreate the same back-and-forth
with Vocabulary that produced the 2022 and 2024 deferrals. A batch
disposition gives Vocabulary one well-formed work item, lets PPIE
publish a single migration note, and avoids the risk of inconsistent
canonicals across the DIC and C4BB families. It also addresses the
practical gap that the ticket itself never names the ValueSet —
forcing the workgroup to enumerate the in-scope artifacts up front.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Decline / "Not Persuasive". Close `FHIR-35555` (and the parallel
Comment-1–6 siblings) on the basis that:

- The ticket originated as an imported "Comment 5" from a 2022 review
  cycle and never named a specific ValueSet, making the request
  un-actionable as filed.
- Two prior PPIE votes (2022-03-08 and 2022-08-03, both 11-0-1)
  already concluded "Considered for Future Use" pending Vocabulary's
  process, and that process still gates the work.
- DIC publishes ValueSets in its own IG namespace consistently with
  other CARIN IGs that are currently published; relocating one value
  set without relocating the rest would create cross-IG inconsistency.

Direct the requester (Joan Harper / Ron G. Parker) to refile a single
focused ticket once Vocabulary publishes a documented THO onboarding
process, naming the specific ValueSet(s) and CodeSystem(s) involved.

#### Justification

Defensible on the grounds that the ticket is too generic to act on,
that its dependency (a formal Vocabulary THO onboarding process) is
external and unresolved, and that two prior PPIE votes already chose
not to act. Declining and asking for a focused refile is a cleaner
disposition than a third "deferred" cycle that everyone knows will
be re-opened at the next STU publication.

---

### Recommendation

**Recommended disposition:** B (Persuasive with Modifier — coordinated
PPIE batch move).

The ticket's history makes Disposition A risky in practice: PPIE has
twice tried to act and twice deferred because Vocabulary's THO
onboarding process was not ready, and the ticket as written does not
name the ValueSet, so a literal "accept" would still require an
additional discovery step before any change could land. Disposition C
is defensible but throws away the substantive agreement reflected in
the comment trail (PPIE *does* agree these belong in THO) and would
simply be re-opened at the next STU cycle, as it was in 2022 and
2024. Disposition B preserves the substantive intent, ties the work
to the parallel `Block-Vote-6` siblings (which keyword similarity
identifies very strongly), and forces the missing prerequisites — a
named ValueSet inventory and a coordinated UTG submission — to be
produced before code changes are made. That is the path most likely
to actually close the ticket family rather than defer it again.
