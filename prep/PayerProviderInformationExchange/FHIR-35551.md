## Summary

FHIR-35551 is a CARIN Digital Insurance Card (DIC) change request asserting
that value set canonical URLs defined by the IG should be anchored under
`terminology.hl7.org` (THO) rather than `hl7.org/fhir`, so that the value
sets live in the cross-IG terminology registry maintained by the Unified
Terminology Governance (UTG) process. The submitter further notes that
only value sets used as a required binding for a FHIR `code` data type
element should remain anchored under an IG's own `hl7.org/fhir/...` URL
namespace; everything else should move to THO. The ticket asks for a
companion UTG ticket to be opened to register the DIC value set(s) in THO
once the URL move is agreed. A previous vote resolved the item
"Considered for Future Use" (2022-03-08, 11-0-1), but that resolution was
reverted and the ticket was reopened after STU publication with a
standing intent to address THO-hosted value sets as the Vocabulary WG
formalizes the onboarding process. The current Jira resolution field is
already marked "Persuasive," indicating the workgroup's prior direction
was to accept the request in principle; this review is about confirming
and operationalizing that disposition.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned for this key.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the canonical URL of each DIC-defined value set whose binding is
**not** a required binding to a `code` data type from the IG's
`http://hl7.org/fhir/us/insurance-card/ValueSet/...` namespace to
`http://terminology.hl7.org/ValueSet/...`, and open a companion UTG
ticket to register those value sets in THO as part of the next DIC
publication. Leave any value set that is used as a required binding on
a FHIR `code` data type element in the IG's own URL namespace, in line
with FHIR guidance. Update every StructureDefinition binding in DIC
that currently points at the old IG canonical to the new THO canonical,
and add a migration note plus redirect/alias in the IG's value-set
history page so existing implementers are not silently broken.

#### Justification

This directly implements the FHIR convention cited in the ticket title:
only value sets used by required bindings on a `code` element belong in
an IG's own URL namespace, while shared / reusable value sets belong in
THO so they can be governed via UTG and reused across IGs. The prior
DIC vote already recorded a "Persuasive" resolution, and the reopen
comments (lynn_laakso, 2022-07-13) record an explicit workgroup
agreement that "there are some codeSystems/ValueSets that we agree
should get added to THO." Accepting as requested aligns DIC with
Da Vinci / CARIN peers that have already migrated reusable value sets
to THO and sets up downstream implementers for cleaner cross-IG reuse.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the value set canonical URLs in the DIC IG namespace for the
current STU, but publish a formal THO mirror (same OID / same codes,
different canonical) and mark the IG-namespaced value sets as
"superseded-by" the THO canonical in a future DIC release once UTG's
onboarding process is formalized. In parallel, tighten DIC's own
authoring guidance so that any *new* value set added going forward is
created directly in THO and referenced from DIC by its THO canonical;
only legacy value sets remain dual-published during the transition.
Still open the UTG registration ticket the submitter asked for, but
scope it as "register the THO mirror," not "move the canonical."

#### Justification

The 2022 deferral was specifically because the Vocabulary WG had not
yet formalized the THO onboarding process; that concern has not fully
evaporated, and moving a published IG's value set canonicals is a
breaking change for anyone who has already bound to the IG URLs
(payers, members' apps, profile authors). A mirror-then-supersede path
achieves the same long-term end state (reusable value sets live in
THO) while avoiding an immediate breaking change to published
artifacts, and it matches the pattern several HL7 IGs have used when
migrating terminology under UTG. It also respects the "required
binding to a `code`" carve-out without requiring a global audit of
every DIC binding in the current publication cycle.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive With Mod" / "Not Persistent,"
keeping the DIC-defined value sets anchored at
`http://hl7.org/fhir/us/insurance-card/ValueSet/...` and not opening a
UTG registration. The rationale would be that the DIC value sets are
sufficiently DIC-specific that hosting them in THO provides little
reuse value, that moving canonical URLs of an already-published STU is
a breaking change for conformant implementations, and that the FHIR
"required binding to a `code`" convention is a guideline for future
IGs rather than a mandate to relocate existing published artifacts.

#### Justification

This disposition is defensible only on the narrow grounds of
publication stability: DIC has already shipped, and changing canonical
URLs of published value sets would invalidate every profile, example,
and payer implementation that dereferences those URLs. It would also
mirror the original 2022-03-08 vote outcome (Considered for Future
Use, 11-0-1), which leaned on the fact that the Vocabulary WG's THO
onboarding process was not yet formalized.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible path for a ticket that has
already been voted "Persuasive" in spirit but was reopened precisely
because the operational mechanism (UTG onboarding) was not ready at
the time. Accepting literally (Disposition A) is directionally correct
but risks an avoidable breaking change to published DIC canonicals and
asks the Vocabulary WG to absorb a bulk migration in a single cycle.
Declining (Disposition C) walks back the workgroup's own reopen action
and leaves DIC out of step with the FHIR convention the ticket cites.
A mirror-then-supersede plan — register the value sets in THO via UTG
now, make new DIC value sets THO-native going forward, and switch
DIC's canonical references to the THO URLs in the next substantive
DIC release — fulfils the submitter's intent, honours the
"Persuasive" vote, and protects existing implementers during the
transition. The one carve-out called out in the ticket (value sets
bound as required to a `code` element stay in the IG namespace) should
be written into the disposition text so the UTG ticket is scoped
correctly from day one.
