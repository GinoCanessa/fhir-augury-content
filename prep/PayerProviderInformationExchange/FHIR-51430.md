## Summary

The reporter (Eric Haas) flags that the LOINC value set
`http://loinc.org/vs/valid-hl7-attachment-requests` (canonical published at
`https://fhir.loinc.org/ValueSet/valid-hl7-attachment-requests`) does not
resolve when expanded inside the Da Vinci CDex IG. The request is two-fold:
(1) follow up with the IG publisher tooling so that the LOINC URI can be
resolved/expanded directly, and then (2) replace the locally maintained
CDex and PDex attachment-code value sets with the official LOINC URI. As a
fallback, if the publisher tooling cannot be made to resolve the LOINC value
set, the CDex and PDex value sets should be refreshed from the latest LOINC
publication, or moved to THO (Terminology.HL7.Org) so they are centrally
maintained rather than duplicated per IG.

The cross-referenced Zulip tracker post for the PIE vote on 2025-07-08
explicitly notes that this ticket is **"waiting for IG publisher updates to
directly access LOINC's terminology server before disposing"** — i.e., the
work group has acknowledged it but deferred a final disposition pending
upstream tooling support. There are no other linked Jira tickets and no
GitHub cross-references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the CDex- and PDex-defined attachment-code value sets with a direct
binding to the LOINC-hosted canonical
`http://loinc.org/vs/valid-hl7-attachment-requests` (resolvable at
`https://fhir.loinc.org/ValueSet/valid-hl7-attachment-requests`). Concretely:
in the CDex IG, change the `cdex-attachment-request` /
`requesting-attachments` value-set bindings (and the equivalent PDex
attachment-code bindings) so they reference the LOINC ValueSet by canonical
URL rather than re-declaring the membership locally. Coordinate with the IG
publisher / FHIR tooling team so that publisher builds resolve the LOINC
canonical against the LOINC FHIR terminology server during IG
generation/validation. Mark FHIR-51430 **Persuasive — pending IG publisher
support** and gate merge on the publisher actually being able to expand the
LOINC value set in a CI build.

#### Justification

Using the LOINC-published value set directly is the FHIR-preferred pattern
for terminology that is owned and maintained by an external authority: it
avoids drift between the LOINC publication cadence and the IG, eliminates
duplicate maintenance in CDex and PDex, and ensures implementers see the
canonical LOINC URI (which is what trading partners are expected to
recognize). The work group has already implicitly endorsed this direction —
the tracker note on 2025-07-08 says the only blocker is publisher /
terminology-server tooling, not the design choice itself. This option
remains the cleanest end-state and should be the target disposition once the
upstream tooling work (tracked in the `#IG creation > LOINC valuesets out
of the box?` Zulip thread) lands.

---

### Disposition B: Alternative Approach — Mirror to THO Now

#### Proposal

Without waiting on IG-publisher / LOINC-terminology-server integration,
publish a stable, IG-resolvable copy of the LOINC attachment-request value
set under THO (Terminology.HL7.Org) — e.g., a THO ValueSet whose compose
references LOINC by code system, drawn from the latest LOINC publication,
with provenance pointing at
`http://loinc.org/vs/valid-hl7-attachment-requests`. Then have CDex and
PDex bind to the THO ValueSet canonical (single source of truth) instead of
each maintaining their own copy. Schedule a refresh of the THO ValueSet
each LOINC release. Mark FHIR-51430 **Persuasive with Modification**.

#### Justification

This addresses the underlying complaint — duplicated, drifting attachment
value sets across CDex and PDex with no resolvable LOINC URI — without
waiting for IG-publisher or LOINC-terminology-server changes that are out
of the WG's control. THO is the HL7-sanctioned home for shared
terminology, the IG publisher already resolves THO canonicals reliably,
and consolidating to a single THO artifact removes the CDex/PDex
divergence today. The trade-off vs. Disposition A is that the canonical
URI implementers see is the THO one rather than the LOINC one, and the
WG takes on a (small) recurring obligation to refresh the THO ValueSet on
each LOINC release. The reporter explicitly listed "move to THO" as an
acceptable fallback in the original ticket.

---

### Disposition C: Decline / Defer

#### Proposal

Mark FHIR-51430 as **Considered — No Change** (or **Deferred**) on the
grounds that the root cause is IG-publisher tooling and the LOINC FHIR
terminology server, not a CDex/PDex specification defect. Leave the
existing CDex and PDex attachment value sets in place and track the
publisher-side work in the `#IG creation > LOINC valuesets out of the
box?` Zulip thread; reopen / re-disposition once the publisher can
resolve `http://loinc.org/vs/valid-hl7-attachment-requests` natively.

#### Justification

The current value sets work for implementers; the failure mode the reporter
describes is a documentation/tooling issue (the LOINC URI not resolving in
the rendered IG), not a clinical or interoperability bug in CDex/PDex
content. Spending CDex spec-change capacity to swap the binding before the
publisher can actually expand the LOINC URI risks publishing an IG whose
own validator can't expand its own bindings — a worse outcome than the
status quo. Deferral is consistent with what the WG already signaled in
the 2025-07-08 PIE tracker post.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — Mirror to THO Now).

Disposition A is the right *long-term* answer and matches the reporter's
preferred outcome, but the work group has already acknowledged on the
2025-07-08 PIE tracker that the IG publisher cannot yet resolve the LOINC
canonical, so adopting A in this ballot cycle would either ship an IG with
unresolvable bindings or require a second disposition pass once tooling
catches up. Disposition C kicks the can on the real complaint —
duplicated, drifting attachment value sets across CDex and PDex — and
leaves the reporter's concern unaddressed. Disposition B closes the
duplication / drift problem now (one THO ValueSet, two IGs binding to it),
keeps the IG publishable today, and leaves a clean migration path: when
the LOINC FHIR terminology server / IG publisher integration lands, the
THO ValueSet can be retired in favor of the direct LOINC canonical (i.e.,
move to A) with a single coordinated change. That sequencing is the lowest-
risk path that still respects the reporter's intent.
