## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the canonical URL of the affected CARIN DIC Value Set from
`hl7.org/fhir/...` to `terminology.hl7.org/ValueSet/...`, deprecate
the old canonical with a tombstone redirect in the IG, and open a
companion UTG ticket against the THO repository to import / register
the Value Set under THO governance. The CARIN DIC IG would then
reference the THO canonical from any binding rather than re-defining
the Value Set in the IG. Coordinate the same change across the
sibling tickets (FHIR-35551/.3/.4/.5, FHIR-35487/.4/.6, FHIR-35578/
.609/.615, FHIR-38530, and the named ADA / AHA NUBC variants) as a
block-vote so the migration is consistent.

#### Justification

This is the long-standing FHIR / Vocabulary WG direction: Value Sets
that are not specific to a single IG — particularly those bound from
a `code` data type with required binding, and those wrapping external
code systems (ADA UNS, AHA NUBC, etc.) — belong in THO, where they
get a single canonical identity, can be reused by any IG, and follow
THO's review and versioning lifecycle. The ticket's own description
spells out exactly this expectation, and the comment history confirms
that the Vocabulary WG has agreed in principle ("we agree should get
added to THO"). Now that STU 1.1 has shipped and the most recent
2025-09-23 comment explicitly asks that the STU publication status be
re-checked, the deferral rationale ("Vocabulary WG hasn't formalized
the process") is the only remaining blocker; if Vocabulary WG has
since formalized that process, the ticket can be acted on as written.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the Value Set's `ValueSet.url` as currently published for STU 1.1
(to preserve canonical stability for implementers who have already
referenced it), but (a) add a `ValueSet.identifier` carrying the
target THO canonical, (b) create the THO-side Value Set in UTG as a
mirror that points back to the IG's canonical, and (c) flip the
canonical only in the next major version of CARIN DIC, with a
documented migration note. The IG keeps shipping today's URL while
THO becomes the system of record going forward.

#### Justification

Changing a published canonical URL is a breaking change for any
implementer who has stored references against the old URL — payer
systems exchanging Coverage / DIC artifacts in production are exactly
the audience most likely to have done so. CARIN's value proposition
depends on stable, well-known identifiers, so the migration should be
phased rather than retroactive. This approach honours the THO
direction without breaking deployed systems and gives Vocabulary WG /
UTG room to complete the formal import process the prior deferrals
were waiting on. Trade-off vs. Disposition A: the IG carries a
short-term canonical-duplication wart, but no implementer is broken.

---

### Disposition C: Decline

#### Proposal

Resolve as "Considered for Future Use" again (or "Persuasive with
Mod" deferring to a future major release), with the rationale that
the Vocabulary WG's process for IG-to-THO Value Set migration is
still not formalized in a way the workgroup can act on
unilaterally, and that any unilateral canonical change risks
breaking deployed CARIN DIC implementations. The ticket would remain
open against a future major-version milestone of CARIN DIC.

#### Justification

This disposition is the de-facto status quo: the ticket has already
been disposed of as "Considered for Future Use" twice and re-opened
twice, each cycle pivoting on whether Vocabulary WG / UTG has
formalized the migration process. If the answer to the 2025-09-23
"check status of STU publication" comment is that THO governance
still hasn't shipped a turn-key import path for IG-owned Value Sets,
then declining-with-deferral is honest: PIE can't complete the
change without UTG, and merging an unbacked canonical change would
just produce a dangling reference. Declining keeps the ticket
visible without committing to action that depends on another WG.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible middle path. Disposition A is
correct in principle and aligns with the WG's stated direction, but
unilaterally swapping a published canonical URL on an STU-1.1
artifact is exactly the kind of change that erodes trust with payer
implementers — and the ticket gives no evidence that the Vocabulary
WG's UTG-import workflow has been formalized to the point where PIE
can drive the THO side cleanly. Disposition C is honest about the
process gap but produces no forward motion and effectively converts
the block-vote into "kick the can again" for a third cycle. B
captures what the workgroup actually wants — THO becomes the system
of record — while honouring the canonical-stability obligation the
IG owes its implementers, and it gives PIE a concrete deliverable
(identifier addition, UTG mirror) that can land now, with the
breaking canonical flip queued for the next major version. As a
practical matter, B should be applied uniformly across the entire
sibling block-vote set (FHIR-35551/.3/.4/.5, FHIR-35401/.402/.404/
.405, FHIR-35487/.489/.491, FHIR-35578/.609/.615, FHIR-38530) so the
migration story is coherent rather than per-Value-Set ad hoc.
