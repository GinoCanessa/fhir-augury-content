## Summary

The reporter asks the US Vital Signs with Qualifying Extensions IG (CIMI) to
allow recording a blood-pressure cuff's physical size as a numeric measurement
(e.g., "22 – 42 cm") in addition to — or instead of — only a coded terminology
concept such as small/medium/large. The motivation is that consumer BP device
manufacturers (Omron, A&D, Withings) do not consistently expose discrete
T-shirt-style cuff-size codes, but they do publish numeric circumference
ranges. The reporter proposes capturing the numeric range and then mapping it
to the existing coded representation via a published reference. The ticket
also raises a secondary design question for the workgroup: when a numeric
upper/lower size with a UCUM unit is supplied, should the equivalent coded
concept (from `bpCuffSizeVS`) also be required, or remain optional?

There are no Jira cross-references and no linked GitHub artifacts for this
ticket. Community context comes from a single Zulip thread (see below) where
implementers raised essentially the same question and pointed at three
existing recording mechanisms (Device.property, the SNOMED-based
`bpCuffSizeVS`, and LOINC 8358-4).

## Details

**Description:**

> In addition to capturing the cuff size as terminology concept (e.g. small,
> medium, large, etc.), we would also like to, optionally, have the ability
> to capture the specified cuff size (e.g. 22 - 42 cm).
>
> It has been our experience that BP device manufacturers like Omron, A&D,
> and Withings, do not always, in a standardized way, identify their cuff
> sizes as discrete sizes. To work this issue, we ask users to specify their
> cuff size in centimetres and the system then uses a published reference to
> map the cuff size into terminology concepts.
>
> Additional consideration - if someone specifies the size of the cuff as a
> numeric upper and lower size value with a unit of measure, should we also
> mandate the inclusion of the terminology concept representation of that
> size?

Daniel Tam (2021-10-08) opened the thread asking how implementers are
recording BP cuff size against the US Core BP profile, and listed three
existing options: (1) attaching cuff size as `Device.property.type` /
`Device.property.valueQuantity` on the referenced Device, (2) using the
SNOMED-based `bpCuffSizeVS` value set published in the US Vitals IG, or
(3) using LOINC code `8358-4` (cuff size, part of LOINC panel `35094-2`).
Øyvind Aassve pointed at a Norwegian national-profile extension on
Simplifier that defines a coded cuff-size value set as an Observation
extension. Ardon Toonstra noted that the Dutch HL7 information model and
in-development R4 BP profile also models a "Cuff" concept directly on the
observation. Jay Lyle (CIMI) closed the visible portion of the thread by
confirming "CIMI has addressed several vitals qualifiers here", positioning
the Qualifying-Extensions IG as the natural home for this concept. No
single mechanism reached consensus; the thread surfaces three viable
locations (Device.property, an Observation extension, and a component with a
coded answer) but does not converge on which is canonical.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an optional cuff-size-quantity capability to the US Vital Signs with
Qualifying Extensions IG that allows recording the cuff's numeric size range
alongside (not in place of) the existing coded representation. Concretely,
define a Qualifying Extension — for example
`StructureDefinition/qe-bp-cuff-size-range` — with `value[x]: Range` bound
to UCUM `cm` (and permitting `[in_i]`), applied at
`Observation.component.extension` on the blood-pressure component(s) of the
US Core / US Vitals BP profile. The existing coded cuff-size mechanism (the
`bpCuffSizeVS` SNOMED value set) remains as the primary qualifying extension
and is **optional** when the numeric range is supplied; conversely, the
numeric range is optional when the coded value is supplied. Document a
non-normative mapping table (small / medium / large / X-large adult →
circumference ranges) so implementers can derive one form from the other.

#### Justification

This is exactly what the reporter and the consumer-device community asked
for: device manufacturers publish numeric ranges (e.g., 22–42 cm), not
SNOMED codes, so a `Range` extension lets that data round-trip without
forcing a lossy mapping at capture time. Using a Qualifying Extension on
`Observation.component` aligns with the IG's existing pattern (Jay Lyle
confirmed on Zulip that CIMI is the right home for "vitals qualifiers"),
and a UCUM-bound `Range` is consistent with how other physical-dimension
qualifiers are represented in FHIR. Keeping both representations optional
but mappable answers the reporter's secondary question (do not mandate the
coded form) and matches FHIR's general preference for "capture what was
observed; let consumers derive equivalents".

---

### Disposition B: Alternative Approach

#### Proposal

Do not extend the BP Observation. Instead, recommend recording cuff
dimensions on the `Device` resource referenced from `Observation.device`,
using `Device.property` with `type` bound to a small CodeableConcept value
set ({cuff-circumference-min, cuff-circumference-max} or a single
"cuff-circumference-range" code) and `valueQuantity` / valueRange in UCUM
`cm`. Publish an IG page that documents this binding and the accompanying
mapping to the existing `bpCuffSizeVS`, but make no normative changes to
the BP Observation profile itself. The coded cuff size on the Observation
remains the only Observation-level representation.

#### Justification

This avoids growing the BP Observation profile with a second representation
of the same physical fact, and it puts a property of the *cuff* (a piece of
equipment) on the *Device* that represents the equipment, which is closer
to the FHIR data-element responsibility model. It is also implementable
today against the base spec without minting a new extension. The trade-off
versus Disposition A is real: many BP submissions, especially home / kiosk
flows, do not include a `Device` reference, so this approach risks losing
the data altogether in exactly the consumer-device scenarios the reporter
cites. It also fragments cuff-size capture between Observation (coded) and
Device (numeric), making validation and downstream use harder.

---

### Disposition C: Decline

#### Proposal

Decline the change. The IG already provides a coded cuff-size qualifying
extension bound to `bpCuffSizeVS`; implementers who need to carry a numeric
range can do so today via `Device.property` on the referenced Device, or
via LOINC `8358-4` as an additional `Observation.component`. No new IG
artifact is required; instead, add an implementation note pointing at these
existing mechanisms.

#### Justification

The mechanisms exist in the base spec and in adjacent terminologies (SNOMED
`bpCuffSizeVS`, LOINC `8358-4`, `Device.property`), and the Zulip thread
shows implementers are already using them. Adding another extension creates
two ways to say the same thing, which the workgroup has historically tried
to avoid for vital signs. The downside is that this leaves the reporter's
core complaint unresolved — manufacturer data is numeric, and forcing a
mapping at capture time is exactly what they want to avoid — so this
disposition is hard to defend on use-case grounds.

---

### Recommendation

**Recommended disposition:** A

The reporter's use case is concrete and current (Omron, A&D, Withings home
BP devices), the data they actually receive is a numeric range in cm, and
the Zulip thread shows multiple national programs (Norway, Netherlands)
have already independently built extensions or model elements to carry this
fact. Disposition A meets the request directly, keeps the existing coded
binding available for systems that already use it, and answers the
secondary question by leaving both forms optional. Disposition B is
architecturally cleaner but materially weakens coverage in exactly the
consumer-device flows that motivated the ticket (no `Device` reference →
no place to record the size). Disposition C is the least defensible: the
two-comment history on this ticket and the explicit "CIMI please propose
definition and vote" / "vote will be taken 2024-02-08" comments make clear
that the workgroup expects to act, not punt. CIMI should ratify a
`Range`-typed Qualifying Extension on `Observation.component`, UCUM-bound,
and publish the small/medium/large mapping table alongside it.
