## Summary

The reporter asks that `DocumentReference.practiceSetting` (referred to in
the ticket as `practiseSetting`) be flagged as part of the resource summary
view. They note that the element's own published `Requirements` text
calls this metadata "an important piece of metadata that providers often
rely upon to quickly sort and/or filter out to find specific content," and
argue that an element so described should appear in summary
serializations (i.e., should carry `isSummary = true`). The ticket is
recorded against the Orders & Observations work group with the highest
priority and is currently in `Triaged` status; no Jira links, no related
Zulip threads, and no GitHub PRs/issues were found in the cross-reference
index.

> Editorial note: `DocumentReference` has historically been managed by
> Structured Documents, not Orders & Observations. The current Jira
> work-group assignment may itself need confirmation as part of triage.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Set `isSummary = true` on `DocumentReference.context.practiceSetting`
(the path is `practiseSetting` as written in the ticket; the actual
published path uses the American spelling). Concretely:

- In the `DocumentReference` StructureDefinition, change the
  `ElementDefinition` for `DocumentReference.context.practiceSetting`
  (R5/R6) so that `isSummary = true`.
- Regenerate the resource snapshot, the summary serialization examples,
  and any rendered summary tables in the spec build.
- Note the change in the resource's change-log / ballot notes for the
  next FHIR Core publication.

No data-model change is required; only the summary flag and the
resulting summary projection change.

#### Justification

The reporter's argument is internally consistent with the resource's own
`Requirements` text: an element documented as something providers "rely
upon to quickly sort and/or filter" content is, by FHIR's general
convention, exactly the kind of metadata that belongs in the summary
view (`_summary=true`, Bundle search summaries, etc.). The cost is low
(one flag, regenerated artifacts), and the change is forwards-compatible
for consumers — adding an element to the summary view does not break
existing parsers or existing instances.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the `isSummary` decision to a holistic review of which
`DocumentReference.context` (R4) / top-level `context.*` (R5/R6)
elements should be marked `isSummary`, rather than flipping a single
flag in isolation. As part of that review:

- Audit `DocumentReference` for elements that are commonly used as
  search/filter axes (e.g., `type`, `category`, `subject`, `date`,
  `author`, `practiceSetting`, `facilityType`, `event`, `period`).
- Confirm which of those are already `isSummary` and which are not.
- Propose a coherent summary set in a single change request, with a
  rationale tied to common search parameters and to the
  `Requirements` text on each element.

The disposition for `practiceSetting` specifically would then be:
"Accept as part of the consolidated summary review" rather than as a
one-off flip.

#### Justification

`isSummary` is a cross-element design concern: the value of any single
element being in the summary depends on what else is in the summary.
Touching one element at a time risks an inconsistent summary projection
where, for example, `practiceSetting` is summary but `facilityType` (a
sibling chosen for similar reasons) is not. Bundling the decision
preserves the coherence of the summary view and gives the workgroup a
single, defensible rationale to publish in the change-log.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave `DocumentReference.practiceSetting` not
marked `isSummary`, on the grounds that the `Requirements` text alone
is not sufficient justification — many `Requirements` strings on FHIR
elements describe utility without implying summary inclusion — and the
element can already be retrieved by clients via the standard
`_elements` parameter or by simply requesting the full resource.

If declined, the response to the reporter should explicitly note:
- the existence of `_elements` for client-driven projection, and
- that the existing `practice-setting` search parameter (where
  defined) already supports the sort/filter workflow the
  `Requirements` text describes.

#### Justification

The summary view is intentionally narrow; expanding it has a cost for
servers that materialize summary representations and for transports
where the summary is the default payload. Without evidence that
servers are routinely returning `DocumentReference` instances in
summary form and that the absence of `practiceSetting` is causing
real interoperability pain, the change is speculative.

---

### Recommendation

**Recommended disposition:** B

The substance of the reporter's argument is sound — `practiceSetting`
is a plausible summary element — but the ticket as written treats
`isSummary` as a per-element decision, when in practice it is a
projection-level decision across the resource. The cleanest path is to
acknowledge the request, treat it as the trigger for a consolidated
summary review of `DocumentReference`, and resolve all related
elements together in a single change. This avoids a piecemeal summary
that drifts over time, gives the workgroup a single rationale to
publish, and still gets the reporter what they actually want
(`practiceSetting` in the summary) — just inside a coherent set rather
than as a one-off flip. As a triage action, it would also be worth
confirming whether this ticket truly belongs to Orders & Observations
or to Structured Documents, since `DocumentReference` has historically
been managed by the latter.
