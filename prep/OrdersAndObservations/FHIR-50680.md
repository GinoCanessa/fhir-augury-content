## Summary

FHIR-50680 asks the Cancer Pathology Data Sharing (eCP) FHIR IG to add a
**Procedure profile**, add guidance ensuring **`Procedure.code.text`** is
populated and marked **Must Support** in all relevant profiles (with
examples updated to match), and to include the new Procedure profile in
the **V2 ORC segment** mapping (per Hans Buitendijk's mapping
spreadsheet referenced in the ticket).

The ticket has had a long disposition arc inside O&O. After multiple
clarification rounds in May–July 2025 (the work group repeatedly asked
what additional constraints beyond US Core 6.1.0 Procedure were needed,
where the resource sits in the document bundle, and what the ORC
mapping should look like), the WG held that the Procedure entry was
already Must Support / optional in the bundle and that procedure
information cannot be required because it is not always retrievable
from a pathology report. On 2025‑08‑21 the WG voted 10‑2‑3 to
"Consider for Future Use" with a motion to not add a profile at this
time but to keep an open issue for further guidance / a possible
profile. On 2025‑09‑10 rikim noted that **v2.0.0 of the IG already
added a Procedure profile and an example**. On 2025‑10‑03 lynn_laakso
**reverted that resolution and reopened the ticket post‑STU2
publication**, so the ticket is back in "Triaged" awaiting a fresh
disposition.

The reporter (Alex Goel) and Andrea Pitkus (apitkus) push hard for
treating Specimen Collection Procedure as a first‑class concept —
arguing CLIA/CAP accreditation, USCDI v1 procedures, and USCDI v4
specimen‑rejection requirements make procedure context essential for
the order‑to‑result workflow (ServiceRequest → Procedure → Specimen →
DiagnosticReport). The CAP Laboratory Accreditation conversation on
2025‑07‑30 clarified that the IG must **support** lab inclusion of
CLIA elements but is not obligated to **enforce** them.

The single cross‑reference recorded for this ticket is a mention of
the FHIR element `Procedure.code.text`, consistent with the request to
mark that element Must Support in relevant profiles.

## Proposed Dispositions

Note: because no `repo-analysis` briefing is in scope for this ticket,
the dispositions name the IG and the affected FHIR element/profile
rather than concrete repo paths. If the WG accepts a code‑touching
disposition, the Cancer Pathology Data Sharing IG repo should be
added to the `repo-analysis` cache and a follow-up `ticket-plan`
run performed.

### Disposition A: Accept As Requested

#### Proposal

Treat the v2.0.0 work as the foundation and complete the original
request in full:

1. **Procedure profile** — keep the Procedure profile that was added
   in v2.0.0, derive it from US Core Procedure 6.1.0, and constrain
   it for the eCP context. Specifically:
   - Mark `Procedure.code` and `Procedure.code.text` as **Must
     Support** on the Procedure profile and on every other eCP
     profile that references Procedure (bundle entries, references
     from Specimen, ServiceRequest, and DiagnosticReport).
   - Keep the bundle entry cardinality for Procedure at `0..*` Must
     Support (per the 2025‑07‑16 WG decision that procedure cannot
     be required because it is not always retrievable).
2. **Examples** — update every shipped eCP example so that any
   Procedure instance carries `code.text`, and add a Specimen
   Collection Procedure example referenced by `Specimen.collection`
   /`basedOn ServiceRequest`.
3. **V2 ORC mapping** — add the Procedure profile to the IG's
   V2‑to‑FHIR mapping page, sourcing the ORC segment mappings from
   the Buitendijk spreadsheet linked in the description and aligning
   with the published V2‑to‑FHIR ORC mapping where one exists.
4. **Guidance** — add an IG narrative section describing how
   Procedure is used in eCP (e.g., specimen collection procedures,
   procedures that produced the specimen referenced in the
   DiagnosticReport), explicitly stating the IG **supports but does
   not enforce** CLIA element inclusion (per the 2025‑07‑30 CAP
   confirmation).

#### Justification

This proposal directly answers the four clarifying questions posed
by the WG (US Core baseline, additional constraints, bundle
positioning, ORC mapping) and is consistent with the v2.0.0 change
already applied. Marking `Procedure.code.text` Must Support is a
low‑burden change that materially improves human readability of
pathology procedures even when codings are absent — this matches
existing FHIR practice for `CodeableConcept.text` on
human‑authored narrative resources. Including Procedure in the V2
ORC mapping is necessary because pathology orders flowing through
ORC carry procedure context (collection method, performer) that
implementers need to round‑trip from V2. Keeping Procedure entry
optional in the bundle (rather than 1..*) honors the 2025‑07‑16
WG conclusion that procedure data is not always retrievable, which
addresses the primary objection raised during deferral discussions
while still fulfilling the structural ask.

### Disposition B: Alternative Approach

#### Proposal

Defer the **profile** work but accept the cross‑cutting **guidance
and mapping** parts of the request:

1. Withdraw or freeze the v2.0.0 Procedure profile (or keep it
   informative only, marked "for future use") and instead rely on
   the base US Core Procedure 6.1.0 profile for any Procedure
   instances in the bundle.
2. Add an IG narrative section that:
   - Tells implementers to populate `Procedure.code.text` whenever
     a Procedure resource is included, and lists the eCP profiles
     that reference Procedure in their Must Support narrative.
   - Documents the recommended pattern for Specimen Collection
     Procedure (Specimen → `basedOn` ServiceRequest, optionally
     pointing at a Procedure that performed the collection), and
     defers a normative `Specimen.collection.procedure` linkage to
     a future R6‑aligned IG version (per lynn_laakso's 2025‑10‑03
     comment).
   - States explicitly that the IG **supports** CLIA element
     inclusion but does not **enforce** it (per the 2025‑07‑30 CAP
     conversation), and adds the disclaimer apitkus requested:
     until the procedure/specimen story is fully expressed, the IG
     should be treated as informative for vendor preparation.
3. Add the V2 ORC mapping section using Hans Buitendijk's
   spreadsheet, but map to the **base Procedure resource** rather
   than to a new eCP‑specific profile.
4. File a follow‑up open issue for an R6 cycle to revisit a true
   Procedure profile (and the `Specimen.collection`↔Procedure
   link) once R6 ballots stabilize.

#### Justification

This reflects the prior WG vote (10‑2‑3 on 2025‑08‑21) that did
not want a new profile yet, while still delivering the parts of the
ticket that have clear consensus: `Procedure.code.text` guidance,
ORC mapping, and a documented Specimen↔Procedure pattern. Using US
Core Procedure 6.1.0 directly avoids creating an eCP profile that
would only redundantly tighten US Core, which was the WG's
unresolved concern in the May–July 2025 clarification thread. The
explicit "supports, not enforces" framing matches the CAP
conversation and addresses apitkus's accreditation worries
without forcing a normative cardinality change. The trade‑off vs
Disposition A is that v2.0.0's profile work would be partially
walked back, which is awkward post‑STU2; if that is unacceptable to
the WG, A is the better path.

---

### Disposition C: Decline

#### Proposal

Close FHIR-50680 as **Persuasive with Mod / Already Addressed**,
on the basis that the v2.0.0 publication of the IG already added a
Procedure profile and an example (rikim, 2025‑09‑10), and that the
remaining asks (`Procedure.code.text` Must Support guidance,
ORC mapping) should be tracked as separate, scoped change
requests rather than reopened under this single umbrella ticket.
Restate the original "Considered for Future Use" outcome
(2025‑08‑21, 10‑2‑3) and direct any further procedure/specimen
linkage work to a new R6‑era ticket.

#### Justification

The Procedure profile asked for has already been added; reopening
the same ticket to also cover element‑level Must Support
guidance, V2 ORC mapping, and possible cardinality changes mixes
distinct work items that have different risk profiles and
different implementer impact. The WG has repeatedly noted (Jul 14,
Jul 16, Aug 21) that there is insufficient new information beyond
US Core 6.1.0 to justify additional normative constraint, and the
CAP Laboratory Accreditation discussion confirmed that the IG is
not obligated to enforce CLIA elements. Splitting follow‑up work
into focused tickets (one for `code.text` MS guidance, one for the
ORC mapping, one for the future R6 Specimen↔Procedure link) gives
each a clean audit trail and respects the prior vote.

---

### Recommendation

**Recommended disposition:** **A — Accept As Requested**.

The 2025‑10‑03 reopen post‑STU2, combined with the v2.0.0 work
that already shipped a Procedure profile and example, makes
Disposition A the path of least disruption: it ratifies what the
IG already publishes and completes the two pieces (Must Support
on `Procedure.code.text` across referencing profiles, plus the V2
ORC mapping that the ticket explicitly calls for via Hans
Buitendijk's spreadsheet) that have not yet been delivered. It
also honors the 2025‑07‑16 WG decision to keep Procedure
**optional** in the bundle, which was the main concern behind the
earlier defer/Future‑Use proposals — this preserves the WG's
position that procedure data is not always retrievable while
still giving CAP/CLIA implementers a clean place to attach it.
Disposition B is the right answer only if the WG concludes the
v2.0.0 profile was a mistake and wants to roll it back, which is
hard to justify after publication. Disposition C is the
cleanest from a process standpoint but effectively asks the
reporter and apitkus to re‑file work that has been on the WG's
docket for five months, and risks losing the V2 ORC mapping ask
entirely; it should only be taken if the WG decides the reopen
itself was procedural and not substantive.
