## Summary

The reporter (Ana Kostadinovska) observes that an Orders & Observations
resource does not follow the agreed FMG/cross-workgroup "BodyStructure
pattern" — namely, a single element typed as `CodeableReference(BodyStructure)`
that supersedes any pre-existing `bodySite`/`bodyStructure` pair. The ticket
flags that the offending element is currently typed `Reference(BodyStructure)`
and asks the workgroup to either (a) refactor the element to follow the
pattern, or (b) add a Notes-section justification for the deviation.

The ticket text itself does not name the specific resource; the augury
metadata exposes only `Work Group = Orders & Observations` and
`Specification = FHIR Core (FHIR)`, with no "Related Artifact" field
surfaced. Reviewers should consult the Jira UI for the artifact link
when triaging. The reporter filed two sibling tickets the same day for
Patient Care resources ([FHIR-54207](https://jira.hl7.org/browse/FHIR-54207),
[FHIR-54208](https://jira.hl7.org/browse/FHIR-54208)), each carrying the
same generic body — these are the only direct cross-references called
out by the reporter and confirmed in the comment by `corey_spears`.

The pattern itself was established and inventoried on the FMG Zulip
("CodeableReference usage in Core") and in the bulk-O&O ticket
[FHIR-53203](https://jira.hl7.org/browse/FHIR-53203) (Applied), which
already migrated `Observation.bodyStructure`, `ServiceRequest.bodyStructure`,
`ObservationDefinition.bodyStructure`, `ImagingSelection.bodySite`,
`ImagingStudy.series.bodySite` and `Specimen.collection.bodySite` to
`CodeableReference(BodyStructure)`. The candidate O&O elements that may
remain `Reference(BodyStructure)` (and therefore most likely to be the
target of this ticket) are anything not covered by the FHIR-53203 sweep
— reviewers should verify against the Jira "Related Artifact" field and
the current build.

## Proposed Dispositions

> **Note for reviewers:** The ticket's body does not name the offending
> resource, and augury does not surface a "Related Artifact" custom
> field. Each disposition below is written so the workgroup can plug in
> the actual resource/element identified from the Jira UI before
> ballot. Where a likely candidate exists (an O&O resource not covered
> by FHIR-53203 / FHIR-50355 and absent from the FMG inventory), the
> recommendation is to refactor in line with the established pattern.

### Disposition A: Accept As Requested

#### Proposal

Refactor the offending O&O element from `Reference(BodyStructure)` to
`CodeableReference(BodyStructure)`, and remove any peer `bodySite`
element that duplicates the structural intent. Specifically:

- Rename the element from its current `…bodyStructure` (or equivalent)
  Reference form to a `CodeableReference(BodyStructure)`-typed element,
  retaining the same path name where backward compatibility allows.
- Mark the prior peer `bodySite` element (if present) as **deprecated**
  in this release with a forwarding note pointing to the new combined
  element, mirroring the treatment applied in
  [FHIR-53203](https://jira.hl7.org/browse/FHIR-53203) and
  [FHIR-50355](https://jira.hl7.org/browse/FHIR-50355).
- Update the resource's Notes section to reference the FMG
  CodeableReference pattern and the precedent tickets, so future
  reviewers understand the typing choice.

#### Justification

This is the disposition the O&O workgroup has applied to every other
resource flagged with the same pattern (FHIR-53203 / FHIR-50355) and it
matches the inventory that FMG accepted as the post-pattern state of
core (Zulip thread). Accepting maintains intra-workgroup consistency,
keeps O&O aligned with the parallel Patient Care work
(FHIR-54957 / FHIR-54992 / FHIR-54703) and avoids implementer surprise
from one O&O resource using a different mechanism for the same
concept. The change is mechanically the same as prior changes, so the
implementation cost (FMM, breaking-change classification) is well
understood.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the element typed as `Reference(BodyStructure)` but document the
deviation explicitly in the resource's **Notes** section (the
escape-hatch the reporter offered). The Notes entry should:

- Cite the FMG CodeableReference pattern and the bulk O&O ticket
  ([FHIR-53203](https://jira.hl7.org/browse/FHIR-53203)).
- State the resource-specific reason the pattern does not apply (e.g.,
  the element semantically requires a *concrete* BodyStructure
  instance and the coded fallback is not meaningful here, or
  the element is already qualified by a sibling
  `bodySite : CodeableConcept`).
- Optionally add an invariant or guidance paragraph aligning with
  FHIR-54614's suggestion that the fields not be considered duplicative.

No element typing or cardinality changes are made.

#### Justification

The pattern as agreed by FMG is a "where applicable" pattern, and the
ticket itself accepts a documented deviation as a valid response. If
the O&O workgroup determines the resource genuinely needs a pure
reference (for example because a coded `BodyStructure` proxy adds no
expressive power for that use case, or because the element is the
*identity* of a body site rather than a reference to one), then a Notes
deviation is preferable to a forced refactor that would (a) burden
implementers with a breaking change without a corresponding semantic
gain, and (b) re-open settled debates of the FHIR-54614 /
FHIR-50355 family for a resource that has already been considered. The
trade-off versus Disposition A is loss of cross-resource uniformity in
exchange for not paying a breaking-change cost.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Resolved – No Change** on the grounds that the
"pattern" cited by the reporter is a recommendation rather than a
normative requirement, and that the resource in question was previously
reviewed and intentionally retained the `Reference(BodyStructure)`
typing — without committing the workgroup to add a Notes paragraph.

#### Justification

If the O&O workgroup confirms that the targeted resource was already
discussed under [FHIR-53203](https://jira.hl7.org/browse/FHIR-53203)
(or FHIR-50355) and consciously left in the current form, the request
has effectively been answered before. A second deviation note
duplicates the audit trail already in those tickets, and accepting (per
Disposition A) would re-litigate a settled decision. Decline is only
defensible if such a prior, traceable decision exists; without it,
silence on the deviation is harder to defend at ballot.

---

### Recommendation

**Recommended disposition:** A

The change requested is small, mechanical, and identical to changes
the O&O workgroup has already accepted for every other resource flagged
with the same pattern (FHIR-53203, FHIR-50355). FMG has already
inventoried the post-pattern state of core, and the parallel Patient
Care tickets (FHIR-54957 / FHIR-54992 / FHIR-54703) make it likely the
pattern will be enforced cross-workgroup anyway — leaving one O&O
resource as an outlier creates a future harmonization debt with no
clear semantic upside. Disposition B remains the right answer if the
workgroup, after identifying the specific element, finds a substantive
reason the pattern is inapplicable (e.g., the element legitimately
needs a *concrete* BodyStructure rather than a coded fallback); in
that case, the Notes paragraph should cite the FMG thread and
FHIR-53203 so the deviation is durable. Disposition C should only be
chosen if a prior conscious decision is already on record for this
resource; otherwise it leaves the inconsistency unexplained.
