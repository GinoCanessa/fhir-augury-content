## Summary

Kevin O'Donnell requests that the `BodyStructure.includedStructure` and
`BodyStructure.excludedStructure` backbone elements be renamed to
`BodyStructure.included` and `BodyStructure.excluded`, respectively. His
rationale is purely readability: because the nested child element is
already named `structure` (e.g. `includedStructure.structure = <brain>`),
the current naming produces a redundant "Structure.structure" sequence
that clutters downstream paths such as
`Observation.bodyStructure.includedStructure.structure`. Shortening the
outer element names removes the repetition while preserving the
semantics of "the structures that are included in / excluded from this
BodyStructure". There are no linked or cross-referenced Jira tickets and
no linked GitHub items. Cross-reference hits are limited to FHIR element
mentions in the ticket body itself. The request is implicitly tied to
the ongoing R6 rework of BodyStructure (see related O&O WG Zulip
discussions), and the element in question is actively being used in R6
usage patterns such as Observation-on-BodyStructure morphology examples.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rename the two BackboneElement children on the `BodyStructure` resource
in FHIR Core:

- `BodyStructure.includedStructure` → `BodyStructure.included`
- `BodyStructure.excludedStructure` → `BodyStructure.excluded`

All nested children (`.structure`, `.laterality`, `.bodyLandmarkOrientation`,
`.spatialReference`, `.qualifier`, `.description`, `.image`, etc.) retain
their current names, so the new paths become e.g.
`BodyStructure.included.structure`, `BodyStructure.included.morphology`,
`BodyStructure.excluded.structure`. Update the StructureDefinition,
invariants, element definitions, narrative, all worked examples, and
any R6 search parameters or constraints that reference the old element
names. Note the rename in the R6 change notes / breaking-change list.
Existing R6 content using `bodyLandmarkOrientation` extension slicing
against `includedStructure` (observed in ballot feedback on Zulip) must
be updated to target `included` in the same release.

#### Justification

The request is a pure readability/clarity improvement with a clean,
self-evident payoff: the current form duplicates the word "structure"
in paths like `BodyStructure.includedStructure.structure` whenever a
structure code is supplied, which is the most common use. The reporter's
own worked examples in the O&O "Observation on BodyStructure Morphology"
Zulip thread demonstrate the friction. Because BodyStructure is still in
an R6 cycle actively under ballot-level review (O&O "R6 BodyStructure
Review" thread explicitly anticipates ballot comments and further
cleanup), this is the correct window to make the rename — deferring it
makes it a far more expensive backward-incompatibility discussion later.
The shorter names also follow the general FHIR convention of keeping
element names noun-only (e.g. `Bundle.entry`, `Composition.section`)
rather than repeating the parent type in the child name.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `includedStructure` / `excludedStructure` as the structural names
but address the repetition in one of two narrower ways:

1. **Rename only the redundant child**, i.e. keep
   `BodyStructure.includedStructure` and
   `BodyStructure.excludedStructure` but rename the inner
   `includedStructure.structure` / `excludedStructure.structure` child
   to `includedStructure.bodyStructure` or `includedStructure.location`,
   eliminating the `Structure.structure` collision without touching the
   backbone element names that have been in draft publications and
   implementation testing; **or**
2. **Add a documentation/stylistic note** to the BodyStructure resource
   narrative showing the preferred short-form aliases (
   "implementers may read `includedStructure` as 'the included
   structures'") and leave the model untouched.

Under either variant, no element on the public StructureDefinition
changes its canonical path, preserving existing profiles, test
fixtures, and tooling.

#### Justification

BodyStructure has been in use across pre-R6 implementations and
cross-IG ballots (see the "Which package to import for cross version
extensions" Zulip thread referencing
`BodyStructure.extension:includedStructure...bodyLandmarkOrientation`
slicing); renaming the BackboneElement breaks those ElementDefinition
IDs, slice names, and downstream profile JSON. Option B.1 localises the
change to the element that actually causes the readability issue
(`.structure`) and leaves the more stable parent-level name in place.
Option B.2 is the lowest-impact response and is defensible if the
workgroup concludes this is a matter of stylistic preference rather than
a substantive model problem. Both variants avoid the need to ripple the
rename through every R6 example and invariant.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that the request is a cosmetic rename
of a BackboneElement that has already shipped in multiple FHIR R6
snapshots, has been adopted by active implementers (confirmed by cross-
IG slicing references such as
`BodyStructure.extension:includedStructure.extension:bodyLandmarkOrientation`
observed in the "implementers" Zulip stream), and is already well
documented. Close the ticket as "Not Persuasive" or "Already Addressed"
with a note that implementers can use the existing names without
ambiguity and that the `BodyStructure.includedStructure` naming pattern
is consistent with other R6 backbone elements that repeat the parent
type in the child name (e.g. compact-descriptor patterns used
elsewhere in the spec).

#### Justification

The request is purely naming; no semantics or interoperability problem
is identified in the ticket body, and no other Jira ticket or Zulip
discussion calls for the rename (the only Zulip hits are usage, not
complaints about verbosity). Breaking the path
`BodyStructure.includedStructure` silently invalidates every downstream
profile, FHIRPath expression, search parameter string, and IG
crosswalk that was authored against the published R6 form. Absent a
stronger signal from implementers or a semantic defect, the cost of the
churn outweighs the readability win.

---

### Recommendation

**Recommended disposition:** A

BodyStructure is still in an R6 ballot cycle with active review — the
O&O "R6 BodyStructure Review" thread explicitly anticipates further
ballot-driven cleanup, and the reporter's own worked examples in
"Observation on BodyStructure Morphology" demonstrate the
`Structure.structure` readability issue in practice. The rename is
small, mechanical, and isolates its impact to a single resource whose
downstream surface area is still modest; it will only grow as R6
matures. Option B.2 (docs-only) does not actually fix the path; Option
B.1 (rename the child instead) creates a more awkward naming
(`includedStructure.bodyStructure`) than the current form. Option C
underweights the fact that we are still in the correct window to make
breaking changes to BodyStructure. Accepting as requested — with a
clear line item in the R6 breaking-change notes and an update to the
already-known `bodyLandmarkOrientation` extension slicing targets — is
the cleanest outcome. The workgroup should confirm during triage that
no O&O-owned IG has already frozen `includedStructure`/`excludedStructure`
at a normative boundary; if it has, fall back to Disposition B.1.
