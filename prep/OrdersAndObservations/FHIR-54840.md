## Summary

`SpecimenDefinition.subject[x]` is a CodeableReference (choice of
`CodeableConcept` or `Reference(Group)`), but the `CodeableConcept`
side has no value-set binding, leaving implementers without guidance
about which codes are appropriate for describing the intended subject
of a kind of specimen. Ulrike Merrick proposes adding (at minimum) an
example binding to a small SNOMED CT value set drawn from the
"qualifier — origin" hierarchy: 278412004 *Human - origin*,
429431000124102 *Environment - origin*, 429411000124108 *Animal -
origin*, 429421000124100 *Food - origin*, and optionally
436631000124109 *Medical device origin* and 436621000124106 *Biologic
product origin*.

A related ticket, [FHIR-54594](https://jira.hl7.org/browse/FHIR-54594)
(*Missing valueset binding for ObservationDefinition.subjectType*,
**Persuasive**, Applied), has already produced a value set for
subject type that the OO Specimen WG flagged as potentially reusable
here. During the 2026-03-23 OO call (Confluence page 441127442), the
group narrowed the choice to two options: (1) add an example value
set, or (2) remove `subject.codeableConcept` entirely and keep only
`.subjectReference` to `Group`, with new guidance steering
implementers to use `Group.type` (`person`/`animal`/etc.) and
`Group.characteristic.value` for finer-grained subject
characterisation. The current resolution on the ticket is **Not
Persuasive with Modification**, with the agreed modification being
option (2) — drop the `CodeableConcept` choice and document the
`Group`-based pattern.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an **example binding** to `SpecimenDefinition.subject[x]`'s
`CodeableConcept` choice, pointing at a new SNOMED CT value set built
from the "qualifier — origin" hierarchy listed by the reporter:
`278412004 | Human - origin`, `429431000124102 | Environment -
origin`, `429411000124108 | Animal - origin`, `429421000124100 | Food
- origin`, `436631000124109 | Medical device origin`, and
`436621000124106 | Biologic product origin`. Keep the existing
CodeableReference choice intact (both `subjectCodeableConcept` and
`subjectReference(Group)` remain valid). Where overlap exists with
the value set produced by FHIR-54594 for
`ObservationDefinition.subjectType`, reuse that value set or compose
this one to share concepts so that ObservationDefinition and
SpecimenDefinition stay aligned.

#### Justification

This is the minimum change required to address the reporter's
original concern that an unbound `CodeableConcept` arm of a
`CodeableReference` provides no implementer guidance. An *example*
binding is non-normative and therefore low-risk: existing
implementations are unaffected, and the related ticket FHIR-54594 has
already established that the workgroup is willing to add an example
binding for the same concept (subject type) on a sibling resource —
so refusing to do the same on `SpecimenDefinition` would be
inconsistent. The SNOMED qualifier-origin concepts are stable,
maintained, and were specifically nominated by the reporter as
suitable.

---

### Disposition B: Alternative Approach (current resolution direction)

#### Proposal

**Remove the `CodeableConcept` arm** of `SpecimenDefinition.subject[x]`
so the element collapses to `Reference(Group)` only (effectively
renaming `subject[x]` to `subjectReference`). Add prescriptive
guidance on the element (and a corresponding usage note on the
`Group` resource) explaining that the intended subject type is
expressed via `Group.type` (`person`, `animal`, `device`,
`medication`, `substance`, `biologicallyDerivedProduct`,
`nutritionProduct`, `location`), and that finer characterisation
(e.g., age categories, species, breed) is expressed via
`Group.characteristic.code` + `Group.characteristic.value`, with
`Group.characteristic.exclude` for negative criteria. Engage with
LOINC to obtain a code for "Subject species or breed" so the animal
case is fully expressible.

This is the resolution direction currently captured on the ticket
("Not Persuasive with Modification") and was the consensus on the
2026-03-23 OO call (Confluence 441127442).

#### Justification

The OO workgroup considered the example-binding route (Disposition A)
and judged that, even with an example binding, the
`CodeableConcept` arm cannot fully express the multi-dimensional
nature of "kind of subject" (species, age, life-stage,
include/exclude, environmental sub-types) without effectively
duplicating the structure that already exists in `Group`. Forcing
implementers through `Group` produces a single, structured way to
describe complex subject populations, avoids two competing modelling
paths, and aligns with how `Group` is already used elsewhere for
research and definitional contexts. Note that this **is** a breaking
change for any implementer using `subject.codeableConcept` today;
because R6 is in ballot/active development for this resource and
because no Applied implementations have been demonstrated, the WG
appears willing to accept that cost. Justification must address two
gotchas: (1) it removes the lightweight "just say it's a human" path
and forces creation of a `Group` even for trivial cases; (2) the
"Subject species or breed" LOINC code is not yet assigned, so the
animal case is documented but not fully realisable on day one.

---

### Disposition C: Decline

#### Proposal

Mark the ticket **Not Persuasive** and leave
`SpecimenDefinition.subject[x]` unchanged: keep both arms of the
CodeableReference, and leave the `CodeableConcept` arm with no
binding. Document, on the element definition, that the
`CodeableConcept` arm is intended for ad-hoc local coding of subject
kind and that implementers wanting structured subject populations
should use the `Reference(Group)` arm.

#### Justification

A `CodeableReference` with an unbound `CodeableConcept` arm is a
recognised FHIR pattern (the binding is genuinely open-ended and
locale-dependent), and the existing
`Reference(Group)` arm already gives implementers an expressive,
structured option when they need it. Adding either an example binding
(A) or removing the choice (B) introduces churn without solving a
demonstrated interoperability failure — no Zulip thread or GitHub
issue surfaces an implementer who is actually blocked. Declining
preserves backward compatibility with anything that may already be
using `subject.codeableConcept` and keeps `SpecimenDefinition`
out of the ballot block.

---

### Recommendation

**Recommended disposition:** B

The workgroup has already converged on Disposition B on the
2026-03-23 call and captured it as the current resolution
("Not Persuasive with Modification"), and the rationale is sound:
`Group` already provides a richer, more structured way to express
"kind of subject" than any single `CodeableConcept` value set could,
and keeping both arms perpetuates two parallel modelling paths for
the same concept. Disposition A is a defensible fallback if the
workgroup decides the breaking change is too disruptive — and the
fact that FHIR-54594 took exactly that route for
`ObservationDefinition.subjectType` is a real consistency argument
that should be weighed. Disposition C is the weakest option: it
leaves the original complaint unresolved and contradicts the
workgroup's own decision on the sibling element. The two follow-ups
the WG should commit to before applying B are (1) confirming there
are no Applied implementations that would break, and (2) opening the
LOINC request for "Subject species or breed" so the animal-subject
guidance is actionable.
