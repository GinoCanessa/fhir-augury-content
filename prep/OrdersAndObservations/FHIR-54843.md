## Summary

The reporter asks the Orders & Observations work group to revisit a piece of
spec guidance that currently reads: *"The bodySite element in any resource can
be used when the resource's code is not implicit about the body structure and
does not require extensive detail."* The sentence is now out of step with how
the various resources actually model body location. Across the spec, three
distinct patterns coexist today: (1) resources where the body element is a
single `CodeableReference(BodyStructure)` (e.g., `ServiceRequest.bodyStructure`,
`ObservationDefinition.bodyStructure`, `DocumentReference.bodyStructure`,
`Specimen.collection.bodySite`, `ImagingStudy.series.bodySite`,
`ImagingSelection.bodySite`, and the `bodySite`/`subSite` shapes on
`Claim.item`, `ClaimResponse.addItem`, and `ExplanationOfBenefit.item`);
(2) resources that still expose two parallel elements — a `bodySite`
(`CodeableConcept`) and a `bodyStructure` (`Reference(BodyStructure)`) —
namely `Procedure`, `Observation`, and `Condition`; and (3) resources where
body location is *only* a `Reference`, e.g., `DeviceAssociation.bodyStructure`.
The ticket therefore asks the work group both to confirm the current
inventory and to rewrite the cross-resource guidance so implementers know
which element to populate when they only have a code, when they have a
referenceable BodyStructure, and how the two relate when both are present.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a documentation-only change against the FHIR core
specification. Concretely:

1. Replace the existing cross-resource sentence with guidance that
   acknowledges the three patterns now present in the spec:
   - **Single `CodeableReference(BodyStructure)`** — populate with a code
     when only a code is known, or with a reference (or both) when a
     `BodyStructure` instance is appropriate. Applies to
     `ServiceRequest.bodyStructure`, `ObservationDefinition.bodyStructure`,
     `DocumentReference.bodyStructure`, `Specimen.collection.bodySite`,
     `ImagingStudy.series.bodySite`, `ImagingSelection.bodySite`, and
     the `bodySite` (with `subSite`) sub-structures on `Claim.item`,
     `ClaimResponse.addItem`, and `ExplanationOfBenefit.item`.
   - **Parallel `bodySite` (CodeableConcept) + `bodyStructure`
     (Reference(BodyStructure))** — `Procedure`, `Observation`,
     `Condition`. Document that `bodySite` is the lightweight option for
     code-only use and `bodyStructure` is for referenceable detail; if
     both are populated, `bodyStructure` is the more detailed and SHALL
     be consistent with `bodySite`.
   - **Reference-only** — `DeviceAssociation.bodyStructure`. Document
     that a `BodyStructure` instance is required to convey location.
2. Audit each affected resource's element-level definition text to ensure
   it consistently points to (or restates) the new cross-resource guidance.
3. Keep the typo `bodyStrucure` cited in the ticket as a *pointer to*
   `bodyStructure`; verify in the spec source that the actual element
   names are correctly spelled (the typo is in the ticket text, not in
   the published resources).

No element renames, type changes, or cardinality changes are proposed.

#### Justification

The reporter is flagging real drift between an old, generic sentence and
the current state of the spec. The text was written when most resources
used `bodySite : CodeableConcept`; today the dominant patterns are
`CodeableReference(BodyStructure)` (single element) and the parallel
`bodySite` + `bodyStructure` pair. A documentation-only fix updates
implementer guidance without touching wire format, schemas, validators,
or downstream IGs, and is consistent with how Orders & Observations has
historically handled cross-resource guidance corrections at the boundary
between BodyStructure adoption and legacy `CodeableConcept` usage. It
also respects the work groups that own the non-O&O resources in the list
(FM owns Claim/ClaimResponse/EoB; II owns ImagingStudy/ImagingSelection;
OO owns most of the rest) by changing only narrative, not their element
definitions.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the spirit of the guidance fix, but split the work into two
deliverables and coordinate the cross-resource patterns rather than just
documenting them:

1. **Now (this ticket):** Update the cross-resource guidance sentence as
   in Disposition A so implementers have correct, current advice in the
   next ballot.
2. **Follow-on (file new tickets, do not bundle here):** Open separate,
   per-resource change requests proposing convergence on a single
   pattern — almost certainly `CodeableReference(BodyStructure)` — for
   `Procedure`, `Observation`, and `Condition`, retiring the parallel
   `bodySite` (`CodeableConcept`) element via the normal deprecation
   path. Likewise, file a ticket asking whether
   `DeviceAssociation.bodyStructure` should also become a
   `CodeableReference(BodyStructure)` so a code-only use case is
   expressible without a separate `BodyStructure` instance.

Each follow-on would go to its owning work group with this ticket as
context.

#### Justification

The current spec contains three distinct shapes for "where on the body."
That is genuinely confusing for implementers, and the existing sentence
is a symptom rather than the disease. However, converging on a single
pattern is a much larger change with backward-compatibility, IG, and
implementation impact, and it would touch resources owned by multiple
work groups. Splitting the work lets the workgroup land the
documentation correction immediately in the current cycle while still
acknowledging — and tracking — the underlying inconsistency without
holding the doc fix hostage to a multi-WG coordination effort.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" on the grounds that the existing sentence is
generic enough to remain technically accurate ("the bodySite element in
any resource can be used when the resource's code is not implicit about
the body structure and does not require extensive detail" still describes
how the `CodeableConcept` form of `bodySite` is intended to be used), and
that resource-specific element definitions already document the actual
element types and their intended use.

#### Justification

If the work group judges that implementers are already correctly served
by the per-element definitions on each resource, the cross-resource
sentence is essentially redundant rather than wrong. Declining avoids
spending ballot capacity on a narrative-only tweak. This is the weakest
of the three options because it does not address the reporter's
substantive observation — that the sentence no longer reflects the
prevalent `CodeableReference(BodyStructure)` pattern and is silent on
the parallel-element resources — but it is defensible if the WG
prioritizes other work for the next cycle.

---

### Recommendation

**Recommended disposition:** A

The request is precise, scoped, and low-risk: it is a guidance-text fix
that reflects what the spec already does, and the reporter has done most
of the inventory work. Disposition A delivers that fix in one place,
keeps it inside Orders & Observations' editorial control, and avoids
opening cross-WG coordination on element renames or convergence. Going
straight to Disposition B (convergence on `CodeableReference`) is
attractive in principle but would entangle FM, II, and OO in
backward-compatibility analysis; that work is better tracked as
follow-on tickets if the WG chooses, but should not block the doc fix
the reporter is actually asking for. Disposition C is hard to defend —
the reporter has shown the sentence is no longer aligned with the bulk
of the resources it claims to describe.
