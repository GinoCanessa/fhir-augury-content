## Summary

Hans Buitendijk has filed this Lab Report ballot item (LABREPORT_BALDEF-2342)
asking the Laboratory Report (FHIR) IG to specify *how* DiagnosticReport and
Observation should be structured when reporting panels (including
microbiology and sensitivities). The proposal is to (a) restrict panel
nesting to one level — the panel code lives on `DiagnosticReport` and the
member test codes live on `Observation`; (b) decompose a multi-level panel
into multiple single-level DiagnosticReports carried together in a Bundle;
(c) if the multi-level structure itself is clinically meaningful, extend
`DiagnosticReport` to carry that structure rather than overloading
`Observation.hasMember`; and (d) reserve `Observation.hasMember` for
relationships between *actual* observations, with sensitivity results being
the deliberate exception (parent obs is a value-less grouper, but is still
considered observation-like rather than a panel). The intent is that
panels are *always* found via `DiagnosticReport` and discrete results
*always* via `Observation`.

This ticket was originally voted Persuasive (2022-09-27) and reverted
2024-10-09 because the motion was only to *include* it as a Lab Report
ballot item, not to actually resolve it. Substantive discussion is still
open, with OO scheduling it for the January Virtual WGM. A 2025-11-14 UV
Lab Report Call note proposes a different mechanism: use the `Group`
resource as the panel grouper, by adding `"panel"` to `Group.type`,
allowing `Observation` in `Group.member`, and allowing `Group` as a target
of `DiagnosticReport.result`. The closely-related ticket
[FHIR-50472](https://jira.hl7.org/browse/FHIR-50472) (Deferred /
Considered for Future Use) — "Do not use Observation to create a Grouper"
— pushes back on Observation-as-grouper from the implementer side and
explicitly references this same Group-based proposal.

## Details

**Description:**

We recommend that the guide defines guidance on how to structure
DiagnosticReport and Observations and when reporting on panels and
micro-biology as follows:

- Panels can only be one level deep (e.g., *[provide example]*) such that
  the panel code is in DiagnosticReport and the test codes are in
  Observation.
- Multi-leveled panels must be reported as individual single-level
  panels. When reporting, a Bundle would include all relevant
  DiagnosticReports (plus related Observations contained or referenced).
- If the structure of a multi-level panel is relevant, then update
  DiagnosticReport to support a structure rather than using
  Observation.hasMember.
- Observation.hasMember should be used for relationships between actual
  observations, not groupers. Groupers should be in DiagnosticReports.
  However, we suggest that sensitivity results use `.hasMember`, not
  `.component`, recognizing that the "parent" Observation would not have
  a value, thus more of a grouper. We see that different though than
  panels.

Taking this approach one can always find panels through DiagnosticReport
and actual test results through Observations.

- If one orders a single observation only, then that test would appear in
  both DiagnosticReport and Observation from a reporting perspective.
- If one orders a panel as a single level or part of a multi-level where
  that is accepted, the panel will always be in DiagnosticReport, not
  sometimes in Observation and sometimes in DiagnosticReport.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative guidance to the Laboratory Report (FHIR) IG that codifies
the reporter's four rules:

1. Panels in this IG are limited to a single level of nesting: the panel
   code is carried on `DiagnosticReport.code` and the constituent test
   codes are carried on `Observation.code` referenced from
   `DiagnosticReport.result`.
2. A clinically multi-level panel MUST be decomposed into multiple
   single-level `DiagnosticReport` instances and exchanged together
   inside a `Bundle` (the existing `DiagnosticReport.result` of the
   parent does not nest other DiagnosticReports — relationships between
   the decomposed reports are conveyed via the Bundle and via
   `basedOn` / `ServiceRequest` linkage).
3. Mark `Observation.hasMember` as **NOT** to be used as a panel
   grouper in the Lab Report IG. Define the *only* sanctioned use of
   `hasMember` as relationships between true observations — explicitly
   call out susceptibility / sensitivity panels as the named exception
   (parent observation is a value-less anchor for the antibiotic
   sensitivity members).
4. Add an issue / open item in the IG noting that *if* preserving
   multi-level panel structure becomes a hard requirement, the
   resolution path is a future change to `DiagnosticReport` itself
   (e.g., a structured `result` or a back-reference) rather than
   re-purposing `Observation.hasMember`.

This change is IG-level (Laboratory Report) only — no change to the
core `DiagnosticReport` or `Observation` resources is required for
items 1–3. Item 4 is recorded as a forward-looking note, not an actual
core change.

#### Justification

This matches the dominant US implementer reality reported in the
*implementers > Laboratory Report - Representing Panels* Zulip thread:
Michele Mottini and "nearly every US EHR" already use a one-level
`DiagnosticReport → Observation` hierarchy, which is exactly what
items 1 and 3 codify. It also addresses Andrea Pitkus's
interoperability concern from the same thread — that "the same result
should not be representable in different ways" — by giving the IG a
single, mandated shape. Treating susceptibility as the named
`hasMember` exception preserves an established microbiology pattern
without re-opening the panel-grouper question. Item 2 keeps multi-level
panels representable today (via Bundle) without a breaking core
change, which directly answers FHIR-50472's "this would be a big change
and would lose DiagnosticReport metadata" objection.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the UV Lab Report Call (2025-11-14) proposal that uses the
`Group` resource as an *explicit* panel grouper, instead of restricting
nesting via prose. Concretely:

1. Submit a coordinated change request against **FHIR Core** to:
   - Add a `panel` code to the value set bound to `Group.type` (or to a
     companion `purpose`/`role` element if `Group.type` cannot accept
     `panel`), and
   - Permit `Observation` in `Group.member.entity` (Group currently
     accepts a fixed list of member resource types).
2. Submit a coordinated change request against **FHIR Core** to add
   `Group` to the allowed target types of `DiagnosticReport.result`.
3. Update the Laboratory Report (FHIR) IG to:
   - Use `Group` (typed `panel`) as the canonical multi-level panel
     grouper, with each `Group.member` pointing at either a child
     `Group` (sub-panel) or an `Observation` (leaf result),
   - Keep flat panels representable as today (`DiagnosticReport.result`
     directly to `Observation`), and
   - Continue to forbid `Observation.hasMember` as a panel grouper,
     reserving it for the susceptibility-style same-observation
     relationships called out in Disposition A.
4. Co-resolve [FHIR-50472](https://jira.hl7.org/browse/FHIR-50472)
   with this disposition, since the same Group-based proposal already
   appears verbatim there.

#### Justification

This honours the underlying request (panels live in DR, results live
in Observation, `hasMember` is not a grouper) while *also* solving
the multi-level / reflex / tiered case that Andrea Pitkus's
implementers-thread comment and apitkus's 2024-10-01 ticket comment
both flag as unsolved by Disposition A's "split into multiple
single-level DRs" rule. `Group` already exists for "an enumerated set
of resources that have some sort of common membership", which matches
LOINC's own definition of *panel* cited in the Zulip thread, so it is
a smaller conceptual stretch than overloading `Observation` or
inventing a new bridging resource (an idea the OO call rejected per
the same thread). The trade-off vs. Disposition A is heavier
machinery: it requires three coordinated FHIR Core changes (Group.type
value-set extension, Group.member type extension, DR.result target
extension), it must clear an OO/Grouping cross-WG review, and it lands
in a future R6+ cycle rather than the current Lab Report ballot.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-47052 as out-of-scope for the current Laboratory Report
(FHIR) IG cycle and defer all panel-grouping guidance to the joint
OO/Grouping work track that already owns
[FHIR-50472](https://jira.hl7.org/browse/FHIR-50472) ("Discussions
about this ticket will continue past the R6 Normative Ballot —
therefore it will be ongoing discussion until a resolution is
proposed."). The Lab Report IG would carry only a non-normative
"Open Issue" pointing readers at FHIR-50472 and the OO/Grouping
discussions, with no new prose constraints on `DiagnosticReport`,
`Observation`, `Observation.hasMember`, or `Group`.

#### Justification

Two strong signals say a stand-alone Lab-Report-IG decision is
premature: (a) the implementers Zulip thread shows no consensus —
US EHR vendors lean Option B (DR-as-grouper), some clinicians lean
Option A (Observation-as-grouper), and Andrea Pitkus's
ServiceRequest-driven proposal is a third axis entirely; and (b) the
Persuasive vote from 2022-09-27 was already reverted (hbuitendijk,
2024-10-09) precisely because the underlying disposition needs more
discussion. Adopting Disposition A unilaterally inside this IG risks
contradicting whatever the joint OO/Grouping resolution lands on
(especially if it adopts Disposition B's `Group`-based shape), forcing
a follow-up breaking change to the IG's normative content. Declining
here is defensible because no implementer is *blocked* by the absence
of IG prose — both Option A and Option B are already in production
somewhere — and the cost of a wrong choice is higher than the cost of
waiting for the January Virtual WGM resolution.

---

### Recommendation

**Recommended disposition:** B

The Group-based proposal in Disposition B is the only option on the
table that satisfies *all three* constraints currently surfacing in
the discussion: (1) the reporter's core ask — panels live in
`DiagnosticReport`, leaf results live in `Observation`,
`Observation.hasMember` is not a panel grouper; (2) Andrea Pitkus's
and apitkus's repeated objection that flattening tiered / reflex /
microbiology panels into a Bundle of single-level DRs loses the
panel-membership relationships that are clinically and PH-reporting
critical; and (3) FHIR-50472's pushback that flipping panels to
`Observation` would be a breaking change for already-deployed
DR-as-panel implementations. Disposition A is the path of least
resistance for the current Lab Report ballot, but it explicitly
defers the multi-level case to a future core change — and the OO/UV
Lab Report Call has already converged on `Group` as that mechanism,
so spending the IG's reconciliation cycle on prose that we know we'll
revisit is poor leverage. Disposition C is the safest answer
politically but leaves implementers with no IG-level direction at
all and effectively re-runs the 2022 → 2024 cycle. Recommend OO take
B forward jointly with FHIR-50472 at the January Virtual WGM, with
Disposition A's IG-prose changes adopted as the *interim* shape of the
IG until the Group/DR.result core changes ballot.
