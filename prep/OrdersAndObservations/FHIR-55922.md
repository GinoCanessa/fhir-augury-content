## Summary

FHIR-55922 asks the FHIR Extensions Pack maintainers to make two related
adjustments to the `device-conformsTo` extension. First, expand the
extension's allowed context targets so that it can be used on
`DeviceDefinition` in addition to `Device`, on the basis that the
underlying `conformsTo` backbone now appears identically in both
resources in R5/R6. Second, attach
`http://hl7.org/fhir/StructureDefinition/version-specific-use` markers
so that the extension is only valid in versions of FHIR where the
native `conformsTo` element does not already exist (R4 and earlier),
i.e., it should not be advertised as usable in R5/R6 where the core
element is present. The reporter also flags an inconsistency: both the
extension and `DeviceDefinition.conformsTo` carry a `source` element,
while `Device.conformsTo` does not, and asks whether that asymmetry is
intentional.

The ticket does not have any cross-references in the Augury index, but
its subject overlaps directly with [FHIR-52946](https://jira.hl7.org/browse/FHIR-52946)
("Device-conformsTo extension overlaps Device backbone element"), which
was reviewed by OO/FMG after FHIR-55922 was filed. FHIR-52946 first
resolved as Persuasive with Modification ("put a ceiling of R4B on this
extension"), was then re-opened, and ultimately resolved as "Not
Persuasive with Modification" with the motion to **remove the
extension entirely**. A comment on 2026-04-13 confirms the extension
"got removed entirely" from the build, via
[HL7/fhir-extensions PR #235](https://github.com/HL7/fhir-extensions/pull/235).
Reviewers should consider FHIR-55922 in the light of that already-applied
resolution — the request to gate the extension by version is essentially
the same intent FHIR-52946 acted on, and the request to expand its
context to `DeviceDefinition` is moot if the extension no longer exists.

## Details

**Description:**

The `device-conformsTo` extension appears to be the definition to use
instead of a related cross-version definition.

The same structure appears in both `Device` and `DeviceDefinition`, but
the extension currently only lists `Device` as a valid context target.

Please expand the context to include `DeviceDefinition`.

Also, please add
`http://hl7.org/fhir/StructureDefinition/version-specific-use` tags so
that the extension is only valid in versions where the elements do not
exist.

As a note, both the extension and `DeviceDefinition.conformsTo` have a
`source` element, while `Device.conformsTo` does not. I am not sure if
that is intentional or an oversight.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir-extensions`, restore (or, if removal has not yet been
released, retain) the `device-conformsTo` extension and modify its
`StructureDefinition`:

- Add `DeviceDefinition` to the extension's `context` (a second
  `context` entry of type `element`, expression `DeviceDefinition`),
  alongside the existing `Device` context.
- Attach
  `http://hl7.org/fhir/StructureDefinition/version-specific-use`
  modifier extensions to the `StructureDefinition` so that the
  extension is declared valid only on FHIR versions where the native
  `conformsTo` backbone does not exist (i.e., R4 and earlier), and
  invalid on R5/R6.
- Decide and document the asymmetry the reporter raised: either add a
  `source` sub-element to `Device.conformsTo` (a separate OO ticket
  against the FHIR core), or constrain the extension's `source`
  sub-element so it behaves consistently across `Device` and
  `DeviceDefinition`.

#### Justification

Both `Device.conformsTo` and `DeviceDefinition.conformsTo` exist in
R5/R6 with the same backbone shape, so an extension that pre-adopts
that shape on R4-era resources should be applicable to both — there is
no design reason to allow it on `Device` only. Adding
`version-specific-use` is the standard, in-pack mechanism for marking
an extension as a back-port for older versions, and it answers the
exact concern FMG/OO acted on in FHIR-52946 (the extension overlaps
the native element in R5/R6). Doing both at once produces a coherent,
self-describing extension whose scope is unambiguous.

---

### Disposition B: Alternative Approach — close as already addressed by FHIR-52946 (R4B-and-earlier ceiling), and split the `source` asymmetry into a separate OO ticket

#### Proposal

Mark FHIR-55922 as **Already Addressed** by the actions taken under
[FHIR-52946](https://jira.hl7.org/browse/FHIR-52946):

- The `device-conformsTo` extension is being scoped to FHIR versions
  R4B and earlier (the originally-applied disposition before reopening),
  which mechanically achieves the "only valid where the elements do
  not exist" outcome FHIR-55922 asks for via
  `version-specific-use`.
- While doing that, also widen the extension's `context` to include
  `DeviceDefinition` (a small additive change inside the same PR
  cycle), so that R4B/R4 implementers can pre-adopt the R5/R6 shape
  on either resource.
- Spin out the `source`-element asymmetry between `Device.conformsTo`
  and `DeviceDefinition.conformsTo` as a new OO ticket against the
  **FHIR core** spec (not the Extensions Pack), since fixing it
  requires a change in `Device`, not in an extension.

#### Justification

FHIR-52946 has already adjudicated the underlying problem (that
`device-conformsTo` overlaps the native element in R5/R6) and FMG/OO
voted on a concrete remedy. Re-litigating in FHIR-55922 risks
contradicting that decision; adopting it explicitly here is cleaner.
A version ceiling is materially equivalent to the
`version-specific-use` annotation FHIR-55922 asks for, and is the
mechanism the pack already uses for "old-version-only" extensions.
Adding `DeviceDefinition` to the context is a minor, additive change
that costs nothing once the version ceiling is in place. The `source`
asymmetry is genuinely a different problem (it is about the core
`Device` resource, not the extension), and conflating it with this
ticket muddies both the disposition and the affected work group.

---

### Disposition C: Decline — superseded by FHIR-52946 (extension removed)

#### Proposal

Resolve FHIR-55922 as **Not Persuasive** on the grounds that the
extension whose context and version-applicability the ticket asks to
modify has already been removed from the Extensions Pack via the
resolution of [FHIR-52946](https://jira.hl7.org/browse/FHIR-52946)
and [HL7/fhir-extensions#235](https://github.com/HL7/fhir-extensions/pull/235).
For implementers needing `conformsTo` semantics on R4-era data, point
them at:

- `Device.specialization` / `Device.property` for the existing
  R4/R4B mechanisms; and
- `DeviceDefinition.regulatoryIdentifier` (R5+) or
  `RegulatedAuthorization` for regulator-issued conformance, where
  applicable (per the
  [implementers > Device - FDA approved/cleared](https://chat.fhir.org/#narrow/stream/implementers/topic/Device%20-%20FDA%20approved%2Fcleared/near/491303326) thread).

The `source`-element asymmetry should be raised as a fresh ticket
against the FHIR core spec for `Device.conformsTo`.

#### Justification

If the extension no longer exists in the pack (the 2026-04-13 comment
on FHIR-52946 states "It got removed entirely"), then neither
expanding its context nor adding `version-specific-use` markers is
actionable — there is nothing to modify. Re-introducing the extension
just to scope it down would directly contradict the FMG/OO motion to
remove it, and the FMG Extensions Pack thread shows a strong
disinclination to make in-place semantic changes to already-published
extensions. Implementers' real needs are already covered by
existing core elements and resources on the FHIR versions where they
apply.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the most defensible because it acknowledges the work
already done under FHIR-52946 without losing the legitimate piece of
this ticket — the request to make the extension applicable to
`DeviceDefinition` on R4-era data, and the request to make the
version-applicability explicit. The originally-applied "R4B ceiling"
on `device-conformsTo` is functionally equivalent to the
`version-specific-use` marking the reporter wants, so OO can deliver
that ask by reaffirming the ceiling rather than by re-opening a
debate. Adding `DeviceDefinition` to the context is a small,
backward-compatible widening that costs essentially nothing inside
the same PR window. Disposition A is too aggressive given that FMG
has signalled it does not want in-place semantic changes to published
extensions and given the standing motion to remove. Disposition C is
clean but throws away the genuinely useful `DeviceDefinition`-context
ask and leaves R4 implementers with no pre-adopt path. Splitting the
`Device.conformsTo` `source`-element asymmetry into a new OO core
ticket is also important under B, because it keeps the Extensions
Pack disposition focused and routes the core-resource fix to the
right surface.
