## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Device module narrative in FHIR Core as requested:

1. **Use Cases section** — Refresh the existing Use Cases on the
   Device module page so they reflect current Device-resource usage
   patterns. Concretely, ensure the listed use cases cover at least:
   (a) implantable / wearable patient-affixed devices referenced from
   `Observation.device`, `Procedure.usedReference`, and
   `DeviceUseStatement`/`DeviceUsage`; (b) point-of-care medical
   equipment that produces observations (e.g., infusion pumps,
   ventilators, monitors) where `Device` is the source/performer of
   measurements; (c) imaging/diagnostic equipment referenced from
   `ImagingStudy` and `DiagnosticReport`; (d) administrative /
   inventory tracking via `Device.identifier` (UDI) and
   `Device.status`; and (e) connectivity/gateway scenarios where a
   `Device` represents a hub that aggregates observations from child
   devices via `Device.parent`.

2. **Device settings guidance** — Add a new subsection to the Device
   module page (e.g., "Representing Device Settings") that explains
   how to convey the configuration / programmed parameters of a
   device. Cover the recommended pattern using
   `Device.property` (with `valueQuantity` / `valueCodeableConcept`)
   for stable settings, contrast it with `Observation` for
   measurement-time settings, and link out to relevant IGs (e.g.,
   PHD, MDI) where deeper guidance already exists. Include a worked
   example for at least one infusion / ventilation use case.

The change is documentation-only, scoped to the Device module
intro/notes pages in FHIR Core; no resource StructureDefinition or
invariant changes are implied.

#### Justification

The request is narrow, additive, narrative-only guidance from the
domain workgroup that owns the resource. The Use Cases section on
module pages is exactly the place readers go to understand intended
scope, and Device's existing list pre-dates significant ecosystem
changes (PHD, MDI, UDI mandates, Device vs. DeviceMetric). Adding
explicit guidance for "device settings" addresses a recurring source
of implementer confusion (settings vs. observations vs. metrics) and
costs nothing in interoperability — it only clarifies an existing
mechanism (`Device.property`). Accepting as requested keeps ownership
with the reporter and the OO workgroup, who are best positioned to
write the prose.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying intent but **broaden and structure the change**
into a small Device-module documentation work item rather than a
single edit:

1. Stand up a short tracking page (or umbrella ticket with child
   tickets) covering: (a) Use Cases refresh, (b) Device settings
   guidance, (c) clarifying the relationship between `Device`,
   `DeviceMetric`, `DeviceDefinition`, and `DeviceUsage`, and
   (d) cross-references to PHD / MDI / Devices-on-FHIR IGs.

2. Before drafting prose, post a short Zulip note in
   `#devices on FHIR` (or `#orders and observations`) inviting
   implementers to nominate the use cases that should appear, so the
   refresh reflects current adoption rather than the original
   authors' assumptions.

3. Place the "device settings" guidance in a reusable location (e.g.,
   the Device module notes page) and link to it from the Device
   resource page, DeviceMetric page, and Observation guidance, so
   the same guidance does not have to be repeated.

4. Treat the result as a coordinated PR against `HL7/fhir` touching
   only Device-module narrative pages; no schema changes.

#### Justification

The ticket as written ("update the use cases" and "add additional
guidance") is open-ended and risks producing prose that one author
finds reasonable but other implementers do not recognise. Splitting
into explicit work items and soliciting community input first avoids
a second round of documentation tickets six months later. It also
ensures the settings guidance is discoverable from neighbouring
resources (`DeviceMetric`, `Observation`) rather than buried only on
the Device module page. The trade-off vs. Disposition A is more
process and a slightly longer cycle time, in exchange for higher
confidence the refresh actually lands the guidance implementers need.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as currently scoped, on the grounds that it does
not articulate a concrete change. Ask the reporter to either:

- supply specific text or a redline of the use-cases section, plus
  the proposed device-settings guidance, or
- file separate, scoped tickets for each concrete edit.

Until a concrete proposal is on the table, mark the ticket
"Persuasive with Mod" / "Not Persuasive — needs more detail" or close
with the request to refile with specifics.

#### Justification

Documentation tickets that say "update X and add guidance about Y"
without supplying the change burden the workgroup with both the
authoring and the design decisions. FHIR's tracker convention favours
concrete proposals (current text → proposed text → rationale).
Accepting as-is risks the workgroup writing prose the reporter then
disagrees with, generating a second tracker. Declining is defensible
*only* if the workgroup does not have an internal champion willing to
own the rewrite; if such a champion exists (the reporter is on OO),
this disposition is overly procedural.

---

### Recommendation

**Recommended disposition:** A

The request is narrow, owned by the domain workgroup, and concerns
narrative pages only — there is no FHIR Core schema risk, no breaking
change, and no inter-spec coordination required. The Device module's
Use Cases section genuinely is dated relative to current Device usage
(UDI, PHD/MDI, Device-as-performer of Observations), and explicit
guidance for device *settings* (vs. observations vs. metrics) is a
known implementer pain point. The reporter (Marti Velezis) is on the
OO workgroup and is well placed to draft the prose, so the
disposition can be acted on directly without further discovery.
Disposition B's process improvements (Zulip canvass, cross-resource
linking) are worth folding into Disposition A as drafting *guidance*
rather than treated as a blocker. Disposition C is too procedural for
a Highest-priority workgroup-owned narrative refresh and should only
be invoked if no champion steps forward to write the text.
