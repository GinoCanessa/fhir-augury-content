## Summary

The ticket reports a binding/semantics mismatch on `Specimen.collection.device[x]` in FHIR Core. The element is described as "Device used to perform collection" (e.g., needle, swab, scalpel, biopsy forceps), but its example binding points to the **Specimen Container** value set (e.g., tube, vial, cup). The reporter (Eric Haas) argues these are semantically different — the *collection device* is the instrument used to obtain the specimen, while the *container* is the vessel that holds it afterward — and recommends rebinding `collection.device[x]` to a value set that actually enumerates collection devices.

The ticket is closely related to [FHIR-49751](https://jira.hl7.org/browse/FHIR-49751), an Applied change which converted `Specimen.processing.device` and `Specimen.container.device` from `Reference(Device)` to `CodeableReference(Device|DeviceDefinition)` to match `Specimen.collection.device`. That earlier ticket made all three device elements use `CodeableReference` but did not curate distinct value sets — leaving `collection.device` pointing at a container-oriented binding, which is exactly the gap FHIR-54488 surfaces. The drafted resolution already on the ticket ("Persuasive with Modification") proposes (1) a combined `specimen-container-type` value set used by `Specimen.container.device`, and (2) a new `collection-device` value set populated with SNOMED CT collection-device concepts (oral specimen swab, urinary catheter, aspiration needle, blood collection needle, cytology brush) for `Specimen.collection.device`, both as example bindings.

## Details

**Description:**

**Element:** `collection.device[x]` — "Device used to perform collection"

**Binding:** Specimen Container (Example)

**Issue:** The element describes a **collection device** (e.g., needle, swab, scalpel, biopsy forceps), but the value set is bound to **Specimen Container** (e.g., tube, vial, cup). These are semantically different, even if it is an example binding.

- **Collection device:** The instrument used to obtain the specimen
- **Container:** The vessel that holds the specimen after collection

**Recommendation:** Rename the binding to a collection-device-appropriate value set.

**Resolution Description (drafted on the ticket):**

We will combine the values from `valueset-specimen-container-type` and `valueset-containerdevice-code`, use that new value set called `specimen-container-type` (and remove "where concept is-a 434711009 (Specimen container (physical object))" from the value set definition) for use in `Specimen.container.device` with example binding.

We will also create a value set for use in `Specimen.collection.device` called `collection-device` with the following value set definition (remains example binding):

Include codes from `http://snomed.info/sct` (latest):

| Code | Display |
|------|---------|
| 1003474008 | Oral specimen collection swab (physical object) |
| 20568009 | Urinary catheter, device (physical object) |
| 706681000 | Aspiration needle (physical object) |
| 700956008 | Blood collection needle, basic (physical object) |
| 22566001 | Cytology brush, device (physical object) |

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already drafted on the ticket. Concretely, in the FHIR Core Specimen resource:

1. **Create a new combined `specimen-container-type` value set** that unions `valueset-specimen-container-type` and `valueset-containerdevice-code`, removing the SNOMED CT `is-a 434711009 (Specimen container (physical object))` constraint so the union is not re-narrowed to physical containers. Bind `Specimen.container.device` to this value set with **example** strength.
2. **Create a new `collection-device` value set** that includes (latest) SNOMED CT concepts for collection instruments — at minimum: 1003474008 (Oral specimen collection swab), 20568009 (Urinary catheter), 706681000 (Aspiration needle), 700956008 (Blood collection needle, basic), 22566001 (Cytology brush). Bind `Specimen.collection.device` to this value set with **example** strength.
3. Leave `Specimen.processing.device` either unbound or with a generic device-type example binding, as established by FHIR-49751.

#### Justification

This precisely matches what the ticket asks for and is consistent with the Specimen WG discussion (Zulip thread above) and FHIR-49751's intent that all three device elements behave the same structurally while carrying semantically appropriate hints. Because the bindings are *example* strength, no implementer is constrained, but the spec stops actively misleading readers who currently see container codes attached to a "device used to perform collection" element. It is a low-risk, ballot-window-friendly fix carrying the existing `SpecimenBlockVote` label.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring two new HL7-curated value sets, **remove the example binding on `Specimen.collection.device`** (leaving it `CodeableReference(Device|DeviceDefinition)` with no value set) and add a non-normative narrative note in the Specimen page pointing implementers at a published external value set (e.g., a SNOMED-curated subset on a terminology server, or one of the existing SNOMED reference sets for collection devices). Apply the same simplification to `Specimen.container.device` — keep the existing `valueset-specimen-container-type` binding only, and drop the `containerdevice-code` value set as redundant once `container.device` is a `CodeableReference`.

#### Justification

HL7 has historically over-curated small example value sets that drift from SNOMED maintenance. The five-code starter list in Disposition A is illustrative but obviously incomplete (no scalpel, no biopsy forceps — both called out in the ticket itself), and curating it inside the spec creates an ongoing maintenance burden against SNOMED CT updates. Leaving the binding open while pointing at SNOMED's own collection-device concepts respects the "example" intent better and avoids HL7 owning a tiny enumeration that is guaranteed to be incomplete. Trade-off: implementers lose the convenience of a clickable example list inside the spec, and tooling that hard-codes example-binding URLs would need to be told the binding is now intentionally unbound.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive**: do not change either binding. Document in the Specimen narrative that the existing `valueset-specimen-container-type` is intentionally reused on `Specimen.collection.device` as a low-fidelity hint, given that example bindings are non-normative and many real implementations will populate either element with codes drawn from broader SNOMED/LOINC sets anyway.

#### Justification

The bindings are already example-strength, so no conformant implementation is harmed by the mismatch — the issue is essentially a documentation clarity problem. Splitting into two new HL7-curated value sets adds spec surface and ongoing terminology maintenance for what is, in practice, a non-binding hint. If implementers want richer device coverage, they can profile and bind locally. **However**, this position runs against the explicit reporter rationale, contradicts the WG's own already-drafted resolution, and ignores the fact that the current binding actively misleads readers about what kind of code belongs there — so this disposition is included for completeness rather than as a serious recommendation.

---

### Recommendation

**Recommended disposition:** A

The ticket has already been triaged as "Persuasive with Modification" and carries the `SpecimenBlockVote` label, indicating the WG has converged on a concrete fix and is moving it through ballot disposition. The drafted resolution is internally consistent: it cleans up the redundancy created by FHIR-49751 (two container-flavored value sets where one will do), and it gives `Specimen.collection.device` a value set that actually matches its definition. Disposition B has a real long-term maintenance argument, but it loses the spec-page affordance that ballot reviewers and new implementers rely on — and FHIR routinely ships small SNOMED-backed example value sets, so doing it here is not a precedent issue. Disposition C should be rejected because the current binding is genuinely misleading on its face, even if non-normative. Recommend approving Disposition A and, as a friendly amendment, expanding the starter `collection-device` value set to also include scalpel and biopsy forceps (concepts the ticket text itself calls out) before block vote.
