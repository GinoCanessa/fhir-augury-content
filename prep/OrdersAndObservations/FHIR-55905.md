## Summary

The ticket asks the COW (Clinical Order Workflow) IG to add explicit
guidance on how a placer can mark an order "complete" — and, more
broadly, on how overall order status is exchanged between placer and
filler. The reporter (Jose Costa-Teixeira) notes that implementers had
suggested a dedicated "Mark complete" Task analogous to the
cancellation-request Task, but earlier investigation concluded a
dedicated Task was unnecessary because **sharing the updated
Coordination Task is sufficient**. The ticket therefore is *not*
asking for a new resource/profile per se; it is asking COW to make
that pattern visible and discoverable.

Three concrete asks:

1. In the Prescription and Dispense (with a single Dispense Task)
   diagram, show the **POST of the (Coordination) Task back to the
   EHR** — making it explicit that this is the normal mechanism for
   exchanging overall status.
2. Add guidance — either on the **"Events, Requests and Tasks"** page
   or, preferably, in **Core Concepts** — explaining the symmetric
   set of lifecycle interactions: orders can be **cancelled by the
   placer**, **cancellation can be requested by the filler**, and
   the order can be **marked complete by the placer** (e.g., when the
   placer notices fulfillment has occurred).
3. Consider whether a **Completion-Request Task** (filler → placer
   asking "please mark this complete") is a useful concept, and if
   not, document **when not to introduce one**.

There are **no Jira cross-references** for FHIR-55905 in the index,
but the reporter explicitly cited
[FHIR-53474](https://jira.hl7.org/browse/FHIR-53474) ("BE: Add support
for tracking progress completion / ratio", Applied / Persuasive with
Modification) in the related Zulip thread, and that ticket establishes
the same architectural pattern this ticket wants to extend: the
**Coordination Task** is the canonical surface for "how is this order
progressing / has it completed" — using `Task.status`,
`Task.businessStatus`, and (per the resolution) `Task.output` as a
ratio. FHIR-55905 should be read as the next logical step in that
program of work: making sure that placer-driven *completion* (not just
progress) is documented alongside cancellation.

### Disposition A: Accept As Requested

#### Proposal

Apply all three asks in the COW IG without altering the resource
model:

1. **Update the "Prescription and Dispense (single Dispense task)"
   diagram** in the COW IG to add an explicit interaction arrow
   showing the **filler POSTing the updated Coordination Task back
   to the placer/EHR** as the canonical signal of overall order
   status (including completion). Add a one-line caption next to
   the arrow stating that this is the normal mechanism for
   exchanging order status.
2. **Add a new sub-section to the COW "Core Concepts" page** (and
   cross-link it from "Events, Requests and Tasks") titled
   "Cancelling and completing orders", structured as a small matrix:
   - placer → cancels the order (already documented under
     `cancelling-and-modifying-requests.html`);
   - filler → requests cancellation of the order (already
     documented);
   - placer → marks the order **complete** by updating the
     **Coordination Task** (`Task.status = completed`, optionally
     with `Task.businessStatus`) and POSTing it back, e.g. when the
     placer observes that fulfillment has occurred but the filler
     has not signalled completion.
   Reuse the wording style of the FHIR-53474 resolution (which
   already pivots on the Coordination Task) for consistency.
3. **Add a short "Completion-Request Task — when *not* to use one"
   note** on the same page. State explicitly that COW does **not**
   define a dedicated Completion-Request Task because the existing
   Coordination Task and its `status`/`businessStatus` already
   cover the bidirectional signalling, and using a separate Task
   would create two competing sources of truth for "is this order
   done?". Provide a single guidance bullet: "If a filler wants the
   placer to formally close the order, the filler updates the
   Coordination Task with an appropriate `businessStatus` and lets
   the placer respond by setting `Task.status = completed`."

#### Justification

The ticket itself rules out a new resource/profile ("after further
investigation this was not needed … sharing the updated Coordination
task was sufficient"), and the related Zulip thread shows the same
WG view: Lloyd McKenzie ("It should be done using Task"), Vassil
Peytchev (Coordination Task surface), and Ralf Herzog ("COW is
generic") all converge on documentation rather than new structure.
This disposition mirrors the *Persuasive with Modification* approach
already taken on the sibling FHIR-53474 ticket — same page family
("Tracking Outputs / Tracking Execution Progress"), same Coordination
Task anchor — so reviewers and implementers will see one coherent
story across cancellation, progress, and completion.

---

### Disposition B: Alternative Approach

#### Proposal

Address only asks #1 and #2, and **explicitly defer ask #3 to a
follow-up tracker** rather than baking a "do not define a Completion
Request Task" statement into the IG now. Concretely:

- Make the diagram change in
  "Prescription and Dispense (single Dispense task)" exactly as in
  Disposition A.
- Add the matrix-style "placer cancels / filler requests
  cancellation / placer marks complete" guidance — but place it as
  a new **pattern page** ("Order Completion Signalling") parallel
  to the existing `cancelling-and-modifying-requests.html` and to
  the `tracking-progress.html` page produced by FHIR-53474, rather
  than embedding it inside Core Concepts. Cross-link from Core
  Concepts and from "Events, Requests and Tasks".
- For ask #3, file a new tracker against COW asking the WG to
  decide whether to *introduce* a Completion-Request Task (the
  reporter's literal framing was "consider the need …"), and link
  it back to FHIR-55905. Do not encode a "do not use" rule until
  that question has been deliberated.

#### Justification

This keeps the editorial change small and low-risk for the current
ballot cycle — adding a diagram arrow and a new pattern page is
non-controversial — while avoiding making a normative-flavoured
statement ("COW does not define a Completion Request Task") on the
basis of a single ticket. A symmetric pattern-page layout
(`cancelling-…`, `tracking-progress`, `order-completion-signalling`)
also makes the COW IG easier to navigate than burying the same
guidance inside Core Concepts. The trade-off vs. Disposition A is
that ask #3 is left open, so an implementer who *does* want to
define their own Completion-Request Task in a downstream IG has no
COW-level guidance to push back on yet — but that downside is
mitigated by the new follow-up tracker.

### Disposition C: Decline

#### Proposal

Decline FHIR-55905 as **already addressed by existing mechanisms**.
Rationale category: redundant with current COW guidance. Specifically:
the Coordination Task is already documented on COW pages such as
`cancelling-and-modifying-requests.html` and the post-FHIR-53474
`tracking-progress.html`, both of which already establish that order
status (including completion) is exchanged by updating the
Coordination Task and POSTing it back. Close the ticket and ask the
reporter to file targeted documentation tickets if any specific
diagram or page is genuinely missing the POST-Task arrow.

#### Justification

Declining is defensible only on a strict reading: the *base FHIR Task
lifecycle* and the *existing COW pages* already give implementers
everything they need to mark an order complete (set
`Task.status = completed` on the Coordination Task and POST it back).
On that reading, the ticket is asking for editorial polish rather
than a substantive change. However, this position ignores the
ticket's central evidence — implementers actually proposed a "Mark
complete" Task because the current guidance is *not* discoverable —
and so this disposition is offered mainly as a foil to A and B.

---

### Recommendation

**Recommended disposition:** A

The ticket is a pure documentation/diagram ask, the reporter has
already done the design analysis (concluding that no new Task type is
needed), and the related Zulip thread shows broad WG agreement that
the Coordination Task is the right surface. Disposition A converts
that consensus into three small, additive edits on existing pages and
mirrors the editorial approach the WG just took on FHIR-53474 —
producing a coherent "cancellation / progress / completion" trio of
pages all anchored on the Coordination Task. Disposition B is a
reasonable fallback if reviewers feel uncomfortable making the
"don't define a Completion-Request Task" statement without a
dedicated discussion, but splitting the ask into two trackers risks
the third question never being resolved. Disposition C should be
rejected: the very existence of the implementer suggestion the
reporter cites is direct evidence that the current guidance is
insufficient.
