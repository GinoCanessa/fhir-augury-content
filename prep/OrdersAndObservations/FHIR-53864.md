## Summary

Lorraine Constable requests that the R6 `BodyStructure` resource be extended
to capture the **origin** of the body structure — a concept currently
represented only in the *Anatomical Inventory and Person Characteristics*
(AIPC) logical model. The motivating use case is organ transplant, where it
is clinically important to know whether tissue/organ material is autologous,
non-autologous, allogenic, zooplastic, or synthetic. The ticket asks the
Orders & Observations workgroup to consider adding an element (and an
accompanying value set bound to those concepts) to `BodyStructure` so that
this information can be recorded directly on the resource rather than via
extensions or out-of-band data. No related Jira tickets, Zulip discussions,
or GitHub items were surfaced via cross-references for this change request.

## Details

**Description:**

> There are some important use cases, in particular, organ transplant where
> the concept of Origin is important. This is currently an attribute of the
> Anatomical Inventory and Person Characteristics logical model. This concept
> is not captured in the current R6 Resource. Please consider the inclusion
> of an element.
>
> The AIPC model includes the following concepts for OriginCodes:
>
> - autologous
> - non-autologous
> - allogenic
> - zooplastic
> - synthetic

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new optional element `BodyStructure.origin` of type `CodeableConcept`
(cardinality `0..1`) to the R6 `BodyStructure` resource. Bind the element
to a new (or imported) value set `BodyStructureOriginCodes` containing the
five concepts already defined in the AIPC logical model: `autologous`,
`non-autologous`, `allogenic`, `zooplastic`, `synthetic`. Use an
`extensible` binding so additional jurisdiction- or use-case-specific
codes (e.g., xenogeneic subtypes, hybrid synthetic-biologic constructs)
can be added without breaking conformance. Align the code system / value
set definition with the AIPC logical model so that conversions between
the logical model and the resource are lossless.

#### Justification

The request is well-scoped to a single resource and a small, closed set
of clinically meaningful codes that are already harmonized in the AIPC
logical model. Adding a typed element (rather than relying on an
extension) makes the concept first-class for transplant workflows,
improves discoverability, and keeps the resource consistent with the
logical model that O&O has already endorsed. `CodeableConcept` with an
extensible binding follows the FHIR convention used elsewhere in
`BodyStructure` (e.g., `morphology`, `includedStructure.structure`) and
preserves room for terminology growth. Because R6 is still in
development, adding an optional `0..1` element is non-breaking for
existing R5 implementers who choose to migrate.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add `origin` to the base `BodyStructure` resource. Instead:

1. Define a standard extension `http://hl7.org/fhir/StructureDefinition/bodystructure-origin`
   typed as `CodeableConcept` and bound (extensible) to the same
   `BodyStructureOriginCodes` value set drawn from AIPC.
2. Publish the value set (and code system, if new) as part of the FHIR
   Core terminology so that IGs and profiles can reuse it consistently.
3. Reference the extension from the `BodyStructure` resource page under
   "Extensions" so implementers in transplant contexts can discover it
   easily.

This preserves the current shape of `BodyStructure` while still giving
transplant implementers a normative, governed mechanism to capture
origin information.

#### Justification

`origin` is highly relevant to a narrow but important clinical domain
(transplant, prosthetics, regenerative medicine) and is not universally
applicable to every `BodyStructure` instance (e.g., an anatomical site
recorded for imaging has no meaningful "origin"). FHIR's modelling
guidance generally favours extensions over core elements when a concept
is domain-specific and the resource already covers most general use
cases. A standard extension keeps the base resource lean, avoids
imposing a new optional element on every implementer's tooling, and
still provides a single canonical mechanism (and value set) so different
IGs do not invent competing extensions. It also lowers the bar for
balloting since the extension can iterate independently of the resource.

---

### Disposition C: Decline

#### Proposal

Decline the request to add an `origin` element to `BodyStructure` in R6
on the grounds that the concept (a) is more naturally represented on
the supplying / procurement artifact (e.g., a `Substance`, `Device`,
`BiologicallyDerivedProduct`, or transplant-specific resource owned by
another workgroup) than on `BodyStructure`, and (b) the AIPC logical
model is the appropriate place for the origin concept until a concrete
implementer ballot comment with mapping evidence demonstrates that
`BodyStructure` itself is the right home. Recommend the reporter raise
a coordinated request with Patient Care / BR&R for representation on
`BiologicallyDerivedProduct` if the underlying need is transplant
material provenance.

#### Justification

`BodyStructure` describes anatomical structures of a specific patient;
the *origin* of grafted/transplanted material is arguably a property of
the **material** being placed, not of the resulting anatomical
structure. FHIR already has resources purpose-built for biologic
provenance (`BiologicallyDerivedProduct`, `Substance`, `Device`) which
are owned by other workgroups and which can be linked to a
`BodyStructure` via existing references and extensions. Adding `origin`
to `BodyStructure` risks duplicating provenance information that should
live on the source resource and creating ambiguity about authoritative
truth when both are present. With no Zulip discussion, no linked
tickets, and no implementer comments demonstrating that the existing
mechanisms are insufficient, the safer path is to decline and redirect.

---

### Recommendation

**Recommended disposition:** B

Disposition B (standard extension) best balances the legitimate clinical
need raised by the reporter against FHIR's preference for keeping core
resources focused. The five OriginCodes are already harmonized in the
AIPC logical model, so publishing them as a governed value set bound to
a standard extension gives transplant implementers a single, normative
mechanism without forcing an optional element onto every `BodyStructure`
consumer (most of whom are recording anatomical sites for imaging,
procedures, or observations where "origin" is meaningless). It also
avoids the cross-resource duplication risk highlighted in Disposition C
while still being more concrete and actionable than a decline. If
implementer feedback during the R6 ballot shows the extension is being
universally adopted in transplant IGs, the workgroup can promote it to
a core element in a future release with strong evidence in hand —
something that is much easier than removing or deprecating a core
element added speculatively now.
