## Summary

The reporter (Elliot Silver) raises a quality-and-clarity concern about the
`DeviceAssociation` resource. Four status-like elements —
`DeviceAssociation.relationship`, `.status`, `.statusReason`, and
`.associationStatus` — are perceived to overlap in scope, and their definitions,
ValueSet bindings, and binding strengths are not yet clearly differentiated.
The ticket asks O&O to (a) sharpen each element's definition so the four are
mutually distinct, (b) confirm `required` bindings are genuinely comprehensive
(and that other bindings are loosened only where appropriate), (c) clarify how
`relationship` interacts with `status` and `period` (e.g., does `relationship`
flip from "implanted" to "explanted" when a device is removed, and which event
does `period` then describe?), (d) tighten the narrative blocks (Boundaries &
Relationships, Scope & Usage, Notes, plus per-element `definition` /
`comment`) so they cover the full intended scope, (e) clarify the relationships
between `subject`, `focus`, and `bodyStructure`, and (f) reconsider whether
`bodyStructure` should be `0..*` and, if so, whether a code is needed to
identify what each repeating structure represents.

No related Jira tickets, GitHub items, or directly-related Zulip threads were
surfaced via cross-references; an opportunistic Zulip search returned only
tangential mentions of `DeviceAssociation` (search semantics, R5 adoption
guidance) that do not address the overlap question.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a quality pass against `DeviceAssociation` and land the
following changes in `HL7/fhir` under `source/deviceassociation/` (resource
StructureDefinition `<differential>` plus the resource's narrative HTML pages):

1. **Disambiguate the four status-like elements** by rewriting `definition` and
   `comment` on each:
   - `relationship` — the *clinical/physical* nature of the link between the
     device and the subject/focus (e.g., implanted, attached, used-with,
     associated-with). It is **not** a workflow state and **not** a reason.
   - `status` — the *lifecycle/operational* state of the association record
     itself (e.g., active, completed, entered-in-error). Aligned with the
     standard FHIR resource-status pattern.
   - `statusReason` — *why* `status` holds its current value (e.g., reason an
     association is `completed` or `entered-in-error`).
   - `associationStatus` — the *clinical state of the association in the real
     world* (e.g., implanted-and-functioning, implanted-but-failing, removed),
     distinct from both record lifecycle and physical relationship type.

2. **Tighten the bindings.** Re-review each ValueSet:
   - `status` → `required` to the standard resource-status code system; confirm
     the codes are truly comprehensive for record lifecycle.
   - `relationship` → `required` (or `extensible` if implementer use cases
     show coverage gaps) and ensure the ValueSet enumerates the complete set
     of physical/clinical link types.
   - `associationStatus` → `extensible` (consistent with most clinical-state
     elements), with the ValueSet expanded to cover the implanted/explanted
     lifecycle plus active-failure states.
   - `statusReason` → `example` (free-form reasons are common).

3. **Clarify the `relationship`↔`status`↔`period` interaction** in the element
   `comment` and in the resource's *Notes* page: a single `DeviceAssociation`
   instance models one relationship (e.g., "implanted"); when the device is
   removed, the existing instance moves to `status = completed` with `period.end`
   marking the end of *that* relationship, and a new `DeviceAssociation` with
   `relationship = explanted` (or equivalent) is created if the explanted state
   itself needs to be tracked. Document this explicitly with an example.

4. **Rewrite the narrative blocks** (Boundaries & Relationships, Scope & Usage,
   Notes) so they describe the full scope of the resource — including
   non-implant uses (wearables, attached external devices, associations with
   non-Patient subjects, body-structure-level associations).

5. **Change `bodyStructure` to `0..*`** and add a sibling code element (e.g.,
   `bodyStructure.role` via a BackboneElement, or model `bodyStructure` as a
   tuple of `{structure, role}`) so that, when multiple structures are listed,
   each one's meaning (primary vs. secondary attachment site, lead position,
   etc.) is explicit.

#### Justification

Each item maps directly to a question raised in the ticket and is grounded in
existing FHIR patterns: the status/statusReason split is the standard pattern
already used across resources (Task, ServiceRequest, etc.); separating
record-lifecycle (`status`) from real-world clinical state
(`associationStatus`) mirrors `Encounter.status` vs. `Encounter.class` and
`MedicationRequest.status` vs. `.intent`. Reaffirming `required` only where
the value space is truly closed — and dropping to `extensible`/`example`
elsewhere — is consistent with the FHIR binding-strength guidance. Promoting
`bodyStructure` to `0..*` plus a role code resolves the ambiguity the reporter
flags without breaking the existing single-cardinality use (a single entry
with no role still parses unambiguously).

The change is confined to a single FhirCore authoring area
(`source/deviceassociation/`); per the `HL7/fhir` briefing the differential
plus narrative HTML pages are hand-edited and the publisher regenerates the
snapshot, so there are no cross-repo dependencies.

### Disposition B: Alternative Approach

#### Proposal

Split the ticket into two resolutions:

1. **Definition + narrative + binding cleanup (in this ticket).** Land items
   1, 2, 3, and 4 from Disposition A — i.e., rewrite `definition` and `comment`
   on `relationship`, `status`, `statusReason`, and `associationStatus`;
   reaffirm or relax bindings; clarify the `relationship`↔`status`↔`period`
   semantics; refresh the narrative pages. **Do not change cardinality or add
   new elements** under this ticket.

2. **Defer the `bodyStructure` cardinality change to a separate ticket.**
   Open a follow-on Change Request to discuss whether `bodyStructure` should
   become `0..*` and, if so, whether the right design is repeating
   `bodyStructure`, a new BackboneElement with a role code, or an extension on
   `BodyStructure` itself. Tag with `BodyStructure` so Patient Care / O&O can
   weigh in on impact to other resources that reference body sites.

Additionally, consider modelling clarity by introducing an **invariant** on
`DeviceAssociation` rather than only prose, e.g., "If `status = completed`
then `period.end` SHALL be present" and "`relationship` and `associationStatus`
SHALL NOT both encode device removal", so the disambiguation is enforceable
rather than purely documentary.

#### Justification

The four sub-questions in the ticket are not equally tractable. Items 1–4 are
spec-quality work that O&O can land within a normal maintenance cycle, with
low implementer impact (definitions and bindings tighten without breaking
existing instances, assuming the reaffirmed `required` ValueSets remain
supersets of today's). Item 5 (`bodyStructure` cardinality and a possible new
role code) is a structural change that affects every implementer who has
already built against `0..1`, and may interact with how other resources
(`Procedure`, `Condition`, `Observation.bodyStructure`) reference body sites.
Splitting prevents the narrative cleanup from being held up by the modelling
debate, and it gives the cardinality change its own visible Jira track for
implementers to follow. Encoding the disambiguation as invariants (rather than
only prose) makes Disposition B more durable than Disposition A — implementers
who skip narrative pages still see the rules at validation time.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Modification → No Change**, on the basis
that the four elements are already documented to play distinct roles
(`relationship` = link type, `status` = record lifecycle, `statusReason` =
reason for that lifecycle value, `associationStatus` = real-world clinical
state) and that the perceived overlap is a documentation discoverability
issue rather than a modelling defect. Decline the cardinality change to
`bodyStructure` on the basis that no implementer use case has been provided
showing `0..1` is insufficient.

#### Justification

`DeviceAssociation` is a relatively new resource (R5) and its element set was
designed deliberately, drawing the lifecycle/clinical-state split from
`Encounter` and `Condition`. A blanket "ensure quality" pass without concrete
counter-examples risks churning the resource without measurable benefit —
particularly the binding-strength changes, which can break downstream IGs
that have already constrained against the current bindings. Without specific
use cases for repeating `bodyStructure`, expanding the cardinality adds
optional complexity that every conformant server must now handle.

---

### Recommendation

**Recommended disposition:** B

This ticket is genuinely a quality pass — the reporter is the resource's own
author flagging concerns ahead of further uptake, which is exactly when O&O
should act — so Disposition C (decline) understates the value. Disposition A
is the right *substance*, but bundling a cardinality change to `bodyStructure`
together with a definition/binding cleanup risks blocking the easy wins behind
a longer modelling discussion. Splitting per Disposition B lets O&O land the
narrative + definition + binding clarifications quickly under
`source/deviceassociation/` (a focused, single-repo edit per the `HL7/fhir`
briefing), while giving the cardinality question its own ticket where Patient
Care and IG authors can be looped in. The invariant-based reinforcement
(rather than prose-only) is the durable way to keep the four status-like
elements from drifting back into ambiguity in future maintenance cycles.
