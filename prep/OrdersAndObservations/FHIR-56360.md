## Summary

This ticket requests a structural change to the `VisionPrescription`
resource: move the `product` `CodeableConcept` element from the
`lensSpecification` backbone element up to the top of the resource. As
designed today, a single `VisionPrescription` instance can mix
prescription products (e.g., glasses and contact lenses) because
`product` is repeated per `lensSpecification`. The reporter argues this
is problematic because (a) in the common case of a single product type
it carries redundant data, (b) it forces oversharing when a user only
wants to share one product (e.g., a SMART Health Cards use case where
the patient wants to share only their contact-lens prescription, not
their reading-glasses prescription), and (c) it bloats payloads. The
proposal is that one `VisionPrescription` instance represents exactly
one product type, aligning usage with the one-request-per-product
pattern seen elsewhere in FHIR.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
found for this ticket.

## Details

**Description:**

Currently, one VP resource can define different products rather than
requiring one resource for a specific prescription type. In most cases
- one prescription type - this adds redundant information. In other
cases - such as sharing a specific prescription type - this means
oversharing. When thinking about a use case with SMART Health Cards,
this issue becomes more visible: a) a user is unable to share e.g.
just their glasses prescription, not their reading glasses, and b) it
bloats the payload.

The proposal is to move the `product` codeable concept out of
`lensSpecification` to the top level. This will necessarily mean one
VP resource per prescription type, which makes usage clearer and
removes the above issues.

**Comments:**

- **mckenzie (2026-04-16):** This would be consistent with
  MedicationRequest. While it's true that multiple meds are often part
  of the same "prescription", each is treated as a distinct Request
  instance. They can be filled independently, and in theory the status
  of an order for contacts might be revoked while not revoking the
  order for reading glasses.

## Proposed Dispositions

> Repo Context is empty because no GitHub cross-references were
> surfaced for this ticket. The change nevertheless clearly targets the
> FHIR Core `VisionPrescription` resource definition (conceptually
> under `HL7/fhir` `source/visionprescription/`), and the dispositions
> are scoped accordingly without making specific claims about file
> paths that would require a current briefing.

### Disposition A: Accept As Requested

#### Proposal

Move `product` from `VisionPrescription.lensSpecification` to the top
of the `VisionPrescription` resource as `VisionPrescription.product`
(1..1, `CodeableConcept`, bound to `vision-product` with the same
strength it has today). Remove `product` from the `lensSpecification`
BackboneElement. Keep `lensSpecification` as 1..* so a single product
(e.g., a pair of glasses) can still carry separate left- and right-eye
specifications. Update the resource narrative, element short/definition
text, examples, and search parameters referencing
`lensSpecification.product` accordingly; add a release-note entry
flagging this as a breaking change for R6 and document the migration
guidance (split existing multi-product instances into one
`VisionPrescription` per product).

#### Justification

Lifting `product` to the top level enforces the intended
"one prescription = one product" semantics at the schema level,
eliminates the oversharing and payload-bloat problems the reporter
describes, and brings `VisionPrescription` structurally in line with
`MedicationRequest`, where each requested product is its own Request
instance (as mckenzie notes in the comment). It also makes
`VisionPrescription` trivially filterable/splittable for SMART Health
Cards-style selective sharing without resorting to custom filtering
logic on `lensSpecification.product`.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `product` on `lensSpecification` but add a new required invariant
on `VisionPrescription` stating that all `lensSpecification` entries
in a single instance SHALL share the same `product` code (e.g.,
`vp-1: lensSpecification.product.coding must be identical across all
lensSpecification elements`). Optionally, also add a top-level
`VisionPrescription.product` computed/derived element or a profile
that constrains `lensSpecification.product` to 1..1 with a fixed value
per instance. Provide guidance that multiple products MUST be
represented as multiple `VisionPrescription` resources.

#### Justification

This achieves the same semantic outcome — one product per resource —
without a breaking structural change, preserving backward
compatibility for any existing `VisionPrescription` implementers who
are already sending a single product per resource. Validators can
enforce the invariant immediately, and implementers who today
(incorrectly or otherwise) mix products in one instance are explicitly
steered toward the new pattern. The trade-off versus Disposition A is
that the schema remains slightly awkward (a repeated `product`
that must all agree) and that oversharing via selective element
filtering at the `lensSpecification` level is not structurally
prevented — only rejected by validation.

---

### Disposition C: Decline

#### Proposal

Leave the structure of `VisionPrescription` unchanged and document
(in the resource's usage notes) that implementations MAY choose to
issue one `VisionPrescription` per product type when separability or
selective sharing matters, and that multi-product instances remain a
legitimate representation where all products were prescribed together.

#### Justification

`VisionPrescription` is a relatively low-traffic resource and the
current shape was deliberately designed to let a single clinical
encounter (e.g., an optometrist visit that produces both a glasses
prescription and a contact-lens prescription) be captured in one
resource. Restructuring for the SMART Health Cards selective-sharing
use case imposes a breaking change on all existing implementers to
serve a use case that can be addressed at the application layer
(split-on-share) or via a profile. No Zulip discussion or other Jira
tickets have been surfaced signalling broad community demand, and
R6-level breaking changes to a mature resource warrant a higher bar
than a single reporter's use case.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest long-term answer and is directly
reinforced by the first commenter's observation that it aligns
`VisionPrescription` with `MedicationRequest`'s established
one-request-per-product pattern. The reporter's SMART Health Cards
motivation is a concrete, realistic sharing scenario, and the current
structure actively encourages oversharing because `product` is
hidden inside a repeating backbone. Disposition B preserves
compatibility but leaves a structurally awkward shape (a repeated
element that must always agree) and still requires implementers to
change behavior, so most of Disposition A's migration pain without
most of its clarity benefit. Disposition C punts a fixable modelling
problem to every downstream implementer. Given that this is filed
against FHIR Core as a Change Request with Highest priority and we
are at an R6 development window where such structural cleanups are
appropriate, the workgroup should accept the change and use the
comment period to collect migration guidance for existing
implementers.
