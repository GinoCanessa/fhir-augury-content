## Summary

The ticket asks the Orders & Observations work group (jointly with Patient
Care) to clarify the documentation of the `start` and `end` input parameters
of the `$docref` operation on `DocumentReference` in FHIR Core (R6), as well
as in the parallel IPA and US Core copies of the operation. The current
parameter description mixes two distinct concepts — "care dates" (the
clinical period the document is *about*) and "record currency dates" (when
the `DocumentReference` itself was created) — and is also silent on the
exact behaviour when neither bound is supplied (a single "most recent
document", or all documents tied to the most recent encounter?). The
reporter (Eric Haas) provides a Grok-generated rewording that fixes both
the meaning ("clinical date range … e.g., encounter or service dates") and
the empty-input case ("returns clinical documents associated with the most
recent encounter or care event"), and asks that all three specifications
(FHIR R6, US Core, IPA) be revised in lockstep.

A companion US Core ticket, [FHIR-52869](https://jira.hl7.org/browse/FHIR-52869)
("clarify $docref operation behavior, following opportunity for implementer
input"), was filed by Isaac Vetter with a slightly different — and more
prescriptive — wording produced by Heath Frankel out of the same IPA
discussion. FHIR-52869 has already been **Applied / Persuasive** against US
Core; it explicitly retitles the parameters as "start context date" / "end
context date", binds them to `DocumentReference.context.period`, and
renames "record currency dates" to "document reference creation dates".
FHIR-52205 is therefore the OO-owned tracker that needs to (a) decide the
authoritative wording for **FHIR R6 base**, and (b) confirm whether IPA
should pick up the US-Core wording verbatim or adopt Eric's alternative
phrasing.

## Details

**Description:**

> See this conversation on Zulip: [#IPA > $docref](https://chat.fhir.org/#narrow/channel/261969-IPA/topic/.24docref/with/537199584)
>
> generated by Grok:
>
> **Issue Summary**
>
> The `$docref` operation's start date parameter definition in FHIR (US
> Core, IPA, and FHIR R6) is ambiguous, leading to implementation disputes.
> The current definition mixes references to "care dates" (relating to the
> care provided) and "record currency dates" (relating to document creation
> or update). Additionally, the behavior when no start or end date is
> provided is unclear, causing confusion about whether the "most recent
> document" or "records relating to the most recent encounter" should be
> returned.
>
> **Proposed Updated Definition**
>
> **start:** The start date and end date parameters define the clinical
> date range (e.g., encounter or service dates) of the care documented. If
> no start date is provided, all clinical documents with care dates before
> the end date are in scope. If neither a start date nor an end date is
> provided, the operation returns clinical documents associated with the
> most recent encounter or care event. Clients **SHOULD** provide date
> values precise to the second with a time offset (e.g.,
> `2025-09-01T00:00:00+10:00`).
>
> **end:** The start date and end date parameters define the clinical date
> range (e.g., encounter or service dates) of the care documented. If no
> end date is provided, all clinical documents with care dates after the
> start date are in scope. If neither a start date nor an end date is
> provided, the operation returns clinical documents associated with the
> most recent encounter or care event. Clients **SHOULD** provide date
> values precise to the second with a time offset (e.g.,
> `2025-09-01T00:00:00+10:00`).
>
> Revise the `$docref` operation documentation in US Core, IPA, and FHIR
> R6 to reflect the proposed definition.

**Comments:**

- **john_moehrke (2026-02-03):** Notes IHE has no stake here. Agrees that

  confusing given the many date-bearing fields, including `lastUpdated`).
  Believes the intent is clinical dates, but flags the practical
  complication that visit notes are often entered well after the visit, so
  the proposed wording may not fully cover late-entered records.

## Related Zulip Discussions

### [IPA > $docref](https://chat.fhir.org/#narrow/stream/IPA/topic/%24docref)

The originating thread. **Grahame Grieve (2025-09-01)** opened by saying
the start-date definition caused a real argument in Australia: it conflates
"the date of the document" with "the date range of the content in the
document". **Eric Haas (2025-09-02)** acknowledged the wording dates back
to Argonaut DSTU2 and committed to filing a US-Core tracker that would be
cloned to IPA and FHIR R6 — i.e., this exact ticket. **Isaac Vetter
(2025-09-10)**, summarising the IPA call, reported emerging consensus that
`start` maps to `DocumentReference.context.period` (the search parameter
`period`) and `end` maps to `DocumentReference.date` (the search parameter
`date`), with the IPA group leaning toward making both refer to the
**care/context period** because that is the more clinically useful axis;
he also raised three open questions (behaviour with `on-demand=true`;
single vs. multiple documents when both bounds are absent; the fact that
`DocumentReference.date` is the *reference* creation date, not the
document body's creation date, which lives in
`DocumentReference.content.attachment.creation`). **Heath Frankel
(2025-09-10, three messages)** pushed back: US Core's CapabilityStatement
just says "date" with the same ambiguity, the `context.period` element is
inconsistently populated across document types (commonly populated for
discharge summaries, frequently empty for pathology results), and a
parameter that targets a sparsely populated element will be useless in
practice; he asked what conformance testers actually expect. He then
proposed the four-bullet edit list that became FHIR-52869. **Brett
Marquard (2025-09-14)**: "Please submit a tracker!" **Isaac Vetter
(2025-09-15)** filed FHIR-52869 and tagged Heath. **Heath Frankel
(2025-12-05)** circled back asking whether equivalent cards exist for R6
and IPA, since FHIR-52869 was Applied only on US Core and R6 is heading to
ballot — that is the gap FHIR-52205 has to close.

Participants: Grahame Grieve, Eric Haas, Heath Frankel, Isaac Vetter,
Brett Marquard.

### [implementers > Document Reference and $docref operation - date Q](https://chat.fhir.org/#narrow/stream/implementers/topic/Document%20Reference%20and%20%20%24docref%20operation%20%20-%20date%20Q)

A 2022 implementer thread that prefigures the same dispute. **Lital
Inghel** asked whether `start`/`end` align with `DocumentReference.date`

to settle.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt Eric Haas's proposed wording verbatim in the `$docref`
`OperationDefinition` for FHIR Core (R6), IPA, and US Core. Concretely,
replace the existing `parameter[name=start].documentation` and
`parameter[name=end].documentation` text with the two paragraphs given in
the ticket description, namely:

> The start date and end date parameters define the clinical date range
> (e.g., encounter or service dates) of the care documented. If no
> {start|end} date is provided, all clinical documents with care dates
> {before the end date | after the start date} are in scope. If neither a
> start date nor an end date is provided, the operation returns clinical
> documents associated with the **most recent encounter or care event**.
> Clients SHOULD provide date values precise to the second with a time
> offset (e.g., `2025-09-01T00:00:00+10:00`).

In FHIR Core, the file to edit is
`source/documentreference/operationdefinition-DocumentReference-docref.xml`
(this is the spec source, not a generated artifact). In US Core the file
is `input/resources-yaml/OperationDefinition-docref.yml` (authoring
source), with the JSON copy at `input/resources/operationdefinition-docref.json`
regenerated. IPA's `$docref` operation page is updated correspondingly.
Re-open / supersede FHIR-52869 so the US Core wording is replaced with
the unified text rather than left as Heath's variant.

#### Justification

This is what the reporter explicitly asked for and the only proposal that
keeps the three specifications **literally identical**, which is the
deliverable Isaac Vetter himself called for ("should be kept in sync with
R6 base and IPA"). The wording resolves both ambiguities in one move:
"clinical date range (e.g., encounter or service dates) of the care
documented" cleanly excludes record-currency dates without committing to
*one specific* element (`context.period` vs. `serviceEvent/effectiveTime`
vs. another), which sidesteps Heath Frankel's sparsity concern about
`DocumentReference.context.period` (Zulip 2025-09-10): a server with
empty `context.period` can still satisfy the contract by reasoning over
encounter or service dates from related resources. The "most recent
encounter or care event" answer to the empty-input case is also more
clinically meaningful than "the most recent document" — it directly
addresses Eric Haas's original Argonaut intent ("the latest document may
not be about the most recent encounter").

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the **already-applied US Core wording from FHIR-52869** as the
canonical text and propagate it unchanged to FHIR R6 and IPA, rather than
introducing Eric's alternative phrasing. Concretely, in the FHIR R6 and
IPA `$docref` `OperationDefinition`s:

1. Rename the prose for the parameters to "start **context** date" / "end
   **context** date" (the parameter `name`s themselves remain `start` /
   `end` for wire compatibility).
2. Change "different from a FHIR RESTful query on DocumentReference by
   patient and type and **date** range" to "… and **period** range".
3. Rewrite the start narrative as: "The **context** date range relates
   to care **service** dates, not **document reference creation** dates —
   e.g., all records relating to care provided in a specific date range.
   If no start **context** date is provided, all documents before the
   end **context** date are in scope."
4. Same edit in the end narrative.

Add a single new sentence — drawn from Eric's wording — covering the
empty-input case: "If neither a start nor end context date is provided,
the operation returns the document(s) associated with the most recent
encounter or care event." Optionally include the SHOULD-precision
guidance from Eric's text. Then close FHIR-52869 (already Applied) as
the prior art and resolve FHIR-52205 by reference, noting that R6 and
IPA now match US Core. The same authoring files apply as in Disposition A.

#### Justification

FHIR-52869 has already been worked through by US Core, reviewed
(`BAM-Reviewed`, `EH-Review`), and Applied — re-litigating the exact
wording risks creating *new* divergence between US Core and the other
two specs, which is the opposite of the stated goal. The "context date"
qualifier is also more discoverable for implementers than Eric's
unqualified "clinical date range", and explicitly contrasting against
"document reference creation date" is more precise than "record
currency", which Grahame Grieve himself flagged on Zulip as
problematic terminology. Heath Frankel's sparsity concern about
`DocumentReference.context.period` (2025-09-10) is real but is properly a
*conformance* matter for the work groups to address separately (e.g., via
implementation guidance or a future tracker on the population obligation
of `context.period` on US Core profiles); it should not block the
documentation clarification. John Moehrke's late-entered-notes concern
(2026-02-03) is unaffected either way — the parameter still refers to
care dates, not entry dates — and arguably is *better* served by the
explicit "context date" framing than by Eric's wording.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-52205 as a duplicate of [FHIR-52869](https://jira.hl7.org/browse/FHIR-52869),
which has already been resolved Persuasive and Applied to US Core, and
take no normative change in FHIR R6 base. Optionally add a non-normative
note on the FHIR R6 `OperationDefinition` page pointing implementers to
the US Core wording as informative guidance, but do not modify the
parameter `documentation` strings in the R6 source. For IPA, defer to
the IPA workgroup to track its own change.

#### Justification

Eric Haas commented "duplicate" on FHIR-52869, which is consistent with
treating FHIR-52205 as the OO-side tracker for an issue already settled
in another work group. Declining keeps the FHIR R6 ballot scope
narrower — Heath Frankel's 2025-12-05 question on Zulip ("understand this
is about to go to ballot?") explicitly raises the cost of late-cycle
text changes to a normative resource. This option is defensible *only*
if OO judges that the existing R6 wording, while ambiguous, is no worse
than what has been in production since DSTU2, and that implementers in
practice already converge on `DocumentReference.context.period` /
`DocumentReference.date` (per the 2022 implementers thread).

---

### Recommendation

**Recommended disposition:** B (adopt the US Core / FHIR-52869 wording in
R6 and IPA, with a single added sentence covering the empty-input case).

The work group's goal here is *consistency across the three specs*, and
FHIR-52869 is the only wording that has already been peer-reviewed and
applied. Re-running the editorial debate on Eric's alternative text in R6
ballot-prep timeframes (raised explicitly by Heath Frankel on
2025-12-05) is a real risk: any divergence introduced now would force a
follow-up tracker on US Core to re-align, which is exactly the loop the
ticket is trying to break. Disposition A is reasonable as a fallback if
the workgroup decides Eric's "clinical date range" framing is genuinely
clearer than "context date range relates to care service dates"; if so,
the workgroup should commit to *also* re-opening FHIR-52869 so all three
specs end up identical. Disposition C should be rejected unless OO has
strong evidence that R6 ballot scope cannot absorb the change — the
thread participants (Grahame, Eric, Heath, Isaac, Brett) are unanimous
that the current wording is broken, and shipping R6 with known-broken
parameter documentation for `$docref` would invite ballot comments that
cost more time than the fix itself.
