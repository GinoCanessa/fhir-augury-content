## Summary

The reporter requests that the `VisionPrescription` resource gain an
explicit field for capturing when the prescription expires. Today, the
specification implies that expiration must be derived from `dateWritten`
plus jurisdiction-specific rules (typically 1–2 years), which leaves
implementers with no interoperable way to represent the actual expiry
date that appears on a prescription. The proposed change is to add a new
top-level `expirationDate` element with cardinality `0..1` to
`VisionPrescription`.

A single comment on the ticket from `mckenzie` proposes an alternative
shape: model validity as an `effectivePeriod` (start/end), aligning
`VisionPrescription` with other Request resources where the period over
which an authorization is in force is already a familiar pattern.

No related Jira tickets, Zulip discussions, or GitHub items were
surfaced via cross-reference or text search.

## Details

**Description:**

> Vision prescription in many jurisdictions have an expiration date,
> usually 1 or 2 years. The current spec intends to use the
> `dateWritten` date to calculate expiration. However, because validity
> depends on jurisdictions, there is a desire to have an explicit field
> for the expiration date.
>
> The proposal is to add a top level `expirationDate` property (0..1)
> to satisfy this need.

**Comments:**

- **mckenzie (2026-04-16):** "I'd actually argue with an
  `effectivePeriod` similar to other Request resources. While it's
  typical that VisionPrescriptions are effective immediately, that
  doesn't have to be the case."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new top-level element `VisionPrescription.expirationDate` of type
`date` with cardinality `0..1` to the `VisionPrescription` resource in
FHIR Core. Document its semantics as: "The date on which this
prescription is no longer valid for dispensing, as determined by the
prescriber and/or jurisdictional rules. When absent, expiration is
implementation- or jurisdiction-defined relative to `dateWritten`." Do
not change `dateWritten`. Add a short paragraph in the resource notes
distinguishing `dateWritten` (when authored) from `expirationDate` (when
no longer valid).

#### Justification

This is the smallest, most additive change that directly satisfies the
reporter's request and matches the wording of jurisdictional
prescription rules, where the expiration is a single calendar date
rather than a period. `0..1` preserves backward compatibility with
existing instances. A `date` (not `dateTime`) is appropriate because
prescription expiry is consistently expressed as a calendar day in
optical practice. No existing element is repurposed, so implementations
that ignore the new element continue to function.

---

### Disposition B: Alternative Approach

#### Proposal

Per the comment from `mckenzie`, add `VisionPrescription.effectivePeriod`
of type `Period` with cardinality `0..1` instead of an `expirationDate`
element. Define `effectivePeriod.start` as the date the prescription
becomes valid (defaulting conceptually to `dateWritten` when omitted)
and `effectivePeriod.end` as the expiration date. This aligns
`VisionPrescription` with the broader Request pattern (e.g.,
`MedicationRequest.dispenseRequest.validityPeriod`,
`ServiceRequest.occurrencePeriod`, `NutritionOrder` timing) where
authorization windows are modeled as periods.

#### Justification

Modeling validity as a `Period` is more expressive than a single
`expirationDate`: it captures both deferred-start prescriptions (e.g.,
"valid starting next week, expires in 2 years") and the common
immediate-start case via an absent or equal `start`. It also yields
better cross-resource consistency with other Request-style resources,
which lowers the cognitive load on implementers writing generic
"in-force prescription" queries and UIs. The trade-off versus
Disposition A is a slightly heavier datatype and a small risk that
implementers populate only `end` and re-derive `start` from
`dateWritten`, but this risk is mitigated by clear notes.

---

### Disposition C: Decline

#### Proposal

Decline the change on the basis that expiration of a `VisionPrescription`
is fully derivable from `dateWritten` plus jurisdictional rules, and
that capturing those rules is out of scope for FHIR Core. Where
jurisdictions need to convey a concrete expiration, recommend an
extension (e.g., `http://hl7.org/fhir/StructureDefinition/vision-prescription-expirationDate`)
defined in a regional IG rather than a core element.

#### Justification

A core change adds a globally-visible element to support a need that is
inherently jurisdictional, which is exactly the case extensions are
designed for. Adding a core element risks implementations diverging on
whether absent means "indefinite", "use `dateWritten`", or
"jurisdiction-defined", which would weaken interoperability rather than
improve it. Existing FHIR practice favors extensions for jurisdictional
data carried on a small number of resources.

---

### Recommendation

**Recommended disposition:** B

Disposition B (`effectivePeriod`) is the strongest middle ground. It
directly addresses the reporter's underlying need — a machine-readable
"this prescription is no longer valid after X" — while also covering
the `mckenzie` use case of deferred-start prescriptions with a single
element, and aligning `VisionPrescription` with the rest of the Request
family (`MedicationRequest.dispenseRequest.validityPeriod`,
`ServiceRequest.occurrencePeriod`, etc.) for cross-resource consistency.
Disposition A is reasonable and lower-risk but locks the workgroup into
a narrower model that would likely attract a follow-up request for a
start date the first time a deferred-start use case appears.
Disposition C undersells a real interoperability gap: even though
*length* of validity is jurisdictional, the *expiration date itself* is
a concrete fact written on the prescription, and pushing that to an
extension imposes burden on every implementer in every jurisdiction
that needs it. Recommend bringing Disposition B to the workgroup with
Disposition A as the documented fallback if `Period` is judged
overkill.
