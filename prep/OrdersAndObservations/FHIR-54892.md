## Summary

The ticket requests that the value `specimen-in-process` be removed from
the `Observation.status` value set (the required binding for
`Observation.status`). The reporter argues that the other codes in
`ObservationStatus` describe the state of the Observation resource
itself (e.g., `registered`, `preliminary`, `final`, `amended`,
`corrected`, `cancelled`, `entered-in-error`, `unknown`), whereas
`specimen-in-process` uniquely describes an intermediate workflow state
of the pre-analytic/analytic procedure that produces one particular
family of Observations (laboratory). The reporter contends that the
existing `registered` status — "the existence of the observation is
registered, but there is no result yet available" — already covers the
"we know the observation is coming, results not yet ready" case, and
that workflow states belonging to the specimen-handling or test-running
process should be represented on `Procedure`, `ServiceRequest`,
`Task`, or other workflow resources rather than on the data resource
itself.

No Jira, Zulip, or GitHub cross-references were returned for this
ticket. The only cross-reference on record is the FHIR artifact
mention of `Observation.status` itself. There are therefore no linked
or directly related tickets to incorporate into the summary.

## Details

**Description:**

Most of the values for Observation.status describe the status of the
observation resource.

One value describes one intermediate workflow status of one procedure
that produces one kind of observations.  Recommend removing
"specimen-in-process" from this codeset.

The status Registered "The existence of the observation is registered,
but there is no result yet available" covers the situation.

It's reasonable to want to know if a specimen is in process, but
Procedure/Workflow resources should be used to track the workflow
status of procedure(s) producing Observation data (e.g. whether
specimens have been collected, prepared, analyzed, or whether images
have been acquired, QA'd, interpreted, etc.).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the code `specimen-in-process` from the `ObservationStatus`
value set (the required binding for `Observation.status`) in FHIR
Core. Concretely:

- Retire `specimen-in-process` from the `ObservationStatus` code
  system / value set bound to `Observation.status`.
- Update `Observation.status` element documentation to clarify that
  the status values describe the state of the Observation resource
  itself, and that workflow states of the specimen-handling or
  test-running process belong on `Procedure`, `ServiceRequest`, or
  `Task`.
- Flag the removal prominently in the migration / breaking-changes
  notes for the next version, and advise implementers who relied on
  `specimen-in-process` to map forward to `registered` for the
  Observation and to represent the in-process specimen handling on a
  `Procedure` or `Task` that references the Observation (or is
  referenced by a `ServiceRequest`).

#### Justification

This aligns `Observation.status` with the design principle that a
resource's `status` element reflects the lifecycle of that resource,
not the lifecycle of an upstream workflow. The other values in
`ObservationStatus` all fit that pattern; `specimen-in-process` is
the single outlier and conflates two orthogonal concerns (the
Observation's data state vs. the specimen-handling procedure's
workflow state). The existing `registered` value genuinely does
cover the "we know a result is coming but it isn't here yet"
situation, and FHIR already provides `Procedure`, `ServiceRequest`,
and `Task` for tracking workflow, which keeps the data resource's
status small and unambiguous.

---

### Disposition B: Alternative Approach

#### Proposal

Retain `specimen-in-process` in `ObservationStatus` but formally
deprecate it rather than remove it, and sharpen the definition of
`registered` and of `Observation.status` itself. Specifically:

- Mark `specimen-in-process` as deprecated in the `ObservationStatus`
  code system, with a note that the code is retained only for
  backwards compatibility and must not be used in new instances.
- Add an explicit recommendation on `Observation.status` to use
  `registered` for Observations whose results are not yet available,
  regardless of where in the pre-analytic pipeline the underlying
  specimen or acquisition currently sits.
- Add a concrete cross-reference from `Observation.status` into the
  workflow guidance, pointing at `ServiceRequest`, `Procedure`, and
  `Task` as the places to record specimen collection, preparation,
  analytic progress, QA, and interpretation.
- Optionally, publish a short IG-level guidance note (or update the
  Observation narrative) describing the migration pattern:
  `specimen-in-process` → `Observation.status = registered` +
  workflow state on `Procedure`/`Task`.

#### Justification

Deprecation rather than removal acknowledges that an installed base
may be using `specimen-in-process` today, particularly in laboratory
interfaces where it has some history. Keeping the code valid but
discouraged avoids turning existing instances into invalid ones and
lets implementers migrate on their own cadence, while still steering
new implementations toward the cleaner separation the reporter is
asking for. This is softer than Disposition A and accepts that a
required-binding change in FHIR Core is a breaking change for any
implementer who has been relying on the current codeset.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep `specimen-in-process` as a valid value
in `ObservationStatus`. Record the disposition as "persuasive with
mod" is not used; the change is declined on the grounds that
`specimen-in-process` is in active laboratory use, that removing it
is a breaking change to a required binding on a heavily-used core
element, and that the "use `Procedure`/`Task` instead" alternative
imposes a higher implementation burden than many laboratory systems
currently carry.

#### Justification

`Observation.status` has a *required* binding, so removing a code is
a breaking change: every message / resource that currently carries
`specimen-in-process` becomes non-conformant and every validator has
to be updated. Laboratory implementations in particular have been
using `specimen-in-process` for some time; demanding that those
systems stand up `Procedure` or `Task` resources to express the same
thing raises the bar meaningfully for simple result-reporting
interfaces. If the workgroup believes the conceptual model issue is
real but not urgent, the defensible path is to leave the code in
place and address it with guidance rather than a breaking codeset
change.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best starting point for the workgroup. The
reporter's conceptual objection is sound: `Observation.status`
really is about the state of the Observation, and
`specimen-in-process` is the only value that talks about the
upstream workflow rather than the data resource. But
`Observation.status` is a required binding on one of the most
heavily-used elements in the specification, and outright removal
(Disposition A) is a real breaking change with validator and
conformance impact. Deprecating the code, sharpening the definition
of `registered`, and pointing implementers at `Procedure`, `Task`,
and `ServiceRequest` for the workflow story captures the intent of
FHIR-54892 without invalidating existing instances. If the
workgroup is willing to absorb the breaking-change cost, Disposition
A can be revisited at the next major version; until then, B is the
lower-risk way to make the same point.
