## Summary

The ticket reports that the human-readable textual summaries on several
Vital Signs profile pages still reflect a previous design in which the
LOINC code identifying the observation was carried on
`Observation.category.coding`. In the current design, the category
coding is the fixed Vital Signs category (`vital-signs` from the
observation-category code system), and it is `Observation.code.coding`
that carries the LOINC concept identifying the specific vital sign
(e.g., `2708-6` for Oxygen Saturation). Two specific occurrences are
called out:

- The Oxygen Saturation profile intro states a fixed
  `Observation.category.coding.code = '2708-6'`, which should be
  `Observation.code.coding.code = '2708-6'`.
- The Blood Pressure profile intro states a fixed
  `Observation.category.coding.system = 'http://loinc.org'`, which
  should be `Observation.code.coding.system = 'http://loinc.org'`.

The reporter further notes that the surrounding text structure should
be reworked, and that the remaining vital signs profile intros should
be audited for similar leftovers from the previous approach. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned for this ticket; the only cross-references are FHIR element
mentions of `Observation.category.coding.code` and
`Observation.category.coding.system`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persuasive with Modification — correct the textual summaries on the
affected Vital Signs profile intros so that the fixed code/system
statements refer to `Observation.code.coding` rather than
`Observation.category.coding`. Concretely:

1. In the Oxygen Saturation profile intro, change the bullet that
   reads "a fixed `Observation.category.coding.code` = '2708-6'" to
   "a fixed `Observation.code.coding.code` = '2708-6'" (and align the
   surrounding system reference to `http://loinc.org` on
   `Observation.code.coding.system`).
2. In the Blood Pressure profile intro, change the bullet that reads
   "a fixed `Observation.category.coding.system` = 'http://loinc.org'"
   to "a fixed `Observation.code.coding.system` = 'http://loinc.org'",
   and similarly correct any sibling code reference.
3. Rework the surrounding prose so the textual summary clearly
   distinguishes the two sets of fixed values: (a) the Vital Signs
   category, fixed on `Observation.category.coding` to
   `system = http://terminology.hl7.org/CodeSystem/observation-category`
   and `code = vital-signs`; and (b) the LOINC concept that identifies
   the specific vital sign, fixed on `Observation.code.coding` to
   `system = http://loinc.org` and the per-profile LOINC code (e.g.,
   `2708-6` for OxSat, `85354-9` for BP panel).
4. Audit the remaining Vital Signs profile intros (Body Height, Body
   Weight, Head Circumference, Body Temperature, Heart Rate,
   Respiratory Rate, Body Mass Index, Vital Signs Panel, Blood
   Pressure components, etc.) for the same category/code mix-up and
   apply the equivalent corrections.

No StructureDefinition or invariant changes are required — the
underlying profiles already constrain the right elements; only the
narrative needs to match.

#### Justification

The narrative as written contradicts the StructureDefinitions, which
is exactly the kind of inconsistency between human-readable summary
and machine-readable profile that confuses implementers and erodes
trust in the spec. The correction aligns the prose with the existing
constraints (Vital Signs category fixed on
`Observation.category.coding`; LOINC code identifying the specific
observation fixed on `Observation.code.coding`) and is purely
documentation — there is no impact on conformance, on existing
instances, or on validators. Reworking the surrounding text and
auditing the remaining profile intros in the same change keeps the
Vital Signs profile narratives internally consistent and avoids
shipping a half-fix that leaves equivalent errors on sibling profiles.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification, but factor the boilerplate. Instead of
hand-editing each profile intro, introduce a shared narrative snippet
(or a small generated table) that documents, in one place, the
"category / code split" rule for Vital Signs:

- `Observation.category.coding` is fixed to the Vital Signs category
  (`http://terminology.hl7.org/CodeSystem/observation-category` /
  `vital-signs`).
- `Observation.code.coding` is fixed to LOINC
  (`http://loinc.org`) with a per-profile code identifying the vital
  sign concept.

Each per-profile intro then references that shared snippet and only
states the per-profile LOINC code(s) and any panel/component
specifics. Apply the corrections from Disposition A as part of the
refactor so the OxSat and BP intros end up correct, and remove the
now-duplicated text from the other vital signs intros.

#### Justification

The reporter's last point — "the other vital sign profiles should be
checked for such remainders of the previous approach" — strongly
suggests the same boilerplate has been copied across many profile
intros, which is exactly how this class of error keeps recurring.
Centralizing the category/code split in one shared narrative makes
the rule authoritative in a single place, prevents future copies from
drifting back to the old wording, and reduces the maintenance burden
the next time the category coding or LOINC system URL needs to be
described differently. The trade-off versus Disposition A is a larger
editorial change with a slightly higher review burden, but no
conformance impact.

---

### Disposition C: Decline

#### Proposal

Not Persuasive — close the ticket without changing the narratives,
on the basis that the StructureDefinitions are normative and already
correct, and that the textual summaries are non-normative
explanatory prose that implementers should not be relying on for
conformance.

#### Justification

Strictly speaking, conformance is governed by the
StructureDefinitions, not by the prose in the profile intros. If a
reviewer takes the position that the narrative is informative only
and that implementers tooling against the actual profiles will
behave correctly regardless of the prose error, declining is
defensible on cost-of-change grounds alone. This option also avoids
opening up a broader rewrite of the Vital Signs intro pages.

---

### Recommendation

**Recommended disposition:** A

The defect is unambiguous: the narrative claims a fixed value on
`Observation.category.coding` that the StructureDefinitions actually
fix on `Observation.code.coding`, and the priority is set to Highest
on a high-visibility profile family (Vital Signs). Disposition C is
hard to justify — leaving prose that contradicts the normative
profiles in place will keep generating questions and Jira tickets.
Disposition B is attractive long-term, but the editorial scope is
larger than the ticket asks for and risks delaying the fix; the
reporter's audit request can be honored within Disposition A by
sweeping all Vital Signs profile intros for the same wording in the
same change. Recommend accepting with the modification scope
described in Disposition A — correct OxSat and BP, rework the
surrounding text so the category-vs-code split is unambiguous, and
audit the remaining Vital Signs profile intros for equivalent
leftovers — and revisit Disposition B's shared-snippet refactor as a
follow-up tracker if the audit confirms the boilerplate is widely
duplicated.
