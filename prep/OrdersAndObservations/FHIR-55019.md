## Summary

FHIR-55019 asks Orders & Observations to add the `List` resource as an
additional allowed target type on `DiagnosticReport.result`, so that a
`List` instance can serve as the "grouper" for laboratory observation
panels (CBC, Chem-7, thyroid, etc.). The reporter argues that a typical
laboratory panel does not have a panel-level result value, which makes
using a value-less `Observation` as the panel grouper an awkward fit and
a recurring source of confusion in the community.

The ticket is the latest in a clustered O&O effort (label
`organizer-grouping`) to settle how lab panels should be represented.
[FHIR-54562](https://jira.hl7.org/browse/FHIR-54562) (Hans Buitendijk)
explicitly enumerates the same option — change `DiagnosticReport.result`
to allow `Group` or `List` — as an alternative to adding an
`Observation.organizer` flag. [FHIR-54897](https://jira.hl7.org/browse/FHIR-54897)
(Kathy Walsh) prefers a new dedicated "Panel" grouper resource, with a
data-absent-reason on `Observation` only as a fallback.
[FHIR-50472](https://jira.hl7.org/browse/FHIR-50472) (deferred,
"Considered for Future Use") objected that turning `Observation` into
the panel grouper would lose `DiagnosticReport`-level metadata, and the
2025-10-14 OO/Grouping call agreed that the discussion will continue
past the R6 Normative ballot. [FHIR-47052](https://jira.hl7.org/browse/FHIR-47052)
(assigned to this ticket's reporter, Robert Hausam) frames the broader
structuring guidance for `DiagnosticReport`/`Observation` panels.
[FHIR-19937](https://jira.hl7.org/browse/FHIR-19937) is the
already-published Genomics precedent that recognised the panel-grouper
ambiguity and asked O&O to align "grouper" semantics on the ordering
side.

## Details

**Description:**

> Adding the List resource as an additional choice for the
> DiagnosticReport.result will allow if to be used as the "grouper" for
> observation panels (e.g., "CBC", "Chem-7", "thyroid" - and many
> others). The typical laboratory panel does not have a result at the
> panel level, and therefore the Observation resource (with 'value' not
> populated) is not an ideal choice to represent this. This has resulted
> in significant discussion, confusion and disagreement regarding the
> proper or best way to represent laboratory panels, and allowing the
> List resource to be used for this purpose could be helpful for
> resolving these issues.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add `List` to the allowed target types of `DiagnosticReport.result` in
FHIR Core (currently `Reference(Observation)`), making it
`Reference(Observation | List)`. The accompanying spec text on the
`DiagnosticReport.result` element and the DiagnosticReport narrative
("Notes" / "Boundaries and Relationships") would be updated to:

- Describe the `List` target as the "panel grouper" usage pattern
  (e.g., CBC, Chem-7, thyroid).
- Require, when a `List` is used, that `List.code` carry the panel
  code and that `List.entry.item` references the panel's member
  `Observation`s.
- Clarify that the `Observation`-only pattern remains valid for
  back-compat, and that `Observation.hasMember` is for genuine
  observation relationships, not panel grouping (aligning with the
  guidance being developed in
  [FHIR-47052](https://jira.hl7.org/browse/FHIR-47052)).

No new resource is introduced; this is a single choice-type widening
on one element plus narrative.

#### Justification

This is the minimum change that resolves the reporter's concern
without inventing new machinery. `List` already exists, already has
`code` + `entry`, and is the pattern other domains use to express
"these things go together for this reason". Crucially, this is exactly
the option enumerated by [FHIR-54562](https://jira.hl7.org/browse/FHIR-54562)
("change `DiagnosticReport.result` to include `Group` or `List` as a
choice"), so adopting it knocks out two `organizer-grouping` tickets
at once. It also avoids the objection raised in
[FHIR-50472](https://jira.hl7.org/browse/FHIR-50472): the panel grouper
is no longer a value-less `Observation`, so `DiagnosticReport`
metadata is preserved on the report itself rather than being squeezed
into a synthetic `Observation`.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of widening `DiagnosticReport.result`, treat the panel
grouper as a structural problem and prefer **flat
`DiagnosticReport`s in a Bundle** as the normative pattern, while
formalising a single, explicit grouper mechanism for the case where
nesting truly is needed:

1. In FHIR Core narrative for `DiagnosticReport`/`Observation`, adopt
   the FHIR-47052 guidance that panels are one level deep, with
   multi-level panels reported as multiple `DiagnosticReport`s in a
   Bundle.
2. For the residual case where the sender must convey "these
   `Observation`s belong to panel X" inside a single
   `DiagnosticReport`, define a standard `dataAbsentReason` code
   (e.g., `grouper` / `not-applicable`) for use on a value-less
   `Observation`, and document that `Observation.hasMember` is the
   correct linkage. Optionally also widen
   `DiagnosticReport.result` to allow `Group` (a clinical-grade
   grouper that already carries membership semantics) rather than
   `List` (which is generic and unordered-by-default).
3. Defer (or decline) the introduction of a brand-new `Panel`
   resource (the FHIR-54897 ask) to avoid R6 churn.

#### Justification

This addresses the same underlying problem — confusion about how to
express "panel" — but does so by clarifying the existing model first
and only adding a single, semantically tighter mechanism. Trade-offs
versus Disposition A:

- **Pro:** `List` is generic and was not designed for clinical
  groupers; `Group` (or a typed `dataAbsentReason`) carries more
  meaning and avoids inviting non-lab uses of `DiagnosticReport.result
  -> List`.
- **Pro:** Aligns with [FHIR-47052](https://jira.hl7.org/browse/FHIR-47052)
  and the bundle-of-`DiagnosticReport` pattern that
  [FHIR-54897](https://jira.hl7.org/browse/FHIR-54897) and existing
  implementers already deploy.
- **Con:** Requires more narrative + terminology work and cuts
  against the literal request of FHIR-55019.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-55019 as **already addressed by existing mechanisms and
as out of scope while the broader O&O grouping discussion is open**.
Specifically: the per-`Observation`-with-`hasMember` pattern (and, if
needed, a `dataAbsentReason` on the panel-level `Observation`) plus
multi-`DiagnosticReport` Bundles already cover the use case, and per
[FHIR-50472](https://jira.hl7.org/browse/FHIR-50472)'s 2025-10-14
OO/Grouping call note the cross-cutting grouping discussion is
explicitly continuing past the R6 Normative ballot. A targeted
choice-type change should be folded into that resolution rather than
landed piecemeal.

#### Justification

`DiagnosticReport.result` has been
`Reference(Observation)` since R2; widening it touches every
implementer and conformance resource (US Core lab profiles,
international lab IGs, every search-include of
`DiagnosticReport:result`). Doing it as a one-off, ahead of the
broader `organizer-grouping` resolution that
[FHIR-50472](https://jira.hl7.org/browse/FHIR-50472),
[FHIR-54562](https://jira.hl7.org/browse/FHIR-54562) and
[FHIR-54897](https://jira.hl7.org/browse/FHIR-54897) are still
arguing about, risks shipping two competing panel-grouper patterns in
the same release. Declining now (with a pointer to the umbrella
discussion) keeps the option open without baking it in.

---

### Recommendation

**Recommended disposition:** A

The `organizer-grouping` cluster has now produced four overlapping
tickets ([FHIR-50472](https://jira.hl7.org/browse/FHIR-50472),
[FHIR-54562](https://jira.hl7.org/browse/FHIR-54562),
[FHIR-54897](https://jira.hl7.org/browse/FHIR-54897), and this one)
plus the broader [FHIR-47052](https://jira.hl7.org/browse/FHIR-47052)
guidance work, all circling the same complaint: a value-less
`Observation` is the wrong shape for a lab panel grouper, and
implementers keep tripping over it. Disposition A is the only option
that actually removes that wrong shape from the toolbox by giving
implementers a first-class alternative target on
`DiagnosticReport.result`, and it is exactly one of the resolutions
[FHIR-54562](https://jira.hl7.org/browse/FHIR-54562) already
enumerates — so it can be presented to the workgroup as a joint
disposition for FHIR-55019 + FHIR-54562, with a note that
[FHIR-50472](https://jira.hl7.org/browse/FHIR-50472)'s objection is
also satisfied. Disposition B is more architecturally satisfying but
relies on the wider grouping conversation reaching consensus on a
"`Group` vs new `Panel` resource" question that
[FHIR-54897](https://jira.hl7.org/browse/FHIR-54897) shows is still
open; Disposition C defers a concrete fix that the community has been
asking for since [FHIR-19937](https://jira.hl7.org/browse/FHIR-19937)
in 2019. The workgroup should adopt A, with the implementation
narrative explicitly cross-referencing FHIR-47052 so the panel-grouper
guidance lands consistently.
