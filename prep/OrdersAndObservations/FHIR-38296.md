## Summary

This ticket asks the Orders & Observations work group to consider how
results produced by an at-home in-vitro test (e.g., a self-administered
COVID test) can be routed to the patient's own physician in addition
to — or alongside — the existing notification to a Public Health
Authority (PHA). Two specific design avenues are floated: (1) using
consumer health platforms such as Apple Health or Google Health/Fit as
a transport / aggregation layer that can then forward results to the
patient's care team, and (2) extending the message that currently goes
to the PHA so that it also identifies (and is delivered or copied to)
the patient's physician.

The request was originally triaged in 2022-12-15 as "Considered for
Future Use — out of scope for this release", then reverted on
2023-03-26 by Lynn Laakso to be re-opened post STU1 publication of the
At-Home In-Vitro Test Report (FHIR) IG. As of 2025-12-05 it is
labelled `Ready-to-Discuss` and `LAB_Call`, with a question from
hbuitendijk to the assignee about whether to discuss it at the January
WGM or on a standing OO Monday call. There are no cross-referenced
Jira tickets, Zulip threads, or GitHub items on file in FhirAugury for
this ticket.

## Proposed Dispositions

> The "At-Home In-Vitro Test Report (FHIR)" IG is published by Orders
> & Observations and is not surfaced as a GitHub repo in the cached
> FhirAugury repo set, so no per-repo briefing applies. The
> dispositions below are scoped to the IG's own profiles, value sets,
> and message definition; concrete file paths cannot be cited from a
> repo-analysis briefing.

### Disposition A: Accept As Requested

#### Proposal

Extend the At-Home In-Vitro Test Report IG so the existing message to
the PHA can also carry the patient's physician (or care team) as a
secondary, optional recipient/copy-to. Concretely:

- Update the IG's `MessageHeader` profile (the one used for the
  at-home test report message) to allow `MessageHeader.destination` to
  repeat, and constrain at most one occurrence to a `receiver`
  representing the PHA endpoint and zero-or-more occurrences to
  represent the patient's physician / PractitionerRole / CareTeam
  endpoint(s). Document the `MessageHeader.destination.receiver`
  binding to `Practitioner | PractitionerRole | CareTeam |
  Organization`.
- Profile the report `Bundle` so the focal `DiagnosticReport` /
  `Observation` is required to carry
  `DiagnosticReport.performer`/`Observation.performer` for the
  reporting service and a separate, optional reference to the
  patient's physician via either `DiagnosticReport.resultsInterpreter`
  or a new `recipient` extension on the report bundle (preferred:
  reuse the existing US Core `Practitioner` / `PractitionerRole`
  profiles where applicable).
- Add narrative guidance describing two delivery modalities:
  (a) duplicate-send of the same FHIR message to the physician
  endpoint where one is known; (b) PHA-mediated forwarding when the
  patient supplies physician details only to the testing app.
- Note (non-normative) that consumer health platforms (Apple
  Health / Google Health) can be used as an *intermediary* that holds
  the report on the patient's device and posts it onward via the same
  FHIR message; do not standardise an Apple/Google-specific transport
  in the IG.

#### Justification

Adding the physician to the existing message is the lowest-friction
path to the requested outcome: it reuses the IG's existing
`MessageHeader` + `Bundle` design rather than inventing a parallel
transport. Modeling the physician via standard FHIR resources
(`Practitioner` / `PractitionerRole` / `CareTeam`) keeps the IG
consistent with US Core and other OO laboratory IGs that already use
the same recipient pattern. Treating Apple / Google Health as an
intermediary rather than a normative endpoint avoids vendor lock-in
and avoids the IG having to track proprietary APIs.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than enlarging the at-home test report message itself, define a
companion **patient-mediated delivery** profile set:

- Add an IG section "Patient-Mediated Delivery to a Physician"
  describing a two-step flow: (1) the at-home testing app produces the
  existing `DiagnosticReport` + supporting resources for the patient;
  (2) the patient (or their app, including consumer health platforms
  acting on their behalf) shares that bundle with their physician's
  EHR using SMART Health Links / `$docref` / a simple FHIR `POST` to a
  patient-nominated endpoint.
- Require `Patient.generalPractitioner` to be populated on the
  at-home report Patient profile when known, so the testing app has
  somewhere to record the physician identity even if delivery is
  asynchronous.
- Optionally, add a single new extension —
  `at-home-report-recipient` (or reuse
  `http://hl7.org/fhir/StructureDefinition/workflow-supportingInfo`
  pattern) — on the report `Bundle` or `MessageHeader` to *advertise*
  intended secondary recipients, without mandating that the producer
  perform the second delivery.
- Leave the PHA message structurally unchanged so existing
  implementers are not impacted.

#### Justification

The literal request ("add physician to message going to PHA") couples
two trust domains — public health reporting and clinical care — into
one transmission, which is operationally awkward: PHAs typically do
not accept multi-destination messages, do not forward to clinicians,
and have legal constraints on re-disclosure. A patient-mediated path
keeps the PHA submission lean and compliant while still letting the
physician receive the same authoritative FHIR payload, and it aligns
the IG with the broader 21st Century Cures / patient-app direction
already taken by US Core and SMART. It also avoids the IG having to
profile multi-destination `MessageHeader` semantics that other OO
messaging IGs do not currently support.

---

### Disposition C: Decline

#### Proposal

Mark the ticket "Persuasive with Mod" → resolved as "Not Persuasive"
for the current STU cycle: the IG should not standardise a
physician-delivery mechanism. Rationale: physician notification of
at-home test results is already addressable today via existing
mechanisms outside the scope of the At-Home In-Vitro Test Report IG
— the patient or app can use US Core `DiagnosticReport-lab` plus any
standard FHIR transport (REST, messaging, SMART Health Links, or
EHR-mediated patient-portal upload) to share the same data with the
physician. Adding it to this IG would expand scope into clinical
results delivery, where Lab Order/Result IGs and US Core already own
the conformance surface. Explicitly mention Apple/Google Health is a
patient-channel concern, not an IG-level concern.

#### Justification

The IG's stated scope is reporting at-home test results to public
health, not clinical results delivery. Re-using existing US Core /
laboratory IGs avoids duplicate, slightly-incompatible profiles for
the same `DiagnosticReport`/`Observation` shape, which is a known
maintenance risk across OO IGs. Declining also respects the original
2022-12-15 "out of scope for this release" vote; the 2023 re-open was
procedural (post-STU1) rather than a substantive change in scope.

---

### Recommendation

**Recommended disposition:** B

The patient-mediated alternative best matches what implementers in
this space (consumer testing apps, payer/PHR integrators) are
actually building and what regulators are pushing toward, and it
avoids overloading the PHA channel with clinical delivery semantics
that PHAs typically cannot honour. It also preserves backwards
compatibility for STU1 implementers — the existing PHA message is
unchanged — while still giving the IG a clear, normative answer to
"how does the physician get a copy?" via `Patient.generalPractitioner`
plus a SMART/REST patient-mediated delivery section. Disposition A
should be the fallback if the work group concludes that PHAs in scope
for this IG *do* commonly forward to clinicians; Disposition C is
defensible but is likely to be re-filed by implementers and would
leave the question that the assignee re-opened in 2023 unresolved.
