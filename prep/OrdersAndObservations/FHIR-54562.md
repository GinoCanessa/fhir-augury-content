## Summary

This ticket pushes back on a separate proposal (tracked under the
`organizer-grouping` label) to add an `Observation.organizer` element
whose purpose would be to flag an `Observation` instance as a pure
"grouper" (e.g., a lab panel header) that never carries an actual
value. The reporter (Hans Buitendijk, on behalf of Orders &
Observations) argues that introducing such an element is unnecessary
and semantically wrong: an `Observation` should be an observation and
should carry a value (or a defined `dataAbsentReason` when it cannot),
not act as a valueless container. The ticket asks the workgroup to
instead solve the panel-grouping problem either by (a) relying on an
existing `dataAbsentReason` code, or (b) introducing a proper grouper
construct — specifically, allowing `Group` or `List` as a choice in
`DiagnosticReport.result`, or adding a separate grouping attribute on
`DiagnosticReport`. The reporter explicitly rejects nesting
`DiagnosticReport` via `.parent` / `.hasMember` (which would dilute the
meaning of `DiagnosticReport`) and rejects leaning on `Composition`
(which is a document-packaging construct, not a grouper). Cross-
reference analysis surfaced only one link: a textual mention of
`DiagnosticReport.result`. No related Jira tickets, Zulip threads, or
GitHub items were discovered for this ticket key.

## Details

**Description:**

The proposal is to add `.organizer` to Observation clarifying that the
Observation instance is serving a grouper function and would never
have an actual value, e.g., for lab panels. This creates a new element
that is not needed as either this can be solved by providing a defined
`dataAbsentReason` to do the same, or introducing a grouper resource
(`Group` or `List`) and change `DiagnosticReport.result` to include
`Group` or `List` as a choice, or add a separate attribute in
`DiagnosticReport`.

We suggest that using `Group` or `List` as neither `DiagnosticReport`
nor `Observation` are well suited and have the meaning of a panel or
like groupers.

- A `DiagnosticReport` provides the scope of the report in response to
  a lab order that may encompass one or more flat and/or nested
  panels. (We prefer to primarily use flat panels.) Thus nesting
  `DiagnosticReport` using a `.parent` or `.hasMember` construct would
  dilute the meaning of `DiagnosticReport`.
- An `Observation` is supposed to be an observation thus should have a
  value, but at times may not for various reasons. But it should not
  be a grouper that never has values as that is not an observation.
  `.hasMember` is still essential for related observations like
  susceptibility tests on a discovered organism, both observations.
- The `Composition` is only necessary if there is a need to package
  and organize the `DiagnosticReport` as a document, so is not a
  proper grouper either.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reject the separate `Observation.organizer` proposal and instead
introduce a proper grouper path on `DiagnosticReport`:

- Change `DiagnosticReport.result` from `Reference(Observation)` to a
  choice of `Reference(Observation | Group | List)` so that a panel
  (or any structural grouping of observations) is expressed via a
  first-class grouping resource rather than a valueless
  `Observation`.
- Clarify guidance on `Observation.hasMember` so it is explicitly
  reserved for semantically-related observations that each carry
  their own meaningful value (e.g., susceptibility tests against a
  discovered organism) and is not the recommended mechanism for
  panel grouping.
- Add narrative in the `DiagnosticReport` and `Observation` resource
  pages stating that valueless "grouper" `Observation`s are
  discouraged; use `Group`/`List` for panel-style grouping, and use
  `dataAbsentReason` only when a real observation genuinely lacks a
  value.

#### Justification

This is consistent with the semantic intent of each resource:
`Observation` remains a unit with a value (or a justified
`dataAbsentReason`), `DiagnosticReport` remains the scope of a
reported order, and `Group`/`List` carry the grouping responsibility
they were designed for. It also avoids minting a new `Observation`
element whose sole purpose is to mark the instance as not-really-an-
observation, which is a meta-flag on a resource whose class semantics
the flag contradicts. Reusing existing grouper resources is a lower-
complexity, more consistent extension point than a new core element.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change `DiagnosticReport.result`'s type. Instead:

- Define (or reuse) a `dataAbsentReason` concept (e.g., `grouper` /
  `panel-header`) that explicitly labels a valueless `Observation` as
  a structural grouper, and document its use on the `Observation`
  and `DiagnosticReport` pages.
- Publish an FHIR pattern page / implementation note describing
  panel grouping via `Observation.hasMember` + this
  `dataAbsentReason` code, with worked examples for flat and nested
  lab panels.
- If a grouping resource is still needed outside the `Observation`
  graph, recommend `List` via an extension on `DiagnosticReport` (or
  via `DiagnosticReport.presentedForm` / `result` narrative) rather
  than changing the `result` type.

#### Justification

This honours the underlying need (clearly distinguishing groupers
from real observations) while avoiding a breaking / cardinality-type
change to `DiagnosticReport.result`, which is heavily implemented.
The reporter acknowledges that `Observation`s sometimes legitimately
lack a value; adding a canonical `dataAbsentReason` for the
"grouper" case standardises current real-world practice without
introducing either a new `Observation.organizer` element or a new
type in `DiagnosticReport.result`. Trade-off vs. Disposition A:
lower implementer disruption, but weaker semantic separation — the
resource class still carries the grouper, just with a coded
rationale.

---

### Disposition C: Decline

#### Proposal

Decline the ticket as duplicative / out of scope: the underlying
design question (whether to add `Observation.organizer`) is owned by
the separate `organizer-grouping` ticket(s), and this ticket is an
argument against that proposal rather than an independent actionable
change request. Redirect the discussion into the primary
`organizer-grouping` thread and close FHIR-54562 as "not persuasive /
discussion captured elsewhere" once the primary ticket is disposed.

#### Justification

The ticket's concrete ask is "do not accept the other proposal, and
consider these alternatives." That is better expressed as a comment
or counter-proposal on the primary organizer-grouping ticket than as
a standalone change request, because approving or declining
FHIR-54562 in isolation does not commit the workgroup to any
specific spec change. Declining here keeps the decision in one
place and avoids parallel, potentially inconsistent resolutions.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the cleanest semantic outcome, but widening
`DiagnosticReport.result` to `Reference(Observation | Group | List)`
is a material breaking change to one of the most widely implemented
R4/R5 elements and would need strong community evidence before
shipping — evidence that the cross-reference and Zulip searches did
not surface here. Disposition C is procedurally tidy but leaves the
underlying panel-grouping question unanswered. Disposition B
captures the reporter's core concern (an `Observation` should stay
an observation, not be re-purposed as a grouper via a new
`.organizer` element) by formalising the `dataAbsentReason` pattern
plus implementation guidance, without forcing a type change on
`DiagnosticReport.result`. It also leaves a clean path to Disposition
A in a later cycle if implementer feedback confirms demand. The
workgroup should use this review to decide explicitly whether to
forward the `Group`/`List`-on-`DiagnosticReport.result` idea as a
follow-on tracker.
