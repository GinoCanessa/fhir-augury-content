## Summary

The ticket raises a definitional question against the Laboratory Report
(FHIR) IG about how Obligations interact with the Must Support flag on
profile elements. In R5, Obligations may be defined on an element when it
is marked Must Support; the reporter (Hans Buitendijk) asks whether the
Laboratory Report IG should require Must Support to be set first when an
Obligation is attached, or whether Must Support is implied by the
presence of an Obligation. Because the Laboratory Report IG is R4-based
and pre-adopts the R5 Obligation machinery, the reporter further asks how
this pre-adoption is meant to function in an R4 context, suggesting at
minimum a cross-reference to the R5 Obligation section and likely
additional explanatory material in the IG. No cross-referenced Jira
tickets, Zulip discussions, or GitHub items were surfaced for this
ticket, so the request stands on its own as an editorial / conformance
clarification against the Laboratory Report IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Laboratory Report (FHIR) IG conformance / profiling guidance
to:

1. State explicitly that any element carrying an Obligation in this IG
   **must** also be flagged Must Support on the same element/profile,
   matching the R5 rule that Obligations apply in the context of Must
   Support.
2. Add a new "Obligations and Must Support" subsection to the IG's
   conformance narrative that explains the R5-on-R4 pre-adoption: it
   should describe how the Obligation extension is being used against R4
   StructureDefinitions, link out to the R5 Obligations specification
   section as the normative reference, and clarify that R4 implementers
   are expected to honour both the Must Support semantics and the
   declared Obligations.
3. Audit every profile in the IG and ensure that, for any element where
   an Obligation has been attached, `mustSupport = true` is also set;
   correct any element where Must Support was implicit.

#### Justification

This matches the literal request and aligns the IG with R5's stated
relationship between Must Support and Obligations, removing the ambiguity
the reporter highlights. Making Must Support explicit (rather than
implied) is consistent with FHIR's general principle that conformance
flags should be visible in the StructureDefinition rather than inferred
by tooling, and it gives implementers a single, deterministic place to
discover element-level expectations. Adding the cross-reference to the R5
Obligations section is low-cost editorial work and is the minimum needed
to make a pre-adoption defensible in an R4-based IG.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than coupling Obligation to an explicit Must Support flag in this
IG, treat the presence of an Obligation as a stronger conformance
statement in its own right and document that contract directly:

1. Add IG narrative stating that, in this IG, any element carrying an
   Obligation is conformant-relevant and SHALL be supported by actors in
   the role(s) named by the Obligation, regardless of the Must Support
   flag value on that element in the underlying R4 profile.
2. Document the R5 pre-adoption as a deliberate "Obligation-first"
   profiling style: Obligations declare the actor-scoped expectation;
   Must Support is reserved for cases where an expectation applies
   uniformly across all actors.
3. Provide a comparison table in the IG showing, for the elements that
   currently carry Obligations, what the equivalent Must Support
   interpretation would be, so R4 implementers without Obligation-aware
   tooling can still derive the expected behaviour.

#### Justification

This addresses the reporter's underlying concern — that R4 implementers
need a clear story for how Obligations bite — without forcing every
Obligation-bearing element to also be Must Support. Doing so preserves
the more precise actor-scoped semantics of Obligations (which Must
Support cannot express) and avoids over-constraining elements that may
only be expected for a subset of actors. It also keeps the IG closer to
the direction R5 is taking, where Obligations are intended to carry the
conformance weight that Must Support historically conflated.

---

### Disposition C: Decline

#### Proposal

Close this ticket as "Not Persuasive — Already Addressed" on the basis
that the R5 specification already defines the relationship between Must
Support and Obligations, and the Laboratory Report IG inherits that
definition by pre-adopting the R5 Obligation extension. No additional IG
text or per-element changes are required; implementers should refer to
the R5 Obligations section for the normative semantics.

#### Justification

The relationship between Must Support and Obligations is a base-spec
concern, not an IG concern. Restating it in every IG that pre-adopts R5
Obligations risks divergence from the base specification as R5 evolves,
and creates maintenance burden for IG editors. Implementers working with
pre-adopted R5 features are already expected to consult the R5
specification for the semantics of those features.

---

### Recommendation

**Recommended disposition:** A

The reporter's concern is real and editorial in nature: the IG
pre-adopts an R5 mechanism without explaining how it should be read
against an R4 baseline, and the silence around Must Support is exactly
the kind of gap that produces inconsistent implementations. Disposition A
gives implementers a single, in-IG answer to "do I also need Must
Support?" and adds the cross-reference the reporter explicitly asked
for, while keeping the IG aligned with the R5 rule that Obligations
operate in the context of Must Support. Disposition B is more expressive
but introduces a profiling style that diverges from how Must Support and
Obligations are described in R5, which would surprise implementers and
tooling vendors and creates a second story to maintain. Disposition C
under-serves R4 implementers, who are the primary audience for an
R4-based IG and cannot reasonably be expected to reconstruct the
pre-adoption semantics from the R5 spec on their own. The workgroup
should confirm whether any Obligation-bearing elements in the current IG
are not yet flagged Must Support, since that audit is what determines
whether Disposition A is purely editorial or also requires profile
edits.
