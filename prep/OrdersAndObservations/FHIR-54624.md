## Summary

The ticket reports an inconsistency in the definition text of the `statusReason`
element on a Request-pattern resource (most plausibly `ServiceRequest`, given
the Orders & Observations work group and the description's reference to "a
service request"). The current definition text states that `statusReason` "can
be used to explain why a service request is **suspended**, cancelled, or on
hold," but `suspended` is not a code in the bound `RequestStatus` value set.
The actual `RequestStatus` codes are `draft`, `active`, `on-hold`,
`entered-in-error`, `ended`, `completed`, `revoked`, and `unknown`. Eric Haas
recommends rewording the definition so that the example statuses it cites
match real codes — e.g., "...can be used to explain why a service request is
on-hold, ended, or revoked..." This is a non-breaking, editorial fix.

No cross-referenced Jira tickets, GitHub items, or Zulip threads were surfaced
for FHIR-54624 specifically.

## Proposed Dispositions

The dispositions below treat the affected resource as `ServiceRequest` in the
HL7/fhir core specification. If on review the same defective wording appears
on `statusReason` of other Request-pattern resources (e.g., `MedicationRequest`,
`NutritionOrder`, `DeviceRequest`, `SupplyRequest`, `CommunicationRequest`,
`Task`), the editorial fix should be applied consistently across all of them
in the same change.

### Disposition A: Accept As Requested

#### Proposal

Edit the `definition` of `ServiceRequest.statusReason` (and any sibling
Request-pattern resource whose `statusReason` carries the same wording) so
that the example status codes it names are members of the bound
`RequestStatus` value set. Concretely, replace the phrase
"…can be used to explain why a service request is **suspended**, cancelled,
or on hold…" with the reporter's suggested wording, e.g.,
"…can be used to explain why a service request is on-hold, ended, or
revoked…". No StructureDefinition shape, cardinality, type, binding, or
invariant changes; this is purely a `definition` (and, if applicable,
`comment`) text edit.

#### Justification

The current wording is factually wrong against the bound value set
(`RequestStatus`), which already does not include `suspended` and uses
`on-hold`, `ended`, and `revoked` for the suspend/cancel-style transitions.
Citing non-existent codes in normative definition text is misleading to
implementers and trivially fixable. The suggested replacement words are
already canonical members of `RequestStatus`, so the fix introduces no new
concepts and no behavior change. Editorial corrections of element
definitions are exactly the class of change the workgroup routinely applies
during ballot reconciliation.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the editorial fix in Disposition A, but generalize the wording instead
of enumerating specific status codes. For example: "Captures the reason for
the current state of the {resource}. This is generally used for non-active
states (e.g., `on-hold`, `entered-in-error`, `ended`, `revoked`) but MAY be
populated in any state. The codes here are distinct from the resource's
own `status` and from `Encounter`-style outcome codes." Apply the same
generalized wording to every Request-pattern resource that carries
`statusReason`, so that the definition does not have to be re-edited every
time `RequestStatus` itself evolves.

#### Justification

Listing example status codes in element-definition prose is what created the
defect in the first place: the prose drifted out of sync with the value set.
A definition that anchors on the *role* of `statusReason` ("explains the
current state") rather than on a specific subset of codes is more robust and
easier to keep consistent across the Request pattern. The trade-off vs.
Disposition A is that reviewers lose the concreteness of seeing example codes
in the definition; that can be partially recovered by moving the example
list into the `comment` field, where it is clearly illustrative rather than
normative.

---

### Disposition C: Decline

#### Proposal

Take no change. Mark the ticket as "Not Persuasive" on the grounds that the
definition's reference to "suspended" is informal English describing a
real-world state, not a code reference, and that the binding to
`RequestStatus` already authoritatively constrains which codes are valid.

#### Justification

A reader who follows the binding to `RequestStatus` will see the actual
allowed codes; the definition prose is illustrative. Declining avoids touching
normative text on a widely-implemented resource for what is arguably a
stylistic issue, and avoids the risk of an editorial change being perceived
as a semantic one by downstream tooling that diff definitions across versions.

---

### Recommendation

**Recommended disposition:** A

The defect is real, narrow, and trivially fixable. The current definition
text names a status (`suspended`) that does not exist in the bound value set,
which is at best confusing and at worst actively misleading to implementers
who rely on element definitions as primary documentation. Disposition A is
the lowest-risk path: it changes only the prose, leaves cardinality, type,
and binding untouched, and aligns the text with codes the value set already
defines. Disposition B is attractive in principle but expands the scope of an
editorial ticket into a Request-pattern-wide rewrite, which is better handled
as its own dedicated cleanup if the workgroup wants it. Disposition C is
hard to defend given the reporter has correctly identified that the cited
status code does not exist. The workgroup should accept the fix and, while
the file is open, sweep the other Request-pattern resources for the same
"suspended" wording so the correction lands consistently in one change.
