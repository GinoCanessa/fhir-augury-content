## Summary

The ticket reports that the **Boundaries and Relationships (B&R)** section of
the `BodyStructure` resource page contains guidance that has been overtaken by
prior changes — specifically, the sentence:

> "If a resource's code value implies a body structure, then the bodyStructure
> resource should be used consistent with the code and to provide any
> additional detail. **The bodySite element in any resource** can be used when
> the resource's code is not implicit about the body structure and does not
> require extensive detail."

According to the reporter (Lorraine Constable), the project previously
**replaced `bodySite` with `bodyStructure` in a number of resources**, so the
phrase "the bodySite element in any resource" is no longer accurate and the
guidance needs to be re-stated to reflect the current model. The text needs an
edit so that it (a) does not falsely imply `bodySite` is universally
available, and (b) gives correct guidance on when to use `bodySite` (where it
still exists) vs. the `bodyStructure` reference / `CodeableReference` element
(where it now exists).

A closely related ticket — [FHIR-54843](https://jira.hl7.org/browse/FHIR-54843)
"Revisit sentence to provide guidance how to use bodyStructure, when only a
code is needed" (Ulrike Merrick, also O&O, filed one day after FHIR-53872) —
quotes the exact same sentence and supplies a comprehensive inventory of the
resources currently using `bodyStructure` (CodeableReference vs. plain
Reference) and `bodySite` (CodeableConcept). FHIR-53872 is the
"problem statement" ticket; FHIR-54843 is the "evidence and inventory"
ticket. They should almost certainly be dispositioned together.

## Proposed Dispositions

> Note: All three dispositions are about narrative-only edits to
> `BodyStructure`'s Boundaries and Relationships section. The ticket has no
> GitHub cross-references and no repo briefings were required.

### Disposition A: Accept As Requested

#### Proposal

Edit the **Boundaries and Relationships** section of the BodyStructure
resource page to replace the obsolete sentence with text that reflects the
post-replacement model. Concretely, change the second paragraph to read
something like:

> "If a resource's `code` value implies a body structure, then the
> `BodyStructure` resource should be used (referenced from a resource's
> `bodyStructure` element, typically a `CodeableReference`) consistent with
> the code and to provide any additional detail. Where the resource exposes a
> `bodySite` element (e.g., `Procedure.bodySite`, `Observation.bodySite`,
> `Condition.bodySite`, `Specimen.collection.bodySite`, etc.), it MAY be used
> when the resource's code is not implicit about the body structure and the
> use case does not require the additional detail expressible via
> `BodyStructure`."

Make the equivalent narrative edit in B&R and check for the same wording on
the `bodySite` data type page so the two stay consistent. No structural
changes; no cardinality, type, or binding changes.

#### Justification

This is what the ticket asks for. The current sentence is factually wrong as
written ("the bodySite element in any resource") because several resources no
longer have `bodySite` and instead have `bodyStructure` only (e.g.,
`DeviceAssociation`, `DocumentReference`, `ObservationDefinition`,
`ServiceRequest`). It is a narrative-only change with no implementer impact,
no breaking effect on instances, and no spec-mechanics risk. It directly
fixes a documentation defect on a resource page, which is exactly the kind of
disposition the workgroup typically applies "Persuasive" to with no further
debate.

---

### Disposition B: Alternative Approach

#### Proposal

Combine FHIR-53872 with [FHIR-54843](https://jira.hl7.org/browse/FHIR-54843)
and apply a single coordinated edit that:

1. Rewrites the offending paragraph in BodyStructure B&R as in Disposition A.
2. Adds a short, normatively-neutral **table** in BodyStructure B&R that
   enumerates how each FHIR R6 resource currently exposes body structure /
   body site, using the inventory from FHIR-54843:
   - `CodeableReference(BodyStructure)` only — `ServiceRequest.bodyStructure`,
     `ObservationDefinition.bodyStructure`,
     `DocumentReference.bodyStructure`, `Specimen.collection.bodySite`,
     `ImagingStudy.series.bodySite`, `ImagingSelection.bodySite`,
     `ExplanationOfBenefit.item.bodySite`, `Claim.item.bodySite`,
     `ClaimResponse.addItem.bodySite`.
   - Parallel `bodySite` (`CodeableConcept`) + `bodyStructure`
     (`Reference(BodyStructure)`) — `Procedure`, `Observation`, `Condition`.
   - `Reference(BodyStructure)` only — `DeviceAssociation.bodyStructure`.
3. Adds explicit guidance that, where both elements exist, `bodySite` is for
   the simple coded case and `bodyStructure` is for the case requiring more
   detail (laterality, qualifier, image, morphology, etc.); where only
   `CodeableReference` exists, the same single element covers both cases.
4. Resolves both FHIR-53872 and FHIR-54843 with a single PR and marks one as
   a duplicate of the other (recommend keeping FHIR-53872 as the editorial
   parent because it was filed first).

#### Justification

The two tickets are clearly about the same paragraph; FHIR-53872 reports the
defect and FHIR-54843 supplies the inventory needed to write the
replacement. Disposing of them together avoids a stale narrative being
re-introduced on the next edit and gives readers a definitive, current
mapping. The table is more durable than prose because the model is
heterogeneous (three different shapes across the affected resources). The
trade-off vs. Disposition A is slightly more editorial work and a longer
section, but the cost is small and the maintenance burden is low because the
table only changes when the model changes.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-53872 as **Not Persuasive** on the grounds that the existing
sentence, while imprecise, is not actually wrong if read in context — every
resource that does not have a `bodySite` element has a `bodyStructure`
element instead, and the surrounding paragraph already states that
`BodyStructure` is the preferred mechanism. Defer any narrative tightening
to a future editorial pass that touches BodyStructure for substantive
reasons.

#### Justification

The change is purely cosmetic: instances are unaffected, validators are
unaffected, and downstream IGs that reference BodyStructure are unaffected.
There is an opportunity cost to spending O&O / FMG cycles on R6 narrative
edits when the substantive ticket queue is large. If the workgroup is
treating R6 as feature-frozen and only landing material defects, this could
plausibly be deferred or rejected.

---

### Recommendation

**Recommended disposition:** B

The narrative is genuinely misleading today — readers being told that "the
bodySite element in any resource can be used" will look for a `bodySite` on
`DeviceAssociation`, `ServiceRequest`, `DocumentReference`,
`ObservationDefinition`, etc., and not find it, which undermines confidence
in the resource page. Disposition A fixes the immediate defect, but
FHIR-54843 (filed by a different reporter one day later, against the same
sentence) demonstrates that a one-line rewrite is not enough — implementers
want to see the actual mapping. Folding the two tickets together and
publishing the inventory as a small table is the highest-value, lowest-risk
change: it definitively closes both tickets, it is a narrative-only edit
(no implementer impact, no breaking change), and it pre-empts a third ticket
on the same paragraph. Disposition C is hard to defend given that the
sentence is observably inaccurate and the fix is bounded.
