## Summary

Cooper Thompson asks the Clinical Order Workflow (COW) IG to explicitly
describe how `Reference` elements should be exchanged when RESTful
create/update interactions are used — most acutely for `Task.create`,
which carries many references but has no documented orchestration for
how those references come to exist on the receiving (fulfiller) server.
The ticket enumerates four candidate patterns: (1) inline the referents
as **contained** resources; (2) **pre-create** the referents on the
fulfiller before `Task.create`; (3) keep references pointing back to the
**Placer's** server and require the fulfiller to resolve them; or
(4) use **logical references** (identifier-only) and let the fulfiller
look up by identifier. The IG currently picks none, so two implementers
who follow the spec literally cannot interoperate.

The request is closely entangled with two sibling COW tickets surfaced
in the comments and resolution. [FHIR-50414](https://jira.hl7.org/browse/FHIR-50414)
("Describe references across servers", reported by Martin von Siebenthal,
later closed as **Duplicate** of this ticket) asks the same question
from the Placer/Filler ownership angle, using the
`ex4-meds-grouped-dispense` example where `Task.focus` points at a
prescription owned by the Placer. [FHIR-50293](https://jira.hl7.org/browse/FHIR-50293)
("Clarify role of intermediaries", reported by Craig Newman) asks
whether intermediaries copy or re-host the `Task` and its references —
which directly affects which server a reference can resolve against.
The ticket is also xref-linked to `Task.create` and `ServiceRequest.specimen`,
the latter being André Pitkus's running example that some references
literally cannot exist at `Task.create` time.

The ticket has been **resolved Persuasive with Modification**: the COW
WG agreed to add guidance on a new "Coordinating across servers" page
(`coordinating-across-servers.html#resolving-references`) stating that
contained resources are out-of-scope for the Coordination Task,
fulfiller-side pre-creation is discouraged, **references to the
Placer's server are the default**, and **logical references are
appropriate for static directory-style resources** (Practitioner,
Location). The remaining open conversation (most recent comments from
Cooper Thompson) is whether that default should be conditional on a
bidirectional channel existing and whether contained resources deserve
re-evaluation as "logical references++".

## Details

**Description:**

> In cases where we are using RESTful create/update interactions, we
> should describe how references should be handled. Doing `Task.create`,
> for example, does not describe the REST orchestration that is needed,
> as Task includes many references. The options I can think of include:
>
> - References would be **contained** resources in the Task.
> - References would be **created on the fulfiller prior to** `Task.create`.
>   If we use this pattern, the interaction diagram should include the
>   steps for making sure those references exist. Some references might
>   be pre-coordinated (e.g., practitioners, locations). But some would
>   probably need to be on-the-fly (e.g., patient, encounter).
> - References would be to the **Placer's server**. In this case, there
>   should be a step where the Fulfiller needs to (probably) resolve
>   those references so they can use the resolved content to match to
>   records internally.
> - References would be **logical references** (i.e., just include
>   identifiers). And the fulfiller would use those identifiers for
>   lookup internally.

**Resolution Description:**

> We agree that we should describe how references should be handled.
> With regards to the four options described in the issue:
>
> - The current scope of this IG does not provide any consideration for
>   the use of contained resources in the Coordination Task Resource.
> - We discourage references to resources on the fulfiller, because
>   determining reliably what resources may or may not be referenceable
>   on the fulfiller's server will degenerate into guessing.
> - References to the Placer's Server is the default expectation.
> - Using logical references for more static resources (e.g.
>   Practitioners, Locations) is appropriate and also expected.
>
> We will change the IG with the description for resolving references
> on the proposed coordinating across servers page
> (`coordinating-across-servers.html#resolving-references`) or
> whatever outcome on [FHIR-50293](https://jira.hl7.org/browse/FHIR-50293).

## Proposed Dispositions

The dispositions below align with the IG-only scope of this ticket.
Because no GitHub repository is cross-referenced, there is no
authoring-root to cite — the actual change will land in the
`HL7/fhir-cow-ig` IG narrative on the new
`coordinating-across-servers` page (per the resolution text), but
that repo was not pre-loaded as a briefing for this prep run.

### Disposition A: Accept As Requested

#### Proposal

Adopt the four-bullet position already written into the resolution and
land it as a new sub-section
`coordinating-across-servers.html#resolving-references` on the
"Coordinating across servers" page of the COW IG (the URL the
resolution itself names). The sub-section MUST state, for the
Coordination `Task`:

1. **Contained resources are out of scope** for the Coordination Task.
2. **References to fulfiller-resident resources are discouraged** —
   the Placer cannot reliably know what is resolvable on the
   fulfiller's server.
3. **The default is references to the Placer's server**, with the
   Fulfiller responsible for resolving (and, per Cooper's "pull-over"
   pattern, optionally re-hosting locally) those references at
   `Task.create` time.
4. **Logical / identifier-only references are expected for static,
   directory-style referents** — Practitioner, Location,
   Organization, etc.

Add an interaction-diagram update to whichever §6.x / Background
section the WG settles on (Ralf's 2025-11-18 proposal: expand
"Requesting additional Info"), showing the pull-over sequence:
Placer POSTs `Task` → Fulfiller resolves placer-side references →
Fulfiller persists local copies → subsequent reads from the
Fulfiller return references that point at fulfiller-local IDs. Add
explicit cross-link from this section to the disposition of
[FHIR-50293](https://jira.hl7.org/browse/FHIR-50293) so the
intermediary-role guidance and the references-resolution guidance
stay consistent.

#### Justification

This is the position the WG already converged on across two COW calls
(2025-05-02 and the November/December 2025 thread) and is reflected
in the recorded resolution. It also matches the Craig-Newman position
("guidance on 2, 3, and 4 with a recommendation to later IG authors")
and aligns with the Persuasive-with-Modification disposition the WG
has already filed. Naming the Placer's server as the default is the
only option that is both (a) compatible with logical references being
the fallback for static referents and (b) doesn't require the Placer
to predict what the Fulfiller's server contains. It does not solve
André Pitkus's "the referent doesn't exist yet" objection — but, as
Cooper noted, those cases simply have **no** reference to resolve at
`Task.create` time and therefore fall outside this guidance section
(Specimen, etc., would be added later via update interactions, which
is its own concern).

---

### Disposition B: Alternative Approach

#### Proposal

Adopt Disposition A's text but **frame it as a decision tree keyed off
the exchange channel rather than as a single default**, addressing
Cooper Thompson's open 2025-12-16 / 2025-11-18 comments:

- **If a bidirectional channel (reverse results feed, Subscription
  Notification with `authorization_hint`, or messaging bundle) is in
  place** → references to the Placer's server are the default, exactly
  as in Disposition A.
- **If exchange is unidirectional** (no reverse channel; Filler has no
  client registration with the Placer) → recommend that the Placer
  send referent content alongside the `Task` — either via a transaction
  Bundle that pre-creates the referents on the Fulfiller, via Messaging
  that bundles the referents, or via a Subscription Notification with
  bundled payload. Logical references for static directory referents
  remain expected in both branches.
- **Contained resources** — keep them out of the recommended set for
  Coordination Task but explicitly note Cooper's "logical
  reference++" framing as a known alternative IGs may opt into,
  rather than ruling them out.
- **Cross-server URL hazards** — add a Gotcha box noting that Placers
  behind API gateways / reverse proxies must publish a stable
  externally-addressable base URL that the Fulfiller can reach, and
  that the Placer is responsible for choosing the correct base URL
  per Fulfiller. Reference the Cooper 2025-11-18 comment.

#### Justification

Disposition A captures the WG's agreed-on default but treats it as
unconditional, which leaves the unanswered comments on the ticket
unresolved and risks setting up implementers in unidirectional or
"intermediary-only" deployments to fail. Ralf's own 2025-11-18 comment
already enumerates three exchange patterns (REST, Messaging,
Subscription Notification with `authorization_hint`); Disposition B
operationalises that enumeration into the resolution text. It also
gives a concrete answer to André Pitkus's "Placer-references aren't
always possible" objection without weakening the default for the
common case. Trade-off vs. A: the page becomes longer and demands
that IG authors / implementers actually pick a branch, but that is
the right engineering forcing-function for a guidance gap that has
already produced two duplicate tickets.

---

### Disposition C: Decline

#### Proposal

Decline to add explicit reference-handling guidance to the COW IG,
on the grounds that `Task` reference semantics are an IG-author and
implementer concern that should be left to downstream IGs (US Core,
Da Vinci PDex/HRex, IHE profiles, etc.), and that the base FHIR
specification's general guidance on `Reference` is already sufficient.

#### Justification

Defensible only on a narrow scope-trimming argument: the COW IG is
positioned as cross-domain workflow plumbing, and prescribing a
single reference-resolution pattern risks colliding with downstream
IGs that have already chosen differently. However, the comment
thread, the duplicate-filing of FHIR-50414, and the explicit
WG-level vote on 2025-05-02 all make this hard to defend — two
implementers independently filed against the same gap, and the WG
has already agreed (resolution = Persuasive with Modification) that
*some* guidance is required for the IG to be implementable. Including
this disposition only for completeness; it should not be the
recommendation.

---

### Recommendation

**Recommended disposition:** B

Disposition B incorporates everything the WG has already agreed to
(and that Disposition A would land verbatim), but extends it to cover
the unidirectional / no-reverse-channel case that Cooper Thompson
raised twice in late-2025 comments and that André Pitkus repeatedly
flagged from the O&O angle. The cost is small — a page-and-a-half of
narrative and a Gotcha box about Placer base URLs — and the upside is
that the IG ships guidance an implementer can actually follow without
having to derive an unwritten "what if there's no reverse feed?"
sub-rule themselves. Disposition B also sets up the
"Coordinating across servers" page to be coherent with the
intermediary-role disposition on
[FHIR-50293](https://jira.hl7.org/browse/FHIR-50293), which will share
the same page and which would otherwise risk contradicting an
unconditional "default = Placer-side references" rule. Disposition A
is the safe fallback if the WG decides to defer the bidirectional /
unidirectional split to a follow-up ticket; Disposition C should not
be entertained given two independent change requests on the same gap.
