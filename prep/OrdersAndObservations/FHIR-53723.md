## Summary

The reporter (Daniel Geil) observes that the `Specimen.container` backbone
element carries far less detail than the `SpecimenDefinition.container`
backbone element. In particular, `SpecimenDefinition.container` describes
container properties such as material, type, and cap, while the equivalent
information on an instance specimen is either missing from
`Specimen.container` or has been pushed onto a referenced device via
`Specimen.container.device`. The ticket asks whether `Specimen.container`
should be expanded to expose comparable detail (material, type, cap) on the
instance side so that the data model is symmetric between the definitional
and instance views of a specimen container.

A subsequent comment from Daniel Rutz reframes the underlying issue as a
broader inconsistency: the definitional resource (`SpecimenDefinition`)
holds the container concept directly and richly, whereas the instance
resource (`Specimen`) splits the same concept across `Specimen.container`,
`Specimen.container.device`, and `Specimen.container.specimenQuantity`
(noting that `specimenQuantity` arguably belongs on the container/device
rather than on the specimen itself). The OO on-FHIR call agreed the topic
warrants further specimen-call discussion and likely some rework, even
though Specimen is targeted for R6 normative. Devices WG (Elliot Silver)
has asked to be involved before any resolution is finalized, since
`Specimen.container.device` straddles the OO/Devices boundary.

There are no cross-referenced Jira tickets, no related Zulip threads, and
no related GitHub PRs/issues â€” only Confluence (the 2026-02-02 specimen
call minutes) and intra-spec FHIR element references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add explicit `material`, `type`, and `cap` elements to
`Specimen.container` in FHIR Core (HL7/fhir, Specimen resource), modeled
after the corresponding elements on `SpecimenDefinition.container`:

- `Specimen.container.type` : CodeableConcept (binding aligned with
  `SpecimenDefinition.typeTested.container.type`'s value set).
- `Specimen.container.material` : CodeableConcept (binding aligned with
  `SpecimenDefinition.typeTested.container.material`).
- `Specimen.container.cap` : CodeableConcept (binding aligned with
  `SpecimenDefinition.typeTested.container.cap`).

Cardinalities `0..1` for each, optional, with mappings/comments noting
that when `Specimen.container.device` is also present and points to a
`Device` representing the physical container, the instance-level fields
on `Specimen.container` describe the actual container used and the
`Device` describes the persistent/registered device. Update the resource
narrative for Specimen.container to call out the alignment with
`SpecimenDefinition.typeTested.container` and to give guidance for when
`specimenQuantity` should be expressed on the container vs. the
specimen.

#### Justification

This delivers what the reporter asked for and restores symmetry between
the definitional and instance views of a specimen container â€” a
recurring FHIR design principle ("the definitional resource and the
instance resource should describe the same concept compatibly"). Today
implementers cannot represent the material/type/cap of the actual
container used unless they go through `Specimen.container.device`,
which forces them to either (a) create transient `Device` resources for
every tube/cup, or (b) lose the data. The OO-on-FHIR call already
indicated the inconsistency is real and that some rework is warranted
even given Specimen's R6 normative target. Devices WG involvement
(per esilver's comments) is required because `Specimen.container.device`
straddles OO/Devices, but adding optional sibling elements does not
disturb the existing Device reference and is therefore a tractable
joint-WG change.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding `material`, `type`, and `cap` directly to
`Specimen.container`, redefine `Specimen.container` to contain a single
"container snapshot" that mirrors the structure of
`SpecimenDefinition.typeTested.container`, in one of two ways:

1. **Reference the definitional concept**: add
   `Specimen.container.specimenDefinition : Reference(SpecimenDefinition)`
   (or a `canonical` thereof) and add a small set of override elements
   (`material`, `type`, `cap`) that capture instance deviations. The
   "shape" of the container is then primarily inherited from the
   referenced `SpecimenDefinition`, with explicit overrides on the
   instance.
2. **Push container detail onto Device**: do not add `material`/`type`/
   `cap` to `Specimen.container` at all. Instead, formally specify (via
   a `Device` profile or guidance in the Specimen narrative) that all
   physical container properties are carried on the `Device` referenced
   by `Specimen.container.device`, and that `Specimen.container.device`
   becomes effectively `1..1` whenever container properties need to be
   expressed. Move `Specimen.container.specimenQuantity` discussion off
   the specimen to the container/device, per Daniel Rutz's comment.

#### Justification

Option B.1 avoids duplicating the SpecimenDefinition container shape on
the instance resource, which keeps the Specimen resource lean and
preserves a single source of truth for "what kind of container this is"
(the definitional resource). It addresses Daniel Rutz's framing of the
problem (inconsistency in *where* the concept lives) more directly than
A, and leaves Devices WG's territory untouched.

Option B.2 fully embraces the existing `Specimen.container.device`
pattern and aligns with the FHIR principle of preferring referenced
resources over duplicating element sets. It would, however, materially
change implementation patterns (every container-bearing specimen would
need a `Device`), which is a larger downstream burden â€” but it cleanly
resolves the OO/Devices boundary question raised by esilver.

Either B variant is more invasive than A in modeling terms but smaller
in element-count terms, and either is harder to land cleanly given
Specimen's R6 normative target.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as **Persuasive â€“ Not Now** (or equivalent), citing that
(a) Specimen is targeted for R6 normative ballot and adding new
elements on `Specimen.container` at this stage risks destabilising the
normative scope, (b) the existing `Specimen.container.device` reference
already provides a path to express container `material`/`type`/`cap`
via a `Device` (or Device profile) representing the physical container,
and (c) implementers requiring richer container data on the instance
can profile `Specimen.container.device.type` and the referenced
`Device` to carry that information today. Recommend that the broader
"definitional vs. instance container modeling" question be tracked as
an R7 work item rather than addressed in the current cycle.

#### Justification

Specimen is a long-standing resource with significant implementation
deployment, and the R6 normative timeline limits the appetite for
adding new optional elements on a backbone that is otherwise stable.
The need described in the ticket is not unmet â€” `Device` plus
`Specimen.container.device` can carry material/type/cap, even if less
ergonomically than direct elements. Declining (or deferring) keeps the
normative scope intact while acknowledging the modeling concern via a
forward-dated work item.

---

### Recommendation

**Recommended disposition:** **A**, with explicit Devices WG sign-off
before the change is finalized.

The OO-on-FHIR call has already acknowledged the inconsistency is real
and that rework is probably warranted; Daniel Rutz's analysis confirms
it is a genuine modeling gap rather than a documentation issue. Option
A is the smallest, most additive change that closes the gap: three
optional `CodeableConcept` siblings on `Specimen.container`, bound to
the same value sets used on `SpecimenDefinition.typeTested.container`,
do not break existing instances and do not disturb
`Specimen.container.device`. Option B is intellectually cleaner but
either introduces a new reference to `SpecimenDefinition` from the
specimen instance (B.1) or substantially changes how implementers
must represent ordinary specimen containers (B.2) â€” both are larger
asks against an R6-normative resource than A. Option C leaves the gap
unaddressed and defers it indefinitely; given that the WG already
agreed rework is warranted, declining is the weakest of the three.

The recommendation is contingent on engaging Devices WG (per esilver's
2026-03-23 and 2026-04-13 comments) so that the new
`material`/`type`/`cap` elements on `Specimen.container` are clearly
positioned relative to `Specimen.container.device` and the underlying
`Device` resource, with narrative guidance on which to use when both
are present.
