## Summary

Ulrike Merrick raises a bundle of follow-up changes to the R6-ballot
Observation resource that depend on the outcome of the still-open
Observation grouping / "panel" representation discussion in Orders &
Observations. Three distinct items are flagged: (1) the example value
set bound to `Observation.code`
(`valueset-observation-codes-observationorboth`) should be tightened or
a rule added so that, when `Observation.organizer = true`, the code is
drawn from LOINC entries flagged as "Order" or "Both" (rather than
"Observation only"); (2) the `obsgrouping` and `gr-result` narrative on
the Observation page (and possibly the Diagnostics Module page) needs to
be updated to reflect whatever grouping mechanism is chosen; and (3) the
existing description of how a "panel" or "battery" is represented (with
`Observation.code` as the panel code, `Observation.value[x]` typically
absent, and members linked via `Observation.hasMember`) needs at minimum
a sentence stating that `Observation.organizer = true` in that case —
and may need broader rewording depending on what the WG decides. The
reporter explicitly notes that if `Observation.organizer` is removed,
item #1 becomes moot.

This ticket is one of a cluster the OO work group is tracking under the
`organizer-grouping` label and is tightly coupled to the parent
discussions in [FHIR-50534](https://jira.hl7.org/browse/FHIR-50534)
(feedback on the `Observation.organizer` ballot note),
[FHIR-54808](https://jira.hl7.org/browse/FHIR-54808) and
[FHIR-54741](https://jira.hl7.org/browse/FHIR-54741) (asking OO to
choose and document a single grouping mechanism for panels in
DiagnosticReport), [FHIR-54525](https://jira.hl7.org/browse/FHIR-54525)
(profile-level use of `organizer = true` for Vital Signs Panel),
[FHIR-55023](https://jira.hl7.org/browse/FHIR-55023) (proposing to
remove `Observation.organizer` entirely if `List` is added to
`DiagnosticReport.result`), and the closely related
[FHIR-50342](https://jira.hl7.org/browse/FHIR-50342) (closed, no change)
asking what `organizer` actually scopes. The ballot-note ticket
[FHIR-53553](https://jira.hl7.org/browse/FHIR-53553) (Applied,
Persuasive) confirms that R6 explicitly invites ballot comments on this
exact area, so FHIR-54887 is best treated as the "carrier" ticket for
the Observation-resource edits that fall out of whatever grouping
decision the WG takes.

## Details

**Description:**

> The value set for [Observation.code](https://hl7.org/fhir/6.0.0-ballot4/observation-definitions.html#Observation.code)
> — which is currently
> [`valueset-observation-codes-observationorboth`](https://hl7.org/fhir/6.0.0-ballot4/valueset-observation-codes-observationorboth.html)
> (example binding only) — should be updated or a rule should be added
> to allow use of LOINC "order" or "both" codes when
> [`Observation.organizer`](https://hl7.org/fhir/6.0.0-ballot4/observation-definitions.html#Observation.organizer)
> = true.
>
> If [`Observation.organizer`](https://hl7.org/fhir/6.0.0-ballot4/observation-definitions.html#Observation.organizer)
> is removed, then this is moot.
>
> **#2:** [`#obsgrouping`](https://hl7.org/fhir/6.0.0-ballot4/observation.html#obsgrouping)
> and [`#gr-result`](https://hl7.org/fhir/6.0.0-ballot4/observation.html#gr-result)
> — this text needs to be updated, if not here, then in the Diagnostics
> Module page.
>
> **#3:** "For grouping related observations such as for a 'panel' or
> 'battery'. In this case the `Observation.code` represents the 'panel'
> code, typically `Observation.value[x]` is not present, and the set of
> member Observations are listed in `Observation.hasMember`. This
> structure permits nested grouping when used with DiagnosticReport
> (e.g. complex micro isolate and sensitivities report) as described in
> the Diagnostics Module" — the text needs to be adjusted depending on
> the grouping outcome. At minimum, a sentence needs to be added to
> indicate `Observation.organizer = true` in this case.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-54887 as the carrier change for the Observation-resource
edits that fall out of the WG's grouping decision, on the assumption
that `Observation.organizer` is retained:

1. **Tighten the binding on `Observation.code`.** Replace (or
   constrain) the existing example binding to
   `valueset-observation-codes-observationorboth` so the value set is
   conditioned on `Observation.organizer`:
   - When `organizer = true`: codes are drawn from LOINC where
     "Order vs. Observation" = `Order Only` or `Both`.
   - When `organizer = false` (or absent): codes are drawn from LOINC
     where "Order vs. Observation" = `Observation Only` or `Both`.
   Implement this either as two value sets and an invariant that
   selects between them, or as a single value set plus a
   FHIRPath-based invariant on `Observation`.
2. **Update the `#obsgrouping` and `#gr-result` narrative** on the
   Observation resource page to (a) name `Observation.organizer = true`
   as the explicit flag for grouper instances and (b) clarify the
   relationship between `hasMember`-based grouping on Observation and
   grouping inside `DiagnosticReport.result`. If the WG prefers to keep
   the canonical guidance on the Diagnostics Module page, leave a
   short pointer in `#obsgrouping`/`#gr-result` and put the full text
   on the Diagnostics Module page.
3. **Update item #3's panel/battery paragraph** to add at minimum a
   sentence such as: *"In this case `Observation.organizer` SHALL be
   `true` to identify the resource as a grouper."* If the WG also
   adopts FHIR-54525, cite the Vital Signs Panel as the worked example.

#### Justification

This is what the ticket literally asks for, and each piece is
internally consistent: item #1 reuses the LOINC "Order vs. Observation"
axis that already underpins the existing
`observation-codes-observationorboth` value set, so it is a refinement
rather than a new modelling concept. The narrative changes in #2 and
#3 only become defensible once the WG has actually picked a grouping
mechanism — they are descriptive, not prescriptive — so this
disposition should be conditional on the resolutions of FHIR-54808 and
FHIR-54741. Reporters of the parent grouping tickets and the ballot
note FHIR-53553 (Persuasive) all assume `organizer` survives
normative; tightening its binding before lock-in is the right time to
do it.

---

### Disposition B: Alternative Approach

#### Proposal

Defer FHIR-54887 and treat the three sub-items as outputs of a single
combined "grouping disposition" on FHIR-54808 / FHIR-54741, then close
FHIR-54887 by reference. Concretely:

- Mark FHIR-54887 as **Persuasive with Modification** with the
  resolution that the substantive changes are tracked under the
  grouping-decision tickets (FHIR-54808, FHIR-54741), and that
  FHIR-54887 will be Applied automatically once those tickets are
  resolved. Optionally split FHIR-54887 into three sub-issues
  (one per numbered item) so each can be tracked against the grouping
  outcome, and close the parent.
- Pre-commit only to the *narrative* hooks that are independent of the
  grouping choice: add a single sentence to `#obsgrouping` /
  Diagnostics Module that flags the grouping discussion is in flight
  and points at the Confluence "Grouping Option Analysis" page. This
  satisfies the reporter's underlying complaint (that the current text
  is silent on grouper semantics) without hard-coding a mechanism the
  WG has not yet picked.
- Leave the `Observation.code` binding untouched until the WG decides
  whether `Observation.organizer` is retained
  (versus FHIR-55023's "remove organizer if List is allowed in
  `DiagnosticReport.result`" branch).

#### Justification

The ticket's own first paragraph admits that #1 is moot if `organizer`
is removed, and #2 and #3 explicitly read "depending on grouping
outcome". Acting on FHIR-54887 in isolation risks (a) tightening a
binding on an element that may be deleted next ballot and (b) baking a
specific grouping mechanism into the Observation page that the WG then
overrides via FHIR-54808 / FHIR-54741. Folding the work into the
grouping-decision tickets keeps a single source of truth for the
Diagnostics Module narrative and avoids the ballot churn of revising
the same paragraphs twice. The cost is that FHIR-54887 stays open
across more than one ballot cycle, which is acceptable for a
ballot-comment ticket whose own author conditioned it on a pending
decision.

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-54887 as **Not Persuasive** with the rationale that the
requested changes are either (a) already covered by the active
grouping-decision tickets FHIR-54808 / FHIR-54741 and the ballot note
FHIR-53553 (so opening a parallel track risks divergent guidance) or
(b) dependent on a hypothetical outcome (`organizer = true` selecting a
narrower LOINC subset) that the WG has not yet endorsed. Direct the
reporter to the grouping-decision tickets and to FHIR-50342, which the
WG already closed as "no action required" on the same scope question,
and close this ticket without spec changes.

#### Justification

There is a defensible but narrow case for this: the ticket is a
collection of conditional follow-ups rather than a single concrete
change, item #1 is potentially superseded by FHIR-55023, and item #3
overlaps with FHIR-50342, which OO already declined on a Block-Vote
Backlog (8-0-2). Declining keeps the ballot-comment surface small and
forces the conversation back into the parent grouping tickets. The
weakness — and it is a real one — is that the reporter's #2 ("the
`#obsgrouping` / `#gr-result` text needs to be updated") is plainly
correct under *any* grouping outcome, so a pure decline leaves a known
documentation defect on a normative-track resource.

---

### Recommendation

**Recommended disposition:** B (Persuasive with Modification — defer
substantive edits to the grouping-decision tickets, but accept a small
"work in progress" narrative hook now).

The reporter is essentially saying "once the grouping question is
settled, the Observation page needs these three follow-up edits", and
that is exactly what Disposition B codifies. Disposition A risks
locking in a tightened `Observation.code` binding and grouping
narrative just before FHIR-55023 / FHIR-55019 may invalidate the very
element the binding is conditioned on; doing the work twice is more
costly to balloters than waiting one cycle. Disposition C is too
strict: item #2 (the `#obsgrouping` / `#gr-result` text is currently
silent on grouper semantics) is genuinely a documentation gap, and the
ballot note in FHIR-53553 (Persuasive, Applied) explicitly invites
exactly this kind of feedback. Folding FHIR-54887 into the grouping
tickets — and adding the small "decision in flight" pointer now —
captures the value of the comment, keeps the grouping narrative in one
place, and lets the WG resolve the value-set tightening at the same
time it decides whether `organizer` survives.
