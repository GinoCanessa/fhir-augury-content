## Summary

The reporter requests that a new core element, `Observation.supportingDevice`,
be added to the `Observation` resource as a `0..*` `Reference(Device)`. The
intent is to formally capture the supporting devices, kits, reagents, and QC
materials that were used or otherwise involved in producing a measurement
(e.g., test kits, calibration kits, QC materials), distinct from the
measurement device already conveyed by `Observation.device`. The driver is
patient safety (e.g., the ability to identify and recall results when a
specific reagent or kit is later found to be defective) and regulatory
alignment with the European Health Data Space (EHDS) and the eHealth Network
guideline on cross-border electronic health data exchange (Directive
2011/24/EU), specifically point A.5.2.5. The reporter explicitly asks for a
**core element** rather than an extension, arguing that an extension would
fragment implementations and prevent EHDS-aligned systems from interpreting
the device involvement consistently. No cross-referenced Jira tickets, Zulip
discussions, or GitHub items were found in the FHIR Augury index for this
ticket at the time of preparation.

## Details

**Description:**

Currently, there is a gap in the Observation resource regarding the capture of
multiple devices or materials involved in a measurement. To ensure patient
safety and meet regulatory requirements for i.e. the European Health Data
Space (EHDS), it is necessary to formally link supporting devices (e.g., Test
Kits, Calibration Kits, ...) to an Observation.

This change should be implemented as a **core element** rather than an
extension to ensure maximum interoperability.

### Summarization of Reasons

- **Patient Safety:** Essential for tracking and recalling results if specific
  reagents or kits are found to be defective.
- **Regulatory Compliance:** Aligns with the *eHealth Network GUIDELINE on
  electronic exchange of health data (Cross-Border Directive 2011/24/EU)*,
  specifically Point A.5.2.5.
- **Standardization:** Avoids the fragmentation caused by custom extensions,
  ensuring that all systems communicating with the EHDS can interpret the
  device involvement.

### Proposed Change

Add `supportingDevice` as a core element to Observation.

- **Element Name:** `supportingDevice`
- **Cardinality:** `0..*`
- **Type:** `Reference(Device)`
- **Definition:** A reference to the supporting devices, kits, or reagents
  (e.g., TestKit, Calibration Kit, QC materials) that were used or involved
  in generating the measurement.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new core element to the `Observation` resource exactly as described in
the ticket:

- **Path:** `Observation.supportingDevice`
- **Cardinality:** `0..*`
- **Type:** `Reference(Device)`
- **Short:** "Supporting device(s) (e.g., test kits, calibration kits, QC
  materials) involved in generating the observation"
- **Definition:** "A reference to the supporting devices, kits, or reagents
  (e.g., TestKit, Calibration Kit, QC materials) that were used or involved
  in generating the measurement, but that are not the primary measuring
  device captured by `Observation.device`."
- **Comments:** Distinguish from `Observation.device` (the producing
  instrument) and from `Specimen.processing.device`/`Specimen.container`
  (specimen-level handling). Multiple supporting devices may be referenced;
  use `Device.type`, `Device.identifier` (UDI/lot/serial) and
  `Device.expirationDate` on the referenced `Device` to convey what is
  needed for recall and traceability.
- **Mapping/Workflow:** Update Observation intro / mappings to mention
  alignment with the eHealth Network guideline A.5.2.5 / EHDS use case.

#### Justification

The request identifies a real semantic gap: `Observation.device` is
generally interpreted as the single device that performed the measurement,
whereas EHDS-style traceability and lot-level recall require recording
*all* devices, kits, and reagents that materially contributed to the
result. Modeling this as a core element keeps the information first-class
and uniformly queryable across implementers, which is the explicit
requirement for EHDS interoperability cited in the ticket. A
`Reference(Device)` array is consistent with the existing pattern used by
`Observation.device`, `Observation.focus`, `Observation.partOf`, and
`Observation.basedOn`, so the change is mechanically simple and
backward-compatible (it is an additive `0..*` element and does not break
any existing instance). The ticket is labeled `R6-Normative-Ballot`, so
landing additive R6 changes prior to normative lock is the appropriate
window for this kind of addition.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new core element. Instead, address the use case by:

1. **Reusing/relaxing `Observation.device`** so that it is `0..*` (today
   it is `0..1`) and explicitly documenting that it covers any device,
   kit, reagent, or QC material that contributed to the result, with
   `Device.type` distinguishing the role (measuring instrument vs. test
   kit vs. calibration kit vs. QC material). A FHIRPath/discriminator
   on `Device.type` could be used by IGs (e.g., EHDS Lab Report) that
   need to slice out the primary measuring device.
2. If broadening `Observation.device` is considered too disruptive at
   this stage of R6, publish a **standard extension**
   `http://hl7.org/fhir/StructureDefinition/observation-supportingDevice`
   of type `Reference(Device)`, `0..*`, owned by Orders & Observations,
   and reference it from the Observation page so EHDS / IG authors get a
   single canonical extension to adopt.

#### Justification

Adding new core elements late in the R6 cycle has downstream cost for
every implementer, validator, mapper, and IG. The `Device` resource
already carries everything needed for recall (UDI, lot, expiration,
type), so the modeling question is really "where is the back-pointer
attached and how many are allowed?" Option B.1 (relax cardinality of
`Observation.device`) avoids introducing two near-identical references
on the same resource, which the working group has historically tried to
avoid because it forces senders and receivers to make policy decisions
about which slot to populate. Option B.2 (a standard extension) is a
non-breaking, low-cost path that gives EHDS the canonical artifact it
needs without committing the base resource permanently; the reporter's
"fragmentation" concern is mitigated when the extension is published as
a *standard* extension by O&O rather than invented per-jurisdiction.

---

### Disposition C: Decline

#### Proposal

Decline the request to add `Observation.supportingDevice` as a core
element. Recommend instead that the reporter (and the EHDS profile
authors) use the existing `Observation.device` reference for the primary
measuring device and convey supporting devices, reagents, kits and QC
materials either via:

- a profile-level extension defined by the EHDS / Laboratory IG, or
- by chaining through `Specimen` (`Observation.specimen` →
  `Specimen.processing.device`, `Specimen.container.device`) and
  `DeviceUsage` / `Procedure.usedReference` for non-specimen workflows.

Rationale category: **already addressed by existing mechanisms**.

#### Justification

The capture of devices, reagents, lots and kits used in a workflow is
already supported by `Device` (with UDI/lot/expiration), `DeviceUsage`,
`Procedure.usedReference`, and `Specimen.processing.device` /
`Specimen.container`. EHDS traceability can be satisfied today by
profiling these existing resources. Adding another reference to
`Observation` increases modeling complexity and creates ambiguity with
`Observation.device`: implementers will reasonably disagree about which
slot a given device belongs in (e.g., a point-of-care analyzer that is
both the measuring device *and* shipped with a single-use reagent
cartridge). Late-cycle additions to a resource heading toward Normative
also raise the bar for justification, and a regulatory citation alone
is not generally accepted as sufficient when the underlying need is
already satisfiable via profiling.

---

### Recommendation

**Recommended disposition:** A

The use case is concrete (lot/kit-level recall and EHDS traceability), the
proposed shape is conservative (`0..*` `Reference(Device)`), and the change
is additive and backward-compatible. `Observation.device` is widely
understood as the single measuring instrument, and overloading it to also
mean "any device/kit/reagent that touched the result" (Disposition B.1)
will create exactly the implementation ambiguity the ticket is trying to
eliminate. A standard extension (Disposition B.2) is technically
sufficient but undermines the EHDS goal that *all* systems be able to
interpret the device involvement without negotiating extension support;
since EHDS is explicitly cited and the ticket is filed against the
R6-Normative-Ballot window, capturing this as a core element now is
materially cheaper than adding it post-Normative. Disposition C
understates the modeling cost of layering profile-level extensions over
`Observation.device` semantics that the community already interprets
narrowly. The workgroup should adopt A, with two clarifications during
drafting: (1) tighten the short/definition so that `supportingDevice` is
unambiguously *not* the primary measuring device, and (2) add cross-links
in the resource intro to `Specimen.processing.device` and `DeviceUsage`
so implementers know when to use which slot.
