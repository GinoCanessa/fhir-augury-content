## Summary

This is a Technical Correction against the **Order Catalog (FHIR)** IG
(under Orders & Observations). In the final R5 release of the FHIR base
specification, `ActivityDefinition.observationRequirement` and
`ActivityDefinition.observationResultRequirement` were retyped from
`Reference(ObservationDefinition)` to `canonical(ObservationDefinition)`.
The Order Catalog IG carries `ActivityDefinition` examples (notably the
two urine 24h creatinine examples) whose XML still uses the
pre-final-R5 `Reference` form for these elements, which makes the
examples fail to validate against the published R5 ActivityDefinition
StructureDefinition. The ticket asks that these examples be updated to
use the `canonical` data type so they conform to the released R5 spec.

No other Jira tickets were surfaced as cross-references; the only
cross-reference returned by FhirAugury is a single GitHub commit on
`HL7/fhir-order-catalog` (504ccbb, by Rob Hausam, 2025-03-06) whose
message names this ticket and which already implements the change in
the two example files. The Jira ticket itself, however, is still in
"Triaged" state with no resolution recorded.

### Disposition A: Accept As Requested

#### Proposal

Apply the technical correction to all `ActivityDefinition` examples in
`HL7/fhir-order-catalog` so that `observationRequirement` and
`observationResultRequirement` use the R5-final `canonical(ObservationDefinition)`
data type instead of `Reference(ObservationDefinition)`. Concretely:

- Confirm and merge the change already implemented by commit
  [504ccbb](https://github.com/HL7/fhir-order-catalog/commit/504ccbbe15bd58e15b5edfe329f170edac9f8a2f),
  which updates `input/examples/example-lab-urine24h-creatinine-clearance.xml`
  and `input/examples/example-lab-urine24h-creatinine.xml`. In XML this
  means replacing each `<observationRequirement><reference value="…"/></observationRequirement>`
  block with `<observationRequirement value="<canonical-url>"/>`, using
  the canonical URL of the referenced `ObservationDefinition` (e.g. the
  IG-published canonical for the corresponding lab procedure / lab
  observation definition).
- Sweep the rest of `input/examples/`, `input/fsh/examples/`,
  `input/resources/`, and any other authoring root identified by the
  briefing for additional `ActivityDefinition` instances or profiles
  that still use the old `Reference` form, and apply the same
  correction.
- Re-run the IG publisher to confirm the examples now validate against
  R5 `ActivityDefinition`.
- Resolve the ticket as **Persuasive — already applied** (or
  **Persuasive with Modification** if the sweep finds additional
  occurrences beyond the two already fixed) and link the commit (and
  any follow-up PR) on the Jira ticket.

#### Justification

This is the minimum change needed to bring the Order Catalog IG's
examples back into alignment with the published R5 base specification.
The R5 ballot/QA cycle deliberately retyped these elements from
`Reference` to `canonical` so that an `ActivityDefinition` would refer
to a logical `ObservationDefinition` artifact (by canonical URL,
optionally version-pinned) rather than to a physical instance — the
Order Catalog IG already relies on canonical references throughout for
its definitional content, so this aligns the examples with both the
spec and the IG's own usage pattern. The author of the ticket has
already produced the fix as a commit, demonstrating both feasibility
and minimal blast radius (two example files, +6/−18 lines).

---

### Disposition B: Alternative Approach

#### Proposal

Take the technical correction from Disposition A *and* extend it with
two additional housekeeping actions, treating the ticket as the trigger
for a broader R5-final consistency pass on the IG's
`ActivityDefinition` content:

- Add an automated check (e.g., an `ignoreWarnings.txt` review plus a
  small `sushi`/`xpath`/`jq` script run from CI) that fails the IG
  build if any `ActivityDefinition.observationRequirement` or
  `observationResultRequirement` element appears with a `Reference`
  child element instead of a canonical value. This prevents regressions
  if examples are later copy-pasted from older sources.
- Add a one-paragraph note to `input/pagecontent/changes.xml` (or the
  equivalent change-log page identified by the briefing) recording that
  these examples were realigned to the R5 final `canonical` typing,
  along with a link to the FHIR R5 `ActivityDefinition` definition.
  Also confirm that any prose in `input/pagecontent/labservices.xml` or
  related lab-service narrative pages that describes how an
  `ActivityDefinition` points to an `ObservationDefinition` uses
  "canonical" language rather than "reference" language.

#### Justification

The literal fix in Disposition A clears the immediate validation
problem but does nothing to prevent the same drift in future examples,
and FHIR-49964 itself is evidence that this drift can survive an
entire ballot cycle unnoticed. A small CI check and an explicit change-
log note cost very little and substantially reduce the chance that a
future example author re-introduces the older `Reference` form. The
trade-off vs. Disposition A is a small amount of additional CI/build
plumbing and a brief narrative edit; given that the ticket is already
classed as a Technical Correction, this remains within scope and does
not require any further workgroup signoff on substantive content.
