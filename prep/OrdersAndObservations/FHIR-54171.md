## Summary

Eric Haas reports that the new `DocumentReference.related` element introduced
in R6 is underdefined and overlaps semantically with several other
relationship elements on the same resource (`basedOn`, `context`, `event`,
`relatesTo`). The element's description ("Any other resource this document
reference was created or to which the creation of this record is tightly
associated") is vague — the phrase "tightly associated" gives implementers no
basis on which to choose `related` over the more specific neighbours, and no
worked examples are provided.

In addition, the ticket calls out a **lossy migration path**: the
"Changes from R4 and R4B to R6" section advertises
`DocumentReference.context.related → basedOn` as the migration, but
`basedOn` is constrained to *request* resources, whereas R4
`context.related` was `Reference(Any)`. Implementations that used
`context.related` to point to non-request resources (Encounter, Observation,
Provenance, Bundle, etc.) are left without a documented migration target. The
ticket suspects the new `related` element may be the actual intended
replacement, but the changelog never says so.

The ticket overlaps strongly with two prior O&O tickets that have already
moved through disposition:
[FHIR-53550](https://jira.hl7.org/browse/FHIR-53550) (Lloyd McKenzie:
"DocumentReference.related shouldn't exist") and
[FHIR-53824](https://jira.hl7.org/browse/FHIR-53824) (Lee Starnes:
"Problems with DocumentReference related/relatesTo"). Both were resolved
**Persuasive with Modification** — `related` is being changed into a
backbone with a `code` (CodeableConcept, example binding to
`AssociatedReferenceType` / `relatedArtifactType`: `documentation`,
`justification`, `citation`, `part-of`) and a `target 1..1 Reference(Any)`,
plus tighter narrative text in Implementation Notes 10.2.5 distinguishing
`relatesTo` (lifecycle only) from `related` (everything else). Resolution PR
[HL7/fhir#4038](https://github.com/HL7/fhir/pull/4038) implements this. The
present ticket arrives after that work and effectively asks the same
clarification *plus* an explicit fix to the R6 migration changelog.

## Details

**Description:**

The new `related` element is described as:

> "Any other resource this document reference was created or to which the
> creation of this record is tightly associated."

But there are already multiple elements that seem to overlap:

- `basedOn` — "Procedure that caused this media to be created"
- `context` — "Encounter the document reference is part of"
- `event` — "Main clinical acts documented"
- `relatesTo` — "Relationships to other documents"

The `related` element lacks a clear distinction; its description is vague
("tightly associated") and doesn't clearly distinguish when to use `related`
vs. the other relationship elements. It provides no guidance on its intended
use case using specific examples.

**Also the Migration Path is Incomplete.**

The "Changes from R4 and R4B to R6" section states:

> "DocumentReference.context.related — Deleted (→ basedOn)"

But `basedOn` only allows **request** resources, while the old
`context.related` could reference any resource. This is a lossy migration —
implementations that used `context.related` to reference non-request
resources have no clear migration path. The new `related` element (added in
R6) might be the intended replacement, but the change log doesn't mention
this. The migration guidance is incomplete.

**Comments:** None on this ticket.

### [FHIR-53550: DocumentReference.related shouldn't exist](https://jira.hl7.org/browse/FHIR-53550)

Lloyd McKenzie's ticket argues that "related" is too fuzzy to be a useful
core element ("created before"/"created after" are technically
relationships, so anything could qualify) and asks for either a complex
structure with a coded relationship type (extensible binding) or that
generic relationships be expressed via explicit extensions instead.
Resolved **Persuasive with Modification** on 2026-03-06; the resolution
text says the disposition for FHIR-53824 fulfils the request by adding a
backbone with a `CodeableConcept`. This is the single most directly
overlapping ticket — FHIR-54171's "underdefined" complaint is the same
problem from a different angle.

Users involved: Lloyd McKenzie (reporter), John Moehrke (champion),
Cooper Thompson, Marti Velezis, Vadim Peretokin, JIRAUSER19407 (jdlnolen).

### [FHIR-53824: Problems with DocumentReference related/relatesTo](https://jira.hl7.org/browse/FHIR-53824)

Lee Starnes's ticket carries the *applied* resolution that operationalises
both FHIR-53550 and (effectively) FHIR-54171. It tightens `relatesTo` to
"document reference lifecycle changes only" and converts `related` into a
backbone with `code` (CodeableConcept, example binding to
`AssociatedReferenceType` using `relatedArtifactType` with seed values
`documentation`, `justification`, `citation`, `part-of`) and
`target 1..1 Reference(Any)`. It also adds Implementation Notes 10.2.5
text explicitly stating that `related` is "intended to be flexible, but
not to be used for the purposes of relatesTo (document lifecycle)" and
"not intended to duplicate the functionality of other elements, e.g.
.subject and .context." Status **Applied**, resolution **Persuasive with
Modification**.

Users involved: Lee Starnes (reporter), John Moehrke (assignee), plus
the FHIR-53550 commenters above.

### [FHIR-41186: DocumentReference no longer supports "Any" context.](https://jira.hl7.org/browse/FHIR-41186)

Cooper Thompson's earlier ticket establishes the *use case* that the
new `related` element exists to satisfy: in R4 you could link a
DocumentReference to any relevant context via `context.related`, and
that capability disappeared in R5 because `subject` and `basedOn` are
type-constrained. Status **Applied**, resolution **Persuasive with
Modification** — this is the ticket that motivated re-introducing a
generic relationship element in R6 in the first place, and is the
direct ancestor of the very migration gap FHIR-54171 calls out.

Users involved: Cooper Thompson (reporter), John Moehrke (assignee).

## Related Zulip Discussions

### [implementers > ✔ Composition relationship to DocumentReference](https://chat.fhir.org/#narrow/stream/implementers/topic/%E2%9C%94%20Composition%20relationship%20to%20DocumentReference)

Vadim Peretokin asked how to link a scanned/printed/signed PDF
(`DocumentReference`) back to its source `Composition` Bundle, given that
`relatesTo` only allows DocumentReference→DocumentReference references.
Kari Heinonen pointed at the new R6 `DocumentReference.related 0..*
Reference(Any)`; Vadim countered that without a coded relationship type
the element feels like a generic catch-all. John Moehrke defended the
element ("for the broad use-case of DocumentReference, anything might be
related") and explained the IHE/XDS heritage (one bucket for any related
identifier when no more-specific element fits). Lloyd McKenzie pushed
back hard, calling the element "an anti-pattern" and announcing he had
filed FHIR-53550. Rik Smithies suggested the `derivation-reference`
extension or `Provenance` as already-existing alternatives. Eric Haas
(this ticket's reporter) observed that R6 has substantially rewritten
FHIR signatures and steered the original use case toward
`Provenance.signature` per §6.1.2.2.9. Vadim ultimately resolved his
own thread by pre-adopting the R6 Signature rules. Consensus: `related`
as currently shipped is too vague; either it gets a coded relationship
type (Lloyd, Vadim) or implementers should reach for extensions /
Provenance for specific semantics (John, Rik).

Participants: Vadim Peretokin, Kari Heinonen, Rik Smithies, John
Moehrke, Lloyd McKenzie, Gino Canessa, Eric Haas, Grahame Grieve.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-54171 as **already resolved by FHIR-53824 / FHIR-53550** and
mark it **Persuasive with Modification — Already Applied**. Concretely:

1. Confirm that the FHIR-53824 resolution (now Applied via
   [HL7/fhir#4038](https://github.com/HL7/fhir/pull/4038)) addresses the
   "underdefined" complaint by:
   - Restructuring `DocumentReference.related` as a backbone with
     `code 0..1 CodeableConcept` (example binding to
     `AssociatedReferenceType` / `relatedArtifactType`: `documentation`,
     `justification`, `citation`, `part-of`) and `target 1..1 Reference(Any)`.
   - Tightening `relatesTo` to "Relationships that this document reference
     has with other document references that already exist" (lifecycle only).
   - Adding Implementation Notes 10.2.5 text explicitly distinguishing
     `relatesTo` from `related` and forbidding overlap with `.subject`
     and `.context`.
2. **Additionally**, fix the migration changelog gap that FHIR-54171
   surfaces (which FHIR-53824 does *not* address): update the "Changes
   from R4 and R4B to R6" entry from
   `DocumentReference.context.related — Deleted (→ basedOn)`
   to a two-line entry that says
   *"Replaced by `DocumentReference.related` (new backbone with coded
   relationship type) for non-request targets; use
   `DocumentReference.basedOn` only for request resources (ServiceRequest,
   etc.)."*

#### Justification

The substantive request — disambiguate `related` from `basedOn`,
`context`, `event`, and `relatesTo` and give it a clearer purpose — has
already been Applied through FHIR-53824, with the disposition explicitly
calling out the same anti-pattern Eric Haas describes here. Closing this
ticket without acknowledging that work would invite duplicate effort and
confuse the changelog audit trail. Conversely, the **migration changelog
half** of FHIR-54171 is a genuinely new finding that FHIR-53824 did not
cover: the changelog still asserts a single-target migration
(`→ basedOn`) that is type-incompatible with most legacy
`context.related` usage. That correction is small, factual, and
backwards-compatible (changelog only, no normative element changes).

---

### Disposition B: Alternative Approach

#### Proposal

Accept the migration-changelog correction (per Disposition A, item 2),
but treat the "underdefined" complaint as a request for **one additional
clarification round on top of FHIR-53824**:

- Add a sentence to the new `related.code` description noting that
  implementers SHOULD populate `code` and that leaving it absent
  (`0..1`) signals "unspecified relationship — receivers may treat as
  uninterpretable."
- Add a worked example to the DocumentReference resource page showing a
  scanned-PDF DocumentReference linking back to its source Composition
  Bundle via `related.code = #documentation, related.target =
  Reference(Bundle)` (the exact scenario from the implementers Zulip
  thread).
- Strengthen the `AssociatedReferenceType` example binding to
  *preferred* if the workgroup judges the four seed codes are
  sufficient; otherwise leave the binding example.

#### Justification

FHIR-53824's resolution introduces the structure but leaves `code` as
`0..1` with an *example* binding, which preserves the very ambiguity
Eric Haas and Lloyd McKenzie are concerned about — implementers can
still send a bare `Reference(Any)` with no relationship semantics. A
SHOULD on populating `code`, plus a canonical worked example sourced
from a real implementer thread, gives the element a clear "happy path"
without forcing a normative break. Trade-off vs. Disposition A: more
editorial work and re-balloting risk for the resource page, but a
materially clearer element. This approach respects the work already
Applied while honouring the spirit of FHIR-54171's request for
"specific examples" of intended use.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-54171 as **Not Persuasive — Already Addressed** and close
without further action. Rationale: the substance of the complaint
(overlap with `basedOn`/`context`/`event`/`relatesTo` and lack of
clear use case) has been adjudicated by FHIR-53550 and FHIR-53824, both
already Applied via PR HL7/fhir#4038. The migration-changelog wording is
auto-generated from element-history metadata and will refresh on the
next regen; no manual ticket is required.

#### Justification

Defensible because the core element-design issue is genuinely closed
and re-litigating it risks reopening a settled disposition. However,
this stance only holds if the changelog claim is true — namely, that
the regenerated "Changes from R4 and R4B to R6" page will, after PR
4038's merge, automatically reflect `context.related → related` (with
the type widening narrative) without manual intervention. If the
changelog is hand-maintained, declining this ticket leaves the lossy
migration claim in the published spec.

---

### Recommendation

**Recommended disposition:** A

The cleanest path is to acknowledge the work already Applied via
FHIR-53824 (so the audit trail is honest) **and** take the small,
clearly-correct fix to the R4→R6 changelog that FHIR-54171 uniquely
identifies. Disposition C is risky because it depends on an
unverified assumption about how the migration changelog is generated,
and leaving the published "→ basedOn" claim in place is precisely the
kind of subtle implementer trap O&O should not ship into R6. Disposition
B is attractive — and the workgroup may well want to layer it on later
— but the SHOULD/example-binding tightening is a separate design
conversation that risks delaying the changelog fix, which can land
immediately as an editorial change. Recommend A; if the workgroup
agrees the `code` element is too permissive, file a follow-up tracker
for B rather than blocking this ticket on it.
