## Summary

The reporter flags a three-way inconsistency between the narrative "Text
Summary" (point 7, *"When using a panel code"*) on the Vital Signs profile
page and what the `vitalsignsbase` profile actually constrains. Point 7
asserts that, when an `Observation` carries a panel LOINC code, there is an
extensible binding on `Observation.code.coding.code` to the **Vital Signs
Units** value set. The reporter argues this is wrong on three counts:

1. The element should be `Observation.component.code.coding.code` (the
   per-component code), not `Observation.code.coding.code` (the panel code
   itself).
2. The bound value set should be **Vital Signs**
   (`valueset-observation-vitalsign-measurement.html`), not **Vital Signs
   Units** (`valueset-ucum-vitals-common.html`); the latter is a UCUM unit
   value set, not a measurement-code value set.
3. Most importantly, the actual `vitalsignsbase` StructureDefinition does
   **not** declare any binding on `Observation.component.code` at all (see
   `vitalsigns-definitions.html#vitalsigns.Observation.component.code`),
   so the narrative describes a binding that does not exist in the
   normative profile.

The cross-reference index surfaces only the FHIR artifact mentions
(`Observation.code.coding.code`, `Observation.component.code.coding.code`,
`Observation.component.code`); there are no linked Jira tickets, Zulip
threads, or GitHub items associated with this ticket.

## Details

**Description:**

Within the "Text Summary" **point 7** ("When using a panel code") says that:

- a LOINC code in `Observation.code.coding.code` which has an
  [extensible](https://hl7.org/fhir/6.0.0-ballot4/terminologies.html#extensible)
  binding to:
  - [Vital Signs Units](https://hl7.org/fhir/6.0.0-ballot4/valueset-ucum-vitals-common.html)
    value set.

Connected with that statement are **three mistakes**:

- Instead of `Observation.code.coding.code` it should be
  `Observation.component.code.coding.code`.
- Instead of
  [Vital Signs Units](https://hl7.org/fhir/6.0.0-ballot4/valueset-ucum-vitals-common.html)
  it should be
  [Vital Signs](https://hl7.org/fhir/6.0.0-ballot4/valueset-observation-vitalsign-measurement.html).
- Most of all, in contrast to the statement in **point 7** of "Text
  Summary", the profile does not specify a value set binding at all for
  [Observation.component.code](https://hl7.org/fhir/6.0.0-ballot4/vitalsigns-definitions.html#vitalsigns.Observation.component.code).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat all three reported mistakes as defects in the Vital Signs profile
narrative and reconcile narrative + StructureDefinition as follows:

1. In the "Text Summary" of the Vital Signs profile page, rewrite point 7
   ("When using a panel code") so that the element referenced is
   `Observation.component.code.coding.code` (the per-component code),
   **not** `Observation.code.coding.code`.
2. In the same point 7, replace the value set reference from **Vital
   Signs Units** (`ucum-vitals-common`, a UCUM units VS) to **Vital
   Signs** (`observation-vitalsign-measurement`, the measurement-code
   VS).
3. In the `vitalsignsbase` StructureDefinition, add an **extensible**
   binding on `Observation.component.code` to the **Vital Signs**
   (`observation-vitalsign-measurement`) value set, so that the
   normative profile actually carries the binding the narrative
   describes. Regenerate the HTML so
   `vitalsigns-definitions.html#vitalsigns.Observation.component.code`
   reflects the new binding.

#### Justification

The reporter has correctly identified an internal inconsistency: the
narrative claims a binding that does not exist in the StructureDefinition
**and** points at the wrong element **and** at the wrong value set
(units, not measurement codes). Picking just one of the three to fix
would leave the page self-contradictory. The Vital Signs panel pattern
in FHIR has always been "the panel code goes on `Observation.code`; each
measured quantity is a `component` whose `component.code` carries the
specific LOINC for that measurement," so binding the **components'**
codes to the Vital Signs measurement value set is the consistent design,
and is what the narrative was clearly trying to say. Adding an
extensible binding (not required) preserves implementer flexibility for
codes outside the Vital Signs VS.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the narrative only and **do not** add a binding to
`Observation.component.code` in the `vitalsignsbase` StructureDefinition.
Specifically:

1. Rewrite "Text Summary" point 7 to describe the *intent* — "when using
   a panel LOINC code, each `Observation.component.code.coding.code`
   should be drawn from the Vital Signs value set" — but phrase it as
   **guidance** rather than as a declared binding.
2. Add an explicit caveat that the binding lives on the *individual*
   vital sign profiles (e.g., `bodyheight`, `heartrate`,
   `bloodpressure`), where `Observation.component.code` is already
   constrained to a fixed LOINC, and that the base `vitalsignsbase`
   profile intentionally leaves `Observation.component.code` unbound.
3. Also correct the value-set URL in the narrative (Vital Signs, not
   Vital Signs Units) and the element path
   (`Observation.component.code.coding.code`).

#### Justification

The Vital Signs profile family already pins each component code on the
**concrete** vital-sign profiles (e.g., systolic / diastolic in
`bloodpressure`); adding an extensible binding on the **base** profile
risks (a) appearing to relax those tighter constraints if a child
profile inherits and re-binds, and (b) implying that *any* component on
*any* vital-sign-derived Observation must come from the Vital Signs VS,
including custom components like device-related metadata. Treating point
7 as documentation rather than a normative binding keeps the base
profile minimally constrained while still correcting the user-visible
errors that triggered the ticket.

---

### Disposition C: Decline

#### Proposal

Decline the request as worded. Acknowledge that "Text Summary" point 7
is imprecise, but treat it as an editorial / documentation issue rather
than a profile-conformance defect, and resolve as **Not Persuasive**
with the explanation that the bindings on individual vital-sign profiles
(bodyheight, heartrate, bloodpressure, etc.) already deliver the
behavior the narrative was trying to describe, and no normative change
is required.

#### Justification

A defensible decline rests on two points: (1) the *normative* artifact
is the StructureDefinition, not the narrative, so the StructureDefinition
is not "wrong" — it simply does not bind `component.code` at the base
profile, which is by design for a base profile that downstream profiles
specialize; and (2) reopening the binding on a base profile this late in
the R6 ballot cycle could surface secondary issues (downstream profile
conformance, cross-IG impact). However, this disposition leaves the
narrative inconsistency in place unless paired with at least an editorial
fix, so a pure decline is the weakest of the three options.

---

### Recommendation

**Recommended disposition:** A

The reporter is documenting an objective internal contradiction: the
narrative asserts an extensible binding that the StructureDefinition does
not contain, and even the element path and value set named in the
narrative are wrong. Disposition A is the only option that leaves the
page self-consistent and matches what point 7 was clearly trying to
communicate (per-component LOINC codes drawn from the Vital Signs
measurement VS). Disposition B is a reasonable fallback if O&O decides,
on review, that adding a base-profile binding has unintended downstream
effects on the concrete vital-sign profiles — in that case the
narrative-only fix is acceptable, but the value-set URL and element path
errors must still be corrected. Disposition C should be avoided: leaving
a normative profile page with a demonstrably wrong element path and a
wrong value set link is a poor outcome regardless of whether the binding
itself is added.
