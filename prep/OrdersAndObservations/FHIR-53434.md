## Summary

The reporter (Finnie Flores, working in the Alberta Health Services context)
asks that a new code, **"semi-urgent"**, be added to the
[`RequestPriority`](http://hl7.org/fhir/ValueSet/request-priority) value set
that is **required**-bound to `ServiceRequest.priority` (and, by inheritance
from the `Request` pattern, to the `priority` element on every other request
resource). The motivation is concrete: Alberta's gastroenterology referral
pathways for colorectal cancer (the Iron Deficiency Anemia and the High Risk
Rectal Bleeding pathways) recognize three clinical urgency levels — routine,
semi-urgent, urgent — each with defined service-time targets. Today
implementers cannot represent the middle level without distorting the meaning
of either `routine` or `urgent`/`asap`, and the required binding leaves no
escape hatch. The Orders & Observations co-chair (Marti Velezis) has asked
that the ticket be linked to a Ballot Vote so it can be discussed during R6
Normative ballot reconciliation; another OO commenter has also signalled that
internal discussions are still under way.

Two cross-referenced Jira tickets give important context. **FHIR-24923**
("Request adding 'elective' to ServiceRequest.priority value set") is the
most direct precedent: in 2024 the OO work group resolved that ticket as
**Not Persuasive**, reasoning that "elective" describes the *type* of service
rather than its *priority*, and that `routine` already covers the use case.
**FHIR-55655** ("Change Binding to Extensible") is an in-flight FHIR
Infrastructure ticket from Eric Haas asking that the binding on every
request-pattern `priority` element be relaxed from required to extensible —
which, if accepted, would let communities like Alberta carry "semi-urgent"
locally without any change to the base value set. That ticket is currently
"Waiting for Input" and Lloyd McKenzie has pushed back firmly, on the same
grounds he uses for `gender` and `status`: implementers should map to the
standard codes and convey extra nuance in an extension.

## Related Jira Tickets

### [FHIR-24923: Request adding "elective" to ServiceRequest.priority value set](https://jira.hl7.org/browse/FHIR-24923)

A 2019 Orders & Observations ticket (reporter Yanyan Hu) asking to add
"elective" to the same `RequestPriority` value set bound to
`ServiceRequest.priority`. Comments from Cerner reviewers showed
"elective" appearing alongside `routine` in HL7 v2 admission types,
NUBC/UB04, HL7 v3 ActPriority, and several EHR local value sets. The
ticket was **Resolved – Not Persuasive** in 2024, with the resolution
text: "The 'elective' value is not considered a priority and is not
about the timing of the Service Request. We will not make this change.
For the purpose of this resource, a 'routine' priority would generally
reflect the priority of a service of type/category 'elective'." This is
the closest direct precedent — the WG previously chose not to expand
the value set, but on conceptual grounds (elective is a type, not a
priority); that argument does not obviously transfer to "semi-urgent",
which *is* a timing/urgency concept.

Users involved: Yanyan Hu (reporter), Michelle Miller, Hans Buitendijk.

### [FHIR-55655: Change Binding to Extensible](https://jira.hl7.org/browse/FHIR-55655)

A FHIR Infrastructure change request (reporter/assignee Eric Haas, 2026)
asking that the binding be changed from required to extensible on every
request-pattern `priority` element — `Request.priority`,
`Communication.priority`, `MedicationRequest.priority`,
`NutritionOrder.priority`, `PlanDefinition.action.priority`,
`ServiceRequest.priority`, `Task.priority`, `ActivityDefinition.priority`,
`CommunicationRequest.priority`, `DeviceRequest.priority`,
`RequestOrchestration.priority`, `RequestOrchestration.action.priority`, and
`VisionPrescription.priority`. Eric argues that downstream profiles that
make `priority` mandatory leave implementers with more or different
priority levels with no escape hatch (citing US Core's experience with
`Location.type`). Lloyd McKenzie has pushed back, arguing that
`extensible` would not actually change the mapping requirement (mapping
to one of the four codes is still required when one applies) and that
the precedent set by `gender` and `status` is to keep these bindings
locked and use extensions for nuance. Status: **Waiting for Input**.
This ticket is highly relevant: if FIA accepts the relaxation,
"semi-urgent" can be carried as a local code without touching the base
value set, eliminating much of the need for FHIR-53434.

Users involved: Eric Haas (reporter/assignee), Lloyd McKenzie.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new code **`semi-urgent`** to `CodeSystem/request-priority` (the
HL7 FHIR `RequestPriority` code system at
`http://hl7.org/fhir/request-priority`) and, by inclusion, to the
`RequestPriority` value set (`http://hl7.org/fhir/ValueSet/request-priority`)
that is required-bound to every `priority` element derived from the
Request pattern (`ServiceRequest.priority`, `MedicationRequest.priority`,
`NutritionOrder.priority`, `Task.priority`, `CommunicationRequest.priority`,
`DeviceRequest.priority`, `RequestOrchestration.priority`,
`VisionPrescription.priority`, etc.).

Concretely:

- Add the concept with `code="semi-urgent"`, a display such as
  *"Semi-Urgent"*, and a definition placing it explicitly between
  `routine` and `urgent` (e.g., "The request should be performed
  sooner than normal scheduling but is not so urgent as to require
  expedited handling — typically aligned with intermediate clinical
  pathway timing targets").
- Update the canonical ordering text in the Request pattern
  documentation so the ladder reads `routine | semi-urgent | urgent |
  asap | stat`.
- Mark the change as introduced in the next published version
  (target R6 / Normative as Marti Velezis requested) and call it out
  in the migration / cross-version notes; existing R4/R5 instances
  are not affected.

Coordinate with FHIR Infrastructure on FHIR-55655 so that the rationale
in that ticket (extensibility) is settled before this addition is
balloted, since the two tickets propose different solutions to the same
underlying problem.

#### Justification

`semi-urgent` is unambiguously a *priority* concept (it describes the
timing/urgency of execution), so it does not run into the categorical
objection that defeated FHIR-24923 ("elective is a type, not a
priority"). The reporter has produced a published, real-world clinical
pathway (Alberta Health Services colorectal cancer referral pathways)
where exactly three urgency levels — routine, semi-urgent, urgent —
are used with defined service-time targets, and the implementer
community has surfaced the same three- to five-level scheme repeatedly
on Zulip (`ServiceRequest.priority more values`,
`Required values for ServiceRequest.priority`). Because the binding is
required, the only conformant option today is to remap `semi-urgent`
onto either `routine` or `urgent`, both of which the reporter has
explicitly rejected as semantically wrong, and Lloyd McKenzie's
"map and convey nuance in an extension" answer requires every
implementer to invent that extension independently. Adding the code
now in the R6 normative window — when value-set additions are still
possible — is the lowest-friction way to make the pattern useful for
the very communities whose pathways already speak this language.

---

### Disposition B: Alternative Approach

#### Proposal

Decline to expand the base value set, but provide a **standard HL7
extension** — for example
`http://hl7.org/fhir/StructureDefinition/request-sub-priority` — that
attaches to any element bound to `RequestPriority` and carries a
finer-grained code (initial value set: `low-routine | high-routine |
low-urgent | high-urgent`, or alternatively a code from a coarse set
including `semi-urgent`). Implementers must still populate the core
`priority` element with one of the four standard codes
(`routine|urgent|asap|stat`); the extension expresses where on the
spectrum within (or between) those codes the request actually sits.
Document the extension as the recommended path for the AHS use case
(`priority="urgent"` or `"routine"` plus
`request-sub-priority="semi-urgent"`).

#### Justification

This is the path Elliot Silver explicitly suggested in the originating
Zulip thread, and it is consistent with how the FHIR core pattern
treats other locked-down codes such as `gender` and `status`: keep
the interoperable concept set stable, push nuance into a clearly named
extension, and rely on profiles or implementation guides to require
the extension where it matters. It avoids touching a value set that
is shared across a dozen request resources (and is therefore expensive
to change and to socialize), it does not delay any AHS implementation
(an extension can be defined and used immediately, including in R4 and
R5, whereas a value-set addition only lands in the next normative
publication), and it cleanly side-steps the live FHIR Infrastructure
debate on FHIR-55655. The trade-off versus Disposition A is that
implementers who only consume the core element will see Alberta's
semi-urgent referrals as either `routine` or `urgent`, with the extra
nuance silently dropped — which is the standard FHIR cost-of-using-an-
extension story and is acceptable here because the priority element
is not mandatory in the first place.

---

### Disposition C: Decline

#### Proposal

Resolve **Not Persuasive**, mirroring the disposition of FHIR-24923.
Document that `RequestPriority` is intentionally a small, stable,
required-bound value set, that the AHS use case can be represented by
mapping `semi-urgent` to the closest existing code (`routine` or
`urgent`) per the implementer's choice, and that any additional
nuance should be expressed using a local or implementation-guide-level
extension. Refer the reporter to FHIR-55655 if they wish to pursue a
broader binding-strength change.

#### Justification

`RequestPriority` is shared across the entire Request pattern; every
addition multiplies the mapping burden on every implementer of every
request resource and increases the risk of regional value-set drift
(every region with its own pathway language could plausibly ask for
its own intermediate level — Lloyd McKenzie has made this argument
repeatedly on Zulip). FHIR-24923 already established the OO precedent
that the WG is reluctant to expand this value set even with strong
implementer support, and the conceptual rationale Lloyd reiterates
(map to the standard codes; convey nuance with extensions) holds for
`semi-urgent` just as it does for `elective`. Declining is defensible
because (a) the request can be fully satisfied with an extension
today, with no spec change, (b) a value-set addition will not help
any R4/R5 implementation, and (c) the broader binding-strength
question is being handled in FHIR-55655 by the right work group
(FIA), and that ticket is the more appropriate place to fix the
class of problem this ticket is an instance of.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested), conditional on
coordination with FIA on FHIR-55655.

The strongest argument for Disposition A is that the conceptual barrier
that defeated the closest precedent — FHIR-24923 — does not apply
here: `elective` is a service category, but `semi-urgent` is squarely
a priority/urgency concept, and the reporter has produced a published,
operational clinical pathway whose semantics genuinely require an
intermediate level. The recurring pattern on Zulip (3–5 priority
levels showing up across legacy systems, lab systems, and now
provincial referral pathways) suggests this is not a one-off; it is a
gap in the value set that implementers keep working around. Marti
Velezis has explicitly asked for this to be considered in R6 ballot
reconciliation, which is the right window to make the addition;
deferring (Disposition C) wastes that window. Disposition B (an
extension) is the safer and more reversible play, but it forces the
nuance the reporter cares about into an optional extension that
downstream consumers will largely ignore, which is a poor outcome for
exactly the clinical-pathway-driven implementations the WG is
nominally trying to support.

The conditional is important: if FIA accepts FHIR-55655 and relaxes
the binding to `extensible`, Disposition A becomes optional — the AHS
use case can be served by a local code system without any change to
the FHIR base. The OO recommendation should be made jointly with FIA
to avoid a situation where both fixes land (which would be coherent
but redundant) or neither lands (which would leave the reporter with
no path forward).
