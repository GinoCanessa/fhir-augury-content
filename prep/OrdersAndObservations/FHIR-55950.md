## Summary

Isaac Vetter, on behalf of the Imaging Integration (InM) FHIRcast working
group, requests that `DiagnosticReport.comparison` (newly added in R6,
currently typed as `Reference(List)`) be restructured. The reporters argue
that referencing a generic `List` resource is too abstract: it adds a layer
of indirection that increases optionality and reduces interoperability,
particularly given that `.comparison` carries more specific semantics than
the looser `.supportingInfo` element. They propose that `.comparison` be
restructured as a BackboneElement (mirroring the pattern used for
`.supportingInfo`) containing a mandatory relationship-type code (bound to
a new value set enumerating comparison types) and a typed reference
restricted to the resources actually used for comparison —
`Observation`, `ImagingStudy`, `Procedure`, and `DiagnosticReport`, with
the possibility of a few others. Example comparison types brainstormed in
the FHIRcast call include comparison of discrete findings via Observation
(e.g., WBC trend for a TB patient, tumor dimension/volume) and prior
studies referenced by the clinician while authoring the report (via
ImagingStudy or DiagnosticReport). The change is targeted at R6 structural
work in time for normative balloting of DiagnosticReport. No
cross-referenced Jira tickets were found.

## Details

**Description:**

The data model of the new DxReport.comparison Reference(List) is too
abstract. The additional layer of indirection increases complexity and
optionality and therefore reduces interop. The .comparison element is
more specific than .supportingInfo, and should be more tightly restrained
in it's references.

We understand and appreciate the goal of maximizing flexibility for
future IG authors towards the goal of taking this resource normative this
year.

Resulting from discussion in the InM FHIRcast WG calls, we ask that the
data type of .comparison be modified to mirror that of .supportingInfo.
It should be a Backbone element, with a mandatory relationship type, and
a reference to definitely Observation, ImagingStudy, Procedure and
DiagnosticReport ... perhaps a few others.

In the FHIRcast call we brainstormed some types of relationships that
should go into this newly proposed valueset –

Types of comparisons:

- Comparison of discrete findings via Observation (e.g. white blood cell
  count for tb patient, dimensions/volume of tumor size)
- Prior studies referenced by the clinician while creating the report
  (via ImagingStudy, or DiagnosticReport)
- ...

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Restructure `DiagnosticReport.comparison` in R6 from
`0..* Reference(List)` to a `0..* BackboneElement` with the following
structure (mirroring `DiagnosticReport.supportingInfo`):

- `DiagnosticReport.comparison` : BackboneElement (0..*)
  - `.type` : CodeableConcept (1..1) — bound (extensible or required, see
    below) to a new value set
    `http://hl7.org/fhir/ValueSet/diagnostic-report-comparison-type`
    seeded with codes for at least:
    - `discrete-finding-comparison` (e.g., trended Observations such as
      WBC, tumor dimension/volume)
    - `prior-study` (a prior imaging study referenced while authoring the
      report)
    - `prior-report` (a prior DiagnosticReport referenced for comparison)
    - `prior-procedure` (a prior Procedure referenced for comparison)
  - `.reference` : Reference(Observation | ImagingStudy | Procedure |
    DiagnosticReport [| others as agreed by OO/InM]) (1..1)

Retire the `Reference(List)` typing. Provide a migration note in the
R5 → R6 changes page explaining the restructure, and add invariants if
needed (e.g., `.type` SHALL be present when `.reference` is present —
trivially true given 1..1 cardinality). Bind the value set as `extensible`
to allow IGs (including the FHIRcast IG) to add jurisdiction- or
domain-specific comparison types while preserving interop on the common
cases.

#### Justification

The request is well-grounded in the design pattern already established by
`DiagnosticReport.supportingInfo` in R5/R6, which pairs a typing
CodeableConcept with a typed Reference. Carrying the relationship type
inline removes the layer of indirection introduced by `List`, which
otherwise forces consumers to dereference a `List` resource just to learn
why the referenced item is being compared. Restricting the target types
to the resources that actually represent comparable clinical artifacts
(`Observation`, `ImagingStudy`, `Procedure`, `DiagnosticReport`) reduces
optionality and improves interop, which is particularly important as
DiagnosticReport is targeted for normative status. The change is
structural and must land in R6 before normative lock; deferring it would
require either holding the resource back from normative or accepting a
problematic abstraction in the normative core. The InM FHIRcast WG, the
primary consumer of comparison semantics in imaging workflows,
explicitly requested this shape.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `DiagnosticReport.comparison` as a typed reference rather than a
BackboneElement, but tighten it as follows:

- Change the type from `Reference(List)` to
  `Reference(Observation | ImagingStudy | Procedure | DiagnosticReport)`
  directly (no inline relationship type, no BackboneElement).
- Define a standard extension
  `http://hl7.org/fhir/StructureDefinition/diagnosticReport-comparisonType`
  (CodeableConcept, bound extensible to the same proposed value set) that
  may be applied to each `.comparison` reference when senders need to
  declare the comparison type explicitly.
- Where senders need to bundle multiple comparators under a single
  semantic grouping, they can still construct a `List` resource and point
  to it via the existing `.supportingInfo` (or via repeated `.comparison`
  entries each carrying the extension).

#### Justification

This approach satisfies the core complaint — the `Reference(List)`
indirection — without introducing a new BackboneElement at the structural
level of a soon-to-be-normative resource. BackboneElements are harder to
profile, harder to query (no first-class search parameter on the inner
type code without additional work), and add downstream burden for
implementers who never need the relationship-type semantics. Modeling the
relationship type as a standard extension defers the question of which
codes are universally applicable (the brainstormed value set is
acknowledged as a starting point, not a finished list), and lets the
FHIRcast IG and other domain IGs evolve the binding without reopening the
core resource. The cost is that the relationship type is no longer
mandatory; for jurisdictions that need it mandatory, a profile can
require the extension.

---

### Disposition C: Decline

#### Proposal

Leave `DiagnosticReport.comparison` as `0..* Reference(List)` for R6.
Document in the resource notes that the referenced `List` is expected to
collect prior Observations, ImagingStudies, Procedures, or
DiagnosticReports used as comparators, and that `List.code` may carry the
comparison type. Defer any structural change to R7.

#### Justification

`Reference(List)` was a deliberate design choice during R6 work to
maximize flexibility while DiagnosticReport moves toward normative. The
existing `.supportingInfo` pattern (BackboneElement with type +
reference) is available when senders need explicit relationship typing,
and `.comparison` was intentionally left looser to avoid prejudging the
value set. The brainstormed comparison-type codes in the ticket are
explicitly incomplete ("…"), which suggests the value set is not mature
enough to bake into a normative structural element. Restructuring a
soon-to-be-normative resource on the basis of a single WG call's
preference, without a vetted value set or implementation experience,
risks locking in a model that does not generalize. R7 can revisit with
implementation evidence.

---

### Recommendation

**Recommended disposition:** A

The reporters' core objection is sound: `Reference(List)` forces every
consumer to dereference a `List` resource just to learn why an item is
being compared, which is a real interop cost on a resource heading for
normative status. The proposed shape mirrors `DiagnosticReport.supportingInfo`,
which is already familiar to implementers and already passes the same
"type + typed reference" pattern test the workgroup has applied
elsewhere. Disposition B (extension on a tightened reference) is
defensible and should be discussed, but it leaves the relationship type
optional and structurally invisible, which undermines the very interop
argument the reporters are making. Disposition C is the riskiest path:
locking `Reference(List)` into normative would be hard to undo in R7 and
would leave the FHIRcast community — the primary articulated user of
this element — without the structure they have asked for. Binding the
new value set as **extensible** (rather than required) preserves the
flexibility the original R6 design was trying to protect, while giving
core consumers a predictable shape to code against. The workgroup should
confirm the target reference list (`Observation | ImagingStudy |
Procedure | DiagnosticReport`, plus any additions) and the initial value
set contents with InM/FHIRcast before merge.
