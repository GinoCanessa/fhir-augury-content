## Summary

The ticket reports a broken / out-of-date hyperlink in the Order Catalog
(FHIR) IG narrative: a sentence describing how the catalog references
its items via `Composition.section.entry` linked to
`http://hl7.org/fhir/composition-catalog.html` ("Catalog profile" in
the base spec), which the reporter says should instead point at
`http://hl7.org/fhir/catalog.html`. While preparing to fix the link,
the work group discovered that the underlying "Catalog" profile on
`Composition` had been renamed to `ProfileForCatalog` in R5 base and
was planned by SDWG for removal in R6. OO accordingly agreed (per the
2024-10-10 OO Main and 2024-10-23 FMG minutes) to move the profile
into the Order Catalog IG itself and re-target every link from the
base-spec URL to the IG-local profile page. The ticket was reopened as
a Change Request with a Persuasive-with-Modification proposal so the
broader rework could be voted in. There are no explicitly linked Jira
tickets; the cross-referenced FHIR artifact is `Composition.section.entry`,
and the ticket is corroborated by an OO Zulip thread
("[fmg] status of ProfileForCatalog") that confirms the IG-side commit
has landed and asks SDWG to deprecate / remove the now-orphaned
`ProfileForCatalog` from core FHIR.

## Details

**Description:**

Fix the link in the following item:

1. The catalog references its items from its
   `Composition.section.entry` elements. This method constrains the
   Composition resource with the
   [Catalog profile](http://hl7.org/fhir/composition-catalog.html),
   which is part of the base standard.

Correct URL should be: <http://hl7.org/fhir/catalog.html>

**Resolution Description:**

Update the links to the previous "Catalog" profile to refer to the new
"Profile for Catalog" page which has now been moved to the IG — the
updated links are in the IG home page (section 1.7) and the
`labservices`, `devices` and `drugs` pages.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir-order-catalog`, edit the narrative paragraph in
`input/pagecontent/index.xml` (§1.7) and any matching paragraphs in
`input/pagecontent/labservices.xml`, `input/pagecontent/drugs.xml`,
and `input/pagecontent/devices.md` to replace the existing
`http://hl7.org/fhir/composition-catalog.html` anchor with the URL
the reporter supplied: `http://hl7.org/fhir/catalog.html`. Make no
other changes — leave the `Catalog` profile in core FHIR alone, do
not move it into this IG, and do not touch
`input/resources/Catalog.StructureDefinition.xml` or
`CatalogHeader.StructureDefinition.xml`. Resolve the ticket as
Persuasive against the originally-requested URL.

#### Justification

This is the most literal reading of the ticket: the reporter named a
specific bad link and a specific replacement, and the change is
trivial to apply. It is consistent with how OO has handled past
narrative-only fixes, keeps the IG's authoring footprint minimal, and
avoids dragging SDWG and a profile relocation into what was filed as
a doc fix.

---

### Disposition B: Alternative Approach (recommended by OO and already pre-applied)

#### Proposal

Adopt the resolution OO actually agreed in the 2024-10-10 OO Main and
2024-10-23 FMG meetings, and which has already been pre-applied on
`HL7/fhir-order-catalog`:

1. **Move the profile into the IG.** Maintain the
   `Catalog` Composition profile inside the Order Catalog IG via
   `input/resources/Catalog.StructureDefinition.xml` (commit
   `be5ee20` aligns the profile name and filename for consistency).
2. **Tighten the profile's extension/constraint set.** Switch the
   `Catalog` profile to use the generic `ValidityPeriod` extension
   instead of `cqm-ValidityPeriod`, and mark it optional + `mustSupport`
   for parity with `CatalogHeader`; drop further unneeded constraints
   (commit `bca0f32`).
3. **Re-target every link.** In `input/pagecontent/index.xml` (§1.7),
   `labservices.xml`, `drugs.xml`, and `devices.md`, replace
   `http://hl7.org/fhir/composition-catalog.html` (and any
   `ProfileForCatalog` URL pointing at the base spec) with the
   IG-local URL `StructureDefinition-Catalog.html`, e.g.
   `https://build.fhir.org/ig/HL7/fhir-order-catalog/StructureDefinition-Catalog.html`
   (commit `bada546`).
4. **Hand off to SDWG.** Forward OO's confirmation (already done on
   the FMG Zulip thread by Rob Hausam, 2025-02-25) for SDWG to
   deprecate / remove the now-orphaned `ProfileForCatalog` from base
   FHIR `Composition` in R6.

Resolve the ticket as Persuasive with Modification.

#### Justification

This matches the Persuasive-with-Modification resolution already on
the ticket and the substance of all three commits referenced in the
comment trail. It also addresses the underlying problem the original
URL fix would have masked: the base-spec profile is on the way out, so
pointing the IG narrative at `http://hl7.org/fhir/catalog.html` only
defers the breakage. Co-locating the profile with the IG that
references it is consistent with FHIR's general "owners maintain their
profiles" pattern, brings `Catalog` into structural alignment with
`CatalogHeader` (same `ValidityPeriod` extension, same must-support
posture), and keeps the IG's links self-contained against future SDWG
moves. The repo-analysis briefing's gotchas — XML/FSH split, no
top-level `sushi-config.yaml`, link-target dependence on the
StructureDefinition id/filename — are explicitly handled by the
pre-applied commits (XML-only edits, name/filename alignment, IG-XML
page wiring left untouched).

---

### Disposition C: Decline

#### Proposal

Persuasive-Not-Persuasive: decline both the literal link swap and the
broader profile relocation in this ticket. Close FHIR-41369 with a
note that any URL change to base-spec `composition-catalog.html` /
`catalog.html` should ride on whatever SDWG decides for the future of
`ProfileForCatalog` (rename, deprecation, or removal) under a
separate cross-WG ticket against `HL7/fhir`, and that no Order Catalog
IG narrative changes will be made in the meantime.

#### Justification

A defensible "do nothing" rationale would be: the ticket as filed
fixes a symptom of a larger SDWG/OO governance question that hasn't
been formally resolved through ballot, and patching links in the IG
risks committing OO to a profile-relocation it hasn't formally voted.
Waiting for SDWG's R6 decision keeps the IG's link targets stable
against an upstream that hasn't moved yet.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the only option that matches what has actually
happened on the ticket, on the OO/FMG meeting record, and in the
repo. The Persuasive-with-Modification resolution is already set,
the three pre-applied commits on `HL7/fhir-order-catalog`
(`bada546`, `be5ee20`, `bca0f32`) implement exactly the four steps
above, and the FMG Zulip thread confirms the IG CI build now renders
the IG-local Catalog profile correctly and that OO has formally
handed the deprecation step over to SDWG. Disposition A "fixes" the
URL to a target that base FHIR is itself about to retire, which would
just generate the same ticket again next cycle. Disposition C
ignores the work the WG has already done and the consensus already
reached in OO Main and FMG. The workgroup should ratify Disposition B
and close FHIR-41369; a separate companion ticket (or SDWG-side
trackers) should carry the core-FHIR removal of `ProfileForCatalog`.
