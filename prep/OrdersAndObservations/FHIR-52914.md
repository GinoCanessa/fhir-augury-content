## Summary

The reporter wants to use `ServiceRequest` to represent Advance Directive
instructions (e.g., CPR, blood transfusion). For Dutch emergency-care use
cases, the Advance Directive must surface the *agreement parties* who
signed off on the directive: a required `HealthProfessional`
(PractitionerRole) plus at least one of a representative/contact person
(RelatedPerson, possibly several) or the patient themselves. The reporter
proposes adding a new element — provisionally named `agreementParties` —
to `ServiceRequest` to carry these references. They note that the
underlying Living Will / POLST-style document and other agreement content
will be captured in a `Consent` resource and linked from the
`ServiceRequest` via `supportingInfo`. The only cross-reference indexed
for this ticket is the ticket's own mention of `ServiceRequest.supportingInfo`;
no related Jira tickets, Zulip threads, or GitHub items have been found.

## Details

**Description:**

We want to utilize the ServiceRequest resource for its intended purpose
(Advanced Directive), such as CPR, blood transfusion, etc.

In the Netherlands there are agreementparties required to display when
this Advance Directive will requested during emergency care cases:

Required:
- HealthProfessional (PractitionerRole)

At least one is required of this option:
- Representative::ContactPerson (ReleatedPerson) - more is possible
- Patiënt

So I'm interested for this information (references), like an
**agreementparties** element, could be added in Servicerequest.

P.S. The LivingWill document like POLST from US and other information
from the agreement will be noticed in the Consent resource and we will
use the ServiceRequest.supportingInfo to make the reference.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new repeating element on `ServiceRequest`, provisionally named
`ServiceRequest.agreementParty`, to carry the parties that have agreed
to / signed off on the requested service when the request represents an
Advance Directive (or, more broadly, any request that requires explicit
agreement of named parties).

Suggested shape:

- `ServiceRequest.agreementParty` — 0..*, BackboneElement
  - `role` — 1..1, CodeableConcept (e.g., authoring health professional,
    representative, patient, witness). Bind to a new value set
    `ServiceRequestAgreementPartyRole`, extensible.
  - `party` — 1..1, Reference(Patient | RelatedPerson | Practitioner |
    PractitionerRole | Organization).
  - Optionally `period` — 0..1, Period (when the agreement was/is in
    force).

Document in the resource notes that for Advance Directive use cases the
expectation (set by profiles such as a Dutch Advance Directive profile)
is at least one entry with role = "health professional" plus at least
one entry with role = patient or representative. The cardinality of
those constraints is left to profiles, not the base resource.

Source change would land in the FHIR Core repo (`HL7/fhir`) under the
`ServiceRequest` resource source (e.g., `source/servicerequest/`),
including the structure definition, examples, and resource narrative.

#### Justification

The use case is concrete and aligns with `ServiceRequest`'s stated scope
of representing requested services, including Advance Directives.
Capturing who agreed to the directive directly on the request — rather
than only via a separately-fetched `Consent` — preserves the
information at the point of clinical use (emergency care), where
implementers may not always traverse `supportingInfo` to a `Consent` to
discover signatories. A typed, role-tagged backbone element gives
profiles (PACIO ADI, Dutch national profiles, etc.) a single, queryable
place to constrain the parties without each project inventing its own
extension.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new core element. Instead:

1. Confirm and document the existing pattern: the agreement / Living
   Will is represented as a `Consent` resource, linked from
   `ServiceRequest.supportingInfo`. The signatories live on
   `Consent.performer`, `Consent.grantor`, `Consent.grantee`, and (where
   relevant) `Consent.verification.verifiedWith` /
   `Consent.provision.actor`, which already cover Practitioner,
   PractitionerRole, RelatedPerson, Patient, and Organization.
2. Define a standard FHIR extension (e.g.,
   `http://hl7.org/fhir/StructureDefinition/servicerequest-agreementParty`)
   on `ServiceRequest` for jurisdictions that need the parties inline on
   the request without a `Consent` round-trip. Same shape as Disposition
   A (role + party reference), but as an extension.
3. Add resource-level guidance to `ServiceRequest` cross-referencing
   `Consent` for Advance Directive scenarios, and pointing implementers
   at the new extension when an inline representation is required.

This keeps any new authoring on the existing FHIR Core source area for
`ServiceRequest` (`HL7/fhir`, `source/servicerequest/`) plus the
standard extensions area.

#### Justification

Most of the requested information is already representable through
existing FHIR mechanisms: `Consent` was designed to carry signatories,
witnesses, grantors, and grantees, and `ServiceRequest.supportingInfo`
already provides the linkage the reporter intends to use. Adding a
first-class element on `ServiceRequest` risks two-place modeling for the
same facts (signatories on both `ServiceRequest.agreementParty` and
`Consent.*`), increasing the chance of divergence between the request
and the underlying agreement document. A standard extension addresses
the emergency-care "I need it inline" concern without a breaking core
change, and lets profiles (e.g., the Dutch Advance Directive profile)
make it required where the jurisdictional rules apply.

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the base `ServiceRequest`
resource. Direct the reporter to:

- Use `Consent` (or a profile of it, e.g., a Dutch Advance Directive /
  Living Will profile, or PACIO ADI work) to capture the agreement and
  its parties.
- Link the `Consent` from `ServiceRequest.supportingInfo`, exactly as
  the reporter already plans to.
- Apply jurisdictional cardinality / participant-role constraints in a
  national or realm-specific profile rather than in core.

#### Justification

`ServiceRequest` is intentionally generic across many request types
(labs, imaging, procedures, referrals, advance directives). Adding a
new element to satisfy one workflow — Advance Directive sign-off — would
sit unused on the vast majority of `ServiceRequest` instances and would
duplicate signatory information already modeled by `Consent`. The
reporter's own design already routes the Living Will document through
`Consent` via `supportingInfo`; the parties belong with that document,
not with the order. The need is real but is a profile-level concern,
not a base-resource concern.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle. It honours the reporter's underlying
need — having the agreement parties available on or from a
`ServiceRequest` representing an Advance Directive — without committing
the base `ServiceRequest` resource to a workflow-specific element that
the majority of implementers will never populate. It avoids the
two-place-modeling problem that Disposition A would introduce
(signatories on both `ServiceRequest.agreementParty` and on
`Consent.*`), while still giving the Dutch profile (and any future
PACIO-aligned ADI profile) a stable, standard extension to constrain.
It also stays compatible with the reporter's already-stated plan to use
`ServiceRequest.supportingInfo → Consent`, so adopting it requires no
rework of their current modeling. Disposition C is defensible and may
end up as the workgroup's choice if the standard-extension overhead is
judged unwarranted, but B better acknowledges the emergency-care
"inline" concern that motivated the request.
