## Summary

The ticket asks the Orders & Observations work group to broaden the value set
bound to `ObservationDefinition.code`. Today the example binding points to the
`observation-codes-observationorboth` value set (LOINCs whose Class Type is
`Observation` or `Both`). The reporter argues that because `ObservationDefinition`
is also used to define **panels** (groupers), and panel LOINCs are typically of
class `Order` or `Both`, the current restriction prevents authoring legitimate
panel definitions. Two alternatives are floated by the reporter:

1. Drop the narrowing entirely — point the example binding at all of LOINC,
   since the binding strength is already `example`.
2. Keep two narrower value sets but introduce an invariant on
   `ObservationDefinition` that selects which LOINC subset is appropriate based
   on whether `.hasMember` is populated (panels → Order/Both; result parameters
   → Observation/Both).

The request sits squarely inside the broader O&O conversation about how panels
and result grouping should be represented. The companion ticket
[FHIR-54887](https://jira.hl7.org/browse/FHIR-54887) makes the symmetric point
on the `Observation` resource itself (allow `Order`/`Both` LOINCs in
`Observation.code` when `Observation.organizer` indicates the instance is a
panel grouper). [FHIR-54741](https://jira.hl7.org/browse/FHIR-54741) and
[FHIR-50534](https://jira.hl7.org/browse/FHIR-50534) frame the underlying
grouping/organizer model that this binding decision depends on.
[FHIR-50603](https://jira.hl7.org/browse/FHIR-50603) raises a parallel
narrowing concern on `DiagnosticReport.code`, and
[FHIR-54593](https://jira.hl7.org/browse/FHIR-54593) (recently `Applied`) shows
the work group is actively tightening up `ObservationDefinition` bindings.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Broaden the example binding on `ObservationDefinition.code` so panel
definitions are not implicitly excluded. Concretely, change the binding's
`valueSet` from
`http://hl7.org/fhir/ValueSet/observation-codes-observationorboth` to a value
set covering all of LOINC — either reuse the existing
`http://hl7.org/fhir/ValueSet/observation-codes` (all-LOINC) value set or
introduce a new `observation-codes-orderobservationorboth` value set that
unions the three LOINC class types (`Order`, `Observation`, `Both`). Keep the
binding strength at `example`. No structural change to `ObservationDefinition`
is required.

#### Justification

The binding is `example` only, so widening it does not constrain conformant
implementers. The current narrowing actively *misleads* implementers building
panel definitions — panels naturally use `Order` or `Both` LOINCs (e.g.
24323-8 Comprehensive metabolic panel), and an "Observation/Both" example
silently excludes the most common real-world panel codes. Disposition A also
mirrors the reporter's primary suggestion ("in the base I think this should
just point to all of LOINC") and is consistent with FHIR's general practice of
keeping example bindings broad on a resource intended to define both single
result parameters and groupers (see also FHIR-54887 on `Observation.code`,
which raises the same problem one layer down).

---

### Disposition B: Alternative Approach

#### Proposal

Keep two purpose-specific LOINC subsets but make the choice context-driven on
the instance, not on the resource definition:

1. Retain `observation-codes-observationorboth` as the example binding for
   "leaf" `ObservationDefinition` instances (no `hasMember`).
2. Add (or reuse) an `observation-codes-orderorboth` value set as the example
   binding for "panel" `ObservationDefinition` instances.
3. Express the choice via an invariant on `ObservationDefinition`, e.g.:
   - `odef-code-panel`: `hasMember.exists() implies code.memberOf('http://hl7.org/fhir/ValueSet/observation-codes-orderorboth')`
   - `odef-code-leaf`: `hasMember.empty() implies code.memberOf('http://hl7.org/fhir/ValueSet/observation-codes-observationorboth')`
   Use `severity = warning` so example-binding semantics are preserved.

This realises the reporter's secondary suggestion ("could we create an
invariant for the binding based on `.hasMember` being populated?").

#### Justification

Disposition B is more *informative* to implementers than Disposition A: it
preserves the educational signal that panels and leaf observables draw from
different LOINC subsets, while still allowing extension in the long tail.
However, it depends on `hasMember` (or `Observation.organizer`, see
FHIR-54741) being the agreed canonical panel marker, and it adds two new
invariants that must be propagated to derived IGs and validators.
Implementation burden is non-trivial, and the warning-severity invariants are
easy for implementers to ignore. Trade-off vs. A: more correctness signal at
the cost of more spec machinery.

---

### Disposition C: Decline

#### Proposal

Decline the change as **persuasive but not actionable in this cycle**: leave
`ObservationDefinition.code` bound to
`observation-codes-observationorboth` until the broader Observation grouping /
panel representation work in FHIR-54741, FHIR-54887, and FHIR-50534 lands.
Once the grouping model is stabilised, revisit the binding holistically across
`Observation.code`, `ObservationDefinition.code`, and `DiagnosticReport.code`
in a single coordinated change.

#### Justification

The binding question is genuinely entangled with the grouping/organizer
decision: if the WG ultimately adopts `Observation.organizer` as the panel
marker (instead of, or in addition to, `hasMember`), any invariant added now
in Disposition B would have to be reworked, and even Disposition A's choice of
target value set may want to be re-aligned with the parallel
`DiagnosticReport.code` discussion in FHIR-50603 (which argues the *opposite*
direction — narrowing rather than broadening). Declining-with-defer keeps
related bindings from drifting and avoids two near-term churn cycles on the
same value sets.

---

### Recommendation

**Recommended disposition:** A

The binding is `example`, the reporter is correct that the current value set
silently excludes panel LOINCs, and there is recent precedent (FHIR-54593,
Applied) for the WG making targeted binding fixes on `ObservationDefinition`
without waiting for the larger grouping debate to conclude. Disposition A is
small, safe, and unblocks panel authoring immediately; Disposition B can be
layered on later (or driven by FHIR-54741) once the WG decides whether
`hasMember` or `organizer` is the canonical panel marker. Disposition C
defers value cheaply gained — implementers building panel
`ObservationDefinition` examples today will keep hitting the same wall, and a
broader example binding does not foreclose the more sophisticated invariant
approach in Disposition B in a future cycle.
