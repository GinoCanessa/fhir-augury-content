## Summary

The ticket documents a SUSHI authoring problem encountered while building
the **Laboratory Report (FHIR)** IG. When the R5 cross-version extension
for `Observation.value[x]` (used to preadopt the R5 `Attachment` data
type onto the R4 `Observation.value[x]` choice) was referenced from FSH,
SUSHI v3.14.0 URL-encoded the `[` and `]` characters in the extension
URL, transforming
`http://hl7.org/fhir/5.0/StructureDefinition/extension-Observation.value[x]`
into
`http://hl7.org/fhir/5.0/StructureDefinition/extension-Observation.value%5Bx%5D`.
The IG Publisher could not resolve the percent-encoded URL, so the build
failed.

The reporter's workaround was to remove `observation-lab.fsh` from the
project temporarily and substitute a hand-authored `StructureDefinition`
JSON resource (with snapshot, since SUSHI requires it for downstream
processing) that contains the un-encoded `[x]` URL. The ticket asks for
guidance / a tracking record so that, once SUSHI is fixed, the project
can revert to the FSH source.

The cross-references attached to the ticket are mentions of the FHIR
artifacts `Observation.value` and `Observation.value[x]`. There are no
linked Jira tickets, no GitHub PRs/issues, and no Zulip threads attached
via cross-reference, but related Zulip discussion exists (see below)
and the only Jira comment (October 2025) reports that the underlying
SUSHI bracket-escaping bug was already fixed in SUSHI v3.16.5.

## Details

**Description:**

> When including the R5 extension for Observation.value[x] (which adds
> the Attachment data type), SUSHI (the latest v3.14.0 version) escapes
> the `[]` bracket characters in the url, transforming the url from
> `http://hl7.org/fhir/5.0/StructureDefinition/extension-Observation.value[x]`
> to
> `http://hl7.org/fhir/5.0/StructureDefinition/extension-Observation.value%5Bx%5D`,
> which the IG Publisher is unable to process, generating a build
> error. A workaround is to temporarily remove the `observation-lab.fsh`
> file and substitute a `StructureDefinition` resource for the profile
> in JSON (including the snapshot which is required by SUSHI), where
> the extension url that contains the `[x]` fragment is able to be
> processed correctly. Once this issue has been addressed in Sushi,
> this temporary fix can be reverted, going back to using
> `observation-lab.fsh` as the source file.

**Comments:**

- **JIRAUSER25848 (2025-10-13):** "It seems that this problem was already
  resolved in current version of SUSHI (v 3.16.5)."

## Related Zulip Discussions

### [shorthand > SUSHI 3.19.1](https://chat.fhir.org/#narrow/stream/shorthand/topic/SUSHI%203.19.1)

A long thread about the SUSHI 3.19.0/3.19.1 release. Giorgio Cangioli
hit two issues building the HL7 Europe `base` IG (an R4 IG that
preadopts R5 elements): (1) a misleading error suggesting the
`hl7.fhir.uv.xver-r5.r5` package, and (2) a warning
`obs-value-1: Reference to an unknown extension - double check that the
URL '...extension-Observation.value[x]' is correct`. Chris Moesel
explains that SUSHI 3.19 has switched to the **official cross-version
(xver) packages** (`hl7.fhir.uv.xver-r5.r4`) and replaces the legacy
`hl7.fhir.extensions.r5`. In the official packages the canonical URL
for the R5 cross-version extension on a choice element **omits the
`[x]` suffix** (it is `.../extension-Observation.value`, not
`.../extension-Observation.value[x]`). SUSHI now warns "Cross-version
extensions for choice elements should omit the `[x]` suffix" and uses
the corrected URL automatically, but advises authors to fix their FSH
source. Patrick Werner confirmed the warning. Issue (1) was traced to
mixing R4 and R5 SUSHI logs across a multi-version build and was
operator error.

Participants: Chris Moesel, Giorgio Cangioli, Patrick Werner,
Notification Bot.

### [IG creation > Preadoption of R5 Obs.value[x] element](https://chat.fhir.org/#narrow/stream/IG%20creation/topic/Preadoption%20of%20R5%20Obs.value%5Bx%5D%20element)

February 2026 thread on the same underlying scenario as FHIR-49664.
Giorgio Cangioli asks whether `hl7.fhir.uv.xver-r5.r4: 0.0.1-snapshot-2`
is sufficient for preadopting the R5 `Observation.value[x]` extension
in an R4 IG; SUSHI demanded `hl7.fhir.extensions.r5: 4.0.1` and even
then validation errors appeared whenever an extension was placed on
those Observations. Lloyd McKenzie pinged Gino Canessa, who confirmed
the next xver release already fixes it. Kevin Mayfield shared a
working FSH pattern that places the cross-version extension *under*
`value[x].extension` (with `value[x] only Attachment` and the
extension URL **without** the `[x]` suffix) and pointed to issue
[hl7-eu/base#30](https://github.com/hl7-eu/base/issues/30). Lloyd
noted you can't add a new type by nesting the extension under `value`
without committing to a type — confirming that the extension is
applied *at the choice element* and the URL must use the published
canonical (no `[x]`).

Participants: Giorgio Cangioli, Lloyd McKenzie, Kevin Mayfield, Gino
Canessa.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat FHIR-49664 as a tracking ticket against the **Laboratory Report
(FHIR)** IG source: keep the temporary JSON `StructureDefinition`
substitution that replaces `observation-lab.fsh` until the IG team
verifies that the SUSHI bracket-escaping bug remains fixed on a current
SUSHI release, then revert the substitution and restore the FSH source
file as the authoring artifact. Resolve the ticket as **Persuasive
with Modification** with a comment that points the IG editors at the
revert step. No spec / model change is required because the root cause
was a SUSHI tooling bug and that bug is reported fixed in SUSHI ≥
3.16.5.

#### Justification

The reporter explicitly framed FHIR-49664 as a workaround placeholder
rather than a request for a normative change ("Once this issue has
been addressed in Sushi, this temporary fix can be reverted"). The
sole follow-up comment confirms SUSHI v3.16.5 already resolves the
bracket-escaping. The Laboratory Report IG retains its intended
authoring approach (FSH), the workaround is unwound, and the ticket is
closed honestly against what actually happened.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve as **Persuasive with Modification**, but in addition to
restoring `observation-lab.fsh` from the JSON workaround, **update the
FSH source to match the current official xver convention**: reference
the cross-version extension as
`http://hl7.org/fhir/5.0/StructureDefinition/extension-Observation.value`
(no `[x]` suffix), depend on `hl7.fhir.uv.xver-r5.r4` rather than the
deprecated `hl7.fhir.extensions.r5`, and apply the extension at
`value[x].extension` per Kevin Mayfield's pattern in the IG-creation
Zulip thread. The disposition note should record this as the
forward-compatible authoring style for any R4 IG preadopting the R5
`Observation.value[x]` Attachment type.

#### Justification

Disposition A only restores the pre-bug status quo. It leaves the
FSH source using the legacy `[x]`-suffixed URL and the deprecated
`hl7.fhir.extensions.r5` dependency, both of which now produce SUSHI
warnings on 3.19+ ("Cross-version extensions for choice elements
should omit the `[x]` suffix" and "Found old-style cross-version
extensions package in dependencies …"). Folding those updates into
the same revert keeps the Laboratory Report IG building cleanly on
current SUSHI/xver tooling and prevents a near-future re-open of an
equivalent ticket. Lloyd McKenzie's and Gino Canessa's confirmations
on Zulip make it clear that the canonical extension URL no longer
carries `[x]`, so authoring against the new form is the correct
target shape.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that it does not
identify a defect or change in the FHIR specification or the Laboratory
Report IG content — it documents a transient SUSHI tooling defect and
a project-local workaround. Direct the reporter to file (or confirm)
a SUSHI issue in the [FHIR/sushi](https://github.com/FHIR/sushi)
repository if regression testing finds the bracket-escaping behavior
again, and otherwise take no action.

#### Justification

The Jira project tracks ballot/specification correction work, not IG
project-internal authoring patches. Because the root cause is a
tooling bug already fixed upstream and there is no normative or IG
content change to make, the ticket has no actionable disposition for
the workgroup. Declining keeps the Orders & Observations backlog
focused on substantive specification work.

---

### Recommendation

**Recommended disposition:** B

The ticket cannot just be left dormant — the workaround it documents
(a hand-authored JSON `StructureDefinition` standing in for
`observation-lab.fsh`) is still sitting in the Laboratory Report IG
source tree and will become harder to unwind the longer it stays.
Disposition A unwinds the workaround, but leaves the FSH source in a
shape that already produces noisy SUSHI warnings on 3.19+ and uses
the deprecated `hl7.fhir.extensions.r5` dependency. Disposition C
ignores the cleanup work entirely. Disposition B closes the
specification-side ticket in a way that matches what actually
happened (SUSHI bug fixed upstream, FHIR-49664 was a tracking
placeholder) **and** packages the small set of authoring changes
needed to land cleanly on the current xver tooling — exactly what the
two related Zulip threads (`shorthand > SUSHI 3.19.1` and
`IG creation > Preadoption of R5 Obs.value[x] element`) describe as
the correct pattern. That gives the IG editors a single, concrete
follow-up rather than a vague "revert when ready" note.
