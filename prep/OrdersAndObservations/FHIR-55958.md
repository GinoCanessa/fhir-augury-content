## Summary

This Change Request asks the Orders & Observations work group to add one
or more **example workflows** to the Clinical Order Workflow (COW) IG
that show how to capture **verbal orders** — i.e., orders that are
issued without prior electronic documentation and are written into the
system either before or after fulfillment has begun. The reporter, Jose
Costa-Teixeira, frames the ticket as a follow-up to
[FHIR-34207](https://jira.hl7.org/browse/FHIR-34207) ("Add attribute to
document confirmation"), which was resolved **Persuasive with
Modification** and Applied via
[HL7/fhir-extensions PR #229](https://github.com/HL7/fhir-extensions/pull/229),
adding a `verbalOrderConfirmation`-style extension on `ServiceRequest`.
With the data structure now in place, the IG lacks normative-side
guidance on **how** implementers should actually use it end-to-end.

The reporter suggests the examples cover:

- Documentation **before** fulfillment has started vs. **after**
  fulfillment is already underway (a posteriori documentation).
- **Lab, pharmacy, and other domains** (since the dynamics differ —
  e.g., lab fillers may legally not be allowed to author the placer
  order, while pharmacy verbal scripts have their own conventions).
- Whether a **filler-order** instance needs to be created prior to
  receipt of the placer `ServiceRequest`/`MedicationRequest`, or whether
  the existing placer pattern is sufficient with backdated `authoredOn`.

The reporter is unsure whether this is a brand-new workflow pattern or
whether it should be folded into the existing **Order Initiation**
section of the COW IG.

## Linked Jira Tickets

### [FHIR-34207: Add attribute to document confirmation](https://jira.hl7.org/browse/FHIR-34207)

The structural predecessor to this ticket. FHIR-34207 was filed against
the FHIR Extension Pack and resolved **Persuasive with Modification**;
its disposition added an extension on `ServiceRequest` to document that
a request confirms a verbal order that may already have been fulfilled.
After extensive debate (see comment thread below), the agreed direction
was to capture verbal-order context using **general-purpose extensions**
(`informationSource`, `recorder`, `event-recorded`, and a possible new
`informationMode` extension), placing them on the **filler-side
Request/Task** rather than the placer's `ServiceRequest`. The PR
implementing this lives at
[HL7/fhir-extensions#229](https://github.com/HL7/fhir-extensions/pull/229).
FHIR-55958 is a direct dependency on this resolution: it requests COW
IG examples that show the agreed pattern in practice.

Hans Buitendijk (reporter), Lloyd McKenzie, Andrea Pitkus, Daniel Rutz,
Marti Velezis, Floyd Eisenberg, Joanie Harper.

## Related Zulip Discussions

### [implementers > Verbal Orders](https://chat.fhir.org/#narrow/stream/implementers/topic/Verbal%20Orders)

A long-running thread (initial messages 2022-03; revived 2026-01) where
implementers ask how to represent verbal orders in FHIR. Floyd
Eisenberg opened it asking whether a verbal order should be modeled as
`MedicationRequest.intent = plan` until signed (then `order`), or
whether `authoredOn` is sufficient. Lloyd McKenzie's position is
consistent with the FHIR-34207 resolution: `authoredOn` is "when the
order was given" (not when it was entered), so the only structural
difference between a verbal order and a standard order is that
`authoredOn` will be earlier than the record creation date, plus
optional metadata about who took the verbal order and when. Josh Mandel
noted that without explicit guidance, implementations will likely
diverge — exactly the gap FHIR-55958 is asking the IG to close.
Joanie Harper (2026-01-14) explicitly asked whether consensus had been
reached on the best representation for both `ServiceRequest` (non-med)
and `MedicationRequest` (med) verbal orders, and Iryna Roy (2026-01-16)
restated the workflow gap: a verbal order is unsigned, sits in the
physician's queue to sign, and implementers want to know whether to
model that with status, a flag, or an extension. The thread shows
**implementer demand for exactly the IG examples this ticket is
requesting**, but does not by itself converge on a canonical approach.

Lloyd McKenzie, Floyd Eisenberg, Josh Mandel, Joanie Harper,
Iryna Roy.

### [workflow > Event-before-Request](https://chat.fhir.org/#narrow/stream/workflow/topic/Event-before-Request)

Tangential thread (2025-01) where Brian Swinkels and Lloyd McKenzie

pattern. Lloyd notes that flagging a request as having originated
verbally helps cue downstream systems that a written order is still
needed. Useful background for the IG section that would document the

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new **"Verbal / Undocumented Orders"** worked-example subsection
to the **Order Initiation** page of the Clinical Order Workflow (COW)
IG (specification key: Clinical Order Workflow (FHIR)). The subsection
should contain **at least three example bundles**, each rendered as
narrative + an example resource bundle:

1. **Lab — verbal order received before any electronic record exists.**
   Filler creates a `ServiceRequest` with `intent = filler-order`,
   `status = active` (or `completed` if already resulted), populates
   `authoredOn` with the time the order was *given verbally*,
   `informationSource` = the clinician who gave the verbal order,
   `recorder` = the lab person who entered it, and `event-recorded`
   = the time of entry. `basedOn` is left empty until a placer
   `ServiceRequest` arrives, at which point the filler updates
   `basedOn` to point at it. Demonstrate the optional
   `verbalOrderConfirmation` extension from
   [HL7/fhir-extensions#229](https://github.com/HL7/fhir-extensions/pull/229).
2. **Pharmacy — verbal medication order from prescriber to pharmacist.**
   `MedicationRequest` with `authoredOn` = time of the verbal call,
   `informationSource` = prescriber, `recorder` = pharmacist,
   `intent = order`, plus the verbal-order-confirmation extension. Show
   the later "co-sign" via a follow-up `Provenance` rather than a status
   change.
3. **Generic / non-lab non-pharmacy — a posteriori documentation after
   fulfillment is already underway.** `ServiceRequest` created after
   the act, with `status = completed`, `authoredOn` backdated, and a
   `Task` whose `status = completed` and whose `Task.input` carries
   the "informally communicated" timestamp + identity, per Lloyd's
   suggestion on FHIR-34207.

Add a short normative narrative *above* the examples that states the
COW IG's recommended pattern: place verbal-order metadata on the
**filler-side Request and/or Task**, not on the placer's
`ServiceRequest`; backdate `authoredOn` rather than introducing a new
status; use general-purpose `informationSource`/`recorder`/`event-recorded`/`verbalOrderConfirmation`
extensions rather than IG-local ones.

The change targets the COW IG repo (likely `HL7/cow` or the
work-group-managed equivalent — the briefing has not been loaded
because no GitHub cross-reference surfaced; confirm at implementation
time and run `repo-analysis` first).

#### Justification

This is exactly what the ticket asks for and exactly what implementers
in the [implementers > Verbal Orders](https://chat.fhir.org/#narrow/stream/implementers/topic/Verbal%20Orders)
thread are asking for (Joanie Harper 2026-01-14, Iryna Roy 2026-01-16).
FHIR-34207's resolution gave the community the structural building
blocks; without worked examples in the COW IG, implementers will
continue to diverge — Josh Mandel explicitly flagged this risk on the
Zulip thread. The proposal is consistent with Lloyd McKenzie's repeated
guidance on FHIR-34207 (verbal-order metadata belongs on the filler
side; reuse general-purpose extensions instead of inventing
status-based mechanisms) and with the "Clinical Order Workflow" IG's
remit to document end-to-end order patterns. Three examples cover the
domains the reporter explicitly called out (lab, pharmacy, other) and
the two timing variants (before vs. after fulfillment).

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a single COW-IG worked-example subsection covering
all domains, **scope the COW IG addition to the cross-domain pattern
only** — one canonical lab example plus one generic "a posteriori
documentation" example — and **defer pharmacy verbal-order examples to
the Pharmacy IG / MedicationRequest documentation** owned by the
Pharmacy WG. Coordinate with the Pharmacy WG to file a sibling ticket
against the appropriate Pharmacy IG (or against `MedicationRequest`
itself) so each WG documents its own resource. The COW IG would link
to those pages from the new "Verbal / Undocumented Orders" subsection.

Concretely: add the same Order-Initiation subsection as in
Disposition A, but with examples (1) lab and (3) generic-non-pharmacy,
plus a sentence that says "for medication verbal orders, see
[Pharmacy IG / MedicationRequest guidance]". File a new Jira against
the Pharmacy WG to produce the parallel pharmacy example.

#### Justification

The verbal-order workflow has materially different regulatory and
authorial constraints in pharmacy vs. lab vs. generic clinical settings
— Andrea Pitkus's comments on FHIR-34207 make this point in detail
(CMS, CLIA §493.1241, ISO 15189:2022 §7.2.3.2 all impose
domain-specific rules). The Pharmacy WG owns `MedicationRequest`
guidance; putting a normative pharmacy example in the COW IG risks
either drift or contradiction with future Pharmacy-WG guidance, and
forces COW IG editors to track pharmacy regulation changes outside
their expertise. Splitting the work is a smaller, more maintainable
change and respects HL7 work-group boundaries. The downside vs.
Disposition A is that implementers won't find all three examples in
one place on day one — but the COW IG can link to the eventual
pharmacy material.

---

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope for the Clinical Order Workflow
IG at this time**, on the grounds that:

1. The structural mechanism for verbal orders already exists
   (FHIR-34207's extensions and `authoredOn` semantics).
2. The Zulip thread shows that even with a worked example,
   implementations will continue to vary by jurisdiction (CMS, ISO,
   CLIA, BC pharmacy rules, Australia's two-nurse confirmation rule)
   and the IG cannot enumerate all of them.
3. The reporter himself notes ("Not sure if this would be a new
   pattern, but perhaps it fits better into order initiation") that
   this may not yet be a settled pattern worthy of normative IG text.

Recommend instead that the WG capture the Lloyd-McKenzie / Hans-Buitendijk
guidance as a short FAQ-style note on the existing Order Initiation
page (no new examples, no new bundles) and close this ticket.

#### Justification

Defensible if the WG judges that adding multi-domain example bundles
risks codifying a pattern that hasn't been validated by implementer
balloting. FHIR examples are sticky — once they ship in an IG, they
become de-facto reference implementations. The FHIR-34207 thread
itself shows unresolved disagreement on filler-vs-placer placement and
on jurisdiction-specific signing requirements; baking that into IG
examples could create downstream rework when those questions are
revisited. Declining keeps the IG silent until the community converges,
which is a conservative-but-defensible posture.

---

### Recommendation

**Recommended disposition:** A (Accept As Requested)

The structural work was completed by FHIR-34207 explicitly so that this
kind of guidance could be written. Implementers are actively asking for
it on Zulip in 2026 — the request is not theoretical. Disposition C
under-serves implementers who will otherwise diverge (Josh Mandel's
exact concern), and Disposition B's split-across-WGs path adds
coordination cost and elapsed time for what is, structurally, the same
pattern in three domains. Disposition A is the option that converts
FHIR-34207's resolution into something implementers can actually use,
and it can be scoped tightly: one Order-Initiation subsection, three
worked examples, no new mechanisms invented. The jurisdictional
nuances Andrea Pitkus raised on FHIR-34207 (CMS, CLIA, ISO 15189) can
be acknowledged in the narrative as "implementations MUST also satisfy
applicable regulatory requirements which may vary by jurisdiction"
without the IG having to enumerate them. If reviewers want to hedge,
mark the new examples explicitly as **non-normative** and label the
narrative as guidance rather than rules — that preserves Disposition
A's value while leaving room for future refinement.
