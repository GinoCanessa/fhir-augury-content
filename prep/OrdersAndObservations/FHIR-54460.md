## Summary

Eric Haas proposes that the FHIR Core `Observation` resource should treat
the presence of `Observation.interpretation` without an
`Observation.value[x]` as clinically illogical and enforce it via an
invariant. The requested constraint is roughly
`interpretation.exists() implies value.exists()`, with an analogous
constraint applied to `Observation.component.interpretation` /
`Observation.component.value[x]`. The argument is semantic: an
"interpretation" (e.g., High, Low, Abnormal) is fundamentally a
classification *of* a measured value, so without a value to classify the
interpretation has no anchor.

No cross-referenced Jira tickets, Zulip discussions, or GitHub items
were returned by the cross-reference index for this key, so the request
appears to be a fresh design proposal rather than a follow-on to prior
community work.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add two new invariants to the `Observation` resource in FHIR Core:

- On `Observation` (root), add e.g. `obs-X`:
  - **Severity:** `error`
  - **Human:** "If Observation.interpretation is present, Observation.value[x] (or dataAbsentReason) SHALL also be present."
  - **Expression:** `interpretation.exists() implies (value.exists() or dataAbsentReason.exists())`
- On `Observation.component`, add e.g. `obs-Y`:
  - **Severity:** `error`
  - **Human:** "If a component has an interpretation, the component SHALL also have a value[x] (or dataAbsentReason)."
  - **Expression:** `interpretation.exists() implies (value.exists() or dataAbsentReason.exists())`

Wire the new invariants into the Observation StructureDefinition (the
authoring root for `Observation` lives under the `HL7/fhir` source
repo's resource definition area) and add the corresponding human
descriptions to the resource's invariants table in the narrative pages.
Add validator test cases covering: interpretation-only (should fail),
interpretation + value (should pass), interpretation + dataAbsentReason
(should pass under the relaxed form), and the same combinations on
`component`.

#### Justification

The semantic argument in the ticket is sound: an "interpretation" code
(High / Low / Abnormal / Susceptible / etc., from the
`observation-interpretation` value set) is by definition a
classification of an observed value, so an interpretation without
either a value or an explicit absent-value reason is incoherent in the
data model. Encoding this as a machine-checkable invariant catches data
quality bugs at validation time rather than relying on prose guidance
that implementers routinely miss. Adding `dataAbsentReason` to the
allowed right-hand side preserves the existing FHIR pattern where
`value[x]` may be legitimately omitted when accompanied by a stated
reason — without that relaxation the new invariant would conflict with
established usage.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding an `error`-level invariant on the base `Observation`
resource, add the same constraint at `warning` severity, and document
the "interpretation requires value or dataAbsentReason" expectation in
the resource notes / scope section. Encourage downstream profiles
(e.g., laboratory, vital signs, US Core, IPS) that need stricter
behaviour to either re-state the invariant at `error` severity or rely
on profile-level slicing.

Optionally pair the warning with a tightened obligation in the
`vital-signs` and laboratory-result profiles where the absence of
`value[x]` alongside an interpretation is least defensible clinically.

#### Justification

A base-level `error` invariant is a normative tightening that may
retroactively invalidate existing instances in production systems —
particularly older lab feeds, registry exports, or analytics pipelines
that historically attached interpretation codes to derived/summary
Observations whose value was not modelled. Demoting to `warning`
preserves the data-quality signal for validators and tooling without
breaking conformance for legacy content, and lets profiles that *do*
own the use case (lab, vitals) escalate to `error` where the semantics
are truly settled. This also avoids edge cases such as panel-style
Observations where the parent carries an overall `interpretation` while
the actual values live on `hasMember` or `component`.

---

### Disposition C: Decline

#### Proposal

Do not add the invariant. Close as "Persuasive with Modifier — Not
Persuasive" / "Not Persuasive", with the rationale that the requested
constraint, while clinically appealing in the common case, is not
universally true across all valid uses of `Observation.interpretation`
and is already addressable by profiles that need it.

#### Justification

`Observation` is intentionally a very broad resource used well beyond
laboratory results — survey scoring, derived/summary observations,
genomics findings, social-determinants assessments, and panel parents
that aggregate component values. There are existing real-world
patterns where an interpretation is meaningful at the parent level
(e.g., overall "abnormal" panel) while the parent's `value[x]` is
deliberately empty because the values live on `component` or
`hasMember`. A normative `error`-level constraint at the base resource
risks invalidating these patterns, and the same goal can be achieved
with profile-specific invariants in domains where the semantics are
unambiguous (the existing `vs-2` family of vital-signs invariants is
the precedent). The ticket also presents the clinical-illogic claim
without supporting use-case analysis or community discussion (no Zulip
or linked tickets), which is thin grounding for a base-spec change.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the data-quality intent that motivated the
ticket while avoiding the realistic compatibility risk of an
`error`-level base-resource invariant. `Observation` is one of the most
widely-implemented resources in FHIR; a new base invariant would force
every conformant producer to revisit historical content and every
validator integration to re-baseline. A `warning`-level base invariant
plus targeted `error`-level constraints in the lab and vital-signs
profiles delivers the validator signal Eric is asking for, gives the
workgroup a clear escalation path if the warning proves uncontroversial
in the next ballot cycle, and keeps the door open for Disposition A on
a future cycle once impact has been measured. The workgroup should
also ask Eric for representative examples of the "illogical"
instances he has encountered in the wild, so the human description and
test cases reflect real failure modes rather than hypothetical ones.
