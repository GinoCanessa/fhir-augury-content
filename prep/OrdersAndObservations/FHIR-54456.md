## Summary

The reporter requests that FHIR provide explicit guidance — and possibly a
dedicated search-expansion parameter — so that searches over resources like
`Observation` and `Condition` using a `bodyStructure` / `bodySite` value can
optionally be expanded along the SNOMED CT anatomical hierarchy. Today, a
query such as `patient=X & bodyStructure=<heart>` will miss observations
coded with more specific descendants (e.g., mitral valve, left ventricle,
descending coronary artery), forcing every client to pre-expand the ~100
descendant codes client-side. The ticket asks O&O to clarify whether
hierarchical anatomical expansion is a server responsibility (and, if so,
how it is invoked) or whether clients are formally expected to do this
themselves.

No formal Jira/Zulip/GitHub cross-references are currently recorded for
this item in the Augury index, but several open and applied O&O tickets
touch the same surface area — most notably
[FHIR-51308](https://jira.hl7.org/browse/FHIR-51308) (Expand on Guidance
for `bodySite`) which explicitly calls out hierarchical query behaviour
(e.g., Lumbar Spine vs. Structure of Vertebral Column), and
[FHIR-54872](https://jira.hl7.org/browse/FHIR-54872) which is actively
producing a new BodyStructure-referencing guidance section that this
ticket's guidance could naturally plug into.

## Details

**Description:**

> Somewhere under BodyStructure or Observation, there should be guidance
> (and possibly an "expand" parameter) for invoking anatomical search
> expansion.
>
> If I want to pull a patient cardiac info form the records, I might
> search for Observations where `patient=X` and `bodyStructure=<heart>`,
> but it won't return any of the observations where
> `bodyStructure=mitral valve`, or `left ventricle`, or
> `descending coronary artery`, etc. etc. Similar issue searching for
> Conditions.
>
> Given that the SNOMED anatomical code tree establishes the relevant
> parent-child relationships, can we offer a search parameter that
> triggers the server to apply the tree to expand the search results to
> relevant sub-anatomy, or is the guidance that it's the responsibility
> of all clients to access the anatomy tree and include the 97 relevant
> sub-anatomy codes in their search?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an explicit FHIR search modifier for terminology-based hierarchical
expansion on anatomy-valued parameters, and document it on both the
`BodyStructure` resource page and the `Observation` search-parameters
narrative. Concretely:

- Recommend use of the already-standard `:below` modifier (and
  `ValueSet/$expand` with a descendent-of filter) for
  `CodeableReference`-typed anatomy search parameters (e.g.,
  `Observation?patient=X&body-structure:below=<SNOMED heart>`), and
  define the expected server behaviour: expand along SNOMED CT `is-a`
  to include descendant anatomy codes before matching.
- If O&O concludes a dedicated parameter is still warranted, introduce
  an `_anatomy-expand=true|false` (or equivalent CapabilityStatement
  flag) that is explicitly scoped to anatomy-valued search parameters,
  rather than a generic `expand` parameter.
- Land the guidance in the new BodyStructure "how to reference"
  section being produced under
  [FHIR-54872](https://jira.hl7.org/browse/FHIR-54872), with worked
  examples for `Observation.bodyStructure` and `Condition.bodySite`.
- Call out explicitly that (a) servers MAY decline expansion and must
  advertise support via `CapabilityStatement`, and (b) expansion
  applies to the **coded** arm of a `CodeableReference`, not to the
  `Reference` arm.

#### Justification

This matches what the reporter actually asked for and leverages FHIR
machinery that already exists — `:below`, `ValueSet/$expand`, and
terminology-aware search — rather than inventing a new mechanism. The
O&O group is already producing a cross-resource BodyStructure-usage
section via FHIR-54872, which is the cleanest place for this guidance
and avoids duplicating it under every consumer (Observation,
Condition, ImagingStudy, ServiceRequest, ImagingSelection). It also
directly answers the "is this the client's or server's job?" question
the reporter raised: servers MAY do it, must declare support, and
clients must otherwise fall back to `$expand` + explicit code lists.

---

### Disposition B: Alternative Approach

#### Proposal

Decline to introduce a new parameter or behaviour; instead, add
non-normative **guidance** to the `BodyStructure` page and the FHIR
search narrative that clarifies existing options, without creating new
surface area:

- Document the pattern of building a SNOMED CT anatomy ValueSet
  (descendant-of the target code) via `$expand` and passing the
  resulting codes as a comma-separated `Observation.bodyStructure`
  query.
- Document the existing `:below` modifier for hierarchical code
  matching and recommend it where both the client and server are
  SNOMED-CT-aware.
- Acknowledge that pre-coordinated anatomy in clinical codes
  (see [FHIR-48789](https://jira.hl7.org/browse/FHIR-48789)) means
  no bodyStructure-only expansion will catch every "cardiac"
  observation, and cross-reference the forthcoming BodyStructure
  referencing guidance from
  [FHIR-54872](https://jira.hl7.org/browse/FHIR-54872).
- Leave server behaviour optional and driven by
  `CapabilityStatement.rest.resource.searchParam` documentation.

#### Justification

Anatomical hierarchical search is not specific to `BodyStructure` —
the same reasoning applies to any hierarchical CodeSystem-backed
search parameter (body site, specimen source, procedure site,
device type, etc.). Solving it **only** for anatomy via a bespoke
`expand` parameter would create an inconsistent precedent and bypass
the generic terminology machinery (`:below`, `$expand`, subsumption
testing) that is already specified. This approach delivers the
educational value the reporter is ultimately after, keeps the spec
surface small, and defers to the terminology layer where the real
parent-child semantics live.

---

### Disposition C: Decline

#### Proposal

Close as *Not Persuasive* on the grounds that the request is already
addressed by existing FHIR mechanisms and is therefore out of scope
for a new BodyStructure/Observation feature:

- Hierarchical code matching is already covered by the `:below`
  search modifier.
- Building a descendant-anatomy ValueSet is already covered by
  `ValueSet/$expand` with SNOMED CT `isa/<<` filters.
- Whether a server performs subsumption-aware matching is already a
  `CapabilityStatement` concern.

Signal that no new search parameter, modifier, or resource change is
warranted.

#### Justification

There is a defensible read in which the reporter has simply not
discovered `:below` + `$expand`, and the appropriate remedy is for the
implementer to use them rather than for the core spec to grow new
knobs. Introducing an anatomy-specific `expand` parameter would
duplicate terminology behaviour already specified elsewhere and set an
awkward precedent for every other hierarchical search parameter in
FHIR.

---

### Recommendation

**Recommended disposition:** B

Disposition B delivers the practical outcome the reporter is asking
for — clear, authoritative guidance that clients and servers can point
at — without inventing a new parameter that would overlap with
`:below` and `$expand`, and without implying that anatomy is uniquely
privileged among hierarchical FHIR search parameters. It is also the
best fit with the O&O work already in flight: FHIR-54872 is about to
introduce a consolidated BodyStructure-referencing section that is the
natural home for anatomical-search-expansion guidance, and FHIR-48789
and FHIR-54890 are actively reshaping how `bodyStructure` /
`CodeableReference` are meant to be used. Disposition A (a dedicated
`_anatomy-expand` parameter) is defensible but risks fragmenting the
terminology story for marginal gain, and Disposition C (pure decline)
misreads the ticket — the reporter explicitly asks *"or is the
guidance that it's the responsibility of all clients…"*, i.e. they
are asking O&O to pick a lane and write it down. The workgroup should
write it down.
