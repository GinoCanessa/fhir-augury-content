## Summary

The ticket asks Orders & Observations to add a `category` element (analogous to
the one already present on `ServiceRequest` and `MedicationRequest`) to three
additional Request-pattern resources: `NutritionOrder`, `VisionPrescription`,
and `DeviceRequest`. The reporter (Lloyd McKenzie) frames `category` as a
general-purpose mechanism for grouping orders along multiple dimensions
(acute/chronic, in-patient/out-patient, billing categorizations such as
Da Vinci's inpatient/outpatient marker, etc.) and explicitly notes a
preference for a core element but accepts an extension if the workgroup
prefers. A 0..* `CodeableConcept` is suggested in the discussion so a single
request can be categorized across multiple dimensions.

The cross-reference index surfaced no related Jira tickets, Zulip threads, or
GitHub items for FHIR-54661 itself; the only hit was a textual mention of
`MedicationRequest.dispenseRequest.dispenser` from one of the comments. The
discussion thread, however, explicitly recalls
[FHIR-43444](https://jira.hl7.org/browse/FHIR-43444) ("Analyze resource for
alignment with workflow pattern"), in which OO previously decided **not** to
add `.category` to DeviceRequest because the candidate categorizations (DME,
HME, PriorAuth, etc.) were not vetted and no clear use case was on the table.
That prior decision (resolved Persuasive with Modification, applied in commit
`24b68f1e49`) is the most directly relevant precedent and should anchor the
disposition discussion.

- **mckenzie (2026-03-23):** Replies that `category` is not single-dimensional
  — a single request can carry several CodeableConcepts categorizing across
  multiple dimensions — and that an experimental value set / code system with
  scattered example codes is sufficient to start.
- **esilver (2026-03-24):** Agrees that if added it should be `0..*`, but
  reiterates difficulty pinning down expected category content; several of the
  candidates feel better expressed as other resources/elements/extensions.
  Asks Lloyd whether his items are appropriate categories. Notes that adding
  `category` implies adding a `category` search parameter.
- **mckenzie (2026-03-24):** Walks through the candidates: therapeutic /
  diagnostic / assistive / life-sustaining → category on the order (the same
  device can be ordered for different purposes); consumable/DME → category on
  the device; in-patient / needed-for-procedure / at-home-use → category on
  the order; fulfill-from-ward-inventory / special-order → instructions, not
  categories; patient-purchase / physician-provided / on-loan / trial-sample →
  dispense rather than order characteristics; needs-delivery-scheduling /
  needs-setup-installation-fitting → instructions or workflow. Notes Da Vinci
  uses a category extension for billing-related categorizations he wouldn't
  necessarily want in OO's value set, and confirms a `category` search
  parameter would be needed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new core element `category : CodeableConcept [0..*]` to each of
`NutritionOrder`, `VisionPrescription`, and `DeviceRequest`, following the
shape of `ServiceRequest.category` and `MedicationRequest.category`. Bind each
to a per-resource example value set seeded with codes that emerged from this
discussion, e.g.:

- **DeviceRequest.category** — example codes for therapeutic, diagnostic,
  assistive, life-sustaining, in-patient-use, needed-for-procedure,
  at-home-use (per Lloyd's 2026-03-24 comment); leave billing-style codes
  (DME/HME/PriorAuth, Da Vinci inpatient/outpatient) to extensions or IGs.
- **NutritionOrder.category** — example codes for acute vs. chronic and
  in-patient vs. out-patient, explicitly *not* duplicating the existing
  oralDiet / enteralFormula / supplement structural distinction.
- **VisionPrescription.category** — example codes covering at minimum
  in-patient vs. out-patient and routine vs. post-procedure, to be expanded by
  Vision SMEs (Michelle Ashafa / Paul Knapp).

Add a corresponding `category` SearchParameter (token) to each resource, named
consistently with the existing `ServiceRequest`/`MedicationRequest` `category`
search parameters. Document in each resource's narrative that `0..*` allows
multi-dimensional categorization (one CodeableConcept per dimension).

#### Justification

This is the most direct and consistent answer: it brings the three remaining
Request-pattern order resources into alignment with their siblings and gives
implementers the same hook they already use on ServiceRequest/MedicationRequest
for grouping (acute/chronic, in-patient/out-patient, Da Vinci-style billing
categorizations, etc.). Lloyd's clarification that `category` is multi-
dimensional and that an experimental example value set is sufficient reduces
the binding burden that has historically blocked this change. A `0..*`
cardinality is consistent with the existing analogous elements and makes the
multi-dimensional usage explicit. Adding the search parameter at the same
time avoids a known follow-up gap.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add `.category` to the three resources in core. Instead, publish a
**single standard extension** `http://hl7.org/fhir/StructureDefinition/request-category`
(`CodeableConcept`, cardinality `0..*`, context = `ServiceRequest`,
`MedicationRequest`, `NutritionOrder`, `VisionPrescription`,
`DeviceRequest`, plus other Request-pattern resources as appropriate). Define
a small example value set with the cross-cutting codes the WG can agree on
today (acute/chronic, in-patient/out-patient, Da Vinci's inpatient/outpatient
billing marker), and leave domain-specific categorization (e.g., DeviceRequest
therapeutic/diagnostic/assistive) to either future ticket work or
domain-specific IG-defined extensions.

Where the existing core `category` element is already published
(ServiceRequest, MedicationRequest), document the standard extension as a
parallel mechanism for Request-pattern-wide categorizations and clarify when
implementers should reach for which.

#### Justification

This honours Lloyd's stated fallback ("ok with an extension if that's
preferred by the WG"), Mark McKenzie's 2026-01-29 note that a *standard*
extension for inpatient/outpatient would be acceptable, and the OO concerns
that surfaced in this ticket — namely (a) Nutrition's objection that a
NutritionOrder.category would duplicate existing structural distinctions and
risks oversimplifying complex orders, and (b) DeviceRequest's prior decision
in [FHIR-43444](https://jira.hl7.org/browse/FHIR-43444) to defer .category
pending vetted use cases. A single Request-pattern extension also avoids
introducing three new core elements (and three new mandatory search
parameters) whose example value sets are admitted to be experimental, and
keeps Da Vinci's billing-style categorizations out of the core spec where
Lloyd himself signalled discomfort with their inclusion. The extension can be
promoted to a core element on each resource individually once a vetted value
set has emerged.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** for all three resources at this time. Specifically:

- For **DeviceRequest**, reaffirm the FHIR-43444 decision: do not add
  `.category` until concrete, vetted use cases and a defensible value set
  exist. Implementers needing a categorization today may use a local
  extension (e.g., Da Vinci's existing one).
- For **NutritionOrder**, decline on the basis of the Nutrition project
  group's analysis: the existing oralDiet/enteralFormula/supplement structure
  already discriminates the most common categorical question, and a
  `category` field is judged likely to duplicate or oversimplify rather than
  clarify.
- For **VisionPrescription**, decline pending input from the Vision SMEs
  (Michelle Ashafa, Paul Knapp) and concrete use cases plus example codes; no
  evidence of demand has been recorded on this ticket.

Suggest the reporter raise per-resource follow-up tickets when concrete
categorization use cases and example codes are available.

#### Justification

OO has explicitly walked this path before for DeviceRequest in
[FHIR-43444](https://jira.hl7.org/browse/FHIR-43444) and concluded the
candidates were not vetted. Nothing in this ticket's discussion vetted them
further: esilver's comments document continued difficulty identifying
appropriate example vocabulary; Lloyd's own walkthrough rejects roughly half
of the candidate codes as belonging on the device, on dispense, or on
instructions rather than on the request; and the Nutrition project group has
formally objected to NutritionOrder.category. Adding a normative element
(plus mandatory search parameter) on the strength of an "experimental
example value set with some scattered codes" risks shipping under-defined
categorization that will be implemented inconsistently — exactly the
"category is inconsistently defined/used" concern OO raised in this thread.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — single standard
Request-pattern category extension).

Disposition B threads the strongest path through the discussion as it stands.
Lloyd explicitly accepts an extension; he also explicitly prefers a *standard*
extension for inpatient/outpatient. A single multi-resource standard extension
delivers the cross-cutting grouping capability that motivated the ticket
(acute/chronic, in-patient/out-patient, Da Vinci's billing marker) without
forcing OO to mint three under-vetted core elements and three new mandatory
search parameters at once — addressing Nutrition's substantive objection that
NutritionOrder.category would duplicate existing structural distinctions, and
respecting OO's standing FHIR-43444 decision on DeviceRequest. It is also
strictly upgradeable: once any of the three resources accumulates a vetted
domain-specific value set (most plausibly DeviceRequest first, given the
therapeutic/diagnostic/assistive list Lloyd endorsed), that resource can be
promoted from extension to core element in a follow-up ticket without
reversing this disposition. Disposition A is defensible if the WG is content
to ship example bindings now; Disposition C is defensible if the WG is
unwilling to commit to *any* shared categorization vocabulary, but it leaves
Da Vinci's existing need unaddressed and is harder to justify given Lloyd's
willingness to accept a lighter-weight mechanism.
