## Summary

The ticket reports that the Obligation tables in the Laboratory Report
(FHIR) IG's Obligation profiles (e.g.,
`StructureDefinition-Patient-obl-uv-lab`) render with an empty **Actor**
column in the *Description & Constraints* section. The reporter assumes
each row corresponds to one of the three documented actor roles, but
because the column is blank, readers cannot tell which actor is bound to
which obligation. The request is to populate the column or otherwise
make the actor-to-obligation mapping clear.

Investigation by the work group concluded the blank column was a
**FHIR IG Publisher rendering bug**, not missing source data — the
underlying `obligation` extensions on `Patient-obl-uv-lab` did contain
`actor` canonicals (e.g., `http://hl7.org/fhir/uv/laboratory/actor-repos-uv-lab`),
they simply weren't being shown. The Publisher fix has since landed; the
remaining IG-side action is cosmetic — surface the actor's `title`
instead of (or in addition to) the canonical/name so the rendered table
is human-readable. The cross-reference index only links this ticket to
the FHIR artifact `Patient.birthDate`, which appears solely as a
boilerplate example inside a quoted JSON snippet in a comment and is
not substantively related.

## Proposed Dispositions

> Note: the ticket is already resolved as *Persuasive with Modification*
> and assigned to Jose Costa-Teixeira. The dispositions below frame the
> options the work group can affirm at vote.

### Disposition A: Accept As Requested

#### Proposal

Persuasive with Modification (current resolution): treat the
empty-Actor column as the originally-reported defect, but resolve it in
two parts.

1. **Publisher fix (already landed)** — confirm the FHIR IG Publisher
   change that renders the `actor` extension on Obligation rows is in
   the version of the Publisher used to build the Laboratory Report IG.
   No IG source change is needed for this leg; it is a tooling
   dependency.
2. **IG cosmetic improvement** — on each Obligation profile in the
   Laboratory Report IG (`Patient-obl-uv-lab`,
   `Specimen-obl-uv-lab`, `Observation-*-obl-uv-lab`,
   `DiagnosticReport-*-obl-uv-lab`, etc.), ensure every
   `obligation` extension's `actor` sub-extension points at an
   ActorDefinition in the companion `actor-repos-uv-lab` (or the
   ballot's actor catalogue) **and** that those ActorDefinitions
   carry a `title`. Where the actor canonical resolves but the
   referenced ActorDefinition has no `title`, add one so the
   Publisher renders a human-readable label rather than the bare
   canonical / id.

Re-publish the IG against a Publisher build that contains the fix and
verify the *Description & Constraints* section now shows a populated
Actor column with titles.

#### Justification

This matches both the reporter's literal ask ("populate … these
tables") and the investigation outcome (rendering bug + minor
authoring polish). It uses the existing Obligation/ActorDefinition
mechanism rather than introducing a new pattern, keeps the IG
consistent with how other Obligation-using IGs render their tables,
and directly addresses giorgio's Zulip-equivalent comment that
human-readable titles should be shown. It is also the
lowest-risk path: the Publisher change has independent QA, and the
IG-side delta is limited to ActorDefinition `title` population.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification, but expand the scope beyond rendering:
add a short **explanatory paragraph** at the top of each Obligation
profile's *Description & Constraints* section that names the actors,
explains how the table should be read (one row per
obligation-binding-to-actor pair, not one row per actor), and links to
the IG's actor catalogue. Combine that with the Publisher fix and the
ActorDefinition `title` polish from Disposition A.

#### Justification

The original ticket text contains two sub-requests: "populate (or
**better explain the usage** of) these tables." Disposition A only
addresses the first half. A boilerplate intro block — possibly
authored once and `{% include %}`-d (or
its FHIR IG equivalent) into each Obligation profile's
`-intro.md` — eliminates the cognitive load that prompted the ticket
in the first place, and makes the IG more self-describing for
implementers who land on a single profile page from search. Trade-off:
a small amount of authoring work and a per-profile narrative file to
maintain. Backwards compatibility is unaffected — narrative-only
changes don't break conformance.

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed by tooling** — close as Not Persuasive
on the grounds that no IG source change is required: the Actor column
was always populated in the StructureDefinition source, the Publisher
bug has been fixed upstream, and a re-publish will resolve the
reporter's complaint without any IG-level edits.

#### Justification

A defensible stance only if (a) every Obligation profile's existing
`actor` canonicals already resolve to ActorDefinitions that carry a
usable `title`, and (b) the work group is comfortable that the
"better explain the usage" half of the ticket is satisfied implicitly
by a populated table. The risk is that this leaves the second half of
the request unanswered and treats a reporter-visible defect as purely
a tooling concern, which is generally weaker than the
Persuasive-with-Modification posture the work group has already taken.

---

### Recommendation

**Recommended disposition:** **A** (with an eye toward folding in
Disposition B's narrative addition if authoring bandwidth allows).

Disposition A reflects what the work group has already converged on,
matches the resolution text recorded on the ticket
("The issue with the Publisher has now been resolved. Will also add
the title for the actors for completeness …"), and is exactly what the
two commenters proposed. It closes the literal defect with a
mechanism-appropriate fix (Publisher + ActorDefinition `title`),
avoids inventing new spec concepts, and is the least disruptive option
for implementers tracking the IG. Disposition B is a worthwhile follow-up
and can be pursued as a non-blocking authoring task; Disposition C
under-serves the "better explain the usage" half of the ask and is
not recommended.
