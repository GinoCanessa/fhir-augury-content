## Summary

The reporter (David Barwin, O&O) flags an informational message emitted
by the FHIR CI build for the `Task:input` and `Task:output` search
parameters. Both parameters are typed as `reference` and use the FHIRPath
expression `Task.input.value.ofType(Reference)` (and the matching
`Task.output.value.ofType(Reference)`), but the build's static analysis
cannot derive any target resource types from those expressions because
`Task.input.value[x]` and `Task.output.value[x]` are declared as the
open `*` choice type. The ticket asks the workgroup to confirm whether
these search parameters are correctly defined and whether they will
function at runtime, given that the build infrastructure cannot infer a
target list. No related Jira tickets, Zulip threads, or GitHub items
were surfaced through the cross-reference, search, or related-keyword
queries — the cross-reference index only links the ticket to the FHIR
elements `Task.input.value` and `Task.input.value.ofType` that it
mentions.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the CI message as a real defect in the `Task-input` and
`Task-output` search parameter definitions and tighten the expressions
so the build can derive a non-empty target list. Replace the current
expression `Task.input.value.ofType(Reference)` with an enumerated
`OR` over the concrete reference types O&O is willing to commit to
indexing — e.g.,
`Task.input.value.ofType(Reference) | Task.input.value.ofType(canonical)`
is **not** sufficient (canonical is not a reference); instead author
the expression as a union over each `Reference(X)` profile that O&O
considers in scope (Patient, Practitioner, PractitionerRole,
Organization, Device, Group, Encounter, ServiceRequest, etc.) and
populate the `SearchParameter.target` element with the matching
resource list. Apply the same fix to `Task-output`. The change lives
in the FHIR core Task resource artifacts (the `searchparameter-*.xml`
files for Task in HL7/fhir under the Task source folder).

#### Justification

The CI message is correct: a `reference`-typed search parameter with
an empty derived target list cannot be served meaningfully by a
server, because the server has no list of resource types to index or
to validate `:[type]` modifiers and `_include`/`_revinclude` chains
against. Per the search-parameter rules in the base spec, a reference
search parameter SHALL declare its target types, and many server
implementations refuse to register a reference SP with no targets.
Enumerating the target types — even if the list is broad — restores
deterministic behavior and silences the warning, while making
implementer expectations explicit.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the search parameters but acknowledge that `Task.input.value[x]`
and `Task.output.value[x]` are intentionally open (`*`), and adjust
the artifacts so that (1) the FHIRPath expression remains
`Task.input.value.ofType(Reference)` / `Task.output.value.ofType(Reference)`
but (2) `SearchParameter.target` is explicitly populated with the
full set of resource types that may legally appear in a `Reference`
(i.e., the full `ResourceType` value set, expressed via the standard
"all resource types" target list used elsewhere in the core spec for
"any reference" parameters such as `Provenance.target` or
`Subscription.topic`-style any-targets). Optionally, also adjust the
build's static analyser to special-case `ofType(Reference)` over an
open choice and populate the target list from the full resource
inventory rather than emitting a warning.

#### Justification

`Task.input.value[x]` and `Task.output.value[x]` are deliberately
open because Task is a generic work-list resource — narrowing the
target list to a hand-picked subset (Disposition A) risks excluding
legitimate uses and would need a workgroup-wide policy call about
which targets are in scope. Declaring "all resource types" is a more
honest reflection of the data type and matches how other "any
reference" parameters are modelled. The build-side tweak is
optional but eliminates the recurring noise without changing the
artifact whenever a similar "open `*` + `ofType(Reference)`" pattern
is added to another resource.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive / Working as Designed**: the CI
output is an *informational* message, not an error, and it is a known
limitation of the static analyser when the `ofType(Reference)`
operator is applied to an open choice. The search parameters are
defined correctly per the existing rules, will work at runtime on any
conformant server (the server walks the FHIRPath expression and
indexes whichever Reference instances it finds), and no spec change
is warranted. Document the limitation in the build-tooling notes and
move on.

#### Justification

The message is explicitly described in the ticket as "an information
message", not a build-breaking error, and the underlying expression
is well-formed FHIRPath. Servers that implement search the way the
spec describes will index `Task.input.value` whenever it happens to
contain a `Reference`, regardless of whether the SearchParameter's
`target` list is enumerated. Tightening either the expression or the
target list (Dispositions A/B) has implementer cost — anyone who has
already published profiles or stored search indexes against
`Task-input`/`Task-output` could see modifier behaviour change — for
no clear runtime benefit.

---

### Recommendation

**Recommended disposition:** B

Disposition B preserves the intentional openness of `Task.input.value[x]`
and `Task.output.value[x]` (which is the whole point of the design —
Task is a generic work-list resource and narrowing its references at
the spec level would be a substantive scope decision the workgroup
has not agreed to), while still resolving the legitimate concern
behind the CI message: a `reference` search parameter with no
declared targets is genuinely under-specified and forces every
server and validator to invent its own behaviour. Populating
`SearchParameter.target` with the full resource-type list — exactly
as is done for other "any reference" parameters in the core spec —
is the smallest change that makes the artifacts internally consistent
and silences the warning without committing O&O to a curated subset
(Disposition A) or papering over the underspecification (Disposition
C). If the workgroup also wants to push the build tooling to stop
warning on the `ofType(Reference)`-over-open-choice pattern, that can
be filed as a separate FHIR-IG-Publisher / build ticket.
