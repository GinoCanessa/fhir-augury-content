## Summary

The submitter (Daniel Pawlak) reports that R6 has introduced an
inconsistency between two closely related request/event resources in how
they represent anatomical location. In R4, both `ServiceRequest` and
`Procedure` used a `bodySite` element (typed as `CodeableConcept`, with
an extension to point at a `BodyStructure` resource). In R6,
`ServiceRequest` has dropped its `bodySite` element entirely and now
only carries `bodyStructure` (a `CodeableReference(BodyStructure)`),
while `Procedure` retains both `bodySite` (`CodeableConcept`) and
`bodyStructure` (`Reference(BodyStructure)`) and uses an invariant to
forbid using both at once on the same resource. The ticket asks the
workgroup to explain the rationale for this divergence — and, by
implication, to consider whether the two resources should be aligned.

No cross-referenced Jira tickets, GitHub items, or Zulip threads were
returned by FhirAugury for `FHIR-53826`. A topical Zulip search did
surface a 2024 implementers thread in which an Estonian implementer
asked exactly this question — what the difference is between
`ServiceRequest.bodySite` and `ServiceRequest.bodyStructure` — without
a definitive answer, suggesting the inconsistency has been a source of
implementer confusion for some time.

## Details

**Description:**

ServiceRequest R6 deletes the bodySite element and instead uses the
bodyStructure element. Procedure has separate bodySite and
bodyStructure elements and uses a constraint to enforce only one being
used at a time for a given resource. In R4 both these resources used
bodySite, now they are inconsistent. What is the reason that these 2
resources represent body location differently?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a request to align the two resources and bring
`ServiceRequest` back into structural parity with `Procedure`. Re-add
`ServiceRequest.bodySite` (`0..* CodeableConcept`, bound to the same
`SNOMED CT body structures` value set used by `Procedure.bodySite`),
keep `ServiceRequest.bodyStructure` as it currently is in R6
(`CodeableReference(BodyStructure)` — or, if alignment with
`Procedure` is preferred, narrow it to `Reference(BodyStructure)`),
and add the same invariant `Procedure` carries — i.e., body site SHALL
NOT be present at the same time as bodyStructure on a single
`ServiceRequest`. Document the parallel modelling in the
`ServiceRequest` and `Procedure` resource notes and cross-link to
`BodyStructure`.

#### Justification

`ServiceRequest` and `Procedure` form a request/event pair in the
workflow pattern; an order placed via `ServiceRequest` is intended to
be fulfilled by a `Procedure`, and downstream systems routinely copy
the requested anatomy into the performed-procedure record. Diverging
modelling between the request and the event forces every implementer
who bridges the two (CDS, scheduling, EHR write-back, reporting) to
write conversion logic for what should be the same concept. The
implementer Zulip thread above is direct evidence that this
inconsistency causes confusion. Restoring `bodySite` keeps the simple,
codeable-only path that the vast majority of orders need, while
`bodyStructure` remains available for the cases that genuinely
require a referenced `BodyStructure` instance (laterality, landmarks,
qualifiers).

---

### Disposition B: Alternative Approach

#### Proposal

Resolve the inconsistency in the opposite direction: keep
`ServiceRequest`'s R6 design (single `bodyStructure` element typed as
`CodeableReference(BodyStructure)`) and propose, in coordination with
Patient Administration, that `Procedure` migrate to the same shape —
collapsing `Procedure.bodySite` and `Procedure.bodyStructure` into a
single `Procedure.bodyStructure : CodeableReference(BodyStructure)`
element and retiring the `bodySite`-vs-`bodyStructure` invariant. Add
a backwards-compatibility note explaining the R4 → R6 migration: an
R4 `bodySite` `CodeableConcept` maps to the `concept` slot of the new
`CodeableReference`, and any R4 `BodyStructure` extension maps to its
`reference` slot.

#### Justification

`CodeableReference` was introduced in R5 specifically to remove the
"do I send a code or do I send a reference?" branching that the
`bodySite` + `bodyStructure` + invariant pattern reproduces. Using a
single `CodeableReference(BodyStructure)` element is consistent with
how anatomy is now represented on other R6 resources
(e.g., `Observation.bodyStructure`, `ObservationDefinition.bodyStructure`,
`DocumentReference.bodyStructure`, `ImagingSelection.bodySite`,
`ImagingStudy.series.bodySite` — all `CodeableReference`). This
direction reduces cardinality, removes an invariant, and brings
`Procedure` into line with the rest of the R6 design rather than
keeping a transitional shape. Cross-WG coordination with Patient
Administration is required (per the triage comment).

---

### Disposition C: Decline

#### Proposal

Decline the request as a "no change required" item: confirm that the
divergence is intentional and a consequence of differing maturity and
WG ownership (`ServiceRequest` owned by O&O, `Procedure` owned by
Patient Administration), document the rationale in both resources'
narrative ("Boundaries and Relationships" / notes), and close as
"Persuasive with Mod" with that documentation as the only change.

#### Justification

`ServiceRequest` and `Procedure` are different artifact types
(request vs. event) and live under different workgroups, so some
modelling drift is unavoidable across major versions. The current R6
shape is internally consistent on each resource, the migration impact
of changing either side again before publication is non-trivial, and
the ticket asks a "why" question rather than reporting a broken
exchange. A documentation-only response acknowledges the inconsistency
without re-opening the modelling debate this late in the cycle.

---

### Recommendation

**Recommended disposition:** B

The R6 move on `ServiceRequest` to a single
`CodeableReference(BodyStructure)` is the more modern shape and is
already mirrored by every other R6 resource that carries anatomy
(`Observation.bodyStructure`, `ObservationDefinition.bodyStructure`,
`DocumentReference.bodyStructure`, `ImagingSelection.bodySite`,
`ImagingStudy.series.bodySite`). Disposition A reverses that direction
on `ServiceRequest` for parity with a `Procedure` shape that is itself
out of step with the rest of R6 — it removes the local inconsistency
but propagates the legacy pattern. Disposition C ignores a real
implementer confusion point (the implementers Zulip thread shows
people are already asking which element to send). Disposition B fixes
the inconsistency in the direction the rest of R6 has already taken,
and the only blocker is a coordination conversation with Patient
Administration, which the triage comment already flagged as
appropriate. The workgroup should expect pushback on migration cost
for `Procedure` and may want to pair this with a clear R4/R5 → R6
mapping note.
