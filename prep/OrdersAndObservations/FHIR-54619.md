## Summary

The ticket flags a substantive asymmetry between the allowed `Reference` target
types on the `basedOn` element and those on the `replaces` element of a
workflow request resource (the description's tables list the reporter-observed
target sets but do not name a single resource — the `basedOn` set
`CarePlan, DocumentReference, ServiceRequest, MedicationRequest, RequestOrchestration, NutritionOrder`
includes `DocumentReference` and omits several common request resources, while
the `replaces` set
`ServiceRequest, MedicationRequest, RequestOrchestration, CarePlan, DeviceRequest, CommunicationRequest, NutritionOrder, VisionPrescription`
matches the canonical "request" resource family from the workflow pattern).
Eric Haas argues this asymmetry is confusing for implementers and asks the
work group to either align the two lists or, if the differences are
intentional, document the rationale in the element definitions.

There are no system-tracked cross-references for this ticket. However, several
other recent O&O change requests are clearly part of the same cleanup theme:
[FHIR-54167](https://jira.hl7.org/browse/FHIR-54167) (DocumentReference's
`basedOn` description still talks about "a procedure") and
[FHIR-54536](https://jira.hl7.org/browse/FHIR-54536) (same issue, separately
reported and already Applied as Persuasive with Modification) confirm that
`basedOn` on `DocumentReference` was recently re-pointed at request resources
— which is exactly the change that produced the asymmetric set being
reported here. [FHIR-40631](https://jira.hl7.org/browse/FHIR-40631) (also
Applied) added `RequestOrchestration` to several request resources' `basedOn`
lists, and [FHIR-43447](https://jira.hl7.org/browse/FHIR-43447) is the broader
workflow-pattern alignment ticket that frames how O&O has been bringing
request resources into line. Together they suggest the asymmetry is not
deliberate — it's a half-finished cleanup that this ticket is asking the
work group to complete.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Align `basedOn` and `replaces` on the request resources called out in the
ticket so that both elements accept the same set of "request" resource
targets. Concretely: take the current `replaces` target list
(`ServiceRequest, MedicationRequest, RequestOrchestration, CarePlan,
DeviceRequest, CommunicationRequest, NutritionOrder, VisionPrescription`) as
the reference workflow-pattern set, and update `basedOn` on each affected
request resource to use the same set, except that `DocumentReference` MAY
remain on `basedOn` only where the resource genuinely needs to record that
it was based on a document (since `replaces` is request-to-request and a
`DocumentReference` would not be replaced). Where `basedOn` currently omits
a resource that appears on `replaces` (e.g., `DeviceRequest`,
`CommunicationRequest`, `VisionPrescription`), add the missing targets.
Update each resource's element-definition text so the natural-language
description and the target list agree, mirroring the cleanup already done
under FHIR-54167 / FHIR-54536.

#### Justification

The work group has already accepted, via FHIR-40631, FHIR-54167, FHIR-54536,
and FHIR-43447, that `basedOn` on the request resources is curated against
the workflow request pattern and that mismatches between text and target
list are bugs. The asymmetry the reporter identified is a leftover of those
incremental fixes — `replaces` was updated to the full workflow set,
`basedOn` was only partially updated on each resource. Bringing them into
parity (with `DocumentReference` as a documented exception on `basedOn`) is
consistent with the workflow pattern, removes implementer confusion, and
finishes work the work group has already endorsed. It is also low-risk
because broadening reference targets is additive for existing data.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the target lists. Instead, document the intentional
asymmetry in (a) the workflow request pattern page and (b) the element
definitions for `basedOn` and `replaces` on each affected resource. Add a
short normative-style sentence to each element's description explaining
why the target sets differ — e.g., `replaces` is restricted to other
request resources because it expresses request-supersession semantics,
while `basedOn` may also reference plans/orchestrations and (where
applicable) the document that triggered the request. Where any single
target is genuinely missing for a known clinical reason (e.g.,
VisionPrescription on `basedOn`), still add it, but treat that as a
narrow targeted fix rather than a wholesale alignment.

#### Justification

The two elements are not synonyms: `basedOn` ("this request was
authorized/derived from X") and `replaces` ("this request supersedes X")
have different semantics, and there are defensible reasons for the lists
to diverge — for example, you would not expect to replace a
`DocumentReference` or `Appointment`, but you might legitimately base a
request on one. Codifying the rationale rather than enforcing symmetry
preserves clinical accuracy and avoids opening "why can I base a
ServiceRequest on a DocumentReference but not replace one?" questions
that symmetric lists would create. This is the lower-impact change and
explicitly addresses the reporter's secondary recommendation ("…or
document the intentional differences").

---

### Disposition C: Decline

#### Proposal

Decline the change request as written. Treat the asymmetry between
`basedOn` and `replaces` as deliberate workflow-pattern semantics that
have already been reviewed under prior tickets (FHIR-40631, FHIR-43447,
FHIR-54167, FHIR-54536), and direct the reporter to the workflow request
pattern page for the rationale.

#### Justification

The two elements model different relationships and existing related
tickets have already been resolved on a case-by-case basis. Re-opening a
generic alignment exercise risks churning element definitions across many
request resources for marginal benefit, and the ticket as written does
not name a specific resource or implementer scenario where the
asymmetry caused harm. Pure-documentation tickets that don't propose a
concrete textual fix are usually better closed with a pointer to the
existing pattern page.

---

### Recommendation

**Recommended disposition:** B

The reporter's substantive concern is implementer confusion, not a bug in
the model — `basedOn` and `replaces` legitimately have different
semantics, and the reporter explicitly accepts a documentation outcome
("…or document the intentional differences"). Disposition B captures
that intent at the right scope: it adds rationale to the element
definitions and the workflow request pattern page so future implementers
can see why the lists differ, while still leaving room to add specific
missing targets (such as VisionPrescription on `basedOn`, per FHIR-40631)
as targeted follow-ups. Disposition A is defensible and consistent with
recent O&O cleanup, but a wholesale "align both lists" change would
overreach the ticket and would force `replaces` to accept targets (e.g.,
`DocumentReference`) where supersession semantics don't apply.
Disposition C is too dismissive given that the reporter is a longtime O&O
contributor and has correctly identified that the prior incremental
fixes left the lists visibly inconsistent. B threads the needle: keep
the design, but make the design legible.
