## Summary

Eric Haas (Orders & Observations) reports that the **extensible** value-set
bindings on `Observation.interpretation` and
`Observation.component.interpretation` reference code systems that may not
be used by every jurisdiction or implementation community. Because
`extensible` requires implementers to use a code from the bound value set
when one of the listed concepts covers their meaning, jurisdictions that
have their own coded interpretation vocabularies (e.g., regional lab
result-interpretation codes) are forced to either pretend a value-set
match exists or argue that nothing in the value set fits. The ticket's
recommendation is to weaken the binding strength on both elements from
`extensible` to `preferred`, which would make
`http://terminology.hl7.org/ValueSet/v3-ObservationInterpretation` (or
its successor) a recommendation rather than a requirement. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the binding strength of both
`Observation.interpretation` and
`Observation.component.interpretation` from `extensible` to
`preferred`, leaving the existing bound value set
(`http://hl7.org/fhir/ValueSet/observation-interpretation`, backed by
`http://terminology.hl7.org/CodeSystem/v3-ObservationInterpretation`)
in place as the recommended vocabulary. No structural change to the
element, cardinality, or type is needed â€” only the `binding.strength`
attribute on each element flips, with a corresponding short note in
the binding description explaining that jurisdictional or
laboratory-specific interpretation code systems may be used instead
when they better fit local practice.

#### Justification

The reporter's concern is consistent with how `extensible` is defined
in the FHIR specification: implementers must use a value from the
bound set whenever one applies, which is precisely the friction point
when a jurisdiction has its own established interpretation vocabulary
(e.g., national lab terminology, regional reference-range flags) that
overlaps semantically with HL7 v3 ObservationInterpretation but is
not aligned to it. `preferred` keeps the HL7 v3 set as the
recommended starting point â€” preserving interoperability for
implementers without local conventions â€” while explicitly permitting
codes drawn from any other code system. Because `preferred` is a
strict relaxation of `extensible`, no instance that was valid under
the current binding becomes invalid, so the change is
backward-compatible for existing implementers and profiles.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the binding strength as `extensible` on both elements, but
introduce (or expand) a jurisdiction-extensible composition: define
the bound value set as a grouping value set that explicitly includes
the existing `v3-ObservationInterpretation` codes plus a
"jurisdictional interpretation codes" supplement, and document in the
binding description that implementers in jurisdictions with local
interpretation vocabularies are expected to extend the value set
(via IG-level profiling) rather than fall back to an arbitrary code.
Optionally pair this with a normative note that the v3 codes remain
the preferred fallback when no jurisdictional code applies.

#### Justification

This preserves the stronger interoperability signal that
`extensible` carries â€” implementers cannot silently ignore the
HL7-curated vocabulary â€” while addressing the reporter's underlying
concern that a jurisdiction may have meaningful interpretation codes
that fall outside the current set. It is more work than Disposition A
(value-set authoring, IG guidance, and possibly UTG coordination),
but it leaves Core FHIR with a meaningful semantic floor for
result interpretation, which aids cross-jurisdictional exchange. The
trade-off is that profiles in jurisdictions without local codes must
continue to use the v3 set as today, and that downstream IGs (e.g.,
US Core, IPS, regional cores) carry the responsibility to express
their own interpretation extensions â€” a heavier coordination cost
than a one-line strength change.

---

### Disposition C: Decline

#### Proposal

Decline the request and retain the `extensible` binding on both
elements, on the grounds that `extensible` already accommodates the
case the reporter describes: per the FHIR binding-strength
definitions, when no concept in the bound value set is suitable for a
given use case, implementers may use a code from a different system.
Document this more clearly in the binding description (or in the
Observation resource notes) rather than weakening the binding.

#### Justification

`extensible` is explicitly designed for exactly this scenario: it
mandates use of a bound code only when one applies, and otherwise
permits any other code. Weakening to `preferred` removes the
semantic floor that downstream consumers rely on when interpreting
`Observation.interpretation` across jurisdictions â€” an `N` (normal)
or `H` (high) flag from the v3 set is meaningful precisely because
it is the recommended-and-expected vocabulary. If jurisdictions are
in fact using the freedom `extensible` already grants, a
documentation clarification is sufficient and avoids destabilising
the analytic and decision-support tooling that depends on a
predictable interpretation vocabulary at the Core level.

---

### Recommendation

**Recommended disposition:** A

`extensible` is, in practice, frequently misread by implementers as
"must come from this set" â€” a perception the reporter is reflecting
rather than misinterpreting. The HL7 v3 ObservationInterpretation
codes were authored from a primarily North-American laboratory
perspective and do not cleanly cover every jurisdiction's
interpretation vocabulary (especially outside lab use, where
`Observation.interpretation` is increasingly applied to imaging,
genomics, and patient-reported observations). Disposition A delivers
the relief the reporter asks for with a one-attribute change, is
fully backward-compatible (every instance valid today remains valid),
and leaves Disposition B available as a follow-up at the IG level for
jurisdictions that want a stronger floor. Disposition C is defensible
on paper but underestimates the real-world drag that an `extensible`
binding to a North-American-leaning code system places on
international and specialty implementers; the workgroup should weigh
that operational reality against the abstract semantic guarantee
`extensible` provides.
