## Summary

The reporter requests that `ServiceRequest` gain a new `dateTime` element
that captures when the request was *actually created* (i.e., first entered
into the system as a draft), distinct from the existing `authoredOn`
(documented as "Date request signed") and `occurrence[x]` ("When service
should occur"). The motivating context is the home-health and hospice
"creation → sign → render" lifecycle, where therapies may be recorded
well before an attending or IDG signs off, and where the gap between
creation and signing is itself clinically and operationally meaningful
(e.g., for identifying long-unsigned drafts). The reporter explicitly
considered and rejected two existing mechanisms: deriving the time from
the surrounding `Encounter` (encounters often span multiple days and a
single encounter can hold many requests) and using
`ServiceRequest.orderDetail.parameter.valuePeriod` (creation is a
singular instant rather than a period).

There are no cross-referenced Jira tickets and no related GitHub items
for this issue. A relevant Zulip thread from 2018
(`#implementers > Search for "New" ServiceRequests`) raises the same
ambiguity from the implementer side: `authoredOn` is searchable but is
defined as "when the request was signed", which makes finding *newly
arrived* draft requests awkward. The ticket effectively asks O&O to
resolve that ambiguity by adding a dedicated creation timestamp.

## Details

**Description:**

ServiceRequest currently has two elements that track dateTimes.
"authoredOn" captures the "Date request signed" and "occurrencedatetime"
captures "When service should occur". Neither of these captures when the
service request was actually created.

The time between creation of a request and the signing off on it can
differ greatly so I don't believe the resource currently reflects the
full timeline of the request. This delay can also be important for
identifying requests still being considered, or that were not signed.
Between the authoredOn element having a cardinality of 0..1 and the
status element it's clear for the request to be documented and not be
signed but there is no indication of how long it has gone unsigned for.

I'm approaching this from a home health and hospice perspective where
therapies may be requested/ordered, the attending or IDG reviews the
request and signs off, and then the service is actually rendered. This
seems like it would be applicable to many other areas of health care as
well in regards to the creation-sign-render lifecycle of service
requests and orders.

It may be possible to capture this through the encounter element but the
actual encounter itself can span multiple days and multiple requests may
be created during a single encounter, each with their own creation time.
The orderDetail.parameter.valuePeriod might also be used but the
creation of request seems to be more of a singular instant in time, as
opposed to a period.

Thank you for the consideration.

Tim Berezny asked how to poll a server for *newly arrived*
ServiceRequests, observing that `_lastUpdated` matches edits as well as
creations and that `ServiceRequest.authoredOn` is documented as "when
the request was signed" rather than created. Eric Haas suggested the
options were polling, `_history`, or FHIR sub/pub. Lloyd McKenzie
clarified that ServiceRequests aren't presumed actionable on existence —
implementers should either use a `Task` and filter on its status, or, if
just tagging actionable requests, use `authoredOn`, which he framed as
"when the Request was transitioned to Active" (i.e., not a true
"created" timestamp either). No consensus was reached on a dedicated
creation-time element; the thread effectively ends with workarounds
rather than a spec change.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new optional element `ServiceRequest.created : dateTime [0..1]`
defined as "The instant the ServiceRequest was first recorded in the
source system, regardless of its authorization status." Tighten the
definition of `ServiceRequest.authoredOn` to clarify that it represents
the moment the request was *signed / authorized* (i.e., transitioned
out of `draft`/`proposal` toward `active`), and add a cross-reference
between the two elements in their respective comments. Mark `created`
as Trial Use, summary=false. No invariants or search parameters are
required initially, though a `created` search parameter would be a
natural follow-up for the "find new requests" use case raised on Zulip.

#### Justification

This directly satisfies the reporter's requirement to model the
creation-sign-render lifecycle and resolves the long-standing
implementer ambiguity flagged in the 2018 Zulip thread (where
`authoredOn` is variously interpreted as "signed" or "transitioned to
Active"). It is additive and backward-compatible: existing instances
remain valid, and consumers that don't care about the draft phase can
continue to use `authoredOn`. The new element is consistent with the
pattern already used in other request resources in FHIR for capturing
distinct lifecycle timestamps. It also positions the resource to
support a `created` search parameter, addressing the "polling for new
requests" workflow that has been an open implementer pain point.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new element to the core ServiceRequest resource.
Instead, address the underlying need in two layered ways:

1. **Clarify `authoredOn`**: amend its definition and short label to
   read "When the request was first recorded (created) in the source
   system" and remove "Date request signed" wording. Document that the
   sign-off event, where it differs from creation, should be captured
   via `Provenance` (with `Provenance.recorded` and an
   `activity = Sign` / `agent.type = author`) targeting the
   ServiceRequest.
2. **Provide implementation guidance**: add a paragraph to the
   ServiceRequest narrative ("Lifecycle and timestamps") explaining
   how to recover signing time from `Provenance` and how
   `meta.lastUpdated` differs from both. Optionally, define a Task-based
   pattern for tracking authorization workflow, mirroring Lloyd's
   Zulip guidance.

#### Justification

ServiceRequest is a heavily-used resource and adding a near-synonym to
`authoredOn` risks long-term confusion about which timestamp to
populate, especially in profiles that have already constrained
`authoredOn`. FHIR already provides `Provenance` for capturing
who-did-what-when on any resource, and `Provenance.recorded` /
`Provenance.signature` are precisely the mechanism for the "signed"
event the reporter wants to distinguish from creation. This approach
keeps the resource lean, leans on existing infrastructure, and turns
the request into a documentation/clarification fix — which also
addresses the implementer ambiguity raised on Zulip. The trade-off
versus Disposition A is that `Provenance` is less convenient to query
than a first-class element and is inconsistently implemented in the
wild.

---

### Disposition C: Decline

#### Proposal

Decline the request. Document the resolution as: the
creation-vs-signed distinction is already representable using existing
FHIR mechanisms — `meta.lastUpdated` (or `_history`) for the moment a
draft instance first appeared on a server, `Provenance` for the
signing event, and `Task` for tracking workflow steps such as "awaiting
signature." Adding a dedicated `created` element to ServiceRequest is
not justified by the volume of demand (one ticket, no cross-references,
one stale Zulip thread) and would create overlap with both
`authoredOn` and `Provenance.recorded`.

#### Justification

The bar for adding a new element to a widely-implemented core resource
is high, and the same information can already be captured. The
reporter's primary use case — knowing how long a request has been
unsigned — is workflow tracking, which is the express purpose of
`Task`. Polling for newly arrived requests is solved by `_history` and
by FHIR Subscription. Adding a near-synonym element risks fragmenting
implementer practice between `created` and `authoredOn` with no clean
migration story.

---

### Recommendation

**Recommended disposition:** B

The ticket exposes a real definitional problem — `authoredOn` is
documented inconsistently and implementers are visibly confused about
whether it means "created" or "signed" — and that problem deserves to
be fixed. However, the cleanest fix is to nail down what `authoredOn`
means and to point implementers at `Provenance` for the signing event,
rather than to introduce a near-synonymous core element that would
permanently divide profiles and implementers between two timestamps.
Disposition B resolves both the reporter's underlying need
(distinguishing creation from signing) and the older Zulip ambiguity,
without growing the resource. If, during workgroup discussion,
implementers indicate that `Provenance` is not realistically queryable
in their environments, Disposition A becomes the correct fallback —
but the workgroup should make that call deliberately, after weighing
the cost of a new core element.
