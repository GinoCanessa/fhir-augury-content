## Summary

Eric Haas proposes removing the status code `cannot-be-obtained` from the
`Observation.status` value set. The ticket argues that this code
semantically overlaps with the already well-established
`Observation.dataAbsentReason` mechanism: both communicate "why there is
no value," but `cannot-be-obtained` does so at the resource/workflow
level while `dataAbsentReason` does so at the value level. The request
contends that the dual mechanism is redundant — the `dataAbsentReason`
pattern is extensively documented in the Observation notes and widely
implemented — and that the new status value can also produce
contradictory instances (e.g., `status = "cannot-be-obtained"` paired
with a present `valueQuantity`). The proposed resolution is simply to
remove `cannot-be-obtained` from the Observation status value set
because it is unnecessary and will confuse implementers. No Jira,
Zulip, or GitHub cross-references were surfaced for this ticket, so the
request stands on its own without explicit linked discussion.

## Details

**Description:**

### status Value "cannot-be-obtained" and dataAbsentReason Overlap

**Issue:** The status code `cannot-be-obtained` semantically overlaps
with `dataAbsentReason`. Both convey "why there's no value," but:

- `status = "cannot-be-obtained"` is a workflow/resource-level statement
- `dataAbsentReason` is a value-level statement

The current approach has been to use `dataAbsentReason`, which is
extensively documented in the Observation's notes section:

```json
{ "status": "final", "dataAbsentReason": { "coding": [ { "code": "error" } ] } }
```

This resource has been widely implemented, and there is no reason to
introduce an alternative mechanism for communicating when a value
cannot be obtained.

In addition, the `cannot-be-obtained` status value can produce
contradictory results:

```json
{ "status": "cannot-be-obtained", "valueQuantity": { "value": 5.0, "unit": "mg/dL" } }
```

**Propose:** Remove the status value `cannot-be-obtained` because it is
not needed and will cause confusion for implementers.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the code `cannot-be-obtained` from the `ObservationStatus`
value set / code system used by `Observation.status` in FHIR Core.
Concretely:

- Delete the `cannot-be-obtained` concept from the
  `ObservationStatus` code system (or retire/deprecate it if the code
  system is immutable for the current version).
- Regenerate the required-bound `ObservationStatus` value set so the
  code is no longer valid.
- In the Observation notes / status-value guidance section, delete
  any reference to `cannot-be-obtained` and strengthen the existing
  guidance that "a value could not be obtained" is expressed via
  `status = final | cancelled` plus `dataAbsentReason` (with the
  appropriate `dataAbsentReason` code such as `error`,
  `not-performed`, `unknown`, or `unable-to-obtain`).
- Add a brief migration note to the Observation change log pointing
  implementers who adopted `cannot-be-obtained` at the new code's
  introduction toward the `dataAbsentReason` pattern.

#### Justification

The reporter's core argument is sound and grounded in existing FHIR
design. `Observation.dataAbsentReason` already carries a well-scoped,
widely implemented vocabulary for "why there is no value," including
codes like `unable-to-obtain` that are nearly synonymous with
`cannot-be-obtained`. Overlapping mechanisms violate the FHIR
principle of "one way to say one thing" and create real interoperability
hazards — the ticket's own example (`status: "cannot-be-obtained"`
alongside a present `valueQuantity`) shows how the new code invites
contradictory instances that implementers must defensively validate.
The historical Zulip thread from 2019 reinforces that the community
previously settled on `cancelled` + `dataAbsentReason` for this
scenario without needing a dedicated status code. Because the code is
newly introduced, removing it now carries the lowest downstream cost.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `cannot-be-obtained` in the `ObservationStatus` value set but
disambiguate it from `dataAbsentReason` with a constraint and
clarifying guidance. Specifically:

- Add an invariant on `Observation` forbidding a value[x] (or
  component.value[x]) when `status = cannot-be-obtained` — e.g.,
  `obs-X: If status is 'cannot-be-obtained', no value[x] may be
  present (valueQuantity, valueString, etc.) on Observation or its
  components`.
- Expand the Observation notes section to draw an explicit line
  between the two mechanisms: `status = cannot-be-obtained` is used
  when the entire observation attempt failed at a workflow level (no
  specimen, test not performed, patient unavailable), whereas
  `dataAbsentReason` is used when a specific value within an otherwise
  successful observation is missing.
- Optionally tighten the `dataAbsentReason` guidance to say that
  `dataAbsentReason = unable-to-obtain` should not be used when the
  whole observation failed — that case should use
  `status = cannot-be-obtained` with no value.

#### Justification

If workgroup members or implementers already rely on
`cannot-be-obtained` as a workflow-level signal, outright removing it
is a breaking change for early adopters. A constraint-plus-guidance
path preserves the new code while eliminating the contradiction the
reporter flagged (the `status = cannot-be-obtained` + `valueQuantity`
case) and gives implementers a crisp decision rule for choosing
between the two mechanisms. This approach is defensible if the
workgroup believes the workflow vs. value-level distinction is
genuinely useful. However, it trades against Disposition A's
simplicity: implementers now have two overlapping ways to express the
same clinical reality and must learn the workgroup's preferred split.

---

### Disposition C: Decline

#### Proposal

Decline the removal request. Keep `cannot-be-obtained` as-is in
`ObservationStatus`, do not add new constraints, and rely on existing
notes guidance.

#### Justification

Declining is defensible only if the workgroup believes the new status
code fills a real gap that `dataAbsentReason` alone cannot address —
for example, distinguishing "we attempted but could not obtain"
workflow states from "the value slot is legitimately empty" at the
data level, analogous to how `cancelled` is kept distinct from
`dataAbsentReason = not-performed`. However, the reporter has
identified a concrete semantic overlap and a real contradiction
failure mode, and no cross-references or Zulip discussion were found
defending the code on its own merits. On the current evidence,
Disposition C is hard to justify and is included primarily for
completeness.

---

### Recommendation

**Recommended disposition:** A

The reporter's case is narrowly technical and strong: the
`dataAbsentReason` mechanism already covers this need, is extensively
documented, and is widely implemented; the new `cannot-be-obtained`
code duplicates it at a different level and invites contradictory
instances. Because the code is new, removal is the lowest-friction
correction — it avoids locking the specification into a two-mechanism
rule that implementers will have to learn and validators will have to
police. Disposition B is a reasonable fallback if the workgroup has
evidence (outside what the cross-reference and search data surfaced)
that adopters already depend on a workflow-level "cannot-be-obtained"
signal; in that case B's constraint plus clarifying guidance would at
least eliminate the contradiction the reporter flagged. Disposition
C should be chosen only if the workgroup is prepared to articulate
why the workflow/value-level split justifies the overlap, which the
available evidence does not support.
