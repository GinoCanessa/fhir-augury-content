## Summary

The ticket reports a FHIR build-time warning raised against the
`DeviceRequest.asNeededFor` element: *"INFORMATION: DeviceRequest.asNeededFor:
An element of type CodeableConcept or Coding must have a binding"*. The
reporter (David Barwin) has confirmed the element currently lacks a value set
binding and, after searching, did not find an existing ticket tracking the
issue. The requested action is implicit but clear — add an appropriate binding
to `DeviceRequest.asNeededFor` so that the build warning is resolved and the
element conforms to the FHIR methodology rule that every `CodeableConcept` /
`Coding`-typed element must declare a binding. No related Jira tickets, Zulip

### Disposition A: Accept As Requested

#### Proposal

Add a value set binding to `DeviceRequest.asNeededFor`. Reuse the same
binding already in place on `MedicationRequest.asNeededFor` (which covers
the analogous "reason the order is as-needed / PRN" concept) so that the
two as-needed-reason elements across the medication and device request
resources share a value set. Concretely:

- Bind `DeviceRequest.asNeededFor` to the `MedicationAsNeededReason`
  (or equivalent existing as-needed-reason) value set at **example**
  strength (matching the binding strength used on
  `MedicationRequest.asNeededFor`).
- Update the resource's structure definition and the element's
  definitional narrative to state that the binding is for documenting
  the reason a device is used on an as-needed basis.
- Re-run the build; the INFORMATION warning should disappear.

#### Justification

The FHIR methodology rule is explicit: every element typed
`CodeableConcept` or `Coding` must carry a binding, even if only at
`example` strength. `MedicationRequest.asNeededFor` already models this
exact concept and carries a binding; reusing that value set keeps the
two as-needed-reason semantics aligned across Pharmacy and O&O
resources, avoids minting a new value set for the same concept, and is
the smallest change that resolves the warning. The HCP_Call label on
the ticket implies a light-touch fix is preferred.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than reusing `MedicationRequest.asNeededFor`'s value set, mint a
new, broader **device-context** `DeviceAsNeededReason` value set (or
rename/rescope the existing as-needed-reason value set to be device-
and medication-agnostic) and bind `DeviceRequest.asNeededFor` to it at
`preferred` strength. Simultaneously retarget
`MedicationRequest.asNeededFor` at that value set so both resources
share one terminology surface.

#### Justification

As-needed reasons for device use (e.g., "during flare-ups", "on
exertion", "when symptomatic") overlap with but are not identical to
medication PRN reasons; a medication-centric value set may coerce
implementers into awkward codings. Consolidating on a single as-needed-
reason value set — and giving it a name that is not medication-
specific — improves cross-resource consistency and sets a cleaner
pattern for any future `*.asNeededFor` elements (e.g., on nutrition or
service request variants). The trade-off versus Disposition A is
scope: this change touches two resources and the terminology artifact,
and is better deferred to a normal ballot cycle than shipped as a
build-warning fix.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — Build warning is INFORMATION
only." Leave `DeviceRequest.asNeededFor` without a binding and add the
element to the build's documented INFORMATION-warning allow-list (or
accept the warning as a known, low-severity item).

#### Justification

The warning is at INFORMATION level, not ERROR, and does not block
publication. If the O&O work group cannot agree on an appropriate
value set — and there is no implementer demand to constrain the
codings used in this element — binding to a thin or speculative value
set risks doing harm (implementers coding against a value set that
doesn't match their actual reasons). Declining keeps the element
maximally permissive until a concrete terminology need is brought
forward.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the least-cost, lowest-risk change that fulfills both
the reporter's implicit ask (resolve the warning) and the FHIR
methodology rule (every CodeableConcept/Coding element must have a
binding). It aligns `DeviceRequest.asNeededFor` with the already-bound
and well-understood `MedicationRequest.asNeededFor` pattern, keeps the
cross-resource semantics consistent, and is scoped to a single resource
edit that O&O can carry without coordinating a multi-resource
terminology reshape. Disposition B is a cleaner long-term design but is
out of proportion for a build-warning fix and should be raised as a
separate methodology/terminology-harmonization ticket if the group
wants to pursue it. Disposition C leaves a methodology-rule violation
in place and is not recommended.
