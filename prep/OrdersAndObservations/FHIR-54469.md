## Summary

This is a broad, sweep-style review request asking the Orders & Observations
work group to verify that the **Device** resource (FHIR Core) is "complete and
accurate" across every facet of its definition: narrative/text content,
structural definition (element types, cardinalities, flags, descriptions,
bindings), terminology bindings and the value sets/code systems they point at,
search parameters, mappings, examples, and the set of core profiles built on
Device. The ticket does not call out a specific defect; rather, it functions
as an umbrella audit task, presumably tied to a maturity / pre-ballot review
of Device. No specific change is proposed — the deliverable is a confirmation
(or a set of follow-on tickets) that each facet has been reviewed and is fit
for purpose.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket key, so the request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a tracking ticket for an O&O-led, end-to-end audit of the
Device resource. Concretely:

1. Walk each facet listed in the description and produce a checklist
   reviewing:
   - Narrative / `text` block of the Device StructureDefinition and the
     resource page narrative.
   - Every element: type choice, cardinality, `isModifier` /
     `isSummary` / `mustSupport` flags, short/definition strings, and
     binding strength.
   - Each `ValueSet` and `CodeSystem` referenced by Device bindings,
     including binding strength (required / extensible / preferred /
     example) and whether the value set is curated by O&O or borrowed.
   - Every Device search parameter: name, expression, type, target,
     comparator/modifier support, and whether common implementer
     queries are covered.
   - All declared mappings (v2, RIM, openEHR, IEEE 11073, etc.) for
     correctness and completeness.
   - Each example instance: that it parses, that it is realistic, and
     that it exercises distinct facets of the resource.
   - Existing core profiles built on Device, confirming each is still
     justified and internally consistent.
2. For every issue found, file a child Jira ticket against
   `Orders & Observations` linked to FHIR-54469 with a specific,
   actionable change request.
3. Resolve FHIR-54469 as "Persuasive" once the audit is complete and
   the child tickets have been opened, regardless of how those child
   tickets are individually dispositioned.

#### Justification

This is exactly what the reporter asked for, and it matches the standard
HL7 pre-ballot / maturity-review pattern: an umbrella "scrub" ticket that
spawns specific, individually-dispositionable tickets. Doing the work in
one pass on a single resource is also more efficient than ad-hoc review,
because the same reviewer keeps cross-cutting concerns (e.g.,
terminology consistency between `type`, `category`, and `status`
bindings) in their head. There is no community pushback on file (no
Zulip discussion, no linked tickets), so accepting at face value is low
risk.

---

### Disposition B: Alternative Approach

#### Proposal

Convert FHIR-54469 from an open-ended audit into a **scoped review
checklist with explicit acceptance criteria and an owner**, then close
this ticket once the checklist is filed. Specifically:

1. Replace the umbrella description with a checklist that names the
   specific deliverables for each facet (e.g., "produce a table of all
   Device bindings with strength + value set canonical + last reviewed
   date"; "confirm every search parameter has at least one example
   query in the resource page").
2. Assign a single named owner from O&O for the audit and set a target
   date tied to the next ballot cycle.
3. Limit the scope of *this* ticket to the FHIR Core Device resource
   only — explicitly excluding `DeviceDefinition`, `DeviceMetric`,
   `DeviceUsage`, `DeviceAssociation`, and `DeviceDispense`, each of
   which should get its own sibling ticket if a similar review is
   wanted.
4. Disposition this ticket as "Persuasive with Modification" — accepted
   in spirit, restructured for tractability.

#### Justification

As written, FHIR-54469 is unbounded: "review everything about Device" can
mean anything from a half-day skim to a multi-month rewrite, and there
is no acceptance criterion that lets the work group decide when it is
done. Without an owner and a checklist, the ticket will likely sit in
"Triaged" indefinitely or get partially worked and then forgotten. The
Device family has grown substantially since R4 (UDI carrier, the split
between `Device` and `DeviceDefinition`, alignment with IEEE 11073);
explicitly scoping this ticket to the base `Device` resource keeps it
finishable, and naming sibling resources avoids implicit promises that
this single ticket covers them.

---

### Disposition C: Decline

#### Proposal

Close FHIR-54469 as **Not Persuasive** on the grounds that it is a
process task rather than a change request, and the same outcome is
better achieved through the existing FMM / maturity-review process and
through specific, defect-driven tickets filed by reviewers and
implementers. Direct the reporter to file individual tickets for any
concrete issues identified during their own review of Device, and to
raise systemic gaps via the O&O work-group call rather than via Jira.

#### Justification

Jira "Change Request" tickets are designed to track a specific proposed
change to the specification. FHIR-54469 proposes no change; it requests
that the work group perform a review. The HL7 process already provides
mechanisms for this (FMM advancement reviews, ballot reconciliation,
work-group standing review of resources), and tracking "review
everything" as a single Jira item creates a ticket that cannot be
cleanly resolved or audited. Declining is also defensible because no
implementer has surfaced a concrete pain point on Zulip or in linked
tickets — there is no evidence of a community-felt defect driving the
ask.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the right balance. The reporter is asking for something
genuinely useful — Device is a mature but heavily-evolved resource, and
a deliberate sweep is overdue — but the ticket as written is too
unbounded to action reliably (Disposition A's risk). At the same time,
declining outright (Disposition C) discards real work-group value: a
single coordinated review will catch cross-cutting issues (binding
inconsistencies, search-parameter gaps, mapping drift) that
defect-by-defect tickets miss. Restructuring FHIR-54469 into a scoped
checklist with a named owner and an explicit Device-only scope gives
O&O something they can actually finish, makes the outcome auditable,
and naturally produces the focused child tickets that the spec process
expects. The work group should pick the owner during the call this
ticket is reviewed in and target the next ballot cycle for completion.
