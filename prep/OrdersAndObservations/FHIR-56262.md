## Summary

The reporter asks the Orders & Observations work group to relax the
cardinality of `BodyStructure.includedStructure.morphology` from its current
0..1 to 0..*. The clinical justification is that a single body site /
structure can simultaneously exhibit several distinct morphological
characteristics — e.g., tissue changes, inflammatory features, and
malformations co-occurring at the same anatomical location. Today, only one
morphology code can be carried per `includedStructure` entry, which forces
authors to either pick one characteristic, repeat the entire
`includedStructure` block per morphology (creating ambiguity about whether
they are the same site), or stretch the meaning of a single SNOMED CT
morphology code to cover multiple findings.

There are no other cross-referenced Jira tickets and no related GitHub
items. Cross-reference resolution surfaced only the artifact mention itself.
A loosely related Zulip thread in *Orders and Observation WG* discusses how
to encode foreign bodies and cysts using `includedStructure.structure` and
`includedStructure.morphology`, which is adjacent context for understanding
how the element is being used in practice.

## Details

**Description:**

> (Abnormal) morphological characteristics of a body structure can be e.g.
> tissue changes, inflammatory features and malformations. It is possible
> that there is more than morphology characteristic at one (the same) body
> site/structure, e.g. malformations, tissue changes and inflammatory
> features could exist all at once at the same time. This requires a
> cardinality of 0..* for BodyStructure.includedStructure.morphology.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the cardinality of `BodyStructure.includedStructure.morphology` from
`0..1` to `0..*` in the FHIR Core R6 build. Update the element definition to
clarify that each repetition represents a distinct morphological
characteristic co-located at the same `includedStructure.structure`, and
that the set is intended to be unordered and additive (not a choice).
Update or add an example of a single `includedStructure` carrying multiple
morphology codes (e.g., a liver lesion that is simultaneously inflammatory
*and* a malformation), and review downstream constraints/profiles
(BodyStructure examples, IPS, imaging IGs that reference BodyStructure) for
any "exactly-one morphology" assumptions that would need updating.

#### Justification

The request is a straightforward cardinality relaxation, which is
non-breaking for existing instances (any 0..1 instance is also a valid 0..*
instance) and aligns with how morphology is modeled in source terminologies:
SNOMED CT does not require a single composite "morphologic abnormality"
concept for multi-feature lesions, and clinicians routinely co-assert
findings such as "inflammation + dysplasia" at one site. The current 0..1
forces the modeller to either pick one code, post-coordinate, or split the
`includedStructure` (which then loses the assertion that it is the *same*
site). 0..* matches existing patterns elsewhere in BodyStructure
(`includedStructure.qualifier` is already 0..*) and is consistent with how
similar concept-set elements behave on related resources (e.g.,
`Condition.bodySite`, `Specimen.subject`'s qualifying codings). The Zulip
discussion shows real authors already pairing structure+morphology in ways
that would benefit from being able to assert multiple morphology codes per
site.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `morphology` at 0..1 in the base resource, but address the
multi-characteristic use case by either (i) introducing a sibling element
`BodyStructure.includedStructure.morphologyQualifier` (0..*, CodeableConcept,
bound to a broader morphology-modifier value set) for additional features
that refine the primary morphology, or (ii) defining a standard extension
`http://hl7.org/fhir/StructureDefinition/bodystructure-additionalMorphology`
(0..* CodeableConcept) on `includedStructure` that implementers needing the
multi-finding pattern can use, with documentation explaining how to combine
it with the primary `morphology`. Either way, document the rule that one
SNOMED CT morphologic-abnormality code is "primary" and others are
modifying / co-occurring.

#### Justification

This preserves the semantic intent that `includedStructure.morphology`
identifies *the* morphologic abnormality of the structure, which simplifies
querying ("give me all BodyStructures whose morphology is 'cyst'") and
indexing — repeating `morphology` 0..* will require search-parameter and
profile authors to handle the "any of N" case explicitly. It also avoids
the open question raised in the Zulip thread about whether multiple
morphology codes describe one finding or several findings at the same site,
by giving the additional codes a distinct, modifier-flavoured slot or
extension. An extension in particular is a low-risk way to gather
implementer experience before committing to a base-resource change.

---

### Disposition C: Decline

#### Proposal

Decline the cardinality change and close as "Persuasive with Mod" / "Not
Persuasive" with the rationale that authors who need to assert multiple
co-occurring morphological features at one anatomical site should repeat
`includedStructure` (one repetition per morphology, sharing the same
`structure` code and any `bodyLandmarkOrientation` / `spatialReference`),
or — when the features are clinically a single composite finding — select a
post-coordinated SNOMED CT morphologic-abnormality expression that captures
them together.

#### Justification

`includedStructure` is itself 0..*, so the multi-finding case is already
representable without a base-resource change. Keeping `morphology` at 0..1
maintains a clear 1:1 link between an asserted body-site repetition and its
morphologic characterization, which is easier to validate, search, and
profile against. Relaxing to 0..* without further guidance leaves it
ambiguous whether a list of morphology codes represents a set of distinct
findings or alternative codings of one finding — an ambiguity that
SNOMED CT post-coordination or repeated `includedStructure` blocks already
resolve cleanly.

---

### Recommendation

**Recommended disposition:** A

The change is small, additive, and non-breaking; the clinical justification
is plausible and aligns with how multi-feature findings are routinely
recorded; and the workaround in Disposition C (repeating `includedStructure`
just to carry a second morphology) overloads `includedStructure` as a
finding container rather than a body-site container, which will hurt
downstream consumers more than it helps. Disposition B (extension /
qualifier element) is worth raising in the BodyStructure discussion that
marti_velezis flagged, but adding a separate "additionalMorphology" slot is
harder to justify than simply allowing the existing element to repeat —
especially given that `includedStructure.qualifier` is already 0..* and sets
the precedent. The work group should pair the cardinality change with a
short normative note on how to interpret multiple morphology codes (set of
co-occurring findings at the same site, not alternative codings of one
finding) and review existing examples / profiles for assumptions that need
updating.
