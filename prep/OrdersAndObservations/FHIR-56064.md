## Summary

The ticket reports a FHIR build QA message against `DeviceDefinition.packaging.type`:
"An element of type CodeableConcept or Coding must have a binding." Because
`packaging.type` is currently declared as a `CodeableConcept` with no value-set
binding, the publisher emits an INFORMATION-level QA issue. The reporter asks
the Orders & Observations work group to add an appropriate binding so that this
element conforms to the FHIR rule that coded elements must declare a binding,
even if the binding is `example`.

This ticket is a per-element follow-up to [FHIR-40530](https://jira.hl7.org/browse/FHIR-40530)
("QA Fixes to structure - DeviceDefinition"), which rolled up all of the
DeviceDefinition publisher QA messages (including this `packaging.type` binding
message). FHIR-40530 was closed "Persuasive with Modification" with the motion
to fix what could be fixed and file individual tickets for the remaining items;
FHIR-56064 is one of those individual spin-outs and is scoped specifically to
the `packaging.type` binding.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an **example** binding to `DeviceDefinition.packaging.type` pointing at a
newly-authored value set, e.g. `http://hl7.org/fhir/ValueSet/device-packaging-type`,
seeded with common package forms used in device distribution (e.g.,
`box`, `blister-pack`, `tray`, `sterile-pouch`, `cassette`, `kit`, `bag`,
`bottle`, `carton`, `case`). The value set would reference a companion
CodeSystem at `http://hl7.org/fhir/device-packaging-type`, authored alongside
the resource (conventionally in `source/devicedefinition/` and
`source/valuesets*`). The binding strength would be `example`, matching the
default HL7 convention for descriptive, open-ended codeable concepts where
the WG does not want to constrain implementers.

#### Justification

FHIR publisher rules require every `CodeableConcept`/`Coding` element to
declare a binding; this INFORMATION message will otherwise persist on every
build. An `example` binding is the lightest-weight fix that (a) satisfies the
tooling rule, (b) gives implementers a concrete starting point, and (c)
imposes no conformance burden on systems that use their own package
terminology. It is consistent with how the parent ticket FHIR-40530 was
already dispositioned ("Persuasive with Modification" — add per-item tickets
and address them), and with how comparable loosely-typed `CodeableConcept`
elements have historically been bound in FHIR Core.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than authoring a new HL7-owned value set, bind
`DeviceDefinition.packaging.type` with **strength `example`** to an existing
external terminology that already enumerates packaging forms — for example
SNOMED CT's "Packaging (physical object)" hierarchy (SCTID 706041008 and
descendants) via a `http://hl7.org/fhir/ValueSet/device-packaging-type`
value set whose compose includes that SNOMED reference set. Alternatively,
bind to the UNECE Recommendation 21 package type codes (used widely in
supply-chain contexts). In either case the binding strength remains
`example`, and no HL7-maintained CodeSystem is created.

#### Justification

The FHIR community generally prefers reusing existing terminology over
minting new HL7 code systems when the domain is already well covered
externally. Device packaging is a supply-chain / regulatory concept (GS1,
UDI, UNECE) rather than a clinical one, so a bespoke HL7 code system adds
maintenance burden without adding coverage. An `example` binding to SNOMED
or UNECE preserves implementer freedom while pointing at a recognized,
maintained source. The trade-off vs. Disposition A is slightly more review
effort up front to pick the right external code system, but lower
long-term maintenance for HL7.

---

### Disposition C: Decline

#### Proposal

Do not add a binding; instead, **suppress** the INFORMATION-level QA message
for `DeviceDefinition.packaging.type` via the DeviceDefinition build QA
exception mechanism (parallel to how other DeviceDefinition QA messages are
suppressed in the `devicedefinition-*-mapping-exceptions.xml` / build exclusion
lists), on the grounds that `packaging.type` is a free-text-style classifier
for which no useful terminology exists and the publisher rule is advisory
only.

#### Justification

The QA message is an INFORMATION, not a WARNING or ERROR, and it does not
block publication. If the WG judges that no existing vocabulary is fit for
purpose and that an invented HL7 code system would be little more than a
string enumeration, declining is defensible — the cost of creating and
maintaining an HL7 CodeSystem for a handful of package form labels may
outweigh the benefit. This is the weakest option because the ticket is
explicitly flagged `HCP_Call` (i.e., surfaced through the HL7 community
hot-list), and leaving a known INFO message in place has historically been
contentious with implementers who expect every coded element to have at
least an `example` binding.

---

### Recommendation

**Recommended disposition:** A

Accepting the request with an `example`-strength binding to a new
`device-packaging-type` value set is the cleanest way to clear the QA
message, aligns with how the parent ticket FHIR-40530 was already
dispositioned (spin out per-item tickets and address them), and matches
the lightweight pattern used elsewhere in FHIR Core for open-ended
`CodeableConcept` classifiers. Disposition B is attractive but forces a
choice among external code systems (SNOMED vs. UNECE vs. GS1) that the WG
may not want to litigate inside a QA-cleanup ticket; it can be folded into
A later by updating the value set's compose. Disposition C leaves a
permanent INFO on every build and conflicts with the `HCP_Call`
signaling that this is an implementer-visible issue the WG should close
rather than suppress.
