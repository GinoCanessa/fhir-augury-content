## Summary

The reporter (BenEaton) finds the definition, requirements, and comments
attached to `ObservationDefinition.name` ambiguous and is asking O&O to clarify
its intended meaning and uniqueness expectations. Two specific points of
confusion are raised: (1) what it means for the `name` value to be "usable for
code generation" — e.g., is it intended to be an identifier that conforms to
the lexical rules of an OO programming language class/identifier name? — and
(2) what the uniqueness scope of `name` is meant to be: globally unique within
a system across all `ObservationDefinition` instances, or unique only in the
weaker sense that `ObservationDefinition.identifier.value` is unique. The
ticket does not propose a specific spec edit; it is a request that O&O tighten
or rewrite the element's `definition`, `requirements`, and `comments` so that
implementers and profile authors have an unambiguous contract. There are no
explicit cross-references on the ticket, no linked GitHub PRs/issues, and no
substantive Zulip discussion focused on this question.

## Details

**Description:**

> I am confused by the definition, requirements, and comments of ObservationDefinition.name.
>
> What does it mean for an identifier to be usable for code generation? Perhaps you could give an example, is it like an object-oriented programming language's class name?
>
> Should the name be unique from all other ObservationDefinition's names within one system, or unique in a way similar to ObservationDefinition.Identifier.value?

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a documentation-only clarification and rewrite the
`definition`, `requirements`, and `comments` fields of
`ObservationDefinition.name` (the StructureDefinition element) to answer
both of the reporter's questions explicitly. Concretely:

1. **Definition** — replace the existing prose with something along the
   lines of: *"A short, human- and machine-friendly token that names this
   ObservationDefinition. It is intended to be a stable symbolic handle
   (similar in spirit to `CodeSystem.name` or `StructureDefinition.name`)
   that implementers can use as a variable, class, or constant name when
   generating code or referring to the definition in scripts. It is not a
   display label — use `title` for that — and it is not the canonical
   business identifier — use `identifier` for that."*
2. **Requirements** — state the intended uniqueness scope. Adopt the
   MetadataResource convention: `name` SHOULD be unique within the scope
   of a single publishing namespace/system, and SHOULD conform to the
   pattern `[A-Z]([A-Za-z0-9_]){0,254}` so it is usable as a token in
   common programming languages. Note that `identifier` remains the
   authoritative business identifier (URI/system+value) and that two
   `ObservationDefinition` resources with different `identifier`s SHOULD
   NOT share a `name` within the same publisher.
3. **Comments** — give one concrete example, e.g. an
   `ObservationDefinition` for "Systolic blood pressure" with
   `name = "SystolicBloodPressure"` and `title = "Systolic Blood
   Pressure"`, and explicitly contrast it with `identifier` and `code`.

No element cardinalities, types, or constraints would change.

#### Justification

The reporter is explicitly asking for clarification, not a structural
change, so a documentation-only edit is the lowest-risk way to resolve
the ticket. The proposed text aligns `ObservationDefinition.name` with
the well-established `MetadataResource.name` convention used by
`StructureDefinition`, `ValueSet`, `CodeSystem`, and similar
canonical/definitional resources, where `name` is the
"machine-friendly" token and `title` is the human-friendly label. This
keeps `ObservationDefinition` consistent with the rest of FHIR's
canonical resources and addresses both ambiguities in one pass without
breaking any existing instance data.

---

### Disposition B: Alternative Approach

#### Proposal

Go beyond pure documentation and additionally:

- Add an invariant on `ObservationDefinition` that constrains `name` to
  the canonical token pattern `[A-Z]([A-Za-z0-9_]){0,254}` (the same
  invariant that `MetadataResource` applies to `name`), with severity
  `warning` so existing instances are not invalidated.
- Mark `name` as the recommended search/sort key for code-generation
  tooling, and add a short cross-reference in the `ObservationDefinition`
  resource narrative to the "Defining Resources" / canonical-resource
  naming guidance page.
- If the resource is not already deriving from `MetadataResource`,
  either align the element definition by reference, or add a note that
  the same rules are intentionally mirrored.

The clarifying prose from Disposition A is still applied; this option
adds machine-checkable expectations on top.

#### Justification

A purely textual clarification leaves implementers free to keep
populating `name` with arbitrary strings (spaces, punctuation, display
phrases), which is the underlying reason the reporter is confused. A
warning-level invariant turns the documented expectation into something
validators and IG publishers can surface, while still being
backward-compatible with existing instances. The trade-off vs.
Disposition A is a small amount of additional implementation work and
the need for a brief tracker/ballot note explaining the new constraint;
the upside is that future questions of this kind become self-answering.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Modifier — No Change Required",
on the grounds that `ObservationDefinition` derives the same
expectations for `name` that all other canonical/definitional resources
inherit from `MetadataResource`, and those expectations are documented
on the canonical-resources page. Respond to the reporter pointing them
to the `MetadataResource.name` definition and to the FHIR canonical
resources naming guidance, rather than editing the
`ObservationDefinition` element.

#### Justification

If the workgroup considers the existing cross-resource guidance
sufficient, restating it on every individual resource risks drift and
maintenance burden. Declining with a pointer to the shared guidance
preserves a single source of truth for `name` semantics across all
canonical resources.

---

### Recommendation

**Recommended disposition:** A

The reporter's questions are real and answerable with text edits alone
— a reviewer reading just the current `ObservationDefinition.name`
element will not be able to tell whether `name` is meant to be a
programming-language token, a display label, or a uniqueness
constraint, and that ambiguity is a defect in the resource
documentation regardless of what `MetadataResource` says elsewhere.
Disposition A clears the ambiguity in one place implementers will
actually look (the resource page itself), aligns the language with the
established canonical-resource convention, and carries essentially zero
risk to existing instance data or downstream profiles. Disposition B is
attractive but is best raised as a follow-up once the prose is settled,
because adding an invariant — even at `warning` severity — is a
behavior change that deserves its own discussion and tracker. Disposition
C under-serves the reporter: the very fact that this ticket was filed
is evidence that the cross-resource guidance is not discoverable from
the `ObservationDefinition` page alone.
