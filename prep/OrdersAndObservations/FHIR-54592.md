## Summary

Ulrike Merrick is asking the Orders & Observations work group to clarify the
guidance on the `ObservationDefinition` resource page that draws a boundary
between `ObservationDefinition` and `Questionnaire`. The current text states
that "ObservationDefinitions are only to be used when describing the contents
of an observation (including observations in order catalogs or panels). When
additional structure, sequence, or rules are needed, the Questionnaire
resource should be used." The reporter argues this implication is misleading
for the order-catalog use case: panels with reflex tests do involve sequence
and rules across tests, but `Questionnaire` cannot reference
`SpecimenDefinition` or `DeviceDefinition`, so it is not actually a viable
substitute for catalog-style structures. The ticket therefore requests
better — and more accurate — boundary guidance between the two resources,
particularly with respect to order catalogs, panels, and reflex testing. No
related Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-reference or search.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Revise the ObservationDefinition boundaries / scope text on the
`ObservationDefinition` resource page (the prose containing the quoted
sentence above) to:

1. Drop the implication that `Questionnaire` is the correct fallback
   whenever "additional structure, sequence, or rules are needed."
2. Explicitly carve out the order-catalog use case — including panels and
   reflex testing — and state that these belong with
   `ObservationDefinition` (and the surrounding catalog
   resources: `PlanDefinition`, `ActivityDefinition`,
   `SpecimenDefinition`, `DeviceDefinition`), not with `Questionnaire`.
3. Re-scope the `Questionnaire` recommendation to its actual sweet spot:
   data capture forms where the structure, branching, or conditional
   logic is about *what to ask a human and how to render it*, not about
   how a laboratory orders, sequences, or reflexes tests.
4. Note the concrete capability gap the reporter identifies — that
   `Questionnaire` items cannot reference `SpecimenDefinition` or
   `DeviceDefinition` — as part of why it is not a substitute for
   catalog/orderable definitions.

#### Justification

The reporter's underlying observation is factually correct:
`Questionnaire.item` does not support typed references to
`SpecimenDefinition` or `DeviceDefinition`, so it cannot represent a
reflex-test panel that needs to bind specimen or device requirements.
Boundary text on a definitional resource should not point implementers
at a resource that demonstrably cannot model the use case it is being
recommended for. Tightening the prose is a low-risk, documentation-only
change that improves accuracy and reduces the chance of implementers
mis-modeling order catalogs as `Questionnaire`s.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than merely tightening the prose on `ObservationDefinition`,
introduce (or expand) a dedicated **Boundaries and Relationships** /
**Scope** subsection that:

- Lays out a small decision matrix of "what are you modeling?" →
  "use which resource(s)?" covering `ObservationDefinition`,
  `Questionnaire`, `PlanDefinition`/`ActivityDefinition`, and
  `SpecimenDefinition`/`DeviceDefinition`.
- Explicitly addresses panels, reflex testing, and orderable
  catalogs as worked examples (these are recurring O&O questions).
- Cross-links from the equivalent boundaries section on
  `Questionnaire` so the guidance is discoverable from both sides.
- Leaves the existing one-liner in place only as a one-sentence
  pointer into the new subsection.

#### Justification

The reporter's confusion is symptomatic of a recurring boundary
question between definitional/orderable resources and form-capture
resources, not just a single bad sentence. A worked-example
boundaries section addresses the root cause across more than one
ticket's worth of confusion and is consistent with how other FHIR
resources document overlapping scope (e.g.,
`Observation` ↔ `DiagnosticReport`,
`Procedure` ↔ `Observation`). The trade-off is a larger editorial
effort and a wider review surface than Disposition A; if the work
group is short on cycles for the current cycle, this can be
sequenced as a follow-up while still applying Disposition A now.

---

### Disposition C: Decline

#### Proposal

Leave the existing boundaries text on `ObservationDefinition`
unchanged on the grounds that the quoted sentence is intended to
caution implementers away from over-loading `ObservationDefinition`
with workflow logic, and that order-catalog reflex modeling is
already covered elsewhere by `PlanDefinition` /
`ActivityDefinition`.

#### Justification

A defensible decline would rest on two claims: (1) the existing
sentence is talking about *behavioral* logic inside a single
observation rather than catalog-level reflex sequencing, and
(2) reflex/sequence behavior is properly the job of
`PlanDefinition`/`ActivityDefinition`, not `ObservationDefinition`
or `Questionnaire`. Under that reading, the current text is
correct as written and the fix is reader interpretation rather
than spec change.

---

### Recommendation

**Recommended disposition:** A (with B as a follow-up if cycles
allow)

The reporter has identified a concrete, verifiable problem: the
current boundary text recommends `Questionnaire` for cases
(reflex panels in order catalogs) where `Questionnaire` cannot
reference `SpecimenDefinition` or `DeviceDefinition` and so
cannot actually model the use case. That makes Disposition C
hard to defend on its merits — the existing text is, at minimum,
ambiguous enough to mislead a careful implementer. Disposition A
is a low-risk, documentation-only change that directly addresses
the reported confusion and is well-scoped for a single ballot
cycle. Disposition B is the better long-term answer but is
heavier and broader in review surface; the work group can adopt
A now and queue B as a follow-up clarifying ticket once the
narrower wording fix is in. Because there are no Zulip threads,
linked tickets, or GitHub PRs surfaced for this ticket, the
work group should also confirm during review that no other
in-flight change is already touching this same prose.
