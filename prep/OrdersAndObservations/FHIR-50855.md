## Summary

FHIR-50855 is a short, comment-type ticket filed by Ralf Herzog against the
Clinical Order Workflow (COW) IG asking that the spec provide **two distinct
constructs** for **Fulfillment Cancellation** versus **Authorization
Cancellation**, instead of folding both use cases into the single existing
Cancellation Request Task profile. The ticket is paired with two siblings
filed in the same WGM-May2025 batch by the same reporter and with closely
related design feedback:

- [FHIR-50854](https://jira.hl7.org/browse/FHIR-50854) — "Placer Initiated
  Cancellations should always use a CancelationRequest task" (closed as
  *Duplicate / Not Persuasive with Modification*). Ralf's own follow-up
  comment on FHIR-50855 ("Is related to the other but not a duplicate")
  almost certainly refers to FHIR-50854; the WG split the two design
  questions deliberately.
- [FHIR-50809](https://jira.hl7.org/browse/FHIR-50809) — "Feedback about
  STU Note" (Persuasive). Reviewer Alan Zhang argued the *opposite*
  position: keep one Cancellation Request Task profile because two
  profiles for a "minus difference" is heavier than the value gained.
  COW agreed on 2025/08/08 to keep a single profile.

A later, closely-related Orders & Observations ticket
[FHIR-56051](https://jira.hl7.org/browse/FHIR-56051) (Applied) generalises
the Task pattern further so that *any* placer/fulfiller request to change
the status of an authorization (cancel, complete, etc.) is conveyed via a
Task — explicitly using the existing Cancellation Request Task as the
template. That direction is incompatible with adding a second
distinct-cancellation profile.

The ticket itself was eventually marked **Retracted** on 2025-08-12 — the
substantive WG decision lives on FHIR-50809.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Split the existing **`cancellation-request-task`** profile in
`HL7/fhir-cow-ig` into two sibling Task profiles:

- `fulfillment-cancellation-request-task` — used when a placer or
  fulfiller is requesting that fulfillment work stop, but the underlying
  authorization remains valid (e.g., today's session is cancelled but
  the standing order endures).
- `authorization-cancellation-request-task` — used when the placer is
  asking the fulfiller (or vice-versa) to revoke the authorization
  itself (the underlying Request goes to `revoked` / `entered-in-error`).

Add a `Task.code` (or `Task.statusReason`) binding distinguishing the two
intents, update section 6.5.2 ("Fulfiller request to cancel the
Request"), refresh examples, and replace the STU Note flagged on
FHIR-50809 with normative guidance.

#### Justification

It is consistent with Ralf's framing that "fulfillment cancellation" and
"authorization cancellation" are semantically distinct events with
different downstream effects (residual obligations, billing, audit) and
warrant explicit profiles to reduce ambiguity at runtime. Two distinct
profiles give implementers a clearer contract and validators a sharper
target. However, this disposition runs against the Zulip and FHIR-50809
consensus that the difference is "minus" and against FHIR-56051's
generalisation direction — those tensions need to be reconciled before
adoption.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a **single** `cancellation-request-task` profile, but make the
fulfillment-vs-authorization intent **explicit on the instance** rather
than at the profile level:

- Define a small required-binding code (e.g.,
  `cancellation-scope` with values `fulfillment-only` and
  `authorization`) carried in `Task.code` *or* a typed extension on
  the existing profile.
- Tighten the STU Note in §6.5.2 ("Fulfiller request to cancel the
  Request") into normative text that explains both scopes and how
  `Request.status` (`revoked` vs `on-hold`/`completed`) should change in
  each case.
- Add at least one worked example per scope (radiologist rejecting
  *only this fulfillment* vs radiologist *revoking the authorization*),
  re-using the use case Jose described in the Zulip thread.

This addresses the underlying need — implementers must be able to
distinguish the two cancellation flavours — without doubling the
profile surface area or fragmenting the existing Cancellation Request
Task ecosystem. It also aligns with FHIR-56051's direction that Task is
the general mechanism for requesting status changes on authorizations.

#### Justification

It honours the COW WG's 2025/08/08 conclusion ("we tend to have only
one") and Alan Zhang's argument in FHIR-50809 that two profiles for a
"minus difference" is overweight, while still solving Ralf's actual
interoperability concern — that a recipient cannot today reliably tell
fulfillment-only from authorization-revoking cancellations from the
profile alone. Carrying the distinction as a coded element keeps the
existing profile, examples, and the FHIR-56051 generalisation path
intact. The race-condition concerns surfaced in FHIR-50840 / FHIR-50854
remain handled by the existing CancelationRequest Task discipline.

---

### Disposition C: Decline

#### Proposal

Decline the request and **leave the resolution as Retracted**. The
substantive design question has already been answered by the WG via
FHIR-50809 (Persuasive — keep one profile), is being further generalised
by FHIR-56051 (Applied — Task is the default mechanism for
authorization-status-change requests), and Ralf himself retracted the
ticket on 2025-08-12. No further normative change is needed beyond what
is already underway in the related tickets.

If reviewers feel implementers still need help disambiguating
fulfillment vs authorization scope, fold a clarifying paragraph into the
guidance work already accepted under FHIR-56051 rather than re-opening
this ticket.

#### Justification

The ticket is already Retracted, the WG has reached and recorded the
opposite conclusion in a sibling ticket (FHIR-50809), and the
generalisation in FHIR-56051 actively moves COW toward *fewer*
specialised cancellation profiles, not more. Re-opening would
re-litigate a closed WG decision and risks creating an inconsistency
with FHIR-56051's "Review Change To Authorization" direction.
Maintaining the status quo also avoids the breaking change that
splitting the profile would impose on existing implementers and on
content already published in the 2025May COW snapshot.

---

### Recommendation

**Recommended disposition:** C (with a fallback to B if reviewers want
to do *anything* on this ticket).

The narrative across FHIR-50809, the Zulip "Intent-driven interaction"
thread, and FHIR-56051 is consistent and recent: COW intends to keep a
single Cancellation Request Task profile and is in fact generalising
that pattern outward to cover other authorization-status-change requests.
Ralf retracted the ticket himself, and the parallel WG decision on
FHIR-50809 (Persuasive — "We tend to have only one") is already applied.
Splitting the profile now would contradict an active applied resolution
in the same workgroup and the same IG, and would impose a breaking
change on early implementers.

If the workgroup nevertheless agrees that implementers cannot today
distinguish "fulfillment-only" from "authorization-revoking"
cancellations from the wire, prefer Disposition B over A: add a coded
`cancellation-scope` element (or extension) and tighten §6.5.2's STU
Note into normative guidance, ideally as a follow-on to FHIR-56051 so
the two changes ship as one coherent story. Avoid Disposition A — it
fragments a profile the WG has just deliberately kept whole.
