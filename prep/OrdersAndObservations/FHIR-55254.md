## Summary

The ticket asks the Orders & Observations work group to evaluate three
DeviceAssociation-owned terminology artifacts as candidates for migration
to Terminology.HL7.org (THO):

- `CodeSystem/deviceassociation-relationship` and the bound
  `ValueSet/deviceassociation-relationship` (used by
  `DeviceAssociation.relationship`).
- `CodeSystem/deviceassociation-status-reason` and the bound
  `ValueSet/deviceassociation-status-reason` (used by
  `DeviceAssociation.statusReason`).
- `CodeSystem/deviceassociation-association-status` (a.k.a.
  `DeviceAssociationAssociationStatusCodes`) and the bound
  `ValueSet/deviceassociation-association-status` (used by
  `DeviceAssociation.associationStatus`).

The ticket is part of the broader THO-candidate sweep: value sets whose
definitions are likely to evolve between releases, or that may also be
useful in CDA, V2, or other specs, should be considered for migration to
THO so they can be versioned and reused outside the FHIR Core release
cycle. The submitter (Lloyd McKenzie) does not require migration, only
that the work group consider it.

The ticket has already been resolved as **Persuasive with Modification**.
The captured resolution is to **move all three code systems to THO while
leaving the value sets in FHIR Core unchanged**. A pulled-from-block-vote
comment by Elliot Silver flags that DeviceAssociation's terminology is
still expected to be cleaned up before next ballot, and that the cleanup
itself may shift bindings and the question of what should ultimately move.

No cross-referenced Jira tickets were returned for FHIR-55254.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate **all three** DeviceAssociation-owned terminologies to THO
end-to-end:

- Move `CodeSystem/deviceassociation-relationship`,
  `CodeSystem/deviceassociation-status-reason`, and
  `CodeSystem/deviceassociation-association-status` from FHIR Core into
  THO under the appropriate `CodeSystem-*` THO authoring location, with
  canonical URLs rebased to `http://terminology.hl7.org/CodeSystem/...`.
- Move the bound `ValueSet/deviceassociation-relationship`,
  `ValueSet/deviceassociation-status-reason`, and
  `ValueSet/deviceassociation-association-status` to THO as well, since
  the ticket frames the value sets (not just the code systems) as the
  THO candidates and notes potential CDA/V2 reuse — which is value-set,
  not code-system, level reuse.
- Update the bindings on `DeviceAssociation.relationship`,
  `DeviceAssociation.statusReason`, and
  `DeviceAssociation.associationStatus` in FHIR Core to point at the new
  THO canonicals, leaving binding strength unchanged.
- Coordinate the THO publication so the canonicals exist before the FHIR
  Core build references them, and add a deprecation/redirect note for
  the old FHIR-Core canonicals.

#### Justification

This is the literal request and matches the THO-candidate program intent
that artifacts "useful in CDA, V2 or other specs" live in THO. Moving
both the code systems and the value sets gives downstream specs a single
canonical reuse point and avoids the awkward split where a value set in
FHIR Core composes a code system in THO. It is also internally
consistent: all three concepts get the same treatment, which is easier
for implementers to reason about than per-artifact carve-outs.

---

### Disposition B: Alternative Approach (Recommended Modification)

#### Proposal

Adopt the resolution already captured on the ticket: **move only the
three code systems to THO**, and **leave the three value sets in FHIR
Core (DeviceAssociation R6) unchanged in location**, simply re-pointing
their `compose.include.system` to the new THO code system canonicals.
Bindings on `DeviceAssociation.relationship`,
`DeviceAssociation.statusReason`, and `DeviceAssociation.associationStatus`
remain pointed at the existing FHIR-Core value set canonicals.

In addition, sequence the work so that Elliot Silver's flagged
DeviceAssociation terminology cleanup lands **before** the THO move:
finalize the codes, descriptions, and binding strengths in FHIR Core
where iteration is cheap, and only then transplant the stabilized code
systems into THO.

#### Justification

This is a smaller, lower-risk change than Disposition A and matches the
work group's already-recorded resolution. Code systems are the part most
likely to be reused cross-spec (V2, CDA, IGs) and most expensive to
move later, so moving them is high value. Value sets, by contrast, are
tightly coupled to the DeviceAssociation element semantics and have not
been called out as needing CDA/V2 reuse, so the marginal benefit of also
moving them is low. Doing the cleanup-then-move sequence directly
addresses Elliot Silver's concern that early migration would freeze
terminology that is still in flux and may cause churn in THO version
history and in any downstream binders.

---

### Disposition C: Decline

#### Proposal

Decline the migration for this ballot cycle. Keep
`deviceassociation-relationship`, `deviceassociation-status-reason`,
and `deviceassociation-association-status` (both code systems and value
sets) in FHIR Core under DeviceAssociation, and revisit only after
DeviceAssociation has matured and the terminology has stabilized.

#### Justification

DeviceAssociation is still relatively new and its terminology is
explicitly flagged as likely to change before next ballot. Migration
under those conditions risks publishing churning content in THO, where
versioning is more visible and where downstream specs may pin specific
versions. The submitter explicitly states migration is not required, and
no cross-spec (CDA/V2) demand for these specific codes has been
surfaced in cross-references or Zulip. Deferring keeps the cost of
iteration inside FHIR Core where it currently belongs.

---

### Recommendation

**Recommended disposition:** B

Disposition B aligns with the resolution already captured on the ticket
("Persuasive with Modification") and directly addresses the only
substantive comment on file — Elliot Silver's concern that the
DeviceAssociation terminology is still in flux. Moving the code systems
to THO captures the cross-spec reuse benefit the THO-candidate program
exists for, while leaving the value sets in FHIR Core preserves the
tight coupling between DeviceAssociation element semantics and the
selected codes, and keeps iteration cheap during R6 stabilization.
Disposition A overreaches given the cleanup still pending, and
Disposition C forfeits the reuse benefit that motivated the ticket in
the first place. The work group should confirm the cleanup-then-move
sequencing and assign an owner for the THO submission.
