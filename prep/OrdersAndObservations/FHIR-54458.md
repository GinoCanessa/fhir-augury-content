## Summary

Eric Haas is asking that the Observation resource's newly added `organizer`
element (introduced in the R6 3rd Draft ballot cycle and currently a Normative
candidate in R6) be renamed from **"organizer"** to **"grouper"**. His
rationale is that "organizer" is a CDA-specific term that connotes structured,
often hierarchical containment, whereas the FHIR use of the element is
explicitly a **flat** grouping of sub-observations (e.g., lab panels/batteries
linked via `hasMember`). He argues that "grouper" more accurately denotes the
actual behavior of the element. No Jira cross-references were surfaced by the
augury index for this ticket, but the request is closely related to
[FHIR-53553](https://jira.hl7.org/browse/FHIR-53553) (R6 Ballot Notes:
Observation, Applied / Persuasive), which introduced the balloter note calling
out `Observation.organizer` as one of the elements now being balloted for
Normative. That ballot note — and the existing Observation narrative and
definitions — currently use the phrase "organizer/grouper", which is the
linguistic artifact this ticket is trying to clean up.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rename `Observation.organizer` to `Observation.grouper` across the
Observation resource definition and all of its supporting spec material,
before R6 Normative lock-in. Concretely, in the HL7/fhir core spec:

- Rename the element `Observation.organizer` → `Observation.grouper` in
  the Observation StructureDefinition (path, id, short, definition,
  comment).
- Update invariant `obs-11` (currently: "if organizer exists and
  organizer = true, then value[x], dataAbsentReason and component SHALL
  NOT be present") to reference `grouper` in both the human text and the
  FHIRPath expression, and rename the invariant's test fixtures
  (`obs-11.*.xml`) accordingly.
- Update the Observation narrative / introduction page (currently
  describes "Observation.organizer" as one of the Normative-candidate
  additions) to use "grouper" throughout.
- Update the R6 ballot note from FHIR-53553 to refer to the renamed
  element and explain the rename rationale.
- Rename the example `observation-example-lipidpanel-organizer.xml` (and
  the example's id/references from the resource list) to
  `...-lipidpanel-grouper.xml`, and update any cross-links.
- Add a rename entry to the R6 change log / diff-from-R5 page and update
  any cross-resource comments that previously said "organizer/grouper".

#### Justification

The element was added only in the most recent R6 draft cycle, so renaming
now avoids locking a suboptimal name into Normative. The reporter's
linguistic point is accurate: in CDA, `organizer` is a structural,
potentially nested act-relationship container, while the FHIR element is
a flat boolean flag that marks a parent Observation as the "header" of a
`hasMember` grouping. Aligning the name with behavior (`grouper`) removes
cross-paradigm confusion for implementers coming from CDA and matches how
the spec already explains the element ("organizer *or grouper*", "an
organizer/grouper observation…"). Doing the rename now, while the
element is still a Normative candidate rather than Normative, minimizes
implementer disruption — the R5 ballot did not ship this element, so
there is no installed base of production FHIR implementations keyed to
`organizer` yet.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the element name `Observation.organizer` (do not rename), but tighten
the supporting material so that the CDA connotation is explicitly
disavowed. Specifically:

- Update the `Observation.organizer` **definition** and **comment** to
  lead with a phrase such as "A boolean flag indicating this Observation
  is the *grouper* (a flat parent) of a set of sub-observations linked
  via `hasMember`. This element is unrelated to the CDA `organizer` act
  and does not imply hierarchy or structured containment."
- Replace the mixed phrase "organizer/grouper" with a single consistent
  phrasing throughout the Observation narrative and the R6 ballot note,
  and add a glossary / mapping-notes entry contrasting FHIR's flat
  grouper semantics with the CDA `organizer` act.
- Optionally add the term `grouper` as a published synonym/alias in the
  element's comment and in the resource introduction so search and
  cross-paradigm readers land in the right place.

#### Justification

A rename, even of a non-Normative candidate, is a breaking change for
every implementation, profile, example, and downstream IG that has
already picked up the R6 3rd Draft element, plus every tool chain that
has already code-generated against it. The confusion the reporter calls
out is real but is primarily a *documentation* problem — the definition
and comment can make the "flat, not hierarchical, not CDA-equivalent"
semantics unambiguous without touching the wire name. This disposition
costs less implementer churn and preserves continuity with the element
already balloted in the R6 3rd Draft, while still resolving the root
concern (readers importing CDA expectations).

---

### Disposition C: Decline

#### Proposal

Decline the rename. Retain `Observation.organizer` as-is, on the grounds
that (a) the element has already been balloted in the R6 3rd Draft and
is a Normative candidate in the current cycle, (b) the existing
definition/comment already explicitly use the phrase "organizer or
grouper" and describe flat `hasMember` semantics, and (c) the request
does not identify a concrete interoperability failure — only a
stylistic/linguistic preference.

#### Justification

"Organizer" is a well-understood, domain-neutral English word for the
parent of a grouping; it is not exclusively a CDA term, even though CDA
uses it. The R6 3rd Draft ballot already introduced this name with the
flat-grouping semantics and the community passed it through that ballot
cycle (see FHIR-53553, which reached *Persuasive*). Renaming a
Normative-candidate element between ballot cycles on stylistic grounds
sets a weak precedent and imposes churn on every early adopter, profile
author, and code generator. If the CDA-connotation concern is real for a
meaningful segment of implementers, the narrative can address it (see
Disposition B) without a wire-level rename.

---

### Recommendation

**Recommended disposition:** A

The window to rename this element with minimal downstream harm is
precisely *now*: it was added only in the R6 3rd Draft, it is not yet
Normative, and the spec's own prose already hedges by writing
"organizer/grouper" and "organizer or grouper" — a strong signal that
the authors themselves did not feel "organizer" alone was
self-explanatory. The reporter's objection is substantive for O&O's
primary audience (lab/clinical observation implementers), many of whom
work in dual CDA/FHIR environments and will import the CDA hierarchical
connotation of `organizer` by default. Fixing that through definition
text alone (Disposition B) leaves a misleading element name permanently
on the wire and in every generated artifact, which is a worse long-term
outcome than a one-cycle rename before Normative lock-in. Declining
(Disposition C) is defensible procedurally but forfeits the last cheap
opportunity to align the name with the behavior. The workgroup should
weigh the churn on early R6-draft adopters against a permanent naming
mismatch going into Normative, and on that balance a rename to `grouper`
is the better call.
