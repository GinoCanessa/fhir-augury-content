## Summary

The ticket asks that `DeviceDefinition.regulatoryIdentifier.identifier` be
changed from 1..1 to a repeating cardinality. The reporter's rationale is
that identifiers "can almost always be assigned by multiple entities over
time," and the current single-cardinality element does not accommodate
that pattern. No cross-referenced Jira tickets were surfaced for this
issue, so the request stands on its own; the scope is limited to the
cardinality of a single element on `DeviceDefinition`. The ticket has
already been discussed on the 2026-03-09 HCP call, and a "Not Persuasive"
resolution has been drafted by O&O on the grounds that the parent
backbone element already repeats and that the sibling context elements
(`type`, `jurisdiction`, etc.) pin a single identifier value per
backbone instance.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the cardinality of `DeviceDefinition.regulatoryIdentifier.identifier`
from `1..1` to `0..*` (or `1..*`) in the base `DeviceDefinition` resource
StructureDefinition. The element would then carry a list of `Identifier`
values that all share the same sibling `type`, `jurisdiction`, and
associated context already captured on the `regulatoryIdentifier`
backbone. No other schema elements change; downstream profiles and IGs
that currently assume a single value would either remain valid (upper
bound relaxation) or need to tighten cardinality in their own profiles.

#### Justification

The `Identifier` datatype is used across FHIR to represent assigned
business identifiers, and most such elements on other resources
(`Patient.identifier`, `Device.identifier`, `Organization.identifier`,
`Substance.identifier`, etc.) are already `0..*`. Allowing the element
to repeat accommodates the real-world case the reporter raises: a single
regulatory artifact (same `type`, same `jurisdiction`) may be issued or
re-issued under multiple assigner-specific identifier values over its
lifecycle (e.g., pre- and post-merger assigning authorities, or national
vs. regional bodies that share a regulatory type). Aligning this element
with the general FHIR convention for `Identifier` removes a modeling
surprise for implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `DeviceDefinition.regulatoryIdentifier.identifier` at `1..1` and
instead document, in the element's comment and/or a short prose note on
the resource page, that when multiple identifier values share the same
`type` + `jurisdiction`, implementers SHOULD repeat the full
`regulatoryIdentifier` backbone once per identifier value. If needed,
introduce a standard extension
(`http://hl7.org/fhir/StructureDefinition/devicedefinition-regulatoryIdentifier-additionalIdentifier`)
of type `Identifier` with cardinality `0..*` that carries the
alternative/additional identifier values keyed to the same backbone
entry, for implementers who truly cannot tolerate backbone repetition.

#### Justification

The current 1..1 modeling keeps the semantics tight: every identifier
value is unambiguously bound to a single `type`/`jurisdiction` pair,
which matches how regulators usually issue identifiers in practice.
Repeating the backbone rather than the inner identifier preserves the
invariant that `type`+`jurisdiction`+`identifier` are a unit, and avoids
downstream tooling ambiguity about whether sibling context still applies
to every member of a repeated list. An extension is a backwards-
compatible escape hatch for the narrow case where multiple values must
truly share context. This matches how FHIR usually handles "one element
is actually a tuple" situations.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Close the ticket as **Not Persuasive** with the drafted resolution: the
parent backbone `DeviceDefinition.regulatoryIdentifier` already has
cardinality `0..*`, so implementers who need to record multiple
identifier values for a device — whether for different types,
jurisdictions, or assigners — repeat the backbone element rather than
the inner `identifier`. The existing model therefore already satisfies
the use case described in the ticket, and no spec change is required.

#### Justification

This is the disposition already drafted by the work group. It is
defensible because: (1) the `identifier` element is not isolated — it
is one of several siblings on `regulatoryIdentifier` that jointly
describe a regulatory artifact, so the correct unit of repetition is
the backbone, not the inner identifier; (2) `Identifier.assigner`,
`Identifier.period`, and `Identifier.use` already carry the
provenance/history nuances the reporter alludes to ("assigned by
multiple entities over time") at the datatype level; (3) relaxing
cardinality at this point in R5/R6 creates a small but real risk of
breaking implementations that assume a single value and access
`identifier` without iterating; (4) no use case has been presented in
which repeating the backbone is unworkable.

---

### Recommendation

**Recommended disposition:** C

Declining as Not Persuasive is the most defensible outcome. The reporter
raises a legitimate property of identifiers in general — that they can
be assigned by multiple entities over time — but FHIR already handles
this at two levels: the `Identifier` datatype carries `assigner`,
`period`, and `use`; and the enclosing backbone
`DeviceDefinition.regulatoryIdentifier` is `0..*`, which is the
appropriate unit of repetition because the sibling `type` and
`jurisdiction` elements scope the meaning of each identifier value. No
concrete implementer use case has been filed that cannot be met by
repeating the backbone, no Zulip conversation or cross-referenced
ticket has been raised against this disposition, and the current draft
resolution from the work group has been circulated for reporter review.
If a credible implementer objection surfaces during the HCP
disposition-approval step, Disposition B (a targeted extension) is the
preferred fallback because it preserves the existing semantics while
offering an opt-in relaxation; Disposition A should be avoided unless
O&O is prepared to accept the tooling-compatibility risk of relaxing a
base-resource cardinality late in the cycle.
