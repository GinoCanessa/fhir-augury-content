## Summary

FHIR-55441 is one of seventeen child tickets created from the parent ticket
[FHIR-53722](https://jira.hl7.org/browse/FHIR-53722) (FHIR Infrastructure,
"Deprecated elements should be withdrawn where applicable in R6"), which
flagged that R6 ballot 4 still carries elements that were intended to be
deprecated. The parent ticket was resolved *Persuasive with Modification* and
established three guideline rules for owning work groups: (1) if an element
has carried the deprecated flag since R4, remove it; (2) if a description
comment marks it deprecated **and** it is trial-use / its resource was
trial-use in R4 / it was added in R5+, remove it; (3) otherwise, add the
deprecated flag.

This child ticket asks **Orders & Observations** to apply that guidance to
`Observation.bodySite` for R6. `Observation.bodySite` was singled out in the
parent ticket as an exception to the "TU / deprecated since R5" pattern,
because its deprecation is tied to the introduction of
`Observation.bodyStructure` as a `CodeableReference` (already documented in
`source/observation/observation-introduction.xml`: *"The Observation.bodySite
element has been deprecated, corresponding with the change to make
Observation.bodyStructure a CodeableReference."*). The single ticket comment
to date confirms intent: "Observation.bodySite is being deprecated as of R6."
The decision the workgroup must make is whether `bodySite` qualifies for
removal under one of the parent's bullets, or whether — because Observation
is normative and `bodySite` is long-standing — it must be retained with the
deprecated flag for backward compatibility.

## Proposed Dispositions

### Disposition A: Accept As Requested — Remove `Observation.bodySite` in R6

#### Proposal

Remove the `Observation.bodySite` element from the `Observation` resource in
`HL7/fhir`. Concretely:

1. Delete the `bodySite` element entry from the `<differential>` of
   `source/observation/observation.xml` (and from the
   `observation-spreadsheet.xml` row if O&O still authors via the
   spreadsheet — sibling StructureDefinition takes precedence per the
   briefing).
2. Remove any `bodySite`-named search parameter authoring from the
   Observation `searchParameter` set, or update its expression to
   `Observation.bodyStructure` if the workgroup wishes to preserve a
   `body-site` search-parameter name.
3. Update `source/observation/observation-introduction.xml` to replace the
   current "has been deprecated" wording with a "removed in R6 — use
   `bodyStructure` (`CodeableReference(BodyStructure)`)" note, including a
   migration sentence pointing to `bodyStructure.concept` for callers that
   used a coded `bodySite`.
4. Add an R5→R6 migration entry pointing implementers at
   `Observation.bodyStructure`, and ensure any examples under
   `source/observation/` no longer populate `bodySite`.
5. Confirm that the deprecated-flag rendering issue noted in FHIR-53722 is
   moot for this element after removal.

#### Justification

The parent ticket established that elements meeting any of three criteria
should be removed; the description's comment in the current Observation
StructureDefinition explicitly marks `bodySite` as deprecated *and* a
direct replacement (`bodyStructure` as `CodeableReference(BodyStructure)`)
exists in R6. Carrying a flagged-deprecated element into a normative
release contradicts the spirit of normativity (FHIR-I's argument in
FHIR-53722) and forces every implementer to reason about an element they
are simultaneously told not to use. `bodyStructure` already provides both
the coded and reference paths, so the migration is purely mechanical for
consumers. FHIR-I has already removed analogous elements
(`Bundle.signature`, `OperationOutcome.issue.location`) under the same
guidance.

---

### Disposition B: Alternative Approach — Retain `Observation.bodySite` with the Deprecated Flag (apply guideline rule 3)

#### Proposal

Keep `Observation.bodySite` in R6 but ensure it is correctly flagged as
deprecated and that the deprecation is rendered in the published
StructureDefinition. Concretely:

1. In `source/observation/observation.xml`, ensure the `bodySite` element
   carries the formal `deprecated` standards-status (not merely a comment
   in the description), so that the publisher emits the deprecation badge
   in the rendered StructureDefinition. Cross-reference the FHIR-I note in
   FHIR-53722 about the flag failing to render in ballot 4 and confirm
   the fix.
2. Tighten the description: state that `bodySite` is deprecated in favour
   of `Observation.bodyStructure` (a `CodeableReference(BodyStructure)`),
   that new producers SHOULD use `bodyStructure`, and that consumers
   SHOULD continue to read `bodySite` for backward compatibility through
   R6.
3. Update `source/observation/observation-introduction.xml` to keep its
   existing "has been deprecated" wording but add explicit guidance that
   `bodySite` will not be removed during R6 to preserve compatibility
   with R4/R5 instances.
4. Note in the resolution that O&O is electing rule 3 of the parent
   guidance (description-comment-deprecated, but the element does not
   meet the removal criteria) because Observation is normative and
   `bodySite` long-predates R5.

#### Justification

`Observation.bodySite` is not analogous to the FHIR-I exemplars that were
removed: it is *not* trial-use, the Observation resource was *not*
trial-use in R4, and the element was *not* added in R5 or later — in
fact it has been part of Observation since DSTU2. Under a strict reading
of the parent ticket's three rules, that places it in rule 3 ("add the
deprecated flag", do not remove). Removing a long-standing element from a
normative resource breaks every R4/R5 instance currently in production,
breaks IG profiles that reference `Observation.bodySite` (US-Core, mCODE,
QI-Core, hl7-eu/laboratory, POCD v2 mappings, the
`structureLaterality` extension in `HL7/fhir-extensions`), and creates
real implementer burden for marginal modelling benefit. The element can
be retained while still steering new use toward `bodyStructure`.
Disposition B explicitly addresses the briefing's noted gotcha that the
deprecated flag isn't rendering in ballot 4 — the fix-the-rendering work
is required for this disposition to deliver value.

---

### Disposition C: Decline — Defer the decision and keep current state for ballot 5

#### Proposal

Decline to apply either change in the current ballot cycle. Leave
`Observation.bodySite` exactly as it stands (description-comment marked
as deprecated, but no formal flag and no removal). File a follow-up
ticket against O&O to revisit after R6 normative is published, and ask
FHIR-I to clarify whether the parent ticket's rule 3 is intended to
override "no element changes after normative" once Observation reaches
normative.

#### Justification

Defensible only if the workgroup judges that (a) it has insufficient time
in the current cycle to assess downstream IG impact, or (b) the
ambiguity in the parent ticket's three rules — specifically how rule 3
interacts with normative elements — should be settled by FHIR-I before
O&O makes a permanent change. Declining is the weakest option because the
status quo (description-comment without a rendered flag) is exactly the
inconsistency the parent ticket was raised to fix; choosing C effectively
kicks FHIR-55441 back to the parent without progress.

---

### Recommendation

**Recommended disposition:** B

A strict application of FHIR-53722's three-rule framework places
`Observation.bodySite` squarely in rule 3: it carries a description
comment indicating deprecation, but does not satisfy any of the removal
criteria (it is not trial-use, the resource was not trial-use in R4, and
the element was not added in R5+). Observation is on the path to
normativity, and `bodySite` is one of its oldest elements; removing it
would break a non-trivial set of downstream artifacts (`US-Core`,
`mCODE`, `hl7-eu/laboratory`, the `structureLaterality` extension that
explicitly augments `bodySite`, and the existing `uv-pocd` v2 mapping)
that current implementers are actively using. Disposition B preserves
backward compatibility, applies the parent ticket's guidance correctly,
and uses the opportunity to fix the rendering gap that prevented the
deprecation from showing in ballot 4 — the same gap FHIR-I called out.
Disposition A is internally consistent with FHIR-I's own choices for
trial-use elements, but `bodySite` is not in that category, so applying
the same treatment requires the workgroup to consciously override rule 3.
If the workgroup does favour removal, the resolution should explicitly
record that override and accept the IG-breakage cost.
