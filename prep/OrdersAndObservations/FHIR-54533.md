## Summary

Bas van den Heuvel ([FHIR-54533](https://jira.hl7.org/browse/FHIR-54533))
asks the Orders & Observations work group to clarify what URL is expected
in `DocumentReference.content.attachment.url` when the attachment
represents a medical image served by a WADO-RS endpoint. The
DocumentReference narrative currently says the `url` "should provide a
reference to a source WADO-RS service for the image," which the reporter
reads as ambiguous — it could mean either the WADO-RS service base URL
or a fully-resolved URL that, when dereferenced, returns the actual
image content. The reporter's position is that pointing at the WADO-RS
*base URL* would be unwise: `Attachment.url` is defined as "uri where
the data can be found," so it must resolve to retrievable content (i.e.,
a specific WADO-RS `studies/.../instances/...` resource, or a
WADO-URI/WADO-RS rendered/metadata endpoint), not a service root that a
client would still have to navigate. The cross-reference engine
identifies no related Jira tickets, no related GitHub PRs/issues, and no
substantive Zulip discussion of this clarification — only one
loosely-related shorthand-stream thread that happens to use a WADO-RS
URL in an example.

## Details

**Description:**

> For this reason, medical images can also be represented in a
> DocumentReference resource, but the
> DocumentReference.content.attachment.url should provide a reference to
> a source WADO-RS service for the image.

What does this mean? That in the case of WADO the baseURL is provided?
I think the URL should point to location where the content can be
retrieved.

If the intent is to point to the baseUrl, I do not think that is a wise
idea. Please clarify.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the DocumentReference narrative (the imaging paragraph that
contains the quoted "should provide a reference to a source WADO-RS
service for the image" sentence — currently in the resource's
boilerplate / scope-and-usage section) to state explicitly that
`DocumentReference.content.attachment.url` MUST be a fully-resolved
WADO-RS (or WADO-URI) resource URL that, when dereferenced, returns
the image content (study, series, instance, rendered, or metadata
representation) — not a WADO-RS service base URL. Add a small
non-normative example showing a `studies/{study}/series/{series}/
instances/{instance}` form with an appropriate `contentType` (e.g.,
`application/dicom`, `application/dicom+json`, or
`multipart/related; type="application/dicom"`). Cross-link to
`ImagingStudy.endpoint` for the case where the caller actually wants
to advertise a WADO-RS service rather than a single retrievable image.

#### Justification

`Attachment.url` is normatively defined as "uri where the data can be
found," so any reading that allows a service base URL would conflict
with the element's own definition and would force every client to
implement WADO-RS service discovery just to render a referenced
attachment. Locking the wording to "fully-resolved WADO-RS resource
URL" removes the ambiguity the reporter flags, aligns the imaging
guidance with how `Attachment.url` is used elsewhere (PDF, JPEG, FHIR
Binary, etc.), and matches observed real-world authoring (e.g., the
shorthand-stream Zulip example uses a concrete `studies/.../metadata`
URL). The change is documentation-only, has no schema impact, and is
non-breaking.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only clarifying `Attachment.url`, redirect callers who
want to expose a WADO-RS *service* to `DocumentReference.context`
(via an `ImagingStudy` reference, whose `endpoint` carries the
WADO-RS service base URL with `connectionType = dicom-wado-rs`) and
keep `content.attachment.url` strictly for the retrievable
representation. Update the DocumentReference narrative to (a) say
`content.attachment.url` MUST resolve to retrievable image content
(same wording as Disposition A), and (b) add a sentence steering
"service-level" use cases to `ImagingStudy.endpoint` /
`Endpoint.connectionType = dicom-wado-rs`. Optionally add a
non-normative example DocumentReference that pairs a single
`content.attachment` (instance-level URL) with a
`context.related`/`context` link to an `ImagingStudy` whose
`endpoint` is the WADO-RS service.

#### Justification

This addresses both halves of the ambiguity the reporter raised: it
not only forbids the base-URL reading of `Attachment.url` (the harm
the reporter is worried about) but also tells implementers where the
service-level URL legitimately belongs. The `Endpoint` resource and
`ImagingStudy.endpoint` already exist precisely for this purpose, so
nothing new has to be invented and the imaging community already
recognises the pattern. The cost is a slightly larger doc edit and a
review pass to make sure the cross-link to `ImagingStudy` /
`Endpoint` is accurate; the benefit is that future readers don't
re-derive the same ambiguity from a different angle ("but where *do*
I put the WADO-RS base URL?").

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive — already addressed by element definition."
Take the position that `Attachment.url` is already defined as the URI
where the data can be found, that a base URL therefore cannot satisfy
that definition, and that no narrative change is required.

#### Justification

Strictly speaking, the normative definition of `Attachment.url`
already rules out a service base URL, so a literal reading of the
spec gives the reporter the answer they want. Declining keeps the
narrative untouched and avoids any risk of re-introducing ambiguity
in the rewrite. The weakness of this disposition is that the
*imaging* paragraph the reporter actually quotes does invite the
misreading — so declining preserves a known foot-gun that costs zero
to fix.

---

### Recommendation

**Recommended disposition:** B

The reporter's question reveals two overlapping problems in the
current text: (1) the imaging paragraph is loose enough to imply a
service base URL is acceptable in `content.attachment.url`, and
(2) there is no signposting telling implementers where a WADO-RS
service URL legitimately *does* belong. Disposition A fixes (1) but
leaves (2) — meaning the next implementer who actually does have a
service URL will either cram it back into `Attachment.url` or file a
follow-up ticket. Disposition B fixes both with the same
documentation-only, non-breaking edit by leaning on the
already-existing `ImagingStudy.endpoint` / `Endpoint(connectionType =
dicom-wado-rs)` pattern, which is the imaging community's
established mechanism. Disposition C is hard to defend: the reporter
has demonstrated, in good faith, that the current paragraph is
ambiguous, and "the element definition technically already says so"
is not a satisfying answer when the offending text is a few lines
above the element definition. Recommend B and let the work group
decide whether the cross-link to `ImagingStudy.endpoint` should be
normative ("MUST use") or merely informative ("see also").
