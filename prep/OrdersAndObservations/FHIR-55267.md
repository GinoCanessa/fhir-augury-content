## Summary

`NutritionOrder.intent` is a mandatory (1..1) element with a required binding to the cross-resource `request-intent` value set. The reporter (Lloyd McKenzie) observes that the bound value set offers no `unknown`, `other`, or equivalent escape-valve concept, which makes it impossible for senders to populate the element honestly when the underlying intent is unknown or doesn't match any of the available codes. The ticket asks that an escape-valve concept be added to the value set so that `NutritionOrder` instances can always be validly constructed.

The same problem appears across every resource that binds `.intent` to `http://hl7.org/fhir/ValueSet/request-intent`. Companion tickets exist for [FHIR-55275](https://jira.hl7.org/browse/FHIR-55275) (ServiceRequest), [FHIR-55262](https://jira.hl7.org/browse/FHIR-55262) (DeviceRequest), and [FHIR-55322](https://jira.hl7.org/browse/FHIR-55322) (CommunicationRequest). Because the value set and code system are owned by FHIR Infrastructure / CDS rather than OO, the OO main call on 2026-02-12 agreed that OO should record its recommendation and refer the change to FHIR-I. A consolidated FHIR-I ticket — [FHIR-56142](https://jira.hl7.org/browse/FHIR-56142) (Request.intent has no escape valve) — was subsequently logged by Michelle Miller and has already been resolved Persuasive (2026-04-13), with FHIR-I committing to add an `Other` concept to the request-intent code system and accompanying authoring guidance. Resolving FHIR-56142 effectively resolves this ticket for `NutritionOrder` as well.

## Proposed Dispositions

### Disposition A: Accept As Requested (Persuasive — defer to FHIR-I via FHIR-56142)

#### Proposal

Mark FHIR-55267 as **Persuasive** with the following note: "The escape-valve concept the reporter requests cannot be added by OO because the bound value set (`http://hl7.org/fhir/ValueSet/request-intent`) and its code system (`http://hl7.org/fhir/request-intent`) are owned by FHIR-I and CDS respectively. The change has been made in FHIR-I ticket FHIR-56142 (Resolved Persuasive 2026-04-13), which adds an `Other` code to the `request-intent` code system and authoring guidance for `Choosing an intent code`. Once FHIR-56142 is applied, `NutritionOrder.intent` will automatically gain the requested escape-valve concept with no change required to the `NutritionOrder` resource itself." Make no edits to the `NutritionOrder` StructureDefinition; close FHIR-55267 as resolved by FHIR-56142 (link as duplicate-of / depends-on).

#### Justification

This matches the OO main-call consensus on 2026-02-12 (record recommendation, defer ownership) and the explicit cross-resource resolution agreed by FHIR-I in FHIR-56142, which lists FHIR-55267 by number as one of the tickets it resolves. It avoids divergent per-resource fixes (which would create inconsistency across `ServiceRequest`, `DeviceRequest`, `CommunicationRequest`, and `NutritionOrder`), respects code-system ownership boundaries, and gives the reporter exactly what they originally asked for: an escape-valve concept inside the bound value set. McKenzie's correction in FHIR-55275 (a `code`-typed element cannot be made extensible, and the request was for a value-set extension, not a binding-strength change) further supports fixing this at the value set/code system layer rather than at the resource binding.

---

### Disposition B: Alternative Approach (Relax cardinality and/or binding on `NutritionOrder.intent`)

#### Proposal

If FHIR-I's resolution were not in place, the alternative would be to change `NutritionOrder.intent` locally: relax cardinality from `1..1` to `0..1`, and/or weaken the binding strength from `required` to one that permits a `data-absent-reason` extension to fill the gap. (Note: because `.intent` is a `code`, the binding cannot be made `extensible`; the realistic local options are cardinality relaxation plus reliance on the `dataAbsentReason` extension, or migrating the element to `CodeableConcept`.) Apply this only to `NutritionOrder` if FHIR-I declines or delays.

#### Justification

This was OO's first-pass resolution path on the parallel ServiceRequest ticket (FHIR-55275, vote 5-0-0 on 2026-02-20) before it was reverted. It has the advantage of being entirely within OO's authority and not waiting on another work group. However, it has serious drawbacks the workgroup already surfaced: (a) `code` cannot be made extensible (JIRAUSER19407, 2026-02-23), so the binding-relaxation half of the original motion is technically infeasible; (b) cardinality relaxation creates an OO-vs-everyone-else inconsistency in the Workflow/Request pattern, which several other resources still keep as `1..1`; (c) it is a substantive, potentially breaking change for implementers already handling `NutritionOrder`; and (d) it does not actually give senders a positive code to communicate "intent is unknown / other" — it just lets them omit the element, which is semantically weaker than what the reporter requested.

---

### Disposition C: Decline (Not Persuasive — existing mechanism suffices)

#### Proposal

Mark FHIR-55267 as **Not Persuasive** on the grounds that the existing `dataAbsentReason` extension already provides a standards-compliant escape valve for required-bound coded elements, and senders that genuinely cannot determine an intent should use that extension rather than relying on a code in the value set itself.

#### Justification

`http://hl7.org/fhir/StructureDefinition/data-absent-reason` is the FHIR-wide mechanism for signalling that a value is unknown / unsupported / not applicable, and it is already permitted on coded elements with required bindings. Declining the request would preserve the current `request-intent` value set, avoid coordination across OO / FHIR-I / CDS, and avoid encouraging "unknown" as a routine intent value (which has historically caused interoperability harm where receivers cannot act on the request). This is a defensible option, but it is weaker than the other two given that (i) FHIR-I has *already* judged the request Persuasive in FHIR-56142, and (ii) McKenzie noted in the FHIR-56142 discussion that "data absent reason doesn't have 'Other'", reinforcing that an in-band `Other` code is genuinely needed for the legacy/forward-compat scenarios called out in the FHIR-56142 resolution.

---

### Recommendation

**Recommended disposition:** A

OO should resolve FHIR-55267 as **Persuasive — resolved by FHIR-56142** with no local change to `NutritionOrder`. This recommendation is anchored in three concrete facts gathered for this review: (1) the OO main call has already agreed (2026-02-12) to defer escape-valve changes that hinge on the `request-intent` code system to FHIR-I; (2) FHIR-I has already resolved FHIR-56142 Persuasive (2026-04-13) with a specific change — adding an `Other` code with a precise definition plus an authoring "Choosing an intent code" note — that explicitly lists FHIR-55267 as a dependent ticket; and (3) the alternative (relaxing `NutritionOrder.intent` locally) was tried on the sister ticket FHIR-55275 and reverted because `code` cannot be made extensible and the original requester's actual ask was a value-set extension, not a binding/cardinality change. Disposition A delivers exactly what the reporter requested, keeps the four sister resources aligned, respects ownership boundaries, and avoids substantive local breaking changes to `NutritionOrder`. The workgroup should also confirm FHIR-55262 (DeviceRequest) is closed in the same way and ping Patient Care to do likewise for FHIR-55322.
