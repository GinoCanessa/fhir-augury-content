## Summary

FHIR-54165 reports a logic mismatch between the human-readable description and
the FHIRPath expression of the `docRef-2` invariant on `DocumentReference`. The
human text reads *"practiceSetting SHALL only be present if context is **not
present**"*, while the FHIRPath says
`practiceSetting.empty() or context.where(resolve() is Encounter).empty()` —
which actually means *"practiceSetting SHALL only be present if context is not
an Encounter"*. As written, the FHIRPath permits `practiceSetting` whenever
`context` references a non-Encounter resource (e.g., `Appointment` or
`EpisodeOfCare`), even though the description implies `context` must be
absent entirely. The reporter (Eric Haas) proposes realigning the description
to *"practiceSetting SHALL only be present if context not an encounter"* so
that the human text matches the executable invariant.

Context from related tickets:

- [FHIR-32580](https://jira.hl7.org/browse/FHIR-32580) is the original (R5)
  request that created the `docRef-1` / `docRef-2` invariants. The invariant
  text proposed there by John Moehrke for `docRef-2` already said
  "practiceSetting SHALL only be present if context is **not present**" with
  expression `practiceSetting.empty() or !context.resolve().as(Encounter).resolve().exists()`,
  i.e., the description/expression mismatch existed at the moment the
  invariant was first authored.
- [FHIR-54170](https://jira.hl7.org/browse/FHIR-54170) (also from Eric Haas,
  already Applied with PR `HL7/fhir#4040`) is a sibling cleanup task: it
  updates `DocumentReference.context`'s description to acknowledge that the
  element now allows `Appointment | Encounter | EpisodeOfCare`. That change
  is what makes the `docRef-2` description in this ticket actually misleading
  in practice — `context` is no longer Encounter-only, so saying "not
  present" hides the fact that practiceSetting is also allowed when context
  is an Appointment or EpisodeOfCare.

## Details

**Description:**

> ### Constraint docRef-2 Logic Error
>
> The constraint `docRef-2` states:
>
> > "practiceSetting SHALL only be present if context is **not present**"
>
> But the FHIRPath expression is:
>
> > `practiceSetting.empty() or context.where(resolve() is Encounter).empty()`
>
> The description says "if context is not present" but the expression says "if
> context is not an Encounter". These are different conditions:
>
> - "not present" means `context.empty()`
> - "not an Encounter" means `context.where(resolve() is Encounter).empty()`
>
> The constraint description doesn't match the FHIRPath expression. The
> FHIRPath allows `practiceSetting` when `context` references an `Appointment`
> or `EpisodeOfCare`, but the description implies `practiceSetting` should
> only be present when `context` is completely absent.
>
> Propose changing the description to "practiceSetting SHALL only be present
> if context not an encounter".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the human description of invariant `docRef-2` on `DocumentReference`
from:

> "practiceSetting SHALL only be present if context is **not present**"

to:

> "practiceSetting SHALL only be present if context is not an Encounter"

Leave the FHIRPath expression
(`practiceSetting.empty() or context.where(resolve() is Encounter).empty()`)
unchanged. Apply the equivalent edit to the rendered DocumentReference
resource page and any derived artifacts (snapshot/differential) so that the
human text and the executable expression agree. While in the file, sanity
check that `docRef-1` (facilityType) does not exhibit the same drift; if it
does, fix it the same way in this ticket.

#### Justification

The reporter is correct that the FHIRPath has always constrained on
"context resolves to an Encounter", not "context is empty" — this is visible
both in the current expression and in the originating
[FHIR-32580](https://jira.hl7.org/browse/FHIR-32580) thread where the
invariant was first authored. With FHIR-54170 now widening
`DocumentReference.context` to allow `Appointment | Encounter |
EpisodeOfCare`, the gap between "not present" and "not an Encounter" has
real implementer impact: under the current human text an implementer might
believe `practiceSetting` is forbidden whenever any `context` is supplied,
even when `context` references an Appointment, but the executable invariant
explicitly permits that. Aligning the description to the FHIRPath is
consistent with the OO decision in FHIR-32580 (option 3: only restrict
co-occurrence with Encounter) and is editorial / non-substantive in impact.

---

### Disposition B: Alternative Approach

#### Proposal

Tighten the FHIRPath instead of relaxing the prose. Change `docRef-2` (and
symmetrically `docRef-1`) so that `practiceSetting` / `facilityType` are
forbidden whenever **any** `context` value is present, matching the original
human text:

- New expression: `practiceSetting.empty() or context.empty()`
- New human text: "practiceSetting SHALL only be present if context is not
  present"

Apply the same shape to `docRef-1` for `facilityType`. Document the
substantive change in the R6 change log and add migration guidance pointing
implementers who today rely on `practiceSetting` alongside an `Appointment`
or `EpisodeOfCare` `context` to either drop `practiceSetting` or move the
setting into the referenced resource.

#### Justification

The original FHIR-32580 discussion was driven by Lloyd McKenzie's concern
that having both an Encounter context **and** the "indirect Encounter
metadata" elements creates duplication and search inconsistency. The same
argument arguably applies to Appointment and EpisodeOfCare: those resources
also model the encounter-like setting (location, service category) that
`practiceSetting` and `facilityType` capture. Tightening to "context is not
present" preserves that no-duplication intent uniformly across all three
allowed `context` target types. The downside is that this is a substantive,
breaking change relative to R5's published invariant, would need OO and FMG
attention rather than an editorial pass, and may invalidate instances
already produced under the current FHIRPath.

---

### Disposition C: Decline

#### Proposal

Decline the change as "Not Persuasive – Editorial only / no implementer
harm". Rationale: warning-level invariants are advisory; the FHIRPath is
normative behaviour and tooling honours the expression, not the description.
Leave `docRef-2` as-is and address description tone (if at all) in a future
batch alongside other DocumentReference editorial issues
(e.g., FHIR-54171).

#### Justification

`docRef-2` is severity `warning`, so no instance becomes invalid based on
the human text; conformance tooling evaluates the FHIRPath. Reviewers may
view the cost/benefit of touching a published invariant — even editorially
— as not worth it given the small population of authors who write
DocumentReference resources with a non-Encounter `context` and
`practiceSetting` simultaneously.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a real and easily verifiable inconsistency
between the human description and the FHIRPath of `docRef-2`, and the
FHIRPath has unambiguously meant "not an Encounter" since the invariant was
authored in [FHIR-32580](https://jira.hl7.org/browse/FHIR-32580). Fixing the
prose (Disposition A) is editorial, has no impact on existing instances or
tooling, and brings the resource into line with the just-Applied widening of
`DocumentReference.context` from FHIR-54170. Disposition B (tightening the
FHIRPath) reopens the FHIR-32580 design debate and would be a substantive
change to a published invariant — out of proportion to a documentation fix,
and better raised as its own change request if OO actually wants that
behaviour. Disposition C understates the harm: even at `warning` severity,
incorrect human text on a published constraint actively misleads
implementers reading the spec. While editing `docRef-2`, it is worth
running the same eye over `docRef-1` (`facilityType`) so the pair stay
consistent.
