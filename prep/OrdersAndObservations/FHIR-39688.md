## Summary

Stephen Chu requests that the FHIR Device resource (and the surrounding
Device-related resources) be reviewed and extended so that the
**function/operating parameters of a device** — for example basal and bolus
dosing on an insulin infusion pump, or the pacing mode on an implantable
cardioverter-defibrillator (ICD) — can be expressed in a "semantically
accurate" structure. Today, the reporter argues, neither the current
settings of such a device nor the changes made to those settings over time
can be captured cleanly in the existing Device resource shape, and that
gap matters for decision traceability, retrospective evaluation of how
well the chosen settings worked, and informing future setting choices.

The cross-references are limited but pointed: the reporter and the
Devices WG (via Elliot Silver) have anchored the discussion on whether
**`ServiceRequest.orderDetail`** (with `parameterFocus` pointing at the
Device and `parameter.code`/`parameter.value[x]` carrying the requested
setting) is sufficient, with **`Task`** raised as the alternative when no
ServiceRequest is in play. A January 2026 OO discussion (per
`hbuitendijk`'s comment) concluded that Device + DeviceMetric +
Observation already cover *recording actual settings*, but FHIR has no
agreed mechanism to *instruct* a device to change a setting — and that
this analysis is too large for R6 ballot reconciliation. There are no
other linked Jira change requests; the conversation has stayed on this
single ticket and the implementers Zulip thread referenced below.

## Details

**Description:**

> Devices (external | implantable), such as insulin infusion pump,
> implantable cardioverter-defibrillator, need to be programmed to
> operate correctly based on patient's clinical status. For example:
> Insulin infusion pump dosage - basal and bolus dosage; Implantable
> cardioverter-defibrillator (ICD) - pacing mode. These device function
> parameters may also need to be adjusted from time to time, based on
> changing patient clinical status. The initial setting and subsequent
> parameters adjustments need to be captured for reasons including:
> decision traceability, evaluation of parameter setting clinical
> effectiveness, to inform future device function parameter setting,
> etc.
>
> A detail analysis of the FHIR Device (and Device related) resources
> lead to the assessment that these requirements are not supported by
> the current Device resource structure. This change request is to
> suggest a detail review of the requirements and to determine the
> optimal change decision(s). An analysis document is uploaded to
> provide more detail use cases analysis and contextual information to
> support this change request.

## Related Zulip Discussions

### [implementers > FHIR resources for devices](https://chat.fhir.org/#narrow/stream/implementers/topic/FHIR%20resources%20for%20devices)

An implementer (Olex) posted a near-identical real-world problem in
May 2024: a medical device receives a few dozen configuration settings
to perform a procedure, a nurse may then adjust some, and the
implementer needs a place to store both the *input preset* and the
*adjusted preset*. They considered `DeviceRequest` but doubted it for
the adjusted settings. Elliot Silver responded that the area is under
active discussion: `DeviceMetric` defines what a setting *is* (not its
value), the *current* value of a setting/measurement belongs in an
**Observation** referencing the DeviceMetric, and the most appropriate
place to express a *requested* setting today is
**`ServiceRequest.orderDetail`** with `parameterFocus` referencing the
Device, `parameter.code` indicating the setting, and `parameter.value[x]`
giving the desired value. Elliot stressed that **FHIR deliberately stays
away from sending settings *to* the device**, because that requires
safety protocols that have not been built into FHIR. Lloyd McKenzie and
Elliot then explored doing the same via **Task** when no ServiceRequest
exists — each setting could be a Task input, or a contained
`Parameters` instance could carry complex/nested settings (Parameters
has an `id` because it inherits from Resource, so it is contained-safe).
No formal consensus was recorded, but the practical guidance settled
into: ServiceRequest.orderDetail for "ordered settings", Observation
(with DeviceMetric) for "actual settings", and Task as a possible
alternate carrier for the request side.

Participants: Olex, Elliot Silver, Lloyd McKenzie.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the ticket as a substantive change request against the **Device**
resource and the surrounding Device family for **R6 (or, more
realistically, R7)**. Specifically:

1. Add a structured way on **Device** (or a sibling resource such as a
   new `DeviceConfiguration` / `DeviceSettings`) to express a *snapshot*
   of the device's function parameters at a point in time, with at
   least: the parameter code (DeviceMetric reference or coded
   identifier), the parameter value (`value[x]` allowing
   Quantity/CodeableConcept/string/etc.), the time the value was set,
   the actor who set it, and a reason/indication.
2. Allow a longitudinal/history representation of those parameter
   snapshots so that adjustments over time are first-class (either by
   making the structure repeating on Device, or by formally pairing
   Device + Observation panels keyed on DeviceMetric and clearly
   normative-binding that pattern in the Device narrative).
3. On **`ServiceRequest`**, expand `ServiceRequest.parameterFocusReference`
   so that **DeviceMetric** is a valid target (per `hbuitendijk`'s OO
   note), so an order can target a specific metric on a specific device.
4. Update Device, DeviceMetric, Observation, and ServiceRequest
   narrative to describe the canonical pattern for "ordered settings",
   "actual settings", and "history of settings" together, with
   examples for the insulin-pump and ICD use cases.

#### Justification

This is what the reporter literally asks for and what the longitudinal
clinical use case (the unresponsive-pump-patient scenario) actually
needs: both the *current* settings and a *history* of how those
settings evolved, in a place that is unambiguously about the device
configuration rather than a side effect of an Observation panel. The OO
January 2026 vote (11 in favor, 0 against, 1 abstain) shows the WG
sees real merit in the substance, and `hbuitendijk`'s summary already
identifies a concrete, low-risk piece — extending
`ServiceRequest.parameterFocusReference` to include DeviceMetric — that
is well within scope of a normal change request. The OO February 2026
revert of "Considered for Future Use" indicates the WG is not ready to
just defer this without further action.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add new structure to Device itself. Instead, formally
document and lightly extend the existing combination of resources to
cover the use case, and split the work in two:

1. **Settings as data (recording what happened):** Confirm and document
   that **Observation referencing DeviceMetric** is the canonical
   carrier for the *actual* current value of a device setting, exactly
   as Elliot Silver described in the implementers Zulip thread. Add a
   short cross-cutting page (or a section in Device / DeviceMetric)
   describing the "Device settings over time" pattern, with worked
   examples for insulin pump basal/bolus and ICD pacing mode. No new
   resource, no new core element.
2. **Settings as a request (asking for a change):** Use
   **`ServiceRequest.orderDetail`** with `parameterFocus` → Device and
   `parameter.code` / `parameter.value[x]` for the requested settings,
   plus **`Task`** with inputs (or a contained `Parameters` instance)
   when there is no clinical order — both already supported, per the
   Zulip discussion. Make the only structural change the one OO has
   already converged on: expand
   `ServiceRequest.parameterFocusReference` to include DeviceMetric.
3. Charter an **HCP/Devices joint IG project** (the ticket already
   carries the `HCP` and `Joint-OO-PC` labels) to validate the pattern
   against real device-management use cases, and only then escalate
   any residual gaps back into core for a future release.

#### Justification

The Devices WG has stated explicitly — both in `esilver`'s 2024 comment
and in the implementers Zulip thread — that **FHIR should not be used
for command-and-control of devices** without a thorough safety
analysis, because mis-set device parameters can cause direct patient
harm. Adding new core "settings" structure to Device risks implying
that core FHIR is the right transport for device programming, which the
Devices WG opposes. The OO January 2026 summary itself concluded that
the request is *too large to address as a ballot comment to R6* and
that *more analysis is needed via an IG project*, which is exactly what
this disposition formalises. It also delivers the only piece both WGs
agree on (DeviceMetric in `parameterFocusReference`) without
overcommitting the Device resource.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — Out of Scope for the Device
Resource**, on the basis that:

- *Recording* device settings is already supported by Observation +
  DeviceMetric (+ Device), and the gap the reporter perceives is
  documentation, not structure.
- *Requesting* a device setting is already supported by
  `ServiceRequest.orderDetail` (and, where appropriate, Task with
  inputs / a contained Parameters), as confirmed by the Devices WG
  and the implementers Zulip thread.
- *Sending* settings *to* a device is intentionally outside FHIR's
  scope per the Devices WG until a dedicated safety analysis is
  completed.

If the reporter wants the documentation gap closed, redirect the work
to a narrower OO/Devices documentation ticket and (separately) to the
HCP IG project.

#### Justification

This is the most defensible position if the WG concludes that the
existing primitives are sufficient and the real problem is that they
are under-documented. It directly mirrors the Devices WG's stated
position and aligns with FHIR's general principle of not adding core
structure when an existing pattern works. It is, however, hard to
square with the OO January 2026 vote (11–0–1 in favor) and with OO's
own observation that there is no agreed mechanism to *instruct* a
device — declining outright would leave that gap on the table for
another release cycle.

---

### Recommendation

**Recommended disposition:** B

The ticket has been open since December 2022 and has cycled through
multiple OO/Devices conversations without converging on a Device-shaped
change, which is a strong signal that bolting new structure onto
Device is not where the value is. Disposition B does three things the
WG has already publicly endorsed: (1) it codifies the
Observation+DeviceMetric pattern that Elliot Silver described in the
implementers thread for *recording* settings, (2) it adopts the one
concrete normative change OO converged on in January 2026 — letting
`ServiceRequest.parameterFocusReference` target DeviceMetric — for
*requesting* settings, and (3) it routes the unresolved
"command-and-control" question to an HCP/Devices joint IG (already
hinted at by the `HCP` and `Joint-OO-PC` labels), where the safety
analysis the Devices WG insists on can actually happen without
blocking R6. Disposition A is attractive because of the strong OO
in-favor vote, but it would commit Device to structure that the
Devices WG is on record opposing without a safety review — exactly
the friction `hbuitendijk` flagged. Disposition C is hard to defend
against the OO vote and the reporter's clinical use case. B threads
the needle: it ships the agreed-on `parameterFocusReference`
extension now, formalises the existing pattern, and creates the
forum where the harder Device-programming question can mature into a
future change request.
