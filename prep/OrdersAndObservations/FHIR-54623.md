## Summary

Eric Haas (Orders & Observations) is asking the workgroup to simplify
`ServiceRequest.patientInstruction` in FHIR Core. Today it is modelled
as a `BackboneElement` (0..1) that contains a single optional choice
child `instruction[x]` (Markdown | Reference). The reporter argues that
wrapping a single optional child in a BackboneElement is unusual,
introduces unnecessary structural overhead, and is a regression from R4,
where `patientInstruction` was a plain `0..* string`. The recommended
fix is to collapse the BackboneElement back into a direct choice element:
`patientInstruction[x] 0..* (Markdown | Reference)`.

Context from related Jira items:

- The current shape was introduced by
  [FHIR-18838](https://jira.hl7.org/browse/FHIR-18838) ("Add support for
  instructional material — STU #40", *Persuasive with Modification*,
  applied in HL7/fhir PR #2600). That ticket's resolution explicitly
  promised "rename patientInstruction to patientInstructions, change
  cardinality to 0..*, make it a **choice element** of `markdown 0..1`
  or `Reference(DocumentReference) 0..1`". The current build matches
  the *intent* (markdown or reference, 0..1 each) but not the *shape*
  (it ended up as a BackboneElement wrapping a single choice child
  rather than a direct choice element). FHIR-54623 is essentially a
  request to deliver the shape originally promised by FHIR-18838.
- [FHIR-54621](https://jira.hl7.org/browse/FHIR-54621) ("Change
  bodyStructure to 0..*") is a companion ticket from the same reporter
  filed the same day, also flagging an R4→R6 ServiceRequest regression.
  It does not directly affect `patientInstruction` but shows that
  Eric Haas is doing a systematic R6 ServiceRequest review, which the
  workgroup may want to handle as a coordinated set.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the reporter's recommendation in HL7/fhir for the R6 build:
flatten `ServiceRequest.patientInstruction` from a BackboneElement back
into a direct choice element.

Concretely, in the `ServiceRequest` StructureDefinition:

- Remove the BackboneElement `ServiceRequest.patientInstruction` and
  its child `ServiceRequest.patientInstruction.instruction[x]`.
- Introduce `ServiceRequest.patientInstruction[x]` with cardinality
  `0..*` and types `markdown` and `Reference(DocumentReference)` (the
  same target types as today's `instruction[x]`).
- Carry over the existing definition / requirements / comments from
  the BackboneElement and its child onto the new choice element.
- Update narrative, examples, and any FHIRPath constraints/invariants
  that referenced `patientInstruction.instruction[x]` to reference
  `patientInstruction[x]` directly.

#### Justification

This shape matches what FHIR-18838 actually resolved to ("make it a
choice element with markdown / Reference choices") and removes the
gratuitous BackboneElement layer that the current build introduced. It
also restores the R4 ergonomics (a single repeating element on the
resource) while keeping the richer R5/R6 capability of `Reference` in
addition to text. The Zulip evidence (IG-publisher narrative exception
on this very element; implementers consistently treating it as a
patient-renderable choice) supports simplification. From a FHIR design
standpoint, BackboneElements are reserved for grouping multiple
related children — a backbone with a single optional child is widely
considered an anti-pattern.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the BackboneElement but justify it by enriching it, rather than
collapsing it. In the `ServiceRequest` StructureDefinition:

- Retain `ServiceRequest.patientInstruction` as a BackboneElement but
  change cardinality to `0..*` (one instruction "block" per piece of
  guidance).
- Keep `instruction[x] 1..1` (markdown | Reference(DocumentReference)) —
  promoting it to required within the block, since an instruction block
  with no instruction is meaningless.
- Add at least one additional sibling element that justifies the
  backbone, e.g. `audience : CodeableConcept 0..1` (patient / caregiver
  / family) or `language : code 0..1` (BCP-47, for translated copies),
  drawn from existing implementer requests on patient-facing
  instructions.

#### Justification

This addresses the reporter's "BackboneElement with a single optional
child is weird" concern without losing the ability to attach metadata
to each instruction (audience, language, when-to-show, etc.) that
implementers regularly need for patient-facing material. It is a
non-breaking superset of the current model (existing data continues
to validate after the cardinality bump and the `instruction[x] 1..1`
promotion, modulo a minor required-vs-optional shift). The trade-off
versus Disposition A is that callers now must always wrap a single
markdown string in a backbone, which is more verbose for the most
common case. It does, however, future-proof the element if O&O / PC
ever want to attach additional facets to an instruction. The Patient
Care thread shows there is appetite for richer instruction modelling.

---

### Disposition C: Decline

#### Proposal

Decline the request and keep `ServiceRequest.patientInstruction` as a
BackboneElement (0..1) wrapping `instruction[x] 0..1` for R6.

#### Justification

R6 is in/near ballot; flattening a BackboneElement into a choice
element is a *breaking* change for every existing R5/R6-draft
implementer that has already coded against
`patientInstruction.instruction[x]`, and it changes the JSON shape
(`{ "patientInstruction": { "instructionMarkdown": "…" } }` becomes
`{ "patientInstructionMarkdown": "…" }`). Tooling — including the IG
publisher exception flagged on Zulip — can usually be fixed with a
generator/cross-version map fix without altering the resource shape.
Since FHIR-18838 was already resolved Persuasive with Modification and
the current shape is *functionally* equivalent (markdown or reference,
optional), the cost/benefit of another shape change this late in the
cycle does not clear the bar.

---

### Recommendation

**Recommended disposition:** A

The current BackboneElement shape is hard to defend on its own merits:
it wraps a single optional child, which the FHIR community generally
treats as a modelling smell, and it diverges from what FHIR-18838
actually resolved to ("a choice element of markdown | Reference"). The
reporter's proposed `patientInstruction[x] 0..* (markdown |
Reference)` is functionally equivalent to today's content but
substantially simpler to author, render, and cross-version-map (R4's
`0..* string` flows naturally into R6's `0..* markdown` choice). The
IG-publisher narrative exception surfaced on Zulip is a concrete sign
that the BackboneElement shape is already causing tooling pain. A pure
decline (Disposition C) just preserves a known wart, and Disposition B
solves the smell only by inventing requirements (audience, language,
…) the workgroup hasn't asked for. If the workgroup later identifies
real per-instruction metadata, it can be added as extensions or via a
future targeted ticket without re-litigating today's shape.

The breaking-change concern raised under Disposition C is real but
manageable while R6 is still in ballot: the cross-version map for
`patientInstruction` should already be touched for the R5→R6 rename
work, and consumers who only ever populate one variant (markdown or a
reference) — which is the dominant pattern in the Zulip threads — will
see a near-trivial migration.
