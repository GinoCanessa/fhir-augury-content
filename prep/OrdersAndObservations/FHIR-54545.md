## Summary

Bas van den Heuvel reports that the current FHIR guidance — illustrated on
DiagnosticReport (and similar event-pattern resources) — tells implementers to
use the Identifier-type codes `PLAC` (Placer Identifier) and `FILL` (Filler
Identifier) on the *report's* `.identifier`. The ticket's title spells the codes
"PLAX/FILL", but the body and the cited value set
([valueset-identifier-type](https://build.fhir.org/valueset-identifier-type.html))
make clear that the codes in question are `PLAC` and `FILL` from
`http://terminology.hl7.org/CodeSystem/v2-0203`. Both codes are defined as
"An identifier for a request" — i.e., they identify a ServiceRequest (the
order), not the resulting event (the report). The reporter therefore argues
that these identifier-type codes do not belong on `DiagnosticReport.identifier`
and that, when a report needs to carry the placer/filler numbers of the
underlying order, those identifiers should appear on the request itself
(e.g., `ServiceRequest.identifier`) and be surfaced on the report via
`DiagnosticReport.basedOn.identifier.type`.

The reporter explicitly links this work to
[FHIR-48692](https://jira.hl7.org/browse/FHIR-48692) ("Clarification on the
Use of Identifiers in FHIR Specification"), a FHIR Infrastructure ticket
already resolved Persuasive-with-Modification. FHIR-48692 added usage notes
to the Request and Event patterns saying exactly this: identifiers are not
classifiers, and a "request" identifier on a result must be conveyed via
`[result].basedOn.identifier`, not `[result].identifier`. FHIR-54545 is the
follow-up that asks Orders & Observations to apply that already-agreed
clarification to the concrete DiagnosticReport guidance/example, and to
remove (or rework) any text that currently instructs implementers to put
`PLAC`/`FILL` directly on `DiagnosticReport.identifier`.

## Details

**Description:**

> The identifier datatype has a type element that may be used to distinguish
> the identifiers assigned by the requester and the performer of the request
> (known as the 'Placer' and 'Filler' in the HL7 Version 2 Messaging
> Standard). **Use the identifier type code "PLAC" for the Placer Identifier
> and "FILL" for the Filler identifier** as is shown in the example below:

According to
[Valueset-identifier-type — FHIR v6.0.0-ballot3](https://build.fhir.org/valueset-identifier-type.html),
PLAC and FILL represent *"An identifier for a request"*. As such they cannot
be identifiers for the report as this is an event. They should be the
identifier for the request (ServiceRequest) and could be included in
`basedOn.identifier.type`.

## Linked Jira Tickets

### [FHIR-48692: Clarification on the Use of Identifiers in FHIR Specification](https://jira.hl7.org/browse/FHIR-48692)

FHIR Infrastructure change request from the same reporter (Bas van den
Heuvel), resolved **Persuasive with Modification** with a vote of 11-0-1.
The agreed resolution adds a "Use of `identifier` and similar elements"
notes section to both the Request and Event patterns, and points the
Identifier datatype page at it. Among the agreed bullets: identifiers are
not classifiers/categories, identifiers must not carry the identity of
*other* business objects, and "if there is a need to convey a 'request'
identifier on a result where the same request might produce multiple
results, the 'request' identifier should never appear in `[result].identifier`
— instead, convey it on `[result].basedOn.identifier`." This is the exact
rule FHIR-54545 wants applied to the DiagnosticReport (and related
event-pattern) guidance/examples.

Reporter: Bas van den Heuvel. Assignee: Grahame Grieve. Commenters:
Lloyd McKenzie (mckenzie). Status: Resolved – change required (still

Implementer thread (Feb 2026) walking through a real lab/genomics workflow

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a non-substantive clarification on the DiagnosticReport
resource (and any other event-pattern resources whose `.identifier`
guidance/examples currently steer implementers to PLAC/FILL on the event):

1. Remove the example/text on `DiagnosticReport.identifier` that says
   *"Use the identifier type code 'PLAC' for the Placer Identifier and
   'FILL' for the Filler identifier"* on the report itself.
2. Replace it with guidance that defers to the new "Use of `identifier`
   and similar elements" notes added by FHIR-48692, and shows the placer
   and filler identifiers as `ServiceRequest.identifier` instances with
   `Identifier.type.coding = PLAC | FILL`, surfaced on the report via
   `DiagnosticReport.basedOn.identifier` (carrying the same
   `Identifier.type` so consumers can still discriminate placer from
   filler without dereferencing the ServiceRequest).
3. Update the JSON example currently rendered on the
   DiagnosticReport.identifier section so it no longer puts PLAC/FILL on
   `DiagnosticReport.identifier`. The replacement example should show
   PLAC/FILL on `DiagnosticReport.basedOn[].identifier` (and, where the
   contained ServiceRequest is shown, on `ServiceRequest.identifier`).
4. Sweep the other event-pattern resources O&O owns (notably
   Observation, ImagingStudy, Specimen) for the same wording/example and
   apply the equivalent edit.

#### Justification

This is exactly what FHIR-48692 already resolved in FHIR Infrastructure,
voted 11-0-1 Persuasive-with-Modification: identifiers are not classifiers
and a "request" identifier on a result must be conveyed via
`[result].basedOn.identifier`, not `[result].identifier`. Lloyd McKenzie's
final comment on FHIR-48692 (2025-11-06) explicitly notes that the
Identifier elements still need to be repointed to the new notes — applying
that on DiagnosticReport is a direct execution of an already-agreed
clarification rather than a new design decision. The Identifier-type
codes `PLAC` and `FILL` in `valueset-identifier-type` are defined as
"An identifier for a request", which is internally inconsistent with using
them on a report's own `.identifier`. The Zulip threads from UK
implementers (NHS NRL, DiagnosticReport.code) show this is a real,
recurring source of implementer confusion that the spec should remove,
not perpetuate.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `DiagnosticReport.identifier` mechanically able to carry PLAC/FILL
(do not impose a constraint), but rewrite the guidance on
`DiagnosticReport.identifier` so it:

1. States that the *normative* home for placer/filler/accession identifiers
   is the underlying request (`ServiceRequest.identifier` with
   `Identifier.type` = PLAC/FILL/ACSN), reflected on the report via
   `DiagnosticReport.basedOn.identifier`.
2. Acknowledges the historical V2 ORU-derived practice of placing the
   filler number in OBR/DiagnosticReport, and notes that systems migrating
   from V2 may continue to populate `DiagnosticReport.identifier` with
   PLAC/FILL for backward compatibility, but that doing so in greenfield
   FHIR designs is discouraged and not a conformant interpretation of
   PLAC/FILL semantics.
3. Adds an inline cross-reference to the new "Use of `identifier` and
   similar elements" section produced by FHIR-48692, and a worked example
   showing the recommended (basedOn-based) shape alongside the
   discouraged-but-tolerated legacy shape.
4. Optionally introduces an explicit best-practice invariant or
   `usage-note` extension on `DiagnosticReport.identifier` rather than
   touching cardinality or bindings.

#### Justification

PLAC/FILL on `DiagnosticReport.identifier` is widespread in deployed
ORU-derived FHIR implementations (the NHS NRL Zulip thread is one of
several examples). Forbidding it outright — even via strongly-worded
guidance — risks breaking real-world content and existing IGs (UK Core,
DataStandards Wales, US/imaging programmes), and would be hard to call
"non-substantive". A scoped guidance rewrite that names the correct shape
as the recommended one, while explicitly tolerating the legacy V2 shape,
addresses the reporter's underlying concern (the spec should not *teach*
implementers to misuse PLAC/FILL) without invalidating instances that
already use that pattern. This also preserves the freedom of an
implementation guide to tighten the rule further if its community is ready.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-54545 as **Not Persuasive** on the grounds that the
clarification it requests is already delivered, in normative form, by
FHIR-48692's resolution at the Identifier datatype / Request & Event
pattern level, and that the existing DiagnosticReport text is acceptable
"shorthand" inherited from V2 messaging. Close without changing the
DiagnosticReport page or example.

#### Justification

FHIR-48692 has already added the necessary cross-cutting guidance to both
the Request and Event patterns (and is being wired into the Identifier
datatype page). One could argue that resource-by-resource sweeps to repeat
the same clarification create churn, increase the risk of
spec-text-vs-pattern-text drift, and add no new normative content.
PLAC and FILL on `DiagnosticReport.identifier` reflect decades of
real-world V2 ORU usage; declining preserves backward compatibility with
that installed base and avoids any appearance of a breaking interpretation
late in the R6 ballot cycle.

---

### Recommendation

**Recommended disposition:** A

The conceptual decision has already been taken in FHIR-48692 with broad
consensus (11-0-1) and Lloyd McKenzie has flagged that the Identifier
elements still need to be repointed to the new notes. FHIR-54545 is the
concrete follow-through on the resource that most visibly *teaches* the
deprecated pattern; leaving the misleading example in place (Disposition C)
re-creates exactly the confusion the FHIR-48692 resolution was meant to
resolve, and Disposition B's "tolerated legacy shape" risks hard-coding a
"wrong-but-OK" path that future implementers will read as endorsement.
The Zulip evidence (NHS NRL and DiagnosticReport.code threads) shows
implementers are *already* discovering the basedOn.identifier.type pattern
on their own and asking whether it is correct — answering "yes, and here
is the example" in the core spec is the smallest, highest-value edit.
The change is bounded (one page plus a short sweep of sibling event-pattern
resources O&O owns), is non-substantive in the FHIR-48692 sense, and aligns
the DiagnosticReport guidance with the value-set-level definition of
`PLAC`/`FILL` as request identifiers. The workgroup should adopt
Disposition A and coordinate the edit with FHIR Infrastructure so the
DiagnosticReport text and the new Identifier-datatype notes ship together.
