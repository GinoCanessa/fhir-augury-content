## Summary

FHIR-48394 requests a standardized FHIR mechanism to *prompt* (interrogate)
a device for its current settings on demand. Today, devices unilaterally
choose when to publish settings and which settings to publish; there is no
defined way for a consumer (clinician, EHR, downstream system) to ask a
device to disclose its current configuration. Use cases include capturing
how a device is configured at the start of a procedure and obtaining the
current configuration of an unconscious patient's insulin pump. The
ticket itself proposes that this could be addressed via a new FHIR
operation, parameterized by the *kinds* of settings of interest (e.g.
end-user configurable, software/firmware version, calibration).

This ticket is the read-side counterpart to its sibling
[FHIR-48395](https://jira.hl7.org/browse/FHIR-48395) ("Mechanism to
request device settings *changes*"), filed by the same reporter on the
same day, which asks for the write-side analog — a way to request that a
device *change* its settings. Both tickets are scoped to the OO
Incubator and tagged Joint-OO-PC. Both sit within the broader analysis
captured in [FHIR-39688](https://jira.hl7.org/browse/FHIR-39688) (Stephen
Chu's longer-running "Device resource needs semantically accurate
structure for setting device function parameters"), where OO has already
observed that Device / DeviceMetric / Observation can convey *actual*
settings but no agreed-upon FHIR mechanism exists to instruct the device
or to interrogate it programmatically. The Devices WG has previously
flagged that device command-and-control over FHIR is high-risk and
warrants careful analysis before any normative mechanism is defined; the
same caution applies to a structured "give me your current settings"
operation, since the response shape, security context, and reliance on
device-side support are non-trivial.

The most recent OO motion (2026-01-30) on this ticket was 11–0–1 (in
favor – against – abstain), and the prior "Considered for Future Use"
resolution from 2025-10-27 was reverted on 2026-02-02, leaving the
ticket back in **Triaged** status pending fresh disposition.

## Details

**Description:**

A request came though from Stephen Chu  to be able to request a device
for its current settings.

At the present, a device chooses when to report its settings, and which
settings are reported. There is no way to ask for the settings. The
request is to be able to obtain device settings on demand. This would be
helpful to record how a device is configured at the start of a
procedure, or to to ask how an unconscious patient's insulin pump is
configured.

A suggestion was made that this could be done via a new operation,
perhaps taking a parameter of "kinds" of settings (end-user configuable,
software/firmware version, calibration, ...).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Define a new FHIR operation `Device/$get-settings` (instance- and
type-level) in the **OO Incubator** specification that, when invoked,
asks the receiving system to return the current settings of the
identified Device(s). The operation would take an input `kind` parameter
(0..*, `code`, bound to a new value set drafted in the Incubator with
codes such as `user-configurable`, `firmware-version`, `software-version`,
`calibration`, `safety-limits`, `all`) and an optional `asOf`
`dateTime`. The output would be a `Parameters` resource carrying:

- `device` — the Device resource as known by the responder, with
  `Device.property` populated for the requested kinds;
- `metric` — 0..* `DeviceMetric` resources for the addressable settings;
- `observation` — 0..* `Observation` resources representing the current
  reported values of those settings (using the existing
  Device/DeviceMetric/Observation pattern OO already endorsed in
  FHIR-39688 comment 64781).

Co-author the OperationDefinition with FHIR-48395 so that
`$get-settings` (read) and the change-request mechanism (write) share a
common `kind` value set, common security guidance, and a common section
in the Incubator narrative on "Device settings interrogation and
control". Status of the operation is **Draft / trial-use within
Incubator**, explicitly not normative in R6.

#### Justification

The reporter's ticket explicitly proposes "a new operation, perhaps
taking a parameter of 'kinds' of settings", and the recent OO motion on
this ticket carried 11–0–1, indicating strong working-group support for
moving the request forward. The Incubator (rather than Core) is the
right home because: (a) FHIR-39688 already concluded the broader problem
needs more analysis than an R6 ballot comment can carry; (b) Devices WG
has flagged command-and-control as high-risk, and an Incubator-scoped
operation lets OO and the Devices WG iterate on the response shape,
security guidance, and conformance expectations without committing the
core spec; and (c) re-using the existing
Device/DeviceMetric/Observation triad keeps the change additive rather
than restructuring the Device resource itself.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** define a new operation. Instead, document a profile-and-Task
pattern in the OO Incubator that solves the same use case using existing
machinery:

- A `DeviceSettingsInterrogationRequest` profile on **Task** (or on
  **ServiceRequest**, if HCP/Devices prefer the order paradigm) whose
  `focus` is the Device, whose `code` is a new code (e.g.
  `device-settings-interrogation`), and whose `input` parameters carry
  the same `kind` codes proposed in Disposition A.
- A response pattern using a **Bundle** (type `searchset` or
  `collection`) of the responder's current `Device`, `DeviceMetric`, and
  `Observation` instances, returned via `Task.output` as a
  `Reference(Bundle)`.
- An IG-level narrative describing the conformance, security, and
  identity-of-responder expectations, and pointing implementers at the
  same artefacts that Disposition A would have built.

#### Justification

This re-uses resources OO has already agreed are the right home for
*actual* device settings (FHIR-39688 comment 64781) and avoids minting a
new operation before the Devices WG has signed off on FHIR-as-control-
channel semantics — a concern Elliot Silver explicitly recorded
(FHIR-39688 comment 64784). It also dovetails naturally with FHIR-48395,
which itself contemplated profiling/modifying Task. The trade-off
versus Disposition A is verbosity and indirection: clients have to round-
trip a Task and chase a Bundle reference rather than calling a single
operation, and there is no first-class capability statement bit
declaring "this server supports device settings interrogation".

---

### Disposition C: Decline

#### Proposal

Decline FHIR-48394 (and, if dispositioned in tandem, FHIR-48395) at
this time, on the basis that:

1. The broader problem of FHIR-as-device-command-and-control is already
   captured under FHIR-39688 (Highest priority, FHIR Core), where OO
   has formally concluded the work is "too big to address as a ballot
   comment to R6" and "more appropriate for an IG project" until
   implementation interest is demonstrated.
2. The current Device / DeviceMetric / Observation triad already
   supports *recording* settings; the gap FHIR-48394 names is not a
   modelling gap but an interaction-pattern gap, which is best resolved
   inside an IG with named implementers, not by adding an operation to
   the spec/incubator before the use cases and risk analysis are
   complete.

Action: close FHIR-48394 as **Considered for Future Use**, and route
the work into the in-flight Device-settings IG analysis tracked under
FHIR-39688.

#### Justification

The Devices WG has explicitly stated (FHIR-39688 comment 64784) that
device command-and-control over FHIR is high-risk and warrants
analysis before standardization, and an interrogation operation
(`$get-settings`) carries similar risk surface (privacy of device
configuration, identity of responder, vendor-specific semantics of
settings codes). Standing up an operation before that analysis is
finished risks publishing something that the Devices WG and field
implementers will not honor. The 2025-10-27 OO motion already chose
"Considered for Future Use" for this exact reason; the 2026-02-02
revert reopened the question but did not introduce new evidence that
the underlying risk has been addressed.

---

### Recommendation

**Recommended disposition:** B

Disposition B (the Task/profile pattern in the OO Incubator) is the
best balance between the strong WG support shown by the most recent
11–0–1 motion and the legitimate, Devices-WG-recorded caution about
turning FHIR into a device control channel before the analysis under
FHIR-39688 is finished. It keeps FHIR-48394 *moving* — implementers
get a documented, conformance-testable pattern they can pilot — while
explicitly using the Device/DeviceMetric/Observation triad OO has
already endorsed for actual settings (FHIR-39688 comment 64781) and
avoiding the minting of a new core/Incubator operation that would have
to be lived with later. It also pairs naturally with FHIR-48395, where
Task/ServiceRequest profiling was already on the table. If the OO–PC
joint discussion concludes that an operation is genuinely needed (e.g.
because the Bundle-via-Task round trip is unacceptable for time-
critical procedure-start use cases), the Incubator can subsequently
graduate the pattern into the operation defined in Disposition A
without disturbing the data model.
