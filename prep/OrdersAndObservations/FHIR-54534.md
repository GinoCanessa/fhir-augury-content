## Summary

The reporter finds the DocumentReference narrative text describing how a
document target can be retrieved via a "service call" (e.g., IHE
XDS.b RetrieveDocumentSet, DICOM WADO-RS, or an HL7 v2 message query) to
be ambiguous. Specifically, the trailing clause that "the way each of
these service calls works must be specified in some external standard
or other documentation" is unclear: the reporter wants to know whether
this means that the source of the document could be a WADO/XDS/V2
endpoint and the consumer is expected to bring out-of-band knowledge of
those external specifications in order to retrieve the bytes — and, if
so, how the consumer is supposed to know which protocol applies. The
ask is to clarify the behavior or rewrite the paragraph to remove the
ambiguity. There are no cross-referenced Jira tickets, Zulip threads,
or GitHub items associated with this ticket at the time of preparation.

## Details

**Description:**

> The document that is a target of the reference can be a reference to
> a FHIR document served by another server, or the target can be stored
> in the [FHIR Binary Resource](https://build.fhir.org/http.html#binary),
> or the target can be stored on some other server system. The document
> reference is also able to address documents that are retrieved by a
> service call such as an
> [XDS.b RetrieveDocumentSet](https://profiles.ihe.net/ITI/TF/Volume1/ch-10.html),
> or a DICOM
> [WADO-RS](ftp://medical.nema.org/medical/dicom/final/sup161_ft.pdf)
> exchange, or an
> [HL7 V2](http://www.hl7.org/implement/standards/product_brief.cfm?product_id=185)
> message query **— though the way each of these service calls works
> must be specified in some external standard or other documentation.**

I'm not sure what is meant by this sentence. Does it mean that the
source could be WADO, XDS, .. and that the user needs knowledge of
these specifications in order to access them?

If that is the case, in what does the user know it has to do so?
Please clarify behavior or update text to be less ambigue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reword the offending sentence in the DocumentReference narrative
(currently in the resource's "Notes" / Boundaries-and-Relationships
prose discussing how the document target may be retrieved) to make
explicit that:

1. `DocumentReference.content.attachment.url` MAY point to an endpoint
   served by a non-FHIR protocol (IHE XDS.b RetrieveDocumentSet, DICOM
   WADO-RS, HL7 v2 message query, etc.) in addition to a FHIR
   `Binary`-style URL.
2. Identifying *which* protocol applies is **out of scope for the
   FHIR DocumentReference resource itself** and is determined by:
   - the URL scheme / well-known endpoint pattern used (e.g., a WADO-RS
     URL, an XDS.b SOAP endpoint, an `mllp:` URL),
   - the `DocumentReference.content.attachment.contentType` /
     `DocumentReference.content.format` codings, and/or
   - the implementation guide or trust framework that the exchange
     partners have agreed to use.
3. Consumers who do not understand the referenced external protocol
   are not required to dereference the URL — the reference is still
   useful as metadata.

Replace the closing clause "though the way each of these service calls
works must be specified in some external standard or other
documentation" with a sentence along the lines of: *"In those cases the
retrieval protocol is defined by the external standard (e.g., IHE
XDS.b, DICOM WADO-RS, HL7 v2 messaging) rather than by FHIR; consumers
identify the applicable protocol from the URL form, the
`content.format` coding, and/or the governing implementation guide."*

#### Justification

The text as it stands is genuinely ambiguous — the reporter, an
implementer, could not determine what behavior was being described,
which is the canonical signal that an editorial clarification is
warranted. The proposed wording does not change the semantics of
DocumentReference; it just names the existing mechanisms (URL form,
`content.format`, IGs) that implementers already use to disambiguate
non-FHIR endpoints. This is consistent with how FHIR generally
delegates protocol-specific behaviors to external standards while
keeping the resource itself protocol-agnostic.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only reword the prose, also tighten the binding signal in
the resource so consumers have a normative way to detect the protocol
without parsing the URL. Specifically:

- Keep the editorial fix from Disposition A.
- Additionally clarify in the element definition for
  `DocumentReference.content.format` (and/or `attachment.contentType`)
  that when the target is reachable only via a non-FHIR service call,
  the format coding **SHOULD** be populated with a code that
  identifies the retrieval protocol (e.g., a code from the existing
  format-code value sets such as the IHE `urn:ihe:iti:xds-sd:*` family
  or DICOM WADO codes).
- Add a short non-normative example to the DocumentReference page
  showing an XDS.b- or WADO-RS-served document with the corresponding
  `content.format` coding.

#### Justification

A pure prose fix (Disposition A) clarifies the *intent* but still
leaves implementers free to omit any machine-readable signal of which
protocol governs the URL, perpetuating the underlying interoperability
problem the reporter is reacting to. Strengthening the guidance on
`content.format` gives consumers a deterministic way to dispatch to
the right retrieval stack, while still being non-breaking (SHOULD, not
SHALL, and only when relevant). The trade-off is more editorial work
and the need to confirm the relevant format-code value set covers the
protocols listed in the paragraph (XDS.b, WADO-RS, V2 query); if that
coverage is incomplete, the disposition implies follow-up coordination
with ITI / Imaging Integration / InM.

---

### Disposition C: Decline

#### Proposal

Decline the change as **Not Persuasive – Working as Designed /
Editorial-only**. The paragraph in question is informative narrative
that already correctly states the situation: FHIR does not redefine
how XDS.b, WADO-RS, or HL7 v2 work, and a consumer that wants to
dereference such a URL must consult the relevant external standard.
No normative behavior changes, and no spec text changes.

#### Justification

DocumentReference deliberately treats `attachment.url` as opaque from
FHIR's point of view; it is a long-standing principle that FHIR does
not re-specify other HL7/IHE/DICOM protocols. The current sentence
communicates exactly that. An implementer who does not recognize a
WADO-RS or XDS.b endpoint is not expected to retrieve it — they can
still rely on the metadata (type, category, date, author, format
coding) without performing the retrieval. Declining keeps the
specification stable and avoids opening a thread of similar
clarification requests for every non-FHIR protocol mentioned in
informative prose.

---

### Recommendation

**Recommended disposition:** A

The reporter is not arguing for a behavior change — they are reporting
that an experienced implementer cannot parse the existing sentence,
which is the textbook trigger for an editorial fix. Disposition A
addresses the actual complaint (ambiguous prose) at minimal cost and
risk: it is non-breaking, requires no value-set work, and does not
touch any cross-workgroup machinery. Disposition B is attractive in
principle but bundles a normative-ish change onto what was filed as a
clarity ticket; that is better tracked as a separate enhancement
ticket once the editorial fix lands, especially given the reporter
has not asked for stronger format-code guidance. Disposition C is
defensible on "FHIR doesn't re-specify other standards" grounds, but
declining a clearly-articulated readability complaint from an
implementer is the kind of disposition that tends to be re-filed and
should be avoided when a low-cost rewording is available.
