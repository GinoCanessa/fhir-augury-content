## Summary

The reporter notes an internal contradiction in the Vital Signs guidance
within FHIR Core. The narrative section (third bullet) states that the
terminology bindings used in the Vital Signs profiles are **extensible**,
while the individual Vital Signs profiles themselves bind their coded
elements (notably `Observation.code`) with strength **preferred**. The
ticket asks Orders & Observations to make the two statements consistent —
either by amending the bullet to say "preferred" or by changing the
profile bindings to "extensible". No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced for this issue; it is a
self-contained editorial / normative consistency request.

## Details

**Description:**

> The third bullet of the section states that the terminology bindings
> are extensible. However the specific profiles specify it as preferred.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Vital Signs narrative page (the page that today carries the
"third bullet" describing the binding policy) so that it states the
binding strength as **preferred**, matching the actual binding strength
declared on `Observation.code` in the Vital Signs profiles
(`vitalsigns`, `bp`, `bodyheight`, `bodyweight`, `bodytemp`, `heartrate`,
`resprate`, `oxygensat`, `headcircum`, `bmi`, `vitalspanel`, etc.). No
profile, value set, or `StructureDefinition` element changes are made;
only the prose bullet on the Vital Signs guidance page changes.

#### Justification

The profile artifacts are the normative definition of the binding
strength; the narrative bullet is a description of the profile family.
Aligning the prose to match the artifacts is the lowest-risk fix, has
no implementation impact, does not break any existing
`StructureDefinition`, value set, or conformance resource, and does not
trigger downstream regeneration of profile snapshots. It is also
consistent with the established editorial practice that prose should
reflect the conformance artifacts rather than override them. This is
the kind of "make the text match the model" correction that O&O
typically resolves as Persuasive with no normative impact.

---

### Disposition B: Alternative Approach

#### Proposal

Change the binding strength on the Vital Signs profile elements
(primarily `Observation.code` in `vitalsigns` and the specific child
profiles) from `preferred` to `extensible`, so that the artifacts match
the existing narrative bullet. This would tighten conformance: any
implementer using a code outside the Vital Signs value set would be
required to demonstrate that no suitable code exists in the bound
value set, rather than merely being encouraged to use one.

#### Justification

If the narrative bullet was written deliberately and reflects the
working group's actual intent for Vital Signs ("you SHALL use these
codes unless none fits"), then the contradiction is a defect in the
profiles, not in the prose, and the profiles should be corrected.
Choosing this disposition would, however, be a normative tightening:
existing implementations that today legitimately use non-Vital-Signs
codes for an `Observation` claiming conformance to a Vital Signs
profile would become non-conformant. That trade-off should be weighed
against community history (LOINC adoption is wide but not universal,
and several jurisdictions deliberately use non-LOINC vital-signs codes).
Backward compatibility and the breaking-change implications across
ballots — especially for IGs that derive from these profiles — are the
key factors against this option.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the basis that the
contradiction is not actionable as written: it does not name the page
or section, does not cite which profiles, and does not propose which
direction the fix should go. Ask the reporter to resubmit with a
specific page anchor and recommended direction.

#### Justification

In principle a workgroup can decline a ticket that lacks specificity,
but in this case the contradiction is real, easy to locate (the Vital
Signs guidance page and the published Vital Signs profiles), and the
fix is small. Declining would leave a known internal inconsistency in
the spec for another ballot cycle and would likely be re-filed.
Declining is therefore the weakest option and is presented mainly for
completeness.

---

### Recommendation

**Recommended disposition:** A

Disposition A (update the narrative bullet to say "preferred") is the
correct call. The Vital Signs profiles have shipped with `preferred`
bindings across multiple FHIR releases and are widely implemented
against that strength; there is no signal in the cross-references,
Zulip, or GitHub that the working group ever intended to require an
`extensible` binding, nor any community pressure to tighten it.
Disposition B would be a normative break for very little benefit, and
Disposition C would leave a real defect unresolved. Accepting and
fixing the prose closes the contradiction, costs essentially nothing
to implementers, and matches the long-standing artifact behaviour.
