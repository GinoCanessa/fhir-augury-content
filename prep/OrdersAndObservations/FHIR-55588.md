## Summary

The ticket asks the Clinical Order Workflow (COW) IG to define how a
fulfiller (e.g., a nurse or other performing clinician) can propose an
*alternative*, *continuation*, or *additional* request while fulfilling
an existing order, and how that proposal should be linked back to the
originating request. The reporter wants explicit guidance on which
elements, resources, and references convey the relationship — should the
fulfiller-originated request use `basedOn` to point back at the original
request, should the linkage flow through `Task.output`, or should a
different mechanism (e.g., `triggeredBy`, `reasonReference`) be used? The
reporter calls out three concrete fulfillment scenarios that the IG
should be able to express: (1) a nurse who is permitted to authorize an
order outright, (2) a nurse acting from a protocol whose action implies
an implicitly approved order, and (3) a nurse who needs an order to be
authorized by another role before acting. There are no linked or
cross-referenced Jira tickets, but the ticket is referenced from a
broader workflow Zulip thread on "intent-driven interaction" where the
reporter notes this issue is in scope of COW.

## Details

## Related Zulip Discussions

### [workflow > Intent-driven interaction](https://chat.fhir.org/#narrow/stream/workflow/topic/Intent-driven%20interaction/near/578941201)

Lionel Cremer described an architecture in which a FHIR Gateway exposes
intent-driven user actions (approve, reject, complete, propose) that are
translated by a FHIR-agnostic backend into coordinated updates across
several persisted resources (ServiceRequest, Task, etc.), without the
"intent" itself being persisted as a resource — and asked which FHIR
workflow pattern best fits. Jose Costa Teixeira explicitly mapped
Lionel's first use case (proposal review) onto FHIR-55588, noting it is
in scope for COW and likely involves an order with `intent = proposal`
and possibly a Task to "please create"; he also pointed at a
cancellation-request task profile in the COW IG for the radiology
revocation case (and at the related ticket FHIR-56051 for the "mark
complete" pattern). Lloyd McKenzie agreed with Jose: standard REST plus
appropriate use of Request `intent` values and `Task` is the recommended
pattern over custom operations or messages — proposals are RESTful
creates, evaluation decisions are status PUTs or POSTs of a new
prescription, completion is a RESTful update on the order, and
substitution/rejection in radiology are status updates. Consensus in
the thread is that COW should describe these flows with existing FHIR
machinery (intent values, status transitions, Task) rather than
introducing a new "intent" resource; the open question for FHIR-55588
remains exactly which linkage element (`basedOn`, `triggeredBy`,
`reason*`, or `Task.output`) the IG should normatively recommend for
fulfiller-proposed alternative/continuation/additional requests.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Clinical Order Workflow IG to add an explicit
"Fulfiller-proposed alternative / continuation / addition" section
under the fulfillment workflow narrative. The section will:

1. Define the three fulfiller scenarios from the ticket (autonomous
   authorization, protocol-driven implicit authorization, requires
   upstream authorization) and map each to a concrete request lifecycle:
   - Autonomous authorization → fulfiller POSTs a new `ServiceRequest`
     with `intent = order` (or `original-order`), `requester` = the
     fulfilling clinician, and `basedOn` pointing back at the
     originating request.
   - Protocol-driven implicit authorization → same as above but with
     `instantiatesCanonical` set to the protocol/PlanDefinition and a
     coded `reasonCode` indicating the protocol trigger.
   - Requires upstream authorization → fulfiller POSTs a `ServiceRequest`
     with `intent = proposal` plus a `Task` (using the COW
     fulfillment-request task pattern) carrying the proposal to the
     authorizing role; on approval the proposal is promoted to an
     order and `basedOn` chains the new order to the proposal and to
     the originating request.
2. Normatively recommend `ServiceRequest.basedOn` (and the equivalent
   element on other request resources) as the primary linkage element
   from a fulfiller-proposed request back to the originating request,
   with `Task.focus` referencing the new request and the originating
   `Task.output` populated with the new request when the proposal arises
   inside an existing fulfillment task.
3. Use `ServiceRequest.reason[Code|Reference]` and (where R5+ is in
   scope) `triggeredBy` to express *why* the new request was raised, and
   give a worked example matching the comment from costateixeira (wound
   care → analgesics for incidentally discovered acute infection).
4. Add at least one example bundle per scenario to the IG examples and
   cross-link to `Task.output` in the COW fulfillment task profile(s).

#### Justification

This is the linkage the reporter and the workgroup discussion converged
on: `basedOn` is the standard, well-understood mechanism for chaining
request resources, and the Zulip workflow thread (Lloyd, Jose) endorses
modelling these flows with standard REST + `intent` + `Task` rather
than inventing new mechanisms. The proposal honours the ticket's
explicit questions ("`.basedOn` the original request? Do we fill
`Task.output`?") by answering both: `basedOn` is the canonical link;
`Task.output` is used when the fulfiller-proposed request is produced
inside an active fulfillment task. Calling out the three nurse scenarios
keeps the guidance grounded in the ticket and gives implementers
unambiguous patterns. Adding a worked example addresses costateixeira's
follow-up comment directly.

### Disposition B: Alternative Approach

#### Proposal

Rather than adding new IG narrative that re-derives the linkage rules,
profile the COW fulfillment Task to formalise the
"fulfiller-proposed-request" pattern and let the request resource carry
only minimal additions:

- Define a new COW Task profile (or extend the existing fulfillment
  task) with `Task.code` = "propose-additional-request" and a constraint
  that `Task.output` SHALL contain the fulfiller-proposed
  `ServiceRequest` (or `MedicationRequest`, etc.).
- Constrain that proposed request to have `intent = proposal`,
  `basedOn` referencing the originating request, and `triggeredBy`
  (R5+) pointing at the `Task` that carried the proposal — collapsing
  the three nurse scenarios into one uniform interaction:
  *fulfiller always proposes via Task.output*, and the *authorization
  policy* (autonomous, protocol-implicit, or requires upstream sign-off)
  is conveyed by `Task.businessStatus` and `Task.owner` rather than by
  branching the request shape.
- Provide a single worked example and a decision table mapping the
  three nurse scenarios to the resulting `Task.status` /
  `ServiceRequest.intent` transitions.

#### Justification

This concentrates the new normative content in one place (a Task
profile) rather than spreading new prose across the fulfillment
narrative, which is easier to validate, easier to test against, and
easier to evolve. It also matches Lloyd's guidance in the Zulip thread
that workflow-initiating intents should be modelled with `Task`, and
keeps the "what's a proposal vs an order" decision out of the request
resource itself. The trade-off vs. Disposition A is that implementers
who are not using Task for fulfillment lose explicit guidance on the
pure-REST case; the IG would have to flag that subset as "not covered
by this profile — use base FHIR `basedOn` semantics."

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Already addressed by base FHIR." The base
specification already defines `ServiceRequest.basedOn` (and parallel
elements on other request resources), `intent` value-set entries
(`proposal`, `plan`, `order`, `original-order`, `instance-order`),
`reasonCode`/`reasonReference`, `triggeredBy` (R5+), and
`Task.output`/`Task.focus` for carrying derived requests. No new IG
mechanism is required; the COW IG can rely on the base definitions
and link to them.

#### Justification

The Zulip thread shows that experienced workflow modellers (Lloyd,
Jose) consider these flows expressible with existing FHIR machinery,
and the ticket itself is phrased as a clarification question ("what are
the elements, resources and references between those?") rather than a
demonstrated gap. Declining with a pointer to the base elements avoids
adding redundant normative text to the COW IG and keeps it consistent
with how other workflow IGs handle proposal/authorization patterns.

---

### Recommendation

**Recommended disposition:** A

The ticket is asking the COW IG for *guidance*, not for a missing
element — but the IG is precisely where that guidance belongs.
Disposition C is technically defensible (the base elements exist) but
leaves implementers to re-discover the same patterns the workgroup
has already discussed, which is exactly the gap the COW IG is meant
to close. Disposition B is attractive architecturally and would
produce cleaner conformance, but it presupposes that every COW
implementer is using `Task` to mediate fulfillment; the ticket's
nurse-autonomous-authorization scenario reads more naturally as a
plain RESTful request flow without a Task wrapper, which Disposition B
would either exclude or force into Task. Disposition A directly
answers the reporter's two literal questions ("`.basedOn`? `Task.output`?"),
covers all three nurse scenarios with concrete request shapes, and
incorporates the Zulip-thread consensus that `intent` + `basedOn` +
`Task` are the right primitives. It also leaves room to fold in the
Task-profile work from Disposition B as a follow-up if the workgroup
later wants to tighten conformance.
