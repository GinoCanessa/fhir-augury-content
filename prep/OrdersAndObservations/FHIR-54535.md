## Summary

The reporter is asking the workgroup to clarify the normative behaviour of
the `_content` search parameter when it is applied to `DocumentReference`,
specifically with respect to the elements
`DocumentReference.content.attachment.data` and
`DocumentReference.content.attachment.url`. The current core specification
states that `_content` "shall search across" these elements, but it is
silent on what that actually means in practice — should a server search
the raw Base64 payload of `attachment.data`, the decoded/de-serialised
payload, the literal text of `attachment.url`, or the (potentially
remote) content the URL points to? The reporter assumes that fetching
and indexing referenced content is not required because it would be
prohibitively expensive, and is asking for the spec to make that
explicit.

No prior Jira tickets are cross-referenced by the indexer, but a 2025-01
Zulip discussion in `#conformance > Behaviour of _content search param
on DocumentReference` covered exactly this question and points back to
the historical change request **FHIR-20417** (John Moehrke's original
proposal that introduced the current "shall" wording on the
DocumentReference page). FHIR-20417 itself is not present in the local
cache, but its existence and intent are surfaced through the Zulip
thread.

## Details

**Description:**

> - The _content search parameter shall search across the
>   DocumentReference.content.attachment.data, and
>   DocumentReference.content.attachment.url.

When searching in `Attachment.data`, does this mean that the search is
in the Base64 content or in the deserialised content? Does it search in
the raw data or does it need to understand the content as well (I hope
not)?

Related to searching on the url, does this mean they search for text in
the url or in the content of the URL? I assume the server is not
required to support searching in the content pointed to by the URL.
That would be very expensive. Please clarify.

## Related Zulip Discussions

### [#conformance > Behaviour of _content search param on DocumentReference](https://chat.fhir.org/#narrow/stream/conformance/topic/Behaviour%20of%20_content%20search%20param%20on%20DocumentReference)

Simone Heckmann opened the topic on 2025-01-10 with essentially the
same question the reporter is now raising in FHIR-54535: does `_content`
on `DocumentReference` search the embedded `attachment.data` (and if so,
the Base64 string or the decoded bytes?), and does it search the
`attachment.url` literal or the dereferenced remote content? John
Moehrke responded that behaviour is highly implementation-dependent:
`_content` is defined as searching "across the entire content", but
since `.data` is Base64 and the URL points to (potentially binary)
remote bytes, expecting a server to do a true full-text search across
all of that is unrealistic — he suggested an Operation would be a
friendlier mechanism if true full-document search is required. Simone
pushed back that, from a client perspective, `_content` should ideally
behave as if the document were always embedded so callers don't have to
reason about server architecture. Gino Canessa noted the answer
depends on context: closed/known deployments can agree on richer
semantics, but in an open ecosystem ambiguity will hurt
interoperability. John then pointed to R5 text on the DocumentReference
page (https://build.fhir.org/documentreference.html#impl) where the
current "SHALL" wording lives, and observed that the SHALL "seems a bit
strong" given that not all servers can implement it; he suggested R6
may need to refine the wording. The thread closed without a hard
consensus but with broad agreement that the current spec text is
under-specified and probably overly strong.

Participants: Simone Heckmann, John Moehrke, Gino Canessa.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `DocumentReference` page in FHIR Core (the "Search Parameters"
/ implementer notes section currently at
`build.fhir.org/documentreference.html#impl`) to spell out exactly what
"`_content` SHALL search across `content.attachment.data` and
`content.attachment.url`" means:

- For `content.attachment.data`: servers SHALL apply `_content` against
  the **decoded** payload when the `contentType` is a text-based media
  type the server understands (e.g., `text/*`, `application/xml`,
  `application/json`, `application/fhir+xml/json`). Searching the raw
  Base64 string is not required and SHOULD NOT be assumed by clients.
  For binary or unsupported media types, the server is not required to
  search inside the payload.
- For `content.attachment.url`: `_content` SHALL match the literal URL
  string only. Servers are NOT required to dereference the URL and
  search the remote payload (and SHOULD NOT do so without an explicit,
  deployment-level agreement, due to performance, security, and
  liveness concerns).
- Clarify in the same paragraph that servers MAY additionally provide
  full-document indexing as an out-of-band capability, but interoperable
  clients cannot rely on it via `_content`.

This narrows but preserves the current SHALL.

#### Justification

This matches what implementers actually do today and what the Zulip
thread converged on as a reasonable common ground. It keeps the
existing search parameter (and existing servers stay conformant for the
text-attachment case), and it removes the two pieces of ambiguity the
reporter is asking about (Base64 vs. decoded, literal URL vs.
dereferenced URL) without inventing new machinery. Importantly, it
explicitly says that dereferencing referenced URLs is not required —
which is the reporter's main concern.

---

### Disposition B: Alternative Approach

#### Proposal

Soften the obligation on `_content` to a SHOULD (or "MAY, if the server
chooses to expose full-document search") and instead define a dedicated
`$find` / `$content-search` operation on `DocumentReference` (or
recommend an existing equivalent) for servers that want to offer real
full-document search across embedded data and/or dereferenced URLs.
Concretely:

- Change `DocumentReference`'s implementer notes so that `_content`
  matches the generic `_content` semantics inherited from the base
  search page (i.e., resource-level text), with no special obligation
  on `attachment.data` / `attachment.url`.
- Add a new operation (or formally bless an existing one) that takes a
  query string and returns matching `DocumentReference` resources, with
  an explicit parameter controlling whether the server may dereference
  remote attachments. Implementation of the operation is optional;
  capability is advertised via `CapabilityStatement`.

#### Justification

This is the path John Moehrke floated on Zulip ("more friendly as an
operation"). It preserves cross-resource consistency for `_content` —
which today is otherwise a server-implementation-detail parameter — and
moves the genuinely expensive, optional, security-sensitive behaviour
(decoding Base64 attachments, dereferencing remote URLs) into an
explicitly opt-in operation that servers can advertise. Trade-off vs.
A: it is a larger spec change and probably breaks any client that
relies on the current SHALL text, but it is more honest about
implementability and avoids forcing every conformant server to claim it
can search inside attachments.

---

### Disposition C: Decline

#### Proposal

Decline to change the spec. Treat the existing language on
`DocumentReference` as sufficient: `_content` is defined at the
resource-level on the base search page, the DocumentReference page
notes that it covers `attachment.data` and `attachment.url`, and the
exact mechanics are intentionally left to server implementations.
Resolve as "Not Persuasive" or "Already Specified", pointing the
reporter at the existing text in `search.html#content` plus
`documentreference.html#impl`.

#### Justification

The base `_content` definition already says behaviour is largely
implementation-defined, and there is no concrete interoperability
problem reported (no client/server pair that broke because of this
ambiguity, just a request for clarification). FHIR has historically
resisted prescribing how a free-text search engine works internally
because real-world servers vary enormously in their indexing
capabilities. Tightening the wording risks unintentionally invalidating
existing conformant implementations.

---

### Recommendation

**Recommended disposition:** A

The reporter is not asking the workgroup to invent new behaviour —
they are asking it to *say what the existing SHALL means*, particularly
that servers do not have to dereference remote URLs. Disposition A
delivers exactly that with minimal disruption: it preserves the
existing search parameter, codifies what the Zulip discussion
converged on (decoded text-attachment search yes; remote-URL
dereferencing no), and doesn't require any new operation. Disposition B
is a more architecturally clean answer and worth keeping on the table
for R6 if the workgroup wants to revisit `_content` semantics broadly,
but it is a much bigger change and would likely break clients that
currently rely on the SHALL wording. Disposition C leaves the original
ambiguity in place and effectively forces the next implementer to ask
the same question on Zulip again — which is the situation FHIR-54535
itself documents. The reporter's "Highest" priority and the recurring
nature of this question on `#conformance` argue for resolving the
ambiguity now rather than deferring.
