## Summary

The ticket asks the Orders & Observations work group to consider adding
an element to the R6 `BodyStructure` resource that captures the
*wholeness* of the described anatomical structure. The reporter notes
that important clinical use cases — in particular organ transplant —
turn on whether the body structure being described is a complete
(whole) organ/part, a partial structure, or (for example) a resected
or divided segment. The concept is already modeled in the HL7
Anatomical Inventory and Person Characteristics logical models, but
is not currently represented in the R6 `BodyStructure` resource.
No explicitly linked Jira tickets, cross-referenced tickets, or
cross-referenced GitHub items were returned for this ticket; the
request therefore stands on its own as an R6 ballot-window change
request against `BodyStructure`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an element to the R6 `BodyStructure` resource that records whether
the described structure is whole, partial, or otherwise
non-intact, aligned with the *wholeness* attribute already defined in
the HL7 Anatomical Inventory and Person Characteristics logical
models. Concretely, add `BodyStructure.includedStructure.wholeness`
(0..1, `CodeableConcept`) bound extensibly to a new value set rooted
in SNOMED CT concepts that distinguish *whole* vs. *partial/resected/
divided* body structures (e.g., descendants of 91723000 |Anatomical
structure| filtered by modifiers such as "whole" / "part of"). Place
the element on `includedStructure` rather than the root so that
multi-structure `BodyStructure` instances (e.g., a transplant donor
organ plus its vascular pedicle) can describe wholeness per included
structure. Mirror the element name and binding used in the
Anatomical Inventory logical model so mappings roll up cleanly.

#### Justification

The reporter's organ-transplant use case is concrete and cannot be
captured today: `includedStructure` carries `structure`, `morphology`,
`bodyLandmarkOrientation`, `spatialReference`, and `image`, none of
which express whether the donated/received organ is intact. Adding the
element on `includedStructure` is consistent with the direction Rob
Hausam opened in the *Observation on BodyStructure Morphology* thread
("if there actually is something missing here, then we could consider
adding a complex element"), and preserves the 0..1 cardinality story
for `Procedure.bodyStructure` that was deliberately defended in
J#51792. Aligning the element with the Anatomical Inventory /
Person Characteristics logical model honours HL7's commitment to
those logical models and avoids divergence between the resource and
the models it is supposed to realise.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a core element. Instead, publish a standard HL7 extension
`http://hl7.org/fhir/StructureDefinition/bodystructure-wholeness`
(0..1, `CodeableConcept`) targeting `BodyStructure.includedStructure`,
bound to the same value set proposed in Disposition A. Accompany the
extension with guidance in the Anatomical Inventory / Person
Characteristics logical-model mapping section of the `BodyStructure`
resource description that explicitly points implementers at the
extension when they need to carry the logical-model *wholeness*
attribute. If post-R6 usage data shows broad uptake, promote the
extension into a core element in a subsequent release.

#### Justification

The request arrived well into the R6 ballot window (created
2026-01-08, with the OO WG already triaging R6 `BodyStructure` ballot
comments as seen in the *R6 BodyStructure Review* thread). A core
addition at this stage risks destabilising a resource whose R6 shape
is actively being defended by the WG (`Procedure.bodyStructure`
cardinality, image-property placement). A standard extension delivers
the transplant use case without incurring breaking-change risk to
implementers already building against the R6 draft, preserves the
logical-model alignment the reporter cares about, and leaves a clean
upgrade path to a core element later. This is the same pattern OO has
used in the past when a single clinical community needs a concept the
broader base doesn't yet.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped. Rationale: insufficient
use cases and insufficient clarity on the value set. The single use
case offered — organ transplant — is real but narrow, and the
Anatomical Inventory / Person Characteristics logical models use
*wholeness* as one attribute among many; importing it in isolation
risks an under-specified element whose binding and cardinality will
have to be revisited as soon as a second use case (e.g., surgical
resection, amputation, partial grafts) arrives. Recommend the
requester work with a transplant-focused implementation community
(e.g., a transplant IG or the Anatomical Inventory project team) to
produce a worked example and a candidate value set, and re-submit
against a future release.

#### Justification

FHIR's stated design principle is that core elements should be added
only when justified by multiple, demonstrated implementer use cases
and when the binding is well understood. The ticket offers one use
case and no proposed value set or cardinality, and there is no Zulip,
GitHub, or Jira discussion showing broader demand for the concept in
`BodyStructure`. The transplant community already has adjacent work
(BiomedicalProduct / specimen handling) that may be a more natural
home for wholeness of a transplanted organ than the anatomic-location
resource.

---

### Recommendation

**Recommended disposition:** B

Disposition B best matches where the resource and the calendar sit.
The request is legitimate — the Anatomical Inventory and Person
Characteristics logical models explicitly model *wholeness*, and the
organ-transplant use case cannot be represented with the R6 elements
as drafted. But the R6 `BodyStructure` is actively being defended in
ballot (the *R6 BodyStructure Review* thread shows the WG pushing back
on late core changes and expecting comments to flow through normal
ballot channels), and Rob Hausam has already signalled in the
*Observation on BodyStructure Morphology* thread that the WG is open
to extension-first solutions for underspecified `includedStructure`
aspects. Publishing a standard extension on
`BodyStructure.includedStructure` gives the transplant community what
they need in R6, keeps the core resource stable, preserves the
hard-won 0..1 `Procedure.bodyStructure` story, and leaves a clean path
to promote the element into the core in a later release once the
value set and additional use cases (surgical resection, amputation,
partial grafts) are validated. Accept-as-requested (A) is the right
eventual destination but is premature without a value set; declining
outright (C) ignores a real, logical-model-aligned gap that the OO
WG has effectively already pre-approved via extension.
