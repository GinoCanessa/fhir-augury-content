## Summary

The reporter notes that pupillary distance (PD) — the distance between the
centers of the pupils — is a routine component of a vision prescription but
has no representation in the `VisionPrescription` resource. PD is needed to
correctly fabricate eyewear (lens optical-center alignment) and, in some
jurisdictions (e.g., Tennessee), is a regulatory requirement on every vision
prescription. The ticket proposes adding two new BackboneElements to the
resource: a top-level `binocularPD` (0..1) capturing combined `near`/`far`
PD in millimeters, and a `monocularPD` (0..1) on each `lensSpecification`
capturing per-eye `near`/`far` PD in millimeters. An invariant is also
proposed to prevent specifying both monocular and binocular PD for the same
quality (near vs. far) on the same prescription, to avoid contradictory or
redundant data. No related Jira tickets, Zulip threads, or GitHub items
were surfaced by the cross-reference index for this ticket.

## Details

**Description:**

Pupillary distance is usually part of a vision prescription, but there is
no place for it today in the VP resource.

The proposal is to:

1. Create a top level `monocularPD` backbone element (0..1), like so:
   ```
   binocularPD:
   {
       near: decimal  // Near binocular pupillary distance, in millimeters
       far:  decimal  // Far binocular pupillary distance, in millimeters
   }
   ```
2. Create a similar backbone element `monocularPD` (0..1) within
   `lensSpecification`, like so:
   ```
   monocularPD:
   {
       near: decimal  // Near monocular pupillary distance, in millimeters
       far:  decimal  // Far monocular pupillary distance, in millimeters
   }
   ```
3. Create an invariant that if `monocularPD` for both left & right eye are
   present, there should NOT be a `binocularPD` of the same quality
   ("near" vs. "far"), or require the math to add up (probably less
   preferred).

> Note: the description text labels the first element as `monocularPD` in
> the prose but uses `binocularPD` in the code block. Based on the structure
> (top-level, single value covering both eyes) and the invariant in item 3,
> the top-level element is clearly intended to be `binocularPD`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify the `VisionPrescription` resource to add pupillary-distance support
substantially as proposed:

1. Add a new top-level BackboneElement `VisionPrescription.binocularPD`
   (0..1) with two children:
   - `near` : `decimal` (0..1) — Near binocular pupillary distance, in
     millimeters.
   - `far`  : `decimal` (0..1) — Far binocular pupillary distance, in
     millimeters.
   Document units in the element definition (millimeters), and add a
   constraint that at least one of `near` / `far` SHALL be present when
   `binocularPD` is supplied.
2. Add a new BackboneElement `VisionPrescription.lensSpecification.monocularPD`
   (0..1) with the same `near` / `far` decimal children and the same unit
   guidance. Because `lensSpecification` already specifies `eye`
   (right/left) via the `vision-eye-codes` value set, a single
   `monocularPD` per `lensSpecification` is sufficient to express per-eye
   PD.
3. Add an invariant on `VisionPrescription` (e.g., `vp-pd-1`):
   "If both `lensSpecification` entries (right and left eye) include
   `monocularPD.near`, then `binocularPD.near` SHALL NOT be present;
   likewise for `.far`." Implement as FHIRPath rather than as a numerical
   sum check (the reporter flags the arithmetic option as less preferred).
4. Fix the wording in the description so that the top-level element is
   named `binocularPD` (per the code block, not the prose).

Authoring touches FHIR Core (the `VisionPrescription` source files,
structure definitions, examples, and the resource intro/notes pages).

#### Justification

PD is a foundational measurement for fabricating corrective lenses, and
its omission is a real interoperability gap — vendors today either lose
the value or carry it in a non-standard extension. The proposed shape
mirrors how clinicians document PD (per-eye when measured, combined when
not), and using `decimal` millimeters matches existing optical-power
elements on `lensSpecification` (`sphere`, `cylinder`, `add`, `prism`),
which keeps the resource internally consistent. The reporter's note about
Tennessee illustrates that this is not theoretical — jurisdictions are
already mandating PD on prescriptions, so a first-class element (versus
an extension) is the right level of standardization for a "Highest"
priority change request from O&O.

---

### Disposition B: Alternative Approach

#### Proposal

Address the same need with a more compact and reusable shape:

- Add a single complex datatype-style BackboneElement for PD that can be
  re-used at both levels, or — preferred — collapse the proposal into a
  per-eye representation only:
  - Add `VisionPrescription.lensSpecification.pdNear` (0..1, `decimal`,
    millimeters) and `VisionPrescription.lensSpecification.pdFar` (0..1,
    `decimal`, millimeters).
  - Do **not** add a top-level `binocularPD`. Where only a binocular
    measurement is available, implementers split it equally across the
    two `lensSpecification` entries (or carry the binocular value as an
    extension on the resource).
- Alternatively, if a top-level binocular value is considered essential,
  add it as a standard FHIR extension on `VisionPrescription` (defined in
  FHIR Core or in the Extensions Pack) rather than a core element, and
  keep only the per-eye fields in the base resource.

In either variant, units are pinned to millimeters in the element
definition, and no cross-element invariant is required because the
"contradiction" scenario the reporter is trying to prevent cannot arise.

#### Justification

The alternative removes the need for the `binocularPD` ↔ `monocularPD`
invariant entirely, which is the most fragile part of the original
proposal (invariants on optional cross-element combinations are easy for
implementers to miss and tend to surface as validation noise). It also
keeps PD next to the other per-eye optical measurements
(`sphere`/`cylinder`/`add`/`prism` already live on `lensSpecification`),
which is where ordering clinicians and lab systems expect to find it.
The trade-off is that purely-binocular measurements lose a first-class
home; the extension fallback addresses that without adding two parallel
ways to say the same thing in core. This option is also less of a
breaking-shape change for the FHIR Core resource graph, which lowers
the cost on downstream profilers and validators.

---

### Disposition C: Decline

#### Proposal

Decline a core resource change and instead direct implementers to use a
standard extension for pupillary distance on `VisionPrescription` —
either an existing extension (if one is already published) or a new
extension defined in the FHIR Extensions Pack rather than in core. Close
the ticket as "addressed via extension" with the extension URL recorded
in the resolution.

#### Justification

`VisionPrescription` is a Maturity Level 0/1 resource with a relatively
narrow user base; adding two new BackboneElements plus an invariant for a
single optical measurement is a non-trivial expansion of the core surface
area for a relatively niche use case. The FHIR extensibility model exists
precisely so that domain-specific data points like PD can be added
without growing every base resource, and a published extension is just as
conformant for jurisdictions like Tennessee as a core element would be.
Declining keeps core lean and avoids carrying a new invariant for a
contradiction that an extension-based design avoids by construction.

---

### Recommendation

**Recommended disposition:** B

The underlying need — first-class, standards-tracked PD on a vision
prescription — is real and well-justified by both clinical practice and
the reporter's jurisdictional example, so a pure decline (C) under-serves
implementers. Disposition A delivers what was asked but introduces a
top-level/per-lens duality plus a cross-element invariant whose only job
is to police that duality; that is a smell, and invariants of this shape
are routinely a source of late-cycle ballot comments. Disposition B keeps
the change inside `lensSpecification` next to the other per-eye optical
parameters, eliminates the need for the invariant, and still gives
binocular-only workflows a documented path (split across eyes, or use a
named extension). Workgroup should confirm with vision-care SMEs whether
binocular-only measurement is common enough to warrant a core
`binocularPD` despite the invariant cost; if it is, fall back to A with
the description fix applied.
