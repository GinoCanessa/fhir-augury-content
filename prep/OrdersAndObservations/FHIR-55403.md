## Summary

The reporter asks the Clinical Order Workflow (FHIR) specification to add
explicit guidance describing the relationship between `CarePlan` and the
order-related resources it overlaps with. The reporter notes that
`CarePlan` also "groups orders" — for example, `CarePlan` is the typical
output of `PlanDefinition $apply` in R4 — and that without spec-level
guidance, implementers are left to guess when a `CarePlan` should be
expected (or used) versus other grouping mechanisms such as
`RequestOrchestration` or a `ServiceRequest` with members. The proposed
edits would touch the **order initiation** and **order grouping**
sections of the workflow narrative. A follow-up comment from the
reporter broadens the ask: rather than a single sentence about
`CarePlan`, the spec should consider a dedicated page or section that
contextualises the boundaries and relationships between the related
resource families — `Request` (e.g., `ServiceRequest`,
`MedicationRequest`), `Task`, `Event`, `CarePlan`, and
`RequestOrchestration` — so an implementer can quickly decide which
resource fits a given scenario.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FhirAugury for this ticket key, so the request stands on
its own at the time of preparation.

### Disposition A: Accept As Requested

#### Proposal

Add a new narrative subsection — and matching cross-links — to the
**order initiation** and **order grouping** pages of the Clinical Order
Workflow (FHIR) specification that explains when implementers should
expect, produce, or consume a `CarePlan` versus a request-side grouping
mechanism. Concretely:

- On the *order initiation* page, add a paragraph noting that
  `PlanDefinition $apply` may yield a `CarePlan` (per R4 behaviour) in
  addition to or instead of `RequestOrchestration` /
  `ServiceRequest`-with-members, and explain how to recognise which is
  which.
- On the *order grouping* page, add a comparison block (table or
  bulleted contrast) covering `CarePlan` vs. `RequestOrchestration` vs.
  a parent `ServiceRequest` with `basedOn` / `partOf` children, with
  one sentence per option describing the intended scope (longitudinal
  plan-of-care vs. single-shot ordered workflow vs. parent/child
  request decomposition).
- Update the relevant page navigation / TOC entries and add anchor
  links from the resource pages (`CarePlan`, `RequestOrchestration`,
  `ServiceRequest`) back to the new narrative.

#### Justification

The ticket is explicitly a guidance request, not a request to change
the resources or their constraints. Workflow pages already exist for
both order initiation and grouping, so the change is editorial: it
extends what is already in scope for the spec and addresses a real
implementer pain point (the `PlanDefinition $apply` → `CarePlan`
output is well-attested from R4). Adding the guidance keeps the
boundary decisions in the workflow IG, which is where implementers
look first, and avoids spreading the explanation across each individual
resource page.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the reporter's broader follow-up comment instead of a localised
edit: introduce a new dedicated page in the Clinical Order Workflow
(FHIR) IG titled something like **"Choosing between Request, Task,
Event, CarePlan, and RequestOrchestration"** that gives a single
contextualised reference for the boundaries and relationships between
these resource families. The order-initiation and order-grouping pages
would then carry short forward-pointers to this new page rather than
inline duplicated guidance. Where appropriate, mirror the same content
(or a link to it) from the workflow module on the FHIR core spec so
core readers find it too.

#### Justification

The same ambiguity that motivates this ticket recurs across many other
order/plan/task discussions, so a single canonical page is more
maintainable than scattered paragraphs in two narrative sections. It
also matches the reporter's stated preference in their follow-up
comment, which suggests a *page or section* rather than spot edits.
The trade-off versus Disposition A is scope: a new page is more work
to draft and review, and risks duplicating material that lives in the
workflow module of the core FHIR spec — so the proposal must be
careful to reference, not re-state, the core workflow definitions.
This is preferable when the workgroup expects similar questions to
keep arriving (e.g., for `Task` vs. `ServiceRequest`).

---

### Disposition C: Decline

#### Proposal

Decline as **already addressed by existing mechanisms**. The base FHIR
specification's *workflow module* and the individual resource pages
(`CarePlan`, `ServiceRequest`, `Task`, `RequestOrchestration`,
`PlanDefinition`) already discuss the intended scope and relationships
between these resources, including the `PlanDefinition $apply`
behaviour. Direct the reporter to those existing pages and close the
ticket with a "Not Persuasive" or "Not Persuasive with Mod" disposition
if no concrete textual gap can be pointed at.

#### Justification

Defensible only if the workgroup judges that the existing core-spec
workflow narrative is sufficient and that re-stating it inside the
Clinical Order Workflow IG would create maintenance burden and
divergence risk. The risk is real: every time the workflow module
changes in core, the IG copy would need to be re-synced. However, the
ticket explicitly identifies two workflow IG sections that, in the
reporter's reading, lack the guidance — so declining requires the
workgroup to assert that those sections do already cover the
boundaries (or that they should not). That makes Decline the weakest
of the three options unless someone in the workgroup can point at
specific existing language that already answers the question.

---

### Recommendation

**Recommended disposition:** B

The reporter's own follow-up comment broadens the ask from a couple of
inline edits to "a page or section explaining the contextualized
boundaries and relationships between resources" — i.e., Disposition B —
and the resource list they enumerate (`Request`, `Task`, `Event`,
`CarePlan`, `RequestOrchestration`) is exactly the long-running set of
overlap questions that the Orders & Observations and FHIR-I workgroups
keep encountering. A dedicated page in the Clinical Order Workflow
(FHIR) IG, with short forward-pointers from the *order initiation* and
*order grouping* sections, gives implementers one canonical place to
land and avoids the maintenance drift that comes with sprinkling the
same guidance into multiple narrative pages. Disposition A is a
reasonable fallback if the workgroup wants to keep the change small for
this ballot cycle; Disposition C should only be selected if a reviewer
can point to existing core-spec text that already answers the
question, which the ticket implies they cannot. Either A or B will
require the workgroup to confirm the authoring repo for the Clinical
Order Workflow (FHIR) IG before the change is drafted, since FhirAugury
did not surface a repo cross-reference for this specification at prep
time.
