## Summary

The reporter (Ralf Herzog) requests that FHIR Core provide a first-class
mechanism for documenting the *reason* a resource is in a particular
`status` — most notably on `Observation` and `DiagnosticReport`, and
potentially on other event-style resources. The motivating use cases are
both around `status = "entered-in-error"`, where today there is no
structured place on either resource to record *why* it was so marked.
The only prior art surfaced on the ticket is the existing
`event-statusReason` extension in the HL7 FHIR Extensions IG
(https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-event-statusReason.html),
which the lone commenter (jdlnolen) raises as an alternative and asks
whether the reporter actually needs this in R6 versus continuing to use
the extension. There are no cross-referenced Jira tickets, Zulip
threads, or GitHub items on this ticket; the only structured
cross-references are mentions of the `Observation.status` and
`DiagnosticReport.status` elements themselves.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new core element `statusReason` of type `CodeableConcept` (0..1)
to both `Observation` and `DiagnosticReport`, sibling to `status`, with
a preferred binding to a value set seeded from common reasons for
`entered-in-error` / `cancelled` / `unknown` (e.g., wrong patient,
duplicate entry, specimen rejected, instrument malfunction, corrected
result superseded). Apply the same change consistently across the
remaining "event pattern" resources where `status` exists but no
`statusReason` does (e.g., `ImagingStudy`, `Specimen`, etc.) so the
Event pattern remains coherent — the OO work group should sweep the
event-pattern resources it owns and propose the analogous change on
others via cross-WG coordination. Retire / mark-as-superseded the
`event-statusReason` extension once the core element is in.

#### Justification

Reason-for-status is a clinically and operationally important piece of
metadata that is most often needed exactly when the resource is in an
exceptional state (`entered-in-error`, `cancelled`, `unknown`). Lifting
it out of the extension into a core element makes it discoverable in
the resource definition, queryable via a standard search parameter, and
consistent with how other event-pattern resources already model status
companion data. The reporter's example (`entered-in-error` on
`Observation` / `DiagnosticReport`) is one of the most common real-world
data-quality patterns; expecting every implementer to know about and
adopt the `event-statusReason` extension is a significant adoption
tax. Promotion from extension to core for elements that are widely
needed is a standard FHIR maturation path.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add `statusReason` to the base resources. Instead:

1. Confirm and document `event-statusReason` as the canonical mechanism
   for capturing the reason of a status change on event-pattern
   resources, and add explicit guidance to `Observation.status` and
   `DiagnosticReport.status` (in their element definitions / "Notes"
   sections) pointing implementers to that extension.
2. Promote `event-statusReason` to **normative** alongside the resource
   in R6 if maturity warrants, and add it to the canonical
   modifierExtension/extension recommendations for Observation and
   DiagnosticReport.
3. Consider adding it to the relevant US Core / IPS / IPA profiles that
   need to mandate it, rather than burdening base FHIR.

This keeps the base resource surface stable and lets the existing,
already-deployed extension carry the load — exactly the path
jdlnolen's comment points to.

#### Justification

The `event-statusReason` extension already exists and already does what
the ticket asks for; the gap is *discoverability and guidance*, not
capability. Promoting an extension to a core element is a breaking
change in spirit (existing extension-based data has to be migrated /
dual-published) and adds element bloat to two of the most heavily used
clinical resources in FHIR. Better discoverability via element-level
guidance + a profile-level requirement gives implementers the same
practical outcome at a fraction of the disruption, and preserves the
"core stays small, extensions carry the variability" design principle
the FHIR community has historically defended.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by an existing mechanism**.
The `event-statusReason` extension in the HL7 FHIR Extensions IG
(https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-event-statusReason.html)
is purpose-built for exactly this use case and is applicable to
Observation, DiagnosticReport, and the rest of the event pattern. No
spec change is needed; close the ticket as "Persuasive with Mod" /
"Not Persuasive" with a pointer to the extension.

#### Justification

FHIR's design explicitly uses extensions for needs that are real but
not universal, and the event pattern already provides a canonical
extension here. The reporter has not (in the ticket text or the single
comment exchange) demonstrated that the extension is insufficient —
only that they would prefer something native. Without a stronger use
case showing why an extension cannot satisfy the requirement (e.g.,
broad adoption data, profiles forced into awkward workarounds, or a
modeling problem the extension cannot represent), a base-spec change
to two heavily used resources is not justified.

---

### Recommendation

**Recommended disposition:** **B**

Disposition B threads the needle the reporter and the commenter are
both pulling on. The reporter is right that "use the extension" is not
a great answer when the need is this common — particularly around
`entered-in-error`, which is a data-integrity primitive that every
implementer eventually hits. But jdlnolen is also right that the
capability already exists and that adding a new core element to two of
FHIR's most-implemented clinical resources mid-cycle is a heavy lift
with real downstream cost (re-tooling, profile updates, search
parameter publication, IG ripple effects, and de-facto deprecation of
existing extension-based data). Promoting the existing
`event-statusReason` extension, calling it out explicitly in the
`status` element definitions on Observation and DiagnosticReport, and
letting the realm/domain IGs (US Core, IPS, IPA, lab IGs) require it
where it matters delivers the practical interoperability the reporter
wants without paying the base-resource churn cost. If, after a cycle
of explicit promotion, implementers still report that the extension is
inadequate, the workgroup can revisit Disposition A on stronger
evidence than what is currently on the ticket.
