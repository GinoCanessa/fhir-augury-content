## Summary

FHIR-55000 asks that the terminology bindings on `BiologicallyDerivedProduct.productCode`
and `BiologicallyDerivedProduct.property.type` be aligned with the ISBT 128 external
code systems (product description codes and ISBT 128 data elements respectively) as
example or extensible bindings. The reporter (Ron G. Parker, comment 26 of the original
discussion) notes that this realignment was already voted on and approved for
terminology but never landed in the build. The companion ticket
[FHIR-53477](https://jira.hl7.org/browse/FHIR-53477) ("Redirect to External
CodeSystems for Product Description and Property Type Codes") is the technical
correction tracking the actual URL/code repointing, and
[FHIR-49979](https://jira.hl7.org/browse/FHIR-49979) ("BDP and BDPDispense code
systems should not be marked as complete") is the upstream work that flagged the
existing FHIR-hosted code systems as inappropriate (they are not complete, and the
codes are ICCBBA/ISBT IP, not HL7's). Together these tickets indicate the
intended path is: stop publishing the BDP product/property code systems in FHIR,
move the canonical content to THO as fragments of the external ISBT 128 systems,
and rebind the FHIR value sets there. The resolution captured on this ticket
(Persuasive with Modification) reflects exactly that: move the two CodeSystems to
THO, keep the FHIR ValueSets but repoint them to the THO equivalents, and register
the existing FHIR codes against the existing ISBT 128 NamingSystems as fragments.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution exactly as already captured on the ticket (Persuasive with
Modification):

1. Remove the two FHIR-hosted CodeSystems from the FHIR Core build:
   - `http://hl7.org/fhir/biologicallyderived-productcodes`
   - `http://hl7.org/fhir/biologicallyderived-product-property-type-codes`
   …and migrate their content to THO as **fragments** of the canonical ISBT 128
   external code systems.
2. Keep the two corresponding FHIR ValueSets (`biologicallyderived-productcodes`
   and `biologicallyderived-product-property-type-codes`) in FHIR Core, but
   change their compose include to reference the THO/ISBT 128 canonicals
   (`https://www.isbt128.org/uri/productdescriptioncode` for `productCode` and
   `https://www.isbt128.org/uri` for property types) instead of the
   FHIR-hosted CodeSystems.
3. Register the existing/migrated FHIR code values against the four NamingSystem
   / ECS_MR entries already published in THO
   (`NamingSystem-ISBT128ProductDescriptionCodes`,
   `ECS_MR-ISBT128ProductDescriptionCodes`,
   `NamingSystem-ISBT128PropertyTypes`,
   `ECS_MR-ISBT128PropertyTypes`), marking the registered subset as fragments.
4. Leave the binding strengths on `BiologicallyDerivedProduct.productCode` and
   `BiologicallyDerivedProduct.property.type` aligned with the existing
   ValueSets (i.e., keep `example`/`extensible` as already configured), since
   the request is about the *target* of the binding, not its strength.
5. Capitalise the leading code characters per FHIR-53477 to match the actual
   external ISBT 128 representation.

#### Justification

This is the path that the workgroup has already endorsed in two adjacent
tickets: FHIR-49979 established that the FHIR-hosted CodeSystems should not be
in FHIR (codes are ICCBBA IP and the systems are not complete), and FHIR-53477
defined the concrete URL/code mechanics. FHIR-53477's HCP comment from Robin
White (ICCBBA) explicitly endorses the "fragment of an external code system
hosted in THO" pattern, which is the THO-recommended way to reference licensed
external terminologies. THO has already published the necessary
NamingSystem/ECS_MR entries, so the dependency is unblocked. Doing exactly what
the resolution says keeps FHIR-55000, FHIR-53477, FHIR-49979 and FHIR-55252
mutually consistent and avoids re-litigating a decision that has multiple
prior approvals.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the same end-state but stage it differently to avoid coupling the FHIR
Core publication to a coordinated THO release: in this cycle, change the
ValueSets to compose against the THO-hosted external code system canonicals
**only** (deleting the FHIR `biologicallyderived-productcodes` and
`biologicallyderived-product-property-type-codes` CodeSystem resources from
FHIR Core), and *do not* attempt to re-publish the historical code subset as a
fragment from FHIR Core. The fragment registration (step 3 of the resolution)
becomes a THO-side maintenance task tracked outside FHIR Core — when the THO
fragments exist, no FHIR-Core change is required because the ValueSet already
points at the external canonical. As an additional protection, change the
binding strength on `BiologicallyDerivedProduct.productCode` and
`BiologicallyDerivedProduct.property.type` to **example** (rather than
extensible) to acknowledge that the listed codes are illustrative until the
THO fragments mature.

#### Justification

The published resolution couples three deliverables (CodeSystem move, ValueSet
rebinding, fragment registration) across two specifications (FHIR Core and
THO). If THO's publication cadence slips, FHIR Core publication can stall on a
dependency it cannot itself fix. This alternative removes that coupling: FHIR
Core ships only the ValueSet rebinding (which it owns), and the
fragment-registration work moves entirely to THO where it belongs. Dropping the
binding strength to `example` reflects the reality that without the THO
fragment, only a small illustrative subset of ISBT 128 codes is meaningfully
reachable from FHIR's perspective; this is also consistent with FHIR-49979's
observation that the codes were never a complete CodeSystem. The trade-off vs.
Disposition A is loss of a guaranteed authoritative listing of the legacy
historical codes from FHIR Core — implementers who relied on
`http://hl7.org/fhir/biologicallyderived-productcodes` would need to update to
the ISBT 128 canonical without a transitional fragment of the same shape.

---

### Disposition C: Decline

#### Proposal

Decline the request as duplicative and already overtaken by adjacent work:
FHIR-53477 (Persuasive, *Ready_to_Apply*) already specifies the URL/code
mechanics, and FHIR-49979 (Persuasive with Modification) already directs
removal of the FHIR-hosted CodeSystems and ValueSet rebinding. Resolve
FHIR-55000 as **Not Persuasive — duplicate**, with a pointer to FHIR-53477 and
FHIR-49979 as the tickets carrying the actual change.

#### Justification

The technical change FHIR-55000 asks for is, mechanically, a subset of what
FHIR-53477 already covers. Closing FHIR-55000 as a duplicate avoids tracking the
same change in three places, reduces the risk of partially-applied edits, and
keeps the change ledger easier to audit. The ticket's value as a *forcing
function* (driving the THO migration) is already preserved by FHIR-49979 and
FHIR-55252, which are higher-priority and more prescriptive. This is a
defensible disposition only if the workgroup is confident FHIR-53477 will
actually be applied in the next FHIR Core publication cycle.

---

### Recommendation

**Recommended disposition:** A

The resolution already on FHIR-55000 (Persuasive with Modification) is precisely
what the workgroup has, in three independent tickets, said it wants:
FHIR-49979 said "stop hosting these CodeSystems in FHIR"; FHIR-53477 said "here
are the exact URLs and code edits"; FHIR-55252 said "this is the THO-candidate
list to migrate"; the ICCBBA SME on FHIR-53477 said "use fragments of the
external system in THO". THO has already published the NamingSystem and ECS_MR
entries the resolution depends on, so the externally-blocking condition that
held FHIR-49979 open has cleared. Disposition B is technically defensible if
the workgroup is worried about cross-spec coordination, but it leaves
implementers without an authoritative FHIR-side fragment of the historical
codes and quietly weakens the binding strength, which is a separate decision
the workgroup has not been asked to make. Disposition C is administratively
clean but loses the explicit binding-target language that FHIR-55000 carries
about `productCode` and `property.type` — language FHIR-53477 does not state in
those exact terms. Accepting as requested is the lowest-risk, lowest-surprise
outcome and lets FHIR-53477 carry the apply step.
