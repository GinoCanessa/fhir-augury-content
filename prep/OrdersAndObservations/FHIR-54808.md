## Summary

The reporter notes that FHIR Core does not currently provide clear,
prescriptive guidance on how Observations (in particular laboratory
panels) should be grouped inside a DiagnosticReport. Several mutually
incompatible patterns are in active use today — using one
DiagnosticReport per panel, using `Observation.organizer` together with
`Observation.hasMember`, or using `Observation.hasMember` alone — and
two further patterns would require Core changes (allowing `Group` or
`List` as a target of `DiagnosticReport.result`). The ticket asks OO to
provide explicit guidance on the [Diagnostics
Module](https://build.fhir.org/diagnostics-module.html) page, and on
the [DiagnosticReport](https://build.fhir.org/diagnosticreport.html)
and [Observation](https://build.fhir.org/observation) resource pages,
so that implementers converge on a single recommended approach. The
ticket points to the OO ["Grouping Option
Analysis"](https://confluence.hl7.org/spaces/OO/pages/403871266/Grouping+Option+Analysis)
Confluence page and the UV Lab Report / Observation Grouping calls for
the underlying analysis. No other Jira tickets were surfaced as
cross-references in FHIR Augury at the time of this prep.

## Details

**Description:**

It is currently not clear how Observations (e.g. Panels) should be grouped
(see page: <https://confluence.hl7.org/spaces/OO/pages/403871266/Grouping+Option+Analysis>).

Since there exist multiple options without changing FHIR Core like

- 1 DiagnosticReport = 1 panel
- `Observation.organizer` indicator and `.hasMember` = 1 panel
- `Observation.hasMember` = 1 panel (without `Observation.organizer`)

and also some options with changes to FHIR Core like

- `Group` referenced in `DiagnosticReport.result` = 1 panel (would need
  `.result` to be able to include `Group`)
- `List` referenced in `DiagnosticReport.result` = 1 panel (would need
  `.result` to be able to include `List`)

There is a need to give more clear guidance on the

- [DiagnosticModule](https://build.fhir.org/diagnostics-module.html) page
- [DiagnosticReport](https://build.fhir.org/diagnosticreport.html) Resource and
- [Observation](https://build.fhir.org/observation) Resource

For details on this discussion take a look at the [Grouping Option
Analysis Confluence Page](https://confluence.hl7.org/spaces/OO/pages/403871266/Grouping+Option+Analysis)
and the discussion of the UV Lab Report Call and the Observation
Grouping Call of OO.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative-style guidance to FHIR Core that picks one preferred
panel-grouping pattern and documents it in three places:

1. The [Diagnostics
   Module](https://build.fhir.org/diagnostics-module.html) page — add a
   "Grouping Observations into Panels" subsection that names the
   recommended pattern, shows a worked example, and explicitly
   deprecates (or strongly discourages) the alternatives listed in the
   ticket.
2. The DiagnosticReport resource page — under "Notes" / "Scope and
   Usage", cross-link to the Diagnostics Module guidance and clarify how
   `DiagnosticReport.result` is expected to point at panel members
   (e.g., reference the panel "header" Observation that uses
   `Observation.organizer = true` and whose members are reached via
   `Observation.hasMember`).
3. The Observation resource page — under the descriptions of
   `Observation.organizer` and `Observation.hasMember`, make the panel
   semantics explicit (when to set `organizer`, what `hasMember` is
   allowed to point at, and how to avoid double-listing members from
   both `DiagnosticReport.result` and `Observation.hasMember`).

The recommended pattern that best aligns with the existing OO direction
and the `organizer-grouping` label on the ticket is:
**`Observation.organizer = true` on a "header" Observation whose
`Observation.hasMember` points at the member Observations, with
`DiagnosticReport.result` referencing the header Observation only.**

#### Justification

This option is the only one in the ticket's first list that uses
elements specifically designed for grouping (`Observation.organizer`
was added to FHIR Core for exactly this purpose), and it does so without
requiring any Core resource changes. It keeps `DiagnosticReport.result`
small and meaningful (one reference per panel rather than one per
analyte), it preserves the round-tripability of panels as standalone
Observation graphs, and it matches how the UV Lab Report IG is
trending. The Zulip thread shows the WG actively trying to converge in
time for R6 ballot reconciliation, which supports a binding guidance
fix in Core rather than leaving it implicit.

---

### Disposition B: Alternative Approach

#### Proposal

Treat this as a documentation-only ticket (no normative change), and
publish the OO "Grouping Option Analysis" content as a new informative
section under the Diagnostics Module page. Enumerate all of the listed
options (1 DR = 1 panel; `organizer` + `hasMember`; `hasMember` only;
plus the Core-change variants using `Group` / `List` in
`DiagnosticReport.result`) with their trade-offs, and recommend
`organizer + hasMember` as the **preferred** pattern without
deprecating the others. Defer any change to `DiagnosticReport.result`'s
allowed targets (no `Group`, no `List`) to a separate ticket gated on
demonstrated implementer demand.

#### Justification

The ticket itself does not demand a Core schema change — it asks for
*clearer guidance*. A pure-documentation pass is lower-risk for R6
ballot reconciliation, avoids breaking implementations that have
already chosen one of the non-preferred patterns, and still gives
implementers an authoritative "OO recommends X" they can point to. It
also keeps the door open for the `Group`/`List` Core change to be
re-evaluated on its own evidence rather than being bundled in. The
trade-off versus Disposition A is weaker convergence — implementations
using "1 DR = 1 panel" or "hasMember only" remain valid — but that may
be acceptable given the late stage of R6.

---

### Disposition C: Decline

#### Proposal

Decline as "already addressed by existing mechanisms". `Observation.organizer`
together with `Observation.hasMember` is already documented in the
Observation resource as the panel-grouping mechanism, and
`DiagnosticReport.result` already supports panel headers by reference.
Close the ticket with a pointer to the existing element definitions and
to the OO "Grouping Option Analysis" Confluence page as the canonical
narrative.

#### Justification

This disposition rests on the position that the variability the
reporter is observing reflects implementer choice, not a gap in the
specification — the building blocks already exist and are documented at
the element level. It also avoids the cost of writing new
cross-resource narrative inside a busy R6 ballot cycle. The downside,
which the Zulip thread underlines, is that implementers are visibly not
converging on their own; declining is essentially a bet that the IGs
(notably the UV Lab Report) will carry the weight of guidance instead
of Core.

---

### Recommendation

**Recommended disposition:** A

The reporter, the OO LAB / UV Lab Report calls, and the Zulip thread
are all explicitly trying to land on a single pattern in time for R6
ballot reconciliation, and the `organizer-grouping` label on the ticket
signals that the WG has already nominated `Observation.organizer` as
the lead. Disposition A turns that direction into normative-strength
guidance in the three pages the ticket calls out, which is the
intervention most likely to actually stop the divergence. Disposition B
is the safe fallback if the WG decides during reconciliation that
strong deprecation language is too aggressive at this point in the
cycle, and Disposition C should be avoided because the discussion
record shows the existing element-level docs are not, in practice,
sufficient to drive convergence. The Core changes implied by the
`Group` / `List` variants in `DiagnosticReport.result` should be left
out of this ticket and, if pursued at all, raised as a separate
follow-up so that R6 ballot reconciliation is not blocked on a new
target-type negotiation.
