## Summary

[FHIR-51528](https://jira.hl7.org/browse/FHIR-51528) asks Orders & Observations
to settle, in FHIR Core, the *appropriate* way to represent the **body position**
of the subject at the time an observation was made (e.g., standing, sitting,
supine). It is the explicit follow-up to
[FHIR-31355](https://jira.hl7.org/browse/FHIR-31355) — "Body position should be
a component, not an extension" — which was applied as *Persuasive with
Modification*: the existing `observation-bodyPosition` extension in the
extensions pack was capped at FHIR R5 (via `version-specific-use` /
`endFhirVersion = 5.0.0`, applied in `HL7/fhir-extensions` PR
[#185](https://github.com/HL7/fhir-extensions/pull/185), commit `e4bb993`), and
a new ticket — this one — was opened against Core to decide what the
post-R5 representation should be. The leading suggestion, voiced by Lloyd
McKenzie on [FHIR-31355](https://jira.hl7.org/browse/FHIR-31355) and echoed in
multiple Zulip threads, is that body position is a *qualifier of the code*
("standing/sitting/supine BP"), and qualifiers of the code belong on
`Observation.component` rather than in an extension. Open questions raised by
reviewers include: how the guidance interacts with `Observation.bodySite` /
`Observation.bodyStructure`; whether a small set of "well-known" qualifier
components (body position, cuff size, susceptibility organism/drug, …) should
be enumerated in Core's Observation guidance; and how to reconcile this with
existing national profiles (notably the HL7 Norway vital-signs profiles, which
currently use extensions) and with US/SHIELD lab-reporting patterns where
implementers have pushed back against using `component` for some qualifiers.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In FHIR Core (HL7/fhir), update the **Observation resource notes / modeling
guidance** to state that *body position* (standing / sitting / supine /
recumbent / lateral, etc.) at the time of the observation is to be conveyed
as an `Observation.component` whose `component.code` is the body-position
LOINC/SNOMED concept and whose `component.value[x]` carries the position
value (typically `valueCodeableConcept`). Add a non-normative example to the
Observation page (and to the vital-signs profile narrative) showing a blood
pressure with a `body position = sitting` component. Do not introduce a new
top-level Observation element; do not bring the deprecated
`observation-bodyPosition` extension forward. Add a back-pointer from the
deprecated extension page to the new Core guidance.

#### Justification

This is the resolution direction already endorsed on
[FHIR-31355](https://jira.hl7.org/browse/FHIR-31355) ("Persuasive with
Modification") and consistently advocated by the resource's primary editor
across the *Blood Type* (2026), *Pathology Lab Reports* (2021), and
*creating spreadsheet in IG* (2018) Zulip threads. Body position qualifies
*what was observed* (the meaning of the BP reading is different at supine vs.
standing), which is the textbook role of `Observation.component`; the
parallel with cuff size — which the WG already treats as a component — is
exact. Implementers are already doing this in the wild (see Igor Bossenko's
FSH profile in the *shorthand* stream), and capping the extension at R5
without giving Core an answer leaves a gap that profile authors will fill
inconsistently.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than issuing body-position-specific guidance, fold this ticket into
the WG's broader "Observation best-practices" effort and publish a single
**generalized "qualifier components" guidance section** on the Observation
resource page. The section would (a) state the rule — code-qualifying
information goes on `Observation.component`, observation-action metadata
(witness, protocol, etc.) goes in extensions — (b) enumerate a non-exhaustive
list of well-known qualifier components (body position, cuff size, fasting
status, susceptibility organism/drug, device, etc.) with example LOINC/SNOMED
codes, and (c) explicitly contrast the component pattern with
`Observation.bodySite` / `Observation.bodyStructure` so implementers know
when to use which. Body position would be the worked example.

#### Justification

Rob Hausam's note in the *Blood Type* thread is that the Observation
modeling discussion is broader than this single qualifier, and several
commenters on FHIR-31355 (Andrea Pitkus, Hans Buitendijk) asked for general
guidance on when a qualifier is a component vs. an extension vs. a
bodySite/bodyStructure. Solving only "body position" leaves the same
question to be re-litigated for the next qualifier. A generalized rule with
body position as the canonical example is cheaper to maintain, more useful
to implementers (notably the HL7 Norway vital-signs profile authors and
SHIELD-aligned lab implementers), and reduces the risk that profile authors
make divergent choices for adjacent qualifiers. The *gotcha* called out by
Andrea Pitkus on FHIR-31355 — that some US/CLSI lab-reporting patterns deliberately *don't* use components — is best addressed by *guidance that names
the boundary*, which Disposition A alone does not do.

---

### Disposition C: Decline

#### Proposal

Decline to add body-position-specific guidance to FHIR Core. Rationale: the
existing Core elements (`Observation.component` with a free choice of code,
`Observation.bodySite`, `Observation.bodyStructure`) already let implementers
represent body position; the deprecated `observation-bodyPosition` extension
has been retired at R5 (FHIR-31355 / fhir-extensions PR #185); profiles
(US Core, IPS, vital-signs profiles, HL7 Norway, openEHR-aligned IGs) are
the appropriate place to make a binding choice for a given realm or use
case. Close FHIR-51528 as *Not Persuasive* with a pointer to the existing
elements and to FHIR-31355's resolution.

#### Justification

Defensible on the grounds that Core already provides the mechanism, and
that prescriptive guidance risks conflicting with realm-specific patterns
that the WG has already heard pushback on (Andrea Pitkus's CLSI / SHIELD
comments on FHIR-31355). Declining keeps Core minimal and pushes the
modeling decision to profiles where the use-case context lives.

---

### Recommendation

**Recommended disposition:** B

The ticket exists *because* FHIR-31355 deliberately deferred the Core
question, so Disposition C — "Core already has the mechanism" — would
re-open exactly the gap the parent ticket was closed to surface; declining
also leaves the deprecated extension's replacement implicit, which is the
worst outcome for implementers reading the spec at R6. Disposition A is
defensible and is the literal request, but the most recent WG conversation
(*Blood Type*, Jan 2026) and the recurring Norway-vs-US-lab tension on
FHIR-31355 both indicate that the *real* question implementers keep asking is
the broader "when is a qualifier a component vs. an extension vs.
bodySite/bodyStructure?" — a question that body position, cuff size, and
susceptibility organism all instantiate. Disposition B answers FHIR-51528's
literal ask (body position becomes a component, with a worked example) while
also producing the re-usable rule the WG would otherwise have to write three
more times. Recommend assigning to the same SMEs already engaged on
FHIR-31355 (Lloyd McKenzie, Hans Buitendijk, Eric Haas) with Andrea Pitkus
consulted on the lab boundary so the SHIELD/CLSI carve-outs are stated, not
discovered later.
