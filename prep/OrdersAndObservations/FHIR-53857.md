## Summary

The Clinical Order Workflow (COW) IG profiles `Task` against R4, where
`Task.statusReason` is a `0..1 CodeableConcept`. The reporter, Oliver Egger,
asks whether the COW IG should add a backport extension that effectively
turns `Task.statusReason` into `0..* CodeableReference`, so a Fulfiller can
both code the reason the status was reached and point at the artifacts that
explain it (other `Task`s, `Communication`s, `CommunicationRequest`s,
etc.). The alternative he raises is to leave `Task.statusReason` alone in
the COW profile and instead require consumers to harvest that information
from `Task.output`. In a follow-up comment he clarifies the COW deployment
model: the `Task` lives on the Fulfiller, so it is the **Placer** (not the
Fulfiller) that would have to read those outputs back. No related Jira
tickets are linked or cross-referenced; the question is unique to the COW
IG today, though it sits squarely in the broader R5/R6 trend of widening
`statusReason`-style elements to `CodeableReference`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the Clinical Order Workflow (FHIR) IG, profile the COW
`coordination-task` (`StructureDefinition-coordination-task`) so that
`Task.statusReason` is widened to `0..* CodeableReference`. Concretely:

- Define a backport extension (e.g., `cow-statusReason` /
  `Task.statusReason.reference`) that carries a list of
  `CodeableReference` entries, each capable of pointing at `Task`,
  `Communication`, `CommunicationRequest` (and any other artifacts COW
  expects fulfilment to depend on).
- In the `coordination-task` profile, slice or constrain
  `Task.statusReason` (`CodeableConcept`, `0..1`) and add the new
  extension at `0..*`, with mapping notes that say "in R5+ this is the
  native `Task.statusReason` `CodeableReference`."
- Add narrative to the COW IG explaining placer/fulfiller responsibilities:
  the Fulfiller populates `statusReason` (concept + references) when it
  changes status; the Placer reads them to learn what the Fulfiller is
  blocked on.

#### Justification

This is the design Lloyd McKenzie endorsed on the
`implementers > negative ack on prescriptions` thread for exactly the
"Fulfiller signals what it needs" use case. It also matches what current
FHIR already does (`Task.statusReason` is `CodeableReference` in the R6
build, per the FMG inventory thread), so the COW IG would simply be
forward-aligning its R4 profile via a backport extension — the standard
R4-IG idiom for picking up an R5+ shape. Consumers gain a single,
predictable place to look for "why did the status change and what does the
Fulfiller need," instead of having to inspect `Task.output` heuristically.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change `Task.statusReason` in the COW profile. Instead, formalise
the existing `Task.output` channel for "what the Fulfiller needs":

- Define COW-specific output codes (or reuse existing ones) for
  `Task.output.type` such as `needs-task`, `needs-communication`,
  `needs-communication-request`, etc.
- Constrain `Task.output.value[x]` to `Reference(Task | Communication |
  CommunicationRequest | …)` for those output codes.
- Document in the COW IG that when `Task.status` enters a state that
  requires placer action, the Fulfiller SHALL populate one or more
  `Task.output` entries with the appropriate code and references, and the
  Placer SHALL inspect `Task.output` (not `Task.statusReason`) to discover
  them.

`Task.statusReason` would remain the unchanged R4 `CodeableConcept`
carrying only the human/coded reason.

#### Justification

This avoids introducing a backport extension and stays purely within R4
mechanics. `Task.output` is already defined for "Information produced as
part of task" and is genuinely `0..*`, so it can carry the heterogeneous
list the ticket envisions without any R5 borrowing. It also keeps a clean
separation between *why* the status was set (a code) and *what artefacts
the placer must look at next* (structured outputs) — the same separation
COW already uses for inputs. The down-side, which Disposition A's
extension avoids, is that placer implementations must learn an output-code
vocabulary instead of dereferencing a single typed element.

---

### Disposition C: Decline

#### Proposal

Mark FHIR-53857 as "Persuasive — Considered for Future Use" / "Not
Persuasive" against the COW IG: do not introduce a `CodeableReference`
backport for `Task.statusReason`, and do not add COW-specific output
slicing either. Refer the requester to the existing R5/R6 definition of
`Task.statusReason`-as-`CodeableReference` and to the standard
inter-version extension pattern (`http://hl7.org/fhir/5.0/StructureDefinition/extension-Task.statusReason`)
that any R4 implementer can already use without IG-level intervention.

#### Justification

The community already has a sanctioned R4-friendly mechanism for this
exact need — inter-version extensions — and Lloyd McKenzie pointed at it
in the `negative ack on prescriptions` thread ("you'd have to use
interversion extensions in R4"). Standing up a COW-bespoke extension or
re-purposing `Task.output` risks divergence from how every other R4 IG
handles backports of R5+ `CodeableReference` widenings, and the ticket
itself is currently `Triaged · Medium` with a single comment and no
broader implementer demand inside Orders & Observations.

---

### Recommendation

**Recommended disposition:** A

The change asked for is small, well-scoped, and aligns the COW R4 profile
with what current FHIR already does for `Task.statusReason`. The relevant
implementer Zulip thread already endorses `CodeableReference` as the right
shape for "the Fulfiller signals what it needs / what blocked status", and
the FMG inventory thread confirms `Task.statusReason` is in fact
`CodeableReference` in the current build — so adopting a backport
extension in the COW IG is the lowest-friction path that keeps R4
implementers and forward-migrating R5/R6 implementers on the same data
shape. Disposition B is workable but quietly invents a COW-specific use of
`Task.output` that other Task-using IGs would not share, and Disposition C
under-serves the use case by leaving each implementer to wire up the
inter-version extension on their own. The workgroup's main open question
should be naming/canonical-URL for the backport extension and whether to
align it byte-for-byte with the existing R5→R4 inter-version extension
rather than mint a COW-only one.
