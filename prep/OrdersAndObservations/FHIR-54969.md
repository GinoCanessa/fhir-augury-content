## Summary

Marti Velezis (Orders & Observations) is asking the workgroup to perform a
comprehensive, side-by-side review of every data element in DeviceDefinition
against the corresponding elements in Device, identify which elements legitimately
overlap (and should be harmonized in name, datatype, cardinality, binding, and
definitional text), and confirm that the elements that are unique to
DeviceDefinition form a complete, coherent definitional model (i.e., no gaps in
characteristics that an implementer would expect a "device kind/model" resource to
carry). The ticket is the latest in a long-running OO programme to clean up the
Device / DeviceDefinition split; it is closely paired with
[FHIR-54471](https://jira.hl7.org/browse/FHIR-54471) ("Ensure DeviceDefinition is
complete and accurate"), which scopes the broader text/structure/binding/example
review of DeviceDefinition. Earlier tickets in the same arc include
[FHIR-38902](https://jira.hl7.org/browse/FHIR-38902) (Device/DeviceDefinition
consistency, persuasive with mod, R5),
[FHIR-38711](https://jira.hl7.org/browse/FHIR-38711) (clean up DeviceDefinition
Scope and Boundary so device categories live on Device and DeviceDefinition only
covers definitional content),
[FHIR-47904](https://jira.hl7.org/browse/FHIR-47904) (rework DeviceDefinition
Scope text that currently leads with Device),
[FHIR-47419](https://jira.hl7.org/browse/FHIR-47419) (harmonize
Device.udiCarrier vs DeviceDefinition.udiDeviceIdentifier and add
deviceIdentifierSystem to DeviceDefinition),
[FHIR-39634](https://jira.hl7.org/browse/FHIR-39634) (intendedUse /
instructionsForUse — added DeviceDefinition.guideline rather than mirroring on
Device), and [FHIR-39388](https://jira.hl7.org/browse/FHIR-39388)
(DeviceDefinition.hasPart semantics, ultimately resolved via an extension in the
Extensions Pack). FHIR-54969 explicitly asks for the missing systematic pass:
enumerate the overlapping elements, decide what the canonical pattern is for each
overlap, and then sweep the unique-to-DeviceDefinition elements for
completeness.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Charter an Orders & Observations work item to produce, before substantive
edits, a single side-by-side matrix of every Device element vs every
DeviceDefinition element. The matrix should record, for each overlapping pair:
element name on each side, datatype, cardinality, modifier/summary flags,
short / long definition, value set binding (system + strength), and current
divergences. For each overlap, the WG then assigns one of: (a) **Harmonize**
— keep on both, align names, types, definitions, and bindings (worked example:
`udiCarrier` / `udiDeviceIdentifier` per
[FHIR-47419](https://jira.hl7.org/browse/FHIR-47419)); (b) **Move to Device
only** — remove or deprecate from DeviceDefinition (instance-only concerns);
(c) **Move to DeviceDefinition only** — remove or deprecate from Device
(definitional-only concerns, mirroring
[FHIR-39634](https://jira.hl7.org/browse/FHIR-39634) and
[FHIR-47905](https://jira.hl7.org/browse/FHIR-47905)); (d) **Keep divergent**
— document why. Then sweep the unique-to-DeviceDefinition elements for gaps
(missing definitional concepts an implementer would expect) and propose
additions either in core or as Extensions Pack extensions following the
[FHIR-39388](https://jira.hl7.org/browse/FHIR-39388) precedent. Track each
matrix row that produces a substantive change as its own follow-up Jira so
ballot scoring is granular. Coordinate the work with
[FHIR-54471](https://jira.hl7.org/browse/FHIR-54471) so the broader
DeviceDefinition QA review and this overlap review share findings.

#### Justification

This is what the ticket literally asks for, and it matches the working
group's established pattern: every prior overlap fix (38902, 47419, 47904,
38711, 39634) was driven by a focused element- or section-level analysis. A
shared matrix prevents the contributors to FHIR-54471 and FHIR-54969 from
duplicating effort or reaching contradictory conclusions, and per
[FHIR-38652](https://jira.hl7.org/browse/FHIR-38652) the WG already plans to
rebuild the DeviceDefinition UML "late in the release" — the matrix gives the
UML rework a stable target. Splitting downstream changes into per-element
follow-up tickets is consistent with how the WG has historically processed
this resource pair (each prior overlap concern is its own ticket).

---

### Disposition B: Alternative Approach

#### Proposal

Decline FHIR-54969 as a standalone item and **roll it into
[FHIR-54471](https://jira.hl7.org/browse/FHIR-54471)** as an explicit
deliverable: extend FHIR-54471's scope to include "produce a Device vs
DeviceDefinition element overlap matrix and reconcile each overlap" as one of
the named review tasks alongside text, structure, bindings, search params,
mappings, examples, and profiles. FHIR-54969 would be resolved
"Persuasive — Tracked under FHIR-54471". Substantive element changes would
still spawn per-change Jira tickets so they can be balloted individually.

#### Justification

FHIR-54471 and FHIR-54969 were filed within four days of each other by two
core OO contributors on essentially the same surface area (DeviceDefinition
content correctness). Carrying both forward risks double-bookkeeping,
divergent conclusions, and ambiguity about which ticket "owns" any given
finding. Folding the overlap review into the broader QA pass keeps a single
home for DeviceDefinition cleanup and matches how the WG previously
consolidated 47904 + 47905 (Scope text + types/categories migration) into a
coordinated change.

---

### Disposition C: Decline

#### Proposal

Resolve FHIR-54969 as **Not Persuasive — Already Addressed**, citing the
chain of recently-applied or published harmonization tickets
([FHIR-38902](https://jira.hl7.org/browse/FHIR-38902),
[FHIR-38711](https://jira.hl7.org/browse/FHIR-38711),
[FHIR-47419](https://jira.hl7.org/browse/FHIR-47419),
[FHIR-47904](https://jira.hl7.org/browse/FHIR-47904),
[FHIR-47905](https://jira.hl7.org/browse/FHIR-47905),
[FHIR-39634](https://jira.hl7.org/browse/FHIR-39634),
[FHIR-39388](https://jira.hl7.org/browse/FHIR-39388),
[FHIR-38652](https://jira.hl7.org/browse/FHIR-38652)) as evidence that the
overlap is being managed continuously through ticket-driven changes, and ask
that any newly-discovered concrete overlap or completeness gap be filed as a
specific Jira ticket rather than handled under an open-ended "review
everything" umbrella.

#### Justification

The ticket as written has no concrete deliverable, no acceptance criteria,
and no assignee, which makes it difficult to score for ballot or to close
cleanly. The WG has demonstrably been doing element-level harmonization on a
ticket-by-ticket basis since 2022, and an unbounded "review and complete"
umbrella may simply collect time without producing change. Declining keeps
the ballot tracker honest and forces concrete findings to surface as
actionable items.

---

### Recommendation

**Recommended disposition:** B

The substance of FHIR-54969 is real and important — the Device /
DeviceDefinition boundary still has element-level inconsistencies the WG has
not finished sweeping — so Disposition C (decline) understates the work that
remains. But spinning up FHIR-54969 as an independent, parallel umbrella to
[FHIR-54471](https://jira.hl7.org/browse/FHIR-54471) (Disposition A in its
pure form) almost guarantees overlapping conclusions and bookkeeping pain,
because the two tickets sit on top of the same resource and were filed days
apart by two of the most active OO contributors on Device. Disposition B
preserves everything Disposition A asks for — the element-by-element matrix,
the per-finding follow-up tickets, the coordination with the
[FHIR-38652](https://jira.hl7.org/browse/FHIR-38652) UML rework — but anchors
it to the single, broader DeviceDefinition QA effort already underway under
FHIR-54471. That gives the workgroup one home, one matrix, one set of
follow-ups, and one closure path, which is the right starting position for the
discussion.
