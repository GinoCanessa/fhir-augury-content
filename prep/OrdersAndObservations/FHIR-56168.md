## Summary

The reporter (MarcoWedekind) observes that on the `inventoryListing.item.item`
element, the allowed target types `Medication` and `InventoryItem` are each
listed twice — i.e., the type-reference list contains duplicate entries for
the same two resources rather than a single entry per resource. This is a
straightforward editorial/spec-correctness defect on the `inventoryListing`
artifact, which is currently being incubated in the HL7 OO Incubator IG
rather than core FHIR. No related Jira tickets were surfaced by
cross-reference search, and no Zulip discussion or GitHub item is currently
indexed for this issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Edit the `inventoryListing.item.item` element in the OO Incubator IG so
that its `Reference` type lists each allowed target resource exactly once.
Concretely, deduplicate the `targetProfile` entries on
`inventoryListing.item.item` so that `Medication` and `InventoryItem` each
appear a single time (and verify no other allowed target — e.g.,
`Device`, `Substance`, `BiologicallyDerivedProduct`, `NutritionProduct` —
is duplicated). The fix is editorial in the StructureDefinition (or its
FSH source) for `inventoryListing` in the OO Incubator IG, with no
behavioral or cardinality change.

#### Justification

Per FHIR's `Reference(...)` semantics, each allowed target resource is
expressed as a single `targetProfile`; duplicate entries are
syntactically valid but carry no additional meaning and are
inconsistent with how every other reference-typed element in the
specification is authored. Removing the duplicates is risk-free
(no instances become invalid, no validator behavior changes) and
aligns with the comment from marti_velezis directing the fix to the
OO Incubator IG. The reporter has correctly identified a defect; the
minimal, faithful response is to delete the duplicates.

---

### Disposition B: Alternative Approach

#### Proposal

In addition to deduplicating the `targetProfile` list, take the
opportunity to revisit and rationalize the full set of allowed target
types on `inventoryListing.item.item`. Specifically, evaluate whether
both `Medication` and `InventoryItem` should be permitted (since
`InventoryItem` is the OO Incubator's own generalized item resource and
in many inventory scenarios is intended to wrap or replace direct
`Medication` references), and whether other product resources (e.g.,
`Device`, `NutritionProduct`, `BiologicallyDerivedProduct`,
`Substance`) belong here. Document the rationale for each retained
target in the element's `comment`/`requirements` text so the
duplicate-entry confusion does not recur.

#### Justification

The duplicate-type defect suggests the type list was edited by hand
without a clear policy for what `inventoryListing.item.item` is meant
to reference. While the OO Incubator is by design exploratory, taking
the cleanup beyond a pure deduplication produces a more durable
artifact and reduces the risk of the same class of mistake when the
artifact is eventually proposed back into core. Trade-off vs.
Disposition A: this is more work and slightly more disruptive to
in-flight implementers of the IG, and a target-list change is a
substantive design decision that the workgroup may want to handle in
its own ticket rather than bundling into a janitorial fix.

---

### Disposition C: Decline

#### Proposal

Decline as "Not Persuasive — already addressed by IG move," on the
grounds that the artifact has been relocated from FHIR core into the
OO Incubator IG (per marti_velezis's comment) and any change to
`inventoryListing.item.item` should be tracked against the OO
Incubator IG repository's own issue tracker rather than via a FHIR
core ticket against an artifact that no longer lives in the FHIR
core spec.

#### Justification

Defensible only as a process-routing call: the work itself still
needs to happen, but arguably belongs to the IG, not to the core
ticket queue. However, declining outright would leave a real
spec defect unfixed unless the reporter is explicitly redirected to
file an equivalent IG issue, and the OO Incubator is still
maintained by the same workgroup. Pure-decline is therefore weak on
merits — the duplicate entries are clearly wrong and there is no
substantive disagreement about the fix.

---

### Recommendation

**Recommended disposition:** A

The defect is unambiguous, the fix is purely editorial, and the
reporter has done the workgroup the courtesy of pointing at exactly the
element that needs cleanup. Disposition A delivers the requested fix
with effectively zero risk: deduplicating `targetProfile` entries on
`inventoryListing.item.item` does not change validation, does not
break any existing instance, and matches the authoring convention used
everywhere else in FHIR. Disposition B's broader re-evaluation of the
allowed target set is worthwhile but is a substantive design
discussion that should be tracked under its own ticket rather than
folded into a janitorial correction. Disposition C is process-correct
in spirit (the artifact lives in the OO Incubator IG now) but the
work is owned by the same workgroup either way, so declining adds
process friction without protecting anything. Apply the deduplication
in the OO Incubator IG's `inventoryListing` StructureDefinition and
close the ticket as Persuasive.
