## Summary

Scott Fradkin reports a casing inconsistency on the FHIRcast `UserHibernate`
event page: the JSON example sets `"hub.event": "userHibernate"` (camelCase),
while the surrounding prose and the published canonical link refer to the
event as `userhibernate` (all lowercase). The cross-reference query against
FhirAugury returned no related Jira tickets, Zulip threads, or GitHub items
explicitly linked to FHIR-37010. A keyword search in Zulip for
`userHibernate FHIRCast` did surface neighbouring activity in the
`fhircast-docs` repository — most notably PR #506 "Fhir 41378", which
addressed an unrelated `userHibernate`/`userHybernate` typo in 2023, and
historical PRs against `3-2-3-UserHibernate.md` driven by FHIR-25892. None of
those changes resolved the camelCase-vs-lowercase mismatch flagged in this
ticket; the cached clone of `HL7/fhircast-docs` still shows the mixed casing
on the page.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Editorial fix in `HL7/fhircast-docs` →
`input/pagecontent/3-2-3-UserHibernate.md`: change the JSON example so the
wire-format event value matches the rest of the page and the published
URL. Specifically, replace

```json
"hub.event": "userHibernate",
```

with

```json
"hub.event": "userhibernate",
```

and audit the same file for any other on-the-wire occurrences of the
event name (e.g. subscription `hub.events` arrays, sample payloads) so the
`hub.event` / `hub.events` token is consistently `userhibernate`. Leave
the prose heading "UserHibernate" and the FSH artefact name
`FHIRcastUserHibernateContext` unchanged — those are display / definition
identifiers, not wire values. Apply the same audit to any sibling pages
(`input/pagecontent/3-2-1-*`, `3-2-2-*`, etc.) that show
cross-references to the userhibernate event in JSON examples.

#### Justification

This is a Technical Correction (the ticket type). The FHIRcast wire
contract elsewhere uses all-lowercase event identifiers — the search
results show `userlogout`, `userhibernate`, `syncerror`, `patient-open`,
`imagingstudy-close`, etc. The published canonical event page itself is
`/events/userhibernate/`. Changing the JSON example to `userhibernate`
brings the page into line with both its own prose link and the FHIRcast
catalogue, and is exactly what the reporter asked for. Risk is minimal:
the example is non-normative copy and the change is purely textual.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the inconsistency as a wider editorial-consistency issue rather than
a one-line example fix. In addition to the change in Disposition A:

1. Normalise the page so the event token is written as `userhibernate`
   in *all* code-formatted (`` ` ``, `<tt>`, fenced JSON) contexts and
   `UserHibernate` only in display headings and prose.
2. Add a short "Naming convention" note near the top of the FHIRcast
   Events section stating that `hub.event` / `hub.events` values are
   always lowercase, hyphen-separated tokens, while page titles and
   prose may use display capitalisation. Place it in the shared events
   intro page (`input/pagecontent/3-events.md` or equivalent in the
   current page tree).
3. Run the same lower-case-on-the-wire audit across the rest of the
   `Event Catalog` pages (`UserLogout`, `SyncError`, `Patient-open`,
   `ImagingStudy-open`, …) to catch any sibling slips.

#### Justification

FHIR-37010 is a symptom of a class problem: there is no single place in
`HL7/fhircast-docs` that codifies "wire identifiers are lowercase, prose
is free." Other tickets (FHIR-25892, FHIR-41378) and PR #506 have all
patched casing/spelling on this same family of pages over the years.
A one-paragraph naming-convention note plus a one-pass audit prevents
the next round of identical tickets and is still a small change. The
trade-off vs. Disposition A is slightly more reviewer load and a
broader diff, but the per-event change is still mechanical.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-37010 as already-superseded editorial drift. Close as
"Not Persuasive" / "No Change" on the basis that the page in its current
form distinguishes between the *display* form (`UserHibernate`,
PascalCase, used in headings/prose) and the *wire* form
(`userhibernate`, lowercase, used in the canonical URL), and that the
`userHibernate` token in the JSON example can be read as a non-normative
typo that downstream readers will infer correctly from the surrounding
material.

#### Justification

The example is non-normative; the published canonical event URL
(`/events/userhibernate/`) is what implementers subscribe against. No
implementer has reported subscription failures attributable to the
mismatch in the years since FHIR-37010 was filed, and there are no
linked Zulip threads or GitHub issues raising it again. If the workgroup
wants to ration its FHIRcast editorial bandwidth, treating this as
editorial noise is defensible. (This option is genuinely the weakest of
the three — it is included for completeness.)

---

### Recommendation

**Recommended disposition:** A

This is exactly the kind of low-cost, high-clarity Technical Correction
the ticket type was designed for. The reporter is right: the JSON example
contradicts both the surrounding prose link and the rest of the FHIRcast
event catalogue, and the cached clone confirms the inconsistency is still
present. Disposition A is a one-token edit with no normative impact, no
breaking change for implementers (the wire value `userhibernate` is
already what the canonical URL and sibling event lists use), and removes
the ambiguity for readers of the spec. Disposition B is a reasonable
"upgrade path" if a maintainer is already in the file — the workgroup
could fold its naming-convention note into the same PR — but it should
not block the basic fix. Disposition C is hard to justify when the fix
is one character of casing in a non-normative example.

Before merging, the workgroup should also commission a
`repo-analysis HL7/fhircast-docs` run so subsequent FHIRcast tickets get
a proper repo briefing.
