## Summary

FHIR-36851 ("FHIRcast Observation Profile id") asks the FHIRcast specification
to require that an `id` element be present on instances conforming to the
FHIRcast Observation profile. The reporter (Eric Martin) supplied a one-line
description: "Profile should specify that an id must be present." The ticket
has been resolved as **Persuasive with Modification**, with the resolution
note pointing to **PR486** in the FHIRcast docs repository (`HL7/fhircast-docs`)
as the change that addresses the request. The augury cross-reference index did
not surface any linked GitHub PR/issue, Zulip thread, or Confluence page for
this ticket, so the only direct link to the implementing change is the
resolution comment text itself.

Keyword similarity surfaces a cluster of related FHIRcast profile/conformance
tickets that touch the same area of the spec — most notably
[FHIR-37232](https://jira.hl7.org/browse/FHIR-37232) ("Why is subject profiled

[FHIR-45644](https://jira.hl7.org/browse/FHIR-45644) DiagnosticReport-update
discussion — which provide context on how the FHIRcast working group has been
tightening profile constraints on event payload resources.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the FHIRcast Observation profile in `HL7/fhircast-docs` so that the
resource `id` element is required (`Observation.id` cardinality `1..1` on the
profile, with a short narrative note explaining that consumers MUST be able
to refer to the Observation by its logical id within the event Bundle and
across follow-up events). The change lives in the FHIRcast docs authoring
content for the Observation profile (the same set of files modified by the
referenced PR486), with no impact on the base FHIR Observation resource.

#### Justification

This is the literal request and matches the Working Group's recorded
resolution ("Persuasive with Modification", "Update is included in PR486").
Requiring `id` on Observations carried by FHIRcast event payloads aligns
with the wider direction taken in tickets like
[FHIR-45644](https://jira.hl7.org/browse/FHIR-45644), which call out that
implicit linkage between event-bundle entries is fragile; making `id`
mandatory gives implementers a stable, explicit handle to reference the
Observation across DiagnosticReport-update and similar events, without
forcing a breaking change on the base Observation resource.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) constraining the FHIRcast Observation
profile in isolation, require `id` on **all** event-payload resource
profiles in `HL7/fhircast-docs` via a single, cross-cutting conformance
rule — e.g., a profile-level invariant or a documented "Event Payload
Resource Conformance" section that states every resource included in a
FHIRcast event Bundle MUST carry an `id`. The FHIRcast Observation
profile then inherits the rule rather than re-stating it locally.

#### Justification

Several keyword-related tickets — [FHIR-37232](https://jira.hl7.org/browse/FHIR-37232)
(subject profiled out), [FHIR-37125](https://jira.hl7.org/browse/FHIR-37125)
(contained resource mandated), and especially
[FHIR-45644](https://jira.hl7.org/browse/FHIR-45644) (implicit linkage
between DiagnosticReport and update entries) — show that the underlying
problem is general to FHIRcast event payloads, not specific to Observation.
A profile-by-profile fix risks drift (one resource profile requires `id`,
others don't) and forces every future payload resource to remember the
rule. A single conformance statement also reduces the per-profile narrative
churn and keeps the FHIRcast docs internally consistent. Trade-off vs. A:
slightly larger scope and more review surface, but it eliminates a class of
follow-on tickets.

---

### Disposition C: Decline

#### Proposal

Do not change the FHIRcast Observation profile to require `id`. Rely on
the base FHIR rule that any persisted/referenced resource has an `id`, and
on the Bundle entry's `fullUrl` and references for resolution within the
event payload, rather than introducing a profile-level cardinality
constraint.

#### Justification

The base FHIR specification already says that any resource that is the
target of a reference must be resolvable, and Bundle entries provide
`fullUrl`-based addressing that does not depend on the resource `id`
element being populated for contained or transient cases. Adding a hard
`id` requirement could conflict with use cases where the Observation is
contained or sent as a transient payload that the receiver never needs to
re-fetch. This disposition is the weakest of the three because the
Working Group has already resolved the ticket as Persuasive with
Modification, but it is included for completeness — it would only be
appropriate if PR486 were found on review to over-constrain the profile
relative to actual implementer use cases.

---

### Recommendation

**Recommended disposition:** A

The Working Group has already adopted Disposition A in substance via
PR486 and recorded the resolution as "Persuasive with Modification", so
the most efficient path is to confirm A: keep the `id`-required
constraint on the FHIRcast Observation profile as merged. Disposition B
is genuinely attractive on architectural grounds — the related tickets
([FHIR-37232](https://jira.hl7.org/browse/FHIR-37232),
[FHIR-37125](https://jira.hl7.org/browse/FHIR-37125),
[FHIR-45644](https://jira.hl7.org/browse/FHIR-45644)) all point at the
same underlying need for stronger conformance rules across FHIRcast event
payload resources — and the working group should consider opening (or
resurrecting) a separate ticket to lift the rule from the Observation
profile up to a cross-cutting "Event Payload Resource Conformance"
statement. Disposition C is not recommended; the implicit-linkage risk
called out by FHIR-45644 outweighs the marginal flexibility of leaving
`id` optional on event-payload Observations.
