## Summary

Reporter Rick Lisseveld observes that the SDOHCC Coordination Platform
CapabilityStatement (and other SDOHCC CapabilityStatements in the US SDOH
Clinical Care IG) lists `searchInclude` entries under the wrong resource
sections. Specifically, the entries declared under `ServiceRequest` and `Task`
(for both `mode = client` and `mode = server`) include `_include` pairs whose
source resource is not `ServiceRequest` or `Task` but rather
`HealthCareService` or `PractitionerRole`. Per the FHIR specification,
`CapabilityStatement.rest.resource.searchInclude` only conveys includes
*originating* from that resource type, so listing `HealthCareService:location`
or `PractitionerRole:practitioner` under `ServiceRequest` / `Task` is
structurally incorrect and confusing for implementers. The reporter proposes
moving each misplaced `_include` to the section for the resource that actually
owns the search parameter (e.g., `HealthCareService:location` belongs under
`HealthCareService`, `PractitionerRole:practitioner` under `PractitionerRole`),
or correcting them to a same-resource include where appropriate. A subsequent
comment confirms the issue persists in STU 2.3. The work group has already
triaged this Persuasive with a resolution to update all SDOHCC CapabilityStatements
(client and server) by reorganizing the `_includes` entries into their
associated profiles. There are no cross-referenced Jira, Zulip, GitHub, or
Confluence items recorded for this ticket.

## Details

**Description:**

Using the SDOHCC Coordination Platform Capability Statement as an example, the
searchInclude sections for the Resources ServiceRequest and Task where
`mode = client` or `mode = server` have `_include` pairs that contain a
Resource reference beyond ServiceRequest and Task.

For ServiceRequest the following `_includes` are part of the listing:
`HealthCareService:location`, `PractitionerRole:practitioner`,
`PractitionerRole:organization`.

For Task the following `_includes` are part of the listing:
`HealthCareService:location`, `ServiceRequest:supporting-info`,
`ServiceRequest:pertains-to-goal`, `ServiceRequest:patient`,
`ServiceRequest:requester`, `ServiceRequest:performer`,
`PractitionerRole:practitioner`, `PractitionerRole:organization`.

The other CapabilityStatements are similarly affected.

As a proposed resolution, the `_includes` should be moved to the appropriate
Resource sections or corrected to match the appropriate resource.

**Resolution Description (already recorded on the ticket):**

> We will proceed with updating all of the SDOHCC Capability Statements, for
> both client and server requirements, by reorganizing the `_includes` elements
> and putting them into their associated profiles.

**Comments:**

## Proposed Dispositions

Because no GitHub repository for the SDOHCC IG is registered with FhirAugury,
the dispositions below cite the conceptual artifact (the SDOHCC
CapabilityStatements) rather than concrete file paths. A briefing-grounded
follow-up should be done once the IG repo is onboarded.

### Disposition A: Accept As Requested

#### Proposal

Update every SDOHCC CapabilityStatement in the US SDOH Clinical Care IG (the
SDOHCC Coordination Platform CapabilityStatement explicitly named in the
ticket, plus the other CapabilityStatements the reporter notes are "similarly
affected"), for both `mode = client` and `mode = server` REST entries, so that
each `searchInclude` entry appears under the resource that actually owns the
search parameter. Concretely:

- Remove `HealthCareService:location`, `PractitionerRole:practitioner`, and
  `PractitionerRole:organization` from the `ServiceRequest`
  `rest.resource.searchInclude` list and from the `Task`
  `rest.resource.searchInclude` list.
- Remove `ServiceRequest:supporting-info`, `ServiceRequest:pertains-to-goal`,
  `ServiceRequest:patient`, `ServiceRequest:requester`, and
  `ServiceRequest:performer` from the `Task` `rest.resource.searchInclude`
  list.
- Add the corresponding entries under the correct
  `rest.resource.searchInclude` blocks (i.e., add `HealthCareService:location`
  under `HealthCareService`, `PractitionerRole:practitioner` and
  `PractitionerRole:organization` under `PractitionerRole`, and the
  `ServiceRequest:*` entries under `ServiceRequest`), creating
  `rest.resource` entries for those resource types in the CapabilityStatement
  if they do not already exist.
- Apply the same correction across all other SDOHCC CapabilityStatements
  affected by the same authoring pattern, in both client and server variants.
- Regenerate IG outputs and verify the rendered CapabilityStatement HTML.

This is the action already captured in the ticket's "Resolution Description"
("reorganizing the `_includes` elements and putting them into their associated
profiles").

#### Justification

The current authoring is structurally incorrect against the FHIR
CapabilityStatement specification:
`CapabilityStatement.rest.resource.searchInclude` is the list of `_include`
values *supported when querying that resource type*, where each value is of the
form `SourceType:searchParam[:TargetType]` and the `SourceType` is the
search-root resource for the query — not an arbitrary type that might appear
in the result graph. Listing `HealthCareService:location` under a
`ServiceRequest` query has no defined semantics for clients building
`?_include=...` parameters against a `ServiceRequest` search and is therefore
misleading. Moving each entry under its rightful resource section makes the
CapabilityStatement both validatable and accurate guidance to implementers.
This is also what the reporter requested verbatim and what the work group has
already endorsed as Persuasive.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substance of the request (the misplaced entries must move), but
expand the scope of the change in two ways before regenerating:

1. Audit `searchRevInclude` entries on the same CapabilityStatements with the
   same lens — confirm that each `RevIncludeTarget:param` is actually a valid
   reverse include from the resource section it lives under — and fix any
   parallel mistakes in the same change.
2. For each `_include` being relocated, confirm that the destination resource
   section also declares the corresponding search parameter under
   `rest.resource.searchParam` (e.g., `HealthCareService` must declare
   `location`), and add the missing search parameter declarations as part of
   the same edit so that the relocated includes are coherent with the rest of
   the CapabilityStatement.

The ticket would still be marked Persuasive, but the implementation work item
is broadened to "fix the placement *and* the supporting declarations across
all SDOHCC CapabilityStatements, both modes."

#### Justification

The defect class (search-related entries placed under the wrong resource
section) is unlikely to be limited to forward `_include`s. Every
`searchInclude` mistake of this shape implies both an authoring habit ("list
all the includes that might appear in the resulting bundle, regardless of
search root") and an absence of validation gating; that habit is just as
likely to have produced wrong `searchRevInclude` entries and missing
`searchParam` declarations. Doing the wider audit in the same pass costs
little extra effort while the author is already in the file, prevents a
near-identical follow-up ticket, and yields CapabilityStatements that are
internally consistent rather than just locally corrected. The trade-off vs.
Disposition A is a slightly larger PR and review surface; the gain is a
durable fix rather than a point fix.

---

### Disposition C: Decline

#### Proposal

Decline to make a change to the SDOHCC CapabilityStatements, on the grounds
that the listed `_include` values — even when placed under an unrelated
resource section — convey practical information about the shape of bundles
returned from these endpoints, and that implementers reading the
CapabilityStatement will understand them as a flat enumeration of "includes
in play across this API" rather than as resource-rooted `_include` parameters.
Close the ticket as Not Persuasive and document the IG's authoring convention.

#### Justification

This disposition would only be defensible if the work group decided that the
SDOHCC CapabilityStatements were being read as informative listings rather
than as machine-checkable conformance statements, and that no implementers
build `_include` request parameters from them. Even then, it conflicts with
the published `CapabilityStatement` definition of
`rest.resource.searchInclude`, would fail any structural validation that keys
on that definition, and contradicts the work group's already-recorded
Persuasive resolution. It is included only for completeness.

---

### Recommendation

**Recommended disposition:** A (with the option to escalate to B if the
authors are willing to take the wider audit in the same change).

The ticket has already been triaged Persuasive and the resolution description
on the ticket is exactly Disposition A. The defect is clear-cut against the
FHIR CapabilityStatement specification — `searchInclude` entries belong under
the resource they originate from — so there is no genuine debate about
whether to change the artifact, only about how widely to sweep. Disposition A
is the minimum-credible fix and matches what the reporter and the work group
have already aligned on. Disposition B is recommended only if an editor is
willing to spend additional time auditing `searchRevInclude` and supporting
`searchParam` declarations across all SDOHCC CapabilityStatements; if that
person is not lined up, the work group should land Disposition A now and
write a follow-up ticket for the wider audit. Disposition C should be
rejected: it contradicts both the spec and the work group's prior Persuasive
disposition. As a housekeeping item, the work group should also answer the
2025-06-18 commenter's question (update the *Raised in Version* field to
STU 2.3 on this ticket rather than asking them to file a duplicate).
