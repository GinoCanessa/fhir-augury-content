## Summary

The reporter asks the International Birth And Child Model (IBCM) authors to
add narrative guidance explaining how the IG's `fetal-demise` status relates
to, and differs from, the `deceased[x]` element on the base FHIR `Patient`
resource. The submitter notes that `Patient.deceased` presupposes a person
who has been born, so it cannot semantically represent the death of a fetus;
the IBCM profile therefore introduces `fetal-demise` to capture this
pre-birth state and that distinction should be explicitly documented so
implementers know which element to populate in which situation. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were returned
by the FHIR Augury index for this ticket, so the scope here is narrowly
editorial/clarification.

## Details

**Description:**

> Add explanation how the fetal-demise status is used in comparison with the
> `deceased` status (boolean) of the standard Patient FHIR resource.
>
> This deceased status can only be used if the patient has been born. A
> fetus has not been born yet and therefore this deceased status cannot be
> used.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a short narrative subsection to the International Birth And Child Model
IG — on the page that defines the `fetal-demise` status (typically the
fetus/child Patient profile page or its status value-set binding page) —
explicitly comparing `fetal-demise` to `Patient.deceased[x]`. The wording
should:

1. State that base `Patient.deceased[boolean|dateTime]` applies only to a
   person who has been born.
2. State that for a fetus that has died in utero, implementers must use the
   IBCM `fetal-demise` status (and, where applicable, the associated date
   element) instead of `Patient.deceased[x]`.
3. Include a short example / decision rule ("If the subject was liveborn,
   use `Patient.deceased[x]`; if the subject never achieved live birth, use
   `fetal-demise`") so the distinction is unambiguous.

No structural change to the profile, value set, or cardinalities is
proposed — this is a documentation-only change.

#### Justification

The request is specifically for an explanation, not a model change. FHIR's
editorial convention is to disambiguate overlapping status concepts
in-place on the defining page so implementers encountering `fetal-demise`
immediately see how it relates to the inherited `Patient.deceased[x]`.
Adding this guidance is consistent with how other IGs (e.g. vital records)
document peri-natal status distinctions and carries no interoperability
risk. It directly resolves the reporter's stated concern with minimal
editorial cost.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) freeform narrative, add a structured
constraint and a cross-linked note:

1. On the IBCM fetus/child Patient profile, add an invariant or usage note
   stating that `Patient.deceased[x]` **SHALL NOT** be populated when
   `fetal-demise` indicates the subject never achieved live birth
   (conversely, `fetal-demise` should not be asserted once live birth is
   recorded).
2. Add a "Comparison with base Patient.deceased[x]" callout box on the
   profile page, with hyperlinks to the base `Patient.deceased[x]`
   definition and to the `fetal-demise` status code/value-set page, so the
   relationship is navigable from both directions.
3. Add a conformance example (one live-born-then-deceased Patient, one
   fetal-demise Patient) so the two patterns are visibly contrasted.

#### Justification

A pure narrative note (Disposition A) relies on implementers reading the
prose. Converting the distinction into a machine-checkable invariant plus a
worked example materially reduces the chance of mis-population in real
systems and makes the guidance testable by validators. The trade-off is
slightly more IG authoring effort and the need to confirm the invariant
does not conflict with valid edge cases (e.g. late-term fetal demise
records that also carry a `deceasedDateTime` for administrative reasons).
If such edge cases exist, the invariant can be relaxed to a `warning`
severity rather than `error`.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** on the grounds that the distinction
between `fetal-demise` (an IBCM-specific status) and
`Patient.deceased[x]` (a base-resource element) is already implicit in
the definitions of each element and does not require additional narrative
in the IG.

#### Justification

The base `Patient.deceased[x]` definition already scopes the element to a
person, and `fetal-demise` is defined within the IBCM profile's own
value set, so a careful reader can derive the distinction from existing
definitions. Declining would be defensible if the workgroup judges the
IG is already at capacity for editorial additions or if the distinction
is considered out of scope for the IBCM profile page. However, this
rationale is weak: the reporter's question demonstrates that the
distinction is *not* obvious to at least one implementer, which is
exactly the signal a clarification ticket is supposed to carry.

---

### Recommendation

**Recommended disposition:** A

The reporter is asking for documentation, and the cheapest, lowest-risk fix
is to provide it. Disposition A fully resolves the stated concern with no
conformance impact and no risk of breaking existing implementations.
Disposition B is attractive in principle but escalates a documentation
request into a conformance change; without any community discussion on
record (no Zulip threads, no linked tickets, no GitHub activity), committing
to a `SHALL NOT` invariant risks over-constraining edge cases the workgroup
has not yet canvassed. Patient Care should adopt A now and, if implementer
feedback later surfaces real mis-population, upgrade to B in a follow-up
ticket. Disposition C should be rejected — the existence of this ticket is
itself evidence that the current text is insufficient.
