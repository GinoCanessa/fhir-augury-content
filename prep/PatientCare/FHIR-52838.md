## Summary

(or, as a minimum, formally deprecated) in the next IPS release after

### Disposition A: Accept As Requested

#### Proposal

After IPS 2.0 publication, retire the IPS "Medications" example value
set entirely from the IPS IG. Concretely:

- In the next IPS release branch, remove the `ValueSet` resource
  defining the IPS Medications example value set from the IG sources
  (and any associated `sushi` / FSH definition, if used).
- Audit the IPS IG for any remaining references to its canonical URL —
  in profiles (e.g., `MedicationRequest (IPS)`,
  `MedicationStatement (IPS)`), bindings, narrative pages, and
  examples — and remove or repoint them. Per the ticket, no normative
  artifact still binds to it.
- Update the IPS Medications section narrative to remove any pointer to
  the example value set.
- Add a short release-note / changelog entry stating that the
  Medications example value set has been retired because it was unused
  and incomplete.

#### Justification

The reporter states the value set is unused by the rest of IPS and is
self-described as incomplete with a plan to remove. Carrying an unused,
incomplete example value set forward in a published IG is a known
implementer hazard: tooling and downstream profiles can pick it up by
canonical URL even when the IG no longer relies on it, and "example"
bindings tend to be mistaken for guidance. Removing it after 2.0
publication respects the existing FMG/PC publication freeze (consistent
with John D'Amore's own comment and the IPS Block-Vote process surfaced
in the IPS Zulip thread above) while delivering the cleanup at the
earliest non-disruptive opportunity.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of deleting the value set in the release immediately following
IPS 2.0, mark it as **deprecated / retired** for one publication cycle
and remove it in the cycle after that:

- Set the IPS Medications example `ValueSet.status` to `retired`.
- Add a `ValueSet.description` and a banner in the IPS Medications
  section narrative explaining that the value set is unused, incomplete,
  retained only for one cycle to give downstream IGs and implementers
  time to migrate, and scheduled for deletion in the following IPS
  release.
- File a follow-up tracker (linked from this ticket) to perform the
  hard delete at that next cycle.
- Leave existing internal references (if any are found in the audit)
  unchanged for this cycle, only updating ones that actively mislead
  implementers.

#### Justification

`status = retired` is the FHIR-native, terminology-friendly mechanism
for signalling "do not use" without breaking the resolvability of the
canonical URL for downstream IGs that may still reference it
transitively. The ticket itself offers this as an acceptable fallback
("…or at least mark as deprecated"). The trade-off versus Disposition A
is a slower clean-up in exchange for a softer deprecation curve for any
out-of-tree consumers that may have picked the value set up by canonical
URL — something an "example" status doesn't formally protect against
but which a measured deprecation respects.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave the IPS Medications example value set in
place as-is. Rationale category: *no implementer harm demonstrated*
combined with *change deferred indefinitely* — i.e., classify the
ticket as "Not Persuasive" on the basis that "example" value sets in
FHIR carry no normative weight, are explicitly non-binding, and removing
one introduces a small backwards-incompatibility risk for any
out-of-tree consumer that resolved the canonical URL.

#### Justification

A defensible decline rests on FHIR's existing semantics: an "example"
binding strength already tells implementers the value set is
illustrative only, so an incomplete example value set is, by
definition, acceptable. Deletion changes a published, resolvable
canonical URL — even a minor risk worth weighing. This disposition is
the weakest of the three because the reporter has explicitly stated
that the value set is unused inside IPS and self-flagged for removal in
its own description; absent a concrete consumer that depends on it,
"leave it alone" mostly defers the housekeeping rather than resolving
it.

---

### Recommendation

**Recommended disposition:** B

Disposition B (retire now, delete next cycle) is the best balance of
the three. It honours the reporter's stated intent (the value set
should go away), it explicitly aligns with the reporter's own fallback
("at least mark as deprecated"), and it respects the post-2.0
publication timing constraint surfaced in the IPS Block-Vote Zulip
discussion. Compared with Disposition A, it gives any out-of-tree IG or
tool that may have picked up the canonical URL one full publication
cycle of warning via `status = retired`, which is the standard FHIR
signal for "stop using this." Compared with Disposition C, it actually
delivers the cleanup the reporter asked for instead of deferring it
indefinitely. The workgroup should confirm during disposition that no
IPS profile or section still binds to this value set (an audit task for
the implementer) before scheduling the hard delete in the cycle after
the retirement release.
