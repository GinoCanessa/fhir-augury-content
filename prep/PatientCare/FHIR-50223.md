## Summary

The reporter identifies an inconsistency in the US SDOH Clinical Care IG
between the canonical task-status code value `cancelled` (two "l"s, as
defined in `http://hl7.org/fhir/task-status` and used by the
`SDOHCCValueSetReferralTaskStatus` ValueSet) and several places in the IG
narrative that spell it `canceled` (one "l"). Because the FHIR code system
literal is `cancelled`, any narrative, image, or table that says `canceled`
risks misleading implementers who copy the term verbatim into instances or
queries. The ticket asks that the IG narrative be brought into alignment
with the code system value. The reporter's comment notes that PR
[HL7/fhir-sdoh-clinicalcare#157](https://github.com/HL7/fhir-sdoh-clinicalcare/pull/157)
is staged to land the corrections once the change is approved. No related
Jira tickets, Zulip threads, or Confluence pages were surfaced via Augury.

## Details

**Description:**

The ValueSet `SDOHCCValueSetReferralTaskStatus` includes the code
`http://hl7.org/fhir/task-status|cancelled` (with two "l"s). However there
are numerous places in the IG that spell `cancelled` with one "l", which
could cause mistakes or confusion.

This issue exists in the following locations:

- `StructureDefinition-SDOHCC-TaskForReferralManagement.html#status-management`
  - including the image and table
- `checking_task_status.html#checking-task-status`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the corrections staged in PR
[HL7/fhir-sdoh-clinicalcare#157](https://github.com/HL7/fhir-sdoh-clinicalcare/pull/157)
to the US SDOH Clinical Care IG narrative so every reference to the
referral Task status uses the canonical spelling `cancelled` (two "l"s)
matching `http://hl7.org/fhir/task-status`. Concretely:

- In the profile-intro markdown backing
  `StructureDefinition-SDOHCC-TaskForReferralManagement.html#status-management`
  (under `input/pagecontent/`), replace every occurrence of `canceled`
  with `cancelled` in the body text and in the status table cells.
- Re-export the status-management diagram from its source asset with the
  corrected spelling and replace the PNG/SVG under `input/images/`.
- In `input/pagecontent/checking_task_status.md` (the source for
  `checking_task_status.html#checking-task-status`), replace every
  occurrence of `canceled` with `cancelled`.
- Run a repo-wide grep for `\bcanceled\b` against `input/` and fix any
  additional narrative hits found beyond the two locations called out
  in the ticket; leave `output/` and `temp/` untouched (regenerated by
  the IG Publisher).

#### Justification

The request is a pure narrative-vs.-code-system alignment fix. The IG
already binds to `http://hl7.org/fhir/task-status`, whose literal value is
`cancelled`; any narrative that says `canceled` is objectively
inconsistent with the spec the IG itself ships and is exactly the kind of
defect that drives implementers to file conformance tickets. The reporter
has already done the engineering work in PR #157, so accepting only
requires review and merge — there is no design judgement at stake. This
matches the Technical Correction ticket type and Medium priority that
Patient Care has assigned.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the substantive intent of A, but expand the scope to a one-pass
"task-status terminology audit" of the entire US SDOH Clinical Care IG
before merging. Specifically:

1. In addition to the two pages the reporter calls out, search every file
   under `input/` (markdown, FSH, JSON resources, examples, and image
   source files such as Drawio/SVG) for any of the `task-status` literals
   (`requested`, `received`, `accepted`, `rejected`, `ready`,
   `cancelled`, `in-progress`, `on-hold`, `failed`, `completed`,
   `entered-in-error`) and confirm the spelling matches the code system
   exactly.
2. Add a short sentence to the IG's narrative conventions (e.g., on the
   IG home page or a `notes`/`conventions` page) stating that all status
   codes in narrative must match the underlying CodeSystem literal
   verbatim.
3. Consider adding a CI lint (e.g., a simple `grep`/Sushi pre-publish
   check) that fails the build if `canceled` appears anywhere in
   `input/`.

#### Justification

The reporter explicitly says the issue exists in "numerous places" and
the ticket only enumerates two; treating this as a focused PR risks
shipping the next revision with a fresh `canceled` typo elsewhere. The
incremental cost over A is small — one extra grep pass plus a one-line
convention note — and the lint guarantees the regression doesn't recur in
future ballots. Trade-off vs. A: a few extra hours of editor time and a
slightly larger PR review surface.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive — Editorial Only / Won't Fix Now**,
on the basis that the spelling difference is a minor narrative blemish
and does not affect machine-readable conformance: instances and
constraints already carry the correct `cancelled` literal because they
are authored against the bound ValueSet, and implementers who follow the
binding rather than the prose will produce conformant Tasks.

#### Justification

Defensible only as a scope/timing argument. The actual normative
artifacts (ValueSet, constraints, `Task.status` binding) are correct, so
no implementer following the conformance rules will produce a
non-conformant resource because of the typo. If the work group is in a
freeze window or wants to bundle narrative-only fixes into a future
errata batch, "decline now, defer to the next publication cycle" is a
coherent position.

---

### Recommendation

**Recommended disposition:** B

This is a Technical Correction with PR #157 already staged, so doing
nothing (Disposition C) wastes work the reporter has already done and
leaves a defect that any reader can spot from the rendered HTML — a poor
look for a published HL7 IG. Disposition A is the safe minimum and
should be accepted at a bare floor. The reporter's own wording —
"numerous places in the IG" — is a strong hint that the two enumerated
URLs are examples, not an exhaustive list, so an audit-and-lint pass
(Disposition B) is the disposition most likely to actually close the
underlying defect for good rather than playing whack-a-mole in a future
ticket. The marginal effort over A is very small, and the CI lint pays
for itself the first time it catches a regression.
