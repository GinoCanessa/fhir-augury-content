## Summary

Grahame Grieve filed FHIR-39563 to ask Patient Care whether
`ClinicalImpression.description` (currently typed as `string`) is expected
to carry more than a single paragraph of free text, and — if so — whether
the element's datatype should be changed from `string` to `markdown`. The
ticket was raised as a follow-up to the related housekeeping ticket
FHIR-38714 ("Correct markdown datatype definition"), which prompted a
sweep across the spec for `string` elements that should arguably be
`markdown` because they carry narrative-style content. The cross-reference
graph also surfaces a long-running implementer Zulip topic
(`#implementers > ClinicalImpression`) where the broader scope and
viability of `ClinicalImpression` itself is being debated; in the
2023-06-15 Patient Care call (recorded in the ticket comments) the WG
expanded the scope of the conversation to "should `description` be
renamed to `reason`?" and "should we keep `ClinicalImpression` at all,
given low implementer uptake?". So while the literal request is a
`string` → `markdown` datatype change, the disposition needs to be
considered alongside the open question of what `description` is *for*
on this resource and whether the resource has enough adoption to justify
breaking changes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Change the datatype of `ClinicalImpression.description` from `string` to
`markdown` in the FHIR Core specification (next normative-or-trial-use
publication). No other element on `ClinicalImpression` is affected; no
cardinality, binding, or definition text changes. Definition text is
left unchanged ("A summary of the context and/or cause of the assessment
- why / where it was performed, and what patient events/status prompted
it."). Add a short release-note entry under the "Substantive Changes"
list for ClinicalImpression noting that instance content that was valid
as `string` remains valid as `markdown` (markdown is a superset of
string at the lexical level).

#### Justification

This is a minimally invasive change that directly answers Grahame's
question: the element is described as a *summary* of context and cause,
which is exactly the kind of free-text narrative that FHIR-38714's
sweep was meant to normalise to `markdown`. It is consistent with the
established FHIR pattern of using `markdown` for elements that may
contain more than a sentence of human-authored prose (e.g. `*.purpose`,
`*.copyright`, many resource `description` elements such as
`CapabilityStatement.description`, `Questionnaire.description`,
`Endpoint.description`). It does not require a rename, does not
prejudice the broader question about whether `description` should
become `reason` or about the resource's future, and is implementer-safe
because plain-string content remains valid markdown.

---

### Disposition B: Alternative Approach

#### Proposal

Defer the datatype change and instead treat FHIR-39563 as the trigger
for a structural cleanup of `ClinicalImpression.description`:

1. Rename the element from `description` to `reason` (per the 2023-06-15
   PCWG minutes: the current text describes *why/where* the assessment
   was performed, which is closer to a `reason` semantic than a
   `description`).
2. Re-evaluate whether the resulting element should remain `string`,
   become `markdown`, or be re-modelled as `CodeableReference(reason)` /
   `Reference(Encounter | Condition | Observation)` so that the
   "context/cause" of the assessment is expressed as data rather than
   prose.
3. Pair this with the existing thread about clarifying `summary`
   vs. `description` so the resource's narrative-bearing elements have
   distinct, non-overlapping definitions.
4. Hold the change until the parallel Zulip/PCWG question about whether
   to retire `ClinicalImpression` is resolved, so we don't invest a
   breaking rename in a resource that may be deprecated.

#### Justification

The 2023 PCWG minutes are explicit that `description` is the *wrong
name* for what the element actually carries, and the implementer thread
shows there is genuine confusion about the resource's scope. Doing the
markdown change in isolation locks in an element name the WG itself has
flagged as misleading and would later require a second breaking change
to fix. A coordinated rename + datatype review costs roughly the same
implementer disruption as Disposition A but addresses the underlying
modelling defect rather than just the surface datatype question. The
trade-off vs. A is timeline: this disposition needs the
`ClinicalImpression`-scope discussion to converge first, so it is
slower.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-39563 as filed and close it with the rationale that the
existing `string` typing is adequate: `ClinicalImpression.description`
is defined as a *summary* of context/cause, not as authored narrative,
and implementers have not reported encountering content that exceeds a
single paragraph or that requires markdown formatting. Direct
implementers who genuinely need formatted narrative for an assessment
to either (a) use `ClinicalImpression.summary` (which is intended for
the findings narrative and is already typed `string` — and could be
revisited under a separate ticket if real markdown demand surfaces),
(b) attach a `DocumentReference`, or (c) use the resource's `text`
narrative.

#### Justification

The Zulip thread shows ClinicalImpression has very low real-world
adoption and that PCWG itself is openly asking whether the resource
should be retired; in that context the cost of any breaking-ish change
(even a string→markdown widening, which forces servers/validators that
strip control characters or enforce single-line semantics to update)
is hard to justify on the back of a hypothetical "if it might contain
more than a paragraph". No implementer in the cross-referenced
discussions has actually requested markdown rendering for this element.
Declining keeps the resource frozen pending the larger
keep-vs-deprecate decision.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the cleanest answer to the literal question on the
ticket and is the only option that can move now without being blocked
on the (still-open) PCWG decisions about renaming `description` and
about the resource's long-term future. The datatype widening from
`string` to `markdown` is implementer-safe (existing string instances
remain valid markdown) and brings the element into line with the
FHIR-38714 sweep's stated rationale and with the established pattern
for narrative-bearing `*.description` elements across the spec.
Disposition B is the *better* answer in the abstract — the WG already
acknowledges the element is mis-named — but it is gated on the
ClinicalImpression-scope discussion and on a `description` → `reason`
rename that has its own design ramifications; pursuing it should be
done as a separate, follow-up ticket so FHIR-39563 itself is not held
hostage to that larger question. Disposition C trades a small amount
of correctness for stalling on a resource whose future is uncertain;
that is defensible only if the WG is genuinely converging on
deprecating `ClinicalImpression`, which the Zulip thread does not yet
support.
