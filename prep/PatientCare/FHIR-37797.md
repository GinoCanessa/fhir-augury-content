## Summary

Stephen Chu reports that the `ClinicalImpression` resource lacks a
semantically unambiguous way to convey, alongside a finding (e.g., that a
patient's clinical condition is improving or worsening), the **clinical
intervention or care context** in which that change occurred — i.e.,
whether the observed pattern is happening *because of* or *despite* a
specific therapy or intervention. Today, exchanging this contextual
information about `ClinicalImpression.finding.item` requires shoehorning
it into elements that were not designed for it. The reporter notes that
`ClinicalImpression.supportingInfo` is the closest candidate, but its
current definition ("Information supporting the clinical impression,
which can contain investigation results.") makes it ill-suited for
asserting an attribution relationship between a finding and an
intervention. The ticket does not propose a specific structural change;
it asks the work group to define a clean way to capture this
finding-to-intervention context in the resource. There are no linked or
cross-referenced Jira tickets, no Zulip discussions, and no GitHub
items associated with this request in the indexed sources.

## Details

**Description:**

> When exchanging information about ClinicalImpressing.finding.item,
> e.g. about whether a patient's clinical condition change pattern is
> improving | worsening, it is useful | important to provide additional
> context, e.g. whether the change is in relation to or despite certain
> clinical intervention or care given to the patient.
>
> ClinicalImpression resource does not have semantically unambiguous
> structure to support this requirement
>
> ClinicalImpression.supportingInfo may be considered
> Definition: Information supporting the clinical impression, which can
> contain investigation results.
> But Definition makes it difficult for this structure to be used for
> this requirement: to provide information about whether the change in
> pattern is related to or despite certain clinical intervention or
> care given to the patient

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a dedicated structure on `ClinicalImpression` to express the
clinical-intervention context that a finding is being interpreted
against. Concretely, introduce a new backbone element such as
`ClinicalImpression.finding.context` (0..*) with sub-elements:

- `intervention` (0..1, `Reference(Procedure | MedicationAdministration |
  MedicationStatement | CarePlan | ServiceRequest | Task)`) — the
  intervention or care plan the finding is being related to.
- `relationship` (1..1, `code`, bound to a new value set, e.g.
  `clinical-impression-finding-context-relationship`, with starter
  codes such as `in-response-to`, `despite`, `unrelated-to`,
  `pending-evaluation-of`).
- `note` (0..*, `Annotation`) — free-text clarification.

Mark the change for the next normative-eligible release of
`ClinicalImpression` and update the resource narrative / mappings /
examples to demonstrate the "improving despite intervention X" and
"worsening in response to intervention Y" patterns.

#### Justification

The reporter's underlying need — distinguishing "finding *because of*
intervention" from "finding *despite* intervention" — is genuinely
information that today cannot be captured unambiguously on
`ClinicalImpression`. `supportingInfo` is documentation/evidence, not
an attribution, and bending its definition would erode it for its
existing use. Modeling the relationship explicitly on
`finding` (rather than on the resource as a whole) keeps the
attribution attached to the specific finding it qualifies, which is the
granularity the reporter calls out. A coded `relationship` element with
a small starter value set follows the precedent of similar contextual
qualifiers elsewhere in FHIR (e.g., `Condition.evidence`,
`Observation.derivedFrom` paired with `interpretation`).

---

### Disposition B: Alternative Approach

#### Proposal

Do not change the base resource. Instead, publish a standard extension
under the Patient Care work group — e.g.,
`http://hl7.org/fhir/StructureDefinition/clinicalimpression-finding-context`
— that can be applied to `ClinicalImpression.finding`. The extension
would carry the same payload as Disposition A
(`intervention` reference + `relationship` code + optional `note`),
bound to a new code system / value set published alongside the
extension.

In parallel, tighten the wording of `ClinicalImpression.supportingInfo`
to make explicit that it is for *evidence* supporting the impression
(investigations, prior notes, etc.) and **not** for asserting a
finding-to-intervention relationship, with a cross-reference to the
new extension.

#### Justification

`ClinicalImpression` has been historically lightly used and has been a
target of multiple model-clarification proposals; adding a new
backbone element is a substantive change with downstream impact on
implementers, profiles, and tooling. An extension achieves the same
expressive outcome without forcing every implementer of
`ClinicalImpression` to consider a new optional structure, and it can
be ballotted faster than a base-resource change. If real-world adoption
of the extension is strong, the work group can later promote it into
the core resource — exactly the lifecycle FHIR encourages for
features whose demand is not yet broadly demonstrated. The companion
clarification of `supportingInfo` directly addresses the reporter's
secondary complaint that today's wording invites misuse.

---

### Disposition C: Decline

#### Proposal

Decline the request as currently scoped. Take the position that the
intended semantics can already be conveyed by combining existing
resources — e.g., representing the intervention as a
`Procedure` / `MedicationAdministration` and the finding as an
`Observation` (with `Observation.derivedFrom` /
`Observation.focus` linking the two, and `interpretation` describing
"improving / worsening"), then referencing those `Observation`
instances from `ClinicalImpression.finding.itemReference`. Recommend
that the reporter raise a follow-up ticket only if a concrete,
implemented use case shows that this approach is insufficient.

#### Justification

`ClinicalImpression` is intentionally a lightweight wrapper around a
clinician's assessment; rich attribution between findings and
interventions is generally modeled in `Observation` (with
`derivedFrom`, `focus`, `interpretation`) and in
`Procedure` / `MedicationAdministration` outcomes. Adding a parallel
mechanism on `ClinicalImpression` risks introducing two ways to say
the same thing, which historically harms interoperability more than
it helps. The ticket also lacks comments, linked tickets, Zulip
discussion, or implementer voices in the indexed sources, suggesting
limited demonstrated demand.

---

### Recommendation

**Recommended disposition:** B

Disposition B threads the needle: it gives the reporter the
expressivity they are asking for, on the right element
(`ClinicalImpression.finding`), without committing the work group to a
base-resource change for which there is currently no comment activity,
no linked tickets, no Zulip discussion, and no GitHub work indexed
against this ticket. A standard extension is the FHIR-canonical way to
introduce a new optional concept whose demand is plausible but
unproven, and it preserves a clean migration path into the core
resource if uptake justifies it. Pairing the extension with a
definition tightening of `ClinicalImpression.supportingInfo` directly
neutralizes the reporter's specific concern that `supportingInfo` is
being conscripted into a role it was not designed for. Disposition A
is the right *eventual* destination if and when implementer evidence
materializes; Disposition C is too dismissive given that the reporter
has identified a real semantic gap, even if the demand signal in the
indexed sources is currently quiet.
