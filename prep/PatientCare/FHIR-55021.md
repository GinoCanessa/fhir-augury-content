## Summary

This R6 ballot comment, raised by Reuben Daniels, asks Patient Care to remove the
`FamilyMemberHistory.sex` element from the base resource and instead recommend
that implementers represent sex/gender for a family member using the Gender
Harmony FHIR extensions in the FHIR Extensions Pack — specifically Recorded Sex
or Gender (RSG), Sex Parameter for Clinical Use (SPCU), and (optionally) Gender
Identity. The reporter argues that the current element is semantically muddled
(name says "sex", binding is `AdministrativeGender`, comment talks about
*genetic* sex with caveats about reporter imprecision), that the Gender Harmony
extensions provide cleaner semantics, and that retaining `sex` alongside the
extensions creates confusing dual paths with no defined relationship between
them. Because `FamilyMemberHistory` is FMM 2 / Trial Use in R5 but is expected
to go normative in R6, this is the practical last window in which a breaking
removal would be tolerable.

The Patient Care WG already triaged this at the January 2026 WGM (joint with
Terminology Infrastructure) and voted 24-for / 2-abstain / 0-against to take it
to the implementer community for input, scheduling an in-person follow-up for
2026-04-09. There are no formal Jira issue links, but the ticket history
explicitly references two prior tickets: [FHIR-17564](https://jira.hl7.org/browse/FHIR-17564)
("Rename FamilyMemberHistory.gender to birthSex" — Published, Not Persuasive
with Modification, 2018) and, surfaced in the implementer Zulip thread,
[FHIR-43360](https://jira.hl7.org/browse/FHIR-43360) ("May rename
FamilyMemberHistory.sex" — Resolved No Change / Not Persuasive, 2024). Both
prior tickets considered renaming the element rather than removing it; both
were closed without change. FHIR-55021 reframes the question as removal +
recommendation of Gender Harmony extensions, and is the most recent and most
concrete proposal on this element.

## Details

**Description:**

The definition of the FamilyMemberHistory resource's sex element is highly
non-specific. It is therefore suggested that it be removed in favour of using
the Gender Harmony specification's Recorded Sex or Gender (RSG) or Sex Parameter
for Clinical Use (SPCU) FHIR extensions.

### [implementers > Question for implementers of FamilyMemberHistory](https://chat.fhir.org/#narrow/stream/implementers/topic/Question%20for%20implementers%20of%20FamilyMemberHistory)

Reuben Daniels opened this thread on 2026-04-07 to gather implementer input
following the WGM motion. The discussion exposed three positions: (1) **Remove
`sex`, recommend Gender Harmony extensions** — Reuben's original proposal;
Cooper Thompson (Epic) confirmed an R6 rename/removal carries no breaking
impact for them and noted the current element is "a semantic mess" because the
property name, value set, and definition comment each describe a different
concept. (2) **Rename `sex` → `gender` and align all "person-like" resources
on Gender Harmony extensions** — Hans Buitendijk argued forcefully that
removing `sex` only from FamilyMemberHistory creates inconsistency with
Patient, RelatedPerson, Practitioner, and Person, all of which carry
`.gender` against the same value set; the consistent fix is to rename and
then layer Gender Harmony extensions across all five. Lloyd McKenzie agreed
that "changing it from 'sex' to 'gender' might be justifiable", and Cooper
agreed if the comment is also updated. (3) **Keep the element as-is** —
Lloyd McKenzie argued the base element captures the 80% lowest-common-
denominator concept (genetically male/female as known to a lay reporter) that
implementers can rely on for interoperability, and that environments needing
finer granularity should layer the Gender Harmony extensions on top. He
maintained the meaning is intentionally distinct from `Patient.gender` (genetic
inheritance vs. administrative purpose), justifying the different name. Hans
closed by noting that, given patient-recall imprecision, status quo may yield
the most usable answer and that both removal and replacement risk losing the
data of interest. Consensus was not reached; the implementer thread surfaced
strong support for *some* change but disagreement on whether removal,
rename-with-extensions, or status-quo-plus-extensions is best, and explicitly
re-opened the FHIR-43360 question.

Participants: Reuben Daniels, Cooper Thompson (Epic), Hans Buitendijk,
Lloyd McKenzie.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the `FamilyMemberHistory.sex` element from the base resource definition
in FHIR R6. In the resource's narrative/notes section, add an informative
paragraph that recommends, when sex/gender of the family member is required,
use of the FHIR Extensions Pack extensions:
[`individual-recordedSexOrGender`](https://hl7.org/fhir/extensions/StructureDefinition-individual-recordedSexOrGender.html)
(RSG, with the SNOMED-based value set defined by the Gender Harmony IG, falling
back to AdministrativeGender if the implementer requires it),
[`patient-sexParameterForClinicalUse`](https://hl7.org/fhir/extensions/StructureDefinition-patient-sexParameterForClinicalUse.html)
(SPCU), and
[`individual-genderIdentity`](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-individual-genderIdentity.html).
Note that these may be used together. Add an R5→R6 substantive change entry
flagging the breaking removal, and document a migration recommendation that
existing R4/R4B/R5 `sex` values should be carried forward as the RSG extension
populated with the same `AdministrativeGender` code (preserving round-trip
fidelity for the trivial mapping case).

#### Justification

Aligns directly with the WGM motion (24-for, 0-against) and the reporter's
argument. R6 is the last realistic opportunity to make a breaking change here
before normative status increases the cost. Removing the element resolves the
documented semantic incoherence (name, binding, and comment describing three
different things) by deferring to extensions whose semantics are explicitly
specified. Cooper Thompson (Epic) confirmed in the implementer thread that an
R6 removal carries no breaking impact for them. The downside — inconsistency
with `Patient.gender`/`Practitioner.gender`/etc. — is real but is partially
mitigated by the fact that those resources can themselves carry the same
Gender Harmony extensions, so harmonisation can proceed by layering rather
than by also touching the base elements on those resources.

---

### Disposition B: Alternative Approach — Rename `sex` to `gender`, fix the comment, and recommend Gender Harmony extensions alongside

#### Proposal

Do **not** remove the element. Instead, in R6:

1. Rename `FamilyMemberHistory.sex` to `FamilyMemberHistory.gender` (0..1, type
   `code`).
2. Keep the extensible binding to `AdministrativeGender`.
3. Rewrite the element's `definition`, `requirements`, and `comment` so that
   all three describe the same concept — the lay-reportable
   administrative-gender-style attribution of the family member, explicitly not
   a precise sex parameter — removing the current "ideally genetic" language.
4. Add an informative note at the resource level recommending the
   `individual-recordedSexOrGender`, `patient-sexParameterForClinicalUse`,
   and `individual-genderIdentity` extensions for implementations that need
   richer or more clinically precise representation, and stating that these
   extensions may be used in addition to (or, where the lay value is unknown,
   instead of) `gender`.
5. Reopen and supersede [FHIR-43360](https://jira.hl7.org/browse/FHIR-43360)
   (whose disposition Hans Buitendijk argued did not properly consider
   cross-resource consistency).

#### Justification

This addresses the implementer-thread majority concern about consistency with
`Patient`, `Practitioner`, `RelatedPerson`, and `Person`, all of which carry a
`gender` element bound to `AdministrativeGender`. Both Lloyd McKenzie and
Cooper Thompson indicated the rename (with comment fix) is acceptable. It
preserves an 80%-rule lowest-common-denominator value that consumers can rely
on without having to special-case extension presence — a real interoperability
benefit Lloyd emphasised — while still letting Gender Harmony adopters layer
RSG/SPCU/Gender Identity on top. It also matches the *patient recall* reality
Hans noted: what a relative knows about their aunt is closer to administrative
gender than to a clinical sex parameter, so the element name and content
align. The cost is keeping a slightly redundant path when extensions are also
present, but no defined relationship between them is required because the
extensions' semantics are distinct.

---

### Disposition C: Decline

#### Proposal

Decline the request. Keep `FamilyMemberHistory.sex` as it is in R5, including
its name, 0..1 cardinality, extensible binding to `AdministrativeGender`, and
existing definition/comment. Treat the Gender Harmony extensions as already
available to implementers who need richer semantics, and note that the prior
ticket [FHIR-43360](https://jira.hl7.org/browse/FHIR-43360) considered this
area in 2023/2024 and resolved Not Persuasive. Rationale category: *existing
mechanism already addresses the need and the proposed change is a breaking
change that loses guaranteed-interoperable data*.

#### Justification

This is the position Lloyd McKenzie advocated and Hans Buitendijk closed the
implementer thread with: removing or replacing `sex` risks losing the
patient-reported value of interest because most reporters cannot distinguish
between administrative gender, recorded sex/gender, gender identity, and SPCU
— so the lowest-common-denominator element is what will actually be populated
in practice. The Gender Harmony extensions are already available on this
resource (FHIR allows extensions on any element/resource), so no spec change
is required to enable them. A prior triage on FHIR-43360 reached the same
conclusion. Declining avoids a breaking change just before normative status,
preserves cross-resource consistency, and avoids the worst-case outcome where
implementers populate neither the (removed) base element nor the (optional)
extensions.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of the three. The implementer thread
demonstrated that the *strongest* objection to A is not the change itself but
the cross-resource inconsistency it creates — `FamilyMemberHistory` would be
the only "person-like" resource without a base sex/gender element, and Hans
Buitendijk's argument about consistency with Patient/Practitioner/
RelatedPerson/Person went unrebutted on its merits. Disposition C, while
defensible, leaves the documented semantic incoherence in place and was
explicitly rejected by the WGM 24-0 motion; Cooper Thompson's "we'd need to
fix *something*" is the right read. Disposition B fixes what is actually
broken (the name/binding/comment mismatch), preserves the 80%-rule
lowest-common-denominator value Lloyd argued is essential for reliable
interoperability, restores consistency with the four sibling resources,
and still channels implementers toward the Gender Harmony extensions for
richer semantics — capturing the substantive intent of the ballot comment
without paying the cost of a removal that several senior implementers
warned against. It does mean re-opening FHIR-43360, but the implementer
thread already provides the new context that justifies revisiting it.
