## Summary

The ticket asks the US Physical Activity (PA) IG to stop using its current
LOINC codes for "moderate physical activity minutes" and "vigorous physical
activity minutes" because, per the reporter, the IG is not allowed to use
those LOINC codes the way it currently does (i.e., as the `Observation.code`
for recording observed/reported minutes of activity at a given intensity).
As an interim fix, the reporter recommends replacing the LOINC codes with
temporary (non-LOINC) codes minted by the IG, with the longer-term plan being
to request properly-scoped LOINC codes from Regenstrief that can be used for
this purpose. No cross-referenced Jira tickets were surfaced for this issue,
so the request stands on its own; it is essentially a terminology-correctness
fix for the PA IG rather than a structural change to its profiles.

## Related Zulip Discussions

### [Physical Activity > Light Intensity](https://chat.fhir.org/#narrow/stream/Physical%20Activity/topic/Light%20Intensity)

Thread (May–Jul 2023) where Adam Burch asked whether the PA IG should add a
code for "light intensity" alongside moderate and vigorous. Paul Chase
explained that the IG is intentionally scoped to the 2018 Physical Activity
Guidelines (150 min/week moderate or 75 min/week vigorous) and that minutes
of moderate and vigorous activity were chosen specifically to inform that
core measure; light-intensity activity is captured only via supporting
per-activity measures. Lloyd McKenzie also clarified that the published
"minutes per week" measure does not differentiate between moderate and
vigorous, and that intensity differentiation is handled through the
detailed per-activity supporting measures. The thread does not directly
discuss the LOINC-licensing concern raised in FHIR-47330, but it
establishes that the IG is firmly committed to the moderate/vigorous
minutes concept — i.e., the codes themselves need to keep working, only
the underlying terminology needs to change.

Participants: Adam Burch, Lloyd McKenzie, Paul Chase, Michael E Campbell.

### [terminology > Placeholder Systems](https://chat.fhir.org/#narrow/stream/terminology/topic/Placeholder%20Systems)

Recent (Apr 2026) terminology-stream discussion started by Gino Canessa
about whether projects should mint "placeholder" code systems when no real
authoritative system exists. Lloyd McKenzie — the reporter of FHIR-47330 —
took a strong line that "placeholder systems aren't a good idea" and that
when no real system exists it is better to omit the system entirely (or
push the placeholder into an extension), accepting the resulting semantic
ambiguity. This is directly relevant to FHIR-47330's interim recommendation
to "replace them with temporary codes": if the IG mints a temporary
HL7/PA-specific `CodeSystem`, the workgroup should weigh Lloyd's more
recent guidance on placeholder systems before doing so, and consider
whether a published, IG-scoped CodeSystem (rather than a true placeholder)
is the right vehicle.

Participants: Gino Canessa, Lloyd McKenzie (and other terminology-stream
contributors).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's recommendation literally: in the US Physical Activity
IG, replace the current LOINC codes used as `Observation.code` for
"minutes of moderate physical activity" and "minutes of vigorous physical
activity" with newly-minted IG-local placeholder codes. Concretely:

- Mint a temporary `CodeSystem` under the IG's canonical (e.g.,
  `http://hl7.org/fhir/us/physical-activity/CodeSystem/temp-mvpa-codes`)
  containing two codes such as `moderate-pa-minutes` and
  `vigorous-pa-minutes`, each with a clear definition that scopes them to
  observed/reported minutes of activity at that intensity over a stated
  period.
- Update the affected Observation profiles, search-parameter
  examples, IG examples, and any CapabilityStatement / measure references
  to use the new codes instead of the offending LOINC codes.
- Add an IG-level note explaining that these are temporary identifiers
  pending a Regenstrief request for properly-scoped LOINC codes, and
  publish a `ConceptMap` from the temporary codes to the eventual LOINC
  codes once they are issued.
- File the LOINC submission with Regenstrief in parallel and track it
  via a follow-up Jira so the temporary codes can be retired in a future
  IG version.

#### Justification

This is exactly what the reporter asked for and is the cleanest way to
remove a known terminology-licensing problem quickly. It keeps the IG
publishable while the slower LOINC submission/approval cycle runs, and
the use of an IG-scoped `CodeSystem` plus a future `ConceptMap` gives
implementers a clear migration path. It also aligns with FHIR's general
guidance to use a real (even if interim) `CodeSystem` rather than leaving
`Observation.code.coding.system` empty, which would harm interoperability
for the very implementers (e.g., the UK supplier in the SNOMED-conversion
Zulip thread) who are already binding to these concepts.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than mint placeholder codes that will later be replaced, take a
two-step terminology approach that minimises churn for implementers:

1. **Submit the new LOINC code request to Regenstrief immediately**
   (moderate-PA minutes/period and vigorous-PA minutes/period, scoped for
   use as `Observation.code`), and publish the request status in the IG's
   terminology page.
2. **In the IG itself, do not introduce IG-local placeholder codes.**
   Instead, redefine the relevant Observation profiles so that
   `Observation.code` is bound (`extensible`) to a value set that
   currently contains only the existing LOINC codes flagged with a
   prominent normative warning ("known terminology issue — see
   FHIR-47330; will be replaced with new LOINC codes once issued"), and
   allow implementers to additionally use a SNOMED CT or local code via
   `extensible` binding while the LOINC issue is resolved. Optionally,
   add an `Observation.method` or similar slot that records the original
   intent, so the underlying clinical concept is preserved even if the
   code itself is contested.

#### Justification

This avoids the cost of two terminology migrations (LOINC → temporary →
new LOINC) and respects Lloyd McKenzie's more recent guidance in the
terminology > Placeholder Systems thread that placeholder/temporary code
systems generally cause more harm than good. It also aligns with the
international reality surfaced in the UK SNOMED-conversion Zulip thread:
implementers are already mapping to non-LOINC equivalents, so an
extensible binding plus a documented LOINC request is closer to how the
ecosystem is actually consuming the IG. The trade-off is that the IG
ships with a known terminology defect for longer; that needs to be
addressed by clear, prominent IG-level documentation and by tracking the
LOINC request as a release-blocking item for the next version.

---

### Disposition C: Decline

#### Proposal

Do not change the codes in the IG at this time. Keep the existing LOINC
codes in `Observation.code`, and instead document — in the IG's
terminology section and in a release note — the licensing/use concern
raised in FHIR-47330, with a pointer to a forthcoming LOINC submission.
Defer any code substitution until Regenstrief has either (a) issued new
codes that can be used as `Observation.code`, or (b) confirmed that the
current usage is in fact disallowed and will not be sanctioned.

#### Justification

The ticket description states only that the IG is "apparently" not
allowed to use the codes this way. Before forcing every existing
implementer (including international consumers like the UK supplier in
the SNOMED-conversion thread) onto a new, temporary, IG-local code system
— which is itself contrary to Lloyd's recent terminology guidance against
placeholder systems — the workgroup should obtain a definitive ruling
from Regenstrief on the actual licensing/usage restriction. If the
restriction is confirmed, Disposition A or B can be executed quickly; if
it is not, the IG avoids an avoidable, disruptive change. The
conservatism cost is reputational only and is mitigated by an explicit
IG-level disclosure of the open issue.

---

### Recommendation

**Recommended disposition:** B

Disposition B best balances the legitimate concern raised in the ticket
with the costs of code churn for an IG that is already in active use
internationally. It addresses the licensing problem head-on (by filing
the LOINC request) while avoiding the two-migration trap of Disposition A
and the inaction risk of Disposition C. It also lines up with the
reporter's own more recent guidance in the terminology > Placeholder
Systems thread that placeholder code systems should generally be
avoided, which makes Disposition A's IG-local temporary `CodeSystem`
harder to defend than it looked when this ticket was filed in 2024. Patient
Care should ratify the LOINC submission as the canonical fix, agree to
ship the next PA IG release with an extensible binding plus a prominent
"known terminology issue" notice, and treat retirement of that notice
(once the new LOINC codes are issued) as a release-blocking item for the
following version.
