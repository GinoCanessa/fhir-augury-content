## Summary

The SDOHCC (US SDOH Clinical Care) IG currently requires a `referencePolicy`
of `literal` on both the client- and server-side CapabilityStatements. The
reporter observes that the required value set bound to
`CapabilityStatement.rest.resource.referencePolicy` in R4 describes
expectations that are inherently server-side (e.g., how a server resolves,
enforces, or manages references). Because a SDOHCC client is acting as a
requestor, it is unclear what a client-side reference-handling expectation
would even mean. The reporter asks the workgroup to either articulate
meaningful client-side semantics for `referencePolicy` or — if no such
semantics can be defined — to stop asserting `referencePolicy` on the client
CapabilityStatements altogether. No cross-referenced Jira tickets, Zulip
threads, or GitHub items were found for this issue.

## Details

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Remove the `referencePolicy` constraint from every client-side
CapabilityStatement published by the SDOHCC IG (e.g., the SDOHCC client
CapabilityStatement resource(s) that currently assert
`rest.resource.referencePolicy = literal`). Retain the `literal`
`referencePolicy` on the server-side CapabilityStatements, since that is
where the R4 value-set semantics actually apply. Update the associated
narrative/conformance prose in the IG so that the "reference handling"
expectation is described only as a server obligation, and any client
obligations (e.g., "clients SHALL send literal references") are expressed
as plain conformance text rather than via `referencePolicy`.

#### Justification

This matches the ticket's literal request and the stated resolution
("Persuasive with Modification — we will update all of the
CapabilityStatements of the Client side requirements to remove the
Reference Policy requirement"). It also aligns with the R4 definition of
`CapabilityStatement.rest.resource.referencePolicy`, whose bound value set
(`reference-handling-policy`) describes behaviors only a server can
exhibit — resolution, enforcement, local/foreign references, deletion
propagation. Asserting those codes on a client CapabilityStatement is
semantically empty and risks being interpreted inconsistently by
implementers and validators. Removing them sharpens the IG's conformance
story without losing any real requirement.

---

### Disposition B: Alternative Approach

#### Proposal

Keep a client-side "reference handling" expectation, but express it
through a mechanism appropriate to clients rather than through
`referencePolicy`. Concretely:

1. Remove `referencePolicy` from the client CapabilityStatement(s) (same
   as Disposition A).
2. Add narrative conformance text in the SDOHCC IG stating the client's
   reference-shape obligations — e.g., "SDOHCC clients SHALL send
   references as literal, absolute or relative URLs and SHALL NOT use
   contained or logical references when referring to SDOHCC-profiled
   resources."
3. Optionally add an invariant or a profile constraint on the relevant
   request/response bundles or on the `Reference` datatype uses in
   SDOHCC profiles, so that the literal-reference expectation is machine
   checkable at the instance level rather than at the CapabilityStatement
   level.

#### Justification

The ticket's underlying concern is not only "client `referencePolicy`
is confusing" but also "if there are reference-handling expectations for
a client, what are they?" Disposition A answers the first half by
deleting the element; Disposition B additionally answers the second half
by relocating the expectation to a place where it is well-defined
(narrative + profile constraints on `Reference.reference`). This avoids
silently losing the intent the original authors had when they added
`referencePolicy=literal` to the client side. Trade-off vs A: slightly
more authoring work and a small risk of scope creep into profiles, but a
stronger, testable conformance story for implementers.

---

### Disposition C: Decline

#### Proposal

Leave the client CapabilityStatements unchanged, retaining
`referencePolicy = literal` on both client and server sides. Rationale:
"already addressed by existing mechanism — `literal` is a reasonable
client-side expectation even if the value set's other codes are
server-centric."

#### Justification

A defensible argument is that `literal` specifically — as opposed to
`logical`, `resolves`, `enforced`, or `local` — *does* carry a meaning
that a client can honor: "when I send a `Reference`, I use a URL rather
than a logical identifier." Under that reading, the current IG is not
incorrect, just awkward, and changing it risks churn in downstream
implementers' conformance tooling. Declining preserves backward
compatibility with existing SDOHCC client implementations and avoids
reopening editorial discussion about which reference-handling guarantees
clients can meaningfully make.

---

### Recommendation

**Recommended disposition:** B

Disposition B matches the resolution already recorded on the ticket
("Persuasive with Modification" — remove `referencePolicy` from client
CapabilityStatements) while also addressing the reporter's secondary
question: *if* there are client-side reference-handling expectations,
what are they? Simply deleting the element (Disposition A) is correct
but leaves the IG silent on a requirement the original authors clearly
intended to express. Re-expressing that requirement as narrative
conformance text — and optionally as a profile-level constraint on
`Reference.reference` — keeps the real-world obligation (literal URL
references) visible to implementers without abusing a CapabilityStatement
element whose value set is server-scoped. Disposition C is weakest
because it ignores both the R4 definition of the bound value set and the
workgroup's already-recorded "Persuasive with Modification" resolution.
