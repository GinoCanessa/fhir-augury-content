## Summary

The ticket asks Patient Care to extend the **Procedure** resource so that
implementers can record the precise sequence of timestamps that occur
during a surgical operation (e.g., patient enters the OR area, anaesthesia
start/stop, incision/cut, suture, patient leaves the OR). The submitter
(via RenÃ© StÃ¶rmer in the originating Zulip thread) reports that German
jurisdictions and other European systems legally require capturing these
"perioperative process times" on a single surgical case, and that today the
core Procedure resource has no first-class slot for them â€” the spec
explicitly notes that "the creation of a resource to support detailed
real-time procedure information awaits the identification of a specific
implementation use-case."

Repeated workgroup discussions (Patient Care 2020-11-05, 2021-04-22, joint
OO/PC 2025-09-17) have framed the request as one of three modelling
options: (1) record each timepoint as an `Observation` whose
`Observation.partOf` or `Observation.subject` references the parent
`Procedure`; (2) add a list of typed `Period`s (or timestamps) directly to
`Procedure` â€” either as a new core element or as a core extension; or
(3) use **sub-Procedures** linked via `Procedure.partOf`, exactly as
already supported by the core resource. The 2025-09-17 minutes leave the
choice open pending more use cases. A closely related ticket,
[FHIR-48449](https://jira.hl7.org/browse/FHIR-48449) ("Need guidance and/or
mechanisms to record a procedure log"), generalises the same problem to
anaesthesia and cath-lab procedure logs â€” that ticket is not formally
linked but is referenced in the comment thread and should be considered
together with this one.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new repeating backbone element to the core **Procedure** resource â€”
e.g. `Procedure.timePoint` (0..*) â€” with the following structure:

- `Procedure.timePoint.code` : `CodeableConcept` (1..1) â€” what the
  timepoint represents (anaesthesia start, incision, suture, patient
  leaves OR, room clean stop, â€¦); bound `extensible` to a new value set
  (e.g., `procedure-time-point-type`) seeded with the BDA/DGAI/BDC/DGCH
  VOPM perioperative codes plus common SNOMED CT equivalents where they
  exist.
- `Procedure.timePoint.time[x]` : `dateTime | Period` (1..1) â€” the
  timepoint or interval.
- `Procedure.timePoint.performer` : `Reference(Practitioner |
  PractitionerRole | Device)` (0..*) â€” optional, for who recorded /
  performed the step.
- `Procedure.timePoint.note` : `Annotation` (0..*).

Add a short narrative section to the Procedure page explaining when to
use `timePoint` (single overarching `Procedure.code`, sub-events that do
not warrant their own Procedure) versus `Procedure.partOf` (sub-events
that themselves have a billable / clinically distinct
`Procedure.code`).

#### Justification

This is what the submitter and the originating implementer (RenÃ©
StÃ¶rmer) explicitly asked for and it matches what European hospital
information systems already do natively. Patient Care has heard the same
request from at least two angles (FHIR-28358 surgical timestamps,
FHIR-48449 procedure log) and the joint OO/PC 2025-09-17 minutes list
"list of Periods to capture this information" as one of the three
options. A first-class element makes the data queryable without
extension overhead, is easier for analytics / quality reporting (waiting
times, OR turnover, anaesthesia duration), and aligns with the existing
pattern of repeating coded sub-elements on other clinical resources
(e.g. `MedicationAdministration.dosage`, `Observation.component`).

### Disposition B: Alternative Approach

#### Proposal

Do not change the core `Procedure` element list. Instead, publish a
**standard core extension** `procedure-timePoint` (URL
`http://hl7.org/fhir/StructureDefinition/procedure-timePoint`) with the
same structure as Disposition A
(`code`, `time[x]: dateTime|Period`, `performer`, `note`), and:

1. Define and publish the value set
   `procedure-time-point-type` covering the standard perioperative time
   points (entry/exit, anaesthesia start/stop, incision/suture, room
   clean start/stop, etc.), based on the VOPM glossary cited in the
   ticket.
2. Update the **Procedure** resource narrative to call out three
   modelling options explicitly â€” (a) sub-Procedures via
   `Procedure.partOf` for distinct billable / clinical sub-procedures,
   (b) the new `procedure-timePoint` extension for fine-grained
   perioperative timestamps on a single Procedure, (c) `Observation`
   referencing the Procedure via `Observation.partOf` /
   `Observation.subject` when the timepoint also carries a measured
   value â€” and indicate when each is preferred.
3. Coordinate with FHIR-48449 so that the procedure-log discussion uses
   the same extension (or an extension of it) rather than inventing a
   parallel mechanism.

#### Justification

This addresses the underlying need without committing the core
`Procedure` resource to a structure (typed timepoint list) that several
voices in the discussion â€” most consistently Jose Costa Teixeira, and
implicitly Lloyd McKenzie â€” believe will keep growing toward what
sub-Procedures already deliver. A standard core extension is normatively
equivalent for implementers who need it (the data is still discoverable
and queryable through the published StructureDefinition) but keeps the
resource's element list stable, avoids breaking changes, and lets the
mechanism prove itself before being promoted into core. It also gives
the workgroup a concrete artifact (the value set) that addresses the
recurring "what timestamps are we even talking about?" question that has
stalled this ticket for five years. The reporter's main objection to
extensions ("bad readability and performance when querying") is a
database-design concern in one specific stack, not a FHIR conformance
concern â€” implementers who care can index extension URLs, and the value
set gives them stable codes to index against.

### Disposition C: Decline

#### Proposal

Decline the request to add a timestamp list to `Procedure`, with the
rationale "already addressed by existing mechanisms." Update the
Procedure resource narrative (and remove the existing "awaits the
identification of a specific implementation use-case" sentence) to state
explicitly that real-time / perioperative event capture is supported
today by:

- using sub-Procedures linked via `Procedure.partOf` (with their own
  `Procedure.code` and `Procedure.occurrencePeriod`) â€” including the
  contained-resource pattern shown in Jose Costa Teixeira's 2020-08-27
  comment for cases where a separate persistent identity is not needed;
- and/or the existing core extensions (`procedure-progressStatus`,
  `procedure-incisionDateTime`).

Close FHIR-28358 as "Persuasive with Mod" pointing at that updated
guidance. Direct any remaining gap (a stable cross-jurisdiction value
set of perioperative time-point codes) to a separate ticket or to the
relevant terminology workgroup, not to a structural change of Procedure.

#### Justification

The community discussion shows the request can be met with what FHIR
already publishes; the substantive disagreement is about ergonomics, not
capability. Patient Care has gone five years without converging on a
structural change, which is itself evidence that no use case has been
strong enough to justify modifying a heavily-used core resource. A core
change here also has consequences for FHIR-48449 (procedure logs) and
for any specification (anaesthesia, cath-lab, surgical) that already
profiles Procedure on top of the current structure. Declining-with-
guidance preserves backward compatibility, gives implementers a clear
answer to the "how do I do this today?" question, and keeps the door
open to revisit if a new use case appears that genuinely cannot be met
by sub-Procedures plus extensions.

### Recommendation

**Recommended disposition:** B

Disposition B is the recommended starting point because it concretely
answers the request â€” implementers get a published, queryable mechanism
*and* a value set of perioperative time points, both of which are
missing today â€” while honouring the strongest counter-arguments from the
Zulip and Patient Care call discussions: that a flat timestamp list on
core `Procedure` is likely to attract more elements over time
(performer, location, status) and re-create the sub-Procedure structure;
and that breaking changes to `Procedure` ripple into every IG that
profiles it. A standard core extension keeps `Procedure` stable, lines
up with the pattern Jose Costa Teixeira and costateixeira have argued
for since 2020, and is the easiest path to also resolve FHIR-48449
coherently. Disposition A (full core element) should be the fallback if
Patient Care, after seeing the use cases Grant and Marti committed to in
the 2025-09-17 minutes, concludes that the perioperative timepoint
pattern is sufficiently universal to belong in core. Disposition C is
the least useful outcome â€” it would close the ticket but leave the
recurring implementer question (what extension? what codes?)
unanswered, exactly the gap the Patient Care WG > Theatre Tracking
thread complained about in 2021.
