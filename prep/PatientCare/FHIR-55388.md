## Summary

The reporter (Igor Bossenko) is asking Patient Care to change `bodyStructure`
to a `CodeableReference` "as in the Observation" and to mark `bodySite` as
deprecated. The intent is to consolidate the historical pattern of carrying a
plain `CodeableConcept` body site alongside a `Reference(BodyStructure)` into a
single `CodeableReference(BodyStructure)` element, matching what `Observation`
(and other recently aligned resources such as `DocumentReference`) already do
in R6.

A reviewer comment on the ticket (Michelle Miller) flags that
[FHIR-50481](https://jira.hl7.org/browse/FHIR-50481) — "Align BodySite
cardinality of ConditionDefinition with Condition Resource" — is the broader
sibling discussion: it covers the same `bodySite`/`bodyStructure` modeling
question in `Condition` and `ConditionDefinition`, and was discussed at the
2026-01 WGM joint OO/PC session. The OO/PC discussion produced three options
(Option 1: `CodeableReference 0..1`; Option 2: `CodeableReference 0..*`; Option
3: keep two elements with an invariant), with an action item to "capture a
ticket for changes to OO resources in alignment with option 3 and combined
search parameters." FHIR-55388 is, in effect, the proposed Option 1 / Option 2
end-state for the resources in scope (which the reporter does not enumerate,
but which by implication includes Procedure, ServiceRequest, Condition, and
ConditionDefinition, all of which still carry both `bodySite` and
`bodyStructure` in R6).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In FHIR Core (FHIR R6), for every resource that currently carries both a
`bodySite` (`CodeableConcept`, typically 0..*) element and a separate
`bodyStructure` (`Reference(BodyStructure)`, typically 0..1) element — at
minimum `Procedure`, `ServiceRequest`, `Condition`, and `ConditionDefinition`
— make the following changes:

1. Add/keep a single element named `bodyStructure` of type
   `CodeableReference(BodyStructure)` with cardinality 0..1, mirroring
   `Observation.bodyStructure`, `ObservationDefinition.bodyStructure`,
   `DocumentReference.bodyStructure`, and `ServiceRequest.bodyStructure`.
2. Mark the existing `bodySite` (`CodeableConcept` 0..*) element as
   **deprecated** (do not remove in R6; keep with a `standards-status =
   deprecated` extension and `Element.modifierExtension`-free guidance to
   prefer `bodyStructure.concept`).
3. Update the bound value sets and search parameters so that the existing
   `body-site` SearchParameter expressions also resolve through
   `bodyStructure.concept` (combined search parameters per the OO/PC
   2026-01 WGM action item).
4. Add a maturity-appropriate change note and a migration mapping from R5
   `bodySite`/`bodyStructure` to the consolidated R6 `bodyStructure`
   `CodeableReference`.

#### Justification

This is the pattern the rest of FHIR Core has already converged on
(`Observation`, `ObservationDefinition`, `DocumentReference`,
`ServiceRequest.bodyStructure`, plus `ImagingSelection.bodySite`,
`ImagingStudy.series.bodySite`, etc., per the FMG `CodeableReference usage in
Core` Zulip listing). Consolidating to a single `CodeableReference` element
removes the "two elements that mean almost the same thing" footgun that Brian
Postlethwaite raised in the OO `R6 BodyStructure Review` thread, eliminates
the need for cross-element invariants ("don't populate both"), and matches
`CodeableReference`'s explicit purpose of carrying either a code or a
reference (or both) for the same concept. Marking `bodySite` deprecated
rather than deleting it preserves R5→R6 round-tripping for implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the WGM Option 3 pattern instead of a type change: keep the two
parallel elements but add a formal invariant.

1. On each affected resource, retain `bodySite` `CodeableConcept` (cardinality
   restored to 0..*) and `bodyStructure` `Reference(BodyStructure)`
   (cardinality 0..1, as Procedure already has per FHIR-51792).
2. Add an invariant per resource: "If `bodyStructure` is present, `bodySite`
   SHALL NOT be present" (or equivalent: `bodyStructure.exists()
   implies bodySite.empty()`), so a single instance never carries both.
3. Define a combined `body-site` SearchParameter that walks both
   `bodySite` and `bodyStructure.resolve().includedStructure.structure`.
4. Do **not** deprecate `bodySite`; document the two-element pattern as the
   PC/OCC-endorsed shape for resources where 0..* `CodeableConcept`
   semantics are required.

#### Justification

This is the option (Option 3) that the joint PC/OCC WGM session already
captured an action item to pursue. It directly addresses Michelle Miller's
concern (in the OO Zulip thread and on FHIR-50481) that
`Procedure`/`ServiceRequest` deliberately separated the elements because
`bodySite` needs 0..* (peri-umbilical + LUQ for abdominal pain; metastatic
sites for cancer) while `bodyStructure` should remain 0..1 (one referenced
`BodyStructure` resource that internally carries multiple
`includedStructure`). It also addresses JD Lyle's argument that overloading
`CodeableReference` with array semantics introduces a precedent of
case-specific design and would complicate the standard for marginal benefit.
The trade-off is the invariant complexity Rob Hausam called out as
"messy/complicated," and the inconsistency with the resources that have
already moved to single-element `CodeableReference` (Observation,
DocumentReference, etc.).

---

### Disposition C: Decline

#### Proposal

Decline FHIR-55388 as written. Do not change the type of `bodyStructure`
to `CodeableReference` and do not deprecate `bodySite` on
Procedure / ServiceRequest / Condition / ConditionDefinition. Close as
"Not Persuasive" and direct the reporter to FHIR-50481 (and the OO/PC WGM
action item) as the canonical thread for any cross-resource alignment.

#### Justification

The current split is the result of explicit, documented PC/OCC decisions
(FHIR-51792 reverting FHIR-41690 to keep Procedure.bodyStructure at 0..1
alongside `bodySite` 0..*). The reporter's request is a one-line proposal
without a use-case analysis or a list of affected resources, and Michelle
Miller's own comment notes that scope was not assessed. Per Amit's WGM
remark, "consistency is not always a persuasive reason for change" once
implementers have shipped against the existing shape — and these resources
(unlike ConditionDefinition, FMM 0) are widely implemented. Declining
preserves R5→R6 backwards behavior, avoids a breaking type change on
heavily-used resources, and lets the focused cross-resource analysis under
FHIR-50481 (and any follow-up captured under the OO action item) drive a
single coherent decision rather than handling each resource piecemeal.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying point is real: the spec currently presents a
mixed pattern (single-element `CodeableReference` on Observation /
DocumentReference / ObservationDefinition / ServiceRequest.bodyStructure
vs. parallel `bodySite` + `bodyStructure` on Procedure / Condition /
ConditionDefinition), and the OO `R6 BodyStructure Review` thread
already anticipates ballot comments on this. However, Disposition A (full
type change to a single `CodeableReference 0..1`) loses the 0..*
`CodeableConcept` semantics that Michelle Miller, Rob Hausam, and the
FHIR-51792 reversion explicitly preserved for Procedure/ServiceRequest,
and JD Lyle's argument against widening `CodeableReference` cardinality
(or against re-engineering the type to fit boundary cases) is well-made.
Disposition C declines an issue the WG has already agreed needs to be
addressed.

Disposition B implements the WGM-endorsed Option 3, keeps the agreed
0..* `CodeableConcept` semantics where they are clinically needed, adds
an invariant to remove the "both elements populated" ambiguity that
prompted FHIR-55388 in the first place, and aligns with the OO/PC action
item to introduce combined search parameters. It also leaves a clean
follow-up path: if implementer feedback during the next ballot cycle
shows that the Observation-style single `CodeableReference` is in fact
sufficient for Procedure/Condition use cases, the WG can collapse to
Disposition A in a later cycle without first having to undo a
deprecation. Note that this skill did not load the `HL7/fhir` repo
briefing (no GitHub cross-references were surfaced for FHIR-55388); the
implementing team should confirm the exact authoring paths and search
parameter definitions against the current `HL7/fhir` source before
acting on the recommendation.
