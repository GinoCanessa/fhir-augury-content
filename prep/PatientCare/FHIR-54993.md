## Summary

Ron G. Parker is asking that the codes `revoked` and `completed` be retained
in the relevant request-status code system / value set rather than removed as
proposed by an earlier ballot comment. He cites real-world use in the Alberta
eReferral implementation, where these codes are used to convey *why* a request
was ended. The ticket is recorded as "Comment 19" — a ballot comment imported
into Jira — and lands in Patient Care because the affected workflow concerns
ServiceRequest. A reviewer comment from `jdlnolen` notes that the underlying
value set is owned by FHIR Infrastructure (FHIR-I), so any disposition will
need cross-WG agreement. No cross-referenced Jira tickets, Zulip threads, or
GitHub items are recorded for this ticket in the Augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Retain the codes `revoked` and `completed` in the request-status code system
and the bound value set used by `ServiceRequest.status` (and any other
request-pattern resources that share the value set, e.g. `MedicationRequest`,
`CommunicationRequest`, `Task` where applicable). Concretely: do **not** apply
the previously proposed deletions; leave the existing concept definitions and
display strings unchanged, and update the value-set version notes to record
that the proposed removals were withdrawn. Because `jdlnolen` notes the
value set is owned by FHIR-I, file a coordination note from Patient Care to
FHIR-I and obtain their explicit concurrence before the change is closed.

#### Justification

The codes are demonstrably in production use (Alberta eReferral) and
represent the natural terminal states of a request workflow — `completed`
(the request was fulfilled) and `revoked` (the request was withdrawn after
acceptance). Removing in-use, semantically distinct codes from a published
status value set is a breaking change for implementers, with no compelling
upside identified in the ticket. Retention is the safer default under FHIR's
backward-compatibility policy for normative status code systems.

---

### Disposition B: Alternative Approach

#### Proposal

Retain `revoked` and `completed` in the value set, but tighten their
definitions and add usage guidance that distinguishes them from neighbouring
codes (e.g., `entered-in-error`, `ended` if applicable, `cancelled`). Where
the original removal proposal cited overlap or ambiguity, address that by
updating the concept `definition` and adding a short comment in the
ServiceRequest narrative on the request-lifecycle page rather than by
deleting the codes. If `revoked` was felt to overlap with another code, add
a SHOULD-level statement clarifying when each code applies. Coordinate the
wording with FHIR-I as the value-set owner.

#### Justification

This preserves implementer investment (matching Disposition A) while
acknowledging that the original removal proposal likely had a real concern —
ambiguity or overlap — that should not simply be ignored. Sharpening
definitions resolves the underlying motivation for removal without breaking
existing instances. It also creates an artefact (clarified definitions +
usage notes) that is durable across ballots, whereas a simple "retain"
disposition leaves the same removal proposal liable to resurface.

---

### Disposition C: Decline

#### Proposal

Decline the request and proceed with the original proposal to remove
`revoked` and `completed` from the request-status value set. Recommend that
Alberta eReferral migrate to the surviving codes (or to a local extension
code system) for the use cases that previously relied on the removed values.

#### Justification

This disposition is only defensible if FHIR-I, as the value-set owner, has
already concluded that the codes are redundant or harmful and that removal
is required for the spec's coherence. Absent such a determination — and
none is cited in the ticket — declining penalises a real implementer for a
change with no documented benefit, which is hard to justify against FHIR's
stability commitments for status terminology.

---

### Recommendation

**Recommended disposition:** B

Disposition B keeps the in-use codes (which is what the reporter and the
Alberta eReferral implementation actually need) while not silently shelving
the concern that motivated the original removal proposal. The single comment
on this ticket already flags a coordination dependency — the value set is
owned by FHIR-I — so the workgroup is going to have to talk to FHIR-I either
way; folding a definition-clarification into that conversation is far cheaper
than re-litigating removal in a later cycle. Disposition A is acceptable as
a fallback if FHIR-I confirms the codes are unambiguous as written.
Disposition C should not be adopted without a documented FHIR-I rationale
for removal, which is not present in the ticket or its (empty) cross-reference
graph.
