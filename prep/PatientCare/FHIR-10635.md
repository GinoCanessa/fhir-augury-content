## Summary

The ticket flags a QA 5a violation in FHIR Core: `Condition` and
`ClinicalImpression` reference each other, creating a circular relationship.
Specifically, `Condition.stage.assessment` points to `ClinicalImpression`,
while `ClinicalImpression.problem` and `ClinicalImpression.finding.item[x]`
point back to `Condition`. Per the DSTU QA 5a guideline, a given relationship
should be represented in only one resource (preferably the one created
second), and reciprocal references like this need to be reconciled. The
ticket has a long disposition history: it was initially deferred (Jan 2017,
7-0-1) on the grounds that `ClinicalImpression` was too immature
(FMM=0) to redesign, an explanatory note was added to the spec, and then in
July 2018 it was voted Not Persuasive (8-0-0). Despite the prior outcomes,
it currently sits in `Triaged` / `Unresolved` for Patient Care to revisit.
There are no explicitly linked sibling Jira tickets in the cross-reference
data; the only substantive related discussion is on Zulip
(`#implementers > Using ClinicalImpression for a cancer therapy decisions`),
where Michelle Miller cites this very ticket while telling implementers that
the circular reference is a known post-STU3 issue.

## Details

> ClinicalImpression as follows:
>
> - `Condition.stage.assessment` (to `ClinicalImpression`)
> - `ClinicalImpression.problem` & `ClinicalImpression.finding.item[x]`
>   (to `Condition`)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Resolve the circular reference in FHIR Core by removing one direction of
the linkage. The most defensible cut is to **remove
`Condition.stage.assessment`** (typed `Reference(ClinicalImpression)`)
and rely on `ClinicalImpression.problem` plus
`ClinicalImpression.finding.item[x]` (which itself points back to
`Condition`/`Observation`) as the single canonical direction. Rationale
for the direction: per QA 5a's "second-created resource owns the
reference" guidance, the clinical assessment is typically authored
*after* the underlying problem and its stage are documented, so the
reference should live on `ClinicalImpression`. The change set would be:

- Remove the `assessment` element from the `Condition.stage` BackboneElement.
- Update `Condition` examples / search params / mappings that mention
  `Condition.stage.assessment`.
- Add a migration note in `Condition` describing how to express the link
  going forward (use `ClinicalImpression.problem` /
  `ClinicalImpression.finding`, with `ClinicalImpression.previous` to chain
  staging assessments over time).
- Replace the existing "known issue" note in both `Condition` and
  `ClinicalImpression` with the new guidance.

#### Justification

- Directly satisfies the QA 5a guideline that motivated the ticket and
  removes a long-standing, explicitly-acknowledged spec defect.
- Keeps the more expressive side of the relationship
  (`ClinicalImpression.finding` already supports referencing both
  `Condition` and `Observation` with a uniform pattern), so no expressive
  power is lost in practice.
- The `Condition.stage.assessment` element is a niche, narrowly-scoped
  back-pointer; removing it has lower implementer impact than removing
  the `ClinicalImpression.problem`/`finding` axis, which is core to
  modelling an assessment.
- A breaking change on `Condition.stage.assessment` is justifiable now
  that `ClinicalImpression` has matured beyond FMM=0 (the explicit
  rationale Patient Care used to defer in 2017 no longer holds), and
  removing a Reference element is a documented R6 / next-major
  refactoring opportunity rather than an STU patch.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both reference axes but **formally exempt this case from QA 5a** by
recasting the two references as semantically distinct (not redundant), and
publish that distinction normatively:

- Retain `Condition.stage.assessment` as the *evidence-of-stage* link
  (the assessments that established this Condition's stage) — narrow
  semantics, scoped to staging only, not "this Condition was discussed
  in this assessment".
- Retain `ClinicalImpression.problem` as the *subject-of-assessment*
  link, and `ClinicalImpression.finding.item[x]` as the *outcomes-of-
  assessment* link.
- Add invariant text to both resources clarifying that the two
  directions are *not* mirror images and that systems MAY populate either
  or both based on local workflow; promote the existing "known issue"
  note to a "Boundaries and Relationships" section that explains the
  distinction.
- Request an explicit MnM exemption against QA 5a for this pair (as
  Michelle proposed in 2017) and record the exemption alongside the
  resource intro so future QA reviews don't re-raise it.

#### Justification

- Preserves backward compatibility for any implementer using
  `Condition.stage.assessment` (the Zulip thread shows there are
  implementers who want both directions for legitimate workflows).
- Acknowledges the substantive point in McKenzie's 2016 comment: the two
  references are not actually pointing at the same relationship —
  staging-evidence is narrower than assessment-subject — so the QA
  violation is more apparent than real.
- Avoids a breaking change in a stable resource (`Condition` is a
  high-maturity, widely implemented resource), satisfying the principle
  that breaking changes to FMM-N resources require strong justification.
- Trades one-time tooling cost (writing the exemption + clearer
  documentation) for ongoing implementer flexibility, which is a good
  trade for a well-known and now well-understood modelling tension.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive — already addressed**. The ticket has already
been adjudicated twice by Patient Care:

- 2017-01-31 (Rob / Elaine): Deferred 7-0-1, with a note added to both
  resources documenting the circular reference as a known issue.
- 2018-07-05 (George / Emma): Voted Not Persuasive 8-0-0.

The original note ("A known issue exists with circular references
between Condition and ClinicalImpression…") is in place and the workgroup
has twice declined to make a normative change. No new use-case evidence,
implementer pain reports, or design proposals have been added since
2019. Re-affirm Not Persuasive and resolve the ticket.

#### Justification

- Two prior workgroup votes (one unanimous against persuasion) already
  represent the WG's considered position; absent new information,
  re-litigating the same issue costs cycles for no incremental benefit.
- The workaround (the explanatory note in both resources) is in the
  published spec, so implementers are not navigating a hidden defect —
  the QA 5a pattern is documented as an intentional, acknowledged
  exception.
- Removing or reshaping `Condition.stage.assessment` would be a breaking
  change to a high-maturity, widely-implemented resource; the historical
  bar for that has not been met by this ticket's evidence.
- The substantive Zulip discussion ended inconclusively for the
  implementer's use case — it does not constitute new normative pressure
  to change the model.

---

### Recommendation

**Recommended disposition:** B

`ClinicalImpression` has matured well past the FMM=0 state that drove the
original deferral, so the "we can't decide yet" rationale that anchors
Disposition C no longer applies — but the Zulip discussion shows that
real implementers genuinely want to navigate both directions of the
relationship, which makes the surgical removal in Disposition A more
disruptive than its strict QA-compliance benefit warrants. Disposition B
threads the needle: it accepts McKenzie's 2016 insight that the two
references are semantically distinct (staging-evidence vs. assessment-
subject), formalises that distinction in the spec, and clears the QA 5a
flag via an explicit, documented MnM exemption rather than by amputating
a useful element. It costs Patient Care one round of doc work and one
formal exemption request, and in return permanently retires the issue
without a breaking change. If Patient Care decides the modelling cost of
keeping both axes outweighs implementer demand, Disposition A is the
correct fallback; Disposition C should only be selected if the WG is
committed to leaving the QA debt in place indefinitely.
