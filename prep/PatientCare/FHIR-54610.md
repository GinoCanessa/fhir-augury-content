## Summary

Tom Page is asking the SDOHCC IG to amend Section 9.2.2 ("Subscription") so
that the narrative explicitly requires the server holding the Task data to
validate patient consent to share information with the subscribing client
before pushing notifications. The reporter supplied concrete suggested
language that inserts the phrase "After the server validates patient consent
to share data with the subscribing entity" into the existing description of
the Subscription-based notification flow. This is the Subscription-side
counterpart to [FHIR-54609](https://jira.hl7.org/browse/FHIR-54609), which
asks for the same consent-validation language to be added to Section 9.2.1
(Polling). Both tickets share the same reporter, work group, ballot label,
and resolution disposition (Persuasive with Modification), and Patient Care
/ Gravity have already replied that SDOHCC is not prescribing consent
practices in this IG and that any added language will be softened to a
permissive ("may", "in cases where") rather than mandatory form. A commenter
also flagged that the topic overlaps with the FAST Consent / FHIR Consent
Management IG dependencies.

## Details

**Description:**

This section needs to incorporate a requirement for the system maintaining
the data to validate patient consent to share information back to the
'client' requesting the update.

**Suggested language:**

> In the subscription mechanism, instead of the client system regularly
> querying the server to see if there are new Tasks or changes to existing
> Tasks, the client creates a [Subscription](http://hl7.org/fhir/uv/subscriptions-backport/STU1.1/StructureDefinition-backport-subscription.html)
> instance on the server that indicates that it wants to be notified about
> changes to Tasks and, in the Subscription, provides filters that describe
> what subset of Tasks it is interested in. **After the server validates
> patient consent to share data with the subscribing entity,** the server
> will then push notifications when there are new Tasks and the client can
> then query for the specific Tasks that have changed.

**Resolution Description (Persuasive with Modification):**

The work group accepted the intent but loosened the language to make consent
validation permissive rather than mandatory:

> "In the subscription mechanism, instead of the client system regularly
> querying the server to see if there are new Tasks or changes to existing
> Tasks, the client creates a [Subscription](http://hl7.org/fhir/uv/subscriptions-backport/STU1.1/StructureDefinition-backport-subscription.html)
> instance on the server that indicates that it wants to be notified about
> changes to Tasks and, in the Subscription, provides filters that describe
> what subset of Tasks it is interested in. **In cases where the server
> validates patient consent to share data with the subscribing entity,** the
> server **may** then push notifications when there are new Tasks and the
> client can then query for the specific Tasks that have changed."

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update Section 9.2.2 of the US SDOH Clinical Care IG to insert the
reporter's exact suggested language into the Subscription-mode narrative,
making consent validation a normative prerequisite to notification:

> "… After the server validates patient consent to share data with the
> subscribing entity, the server will then push notifications when there
> are new Tasks …"

No new artifact is created; this is a narrative change to the
Subscription-mode description in Section 9.2.2, optionally accompanied by
a non-normative note that the Consent resource (or the SDOHCC Consent
profile) is the expected vehicle for capturing the patient's authorization.

#### Justification

The request directly addresses a real privacy gap: the current §9.2.2 text
describes a server pushing Task updates to a subscribing client without any
mention of authorization to disclose. Tying notification to a validated
consent aligns the IG with widely-stated FHIR privacy expectations and with
the direction of the FAST / FHIR Consent Management IG that one commenter
flagged. The reporter has already supplied implementable language, so the
editorial cost is low and the consistency with sibling ticket FHIR-54609
(which asks the same of §9.2.1) is preserved.

---

### Disposition B: Alternative Approach (Recommended — matches the recorded resolution)

#### Proposal

Adopt the loosened, permissive form of the language already captured in the
ticket's Resolution Description, mirroring the approach taken for
FHIR-54609:

> "In the subscription mechanism, instead of the client system regularly
> querying the server to see if there are new Tasks or changes to existing
> Tasks, the client creates a Subscription instance on the server that
> indicates that it wants to be notified about changes to Tasks and, in the
> Subscription, provides filters that describe what subset of Tasks it is
> interested in. **In cases where the server validates patient consent to
> share data with the subscribing entity,** the server **may** then push
> notifications when there are new Tasks and the client can then query for
> the specific Tasks that have changed."

In parallel, defer any prescriptive consent mechanics to a single
higher-level statement in the IG (the global consent statement that
Gravity has indicated it will draft, per JIRAUSER24103's 2026-03-18
comment), and cross-reference the FAST FHIR Consent Management IG from
that higher-level section rather than from §9.2.2.

#### Justification

This is the position SDOHCC editors have already articulated on the
ticket: "We are not prescribing any consent practices or guidelines in this
IG. The SDOHCC Consent profile is not a complete treatment of the issues
related to consent; it is a starting point to test the viability of
exchanging consent information." Using "in cases where … may" acknowledges
the privacy concern in the Subscription narrative without obligating every
SDOHCC-conformant server to implement consent validation as a precondition
to notifications — which would be a normative change well beyond the scope
the work group has scoped for this IG and would create a hard dependency on
a consent infrastructure that is not yet stable. It also keeps §9.2.2
parallel with the resolution already accepted for §9.2.1 in FHIR-54609.

---

### Disposition C: Decline

#### Proposal

Reject the change on the grounds that consent management is explicitly
out of scope for the SDOHCC IG and is being handled by the FAST FHIR
Consent Management work; leave §9.2.2 untouched and respond to the
reporter pointing them at that IG and at any forthcoming Gravity
global-consent statement.

#### Justification

Defensible because: (a) the SDOHCC editors have already stated on the
record that the IG does not prescribe consent practices; (b) Subscriptions
in FHIR carry no built-in consent semantics, so embedding a normative
"server validates consent" step in narrative without a corresponding
conformance artifact (extension, search parameter, OperationOutcome
behaviour) risks unimplementable text; (c) a commenter has flagged that
FAST Consent / FHIR Consent Management is the proper home for this
requirement. The cost is that the privacy gap the reporter identified
remains undocumented in §9.2.2.

---

### Recommendation

**Recommended disposition:** B

Disposition B matches the resolution the work group has already recorded
on the ticket (Persuasive with Modification) and the editor stance posted
in the comments of FHIR-54609, and it keeps §9.2.2 wording symmetric with
the §9.2.1 wording approved for sibling ticket FHIR-54609. Disposition A
is too strong: making consent validation a normative precondition to
Subscription notifications introduces a conformance obligation that
SDOHCC has explicitly disclaimed and that has no defined mechanism in the
Subscriptions R5 Backport profile referenced by the section. Disposition C
under-serves the reporter's legitimate privacy concern and would leave
§9.2.1 (which is being amended) inconsistent with §9.2.2. Pairing the
permissive narrative tweak with a pointer to the forthcoming
Gravity-level consent statement and the FAST Consent Management IG gives
implementers the signal without overreaching.
