## Summary

Stephen Chu requests that `AllergyIntolerance.reaction.onset` (currently
`0..1 dateTime`) be replaced by a choice element `AllergyIntolerance.reaction.effective[x]`
with two type options: `effectiveDateTime` and `effectivePeriod`. The
clinical motivation is to allow recording of the *duration* of an
individual reaction (onset + abatement) rather than only its starting
instant, since reaction duration is itself clinically meaningful (e.g.,
how long a rash, swelling, or anaphylactoid manifestation persisted).

The label `OnsetDiagnosisAbatementDates` indicates this ticket is part
of a coordinated cross-resource campaign asking Patient Care to align
onset/abatement modelling across clinical-statement resources. The
ticket has no cross-referenced Jira, Zulip, or GitHub items in the
Augury index, but two label-adjacent items provide useful context:
[FHIR-53631](https://jira.hl7.org/browse/FHIR-53631) (AU Patient
Summary) confirms that AUCDI maps "date/time of onset of first
reaction" to `AllergyIntolerance.onset[x]` (the resource-level onset),
not to `reaction.onset`, leaving reaction-level *duration* unaddressed
in the base resource. [FHIR-52939](https://jira.hl7.org/browse/FHIR-52939)
(Patient Care, Not Persuasive) is the most recent Patient Care
disposition on the adjacent `allergyintolerance-duration` extension; PC
declined to add an estimate-vs-precise indicator because "most timing
elements are assumed to be estimates." Russell Leftwich's comment on
that ticket explicitly notes that the extension is on the *reaction*
element, that "It is the duration of the reaction and not the duration
of onset," and that "the time from administration to the onset of the
reaction would be clinically significant" — directly supporting the
clinical premise of FHIR-54072 while raising questions about whether a
new modelling change is needed at all given the existing extension.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In FHIR Core, modify the `AllergyIntolerance` resource definition so
that `AllergyIntolerance.reaction.onset` (currently `0..1 dateTime`) is
renamed to `AllergyIntolerance.reaction.effective[x]` with `0..1`
cardinality and the type set as a choice of `dateTime | Period`. The
StructureDefinition change touches:

- `reaction.onset` element — renamed and re-typed.
- Element short/definition/comments updated to describe the choice and
  the meaning of `effectivePeriod` (start = reaction onset; end =
  reaction abatement / resolution).
- A backwards-compatibility note in the resource Notes / maintenance
  page documenting that R5 `reaction.onset` (dateTime) maps to
  `reaction.effective.ofType(dateTime)` in the next release; provide a
  forward/backward conversion rule for FHIRPath/StructureMap.
- The `allergyintolerance-duration` extension (which today extends
  `reaction` to express reaction duration) is marked **deprecated** in
  favor of computing duration from `reaction.effectivePeriod`. Cross-
  reference FHIR-52939 in the deprecation note.

#### Justification

The proposal is internally consistent with the precedent already set at
the resource level by `AllergyIntolerance.onset[x]`, which uses a
choice including `Period` precisely because allergy timing is often
imprecise (see the *implementers > AllergyIntolerance onset[x]* thread
where Russell Leftwich endorses `onsetPeriod` for vague patient-reported
timing). It also resolves a known modelling gap surfaced in FHIR-52939
and in the *Reaction are separate diagnoses* thread: the lack of a
first-class way to express reaction duration. Using `effective[x]` with
`Period` semantics is the FHIR-idiomatic mechanism for "an event with a
known/estimated start and end" (cf. `Procedure.performed[x]`,
`Observation.effective[x]`). Renaming `onset` → `effective` aligns the
element with this idiomatic naming convention, which is the
established Core pattern when the element represents the time the event
occurred rather than the start of an ongoing condition.

---

### Disposition B: Alternative Approach

#### Proposal

Leave `AllergyIntolerance.reaction.onset` unchanged as `0..1 dateTime`
to preserve backward compatibility, and instead address the
duration/abatement use case by:

1. Adding a sibling element `AllergyIntolerance.reaction.abatement[x]`
   (`0..1`, choice of `dateTime | Period | dateTime + duration`), or
   most simply `0..1 dateTime` — paralleling `Condition.abatement[x]`.
2. Promoting the existing `allergyintolerance-duration` extension into
   the core resource as `AllergyIntolerance.reaction.duration` (`0..1
   Duration`), or keeping it as an extension and adding documentation
   that `onset + duration` is the canonical way to express a bounded
   reaction.
3. Clarifying in the element definition of `reaction.onset` that
   imprecise/period-style timing should continue to use the existing
   resource-level `AllergyIntolerance.onset[x]` for the *first*
   reaction (consistent with the AUCDI mapping in FHIR-53631), and that
   the new `abatement[x]` covers the upper bound of an individual
   reaction.

#### Justification

This achieves the clinical objective (capturing reaction duration)
without a breaking rename or type change. Renaming `reaction.onset` to
`reaction.effective[x]` in a published resource is a substantive
breaking change for every implementer that has bound to
`AllergyIntolerance.reaction.onset` — terminology services, profiles
(US Core, IPS, AU Core, IPA, QI-Core), validators, mappings, and
search-parameter expressions all reference the existing element name.
The `Condition.onset[x]` / `Condition.abatement[x]` pair is the
established Core pattern for "a clinical event with a start and end" on
a clinical-statement resource, and mirroring it on `reaction` is more
familiar to implementers than introducing `effective[x]` semantics on a
backbone-element nested inside another clinical resource. Disposition B
also explicitly resolves the disposition tension with FHIR-52939: the
deprecated estimate-indicator question goes away because `Period`
already conveys that both endpoints are estimates by virtue of being
recorded at all (per PC's own reasoning).

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive – Existing Mechanism**. Decline the
rename/re-type of `reaction.onset` and decline to add new core
elements. Document that:

- Reaction *duration* is already addressable via the
  `allergyintolerance-duration` extension on `reaction` (re-affirmed in
  FHIR-52939, Russell Leftwich's comment).
- Reaction *abatement* (when the manifestation resolved) can be
  captured via `reaction.note` or, for richer semantics, by referencing
  an `Observation` from the relevant manifestation in
  `reaction.manifestation` (codeableReference in R5).
- Resource-level first-reaction onset is already mapped via
  `AllergyIntolerance.onset[x]` (see FHIR-53631).

#### Justification

The change requested is breaking, the underlying clinical need has
existing (if scattered) mechanisms, and Patient Care's most recent
related decision (FHIR-52939, March 2026, Not Persuasive) explicitly
declined to enrich reaction-timing modelling on the grounds that
reaction timing is broadly assumed to be estimates and the additional
precision is not clinically actionable. Russell Leftwich's comment
there even pre-empts this ticket: he distinguishes "time from
administration to the onset of the reaction" (clinically significant)
from duration precision (not needed beyond minutes/hours/days). On
that basis, a defensible position is that the existing extension plus
codeableReference observations covers every actionable use case
without churning the base resource.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the clinical value (reaction duration /
abatement) while avoiding a hard backward-incompatible rename of a
long-established, widely-profiled element. Renaming
`AllergyIntolerance.reaction.onset` to `reaction.effective[x]` would
ripple through every downstream IG and search parameter that currently
references it; the cost is hard to justify when adding
`reaction.abatement[x]` (and optionally promoting the existing
`allergyintolerance-duration` extension) achieves the same clinical
expressivity using the well-understood `Condition.onset[x]` /
`Condition.abatement[x]` pattern. Disposition B also lets Patient Care
reconcile its prior FHIR-52939 decision: rather than reversing course,
the WG can frame the new core elements as the *replacement* mechanism
that absorbs the duration-extension question, which is a cleaner
narrative for ballot reviewers. Disposition A should remain on the
table only if the WG concludes that a clean break with the
`reaction.onset` element name is worth the implementer cost; Disposition
C is hard to defend now that AUCDI (FHIR-53631) has flagged real
implementer demand at the resource level and the *separate diagnoses*
Zulip thread has documented the reaction-level temporal-modelling gap.
