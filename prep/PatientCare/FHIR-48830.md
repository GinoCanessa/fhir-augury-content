## Summary

FMG (the FHIR Management Group) has reviewed every profile that currently
ships in the FHIR Core specification against its
[profile-approval criteria](https://confluence.hl7.org/pages/viewpage.action?pageId=256187125#FHIRArtifactGovernanceProcess-ProfileApprovalProcess)
and produced [a per-profile proposal](https://docs.google.com/spreadsheets/d/1A-VFJeOucscIrAP3TvWzXujYvD2DIubBk-OFD6l5Ea8/edit?usp=sharing)
identifying profiles that should be moved out of Core (typically into an
existing IG such as CRMI, CDS, EBM, Genetics) or deleted. The motivation is
to keep Core lean for R6 and beyond, since IGs can continue to iterate after
Core stabilises.

This ticket asks **Patient Care** to make a decision on the single Core
profile FMG attributes to PC: **FamilyMemberHistoryForGeneticsAnalysis**
(`familymemberhistory-genetic`). Grahame Grieve's "GG" column in the
spreadsheet recommends moving it out of Core, most plausibly to a Genetics IG
owned/co-owned by Clinical Genomics. PC must either (a) accept the move and
coordinate the new home, (b) propose an alternative (e.g., delete, or keep
Core with a justification against the criteria), or (c) defend keeping it in
Core.

Sister tickets created from the same Zulip review and explicitly named in
the FMG thread are
[FHIR-48827](https://jira.hl7.org/browse/FHIR-48827) (Vocab),
[FHIR-48828](https://jira.hl7.org/browse/FHIR-48828) (FHIR-I:
ActualGroup, DocumentStructure, GroupDefinition, ShareableTestScript),
[FHIR-48829](https://jira.hl7.org/browse/FHIR-48829) (Security:
ProvenanceRelevantHistory), and the already-approved
[FHIR-48393](https://jira.hl7.org/browse/FHIR-48393) (CDS/CQI removals). As
of the latest activity in the source Zulip thread (May 2025, R6 Ballot 3),
`FamilyMemberHistoryForGeneticsAnalysis` is one of four profiles Lloyd
McKenzie still flagged as unresolved, so PC's disposition here is on the
critical path for closing out the FMG core-profile cleanup.

## Proposed Dispositions

Because no GitHub cross-reference was surfaced and no `repo-analysis`
briefing applies, the dispositions name the affected Core artifact by its
canonical URL and published location
(<https://build.fhir.org/familymemberhistory-genetic.html>) rather than by
source-tree path. Whichever option PC adopts, the actual edit will land in
the FHIR Core source repository (HL7/fhir) under whatever directory hosts
the `familymemberhistory-genetic` `StructureDefinition` and its supporting
intro/example files; FMG and the Core publication tooling will need to
coordinate the canonical-URL handling, mirroring the technical issue
Grahame Grieve flagged on 20 May 2025 for the Vocab profiles being
relocated to CRMI.

### Disposition A: Accept As Requested

#### Proposal

Patient Care endorses removing
`FamilyMemberHistoryForGeneticsAnalysis`
(`http://hl7.org/fhir/StructureDefinition/familymemberhistory-genetic`)
from the FHIR Core specification for R6, and supports relocating the
profile, its narrative
(`familymemberhistory-genetic.html`), any examples and the
narrative reference from `FamilyMemberHistory` to a Clinical Genomics
implementation guide (the existing
[Genomics Reporting IG](https://build.fhir.org/ig/HL7/genomics-reporting/)
is the most plausible host; if Clinical Genomics prefers a new "Family
History for Genetics" IG, PC defers to that choice). PC asks FMG to
coordinate with Clinical Genomics on:

1. Which canonical URL the relocated profile will carry. Following Bryn
   Rhodes / Grahame Grieve's discussion on 20 May 2025, the simplest
   path is to retire the existing
   `http://hl7.org/fhir/StructureDefinition/familymemberhistory-genetic`
   canonical and republish under the IG's canonical
   (`http://hl7.org/fhir/uv/genomics-reporting/StructureDefinition/...`),
   with a redirect / deprecation note in R6 Core.
2. Removing all build references to the profile from the
   `FamilyMemberHistory` resource intro and the Patient Care list of
   profiles.
3. Adding an entry to the R6 substantive-changes list noting the move so
   implementers using the Core canonical can find the IG version.

PC's role beyond the endorsement is review-only; Clinical Genomics owns
the IG-side artifacts.

#### Justification

The profile clearly fails the Core profile-approval criteria as
clarified by FMG: it has a single, narrow domain (genetics-driven
family-history capture) that is fully served by an IG audience, it is
not referenced from any other Core artifact in a normative way, and no
one in the FMG thread (PC, Clinical Genomics, FMG, FHIR-I or any
implementer) has stepped forward to argue it belongs in Core. Every peer
WG that has dispositioned an analogous profile in the same review
(CDS/CQI via FHIR-48393, Vocab via FHIR-48827, FHIR-I via FHIR-48828,
Devices via FHIR-46029) chose relocation or deletion, so accepting the
move keeps PC consistent with the established R6 cleanup pattern and
avoids being the lone WG that blocks closure of the FMG initiative.
Putting the profile under Clinical Genomics also matches the
profile's actual subject-matter expertise; PC owns
`FamilyMemberHistory` as a resource but the genetics-specific constraints
are authored and maintained by the Clinical Genomics community.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than relocate to an existing or new IG, **delete the
`FamilyMemberHistoryForGeneticsAnalysis` profile outright in R6 Core
and do not republish it elsewhere**. In its place:

1. Add a short non-normative note to the `FamilyMemberHistory` resource
   intro pointing implementers needing genetics-specific structure to the
   Genomics Reporting IG (which already provides
   `FamilyMemberHistory`-adjacent artifacts such as family-pedigree
   observations) and to SDC/StructureMap patterns for collecting genetics
   pedigree data.
2. Retire the
   `http://hl7.org/fhir/StructureDefinition/familymemberhistory-genetic`
   canonical with a "deprecated, no successor" status so that any tooling
   following the canonical lands on a clear retirement page.
3. Coordinate with Clinical Genomics so they confirm in writing that they
   do **not** wish to inherit the profile.

#### Justification

The profile has had effectively zero authoring activity since DSTU
(it is one of the long-lived Core profiles flagged in the FMG sweep),
no implementer in the FMG thread has spoken up to defend the existing
constraints, and the genetics community's centre of gravity has shifted
to Genomics Reporting / pedigree-as-Observation patterns rather than a
`FamilyMemberHistory` profile. Republishing a profile no one is using
just to preserve the artifact carries a real cost: an IG that adopts it
inherits the maintenance burden and the same governance question
recurs at the next IG ballot. Deletion is a cleaner outcome and
mirrors what CDS/CQI did for several of their unused Core profiles
under FHIR-48393. The trade-off vs. Disposition A is breaking any
implementer currently conforming to the Core canonical; that risk is
real but small (the profile is not referenced from any normative
material) and is mitigated by the deprecation note and the substantive
change-log entry.

---

### Disposition C: Decline

#### Proposal

Patient Care declines to remove
`FamilyMemberHistoryForGeneticsAnalysis` from FHIR Core and asks FMG to
keep it in R6 Core as-is. PC takes the position that the profile meets
the Core criteria as a canonical example of how to constrain
`FamilyMemberHistory` for a recognised cross-cutting use case (family
pedigree for genetics), that there is implementer reliance on the
existing canonical URL that PC is unwilling to break in R6 without
evidence, and that no Genetics-owned IG has formally accepted the
artifact.

#### Justification

Defensible only if PC can point to (a) at least one implementer or
adopter who relies on the existing canonical and (b) a reading of the
FMG criteria under which "canonical genetics constraints on
`FamilyMemberHistory`" qualifies as Core (as opposed to IG) material.
Neither piece of evidence appears anywhere in the FMG thread, the
ticket comments, or the cross-reference index. Declining therefore
puts PC in the awkward position of being the only WG in the FMG sweep
that resisted relocation without a documented case, and it leaves
Lloyd McKenzie's R6 Ballot 3 "still unresolved" list permanently
populated. The justification this disposition would need to carry is
real — backward compatibility for the Core canonical — but is hard to
sustain absent any voice in the community making it.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the right balance: it honours the FMG profile-approval
criteria, aligns PC with every peer WG that has already dispositioned
its share of the cleanup (CDS/CQI, Vocab, FHIR-I, Devices), keeps the
genetics constraints alive for the implementers who do use them, and
puts ownership where the subject-matter expertise sits (Clinical
Genomics). The chief risk — canonical-URL churn for any current adopter
— is the same risk the Vocab/CRMI relocation is already solving in the
same Zulip thread, so PC can ride that solution rather than invent its
own. Disposition B (outright delete) is a reasonable fallback if
Clinical Genomics declines to inherit the artifact; PC should explicitly
ask them as part of executing A so that the fallback path is clear.
Disposition C should only be revisited if a PC member or an implementer
surfaces a concrete Core-criteria argument that has not appeared
anywhere in the record so far.
