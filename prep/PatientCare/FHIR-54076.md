## Summary

The reporter requests that the target type list of `CarePlan.addresses`
(currently `0..* CodeableReference(Condition | Procedure |
MedicationAdministration)`) be expanded to include `ImmunizationRecommendation`.
The motivating use case is care planning for patients with one or more chronic
conditions whose plans of care need to address vaccination needs that are
driven by condition-specific recommendations (for example, additional or
accelerated vaccinations indicated by a chronic disease). Today, a CarePlan can
reference the underlying `Condition` being addressed but cannot directly point
at an `ImmunizationRecommendation` resource that captures *what* vaccinations
that condition implies, forcing implementers to bridge the link out-of-band or
through an `Activity`/goal narrative. The single cross-reference recorded for
this ticket is to the FHIR element `CarePlan.addresses` itself; no other Jira
tickets, Zulip threads, or GitHub items were surfaced.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Modify `CarePlan.addresses` in FHIR Core from:

```
CarePlan.addresses : 0..* CodeableReference(Condition | Procedure | MedicationAdministration)
```

to:

```
CarePlan.addresses : 0..* CodeableReference(Condition | Procedure | MedicationAdministration | ImmunizationRecommendation)
```

Concretely, in the `CarePlan` resource definition add `ImmunizationRecommendation`
to the allowed target types of the `addresses` element's `Reference`, update
the element short/definition/comments to mention immunization recommendations
as a valid "thing the plan addresses", and refresh examples / narrative to
include at least one care plan that addresses an `ImmunizationRecommendation`
alongside one or more `Condition` references.

#### Justification

The change is additive at the reference-target level and does not alter
cardinality, type (still `CodeableReference`), or existing target types, so it
is backward compatible for instances and for servers that already accept the
existing targets. `ImmunizationRecommendation` is an existing normative-track
resource owned by a sibling workgroup that is explicitly designed to capture
"what immunizations are recommended for this patient", which is a clean
semantic fit for "what the care plan is addressing". The pattern of expanding
`addresses` over time is already established — `Procedure` and
`MedicationAdministration` were added to the original `Condition`-only target
list — so this proposal continues an existing trajectory rather than
introducing a new one.

---

### Disposition B: Alternative Approach

#### Proposal

Do not change `CarePlan.addresses`. Instead, model the link from the
`CarePlan.activity` side: add or document use of an extension (e.g.,
`careplan-addressesImmunizationRecommendation`) on `CarePlan` that carries a
`Reference(ImmunizationRecommendation)`, and/or rely on
`CarePlan.activity.plannedActivityReference` pointing at the
`ImmunizationRecommendation` (or at concrete `ImmunizationRequest` /
`Immunization` resources derived from it). Where the underlying driver is a
chronic condition, keep the existing `CarePlan.addresses` →
`Reference(Condition)`, and let the ImmunizationRecommendation be reachable
via `ImmunizationRecommendation.recommendation.supportingPatientInformation`
or via the activity tree.

#### Justification

`CarePlan.addresses` is intended to enumerate the *problems / things being
managed* by the plan; an `ImmunizationRecommendation` is arguably the
*response* to a problem (the chronic condition) rather than the problem
itself. Modelling it on the activity side preserves the current semantics of
`addresses` and avoids subtle re-interpretation of the element by existing
implementers. An extension keeps the door open for jurisdictions / IGs that
want the direct link without forcing the change into the base resource, and
avoids cross-WG coordination cost (Patient Care owns CarePlan; Public Health
WG owns ImmunizationRecommendation). The trade-off vs. Disposition A is that
implementers lose a single uniform "what this plan addresses" query and must
know to look at activities / extensions for the immunization-driven case.

---

### Disposition C: Decline

#### Proposal

Decline the change to `CarePlan.addresses`. Rationale: the underlying need —
representing that a care plan accounts for vaccination requirements driven by
a chronic condition — is already addressable today by (1) listing the
`Condition`(s) under `CarePlan.addresses`, (2) representing the recommended
immunizations as `CarePlan.activity` entries (which can reference
`ImmunizationRecommendation`, `ImmunizationRequest`, or planned
`Immunization`), and (3) using `ImmunizationRecommendation` itself, which
already links to the patient and supporting clinical information.

#### Justification

Adding new target types to a widely-implemented element such as
`CarePlan.addresses` is not free: every server that validates references, every
profile that constrains `addresses`, and every consumer that switches on the
target type must be revisited. With existing mechanisms covering the use case,
the marginal benefit of a direct `addresses → ImmunizationRecommendation` link
does not clearly outweigh the implementation churn, especially in the absence
of community discussion (no Zulip threads, no linked tickets) demonstrating
broader demand.

---

### Recommendation

**Recommended disposition:** A

The requested change is small, additive, semantically clean, and continues an
established pattern of broadening `CarePlan.addresses` as new resource types
mature. `ImmunizationRecommendation` is a natural member of the "things a care
plan is addressing" set when the plan exists specifically to manage
condition-driven vaccination needs, and forcing implementers to route this
through extensions (Disposition B) or activity-side modelling (Disposition C)
introduces avoidable variability across implementations. Disposition A does
require coordination with the Public Health WG (owner of
`ImmunizationRecommendation`) for sign-off and minor cross-resource narrative
updates, and the workgroup should confirm that no profile in the wild
constrains `CarePlan.addresses` in a way that would be disturbed; both are
manageable. Patient Care should accept the change, loop in Public Health for
concurrence, and update CarePlan's element definition, narrative, and at
least one example to reflect the new target type.
