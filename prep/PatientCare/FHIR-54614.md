## Summary

The ticket challenges constraint **prc-1** on the `Procedure` resource —
*"bodyStructure SHALL only be present if Procedure.bodySite is not present"* —
arguing that it is overly restrictive. The reporter (Eric Haas) notes that
`Procedure.bodySite` is `0..*` (multiple `CodeableConcept`s) while
`Procedure.bodyStructure` is `0..1` (a single `Reference(BodyStructure)`), so
the mutual-exclusivity rule produces an "all-or-nothing" outcome: a procedure
must either describe **every** site as a code, or describe **one** site as a
detailed `BodyStructure` reference, with no way to mix coded sites and a
referenced structure when both are clinically meaningful (e.g., several coded
sites plus one site needing morphology/laterality detail).

The ticket also flags duplication between the core element
`Procedure.bodyStructure` and two extensions defined in the FHIR Extensions
pack — [BodyStructure Reference](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-bodySite.html)
and [Procedure Target Body Structure](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-procedure-targetBodyStructure.html) —
both of which appear to cover the same need now that `Procedure` has a native
`bodyStructure` element.

The reporter offers two recommendations: (1) collapse `bodySite` and
`bodyStructure` into a single `CodeableReference`-typed element with guidance
that the same anatomic site should not be described both ways, and
(2) remove `Procedure` from the `context` of the two duplicating extensions.

No comments and no Jira / Zulip / GitHub cross-references were returned for
this ticket; the only system links found are the FHIR-element mentions of
`Procedure.bodySite` and `Procedure.bodyStructure` themselves.

## Details

**Description:**

> ### bodySite vs bodyStructure Constraint is Overly Restrictive
>
> **Constraint (prc-1):** "bodyStructure SHALL only be present if Procedure.bodySite is not present"
>
> **Issue:**
>
> - This enforces mutual exclusivity
> - Their cardinalities are different. `Procedure.bodySite` allows multiple codes, but `Procedure.bodyStructure` allows only a single reference
> - If they are indeed mutually exclusive, then why are the cardinalities not the same?
> - There are valid use cases for both, where some sites are coded only, and some sites need a detailed structure (`bodyStructure` reference to BodyStructure resource with morphology, laterality, etc.)
> - The extensions [BodyStructure Reference](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-bodySite.html) and [Procedure Target Body Structure](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-procedure-targetBodyStructure.html) duplicate `Procedure.bodyStructure`
>
> The constraint forces all-or-nothing: either ALL sites as codes or ONE site as a BodyStructure reference
>
> **Recommendation:**
>
> 1. Combine them into a single element using the `CodeableReference` datatype with guidance that they shouldn't describe the SAME site in both ways.
> 2. Remove Procedure from the context of the extension, [BodyStructure Reference](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-bodySite.html) and [Procedure Target Body Structure](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-procedure-targetBodyStructure.html).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply both reporter recommendations to `Procedure` and the related extensions:

1. **`HL7/fhir` (FHIR Core, R6 build):** Remove the existing `Procedure.bodySite`
   (`0..* CodeableConcept`) and `Procedure.bodyStructure` (`0..1 Reference(BodyStructure)`)
   elements and replace them with a single element
   `Procedure.bodySite : 0..* CodeableReference(BodyStructure)`. Drop invariant
   **prc-1**. Add element-definition guidance stating that `concept` and
   `reference` SHOULD NOT describe the same anatomic site, and that multiple
   repetitions are used for procedures involving more than one site.
2. **`HL7/fhir-extensions`:** Remove `Procedure` from the `context` of
   `StructureDefinition-bodySite` (BodyStructure Reference) and of
   `StructureDefinition-procedure-targetBodyStructure`. If `Procedure` is the
   only remaining context on `procedure-targetBodyStructure`, retire that
   extension.

#### Justification

Pattern-consistency: `CodeableReference(BodyStructure)` is exactly the modern
FHIR pattern for "either a code, a referenced resource, or both, repeated for
multiple instances" and is already used elsewhere in R6 (e.g., on Condition,
Observation in places, MedicationRequest, etc.). It naturally fixes the three
issues the reporter raises — mismatched cardinality, the all-or-nothing
mutex, and duplication with the two extensions — in one move. Removing
`Procedure` from the extension contexts prevents implementers from continuing
to use redundant extensions once core supports the same capability natively.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both elements but relax the mutex and align cardinality so the existing
shapes can co-exist additively, without a breaking element-shape change:

- **Drop invariant `prc-1`** from `Procedure` so `bodySite` and `bodyStructure`
  may be populated together.
- **Widen `Procedure.bodyStructure`** from `0..1 Reference(BodyStructure)` to
  `0..* Reference(BodyStructure)` so its repetition matches `bodySite`.
- Add element-definition guidance ("`bodySite` and `bodyStructure` SHOULD NOT
  describe the same anatomic site") and a worked example showing several
  coded sites plus a referenced `BodyStructure` for one site needing
  morphology/laterality detail.
- In `HL7/fhir-extensions`, remove `Procedure` from the `context` of the
  duplicating extensions as in Disposition A — that part is uncontroversial
  once core supports the use case directly.

#### Justification

This addresses the reporter's clinical concern (mixed coded + structured
sites in a single Procedure) and the cardinality mismatch without changing
either element's *type*. That makes the change far less disruptive: existing
instances remain valid, search parameters and profiles that target
`Procedure.bodySite` (a very common pattern in IGs such as US Core) continue
to work unchanged, and implementers don't have to migrate from
`CodeableConcept`/`Reference` to `CodeableReference`. The trade-off vs.
Disposition A is that the model retains two parallel elements rather than the
unified `CodeableReference` shape, but that is the same trade-off many other
R5/R6 resources already live with.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope for this cycle on the grounds that
`Procedure.bodyStructure` was deliberately added as a *narrow, mutually
exclusive* alternative to coded `bodySite` (hence prc-1 and `0..1`
cardinality), and that implementers needing the "coded sites + one detailed
structure" pattern can already achieve it via the existing
[BodyStructure Reference](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-bodySite.html)
extension on `Procedure.bodySite`.

#### Justification

Defensible because (a) the requested change is breaking — converting
`Procedure.bodySite : CodeableConcept` to `CodeableReference` would invalidate
existing instances and profiles, and even Disposition B's removal of `prc-1`
plus widening of `bodyStructure` is an FMM-level shape change for a heavily
implemented resource — and (b) an existing extension mechanism already
satisfies the underlying use case. The cost of this position is that it
preserves the current confusion between core `bodyStructure` and the two
duplicating extensions and leaves the prc-1 awkwardness in place.

---

### Recommendation

**Recommended disposition:** B

Disposition A is the cleanest model and matches where FHIR has been heading
with `CodeableReference`, but the cost of replacing
`Procedure.bodySite : CodeableConcept` (an element used in nearly every
clinical IG, including US Core) is high: every dependent profile, slicing
declaration, search-parameter expression, and implementation library would
need to be revisited. Disposition C under-serves a real, repeatedly-reported
use case and leaves the duplicating extensions in place. Disposition B
removes the invariant the reporter actually objects to (`prc-1`), aligns the
cardinality, and retires `Procedure` from the duplicating extension contexts
— delivering the requested behaviour with the smallest realistic disruption.
If the work group later decides to converge on `CodeableReference` for
`bodySite` across resources, Disposition B is a forward-compatible stepping
stone toward that.
