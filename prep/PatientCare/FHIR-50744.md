## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Tighten the security-guidance language in the PACIO Transitions of Care
IG by replacing the recommendation-level verb (SHOULD) with the
requirement-level verb (SHALL) for the relevant security statement(s).
Concretely:

- Locate the SHOULD-level security statement(s) in the IG (most likely
  in the IG's "Security and Privacy" / "Security Considerations" page,
  and any equivalent text on profile pages).
- Reword each identified statement so the conformance verb becomes
  SHALL, ensuring the surrounding sentence still reads as a testable
  conformance rule (e.g., "Implementations SHALL …" rather than
  "Implementations SHOULD …").
- Update any companion narrative (e.g., "is recommended" → "is
  required") so the prose matches the new normative weight.
- If the IG references an external security profile (e.g., SMART App
  Launch, IUA, FHIR Security & Privacy Module), state that conformance
  to that profile is now SHALL rather than SHOULD.

Because the ticket does not pin down the exact paragraph(s), the
preliminary step is to confirm with the reporter (Celine Lefebvre) which
SHOULD statement(s) are in scope before merging the wording change.

#### Justification

This matches the reporter's literal request and the stated rationale
("It should be required to follow security guidance"). Treating
security guidance as mandatory is consistent with how most HL7
specifications handle baseline security expectations, and a SHALL is
testable by IG validators and certification programs. The Highest
priority assigned to the ticket suggests the work group already views
the recommendation as too weak.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than a blanket SHOULD → SHALL substitution, refactor the
security section so each individual expectation carries the appropriate
conformance verb:

- Split the current omnibus security guidance into discrete, atomic
  statements (e.g., transport security, authentication, authorization,
  audit, consent, de-identification).
- Mark statements that have a clear, testable, universally-applicable
  obligation as SHALL (e.g., "Servers SHALL require TLS 1.2 or
  higher"; "Servers SHALL authenticate the requesting principal").
- Leave context-dependent or environment-specific guidance as SHOULD
  with an explicit rationale (e.g., guidance that depends on the
  trust framework or jurisdictional policy).
- Where a statement is really "conform to profile X", express it as
  "Implementations SHALL conform to [profile X]" and let that profile
  carry its own conformance verbs.

#### Justification

A blanket SHALL on a generic "follow security guidance" sentence can
create unintentionally untestable conformance — every implementer must
be able to demonstrate compliance, and "follow guidance" is hard to
audit. Splitting and re-leveling preserves the reporter's intent
(security is mandatory) while keeping the IG implementable and
testable. It also reduces the risk of breaking existing implementers
who pass today's SHOULD-level bar but would fail an under-specified
SHALL.

---

### Disposition C: Decline

#### Proposal

Do not change the conformance verb. Keep the existing SHOULD-level
security guidance and, if needed, strengthen the surrounding narrative
to emphasize that adopting the guidance is strongly expected.

#### Justification

A SHALL on security guidance imported from outside the IG (e.g.,
references to FHIR Security & Privacy Module or to the SMART App
Launch / IUA frameworks) can create conformance-verb conflicts: the
IG would assert SHALL while the upstream specification leaves the
behavior at SHOULD or context-dependent. Without a specific section
identified by the reporter, a global SHALL also risks turning advisory
text (which the IG cannot test) into normative text (which it must
test). This disposition is defensible if the existing guidance is
already known to point at a normative external profile that itself
imposes the obligation — in which case the SHOULD in the PACIO IG is
informative, and tightening it would create a redundant or conflicting
requirement.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need — that PACIO Transitions of Care
implementations be required, not merely encouraged, to honor the IG's
security expectations — is reasonable and aligns with the Highest
priority on the ticket. However, the request as written is too coarse:
"change SHOULD to SHALL" on unspecified text risks creating
untestable conformance rules or conflicts with referenced external
security profiles. Disposition B captures the reporter's intent (move
the bar from advisory to required) while forcing the security guidance
to be expressed as discrete, testable statements at the right
conformance level. As a procedural note, the work group should ask the
reporter to identify the specific SHOULD statement(s) they have in
mind so the refactor lands on the correct text; the disposition
discussion can proceed in parallel.
