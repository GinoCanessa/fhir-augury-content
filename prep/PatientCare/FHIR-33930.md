## Summary

The reporter is flagging an apparent profiling error in the Ophthalmology
Retinal (FHIR) implementation guide: a `value[x]` choice element in one of
the IG's profiles has been declared with a single `Quantity` data type
applied to the choice element itself, rather than expressing the
restriction in the conventional FHIR way (either by renaming the element
to `valueQuantity`, by slicing the choice on type, or by listing the
allowed types under `ElementDefinition.type` so each child is a
type-specific choice). The reporter notes "this is in other profiles,"
suggesting the same anti-pattern recurs across several profiles in the
guide.

Two sibling tickets opened minutes apart by the same reporter against
the same specification reinforce that this is a systemic authoring
issue in the IG rather than a one-off:

- [FHIR-33931](https://jira.hl7.org/browse/FHIR-33931) — `value[x]`
  appears as a choice element but its child elements are not choices
  ("This is in other profiles").
- [FHIR-33932](https://jira.hl7.org/browse/FHIR-33932) —
  `bodySite.value[x]` looks like a choice element without any choice
  children.

Together the three tickets describe the same underlying defect pattern:
profiles in the Ophthalmology Retinal IG are using `value[x]` (and at
least one nested `bodySite.value[x]`) without correctly constraining the
choice. FHIR-33930 specifically targets the case where `value[x]` has
been collapsed onto a single `Quantity` type expressed at the choice
level instead of via a renamed/sliced child.

## Proposed Dispositions

> Important: no GitHub repository for the Ophthalmology Retinal (FHIR)
> IG is cross-referenced in Augury, so the dispositions below cannot
> name a specific authoring root or file path. They identify the
> Ophthalmology Retinal IG as the target and describe the change in
> profile-authoring terms; the assignee will need to map them onto the
> actual `StructureDefinition` resources in that IG.

### Disposition A: Accept As Requested

#### Proposal

Treat the affected profile element as defective and re-author it to
conform to FHIR's normal choice-constraint conventions in the
Ophthalmology Retinal IG. Specifically, in the `StructureDefinition`(s)
that currently declare a `value[x]` element with `ElementDefinition.type
= [{ code: "Quantity" }]` (and no typed children/slices):

- Rename the element from `value[x]` to `valueQuantity` so the
  differential reads as a single typed element (the conventional way to
  pin a choice to one type), **or**
- Keep `value[x]` and slice it on `$this.type` with a single `Quantity`
  slice (the conventional way to allow a future re-broadening),

and apply the same fix to all other profiles in the IG that exhibit the
same pattern, per the reporter's note "this is in other profiles." The
fix should be coordinated with FHIR-33931 and FHIR-33932 so that
`value[x]` (and `bodySite.value[x]`) are corrected together rather than
in three separate passes.

#### Justification

The reporter's description is correct as a matter of FHIR profiling
convention: `value[x]` is a choice element, and pinning it to a single
type is normally expressed either by renaming to `valueQuantity` or by
type-slicing — not by writing the type onto the choice itself. Both
patterns are used widely in the core spec and in mature IGs (e.g., the
Vital Signs profiles referenced above), so adopting them here improves
consistency, makes the IG work correctly with downstream validators and
profile-aware tooling, and removes a class of confusing error/warning
output for implementers consuming the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than choosing a single rewrite shape across the whole IG, do a
two-step disposition:

1. **Audit and reclassify.** Sweep every profile in the Ophthalmology
   Retinal IG for `value[x]` (and `*.value[x]`) elements that have a
   single declared type or no typed children, producing a checklist
   that includes the profiles flagged by FHIR-33930/33931/33932.
2. **Fix per-profile by intent.** For each occurrence, choose the
   smallest correct authoring pattern based on the *intended*
   semantics:
   - If the profile is meant to permit only `Quantity` (or only one
     other type) for that observation, rename to the typed element
     (`valueQuantity`, etc.).
   - If the profile is meant to permit several types but constrain
     them, keep `value[x]` and add typed slices (`valueQuantity`,
     `valueCodeableConcept`, …) so each allowed type is expressed
     via a child slice, restoring the "choices on the children" shape
     the reporter expects.
   - If the original authoring intent cannot be recovered from the
     guide, default to the typed-element rename (Disposition A's
     first option) and flag the profile in the IG's release notes
     so reviewers can confirm.

#### Justification

The three sibling tickets together strongly suggest a systemic
authoring style problem, not a single typo. A blanket mechanical
rewrite (Disposition A) risks losing the distinction between "intended
to be `Quantity`-only" and "intended to be a constrained polymorphic
element," which matters for downstream profiling and for any
`Observation` resources already created against this IG. Doing the
audit first preserves intent, gives the workgroup a single coordinated
fix across FHIR-33930/33931/33932, and produces an artifact (the
checklist) that future ballot reviewers can use to confirm the IG no
longer contains the anti-pattern. The cost is one audit pass, which the
WG would have to do anyway to be sure all instances of "this is in
other profiles" have been addressed.

---

### Disposition C: Decline

#### Proposal

Decline the change request on the grounds that the publishing tooling
(IG Publisher / FHIR Validator) accepts `value[x]` with a single typed
constraint as a valid way to pin a choice element to one type, and that
the existing profiles therefore parse and validate correctly even if
the authoring style is unconventional. Mark FHIR-33930 (and by
extension FHIR-33931 and FHIR-33932) as "Not Persuasive — authoring
style preference, not a defect."

#### Justification

Strictly speaking, restricting `value[x]`'s `ElementDefinition.type` to
a single entry is a legal differential and produces a snapshot in which
only `Quantity` is allowed. A reviewer could argue that the reporter is
asking for a stylistic conformance with how the core spec's Vital Signs
profiles happen to be written, rather than reporting a defect that
breaks conformance, validation, or interoperability. If the WG's bar
for accepting changes to the Ophthalmology Retinal IG is "must
demonstrably break something," this ticket arguably does not clear it.

---

### Recommendation

**Recommended disposition:** B

The reporter's complaint is technically correct and consistent with how
the FHIR community expresses constrained choice elements, so simply
declining (Disposition C) leaves the IG with a known anti-pattern that
will keep generating questions and bug reports — the existence of two
sibling tickets filed minutes apart already shows that. At the same
time, Disposition A's mechanical rewrite is risky because the three
tickets explicitly hint that the pattern recurs in unspecified "other
profiles," and we don't know in each case whether the original author
meant "Quantity only" or "constrained polymorphic." Disposition B
captures the right outcome — bring the IG into line with conventional
`value[x]` profiling — while forcing an explicit, intent-preserving
audit that closes FHIR-33930, FHIR-33931, and FHIR-33932 together and
gives the workgroup a single defensible artifact for the IG's next
ballot. The assignee should be someone with both Ophthalmology Retinal
IG editorial rights and familiarity with `Observation.value[x]` slicing
patterns.
