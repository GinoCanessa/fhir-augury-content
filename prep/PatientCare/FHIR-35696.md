## Summary

Virginia Lorenzi (NewYork-Presbyterian) has flagged that the value set
currently bound to `Task.status` in the US SDOH Clinical Care IG is identical
to (or substantially overlaps with) the value set used by the "Patient Request
for Corrections" pattern. She is asking Patient Care whether there is a
benefit to converging the two onto a single, shared value set rather than
maintaining parallel definitions.

The ticket is purely an opportunity / harmonization question — it does not
specify a target value set, a binding strength, or a preferred consumer. The
cross-reference index surfaces only FHIR artifact mentions (`Task.status`,
`Patient.status`); no related Jira tickets, Zulip threads, GitHub PRs, or
Confluence pages were found. Comment history shows the WG previously voted
7-0-0 to mark this "Considered for Future Use" on 2022-04-25 (with a note that
both SDOH and Patient Request for Corrections were still receiving feedback
and might diverge further), then re-opened it post-STU2 publication on
2022-11-21 to revisit once the implementer community had more experience.

## Proposed Dispositions

The dispositions below reason from the ticket text and FHIR core / SDOH-CC
spec knowledge only, since no GitHub repo briefings are in scope. Any of the
"change" dispositions would, when actioned, require the SDOH-CC IG repo's
authoring root for `Task` profiles / SDOH value sets to be identified before
a concrete PR could be opened.

### Disposition A: Accept As Requested

#### Proposal

Converge the SDOH-CC `Task.status` binding and the "Patient Request for
Corrections" status binding onto a single shared ValueSet defined in the
SDOH-CC IG (or, preferably, lifted to a common terminology module if both
artifacts continue to live in the IG). Concretely:

1. Define one canonical ValueSet — e.g.,
   `ValueSet/SDOHCC-TaskStatusValueSet` (name TBD by the WG) — whose
   composition is the union/intersection currently shared between the two
   bindings.
2. Update the SDOH-CC `Task` profile to bind `Task.status` to the new
   canonical ValueSet (binding strength `required`, matching the base
   `Task.status` binding strength).
3. Update the "Patient Request for Corrections" profile/extension/operation
   (whichever artifact carries the parallel status) to point at the same
   canonical ValueSet.
4. Mark the now-redundant duplicate ValueSet as `retired` with a
   `replacedBy` extension pointing at the canonical one to preserve
   traceability for STU2 implementers.

#### Justification

This is the literal request and is the cleanest outcome from a terminology
governance standpoint: one concept set, one canonical URL, one set of
translations. It eliminates the risk of the two sets drifting silently (which
is exactly what Lynn flagged as a concern in 2022-11). Because base
`Task.status` is itself a `required`-bound coded value set in core FHIR, the
SDOH binding is effectively constrained anyway — formalizing the shared
SDOH-side ValueSet does not loosen conformance, it just removes a duplicate
definition.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than creating a new shared SDOH-side ValueSet, **rebind both SDOH-CC
`Task.status` and the Patient Request for Corrections status element directly
to the base FHIR `http://hl7.org/fhir/ValueSet/task-status` ValueSet** (or
the closest core equivalent for the corrections workflow) with a `required`
binding, and remove the IG-local copies entirely. If the corrections workflow
needs additional codes that core `task-status` does not provide, capture them
in a small `additional`-binding extension or an `Coding`-typed companion
extension rather than forking a parallel ValueSet.

#### Justification

The core FHIR `task-status` ValueSet is already maintained, versioned, and
translated by HL7 International; reusing it directly avoids the IG taking on
indefinite stewardship of a near-clone. It also makes Task instances
emitted by SDOH-CC implementers indistinguishable (at the status level) from
Task instances anywhere else in FHIR, which is friendlier to generic Task
clients and workflow engines. The trade-off vs. Disposition A is loss of
SDOH-specific labels/definitions if the WG had previously customized any
display strings — those would need to be carried as designations on the
core ValueSet or as documentation in the profile narrative.

---

### Disposition C: Decline

#### Proposal

Persistent Disposition: **Not Persuasive — leave the two value sets
distinct.** Document in the resolution that the SDOH `Task.status` binding
and the Patient Request for Corrections status binding are intentionally
separate to preserve independent evolution paths, and close the ticket
without spec change. Add a non-normative note in both profiles cross-linking
the two sets so future readers understand the relationship.

#### Justification

Defensible on the same grounds the WG used in November 2022 when it put the
ticket on hold: both SDOH-CC and Patient Request for Corrections were still
receiving implementer feedback, and forced convergence risks locking in a
compromise that constrains either side. A divergence today is cheap to fix
later (adding a code to either set is non-breaking); a premature merge is
expensive to undo because it creates implementer-visible binding churn. If
no implementer has actually reported pain from the duplication in the
intervening years, "do nothing" is a legitimate answer to a pure-opportunity
ticket.

---

### Recommendation

**Recommended disposition:** B

The original 2022 hold was justified by uncertainty about whether the two
workflows would diverge; with STU2 of SDOH-CC now published and the
implementer-community feedback window the WG was waiting for largely past,
the case for two parallel near-identical bindings is weak. Disposition A
(harmonize on a single SDOH-side ValueSet) solves the immediate duplication
but leaves the IG owning yet another `task-status`-like artifact that has to
be maintained, translated, and version-aligned with the base. Disposition B
(rebind both to the base FHIR `task-status` ValueSet, with extensions for any
true SDOH-specific additions) goes one level further: it eliminates the
duplication *and* the stewardship burden, and produces Task instances that
are interoperable with non-SDOH Task tooling out of the box. It is the option
most consistent with the FHIR principle of "reuse base concepts; profile
only what is genuinely different." Disposition C should only be selected if
Patient Care surfaces concrete implementer feedback — which neither the
ticket nor the cross-reference / Zulip / GitHub indices currently show — that
the two workflows actually need to diverge.
