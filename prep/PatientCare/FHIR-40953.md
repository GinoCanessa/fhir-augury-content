## Summary

The reporter (Nick Radov) requests that LOINC code [80404-7](https://loinc.org/80404-7) — "R-R interval.standard deviation (Heart rate variability)" — be added to the Physical Activity IG's activity value set, accompanied by an example measure. The rationale is that exercise physiologists treat heart rate variability (HRV) as a critical signal for monitoring stress and adapting care/exercise plans (e.g., shifting toward rest and recovery when HRV drops), and most contemporary high-end consumer wearables already capture it. The reporter further raises the longer-term possibility of including HRV in the vital signs panel.

The ticket has had a contested lifecycle. Patient Care initially disposed it as "Considered for Future Use" (vote 6-0-1, Jay Lyle / Rob Eastwood, 2023-07-13), reasoning that HRV was deliberately excluded from the first STU release because the experts felt there was too much complexity around using it appropriately to mandate Must-Support, and noting that the time-based Observation profile already has an *extensible* value set so trading partners can already exchange the LOINC code today. Lloyd McKenzie reviewed and approved that disposition on 2023-06-06; Nick Radov accepted deferral. Lynn Laakso then reverted the resolution post-STU publication on 2023-08-25, returning the ticket to Triaged for re-consideration in a future release of the IG. The ticket therefore needs a fresh disposition for the next ballot cycle.

## Details

**Description:**

Please consider adding LOINC code [80404-7](https://loinc.org/80404-7) (R-R interval.standard deviation (Heart rate variability)) to the activity value set with an associated example measure. Exercise physiologists consider HRV to be a critical measure for tracking stress and planning physical activity; for example, if a patient's HRV is temporarily low due to other stressors then it may be advisable to alter the care plan to focus more on rest and recovery instead of hard exercise until the HRV is back up in the normal range. Most high-end consumer smart watches and wearable fitness trackers released in the past few years do track HRV so the data is available for many patients. At some point I think we might even want to consider adding it to the vital signs panel.

## Proposed Dispositions

Because there is no cached repo briefing for the Physical Activity IG, the dispositions below name the conceptual artifacts (value set, profile, example) but do not pin file paths. Implementers should resolve the actual paths from the IG's authoring root once the briefing is generated.

### Disposition A: Accept As Requested

#### Proposal

Add LOINC `80404-7` ("R-R interval.standard deviation (Heart rate variability)") to the Physical Activity IG's activity value set (the value set bound — extensibly — to the time-based `Observation` profile), and publish a new example `Measure`/`Observation` instance demonstrating SDNN-style HRV reporting against that profile. Add a short narrative section in the IG explaining: (a) what HRV is and why it is reported in physical-activity contexts, (b) that 80404-7 is the SDNN-style code (R-R standard deviation) and is *not* a substitute for the broader HRV panel, and (c) acceptable measurement-context guidance (time window, device class, posture/state) so that consumers can interpret the value.

#### Justification

This is the literal request, comes from a domain-credentialled reporter, and matches the direction the work group already signalled — Lynn Laakso explicitly re-opened the ticket "post STU publication" so HRV could be picked up in a future release, and the original deferral comment promised exactly that ("We will absolutely be looking at introducing standardization of this measure in future releases"). HRV is well-established in exercise physiology (the cited *Frontiers in Public Health* review supports this), and consumer wearables commonly emit it, so there is realistic implementer demand — confirmed by the parallel `#terminology` Zulip thread, where implementers were already trying to send HRV via `Observation` and were stuck on terminology choice. Adding it to the value set with a worked example resolves that ambiguity authoritatively and is consistent with the precedent set in [FHIR-40952](https://jira.hl7.org/browse/FHIR-40952), where the work group actively curated the activity value set based on consumer-wearable realities.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a single SDNN code (80404-7) in isolation, define a small **HRV measure family** in the IG: (1) add LOINC 80404-7 (SDNN / rMSSD-style R-R standard deviation) to the activity value set, (2) add a non-normative companion table in the IG enumerating the other commonly reported HRV statistics (SDANN 18505-8, pNN50 76644-4, and the SNOMED-coded frequency-domain measures HF / LF / VLF) with explicit guidance that they are out of scope for the first round of additions but recognised, and (3) require that any HRV `Observation` carry `Observation.method` (or an `Observation.device` reference) plus `effectivePeriod` so the consumer can tell *how* the value was derived. Publish one example `Observation` + one example `Measure`. Defer any change to the vital signs panel to a separate, future ticket.

#### Justification

This addresses the underlying need (let implementers exchange HRV in this IG without guessing) while directly answering the legitimate concerns the work group raised in the original deferral — namely "enough complexity around using it appropriately that it wasn't something we wanted to mandate". The Zulip thread is concrete evidence of that complexity: implementers were collapsing SDNN and rMSSD onto the same LOINC code and mixing in SNOMED-coded frequency-domain measures without guidance. Requiring `method`/`device`/`effectivePeriod` on HRV `Observation`s is consistent with how the IG already handles other measurement-quality-sensitive metrics (cf. [FHIR-40952](https://jira.hl7.org/browse/FHIR-40952)'s rationale for preferring mean-over-pedometer over peak heart rate). It also keeps the change cleanly scoped to the activity profile and avoids touching vital signs, which is a much larger cross-IG conversation.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Decline the request and close the ticket as **Not Persuasive**, on the grounds that (a) the time-based `Observation` profile already binds its code value set *extensibly*, so any implementer with a willing trading partner can already send LOINC 80404-7 today — exactly as Lloyd McKenzie noted in the original disposition comment — and (b) the prior 6-0-1 work-group vote determined that HRV's interpretive complexity (device class, time window, frequency- vs. time-domain methods, posture, autonomic state) is not yet ready to be mandated in this IG. Direct the reporter to revisit once a broader HRV terminology and measurement-context guideline exists (e.g., from a Vital Signs / Devices joint effort).

#### Justification

The original deferral was substantive, not procedural: a 6-0-1 work-group vote (Jay Lyle / Rob Eastwood) explicitly concluded that HRV's appropriate-use complexity outweighed its near-term value as a Must-Support measure, and the reporter himself agreed at the time ("I would be fine with deferring to a future release"). Nothing in the ticket's history materially changes that calculus — no new clinical evidence, no new device-class guidance, and the parallel Zulip thread *underscores* the very ambiguity (one LOINC code being mapped to multiple distinct HRV statistics by implementers) that the original vote cited. The extensible binding on the time-based `Observation` profile means declining does not block any implementer — it merely declines to bless a single code in isolation. Standardising HRV poorly is arguably worse than leaving it under the extensible binding until a coherent panel can be defined.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach)

Disposition A is the path of least friction and is well-aligned with the work group's stated intent to revisit HRV after STU, but it inherits exactly the problem the original 6-0-1 vote called out: dropping a single LOINC code into the value set without measurement-context guidance leaves implementers in the same place as the `#terminology` thread, where SDNN and rMSSD were being collapsed onto the same code. Disposition C is defensible, but Lynn Laakso explicitly re-opened this ticket post-STU to *make progress*, and the prior disposition comment promised future work — declining now would be inconsistent with that signal and with the FHIR-40952 precedent of actively curating the value set. Disposition B threads the needle: it gives implementers the canonical SDNN code they have been asking for, names the rest of the HRV family so the IG isn't silent on it, and anchors interpretability with `method` / `device` / `effectivePeriod` requirements so the same data quality concerns the work group raised on FHIR-40952 do not recur for HRV. The vital-signs panel question, which the reporter raised as an aspiration, should remain explicitly out of scope for this ticket and be tracked separately if there is appetite.
