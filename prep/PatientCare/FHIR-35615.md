## Summary

The reporter (Joan Harper, on behalf of Ron G. Parker per "Comment 2") asks that
the canonical URL for a Value Set in the US SDOH Clinical Care IG be anchored
under `terminology.hl7.org` (THO) rather than under `hl7.org/fhir`, on the
basis that value sets — except those bound `required` to a `code` data type —
belong in THO. As part of the change, a corresponding UTG ticket should be
opened to register the Value Set in THO. The ticket was originally resolved
"Considered for Future Use" on 2022-03-28 (vote 8-0-0) with the rationale
that the value set and its related code system were still in development and
would be moved to THO in a future release; it was then re-opened on
2022-11-21 by Lynn Laakso to be addressed post-STU2 publication. No
cross-referenced Jira tickets, Zulip threads, or GitHub items were surfaced
by the augury index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Persistent — apply the requested terminology relocation in the next
US SDOH Clinical Care publication:

1. For each Value Set authored in the US SDOH Clinical Care IG that is **not**
   bound `required` to a `code` data type, change the `ValueSet.url` (and any
   companion `CodeSystem.url`, where the code system is also IG-local and
   intended for shared use) from a `hl7.org/fhir/...` anchor to the
   appropriate `terminology.hl7.org/...` anchor.
2. File the corresponding UTG ticket(s) to register each affected Value Set
   (and code system) in THO so the new canonical URLs resolve.
3. Update all internal references in the IG (profiles' `ValueSet` bindings,
   narrative, ConceptMaps) to point at the new THO canonical URLs and add
   redirects/release notes describing the URL change.

#### Justification

This matches the stated HL7 terminology governance pattern: shared
terminology content lives in THO, while only `code`-typed required bindings
are anchored under the spec's own URL space. The original 2022-03-28
disposition (Considered for Future Use, 8-0-0) explicitly committed to
"move it to THO in a future release," and Lynn Laakso re-opened the ticket
post-STU2 specifically to honour that commitment. Doing the move now —
rather than letting the IG accumulate more `hl7.org/fhir`-anchored value
sets — minimises future churn for implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Move the value sets to THO, but stage the change to protect existing
implementers:

1. Mint the new `terminology.hl7.org/...` canonicals in THO via UTG and
   publish them as the authoritative artifacts.
2. Retain the existing `hl7.org/fhir/...`-anchored ValueSet/CodeSystem
   resources in the IG as **deprecated aliases** for one publication cycle,
   with `status = retired` (or an explicit deprecation extension) and
   narrative pointing implementers at the new THO canonicals.
3. Publish a ConceptMap or migration note that maps old → new canonicals,
   and only remove the deprecated aliases in the publication after next.

#### Justification

Canonical URL changes are silently breaking for any consumer that has
hard-coded the old `hl7.org/fhir/...` URLs in profiles, validators, or
content. A staged deprecation cycle delivers the governance correctness
the ticket asks for, while giving downstream IGs and implementers one
clearly-labelled release to migrate. The trade-off vs. Disposition A is
extra authoring effort and one extra release of duplicated artifacts, in
exchange for materially better backward compatibility.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Mod / Not Persuasive** on the grounds that the
value sets in question are scoped to and only meaningful within the US
SDOH Clinical Care IG, and therefore appropriately live under the IG's
own `hl7.org/fhir/...` URL space; no UTG / THO move is required.

#### Justification

This disposition is only defensible if the affected value sets really are
IG-internal and not reused by other specifications. Given that Patient
Care has already voted (8-0-0) to move them to THO and re-opened the
ticket specifically to do so, declining now would contradict the
workgroup's prior decision and the THO governance pattern. It is
documented here for completeness rather than as a serious option.

---

### Recommendation

**Recommended disposition:** B

The workgroup has already committed (8-0-0 in March 2022, reaffirmed by
the November 2022 re-open) to relocating these value sets to THO, so
Disposition C is effectively off the table. The remaining question is
*how* to make the move. Disposition A is the literal request and is
correct in direction, but a bare canonical-URL swap is a silently
breaking change for every downstream consumer that resolves or compares
on `ValueSet.url`. Disposition B delivers exactly the governance outcome
the ticket asks for while giving implementers one clearly-labelled
deprecation cycle to migrate, at modest cost to the authoring team. The
workgroup should confirm which value sets are in scope (only those
**not** bound `required` to a `code` data type, per the ticket title)
and authorise the corresponding UTG submissions before the next
publication.
