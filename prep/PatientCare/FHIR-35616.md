## Summary

The ticket asks the US SDOH Clinical Care (SDOH-CC) FHIR IG to move the
canonical URL of a Value Set out of the IG's `hl7.org/fhir/...` namespace
and anchor it under `terminology.hl7.org` (THO). The reporter notes that
a companion UTG ticket would be needed to register the Value Set in THO.
The ticket title further qualifies the rule: only value sets that are
the target of a **required** binding to a `code` data type should remain
anchored under `hl7.fhir.org` (i.e., the IG namespace); all others belong
in THO. The ticket was originally resolved as "Considered for Future
Use" on 2022-03-28 by an 8-0-0 vote (Dieterle/Hill) on the rationale
that the Value Set was being developed for the SDOH IG use case and
could be moved to THO in a future release if other IGs needed it. That
disposition was later reverted by Lynn Laakso to "re-open post STU2
publication," and the ticket is now Triaged on the Block-Vote-8 list for
fresh disposition. No related Jira tickets, Zulip threads, or GitHub
items were surfaced through cross-references.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

For the SDOH Clinical Care (SDOH-CC) FHIR IG, audit every Value Set
defined in the IG and reclassify them by binding strength and target
type:

1. **Move to THO.** Any Value Set that is *not* the target of a
   **required** binding to a `code`-typed element is republished under
   `http://terminology.hl7.org/ValueSet/...` (and any backing CodeSystem
   under `http://terminology.hl7.org/CodeSystem/...`). Open the matching
   UTG ticket(s) so THO formally takes stewardship and assigns the new
   canonical URLs.
2. **Keep in the IG.** Only Value Sets that *are* required-bound to a
   `code` element remain at `http://hl7.org/fhir/us/sdoh-clinicalcare/
   ValueSet/...` (i.e., the IG namespace), since FHIR currently expects
   such bindings to resolve from the IG that defines the constrained
   profile.
3. **Update bindings and references.** In the SDOH-CC profiles, repoint
   every `binding.valueSet` reference and any narrative/IG page links to
   the new THO canonical URLs for items moved in step 1, and add a
   migration note in the IG change log so existing implementers can
   follow the rename.

#### Justification

This is the standard FHIR/HL7 governance pattern: terminology that is
reusable across IGs lives in THO under `terminology.hl7.org`, and only
IG-specific or implementation-bound `code` enumerations stay in the IG
namespace. The reporter's framing matches that pattern exactly. The
original 2022 deferral was justified only on "no other IG needs it
yet"; that argument weakens once the IG is post-STU2 and the cost of
later renames grows. Moving now, before broader adoption locks the
URLs, minimizes downstream breakage and aligns SDOH-CC with peer IGs
that route shared value sets through THO.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a **scoped / phased** version of A:

1. Define explicit criteria in the IG for "THO-eligible" Value Sets
   (reusable across domains, no IG-specific semantics, stable codes from
   external code systems such as SNOMED CT, LOINC, ICD-10-CM).
2. In this cycle, move only the subset that clearly meets those criteria
   (and file UTG tickets for them), while leaving SDOH-specific
   composition value sets — those built from SDOH-CC-curated content,
   in-development codes, or Gravity-specific groupings — in the IG
   namespace for now.
3. Add a deprecation/redirect note in the IG so the moved value sets'
   old IG-namespace URLs resolve (or are clearly documented as
   superseded) to the new THO URLs, and re-evaluate the remainder at
   the next ballot cycle.

#### Justification

This avoids the worst-case implementer impact (a flag-day rename of
every Value Set in the IG) while still honoring the principle that
broadly-reusable terminology belongs in THO. It also gives the Gravity
Project room to keep iterating on its own composition value sets
without forcing premature THO governance on content that is still
stabilizing. The trade-off vs. A is incompleteness: some value sets
that "should" be in THO will remain in the IG namespace until the next
cycle, and implementers will see two governance regimes side by side
during the transition.

---

### Disposition C: Decline (Persistent / Not Persuasive)

#### Proposal

Re-affirm the original 2022 disposition: keep the SDOH-CC Value Sets
anchored in the IG namespace (`http://hl7.org/fhir/us/sdoh-
clinicalcare/ValueSet/...`) for the current ballot cycle, on the basis
that they were authored to meet SDOH-CC use cases and there is no
documented external demand to consume them from another IG. Re-evaluate
only when a concrete cross-IG reuse case is filed.

#### Justification

The work to move value sets to THO is not free: it requires UTG
tickets, THO editorial review, coordination with the Vocabulary WG,
binding updates across SDOH-CC profiles, and a migration note for
implementers who already point at the IG-namespace URLs. The reverted
2022 vote (8-0-0) explicitly weighed those costs and judged them
unjustified absent cross-IG interest. With no cross-references, no
linked tickets, and no Zulip discussion surfaced for this ticket, there
is no new evidence that the original cost/benefit calculus has changed.
The risk of declining is that a future cross-IG consumer forces a more
disruptive rename later; that risk is real but currently hypothetical.

---

### Recommendation

**Recommended disposition:** B

Disposition B best reflects standard HL7 governance (shared terminology
belongs in THO) while respecting the practical reality the original
voters identified — SDOH-CC owns value sets that are still maturing and
that mix "obviously THO-eligible" content with IG-specific composition
logic. A blanket "move everything that isn't required-bound to a code"
rule (Disposition A) is correct in principle but heavy on coordination
with UTG and on implementer churn for a post-STU2 IG; doing it as a
single batch invites mistakes. A blanket decline (Disposition C)
preserves the 2022 status quo but ignores Lynn Laakso's reopen note,
which signals that Patient Care now intends to revisit this rather than
let it stand. A scoped move — clear criteria, file UTG tickets only for
the unambiguous cases this cycle, defer the rest — gives the workgroup
a defensible, incremental path that they can extend at the next cycle
once they see how implementers respond.
