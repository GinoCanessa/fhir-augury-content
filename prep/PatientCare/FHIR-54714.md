## Summary

Jose Costa-Teixeira asks Patient Care to add a first-class element (or
extension of higher rank than `Procedure.method`) on `Procedure` to
capture the **surgical approach** — the access route used to perform a
procedure, e.g., transaxillary, vaginal, abdominal, open, closed,
transanal, or scrotal approach. The reporter argues that
`Procedure.method` does not truly fit this concept, since "method" and
"approach" are distinct facets of how a procedure is performed. The
suggested binding would draw from the SNOMED CT subhierarchy
`< 103379005 |Procedural approach (qualifier value)|`.

A clarifying comment from michelle.m.miller flags an existing extension
in the cross-version FHIR Extensions Pack —
`procedure-approachBodyStructure` — and asks for an assessment of
overlap before deciding whether a new core element / extension is
warranted. There are no explicitly cross-referenced Jira tickets, but
several semantically related Patient Care tickets exist around
qualifiers on Procedure / BodyStructure (laterality, body-location
qualifier, treatment-intent timing, and the BodyStructure pattern
discussion), all of which inform the broader design question of how
Procedure should accept qualifier-style codes.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new core element `Procedure.approach` of type
`CodeableReference(BodyStructure)` (cardinality `0..*`), bound preferred
to a new value set
`http://hl7.org/fhir/ValueSet/procedure-approach`
defined as `< 103379005 |Procedural approach (qualifier value)|` in
SNOMED CT. Place the element after `Procedure.bodySite` in the
resource definition; cross-link to `Procedure.method` in the element
short/definition to make the distinction explicit ("how access was
gained" vs. "the technique applied once accessed"). Mark the existing
cross-version extension
`http://hl7.org/fhir/StructureDefinition/procedure-approachBodyStructure`
as superseded by the new core element in the Extensions Pack
`Obligations`/`comments`, and add a mapping note.

#### Justification

Mirrors V3/CDA's long-established `approachSiteCode` on Procedure (Gay
Dolin's point), satisfies the 80% rule that Lloyd implicitly invited a
change request against, and removes the friction Jose described
(extensions are easy to miss, especially in surgical IGs). Modelling it
as `CodeableReference(BodyStructure)` keeps parity with the existing
extension's typing while allowing either an inline coded approach
qualifier or a referenced BodyStructure instance, which is consistent
with how Patient Care has been moving qualifier patterns
(see FHIR-32340, FHIR-32345, FHIR-32413, FHIR-44969). Promoting from
extension to core is a non-breaking additive change for existing
implementers and gives surgical-IG authors a stable target.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new core element. Instead:

1. Promote `procedure-approachBodyStructure` from the FHIR Extensions
   Pack into the **standard** extensions registry with stronger
   visibility (a "Best Practice" obligation and a clear pointer from
   the `Procedure.bodySite` and `Procedure.method` element notes).
2. Loosen its target from `BodyStructure` only to
   `CodeableReference(BodyStructure)` so that implementers who want a
   pure qualifier code (e.g., "Open approach") can use it without
   having to instantiate a BodyStructure resource.
3. Add a dedicated value set
   `procedure-approach` bound preferred, scoped to
   `< 103379005 |Procedural approach (qualifier value)|`.
4. Optionally generalise the extension across resources that already
   carry approach-style information (e.g., it could be re-used wherever
   a `bodySite` is present), echoing the qualifier-generalisation work
   in FHIR-32340 / FHIR-44969.

#### Justification

Surgical approach is a high-value but ultimately **specialty** concept
— general-practice, lab, imaging-order and many other Procedure
consumers will never populate it, which weighs against core-element
promotion under the 80% rule. Daniel Venton's Zulip point also
remains valid: many operational systems carry approach inside
precoordinated procedure codes (e.g., "Laparoscopic appendectomy"), so
a separate slot is useful but not universal. A first-class extension
with a stable URL and an explicit value set delivers the missing
discoverability Jose and Gay called out, without committing R6/R7 to a
new core element that could be hard to retire if the precoordination
camp prevails. It also dovetails with FHIR-44969 / FHIR-32340's broader
"qualifier extension" generalisation rather than adding a one-off
element.

---

### Disposition C: Decline

#### Proposal

Decline the request as **already addressed by an existing mechanism**.
Resolve as "Persuasive with Modification → No Change", point the
reporter at
`http://hl7.org/fhir/StructureDefinition/procedure-approachBodyStructure`,
and add a discoverability cross-link from `Procedure.bodySite` and
`Procedure.method` to that extension in the resource notes. Optionally
add the SNOMED CT `Procedural approach` value set as the suggested
binding in the extension's definition.

#### Justification

A standard extension already exists and covers exactly the concept the
ticket describes; the friction reported is primarily *findability*
(Jose's own note that he didn't find it because of stale extension
registry filters) rather than a true gap. Patient Care has historically
preferred to keep low-prevalence Procedure modifiers in the Extensions
Pack to avoid bloating the core resource, and the precoordination
argument (Daniel Venton) is genuine: a non-trivial fraction of
implementers never need a separate approach slot. Adding a core element
also conflicts with the broader "qualifier-as-extension" pattern that
FHIR-44969 / FHIR-32345 / FHIR-32340 are still working through; making
a unilateral exception for surgical approach pre-empts that
harmonisation.

---

### Recommendation

**Recommended disposition:** B

The Zulip discussion is genuinely split between "promote to core" and
"leave it post-coordinated," which makes a hard accept (A) premature,
but the existing extension is clearly under-discovered and
under-specified (no value-set binding to
`Procedural approach (qualifier value)`, target restricted to
`BodyStructure`), which makes a hard decline (C) unsatisfying. Option B
captures the genuine fix Jose is asking for — a discoverable, properly
bound, FHIR-native way to record surgical approach on `Procedure` —
while preserving Patient Care's stated direction of treating qualifier
dimensions as extensions until the cross-resource qualifier pattern
work (FHIR-32340, FHIR-32345, FHIR-44969) settles. It is also the
lowest-risk path with respect to the precoordination concern, since
implementers whose procedure codes already carry the approach are not
forced to populate a new core slot.
