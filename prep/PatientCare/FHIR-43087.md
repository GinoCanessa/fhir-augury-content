## Summary

The reporter requests additional documentation in the US SDOH Clinical Care
IG (and notes it may also apply to the SDC `extract` documentation) to
clarify two questions about extracting `Observation` resources from a
`QuestionnaireResponse`:

1. **When should extracted Observations be grouped via `hasMember`?**
   The IG's "Assessment Instrument Support" page states that grouping is
   "generally redundant but is not prohibited" when each Observation can
   `derivedFrom` the QuestionnaireResponse, but also says a grouping
   Observation **MUST** be used when results are represented "only as
   Observation data." The reporter intends to retain and reference the
   QuestionnaireResponse and asks whether grouping is still recommended,
   and whether doing both (a grouping Observation **and** `derivedFrom`)
   is preferable. The NHANES grouping example is cited as a case that
   shows grouping but no `derivedFrom`.
2. **How should nested `Questionnaire` groups (e.g., PRAPARE) be reflected
   in extracted Observations?** Should the `hasMember` hierarchy mirror
   the full Questionnaire group structure, or is it acceptable to start
   grouping from a deeper subtree (such as the LOINC code
   `93041-2 — Money and Resources` panel)?

There are no related Jira tickets. The originating Zulip thread on
`#questionnaire > Item groups and observation extraction` led directly to
this filing at Lloyd McKenzie's suggestion that "more clarification in the
spec would be useful." Brian Postlethwaite noted that the SDOH IG also
ships StructureMaps that can drive extraction (with Java and .NET
engines), which is relevant context for how guidance might be expressed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Expand the US SDOH Clinical Care IG's
[`assessment_instrument_support.html`](https://build.fhir.org/ig/HL7/fhir-sdoh-clinicalcare/assessment_instrument_support.html)
narrative with an explicit decision matrix and worked examples covering:

1. **When to use a grouping Observation vs. relying on
   `Observation.derivedFrom(QuestionnaireResponse)`.**
   Add three named patterns:
   - **QR-only:** keep the QuestionnaireResponse as the sole grouping
     mechanism, no per-question Observations.
   - **Observations-only (grouping Observation REQUIRED):** restate the
     existing MUST and show the NHANES example.
   - **Hybrid (QR retained AND Observations extracted):** state
     explicitly that both `hasMember` grouping **and** `derivedFrom` on
     each member are RECOMMENDED (not just permitted), and explain that
     redundancy here is intentional — it lets Observation-only consumers
     work without resolving the QR while preserving full provenance for
     consumers that do.
2. **How nested Questionnaire `item.type = group` items map to
   Observation `hasMember` hierarchies.** Add normative-style guidance
   that the `hasMember` tree SHOULD mirror the nesting of `group` items
   in the source `Questionnaire` (parent group → grouping Observation;
   child leaf items → member Observations; child groups → nested
   grouping Observations), with the PRAPARE Questionnaire and the US
   Core PRAPARE panel examples (`93025-5` root with `93039-6`, `93040-4`,
   `93041-2`, `93042-0`, `93043-8` children) cited as the canonical
   shape. Explicitly state that "starting grouping at a deeper subtree
   only" is **not** recommended when extracting the whole instrument,
   because it loses the instrument-level context that the existing MUST
   is intended to preserve.
3. Cross-link the section to the SDC `$extract` operation documentation
   so the same patterns apply to StructureMap-based and
   observation-extraction-based pipelines, picking up Brian
   Postlethwaite's point about the existing StructureMaps shipped with
   the IG.

#### Justification

The reporter's confusion stems from genuinely ambiguous spec text — the
existing "generally redundant but is not prohibited" clause does not
tell an implementer what to *prefer*, and the NHANES example (no
`derivedFrom`) appears to contradict the worked PRAPARE pattern in US
Core (which uses panel grouping with `derivedFrom`). Lloyd's Zulip reply
already conceded that "more clarification in the spec would be useful,"
which is an implicit endorsement of this disposition. Adding a decision
matrix plus the PRAPARE/NHANES contrast aligns the IG with the existing
US Core examples the reporter himself surfaced in his comment, and
keeps the SDC extract documentation consistent. None of this requires a
breaking change — it codifies preferred patterns without invalidating
existing conformant resources.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than expanding only the SDOH IG narrative, lift the guidance one
level: add a short normative section to the **SDC** `$extract` /
observation-extraction documentation that defines the three patterns
(QR-only, Observations-only, Hybrid) and the "mirror the `group`
hierarchy in `hasMember`" rule once, then have the SDOH Clinical Care
IG (and any other IG that extracts Observations from Questionnaires)
link to that section and supply only domain-specific examples (NHANES,
PRAPARE). Optionally, ship a worked StructureMap in the SDOH IG that
implements the Hybrid pattern end-to-end on PRAPARE so that the guidance
is executable, not just prose.

#### Justification

The underlying question is not SDOH-specific — any IG that extracts
Observations from a QuestionnaireResponse will hit it, and the reporter
already noted the ticket "is potentially also relevant to the SDC extract
documentation." Putting the canonical rules in SDC avoids drift between
IGs and removes the contradiction that arises when the NHANES example
(no `derivedFrom`) and the US Core PRAPARE panel examples (with
`derivedFrom`) sit under different ownership. Brian Postlethwaite's
note about existing extraction StructureMaps and engines argues for
making the guidance executable: a reference StructureMap is harder to
misread than prose. Trade-off vs. Disposition A: it crosses a
work-group boundary (FHIR-I / SDC owns SDC; Patient Care owns SDOH
Clinical Care), so it requires coordination and is slower to land — but
the result is more durable.

---

### Disposition C: Decline

#### Proposal

Decline as "already addressed by existing mechanism / guidance." Close
with a comment pointing the reporter at: (a) the existing
`assessment_instrument_support.html` text (which already states that
grouping is permitted-but-redundant when `derivedFrom` is present, and
required when results are Observation-only), and (b) the US Core PRAPARE
panel examples the reporter found in his own follow-up comment, as a
worked illustration of the Hybrid pattern.

#### Justification

A defensible-but-weak reading is that the spec already says everything
needed: the MUST for Observation-only is explicit, the
permitted-but-redundant clause covers the Hybrid case, and the US Core
PRAPARE examples demonstrate the nested `hasMember` shape. Declining
avoids editorial churn in a published IG. However, this reading is
undermined by the fact that an engaged implementer (the reporter) read
the same text and could not derive a clear answer, and by Lloyd
McKenzie's own Zulip acknowledgement that clarification would be useful
— so this disposition is included for completeness rather than as a
strong candidate.

---

### Recommendation

**Recommended disposition:** A

Accepting as requested is the lowest-risk, highest-value option. The
ambiguity is real (an experienced implementer asked, an HL7 chair
agreed clarification is needed, and the NHANES vs. US Core PRAPARE
examples model two visibly different shapes), the change is purely
narrative plus examples, and there is no normative cost — no existing
conformant resource becomes non-conformant. Disposition B is genuinely
better in the long run because the guidance belongs in SDC, but it
requires cross-WG coordination and will not move on this ticket's
timeline; Patient Care can land Disposition A now and file a follow-up
against SDC to lift the same wording later. Disposition C is
unattractive because the very existence of this ticket and the Zulip
thread are evidence that the current text does not give implementers a
clear answer.
