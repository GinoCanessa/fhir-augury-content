## Summary

The reporter notes that the US SDOH Clinical Care IG (the "SDOHCC" IG) is
silent on a fundamental architectural question that arises whenever a
referral flows from a Referral Source through an intermediary
Coordination Platform (CP) to a Referral Target (and back again): when
the referralTask / patientTask references other instances — for example
ServiceRequest, Patient, Condition, Observation, Procedure,
Practitioner, Location, HealthcareService — is the CP expected to
**clone** those instances into its own store, or to **proxy** them in
place at the source? The ticket flags this as a significant unexplored
implementation complexity that the IG should address explicitly,
because the choice has substantial downstream implications for
identity, provenance, consent, update propagation, and capability
statements on each actor. The Augury cross-reference index returned no
direct cross-referenced Jira tickets, Zulip threads, or GitHub items
for this ticket key, so this Summary is drawn from the ticket itself.

## Details

**Description:**

There is considerable unexplored complexity in the IG that relates to
moving data instances that are linked directly/indirectly to the
referralTask and patientTask between the referral source and the
referral target via the intermediary CP (and in reverse).
Is the intermediary expected to clone the referred to instances (e.g.,
ServiceRequest, Patient, Condition, Observation, Procedure,
Practitioner, Location, HealthCareService) or provide a proxy interface
to them?   The IG is fairly silent on this, but clearly this will
represent considerable complexity for implementors.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new normative section to the SDOHCC IG (under the Coordination
Platform / Referral Workflow narrative pages) titled something like
*"Handling of referenced resources across the Source → CP → Target
boundary"*. The section would:

1. Enumerate every resource type that the SDOHCC profiles allow as a
   reference target from `Task` (referralTask, patientTask), the
   `ServiceRequest` payload, and any nested elements — i.e., the list
   already named in the ticket: `ServiceRequest`, `Patient`,
   `Condition`, `Observation`, `Procedure`, `Practitioner`, `Location`,
   `HealthcareService`.
2. For each, state which actor is **authoritative** for the instance,
   and require one of two behaviours when the instance crosses the
   boundary:
   - **Clone** (CP/Target creates a new resource in its own store, with
     a `Provenance` linking back to the source identifier and a stable
     `identifier` that lets downstream actors correlate the copy with
     the original); or
   - **Proxy** (CP/Target stores only the absolute reference URL and
     resolves it on read against the source's RESTful endpoint, with
     authentication and consent obligations spelled out).
3. Update the SDOHCC `CapabilityStatement` profiles (TaskReferral
   Management, Coordination Platform, Source, Target) to declare
   whether each actor publishes resolvable URLs for these resource
   types and whether it accepts cloned copies on write.
4. Add reverse-direction guidance describing how updates made on the
   Target side (e.g., a new `Procedure` documenting completion) are
   propagated back to the Source — either by `Provenance`-linked
   clones flowing in reverse, or by the Source dereferencing
   Target-hosted URLs.

#### Justification

This is exactly what the ticket asks for and what implementers have
been signalling they need; without an explicit decision, two
conformant CPs can implement this incompatibly and break
interoperability. FHIR itself supports both reference styles
(absolute URL vs. logical reference + `identifier`), so the IG can
constrain the choice without fighting the base spec. Several
keyword-adjacent tickets (FHIR-50069 on `searchInclude` mistakes,
FHIR-54910 on glossary gaps, FHIR-55373 on confusion about which task
is being referred to) reinforce that the IG's current silence is
producing real downstream defects, not just hypothetical ones.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than committing the IG to a single clone-vs-proxy rule per
resource, publish an **architectural appendix + capability
declaration**:

- Add a non-normative architecture appendix that documents both
  patterns (clone, proxy) with sequence diagrams for forward and
  reverse flows, and lists the trade-offs (identity stability,
  consent surface, update propagation, storage cost, dereference
  availability).
- Add a new `Extension` (e.g., `sdohcc-resource-handling-mode`) on the
  CP / Target `CapabilityStatement` whose value set is `clone | proxy
  | both`, declared per referenced resource type. Implementers MUST
  publish their mode; consumers MUST honour it.
- Require, for **either** mode, a uniform identifier-correlation rule:
  any clone MUST carry a `Provenance.entity.what` link to the source
  resource and a stable business `identifier`, so a downstream actor
  can always answer "is this the same Patient/Condition/etc. as the
  one over there?".

#### Justification

The clone-vs-proxy choice is genuinely context-dependent: large CPs
serving many small Sources will often need to clone for performance
and offline access, while tightly-coupled state HIE deployments may
prefer proxy semantics to avoid stale data. Forcing one rule risks
ruling out legitimate deployments that the IG was designed to
support. Declaring the mode in the `CapabilityStatement` keeps
interoperability deterministic without picking a winner, and the
shared `Provenance` + `identifier` rule is cheap to require and
solves the hard part (correlation) regardless of mode. This matches
how other IGs (e.g., the FAST National Directory work that
FHIR-54021 touches) handle cross-actor resource ownership.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Mod / Considered for Future Use**: do not
make any normative change in the current STU cycle, but add a single
short paragraph to the IG noting that the question is open and is
being tracked for the next major version. Continue to allow
implementers to choose either pattern for now.

#### Justification

This ticket has already been deferred once (the 7-0-0 vote on
2023-06-22 to "Consider for Future Use") and was re-opened only
because the original resolution wasn't applied as intended, not
because new evidence emerged. A change of this scope — touching every
referenced resource type, every actor's capability statement, and the
reverse flow — is large enough that doing it under the current
Block-Vote-1 batch risks producing under-baked normative text. If no
implementer ballot comments in the current cycle have surfaced
concrete interop failures, deferring again with an explicit
"non-decision is documented" note is defensible and lets the WG
collect more field experience first.

---

### Recommendation

**Recommended disposition:** B

The ticket is real and implementers will keep tripping on it, so
declining outright (C) just kicks the can down the road again — and
the prior 2023 deferral already showed that path doesn't produce
follow-through. But mandating a single clone-or-proxy rule per
resource type (A) is too prescriptive: in practice, both patterns are
in use across SDOHCC pilots and locking one in mid-STU could orphan
existing deployments. Disposition B threads the needle: it gives
implementers an architectural map, a CapabilityStatement-level
declaration so any two actors can negotiate behaviour deterministically,
and a uniform `Provenance` + business-`identifier` correlation rule
that solves the genuinely hard interop problem (knowing two resources
refer to the same real-world thing) regardless of which pattern an
actor picked. It is also the disposition most consistent with how
adjacent issues in the keyword neighbourhood (consent across actors,
HealthcareService alignment with FAST NDH) are being resolved.
