## Summary

Daniel Vreeman, after reviewing both PACIO implementation guides — the
Functional Status IG and the Cognitive Status IG — argues that the case for
maintaining two separate IGs is not evident. He observes that the two guides
share more than 80% of their narrative, that their driving use cases are
indistinguishable, and that their profile detail and element bindings are
fundamentally the same. The handful of genuine differences (e.g., a few
extensions) and the practice of giving sub-sections of the same underlying
CMS patient-assessment instruments (MDS, OASIS, etc.) different profile
names is described as added burden for both implementers and receivers, with
no rationale documented in the IGs themselves. The ticket proposes merging
the two IGs into a single "common core" IG for functioning data, with the
small number of domain-specific profiles organized as sub-sections of that
common core, and frames this as a more robust foundation for future PACIO
content.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FHIR Augury for this ticket, so the review rests on the ticket
description and its three internal comments.

> basically the same. Splitting them into two IGs and having different
> profile names for subsections of content drawn from the same overall

> MDS, OASIS, etc) imposes additional burden on implementers (and
> receivers). The rationale for splitting the IGs was not described in the
> IG. We propose merging the two IGs into a common core for functioning
> data, with the small number of domain-specific profiles that are needed
> being organized into subsections. Looking ahead to the other kinds of
> content being contemplated by the PACIO project, we believe this will be
> a more robust foundation.
>
> Is there value in having two IGs? Please stongly consider merging the two
> IGs.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Merge the **PACIO Functional Status IG** and **PACIO Cognitive Status IG**
into a single combined IG (e.g., "PACIO Functioning IG"). The merged IG
would:

- Define one shared narrative section covering the common use cases (CMS
  patient assessment instruments such as MDS, OASIS, IRF-PAI, etc.) and the
  general FHIR design (Observation-based functioning measurements,
  groupings, statuses, provenance).
- Collapse the duplicated profiles (Observation, ObservationGroup-style
  profiles, related extensions) into a single set of base profiles in the
  combined IG, with **sub-sections** ("Functional Status profiles",
  "Cognitive Status profiles") holding only the small number of profiles
  that genuinely differ — typically driven by domain-specific extensions
  and value sets.
- Re-version both existing IGs as deprecated successors that point at the
  merged IG (or publish the merged IG as PACIO Functioning STU2 and mark
  the two STU1 IGs as superseded).

#### Justification

The reporter's observation that the two IGs duplicate >80% of narrative and
profile content is consistent with the comment thread: the only durable
difference identified by the working group is the *value set / code system*
applied to otherwise-identical Observation profiles. Combining them
eliminates parallel-maintenance overhead, reduces implementer confusion
about which IG to conform to, and aligns with the FHIR community's general
preference for a single canonical profile per concept where the structural
content is the same. It also positions the PACIO project to add additional
"functioning" sub-domains (the reporter explicitly cites "other kinds of
content being contemplated") without spinning up a new IG each time.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the two IGs structurally separate but **factor out a shared "PACIO
Functioning Common" IG** that holds:

- The shared narrative (use cases, CMS instrument background, conformance
  patterns, security/provenance guidance).
- The shared base profiles, extensions, and shared value sets.

The Functional Status IG and Cognitive Status IG then depend on the common
IG and contribute only the genuinely domain-specific profiles, value sets,
and extensions. As an enabling step, adopt the run-time value set / code
system selection mechanism that PACIO and Gravity are investigating
(referenced in lynn_laakso's 2021-11-04 comment) so that profiles which
differ only by binding can be expressed once in the common IG with a
context-sensitive binding rather than duplicated in each child IG.

#### Justification

This addresses the reporter's underlying concern (duplication and
implementer burden) without forcing a single combined IG, which preserves
existing branding, ballot history, and downstream references to the two
STU1 IGs. It also accommodates the working group's observation that
Functional and Cognitive Status may end up using *different* focused LOINC
value sets in the future — a divergence that argues for keeping
domain-scoped artifacts identifiable. The cost is added IG-dependency
machinery and the need for the run-time value-set-selection mechanism to
mature; both are tractable but not free.

---

### Disposition C: Decline

#### Proposal

Decline the merge request. Retain the **PACIO Functional Status IG** and
**PACIO Cognitive Status IG** as separately published IGs. Document
explicitly in each IG's introduction why the two are kept apart (distinct
clinical domains, distinct stakeholder communities, divergent LOINC
sub-sets anticipated) and add a cross-reference between the two so readers
can navigate. Treat narrative duplication as an editorial issue to be
reduced, not a structural one.

#### Justification

The 2021-06-28 vote (6-0-0, "Considered for Future Use") and the
lynn_laakso comment together capture the working group's standing position:
the two IGs are expected to *diverge* in their value sets even though their
profile structure is currently similar, and a run-time mechanism for
selecting the correct value set is needed before consolidation is
technically feasible. Until that mechanism exists, merging would either
(a) lose the domain-specific binding intent or (b) require fragile
profile-level discrimination inside a single IG. Declining preserves
optionality and avoids reworking STU1 publications that have already shipped.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying complaint — that implementers and receivers are
asked to learn and conform to two near-identical IGs — is real and is not
adequately answered by Disposition C's "document why they're separate"
remedy. At the same time, the working group's 2021 reasoning about
divergent LOINC value sets and the need for a run-time
code-system/value-set selection mechanism is a substantive technical
constraint that Disposition A glosses over: a hard merge today would either
collapse the value-set distinction prematurely or import the duplication
into the merged IG. Disposition B (a shared "PACIO Functioning Common" IG
plus thin domain IGs) directly removes the duplication the reporter
identified, keeps the door open for the divergent-binding work that PACIO
and Gravity are pursuing, and avoids invalidating STU1 publication
artifacts. The workgroup should treat the run-time binding mechanism as the
gating dependency and set Disposition B as the target end-state, with a
clear migration path from the current two-IG arrangement.
