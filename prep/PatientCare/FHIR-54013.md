## Summary

The ticket, titled *"Make criticality an extension. Add rationale."*, asks
Patient Care to remove the `AllergyIntolerance.criticality` element from the
core FHIR resource and re-expose it as a standard extension that **requires** a
rationale (a `CodeableConcept`). The reporter argues that criticality is rarely
implemented (failing the 80/20 rule), is regularly conflated with reaction
severity by source systems that record "severity" at the intolerance level, and
in routine practice is almost always implicit from the substance/manifestation
pair — anaphylaxis is high‑risk, morphine‑induced nausea is not. Criticality
therefore only carries information when it diverges from the norm, and even
then it should be accompanied by a rationale (with a value‑set code such as
"Value chosen based on common experience with this substance and this symptom;
i.e., no special information" available for systems that must populate the
field). The cross-reference query returned no related Jira tickets, so the
request stands on its own; however, the supporting Zulip context (see below)
shows Jay Lyle has been raising this concern with Patient Care since 2023, and
the issue is connected to ongoing v2-to-FHIR mapping pain around AL1.4 →
`AllergyIntolerance.criticality`.

## Details

**Description:**

The distinction between allergy criticality and reaction severity is clear
within the FHIR specification. However, many systems record a property called
"severity", but they do so at the intolerance level, and few capture
"criticality". This has led to confusion in attempts to map data from these
systems into FHIR.

First, since the "criticality" property is not often implemented, it seems like
an obvious candidate for exclusion by the 80/20 rule.

For systems that do use it, it seems valuable, but only in rare cases. Most
providers expect that anaphylaxis carries a high level of risk, and that nausea
in reaction to morphine is typically not dangerous. If these examples were
marked "critical" and "not critical", all we have done is increase the
documentation burden. Only in the case of divergence from the norm — e.g.,
that the nausea is judged to be critical — does this assertion have value, and
even there, it would seem to require some supporting evidence to help the
consuming provider process the assertion.

In addition to removing criticality from the core spec to an extension, the
extension should require a rationale. A codable concept would support text in
the near term and computable values if and when they are identified. The
rationale value set, when built, should include a value for "Value chosen
based on common experience with this substance and this symptom; i.e., no
special information", in case any system actually requires its use.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the next non-normative cycle of FHIR Core, mark
`AllergyIntolerance.criticality` (currently 0..1 `code` bound to the required
`allergy-intolerance-criticality` value set) as deprecated, then remove it in
the following cycle. Replace it with a standard extension —
`http://hl7.org/fhir/StructureDefinition/allergyintolerance-criticality` —
defined on `AllergyIntolerance` (root), 0..1, with a complex structure:

- `value` : `code`, required, bound to the existing
  `allergy-intolerance-criticality` value set (preserves codes
  `low | high | unable-to-assess`).
- `rationale` : `CodeableConcept`, **required (1..1)**, bound (preferred or
  extensible) to a new Patient‑Care‑maintained value set
  `allergy-intolerance-criticality-rationale`. The value set must include a
  concept equivalent to *"Value chosen based on common experience with this
  substance and this symptom; i.e., no special information"* so that systems
  that must populate the element have a non-null escape hatch, plus codes for
  the principal evidence types (e.g., prior anaphylaxis, prior life-threatening
  reaction, family history, provoked challenge result, non-clinical default).

Publish a migration note in the AllergyIntolerance resource page explaining
why criticality is leaving core, and update the v2→FHIR concept map
`segment-al1-to-allergyintolerance` so that AL1.4 only maps to
`AllergyIntolerance.reaction.severity`, never to criticality.

#### Justification

This is exactly the path Lloyd McKenzie articulated on the Patient Care WG
Zulip thread: when an element is conceptually defensible but rarely populated
by industry, FHIR Core's convention is to demote it to an extension until
practice changes. The reporter, an openEHR clinical lead (Heather Leslie),
the AU Sparked accelerator's clinical design group, and the v2→FHIR mapping
discussion (Smithies, Oemig) all corroborate that real source systems do not
carry criticality and that conflating it with severity actively breaks
mappings. Requiring `rationale` on the extension addresses Russell Leftwich's
substantive concern from the same thread — that bare criticality codes drive
patient harm — by forcing the asserter to record evidence whenever they
diverge from the substance/manifestation default. Preserving the existing code
system as the extension's `value` binding keeps already-deployed criticality
codes round-trippable.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `AllergyIntolerance.criticality` in the core resource (no breaking
change), but:

1. Re-tag the element in the resource definition to soften it (note that it
   is rarely populated, refer to the rationale extension below for guidance,
   and document the substance/manifestation defaults).
2. Define a sibling **`allergyintolerance-criticality-rationale`** standard
   extension on `AllergyIntolerance` (0..1, `CodeableConcept`, bound to the
   same new rationale value set proposed in Disposition A).
3. Add an invariant or maturity guidance recommending that whenever
   `criticality` is present **and** diverges from the substance/manifestation
   default, the rationale extension SHOULD be populated.
4. Update the v2→FHIR concept map to stop routing AL1.4 to criticality and
   restrict it to `reaction.severity`, regardless of any "if severity was used
   equivalent to criticality" caveat.

#### Justification

This addresses the reporter's substantive complaint — confusion in mapping and
the absence of evidence for asserted criticality — without a breaking change
to a normative element. R5 implementers, US Core, AU Core profiles, IHE
content profiles and CDA-on-FHIR transforms all reference
`AllergyIntolerance.criticality` directly; removing the element forces every
downstream IG to re‑author allergy profiles, regenerate examples, and re‑bind
extensions, which is a cost that should not be paid lightly when a softer
intervention exists. It also respects Russell Leftwich's argument that
criticality, when correctly used, encodes information that severity alone
cannot — the clinical risk on re‑exposure — and keeps the element available to
the systems Heather Leslie's group might one day choose to capture it.
Trade‑off vs. A: B does not reduce documentation burden because it does not
remove the element from the 80/20 calculus; it only mitigates misuse.

---

### Disposition C: Decline

#### Proposal

Decline the request as written. Rationale: criticality and reaction severity
are distinct, deliberately modeled clinical concepts; removing criticality
from the core resource is a normative breaking change to a long-standing
element that downstream IGs (US Core, AU Core, IPS, IHE PCC, CDA‑on‑FHIR maps,
v2→FHIR maps) all bind to. The reported confusion is fundamentally a
mapping/source-data problem (v2 AL1.4 does not have a criticality field),
which is more appropriately addressed by tightening the v2→FHIR concept map
and by clarifying the resource's element documentation, not by removing the
element. Track the rationale-coding aspect of the request as a separate, more
narrowly scoped enhancement (a new sibling extension carrying a rationale
code) without disturbing the existing element.

#### Justification

Removing or demoting a normative element on a widely-implemented core resource
is a high-cost change that should require evidence the element causes harm in
the field, not merely that it is under-used. Russell Leftwich's argument that
the criticality/severity distinction matters clinically is on the record and
was not refuted in the Zulip thread, only counter-balanced by adoption data.
Where criticality codes are misapplied (e.g., Grant Smith's v2→FHIR mapping
yielding "high" criticality from "moderate" severity), the defect is in the
concept map, not in the core resource: AL1.4 should never have been allowed to
target criticality, and the v2→FHIR maintainers can fix that without Patient
Care touching the core spec. A rationale extension is a low-cost win that
should be pursued independently.

---

### Recommendation

**Recommended disposition:** B

Disposition B captures the substance of the reporter's request — getting
rationale onto every meaningful criticality assertion and unblocking v2→FHIR
mappers — without taking on the disproportionate downstream cost of removing
a normative element from `AllergyIntolerance`. The Zulip record is genuinely
mixed: clinicians and openEHR/AU Sparked back the reporter, but Russell
Leftwich (a domain expert who authored the original model) and Lloyd
McKenzie's "extension until practice changes" rule both apply *prospectively*;
they don't compel removing an element that has already shipped normatively
and is referenced by US Core, AU Core, IPS, and several IHE/CDA-on-FHIR
mappings. B also lets Patient Care fix the actual proximate harm (v2→FHIR
AL1.4 routing to criticality) immediately by patching the concept map, while
the new rationale extension and softened guidance signal to implementers that
criticality should be used sparingly and with evidence. If the workgroup
accepts the principle that 80/20 alone justifies demotion of normative
elements, escalate to Disposition A — but in that case the workgroup should
plan a coordinated migration with US Core and AU Core, not just edit the core
resource.
