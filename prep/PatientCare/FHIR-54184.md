## Summary

The ticket asks Patient Care to extend the choice list for
`FamilyMemberHistory.condition.onset[x]` by adding a `dateTime` (i.e.
`onsetDateTime`) option alongside the current `Age`, `Range`, `Period`, and
`string` choices. The reporter, Stephen Chu, frames the request from a
public-health use case in which `FamilyMemberHistory` is used to capture
communicable diseases of family members in contact with the subject; in that
context, the actual calendar date a relative's condition first manifested is
clinically meaningful and is not adequately represented by an age, a range, a
period, or free-text string. The cross-references surfaced for the ticket are
limited to mentions of the `FamilyMemberHistory.condition.onset[x]` element
itself; no related Jira tickets, Zulip threads, or GitHub items were returned
by the orchestrator.

## Details

**Description:**

FamilyMemberHistory.condition.onset[x] includes the following choices:
- onset[x]   0..1   When condition first manifested
  - onsetAge — Age
  - onsetRange — Range
  - onsetPeriod — Period
  - onsetString — string

FamilyMemberHistory is useful from the Public Health context for capturing
communicable diseases of family members in contact with the subject. Knowledge
of onsetDate is clinically useful.

Suggest adding: onsetDate to the choice list.

## Proposed Dispositions

The dispositions below name `FamilyMemberHistory` and the
`FamilyMemberHistory.condition.onset[x]` element specifically; because no
GitHub cross-references were surfaced, a per-repo briefing was not loaded for
this ticket and detailed repo paths are not asserted here. Reviewers should
expect any accepted change to land in the FHIR Core authoring source for the
`FamilyMemberHistory` resource (resource definition, structure-definition
generation inputs, and the resource intro/notes pages).

### Disposition A: Accept As Requested

#### Proposal

Add `dateTime` to the type choices of `FamilyMemberHistory.condition.onset[x]`
so that the element becomes:

- `onsetAge` — Age
- `onsetRange` — Range
- `onsetPeriod` — Period
- `onsetString` — string
- **`onsetDateTime` — dateTime** *(new)*

The change is additive: the `onset[x]` cardinality (0..1) and definition
("When condition first manifested") remain unchanged, the existing four
choices are preserved, and a `dateTime` option is appended. Update the
`FamilyMemberHistory` resource notes / element definition to mention the new
choice and to describe the public-health / contact-tracing use case that
motivates representing a known calendar onset date for a relative's condition.
No constraints, invariants, search parameters, or operations need to change,
since `dateTime` is already a permitted variant of the `*[x]` polymorphic
machinery used elsewhere in FHIR (e.g. `Condition.onset[x]`,
`AllergyIntolerance.onset[x]`).

#### Justification

Adding `dateTime` is consistent with the analogous
`Condition.onset[x]` and `AllergyIntolerance.onset[x]` elements, both of which
already include a `dateTime` choice alongside `Age`, `Period`, `Range`, and
`string`. The current `FamilyMemberHistory.condition.onset[x]` choice list is
therefore an outlier within Patient Care's onset modeling. The public-health
contact-tracing scenario described by the reporter (knowing the calendar date
a relative's communicable disease first manifested) cannot be represented
accurately by `Age` (granularity / unknown DOB), `Period` (implies a window
with start *and* end), `Range` (numeric range, not a calendar value), or
`string` (not computable). Because `dateTime` already supports partial
precisions (year, year-month, full date, full instant), it covers both the
"only the year is known" and "exact date is known" cases without further
modeling. The change is purely additive and therefore non-breaking for
existing instances and implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding `dateTime` to the core choice list, document an
extension-based pattern for capturing a calendar onset date on
`FamilyMemberHistory.condition.onset[x]` and publish a standard extension
(e.g. `condition-familyMemberHistory-onsetDateTime` or reuse of an existing
"onset-as-date" extension) that implementers can attach when a calendar value
is known. Update the `FamilyMemberHistory` resource notes to point to this
extension and to explain the public-health use case.

Optionally — if the workgroup wants to keep the surface API simple — broaden
this disposition to also align the type list across all "onset"-style
elements in Patient Care (i.e. confirm that `Condition`, `AllergyIntolerance`,
`Observation`, and `FamilyMemberHistory` all expose the same set of onset
choices) and handle them as a coordinated set under the
`OnsetDiagnosisAbatementDates` label rather than as a one-off addition here.

#### Justification

An extension avoids any change to the core resource's polymorphic type list,
which can be sensitive in normative contexts and can ripple into validators,
profiles, and code generators that enumerate the `*[x]` choices. It also
gives the workgroup time to handle the request as part of the broader
`OnsetDiagnosisAbatementDates` campaign so that
`Condition.onset[x]`, `AllergyIntolerance.onset[x]`,
`FamilyMemberHistory.condition.onset[x]`, and any abatement / diagnosis-date
counterparts are tightened together rather than drifting further apart.
Trade-off vs. Disposition A: an extension is less ergonomic for implementers
than a first-class choice and is harder to discover; for a clinically obvious
data point like "date the relative's condition started", a standard extension
is arguably underpowered.

---

### Disposition C: Decline

#### Proposal

Decline the request and direct implementers who need a calendar onset date
for a family member's condition to either (a) approximate it via
`onsetPeriod` (with `start` set to the known date and `end` omitted), (b)
record it in `onsetString`, or (c) carry the value on a separate `Condition`
resource for the family member that does already support `onsetDateTime`.

#### Justification

A decline is defensible only on the grounds that `FamilyMemberHistory` is a
summary, recall-based resource and that the existing `Period`, `Age`, and
`string` options already cover the spectrum from precise to imprecise
recollection. Adding another choice grows the polymorphic surface and the
combinatorial space implementers must validate, with limited incremental
clinical value beyond what `onsetPeriod` (start-only) and `onsetString` can
already express.

---

### Recommendation

**Recommended disposition:** A

Adding `dateTime` to `FamilyMemberHistory.condition.onset[x]` is the cleanest
and most consistent fix. The sibling onset elements on `Condition` and
`AllergyIntolerance` already include `dateTime`, so this change aligns
`FamilyMemberHistory` with the rest of the Patient Care onset pattern rather
than introducing a new mechanism. The change is additive and non-breaking,
the public-health contact-tracing use case is compelling and not well served
by `Period` (which implies an end), `Age` (which often cannot be derived for
a relative), or `string` (which is not computable). An extension-based
alternative (Disposition B) solves the same problem with strictly more
implementer friction, and a decline (Disposition C) leaves
`FamilyMemberHistory` as an inconsistent outlier within the
onset/diagnosis/abatement family of elements that the `OnsetDiagnosisAbatementDates`
label is tracking. Workgroup should consider sweeping any peer tickets under
that label at the same review to keep the onset modeling coherent across
Patient Care resources.
