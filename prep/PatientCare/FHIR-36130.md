## Summary

Reporter Richard Townley-O'Neill flags that the `observation-visual-acuity`
value set in the FHIR Eyecare IG (HL7/fhir-eyecare-ig) is too narrow because
it is composed using a SNOMED CT filter of `concept descendent-of 363983007
(Visual acuity)`. As written, the value set excludes the root concept itself
(`363983007 |Visual acuity|`), even though that concept is a perfectly valid
code to bind to `Observation.code` for visual-acuity observations. The
reporter requests that the `compose.include.filter.op` be changed from
`descendent-of` to `is-a` so the root concept is included alongside its
descendants. No related Jira tickets, Zulip threads, or GitHub items were
surfaced via cross-references; the only cross-reference recorded is a FHIR
artifact mention of `Observation.code`.

## Details

**Description:**

The valueset
[ValueSet-observation-visual-acuity](https://build.fhir.org/ig/HL7/fhir-eyecare-ig/ValueSet-observation-visual-acuity.html)
is defined as:

- Include codes from `http://snomed.info/sct` where concept descends from
  `363983007` (Visual acuity)

It should include the root of this tree, `363983007 |Visual acuity|`.

Make `compose.include.filter.op = 'is-a'` instead of `'descendent-of'`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the FHIR Eyecare IG (`HL7/fhir-eyecare-ig`), modify the
`observation-visual-acuity` ValueSet so that its single
`compose.include` block uses the SNOMED CT filter operator `is-a`
instead of `descendent-of` against concept `363983007 |Visual acuity
(observable entity)|`. Concretely, the `compose.include.filter` entry
changes from:

```json
{ "property": "concept", "op": "descendent-of", "value": "363983007" }
```

to:

```json
{ "property": "concept", "op": "is-a", "value": "363983007" }
```

This adds the root concept `363983007 |Visual acuity|` to the value set
while keeping every descendant currently included. Regenerate the IG
expansions and update the published HTML. No structural change is
required to `Observation.code`'s binding — only the value set definition
is touched.

#### Justification

The reporter's underlying point is correct under standard SNOMED CT and
FHIR ValueSet semantics: `descendent-of` excludes the seed concept,
while `is-a` includes it. `363983007 |Visual acuity|` is itself a
clinically valid code for an eye-care visual-acuity observation; there
is no good reason for the value set to forbid it while admitting more
specific descendants. The change is local to one value set in the
Eyecare IG, is mechanically simple, and aligns with the pattern used by
many other SNOMED-based FHIR value sets that intentionally include the
root.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing `descendent-of` filter and add a second
`compose.include` entry that lists `363983007 |Visual acuity|` as an
explicit `concept` (or, equivalently, add a parallel `compose.include`
with `op = "="` / `op = "is-a"` scoped only to that single root
concept). The net membership of the value set becomes the same as
Disposition A, but the intent — "all proper descendants, plus the
explicitly named root" — is visible to readers of the source.

#### Justification

This option preserves the original author's intent of a tree-walk
beginning below the root, while still admitting the root concept the
reporter wants. It is useful if the work group wants the ValueSet
source to make the inclusion of the root explicit (for review and
auditability) rather than relying on readers to know the difference
between `is-a` and `descendent-of`. The trade-off is a slightly more
verbose `compose` and one extra include block to maintain. Functionally
it is equivalent to Disposition A from an expansion perspective; the
choice is editorial.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the value set as currently defined
(`descendent-of 363983007`). Document in the IG (or in a ticket
response) that `363983007 |Visual acuity|` is intentionally excluded
because the value set is meant to bind `Observation.code` only to a
*specific* visual-acuity measurement (e.g., distance, near, corrected),
not to the generic top-level concept.

#### Justification

This disposition is defensible only if the Eyecare work group can
articulate a clinical reason that a generic "Visual acuity"
`Observation.code` is not useful — for example, if downstream profiles
rely on always being able to discriminate the subtype of visual-acuity
measurement. Absent such a stated requirement, declining penalises
implementers who legitimately need to record a visual-acuity
observation without committing to a more specific subtype, and forces
them either to invent a workaround code or to use an extension. The
"breaking-change" risk of accepting is essentially zero, since the
value set is being broadened, not narrowed.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a straightforward terminology-binding
defect: the current `descendent-of` filter excludes a clinically valid
root concept that implementers may reasonably need. Switching to `is-a`
is a one-line change to a single value set in `HL7/fhir-eyecare-ig`,
broadens (rather than breaks) existing bindings, and aligns the value
set with common practice for SNOMED-rooted FHIR value sets.
Disposition B achieves the same outcome but adds editorial overhead
without functional benefit; it should only be preferred if Patient Care
explicitly wants the root inclusion called out in source. Disposition C
should be chosen only if the work group can point to a concrete
clinical reason to forbid the root concept, which has not been
established in this ticket or in any surfaced discussion.
