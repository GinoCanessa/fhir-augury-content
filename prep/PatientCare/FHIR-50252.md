## Summary

The ticket asks the Gravity SDOH Clinical Care (SDOHCC) IG to update the
parent profiles of a defined set of SDOHCC profiles so that, where
appropriate, they derive from the corresponding US Core 7.0.0 profile,
and to add or update references that are required to point at US Core
profiles. The five SDOHCC profiles named for re‑parenting are SDOHCC
Goal, SDOHCC Location, SDOHCC Observation Assessment (which would derive
from US Core Simple Observation), SDOHCC Observation Screening Response
(which would derive from US Core Screening Assessment), and SDOHCC
ServiceRequest. The change is scoped to the SDOHCC IG only; it is part
of the Gravity team's broader STU2.3 alignment with US Core 3.1.1, 6.1.0
and 7.0.0 (announced on Zulip by the same reporter, Aaron Nusstein, in
the `Gravity sdoh-cc` stream). No cross‑referenced Jira tickets, GitHub
PRs/issues, or other Jira tickets were surfaced for FHIR‑50252 in the
FhirAugury cross‑reference index.

## Details

**Description:**

The following SDOHCC profiles will have their parent profile be updated
to the associated US Core 7.0.0 profile. If needed, references that
require a US Core profile will be added/updated to the SDOHCC profile:

- SDOHCC Goal
- SDOHCC Location
- SDOHCC Observation Assessment (US Core Simple Assessment)
- SDOHCC Observation Screening Response (US Core Screening Assessment)
- SDOHCC ServiceRequest

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Within the US SDOH Clinical Care (Gravity SDOHCC) IG, update the
`baseDefinition` of the five named profiles to point at the US Core
7.0.0 canonical, and add or tighten the corresponding profile-bound
references on the SDOHCC profiles:

- SDOHCC Goal → `baseDefinition` =
  `http://hl7.org/fhir/us/core/StructureDefinition/us-core-goal|7.0.0`,
  with reference targets on `subject` constrained to US Core Patient.
- SDOHCC Location → `baseDefinition` =
  `http://hl7.org/fhir/us/core/StructureDefinition/us-core-location|7.0.0`.
- SDOHCC Observation Assessment → `baseDefinition` =
  `http://hl7.org/fhir/us/core/StructureDefinition/us-core-simple-observation|7.0.0`,
  with `subject` and `performer` constrained to the matching US Core
  Patient / Practitioner / PractitionerRole / CareTeam / Organization
  profiles.
- SDOHCC Observation Screening Response → `baseDefinition` =
  `http://hl7.org/fhir/us/core/StructureDefinition/us-core-observation-screening-assessment|7.0.0`,
  with `subject`, `performer`, `derivedFrom` and `hasMember` reference
  targets aligned to the US Core 7.0.0 screening-assessment expectations.
- SDOHCC ServiceRequest → `baseDefinition` =
  `http://hl7.org/fhir/us/core/StructureDefinition/us-core-servicerequest|7.0.0`,
  with `subject`, `requester`, and `performer` reference targets bound
  to the corresponding US Core profiles.

Bump the `dependsOn` declaration for `hl7.fhir.us.core` in the IG
manifest to `7.0.0` (or add it as an additional supported version,
consistent with the multi-version strategy described in the Zulip
announcement) and refresh the IG's must-support / invariants test
fixtures accordingly.

#### Justification

This is exactly what the reporter and the Gravity team announced on
Zulip: align the SDOHCC profiles with US Core 7.0.0 so SDOHCC content
can be exchanged in environments standardised on the most recent US
Core. Re-parenting (rather than reinventing constraints) keeps SDOHCC
in lockstep with US Core's evolving must-support, invariants and
binding strengths and is the conventional FHIR approach for
realm-specific IGs that build on US Core. Doing it for the precise set
of profiles named in the ticket — and only those — keeps the change
auditable and limits regression risk for the rest of the IG.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than re-parent the SDOHCC profiles directly to a single US Core
7.0.0 version, publish the SDOHCC profiles against US Core 7.0.0 as the
default baseline but keep the existing US Core 3.1.1 / 6.1.0 derivations
available via parallel profile variants (e.g.,
`SDOHCC-Goal`, `SDOHCC-Goal-USCore3`, `SDOHCC-Goal-USCore6`) or via a
companion "compatibility" IG package. The base IG would declare US Core
7.0.0 as its primary `dependsOn`; the older versions would be expressed
as additional package dependencies referenced only by the variant
profiles.

#### Justification

The Zulip announcement explicitly states that the Gravity team intends
SDOHCC STU2.3 to be compatible with US Core 3.1.1, 6.1.0 *and* 7.0.0.
A hard re-parent to 7.0.0 alone (Disposition A) makes it harder for
implementers still pinned to 3.1.1 / 6.1.0 — typically those tied to
specific CMS / ONC certification cycles — to keep using SDOHCC without
forking. Variants make the multi-version intent first-class in the
artifacts instead of relying on prose, at the cost of more profiles to
publish and test, more must-support divergence to track, and additional
authoring complexity in the IG sushi config.

---

### Disposition C: Decline

#### Proposal

Do not adopt the parent re-pointing in this ticket. Keep SDOHCC profiles
on their current FHIR base resources and document US Core 7.0.0 as a
non-normative compatibility statement, leaving any tighter US Core
binding to implementers' realm guides or to a future major SDOHCC
revision aligned to a single US Core version.

#### Justification

Re-parenting onto US Core forces every SDOHCC implementer — including
those outside the US realm or those who consume SDOHCC alongside a
non-US-Core-conformant Patient/Practitioner stack — to also conform to
US Core's must-support and invariant set, which is a meaningfully
broader contract than today's SDOHCC. If the workgroup believes
SDOHCC's value is in being a domain (SDOH) IG that can sit on top of
plain core FHIR (or any national base), and that US Core alignment is
better expressed as conformance guidance rather than profile lineage,
declining is defensible. This option is the weakest because it
contradicts the publicly announced direction of the Gravity STU2.3
publication, but it is included to make the cost of the lineage choice
explicit.

---

### Recommendation

**Recommended disposition:** A

The ticket is a narrow, intentional re-parent of five named profiles
inside an IG whose stewards have already announced — on Zulip, in the
public peer-review window — that aligning with US Core (including 7.0.0)
is the explicit goal of STU2.3. Disposition A delivers exactly that with
the smallest possible surface area: five profile `baseDefinition`
changes plus the reference tightening that the description mentions,
nothing more. Disposition B's variant strategy is technically defensible
if the workgroup wants the multi-version compatibility goal to be
first-class in the artifacts, but it materially expands authoring,
publication and conformance-testing burden for the same implementer
benefit, and the Gravity team has indicated the existing single-profile
approach is sufficient. Disposition C should be discussed only if the
workgroup wants to revisit whether SDOHCC should depend on US Core at
all — that is a strategic question well beyond the scope of this
change request, and resolving FHIR‑50252 should not be blocked on it.
