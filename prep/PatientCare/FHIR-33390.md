## Summary

Ricardo Quintano requests a structured way to record parameters (e.g.,
amount, volume, gauge size, and other characteristics) of the focal
devices manipulated or implanted during a procedure. Today
`Procedure.focalDevice` only carries `action` and `manipulated`, with no
slot for the device-specific settings or measurements that were applied
when the action took place. The reporter proposes adding a repeating
`focalDevice.parameter` BackboneElement with `code 1..1 CodeableConcept`
and `value[x] 1..1` (Quantity | Ratio | Range | boolean |
CodeableConcept | string | Period), and aligning `focalDevice.manipulated`
with the new CodeableReference(Device | DeviceDefinition) shape — the
same pattern applied to the request side in
[FHIR-31584](https://jira.hl7.org/browse/FHIR-31584).

The ticket sits inside a closely-coupled cluster. Its sibling
[FHIR-33391](https://jira.hl7.org/browse/FHIR-33391) (Patient Care,
Resolved – Not Persuasive) asked for a generic `Procedure.details`
parameter list and was declined with the rationale that procedural
parameter detail belongs in `Procedure.report`/notes. The Orders &
Observations counterpart
[FHIR-32265](https://jira.hl7.org/browse/FHIR-32265) (Applied,
Persuasive with Modification) added a structured
`ServiceRequest.orderDetail` BackboneElement with `parameterFocus
0..1 CodeableReference(...)`, `parameter 1..*`, `parameter.code 1..1`,
and `parameter.value[x] 1..1` to express the ordered settings of
device-bearing services such as mechanical ventilation. The latest
status note on this ticket — "CONSIDERED (pending OO coordination on
ServiceRequest.orderDetail)" — and the Sept-2024 WGM label make clear
that PC has parked this ticket waiting for a joint OO/PC discussion so
that the *performed* side (Procedure.focalDevice.parameter) is modeled
consistently with the *requested* side already in
ServiceRequest.orderDetail.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a `parameter` BackboneElement to `Procedure.focalDevice` in FHIR
Core, mirroring the OO resolution applied for
`ServiceRequest.orderDetail` in FHIR-32265:

- `Procedure.focalDevice.parameter 0..*` BackboneElement — "Parameters
  recorded for the focal device as it was used in the procedure."
- `Procedure.focalDevice.parameter.code 1..1 CodeableConcept` — what
  the parameter is (e.g., gauge size, tidal volume, FiO₂, insertion
  depth).
- `Procedure.focalDevice.parameter.value[x] 1..1` with the same type
  set as `ServiceRequest.orderDetail.parameter.value[x]`: `Quantity |
  Ratio | Range | boolean | CodeableConcept | string | Period`.

Also align `Procedure.focalDevice.manipulated` with the modern
device-handle pattern by retyping it as `CodeableReference(Device |
DeviceDefinition) 0..1` (the reporter's table calls this out
explicitly), so that catalog-level (DeviceDefinition) and
instance-level (Device) entries can both be recorded.

Inherit FHIR-32265's pink-ribbon implementer note verbatim — "the
structure is subject to further feedback based on use; modeling
creates a challenge when there is no focus, while an alternate
structure would yield a requirement to repeat the focus attribute on
each code/value pair" — so the performed-side and requested-side
notes stay symmetric.

#### Justification

This is the option the reporter asked for, and it is the option the
ticket's own history converges on: PC has explicitly held the ticket
open ("CONSIDERED, pending OO coordination on
ServiceRequest.orderDetail") so that Procedure mirrors what OO did on
ServiceRequest. The parent–child symmetry between
`ServiceRequest.orderDetail.parameter[Focus]` and a new
`Procedure.focalDevice.parameter` keeps round-tripping from order to
result coherent and preserves the design choices already accepted by
the community for the requested side. It also addresses real
documented gaps (mechanical ventilation, infusion settings, gauge
size, insertion depth) that the existing
`focalDevice.action`/`manipulated` cannot carry. The pink-ribbon
implementer-note convention from FHIR-32265 explicitly invites the
exact downstream feedback OO/PC anticipated, so accepting now does not
foreclose later refinement.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of adding a new BackboneElement under `focalDevice`, lift the
parameter pattern up one level on Procedure as a generalized "applied
detail" element with an explicit focus, modeled directly on the OO
shape that already exists:

- `Procedure.orderDetail 0..*` (rename to `Procedure.appliedDetail` if
  PC prefers — name TBD) BackboneElement.
- `Procedure.appliedDetail.parameterFocus 0..1
  CodeableReference(Device | DeviceDefinition | Medication |
  BiologicallyDerivedProduct | Substance)` — points at the focal
  device (or other focus) the parameters apply to.
- `Procedure.appliedDetail.parameter 1..*` BackboneElement with
  `code 1..1 CodeableConcept` and `value[x] 1..1` (same type set as
  ServiceRequest.orderDetail).
- Add an invariant that, when `parameterFocus` references a Device,
  that Device SHALL also appear in `Procedure.focalDevice.manipulated`
  (or a `Procedure.used` slot) so the focal device is not re-listed
  inline.

This matches OO's pattern 1:1, keeps `focalDevice` focused on
"action + which device" (its current contract), and gives PC a place
to record parameters whose focus is *not* a device (the FHIR-33391
use case — incision length, dressing pack-method, ET tube insertion
depth, patient positioning) without reopening 33391's
`Procedure.details` debate.

#### Justification

This option directly answers the structural concern Rob Hausam raised
on the 2023-08-10 PC call — having a 0..* `parameter` nested inside a
0..* `focalDevice` makes "no-focus" modeling awkward and forces the
focus attribute to be repeated on every code/value pair if you flip
the nesting. By promoting parameters to a top-level BackboneElement
with an explicit `parameterFocus`, the model is symmetrical with
ServiceRequest.orderDetail (so request → performance round-tripping is
trivially mappable), avoids two levels of unbounded nesting under
`focalDevice`, and provides a single landing spot that can absorb
both the device-parameter use cases here and the broader procedural
parameter use cases that motivated FHIR-33391 — without re-litigating
33391's specific `Procedure.details` proposal. The trade-off vs. A is
that this is a larger change (new top-level Procedure element) and
may surprise implementers expecting parameters to live with the
device they are about; the `parameterFocus` pointer + invariant
mitigate that.

---

### Disposition C: Decline

#### Proposal

Resolve **Not Persuasive** for the same reason FHIR-33391 was
declined: device parameters used during a procedure are part of the
operative narrative and should be captured in `Procedure.report` (or
in a Device/DeviceMetric-bearing Observation linked from
`Procedure.report`) rather than added as structured fields on
`Procedure.focalDevice`. Where structured device settings are
required at *order* time, implementers already have
`ServiceRequest.orderDetail.parameter`/`parameterFocus` from
FHIR-32265 and can carry the as-applied values via Observations
referencing the same Device. Add a one-line implementer note on
`Procedure.focalDevice` pointing at this guidance.

#### Justification

PC has already declined the structurally adjacent FHIR-33391 on
exactly this rationale (parameter detail belongs in the note /
Procedure.report, not the Procedure record), and accepting A or B
without revisiting 33391 risks an inconsistent PC stance: device
parameters get a structured slot, but other procedural parameters
(incision length, positioning, ET tube depth) do not. Declining
preserves Procedure's current scope, keeps the resource lean, and
relies on existing well-understood mechanisms (Observation referencing
Device, Procedure.report). The downside is that as-performed
device-setting data (e.g., the actual ventilator settings used) ends
up in narrative or in loosely coupled Observations rather than
travelling with the Procedure itself, which is what implementers like
the reporter have repeatedly said they want.

---

### Recommendation

**Recommended disposition:** A

A is the option the ticket's whole history points at. PC explicitly
held this ticket "CONSIDERED, pending OO coordination on
ServiceRequest.orderDetail" precisely so that, once OO landed
FHIR-32265, PC could mirror it on `Procedure.focalDevice` — and the
Sept-2024 WGM joint session was scheduled for that purpose. Accepting
the reporter's structure (with the FHIR-32265-style implementer note)
gives a same-shape modeling on the request and performance sides,
which is the highest-value behavior for implementers exchanging
device-bearing orders and resulting Procedures.

B is genuinely tempting because it answers Rob Hausam's "no-focus /
repeated-focus" concern more cleanly and would gracefully absorb some
of the FHIR-33391 use cases. However, it is a materially larger
change to Procedure and would likely require re-balloting
`Procedure.focalDevice` consumers; if PC wants to head in that
direction, the cleaner path is to accept A now (so the
device-parameter gap is closed) and file a follow-on ticket to
generalize the pattern in a future cycle.

C should be rejected: PC's prior "report/notes is enough" stance on
FHIR-33391 has been overtaken by OO's adoption of structured
parameters on the request side. Forcing only the *performed* side
back into narrative would create an asymmetric round-trip and
contradict the joint OO/PC alignment that the current "CONSIDERED"
state is explicitly waiting on.
