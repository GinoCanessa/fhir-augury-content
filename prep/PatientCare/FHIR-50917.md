## Summary

The PACIO Transitions of Care (ToC) Implementation Guide currently scopes
its data elements to those catalogued in the CMS Data Element Library
(DEL). May Terry, on behalf of Terrence O'Malley and James Patterson,
asks the Patient Care work group to provide further guidance on how the
IG should support and represent clinically relevant elements that are
**not** present in the DEL — i.e., elements needed for transitions of
care that fall outside the DEL's current catalog. The ticket does not
yet propose a specific representation; it asks the work group to decide
the IG's policy and authoring pattern for such "non-DEL" elements
(e.g., whether they are added to existing PACIO ToC profiles, carried
via extensions, or surfaced through a companion vocabulary). No
cross-referenced Jira tickets, Zulip threads, or GitHub artifacts were
found via the augury index.

### Disposition A: Accept As Requested

#### Proposal

Add an explicit "Non-DEL Elements" section to the PACIO Transitions of
Care IG narrative that:

1. Defines the criteria a data element must meet to be included in the
   IG when it is **not** present in the CMS DEL (e.g., demonstrated
   transitions-of-care use case, sponsor, mapping target).
2. Specifies the authoring pattern: such elements are represented as
   first-class slices/elements on the existing PACIO ToC profiles
   (e.g., the assessment Observation profile family used by the IG) so
   they are indistinguishable to consumers from DEL-sourced elements.
3. Requires each non-DEL element to carry a provenance marker — either
   an `Observation.derivedFrom` reference, a code from a "source
   catalog" CodeSystem, or an extension such as
   `data-element-source` — so implementers can tell DEL vs. non-DEL
   elements apart at runtime.
4. Seeds the section with the specific elements Terrence O'Malley and
   James Patterson have flagged (to be enumerated during the work
   group call) so the guidance lands with concrete examples.

#### Justification

The ticket explicitly asks for *guidance on supporting / representing*
non-DEL elements; producing dedicated narrative plus an authoring
pattern is the most direct response. Treating non-DEL elements as
first-class on the existing profiles avoids fragmenting the IG into
"core" and "extras" surfaces, which is consistent with how other HL7
IGs (e.g., US Core) absorb elements that originate outside their
nominal source vocabulary. A provenance marker preserves traceability
back to the DEL boundary, which is the property reviewers cared about
when the DEL was originally chosen as the scoping mechanism.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than embedding non-DEL elements in the core PACIO ToC profiles,
publish them through a **companion extension pack** in the same IG:

- Define a small set of FHIR extensions (one per non-DEL element, or
  one parameterised extension keyed by a `data-element` code) that can
  be applied to the existing PACIO ToC Observation / Condition /
  whatever profiles.
- Maintain the DEL-derived element set as the IG's normative content
  and present the extensions as "supplementary elements supported by
  PACIO ToC but not yet in the DEL."
- Provide a CapabilityStatement note describing how a server SHALL or
  SHOULD advertise support for the extensions.

#### Justification

This keeps the DEL boundary crisp inside the profile definitions and
treats non-DEL content as additive, which lowers the bar for
implementers who only target DEL-aligned content. It also lets the IG
ship now without waiting for the DEL to absorb the missing elements:
when an element is later added to the DEL, the corresponding extension
can be deprecated and the element promoted into the profile without a
breaking change to existing extension users (the extension can be
retained as a no-op alias). Trade-off vs. Disposition A: implementers
have to opt in to two surfaces (profile + extension pack), which is
mildly more complex but better signals "this element is provisional
relative to the DEL."

---

### Disposition C: Decline

#### Proposal

Decline the request as out of scope for the IG itself, on the basis
that the gap is in the **DEL**, not in PACIO ToC. Recommend instead
that May Terry, Terrence O'Malley, and James Patterson submit the
missing elements to CMS for inclusion in the DEL, and that the PACIO
ToC IG continue to scope its content strictly to DEL-listed elements
until the DEL catches up.

#### Justification

Defensible if the work group concludes that the IG's value
proposition rests on a 1:1 correspondence with the DEL, and that
introducing non-DEL elements (whether on profiles or via extensions)
would erode that contract and create downstream confusion for
implementers who chose the IG specifically because of the DEL
boundary. The cost is real: the elements stay un-representable in
PACIO ToC FHIR exchanges until the DEL adds them, which is the very
problem the ticket is trying to solve.

---

### Recommendation

**Recommended disposition:** B

Disposition B (companion extension pack) gives the work group a way to
say "yes" to Terrence O'Malley's and James Patterson's request without
muddying the DEL alignment that the IG was built around. The non-DEL
elements become real, exchangeable FHIR content immediately, but the
profile cores keep their current DEL-scoped definitions, which
preserves the implementer contract that the IG has already shipped.
Disposition A is the most user-friendly but rewrites the IG's scoping
story in a way that may surprise existing adopters; Disposition C
preserves the scoping story but leaves the underlying clinical gap
unaddressed for as long as DEL governance takes to catch up — which is
exactly the situation the reporters are trying to escape. The work
group should confirm Disposition B by enumerating the specific
non-DEL elements in scope on the next call so the extension surface
can be sized before any spec work begins.
