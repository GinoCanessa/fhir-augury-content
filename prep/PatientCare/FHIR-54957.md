## Summary

The reporter requests that `Procedure.bodySite` (a `CodeableConcept`) be
deprecated and that `Procedure.bodyStructure` (currently a
`Reference(BodyStructure)`) be promoted to a `CodeableReference(BodyStructure)`
so that Procedure follows the same pattern that other FHIR resources have
already adopted for body-site capture. The argument is consistency: several
resources (e.g., `Observation.bodyStructure`,
`ServiceRequest.bodyStructure`, `ObservationDefinition.bodyStructure`,
`DocumentReference.bodyStructure`, `DeviceAssociation.bodyStructure`) have
either been migrated to `CodeableReference(BodyStructure)` or are headed
that way, and Procedure is now an outlier with two parallel (and
mutually-exclusive) elements.

This ticket is closely linked to a cluster of Patient Care / Orders &
Observations tickets filed in the same window:

- [FHIR-54992](https://jira.hl7.org/browse/FHIR-54992) — same proposal, framed as
  "combine `Procedure.bodySite` and `Procedure.bodyStructure` into a single
  `CodeableReference` element" (Ron G. Parker).
- [FHIR-54703](https://jira.hl7.org/browse/FHIR-54703) — explicitly asks to
  harmonize `Procedure.bodyStructure` with `Observation.bodyStructure`
  (which is already `CodeableReference(BodyStructure)`), including
  deprecating `Procedure.bodySite` and adding the same binding.
- [FHIR-54614](https://jira.hl7.org/browse/FHIR-54614) — flags the existing
  `prc-1` constraint ("`bodyStructure` SHALL only be present if
  `Procedure.bodySite` is not present") as overly restrictive and proposes
  collapsing the two into a single `codeableReference` element with
  guidance.
- [FHIR-54843](https://jira.hl7.org/browse/FHIR-54843) (Orders & Observations) —
  surveys all `bodySite`/`bodyStructure` element pairs across resources and
  identifies Procedure (alongside Observation and Condition) as still
  carrying both elements.
- [FHIR-50355](https://jira.hl7.org/browse/FHIR-50355) — the analogous
  ServiceRequest ticket, already **Applied** with resolution
  *Persuasive with Modification*; sets the precedent for migrating
  resources to a single `CodeableReference(BodyStructure)` element.

Taken together, the request is one of several converging asks to finish
the pattern migration across `bodyStructure`-bearing resources, with
Procedure being the most prominent remaining outlier alongside Condition.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In FHIR Core, on the `Procedure` resource:

1. Mark `Procedure.bodySite` (`0..* CodeableConcept`) as **deprecated** in
   the next version, retaining the element for backward compatibility but
   discouraging new use.
2. Change the type of `Procedure.bodyStructure` from
   `Reference(BodyStructure)` to `CodeableReference(BodyStructure)`,
   adopting the same value-set binding used by
   `Observation.bodyStructure` (SNOMED CT Body Structures, per the
   harmonization noted in FHIR-54703).
3. Update or retire the `prc-1` invariant so that the deprecated
   `bodySite` and the new `bodyStructure` are not in mutual conflict
   during the deprecation window; clarify in the element definition that
   new content should populate `bodyStructure` only.
4. Update the element documentation to direct implementers to the new
   `bodyStructure` element and explain the migration path.
5. Coordinate with the maintainers of the
   `bodySite` and `procedure-targetBodyStructure` extensions to remove
   `Procedure` from their context (per FHIR-54614), because the new
   `CodeableReference` makes them redundant.

This change should be dispositioned **jointly with FHIR-54992 and
FHIR-54703**, all of which ask for substantively the same outcome, and
should be coordinated with FHIR-54614 (which targets the `prc-1`
constraint) and FHIR-54843 (which inventories sibling resources).

#### Justification

This is the consistent, already-validated pattern. ServiceRequest
([FHIR-50355](https://jira.hl7.org/browse/FHIR-50355)) was migrated to a
single `CodeableReference(BodyStructure)` and resolved
*Persuasive with Modification*; Observation has likewise moved to
`CodeableReference(BodyStructure)`. Leaving Procedure with two
mutually-exclusive elements of differing cardinality (the issue
surfaced in FHIR-54614) is both confusing for implementers and an
artificial divergence from sibling resources that capture the same
clinical concept. `CodeableReference` is exactly the FHIR datatype
designed for "code or reference, not both", which makes the `prc-1`
constraint unnecessary going forward. Keeping `bodySite` as deprecated
(rather than removing it) preserves backward compatibility for existing
implementations and profiles.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** deprecate `Procedure.bodySite`. Instead:

1. Change `Procedure.bodyStructure` to `CodeableReference(BodyStructure)`
   with a SNOMED CT Body Structures binding (the harmonization piece of
   FHIR-54703).
2. Relax `prc-1` so that `Procedure.bodySite` and
   `Procedure.bodyStructure` may co-occur, with definitional guidance
   that they must not describe the *same* anatomical site (per the
   FHIR-54614 recommendation, but kept as two elements).
3. Add normative guidance on when to use which: `bodySite` for the
   simple coded case (and the multi-site case, since
   `bodyStructure` is `0..1`), `bodyStructure` for the
   richer/post-coordinated case requiring laterality, morphology, etc.

#### Justification

This addresses the *underlying* concern (Procedure is the only resource
in its peer group with a non-`CodeableReference` `bodyStructure`)
without breaking the many existing profiles, IGs, and production systems
that already populate `Procedure.bodySite` (e.g., AU Core, PS-ON,
several EU Base profiles surfaced in the related Zulip threads). It also
preserves the multi-site capability that `bodyStructure` (`0..1`) cannot
represent today — a point FHIR-54614 raises explicitly. The cost is that
Procedure remains structurally different from Observation/ServiceRequest
and continues to carry two semantically overlapping elements, which
implementers will continue to find confusing.

---

### Disposition C: Decline

#### Proposal

Decline the deprecation of `Procedure.bodySite` and the type change on
`Procedure.bodyStructure`, on the grounds that (a) `Procedure` is widely
deployed and a type change on `bodyStructure` from
`Reference(BodyStructure)` to `CodeableReference(BodyStructure)` is a
breaking change for every consumer expecting the old type, and (b) the
existing extensions
[`bodySite`](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-bodySite.html)
and
[`procedure-targetBodyStructure`](https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-procedure-targetBodyStructure.html)
already provide the missing capability for implementers who need it.
Limit the change to clarifying guidance on when to use each element
and on the `prc-1` constraint (which can be addressed by FHIR-54614 and
FHIR-54843 alone).

#### Justification

A type change on a base-resource element is the single most disruptive
change Patient Care can make to Procedure. Existing instances and
profiles populate `Reference(BodyStructure)` and any tooling, validators,
mapping artefacts, and downstream IGs will need to be updated. If the
core need is the multi-site / overly-restrictive-constraint problem,
that can be solved with a constraint relaxation and clarifying guidance
without touching element types. The standalone extensions already let
implementers attach a `BodyStructure` reference where needed.

---

### Recommendation

**Recommended disposition:** A

The Patient Care work group should adopt Disposition A and dispose of
this ticket jointly with FHIR-54992, FHIR-54703, and (for the
constraint piece) FHIR-54614, with FHIR-54843 supplying the
cross-resource inventory. The pattern has already been validated on
ServiceRequest (FHIR-50355, *Persuasive with Modification*, Applied) and
on Observation, so harmonising Procedure is a continuation of an
established direction rather than a novel design choice; declining
would leave Procedure as the most prominent outlier in the family of
`bodyStructure`-bearing resources. The legitimate concern raised in
Disposition B about losing the multi-site capability of the existing
`bodySite` (`0..*`) is real and should be addressed by either
(i) raising the cardinality of the new `CodeableReference` element to
`0..*`, or (ii) keeping `bodySite` as **deprecated but valid** for the
multi-site case during the transition. The breaking-change concern in
Disposition C is mitigated by deprecating rather than removing
`bodySite` and by signalling the migration well in advance; deferring
the harmonisation indefinitely will only make the eventual migration
more painful as more profiles accumulate around the current shape.
