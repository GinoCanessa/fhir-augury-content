## Summary

The ticket asks the Patient Care work group to broaden
`AdverseEvent.subject` (currently `1..1 Reference(Patient | Group |
Practitioner | RelatedPerson)`) to additionally allow `CareTeam` and
`Organization`. The reporter argues that an adverse event can affect an
entire care team (e.g., an OR team whose lead aprons / RPPE were
contaminated with the COVID virus during a procedure) or an entire
department, room, or organization (e.g., a radioactive-pharmaceutical
leak contaminating a department). Allowing `CareTeam` and `Organization`
as the subject would let implementers record the event against the
affected collective entity instead of having to instantiate a separate
`AdverseEvent` per individual.

No related Jira tickets, GitHub items, or substantive Zulip threads were
surfaced via cross-references; one weakly-related Zulip message from
2019 was found by keyword search and is recorded below.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update `AdverseEvent.subject` in FHIR Core (HL7/fhir) from
`1..1 Reference(Patient | Group | Practitioner | RelatedPerson)` to
`1..1 Reference(Patient | Group | Practitioner | RelatedPerson |
CareTeam | Organization)`. This entails:

- Adding `CareTeam` and `Organization` to the `subject` element's
  allowed target profiles in the AdverseEvent StructureDefinition.
- Updating AdverseEvent narrative/notes to describe the new collective
  use cases (a CareTeam exposed to a contaminated PPE batch; an
  Organization/department subject to a radioactive contamination
  incident).
- Reviewing AdverseEvent invariants and search parameters
  (`AdverseEvent-subject`, the `subject` chain, and any `patient`
  compartment-related guidance) to confirm they still behave correctly
  when the referenced resource is not a Patient — in particular,
  ensuring that compartments and patient-scoped queries continue to
  return only Patient-rooted events.

#### Justification

The reporter's two scenarios (CareTeam-wide PPE contamination,
Organization/department radioactive leak) are real patient-safety
events that today have no clean home: forcing the recorder to either
fan out the event into N per-individual `AdverseEvent` instances or to
relegate the collective subject into `AdverseEvent.contributor` /
`location` loses the semantic that the *collective* was the affected
party. Allowing `CareTeam` (a defined collective of practitioners) and
`Organization` (a department / facility) as `subject` mirrors the
pattern already adopted on neighbouring resources (e.g.,
`Communication.subject`, `DocumentReference.subject`, several `*.for`
elements) and keeps the cardinality at `1..1`, so the change is
strictly additive at the type level.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `AdverseEvent.subject` narrow and instead address the underlying
need by clarifying existing mechanisms:

1. Document in the AdverseEvent narrative that a `Group` of type
   `practitioner` is the recommended way to represent an affected care
   team when the event is logged against the collective (rather than
   adding `CareTeam`, which is a care-coordination resource not
   designed to be the "subject of harm"). Optionally add an example
   instance.
2. Document that physical-environment events (radiation leak, spill)
   should be recorded with `AdverseEvent.location` set to the affected
   room/department `Location`, with the *exposed people* (or a `Group`
   of them) as the `subject` — addressing bheale's point that "a room
   can be the *site* of a leak, not the *experiencer* of one".
3. If implementers still need to attribute an event to an `Organization`
   (e.g., a facility-wide regulatory finding), introduce an extension
   `adverseEvent-affectedOrganization : Reference(Organization)` rather
   than widening the core `subject` target set.

#### Justification

This approach preserves the conceptual integrity of `subject` as "the
party that experienced harm" — a constraint that downstream consumers
(safety registries, patient-compartment scoping, analytics) currently
rely on. `Group` already exists precisely to represent an unspecified
or anonymous collective of people, and `Location` already exists to
represent a contaminated physical place; widening `subject` to
`CareTeam` and `Organization` blurs the line between "who was harmed"
and "who is involved / where it happened", which the resource
intentionally separates via `contributor`, `recorder`, `location`, and
`actuality`. An extension keeps the door open for the rarer
organization-as-subject case without forcing every implementer to widen
their value-handling code.

---

### Disposition C: Decline

#### Proposal

Decline the request. The two motivating scenarios are already
expressible with existing AdverseEvent elements:

- **Contaminated RPPE / OR care team** — record one `AdverseEvent` per
  affected practitioner with `subject = Practitioner`, or a single
  event with `subject = Group` (a Group of those practitioners). The
  care team itself is a coordination construct, not the harmed party.
- **Radioactive leak in a department** — record `subject` as the
  exposed person(s) (Patient/Practitioner/Group) and use
  `AdverseEvent.location = Location` for the affected room/department,
  per bheale's comment.

Resolve as "Not Persuasive" / "No Change" with a Resolution Description
pointing to `Group`, `location`, and `contributor` as the appropriate
existing mechanisms.

#### Justification

`AdverseEvent.subject` semantically represents the party that
experienced the adverse event. `CareTeam` is a care-coordination
resource (membership and roles), not a clinical/affected entity, and
allowing it as `subject` would create modelling ambiguity with
`AdverseEvent.contributor`. `Organization` as a "harmed party" is
similarly a category mismatch — organizations don't experience harm in
the clinical sense the resource models; their *people* and their
*places* do, and both already have homes (`subject` for people,
`location` for places). Per the reporter comments from bheale, the
existing `Group` and `Location` elements already cover the cited cases
without expanding the target type set.

---

### Recommendation

**Recommended disposition:** B

The reporter's use cases are legitimate patient-safety scenarios and
should not be brushed aside (so a flat decline is too strong), but
widening `subject` to include `CareTeam` and `Organization`
(Disposition A) conflates "who was harmed" with "who was involved /
where it happened", which AdverseEvent intentionally separates via
`subject`, `contributor`, and `location`. bheale's review comments —
that `Group` likely already covers the CareTeam case, and that a room
is the *site* of a leak rather than its experiencer — strongly suggest
the right move is to clarify and document the existing model (and, if
needed, offer an extension for the organization-as-subject edge case)
rather than expand the core target set. This also avoids a wider blast
radius on patient-compartment scoping and consumer code that today
assumes `subject` resolves to a person-shaped resource. The work group
should confirm whether a `Group` of practitioners is acceptable for the
RPPE scenario; if so, a documentation/example update is sufficient and
no breaking type-set change is needed.
