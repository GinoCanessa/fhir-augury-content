## Summary

Lloyd McKenzie reports that the US Physical Activity FHIR IG currently
binds activity-type observations to the example LOINC answer list
[LL734-5](https://loinc.org/LL734-5), which has several quality issues:
it lists both "running" and "jogging" (a confusing distinction), it
includes the ambiguous code "mixed", and overall its membership is not
aligned with the activity-type vocabularies that wearables/phones
already expose via Apple HealthKit (iOS) and Android Health Connect.
The ticket asks the workgroup to revamp the value set — drop "jogging"
and "mixed", and add codes that correspond to well-defined concepts
present in both iOS and Android — so that data captured on consumer
devices can be mapped into the IG without semantic loss. The proposed
disposition recorded on the ticket is "Persuasive with Modification":
align with Apple & Android **but avoid product/trademarked names** (the
examples called out are "CrossFit", "Zumba", and "Spinning"). The
workgroup approved this disposition on the March 5 PA call and the
ticket is labeled `Ready-For-Vote`.

The closely related ticket
[FHIR-40949](https://jira.hl7.org/browse/FHIR-40949) made essentially
the same critique a year earlier — that LOINC `73985-4` / LL734-5 does
not cover popular activities (elliptical, pickleball, HIIT, yoga, etc.)
and should be reconciled with an industry source (in that case the
Garmin FIT SDK `sport` / `sub_sport` enumerations). FHIR-40949 was
closed "Not Persuasive" at the time, so FHIR-44892 represents the
workgroup revisiting the same problem with a sharper, vendor-neutral
direction (Apple + Android, not a single SDK).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the US Physical Activity FHIR IG, replace the example binding to
LOINC answer list `LL734-5` on the activity-type element with a new IG-
defined value set (e.g.
`http://hl7.org/fhir/us/physical-activity/ValueSet/activity-type`)
whose membership is curated to:

1. **Remove** the codes for `jogging` and `mixed` from the current
   list.
2. **Add** codes for the activity concepts that appear in **both**
   Apple HealthKit (`HKWorkoutActivityType`) and Android Health Connect
   (`ExerciseSessionRecord` exercise types) and that have well-defined
   semantics — e.g. cycling, elliptical, hiking, HIIT, pilates, rowing,
   stair-climbing, strength training, swimming (pool/open-water), tai
   chi, walking, yoga, and similar.
3. **Exclude** any codes whose primary identifier is a product name or
   trademark (e.g. CrossFit®, Zumba®, Spinning®), per the workgroup's
   recorded modification.

Where LOINC already publishes a code for the concept, prefer the LOINC
code; where it does not, file a LOINC submission and, in the interim,
mint a code in an IG-owned CodeSystem (e.g.
`CodeSystem/activity-type`). Keep the binding strength as `example` (or
`preferred`) to remain consistent with the IG's current modeling of
activity type.

#### Justification

This is exactly what the reporter asked for and what the workgroup
recorded as the disposition on the March 5 PA call ("Will go with Apple
& Android, but avoid making use of codes that are product names or
trademarks"). Aligning with the two dominant consumer-device
vocabularies directly addresses the implementer pain point that
motivated the ticket and FHIR-40949: mobile/wearable data is the
principal source of physical-activity observations, and unmappable
activity types currently force implementers to invent local codes or
collapse to "other". Excluding trademarks avoids the IP and
sustainability concerns the workgroup explicitly raised.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the example binding to `LL734-5` as-is (no value-set replacement),
but extend the IG with a clearly named **secondary** value set —
`activity-type-consumer-device` — that enumerates the Apple/Android-
aligned, non-trademarked concepts. Document on the activity-type
element that implementers SHOULD use a code from `LL734-5` when one
fits, and OTHERWISE MAY use a code from `activity-type-consumer-device`.
Concurrently, file LOINC change requests against LL734-5 to (a) retire
or deprecate `jogging` and `mixed`, and (b) add the missing
Apple/Android-aligned concepts, so that the long-term steady state is a
single LOINC-backed list.

#### Justification

This preserves the existing LOINC binding (no breaking change for
implementers already using LL734-5), avoids the IG taking on permanent
maintenance of an activity-code system that is really LOINC's domain,
and still gives consumer-device implementers a sanctioned place to put
codes for elliptical, pickleball, HIIT, yoga, etc. The trade-off vs.
Disposition A is slower convergence: implementers face two value sets
in the interim, and the cleanup depends on LOINC accepting the
submitted changes (FHIR-40949 shows that prior LOINC-direction
suggestions have stalled). It is a defensible middle path if the
workgroup is uncomfortable diverging from LOINC even on an `example`
binding.

---

### Disposition C: Decline

#### Proposal

Close the ticket without changing the value set. Note that the binding
is already `example`, so implementers are free to use any code that
fits their use case, including codes from HealthKit or Health Connect
mappings; the IG does not need to enumerate them. Direct vocabulary
expansion requests to LOINC via their submission process (as was done
in the disposition of FHIR-40949).

#### Justification

This was effectively the workgroup's prior position on FHIR-40949
("Not Persuasive"). The argument is that an `example` binding imposes
no constraint, so the "jogging vs. running" and "mixed" complaints are
cosmetic, and that owning a parallel activity-type vocabulary inside
the IG creates a maintenance burden and a divergence from LOINC that
HL7 has historically tried to avoid. However, this rationale is harder
to defend now that the workgroup itself has recorded a "Persuasive with
Modification" disposition on FHIR-44892, and it leaves the implementer
problem unsolved.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already reviewed and approved this direction on the
March 5 PA call, the ticket is labeled `Ready-For-Vote`, and the
modification ("avoid product/trademark names") is captured in the
resolution text — so Disposition A simply codifies what PC has agreed
to and discharges a complaint that has now been raised twice
(FHIR-40949 and FHIR-44892) by different reporters against the same
value set. The risk of accepting is low because the binding is example-
strength: nothing in existing implementations breaks, and consumer-
device data finally has a sanctioned mapping target. Disposition B is
the right fallback if the committee wants to defer to LOINC for the
canonical list, but it leaves implementers in the two-value-set
interim for an indeterminate period; Disposition C should be rejected
because it reverts to the FHIR-40949 outcome the workgroup has now
chosen to revisit.
