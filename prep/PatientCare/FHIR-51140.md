## Summary

The submitter notes a conformance asymmetry in the SDOHCC Coordination
Platform CapabilityStatement (US SDOH Clinical Care IG, STU2.2): the
`Organization` resource is listed as **SHALL Support** on the Client side
but only **SHOULD Support** on the Server side. A client that is required
to support `Organization` cannot reliably operate against a server whose
support for the same resource is optional, so the asymmetry appears to be
an authoring inconsistency rather than an intentional design choice. The
ticket asks the work group to explain or correct the mismatch. No related
Jira tickets are cross-referenced, but the request was rolled into the
"Gravity SDOHCC STU3 – Block Vote 2" disposition cycle (see Zulip below),
where the proposed resolution is to upgrade the Server side to **SHALL
Support** to match the Client side. The ticket has been triaged with
resolution "Persuasive".

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the resolution already drafted on the ticket: in the SDOHCC
Coordination Platform CapabilityStatement
(`CapabilityStatement-SDOHCC-CoordinationPlatform`), change the Server
interaction's Resource Conformance entry for `Organization` from
**SHOULD Support** to **SHALL Support**, so it matches the existing
**SHALL Support** designation on the Client side. No other resources or
interactions are touched; published narrative for §Organization2-11 is
regenerated from the updated CapabilityStatement during IG build.

#### Justification

A client conformance of SHALL paired with a server conformance of SHOULD
is internally inconsistent for an interoperability profile: a fully
conformant Coordination Platform client cannot count on any conformant
server actually exposing `Organization`, which defeats the point of
designating the resource as mandatory on the client side. The Gravity
Project has already proposed this fix and queued it for Patient Care
Block Vote 2 with a "Persuasive" disposition (see the Zulip thread),
indicating the SDOHCC author team views the asymmetry as a defect
rather than an intentional design. Tightening from SHOULD to SHALL is
the conservative direction (it strengthens, not weakens, an existing
expectation) and aligns with how sibling tickets FHIR-51138 and
FHIR-51139 are being handled in the same block vote.

---

### Disposition B: Alternative Approach

#### Proposal

Restore symmetry by relaxing the Client side rather than tightening the
Server side: change the Client Resource Conformance for `Organization`
in the SDOHCC Coordination Platform CapabilityStatement from
**SHALL Support** to **SHOULD Support**. Optionally pair this with
guidance text in the CapabilityStatement narrative explaining when a
client is expected to consume `Organization` (e.g., only when the
server's CapabilityStatement advertises it, discovered at runtime).

#### Justification

If implementation experience from Gravity reference servers shows that
many Coordination Platform deployments do not in fact expose
`Organization` (e.g., they delegate organization data to a directory or
identity service), then forcing servers to SHALL Support could create
unexpected conformance breakage in the field for STU3. Relaxing the
client side preserves backward compatibility for existing servers,
keeps the asymmetry resolved, and explicitly defers the
"must-be-exposed" decision to the realm/use-case profile that uses this
CapabilityStatement. The trade-off vs. Disposition A is reduced
guarantees for client implementers, who would need a runtime
discovery/fallback path when `Organization` is not advertised.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** with a clarifying note added to
the SDOHCC Coordination Platform CapabilityStatement narrative
explaining that the Client SHALL / Server SHOULD pattern is intentional:
clients are required to be **capable** of consuming `Organization` data
when a server provides it, while servers are not required to surface it
because some Coordination Platform deployments may legitimately defer
organization data to an external directory.

#### Justification

The conformance asymmetry only looks like a defect if one assumes
strict end-to-end symmetry between client and server SHALLs. FHIR
CapabilityStatements routinely encode "be tolerant in what you accept"
patterns where the client is required to handle a resource it
encounters, even if no single server is required to produce it. If
that is the SDOHCC author intent, the fix is documentation, not a
change in conformance level. This option avoids tightening server
requirements late in the STU cycle, which could destabilize existing
implementations.

---

### Recommendation

**Recommended disposition:** A

The ticket is a small, surgical conformance fix to a single
CapabilityStatement entry (`Organization` server interaction), the
SDOHCC project team has already converged on the "tighten Server to
SHALL" direction and queued it for Block Vote 2 with a Persuasive
disposition, and the in-ticket comment explicitly acknowledges the
mismatch as incorrect. Disposition B (relax the Client) would preserve
symmetry but weakens an existing expectation that implementers may
already be relying on, and Disposition C (document the asymmetry) only
holds up if there is a stated design rationale for the split, which is
not evident in the ticket, comments, or the Zulip thread. Accept as
requested, apply the SHOULD→SHALL change in the Server section of the
SDOHCC Coordination Platform CapabilityStatement, and bundle the
publish into the Block Vote 2 batch already in flight.
