## Summary

FHIR-54987 asks Patient Care to combine the existing `Condition.bodySite`
(`CodeableConcept`) and the newly added `Condition.bodyStructure`
(`Reference(BodyStructure)`) elements into a single `CodeableReference`
element on `Condition`. The ticket was imported as "Comment 13" by Ron G.
Parker and lodged at the highest priority against FHIR Core. It directly
revisits the design decision documented in the resolution of
[FHIR-39630](https://jira.hl7.org/browse/FHIR-39630), where Patient Care
explicitly weighed `CodeableReference` (Option 1) against a second,
mutually-exclusive `bodyStructure Reference(BodyStructure)` element
(Option 2) and selected Option 2 — which is what is now in R6. A near-identical
companion ticket, [FHIR-54988](https://jira.hl7.org/browse/FHIR-54988)
("…2", Comment 14), was filed at the same time against Patient
Administration and has already been closed as **Duplicate**, leaving
FHIR-54987 as the live request. Adjacent open tickets,
[FHIR-54843](https://jira.hl7.org/browse/FHIR-54843) (Orders &
Observations — guidance on when to use `bodyStructure` vs `bodySite`)
and [FHIR-52977](https://jira.hl7.org/browse/FHIR-52977) (Patient Care —
clarifying R5/R6 version applicability for the `bodyStructure`
extension), show that the broader `bodySite` / `bodyStructure` story
across resources is being re-examined in this cycle.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In FHIR Core (R6), retire `Condition.bodySite` (`CodeableConcept`) and
`Condition.bodyStructure` (`Reference(BodyStructure)`) and replace them
with a single `Condition.bodySite` of type `CodeableReference(BodyStructure)`,
with cardinality `0..*` (preserving the current `bodySite` cardinality).
Drop the existing mutual-exclusion invariant
(`bodySite.exists() implies bodyStructure.empty()`) introduced under
FHIR-39630 — `CodeableReference` already encodes the
"code-or-reference" choice intrinsically. Update the `Condition`
narrative and the cross-resource bodySite/bodyStructure guidance
sentence flagged in FHIR-54843 to describe the new shape, and coordinate
with Patient Administration (FHIR-54988 reporter) so both ballot
comments are dispositioned consistently.

#### Justification

This is the literal ask of the ticket and aligns `Condition` with the
modern FHIR pattern of using `CodeableReference` whenever an element can
carry either a code or a reference to a richer resource — the same
pattern recently adopted on several R5/R6 elements. It eliminates the
awkward two-element + invariant arrangement that PC explicitly flagged
as the trade-off when accepting Option 2 in FHIR-39630. Because
`Condition.bodyStructure` is brand-new in R6 (per FHIR-52977), the
Option-2 shape has not yet had a normative release, so the breaking
change concern that drove the 2023 decision is materially smaller now
than it was then.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing two-element shape on `Condition` (`bodySite` as
`CodeableConcept` and `bodyStructure` as `Reference(BodyStructure)` with
the mutual-exclusion invariant from FHIR-39630), but: (1) tighten the
element comments and the shared bodySite/bodyStructure guidance
sentence (in coordination with FHIR-54843) to make the
"use-`bodyStructure`-when-you-need-the-resource, otherwise use
`bodySite`" rule explicit and consistent across `Condition`,
`Observation`, `Procedure`, and any other resources O&O / PC enumerate;
and (2) open a cross-workgroup tracker (PC + O&O + FHIR-I) to evaluate
a `CodeableReference` migration **uniformly across all anatomic-site
elements** in a future major release rather than only on `Condition`.

#### Justification

The PC 10/5/2023 minutes on FHIR-39630 show this was a deliberate
choice, not an oversight: Option 1 was rejected primarily because it is
a breaking change and because the pattern needed community feedback.
Doing the consolidation only on `Condition` would create a third shape
("code-only", "code-or-separate-reference", "CodeableReference")
across the resources that carry an anatomic site, which directly
contradicts the harmonization motivation behind FHIR-54843. A
guidance-only fix here, plus a coordinated cross-resource workstream,
gets the consistency benefit without locking PC into a unilateral
breaking change.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** (or **Persuasive with Modification**
pointing at FHIR-54843 and the mutual-exclusion invariant). Rationale:
the requested consolidation was already considered and explicitly
rejected by Patient Care under [FHIR-39630](https://jira.hl7.org/browse/FHIR-39630)
in October 2023, where Option 1 (`CodeableReference`) was tabled in
favour of Option 2 (separate `bodyStructure` element + invariant) on
breaking-change and community-feedback grounds. No new use cases or
implementation evidence are presented in FHIR-54987 (the description is
a single sentence imported as "Comment 13"), and the imported companion
ticket FHIR-54988 has already been dispositioned as **Duplicate**, so
the request stands on its own without additional support.

#### Justification

FHIR's process discourages re-litigating recently-resolved Persuasive
dispositions in the absence of new information. The current Option-2
shape is functionally equivalent for implementers (it can carry either
a code or a reference, and the invariant prevents ambiguity), and the
only argument offered by the ticket is stylistic. Declining keeps R6
stable for early adopters who have already begun profiling
`Condition.bodyStructure` (e.g. the eu-core work referenced in
FHIR-53439) and avoids a breaking change to a Maturity-Model 5 resource
on the basis of one sentence.

---

### Recommendation

**Recommended disposition:** B

The literal `CodeableReference` consolidation in Disposition A is the
right long-term shape and matches where FHIR is heading on
code-or-reference elements, but doing it **only on `Condition`**
amplifies exactly the cross-resource inconsistency that FHIR-54843 is
already complaining about, and it overturns a deliberate Patient Care
decision (FHIR-39630) without bringing new evidence. Disposition C is
defensible procedurally but leaves the substantive concern about
inconsistency unaddressed. Disposition B captures the ticket's
underlying need — a cleaner, more uniform anatomic-site representation
— while routing it through the cross-WG conversation that O&O has
already opened, and it tightens the immediate guidance so implementers
in R6 stop having to guess between `bodySite` and `bodyStructure`. If
PC wants to signal openness to a future Option-1 migration, the
disposition write-up should explicitly say so and reference both
FHIR-39630 and FHIR-54843 so the cross-WG tracker has a clear mandate.
