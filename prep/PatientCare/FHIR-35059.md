## Summary

The submitter (Spencer LaGesse) asks the IPS work group to clarify what
`Composition.encounter` is intended to mean inside the International Patient
Summary (IPS) profile. The reporter's reading is that populating
`Composition.encounter` would narrow the scope of the IPS document to a
single referenced encounter, while leaving it absent would leave the
document scoped to the patient's overall record. The ticket asks IPS to
either (a) provide narrative explaining how an encounter‑scoped IPS should
be constructed differently from a patient‑scoped IPS, or (b) — if
encounter‑scoped IPS documents are out of scope — constrain
`Composition.encounter` to `0..0` in the IPS Composition profile.

The ticket has been Triaged by Patient Care / IPS but is currently
**Unresolved**. Comment history shows it was originally taken to a Sept 2022
WGM block vote (`block-vote-2`) with a recommended disposition of
*Considered for Future Use* (vote 14‑0‑16), but Lynn Laakso reverted that
resolution post‑STU publication so the topic could be re‑investigated. On
the 2023‑03‑22 IPS call, the group noted that `Composition.encounter` had
previously been `mustSupport` in IPS but that constraint had since been
removed, and the group agreed to add a comment along the lines of:
*"Generally an IPS document is not expected to be based on a single
encounter, so the use of this element is discouraged as its meaning would
be unclear in context of a medical summary."* Thomas Zhou's feedback (also
recorded in the comments) was that "the IPS is Act‑agnostic; with this
setting, the encounter might not fit in the profile. However, there is no
reason to stop using the attribute for any business requirement implemented
in any use‑case project." — i.e., discourage but do not prohibit.

The cross‑reference index surfaces only the Sept 2022 IPS block‑vote Zulip
thread, the Confluence page for the Sept 2022 Patient Care WGM agenda, and
the FHIR artifact `Composition.encounter`. No related Jira tickets are
linked, and no GitHub PRs/issues/commits are linked to this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested (clarify with narrative; do *not* constrain cardinality)

#### Proposal

Adopt the action the IPS group already agreed to on the 2023‑03‑22 call:
add a normative comment / narrative to the IPS Composition profile element
`Composition.encounter` (in the IPS IG, repo `HL7/fhir-ips`) stating:

> *"Generally an IPS document is not expected to be based on a single
> encounter, so the use of this element is discouraged as its meaning would
> be unclear in the context of a medical summary."*

In addition, add a short paragraph in the IPS Composition profile page
explaining that the IPS is intended as a patient‑scope summary and that
encounter‑scoped variants are out of the IG's intended primary scope — but
that implementers are not prohibited from referencing an encounter when a
specific business case requires it (Thomas Zhou's point). Leave
`Composition.encounter` cardinality at the FHIR R4 default of `0..1`.

#### Justification

This matches the consensus already recorded by the IPS group on the
2023‑03‑22 call and respects Thomas Zhou's caution that the IPS is
Act‑agnostic — narrowing cardinality to `0..0` would forbid implementers
from using `Composition.encounter` for legitimate downstream (non‑IPS)
business needs, which the group has explicitly said it does not want to
do. It is also the lowest‑risk change: a comment / narrative addition is
non‑breaking, requires no profile cardinality change, and directly answers
the submitter's "what does this mean?" question.

---

### Disposition B: Alternative Approach (constrain `Composition.encounter` to `0..0` in the IPS Composition profile)

#### Proposal

Tighten the IPS Composition profile (`HL7/fhir-ips`, IPS Composition
StructureDefinition) so that `Composition.encounter` has
`max = 0`, with a constraint description explaining that an IPS document
is by definition patient‑scope and not encounter‑scope. Update the
companion narrative in the IPS Composition profile page to state that
encounter‑scoped patient summaries should use a different document
profile (e.g., a discharge‑summary profile) rather than IPS.

#### Justification

This is the second branch the submitter explicitly asked the group to
consider ("if documents scoped to a particular encounter are not in scope
for IPS, should `Composition.encounter` be fixed with a maximum
cardinality of 0?"). It resolves the ambiguity by making the rule
machine‑checkable rather than narrative‑only, which is helpful for
implementers and validators. The trade‑offs are real, however: (a) it is
a *breaking* profile constraint for any implementation that has populated
`Composition.encounter` against the IPS Composition profile; (b) it
contradicts Thomas Zhou's explicit feedback that the IPS should remain
Act‑agnostic and not block legitimate downstream uses; and (c) the group
has already, on the 2023‑03‑22 call, opted for the narrative path rather
than the cardinality path.

---

### Disposition C: Decline (Not Persuasive — already resolved by removal of mustSupport plus prior block vote)

#### Proposal

Close FHIR‑35059 as *Not Persuasive* on the grounds that the underlying
ambiguity has effectively been addressed: `Composition.encounter` is no
longer `mustSupport` in the IPS Composition profile (per the 2023‑03‑22
call notes), and the Sept 2022 WGM already voted 14‑0‑16 to treat the
issue as *Considered for Future Use*. Make no profile or narrative change.

#### Justification

A defensible "do nothing" position can be argued from the existing record:
the WGM has already voted, mustSupport has already been removed, and the
ticket has been open for over three years without a normative change being
made. However, this disposition leaves the submitter's question — and the
implementer ambiguity that John Moehrke's 2023 *Purpose of use for IPS*
thread also flags — unanswered, and contradicts Lynn Laakso's explicit
re‑open of the ticket post‑STU publication.

---

### Recommendation

**Recommended disposition: A — Accept As Requested (narrative clarification, no cardinality change).**

Disposition A is the strongest fit for three reasons. First, it is the
disposition the IPS work group has *already agreed to in principle* on the
2023‑03‑22 call (the comment text is even drafted in the meeting notes);
adopting it simply finishes work that was started. Second, it directly
answers the submitter's primary ask ("clarify the meaning") without
foreclosing the legitimate downstream use cases Thomas Zhou flagged and
John Moehrke's discharge‑summary scenario implies — both of which would be
broken by Disposition B's `0..0` constraint. Third, it is non‑breaking and
low‑risk: a narrative + comment edit in `HL7/fhir-ips`'s IPS Composition
profile authoring source is a small, reviewable change that can ship in
the next IPS publication. Disposition B should be revisited only if the
group decides — separately and with new use‑case input — that IPS must be
strictly patient‑scope at the schema level. Disposition C should be
rejected because it ignores the explicit re‑open recorded on the ticket.
