## Summary

Heath Frankel reports an inconsistency in the IPS (International Patient
Summary) profiles between the `^comment` and the `^short` description on
three status-bearing elements: `AllergyIntolerance.verificationStatus`,
`MedicationStatement.status`, and `MedicationRequest.status`. The
`^comment` states that the `entered-in-error` code is **excluded** from
the profile's value-set binding, but the `^short` description still
lists `entered-in-error` as an allowed value, so the two pieces of
metadata contradict each other. The ticket proposes two coordinated
changes against the IPS guide: (a) a documentation/short-description
fix (PR HL7/fhir-ips#82) that aligns the `^short` text with the
`^comment` by removing `entered-in-error` from the listed codes, and
(b) a normative change (PR HL7/fhir-ips#83) that introduces new
ValueSets excluding `entered-in-error` and re-binds the three IPS
profile elements to those ValueSets, so that instances using
`entered-in-error` actually fail validation rather than just being
discouraged in prose. No related Jira tickets were surfaced by the
cross-reference index.

The ticket has already been discussed on the 2025-09-03 IPS call; the
recorded resolution is "Considered for Future Use" — i.e., the issue is
acknowledged but not slated for the current cycle.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt **both** PRs against `HL7/fhir-ips`:

1. **PR #82 (documentation):** Update the `^short` description on the
   IPS profiles for the three elements to match the `^comment`:
   - `AllergyIntolerance.verificationStatus` → `"unconfirmed | confirmed | refuted"`
   - `MedicationStatement.status` → `"active | completed | intended | stopped | on-hold | unknown | not-taken"`
   - `MedicationRequest.status` → `"active | on-hold | cancelled | completed | stopped | draft | unknown"`
2. **PR #83 (normative):** Define three new IPS-scoped `ValueSet`
   resources (e.g., `AllergyIntoleranceVerificationStatusCodes-IPS`,
   `MedicationStatementStatusCodes-IPS`,
   `MedicationRequestStatusCodes-IPS`) that enumerate exactly the
   codes listed above (i.e., the base R4 status code system minus
   `entered-in-error`), and re-bind the three IPS profile elements
   to these ValueSets, preserving the binding strength used in core
   FHIR (`required`).

#### Justification

This is the cleanest fix: it resolves the prose inconsistency the
reporter identified *and* makes validators actually enforce the IPS
intent. The IPS clearly does not want `entered-in-error` records to
appear in a patient summary — a summary is, by definition, the set of
clinically relevant assertions, not erroneous entries that have been
retracted. Today the profile only *says* this in `^comment`; making
the binding match the comment closes the conformance gap. Both PRs are
already authored by the reporter, which lowers implementation cost.

The main feasibility caveat (and the reason this disposition deserves
explicit scrutiny on the IPS call): the base FHIR R4 bindings on these
status elements are `required`. FHIR's profiling rules allow a profile
to constrain a `required` binding to a *subset* ValueSet of the same
code system, which is exactly what removing `entered-in-error` does —
so PR #83's approach is technically permissible. The IPS reviewer
should still confirm with FHIR Infrastructure / MnM that this
constraint pattern is acceptable for the published IPS guide.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the documentation portion (PR #82 — align `^short` with
`^comment`) but **decline the new ValueSets** in PR #83. Instead, add
a profile-level `invariant` (FHIRPath constraint) on each of the three
IPS profiles that asserts the element's `code` is not
`entered-in-error`, e.g.:

```
context: AllergyIntolerance / MedicationStatement / MedicationRequest
expression: <element>.coding.where(system = '<status-system>' and code = 'entered-in-error').empty()
severity: error
```

This keeps the existing required binding to the core ValueSet
unchanged, while still causing IPS-conformant validation to reject
`entered-in-error`. Optionally, the IPS narrative pages could note
that the constraint exists.

#### Justification

This avoids minting three new ValueSets that exist solely to remove a
single code, which some implementers and tooling find heavier than the
problem warrants (extra terminology artifacts to publish, version, and
consume). It also sidesteps any concern that re-binding a `required`
binding to a profile-specific ValueSet might be misread by downstream
profilers as a wider semantic shift, and it leaves the existing
ValueSets exactly where consumers expect them. The trade-off is that
constraint expressions are slightly less discoverable than a binding
in tooling UIs (IDEs and reference renderings tend to lead with the
binding), and validators implement invariants less uniformly than
bindings.

---

### Disposition C: Decline

#### Proposal

Decline the normative change (PR #83) and accept only the
documentation alignment (PR #82) — i.e., keep the existing required
bindings to the base FHIR status ValueSets, fix the `^short` text to
match the `^comment`, and rely on the `^comment` plus IPS narrative
guidance to discourage `entered-in-error`. Tag the issue as
"Persuasive — Documentation Only", which is consistent with the
existing IPS-call resolution of "Considered for Future Use".

#### Justification

The IPS profiles already document the exclusion in `^comment`; the
substantive integrity gap is small in practice because an
`entered-in-error` record is, by clinical workflow, supposed to be
suppressed by the source system before being summarized. Profiles
across HL7 are inconsistent about whether they enforce
`entered-in-error` exclusion at the binding layer or only call it out
in prose, and changing IPS unilaterally adds churn for implementers
without a clear interoperability win. Declining the binding change
also avoids any debate about constraining a `required` binding in a
published international IG, and matches the workgroup's recorded
resolution.

---

### Recommendation

**Recommended disposition:** **A**

The reporter has correctly identified an internal contradiction in the
IPS profiles (`^short` says one thing, `^comment` says the opposite)
*and* has supplied PRs for both the cosmetic and the normative fix.
Disposition A turns the IPS's already-stated intent into something
validators actually enforce, which is the whole point of a profile
that constrains base FHIR. Disposition B (invariants) achieves a
similar runtime outcome but is harder for downstream implementers to
discover and is unevenly supported by tooling, while Disposition C
leaves the spec saying contradictory things in different fields and
materially weakens the value proposition of constraining at all. The
one open question — whether constraining a `required` binding to a
profile-specific subset ValueSet is acceptable for an HL7
international IG — should be confirmed with FHIR Infrastructure / MnM
before merge, but is well-precedented elsewhere. The current
"Considered for Future Use" resolution is conservative; given that
the reporter has already done the implementation work, Patient Care
should reopen and progress this as Disposition A in the next IPS
cycle.
