## Summary

The submitter (Bryn Rhodes) proposes that the **US Public Health Profiles
Library (FHIR) IG** stop defining its own IG-local "expansion parameters"
extension and instead reuse the cross-IG
[`cqf-expansionParameters`](https://hl7.org/fhir/extensions/StructureDefinition-cqf-expansionParameters.html)
extension from the FHIR Extensions Pack. That extension already supports
specifying expansion parameters by reference from a `Library`, which is the
same use case the US PH-specific extension is solving. The ticket asks the
work group to either remove or deprecate the IG's own variant in favor of
the shared one.

This ticket sits inside a small cluster of related, same-author requests
that all push the US Public Health Profiles Library toward shared
extensions instead of IG-local ones:

- [FHIR-49700](https://jira.hl7.org/browse/FHIR-49700) — a near-duplicate
  request against the same IG, filed against the Public Health WG track,
  asking to replace the US PH-specific extension with
  `cqf-expansionParameters` and remove the local one.
- [FHIR-49781](https://jira.hl7.org/browse/FHIR-49781) — a broader
  housekeeping ticket (also Patient Care WG) listing many US PH IG
  extensions that should either move to the extensions pack or be
  replaced by existing extension-pack equivalents; the "Expansion
  parameters" line item there explicitly calls for replacement with
  `cqf-expansionParameters`. FHIR-49706 is effectively the focused
  version of that one bullet.
- [FHIR-52764](https://jira.hl7.org/browse/FHIR-52764) — same
  "consider re-using `cqf-expansionParameters`" recommendation against
  the same IG, filed from the Public Health side and already
  **Published**, indicating the WG has accepted this direction
  elsewhere.
- [FHIR-53319](https://jira.hl7.org/browse/FHIR-53319) — a related but
  distinct CRMI/Terminology Infrastructure ticket that clarifies the
  semantic distinction between `cqf-expansionParameters` ("parameters
  that *should be used* to expand"), the new `expansion-parameters`
  extension ("parameters that *were used* to expand"), and
  `cqf-inputParameters`. Useful context to make sure the right one of
  the three is adopted here.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Replace the US Public Health Profiles Library's IG-local "expansion
parameters" extension with the FHIR Extensions Pack extension
[`cqf-expansionParameters`](https://hl7.org/fhir/extensions/StructureDefinition-cqf-expansionParameters.html).
Concretely:

1. In every profile (e.g., the IG's `Library`/manifest profile and any
   `ValueSet` or terminology-related profiles) that currently allows or
   constrains the IG-local extension, change the allowed extension URL
   to `http://hl7.org/fhir/StructureDefinition/cqf-expansionParameters`.
2. Update the IG's narrative and examples to reference the
   extensions-pack extension and to point at its canonical
   `StructureDefinition` rather than the IG-local one.
3. Mark the IG-local `StructureDefinition` `status = retired` (and add a
   `replaced-by` mapping to `cqf-expansionParameters`) for this ballot
   cycle, then remove it in the following cycle once downstream
   implementers have migrated.
4. Coordinate with FHIR-49700 (Public Health) and FHIR-49781 (the
   bulk-replace ticket) so the same change is applied once, not three
   times.

#### Justification

The extension being replaced is functionally equivalent to
`cqf-expansionParameters`: both let an artifact reference a `Library`
that carries the parameters to use when expanding value sets. Reusing
the extensions-pack version aligns the IG with the rest of the FHIR
ecosystem, removes a maintenance burden from the WG, and is consistent
with the disposition already accepted on **FHIR-52764 (Published)** for
the same IG. A retire-then-remove path (rather than immediate deletion)
preserves backward compatibility for existing implementers in line with
HL7 deprecation conventions.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the *direction* of the request but **stage the change** and
**confirm semantics first**:

1. Before swapping, confirm against
   [FHIR-53319](https://jira.hl7.org/browse/FHIR-53319) that the US PH
   IG's intent matches the "*should be used* to expand value sets"
   semantics of `cqf-expansionParameters`, rather than the "*were used*
   to expand" semantics of the upcoming `expansion-parameters`
   extension. If the IG's actual intent is the latter, wait for
   `expansion-parameters` to land in the extensions pack and adopt that
   one instead.
2. Once the right target is confirmed, **deprecate** the IG-local
   extension (`status = retired`, narrative note, `replaced-by`
   mapping) in this ballot cycle, but do not yet remove it. Only adopt
   the extensions-pack extension in profiles/examples after a deprecation
   window of one published cycle.
3. Roll FHIR-49706, FHIR-49700, and the relevant line of FHIR-49781
   into a single coordinated change set (one PR per affected repo) so
   the IG, narrative, and examples all move together.

#### Justification

This preserves the request's intent while protecting against two real
risks visible in the related-ticket context: (a) picking the wrong
member of the `cqf-expansionParameters` / `expansion-parameters` /
`cqf-inputParameters` family — a distinction explicitly flagged in
FHIR-53319 — and (b) breaking existing US PH implementers by deleting
the IG-local extension in the same cycle the replacement is adopted.
Deprecate-then-remove is the standard FHIR pattern for this class of
change and is cheap to do here.

---

### Disposition C: Decline

#### Proposal

Close FHIR-49706 as **Persuasive — handled elsewhere**, and rely on
**FHIR-49781** (the bulk extensions-cleanup ticket) to drive the
actual change to the IG-local "expansion parameters" extension. Add a
comment cross-referencing FHIR-49781, FHIR-49700, and FHIR-52764 so
the disposition is traceable.

#### Justification

This is not a "do nothing" decline; it is a "do not double-track"
decline. The same change is already on the work plan via FHIR-49781,
which scopes the broader extensions-pack alignment for this IG, and
the parallel Public Health ticket FHIR-52764 has already been
**Published** with the same direction. Carrying FHIR-49706 separately
risks two PRs touching the same profiles and narrative pages, with no
additional information beyond what the other tickets already capture.

---

### Recommendation

**Recommended disposition:** B

The substance of the request is sound — the IG-local extension and
`cqf-expansionParameters` overlap, and the Public Health side has
already accepted the same direction on FHIR-52764 — so an outright
decline (C) would leave the IG inconsistent with its own sibling
disposition. Disposition A is close to correct but a little too
mechanical: FHIR-53319 makes it clear that there are now *three*
closely-named extensions in this space, and the safe move is to
explicitly confirm the IG's intent maps to `cqf-expansionParameters`
("should be used") rather than the forthcoming `expansion-parameters`
("were used") before profiles and examples are rewritten. Pairing that
check with a one-cycle deprecation window for the IG-local extension
costs little, follows standard FHIR deprecation practice, and lets
FHIR-49706, FHIR-49700, and the relevant slice of FHIR-49781 be
executed as a single coordinated change rather than three uncoordinated
ones.
