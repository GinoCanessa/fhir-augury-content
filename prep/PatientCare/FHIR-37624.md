## Summary

The IPS ValueSet `http://hl7.org/fhir/uv/ips/ValueSet/healthcare-professional-roles-uv-ips`
binds to a CodeSystem identified as `urn:oid:2.16.840.1.113883.2.9.6.2.7`. That OID
sits in HL7 Italy's OID branch, but the underlying classification is ISCO-08
("International Standard Classification of Occupations, 2008"), which is owned and
published by the International Labour Organization (ILO), an agency of the United
Nations. The reporter (Patrick Werner) argues that an internationally-scoped FHIR
IG should not bind to a country-specific OID for an internationally-published code
system, and asks that the IPS use a canonical URL in the ILO space and/or a
non-country-specific OID for ISCO-08.

There are no cross-referenced Jira tickets. The ticket was discussed at the IPS
weekly call on 2022-09-28 and surfaced for individual (not block) discussion at
the 2022-10-10 Patient Care co-chair call. A vote was taken on 2022-10-10
(John D'Amore / Rob Hausam, 6-0-0) to mark the ticket "Considered for Future
Use" with an action to ask HTA (the HL7 Terminology Authority) to finish the
canonical-URL determination, update the copy hosted on tx.fhir.org and inside
the IPS IG if needed, and look at supplying the complete code system content
rather than only a fragment. That resolution was subsequently reverted by
lynn_laakso on 2022-11-22 ("re-open post STU update publication"), which is why
the ticket is back in `Triaged / Unresolved`.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Re-bind the IPS ValueSet
`http://hl7.org/fhir/uv/ips/ValueSet/healthcare-professional-roles-uv-ips` to
an HTA-published, internationally-scoped representation of ISCO-08 rather than
the Italian-OID CodeSystem `urn:oid:2.16.840.1.113883.2.9.6.2.7`. Concretely:

1. Coordinate with HTA to publish (or formally identify) ISCO-08 with a
   non-country-specific identifier — either a canonical URL in the ILO/HTA
   namespace (e.g., something under `http://terminology.hl7.org/CodeSystem/`
   or an ILO-owned URL) and/or a globally-scoped OID for ISCO-08.
2. In `HL7/fhir-ips`, update the `compose.include.system` (and any
   `valueSet` of equivalent shape) of `healthcare-professional-roles-uv-ips`
   to point at the new identifier.
3. Replace the current fragment hosted on tx.fhir.org with the complete
   ISCO-08 content (the 2022-10-10 vote already called this out).
4. Add an IG narrative note explaining the binding change for STU implementers
   and link to the HTA decision.

#### Justification

This is exactly what the reporter is asking for, and it aligns with HL7
governance: when an external SDO owns a code system, the canonical identifier
should reflect that SDO, not whichever affiliate happened to publish the first
HL7 mirror of it. ISCO-08 is unambiguously an ILO publication, so binding an
international IG (IPS) through an Italian-affiliate OID is a defect in scope.
The 2022-10-10 Patient Care vote already endorsed the substantive approach
(let HTA finish the canonical-URL determination, then update IPS and
tx.fhir.org accordingly); the only reason the ticket is now re-opened is that
the change had to be deferred past the prior STU publication.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the existing OID-based CodeSystem identifier as a stable URI for already-
deployed implementations, but introduce a new HTA-published canonical URL for
ISCO-08 and bind the IPS ValueSet to that new canonical going forward, with
an explicit equivalence/migration note. Specifically:

1. Have HTA publish ISCO-08 under a stable canonical URL (e.g.
   `http://terminology.hl7.org/CodeSystem/ISCO-08` or an equivalent
   ILO-derived URL).
2. Add a `ConceptMap` (or a documented equivalence statement) between the new
   canonical CodeSystem and `urn:oid:2.16.840.1.113883.2.9.6.2.7`, asserting
   that the codes are identical.
3. Update `healthcare-professional-roles-uv-ips` to compose from the new
   canonical CodeSystem.
4. Leave the OID resolvable on tx.fhir.org so existing instances that already
   carry `system = "urn:oid:2.16.840.1.113883.2.9.6.2.7"` keep validating.

#### Justification

This still fixes the underlying scoping defect but treats it as a deprecation
rather than a hard switch, which lowers risk for implementers who have already
shipped IPS documents with the OID-based system identifier. It also formalises
the HTA-vs-Italian-affiliate boundary explicitly via a ConceptMap, which is
useful for any future HL7 IG that wants to reference the same terminology. The
trade-off vs. Disposition A is more terminology artifacts to maintain; the
upside is no behavioural break for existing instances.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Modification - keep current binding"
(or "Not Persuasive") on the grounds that the OID
`urn:oid:2.16.840.1.113883.2.9.6.2.7` is, in practice, a stable identifier for
ISCO-08 content even if it lives in an Italian-affiliate OID branch, and that
the responsibility for re-publishing ISCO-08 with a clean canonical URL lies
with HTA rather than with the IPS IG. Document the situation in the IG
narrative and move on.

#### Justification

Defensible only if (a) HTA is unwilling to publish ISCO-08 under a
non-country-specific identifier in a reasonable timeframe, or (b) there is
strong implementer evidence that re-pointing the binding would break existing
deployments. Neither condition is currently established in the ticket, in the
Zulip thread, or in the 2022-10-10 minutes — the workgroup actually voted in
the opposite direction. So this disposition exists mainly as a fallback in
case HTA cannot deliver a canonical URL.

---

### Recommendation

**Recommended disposition:** A

The reporter's argument is straightforward and uncontested: ISCO-08 is an ILO
publication, the IPS is an international IG, and binding through an Italian-
affiliate OID is a scope mismatch. The 2022-10-10 Patient Care vote (6-0-0)
already endorsed this substantive direction; the ticket only re-opened because
the change missed the prior STU publication window, not because the workgroup
changed its mind. Disposition A is therefore the cleanest path: ask HTA to
finish the canonical-URL determination for ISCO-08, then re-bind
`healthcare-professional-roles-uv-ips` to that canonical and host the full
code system on tx.fhir.org. If HTA signals that existing implementers would
be meaningfully disrupted by retiring the OID, fall back to Disposition B
(keep the OID resolvable, add a ConceptMap, bind forward to the new
canonical). Disposition C should only be selected if HTA explicitly declines
to publish a non-country-specific identifier.
