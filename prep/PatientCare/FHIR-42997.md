## Summary

The reporter requests that the constraint on the `allergyintolerance-substanceExposureRisk` extension (currently published in the FHIR Extensions Pack) be relaxed so that it can coexist with `AllergyIntolerance.code`. Today the extension is defined to prohibit `AllergyIntolerance.code`, which conflicts with US Core, where `AllergyIntolerance.code` is a required element. As a result, AllergyIntolerance instances that use the substance-exposure-risk pattern (typically to record that a patient is at risk from a class of substances without a specific identified allergy) cannot conform to US Core. The reporter offers two solutions and an explicit fallback: (1) relax the extension constraint — possibly with conditional logic so `.code` is only allowed when populated with a Data Absent Reason ("absent") value; (2) broaden the US Core profile so that presence of this extension satisfies the `.code` requirement (again ideally restricted to "absent" codes); or (3) accept the gap and treat it as an edge case where such instances simply fall outside US Core. The reporter notes that the third option is tolerable but not ideal. The cross-referenced FHIR artifact `AllergyIntolerance.code` is the element directly in tension with the extension's constraint. The supporting Zulip thread (`Patient Care WG > substanceExposureRisk constraint`) restates the same three options without further discussion. There are no other linked or cross-referenced Jira tickets and no related GitHub items.

## Details

**Description:**

The substanceExposureRisk extension prohibits the use of AllergyIntolerance.code, which US Core requires.

Absent specific substances is an edge case, so having these instances fall outside of US Core seems tolerable.

Ideally, though, we'd
- Relax the extension constraint (which might require or suggest some fancy conditional logic wrt DAR values in .code), or
- Broaden the US Core requirement to allow this extension in lieu of .code (which also would ideally only allow "absent" values).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **FHIR Extensions Pack**, modify the `allergyintolerance-substanceExposureRisk` extension's StructureDefinition (canonical `http://hl7.org/fhir/StructureDefinition/allergyintolerance-substanceExposureRisk`) to relax the invariant that prohibits `AllergyIntolerance.code` when this extension is present. Replace the blanket prohibition with a conditional invariant whose semantics are: when the extension is present, `AllergyIntolerance.code` SHALL either be absent or SHALL use a code from the FHIR Data Absent Reason value set (e.g., `unknown`, `not-asked`, `asked-unknown`, or — most naturally for this case — a code such as `unsupported` / `temp-unknown` as appropriate). Author this change in the Extensions Pack source for that extension (the SD JSON/XML for `allergyintolerance-substanceExposureRisk` plus its narrative/intro page), update the description and constraint expression (FHIRPath) accordingly, and add an example AllergyIntolerance instance that uses both the extension and `AllergyIntolerance.code` populated with a DAR code so US Core-conformant instances become possible without semantic conflict.

#### Justification

This option directly addresses the asymmetry the reporter identified: US Core makes `AllergyIntolerance.code` mandatory, but the substance-exposure-risk pattern is precisely the case where a specific substance code is not (yet) known. Allowing `.code` only with a Data Absent Reason preserves the original intent of the extension (no confirmed substance) while removing the hard conflict with US Core. The change is fully scoped to a single extension StructureDefinition in the Extensions Pack — the venue the reporter and the original triage comment confirmed as correct (jlyle, 2023-11-26) — and does not require any change to the core `AllergyIntolerance` resource. Constraining `.code` to DAR values keeps the semantic guard the original constraint was protecting (preventing implementers from putting a real substance code there and silently contradicting the extension), so it is a conservative relaxation rather than an open hole.

---

### Disposition B: Alternative Approach

#### Proposal

Leave the extension's constraint largely as-is in the **FHIR Extensions Pack** but coordinate with the **US Core** work group to update the US Core `AllergyIntolerance` profile so that, when the `substanceExposureRisk` extension is present, the `AllergyIntolerance.code` requirement is satisfied either by (a) a DAR code in `.code` or (b) presence of the extension itself. Concretely, weaken the US Core `.code` cardinality / binding rule on the AllergyIntolerance profile from "1..1, required code" to "1..1 with allowance for Data Absent Reason when the substanceExposureRisk extension is present", or add a profile-level invariant equivalent. The Extensions Pack would only need a narrative cross-reference noting the US Core interaction; no normative change to the extension's constraint.

#### Justification

The substance-exposure-risk pattern is inherently a US-realm / US-Core integration concern — it surfaced because US Core makes `.code` mandatory, not because the extension itself is broken on its own terms. Solving it in US Core localizes the change to the profile that created the conflict, keeps the extension's stronger semantic guard intact for non-US-Core implementers, and follows the FHIR design principle that a profile should constrain or relax requirements for its own realm rather than pushing realm-specific accommodations back into base artifacts. The trade-off versus Disposition A is that the change must be coordinated cross-WG (Patient Care owns the extension, CG/US Realm Steering owns US Core) and any other realm profile that inherits the same conflict would need a similar fix. It also leaves implementers who do not use US Core unable to combine the extension with `.code` even when they have a legitimate "absent" rationale.

---

### Disposition C: Decline

#### Proposal

Decline the change and treat substance-exposure-risk instances as **out of scope for US Core**. Close FHIR-42997 as "Not Persuasive — edge case", with a note in the extension's narrative explicitly acknowledging that AllergyIntolerance resources using `substanceExposureRisk` will not conform to US Core's `AllergyIntolerance` profile and SHOULD be exchanged using the base `AllergyIntolerance` resource (or a non-US-Core profile) when needed.

#### Justification

This is the option the reporter explicitly described as "tolerable" in the original description. Substance-exposure-risk without an identified substance is genuinely an edge case in clinical exchange; the volume of instances affected is low, and implementers already have a clean fallback (use base `AllergyIntolerance` rather than the US Core profile). Declining avoids any normative change to a published Extensions Pack artifact, preserves the extension's clean semantics ("we are documenting risk in the absence of a coded substance"), and avoids dragging US Core into a profile change for a marginal use case. The downside is that the conflict the reporter raised remains structurally present — implementers who must conform to US Core simply cannot use this extension — but the workgroup can defensibly judge that cost acceptable for an edge case.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the most defensible because it solves the problem at its actual source: the extension's invariant is the artifact that mechanically blocks `AllergyIntolerance.code`, and the FHIR Extensions Pack is — as confirmed by the reporter and triage comment — the correct venue for the fix. Constraining the relaxed `.code` to Data Absent Reason values preserves the original semantic intent (no real substance code may sneak in alongside a "risk without identified substance" assertion) while unblocking US Core conformance without requiring a coordinated cross-WG profile change. Disposition B is a reasonable fallback if Patient Care decides the constraint must remain absolute, but it pushes a realm-specific patch into US Core for a problem rooted in a base extension and would have to be repeated by every other AllergyIntolerance profile that inherits the same conflict. Disposition C is honest about scope but leaves a known structural conflict in place between two HL7 artifacts that are intended to interoperate, and the reporter's "tolerable" framing should not be over-read as a preference. Patient Care should adopt Disposition A and engage the Extensions Pack maintainers to draft the conditional invariant and example instance; the workgroup may also want to ping Cross-Group Projects / US Realm Steering as a courtesy so US Core's `AllergyIntolerance` profile narrative can be updated to reference the relaxed constraint.
