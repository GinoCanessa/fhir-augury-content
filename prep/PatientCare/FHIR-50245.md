## Summary

The reporter (Peter Muir) observes that the PACIO Transitions of Care
Implementation Guide enumerates five traditional Post-Acute Care (PAC)
settings — Hospice, Home Health, Skilled Nursing Facility (SNF), Long
Term Care Hospital (LTCH), and Inpatient Rehabilitation Facility (IRF) —
but does not model the eventual transition of the patient back to
Primary Care Provider (PCP) care. The ticket asks the IG to add this
"return to PCP" scenario as a recognized transition type. To avoid
adding clinician burden (an explicit IG design tenet), the reporter
suggests the profile should be standardized in a way that allows
receiving systems to "fold" or collapse data according to the
receiver's preference, so that the additional scenario doesn't force
every receiver to consume an overwhelming amount of detail.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items recorded for this ticket in the FHIR Augury index, so the
context above is taken solely from the ticket itself.

## Details

**Description:**

"The five traditional Post-Acute Care (PAC) settings, Hospice, Home
Health, Skilled Nursing, Long Term Care Hospitals, and Inpatient Rehab
facilities" does not include an ultimate return to PCP care. This
scenario should be considered since can eventually occur. However, an
overwhelming amount of detail would be counterproductive (see Reducing
Clinician Burden). Consider capability of 'folding' the data as per a
default receivers preference. While this would be done within a
receiving system, the profile should be standardized.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a sixth recognized post-acute transition scenario to the PACIO
Transitions of Care IG: **Discharge / Return to Primary Care Provider
(PCP)**. Concretely:

- Extend the IG's narrative inventory of supported PAC transition
  patterns (currently Hospice, Home Health, SNF, LTCH, IRF) with an
  explicit "Return to PCP" pattern, including a use-case section
  describing the typical clinical context (e.g., end of post-acute
  episode, patient resuming community-based primary care).
- In the Encounter / EncounterHistory or transition-summary profile
  used by the IG to characterize the destination setting, add a bound
  code (or extend an existing value set) representing "Primary Care"
  as a discharge disposition / next care setting. Where the IG today
  binds a value set for `Encounter.serviceType`,
  `Encounter.location.physicalType`, `dischargeDisposition`, or an IG
  extension that names the receiving setting, add a "Primary Care
  Provider" concept and document its semantics.
- Define a corresponding **transition-summary profile variant (or
  conformance section) for the Return-to-PCP scenario** that scopes
  the data set to what a PCP typically needs at handoff (problem
  list, medications, recent results, follow-up tasks, advance care
  planning), rather than the full PAC-to-PAC payload.
- Standardize a **receiver-driven "folding" mechanism**: define
  `MustSupport` slices and an explicit guidance section that lets
  conformant receivers collapse repeated/long-tail elements (e.g.,
  per-visit assessments) to summary form without losing conformance.
  This is presentation/consumption guidance — the sender still emits
  the standard payload — so the profile remains a single
  standardized contract.

#### Justification

The request reflects a real clinical workflow gap: every PAC episode
ultimately ends, and a meaningful fraction end with the patient
returning to ambulatory primary care rather than to another PAC
setting. Modeling this explicitly improves the IG's completeness as
a transitions-of-care guide and aligns with the Patient Care WG's
broader interest in continuity of care. The "folding" guidance
directly answers the reporter's stated concern about clinician burden
("Reducing Clinician Burden" is an explicit PACIO design tenet) by
keeping a single standardized payload while letting receivers present
a tailored view. Standardizing the receiver-side collapse rules (vs.
leaving them implementation-defined) is what preserves
interoperability — it ensures two PCP systems consuming the same
PACIO bundle render comparable summaries.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than extending the IG with a new sixth profile/scenario, treat
"Return to PCP" as a **reuse of existing transition mechanisms with an
added discharge-disposition code** plus a separate, lightweight
companion artifact:

- Add only the missing terminology — a "Primary Care" / "Return to
  PCP" concept in whichever value set the IG currently uses for the
  next care setting / discharge disposition (likely a binding on
  `Encounter.dischargeDisposition` or an IG-defined extension on the
  transition summary).
- Do **not** create a sixth transition-summary profile. Instead,
  publish a short **Implementation Guidance page** (non-normative)
  describing how to use the existing PAC transition-summary profile
  when the destination is a PCP, including recommended `MustSupport`
  expectations for that audience and a worked example bundle.
- Defer the "data folding" capability to a separate ticket aimed at
  the IG as a whole (it applies to all five existing scenarios too,
  not only the new one), or hand it to US Core / IPS where summary
  profiles already exist.

#### Justification

The five existing PAC scenarios are each motivated by setting-specific
regulatory data sets (e.g., MDS for SNF, OASIS for Home Health, IRF-PAI
for IRF). Primary care does not have an equivalent mandated
post-acute data set, so the marginal value of a sixth peer profile is
limited; the same payload structure works, only the disposition code
changes. This approach is much smaller in scope, avoids creating a
new normative profile that implementers must support, and keeps the
IG's published surface area focused. It also separates two distinct
concerns — "name the scenario" (small, local change) and "let
receivers fold data" (cross-cutting IG-wide concern) — which is
better governance than bundling them. The trade-off vs. Disposition A
is reduced explicitness: implementers won't see a dedicated
"Return-to-PCP" profile, only guidance, which may slow adoption.

---

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope for the PACIO Transitions of
Care IG**. The IG is intentionally scoped to transitions among the
five regulated post-acute settings, where setting-specific data
requirements (MDS, OASIS, IRF-PAI, etc.) drive the need for
standardization. Transitions back to ambulatory primary care are
already addressed by general-purpose continuity-of-care artifacts
(US Core `DischargeSummary` Composition, IPS, C-CDA Continuity of
Care Document) and do not require a PACIO-specific profile. The
reporter should be directed to those artifacts; if a gap exists
there, it should be raised against US Core or IPS.

#### Justification

PACIO's value proposition is harmonizing the *post-acute-specific*
assessment data that does not flow well through general continuity
artifacts. A PCP receiving a patient at the end of a post-acute
episode is the canonical use case for US Core / IPS, not for a
PACIO-specific transition profile. Adding a sixth scenario risks
turning PACIO into a general transitions-of-care IG, duplicating
work already done elsewhere and increasing implementer burden
(the opposite of the "Reducing Clinician Burden" tenet the reporter
cites). Declining keeps the IG's scope coherent.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the best balance of responsiveness and scope
discipline. The reporter has identified a genuine gap — the IG
currently reads as if PAC episodes only ever transition to other PAC
settings, which is clinically inaccurate — and that gap is cheaply
closed with a discharge-disposition code plus a guidance page. A
full sixth profile (Disposition A) is over-engineered: unlike the
existing five settings, primary care has no mandated post-acute data
set to standardize against, so a peer profile would mostly restate
the existing transition-summary structure. Outright declining
(Disposition C) is too dismissive: while US Core and IPS do cover
ambulatory handoff in general, the reporter is specifically asking
PACIO to acknowledge this terminal transition as part of the PAC
journey, and a value-set entry plus guidance does that without
ceding the use case. The "data folding" idea the reporter raises is
real but cross-cutting; it deserves its own ticket so the WG can
decide whether to standardize a receiver-presentation contract
across the whole IG rather than only for the new scenario.
