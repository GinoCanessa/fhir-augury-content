## Summary

Lloyd McKenzie requests guidance/profiling in the US Physical Activity FHIR
IG for representing strength-based exercise (resistance training)
observations. A single strength-training observation generally needs to
capture four facets: the exercise type (e.g., lateral press) on
`Observation.code`, the body site (e.g., left leg) on `Observation.bodySite`,
the amount of weight/resistance, and the number of repetitions. Because
"weight without reps" or "reps without weight" is not clinically meaningful,
the weight and rep count must be carried on the same `Observation` instance.
The ticket asks the work groups to define a structure (deciding which value
goes on `Observation.value[x]` versus `Observation.component`, and whether
two components are needed) and to nominate the standard codes for those
components. The ticket was originally directed to Orders & Observations for
core-profile feedback, but at the OO 2025-09-15 WGM it was returned to
Patient Care because PC owns the Physical Activity IG. There are no
cross-referenced Jira tickets, Zulip threads, or GitHub items for this
ticket; the only cross-references discovered are inline mentions of
`Observation.code` and `Observation.bodySite` in the description.

## Details

**Description:**

When representing strength-based exercises, there's typically a need to
capture 3 things:

- what type of exercise - e.g. lateral press (Observation.code)
- what part of body - e.g. left leg (Observation.bodySite)
- how much weight/resistance
- how many reps

The weight and reps really need to be captured as part of a single
Observation because knowing reps without weight or weight without reps
doesn't buy you much.

We need to define a structure that defines how to use components to manage
this and to nail down which is value and which is component, or whether we
have two components.  We also need to define the standard codes for the
components to be used.  Should seek OO feedback on these questions, so
assigning it there first.

**Resolution (Not Persuasive with Modification, 2024-04-16):**

Several concerns with doing this at this point:

- Lots of providers are not yet ready to handle this sort of data (formal
  evidence suggesting this)
- We don't have much evidence around the use of this sort of data
- Keep it 'simple' for now to get people on board. Look at ramping up once
  there's broader adoption and more demand
- We don't necessarily have 'standardized' questions that are reliable
  tools for gathering this sort of data
- Concerned about it increasing costs for software development

However, we may want to revisit this in the future.

We will add an open issue into the IG highlighting that this release
doesn't have specific guidance around detailed strength-based training
(and some of the reasons why), but are actively soliciting feedback on
what the community would like to see in this space.

We will add an issue marked as "considered for future use" for us to
revisit this in the future when we can evaluate feedback from implementers
and reconsider.

We're also open to testing at connectathons.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new strength-training profile (e.g., `pa-strength-exercise-observation`)
to the US Physical Activity IG that constrains `Observation` as follows:

- `Observation.code` — bound (extensible) to a value set of strength-exercise
  types (lateral press, leg press, bench press, squat, biceps curl, …),
  preferring LOINC where available with SNOMED CT codes as fallback.
- `Observation.bodySite` — bound (extensible) to the existing PA body-site
  value set (or SNOMED body-structure subset), with `bodySite` constrained
  to the laterality-bearing concepts where relevant.
- Disallow `Observation.value[x]`. Require `Observation.component` slicing
  on `component.code` with two named slices, both 1..1:
  - **`weight`** — `component.code` fixed to a single concept for
    "exercise resistance/weight" (LOINC if available, otherwise mint a PA
    code), `component.valueQuantity` with UCUM units constrained to
    `kg | lb_av`.
  - **`repetitions`** — `component.code` fixed to a single concept for
    "exercise repetitions per set", `component.valueQuantity` with UCUM
    `{reps}` (unit-less count, integer).
- Optionally allow a third 0..1 component slice **`sets`** (UCUM `{sets}`)
  so that a single Observation can carry "3 × 10 reps @ 50 kg" without
  forcing one Observation per set.
- Add IG narrative and an example Observation showing the canonical
  encoding, plus a note that the weight and reps MUST be co-resident on a
  single Observation.

#### Justification

This is the encoding the reporter explicitly asked for and matches existing
US Physical Activity IG patterns that already use `Observation.code` /
`bodySite` / components for activity measurements. Co-locating weight and
reps on a single `Observation` honours the reporter's clinical point that
the two values are only meaningful together, and using component slicing
(rather than `value[x]` plus a single component) keeps the structure
symmetric, extensible to a `sets` slice, and consistent with how multi-
parameter observations (e.g., blood pressure) are profiled across FHIR.
The OO feedback at WGM 2025-09-15 confirmed that PC owns this IG and is
free to define the profile without an OO core-profile dependency, removing
the original reason for the cross-WG referral.

---

### Disposition B: Alternative Approach

#### Proposal

Persist the current resolution ("Not Persuasive with Modification") but
strengthen and operationalise the IG-side follow-up that the resolution
already promised:

- Add an explicit "Open Issue" page section in the US Physical Activity IG
  describing the strength-training gap, listing the four data elements the
  reporter identified (exercise type, body site, weight, reps) and the
  open design questions (value vs. components, code system selection,
  whether `sets` is in scope).
- Add a "Considered for Future Use" entry tracking the same gap so it is
  re-surfaced in the next IG cycle.
- Stand up a connectathon track for strength-training observations and
  publish a non-normative example bundle in the IG showing one plausible
  component-slicing encoding (without binding it as a profile), so that
  implementers can experiment without the IG endorsing a single shape
  yet.
- Keep the formal profile work deferred until connectathon feedback and
  implementer demand have been evaluated.

#### Justification

This matches the WG's already-approved disposition (Apr. 16 PA call) and
addresses the substantive concerns the WG raised: limited provider
readiness, thin evidence base, lack of standardised questionnaires, and
risk of inflating implementation cost. It still gives implementers a
concrete artifact (the example bundle and connectathon track) to
experiment against, and it preserves a documented path back to a profile
once evidence accumulates — which is what Disposition A would commit the
WG to in advance of that evidence.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** with no IG-side action: do not add
a strength-training profile, do not add an "Open Issue" / "Considered for
Future Use" entry, and do not stand up connectathon work. Direct any
implementer interested in strength-training observations to file a
separate, use-case-backed change request once concrete real-world demand
materialises.

#### Justification

The resolution narrative concedes that the WG has no evidence base, no
standardised data-collection instrument, and no clear provider demand.
A pure decline avoids signalling that strength-training profiling is
actively on the IG's roadmap and avoids creating "considered for future
use" debt that may never be retired. It is the most defensible position
if the WG wants to keep the IG's scope tightly bounded to areas with
demonstrated demand.

---

### Recommendation

**Recommended disposition:** B

The current resolution is already "Not Persuasive with Modification", the
WG has already approved that disposition (Apr. 16 PA call), and the OO
review at WGM 2025-09-15 explicitly returned the work to Patient Care
without changing the substance. Disposition B is the smallest delta that
honours that approved disposition while making the follow-up commitments
in the resolution text concrete and tractable (Open Issue page,
"Considered for Future Use" entry, connectathon track, non-normative
example bundle). Disposition A would commit the WG to a profile ahead of
the implementer evidence the WG itself flagged as missing, and the
reporter's own framing — "we need to define a structure … should seek OO
feedback first" — supports a deliberate, evidence-led path rather than an
immediate normative profile. Disposition C is too austere given that the
WG has already publicly committed (in the resolution text) to documenting
the gap and inviting feedback; backing away from that commitment would
weaken the IG's relationship with the strength-training implementer
community.
