## Summary

Gay Dolin (reporter) requests that the US Physical Activity (PA) IG promote
`CarePlan` to the role of "cornerstone" profile that ties together the rest of
the IG's profiles. Specifically, the PA `CarePlan` profile should be able to
reference all of the other PA profiles — Goals, ServiceRequests, Observations,
Procedures, etc. — to provide an aggregated, comprehensive view of a patient's
physical-activity plan. The reporter does not want every reference to be
required, but does want the relevant references flagged as **Must Support** on
the PA `CarePlan` profile. The request additionally asks the IG authors to
explicitly call out where outcomes of physical-activity activities are tracked
inside the CarePlan (i.e., on R4's `CarePlan.activity.outcomeReference` /
R5's `CarePlan.outcome`).

The ticket was reviewed and approved by the project group on a 2023-05-02
conference call and was originally dispositioned as *Considered for Future Use*
(vote 6-0-1) on 2023-06-15 with the rationale that not every PA workflow even
produces a CarePlan and that connecting these relationships was not critical
to the immediate STU use cases. That resolution was reverted on 2023-08-25
("re-open post STU publication"), leaving the ticket back at **Triaged /
Unresolved** for Patient Care to redispose. The reporter pointed reviewers at
the MCC `CarePlan` profile
(`StructureDefinition-mccCarePlan`) as a precedent for this style of "umbrella
CarePlan" pattern.

Keyword-related tickets reinforce the context: **FHIR-39902** (Gay Dolin's own
US Core CarePlan Note-to-Balloters feedback) discusses how `CarePlan.activity.reference`
and `CarePlan.supportingInfo` should be used to point at PlanDefinition,
assessments, and other supporting resources; **FHIR-39852** and **FHIR-42691**
(MCC) flag the implementability concerns of *requiring* `outcomeReference` and
restricting it to a closed set of resources — directly relevant to the
"Must Support outcomes" half of this ticket; **FHIR-41911** (MCC) covers the
general "aggregated CarePlan referencing many other artifacts" pattern; and
**FHIR-40971** (PA IG) requested filling in the empty example PA CarePlan,
which the workgroup will need to align with whatever Must-Support set is
adopted here.

## Proposed Dispositions

The dispositions below are scoped to the **US Physical Activity (FHIR)** IG's
PA `CarePlan` profile. Because no PA-IG repo briefing is cached, the proposals
identify the spec-level artifact (the PA `CarePlan` StructureDefinition and
its example) without naming concrete file paths. If/when the PA repo is added
to the analysis cache, the recommendation should be re-pinned to the actual
authoring path.

### Disposition A: Accept As Requested

#### Proposal

In the PA IG, modify the PA `CarePlan` profile to:

1. Add Must-Support flags on the elements that link to the other PA profiles —
   at minimum `CarePlan.goal` (-> PA Goal profile),
   `CarePlan.activity.reference` / R5 `CarePlan.activity.plannedActivityReference`
   (-> PA ServiceRequest, PA Procedure, PA Task as applicable),
   `CarePlan.addresses` (-> PA Condition / Health Concern), and
   `CarePlan.supportingInfo` (-> PA Observation, PA Questionnaire/Assessment).
2. Keep all of these references **0..\*** (not required); only Must-Support is
   added, matching the reporter's explicit request "I would not make it
   required to reference all - but I suggest that they all be Must Support."
3. Add narrative on the PA `CarePlan` page that calls out where outcomes are
   tracked: in R4 publication, on
   `CarePlan.activity.outcomeReference` (-> PA Observation/Procedure that
   represents the result of a planned PA activity), and on
   `CarePlan.activity.outcomeCodeableConcept` for coded outcomes; and add a
   mapping note pointing to the R5 rename to `CarePlan.outcome` so
   implementers tracking R5 work see the forward-compatibility path.
4. Update the empty PA CarePlan example (FHIR-40971) so it actually populates
   each Must-Support reference, demonstrating the cornerstone pattern.
5. Cite the MCC `mccCarePlan` profile in the PA `CarePlan` page as the
   precedent the reporter referenced.

#### Justification

This is exactly what the reporter asked for and matches the project group's
2023-05-02 approval. The MCC `mccCarePlan` precedent (cited in the ticket and
reinforced by FHIR-41911) shows that the cornerstone-CarePlan pattern is an
established, ballot-tested approach for HL7 care-coordination IGs. Must-Support
(rather than `1..1`) is the lightest-touch mechanism that still tells
implementers "if you have this data, send it on the CarePlan" — and Must-Support
on a 0..* reference does **not** force every system to actually have a
CarePlan, which is the concern Lloyd McKenzie raised in the original
*Considered for Future Use* vote. The outcomes-highlighting work has zero
implementation cost (it is documentation + Must-Support flags on existing
elements) and addresses a known balloter pain point.

The MCC outcomes lessons (FHIR-39852, FHIR-42691) are explicitly heeded by
*not* tightening outcome cardinality and by surfacing both
`outcomeReference` and `outcomeCodeableConcept` so implementers have a path
for narrative/coded outcomes alongside the structured ones.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt the spirit of the reporter's request but split it into two narrower
changes, and add an explicit guidance section instead of broadly flagging
Must-Support on every reference element:

1. Add a new "Using the CarePlan as the entry point" page (or section in the
   existing CarePlan page) of the PA IG that diagrams the relationships between
   the PA CarePlan and each other PA profile, with prose stating "when present,
   these references SHOULD be populated to give consumers an aggregated view."
   No new conformance flags.
2. On the PA `CarePlan` profile, only add Must-Support to a small, scoped set
   that the workgroup is actually willing to test against an example —
   `CarePlan.goal` and `CarePlan.activity.reference` (planned activity) — and
   leave the other links as plain references with documentation.
3. Resolve FHIR-40971 (empty example) by building one rich example that
   exercises the documented pattern, rather than mandating Must-Support on
   the whole reference set.
4. Defer Must-Support on `outcomeReference` to a separate, focused tracker
   so the WG can take advantage of MCC's experience (FHIR-39852, FHIR-42691)
   and consider also recommending `outcomeCodeableConcept`.

#### Justification

This addresses Lloyd's original concern that "in some cases there won't even
*be* a CarePlan, and many systems will not produce these relationships" —
Must-Support carries a real conformance test burden, and applying it
broadly to a half-dozen references on a profile that some implementers may
not produce at all risks creating non-conformant CarePlan instances in the
wild. A guidance page + narrow Must-Support set captures the cornerstone
intent at lower implementation risk, defers the messier outcome-modelling
question to a tracker that can borrow MCC's lessons learned, and still gives
the workgroup something concrete to ship in the next PA publication. The
trade-off vs. Disposition A is reduced normative weight (less of the
reference set is conformance-tested) in exchange for materially lower
implementer pushback during ballot.

---

### Disposition C: Decline

#### Proposal

Re-affirm the original 2023-06-15 disposition of *Considered for Future Use /
Postpone* with rationale "insufficient implementation experience". State in
the disposition that PA implementers are not expected to centralise their
data on a `CarePlan` resource in this STU (per Lloyd's original vote
comment), and that the WG will revisit cornerstone-CarePlan and outcome
Must-Support flags once at least N PA implementations have produced
CarePlans in production. Close FHIR-41166 with a forward-pointer to a
new tracker scoped to "PA CarePlan Must-Support review for next ballot."

#### Justification

Defensible because nothing has materially changed since the original
*Considered for Future Use* vote (which was 6-0-1 — broad consensus). The
re-open comment on 2023-08-25 is a procedural reopen ("re-open post STU
publication") rather than new technical evidence; the underlying argument
that "these relationships are not critical to immediate use-cases and many
PA workflows skip CarePlan entirely" remains valid. Adding Must-Support
without implementation evidence risks repeating MCC's `outcomeReference`
mistake (FHIR-39852, FHIR-42691) where the WG locked in conformance
expectations that turned out to be unimplementable.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying need — a discoverable, aggregated view of a
patient's physical-activity plan, with outcomes clearly located — is
real and was endorsed by the project group on the 2023-05-02 call. Pure
Decline (C) wastes that signal and ignores the precedent set by MCC
(FHIR-41911) and Gay Dolin's own US Core feedback (FHIR-39902). However,
Pure Accept (A) goes broader on Must-Support than the workgroup has
implementation evidence to defend, which is exactly the failure mode MCC
hit with `outcomeReference` (FHIR-39852, FHIR-42691) and which prompted
Lloyd's original *Considered for Future Use* vote. Disposition B threads
the needle: it ships the cornerstone-CarePlan **guidance** and a populated
example now (also closing FHIR-40971), it adds Must-Support only where the
WG is confident enough to test it, and it punts the outcomes-modelling
question to a focused tracker that can incorporate MCC's lessons. This
gives the next PA publication something visible and reviewable for
balloters without risking another over-constrained-outcome ballot
comment in the next cycle.
