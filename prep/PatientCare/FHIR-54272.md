## Summary

The reporter (Stephen Chu) observes that the `Goal` resource currently
provides `Goal.start[x]` (0..1) for the start of the goal and
`Goal.target.due[x]` (within the 0..* `Goal.target` backbone) for the due
date of each individual target, but offers no element that captures the
overall due / end date of the goal itself. When a `Goal` instance has no
`Goal.target` at all, there is no way to record or compute when the goal
is expected to be complete. The change request asks Patient Care to add a
new `Goal.due[x]` element at the root of the `Goal` resource, paired with
an invariant that requires `Goal.due[x]` to be on or after the latest
`Goal.target.due[x]` value when targets are present. No related Jira
tickets, Zulip threads, or GitHub items were surfaced by the
cross-reference search; the four cross-references returned are all
mentions of FHIR artifacts (`Goal.start`, `Goal.target.due`, `Goal.due`,
`Goal.due[x]`) inside the ticket itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new optional element `Goal.due[x]` to the `Goal` resource in FHIR
Core with cardinality `0..1` and the same allowed types as
`Goal.target.due[x]` (i.e., `date` and `Duration`). Add a definition
along the lines of "The date or time-after-start by which the overall
goal is expected to be achieved; intended for use when no individual
`Goal.target.due[x]` is supplied or when the overall goal end date
differs from any single target's due date." Add an invariant on `Goal`
(e.g., `gol-2`) along the lines of:

```
target.due.exists() implies
  due.ofType(date).empty() or
  target.due.ofType(date).empty() or
  due.ofType(date) >= target.due.ofType(date).max()
```

with an analogous clause for the `Duration` choice. The work would land
in HL7/fhir under the Goal resource's authoring inputs (StructureDefinition
JSON, the resource-level invariants table, and the narrative description
of the new element).

#### Justification

The reporter's observation is factually correct: the current model lets
implementers state when a goal *starts* and when each *target* is due,
but provides no element for the overall goal end / due date when targets
are absent. Adding `Goal.due[x]` mirrors the symmetry already established
by `Goal.start[x]` and re-uses the type set already accepted on
`Goal.target.due[x]`, keeping the design internally consistent. The
proposed invariant prevents the obviously-wrong state in which the
overall goal is "due" before one of its constituent targets is due,
which is the scenario the reporter explicitly calls out.

---

### Disposition B: Alternative Approach

#### Proposal

Do not add a new root-level element. Instead, define a standard pattern
(documented in the `Goal` resource narrative and/or via a
`goal-overall-due` extension published in the FHIR Extensions Pack) for
representing the overall goal due date. The pattern would be: when an
overall due date is needed, populate a single `Goal.target` whose
`measure` is omitted (or set to a sentinel) and whose `Goal.target.due[x]`
carries the overall due date; or, alternatively, attach the
`goal-overall-due[x]` extension at the `Goal` root. Update the resource
narrative to make this guidance explicit and add a short example.

#### Justification

`Goal.target` already carries `due[x]`, and many implementations have
historically modelled "overall due" by either using a single target or
applying an extension. Choosing this route avoids a breaking-feeling
core change in a mature resource, defers schema churn, and keeps a
single locus for "due" semantics (`Goal.target.due[x]`). The trade-off
versus Disposition A is real: implementers who treat `Goal.target` as
strictly "measurable sub-goals" find this overloaded, and an extension
is less discoverable than a first-class element. This option is the
right answer if Patient Care concludes that the existing target
mechanism, plus guidance, is sufficient and that the cost of adding a
new root element is not warranted.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the overall goal end date can
already be expressed using `Goal.target.due[x]` (with a single target
where necessary), and that for goals that genuinely have no notion of
"due" (open-ended aspirational goals), the absence of a due date is
itself meaningful information. Mark the ticket as "Persuasive with
Mod — no change", recommending the reporter use `Goal.target` plus
narrative when an overall due date must be captured.

#### Justification

This is defensible if the workgroup judges that (a) every concrete
goal worth tracking can be expressed with at least one `Goal.target`,
and that targetless goals are by definition open-ended and should
*not* carry a due date, and (b) adding another `[x]` element to a
mature resource increases implementation burden, validator surface,
and mapping work for downstream IGs (US Core, IPS, ICHOM, etc.) for
limited additional expressiveness. Declining preserves backward
compatibility and avoids invariant churn, but it leaves the reporter's
specific use case — a goal with no targets at all — without a clean
representation.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a genuine modelling gap: `Goal.start[x]`
exists at the root of `Goal`, but its symmetric counterpart does not,
and the workaround in Disposition C (always create a `Goal.target` even
when none is conceptually appropriate) overloads `Goal.target` and
weakens the semantics of "target" as a measurable sub-goal. Disposition
B's extension pattern is a viable middle ground but trades discoverability
and validator support for a conservative core. Given that `Goal` is
widely used in care-planning IGs that already need to express overall
goal end dates (chronic-disease management, behavioural-health plans,
care plans with single overarching objectives), promoting the element to
the core resource — with the invariant the reporter proposed — is the
cleanest fix and the one most consistent with the existing `Goal.start[x]`
design. The workgroup should confirm the type set (`date | Duration`,
matching `Goal.target.due[x]`) and the exact invariant wording during
discussion.
