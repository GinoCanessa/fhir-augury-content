## Summary

The reporter requests that the US SDOH Clinical Care (SDOHCC) FHIR Implementation Guide
declare the US Core IG at versions **6.1.0** and **7.0.0** as dependencies for its profiles.
SDOHCC profiles already conform to and reference US Core resource profiles (e.g.,
US Core Patient, US Core Condition, US Core Observation), so explicit dependency
declarations would align SDOHCC with current US Core releases used by implementers
in the United States. The request does not specify whether one specific version
should be pinned or whether multiple US Core versions should be supported simultaneously
(i.e., a multi-version dependency strategy), which is a key disposition question.
There are no cross-referenced Jira tickets, and no GitHub cross-references were
returned by the orchestrator at the time of preparation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add **both** US Core 6.1.0 and US Core 7.0.0 as declared dependencies of the SDOHCC
IG package, listing each as an entry under `dependencies` in `sushi-config.yaml`
(or the equivalent `dependsOn` entries in the generated `ImplementationGuide`
resource), e.g.:

```yaml
dependencies:
  hl7.fhir.us.core:
    id: uscore610
    version: 6.1.0
  hl7.fhir.us.core7:
    id: uscore700
    version: 7.0.0
```

Update SDOHCC profiles (Patient, Condition, Observation, Goal, Procedure, Task,
ServiceRequest, Encounter, etc.) to make the parent profile reference explicit
against the chosen US Core baseline, and verify that downstream IGs (e.g.,
Gravity-related profiles) continue to validate.

#### Justification

This matches the literal request and aligns SDOHCC with the two US Core releases
most widely adopted in production today (6.1.0, the long-standing baseline tied
to USCDI v3, and 7.0.0, the current STU release tied to USCDI v4). Multi-version
US Core dependencies are an established pattern in HL7 US-realm IGs and let
implementers pick the US Core version that matches their certification target
without forking SDOHCC.

---

### Disposition B: Alternative Approach

#### Proposal

Declare a **single** US Core dependency at the most recent normative-aligned
version (US Core 7.0.0) as the SDOHCC baseline, and publish a short *Conformance
with US Core* page in the SDOHCC IG documenting:

1. that SDOHCC profiles are designed to be compatible with US Core 6.1.0 *and*
   7.0.0 because they constrain only elements stable across both;
2. guidance for implementers still on US Core 6.1.0 explaining which SDOHCC
   profiles can be used unchanged versus those that require local re-binding;
3. a published validation matrix demonstrating compatibility.

This avoids the IG-publisher / validator complexity of carrying two simultaneous
US Core dependencies in one package while still answering the underlying need.

#### Justification

Carrying two simultaneous US Core dependencies in one IG package can produce the
sort of duplicate-derivation and slicing warnings observed in the related Zulip
thread (*"Odd warnings after updating a dependency version"*), and forces the
IG publisher to reconcile two parallel canonical bases for shared resources
(Patient, Observation, etc.). A single declared dependency plus a documented
compatibility matrix is lower-risk for the IG build pipeline and gives
implementers explicit guidance instead of implicit dual support.

---

### Disposition C: Decline

#### Proposal

Do not add US Core 6.1.0 and 7.0.0 as new SDOHCC dependencies in the current
publication cycle. Document that SDOHCC currently depends on its existing US Core
baseline (whatever version is already declared in the SDOHCC `sushi-config.yaml`
at the time of triage) and defer any change to a future release tied to a
broader SDOHCC re-publication and US Core alignment review.

#### Justification

Defensible if (a) the SDOHCC IG is between formal publication cycles and adding
or changing dependencies would force an unplanned republication, or (b) the
workgroup determines that the SDOHCC profiles do not in practice constrain
US Core-specific elements and therefore do not require a hard package
dependency. Since the ticket provides no use-case detail beyond the one-line
request, declining-with-deferral is a reasonable holding position until the
US Core alignment scope is scoped in a planning ticket.

---

### Recommendation

**Recommended disposition:** A

The reporter has named two specific US Core versions (6.1.0 and 7.0.0), which are
exactly the two versions most US-realm implementers are running today: 6.1.0 for
USCDI v3 / ONC §170.215 incumbents, and 7.0.0 for USCDI v4 adopters. Refusing to
declare either dependency (Disposition C) leaves SDOHCC under-specified relative
to the realm it targets, and pinning to only one (Disposition B) forces a hard
choice on implementers who have legitimate reasons to be on either US Core
version. Disposition A is also the explicit pattern endorsed by other HL7
US-realm IGs that need to bridge a USCDI version transition. The Patient Care
workgroup should accept the request as stated, confirm with the SDOHCC IG editors
that the IG publisher build is green with both dependencies declared, and use
the *Odd warnings after updating a dependency version* Zulip thread as a known
issue to validate against during republication.
