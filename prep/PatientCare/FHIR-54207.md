## Summary

The reporter observes that, following community discussions, the FHIR community
has aligned on a "pattern" for referencing `BodyStructure` from other resources:
where applicable, the resource should expose a single
`CodeableReference(BodyStructure)` element (carrying either an inline body-site
code or a reference to a `BodyStructure` instance), and any other
`bodyStructure`/`bodySite` representations in the resource should be removed
in favor of that single element.

The ticket asserts that *"this resource"* (a Patient Care resource) does not
follow that pattern, and asks the work group to either (a) update the resource
to conform, or (b) document in the resource's Notes section why it deviates.
The ticket text does not explicitly identify *which* Patient Care resource is
being challenged, and the FHIR Augury snapshot for the ticket carries no
"related artifacts" metadata, so the work group will need to confirm the target
resource from context (see the Keywords / Recommendation sections for the
likely candidates).

The ticket is one of three filed simultaneously by the same reporter:
[FHIR-54206](https://jira.hl7.org/browse/FHIR-54206) (Orders & Observations,
which explicitly calls out an existing element typed `Reference(BodyStructure)`
that should be re-typed to `CodeableReference(BodyStructure)`),
**FHIR-54207** (Patient Care, this ticket), and
[FHIR-54208](https://jira.hl7.org/browse/FHIR-54208) (also Patient Care).
Together they ask each owning work group to do the same conformance review
against the BodyStructure pattern.

## Proposed Dispositions

> **Caveat:** All three dispositions below are written generically because
> the ticket does not name the specific Patient Care resource under review.
> Before the work group acts, the most important step is to confirm with the
> reporter (Ana Kostadinovska) which resource is intended — the Zulip
> inventory above suggests the likely candidates are `Procedure`,
> `Condition`, `AllergyIntolerance.reaction`, `FamilyMemberHistory`,
> `AdverseEvent`, or `ImagingSelection`, since the resources Patient Care
> co-stewards that *already* use `CodeableReference(BodyStructure)` (e.g.,
> `Observation.bodyStructure`, `ServiceRequest.bodyStructure`,
> `DocumentReference.bodyStructure`) are not in scope.

### Disposition A: Accept As Requested

#### Proposal

Identify the Patient Care resource the reporter intended (e.g., `Procedure`),
then in the FHIR Core build replace the existing body-site representation
with a single `CodeableReference(BodyStructure)` element. Concretely, for
the most likely candidate `Procedure.bodySite`, retype the element from
`CodeableConcept` to `CodeableReference(BodyStructure)` (preserving the
existing binding on the `concept` arm), update `Procedure.fsh`/the resource
StructureDefinition accordingly, regenerate examples, and update the
narrative on the resource's intro page to point at the BodyStructure
referencing pattern. The same shape would apply to `Condition.bodySite`,
`AllergyIntolerance.reaction.*` body-site references, etc., depending on the
confirmed target.

#### Justification

This is the literal request and aligns the resource with the cross-resource
pattern the community has converged on (one element, dual code/reference via
`CodeableReference(BodyStructure)`). It matches the existing precedent set
by `Observation.bodyStructure`, `ObservationDefinition.bodyStructure`,
`ServiceRequest.bodyStructure`, and `DocumentReference.bodyStructure`, all
of which are listed as `CodeableReference` in the inventory Gino Canessa
posted on the FMG Zulip thread. Convergence reduces implementer cognitive
load and makes downstream guidance (IGs, profiles, terminology bindings)
consistent across resources that all describe anatomical sites.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than retype the existing element on the resource, leave the current
element shape in place and meet the pattern *additively*: deprecate
(or constrain via guidance) the existing typed body-site element in favor of
a sibling `CodeableReference(BodyStructure)` element introduced alongside
it, with cross-element invariants ensuring at most one is populated. For a
substantive deviation from the pattern, instead document the rationale in
the resource's **Notes** section (as the ticket itself offers as an
alternative) — for example, on `Condition.bodySite` if Patient Care wants
to retain `CodeableConcept` for compatibility with very large existing
implementer bases.

#### Justification

The pattern is normative aspiration, but several Patient Care resources are
either Normative or carry significant FMM and have heavy real-world use
where retyping `bodySite`/`reaction.*` from `CodeableConcept` to
`CodeableReference(BodyStructure)` is a breaking change for implementers
with no `BodyStructure` workflow today. An additive path (or an explicit
documented deviation) preserves the migration runway. The ticket text
itself sanctions the "document the deviation" route, so this is firmly
within the bounds of what the reporter asked for.

---

### Disposition C: Decline

#### Proposal

Decline as **insufficiently specified**: close as "Not Persuasive — please
re-file with the specific resource and element identified". The ticket does
not name the resource, does not name the element, and the FHIR Augury
snapshot of the ticket carries no related-artifact metadata that
disambiguates. Without that, any change Patient Care makes risks targeting
a resource the reporter did not intend, and any "documented deviation"
written in the Notes section would be answering a question the reporter did
not actually ask.

#### Justification

Spec changes (especially retyping an element) are non-trivial, and the
sibling ticket FHIR-54206 — which *does* concretely identify its target by
naming the offending `Reference(BodyStructure)` typing — shows that the
reporter is capable of being specific when they intend to be. Acting
without that specificity would either create churn against the wrong
resource or force the work group to invent a target. Declining and asking
for clarification is the cheapest way to recover the missing information,
and the reporter is active and responsive (filed three tickets in one
sitting, and the comment thread shows engagement).

---

### Recommendation

**Recommended disposition:** C, with an immediate path to A or B once
clarification arrives.

The blocker for FHIR-54207 is not philosophical — Patient Care broadly
agrees with the BodyStructure pattern, and the precedent of
`Observation.bodyStructure` / `ServiceRequest.bodyStructure` /
`DocumentReference.bodyStructure` already in `CodeableReference` form makes
that clear — the blocker is that the ticket does not say which resource it
is about, and it is a sibling of FHIR-54208 (also Patient Care, also
unspecified). Trying to dispose of both in parallel without clarification
risks Patient Care guessing two resources and getting one (or both) wrong.
Asking the reporter to point at the specific resource and element (and
ideally update both 54207 and 54208 in one pass) costs little and unblocks
a clean disposition. Once the targets are confirmed, the work group should
default to **Disposition A** for resources where retyping is low-impact
(e.g., trial-use, low FMM, light implementer base), and to **Disposition B**
(documented deviation) for resources where breaking the existing
`bodySite`/equivalent element would be disruptive.
