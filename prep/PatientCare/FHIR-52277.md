## Summary

The reporter asks the IPS work to clarify how an `Immunization` instance
should be populated when it is being used to assert that a patient has
**no known immunizations**. The Immunization (IPS) profile already
points implementers at SNOMED CT code `787482006 | No known
immunisations |` (available via the preferred-bound *Vaccines - IPS*
value set on `Immunization.vaccineCode` / `Immunization.code`), but the
profile does not say what values should be used for the other elements
that an `Immunization` instance must / typically carries when the
resource is being used as a positive assertion of absence rather than as
a record of an actual administration. Specifically, the ticket asks for
guidance — and a worked example — covering at least:

- `Immunization.status` — what code is appropriate when the resource
  represents "no known immunizations" (`completed` is semantically
  awkward; `not-done` requires `statusReason`; `entered-in-error` is
  clearly wrong).
- `Immunization.occurrence[x]` — what this date/period means in the
  absence of an actual event. Is it the date the assertion was made
  (i.e., date of recording / attestation), or should it be omitted /
  represented some other way?

## Details

**Description:**

Consider clarifying how to represent no known immunizations in IPS. At
this time there is the following guidance on `Immunization.vaccineCode`
in the Immunization (IPS) profile:

> The IPS assumes that either the type of the vaccine for particular
> disease or diseases (e.g. MMR vaccine) against which the patient has
> been immunised is provided; or the known absent / unknown code. Other
> coded information can be provided as well as: the IDMP Pharmaceutical
> Product Identifiers (PhPID), Medicinal Product Identifier (MPID),
> Packaged Medicinal Product Identifier (PCID), when available, or
> equivalent coded concepts; the WHO ATC codes; or any other kind of
> code that that identifies, classifies or cluster the administered
> product.

This statement refers to SNOMED code `787482006 | No known
immunisations |`, which is available in the IPS value set for
`Immunization.code` ([Vaccines - IPS](https://build.fhir.org/ig/HL7/fhir-ips/ValueSet-vaccines-uv-ips.html), preferred binding).

What is missing is guidance on the expected population of the following
elements when populating an `Immunization` instance to represent no
known immunizations:

- `Immunization.status`
- `Immunization.occurrence[x]` — what does this mean semantically; is
  it date of assertion?

Please provide clarification and an example.

**Resolution Description:** Discussed on 9/3 IPS call. Will revisit
after 2.0 publication. (Resolution: *Considered for Future Use*.)

**Comments:**

- **JIRAUSER25409 (2025-10-17):** Notes that in SNOMED CT,
  `787482006 | No known immunizations |` is an inferred child of
  `787480003 | No known procedures |`, because *Immunization* is-a
  *Procedure*. So asserting "no known immunizations" implies "no known
  procedures" under SNOMED's description logic. The commenter warns
  that negation of things like Procedures and Immunizations is broken
  in SNOMED CT (no disjoint classes), and any analytics that
  categorise `Immunization` resources using ECL may make incorrect
  inferences from the chosen code.

### [IPS &gt; Interpreting 'Known Absent' Information in Composition (2023-11)](https://chat.fhir.org/#narrow/stream/IPS/topic/Interpreting%20%27Known%20Absent%27%20Information%20in%20Composition/near/402895583)

Thomas Zhou discusses the subtle difference between
`Composition.section.emptyReason` (a *document/section* signal that the
section is empty after reasonable investigation) and a clinical "no
known X" resource entry inside the section. The thread argues that a
clinical "no known" assertion in `composition.section.entry` (e.g., a
`No Known Allergy` AllergyIntolerance) carries clinical weight that
`emptyReason` does not. The same distinction applies directly to
Immunization and reinforces the case that IPS should standardise the
*entry-level* representation rather than relying on section-level
emptyReason alone.

Participants: Thomas Zhou and other IPS contributors.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **Immunization (IPS) profile** narrative (in the HL7
`fhir-ips` IG) to add an explicit "Representing No Known Immunizations"
sub-section and a worked example. Concretely:

- On `Immunization.vaccineCode` (and the `Immunization.code` slice that
  IPS uses), add narrative reaffirming that SNOMED CT
  `787482006 | No known immunisations |` (from the *Vaccines - IPS*
  value set) is the preferred code for the absent assertion.
- On `Immunization.status`, add narrative stating that the value
  **SHALL be `completed`** when the resource is being used as a
  positive assertion of "no known immunizations" (the assertion itself
  has been completed; `not-done` is reserved for a vaccine that was
  ordered/intended but not administered).
- On `Immunization.occurrence[x]`, add narrative stating that the
  element **carries the date the absence was asserted / recorded**
  (date of attestation), and recommending `occurrenceDateTime` for
  this purpose. Note that this is a deliberate IPS-specific
  interpretation of `occurrence[x]` for the absent-assertion case.
- Add a published example instance (`Immunization-no-known-uv-ips` or
  similar) under `fhir-ips` with all of the above populated, and
  reference it from the profile page.
- Reflect the comment from JIRAUSER25409 by adding an implementation
  note that consumers using SNOMED ECL must not rely on the
  inferred-parent relationship (`787482006 → 787480003 | No known
  procedures |`) to draw conclusions about the patient's procedure
  history.

#### Justification

This is what the reporter literally asked for and what the IPS Project
team's 2025-10-29 special call agenda is converging on (it bundles
"No known medications / No known immunizations / No known procedures"
as a single piece of work). Pinning `status = completed` and treating
`occurrence[x]` as the assertion date is consistent with how the same
absent-assertion pattern is being discussed for the related
required-if-known sections, and it stays inside the IPS IG's narrative
authoring surface (no base-spec change required). The published example
addresses the most concrete part of the ticket ("Please provide
clarification and an example.") and gives implementers something
unambiguous to copy. The SNOMED ECL caveat directly addresses the only
substantive comment on the ticket.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of leaning on `Immunization` to carry a "no known immunizations"
assertion at all, recommend representing the absence at the
**Composition section level**, and reserve resource-level "no known"
codes for the cases where the section pattern cannot be used. Concretely
in the IPS IG:

- In the Immunizations section narrative, document a primary pattern:
  when there are no known immunizations, the section MAY be conveyed
  with `Composition.section.emptyReason` populated from the IPS
  absent-unknown CodeSystem (mirroring how the IPS handles other
  required-if-known sections).
- Document a secondary pattern (resource-level assertion) using an
  `Immunization` resource with `vaccineCode = 787482006`,
  `status = completed`, and `occurrenceDateTime` = date of assertion,
  but mark it as appropriate only when a section-level emptyReason is
  not feasible (e.g., when the immunization data is being shipped
  outside an IPS Composition).
- Add the worked example for the secondary pattern as in Disposition
  A.

#### Justification

This separates the two questions FHIR-52277 conflates: *how to signal
that a section is empty* vs. *how to encode a clinical "no known"
assertion as a resource*. The Zulip thread "Interpreting 'Known Absent'
Information in Composition" explicitly argues that
`Composition.section.emptyReason` is the right place for the
section-empty signal, while resource-level "no known" codes carry
clinical weight. This approach also dodges the SNOMED-ECL issue raised
in the ticket comment, because most consumers will see an empty section
rather than an `Immunization` whose code's parent implies "no known
procedures". The trade-off is that implementers consuming IPS will need
to handle two patterns, and the reporter's request for a single clear
recipe for `Immunization.status` / `Immunization.occurrence[x]` is only
partially answered.

---

### Disposition C: Decline

#### Proposal

Decline as out-of-scope for the current IPS publication cycle, on the
grounds that the question is part of a broader, in-flight design
conversation across the IPS *required-if-known* sections (immunizations,
procedures, devices) and across IPS / Australian PS / national IPS
flavours. Per the existing resolution note ("Discussed on 9/3 IPS call.
Will revisit after 2.0 publication"), keep the issue in the
*Considered for Future Use* bucket and address it as part of a
coordinated post-2.0 work item that covers no-known medications,
no-known immunizations, and no-known procedures together, rather than
patching `Immunization (IPS)` in isolation.

#### Justification

The IPS Project team has already explicitly slotted this exact topic
(plus medications and procedures) into a special call agenda
(2025-10-29) driven by AU PS reconciliation feedback. Producing
narrative + examples for `Immunization` only — before that call's
output is folded back in — risks committing to a representation that
later has to be reversed for cross-section consistency, and could
create churn for downstream IGs (e.g., AU PS) that are awaiting the
joint outcome. The current resolution ("Considered for Future Use",
revisit after 2.0) already reflects this position.

---

### Recommendation

**Recommended disposition:** A

The reporter's ask is narrow and concrete — "what goes in
`Immunization.status` and `Immunization.occurrence[x]`, plus an
example" — and the IPS profile already commits to using SNOMED
`787482006` on `Immunization.vaccineCode`. Disposition B is
intellectually cleaner but reopens a settled design choice (resource-
level absent codes are already in the IPS profile and value set), and
Disposition C leaves implementers without guidance for an unbounded
period after 2.0. Disposition A delivers the requested narrative and
worked example, can be implemented entirely within the `fhir-ips`
authoring narrative for the Immunization (IPS) profile (no base-spec
change), and folds in the SNOMED-ECL caveat raised in the ticket
comment. It is also straightforward to extend later — the same
`status = completed` + `occurrenceDateTime = assertion date` pattern,
documented for Immunization, can be reused by the parallel "no known
medications" and "no known procedures" work coming out of the
2025-10-29 IPS / AU PS joint call, so adopting A now does not block
the broader cross-section harmonisation Disposition C is worried
about.
