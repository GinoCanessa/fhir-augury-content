## Summary

The ticket reports that the IPA (International Patient Access) Client and
Server CapabilityStatements declare `_revinclude=Provenance:target` on every
`CapabilityStatement.rest.resource` entry, but never attach a
`capabilitystatement-expectation` extension to those `_revInclude` entries.
Because IPA otherwise uses the expectation extension to mark conformance
strength (SHALL / SHOULD / MAY) for searches and includes, the absence of an
expectation here leaves implementers unsure whether `_revinclude=Provenance:target`
is a normative requirement, an encouraged capability, or merely an example.

The reporter (working on a CapabilityStatement comparison between AU Core and
IPA) asks the IPA work group to either tag each `_revinclude=Provenance:target`
entry with an explicit `capabilitystatement-expectation` (e.g., SHALL / SHOULD /
MAY), or to add narrative guidance explaining that the lack of expectation is
intentional. The IPA discussion (see the linked Zulip thread) and the recorded
"Persuasive with Modification" resolution converge on clarifying that there
is no SHALL requirement for `_revinclude=Provenance:target` in IPA — Provenance
is valuable but not part of the critical IPA data set, and downstream IGs
(e.g., IHE mXDE) can elevate the requirement when their use cases need it.

There are no other Jira tickets cross-referenced. The single related FHIR
artifact reference is to `CapabilityStatement.rest.resource`, which is the
element on which the missing expectation lives. There are no GitHub
cross-references, so no repository-level context applies.

## Details

**Description:**

The `_revinclude=Provenance:target` is declared across all
`CapabilityStatement.rest.resource` sections in the IPA specification.
However, the IPA CapabilityStatements do not include a
`capabilitystatement-expectation` extension to indicate whether support
for this `_revInclude` is **SHALL**, **SHOULD** or **MAY**. This makes it
unclear if IPA clients and servers are expected to support this
`_revInclude`, or is it possibly just an example?

Propose that the IPA CapabilityStatement clarify the conformance level for
`_revinclude=Provenance:target` by including the
`capabilitystatement-expectation` extension with an appropriate expectation
code (e.g. SHALL, SHOULD), or alternatively provide some guidance in the
documentation if the expectation level is intentionally unspecified.

**Resolution Description (from snapshot):**

Provenance is obviously important in a wide variety of use-cases; however, it
doesn't constitute the critical data set that's otherwise profiled and
described by IPA. Similarly, other national cores and (we believe) production
systems don't require/support it. We will clarify that there is no requirement
in IPA for implementers to support. See related discussion on Zulip for
additional background.

**Comments:**

- **isaac.vetter (2025-04-23):** Thanks DusicaBojicic for raising this. The
  IPA call discussed it and felt that, while Provenance is broadly important,
  it isn't part of the critical IPA data set — other national cores and
  production systems generally don't require/support it. The group is leaning
  toward clarifying that there is no SHALL requirement, while still
  encouraging support via MAY or SHOULD. Asks the reporter for a
  recommendation.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In every `CapabilityStatement.rest.resource` block of the IPA Client and IPA
Server CapabilityStatements that declares `searchRevInclude` of
`Provenance:target`, attach the
`http://hl7.org/fhir/StructureDefinition/capabilitystatement-expectation`
extension to the `searchRevInclude` element with `valueCode = SHALL`. Apply
the same edit consistently across every resource entry in both the client and
server CapabilityStatements so the conformance signal matches the rest of the
IPA expectation pattern. No narrative changes are required beyond a brief note
in the IPA CapabilityStatement page explaining that Provenance reverse-include
support is now mandatory.

#### Justification

This is the most literal reading of the request: it removes ambiguity by
giving `_revinclude=Provenance:target` the same expectation-extension
treatment that every other search/include in IPA already receives. It is also
internally consistent with how IPA expresses conformance throughout its
CapabilityStatements. However, the IPA call (per Isaac's comment and John
Moehrke's Zulip reply) explicitly judged Provenance to be outside the
critical IPA data set, and the recorded resolution is "Persuasive with
Modification" — so SHALL is stronger than the work group is willing to
require. This disposition is therefore a viable shape but at an expectation
level the work group has already softened away from.

---

### Disposition B: Alternative Approach

#### Proposal

Attach the `capabilitystatement-expectation` extension to every
`searchRevInclude` of `Provenance:target` in both the IPA Client and IPA
Server CapabilityStatements, but set `valueCode = MAY` (with a serious
consideration of `SHOULD` if the IPA work group prefers stronger
encouragement). In parallel, add a short narrative paragraph in the IPA
CapabilityStatement / conformance page explaining: (1) Provenance is not part
of the IPA critical data set, (2) servers and clients are not required to
support `_revinclude=Provenance:target`, (3) implementers — and downstream
IGs such as IHE mXDE — are encouraged to support it where their use cases
benefit, and (4) downstream IGs that depend on IPA may strengthen the
expectation (e.g., to SHALL) by deriving from these CapabilityStatements.
Apply the change uniformly across every resource block so behaviour is
consistent, and update any CapabilityStatement-derived conformance tables
generated by the IG publisher.

#### Justification

This matches the resolution already recorded on the ticket
("Persuasive with Modification") and the explicit consensus reached on the
linked Zulip thread between Isaac Vetter, John Moehrke, and the reporter
Dusica Bojicic. It also addresses the original problem the reporter
raised — ambiguity in CapabilityStatement comparisons (AU Core vs. IPA) —
by emitting an explicit, machine-readable expectation rather than relying on
silence. Calling out the downstream-IG elevation path (e.g., mXDE) directly
in the narrative protects the use cases John raised without forcing a SHALL
into IPA itself. This disposition trades a small amount of authoring effort
across many resource blocks for a clear, durable conformance signal and
removes the most common source of confusion for implementers comparing IPA
to other national cores.

---

### Disposition C: Decline

#### Proposal

Do not modify the IPA CapabilityStatements. Instead, add a single
clarifying note to the IPA CapabilityStatement / conformance documentation
stating that `_revinclude=Provenance:target` is intentionally listed without
a `capabilitystatement-expectation` extension because Provenance is not part
of the IPA critical data set, and that implementers should treat the absence
of an expectation as "no requirement" (effectively MAY). Close the ticket
as documentation-only, with no machine-readable change to the
CapabilityStatement instances.

#### Justification

This is the lightest-touch option and avoids touching every
`CapabilityStatement.rest.resource` block. It is defensible because the IPA
work group has already concluded there is no SHALL requirement, so a prose
clarification arguably conveys the same information. However, the original
report was prompted precisely by an automated CapabilityStatement comparison
(AU Core ↔ IPA) that cannot read narrative — leaving the expectation
unspecified means tooling will keep flagging this gap. Declining the
machine-readable fix preserves the very ambiguity the ticket is trying to
remove, so this option is hard to justify against the recorded resolution.

---

### Recommendation

**Recommended disposition:** B

Disposition B implements the resolution the IPA work group has already
recorded ("Persuasive with Modification" — clarify that there is no SHALL
requirement) in the form the original reporter actually needs: a
machine-readable `capabilitystatement-expectation` (MAY, optionally SHOULD)
attached to the `Provenance:target` reverse-include in every IPA Client and
IPA Server CapabilityStatement resource block, plus a short narrative
explaining the reasoning and the path for downstream IGs (e.g., IHE mXDE)
to elevate the requirement. It directly resolves the AU Core ↔ IPA
comparison gap that motivated the ticket, matches the on-thread consensus
between Isaac Vetter, John Moehrke, and Dusica Bojicic, and avoids both
the over-commitment of Disposition A (forcing SHALL where the work group
has explicitly stepped back from it) and the under-delivery of Disposition C
(leaving the machine-readable conformance signal absent — the very condition
the reporter and downstream tooling are trying to fix).
