## Summary

This Change Request, raised by Smita Hastak on behalf of the BR&R WG sponsored
Pharmaceutical Quality / Chemistry, Manufacturing & Controls (PQ/CMC) project,
asks Patient Care to broaden the **description and scope of the Procedure
resource** (section 9.3 of the FHIR Core spec) so that `Procedure` can also
represent the manufacturing of a **drug product (medication)** and a **drug
substance (substance)**. Concretely, the reporter proposes adding `medication`
and `substance` to the enumerated list of subjects in the resource narrative
("An action that is or was performed on or for a patient, practitioner, device,
organization, *medication, substance* or location…") and appending a sentence
clarifying that "*This can be manufacturing of a drug product (medication) or a
drug substance.*" The only cross-reference recorded against this ticket is to
the BR&R WG's PQ/CMC Confluence project page, which is the project context but
not a separate discussion of this specific edit. No related Jira tickets,
Zulip threads, or GitHub items were found.

## Details

**Description:**

> For the BR&R WG sponsored PQ/CMC project, we plan to leverage the Procedure
> resource to support the drug product (medication) and drug substance
> (substance) manufacturing process. To support this use case, we request that
> the description and scope of this resource be broadened. Please see suggested
> edits to the resource description below (edits in blue font).
>
> An action that is or was performed on or for a patient, practitioner, device,
> organization, **medication, substance** or location. For example, this can be
> a physical intervention on a patient like an operation, or less invasive like
> long term services, counseling, or hypnotherapy. This can be a quality or
> safety inspection for a location, organization, or device. This can be an
> accreditation procedure on a practitioner for licensing. **This can be
> manufacturing of a drug product (medication) or a drug substance.**

(Bold text above corresponds to the additions the reporter highlighted in blue
in the original ticket.)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's wording verbatim in the Procedure resource description /
scope (FHIR Core, section 9.3 — Procedure):

1. In the first sentence of the resource description, add `medication` and
   `substance` to the enumerated list of subjects, so it reads:
   *"An action that is or was performed on or for a patient, practitioner,
   device, organization, medication, substance or location."*
2. Append a new sentence to the same paragraph:
   *"This can be manufacturing of a drug product (medication) or a drug
   substance."*

This is a **narrative-only** change to the Procedure resource page (the
resource scope text); it does not modify the StructureDefinition, the
cardinality or type of `Procedure.subject`, or any invariants. No new code
system, value set, or extension is introduced as part of this disposition.

#### Justification

The change is narrative and additive. It does not alter the wire format,
existing implementations, conformance, or downstream profiles, so backward
compatibility risk is essentially nil. Broadening the documented scope to
acknowledge a new, real regulatory use case sponsored by another HL7 work
group (BR&R WG, PQ/CMC, FDA submissions) is consistent with how Procedure's
scope has historically been described as deliberately broad ("a physical
intervention… a quality or safety inspection… an accreditation
procedure…"). Recognising manufacturing-of-medication / manufacturing-of-
substance in the prose helps PQ/CMC implementers cite Procedure as the
intended resource for their use case without waiting on a structural change.

### Disposition B: Alternative Approach

#### Proposal

Accept the **intent** but make two adjustments before merging:

1. **Confirm `Procedure.subject` actually permits `Medication` and
   `Substance` references** before changing the prose. Today
   `Procedure.subject` is typed as `Reference(Patient | Group)` in R5
   (and historically `Reference(Patient | Group)` in R4 / R4B). If the
   PQ/CMC use case really needs the *subject* of the Procedure to be a
   Medication or a Substance, the narrative change in Disposition A is
   misleading on its own — implementers would read "medication, substance
   or location" in the description but be unable to express it without
   either (a) extending `Procedure.subject` to allow `Reference(Medication
   | Substance | Location)` (a normative-track change), or (b) using a
   standard extension such as `procedure-targetBodyStructure`-style
   patterns or a Patient-Care-defined `procedure-subject` extension.
2. Make the narrative edit **conditional on choosing a target mechanism**:
   either expand `Procedure.subject`'s allowed reference types in a
   future ballot cycle, or define an extension under the BR&R/PQ/CMC IG
   that constrains/extends `Procedure` for manufacturing. The Procedure
   resource description would then describe both options ("manufacturing
   of a drug product or drug substance is supported in this release via
   the `<extension-name>` extension; subject-typing changes are tracked
   under FHIR-XXXXX").

This keeps the door open to PQ/CMC's use case without misrepresenting
current resource capability.

#### Justification

The literal request changes only the prose, but the prose lists subjects
that the current StructureDefinition does not actually allow as
`Procedure.subject`. Accepting Disposition A in isolation creates a known
spec/implementation mismatch — implementers will follow the broadened scope
text, attempt `Procedure.subject = Reference(Medication/...)`, and fail
validation. Disposition B preserves the BR&R WG's manufacturing use case
while keeping the resource description faithful to what the
StructureDefinition allows in this release, and gives Patient Care a clear
hand-off to either a normative widening of `Procedure.subject` or an IG-
level extension owned by BR&R / the PQ/CMC project. It also explicitly
involves the BR&R WG (the sponsoring WG) in choosing the mechanism rather
than only the prose.

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that **manufacturing of a drug product
or drug substance is out of the documented scope of the Procedure
resource** as currently specified, and that the appropriate place to model
manufacturing of a medicinal product is the BR&R-owned medication /
substance resource family — for example, the
**`MedicinalProductDefinition`** family in R5
(`MedicinalProductDefinition`, `ManufacturedItemDefinition`,
`PackagedProductDefinition`, `Ingredient`, `RegulatedAuthorization`,
`SubstanceDefinition`) — together with any PQ/CMC-specific implementation
guide the BR&R WG produces. Recommend that BR&R model the manufacturing
process through that resource family or via a PQ/CMC IG, rather than
broadening `Procedure`'s documented scope.

#### Justification

`Procedure` is, by long-standing design, a record of an action performed
*on a patient (or group)*; its `subject` is `Reference(Patient | Group)`.
Broadening the prose to imply manufacturing-of-a-medication is a `Procedure`
without also changing the StructureDefinition would be misleading, and
broadening the StructureDefinition itself is a substantive scope change
that touches many existing profiles, search parameters
(`Procedure.subject` reference targets), and implementer expectations
across the ecosystem. The R5 `MedicinalProductDefinition` family was
specifically introduced to describe regulated manufactured items, their
ingredients, and their regulatory authorizations — i.e., the PQ/CMC space.
Routing the use case there avoids stretching `Procedure` semantics and
keeps responsibility within the BR&R WG, which sponsors PQ/CMC.

### Recommendation

**Recommended disposition:** B

The reporter's underlying need — a place in FHIR Core to clearly indicate
that manufacturing of a drug product or drug substance is a recognised
use case — is legitimate and is sponsored by another HL7 WG (BR&R), so an
outright decline (Disposition C) is heavier than warranted; it would push
PQ/CMC entirely off Procedure without first asking whether Patient Care
is willing to accommodate the use case. However, accepting the narrative
change exactly as written (Disposition A) creates a documentation /
StructureDefinition mismatch, because `Procedure.subject` does not
currently allow `Medication` or `Substance` references. Disposition B
keeps Patient Care's prose honest by tying the narrative broadening to a
concrete mechanism — either a future widening of `Procedure.subject`'s
allowed reference types or a BR&R-defined extension on `Procedure` for
PQ/CMC — and gives the workgroup a clean way to coordinate with BR&R
before merging the description change. Patient Care should respond to
the reporter asking which mechanism BR&R prefers, and resolve this
ticket with a follow-up either to extend `Procedure.subject` or to
register an extension/IG, in addition to (or instead of) the prose
edit.
