## Summary

The submitter (Gabriel Leonhartsberger) is asking the Patient Care work group
to refine the *context of use* and version applicability metadata of the
`bodySite` extension (`http://hl7.org/fhir/StructureDefinition/bodySite`) in
the FHIR Extensions Pack so that it is only advertised on resources/versions
where there is no native equivalent. As FHIR has evolved, several resources
have grown a first-class `bodyStructure` (or `CodeableReference(BodyStructure)`)
element, which makes the extension redundant — and potentially misleading —
on those versions. The reporter proposes per-resource version cut-offs:

- `Condition` — extension relevant up to R5 (native `bodyStructure` added in R6)
- `Observation` — extension relevant up to R4B (native `bodyStructure` added in R5)
- `Procedure` — extension relevant up to R5 (native `bodyStructure` added in R6)
- `Specimen` — extension relevant up to R4B (`bodySite` became
  `CodeableReference(BodyStructure)` in R5)
- `MedicationAdministration` — extension still relevant (no version has added
  a native body-site element)
- `Dosage` — extension still relevant (no version has added a native body-site
  element)

The first comment (Gabriel) confirms the extension in question is the
canonical `bodySite` extension. Grahame Grieve replied that once the specific
extension is identified he can advise how to express the per-context /
per-version applicability in the extension definition itself. There are no
explicitly-linked Jira tickets and no related GitHub items. The only
cross-reference is the FMG "Extensions Pack" Zulip topic where this ticket is
listed (alongside many sibling Extensions Pack tickets) for triage.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `bodySite` extension (`http://hl7.org/fhir/StructureDefinition/bodySite`)
in the FHIR Extensions Pack so that its `StructureDefinition.context` /
applicability is constrained per the reporter's table:

- `Condition` — context applies for `fhirVersion` ≤ R5 only
- `Observation` — context applies for `fhirVersion` ≤ R4B only
- `Procedure` — context applies for `fhirVersion` ≤ R5 only
- `Specimen` — context applies for `fhirVersion` ≤ R4B only
- `MedicationAdministration` — context retained for all FHIR versions
- `Dosage` — context retained for all FHIR versions

Mechanically this should be expressed using the per-context version-bounding
that Grahame indicated can be done in the extension definition (per-context
`fhirVersion` constraints / `extension-context-fhirVersion` style metadata),
plus a note in the extension's narrative explaining that on later versions
implementers should use the native `bodyStructure` element (or
`CodeableReference(BodyStructure)` for `Specimen` ≥ R5). Add the cut-over
guidance to the extension's "Notes" / "Usage" panel.

#### Justification

This is exactly the policy already adopted across the Extensions Pack: an
extension exists to back-port functionality that is otherwise unavailable in
a given FHIR version, and once the core spec carries an equivalent element
the extension should *not* be advertised on that resource/version. The
reporter's per-resource version mapping is correct against the published
specs (`Observation.bodyStructure` in R5; `Condition.bodyStructure` and
`Procedure.bodyStructure` in R6; `Specimen.bodySite` typed as
`CodeableReference(BodyStructure)` in R5). Keeping the extension applicable
on `MedicationAdministration` and `Dosage` is consistent with there being no
native equivalent there. Grahame's comment on the ticket effectively
pre-approves this approach pending confirmation of which extension was
meant — which the reporter has now supplied.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than narrowing `context` per resource/version, **deprecate the
`bodySite` extension wholesale on resources that have a native equivalent in
*any* currently-published version**, and replace its narrative with a
"Migration Guidance" section that points implementers to the native element
on each resource/version. Concretely:

- Remove `Condition`, `Observation`, `Procedure`, and `Specimen` from
  `context` entirely (mark them deprecated rather than version-bounded).
- Retain `MedicationAdministration` and `Dosage` as the only active contexts.
- Add a usage table in the extension's page that says "for *Condition* on
  R4/R4B/R5, use this extension; on R6+, use `Condition.bodyStructure`" so
  the historical guidance is preserved without keeping the extension
  formally advertised on those contexts in the current Extensions Pack
  build.

#### Justification

The Extensions Pack is published as a single artifact rather than per FHIR
version, and tooling support for *per-context* `fhirVersion` constraints is
uneven (validators, IG publisher, profile editors). A simpler "deprecate the
context, document the migration" model is easier to implement, easier to
test, and harder to misuse — implementers on older versions are still served
by reading the migration table, while implementers on R5/R6 see a clean
extension that is no longer advertised on resources where the core spec
covers the use case. The trade-off vs. Disposition A is that a strict R4
implementer who validates against the current Extensions Pack would no
longer see `Condition` listed as a valid context at all, so they would need
to rely on the documented migration table instead of a machine-checkable
context. This is the same trade-off that several sibling Extensions Pack
tickets (e.g. FHIR-52827 "Many extensions are deprecated based on R6",
FHIR-44371) are wrestling with.

---

### Disposition C: Decline

#### Proposal

Decline the request and leave `bodySite`'s `context` as currently published
(applicable to `Condition`, `Observation`, `Procedure`, `Specimen`,
`MedicationAdministration`, `Dosage`). Document — in the extension's page
narrative only — that on FHIR versions where the resource has a native
`bodyStructure`/`CodeableReference(BodyStructure)`, implementers SHOULD
prefer the native element.

Rationale category: *already addressed by existing mechanism* — the FHIR
core specs themselves already make the native element the obvious choice on
R5+/R6+; the extension's continued presence on `Observation` etc. is harmless
in older-version contexts and is widely deployed.

#### Justification

A defensible reason to decline is backward-compatibility risk: tightening
`context` on a long-published extension can break existing R4 instances and
profiles that reference the extension on these resources, or that derive
from it. Some implementers may also be using the extension on R5+ data for
historical / migration reasons (e.g. data that originated as R4 and is being
projected forward). A narrative-only clarification preserves all existing
behavior while still steering new implementations toward the native element.
The clear weakness is that it leaves the extension's machine-readable
metadata misaligned with the spec's own advice and does nothing to address
the reporter's underlying concern that the extension is being recommended in
contexts where it is no longer the right answer.

---

### Recommendation

**Recommended disposition:** A

The reporter's per-resource version table is factually correct and mirrors
the policy the Extensions Pack already applies elsewhere: extensions are
meant to back-port, not to compete with, native FHIR elements. Grahame
Grieve has already signalled on the ticket that the per-context /
per-version applicability *can* be expressed in the extension definition,
which removes the main objection that would otherwise push the work group
toward Disposition B. Disposition A also has the smallest "policy surface"
— it adopts the change the reporter actually asked for, keeps the
machine-checkable `context` aligned with the human-readable guidance, and
avoids the broader "deprecate vs. version-bound" debate that sibling tickets
(FHIR-52827, FHIR-44371) are tracking. If, during implementation, the
per-context `fhirVersion` mechanism turns out to be unworkable in tooling,
the work group can fall back to Disposition B as a documented compromise;
Disposition C should only be picked if the work group has concrete evidence
of implementer reliance on the current contexts that would be broken by
narrowing them.
