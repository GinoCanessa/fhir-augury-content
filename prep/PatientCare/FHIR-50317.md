## Summary

The PACIO Transitions of Care (TOC) Implementation Guide currently fixes
`Composition.category.code` to LOINC `42348-3` ("Advance healthcare
directives"). The reporter (Rob McClure) points out that this is
semantically wrong: a PACIO Transitions of Care document is not an
advance healthcare directive, so binding the TOC Composition profile's
category to that LOINC code mislabels every conformant TOC document.
The ticket asks the work group to correct the fixed LOINC code on the
TOC Composition profile so that it identifies a Transitions of Care
document rather than an advance directive. The ticket has already been
worked through to a "Persuasive" resolution proposing LOINC `18761-7`
("Transfer Summary Note") as the replacement, and is on Block-Vote-1.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items in the Augury index for this issue; the only cross-reference is
to the FHIR artifact `Composition.category.code` itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already drafted on the ticket: change the fixed
value of `Composition.category.code` on the PACIO TOC Composition
profile from LOINC `42348-3` ("Advance healthcare directives") to
LOINC `18761-7` ("Transfer Summary Note"). Concretely, in the TOC
Composition profile (StructureDefinition), update the
`Composition.category.coding` slice that fixes
`system = http://loinc.org` so that `code = 18761-7` and
`display = "Transfer summary note"`, regenerate the IG, and update any
narrative, examples, and ValueSet expansions that reference `42348-3`
in the TOC Composition context. Add a brief change-log entry noting
the technical correction and the rationale (the prior code mislabeled
TOC documents as advance directives).

#### Justification

The current binding is factually incorrect: a Transitions of Care
document is not an advance healthcare directive, so every conformant
TOC instance is being mis-typed at the document level. LOINC `18761-7`
"Transfer Summary Note" is a recognized Clinical LOINC document type
whose scope description ("generated by the originating institution …
the facility the patient is being transferred out of") aligns with
PACIO TOC's stated use case. The resolution is already marked
**Persuasive** and the ticket carries the **Block-Vote-1** label,
indicating the work group has converged on this remedy. Treating it as
a Technical Correction is appropriate because the change fixes a
factual error rather than introducing new functionality.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of fixing `Composition.category.code` to a single LOINC code,
replace the fixed value with a small **required ValueSet binding** on
`Composition.category` that enumerates the document types the PACIO
TOC IG is intended to cover (minimally LOINC `18761-7` "Transfer
Summary Note", with room to add other LOINC transfer/transition
document codes such as `18842-5` "Discharge Summary" if the
work group considers them in scope). Slice `Composition.category` so
that one slice is required and bound to that ValueSet, and explicitly
prohibit `42348-3` via an invariant or by virtue of the bound
ValueSet. Update profile narrative and examples accordingly.

#### Justification

A fixed code at the Composition level is brittle: if PACIO ever needs
to express a closely related transition document (e.g., discharge
summary, referral note) under the same IG, another technical
correction will be required. A small ValueSet-bound `category` slice
preserves the strong typing the original authors wanted while leaving
room to evolve without re-balloting a fixed value. It also makes the
intent of the IG ("these document codes are in-scope") explicit in a
single artifact (the ValueSet) rather than buried in the
StructureDefinition. The trade-off versus Disposition A is more
implementation churn and a slightly looser conformance contract; for
implementers already targeting the existing fixed value, migration is
no harder than for Disposition A.

---

### Disposition C: Decline

#### Proposal

Decline to change the fixed code at this time, on the basis that the
mis-bound value is a known errata that can be addressed via an IG
errata page rather than a normative change to
`Composition.category.code`, and that any change to a fixed value on a
balloted profile is a breaking change for existing implementers.

#### Justification

This position is hard to defend on the merits: the current code
genuinely mislabels TOC documents as advance directives, which has
real safety/communication implications (a downstream system filtering
on `42348-3` would either pull in TOC documents as advance directives
or, conversely, hide TOC documents from advance-directive workflows
when the reverse is intended). Errata-only handling would leave the
machine-readable artifact wrong while only the human-readable narrative
is corrected, which is exactly the opposite of how FHIR conformance is
consumed. Decline is listed for completeness but is not recommended.

---

### Recommendation

**Recommended disposition:** A

The ticket is a clear factual correction with an already-agreed
resolution ("Persuasive"), is queued on **Block-Vote-1**, and the
proposed code (`LOINC 18761-7` "Transfer Summary Note") matches the
stated PACIO TOC use case. Accepting as requested gets the IG to a
correct, conformant state with the smallest possible change to
existing implementations: anyone who built against `42348-3` was
building against an incorrect fixed value and must update regardless
of which option is chosen, and switching to another single fixed code
keeps the implementation contract just as tight as it is today.
Disposition B (ValueSet binding) is defensible if the work group
believes PACIO TOC will soon need to cover additional transition
document types, but that is a scope expansion that should be raised
as its own ticket rather than bundled into a technical correction;
holding the fix hostage to that broader discussion would delay an
already-converged remedy. Disposition C should be rejected because
leaving an incorrect fixed LOINC code in a normative profile is not a
defensible end state.
