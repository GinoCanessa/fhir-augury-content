## Summary

In the US SDOH Clinical Care IG (STU2.1), the `SDOHCC-Condition` profile
declares the invariant `SDOH-Con-1`, which is intended to enforce that a
Condition has at most one asserter — counted across the standard
`Condition.asserter` element and the IG's device-asserter extension.
The FHIRPath expression for the invariant references the extension by
the wrong canonical URL
(`http://hl7.org/fhir/us/sdoh-clinicalcare/extension-Condition.asserter`),
which does not match the actual extension canonical
(`http://hl7.org/fhir/us/sdoh-clinicalcare/StructureDefinition/SDOHCC-ExtensionConditionAsserter`).
As a result, `extension(...)` returns no matches and the invariant
silently degenerates to `asserter.count() <= 1`, never catching the
"both asserter and device-asserter extension supplied" case it was
designed to prevent. The reporter (Corey Spears) asks that the
invariant expression be corrected to use the real extension canonical.
The issue was first surfaced by Brian Postlethwaite while reviewing
the profile, then opened as a ticket; a fix has already been proposed
upstream as PR
[HL7/fhir-sdoh-clinicalcare#150](https://github.com/HL7/fhir-sdoh-clinicalcare/pull/150).

There are no other Jira tickets cross-linked to this issue.

## Details

**Description:**

> In the [SDOHCC Condition profile](http://hl7.org/fhir/us/sdoh-clinicalcare/STU2.1/StructureDefinition-SDOHCC-Condition.html),
> the invariant `SDOH-Con-1` describes the requirement that there be a
> maximum of one asserter (either `Condition.asserter` or the extensions
> for a device asserter). However, this invariant expression uses the URL
> `http://hl7.org/fhir/us/sdoh-clinicalcare/extension-Condition.asserter`
> to identify the extension. However, the canonical for the extension is
> `http://hl7.org/fhir/us/sdoh-clinicalcare/StructureDefinition/SDOHCC-ExtensionConditionAsserter`,
> so this invariant will not work as intended.
>
> Change the SDOH-Con-1 expression to:
>
> `extension('http://hl7.org/fhir/us/sdoh-clinicalcare/StructureDefinition/SDOHCC-ExtensionConditionAsserter').count()+asserter.count()<=1`
>
> There is an invariant that I expect is intending to reference the
> asserter extension, however uses the wrong canonical URL. Invariant:
>
> `extension('http://hl7.org/fhir/us/sdoh-clinicalcare/extension-Condition.asserter').count()+asserter.count()<=1`
>
> Uses: `http://hl7.org/fhir/us/sdoh-clinicalcare/extension-Condition.asserter`
> However the correct canonical URL should be:
> `http://hl7.org/fhir/us/sdoh-clinicalcare/StructureDefinition/SDOHCC-ExtensionConditionAsserter`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the reporter's exact change in the SDOH Clinical Care IG
(`HL7/fhir-sdoh-clinicalcare`): update the `SDOH-Con-1` invariant on
the `SDOHCC-Condition` profile so its FHIRPath expression references
the real extension canonical, i.e. change

```
extension('http://hl7.org/fhir/us/sdoh-clinicalcare/extension-Condition.asserter').count() + asserter.count() <= 1
```

to

```
extension('http://hl7.org/fhir/us/sdoh-clinicalcare/StructureDefinition/SDOHCC-ExtensionConditionAsserter').count() + asserter.count() <= 1
```

Apply this change in the FSH source that declares the invariant on
`SDOHCC-Condition` (verify the exact path via PR
[#150](https://github.com/HL7/fhir-sdoh-clinicalcare/pull/150)) and
release the corrected `StructureDefinition-SDOHCC-Condition` in the
next IG publication. Optionally add a publication-note / change-log
entry calling out the constraint behaviour change so implementers can
re-validate existing Condition instances.

#### Justification

This is a straightforward erratum: the extension URL referenced in the
expression simply does not exist on any instance, so the
`extension(...).count()` term is always `0` and the constraint is not
enforcing what the IG narrative says it enforces. Fixing the URL
restores the constraint to its documented intent without changing the
profile's structure, cardinalities, or any other artifact, and matches
the canonical that the IG itself publishes for
`SDOHCC-ExtensionConditionAsserter`. The reporter is the IG's own
contributor and an upstream PR (#150) already encodes exactly this
change, so the change is low-cost and ready to merge.

---

### Disposition B: Alternative Approach

#### Proposal

Fix the invariant, but generalise the expression so it survives any
future renames of the extension canonical and keeps the
implementer-friendly behaviour of "treat any
`SDOHCC-ExtensionConditionAsserter`-shaped extension as an asserter".
Concretely, replace the hard-coded URL lookup with a slice-/profile-
aware form, e.g.:

```
(extension.where(url.toString().endsWith('SDOHCC-ExtensionConditionAsserter')).count() + asserter.count()) <= 1
```

or, if the profile slices the extension by the canonical (preferred
SUSHI pattern), assert the constraint against the slice name instead
of the raw URL:

```
(extension('SDOHCC-ExtensionConditionAsserter').count() + asserter.count()) <= 1
```

— having first declared the slice in the profile so the short-name
form resolves. Apply in the same FSH file as Disposition A; supersede
PR #150 with the slice-based form, or amend PR #150.

#### Justification

The underlying defect is that the invariant referenced a URL by hand
and got it wrong; that's a class of bug, not a one-off. Driving the
constraint off the slice (or off a tail-match) reduces the chance of
the same defect recurring next time the extension is renamed,
re-published, or moved into a sibling IG. The trade-off is added
profile complexity (a slice has to exist and be authored correctly)
and a slightly more permissive matcher; against that, the recipe
matches the SUSHI-idiomatic style for "constrain by named extension"
and is easier for implementers to read.

---

### Disposition C: Decline

#### Proposal

Decline as a profile change and instead resolve the ticket by
publishing an erratum / known-issue entry on the SDOH Clinical Care
STU2.1 page that (a) documents that `SDOH-Con-1` does not currently
constrain the device-asserter extension, and (b) advises implementers
to enforce the asserter-cardinality rule themselves until the next
substantive IG release.

#### Justification

The rationale would be that STU2.1 is published, the constraint is
"merely advisory" in current behaviour (it never fires for the
extension case), and altering an invariant on a published artifact is
a behaviour change for downstream validators that may flag previously
"passing" instances. This is a defensible position for declining
*now*, but it is weak: the expression as written is objectively a bug
relative to the IG's own narrative, and "do nothing" leaves a known-
incorrect constraint in the published spec.

---

### Recommendation

**Recommended disposition:** A

The ticket is a narrowly-scoped erratum on a single FHIRPath string
inside one invariant in one profile. The reporter has supplied the
exact corrected expression, the bug is independently confirmed in the
linked Zulip thread by Brian Postlethwaite, and an upstream PR
([#150](https://github.com/HL7/fhir-sdoh-clinicalcare/pull/150))
already encodes the change. Disposition B's slice-based form is
attractive in principle but is best handled as a follow-up
hardening — it expands the scope from "errata" to "profile redesign of
the asserter slot" and would benefit from its own ticket. Disposition
C is not really viable because the IG's published narrative already
asserts the constraint; leaving the broken expression in place
contradicts the spec's own text. Patient Care should accept FHIR-43313
as requested, merge PR #150 (or the equivalent FSH edit), and ship
the fix in the next SDOH CC publication, with a brief change-log
entry warning implementers that the constraint will now actually
fire.
