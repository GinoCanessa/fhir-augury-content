## Summary

The reporter requests that a `dosageInstructions` element with cardinality
`0..*` be added to the `Procedure` resource so that blood products (and other
non-traditional "administered" items) can be administered through one or more
procedures linked back to a `ProcedureRequest`/`ServiceRequest`. The underlying
clinical drivers are blood transfusions and other applications of biologically
derived products (BDPs) such as cellular therapies (e.g., CD34+ stem cells
dosed in cells/kg over multiple days), where today the `Procedure` resource
cannot carry the dose actually delivered and `MedicationAdministration` does
not fit cleanly because the "product" is not a coded medication.

The discussion has expanded well beyond the original literal request and is
now tightly coupled to several related Patient Care and Orders & Observations
trackers: [FHIR-12993](https://jira.hl7.org/browse/FHIR-12993) (asks for a
generic Administration / NonMedicationAdministration resource that would cover
blood products, oxygen/ventilator therapy, dialysis, line/tube placement,
etc.), [FHIR-8458](https://jira.hl7.org/browse/FHIR-8458) (asks how to express
quantity for a transfusion — Procedure or MedicationAdministration?),
[FHIR-51207](https://jira.hl7.org/browse/FHIR-51207) (raises that the BDP
resource family is incomplete — application/transplant/transfusion are recorded
inconsistently as Procedure, Observation, or MedicationStatement), and the
companion OO ticket
[FHIR-55923](https://jira.hl7.org/browse/FHIR-55923) (requests an extension on
`ServiceRequest` aligning with `MedicationDispense.dosageInstructions` using
the new `DosageDetails` data type, with Procedure expected to follow). Recent
PCWG and joint OO/Pharmacy/PC meetings have converged on (a) waiting for OO to
land its non-medication-product resources before adding any
`Procedure.dosage`-style structure, and (b) preferring an extension on
`Procedure` (modelled on `MedicationAdministration.dosage`) over a normative
backbone, because R6 is approaching normative status for Procedure.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a `dosage` element to `Procedure` modelled on
`MedicationAdministration.dosage`, but as a true core element rather than the
literal `dosageInstructions` (which is a request-side type). Concretely,
extend the existing `Procedure.used` backbone (already
`CodeableReference(Device | Medication | Substance |
BiologicallyDerivedProduct)`) so that each `Procedure.used` entry can carry a
`dosage` BackboneElement (`Procedure.used.dosage 0..1 BackboneElement`) that
mirrors `MedicationAdministration.dosage` (dose, rate[x], route, method, site,
text). This preserves a 1:1 association between the substance/product
"used" and the dose actually delivered, addressing the inference-safety
concern raised by chuscmi. Cardinality remains `0..1` per used item, with
`Procedure.used` itself being `0..*` so that multi-dose procedures (e.g.,
cellular therapy across days) are representable.

#### Justification

Aligns with the modelling already sketched in PC's 2025-10-16 minutes
(`Procedure.used.item` + `Procedure.used.dosage` like MedAdmin) and keeps the
dose tied to the specific substance/product, satisfying transfusion-lab
traceability requirements raised by JIRAUSER18752. Reuses an existing
backbone rather than introducing a parallel concept, and avoids the
costateixeira concern that "DosageInstructions" is the wrong (request-side)
type by using the event-style `MedicationAdministration.dosage` shape (or, if
preferred, the R6 `DosageDetails`). Lets implementers exchange transfusion
and cellular-therapy dosing today via `Procedure` without inventing a new
resource.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** modify the `Procedure` resource itself. Instead:

1. Publish a non-normative **extension** on `Procedure` (in the FHIR
   Extensions Pack) with the same shape as `MedicationAdministration.dosage`,
   bound to `Procedure.used.item` so each used product can carry an
   administered dose. This mirrors the OO direction in
   [FHIR-55923](https://jira.hl7.org/browse/FHIR-55923) for `ServiceRequest`
   and is exactly the path PC's 2025-10-16 minutes converged on.
2. Hold any base-resource change on `Procedure` until OO finishes the
   `NonMedicationProductAdministration` / `NonMedicationProductDispense` /
   `NonMedicationProductRequest` resource family
   ([FHIR-51207](https://jira.hl7.org/browse/FHIR-51207),
   [FHIR-12993](https://jira.hl7.org/browse/FHIR-12993)). The longer-term
   recommendation is that BDP/blood transfusions move to those new resources
   rather than being squeezed into `Procedure`.
3. Close FHIR-13047 as Persuasive with Modification once the extension is
   published, and link it to the OO resource-proposal tickets for the
   long-term path.

#### Justification

Procedure is targeted to be normative in R6; PC's own discussion (Russ /
Stephen) flagged that adding a backbone element now would lock in a structure
before the modelling question (Procedure vs. MedicationAdministration vs.
NonMedicationProductAdministration) is resolved. An extension is reversible,
costs implementers nothing today, and matches the OO pattern for
`ServiceRequest.dosageInstructions` extension, keeping the request and event
sides symmetrical. It also respects costateixeira's position that
administration semantics belong outside Procedure proper, and the
JIRAUSER18752/Elliot Silver point that BDP traceability really wants its own
resource family.

---

### Disposition C: Decline

#### Proposal

Decline the request to add `dosageInstructions` (or any dosage element)
directly to `Procedure`. Direct implementers to:

- Use `MedicationAdministration` (with `medication.concept`/textual
  CodeableReference where no Medication code exists) and
  `MedicationAdministration.partOf Reference(Procedure)` to record perioperative
  medications and transfusions delivered as part of a surgical procedure.
- Follow the OO BDP resource family
  ([FHIR-51207](https://jira.hl7.org/browse/FHIR-51207),
  [FHIR-12993](https://jira.hl7.org/browse/FHIR-12993)) for biologically
  derived product administration once available.
- For now, capture quantity/dose narratively in `Procedure.note` or via
  `Procedure.used.item` plus a paired `MedicationAdministration` /
  `Observation` when needed.

Mark FHIR-13047 as **Not Persuasive — addressed by other mechanisms** and
keep it linked to the BDP / NonMedicationAdministration tickets for tracking.

#### Justification

Per costateixeira and chuscmi, `Procedure` is an event resource and was not
designed to carry administration semantics; conflating dose with a procedure
introduces inference-only coupling between `Procedure.used*` and any new
dosage field, which is clinically risky for transfusion. Both R4 and R5
`MedicationAdministration.medication` already accept a free-text
CodeableConcept/CodeableReference, so the "no medication code for BDPs"
objection is partially mitigated today. Adding the field invites duplication
with `MedicationAdministration.partOf Reference(Procedure)` and risks two
divergent ways to record the same clinical event right before R6 normative.

---

### Recommendation

**Recommended disposition:** B

After nine years of discussion, every recent PC and joint PC/OO/Pharmacy
conversation (Sept 2025 WGM, 2025-10-16 PC call, 2026-04-02 PC call) has
landed on the same compromise: an extension now, base-resource changes later,
once OO's NonMedicationProduct resource family is in place. Disposition B
codifies that consensus, keeps PC and OO symmetric (mirroring FHIR-55923's
extension on ServiceRequest), and avoids freezing a structure into Procedure
just before it goes normative in R6. Disposition A is the cleanest model of
the underlying clinical request and remains a valid Post-R6 evolution path,
but accepting it now contradicts the explicit warnings from Stephen, Russ,
costateixeira, and the "hold until OO lands the substance resources"
guidance. Disposition C reflects costateixeira's pure-event-modelling
position but ignores the practical barrier that BDPs and cellular therapies
have no Medication code, leaving real implementers without a workable answer
for years. Recommend Disposition B with a follow-up tracker (or revival of
[FHIR-12993](https://jira.hl7.org/browse/FHIR-12993)) to drive the long-term
NonMedicationProductAdministration work, to be confirmed at the Rotterdam
WGM where OO has already asked to pick the topic up again.
