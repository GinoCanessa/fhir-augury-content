## Summary

The reporter notes that `ClinicalImpression` is classified as an event-type
resource but is missing two of the standard Event pattern elements —
`ClinicalImpression.partOf` and `ClinicalImpression.basedOn` — and asks
Patient Care to consider aligning the resource with the rest of the Event
pattern for consistency. The original use case driving the request was
A&E (accident & emergency) triage in Sweden, where the full triage
assessment was being modelled as a `ClinicalImpression` (see the Cambio
implementation guide linked under Related URLs). The resource is at a
low FMM level and the workgroup has explicitly invited implementer
feedback, which is what prompted the request. In a 2023 follow-up the
reporter indicated they had moved off that project, that nothing is
currently blocking them, and that the workgroup is free to close the
ticket if there is no broader interest in the consistency change. No
related Jira tickets, Zulip threads, or GitHub items were surfaced by
the cross-reference index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Align `ClinicalImpression` with the Event resource pattern by adding
the two missing Event-pattern elements:

- `ClinicalImpression.basedOn` — `Reference(CarePlan | ServiceRequest |
  RequestOrchestration)` (intersection of the Event pattern's
  `basedOn` targets with what plausibly authorises a clinical
  assessment), `0..*`, summary, modifier=false, normal binding
  semantics. Description aligned with the Event pattern: "A plan,
  proposal or order that this ClinicalImpression fulfils in whole or
  in part."
- `ClinicalImpression.partOf` — `Reference(ClinicalImpression |
  Procedure)` (or at minimum `Reference(ClinicalImpression)` to allow
  decomposition of larger assessments such as a full triage
  assessment), `0..*`, summary, modifier=false. Description aligned
  with the Event pattern: "A larger event of which this particular
  ClinicalImpression is a component or step."

Both elements would be added to the resource definition (StructureDefinition
+ resource intro page) for ClinicalImpression in the next FHIR release,
with no change to existing element cardinalities or bindings, so the
change is additive and backward-compatible for current implementers.

#### Justification

The Event pattern is the documented baseline for event-type resources,
and consistent population of `basedOn` / `partOf` is what makes the
generic workflow chain (request → fulfilment → component events)
queryable across resources. `ClinicalImpression` already self-identifies
as an event-type resource, and the omission of these two elements is a
genuine inconsistency rather than a deliberate design choice. Both
elements are additive and optional, so the change carries no breaking
risk for existing implementers, and it directly supports the reporter's
real-world A&E triage use case where a full triage assessment composes
multiple sub-assessments. It also matches Patient Care's stated posture
of inviting implementer feedback at this FMM level.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than committing to the full Event-pattern alignment in one pass,
add only `ClinicalImpression.partOf` (`Reference(ClinicalImpression)`,
`0..*`, summary) in this cycle, and defer `basedOn` pending an explicit
use case showing what authorising request would drive a
`ClinicalImpression`. `partOf` directly serves the reporter's triage
composition case (a "full triage assessment" containing component
assessments) and is unambiguous; `basedOn` for `ClinicalImpression` is
less obviously motivated because clinical impressions are typically
formed by a clinician at the point of care rather than fulfilling a
discrete order, and committing to a `basedOn` target list now risks
needing to revise it later. Document the partial alignment explicitly
in the resource intro and open a follow-up tracker for `basedOn` once
implementer demand is shown.

#### Justification

This preserves the consistency benefit where it has a clear use case
(composition of assessments) without locking the spec into a `basedOn`
target list before the workflow semantics are well understood. It also
keeps the change scope small enough to not contend with other Patient
Care priorities at the current FMM, and respects the principle that
event-pattern adoption should be earned per element rather than
applied wholesale when only some elements are justified by real
implementations.

---

### Disposition C: Decline

#### Proposal

Decline the change as **not pursued** and close the ticket. Rationale:
the original reporter has explicitly stated in his 2023-07-28 comment
that he is no longer on the implementing project, that nothing is
blocking him, and that the workgroup is free to close the ticket;
no Zulip discussions, linked tickets, or GitHub items have surfaced in
the years since filing to indicate broader implementer demand for
Event-pattern alignment of `ClinicalImpression`. Implementers needing
composition or fulfilment links today can use the `extension`
mechanism, or model a containing assessment with a `Composition` /
`Encounter` / `Procedure`, until concrete demand justifies a core
change.

#### Justification

A request with no demonstrated second implementer in five years, an
inactive original reporter who has signalled it is OK to close, and no
community discussion is a defensible candidate to close from triage.
Adding optional elements is technically cheap, but each element carries
a long-tail cost (mapping tables, examples, search-parameter review,
implementer questions) that is not justified by a single historical
use case. Declining now does not preclude reopening if Patient Care
sees the request resurface from active implementers.

---

### Recommendation

**Recommended disposition:** A

Even though the reporter has invited the workgroup to close the ticket,
the underlying observation is correct and uncontroversial: `ClinicalImpression`
self-identifies as an event-type resource and the absence of `basedOn`
and `partOf` is an inconsistency with the Event pattern rather than a
deliberate design. Both additions are optional, additive, and
backward-compatible, so the implementer cost of adopting them is
essentially zero, while the consistency payoff — uniform workflow
linkage across event resources, a single mental model for
implementers, and direct support for the reporter's documented triage
composition use case — is real. Disposition B is a reasonable fallback
if Patient Care wants to defer `basedOn` until a concrete authorising
request use case is demonstrated, and should be the workgroup's
secondary choice. Disposition C is defensible on housekeeping grounds
but trades a small short-term saving for a known, persistent
inconsistency in the spec, which is the weaker outcome for
implementers in the long run.
