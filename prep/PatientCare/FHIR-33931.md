## Summary

The reporter observes that in the Ophthalmology Retinal FHIR Implementation
Guide one or more profiles expose a `value[x]` element — which by FHIR
convention is a polymorphic *choice* element — but the children rendered
underneath it are not the expected `valueQuantity`, `valueCodeableConcept`,
etc. choices. The ticket notes the issue is present "in other profiles" as
well, suggesting a recurring authoring pattern in the IG rather than a
one-off slip. No related Jira tickets, Zulip discussions, or GitHub items
were surfaced via cross-references or search, so the only available context
is the ticket itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the report as a defect in the Ophthalmology Retinal IG profiles. For
each profile in the IG that constrains an `Observation.value[x]` (or any
other `[x]` choice element), audit the differential and snapshot:

1. If the intent is to constrain the choice to a single type, replace the
   `value[x]` element in the differential with a typed slice / constraint
   that fixes `type` to the chosen datatype (e.g., `valueQuantity` only),
   and remove any child element definitions that were authored under
   `value[x]` as if it were a backbone element.
2. If the intent was to allow several types, declare the permitted types
   on `value[x].type` and define per-type slices using the standard
   choice-element slicing pattern (`discriminator.type = type`,
   `discriminator.path = "$this"`).
3. Re-run the IG publisher and confirm the rendered profile no longer
   shows non-choice children directly under `value[x]`.

#### Justification

This matches what the reporter literally asks for: the profiles do not
conform to the FHIR convention for `[x]` choice elements, and the IG
publisher's rendering reflects that. Fixing it in-place keeps the IG aligned
with the base FHIR rules for polymorphic elements (R4 §"Choice of
Datatypes") and with how every other published HL7 IG handles `value[x]`,
so implementers will see a predictable, tool-friendly profile.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than only fixing the affected profiles in isolation, address the
root cause: introduce an authoring convention / template for the
Ophthalmology Retinal IG that documents how to constrain `value[x]` (and
other choice elements) and add a CI check (e.g., a Sushi/FSH lint or a
small validator script invoked from the IG build) that fails the build
when a profile defines child elements directly under an unsliced `[x]`
element. Then apply the per-profile fixes from Disposition A under that
new guard rail.

#### Justification

The reporter explicitly notes the same mistake recurs "in other
profiles," which is a strong signal of a systemic authoring issue rather
than a single bad profile. Pairing the cleanup with a guard rail prevents
regression as the IG continues to evolve and is consistent with how other
HL7 IGs have hardened their build to catch choice-element misuse. The
trade-off vs. A is a larger up-front investment, but it materially
reduces the chance that the same defect re-appears in the next ballot
cycle.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Not Persuasive — already addressed by base FHIR
rules" if, on re-inspection, the profiles in question are actually
conformant and the reporter was misled by an IG-publisher rendering
quirk or a stale snapshot. In that case, document the finding in the
ticket and leave the profiles unchanged.

#### Justification

This is only defensible if a fresh build of the IG demonstrates that the
profiles do conform to choice-element rules and the apparent "non-choice
children" are an artifact of how the differential was being viewed. Given
the reporter's specificity ("This is in other profiles"), declining is
unlikely to be the right answer, but it is included for completeness so
the workgroup can rule it out explicitly.

---

### Recommendation

**Recommended disposition:** B

The reporter's observation that the pattern repeats across multiple
profiles points to an authoring habit in the Ophthalmology Retinal IG
rather than a one-line typo, and the priority is set to Highest with the
ticket sitting at Triaged since 2021. Disposition A would clean up the
visible symptom but leaves the door open for the same mistake to creep
back in as new profiles are added; Disposition C is hard to justify on
the face of the report. Disposition B does the work A would do *and*
adds a cheap guard rail (lint / build check plus a short authoring
note), which is the more durable fix and the better starting point for
the workgroup discussion.
