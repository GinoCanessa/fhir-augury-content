## Summary

The reporter (Cooper Thompson, on behalf of an implementer audience) is
asking Patient Care to define how a **post-operative diagnosis** should be
recorded for surgical procedures in FHIR. The reporter notes the working
assumption that `Procedure.reason` is the appropriate location for the
**pre-operative** diagnosis, but no clear, documented home has been
identified for the **post-operative** diagnosis — i.e., the diagnosis as
understood after the procedure is complete (which may differ from the
indication that prompted the procedure). The Patient Administration work
group is currently drafting cross-spec guidance on where various types of
diagnoses should be represented and is asking PC to settle this specific
case so PA can reference it. There are no Jira links, no related Zulip
threads, and no related GitHub items recorded for this ticket; the only
cross-reference on file is to the FHIR artifact `Procedure.reason`
itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add explicit guidance to the **Procedure** resource (in HL7/fhir, under
the FHIR Core authoring root for the Procedure resource — typically
`source/procedure/`) that:

1. Confirms `Procedure.reason` (R5+) is the location for the
   **pre-operative / indication** diagnosis — i.e., the condition that
   prompted the procedure, as understood at the time of the procedure.
2. Defines a recommended location for the **post-operative diagnosis**
   — the diagnosis as established by or refined during/after the
   procedure. Recommend recording it as a separate `Condition` resource
   (with `Condition.category = encounter-diagnosis` and
   `Condition.encounter` referencing the procedure's encounter) and
   linking it from `Procedure.report` (a DiagnosticReport) and/or via
   `Encounter.diagnosis.condition` with an appropriate `use` code (e.g.,
   `post-op` from a value set extended for that purpose).
3. Adds resource-level narrative (Notes / Boundaries section) and a
   short worked example illustrating both the pre-op and post-op
   diagnoses for the same procedure, so PA can cross-reference this
   guidance from its forthcoming cross-spec "where to put diagnoses"
   page.

#### Justification

This directly answers the question PA is waiting on and keeps the
diagnosis representation aligned with existing FHIR patterns: pre-op
intent stays on `Procedure.reason` (which the reporter already endorses),
while the post-op diagnosis lives where other clinically-established
diagnoses live — as a `Condition` tied to the encounter, optionally
referenced from the procedure's `DiagnosticReport`. This avoids
overloading `Procedure.reason` with two semantically distinct kinds of
diagnosis and reuses mechanisms (`Encounter.diagnosis.use`,
`Condition.category = encounter-diagnosis`) that implementers already
understand.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than relying solely on `Encounter.diagnosis` and a separate
`Condition`, introduce a structured slot on `Procedure` itself for the
post-operative diagnosis. Two concrete shapes are possible:

- **B1 (extension):** Define a standard extension
  `procedure-postOpDiagnosis` (`Reference(Condition | CodeableReference)`),
  registered against `Procedure`, with documented expectation that the
  referenced `Condition` carries `category = encounter-diagnosis` and is
  scoped to the procedure's encounter. Ship it in the FHIR Core
  extensions pack so PA / specialty IGs can adopt it uniformly.
- **B2 (core element, R6 candidate):** Add a new element
  `Procedure.diagnosis` of type `BackboneElement` with `use`
  (CodeableConcept; binding to a value set distinguishing pre-op,
  post-op, principal, etc.) and `condition` (CodeableReference(Condition)),
  mirroring the `Encounter.diagnosis` pattern. Mark it Trial Use and
  request Patient Care + Patient Administration joint stewardship.

In either variant, `Procedure.reason` continues to carry the pre-op /
indication diagnosis as today.

#### Justification

The "Condition + Encounter.diagnosis" path in Disposition A is correct
but indirect: implementers who start at `Procedure` have to traverse to
`Encounter` to find the post-op diagnosis, which is awkward for surgical
workflows where the post-op diagnosis is conceptually an attribute *of*
the procedure. A standard extension (B1) is low-risk and can be done
without a breaking change, and provides the "obvious place to look" that
the ticket is really asking for. A core element (B2) is more invasive
and would need wider review, but parallels `Encounter.diagnosis` and
would future-proof the answer if multiple diagnosis kinds (principal,
admitting, post-op, complication) recur on `Procedure`.

---

### Disposition C: Decline

#### Proposal

Decline to add new structure or new normative guidance on `Procedure`.
Respond to PA that the existing FHIR machinery is already sufficient and
that the answer is documentation-only on PA's side: post-op diagnosis is
recorded as a `Condition` with `category = encounter-diagnosis` linked
to the same `Encounter` as the `Procedure`, optionally referenced from
the procedure's `DiagnosticReport` (`Procedure.report`) or from
`Procedure.complication` / `Procedure.complicationDetail` when the
post-op finding is a complication. No change to the FHIR Core spec is
required.

#### Justification

`Procedure` already exposes several anchors for post-procedure
clinical findings (`Procedure.outcome`, `Procedure.complication`,
`Procedure.complicationDetail`, `Procedure.report`), and `Encounter`
already has a typed `diagnosis` backbone with a `use` element for
distinguishing kinds of diagnoses. Adding another spec-level mechanism
risks fragmenting how implementers represent diagnoses and creates
redundant places to look. Declining keeps the surface area small and
pushes the work to where it actually belongs — PA's cross-spec
documentation page — without requiring any FHIR Core change.

---

### Recommendation

**Recommended disposition:** A

Disposition A gives PA the concrete answer they explicitly asked for
("once you figure this out please let PA know") without inventing new
structure: pre-op diagnosis stays on `Procedure.reason`, post-op
diagnosis is a `Condition` with `category = encounter-diagnosis`
attached to the procedure's `Encounter` (and optionally surfaced via
`Procedure.report`). It reuses well-established patterns
(`Encounter.diagnosis`, `Condition.category`) and is achievable as a
narrative / examples change in the Procedure resource page, which is
low-risk and ballotable in the current cycle. Disposition B is worth
revisiting as a follow-up if implementer feedback shows that pointing
from `Procedure` into `Encounter.diagnosis` is consistently missed in
the wild — at which point a standard extension (B1) is the lighter next
step. Disposition C is technically defensible but ignores the
documentation gap that motivated the ticket: the whole reason PA is
asking is that today implementers cannot tell from the spec where
post-op diagnoses go, so "no change" leaves the original problem
unsolved.
