## Summary

The ticket reports that the `CareTeam` resource's `date` SearchParameter
is published with an empty FHIRPath `Expression` cell in the resource's
search-parameter table, even though the parameter has a name (`date`),
type (`date`), and description ("A date within the coverage time period.").
Without an expression, implementers cannot determine which CareTeam
element the `date` parameter resolves to, which makes the parameter
effectively unimplementable and breaks server conformance tooling that
generates code from `SearchParameter.expression`. The reporter notes
that the description still references the older "coverage time period"
wording — a leftover from when CareTeam carried a `coverage[x]` element
that was later renamed to `effective[x]` on `CareTeam.participant` —
and proposes that the expression should resolve to either
`CareTeam.period` or `CareTeam.participant.effective` (or a union of
both). The fix is to populate `SearchParameter.expression` for the
CareTeam `date` parameter so the published table no longer has a blank
expression cell.

There are no other Jira tickets cross-referenced from this issue. The
FHIR-side cross-references surfaced are the two candidate target
elements named in the description (`CareTeam.period` and
`CareTeam.participant.effective`); both are mentions inside the
ticket body rather than independent issues.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Populate `SearchParameter.expression` for the CareTeam `date`
search parameter so that the published expression cell is no longer
blank. Concretely, set the expression to the union of the two date-
bearing elements that the parameter is intended to cover:

```
CareTeam.period | CareTeam.participant.period
```

(or `CareTeam.period | CareTeam.participant.effective` if the
post-rename element name on `CareTeam.participant` is `effective[x]`
rather than `period`). Update the description from "A date within the
coverage time period." to "A date within the time period of the team
or any of its participants." so the wording matches the resolved
expression and no longer references the obsolete `coverage[x]` element.
The change is to the CareTeam `date` SearchParameter definition in
the FHIR Core spec source (the per-resource search-parameter
definition file under the CareTeam resource source folder); the
published HTML table is regenerated from that definition and will
fill in automatically.

#### Justification

The empty expression is a concrete defect in the published
specification: every other search parameter of type `date` in FHIR
Core has a non-empty expression, and tooling that consumes the search-
parameter registry (server scaffolders, capability-statement
validators, IG QA) treats a missing expression as an error. The ticket
correctly identifies that CareTeam carries date-typed elements
(`CareTeam.period` on the team itself and a date-typed element on
`CareTeam.participant`) that the `date` parameter is naturally meant
to cover; combining them in a union expression matches the established
FHIR pattern for "date" search parameters that span both a top-level
period and per-participant timing (e.g., `Encounter.period |
Encounter.participant.period`-style unions used elsewhere). Because
this is filling in a missing definition rather than changing existing
behavior, it is non-breaking and improves conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than picking a union, redefine the parameter to target only
`CareTeam.period` and add a separate, narrower `participant-date`
search parameter for `CareTeam.participant.period` /
`CareTeam.participant.effective`. The CareTeam `date` parameter would
then be:

```
expression: CareTeam.period
description: The time period during which the care team is or was active.
```

and a new `participant-date` parameter (type `date`) would target the
participant-level element. Also clean up the description on `date` to
remove the "coverage time period" wording.

#### Justification

Treating `date` as a single union across the team and its participants
conflates two distinct query intents: "which teams were active in
range X" versus "which teams had a participant active in range X."
Implementers and writers of CapabilityStatements often want to expose
the team-level filter without taking on the cost of indexing a
nested participant element. Splitting the parameter follows the same
modeling choice already used on other resources where a top-level
date and a nested-participant date are distinguished (e.g., the
`date` vs. participant-scoped date parameters on Appointment). The
trade-off versus Disposition A is that this introduces a brand-new
search parameter, which is a larger surface change and may require
ballot, whereas Disposition A is a pure defect fix.

---

### Disposition C: Decline

#### Proposal

Decline on the basis that the search parameter can be removed
entirely rather than fixed. If the parameter has shipped without an
expression for one or more releases without implementer demand
sufficient to specify its semantics, retire `CareTeam.date` from the
published list in the next release and direct implementers who need
date-based CareTeam queries to use `_lastUpdated` or to define their
own search parameter at the IG / server level.

#### Justification

This is defensible only if there is no demonstrated implementer
demand and if removing the parameter is judged less harmful than
guessing at a definition that may not match what existing servers
have already implemented locally. Removing rather than fixing avoids
the risk that any newly-chosen expression silently changes results
returned by servers that had inferred a different binding. The reason
this is the weakest disposition is that the ticket itself shows there
*is* implementer interest (the reporter is a longtime IG author
flagging this exact problem), and FHIR Core has consistently treated
empty `SearchParameter.expression` as a defect to repair, not as
grounds to retire the parameter.

---

### Recommendation

**Recommended disposition:** A

Accept the request as filed and populate the expression with the
union `CareTeam.period | CareTeam.participant.period` (adjusting the
participant-side element name to whatever the current CareTeam
structure uses post-`coverage[x]`/`effective[x]` rename). This is a
defect fix: the published expression cell is blank today, the ticket
correctly identifies the candidate target elements, and the union form
matches the established pattern for date parameters that span both a
top-level period and per-participant timing. Disposition B is
defensible but introduces a new parameter and so carries scope and
ballot cost that is hard to justify when the immediate problem is a
missing expression on an already-published parameter; it can be
revisited as a follow-up enhancement if implementers ask for the
finer-grained query. Disposition C should not be the workgroup's
starting point because the parameter's name, type, and intent are
clear from the description, and silently retiring it would be
disruptive to anyone who has already implemented it. The description
text should also be cleaned up in the same change to drop the
obsolete "coverage time period" phrasing.
