## Summary

The submitter requests that the `bodyStructure` element on `Procedure` be
harmonized with the equivalent element on `Observation`. In `Observation`,
`bodySite` has been deprecated and replaced by `bodyStructure`, typed as
`CodeableReference(BodyStructure)`. In `Procedure`, a `bodyStructure`
element was added, but it is typed as a plain `Reference(BodyStructure)`
and `Procedure.bodySite` was **not** deprecated, leaving the two resources
inconsistent.

The reporter asks Patient Care (and originally FHIR-I) to consider:
(a) whether `Procedure.bodySite` should be deprecated like
`Observation.bodySite`; (b) clarification of the boundary between
`bodySite` and `bodyStructure` (when each should be used); (c) cardinality,
which is being addressed separately in
[FHIR-54621](https://jira.hl7.org/browse/FHIR-54621); and (d) changing
`Procedure.bodyStructure` to `CodeableReference(BodyStructure)` with the
same value-set binding that `Observation.bodyStructure` carries.

`FHIR-54621` (O&O, Eric Haas) is closely related: it argues that the
current `Procedure.bodyStructure` 0..1 cardinality is a regression from
R4 `bodySite` 0..* and a capability loss for bilateral procedures,
multi-site biopsies, and procedures spanning multiple anatomical regions.
That ticket recommends restoring 0..* and removing the now-redundant
`procedure-targetBodyStructure` extension. Any harmonization decision
reached here should be coordinated with that cardinality decision and with
O&O, since `Observation` is normative and the change burden therefore
falls on `Procedure`.

## Details

**Description:**

In the Observation resource Observation.bodySite was deprecated and
Observation.bodyStructure which is of type `CodeableReference(BodyStructure)`
was introduced.

In the Procedure resource the Procedure.bodyStructure was also introduced,
however, it is of type `Reference(BodyStructure)` and Procedure.bodySite is
not deprecated.

These two resources should be harmonized and follow the same approach.
Aspects that should be considered:

- Decision whether `bodySite` should be deprecated
- Clarification of boundaries (when to use which)
- Cardinality (addressed in
  [FHIR-54621](https://jira.hl7.org/browse/FHIR-54621))
- Change `Procedure.bodyStructure` to a `CodeableReference` and add the
  same binding as in `Observation.bodyStructure`

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Harmonize `Procedure` with `Observation` on the Procedure side:

1. Change the type of `Procedure.bodyStructure` from
   `Reference(BodyStructure)` to `CodeableReference(BodyStructure)`.
2. Apply the same value-set binding (and binding strength) used on
   `Observation.bodyStructure` (the SNOMED CT body-site / body-structure
   value set) to `Procedure.bodyStructure`.
3. Deprecate `Procedure.bodySite` with a forward-pointer note directing
   implementers to `Procedure.bodyStructure`, mirroring the
   `Observation.bodySite` → `Observation.bodyStructure` deprecation
   pattern.
4. Update the Procedure resource notes / boundaries-and-relationships
   text to describe when each element should be used and to explain the
   coded-or-referenced semantics introduced by `CodeableReference`.
5. Coordinate with FHIR-54621: lock in the cardinality on
   `Procedure.bodyStructure` (presumably 0..*) before publishing the
   retyped element so the change ships once.

#### Justification

This is exactly what the submitter requests and what FHIR-I has already
endorsed. `Observation` is normative, so the burden of alignment must
fall on `Procedure`. `CodeableReference(BodyStructure)` is the modern
pattern for "code or reference" body-site data and is already the
agreed Observation pattern; using it on Procedure removes a per-resource
inconsistency that would otherwise force implementers and profile
authors to handle the same concept two different ways. Deprecating
`Procedure.bodySite` (rather than removing it) preserves backward
compatibility for R4/R5 implementers while signalling the migration
direction. Settling cardinality first via FHIR-54621 avoids a
double-breaking change on the same element.

---

### Disposition B: Alternative Approach

#### Proposal

Retype `Procedure.bodyStructure` to `CodeableReference(BodyStructure)`
with the same binding as on `Observation.bodyStructure`, **but do not
deprecate `Procedure.bodySite` in this cycle**. Instead:

- Keep `Procedure.bodySite` as-is and add explicit boundaries-and-
  relationships text stating that `bodyStructure` is preferred, with
  `bodySite` retained for legacy/simple coded-only usage.
- Tighten the coded binding on `bodySite` and `bodyStructure.concept` so
  they share the same value set, removing the practical reason to use
  `bodySite` going forward.
- Defer the deprecation of `Procedure.bodySite` to the next cycle, after
  the harmonized `bodyStructure` element has been exercised by trial-use
  implementers.

#### Justification

`Procedure` is not normative, but it is widely implemented. Deprecating
`bodySite` in the same cycle that retypes `bodyStructure` and (per
FHIR-54621) widens its cardinality stacks three coupled changes on one
element family, which raises the conversion cost for every existing
Procedure consumer (USCDI, IPS, IPA, regional IGs, etc.). A staged
approach — retype + bind now, deprecate later — gives implementers a
working harmonized target while still resolving the inconsistency
called out in this ticket. It also leaves a clean audit trail: this
ticket lands the harmonization, and the deprecation can be its own
ticket with its own impact analysis.

---

### Disposition C: Decline

#### Proposal

Decline to retype `Procedure.bodyStructure`. Keep
`Procedure.bodyStructure` as `Reference(BodyStructure)`, keep
`Procedure.bodySite` as the coded entry point, and document the
intentional asymmetry with `Observation` in the
boundaries-and-relationships section of `Procedure`. Address
cardinality only via FHIR-54621.

#### Justification

`Observation` and `Procedure` model different things: an `Observation`
is a finding whose subject is often itself an anatomical site (the
"what was observed"), so collapsing code+reference into
`CodeableReference` directly on the observed site is natural. A
`Procedure` performed *at* a site is a different relationship; keeping
`bodySite` (coded) and `bodyStructure` (reference) as distinct elements
arguably preserves a clearer authoring model and avoids the
known-confusing "is the code or the reference authoritative?" question
that `CodeableReference` introduces. Declining preserves backward
compatibility and avoids breaking changes for a non-normative resource
that is already widely deployed.

---

### Recommendation

**Recommended disposition:** A (with explicit coordination with FHIR-54621)

Disposition A is the only outcome that actually delivers the
harmonization the ticket asks for and that FHIR-I has already endorsed.
The status quo forces every cross-resource consumer (Patient Summary,
US Core, IPS, terminology services, SDC question logic, FHIRPath
queries that traverse "what part of the body") to special-case
`Procedure` versus `Observation` for the same clinical concept, which
is a long-term tax on implementers far larger than the one-time R6
breakage. Disposition B is a reasonable fallback if Patient Care wants
to soften the implementer impact, but it leaves `Procedure.bodySite`
formally first-class for at least one more cycle, which propagates the
inconsistency into downstream IGs that are being authored right now.
Disposition C should be declined: the asymmetry it tries to defend is
not a deliberate design — it is an artifact of `Observation` going
normative before `Procedure` adopted `CodeableReference`. The work
should be sequenced behind FHIR-54621's cardinality decision so the
retype, the binding, and the cardinality change ship as a single
coherent update to `Procedure.bodyStructure`.
