## Summary

The ticket asks Patient Care to relax the cardinality of
`ClinicalImpression.encounter` from `0..1` to `0..*`. The reporter's
rationale is that a single clinical assessment is not always confined to
one encounter — it may evolve over time and span multiple encounters
while still representing the same logical assessment event (e.g., an
admission assessment that is refined across rounds, a multi-visit
diagnostic work-up, or a longitudinal psychiatric evaluation). Changing
the cardinality would let a single `ClinicalImpression` instance link
all the encounters that contributed to it without forcing implementers
to either pick one "primary" encounter or fragment the assessment into
multiple resource instances.

There are no Jira-side links, no Zulip cross-references, and no GitHub
cross-references on this ticket. Cross-reference discovery surfaced only
the FHIR artifact `ClinicalImpression.encounter` itself. Sibling Patient
Care tickets on the same resource provide background:
[FHIR-8201](https://jira.hl7.org/browse/FHIR-8201) is the original
(persuasive) request that introduced the encounter linkage to
`ClinicalImpression`; [FHIR-46597](https://jira.hl7.org/browse/FHIR-46597)
(also from Russell Leftwich, Highest priority, Triaged) requests adding
a `category` element to better classify assessment types; and
[FHIR-37754](https://jira.hl7.org/browse/FHIR-37754) (Michelle Miller,
Triaged) questions whether `ClinicalImpression.effective` is sufficient
to express assessment timing across change patterns. Together these
suggest an ongoing Patient Care effort to modernise `ClinicalImpression`
for longitudinal use, into which the cardinality request fits.

Reporter: Michelle Miller.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In FHIR Core (HL7/fhir), change the cardinality of
`ClinicalImpression.encounter` from `0..1` to `0..*`. Update the element
definition and short to reflect that a single clinical impression may
span multiple encounters (e.g., "The Encounter(s) during which this
ClinicalImpression was created or to which the creation of this record
is tightly associated."). Update any element comment that implies a
single encounter, refresh the resource narrative, and revise examples
to include at least one multi-encounter case. Note the breaking change
in the version-history / change-notes for `ClinicalImpression`.

#### Justification

The current `0..1` cardinality forces implementers to either pick a
single "primary" encounter or fragment a logically single assessment
into multiple `ClinicalImpression` resources, neither of which is
faithful to longitudinal clinical workflows. The pattern of allowing
`0..*` references to `Encounter` is already used elsewhere in the spec
(e.g., `Goal.addresses` / care-coordination resources frequently
reference multiple encounters via supporting structures), so the change
is consistent with existing modelling. This is a relaxation of
cardinality, which is largely additive for producers (existing
single-encounter instances remain valid against the wire form change),
though it is a breaking change for consumers that have hard-coded a
scalar reference.

---

### Disposition B: Alternative Approach

#### Proposal

Keep `ClinicalImpression.encounter` at `0..1` (the "primary" or
"originating" encounter) and instead model additional encounters that
contributed to the assessment via a new repeating element such as
`ClinicalImpression.contributingEncounter : Reference(Encounter) 0..*`,
or via a standard extension
(`http://hl7.org/fhir/StructureDefinition/clinicalimpression-contributingEncounter`)
in the same release. Document the relationship between
`encounter` and `contributingEncounter` (the former is the encounter of
record; the latter enumerates additional encounters whose data fed into
the assessment). Coordinate with FHIR-37754 so that
`effective[x]`/last-assessed semantics line up with the multi-encounter
model.

#### Justification

This avoids a breaking change for the (likely large) population of
implementers and downstream tooling that treat
`ClinicalImpression.encounter` as scalar — including profiles, search
parameter consumers, and code generators that emit a non-collection
field. It also preserves a clear "encounter of record" semantic, which
is useful for billing, attribution and provenance, while still letting
producers express the longitudinal reality the reporter describes.
Using an extension (rather than a new core element) is even safer if
the use case is still being validated, which aligns with Michelle
Miller's WGM comment asking for a concrete use case before committing
to a model change.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** (or **Not Persuasive with Modifier** —
see below) and leave `ClinicalImpression.encounter` at `0..1`.
Rationale: the longitudinal-assessment use case can already be modelled
today by (a) leaving `encounter` unset and using `effectivePeriod` to
express the time span, (b) recording successive assessments as separate
`ClinicalImpression` instances linked via
`ClinicalImpression.previous`, or (c) using the existing
`Encounter.partOf` chain to group encounters that share an assessment.
Encourage the reporter to bring back a concrete use case (per Michelle
Miller's Sept 2025 WGM comment) showing why those mechanisms are
insufficient.

#### Justification

The reporter's description states the use case in the abstract
("assessments can take place over time") but does not show a workflow
that the existing model cannot express. The Sept 2025 WGM comment
explicitly asks for that use case. Changing core cardinality is a
breaking change for consumers and triggers ripple effects in profiles,
search parameters, and downstream code, so the bar for accepting it
should be higher than "this could happen". Declining now does not
foreclose the change — if a stronger use case arrives, the ticket can
be reopened with the evidence in hand.

---

### Recommendation

**Recommended disposition:** B

The reporter's underlying concern — that one clinical assessment can
legitimately span multiple encounters — is real and recurs in
longitudinal-care settings, so a flat decline (C) is hard to defend
once the use case is articulated, and the parallel work on
`ClinicalImpression.category` (FHIR-46597) and `effective`/last-assessed
semantics (FHIR-37754) suggests Patient Care is already modernising
this resource for longitudinal use. However, accepting the cardinality
change as requested (A) is a breaking change for every consumer that
treats `ClinicalImpression.encounter` as scalar, and Michelle Miller's
WGM comment shows the work group is not yet convinced the use case
warrants that cost. Disposition B preserves the "encounter of record"
semantic, avoids breaking existing consumers, and still lets producers
faithfully record multi-encounter assessments — either via a new
`contributingEncounter` element or a standard extension while the use
case is firmed up. It also dovetails with FHIR-37754 and FHIR-46597 so
the longitudinal-assessment story can be told coherently in one
release. The work group should ask Russell Leftwich for the concrete
use case requested at the Sept 2025 WGM and use it to choose between
the new-element and extension variants of B.
