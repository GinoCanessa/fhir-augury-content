## Summary

Craig Newman (Patient Care) is asking, against the Eye Care / Ophthalmology
Retinal IG, whether the **Observation Base** profile's reference to a
BodyStructure (via the `.bodySite` extension's value reference) was
*intentionally* permissive — allowing **either** the unprofiled
`BodyStructure` resource **or** the IG's **Body Structure Eye** profile —
or whether all conformant Observations are expected to use the Body
Structure Eye profile only. If both are intentional, he is asking the WG
to provide guidance on when an implementer should use each option.

This is a clarification/conformance question rather than a structural
change request: the underlying mechanism (Observation referencing a
BodyStructure-typed resource through the standard extension on
`Observation.bodySite`) is not in dispute. What is missing is IG-level
narrative explaining the design intent and selection criteria. The same
ambiguity is raised from an adjacent angle in the related ticket
**[FHIR-36110](https://jira.hl7.org/browse/FHIR-36110)** ("VisualAcuity
examples should use CodeableConcept for body sites"), which argues that
the IG's examples push implementers toward `BodyStructure`/Body Structure
Eye even where a simple CodeableConcept would suffice — i.e., the IG's
typing is more elaborate than the use cases require, and there is no
guidance distinguishing when to use the rich form versus the plain form.

The 2024 Zulip thread "[Eye Care] The Two-Eyes Problem" (Lars Fuhrmann)
re-opens essentially the same modeling question for Eye Care 2.0,
proposing a layered approach (coarse "right/left/both eye" via
`Observation.bodySite` CodeableConcept; richer detail via a referenced
BodyStructure when needed). That suggests the ambiguity flagged in this
ticket is still live, and that the IG would benefit from explicit
selection guidance regardless of which way the WG resolves it.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the current "either base BodyStructure **or** Body Structure Eye"
typing as **intentional**, and add explicit usage guidance to the Eye
Care IG's `Observation Base` profile narrative (and to a new IG-level
"Body site and laterality" page) that says:

- The `bodySite` extension's `value` reference SHOULD point to an
  instance conforming to **Body Structure Eye** when the BodyStructure
  describes ophthalmic anatomy (e.g., specific quadrants of the retina,
  clock-hour positions, anterior/posterior segment landmarks).
- It MAY point to an unprofiled `BodyStructure` resource when the body
  site is **not ophthalmic** (e.g., eyelid, brow, peri-orbital skin) or
  when the producing system is a non-ophthalmology actor that does not
  carry the Body Structure Eye profile.
- Implementers SHOULD prefer a coded `Observation.bodySite`
  CodeableConcept (e.g., SNOMED *Entire right/left/both eye*) for the
  coarse "which eye" case, and reserve the BodyStructure reference for
  finer-grained anatomy — picking up the layered model proposed in the
  "Two-Eyes Problem" Zulip thread.

Mark the Body Structure Eye profile as Must-Support **conditional on
ophthalmic content**, not unconditionally.

#### Justification

This matches the literal ask in the ticket (clarify intent + provide
guidance) and is consistent with the Zulip community thread, where the
participants explicitly prefer a layered approach and warn against
forcing a heavyweight BodyStructure reference for cases a simple
CodeableConcept would cover. It is also consistent with the broader
WG-wide direction visible in the related tickets (FHIR-50355, FHIR-54614,
FHIR-54843), all of which push toward documenting *when* to use
BodyStructure vs `bodySite` rather than restricting one of them.
Crucially, this option is **non-breaking** — every existing instance
remains valid — which is appropriate given the IG's STU status.

---

### Disposition B: Alternative Approach — Tighten by Slicing the Reference Type

#### Proposal

Keep both target types allowable, but **slice the BodyStructure reference
on the `bodySite` extension** in the `Observation Base` profile so the
profile makes the choice explicit at conformance level rather than only
in narrative:

1. Slice the extension's `value[x]` (or the underlying
   `Reference.profile`) into two named slices:
   - `eyeStructure` — `Reference(BodyStructure | BodyStructureEye)` with
     `aggregation = referenced` and a discriminator on
     `target.profile`, **bound by Must-Support** to the Body Structure
     Eye profile when the BodyStructure represents ocular anatomy.
   - `nonEyeStructure` — `Reference(BodyStructure)` for adjacent / non-
     ocular sites used in eye-care workflows (lids, brows, orbit).
2. Add an invariant on the Observation profile that fires when
   `BodyStructure.location` is in the SNOMED *eye* hierarchy and the
   referenced resource does not conform to Body Structure Eye, raising
   a `warning`-severity issue rather than `error` (so existing
   instances are not invalidated).
3. Pre-adopt R5 `Observation.bodyStructure` via the documented R4
   forward-compatibility extension (per Richard Townley-O'Neill's Zulip
   suggestion) so future migration to R5 is mechanical.

#### Justification

This addresses the same need as Disposition A but moves the "when to
use which" decision from narrative into machine-checkable conformance,
which is more useful to implementers and validators. The slicing
pattern is well-established in HL7 IGs and avoids the brittleness of
either (a) requiring Body Structure Eye unconditionally — which would
break legitimate non-ocular references — or (c) dropping the Eye
profile entirely. The trade-off versus A is implementation cost: the
profile becomes more complex and the IG must publish slice-specific
examples; existing tooling that does not honor `Reference.profile`
discriminators may emit false-positive warnings. The warning-severity
invariant keeps backward compatibility while still nudging publishers
toward the richer profile.

---

### Disposition C: Decline / Defer to Eye Care 2.0

#### Proposal

Decline to make a normative change against the current Ophthalmology
Retinal (FHIR) IG release. Rationale: (i) the ticket has sat in
**Triaged** status since 2021 with no comments and no Assignee, and
(ii) the same modeling question has been re-opened from scratch in the
Eye Care 2.0 Zulip discussion ("Two-Eyes Problem"), which is
considering a more thorough redesign of how laterality and body site
are represented (potentially shifting the coarse case to
`Observation.bodySite` as a CodeableConcept and the rich case to R5's
`Observation.bodyStructure`). Roll FHIR-33496 — together with the
closely-related FHIR-36110 — into the Eye Care 2.0 work item set and
close this ticket as *Persuasive with Modification — addressed in next
version*, with a pointer to the consolidated Eye Care 2.0 ticket.

#### Justification

A tactical guidance patch to a profile that the WG is already planning
to redesign creates churn for implementers (who would have to learn one
set of guidance now and a different set in 2.0) and risks anchoring the
2.0 redesign to a model the WG is no longer comfortable with. Closing
this ticket cleanly with a forward pointer is defensible because (a)
the literal request is for *guidance*, not a normative constraint, and
the Zulip thread already documents the relevant trade-offs publicly,
and (b) HL7 disposition practice supports rolling a clarification
ticket forward when the surrounding artifact is being substantively
re-worked.

---

### Recommendation

**Recommended disposition:** A.

The ticket is, on its face, a request for clarification — the reporter
is not asking for a new constraint, only for confirmation of intent and
selection guidance. Disposition A delivers exactly that, in a way that
is non-breaking, minimally disruptive to existing implementers, and
consistent with the position the Eye Care community arrived at on
Zulip. It also disposes of the closely-related FHIR-36110 in the same
narrative pass (by acknowledging that a coded CodeableConcept on
`Observation.bodySite` is the right answer for the coarse
right/left/both case).

Disposition B is the more rigorous answer and is worth holding in
reserve if the WG decides the guidance must be machine-checkable, but
the slicing + invariant work is non-trivial for an STU-stage IG and
should not gate this clarification. Disposition C is reasonable if the
WG has already committed to an Eye Care 2.0 ballot timeline that will
land before any guidance from this ticket would reach implementers; if
not, deferring leaves the ambiguity in the field for another release
cycle, which is the situation the reporter raised in the first place.
