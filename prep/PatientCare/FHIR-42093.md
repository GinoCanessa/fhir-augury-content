## Summary

The reporter (Gay Dolin) flags that several `Observation` example
instances published with the International Patient Access / International
Birth And Child Model materials are missing the `Observation.value[x]`
element. Two examples are called out by name — **Fundal Height** and
**Reducing Substances** — and the reporter asks whether other examples
are similarly affected ("Others?"). Since these examples are intended to
illustrate how to author birth- and child-related observations, missing
result values undermine their usefulness for implementers and may even
trigger validation warnings against profiles that expect a value (or one
of `dataAbsentReason`, a component, or `hasMember`). No cross-referenced
Jira tickets, Zulip discussions, or GitHub items were located for this
issue.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Audit every `Observation` example shipped with the International Birth
And Child Model and ensure each instance carries a valid
`Observation.value[x]` (or, where genuinely absent, an explicit
`dataAbsentReason`, `component`, or `hasMember` per the
`obs-7`/`obs-6` invariant family). Concretely:

- For the **Fundal Height** example, add `valueQuantity` with an
  appropriate UCUM unit (`cm`) and a clinically plausible measurement.
- For the **Reducing Substances** example, add either a
  `valueCodeableConcept` (e.g., a SNOMED CT/LOINC-coded result such as
  "Negative") or a `valueQuantity` with the relevant unit, depending on
  the LOINC code's expected answer type.
- Sweep the remaining Observation examples in the IG and add values (or
  a justified `dataAbsentReason`) wherever the current instance has
  neither.
- Re-run the IG publisher to confirm the examples validate against the
  IG's Observation profile(s) without missing-value warnings.

#### Justification

Examples in a published IG are normative-by-practice for implementers —
they get copied, adapted, and used as fixtures. Shipping examples that
violate the base `Observation` invariants (which require one of
`value[x]`, `dataAbsentReason`, `component`, or `hasMember`) teaches
the wrong pattern and can fail downstream validators. The fix is
mechanical, scoped to example artifacts only, and does not alter any
profile, terminology, or narrative — making it a low-risk technical
correction wholly consistent with the ticket's "Technical Correction"
type and "Highest" priority.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (only) patching the named examples, treat this as a
systemic example-quality problem in the IG:

1. Add an automated check to the IG build (e.g., a FHIR Validator
   profile run plus a small FSH/QA script) that fails the build when an
   `Observation` example lacks `value[x]`, `dataAbsentReason`,
   `component`, and `hasMember`.
2. Fix Fundal Height, Reducing Substances, and any other examples the
   check surfaces.
3. Document the rule in the IG's "Authoring examples" guidance so future
   contributions can't regress.

#### Justification

The reporter's "Others?" explicitly invites a broader sweep, and a
manual one-off fix (Disposition A) won't prevent the same defect from
re-appearing the next time examples are added or regenerated. Wiring
the check into CI converts a recurring editorial burden into a build-
time guarantee, at the cost of a small one-time tooling investment.
Trade-off vs. A: more effort up front, but durable; vs. C: actually
addresses the implementer-facing harm.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive / Already Addressed** on the basis that
`Observation` examples are illustrative only, the base resource's
invariants already document the requirement, and implementers are
expected to validate their own instances rather than copy examples
verbatim.

#### Justification

This is the weakest path and is included only for completeness. It
would be defensible *only* if the affected IG had been retired or if
the examples in question had already been corrected in a later
publication — neither of which is evidenced by the available data. In
the absence of such evidence, declining a "Highest" priority Technical
Correction from a known Patient Care contributor is hard to justify.

---

### Recommendation

**Recommended disposition:** B

Disposition A fixes the two named examples but leaves the reporter's
"Others?" question unanswered and lets the same defect recur. Disposition
C is not defensible for a "Highest" priority Technical Correction with
a concrete, reproducible defect. Disposition B does everything A does
**and** answers "Others?" definitively by letting the IG build itself
enumerate the offending examples, then prevents regression with a
minimal CI check. Given that Patient Care ships several Observation-
heavy IGs, the small tooling investment will pay back beyond this single
ticket. If the workgroup judges the CI work out of scope for this
ticket, fall back to A and open a follow-up ticket for the automated
check.
