## Summary

Stephen Chu requests that the AllergyIntolerance resource gain a first-class
`abatement[x]` element analogous to the existing `onset[x]`. The reporter
points out an internal inconsistency in the resource: `clinicalStatus`
already includes a `resolved` concept (and the value set is `active |
inactive | resolved`), yet there is no native way to record *when* a
documented allergy resolved or abated. The reporter cites clinical
literature that many allergies — most notably penicillin and cow's milk —
genuinely resolve over time (50% of penicillin allergies within 5 years,
80% within 10 years; ~80% of cow's milk allergies resolve by age 3–5),
arguing the resource should be able to capture that abatement timing
directly rather than requiring an extension.

There are no other Jira tickets cross-referenced to this request. The
cross-reference index links it only to the FHIR artifacts
`AllergyIntolerance.abatement[x]` and `AllergyIntolerance.clinicalStatus`,
confirming that the ticket is scoped to AllergyIntolerance in FHIR Core
rather than to any IG. Community discussion on chat.fhir.org (see
*Related Zulip Discussions*) has, since at least 2016, repeatedly pointed
implementers at the standard `allergyintolerance-abatement` extension as
the work-around, with one of those threads ending with an explicit
recommendation that anyone who would prefer a native element should
"submit a Jira ticket" — which is effectively what FHIR-54183 now does.

## Related Zulip Discussions

### [implementers > AllergyIntolerance end date](https://chat.fhir.org/#narrow/stream/implementers/topic/AllergyIntolerance%20end%20date)

Fieran Mason-Blakley asked in July 2024 whether AllergyIntolerance is
intended to ever "end" in FHIR R5, since they were migrating from a system
with an explicit allergy end-date. Peter Jordan answered that the standard
work-around is `clinicalStatus = resolved` plus the
`allergyintolerance-abatement` extension, and explicitly suggested filing
a Jira ticket if a native element is preferred. Daniel Venton questioned
whether allergies ever truly resolve, prompting a long, authoritative
clinical reply from Russell Leftwich (board-certified allergist /
immunologist) explaining that allergies *do* genuinely resolve at the IgE
level — particularly childhood allergies such as cow's milk — and citing
the AAAAI's penicillin-allergy delabeling initiative (>90% of patients
labelled as penicillin-allergic show no reaction on challenge). Grahame
Grieve agreed that genuine mechanistic resolution is real but "pretty
rare." Russell also cautioned that *identifying the time of abatement* is
clinically hard: typically the only datable events are reactions and
non-reactions on re-exposure, and patients rarely return to declare an
allergy gone. The thread reached rough consensus that the extension is
the right tool today, while acknowledging the use case for a native
element is legitimate.

Participants: Fieran Mason-Blakley, Peter Jordan, Daniel Venton, Grahame
Grieve, Russell Leftwich, John Silva.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new element `AllergyIntolerance.abatement[x]` to the
AllergyIntolerance resource in FHIR Core (R6), modelled on the existing
`Condition.abatement[x]` and on the in-resource `AllergyIntolerance.onset[x]`.
Concretely:

- Cardinality `0..1`.
- Type choice: `dateTime | Age | Period | Range | string` (mirroring
  `Condition.abatement[x]` so the two clinical "resolved" concepts use
  the same shape).
- Definition: "The estimated or actual date/time/age the allergy or
  intolerance is considered to have abated. Should typically only be
  populated when `clinicalStatus` is `inactive` or `resolved`."
- Add a `Warning` invariant or guidance text encouraging
  `clinicalStatus in ('inactive','resolved')` when `abatement[x]` is
  populated (parallel to the guidance on `Condition.abatement[x]`).
- Deprecate (mark as "preferred replacement") the
  `http://hl7.org/fhir/StructureDefinition/allergyintolerance-abatement`
  extension in `HL7/fhir-extensions`, with a note that implementers on
  R5 and earlier should keep using the extension.
- Add a search parameter `abatement-date` consistent with Condition's
  `abatement-date` to enable date-range queries on resolved allergies.

#### Justification

The reporter's observation is internally consistent: the resource already
admits a `resolved` clinical status but provides no way to record when
that resolution occurred, forcing implementers to either lose the
information or adopt the standard extension. Adding `abatement[x]`
restores symmetry with `onset[x]` within the same resource and with
`Condition.abatement[x]` across the patterns. The Zulip thread
*AllergyIntolerance end date* contains an explicit nudge from Peter
Jordan that anyone wanting a native element should file a Jira ticket;
this ticket is that follow-through. Russell Leftwich's clinical
testimony in the same thread directly supports the use case (genuine
mechanistic resolution does occur, especially in pediatric allergies and
following immunotherapy or delabeling protocols).

---

### Disposition B: Alternative Approach

#### Proposal

Do not add `abatement[x]` to AllergyIntolerance core. Instead:

1. Promote the existing `allergyintolerance-abatement` extension to
   "Standard" / "Trial Use" status in `HL7/fhir-extensions`, ensuring it
   stays available for R5 and R6 alike, and broaden its type set to
   match `Condition.abatement[x]` (`dateTime | Age | Period | Range |
   string`) instead of just `dateTime`.
2. Add a normative cross-link from the AllergyIntolerance resource page
   (Boundaries / Notes section) to the extension, so implementers
   discover it without having to search.
3. Tighten the textual definition of `AllergyIntolerance.clinicalStatus
   = resolved` and `inactive` to explicitly recommend the extension when
   a date is known.

#### Justification

Russell Leftwich's clinical commentary cuts both ways: while abatement
is a real biological event, *dating* it is intrinsically uncertain
("only reactions and non-reactions on re-exposure" are dateable; patients
rarely report resolution). Coupled with Grahame Grieve's observation
that genuine resolution is "pretty rare," the data quality on a native
`abatement[x]` element will likely be poor and inconsistent — the same
problem that makes the existing penicillin-allergy data so untrustworthy.
Keeping the concept in an extension preserves the option for implementers
who need it, avoids a normative element that few implementers will
populate, and avoids any breaking change or migration burden for R5
implementations. It also leaves the existing
`StructureDefinition-allergyintolerance-abatement` URL stable, which is
already in use (e.g., it is referenced in the IPS conformance materials
that surfaced in the Zulip search).

---

### Disposition C: Decline

#### Proposal

Decline FHIR-54183 on the grounds that the need is *already addressed*
by the standard `allergyintolerance-abatement` extension and by the
`clinicalStatus = inactive | resolved` codes. Resolve as
"Persuasive with Mod / Not Persuasive — existing mechanism" with a
comment directing the reporter to the extension and to the prior Zulip
discussion.

#### Justification

Three independent Zulip threads spanning 2016 → 2024 have converged on
the same answer: use `clinicalStatus = resolved` plus the
`allergyintolerance-abatement` extension. The mechanism exists, is
documented, and is in active use. Adding a duplicate native element
would create two ways to express the same fact (extension *and* core
element) for the lifetime of any system that supports both R5 and R6,
which is a known source of interoperability pain. The reporter's
literature citations argue that abatement *happens*, not that the
current representation is insufficient.

---

### Recommendation

**Recommended disposition:** A

The ticket identifies a real and long-standing internal inconsistency:
`clinicalStatus` already includes `resolved`, and the resource already
carries `onset[x]`, so the absence of `abatement[x]` is a structural
anomaly, not a deliberate design choice. The fact that this gap has been
re-discovered by implementers in 2016, 2024, and again in 2026 (this
ticket) is itself evidence that an extension is not a sufficient
solution — implementers keep failing to find it. The
`Condition.abatement[x]` precedent makes the modelling decision
essentially mechanical, including the type choice and the
`clinicalStatus` coupling.

The Disposition B / C arguments around data quality (Russell Leftwich's
and Grahame Grieve's caveats) are real but apply equally to *any*
representation of allergy abatement — they argue for clear *guidance*,
not for keeping the field hidden in an extension. The R6 ballot is the
right window to add this; deferring will simply push the same ticket into
R7. The Patient Care WG should accept the request, with the explicit
addition that the existing `allergyintolerance-abatement` extension be
formally marked as superseded by the new core element going forward
while remaining valid for pre-R6 use.
