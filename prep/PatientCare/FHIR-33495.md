## Summary

The reporter (Craig Newman) flags a documentation/profile inconsistency in
the **Observation Base profile** of the *Ophthalmology Retinal (FHIR)*
Implementation Guide: the profile narrative discusses a **laterality
extension**, but no such extension is actually declared on the profile's
`StructureDefinition`. The ticket asks the Patient Care work group (in
practice, the Ophthalmology / Eye Care project team that owns the IG) to
clarify which extension is intended — i.e. either fix the narrative,
declare/slice the missing extension on `Observation.bodySite` (or the
profile root), or point the reader at an existing standard extension
(e.g. the core `bodySite` reference extension or a dedicated
laterality / body-side extension).

There are no incoming or outgoing cross-references registered for
FHIR-33495 in Augury, but several closely-related laterality tickets
exist in the corpus and are summarised under "Related Jira Tickets" below.
A more recent **Eye Care** Zulip thread ("The Two-Eyes Problem")
substantively re-opens the same modelling question for ophthalmic
observations and is summarised under "Related Zulip Discussions". Together
these show that the Eye Care community is still actively debating
*where* laterality belongs on `Observation` (in the code, in
`bodySite`, via the standard BodyStructure-reference extension, or via
`Observation.bodyStructure` pre-adopted from R5), which directly informs
how this clarification ticket should be resolved.

*Participants:* Alexander Zautke (reporter), Giorgio Cangioli, Martina

*Participants:* Alexander Henket (reporter).

Older ticket calling out narrative-vs-profile clarity issues around

### [FHIR-20236: The Base Observation profile is missing a StructureDefinition. - PHD #24](https://jira.hl7.org/browse/FHIR-20236)

Comparable IG-side issue where a Base Observation profile referenced in

## Related Zulip Discussions

### [Eye Care > The Two-Eyes Problem: Handling Laterality in Ophthalmic IGs.](https://chat.fhir.org/#narrow/stream/Eye%20Care/topic/The%20Two-Eyes%20Problem%3A%20Handling%20Laterality%20in%20Ophthalmic%20IGs./near/484145367)

A late-2024 Eye Care thread by Lars Fuhrmann that re-opens exactly the
modelling question this ticket touches. Lars proposes binding
`Observation.bodySite` to a small value set (right eye / left eye / both
eyes) for coarse "which-eye" sorting, and using either the standard
BodyStructure-reference extension or `Observation.bodyStructure`
pre-adopted from R5 (per the R4 cross-version extension technique) for
detailed location. He explicitly references the Eyes on FHIR base
Observation profile and its must-support flag for the BodyStructure
reference extension. Richard Townley-O'Neill responds suggesting (a)
preadopting R5 `Observation.bodyStructure` rather than continuing to use
the extension, and then (b) noting `obs-8` (which forbids both `bodySite`
and `bodyStructure` simultaneously) and proposing a multi-coding
`CodeableConcept` workaround. The thread ends without a binding
decision but with broad agreement that (i) laterality should not be
trapped inside `Observation.code`, and (ii) some form of profile-level
guidance is needed; it does not resolve which *extension* the
Ophthalmology Retinal IG's Observation Base profile should reference.
*Participants:* Lars Fuhrmann, Richard Townley-O'Neill.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reconcile the Observation Base profile of the *Ophthalmology Retinal
(FHIR)* IG with its narrative by **declaring an explicit laterality
mechanism on the profile**. Concretely:

1. On the `observation-base` profile, add a `must-support` extension
   slice on `Observation.bodySite.extension` (or on the profile root,
   per IG convention) bound to the FHIR core
   `http://hl7.org/fhir/StructureDefinition/bodySite` reference
   extension, which references a `BodyStructure` resource carrying
   laterality. (This is the same extension the more recent Eyes on FHIR
   build already marks as must-support, per the Zulip thread.)
2. Update the profile narrative to name that extension URL explicitly
   and to document the expected `BodyStructure` content (laterality
   coded with right / left / bilateral SNOMED CT concepts).
3. Add at least one example instance demonstrating the bound extension
   so implementers have a concrete reference.

#### Justification

This is the literal request: the narrative already promises a
laterality extension, so adding/declaring it on the StructureDefinition
removes the inconsistency while preserving the IG author's stated
intent. Reusing the **core** `bodySite` reference extension avoids
inventing yet another laterality extension — a course of action the
community has explicitly rejected in FHIR-51264 ("Extension for
laterality duplicates BodyStructure.locationQualifier", *Not
Persuasive*) — and it keeps the Ophthalmology profile aligned with the
broader Eye Care direction documented in the *Two-Eyes Problem* Zulip
thread, which already discusses this very extension and its
must-support flag in a successor build.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than declaring an extension at all, **resolve the inconsistency
in the opposite direction** by removing the laterality-extension
language from the narrative and instead **pre-adopt `Observation.bodyStructure`
from R5** on the Observation Base profile using the standard R4
cross-version extension technique
(`http://hl7.org/fhir/5.0/StructureDefinition/extension-Observation.bodyStructure`,
per <https://hl7.org/fhir/R4/versions.html#extensions>). The profile
narrative would then describe how implementers carry laterality:

- Coarse "which-eye" laterality goes in `Observation.bodySite` bound to
  a small value set (right eye / left eye / both eyes), consistent with
  the OD/OS/OU pattern Lars Fuhrmann proposes in the *Two-Eyes Problem*
  thread.
- Detailed anatomical positioning (clock-hour, segment, etc.) goes via
  the pre-adopted `Observation.bodyStructure` referencing a profiled
  `BodyStructure` resource whose `locationQualifier` (R4) /
  `laterality` (R5) carries the side concept.
- The R5 invariant `obs-8` (mutually exclusive `bodySite` /
  `bodyStructure`) would be relaxed via an IG-level note covering R4
  usage, mirroring the discussion in the same Zulip thread.

#### Justification

This avoids stacking a new must-support extension onto a profile that
has already gone through editorial drift, keeps the Ophthalmology IG
forward-compatible with R5 (which is where the WG is heading), and
respects the precedent in FHIR-43136 / FHIR-51264 that
`BodyStructure.locationQualifier` (R4) ↔ `BodyStructure.laterality`
(R5) is the *core* place to express body-side. The cost is that the
current narrative needs more substantial rewriting and IG authors must
choose between `bodySite` and `bodyStructure` per resource — a real
modelling decision, but one the Eye Care project team is already
debating openly in Zulip.

---

### Disposition C: Decline

#### Proposal

Decline the change as **out of scope for the current ballot of the
Ophthalmology Retinal (FHIR) IG**, on the grounds that:

- The IG version this ticket was filed against is no longer the active
  authoring target; the Eye Care project team has moved to a successor
  Eyes on FHIR build (referenced in the *Two-Eyes Problem* Zulip
  thread) where the BodyStructure-reference extension is already
  flagged must-support.
- The fundamental modelling question (laterality at `Observation.code`
  vs. `Observation.bodySite` vs. an extension vs.
  `Observation.bodyStructure`) is unresolved at the project-team level
  per that same Zulip thread, so any clarification added now risks
  being undone within one ballot cycle.

Recommend the reporter re-file (or transfer) against the active Eyes on
FHIR IG and that the Eye Care project team take the modelling decision
through normal ballot before returning to the editorial fix.

#### Justification

Defensible only if the WG agrees that the original *Ophthalmology
Retinal (FHIR)* IG is effectively superseded — otherwise the narrative
remains broken in a published artifact, which is precisely the class of
defect FHIR-25077 and FHIR-20236 were filed (and accepted) to correct.
Declining without a successor disposition leaves implementers with a
profile whose narrative is provably wrong, so this option is only
honest if accompanied by a clear pointer to the successor IG.

---

### Recommendation

**Recommended disposition:** A

The ticket is a five-year-old, *Highest*-priority editorial defect in a
published profile: the narrative names a construct that does not exist
on the `StructureDefinition`. That is exactly the kind of defect the
work group should fix at source, not defer. Disposition A is the
minimum-disruption change that delivers the requested clarification,
reuses an existing core extension (avoiding the FHIR-51264 anti-pattern
of minting new laterality extensions), and aligns with the
must-support flag the Eye Care community has already adopted in the
successor build per the *Two-Eyes Problem* Zulip thread. Disposition B
is technically more elegant and forward-looks toward R5, but it
expands the ticket from "make the narrative match the profile" into
"redesign laterality handling for the IG", which is properly an Eye
Care project-team decision and should be tracked separately rather
than buried inside a clarification ticket. Disposition C should only
be selected if the WG can simultaneously confirm the IG is superseded
and point the reporter at the successor — otherwise it leaves a known
defect unaddressed.
