## Summary

The Ophthalmology Retinal IG's `ObservationBase` profile slices
`Observation.category` as **Unordered, Open by `value:coding.code`,
`value:coding.system`**. Because the discriminators are `value`-based on the
code/system pair, every `coding` in any matching `category` slice must
literally match the slice's fixed code+system — no alternate ("translation")
codings can appear in the same `CodeableConcept`. As a result, profiles
derived from `ObservationBase` that add their own category slices cannot
permit multiple alternate codings within a category slice; implementers are
forced to record alternate codings as additional separate `category`
entries, which loses the semantics that all of those codings describe the
*same* category concept.

The reporter (Richard Townley-O'Neill) explicitly cross-references
[FHIR-36093](https://jira.hl7.org/browse/FHIR-36093) "VitalSIgns.category
slicing limits alternate codings", which raises the identical problem
against the FHIR Core `vitalsigns` profile. That sibling ticket has since
been resolved as **Applied / Not Persuasive with Modification**: the Vital
Signs base profile was re-sliced as **Unordered, Open by
`pattern:$this`** (using a `patternCodeableConcept` containing the
`vital-signs` coding), which lets `category:VSCat.coding` carry multiple
alternate translation codings while still requiring the canonical coding to
be present. The same fix pattern is the obvious template for
`ObservationBase` in the Ophthalmology Retinal IG.

The reporter also offered an alternative resolution in a comment: drop
slicing on `Observation.category` entirely and replace it with a
FHIRPath constraint (e.g. `category.coding.where(system='…observation-category'
and code='exam').exists()`) that asserts the required category coding is
present somewhere, without prescribing how derived profiles must structure
their own slicing.

## Details

**Description:**

[ObservationBase](http://build.fhir.org/ig/HL7/fhir-eyecare-ig/StructureDefinition-observation-base.html).category
is sliced this way: Slice: Unordered, Open by `value:coding.code`,
`value:coding.system`.

That means that no alternate codings are allowed in any slice.

If a profile is derived from `Observation.Base` and the profiler adds a
slice to category, they are unable to allow multiple alternate codings in
the new slice.

They will have to record alternate codings of a category as extra
categories.

See [FHIR-36093](https://jira.hl7.org/browse/FHIR-36093) for the same
problem with VitalSigns.

**Comments:**

### richardt (2022-02-25)

Replace the slicing of `Observation.category` with a constraint:

```xml
<constraint>
  <key value="ob-1"/>
  <severity value="error"/>
  <human value="At least one category is vital-signs"/>
  <expression value="category.coding.where(system='http://terminology.hl7.org/CodeSystem/observation-category' and code='exam').exists()"/>
</constraint>
```

(The reporter's example uses the eye-exam category code rather than
`vital-signs`; the structural pattern is the same as the one proposed and
later refined in FHIR-36093.)

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `ObservationBase` profile in the Ophthalmology Retinal IG
(`HL7/fhir-eyecare-ig`, profile
`StructureDefinition/observation-base`) so that `Observation.category`
is sliced **Unordered, Open by `pattern:$this`** instead of
`value:coding.code` + `value:coding.system`. Each existing category slice
is redefined to carry a `patternCodeableConcept` whose `coding` fixes the
slice's required `system`+`code` pair, leaving `coding` cardinality open
so derived profiles (and instance authors) may include additional
translation codings inside the same `CodeableConcept`. This mirrors the
fix already adopted in FHIR-36093 for the Vital Signs base profile.

#### Justification

This is the standard FHIR pattern for "the slice must contain a specific
coding but may carry alternate translations." It is exactly the
modification the Orders & Observations work group ultimately accepted in
FHIR-36093 for the same problem on FHIR Core's `vitalsigns` profile, so
adopting it here is consistent with cross-spec precedent and avoids
diverging slicing styles between FHIR Core observation profiles and the
Eye Care IG. It directly resolves the reporter's complaint without forcing
implementers to scatter alternate codings across multiple `category`
entries, and it is non-breaking for instances that today supply only the
canonical coding.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of (or in addition to) re-slicing, replace the value-based slicing
on `Observation.category` with one or more FHIRPath `constraint` elements,
as the reporter suggested in his comment, e.g.:

```xml
<constraint>
  <key value="ob-1"/>
  <severity value="error"/>
  <human value="At least one category coding identifies an eye exam"/>
  <expression value="category.coding.where(system='http://terminology.hl7.org/CodeSystem/observation-category' and code='exam').exists()"/>
</constraint>
```

The constraint asserts the required category coding(s) are present
somewhere in `Observation.category` without prescribing the slicing
structure that derived profiles must use. Derived eye-care profiles are
then free to add their own slices (or none) and to allow any number of
alternate codings inside any category `CodeableConcept`.

#### Justification

This addresses the *underlying* need — guaranteeing the presence of a
required category coding — while leaving derivation maximally flexible,
which is the dimension the reporter is actually constrained on. It also
sidesteps known gotchas with `pattern:$this` slicing and downstream
profile authoring (every derivative must understand the slicing model
before adding its own slices). The trade-off vs. Disposition A is that
constraints are less self-documenting in the rendered profile view than
named slices, validators surface them as constraint failures rather than
slice-membership failures, and Disposition A has the advantage of having
already been vetted and applied for the directly analogous Vital Signs
case. A hybrid is possible: keep `pattern:$this` slicing for
discoverability and add a constraint as belt-and-braces.

---

### Disposition C: Decline

#### Proposal

Decline the ticket on the basis that the existing slicing is intentional
and that derived profiles wishing to carry alternate codings should record
each coding as a separate `Observation.category` entry, which is already
permitted (`category` cardinality is `0..*`). Rationale category:
*already addressed by an existing mechanism*.

#### Justification

The defensible form of this argument is that a `CodeableConcept` whose
codings are all "translations of one another" and a repeating `category`
whose entries each carry a single canonical coding are semantically
equivalent for category-style metadata, and that value-based slicing makes
profile validation simpler and more predictable than `pattern:$this`. This
position was essentially the initial OO response on FHIR-36093 ("we
believe what you are attempting can be done"), but OO ultimately moved off
it once concrete validator failures were demonstrated. Choosing this
disposition here would reintroduce divergence from the FHIR Core Vital
Signs precedent, so it is the weakest of the three options unless Patient
Care has a specific reason to keep the Eye Care `ObservationBase` slicing
strict.

---

### Recommendation

**Recommended disposition:** A

The directly analogous ticket FHIR-36093 against FHIR Core's `vitalsigns`
profile was filed by the same reporter, debated at length in Orders &
Observations, and ultimately resolved by exactly the change Disposition A
proposes here — re-slicing `Observation.category` as **Unordered, Open by
`pattern:$this`**. Adopting the same pattern in `ObservationBase` keeps
the Eye Care IG aligned with FHIR Core's resolution of the underlying
issue, fully addresses the reporter's complaint, and is non-breaking for
existing instances. Disposition B (FHIRPath constraint) is a reasonable
fallback or supplement and should be considered if Patient Care wants to
avoid `pattern:$this` for IG-specific reasons, but on its own it loses the
self-documenting slice structure. Disposition C is hard to defend in light
of the FHIR-36093 outcome and is recommended only if Patient Care
explicitly disagrees with that precedent.
