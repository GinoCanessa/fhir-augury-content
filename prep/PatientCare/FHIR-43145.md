## Summary

Lloyd McKenzie (the reporter) is asking the PACIO Functional Status IG
maintainers to address two coupled defects in how the IG defines its
extensions. First, the extension `Context` declarations in this IG are too
loose and should be tightened so that each extension is restricted to the
resources/elements where its use actually makes sense — the same feedback
that was previously raised against the sibling PACIO Cognitive Status IG.
Second, the IG defines extensions that appear to be byte-identical to
extensions already defined in another IG (almost certainly PACIO Cognitive
Status). Defining the same extension URL/shape in two separate IGs is an
anti-pattern; the request is to resolve the duplication by either making one
PACIO IG depend on the other and reusing the existing definitions, or by
moving the shared extensions out into a "core" / shared extensions IG that
both PACIO IGs depend on. No related Jira tickets, Zulip threads, or GitHub
items were surfaced via the cross-reference index for this ticket.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Take both corrective actions in the next publication of US PACIO Functional
Status:

1. **Tighten `Extension.context` on every extension defined in the IG.**
   For each StructureDefinition whose `kind = "complex-type"` and `type =
   "Extension"`, replace any over-broad context entries (e.g., `Element`,
   `DomainResource`, or unconstrained `Resource`) with the minimal set of
   `(type, expression)` pairs that match the resources/elements where the
   extension is actually intended to be used in the PACIO Functional Status
   profiles (typically `Observation`, `Observation.component`, and the
   functional-status-related profiles defined in this IG).
2. **De-duplicate extensions shared with PACIO Cognitive Status.** Identify
   every extension whose URL and/or content matches an extension already
   defined in PACIO Cognitive Status. Remove the duplicate definition from
   PACIO Functional Status and add an `dependsOn` entry in the IG's
   `ImplementationGuide` resource pointing at PACIO Cognitive Status (or,
   if the dependency direction is wrong, do the reverse and remove the
   duplicate from Cognitive Status — Patient Care should pick a direction).
   Update every profile/example in PACIO Functional Status that references
   the now-removed local extension URL to reference the canonical extension
   URL from the dependency.

#### Justification

Both halves of the request align with long-standing FHIR design guidance.
`Extension.context` exists precisely so that tooling, validators, and
implementers can detect inappropriate use of an extension; leaving it open
defeats the validator and invites misuse. Defining the same extension URL
(or worse, two different URLs for the same concept) in two IGs from the
same project family causes interoperability friction: implementers can't
tell which URL is "the" canonical one, instances built against one IG fail
to validate against the other, and downstream IGs that want to use both
PACIO IGs end up with conflicting StructureDefinitions. The IG-dependency
mechanism is the FHIR-native way to share definitions across IGs, and the
PACIO family is exactly the kind of related-IG cluster it was designed for.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of making one PACIO IG depend on the other, **factor the shared
extensions out into a small "PACIO Common" (or "PACIO Shared Extensions")
IG** and have both PACIO Functional Status and PACIO Cognitive Status take
a dependency on it. As part of the same release cycle, still tighten
`Extension.context` on every shared extension so the contexts cover the
union of valid use sites across both PACIO IGs (and only those sites). The
duplicate definitions are then deleted from both consumer IGs and replaced
with references to the canonical URL in the shared IG.

#### Justification

A peer "common" IG avoids creating an asymmetric dependency between two
IGs that are conceptually siblings — there is no obvious reason Functional
Status should depend on Cognitive Status or vice versa, and locking in
either direction tends to age badly when one of the two IGs needs to
release independently. A small shared-extensions IG also gives PACIO a
natural home for any future extensions that turn out to apply to more than
one functional-status-style domain (ADL/IADL, mobility, cognition, etc.)
without forcing yet another bilateral dependency. The trade-off versus
Disposition A is one extra published artifact and one extra review/ballot
cycle for the new common IG, which is real overhead but pays for itself
quickly once a third PACIO-style IG appears.

---

### Disposition C: Decline

#### Proposal

Close the ticket as "Persuasive with Modification — partial" or "Not
Persuasive": agree only to tighten `Extension.context` in the next minor
release, but decline to restructure the IG to remove duplicate extension
definitions, on the grounds that PACIO Functional Status was published
standalone and changing its dependency graph or moving extension URLs
would be a breaking change for implementers who have already built
against the current published version.

#### Justification

This is the most conservative option and is defensible only if Patient
Care concludes that the implementer base of PACIO Functional Status is
large enough that any change to extension canonical URLs would cause more
harm than the duplication causes. It is the weakest disposition because it
leaves a known anti-pattern (the same extension defined in two places) in
the published artifact set, but it would be the right answer if a survey
of implementers showed strong attachment to the current URLs and no real
pain from the duplication.

---

### Recommendation

**Recommended disposition:** A

Disposition A is the most direct answer to what Lloyd actually asked for
and matches established FHIR practice for cross-IG extension reuse. The
PACIO Functional Status and PACIO Cognitive Status IGs are part of the
same project family and are already conceptually coupled, so an explicit
`dependsOn` between them is honest about the relationship and is the
lowest-overhead way to eliminate the duplicate StructureDefinitions in
this ballot cycle. Disposition B (a shared "PACIO Common" IG) is the
cleaner long-term architecture and the workgroup should keep it on the
table if a third PACIO IG is anticipated, but it adds a new ballotable
artifact and is best deferred to a future cycle. Disposition C should
only be selected if the workgroup has evidence that breaking the existing
extension URLs would cause real implementer harm; absent that evidence,
leaving duplicate extension definitions in place is exactly the kind of
defect the request is asking the WG to clean up.
