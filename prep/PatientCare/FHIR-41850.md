## Summary

The ticket is a small editorial / technical correction against the
**International Birth And Child Model (IBCM) FHIR IG**. On the page that
discusses design options for modeling a fetus in FHIR, "Option 3" is
referred to inconsistently — once with one label (e.g., "Patient
resource") and again with a different label (e.g., "Patient extension")
when the same option is cited as the project's Final decision. The
reporter asks the IG editors to pick one label for Option 3 and use it
consistently in both the option definition and the Final decision
callout, so readers do not mistake the Final decision for a fourth,
distinct option.

A closely related ticket on the same page, [FHIR-42065](https://jira.hl7.org/browse/FHIR-42065)
("Thought Process - strange title - should title be 'Fetal Data
Model'"), independently raises the same Option 3 naming problem ("Option
3: should be called 'Patient Resource Extension' otherwise, at first
glance it seems as if the 'Final Choice' is different than any in the
list"). FHIR-42065 has been resolved as **Persuasive with Modification**
(Applied), which gives the work group a concrete precedent for how to
dispose of FHIR-41850.

## Details

**Description:**

> Suggestion to use same title for Option 3 when referring to it,
> otherwise can be confusing. Thus either use "Patient resource" or
> "Patient extension" in both occasions (here as a Final decision, and
> in the definition of Option 3).

## Proposed Dispositions

The dispositions below all target the same single page in the IBCM IG —
`input/pagecontent/thought-process.html` (or its `.md` equivalent under
the IG's authoring root). Because there are no GitHub cross-references
and no cached IBCM repo briefing to draw from, the file path is named
generically; the IBCM editors should confirm the exact authoring path
in `HL7/fhir-ibcm-ig` (or whatever repository currently hosts the IG
sources) before applying the change.

### Disposition A: Accept As Requested

#### Proposal

Edit `thought-process.html` (section "Design Options for Modeling a
Fetus in FHIR") so that Option 3 and the Final decision callout use a
single, identical label. To stay aligned with how Option 3 is
implemented (as an extension on `Patient`), use **"Patient Resource
Extension"** (the label adopted via FHIR-42065's Persuasive with
Modification resolution) in:

1. The heading / first sentence that defines Option 3, and
2. The "Final decision" sentence that selects this option.

Also scrub the surrounding paragraphs so that any back-references
(e.g., "Option 3 (Patient extension)…") use the same wording. No
normative content changes; this is text-only.

#### Justification

The reporter's request is purely editorial and aligns with FHIR-42065,
which has already been resolved Persuasive with Modification on the
same page. Using the same label that was applied for FHIR-42065
guarantees both tickets converge on a single, consistent term and
removes the risk of re-introducing the ambiguity from a different
direction. The change has no downstream impact on conformance
resources, profiles, or examples.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-41850 as **a duplicate of FHIR-42065** and close it with a
"Not Persuasive — Already Applied" / "Duplicate" resolution that
points the reporter at FHIR-42065 (and the resulting Applied wording).
Before closing, the IG editor verifies that the published
`thought-process.html` already uses a single consistent label for
Option 3 in both the option definition and the Final decision; if any
residual inconsistency remains, fold a small follow-up edit into the
next IG erratum rather than reopening this ticket.

#### Justification

FHIR-42065 raises exactly the same Option 3 naming problem against the
same page and has already been resolved and applied. Tracking the same
editorial fix under two tickets risks divergent wording (the editor
could pick "Patient extension" for FHIR-41850 and "Patient Resource
Extension" — the FHIR-42065 wording — at the same time). Marking
FHIR-41850 as a duplicate is the lowest-cost way to keep the
specification self-consistent and to keep the ticket queue honest
about who actually drives the change.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive** on the basis that the "Final decision"
callout is intentionally worded differently from the option title in
order to emphasize the chosen mechanism (an extension) rather than the
option number. Leave the page unchanged.

#### Justification

This is the weakest of the three options. The only defensible
rationale for declining is that the editors actively prefer
heterogeneous wording for stylistic reasons, but that preference is
hard to justify when (a) the reporter explicitly flags the wording as
confusing and (b) FHIR-42065 has already been accepted (Persuasive
with Modification) on the same point. Recorded here for completeness
so the work group can explicitly reject it.

---

### Recommendation

**Recommended disposition:** B

FHIR-42065 has already been resolved Persuasive with Modification and
applied against the same `thought-process.html` page, with an explicit
ask to rename Option 3 to "Patient Resource Extension" so the Final
decision is no longer mistaken for a fourth option. That is the same
fix FHIR-41850 is requesting. The cleanest outcome for the work group
is to mark FHIR-41850 as a duplicate of FHIR-42065 (Disposition B),
record FHIR-42065's wording as the canonical label, and — if the
editor finds any residual inconsistency on the published page — pick
it up under the existing FHIR-42065 application rather than opening a
parallel edit thread here. If the work group prefers to keep
FHIR-41850 open for traceability (e.g., because the reporter is
external and should see a direct response), Disposition A is a fully
acceptable fallback so long as the editor reuses **"Patient Resource
Extension"** verbatim for both the option title and the Final
decision.
