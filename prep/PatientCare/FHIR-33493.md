## Summary

Craig Newman reports an inconsistency in the Ophthalmology Retinal (FHIR) IG:
the human-readable **Profiles** page lists 14 profiles, while the IG's
**Artifact list** has 15. He believes the missing entry is the *Eye Region
Finding* profile, but the names differ between the two pages, making
verification difficult. He also notes a related problem in Jira itself: the
**artifact picker on the Jira project** uses different names than the profiles
in the published IG, which makes it hard for commenters to select the correct
artifact when filing a change request. Craig proposes either syncing the two
lists (and the Jira artifact list) so all three use consistent names, or
removing the Profiles page entirely since it adds little beyond grouping. No
related Jira tickets, Zulip discussions, or GitHub items were surfaced by
cross-reference or search.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Reconcile the three places where Ophthalmology Retinal profiles are
enumerated so that they share a single source of truth and identical display
names:

1. In the IG source, regenerate the **Profiles** page from the same
   StructureDefinition inventory that feeds the **Artifact list** (typically
   the IG publisher's auto-generated `artifacts.html`). The Profiles page
   should list all 15 profiles, including *Eye Region Finding*, using each
   profile's `StructureDefinition.title` (or `name` if `title` is absent) as
   the display label.
2. Update the **Jira artifact picker** for the Ophthalmology Retinal project
   in HL7 Jira so that the artifact options exactly match the IG's profile
   `title` values. This is an HL7 Jira admin action (typically performed by
   the FMG / publishing tooling team) rather than an IG source change.
3. Add a brief automated check (or manual publisher checklist item) so future
   profile additions update all three lists together.

#### Justification

The discrepancy is a straightforward documentation/quality defect:
implementers using the Profiles page see a different inventory than they get
from the Artifact list or the Jira picker, which directly undermines the
ability to file accurate change requests against the IG. Driving both the
Profiles page and the Jira picker from the canonical
StructureDefinition set eliminates drift, follows the FHIR convention of
using IG-publisher-generated artifact lists as the authoritative inventory,
and addresses both halves of the reporter's concern (page sync + Jira sync)
without introducing any normative change.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than maintaining two parallel listings, **remove the hand-curated
Profiles page** and rely solely on the IG-publisher-generated **Artifact
list** as the single inventory of profiles. Replace the Profiles page with a
short narrative section (e.g., on the IG home page or a "Conformance
Overview" page) that explains the *grouping* of related profiles — which is
the only value the Profiles page currently adds — and links into the
Artifact list for the actual enumeration. Separately, still update the Jira
artifact picker to match the IG's profile titles, since that defect exists
independently of which IG page hosts the list.

#### Justification

The reporter himself flags the option of removing the Profiles page on the
grounds that it "doesn't add any new information (other than grouping
related profiles together)." Eliminating the duplicate listing structurally
prevents future drift, reduces maintenance burden on the IG editors, and is
consistent with how many HL7 IGs lean on the auto-generated artifact index
as the canonical inventory. Preserving the *grouping* commentary as a short
narrative keeps the only genuinely useful content from the original page.
The Jira-picker fix is retained because it is orthogonal to the IG-page
question.

---

### Disposition C: Decline

#### Proposal

Decline the change on the grounds that the Ophthalmology Retinal IG is a
small, low-traffic specification, the discrepancy is cosmetic rather than
normative, and no implementers have reported confusion in the four years
since the ticket was filed. Close as *Persuasive with Mod / Not Persuasive*
with a note that editors may sync the lists opportunistically during the
next IG publication cycle but that no formal action is required.

#### Justification

There is no normative content at stake — every profile that exists in the
IG is reachable through the Artifact list, so no implementer is being
deprived of a profile definition. No Zulip discussion, linked tickets, or
downstream GitHub activity has accumulated around this issue, suggesting
limited real-world impact. Patient Care has higher-priority Ophthalmology
Retinal work to invest editorial cycles in, and a documentation-only
cleanup may not justify a formal disposition.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses both halves of Craig's concern — the IG-internal
inconsistency *and* the Jira-picker mismatch — while structurally preventing
the same defect from reappearing the next time a profile is added or
renamed. Maintaining two hand-aligned lists (Disposition A) reliably drifts
again as soon as the next profile is added, and the reporter explicitly
acknowledges that the Profiles page contributes little beyond grouping.
Declining (Disposition C) leaves the Jira-picker problem unsolved, which is
the more impactful of the two issues because it directly degrades the
quality of incoming ballot/change-request comments. Replacing the
hand-maintained Profiles page with a short narrative + canonical Artifact
list, and aligning the Jira picker to the IG's profile titles, is the
lowest-maintenance path that fully satisfies the reporter.
