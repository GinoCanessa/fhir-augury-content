## Summary

The reporter (Virginia Lorenzi) finds the Capacity Status / pre-referral
capacity query workflow in the US SDOH Clinical Care (Gravity) IG hard to
follow from the diagrams alone. The "direct query" path is clear enough,
but it was not obvious from the diagram that *"GET HealthcareService"* is
how a system queries for capacity status. The "indirect query" path
references `Task`, `ServiceRequest`, and `Subscription` without explaining
how those interactions ultimately yield a capacity status. The ticket asks
the workgroup to add explanatory text alongside the numbered steps in the
workflow diagrams so a reader can follow what each numbered interaction is
actually doing and how it relates to capacity status.

The ticket is already triaged as **Persuasive** by the Gravity team and
bundled into the Patient Care WG block vote (Block Vote 2) tracked in the
Zulip thread *"Gravity SDOHCC STU3 - Block Vote 2"*. The proposed
resolution is to add more descriptive text explaining where Capacity Status
fits into the workflow diagrams on the Capacity Status page as part of a
broader diagram update pass. There are no cross-referenced Jira tickets and
no related GitHub items.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Disposition the ticket as **Persuasive** and update the Capacity Status
page of the US SDOH Clinical Care (Gravity SDOHCC) IG so that each
numbered step in both the *direct query* and *indirect query* sequence
diagrams is accompanied by a short narrative description. Concretely:

- For the direct-query diagram, add prose that explicitly states that
  step "GET HealthcareService" is the interaction by which a consumer
  retrieves capacity status (and identifies which element(s) on
  `HealthcareService` carry that status, e.g., the capacity-status
  extension or designated element used by the IG).
- For the indirect-query diagram, add prose for each numbered interaction
  describing the role of `Task`, `ServiceRequest`, and `Subscription`,
  and trace the end-to-end path that ultimately delivers a capacity
  status back to the requester (e.g., subscription notification carrying
  an updated `HealthcareService` or `Task` output).
- Roll this into the in-flight "diagrams at-large" refresh referenced in
  the resolution description so the wording is consistent across the IG
  rather than only on this one page.

#### Justification

This is what the reporter asked for and what the Gravity authors have
already committed to in the resolution description. The change is
narrative-only, scoped to a single IG page (plus the broader diagram
refresh already underway), and carries no impact on conformance,
profiles, value sets, or examples. It directly addresses an
implementer-facing usability defect: workflow diagrams that name FHIR
interactions without explaining the *intent* of those interactions are a
common cause of misimplementation in IGs that mix direct REST queries
with messaging/subscription patterns. Persuasive is consistent with how
Patient Care typically dispositions "improve narrative around an existing
diagram" tickets.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the underlying request but address it more structurally:
**Persuasive with Modification.** In addition to annotating the numbered
diagram steps, restructure the Capacity Status page so that:

1. A short "How to read these diagrams" preamble defines the conventions
   used (actors, lifelines, what a numbered arrow means, how each
   interaction maps to a FHIR REST/messaging operation).
2. Each diagram is followed by a numbered table — *Step → Actor →
   Interaction → FHIR resource(s) → Purpose* — so the prose stays
   parallel to the figure and is easy to maintain when the figure
   changes.
3. The indirect-query section explicitly walks through the
   `ServiceRequest` → `Task` → `Subscription` notification → updated
   `HealthcareService` (or equivalent) chain in a single end-to-end
   paragraph, before the diagram, so a reader can grasp the flow without
   reverse-engineering it from the figure.

#### Justification

The literal request — "add more text next to the numbers" — fixes this
one diagram but leaves the underlying problem (diagrams that aren't
self-explanatory) intact elsewhere in the IG. A Step/Actor/Interaction
table is a pattern other HL7 IGs (e.g., Da Vinci, US Core workflow
narratives) use successfully and is cheaper to maintain than free-form
prose tied to specific arrow numbers, because authors don't have to
re-stitch the prose every time a diagram is renumbered. The trade-off
versus Disposition A is more authoring effort up front, but better
long-term maintainability and a template Gravity can reuse on other
workflow pages during the "diagrams at-large" refresh.

---

### Disposition C: Decline

#### Proposal

Disposition as **Not Persuasive**, on the grounds that the diagrams are
sequence/workflow diagrams whose numbered arrows are explained by the
existing surrounding narrative, and the explicit FHIR interaction names
(`GET HealthcareService`, `Task`, `ServiceRequest`, `Subscription`)
already convey the technical semantics to a FHIR-literate implementer.

#### Justification

This would only be defensible if the workgroup believed the existing
narrative on the Capacity Status page already adequately explains each
numbered step, and that the reporter's confusion reflected unfamiliarity
with FHIR rather than a defect in the IG. The ticket's resolution
description, however, explicitly concedes that more descriptive text is
needed and that Gravity is already updating the diagrams; declining
would directly contradict the authors' own assessment and the
disposition already proposed in the Block Vote 2 slate.

---

### Recommendation

**Recommended disposition:** A

The Gravity authors have already triaged this as Persuasive, scoped the
fix to narrative around the Capacity Status diagrams, and folded it into
an in-flight diagram refresh; it is on the Block Vote 2 slate without
visible dissent on Zulip. The change is narrative-only, has no
conformance impact, and addresses a real implementer-facing readability
gap (the indirect-query path through `Task` / `ServiceRequest` /
`Subscription` genuinely is hard to follow from arrows alone). Disposition
B is attractive for long-term maintainability and the workgroup may want
to ask Gravity to *also* adopt a Step/Actor/Interaction table as part of
the broader refresh, but that is better handled as a follow-up
authoring-style suggestion than as a condition on this ticket. Disposition
C is not defensible given the authors' own resolution text.
