## Summary

The ticket asks the IPS authors to reword the element-definition text for
`MedicationRequest.medication[x]` and `MedicationStatement.medication[x]`
in the International Patient Summary (FHIR) IG. The current wording —
"…strongly encouraged that the reference to a medication resource is
used, **limiting** the usage of the medicationCodeableConcept only to
the cases in which no other information than a simple code is
available" — uses the word "limiting", which the reporter argues reads
like a conformance constraint when in fact the IPS authors only intend
a recommendation. The request is to rephrase the sentence (e.g., change
"limiting the usage of the medicationCodeableConcept only to cases" to
"medicationCodeableConcept is only suggested for cases") so that
readers understand it as guidance rather than a rule.

There are no cross-referenced Jira tickets — the only cross-references
are recurring IPS agenda posts on Zulip that simply list this ticket
alongside other IPS items being worked through on the project call.

## Details

**Description:**

In the element definition for `MedicationRequest.medication[x]` and
`MedicationStatement.medication[x]`:

> Identifies the medication being administered or the reason for absent
> or unknown Medication. This is either a link to a resource
> representing the details of the medication or a simple attribute
> carrying a code. To improve global interoperability, it is strongly
> encouraged that the reference to a medication resource is used,
> limiting the usage of the medicationCodeableConcept only to the cases
> in which no other information than a simple code is available.

The usage of the word "limiting" is misleading because it suggests
there is a conformance requirement or constraint.

Suggest rewording to clarify this is a recommendation, not a rule;
e.g., change "limiting the usage of the medicationCodeableConcept only
to cases" to "medicationCodeableConcept is only suggested for cases".

**Comments:**

- **JIRAUSER22800 (2025-11-25)** — Discussed on the IPS Call of
  Nov 25, 2025. The intent is *not* to constrain the use of
  `medicationCodeableConcept` only to "the cases which no other
  information than a simple code is available". The group was not
  greatly concerned about the word "limiting", but agreed that
  "recommended" might convey the intent better (with the caveat that
  "recommended" might come across as too positive vs. the deterrent
  the original wording aims for). Discussion also probed what
  "simple code" actually means — the implication is that the
  medication description should not be smuggled into
  `medicationCodeableConcept.text`. The intent is more about cases
  where there is a structured definition of the medication including
  form, ingredients and their strength, per ISO 27269:2021 Table 17
  (Medication Summary, Part 2). Two candidate replacements were
  drafted:

  1. "To improve global interoperability, it is strongly encouraged
     that the reference to a medication resource is used, **with the
     use** of medicationCodeableConcept **recommended** only in cases
     in which no other information than a simple code is available."
  2. "To improve global interoperability, it is strongly encouraged
     that the reference to a medication resource is used, it is
     **recommended** that the usage of the medicationCodeableConcept
     is **only used in** cases **where** no information **other** than
     a **medication name and** code is available."

- **gdolin (2025-12-03)** — December 3 note: the IPS spec text needs
  to be cross-checked against the ISO spec to ensure the clarifying
  wording does not conflict with ISO 27269, and to consider whether
  any reciprocal changes are needed in ISO.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the IPS IG profiles for `MedicationRequest` and `MedicationStatement`,
update the `medication[x]` element-definition text by replacing the
clause "limiting the usage of the medicationCodeableConcept only to the
cases in which no other information than a simple code is available"
with "medicationCodeableConcept is only suggested for cases in which no
other information than a simple code is available." Apply the change in
both the `MedicationRequest (IPS)` and `MedicationStatement (IPS)`
profile narratives so they remain in lock-step. No structural or
cardinality changes; this is a narrative-only edit to the element
definition.

#### Justification

The sentence sits in narrative documentation, not in a `constraint`,
binding, slicing, or `mustSupport` flag, so swapping "limiting" for
"only suggested" is a pure clarification and preserves all conformance
behaviour and instance validity. The reporter's literal proposal
removes the most obvious "this looks like a constraint" trigger word,
which is exactly the misreading the IPS team confirmed they want to
avoid. The Nov 25 IPS-call notes explicitly agreed the intent is not
to constrain `medicationCodeableConcept`, so accepting the requested
wording aligns the text with the IPS authors' actual intent without
opening any new design questions.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt one of the two reworded sentences drafted on the Nov 25 IPS call
rather than the reporter's literal suggestion, and pair the change
with a brief inline elaboration of what "simple code" means. Concretely,
replace the trailing clause with:

> "…it is strongly encouraged that the reference to a medication
> resource is used, with the use of `medicationCodeableConcept`
> recommended only in cases where no information other than a
> medication name and code is available — i.e., where a structured
> definition of the medication (form, ingredients, strength, per
> ISO 27269:2021 Medication Summary, Part 2) is not available."

Apply the change to both `MedicationRequest (IPS)` and
`MedicationStatement (IPS)` profile element definitions, and run the
final wording past the ISO 27269 editors per gdolin's Dec 3 comment to
make sure the IPS clarification does not contradict the ISO source and
to flag any reciprocal text change ISO may want to make.

#### Justification

This option addresses two distinct concerns the IPS group raised that
the reporter's literal edit does not fully solve: (1) the call group
was lukewarm on "limiting" being the real problem and felt
"recommended" carries the right normative weight, and (2) several
attendees noted that "simple code" is itself ambiguous, with
implementers sometimes interpreting it as license to dump the
medication description into `medicationCodeableConcept.text`. Pulling
in the ISO 27269 framing makes the *why* explicit (structured product
information vs. a bare code) and answers the underlying interop intent
behind the original recommendation. The ISO cross-check directly
addresses gdolin's standing concern that the IPS narrative not drift
from its ISO source.

---

### Disposition C: Decline

#### Proposal

Decline the change as not persuasive. The current sentence already
sits in informative narrative, the surrounding paragraph clearly
frames the guidance as "strongly encouraged" rather than required, and
no implementer evidence has been brought forward of an actual
validator, conformance test, or downstream profile reading "limiting"
as a constraint. Mark as "Persuasive with Mod – No Change" or
"Not Persuasive" depending on workgroup preference, and instead add a
short FAQ entry (or a one-line clarification in the IPS Profiles
overview page) noting that the medication-resource recommendation is
guidance, not a conformance rule.

#### Justification

The IPS R1 and R2 text has shipped with this wording for multiple
publication cycles without a documented misinterpretation in the wild.
The Nov 25 IPS-call discussion itself concluded there was "not a lot
of concern about the term 'limiting'", and gdolin's Dec 3 note flags a
non-trivial downstream cost (ISO 27269 cross-check) for what is
effectively a stylistic edit. Declining preserves text stability and
keeps the IPS narrative in lock-step with ISO 27269 without requiring
a coordinated change.

---

### Recommendation

**Recommended disposition:** B

The Nov 25 IPS-call discussion is unusually clear about *why* the
sentence exists: the goal is to push implementers toward structured
medication information (per ISO 27269:2021 Table 17) rather than a
free-text dump in `medicationCodeableConcept.text`, and the word
"limiting" is only a symptom of that goal being implicit. Disposition A
fixes the surface symptom but leaves "simple code" — the term that the
IPS authors themselves flagged as the more confusing phrase —
untouched, so a future ticket on the same paragraph is likely.
Disposition C ignores reporter feedback that the current phrasing
*does* read as a constraint and ducks gdolin's already-identified ISO
cross-check work, which the IPS group will have to do anyway the next
time this paragraph is revisited. Disposition B costs essentially the
same editorial effort as A, captures both the "recommended vs.
limiting" softening *and* the structured-vs-bare-code intent, and
explicitly couples the change to an ISO 27269 review so the IPS and
ISO texts stay aligned. That is the disposition most consistent with
both the reporter's request and the IPS team's own expressed intent on
the Nov 25 call.
