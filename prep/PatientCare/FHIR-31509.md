## Summary

The reporter recommends that the PACIO Cognitive Status IG explicitly differentiate between **self-report measures** (patient-reported outcomes) and **performance-based measures** when capturing cognitive/functional status. The reporter notes that for the same activity, the two measure types can yield different findings in the same individual: a self-report captures the patient's *perception* of how an impairment limits activity and participation, whereas a performance-based measure provides clinician-observed *data* on the level of impairment in body structure or function. The ticket asks the IG to make this distinction visible to implementers and consumers of cognitive status data so that the two kinds of findings are not conflated. No cross-referenced Jira tickets were returned for this issue.

## Details

**Description:**

> Recommend including differentiation between self-reported measures (Patient reported outcomes) and performance based measures.
>
> The findings from a self-report measure and from a performance-based measure for the same activity may be different in the same individual. The difference between these 2 types of measures reveals a distinction between the individual's perceived or self-measured ability to perform a task or activity and the clinically or professionally measured performance of a task or activity. Self-report measures relay information to the clinician about the individual's *perception* of how his or her impaired body function or structure is limiting activities and participation. Performance-based measures provide *data* to the clinician about the level of impairment in body structure or functions.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **US PACIO Cognitive Status (FHIR) IG** to explicitly differentiate self-report measures from performance-based measures on every cognitive-status `Observation` (and corresponding `QuestionnaireResponse`-backed) profile. Concretely:

- Add a required (or `must-support`) coded element capturing the **measure type** — either by:
  - Constraining `Observation.method` to a small bound value set such as `{self-reported, performance-based, observer-reported, mixed}` (using SNOMED CT and/or a PACIO-defined code system as appropriate), or
  - Defining a PACIO `measureType` extension on the cognitive-status profiles where `Observation.method` is otherwise in use.
- Where assessments are sourced from instruments, ensure `Observation.derivedFrom` references the `QuestionnaireResponse`, and add narrative guidance that the instrument's own classification (self-report vs. performance-based) MUST be reflected in the chosen `method` code.
- Update the IG narrative pages for the cognitive-status profiles, the assessment-instrument guidance page, and any example instances to demonstrate both flavors so implementers can see a self-report case and a performance-based case side-by-side.
- Add a section to the IG's "Interpreting Results" / guidance page explaining that the two measure types can legitimately disagree for the same activity in the same patient, and that consumers SHOULD NOT merge them as if equivalent.

#### Justification

The request is well-grounded in functional-assessment science (and aligned with the WHO ICF framing the ticket alludes to): self-report and performance-based instruments measure different constructs (perception of limitation vs. observed performance) and routinely diverge. Surfacing the distinction in a structured element rather than narrative-only is consistent with how FHIR generally encodes provenance/method on `Observation` and lets downstream systems (CDS, registries, quality measures) filter or stratify by measure type. Using `Observation.method` keeps the change additive and aligned with existing FHIR patterns rather than inventing new structure.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than constraining `Observation.method` on every cognitive-status profile, address the underlying need at the **assessment-instrument / value-set layer**:

- Tag each instrument in the PACIO assessment-instrument catalog (as `Questionnaire` resources or as concepts in the instrument value set) with a `measureType` designation (`self-report` | `performance-based` | `observer-reported`).
- Have the cognitive-status `Observation` profiles continue to use `derivedFrom` → `QuestionnaireResponse` → `Questionnaire`, and instruct consumers to retrieve the measure type from the referenced instrument rather than from the `Observation` itself.
- Optionally expose a derived/computed extension on the `Observation` for convenience, but treat the instrument as the source of truth.
- Update IG narrative to make the lookup pattern explicit, with worked examples.

#### Justification

In PACIO, the measure type is a property of the *instrument*, not of an individual observation — every result from PROMIS Cognitive Function Short Form is self-reported by definition, and every result from the Montreal Cognitive Assessment is performance-based. Encoding the type once on the instrument avoids repetition, removes the risk of an implementer mis-classifying a single result, and prevents drift between an `Observation.method` value and the instrument's actual nature. It also keeps the cognitive-status profiles unchanged, which is friendlier to existing implementers. The trade-off vs. Disposition A is that consumers must dereference the instrument to obtain the type, which adds a fetch step and is harder for ad-hoc query tools.

---

### Disposition C: Decline

#### Proposal

Decline the request on the basis that the distinction is already representable using existing FHIR machinery — implementers can populate `Observation.method` with a SNOMED CT code (e.g., self-reported vs. clinician-measured) today, and the assessment instrument referenced via `Observation.derivedFrom` / `QuestionnaireResponse.questionnaire` already disambiguates the source. Recommend adding a short non-normative note to the IG pointing this out, but make no profile or value-set changes.

#### Justification

The request describes a real clinical concern but does not identify a structural gap in the IG; the FHIR `Observation` resource already supports the differentiation through `method`, `performer`, and `derivedFrom`. A normative IG-level constraint risks over-specifying and may not align with how individual instruments classify themselves (some instruments are mixed or have observer-reported variants). Declining preserves implementer flexibility and avoids a breaking constraint on existing PACIO conformant systems.

---

### Recommendation

**Recommended disposition:** B

The distinction the reporter is asking for is genuinely important and should be visible in conformant data, so a flat decline (Disposition C) leaves a real interoperability gap — two systems could both validate against the IG yet disagree on whether a result is self-reported or performance-based. However, Disposition A pushes the type onto every individual `Observation`, which is repetitive and fragile because the measure type is fundamentally a property of the *instrument*, not of a one-off observation. Disposition B encodes the fact once, at the instrument, where it is already known and immutable, and uses the existing `derivedFrom` chain to make it discoverable. It is the minimum change that closes the interoperability gap without imposing per-observation authoring burden or risking mis-classification. Patient Care should confirm with the PACIO project team whether the assessment-instrument catalog is the right anchor point and, if so, advance Disposition B; an `Observation.method` `must-support` constraint can be layered on later as a convenience without invalidating the instrument-level tagging.
