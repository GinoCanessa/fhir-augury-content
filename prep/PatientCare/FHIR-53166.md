## Summary

Eric Haas proposes adding `DocumentReference` as an additional allowed
target type on `Procedure.basedOn`. The motivating use case is that a
`DocumentReference` may carry a scanned paper order — for example a
Portable Medical Order (PMO) — and a clinician performing a `Procedure`
needs a way to record that the procedure is fulfilling that paper
artifact. Today `Procedure.basedOn` permits only `CarePlan` and
`ServiceRequest`, neither of which cleanly represents an unstructured /
externally-issued paper order. The reporter also asks PC to consider
extending the same change to other Patient Care request/event resources
where the same logic applies.

This ticket sits inside a small family of nearly-identical "add
`DocumentReference` to `.basedOn`" requests filed in October 2025:

- [FHIR-53165](https://jira.hl7.org/browse/FHIR-53165) — same proposal
  for `ServiceRequest.basedOn` (OO). **Applied / Persuasive** — the
  precedent already exists in core for the parent request resource.
- [FHIR-52929](https://jira.hl7.org/browse/FHIR-52929) — Hans
  Buitendijk's earlier ticket that drove the OO change, explicitly
  citing PMOs / advance directives. **Applied / Persuasive with
  Modification** for `Observation.basedOn`.
- [FHIR-53163](https://jira.hl7.org/browse/FHIR-53163) — Lisa Nelson's
  parallel ask for `Composition.basedOn`. **Not Persuasive** by
  Structured Documents.
- [FHIR-53164](https://jira.hl7.org/browse/FHIR-53164) — Eric's broader
  ask to add `DocumentReference` to the **Requests pattern** itself.
  **Resolved – No Change / Not Persuasive** by FHIR Infrastructure.

The disposition here therefore turns on whether PC follows OO's
ServiceRequest / Observation precedent (accept) or aligns with FHIR-I's
refusal to generalize the Requests pattern (decline / scope to a
profile/extension).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add `Reference(DocumentReference)` to the allowed target set on
`Procedure.basedOn`, alongside the current `CarePlan` and
`ServiceRequest` targets. Specifically:

- Update the Procedure StructureDefinition so
  `Procedure.basedOn.targetProfile` includes
  `http://hl7.org/fhir/StructureDefinition/DocumentReference`.
- Add an MS=false flag extension on the new `DocumentReference` target
  to stay consistent with the per-target MS-flag pattern established
  by [FHIR-48848](https://jira.hl7.org/browse/FHIR-48848).
- Extend the element short/definition text on `Procedure.basedOn` to
  clarify that a `DocumentReference` target represents an
  externally-authored / unstructured order (e.g., a scanned Portable
  Medical Order) that the Procedure is fulfilling, contrasted with the
  structured `ServiceRequest` / `CarePlan` cases.
- File a follow-on PC sub-task to evaluate the same addition on other
  PC `.basedOn` / request-event resources the reporter alludes to
  (e.g., PC-owned event resources whose `.basedOn` could legitimately
  be fulfilled by a paper order).

#### Justification

This option directly mirrors what Orders & Observations has already
accepted for the parent request resource ([FHIR-53165, Applied
Persuasive](https://jira.hl7.org/browse/FHIR-53165)) and for
`Observation.basedOn` ([FHIR-52929, Applied Persuasive with
Modification](https://jira.hl7.org/browse/FHIR-52929)). Refusing it on
the matching event resource (`Procedure`) would create an awkward
asymmetry: implementers could record that a `ServiceRequest` was based
on a scanned PMO, but they could not record that a Procedure satisfied
the same PMO directly. Portable Medical Orders, advance directives,
and faxed/scanned referrals are real and common workflows in PC
domains (palliative care, home health, EMS), and the existing
workarounds (creating a synthetic `ServiceRequest` purely to wrap a
DocumentReference) impose unnecessary modeling overhead on
implementers. The MS-flag extension carry-forward addresses the
internal-consistency gotcha flagged by FHIR-48848.

---

### Disposition B: Alternative Approach

#### Proposal

Decline to add `DocumentReference` to base `Procedure.basedOn`, but
instead define a standard extension —
`http://hl7.org/fhir/StructureDefinition/procedure-basedOnDocument`
(or similarly named) — of type
`Reference(DocumentReference | Composition)` to be slotted onto
`Procedure` (or, by convention, alongside `Procedure.basedOn`). PC
would publish guidance describing when to use the extension (paper
PMO, advance directive, externally-authored order document) versus
the structured `CarePlan` / `ServiceRequest` targets that remain in
the core element. Implementation guides that need the linkage today
(e.g., US Core, hospice/home-health IGs) would adopt the extension.

#### Justification

This route preserves the conservative shape of `Procedure.basedOn`
that PC has historically maintained and aligns with the signal sent by
[FHIR-53164](https://jira.hl7.org/browse/FHIR-53164) (FHIR-I declined
to add `DocumentReference` to the Requests pattern at all) and
[FHIR-53163](https://jira.hl7.org/browse/FHIR-53163) (SDWG declined
the same change for `Composition.basedOn`). It also avoids the
reverse-search and validator footprint of widening a base reference
choice list across every Procedure server, and it limits the change
blast radius to consumers who explicitly opt in. The cost is that
implementers must learn a new extension and that interop with the OO
side becomes uneven (ServiceRequest uses a core element; Procedure
uses an extension), which is a real downside to weigh against the
schema conservatism.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Resolve as **Not Persuasive**. Do not add `DocumentReference` to
`Procedure.basedOn`. Direct the reporter to the existing pattern of
representing the paper order as a `ServiceRequest` whose
`.supportingInfo` (or, in some workflows, `.basedOn`) points at the
`DocumentReference`, and then making the `Procedure.basedOn` reference
that `ServiceRequest`. Cite [FHIR-53164](https://jira.hl7.org/browse/FHIR-53164)
and [FHIR-53163](https://jira.hl7.org/browse/FHIR-53163) as
precedent for not generalizing this beyond OO's already-accepted
ServiceRequest / Observation cases.

#### Justification

There is a defensible architectural argument that `.basedOn` should
point at *request* resources only, and that documents which carry
requests should be normalized into a request resource at intake rather
than being referenced directly by event resources. Both FHIR-I (on the
Requests pattern itself) and SDWG (on Composition.basedOn) have
declined this generalization, and PC would be on consistent ground in
doing the same. The downside, and the reason this is not the
recommended option, is that it pushes implementation cost onto every
party that currently has to relate a Procedure back to an
externally-authored paper order, and it diverges from the OO sibling
decision on `ServiceRequest.basedOn`.

---

### Recommendation

**Recommended disposition:** A — Accept As Requested.

The deciding factor is symmetry with
[FHIR-53165](https://jira.hl7.org/browse/FHIR-53165): OO has already
accepted (Applied / Persuasive) the identical change on the parent
request resource, motivated by the same Portable Medical Order /
advance-directive use cases that PC sees in palliative, home health,
and EMS workflows. Allowing `ServiceRequest.basedOn → DocumentReference`
while disallowing `Procedure.basedOn → DocumentReference` would force
implementers into a synthetic-`ServiceRequest` workaround to express a
fact (this Procedure fulfilled that paper order) that the requesting
side can already model directly. The extension-only path
(Disposition B) is defensible and respects FHIR-I's signal on the
Requests pattern, but it introduces a per-resource modeling
inconsistency that is hard to justify to implementers when OO has
already crossed the line at the core-element level. Disposition C
preserves architectural purity but at a real workflow cost in PC's
own domains. PC should accept the change, carry the MS-flag extension
forward per [FHIR-48848](https://jira.hl7.org/browse/FHIR-48848), and
sharpen the element definition so the `DocumentReference` target is
clearly scoped to externally-authored / unstructured orders rather
than becoming a generic escape hatch.
