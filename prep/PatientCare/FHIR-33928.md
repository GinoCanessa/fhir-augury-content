## Summary

The reporter, raising the change against the **Ophthalmology Retinal (FHIR)**
implementation guide, asks whether `Observation` status supports values other
than `final` or `amended` — specifically calling out `pending`, `in-progress`,
and `unknown`. If such values are supported, they request that the relevant
value set be updated (presumably the IG's profiled binding) to include them.
The reporter also notes that the `code` data type does not support extensions,
contrasting this with "other profiles" that apparently do allow extensions on
the status element. No comments, links, cross-references, or Zulip threads were
found via the augury cross-reference and search APIs, so the request stands on
its own without recorded follow-up discussion.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Ophthalmology Retinal IG's `Observation` profile so that its
`Observation.status` binding reflects the full set of workflow states the
reporter expects. Concretely:

- Confirm the profile is bound to the base FHIR `ObservationStatus` value set
  (`http://hl7.org/fhir/ValueSet/observation-status`), which already includes
  `registered`, `preliminary`, `final`, `amended`, `corrected`, `cancelled`,
  `entered-in-error`, and `unknown`. If the IG currently narrows this binding
  to `final | amended`, broaden it back to the base value set (or to an
  explicitly enumerated subset that includes `registered`, `preliminary`,
  `unknown` alongside `final` and `amended`).
- Document in the profile's narrative which statuses are expected to appear in
  ophthalmology retinal observations and the workflow semantics for each.
- Note explicitly that `in-progress` is **not** a member of `ObservationStatus`
  in the base spec; the closest equivalents are `registered` (created but no
  data yet) and `preliminary` (data captured but not verified).

#### Justification

The reporter's confusion suggests the IG is either silently narrowing the
status binding or under-documenting which statuses are acceptable. The base
`ObservationStatus` value set already provides the workflow states being asked
for (apart from `in-progress`), so widening / clarifying the binding aligns
the IG with FHIR's required core binding for `Observation.status` — which is
itself a `required` binding in the base spec and cannot legally be replaced by
a narrower required binding without a different status code system. This
disposition matches reporter intent at the lowest cost.

---

### Disposition B: Alternative Approach

#### Proposal

Decline to change the value set, but **clarify documentation** in the
Ophthalmology Retinal IG to address both halves of the ticket:

1. Add a "Status semantics" subsection to the `Observation` profile page that
   enumerates each base `ObservationStatus` code and the ophthalmology-specific
   workflow event it represents (e.g., `registered` = imaging session booked,
   `preliminary` = capture complete pre-read, `final` = signed off,
   `amended` / `corrected` = post-sign-off changes, `cancelled` /
   `entered-in-error` for retraction, `unknown` for legacy migrated data).
   Note that `in-progress` is intentionally absent because base
   `Observation.status` does not model multi-step task progress — that is
   `Task.status` / `ServiceRequest.status` territory.
2. Address the "code data type does not support extensions" concern by
   documenting the standard FHIR pattern: primitive elements can be extended
   via the sibling `_status` element (e.g.,
   `"_status": { "extension": [...] }`). Point to the
   `data-absent-reason` and `iso21090-nullFlavor` extensions as the canonical
   way to attach reason-style metadata to a primitive `code` value.

#### Justification

Most of what the reporter is asking for already exists in the base spec — the
issue is discoverability inside the IG, not the value set. A documentation
update avoids touching a `required` binding (which is hard to change without
breaking conformance) and corrects the misconception that primitives cannot be
extended, which would otherwise re-surface in future tickets. It also keeps
the IG aligned with base FHIR rather than introducing IG-local divergence.

---

### Disposition C: Decline

#### Proposal

Close as **Not Persuasive / Already Addressed**. The base FHIR
`ObservationStatus` value set (`http://hl7.org/fhir/ValueSet/observation-status`)
already includes `registered`, `preliminary`, `final`, `amended`, `corrected`,
`cancelled`, `entered-in-error`, and `unknown`; `Observation.status` is a
`required` binding to that value set in the base spec, which the IG cannot
narrow further without re-binding to a different code system. `in-progress` is
deliberately not part of `ObservationStatus` — workflow progress belongs on
`Task` / `ServiceRequest`, not on the result resource. Extensions on the
primitive `code` value are supported via the standard `_status` sibling
element pattern, so the second half of the request is also based on a
misunderstanding.

#### Justification

Nothing in the ticket identifies a concrete defect in the IG or the base spec.
Both concerns reflect gaps in the reporter's familiarity with FHIR's
status-code modelling and primitive-extension pattern rather than a missing
feature. Closing as not persuasive is defensible, but only if the disposition
text itself answers the reporter's questions so they don't re-file.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the underlying confusion without making structural
changes the IG cannot legally make (the base `Observation.status` binding is
`required`, which constrains how much the IG can profile it). It also
explicitly resolves the secondary concern about extending primitive `code`
values, which Disposition A leaves on the table and Disposition C closes out
without educating the reporter. A targeted documentation update inside the
Ophthalmology Retinal IG — listing the workflow statuses with
ophthalmology-specific semantics and pointing to the `_status` extension
pattern — gives implementers a concrete artifact to follow and pre-empts
re-files of the same question. If the workgroup later finds the IG is in fact
narrowing the binding (which would warrant Disposition A's value-set work),
that can be handled as a follow-up on top of the documentation fix.
