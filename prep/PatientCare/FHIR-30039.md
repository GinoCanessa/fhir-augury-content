## Summary

The ticket asks the PACIO Functional Status IG to narrow the binding on
`Observation.code` for its Functional Status profiles. The current binding
is `extensible` against the entire LOINC code system, which the reporter
considers too broad to be useful guidance for implementers. The request
is to create (or adopt) a named value set that enumerates the LOINC codes
covering the CMS post-acute-care assessment "Section GG" functional
domains, and bind `Observation.code` to that value set with an
`extensible` (or `example`) strength instead.

No related Jira tickets, Zulip threads, or GitHub items were surfaced by
cross-reference or search. The only cross-reference recorded is an
outgoing mention of the FHIR element `Observation.code`, which is the
direct target of the proposed change.

## Proposed Dispositions

(No GitHub cross-references exist for this ticket, so the dispositions
below are framed at the IG-artifact level — the PACIO Functional Status
IG's `Observation` profile(s) and accompanying `ValueSet` resource(s) —
rather than against a specific repo path. If the workgroup adopts a
change-bearing disposition, the implementing PACIO repository and file
locations should be confirmed before a PR is opened.)

### Disposition A: Accept As Requested

#### Proposal

Define a new `ValueSet` in the PACIO Functional Status IG that
enumerates the LOINC codes corresponding to the CMS Section GG
functional-status domains used by the IRF-PAI / SNF MDS / LTCH CARE /
OASIS post-acute assessment instruments (mobility, self-care, and
related GG items). Re-bind `Observation.code` on each Functional
Status `Observation` profile from the current `extensible` binding
against the entire LOINC code system to this new named value set, with
strength `extensible` (preferred) or `example` (fallback). Publish the
value set with a stable canonical URL under the IG's namespace and
include narrative documenting source mapping (Section GG item ↔ LOINC
code) and the policy for adding codes (extension rule for the
extensible binding, or "guidance only" rule for example).

#### Justification

`extensible` against the full LOINC code system gives implementers no
meaningful enumeration target — every LOINC code satisfies the
binding, defeating the purpose of an extensible binding. Naming a
value set that captures the in-scope GG domains is consistent with
how other US-realm post-acute IGs scope LOINC bindings (e.g.,
condition-specific assessment IGs and US Core's pattern of binding to
named value sets rather than entire systems). Section GG is a
well-defined external authority, so the value set has an objective
inclusion rule and is auditable. Comment history shows the only
reason this ticket was previously deferred was that no suitable value
set existed; if the HL7 functional-status work referenced by Lynn
Laakso has matured, adopting it (or its equivalent) is the requested
fix.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the base `Observation.code` binding loose but add a tighter
binding via a profiled `ObservationDefinition` (or, more practically,
split the Functional Status profiles into a small set of
domain-specific profiles — e.g., `MobilityObservation`,
`SelfCareObservation`) and bind `Observation.code` on each
narrow profile to a small `required` or `extensible` value set
containing only the LOINC codes for that domain. Optionally retain
the current broad binding on a parent profile so legacy producers are
not broken, and publish the new narrow profiles as the recommended
authoring targets. Document the migration path from the broad profile
to the domain-specific profiles in the IG's "must support" / capability
statement.

#### Justification

This addresses the underlying complaint — that "all of LOINC" is not
useful guidance — without forcing a single monolithic value set that
will need maintenance as Section GG changes. Splitting by domain
gives implementers stronger conformance signals (a Mobility
Observation can't accidentally carry a self-care code), aligns with
FHIR's preference for narrow, composable profiles, and avoids the
governance burden of the IG owning a cross-domain value set that
overlaps with future HL7 terminology work. Trade-off vs. Disposition A:
more profiles to maintain and a heavier change for existing
implementers, but a more durable terminology shape.

---

### Disposition C: Decline

#### Proposal

Decline the change and retain the existing `extensible` binding to
LOINC, on the grounds that the prior PACIO community discussion (cited
in Lynn Laakso's 2021-08-09 vote note) concluded no suitable value set
exists, and no new external value set has been brought forward in this
ticket. Recommend the requester re-file once a candidate value set
(from the referenced HL7 functional-status work or elsewhere) is
available for adoption.

#### Justification

The original "Considered for Future Use" disposition was explicitly
predicated on the absence of a fit-for-purpose value set, and the
ticket was re-opened administratively after STU1 publication rather
than because a candidate value set had landed. Adopting a value set
the IG itself authors creates an ongoing maintenance liability that
PACIO has previously chosen to avoid. Declining preserves backward
compatibility for STU1 implementers and defers the change until the
external dependency is real. This is the weakest of the three options
because it leaves the original usability problem unresolved.

---

### Recommendation

**Recommended disposition:** A

The reporter's underlying point is correct: an `extensible` binding to
the entire LOINC code system provides no actionable conformance
guidance, and Section GG is a well-defined, externally-governed scope
that the PACIO IG already builds its profiles around. The 2021
deferral was conditional on a value set becoming available; the
re-opening comment in November 2021 signals the workgroup is now
willing to revisit. A named, IG-published value set scoped to Section
GG (Disposition A) is the most direct fix, is consistent with how
other US-realm post-acute IGs constrain LOINC, and gives implementers
something to validate against. Disposition B is a stronger long-term
shape but is a much bigger change to the IG and to existing
implementations; it can be considered as a follow-up once the value
set in Disposition A is in place. Disposition C should only be
selected if the workgroup confirms that no acceptable code list
(neither the previously-referenced HL7 functional-status work nor an
IG-authored Section GG list) can be produced in this cycle.
