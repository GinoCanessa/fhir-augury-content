## Summary

Jay Lyle proposes a small wording change to the definition of
`Linkage.item.type` in FHIR Core. The current text — *"Distinguishes which
item is 'source of truth' (if any) and which items are no longer considered
to be current representations."* — is a two-way contrast that effectively
hides a third state (current items that are not the source of truth). The
ticket asks that the definition be rewritten to: *"Classifies items as the
'source of truth' (if any), current but not source of truth, and no longer
considered to be current."* This restates the element's intent in terms of
the three values it actually carries (`source`, `alternate`, `historical`)
rather than as a binary distinction.

The Patient Care call on 2025-07-17 (captured in the only comment on the
ticket) accepted the new definition wording but added two adjacent ideas
that grew the scope: (1) renaming the element from `type` to `role`, and
(2) revising the bound code system / value set to support a concern-tracking
use case. The work group decided to **defer the ticket** until item (3),
the value-set work, is logged and resolved (Stephen was to file a separate
Jira with prior documentation).

The cross-referenced ticket [FHIR-25791](https://jira.hl7.org/browse/FHIR-25791)
("Expand scope of Linkage") was independently resolved **Not Persuasive** on
2025-10-23 because no concrete use cases were produced; that closure does
not block this ticket but is relevant context — it shows the work group
recently chose *not* to expand `Linkage.item.type`'s binding strength or
add new structural elements without use cases. The Zulip thread
[*Problem Verification Statuses*](https://chat.fhir.org/#narrow/stream/implementers/topic/Problem%20Verification%20Statuses/near/523835138)
referenced the use of `Linkage` (with `Linkage.item.type = alternate`) as
one option for grouping differential-diagnosis Conditions, which is the
"current but not source of truth" case that the proposed wording would make
explicit.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `definition` of `Linkage.item.type` in the FHIR Core
specification to read:

> Classifies items as the "source of truth" (if any), current but not
> source of truth, and no longer considered to be current.

This is a pure definition-text change on the existing element; the
element name (`type`), cardinality (`1..1`), data type (`code`), and
binding (`LinkageType` value set: `source | alternate | historical`)
remain unchanged. No structural or terminology changes are made, and
the rename-to-`role` and value-set rework discussed in the 2025-07-17
call are explicitly **not** part of this disposition — they belong to
follow-up tickets.

#### Justification

The current wording presents the element as a binary distinction
(source-of-truth vs. no-longer-current) but the bound value set carries
three codes (`source`, `alternate`, `historical`). The proposed wording
mirrors the actual three-way classification and matches the very use
case raised on Zulip (using `alternate` to mark differential-diagnosis
Conditions that are current but not the source of truth). Because this
is a non-normative definition clarification with no impact on
conformance, instances, schemas, or terminology, it can be applied as a
correction even without resolving the broader rename / value-set
discussion that the work group deferred.

---

### Disposition B: Alternative Approach

#### Proposal

Bundle the definition rewrite together with the rename of
`Linkage.item.type` → `Linkage.item.role`, treating it as a single
coordinated change. Specifically:

1. Rename the element from `type` to `role` (with the standard
   backward-compatibility note in the change log).
2. Apply the new definition text from Disposition A.
3. Update the short label and any examples that reference the element
   path.
4. Continue to **defer** the value-set / concern-tracking rework
   (call item #3) to the separate Jira Stephen agreed to file.

#### Justification

The 2025-07-17 call explicitly raised renaming the element to `role`
because "type" reads as a structural classifier whereas the codes
(`source | alternate | historical`) describe the *role* an item plays
within a linkage. Coupling the rename with the definition change avoids
asking implementers to absorb two consecutive churn events on the same
element. The trade-off vs. Disposition A is that a rename is a
breaking change for any consumer that addresses the element by path
(profiles, search parameters, mappings, examples) and therefore
warrants a separate substantive review — which is precisely why the
work group deferred. Choosing this option means re-opening that
discussion now rather than later.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Persuasive with Modification → Defer** (or
equivalent), citing the existing 2025-07-17 call decision: defer this
ticket until the related code-system / value-set ticket (call item #3)
is filed and resolved, then re-evaluate the definition wording in
context. Make no immediate change to the specification.

#### Justification

The 2025-07-17 minutes already record a deferral decision pending
Stephen's follow-up Jira. Acting on the definition text now risks
having to re-revise it once the value-set semantics are settled (for
example, if new codes are added or `alternate`/`historical` are
re-scoped, the three-way phrasing in the proposed definition could
become stale). Given that Patient Care has explicitly chosen to wait,
the most defensible course is to honour that decision rather than
applying a piecemeal text fix.

---

### Recommendation

**Recommended disposition:** A

The change requested is a pure definition clarification: it does not
alter the element's name, type, cardinality, binding, or any conformance
artifact, and it more accurately reflects the three codes already in the
bound `LinkageType` value set. The Zulip *Problem Verification Statuses*
thread shows implementers reasoning about exactly the "current but not
source of truth" case (`alternate`) that the present wording obscures, so
the clarification has demonstrated value. The deferral recorded on
2025-07-17 was driven by the rename-to-`role` and value-set questions
(items #1 and #3), neither of which are coupled to the definition text;
nothing in the value-set rework would be invalidated by tightening the
wording today, and if it later is, the definition can be revised again
with no breaking impact. Disposition B is appealing but reopens the
deferred rename discussion that the work group consciously parked, and
Disposition C unnecessarily delays a low-risk fix that improves
implementer comprehension of an existing element.
