## Summary

The reporter asks the Patient Care / SDOH Clinical Care work group to review whether the `code` SearchParameter (`http://hl7.org/fhir/SearchParameter/Group-code`, expression `Group.code`) — currently advertised in the SDOHCC `CoordinationPlatform` and `ReferralSource` CapabilityStatements for the `Group` resource — is the correct search parameter for the SDOHCC-Group profile, or whether it should be replaced by (or augmented with) the `characteristic` SearchParameter (`http://hl7.org/fhir/SearchParameter/Group-characteristic`, expression `Group.characteristic.code`). The mismatch is that the SDOHCC-Group profile does not require or Must-Support `Group.code`, but it does mandate two slices on `Group.characteristic` (`HasSDOHCondition` and `HasPayer`) whose `code` is required — so the advertised search parameter targets an element implementers are not expected to populate, while the element they are required to populate has no advertised search. The reporter offers two possible resolutions: (1) add or substitute `Group-characteristic` in the CapabilityStatements, or (2) keep `Group-code` but make `Group.code` Must Support / required in the SDOHCC-Group profile. The issue has existed since STU1.

The reporter also flagged a closely related but separate ticket — [FHIR-52375](https://jira.hl7.org/browse/FHIR-52375) — covering the narrative and `valueReference` expectations for `Group.characteristic:HasSDOHCondition` (including the deeper R4 limitation around `Group-characteristic-value` composite search and the way R5 split it into `Group-characteristic-value` + `Group-characteristic-reference`). FHIR-52375 explicitly notes overlap with this ticket and is currently `Submitted` / `Unresolved`, while FHIR-52368 has already been dispositioned `Persuasive` with the work group agreeing to remove `code` and replace it with a `characteristic.code` parameter at conformance `SHALL` in the SDOHCC CoordinationPlatform CapabilityStatement.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt the resolution already recorded on the ticket: in the **SDOHCC-CoordinationPlatform** `CapabilityStatement` (and, for parity, **SDOHCC-ReferralSource**), **remove the `code` SearchParameter entry for the `Group` resource and replace it with a `characteristic-code` entry at conformance `SHALL`**, referencing `http://hl7.org/fhir/SearchParameter/Group-characteristic` (expression `Group.characteristic.code`). No structural change to the `SDOHCC-Group` profile is required; the existing `HasSDOHCondition` and `HasPayer` slices on `Group.characteristic` already require `code`, so a `SHALL`-level `characteristic-code` search is implementable today. Update the IG narrative under the SDOHCC-Group page (search parameter list / capability tables) to describe sample queries against `characteristic-code` rather than `code`.

#### Justification

This is exactly what the work group already agreed to ("Persuasive" with the resolution text on the ticket) and it aligns the advertised search surface with the elements implementers are actually required to populate. `Group.code` is neither required nor Must Support in the SDOHCC-Group profile, so advertising `Group-code` at any conformance level above MAY is misleading and gives implementers a search parameter that will return empty results against conformant data. Replacing it with `characteristic-code` (`Group.characteristic.code`) — which the profile makes mandatory through the `HasSDOHCondition` and `HasPayer` slices — is internally consistent with the profile's slicing on `Group.characteristic`, matches the IG's stated cohort-identification use cases (find groups by SDOH category code, find groups by payer-coverage code), and is a non-breaking change for any implementer that was actually populating the SDOHCC-required slices. It also keeps the disposition narrowly scoped, leaving the wider `valueReference` / composite-search redesign to its sibling ticket [FHIR-52375](https://jira.hl7.org/browse/FHIR-52375), which is the right place to handle it.

---

### Disposition B: Alternative Approach

#### Proposal

Address the mismatch from both directions and resolve FHIR-52368 jointly with [FHIR-52375](https://jira.hl7.org/browse/FHIR-52375). Concretely: (1) in the `SDOHCC-CoordinationPlatform` and `SDOHCC-ReferralSource` CapabilityStatements, **remove `Group-code`** and **add `Group-characteristic-code` at `SHALL`** as in Disposition A; (2) **also add `Group-characteristic-value` (or, if backporting the R5 split, an SDOHCC-defined `Group-characteristic-reference`) at `SHOULD`** so implementers have a documented, profiled way to query the `HasSDOHCondition` and `HasPayer` slices by `valueReference`; (3) optionally, add a profile-level invariant or narrative example showing the canonical "AND" query (`characteristic-code=...&characteristic-code=...` plus the appropriate value parameter) that the IG's cohort narrative implies. Keep `Group.code` unchanged in the `SDOHCC-Group` profile — do **not** raise it to Must Support, since doing so would force implementers to populate an element the IG does not otherwise use.

#### Justification

The `Group-code`/`Group-characteristic` mismatch in this ticket is a symptom of the same underlying issue FHIR-52375 raises about narrative-vs-conformance for the SDOHCC-Group cohort use case. Fixing only the `code → characteristic-code` swap leaves implementers without a conformance-anchored way to actually retrieve groups by SDOH category or payer reference, which is the use case the IG narrative describes. Doing both at once is more work but produces a coherent search surface and avoids a second, near-identical CapabilityStatement edit when FHIR-52375 is dispositioned. The trade-off vs. Disposition A is scope creep: this ticket was already approved with a narrower resolution, and bundling FHIR-52375's open questions in could re-open the discussion (especially around `valueReference` typing, fixed-value vs. category-code semantics, and whether to introduce R5-style `Group-characteristic-reference`). Reject the second sub-option from the ticket ("make `Group.code` Must Support") in either path — it papers over the mismatch by mandating an element the IG has no functional use for.

---

### Disposition C: Decline

#### Proposal

Decline as out of scope / no implementer harm: leave the `SDOHCC-CoordinationPlatform` and `SDOHCC-ReferralSource` CapabilityStatements as-is with the `Group-code` search parameter, on the basis that `Group-code` is part of the FHIR R4 core search-parameter set and is therefore implicitly available regardless of CapabilityStatement listing, and that the SDOHCC-Group profile's mandatory slices on `Group.characteristic` are documented in the profile itself.

#### Justification

This is defensible only if the work group concludes the CapabilityStatement is purely informative and that downstream implementers will not be misled. The ticket has been triaged as `Persuasive`, the narrative concern is concrete (the advertised search targets an element the profile does not require), and there is a clean low-cost fix available — so a decline is hard to justify on technical grounds. Including this disposition mainly so the work group can explicitly weigh and reject it; it should not be the recommended path.

---

### Recommendation

**Recommended disposition:** A

The work group has already reached `Persuasive` with a specific, surgical resolution: remove `Group-code` and replace it with `characteristic-code` at `SHALL` in the SDOHCC CoordinationPlatform CapabilityStatement (and the same change should be applied to `SDOHCC-ReferralSource` for symmetry, since the ticket flags both). Disposition A executes that resolution with no scope creep, no structural change to the `SDOHCC-Group` profile, and no risk of breaking conformant data — the `HasSDOHCondition` and `HasPayer` slices already make `Group.characteristic.code` mandatory, so a `SHALL`-level `characteristic-code` search is satisfiable on day one. Disposition B is technically attractive but conflates this ticket with [FHIR-52375](https://jira.hl7.org/browse/FHIR-52375), which is still `Submitted`/`Unresolved` and carries genuinely contested questions about `valueReference` typing and composite-search backports from R5; resolving FHIR-52368 narrowly preserves the work group's existing decision and lets FHIR-52375 be handled on its own merits. Disposition C ignores the documented mismatch and is not consistent with the existing triage outcome.
