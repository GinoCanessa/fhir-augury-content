## Summary

Caleb Wan (Patient Care WG) requests that the **US PACIO Cognitive Status
FHIR IG** be updated to incorporate the same set of resolution changes that
Patient Care has already approved for the parallel **US PACIO Functional
Status FHIR IG**. The ticket explicitly cross-references four previously
resolved Functional Status ballot comments — all already marked Persuasive
or Persuasive with Modification — and asks the Cognitive Status IG to be
brought into alignment so the two PACIO IGs remain consistent in
profiling, terminology, and narrative guidance. The four sister tickets
covered by this request are:

- [FHIR-30031](https://jira.hl7.org/browse/FHIR-30031) — switch IG
  profiles to reference US Core profiles (Encounter, Location,
  Organization, Patient, Practitioner, PractitionerRole) instead of base
  FHIR resources.
- [FHIR-30032](https://jira.hl7.org/browse/FHIR-30032) — define the
  acronym "HIT" on first use in §1.1 (resolved by spelling it out as
  "Health Information Technology (Health IT)").
- [FHIR-29754](https://jira.hl7.org/browse/FHIR-29754) — clarify the
  AssistanceRequired (formerly AssistanceNeeded) extension, bind it to
  LOINC answer list LL4309-2, and rename the *Prior Level of Function*
  profile to *Narrative History of Functional Status*.
- [FHIR-30047](https://jira.hl7.org/browse/FHIR-30047) — clarify the
  intended use of `Observation.code`, `Observation.value`, and
  `hasMember` vs. `component` for the BundledFunctionalStatus / panel
  profiles.

The ticket was filed on 2021-09-01 and is recorded with resolution
**Persuasive** ("Will apply the already approved resolution changes from
Functional Status IG to this IG."), with current status **Triaged**.

## Related Jira Tickets

### [FHIR-30031: USCore Profiles are not used in IG Profiles where available](https://jira.hl7.org/browse/FHIR-30031)

Filed by David Pyke against the PACIO Functional Status IG. Many IG
profiles bound to base FHIR resources where US Core equivalents exist;
because the IG depends on US Core, the resolution committed to using US
Core Encounter, Location, Organization, Patient, Practitioner, and
PractitionerRole. Resolved Persuasive (2021-06-07), Published.
Replicating it in Cognitive Status means swapping the same set of base
references in the Cognitive Status profiles to their US Core equivalents.

Reporter: David Pyke. Commenters: jiraadmin (import note).

### [FHIR-30032: Define Acronym](https://jira.hl7.org/browse/FHIR-30032)

Filed by Karin Frank against the PACIO Functional Status IG. The acronym
"HIT" appears in §1.1 without expansion; resolution was to change "HIT"
to "Health Information Technology (Health IT)" on first use. Resolved
Persuasive (2021-06-02), Published. Same narrative fix applies wherever
the Cognitive Status IG uses "HIT" before defining it.

Reporter: Karin Frank. Commenters: jiraadmin (import note).

### [FHIR-29754: AssistanceNeeded extension lacks values and explanation](https://jira.hl7.org/browse/FHIR-29754)

Filed by Mark Kramer against PACIO Functional Status IG. Asked for a
clearer definition of the AssistanceNeeded extension, a value set, and
consistency with FunctionalStatus. Resolved **Persuasive with
Modification** (2021-08-13): the extension's description was tightened,
it was bound to LOINC answer list **LL4309-2**, the *Prior Level of
Function* profile was renamed to *Narrative History of Functional
Status*, and the rationale for keeping the extension scoped to the
narrative profile (vs. the structured ones) was documented. Cognitive
Status carries the analogous extension/profile pattern, so the same
narrative tightening, value-set binding, and (where present) profile
rename need to be mirrored.

Reporter: Mark Kramer.

### [FHIR-30047: BundledFunctionalStatus code and value purpose](https://jira.hl7.org/browse/FHIR-30047)

Filed by Scott M. Robertson against PACIO Functional Status IG. Asked
what `Observation.code` and `Observation.value` mean for the bundled
profile, and whether `component` could substitute for `hasMember`.
Resolved **Persuasive with Modification** (2021-08-13): added narrative
to `Observation.code` (use a LOINC code for the instrument/screen/panel
title), added narrative to `Observation.value` on both the bundled and
Functional Status profiles (prefer CodeableConcept; leave unused when
summary scores live under `hasMember`; allow Ordinal Value Extension for
ordinals), and clarified that `hasMember` — not `component` — is the
right grouping mechanism for multi-observation, multi-performer, multi-day
assessments such as a CMS MDS. The Cognitive Status IG defines an
analogous bundled-assessment profile (e.g., for cognitive screens such as
MoCA / MMSE panels), which needs the same narrative refinements and the
same `hasMember`-vs-`component` guidance.

Reporter: Scott M. Robertson. Commenters: jiraadmin (import note).

## Proposed Dispositions

Because no `repo-analysis` briefing exists for the PACIO Cognitive
Status IG repo, the dispositions below identify *what* must change at
the IG-artifact level (profiles, extensions, narrative, value-set
bindings) but cannot yet name authoring roots or specific file paths.
Patient Care should treat that as a follow-up step (see Recommendation).

### Disposition A: Accept As Requested

#### Proposal

Apply, in the PACIO Cognitive Status IG, the four already-approved
Functional Status resolutions verbatim where the same construct exists,
and by direct analogy where the construct is renamed for the cognitive
domain:

1. **FHIR-30031 mirror** — Re-target every Cognitive Status profile that
   currently slices/types references against base FHIR `Patient`,
   `Encounter`, `Practitioner`, `PractitionerRole`, `Location`, or
   `Organization` to the corresponding **US Core** profiles. Add a
   dependency on the US Core version that PACIO Functional Status pinned
   in its FHIR-30031 fix (so both IGs converge on the same US Core
   anchor).
2. **FHIR-30032 mirror** — In the Cognitive Status IG narrative
   (§1.1 / introductory section), replace any standalone occurrence of
   "HIT" with "Health Information Technology (Health IT)" on first use,
   then continue using "Health IT".
3. **FHIR-29754 mirror** — For the Cognitive Status counterpart of the
   AssistanceRequired extension (and any "Prior Level of …" narrative
   profile): tighten the extension description with the same wording
   adopted for Functional Status, bind it to **LOINC LL4309-2**, and —
   if a "Prior Level of Cognitive Function" profile exists — rename it
   to "Narrative History of Cognitive Status" (matching the Functional
   Status precedent of "Narrative History of Functional Status").
4. **FHIR-30047 mirror** — On the BundledCognitiveStatus (or
   equivalently named) profile, copy the Functional Status narrative
   for `Observation.code` (LOINC code for the instrument/screen/panel
   title) and `Observation.value` (prefer CodeableConcept; leave unused
   when summary scores live under `hasMember`; ordinal values via the
   Ordinal Value Extension). On any per-screen Cognitive Status
   observation profile, add the same `hasMember`-vs-`component`
   guidance — `hasMember` is the right grouping mechanism for
   multi-observation, multi-performer, multi-day assessments.

Package these as a single IG version bump with a changelog entry that
points back to FHIR-33342 and the four source tickets.

#### Justification

The four source resolutions are already approved Patient Care positions
(three Persuasive, one Persuasive with Modification) and are Published in
the Functional Status IG. The Cognitive Status IG is the sister PACIO
publication and shares the same modeling spine (US Core anchoring, an
AssistanceRequired-style extension, a bundled assessment profile). Not
mirroring the resolutions would create a divergence between two IGs that
implementers explicitly use together, which is exactly the kind of
inconsistency the four tickets were filed to remove. The current
ticket's own resolution text — "Will apply the already approved
resolution changes from Functional Status IG to this IG." — confirms
that this is the WG's intended outcome.

---

### Disposition B: Alternative Approach

#### Proposal

Accept the *intent* of mirroring, but split the four mirrors into
**separate child tickets per resolution** (one per source ticket) and
re-validate each against the Cognitive Status IG before applying:

- Verify that every base-resource reference in Cognitive Status actually
  has a US Core equivalent the WG wants to pin (FHIR-30031 mirror).
- Confirm that `LL4309-2` is semantically appropriate for cognitive
  assistance contexts before binding it (FHIR-29754 mirror) — the
  Functional Status binding was made for ADL-style assistance and may or
  may not apply unchanged to cognitive support.
- Confirm that the Cognitive Status bundled profile uses the same
  `hasMember`/`component` shape the FHIR-30047 narrative was written
  against (FHIR-30047 mirror).
- Apply FHIR-30032 (the HIT acronym fix) immediately as a pure narrative
  patch — no analysis needed.

Track each child as a normal Patient Care ballot item against the
Cognitive Status spec; close FHIR-33342 once the children are resolved.

#### Justification

"Apply verbatim" is operationally clean but quietly assumes that the
Cognitive Status IG's structures are identical to Functional Status's.
Two of the four resolutions (FHIR-29754 — value-set binding and profile
rename; FHIR-30047 — narrative tied to specific `Observation.code` /
`Observation.value` semantics) are non-trivially shaped by Functional
Status's *modeling*. If Cognitive Status diverges in any of those
specifics, blanket replication would either (a) bind a value set that is
contextually wrong for cognition or (b) document an
`Observation.code`/`hasMember` pattern the Cognitive Status profile
doesn't actually use. Splitting the work preserves the WG's intent while
giving each mirror its own review surface.

---

### Disposition C: Decline

#### Proposal

Close FHIR-33342 without further action and instead require any
Cognitive Status IG change to be filed as a fresh ballot comment against
the Cognitive Status spec directly, citing the relevant Functional
Status resolution as precedent.

#### Justification

A single umbrella ticket that says "apply four other tickets' fixes to a
different IG" leaves an audit gap: there is no per-change reviewer
comment, no per-change resolution text scoped to the Cognitive Status
spec, and no per-change ballot record. Patient Care has historically
preferred one ticket per substantive change, and the four source
tickets' resolutions are already public — implementers and balloters
can find them. Declining and requiring per-change tickets enforces that
discipline. This option is only defensible if the WG values audit
granularity over the modest delivery delay it implies.

---

### Recommendation

**Recommended disposition:** B

The intent of FHIR-33342 — keep PACIO Cognitive Status aligned with
PACIO Functional Status — is unambiguously correct, and the ticket's own
recorded resolution ("Persuasive — will apply the already approved
resolution changes…") confirms Patient Care has already reached
consensus on that intent. Disposition C (decline) throws that consensus
away for procedural reasons and is hard to justify. Disposition A
(verbatim mirror) is the cheapest path but conflates four substantively
different changes — one is a pure narrative fix (HIT acronym), one is a
profile-reference change (US Core anchoring), and two carry semantic
commitments (a LOINC value-set binding and `Observation.code` /
`hasMember` narrative) that should be re-checked against the Cognitive
Status IG's actual structure before being copied across. Disposition B
preserves the WG's existing decision while giving the two semantically
loaded mirrors their own review surface, and it lets FHIR-30032 land
immediately as a trivial patch. As a follow-up, the PACIO Cognitive
Status IG repository should be added to the FHIR Augury repo catalog
and a `repo-analysis` briefing generated, so the next iteration of this
prep can name the specific authoring roots and file paths each child
ticket would touch.
