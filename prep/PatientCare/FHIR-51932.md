## Summary

The reporter asks whether the IPS Social History profile for Alcohol Use
(`ObservationAlcoholUseUvIps`) is insufficiently restrictive on its
`value[x]` choice. The profile is currently bound to LOINC code
[74013-4](https://loinc.org/74013-4/) ("Alcoholic drinks per day"), which
the reporter notes is a strictly *quantitative* concept — either a count
of drinks or an estimated volume of alcohol consumed. Given that, the
reporter questions why `value[x]` is not narrowed to `valueQuantity` in
the IPS profile. The IPS workgroup triaged the ticket on the 8/27 call
and resolved it as **Considered for Future Use**, deferring action until
after the 2.0 publication.

The same Social History profiles are the subject of several closely
related items raised in the same window:

- [FHIR-51969](https://jira.hl7.org/browse/FHIR-51969) (Heath Frankel)
  asks for `value[x]` to be flagged Must-Support with
  `SHALL:populate-if-known / SHALL:handle / SHOULD:display` obligations
  on `ObservationAlcoholUseUvIps` and `ObservationTobaccoUseUvIps`,
  mirroring how the Pregnancy Hx profiles are constrained.
- [FHIR-51934](https://jira.hl7.org/browse/FHIR-51934) (Alex Goel — same
  reporter) asks that the Alcohol/Tobacco Social History profiles fix
  the Observation `category` to the `social-history` code.
- [FHIR-40802](https://jira.hl7.org/browse/FHIR-40802) (John D'Amore,
  2023) is the long-standing "Social History Alcohol is too restrictive"
  ticket that flags the *opposite* problem with the same LOINC binding —
  74013-4 is "drinks per day", which prevents recording "drinks per
  week" without lossy conversion.
- [FHIR-40804](https://jira.hl7.org/browse/FHIR-40804) (Persuasive,
  Published) is the parallel Smoking Status restructuring that the
  Alcohol profile is expected to follow.

Together these tickets show that the IPS Alcohol Use profile is
under active scrutiny for both its code binding (40802) and its value
shape / Must-Support flagging (51932 and 51969), and that all three are
currently parked behind the IPS 2.0 publication.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Narrow `value[x]` on `ObservationAlcoholUseUvIps` (the IPS Social
History Alcohol Use entry profile) to `valueQuantity` only. Concretely,
in the IPS structure definition for the profile, slice/constrain
`Observation.value[x]` so that only the `Quantity` choice is allowed
(remove the other `value[x]` choices from the profile via type
restriction). Document the rationale as: the bound LOINC code 74013-4
is intrinsically quantitative (counts or volume), so non-Quantity
choices are not meaningful for this concept. Apply Must-Support to
`valueQuantity` consistent with the parallel proposal in
[FHIR-51969](https://jira.hl7.org/browse/FHIR-51969).

#### Justification

Aligning the allowed datatype with the semantics of the bound LOINC
code is consistent with normal FHIR profiling practice (when the code
fixes the meaning, restricting `value[x]` removes ambiguity for
implementers and validators). It also pairs cleanly with FHIR-51969,
which already proposes Must-Support / obligations on the same
`value[x]`; doing both at once gives the workgroup a coherent revised
profile rather than two narrow patches. The current openness of
`value[x]` is not justified by any cited use case in this ticket or in
51969.

---

### Disposition B: Alternative Approach

#### Proposal

Decline the type narrowing in `ObservationAlcoholUseUvIps` itself, and
instead address the underlying need by restructuring the LOINC binding
together with [FHIR-40802](https://jira.hl7.org/browse/FHIR-40802).
Concretely: broaden the code binding (e.g., to a small value set that
includes period-qualified alcohol-use codes, or to a binding that lets
the period be expressed without forcing a per-day denominator), then
re-evaluate `value[x]`. If after that change the bound concepts remain
purely quantitative, narrow `value[x]` to `valueQuantity`; if some
permitted codes are categorical (e.g., "current drinker / former
drinker"), keep `value[x]` open but slice it by code. This work is
naturally bundled with 51969 (MS/obligations) and 51934
(`category` fix) into a single post-2.0 Social History profile pass.

#### Justification

Narrowing `value[x]` in isolation locks the profile to today's narrow
LOINC binding and would have to be reverted if 40802 succeeds in
broadening the code binding. Doing both together avoids a churn cycle
and produces an internally consistent profile. It also matches the
precedent set by [FHIR-40804](https://jira.hl7.org/browse/FHIR-40804)
for Smoking Status, where the binding was reconsidered before
constraint tightening, and it respects the workgroup's stated decision
to defer Social History changes until after IPS 2.0.

---

### Disposition C: Decline

#### Proposal

Persist the current ticket disposition ("Considered for Future Use,"
deferred to after IPS 2.0 publication) without committing to a profile
change. Document on the ticket that 51932 will be revisited only as
part of the consolidated post-2.0 Social History profile review along
with FHIR-40802, FHIR-51934, and FHIR-51969, and close 51932 if the
broader review supersedes it.

#### Justification

The IPS workgroup has already triaged this issue and explicitly chosen
to defer; re-opening it now risks fragmenting a coordinated Social
History review across at least four overlapping tickets. There is no
new evidence in the ticket beyond the reporter's analytical question,
no Zulip or GitHub discussion driving urgency, and no implementer
report of harm caused by the current openness of `value[x]`. Declining
is therefore defensible on process grounds: act on the bundle, not the
individual question.

---

### Recommendation

**Recommended disposition:** B

Disposition B is the strongest because it directly engages the tension
between this ticket and FHIR-40802. Accepting A in isolation would
implicitly endorse the current LOINC 74013-4 binding as correct, which
40802 explicitly disputes; the workgroup would then very likely have
to re-open the profile to revert the narrowing. Declining outright (C)
defers a real ambiguity in the profile and leaves implementers with an
under-specified `value[x]`. B preserves the workgroup's stated
post-2.0 timing, bundles 51932 with 40802, 51934, and 51969 (all on
the same Social History entry profiles, all deferred), and produces a
single coherent revision rather than a sequence of patches. The
recommendation to the workgroup is therefore: keep this ticket open,
link it explicitly to 40802 / 51934 / 51969, and resolve all four
together as part of the post-2.0 Social History profile pass.
