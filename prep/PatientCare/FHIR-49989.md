## Summary

The BR&R-sponsored Pharmaceutical Quality / Chemistry, Manufacturing & Controls
(PQ/CMC) project intends to use the `Procedure` resource to represent drug
product and drug substance manufacturing process requirements. Currently
`Procedure.subject` only references patient-centric subjects
(`Patient | Group`, plus the broader patient-context targets adopted in recent
builds). The submitter is asking Patient Care to broaden the choice list on
`Procedure.subject` by adding four product-oriented references:
`Medication`, `Substance`, `BiologicallyDerivedProduct`, and
`NutritionProduct`. The ticket notes that `Observation.subject` already
permits this same set, so the precedent for "subject = the thing the action is
performed on, not just a person" exists in the spec.

This ticket is one of four sibling tickets filed by the same reporter on the
same day in support of the PQ/CMC project:

- [FHIR-49990](https://jira.hl7.org/browse/FHIR-49990) — broaden the
  **Procedure resource description** (§9.3) to acknowledge medication,
  substance, etc. as valid subjects.
- [FHIR-49991](https://jira.hl7.org/browse/FHIR-49991) — broaden the
  **Procedure Scope and Usage** (§9.3.1) to call out drug manufacturing as
  an in-scope use case.
- [FHIR-49992](https://jira.hl7.org/browse/FHIR-49992) — add a
  `manufacturing-process` code to the `plan-definition-type` value set
  (CDS workgroup; already **Resolved – Not Persuasive**).
- **FHIR-49989 (this ticket)** — the structural change on
  `Procedure.subject` that the prose changes in 49990/49991 would describe.

In effect, FHIR-49989 is the load-bearing change of the PQ/CMC bundle: the
prose tickets only make sense if `Procedure.subject` is actually broadened.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir`, modify the StructureDefinition for `Procedure` to add
`Medication`, `Substance`, `BiologicallyDerivedProduct`, and
`NutritionProduct` to the target profile list of `Procedure.subject`.
Concretely:

- Update `source/procedure/structuredefinition-Procedure.xml` (and the
  generated profile/snapshot artifacts) so `Procedure.subject.type[].targetProfile`
  becomes
  `Patient | Group | Device | Practitioner | Location | Organization | Medication | Substance | BiologicallyDerivedProduct | NutritionProduct`
  (i.e., the existing patient-context set plus the four product-context
  resources, mirroring `Observation.subject`).
- Update the element short/definition text on `Procedure.subject` to say
  "Who or what the procedure was performed on or for" (it already does)
  and add a sentence noting that for manufacturing-style procedures the
  subject may be the product being produced.
- Coordinate with FHIR-49990 / FHIR-49991 so the resource description and
  Scope & Usage sections in `Procedure` are updated in the same release
  to advertise the broadened scope.
- Add a release-notes / change-log entry under the Patient Care work
  group indicating that `Procedure.subject` gained four product-oriented
  reference targets.

#### Justification

The request is internally consistent and well-precedented:

- `Observation.subject` already permits exactly this set, so accepting it
  on `Procedure.subject` keeps the two most-used "an action / a finding
  on something" resources symmetric. Implementers who already accept
  Medication/Substance/BiologicallyDerivedProduct/NutritionProduct as
  Observation subjects pay near-zero additional cost to do the same on
  Procedure.
- BR&R's own [FHIR-48630](https://jira.hl7.org/browse/FHIR-48630)
  established that this four-resource bundle is the right "product
  subject" set, and that change was applied as Persuasive.
- The change is **purely additive** to the choice list on a Reference
  element. It does not break any existing instance: a Procedure whose
  subject today references a Patient remains valid, and validators
  simply gain four additional acceptable targets. There is no migration
  burden for existing implementers.
- The PQ/CMC use case (drug manufacturing represented as a Procedure
  performed on a Medication/Substance) is a legitimate clinical-adjacent
  workflow already being modelled by an HL7-sponsored project, and
  rejecting it forces that project to either profile around the spec or
  invent an extension on every Procedure instance, both of which are
  worse outcomes than a one-line target-profile addition.

### Disposition B: Alternative Approach

#### Proposal

Rather than widening `Procedure.subject` to the four product resources
directly, address the underlying need by:

1. **Recommend the PQ/CMC project model the manufacturing relationship
   on a different element**, specifically `Procedure.focus`
   (`Reference(Any)`), which exists precisely for "the thing the action
   was about, when it isn't the patient/subject." The Procedure would
   carry `subject = Organization` (the manufacturer / facility) and
   `focus = Reference(Medication | Substance | BiologicallyDerivedProduct
   | NutritionProduct)`.
2. If `Procedure.focus` is judged insufficient, accept the request but
   only for **`Medication` and `Substance`** in the base spec, and ask
   PQ/CMC to handle `BiologicallyDerivedProduct` and `NutritionProduct`
   via a profile (a constrained `Procedure` profile in the PQ/CMC IG
   that adds those two to `subject` via a profiled type).
3. Coordinate with O&O (steward of `BiologicallyDerivedProduct` and
   `NutritionProduct`) and Pharmacy (steward of `Medication`) before
   accepting any additions, so `Procedure.subject` does not gain target
   profiles their owners haven't endorsed.

#### Justification

- `Procedure.focus` is already `Reference(Any)` and is the spec-blessed
  escape hatch for "the thing this Procedure was about, when it isn't
  the subject." Using it costs zero core-spec change and keeps
  `Procedure.subject` semantically anchored to "who/what the procedure
  was performed on" in the patient-context sense.
- Cross-WG ownership is real: `Medication` belongs to Pharmacy,
  `BiologicallyDerivedProduct` and `NutritionProduct` to O&O. Patient
  Care should not unilaterally promote those resources into a structural
  position on Procedure without those workgroups' sign-off, and FHIR-51842
  (just-applied O&O ticket on BDP/NutritionProduct boundaries) shows
  those owners are actively cleaning up that part of the spec.
- A profile-based approach localises the broadening to the IG that
  needs it, which is the FHIR methodology preference for narrow,
  domain-specific scope expansions.

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope for `Procedure.subject`**, on the
grounds that (a) representing a manufacturing process as a clinical
`Procedure` stretches the resource's stated purpose beyond
patient-centred care, and (b) the use case is already addressable
through existing mechanisms (`Procedure.focus`, profiling, or modelling
the manufacturing process on `PlanDefinition` /
`ManufacturedItemDefinition` / a regulatory IG). Note in the resolution
that the sibling description/scope tickets (FHIR-49990, FHIR-49991)
should be declined for the same reason if this ticket is declined,
since they only describe a broadening that would not occur.

#### Justification

- Procedure has historically been a clinical workflow resource
  ("an action that is or was performed on a patient"). Promoting it to
  also model industrial manufacturing dilutes its semantics and risks
  making conformance / search behaviour confusing for the dominant
  patient-care implementer base that Patient Care serves.
- The CDS workgroup already declined the parallel
  [FHIR-49992](https://jira.hl7.org/browse/FHIR-49992) ask
  (`manufacturing-process` code in `plan-definition-type`) as Not
  Persuasive, suggesting at least one steward did not see the PQ/CMC
  use case as belonging in their core artifact. Declining FHIR-49989 is
  consistent with that signal.
- The need can be met without a base-spec change: `Procedure.focus`
  accepts any reference, and the PQ/CMC IG can constrain `Procedure`
  (or use `PlanDefinition` / `ActivityDefinition` /
  `ManufacturedItemDefinition`) to express the manufacturing
  relationship.

### Recommendation

**Recommended disposition:** A (Accept As Requested), with the cross-WG
coordination conditions from Disposition B folded in as a precondition
to merging.

The change is small, additive, breaks nothing, and brings
`Procedure.subject` into line with `Observation.subject` — which the
ticket itself correctly identifies as the precedent. BR&R's recently
applied FHIR-48630 already validated treating
Medication/Substance/BiologicallyDerivedProduct/NutritionProduct as a
coherent "product subject" set, so accepting the same four targets on
Procedure is consistent with where the methodology is already heading.
Disposition B's `Procedure.focus` workaround is technically valid, but
it pushes a sponsored HL7 project (PQ/CMC) into a less natural
modelling pattern purely to avoid a one-element widening, which is
not a good trade. Disposition C is hard to justify when the symmetric
change has already been accepted on `Observation.subject` and on
`ClinicalUseDefinition.subject`.

The Patient Care discussion should focus on the **operational
preconditions**, not the principle: (1) confirm Pharmacy is comfortable
seeing `Medication` listed as a `Procedure.subject` target; (2) confirm
O&O is comfortable with `BiologicallyDerivedProduct` and
`NutritionProduct` being added, especially in light of the
boundary-cleanup work just landed via FHIR-51842; and (3) commit to
applying FHIR-49990 and FHIR-49991 in the same release so the resource
narrative is consistent with the new structural choice.
