## Summary

Rik Smithies asks Patient Care / IPA to clarify the intent and reach of
the IPA `Observation` constraint **`ipa-obs-1`**:

> `(component.empty() and hasMember.empty()) implies (dataAbsentReason or value)`

Specifically, does this constraint deliberately rule out the **4th
Observation pattern** described in the core FHIR spec
([`observation.html#code-interop`](https://build.fhir.org/observation.html#code-interop)) —
the pattern where `Observation.code` carries a self-contained finding
(typically a SNOMED CT *finding* concept) and `value[x]` is omitted
entirely (no value, no `dataAbsentReason`, no `component`, no
`hasMember`)?

The submitter notes two distinct questions: (1) *does* `ipa-obs-1`
actually preclude pattern #4 (the chat thread shows interpreters
disagree — Lloyd McKenzie says yes, Rob Hausam says it could be argued
either way if a `dataAbsentReason` is included), and (2) was that
preclusion *intended*? He argues that pattern #4 is the natural
representation for SNOMED CT *findings* (e.g., "Bus/coach driver
(occupation)", "rash", DNR-status codes) which are designed by SNOMED
to stand alone and which are extremely common in UK GP data and other
SCT-heavy jurisdictions. Forcing such data into IPA pattern #3
(`code` + boolean `value`) requires per-instance code mapping that
sources cannot reasonably perform, so an unintended exclusion of
pattern #4 in IPA (and possibly IPS, which carries the same
constraint) would create a real interoperability impedance with
existing FHIR data.

No earlier or sibling Jira tickets were surfaced by the cross-reference
search; the only related artifact is the originating Zulip thread (see
below), where IPA leadership (Vassil Peytchev) confirms the constraint
is intended to disallow #4 but the chat does not reach consensus on
whether that intent should stand for SCT-driven implementations.

## Details

**Description:**

IPA has a constraint ipa-obs-1:
If there is no component or hasMember element then either a value\[x\]
or a data absent reason must be present:
`(component.empty() and hasMember.empty()) implies (dataAbsentReason or value)`

Is the intention of IPA to rule out the 4th Observation style here?:
<https://build.fhir.org/observation.html#code-interop>
the 4th one, where the value element is omitted

One question is does `ipa-obs-1` rule this out (differing opinions
expressed on chat). Another question is if it is intended to rule this
out.

Some clarification in the standard would be good, and/or a change, if
the constraint is determined to not give the behaviour that was is
intended.

The 4th Observation type is commonly used for SNOMED CT codes and
excluding this in IPA (and possibly IPS too) would make things
difficult for places that use SCT. If FHIR in general allows the 4th
type, then having it ruled out by IPA would lead to difficulties
interoperating between different FHIR representations. A deliberate
choice to do this (though perfectly legal) would be an overreach in a
standard with wide international applicability, imho.

**Comments:**

The Jira ticket itself has no comments yet — all substantive discussion
is on the Zulip thread captured below.

## Related Zulip Discussions

### [implementers > HL7 IPA Observation - only have a code, no value](https://chat.fhir.org/#narrow/stream/implementers/topic/HL7.20IPA.20Observation.20-.20only.20have.20a.20code.2C.20no.20value)

This is the originating thread (Zulip item `484612925`, started
2024-11-23) that Rik Smithies explicitly references when filing
FHIR-48911 ("I've raised <https://jira.hl7.org/browse/FHIR-48911> for
IPA to clarify what it is trying to constrain"). Kevin Mayfield opens
by describing UK GP-system facades that surface SNOMED CT codes alone
(no value, no parent code) and asks how to make them IPA-compliant
given `ipa-obs-1`.

The conversation crystallises around three positions:

- **Pattern #4 is prohibited and that is intentional** — Lloyd
  McKenzie and Vassil Peytchev (IPA editor) state plainly that IPA
  prohibits the code-only pattern, that putting a `dataAbsentReason`
  in to "rescue" #4 changes the semantics ("I don't have an answer"
  rather than "the code *is* the assertion"), and that data sources
  should convert to pattern #3 (`code` + boolean `value`=true) or
  pattern #1/#2 (observable-entity `code` + coded `value`). They view
  this as a deliberate IPA constraint to drive recipient consistency,
  consistent with similar tightening in IPS and US Core.
- **`ipa-obs-1` may not technically preclude #4** — Rob Hausam reads
  the constraint more narrowly: because `Observation.dataAbsentReason`
  is a separate element from `value[x]`, an instance with a
  `dataAbsentReason` (with extensible coding such as the V3
  `ASSERTION` code) and no `value[x]` *does* satisfy the invariant
  while still effectively expressing pattern #4. He suggests the spec
  text could be clarified rather than the constraint changed.
- **Pattern #4 should be permitted in IPA for SCT findings** —
  Rik Smithies argues at length that SNOMED CT *findings* (and
  related semantic-tag classes such as situation, event, social
  context) are by SCT design self-contained codes; coercing them into
  pattern #3 with `value=true` either misrepresents the data or
  forces ad-hoc, locally-invented "question" codes that are
  themselves not standardised. John Moehrke supports raising a
  comment to IPA to remove or relax the constraint, noting IPA is
  "much less mature" than the core spec and that a new DAR code
  ("never applicable") could otherwise close the gap.

The thread does not reach consensus. Vassil and Lloyd defend the
constraint on interoperability grounds; Rik and John want it relaxed
or formally clarified; Rob proposes a textual fix; Kevin and others
sit between the positions. Rik filing FHIR-48911 is the action item
that closes the chat.

**Participants:** Kevin Mayfield, Rik Smithies, Lloyd McKenzie,
René Spronk, John Moehrke, Rob Hausam, Vassil Peytchev,
Grahame Grieve, Richard Townley-O'Neill, Jeremy Rogers.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Patient Care / IPA formally **clarifies the intent and tightens the
text** of `ipa-obs-1` in `StructureDefinition-ipa-observation` so that
the constraint unambiguously prohibits pattern #4 (code-only
`Observation` with no `value[x]`, no `component`, no `hasMember`, and
no `dataAbsentReason`):

- Keep the existing FHIRPath
  `(component.empty() and hasMember.empty()) implies (dataAbsentReason or value)`
  unchanged.
- Add a `human` description and a paragraph in the IPA Observation
  profile narrative that explicitly says: "IPA does not permit the
  code-only Observation pattern described as option #4 in the core
  Observation guidance (`observation.html#code-interop`); senders
  must use one of patterns #1, #2 or #3."
- Add an example illustrating the recommended substitution
  (`code` = SNOMED finding, `value` = `boolean true`) for
  SCT-finding-only sources, mirroring the IPS pattern.
- File a parallel clarifying note (or cross-reference) on the IPS
  Observation profile so the two IGs remain aligned.

#### Justification

This is the position explicitly stated on the Zulip thread by the
IPA/IPS contributors (Vassil Peytchev, Lloyd McKenzie): the
constraint *is* intended to disallow code-only Observations, in order
to give IPA recipients a single, interoperable shape to consume. The
ambiguity that prompted the ticket (Rob Hausam reading the constraint
as *not* technically precluding #4 when a `dataAbsentReason` is
present) is purely a documentation gap, not a design disagreement
inside IPA, so a wording fix plus one example is sufficient. Keeping
parity with IPS preserves the broader "international patient summary
+ access" coherence that IPA was built on.

---

### Disposition B: Alternative Approach

#### Proposal

**Relax `ipa-obs-1` to permit pattern #4 explicitly for SNOMED CT
finding-class codes, and keep it prohibited otherwise.**

Concretely:

- Replace `ipa-obs-1` with two invariants (or a single invariant with
  an `or`-branch) on the IPA `Observation` profile:
  - `ipa-obs-1a`: existing rule applies whenever
    `Observation.code` does **not** contain a SNOMED CT concept whose
    semantic tag is one of `finding`, `disorder`, `event`, `situation`
    or `social context`.
  - `ipa-obs-1b`: when `Observation.code` *does* contain such a
    SNOMED CT concept, the requirement for `value[x]`, `component`,
    `hasMember`, or `dataAbsentReason` is dropped (i.e. pattern #4 is
    allowed).
- Document the rationale in the profile narrative, citing
  `observation.html#code-interop` rule "for case 4: preferably a
  SNOMED CT concept that is-a `404684003 | Clinical finding |`,
  `413350009 | Finding with explicit context |`, or `272379006 | Event |`".
- Encourage but do not require sources to use pattern #3 instead.

#### Justification

This addresses Rik Smithies's substantive concern: SNOMED CT findings
are by design standalone codes, and data sources (UK GP systems, but
also any SCT-native EHR) cannot reasonably re-shape every legacy
finding into a `code`+`value=true` pair on the way out. The base
FHIR Observation guidance not only allows but *recommends* pattern #4
exactly for these SCT semantic tags, so an IPA carve-out aligned to
those tags stays consistent with the core spec while preserving IPA's
stricter shape for everything else (laboratory results, vital signs,
question/answer observations, etc.). The trade-off versus
Disposition A is added recipient complexity (two valid shapes for
finding-typed Observations) in exchange for not blocking a large
class of real-world SCT data from flowing through IPA at all.

---

### Disposition C: Decline

#### Proposal

Close FHIR-48911 as **Not Persuasive — Working as Designed**, with a
short response noting that:

- `ipa-obs-1` already enforces the intended IPA rule (no code-only
  Observations); the FHIRPath is correct, and Lloyd McKenzie's
  reading is the authoritative one.
- Sources that today rely on pattern #4 for SCT findings are expected
  to adapt to pattern #3 (`code` + `value` = boolean `true`) when
  exposing data through an IPA-conformant endpoint, exactly as US
  Core, IPS and other downstream IGs require constrained shapes.
- No spec change or text clarification is needed because the
  observation-pattern guidance lives in core FHIR, not in IPA.

#### Justification

This is defensible if Patient Care concludes the existing wording is
already clear enough for implementers who read the constraint
literally, and that the cost of either editing the IPA narrative
(Disposition A) or carving out an SCT-finding exception
(Disposition B) is not justified by a single reporter's concern.
However, it ignores the documented disagreement in the Zulip thread
(Rob Hausam's reading of the constraint differs from Lloyd's), and it
leaves the SCT-finding interoperability problem entirely to data
sources, which is the exact "overreach in a standard with wide
international applicability" the submitter warns against. Decline
should therefore only be selected if Patient Care has appetite to
push back on the SCT-source community.

---

### Recommendation

**Recommended disposition: A (Accept As Requested).**

The Zulip thread shows that the IPA editors (Vassil Peytchev) and the
core FHIR design lead for Observation (Lloyd McKenzie) both intend
`ipa-obs-1` to prohibit pattern #4, and both already responded in
that voice on the thread. The disagreement that triggered the ticket
is a *documentation* problem (Rob Hausam's reading of the bare
FHIRPath leaves room for "code + DAR + no value" to slip through),
not a design problem inside IPA. Disposition A closes that
documentation gap with the smallest possible change: tighten the
human description, add an example of the IPA-preferred substitution,
and cross-link to the IPS Observation profile so the two IGs remain
aligned.

Disposition B (carve-out for SNOMED finding tags) is genuinely the
most generous answer to Rik Smithies's underlying SCT concern, but
it bifurcates the IPA shape that recipients have to handle and
re-imports the very ambiguity IPA constrained out — recipients would
need terminology-server lookups to know which shape any given
Observation might take. It would be defensible if the Patient Care
WG has clear evidence of widespread SCT-only data sources that
*cannot* perform the pattern-#3 substitution, but the chat thread
itself is a single supplier (Virtually Healthcare's facade); without
broader evidence, Disposition B's complexity cost outweighs its
benefit.

Disposition C (Decline) is not recommended: at minimum the editors
should tighten the wording given that two senior reviewers read the
same constraint differently in public.

If the WG ultimately wants to leave the door open for SCT findings,
the cleaner path is to land Disposition A now (clarify intent), and
file a separate, scoped ticket against IPA or core FHIR proposing a
new `dataAbsentReason` code (the "never applicable" / "assertion"
code John Moehrke suggested) — that is a smaller and more
broadly-useful change than rewriting `ipa-obs-1`.
