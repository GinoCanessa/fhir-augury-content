## Summary

The ticket asks the US SDOH Clinical Care (SDOH-CC) IG team to relocate the
canonical anchor for an IG-defined Code System away from `hl7.org/fhir` and
into `terminology.hl7.org` (THO). The reporter (originally a comment from
Ron G. Parker, imported into the ticket) notes that, per HL7 publishing
practice, code systems should live under THO so they can be discovered,
versioned, and reused outside the IG that first introduced them. The ticket
also calls for a companion UTG (Unified Terminology Governance) ticket to
register the affected Code System(s) in THO so that the new canonical URL is
backed by a real terminology artifact. The ticket was originally dispositioned
as **Considered for Future Use** on 2022-03-28 (vote 8-0-0, Dieterle/Hill) on
the rationale that the IG's value sets and code systems were still under
active development; that resolution was reverted post STU2 publication and
the ticket was re-opened so it can be reconsidered for the next SDOH-CC
release. No cross-referenced Jira tickets, Zulip threads, or GitHub items
were surfaced by the augury index.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Migrate the canonical URL(s) of the IG-defined `CodeSystem` resource(s) in
the **US SDOH Clinical Care** IG from the `http://hl7.org/fhir/...` namespace
to the `http://terminology.hl7.org/CodeSystem/...` namespace, and file the
corresponding **UTG** ticket(s) so that each affected `CodeSystem` is
registered and physically published under `terminology.hl7.org`. Concretely:

- Identify every `CodeSystem` resource currently authored in the SDOH-CC IG
  whose canonical begins with `http://hl7.org/fhir/...`.
- For each, open a UTG change request to register the code system under
  `http://terminology.hl7.org/CodeSystem/<id>` (preserving identifiers and
  concept content where possible).
- Once UTG accepts the registration, remove the `CodeSystem` from the
  SDOH-CC IG (or convert its IG copy to a stub that points at the THO
  canonical), and update every internal reference — `ValueSet.compose.include.system`,
  profile `binding`s, examples, narratives, and prose — to the new THO
  canonical.
- Add release notes / migration guidance describing the canonical change
  and noting that implementers MUST switch references at the next IG
  upgrade.

#### Justification

This matches HL7 publishing policy that terminology artifacts should be
hosted under THO so they can be reused, versioned, and governed
independently of any one IG, and avoids the long-term anti-pattern of
"orphan" code systems anchored under the core FHIR namespace. The
reverted 2022 resolution was explicitly conditional on the SDOH-CC code
systems being immature ("under development or constrained to this
implementation guide … we will revisit … with the next release of the
IG"), and the re-open comment confirms STU2 has now shipped, so the
maturity blocker the workgroup originally cited has been cleared. The
Patient Care / Gravity IG call already approved this direction on
2022-03-02, providing prior workgroup support.

---

### Disposition B: Alternative Approach

#### Proposal

Adopt a **phased migration** rather than a single cutover. Specifically:

- For the next SDOH-CC publication, only register and migrate code
  systems that are **stable** (no active concept churn since STU2). Each
  migrated `CodeSystem` gets a new THO canonical and the IG keeps a
  deprecation alias / mapping note from the old `hl7.org/fhir` canonical.
- Code systems still undergoing active concept work remain in the IG
  namespace for this release with a documented intent to migrate in the
  release after, so editorial churn doesn't have to round-trip through
  UTG governance for every concept change.
- File one UTG umbrella ticket that lists all SDOH-CC code systems and
  their target THO canonicals, with a per-CodeSystem sub-ticket created
  when that CodeSystem is ready to migrate.

#### Justification

Honors the original 2022 concern (immaturity of some artifacts) without
indefinitely deferring the request. UTG registration imposes external
governance overhead that is awkward for a code system whose concepts are
still being defined; a phased plan keeps editorial velocity for the
unstable pieces while still moving the IG toward THO-anchored canonicals
for the stable ones. It also gives implementers a smaller, predictable
change surface per release rather than one large URL break.

---

### Disposition C: Decline

#### Proposal

Persistent the prior **Considered for Future Use** posture: leave the
SDOH-CC code systems anchored at `http://hl7.org/fhir/...` for the
foreseeable future and close the ticket, on the basis that the IG's
terminology is still domain-constrained to SDOH-CC and is not currently
being reused outside the IG.

#### Justification

If the workgroup judges that the SDOH-CC code systems are not yet
sufficiently stable or sufficiently *cross-IG* in use to justify the
governance cost of UTG registration, declining preserves implementer
stability (no canonical URLs change, no `ValueSet.include.system`
rewrites, no broken references in deployed systems) and avoids
introducing UTG as a bottleneck for ongoing concept additions. This is
defensible only if the workgroup intends to re-confirm the same
rationale recorded in the reverted 2022 disposition.

---

### Recommendation

**Recommended disposition:** B

The original 2022 vote correctly identified that wholesale UTG migration
of *every* SDOH-CC code system is risky while concepts are still in
flux, but the 2022-11 re-open after STU2 shipping is a clear signal
that the workgroup no longer wants to keep deferring the move. A phased
migration captures both: stable code systems move to THO now (which is
what the ticket literally asks for), while still-evolving code systems
get an explicit deferral with a named next-release target rather than a
permanent home under `hl7.org/fhir`. This avoids both the implementer
shock of a single large canonical-URL break (Disposition A's main risk)
and the governance debt of leaving SDOH-CC's terminology permanently
anchored in the wrong namespace (Disposition C's main risk). It also
gives the Gravity IG team a concrete, defensible answer to the UTG
work-item question the ticket raises.
