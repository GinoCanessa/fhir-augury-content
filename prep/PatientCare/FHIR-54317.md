## Summary

The ticket reports a binding-strength inconsistency in the `Goal` resource:
the same priority value set is bound at two different strengths in two
different places. At the `Goal` level the priority codes are bound as
**Preferred**, while at `Goal.acceptance` (the nested element capturing
acceptance/agreement with the goal) the same value set is bound only as
**Example**. Reporter Eric Haas argues this is incoherent — if the codes
are good enough to be Preferred at the outer level, they should be
Preferred at the nested level too — and proposes harmonising both
bindings to **Preferred**.

No related Jira tickets were surfaced via cross-references. The only
cross-reference returned is to the FHIR artifact `Goal.acceptance`
itself (a "mentions" link recognising the element discussed in the
ticket), confirming the scope is contained to the `Goal` resource in
FHIR Core.

## Proposed Dispositions

> `acceptance.priority` element. Dispositions name that artifact

### Disposition A: Accept As Requested

#### Proposal

Strengthen the binding on `Goal.acceptance.priority` from **Example**
to **Preferred**, using the same value set already bound at
`Goal.priority` (the goal-priority value set). Update the
`StructureDefinition` for `Goal` so both bindings reference the same
value set with the same `Preferred` strength, and refresh any generated
narrative / element table that documents the binding strength on the
`acceptance.priority` element. No code-system or value-set content
changes are required; this is a pure binding-strength alignment.

#### Justification

The two elements model the same concept ("priority of this goal" vs.
"priority of acceptance of this goal") and draw from the same code
domain. Asymmetric binding strengths on the same value set within a
single resource are difficult for implementers to justify and lead
publishers and validators to treat the two elements differently
without a substantive reason. Aligning both at Preferred matches the
strength already deemed appropriate at the outer level, preserves the
ability for systems to use other codes when needed (Preferred does
not forbid alternates), and is the minimum change that resolves the
inconsistency the reporter identified.

---

### Disposition B: Alternative Approach

#### Proposal

Resolve the inconsistency by levelling **down** rather than up: keep
`Goal.acceptance.priority` at **Example** and weaken `Goal.priority`
to **Example** as well, *or* introduce a clarifying note in the
`Goal.acceptance.priority` element definition that explains why the
nested binding is intentionally weaker (e.g., acceptance priority is
captured per-acceptor and may legitimately use organisation-specific
codes the publisher does not want to bias). Optionally, scope a
narrower acceptance-priority value set if a distinct semantic is
intended, and bind it Preferred.

#### Justification

The reporter's framing assumes the two priorities are the same
concept; they may not be. Goal-level priority is the publisher's
ranking of the goal, while `acceptance.priority` is the priority an
individual participant assigns when they accept (or decline) the
goal — a more subjective, per-actor signal where the workgroup may
have deliberately wanted maximum vocabulary freedom. If that was the
original intent, the right fix is to *document* the asymmetry rather
than erase it; if the asymmetry was unintentional but Preferred is
considered too strong for either element (e.g., low real-world
implementer uptake of the goal-priority codes), levelling down keeps
both bindings honest and avoids implying a maturity the codes have
not earned.

---

### Disposition C: Decline

#### Proposal

Leave both bindings as they currently stand: `Goal.priority` =
Preferred, `Goal.acceptance.priority` = Example. Close the ticket as
"Not Persuasive" with a note that the difference in binding strength
is intentional (or at minimum tolerable) and does not warrant a
normative change.

#### Justification

A binding-strength change — even from Example to Preferred — is a
substantive normative signal to implementers and validators, and may
require regression testing of downstream IGs and profiles that
constrain `Goal`. Without evidence of implementer harm (confusion,
incorrect validator behaviour, mis-bindings in derived profiles), the
change is cosmetic. Declining keeps the existing publication stable
and avoids a churn-vs-benefit imbalance for a ticket whose only stated
rationale is symmetry.

---

### Recommendation

**Recommended disposition:** A

The reporter's observation is correct on its face: two elements that
draw from the same value set within the same resource carry different
binding strengths with no documented rationale, and that is the kind
of low-grade inconsistency that erodes trust in the spec for careful
implementers. Disposition A is the smallest defensible fix — it
aligns the two bindings at the strength the workgroup has already
endorsed for the outer element, preserves implementer flexibility
(Preferred still permits alternate codes), and requires no value-set
content work. Disposition B is worth raising in discussion only if a
Patient Care member can articulate a deliberate semantic difference
between goal priority and acceptance priority; absent that
articulation, levelling down would weaken a binding the workgroup
already chose to keep at Preferred. Disposition C is the weakest
option here because the inconsistency is real and the cost of fixing
it is low.
