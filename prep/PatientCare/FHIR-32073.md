## Summary

The reporter (Elliot Silver) observes that the International Patient Summary
(IPS) FHIR IG is internally inconsistent in how it constrains `Reference`
targets to IPS-specific profiles. Specifically, in the IPS Diagnostic Report
profile (and similar profiles elsewhere in the IG), the `performer` element is
profiled to require IPS-specific reference targets (IPS Practitioner, IPS
Organization, etc.), while a closely related element on the same resource —
`resultsInterpreter` — is **not** constrained, so it is still allowed to point
at the un-profiled base FHIR `Practitioner`, `Organization`, etc. The reporter
notes this pattern recurs in "a couple of places throughout the IG" and asks
whether the inconsistency is intentional. The IPS work group originally
resolved the ticket as "Considered for Future Use" on 2022-10-10 (vote 6-0-0,
moved by John D'Amore / Rob Hausam), noting the inconsistency was by design
for this version but worth revisiting; the resolution was then reverted by
Lynn Laakso on 2022-11-22 to re-open the ticket "post STU update publication",
and it has remained in **Triaged** status ever since.

No directly-linked Jira tickets were returned by cross-reference. The only
cross-reference is a Zulip block-vote announcement that grouped this ticket
with several other IPS items for the 2022-10-10 Patient Care meeting.

## Details

**Description:**

The performer element is profiled to reference IPS Practitioner, IPS

I've noticed this in a couple of places throughout the IG. Is this a

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Make the IPS reference targeting consistent across IPS profiles by
constraining every participant-style `Reference` element on every IPS
profile to the IPS variants (e.g., `Practitioner` →
`http://hl7.org/fhir/uv/ips/StructureDefinition/Practitioner-uv-ips`,
`Organization` → IPS Organization, `PractitionerRole` → IPS
PractitionerRole, `Patient` → IPS Patient, `Device` → IPS Device, etc.).
Concretely, in the IPS DiagnosticReport profile add `targetProfile`
constraints on `DiagnosticReport.resultsInterpreter` matching those
already on `DiagnosticReport.performer`. Sweep the rest of the IG for
analogous omissions — at minimum review every reference element that sits
adjacent to an already-IPS-constrained reference (e.g., performer/recorder/
asserter/author/attester/witness/informationSource/onBehalfOf groups on
Composition, AllergyIntolerance, Condition, Immunization, MedicationStatement,
MedicationRequest, Procedure, etc.) and apply matching `targetProfile`
constraints. Add a short authoring rule to the IG's contributor guidance so
that future participant references are constrained to IPS profiles by
default.

#### Justification

This is what the reporter is literally asking for, and it directly improves
internal consistency of the IG — a known quality concern flagged by the work
group itself in the original 2022 vote ("we may want to revisit this... and
make the references as consistent as possible"). Constraining
`targetProfile` does not narrow what wire-level resources may appear (a
server/sender can still satisfy IPS profiles using their existing
Practitioner/Organization instances); it just declares the validation
expectation, which is the mechanism IPS already uses for `performer`. Doing
this comprehensively also discharges the reporter's broader observation
("a couple of places throughout the IG") rather than fixing only the
specific example.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than constraining every participant `Reference` to an IPS profile,
**publish an explicit IG-wide reference-targeting policy** in the IPS
narrative (e.g., a "Reference targets" section in the IG home / "Profiles"
overview) stating the rule the IG follows — e.g., "All references to
participants (Practitioner, Organization, PractitionerRole, Patient,
Device) within IPS profiles SHALL/SHOULD resolve to the corresponding IPS
profile when one exists; profiles in this IG do not always re-state this
constraint at the element level." Then fix only the obvious local
inconsistencies — most notably
`DiagnosticReport.resultsInterpreter` to align with
`DiagnosticReport.performer` — and leave the rest to the published policy.
Optionally add a FHIRPath invariant or a validator-friendly
`obligation`/`mustSupport` note rather than a hard `targetProfile`
constraint.

#### Justification

A blanket profile-targeted sweep (Disposition A) is non-trivial: it
touches many StructureDefinitions, can interact with IPS profiles' own
`mustSupport` slicing, and risks invalidating real-world IPS instances
that today reference base resources (especially across organizational
boundaries where a sending system may not have asserted IPS conformance
on its Practitioner/Organization records). A narrative policy plus
targeted local fixes preserves implementer flexibility while still
addressing the reporter's "is this conscious?" question with a clear
"yes, and here is the rule." This also respects the IG's previous
deliberate choice (the 2022 "by design" note) by codifying it instead of
overturning it wholesale.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Resolve as **Not Persuasive** with the rationale that the existing IPS
profiles are the result of a deliberate scoping decision: where IPS
*requires* a reference to be to an IPS-conformant resource (e.g.,
`performer` on a DiagnosticReport-IPS), the profile says so; where IPS
deliberately leaves room for non-IPS references (e.g.,
`resultsInterpreter`, which may be a consultant or external organization
not under IPS conformance), the profile intentionally does not constrain
the target. Document this rationale in the resolution comment but make no
spec changes.

#### Justification

This matches the original 2022-10-10 work-group judgement ("This was done
by design") and avoids re-litigating a decision that had a 6-0-0 vote.
Tightening every participant reference to IPS profiles risks producing
false-negative validation on perfectly reasonable IPS instances that
include external participants whose source systems do not assert IPS
conformance — a meaningful concern given that IPS is explicitly designed
for cross-border / cross-organisational summary exchange. A "Not
Persuasive" close keeps the IG stable and pushes any genuine consistency
work to a future major version where breaking changes are in scope.

---

### Recommendation

**Recommended disposition:** B

The ticket is genuinely valid — the IG *is* internally inconsistent in a
way that is reasonable for an implementer to flag, and the work group
itself acknowledged as much when it first considered the ticket and again
when Lynn Laakso re-opened it post-STU. Pure decline (C) ignores that
acknowledgement and leaves the inconsistency for the next reader to
re-discover. A blanket sweep (A) is the most thorough fix but is high
effort and carries real interoperability risk for a cross-jurisdictional
summary IG, where senders frequently include participants whose source
systems will never assert IPS conformance on Practitioner/Organization
records. Disposition B threads the needle: it states the IG's actual
intent in narrative form (so the reporter's "is this conscious?" question
gets a documented "yes"), it fixes the most defensible local
inconsistency (`resultsInterpreter` aligned with `performer` on the same
profile), and it leaves room for a future major version to do the
broader sweep with proper migration guidance. It also has the smallest
chance of breaking existing IPS implementations, which should weigh
heavily for a published, in-use IG.
