## Summary

The reporter requests that the Ophthalmology Retinal (FHIR) implementation
guide embed its use cases directly into the profile/IG content rather than
linking out to an external Google Doc. The concern is that some
organizations and entities cannot reach Google Docs (for example, due to
corporate firewalls, regional access restrictions, or institutional
policies that block Google services), which leaves them unable to read
the use cases that motivate and explain the profile. No cross-referenced
Jira tickets, Zulip threads, or GitHub items were surfaced for this
ticket, so the request stands on its own as an editorial / publishing
issue against the Ophthalmology Retinal IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the Ophthalmology Retinal (FHIR) IG so that the use-case content
currently hosted in the external Google Doc is inlined into the IG's
narrative pages. Concretely:

- Add (or expand) a "Use Cases" page in the IG's narrative section that
  contains the full text, diagrams, and any actor/workflow descriptions
  that today live in the Google Doc.
- Replace the existing Google Docs hyperlink with an in-IG link to the
  new Use Cases page. Keep the Google Doc URL only as a "source"
  citation/footnote, not as the primary access path.
- If the use cases include diagrams or tables, port them in as PlantUML
  / images / markdown tables checked into the IG source so they render
  in the published HTML and remain available offline.
- Re-publish the IG so the consolidated use cases are part of the
  balloted/normative artifact.

#### Justification

This is exactly what the reporter asks for, and it aligns with standard
HL7 publishing practice: ballot- and standards-quality content should
live inside the IG so that it is versioned with the IG, accessible to
all implementers regardless of network policy, indexed by the FHIR IG
toolchain, and preserved as the IG evolves. Externally-hosted Google
Docs are mutable, can be deleted or have permissions changed at any
time, are not part of the published artifact reviewers are voting on,
and are unreachable from environments that block Google services
(common in healthcare IT and several jurisdictions). Inlining the
content removes a real accessibility barrier and improves durability
of the spec without changing any computable conformance.

---

### Disposition B: Alternative Approach

#### Proposal

Inline a self-contained **summary** of each use case in the IG (one or
two paragraphs per use case, plus actor list and primary flow), and
move the long-form / supporting material out of Google Docs into a
durable, openly-accessible location that is still separate from the
IG — for example, a markdown file in the IG's source repository (under
`input/pagecontent/` or an equivalent supplementary folder) or a page
in the Confluence space owned by the work group. The IG would link to
that durable location for the full text.

#### Justification

If the full Google Doc material is large, contains many diagrams, or is
expected to keep evolving outside the IG ballot cadence, inlining
everything verbatim may bloat the IG and force an IG re-publish each
time the use case narrative is tweaked. A two-tier approach — a
ballot-quality summary inside the IG plus a longer companion document
in a non-Google, openly-reachable location — addresses the reporter's
core complaint (Google Docs is not universally accessible) without
overcommitting the IG to carrying every supplementary detail. It also
preserves an editable working surface for the authors between
publication cycles.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the use cases in the existing external
Google Doc, on the basis that use-case narrative is supplementary
authoring material rather than normative IG content, and that the
existing link is sufficient for the IG's intended audience.

#### Justification

This disposition is hard to defend. The only credible grounds for
declining would be that the use cases are explicitly out of scope for
the IG itself, or that the work group has already committed to
maintaining them externally. Neither is asserted in the ticket or
surfaced in any related discussion. The reporter's accessibility
concern is concrete (some organizations cannot reach Google Docs at
all) and would mean those implementers cannot read the use cases that
justify the profile — a real harm with no offsetting benefit. Decline
should only be selected if the work group decides the use cases are
genuinely informative-only background and intentionally outside the
balloted IG.

---

### Recommendation

**Recommended disposition:** A

Accepting the request as written is the cleanest fix and matches HL7's
general expectation that an IG be self-contained and readable by all
implementers from the published artifact. The change is editorial — it
does not affect any FHIR resource, profile element, cardinality,
binding, or invariant — so the cost is purely authoring effort, and
the benefit (every implementer can read the use cases, content is
versioned and balloted with the IG, no dependency on a third-party
service that some organizations cannot reach) is large and durable.
Disposition B is a reasonable fallback if the Google Doc turns out to
be very large or to contain material the work group genuinely wants to
keep evolving outside the ballot cadence; in that case the IG should
still carry at least a self-contained summary so the narrative is not
gated on Google Docs access. Disposition C should only be chosen if
the work group makes an explicit decision that the use cases are out
of scope for the IG.
