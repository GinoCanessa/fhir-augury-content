## Summary

The ticket requests adding a new level-2 code, `respnet-case-report-message`,
to the message-type CodeSystem maintained in the **US Public Health Profiles
Library (USPHPL)**. The new code is intended to support the RESP-NET
(US Respiratory Virus Hospitalization Surveillance Network) FHIR IG so that a
MessageHeader sent from a clinical care system to a Public Health Authority
(PHA) can declare that the payload is a RESP-NET case report. The reporter
proposes:

- **Code:** `respnet-case-report-message`
- **Display:** "Indicates a message containing a resp-net case report data sent
  from clinical care to PHA."
- **Definition:** "Indicates that resp-net data are present in the message
  being sent from clinical care to PHA."

Although filed against the **Patient Care** work group, the artifact actually
lives in **Public Health** (the USPHPL message-type CodeSystem), so this
ticket should likely be reassigned at triage. Context comes from related
RESP-NET work — particularly [FHIR-43680](https://jira.hl7.org/browse/FHIR-43680)
(scope of the RESP-NET disease-burden use case) and
[FHIR-45328](https://jira.hl7.org/browse/FHIR-45328) (tagging RESP-NET bundles
by use case) — which establish that RESP-NET reports are a distinct PHA-bound
flow that warrants its own message-type identifier. The closely-analogous
[FHIR-33479](https://jira.hl7.org/browse/FHIR-33479) (cancer case-report
message) and [FHIR-38348](https://jira.hl7.org/browse/FHIR-38348)
(USPHPL/MedMorph alignment) confirm the established pattern of one
clinical-care → PHA message-type code per content IG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new concept to the **US Public Health MessageType CodeSystem**
(USPHPL `CodeSystem/us-ph-messageheader-message-types`, the artifact aligned
with MedMorph/eCR per FHIR-38348) and include it in the bound
MessageType ValueSet:

- **Code:** `respnet-case-report-message`
- **Display:** "RESP-NET Case Report Message"
- **Definition:** "Indicates that RESP-NET data are present in the message
  being sent from clinical care to a Public Health Authority (PHA)."
- **Hierarchy:** level 2 under the existing top-level "case-report" / PHA
  message group, sibling to the existing per-IG case-report codes
  (e.g., cancer-report-message, ecr-case-report-message).

In the RESP-NET IG, fix or strongly bind `MessageHeader.event.code` to the new
code in the RESP-NET MessageHeader profile so RESP-NET messages are
self-identifying. Re-route the ticket from Patient Care to **Public Health**
at triage, since USPHPL is owned by Public Health.

#### Justification

The MessageType CodeSystem in USPHPL is the established home for content-IG
PHA-bound message types (FHIR-38348). The precedent in FHIR-33479 explicitly
endorses a distinct code per IG/exchange-direction so that the receiving PHA
can dispatch on `MessageHeader.event.code` without ambiguity. Tagging
RESP-NET bundles by use case was already accepted in FHIR-45328, and the
disease-burden flow described in FHIR-43680 is plainly a different exchange
than eCR or cancer reporting. The proposed code, display, and definition are
internally consistent with siblings already in the CodeSystem and require no
breaking change — only an additive concept and ValueSet membership.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than adding a single hard-coded RESP-NET code to the shared USPHPL
MessageType CodeSystem, define a **RESP-NET-local CodeSystem** in the
RESP-NET IG (e.g., `CodeSystem/resp-net-message-types`) that contains
`respnet-case-report-message`, then **extend** the USPHPL MessageType
ValueSet to compose-include that RESP-NET CodeSystem. Bind
`MessageHeader.event.code` in the RESP-NET MessageHeader profile to either
(a) the RESP-NET CodeSystem directly, or (b) the extended USPHPL ValueSet.

#### Justification

This pattern keeps IG-specific vocabulary in the IG that owns it, avoiding
USPHPL accumulating one new code per downstream content IG over time
(MedMorph, eCR, Cancer, RESP-NET, FluSurv-NET, etc.). It is also
backward-compatible at the ValueSet level: implementers binding to the
USPHPL ValueSet still see `respnet-case-report-message` via composition. The
trade-off vs. Disposition A is process complexity (two artifacts to
synchronize, ValueSet expansion across systems) and a departure from the
current USPHPL/MedMorph pattern, which co-locates these codes by design
(FHIR-38348). It also requires Public Health WG sign-off on the
"include-by-composition" pattern.

---

### Disposition C: Decline

#### Proposal

Decline the request as duplicative of an existing mechanism. RESP-NET bundles
are already disambiguated by the use-case tag added under FHIR-45328 and by
the RESP-NET-specific `Composition.type` adopted under FHIR-43493; receivers
can dispatch on those signals plus `MessageHeader.definition` /
profile URL without needing a new MessageType code.

#### Justification

Defensible only on minimalism grounds: every additional code in a shared PHA
CodeSystem is one more thing PHAs have to map and validate. If receivers can
already route reliably on profile URL + Composition.type + bundle tag, a
distinct event code is arguably redundant.

---

### Recommendation

**Recommended disposition:** A

`MessageHeader.event.code` is the canonical FHIR signal for "what kind of
message is this" and is the field PHAs are most likely to filter on at the
edge — well before they parse the bundle's Composition or PlanDefinition tags.
The USPHPL MessageType CodeSystem already exists precisely so that each
PHA-bound content IG can declare itself there (see the cancer-report-message
and eCR precedents and the alignment work in FHIR-38348). The proposed code,
display, and definition are well-formed, additive, non-breaking, and
consistent with siblings; declining (Disposition C) would force PHAs into a
second-best dispatch path that prior tickets have rejected, and Disposition B
introduces multi-artifact coordination cost that USPHPL's design has
deliberately avoided. The only adjustment needed at triage is to **reassign
the ticket from Patient Care to Public Health**, since USPHPL is the affected
artifact's owner.
