## Summary

Within the SDOHCC Coordination Platform `CapabilityStatement`
(`CapabilityStatement-SDOHCC-CoordinationPlatform`), several resources —
notably `CareTeam`, `Location`, `Questionnaire`, `QuestionnaireResponse`,
and `RelatedPerson` — are listed with **resource conformance = SHALL** and
have at least one **search parameter (e.g., `_id`) marked SHALL**, while
the interaction summary for those same resources only lists `read` and
`search-type` as **MAY** (or below `SHALL`). The reporter (Rick
Lisseveld) is asking what conforming behavior is actually expected:
how can a system claim to "SHALL support" a resource and a search
parameter on it, while at the same time the very interaction
(`search-type`) needed to exercise that search parameter is only `MAY`?
Concretely: how does an implementer demonstrate conformance to "SHALL
support `_id`" without supporting `search-type`? A follow-up comment
(2025-06-18) confirms the same inconsistency persists in the current
STU 2.3 version of the IG, and asks whether the existing ticket should
be re-versioned or a new one filed against STU 2.3.

There are no cross-referenced Jira tickets, Zulip threads, or GitHub
items associated with this ticket in the cache.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat this as a defect in
`CapabilityStatement-SDOHCC-CoordinationPlatform` and **align the
interaction summary with the SHALL-marked search parameters**. For each
resource the reporter calls out (`CareTeam`, `Location`,
`Questionnaire`, `QuestionnaireResponse`, `RelatedPerson`), set
`CapabilityStatement.rest.resource.interaction` for `search-type`
(and, where `_id` read access is implied, also `read`) to a conformance
expectation of **SHALL** (i.e., `extension[capabilitystatement-expectation]
= SHALL` on the matching `interaction` entry). Carry the change forward
into the current build (STU 2.3+) so the rendered interaction summary
matches the search-parameter expectations. No change to the SHALL
search parameter rows themselves.

#### Justification

This is the most direct reading of "Resource SHALL be supported and
`_id` search parameter SHALL be supported": there is no interoperable
way to demonstrate support for a SHALL search parameter without
supporting the `search-type` interaction. FHIR R4 §3.2.1 ties searchable
parameters to the `search-type` interaction, and the IG publisher
renders the per-resource "Interaction Summary" directly from the
`interaction.extension[expectation]` values, so the inconsistency the
reporter sees is purely a matter of those expectation extensions being
under-specified relative to the search parameter expectations. Bumping
`search-type` to SHALL on those resources removes the ambiguity and
makes the CapabilityStatement testable.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than promoting the interactions to SHALL, **demote the `_id`
search-parameter expectation (and any other search parameters whose
interaction is intentionally optional) to match the interaction
summary** — i.e., mark the `_id` `searchParam` row as `SHOULD` (or
`MAY`) on resources where `search-type` is intentionally optional —
and add narrative to the IG section preceding the CapabilityStatement
explicitly stating that, for the Coordination Platform actor, only the
interactions/search parameters carrying SHALL form the conformance
floor. Optionally, also add a `messages` block (or capability-statement
markdown narrative) explaining that "SHALL" on a search parameter is
contingent on supporting the corresponding interaction.

#### Justification

It is possible the SHALL on `_id` was inherited from a base profile or
from copy-paste rather than an intentional design decision — if the
authors actually intend `search-type` to be optional for those
resources, the search parameter cannot honestly remain SHALL.
Down-leveling the search parameter preserves the (apparently)
intentional optionality of `search-type` for `Location`,
`Questionnaire`, `QuestionnaireResponse`, and `RelatedPerson` (which
may be referenced data the platform only needs to expose by reference,
not search). It also avoids forcing implementers to stand up a full
search endpoint for resources they only need to round-trip by id.
Trade-off vs. Disposition A: it weakens the testable conformance
surface and may surprise consumers who today read "_id SHALL" as a
guarantee.

---

### Disposition C: Decline

#### Proposal

Resolve as **Not Persuasive — Already Addressed**: keep the
CapabilityStatement as written and clarify in the IG narrative (and/or
in the issue resolution) that the resource-level expectation
(`SHALL`/`SHOULD`/`MAY`) and the per-interaction expectation are
deliberately independent axes — a server "supports" a resource by
holding/exposing instances of it (e.g., via references from other
resources, includes, transactions, or messaging), and the
search/read interactions describe the additional API surface that may
or may not be present. Point implementers to FHIR R4 conformance
guidance and close.

#### Justification

CapabilityStatement does, in fact, allow resource-level support to be
asserted without all CRUD interactions being SHALL — for example,
servers that only return resources via `_include` or via Bundle
transactions still "support" the resource type. If the SDOH-CC authors
intentionally modeled the Coordination Platform that way, the
inconsistency the reporter perceives is actually working as designed
and only needs documentation. Trade-off: this leaves the `_id`-SHALL /
`search-type`-MAY combination as a real, persistent contradiction —
because `_id` is by definition a search parameter, this rationale
collapses for that specific row, which makes the "decline" disposition
hard to defend in full.

---

### Recommendation

**Recommended disposition:** A

The reporter has identified a genuine internal contradiction inside
`CapabilityStatement-SDOHCC-CoordinationPlatform`: a SHALL search
parameter on a resource whose `search-type` interaction is only MAY is
not implementable or testable as stated, and the comment thread
confirms it has survived from STU 2.2 into STU 2.3 without an
explanatory narrative. Disposition A produces the most defensible,
testable outcome: every resource called out (`CareTeam`, `Location`,
`Questionnaire`, `QuestionnaireResponse`, `RelatedPerson`) needs its
`search-type` interaction expectation raised to SHALL so the existing
SHALL search parameters become exercisable, with `read` raised
alongside where the IG already implies id-based access. Disposition B
is a reasonable fallback if the work group decides the optionality of
`search-type` was intentional for some of these resources, but in that
case the `_id` SHALL is the wrong row to keep — the work group should
explicitly choose, per resource, which axis is authoritative.
Disposition C is the weakest option here precisely because `_id` is a
search parameter; deferring to "resource support need not imply search
support" does not rescue the `_id`-specific contradiction. The work
group should also confirm whether the fix lands against STU 2.3 (the
version where the issue still reproduces, per the 2025-06-18 comment)
rather than the STU 2.2 version originally cited.
