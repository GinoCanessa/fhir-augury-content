## Summary

The reporter, working against the Ophthalmology Retinal (FHIR) implementation
guide, observes that `bodySite.value[x]` on one of the IG's Observation
profiles is declared as a polymorphic choice element but no concrete `value*`
child types (e.g., `valueCodeableConcept`, `valueReference`) are defined. In
standards FHIR, `Observation.bodySite` is `0..1 CodeableConcept` and has no
`value[x]` child — so the appearance of `bodySite.value[x]` in the IG's
StructureDefinition is almost certainly an authoring/modelling artifact in
the IG itself, not a base-spec defect.

This ticket is one of a tightly-related cluster filed by the same reporter
against the same IG on the same day:
[FHIR-33931](https://jira.hl7.org/browse/FHIR-33931) reports the same
"choice element with no choices" pattern on `value[x]` more generally
across the IG's profiles, and
[FHIR-33930](https://jira.hl7.org/browse/FHIR-33930) reports a complementary
modelling issue where `value[x]` is constrained directly to `Quantity`
instead of being narrowed via a `valueQuantity` child slice.
[FHIR-33496](https://jira.hl7.org/browse/FHIR-33496) (Craig Newman) provides
related context, asking whether the Body Structure Eye profile is the
intended target for `Observation.bodySite` references in this IG. Together
the four tickets indicate that the IG's Observation profiles need a coherent
clean-up of how `bodySite` and `value[x]` are sliced and typed.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Treat the report as a profile-authoring defect in the Ophthalmology Retinal
IG and correct the affected Observation profile so that `bodySite` is
modelled the way base FHIR defines it. Concretely: remove the spurious
`bodySite.value[x]` element from the StructureDefinition and constrain
`Observation.bodySite` directly (e.g., `0..1 CodeableConcept`, with a bound
ValueSet for ophthalmic / retinal sites and, where reference semantics are
required, an `Observation.bodySite` extension such as
`http://hl7.org/fhir/StructureDefinition/bodySite` carrying a
`Reference(BodyStructure | EyeBodyStructure)`). Regenerate the IG's
snapshots and validation artifacts so `bodySite` no longer appears to be a
polymorphic choice. Resolve in coordination with FHIR-33931 (the same
"choice with no children" pattern on `value[x]`) and FHIR-33930 (data type
mis-placed on the choice rather than on its children).

#### Justification

`Observation.bodySite` in base FHIR is a single `CodeableConcept`, not a
polymorphic element, so a `bodySite.value[x]` child in a derived profile
cannot be expressing legitimate FHIR semantics — by definition any choice
element must have at least one concrete typed child or it is invalid as a
choice. Removing the malformed element brings the IG into line with the
base spec, restores publishable snapshots, and makes downstream
validation and tooling behave correctly. Bundling the fix with the
sibling tickets (FHIR-33930 / FHIR-33931) is appropriate because they
describe the same mis-modelling pattern; fixing them together avoids
re-balloting the same profiles repeatedly.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than removing the element outright, interpret the original intent
as "the IG wants to capture a body site that may be either coded or a
reference to a `BodyStructure`/Body Structure Eye instance" and replace
the malformed `bodySite.value[x]` with the standard FHIR mechanism for
that intent: keep `Observation.bodySite` as `CodeableConcept` and add the
`bodySite` extension
(`http://hl7.org/fhir/StructureDefinition/bodySite`) on the IG's
Observation profile, typed as `Reference(BodyStructure |
EyeBodyStructure)` with cardinality `0..1`. Document, in the profile's
narrative, when implementers should populate the coded value vs. the
reference extension, and (per FHIR-33496) clarify whether the EyeBody
Structure profile is required or merely permitted.

#### Justification

This preserves the apparent author intent — that the IG needed to express
both a code and a reference to a structured body-site resource — without
introducing a non-conformant `value[x]` on `bodySite`. It uses the
mechanism FHIR has standardised for exactly this case (the `bodySite`
extension), so it works with off-the-shelf tooling, validators, and IG
publishers. It also gives a natural home for the FHIR-33496 guidance
question, which Disposition A leaves unanswered. The trade-off versus A
is more spec text and one more extension to maintain, in exchange for
not silently dropping a capability the original authors may have wanted.

---

### Disposition C: Decline

#### Proposal

Decline the change as out of scope for the base FHIR specification and
close as "Not Persuasive — IG Authoring Issue", with a note that the
Ophthalmology Retinal IG project should fix the affected
StructureDefinition in its own repository and publish a corrigendum.

#### Justification

The report describes a defect in a downstream implementation guide's
profile, not a defect in core FHIR. Patient Care does not own the IG's
content and cannot ship a fix for it through the FHIR core ballot
process. Declining at the FHIR-core / WG level and routing the issue to
the IG maintainers is defensible because the underlying base resource
(`Observation`) is already correctly defined and no change to it is
warranted by this report.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the actual mis-modelling that triggered the
ticket while preserving the apparent design intent of the IG authors —
that an Observation in this IG needs both a coded body-site and an
optional reference to a structured (eye-specific) BodyStructure
instance. Disposition A is essentially a strict subset of B (it removes
the malformed element but says nothing about the reference capability)
and would likely just generate a follow-up ticket asking how to attach
a `BodyStructure` reference; B forecloses that by adopting the standard
FHIR `bodySite` extension up front and lets the same change set close
out FHIR-33496's "is the eye profile required?" question. Disposition C
is procedurally clean but leaves a Highest-priority, Triaged ticket on
the WG's books with no actual remediation, and Patient Care is in fact
the sponsoring WG for the Ophthalmology Retinal IG, so the "not our
problem" framing does not really apply. The workgroup should also
sequence FHIR-33930 and FHIR-33931 alongside this one, since the same
edit pass to the IG's Observation profiles will resolve all three.
