## Summary

The reporter found an inconsistency in the US SDOH Clinical Care (SDOHCC) IG: one or more `CapabilityStatement` resources advertise `ServiceRequest:pertains-to-goal` as a `searchInclude`, but there is no corresponding `SearchParameter` definition for `pertains-to-goal` either in the SDOHCC IG itself or in the base FHIR R4 search registry. Because `_include` targets must reference a defined reference-typed search parameter, the current CapabilityStatement is internally inconsistent and an implementer cannot satisfy the listed `_include` against any conformant server. The ticket proposes either (a) defining the missing `SearchParameter` (and deciding whether it should also be exposed as a regular searchable parameter), or (b) removing `pertains-to-goal` from every affected `CapabilityStatement.searchInclude`. The ticket has already been triaged with resolution **Persuasive**, with a resolution note indicating the workgroup intends to follow Option 1 (define the search parameter). No cross-referenced Jira tickets, Zulip threads, or GitHub items were found via FhirAugury for this issue.

## Details

**Description:**

There are two issues identified:

1. The `searchInclude` for `ServiceRequest` lists `ServiceRequest:pertains-to-goal` as a search parameter, but the search parameter is not listed as a search parameter under the resource.
2. There is no definition for the search parameter `pertains-to-goal` in the IG or the base R4 specification search registry.

Potential Remediation:

1. If the search parameter is undefined, it should be defined.
2. Determine if the search parameter is only relevant as an `_include` or if it should also be added to the list of search parameters for `ServiceRequest` (the reporter is unsure whether a search parameter can exist only as an `_include`).

or

1. Remove the search parameter from the `searchInclude` for all affected `CapabilityStatement`s.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Adopt Option 1 from the ticket (which matches the recorded **Persuasive** resolution): define a new `SearchParameter` resource in the US SDOH Clinical Care IG with id/code `pertains-to-goal`, `base = [ServiceRequest]`, `type = reference`, `target = [Goal]`, and an `expression` along the lines of `ServiceRequest.extension('http://hl7.org/fhir/us/sdoh-clinicalcare/StructureDefinition/SDOHCC-ExtensionPertainsToGoal').value.as(Reference)` (or whatever the IG's existing "pertains to goal" extension URL actually is). Then update every SDOHCC `CapabilityStatement` that already lists `ServiceRequest:pertains-to-goal` under `searchInclude` to also list it under `searchParam` with the same canonical, so the `_include` is internally consistent. Add a short note to the IG's "Search and Includes" narrative explaining how clients use the parameter (filter on `pertains-to-goal`) versus the include (`_include=ServiceRequest:pertains-to-goal`).

#### Justification

Per FHIR R4 conformance rules, an `_include` value of `SourceType:searchParam` must resolve to a defined reference-typed search parameter on the source resource â€” the parameter being only listed under `searchInclude` is not a valid mechanism to expose it. Defining the parameter resolves the existing internal inconsistency in the SDOHCC CapabilityStatements, makes the IG implementable as advertised, and aligns with the reporter's confirmation that the defect still reproduces in STU 2.3. It also matches the workgroup's already-recorded Persuasive disposition, so this option carries the lowest risk of re-litigating the decision.

---

### Disposition B: Alternative Approach

#### Proposal

Remove `ServiceRequest:pertains-to-goal` from `CapabilityStatement.rest.resource.searchInclude` in every SDOHCC CapabilityStatement instead of defining a new search parameter. Where servers genuinely need to return the linked `Goal` alongside the `ServiceRequest`, document the use of `_include=ServiceRequest:based-on` (only if the IG is using `ServiceRequest.basedOn` to point at the Goal, which it normally is not) or recommend a follow-up read of the referenced `Goal` resource. Update the IG narrative to explain why no dedicated search parameter or include is provided.

#### Justification

This is the second remediation path the reporter proposed. It avoids minting a new IG-local search parameter (and the associated registration, FHIRPath expression, and conformance test surface), which is preferable when the linkage from `ServiceRequest` to `Goal` is already covered by an existing extension that clients can chase by reference rather than by `_include`. It also avoids any ambiguity about whether the parameter is "include-only" â€” a state the FHIR search framework does not really support. The trade-off is reduced ergonomics for clients that wanted a single-call retrieval; given there is no Zulip evidence of implementer demand for the include, that cost may be acceptable.

---

### Disposition C: Decline

#### Proposal

Decline to make any normative change in this ballot cycle and instead annotate the affected CapabilityStatements with an editorial note acknowledging that `ServiceRequest:pertains-to-goal` is currently advertised as a `searchInclude` without a backing `SearchParameter` definition, deferring the structural fix to a future SDOHCC release.

#### Justification

This option is only defensible if the workgroup has insufficient bandwidth to validate a new SearchParameter (FHIRPath expression, target types, naming) before the ballot freeze, *and* there is no implementer pressure to use the include today. Since the ticket has already been triaged as **Persuasive** with the workgroup signalling intent to act, declining would contradict that recorded position and would leave the IG demonstrably inconsistent with FHIR R4 conformance rules. It is included for completeness, not as a serious recommendation.

---

### Recommendation

**Recommended disposition:** A

The workgroup has already recorded a Persuasive resolution pointing at Option 1, the defect is reproducible in STU 2.3 (per the reporter's June 2025 comment), and a missing `SearchParameter` for an advertised `_include` is a straightforward conformance bug rather than a design debate. Defining the `pertains-to-goal` `SearchParameter` and adding it to every affected SDOHCC `CapabilityStatement.searchParam` (alongside the existing `searchInclude`) is the smallest change that makes the IG self-consistent and implementable. Disposition B (remove the include) would also resolve the conformance issue but quietly removes a feature the IG already advertises, which is more disruptive to anyone who built against earlier ballots; it should only be chosen if the workgroup concludes there is no real `Goal`-linking use case to support. Disposition C should not be selected â€” it preserves a known-broken CapabilityStatement.
