## Summary

This ticket recommends merging the two existing `Procedure` elements
`Procedure.bodySite` (a `CodeableConcept`) and `Procedure.bodyStructure`
(a `Reference(BodyStructure)`) into a single `CodeableReference(BodyStructure)`
element on `Procedure`. The request was imported from Comment 18 by Ron G.
Parker and aligns `Procedure` with the broader R6 trend of replacing paired
"code/reference" elements with a single `CodeableReference`. No other Jira
tickets are linked or cross-referenced, but the same concern was raised
independently by Brian Postlethwaite in the OO WG "R6 BodyStructure Review"
Zulip thread, where reviewers noted that other resources have already moved
to `CodeableReference` for body-site information while `Procedure` has not.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In R6 `Procedure`, replace the two existing elements
`Procedure.bodySite` (`0..* CodeableConcept`) and
`Procedure.bodyStructure` (`0..* Reference(BodyStructure)`) with a single
combined element typed `CodeableReference(BodyStructure)`. Concretely:

- Add `Procedure.bodySite : CodeableReference(BodyStructure) 0..*` (reusing
  the existing element name `bodySite` is preferred for continuity with
  pre-R6 implementations and with the analogous element on other resources
  such as `Condition.bodySite`).
- Remove `Procedure.bodyStructure` (the dedicated `Reference(BodyStructure)`
  element introduced more recently).
- Update the element definitions, short/definition text, comments, and
  bindings (the existing SNOMED CT body-structure value set continues to
  apply to the `concept` half of the `CodeableReference`).
- Carry the change through the StructureDefinition, examples, search
  parameters (the existing `body-site` SP becomes a `reference` /
  `codeable-reference` SP per FHIR R6 search-parameter handling for
  `CodeableReference`), and any narrative.
- Note the breaking change in the R6 change-log / migration guidance for
  `Procedure`.

#### Justification

This is the explicit ask of the ticket and directly addresses the
inconsistency that Brian Postlethwaite flagged in the OO "R6 BodyStructure
Review" thread: most other resources that need to point at an anatomical
location have already migrated to `CodeableReference(BodyStructure)`, and
`Procedure` is now an outlier. Collapsing the two parallel elements into one
removes the modelling ambiguity ("when do I use which?"), aligns Procedure
with the rest of Core (per the FMG `CodeableReference usage in Core`
discussion), and matches the long-standing PC/OO/II direction of moving away
from a free-floating `bodySite : CodeableConcept` toward `BodyStructure`.

### Disposition B: Alternative Approach

#### Proposal

Keep the structural change but stage it more conservatively to soften the
breaking impact on existing R5 implementers:

- Retype `Procedure.bodySite` from `CodeableConcept` to
  `CodeableReference(BodyStructure)` (the `concept` slot preserves the
  existing R5 usage path).
- Mark `Procedure.bodyStructure` as **deprecated** in R6 (rather than
  removing it), with a `replacedBy`/standards-status note pointing at the
  new `bodySite` and a clear plan to remove it in R7.
- Add an invariant or guidance discouraging populating both
  `bodySite.reference` and `bodyStructure` for the same row.

#### Justification

This delivers the same target shape (a single `CodeableReference` for body
site on `Procedure`) without immediately invalidating instances or profiles
that already use `Procedure.bodyStructure`. Given that `Procedure` is one of
the most widely implemented resources, a one-release deprecation window is
consistent with how FHIR has historically handled element retirements, and
it gives jurisdictional IGs (US Core, IPS, AU Base, etc.) time to migrate
profiles and mappings. The downside is two releases of mild duplication and
the need for a clear "use `bodySite`, not `bodyStructure`" message.

### Disposition C: Decline

#### Proposal

Do not adopt the change for R6. Retain `Procedure.bodySite`
(`CodeableConcept`) and `Procedure.bodyStructure` (`Reference(BodyStructure)`)
as-is, on the basis that:

- The two elements convey distinct, complementary information (a free-text /
  coded site vs. a structured `BodyStructure` instance with laterality,
  qualifiers, and image references), and merging them into a
  `CodeableReference` may obscure that distinction for implementers who
  consciously chose one over the other.
- The change is breaking against R5 instances and against any pre-existing
  R6 ballot content.

#### Justification

A defensible (if minority) position. Some implementers have argued that
keeping a dedicated `Reference(BodyStructure)` makes the strongly-typed
relationship discoverable at the schema level, and that `CodeableReference`
hides whether servers support `BodyStructure` retrieval. If PC concludes
the consistency win does not outweigh the migration cost mid-cycle, the
ticket can be deferred to R7 and re-evaluated alongside other
`CodeableReference` harmonisation work.

### Recommendation

**Recommended disposition:** A

The OO "R6 BodyStructure Review" thread shows this is already a
well-understood inconsistency that PC/OO are expecting to surface as ballot
comments, and the FMG `CodeableReference usage in Core` discussion confirms
`CodeableReference(BodyStructure)` is the established R6 pattern for
anatomical-site references across Core. Because R6 is already a
breaking-change window relative to R5, doing the full merge now (Disposition
A) avoids leaving `Procedure` as a permanent outlier and avoids carrying a
deprecated `bodyStructure` element forward to R7. Disposition B is a
reasonable fallback if PC hears strong implementer pushback during ballot;
Disposition C should only be selected if PC concludes that
`bodySite : CodeableConcept` and `bodyStructure : Reference(BodyStructure)`
genuinely model different concepts on `Procedure` — the available evidence
suggests they do not.
