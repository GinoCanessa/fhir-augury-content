## Summary

Stephen Chu proposes adding a new element, `Flag.criticality`, to the FHIR
Core `Flag` resource so that a flag (a warning or notification displayed
to clinicians or other care participants) can carry an explicit
indication of how serious the warning is. The reporter notes that
because a `Flag` may concern a patient, clinician, organization, or even
a population (e.g., an Ebola outbreak associated with a `Location`), the
recipient benefits from knowing whether the warning represents a
"high" risk, a "low" risk, or an "unknown" risk. The suggested value
set has three codes — `high`, `low`, and `unknown` — mirroring the
intent (though not the exact codes) of the existing
`AllergyIntolerance.criticality` element. The ticket does not specify
cardinality, binding strength, or whether a new value set should be
defined or an existing one reused. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were found by FhirAugury for this
ticket; the only cross-reference is the self-mention of the proposed
element name `Flag.criticality`.

## Details

**Description:**

> Flag - scope statements include: "A flag is a warning or notification
> of some sort presented to the user - who may be a clinician or some
> other person involved in patient care. It usually represents
> something of sufficient significance to warrant a special display of
> some sort - rather than just a note in the resource."
> Examples include: An outbreak of Ebola in a particular region
> (subject=Location) so that all patients from that region have a
> higher risk of having that condition.
>
> As the "warning or notification" may have impact on clinician,
> patient, some other person involved in patient care, or perhaps the
> healthcare organization or broader, it is useful to consider adding:
> `Flag.criticality`
>
> Suggested criticality value set:
> - **high** — indicates that the warning or notification may represent
>   very high or very serious risks to patient, or healthcare
>   personnel, or broader
> - **low** — indicates that the warning or notification may represent
>   low risks to patient, or healthcare personnel, or broader
> - **unknown** — risk implications of the warning or notification is
>   not assessed or not known

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new element `Flag.criticality` to the `Flag` resource in FHIR
Core with cardinality `0..1`, type `code`, and a **required** binding
to a new value set `http://hl7.org/fhir/ValueSet/flag-criticality`
defined with the three codes proposed by the reporter:

- `high` — the warning may represent very high or very serious risks
  to patient, healthcare personnel, the organization, or broader.
- `low` — the warning may represent low risks to patient, healthcare
  personnel, the organization, or broader.
- `unknown` — risk implications of the warning are not assessed or not
  known.

The element would be optional but, when present, would constrain the
display layer to a fixed three-level severity ladder. Companion
guidance under the Flag resource notes section would explain the
distinction between `Flag.status` (active/inactive/entered-in-error),
`Flag.category` (clinical/administrative/etc.), and the new
`Flag.criticality` (severity of the *risk being flagged*).

#### Justification

The reporter's underlying observation is sound: today a `Flag` carries
no first-class severity signal, so consuming systems cannot
deterministically render a high-risk Ebola-exposure flag differently
from a routine administrative reminder. A required binding to a small,
fixed code set (mirroring the precedent set by
`AllergyIntolerance.criticality`, which also uses `high`/`low`/`unable-to-assess`)
gives implementers a reliable hook for visual prioritisation without
forcing them to parse `Flag.code` text. Cardinality `0..1` preserves
backward compatibility — existing instances remain valid — and the
required binding ensures interoperability of the severity signal where
it is supplied.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add `Flag.criticality` to the base `Flag` resource. Instead:

1. Define a standard extension
   `http://hl7.org/fhir/StructureDefinition/flag-criticality` of type
   `code`, with the same proposed value set
   (`flag-criticality`: `high` | `low` | `unknown`) and an
   **extensible** binding.
2. Document the extension in the Flag resource page's "Extensions"
   section and in the Patient Care WG's profile guidance.
3. Re-evaluate promotion to a core element after at least one ballot
   cycle of real-world use, contingent on observed uptake and
   implementer feedback.

Optionally, point implementers at `AllergyIntolerance.criticality` and
`DetectedIssue.severity` as the existing severity-style codings they
should align with conceptually.

#### Justification

`Flag` is a relatively lightweight "notification" resource and the
WG has historically been conservative about widening its core schema.
The use cases for severity on a flag are real but heterogeneous —
some implementers will want a 3-level scale, others a 5-level scale,
others a numeric score, and several will want to reuse existing
severity terminologies (SNOMED CT, IHE, organisation-specific). A
standard extension lets all of those communities adopt the
`high`/`low`/`unknown` ladder where it fits without baking a single
opinion into the core resource. Promotion to a first-class element
later remains cheap (extensions are routinely promoted at major
versions); demotion is not. This path also avoids the FMM/maturity
churn that adding a required-bound element to an established resource
would trigger.

---

### Disposition C: Decline

#### Proposal

Decline the request. Document the rationale on the ticket and close
as "Persuasive with Mod" only if the WG is willing to add a single
sentence to the Flag resource notes pointing implementers at
`Flag.code` (with appropriate concept selection) and existing
extensions on related resources for severity expression. Otherwise
close as "Not Persuasive — already addressed by existing mechanism".

Specific rationale: the severity / criticality of the situation a flag
describes is intrinsic to the *concept* the flag carries, and that
concept is already conveyed by `Flag.code` (a `CodeableConcept`).
Implementers who need to communicate "high-risk Ebola exposure" should
encode that severity into the chosen flag concept rather than via a
parallel orthogonal axis, which risks contradiction
("`Flag.code` = high-risk-Ebola, `Flag.criticality` = low") and
duplicates information that terminology already carries.

#### Justification

Adding a new core element imposes a permanent cost on every
implementer of the resource — schema changes, validator updates,
mapping work — and the proposal does not yet include enough use-case
evidence (no Zulip discussion, no linked use cases, no profile
demonstrating need) to justify that cost. The
`AllergyIntolerance.criticality` precedent is not directly comparable:
an allergy is itself a long-lived clinical assertion whose criticality
is computed independently of the substance, whereas a `Flag`'s
severity is generally a property of the underlying coded concept.

---

### Recommendation

**Recommended disposition:** B (Alternative Approach — standard
extension first, promote later if warranted).

The reporter's observation is legitimate and the use case (varied risk
levels for varied notification audiences) is realistic, so an outright
decline (C) understates the value. However, accepting the change as a
required-bound core element on the first pass (A) commits the WG to a
specific 3-level severity ladder before there is any cross-referenced
implementer or Zulip discussion to confirm that ladder is the right
one — and this ticket has neither linked Jira tickets nor Zulip
threads to draw on. A standard extension with the same proposed value
set captures the value immediately, lets implementations exercise the
codes, and preserves a clean promotion path at the next major version
if uptake confirms the need. Patient Care should explicitly invite
implementer feedback when balloting the extension and revisit
promotion to a core element on that evidence.
