## Summary

The ticket questions a profiling decision in the FHIR International Patient
Summary (IPS) implementation guide for the Results Radiology Observation
profile. As currently profiled, all radiology measurement values are required
to live on `Observation.component.value[x]` rather than on the top-level
`Observation.value[x]`. The reporter (Elliot Silver) is not clear on the
rationale — is the assumption that each radiology study yields a single
Observation whose components carry the individual measurements/findings, or
is it intended to provide higher-level grouping context? The reporter also
flags that the bound code system (Results Radiology Measurement Observation
GPS-DICOM value set) communicates the kind of measurement (length, width,
volume, etc.) but not the anatomical/lesion subject of the measurement, so
implementers cannot say "tumour #2 measures 30 cu. mm" or distinguish "length
of femur" from "length of metacarpal." The ticket asks that, unless the
component-only design is justified and explained for implementers, the
`component.value[x]` profiling be raised up to `value[x]` so simple,
single-measurement radiology results can also be represented. Discussion with
Imaging Integration WG is suggested. There are no other Jira tickets
cross-referenced to FHIR-32811 in the data store.

## Details

**Description:**

It isn't immediately clear to me why all radiology observations are pushed
into Observation.component.value[x] rather than being allowed in
Observation.value[x]. Is the intent that there is only a single Observation
from each study, and the components contain all the measurements and
findings? Is this approach needed in order to provide some higher-level
context (all the following are from a previous study...)?

Unless clarified for implementers, the component.value[x] profiling should
be raised up to value[x].

Suggest discussion with II WG.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the IPS Results Radiology Observation profile (`Observation` profile in
the HL7 IPS IG, `StructureDefinition/Observation-results-radiology-uv-ips`),
relax the constraint that pushes the measured value into
`Observation.component.value[x]` so the profile **also permits the value to
appear at the top-level `Observation.value[x]`**. Concretely:

- Remove the `0..0` (or otherwise prohibitive) restriction on
  `Observation.value[x]` and re-open it to the same data types currently
  allowed on `Observation.component.value[x]` (Quantity, CodeableConcept,
  string, etc., as appropriate for radiology measurements/findings).
- Lower the cardinality of `Observation.component` from "required for the
  measurement" to optional (e.g., `0..*`) so a single-measurement radiology
  observation does not need a component slice just to carry its value.
- Add narrative guidance to the profile page explaining when to use the
  top-level value (a single measurement/finding per Observation) versus
  components (a panel of measurements grouped under one Observation, e.g.,
  measurements from a single study or a single lesion).
- Keep the existing GPS-DICOM value-set binding available for both
  `Observation.code` (top-level) and `Observation.component.code` so
  implementers can use the same vocabulary either way.

#### Justification

This restores parity with the base FHIR Observation pattern and with the
broader Results Observation (IPS) profile, where simple single-value
observations live on `value[x]` and only multi-component panels use
`component`. Forcing every radiology finding through `component` adds
ceremony that has no clinical payoff for the common "one number, one
unit" case (e.g., "abdominal aorta diameter = 1.8 cm"). It also resolves
the reporter's secondary concern: when a profile only allows components,
implementers must invent a parent code for the top-level Observation that
carries no measurement, which is awkward and inconsistent with how other
IPS Results profiles behave. Patient Care has already procedurally agreed
to revisit this post-publication (Lynn Laakso, 2022-11-22), and the
reporter (Elliot Silver) is the author of multiple IPS clarification
tickets, so accepting here aligns with the work group's stated intent.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the component-centric data shape for the Results Radiology
Observation profile, but address the underlying interoperability
shortcoming by:

1. Adding **explicit profile narrative** that documents the
   "one Observation per study, components for each measurement/finding"
   design intent, so implementers stop questioning it.
2. **Permitting `Observation.bodySite` and/or `Observation.component.focus`
   on each component** (or recommending the use of `Observation.focus` →
   `BodyStructure`) to carry the *subject* of the measurement (e.g., "left
   femur," "tumour #2"), which is the gap Elliot Silver actually identified
   in his 2021-05-30 follow-up comment.
3. Optionally, define a **lightweight extension** on
   `Observation.component` that carries a structured anatomical/lesion
   reference for the case where a coded `bodySite` is too coarse.
4. Coordinate with Imaging Integration WG (as the original ticket
   suggested) to ensure alignment with DICOM SR-derived patterns already
   in use elsewhere in the IG.

This leaves the value on `component.value[x]` but makes single-value usage
viable by allowing a degenerate case of one component, and makes
multi-measurement usage usable by adding a way to identify *what* each
measurement is of.

#### Justification

The component-centric pattern was deliberately chosen to mirror DICOM
Structured Report measurement groups, where one report yields one
Observation with many measurement rows. Keeping that shape preserves
round-trippability with imaging systems and avoids a breaking change for
implementers who have already built to the current profile. The reporter's
strongest substantive objection is not actually about value-vs-component;
it is that the profile cannot say *what was measured* — and that gap is
better closed with `bodySite` / `focus` than by relaxing `value[x]`. This
disposition addresses the real interoperability issue without abandoning
the design rationale, and it teaches future readers via the narrative
fix.

---

### Disposition C: Decline

#### Proposal

Do not change the profile constraint. Mark the ticket "Not Persuasive"
(or "Considered for Future Use," consistent with the 2022 Sept WGM block
vote) on the basis that:

- The component-centric shape is a deliberate alignment with DICOM SR
  measurement groups and is shared with adjacent IPS radiology profiles.
- Implementers needing to send a single measurement can populate a single
  `component` element; the cost is one extra level of nesting, not lost
  expressivity.
- The "what was measured" gap raised in the reporter's follow-up is a
  separate concern that should be raised as its own ticket against the
  GPS-DICOM-derived value set / Imaging Integration WG, not solved by
  changing this cardinality.

#### Justification

This profile has shipped in published IPS releases and has been re-voted
by Patient Care (block-vote-2, Sept 2022 WGM, 14-0-16) under the
Considered-for-Future-Use disposition. Loosening `value[x]` after
publication is a substantive change that risks fragmenting implementer
behaviour (some senders putting the value on top-level, some on
component, with the same `code`), which makes downstream rendering and
validation harder rather than easier. Declining preserves a single,
predictable shape, and the underlying clarity issue can be addressed via
narrative-only edits without touching the StructureDefinition.

---

### Recommendation

**Recommended disposition:** B

The work group already agreed in 2022 to revisit this post-STU rather
than slam it shut (Laakso re-open comment, 2022-11-22), so a flat decline
(C) under-serves the reporter and the prior commitment. A pure relaxation
(A) does fix the immediate question but ignores the substantive gap the
reporter raised in his follow-up — namely that even with `value[x]`
unlocked, the profile still cannot say *what* was measured — and
introduces two valid shapes for the same data, which IPS implementers
will have to choose between. Disposition B keeps the published shape
stable, addresses the real interoperability shortcoming (`bodySite` /
`focus` on the component, plus explicit narrative on the
one-Observation-per-study intent), and explicitly loops in Imaging
Integration WG as the original ticket suggested. That gives Patient Care
something concrete to take to a joint discussion without committing
to a breaking-style profile change before that conversation has happened.
