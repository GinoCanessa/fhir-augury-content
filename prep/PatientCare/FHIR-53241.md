## Summary

The reporter notes an inconsistency in FHIR Core for the `Condition.clinicalStatus`
value set (`http://hl7.org/fhir/ValueSet/condition-clinical`). The value set's
**definition text** describes itself as the *"Preferred value set for Condition
Clinical Status,"* yet on `Condition.clinicalStatus` the value set is bound with
**`required`** strength rather than `preferred`. The reporter believes the wording
is a leftover artifact from a prior change (or a typo) and asks the work group to
reconcile the two. No alternate use cases or supporting examples are supplied; the
ticket is purely an editorial/binding-coherence issue. There are no linked Jira
tickets and no GitHub PRs, issues, or commits cross-referenced to this ticket.

## Details

**Description:**

> Definition of the Valueset:
>
> Preferred value set for Condition Clinical Status.
>
> However it is only used as a required (not preferred) binding.
>
> This seems inconsistent, possibly left from previous changes or possibly a typo.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Accept the ticket as an editorial fix in `HL7/fhir`. Update the
`condition-clinical` ValueSet's `description` (and any companion narrative on
`valueset-condition-clinical.html`) so it no longer characterizes itself as a
*"Preferred value set."* Replace the wording with text that matches the actual
binding strength on `Condition.clinicalStatus`, e.g. *"Required value set for
Condition Clinical Status — codes describing the clinical status of a
condition."* Do not change the binding strength on `Condition.clinicalStatus`
itself; the change is purely to bring the ValueSet's self-description in line
with how Condition uses it.

#### Justification

This matches exactly what the reporter and the only Zulip respondent
(Michelle Miller, Patient Care WG co-chair) treated as the underlying defect:
text drift between a ValueSet's description and the binding strength used on
the only resource element that consumes it. It is the lowest-risk fix
(narrative-only, no schema or instance impact), preserves the long-standing
`required` binding that implementers already conform to, and removes a
genuine source of reader confusion in the published spec.

---

### Disposition B: Alternative Approach

#### Proposal

Treat the wording mismatch as a *symptom* and re-examine the binding intent.
Either (i) keep `Condition.clinicalStatus`'s binding `required` and apply
Disposition A's text fix, **plus** add a one-line normative note to the
ValueSet page explaining *why* it is required (so future readers do not
re-introduce the "preferred" wording), or (ii) if Patient Care still believes
the original *intent* was a `preferred` binding (the wording having been
correct and the binding having drifted), open a companion change to relax
`Condition.clinicalStatus` from `required` to `preferred` in the next
backwards-compatible release window — with a concrete migration note for
implementers and a check against US Core / IPS / international Patient
profiles that depend on the current strength.

#### Justification

The reporter explicitly hedges ("possibly left from previous changes or
possibly a typo"), so it is worth letting the work group confirm which side
of the inconsistency is correct rather than assuming the binding is canonical
by default. Path (i) is essentially Disposition A with a forward-looking
guard; path (ii) is harder — relaxing a long-standing `required` binding on a
widely implemented status element would be a breaking change for downstream
profiles and should not be done without an explicit Patient Care decision and
HTA cross-check.

---

### Disposition C: Decline

#### Proposal

Decline the change as a non-issue: the `description` element on a ValueSet is
narrative metadata about the *value set itself*, not about how any particular
resource binds to it; binding strength is conveyed by the
`ElementDefinition.binding.strength` of `Condition.clinicalStatus`, which is
correctly set to `required`. Close as "Not Persuasive" with a note pointing
the reporter to the binding element as the authoritative source of strength.

#### Justification

This position is technically defensible — `ValueSet.description` is free
text about the value set's purpose and content, and FHIR formally derives
binding strength from the ElementDefinition, not from prose. Implementers
and validators consult the binding element, not the ValueSet description, so
no conformance behaviour changes either way.

---

### Recommendation

**Recommended disposition: A**

The ticket is a low-cost, low-risk editorial fix and the Zulip thread already
shows alignment between the reporter and a Patient Care co-chair that the
"Preferred" wording is unintentional. Disposition C is technically correct
that binding strength lives on the ElementDefinition, but published spec text
that contradicts the binding it is paired with is a genuine reader hazard and
exactly the kind of inconsistency the work group should clean up rather than
defend on a technicality. Disposition B's path (ii) — relaxing the binding
itself — is out of proportion to what was reported and would be a breaking
change for every profile and implementation that depends on the current
`required` strength; if Patient Care wants to revisit binding strength, that
should be a separately scoped ticket with use cases and a profile impact
review, not bundled into this editorial correction. Apply Disposition A;
optionally fold in B (i)'s one-line "why this is required" sentence to
prevent the wording from drifting again.
