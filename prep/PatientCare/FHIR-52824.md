## Summary

The reporter (Robert Hausam) asks the Patient Care work group to scrub the
example Bundles published in the International Patient Summary (FHIR) IG
to remove uses of the now-deprecated `other` code from the
`organization-type` value set on `Organization.type`. Because no other code
in `organization-type` is an obvious replacement for `other`, the proposed
correction is to delete the offending `Organization.type` element entirely
from the Organization resources contained in the IPS example Bundles
rather than substitute another code. The reporter has noted in a comment
that the change has already been pre-applied (git commit `db0b53c7`),
indicating this is a clean-up technical correction expected to be
ratified at review rather than re-debated. There are no other Jira
tickets cross-referenced from FHIR-52824.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Apply the change exactly as described: in every example Bundle in the
International Patient Summary (FHIR) IG that contains an `Organization`
resource carrying `Organization.type` with the deprecated
`organization-type#other` code, delete the `type` element entirely
(do not substitute another code). Confirm the pre-applied git commit
`db0b53c7` (referenced in the reporter's comment) covers all such
example Bundles and contains no collateral edits, then close the
ticket as Persuasive — Applied. No structural, terminology, or
constraint change to IPS profiles is required; this is purely an
example-content correction.

#### Justification

The `other` code in the HL7-maintained `organization-type` value set
has been deprecated, so continuing to ship it in normative-track
examples teaches implementers to use a deprecated code, and would
trigger validator warnings for downstream IPS implementations.
Removing the `type` element is consistent with FHIR's general guidance
that `Organization.type` is `0..*` and should be omitted when no
appropriate code exists, rather than carrying a placeholder. Because
the reporter has already prepared the commit, accepting as requested
is the lowest-risk path and matches the standard handling of
"Technical Correction" tickets.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than silently dropping `Organization.type`, replace each
deprecated `organization-type#other` coding with an explicit
`data-absent-reason` extension on `Organization.type` (e.g.,
`unknown` or `not-applicable`), or with a locally-defined IPS example
code (e.g., a coding from a more specific value set such as
`http://terminology.hl7.org/CodeSystem/v2-0331` Provider Role where
the example organization plays a clinical role). This preserves the
modeling intent of the original examples — that the example author
deliberately tagged the organization's role — while moving off the
deprecated code.

#### Justification

Deletion loses information: a reader of the post-fix examples cannot
tell whether the omission means "type is unknown" or "type is
deliberately unspecified". Using `data-absent-reason` makes that
distinction explicit and gives implementers a worked example of how
to handle the same situation in their own data when they previously
would have used `other`. The trade-off is more change surface (the
examples grow rather than shrink) and a small risk that the chosen
replacement coding raises questions of its own at review.

---

### Disposition C: Decline

#### Proposal

Decline as Not Persuasive — Out Of Scope For This Cycle. The IPS
examples remain syntactically valid; deprecated codes are still
resolvable in the value set and produce, at worst, a validator
warning. Defer the change to the next IPS publication cycle and
batch it with any other example refresh work, rather than producing a
technical-correction publication for an examples-only change.

#### Justification

A defensible argument can be made that this is cosmetic: it changes
no profiles, no constraints, no narrative, and no conformance
requirements, and it does not affect a single production IPS
instance. If the work group is rate-limiting technical-correction
republications of IPS, bundling this with the next planned refresh is
cheaper than a one-off correction. This is the weakest disposition
because the fix is already pre-applied and the cost of accepting it
is essentially zero.

---

### Recommendation

**Recommended disposition:** A

The change is mechanical, narrowly scoped to example content, and the
reporter has already prepared the commit (`db0b53c7`). No profile,
constraint, terminology binding, or normative text is affected, so
the disposition risk is bounded to "did we miss any other example
Bundle containing the deprecated code?" — a question the work group
can answer quickly by reviewing the diff. Disposition B (replace
with `data-absent-reason`) is principled but introduces design
debate the ticket does not call for and expands the change surface
unnecessarily. Disposition C (decline / defer) leaves a known
deprecated code in published examples for another cycle, which
contradicts the purpose of having a "Technical Correction" track.
Recommend the work group confirm the pre-applied commit covers all
affected example Bundles and resolve as Persuasive — Applied.
