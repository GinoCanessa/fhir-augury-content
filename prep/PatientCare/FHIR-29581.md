## Summary

The reporter (Craig Newman) is asking the PACIO Functional Status work group
to clarify the design intent behind two related constraints on
`Observation.value` in the **FunctionalStatus** and **BundledFunctionalStatus**
profiles. Specifically: (1) both profiles constrain `Observation.value[x]` to
`CodeableConcept`, but no value set is bound, leaving implementers without
guidance on what codes are appropriate to send; and (2) it is unclear what
`Observation.value` is supposed to mean on a *bundled* (panel-style) instance
when the individual member observations — which carry the actual scored values
— live in `Observation.hasMember`. The request is for the IG to clarify the
usage of `.value` on each profile (and, by implication, either bind a value
set, drop/relax the constraint, or document the semantics on the bundled
profile). No related Jira tickets, Zulip threads, or GitHub items were
surfaced for this ticket; the only structured cross-reference is to the FHIR
artifact `Observation.value` itself.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update both the **FunctionalStatus** and **BundledFunctionalStatus** profiles
in the US PACIO Functional Status IG to clarify `Observation.value[x]`:

1. On **FunctionalStatus** (the individual-assessment profile): keep the
   `CodeableConcept` type slice but add an explicit `extensible` (or
   `preferred`) binding to a value set of allowable functional-status answer
   codes (e.g., a PACIO-curated set drawn from LOINC answer lists / SNOMED CT
   for ADL/IADL responses). If no consensus value set is yet ready, document
   the *intent* of the type constraint in the element definition (e.g., "the
   coded result of the assessment item, e.g., LOINC answer-list code") and
   flag the binding as `example` until a curated set is published.
2. On **BundledFunctionalStatus** (the panel/grouping profile): explicitly
   prohibit `Observation.value[x]` (`max = 0`) and add a definition note
   stating that scored results live on the member observations referenced by
   `Observation.hasMember`, mirroring the long-standing FHIR pattern for
   panel observations (e.g., vital-signs panel, blood-pressure panel).

#### Justification

The reporter raises two distinct issues, and they have different "right"
answers under existing FHIR conventions. For the bundled profile, the FHIR
core guidance for panel-style Observations is already that the panel header
should not carry `value[x]`; making this explicit in PACIO is a low-risk
clarification that aligns with `vitalspanel` and similar profiles. For the
individual profile, leaving a `CodeableConcept` type constraint with no
binding is the situation the ticket explicitly calls out as confusing —
either document the intent or bind a value set. The 2021 deferral comment
notes that an HL7 group was working on a Functional Status value set; if that
work has produced (or can produce) even a starter set, an `extensible` or
`preferred` binding fulfils the original intent of deferring.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than binding a value set on `FunctionalStatus.value[x]`, *relax* the
type constraint to `CodeableConcept | Quantity | integer` (the union actually
used by real functional-status instruments — e.g., Barthel Index uses an
integer score, FIM uses scored ordinals, while many ADL items are coded
answers). On **BundledFunctionalStatus**, do the same `value[x]` prohibition
proposed in Disposition A. In the IG narrative, add a "Choosing the right
value type" section that maps common instruments (Barthel, FIM, Katz ADL,
Lawton IADL, etc.) to the recommended `value[x]` choice and pointer to the
relevant LOINC code on `Observation.code`.

#### Justification

The original `CodeableConcept`-only constraint is the deeper problem the
ticket exposes: it was never well-justified, and the absence of a binding
suggests the authors were uncertain about the answer space. Functional-status
instruments in real use are heterogeneous — some are scored numerically, some
are coded answer lists, some are free-text observations rolled up to
ordinals. A single typed constraint without a binding penalises implementers
who carry numeric scores. Relaxing the type and explaining the choice in
narrative trades a bit more flexibility for substantially clearer
implementer guidance, and is more durable than waiting for a single value
set to cover every instrument PACIO will ever profile.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive – Already Addressed**, on the grounds
that (a) the `CodeableConcept` constraint was a deliberate scoping choice
for STU1 to keep functional-status answers code-based, (b) the bundled
profile's intended use of `value` is implicitly covered by the FHIR core
panel pattern (header observations should not carry `value`), and (c) any
value-set binding work is dependent on the external HL7 effort cited in the
2021 deferral comment, which has not yet produced a publishable artifact.

#### Justification

A defensible decline rests on the original 2021 vote (6 – 0 – 1 to defer
pending external value-set work) and on the fact that the IG is already
published as STU. Re-opening core typing decisions on `Observation.value[x]`
between STU cycles risks breaking existing implementers without a clearly
better answer in hand. If the external work group has not delivered a value
set, the IG cannot bind one, and editorial-only clarifications can be
captured as a documentation-only ticket against the next minor publication
rather than as a profile change.

---

### Recommendation

**Recommended disposition:** A

The ticket is fundamentally a *clarity* request, not a re-litigation of the
profile design, and Disposition A answers it directly with the smallest
defensible change: make the bundled profile's "no `value` on the header"
behaviour explicit (which is just codifying standard FHIR panel semantics),
and either bind or at least *describe* the intended answer space on the
individual profile. Disposition B has merit if the work group is willing to
revisit the type constraint, but that is a larger change with backward-
compatibility implications and should be split into its own ticket.
Disposition C is hard to defend now that the ticket has been explicitly re-
opened post-STU1 with the comment "re-opening for consideration"; the
original deferral rationale (waiting on an external value set) does not
preclude the editorial fixes described in A. Recommend the Patient Care /
PACIO group adopt A as the working disposition and use the discussion to
decide whether the value-set binding is `example`, `preferred`, or
`extensible` based on the current state of the external HL7 work.
