## Summary

The reporter (Daniel Vreeman) asks the PACIO Functional Status IG authoring
team to publish a forward-looking "future roadmap" alongside the guide. The
current scope of the IG is perceived as unclear, and the request is that a
roadmap section explicitly describe what is *not* covered today and what the
project intends to address in subsequent releases — for example Instrumental
Activities of Daily Living (IADLs), communication, sensory impairments, and
other post-acute care (PAC) domains. The intent is to help implementers and
reviewers calibrate expectations about the IG's coverage and the project's
trajectory.

The ticket has a noteworthy lifecycle: it was previously resolved as
"Considered for Future Use" on 2021-06-28 (Wan/Lyle, 6-0-0) on the basis
that emerging FHIR work on run-time filtered value sets would likely subsume
separate cognitive- / functional-status profiles, and was then **re-opened
post-STU1** by Lynn Laakso on 2021-11-04 for renewed consideration. No
related Jira tickets, Zulip threads, or GitHub items were surfaced by the
cross-reference index, so this ticket appears to stand on its own in the
current corpus.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a dedicated **"Future Roadmap"** narrative page to the US PACIO
Functional Status IG that:

1. States the **explicit in-scope content** of the current release
   (resources, profiles, value sets, and care domains actually covered
   by STU1).
2. States the **explicit out-of-scope content** for the current release
   so readers do not assume coverage that is not present.
3. Lists **anticipated future scope items**, at minimum: Instrumental
   ADLs (IADLs), communication, sensory impairments, and any other PAC
   domains the project intends to take up next, each with a short
   paragraph on intent and dependencies (e.g., "pending availability of
   run-time filtered value sets in core FHIR").
4. Adds a clear disclaimer that the roadmap is non-normative and
   subject to change between ballot cycles.

Mechanically, this is an IG narrative change: a new page (e.g.,
`input/pagecontent/future-roadmap.md`) registered in the IG's
`ImplementationGuide` resource under `definition.page`, with a link
from the IG home page and the scope/introduction page. No FHIR
artifacts (resources, profiles, value sets) need to change.

#### Justification

The request is purely about narrative clarity, which is one of the
cheapest and lowest-risk changes an IG can make. A roadmap page is a
well-established pattern in HL7 IGs (e.g., US Core, Da Vinci IGs all
publish roadmap or future-considerations sections), so it aligns with
existing community practice. The 2021-06-28 disposition note
("Considered for Future Use") explicitly identified concrete future
directions (run-time filtered value sets and their effect on PAC
profiles); capturing that reasoning *in the IG itself* — rather than
only in Jira — is exactly what the reporter is asking for and would
prevent implementers from re-litigating scope questions in tracker
tickets and Zulip threads later. Lynn Laakso's re-opening of the
ticket post-STU1 signals that the workgroup itself sees value in
revisiting this for the next publication cycle.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing a standalone "Future Roadmap" page, **expand
the existing Scope / Introduction section** of the IG to:

- Tighten the in-scope statement (what STU1 covers and the population
  it targets).
- Add an explicit "Out of scope for this release" subsection enumerating
  domains intentionally deferred (IADLs, communication, sensory
  impairments, etc.).
- Add a short, *non-normative* "Future considerations" paragraph that
  cross-links to the PACIO project's external project plan or
  Confluence page, where the more volatile roadmap detail lives and can
  be updated between ballots without re-publishing the IG.

This keeps roadmap-style content in the IG narrative in a constrained,
maintainable form, while delegating the "what's coming next" detail to
project artifacts that update on a faster cadence than the IG.

#### Justification

A roadmap embedded directly in a published IG can become misleading
quickly: if the listed future work shifts (which is likely for an
incremental project that depends on still-evolving core FHIR
capabilities such as run-time filtered value sets), the IG ends up
publishing stale forward-looking statements that implementers may treat
as commitments. Anchoring the IG narrative on *scope clarity* (which
was the reporter's underlying complaint — "the scope of the present IG
is unclear") and pointing to the PACIO project plan for the
forward-looking detail addresses the substantive issue without binding
the IG's publication cadence to roadmap volatility. It also avoids the
implication that future roadmap items are normative balloted content.

---

### Disposition C: Decline

#### Proposal

Decline the request as **out of scope for normative IG content**. Note
in the resolution that scope clarification (separate from a future
roadmap) is reasonable and may be addressed by other tickets, but that
forward-looking roadmap content belongs on PACIO project pages
(Confluence / project site), not in the published IG.

#### Justification

The original 2021-06-28 disposition (vote 6-0-0) implicitly took this
line by deferring the work pending core FHIR developments. Roadmaps
tend to date faster than IG release cycles, and there is a real risk
that implementers treat a roadmap section as a forward commitment of
the workgroup. Declining keeps the IG focused on currently-balloted
artifacts and avoids creating a maintenance burden (refreshing the
roadmap each ballot) that has not been resourced. The cross-reference
search returned no companion tickets or community discussion since
re-opening, which weakens the case that this remains an active
community ask.

---

### Recommendation

**Recommended disposition:** B

Disposition B addresses the reporter's underlying concern — that the
IG's scope is unclear — while sidestepping the well-known failure mode
of a roadmap section that becomes stale between ballots. The reporter's
own framing pairs "recommendations about defining scope" with "future
roadmap" as two halves of the same problem, and tightening the
Scope / Introduction with an explicit out-of-scope list captures most
of the value at the lowest maintenance cost. Cross-linking to the PACIO
project plan preserves the spirit of "what's coming" without making the
IG itself the source of truth for evolving plans, which is especially
important given the 2021 rationale that core FHIR work on run-time
filtered value sets may materially reshape PACIO's trajectory.
Disposition A remains a defensible second choice if the workgroup
wants the roadmap visible inside the IG; Disposition C is hard to
justify given that Lynn Laakso's re-opening of the ticket post-STU1
signals continued workgroup interest.
