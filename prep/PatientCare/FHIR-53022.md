## Summary

The reporter notes that the Extensions Pack entry for the **Procedure target
body structure** extension (which lets a `Procedure` reference a
`BodyStructure` resource rather than just a coded `Procedure.bodySite`) does
not explain *when* an implementer should reach for it instead of the existing
core elements `Procedure.bodySite` (CodeableConcept), `ServiceRequest.bodySite`
(CodeableConcept), or `ServiceRequest.bodyStructure` (Reference(BodyStructure),
added in R5). The request is for the extension's description / definition /
"Notes to Implementers" to spell out the boundary between these four ways of
expressing where on the body a procedure was performed or is requested,
including (a) when the richer `BodyStructure` reference is preferred over a
plain code, and (b) what the extension means for a `Procedure` once R6 is in
play. There are no explicitly linked or cross-referenced Jira tickets, but
sibling Patient Care extensions-pack tickets discussed in the same FMG round
(e.g. FHIR-52938, FHIR-52939, FHIR-52955, FHIR-52977, and the OO siblings
FHIR-53023 / FHIR-53024 covering the equivalent ServiceRequest /
DiagnosticReport extensions) frame this as part of a broader cleanup of
extension descriptions and boundary text in the Extensions Pack ballot.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the **`procedure-targetBodyStructure`** extension definition in the
Extensions Pack so that both its short description and its
`ElementDefinition.comment` explicitly state when to use it. Concretely:

- Reword the extension's short description to read along the lines of
  *"Anatomical location targeted by the procedure, expressed as a reference
  to a BodyStructure resource. Use when the body site requires more detail
  than a single code on `Procedure.bodySite` can carry (e.g. patient-specific
  anatomy, laterality, multiple parts, image-derived morphology, or a
  longitudinally identified structure)."*
- Add a "Boundary and Relationships" / "Notes to Implementers" paragraph that
  contrasts the extension with the three core elements named in the ticket:
  - `Procedure.bodySite` (CodeableConcept) — preferred when a coded site is
    sufficient.
  - `ServiceRequest.bodySite` (CodeableConcept) — equivalent on the order
    side.
  - `ServiceRequest.bodyStructure` (Reference(BodyStructure)) — the R5+ core
    parallel that this extension pre-adopts on `Procedure` until the
    equivalent element is added to `Procedure` in R6.
- Add a "Notes to Balloters" entry stating that for R6 the extension is
  expected to be retired in favour of a native `Procedure.bodyStructure`
  element (mirroring `ServiceRequest.bodyStructure`), so implementers
  building today on R4/R5 can use the extension and migrate cleanly.

#### Justification

This is exactly what the reporter is asking for, and aligns with the broader
FMG ballot work cleaning up Extensions Pack descriptions (see sibling tickets
FHIR-52938/52939/52955/52977 for Patient Care and FHIR-53023/53024 for OO).
It also follows the FHIR convention that a coded element and a
`Reference(BodyStructure)` serve different precision levels — the same
boundary text already used for `ServiceRequest.bodySite` vs
`ServiceRequest.bodyStructure` in R5 can be carried over almost verbatim,
giving consistency across resources. The change is editorial / metadata only,
so risk to implementers is minimal.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than (only) editing prose, treat FHIR-53022, FHIR-53023 and FHIR-53024
as a single coordinated change and:

1. Define a shared "BodyStructure target" guidance block once (e.g. in the
   Extensions Pack's general usage notes or in the BodyStructure reference
   page) and have the `procedure-targetBodyStructure`,
   `servicerequest-bodyStructure`-style, and `diagnosticreport`-equivalent
   extensions all link to it.
2. Add a Patient-Care/OO co-signed "Notes to Balloters" entry committing to
   the R6 plan: introduce native `Procedure.bodyStructure` (and matching
   `DiagnosticReport.bodyStructure`) elements parallel to
   `ServiceRequest.bodyStructure`, and deprecate the extensions on the same
   schedule.
3. Update the extension's `context` and `element.short`/`comment` text to
   reference the shared guidance.

#### Justification

The reporter raised this against `Procedure`, but the root problem is the
same on `ServiceRequest` and `DiagnosticReport` (which is why the FMG
roll-up groups them). Solving each ticket independently risks three subtly
different boundary statements. Defining the boundary once and linking from
each extension is more maintainable, gives implementers a single canonical
place to look, and keeps the message about R6 native-element migration
consistent across resources. The trade-off vs A is that it requires
coordination with OO (and possibly OOM/FMG) and is a slightly larger edit;
the upside is far better long-term clarity.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as "Persuasive with Modifier — No Change" with a note that
the boundary is already implicit: `Procedure.bodySite` is a code, the
extension carries a reference to `BodyStructure`, and the choice is the same
as for `ServiceRequest.bodySite` vs `ServiceRequest.bodyStructure` in R5,
which is already documented on `BodyStructure` and on the ServiceRequest
element. Direct readers there rather than duplicating guidance on the
extension.

#### Justification

The mechanism distinction (CodeableConcept vs Reference(BodyStructure)) is a
general FHIR pattern, not specific to this extension, and `BodyStructure`
itself documents when a referenced structure is appropriate. Repeating that
text on every extension that points at `BodyStructure` adds maintenance
burden and risks divergence. Declining keeps the extension definition lean.

---

### Recommendation

**Recommended disposition:** B

This ticket is one of three nearly-identical requests from the same FMG
ballot round (FHIR-53022 for Procedure, FHIR-53023 for ServiceRequest after
R6, FHIR-53024 for DiagnosticReport). The reporter's underlying complaint —
"I can't tell which of the body-site mechanisms to use" — is real, and a
work-group-level answer that solves it once for all three extensions will
serve implementers far better than three independent prose edits. Disposition
A is acceptable as a fallback if cross-WG coordination is impractical in the
ballot window, but the cleanest outcome is the shared boundary block plus an
explicit R6 deprecation/migration note in "Notes to Balloters". Disposition C
underestimates how often implementers ask exactly this question and would
leave the Extensions Pack ballot with the same gap the reporter identified.
