## Summary

Saul Kravitz asks whether the US SDOH Clinical Care (SDOHCC) IG should
**require** support for FHIR `Subscription`, citing the load that
polling places on both polling and polled systems and noting that the
HL7 Physical Activity IG already requires Subscription support. The
ticket targets the SDOHCC exchange flows (notably the Task-based
referral / status pattern between care providers and CBROs) where
status changes today are typically discovered via polling.

The cross-reference index returns **no items** (no linked Jira tickets,
no Zulip threads, no GitHub items reference this ticket directly). A
keyword-based Jira search nevertheless surfaces a tightly-coupled
sibling on the same IG — [FHIR-31240](https://jira.hl7.org/browse/FHIR-31240)
("Add support for subscription"), reported by Robert Dieterle for the
same SDOHCC Task flow — which has already been **Published with
"Persuasive with Modification"** in a later block vote. That outcome
materially changes the framing of FHIR-40826: the IG has since moved in
the requested direction, but stopped short of a `SHALL`. The ticket was
originally dispositioned "Considered for Future Use" (Kravitz/Jones
7-0-0 on 2023-06-22) and then re-opened post STU2.1 publication by
lynn_laakso for re-consideration, hence the current `Block-Vote-1`
label.

## Proposed Dispositions

The dispositions below reflect that there is no GitHub repo briefing
to anchor concrete file paths against; any change would land in the
SDOHCC IG sources owned by the Patient Care / Gravity Project team.
The resolution must also remain consistent with the prior
"Persuasive with Modification" outcome for the sibling
[FHIR-31240](https://jira.hl7.org/browse/FHIR-31240).

### Disposition A: Accept As Requested

#### Proposal

Persuasive. Update the SDOHCC IG to **require** Subscription support
for SDOHCC Task status changes:

- Add a `SHALL` conformance statement on the SDOHCC server
  CapabilityStatement requiring support for the R5 Topic-Based
  Subscription R4 Backport (`SubscriptionTopic` + `Subscription`).
- Define (or reference) an SDOHCC `SubscriptionTopic` keyed on changes
  to `Task.status` for SDOHCC Task profiles, mirroring the pattern
  used by the HL7 Physical Activity IG that the reporter cites as
  precedent.
- Update the IG narrative pages on referral workflow / CBRO interaction
  to replace "poll for status" guidance with subscription-based
  notification as the normative path, retaining polling only as a
  documented fallback.

#### Justification

This is what the ticket literally asks for and matches the precedent
of the Physical Activity IG. Subscription-based notification removes
the well-known scaling burden of polling on both the requester and
the CBRO. The HL7-wide trend is consistent with this direction:
[FHIR-45939](https://jira.hl7.org/browse/FHIR-45939) and
[FHIR-45936](https://jira.hl7.org/browse/FHIR-45936) on Da Vinci PCT
both received "Persuasive with Modification" for the same underlying
complaint. A `SHALL` here gives implementers a clear, testable
conformance target.

---

### Disposition B: Alternative Approach

#### Proposal

Persuasive with Modification. Adopt subscription support but at
`SHOULD` strength, scoped to the R5 Topic-Based Subscription R4
Backport, and ship a normative `SubscriptionTopic` rather than a bare
`SHALL Subscription` requirement:

- Define an SDOHCC `SubscriptionTopic` for SDOHCC Task status changes
  (and, where appropriate, for new `ServiceRequest` / referral events).
- Mark server support for this topic as `SHOULD` in the
  CapabilityStatement, with a documented expectation that it will be
  promoted to `SHALL` in a subsequent release once implementation
  experience is gathered.
- Explicitly *deprecate* polling as the recommended pattern in the
  narrative without removing it; servers that cannot yet host
  subscriptions remain conformant.
- Cross-reference the
  [FHIR-31240](https://jira.hl7.org/browse/FHIR-31240) resolution so
  this disposition is consistent with the already-published
  Subscription guidance in the IG.

#### Justification

The IG has *already* gone partway in this direction:
[FHIR-31240](https://jira.hl7.org/browse/FHIR-31240) was published as
"Persuasive with Modification" — i.e., the WG accepted the need but
declined a `SHALL`. The MedMorph experience captured in
[FHIR-30716](https://jira.hl7.org/browse/FHIR-30716) shows the concrete
risk of binding `SHALL` to bare R4 Subscription — implementers found
the R4 criteria semantics not obviously implementable, which is what
prompted the R5 redesign and the R4 backport. Using
`SubscriptionTopic` (the modern, tested pattern) at `SHOULD` strength
captures the reporter's intent, avoids re-litigating the issue
MedMorph hit, gives small CBROs a phase-in path, and keeps the door
open to tightening to `SHALL` once topic-based servers are widespread
in the SDOHCC implementer community.

---

### Disposition C: Decline

#### Proposal

Not Persuasive — keep the previous "Considered for Future Use"
disposition. Decline to add a new conformance requirement on
Subscription, on the basis that the underlying need is already
addressed by [FHIR-31240](https://jira.hl7.org/browse/FHIR-31240)'s
published "Persuasive with Modification" change, and that mandating
Subscription would impose a breaking conformance burden on already-
deployed SDOHCC servers and CBROs that today rely on polling.

#### Justification

The IG has already accepted Subscription as a *supported* mechanism
for Task status notification via FHIR-31240; layering an additional
`SHALL` from this ticket on top of that risks a breaking change for
implementers in the field. Polling, while inefficient, is a
universally implementable fallback. Implementer pushback documented
in [FHIR-30716](https://jira.hl7.org/browse/FHIR-30716) shows the WG
has reason to be cautious about hard-requiring Subscription in IGs
that target a heterogeneous implementer population (which SDOHCC
explicitly does — many CBROs are small organisations).

---

### Recommendation

**Recommended disposition: B (Alternative Approach — Persuasive with
Modification).**

Disposition B is the only option that is simultaneously consistent
with the WG's prior decision on the sibling ticket
([FHIR-31240](https://jira.hl7.org/browse/FHIR-31240) →
"Persuasive with Modification"), responsive to the reporter's
concrete concern about polling burden, and defensible against the
implementability pushback captured in
[FHIR-30716](https://jira.hl7.org/browse/FHIR-30716). Disposition A
would re-open a debate the WG has already resolved against `SHALL`,
and would impose a hard breaking-change risk on CBRO implementers.
Disposition C ignores that the broader HL7 community has shifted
toward Subscription as the default notification mechanism (see the
PCT tickets [FHIR-45939](https://jira.hl7.org/browse/FHIR-45939) and
[FHIR-45936](https://jira.hl7.org/browse/FHIR-45936)) and would leave
SDOHCC trailing the Physical Activity IG precedent the reporter
explicitly cited. Defining an SDOHCC `SubscriptionTopic` (Task status
changes) and requiring `SHOULD`-level support, with an explicit
roadmap toward `SHALL` in a future release, gives implementers a
clear target without breaking conformance for current deployments.
