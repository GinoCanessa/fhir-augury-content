## Summary

The reporter notes that the published examples for the VisualAcuity profile
(in the Ophthalmology Retinal FHIR IG) carry the body site as
`bodySite.extension(bodySite)` — i.e., they use the `bodySite` extension
that points to a `BodySite` resource — when a plain `Observation.bodySite`
`CodeableConcept` would be sufficient. Because visual acuity always applies
to a whole eye structure (not to a sub-anatomical part), the richer
`BodySite` resource is unnecessary, and the wider audience for this profile
(e.g., GPs performing child health checks, not just ophthalmologists) is
better served by simple SNOMED CT codes. The reporter proposes using
`244486005 | Entire eye |`, `362503005 | Entire left eye |`, and
`362502000 | Entire right eye |` directly in `Observation.bodySite` in the
examples.

No linked, cross-referenced, or related Jira tickets, Zulip threads, or
GitHub items were surfaced for this ticket.

## Details

**Description:**

The examples of visual acuity use bodySite.extension(bodySite) for the
body site.

There is no need for the richness and complexity of the BodySite resource
for visual acuity, as visual acuity applies to the whole eye structure and
not to some parts of it.

VisualAcuity is used much wider than just Ophthalmology, for example in
child health checks performed by a GP. bodySite (as CodeableConcept) is
permitted and should be used here in the examples.

SNOMED CT has suitable concepts:

- 244486005 | Entire eye |
- 362503005 | Entire left eye |
- 362502000 | Entire right eye |

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the VisualAcuity example instances in the Ophthalmology Retinal IG
so that the body site is conveyed via `Observation.bodySite`
(`CodeableConcept`) rather than via the `bodySite` extension referencing a
`BodySite` resource. Use the SNOMED CT codes the reporter proposes:

- Whole-eye examples → `244486005 | Entire eye |`
- Left-eye examples → `362503005 | Entire left eye |`
- Right-eye examples → `362502000 | Entire right eye |`

Remove the now-unused `BodySite` example resources (or leave them only if
another example genuinely needs them), and update any narrative in the
profile / example pages that walks through the old extension-based
pattern.

#### Justification

`Observation.bodySite` is the FHIR-standard slot for the anatomical site
of an observation, and it accepts a `CodeableConcept`. Reserving the
`bodySite` extension (which references the heavier `BodySite` resource)
for cases that need richer anatomical detail is consistent with how the
core Observation guidance and most other observation IGs handle simple
whole-organ sites. Using SNOMED CT "Entire eye / left / right" concepts
is unambiguous, internationally recognised, and matches existing
clinical practice in both ophthalmology and general primary-care
workflows. Aligning the examples with the simpler pattern lowers the
implementation bar for the broader (non-ophthalmology) audience the
profile already targets.

---

### Disposition B: Alternative Approach

#### Proposal

Keep both representations, but make the simple `Observation.bodySite`
`CodeableConcept` form the *primary* example pattern across the
VisualAcuity examples, and retain **one** illustrative example that uses
the `bodySite` extension referencing a `BodySite` resource — clearly
labelled as an "advanced" pattern for cases where richer anatomical
context is required (e.g., a partially affected region of the eye, or
linkage to a specific patient-anatomy record). Update the profile
narrative to state the policy: prefer `Observation.bodySite` with the
SNOMED CT "Entire eye / left / right" codes; use the `bodySite`
extension only when the simple coded site is insufficient.

#### Justification

This preserves the implementer-friendly default the reporter is asking
for while still showing how the extension is intended to be used. It
avoids the (small) risk that an existing implementer who copied the
current example loses their reference, and it documents an explicit
guidance rule rather than just changing examples silently. The trade-off
versus Disposition A is slightly more authoring effort and one extra
example to maintain, in exchange for clearer guidance on *when* the
extension is appropriate.

---

### Disposition C: Decline

#### Proposal

Leave the examples as they are. Both `Observation.bodySite`
(`CodeableConcept`) and the `bodySite` extension referencing the
`BodySite` resource are valid FHIR mechanisms; the existing examples are
not incorrect, and the profile already permits implementers to choose
the simpler form. Optionally add a one-line note in the profile
narrative acknowledging that `Observation.bodySite` with a SNOMED CT
"Entire eye" code is also acceptable.

#### Justification

The technical content of the IG is not wrong — the examples just choose
the more elaborate of two valid options. Changing example resources
that have presumably been in the IG for some time can confuse
implementers who already mirror them, and the cost/benefit of an
examples-only edit is debatable when no implementers have publicly
flagged interoperability problems (no related Zulip threads, GitHub
items, or linked tickets surfaced for this issue).

---

### Recommendation

**Recommended disposition:** A

The reporter's argument is straightforward and well-grounded: visual
acuity is intrinsically a whole-eye observation, the wider clinical
audience for the profile (notably primary-care child health checks) has
no need for the `BodySite` resource's richness, and SNOMED CT already
provides clean "Entire eye / left / right" codes. The current examples
push implementers toward an unnecessarily heavy pattern, which is
exactly the kind of friction examples are supposed to remove. There is
no countervailing signal from related discussion (no linked tickets,
Zulip threads, or GitHub items were found) that the existing examples
are doing useful work. Disposition B is a defensible fallback if Patient
Care wants to keep one "advanced" example, but in the absence of a
concrete use case for the extension within visual acuity, the simpler
fix in Disposition A is the better default and matches the reporter's
intent exactly.
