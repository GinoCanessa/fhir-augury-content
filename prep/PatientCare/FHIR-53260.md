## Summary

The reporter observes that the IPS profiles for `AllergyIntolerance`,
`MedicationRequest`, and `MedicationStatement` carry the informal comment
*"In the scope of the IPS the entered-in-error concept is not allowed."* on
the status elements (`AllergyIntolerance.status`, `MedicationRequest.status`,
`MedicationStatement.status`). The text reads like a normative constraint,
but nothing in the profiles enforces it: the bound value sets still include
`entered-in-error`, and no invariants prohibit the code. The ticket asks the
IPS project to clarify its intent and either (a) make the prohibition
enforceable or (b) remove/soften the comment so implementers are not misled.

Triage discussion on the 25 Nov IPS call confirmed that the intent *is* to
disallow `entered-in-error`, and that the appropriate fix is to (i) mint
IPS-specific value sets that are subsets of the base FHIR status value sets
minus `entered-in-error`, (ii) bind those value sets to the respective
elements, and (iii) adjust the short descriptions to reflect the restriction.
A notable correction emerged during triage: the affected AllergyIntolerance
element is **`verificationStatus`** (not `status` as originally written in
the ticket description). The resolution has been set to **Persuasive** and
the ticket is labelled **Ready-for-vote**. No related Jira tickets were
surfaced by cross-reference analysis; all cross-references resolve to the
four FHIR element paths above plus several IPS/Australia Zulip threads that
place the item on the project-call agendas.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Implement the resolution recorded on the 25 Nov IPS call exactly as
stated, in the **International Patient Summary (FHIR) IG**:

1. Create three new IPS-specific value sets, each a subset of the
   corresponding base FHIR status value set with `entered-in-error`
   excluded:
   - e.g. `ValueSet/allergyintolerance-verification-status-ips` based
     on `allergyintolerance-verification` (http://hl7.org/fhir/ValueSet/allergyintolerance-verification)
     minus `entered-in-error`.
   - `ValueSet/medicationrequest-status-ips` based on
     `medicationrequest-status` minus `entered-in-error`.
   - `ValueSet/medication-statement-status-ips` based on
     `medication-statement-status` minus `entered-in-error`.
2. In the IPS profiles, change the `binding` on:
   - `AllergyIntolerance.verificationStatus` (IPS AllergyIntolerance profile)
   - `MedicationRequest.status` (IPS MedicationRequest profile)
   - `MedicationStatement.status` (IPS MedicationStatement profile)

   from the base FHIR value set to the new IPS value sets with
   `strength = required` (preserving the current binding strength on
   those elements).
3. Replace the free-text profile comment *"In the scope of the IPS the
   entered-in-error concept is not allowed."* with an updated `short`
   definition that names the restriction (exact wording to be supplied
   before ballot vote, per the resolution).
4. Remove the now-redundant comment, or retain it only as a pointer to
   the value-set binding for implementer readability.

#### Justification

This is the disposition the IPS project itself recorded as its
resolution (Persuasive, Ready-for-vote), so accepting as requested
aligns the spec with the stated intent. Expressing the prohibition as a
bound value set with required strength is the normatively enforceable
FHIR mechanism; it makes the restriction machine-checkable by validators
rather than relying on prose commentary that implementers can miss.
Correcting `AllergyIntolerance.status` → `.verificationStatus` during
the same change closes the documentation bug the reporter flagged and
avoids re-opening the issue for a second pass. The pattern (IPS-scoped
value set subsetting a base FHIR value set) is already used elsewhere
in the IPS IG, so this change is consistent with existing patterns and
should not surprise implementers.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than introducing three new value sets, express the prohibition
as an **invariant** on each IPS profile:

- `AllergyIntolerance (IPS)`: invariant `ips-allergy-1` — `verificationStatus.coding.where(system='http://terminology.hl7.org/CodeSystem/allergyintolerance-verification' and code='entered-in-error').exists().not()`
- `MedicationRequest (IPS)`: invariant `ips-medreq-1` — `status != 'entered-in-error'`
- `MedicationStatement (IPS)`: invariant `ips-medstmt-1` — `status != 'entered-in-error'`

Keep the existing base FHIR value-set bindings and upgrade the short
definitions / comments to explicitly reference the invariant.

#### Justification

Invariants are the most direct way to say *"this one code is not
allowed"* without spawning three new value-set artifacts to version and
maintain. For `status` (required, bound to a code data type with a fixed
value set) the invariant is trivial to state and evaluate. This avoids
the maintenance burden of keeping IPS-subset value sets synchronised
with future changes to the base FHIR value sets (e.g., if a new status
code is added in R6, the IPS subset value sets would need republishing;
an invariant targeting only `entered-in-error` would not). The trade-off
is that validators must evaluate FHIRPath, and the restriction is less
discoverable in tooling that only surfaces bindings; many IPS
implementers rely on binding-driven UIs for code pick-lists.

---

### Disposition C: Decline

#### Proposal

Close the ticket as **Not Persuasive** and simply **remove** the
free-text comment *"In the scope of the IPS the entered-in-error
concept is not allowed."* from the three profiles, without adding a
value-set or invariant restriction. Document in the IPS guidance that
`entered-in-error` remains a valid status code but SHOULD NOT be used
to communicate clinically meaningful entries in an IPS document.

#### Justification

An IPS is a *point-in-time snapshot* of a patient's summary; in
practice implementers would not deliberately include an
`entered-in-error` entry in the summary payload anyway, so the
restriction is largely theoretical. Declining avoids introducing new
normative artifacts (three value sets or three invariants) that have
to be maintained across IPS revisions and aligned with downstream IGs
(AU PS, EU PS, etc.) that may already profile these elements
differently. It also sidesteps the risk that a strict prohibition
breaks legitimate edge cases — e.g., a receiving system that retains
a prior version of an entry that has since been marked
`entered-in-error` at the source and wants to surface that status for
reconciliation.

---

### Recommendation

**Recommended disposition:** A

The IPS project has already discussed this on a call, agreed that the
intent genuinely is to disallow `entered-in-error`, and set resolution
to Persuasive with the ticket labelled Ready-for-vote. Disposition A
implements exactly that consensus using the standard FHIR mechanism
(required binding to a scoped value set), which makes the restriction
enforceable by off-the-shelf validators without relying on
comment-reading humans or FHIRPath evaluation. Disposition B's
invariant-based alternative is technically lighter but is less
discoverable in tooling and is inconsistent with how the IPS IG
typically narrows code sets elsewhere. Disposition C is defensible only
if the project wanted to walk back the recorded intent, which it does
not — the 25 Nov resolution is explicit. The only open item for
Disposition A is confirming the exact short-description wording before
the vote, which the resolution already calls out as an action.
