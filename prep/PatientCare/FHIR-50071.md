## Summary

The ticket flags an internal inconsistency in the US SDOH Clinical Care
(SDOHCC) IG's CapabilityStatement: the `_lastUpdated` search parameter is
asserted as **SHALL** for the `Organization` and `Device` resources, but
only **SHOULD** for most other resources in the same CapabilityStatement.
At the same time, `meta.lastUpdated` is not enforced by either cardinality
or Must Support anywhere in the IG, so even the resources where the search
parameter is claimed to be mandatory cannot reliably support it. The
reporter asks whether the conformance expectation for `_lastUpdated`
should be lowered to **SHOULD** in the places where it is currently
**SHALL**, to bring the CapabilityStatement into line with both the
underlying data requirements and the rest of the IG.

The ticket has been triaged with a resolution of **Persuasive** and a
resolution description that narrows the fix further: change
`_lastUpdated` from **SHALL** to **SHOULD** in the **Client Interaction of
Device** and the **Server Interaction of Organization** in the SDOHCC CP
CapabilityStatement.

No related Jira tickets, Zulip threads, or GitHub items were surfaced via
cross-reference or search.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In the **SDOHCC CP CapabilityStatement** (`CapabilityStatement-SDOHCC-CP`)
of the US SDOH Clinical Care IG, change the `_lastUpdated` search
parameter conformance expectation from **SHALL** to **SHOULD** in:

- the **Client** interaction entry for the `Device` resource, and
- the **Server** interaction entry for the `Organization` resource.

Leave the `_lastUpdated` expectation as **SHOULD** for the other
resources where it is already **SHOULD**. Do not introduce any new
cardinality or Must Support constraint on `meta.lastUpdated`. This is
exactly the change captured in the ticket's resolution description.

#### Justification

`_lastUpdated` is defined to operate on `Resource.meta.lastUpdated`
(R4 / R5 base). Asserting **SHALL** support for searching by
`_lastUpdated` while leaving `meta.lastUpdated` unconstrained (no Must
Support, optional cardinality) is internally inconsistent: a conformant
server may legitimately omit `meta.lastUpdated`, in which case the search
parameter cannot meaningfully be supported. Lowering the two outliers to
**SHOULD** brings these resources into line with how the IG handles
`_lastUpdated` everywhere else and with the actual data-element
expectations. It also addresses the first comment's concern that the
client/server mismatch on these two resources looks accidental rather
than designed. Because the change only relaxes a conformance expectation
(SHALL → SHOULD), it is non-breaking for existing implementers: anything
that previously satisfied SHALL still satisfies SHOULD.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the **SHALL** expectations on `_lastUpdated` for `Organization`
(server) and `Device` (client), but make the underlying data element
support explicit by adding **Must Support** to `meta.lastUpdated` on the
SDOHCC profiles for `Organization` and `Device` (and ideally on every
resource where the IG asserts even SHOULD-level support for
`_lastUpdated`). Optionally tighten `Resource.meta.lastUpdated` to
`1..1` Must Support on those profiles so the search parameter is
unambiguously supportable.

#### Justification

This addresses the same internal inconsistency, but in the opposite
direction: instead of weakening the conformance statement, it strengthens
the data requirement so the conformance statement becomes truthful. For
SDOH workflows where consumers may want to incrementally pull records
that changed since a last-known timestamp (a common need for care
coordination and registry updates), stronger guarantees on
`meta.lastUpdated` and `_lastUpdated` are arguably more useful than
weaker ones. The trade-offs vs. Disposition A are: (1) it imposes a new
implementation burden on existing SDOHCC servers and clients, which is a
breaking change for the IG; (2) it is broader in scope than what the
ticket strictly asked for; and (3) Patient Care has not, on this thread,
documented use cases that justify hardening over relaxing.

---

### Disposition C: Decline

#### Proposal

Decline the change and keep the SDOHCC CP CapabilityStatement as
currently published, on the grounds that `_lastUpdated` is a search
parameter defined by the FHIR core and its server-side support is a
property of the endpoint rather than of any specific resource profile.
Implementers who cannot satisfy the `Organization`-server / `Device`-
client SHALLs can declare a deviation in their own derived
CapabilityStatement.

#### Justification

This rationale would lean on the position that CapabilityStatement
expectations are a deployment concern, not a structural-data concern, and
that the existing text — even if visibly inconsistent — has been
published and any change introduces churn for existing implementers.

This justification is weak: the inconsistency is real, the reporter and
the first triage comment both flag it as likely incorrect, and the
resolution has already been classified as **Persuasive**. Declining
would also leave a SHALL conformance claim that no SDOHCC server or
client can provably meet given the current `meta.lastUpdated` data
requirements. Included here for completeness.

---

### Recommendation

**Recommended disposition:** A

The work group has already classified the ticket as **Persuasive** with
a precisely-scoped resolution that matches Disposition A: lower
`_lastUpdated` from SHALL to SHOULD in the **Device** client interaction
and the **Organization** server interaction of the SDOHCC CP
CapabilityStatement. This is the minimum, non-breaking change that
removes the inconsistency the reporter identified, is consistent with how
`_lastUpdated` is treated for the other resources in the same
CapabilityStatement, and avoids the collateral implementer burden that
Disposition B (adding Must Support / tighter cardinality on
`meta.lastUpdated`) would impose. Disposition B is worth keeping on the
table as a longer-term hardening discussion — separately ticketed —
once Patient Care has explicit SDOH use cases that justify mandating
last-updated semantics across the IG. Disposition C is not defensible
given the agreed Persuasive resolution.
