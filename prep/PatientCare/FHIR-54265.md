## Summary

The reporter observes that the `Goal` resource carries two status-bearing
elements — `Goal.lifecycleStatus` (the workflow state of the Goal itself)
and `Goal.achievementStatus` (whether the Goal is being achieved) — but
the existing `Goal.statusReason` element does not, by name alone, make
clear which of those two statuses it explains. The reporter therefore
asks that `Goal.statusReason` be renamed to `Goal.achievementStatusReason`
to align with the status it actually qualifies and to improve semantic
clarity for implementers. No related Jira tickets were surfaced through
cross-references; the ticket stands on its own as a naming/clarity
change against the `Goal` resource in FHIR Core.

## Details

Goal resource includes 2 "status" element:

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Rename `Goal.statusReason` to `Goal.achievementStatusReason` in the FHIR
Core `Goal` resource definition. This is a breaking change to the
resource's element set:

- Update the element name in the resource definition (StructureDefinition,
  XML/JSON schemas, generated code/examples).
- Update the element's short/definition text to make explicit that it
  explains the value held in `Goal.achievementStatus`.
- Update all examples, search parameters (if any reference the path),
  invariants, and narrative that mention `Goal.statusReason` to use the
  new name.
- Add a change-note / backward-compatibility entry to the Goal resource
  history page indicating the rename and the prior name.

#### Justification

The reporter's observation is factually correct: `Goal` carries two
distinct status concepts (`lifecycleStatus` and `achievementStatus`),
and the unqualified name `statusReason` is genuinely ambiguous.
`Goal.achievementStatusReason` follows the naming pattern already used
elsewhere in FHIR where a `*StatusReason` element pairs with a specific
`*Status` element (and mirrors the `lifecycleStatus` / `achievementStatus`
split that already exists on `Goal`). Renaming brings the element name
into alignment with its definition and removes a class of
implementer-confusion errors at parse/mapping time rather than at
runtime.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the element name `Goal.statusReason` unchanged, but tighten the
element's definition, short description, and any usage notes to state
unambiguously that it explains `Goal.achievementStatus` (and not
`Goal.lifecycleStatus`). Optionally, add a note in the Goal resource
narrative that contrasts `lifecycleStatus` (and the lack of a paired
`lifecycleStatusReason`) with `achievementStatus` + `statusReason`.
If a stronger signal is desired without a breaking rename, consider
adding a non-normative alias / "also known as" entry pointing
`achievementStatusReason` → `statusReason`.

#### Justification

`Goal.statusReason` is an existing, normatively-named element that has
been in the spec across multiple versions and is referenced by
implementers, profiles, IGs, mappings, and stored instance data. A
rename is a hard breaking change that ripples into every system that
parses or produces `Goal` resources, into derived IGs (e.g., US Core
Goal), into terminology bindings keyed off the element path, and into
historical instance data. A definitional clarification achieves the
reporter's stated goal — removing the need to read the definition to
disambiguate — at a tiny fraction of the implementer cost, while
preserving wire-format and tooling compatibility.

---

### Disposition C: Decline

#### Proposal

Decline the rename and make no change. Rationale category: breaking
change not justified by the size of the clarity benefit, given that the
element's definition already states which status it explains and that
no implementer pain has been reported beyond the abstract observation
of ambiguity.

#### Justification

FHIR Core element renames carry an outsized cost relative to the
clarity benefit when the element is already well-defined in its own
`definition` text. There is no Zulip thread, linked ticket, or
implementer report attached to this CR that demonstrates real-world
confusion or incorrect implementations stemming from the current name;
the ticket reads as a stylistic suggestion. Declining preserves
backward compatibility for every consumer of `Goal` (including US Core
and other downstream IGs that profile or reference `Goal.statusReason`)
and avoids triggering migration work across the ecosystem for a
cosmetic gain.

---

### Recommendation

**Recommended disposition:** B

The reporter has identified a real readability issue — the bare name
`statusReason` does not telegraph which of `Goal`'s two statuses it
qualifies — and the workgroup should acknowledge that. However,
`Goal.statusReason` is a long-standing, normatively-named element that
is referenced by downstream IGs, profiles, mappings, terminology
bindings, and existing instance data; a rename is a breaking change
whose implementer cost is large compared with the size of the clarity
gain. Disposition B captures the benefit the reporter is asking for
(no need to read the definition to know which status is being
explained) by tightening the definition and short text, without
forcing every Goal-touching system, IG, and instance store to migrate.
If the workgroup later concludes that the ambiguity is causing
demonstrable implementer harm, a rename can be revisited as a
deliberate, ecosystem-wide migration; in the meantime, B is the
proportionate response.
