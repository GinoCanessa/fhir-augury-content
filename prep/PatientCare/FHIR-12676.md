## Summary

The submitter (Alexander Henket, Nictiz / Netherlands) is asking Patient
Care for guidance — and potentially for a new resource or formal
profiling pattern — to model a **General Practitioner (GP) encounter
report** in FHIR using the **SOAP** method (Subjective, Objective,
Assessment, Plan; locally "SOEP"). In real-world Dutch GP practice, a
single Encounter typically contains multiple "deelcontacten" — one
S-O-A-P tuple per problem — and each tuple may be linked to a
longer-running Episode/problem. Each of S, O, A, P is principally text
but can be coded (e.g., ICPC-1-NL) or replaced by a structured
resource (Observation for O; CarePlan, MedicationRequest, ServiceRequest
for P). The submitter's gut feeling is that an `EncounterReport`
resource is missing and that `Composition` is at best a fallback.
Lloyd McKenzie's initial response pointed to `ClinicalImpression`
(formerly ClinicalAssessment), but the submitter argues that resource
covers only A (and partially O), not S or P. The ticket carries the
`Discuss-at-WGM` label and was earmarked for joint Patient Care / SD /
FHIR-I quarters at the Sept 2017 (St. Louis) and Jan 2018 (New Orleans)
WGMs but was ultimately auto-deferred.

There are no linked or otherwise cross-referenced Jira tickets, and no
related GitHub items. The substantive context lives entirely in two
Zulip threads (`#implementers > GP contact report` from Jan 2017 and
`#implementers > GP Encounter Report` from Jan 2018), both of which the
submitter started.

inconclusively with Lloyd reiterating that `ClinicalImpression` was
the intended target but had not been profiled to prove it, and

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Introduce a new `EncounterReport` (or equivalently named) resource in
the FHIR Core Patient Care domain to formally represent a problem-
oriented S-O-A-P journal entry attached to an `Encounter`. The resource
would carry, at minimum:

- `subject`, `encounter`, `episodeOfCare` (0..*) for context;
- a repeating `problem` / `concern` group, each containing
  - `problem` → `Reference(Condition)` with an optional ICPC-coded
    `CodeableConcept`,
  - one occurrence each of `subjective`, `objective`, `assessment`,
    `plan`, where each occurrence carries a narrative `text` plus
    `entry` 0..* `Reference(Any)` (typed-allowed list: `Observation`
    for O; `Condition` / `ClinicalImpression` for A; `CarePlan` |
    `MedicationRequest` | `ServiceRequest` | `NutritionOrder` |
    `Communication` for P);
- coding slots for the section-level concept (LOINC 67781-5 default).

Patient Care would own the resource; Structured Documents would be
consulted on the `Composition`/section overlap.

#### Justification

The Zulip threads converge on a finding that none of the existing
resources cleanly model a problem-oriented S-O-A-P journal entry as
implemented in real GP systems: `ClinicalImpression` covers A (and
partially O), `Composition` is generic and document-oriented,
`Observation` is awkward for narrative S and not well suited for
P, and `CarePlan` is overkill for the typical "P". A purpose-built
resource is what the submitter explicitly asks for and what the Swiss
(in4medicine/elexis) and Dutch (Nictiz) implementer experiences both
imply is missing. The grouping of multiple S-O-A-P tuples per Encounter
(Dutch *deelcontact*; Lloyd McKenzie agreed Encounters cannot be split
per problem because of billing semantics) is a structural requirement
that does not fit naturally into any existing resource without an
extension.

---

### Disposition B: Alternative Approach

#### Proposal

Do **not** add a new resource. Instead, formally adopt the existing
`Composition`-based pattern that PC, SD, and FHIR-I had already started
exploring (per Eric Haas and Michelle Miller in the 2018 Zulip thread)
and publish it as a Patient Care **implementation guidance page** plus
a **profile** on `Composition`:

- `Composition.type` = LOINC `67781-5` (Summarization of encounter
  note Narrative);
- `Composition.encounter` = the `Encounter`;
- one `Composition.section` per *deelcontact* (problem), with
  - `section.code` = problem-level concept (optional ICPC extension),
  - nested `section` entries with `code` = S / O / A / P,
    `section.text` carrying the narrative (per SD's guidance that
    `section.text` alone is sufficient), and `section.entry`
    referencing `Observation` / `Condition` / `ClinicalImpression` /
    `CarePlan` / `MedicationRequest` / `ServiceRequest` etc. when
    structured detail exists;
- link the per-problem `Condition` (category `problem-list-item` or
  `encounter-diagnosis`) and any `EpisodeOfCare` via the section's
  `focus` / `entry`.

Document this pattern in the Patient Care general-practice / encounter
guidance and provide it as a normative example/profile (echoing
Henket's `gp-EncounterReport` Simplifier profile).

#### Justification

Adding a Core resource is expensive, requires multi-WG agreement,
implementer demand evidence, and carries long compatibility
obligations. The Zulip discussion shows that the building blocks
(`Composition`, `Encounter`, `Condition`, `Observation`, `CarePlan`,
`MedicationRequest`, `ServiceRequest`, optional `ClinicalImpression`)
are already sufficient when assembled with clear guidance, and that
both PC and SD were already moving in this direction. A documented
pattern + profile is reversible if implementer experience later shows
a dedicated resource is necessary; a new Core resource is not.
This also avoids the open issue (raised in the 2018 thread) that
`ClinicalImpression` has few proponents — by relegating it to an
optional A-section entry rather than the load-bearing structure.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Mod / Not Persuasive — Already Addressed**:
the request is for guidance on a pattern that is already supportable
with `Encounter` + `Composition` + `Condition` + `Observation` +
`ClinicalImpression` + `CarePlan` + `MedicationRequest` /
`ServiceRequest`, exactly as Lloyd McKenzie outlined in 2017 and as
the in4medicine implementation demonstrates. No spec change is
required; any remaining gap is a documentation/guidance concern that
should be tracked as a separate, more narrowly scoped ticket against
the Patient Care implementation guidance pages, or pursued in an IG
(e.g., a Dutch GP IG or International Patient Summary companion).

#### Justification

The ticket is from 2017, was auto-deferred, and has had no substantive
movement since 2018. No subsequent ballot comments, downstream Jira
tickets, or PRs reference it. Two implementer communities (Dutch GPs
via Nictiz, Swiss GPs via in4medicine) reported workable — if
imperfect — implementations using existing resources, and PC/SD work
on Clinical Notes / Composition guidance has continued in parallel
through other tickets. Re-opening a Core resource discussion eight+
years later, against a single original requester and no fresh
implementer pressure, is hard to justify against the cost of resource
addition.

---

### Recommendation

**Recommended disposition:** B

The Zulip record makes two things clear: (1) implementers genuinely
struggle to model multi-problem GP encounters in a uniform way, so
"do nothing" (Disposition C) would leave a real gap and would be
inconsistent with the original `Discuss-at-WGM` intent; but (2) the
WG and SD had already aligned on a `Composition`-based pattern by
early 2018, with section-level narrative permitted and structured
entries optional, and the submitter himself published a working
profile on Simplifier using exactly that approach. A new resource
(Disposition A) carries a substantial Core-spec cost — multi-WG
sign-off, breaking-change risk, FMM lifecycle — for benefits that a
profile + IG-style guidance can capture today and revisit later if
needed. Disposition B both honours the original ask (clear, FHIR-
sanctioned guidance for the SOAP/SOEP encounter report) and respects
the WG's de-facto direction, with a much lower risk profile and a
clean upgrade path to Disposition A if real-world implementer demand
later materialises.
