## Summary

The reporter asks the PACIO Personal Functioning and Engagement (PFE)
IG to define a new `Observation` profile that captures a *derived
result* — a clinical interpretation (typically a condition/problem
characterization) computed from a Questionnaire response — modelled
after the [SDOHCC ObservationAssessment](http://build.fhir.org/ig/HL7/fhir-sdoh-clinicalcare/StructureDefinition-SDOHCC-ObservationAssessment.html)
profile published by HL7 Cross-Group Projects. The motivating clinical
example is the Berg Balance Scale: a series of scored balance tasks is
captured (e.g., via a `QuestionnaireResponse`), the item scores are
summed, and the total score is mapped to a categorical fall-risk
determination (41–56 = low, 21–40 = medium, 0–20 = high). The PACIO
PFE IG today has no profile that explicitly represents that derived
classification distinct from the underlying score Observation, so this
ticket requests a profile to standardize how that derivation is
exchanged. The ticket was originally resolved as *Considered for Future
Use* (out of scope for STU-1) on 2023-08-24 (vote 7-0-0, Brian Meshell
/ Dave Hill) and re-opened post STU-1 publication by Lynn Laakso for
disposition in STU-2.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add a new profile in the **PACIO Personal Functioning and Engagement
(PFE) IG** named, e.g., `PFEObservationDerivedAssessment` (or
`PFEAssessmentInterpretation`), modelled directly on
`SDOHCC-ObservationAssessment`. Concretely:

- `Observation.status`: required (`final` / `amended` / `corrected`).
- `Observation.category`: must include `survey` (and PFE's existing
  PFE category if one is bound).
- `Observation.code`: bound (extensible) to a value set of LOINC
  assessment codes used by PFE (Berg Balance Scale total = LOINC
  76437-3 as the seed concept; bind a value set, not a single code, so
  additional instruments can be added without re-profiling).
- `Observation.subject`: required, `Reference(Patient)`.
- `Observation.effective[x]`: required.
- `Observation.value[x]`: must support `valueCodeableConcept` bound
  (extensible) to a "PFE Assessment Interpretation" value set seeded
  with the Berg Balance Scale categories (low / medium / high fall
  risk) — this is the *derived* clinical determination.
- `Observation.derivedFrom`: must support, `Reference(Observation |
  QuestionnaireResponse)`, with a constraint requiring at least one
  reference (this is what makes the profile "derived").
- `Observation.hasMember`: must support, `Reference(Observation)` for
  panel-style assemblies of the contributing scored item Observations.
- Optional but encouraged: profile-level `obeys` invariant requiring
  that `derivedFrom` resolve to either a panel `Observation` or a
  `QuestionnaireResponse` whose `questionnaire` is one of the
  PFE-recognized instruments.

Add a usage page under PFE that walks through the Berg Balance Scale
example end-to-end (QuestionnaireResponse → per-item Observations →
panel/total-score Observation → derived `PFEObservationDerivedAssessment`
with the categorical fall-risk interpretation).

#### Justification

This is the smallest concrete addition that satisfies the requester's
stated need and aligns PFE with an already-balloted, in-the-wild
HL7 pattern (`SDOHCC-ObservationAssessment`). It re-uses core FHIR
machinery (`derivedFrom`, `hasMember`, `valueCodeableConcept`) rather
than inventing PFE-only structure, which keeps the profile
implementer-friendly and consistent with US Core Observation slicing
expectations. The reporter's example (Berg Balance Scale → fall-risk
band) is exactly the use case SDOHCC handles today, so adopting the
same shape minimises implementer cognitive load for systems already
exchanging SDOHCC assessments. Patient Care has explicitly re-opened
the ticket for STU-2, signalling appetite to take it on.

---

### Disposition B: Alternative Approach

#### Proposal

Instead of authoring a brand-new PFE profile, **reuse
`SDOHCC-ObservationAssessment` by reference** in the PFE IG. The PFE
guidance page would:

1. State that, when an instrument's score yields a categorical
   clinical interpretation (e.g., Berg Balance Scale → fall-risk
   band), implementers SHALL use
   `SDOHCC-ObservationAssessment` as the *derived* Observation,
   populating `derivedFrom` with the PFE score Observation or
   `QuestionnaireResponse`.
2. Publish a small PFE value set ("PFE Assessment Interpretation
   Codes") and require it as the binding for
   `Observation.valueCodeableConcept` *in this PFE-scoped use* via
   a usage rule rather than a new StructureDefinition.
3. Provide example resources and a narrative crosswalk in PFE.

If a thin wrapper is still needed for IG-scoped conformance tooling,
publish a derived profile that *only* tightens
`Observation.code` and `Observation.valueCodeableConcept` bindings on
top of `SDOHCC-ObservationAssessment` (a one-step derivation rather
than a sibling profile).

#### Justification

PACIO and SDOHCC already collaborate, and duplicating a near-identical
profile in two IGs increases the maintenance surface and the risk of
quiet drift between them (different `mustSupport` sets, different
slicing on `category`, etc.). Re-using SDOHCC's profile keeps a single
source of truth for the "derived clinical assessment from scored
input" pattern across HL7 and lets PFE focus its STU-2 work on the
PFE-specific value sets. The trade-off versus Disposition A is that
implementers consuming the PFE IG must follow a cross-IG dependency,
and PFE's conformance tests must validate against an externally-owned
profile — which is acceptable because SDOHCC is HL7-published and
balloted.

---

### Disposition C: Decline

#### Proposal

Close as **Persuasive with Mod / Not Persuasive — already addressed by
existing mechanism**, on the grounds that the base FHIR
`Observation` resource (with `derivedFrom`, `hasMember`,
`valueCodeableConcept`, and `interpretation`) plus the existing
score-Observation pattern documented in PFE already supports the Berg
Balance Scale example without a new profile. Direct the reporter to
use `SDOHCC-ObservationAssessment` directly when an SDOH-style derived
assessment is needed, and document a one-paragraph note in PFE
pointing at that profile.

#### Justification

A new profile only adds value if it tightens cardinality, bindings,
or `mustSupport` beyond what implementers can reasonably infer from
base FHIR. The ticket as written does not enumerate PFE-specific
constraints that wouldn't already be satisfied by base `Observation`
or `SDOHCC-ObservationAssessment`, and PFE STU-1 has shipped without
this profile, suggesting current implementers have not been blocked.
Adding a profile carries downstream cost (publishing, validating,
versioning, IG-package size) that should be paid only when the
constraints are concrete. Declining preserves PFE STU-2 scope for
items with clearer implementer demand.

---

### Recommendation

**Recommended disposition:** A

The ticket has been explicitly re-opened by the work group for STU-2,
the requester has supplied a concrete clinical example (Berg Balance
Scale → fall-risk band), and an analogous, balloted profile
(`SDOHCC-ObservationAssessment`) already proves the shape works.
Disposition B (re-use SDOHCC by reference) is attractive on
maintenance grounds but creates a hard cross-IG runtime dependency for
PFE implementers and forces conformance tooling to chase an
externally-owned profile — a real friction point for the rehab/PT
vendor community PFE primarily targets. Disposition C
under-serves the reporter: while base `Observation` *can* express the
derivation, the value of the PFE IG is precisely to remove that
ambiguity for assessment instruments. Authoring a PFE-owned profile,
seeded from the SDOHCC structure but bound to PFE's own assessment-
code and interpretation value sets, gives implementers the clearest
target and keeps PFE self-contained for STU-2. The work group should
also consider opening a parallel coordination ticket with SDOHCC to
keep the two profiles aligned over time.
