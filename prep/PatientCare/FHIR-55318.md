## Summary

The submitter (Lloyd McKenzie) reports that `CarePlan.intent` is a mandatory
(`1..1`) element with a required binding to the `request-intent` value set,
but that value set does not include an `unknown`, `other`, or equivalent
"escape valve" code. Because the element is mandatory and the binding
strength is required, an instance author who genuinely does not know — or
cannot fit — the intent of a CarePlan into one of the existing concepts
(`proposal`, `plan`, `order`, `option`, `directive`) has no conformant
value to send, which can make a `CarePlan` resource impossible to populate
in some real-world scenarios. The ticket asks Patient Care to add an
`unknown`, `other`, or similar concept to the bound value set so that the
required-binding/mandatory-element combination always has a usable answer.

The cross-reference search returned only the ticket's own mention of
`CarePlan.intent` (no linked Jira tickets, no Zulip threads, and no GitHub
items). The `UnknownOther` label indicates the issue is part of a broader
pattern of value-set "escape valve" tickets being tracked across the
specification.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add an `unknown` concept to the `RequestIntent` code system (and therefore
to the `request-intent` value set bound to `CarePlan.intent`,
`ServiceRequest.intent`, `MedicationRequest.intent`, `NutritionOrder.intent`,
`DeviceRequest.intent`, etc.). Define it as: "The intent of the request is
not known." Apply the change to the shared `RequestIntent` artifact rather
than creating a CarePlan-specific value set, so that every resource in the
request pattern continues to share a single intent vocabulary. Mark
`unknown` as a `notSelectable` discouraged code (or document it as the
"data absent" fallback) so it is only used when no specific intent can be
determined.

#### Justification

This directly resolves the reporter's concern with the smallest possible
delta to the spec: one new code in one shared code system. It is consistent
with how FHIR has handled the same pattern elsewhere (e.g., `unknown` in
`AdministrativeGender`, `data-absent-reason` extensions on optional
elements that became required, and the `unknown` code recently added to
several status value sets). It preserves the required-binding strength
(which Patient Care has historically valued for the request pattern) while
guaranteeing that a conformant value always exists. Because every
request-pattern resource shares `RequestIntent`, fixing it once
automatically closes the same gap for every sibling resource that is also
mandatory + required-bound on `intent`.

---

### Disposition B: Alternative Approach

#### Proposal

Leave the `RequestIntent` code system unchanged and instead permit
`CarePlan.intent` (and other `intent` elements in the request pattern) to
carry a `data-absent-reason` extension when the intent is genuinely
unknown. Either (i) relax the cardinality of `.intent` to `0..1` and use
the standard "missing required element" pattern (modifierExtension +
`dataAbsentReason`), or (ii) keep cardinality `1..1` but explicitly
document in the element definition that an `unknown`/`asked-unknown`
`dataAbsentReason` extension on `intent.extension` satisfies the required
binding's "must supply a coded value" intent. Add a constraint /
documentation block on `CarePlan.intent` that calls out this pattern.

#### Justification

`intent` is foundational to the FHIR request pattern: workflow engines,
state machines, and many profiles branch on whether a resource is a
`proposal`, `plan`, or `order`. Adding an `unknown` code into the
vocabulary risks making it acceptable to send "I don't know if this is a
plan or an order" in routine traffic, which would silently break those
state machines. Routing the genuinely-unknown case through the
`data-absent-reason` extension keeps the vocabulary semantically clean,
forces senders to be explicit that data is missing, and matches FHIR's
established mechanism for "required element, no usable value." It also
avoids a code-system version bump for what is, in practice, a rare edge
case.

---

### Disposition C: Decline

#### Proposal

Decline the request. Keep `CarePlan.intent` as `1..1` with a required
binding to the unchanged `request-intent` value set. Document in the
ticket resolution that any instance of `CarePlan` necessarily reflects
some intent on the part of the author or system that produced it, and
that authors must select the closest of `proposal`, `plan`, `order`,
`option`, or `directive` — with `proposal` being the appropriate default
when the intent is otherwise indeterminate.

#### Justification

The five existing codes were deliberately chosen to be exhaustive over
the request lifecycle, and adding `unknown` would weaken the contract
that downstream workflow logic depends on. If a system has enough
information to instantiate a `CarePlan` at all, it generally has enough
context to choose between the existing concepts (e.g., a record imported
from a legacy system without explicit intent metadata is, by default, at
worst a `proposal`). Declining is defensible if Patient Care concludes
that the absence of an escape valve is a feature, not a bug, and that
the `data-absent-reason` extension already handles the truly-unknown
case without changing the vocabulary.

---

### Recommendation

**Recommended disposition:** A

The `UnknownOther` label on this ticket signals that it is part of a
program-wide effort to ensure every mandatory + required-bound element in
FHIR has a usable fallback value. Disposition A is consistent with that
program, fixes every request-pattern resource in one stroke (because
`RequestIntent` is shared), and matches the precedent already set for
status and gender value sets. Disposition B is technically defensible but
adds implementer burden (every system has to learn the
`dataAbsentReason`-on-required-element pattern) for a problem that a
single new code resolves cleanly. Disposition C ignores the legitimate
real-world scenarios — clinical data migrations, narrative-only imports,
and cross-jurisdictional exchange — where the original intent simply was
not captured. The workgroup should accept the change, with the
clarification that `unknown` is the value of last resort and should not be
used when any of the existing concepts plausibly applies.
