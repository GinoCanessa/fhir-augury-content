## Summary

The ticket reports an internal inconsistency in the definition of
`Procedure.performer`. The element's narrative description states that it
represents "The people who performed the procedure", but the underlying
type binding for `Procedure.performer.actor` allows references to
`Organization` and `Device` (in addition to `Practitioner`,
`PractitionerRole`, `Patient`, `RelatedPerson`, `CareTeam`, and
`HealthcareService`). Those targets are not "people", so the description
mis-characterises the cardinality and types that the schema actually
permits. The reporter is asking for the description text to be corrected
so that it accurately reflects the full set of allowed actor types.

No cross-referenced Jira tickets, Zulip threads, or GitHub items were
returned by FhirAugury for this ticket key, so the request is treated as a
standalone editorial fix to the `Procedure` resource's `performer` element
description.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the short and formal definitions of `Procedure.performer` (the
BackboneElement) so they no longer restrict the description to "people".
Replace wording such as "The people who performed the procedure" with
language that matches the actual `performer.actor` target list — for
example: "Indicates who or what performed the procedure and how they were
involved." `performer.actor` itself should also be updated to read along
the lines of "Indicates who or what performed the procedure" rather than
naming "people". No structural change is made: cardinality, type
bindings, and the existing `function` / `onBehalfOf` siblings are
untouched.

#### Justification

This is a pure editorial/definition fix that brings the human-readable
description into line with the machine-readable type binding, which
already permits `Practitioner`, `PractitionerRole`, `Organization`,
`Patient`, `RelatedPerson`, `Device`, `CareTeam`, and
`HealthcareService`. Other resources that share the same actor pattern
(e.g., `ServiceRequest.performer`, `MedicationAdministration.performer`)
already use neutral "who" wording, so this aligns `Procedure` with the
established pattern in Patient Care–owned resources. No implementer
behaviour changes; this is the lowest-risk way to resolve the reported
inconsistency.

---

### Disposition B: Alternative Approach

#### Proposal

Take the editorial fix from Disposition A and additionally tighten the
guidance: rather than just neutralising the wording, add an explicit
sentence to the element comment enumerating the kinds of actors that are
expected (a person, an organisation acting as the performer of record, or
a device that automatically performed the procedure), and add a short
usage note clarifying when `performer.actor = Device` or `Organization`
is appropriate versus when `performer.onBehalfOf` should be used to
attribute the act to an organisation while still naming a human actor.
Optionally cross-reference `Procedure.performer.function` for capturing
the role the actor played.

#### Justification

The original ticket only flags the wording, but the underlying confusion
is broader: implementers regularly ask when to use `Device` /
`Organization` as the actor versus reserving those for `onBehalfOf`.
Patient Care has answered variants of this question elsewhere; folding
that guidance into the element comment costs little and reduces the
chance of the same ticket being re-raised by another reader who hits the
same inconsistency. The trade-off versus Disposition A is a slightly
larger editorial diff and a lightweight content review, but no normative
or schema impact.

---

### Disposition C: Decline

#### Proposal

Close as "Not Persuasive" / "Editorial — Already Conveyed Elsewhere",
on the grounds that the formal definition of `performer.actor` and its
type binding are already authoritative, and the prose phrase "people who
performed the procedure" can be read as a colloquial summary rather than
a normative restriction. No spec change is made.

#### Justification

Defensible only if the work group considers the colloquial summary
sufficient and is unwilling to spend editorial cycles on `Procedure`.
However, this is hard to justify given the cost of a definition tweak is
trivial, the inconsistency is real and discoverable from the published
StructureDefinition, and leaving it in place invites duplicate tickets.
This option exists mainly to be rejected.

---

### Recommendation

**Recommended disposition:** A

The reported defect is a clear narrative-vs-binding mismatch in a
high-traffic, mature resource. Disposition A resolves it with a minimal,
self-contained editorial change to `Procedure.performer` and
`Procedure.performer.actor` definitions, with no cardinality, binding,
or structural impact and therefore no implementer burden. Disposition B
is attractive but expands scope into usage guidance the original
reporter did not ask for and would benefit from a separate ticket if the
work group wants to pursue it. Disposition C should be rejected because
the inconsistency is real, cheap to fix, and likely to recur if left in
place. Note: although the ticket's `specification` field is recorded as
"Ophthalmology Retinal (FHIR)", the described element lives in the FHIR
core `Procedure` resource (Patient Care–owned), which is where the fix
should be applied.
