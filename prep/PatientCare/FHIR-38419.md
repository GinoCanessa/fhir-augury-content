## Summary

FHIR-38419 is a TSMG-sourced ballot comment against the US PACIO Personal
Functioning and Engagement (PFE) FHIR IG. The reporter (Joan Harper, with
"Comment 2 imported by Ron G. Parker") asserts that any locally-defined
Code System in the PFE IG must have its canonical URL anchored under
`terminology.hl7.org` (THO) rather than `hl7.org/fhir`, per HL7 best
practice and the R5 terminologies guidance. The ticket asks the
workgroup to (a) move the Code System's canonical URL to a THO-rooted
namespace and (b) open a corresponding UTG ticket so the Code System is
formally registered in THO. The ticket has been disposed of twice as
"Considered for Future Use" (2023-06-22 and 2024-12-12) and reopened
both times after each STU publication; it is again open against
STU2-Block-Vote-2.

The most recent activity (2025-07-23, lynn_laakso) clarifies that the
specific artifact originally targeted — the PFE Functioning Category
Code System — is now planned for deletion per
[FHIR-38267](https://jira.hl7.org/browse/FHIR-38267) (Persuasive), since
PFE STU-2 instead uses a value set derived from US Core 6.1.0 simple
observation. The remaining locally-defined PFE artifact that could
plausibly be migrated to THO is the PFE US Core VS (and the related
`pfe-category-cs` for ICF concepts), but the comment explicitly
recommends waiting until that artifact has been tested further before
migrating it to THO. A parallel terminology-stream Zulip discussion
([THO External Code System for ICF Concepts](https://chat.fhir.org/#narrow/stream/terminology/topic/THO%20External%20Code%20System%20for%20ICF%20Concepts/near/483480918))
shows the PACIO team also pursuing the THO-anchoring path for the ICF
concepts via the external-code-system request process.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Move the canonical URLs of all PFE-defined Code Systems off
`http://hl7.org/fhir/...` and onto the `http://terminology.hl7.org/...`
namespace, and open a UTG ticket (HL7/UTG repository) to register each
PFE Code System (and its concept content) in THO. Concretely:

- Delete `pfe-functioning-cs` from the PACIO PFE IG entirely (per
  FHIR-38267, since it is being replaced by the PFE US Core VS based on
  US Core 6.1.0 simple-observation categories).
- For any remaining locally-defined PFE Code System still in scope for
  STU-2 (notably `pfe-category-cs` for ICF and the PFE US Core VS's
  underlying CodeSystem reference, if any), change the
  `CodeSystem.url` to a THO-anchored canonical (e.g.,
  `http://terminology.hl7.org/CodeSystem/pfe-category-cs`) in the IG's
  source FSH/JSON, and update all `ValueSet.compose.include.system`
  references and narrative.
- File a UTG ticket against HL7/UTG that contains the CodeSystem
  resource(s) so TSMG can review and publish them under THO.

#### Justification

This is the literal action the ticket requests, is consistent with the
R5 terminologies guidance the description cites, and aligns with the
direction Brian Meshell already initiated on the terminology Zulip
thread for the ICF code system. Deleting `pfe-functioning-cs` is
already the agreed outcome of FHIR-38267 (Persuasive), so doing it
under FHIR-38419 simply makes the URL-anchoring rationale explicit. For
the remaining artifacts, anchoring in THO is the long-standing TSMG
best practice referenced in the description's Confluence deck.

---

### Disposition B: Alternative Approach — Persuasive with Mod / Phased

#### Proposal

Resolve FHIR-38419 as **Persuasive with Modification** for STU-2 with a
two-part response:

1. For `pfe-functioning-cs`: declare it removed by FHIR-38267 (no
   URL change is needed because the artifact ceases to exist in
   STU-2). Cite FHIR-38267 as the implementing change.
2. For the remaining locally-defined PFE terminology that *is* still
   shipped in STU-2 (PFE US Core VS and the supporting code system, if
   any, plus `pfe-category-cs` for ICF): commit in the resolution to
   migrate canonical URLs to THO in STU-3, contingent on (a) the
   external ICF code system being accepted onto the THO External
   Terminologies page via the documented request process Robert
   McClure outlined on Zulip, and (b) the PFE US Core VS having
   sufficient implementation experience. File a tracking UTG ticket
   now so the work is queued, but do not block STU-2 publication on
   it.

#### Justification

This honors the substance of the ballot comment without forcing a
URL-namespace change against an artifact whose content is still being
validated in production. It aligns with the explicit recommendation in
the 2025-07-23 comment ("we recommend waiting until this new VS has
been tested more before it is mature enough to migrate to the THO")
and with Robert McClure's Zulip guidance that THO-hosted external code
systems require an out-of-band requestor process — i.e., the migration
is not unilaterally in PACIO's gift to deliver inside an STU-2 cycle.
Splitting the response also disambiguates the two artifacts the
discussion has conflated across the 2023, 2024, and 2025 reopenings.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Resolve as **Not Persuasive** on the grounds that the underlying
artifact the comment originally targets (`pfe-functioning-cs`) is
already being removed via FHIR-38267, and the remaining PFE
terminology is intentionally hosted in the IG namespace because it is
not yet stable enough for THO publication. State explicitly that the
PFE IG is not committing to any URL move during STU-2 and that future
THO migration will be tracked separately, outside the STU-2 ballot
disposition.

#### Justification

The comment was written against a code system that no longer exists in
PFE STU-2. The IG team has now stated three times (2023, 2024, 2025)
that the remaining PFE terminology is still being experimented with
and is not ready for THO. From a process standpoint, anchoring an
unstable code system in THO would create a stable identifier for an
unstable artifact, which TSMG generally discourages. There is also no
evidence in the cross-reference index, the Zulip thread, or the
ticket's own comments that a UTG ticket is currently actionable for
this specific artifact under STU-2 timing.

---

### Recommendation

**Recommended disposition:** B (Persuasive with Modification / Phased).

The literal request in FHIR-38419 cannot be cleanly accepted as written
(Disposition A) because the original artifact is being deleted by
FHIR-38267 and the remaining PFE terminology is, by the IG team's own
recent statement, not stable enough to take a permanent THO canonical
URL. At the same time, declining outright (Disposition C) ignores the
substantive direction the PACIO team is already pursuing on the
terminology Zulip thread — Brian Meshell explicitly asked TSMG how to
get an ICF-anchored code system onto THO for STU-2. Disposition B
captures both realities: it closes out the original artifact through
FHIR-38267, commits to THO anchoring for the remaining code systems
when they mature (with a tracked UTG ticket so the work is not lost),
and does so without holding STU-2 hostage to an external code-system
request process the IG team does not control. It also gives the
workgroup a clean "two artifacts, two answers" framing that prevents a
fourth re-open at the next publication cycle.
