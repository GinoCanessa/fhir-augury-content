## Summary

The reporter (Gay Dolin) requests that the IPS guide add explicit guidance on
how to populate the mandatory `status` and `date` (i.e. `effective[x]`) elements
on the IPS Medication-related profiles — Medication (IPS), MedicationRequest
(IPS), and MedicationStatement (IPS) — when the resource is being used to
convey "No Known Medications" via SNOMED CT concept `787481004 | No known
medications` from the [Medications - IPS ValueSet](http://hl7.org/fhir/uv/ips/ValueSet/medication-uv-ips).
Today the value set permits the "no known medications" code, but implementers
have no normative wording telling them what to put in the other mandatory
elements, leading to inconsistent representations on the wire.

This ticket was opened as a follow-up to
[FHIR-52278](https://jira.hl7.org/browse/FHIR-52278) ("Clarify guidance on
representing no known medications"). FHIR-52278 was resolved
*Persuasive with Modification* with two work items: (1) IPS will create a new
value set on Medication (IPS) that excludes the `is-a 787481004` branch, and
(2) a separate ticket (this one — FHIR-53568) was to be raised by Gay Dolin to
work the remaining "guidance on status + date" piece for the resources where
"no known medications" remains permissible. FHIR-53568 is therefore the
narrow, surviving slice of FHIR-52278 that focuses purely on documenting the
expected `status`/`effective[x]` population pattern (and ideally an example).

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Add normative guidance to the IPS narrative for the resources where the
"no known medications" representation is still permitted (MedicationStatement
(IPS) being the recommended carrier per FHIR-52278's resolution; and, while
still allowed, MedicationRequest (IPS) by symmetry). Concretely:

- In the **MedicationStatement (IPS)** profile narrative (and in the
  cross-cutting "Empty Sections and Missing Data" page of IPS), add a
  subsection titled "Representing 'No Known Medications'" that states:
  - `medication.concept` SHALL be a CodeableConcept containing SNOMED CT
    `787481004 | No known medications (situation)` (or another code from the
    new "Medications with no known medications" value set being created via
    FHIR-52278 once available).
  - `status` SHALL be `active` (the assertion of "no known medications" is a
    currently-true statement about the patient).
  - `effective[x]` SHALL be populated with `effectiveDateTime` (or
    `effectivePeriod` with a `start` only) set to the date/time at which the
    statement was made / known to be true. `effective[x]` SHOULD NOT be
    omitted on the basis that no medication exists, because the IPS profile
    constrains it to MS 1..1 (`populate-if-known`) and the statement itself
    has a known temporal anchor.
  - `subject` SHALL reference the patient.
- Add an inline JSON example ("MedicationStatement-uv-ips-no-known-medications")
  that materialises the rules above so implementers have a copy/paste anchor.
- For **MedicationRequest (IPS)**, either (i) repeat the same status/date
  rules with `status = active` and `authoredOn` = date the statement was
  made, or (ii) add a sentence cross-referencing the MedicationStatement
  pattern as preferred (since FHIR-52278 already discourages MedicationRequest
  for this purpose).
- Cross-link from the new wording to FHIR-53425 (the R6 MedicationStatement
  "we don't recommend this approach" note) so IPS readers see the broader
  Pharmacy WG context.

#### Justification

This is exactly the gap implementers (AU PS in particular) have hit and
documented in the Medication > "No current medications" Zulip thread and the
AU Patient Summary poll. Picking `active` + a real `effectiveDateTime` /
`authoredOn` is consistent with what AU PS already settled on, with FHIR-52278's
resolution direction, and with the existing MS 1..1 `populate-if-known`
constraint on `MedicationStatement.effective[x]`. Filling this gap *before* a
significant number of vendors implement divergent patterns is the explicit
motivation Heath Frankel gives in the FHIR-52278 thread.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than enriching the per-resource narrative with `status`/`date` rules,
use the IPS *section-level* "Empty Sections and Missing Data" mechanism as
the primary recommended pattern for "no known medications" and treat the
`787481004`-bearing MedicationStatement as a fall-back:

- Update the "Empty Sections and Missing Data" page to recommend
  `Composition.section[medications].emptyReason` with code
  `nilknown` (or the closest IPS-aligned data-absent reason) as the canonical
  way to communicate "no known medications".
- Retain the MedicationStatement-with-SNOMED-787481004 pattern as legacy /
  permitted, and *only there* document the `status = active` +
  `effective[x] = date asserted` rule, marked clearly as "deprecated in favour
  of the section emptyReason approach".
- Coordinate the wording with FHIR-53425 (R6 Pharmacy guidance) and the
  Resolution of FHIR-52278.

#### Justification

This shifts the "No Known X" semantics out of the resource layer (where
`MedicationStatement` was never really designed to represent absence) and into
the document/composition layer, which is structurally a better fit for "this
section is intentionally empty". It also avoids forcing implementers to
populate a `MedicationStatement.effective[x]` whose meaning ("when was the
patient's lack of medications true?") is awkward. Trade-off: it diverges from
the AU PS implementation pattern that FHIR-52278's commenters described
(MedicationStatement + `1234391000168107`), and the IPS Pharmacy WG has on
record (FHIR-53425) that MedicationStatement *can* carry this — so Disposition
B re-opens a settled debate.

---

### Disposition C: Decline

#### Proposal

Decline FHIR-53568 on the basis that the substantive design decision is
already being addressed by FHIR-52278 (new value set excluding `is-a
787481004` on Medication (IPS)) and FHIR-53425 (R6 MedicationStatement
narrative + value-set update + "not recommended" note). Close FHIR-53568 as
*Not Persuasive — Already Addressed*, with a comment pointing to those two
tickets and to the IPS "Empty Sections and Missing Data" page.

#### Justification

The narrow ask — "tell us how to populate `status` and `date`" — is implicitly
answered the moment FHIR-52278's value-set surgery and FHIR-53425's R6
narrative land: the use of `787481004` on MedicationStatement is being
de-recommended, and the existing 1..1 MS constraint on
`MedicationStatement.effective[x]` already tells implementers it must be
populated when known. Spending workgroup cycles on a parallel, IPS-only
narrative risks contradicting the R6 wording that Pharmacy WG is currently
finalising.

---

### Recommendation

**Recommended disposition:** A

FHIR-52278's resolution explicitly carved this work out as "a new tracker
will be created (Gay will create) to discuss and resolve the guidance on
status + date" — that tracker is FHIR-53568, so declining (C) effectively
unwinds a decision the IPS WG has already taken. Disposition B is
architecturally cleaner but reopens the AU/Pharmacy/IPS debate that
FHIR-52278 just closed and would leave the pre-R6 community without
implementable guidance for the MedicationStatement pattern they are already
using. Disposition A delivers the concrete sentences and example the reporter
asked for, aligns with the AU PS worked example (`status = active`,
`effectiveDateTime = date of statement`), is consistent with the existing MS
1..1 `populate-if-known` constraint on `MedicationStatement.effective[x]`,
and can — and should — explicitly point to FHIR-53425 so IPS and R6
MedicationStatement narratives stay in lock-step.
