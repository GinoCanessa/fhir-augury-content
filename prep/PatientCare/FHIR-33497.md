## Summary

This ticket reports an internal inconsistency within the Ophthalmology
Retinal FHIR Implementation Guide in how `DiagnosticReport.code` is
constrained across two sibling profiles. The Visual Field Report profile
constrains the report's coded type by **slicing `DiagnosticReport.code.coding`**
and fixing values inside one of the slices. The OCT Report profile, in
contrast, constrains the same element by **directly fixing
`DiagnosticReport.code.coding.code`** (and presumably its `system`).
The reporter (Craig Newman) asks whether the divergence is intentional
— if it carries a meaningful semantic difference, the IG should explain
it; if not, the two profiles should be aligned to the same pattern so
implementers see one consistent constraint style across the IG.

A follow-up comment from the reporter notes that slicing-to-fix is an
established pattern elsewhere in FHIR (e.g., `Observation.category` in
the base Observation resource), so either approach is technically
defensible — the question is consistency within this IG, not whether
slicing is legitimate.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Align the Visual Field Report and OCT Report profiles in the
Ophthalmology Retinal IG so that `DiagnosticReport.code` is constrained
the same way in both. Concretely: rewrite the Visual Field Report
profile to drop the slice on `DiagnosticReport.code.coding` and instead
constrain the element using the same mechanism the OCT Report profile
uses today — i.e., a direct fixed/pattern value on
`DiagnosticReport.code` (using `patternCodeableConcept` with the
intended LOINC `system` + `code`, or `fixedCodeableConcept` if a single
exact code is required). Remove the slice discriminator and the slice
definition. Verify the Visual Field LOINC code(s) currently fixed
inside the slice are preserved as the pattern/fixed value on the
unsliced element.

#### Justification

The reporter's complaint is purely about implementer experience: two
sibling profiles in the same IG, constraining the same element of the
same resource, should not require validators and authors to reason
about two different constraint styles unless the styles encode
different meaning. The simpler "fixed/pattern coding" form is
sufficient when each profile's report type is a single concept, which
is what the OCT profile already assumes. Adopting the OCT pattern
across the IG produces a single, easy-to-explain rule for IG
implementers ("each report profile fixes its own report code"), reduces
the surface area for slicing-related authoring mistakes, and is
backwards-compatible for any instance that already conforms to the
existing Visual Field slice (a single fixed coding still validates
against a `patternCodeableConcept` with the same `system`+`code`).

---

### Disposition B: Alternative Approach

#### Proposal

Take the inverse direction: standardise the IG on the **slicing**
pattern instead of the fixed-coding pattern. Update the OCT Report
profile (and any other report profile in the IG that uses the
fixed-coding form) to slice `DiagnosticReport.code.coding` on
`system`+`code` (closed slicing, `min=1`) and fix the report code
inside that slice, mirroring the existing Visual Field Report profile.
Add a short authoring note in the IG narrative (e.g., on the
"Profiling conventions" or equivalent page) documenting that all
report-type bindings in this IG use the slicing form, and citing
`Observation.category` as the precedent the reporter raised.

#### Justification

Slicing-to-fix is forward-compatible: it leaves room for future
profiles or derived profiles to add additional codings (e.g., a
secondary classification system, a national extension code) alongside
the fixed primary code without re-opening the constraint, whereas a
top-level `patternCodeableConcept` already permits that, but a
top-level `fixedCodeableConcept` does not. If the IG anticipates
multi-coded report types (LOINC + a national or specialty code system),
preserving the slicing form across all profiles avoids a future
breaking refactor. The trade-off versus Disposition A is more authoring
ceremony today in exchange for extensibility tomorrow, and it aligns
with the precedent the reporter himself cited
(`Observation.category`).

---

### Disposition C: Decline

#### Proposal

Decline to change either profile. Instead, add a short clarifying
paragraph to the Ophthalmology Retinal IG documenting that both
constraint styles are used, that they are semantically equivalent for
the purposes of this IG (a conformant instance must carry the
specified `system`+`code` on `DiagnosticReport.code.coding`), and that
the choice is an authoring-style artefact rather than a substantive
difference. Close the ticket as "Persuasive with Modification – not
related" / "Not Persuasive" with that narrative addition as the only
change.

#### Justification

Both constraint forms validate the same set of instances for the
report-code use case at hand, so no implementer is actually blocked —
the cost of the inconsistency is cognitive, not functional. Touching
two profiles in a published IG to fix a stylistic divergence risks
introducing churn (regenerating snapshots, retesting validation,
updating example instances, refreshing IG narrative cross-links) for
no behavioural gain. A short narrative note is a much cheaper way to
remove the implementer surprise the ticket actually describes.

---

### Recommendation

**Recommended disposition:** A

The ticket is, at its core, a request to remove an inconsistency that
serves no documented purpose. Disposition A does that with the smallest
authoring change (the OCT profile is already in the simpler form, so
only the Visual Field profile needs to be rewritten) and the lowest
cognitive load on implementers — every report profile in the IG ends
up looking the same. Disposition B is defensible only if Patient Care
and the Ophthalmology project actually anticipate multi-coded report
types in this IG; absent that signal, it adds slicing complexity with
no concrete payoff. Disposition C is the cheapest in author-hours but
explicitly preserves the inconsistency the reporter is asking the WG
to remove, so it does not really resolve the ticket — it just
documents the problem. Patient Care should confirm with the
Ophthalmology project leads whether multi-coded report types are
planned; if "no" (which is the default assumption for a
single-modality report profile), Disposition A is the right call.
