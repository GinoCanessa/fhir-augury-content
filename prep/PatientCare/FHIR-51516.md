## Summary

Rick Lisseveld reports that the `SDOHCC-ObservationScreeningResponse`
profile in the US SDOH Clinical Care IG (STU 2.3) tells implementers
that, when assessment-instrument context is needed, an Observation
generated from a QuestionnaireResponse "can reference QuestionnaireResponse
via `Observation.derivedFrom`" (section 14.13.1.2 Usage). However, the
profile's `Observation.derivedFrom` reference target for
QuestionnaireResponse is only the US Core 7.0 QuestionnaireResponse
profile, which (the reporter asserts) is not based on the SDC
QuestionnaireResponse profile defined in the Structured Data Capture
(SDC) IG. The reporter asks whether the SDC QuestionnaireResponse
profile should also be added as an allowed reference target on
`Observation.derivedFrom`, so that Observations derived from SDC
QuestionnaireResponses can be conformantly referenced.

The ticket has been triaged with a current resolution of **Not
Persuasive**: the WG's draft resolution holds that the SDOHCC OSR
profile already allows references to the US Core 7 QuestionnaireResponse
profile, which (per the resolver) is itself based on the SDC
QuestionnaireResponse profile, so no change is required. The
cross-reference index surfaces only one related artifact —
`Observation.derivedFrom` — and no linked Jira tickets, Zulip threads,
or GitHub items were found.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the `SDOHCC-ObservationScreeningResponse` profile so that the
`Observation.derivedFrom` element's reference target slice for
`QuestionnaireResponse` accepts **both** the US Core 7.0
QuestionnaireResponse profile **and** the SDC QuestionnaireResponse
profile (`http://hl7.org/fhir/uv/sdc/StructureDefinition/sdc-questionnaireresponse`).
Concretely, add a second target profile to the existing
`Reference(QuestionnaireResponse)` type on `Observation.derivedFrom` in
the SDOHCC-ObservationScreeningResponse StructureDefinition (and any
companion guidance text in section 14.13.1.2 Usage), so that an
Observation produced from an SDC-conformant QuestionnaireResponse can
be referenced without coercing the source instance through the US Core
profile.

#### Justification

The reporter's concern is that the IG's narrative explicitly invites
implementers to point `derivedFrom` at a QuestionnaireResponse, but
its formal type slicing only names the US Core 7.0 profile. If — as
the reporter asserts — the US Core 7.0 QuestionnaireResponse profile
is not derived from (and does not subsume) the SDC profile, then an
SDC-conformant QuestionnaireResponse instance is not guaranteed to
also satisfy the US Core profile, and a strict validator would flag
the reference as non-conformant. Explicitly listing the SDC profile as
an additional allowed target eliminates that ambiguity and aligns the
profile's machine-readable conformance with its narrative guidance,
matching the broader FHIR pattern of enumerating every profile a
target may conform to when more than one is acceptable.

---

### Disposition B: Alternative Approach

#### Proposal

Rather than modifying the type slicing, **clarify the IG narrative and
the resolution itself**: keep `Observation.derivedFrom` referencing
only the US Core 7.0 QuestionnaireResponse profile, but add (a) an
explicit note in section 14.13.1.2 Usage stating that any
QuestionnaireResponse referenced from a SDOHCC OSR must conform to the
US Core 7.0 QuestionnaireResponse profile, and (b) guidance (or a
worked example) showing how an SDC-conformant QuestionnaireResponse
can also assert conformance to the US Core 7.0 profile (e.g., by
listing both canonicals in `meta.profile`) so SDC-driven workflows can
still satisfy the constraint. If the WG additionally wants to leave
the door open to SDC, it could reference SDC informatively in the
Usage section without changing the binding.

#### Justification

This preserves the current resolver's position — that US Core 7.0 is
the canonical target — while addressing the underlying authoring
confusion that prompted the ticket. It avoids broadening the allowed
profile set (which would obligate SDOHCC consumers to handle two
QuestionnaireResponse shapes) and keeps validation deterministic, but
gives implementers a documented path for SDC-sourced data. It is also
the lower-risk change for STU maintenance: a narrative clarification
is non-breaking, whereas adding a target profile is a substantive
profile change that may require renotification of conformance tooling.

---

### Disposition C: Decline (Not Persuasive)

#### Proposal

Close the ticket as **Not Persuasive**, consistent with the current
draft resolution: state that the SDOHCC OSR profile already permits
references to the US Core 7.0 QuestionnaireResponse profile, that the
US Core 7.0 QuestionnaireResponse profile is itself derived from /
compatible with the SDC QuestionnaireResponse profile, and that
therefore SDC-shaped QuestionnaireResponse instances are already
acceptable as `derivedFrom` targets without any change to
SDOHCC-ObservationScreeningResponse.

#### Justification

If the lineage assertion holds — i.e., US Core 7.0
QuestionnaireResponse is a derivative/refinement of (or otherwise
fully compatible with) the SDC QuestionnaireResponse profile — then
adding SDC as a second target profile is redundant and increases the
profile surface area without enabling any instance that wasn't
already valid. Declining is also consistent with the WG's stewardship
of STU profile churn: every additional reference target is a forward
compatibility commitment, and the WG should not pay that cost when
the existing target already covers the use case. The reporter's
concern is best resolved by clarifying the lineage in the response
rather than expanding the binding.

---

### Recommendation

**Recommended disposition:** B

The factual hinge of the ticket is whether US Core 7.0
QuestionnaireResponse actually subsumes SDC QuestionnaireResponse. The
reporter explicitly asserts it does not ("which is not the base of the
SDC QuestionnaireResponse in the IG"); the current Not-Persuasive
resolution asserts it does. That disagreement is exactly the kind of
ambiguity that will recur with implementers, so closing the ticket
without addressing it (Disposition C) is likely to generate repeat
tickets even if the resolver is technically correct. Disposition A
(adding SDC as a second target) is defensible but is a substantive
profile change late in STU 2.3 and obligates every SDOHCC consumer to
handle two QuestionnaireResponse shapes — a non-trivial cost if the
lineage really does collapse them. Disposition B threads the needle:
keep the binding stable, but use the IG narrative (and the ticket
resolution itself) to state the lineage explicitly and to show
SDC-driven implementers how to satisfy the constraint via
`meta.profile`. If, on review, the WG confirms that US Core 7.0
QuestionnaireResponse genuinely is **not** SDC-compatible, the
recommendation should fall back to Disposition A.
