## Summary

The reporter requests that a conformance statement in the US
Person-Centered Outcomes (PCO) FHIR IG be strengthened from `SHOULD` to
`SHALL`. Specifically, the current sentence — "Data Consumer Systems
SHOULD be capable of storing the data elements for other uses (such as
record keeping of data used for clinical use)" — would be changed to
`SHALL`, with a carve-out: "unless for privacy or security reasons."
The intent is to elevate persistence of received PCO data from a
recommendation to a requirement, while preserving an exception for
privacy/security-driven non-storage. No cross-referenced Jira tickets,
Zulip threads, or GitHub items were surfaced for this ticket, so the
request stands on its own.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Update the PCO IG conformance statement on Data Consumer Systems to
read: "Data Consumer Systems **SHALL** be capable of storing the data
elements for other uses (such as record keeping of data used for
clinical use), **unless prevented by privacy or security
requirements**." Apply the change to the IG's formal specification /
conformance page (the page corresponding to the PACIO PFE
`formal_specification.html` cited in the comment), and keep the
existing PACIO-aligned wording elsewhere in the section consistent.
No structural FHIR artifact (profile, extension, capability statement)
needs to change — this is a normative-prose change in the IG
narrative.

#### Justification

The ticket's underlying need is interoperability: if storage of
received PCO data is only "recommended," Data Provider Systems cannot
rely on Consumers to retain the data they share, which undermines the
"other uses" use cases the IG explicitly calls out (clinical record
keeping). Tightening to `SHALL … unless for privacy or security
reasons` aligns conformance with the IG's stated intent, while the
privacy/security carve-out preserves a defensible exception consistent
with HL7 security and privacy guidance. Per RFC 2119, this kind of
"requirement with explicit exception" is a well-formed `SHALL`
statement.

---

### Disposition B: Alternative Approach

#### Proposal

Keep the base conformance statement as `SHOULD`, but add an explicit
**conditional `SHALL`** alongside it that scopes the requirement to
the use cases the reporter cares about. For example: "Data Consumer
Systems **SHOULD** be capable of storing the data elements for other
uses (such as record keeping of data used for clinical use). Data
Consumer Systems that retain received PCO data for clinical
record-keeping purposes **SHALL** persist the data elements as
received, unless prevented by privacy or security requirements."
Optionally, define a CapabilityStatement-level expectation (e.g., an
`expectation` extension of `SHALL` on the relevant interaction) only
for Consumers that declare a `record-keeping` profile/role, leaving
non-record-keeping Consumers under `SHOULD`.

#### Justification

The committee comment notes the SHALL/SHOULD wording was deliberately
copied from PACIO PFE and is shared across Patient Care and Patient
Empowerment IGs. Unilaterally promoting `SHOULD` → `SHALL` in PCO
diverges from that harmonized baseline and could fragment conformance
across closely related IGs. A conditional `SHALL` keeps PCO aligned
with PACIO/PE for the general case while still giving the reporter a
hard requirement for the scenario they actually need to rely on
(clinical record-keeping). It also lets the change land in PCO
without first negotiating a coordinated change across every IG that
inherited the PACIO wording.

---

### Disposition C: Decline

#### Proposal

Mark the ticket as **Persuasive with Modification — Deferred** /
**Not Persuasive for this ballot cycle**, and resolve by leaving the
text at `SHOULD`. Record the rationale that PCO's conformance verbs
were intentionally aligned with PACIO PFE and other Patient
Care/Patient Empowerment IGs, and that any change should be made
across that family of IGs together rather than in PCO alone. Track
the request for the next IG update cycle, contingent on a
cross-IG (PACIO + PC + PE) discussion.

#### Justification

This mirrors the position already stated in the lone committee
comment on the ticket: the wording was copied for consistency, and
diverging unilaterally creates implementer surprise across IGs that
intentionally share conformance language. Promoting `SHOULD` to
`SHALL` is also a backwards-incompatible conformance change — systems
that were conformant under the current text could become non-conformant
overnight, with the only escape hatch being a self-asserted "privacy
or security" exception that is not currently testable. Until the
exception is defined precisely enough to be evaluated (who decides,
how is it documented, how is it surfaced in a CapabilityStatement),
declining is defensible.

---

### Recommendation

**Recommended disposition:** B

Disposition B best reconciles the reporter's legitimate concern
(Consumers silently dropping data the IG was designed to share) with
the committee's stated constraint (keep PCO's conformance language
aligned with PACIO and the broader PC/PE IG family). A blanket
`SHOULD → SHALL` change (Disposition A) is a backwards-incompatible
conformance shift that the committee has already flagged as needing
cross-IG coordination, and it leans on a `unless for privacy or
security reasons` exception that is currently too vague to test.
Outright declining (Disposition C) leaves a real interoperability gap
that the reporter has correctly identified. A scoped, conditional
`SHALL` — applied only to Consumers that hold themselves out as
performing clinical record-keeping — gives implementers a hard
obligation where it matters, preserves PACIO-aligned language for the
general case, and creates a clean handoff for the cross-IG
harmonization conversation the committee said it wants to have in a
future update.
