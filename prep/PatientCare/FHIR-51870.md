## Summary

The Bundle (IPS) profile defines a sliced `Bundle.entry` listing every
profile that may appear in an IPS document. The two IPS Device profiles
— [Device (IPS)](https://build.fhir.org/ig/HL7/fhir-ips/StructureDefinition-Device-uv-ips.html)
and [Device — Performer or Observer (IPS)](https://build.fhir.org/ig/HL7/fhir-ips/StructureDefinition-Device-observer-uv-ips.html)
— are advertised by the [IPS Server CapabilityStatement](https://build.fhir.org/ig/HL7/fhir-ips/CapabilityStatement-ips-server.html)
and are referenced from DeviceUseStatement (IPS), Observation Results —
Radiology (IPS), and Procedure (IPS), but they are **not present** in
the Bundle (IPS) entry slicing. The reporter asks that they be added so
the Bundle profile is internally consistent with the rest of the IG.

A first attempt to apply the change was made (originally proposed as
Persuasive / Non-substantive) but had to be reverted: adding both
profiles as separate Bundle.entry slices produced a slicing
discrimination failure in the IPS examples, because the current
`(type, profile)` discriminator pair on `Bundle.entry.resource` does not
provide enough distinguishing information between `DeviceUvIps` and
`DeviceObserverUvIps`. Reviewed on the 8/27/2025 IPS call, the work
group deferred the change pending an alternative discrimination
approach (Heath Frankel taking the action), and the final resolution is
*Considered for Future Use* — to be revisited after IPS 2.0
publication.

No incoming or outgoing cross-references were registered in FHIR Augury
for this ticket (`cross-referenced` returned 0 hits), so there are no

through the augury index. There is, however, a directly-relevant

## Details

**Description:**

The Bundle (IPS) profile is missing reference to the two IPS Device profiles:

- [Device (IPS)](https://build.fhir.org/ig/HL7/fhir-ips/StructureDefinition-Device-uv-ips.html)
- [Device - Performer or Observer (IPS)](https://build.fhir.org/ig/HL7/fhir-ips/StructureDefinition-Device-observer-uv-ips.html)

As these profiles are listed as supported profiles in the
[IPS Server Capability Statement](https://build.fhir.org/ig/HL7/fhir-ips/CapabilityStatement-ips-server.html)

[DeviceUseStatement (IPS)](https://build.fhir.org/ig/HL7/fhir-ips/StructureDefinition-DeviceUseStatement-uv-ips.html),

and [Procedure (IPS)](https://build.fhir.org/ig/HL7/fhir-ips/StructureDefinition-Procedure-uv-ips.html),
these two Device profiles should be included in the Bundle (IPS) entry
slices.

having two slices for each introduces validator noise. Will revisit
after 2.0 publication since there may be multiple options to address.

## Related Zulip Discussions

### [conformance > Slicing Bundle.entry](https://chat.fhir.org/#narrow/stream/conformance/topic/Slicing%20Bundle.entry)

Josh Mandel opened the thread on 2024-05-06 using the IPS
`Bundle-uv-ips.json` as a worked example of `Bundle.entry` slicing with
two discriminators (`type` on `resource` plus `profile` on `resource`).
Chris Moesel explained the rationale for keeping `type` alongside
`profile`: it lets a slice constrain to a resource type without naming
a profile, which validators expect when only `profile` is used. Grahame
Grieve confirmed the converse case is exactly the problem here —
"there's more than one profile for some types of resource, so type of
resource is not enough to discriminate." Josh experimented with
removing either the `type` discriminator or the `profile` element on
Device, and found that removing only one still validates, but
combining a `type` discriminator with a `pattern`-based discriminator
(as he did in his CGM example) is the recommended way to disambiguate
profiles that share a resource type without paying the
profile-discriminator cost. Chris confirmed that profile discriminators
are expensive and that for cases where multiple profiles share a type
(or where one slice is type-only), `type + pattern` is preferable. The
thread closed with the participants noting that the discussion had
crossed from methodology into IPS-specific design and should continue
on `#IPS` — which is the same loop FHIR-51870 has now landed in.

Participants: Josh Mandel, Chris Moesel, Grahame Grieve.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

In `HL7/fhir-ips`, edit `input/profiles/Bundle-uv-ips.json` to add two
new `Bundle.entry` slices to the existing slicing definition:

- `Device` — bound to the `Device-uv-ips` profile.
- `DeviceObserver` — bound to the `Device-observer-uv-ips` profile.

Both slices should follow the existing pattern in Bundle (IPS):
`min: 0`, `max: *`, `Bundle.entry.resource.type = Device`, with
`Bundle.entry.resource.profile` constrained to the corresponding
canonical. Update Bundle (IPS) examples that include any Device
instance to populate `Bundle.entry.resource.meta.profile` with the
exact IPS Device profile canonical so the existing
`(type, profile)` discriminator pair can resolve them. No change to
the discriminator block itself, no change to the IPS Server
CapabilityStatement (which already lists both profiles), and no change
to the referencing profiles (DeviceUseStatement, Observation Results -
Radiology, Procedure).

#### Justification

This is the literal, minimal change asked for and matches the existing
pattern used by every other supported IPS resource. Bundle (IPS)
slicing is `open`, so the practical impact is documentation and reader
clarity rather than a new conformance constraint. Aligning Bundle
(IPS) with the IPS Server CapabilityStatement removes an obvious
internal inconsistency and lets FHIR validators surface a useful
"unknown slice" warning for Device entries that omit the profile tag.
The slicing-error problem that caused the original revert is solvable
by ensuring the IPS Device examples carry `meta.profile` for the
correct IPS Device canonical — which is the standard recipe used
throughout the IG.

---

### Disposition B: Alternative Approach — change the discriminator strategy

#### Proposal

Rather than adding the two Device slices under the current
`{type:type on resource, type:profile on resource}` discriminator pair,
restructure the Bundle (IPS) entry slicing to use
`type + pattern` discrimination for any group of profiles that share a
resource type but cannot be told apart by profile alone:

- Keep the existing `type` discriminator on `Bundle.entry.resource`.
- Replace the `profile` discriminator with a **`pattern` discriminator**
  on a stable element that distinguishes the IPS Device profiles
  (e.g., `Device.type`, or a pattern on
  `Device.specialization.systemType` for the observer Device).
- Add the two Device slices (Disposition A) once the discriminator can
  separate them.

This is the approach recommended by Chris Moesel and Josh Mandel in
the `#conformance > Slicing Bundle.entry` thread, and is the
investigation Heath Frankel was assigned on the 8/27 IPS call.

#### Justification

This addresses the root cause that blocked the original Persuasive
resolution: the existing discriminator pair cannot distinguish
`DeviceUvIps` from `DeviceObserverUvIps`, so adding them as slices
under the current scheme creates a real slicing error in instances —
not just a stylistic one. `type + pattern` is cheaper than profile
discrimination and is the documented pattern for distinguishing
profiles that share a resource type without unique profile markers.
The trade-off is scope: it touches the discriminator design of every
existing Bundle (IPS) slice and may require example updates across
the IG, so it is realistically post-2.0 work — which matches the
ticket's current resolution timing.

---

### Disposition C: Decline (defer to post-2.0, no spec change for IPS 2.0)

#### Proposal

Hold the current resolution: keep Bundle (IPS) as-is for IPS 2.0 and
do not add Device or Device-Observer slices. Treat the ticket as
*Considered for Future Use* with an explicit follow-up tracker for
post-2.0 (linked to the Heath Frankel investigation) so the
discriminator-redesign work in Disposition B can be picked up cleanly
when the publication freeze ends.

#### Justification

Bundle (IPS) slicing is open, so the omission does not block
implementers from including Device resources in IPS bundles. The
attempt to apply the change broke validation against the IPS
examples, which is the exact harm the IPS 2.0 ballot cycle is trying
to avoid late in the cycle. Declining for now preserves the ballot
state, keeps the existing examples valid, and defers the structural
discriminator question (Disposition B) to a release where it can get
the design attention it needs. The cost is a documented inconsistency
between Bundle (IPS) and the IPS Server CapabilityStatement, which
the WG has already acknowledged on the 8/27 call.

---

### Recommendation

**Recommended disposition:** **C** for IPS 2.0, with a committed
follow-up to land **B** post-2.0.

The work group has already converged on this position: the original
Persuasive resolution was reverted specifically because adding the
slices under the current discriminator scheme breaks the IPS Bundle
examples, and the 8/27/2025 IPS call delegated a Heath-Frankel
investigation into a different discriminator approach. The
`#conformance > Slicing Bundle.entry` discussion supports the same
conclusion from a methodology angle — `type + profile` cannot
discriminate same-resource-type profiles, and the right fix is
`type + pattern`. Trying to ship Disposition A inside the IPS 2.0
window risks shipping a profile that fails its own examples;
Disposition B is the correct long-term answer but is too large a
discriminator change to land late in the ballot. Disposition C
preserves IPS 2.0 stability, leaves the supported-profile
acknowledgement in the IPS Server CapabilityStatement (so implementers
are not surprised), and explicitly schedules the structural fix —
exactly what the existing resolution comment commits to. The work
group should decide whether to attach a placeholder ticket for the
post-2.0 Disposition B work so it does not get lost.
