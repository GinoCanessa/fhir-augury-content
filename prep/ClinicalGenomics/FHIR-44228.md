## Summary

Ron G. Parker requests that the Clinical Genomics work group anchor the
CodeSystem and ValueSet URLs used by the Genomics Reporting IG in
**terminology.hl7.org (THO)** rather than under `hl7.org/fhir`, in line
with the TSC-approved [Terminology Expectations for IG
Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy. The reporter asks that the IG initiate the UTG process to
migrate the affected code system(s) into THO; if the work group
believes a code system is too tightly coupled to the IG to migrate, it
must request a formal exception from TSMG. Ticket history indicates
this concern was raised against a specific IG-defined CodeSystem that
is the target of a *required* binding (Joel Schneider previously pulled
it from a block vote for that reason) and that Patrick Werner has been
investigating whether the underlying *cis*/*trans* concepts could be
sourced from LOINC, NCIt, or another existing CS, possibly restructured
or removed. The work group previously resolved the issue as
"Considered for Future Use" (Bob D / Aly K, 13-0-1) on the grounds that
the CS was still tightly coupled to the IG, then reverted that
resolution in December 2024 to re-open it post STU3 publication. No
GitHub cross-references were found, and the only related Zulip thread
is a Clinical Genomics block-vote announcement that proposed the same
deferred outcome.

## Proposed Dispositions

### Disposition A: Accept As Requested

#### Proposal

Initiate the UTG process to migrate the IG-defined CodeSystem(s)
referenced by FHIR-44228 (the *cis*/*trans* CS that Patrick Werner is
investigating, and any sibling CodeSystems anchored at `hl7.org/fhir/...`
inside the Genomics Reporting IG) from `hl7.org/fhir/...` to
`http://terminology.hl7.org/CodeSystem/...`. Concretely: (1) submit a
UTG proposal for each CS, (2) update the IG's CodeSystem resource
`url`s, (3) update every dependent ValueSet `compose.include.system`
and every dependent ElementDefinition `binding.valueSet`/CodeableConcept
canonical reference, (4) leave a redirect/aliasing note in the IG so
that existing implementations continue to validate, and (5) close the
ticket as **Persuasive** once UTG accepts the submission. Companion
ticket FHIR-44230 should be progressed in lockstep.

#### Justification

This is the action the TSC-approved [Terminology Expectations for IG
Developers](https://confluence.hl7.org/display/TSMG/Terminology+Expectations+for+IG+Developers)
policy requires, and Kevin Power's 2025-05-26 walk of the TSMG decision
tree concluded that the CodeSystem qualifies for THO (no suitable
external or existing THO CS, not example-strength only, not so tightly
bound to the IG that it must ship with the IG). Lynn Laakso re-opened
the issue specifically to act on it post-STU3, so deferring again would
ignore the work group's own correction. The fact that the CS is the
target of a *required* binding (per Joel Schneider's 2024-02-15 note)
makes the migration *more* important, not less, because a required
binding to a non-THO URL is exactly the pattern TSMG is trying to
eliminate.

---

### Disposition B: Alternative Approach

#### Proposal

Treat FHIR-44228 as a request to **eliminate** the IG-defined CS
rather than to relocate it. Pursue Patrick Werner's investigation: map
the *cis*/*trans* (and any other affected) concepts to existing codes
in **LOINC**, **NCIt**, or **Sequence Ontology**, restructure the
affected ValueSets to compose those external systems instead, and
**delete** the IG-defined CodeSystem. Where a 1:1 external code is not
available, file an HTA request (analogous to the entries in FHIR-43478
for ClinVar / RefSeq / dbSNP / SequenceOntology) rather than coining a
new HL7 code. Resolve as **Persuasive with Modification**, citing
Patrick Werner's 2026-01-30 plan.

#### Justification

The underlying TSMG goal is to stop IG-local terminology fragmentation,
not specifically to move HL7-coined codes to THO; using the appropriate
external code system achieves the goal more cleanly and aligns with
Genomics Reporting's broader pattern of leveraging LOINC, HGNC, HGVS,
SequenceOntology, etc. (see the FHIR-43478 External Code Systems
appendix work). It also avoids the long-tail UTG maintenance burden of
HL7 owning a small genomics-specific CS. The trade-off versus
Disposition A is timeline risk and a potential breaking change to
existing implementers' bindings; the work group should commit to a
deprecation path in the next ballot if it picks this option.

---

### Disposition C: Decline

#### Proposal

Re-resolve as **Considered for Future Use** (or **Not Persuasive with
Modification**), with a documented TSMG exception request stating that
the CodeSystem is tightly coupled to the IG and should remain anchored
under `hl7.org/fhir/...` for the current STU cycle, to be revisited
when the affected resources reach FMM ≥ 3. This restores the 2024-09-03
disposition (vote Bob D / Aly K, 13-0-1) that Lynn Laakso reverted in
December 2024.

#### Justification

This was the work group's prior consensus and the only one with a
recorded vote. If the CS is genuinely tightly coupled to IG-internal
structures (as the original rationale claimed), the TSMG policy itself
provides for an exception. Declining would also avoid the breaking-
change risk to the *required* binding Joel Schneider flagged. However,
this position is hard to defend after Kevin Power's 2025-05-26 decision-
tree walk explicitly concluded the CS is **not** so tightly coupled
that it cannot evolve independently — choosing C requires the work
group to either rebut that analysis or re-classify the CS.

---

### Recommendation

**Recommended disposition:** B

The TSMG policy goal is to anchor terminology in the right place, not
to grow HL7-owned content in THO for its own sake. Patrick Werner's
2026-01-30 note (looking for the concepts in LOINC / NCIt and
restructuring or deleting the IG CS), combined with Kevin Power's
decision-tree analysis showing the concepts are not uniquely HL7's,
points strongly at *eliminating* the IG-defined CodeSystem rather than
just relocating it. This achieves the TSC/TSMG intent of FHIR-44228,
matches the External Code Systems clean-up the work group already
committed to in FHIR-43478, and avoids both the long-term UTG
maintenance burden of a small HL7-owned CS (Disposition A) and the
weak-rationale deferral that the work group itself previously reverted
(Disposition C). The block-vote-era "tightly coupled to the IG" framing
no longer holds up given the work group's own subsequent analysis, so
Disposition C should not be the starting recommendation. Disposition A
remains the safe fallback if Patrick Werner's external-code-system
mapping does not pan out before STU4.
